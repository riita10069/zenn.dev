# 6.4 学習戦略とトレーニングテクニック

アーキテクチャを決めても、学習戦略を間違えれば数 pt の差で量産可否が分かれます。本節では性能を押し上げる代表的なテクニックを扱います。自己教師あり事前学習、マルチタスク損失の自動重み付け、Mixed Precision、Gradient Checkpointing、長尾対策、知識蒸留 (Knowledge Distillation) です。

ここで先に用語を整理します。**Mixed Precision (混合精度)** は学習の途中計算を FP32 (32 ビット浮動小数) ではなく FP16 / BF16 / FP8 などの低精度で行う技法、**Gradient Checkpointing** は順伝播の中間活性化を一旦捨てて逆伝播時に再計算しメモリを節約する技法、**Knowledge Distillation (知識蒸留)** は大きな教師モデルの出力分布を小さな生徒モデルに学ばせる技法、**Curriculum Learning** は易しい例から難しい例へ段階的に学習させる訓練設計です。

## 自己教師あり事前学習：MAE / DINOv2 / CLIP

フリートで集めたログの大半は未ラベルです。ラベル付与には人手コストがかかるため、ラベルなしのまま使える表現を学んでおくと、後段のラベル予算を大幅に節約できます。これが自己教師あり学習 (Self-Supervised Learning, SSL) です。**MAE (Masked Autoencoder)** は画像の一部を隠して復元させる手法、**DINOv2** は自己蒸留で頑健な特徴を学習する手法、**CLIP** は画像とテキストを対照学習で結びつける手法です。代表的手法は次の通りです。

| 手法 | 原理 | 自動運転での使いどころ | 参照 |
|---|---|---|---|
| MAE | 画像の高比率マスクを復元 | 大量の無ラベル車載カメラで ViT を事前学習 | [P15](references#p15) |
| DINOv2 | 自己蒸留 + 画像レベル/パッチレベル一貫性 | 凍結バックボーンでも強い特徴、ラベル少時に有効 | [P16](references#p16) |
| CLIP | 画像-テキストの contrastive 学習 | zero-shot 分類・データ検索・難例マイニング | [P17](references#p17) |

DINOv2 [P16](references#p16) はラベルなしで頑健な特徴を学習し、**バックボーンを凍結したまま線形ヘッドだけ学習しても高精度** を出せることが報告されており、ラベル予算が限られる新地域展開で有効です。MAE [P15](references#p15) は ViT のスケーラブルな事前学習に適し、CLIP [P17](references#p17) は第 4 章の意味ベース検索・zero-shot 難例抽出に活用できます。

少ラベル時の実装上の定石は、(1) DINOv2 の事前学習済み ViT バックボーン (例：`dinov2_vitl14`、出力次元 1024) を読み込み、すべてのパラメータの勾配計算を停止して凍結する、(2) その上にタスク固有の検出ヘッドだけを新規に構築する、(3) ヘッドのパラメータのみを AdamW などのオプティマイザに渡し、比較的高い学習率 (例：1e-3) で短期間に収束させる、という流れです。

Closed-Loop の観点では、新 ODD 追加時に「その地域の無ラベルデータで追加事前学習 → 少量ラベルでファインチューニング」という運用が、ラベルコストを抑えつつ分布シフトに追従する有効な手段です。

## マルチタスク学習と不確実性重み付け

1 つのバックボーンから検出・セグメンテーション・車線検出などを同時学習する **マルチタスク学習 (Multi-Task Learning)** は、特徴共有によりデータ・推論効率を高めます。課題は損失バランスです。タスクごとに損失のスケールが違うため、固定重みでは調整が難しい問題があります。Kendall & Gal の **不確実性 (homoscedastic uncertainty) に基づく自動重み付け** ("Multi-Task Learning Using Uncertainty to Weigh Losses for Scene Geometry and Semantics", CVPR 2018) は、タスクごとの不確実性を学習可能パラメータにして自動でバランスを取る手法です。各タスク $i$ の学習可能な不確実性 $\sigma_i$ を導入し、損失を次のように定義します。

$$
\mathcal{L} = \sum_{i} \frac{1}{2\sigma_i^2}\, \mathcal{L}_i + \log \sigma_i
$$

第 1 項はタスクごとの損失を不確実性で割って自動スケールし、第 2 項は $\sigma_i$ が無限に大きくなる退化を防ぐ正則化です。

実装上は、$\sigma_i$ そのものを学習対象にすると 0 付近で数値が不安定になるため、$\log \sigma_i$ をモデルの学習可能パラメータ (タスク数と同じ次元のベクトル) として保持します。順伝播では、各タスクの損失 $\mathcal{L}_i$ に対し $\exp(-2 \log \sigma_i)$ を係数として掛けて 0.5 を乗じた項と、正則化項 $\log \sigma_i$ を足し合わせ、全タスク分を総和して総合損失とします。学習が進むにつれ、難しいタスクの $\sigma$ は大きくなり、その損失の寄与が自動的に下がる挙動になります。

実装上は、勾配ノルムを揃える GradNorm (タスクごとの勾配ノルムを動的に均一化する手法) などと比較し、検証 mAP/NDS が改善する方を採用します。データ中心の観点では、マルチタスク化により同一シーンに複数タスクのラベルを揃える必要が生じます。どのタスクを蒸留や疑似ラベル (pseudo-label) で代替するかを、ラベルコストと合わせて判断します。

## Mixed Precision と Loss Scaling

大規模モデルを FP32 (32 ビット浮動小数) のまま学習すると、メモリ・速度ともに不足します。**Mixed Precision** は計算の一部を FP16 / BF16 / FP8 に落とし、精度を保ったまま高速化する技法です。FP16 は仮数部 10 ビット・指数部 5 ビット、BF16 は仮数部 7 ビット・指数部 8 ビット (FP32 と同じ動的範囲)、FP8 (E4M3 / E5M2) は H100 以降の最新 GPU で対応する 8 ビット形式を指します。

| 形式 | ビット | 動的範囲 | Loss Scaling | 主な用途 |
|---|---|---|---|---|
| FP32 | 32 | 広い | 不要 | 基準・小規模 |
| FP16 | 16 | 狭い (勾配アンダーフロー) | **必須** | 一般的な高速化 |
| BF16 | 16 | FP32 並 (仮数部が短い) | 不要なことが多い | A100/H100 で安定 |
| FP8 (E4M3/E5M2) | 8 | 非常に狭い | スケーリング必須 | H100 以降の最先端 |

FP16 は動的範囲が狭いため、小さな勾配がアンダーフロー (極小値が表現できずゼロになる現象) を起こします。これを避けるには、損失を定数倍してから逆伝播し、勾配を割り戻す **Loss Scaling (損失スケーリング)** が必須です。一方 BF16 は指数部が FP32 と同じで動的範囲が広いため、Loss Scaling なしでも安定することが多く、対応 GPU (A100 / H100) では BF16 が扱いやすい選択です。

PyTorch の混合精度 API を使う場合の実装手順は、(1) 動的 Loss Scaling を担う GradScaler を 1 つ作成する、(2) 各イテレーションで `optimizer.zero_grad()` の後に `autocast` コンテキストを開き、その中で順伝播と損失計算を行う (BF16 なら `dtype=torch.bfloat16`、FP16 なら `torch.float16`)、(3) 損失をスケールして逆伝播し、`scaler.step(optimizer)` でアンダーフローが検出されたステップを自動スキップ、(4) `scaler.update()` でスケール係数を自動調整、という 4 段で実装します。FP16 ではこの仕組みが必須ですが、BF16 では Loss Scaling を実質無効化してもほぼ問題なく動作します。

## Gradient Checkpointing のメモリ-計算トレードオフ

層が深くバッチサイズが大きいと、順伝播で生成される **活性化 (activation、各層の中間出力)** が GPU メモリを圧迫します。これを解決するのが **Gradient Checkpointing (勾配チェックポイント)** です。順伝播時に中間活性化を捨て、逆伝播時に必要な分だけ再計算します。メモリは大幅に削減できますが、順伝播を一部やり直すため計算時間が増えるトレードオフがあります。

| 設定 | 活性化メモリ | 計算時間 | 使いどころ |
|---|---|---|---|
| なし | 大 (層数 $N$ に比例) | 基準 | メモリに余裕あり |
| 選択的 ($\sqrt{N}$ 等間隔でセグメント化) | 小（$O(\sqrt{N})$）| +20〜35% 程度 | バッチサイズ/解像度を上げたい |
| 全層 checkpoint | 最小（$O(1)$ 近傍。1 層分の活性化のみ）| +35〜50% 程度 | メモリが極端に逼迫した場合のみ |

実装は、Transformer ブロックや Conv ブロックなど「再計算しても安全な単位」をラッパーで包み、学習中だけ checkpoint を有効化する設計が扱いやすいです。具体的には、対象ブロックの `forward` を PyTorch の `torch.utils.checkpoint.checkpoint` で呼び出すように差し替え、評価時は通常の forward に戻す分岐を入れます。`use_reentrant=False` を指定すると、最近の PyTorch で推奨されている非リエントラント実装が使われ、外側で `torch.compile` などと組み合わせる際の互換性が高まります。

BEV/Occupancy 系は高解像度特徴で活性化が大きくなりやすく、Gradient Checkpointing でバッチサイズや BEV 解像度を引き上げられることが多いです。第 6.5 節の FSDP と組み合わせると、さらに大きなモデルを学習できます。

## 長尾分布への対処：Focal Loss と Class-Balanced Loss

自動運転のデータでは日常シーンが大多数を占め、インシデントや希少クラス (歩行者・自転車) は稀です。これを「長尾分布 (long-tailed distribution)」と呼び、放置すると学習が多数派に引きずられて少数派の recall (再現率) が落ちます。代表的な対策が **Focal Loss** と **Class-Balanced Loss** です。Focal Loss は易しい例の損失を抑え難しい例に集中させる手法、Class-Balanced Loss はクラスごとの「有効サンプル数」で損失を重み付けする手法です。

- **focal loss** [AL5](references#al5)：易しい (well-classified) サンプルの損失を抑え、難しいサンプルに集中させます。
$$
\mathrm{FL}(p_t) = -\alpha_t (1 - p_t)^{\gamma} \log(p_t)
$$
$(1-p_t)^\gamma$ が易例の寄与を減衰させ、$\gamma$ (通常 2.0) が集中度を制御します。

- **class-balanced loss** [AL6](references#al6)：サンプル数 $n$ の「有効サンプル数」$ (1-\beta^n)/(1-\beta) $ の逆数で各クラスを重み付けし、頻度の偏りを補正します。

focal loss を実装する際は、(1) ロジットからシグモイドで確率 $p$ を求め、(2) 通常のバイナリ交差エントロピー (reduction なし) を要素ごとに計算し、(3) 正解ラベルが 1 のときは $p$、0 のときは $1-p$ を $p_t$ として取り出し、(4) `alpha * (1 - p_t)^gamma * ce` を要素ごとの損失とし、最後にバッチで平均する、という流れで構成します。$\alpha$ は 0.25、$\gamma$ は 2.0 を起点に調整します。

これらは Curriculum Learning や難例マイニングと併用します。難例のみに集中して分布が偏らないよう、ランダムサンプルとのバランスを保つことが重要です。

## Curriculum Learning と難例マイニング

**Curriculum Learning (カリキュラム学習)** は、簡単な例から難しい例へ段階的に学習させる訓練設計です。人間の学習過程に着想を得ており、いきなり難しい例を見せると勾配が暴れて発散しやすいのを避けられます。自動運転では次のように設計できます。

- ステージ 1：白昼・晴天・単純道路。
- ステージ 2：夜間・雨天・複雑交差点・工事区間。
- ステージ 3：インシデント/ヒヤリハットの極端ケース、稀な長尾シナリオ。

難例マイニング (hard example mining) は、モデルが誤った/損失が高いサンプルを次サイクルで優先サンプリングします。第 4 章の active learning・error mining と密接に連携し、Closed-Loop に自然に組み込まれます。オフライン (評価/リプレイから誤検出シーン抽出) とオンライン (学習中に損失上位を記録) の両方を使い分けます。

## 知識蒸留と温度調整

**知識蒸留 (Knowledge Distillation, KD)** は、巨大な「教師モデル」の出力分布を模倣するように小型の「生徒モデル」を学習させる手法です。教師の確率分布には正解ラベル以外の「クラス間の類似性 (dark knowledge)」が含まれており、これを生徒に伝えると、ハードラベルだけで学習するよりも精度を保ちやすいことが知られています。車載向けの精度-レイテンシ調整に有効です。教師のソフトラベルを温度 $T$ で平滑化し、KL ダイバージェンスで一致させます。

$$
\mathcal{L}_{KD} = (1-\lambda)\,\mathcal{L}_{CE}(y, \sigma(z_s)) + \lambda\, T^2 \cdot \mathrm{KL}\big(\sigma(z_t/T)\,\|\,\sigma(z_s/T)\big)
$$

$T$ を大きくするとクラス間の関係 (dark knowledge) が強調され、$T^2$ は蒸留項の勾配スケールを補正します。実務では $T=2\sim4$、$\lambda=0.5\sim0.9$ あたりから探索します。

蒸留損失の実装手順は、(1) 生徒のロジットと正解ラベルの通常の交差エントロピー損失を計算する、(2) 生徒・教師のロジットをそれぞれ温度 $T$ で割り、生徒側は log-softmax、教師側は softmax を取って KL ダイバージェンスを計算する (バッチで平均)、(3) KL 項に $T^2$ を掛けて勾配スケールを揃える、(4) 通常損失に重み $1-\lambda$、蒸留項に重み $\lambda$ を掛けて足し合わせる、という流れになります。

Closed-Loop では、大規模モデルで新アーキテクチャ/戦略を検証し、改善が確認できたら蒸留・量子化 (第 6.6 節) で車載モデルに反映する流れが定着しています。

## オプティマイザ・スケジューラ・正則化

大規模で長時間の学習には、安定性と収束性が重要です。

- **オプティマイザ**：AdamW、大バッチ向けに LAMB、メモリ節約に AdaFactor や 8-bit Adam [T4](references#t4)。
- **スケジューラ**：warmup 付き cosine decay で初期安定化と後半微調整を両立。
- **正則化**：weight decay、Dropout、Stochastic Depth、ラベルスムージング。

8-bit optimizer [T4](references#t4) はオプティマイザ状態を 8bit に量子化し、状態メモリを大幅削減します。第 6.5 節の分散学習と組み合わせ、メモリ制約下でのバッチ拡大に寄与します。

## 学習戦略を組み合わせる際の効果検証の難しさ

学習戦略は単独でも効くものが多く、闇雲に全部入れたくなりますが、複数を同時投入すると効果が干渉して何が効いたのか分からなくなる典型的な落とし穴があります。設計判断としては「効果が独立しやすいものから順に積む」「各段階で評価指標の何を見るかを先に決める」の二つが要点です。

まず Mixed Precision は最も副作用が少ない高速化なので、最初に有効化します。BF16 対応 GPU (A100 / H100) では Loss Scaling を実質無効化したまま速度・メモリ改善が得られるため、安全側の選択です。FP16 では動的範囲の狭さから勾配アンダーフローが起き、Loss Scaling なしでは学習が収束しないため、対応 GPU で BF16 を選ぶ判断が現実的です。次に Gradient Checkpointing は計算時間と引き換えにメモリを削減する純粋なトレードオフで、バッチサイズや BEV 解像度を上げたい場合に限定して Transformer 層単位で導入します。両者は独立に効くため、順次導入しても効果切り分けが容易です。

DINOv2 等の自己教師あり事前学習は、新 ODD 投入時のラベル予算を最も大きく節約できる打ち手です。新地域の無ラベル車載カメラで追加事前学習し、少量ラベルでヘッドだけ学習する 2 段プロトコルにすると、ラベル付与コストを桁単位で削減できることが報告されています。バックボーン凍結のまま線形ヘッドを学ぶ構成は、効果が出ているか出ていないかを早く判定できる検証フレンドリーな設計でもあります。

マルチタスク化したときに損失バランスをどう決めるかは、設計判断として最も難しい部類です。固定重みでまず動かし、不確実性重み付け ($\sigma_i$ 学習可能) で自動バランスを試し、勾配ノルムを揃える GradNorm と比較する、という順序が効果検証しやすい流れです。検証 NDS が改善した手法だけ採用するという原則を守らないと、論文で報告された改善が自社データでは効かないケースを掴みきれません。

長尾対策は Focal Loss・Class-Balanced Loss・難例マイニングを順に試し、各段階で安全クリティカルクラス (歩行者・自転車) の recall 変化を独立に確認するのが要点です。全体の mAP/NDS は改善しているのに歩行者 recall が落ちている、という静かな劣化は ODD 別評価でしか検出できず、車載化後にインシデントとして顕在化します。難例だけに集中すると分布が偏るため、ランダムサンプルとのバランスを保つことも忘れがちな注意点です。

知識蒸留は、6.6 節の量子化・枝刈りと組み合わせて車載モデルを作る最終段で効きます。温度 $T=2\sim4$ と重み $\lambda=0.5\sim0.9$ を起点に探索し、$T^2$ で勾配スケールを揃える理屈を踏まえれば、教師-生徒間のクラス間類似性の伝達がハードラベル単独より効率的になります。

## 本節の振り返り

学習テクニックは、自己教師あり事前学習でラベル予算を節約し、不確実性重み付けでマルチタスクの損失バランスを自動化し、Mixed Precision と Gradient Checkpointing でメモリと速度のトレードオフを管理し、Focal/Class-Balanced Loss で長尾の見落としを抑え、知識蒸留で車載モデルへ精度を引き継ぐ——という、本書の Closed-Loop の各段階に対応する独立技術の集合です。重要なのは「全部入れて改善」ではなく、効果が独立しやすいものから段階的に積み、検証 NDS と安全クリティカルクラスの recall を必ず分解して確認する規律です。とくに BF16 + Loss Scaling 不要、$\sigma_i$ 学習可能なマルチタスク重み付け、Focal の $(1-p_t)^\gamma$ による易例減衰、KD の温度 $T^2$ による勾配スケール補正は、いずれも数式上の意味を理解しないと効果検証ができない種類の技法で、本書のデータ中心の主張を支える定量基盤の一部です。
