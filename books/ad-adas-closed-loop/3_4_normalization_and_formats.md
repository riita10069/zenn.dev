# 3.4 データ正規化とフォーマット設計

この節では、クラウド側でのデータ正規化 (data normalization) とフォーマット設計を、具体的なツールと数値に踏み込んで扱います。MCAP / rosbag2 / WebDataset / Parquet / Zarr / HDF5 の選定マトリクス、バイナリとメタデータの分離、WebDataset のシャード設計式、画像・点群・CAN ログの圧縮率を整理します。

一度正規化したフォーマットは、第4章以降のシーン検索・データ選択・ラベリング・学習・シミュレーション・運用監視のすべてで共通の土台になります。ここでの設計が Closed-Loop 全体の速度とコストを左右します。

## フォーマット選定マトリクス

ロガー固有フォーマットを共通フォーマットへ正規化する際、用途ごとに最適なフォーマットが異なります。

フォーマットを並べて比較します。各行のかっこ内は ML 出身者向けの 1 行説明です。

| フォーマット | 種別 | 強み | 弱み | 主用途 |
|---|---|---|---|---|
| MCAP [ST4](references#st4)（センサログ用の汎用コンテナ、ROS 2 公式記録形式）| 自己記述型コンテナ | 高速シーク、チャンク圧縮、スキーマ埋込、ROS 2 公式 | 学習直読には不向き | 中間形式・可視化・再生 |
| rosbag2 (.db3) [ST5](references#st5)（ROS 2 旧来の SQLite ベースのログ形式）| SQLite ストレージ | ROS 2 ツール互換 | シーク性能・移植性が MCAP に劣る | レガシー（rosbag2 v0.15+ は MCAP backend がデフォルトのため新規プロジェクトでは MCAP を選択）|
| WebDataset [ST6](references#st6)（tar 形式の学習用シャードフォーマット）| tar シャード | 連続 I/O 高速、S3 親和、学習直読 | ランダムアクセス不可 | 大規模学習 |
| Parquet [ST7](references#st7)（列指向の分析用ファイル形式、Spark / Trino 標準）| 列指向 | 述語プッシュダウン、メタデータ集計 | バイナリ大容量に不向き | メタデータ・CAN 信号 |
| Zarr [ST8](references#st8)（チャンク化された N 次元配列形式、numpy 互換）| チャンク配列 | N 次元配列、部分読込、並列 | エコシステム限定 | Occupancy / voxel グリッド |
| HDF5（科学計算で広く使われる階層配列フォーマット）| 階層配列 | 成熟、自己記述 | 並列書込・クラウド I/O が弱い | 研究・単一ノード |

### MCAP を中間形式の標準に据える

MCAP [ST4](references#st4) は 2023 年に ROS 2 の推奨記録フォーマットとなった自己記述型コンテナで、チャンク単位の圧縮（zstd/lz4）、メッセージインデックスによる高速シーク、スキーマの埋め込みを備えます。rosbag2 のデフォルトだった SQLite3 (.db3) [ST5](references#st5) と比べ、シーク性能・移植性・圧縮効率で優れるため、**ロガー → MCAP（中間形式・可視化・再生用）→ WebDataset（学習用）**という多段構成が現代的です。

ROS 2 ロガー側での記録は、`ros2 bag record` コマンドに対し、ストレージプラグインを `mcap`、圧縮モードをファイル単位、圧縮フォーマットを `zstd`（高圧縮率と高速デコードを両立する圧縮ライブラリ）に指定し、対象トピックを全件 (`-a`) で取り込む構成にします。これで rosbag2 の出力を SQLite3 から MCAP + zstd に切り替えられます。MCAP は Foxglove Studio [ST11](references#st11) でそのまま可視化でき、3.9 節のデータブラウザと直結します。フリート全体で数十 PB を扱う場合、MCAP を最終形式にはせず、学習向けに WebDataset へ再正規化します。

### WebDataset のシャード設計

WebDataset [ST6](references#st6) は tar アーカイブをシャーディングするフォーマットで、同一ベースネームのファイル群（`12345.jpg` + `12345.json` + `12345.lidar.bin`）が 1 サンプルを構成します。tar は連続アクセスに最適化され、S3 互換ストレージで高スループットを出せます。

シャード数とサイズは、ワーカー並列とシャッフル品質から逆算します。

```
シャード数 = 総サンプル数 / シャードあたりサンプル数
推奨: シャードあたり 1,000〜10,000 サンプル、1 シャード 100MB〜1GB
制約: シャード数 ≥ (GPU 数 × DataLoader worker 数)   # 各workerが独立シャードを担当
```

例：1,000 万サンプル、シャードあたり 5,000 サンプルなら 2,000 シャード。これを 8 GPU × 8 worker = 64 並列で読む場合、2,000 ≫ 64 を満たすので十分です。シャードが少なすぎるとシャッフルが偏り、多すぎるとオープン/クローズのオーバーヘッドが増えます。

学習側の DataLoader は、シャード URL をワイルドカード（例：`s3://bucket/train-{000000..001999}.tar`）で指定し、シャード単位のシャッフルを有効化したうえで、各サンプルのキー（`cam.jpg` / `lidar.bin` / `label.json` のような同一ベースネーム）を画像・点群・ラベルの 3 つ組としてデコードする実装にします。S3 への並列ストリーミング読み出しは WebDataset 公式ライブラリで標準サポートされており、シャードあたりの順次走査と全体のシャッフルが両立します。

規模別の出発点としては、TB 級（数十車両、数千時間まで）なら MCAP のまま学習読み込みも可能で再正規化は不要、数百 TB〜PB 級になると MCAP（中間）+ WebDataset（学習用）の二段構成と Parquet メタデータの併用が現実的です。Occupancy / voxel grid のような体積データを多用する場合は Zarr で別管理し、Iceberg のメタデータからストア URI を参照させます。

## バイナリとメタデータの分離

どのフォーマットでも、バイナリ（画像・点群・波形）とメタデータ（タイムスタンプ、センサ ID、ODD 属性、ラベル状態）を論理的に分離します。バイナリはオブジェクトストレージ、メタデータは Parquet [ST7](references#st7) やカタログ（3.7 節）に保存します。実運用のクエリは「メタデータで数百万件に絞る → 数千〜数万件を抽出 → 対応バイナリだけ読む」流れが主流のため、メタデータ設計が検索性能を決めます。

Frame メタデータの Parquet スキーマには、最低限次のカラムを持たせます。

| カラム | 型 | 役割 |
|---|---|---|
| `frame_id` / `drive_id` / `scene_id` | string | 階層 ID（3.6 節の Iceberg テーブルと結合可能にする） |
| `ts_utc` | timestamp(ns) | 補正後の基準クロック時刻 |
| `sensor_config_id` | string | 適用センサ構成のバージョン |
| `odd_segment` / `weather` | string | ODD・気象タグ（カタログのタクソノミと整合） |
| `sync_quality` / `label_status` | string | 同期品質ラベル・ラベリング状態 |
| `blob_uri` | string | オブジェクトストレージ上の本体 URI |
| `codec` | string | 圧縮方式・レベル（後述） |

このスキーマで Parquet として保存しておけば、Iceberg / Trino / Spark から述語プッシュダウンによる効率的なフィルタが可能になります。

## 圧縮方式と削減率

用途ごとに圧縮ポリシーを分けます。長期保存は可逆寄り、学習用はデコード速度重視、プレビューは非可逆で軽量にします。

| データ | 推奨方式 | 概算削減率 | 典型エンコードパラメータ | 備考 |
|---|---|---|---|---|
| カメラ画像 | H.265 / JPEG-LS（JPEG-LS は可逆 JPEG）| 圧縮後 25〜40%（60〜75% 削減） | H.265 crf=23, JPEG-LS predictor=3 | 学習静止画は JPEG、アーカイブは元コーデック保持 |
| LiDAR 点群 | Draco [ST12](references#st12)（Google 製の点群圧縮ライブラリ）| 約 10:1（90% 削減） | quantization=14 bit | 量子化ビット数で可逆性を調整（8 bit で 80%、14 bit で 95% 削減）|
| LiDAR（可逆要件） | LAZ / カスタムバイナリ+zstd（LAZ は LiDAR 業界の可逆圧縮形式）| 約 2〜3:1 | zstd level=12 | 再利用性重視 |
| CAN/車両バス | zstd | 80〜90% 削減 | level=9〜12 | 高頻度スカラは冗長性大 |
| Occupancy グリッド | Zarr + blosc（blosc は数値配列向けの高速圧縮）| 用途依存 | blosc clevel=5, shuffle | 疎なボクセルで高圧縮 |

### zstd レベルの選択

zstd は圧縮レベル（1〜22）で速度と圧縮率を調整できます。CAN ログのようなテキスト/スカラ系は高レベルでも高速に 80〜90% 削減できますが、学習シャードでは I/O とデコードのバランスが重要です。

| zstd レベル | 相対速度 | 用途 |
|---|---|---|
| 1〜3 | 最速 | オンライン/リアルタイムログ、lz4 代替 |
| 9〜12 | 中速 | 学習シャード（I/O バインド時） |
| 19〜22 | 低速・高圧縮 | コールドアーカイブ、CAN 長期保存 |

いずれの場合も「どのデータがどの圧縮方式・レベル・バージョンで保存されているか」をメタデータ（上記 `codec` フィールド）に記録し、将来のフォーマット移行や再エンコードに備えます。圧縮方式の変更自体も、性能・コスト・安全性への影響を評価しながら進める Closed-Loop の設計項目で、変更時は新旧で同一 Drive を二重保存して再現実験を行い、性能差が 1 mAP ポイント以内に収まることを確認してから本番反映する運用が安全です。

## 本節の振り返り

フォーマットは「一度決めると後戻りが極めて高くつく」設計判断です。MCAP と WebDataset を二段で持つ設計には深い意味があります。MCAP は人間と ROS 2 ツールが扱う「再生・可視化・デバッグ用」の形、WebDataset は GPU が連続 I/O で読みたい「学習用」の形で、両者を一つにまとめようとすると必ずどちらかが歪みます。実務でよく見る失敗は、ロガー出力の MCAP をそのまま学習に使い続け、ランダムアクセスやシャード並列で I/O ボトルネックを引き起こすケース、もう一つは WebDataset シャードを並列度から逆算せずに「シャード数 < GPU × worker 数」となり、一部 worker が遊ぶケースです。後者は GPU 利用率が伸びない原因として顕在化しますが、最初は学習コードの問題に見えて切り分けに時間がかかります。データ中心 AI の主張は「学習コードよりデータの形を整えるほうが効く」ですが、本節の WebDataset シャード設計と Parquet メタデータ分離はその実装そのものです。バイナリとメタデータを分け、`codec` まで含めてメタデータ化することで、将来の再エンコードや形式移行が「データ全量の再アップロード」ではなく「メタデータ駆動の差分処理」で完結します。
