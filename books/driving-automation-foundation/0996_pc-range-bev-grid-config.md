# point cloud range と BEVグリッド解像度

## ひとことで言うと

鳥瞰図（BEV, Bird's-Eye-View、車を真上から見下ろした地図のような表現）を作るとき、「この地図は車の周り何メートル四方をカバーするか」を決めるのが point cloud range（`pc_range`）、「その範囲を縦横いくつのマス目に区切るか」を決めるのが BEV グリッド解像度（`bev_h × bev_w`）です。前者が物理的な広さ、後者が地上分解能（マス1個あたり何メートルか）を決めます。この2つの設定だけで、BEV 空間の地図としての性質（どこまで見えるか・どれだけ細かいか・メモリと計算量がいくらか）がほぼ確定します。地味なハイパーパラメータに見えて、認識精度・計算コスト・適用シーンを左右する根幹の設計判断です。

## 直感的な理解

カメラ画像はそのままでは「斜めから見た景色」です。自動運転の計画（どこをどう走るか）を立てるには、これを「真上から見た地図」に直すと格段に扱いやすくなります。前車との距離、車線の位置、交差点の形が、地図上の座標として直接読めるからです。

このとき必ず決めねばならないのが「地図の範囲」と「地図の細かさ」です。地図には必ず縁があります。無限に広い地図は作れないので、「車を中心に前後左右・上下何メートルまでを地図にするか」を決めます。これが `pc_range` です。pc は point cloud（点群）の略で、もともと LiDAR 点群を扱う範囲を指す慣習的な名前ですが、カメラのみの手法でも「BEV がカバーする 3 次元の箱の範囲」を表す変数名として広く使われます。

そして、その範囲を「何マスに区切るか」が解像度 `bev_h`（縦のマス数）× `bev_w`（横のマス数）です。同じ広さでもマスを細かくすれば細い車線や小さな物体まで表現できますが、マス数は縦×横で増えるため、メモリと計算量が二乗のオーダーで膨らみます。「広く見たい」「細かく見たい」「軽くしたい」は同時には満たせないトレードオフです。範囲を一定にして解像度だけ上げると分解能は細かくなるがメモリが増え、解像度を一定にして範囲だけ広げると遠くまで見えるが1マスが粗くなる、という二者択一が常につきまといます。

## 基礎: 前提となる概念

**自車座標系（ego frame）**: 自車を原点 (0,0,0) とした座標系です。慣習として X=前方向、Y=左方向、Z=上方向（右手系）、単位はメートルです。`pc_range` はこの座標系で範囲を指定します。

**地上分解能（ground sampling distance）**: BEV グリッドの1マスが地上で何メートルに相当するかです。範囲をマス数で割れば求まります。小さいほど細かい（高解像度）。これがカメラ画像でいうところの「ピクセルあたりの実寸」に相当し、表現できる最小の物体サイズを規定します。

**ボクセル/ピラー（voxel / pillar）**: 3D 空間を区切った立方体がボクセル、縦方向（高さ）に積んだ柱状の単位がピラーです。BEV では平面のマスごとに高さ方向の柱を立て、その柱に沿って数点をサンプルすることがあります。

**クエリ数とメモリ**: BEV グリッドの各マスを1つのクエリ（学習可能な特徴ベクトル）とすると、クエリ数は `bev_h × bev_w` です。各クエリが C 次元（例 256）の特徴を持つので、BEV 特徴マップは `[B, C, bev_h, bev_w]`。アテンションの計算・中間テンソルはクエリ数に比例（自己注意なら二乗）して効くため、解像度がメモリ予算を直接決めます。

## 仕組みを詳しく

### pc_range の中身

`pc_range` は6つの数 `[x_min, y_min, z_min, x_max, y_max, z_max]` で、自車座標系の箱の範囲を表します。代表的な対称設定（BEVFormer 原論文の nuScenes 設定）は次のとおりです。

```
pc_range = (-51.2, -51.2, -5.0, 51.2, 51.2, 3.0)
            x_min  y_min  z_min  x_max  y_max  z_max
```

- X（前後）: -51.2m 〜 +51.2m（前後対称）
- Y（左右）: -51.2m 〜 +51.2m（左右対称、合わせて 102.4m 四方の正方形）
- Z（高さ）: -5.0m 〜 +3.0m（地面より下 5m、上 3m。坂・トンネルの下り、高架・歩道橋を含む余裕）

一方、運転計画では前方の重要度が高いため、X を前方寄りに非対称化する設計もよく使われます。たとえば後方を浅く前方を厚く取る `(-60, -60, -5, 120, 60, 3)`（後方60m・前方120m）のような「forward-biased（前方寄り）」設定です。どちらが正しいというものではなく、「対称な周辺認識を重視するか、前方計画を重視するか」で選ぶ設計判断です。X 方向だけ非対称にすると、前方の遠距離まで地図に収められる代わりに、同じマス数なら1マスが粗くなります。高速道路のように前方の遠距離が重要なシーンでは前方寄り、市街地の交差点のように全方位が重要なシーンでは対称、という使い分けになります。

### bev_h × bev_w（解像度）

例として `bev_h = 450, bev_w = 300` なら、合計 450×300 = 135,000 マス。各マスに 256 次元特徴が対応し、BEV 特徴マップは `[B, 256, 450, 300]` です。

1マスあたりの地上分解能を計算します（前方寄り pc_range `(-60,-60,-5,120,60,3)` の場合）。

- X 方向: 範囲 (120 - (-60)) = 180m を 450 マスで割る → 1マス 0.40m
- Y 方向: 範囲 (60 - (-60)) = 120m を 300 マスで割る → 1マス 0.40m

約 40cm 四方のマスで世界を区切ることになります。40cm あれば、車線幅（約3.5m）は約9マス、乗用車（全長約4.5m）は約11マスで表現でき、計画に十分な細かさです。逆に言えば、40cm より小さい物体（細い縁石、薄い路面標示）は1マスに潰れて表現が曖昧になります。参考に BEVFormer 原論文は 200×200 で 102.4m 四方を区切るので 1マス約 0.51m、上記設定はやや高解像度です。

解像度がメモリに直結する具体例として、BEV 融合を高解像度（450×300 程度）で動かすと、バッチサイズによってはミドルレンジ GPU の VRAM を容易に超えます。一般に、解像度を上げてマス数が増えるほど、また 1 マスあたりの柱サンプル点数を増やすほど、中間テンソルが線形〜二乗で膨らみ、バッチサイズの上限が下がります。「解像度・柱点数・バッチサイズ・VRAM」が互いに引っ張り合う関係にあり、どれか1つを上げると他のどれかを下げる必要があります。

### Z 方向の使われ方: ピラーサンプリング

Z 範囲（例 -5m〜+3m）は、平面マスごとに「縦の柱（pillar）」を立て、その柱に沿って何点かをサンプルするのに使われます（例 `num_points_in_pillar = 4` で高さ違い4点）。

なぜ高さ方向に複数点を取るかというと、カメラ画像だけでは奥行き（深さ）が曖昧だからです。「この BEV マスに映っているのは地面の物か、高い建物か」が一意に決まりません。そこで複数高さを試しに投影し、アテンションの重みで「どの高さが効くか」をデータから学ばせます。これは明示的な深度推定をせずに深度の曖昧さを吸収する仕組みです。点数を増やすと表現は増えますが、サンプルと集約のコストが線形に増えます。

### 設定が下流に与える影響（処理の流れ）

```
pc_range と (bev_h, bev_w)
   │
   ▼
参照点の生成:
   [0,1] の正規化グリッドを作り、pc_range で実メートルにスケール
   → reference_points_3d: [bev_h*bev_w, num_z, 3]
   │
   ▼
2D 投影:
   3D 点をカメラ行列で画像 2D 座標へ投影
   → 「BEV のこのマスは画像のここ」という対応を得る
   │
   ▼
集約: その座標で grid_sample し特徴を集約
   → BEV 特徴マップ [B, C, bev_h, bev_w]
```

設計が良ければ、解像度を変えても下流の軌跡計画モジュールは作り直し不要です。下流が正規化座標（0〜1）で `grid_sample` を使う設計なら、(8,8) でも (450,300) でもそのまま動きます。範囲と解像度を「物理」と「離散化」として疎結合に保つことが、再利用性の鍵です。逆に、下流がピクセル単位の絶対座標を前提にしていると、解像度を変えるたびに後段の係数を直す必要が生じ、保守性が悪化します。

## 手法の系譜と主要論文

**Lift, Splat, Shoot（Philion & Fidler, ECCV 2020, arXiv:2008.05711）**。各画素に深度分布を予測し、特徴を 3D へ持ち上げ（lift）、BEV ボクセルへばらまく（splat）手法です。やはり BEV のボクセル範囲と解像度を事前に固定値で与えます。深度を密に予測するためメモリが重く、`pc_range` を広く・解像度を高くするとコストが急増するトレードオフがあります。LSS は範囲・解像度を「設計者が選ぶハイパーパラメータ」として明示的に切り出した初期の代表例です。

**BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）**。BEV グリッド 200×200、`pc_range` 約 [-51.2, 51.2]（nuScenes）を採用しました。核は「学習可能な BEV クエリがカメラ画像の投影位置から特徴を集める」ことで、その前提として物理範囲と解像度を固定で与えます。解像度を上げると遠方の小物体は捉えやすくなりますが、クエリ数が二乗で増えメモリ・計算が重くなります（200×200=4万 → 450×300=13.5万、約3.4倍）。

**BEVDet（Huang et al., 2022, arXiv:2112.11790）**。LSS 系の代表で、BEV 範囲・解像度の選び方が検出精度に直結することを実験で示しました。範囲を前方に伸ばすほど遠方検出に有利ですが、近傍の解像度を犠牲にしうるという関係を議論しています。前方非対称 pc_range は、この「前方を重視する」発想と整合します。続く BEVDet4D は時系列を加え、範囲・解像度の設定が時間方向の整合（過去フレームを現在の範囲へ整列する処理）にも関わることを示しました。

系譜として共通するのは「BEV は連続な物理空間を有限の格子に離散化したもの」という認識です。範囲（どこまで）と解像度（どれだけ細かく）を分けて指定する設計は、これらの手法すべてに受け継がれています。

## 論文の実験結果（定量データ）

**BEVFormer（nuScenes 3D 検出）**: BEV 解像度 200×200、`pc_range` [-51.2, 51.2] で NDS 約 0.517、mAP 約 0.416 を報告。NDS（nuScenes Detection Score）は検出精度と位置・速度・属性誤差を総合した 0〜1 のスコアで高いほど良く、mAP（平均適合率、0〜1）も同様です。論文のアブレーションでは、BEV クエリ数（解像度）を下げると精度が低下する傾向が示され、解像度が精度の上限を規定することが分かります。一方で解像度を上げるほど良いわけではなく、画像特徴の有効解像度（バックボーンが出す特徴マップの細かさ）や計算予算とのバランスで頭打ちになります。BEV を細かくしても、元の画像特徴が粗ければ詰め込む情報がない、というわけです。

**LSS（nuScenes BEV セグメンテーション）**: 車両 BEV セグメンテーションの IoU（予測と正解の重なり率、0〜1 で高いほど良い、車両でおよそ 0.32 程度）で当時のベースラインを上回りました。論文では BEV グリッド解像度・範囲を変えた分析があり、解像度を上げると細部のセグメンテーションが改善する一方、メモリと計算が急増することが示されます。

**BEVDet 系の知見**: 入力画像解像度・BEV 解像度・範囲を変えるアブレーションで、いずれも検出 mAP/NDS に明確な影響が出ます。とくに「遠距離物体は範囲を伸ばさないとそもそも地図に入らない」「近距離の小物体は解像度を上げないと潰れる」という、範囲と解像度の役割分担が定量的に確認されています。これは「pc_range で何が地図に入るか」「bev_h×bev_w で何が表現できるか」を別々に決める意義の実証です。また入力画像解像度を上げると mAP が顕著に伸びる（BEVDet で数ポイント規模）一方、計算量も比例して増えることが示され、3つの解像度（画像・BEV・範囲）が独立に効くことが分かります。

**メモリの定量関係**: クエリ数は `bev_h × bev_w` で二乗的に増えるため、解像度を 200×200 から 450×300 に上げると約 3.4 倍のクエリ数になり、アテンションの中間テンソルとメモリも比例して増えます。高解像度かつ大バッチでメモリ不足（OOM）に至るのはこの関係の必然で、解像度・バッチ・VRAM が三つ巴のトレードオフにあることを示します。

## メリット・トレードオフ・限界

**メリット**

- 物理範囲（pc_range）と解像度（bev_h×bev_w）を分けて指定でき、「どこまで見るか」と「どれだけ細かく見るか」を独立に調整できる。
- 前方非対称な pc_range により、計画上重要な前方を厚くカバーできる（用途に合わせた設計が可能）。
- 下流が正規化座標で動く設計なら、解像度を変えても後段モジュールを作り直さなくてよい。

**トレードオフ・限界**

- 解像度を上げるとクエリ数が縦×横で増え、メモリ・計算量が二乗オーダーで急増する（高解像度＋大バッチは OOM になりやすい）。
- 範囲を広げると同じマス数では1マスが粗くなる（分解能低下）。範囲と解像度は本質的にトレードオフ。
- `pc_range` は「車に対する世界の広さ」を固定で決め打ちするため、想定外に遠い／高い物体は地図の外に出て表現できない（高速道路の遠方車両、跨線橋の上の構造物など）。
- 一様な格子は「近距離は細かく、遠距離は粗くてよい」という運転の実情に合っておらず、遠距離の無駄なマスにメモリを割きがち。

**未解決の課題（研究視点）**: 一様格子の非効率を解消する非一様/極座標 BEV（近距離を密、遠距離を疎にする）、解像度を動的に変える適応的グリッド、範囲外の物体を扱う長距離 BEV などが研究テーマです。また、固定 pc_range はデータセットのセンサ配置・走行環境に依存して最適値が変わるため、汎化（別環境への転移）を考えると「範囲・解像度をどうデータから決めるか」自体が課題になります。あるデータセットで調整した範囲・解像度を別環境にそのまま持ち込むと、カメラの画角や搭載高さが違うために性能が落ちることがあり、設定の転移性は実用上の悩みどころです。

## 発展トピック・研究の最前線

近年は、平面 BEV の解像度・範囲という2D 的な離散化を超えて、3D 占有グリッド（occupancy grid）で高さ方向も陽に区切る手法が広がっています。これは Z 方向のピラーサンプリングを「数点」から「密なボクセル列」へ拡張したもので、立体構造（高架下の通行可否など）を表現できる代わりにメモリが一段と重くなり、解像度・範囲の設計がより切実になります。また、極座標や対数距離スケールの BEV（近距離を密に、遠距離を粗に配分する）は、一様格子のメモリ非効率を緩和する方向として注目されています。時系列で複数フレームの BEV を重ねる手法では、自車の移動に応じて過去 BEV を現在の `pc_range` に整列（warp）させる必要があり、範囲と解像度が時間方向の整合性にも関わってきます。さらに、長距離（前方 200m 超）を扱う高速道路向けの BEV では、遠方を粗く・近傍を密にする可変解像度や、複数スケールの BEV を併用する設計が研究されており、「一様格子という前提そのものを崩す」流れが進んでいます。
