# 動画フレームデコードライブラリ比較

## ひとことで言うと
mp4 などの動画ファイルから「○秒目の1枚」を取り出すには、専用のデコードライブラリ(動画を生のピクセルに展開するプログラム)が必要です。代表的なものに PyAV・Decord・TorchCodec などがあり、それぞれ「狙ったフレームへ速く頭出しできるか」「並列処理と相性が良いか」「出力がそのままテンソルか」「GPU でデコードできるか」が違います。学習で必要なのは各タイミングの特定の1フレームだけなので、この「1枚を速く正確に取り出す」性能が、データパイプライン全体の速度を左右します。

## 直感的な理解
動画は、巻物のように先頭から順に巻かれたテープだと思ってください。「100コマ目だけ見たい」と思っても、テープを途中だけ切り出すことはできず、近くの「区切りの良い場所」まで早送りして、そこから順に再生して100コマ目にたどり着く必要があります。この「早送り + そこから再生」が速いか遅いかが、ライブラリの差です。

なぜ単純に「100コマ目」を直接読めないのか。動画は容量を小さくするために、隣り合うコマの差分だけを記録しているからです。ほとんどのコマは「前のコマからどれだけ変わったか」しか持っておらず、単独では絵になりません。一定間隔に置かれた「完全な絵を持つコマ」までさかのぼり、そこから差分を順に足していって、初めて目的のコマが復元できます。この差分記録こそが、動画ファイルが静止画の連番より圧倒的に小さく済む理由であり、同時にランダムアクセスを難しくする原因です。

学習データローダの観点では、この「頭出し(シーク)」を何百万回も繰り返すことになります。1回あたり数ミリ秒の差でも、500万回繰り返せば数千秒〜数時間の差になり、学習時間が数倍変わります。だからデコードライブラリの選択は、地味ですが効きます。

## 基礎: 前提となる概念

動画の圧縮構造を、用語とともに丁寧に説明します。

- コンテナ(container): mp4, mkv, webm などのファイル形式。中に映像ストリーム・音声ストリーム・メタデータをまとめた「入れ物」です。拡張子はコンテナを指し、中身の圧縮方式(コーデック)とは別の概念です。同じ .mp4 でも中の映像が H.264 のことも H.265 のこともあります。
- コーデック(codec、coder-decoder の略): 映像の圧縮方式。H.264(AVC, Wiegand et al. 2003)、H.265(HEVC, Sullivan et al. 2012)、AV1、VP9 など。新しいものほど圧縮率が高い(同じ画質で容量が小さい)代わりに、デコードの計算が重くなる傾向があります。
- フレーム(frame): 動画の1コマ。圧縮の都合で3種類あります。
  - Iフレーム(キーフレーム、Intra frame): その1枚だけで完全な画像に復元できるフレーム。一定間隔(例: 数十フレームごと、これを GOP = Group of Pictures と呼ぶ)に置かれます。静止画の JPEG に近い独立した絵です。
  - Pフレーム(Predicted frame): 直前のフレームとの差分(動きベクトル + 残差)だけを持つ。前を復元してからでないと作れません。
  - Bフレーム(Bi-directional frame): 前後両方のフレームを参照して差分を持つ。最も圧縮効率が高いが、復元順が表示順とずれるため処理が複雑。
- シーク(seek): 目的の位置まで頭出しする操作。任意フレームを取るには、その手前のキーフレームまで戻り、そこから順にデコードします。シークのコストはキーフレーム間隔(GOP の長さ)に依存します。GOP が長い(キーフレームが疎)ほど、1フレーム取り出すのに余計に多くのフレームを展開する必要があり、遅くなります。
- 近似シーク vs 正確シーク: 多くのライブラリは「キーフレームへの高速シーク(近似)」と「指定フレームへの正確シーク(手前のキーフレームから順に展開)」の両方を持ちます。前者は速いが指定とずれ、後者は正確だが遅い、という根本的なトレードオフがあります。

たとえば GOP = 60(60フレームごとにキーフレーム)で、欲しいのが59番目のフレームなら、直前のキーフレームから59枚ぶん展開してようやく目的に届きます。これが「1フレーム取り出すのに動画全体に近い計算がかかる」最悪ケースの正体です。平均すると、ランダムな1フレーム取得は「GOP の半分ぶん」のフレーム展開を要する、と見積もれます(GOP=60 なら平均30フレーム展開)。

学習で必要なのは、各サンプル点に対応する特定の1フレームだけ(理由は [学習のIO/CPUボトルネック](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0805_dataloader-io-bottleneck))。だからこそ「狙ったフレームへ速く正確にシークして取り出せる」性質が重要になります。

## 仕組みを詳しく

主要な選択肢を、特性ごとに比較します。

PyAV

- 何か: FFmpeg(動画処理の事実上の標準である C 言語ライブラリ群、libavcodec / libavformat)を Python から直接叩くための薄いラッパー(包み)です。
- 特性: FFmpeg の機能をほぼそのまま使えるため柔軟で確実。あらゆるコーデックに対応し、低レベルのシーク制御もできます。ただし API が低レベル(コンテナを開く、ストリームを選ぶ、パケットを読む、デコードする、を自分で書く)で、フレーム単位の正確なシークはキーフレームを意識して実装しないと遅くなりがちです。`seek` でキーフレームに飛んでから目的フレームまで `decode` を回す、という作法を自分で書く必要があります。
- 使いどころ: 確実に動かしたいオフライン前処理スクリプト向き。コーデック互換性が最重要なときの保険。

Decord

- 何か: 深層学習の動画読み込みに特化したライブラリ(DMLC コミュニティ発)。ランダムアクセス(任意フレームへの頭出し)を高速化する設計です。
- 特性: `vr = VideoReader(path); frame = vr[i]` のように「i 番目のフレーム」を添字で直接取れる使い勝手の良さが特徴。内部でフレームのタイムスタンプ索引を持ち、シークを効率化します。バッチ取得(`vr.get_batch([i, j, k])`)もでき、複数フレームを取るときに重複するデコードをまとめられます。動画行動認識(action recognition)の学習で広く使われてきました。
- 注意: 開発の活発度やメンテ状況は時期により変動します。新しいコーデックや環境で詰まることがあり、採用時は最新状況を確認すべきです(この情報は 2026-06 時点)。

TorchCodec

- 何か: PyTorch 公式系の比較的新しい動画/音声デコーダ(2024年頃登場)。デコード結果をそのまま PyTorch テンソルとして受け取れ、CPU デコードに加え GPU デコード(NVDEC 利用)にも対応します。
- 特性: 内部で FFmpeg を使いつつ、フレーム単位・時刻単位の取得 API(`decoder.get_frame_at()`、`decoder[i]` など)を提供。出力が直接テンソルなので、PIL や numpy を経由する変換コストがない。PyTorch エコシステムとの統合が良く、torchvision の旧動画 API の後継として位置づけられています。
- 落とし穴: デコーダオブジェクトが DataLoader の複数ワーカー間でうまくシリアライズ(プロセス間で渡せる形に変換)できないことがあり、`num_workers>0`(並列読み込み)と素直に組み合わせにくい場合があります。回避策として、各ワーカーの中でデコーダを生成し直す(ファイルパスだけを渡す)設計がよく使われます。

「フレームインデックスから1枚取る」イメージを並べると、

- PyAV: コンテナを開き、目的時刻へシークし、キーフレームからデコードを進めて該当フレームを拾う(手続きが多い)
- Decord: `vr = VideoReader(path); frame = vr[frame_idx]`(添字で一発)
- TorchCodec: デコーダを作り、`decoder.get_frame_at(...)` 系でテンソルを直接取得

採用の判断軸は概ね次の5つです。

1. 狙ったフレームへのシーク速度(GOP 構造や索引の持ち方に依存)
2. DataLoader の並列ワーカーとの相性(プロセス間でデコーダを渡せるか)
3. 出力がそのままテンソルか(PIL/numpy を経由しないか)
4. GPU デコード対応の有無(CPU を解放できるか)
5. フレーム対応の正確性(指定 frame_idx と実際に返るフレームが一致するか)

## 手法の系譜と主要論文

これらは論文ではなくソフトウェアなので、一次ドキュメントと背景を系譜として描きます。

- FFmpeg(2000年〜): 動画処理の基盤。ほぼすべての上位ライブラリが内部で FFmpeg(の libavcodec / libavformat)に依存しています。コーデックのデコード、シーク、フィルタを提供する「土台」です。
- H.264/AVC(Wiegand, Sullivan, Bjøntegaard, Luthra, "Overview of the H.264/AVC video coding standard", IEEE TCSVT 2003)と H.265/HEVC(Sullivan, Ohm, Han, Wiegand, "Overview of the High Efficiency Video Coding (HEVC) Standard", IEEE TCSVT 2012): デコードの重さとシークの難しさを規定する圧縮標準そのもの。HEVC は AVC に対しおおむね同画質で半分程度のビットレートを達成する一方、デコード計算量は増えました。コーデックの世代がデコード速度を左右する、という事実の根拠です。
- PyAV(2010年代〜): FFmpeg の Python バインディング。FFmpeg の低レベル機能を Python から型安全に使えるようにしたもの。柔軟性と確実性を重視する前処理で定番化しました。
- Decord(2019年頃〜): 動画系深層学習(行動認識など)の需要から生まれた、ランダムアクセス最適化リーダー。「学習で任意フレームに速くアクセスしたい」という用途を明示的に狙いました。多くの動画認識モデルの公式実装が Decord を採用しています。
- DALI の動画リーダー(NVIDIA, 2018年〜): GPU 上で動画をデコード(NVDEC)し、データ拡張まで GPU で完結させる方向。CPU を律速から外す思想。
- TorchCodec(PyTorch, 2024年〜): PyTorch 公式として、torchvision の古い動画 API を置き換える形で登場。テンソル直出力・GPU デコード・明快な API を掲げ、PyTorch ネイティブな動画 I/O の標準を狙っています。

系譜の流れとしては、「FFmpeg という土台 → Python から使う(PyAV) → 学習用にランダムアクセスを速くする(Decord) → GPU でデコードして CPU を解放する(DALI/NVDEC) → PyTorch にネイティブ統合する(TorchCodec)」と理解できます。一貫した方向性は「学習という用途のために、汎用の動画処理基盤をどんどん特化・高速化していく」流れです。

## 論文の実験結果(定量データ)

これらはライブラリなので、論文の数値ではなくベンチマーク的な事実を整理します。

- シーク速度の一般傾向: Decord は内部にフレームタイムスタンプの索引を持つため、ランダムアクセス(バラバラなフレームを多数取る)で PyAV の素朴な実装より速い、というのが行動認識コミュニティの一般的な観測です。差は GOP 構造に強く依存し、キーフレームが密な動画では差が縮み、疎な動画では Decord 系の索引が効きます。報告される差は典型的に数倍のオーダーですが、後述の通り条件依存です。
- GPU デコードの効果: NVDEC を使う GPU デコード(DALI / TorchCodec)は、CPU が律速のときにデコードを GPU の専用ハードウェアブロックにオフロードできます。NVIDIA のベンチマークでは、高解像度動画の学習で CPU デコードがボトルネックだったケースのスループットが数倍に改善する例が示されています。ただし NVDEC は GPU 上の専用回路なので、同時にデコードできるストリーム数に上限があり(GPU 世代により数本〜十数本程度)、計算用の SM(GPU コア)とは別資源です。デコードと学習計算が同じ GPU を奪い合うわけではない点が利点でもあり、ストリーム数上限が制約でもあります。
- 出力形式のオーバーヘッド: PIL/numpy を経由する経路では、デコード後に色空間変換(YUV→RGB)やテンソル化のコピーが発生します。テンソル直出力(TorchCodec)はこのコピーを省けるため、小さいながら一定の高速化に寄与します。高解像度・高スループットになるほどこのコピーコストが効いてきます。

定量比較で注意すべきは「測定条件で結論がひっくり返る」点です。コーデック(H.264 vs H.265 vs AV1)、解像度、GOP 長、取り出すフレームがランダムか連続か、ワーカー数、ストレージ速度——これらすべてが効くため、「自分のデータ・自分の環境で実測する」のが鉄則です。あるベンチで Decord が速くても、別のコーデックでは TorchCodec が速い、ということが普通に起きます。ベンチマークでは「ウォームアップ後の定常スループット」と「初回のデコーダ初期化コスト」を分けて測ることも重要で、初期化コストは1サンプルしか取らない用途では無視できない大きさになります。

## メリット・トレードオフ・限界

PyAV

- 利点: FFmpeg の全機能・高い確実性。あらゆるコーデックに対応。オフライン前処理に堅実。
- 欠点: 低レベルでコード量が増える。フレーム単位シークの最適化を自分で考える必要。

Decord

- 利点: 添字アクセスで簡単、ランダムアクセスが速い。バッチ取得が便利。
- 欠点: メンテ状況が時期依存。新しいコーデックや環境で詰まることがある。

TorchCodec

- 利点: テンソル直出力、GPU デコード(NVDEC)、PyTorch 統合が良い。
- 欠点: DataLoader の複数ワーカーとのシリアライズ問題。比較的新しく実績が積み上がり途中。

共通のトレードオフと限界:

- どれを使っても「学習ループ内でデコードする限り」CPU/IO が律速になりやすい。最速はそもそもデコードしない事前抽出([raw動画とpre-extractedの二段構成](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0870_raw-vs-extracted-dual-mode))です。デコードライブラリの議論は、しばしば「学習時はデコードしない」という設計判断を後押しする結果に至ります。
- GPU デコードは魅力的ですが、NVDEC の同時ストリーム数上限、対応コーデックの制限、計算用 GPU 資源との取り合い、という制約があります。マルチカメラ(7台同時など)では、ストリーム数上限に早々に当たることがあります。
- 正確性の落とし穴: ライブラリやシーク方式によって「frame_idx で指定したフレーム」が1〜2フレームずれることがあります(キーフレーム境界やタイムスタンプ丸めの差)。学習ラベルと画像の時刻整合がシビアな用途(自動運転の軌道予測など)では、デコーダ間でフレーム対応が一致するかを検証する必要があります。たとえば「同じ frame_idx を PyAV と Decord で取って、ピクセルが一致するか」を抜き取りでチェックする、といった整合性テストが望まれます。
- 可変フレームレート(VFR)の罠: スマホ録画など可変フレームレートの動画では「i 番目のフレーム」と「t 秒のフレーム」の対応が単純な掛け算で計算できず、ライブラリ間でずれやすい。時刻ベースのシーク(`get_frame_at(seconds)`)を使う方が安全な場合があります。

## 発展トピック・研究の最前線

- GPU デコードの学習直結: NVDEC でデコードしたフレームを GPU メモリに置いたまま前処理・学習に渡し、CPU↔GPU 転送を最小化する流れ。動画基盤モデルの大規模学習で重要性が増しています。デコード → リサイズ → 正規化 → モデル投入まで GPU 上で完結させる「ゼロコピー」志向です。
- ハードウェア対応の進化: AV1 など新コーデックの GPU デコード対応が進行中(NVDEC は世代により対応コーデックが異なる)。圧縮率の高い新コーデックは保存容量を減らせる一方、デコード負荷が上がるトレードオフがあります。
- 圧縮ドメイン学習: 動画を完全にデコードせず、圧縮表現(動きベクトルや残差)を直接モデルに入れる研究(CoViAR: Wu et al., "Compressed Video Action Recognition", CVPR 2018 など)。デコードコストそのものを削る発想で、行動認識で精度を保ちつつ大幅な高速化を報告しています。
- 「デコードを避ける」設計との分業: 学習時はあらかじめ抽出した画像を読み([フレーム保存フォーマットの選択](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0858_frame-storage-format-tradeoffs))、動画デコードは前処理段階に限定する、という役割分担が大規模学習では一般的になりつつあります。デコードライブラリは「前処理を速くする道具」として位置づけられます。
