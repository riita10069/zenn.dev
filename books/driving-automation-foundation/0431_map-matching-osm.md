# OSMマップマッチング

## ひとことで言うと
車載 GPS が記録した「ザラついた緯度・経度の点列」を、OpenStreetMap のような道路地図に重ね合わせ、「この車は実際にはこの道を、この順番で走った」というきれいなルートに直す処理です。地図を一緒に保存していないデータセットでも、GPS さえあれば後付けで地図っぽい入力を作るために使えます。マップマッチング (map matching) は、ナビゲーション、交通解析、自動運転の学習データ整備のいずれでも基礎となる技術です。

## 直感的な理解
スマホの地図アプリで歩いていると、自分の現在地を示す青い点が建物の中や川の上に飛んだりするのを見たことがあるはずです。GPS の測位には数メートルから十数メートルの誤差があり、生のままだと「人や車が物理的にあり得ない場所にいる」記録になってしまいます。しかし現実には人や車は道や歩道の上を動いています。「観測された点」と「現実に通れる場所(道路網)」のギャップを埋め、点を最も妥当な道の上に貼り直すのがマップマッチングの仕事です。

自動運転の学習で地図を入力に加えたい理由も直感的です。カメラ映像だけでは「この先がカーブか直進か」「ここが交差点か」を映っている範囲でしか読めません。地図があれば視界の外の道の構造まで先回りで分かり、経路に沿った運転を学びやすくなります。ところが多くの公開運転データセットはカメラ映像と GPS 軌跡しか持たず、地図を同梱していません。「GPS はあるのだから、無料地図に貼り付けて地図画像を後から合成すればよい」というのがこの前処理の発想です。

## 基礎: 前提となる概念
いくつかの用語を順に噛み砕きます。

GPS (Global Positioning System) は、複数の衛星からの電波の到達時間差から受信機の位置を推定する仕組みです。民生用受信機の水平誤差は良い条件で数メートル、ビルに囲まれた市街地(マルチパスといって電波が建物に反射して経路が伸びる現象が起きる)では十数メートル以上になります。出力は緯度 (latitude)・経度 (longitude) の時系列で、1 Hz(毎秒1点)から 10 Hz 程度でサンプリングされるのが一般的です。サンプリング間隔が長い(点が疎な)ほどマッチングは難しくなります。

道路ネットワーク (road network) は、道のつながりを「グラフ (graph)」として表したものです。グラフとは、点 (ノード, node) と、点どうしをつなぐ線 (エッジ, edge) の集合です。道路網では交差点や道の分岐点をノード、その間の道路区間をエッジとします。例えば交差点 A と交差点 B が1本の道でつながっていれば、ノード A とノード B の間にエッジが1本あります。各エッジは「長さ (length)」「制限速度」「一方通行か」といった属性 (attribute) を持ちます。一方通行を表すために、道路網は通常「有向グラフ」(エッジに向きがある)で表します。

OpenStreetMap (OSM) は、世界中の地図データを誰でも編集・利用できる無料のオープンデータプロジェクトです。道路・建物・land use などがタグ付きのジオメトリで保存されています。OSM のデータはオープンデータベースライセンス (ODbL) で提供され、帰属表示を条件に自由に使えるため、研究用途で広く採用されます。

OSMnx は、OSM のデータをダウンロードして NetworkX のグラフとして扱える Python ライブラリです (Geoff Boeing, 2017)。道路網の取得・解析・可視化・最短経路計算を数行で書けます。読みは「オーエスエムエヌエックス」です。

マップマッチングとは、観測した位置列を道路ネットワーク上の最も妥当な位置・経路に対応づける処理の総称です。大きく2つの操作からなります。(1) スナップ (snap): 各観測点を最も近い道路上の点(またはノード)に貼り付ける。(2) ルート復元 (route reconstruction): 貼り付いた点どうしを、道路網の上で実際に通れる連続経路として結ぶ。

## 仕組みを詳しく
簡易な OSMnx ベースのパイプラインを段階的に見ます。

ステップ1: 生の GPS データを用意する。緯度・経度の表を作ります。例として4点だけ示します。

```
latitude : 37.7749, 37.7752, 37.7759, 37.7765
longitude: -122.4194, -122.4185, -122.4173, -122.4162
```

これはサンフランシスコ付近の座標です。実運用では数百から数千点の表になります。

ステップ2: その場所の道路ネットワークをダウンロードする。点列の重心(緯度・経度それぞれの平均)を中心に、半径数百メートルぶんの「車が走れる道」のグラフを OSM から取得します。

```python
center = (df.latitude.mean(), df.longitude.mean())
graph = ox.graph_from_point(center, dist=800, network_type="drive")
```

`network_type="drive"` は「歩道や自転車専用道を除き、車が走れる道だけ」を意味します。返ってくる `graph` はノード(交差点)とエッジ(道路区間)からなる有向グラフです。

ステップ3: スナップ。各 GPS 点について、グラフの中で最も近いノードを探します。`ox.nearest_nodes(graph, X=経度, Y=緯度)` がそれを返します。フラフラした生の点が道路上の確定したノードに吸着し、歩道上に外れた点も最寄りの道路ノードへ引き寄せられます。

```python
matched = [ox.nearest_nodes(graph, X=lon, Y=lat)
           for lat, lon in zip(df.latitude, df.longitude)]
```

ステップ4: ルート復元。スナップ後のノードは飛び飛びで、間の道のつながりが分かりません。そこで連続する2ノード間を、距離最小の経路で結びます。

```python
route = []
for a, b in zip(matched[:-1], matched[1:]):
    path = ox.shortest_path(graph, a, b, weight="length")
    if path:
        route.extend(path[:-1])  # 端点の重複を避けて連結
route.append(matched[-1])
```

`weight="length"` は「道の物理的な長さの合計が最小になる経路を選べ」という指定です。これを全区間でつなげると、交差点から交差点までの連続したルートが復元できます。

ステップ5: 描画して画像化する。道路網を暗い背景に灰色の線で描き、その上に生の GPS 点を赤、復元ルートを青の線で重ねます。PNG としてメモリ上のバッファに書き出し、画像オブジェクトに読み込めば、後段モデルに食わせられる地図画像が1枚できます。最終的な形は、縦横数百ピクセルの RGB 画像、テンソルでいえば `(3, H, W)` のラスタ画像です([ラスタ地図とベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0433_rasterized-vs-vectorized-maps))。

before / after で言うと、before は道路から外れた赤い点がバラバラに散らばっている状態、after は青いルート線が実際の道路に沿ってなめらかにつながっている状態です。

なぜこの単純な貪欲 (greedy) 法でそれなりに動くのかというと、目的が「学習用の地図っぽい画像を生成する」ことであり、ナビのように厳密な道路特定が要らないからです。逆に厳密さが必要な場面では、後述の確率的手法が要ります。具体的には、並走する2本の道(高速道路と側道、本線と分岐車線)が近接していると、最近傍スナップは隣の道へ飛び移りやすく、ルート復元が大回りや逆走を含むことがあります。

## 手法の系譜と主要論文
マップマッチングは GPS の普及とともに長く研究されてきた分野です。代表的な流れを整理します。

最近傍スナップ法(幾何ベース)。各観測点を最も近いエッジまたはノードに貼る最も素朴な方法です。実装が容易で高速ですが、交差点付近や並走道路では隣の道に飛び移りやすく、時系列の整合性を考えないため経路が不自然になりがちです。OSMnx ベースの簡易パイプラインはこの系統にあたります。

Hidden Markov Model (HMM, 隠れマルコフモデル) ベース。Newson and Krumm, "Hidden Markov Map Matching Through Noise and Sparseness" (ACM SIGSPATIAL 2009, Microsoft Research) が代表的で、被引用数が非常に多い基礎論文です。HMM とは「観測の裏に隠れた本当の状態があり、状態は時間とともに遷移する」と仮定する確率モデルです。ここでは各時刻の「本当にいる道路区間」が隠れ状態、GPS 観測点が観測にあたります。

HMM の中身を少しだけ式で見ます。各観測点 z_t に対し近くの候補道路 r_i を複数挙げ、(1) 観測点と道路の近さを表す放出確率 (emission probability) を、誤差が正規分布に従うと仮定して `p(z_t | r_i) ∝ exp(-d(z_t, r_i)^2 / (2σ^2))` で与えます(d は点から道路への垂線距離、σ は GPS 誤差の標準偏差)。(2) ある道路区間 r_i から次の r_j への移り変わりの自然さを表す遷移確率 (transition probability) を、「2点間の直線距離」と「道路網上を実際に走った経路距離」の差が小さいほど高くなるよう定義します。直線で 50m なのに道路上を辿ると 500m もかかる遷移は、不自然なので低い確率になります。これらを掛け合わせて系列全体の尤度を作り、Viterbi アルゴリズム(最も尤もらしい状態列を効率的に求める動的計画法)で全体最適な道路列を選びます。単純な最近傍法と違い、時系列のつながりを考慮するため、間引かれた粗い (sparse) GPS でも正しい道を当てられるのが利点です。トレードオフは候補組み合わせの評価で計算量が増えること、放出・遷移確率のパラメータ(特に σ と距離差のスケール)調整が要ることです。

その後の発展として、Lou et al. "Map-Matching for Low-Sampling-Rate GPS Trajectories" (ST-Matching, ACM SIGSPATIAL 2009, Microsoft Research Asia) は時空間制約(距離だけでなく制限速度から見た移動の妥当性)を加えて低サンプリングレートに強くしました。近年は深層学習を用いた手法(例えば DeepMM, Zhao et al. 2019)が、軌跡を系列変換問題として解く方向で研究されています。実務では HMM ベースが頑健さと実装コストのバランスから今も標準的に使われ、Valhalla や GraphHopper, OSRM といったオープンソースルーティングエンジンにも HMM マッチャが組み込まれています。

可視化・前処理ツールとしては OSMnx (Boeing, "OSMnx: New methods for acquiring, constructing, analyzing, and visualizing complex street networks", Computers, Environment and Urban Systems, 2017) が研究コミュニティで広く使われ、道路網取得から最短経路、描画までを一貫して扱えるため、学習データ用の地図画像生成にしばしば採用されます。

## 論文の実験結果(定量データ)
マップマッチングの精度は通常「ルート一致率 (route mismatch fraction, 誤って選んだ区間長の割合)」や「正しくマッチした点の割合 (accuracy)」で測られます。

Newson and Krumm (2009) の評価では、サンプリング間隔を変えて頑健性を調べています。報告によれば、サンプリング間隔が 30 秒程度までは誤マッチ率が数パーセント以下と低く保たれ、間隔を 60〜120 秒と粗くしても破綻せずに正しい経路を復元できることが示されました。これは、単純な最近傍法だと点が疎になるほど隣の道に飛びやすく急激に悪化するのに対し、遷移確率で経路の連続性を縛る HMM が粗いデータに強いことを意味します。この論文は併せて「グラウンドトゥルース付きの GPS データセット」(約 7〜8 時間ぶんの走行に正解経路を付与したもの)を公開し、後続研究の標準ベンチマークになりました。

指標の意味を補足します。誤マッチ率が 1% というのは「復元した経路の総距離のうち、正解と食い違う区間が 1% だけ」という意味で、ナビや交通解析では十分実用的な水準です。一方、自動運転の学習用に地図画像を作るだけなら、数パーセントの誤マッチがあっても大局的な道の形は保たれるため、簡易な最近傍スナップでも目的を満たせることが多い、という使い分けの判断につながります。Lou et al. の ST-Matching は、サンプリング間隔が 2 分を超えるような極端に疎な軌跡でも、時空間制約のおかげで HMM 系の高い精度を維持できることを報告しており、「疎なほど時間方向の制約が効く」という傾向を裏づけました。

アブレーション的に言えば、HMM から遷移確率を外して放出確率だけにすると最近傍法に退化し、交差点付近で経路が頻繁に隣の道へジャンプして誤マッチ率が跳ね上がります。逆に放出確率を外すと、観測点とまったく無関係な経路を選びかねません。両方の項が揃って初めて頑健になる、という構造です。放出確率の σ を実際の GPS 誤差より大きく取りすぎると候補が広がりすぎて遅く・不正確になり、小さく取りすぎると正しい道が候補から漏れる、というパラメータ感度もあります。

## メリット・トレードオフ・限界
メリット。地図を持たないデータセットでも、GPS さえあれば地図入力を後付けで合成でき、ベンチマークの幅が広がります。OSM は無料で世界中をカバーし、ライセンス上も扱いやすいです。OSMnx を使えば実装が短く、道路網取得から描画まで一貫します。

トレードオフと限界。簡易な最近傍スナップは GPS の大きなずれや並走道路で道を取り違えやすく、頑健さでは HMM 系に劣ります。OSM が実世界と食い違う地域(新設道路、私道、地図未整備地域)では破綻します。レンダリングした画像は色やスタイルがスクリプト依存で、実車のナビ地図とは見た目が異なり、学習時と推論時で地図のドメイン(見た目の分布)がずれる懸念があります。さらに、OSM は車線数・停止線・信号といった車線レベルの精密情報に乏しく、自動運転用の HD 地図のような精度は得られません([ラスタ地図とベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0433_rasterized-vs-vectorized-maps))。GPS 自体が取得できない区間(トンネル、屋内、高層ビル街)ではそもそも入力が欠ける、という根本的な限界もあります。

未解決の課題としては、生成地図のドメインギャップをどう埋めるか(学習時に多様なレンダリングスタイルを混ぜる、あるいはベクトル地図に統一する)、低サンプリング・大誤差の GPS でも頑健に復元する軽量手法をどう設計するか、合成地図とカメラ画像の空間整合([地図BEVと画像BEVの座標整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0430_map-image-bev-alignment))をどう保証するか、などが挙げられます。

## 発展トピック・研究の最前線
近年は、マップマッチングそのものを深層学習で解く流れ(軌跡を系列変換として扱う、グラフニューラルネットで道路網を埋め込む)が進んでいます。また、GPS だけでなく IMU(慣性計測装置)やホイールオドメトリ、カメラ・LiDAR の SLAM 結果を融合して測位精度を上げ、マッチングの前提を底上げするアプローチも一般的です。自動運転の学習データ整備という観点では、合成したラスタ地図に頼らず、HD 地図(ベクトル形式)を直接取り込んでオンラインで地図要素を推定する方向(MapTR 系)が主流になりつつあり、GPS 起点の地図合成は「地図を持たないデータセットへの橋渡し」という補助的な位置づけに収まっていく可能性があります。とはいえ、世界中をカバーする無料地図から学習用の地図信号を安価に作れる利点は大きく、特に多地域・大規模データの整備では今後も実用的な選択肢であり続けます。
