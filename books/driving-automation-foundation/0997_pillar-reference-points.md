# ピラー参照点 (num_points_in_pillar)

## ひとことで言うと
BEV(車を上から見下ろした地図状の平面)の1マスごとに、地面から空に向かって縦に何点かのサンプル点を並べる、という考え方です。「そのマスに物体があるとして、その物体の高さがどれくらいか分からない」ので、複数の高さに点を置いて取りこぼしを防ぎます。この縦に並んだ点の列を「ピラー(pillar=柱)」と呼び、点の数を慣習的に num_points_in_pillar というパラメータで表します。

## 直感的な理解
真上から見た地図(BEV)を作りたいとします。地図の1マスは地面の (前後X, 左右Y) の位置を表します。ところが現実の物体は、地面の上に高さを持って立っています。

例えば「自車の前方20m・左3m」の1マスを考えると、そこには地面の白線(高さ約0m)もあれば、歩行者の頭(約1.7m)も、トラックの荷台の上面(約2.5m)もありえます。地図のマスは平面上の1点ですが、現実にはそのマスの真上に高さ方向のいろいろな物が積み重なっているのです。

ここで、もし「マスの中心、高さ0mの1点だけ」をカメラ画像と照合すると、地面のものしか拾えず、空中にある信号機・看板・トラックの上部を取りこぼします。そこで「1マスの真上に、低い位置から高い位置まで縦に複数の点を立てておき、まとめてカメラ画像から特徴を集めよう」と考えます。これがピラー参照点です。串団子のように、1マスから上に串を刺し、串に刺さった団子(点)それぞれで画像を覗きにいく、というイメージです。

## 基礎: 前提となる概念
**深度不定性・高さ不定性(depth / height ambiguity)**: カメラ画像は3次元の世界を2次元に潰した結果なので、「画像のこのピクセルが世界のどの距離・どの高さの点か」という奥行きと高さの情報が失われています。1枚の写真からは、近くの小さい物体と遠くの大きい物体を原理的に区別できません。これがBEV構築の根本的な難しさです。

**参照点(reference point)**: 「画像のどこを見にいくか」を決める、あらかじめ計算しておく3D点のことです。各BEVセルから何点かの参照点を3D空間に置き、それをカメラに射影して、その投影先の画像特徴をサンプリングします。参照点は幾何的に固定で、学習で動かさない(あるいはアテンションで微調整する)値です。

**buffer(バッファ)**: ニューラルネットの実装で、「学習はしないが状態として保持し、GPU移動やモデル保存に追従する値」を指します。参照点は幾何で決まり学習対象ではないので、重み(学習するパラメータ)ではなくbufferとして登録するのが定石です。

**正規化座標と meshgrid**: 各軸を [0,1] の範囲で表した座標を正規化座標と呼びます。meshgrid は「縦の点列と横の点列と高さの点列から、すべての組み合わせの格子点を作る」操作です。

## 仕組みを詳しく
参照点の生成を段階的に追います。

**ステップ1: グリッドと点数を決める。** 縦 bev_h マス × 横 bev_w マスのBEVグリッドを用意し、1マスあたり num_points_in_pillar 点を立てると決めます。例として bev_h=450, bev_w=300(=135,000マス)、num_points_in_pillar=4 とします。

**ステップ2: 各軸を [0,1] に正規化した格子点を作る。** 各マスの中心を取るのが定石です。

```
xs = linspace(0.5, bev_w − 0.5, bev_w) / bev_w   # 横方向、各マス中心
ys = linspace(0.5, bev_h − 0.5, bev_h) / bev_h   # 縦方向、各マス中心
zs = linspace(0.5, num_points_in_pillar − 0.5, num_points_in_pillar) / num_points_in_pillar
```

linspace(0.5, N−0.5, N)/N は「N個のマスに区切り、各マスの真ん中を取る」表現です。num_points_in_pillar=4 なら zs = [0.125, 0.375, 0.625, 0.875]。これは「ピラーを4等分し、各区間の中央に点を置く」ことを意味します。端(0.0 や 1.0)ではなく中心を取るのは、端は隣のマスとの境界で曖昧になりやすいからです。

**ステップ3: 3次元格子を作って並べ替える。** meshgrid で全組み合わせを作り、stack して reshape します。tensor の形を追うと理解が深まります。

```
meshgrid(ys, xs, zs)  → 各々 [450, 300, 4]
stack([gx, gy, gz])   → [450, 300, 4, 3]   (縦450 × 横300 × 高さ4点 × 座標3次元)
reshape               → [135000, 4, 3]
```

[135000, 4, 3] の読み方は「135,000個のマス、それぞれに4個の点、各点は (x,y,z) の3座標」です。この時点では値はすべて [0,1] の正規化値です。これを buffer として保存します(学習で動かさない固定の幾何)。

**ステップ4: 使うときにメートル座標へ拡大する。** 実際にカメラへ射影する段階で、知覚範囲 pc_range(例: (−60,−60,−5, 120,60,3) メートル。[BEVグリッドとpc_range設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0927_bev-grid-and-pc-range))を使って正規化値を実世界座標(自車中心ego座標、典型的にX=前方・Y=左・Z=上)に引き伸ばします。高さの例で見ると、Z範囲が [−5, 3](8m幅)のとき、

```
z_norm = 0.125 → −5 + 0.125 × 8 = −4.0 m
z_norm = 0.375 → −5 + 0.375 × 8 = −2.0 m
z_norm = 0.625 → −5 + 0.625 × 8 =  0.0 m
z_norm = 0.875 → −5 + 0.875 × 8 = +2.0 m
```

このように、1マスの真上に高さ −4.0m から +2.0m まで4点が縦に散らばります(具体値はpc_rangeと点数の組み合わせ次第)。

**before/after で見る効果。**

```
before(高さ1点 Z=0 だけ):
  前方20mのマス → 地面しか見ない → トラック荷台・信号機を取りこぼす ✗

after(ピラー4点 −4m〜+2m):
  同じマスで複数の高さをカバー
     ・+2.0m → トラック荷台の高さに当たる ✓
     ・ 0.0m → 路面・白線に当たる ✓
  どれかの点が物体表面に当たれば、その点をカメラに射影して特徴を拾える
```

各参照点はカメラの射影行列([カメラ射影 (内部・外部パラメータ)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0942_camera-projection-intrinsic-extrinsic))で画像平面に落とされ、grid_sample([grid_sampleによる特徴サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0969_grid-sample-feature-sampling))でその位置の画像特徴を取り出します。同じマスの4点から拾った特徴を集約することで、高さの異なる物体を取りこぼしにくくなります。

## 手法の系譜と主要論文
- **Lang et al. (CVPR 2019)「PointPillars」(arXiv:1812.05784)**。「ピラー」という用語の出どころ。LiDAR点群を、3Dボクセルではなく地面から立つ柱(ピラー)に縦方向をまとめて2D BEVに落とし、高速化しました。高さ方向をまとめる発想がここで定着しました。

- **Philion & Fidler (ECCV 2020)「Lift-Splat-Shoot (LSS)」(arXiv:2008.05711)**。対照的アプローチ。各ピクセルに離散的な深度の確率分布を予測し(「このピクセルは深度5mに30%、10mに50%…」)、特徴を奥行き方向に持ち上げてBEVに貼ります。動機は高さ・奥行きの不定性を確率で陽に扱うこと。利点は物理的な解釈しやすさ。弱点は深度予測が外れるとBEV全体がずれること、深度ビン数だけ計算が増えること。

- **Li, Wang, Lu et al. (ECCV 2022)「BEVFormer」(arXiv:2203.17270)**。ピラー参照点を明示的に導入した本トピックの直接の源流。各BEVクエリ(マス)に対し高さ方向に複数(論文では4点)の3D参照点を立て、それらを各カメラに射影して特徴を集める「Spatial Cross-Attention(SCA)」を提案しました。動機は、LSSのように深度分布を予測する代わりに「高さ方向に点をばらまいて当たったものを使う」方が、深度予測の誤差に頑健だと考えたこと。深度を予測しないので深度誤差に振り回されない、というのが新規性の核です。

- **Liu et al. (ICRA 2023)「BEVFusion」(arXiv:2205.13542)**。カメラとLiDARを共通のBEV表現で融合する枠組みで、BEVグリッドの高さ方向サンプリングの考え方を引き継いでいます。マルチモーダル時代でもピラー的な高さ表現が基盤として残っていることを示します。

系譜としては、PointPillars(LiDARで高さをまとめる)→ LSS(カメラで深度を予測)→ BEVFormer(カメラで高さに点をばらまく)という流れで、「カメラの高さ不定性をどう扱うか」が一貫したテーマです。BEVFormerのピラー方式は「深度を予測せず点をばらまく」点でLSSの弱点を避ける設計です。

## 論文の実験結果(定量データ)
BEVFormer(ECCV 2022)はnuScenes(6カメラの大規模自動運転データセット)の3D物体検出で評価され、NDS(nuScenes Detection Score、0〜1で大きいほど良い)で当時のマルチカメラ手法の上位、報告でおよそ0.51前後を達成しました。これは深度を陽に予測しないにもかかわらず、深度予測ベースの手法と同等以上、という点で重要です。

高さ方向の参照点数(num_points_in_pillar に相当)のアブレーションでは、点を1点から増やすと高さのある物体や立体物の検出mAP(mean Average Precision、0〜1で大きいほど良い)が改善し、4点程度で性能とコストの折り合いが良くなる、という飽和傾向が報告されています。点を増やすほど射影とサンプリングの回数が線形に増えるので、それ以上はメモリと計算だけが増えて性能はほぼ頭打ちになります。「4点」という設定は、この飽和点を踏まえた実用的な選択です。

指標の意味を補足します。mAPは検出の適合率(predictした物体がどれだけ正解と一致するか)を様々な許容誤差で平均した値で、立体物クラスでmAPが上がるということは、まさに「高さのある物体を正しい位置に検出できるようになった」ことを意味します。1点だけ(=地面しか見ない設定)で立体物mAPが低いのは、ピラー参照点の必要性を数値で裏づける結果です。

LSS(ECCV 2020)のアブレーションでは、深度分布を「単一深度の決め打ち」に退化させるとBEVセグメンテーションのIoU(重なり率、0〜1で大きいほど良い)が大きく下がることが示されました。これは「高さ・深度を1点で決め打ちするのは危険で、複数候補を持つことが効く」という、ピラー方式とも共通する教訓を別角度から示しています。

## メリット・トレードオフ・限界
**メリット**: (1) 高さ不定性に頑健で、明示的な深度予測が不要なため深度誤差の影響を受けにくい。(2) 参照点は固定の幾何計算だけで作れ、学習を要しない(あるいはアテンションで微調整するだけ)。(3) 立体的な物体(高い車両、信号機、トラック上部)を取りこぼしにくい。

**トレードオフ・限界**: (1) num_points_in_pillar を増やすと、射影・grid_sample・重み計算がすべて点数に比例して増える(メモリと速度の直接コスト)。(2) 高さ方向の解像度は点数で決まる粗いもので、4点では高さ8mを4分割する程度にすぎない。(3) 物体の正確な高さは依然分からない(複数候補を試すだけで、推定値そのものは得られない)。(4) どの高さに点が当たったか自体は出力されないため、高さ推定タスクには別の仕組みが要る。

**研究上の未解決課題**: 点を等間隔にばらまくのは無駄が多い(多くの点は空中の何もない場所に当たる)ため、画像から推定した深度・高さに応じて点を適応的に配置する、あるいは深度予測(LSS系)と点ばらまき(BEVFormer系)を融合して両者の長所を取る方向が研究されています。等間隔配置の無駄を定量化すると、市街地シーンでは画像の上半分の多くが空・建物上部で占められ、地表付近の物体が密集する高さ帯(おおむね 0〜2m)に点が少ししか割り当てられない、という偏りが生じます。高さ方向の点配置を学習で偏らせる(地表付近に点を集中させる)だけでmAPが改善する、という報告もあります。

## 発展トピック・研究の最前線
最近のマルチカメラ3D検出では、ピラー参照点をアテンションで動的にずらす(deformable attention で参照点位置を学習補正する)ことで、固定点の粗さを補う手法が主流です。また、深度予測ヘッドを併設して点の重みを深度確率で重み付けする、深度とピラーのハイブリッドも登場しています。さらに、点をばらまく代わりに連続的な高さ表現(暗黙関数やボリュームレンダリング的なサンプリング)で高さ方向を扱う研究も進んでおり、「離散点 vs 連続表現」が次の論点になっています。
