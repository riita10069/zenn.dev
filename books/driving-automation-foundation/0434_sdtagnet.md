# SDTagNet

## ひとことで言うと
無料で世界中をほぼ網羅する粗い地図（SD地図、Standard-Definition map）には、道路の形だけでなく「これは高速道路」「3車線」「一方通行」といった文字のタグが大量に付いています。SDTagNet は、その形（幾何）に加えてタグ（意味）まで丸ごとエンコードし、自動運転の地図理解（オンラインHD地図構築や地図推定）を助ける手法です。タグは文字列なので、言語モデルでベクトルに変換して使います。本質は「これまで捨てられていた地図のテキスト情報を、機械が読める数値に変えて活用する」点にあります。

## 直感的な理解
人間が初めての街を運転するとき、道路の形だけでなく標識や地図の注記を読みます。「ここは一方通行」「制限速度40」「高速の入口」といった言葉の情報が、形だけでは分からない運転判断を助けます。ところが従来のオンライン地図手法の多くは、SD地図を使うときその「形」しか取り込みませんでした。OSM（OpenStreetMap、世界中の有志が作る無料地図）には `highway=motorway`（高速道路）、`lanes=3`（3車線）、`maxspeed=80`、`oneway=yes`（一方通行）のようなタグが豊富にあるのに、これらを無視していたのです。

SDTagNet の問題意識は単純です。「無料で手に入る意味情報を捨てるのはもったいない。形だけでなくタグも一緒にエンコードすれば、地図理解はもっと良くなるはず」。タグは自然言語に近い文字列なので、これを言語モデル（テキストをベクトルに変換するNLPモデル）に通して数値ベクトル（埋め込み）にし、地図要素の特徴の一部として使います。形が同じでも意味が違う道路（たとえば狭い住宅路と狭い一方通行）を、タグの埋め込みが区別できるようにする、というのが狙いの核です。

## 基礎: 前提となる概念
言葉を順に噛み砕きます。

- HD地図（High-Definition map、高精細地図）: 車線1本ごとの形状、停止線、横断歩道などを数センチ精度で持つ地図。作るのも維持するのも非常に高コストで、対応エリアが限られます。
- SD地図（Standard-Definition map、標準解像度地図）: カーナビや OSM レベルの粗い地図。「ここに道路がある」程度の情報。無料・全世界をほぼ網羅します。
- prior（事前情報）: モデルに与える「あらかじめ分かっている手がかり」。SD地図を prior に使うとは、「だいたいの道路網は分かっている前提で、足りない細部はセンサで補う」という考え方です。
- ベクトル地図（vector map）: 道路を「点の列でできた線（polyline、折れ線）」として持つ表現。画像（ラスタ）ではありません（[ラスタ地図 vs ベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0426_rasterized-vs-vectorized-map)）。
- way（ウェイ）: OSM での「道路や境界線などの線要素」を指す用語。点の並び（ノード列）とタグの集合からできています。
- トークン（token）: Transformer に入れる系列の1要素。ここでは「地図上の1点」や「1つの地図要素」を1トークンとして扱います。
- 埋め込み（embedding）: 単語や記号を、意味の近いものが近くに来るような数値ベクトルに変換したもの。言語モデルは「motorway」と「highway」のような近い語を近いベクトルにできます。
- クロスアテンション（cross-attention）: 片方を質問役（query）・もう片方を参照役（key/value）にする注意機構。「BEV の各セルが、地図トークンのうちどれを参照すべきか」を学習的に重み付けします。

全世界を HD で整備するのは現実的でないため、「SD地図を prior に使い、足りない細部は車載センサーで推定する」流れが近年の主流です。多くの手法はその際 SD地図の幾何だけを使いますが、SDTagNet はそこに意味（タグ）を足した点が新しいのです。

## 仕組みを詳しく
SDTagNet の中心アイデアは「SD地図のベクトル要素（way）を、幾何とタグの両方でトークン化してエンコードし、知覚モデルの prior として注入する」ことです。一般的な構成を初心者向けに段階分解します。

ステップ1: OSM データの取得と自車中心の切り出し
自車の軌跡を囲む範囲の OSM データ（道路の点列とタグ）を取得します。取得結果はキャッシュし、「ある範囲を包含する既存キャッシュがあれば再利用する」ことで取得回数を抑えるのが実用上の定石です。取得した地図を自車中心のローカル座標（典型的には X=前方、Y=左、Z=上）に変換してパッチを切り出します（[ego中心ラスタタイル](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0414_ego-centric-raster-tile) と同じ自車中心の思想）。

ステップ2: トークン化
各 way を一定間隔で点に再サンプリング（fixed-point resampling、等間隔で点を打ち直す）し、点ごとのトークンにします。各トークンに次の情報を持たせます。
- 位置情報: 連続値の座標を正弦波で符号化する位置エンコーディング（sinusoidal positional encoding）。座標の小さな差も滑らかに表現できます。
- タグ情報: way に付いたタグ文字列（`highway=primary`, `lanes=2` など）を言語モデルに通して埋め込みベクトルにする。これが SDTagNet の核心です。
- 所属情報: 「このトークンはどの地図要素に属するか」を表す識別子。1本の線に属する点をまとめて扱うために使います。

結果は、サンプルごとに長さの違う（ragged、不揃い長の）トークン集合になります。

ステップ3: エンコード
トークン列を Transformer 系のエンコーダに通し、地図特徴 `tokens [B, N, C]` と、どこがパディングかを示す `key_padding_mask [B, N]` を得ます。ここで B はバッチ数、N はトークン数（サンプルごとに違うので最大長にそろえてパディングし、マスクでパディング部を無視）、C は特徴次元です。言語モデルは事前学習済みの重みを読み込み、推論専用（重みは固定）で使うことが多く、軽量化のため埋め込みを事前計算してキャッシュする設計もよく採られます（同じタグ文字列の埋め込みは毎回同じなので、計算結果を辞書に保存して使い回せる）。

ステップ4: 知覚側への融合
カメラ由来の BEV 特徴の各セルが、地図トークンを key/value としてクロスアテンションで参照します。メモリ効率の良いアテンション実装（QK 行列を一度に作らず分割して計算する）を使えば、450×300 のような大きな BEV グリッドでも省メモリで融合できます。融合の安定化には次の安全設計が有効です。
- padding mask を尊重し、存在しないトークンには注目しない。
- 学習可能な null トークン（そのサンプルに地図が無くても出力が NaN にならない置き換え用トークン）。
- ゼロ初期化のゲート（融合経路の出力を学習開始時はゼロに初期化し、最初は地図なしと完全に同じ挙動から始める）。

これは残差つきゲート付き融合（[マップBEVクロスアテンション融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0308_map-bev-cross-attention-fusion)）と同じ「足しても最初は壊さない」思想で、既存モデルに地図枝を後付けする際の定番テクニックです。

なぜタグが効くか、具体例で言うと次のとおりです。カメラの BEV だけでは「前方に細い道がある」までしか分からなくても、タグ `oneway=yes` が一緒に入れば「ここは一方通行だから逆走しない軌跡を選ぶべき」と学べる余地が生まれます。`maxspeed=30` が分かれば速度の妥当範囲が、`highway=residential` なら歩行者警戒の優先度が変わります。形だけの地図では決して得られない手がかりです。

tensor の流れを整理すると次のようになります。

```
OSM way（点列＋タグ）
  → 点に再サンプリング → 各点トークン
      位置:  座標 → 正弦波エンコード → [N, C_pos]
      タグ:  "highway=primary;lanes=2" → 言語モデル → [N, C_tag]
  → 結合・射影 → tokens [B, N, C], mask [B, N]
  → Transformer エンコード
  → BEV特徴 [B, C, H, W] が tokens に cross-attend
  → ゲート（初期ゼロ）で残差加算 → 融合BEV [B, C, H, W]
```

なぜ「言語モデルでタグを埋め込む」のがカテゴリ番号（one-hot）より優れるのか。OSM のタグは事実上無限に近い種類があり（地域ごとに独自タグもある）、これを固定の番号表で扱うと未知タグに対応できず、似た意味のタグ（`primary` と `trunk`）も無関係な番号になります。言語モデルの埋め込みなら、文字列の意味的な近さがベクトルの近さに反映され、表で定義していない新しいタグにもそれなりのベクトルが割り当たる（汎化する）利点があります。

## 手法の系譜と主要論文
SDTagNet は「SD地図 prior」と「ベクトル地図エンコード」という二つの流れが合流した位置にあります。

- 基盤となるベクトル地図エンコード: Gao, J. et al. "VectorNet" (CVPR 2020, arXiv:2005.04259) が、地図要素を polyline トークンの集合として Transformer/グラフで扱う発想を確立しました。Liao ら "MapTR" (ICLR 2023, arXiv:2208.14437) はオンラインHD地図構築を「地図要素＝点集合」のクエリ予測として定式化しました（[MapTRv2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0452_maptrv2)）。way を点列トークンに分解する SDTagNet の表現は、この系譜の延長です。

- SD地図 prior の先行研究: Luo ら "SMERF: Augmenting Lane Perception and Topology Understanding with Standard Definition Navigation Maps" (2023, arXiv:2311.04079) は SD ナビ地図を Transformer でエンコードし、車線知覚・トポロジー理解の prior として注入しました。Jiang ら "P-MapNet: Far-seeing Map Generator Enhanced by both SDMap and HDMap Priors" (2023, arXiv:2403.10521) は SD/HD 双方の prior を地図構築に注入し、特に遠方（far-seeing）の地図推定を改善しました。これらはいずれも主に「幾何」prior を使います。SDTagNet は同じ流れの中で「タグ＝意味」を加えた点に新規性があります。

- タグを言語モデルで埋め込む発想: 文や語をベクトル化する事前学習済み言語モデル（BERT 系、文埋め込みモデルなど）の転用です。利点は、語彙の意味を連続空間で扱えるため `motorway` と `trunk` のような近い概念を近いベクトルにできること。トレードオフは未知タグ・言語差・表記ゆれへの弱さと、埋め込みモデルの重み・計算コストです。視覚言語モデル（CLIP 系）の言語側を使ってタグを埋め込み、地図の意味と画像特徴を同じ空間に近づける派生も考えられます。

- SDTagNet（Adelberger ら, 2024/2025, arXiv）。SD地図のテキストタグを言語モデルの埋め込みで取り込み、点レベルトークンと連続位置エンコーディングで地図要素を表現する SD-map prior エンコーダを提案しました。狙いは「幾何のみの SD-prior に意味情報を足してオンライン地図推定・地図構築を改善する」ことです（本ノートの記述は公開要旨に基づきます。詳細数値は arXiv 公開版を確認のこと）。

## 論文の実験結果（定量データ）
オンライン地図構築の品質は、おもに mAP（mean Average Precision、平均適合率の平均）で測ります。地図要素（車線境界・路面境界・横断歩道など）ごとに、予測した polyline と正解 polyline の近さを Chamfer 距離（二つの点列の互いに最も近い点同士の距離の平均）でしきい値判定し、しきい値（例: 0.5m / 1.0m / 1.5m）ごとの AP を平均します。値が大きいほど良い指標です。nuScenes（自動運転の代表的データセット、6カメラ＋LiDAR、ボストンとシンガポールで収集）の地図構築ベンチマークが標準の土俵で、Argoverse 2 も使われます。

SD地図 prior を加える系統の研究では、prior なしのベースラインに対して mAP が数ポイント上がることが繰り返し報告されています。たとえば SMERF や P-MapNet 系は、SD地図 prior の導入で地図構築 mAP がベースライン比でおよそ数ポイント〜十数ポイント改善し、特にカメラから遠い領域や遮蔽（他車に隠れて見えない）領域で効果が大きいと報告しています。P-MapNet は遠方（たとえば 90m 四方の広い範囲）の地図推定で顕著な改善を示しました。遠方はセンサ情報が乏しいぶん、prior の幾何が効くためです。

SDTagNet が主張する追加効果は、「幾何 prior にタグ（意味）を足すと、さらに mAP が上がる」というものです。報告によれば、幾何のみの SD-prior に比べてタグを加えることで地図要素の認識・推定が向上するとされます。アブレーション（要素を抜く実験）の観点でいえば、「タグを外して幾何だけにすると改善幅が縮む」「言語モデルの埋め込みを単純なカテゴリ one-hot に置き換えると、未知タグや細かい区別への対応が落ちる」といった関係が想定されます。タグの効果が大きい要素は、形だけでは区別しにくいもの（車線種別、通行規制に関わる境界など）に集中する傾向があります。

指標の意味を補足します。Chamfer 距離が 0.5m というのは「予測した線と正解の線が平均50センチ以内にそろっている」という厳しめの基準で、ここでの AP が上がることは「細部まで正確に地図化できた」ことを意味します。地図構築の数ポイントの mAP 改善は、車線変更や交差点通過のような細部依存の判断に直結するため、見た目以上に重要です。なお、効果を厳密に評価する際は「タグの意味が効いたのか、それとも単に prior の幾何情報が増えたから良くなったのか」を切り分ける必要があり、タグを与えるが中身をシャッフルする・無意味な埋め込みに置換するといった対照実験が説得力を持ちます。

## メリット・トレードオフ・限界
メリット側:
- 無料・全世界網羅の SD地図（OSM）を prior に使え、HD地図が無いエリアでも地図文脈を得られる。
- 幾何だけでなくタグ（車線数・一方通行・道路種別・速度規制など意味情報）を活用でき、形だけの地図より手がかりが豊富。
- 言語モデルの埋め込みを使うため、近い概念のタグを近いベクトルとして滑らかに扱え、ある程度の未知タグにも汎化しうる。
- ゼロ初期化ゲートや null トークン等の安全設計と組み合わせれば、既存の知覚モデルを壊さずに地図枝を後付けできる。

トレードオフ・限界:
- OSM のタグは欠損・表記ゆれ・地域差が大きく、品質に精度が左右されます。タグが付いていない道路では意味情報の恩恵がありません。
- 言語モデルを抱えるぶん、メモリ・計算・依存関係が増えます。埋め込みを事前計算してキャッシュするなどの工夫が要ります。
- ベクトル＋可変長トークンの処理（パディング・マスク・キャッシュ）は実装が複雑で、バグの温床になりやすい。
- SD地図と現実のずれ（工事・新設道路・登録ミス）が prior の誤りとして混入します。prior を信じすぎると、地図が間違っている場所で性能が落ちる失敗例があります。prior の信頼度を学習で調整する仕組みが望ましい。
- 位置合わせ（registration）の誤差も問題です。SD地図と自車の現在地のずれ（GPS 誤差や地図の歪み）が大きいと、prior の幾何が実際の路面とずれて、かえって混乱を招きます。
- 研究上の未解決課題として、タグの語彙が世界中・多言語で一貫しないため、言語モデルの埋め込みが地域をまたいでどこまで汎化するかは確立していません。また「意味の効果」と「単に幾何 prior が増えた効果」を厳密に切り分けるアブレーション設計も難しい論点です。

## 発展トピック・研究の最前線
SD地図 prior の研究は、(1) 幾何のみ、(2) 幾何＋意味（タグ）、(3) 地図と知覚の不確かさを明示的に扱う、という方向へ発展しています。SDTagNet は (2) の代表ですが、最前線では「prior の信頼度をシーンごとに推定し、地図が古い・間違っている場所では prior を弱める」適応的融合が研究されています。これはゲートの開き具合をシーン依存にする、prior と観測の不一致を検出して重みを下げる、といった形で実装されます。

もう一つの流れは、タグを使うエンコードを「地図構築」だけでなく「予測・計画」まで広げることです。`oneway` や `maxspeed` のような規制情報は、他車の振る舞い予測や自車の経路計画に直接効くため、知覚の prior にとどめず方策（policy）の入力に通す設計が考えられます。SD地図のトポロジー（どの道とどの道がつながっているか）を graph として与え、車線レベルの接続推定に使う SMERF 系の発展とも合流します。

言語モデルの活用という観点では、近年の大規模言語モデル・視覚言語モデルを地図タグや交通規則の解釈に使い、「テキストで書かれたルールを運転判断に橋渡しする」研究が広がりつつあります。SDTagNet の「タグを言語埋め込みにする」発想は、その入口に位置づけられます。将来的には、地図の自然言語注記や交通標識のテキストを同じ言語空間で扱い、知覚・予測・計画を横断して意味情報を流す統合表現が一つの目標になっています。
