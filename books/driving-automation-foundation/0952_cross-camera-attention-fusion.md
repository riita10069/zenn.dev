# CrossAttentionViewFusion（カメラ間アテンション融合）

## ひとことで言うと

複数のカメラで撮った映像を1つにまとめるとき、「各空間位置ごとに、すべてのカメラの特徴がお互いを見合って、いまこの場所にとってどのカメラが重要かを学習で動的に重み付けする」仕組みです。Transformer の中核であるアテンション（注意機構）をカメラ同士の間に適用したもので、単純にカメラ特徴を積み重ねて固定の重みで混ぜる方式（チャネル連結）よりも賢く、走行状況に応じてカメラの寄与を毎回変えられます。

## 直感的な理解

自動運転車は前方・側方・後方など複数のカメラを積んでいます。これらをどう統合するかが問題です。素朴な方法は「全カメラの特徴を縦に積み重ねて、固定の重みで混ぜる」ことですが、これには2つの弱点があります。1つは、混ぜる重みが学習後に固定されてしまうこと。もう1つは、カメラの並べる順番に結果が依存してしまうことです。

しかし現実の運転では、重要なカメラは状況で変わります。交差点では左右のカメラが死活的に重要ですが、高速道路の巡航では前方カメラが支配的です。さらに、隣り合うカメラの視野が重なる領域では、複数カメラの情報を突き合わせた方が正確に物体を捉えられます。

人間も同じことをしています。直進中はほとんど前を見ていますが、合流のときは無意識にサイドミラーと首振りで側方へ注意を移します。「いまどこを重点的に見るか」を状況で切り替えているわけです。カメラ間アテンション融合は、この「重点の動的な切り替え」をニューラルネットワークで実現します。固定の重みではなく、入力された画像内容に応じて毎回重み付けが変わるのが本質です。

## 基礎: 前提となる概念

専門用語を順に噛み砕きます。

- アテンション（attention、注意機構）= 「複数の入力のうち、いまの目的にとってどれが重要かを数値で重み付けし、重要なものを多く取り込む」仕組みです。Transformer の中核技術です。
- query / key / value（クエリ・キー・バリュー、Q/K/V）= アテンションの3要素です。query は「探したい問い」、key は「各候補の見出し」、value は「各候補の中身」です。query と各 key の一致度（内積）を計算し、その一致度で value を重み付き合計します。式で書くと `Attention(Q,K,V) = softmax(QKᵀ/√d) V` です。ここで `d` は key の次元で、`√d` で割るのは内積が次元に比例して大きくなりすぎて softmax が飽和するのを防ぐためです（スケーリング）。softmax は複数の数値を合計1の確率に変換する関数で、これが「重み」になります。
- self-attention（自己アテンション）= query・key・value をすべて同じ入力から作るアテンションです。「集合の中の各要素が、他の全要素（と自分）を見て関係を測る」ことになります。
- Multihead Attention（マルチヘッドアテンション）= アテンションを複数の「ヘッド（観点）」で並行に行い、観点ごとに違う関係を捉えます。1つのヘッドは色の一致を、別のヘッドは形の一致を見る、というように分担できます。各ヘッドの出力を連結して元の次元に戻します。
- 順序不変（permutation invariant）= 入力の順番を入れ替えても出力が変わらない性質です。アテンションはこの性質を持ちます。これは利点でもあり（カメラ順に縛られない）、欠点でもあります（どれが何番カメラか区別できない）。後者は位置埋め込みで補います（[学習可能なカメラ位置埋め込み](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0975_learnable-view-position-embedding) 参照）。
- 残差接続（residual connection）= サブブロックの出力を入力にそのまま足し戻す配線です。`x = x + F(x)` の形をとり、F が何も学習していない初期状態でも入力が素通りでき、勾配が深い層まで届きやすくなります。
- FFN（Feed-Forward Network）= 各トークンを独立に2層の全結合層で変換する部品です。アテンションがトークン間の関係を、FFN がトークン内の非線形変換を担当する、という役割分担です。

## 仕組みを詳しく

カメラ間アテンション融合の典型的な処理を、具体的な tensor shape で追います。バックボーン（特徴抽出器）が各カメラ画像から特徴マップを出した状態を入力とします。バッチサイズ B=8、カメラ台数 V=7、チャネル数 C=256、特徴マップの空間サイズ H=W=8 という例を使います。

入力は「V台ぶんの特徴マップ」で、形は `[B*V, C, H, W] = [56, 256, 8, 8]` のように、バッチとカメラを1つの軸にまとめている場合が多いです。

ステップ1: view 軸を復元して並べ替える。
```
[B*V, C, H, W] = [56, 256, 8, 8]
→ reshape  [B, V, C, H, W] = [8, 7, 256, 8, 8]
→ permute  [B, H, W, V, C] = [8, 8, 8, 7, 256]
→ reshape  [B*H*W, V, C]   = [512, 7, 256]
```
意味は重要です。最終的な `[512, 7, 256]` は「512個の空間位置（=B*H*W）それぞれについて、7台のカメラ特徴（各256次元）を7個のトークンとして並べた」ものです。アテンションはこの「各位置での7トークンの列」の内部で独立に行われます。つまり「同じ空間位置に対応する7台のカメラ特徴同士」を比較しているわけです。

ステップ2: カメラ位置埋め込みを足す。
```
x = x + view_embed   # view_embed: [1, V, C] = [1, 7, 256]
```
これがないとアテンションは7トークンを区別できません（順序不変なので「どれが何番カメラか」が消えます）。詳細は [学習可能なカメラ位置埋め込み](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0975_learnable-view-position-embedding) を参照してください。

ステップ3: pre-norm マルチヘッド self-attention。
```
x_norm = LayerNorm(x)
attn_out = MultiheadAttention(x_norm, x_norm, x_norm)   # Q=K=V=x_norm
x = x + attn_out                                        # 残差
```
query=key=value をすべて同じ `x_norm` にしているので、これは「7台のカメラトークンが互いを見合う self-attention」です。各カメラトークンが他の6台＋自分を見て、一致度に応じた重み付き合計を作ります。num_heads=8 なら8つの観点から並行に関係を捉えます。これにより「この位置では3番カメラと5番カメラの情報が整合しているから強める」といった動的重み付けが起きます。LayerNorm をアテンションの手前に置く構成（pre-norm）は学習を安定させます（[Pre-Norm Transformer ブロック構成](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1003_pre-norm-transformer-block) 参照）。

ステップ4: FFN と残差。
```
x = x + FFN(LayerNorm(x))   # FFN: Linear(256→512) → GELU → Linear(512→256)
```
各トークンを独立に非線形変換し、表現力を高めます。

ステップ5: view 方向に平均プール。
```
x.mean(dim=1)   # [512, 7, 256] → [512, 256]
```
7台のトークンを1つに集約します。重要なのは、アテンションで「カメラ同士が情報交換した後」の特徴を平均するので、生の特徴を単純平均するよりも情報がリッチに統合されている点です。

ステップ6: 空間に戻す。
```
[512, 256] → reshape [B, H, W, C] → permute [B, C, H, W] = [8, 256, 8, 8]
```
出力はチャネル連結方式と同じ `[B, C, H, W]` 形式になり、下流のモジュールから見ると差し替え可能なインターフェースになります。

チャネル連結との決定的な違いをまとめると、連結は「7台を縦に積んで 1×1 畳み込みで固定重み混合」、アテンション融合は「7台が互いを見て内容に応じて毎回重みを計算」です。後者は（位置埋め込みを抜けば）順序不変で、状況適応的です。

## 手法の系譜と主要論文

カメラ間・視点間にアテンションを使う考え方は、Transformer ベースの3D知覚の発展とともに主流になりました。

- Vaswani et al., "Attention Is All You Need"（NeurIPS 2017、arXiv:1706.03762）。Multihead Attention・残差・FFN・LayerNorm という本手法が使う全部品の原典です。提案動機は、再帰や畳み込みを使わずに系列内の任意の2要素間の関係を直接モデル化すること。効果は長距離依存の獲得と並列計算のしやすさ。トレードオフはトークン数の2乗に比例する計算量です。

- Carion et al., "DETR: End-to-End Object Detection with Transformers"（ECCV 2020、arXiv:2005.12872）。物体検出をクエリベースのアテンションに置き換え、後段の NMS（非最大抑制）を不要にしました。「学習可能なクエリが特徴を相互参照する」設計は後続の3D知覚の出発点になりました。

- Liu et al., "PETR: Position Embedding Transformation for Multi-View 3D Object Detection"（ECCV 2022、arXiv:2203.05625）。複数カメラの特徴に3D座標由来の位置埋め込みを付与し、物体クエリがアテンションで多視点特徴を直接参照できるようにしました。動機は、明示的な幾何投影なしに多視点を統合すること。カメラ間アテンション融合で使われる学習可能な view 埋め込みと「クエリが多視点を参照する」発想は PETR 系の簡略版とみなせます。PETR の位置埋め込みは3Dカメラ幾何から計算するのに対し、カメラ識別だけを目的とする実装では単純な学習可能ベクトルにとどまります。

- Li et al., "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers"（ECCV 2022、arXiv:2203.17270）。BEV（鳥瞰図）クエリが spatial cross-attention で多視点画像特徴を参照し、temporal self-attention で過去フレームを参照します。「クエリが複数視点をアテンションで集める」点で同系統ですが、BEVFormer はカメラの投影行列を使って各 BEV セルが参照すべき画像位置を絞り込む点でより幾何的です。

- Hu et al., "UniAD: Planning-oriented Autonomous Driving"（CVPR 2023 Best Paper、arXiv:2212.10156）。検出・追跡・地図・予測・計画をクエリベースのアテンションで一気通貫に繋ぐ統合フレームワークです。各タスクを別々に最適化する従来の弊害を避けることが動機で、クエリが特徴を相互参照する設計がカメラ間アテンションと思想的に通じます。

系譜としては「視点間の関係を固定結合（連結＋畳み込み）で扱う段階 → アテンションで動的に扱う段階 → さらにカメラ幾何（投影）を組み込んで参照先を絞る BEV 段階」へと進化してきました。カメラ間 self-attention は、幾何を使わずにアテンションだけで統合する中間的な位置にあります。

## 論文の実験結果（定量データ）

アテンション系の多視点統合がどれだけ効くかは、3D物体検出のベンチマーク nuScenes でよく示されています。指標の意味から説明します。

- nuScenes は約140万枚の画像と39万のアノテーション付きフレームを持つ大規模な自動運転データセットです。
- mAP（mean Average Precision、平均適合率）は「検出の正確さ」を測る指標で、0〜1（または%）で高いほど良いです。
- NDS（nuScenes Detection Score）は mAP に加えて位置・大きさ・向き・速度・属性の誤差を統合した nuScenes 独自の総合指標で、こちらも高いほど良いです。

報告されている代表値（カメラのみ、検証セット）の目安です。

- PETR は ResNet-101 バックボーンで mAP およそ 0.357、NDS およそ 0.421。これは、当時の素朴な多視点統合（カメラごとに独立に検出して後で統合する方式や単純連結）を NDS で数ポイント上回る水準でした。
- BEVFormer は同程度のバックボーンで NDS およそ 0.517、mAP およそ 0.416 と報告され、temporal self-attention（過去フレーム参照）を加えることで速度推定の誤差が大きく下がり NDS が伸びました。
- アブレーション（ある要素を抜く実験）の知見として、BEVFormer 系では「temporal attention を外すと NDS が数ポイント低下し、特に物体速度の推定精度（mAVE）が大幅に悪化する」ことが報告されています。これは「複数視点や時間方向の相互参照が、単一視点では取れない情報（とくに動き）を補っている」ことの定量的な証拠です。
- spatial cross-attention（多視点参照）を単純な平均プーリングに置き換えると mAP が明確に落ちることも示されており、「カメラ間で内容に応じて重み付けする」効果が単純平均を上回ることを裏付けます。

これらは物体検出での結果ですが、本質は「視点間の関係を固定結合ではなくアテンションで動的に捉えると、単一視点では取れない情報が得られる」という点で、運転特徴の融合にも同じ論理が当てはまります。

## メリット・トレードオフ・限界

メリット
- 状況適応的。各空間位置で重要なカメラを内容に応じて動的に重み付けできる（固定重みと対照的）。
- 順序不変（位置埋め込みを除けば）。カメラの並び順に本質的に依存しない。
- 視野の重なる領域で、複数カメラの情報を突き合わせて整合を取れる。
- Transformer の確立された部品（残差・LayerNorm・FFN）で学習が安定しやすい。

トレードオフ・限界
- 計算とメモリが重い。`B*H*W` 個の位置でそれぞれ V トークンのアテンションを走らせるため、特徴マップの格子サイズが大きいとコストが急増します。アテンションはトークン数の2乗に比例するため、ここでは位置数 × V² に効いてきます。
- 幾何を明示的に使わない。学習可能な view 埋め込みは抽象的なカメラ識別子にすぎず、実際のカメラ姿勢（外部パラメータ）を投影に使う BEV 方式ほど3D整合は取れません。
- 学習が難しめ。アテンションは連結より表現力が高いぶん、データが少ないと過学習・不安定になりやすいです。
- 最後に view 方向を平均プールするため、「特定カメラだけを強く残す」表現はアテンション段で作れても最終集約で薄まりうる、という構造的な情報損失があります。
- 未解決の課題として、カメラ台数や配置が異なる車種・データセット間でどう汎化させるか、計算量を抑えつつ表現力を保つ効率的アテンション（線形アテンション等）の適用、幾何情報をどこまで明示的に注入すべきかのバランスが、研究上のオープンな論点として残っています。

## 発展トピック・研究の最前線

カメラ間アテンションを起点に、研究は2方向へ進んでいます。1つは幾何を強く組み込む方向で、BEV 空間でのクロスアテンション（投影行列で参照先を絞る）や、3D点群と画像を融合するマルチモーダルアテンションがこれにあたります。もう1つは計算効率の方向で、deformable attention（参照点を少数に絞る変形可能アテンション、Deformable DETR 由来）や線形計算量のアテンションが、高解像度・多カメラでも回る統合手法として研究されています。さらに、エンドツーエンド運転（知覚から計画まで一気通貫）の文脈では、カメラ特徴の融合と軌道計画を同じアテンション機構で繋ぐ流れ（UniAD やその後継）が最前線です。
