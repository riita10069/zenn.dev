# マルチスケール特徴のプーリング連結

## ひとことで言うと

画像認識ネットワーク（backbone）は、処理が進むにつれて「細かいが範囲の狭い特徴」から「粗いが範囲の広い特徴」まで、解像度の異なる複数段階（典型的には4段階）の中間出力を作ります。それらをすべて同じ小さな格子サイズに揃えてから、チャネル方向（特徴の種類の方向）に積み重ねて1つの分厚い特徴にまとめる処理です。細かい情報と大局的な情報の両方を1つの表現に同居させ、後段が両方を同時に使えるようにするのが狙いです。

## 直感的な理解

地図を読む状況を考えてください。市街地図（細かいが範囲が狭い）と、県全体の広域図（粗いが範囲が広い）は、どちらか一方だけでは目的地にたどり着けません。細い路地は市街図にしか載っておらず、全体の方角は広域図でしか分かりません。だから両方を見比べます。

画像認識でもまったく同じことが起きます。浅い層は高解像度で「車線の細い線」「遠くの小さな歩行者」のような細部をよく見ていますが、それが何なのかという意味づけは弱い。深い層は低解像度で「ここに車がある」という意味を強く持ちますが、細かい位置は失われています。自動運転では「遠くの小さな物体（細かさが必要）」と「全体の状況（大局が必要）」の両方が必要なので、複数の段階をまとめて使いたい。しかし段階ごとに解像度もチャネル数もバラバラなので、そのままでは1つに束ねられません。そこで「全部を同じ格子サイズへ揃えてからチャネル方向に連結する」という処理が要るのです。

## 基礎: 前提となる概念

用語を噛み砕きます。

- 特徴マップ（feature map）。画像を縮小して情報を凝縮した格子状の数値で、形は `[チャネル数 C, 高さ H, 幅 W]` です。

- チャネル（channel）。特徴の種類の本数です。たとえば「縦エッジに反応するチャネル」「赤い物体に反応するチャネル」が別々に存在します。深い層ほどチャネル数が増え、より抽象的な特徴を表します。

- スケール（scale）。解像度の段階のことです。浅い層は高解像度（H, W が大きい＝細かい）、深い層は低解像度（H, W が小さい＝粗い）。

- プーリング（pooling）。格子を区画に区切り、各区画を1つの値に縮約して解像度を下げる操作です。最大値を取れば max pooling、平均を取れば average pooling です。

階層型の backbone（Swin Transformer や ResNet など）は、画像を処理するうちに複数のステージを通り、各ステージで解像度を半分にしながらチャネル数を倍にしていきます。Swin Transformer の Tiny 構成を例にとると、各ステージの出力はおよそ次のようになります（入力 256×256 のとき）。

| ステージ | チャネル C | 解像度の目安 | 性質 |
|---|---|---|---|
| stage1 | 96  | 64×64 | 高解像度・細かい・局所的（縁、テクスチャ） |
| stage2 | 192 | 32×32 | 中解像度 |
| stage3 | 384 | 16×16 | 中〜低解像度 |
| stage4 | 768 | 8×8   | 低解像度・粗い・大局的（物体の意味） |

ここで問題が起きます。最後のステージ（768ch, 8×8）だけを使うと大局的な情報は得られますが、細かい位置情報が失われます。逆に最初のステージ（96ch, 64×64）だけだと細かいですが「これは車だ」という意味的な判断材料が足りません。だから全段階を使いたい。しかし解像度もチャネル数もバラバラなので、同じ格子へ揃える前処理が要ります。この発想の源流が FPN（Feature Pyramid Network、特徴ピラミッドネットワーク）です。

## 仕組みを詳しく

典型的な実装は次の3ステップで構成されます。

```python
self.pool = nn.AdaptiveMaxPool2d(image_feature_size)   # 揃える格子サイズ
self.channel_proj = nn.Sequential(
    nn.Conv2d(backbone_channels, embed_dim, kernel_size=1),  # 例: 1440 → 256
    nn.GELU(),
)

def forward(features):                  # features は backbone の4ステージ出力
    for i in range(len(features)):
        features[i] = self.pool(features[i])    # 各スケールを同じ格子へ
    fused = torch.cat(features, dim=1)          # チャネル方向に連結
    fused = self.channel_proj(fused)            # チャネル圧縮
    return fused
```

ステップごとに追います（説明のため処理する枚数を N=56 とします。これは前段で B×V を畳んだ枚数です。[backbone処理のための B*V reshape](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0918_backbone-batch-view-reshape) 参照）。

1. AdaptiveMaxPool2d（適応的マックスプーリング）。これは「出力の格子サイズを指定すると、入力の解像度が何であってもその格子に収まるよう自動で区画を区切り、各区画の最大値を取る」操作です。これで4つのバラバラな解像度を全部 `S×S` に揃えます。最大値を取る（average ではない）のは「その区画で最も強く反応した特徴を残す」ためで、小さく強い反応が平均化で薄まりにくい利点があります。

   - stage1: `[56, 96, 64, 64]` → `[56, 96, S, S]`
   - stage2: `[56, 192, 32, 32]` → `[56, 192, S, S]`
   - stage3: `[56, 384, 16, 16]` → `[56, 384, S, S]`
   - stage4: `[56, 768, 8, 8]` → `[56, 768, S, S]`

   S は揃える格子サイズで、設計によって 7 や 8 などが選ばれます。

2. チャネル方向の連結（`torch.cat(..., dim=1)`）。dim=1 はチャネル軸です。4つを積み重ねると `96 + 192 + 384 + 768 = 1440` チャネルになります。

   - 結果: `[56, 1440, S, S]`

   この 1440 が4ステージの合計チャネル数です。実装では各ステージのチャネル数を `sum(feature_channels)` として自動算出するのが一般的で、こうしておくと backbone を差し替えても合計が自動で合います。

3. 1×1 Conv によるチャネル圧縮（`channel_proj`）。1440チャネルは後段で扱うには分厚すぎるため、1×1 の畳み込み（各位置のチャネル方向だけを混ぜる線形変換）で `embed_dim`（例: 256）へ圧縮します。続く GELU（なめらかな活性化関数。負の入力を完全には0にせず少し残す ReLU の親戚で、Transformer 系で標準的に使われます）で非線形性を加えます。

   - 結果: `[56, 256, S, S]`

この `[56, 256, S, S]` が、次のカメラ間融合に渡される「各画像ごとの統合済み特徴」です。

なぜ「全部を同じ格子へ揃えてから連結」なのか。`torch.cat` でチャネル方向に積むには、連結する全テンソルの H と W が一致していなければなりません（チャネル軸以外の形が揃っていないと連結できないのが規則だからです）。解像度が違うままでは連結できないので、プーリングで先に揃える必要があります。

## 手法の系譜と主要論文

マルチスケール特徴を1つに統合する考え方は、物体検出とセグメンテーションの分野で確立されてきました。系譜をたどります。

- Lin et al., "Feature Pyramid Networks for Object Detection"（CVPR 2017, arXiv:1612.03144）。出発点です。深い層（意味は強いが粗い）を上から下へ最近傍補間で拡大し、対応する浅い層へ足し込み、各解像度に「意味も位置も持つ」特徴ピラミッドを作りました。提案理由は「小さい物体は高解像度の層でしか見えないが、その層は意味が弱い」という問題の解決です。

- Liu et al., "Path Aggregation Network (PANet)"（CVPR 2018, arXiv:1803.01534）。FPN のトップダウン経路に加えて、浅い層から深い層へ向かうボトムアップ経路を足しました。浅い層の正確な位置情報を上位へも伝えることで、マルチスケール情報の双方向の流れが精度に効くことを示しました。

- Wang et al., "HRNet (Deep High-Resolution Representation Learning)"（TPAMI 2020 / CVPR 2019, arXiv:1908.07919）。高解像度の表現を最後まで保ち続け、複数解像度の枝を並列に走らせて相互に融合し続けました。「高解像度を途中で捨てない」ことの重要性を示し、本トピックの「浅い高解像度ステージも使う」動機と直結します。

- Liu et al., "Swin Transformer"（ICCV 2021, arXiv:2103.14030）。Transformer に階層構造（ステージごとに解像度を半減・チャネルを倍化）を導入し、CNN のような4段階のマルチスケール特徴を Transformer でも自然に得られるようにしました。本トピックが対象とする「4ステージの中間出力」はこの階層型 backbone が前提です。

プーリングで同じ格子に潰してチャネル連結する方式は、これら精緻な FPN 系に比べると最小限の簡略版です。解像度別の出力を残さず、強制的に1つの格子へ落としてチャネル連結するだけで、FPN 的な動機（マルチスケール統合）を軽量に実現します。実装が単純でメモリ・計算が軽く、後段の BEV 変換やアテンションに計算予算を回せる利点があります。

## 論文の実験結果(定量データ)

マルチスケール統合の価値は、FPN の実験で最も明快に示されています。Lin et al.（2017）は COCO 物体検出ベンチマーク（80カテゴリの大規模検出データセット）で、Faster R-CNN に FPN を組み込むと、最深層の特徴のみを使うベースラインに対して AP（Average Precision、検出精度の総合指標で高いほど良い）が大幅に向上することを示しました。報告では、特に小物体の AP（APs）が顕著に改善しています。アブレーション（要素を抜く実験）では、トップダウン経路を除くと精度が大きく落ち、横方向接続（浅い層との合流）を除いても落ちることが示され、「複数スケールを実際につなぐこと」が効いていると確認されました。

PANet はインスタンスセグメンテーションの COCO で、FPN だけのベースラインに対してボトムアップ経路の追加でさらに数ポイントの AP 改善を報告し、双方向のマルチスケール流れの有効性を示しました。HRNet は姿勢推定（COCO keypoint）で、高解像度を保ち続ける設計が、いったん落として戻す設計より高い AP を出すことを示しました。これらの数値の意味は「細かい段階を捨てずに使うほど、小さい対象や正確な位置が要る課題で精度が上がる」ということで、自動運転で遠方の小物体を見落とさないことに直結します。

プーリング連結方式そのものについては、average と max の選択がしばしばアブレーションされます。一般に max は小さく強い信号を残しやすく、average は領域全体の傾向を滑らかに捉えます。どちらが良いかは課題依存で、小物体重視なら max が有利という報告が多い一方、ノイズの多い入力では average の方が安定することもあります。

## メリット・トレードオフ・限界

メリット。細かい情報（浅いステージ）と大局的な情報（深いステージ）の両方を1つの特徴に同居させられます。実装が単純で、プーリングで揃えて連結するだけです。チャネル数を自動検出すれば backbone 非依存にでき、Swin から ResNet や ConvNeXt へ差し替えても動きます。AdaptiveMaxPool は最大値を取るため、小さく強い反応が平均化で薄まりにくい利点もあります。

トレードオフと限界。第一に、プーリングで空間解像度を大きく落とすため、細かい位置情報の一部は失われます。FPN のように解像度ごとに残すわけではありません。第二に、max プーリングは「その区画で一番強い1点」だけを残すので、弱いが重要な信号を取りこぼす可能性があります。第三に、4ステージを全部メモリに載せてから連結するため、瞬間的なメモリ使用が増えます。第四に、1440→256 のような圧縮で情報が削られます。embed_dim を小さくしすぎると表現力が落ち、大きすぎると後段が重くなるトレードオフがあります。

研究上の未解決課題としては、プーリングという情報破壊的な縮約に代えて、各スケールへ学習可能な重みを付けて加重統合する方式（BiFPN の重み付き融合など）や、スケール間でアテンションを張る方式が提案されており、軽さと表現力のバランスをどう取るかは今も探求が続いています。

## 発展トピック・研究の最前線

Tan et al. の "EfficientDet"（CVPR 2020, arXiv:1911.09070）が提案した BiFPN は、各スケールの寄与を学習可能なスカラ重みで正規化して融合し、しかも双方向経路を繰り返し積むことで、固定の連結より高い精度・効率を達成しました。これはプーリング連結の「全スケールを等価に扱う」前提への直接的な改良です。また Transformer 系では、各スケールをトークン列として deformable attention で相互参照する Deformable DETR 系のマルチスケール統合が主流化しています。BEV 系の自動運転モデルでは、マルチスケール画像特徴をそのまま BEV クエリのキー・バリューとして使い、空間統合と視点変換を一体化する流れが強まっており、「いつ・どの段で・どの方法でスケールを混ぜるか」という設計自由度が研究の前線になっています。
