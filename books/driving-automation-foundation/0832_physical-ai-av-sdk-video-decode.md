# 動画 (.mp4) からの効率的なフレームデコードとセンサー時刻同期

## ひとことで言うと
カメラの映像は `.mp4` のような圧縮された動画ファイルで保存されています。モデルに渡すには、その中から「ある瞬間の 1 枚の絵 (フレーム)」を取り出して数値の画像テンソルに直す必要があります。このノートは、(1) 圧縮動画から狙ったフレームだけを効率よく取り出す「シークデコード」と、(2) レートの違う複数センサー (カメラと自車運動) を時刻で揃える「時刻同期」という、自動運転データ前処理の 2 つの基礎技術を扱います。

## 直感的な理解
動画から 1 枚の絵を取り出すのは、一見すると「ファイルの 1000 番目の画像を読むだけ」に思えます。ところが動画は 1 枚 1 枚を独立した画像として保存していないので、そう単純にはいきません。多くのフレームは「直前のフレームとの差分」としてしか記録されていないからです。1000 番目だけ欲しくても、その手前のキーとなるフレームから順に復元しないと正しい絵になりません。

もう 1 つの直感は時刻合わせです。車には複数のセンサーが載っていますが、それぞれ独立した時計とレートで動いています。カメラは秒 30 枚、自車運動センサーは秒 100 回、というように。「カメラのこの絵」と「自車運動のこの行」は厳密には同じ瞬間ではありません。学習データを作るには、別々のリズムを刻むセンサーから「だいたい同じ瞬間」のペアを取り出して揃える必要があります。これが時刻同期です。

## 基礎: 前提となる概念

- フレーム (frame): 動画を構成する 1 枚 1 枚の静止画です。30fps (frames per second) の動画なら 1 秒あたり 30 枚あります。
- デコード (decode): 圧縮された動画データを、扱える生のピクセル配列 (RGB の数値の格子) に展開する処理です。逆の圧縮処理をエンコードと呼びます。
- 動画コーデック (codec) と GOP 構造: H.264 や HEVC (H.265) などの圧縮方式は、フレームを 3 種類に分けます。I フレーム (Intra, 単独で復元できる完結した絵。キーフレームとも呼ぶ)、P フレーム (Predicted, 前のフレームとの差分)、B フレーム (Bi-directional, 前後両方を参照する差分)。I フレームから次の I フレームまでのまとまりを GOP (Group of Pictures) と呼びます。P/B フレームは単独では復元できず、参照元をたどる必要があります。
- シーク (seek): 動画の途中の任意の位置に飛ぶことです。本を最初から 1 ページずつめくらず、目次から目的のページへ直接飛ぶイメージです。動画では「目的フレームの直前の I フレームまで飛び、そこから目的フレームまで順にデコードする」のが正しいシークです。
- SDK (Software Development Kit): あるデータやサービスを扱うために提供される、関数やクラスをまとめた開発キットです。データセット提供者が独自フォーマットの正しい読み方を SDK として配布することがあります。

問題の核心はこうです。「動画の 1000 番目のフレームだけ欲しい」とき、素朴に作ると「先頭から 1000 枚デコードしてやっと目的の 1 枚」となり非常に遅くなります。正しくは「1000 番目の直前の I フレームに飛んで、そこから数枚だけデコードする」べきです。シークデコードを提供するリーダー (たとえばデータセット付属の `SeekVideoReader` のようなクラス) は、この「目的フレームへ効率よく飛んでデコードする」処理を肩代わりしてくれます。

データセットが独自のディレクトリ構造とファイル命名規則を持つ場合、専用 SDK を使うのが公式の読み方です。たとえば `pip install git+https://github.com/<provider>/<sdk>.git` のように配布され、特定の Python バージョン以上を要求することがあります。

## 仕組みを詳しく

カメラフレーム読み込みは、複数台のカメラそれぞれについて次の手順を踏みます。

### ステップ1: どのフレームを取るか決める (時刻あわせ)

モデルが「今この瞬間」とみなすのは自車運動 (egomotion) のある行の時刻です。その時刻 `egomotion_timestamp_us` (マイクロ秒) に最も近いカメラフレームを探します。

```python
def egomotion_ts_to_frame_idx(egomotion_timestamp_us, camera_timestamps_us):
    return int(np.argmin(np.abs(camera_timestamps_us - egomotion_timestamp_us)))
```

`np.argmin(np.abs(...))` は「差の絶対値が最小になる位置」を返します。つまり基準時刻に一番近いカメラフレームの番号 (0 始まり) を求めています。

なぜこの処理が要るのか。自車運動 (約 100Hz) とカメラ (約 30Hz) は別々のセンサーで、記録レートも開始タイミングも完全には一致しません。両者が同じクリップ基準時刻 (t=0) のマイクロ秒で記録されていれば、時刻が一番近いフレームを「対応するフレーム」とみなして整列できます。これを時刻ベースの最近傍アライメント (nearest-neighbor alignment) といいます。

数値例で見ます。カメラフレームの時刻が `[0, 33000, 66000, 100000, ...]` マイクロ秒 (= 0ms, 33ms, 66ms, 100ms, 約 30fps) で、基準時刻が `70000` (= 70ms) のとき、差の絶対値は `[70000, 37000, 4000, 30000, ...]`。最小は 3 番目 (66ms) なので `frame_idx = 2` が選ばれます。実際の時刻ずれは 4ms。30fps なら最近傍でも最大で約 ±16.7ms (フレーム間隔の半分) ずれます。

### ステップ2: そのフレームをデコードする

```python
video_data = io.BytesIO(video_path.read_bytes())
reader = SeekVideoReader(video_data=video_data)
try:
    indices = np.array([frame_idx], dtype=np.int64)
    rgb_frames = reader.decode_images_from_frame_indices(indices)
finally:
    reader.close()
```

`decode_images_from_frame_indices` に欲しいフレーム番号の配列を渡すと、そのフレームだけを RGB のピクセル配列としてデコードして返してくれます。内部では「指定フレームの直前の I フレームへシークし、そこから順にデコードして目的フレームに到達する」処理が走ります。`try/finally` で必ず `reader.close()` を呼ぶのは、デコーダが内部で確保するリソース (メモリやファイルハンドル、場合によっては GPU デコーダのコンテキスト) を確実に解放するためです。これを怠ると長時間の学習でリソースリークを起こします。

### ステップ3: 画像にして前処理する

```python
pil_frame = Image.fromarray(rgb_frames[0])
camera_tensors.append(transform(pil_frame))  # (3, H, W)
```

NumPy のピクセル配列を PIL (画像処理ライブラリ) の Image に変換し、`transform` (リサイズ・正規化など) を通して `(3, 256, 256)` のテンソルにします。この `transform` は backbone の設定から導出するのが安全です ([モデル設定から前処理を自動導出する (timm の resolve_model_data_config)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0840_timm-resolve-model-data-config))。

### ステップ4: 複数枚にまとめる

各カメラぶんのテンソルを集めたあと、必要なら追加面 (地図タイルなど) を足し、最後に積み重ねて `(N, 3, 256, 256)` を作ります。

```python
return torch.stack(camera_tensors, dim=0)  # (N, 3, H, W)
```

### 既知のボトルネックと回避策

素朴な実装には次の落とし穴があります。

```python
video_data = io.BytesIO(video_path.read_bytes())  # ファイル全体をメモリに読み込む
```

`read_bytes()` は `.mp4` ファイルを丸ごとメモリに読み込みます。1 フレームしか要らないのに毎回ファイル全体を読むのは無駄が大きく、`__getitem__` が何百万回も呼ばれる学習では深刻な速度低下になります。回避策は、学習前に「必要なフレームだけを抽出してテンソルや圧縮画像として保存しておく前処理工程」を設けることです。これにより学習中のデコードコストを一度きりの前処理に前倒しでき、スループットが数倍変わることもあります。

## 手法の系譜と主要論文

動画デコードと時刻同期は研究テーマというより工学的な基盤技術ですが、関連する系譜があります。

- 動画圧縮標準 (H.264/AVC: 2003, HEVC/H.265: 2013): 差分符号化と GOP 構造を標準化し、「キーフレームからの順次デコード」という制約を生みました。シークデコードはこの制約への対処です。
- GPU 動画デコード (NVIDIA NVDEC, DALI ライブラリ): NVIDIA は GPU 上で動画をデコードする専用ハードウェア (NVDEC) とデータローディングライブラリ (DALI) を提供しています。狙いは「動画デコードを CPU から GPU にオフロードして学習のデータ供給を速くする」ことです。効果はスループット向上ですが、トレードオフとして「GPU メモリと処理を学習以外に使う」「対応 GPU・ドライバへの環境依存が増える」点があります。データセット付属の効率的シークデコーダもこの系譜にあります。
- マルチセンサー時刻同期 (KITTI: Geiger et al., CVPR 2012 / nuScenes: Caesar et al., CVPR 2020, arXiv:1903.11027): これらのデータセット論文は、LiDAR・カメラ・IMU など異なるレートのセンサーをどう時刻同期するかを論じました。nuScenes は LiDAR の回転位置に合わせてカメラのシャッターを切る「キーフレーム同期」を採用し、センサー間ずれを抑えました。最近傍タイムスタンプによる整列は最も単純な同期方式で、利点は実装が簡単なこと、トレードオフは「真の同時刻ではなく最大でフレーム間隔の半分ずれる」ため高速走行時に位置ずれが生じうることです。より厳密には時刻補間や動き補償 (ego-motion compensation) を行う研究もあります。

## 論文の実験結果 (定量データ)

- 時刻同期誤差の影響: 一般に 30fps カメラでの最近傍同期は最大約 16.7ms のずれを生みます。時速 72km/h (= 20m/s) で走行中なら、16.7ms は約 0.33m の移動に相当します。3D 物体検出や軌道予測では、この程度のずれが検出位置や速度推定の誤差として下流に伝播します。nuScenes 系の研究では、センサー間の精密同期が検出の mAP (mean Average Precision, 高いほど良い) を数ポイント左右することが報告されています。指標 mAP は「検出のあたり具合を再現率の各水準で平均したもの」で、0〜1 のうち高いほど検出が正確という意味です。
- デコード I/O の影響: 動画デコードがデータローディングのボトルネックになると、GPU が遊んでデータ待ちになります (GPU 使用率が下がる)。NVIDIA DALI による GPU デコードは、CPU デコードに比べてデータローディングのスループットを数倍に上げると報告されています。事前フレーム抽出によるキャッシュ化も同様に、学習 1 エポックの所要時間を数分の一にできる場合があります。これらは「指標の改善」ではなく「同じ精度に到達するまでの時間 (wall-clock)」の改善です。

アブレーション的な観点では、最近傍同期を時刻補間 + 動き補償に置き換えると、高速・急旋回シーンでの軌道予測誤差 (ADE/FDE, メートル、小さいほど良い) がわずかに改善する一方、実装と計算コストが増えます。多くの実装はまず最近傍同期から始め、必要に応じて高度化します。

## メリット・トレードオフ・限界

メリット
- 公式 SDK やシークデコーダを使うことで、データセットの独自フォーマットを正しく確実に読める。
- シークデコードにより、欲しい 1 フレームだけを狙って取り出せる (先頭から全デコードを避けられる)。
- 時刻ベースの最近傍アライメントは実装が単純で、レートの違うセンサー同士を素直に整列できる。

トレードオフ・限界
- 素朴な実装はファイル全体を毎回メモリに読み込むため学習時のボトルネックになる。事前フレーム抽出工程が望ましい。
- 最近傍アライメントは厳密な同時刻保証がなく、高速走行時に最大でフレーム間隔の半分の位置ずれが残る。動き補償しない限りこのずれは消えない。
- 専用 SDK への依存と特定 Python バージョンの要求が環境構築の制約になる。GPU デコードを使うとドライバ・GPU 世代への依存がさらに増える。
- 未解決の課題として、同期ずれを下流タスク (検出・予測) の精度に与える影響として定量化し、どこまで厳密同期に投資すべきかの一般的な指針は確立していません。

## 発展トピック・研究の最前線
- 動き補償付き時刻同期: センサー間の時刻ずれを、自車運動を使って幾何的に補正する手法。高速・急旋回シーンでの精度を稼ぎます。
- フレーム事前抽出とシャード化: 動画を学習用フレーム列に展開し WebDataset 形式の `.tar` シャードにパックして I/O を最適化する流れ ([大規模自動運転データセットの PyTorch Dataset 化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0828_nvidia-physical-ai-dataset-parser))。
- ニューラル動画コーデックと学習可能デコード: 圧縮表現を直接モデルに食わせる (デコードを省く) 研究も登場しています。
