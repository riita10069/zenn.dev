# UniAD プランニングヘッド

## ひとことで言うと
自動運転は普通「周りを認識する → 物体が次にどう動くか予測する → 自車の進路を計画する」という複数の処理に分かれています。UniAD(ユニエーディー、Unified Autonomous Driving)は、これらの処理をそれぞれのモジュールとして残しつつ、全体を一つのニューラルネットとして End-to-End(一気通貫)で学習します。しかも各モジュールを クエリ(Transformer が情報を取りに行くための質問ベクトル)で繋ぐことで情報を劣化させずに後段へ流し、「最終的に良い計画(プランニング)ができること」をゴールに据えて、認識や予測の途中処理までそのゴールに合わせて最適化します。このノートはその最後の「計画を出す部分(プランニングヘッド)」に焦点を当てます。

## 直感的な理解
従来の自動運転は、認識・予測・計画を独立したモジュールに分け、それぞれ別々に開発・学習していました(モジュール型パイプライン)。整理しやすく開発分担もしやすいのですが、三つの深刻な問題がありました。

情報の劣化(information loss)。各モジュールは前段の出力を「確定した答え」として受け取ります。たとえば認識が「ここに歩行者がいる」と決め打ちすると、その確信度や曖昧さは捨てられ、計画側は「本当にいるのか怪しい」という情報を使えません。段を経るごとに情報がやせ細ります。

誤差の蓄積(error accumulation)。認識が間違えると、その間違いがそのまま予測・計画へ伝わり、増幅されます。途中で訂正する仕組みがありません。

目的のズレ(objective misalignment)。認識モジュールは「物体検出の精度」を最適化しますが、それが必ずしも「安全な計画」に直結しません。本来のゴールは「安全に運転すること」なのに、各モジュールは自分のローカルな指標を磨いているだけ、という状態になります。たとえば遠くの小さな物体の検出精度を上げても、その物体が自車の進路に関係なければ計画は良くなりません。逆に進路上の物体を見落とすのは致命的です。専用指標はこの「計画にとっての重要度」を区別しません。

UniAD の問題意識は「最終目的は計画(運転)なのだから、全モジュールを計画というゴールに向けて一緒に最適化すべきだ」という点です。そこで認識から計画までを一つのネットワークにし、各段をクエリで繋ぐことで、確信度や曖昧さを含んだ豊かな情報を劣化させずに後段へ流す設計にしました。論文のタイトル "Planning-oriented"(計画指向)はこの主張そのものです。

## 基礎: 前提となる概念
BEV 特徴(Bird's Eye View 特徴): 複数カメラの画像から作る「上空から見下ろした周囲の地図」状の特徴です。UniAD は BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)という手法で、各カメラ画像の特徴を空間的に持ち上げて鳥瞰格子(例 200×200 のグリッド、各マスが特徴ベクトルを持つ)に集約し、全タスクが共有する土台にします。地面に貼った格子の各マスが、その場所の情報を表すベクトルを持つイメージです。BEVFormer は時間方向の複数フレームも融合するため、速度や動きの手がかりも BEV に含まれます。

クエリ(query): Transformer デコーダが情報を取りに行く質問ベクトルです。N 個のクエリを用意し、それぞれが入力特徴にアテンション(注目)をかけると、N 個の出力(N 台の車、N 個の地図要素、N 本の未来軌道など)を並列に得られます。重要なのは、クエリは単なる座標値ではなく「豊かな特徴ベクトル」だという点です。これにより確信度や属性が後段へそのまま伝わります。

微分可能(differentiable)と誤差逆伝播: ネットワーク全体が微分可能なら、最後の損失(計画の誤差)を入力側まで逆向きに伝えて(誤差逆伝播)、すべての重みを同時に更新できます。これが「計画の誤差で認識・予測まで調整する」ことを技術的に可能にしています。モジュール型パイプラインでは段の境界が微分不可能(検出ボックスのしきい値処理など)なので、これができませんでした。

開ループ評価と閉ループ評価: 開ループ(open-loop)は、実際に車を走らせず、記録済みの走行データに対して「モデルが出す軌道が正解軌道とどれだけ近いか」を測る評価です。手軽ですが、モデルの出力が次の状況に影響しないので、誤差が蓄積する実走行の難しさを測れません。閉ループ(closed-loop)は、シミュレータ等でモデルの出力を実際に実行し、その結果が次の入力になる本物の運転評価です。UniAD は nuScenes の開ループで評価され、この点が後に大きな論争を呼びます。

## 仕組みを詳しく
UniAD は BEV 特徴を土台に、五つのタスクをクエリで連結します。流れは「知覚 → 予測 → 計画」で、各段の出力(クエリ)が次段の入力になります。

```
複数カメラ → BEVFormer → BEV 特徴(共有の地図)
                          │
   知覚 ┌─ TrackFormer : 物体検出+追跡 → エージェントクエリ(各車/歩行者を表す)
        └─ MapFormer   : 車線/横断歩道 → 地図クエリ
                          │
   予測 ┌─ MotionFormer: 各エージェントの複数未来軌道(確率的)
        └─ OccFormer   : 将来の占有(どのマスがいつ塞がるか)
                          │
   計画 └─ Planner head: ego query が上記すべてに注目 → 自車 waypoint
                          │  + OccFormer 占有で衝突回避の後最適化
                          ▼
                   自車軌道 [(x1,y1),...,(x6,y6)]
```

知覚。TrackFormer が物体を検出しフレームをまたいで同一物体を追跡し、各物体を表すエージェントクエリを出します。MapFormer が車線・横断歩道などの地図要素をクエリで表します。

予測。MotionFormer がエージェントクエリと地図クエリを使い、他車・歩行者が今後どう動くかを複数候補の未来軌道として確率的に予測します(1つの物体に複数の進路候補を出すのは、右折か直進か分からない不確実性を表すためです)。OccFormer が周囲の占有(将来どのマスが塞がるか)を BEV グリッド上で予測し、検出できない物体も含めた「通れない領域」を捉えます。

計画(このノートの主題)。プランニングヘッドは、自車を表す ego query(自車クエリ)を用意し、それが MotionFormer / OccFormer の出力に対してアテンションをかけます。つまり「周りの車がこう動く」「ここが将来塞がる」を直接見ながら自車の進路を決めます。出力は自車座標系の waypoint 列で、たとえば未来3秒を 0.5 秒刻みで6点、`[(x1,y1),...,(x6,y6)]`(単位メートル)のように出します。ego query にはナビ用の高レベル指令(左折・直進・右折)も与えられ、どの方向へ進むかの大局を決めます。

さらに重要なのが衝突回避の後最適化です。OccFormer が出した占有予測と照らし合わせ、予測軌道が将来塞がる領域に入りそうなら、占有を使ったコスト関数(衝突しそうな軌道ほど高コスト)で軌道を回避方向へ最適化し直します。ニューラルネットの素の出力よりも衝突を避けやすくなる仕掛けで、学習で得た軌道に「占有マップとぶつからない」という明示的制約を後付けで足すイメージです。

なぜクエリで繋ぐのか(設計の核心)。各段の出力を「確定した数値(検出ボックスの座標)」ではなく「クエリという豊かな特徴ベクトル」で渡すと、確信度や曖昧さを含んだ情報が後段へ流れます。計画ヘッドは「この物体は存在が怪しいが念のため避ける」といった微妙な判断ができます。これがモジュール型パイプラインの情報劣化を防ぐ鍵です。

学習。五タスクすべてに教師信号を与えますが、全体は微分可能なので、計画の損失が認識・予測の重みにまで伝わり、認識・予測が「良い計画のために」調整されます。学習は段階的(stage-wise)に行い、まず知覚(TrackFormer/MapFormer)を学ばせてから予測・計画を加える、という工夫もなされています。これは五タスクを最初から同時に学ぶと勾配が衝突して不安定になるためで、共同最適化の難しさを物語っています。

## 手法の系譜と主要論文
UniAD は Yihan Hu, Jiazhi Yang, Li Chen, Keyu Li, Chonghao Sima ら(上海 AI Lab / 香港大ほか)による "Planning-oriented Autonomous Driving"(CVPR 2023, arXiv:2212.10156)で提案され、CVPR 2023 の Best Paper Award を受賞しました。タイトルどおり「計画指向(planning-oriented)」を旗印にした点が象徴的です。

系譜の前史には、占有とコストボリュームで解釈可能な計画を行う ST-P3(Hu et al., ECCV 2022)や、鳥瞰特徴を作る BEVFormer(Li et al., ECCV 2022)があります。UniAD はこれらを束ね、知覚・予測・計画の五タスクをすべてクエリで繋ぐ単一ネットワークに統合しました。それ以前にも FIERY(Hu et al., ICCV 2021)が BEV 上で未来の占有を予測する流れを作っており、UniAD の OccFormer はこの系譜上にあります。

UniAD 以後は二方向に展開しました。一つは軽量化で、VAD / VADv2(Jiang et al., ICCV 2023 / 2024, arXiv:2303.12077 ほか)が、UniAD の重い密な表現を捨てベクトル化表現(物体や地図を密なグリッドではなくポリラインのベクトルで表す)で高速化しました。VAD は UniAD 比で大幅な推論高速化を報告しています。VADv2 はさらに軌道を語彙(候補集合)から確率的に選ぶ方向へ進み、→ [軌道語彙(クラスタアンカー)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0205_trajectory-vocabulary-anchors) に繋がります。もう一つは設計の再検討で、UniAD の「逐次にクエリを繋ぐ」設計が本当に最善かを実験で問い直した ParaDrive(→ [ParaDrive / 並列パラダイム](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0196_paradrive-parallel-planning), Weng et al., CVPR 2024)があります。

## 論文の実験結果(定量データ)
評価は nuScenes(ヌーシーンズ。自動運転の代表的な実世界データセット。多数の都市シーンに 3D 物体・地図のアノテーションが付く)で行われました。UniAD は各段の指標を従来の専用手法と比較し、追跡・地図・動き予測・占有予測・計画のすべてで改善を示しました。たとえば多物体追跡(AMOTA)や動き予測(minADE/minFDE)、占有予測(IoU)で、それぞれの単機能 SOTA を上回るか同等の値を出し、「統合してもタスクごとの精度は犠牲にならない」ことを示しました。

計画の主要指標は二つです。L2 誤差(予測軌道と正解軌道の各時刻の座標距離。単位メートル。小さいほど正解に近い)と、衝突率(collision rate。予測軌道に沿って走ると他物体に衝突する割合。低いほど安全)です。UniAD は、自車軌道のみを使う単純手法や、占有を使わない計画ヘッドに比べて、平均 L2 誤差を下げ(報告では平均 L2 が ST-P3 など先行手法より小さい水準)、特に将来時刻(3秒先)での衝突率を明確に低減したと報告されています。占有予測を計画に直接使う後最適化が、衝突率の改善に効くことがアブレーションで確認されました。

アブレーションの要点。(a) MotionFormer / OccFormer など予測モジュールを外して計画ヘッドへの入力を減らすと、計画の衝突率が悪化する。(b) 占有を使った衝突回避の後最適化を外すと衝突率が上がる。(c) 各タスクを別々に学習するより、計画損失を端から端まで流して共同最適化した方が計画が良くなる。これらは「全部を一緒に、計画を目的に学習する」という主張を支える結果です。

ただし、後続研究から強い批判も出ました。AD-MLP / "Rethinking the Open-Loop Evaluation of End-to-End Autonomous Driving in nuScenes"(Zhai et al., 2023, arXiv:2305.10430)は、自車の状態(速度・加速度・過去軌道など)だけを入れた単純な MLP(センサー画像を一切見ない)でも nuScenes の計画 L2・衝突率が UniAD に匹敵するか上回るほど高く出ることを示し、nuScenes の開ループ計画評価が「ほぼ直進・等速運動すれば良いスコアが出る」性質を持つことを指摘しました。BEV-Planner(Li et al., 2024, arXiv:2312.03031)も自車状態への依存が計画スコアを支配していることを確認しています。つまり UniAD の計画優位は、開ループ指標の偏りによって誇張されている可能性があるということです。この指摘は、計画評価を閉ループへ移す必要性という分野全体の課題を浮き彫りにしました。なお、これは UniAD の知覚・予測タスクの価値や統合という発想そのものを否定するものではなく、あくまで「計画スコアの解釈には注意が要る」という評価プロトコル批判です。

## メリット・トレードオフ・限界
メリット。全段を計画というゴールに揃えて共同最適化するので、計画に資する認識・予測表現が学べる。クエリで段を繋ぐことで情報劣化・誤差蓄積を抑える。占有予測を計画の衝突回避に直接使い、安全側の軌道を選びやすい。追跡・地図・占有という解釈可能な中間出力を持つので検査しやすい。一つのネットワークで知覚から計画まで賄えるため、別々のモジュールを保守するより全体最適を取りやすい。

トレードオフと限界。第一に、巨大・高コストで学習も推論も重く、車載のリアルタイム要件には工夫が要ります(密な BEV と五タスクのデコーダがメモリと計算を食う)。第二に、五タスク分の教師ラベル(追跡・地図・動き・占有・軌道)が必要でデータ整備が大変です。第三に、一体型ゆえモジュールを一部だけ差し替えるのが難しく、どこが原因で性能が落ちたかのデバッグも難しい。第四に、上述の通り nuScenes の開ループ評価では計画の優位が過大評価されうるため、閉ループ評価での再検証が不可欠です。第五に、共同最適化は学習が不安定になりやすく、段階的学習や損失の重み調整といったノウハウに依存します。

未解決課題としては、(1) 計画性能を正しく測る閉ループ・反応的ベンチマークの確立、(2) 巨大モデルの効率化(ベクトル化・蒸留・スパース化)、(3) 共同最適化の安定化と、各タスク損失の自動バランス、(4) 自車状態に頼らず本当にシーンを見て計画する性質をどう測り・どう促すか、が挙げられます。

## 発展トピック・研究の最前線
UniAD は「計画指向の統合 End-to-End」というパラダイムを確立し、その後の研究の基準点になりました。VAD 系の軽量化、ParaDrive による設計の実証的再検討、語彙ベース計画(VADv2 / Hydra-MDP)、そして大規模言語モデルや視覚言語モデルを計画に組み込む VLM ベースの運転(言語で交通状況を推論し説明可能性を高める方向)へと展開しています。評価面では、nuScenes 開ループの限界を受けて、NAVSIM のような非反応的だが衝突を正しく測る評価や、CARLA / nuPlan の閉ループ評価へ重心が移りつつあります。UniAD が提起した「計画を最終目的に全段を最適化する」という問いは、形を変えながら今も研究の中心にあります。
