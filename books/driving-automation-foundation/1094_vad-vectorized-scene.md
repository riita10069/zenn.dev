# VAD (Vectorized Scene Representation)

## ひとことで言うと

自動運転の E2E(end-to-end、端から端まで一気通貫)モデルの一つで、周囲のシーン(車・歩行者・車線)をすべて「点をつないだ折れ線(ベクトル)」で表すのが特徴です。UniAD のように画素を1マスずつ細かく塗る「密な表現」を避けることで、計画(自車がどう動くか決めること)を高速かつ高精度に行います。売り文句は「軽くて速いのに、当時の旗艦 UniAD と同等以上の計画性能」。VAD は Vectorized Autonomous Driving の略です。

## 直感的な理解

なぜベクトルなのか。人間が運転するとき、頭の中にあるのは画素ごとの塗り絵ではありません。「あの車線はあっちに伸びている」「前の車はこの軌跡で進みそうだ」「右の車線境界はここにある」という、線と矢印でできた幾何的なイメージです。運転の判断に必要なのは、世界の隅々まで塗りつぶした密な画像ではなく、進路に関わる要素の位置と向きと将来の動きという、ずっと少ない情報です。

UniAD はこの判断を作るために、地図を画素ごとに塗り(ラスターセグメンテーション)、未来の占有を画素・マス単位で密に予測しました。情報は豊富ですが計算がとても重く、推論が遅くなります。実車で1秒に何度も判断する必要を考えると、これは深刻な問題です。VAD はここで「計画を決めるのに本当に密な表現が要るのか」と問い、答えを「要らない、ベクトルで十分」としました。シーン全体を折れ線で表せば、密な計算が消えて軽く・速くなり、しかもベクトルは「この線を越えるな」「この軌跡と近づくな」という形で計画に直接制約として効かせやすいのです。

## 基礎: 前提となる概念

理解には密表現と疎(ベクトル)表現の違いを押さえる必要があります。

密表現(dense)とは、空間を細かい格子に区切り、すべてのマスに値を持たせる方式です。例えば 200×200 のマス目それぞれに「車線か否か」「将来塞がるか否か」を持たせます。表現力は高いですが、マス数に比例して計算とメモリが増えます。一方、疎表現(ベクトル)は、意味のある要素(車線1本、軌跡1本)だけを少数の点列で表します。例えば車線1本を20点で表せば、200×200=4万マスではなく20点で済みます。計画に関係する要素だけを保持するので圧倒的に効率的です。

次に、各構成要素の表し方です。地図要素(車線・路肩・横断歩道)は MapTR と同じく折れ線(ポリライン)で表します。エージェント(他車・歩行者)は、位置・向きに加えて「将来の軌跡」を折れ線で持ちます。自車の計画も将来軌跡という折れ線です。つまりシーンの登場人物すべてがベクトルで統一されます。

最後に、計画における「制約」という概念です。ニューラルネットが出した自車軌跡は、そのままでは交通ルールや物理を破るかもしれません。そこで「他車の軌跡に近づきすぎない」「車線境界を越えない」といった条件を、学習時の損失や推論時の調整として加えます。VAD の工夫は、すべてがベクトルなので、これらの制約を「点と点の距離」「ベクトル同士の角度」という軽い計算で表せる点にあります。

## 仕組みを詳しく

VAD のパイプラインをステップで分解します。

ステップ1は BEV 特徴を作ることです。6台のカメラ画像から、BEVFormer 系の手法で BEV(真上視点)特徴を作ります。例えば 256×200×200。ここまでは UniAD と同様です。

ステップ2はベクトル化マップ(Vectorized Map)です。車線・路肩・横断歩道などを、ラスター(塗り絵)ではなく折れ線として予測します。これは MapTR と同じベクトル地図の考え方で、各地図要素を例えば20点の (x, y) 列で表します。マップクエリ(地図を問い合わせるベクトル)を Transformer デコーダに通して点列を出します。

ステップ3はベクトル化エージェント(Vectorized Agent)です。周囲の車や歩行者について、未来の動きを折れ線(将来軌跡)として予測します。各エージェントに対しクエリを使い、「この先 T 秒間にどこを通るか」を点列で、複数の可能性(マルチモーダル、例えば6本)出します。エージェントは点ではなくベクトル(位置 + 向き + 将来軌跡)として保持されます。

ステップ4がベクトル化された計画(Planning)とベクトル制約です。自車(ego)のクエリを用意し、上記の地図ベクトル・エージェントベクトルにアテンション(関連する情報に重みをつけて参照する仕組み)を向けて、自車の将来軌跡を点列で出します。

VAD のもう一つの核心が「ベクトル化された計画制約(vectorized planning constraints)」で、3種類あります。第一に自車-エージェント衝突制約で、予測した自車軌跡が他エージェントの予測軌跡と近づきすぎないようにします(横方向・縦方向の安全距離を点同士の距離で測る)。第二に自車-境界制約で、自車軌跡が車線境界(road boundary のベクトル)を越えないようにします。第三に自車-車線方向制約で、自車の進む向きが走行中の車線の向き(車線ベクトルの方向)と大きくずれないようにします。これらはすべてベクトル同士の距離・角度で計算できるので軽く、計画を物理的・交通ルール的に妥当な方向へ誘導します。密な占有グリッドで衝突判定する方式より計算が大幅に軽くなります。

ASCII で全体像を示します。

```
6カメラ → BEVエンコーダ → BEV特徴 [256x200x200]
                │
     ┌──────────┼──────────┐
     ▼          ▼          ▼
ベクトル地図  ベクトルエージェント   ego-query
(車線=折れ線) (他車の将来軌跡=折れ線)  │
     └──────────┴──────────┘
                ▼ (アテンション + ベクトル制約)
          自車の将来軌跡(折れ線)
```

## 手法の系譜と主要論文

VAD は UniAD への直接の応答として登場しました。系譜を追います。

旗艦が UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)です。検出・追跡・地図・予測・計画をクエリで連結し、密な占有やラスター地図を駆使して多タスクを統合しました。性能は高いが重い、という特徴がありました。

ベクトル地図表現の基盤を与えたのが MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)です。順序不変な点集合で地図要素を折れ線として出す手法で、VAD のベクトル地図はこの系譜の上にあります。

そして VAD(Jiang et al., ICCV 2023, arXiv:2303.12077)です。提案は「シーン全体(地図・エージェント・計画)をベクトル化し、ベクトル制約で計画を誘導する効率的 end-to-end」。動機は UniAD の密表現が重い問題を解くことでした。密(UniAD)から疎(VAD)への転換で効率を勝ち取った、という対比で語られます。

後継が VADv2(Chen et al., 2024, arXiv:2402.13243)です。計画を「決定的に1本の軌跡を出す」のではなく「行動の確率分布を学ぶ(probabilistic planning、確率的計画)」形に拡張しました。大量の候補軌跡(語彙、vocabulary、例えば4000本超)をあらかじめ用意し、シーンに対して各候補の確率を出して選びます。これにより運転の不確かさ(複数の妥当な選択肢がある状況、例えば交差点で直進も右折も妥当な場面)を自然に扱い、closed-loop(実走行に近い)評価で性能を伸ばしました。この「軌跡語彙から確率的に選ぶ」発想は、後の Hydra-MDP などにも受け継がれています。

## 論文の実験結果(定量データ)

評価は nuScenes(屋外運転の大規模公開データ、約1000シーン、6カメラ、自車軌跡付き)の open-loop 計画で行われました。指標は L2 誤差(予測軌跡と人間の実際の軌跡の平均変位、メートル単位、小さいほど人間に近い)と衝突率(予測軌跡が他物体とぶつかる割合、小さいほど安全)です。

報告によると、VAD は UniAD と同等以上の計画精度(L2 と衝突率)を達成しつつ、推論速度を大きく改善しました。具体的には、VAD-Base が UniAD のおよそ数倍速い推論速度(報告では FPS で UniAD の2〜3倍規模)を示し、軽量版の VAD-Tiny はさらに高速で、報告では十数 FPS に達しました。UniAD が1秒に1〜2回程度の処理速度だったことを考えると、実車のリアルタイム要件に大きく近づいた成果です。「ベクトル表現は計画にとって密表現と同等に有効で、かつずっと効率的」というのが結論です。

アブレーションでは、3つのベクトル制約(自車-エージェント、自車-境界、自車-車線方向)それぞれが衝突率や L2 に寄与することが示されました。特に自車-エージェント衝突制約と境界制約を外すと衝突率が悪化し、これらが安全性に効いていることが確認されています。また、地図やエージェントのベクトル予測を補助タスクとして学習することが計画精度の底上げに寄与する、という相乗効果も報告されています。VADv2 では、軌跡語彙のサイズが性能に影響することが示され、語彙が小さすぎると表現できる行動が粗くなり、closed-loop での運転スコア(CARLA の Town05 などでの達成率)が下がることが確認されました。

注意点として、UniAD の項でも触れた通り、nuScenes の open-loop L2 指標は自車の過去運動だけからかなり当たってしまうという後年の指摘があり、VAD の open-loop 数値も同じ留保のもとで読む必要があります。VADv2 が closed-loop 評価に踏み込んだのは、この限界への応答でもあります。

## メリット・トレードオフ・限界

メリットは、ベクトル表現で密な地図・占有を避けるため UniAD より大幅に高速・軽量で実車向きなこと、ベクトル制約(衝突・境界・車線方向)を計画に直接かけられて安全・交通ルール的に妥当な軌跡を出しやすいこと、出力(自車軌跡・地図・他車軌跡)がすべて折れ線で人間にも解釈しやすく後段で扱いやすいことです。

トレードオフと限界もあります。不定形・想定外の障害物(落下物、特殊形状の工事車両など、密な占有グリッドが得意とする対象)の表現力は密表現に劣りえます。ベクトル地図・軌跡の教師ラベルが必要でデータ作成コストがかかります。open-loop 評価中心の初期 VAD は実走行での因果的妥当性に課題があり、VADv2 で確率的計画として補強されました。ベクトル要素を固定点数で表すため、極端に複雑な形状の表現には上限があります(MapTR と共通の制約)。

## 発展トピック・研究の最前線

研究の最前線は3方向に整理できます。第一に確率的・マルチモーダル計画の深化で、VADv2 の軌跡語彙の考え方を発展させ、複数の妥当な運転候補をスコア付けして選ぶ枠組み(Hydra-MDP など)が closed-loop ベンチマークで成果を上げています。第二にさらなる疎化で、BEV 特徴自体を密に持つことすらやめ、完全に疎なクエリだけでシーンを表す SparseDrive 系が、効率を一段押し進めています。第三に評価基盤の刷新で、open-loop の限界を踏まえた CARLA ベースの closed-loop ベンチマーク(Bench2Drive)や、産業界の NAVSIM のような疑似 closed-loop 評価が標準になりつつあります。VAD は「密から疎への転換」という設計思想を確立した点で、効率的 E2E の系譜の起点として参照され続けています。
