# Lanelet2

## ひとことで言うと
Lanelet2 は、車線・信号・規制(止まれ、進入禁止、制限速度など)と、それらの関係性を、レイヤ(層)に分けて構造的に表現する HD 地図(High Definition map=車線レベルまで精密な地図)のフォーマットです。フォーマットとは「地図データをどういう約束ごとで書き表すか」という規格のことです。重要なのは、これは「地図を絵として描く」ものではなく、「機械が運転の判断に使える形で地図の意味と関係を記録する」ものだという点です。地図をオンラインで作る手法とは目的が違い、こちらは「すでに作った地図を、自動運転ソフトが読める形でどう持つか」を定めます。

## 直感的な理解
人間がカーナビの画面を見て運転するとき、頭の中では「次の交差点は左に2車線あって、自分は右車線にいるから、左折するには車線変更が要る」「あの信号は自分の車線用だ」というふうに、絵そのものではなく意味と関係を読み取っています。自動運転ソフトに地図を渡すときも同じで、ただのピクセル画像や点の集まりを渡しても「次にどの車線へ進めるか」「この信号は自分の車線にかかるか」には答えられません。

たとえば次のような問いに、地図が答えられないと自動運転は成立しません。

- いま走っている車線の次に、どの車線へつながっているか(右折専用レーンはどれか)。
- この交差点に対応する信号機はどれか。複数ある信号のうち、自分の進路を支配するのはどれか。
- ここの制限速度はいくつか。一方通行か。
- この横断歩道は、どの車線を横切る形で存在するか。

これらに答えるには、「この線は車線の左境界」「この車線片とこの信号は紐づいている」といった意味と関係を地図データそのものに埋め込む必要があります。Lanelet2 はまさにこの「意味と関係を持った地図」を表現するための規格です。地図を「眺めるための絵」から「問い合わせに答えるデータベース」へ昇格させる、と考えると本質をつかめます。

## 基礎: 前提となる概念
先に、登場する用語を知らない人向けに噛み砕いておきます。

- HD 地図と SD 地図: HD 地図は車線の境界線・停止線・信号との対応まで数十 cm 精度で持つ地図です。SD 地図(Standard Definition map=標準精度地図)はカーナビが使う粗い地図で、道路を中心線 1 本くらいでしか持ちません。Lanelet2 は HD 地図側の表現です。詳しくは [ナビ地図 vs HD地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0422_nav-map-vs-hd-map) を参照してください。
- OpenStreetMap(OSM): 誰でも編集できる無料の世界地図プロジェクトです。OSM は地図データを「ノード(点)・ウェイ(点をつないだ線)・リレーション(要素同士の関係)・タグ(key=value の属性)」という素朴なデータモデルで表します。Lanelet2 はこの OSM のデータモデルをそのまま土台に借りています。だから Lanelet2 地図はファイルとしては OSM の XML(`.osm` 形式)に見えます。
- セマンティクス(意味づけ): 「この線はただの線」ではなく「この線は車線の右境界である」と意味を与えることです。Lanelet2 の核心はこのセマンティクスにあります。意味づけは「タグ」で行います。たとえば LineString に `type=line_thin, subtype=solid` を付けると「これは細い実線(=車線変更不可の境界)」という意味になります。
- レイヤ構造: 地図を役割の違う層に分けて積み重ねる考え方です。一番下の層は素朴な幾何(点と線)、その上の層はその幾何に意味づけした要素(車線片・規制)、さらに上の層は要素同士の関係(この車線の次はこの車線、この車線にこの信号がかかる)です。下から上へ段階的に意味を重ねるので、複雑な交通環境を整理して表せます。
- ルーティンググラフ(routing graph): 道路網を「ノード(車線片)」と「エッジ(つながり)」の数学的なグラフとして表したものです。グラフにしておくと、経路探索が「グラフ上の最短経路問題」という枯れたアルゴリズムにそのまま乗ります。

## 仕組みを詳しく
Lanelet2 は地図を次のプリミティブ(基本構成要素)で組み立てます。下層の幾何から上層の意味・関係へと積み上がっていきます。

1. Point(点)。緯度経度、あるいは局所平面座標(自車基準の x,y)を持つ 1 点です。すべての土台になります。
2. LineString(線)。点を順につないだ折れ線です。車線の左境界・右境界、停止線、横断歩道の縁などに使います。
3. Polygon(多角形)。線で囲んだ面です。駐車禁止エリアや特定領域の指定に使います。
4. Lanelet(車線片)。「左境界の LineString」と「右境界の LineString」のペアで、1 つの走行可能な車線片を表します。これがフォーマット名の由来です。実際の道路は、進行方向ごと・区間ごとに細かい Lanelet に分割し、それらを前後・左右の関係でつないで道路網(Lanelet のグラフ)にします。
5. Area(エリア)。駐車場や広場のように、車線という細長い帯では表しにくい「面として走れる空間」を表します。
6. RegulatoryElement(規制要素)。信号・一時停止・制限速度・優先関係(右方優先など)を表す要素で、Lanelet に紐づけて「この車線にこの規制がかかる」と表現します。

具体例で動きを追います。十字路で「いまの車線から直進すると次にどの車線へ入れるか」を知りたいとき、現在の Lanelet が持つ「後続(successor)」関係をたどれば次の Lanelet が分かります。信号待ちなら、その Lanelet に紐づく RegulatoryElement(種類は traffic_light)を見れば、自分の進路を支配する信号が一意に分かります。「この線が左境界」「この車線の後続はこれ」「この信号はこの車線用」という関係が、すべて id の参照で結ばれているのがポイントです。

データの持ち方を具体的に書くと、各 Point に一意の id が振られ、LineString は Point の id 列を、Lanelet は左右の LineString の id を、RegulatoryElement は対象 Lanelet と信号などの id を参照します。ファイルとしては OSM XML を拡張したテキストで、`type=lanelet` や `subtype=solid`(実線)、`subtype=traffic_light` といったタグで意味を区別します。境界線の subtype(solid=実線で越えられない、dashed=破線で車線変更可など)は、走行可能性の判断に直結する重要なセマンティクスです。

```
   左境界 LineString (id=100, subtype=solid): pt1 - pt2 - pt3 ...
   ┌───────────────────────────────────────┐
   │   Lanelet (id=300)  走行可能な車線片     │ → successor: Lanelet(id=305)
   │                                         │ → left/right: 隣接車線への参照
   └───────────────────────────────────────┘ → regulatory: traffic_light(id=900)
   右境界 LineString (id=101, subtype=dashed): pt4 - pt5 - pt6 ...
```

座標系にも注意が要ります。Lanelet2 は緯度経度(WGS84)で点を持ちますが、実際の幾何計算は平面に投影してから行います。一般的には UTM(横メルカトル図法)や、地図中心を原点とする局所メートル座標へ投影します。投影しないと「2 点間の距離」や「車線の幅」をメートルで素直に測れないためです。これは「地球は丸いが、局所的には平面とみなして計算する」という、測量で一般的な近似です。

なぜこの設計かというと、自動運転の意思決定はすべて「どの車線にいて、次にどこへ行けて、どんな規制がかかるか」という関係の問い合わせだからです。幾何だけ持っていても、その都度「この線とこの線は同じ車線か」を計算で復元しなければならず、現実的でも安全でもありません。意味と関係を地図に最初から埋めておくことで、経路探索・信号認識対象の決定・走行可能領域の判定が、すべて単純なグラフ探索と参照に還元されます。

ルーティングモジュールの動作も具体化しておきます。Lanelet 同士の幾何的な接触(片方の終端の境界点が、もう片方の始端の境界点と一致する)を判定して「後続」エッジを張り、隣接して同じ向きに走れる車線片の間には「左右移動可能」エッジを張ります。こうしてできたグラフ上で、出発 Lanelet から目的 Lanelet までをダイクストラ法や A* で探索すれば、車線変更を含む経路が求まります。「人間が地図を見て頭の中でやっている経路の組み立て」を、グラフ探索という機械的手続きに落とし込んだものだと理解できます。

## 手法の系譜と主要論文
- liblanelet / 初代 lanelet(Philipp Bender, Julius Ziegler, Christoph Stiller, IV 2014, "Lanelets: Efficient Map Representation for Autonomous Driving")。これが起源です。1 本の道を「左右の境界線のペア」という最小単位(lanelet)に分割するアイデアを提示しました。当時の動機は、車線レベルの計画に使える軽量な地図表現が欲しかったことです。この論文は Bertha Benz Memorial Route(独 Karlsruhe〜Pforzheim、約 103km の歴史的自動運転デモ走行)で用いられた地図表現の基礎になりました。ただし規制や複雑な関係表現は限定的でした。
- Lanelet2(Fabian Poggenhans, Jan-Hendrik Pauls, Johannes Janosovits, Stefan Orf, Maximilian Naumann, Florian Kuhnt, Matthias Mayr, ITSC 2018, "Lanelet2: A High-Definition Map Framework for the Future of Automated Driving")。初代を大幅に拡張した後継です。動機は、自動運転の意思決定に必要な意味・規制・関係まで一貫して表せ、かつ拡張可能で実装(C++/Python ライブラリ)が揃った HD 地図フォーマットが当時不足していたことでした。新規性は、(1) OSM データモデルを土台にして既存の編集ツール資産(JOSM など)を流用できるようにしたこと、(2) RegulatoryElement という抽象で信号・速度・優先などの規制を統一的に扱い、ユーザが独自の規制も足せる拡張性を持たせたこと、(3) 幾何・意味・関係をレイヤ分離し、ルーティング(経路探索)用のグラフ構造を標準で備えたことです。
- 同時代の競合・並走規格として OpenDRIVE(ASAM が標準化、2000 年代後半から発展、現在 1.7/1.8 系)があります。OpenDRIVE は道路を「リファレンスライン+横方向オフセット」で連続的に表すパラメトリックな表現で、シミュレータ(CARLA など)で広く使われます。Lanelet2 が「離散的な車線片のグラフ」なのに対し、OpenDRIVE は「連続曲線(直線・円弧・クロソイド曲線・3 次多項式)で道路形状を記述」する設計で、両者は変換しあえる関係にあります。自動運転スタックでは、実車走行の地図に Lanelet2、シミュレーション環境に OpenDRIVE、という使い分けがよく見られます。
- なお、HERE や TomTom などの商用 HD 地図、NDS(Navigation Data Standard)といった業界標準も並立しています。これらはオープンな Lanelet2/OpenDRIVE とは出自が異なりますが、「意味と関係を持った車線レベル地図」という思想は共通しています。

## 論文の実験結果(定量データ)
Lanelet2 は予測精度を競う機械学習モデルとは性質が異なり、「地図表現の枠組み」を提案する論文なので、ベンチマークの精度スコアという形の評価とは馴染みません。ITSC 2018 の原論文での評価は、主に表現力(どれだけ多様な交通シーンを正しく記述できるか)と実用性(ライブラリの機能・性能)を示すものになっています。報告されている定性・定量の要点は次のとおりです。

- ルーティング性能。標準で提供される routing module は、Lanelet グラフ上の経路探索(車線変更を含む到達可能性判定や最短経路)を行えます。論文では実都市規模の地図に対して経路探索が実用的な時間で動くことが示されています。グラフのノード数は地図中の Lanelet 数に比例し、典型的な都市の一区画でも数千ノード規模で、ダイクストラ系探索がミリ秒〜数十ミリ秒で完了する水準です(報告による)。リアルタイム制御ループ(数十 Hz)の中で繰り返し問い合わせても破綻しない速度感です。
- 表現の網羅性。論文は、信号交差点・環状交差点・複数車線・優先関係・速度規制など、実世界の主要な交通要素を Lanelet と RegulatoryElement の組み合わせで記述できることをケーススタディで示しました。「どれだけ精度が上がったか」ではなく「従来表現では書けなかった関係を一貫して書けるようになった」ことが成果です。
- 拡張性の実証。RegulatoryElement を継承して独自規制を定義できる設計により、後続の研究・実プロジェクトが、信号と一時停止以外の地域固有ルール(国ごとに異なる優先規則など)を追加できることが実例で示されています。

定量的な「ベースライン比でどれだけ良いか」という数字が出にくいのは、Lanelet2 が予測器ではなくデータモデルだからです。研究者として読むときは、「精度ベンチ」ではなく「表現の完全性・実装の成熟度・他規格との相互運用性」で評価する対象だと理解するのが正しい読み方です。逆に、機械学習の文脈では Lanelet2 は「評価指標を出すモデル」ではなく「教師ラベル(ground truth)の供給源」として現れます。オンライン地図構築モデルを学習・評価するとき、Lanelet2 由来の車線ベクトルを正解として mAP を測る、という使われ方が典型です。

## メリット・トレードオフ・限界
- メリット。意味と関係を構造的に持てるので、経路計画・信号認識対象の決定・規制順守の判定に直接使えます。グラフ探索に還元できるため実装が堅牢です。
- メリット。オープンで C++/Python ライブラリが揃い、実運用での実績があります。OSM データモデル由来なので既存の地図編集ツール(JOSM)を流用でき、OpenDRIVE との相互変換も整備されています。
- メリット。レイヤ構造で拡張しやすく、独自の規制要素を後付けできます。地域固有のルールを汚染なく追加できます。
- トレードオフ。地図を事前に作り込む必要があります。専用計測車(LiDAR + 高精度 GNSS/IMU)で計測し、人手で意味づけ・規制紐づけを編集するコストがかかります。オンライン地図構築のような「センサから自動で地図を起こす・鮮度を保つ」性質はありません。
- トレードオフ。大規模化すると作成・保守コストが上がります。道路工事や信号の変更があれば地図を更新する運用負荷がかかり、地図が古いと自動運転が誤った前提で走る危険があります。これは「HD 地図の鮮度問題」と呼ばれ、商用展開での主要なコスト要因です。
- 限界。表現が車線中心なので、車線として書きにくい場所(広い駐車場、未舗装路、私有地)は Area などで補いますが、扱いがやや難しく、表現の自然さで OpenDRIVE のパラメトリック表現に劣る場面があります。
- 限界。離散グラフ表現ゆえ、なめらかな曲率の連続的記述は苦手で、滑走するような軌道生成にはスプライン補間など追加処理が要ります。逆に OpenDRIVE はクロソイド曲線で曲率を連続的に記述できるため、この点では補完関係にあります。

研究上の未解決課題として、HD 地図の鮮度問題(作った瞬間から古くなる)を、オンライン地図構築や走行ログからの自動更新とどう組み合わせるか、という統合課題があります。事前作成型の Lanelet2 と、走りながら作るオンライン地図のハイブリッドが研究の焦点の一つです。さらに、人手の意味づけ作業を機械学習で半自動化する「地図の自動アノテーション」も活発な領域です。

## 発展トピック・研究の最前線
最近の流れは「事前作成 HD 地図への依存をどう減らすか・どう補うか」に向かっています。一方の極が、HD 地図を捨ててカメラからオンラインで車線地図を起こすオンライン地図構築です([HDMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0442_hdmapnet)、[MapTRv2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0452_maptrv2))。もう一方が、粗い SD 地図(OSM など)を弱い事前情報として注入し、見えない先の道路接続を補う手法です([OpenStreetMap SD地図事前情報](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0432_openstreetmap-sd-map-prior))。Lanelet2 のような事前作成 HD 地図は、これらと対立するというより、(1) 学習・評価の教師ラベル源として、(2) 安全保証の効く運用基盤として、依然中心的な役割を持ち続けています。実務では、Lanelet2 の精密で意味づけされた地図を真値とし、オンライン手法でその鮮度の穴を埋める、という二段構えが現実的とされています。

別の最前線として、地図を「固定の入力」ではなく「学習可能なベクトルトークン」として end-to-end の運転 Transformer に渡し、知覚から計画までを一気通貫で扱う流れがあります。この文脈では、Lanelet2 のベクトル要素(車線中心線・境界・規制位置)を、そのまま Transformer のトークン列に変換して注入できることが利点になります。離散・構造化された Lanelet2 表現が、ベクトル系の学習パイプラインと素直に接続できるという意味で、古いフォーマットが新しい学習法に再利用されているのが現状です。
