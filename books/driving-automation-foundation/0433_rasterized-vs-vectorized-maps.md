# ラスタ地図とベクトル地図

## ひとことで言うと
コンピュータで地図を表す方法は大きく2種類あります。1つは「画素 (ピクセル) の絵」として持つラスタ地図、もう1つは「点と線の数式的なデータ」として持つベクトル地図です。ナビ地図は手軽さからラスタで、自動運転用の高精度地図 (HD 地図) は精度のためベクトルで扱われることが多く、両者には別々のエンコーダを使い分けるのが現代的な設計です。どちらが優れているという話ではなく、表現するものと求める精度に応じて選ぶ、というのが要点です。

## 直感的な理解
同じ「地図」でも、目的によって必要な情報の質が違います。知らない街でだいたいの道順を知りたいだけなら、スマホのナビ画面のような絵地図(ラスタ)で十分です。「この大通りを進んで2つ目を右」が読めればよく、車線が何本あるかまでは要りません。一方、車が自分で「今いる車線の中心はどこか」「停止線はあと何メートル先か」を判断するには、車線1本ずつの正確な形が必要で、ぼやけた絵地図では足りません。ここで精密なベクトル地図 (HD 地図) が要ります。

ラスタとベクトルは、写真とフォントの関係に似ています。写真を拡大すると画素が荒れますが (ラスタ)、フォントの輪郭は数式で定義されているのでいくら拡大しても滑らかです (ベクトル)。地図も同じで、ラスタは解像度で精度が頭打ちになり、ベクトルは幾何が正確に保てます。もう1つの本質的な違いは「意味が明示されているか」です。ラスタ地図では「ここが車線だ」という情報は色の見た目に埋もれていてデータとして取り出せませんが、ベクトル地図では「これは車線中心線」というラベルが構造として直接付いています。

## 基礎: 前提となる概念
ラスタ (raster) 地図は、縦横に並んだ画素の格子で地図を表したものです。1画素ごとに色 (RGB) が入っているだけで、「ここが車線だ」という構造的意味はデータとして明示されません。意味はあくまで「絵の見た目」に埋め込まれています。テンソルでいえば `(3, H, W)` の画像です。複数の意味を別チャンネルに分けて持つ「マルチチャンネルラスタ」(例えば「車線」チャンネル・「横断歩道」チャンネル・「他車」チャンネルを別の2値画像として重ねる)もよく使われ、これだと意味が多少分離されますが、依然として解像度で精度が制限されます。

ベクトル (vector) 地図は、「この車線中心線は座標 (x1,y1),(x2,y2),... を結んだ折れ線」「この横断歩道はこの多角形」というふうに、地図要素を点と線(ジオメトリ)の集合として持ちます。各要素にはクラス(車線、停止線、境界線など)のラベルと、要素どうしのつながり(どの車線の先にどの車線へ行けるか、というトポロジー)が付くこともあります。HD 地図 (High-Definition map) はこの形式で、横方向の精度はセンチメートル級に達します。

ポリライン (polyline) は複数の点をつないだ折れ線で、ベクトル地図で車線や境界を表す基本単位です。詳しくは [ベクトル地図のポリライン符号化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0465_vectorized-map-polyline-encoding)。

ラスタとベクトルの違いの本質は、情報の質と粒度です。ラスタなナビ地図は「だいたいこの道を行く」という大局的な経路ガイドには十分ですが、車線数や停止線の位置といった精密情報はぼやけています。ベクトルな HD 地図は車線レベルで正確ですが、構造化されていてそのままでは画像用エンコーダに通せません。だから「ラスタにはラスタの、ベクトルにはベクトルの、それぞれ適したエンコーダ」を用意して使い分ける必要が出てきます。

## 仕組みを詳しく
2系統のエンコーダの典型的な使い分けを見ます。

ラスタ地図(ナビ地図)の場合。入力は地図を描いた RGB 画像、例えば `(B, 3, 256, 256)` です。これを画像バックボーン(CNN や Vision Transformer)に通し、BEV 特徴 `(B, embed_dim, h, w)`(embed_dim は例えば 256)を作ります。画像をそのまま畳み込み系ネットで特徴に潰す素直な流れで、実装が容易です。CNN は局所的な模様(線・角・テクスチャ)を捉えるのが得意なので、絵として描かれた道の形はそれなりに拾えます。

ベクトル地図(HD 地図)の場合。入力は「固定数のインスタンス × 固定数の点 × 座標」という構造化テンソル、例えば `(B, M, N, 2)`(M はインスタンス数、N は1要素あたりの点数、2 は x,y 座標)です。これをポリライン専用エンコーダ(VAD の map モジュールや MapTR 系)に通し、点を埋め込み、ポリライン内の点どうしやポリライン間の関係を attention やグラフで集約し、最後に BEV 特徴 `(B, embed_dim, h, w)` にそろえます。具体的な符号化形式は [ベクトル地図のポリライン符号化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0465_vectorized-map-polyline-encoding)。

before / after のイメージは次の通りです。

```
ラスタ:  地図画像 (B,3,256,256) ──CNN/ViT──▶ (B,256,h,w)
ベクトル: 地図ファイル ──parse──▶ ポリライン集合 (B,M,20,2) ──VAD系──▶ (B,256,h,w)
```

両方とも最後は同じ `(B,256,h,w)` の BEV 特徴になり、画像 BEV と融合できます([独立マップエンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0435_separate-map-encoder))。「最終的に共通の BEV 特徴に落とす」という設計のおかげで、地図がどちらの形式で来ても下流(融合・計画)は同じ作りで済みます。

HD 地図のファイルフォーマットには主に2系統があり、両対応が望まれます。

Lanelet2 は、車線 (lanelet) を左右の境界線の組として表し、つながり(どの車線の次にどの車線へ行けるか)や交通規則(信号・優先・速度制限)も持つフォーマットです (Poggenhans et al., ITSC 2018)。OpenStreetMap の XML を拡張した形式で、研究・開発のオープンソース基盤で標準的に使われます。「左右2本の境界線で1車線を定義する」という発想が特徴で、車線中心線はその2本から計算で導けます。

OpenDRIVE (.xodr) は、自動車業界とシミュレータ(CARLA など)で広く使われる道路記述フォーマットです (ASAM が標準化)。道路を参照線 (reference line) とそこからのオフセットで数式的に表します。参照線は直線・円弧・クロソイド曲線(曲率が距離に比例して変化する、実際の道路設計に使われる曲線)・3次多項式などのプリミティブを連結して定義されます。連続曲線をパラメータで持つため、任意の点で滑らかに座標と曲率を計算できるのが強みです。

両対応にするのは、用途によって流通する形式が違うためです。研究・開発基盤では Lanelet2、シミュレーションや一部の商用データでは OpenDRIVE が一般的です。どちらで来ても、最終的に「ポリラインの集合」という共通の中間表現に変換してからベクトルエンコーダに渡す、という設計が定石です。これにより、フォーマットの違いをパーサで吸収し、下流のネットワークは1種類の表現だけ扱えばよくなります。実装上は、OpenDRIVE の連続曲線を等間隔サンプリングしてポリライン化する、Lanelet2 の左右境界から中心線を補間する、といった変換が前処理に入ります。

## 手法の系譜と主要論文
地図表現はラスタからベクトルへと進化してきました。系譜をたどります。

ラスタ系の原点は ChauffeurNet (Bansal et al., RSS 2019, Waymo) です。地図・経路・他車・自車軌跡を複数のラスタ画像チャンネルとしてレンダリングし、CNN で処理して運転を模倣学習しました。環境を上から見たラスタ画像群に統一して表現する発想で、実装が単純で多様な情報を1枚(複数チャンネル)の絵に詰められるのが利点です。トレードオフは画素解像度で精度が頭打ちになり、車線の正確な幾何が表しにくいことです。同時期の軌跡予測分野でも、地図と物体をラスタ化して CNN に通す手法(例えば MultiPath や CoverNet 系)が主流でした。

ベクトル系への転換点は VectorNet (Gao et al., CVPR 2020, Waymo) です。地図と物体の軌跡を「ポリライン(点列)」としてベクトル化し、ポリライン内をサブグラフ、ポリライン間をグローバルグラフで集約しました。「ラスタ化して CNN に通すより、もともと構造を持つ地図はベクトルのまま扱う方が情報を失わず効率的」という主張で、メモリ・計算の効率と精度を両立させました。トレードオフはベクトルデータの整備が前提になることです。

オンライン地図構築への発展が HDMapNet (Li et al., ICRA 2022) と MapTR (Liao et al., ICLR 2023) / MapTRv2 (IJCV 2024) です。HDMapNet はまずセンサから BEV のセマンティックセグメンテーション(画素ごとのクラス分類)を作り、それを後処理でベクトル化する2段構えでした。MapTR は地図を「固定数のポリライン要素」として end-to-end に直接検出し、各要素を等間隔点で表し、点順序の曖昧さ(同じ車線でも始点と終点を入れ替えても同じ形)を扱う順列同変なマッチングを導入しました。これにより、あらかじめ作った HD 地図に頼らず、その場のセンサからリアルタイムにベクトル地図を生成できるようになりました。詳細は [ベクトル地図のポリライン符号化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0465_vectorized-map-polyline-encoding)。

計画との統合が VAD (Jiang et al., ICCV 2023) です。MapTR 流のベクトル地図表現を、地図予測だけでなく軌跡計画まで一体で使い、ベクトル表現の効率性を計画の高速化に直結させました。「ベクトル地図には VAD/MapTR 系の map モジュール」という設計指針はこの系譜から来ています。

要するに、地図表現はラスタ (ChauffeurNet 系) からベクトル (VectorNet → HDMapNet → MapTR → VAD 系) へと進化し、ラスタは手軽さ、ベクトルは精度という別々の価値を持つため、現代的なシステムでは両方を使い分けます。

## 論文の実験結果(定量データ)
ベクトル地図構築の精度は mAP (mean Average Precision, 平均適合率) で測られます。ここでは、推定した車線・境界・横断歩道などのポリラインが正解とどれだけ一致するか(チャンファー距離という点列の近さの閾値で判定)を、クラス平均で表します。値が大きいほど良いです。チャンファー距離は2つの点集合間の「各点から相手の最近傍までの距離」を双方向に平均したもので、点列の形の近さを測ります。

HDMapNet は、ラスタのセグメンテーションを経由する2段方式で、当時のラスタ系ベースラインに対して地図構築の精度向上を示しました。MapTR の報告では、nuScenes のオンライン地図構築で、このセグメンテーション系を含む旧来手法を上回る mAP(報告でおよそ 50 台)を、リアルタイムに近い速度で達成しました。MapTRv2 はさらに改良され、報告によれば nuScenes でおよそ 60 台後半の mAP に到達し、推論速度も実用的水準を保ちました。ラスタ化してセグメンテーションで地図を作る旧来手法に比べ、ベクトルのまま end-to-end に扱うことで精度と速度の両方を改善した、というのが定量的な要点です。

計画への効果としては、VAD が nuScenes 計画タスクで、ラスタや稠密 BEV を扱う構成と同等以上の計画精度(平均 L2 誤差でおよそ 0.7m 前後)を、報告によりベースラインの数倍の推論速度で達成しました。L2 誤差は予測軌跡と人間の走行軌跡の位置ずれ(メートル)で、小さいほど良い指標です。ベクトル地図が「精度を保ちつつ計算を節約する」効果を持つことを示しています。

アブレーション的には、ラスタ表現で解像度を下げると車線の細い構造が潰れて地図 mAP が落ちること、逆にベクトルで点数 N やインスタンス数 M の上限を絞りすぎると複雑な交差点で要素が溢れて精度が頭打ちになることが報告されており、それぞれの表現の限界が数値で確認されています。横断歩道のように面で広がる要素はラスタが比較的得意、車線分割線のように細く長い要素はベクトルが得意、という傾向もアブレーションから読み取れ、クラスによって向き不向きがあることが分かります。

## メリット・トレードオフ・限界
メリット。ラスタは入手・生成が容易(ナビ画面のキャプチャや GPS からの合成 [OSMマップマッチング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0431_map-matching-osm) で作れる)で、大局的な経路情報を手軽に与えられ、CNN という枯れた道具で処理できます。ベクトルは車線レベルで正確、構造を保持でき、計画に直接効く精密情報を渡せ、メモリ効率も良いです。両方を使い分けることで、データセットや地図の有無に応じて柔軟に対応できます。

トレードオフと限界。ラスタは解像度で精度が頭打ちになり、車線の正確な幾何や接続関係を表しにくいです。ベクトルはデータ整備のコストが高く、Lanelet2/OpenDRIVE のパーサや前処理が要ります。2系統のエンコーダを保守する負担、フォーマットごとの変換コードも増えます。さらに、ラスタとベクトルで地図の見た目・分布(ドメイン)が違うため、学習・評価で混在させると分布のずれが生じうる点も注意が必要です。どちらの形式でも、地図が現実と食い違う(古い、工事中、未整備)場合の頑健性は別途確保する必要があります。

未解決の課題としては、ラスタとベクトルのどちらに統一すべきか(あるいは両方を融合すべきか)、異なるフォーマット・地域の地図をまたいで汎化する表現をどう作るか、事前 HD 地図とオンライン推定地図をどう整合させるか、地図への過依存(地図を信じすぎて目の前の障害物を軽視する)をどう防ぐか、が研究的に重要です。

## 発展トピック・研究の最前線
近年は、事前作成の HD 地図への依存を減らす「マップレス運転 (mapless driving)」が活発です。MapTR 系のオンライン地図構築を計画に直結させ、車が走りながら必要なベクトル地図を生成します。さらに、地図要素を画素でも固定ポリラインでもなく「クエリ可能なベクトル場・暗黙関数」として表す研究、車線のトポロジー(つながり)をグラフとして明示的に予測する研究(TopoNet など)も進んでいます。標準化の面では、Lanelet2 と OpenDRIVE の相互変換や統一中間表現の整備が進み、フォーマット間のギャップを下流ネットワークから隠す方向に向かっています。標準ナビ地図 (SD map) を弱い事前情報としてオンライン地図構築に与え、精度を底上げする研究も登場しており、「軽いラスタ的ヒント + 精密ベクトル推定」というハイブリッドが1つの潮流です。
