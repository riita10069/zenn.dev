# System1/System2分離（Reactive/Reasoning）

## ひとことで言うと
人間の思考に「とっさの直感的な判断（System1）」と「じっくり考える熟考（System2）」の2種類があるように、知能を持つエージェントも「高頻度で反射的に動く部分」と「低頻度でじっくり推論する部分」に分ける、という設計思想です。自動運転においては、ミリ秒単位で安全を守る制御層と、秒単位で先を読む計画・推論層を、別々の頻度・別々の重さで動かします。こうして応答速度（速く動けること）と知能の深さ（深く考えられること）を同時に成り立たせようとする二系統アーキテクチャです。

## 直感的な理解
運転という行為には、性質のまったく異なる2種類の判断が混ざっています。

1つ目は「目の前に飛び出した自転車を避ける」「車線の中央を保つ」「先行車が急ブレーキしたら踏む」といった、ミリ秒単位の素早い反応です。これは複雑な推論をしている暇がなく、見たものに対してほぼ反射的に操作を返す必要があります。0.1秒の遅れが事故に直結する領域です。

2つ目は「この先で渋滞しているから1本裏道に入ろう」「対向車が来そうだから先に右折を譲ろう」「あの歩行者は子どもだから飛び出すかもしれない」といった、数秒から数十秒スパンの予測と計画です。これは状況を理解し、未来を想像し、選択肢を比べる、いわば熟考です。

これらを1つのニューラルネットワークに同じ頻度で全部やらせると不都合が起きます。熟考のための重い計算（例えば大規模言語モデルによるシーン理解）を毎フレーム（10Hz = 0.1秒ごと）走らせると計算が間に合わず、反射が遅れます。逆に反射に合わせて計算を軽くすると、深い推論ができません。そこで「反射は速く小さく、推論は遅く重く」と役割と動作頻度を分けるのが System1/System2 分離です。これは車の世界に限らず、ロボット制御全般・ゲームAI・対話エージェントなど、応答速度と推論の深さがトレードオフになるあらゆる場面に現れる普遍的な構図です。

身近な例えで言えば、熟練したドライバーがハンドルを握りながら「次の交差点をどう曲がるか」を考えているとき、車線維持や微妙なアクセル調整は無意識（反射）で行われ、意識（熟考）はもっと先の戦略に向いています。脳が同時に2つの時間スケールで動いているのと同じ構造を、機械の中に作るわけです。

## 基礎: 前提となる概念
まず「頻度（Hz）」を理解しておきます。Hz（ヘルツ）は1秒間に何回実行するかの単位です。10Hz は0.1秒ごとに1回、1Hz は1秒に1回です。自動運転の制御ループは一般に10〜20Hz で回り、知覚（カメラ画像から物体を検出する処理）もこのくらいの頻度が要求されます。一方、言語的・記号的な推論は重いので、それを毎フレーム回すのは現実的でないことが多いのです。具体的な数字感を持っておくと、大規模 Vision-Language Model（VLM）の1回の推論は数百ミリ秒〜数秒かかることがあり、これはそのまま10Hz の反射ループには載せられないことを意味します。

次に「エンドツーエンド（end-to-end, E2E）」という言葉です。従来の自動運転は「知覚→予測→計画→制御」と機能ごとにモジュールを分け、それぞれを別々に作って繋いでいました（モジュラー方式）。これに対しエンドツーエンドは、センサ入力（カメラ画像など）から運転操作（軌跡やステアリング）まで、1つの学習可能なネットワークで一気通貫に学ぶ方式です。中間表現を人間が設計しない分、データから最適な特徴を学べる可能性がありますが、ブラックボックス化しやすいという弱点もあります。

「世界モデル（world model）」も重要です。これは「いま自分が行動したら、次に世界がどう変わるか」を内部で予測するモデルです。人間が「ここでブレーキを踏んだら前の車との距離はこう縮まる」と頭の中でシミュレートするのと同じです。System2 側はしばしばこの世界モデルを内包します。

最後に「軌跡（trajectory）」です。自動運転モデルの出力としてよく使われるのは、未来数秒間の自車の通る点列（例えば0.5秒刻みで6点、各点が x, y 座標）です。これを「プランニングされた軌跡」と呼び、その後の低レベル制御器（PID やモデル予測制御）がこの軌跡を追従します。

## 仕組みを詳しく
System1/System2 分離アーキテクチャは、典型的には次の2系統で構成されます。

System1（Reactive、反射側、高頻度 ~10Hz）の役割は、現在の観測から直接、安全な短期軌跡を出すことです。一般的な処理の流れをテンソルの形（shape）とともに示します。`B` はバッチサイズ、`V` はカメラ台数（サラウンドビューなら6〜7）、`H, W` は画像の縦横です。

```
入力:
  camera_images : [B, V, 3, H, W]    V台のカメラ画像
  map / nav     : [B, 3, H_map, W_map] ナビ用のBEVマップ画像
  ego_history   : [B, T, D_ego]       自車の過去の速度・操舵などの履歴
  context_vec   : [B, D_ctx]          System2から渡される視覚文脈ベクトル

処理:
  1. Backbone が各カメラ画像から特徴を抽出     → [B, V, C, h, w]
  2. View Fusion が多視点特徴をBEV空間に統合    → [B, C_bev, H_bev, W_bev]
  3. Map とBEVを融合                            → [B, C_bev, H_bev, W_bev]
  4. Temporal Memory が履歴と context を圧縮    → [B, D_mem]
  5. Planner が融合特徴と文脈から軌跡を出力     → [B, T_future, 2]
```

ここで「BEV（Bird's-Eye-View）」は鳥瞰図、つまり真上から見下ろした2次元平面のことです。複数のカメラ画像をこのBEV平面に投影して統合すると、車の周囲を1枚の地図のように扱えて、計画が立てやすくなります。

System2（Reasoning / World Model、熟考側、低頻度 ~1Hz）の役割は、重い視覚理解や未来予測を行い、その成果を圧縮した文脈ベクトル（上の `context_vec`）として System1 に供給することです。System2 が世界モデルとして実装される場合、「数フレーム後の世界の見え方（特徴）」を予測し、その予測誤差を自己教師あり学習の信号にします。ラベル（人間の運転正解）が無くても「世界の動き方そのもの」を学べるのが利点です。

2系統をつなぐインターフェースが設計の肝です。System2 → System1 へ何を渡すかにはいくつかの形があります。(a) 固定長の視覚文脈ベクトル1本を渡す、(b) 言語的な指示（「前方の工事を避けて左に寄れ」のようなテキストやメタアクション）を渡す、(c) 高レベルな経路点（waypoint）や運転意図（go straight / turn left など）を渡す、の3系統です。いずれにせよ、System2 がいくら深く理解しても、System1 に渡せる情報は限られているので、ここが情報のボトルネックであり、何を詰めるかが性能を左右します。

図にするとこうです。

```
[ System2 / 推論 (低頻度 ~1Hz) ]
  multi-cam → 重い視覚理解(VLM等) → 文脈/意図に圧縮 → 将来特徴を予測
        │  (context_vec / 言語指示 / waypoint を供給)  │ (自己教師あり損失で学習)
        ▼
[ System1 / 反射 (高頻度 ~10Hz) ]
  cameras + map + ego履歴 → BEV融合 → Temporal Memory → Planner → 軌跡
```

なぜこの設計なのか。重い視覚理解は1秒に1回でも実用上十分な場合が多く、その成果を、速く回る System1 が毎フレーム再利用すれば、計算予算を大幅に節約できます。脳が遅い熟考の結論を、速い反射が参照しながら動くのと同じ構造です。さらに、両者が非同期に動けること（System2 の最新結果がまだ来ていなくても System1 は直前の結果を使って動き続けられる）が、リアルタイム性を保つ上で本質的です。

## 手法の系譜と主要論文
出発点は認知心理学です。Daniel Kahneman は著書 "Thinking, Fast and Slow"（2011）で、人間の思考を System1（速い・自動的・直感的・努力を要さない）と System2（遅い・意識的・論理的・努力を要する）の2系統で説明しました。これは工学的手法ではなく概念枠ですが、「反射と熟考を分ける」アーキテクチャの命名と着想の源になりました。重要なのは、System2 が常に上位で正しいわけではなく、多くの日常判断は System1 が担い、System2 はコスト高なので必要なときだけ起動する、という非対称性です。

次に Yann LeCun の "A Path Towards Autonomous Machine Intelligence"（2022, ポジションペーパー）が、自律エージェントの設計図として世界モデル・行動価値・コンフィギュレータなどのモジュール分割を提案しました。特に JEPA（Joint-Embedding Predictive Architecture、生のピクセルを再現せず特徴空間で未来を予測する枠組み）を中心に据え、「ピクセル再構成は予測不能な細部に労力を割き非効率。特徴空間で予測すれば本質だけ学べる」と論じました。これが System2 を世界モデルとして実装する際の理論的支柱になっています。トレードオフは collapse（予測対象がすべて同じ定数ベクトルに潰れて学習が無意味になる現象）で、ターゲット側を固定または stop-gradient する安定化が必須です。

自動運転に話を移すと、まず Hu らの UniAD（CVPR 2023 Best Paper, arXiv:2212.10156）が、知覚・予測・計画をTransformerの query で連結した統合エンドツーエンドを提案し、計画指向で全タスクを束ねる潮流を作りました。これは厳密な System1/System2 分離ではありませんが、「計画という最終目的に向けて重い理解を組織化する」という発想の前段です。

そして大規模言語・視覚言語モデルの登場で、明示的な slow/fast 二層構成が現れます。Tian らの DriveVLM（2024, arXiv:2402.12289）は、VLM（画像を見て言語で推論できる大規模モデル）が低頻度でシーンを記述・推論し（slow）、軽量な従来型プランナが高頻度で軌跡を出す（fast）「DriveVLM-Dual」を提案しました。これは本トピックのほぼ教科書的な実装例です。Sima らの DriveLM（ECCV 2024, arXiv:2312.14150）は、運転を Graph Visual Question Answering（知覚→予測→計画を問答の連鎖で繋ぐ）として定式化し、推論側の構造を与えました。さらに Jiang らの Senna（2024, arXiv:2410.22313）は、VLM が「メタアクション」（go straight, turn left, decelerate 等の高レベル運転意図を言語で出す）を生成し、それを下流の end-to-end プランナが軌跡に翻訳する分業を明確化し、言語が System2→System1 のインターフェースとして機能することを示しました。Wayve の LINGO 系など各社の「reasoning + control」二段構成も、共通する動機は一貫して「言語的・記号的推論は重いので毎フレームは回せないが、価値があるので低頻度で取り入れたい」というものです。

## 論文の実験結果（定量データ）
DriveVLM（Tian et al., 2024）は nuScenes データセット（自動運転研究で広く使われる、6カメラ・LiDAR を備えた都市走行データ集）と独自の SUP-AD ベンチマークで評価されました。nuScenes の計画タスクでは、計画した軌跡が実際の人間の運転からどれだけずれるかを測る L2 誤差（メートル単位、小さいほど良い）と、衝突率（collision rate、計画軌跡が他物体とぶつかる割合、小さいほど良い）が主指標です。DriveVLM-Dual は、VLM による slow な推論を従来型の fast プランナと組み合わせることで、いずれか単独より計画精度を改善し、3秒先の平均 L2 誤差をベースラインから明確に下げ、特に難しいコーナーケース（工事中の道、特殊車両、複雑な交差点など）での性能向上を報告しています。論文の主張の核は、VLM 単独では推論は深いが実時間制御に間に合わず、軽量プランナ単独では難局面で破綻するため、両者の組み合わせが最良という点です。これはまさに「どちらか一方では足りず、頻度を分けた協調が効く」という System1/System2 分離の有効性を示すデータです。

Senna（Jiang et al., 2024）は、VLM が出すメタアクションで end-to-end プランナを条件付けることで、nuScenes の計画 L2 誤差と衝突率の両方を、メタアクション無しのベースラインから下げたことを報告しています。さらに大規模な無ラベル走行データでの事前学習を加えると、データ分布の異なる別ドメインへの汎化（cross-domain での planning 性能維持）が改善したと報告しており、これは「高レベル意図を言語で表す抽象化が、ドメイン差に頑健な橋渡しになる」ことを示唆します。

DriveLM（Sima et al., 2024）は、Graph VQA 形式の質問応答精度と、それを下流の計画に繋いだときの汎化性能を評価しました。質問の連鎖（あの物体は何か→次にどう動くか→だから自車はどうすべきか）を踏ませることで、未知のシーンや指示への汎化（zero-shot 的な状況対応）が、単純なエンドツーエンドより向上したことを報告しています。ここでの教訓は、推論を中間ステップに分解して明示すると、ブラックボックスな直結より頑健になりうる、という点です。

アブレーションの観点で重要なのは、DriveVLM-Dual から slow ブランチ（VLM 推論）を抜くと難局面の性能が落ち、fast ブランチを抜くと実時間性が失われる、という相補性です。これは「2系統それぞれが別の貢献をしている」ことの直接的証拠で、片方だけでは代替できないことを示します。一方で、低頻度の推論結果が古くなる影響（latency による劣化）も課題として認識されており、slow ブランチの更新頻度をどう設定するかが性能と計算量のトレードオフを決めます。

UniAD（Hu et al., 2023）は nuScenes で、計画の平均 L2 誤差を当時の代表的手法から約1mから0.3m台へと大幅に下げ、衝突率も低減したことを報告し、CVPR Best Paper を受賞しました。これは「最終目的（計画）に向けて重い理解タスクを組織化する」設計の有効性を示し、後続の slow/fast 分割の土台となりました。

## メリット・トレードオフ・限界
メリットは明快です。第一に、安全に直結する反射を軽く高頻度に保てるため応答遅延を抑えられます。第二に、重い視覚理解・推論を低頻度に回せるため、全体の計算量を抑えつつ表現力を確保できます。第三に、モジュールが分かれているため System2 を後から追加・差し替えしやすく、段階的開発に向きます。実際、多くのプロジェクトが「まず安全な反射層を完成させ、推論層を後から差し込む」順序で進めます。

トレードオフと限界も無視できません。最大の問題は2系統間のインターフェース設計です。固定長ベクトル1本にどれだけ意味を詰められるか（あるいは言語指示で表現を量子化することの粗さ）が上限を決め、ここが情報のボトルネックになります。次に、低頻度の System2 の判断が古くなるリスクです。1秒前の理解で動くと、急変するシーン（急な割り込み等）で判断が陳腐化します。さらに、System2 を世界モデルとして自己教師あり学習する場合、JEPA 系特有の collapse など不安定要因があり、安定化の工夫（ターゲット凍結、stop-gradient、非対称な予測器）が必須です。VLM を slow 側に使う場合は幻覚（hallucination、もっともらしいが誤った記述）という固有のリスクも加わります。

研究上の未解決課題としては、(1)「どの判断を反射に、どれを熟考に振るか」の線引きが自明でなく設計依存であること、(2) slow ブランチの最適な起動頻度を状況に応じて動的に変える方法（簡単な道では起動を減らし、難局面で増やす）が未確立であること、(3) 2系統の協調を end-to-end でどう学習するか（slow と fast を同時に学ぶと勾配が干渉する）、(4) 推論側の VLM の出力が幻覚を含むときに反射側がそれをどう無視・補正するか、などが挙げられます。

## 発展トピック・研究の最前線
最前線では、slow ブランチを大規模 VLM から知識蒸留して軽量化し、より高頻度で回せるようにする方向が活発です。また、slow ブランチの起動を「いつ熟考すべきか」の信頼度推定で動的にゲートする研究（不確実なときだけ熟考する、adaptive computation の運転版）も登場しています。世界モデル側では、生成的世界モデル（GAIA など、未来の運転映像そのものを生成するモデル）と JEPA 的な特徴空間予測のどちらが下流計画に効くかの比較が論点です。さらに、言語を中間表現に使う「chain-of-thought 運転」（推論過程を言語で展開してから行動を決める）が解釈性と性能の両面で注目されています。長期的には、System1 と System2 を別ネットワークではなく1つのモデル内で動的に切り替える（同じパラメータで速いパスと遅いパスを切り替える）統合アーキテクチャが理想とされ、これは大規模モデルの adaptive computation 研究と合流しつつあります。
