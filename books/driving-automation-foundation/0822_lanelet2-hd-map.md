# Lanelet2 HDマップ

## ひとことで言うと

Lanelet2(レーンレットツー)は、自動運転用の高精度地図(HD マップ)を表現するオープンな標準形式です。道路を画像(ドットの集まり)ではなく、「車線の中心線・左右の境界線・停止線・横断歩道・交通規制」といった線や点の集まり(ベクトル)で記述します。さらに「どの車線からどの車線へ進めるか」という接続性や、「ここは赤信号で止まれ」という交通ルールも一体で持ちます。学習に使うときは、この地図を自車中心の俯瞰図(BEV、真上から見た図)の画像に描き起こして(ラスタ化して)、カメラ画像と並べてモデルに入力するのが典型的な使い方です。

## 直感的な理解

カメラ映像だけで運転するのは、人間でも難しい場面があります。車線の白線が消えかかっている、交差点で自分の進める車線が分かりにくい、見通しの悪い場所で停止線の位置が見えない、といった状況です。

人間はこういうとき、頭の中の事前知識を使います。「ここは片側2車線で、次の信号に右折レーンがある」「この交差点は直進が優先」といった、目に見えていなくても知っている道路の構造とルールです。自動運転でも同じで、カメラから読み取れる即時の情報に加えて、あらかじめ用意された詳しい地図を持っていると、判断が格段に安定します。霧で前が見えなくても、地図があれば「この先カーブ」と分かるのと同じです。Lanelet2 は、この「頭の中の地図」を計算機が扱える形に標準化したものです。

## 基礎: 前提となる概念

用語を噛み砕きます。
- HD マップ(High Definition Map、高精度地図): カーナビの地図よりはるかに精密で、車線を1本ずつ、白線の種類、停止線や横断歩道の位置などをセンチメートル級で記述した地図。カーナビ地図が「道路」を1本の線で表すのに対し、HD マップは車線ごとに左右境界と中心線まで持つ。
- ベクトル表現: 道路を画像のドット(ラスタ)ではなく、点の座標と、それを結ぶ線で表す方法。「この点とこの点を結ぶ線が車線境界」のように記述する。拡大しても劣化せず、各要素に意味(これは停止線、これは中心線)というラベルを持たせられる。
- ラスタ表現: 道路をピクセルの格子(画像)で表す方法。畳み込みニューラルネット(CNN、画像を小さなフィルタでなぞって特徴を抽出するネット)でそのまま処理しやすいが、解像度で精度が頭打ちになる。
- BEV(Bird's Eye View、鳥瞰図): 真上から見下ろした俯瞰図。自車を中心に周囲の道路構造を平面で表す。地図とセンサー情報を統一座標で重ねるのに適する。
- ラスタ化(rasterization): ベクトル表現(意味つきの線)をラスタ表現(ピクセル画像)に変換する処理。線を太さと色を付けて画像に「焼く」イメージ。
- トポロジー(接続性): 「この車線の次にこの車線へ行ける」という、車線同士のつながりの構造。経路探索(ルーティング)に必須。幾何(どこに線があるか)だけでは「この車線からあの車線へ移れるか」は分からないので、別途グラフとして持つ。

Lanelet2 はオープンソースの自動運転スタックの標準地図形式としても採用され、研究と実装の両面で広く使われています。

## 仕組みを詳しく

Lanelet2 の地図は、いくつかの基本要素を階層的に組み合わせて道路を記述します。

1. ポイント(Point): 緯度経度または地図ローカル座標を持つ1点。すべての基礎。
2. リネストリング(LineString): ポイントを順につないだ折れ線。車線の境界線・中心線・停止線などを表す。`type`(road_border=道路の縁, line_thin=細線, stop_line=停止線 など)と `subtype`(solid=実線, dashed=破線 など)の属性を持つ。
3. レーンレット(Lanelet): 「1本の車線の区間」。左境界・右境界・中心線(centerline)を持つ。これが Lanelet2 という名前の由来で、地図の主役。
4. 規制要素(RegulatoryElement): 信号、停止線、速度制限、優先関係など、車線にかかる交通ルール。これにより「幾何」だけでなく「どう走ってよいか」を表せる。
5. エリア(Area): 駐車場や交差点内の領域など、面で表す要素。

Lanelet2 はこれらを3層(物理層=実在する線や面、関係層=規制やトポロジー、トポロジー層=ルーティング用のグラフ)で構造化します。データは XML 形式(`.osm` ファイル、OpenStreetMap 由来の書式)に格納されるのが一般的です。

学習に使うときは、この地図から自車中心の BEV 画像を描き起こします。典型的なラスタ化のステップ:

1. 地図を読み込む。同じ地図を何度もパースしないよう、結果をキャッシュするのが定石(`lru_cache` など。パースは重いので、フレームごとに読み直すと学習が極端に遅くなる)。
2. 自車位置(ego_x, ego_y)を中心に、半径数十m(例 60m)の範囲にある車線(レーンレット)を取り出す。この注目範囲を ROI(region of interest)と呼ぶ。
3. 取り出した線を、自車中心の座標へ変換して画像に描く。座標変換は「自車を原点に移し、自車の向き(yaw)が画面の上を向くように回転させ、ピクセル座標に直す」処理。これで「自分の進行方向が常に上」という見やすい BEV になる。
4. 要素の種類ごとに色と描き方を変える。道路の縁は緑、車線の区切りは下地+色付き細線、中心線は破線、停止線は赤、横断歩道は黄、というように意味を色で符号化する。色チャンネルにセマンティクス(意味)を載せることで、CNN が「赤い線=停止線」と学べる。
5. いったん高解像度(例 2048×2048)で描いてから出力サイズ(例 256×256)に縮小する。細い線をなめらかに表現するための工夫(アンチエイリアス効果。直接 256 に描くと細線が消えたりギザギザになる)。

座標変換の核となる計算(自車を原点・進行方向を上に揃える回転):

```python
rel = pts[:, :2] - ego                 # 自車を原点に
c, s = np.cos(-yaw), np.sin(-yaw)       # 自車の向きを上に揃える回転
x_rot = c * rel[:, 0] - s * rel[:, 1]
y_rot = s * rel[:, 0] + c * rel[:, 1]
col = cx - y_rot * scale                # 左(+Y)→画面左
row = cx - x_rot * scale                # 前(+X)→画面上
```

`-yaw` で回すのは、世界座標での自車の向き yaw を打ち消して進行方向を常に上向きに固定するためです。`scale` はメートルをピクセルに変換する係数(例: 60m を 256px に収めるなら scale ≈ 256/120)。

出力は (H, W, 3) の RGB 画像で、これをカメラ画像と同じ正規化に通して (3, H, W) のテンソルにします。モデルから見ると「もう1枚のカメラ画像」のように扱える、地図をラスタ化したタイルになります。CNN や ViT(Vision Transformer、画像をパッチに分けて Transformer で処理するモデル)といった画像系バックボーンの資産をそのまま使える点が、ラスタ化の最大の実利です。

## 手法の系譜と主要論文

- Lanelet2(Poggenhans, Pauls, Janosovits ほか; KIT/FZI, ITSC 2018, "Lanelet2: A high-definition map framework for the future of automated driving")。車線中心の HD マップフレームワークを提案。道路を Lanelet(車線区間)+ RegulatoryElement(規制)+ 物理/関係/トポロジーの階層で表す。動機は、自動運転には「幾何(どこに道があるか)」だけでなく「意味(どう走ってよいか=規制・優先)」と「接続性(どの車線からどの車線へ)」を一体で扱える地図が必要だったこと。従来の地図は幾何中心でルーティングや交通ルールの表現が弱かった。経路探索・交通ルール適用・予測まで1つの地図で扱え、オープンソースで広く採用されました。前身として、より単純な初代 lanelet 概念(ROS 由来)を発展させたものです。
- HD マップをラスタ化してモデル入力にする系譜の代表が ChauffeurNet(Bansal ほか, Waymo, RSS 2019)です。地図と動的物体を BEV ラスタにして畳み込みネットに入れる方式。地図の文脈を画像処理の枠組みで扱える利点がある一方、ラスタ化は解像度で精度が頭打ちになり、細い線や遠方情報が潰れやすい欠点があります。
- 対する潮流が、地図を画像化せずベクトルのまま使う手法です。VectorNet(Gao ほか, CVPR 2020)は折れ線をベクトル集合として階層グラフでエンコードし、LaneGCN(Liang ほか, ECCV 2020)は車線をノードとするグラフ畳み込み(隣接する車線ノード間で情報を伝播させる)で接続性を直接モデル化します。解像度に縛られず細部や接続性を保てる利点がありますが、実装が複雑で画像系バックボーンの資産が使えません。
- さらに近年は、HD マップ作成・維持コストそのものを下げるオンライン地図構築(車載センサーから走行中に局所地図を推定する)が活発で、HDMapNet(Li ほか, ICRA 2022)や MapTR(Liao ほか, ICLR 2023)が代表です。これは「事前地図に依存しすぎる」という Lanelet2 系の限界への応答という位置づけです。

## 論文の実験結果(定量データ)

- Lanelet2(ITSC 2018)は主に枠組みとライブラリの提案で、ベンチマーク数値より「経路探索・規制適用・予測を単一地図で扱える」という機能網羅性を示しました。実用評価としては、ルーティング(出発点から目的地までの車線レベル経路探索)が交通ルールを尊重して計算でき、実車スタック上で走行に供されたことが報告されています。
- ラスタ vs. ベクトルの定量比較は VectorNet(CVPR 2020)が明確に示しました。Argoverse 予測ベンチマークで、ベクトル符号化はラスタ化 CNN ベースラインに対し、計算量(FLOPs・メモリ)を大幅に削減しつつ予測誤差(ADE/FDE、メートル、小さいほど良い)を同等以上に下げたと報告しています。ADE/FDE は予測軌道が正解からどれだけずれたかの距離で、ここでの改善は「同じ精度を少ない計算で出せる」ことを意味します。アブレーションでは、車線グラフ(地図のトポロジー)を入力から除くと予測誤差が悪化することが示され、地図の接続性情報の寄与が定量化されています。
- LaneGCN(ECCV 2020)は Argoverse で、車線グラフ畳み込み(LaneConv)とエージェント-車線間の注意機構を導入し、当時の最先端を更新しました(報告では Argoverse motion forecasting でトップ級の精度)。アブレーションで、車線トポロジーの伝播(LaneConv)を抜くと精度が落ちることが示され、ベクトル地図の接続性をモデルに通す価値が裏づけられました。
- オンライン地図側では、MapTR(ICLR 2023)が nuScenes で、車線などのベクトル地図要素をエンドツーエンドで推定し、従来手法比で地図要素の検出精度(mAP、平均適合率。検出の正確さを 0〜100% で測る指標)を大きく上げつつリアルタイムに近い推論速度(報告では数十 FPS 級)を達成したと報告しています。これは「事前 HD マップが無い/古い」状況への現実的な代替を定量的に示した結果です。

## メリット・トレードオフ・限界

メリット
- ベクトル表現なので意味(車線・停止線・規制)を持たせられ、拡大しても劣化しない。
- 車線の接続性・交通ルールまで一体で表せ、ルーティングや予測にそのまま使える。
- オープン標準なので、学習したモデルを実車スタックへ接続するときの地図と整合しやすい。
- ラスタ化すれば画像系バックボーンでそのまま扱え、カメラ特徴と素直に融合できる(実装が簡単)。

トレードオフ・限界
- HD マップの作成・更新コストが高い。現実の道路工事や標示変更に地図が追従していないと、かえって誤った前提を与え、地図を信じたモデルが危険な判断をしうる(消えた車線を「ある」と思い込む等)。
- ラスタ化は解像度で精度が頭打ち。高解像度で描いて縮小する工夫をしても、細線や遠方は潰れやすい。ベクトルのまま使う手法はこれを避けられるが実装が重い。
- 自車のグローバル位置(地図上のどこにいるか)が正確に分かっている前提。位置推定(localization)が数十cmずれると地図と現実がずれ、入力が誤情報になる。
- 地図を持つデータセットに限られ、地図なしデータでは使えない。これがオンライン地図構築の研究動機にもなっている、分野の未解決課題です。

## 発展トピック・研究の最前線

研究の流れは「事前 HD マップへの依存を減らす」方向に強く向かっています。HDMapNet・MapTR・VectorMapNet などのオンライン地図構築は、車載カメラ/LiDAR から走行中にベクトル地図を推定し、古い地図や未整備地域でも動けるようにします。また地図表現自体を学習する潮流として、ラスタとベクトルの中間にあたるクエリベースの地図トークン(地図要素を Transformer のクエリ=問い合わせベクトルで表す)が広がっています。予測・プランニングと地図を一体で解く end-to-end モデル(地図を中間表現として暗黙に持つ)も活発で、Lanelet2 のような明示的ベクトル地図は、学習の教師・評価基準・安全層(モデル出力が地図上のルールに反していないか検査する層)としての役割へと位置づけが移りつつあります。
