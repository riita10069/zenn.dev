# 自車座標系へのラスタ化

## ひとことで言うと
地図(道路や走行経路)を、自分の車を原点に置き・進行方向が常に画像の「上」を向くように座標変換し、それをピクセルの絵(ラスタ画像)に焼き込む処理です。これにより、どんな場面でも「自車は絵の真ん中で前を向いている」という統一された見え方になり、機械学習モデルが地図を効率よく学習できるようになります。予測・運転モデルへ地図を入力する際の、最も基本的で核になる前処理です。

## 直感的な理解
カーナビを思い出してください。多くのナビは「進行方向が常に画面の上」になるよう、曲がるたびに地図がくるくる回ります。北を上に固定したモードより、こちらの方が「次に右へ曲がる」「この先で左に分岐」が直感的に分かります。これは人間にとってだけでなく、機械にとっても同じ理屈で都合がよいのです。

仮に「北を上」に固定したまま学習させると、北へ直進する場面では経路が画面の上へ伸び、南へ直進する場面では下へ伸びます。人間は「どちらも直進」と理解できますが、モデルにとっては絵がまるで上下逆さまの別物に見え、「直進とは何か」を学ぶのに余計な労力がかかります。専門的に言えば、モデルが学習すべき写像(入力→出力の対応関係)が、向きの数だけ「水増し」されてしまい、同じことを学ぶのに多くのデータと時間を要するのです。さらに、世界の絶対座標(緯度経度)をそのまま使うと、ある街で学んだ感覚が別の街では通用しにくくなります。これらの無駄を消すために「自車中心・進行方向上向き」へ正規化するのが、本トピックの座標変換です。これは画像認識で言う「データ拡張(augmentation)で不変性を学ばせる」のと逆の発想で、最初から不変な座標系に直して問題を簡単にしてしまう、という考え方です。

## 基礎: 前提となる概念

緯度・経度。地球上の位置を「角度」で表す座標です。たとえば北緯35.68度・東経139.77度のように表します。緯度は赤道からの南北の角度、経度はグリニッジからの東西の角度です。注意点は、これは長さ(メートル)ではなく角度だということ。経度1度の実距離は赤道で約111km ですが、高緯度ほど経線が縮むので短くなります。

ヘディング(heading)。自車が向いている方位角です。本トピックでは「ラジアン(角度を弧の長さで測る単位、360度=2π)で、北を0として反時計回りに増える」という規約をよく使います。heading=0 なら真北、π/2(90度)なら真西を向いている、という具合です。実装によっては「東を0として反時計回り」(数学の標準)や「北を0として時計回り」(コンパス方位)を使うこともあり、規約の違いは符号や 90度のずれとして現れるため、必ずどの規約かを確認するのが事故防止になります。

相対座標と絶対座標。絶対座標は地球全体で一意な座標(緯度経度や都市座標)。相対座標は「自車から見て東に何m、北に何m」のように、自車を基準に測り直した座標です。場所が変わっても意味が変わらないのが相対座標の利点です。専門的にはこれを「自車座標系(ego frame)」と呼びます。

回転行列。平面上の点を原点まわりに角度θ回す操作を、掛け算1回で表す2×2の行列です。点 (x, y) を角度θ回した先 (x', y') は次式になります。
```
x' = x·cosθ − y·sinθ
y' = x·sinθ + y·cosθ
```
この式の各記号は、x,y が回す前の座標、x',y' が回した後の座標、θ が回転角です。回転行列は直交行列(逆回転は転置で得られる)で、長さを変えないという良い性質を持つため、距離の歪みを生みません。

ラスタ化(rasterization)。点や線・面で表されたベクトル情報を、ピクセルの格子に塗って画像にする操作です。解像度(1ピクセルが何メートルか)を決めると、メートル座標が整数のピクセル位置に対応づきます。

同次変換(homogeneous transform)。平行移動と回転を1つの行列で表す手法です。3×3の行列に座標 (x, y, 1) を掛けることで、「中心を自車へ移す → 向きをそろえる → ピクセルへ写す」という一連の操作を、行列の積として1回でまとめられます。実装では各点に個別に式を当てるより、この行列を点群全体に一括適用する方が高速です。

## 仕組みを詳しく

処理は大きく3ステップです。

ステップ1: 緯度経度を「自車から見たメートル」に変換する。自車の周り数百メートル程度の狭い範囲なら、地球の丸みを無視して平面とみなしてよい近似が使えます。これを等距円筒近似(equirectangular approximation)、より一般には局所接平面近似(local tangent plane)と呼びます。次式で、緯度経度を「東向き何メートル(x)・北向き何メートル(y)」に直します。

```
deg_to_m = EARTH_RADIUS_M * π / 180        (緯度経度1度あたりのメートル数)
x_east  = (経度 − 自車経度) * cos(自車緯度) * deg_to_m
y_north = (緯度 − 自車緯度) * deg_to_m
```

ここで EARTH_RADIUS_M は地球半径(約6,378,137メートル)です。肝は経度側に `cos(自車緯度)` を掛けている点。前述のとおり経度1度の実距離は緯度が高いほど短くなるので、cos を掛けて縮めて補正しています。

具体例。自車が(緯度35.0, 経度139.0)、ある道路点が(緯度35.001, 経度139.002)のとき、
- y_north = (35.001 − 35.0) × 6,378,137 × π/180 ≈ +111.3 m(北へ約111m)
- x_east = (139.002 − 139.0) × cos(35°) × 6,378,137 × π/180 ≈ +182.4 m(東へ約182m)

こうして自車を原点 (0,0) とした「東(x)・北(y)」のメートル平面ができます。なお、もともと地図が「UTM(横メルカトル図法に基づく平面直角座標)」や「都市固有のメートル座標」で与えられている場合は、ステップ1の緯度経度変換は不要で、単に「自車位置を引く(平行移動)」だけで済みます。nuScenes のように地図がメートル座標で提供されるデータセットでは後者の経路を取ります([nuScenes Map API](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0454_nuscenes-map-api))。

ステップ2: 進行方向が「上」を向くように回転させる。ステップ1の平面はまだ「北が上」です。これを、自車のヘディングが画像の上(+y)を向くように回します。使うのは `−heading` だけの回転です。

```
cos_h = cos(−heading) ,  sin_h = sin(−heading)
x_local = x_east * cos_h − y_north * sin_h
y_local = x_east * sin_h + y_north * cos_h
```

なぜマイナスか。自車が北から heading だけ回った向きを向いているとき、世界の方を逆向き(−heading)に回せば、自車前方がちょうど真上に来ます。heading=0(北向き)なら回転ゼロで北が上のまま、という整合性も保てます。直観的には「自分が右を向いたとき、景色は左へ回って見える」のと同じで、自車の回転と世界の見かけの回転は符号が逆になる、ということです。

before/after で見ると、
- before(north-up): 北東に進んでいると、経路は絵の中で斜め右上へ伸びる。
- after(ego-up): どの方角に進んでいても、進行方向の経路は常に絵の「上」へ伸びる。

ステップ3: メートル平面をピクセルに離散化する(ラスタ化)。回転後の (x_local, y_local) を、固定の表示範囲とピクセル格子に焼き込みます。たとえば表示範囲を ±50m(つまり100m四方)、出力を256×256ピクセルにすると、解像度は 100m ÷ 256 ≈ 0.39m/px になります。メートルからピクセルへの対応は次のように書けます。

```
col =  (x_local + 50) / 0.39      (左端 −50m が 列0、右端 +50m が 列256)
row =  (50 − y_local) / 0.39      (上端 +50m が 行0、下端 −50m が 行256)
```

ここで row 側は「上が +y」になるよう符号を反転している点に注意してください。画像座標は「行番号が下に向かって増える」のに対し、メートル座標は「y が上に向かって増える」ので、両者をそろえるために反転が要ります。この符号ミスは可視化で「地図が上下逆さま」になる典型バグです。また、縦横のメートルスケールを必ず等しく(aspect equal)しておくことが重要で、これを怠ると距離が縦横で歪み、まっすぐな道が斜めに潰れて見えてしまいます。

塗る内容は用途次第です。典型的には、道路網を灰色の線で、走行予定の経路を別の色の線で、車線や横断歩道などの地図要素をさらに別の色や別チャンネルで描きます(配色やチャンネルの設計は[地図チャンネル符号化方式](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0419_map-channel-encoding-scheme)の主題)。最終的に得られる画像は、形状 `(3, H, W)`(RGB3枚)や `(K, H, W)`(K個の意味レイヤ)のテンソルになり、カメラ画像と同じ要領で画像処理バックボーン(ImageNetなどで事前学習された特徴抽出器)へ流せます。

マップマッチングという補助処理もよく使われます。GPS は誤差で道路から数メートル外れることがあるので、各GPS点を最寄りの道路網上の点に吸着(スナップ)させ、点と点を最短経路でつないで滑らかな経路に直します。失敗した場合は「道路網と生のGPS点だけを描く」フォールバックにするのが一般的です。

なぜオフライン前処理にするのか。道路網を外部の地図サービス(OpenStreetMap など)から取得するのには通信と時間がかかります。学習のたびに毎回取得・描画すると、データ読み込みが学習のボトルネックになります。そこで多くの実装では、全クリップ分のタイルを事前に描いてキャッシュ(保存)しておき、学習時はキャッシュ画像を読むだけにします。これにより GPU が画像描画待ちで遊ぶ(GPU 使用率が下がる)のを防げます。

## 手法の系譜と主要論文

「自車中心・進行方向上向きにラスタ化する」のは、特定の1論文の発明というより、ラスタ系の予測・運転モデルで広く共有される定石です。系譜を追います。

ChauffeurNet(Bansal, Krishnan, Ogale; Waymo, 2018/RSS 2019, "ChauffeurNet: Learning to Drive by Imitating the Best and Synthesizing the Worst", arXiv 1812.03079)。自車中心のトップダウン(真上視点)画像を入力に、人間の運転を模倣して学習させました。地図・他車・信号・自車軌跡を複数の白黒チャンネルに分け、自車を画像の決まった位置・決まった向きに固定して描きます。場面によらず一貫した入力にすることで学習を安定させるという、本トピックの動機そのものを体現した代表例です。自車を画像内の固定位置(たとえば下から1/4、横は中央)に置くという慣習も、ここで広く知られるようになりました。

MultiPath(Chai, Sapp, Bansal, Anguelov; Waymo, CoRL 2019, "MultiPath: Multiple Probabilistic Anchor Trajectory Hypotheses for Behavior Prediction", arXiv 1910.05449)。同様に自車中心のラスタBEVを入力にし、将来の軌跡を複数候補(マルチモーダル)として確率つきで予測します。あらかじめ用意した代表的な軌跡(アンカー)からの差分を学ぶことで、右折・直進・左折のような分岐する未来を1つに潰さず扱えるようにしました。「未来は複数ありうる」ことを正面から扱った初期の重要例です。

CoverNet(Phan-Minh, Grigore, Boulton, Beijbom, Wolff; nuTonomy, CVPR 2020, "CoverNet: Multimodal Behavior Prediction using Trajectory Sets", arXiv 1911.10298)。やはり自車中心ラスタを入力に、将来軌跡を「あらかじめ用意した軌跡集合の分類問題」として解きます。回帰ではなく分類にすることで、物理的にありえない軌跡を出さないようにした点が特徴です。軌跡集合の細かさ(本数)を変えると、表現できる軌跡の解像度と分類の難しさがトレードオフになる、という設計思想を示しました。

これらを下支えしたのが nuScenes / Argoverse の地図API([nuScenes Map API](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0454_nuscenes-map-api), [Argoverse HD Map](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0438_argoverse-hd-map))です。いずれも「自車の位置・向きを与えると、自車中心のマスク/タイルを切り出す」機能を持ち、内部で本トピックと同じ座標変換(中心移動 → heading回転 → 解像度離散化)を行っています。つまり API のマスク生成関数は、本トピックの3ステップをカプセル化したものだと理解できます。

## 論文の実験結果(定量データ)

軌跡予測の評価指標を押さえると、本前処理の効果が読めます。代表指標は ADE と FDE です。ADE(Average Displacement Error、平均変位誤差)は、予測軌跡と正解軌跡を各時刻で比べた距離の平均(メートル)。FDE(Final Displacement Error、最終変位誤差)は、予測の終端時刻だけで見た距離です。いずれも小さいほど良い指標です。マルチモーダル予測ではしばしば minADE_k(k本の候補のうち最良のものの ADE)を使い、「正解に最も近い候補がどれだけ当たったか」を測ります。なぜ「最良の候補」で測るかというと、未来は複数ありうるので、出した候補のどれか1本が当たっていれば良し、とする評価が現実的だからです。これに加え、当てるべき確率の付け方を見る指標(miss rate、外した割合)も併用されます。

MultiPath や CoverNet は nuScenes と自社データで、自車中心ラスタBEVを入力にした手法が、座標列をそのまま並べただけの単純なベースラインより minADE/FDE を明確に下げると報告しました。報告によれば、nuScenes 予測ベンチマーク(過去2秒から未来6秒を予測)で、これらラスタ系は minADE_5 で数メートル規模、minADE_1 でより大きい誤差を達成し、当時の上位を占めました。重要なのは「自車中心へ正規化したこと」がこの改善の前提になっている点です。アブレーション的に言えば、入力を北固定や絶対座標に戻すと、同じネットワークでも学習が遅くなり誤差が悪化することが経験的に知られています。これは「向きと位置をそろえる正規化」が、特徴量学習の難度を大きく下げているためです。地図ラスタを入力から外す(エージェントの過去軌跡だけにする)アブレーションでも誤差が悪化し、「地図文脈が将来軌跡の制約として効く」ことが確認されています。

ChauffeurNet は、地図・信号・他車を含む自車中心ラスタを入力にすることで、生のカメラだけより複雑な交差点や信号交差点での挙動が安定したと報告しました。とくに、わざと危険な状況(車線からのはみ出しなど)を合成して学習に混ぜる工夫と組み合わせると、閉ループ(モデルが運転し続ける評価)での車線逸脱や衝突が減ったとされています。これも入力が「いつも同じ見え方」に正規化されているからこそ、稀な状況の学習信号がぶれずに効いた、と解釈できます。

## メリット・トレードオフ・限界

メリット。第一に、場所(緯度経度の絶対値)に依存しない相対座標になり、学習したモデルが別の土地でも機能しやすくなります。第二に、進行方向を上に固定することで向きの違いを正規化し、学習が安定・効率化します。第三に、メートルスケールで縦横を等しくするので距離の歪みがありません。第四に、出力をカメラ画像と同じ `(3,H,W)` テンソルにできるので、既存の画像バックボーンへそのまま流せます。

トレードオフと限界。ラスタ化である以上、解像度(画像サイズと表示範囲で決まる)より細かい位置差は表現できません。たとえば 0.4m/px なら、それ未満の車線の微妙なずれは1ピクセルに丸められて消えます。等距円筒近似は半径数百m程度でのみ妥当で、広範囲では歪みが出ます(数km四方になると緯度方向の補正だけでは足りず、正しくは UTM などの図法へ移すべきです)。heading の精度に強く依存し、GPS軌跡の末端区間から方位を推定するような場合は停車中や低速時に方位が乱れがちで、IMU(慣性計測装置)など正確な情報源があればそちらが望ましいです。道路網の取得が遅いためオフライン前処理が事実上必須で、その場で地図がない場所には対応しにくいのも限界です。マップマッチングは範囲外や道路分断で失敗しうり、失敗時の生GPSのみ描画はモデルにとって情報の薄い入力になります。

研究上の課題としては、ラスタの解像度・範囲・配色という設計パラメータが性能に効くのに、その最適値はデータセット依存で、原理的な指針が乏しいことが挙げられます。広い範囲(高速道路の遠方)と高い解像度(交差点の細かい車線)は同じタイル内では両立しにくく、マルチスケールで複数解像度を併用する工夫も生まれていますが、決定打はありません。この「ラスタ化の情報損失」を根本から避けるべく、点と線の座標で地図を扱うベクトル表現([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))へ進む流れが生まれています。

## 発展トピック・研究の最前線

ラスタBEVは実装が単純で強力ですが、解像度に縛られる弱点があります。これを乗り越える方向として、地図や軌道を折れ線・曲線の座標で表すベクトル表現、それを集合予測で出すDETR型デコーダ([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))、さらにカメラ特徴を学習可能な投影でBEVへ持ち上げる手法(Lift-Splat 系、[ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction))が広がっています。後者は「手書きの座標変換」ではなく「学習で最適な投影」を獲得する点で本トピックの一歩先です。とはいえ、入力を自車中心・進行方向上向きに正規化するという発想自体は、ベクトル系・学習投影系でも(座標系の取り方として)受け継がれており、本トピックは今も多くの手法の土台になっています。実際、Transformer ベースの予測・地図構築モデルでも「自車を原点とした座標で位置エンコーディングを与える」のが標準で、座標系の正規化という発想はラスタ・ベクトルの別を問わず生き続けています。さらに近年は、エージェントごとにそれぞれの座標系を取り直して相互作用を不変に扱う「query-centric」な設計も提案され、自車中心化の発想がマルチエージェントへ一般化しています。
