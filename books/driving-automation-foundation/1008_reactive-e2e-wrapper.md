# 薄いラッパー（トップレベルモデルの責務分離）

## ひとことで言うと
深層学習モデルの最上位クラスを、実際の計算をほとんど自分でやらない「束ねるだけの薄い入れ物（ラッパー）」にする設計です。トップは複数のサブシステム（反射的に動く制御系統と、低頻度で推論する系統など）を並べて保持し、入力を適切なサブシステムへ受け渡すだけにします。中身を空っぽに近くすることで、「誰が何の責務を持つか」をコード構造そのもので明示でき、新しい系統を後から最小の変更で差し込めます。

## 直感的な理解
大きなモデルのコードでは、最上位クラスにあれもこれも詰め込みがちです。すると1つのクラスがカメラ処理もマップ融合も軌跡予測も時系列メモリも全部抱え、数百〜数千行に膨らみ、どこが何の役割か分からなくなります。これを「責務が混ざった神クラス（God class）」と呼びます。神クラスは、一部を直したいだけでも全体を理解しなければならず、テストも全部まとめてしか書けず、機能追加のたびに肥大化していきます。

会社組織に例えると、社長が経理も営業も製造も全部自分でやっている状態が神クラスです。これでは社長が何の専門家なのか分からず、新しい事業部を増やすのも一苦労です。健全な組織は、社長は「事業部を統括する」だけで、各事業部がそれぞれの専門を担います。薄いラッパー設計は、モデルのトップクラスを「社長」にして、計算の実務を各サブモジュール（事業部）に委ねる発想です。

二系統アーキテクチャ（高頻度の反射系統と低頻度の推論系統に分ける設計、別ノート参照）を採るなら、この方針をコードで素直に表すには「トップは2つの系統を並べて持つだけ」「各系統の中身はそれぞれのクラスに閉じ込める」のが自然です。トップクラスを見れば、システム全体が何系統で構成されているかが一目で分かる、という状態を作れます。

## 基礎: 前提となる概念
まず「クラスとコンストラクタとforward」を押さえます。深層学習フレームワーク（PyTorch など）では、モデルは `nn.Module` を継承したクラスとして書きます。`__init__`（コンストラクタ、初期化メソッド）で使う部品（層やサブモデル）を生成して保持し、`forward`（順伝播メソッド）で入力をそれらの部品に通して出力を作ります。学習時はこの forward の出力から損失を計算し、誤差逆伝播でパラメータを更新します。

次に「コンポジション（合成）」です。あるクラスが別のクラスのインスタンスを部品として内部に持つことをコンポジションと呼びます。PyTorch のモデルは入れ子のコンポジションで構成され、大きなモデルは小さなモジュールの木構造になります。薄いラッパーはこの木構造の根（ルート）にあたり、根が直接計算するのではなく、子（サブシステム）に委譲します。

「委譲（delegation）」も重要です。自分宛に来た仕事を、適切な担当に丸投げして結果をそのまま返すことです。薄いラッパーの forward は典型的に委譲だけを行います。

最後に設計原則を2つ。「単一責任の原則（Single Responsibility Principle, SRP）」は、1つのクラスは1つの責務だけを持つべき、という指針です。「開放閉鎖の原則（Open/Closed Principle）」は、拡張に対して開き（新機能を足せる）、修正に対して閉じる（既存コードを変えなくてよい）べき、という指針です。薄いラッパーは両者を実践する手段です。

## 仕組みを詳しく
薄いラッパーがやることは典型的に2つだけです。

1つ目、コンストラクタで各サブシステムを生成して保持します。例えば反射系統を1つ作って `self.reactive = ReactiveModel(...)` のように持ちます。このとき重要なのは、ラッパー自身は新しい計算層（畳み込みや Attention など）を一切持たないことです。受け取ったハイパーパラメータ（backbone の種類、視点数、埋め込み次元、プランナの種類など）は、ほぼそのままサブシステムへ横流しします。

2つ目、forward で受け取った入力をそのままサブシステムに渡し、その戻り値をそのまま返します。擬似コードで書くと:

```python
class TopLevelModel(nn.Module):
    def __init__(self, backbone, num_views, embed_dim, planner_mode, **kw):
        super().__init__()
        self.reactive = ReactiveModel(backbone, num_views, embed_dim,
                                      planner_mode, **kw)
        # 将来: self.reasoning = ReasoningModel(...) をここに並べる

    def forward(self, camera, map_input, context_vec, ego_history,
                mode="infer", target=None, **kw):
        out = self.reactive(camera, map_input, context_vec, ego_history,
                            mode=mode, trajectory_target=target, **kw)
        return out
```

入出力の形（shape）の例。`B` はバッチサイズ、`V` は視点（カメラ）数です。

```
入力:
  camera      : [B, V, 3, H, W]
  map_input   : [B, 3, H_map, W_map]
  context_vec : [B, D_ctx]  または [B, T, D_ctx]   推論系統からの文脈
  ego_history : [B, T, D_ego]
戻り値（mode依存）:
  mode="train": 損失スカラー（プランナ内部で計算）
  mode="infer": 軌跡 [B, T_future * num_signals]
```

ここで「戻り値の意味が mode で変わる」設計はよく使われます。学習ループ側は「どのプランナを使っているか」を知らなくてよく、`mode="train"` を渡せば常に損失が返る、という統一された窓口（uniform interface）になります。プランナが GRU でも Flow Matching でも、損失計算はプランナ内部に隠蔽されるため、学習ループは無改変で済みます。

将来の拡張点を構造として宣言しておくのも、薄いラッパーの重要な役割です。実装より先に「ここに推論系統が入る」「その役割は文脈ベクトルを作って反射側に渡し、将来特徴を自己教師あり損失に使うこと」をコメントやプレースホルダで示しておくと、設計意図が後から読む人に伝わります。

before/after で効果を示すと、もしラッパーに中身を全部書いていたら（before）、新しい系統を足すときにトップクラスを大改修する必要があります。薄いラッパーにしておけば（after）、`self.reasoning = ReasoningModel(...)` を1行足し、forward で呼ぶだけで済みます。

```
before: トップに全部 → 系統追加 = トップを大改修 + テスト全書き直し
after : トップは束ねるだけ → 系統追加 = 1行追加 + forwardで呼ぶだけ
```

## 手法の系譜と主要論文
これはソフトウェア設計の一般パターンであり、特定の機械学習論文に紐づくものではありません。背景にある原則と系譜を辿ります。

起点は GoF（Gang of Four）の "Design Patterns: Elements of Reusable Object-Oriented Software"（Gamma, Helm, Johnson, Vlissides, 1994）です。ここで定義された Facade パターンは、複雑なサブシステム群に対してシンプルな統一窓口（facade）を1つ提供する設計です。薄いラッパーは、複数の系統への統一窓口になる Facade に近い位置づけです。提案動機は「利用側（学習ループ・推論スクリプト）に内部構造を意識させない」ことです。

次に Robert C. Martin らが整理した単一責任の原則（SRP）と開放閉鎖の原則（OCP）("Agile Software Development", 2002 ほか）が、なぜラッパーを薄くするのかを正当化します。ラッパーは「系統を束ねる」だけ、各サブシステムは「自分の計算をする」だけ、と責務を分けるのが SRP の実践であり、系統を足してもラッパー以外を変えないのが OCP の実践です。

深層学習の文脈では、PyTorch の `nn.Module` のコンポジショナル設計そのものがこの思想の土台です。大きなモデルを小さなモジュールの木として組み、根を薄くする習慣は、torchvision・Hugging Face Transformers・各種検出フレームワークに広く見られます。例えば Transformers ライブラリの `XxxForCausalLM` のようなトップクラスは、本体（base model）とタスク用ヘッドを束ねるだけの薄いラッパーになっており、本トピックの典型例です。

## 論文の実験結果（定量データ）
設計パターンには論文のベンチマーク数値のような定量評価は通常付きませんが、ソフトウェア工学の実証研究から間接的な根拠を挙げられます。神クラス（God class）や大きすぎるクラスは「コードの臭い（code smell）」として知られ、複数の実証研究（例えば Olbrich らの変更頻度・欠陥率の分析）で、こうしたアンチパターンを含むクラスは変更頻度が高く欠陥が混入しやすいことが報告されています。つまり責務を分けて各クラスを小さく保つことは、保守コストと不具合率の低減という測定可能な効果に結びつきます。

機械学習プロジェクトに即して言えば、薄いラッパー化の効果は「新系統を追加する際の差分行数」「既存テストの再利用率」で測れます。トップに全部書く設計では系統追加が数百行規模の改修になりうるのに対し、薄いラッパーでは追加行数が数行〜十数行に収まり、既存サブシステムのテストはそのまま再利用できます。アブレーション的に言えば、ラッパーを薄くする要素（委譲のみ・新層を持たない・統一窓口）を1つでも崩すと、その分だけ結合度が上がり拡張コストが増える、という関係です。

定量的な傍証としては、ソフトウェアメトリクスの古典である結合度（coupling、モジュール間の依存の強さ）と凝集度（cohesion、1モジュール内の責務のまとまり）の研究があります。Chidamber と Kemerer の CK メトリクス（1994, IEEE TSE）は CBO（Coupling Between Objects、あるクラスが依存する他クラス数）や RFC（Response For a Class、そのクラスに対する呼び出しで起動しうるメソッド総数）を定義し、後続の複数の実証研究でこれらの値が高いクラスほど欠陥密度が高いことが繰り返し報告されています。薄いラッパーは、トップクラスの責務を「束ねるだけ」に絞ることで、トップの凝集度を高く保ち、サブシステム実装への依存を委譲1点に局所化します。これは CK メトリクス上は「トップクラスの RFC を小さく保ち、変更の波及を抑える」効果として説明できます。

## メリット・トレードオフ・限界
メリットは、トップクラスを読むだけで全体の系統構成が分かり設計意図が明確になること、新しい系統を後から最小の変更で差し込める拡張性、各サブシステム単体でのテスト・差し替えのしやすさ、そして学習ループや推論スクリプトが1つの窓口だけ見ればよく内部の系統数が増えても呼び出し側を変えずに済むこと、です。

トレードオフと限界もあります。第一に、系統が1つしかない段階では「無駄な1段」に見えます。Facade の恩恵は束ねる対象が複数になって初めて顕在化するため、初期は冗長に感じられます。第二に、forward の引数がサブシステムにそのまま横流しされる場合、引数定義が二重管理になりやすく、片方を変えたらもう片方も合わせる手間が生じます（これは `**kwargs` の素通しで緩和できますが、今度は型チェックが効かなくなります）。第三に、戻り値の意味を mode で変える設計は柔軟ですが、呼び出し側が mode を取り違えると分かりにくいバグになります。

研究・開発上の課題としては、ラッパーが「薄すぎて何もしない層」になるか「便利機能を足しすぎて再び神クラス化する」かの境界をどこに引くか、という設計判断が残ります。また、複数系統を協調させる学習（反射系統と推論系統を同時に学ぶ）では、ラッパーが損失の集約や勾配の調整という新たな責務を持ち始め、薄さを保てなくなることがあります。このとき、集約ロジックを別の「損失コンビネータ」クラスに切り出して薄さを維持するか、ラッパーに持たせるかが判断点になります。

## 発展トピック・研究の最前線
拡張点を構造で宣言する考え方は、近年の大規模モデルでは「アダプタ」「LoRA モジュール」「タスクヘッドの動的登録」といった形でさらに発展しています。トップクラスを薄く保ったまま、実行時に系統やヘッドを差し替える設計（プラグイン的構成）は、マルチタスク学習やマルチモーダル統合で標準になりつつあります。これは [Component Registry / Plugin 設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0948_component-registry-pattern) とも密接に関わり、「何を束ねるか」を文字列キーで動的に決める Registry と、「束ねるだけ」の薄いラッパーは、拡張性を支える両輪です。さらに、複数系統を非同期・別頻度で動かす実システム（reactive 系統は10Hz、reasoning 系統は1Hz）では、ラッパーが単なる関数呼び出しでなくスケジューラ的な役割を持つことになり、ソフトウェア工学とリアルタイムシステム設計の交点が研究対象になります。
