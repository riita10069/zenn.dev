# SwinV2の4段階特徴マップ形状

## ひとことで言うと
Swin Transformer V2(SwinV2、画像から特徴を取り出す高性能なニューラルネット)は、画像を一気に1つの特徴へ潰すのではなく、解像度を段階的に下げながら4段階(stage 0〜3)の特徴マップ(画像のような数値配列)を出します。256×256 の画像を入れると、stage 0 は細かく広い特徴、stage 3 は粗く小さいが意味の濃い特徴になります。この「4段階の具体的な形(縦・横・チャンネル数)」を正しく知っておくと、後段の部品(視点融合や検出ヘッド)の設計や、テスト用の代用 backbone を作るときに必須の知識になります。

## 直感的な理解
地図を読むときを思い浮かべてください。番地を探すには細かい縮尺の地図が要りますが、「どの方角に向かうか」を決めるには大まかな広域地図が便利です。片方だけでは道に迷います。ニューラルネットも同じで、浅い層は細かい模様(エッジ・テクスチャ)を、深い層は大きな意味(これは車だ、人だ)を捉えます。

自動運転のように「遠くの小さな歩行者」と「目の前の大きなトラック」を同時に扱いたいタスクでは、1つの解像度の特徴だけでは足りません。小さい物体は細かい解像度の特徴に、大きい物体は粗い解像度の特徴によく写ります。だから複数解像度の特徴を後段にまとめて渡す必要があり、SwinV2 が出す4段階の階層特徴はこの要求にちょうど合います。ただし、複数段階を後段に流すには「各段階がどんな形で出るか」を1次元の狂いもなく知っておく必要があります。形を1つ取り違えると、エラーで止まるか、最悪エラーすら出ずに静かに間違った計算が進みます。

## 基礎: 前提となる概念
- 特徴マップの形 `[B, C, H, W]`: B はバッチ(同時処理サンプル数)、C はチャンネル数(特徴の種類の数)、H は縦、W は横です。
- channels-first / channels-last: 同じ特徴マップでも、メモリ上の次元の並びに2流派があります。`[B, C, H, W]` を channels-first、`[B, H, W, C]` を channels-last と呼びます。Transformer 系は内部で channels-last(`[B, H, W, C]`、各空間位置を1つのトークンとして並べる)で計算することが多く、CNN の畳み込み層は channels-first(`[B, C, H, W]`)を期待します。両者をつなぐときはどこかで `permute`(次元の並べ替え)が要ります。
- トークン(token): Transformer が扱う処理単位。画像の場合、小さなパッチ1枚を1トークンとして扱います。64×64 の特徴マップなら 4096 トークンです。
- ダウンサンプリング(downsampling): 解像度を間引いて小さくすること。情報を圧縮しつつ受容野(1つの特徴が見ている入力範囲)を広げます。
- 受容野(receptive field): 出力の1点が入力のどれだけ広い範囲を見ているか。深い層ほど広く、これが「大きな意味」を捉える根拠になります。
- 注意機構(attention): 画像の各部分が互いを見比べながら、どこに着目すべきかを学習で決める仕組み。Transformer の中核です。各トークンが他の全トークンとの関連度を計算します。

昔の CNN も深い層ほど解像度を下げて意味を濃くしていました。素の Vision Transformer(ViT, Dosovitskiy et al. ICLR 2021)は、最初に画像を 16×16 のパッチに切って以降は解像度を変えず「単一解像度・全トークン全対全の注意」で処理する設計でした。Swin Transformer は、これを注意機構を保ちつつ CNN のように階層的(hierarchical、段階的に解像度を下げる)に作り直した点が新しいところです。

## 仕組みを詳しく
SwinV2-tiny に256×256の画像を入れたときの4段階の形は次のとおりです(channels-first 表記、複数カメラを扱う文脈では先頭が `B*V`)。

```
入力画像:   [B*V,   3, 256, 256]   (RGB 3チャンネル)
  ↓ Patch Embedding(4x4 パッチに分割し各パッチを1トークン化)
Stage 0:    [B*V,  96,  64,  64]   解像度は 1/4、チャンネル 96
  ↓ Patch Merging(2x2 統合)
Stage 1:    [B*V, 192,  32,  32]   解像度は 1/8、チャンネル 192
  ↓ Patch Merging
Stage 2:    [B*V, 384,  16,  16]   解像度は 1/16、チャンネル 384
  ↓ Patch Merging
Stage 3:    [B*V, 768,   8,   8]   解像度は 1/32、チャンネル 768
```

読み解きのポイント:

- 解像度(H, W)は段ごとに半分ずつ縮む(64 → 32 → 16 → 8)。Swin ではこれを Patch Merging(隣り合う 2×2 のトークンをまとめて1つにし、チャンネルを 4 倍に連結してから線形変換で 2 倍に圧縮する)で実現します。空間は 1/4(縦横それぞれ 1/2)、チャンネルは 2 倍になる操作です。
- チャンネル(C)は段ごとに倍々で増える(96 → 192 → 384 → 768)。解像度を半分にする代わりに表現の豊かさ(チャンネル)を倍にして、情報量(空間サイズ×チャンネル)のバランスを取る設計思想です。
- 最初に解像度が一気に1/4になるのは、入力直後の Patch Embedding(画像を 4×4 のパッチに区切り、各パッチを1つのトークンにする)が原因です。だから stage 0 がいきなり 256→64 になります。

一言で言えば「縦横は半分ずつ、チャンネルは倍ずつ」という規則的な階層構造です。「stage0-3 が [64,64,96] 〜 [8,8,768]」という表現は、この4段階を channels-last 風(H, W, C)で並べたときの両端を指します。

なぜ window(窓)で注意を計算するのか。画像全体で注意を計算すると、計算量がトークン数 N の2乗(O(N²))で爆発します。256×256 を1段目の解像度64×64で考えても 4096 トークンあり、その全対全の注意(4096×4096 の関連度行列)は重すぎます。Swin は画像を小さな窓(SwinV2 では 8×8 など)に区切り、窓の中だけで注意を計算します。窓のサイズを M、トークン数を N とすると、計算量は窓内 M² × 窓数(N/M²)= O(N·M²) となり、画像全体に対して線形(O(N))に収まります。ただし窓を固定すると窓の境界をまたぐ関係が切れるので、層ごとに窓の位置を半分(M/2)ずらす Shifted Window で境界情報を混ぜます。

### "tiny" の意味とサイズによる数値の違い
SwinV2-tiny の "tiny" はモデルの大きさの呼称です。tiny / small / base / large の順に大きく(チャンネルや層が多く)なります。stage 0 の基準チャンネル(embed dim)が tiny=96、small=96、base=128、large=192 で、最終 stage 3 のチャンネルはその 8 倍(tiny=768、base=1024、large=1536)になります。tiny は軽量さ重視の選択肢で、エッジや車載のように計算資源が限られる環境で好まれます。パラメータ数は Swin-T で約2,800万、Swin-B で約8,800万、Swin-L で約1.97億です。

### なぜ形を「知って」いる必要があるのか
後段の視点融合や特徴ピラミッド処理は「4段階・各段ごとに決まったチャンネル数の特徴リスト」を前提に組まれます。そのため、テスト用の代用 backbone([モックbackboneによる軽量テスト](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1127_mock-backbone-stub))はこの4段階の形をそっくり真似て返す必要があります。

```python
s0 = self.stage0(x)   # [B*V, 96, 64, 64]
s1 = self.stage1(s0)  # [B*V, 192, 32, 32]
s2 = self.stage2(s1)  # [B*V, 384, 16, 16]
s3 = self.stage3(s2)  # [B*V, 768,  8,  8]
return [s0, s1, s2, s3]
```

形を1つでも外すと、後段は形不一致でエラーになるか、最悪「無言で間違った計算」をします。特に channels-first と channels-last を取り違えると、`[B, 64, 64, 96]`(channels-last)を `[B, C=64, H=64, W=96]`(channels-first)と誤解しても要素数が同じため `reshape` が通ってしまい、エラーすら出ずチャンネルと空間を混同したまま学習が進み、精度が出ない原因が掴めなくなります。だから「正しい形を正確に把握する」ことが、軽量テストと正しい結線の前提になります。多くの実装ライブラリでは、Swin は内部 channels-last のため、CNN 系後段に渡す直前で `permute(0, 3, 1, 2)` を入れて channels-first に直します。

## 手法の系譜と主要論文
階層的な多解像度特徴という発想自体は CNN 時代からあります。Lin et al. "Feature Pyramid Networks for Object Detection"(FPN, CVPR 2017, arXiv:1612.03144)は、CNN が自然に出す多解像度特徴を上位(粗い・意味が濃い)から下位(細かい)へトップダウンに融合し、小さい物体の検出精度を底上げしました。SwinV2 の4段階出力は、まさにこの FPN 系の後段に渡しやすい形をしています。

Dosovitskiy et al. "An Image is Worth 16x16 Words"(ViT, ICLR 2021, arXiv:2010.11929)は、画像を 16×16 のパッチ列に変換して純 Transformer で分類し、大規模事前学習(JFT-300M)のもとで CNN を上回ることを示しました。ただし単一解像度・全対全注意のため計算量が画素数の2乗で、検出やセグメンテーションのような密な予測タスクや高解像度入力には不向きでした。

Liu et al. "Swin Transformer: Hierarchical Vision Transformer using Shifted Windows"(Ze Liu ら, Microsoft Research Asia, ICCV 2021, arXiv:2103.14030)が、この弱点を解いた原典です。核は2点。(1)窓内に限った注意で計算量を線形に抑える。(2)層ごとに窓をずらす Shifted Window で境界をまたぐ情報も混ぜる。これにより CNN のような階層的多解像度特徴を出しつつ、Transformer の表現力を得て、検出・セグメンテーションの汎用 backbone として広く採用されました。

Liu et al. "Swin Transformer V2: Scaling Up Capacity and Resolution"(CVPR 2022, arXiv:2111.09883)が改良版です。V1 を高解像度・大規模で安定して学習させるための工夫が中心で、(1)注意を scaled cosine attention に変えて大モデルでの注意値の暴れ(一部のトークンに注意が過集中する不安定)を抑える、(2)位置の符号化を log-spaced continuous position bias にして、低解像度で学習したモデルを高解像度へ転用しても破綻しにくくする、(3)学習を安定させる残差後正規化(res-post-norm)を導入。高解像度カメラを扱う自動運転では、この高解像度耐性が魅力です。

比較対象として、Liu et al. "A ConvNet for the 2020s"(ConvNeXt, CVPR 2022, arXiv:2201.03545)は、同じ4段階・解像度1/4〜1/32という階層構造を純粋な CNN だけで実現し、Swin に匹敵する精度を出せると示しました。stage の解像度比とチャンネル比を Swin に揃えているため、SwinV2 と ConvNeXt を入れ替え可能にできます(両者が同じ4段階の特徴形状を持つため)。

## 論文の実験結果(定量データ)
指標の意味から説明します。

- ImageNet top-1 accuracy: ImageNet-1K(1000クラスの画像分類データセット)で、モデルが最も確からしいと答えた1クラスが正解と一致した割合。画像認識の事実上の標準指標です。
- COCO box AP / mask AP: 物体検出・インスタンスセグメンテーションの精度。Average Precision で、検出枠やマスクの正確さを IoU(重なり率)のしきい値を変えながら平均した値。高いほど良い。
- ADE20K mIoU: 意味的セグメンテーション(画素ごとのクラス分類)の精度。クラスごとの IoU の平均(mean IoU)。

主な数値(報告による):

- Swin V1(ICCV 2021): Swin-T(tiny)で ImageNet-1K top-1 約81.3%、Swin-S 約83.0%、Swin-B(base)で約83.5%。当時の同規模 CNN(ResNet-50 の約76%台、ResNet-101 の約78%)や DeiT を明確に上回りました。COCO 物体検出では、Swin-L backbone を使った検出器が box AP 約58.7 / mask AP 約51.1 と当時の最高水準を更新。ADE20K セグメンテーションでも mIoU 約53.5(Swin-L, UperNet)と大きく押し上げました。
- Swin V2(CVPR 2022): 約30億パラメータの SwinV2-G まで安定して学習でき、ImageNet-V2 で top-1 約84.0%(当時世界最高)、COCO test-dev で box AP 約63.1、ADE20K で mIoU 約59.9、Kinetics-400 動画分類で top-1 約86.8% などを達成。低解像度(192×192 など)で事前学習し高解像度(最大 1536×1536)へ転移しても性能が崩れにくいことを実証しました。
- ConvNeXt(CVPR 2022): ConvNeXt-T で ImageNet-1K top-1 約82.1%、ConvNeXt-B で約83.8%(IN-1K 学習)〜約85.8%(IN-22K 事前学習)。同規模の Swin と「ほぼ互角〜やや上」で、純 CNN でも Transformer に並べることを示しました。

アブレーション(どの要素を抜くとどうなるか):

- Swin V1 の論文では、Shifted Window を外して固定窓にすると ImageNet top-1 が約1.1ポイント、COCO box AP が約2.3ポイント、ADE20K mIoU が約2.8ポイント低下したと報告。境界をまたぐ情報の重要性を示します。相対位置バイアスを外した場合も同様に精度が落ちます。
- Swin V2 の論文では、scaled cosine attention と log-spaced continuous position bias を外すと、特に学習解像度と推論解像度が大きく異なる高解像度転移の場面で精度が顕著に劣化。これらが「高解像度耐性」の主因であることを裏づけます。

これらの差は数値としては数ポイントですが、検出・セグメンテーションでは「小さい歩行者を見落とすか否か」に直結するため、自動運転の安全性の観点で意味を持ちます。1ポイントの mIoU 改善が、特定クラス(歩行者・自転車)の見落とし率を実用上無視できない幅で減らすことがあります。

## メリット・トレードオフ・限界
メリット:
- 階層的な多解像度特徴により、遠近・大小さまざまな物体を同時に捉えやすい。FPN 系の後段にそのまま渡せる。
- 4段階の形が規則的(縦横は半分、チャンネルは倍)なので、後段の設計やテストの代用 backbone 化がしやすい。
- ConvNeXt と同じ形状なので、backbone を差し替えても後段を変えずに済む。実験で backbone を比較しやすい。

トレードオフと限界:
- channels-last と channels-first の並び替えが必要で、扱いを誤ると無言のバグになる。
- 4段階すべてを後段に渡すとメモリ消費が増える。特に stage 0(64×64×96 = 約39万要素/画像)は要素数が大きく、高解像度入力ではここがメモリのボトルネックになりやすい。マルチカメラで `B*V` をまとめると、stage 0 のテンソルが GPU メモリを最も圧迫する。
- 段階数の違う backbone(たとえば stem を含め5段階を返す ResNet 系の特徴抽出設定)とは形が合わず、後段や代用 mock の前提が崩れる。
- 未解決の課題として、窓サイズと入力解像度の相性がある。学習時と推論時で解像度が大きく変わると窓の数や境界の扱いが変わり、V2 の連続位置バイアスでも完全には吸収しきれない場合がある。窓サイズを推論時に変える研究もあるが、再学習なしの一般解は確立していない。

## 発展トピック・研究の最前線
- ViT-Adapter / 階層化 ViT: 階層構造を持たない素の Vision Transformer に、後付けで多解像度特徴を取り出す機構を足す研究(Chen et al. "Vision Transformer Adapter for Dense Predictions", ICLR 2023)。Swin の階層性の利点を別経路で得ようとする流れ。
- BEV 認識との接続: 多解像度の画像特徴を BEV(俯瞰)空間へ持ち上げる手法(BEVFormer, Lift-Splat-Shoot など)では、どの段階の特徴を BEV 変換に使うかが精度と計算量を左右する。SwinV2 の4段階のうち中間段(stage 1〜2)がよく使われる。
- 効率化系 backbone: MobileViT、EfficientFormer、FastViT など、車載・エッジ向けに同様の階層特徴を軽量に出す系統。tiny でも重い場合の代替候補として研究が進む。
- 基盤モデル backbone: DINOv2 や SAM の画像エンコーダなど、大規模自己教師あり学習で得た汎用特徴を backbone とする流れ。これらは段階数や出力形が Swin と異なることが多く、後段や mock の設計を都度合わせ直す必要がある。
