# カメラキャリブレーション抽出（BEV用）

## ひとことで言うと

カメラで撮った2D画像の上のある点が、実世界（車を中心とした3D空間）のどこに対応するかを計算するには、カメラの「内部の性質（焦点距離など）」と「車のどこに・どの向きで付いているか」という2種類の情報が必要です。これらをキャリブレーション情報から読み取り、3D座標を画素座標へ変換する `[3,4]` の行列を視点（カメラ）ごとに作る処理を、ここでは「カメラキャリブレーション抽出」と呼びます。鳥瞰図（BEV、Bird's-Eye-View）を複数カメラから正確に組み立てるために不可欠な前処理です。

## 直感的な理解

複数のカメラで車のまわりを撮っているとします。前方・斜め前・側方・後方など、それぞれが別の方向を向いています。これらを上空から見下ろした一枚の地図（BEV）に貼り合わせたい。ところが、各カメラの画像はあくまで「そのカメラから見た2D投影」であり、画像のどの画素が地面のどこに対応するかは、画像だけ見ても分かりません。

人間なら「このカメラは右斜め前を向いていて、地面はこのあたり」と直感で補正できますが、それは私たちが「カメラがどこに・どの向きで付いているか」を暗黙に知っているからです。コンピュータに同じことをさせるには、その「カメラの取り付け情報」と「カメラのレンズの拡大率」を数値で与え、3D空間の点と画素を結ぶ式を立てる必要があります。これがキャリブレーションです。キャリブレーションが間違っていると、複数カメラのつなぎ目で同じ物体が二重に見えたり、地面が斜めにずれたりして、BEVが幾何的にデタラメになります。

## 基礎: 前提となる概念

カメラがどう3D点を2D画像に写すかは「ピンホールカメラモデル」という単純な数学モデルで表せます。これを構成する2種類のパラメータを押さえます。

内部パラメータ（intrinsic、イントリンシック）はカメラ単体の性質です。焦点距離 `fx, fy`（レンズがどれだけ拡大して写すか、画素単位）と主点 `cx, cy`（光軸が画像面と交わる点の画素座標、ほぼ画像中心）からなる `3×3` 行列 `K` で表します。これは「カメラ座標系の3D点（カメラから見た右・下・前方）を、その画像の画素座標へ写す」変換です。

```
    [ fx   0   cx ]
K = [  0  fy   cy ]
    [  0   0    1 ]
```

外部パラメータ（extrinsic、エクストリンシック）はカメラが車のどこに、どの向きで取り付けられているかを表す `4×4` の剛体変換（回転 `R` と並進 `t` を組み合わせたもの）です。「車を中心とした座標系（ego frame、エゴフレーム）の3D点」と「カメラ座標系の3D点」を相互に変換します。回転 `R` は 3×3、並進 `t` は 3次元ベクトルで、合わせて `[[R, t], [0, 0, 0, 1]]` という 4×4 になります。

同次座標（homogeneous coordinates）も前提知識です。3D点 `(x, y, z)` を `(x, y, z, 1)` という4次元で表すと、回転と並進をまとめて1つの行列の掛け算で扱えます。投影の結果 `(u', v', w)` が出たら、最後の `w` で割る操作（透視除算、perspective division）で実際の画素座標 `(u, v) = (u'/w, v'/w)` が得られ、`w` そのものが深度（カメラからの距離）になります。

座標規約（convention）にも注意が要ります。軸の取り方はデータセットごとに違い、例えば RDF（Right-Down-Forward、右・下・前）と呼ばれる規約や、x前方・y左・z上の規約などがあります。さらに extrinsic が `ego→cam`（エゴ座標からカメラ座標への変換）として与えられるか、その逆の `cam→ego` として与えられるかもデータセットで異なります。ここを取り違えると、向きが反転して全カメラが一律にずれます。

## 仕組みを詳しく

intrinsic `K` と extrinsic `T` を掛け合わせると、「車中心の3D点を画素座標へ写す」1本の変換 `P`（射影行列、projection matrix）が得られます。形は `[3,4]` です。

```
P = K @ T_ego→cam[:3, :]      （K は 3×3、T_ego→cam の上3行は 3×4、結果は 3×4）
```

なぜ `[3,4]` なのか。3D点を同次座標 `[x, y, z, 1]ᵀ`（4次元）で表し、`P`（3×4）を掛けると `[u', v', w]ᵀ`（3次元）になります。透視除算で `[u, v] = [u'/w, v'/w]` が画素座標、`w` が深度です。

実際の抽出は次のステップで行うのが典型です。

ステップ1: キャリブレーション情報を読む。視点ごとに `K`（3×3 intrinsic）と extrinsic（4×4）を取得します。多くのデータセットはこれを YAML や JSON、あるいは専用のキャリブレーションファイルに格納しています。extrinsic が `cam→ego` で与えられているか `ego→cam` かをここで確認します。

ステップ2: intrinsic を画像変換に合わせてスケールする。これが最大の落とし穴です。`K` は撮影時の元解像度（例 1920×1080）の画素座標を前提にしていますが、ニューラルネットに入れる前に画像を例えば 256×256 にリサイズ・クロップします。すると焦点距離も主点も画素単位で変わります。リサイズ倍率 `s` を掛けるなら `fx, fy, cx, cy` をすべて `s` 倍し、中央クロップでオフセット `(ox, oy)` だけ切り取るなら主点を `cx -= ox; cy -= oy` と補正します。

```
Resize(倍率 s)   : fx *= s; fy *= s; cx *= s; cy *= s
CenterCrop(ox,oy): cx -= ox; cy -= oy
```

この補正を忘れると、リサイズ後の画像に対して古い座標系の `K` を使うことになり、投影が数十画素ずれます。BEVのように複数カメラを重ねる用途では、このずれがそのままつなぎ目のゴーストになります。

ステップ3: extrinsic の向きをそろえる。BEVは ego frame の3D点を基準にサンプリングするので、必要なのは `ego→cam`。手元の extrinsic が `cam→ego` なら逆行列を取ります。

```python
T_ego_to_cam = np.linalg.inv(extrinsic_cam_to_ego)   # (4,4)
P = K_scaled @ T_ego_to_cam[:3, :]                    # (3,4)
```

ステップ4: 全視点ぶん積み上げて `(V, 3, 4)` のテンソルにする（`V` はカメラ数）。これが BEV 融合モジュールへ渡す射影パラメータになります。

BEV側での使われ方（数値の流れ）を追います。鳥瞰図のグリッド（例えば前方120m・左右60mの範囲を 450×300 マスに区切る）の各マスに対応する3D基準点を作り、同次座標にして各カメラの `P` を掛けます。透視除算で画素座標 `[u, v]` を得て、画像サイズで割って `[0,1]` に正規化し、`grid_sample`（指定座標で特徴マップを双線形補間して取り出す操作）で各カメラの特徴をBEVグリッドへサンプリングします。`P` が恒等行列のようなデタラメだと、この投影が無意味になり、幾何的に正しいBEVが作れません。実装によってはこの場合に、投影の代わりに学習可能なパラメータでカメラ特徴をBEVへ写す「擬似投影」に切り替えることもありますが、それは幾何的整合を諦めた近似です。

数値例で確認します。前方20m・右0m・地面高(z=0) の点を考えます。ego frame で `[20, 0, 0, 1]ᵀ`。前方カメラの `P` を掛けると、例えば `[u', v', w] = [1280·0.5+640·25, 720·0+360·25, 25]` のような値が出て（数字は説明用）、`w=25`（25m先）で割ると画素 `[u, v] = [640, 360]`（画像中央付近）になる、といった具合です。これが正しく出れば、その地面の点が前方カメラの中央に写っているという物理と一致します。

## 手法の系譜と主要論文

カメラの内部パラメータを推定する古典が、Zhang のチェッカーボード法（Z. Zhang, "A Flexible New Technique for Camera Calibration", ICCV 1999 および PAMI 2000）です。既知の格子パターン（チェッカーボード）を複数枚撮り、その対応点から `K` とレンズ歪み係数を最小二乗で推定します。データセットに付属するキャリブレーション情報は、この種の手法の出力であることがほとんどです。歪み補正（distortion correction）はこのモデルの拡張で、広角レンズの樽型歪みなどを多項式で補正します。

多視点カメラからBEVを作る現代的な系譜は、LSS（Lift, Splat, Shoot; J. Philion & S. Fidler, ECCV 2020, arXiv:2008.05711）から本格化します。各カメラの画素ごとに深度の確率分布を予測し、intrinsic/extrinsic を使って画素を3D点群へ「持ち上げ（lift）」、BEVグリッドへ「撒く（splat）」、最後にタスクを解く（shoot）という3段構成です。正しいキャリブレーション行列が投影の前提であることを明確に位置づけました。トレードオフは、深度予測の不確かさがそのままBEVのにじみになる点です。

BEVFormer（Z. Li et al., ECCV 2022, arXiv:2203.17270）は深度を明示予測せず、BEVクエリ（学習可能なBEVグリッドのベクトル）が3D基準点を各カメラ画像へ投影し、その位置の特徴を集める「空間クロスアテンション（spatial cross-attention）」を提案しました。投影には intrinsic@extrinsic が必須で、calibration が正確なほど、複数カメラに写る同じ実点が正しく重なります。多くの簡易BEV融合実装はこの考え方の縮小版です。

Cam2BEV（L. Reiher et al., ITSC 2020, arXiv:2005.04078）は、路面が平面だと仮定してホモグラフィ（2Dの射影変換）でカメラ画像をBEVへ変形します。平面仮定のぶん calibration の使い方は単純化されますが、人や車のような立体物が引き伸ばされて崩れる弱点があり、立体を扱うには完全な3D投影（intrinsic と extrinsic の両方）が要ることを裏から示しています。

## 論文の実験結果（定量データ）

LSS は nuScenes（6カメラ、1000シーンの自動運転データセット）で、car セグメンテーションの IoU（Intersection over Union、予測領域と正解領域の重なり率。1に近いほど良い）を評価し、当時の単眼ベース手法を上回りました。報告では、複数カメラを正しい投影で統合することで、単一カメラからは見えない死角を補完できることが定量的に示されています。

BEVFormer は nuScenes 検出ベンチマークで、NDS（nuScenes Detection Score、検出の総合指標。0〜1で大きいほど良い）でおよそ 0.56、mAP（mean Average Precision、平均適合率）でおよそ 0.42 を報告し、当時のカメラのみ手法で最高水準でした。BEV map segmentation でも高い IoU を示しています。重要なアブレーション（ある要素を抜くと性能がどう変わるかの検証）として、時系列のBEV情報を統合する temporal self-attention を外すと検出性能が明確に落ちることを示し、また空間クロスアテンションが正しい投影に依存することを構造として裏づけました。

キャリブレーション誤差の影響を直接調べた研究も複数あり、報告によると extrinsic の角度がおよそ1〜2度ずれるだけで、遠方（数十m先）の投影位置は数十cmから1m規模でずれます。BEVのように複数カメラを重ねる用途では、この誤差がつなぎ目で物体の二重像となって現れ、下流タスク（検出・セグ・軌道予測）の精度を目に見えて下げます。逆に言えば、calibration の質はモデルの工夫と同じくらい性能に効く要素です。

## メリット・トレードオフ・限界

正しい射影行列があれば、複数カメラの映像を幾何的に整合したBEVへ正確に貼り合わせられます。BEVFormer 系のように深度を別途学習しなくても、幾何だけで投影でき、この場合 calibration の質がそのまま精度に直結します。一方で限界も多くあります。

calibration が間違っている、あるいは古いと、複数カメラのつなぎ目で物体が二重・ずれて見え、学習がかえって有害になります。リサイズ・クロップに合わせた intrinsic 補正は忘れやすく、見落とすと全体が一律にずれます。外部パラメータは車体のたわみ・温度変化・経年でわずかにずれるため、固定値で扱うと長期では誤差が蓄積します。これに対し、走行中に extrinsic を逐次推定するオンラインキャリブレーション（online/self-calibration）の研究があります。

さらに、平面ホモグラフィのような単純化は立体物を崩し、LSS 系の深度予測は不確かさを持ち込みます。座標規約（RDF か否か、ego→cam か cam→ego か）の取り違えは現場で頻発する失敗で、可視化で1点の投影を必ず目視確認するのが実務上の鉄則です。未解決の課題としては、安価なセンサや振動環境での頑健な自動キャリブレーション、カメラとLiDARなど異種センサ間の高精度な相互キャリブレーションが挙げられます。

## 発展トピック・研究の最前線

近年は calibration への依存を減らす方向と、より正確に使う方向の両方が進んでいます。前者では、明示的な射影行列を使わずデータから視点関係を学ぶ手法（BEVFormer の派生や、Transformer に幾何を暗黙に学ばせる試み）が増えています。後者では、複数フレームにわたる整合性から extrinsic を継続補正するオンラインキャリブレーション、LiDARと画像の対応からキャリブレーションを自動化する手法が活発です。BEV表現そのものを3Dの占有グリッド（occupancy）へ拡張する流れ（occupancy prediction）でも、正確な投影は依然として土台です。研究を始めるなら、まず1台のカメラで「ある3D点が画像のどこに写るか」を `P` で計算し、実画像と重ねて目視確認するところから入ると、抽象的な行列が一気に具体的になります。
