# 推論レイテンシ計測の方法論

## ひとことで言うと

「このニューラルネットワークは 1 回の推論に何ミリ秒かかるのか」を正しく測るのは、初心者が思うよりずっと難しい問題です。GPU は CPU から見ると非同期(裏で勝手に進む)に動くので、素朴にストップウォッチを当てると、本当の計算時間ではなく嘘の値が出ます。正しく測るには、`torch.cuda.synchronize()` のような命令で計測のタイミングをそろえ、最初の数回は捨て(ウォームアップ)、平均だけでなく p99(99 パーセンタイル)やジッタ(ばらつき)まで見ます。さらに「モデル単体の時間」と「前処理込みのパイプライン時間」を分けて報告します。このノートは、信頼できるレイテンシ計測の作法を、なぜそうするのかの理由とともにゼロから積み上げて説明します。

## 直感的な理解

レイテンシ(latency)とは「入力を入れてから答えが返るまでの時間」、日本語で言えば遅延です。リアルタイムで動くシステム、たとえばロボットや自動運転車では、この遅延が安全に直結します。1 秒間に 10 回判断したい(10Hz と言います)なら、1 回の推論は最低でも 100 ミリ秒(ms)以内に終わらなければ、次のフレームが来てしまって間に合いません。

ここで多くの人がやってしまう失敗が、こういうコードです。

```python
import time
t0 = time.time()
output = model(input)   # GPU 推論したつもり
t1 = time.time()
print((t1 - t0) * 1000, "ms")   # 例えば 3ms と表示される
```

「3ms? このモデルめちゃくちゃ速いじゃないか」と喜んではいけません。これは罠です。実際に GPU が計算を終えるには 80ms かかっているのに、3ms と表示されてしまうことがあります。なぜそんなことが起きるのか。それを理解するのがこのトピックの出発点です。身近なたとえで言えば、レストランで「注文を伝えた瞬間(3ms)」を「料理ができた時刻」だと勘違いしているようなものです。注文を伝えるのは一瞬ですが、料理が出てくるまでには時間がかかります。

## 基礎: 前提となる概念

### CPU と GPU の関係、そして「非同期実行」

深層学習の計算は GPU(Graphics Processing Unit、画像・並列計算が得意な専用チップ)上で行われますが、プログラム全体を制御しているのは CPU(Central Processing Unit、汎用の頭脳)です。CPU は「この行列を掛けて」「この畳み込みをして」という指示を一つずつ GPU に送ります。

ここが肝心なところです。CPU が `model(input)` を呼ぶと、その計算の指示は GPU のキュー(待ち行列)に積まれるだけで、CPU はその完了を待たずにすぐ次の行に進みます。これを非同期実行(asynchronous execution)と呼びます。「指示を出したら、終わるのを待たずに次へ行く」方式です。

なぜこんな設計なのか。CPU と GPU を並行して働かせ、全体のスループット(処理量)を上げるためです。CPU が次々と指示を積み(これをカーネルローンチ、kernel launch と呼びます)、GPU はそれを順番に消化していく。両者が同時に動くので無駄がありません。しかしこの効率の良さが、計測のときには罠になります。`time.time()` で測っているのは CPU 側の時刻なので、「指示をキューに積むのにかかった時間(数ミリ秒)」しか測れず、GPU が実際に計算を終える時間は見えないのです。

### CUDA カーネルとウォームアップ

GPU 上で動く小さなプログラムを CUDA カーネル(CUDA kernel)と呼びます。深層学習フレームワークは、初めて呼ばれた演算について、その場で最適な GPU 機械語を生成したり(JIT コンパイル、Just-In-Time compilation)、必要なメモリ領域を確保したり(初回はメモリアロケータがキャッシュを持っていないので OS から確保する)、cuDNN(NVIDIA の深層学習ライブラリ)が複数のアルゴリズム候補を試して一番速いものを選んだり(`torch.backends.cudnn.benchmark=True` でのアルゴリズム探索)します。これらは初回だけ発生する一度きりのコストです。

その結果、最初の 1 回の推論は異常に遅くなります。報告例では、あるモデルの連続推論で 1 フレーム目だけ約 596ms かかり、それ以降は平均 約 81.6ms で安定した、つまり初回が約 7 倍遅かったという事例があります。この初回値をそのまま平均に混ぜると、数値が大きく歪みます。だから本計測の前に「捨て走り」を数回行って、定常状態(steady state、安定した速度)にしてから測ります。これがウォームアップ(warmup)です。GPU のクロックが省電力状態から最大に上がるまでにも時間がかかるため、ウォームアップはアルゴリズム探索だけでなく「GPU を暖機する」意味でも必要です。

### パーセンタイルとジッタ

統計の言葉を 2 つ押さえます。パーセンタイル(percentile)は「全体を小さい順に並べたとき、下から何%の位置にあるか」を表します。p99(99 パーセンタイル)は「100 回測ったうち 99 回はこの値以下」という意味で、ほぼ最悪に近いケースを表します。ジッタ(jitter)はばらつきの大きさで、標準偏差 σ(シグマ)で表すことが多いです。σ が小さいほど毎回ほぼ同じ時間で終わる安定したシステム、大きいほど「速いときと遅いときの差が大きい」不安定なシステムです。

なぜテール(分布のすそ、p99 など)を見るのか。リアルタイムシステムでは「平均で間に合う」では不十分で、「最悪でも締切を守れるか」が安全の条件だからです。1000 フレームのうち 10 フレームだけ締切を破るシステムは、平均は良くても、その 10 回のどれかが交差点での歩行者検知だったら事故になりかねません。

## 仕組みを詳しく

### torch.cuda.synchronize() で時間を区切る

`torch.cuda.synchronize()` は「GPU のキューに積まれた仕事が全部終わるまで、CPU をその行で待たせる」命令です。これを計測の前後に置くことで、GPU の本当の計算時間を切り出せます。

```python
import torch, time

# --- ウォームアップ ---
for _ in range(10):
    _ = model(input)
torch.cuda.synchronize()          # ウォームアップの残りを全部終わらせる

# --- 本計測 ---
latencies = []
for _ in range(100):
    torch.cuda.synchronize()      # (1) それまでの GPU 仕事を全部終わらせる
    t0 = time.perf_counter()      # (2) 計測開始
    output = model(input)         # (3) 推論(指示をキューに積む)
    torch.cuda.synchronize()      # (4) 推論が GPU 上で本当に終わるまで待つ
    t1 = time.perf_counter()      # (5) 計測終了
    latencies.append((t1 - t0) * 1000)   # ms
```

ポイントは「前後の両方」で synchronize することです。(1) で前に同期しないと、直前の処理の残り(前のループの後始末など)が今回の計測時間に混ざります。(4) で後ろに同期しないと、まだ GPU 上で計算が終わっていないのに (5) で止めてしまい、不当に短い値が出ます。前後の両側を同期して初めて「モデルの純粋な GPU 計算時間」が分離できます。

なお `time.time()` ではなく `time.perf_counter()` を使うのは、後者が単調増加でより高分解能な計測用クロックだからです。`time.time()` はシステム時刻なので NTP 同期などで巻き戻ることがあり、計測には不向きです。さらに正確を期すなら、CPU を介さず GPU 上に時刻イベントを刻む `torch.cuda.Event` を使う方法もあります。

```python
start = torch.cuda.Event(enable_timing=True)
end   = torch.cuda.Event(enable_timing=True)
start.record()
output = model(input)
end.record()
torch.cuda.synchronize()
latency_ms = start.elapsed_time(end)   # GPU 上で測った経過時間 (ms)
```

`Event` 方式は CPU 側のオーバーヘッド(Python のループや時刻取得の遅延)を計測区間に含めないので、純粋な GPU 実行時間としてはより厳密です。一方 `synchronize` + `perf_counter` 方式は、CPU の指示発行コストも含めた「実際にユーザーが体感するレイテンシ」に近い値になります。何を測りたいかで使い分けます。サーバ推論で「カーネルローンチのオーバーヘッドも含めた end-to-end」を見たいなら後者、「カーネル自体の重さ」を見たいなら前者です。

### 統計量を複数取る

1 回測って終わりではなく、数百回測って分布を見ます。報告すべき指標は次のとおりです。具体的な数値例として、ある GPU 上のモデル推論の計測結果を併記します。

- 平均(mean): ならした代表値。例: 81.6ms。
- 中央値(median, p50): 真ん中の値。外れ値の影響を受けにくい。例: 79.4ms。
- p99: 100 回中 99 回はこれ以下、という値。最悪近辺の保証に使う。例: 104.2ms。
- ワーストケース(worst): 一番遅かった値。例: 初回の 596ms。
- ジッタ(σ、標準偏差): ばらつきの大きさ。例: 18.8ms。

なぜ平均だけではダメか。平均が同じ 80ms でも、σ が 5ms のモデル A と σ が 30ms のモデル B では、安全性がまったく違います。A は常に 75〜85ms で安定していますが、B はたまに 130ms になり、10Hz(100ms 締切)を破ってしまう。リアルタイムシステムでは「最悪でも何 ms で終わるか」が締切を守れるかどうかを決めるので、p99 とワーストとジッタを並べて報告することが必須です。

平均と中央値の関係も情報になります。平均が中央値より明らかに大きいなら、分布が右に長い裾を持つ(時々遅いフレームがある)ことを意味します。逆に両者がほぼ一致するなら対称的で安定した分布です。

### モデル時間とパイプライン時間を分ける

実運用の推論は、モデルの forward だけではありません。ファイルからの画像読み込み、JPEG のデコード、リサイズ、正規化、テンソルへの変換、CPU から GPU へのデータ転送(PCIe 経由)、そしてモデル forward、最後に出力の後処理(NMS や軌道のデコードなど)、という一連の流れ(パイプライン)全体に時間がかかります。

これらを混ぜて測ると「モデルが遅いのか、前処理が遅いのか」が分からなくなります。報告例では、入力を作り置きしてモデルだけを回す純粋なストレステストで 約 67.7ms、実データを読み込みながら回すパイプライン込みで 約 81.6ms と、約 14ms の差がありました。この差はデコードと前処理のコストです。「モデル単体の速さ」と「実データを流したときの速さ」は別物として分けて報告するのが、誠実かつ有用な計測です。ボトルネックがモデルなら量子化や蒸留で対処し、前処理なら並列デコード・GPU 前処理(NVIDIA DALI など)・データ転送のピンメモリ化で対処する、と打ち手が変わるからです。

### before / after の数字感

```
素朴な計測 (synchronize なし, 初回込み)
  → 3ms (嘘) または 596ms (初回汚染) → 信用できない
        ↓ 正しい方法論
synchronize 両側 + ウォームアップ + 統計量
  → mean 81.6ms / median 79.4ms / p99 104.2ms / σ 18.8ms / worst 596ms(別枠報告)
```

## 手法の系譜と主要論文

レイテンシ計測の作法は単一の論文というより、フレームワーク公式ドキュメントと業界標準ベンチマークで確立された共通知識です。

PyTorch 公式ドキュメントの CUDA Semantics の章は、CUDA 演算が非同期であること、正確な時間計測には `torch.cuda.synchronize()` または `torch.cuda.Event` が必須であることを明記しています。これは「非同期実行で計測がずれる」という根本問題への公式回答です。同様の注意は TensorFlow や JAX のプロファイリング文書にも共通して現れ、フレームワークを問わず GPU 計測には同期が要るという普遍的な原則です。

MLPerf Inference(Reddi ら, 2020, ISCA, "MLPerf Inference Benchmark"、MLCommons が策定)は、推論性能を業界横断で公平に比較するためのルールを定めた標準ベンチマークです。十分なウォームアップ、十分な反復回数を要求し、単一の平均だけでなくテール遅延(tail latency、分布のすそにあたる p90 / p99 / p99.9)を報告させます。さらに「Single-stream(1 件ずつ)」「Multi-stream」「Server(リクエストがポアソン分布でランダムに来る)」「Offline(まとめて処理)」といったシナリオを分け、それぞれに合った指標(レイテンシ制約下のスループットなど)を使います。これは「平均だけ良くてもテールが悪いシステムは実運用で破綻する」という問題意識から来ています。

テール遅延の重要性を体系的に論じた古典が Dean & Barroso(2013, CACM, "The Tail at Scale")です。これは GPU 推論に限らず、大規模分散システムにおいて「個々のコンポーネントが時々遅くなるだけで、全体の応答時間のテールが大きく悪化する」ことを論じ、p99 を設計の中心に据える思想を広めました。推論システムの SLA(Service Level Agreement)がしばしば「p99 < X ms」で書かれるのはこの流れです。

リアルタイムシステム・組込み・ロボティクスの文献では、平均ではなく最悪ケース実行時間(WCET, Worst-Case Execution Time)とジッタを重視する伝統があります。航空・自動車の安全規格(DO-178C, ISO 26262 など)では「締切を必ず守れること」が要求されるため、平均値は意味をなさず、最悪値の上界保証が問われます。自動運転の推論評価が p99・ワースト・ジッタを並べるのは、この思想の継承です。

プロファイリングツールの系譜としては、NVIDIA の nvprof(旧世代)から Nsight Systems / Nsight Compute(現行)への発展があります。これらはカーネル単位の実行時間、GPU 占有率、メモリ転送のタイムラインを可視化し、「どのカーネルが時間を食っているか」をミリ秒よりさらに細かく分解できます。PyTorch には `torch.profiler` が統合されており、CPU と GPU の各演算の時間配分を見られます。粗い計測(全体の ms)から細かい計測(カーネル単位)へ、という二段構えが実務の定石です。

## 論文の実験結果(定量データ)

MLPerf Inference の各ラウンドの提出結果は、同じモデル(ResNet-50, BERT, RetinaNet, 3D-UNet など)を異なるハードウェアで動かしたときのレイテンシとスループットを、共通ルールのもとで比較できる貴重なデータ集です。たとえば ResNet-50 の Single-stream レイテンシは、ハイエンド GPU で 1ms 未満、エッジ向け SoC で数ミリ秒〜十数ミリ秒と、ハードウェアによって桁が変わることが示されています。同じモデルでも測る環境で値がまったく違うので、「絶対値の比較には環境の明記が必須」というのがこのデータの教訓です。

ウォームアップの効果を定量化した報告例では、初回フレームが約 596ms、2 フレーム目以降が平均 約 81.6ms と、約 7 倍の差がありました。仮にウォームアップなしで 100 フレームの平均を取ると、初回の 596ms が混ざって平均が約 86.7ms になり、定常値 81.6ms より約 6% 高く見積もられます。フレーム数が少ないほどこの汚染は深刻で、10 フレームなら平均が約 133ms と、実態の 1.6 倍に膨らみます。これがウォームアップを必須とする定量的な根拠です。「少ない反復で測ると初回汚染がそのまま結果を支配する」ことが分かります。

アブレーション的に「synchronize を外すとどうなるか」を考えると、前後の同期を両方外した場合、計測値は「CPU が指示を発行する時間」だけになり、モデルによっては実測 80ms が見かけ上 2〜5ms になります。つまり 95% 以上の過小評価です。後ろの同期だけ外した場合も、最後の forward が GPU 上で完了する前に時刻を止めるので、やはり過小評価になります。前の同期だけ外した場合は、直前のループの残りが混入して過大評価になります。両側の同期が揃って初めて正しい値が出る、というのは「どれを抜くと壊れるか」というアブレーションそのものです。

## メリット・トレードオフ・限界

メリット。synchronize で GPU の本当の時間を測れるので数値が信頼できます。ウォームアップで初回の異常値を排除し、定常状態どうしで公平に比較できます。p99 とジッタを見れば「たまに締切を破るか」という安全評価ができます。モデル時間とパイプライン時間を分ければ、ボトルネックの所在が分かり、打ち手が決まります。

トレードオフと限界。synchronize は CPU を待たせるので、計測自体がやや重く、本番コードに残すとスループットが落ちます(計測用フラグで切り替えるのが定石)。また同期を毎反復入れると、本来 CPU/GPU が重なって隠れていたカーネルローンチのオーバーヘッドが露出し、パイプライン全体としては実運用より遅く見えることがある、という逆の歪みもあります。p99 を信頼するには十分なサンプル数(数百〜数千回)が必要で、計測に時間がかかります。最も厄介なのは環境依存の大きさで、GPU の世代、ドライバ、クロック設定、温度(サーマルスロットリングで遅くなる)、同居する他プロセスの影響を受け、同じコードでも数字がぶれます。再現性のために GPU クロックを固定する(`nvidia-smi -lgc`)、計測中は他プロセスを止める、十分に冷えた状態から測る、といった配慮が要ります。さらに、1 つの数字はバッチサイズ・解像度・系列長・精度(FP32/FP16/INT8)という条件の組合せのうち 1 点でしかなく、条件が変われば値も激変します。だから「どの条件で測ったか」を必ず併記しないと、数字は無意味になります。

未解決・難しい論点として、テール遅延の原因切り分け(カーネル起動の遅延なのか、メモリ確保なのか、OS のスケジューリングなのか、GC のような周期的処理なのか)は依然として職人芸の領域です。また、複数モデルを同一 GPU で同時に動かす(マルチテナント、MPS や MIG)状況では、相互干渉でジッタが増え、単独計測の値が当てにならなくなります。エッジでの動的電力管理(DVFS、Dynamic Voltage and Frequency Scaling)も、負荷に応じてクロックが変動するため計測を難しくします。

## 発展トピック・研究の最前線

レイテンシを「測る」だけでなく「削る」方向に進むと、量子化(FP32 を INT8 などに落とす)、知識蒸留(大きなモデルの知識を小さなモデルに移す)、構造的枝刈り(不要なチャンネルを削る)、演算子融合(複数の演算を 1 カーネルにまとめる)、グラフコンパイル(TensorRT, torch.compile, ONNX Runtime, CUDA Graphs)といった最適化群があります。特に CUDA Graphs はカーネルローンチのオーバーヘッドをまとめて削る技術で、小さなカーネルが多数並ぶモデルでテール遅延を縮めます。これらはいずれも「正しいレイテンシ計測」がなければ効果を検証できないので、計測方法論はその土台です。

最近は「精度とレイテンシのパレートフロンティア(どちらかを良くすると他方が悪くなる境界)」を体系的に探索する研究や、ハードウェアを意識したニューラルアーキテクチャ探索(HW-aware NAS)が盛んです。ここでは実測レイテンシを最適化の目的関数に組み込むため、高速かつ正確な計測、あるいはレイテンシ予測モデル(計測せずに構成からレイテンシを推定するモデル)が鍵になります。サーバ側では、リクエストをまとめて処理する動的バッチング(dynamic batching)や連続バッチング(continuous batching、特に LLM 推論)が、レイテンシとスループットのトレードオフを動的に調整する技術として発展しています。自動運転のようなエッジ・リアルタイム系では、入力レートが一定でバッチングの自由度が低いため、単一フレームのテール遅延をいかに削るかが中心課題であり続けています。
