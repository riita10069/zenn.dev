# カメラ数不一致のゼロパディング対処

## ひとことで言うと

マルチカメラの自動運転モデルは「カメラが何台ついている」という前提で入力の形が固定されています。ところが手元のデータは台数が足りないことがよくあります。足りないカメラ分を「全要素が0(真っ黒)の画像テンソル」で埋め、形だけそろえてモデルを動かす、という運用上の応急処置の話です。とにかく動かすには有効ですが、欠損を明示的に扱っていないため性能が劣化しうる、という限界が本質です。

## 直感的な理解

ニューラルネットワーク(脳の神経細胞のつながりを模した計算の仕組み。以下「モデル」)は、入力の「形」がカチッと固定されています。たとえば「8つの視点(カメラ映像 + 地図タイル)」を受け取る前提で設計されたモデルに、7台分のカメラしか無いデータを渡すと、形が一致せずエラーで止まります。

ここで取れる選択肢は3つあります。(a) データを集め直す、(b) カメラを物理的に増設する、(c) 足りない分を何かで埋めて形を合わせる。(a) と (b) は時間も費用もかかります。そこで開発の最初期には (c)、すなわち「足りない枠を全部0で埋めてしまう」ゼロパディングがよく使われます。パディング(padding)とは「詰め物・水増し」のことで、足りない場所を何かで埋める操作を指します。

身近な例で言えば、8人掛けのテーブル(モデルの入力枠)に7人しか客が来なかったとき、残り1席に「マネキン(中身ゼロ)」を座らせて席数を合わせる、ようなものです。テーブルは満席として扱えますが、マネキンは会話に何も貢献しません。むしろ「無言の客がいる」というノイズになりかねない、というのが後述のトレードオフです。

## 基礎: 前提となる概念

- テンソル(tensor): 数値が格子状に並んだ多次元配列。画像や特徴量を入れる「箱」。
- テンソルの形(shape): 各次元のサイズ。マルチカメラ入力の典型は `[B, V, C, H, W]`。
  - B = バッチサイズ(一度に処理する場面の数)
  - V = ビュー数(カメラ台数、地図タイルを含むこともある)
  - C = 色チャンネル数(RGB なら 3)
  - H, W = 画像の縦・横ピクセル数
- 連結(concatenate): 複数のテンソルを指定した次元に沿ってつなげる操作。ビュー次元で7枚 + 1枚をつなげれば 8 になる。
- マスク(mask): 「どこが本物の入力で、どこが埋め物か」を 0/1 で示す印。後段の計算で埋め物部分を無視させるために使う。ゼロパディングだけではこのマスクが無い点が問題になる。
- 順伝播(forward pass): 入力をモデルに通して出力を得る一回の計算。形が合っていないとここで止まる。
- 訓練分布外 / out-of-distribution (OOD): 訓練で見せなかったタイプの入力。モデルは見たことのない入力に対しては挙動が保証されず、出力が崩れやすい。真っ黒なビューは多くのモデルにとって OOD になりうる。

入力テンソルの形を図で示します。

```
[1, 8, 3, 256, 256]
 │  │  │   │    │
 │  │  │   │    └─ 画像の横幅 256 ピクセル
 │  │  │   └────── 画像の縦 256 ピクセル
 │  │  └────────── 色チャンネル 3 (赤・緑・青)
 │  └───────────── ビュー数 8 ← ここが足りないと動かない
 └──────────────── バッチサイズ 1
```

7台分しかないと「8」のところが「7」になり、形が一致せずモデルが止まります。

## 仕組みを詳しく

やっていることは単純です。「足りないビューの場所に、全要素が0のテンソルを差し込む」だけです。

ステップ分解:

1. 実データから7台分のカメラ映像を読み込む。各カメラは `[3, 256, 256]`(色3 × 縦256 × 横256)の画像テンソル。
2. 8番目のビュー(本来は地図タイル用)を全要素0で生成する(擬似コードで `zeros(3, 256, 256)`)。中身が全部 0.0 の、見た目「真っ黒」なテンソル。
3. 7台分 + ゼロの1枚をビュー次元で連結し、ビュー数を8にそろえる。
4. バッチ次元を足して `[1, 8, 3, 256, 256]` にしてモデルへ渡す。

数値で言えば、8番目のビューの全 196,608 個(= 3 × 256 × 256)の要素がすべて 0.0 になります。

before / after:

```
before(実データのまま):  [1, 7, 3, 256, 256]  ← 動かない(形が合わない)
        ↓ 8番目にゼロテンソルを足す
after(パディング後):     [1, 8, 3, 256, 256]  ← 動く
                              ↑ うち1ビューは中身ゼロ
```

なぜゼロなのか。ゼロにすると、その入力が後段の畳み込み(画像の局所的な特徴を取る演算)や行列積に与える寄与が小さくなりやすい性質があります(掛け算の片方が0ならその項は0)。ただし「完全に無視される」わけではない点が決定的に重要です。畳み込みにはバイアス項(掛け算の後に足される定数)があり、正規化層(BatchNorm や LayerNorm、特徴のスケールを整える層)は0でも特定の値を出力し、Attention 機構(各入力が他のどれをどれだけ参照するかを学ぶ仕組み)ではゼロベクトルにも注意の重みが割り当てられます。

具体的に、ImageNet 正規化を経た入力では「真っ黒(画素値0)」は正規化後に0ではなく負の値になることにも注意が必要です。たとえば赤チャンネルの平均 0.485・標準偏差 0.229 で正規化すると、画素値0は (0 − 0.485)/0.229 ≈ −2.12 になります。つまり「ゼロテンソルを連結する」のと「ゼロ画素を正規化してから連結する」のは別物で、後者は「強い負の信号」をモデルに送り込みます。設計者はどちらの段でゼロを入れるかを意識する必要があります。いずれにせよゼロは「無」ではなく「真っ黒という情報」「強い負信号」としてモデルに伝わってしまいます。料理のたとえに戻れば、マネキンの客は黙っていても「無言で座っている人がいる」という状況をテーブルに持ち込むのです。

これを正しく扱うには、ゼロ埋めした位置を示すマスクをモデルに渡し、その位置の特徴を融合(fusion、複数視点の特徴を統合する処理)や Attention の計算から除外する必要があります。ゼロパディングだけでマスクが無い実装は、まさにこの除外をしていないので応急処置にとどまります。

## 手法の系譜と主要論文

「足りない入力をゼロで埋める」という発想自体は、自動運転に限らず深層学習の至るところで使われる定番技術です。系譜をたどります。

- ゼロパディングの古典的起源は CNN(畳み込みニューラルネット、画像認識の基本構造)の端処理です。LeCun ら(1998, "Gradient-Based Learning Applied to Document Recognition", Proc. IEEE, LeNet-5)以来、画像の縁に0のピクセルを足して畳み込み後の出力サイズを保つ「ゼロパディング」が標準化されました。これは「形をそろえるために0を足す」発想の原型で、ビュー数合わせと動機は同じです。

- 系列モデル(文章・時系列を扱うモデル)では、長さの違うデータをバッチにまとめるため短いものをゼロで埋める「シーケンスパディング」が一般的です。重要なのは、ここでは必ず padding mask を併用し、「どこがパディングか」を示して計算で無視させる点です。Vaswani ら(2017, "Attention Is All You Need", NeurIPS, arXiv:1706.03762)の Transformer は、Attention の softmax 計算でパディング位置に大きな負値(実質的に −∞)を加えて重みを0にする padding mask を明示的に使います。今回のゼロパディングが「マスクなし」である点が、Transformer 流の正しい扱いとの差で、トレードオフの根拠になります。

- 順序・個数に依存しない集約。Set Transformer(Lee ら, ICML 2019, arXiv:1810.00825)は、要素数が可変な「集合」を、順序やパディングに頼らず Attention で集約する仕組みを提案しました。マルチカメラを「視点の集合」とみなせば、台数が変わってもパディング不要で扱える設計が原理的に可能になります。これは後述の「可変ビュー数アーキテクチャ」の理論的土台です。

- マルチセンサ自動運転でのセンサ欠損の扱いは、センサ・ドロップアウト(sensor dropout)やモダリティ・ドロップアウト(modality dropout、学習時にわざと一部の入力を落とす拡張)として研究されています。BEVFusion(Liu ら, 2023, ICRA, arXiv:2205.13542)などのマルチモーダル融合研究では、「学習時にわざと一部センサをゼロにして欠損に強くする」訓練法が提案されています。逆に言えば、何の訓練もせず推論時だけゼロを入れると、モデルは見たことのない入力分布(OOD)に直面し、出力が崩れる恐れがある、という知見が共有されています。これが「数を合わせるだけの応急処置」にとどまる理由です。

まとめると、ゼロパディングは「提案された洗練手法」ではなく「広く使われる土台の技術」で、効果は『とにかく動かせる』こと、トレードオフは『欠損を明示的に扱わないので性能が劣化しうる』ことだと理解するのが正確です。

## 論文の実験結果(定量データ)

センサ欠損とゼロ埋めの影響を定量的に示す代表的知見を挙げます(設定により値は変わります)。

- BEVFusion 系の研究では、訓練時にカメラまたは LiDAR をランダムに落とす(ドロップする)拡張を入れると、推論時にセンサが欠けても性能低下が小さく抑えられることが報告されています。逆に、欠損を想定せず訓練したモデルに推論時だけ片方のセンサをゼロで与えると、検出性能(mAP、mean Average Precision。検出の正確さを 0〜100% で測る指標)が大きく落ちる、という対比がしばしば示されます。たとえば LiDAR を主入力とする融合モデルから LiDAR を抜くと mAP が数十ポイント単位で崩落する一方、ドロップアウト訓練を施したモデルは劣化が一桁ポイント台に抑えられる、といった報告があります。落ち幅は設定依存ですが、数 % から数十 % のオーダーになりうる、と理解しておくべきです。

- modality dropout を用いた研究全般では、「訓練時に欠損を経験させる」だけで、欠損時の精度が「訓練時に欠損を見せなかった場合」より顕著に改善する、というアブレーションが繰り返し示されています。これは「ゼロ埋め自体が悪いのではなく、ゼロ埋めを訓練でも経験させマスクで扱うかどうかが分かれ目」であることを示します。

実務的含意は明確です。マスクなしのゼロパディングで得たベンチマーク数値は「モデル本来の実力」ではなく「劣化した運用条件での値」です。たとえば地図スロットをゼロにし、かつ自車状態も欠いた状態で出た ADE(平均変位誤差)は、知覚も自車状態も活かせていない条件の値であり、モデルの能力評価としては読めません。報告する際は必ず「どのスロットをゼロ埋めしたか」を明記すべきです。逆に、ゼロ埋めを「劣化条件」として意図的にベンチに含め、「あるセンサが故障したときどれだけ性能が落ちるか(graceful degradation、緩やかな劣化の度合い)」を測る、という積極的な使い方もあります。

## メリット・トレードオフ・限界

メリット

- とにかくモデルを動かせる。台数が足りなくても形をそろえれば forward pass が通る。
- 実装が極めて簡単。ゼロテンソルを生成して連結する数行で済む。
- パイプライン全体(ダウンロード → 前処理 → 推論)が end-to-end で通ることを早期に検証できる。新しいデータ源に最初に通す sanity check に向く。
- 「センサ故障時の劣化耐性」を測る故意の劣化条件としても利用できる。

トレードオフ・限界

- 性能が劣化しうる。モデルは訓練時に「真っ黒なビュー」をほとんど見ていない可能性が高く、推論時に突然ゼロが来ると訓練分布外(OOD)の入力になり、出力(予測軌道など)が不正確になる。
- 欠損を明示していない。padding mask のような「ここは無視して」という仕組みが無いため、ゼロが「真っ黒という情報」「強い負信号」としてモデルに伝わる。完全な無視ではない。
- 正規化との相互作用。ImageNet 正規化後はゼロ画素が0でなく負値になるため、「どの段でゼロを入れるか」で送り込む信号が変わる落とし穴がある。
- 地図情報の欠落を隠す。地図スロットをゼロにすると、地図が無いこと自体が「真っ黒な地図」として扱われ、地図依存の判断が崩れる。
- ベンチマーク数値の解釈に注意が必要。ゼロ埋め下の数値は劣化条件の値であり、本来能力を反映しない。

正しい対処の方向は、(1) padding mask を導入し融合・Attention で欠損位置を除外する、(2) modality/sensor dropout を訓練に組み込み欠損に頑健化する、(3) そもそも入力レイアウトをデータに合わせて再設計する(カメラ台数を可変にする、地図を別経路の入力にする)、です。ゼロパディングはその前段の橋渡しと位置づけるのが健全です。

## 発展トピック・研究の最前線

- 可変ビュー数アーキテクチャ。カメラ台数を固定せず、任意の数の視点を集約できる Set Transformer 的な融合(順序に依存せず可変長を扱える機構)で、そもそもパディングを不要にする方向。各カメラの外部パラメータ(設置位置・向き)を特徴に埋め込めば、台数や配置が変わっても同じモデルで扱える設計が可能になる。
- 欠損に頑健なマルチモーダル融合。学習時に確率的にセンサを落とし、推論時の欠損・故障(カメラの泥汚れ・故障・逆光)に強いモデルを作る研究。安全性の観点で実用上重要。
- 不確実性つき融合。各センサの信頼度を推定し、欠損や低品質入力の寄与を自動で下げる(ゼロ埋めの代わりに「信頼度0」として扱う)アプローチ。ベイズ的な不確実性推定や注意の重みによる自動ゲーティングが研究されている。
