# 学習可能BEVクエリ

## ひとことで言うと

車を真上から見下ろした地図(BEV = Bird's-Eye-View、鳥瞰図(ちょうかんず)。ヘリコプターから地面を見下ろしたような視点)の各マスに、「このマスは、カメラ画像のどこを・どんなふうに見たいか」を表す小さなベクトル(数値の並び)を1つずつ持たせます。このベクトルは最初はでたらめなランダム値ですが、ニューラルネットワークの学習(訓練)を通じて自動的に良い値へ育っていきます。これを「学習可能 BEV クエリ(learnable BEV queries)」と呼びます。「クエリ(query)」とは「問い合わせ」のことで、各マスが画像に対して「私のための情報をください」と能動的に問い合わせる主体になる、というイメージです。

## 直感的な理解

複数台のカメラ画像を1枚の BEV 地図にまとめたいとします。素朴には「BEV のマスに対応する3D座標を計算し、それをカメラ画像へ投影して、そこの画像特徴をそのままコピーする」だけでも一応動きます(これに近いのが [逆透視変換 (IPM)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0280_inverse-perspective-mapping))。しかしこの素朴方式には大きな弱点があります。

- すべてのマスを同じルールで処理するので、マスごとの事情(自車の真横は重要、遠方は不確実、交差点中央は注意が要る、など)を反映できない。
- どの高さ・どの位置をどれくらい重視して特徴を拾うか、というのが固定で、データから学べない。

人間が手で「このあたりは重視」とルールを書くのは大変で、しかも最適とは限りません。そこで発想を逆転させます。各マスに小さなパラメータ(学習可能ベクトル)を持たせ、「そのマスにとって最適な情報の集め方」を学習でデータから自動的に決めさせるのです。マスが受け身の容れ物ではなく「主役(クエリ)」になって、画像から欲しい情報を能動的に引き出す。これは Transformer(後述)のアテンション(注意機構)の考え方を BEV に持ち込んだものです。

身近なたとえで言えば、地図の各マスが「自分専属の調査員」を雇うようなものです。前方20m・左3mのマスの調査員は「歩行者をよく見つけるコツ」を学習で身につけ、遠方50mのマスの調査員は「車のテールランプを手がかりにする」コツを身につける。マスごとに違う専門知識をベクトルに蓄えられるのが、固定ルールとの決定的な違いです。

## 基礎: 前提となる概念

このトピックを理解するには、いくつかの前提知識が必要です。順に噛み砕きます。

埋め込み(embedding): 「ID番号を入れると、対応するベクトルを返す辞書」のような仕組みです。たとえば単語の意味をベクトルで表すとき、「犬→[0.2, -0.5, ...]」のように対応表を作ります。この対応表の中身(各ベクトル)が学習対象のパラメータになり、誤差逆伝播(後述)で更新されます。BEV クエリでは「マスの番号→そのマス専用のベクトル」という辞書になります。

Transformer とアテンション: アテンションとは「複数の情報源の中から、いま必要なものを重み付きで選んで混ぜる」仕組みです。三つの役者がいます。クエリ(query, 何を探しているか)、キー(key, 各情報源が何を持っているか)、バリュー(value, 各情報源の中身)。クエリとキーの一致度(内積)を計算し、それを重みにしてバリューを加重平均します。BEV クエリ方式では、BEV のマスがクエリ、カメラ画像特徴がキーとバリューになります。

誤差逆伝播と微分可能性: ニューラルネットは「出力の誤差(損失)を、各パラメータで微分して、誤差が減る方向にパラメータを少しずつ動かす」ことで学習します。だから途中の処理がすべて微分可能(なめらかにつながっている)である必要があります。クエリも投影もアテンションも微分可能なので、BEV クエリは end-to-end(入力画像から最終出力まで一気通貫)で学習できます。

カメラ幾何(intrinsic / extrinsic): カメラの内部パラメータ(intrinsic, 焦点距離・画像中心をまとめた3×3行列 K)と外部パラメータ(extrinsic, 車体に対する取り付け位置と向きを表す回転 R・並進 t)を使うと、3D空間の点が画像のどのピクセルに写るかを計算できます。これを「投影(projection)」と呼びます。BEV クエリ方式は、各 BEV マスに3D座標を仮定し、この投影でカメラ画像上のサンプリング位置を求めます。

## 仕組みを詳しく

学習可能 BEV クエリの実装は、概念のわりに驚くほど簡潔です。深層学習フレームワークでは、埋め込み層(典型的には `nn.Embedding` に相当する層)を1つ用意するだけで作れます。

```
bev_queries = Embedding(num_cells = bev_h * bev_w, dim = embed_dim)
```

ここで `bev_h * bev_w` がマスの総数、`embed_dim` が1マスあたりのベクトルの長さ(チャネル数)です。たとえば BEV を縦200×横200=40000マス、各ベクトルの長さを256とすると、40000×256 ≈ 1024万個のパラメータを持ちます。これらは最初ランダムに初期化され、学習で更新されます。

forward(順伝播)での使われ方を、tensor の shape(形状)で追います。バッチサイズを B、マス数を N = bev_h × bev_w、チャネルを C = embed_dim とします。

```
queries = bev_queries.weight             # [N, C]    学習済みベクトル群
queries = queries.expand(B, N, C)        # [B, N, C] バッチ方向に複製
```

ここから、各マスのクエリベクトルが「どう情報を集めるか」の指示を生成します。代表的な BEV クエリ方式(変形可能アテンション系)では2種類の指示を出します。

1. サンプリングオフセット(sampling offsets): クエリ → 全結合層 → 「投影した基準点から、どれだけずらして特徴を拾うか」の微調整量。3D点を画像に投影しただけだと位置がずれることがあるので、その補正をデータから学ぶ。

```
offsets = Linear(queries)   # [B, N, num_points * 2]   (画像上の x,y ずらし量)
```

2. アテンション重み(attention weights): クエリ → 全結合層 → 「複数のサンプリング点・複数の高さ・複数カメラのうち、どれを重視するか」の重み。

```
attn_weights = Linear(queries)  # [B, N, num_cameras * num_points]
attn_weights = softmax(attn_weights)
```

つまり各マスのクエリは「私のための特徴は、投影点からこれくらいずらした位置の、この高さ・このカメラを重視して拾ってきて」という指示を出します。指示に従って画像特徴をサンプリングし([grid_sample バイリニア特徴抽出](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0967_grid-sample-bilinear))、見えるカメラだけを残す可視性マスク([可視性マスク(Visibility Masking)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0708_visibility-masking))を掛け、加重平均して集約します。最後にクエリ自身を残差(residual, 元のクエリを足し戻す)として加え、正規化と前向き層(FFN)を通します。

```
output = queries + output_proj(aggregated)   # 残差接続
output = norm(output)
output = output + ffn(output)
output = norm(output)
```

なぜ残差で足し戻すのか。残差接続は深いネットワークでも勾配が流れやすくし、「クエリが持つもとの知識を壊さずに、画像から拾った情報を上乗せする」働きをします。

数値例: マス「前方20m・左3m」のクエリが、学習の結果「投影点よりやや上(空に近い側)を見ると、その位置に立つ歩行者の頭が拾える」と学んでいたとします。すると offsets はその方向を指し、attn_weights は歩行者の高さの点を重視します。マス「前方50m・正面」のクエリは「先行車のテールランプの高さを重視」と学ぶかもしれません。こうしたマス固有の見え方の知識がベクトルに蓄積されます。手書きできるルールではなく、大量データからの学習でしか得られないものです。

なぜ高さ方向に複数点をサンプリングするのか。BEV のマスは地面上の (x, y) しか指定しません。しかし実世界はそのマスの上に高さ方向の柱(歩行者なら0〜2m、トラックなら0〜4m)が立っています。1つの高さだけ見ると物体の一部しか拾えないので、複数の高さ(柱状の参照点、典型的には4点程度)をサンプリングして、どの高さに情報があるかをアテンションで重み付けします。これが3Dの立体構造を BEV に押し込む工夫です。

## 手法の系譜と主要論文

DETR(Carion et al., ECCV 2020, arXiv:2005.12872, Facebook AI): 「学習可能クエリ」という発想の源流です。物体検出を「100個の学習可能オブジェクトクエリが、画像特徴から物体を引き出す」問題として定式化しました。それまでの検出器が必要としていた手作りの後処理(NMS = Non-Maximum Suppression, 重複検出をまとめる処理、アンカー設計)を、クエリとハンガリアンマッチング(予測と正解を最適に1対1対応づける割り当てアルゴリズム)で置き換えた点が新規性です。クエリが「学習する問い合わせベクトル」であるという考え方は、ここから広がりました。

Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159): DETR の弱点(全ピクセルへのアテンションは計算が重く、収束が遅い。DETR は約500エポックの学習を要した)を、変形可能アテンション(deformable attention)で解決しました。全位置を見るのではなく、各クエリが少数のサンプリング点(典型的には4点)だけを参照し、その位置をオフセットで学習する。学習を約10倍速くし(50エポックで同等精度)、BEV クエリ方式の「サンプリングオフセット + アテンション重み」の直接の祖先になりました。

DETR3D(Wang et al., CoRL 2021, arXiv:2110.06922): クエリ系を3D空間に持ち込んだ橋渡しです。各オブジェクトクエリが3D参照点を予測し、それをカメラ幾何で全カメラ画像へ投影して特徴を集める「sparse(疎)」な手法を示しました。BEV グリッドのような密な表現を持たず、物体がありそうな少数の3D点に直接クエリを置く点が、後のスパース系の流れを準備しました。

BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270): 学習可能 BEV クエリを明確に打ち出した代表論文です。BEV 平面をグリッドに切り、各セルに学習可能クエリを置きました。各クエリは2種類のアテンションの起点になります。空間クロスアテンション(Spatial Cross-Attention, 複数カメラ画像を参照し、変形可能アテンションで効率化)と、時間自己アテンション(Temporal Self-Attention, 過去フレームの BEV を参照して動きや遮蔽に強くする)。深度を一切明示的に予測せず、投影とアテンションだけで複数カメラ・複数時刻を1枚の BEV に統合できる点が新規性です。これは深度推定([深度分布スプラッティング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0277_depth-distribution-splatting))とは異なる「pull(引っ張る)」系の代表となりました。

UniAD(Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156): BEVFormer を BEV エンコーダとして使い、その上に検出・追跡・地図・予測・計画を1本のネットワークに積み上げました。BEV クエリで作った BEV 特徴が、下流すべてのタスクの共通の土台になる、という使われ方を示し、エンドツーエンド自動運転の旗艦的研究になりました。

StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926): 密な BEV グリッドを持たず、物体クエリを時間方向に「ストリーミング」で伝播する設計です。前フレームのオブジェクトクエリを次フレームへ持ち越し、長期の時間情報を低コストで蓄積します。密 BEV クエリの計算コストを避けつつ時間統合を実現する、スパース系の到達点の一つです。

## 論文の実験結果(定量データ)

評価は主に nuScenes(ヌーシーンズ)という大規模自動運転データセットで行われます。これは6台のサラウンドカメラ・LiDAR・レーダーを積んだ車で集めた、ボストンとシンガポールの1000シーン(各20秒)のデータです。

3D物体検出の主要指標を噛み砕きます。NDS(nuScenes Detection Score)は、検出の総合点で、mAP に加えて位置・大きさ・向き・速度・属性の5種の誤差(mATE, mASE, mAOE, mAVE, mAAE)をまとめた0〜1の指標(高いほど良い)。mAP(mean Average Precision)は、物体を正しく見つけられたかの平均適合率で、ここでは中心位置が一定距離(0.5/1/2/4m)以内かで正否を判定します(高いほど良い)。

BEVFormer は、nuScenes test セットで報告によりおよそ NDS 0.569、mAP 0.481(大型バックボーン使用時)を達成し、当時のカメラのみ手法で最高水準でした。重要なアブレーション(構成要素を抜く実験)として、時間自己アテンション(過去フレーム参照)を外すと速度推定の誤差 mAVE が大きく悪化し(報告では概ね 0.4 台から 0.5 超へ)、NDS が数ポイント下がったと報告されています。これは「時間情報が、瞬間の1フレームでは取れない速度や遮蔽下の物体に効く」ことを定量的に示しています。空間クロスアテンションを単純なグローバルアテンションから変形可能アテンションに変えることで、メモリと計算を大きく削減しつつ精度を保てたことも示されました。

DETR3D は nuScenes で NDS 約0.43(ResNet-101 系)を報告し、密 BEV を持たない疎なクエリでも複数カメラ統合が成立することを示しました。StreamPETR はストリーミング時間統合により、同程度のバックボーンで NDS 0.55 超(test)に達し、密 BEV を持たないスパース系でも時間情報を活かせば BEVFormer に匹敵することを示しました。

UniAD は、各タスク専用に作った従来手法と比べ、計画(自車の将来軌道予測)の平均 L2 誤差と衝突率を大幅に下げたと報告しています。具体的には、複数タスクを共有 BEV 特徴で繋ぐことで、検出・追跡・地図・動き予測の各指標が、それらを別々に学習した場合より改善したことを示し、「BEV クエリで作った共通表現を end-to-end で繋ぐ」価値を定量的に裏付けました。

これらの数値の意味: NDS が0.5から0.57に上がるというのは、誤検出と位置ずれが目に見えて減るということで、自動運転では「前方の車を取りこぼさない」「歩行者の位置を1m単位で当てる」に直結します。時間情報を抜くと悪化するというアブレーションは、BEV クエリ単体ではなく「クエリ + 時間アテンション」のセットで価値が出ることを教えてくれます。

## メリット・トレードオフ・限界

メリット

- マスごとに固有の「情報の集め方」を学習でき、手書きルールが不要。
- Transformer のアテンション機構と自然につながり、時間方向(過去フレーム)・空間方向(複数カメラ)の拡張がしやすい。
- 深度を明示的に予測しなくても、クエリ + 投影 + アテンションだけで複数カメラを統合できる。深度推定の難しさを回避できる。
- BEV 特徴が下流の複数タスクの共通土台になり、マルチタスク学習と相性が良い。

トレードオフ・限界

- マス数 × embed_dim ぶんのパラメータが必要。高解像度 BEV ではメモリと計算が重く、バッチサイズや解像度に厳しい上限がかかる(高解像度・大バッチは大容量 GPU でもメモリ不足になりやすい)。
- 学習しないと意味のあるベクトルにならない。十分なデータと学習時間が前提。
- 各マスが独立にパラメータを持つため、学習データに無い構図・未観測の位置への汎化は保証されない。
- 正確なカメラキャリブレーションへの依存が残る(投影位置が幾何に基づくため、extrinsic がずれると拾う位置がずれる)。

研究上の未解決課題: 高解像度化に伴うメモリ爆発をどう抑えるか(スパース化や階層化の研究が続く)、クエリの初期化や事前知識の入れ方(マップ事前情報をクエリに焼き込む)、キャリブレーション誤差や時刻同期のずれへのロバスト性、そして長距離(100m超)での精度低下が代表的なオープン問題です。

## 発展トピック・研究の最前線

スパース化の流れ: 密な BEV グリッドはマスが多すぎてコストが高いので、物体がありそうな場所だけにクエリを置く「スパースクエリ」系(PETR, Sparse4D, StreamPETR など)が発展しました。グリッド全体を持たず、3D空間に直接クエリを配置して3D位置エンコーディングで幾何を注入する設計です。

時間の長期化とストリーミング: BEVFormer は直前フレームを参照しますが、より長い履歴を効率よく持ち越す「ストリーミング」設計(前フレームのクエリ状態を伝播する)が精度と効率の両面で進みました。8フレーム以上の長期時間融合で遠方・遮蔽の検出が改善することが報告されています。

占有予測(occupancy)への展開: BEV(2D俯瞰)から、高さ方向も含む3Dボクセルの占有を予測する Occupancy Network 系(TPVFormer, SurroundOcc など)へ発展しています。BEV クエリを3Dボクセルクエリ(あるいは三平面 TPV クエリ)に拡張し、より細かい立体構造を表現します。

エンドツーエンド計画との統合: UniAD 以降、BEV クエリで作った表現の上に計画を直接乗せる研究(VAD, GenAD, VADv2 など)が活発で、検出・予測・計画を1つの微分可能パイプラインにする方向が主流になりつつあります。
