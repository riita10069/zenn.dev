# オンラインHDマップ構築

## ひとことで言うと

事前に人手で作り込んだ高精細地図(HDマップ)に頼らず、走行中に車載センサ(主にカメラ、補助的にLiDAR)から、車線の境界線・道路の縁石・横断歩道といった「地図要素」をその場でリアルタイムに推定する技術です。これにより、地図のメンテナンスコストや、地図が古くて現実と食い違う問題を減らし、地図のない場所でも自動運転スタックが必要とする道路構造を供給できます。

「HDマップ(High-Definition map、高精細地図)」とは、車線がどこにあるか、どちらに繋がるか、停止線・横断歩道・標識がどこにあるかを、数センチ〜数十センチの精度で記述した地図です。自動運転車はこれを使って「自分がどの車線にいて、次にどう曲がるべきか」を判断します。「オンライン」とは「走りながらその場で」という意味で、事前作成済み地図を読み込む「オフライン」と対比されます。

## 直感的な理解

人間のドライバーは、初めて走る道でも、目の前の白線や縁石を見れば「この車線をたどればいい」と即座に理解できます。事前に道路台帳を暗記しているわけではありません。オンラインHDマップ構築は、この「見た情報から走行可能な構造を即座に組み立てる」能力を機械にやらせるものです。

従来の自動運転は、専門の計測車で道路を事前にスキャンし、人手で精密な地図を作っていました。これは精度が高い反面、

- 地図を作る・更新するコストが莫大
- 工事や車線変更で現実が変わると地図が古くなる
- 地図のない新しい地域では走れない

という弱点がありました。オンライン構築は、車に積んだカメラの映像から、その場で「車線の境界線はここ、横断歩道はここ」と推定して出力します。出力の形が重要で、ただ画像のどこが車線かを塗る(セグメンテーション)のではなく、後段の経路計画がそのまま使える「線(ポリライン)」や「閉じた領域(ポリゴン)」というベクトル形式で出すのが現代的なアプローチです。

## 基礎: 前提となる概念

### BEV(鳥瞰図)表現

オンラインHDマップは「BEV(Bird's-Eye-View、鳥瞰図)」、つまり真上から見下ろした座標系の上に地図要素を描きます。理由は単純で、地図も経路計画も「真上から見た平面の世界」で考えるからです。

カメラは前向き・斜め向きに付いていて、画像は遠近のついた斜めの視点です。これを真上から見た平面に変換する必要があります。複数カメラの画像特徴をBEV平面に投影する処理が、現代手法の心臓部です(詳しくは[BEV知覚レンジ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0263_bev-perception-range)を参照)。

### ラスタ表現 vs ベクトル表現

地図要素の表し方には2種類あります。

- ラスタ(raster)表現: BEV平面を格子(ピクセル)に分け、各セルが「車線か/縁石か/横断歩道か」を持つ。画像のセグメンテーションと同じ形式。
- ベクトル(vector)表現: 地図要素を点の並び(ポリライン)や多角形(ポリゴン)で表す。例えば車線境界を `[(x1,y1), (x2,y2), ..., (xN,yN)]` という座標列で表現する。

経路計画は「この線をたどる」という形で道路を扱うので、ベクトル表現の方が後段に直結します。ラスタ出力からベクトルを取り出すには後処理が必要で、誤差や繋がりの破綻が起きやすいという問題があります。

### Transformer と集合予測

近年の手法はTransformer(注意機構で要素間の関係を学ぶネットワーク)を使い、地図要素を「集合」としてまとめて予測します。物体検出のDETR(Carion et al., 2020)という手法の考え方を地図に応用したもので、「学習可能なクエリ(問い合わせベクトル)」の各々が1つの地図要素(1本の車線など)を担当し、画像特徴を参照しながら座標列を直接出力します。

## 仕組みを詳しく

### HDMapNet: BEVセグメンテーションから始める

HDMapNet(Li et al., ICRA 2022, arXiv:2107.06307)は、複数カメラ(必要に応じてLiDARも)の特徴をBEVに変換し、BEV平面上でセマンティックセグメンテーション(各ピクセルのクラス分け)を行います。さらにインスタンス埋め込みと方向予測を組み合わせ、後処理でベクトル化します。

```
6カメラ画像 → 画像特徴抽出 → BEVへ変換 → BEV上でセグメンテーション
            → (後処理: クラスタリング・追跡) → 車線/縁石/横断歩道のベクトル
```

この方式は、出力がピクセル単位なので後処理が必要で、線の繋がりが切れたり、複数の線が混ざったりする弱点がありました。それでも「事前地図なしでHDマップ要素を学習推定する」という枠組みと、その評価指標を提示した点で基礎を築いた重要研究です。

### VectorMapNet: ベクトルを直接出力する

VectorMapNet(Liu et al., ICML 2023, arXiv:2206.08920)は、ラスタを介さず地図要素をベクトル(ポリライン)として直接生成する最初期の手法です。要素を検出し、その後で各要素の点列を自己回帰的(1点ずつ順番に)に生成しました。後処理を減らした一方で、点を順番に出す生成は遅く、点の順序付けが難しいという課題がありました。

### MapTR: 並列に、順序の曖昧さを解決して

MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)は、現在広く参照される代表手法です。鍵となる工夫は次の通りです。

第一に、各地図要素を「固定数の点の並び」として表し、Transformerのクエリで並列に(1点ずつではなく一気に)予測します。これにより高速かつ端から端まで学習可能(エンドツーエンド)になりました。

第二に「順序の曖昧さ(permutation ambiguity)」への対処です。車線の境界線は、左端から右端へ点を並べても、右端から左端へ並べても、同じ線を表します。閉じた図形(横断歩道の四角形など)はどの頂点から始めてもよい。素朴に1つの順序を正解とすると、同じ線なのに「順序が逆」というだけで大きな損失が出てしまい学習が不安定になります。MapTRは「点列を巡回的・両方向に等価とみなす」マッチングを導入し、正解との対応付けで最も損失が小さい並べ方を選ぶことで、この曖昧さを吸収しました。

```
素朴な方法: 正解 [A,B,C,D] と予測 [D,C,B,A] → 順序違いで大損失(同じ線なのに)
MapTR:      等価な並べ方をすべて候補にし、最小損失の対応を採用 → 安定して学習
```

第三に、点同士・要素同士の整合を測る損失を組み合わせ、線がガタつかず滑らかに繋がるようにしています。

### 後段との関係

出力されたベクトル地図要素(車線中心線、境界、横断歩道など)は、経路計画や予測モジュールにそのまま渡されます。オフラインHDマップの代わりとして使える形を、人手の後処理なしで出すことが目標です。

## 手法の系譜と主要論文

- HDMapNet(Li et al., ICRA 2022, arXiv:2107.06307): BEVセグメンテーション+後処理ベクトル化。タスクと評価指標を確立した基礎研究。
- VectorMapNet(Liu et al., ICML 2023, arXiv:2206.08920): ラスタを介さずベクトルを自己回帰生成する初期のエンドツーエンド手法。
- MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)とMapTRv2(arXiv:2308.05736): 並列予測と順序曖昧性の解決で速度と精度を両立。本分野の標準的ベースライン。
- StreamMapNet(Yuan et al., WACV 2024, arXiv:2308.12570): 時系列(過去フレーム)の情報を伝播させ、遠方や遮蔽に強い時間的に一貫した地図を構築。
- 標準地図(SDマップ)併用の流れ(2024年前後): カーナビ用の粗い地図(車線数や道路接続の概略を持つ)を事前情報として与え、推定を安定させる研究。完全に地図なしと、精密な事前地図の中間を狙う。

系譜は「ラスタ+後処理(HDMapNet)→ ベクトル直接生成(VectorMapNet)→ 並列・安定なベクトル生成(MapTR)→ 時間情報と事前情報の活用(StreamMapNet、SDマップ併用)」と整理できます。

## 論文の実験結果(定量データ)

主な評価データセットはnuScenes(都市運転の大規模マルチセンサデータセット)で、評価指標はmAP(mean Average Precision、平均適合率の平均)です。地図要素ではChamfer距離(2つの点集合がどれだけ近いか。小さいほど一致)のしきい値で「当たり/外れ」を判定してAPを計算します。値が大きいほど良い指標です。数値は各論文の報告に基づきます。

- HDMapNet(2022): nuScenvesでの地図セグメンテーション/ベクトル化を提示。後続手法のベースラインとして、ベクトル指標で比較的低めのmAP(数十パーセント未満の領域)を示しました。
- VectorMapNet(2023): HDMapNetを上回るベクトルmAPを報告。ただし自己回帰生成のため推論が遅く、報告ではフレームあたりの処理が遅い(リアルタイムには届きにくい)とされています。
- MapTR(2023): nuScenesでベクトル化mAPをVectorMapNetから大きく改善し、報告ではおよそ50パーセント前後のmAPを達成。同時に並列予測により推論が高速化し、リアルタイムに近い速度(報告でおよそ毎秒十数枚)を実現しました。アブレーションでは、順序曖昧性を扱うマッチングを外すとmAPが明確に低下することを示し、この工夫の効果を裏付けています。
- MapTRv2(2023): 補助的な損失や3D化への拡張などで、MapTRからさらに数ポイント〜十数ポイントmAPを改善したと報告。
- StreamMapNet(2024): 時系列伝播の導入で、特に長距離・遮蔽下の地図構築で従来手法を上回るmAPと時間的一貫性を報告しました。

数値の意味を補足すると、ベクトル地図のmAPが30パーセント台から50パーセント台へ上がるというのは、推定した車線や横断歩道のうち「正解の位置・形に十分近い」と判定される割合が大きく増えるということです。ただしmAP 50パーセントは「半分は基準を満たさない」とも読めるため、安全に直結する自動運転用途としては依然として精度に余地があります。

## メリット・トレードオフ・限界

メリット:

- 事前HDマップの作成・更新コストを削減でき、地図のない地域でも走行に必要な構造を供給できる。
- 工事や車線変更など、地図が現実と食い違う状況に強い(その場で見て推定するため)。
- ベクトル出力なら後段の計画モジュールに直結し、後処理を減らせる。

トレードオフと限界:

- 認識距離と遮蔽。カメラの視界が遮られる交差点や、遠方の車線は推定が不安定。事前地図なら見えない先まで分かるが、オンライン推定は見える範囲に縛られる。
- 精度。最先端でもmAPは安全用途として十分とは言い切れず、誤った車線接続や境界のズレが致命的になりうる。
- 一貫性。フレームごとに独立推定するとちらつき(同じ車線が出たり消えたり)が起きる。時系列処理で緩和するが完全ではない。
- 接続関係(トポロジー)の難しさ。どの車線がどの車線に繋がるか(交差点の流れ)を正しく出すのは形状推定よりさらに難しく、未解決度が高い。
- そのため実運用では「オンライン推定」と「軽量な事前地図(SDマップ)」を併用し、互いの弱点を補う設計が現実的とされています。

## 発展トピック・研究の最前線

- 車線トポロジー推定: 形状だけでなく「車線同士・車線と信号や標識の接続関係」をグラフとして出す方向(交差点の走行に必須)。
- SDマップ事前情報の活用: 粗いナビ地図を条件として与え、推定の安定性と認識距離を改善する。
- 時間的・地図的メモリ: 過去に推定した地図を保持・更新し、長距離走行で一貫した地図を維持する。
- エンドツーエンド運転への統合: 地図構築を独立モジュールにせず、知覚から計画まで一体で学習する流れ。地図を中間表現として共有する設計が研究されています。
