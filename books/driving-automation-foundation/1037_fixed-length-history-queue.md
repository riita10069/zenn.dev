# 固定長履歴キュー(queue_len)

## ひとことで言うと
「過去をどれだけ覚えておくか」を一定の本数 `queue_len` に固定し、新しいフレームが来たら一番古いフレームを捨てて常に直近 T フレームだけを保持する仕組みです。BEVFormer の固定長 BEV キューに代表される設計で、計算量とメモリを一定に抑えながら時系列の手がかりを使えるようにします。リアルタイムで動き続ける知覚システムにとって、1フレームあたりのコストが一定であることは死活的に重要で、固定長キューはそれを保証する標準パターンです。

## 直感的な理解
あなたが運転しながら過去の出来事を覚えておくとして、走り続けるほど記憶がどんどん増え続けたら、頭がパンクします。実際の人間は「直近の数秒」だけを鮮明に覚え、古いものは自然に忘れます。これが固定長の発想です。

機械でも同じ問題が起きます。すべての過去を覚えると、時間が経つほど履歴が際限なく長くなり、計算量とメモリが青天井に増えます。特に self-attention は系列長の2乗に比例して重くなるので、無制限の履歴は実用になりません。自動運転は毎秒10フレーム以上を処理し続けるため、1時間走れば数万フレーム、フレームごとのコストが一定でないと、走り続けるうちに必ず処理が追いつかなくなります。「直近の固定本数だけ覚える」という割り切りが、この破綻を防ぎます。固定長キューは「忘れること」を機能として組み込んだ設計だと言えます。

## 基礎: 前提となる概念
「キュー（queue）」はスーパーのレジ待ち行列のように、先に入ったものが先に出ていく（FIFO: First In First Out）データの並びです。新しいものを後ろに追加し、古いものを前から取り出します。「固定長キュー」は、その行列の最大人数を決めておき、超えたら先頭から押し出す、という制約を加えたものです。長さを `queue_len` に固定すれば、何フレーム処理しても行列の中身は常に `queue_len` 本で、1回あたりのコストが一定になります。実装としては「リングバッファ（環状の固定長配列。末尾まで書いたら先頭に戻って上書きする）」がよく使われます。

ここで「計算量がどう効くか」を押さえます。self-attention は系列の各要素が他の全要素と関連度を計算するため、要素数 T に対して T×T の総当たり、つまり計算量は O(T²)（T の2乗のオーダー）です。T を半分にすれば計算は4分の1、4分の1にすれば16分の1になります。だから「どこまで遡るか」は性能だけでなく速度を直接左右する設計パラメータです。メモリも同様で、attention 行列は T×T の大きさを持つため、長い履歴はメモリも2乗で食います。

もう一つ、再帰型（RNN/GRU）との対比を理解しておくと立体的になります。再帰型は隠れ状態という1本のベクトルに過去をすべて要約し続けるので、メモリ消費は履歴長に依存せず一定です。ただし古い情報ほど薄れて「忘れる」性質があります。一方、固定長キュー＋attention は「直近 N 本を明示的に全部見る」ため、N の範囲内では何も忘れませんが、N より古い情報は完全に消えます。前者は「実質無限長だが曖昧」、後者は「範囲内は完璧だが範囲外はゼロ」という対照的な記憶の形をしています。この対比は、メモリ設計の根本的なトレードオフ（圧縮による曖昧化 vs 切り捨てによる範囲制限）を示しています。

## 仕組みを詳しく
固定長キューを attention 型の時系列メモリに組み込むには、キュー長を制御するパラメータ `queue_len`（整数、または「制限なし」を表す `None`）を用意します。挙動は次の2通りです。

- `queue_len=None`: 切り詰めず、与えられた全履歴を使う。
- `queue_len=N`（正の整数）: 直近 N ステップだけ残す。

実装上の本質は、連結済みの履歴テンソル `joint`（形 `[B, T, D]`、B=バッチ、T=時間ステップ数、D=特徴次元）に対して末尾 N 本を取るスライス操作です。

```
if queue_len is not None:
    joint = joint[:, -queue_len:, :]
```

履歴は「古い→新しい」の順に並んでいるので、末尾から `queue_len` 本を取る `[:, -queue_len:, :]` で「一番新しい N フレーム」が残ります。数値例で追います。T=64（過去64フレーム）が入ってきたとき、

- `queue_len=None` なら `[B, 64, D]` のまま attention にかける。
- `queue_len=8` なら `[B, 8, D]` に切り詰めてから attention にかける。古い56フレームは捨てられる。

```
全履歴:   [ f0 f1 f2 ........................ f62 f63 ]   T=64
queue_len=8:                          [ f56 ... f63 ]   末尾8本だけ
```

後段の self-attention は系列長の2乗で重くなるので、64→8 に切り詰めると計算量はおよそ (64/8)² = 64倍軽くなります。逆に言えば、覚える範囲を狭めるほど速くなるが遠い過去は見えなくなる、という直接のトレードオフがここに現れます。

実用上の注意として、`queue_len` は最低でも1にする必要があります（少なくとも1フレームは見ないと話にならない）。1未満を渡したらエラーにするのが安全な実装です。また `queue_len=1` は事実上「最新フレームだけ」になり、NoMemory（[NoMemory（時系列メモリ無しのベースライン）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0989_no-memory-default-baseline)）とほぼ同じ振る舞いになります。つまり `queue_len` は「履歴ゼロ（=1）」から「全履歴（=None）」までを連続的につなぐノブだと理解できます。

なお、履歴がすでに長さ T のテンソルとしてまとめて渡される構成では、`queue_len` は「リングバッファのように毎フレーム push/pop する逐次キュー」ではなく、「与えられた履歴テンソルのうち末尾何本を attention 対象にするか」という上限の指定として働きます。固定長キューの思想（直近 N 本だけ使い、コストを一定に保つ）を、テンソルのスライスで表現していると理解すると正確です。オンラインで毎フレーム処理する本番システムでは、実際にリングバッファで直近 N 本を保持し、新フレームが来るたびに最古を捨てる実装になります。学習時（オフライン、バッチでまとめて処理）と推論時（オンライン、1フレームずつ）で「見ている範囲」を一致させることが、学習と本番の挙動を揃えるうえで重要です。

## 手法の系譜と主要論文
- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270): 過去の BEV 特徴を固定長のキューに保持し、現在フレームがその直近数フレームを temporal self-attention で参照します。固定長にする理由は「無制限に過去を溜めると計算・メモリが増え続けリアルタイムに回らないから」。効果は一定コストで速度推定や遮蔽耐性を得ること。トレードオフはキューが短いと長期文脈を失い、長いと重くなる点です。`queue_len` はまさにこのキュー長を露出させたパラメータに当たります。

- SOLOFusion (Park et al., ICLR 2023, arXiv:2210.02443): 「履歴は長いほど精度が上がるが計算が増える」という固定長キューの本質的トレードオフを定量化し、短期は高解像度・長期は低解像度で持つ折衷案を示しました。`queue_len` をどの値にするかの判断材料を与える、最も直接的な研究です。長期履歴を低解像度・疎に持つことで、固定の単一 `queue_len` では避けられない「長くすると重い」問題を緩和しています。

- StreamPETR (Wang et al., ICCV 2023, arXiv:2303.11926): 最近フレームのメモリを使い回すことで、フレームごとのコストを一定にしつつ時系列情報を活用する設計を採っています。物体クエリを次フレームへ持ち越す「再帰的な固定長メモリ」とも言え、固定長メモリが現代の時系列知覚で標準的なパターンであることを示します。

- Gu & Dao (2023, "Mamba", arXiv:2312.00752): 状態空間モデルにより、系列長に対し線形コスト（O(T)）で長期依存を扱う方向を示しました。固定長キュー＋2乗 attention の代替として、「長期化したいが O(T²) を避けたい」という固定長キューの動機に対する一つの回答です。

系譜としては、固定長キューは「無制限履歴は破綻する」という現実的制約から生まれた共通解です。BEVFormer がこのパターンを自動運転知覚に定着させ、SOLOFusion がそのトレードオフを定量化し、StreamPETR 系が固定長メモリの中身をより効率的（物体／クエリ単位）にし、Mamba 系がそもそもの2乗コストを線形に落とす、という流れで洗練されてきました。

## 論文の実験結果（定量データ）
固定長キューの効果を最も明快に示すのが SOLOFusion の履歴長アブレーションです。指標は nuScenes の NDS（nuScenes Detection Score、各種誤差をまとめた総合点。高いほど良い）と mAVE（mean Average Velocity Error、速度誤差。低いほど良い）です。

SOLOFusion は履歴長（参照する過去フレーム数）を 1、2、4、8、16 と増やしていくと NDS が単調に上昇し、長期履歴を使うほど精度が伸びることを示しました。報告によれば、短期のみのベースラインから長期履歴版へ移ると NDS がおよそ数ポイント（数%）改善し、速度誤差 mAVE も着実に低下します。これは「`queue_len` を大きくするほど精度が上がる」という固定長キューの性質を、定量的に裏付けた結果です。同時に、フレームあたりの計算量は履歴長に応じて増えるため、論文は短期高解像度＋長期低解像度という非対称な持ち方でコストを抑える工夫を提案しました。注目すべきは、精度の伸びが履歴長に対して逓減（だんだん鈍る）していく点で、ある長さを超えると追加のフレームから得られる情報が薄くなるため、無限に長くする意味は薄いことも示唆しています。

BEVFormer も、参照する過去フレーム数を増やすと速度・遮蔽関連の指標が改善する一方、推論時間とメモリが増えることを報告しています。これらの数値が伝えるのは、`queue_len` が「精度と速度を直接トレードオフするノブ」であり、最適値はタスク・データ・計算予算に依存して決まる、という実務的な教訓です。なぜこの調整が重要かというと、自動運転では「速度推定の正確さ（mAVE）」が安全性に直結する一方、「1フレームあたりの遅延」がリアルタイム性に直結し、両者を同時に満たす点を `queue_len` で探す必要があるからです。

## メリット・トレードオフ・限界
メリット

- フレームあたりの計算量・メモリを一定に保て、リアルタイム処理に向く。走行時間に依存せずコストが安定する。
- attention の2乗コストを `queue_len` で直接コントロールできる。速度予算に合わせて精度を調整できる。
- BEVFormer 由来の実績ある設計パターンで、直近の重要な手がかりに集中できる。
- `queue_len=1` で NoMemory 相当、`None` で全履歴と、単一パラメータで幅広い設定を表現できる。

トレードオフ・限界

- `queue_len` より古い情報は完全に失われる。長期依存（数十秒前の文脈、一度通り過ぎた標識など）に弱い。
- 適切な `queue_len` はデータとタスク依存で、決めるには評価実験が必要。
- 短すぎると時系列メモリの利点が薄れ、長すぎると重くなるという調整の手間が残る。
- テンソルのスライスによる「上限指定」と、毎フレーム push/pop する逐次キューは挙動が一致するが実装が異なる。オフライン評価とオンライン推論で同じ範囲を見ているか、整合性に注意が要る。

研究上の未解決課題として、「固定長で切ると長期文脈を失う」問題への対処があります。SOLOFusion の多時間スケール（短期は密、長期は疎）や、再帰型メモリと固定長キューのハイブリッド、Mamba のような線形コストの長期記憶が、この限界を緩める方向として研究されています。

## 発展トピック・研究の最前線
固定長キューの自然な発展は「効率の良い長期化」です。素朴に `queue_len` を伸ばすと O(T²) で破綻するため、(1) 過去を低解像度・疎な表現で持つ（SOLOFusion の長期ストリーム）、(2) 物体やクエリ単位で過去を伝播させ全フレーム attention を避ける（StreamPETR、Sparse4D 系）、(3) 線形 attention や状態空間モデル（Mamba 系）で系列長に対し線形コストにする、といったアプローチが探求されています。

もう一つの最前線は、固定長キューの長さを学習や状況に応じて動的に変える適応的キューです。直線で単調な走行では短いキュー、交差点や合流のような複雑な状況では長いキュー、というように `queue_len` をシーンに応じて変えられれば、計算予算を必要な場面に集中できます。これは計算効率と安全性を両立させる実用的な方向として注目されています。情報量（たとえば直近フレームの不確実性）に応じてキュー長を伸縮させる、という制御理論的なアイデアとも接続します。
