# 軌道予測のための MLP ヘッド

## ひとことで言うと

カメラ画像から作った「現在の場面の特徴」と「過去の場面の記憶」「過去の自分の動きの記憶」を1本の長いベクトルにつなげ、それを多層パーセプトロン（MLP、全結合層を重ねた最も素朴なニューラルネット）に通して、これから数秒間どう走るか（軌道）を一気に出力する、出力部品（ヘッド）の話です。エンドツーエンド運転で最も古典的かつ基本的な出力設計です。

## 直感的な理解

自動運転モデルは、最終的に「これからどう動くか」を数値で出さなければ車を制御できません。ところが画像を解析した結果（特徴マップ）は、例えば `[B, 256, 8, 8]` のような数値の塊で、人間が見ても何を意味するか分かりません。この塊から「アクセルとハンドルをどう操作するか」という運転の答えを取り出す部品が必要です。それがヘッド（head、本体の特徴抽出器の先につける出力専用の小さなネットワーク）です。

単に現在の画像だけを見て運転を決めると問題が起きます。止まっている車と動き出した車の区別がつかない、急に飛び出した歩行者の動く向きが読めない、といった具合です。人間も「さっきまでどうだったか」を覚えて運転しています。だから軌道予測ヘッドは、現在の視覚特徴に加えて、過去の場面の記憶（visual history）と過去の自分の動きの記憶（egomotion history）を一緒に入力します。これらを全部つなげて MLP に流し込み、未来の運転操作の系列を一度に吐き出すのが、このヘッドの役割です。

## 基礎: 前提となる概念

- MLP（Multi-Layer Perceptron、多層パーセプトロン）= 全結合層（Linear、入力の全要素に重みを掛けて足す層）を活性化関数を挟みながら重ねた、最も基本的なニューラルネットです。任意の連続関数を近似できる表現力を持ちます。
- 回帰（regression）= 連続値を予測すること。分類（カテゴリを当てる）の対義語で、軌道のように実数の系列を出すのは回帰です。
- ヘッド（head）= バックボーン（特徴抽出器）の先に付ける、タスク専用の小さな出力ネットワークです。同じバックボーンに検出ヘッド・セグメンテーションヘッド・軌道ヘッドなどを付け替えられます。
- 活性化関数（activation）= 層と層の間に挟む非線形関数です。これがないと何層重ねても1つの線形変換に潰れてしまい表現力が出ません。GELU や ReLU が代表例で、GELU は入力を滑らかにゲートする関数です。
- Dropout = 学習時に一部のニューロンの出力をランダムにゼロにして、特定の経路に依存しすぎる過学習を防ぐ手当てです。確率（例えば0.25）で落とします。推論時には無効化します。
- egomotion（自車運動）= 自分の車自身の動き（加速度・曲率・速度など）のことです。「ego」は自己の意味で、周囲の物体の動きと区別するために使います。

## 仕組みを詳しく

典型的な軌道予測 MLP ヘッドは、3種類の入力を受け取ります。

- 現在の視覚特徴: 複数カメラを融合した特徴マップ。形は `[B, C, 8, 8]`（C はチャネル数、8×8 は空間の縦横）。
- 視覚履歴（visual history）: 過去の各フレームの視覚特徴を圧縮して並べた履歴ベクトル。形は例えば `[B, 896]`。
- 自車運動履歴（egomotion history）: 過去の自分の動き（加速度・曲率など）の履歴。形は例えば `[B, 256]`。

処理の流れをステップで分解します（C=256 の例）。

ステップ1: チャネル圧縮。
```
reduce_channels = Conv2d(C, 3, kernel=3, stride=1, padding=1)
# [B, 256, 8, 8] → [B, 3, 8, 8]
```
3×3 の畳み込みでチャネル数を一気に3まで減らします。分厚い特徴を軽い3チャネルへ絞り、後段の全結合層が扱える小さなベクトルにする狙いです。

ステップ2: 平坦化。
```
[B, 3, 8, 8] → flatten → [B, 192]   (3×8×8 = 192)
```
これが現在シーンの視覚特徴ベクトルです。

ステップ3: 連結（concat）。
```
[B, 192]（現在視覚） + [B, 896]（視覚履歴） + [B, 256]（egomotion履歴）
→ [B, 1344]   (192 + 896 + 256 = 1344)
```
3種類の入力を横に並べて1本の長いベクトルにします。

ステップ4: MLP 本体。
```
fc1: 1344 → 1344    (同じ幅で一度こねる)
fc2: 1344 → 1164
fc3: 1164 → 128
# 各層の間に GELU と Dropout(p=0.25)
```

ステップ5: 出力。
```
trajectory: [B, 128]
```
これは「64ステップ ×（加速度, 曲率）の2値 = 128」と解釈されます。10Hz（1秒間に10回）で64ステップ先まで、つまり6.4秒先までの運転操作の系列を一度に出力します。軌道を位置（x, y）ではなく（加速度, 曲率）の制御量で表すと、車両の運動学的な制約を満たしやすく滑らかになる利点があります（[加速度・曲率による軌道表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0108_acceleration-curvature-trajectory) 参照）。

ステップ6: 副産物。
```
compress_vision = Linear(192, 14)
# 現在の視覚特徴を長さ14に圧縮して返す
```
これは次のフレームで視覚履歴に積むためのものです。各フレームの視覚特徴を短いベクトルに圧縮して時系列に溜めることで、過去の見え方を軽量に記憶します（[圧縮視覚特徴ベクトルと視覚履歴バッファ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0949_compressed-visual-feature-history) 参照）。

まとめると、このヘッドは「現在の見え方 + 過去の見え方 + 過去の動き」を全部つないで、素朴な全結合の積み重ねで一気に未来の運転を回帰する設計です。

## 手法の系譜と主要論文

- Bojarski et al., "End to End Learning for Self-Driving Cars"（NVIDIA, 2016、arXiv:1604.07316）。通称 PilotNet。カメラ画像から直接ハンドル操作を回帰するエンドツーエンド運転の出発点です。畳み込みで特徴を作り、その後を全結合層（MLP）で操作量に落とす構造は、まさにこの軌道 MLP ヘッドの祖先です。実装によっては全結合層の幅（1164ユニットなど）を PilotNet からそのまま踏襲することがあります。効果はシンプルさと実装の容易さ、トレードオフは説明性の低さと長期予測の弱さでした。

- Codevilla et al., "End-to-end Driving via Conditional Imitation Learning"（ICRA 2018、arXiv:1710.02410）。単純な回帰ヘッドだけだと交差点で「右に行くか左に行くか」のような分岐を学べないと指摘し、ナビゲーションコマンド（直進・右折・左折）で条件分けする工夫を入れました。単一 MLP ヘッドの限界（マルチモーダルな未来を表現しづらい）を明確に示した点で重要です。

- Bansal et al., "ChauffeurNet"（RSS 2019、arXiv:1812.03079）。模倣学習だけでは分布のずれ（学習時に見ない状況での失敗）が起きることを示し、合成した摂動データや補助損失で頑健性を高めました。MLP ヘッドの軌道回帰を実用化するうえでの課題（誤差の蓄積）を浮き彫りにしました。

- Chen et al., "Learning by Cheating"（CoRL 2019、arXiv:1912.12294）。特権情報（鳥瞰図の地図など）で教師モデルを学習し、それを画像ベースの生徒モデルに蒸留する2段階学習を提案しました。軌道を直接出すヘッド設計に過去情報や特権情報を足す効果を議論しており、履歴を連結する発想と通じます。

- Hu et al., "UniAD: Planning-oriented Autonomous Driving"（CVPR 2023 Best Paper、arXiv:2212.10156）。知覚・予測・計画をクエリで一気通貫に繋いだうえで、最終的な計画ヘッドは依然として「特徴を集約して MLP で軌道を回帰する」素朴な構造を残しています。つまり、いくら前段がリッチなアテンションになっても、最後に未来軌道へ落とす出口は本トピックと同じ MLP 回帰の系譜にあるという点が重要です。違いは「MLP に何を食わせるか」で、UniAD では検出・追跡・占有・動き予測を経た計画クエリ（plan query）を入力にします。

- Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving"（ICCV 2023、arXiv:2303.12077）。シーンをベクトル化（車線や物体を点列で表現）して軽量化したうえで、自車の計画はやはり MLP ヘッドで軌道を回帰します。動機は UniAD のラスタ表現が重いことで、VAD はベクトル表現により大幅な高速化と同等以上の安全性を狙いました。出口が MLP 回帰である点は変わらず、入力表現と補助制約（衝突・境界逸脱）の設計で性能を伸ばしています。

系譜としては「画像→操作の単純回帰（PilotNet）→ コマンド条件付き回帰（Conditional IL）→ 補助損失・特権蒸留による頑健化（ChauffeurNet, LbC）→ リッチな知覚クエリを入力にした MLP 計画ヘッド（UniAD, VAD）」と進み、その先で「単一の決定的な軌道を出す回帰」から「複数の可能な未来を確率的に生成する」方向（拡散モデルや Flow Matching を使う軌道生成）へと移っています。注目すべきは、これだけ進化しても「最終的に未来軌道へ落とす出口は MLP 回帰のまま」という設計が長く生き残っていることです。これは MLP ヘッドが単純さゆえに堅牢で、入力さえ良ければ十分機能するからです。

## 論文の実験結果（定量データ）

- PilotNet（2016）は、自律走行率（autonomy、人間の介入なしに走れた時間の割合）を指標とし、実車および シミュレーションで「およそ98%の自律走行率」を報告しました。指標の意味は「総走行時間のうち、人間がハンドルを取らずに済んだ割合」で、100%に近いほど良いです。ただしこれは比較的単純な道路条件での結果で、複雑な交差点や分岐は扱えませんでした。

- Codevilla et al.（2018）は CARLA シミュレータで、コマンド条件なしの素朴な回帰ヘッドが交差点で目的方向を選べず成功率（指定ルートを完走できた割合、success rate、高いほど良い）が大きく落ちることを定量的に示しました。コマンド条件付きにすると交差点を含むタスクの成功率が顕著に改善し、「単一 MLP の表現上の限界」と「条件付けによる解決」を数値で対比しました。これは軌道 MLP ヘッドが抱える「マルチモーダル性を表せない」弱点の直接的な証拠です。

- マルチモーダル性の限界を別角度から見ると、単一の決定的軌道を回帰するヘッドは、右にも左にも行きうる分岐で「両者の平均（=どちらでもない直進や中央）」に潰れる傾向があります。後年の確率的軌道生成（複数モードを出せる手法）との比較研究では、分岐シーンでの軌道の妥当性（minADE/minFDE、複数候補のうち最良のものと正解との平均/最終地点の距離。小さいほど良い）が改善することが繰り返し報告されており、これが回帰ヘッドから生成系ヘッドへ移行する定量的な動機になっています。

- 現代のエンドツーエンド計画では、nuScenes 上で「L2 誤差（予測軌道と実際の軌道の平均ユークリッド距離、メートル、小さいほど良い）」と「衝突率（collision rate、予測軌道が他物体と衝突する割合、%、小さいほど良い）」を1秒・2秒・3秒先で報告するのが定番です。UniAD（2023）は MLP 計画ヘッドを持ちつつ、平均 L2 誤差およそ 1.03m、平均衝突率およそ 0.31% を報告し、当時の単純なベースライン（知覚なしで画像から直接 MLP 回帰する方式）を明確に下回りました（誤差・衝突ともに小さいほど良い）。VAD（2023）はベクトル表現により平均 L2 をおよそ 0.72m まで下げつつ推論を数倍高速化したと報告しています。ここで効いているのは MLP ヘッドそのものの改良ではなく「MLP に渡す入力（知覚クエリ）の質」と「衝突・境界逸脱を罰する補助損失」であり、回帰ヘッドの限界を入力と損失の工夫で押し上げている構図が読み取れます。

- アブレーションの一般的知見として、計画 MLP ヘッドに過去軌道や ego 状態（速度・操舵）を連結すると短期 L2 が下がる一方、それらに過度に依存すると「過去の動きをそのまま外挿するだけ」のショートカット学習に陥り、停止物体への急ブレーキや分岐での誤判断が起きやすいことが報告されています。これは「履歴を入れれば入れるほど良い」わけではないという、入力設計のトレードオフを示します。

これらの数値が示す本質は、「素朴な MLP 回帰ヘッドはシンプルで強力な出発点だが、(1)分岐のような状況依存の選択、(2)複数の正解が併存するマルチモーダル性、の2点で原理的な限界を持つ。ただし入力表現と補助損失を工夫すれば現代の最先端計画でも主力の出口として通用する」ということです。

## メリット・トレードオフ・限界

メリット
- 構造が単純で実装・デバッグが容易。全結合層だけなので推論が高速。
- 現在 + 過去視覚 + 過去動作を明示的に連結するので、停止/発進や動きの文脈をある程度扱える。

トレードオフ・限界
- 1ショットの回帰なので「右にも左にも行きうる」といった複数の正解（マルチモーダル）を表現できず、平均的でぼやけた軌道に潰れやすい。
- チャネルを3まで絞る圧縮は情報を大きく捨てるため、細かい空間情報が落ちる。
- 履歴を固定長ベクトルに丸めて連結するだけなので、長期の時間依存を厳密には扱えない。
- 模倣学習特有の分布シフト（学習時に見ない状態に入ると誤差が雪だるま式に増える）に弱く、補助損失やデータ拡張なしでは頑健性が不足しがち。
- 未解決の課題として、マルチモーダルな未来をどう表現するか（生成系ヘッドへの移行）、長期の時間文脈をどう効率的に保持するか、安全制約をヘッドにどう組み込むか、が研究上の焦点です。

## 発展トピック・研究の最前線

軌道予測ヘッドの研究は、決定的回帰から確率的生成へと移っています。アンカーベースのマルチモーダル予測（複数の候補軌道を出して各々の確率を付ける、MultiPath など）、混合密度ネットワーク（出力を確率分布の混合で表す）、そして最近は拡散モデル（ノイズから軌道を段階的に生成）や Flow Matching（確率流で軌道を生成）を使う手法が、マルチモーダルな未来を自然に表現できる方向として注目されています。時系列の扱いも、固定長ベクトル連結から GRU/LSTM などの再帰や Transformer による系列モデリングへと洗練が進んでいます。これらの後継的なプランナ設計については [GRU 軌道プランナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0127_gru-trajectory-planner) や [Flow Matching 軌道プランナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0123_flow-matching-trajectory-planner) を参照してください。
