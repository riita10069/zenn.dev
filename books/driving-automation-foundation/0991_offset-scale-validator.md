# offset_scale バリデータ — ハイパーパラメータの入力検証と早期失敗

## ひとことで言うと

`offset_scale` という設定値(deformable アテンションで予測したずれ幅を縮める係数)に、おかしな値が混入しないようにする入力チェックの話です。具体的には bool(真偽値)・文字列のような非数値・負の数・NaN(非数)・inf(無限大)をはっきり拒否します。設定値を受け取った瞬間に弾くことで、後段の計算で初めて表面化する原因不明なバグ(損失が NaN になる、学習が発散するなど)を防ぎます。これは「早期失敗(fail fast)」というソフトウェア工学の原則を、機械学習のハイパーパラメータに適用した具体例です。

## 直感的な理解

機械学習モデルには「人間が手で決める設定値」が無数にあります。これをハイパーパラメータと呼びます。学習率(learning rate)、バッチサイズ、層の数などが代表例です。`offset_scale` もそのひとつで、後述する deformable アテンションで「予測したサンプリング位置のずれをどれだけ小さく抑えるか」を決める係数です。

ここで問題になるのは、こういう設定値の多くが「設定された瞬間には何のエラーも出ず、ずっと後の計算で初めて壊れる」ことです。たとえば台所で塩と砂糖を取り違えたとして、容器に入れた瞬間は何も起きません。料理を口に入れて初めて「何かおかしい」と気づきます。そして「なぜまずいのか」の原因を、できあがった料理から逆算するのは難しい。プログラムでも同じで、おかしな設定値が forward 計算(モデルが入力から出力を計算する処理)の奥深くで初めて NaN を生むと、「なぜか損失が NaN」という症状だけが見えて、真犯人の設定値にたどり着くのに何時間もかかります。

入力検証(input validation)の発想は「容器に入れる瞬間に塩か砂糖かを確かめる」ことです。設定値を受け取った構築時(オブジェクトを作る `__init__` の中)に即座に検査し、ダメなら「offset_scale は有限の非負数でなければなりません。受け取った値は -0.1 です」のように、何がどうダメかを名指しするエラーを出して止めます。症状ではなく原因の場所でエラーが出るので、デバッグが劇的に速くなります。

## 基礎: 前提となる概念

このトピックを理解するには、まず deformable アテンションと `offset_scale` の役割を押さえる必要があります。

アテンション(attention)とは、ニューラルネットワークが「入力のどこに注目するか」を学習で決める仕組みです。通常のアテンションは入力の全位置を見比べますが、画像のように位置の数が膨大だと計算量が爆発します。そこで deformable(変形可能な)アテンションは「全部見るのをやめて、注目すべき少数の位置を予測して、そこだけサンプリング(値を読み取る)する」という方針を取ります。

このとき「どこを読むか」を、基準点からの相対的なずれ(offset)としてネットワークが予測します。ずれは2次元のベクトル(横方向 dx、縦方向 dy)で、入力の任意の位置を指すことができます。問題は、学習の初期段階ではネットワークがまだ何も学んでおらず、でたらめに大きなずれを予測しがちなことです。基準点から遠く離れた位置を読むと、勾配(学習を進めるための信号)が安定せず、学習が発散します。

そこで予測したずれに小さな係数を掛けて抑え込みます。式で書くと:

```
offset_used = offset_predicted * offset_scale
```

`offset_scale` が 0.1 なら、予測したずれを 10 分の 1 に縮めます。学習初期の暴れを抑え、徐々に必要な範囲を学べるようにする安定化テクニックです。典型的な実装では 0.1 前後の小さな正の値を既定値にします。

ここで重要なのは、`offset_scale` が**学習の安定性を直接左右するハイパーパラメータ**だということです。だからこそ、おかしな値が入ると深刻な事故になります。

- 負の数(例 -0.1): ずれの符号が反転し、意図と逆方向にサンプリングしてしまう。エラーは出ないが学習が静かに壊れる。最も発見しにくいタイプのバグです。
- NaN(Not a Number、非数): 0/0 や inf-inf などの不定形から生まれる特殊な値。NaN を含む計算結果はすべて NaN になり(NaN は伝播する)、損失も勾配も NaN になって学習が完全に死にます。
- inf(無限大): サンプリング座標が無限大になり、グリッドサンプリング(座標から値を補間して読む操作)が破綻します。
- 文字列(例 "0.1"): 一見数値に見えますが型が違うので、掛け算の瞬間に型エラーになります。ただしエラーが出る場所が「掛け算の行」なので、原因が設定値だと分かりにくい。
- bool(True / False): これが Python 特有の落とし穴です。後述します。

## 仕組みを詳しく

バリデータの典型的な実装は、2段構えのチェックになります。一般化したコードで示します。

```python
import math

def validate_offset_scale(offset_scale):
    # 段1: 型チェック(bool を明示的に除外するのが肝)
    if not isinstance(offset_scale, (int, float)) or isinstance(offset_scale, bool):
        raise ValueError(
            f"offset_scale must be a finite non-negative number, got {offset_scale!r}.")
    # 段2: 値チェック(NaN/inf/負を弾く)
    if not math.isfinite(offset_scale) or offset_scale < 0:
        raise ValueError(
            f"offset_scale must be a finite non-negative number, got {offset_scale!r}.")
```

### 段1: 型チェックと bool の罠

`isinstance(x, (int, float))` は「x が整数または浮動小数点数か」を判定します。文字列 "0.1" や `None` はここで弾かれます。

ここでトリッキーなのが bool です。Python では `bool` 型は `int` 型のサブクラス(派生型)として定義されています。歴史的な経緯で、`True` は内部的に整数 1、`False` は整数 0 として振る舞います。実際に確かめると:

```python
>>> isinstance(True, int)
True
>>> True == 1
True
>>> True * 5
5
```

つまり `isinstance(offset_scale, (int, float))` だけでは `True` / `False` が「整数」として通り抜けてしまいます。`offset_scale=True` が黙って 1.0 として受理されると、設定ミス(本当は 0.1 を書きたかった)に誰も気づけません。これを防ぐため `isinstance(offset_scale, bool)` で明示的に除外します。「数値型であること、かつ bool でないこと」という二重の条件が必要なのです。この落とし穴は Python のハイパーパラメータ検証で繰り返し問題になる古典的なポイントです。

### 段2: 値チェック

型が通っても、値そのものが不正なことがあります。`math.isfinite(x)` は「x が有限か」を判定し、NaN と inf に対しては両方 False を返します(NaN も inf も「有限ではない」ため)。これ一発で NaN と inf を同時に弾けます。残る `offset_scale < 0` で負の数を弾きます。

この2段を通過すると、残るのは「有限で非負の実数」だけです。0 も(非負なので)通過します。0 を許すかどうかは設計判断で、0 にすると deformable のずれが完全に無効化される(基準点しか読まない)ので機能的には縮退しますが、不正値ではないので通す、という割り切りが一般的です。

### 入力ごとの判定表

| 入力 | 判定 | 理由 |
|---|---|---|
| `0.1`(正常) | 通過 | 有限・非負の float |
| `0`(正常) | 通過 | 有限・非負(0 は非負) |
| `True` | 拒否 | bool は明示除外 |
| `"0.1"` | 拒否 | int/float でない |
| `-0.1` | 拒否 | 負の数 |
| `float("nan")` | 拒否 | isfinite が False |
| `float("inf")` | 拒否 | isfinite が False |
| `None` | 拒否 | int/float でない |

### どこで呼ぶか

この検証は、`offset_scale` を受け取るあらゆる構築点(コンストラクタ)の冒頭で呼びます。引数を受け取った直後に検証し、通ったら初めて `self.offset_scale = offset_scale` として保存します。1つの値が複数のモジュール(たとえばプランナと融合モジュール)で使われるなら、それぞれの `__init__` で検証します。検証ロジックは1か所の関数にまとめ、各所はそれを呼ぶ形にすると、ルールの重複(片方だけ直して食い違う事故)を避けられます。

## 手法の系譜と主要論文

`offset_scale` という概念そのものは、deformable アテンションの系譜から来ています。入力検証は工学実務なので「論文」はありませんが、検証対象が生まれた背景を系譜として描きます。

- Dai et al. "Deformable Convolutional Networks" (ICCV 2017, arXiv:1703.06211): 畳み込みのサンプリング位置を固定格子ではなく学習可能なオフセットでずらす、という発想の起点。ここで「予測したオフセットを使ってサンプリングする」枠組みが確立しました。
- Zhu et al. "Deformable DETR" (ICLR 2021, arXiv:2010.04159): この発想をアテンションに持ち込み、各クエリが少数のサンプリング点だけを参照する deformable attention を提案。物体検出 Transformer(DETR)の学習を 10 倍速くし、計算量を抑えました。予測オフセットを安定化のためにスケーリングする実装上の工夫がここで広く知られるようになります。
- Li et al. "BEVFormer" (ECCV 2022, arXiv:2203.17270): 複数カメラの特徴を Bird's-Eye-View(俯瞰図)の格子に集約する際に spatial cross-attention として deformable attention を使用。`offset_scale` 相当の係数で、学習初期にサンプリング点が画像外へ飛ぶのを抑えます。自動運転の知覚で deformable attention が定番化する契機になりました。

入力検証側の設計原則の系譜は次の通りです。

- Meyer "Design by Contract"(Eiffel 言語、1986 年以降; 体系化は "Object-Oriented Software Construction" 1988/1997): 関数やオブジェクトが前提とする入力条件(事前条件、precondition)を入口で明示的に検証する、という考え方。「契約による設計」と訳されます。`offset_scale` の検証は事前条件チェックそのものです。
- Hunt & Thomas "The Pragmatic Programmer"(1999): 「早期に、頻繁にクラッシュせよ(Crash Early)」という格言で fail fast を広めました。壊れた状態のまま処理を続けるより、異常を検知した瞬間に止めて原因を明示する方が、結局デバッグが速いという主張です。

## 論文の実験結果(定量データ)

入力検証自体に定量実験はありませんが、検証対象である offset スケーリングの効果は手法側の論文で測られています。指標の意味とあわせて紹介します。

Deformable DETR(Zhu et al. 2021)では、COCO という物体検出のベンチマークデータセットで評価しています。COCO は約 12 万枚の画像に物体の位置と種類を注釈したデータで、検出精度は AP(Average Precision、平均適合率。検出した枠が正解とどれだけ重なり、かつ取りこぼしが少ないかを 0〜100 で表す指標。大きいほど良い)で測ります。

- 元の DETR は学習に 500 エポック(全データを 500 周)を要し AP は約 42 でした。Deformable DETR は 50 エポックで AP 43.8 を達成し、学習を約 10 倍高速化しました。これは deformable attention が「全位置ではなく少数のサンプリング点だけ見る」ことで効率と収束性の両方を改善した結果です。
- このとき、サンプリングオフセットを適切に抑えること(初期の暴走を防ぐこと)が安定収束に効く、というのが実装上の知見です。オフセットを抑えずに学習すると、初期にサンプリング点が画像外へ飛び、勾配が不安定になって収束が悪化します。

BEVFormer(Li et al. 2022)は nuScenes(自動運転向けの大規模マルチセンサーデータセット。6 カメラ・LiDAR・レーダーを搭載した車両で都市部を走行収集)で評価し、NDS(nuScenes Detection Score、検出精度と位置・速度・属性推定の正確さを総合した 0〜1 のスコア。大きいほど良い)51.7、mAP(mean AP)41.6 を報告しています。これは当時のカメラのみ手法で最高水準でした。spatial cross-attention の deformable サンプリングが、複数カメラの特徴を BEV 格子へ正確に集約するのに寄与しています。

なお `offset_scale` の値そのもののアブレーション(値を変えると精度がどう変わるか)は論文によって明示の有無が分かれますが、実務上は「小さすぎると deformable の柔軟性が活きず、大きすぎると初期発散」という U 字のトレードオフがあり、0.1 前後が経験的な安定領域として使われます。

## メリット・トレードオフ・限界

メリット:
- おかしな設定値を構築時に即座に弾けるので、後段の原因不明な NaN・発散を防げる。エラーが症状の場所(forward の奥)ではなく原因の場所(構築時)で出るため、デバッグが速い。
- bool の混入という Python 特有の落とし穴を明示的に塞いでいる。`True` が黙って 1.0 になる事故を防ぐ。
- エラーメッセージに不正な値そのもの(`{offset_scale!r}` の形で型と値を可視化)を含めるので、何を直せばよいか一目で分かる。

トレードオフと限界:
- 検証は構築時の一度きり。`self.offset_scale` を後から手で書き換えれば検証は効かない(ただし通常の使い方では問題にならない)。
- 「有限・非負」しか見ない。極端に大きい正の値(たとえば 100.0、これは事実上発散を招く)は通してしまう。範囲の上限までは検証していない。上限を設けるかは「設定の自由度」と「安全側に倒す」のトレードオフで、上限を厳しくしすぎると正当な実験設定まで弾く弊害もあります。
- 検証は値の妥当性しか見ず、その値が「良い学習結果を生むか」は保証しない。`offset_scale=0.1` が通っても、それがそのモデル・データに最適とは限りません。これはハイパーパラメータ探索(後述)の領域です。

研究上の未解決課題として、ハイパーパラメータの妥当範囲を人手のルールではなくデータから自動推定する、あるいは学習中に適応的に調整する(たとえばオフセットの大きさを監視して動的に scale を絞る)といった方向があります。固定の検証ルールは安全ですが、最適値を見つける作業は別途必要です。

## 発展トピック・研究の最前線

- ハイパーパラメータ最適化(HPO): `offset_scale` のような値の最適値を自動探索する研究分野。ベイズ最適化(Snoek et al. 2012)、Hyperband(Li et al. 2018)、Population Based Training(Jaderberg et al. 2017)などがあり、Optuna や Ray Tune といったツールで実用化されています。検証は「許される範囲」を定め、HPO は「その範囲で最良の点」を探す、という補完関係です。
- 契約による設計の機械学習への応用: 入力テンソルの shape・dtype・値域を関数の入口で検証する考え方は、近年 typeguard や jaxtyping(配列の shape を型注釈で表す)のようなツールに発展しています。深層学習は shape の不一致や暗黙の型変換でバグが出やすいため、入口検証の重要性が再評価されています。
- 数値安定性の自動診断: NaN の発生源を自動で特定する仕組み(PyTorch の `torch.autograd.detect_anomaly()` など)は、検証をすり抜けた数値崩壊を捕まえる補完手段です。入力検証(予防)と異常検知(事後)を組み合わせるのが堅牢な学習基盤の定石です。
