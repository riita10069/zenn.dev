# MoCo (Momentum Contrast)

## ひとことで言うと
MoCo は、ラベル(正解の名前付け)を一切使わず、大量の画像だけから「良い特徴ベクトル」を取り出す方法を学習する技術です。「同じ画像を少し加工した2枚は似た特徴、別の画像は違う特徴」というルールだけで学ぶ「対照学習(コントラスティブ学習, contrastive learning)」の一種です。その中でも MoCo は「モメンタムエンコーダ(momentum encoder)」と「キュー(queue, 待ち行列)」という2つの仕組みを導入し、巨大なバッチを使えない計算資源の少ない環境でも、大量の負例(似ていないべき相手)を確保して安定して学習できるようにしたのが特徴です。ラベルなし学習が、初めて教師あり事前学習の転移性能を一部のタスクで上回ったことを示した、自己教師あり学習(self-supervised learning, SSL)の歴史的な転換点となった手法です。

## 直感的な理解
あなたが何の知識もなく、外国語のニュース映像を大量に見せられたとします。意味は分からなくても、「同じ人物が別の角度で映っている2カット」は何となく結びつけられ、「まったく別の人」とは区別できるはずです。人間は明示的なラベルがなくても、見た目の連続性や類似性から自然に「同じものは同じ、違うものは違う」という構造を作れます。

対照学習はこれを機械にやらせます。1枚の画像を2通りに加工した2枚は「同じものの別の見え方」なので特徴空間で近くに、まったく別の画像は遠くに配置するよう学習します。ここで効くのが「どれだけ多くの違う相手と比べさせるか」です。たった数枚の中で「これとあれは違う」と言っても区別は粗いままですが、何万枚もの「違う相手」と同時に比べさせると、特徴の境界が鋭く磨かれます。MoCo は、この「大量の比較相手(負例)を、少ない計算資源でどう確保するか」という問題に対する、エレガントな構造的回答です。

ポイントは、MoCo が「アルゴリズムを増やす」のではなく「データ構造(キュー)と更新規則(モメンタム)を足す」だけで問題を解いたことです。学習則そのものはシンプルな分類問題のままで、負例の供給だけを切り離した、という発想の節約が美しい点です。

## 基礎: 前提となる概念

ここで使う用語を先に噛み砕いておきます。

- 教師あり学習(supervised learning): 「この画像は猫」「これは車」という正解ラベルを人間が付け、それを当てるよう学習する方式。ImageNet という有名なデータセットは約128万枚に人手でラベルが付いています。ラベル付けは非常に高コストで、たとえば走行映像は1日で数百万フレームにもなり、すべてに「歩行者」「信号」と人間が付けるのは現実的ではありません。
- 自己教師あり学習(self-supervised learning, SSL): データ自身から擬似的な正解(pretext task, 口実タスク)を自動生成して学ぶ方式。人手ラベルが要りません。
- データ拡張(data augmentation): 画像をランダムに切り抜く(random crop)、色を変える(color jitter)、左右反転する、ぼかす(Gaussian blur)、グレースケール化するなどの加工。1枚から複数の「見え方(view)」を作るのに使います。対照学習の心臓部です。
- 正例(positive pair): 似ているべきペア。MoCo では「同じ画像の別加工版」。
- 負例(negative): 似ていないべき相手。MoCo では「別の画像から作った特徴」。
- 埋め込み・特徴ベクトル(embedding): 画像を数百次元の数値の並びに変換したもの。意味が近い画像ほどベクトルが近くなるよう学習します。
- バッチサイズ(batch size): 1回の学習ステップでまとめて処理する画像枚数。大きいほど多くの負例を一度に使えますが、GPU メモリを食います。
- 線形評価(linear probing): 学習済みの特徴抽出器を凍結し、その上に線形分類器1層だけを教師ありで足して精度を測る、SSL の標準的な評価。特徴そのものの良さを直接測れます。

対照学習の精度は負例数とともに上がることが経験的に知られています。素朴に負例を増やすにはバッチを大きくしますが、たとえば SimCLR(Chen et al., ICML 2020)はバッチ 4096 を多数の TPU で同時に回す必要があり、一般の研究室には手が出ませんでした。MoCo はこの壁を「アーキテクチャの工夫」で越えます。

## 仕組みを詳しく

MoCo の核心は3つの部品です。

### 1. クエリエンコーダとキーエンコーダ(2つのネットワーク)

同じ構造のネットワーク(たとえば画像から特徴を出す ResNet-50)を2つ用意します。

- クエリエンコーダ f_q: 加工画像 A1 を入れて特徴ベクトル q を出す。これは普通に勾配で学習する(backprop で更新)。
- キーエンコーダ f_k: 加工画像 A2 や過去の画像を入れて特徴ベクトル k を出す。こちらは勾配で直接学習しない。

数値で見ます。入力画像が tensor shape [3, 224, 224](3 は RGB チャンネル数、224x224 は縦横ピクセル)だとします。ResNet-50 を通すと global average pooling 後に [2048] の特徴になり、射影ヘッド(projection head、特徴を小さなベクトルに変換する小さな MLP)を通すと [128] の埋め込みが出ます。q も k もこの 128 次元ベクトルです。最後に L2 正規化(L2 normalization, ベクトルの長さを 1 にそろえる操作)して使うので、2 ベクトルの内積がそのままコサイン類似度(角度の近さ、−1〜+1)になります。

### 2. キュー(待ち行列)で負例をためる

ここが MoCo 最大の発明です。負例を大量に使うために、過去のミニバッチで計算したキーベクトル k を「キュー」と呼ばれる固定長の保管庫にためます。

イメージは長さ 65536(=約6万5千)、形 [65536, 128] の保管庫です。

```
新バッチ256個のキー  →  [ 新256 | 古い65280個 ... ]  →  最古256個を捨てる
                          ↑ FIFO(先入れ先出し, First-In First-Out)
```

- 新しいミニバッチ(たとえば256枚)を処理するたびに、その256個のキーをキューに入れる(enqueue)。
- 同時にキューの最古256個を捨てる(dequeue, FIFO)。

こうすると、1ステップで実際にネットワークに通すのは256枚だけなのに、負例としては「キューにたまった65536個」を使えます。バッチサイズと負例数を切り離したわけです。これにより小さなバッチでも大量の負例で学習できます。キューは勾配計算の対象外(単なる保存済みベクトル)なので、6万個保持してもメモリ負荷は埋め込みベクトルの分だけ(65536×128×4バイト ≒ 33MB)で済みます。これは ResNet を6万回 forward する計算量とは比較にならないほど軽量です。

### 3. モメンタムによるキーエンコーダの更新

キューには「過去の」キーが入っています。もしキーエンコーダが学習のたびに大きく変わると、キュー内の古いキーは「古いエンコーダが出した特徴」になり、今のクエリと比べても整合が取れなくなります(不整合, inconsistency)。クエリは最新のエンコーダ、キューの中身はバラバラの世代のエンコーダ、という状態では「似ている・似ていない」の比較が意味を失います。

そこで MoCo は、キーエンコーダを勾配で直接更新せず、クエリエンコーダのパラメータをゆっくり真似させます。これがモメンタム更新です。

```
θ_k ← m · θ_k + (1 − m) · θ_q
```

- θ_k はキーエンコーダのパラメータ、θ_q はクエリエンコーダのパラメータ。
- m はモメンタム係数で、論文では 0.999 という 1 に極めて近い値。
- 毎ステップ「99.9%は今のまま、0.1%だけクエリに寄せる」という超ゆっくりな更新になる。

論文のアブレーションでは m=0.999 が最良で、m=0.9 に下げると精度が大きく落ち(約 6 ポイント低下と報告)、m=0(毎回クエリをそのままコピー=普通のシャム型)では学習が崩壊しました。これは「キーエンコーダが滑らかに進化する辞書(dictionary)である」ことが整合性に決定的だと示しています。あまりに速く変わると辞書がブレ、固定されると古さで使えなくなる。0.999 はその絶妙なバランス点です。

### 4. 損失関数(InfoNCE)

学習目標は、クエリ q を対応する正しいキー k+(同じ画像の別加工)に近づけ、キュー内の負例 k− から遠ざけることです。これを数式にしたものが InfoNCE 損失(Oord et al., 2018 で定式化された対照学習の標準損失)です。

```
L = −log( exp(q·k+ / τ) / ( exp(q·k+ / τ) + Σ_{k−} exp(q·k− / τ) ) )
```

- q·k は2ベクトルの内積(L2 正規化済みなのでコサイン類似度。似ているほど大きい)。
- τ(タウ)は温度パラメータ(temperature)で論文では 0.07。小さいほど「正解とそれ以外の差」を強調し、勾配が鋭くなる。
- 分子は正例との類似度、分母は正例+全負例の類似度の合計。
- 実質「65537クラス分類(正例1個+負例65536個の中から正解を1つ当てる)」と同じ形で、ソフトマックス・クロスエントロピーとして計算できます。

τ の役割を数値で見ます。q·k+=0.9、ある負例 q·k−=0.7 のとき、τ=1 なら exp(0.9)/exp(0.7) の比は約 1.22 倍にしかなりませんが、τ=0.07 なら exp(0.9/0.07)/exp(0.7/0.07)=exp(2.86)≈17 倍になります。小さい τ がわずかな類似度差を大きな確率差に拡大し、「ギリギリ似ている難しい負例(hard negative)」を強く押し離す効果を生みます。

### 全体の流れ

```
画像 A ──拡張──> A1 ──f_q──> q ┐
        └─拡張─> A2 ──f_k──> k+ ┘── InfoNCE(q, k+, キュー65536個) ── 勾配 ─> f_q更新
                                  キュー ─ 負例 k−                  └ モメンタム ─> f_k更新
                                                                   k+ をキューへ enqueue、最古を dequeue
```

学習が終わったら、キーエンコーダ・キュー・射影ヘッドは捨て、クエリエンコーダ本体(ResNet など)を「事前学習済み特徴抽出器」として下流タスク(分類・検出・セグメンテーション)に使います。射影ヘッドを捨てるのは、ヘッド直前の特徴のほうが下流で転移性能が高いという経験則によります。

なお実装の落とし穴として BatchNorm のリーク(BN leakage / information leakage)があります。バッチ正規化はミニバッチ内の統計(平均・分散)を共有するため、その統計を通して正例の情報がモデルに漏れ、「画像の中身を見ずにバッチ統計でカンニング」して学習が崩れることがあります。MoCo は ShuffledBN(Shuffling BN, 複数 GPU 間でサンプルの並びをシャッフルしてからキーエンコーダに通し、BN 統計のリークを防ぐ)を導入してこれを回避しています。

## 手法の系譜と主要論文

対照学習の系譜は、負例をどう確保するかをめぐる試行錯誤の歴史として読めます。

### 前史 — メモリバンク vs end-to-end

MoCo 以前は2つのアプローチがありました。(1) メモリバンク方式(Wu et al., CVPR 2018, Non-Parametric Instance Discrimination)は全データ(全画像)の特徴を1つずつテーブルに保存して再利用しますが、各特徴は最後にそのサンプルが現れたとき(数エポック前かもしれない)に更新されたきりで、時間が経って大きな不整合を抱えます。(2) end-to-end 大バッチ方式(SimCLR の前身的発想)はバッチ内の他サンプルをその場で負例にするので整合は完璧ですが、負例数=バッチサイズに縛られ高コスト。MoCo はこの両者の良いとこ取り(メモリバンクの大量性 + end-to-end の整合性)を狙った設計として提案されました。キューが「大量性」を、モメンタムが「整合性」を担います。

### MoCo v1 — He et al., CVPR 2020「Momentum Contrast for Unsupervised Visual Representation Learning」

著者は Kaiming He ら(Facebook AI Research, FAIR)。上記のキュー+モメンタムエンコーダの枠組みを提案。動機は「メモリバンクの整合性の悪さ」と「大バッチの高コスト」の両方を、キューでバッチと負例数を切り離し、モメンタムで整合を保つことで同時に解決する点にありました。新規性は「辞書(負例集合)を、滑らかに進化する動的キューとして捉えた」概念的フレーミングにあります。

### SimCLR — Chen et al., ICML 2020「A Simple Framework for Contrastive Learning」

MoCo とほぼ同時期の Google Brain の手法(Ting Chen, Geoffrey Hinton ら)。キューを使わず、巨大バッチ(最大 8192、典型 4096)内の他サンプルを負例にします。代わりに(1)強いデータ拡張の組合せ(特に random crop + color distortion が決定的)、(2)非線形 MLP 射影ヘッド、(3)大バッチ・長時間学習が精度に決定的だと体系的に示しました。MoCo の「少資源」とは対極ですが、ここで確立された拡張・射影ヘッドの知見が次の MoCo v2 に逆輸入されます。両者は競合というより相補的な「教科書」になりました。

### MoCo v2 — Chen et al., 2020「Improved Baselines with Momentum Contrastive Learning」

SimCLR の知見を MoCo に取り込んだ短い技術報告。具体的には(1)射影ヘッドを線形1層から 2層 MLP(隠れ層 + ReLU)へ、(2)データ拡張を強化(color jitter を強め、Gaussian blur を追加)、(3)コサイン学習率スケジュール(cosine LR schedule)。動機は「SimCLR の高精度を、MoCo の小バッチ・低コストのまま追いつかせる」こと。本質構造(キュー+モメンタム)は変えていません。

### MoCo v3 — Chen et al., ICCV 2021「An Empirical Study of Training Self-Supervised Vision Transformers」

バックボーンを ResNet から Vision Transformer(ViT、画像を 16x16 などの小パッチに分けてトークン列として Transformer で処理するモデル)へ替えた研究。興味深いことにキューを廃止し、end-to-end 方式(バッチ内負例)に戻しつつ、対称化損失(クエリとキーの役割を入れ替えた損失も足す)と予測器(predictor、BYOL 由来の追加 MLP)を追加しました。動機は「台頭する ViT を自己教師ありで安定学習するレシピを確立すること」。ViT の SSL 学習は不安定で、学習曲線が突然跳ねて精度が落ちる現象(training instability)が観測されましたが、論文は「パッチ埋め込み層(patch projection, 最初の線形射影)を学習せずランダム初期値で凍結すると劇的に安定する」という実用的な発見を報告しました。ここで MoCo の系譜は [BYOL (Bootstrap Your Own Latent)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0033_byol_note) / DINO といった「予測器・自己蒸留」系と合流していきます。

## 論文の実験結果(定量データ)

評価の中心は ImageNet 線形評価(linear probing)です。事前学習で得た特徴抽出器を凍結し、その上に線形分類器1層だけを教師ありで学習して Top-1 精度(正解クラスが最上位スコアに来た割合)を測ります。特徴自体の良さを直接測れるのが利点です。

- MoCo v1(ResNet-50, 200 epoch): ImageNet 線形評価で約 60.6%。同時期のメモリバンク系や他の SSL を上回りました。
- MoCo v2(ResNet-50): 200 epoch で約 67.5%、800 epoch で約 71.1%。v1 から大幅改善し、SimCLR(同 200 epoch・条件で約 66.6%)をバッチ256という小さな構成で上回りました。SimCLR がバッチ4096を要したのに対し、MoCo v2 は 8〜16GB GPU 級の現実的な構成でこれを達成した点が実務的に重要です。
- 転移性能(MoCo v1): PASCAL VOC 物体検出や COCO 検出・インスタンスセグメンテーションで、ImageNet 教師あり事前学習を上回るか同等の AP(Average Precision、検出の正確さの指標。検出枠と正解枠の重なりがしきい値以上で正解とし、適合率と再現率の調和で測る)を示しました。たとえば VOC 検出で教師あり事前学習より AP が数ポイント高い結果が報告されています。これが「ラベルなし事前学習が教師ありを下流タスクで上回りうる」ことを初めて広く示した結果で、分野に大きな衝撃を与えました。
- MoCo v3(ViT-B/16, 300 epoch): ImageNet 線形評価で約 76% 前後、ViT-L ではさらに高い精度。ViT の安定 SSL レシピを確立しました。

アブレーションの要点。(a)キューの長さ K を 256 → 16384 → 65536 と増やすほど精度が単調に上がり、負例数が直接効くことを示しました(ただし飽和傾向もある)。(b)モメンタム m は 0.999 が最良で、0.99 ですでにやや低下、0.9 では数ポイント低下、0(コピー)では崩壊。(c)温度 τ は 0.07 付近が良く、大きすぎると正負の区別が鈍り、小さすぎると勾配が過敏になります。(d)MoCo v2 のアブレーションでは MLP ヘッド追加で約 +6%、Gaussian blur を含む強い拡張で追加の改善が得られ、「射影ヘッドと拡張が精度の大部分を担う」ことが示されました。この「ヘッドと拡張が効く」という発見は、後の SSL 全般に通底する重要な経験則です。

## メリット・トレードオフ・限界

メリット:
- 人手ラベル不要で、大量の未ラベル画像から良い特徴を学べる。ラベル付けコストの壁を回避できる。
- キュー+モメンタムにより小バッチ・少 GPU でも大量の負例で学習でき、計算資源の少ない環境に優しい。これが SimCLR との決定的な実務差。
- 分類・検出・セグメンテーションへの転移性能が高く、特に密予測(検出系)では教師ありを上回ることもある。

トレードオフと限界:
- 性能が負例の質と量に依存し、温度 τ・モメンタム m・キュー長 K など調整項(ハイパーパラメータ)が多い。
- データ拡張の設計が性能を大きく左右する。弱すぎると課題が簡単すぎて学べず(正例どうしが見た目そっくりで自明)、強すぎると正例が別物になり崩壊に近づく。拡張設計には人間の事前知識(invariance の選択。たとえば「色は変わっても同じ物体」という仮定)が入り込み、その妥当性はドメイン(自然画像か走行映像か医療画像か)で変わります。たとえば色ジッタは自然画像では有効でも、信号機の色が意味を持つ走行ドメインでは破壊的になりうる。
- BatchNorm リークなど実装上の落とし穴があり ShuffledBN 等の対策が要る。
- 偽陰性(false negative)問題: 「実は同じカテゴリの別画像(別の柴犬の写真など)」を負例に混ぜると、本来近いはずのものを無理に遠ざけてしまい特徴が劣化する。負例をランダムに選ぶ限りこれを完全には避けられず、対照学習一般の未解決課題です。これが「負例なし」手法 [BYOL (Bootstrap Your Own Latent)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0033_byol_note) が生まれた直接の動機でもあります。
- インスタンス識別という pretext task は「各画像は1つの独自クラス」と仮定するため、本来意味的に近い画像どうしの関係を明示的には学べない、という原理的限界もあります。

## 発展トピック・研究の最前線

MoCo の後、自己教師ありは大きく3方向に分岐しました。(1)負例を捨てる予測ベース系([BYOL (Bootstrap Your Own Latent)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0033_byol_note)、SimSiam(Chen & He, CVPR 2021)、DINO(Caron et al., ICCV 2021))。負例の偽陰性問題と大量性の要求を構造的回避で解決します。(2)クラスタリングを併用する系(SwAV, Caron et al., NeurIPS 2020。オンラインクラスタ割当を予測しあう)。(3)生成・再構成ベース系(MAE = Masked Autoencoders, He et al., CVPR 2022。画像の 75% を隠して残りから復元させる。対照学習とは別系統だが ViT で極めて強く、拡張への依存も小さい)。さらに DINOv2(Oquab et al., 2023)は対照・自己蒸留・正則化を統合し、凍結特徴のまま深度推定・セグメンテーション・対応点マッチングなど多様な密予測タスクで強い汎用視覚基盤モデルとなりました。対照学習の枠組みは画像とテキストにも拡張され、[CLIP 画像エンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0372_clip-image-encoder) や [SigLIP](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0407_siglip_note) へと発展します。研究の最前線では「拡張に依存しない SSL」「ドメイン特化データ(走行映像・医療・衛星)での SSL の有効性」「SSL 特徴の理論的理解(なぜ転移するのか、何の不変性を学んでいるのか)」が引き続き重要なテーマです。
