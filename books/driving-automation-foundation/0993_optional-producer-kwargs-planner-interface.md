# kwargs 経由のオプショナル入力でプランナ I/F を不変に保つ設計

## ひとことで言うと

経路を計画する部品（プランナ）の呼び出し方（インターフェース、略して I/F）を一切変えずに、新しい追加情報を「余った引数（kwargs）」として渡せるようにする設計のことです。新しい機能を足しても、その部品を使っていた古いコードはそのまま動き続けます。機械学習のモデルを長く育てていくと「あとから文脈情報を足したい」場面が必ず来ますが、そのたびに全コードを書き換えるのを避けるための、地味だが極めて実用的なソフトウェア設計の知恵です。

## 直感的な理解

たとえば「住所を渡すと地図を返す」関数があるとします。あるとき「移動手段（徒歩か車か）も考慮させたい」と思ったとして、関数の引数に `transport` を必須で足してしまうと、過去にこの関数を呼んでいた全ての場所が「引数が足りない」というエラーで一斉に壊れます。地図アプリのどこか一箇所を直したいだけなのに、アプリ全体が動かなくなるわけです。

これと同じことが、自動運転モデルの軌跡プランナでも起きます。プランナは「周囲を要約した特徴」を入力に「数秒先の走行経路」を出す関数で、学習コード・推論コード・テスト・評価スクリプトなど無数の場所から呼ばれます。ここに「危険物の検出結果も渡したい」「シーンの意味理解の結果も渡したい」と新しい情報を必須引数で足すたびに、呼び出し側を全部書き直す羽目になります。しかも、その新情報を使わない古い構成まで巻き添えで壊れます。

解決策はシンプルです。必須の入力だけを固定の引数にしておき、それ以外の追加情報は「名前付きで好きに足せる受け皿」に流し込む。これが Python でいう `**kwargs`（キーワード引数の受け皿）です。新情報を使うプランナだけが受け皿の中を覗き、使わないプランナは無視する。こうすれば呼び出し側の契約は永久に変わりません。研究の現場では「とりあえず新しい入力を一つ試したい」という実験が日常的に発生するので、この受け皿があるかないかで実験の立ち上げ速度が桁違いに変わります。

## 基礎: 前提となる概念

理解の前提となる用語を噛み砕きます。

関数のシグネチャ（signature、署名）とは「関数名・引数の並び・型・戻り値」の組み合わせ、つまり「この関数はこう呼べばこう返す」という対外的な約束のことです。シグネチャを変えると、その関数を呼んでいる全箇所が影響を受けます。プログラミングでは、この「対外的な約束」を契約（contract）と呼びます。契約が安定していれば、契約の内側（実装）は自由に変えても外側（呼び出し側）に波及しません。

キーワード引数（keyword argument）とは、`func(x=3)` のように名前を付けて渡す引数です。これに対し `func(3)` は位置引数（positional argument）と呼びます。Python では関数定義の末尾に `**kwargs` と書くと、明示的に宣言しなかった名前付き引数をすべて辞書（dict）として受け取れます。`func(a=1, b=2)` を `def func(**kwargs)` で受けると `kwargs == {"a": 1, "b": 2}` になります。これは「想定外の引数も静かに飲み込める柔軟な受け皿」です。多くの言語に類似の機構があり、JavaScript の rest parameter（`...args`）や可変長引数（variadic arguments）も同じ役割を果たします。

後方互換性（backward compatibility）とは「新しいバージョンが、古い呼び出し方を壊さない」性質のことです。ライブラリやモデルを長期間メンテナンスするうえで最重要の性質の一つです。逆に「壊す変更」を破壊的変更（breaking change）と呼び、利用者全員に修正を強いるため、バージョン番号のメジャー更新で告知するのが慣例です。

producer と consumer という言葉も使います。producer は「情報を作って渡す側」、consumer は「その情報を受け取って使う側」です。本トピックの肝は「producer 側は新機能をオプションとして自由に追加してよいが、consumer（プランナ）の I/F は不変に保つ」という非対称な役割分担にあります。producer の自由度を最大化し、consumer の安定性を最大化する、という二つの要求を同時に満たすのがこの設計です。

抽象基底クラス（abstract base class）も登場します。これは「この種類の部品はこう呼べる、という共通の型枠」を定義するだけで、中身の実装は持たないクラスです。複数のプランナ実装（Bezier 曲線でなめらかな軌跡を出すもの、Flow Matching や拡散で軌跡分布を生成するもの等）が、同じ基底クラスの I/F を共有することで「どれに差し替えても周りのコードが変わらない」状態を作ります。この「差し替え可能性」こそ、研究で複数手法を公平に比較するための前提になります。

## 仕組みを詳しく

中心となる設計を before / after で示します。素朴に引数を増やすと、こうなります。

```python
# 悪い例: 必須引数を足す
def forward(self, bev_features, visual_history, egomotion_history, scene_context):
    ...
```

この変更を入れた瞬間、抽象基底クラスと全プランナ実装と全呼び出し箇所を一斉に書き換える必要が生じます。`scene_context` を渡していなかった既存の呼び出しは「引数が足りない」エラーで全滅します。新情報を使わないプランナ（たとえば単純な Bezier プランナ）まで、無意味な引数を受け取らされます。これは前述のオープン・クローズドの原則に真っ向から反する変更です。

kwargs パターンでは、こう書きます。

```python
# 良い例: 必須は固定、追加はオプショナルな受け皿に
def forward(self, bev_features, visual_history, egomotion_history, **kwargs):
    scene_context = kwargs.get("scene_context")   # 無ければ None
    if scene_context is not None and scene_context.is_reliable:
        # この情報を使うプランナだけが条件づけに利用する
        ...
    # 使わないプランナは kwargs を一切触らない
```

呼び出し側は、追加情報があるときだけ名前付きで足します。

```python
# 追加文脈なし（従来どおり、何も壊れない）
planner(bev_features, visual_history, egomotion_history)

# 追加文脈あり（新しいプランナだけが使う）
planner(bev_features, visual_history, egomotion_history, scene_context=ctx)
```

tensor shape の観点で整理します。必須入力はたとえば BEV 特徴 `[B, C, H, W]`（B はバッチ数、C はチャネル数、H と W は鳥瞰グリッドの縦横）、視覚履歴 `[B, T, D_v]`、自車運動履歴 `[B, T, D_e]`（T は過去フレーム数、D は特徴次元）。これらは固定です。一方 `scene_context` は意味理解モジュールが作る `[B, D_ctx]` の潜在ベクトルかもしれないし、構造化された型付きオブジェクトかもしれない。kwargs はこの「形も種類も将来変わりうるもの」を、I/F を凍結したまま流すための導管になります。具体例で言えば、`D_ctx = 256` の reasoning latent を渡すこともあれば、検出された交通参加者のリストを渡すこともあり、その違いを呼び出し側が意識せずに済みます。

この設計には、もう一段の判断が伴うのが普通です。それは「forward は常に同じ意味の値を返す」という原則です。学習時と推論時で forward の戻り値の意味が変わる（あるときは軌跡、あるときは損失）と、呼び出し側が場合分けを強いられ、バグの温床になります。そこで多くの実装では、推論の forward は常に完成した軌跡 `[B, num_timesteps * num_signals]`（時間ステップ数 × 各時刻の信号数、たとえば x, y, heading）を返し、学習用の目的関数は `compute_loss()` のような別メソッドに分離します。これは kwargs パターンとは独立した話ですが、根は同じ「契約を安定させる」思想です。入力側を kwargs で安定させ、出力側を「意味を変えない」で安定させる、という両側からの規律です。

なぜこの設計が「正しい」と言えるのか。組み合わせ爆発（combinatorial explosion）を防げるからです。プランナの種類が P 通り、追加文脈の有無や種類が K 通りあるとき、もし I/F が文脈ごとに変わるなら P × K 通りの呼び出しパターンを面倒見る必要があります。たとえば P=4、K=5 なら 20 通り。kwargs で I/F を一本化すれば、呼び出しパターンは P 通り（文脈は中で吸収）に抑えられます。この削減は、実装する側の認知負荷にも、テストすべき経路の数にも直結します。

## 手法の系譜と主要論文

この設計はモデルの数式というより、ソフトウェア工学とモデル設計の交差点にある原則です。系譜を追います。

土台はオープン・クローズドの原則（Open-Closed Principle, OCP）です。Bertrand Meyer が "Object-Oriented Software Construction"（1988）で定式化し、「ソフトウェアの実体は拡張に対して開いており、修正に対して閉じているべき」と述べました。つまり「新機能を足すときに既存コードを書き換えなくて済む」設計を良しとする。kwargs によるオプショナル入力は、consumer 側を「修正に対して閉じた」まま、producer 側を「拡張に開く」OCP の具体的な一手段です。Robert C. Martin が "Agile Software Development: Principles, Patterns, and Practices"（2003）で SOLID 原則の頭文字の一つ（O）として再定式化し、広く普及させました。

機械学習側の系譜では、条件づけ（conditioning）の歴史と結びつきます。Codevilla et al. "End-to-end Driving via Conditional Imitation Learning"（2018, ICRA, arXiv:1710.02410）は、運転ネットワークに「右折・左折・直進」という高レベルコマンドを条件入力として注入しました。動機は明快で、同じ交差点画像でも運転手が行きたい方向によって正解操作が変わるため、画像だけからは正解が一意に決まらない（これを「ナビゲーションの曖昧性」と呼びます）。新規性は、追加文脈で出力を分岐させる branched architecture（コマンドごとに別の出力ヘッドを用意する構造）の提案です。これは「追加の文脈入力でモデルの振る舞いを切り替える」という、kwargs パターンが運ぶ中身そのものの原型です。

条件づけの注入方法を洗練させたのが Perez et al. "FiLM: Visual Reasoning with a General Conditioning Layer"（2018, AAAI, arXiv:1709.07871）です。条件ベクトルから各特徴チャネルのスケール γ とシフト β を生成し、`output = γ * feature + β` で特徴を変調します。動機は、条件を単純に連結（concat）するより、特徴空間を直接アフィン変換する方が表現力と学習効率が高いこと。これは「kwargs で運ぶ中身をどう注入すれば効くか」という、本トピックの実利的な続きにあたります。

拡散・フローベース方策での一般化として、Chi et al. "Diffusion Policy: Visuomotor Policy Learning via Action Diffusion"（2023, RSS, arXiv:2303.04137）は、観測を条件としてロボット行動の分布を生成しました。ここでは観測がそのまま条件（conditioning）として denoising（ノイズ除去）の各ステップに注入されます。プランナの forward が「モードによって戻り値の意味が変わってはいけない」という設計上の戒めは、こうした生成方策の実装で繰り返し学ばれてきた教訓です。

この二つの流れ（OCP というソフト工学側、conditional policy というモデル側）が合流した結果が「producer はオプショナル・consumer の I/F は不変」という拡張規約です。同じ規約は時系列メモリの基底クラスにも適用でき、`forward(self, visual_history, egomotion_history, **kwargs)` の形でモデル全体の共通様式になります（[[B,T,feat] 履歴インターフェース](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0936_bt-feat-history-interface) 参照）。

## 論文の実験結果（定量データ）

ソフトウェア設計原則そのものはベンチマークになじみませんが、それが運ぶ「条件づけ」の効果は定量化されています。

Codevilla et al.（2018）は、CARLA シミュレータと実車（小型ロボットトラック）の両方で評価しました。評価指標は「指定ルートを成功裏に完走できた割合（success rate）」と「介入なしで走れた距離」です。条件入力なしのベースライン（画像だけから操作を回帰する素朴な模倣学習）は、交差点で行き先が定まらず、しばしば直進してしまう、あるいは進路を迷う挙動を示しました。コマンドを条件入力として注入した branched モデルは、CARLA の動的ナビゲーションタスクで成功率が顕著に向上したと報告されています。最重要のアブレーション（要素を抜く実験）として、条件を入力の途中に単に concat（連結）する方式より、コマンドごとに出力ヘッドを分岐させる branched 方式の方が成功率が高い、という結果が示されました。これは「追加文脈をどう注入するか」が性能を左右することを示し、kwargs で運ぶ中身の表現・注入方法を慎重に設計すべきという教訓につながります。

FiLM（Perez et al. 2018）は視覚的推論ベンチマーク CLEVR で評価しました。CLEVR は合成画像に対する論理的な質問（「赤い立方体の左にある球と同じ大きさの物体は何個あるか」など）に答えるタスクで、複数の条件を組み合わせた推論能力を測ります。FiLM は全体精度およそ 97.7% を達成し、当時の素朴な条件づけ手法を上回りました。アブレーションでは、γ と β のうち一方だけを使うより両方を使う方が精度が高く、特徴のスケールとシフトの双方による変調が効くことが示されました。これは「条件を単純に連結するだけでなく、特徴を能動的に変調すると効く」という、注入方式の重要性を裏付ける数値です。

Chi et al.（2023）は、シミュレーションと実機ロボットの計 15 タスクで評価しました。比較対象は LSTM-GMM、IBC（暗黙的行動クローニング）、BET など既存の行動方策です。指標はタスク成功率です。Diffusion Policy は既存手法に対して平均でおよそ 46.9% の相対的な成功率向上を報告しました。マルチモーダル（複数の正解動作がありうる状況）や精密操作で特に強く、観測を条件として行動分布を生成する枠組みの有効性を示しました。

これらの数値が共通して示すのは「追加の文脈で条件づけると性能が上がる、ただし注入方式によって効き方が大きく変わる」という事実です。kwargs パターンは、その性能向上をもたらす条件入力を、既存システムを壊さずに後付けできるようにする受け皿だと位置づけられます。受け皿の存在が性能を上げるわけではなく、受け皿のおかげで「効く注入方式を低コストで探索できる」ことに価値があります。

## メリット・トレードオフ・限界

メリット。既存の呼び出し側と既存プランナ実装を書き換えずに新しい入力を足せます（後方互換）。producer を完全オプションにできるため、追加文脈を渡さないデフォルト構成の挙動が一切変わりません。プランナの種類 × 追加文脈の有無という組み合わせ爆発を防げます。研究開発のように「とりあえず新しい入力を試したい」局面で、実験の立ち上げコストを劇的に下げます。

トレードオフと限界。第一に、`kwargs` は型チェックが効きにくいことです。`scene_context` を `scene_ctx` とタイプミスしても静的解析が捕まえてくれず、`kwargs.get("scene_context")` が静かに None を返して「なぜか新機能が効かない」という発見しづらいバグになります。これは実際に研究コードで頻発する事故で、半日デバッグした末にタイプミスだった、という類の損失を生みます。第二に、シグネチャから「何が渡されうるか」が読み取れず、ドキュメントやコメントへの依存が増えます。インターフェースの自己記述性（self-documenting）が犠牲になるわけです。第三に、kwargs は何でも飲み込むため、本来エラーにすべき誤った引数も黙って受理してしまう「過度の寛容さ」が混乱を招きます。

研究上の未解決の論点として、「いつ kwargs の柔軟さを捨てて明示的な型に昇格させるべきか」があります。実験段階では kwargs が便利でも、設計が固まったら TypedDict や dataclass で型付きにし、IDE 補完と静的検査の恩恵を取り戻すのが定石です。柔軟性と安全性のトレードオフをどの開発フェーズで切り替えるかは、定量化しづらく経験則に委ねられている領域です。一般論として、探索フェーズでは柔軟性（kwargs）、安定運用フェーズでは安全性（型）に寄せる、という指針がよく採られます。

## 発展トピック・研究の最前線

条件づけの注入方法は、それ自体が活発な研究分野です。単純な concat に対し、前述の FiLM は条件から各特徴チャネルのスケールとシフトを生成して変調する手法で、画像 QA や強化学習で条件づけを効率化しました。Transformer 系では cross-attention（クロスアテンション、条件を key/value として注入する注意機構）で条件を注入するのが主流で、生成モデルの分野では classifier-free guidance（条件あり予測と条件なし予測を内挿して条件への従い度合いを推論時に調整する技法）も標準になりました。kwargs で運ぶ「中身」をどう表現し、どこでどう注入するかが、まさにこうした研究の対象になります。

ソフトウェア工学側では、依存性注入（Dependency Injection、依存する部品を外から渡す設計）やプラグインアーキテクチャ、レジストリパターン（名前で実装を引く辞書）と組み合わせると、「新しいプランナや新しい文脈プロバイダをコード本体を触らず追加できる」拡張可能なシステムが構築できます。これは大規模な研究コードベースで、複数の手法を公平に比較するための土台になります。実験管理ツール（設定ファイルから構成を組み立てる Hydra のような仕組み）とも相性がよく、「設定だけ変えて手法を切り替える」運用が、契約の安定性に支えられて成立します。
