# nuScenes Map API

## ひとことで言うと
nuScenes(ニュー・シーンズ)は自動運転の代表的な公開データセットで、その中に「地図を扱うためのライブラリ(API)」が同梱されています。車線・歩道・横断歩道・停止線などが「レイヤ(層)」ごとに分かれたベクトル地図を、座標や範囲を指定して取り出したり、自車を中心にした白黒のマスク画像に焼き込んだりできます。とくに「カメラ画像だけから真上視点の地図を予測する」研究(BEV マップセグメンテーション)の評価では、世界中の研究者が同じ条件で性能を比べる事実上の標準基盤になっています。

## 直感的な理解
自動運転車は、カメラやLiDAR(レーザーで周囲の距離を測るセンサ)から「いま周りに何があるか」を理解します。ですが運転判断に本当に使いたいのは、目の前に斜めに映る道路そのものではなく、「真上から見たレイアウト」です。「自分は左から2番目の車線にいて、20m先に横断歩道があり、その先で道が右に分岐している」という俯瞰の地図が頭の中にあると、ハンドルやアクセルをどう操作すべきかが一気に決まります。

人間のドライバーは、前方の景色を見ながら無意識にこの俯瞰図を脳内で組み立てています。これを機械にやらせるのが「BEV マップセグメンテーション」です。BEV は Bird's-Eye-View(鳥の目から見た図、鳥瞰図)の略で、真上から見下ろした表現を指します。セグメンテーションは「画像の1ピクセルずつに、それが何の種類か(道路なのか歩道なのか横断歩道なのか)というラベルを塗り分ける」タスクのことです。つまり「カメラ画像を入力に、各ピクセルが何かを塗り分けた真上視点の地図を出力させる」という課題です。

ここで困るのが「正解の地図がないと採点できない」ことです。AIが描いた俯瞰地図が当たっているかどうかは、その瞬間に自車の周りが実際どうなっていたか(本物の車線・横断歩道の配置)と比べないと分かりません。nuScenes はこの「本物の地図」を、緻密なベクトル地図として後から追加で提供し(map expansion パック)、それを座標指定で簡単に切り出せる API を用意しました。これにより誰でも「自車周辺の正解BEV地図」を同一手順で作れるようになり、手法の公平な比較が成り立つようになったのです。

## 基礎: 前提となる概念

理解に必要な言葉を先に噛み砕いておきます。

ベクトル地図とラスタ地図。ベクトル地図は、地図の要素を「点の座標」と「それを結ぶ線・面」で表します。たとえば横断歩道は四隅の座標で囲まれた多角形(ポリゴン)、車線中心線は点を並べた折れ線(ポリライン)です。データが軽く、拡大しても劣化しません。一方ラスタ地図は写真と同じく「ピクセルの格子(マス目)」で表します。ニューラルネット(脳の神経をまねた計算装置、以後「モデル」)が直接食べられるのはこのラスタ形式です。nuScenes が持っているのはベクトル地図なので、学習・評価に使うにはラスタ化(マスク画像化)という変換が要ります。

レイヤ。地図要素を種類ごとに別の層に分けて持つ考え方です。横断歩道だけの層、車線だけの層、というふうに分かれていると、必要な種類だけを取り出せます。

マスク画像。ある層について「ここは横断歩道なら1、そうでなければ0」と各ピクセルを0か1で塗った白黒画像です。複数のレイヤをそれぞれマスク化して重ねると、形状 `(レイヤ数, 縦, 横)` のテンソル(数値を多次元に並べた配列)になり、これがモデルの正解になります。

座標系。地図は通常、その都市内で固定された平面座標(原点からのメートル)で表されています。一方センサや自車は刻々と動きます。だから「いまの自車位置・向き」を地図に重ねて、自車中心の切り出しをする必要があります([自車座標系へのラスタ化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0415_ego-pose-rasterization-coord) で詳述)。nuScenes ではこの「いまの自車位置・向き」を、各サンプルに付随する `ego_pose`(平行移動の3次元ベクトルと、向きを表すクォータニオン)として持っています。クォータニオンとは、3次元の回転を4つの数で表す方法で、向きをジンバルロック(特定姿勢で自由度が失われる現象)なしに扱える表現です。

評価指標 IoU。Intersection over Union(共通部分 ÷ 和集合)の略で、予測した領域と正解の領域がどれだけ重なっているかを 0〜1 で測ります。完全一致なら 1.0、まったく重ならなければ 0.0 です。式で書くと、予測領域を P、正解領域を G とすると IoU = |P∩G| / |P∪G| で、分子が「両方とも塗った画素数」、分母が「どちらか一方でも塗った画素数」です。BEV セグメンテーションでは多くの場合この IoU をクラスごとに計算し、平均(mIoU)を主指標にします。なぜ単純な「正解率(accuracy)」を使わないかというと、BEV では道路や背景が画面の大半を占めるため、車線のような細い要素を全く塗らなくても正解率は高く出てしまう(クラス不均衡)からです。IoU は塗りすぎ(偽陽性)も塗り足りなさ(偽陰性)も両方ペナルティにするので、細い要素の評価に向いています。

## 仕組みを詳しく

nuScenes Map API の中心は `NuScenesMap` というクラスです。地図は4都市(ボストン・シーポート、シンガポールのワン・ノース、クイーンズタウン、ホランド・ビレッジ)ごとに用意され、それぞれに対応するインスタンスを作ります。主なレイヤは次の通りです。

```
drivable_area   走行可能領域(車が走ってよい面)
road_segment    道路区間
road_block      道路ブロック
lane            車線(面)
ped_crossing    横断歩道
walkway         歩道
stop_line       停止線
carpark_area    駐車場
road_divider    道路中央の分離線
lane_divider    車線間の境界線
traffic_light   信号機(位置)
```

これらのレイヤは内部的に2種類に分かれます。多角形で表される「面の要素」(drivable_area, lane, ped_crossing, walkway など)と、折れ線で表される「線の要素」(road_divider, lane_divider など)です。マスク化のときは前者を塗りつぶし、後者を一定の太さの線として描きます。map expansion v1.3 ではさらに、車線の中心線をつないだ有向グラフ(どの車線からどの車線へ進めるかの接続関係)も提供され、軌跡予測や経路計画にも使えるようになりました。

API でできる代表的な操作は次の3つです。

第一に点問い合わせ。`layers_on_point(x, y)` のように座標を渡すと、その点がどのレイヤに属するか(例: ここは drivable_area か)を返します。

第二に範囲取得。矩形パッチ `(x_min, y_min, x_max, y_max)` を渡すと、その範囲に含まれる各レイヤの要素(多角形・折れ線)を取り出せます。内部では空間インデックス(R-tree のような、矩形範囲での絞り込みを高速化するデータ構造)を使い、都市全体の数万要素から該当範囲だけを素早く拾います。

第三にマスク生成。これが評価で最もよく使う機能です。自車の位置と向き、切り出すパッチの大きさ、解像度を与えると、自車中心の白黒マスクをレイヤごとに生成してくれます(`get_map_mask` 系の関数)。

正解BEV地図を作る具体例を、数値を入れて追ってみます。ある瞬間の自車が地図座標で (x=965.3, y=1703.1)、向き(heading)が約0.8ラジアン(北から反時計回りに約46度)だとします。

1. 自車を中心に、前後左右それぞれ 50m、つまり 100m × 100m の正方形パッチを地図から切り出す。
2. 自車の進行方向が画像の上を向くようにパッチを回転させる([自車座標系へのラスタ化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0415_ego-pose-rasterization-coord) と同じ座標変換)。
3. 解像度を 0.5m/px に決めると、100m ÷ 0.5m = 200 なので 200×200 ピクセルのマスクになる。
4. レイヤごとにマスク化すると、たとえば6レイヤ選べば `(6, 200, 200)` のテンソルが得られる。

この `(6, 200, 200)` が「正解のBEV地図」です。AIがカメラ画像から予測した同じ形 `(6, 200, 200)` の出力と1ピクセルずつ比較し、レイヤごとに IoU を計算します。before/after でイメージすると、before は緯度経度や都市座標でバラバラな向きのベクトル地図、after は「自車が常に中央・前方が上」に正規化された200×200の白黒6枚、という具合です。

なぜ解像度や範囲を自分で決めるのか。これは手法によって都合が違うからです。広い範囲を見たい手法は 100m×100m を取り、近距離精度を重視する手法は 30m×30m を細かく取ります。研究でよく使われる設定は二系統あり、一つは前方/側方 ±50m を 0.5m/px で 200×200、もう一つは縦100m×横100m を 0.5m とするものです。Lift-Splat-Shoot 系では前後 ±50m を 0.5m/px、CVT(Cross-View Transformer)系では前後 ±50m を 0.25m/px(つまり 400×400)など、論文ごとに微妙に違います。同じデータセットでも設定が違うと数値が比較できなくなるため、論文では「どの範囲・どの解像度・どのレイヤで測ったか」を必ず明記するのが作法になっています。逆に言うと、IoU の数値だけ見て手法を比べるのは危険で、評価設定をそろえて初めて意味のある比較になります。

## 手法の系譜と主要論文

nuScenes Map API は「カメラから俯瞰地図を作る」研究の評価基盤として育ってきました。系譜を追います。

nuScenes 本体(Caesar, Bankiti, Lang, Vora, Liong, Xu, Krishnan, Pan, Baldan, Beijbom; nuTonomy/Motional, CVPR 2020, "nuScenes: A Multimodal Dataset for Autonomous Driving", arXiv 1903.11027)。6カメラ(全周)・1 LiDAR・5レーダーを時刻同期した、当時としては大規模なマルチモーダルデータセットです。1000シーン(各20秒)、約140万枚のカメラ画像、約39万 LiDAR スイープ、140万個の3D物体ボックスを含みます。なぜ作ったか。それまでは KITTI のようにカメラとLiDARが中心の中規模データが主で、レーダーまで含めた全周囲・大規模・多都市のデータが乏しく、マルチセンサ手法を公平に比べられなかったからです。地図はこの数年後に拡張パックとして追加され、Map API が整備されました。

Lift-Splat-Shoot(Philion & Fidler; NVIDIA, ECCV 2020, "Lift, Splat, Shoot: Encoding Images from Arbitrary Camera Rigs by Implicitly Unprojecting to 3D", arXiv 2008.05711)。複数カメラそれぞれについて、各ピクセルの深度(距離)を確率分布として推定し、画像特徴を3D空間に「持ち上げ(lift)」、共通のBEV平面に「叩き落とす(splat)」ことで、カメラだけからBEV特徴を作る手法です。出力は drivable area や車線などのBEVセグメンテーションで、その正解生成に nuScenes Map API が使われました。「カメラのみで俯瞰地図」という路線の出発点であり、後続のほぼすべてが比較対象にする基礎になりました。

CVT / Cross-View Transformer(Zhou & Krähenbühl; UT Austin, CVPR 2022, "Cross-view Transformers for Real-time Map-view Semantic Segmentation")。深度を明示的に推定する代わりに、カメラ視点とBEV視点をアテンションで直接結びつける手法です。カメラの内部・外部パラメータ(較正情報)を位置エンコーディングに織り込み、軽量かつ高速にBEVセグメンテーションを行います。正解は同じく nuScenes Map API のマスクです。

BEVFormer(Li, Wang, Lu, Xie, Li, Wang, Zhou, Dai; Shanghai AI Lab/NJU, ECCV 2022, "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers", arXiv 2203.17270)。学習可能なBEVクエリと変形アテンション(画像の要点だけ見るアテンション)で、複数カメラ・複数時刻の特徴をBEV格子に統合します。空間方向だけでなく時間方向も束ねるのが新規性で、検出とBEVセグメンテーションの両方で精度を押し上げました。

HDMapNet(Li, Wang, Li, Xie, Zhou; Tsinghua/MIT, ICRA 2022, "HDMapNet: An Online HD Map Construction and Evaluation Framework", arXiv 2107.06307)。BEVセグメンテーションの一歩先、すなわち「ピクセル塗り分け」ではなく「車線などのベクトル地図要素そのもの」をその場で推定する課題(オンライン地図構築)を、nuScenes 地図を正解に定量評価する枠組みを初めて整えました。この流れがのちの MapTR 系([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))へつながります。

## 論文の実験結果(定量データ)

ここでは指標の意味と、おおよその数値感を初心者にも分かる形で示します。具体値は論文・条件で揺れるため「報告による」と添えます。

BEVセグメンテーションの主指標は IoU(%)です。たとえば drivable area(走行可能領域)の IoU が 70 なら、予測した「走れる面」と本物の「走れる面」が約7割重なっている、ということです。Lift-Splat-Shoot は nuScenes で drivable area の IoU をおよそ72〜73%、車線(lane boundary)では 20%弱、車両(vehicle)のBEVセグメンテーションでは30%台と報告し、単眼ベースの素朴な手法を明確に上回りました。なぜこの差が重要かというと、drivable area の IoU が数%上がるだけでも「走ってよい面の境界をより正確に引ける」ことを意味し、車線逸脱や縁石への接近の判断に直結するからです。逆に車線や横断歩道の IoU が20%台と低いのは、これらが画面の中で非常に細い領域だからで、IoU は「細い対象ほど数値が出にくい」性質を持つことの典型例です。

CVT は同じ nuScenes 上で、車両セグメンテーション IoU をおよそ36〜37%まで引き上げつつ、推論を数十FPSのリアルタイム級で達成したと報告しています。深度を明示推定する Lift-Splat 系と同等以上の精度を、より軽量に出せた点が貢献です。

BEVFormer は、同じ nuScenes 上で複数時刻の情報を使うことにより、単一時刻のみのベースラインに比べてBEVセグメンテーション・検出ともに数ポイント改善したと報告しています。とくに3D物体検出の総合指標 NDS(nuScenes Detection Score、検出の正確さ・位置・速度・属性などを総合した0〜1の指標)で、当時のカメラのみ手法のトップ級(NDS おおよそ0.56前後)に達しました。アブレーション(構成要素を1つ抜いて効果を測る実験)では、時間方向の融合(過去フレームのBEV特徴を引き継ぐ機構)を外すと、とくに移動物体や遮蔽された領域での性能が落ちることが示されています。これは「1枚の画像では見えない物陰や、止まって見える対象の速度推定に、過去フレームが効く」ことを意味します。

HDMapNet はオンライン地図構築の評価指標として、ピクセルIoUに加え、ベクトル要素同士の一致を測る指標(セマンティックなインスタンス検出 AP など)を導入しました。カメラのみよりカメラ+LiDARを併用した方が、車線や横断歩道の抽出精度が明確に高い(報告では数ポイントから二桁ポイントの差)ことを示し、「センサを足せば足すほど地図は正確になるが、その分コストとセンサ較正の手間が増える」というトレードオフを定量的に見せた点が貢献です。

これらの実験はいずれも、正解地図の生成に nuScenes Map API を用いています。つまり API は「数値を出すための採点台」として、研究の比較可能性そのものを支えています。注意すべきは、上記の数値が評価設定(範囲・解像度・どのレイヤを drivable とみなすか)に敏感なことです。たとえば「visibility(可視性)でフィルタするか」「自車後方を含めるか」で IoU が数ポイント動くため、論文間の数値比較は同一設定下でのみ厳密に成り立ちます。

## メリット・トレードオフ・限界

メリットは明快です。多層のベクトル地図を座標指定で取り出せ、自車中心のマスク生成まで組み込まれているため、誰でも同一手順で正解BEV地図を作れます。これが公平なベンチマークを成立させ、手法の優劣を再現可能に比較できるようにしました。

トレードオフと限界もあります。第一に地理的偏り。データは米国ボストンとシンガポールの限られた地域・期間で収集されており、その外には地図がありません。シンガポールは左側通行、ボストンは右側通行で、両者を混ぜた評価には注意が要ります。第二に前処理の必要。地図はベクトルなので、用途に応じた範囲・解像度・回転の設定とマスク化が毎回必要で、設定差が数値差を生みます。第三に静的地図の限界。地図は収集時点で固定され、工事や車線変更といった現実の変化は反映されません。第四にラベルの粒度。提供レイヤは有限で、たとえば「一時停止の優先関係」や「車線の合流の細かな規則」までは表現されません。第五に評価指標自体の限界で、IoU は「形が合っているか」を測りますが「つながり(トポロジー)が合っているか」は測れません。車線が1ピクセルだけ途切れて2本に分かれても IoU はほとんど下がらないのに、経路計画上は致命的、という乖離が起きえます。

研究上の未解決課題としては、こうした単一データセットへの過適合を避けるための、複数データセットをまたいだ汎化(別の都市・国でも通用するか)の評価がいまだ難しいことが挙げられます。nuScenes は地理的に train と val が場所で重ならないよう分割されていますが、それでも同一都市・同一センサ構成内での評価にとどまります。また「俯瞰地図を正確に塗れること」が「安全な運転判断」に本当に結びつくのか、すなわち地図セグメンテーションの IoU と最終的な運転性能の相関は、まだ十分に解明されていません。

## 発展トピック・研究の最前線

近年は、ピクセル塗り分け(セマンティックBEV)から、車線などをベクトル要素として直接出すオンライン地図構築([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))へ重心が移っています。さらに、地図を固定の正解とみなさず、車線の接続グラフ(どの車線からどこへ進めるか)まで含めて出す「ベクトル化+トポロジー推定」へと広がっています。nuScenes はこの新課題にも対応するため、車線中心線グラフを提供する方向で拡張されてきました。評価の側でも、形だけの IoU/AP に加え、トポロジー一致を測る指標(車線の接続が正しいかを採点する TOP 系の指標)が提案され、地図評価の枠組み自体が深化しています。

もう一つの潮流は、地図を明示的に作らず end-to-end の運転モデルの内部表現として暗黙的に持つ流れです。この場合 nuScenes Map API は、補助的な学習信号(地図セグメンテーションを補助タスクにして特徴を整える)や、デバッグ用の可視化基盤として使われます。地図セグメンテーション性能と運転性能の関係を測るうえでも、Map API は引き続き重要な参照点であり続けています。加えて、自前データセットを構築する際にも「nuScenes と同じレイヤ定義・同じマスク生成手順をまねる」ことで既存手法をそのまま走らせられるため、API の設計は事実上の業界標準として影響を残しています。
