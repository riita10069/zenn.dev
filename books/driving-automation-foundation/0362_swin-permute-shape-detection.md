# テンソル形状によるchannels-last検出とpermute

## ひとことで言うと

Swin Transformer のような階層型 Transformer は特徴を `[B, H, W, C]`(チャネルが最後)の順で出しますが、ResNet や ConvNeXt のような畳み込み系は `[B, C, H, W]`(チャネルが2番目)で出します。下流が後者を期待しているなら、前者だけ軸の並べ替え(permute)が必要です。このとき「バックボーン名が swin かどうか」という文字列で判定するのではなく、「テンソルの実際の形を見て、チャネルがどの軸にあるかを判定する」方式にする、という設計の話です。名前という間接的な手がかりではなく、形状という直接の事実を見ることで、表記ゆれやライブラリの仕様変更に強くなります。

## 直感的な理解

身近な例で言うと、荷物の中身を「箱に貼られたラベルの文字」で判断するか、「実際に箱を開けて中を見て」判断するかの違いです。ラベル(バックボーン名)は人が書いた間接情報で、表記ミスや古い情報のリスクがあります。中身(テンソルの形)は事実そのものです。

なぜこれが問題になるか。下流の畳み込みやプーリングは channels-first `[B, C, H, W]` を大前提にしています。もし channels-last `[B, H, W, C]` のテンソルをそのまま渡すと、エラーで止まればまだ良いほうで、最悪の場合は「縦幅 H をチャネルとして」処理を続けてしまい、エラーも出ないまま意味のない結果を出し続けます。たとえば `[4, 64, 64, 96]`(B=4, H=64, W=64, C=96)を channels-first だと思い込むと、「C=64, H=64, W=96」と解釈してしまい、96 個あるはずの特徴チャネルが 64 個に化け、空間の縦軸が特徴チャネルに混入します。畳み込みの重みの形が偶然合ってしまえばエラーにすらならず、学習は「ノイズを学習させられている」状態のまま発散も収束もしない、という最悪のデバッグ地獄になります。この種の「静かに間違う」バグは、学習が収束しない原因として最も発見が難しい部類です。

だから「どの軸がチャネルか」を確実に判定して、必要なら並べ替える必要があります。名前ベースの判定は、その確実性を「文字列が一致すること」という脆い条件に賭けてしまっています。形状ベースは、賭けの対象を「データの実際の形」という動かしようのない事実に移します。

## 基礎: 前提となる概念

- channels-first(チャネル先): `[B, C, H, W]` の並び。`B` はバッチ(画像の枚数)、`C` はチャネル(特徴の種類数)、`H`/`W` は縦横の解像度です。畳み込み系の標準で、`nn.Conv2d` や `F.grid_sample`(座標を指定して特徴をサンプリングする関数)、`F.interpolate`(リサイズ)はこの並びを前提にします。NCHW とも呼ばれます。

- channels-last(チャネル後): `[B, H, W, C]` の並び。NHWC とも呼ばれます。Transformer 系は内部でトークン列(各位置を1ベクトルとみなす列)を扱うので、チャネルを末尾に置くこの形が自然なことが多く、Swin V2 はこの形で各段の特徴を返します。

- permute(順列入れ替え): テンソルの軸の順番を入れ替える操作です。`f.permute(0, 3, 1, 2)` は「新しい軸0には元の軸0を、新しい軸1には元の軸3を、新しい軸2には元の軸1を、新しい軸3には元の軸2を」という指定で、`[B, H, W, C]` を `[B, C, H, W]` に並べ替えます。重要なのは、permute は中身のデータをコピーして動かすのではなく「見え方(ストライド、各軸を1進むのにメモリ上で何要素跳ぶか)」だけを変える、ほぼ無コストの操作だという点です。ただし後段で連続メモリを要求する処理(一部の畳み込み実装や `.view()`)に渡す前には `.contiguous()` で実体化が要ることがあり、ここで初めてコピーのコストが発生します。

- なぜ Swin だけ channels-last なのか: Swin はウィンドウ内自己注意(画像を小窓に区切って窓内だけ注意する。全画素ではなく窓内に限ることで計算量を画素数に対して線形に抑える工夫)を使う Transformer です。注意計算はトークンの列に対して行うので、各位置を1ベクトルとして末尾にチャネルを置くのが計算上自然です。一方 ConvNeXt は畳み込みベースなので、たとえ Swin と同じ「Tiny で 96/192/384/768」というチャネル設計でも channels-first で返します。この「設計は似ているのに軸の並びが違う」状況こそ、形状ベース判定が役立つ典型例です。

## 仕組みを詳しく

前提として、バックボーン構築時に各段の正しいチャネル数(例 `[96, 192, 384, 768]`)を動的取得で知っているとします。これを「正解の物差し」として、forward で返ってきた各特徴マップの形と照合します。

```python
def forward(image):
    features = backbone(image)
    permuted = []
    for f, expected_c in zip(features, feature_channels):
        # 4次元で、dim=1 が期待チャネルでなく、最後の次元が期待チャネルなら
        # それは channels-last なので permute して channels-first に直す
        if f.dim() == 4 and f.shape[1] != expected_c and f.shape[-1] == expected_c:
            permuted.append(f.permute(0, 3, 1, 2).contiguous())
        else:
            permuted.append(f)
    return permuted
```

判定ロジックを言葉で言うと「期待しているチャネル数が、軸1ではなく最後の軸にあるなら、チャネルが後ろに来ている=channels-last なので並べ替える」です。

具体例(Swin V2 の stage 0、期待チャネル 96)

- 返り値の形が `[4, 64, 64, 96]` → `shape[1]=64≠96` かつ `shape[-1]=96==96` なので channels-last と判定 → `permute(0,3,1,2)` で `[4, 96, 64, 64]` に直す。

具体例(ResNet-50 / ConvNeXt の stage 0)

- 返り値の形が `[4, 256, 64, 64]` → `shape[1]=256==256`(期待チャネル)なので何もしない(すでに channels-first)。

この方式は段ごとに独立して判定するので、仮に一部の段だけ並びが違うような変則的なバックボーンでも段単位で正しく直せます。名前は一切見ません。

before/after を一言で言うと、before は `if backbone_name == "swin": permute` という文字列分岐、after は「期待チャネルがどの軸にいるかを各段で照合して並べ替える」形状分岐です。

なぜ名前ベースが脆いか、3つの観点で整理します。

1. 表記ゆれに弱い。`swin_v2_tiny`、`swinv2`、`Swin`、`microsoft/swinv2-tiny` など命名は多様で、判定の文字列マッチを少しでも外すと並べ替えが行われず、軸がずれたまま下流に流れます。
2. ライブラリ実装の都合に依存する。「名前が swin なら channels-last」という仮定は、ライブラリが将来 Swin の出力を channels-first に変えたら逆に壊れます。実際、同じ Swin でもライブラリやバージョンによって出力規約が違うことがあります。形状ベースなら、出力が変わってもその場の形を見るので自動で追従します。
3. 拡張性が低い。名前に swin を含まないが channels-last を返す新しい Transformer 系バックボーンを足すたびに、また名前を if に書き足す羽目になります。

## 手法の系譜と主要論文

これは単独の研究テーマではなく、異なる設計のバックボーンが混在する現代の実務から要求された堅牢化です。背景となる研究を系譜で示します。

- Vision Transformer(Dosovitskiy et al., ICLR 2021, arXiv:2010.11929): 画像を 16x16 などのパッチに分割しトークン列として Transformer に入れる発想を導入しました。ここから「画像特徴をトークン列(末尾チャネル)として扱う」流れが本格化します。ViT は階層構造を持たず単一解像度のトークン列 `[B, N, C]` を出すため、後段で空間特徴へ戻すには N を H×W に再整形する処理が要る、という別種の軸問題も生みました。

- Swin Transformer(Liu et al., ICCV 2021, arXiv:2103.14030)と Swin V2(Liu et al., CVPR 2022, arXiv:2111.09883): ウィンドウ内自己注意とシフトウィンドウ(窓の境界をずらして窓間の情報も混ぜる工夫)で、Transformer を階層型(多段で解像度を下げる)にし、検出やセグメンテーションのバックボーンとして使えるようにしました。トークン列を扱う都合上、中間特徴は channels-last `[B, H, W, C]` が自然です。Swin V2 はさらに大解像度・大規模学習での安定化(残差後正規化、スケール化コサイン注意)を入れ、3 万画素級の入力にも対応しました。

- ConvNeXt / ConvNeXt V2(Liu et al., CVPR 2022, arXiv:2201.03545 / Woo et al., CVPR 2023, arXiv:2301.00808): 畳み込みを Transformer の設計思想(大きいカーネル、LayerNorm、GELU、段比率の見直し)で現代化したモデルで、channels-first `[B, C, H, W]` を返します。Swin と性能・チャネル設計が近いため、両者を差し替えて比較する需要が高く、軸の並びの違いを安全に吸収する仕組みの価値が顕在化しました。

- timm(Wightman, 2019〜): これら多様なバックボーンを統一 API で扱えるようにした一方、出力の軸の並びはモデル固有のまま残ります。だからこそ「ライブラリが返す形そのものを見る」形状ベース判定が、ライブラリ横断で安全に働きます。

## 論文の実験結果(定量データ)

permute 判定そのものに固有のベンチマークはありませんが、これが支える「Swin と ConvNeXt を同一条件で比較する」研究の数値を示します。

ImageNet-1K 分類のトップ1精度(画像を正しいクラスに分類できた割合、高いほど良い)で、Swin-T が約 81.3%、ConvNeXt-T が約 82.1% と報告されています。両者は同程度のパラメータ数(約 28〜29M)・計算量(約 4.5 GFLOPs、224x224 入力)で設計されており、ConvNeXt の原論文はこの公平な比較を通じて「適切に現代化した畳み込みは Swin と同等以上」と主張しました。この比較は、軸の並びが違う2モデルを同じ下流で扱えて初めて成立します。

COCO 物体検出では、Swin-T バックボーンの Mask R-CNN が box AP 約 46 前後(マルチスケール学習時)、ConvNeXt-T も同程度を達成し、いずれも ResNet-50 バックボーン(同条件で box AP 約 41 前後)を上回ると報告されています。AP は検出の正確さの指標で、約 5 ポイントの差は実用上はっきり体感できる差です。ADE20K のセマンティックセグメンテーション(mIoU、領域の重なり度合いを 0〜100 で測る指標)でも Swin と ConvNeXt は接戦で、いずれも ResNet を大きく上回ります。ここでも、channels-last の Swin と channels-first の ConvNeXt を同じ検出・分割ヘッドに接続する必要があり、軸の正規化が前提になります。

アブレーション的に言えば、「形状ベース判定を外して名前ベースに戻す」と、表記が一致している間は精度に差は出ませんが、命名が想定外だったり段ごとに軸が混在したりすると、permute が抜けて学習がそもそも収束しなくなります。これは「精度がわずかに下がる」のではなく「学習が成立しない」という、定量比較以前の破綻として現れます。つまりこの仕組みの価値は AP の数字に現れるのではなく、「比較できる土俵を成立させる」という前提の保証にあります。

## メリット・トレードオフ・限界

メリット

- バックボーン名の表記ゆれや改名に一切依存しない。
- ライブラリが軸の並びを将来変えても、形状ベースなので自動で追従する。
- 新しい channels-last 系バックボーンを足しても判定コードを書き換えなくてよい。
- 段ごとに判定するので、段によって並びが違う変則的なバックボーンにも対応できる。

トレードオフ・限界

- 「期待チャネル数が軸1にも最後の軸にも一致する」という稀なケース(例 H や W がたまたまチャネル数と同じ値になる、入力解像度を特定にすると起こりうる。チャネル 64 で空間 64x64 など)では誤判定の理論的余地がある。実用的な解像度ではほぼ起きないが、ゼロではない。回避するには、軸の並びを明示的に持つメタ情報があればそれを優先する、もしくは入力解像度をチャネル数と異なる値に固定して初期化するのが堅い。
- 出力が4次元 `[B, ?, ?, ?]` であることを前提にしている。3次元トークン列 `[B, N, C]`(N はパッチ数)を返す純 ViT 系には、別途「N から H, W を復元する」処理が要る(N = H×W という関係から正方なら sqrt で逆算、非正方ならアスペクト比を別途保持)。
- 構築時に求めた期待チャネル数が正しいことに依存する(その取得自体は動的取得で担保される)。期待チャネルが間違っていれば判定も間違うので、上流の動的取得とセットで信頼性が決まる。
- permute 後に `.contiguous()` を呼ぶ実装ではメモリコピーが発生し、巨大特徴では無視できないコストになりうる。可能なら下流を非連続テンソルでも動く処理にして実体化を遅らせる。

## 発展トピック・研究の最前線

- PyTorch には memory_format としての channels_last(`torch.channels_last`)があり、これはここで言う軸の並べ替えとは別概念(メモリ上の物理配置を NHWC 風にして畳み込みを高速化する最適化で、テンソルの論理的な形状は NCHW のまま)である点に注意。両者を混同しないことが、性能最適化を学ぶ上で重要。Tensor Core を使う混合精度学習では memory_format の channels_last が速度を稼ぐが、これは軸の意味を変えるものではない。
- 純 Transformer 系を検出・自動運転に使う流れ(DINOv2 などの自己教師ありバックボーン)では、トークン列 `[B, N, C]` をいかに空間特徴 `[B, C, H, W]` に戻すかが共通課題で、ここは形状検査だけでなく位置エンコーディングの設計とも絡む。CLS トークンの除去、パッチ数からの空間復元、レジスタトークンの扱いなど、実装上の落とし穴が多い。
- より根本的には「特徴の軸規約をアーキテクチャ全体で型として明示する」設計(named tensor, jaxtyping のような形状注釈)が、こうした暗黙の並び依存を静的に検出する方向として研究・実装の両面で進んでいる。実行前に「ここは [B, C, H, W] のはず」と型で書いておけば、誤った形のテンソルが渡された時点で落とせる。
