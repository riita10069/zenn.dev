# UniAD (計画指向BEV)

## ひとことで言うと

自動運転に必要な「物体検出」「追跡」「地図作成」「未来予測」「経路計画」という、本来は別々に作られていたタスクを、バラバラのモジュールではなく1つの大きなニューラルネットの中にまとめ、しかもすべてを「最終的に安全に運転する(=計画する)」という目的に向けて協調させた、E2E(end-to-end、端から端まで一気通貫)自動運転の代表作です。タスク同士は「クエリ」という問い合わせ用ベクトルで情報をやり取りします。コンピュータビジョンの最高峰会議 CVPR 2023 で最優秀論文賞を受賞しました。

## 直感的な理解

なぜタスクを1つにまとめる必要があるのか。自動運転を作る素朴な発想は、人間がやっている処理を順番に並べることです。まず周りの車を見つける(検出)、それに名前(ID)を付けて追い続ける(追跡)、道の形を把握する(地図)、相手がどう動くか読む(予測)、それを踏まえて自分の進路を決める(計画)。これを独立した部品として作り、出力をバケツリレーのようにつなぐのが伝統的な「パイプライン方式」です。

ここに問題があります。たとえるなら、リレー競走で各走者が自分の区間のタイムだけを気にして全力で走っても、バトンの受け渡しが下手だとチーム全体は速くなりません。検出器は「検出が当たること」を目的に最適化されますが、運転にとって本当に大事なのは「自分の進路に関わる相手を逃さないこと」です。遠くの無関係な車を完璧に検出しても運転は1mmも良くなりません。各部品の目的が、最終ゴールである「良い運転」とズレているのです。UniAD は「全部品を、良い運転というゴールから逆算して一緒に鍛え直す」という発想で、このズレを正面から解こうとしました。

## 基礎: 前提となる概念

理解の土台として、まずパイプライン方式の2つの弱点を整理します。

弱点1は情報の欠落と誤差の伝播です。各部品は「次の部品が使いやすい形」(検出なら3Dの箱、地図ならベクトル)に情報を圧縮して渡します。すると、計画に本当は役立つはずの微妙な手がかり(画像の質感や、わずかな動きの兆候)が途中で捨てられます。さらに前段が間違えるとその誤差が後段に伝わり、誰も全体最適を見ていないので、各部品が自分の精度を上げても全体が良くなる保証がありません。

弱点2は目的のズレです。前述の通り、部品ごとの目的(検出精度など)と、システムの目的(安全な運転)が一致していません。

ここで一つの極端な解として「画像 → ハンドル操作」を直接学ぶ素朴な end-to-end(ブラックボックス方式と呼ばれます)があります。これは目的のズレは消えますが、途中で何を考えているか分からず、事故時に原因を追えず、安全性の検証ができません。UniAD はこの中間を狙います。全タスクを1つのネットに入れつつ、検出・追跡・地図・予測・計画という解釈可能な中間タスクを明示的に残し、それらをすべて計画指向(planning-oriented)に並べて協調させるのです。

もう一つの前提が BEV(Bird's-Eye-View、真上から見下ろした視点)特徴です。複数カメラ画像を BEVFormer という手法で1つの俯瞰特徴マップに変換したもので、例えば形は 256×200×200(チャンネル256、縦200マス、横200マス)です。すべての下流モジュールがこの共有 BEV 特徴を土台にします。

最後にクエリ(query)です。Transformer において、クエリとは「この物体について教えて」「この未来について教えて」と尋ねる問い合わせベクトルです。UniAD の設計思想の核心は、情報を密な特徴マップではなく疎な(数の少ない)クエリで渡す点にあります。これにより、モジュール間で必要な情報だけを選んで伝えられます。

## 仕組みを詳しく

UniAD は共有 BEV 特徴の上に、6つの段を Transformer で積み上げます。

第1段は BEV エンコーダ(共有の土台)です。6台のカメラ画像を BEVFormer に通し、BEV 特徴 B(256×200×200)を作ります。以降のすべてのモジュールがこの B を参照します。

第2段は TrackFormer(検出 + 追跡の統合)です。物体検出と追跡を1つの Transformer で同時に行います。新しく現れた物体用の「検出クエリ」と、前フレームから引き継ぐ物体用の「追跡クエリ」を持ち、各エージェント(車・歩行者など)に対応する「エージェントクエリ」を出力します。従来は別々だった検出と追跡が一体になり、ID の引き継ぎ(同じ車を同じ車だと認識し続けること)が自然に行えます。

第3段は MapFormer(地図のセグメンテーション)です。車線・路肩・横断歩道などの地図要素をクエリで表現します(パノプティック・セグメンテーション、つまり物体ごとと領域ごとの両方を統一的に扱う考え方)。出力される「地図クエリ」は後段で道路構造の文脈として使われます。

第4段は MotionFormer(エージェントの未来予測)で、見どころの一つです。TrackFormer のエージェントクエリと MapFormer の地図クエリを参照し、周囲の各物体がこの先どこへ動くかを予測します。重要なのは、自車(ego vehicle、自分の車)も1つのエージェントとして同じ枠組みで予測する点(ego-query)です。さらに「エージェント同士」「エージェントと地図」「エージェントとゴール地点」の3種類の相互作用をアテンションで明示的にモデル化し、複数の未来候補(マルチモーダル、例えば6本の軌跡)を出します。マルチモーダルとは「相手は直進するかもしれないし右折するかもしれない」という複数の妥当な未来を同時に持つことです。

第5段は OccFormer(占有グリッドの未来予測)です。将来どのマスが物体に占有されるか(occupancy)を時系列で予測します。物体単位の予測(MotionFormer)とは別に、画素・マス単位で「ここは通れる/通れない」を密に予測することで、検出漏れした物体や形が定まらない障害物にも備えます。

第6段は Planner(計画)です。MotionFormer の ego-query(自車の意図を含むクエリ)を受け取り、自車の走行軌跡を出します。さらに OccFormer の占有予測を使い、「予測した自車軌跡が障害物の占有マスとぶつからないか」を最適化で調整します。これで「周囲の未来予測と矛盾せず、衝突しない計画」が得られます。

学習のしかたにも工夫があります。全部を一気に学習すると不安定なので、論文では段階的に学習します。まず認識系(TrackFormer, MapFormer)を数エポック先に学習して土台を安定させ、その後に予測・計画系を含めて全タスクを end-to-end で同時学習します。各タスクの損失を重み付きで足し合わせるので、計画の誤差が逆伝播して、検出や予測の特徴までも「良い計画に役立つ方向」へ調整されます。これが「計画指向」という言葉の技術的な意味です。

## 手法の系譜と主要論文

E2E 自動運転には大きく2つの源流があります。一つは1989年の ALVINN(Pomerleau)や2016年の NVIDIA PilotNet に代表される、画像から直接操作を出すブラックボックス模倣学習。もう一つが、解釈可能な中間表現を経由する流れです。UniAD は後者の系譜の到達点です。

直接の前身が ST-P3(Hu et al., ECCV 2022, arXiv:2207.07601)です。空間-時間特徴を使い、知覚・予測・計画を一貫して学習する解釈可能な end-to-end でした。UniAD はこれをクエリベースで全面的に再設計し、性能を大きく伸ばしたと位置づけられます。

土台となる BEV エンコーダが BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)です。時空間アテンションで複数カメラ・複数時刻を1つの BEV にまとめる手法で、UniAD はこれを共有エンコーダとして採用しました。

そして UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)です。提案は「全タスクをクエリで連結し、計画を最終目的に据えて協調最適化する統合フレームワーク」。動機は、独立パイプラインの誤差伝播と目的ズレ、素朴ブラックボックス end-to-end の非解釈性、その両方を避けることでした。

UniAD の直後に出たのが VAD(Jiang et al., ICCV 2023, arXiv:2303.12077)で、軽量化路線です。UniAD が密な占有やセグメンテーションを多用して重いのに対し、シーンをベクトル化して計画することで、同等以上の計画性能を大幅に高速・軽量に達成しました。さらに後年には、計画を最終目的にすると認識性能が犠牲になりうるという批判や、open-loop 指標の妥当性への疑問(AD-MLP、BEV-Planner などの研究)が提起され、E2E 評価のあり方そのものが議論される段階に入りました。

## 論文の実験結果(定量データ)

評価は nuScenes(屋外運転の大規模公開データ、約1000シーン、6カメラ + LiDAR、3D アノテーションと自車軌跡を含む)で行われました。UniAD の特徴は、検出・追跡・地図・予測・占有・計画の全タスクを1つのモデルで同時に評価し、それぞれで当時の最先端に並ぶか上回った点です。

最重要の計画指標が衝突率(collision rate、予測した自車軌跡が他物体とぶつかると判定される割合)と L2 誤差(予測軌跡と人間の実際の運転軌跡の平均変位、メートル単位、小さいほど人間に近い)です。報告によると、UniAD は平均衝突率を、前身 ST-P3 や同時期の手法に比べて大きく(相対的に半分以下の水準まで)下げました。具体的には1秒・2秒・3秒先の平均衝突率を ST-P3 のおよそ0.7%前後の水準から0.3%前後の水準へ改善したと報告されています。衝突率は「実際にぶつかる危険」を直接測る最も切実な指標なので、この改善は大きな意味を持ちます。

アブレーション(構成要素を抜く実験)が UniAD の主張の中核です。論文は、(1) MotionFormer を外す、(2) OccFormer を外す、(3) 地図や追跡の段を外す、といった形で下流タスクへの寄与を測りました。結果、予測(MotionFormer)と占有(OccFormer)を計画段の手前に入れることで衝突率が一貫して下がり、「上流タスクを計画指向につなぐほど計画が良くなる」という主張を定量的に裏づけました。特に占有予測を計画の衝突チェックに使う構成が、衝突率削減に効くことが示されています。逆に、これらのモジュールが互いに切り離されていると、各タスク単体の精度は出ても計画は改善しませんでした。なお後年の研究(BEV-Planner ら)は、自車の過去の運動状態だけからでも nuScenes の L2 がかなり当たってしまうことを指摘し、open-loop の L2 指標が計画能力を過大評価しうると警告しています。この点は UniAD の成果を否定するものではありませんが、評価指標の読み方には注意が要ります。

## メリット・トレードオフ・限界

メリットは、全タスクを計画指向に協調させることで計画の質(衝突率など)が単独最適化より良くなること、検出・追跡・地図・予測という解釈可能な中間出力が残るので「なぜその運転をしたか」を検証しやすく安全性の観点で重要なこと、自車も周囲エージェントも同じ枠組みで未来予測するため譲り合いなどの相互作用が自然に表現できることです。

トレードオフと限界もあります。モデルが巨大で学習・推論が重く、実車のリアルタイム要件には追加の最適化が要ります。検出・追跡・地図・予測・占有・計画とほぼ全種類の教師ラベルが必要で、データ作成コストが非常に高い。段階的学習やタスク損失の重み調整など学習が繊細で、再現が難しい面があります。評価が nuScenes の open-loop(記録データ上での評価で、自車の行動が環境に影響を返さない設定)中心であり、実走行(closed-loop、自分の行動が次の観測を変える設定)での因果的妥当性は別途議論が続いています。さらに前述の通り、open-loop の L2 指標自体が計画能力を過大評価しうるという指摘があり、closed-loop シミュレータ(CARLA など)での再評価が今後の重要課題です。

## 発展トピック・研究の最前線

研究の最前線は3方向に整理できます。第一に効率化で、VAD 系のベクトル表現や、近年の SparseDrive(疎な表現で密な BEV を避ける)など、UniAD の重さを削る流れです。第二に評価の刷新で、open-loop の限界を踏まえ、CARLA ベースの closed-loop ベンチマーク(Bench2Drive など)や、自車の過去運動への近道を断つ評価設計が広がっています。第三に大規模化・言語統合で、運転を言語モデル(VLM、Vision-Language Model)と結びつけ、運転判断を自然言語で説明させたり指示で制御したりする方向(DriveGPT、LMDrive など)が急速に伸びています。UniAD はモジュール統合の到達点として、これらすべての研究が参照する基準点であり続けています。
