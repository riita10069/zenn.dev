# 特徴空間で未来を予測するヘッド(JEPA 風 future-state head)

## ひとことで言うと
未来予測ヘッド(future-state head)とは、ワールドモデル(world model、世界が次にどうなるかをネットワーク内部で予測する仕組み)の出力末端で、「現在のシーンを要約した特徴から、数秒先の特徴を当てる」部品です。ピクセル(画素そのもの)ではなく特徴空間で未来を予測するのが要点で、これは JEPA (Joint-Embedding Predictive Architecture, 結合埋め込み予測アーキテクチャ) と呼ばれる設計思想に連なります。さらに、自車が「どう動くつもりか」を要約したベクトルを条件として与えることで、計画と整合した未来を予測できる点が、自動運転の文脈で重要になります。

## 直感的な理解
自動運転で「次に何が起きるか」を予測できると、計画(planning、どう走るかを決めること)の質が上がります。「前の車が減速しそう」と分かれば早めにブレーキを準備できますし、「右から自転車が出てきそう」と分かれば速度を落とせます。

未来予測には大きく2つのやり方があります。

1. 未来のカメラ映像そのもの(ピクセル)を生成する。直感的で「見れば分かる」反面、未来のピクセルは本質的に予測不能な細部(影の揺れ、葉の動き、対向車の正確なテクスチャ)を大量に含みます。これを律儀に当てようとすると、損失を下げる最も安全な答えは「全部を平均してぼかす」ことになり、肝心の動きや構造の情報が薄れます。これを mode averaging(複数のあり得る未来を平均してぼやける現象)と呼びます。
2. 未来の「特徴」を予測する。ピクセルの細部は捨て、「車が前方にいる」「車線がこう曲がる」「自由空間がこちらに広がる」といった意味だけを当てます。予測不能な細部を当てる無駄が消え、表現が安定します。

未来予測ヘッドは2の路線です。さらにここで決定的に重要なのが「自分の意図を条件にする」ことです。未来は「自分がどう動くか」で変わります。同じ現在シーンでも、右折する計画と直進する計画では数秒後に見える世界はまったく違います。そこで「プランナーが今どんな軌道を描こうとしているか」を要約したベクトル(以下 ego ベクトル)を手がかりに与え、「その計画で進んだ未来」を予測させます。これにより予測が計画と整合し、後述するワールドモデル的な計画への布石になります。

## 基礎: 前提となる概念
特徴マップ(feature map): 畳み込みネットワークが画像から取り出した中間表現で、形は [C, H, W] です。C はチャンネル数(検出した特徴の種類数のようなもの)、H×W は空間グリッドです。ここに複数サンプルをまとめると [B, C, H, W] になり、B はバッチサイズ(同時処理する例の数)です。たとえば [4, 256, 50, 50] なら「4枚の画像を、256 種類の特徴で、50×50 のマス目に要約した」もの。

BEV (Bird's-Eye-View, 鳥瞰図): カメラ画像の特徴を上空から見下ろした2次元グリッドに投影した表現です。各マスが「自車を中心とした地面上のある区画」に対応し、複数カメラや地図情報を同じ座標系に統合しやすいため、計画・予測でよく使われます。前方カメラの「奥に小さく写る車」と後方カメラの車を、同じ地面座標の上で扱えるのが利点です。

埋め込み次元(embedding dimension, embed_dim): 特徴のチャンネル数 C のことで、表現の容量を決めます。256 程度がよく使われます。大きいほど表現力は増すがメモリと計算が増えます。

条件付け(conditioning): あるベクトル(ここでは ego ベクトル)をネットワークに注入し、出力をその情報で変調することです。最も単純な注入法が per-channel bias(チャンネルごとのオフセット)で、特徴マップの各チャンネルに、ベクトルから計算した1つの値を空間全体に一律で足します。より高度には FiLM(特徴をスケール+シフトで変調)やクロスアテンションがあります。

stop-gradient / 凍結: 学習の正解(ターゲット)を作る経路に勾配を流さないことです。未来予測ヘッドが「予測」を出す一方、その正解をどう作るかが JEPA の肝で、これは [凍結 backbone によるターゲット生成](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1211_frozen-backbone-target-encoder) で詳述します。

## 仕組みを詳しく
典型的な実装の形(tensor shape)を追います。embed_dim は 256、ego ベクトルの次元も 256 とします。

入力:
- 融合特徴 fused_features: [B, C, H, W]、C=256。これは「複数カメラの画像特徴 + ナビ地図の特徴」を融合した現在シーンの俯瞰特徴で、H×W は BEV グリッド解像度(例 450×300)。
- ego ベクトル ego_hidden: [B, 256]。プランナー(GRU や Flow Matching ベースの軌道生成器など)が出す意図要約。

処理ステップ:

1. ego ベクトルを線形層(Linear(256, 256))に通し、[B, 256, 1, 1] に整形する。これを per-channel bias と呼ぶ。
   ```
   bias = ego_proj(ego_hidden).view(B, 256, 1, 1)
   ```
2. その bias を fused_features の全空間位置(H×W のすべてのマス)に同じ値でブロードキャスト(broadcast、形を自動で合わせて足す)して加える。これでシーン特徴にプランナーの意図が注入される。
   ```
   conditioned = fused_features + bias   # [B, 256, H, W]
   ```
3. 畳み込み層(Conv2d、画像の局所パターンを抽出する層)を2段通してチャンネルを増やす。1段目で 256→512、GELU 活性化(滑らかな非線形変換)を挟み、2段目で 512→1024(=4×256)。
   ```
   x = conv1(conditioned)   # [B, 512, H, W]
   x = GELU(x)
   x = conv2(x)             # [B, 1024, H, W]
   ```
4. 1024 チャンネルを4つに分割(chunk)し、[B, 256, H, W] の特徴マップを4枚得る。それぞれが将来の各タイムステップ(例: +1.6 / +3.2 / +4.8 / +6.4 秒)の予測。
   ```
   future = chunk(x, 4, dim=1)   # 4 × [B, 256, H, W]
   ```

ASCII 図:
```
fused_features [B,256,H,W] ──┐
                             (+) ── Conv 256→512 ─ GELU ─ Conv 512→1024 ── chunk×4
ego_hidden [B,256] ─ Linear ─┘                                              │
   (per-channel bias, 全マスへブロードキャスト)                            ▼
                                              4枚の未来特徴 各[B,256,H,W]
                                              t+1.6s / +3.2s / +4.8s / +6.4s
```

この設計で必ず検証したい性質が3つあります。(a) 出力が4枚で形が入力どおりであること、(b) ego ベクトルを変えると4枚すべての出力が変わること(意図がちゃんと効いている証拠)、(c) 4枚が同じデータの使い回しでなく本当に異なること(時間方向に違う未来を出している証拠)。(b) と (c) が崩れていると、条件付けや時間方向の予測が機能していないサインです。実装をデバッグするときは、同じ fused_features に対して ego ベクトルだけ二通り入れて出力の差分ノルムが十分大きいかを確認するのが定番です。

なぜ chunk で4ステップを「一気に」出すのか、という設計判断も重要です。各時刻を別々の畳み込みで出す(時刻ごとに独立ヘッド)案もありますが、一気に出すと共有された特徴計算を再利用でき軽量です。一方で、時間方向に再帰的(t を予測してそれを入力に t+1 を予測)にすると長期予測ができますが、誤差が累積しやすくなります。一括 chunk は「短期数ステップを安定に、再帰なしで」出す折衷です。

メモリ上の注意も実務的に重要です。450×300 の全解像度で 4×256 チャンネルを fp32(32ビット浮動小数)で保持すると、1サンプルあたり 450×300×1024×4 バイト ≈ 550MB に達します。バッチを少し増やすだけで容易に数十 GB を食うため、bf16(16ビット混合精度)や、ヘッド前に空間解像度を下げる、あるいは時間ステップを別々に計算する、といった工夫が事実上必須です。45GB クラスの GPU でも、BEV 450×300・バッチ4 でこのヘッドを有効化すると 20GB 超、バッチ8 で容易に OOM(out of memory, メモリ不足)になる、という規模感が報告されています。研究で高解像度 BEV の未来予測を扱うときは、まずこのメモリ見積もりを立てるのが鉄則です。

## 手法の系譜と主要論文
- Yann LeCun, "A Path Towards Autonomous Machine Intelligence" (2022) が JEPA という枠組みを提示しました。「世界の予測は予測不能な細部まで生成するのでなく、抽象的な表現空間で行うべき」という主張で、未来予測ヘッドの思想的な母体です。LeCun はピクセル生成型(autoregressive / generative)を「エネルギーを無駄に使う」と批判し、表現空間で予測する JEPA を自律知能への道筋として提唱しました。詳細は [JEPA (Joint-Embedding Predictive Architecture)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0064_jepa-architecture)。
- Mahmoud Assran et al., "I-JEPA: Self-Supervised Learning from Images with a Joint-Embedding Predictive Architecture" (CVPR 2023, arXiv:2301.08243) が画像で JEPA を実装し、マスクした領域の特徴を文脈から予測する形で、ピクセル再構成(MAE)に匹敵以上の表現品質を得ました。ターゲットを EMA(指数移動平均)で更新し stop-gradient を併用して表現崩壊を防ぐ仕組みが、未来予測ヘッドの「正解の作り方」の手本です。
- Adrien Bardes et al., "V-JEPA" (2024) が動画へ拡張し、時間方向にマスクした部分の特徴を予測する形で、動きの理解(action recognition など)に強い表現を得ました。未来予測ヘッドの「時間方向に特徴を予測する」発想に直結します。続く V-JEPA 2 では行動入力を加えてロボット操作の計画に使う流れも生まれています。
- 自動運転の世界モデルでは2系統が並走します。生成系の GAIA-1 (Hu et al., Wayve 2023, arXiv:2309.17080) は行動を条件にカメラ映像そのものを生成し「見て分かる」反面、計算が極めて重くぼかし問題を抱えます。潜在系の MILE (Hu et al., NeurIPS 2022) は潜在(特徴的)空間で世界ダイナミクスと方策を同時に学び、特徴空間で予測する点が未来予測ヘッドに近い発想です。未来予測ヘッドは「潜在で予測し、かつプランナーの意図を条件にする」点で、両者の中間かつ JEPA 寄りの設計と位置づけられます。

## 論文の実験結果(定量データ)
特徴空間予測が「ぼかし問題」を避けつつ良い表現を作ることは、画像・動画の表現学習で繰り返し示されています。指標の読み方も含めて説明します。

- I-JEPA (CVPR 2023): ImageNet-1k の線形評価(linear probing、エンコーダを凍結し線形分類器だけ学習して精度を測る方式。特徴がどれだけ意味を持つかを直接測れる)で、ViT-Huge を用いて約 72–73% top-1 を達成しました。ピクセル再構成型の MAE と同等以上の精度を、より少ない学習計算(GPU 時間)で得られると報告され、「ピクセルを当てない方が効率的に良い特徴が得られる」ことの定量的根拠になりました。アブレーションでは、ターゲットを stop-gradient にしないと表現が崩壊(精度が著しく低下し、ほぼランダム水準に落ちる)すること、予測対象を十分に大きなブロックにすると下流性能が上がることが示されています。後者は「小さすぎる領域を当てるとテクスチャの丸暗記になり、意味的な予測にならない」ことを意味します。
- V-JEPA (2024): 動画の凍結評価で、Kinetics-400(動作分類)や Something-Something-v2(時間的推論を要する動作分類、SSv2)で当時の自己教師あり動画手法を上回る精度を報告しました。特に「単一フレームの見た目だけでは解けず、時間的な手がかりが必須」な SSv2 で強いことは、特徴空間での未来/欠損予測が動きの情報を保持できる証拠です。SSv2 は「何かを左に動かす/右に動かす」のように、フレームの順序を逆にすると答えが変わるタスク群なので、時間理解の良いリトマス試験紙になります。
- MILE (NeurIPS 2022): CARLA のクローズドループ評価で、当時の手法に対し走行スコアを大きく改善し、潜在空間でのワールドモデル学習が運転計画に有効であることを示しました(報告では先行手法比で運転性能・違反指標が改善)。これは「特徴空間で未来を予測すること」が表現学習だけでなく実際の運転にも効く、という橋渡しの証拠です。

これらが示す共通のメッセージは、未来予測ヘッドの設計選択(特徴空間で予測する、ターゲット経路を勾配から切る、意図で条件付ける)それぞれに実証的裏付けがある、ということです。なお、ランダムなノイズ入力で動作確認すると特徴空間予測は数値的に不安定になりやすく(正解構造のない入力では損失が発散・NaN 化しやすい)、実データでの数値安定性は別途確認が必要、という実務上の注意もあります。スモークテスト(ランダムテンソルで形だけ確認するテスト)で loss が nan になっても、それ自体はコードの欠陥とは限らない、というのは現場でよく遭遇する落とし穴です。

## メリット・トレードオフ・限界
- メリット: 未来をピクセルでなく特徴で予測するためぼけにくく、計算も生成系(拡散モデルや自己回帰生成)より軽い。
- メリット: ego ベクトルでの条件付けにより、予測がプランナーの計画と整合する。「自分が右折する未来」を当てに行ける。
- メリット: 条件付け経路が固定なら、内部のプランナー(GRU / Flow Matching など)を差し替えても同じ仕組みで動く。モジュール性が高い。
- トレードオフ: 全解像度では VRAM 消費が大きく、bf16 や解像度削減・時間ステップ分割が事実上必須。
- トレードオフ: ego ベクトルを全空間に一様な bias として足すだけなので、空間的に偏った影響(画面右側だけ大きく変える等)は表現しづらい。これを乗り越えるのが FiLM やクロスアテンションへの拡張。
- トレードオフ: 予測損失の正解(ターゲット)をどう安定に作るかが本質的に難しく、ここを誤ると表現崩壊する([凍結 backbone によるターゲット生成](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1211_frozen-backbone-target-encoder))。
- トレードオフ: 一括 chunk による短期予測は安定だが、長期(10 秒超)の未来は表現できない。再帰展開すると長期は出せるが誤差が累積する。
- 未解決課題: 特徴空間予測の品質を直接測る指標が乏しく、「良い未来特徴」を可視化・評価しにくい(ピクセルなら目で見えるが、特徴は見えない)。下流(計画)への寄与を分離して評価する方法、長期予測での誤差累積の抑制が課題として残る。

## 発展トピック・研究の最前線
未来予測ヘッドの自然な発展は、(1) 条件付けを bias 加算から、空間的に異なる影響を表現できる方式(クロスアテンションや FiLM 的な乗算変調)へ拡張すること、(2) 単一の点予測でなく未来の不確実性(分岐する複数シナリオ)を表現する確率的ヘッドへ拡張すること、たとえば未来を複数モード(右折・直進・左折)の混合分布として出すこと、(3) [行動条件付きワールドモデル](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0003_action-conditioned-world-model) として計画軌道そのものを条件に入れ、複数の候補軌道それぞれの未来を試走させて最良を選ぶ「計画と予測の結合」へ進むこと、です。

研究の最前線では、JEPA 系を自動運転・ロボティクスの行動条件付き予測へ持ち込む動き(V-JEPA を行動入力に拡張する流れ、V-JEPA 2 など)が活発で、生成系ワールドモデル(GAIA-1 系や運転ドメインの拡散世界モデル)との「予測の質 対 計算コスト」のトレードオフが中心的な論点になっています。生成系は「人間が見て検証できる映像」を出せる点で説明性に優れますが、特徴空間予測は「計画に必要な意味だけを安く予測する」点で効率に優れます。どちらが運転に最適かはまだ決着しておらず、近年は両者を組み合わせる(意味は特徴空間で、検証用に一部だけピクセルで)ハイブリッドも模索されています。
