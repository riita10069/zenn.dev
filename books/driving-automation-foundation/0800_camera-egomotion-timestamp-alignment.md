# カメラとegomotionのタイムスタンプ整合

## ひとことで言うと
車に積んだカメラ (約30Hz = 1秒に30枚撮る) と、車の運動を測る装置 (約100Hz = 1秒に100回記録する) は、別々のセンサーなので記録のタイミングがピッタリ揃っていません。このノートは「ある瞬間の運動の記録」に対して「一番近い時刻に撮られたカメラ画像」をマイクロ秒単位の時刻で探し出して結びつける処理 — マルチセンサーの時刻同期 — の話です。これを誤ると、モデルは「見た景色」と「その瞬間の運動」を取り違えて学習してしまいます。

## 直感的な理解
2人の友達が同じ花火大会をそれぞれ別の動画で撮ったとします。1人は1秒30コマ、もう1人は1秒100コマで撮りました。後で「同じ瞬間の2人の映像」を並べたいとき、コマの数も撮り始めも違うので、片方のあるコマに「もう片方で時刻が一番近いコマ」を探して対応づけるしかありません。これがタイムスタンプ整合 (最近傍マッチング) です。

自動運転では、これがもっと深刻です。モデルに学ばせたいのは「この景色 (カメラ画像) を見たとき、車はどう動いていたか / これからどう動くべきか」という対応関係です。画像と運動が「同じ瞬間のもの」でなければ、モデルは間違った対応を覚えてしまいます。たとえばカメラ画像は交差点の3メートル手前なのに、運動の記録は交差点を通過した後だったら、「交差点手前ではアクセルを踏む」という誤った教訓を学んでしまうかもしれません。時刻のズレは、ラベルノイズとして静かに学習を蝕みます。

## 基礎: 前提となる概念

- タイムスタンプ (timestamp): あるデータが「いつ」のものかを表す時刻の刻印です。マイクロ秒 (= 100万分の1秒) 単位で記録されることが多いです。Unix エポック (1970年1月1日) からの経過時間で表すのが一般的。
- clip (クリップ): 1本の走行を切り出した区間です。クリップの先頭を t=0 とした「相対時刻」でタイムスタンプを表すと、クリップをまたいだ巨大な絶対時刻 (16桁を超える) の数値誤差を気にせずに済みます。
- egomotion (エゴモーション): 自車の運動状態 (速度、加速度、車体の向き=ヨー、曲がり具合=曲率など) の集まり。IMU や GPS から計算されます。
- IMU (Inertial Measurement Unit, 慣性計測装置): 加速度センサーとジャイロセンサー (角速度センサー) を組み合わせ、車の加速度や回転を測る装置。高頻度 (100Hz 以上) で動きます。GPS が秒に数回しか更新されないのを IMU が補間します。
- 最近傍マッチング (nearest-neighbor matching): 複数の候補の中から「目標値に最も近いもの」を1つ選ぶこと。ここでは「時刻が最も近いフレーム」を選びます。
- デコード (decode): 圧縮された動画 (.mp4 など) から、特定のフレームの画像を復元すること。動画は前後フレームの差分で圧縮されているので、任意の1枚を取り出すには近くのキーフレームから再生する必要があり、ランダムアクセスは意外に重い処理です。
- クロック (clock): センサーごとの内部時計。別々のセンサーは別々のクロックで動き、温度ドリフトなどでわずかにずれます。完全に同期したクロックを持つことは現実には難しい。

具体的な数字で考えます。100Hz なら記録は10ミリ秒 (0.010秒) ごと、30Hz なら約33.3ミリ秒ごとです。ある運動記録の時刻が、カメラのフレームとフレームのちょうど中間に来ることは普通に起こります。そこで「どのカメラ画像をその瞬間の代表として使うか」を決める仕組みが必要になります。

## 仕組みを詳しく

整合の中心は「最近傍マッチング」です。やっていることは「時刻が一番近いものを選ぶ」だけのシンプルな処理です。

### 手順

1. 共通の時間軸を使う。egomotion もカメラも、タイムスタンプを「マイクロ秒単位」かつ「同じクリップ先頭を t=0 とした相対時刻」で表します。両方が同じ基準で測られているので、単純に数値を引き算して比べられます。整数マイクロ秒で持つと浮動小数点の丸め誤差も避けられます。
2. 注目する運動記録の時刻を1つ決める。これは「サンプル点」の時刻 (例: 整数のマイクロ秒値) です。
3. 全カメラフレームの時刻配列と引き算し、絶対値が最小のインデックスを返す。

$$
\text{frame\_idx} = \arg\min_{j} \left| t^{cam}_j - t^{ego} \right|
$$

ここで $t^{cam}_j$ は $j$ 番目のカメラフレームの時刻、$t^{ego}$ は注目する運動記録の時刻です。$\arg\min$ は「差の絶対値が最小になる $j$」を返します。時刻配列がソート済みなら二分探索 ($O(\log N)$) で高速に求められますが、線形探索 ($O(N)$) でもフレーム数が数千程度なら十分実用的です。

### 数値例

運動記録の時刻が $5{,}000{,}000$ マイクロ秒 (= 5.000秒) だとします。カメラのフレーム時刻が

```
[..., 4_966_000, 4_999_000, 5_033_000, ...]   (マイクロ秒)
```

なら、差は

```
[...,   34_000,    1_000,    33_000, ...]
```

最小は2番目 (差1ミリ秒) なので、そのフレームが選ばれます。ズレは最大でもフレーム間隔の半分、つまり30Hz なら約16.7ミリ秒に収まります。これが最近傍マッチングの誤差上限です。

### 選んだフレームの取り出しと多カメラ整合

選ばれたフレーム番号で、シーク可能な動画リーダーを使って .mp4 から1枚だけデコードします。複数台 (例: 7台) のカメラそれぞれについて、同じ運動記録の時刻で最近傍フレームを選ぶので、複数枚の画像が「だいたい同じ瞬間」に揃います。各カメラのデコードは独立なので並列化できます。

```
egomotion時刻 t_ego = 5.000s
        │
        ├─→ カメラ1: 最近傍フレーム選択 → デコード → (3, H, W)
        ├─→ カメラ2: 最近傍フレーム選択 → デコード → (3, H, W)
        ├─→ ...
        └─→ カメラ7: 最近傍フレーム選択 → デコード → (3, H, W)
                                          ↓
                          全部 stack → (7, 3, H, W)
```

注意すべきは、7台のカメラがそれぞれ独立に最近傍を選ぶため、選ばれた7枚が「完全に同一時刻」とは限らない点です。各カメラのシャッターが揃っていなければ、7枚の間にも最大で数十ミリ秒のズレが残ります。

### 実装上の工夫

- タイムスタンプ配列のキャッシュ: データを1件取り出すたびにタイムスタンプのファイルを読み直すのは無駄なので、一度読んだら使い回します。最近傍マッチング自体は引き算と argmin だけなので軽いですが、ディスク I/O は重いので、ここを節約するのが効きます。データローダのワーカーごとにキャッシュを持たせる設計が定番です。
- 単調増加チェック: 時刻が「厳密に単調増加 (必ず増えていき、戻ったり止まったりしない)」かどうかを検証し、おかしいクリップはスキップします。最近傍マッチングは時刻がきれいに並んでいることを前提にしているので、巻き戻りや重複のある壊れたメタデータを弾く必要があります。時刻が逆行しているのは、ログ収集中のバッファリングやクロック補正の痕跡であることが多いです。

## 手法の系譜と主要論文

マルチセンサーの時刻同期そのものを単独主題にした論文は多くありませんが、実車データセットの標準的な前処理として各データセット論文が設計を記述しています。

- KITTI (Geiger et al., CVPR 2012): 自動運転データセットの草分け。カメラ・LiDAR (Velodyne HDL-64E)・GPS/IMU を搭載し、各センサーのタイムスタンプを記録しました。LiDAR の回転 (10Hz) に合わせてカメラのシャッターをトリガーする「ハードウェア同期」を採り、LiDAR が前方を向いた瞬間にカメラを切る設計で、ソフトウェアでの後処理マッチングをある程度不要にしました。これは「センサーを物理的に同期させる」アプローチです。
- nuScenes (Caesar et al., CVPR 2020): 6台のカメラ・1台のLiDAR・5台のレーダーを持ち、各センサーのタイムスタンプを記録します。カメラの露光を LiDAR が各カメラの視野中心を通過する瞬間にトリガーすることで、LiDAR スイープとカメラの時刻ずれを最小化する設計です。なぜそうしたかというと、センサーごとにレートも露光タイミングも違うため、何らかの基準時刻に寄せないと物体検出のアノテーションが一貫しないからです。効果は全センサーを共通時刻軸で扱えること。トレードオフは、基準に最も近いフレームを採るだけだと最大で数十ミリ秒のズレが残り、高速移動時には画像と点群が空間的にずれる点です。
- Waymo Open Dataset (Sun et al., CVPR 2020): カメラのローリングシャッター (CMOS センサーが画像の上から下へ少しずつ撮るので、1枚の中でも上端と下端で時刻が違う現象) まで考慮した正確なタイムスタンプを提供し、各カメラフレームに行ごとの露光時刻情報を持たせました。理由は、自動運転では数十ミリ秒のズレでも高速域では位置誤差になるためです。効果は厳密な時空間整合。トレードオフは、扱うデータと処理が複雑になることです。

「最近傍1フレーム」方式は最も単純な方式で、フレーム内補間やローリングシャッター補正は行いません。30Hz カメラ・市街地速度の範囲では実用上問題になりにくいという割り切りです。

## 論文の実験結果(定量データ)

時刻ズレが性能にどう効くかは、速度から物理的に見積もれます。

- 速度と位置ズレの関係: 時速72km は秒速20m です。10ミリ秒 (0.010秒) のズレは $20 \times 0.010 = 0.2$ m = 20cm の位置ずれに相当します。30Hz カメラの最近傍マッチングが残す最大ズレ約16.7ミリ秒では、時速72km で約33cm ずれます。物体検出のバウンディングボックス (車を囲む四角) にとって33cm は無視できない大きさで、隣車線の車と自車線の車を取り違える原因になりえます。Waymo がローリングシャッター補正まで入れたのは、まさにこのオーダーの誤差を消すためです。高速道路 (時速108km = 秒速30m) では同じ16.7ミリ秒が50cm に拡大します。
- 時刻同期の質と検出性能: 同期がずれたデータで学習・評価すると、LiDAR 点群とカメラ画像の対応がずれ、センサーフュージョン型の物体検出 (例: PointPillars, Lang et al., CVPR 2019 のようなマルチモーダル拡張や、カメラ・LiDAR 融合手法) の精度が落ちることが知られています。融合手法は点群を画像に投影して特徴を貼り合わせるので、時刻ずれは投影位置のずれに直結します。nuScenes のように基準時刻に揃える設計は、この同期誤差を一定範囲に抑えることで、3D 検出の指標 mAP (mean Average Precision, 0〜1 で大きいほど良い。検出の適合率と再現率を総合した値) を安定して測れるようにしています。逆に言えば、同期が悪いデータセットでは、モデルの良し悪し以前にラベルとセンサーの対応が崩れて評価が信頼できなくなります。
- アブレーション的観点: 補間や補正を入れない単純最近傍は実装コストがほぼゼロですが、高速・高頻度シーンで誤差が増えます。Waymo の行ごと露光時刻のような精密同期は、低速の市街地では効果が小さく、高速道路や近接物体で効きます。「どこまで凝るか」はシーンの速度分布とタスクの要求精度で決めるべき、というのが実務的な結論です。

## メリット・トレードオフ・限界

メリット

- 実装が極めて単純 (引き算と argmin だけ) で、依存も少なく、バグが入りにくい。
- 共通のマイクロ秒・クリップ相対時刻という同じ物差しで比較するので、センサーごとのレート差を意識せず一律に扱える。
- タイムスタンプ配列をキャッシュすれば、データ取り出しのたびにファイルを読み直さず高速。ソート済みなら二分探索で $O(\log N)$。

トレードオフ・限界

- 最近傍を選ぶだけなので、最大でフレーム間隔の半分 (30Hz で約16.7ms) のズレが残る。高速走行時は画像と運動に無視できない位置ズレ (時速72km で約33cm) が出る。
- フレーム間の補間 (2枚を混ぜて中間時刻の画像を作る、または前後フレームから運動を補間する) はしないので、滑らかさより単純さを優先している。
- ローリングシャッター補正をしないので、1枚の画像内の時刻差 (上端と下端で数ミリ秒) は無視される。
- 複数カメラ間でも各々独立に最近傍を選ぶため、カメラ同士の時刻が完全には揃わない。
- タイムスタンプが単調増加でないクリップはスキップする方針なので、壊れたメタデータのクリップは丸ごと使えなくなる (データ歩留まりの低下)。

未解決の課題として、(1) 学習可能な時刻オフセット推定でハードウェア同期の不完全さを自動補正する、(2) イベントカメラ (画素ごとに輝度変化を時刻分解能マイクロ秒オーダーで非同期に出力するセンサー) との融合で同期問題そのものを軽くする、といった研究方向があります。

## 発展トピック・研究の最前線

- ハードウェア同期 vs ソフトウェア同期: KITTI のようにセンサーを物理的にトリガーで揃える方式は精度が高いが、ハードウェア構成 (共通クロック線、トリガー配線) に縛られます。後処理マッチングは柔軟だが誤差が残ります。両者のハイブリッド (粗くハードウェアで揃え、細かくソフトウェアで補正) が実用車では一般的です。
- 連続時間表現: 離散フレームではなく、連続時間バッチ推定 (Furgale et al., Kalibr の理論基盤) や Neural ODE で「任意時刻の状態」を扱えれば、最近傍マッチングのズレを根本的に減らせます。スプライン (滑らかな補間曲線) で軌道を連続関数として表し、任意時刻のセンサーと整合させる方式です。
- 時空間キャリブレーション: 時刻同期 (temporal calibration) と、センサー間の位置・姿勢合わせ (spatial/extrinsic calibration) は本来セットで最適化すべき問題です。Kalibr (Furgale et al., IROS 2013) はカメラ・IMU の外部パラメータと時刻オフセットを同時推定する代表的ツールで、両者を分けて解くより精度が高いことを示しました ([カメラ校正 (内部/外部パラメータ)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0938_camera-calibration-intrinsics-extrinsics))。
