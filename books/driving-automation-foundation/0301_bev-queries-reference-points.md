# BEVクエリと3D参照点・pillarサンプリング

## ひとことで言うと

鳥瞰図(上から見た地図状の特徴マップ)の各マス目に「自分は3D空間のどこを見ればいいか」を表す学習可能な問い合わせベクトル(BEVクエリ)を持たせ、そのマスの真上に縦に積んだ複数の高さの点(pillar、柱)を使って周囲のカメラ画像へ問い合わせる仕組みです。1マスを複数の高さで見ることで、深度(奥行き)を別に予測する専用モジュールを使わなくても「どの高さに物がいるか」を暗黙的に学習できます。多視点カメラから鳥瞰図を作るカメラベース知覚の代表的な設計のひとつです。

## 直感的な理解

自動運転では、複数のカメラ(前後左右に取り付けた6〜8台が典型)が撮る別々の画像を、1枚の統一された「上から見た地図」にまとめたい場面が多くあります。なぜなら、運転の判断(どこを走るか、何を避けるか)は周囲を俯瞰した座標系で考えるのが自然だからです。この俯瞰地図を鳥瞰図(Bird's Eye View, BEV)と呼びます。

ところが各カメラ画像は2Dで、奥行きが失われています。「画像のこのピクセルに写った車は、手前5mなのか奥30mなのか」が分からないと、鳥瞰図のどのマスに置けばいいか決められません。これが多視点カメラ知覚の根本問題です。

解き方には大きく2つの流派があります。ひとつは「各ピクセルの奥行きを予測して、その奥行きに従って3Dへ持ち上げる」前向きの方法(Lift-Splat-Shoot 系)。もうひとつは「鳥瞰図のマスが3Dのどこにあるかは設計上わかっているので、その3D点をカメラに投影し、当たったピクセルの特徴を取ってくる」後ろ向きの方法(BEVFormer 系)です。本ノートは後者を扱います。後者の利点は、奥行きを直接当てなくてよいことです。

さらに、ひとつのマスの上には地面の白線(低い)、車のボディ(中くらい)、信号機(高い)など、異なる高さの物が並びます。マスがひとつの高さだけ見るとその高さの物しか拾えません。そこでマスの上に複数の高さの点を縦に積んで(これが pillar、柱)、すべての高さに問い合わせ、最後に「どの高さが重要か」をアテンションで選ばせます。これが「暗黙の深度学習」の正体です。

## 基礎: 前提となる概念

**鳥瞰図(BEV)特徴マップ**: `[B, C, H, W]` の形をした、上空視点の格子データです。B はバッチ、C はチャネル(各マスの特徴次元、たとえば 256)、H×W は地面を区切ったマス目の数です。たとえば H=450, W=300 なら 135,000 マスで、前後・左右の各位置に256次元の特徴が入ります。

**学習可能埋め込み(learnable embedding)**: `nn.Embedding(N, D)` は N 個の D 次元ベクトルを学習で獲得するテーブルです。BEVクエリはこれで作り、各マスに「私はどんな特徴を見たいか」という問い合わせベクトルを持たせます。最初はランダムですが、学習が進むと位置ごとに役割を獲得します。

**カメラの射影行列(projection matrix)**: 3D空間の点をカメラ画像の2Dピクセルへ変換する行列です。カメラの位置・向き(外部パラメータ、extrinsics)と焦点距離・主点(内部パラメータ、intrinsics)から作られ、`[3, 4]` の形をします。3D点を同次座標 `[x, y, z, 1]` にしてこの行列を掛けると、奥行きでスケールされたピクセル座標が得られます。これを得るにはカメラ校正(キャリブレーション、各カメラの正確な位置と特性の測定)が必須です。

**透視除算(perspective division)**: 射影行列を掛けた結果 `[u', v', w']` の u', v' を w'(奥行きに相当)で割ると、最終的なピクセル座標 `[u, v] = [u'/w', v'/w']` になります。w' が正なら点はカメラの前方、負や0なら背後や同一平面で、可視判定に使います。

**点群範囲(point cloud range, pc_range)**: 鳥瞰図が現実世界の何メートル四方をカバーするかの定義です。`[x_min, y_min, z_min, x_max, y_max, z_max]` の6数で与え、正規化座標 `[0,1]` と実メートル座標を行き来する変換に使います。

## 仕組みを詳しく

多視点融合の典型的な実装を段階的に追います。

**1. BEVクエリの用意**

```python
self.bev_queries = nn.Embedding(bev_h * bev_w, embed_dim)  # 例: 450*300 個, 各256次元
```

鳥瞰図の各マスに1つずつ学習可能な256次元ベクトルを割り当てます。450×300 なら 135,000 個です。

**2. 3D参照点(pillar)の生成**

各マスに縦の点列(pillar)を割り当てます。`num_points_in_pillar=4` なら1マスにつき高さ方向に4点を置きます。

```python
xs = linspace(0.5, bev_w-0.5, bev_w) / bev_w   # 正規化した前後位置 [0,1]
ys = linspace(0.5, bev_h-0.5, bev_h) / bev_h   # 正規化した左右位置 [0,1]
zs = linspace(0.5, 4-0.5, 4) / 4               # 正規化した高さ 4段 [0,1]
```

`meshgrid` で全組み合わせを作ると `reference_points_3d: [bev_h*bev_w, 4, 3]` を得ます。「135,000マス × 各4高さ × (x,y,z)」の正規化3D点です。`0.5` 始まりはマスの中心を指すための半マスオフセットです(端ではなく中央を代表点にする)。

**3. 正規化座標から実世界座標へ**

`[0,1]` の正規化座標を pc_range で実メートルに直します。

```python
ref_world[...,0] = ref[...,0] * (x_max - x_min) + x_min   # 例 X: -60m〜120m
ref_world[...,1] = ref[...,1] * (y_max - y_min) + y_min   # 例 Y: -60m〜60m
ref_world[...,2] = ref[...,2] * (z_max - z_min) + z_min   # 例 Z: -5m〜3m
```

4つの高さ点が Z=-5m〜3m に等間隔で並ぶので、路面付近から車高の上端あたりまでをカバーします。前後方向(X)を左右(Y)より広くとる非対称な範囲は、運転で前方をより遠くまで見る必要があるためです。

**4. カメラへの投影とサンプリング**

実世界の3D点を同次座標 `[x,y,z,1]` にし、`camera_params: [B, V, 3, 4]`(V はカメラ台数、ego座標→ピクセル座標の射影行列)で各カメラに投影します。透視除算でピクセル座標 `[u,v]` を得て、奥行きが正かつ画面内にある点だけを可視とします。そして可視な点について、微分可能なサンプリング(`F.grid_sample` による双線形補間)でカメラ特徴を取り出します。詳細は [grid_sampleによるdeformable cross-attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0303_deformable-cross-attention-grid-sample) を参照。

**5. 高さ方向のアテンション(暗黙の深度学習)**

4つの高さ点それぞれから特徴を取った後、各高さの重要度を softmax で重み付けします。

```python
attn_weights = attention_weights(queries).reshape(B, N, V, 4).softmax(dim=-1)
```

ここが暗黙の深度学習の核心です。あるマスに車があるなら、車のボディの高さに当たる点の重みが学習で大きくなり、白線があるなら地面の高さの点が大きくなります。深度ラベルを一切与えなくても、軌跡や検出などの最終的な学習信号から逆算して「どの高さを見るべきか」がアテンション重みに反映されます。深度を「確率分布として陽に出す」のではなく「どの高さの特徴を重視するかという重み」として暗黙に表現するわけです。

**数値例**

`V=8(8カメラ), bev_h=450, bev_w=300, num_points_in_pillar=4` のとき:
- 参照点総数: `450 × 300 × 4 = 540,000` 個の3D点
- 各点を8カメラに投影: `540,000 × 8 = 4,320,000` 回の投影
- ただし可視マスクで大半は枠外として除外され、各マスは実際に映っているカメラ・高さの特徴だけを集約します。前方マスは前カメラに、後方マスは後カメラにしか映らないため、有効な投影は総数のごく一部です。

```
[BEVマス (h,w)]
      |  上に4点積む (pillar)
      v
   z=3m  o ──投影──> カメラ画像のピクセル ──grid_sample──> 特徴
   z=1m  o ──投影──>      〃
   z=-1m o ──投影──>      〃
   z=-5m o ──投影──>      〃 (地面)
                          ↓ softmax で高さ重み付け集約
                     このマスの BEV 特徴 (256次元)
```

## 手法の系譜と主要論文

- **OFT / Orthographic Feature Transform**(Roddick et al., BMVC 2019, arXiv:1811.08188): 画像特徴を鳥瞰図の格子へ写す初期の試み。各 BEV セルに対応する画像領域を積分して特徴を集める発想を示し、後の「BEVクエリが画像を見に行く」方式の先駆けです。

- **Lift-Splat-Shoot / LSS**(Philion & Fidler, ECCV 2020, arXiv:2008.05711): 各ピクセルに離散的な深度分布を陽に予測し(Lift)、その分布に従って画像特徴を3D空間にばらまき(Splat)、鳥瞰図に集約する前向き手法です。幾何が明示的で扱いやすい一方、深度予測の誤差が鳥瞰図に直接乗る欠点があります。pillar+アテンション方式はこの深度予測を回避する選択です。

- **DETR3D**(Wang et al., CoRL 2021, arXiv:2110.06922): 3D参照点を各カメラに投影して特徴をサンプルする「3D-to-2D query」の考え方を確立しました。疎な物体クエリが3D空間に参照点を持ち、それを画像へ投影して特徴を集めます。pillar サンプリングはこの発想を密な BEV グリッドへ広げた発展形と位置づけられます。

- **BEVFormer**(Li et al., ECCV 2022, arXiv:2203.17270): 学習可能な BEV クエリと、各クエリに対応する pillar 状の3D参照点を導入しました。pillar の各点を全カメラに投影し、当たったカメラの特徴を deformable な spatial cross-attention で集約します。深度を陽に予測せず、幾何で位置を決めつつ高さ方向の曖昧さをアテンションで吸収します。さらに過去フレームの BEV を参照する temporal self-attention で時系列も扱います。

- **BEVDepth**(Li et al., AAAI 2023, arXiv:2206.10092): LSS 系に立ち戻り、LiDAR を教師として深度予測を明示的に監督することで、深度の質が鳥瞰図精度を支配することを定量的に示しました。「深度を陽に当てるか暗黙に学ぶか」という2流派の対比を最も明確にした研究です。

この系譜は「画像→BEV 写像の素朴な試み(OFT) → 陽な深度持ち上げ(LSS) → 3D参照点投影(DETR3D) → 密な BEV クエリ+pillar(BEVFormer) ⇔ 陽な深度監督(BEVDepth)」と、前向き派と後ろ向き派が並走しながら発展しています。

## 論文の実験結果(定量データ)

評価は主に nuScenes(6カメラ・1台のLiDAR・全方位レーダーを備えた都市走行データセット、1000シーン)で行われます。代表的な指標は次の2つです。

- **mAP(mean Average Precision)**: 物体検出の平均精度。予測した3D枠が正解とどれだけ重なるかをクラス平均した0〜1の値で、高いほど良い。
- **NDS(nuScenes Detection Score)**: mAP に位置・大きさ・向き・速度・属性の誤差を加味した総合スコア。0〜1で高いほど良く、nuScenes での標準的な総合指標です。

報告されているおおよその数値:
- BEVFormer はカメラのみで NDS 約 0.51、mAP 約 0.42(ResNet101 系バックボーン、過去フレームを使う設定)を達成し、当時のカメラ単体手法の最上位に入りました。temporal self-attention(過去 BEV の参照)を加えると、特に速度誤差(mAVE)が大きく下がる(物体の動きを時間差から推定できるため)と報告されています。
- アブレーションでは、spatial cross-attention(pillar サンプリング)を素朴な全画素アテンションに置き換えると計算量が跳ね上がる一方で精度向上は限定的で、deformable な疎サンプリングが効率と精度を両立していることが示されています。pillar の点数(num_points_in_pillar)を増やすと高さ方向の表現力が上がり精度がわずかに改善しますが、計算量と引き換えで、4点前後で実用的な飽和に達するのが一般的な傾向です。
- BEVDepth は明示的な深度監督を加えることで、同等のバックボーンの LSS 系ベースラインに対し NDS を数ポイント(報告で 0.47→0.52 規模)押し上げました。これは「深度を正しく当てられれば BEV 精度が直接上がる」ことの定量的証拠であり、逆に言えば暗黙手法は深度教師がある状況では幾何精度で劣りうることを示唆します。

指標差の意味: NDS が 0.01 上がるとは、検出の見落とし・誤検出・位置や速度の誤差が総合的に縮むことを意味し、nuScenes リーダーボードでは順位を大きく動かす差です。速度誤差(mAVE)の改善は、衝突予測や経路計画の安全余裕に直結するため特に重視されます。

## メリット・トレードオフ・限界

メリット:
- 深度を陽に予測しなくても、高さ方向アテンションで暗黙に学習できる。深度予測モジュールも LiDAR 等の深度教師も不要。
- pillar で複数高さを見るため、地面の白線から信号機まで異なる高さの物体を1マスで扱える。
- 各マスが独立した学習可能クエリを持つので、場所ごとに見るべき特徴を変えられる(前方は遠方、側方は近接など)。

トレードオフ・限界:
- カメラ校正(camera_params)が必須。校正がずれると投影位置がずれ、誤った特徴を拾う。校正なしでは意味のある投影ができない。
- 深度を陽に持たないため、深度教師がある状況では LSS/BEVDepth 系より幾何精度で劣る可能性がある。
- `num_points_in_pillar` を増やすと表現力は上がるが計算量・メモリが線形に増える。実装では4点前後に抑えるのが定石。
- 投影が幾何に強く依存するため、未知の高さの物体(極端に高い/低い)が pillar の範囲外にあると拾えない。pc_range の Z 範囲設計が知覚の上限を決める。

未解決の研究課題: pillar の高さ点を等間隔ではなく学習で適応配置する、暗黙手法に弱い深度監督を補助損失として加えてハイブリッド化する、校正誤差にロバストな投影など、暗黙派と陽派の長所を統合する方向が活発に研究されています。

## 発展トピック・研究の最前線

- **時系列 BEV(temporal BEV)**: BEVFormer の temporal self-attention、BEVFormer v2、SOLOFusion など、過去フレームの BEV を蓄積・整列して速度推定と遮蔽補完を改善する流れ。長時間の履歴を効率よく保持する手法が焦点です。
- **疎クエリ系(sparse query)**: 密な BEV グリッドの計算量を避け、物体に対応する疎な3Dクエリだけを保持する Sparse4D / StreamPETR。pillar サンプリングの考え方を疎クエリに適用し、効率を大きく上げています。
- **オクパンシー予測(occupancy prediction)**: BEV の2Dグリッドを3Dボクセルへ拡張し、空間を「占有/非占有」で密に予測する流れ(Occ3D, TPVFormer など)。pillar の高さ離散化を本格的な3D表現へ発展させたものと見ることができます。
