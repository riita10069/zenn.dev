# ラスタライズドマップエンコーダ

## ひとことで言うと

ナビ用の進路を描いた俯瞰(BEV: Bird's Eye View、真上から見下ろした視点)の地図画像を、画像認識ネットワーク(畳み込みや Transformer 系のバックボーン)に通して、空間的な特徴の地図に変換するモジュールです。出力の形を `(B, C, h, w)` という4次元テンソルにそろえ、カメラから作った BEV 特徴とまったく同じ形にします。同じ形にそろえることで、後段でこの2つを直接足したり、注意機構で混ぜたりできるようになります。これは「地図というモダリティ(情報の種類)を、カメラと同じ土俵に乗せる」ための前処理だと考えてください。

## 直感的な理解

カメラ映像だけを見ている自動運転モデルを想像してください。前方に交差点があるのは見えますが、「ここで右に行くべきか、左に行くべきか、直進すべきか」はカメラ画像だけからは決まりません。これは人間でも同じで、初めて訪れる街では、目で見える景色に加えてカーナビの「次の交差点を右」という指示が要ります。自動運転モデルにとってのカーナビ指示にあたるのが、ナビ地図です。

ここで問題になるのが、地図画像という生のピクセルを、そのままカメラ特徴と混ぜられないことです。地図画像はたとえば `(B, 3, 256, 256)`(B はバッチ数=同時に処理する枚数、3 は RGB の3色、256×256 はピクセル解像度)という形をしています。一方カメラから作った BEV 特徴は `(B, 256, h, w)` のような形で、チャンネル数(3 対 256)も解像度(256×256 対 h×w)も違います。形が違うものは足せませんし、混ぜられません。

そこで「地図画像を、カメラ BEV 特徴とまったく同じ形 `(B, C, h, w)` の特徴マップ(feature map: 画像をネットワークに通したときの中間出力で、各位置がその場所の特徴ベクトルを持つ3次元配列)に変換する専用エンコーダ」が要ります。これがラスタライズドマップエンコーダです。形をそろえれば、後段の融合モジュール([マップBEV残差フュージョン（per-channel alpha）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0309_map-bev-residual-fusion) や [マップクロスアテンションフュージョン](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0310_map-cross-attention-fusion))でそのまま合流できます。

「ラスタライズド(rasterized)」という言葉が指すのは、地図を写真と同じピクセルの絵として描いておくことです。道路を灰色、進路線を青、横断歩道を白、というように色を割り当てて1枚の画像にする。すると地図を「自然画像と同じ画像処理」に乗せられます。この発想は軌跡予測の分野で確立しており、Cui et al.(ICRA 2019, arXiv:1809.10732)は周囲の道路と他車を多チャンネルのラスタ画像に描いて CNN に通し、将来軌跡を予測しました。地図をラスタにする最大の利点は、まさにこの「画像処理の道具一式がそのまま使える」ことにあります。

## 基礎: 前提となる概念

理解に必要な用語を一つずつ噛み砕きます。

- ラスタライズド地図(rasterized map): 地図を「ピクセルの絵」として表したもの。道路・進路・横断歩道などを色付きの画像で描いたものです。「ラスタ(raster)」とは画像をピクセルの格子として持つ方式を指す言葉です。対義語はベクトル地図(vectorized map)で、こちらは線や点の座標の集まりとして地図を表します(Lanelet2 や OpenDRIVE などの HD マップ形式)。ラスタは「写真と同じ画像処理に乗せやすい」のが利点、ベクトルは「車線中心線などの厳密な幾何を保持できる」のが利点です。
- エンコーダ(encoder): 生データ(ここでは画像)を、機械が扱いやすい数値の特徴ベクトルに変換する部品。逆にベクトルから出力を作る部品をデコーダと呼びます。
- バックボーン(backbone): 画像から特徴を抜き出す「土台」となるネットワーク。ResNet、Swin Transformer などが代表例です。多くの場合、画像分類などで事前学習した重みを使い回します。
- 特徴マップ(feature map): 画像をネットワークに通したときの出力で、`(チャンネル数 C, 高さ H, 幅 W)` の3次元配列(バッチを含めると4次元 `(B, C, H, W)`)。チャンネルとは「同じ画素位置に並ぶ複数の特徴量」で、各チャンネルが「縦エッジに反応する」「赤い領域に反応する」のように別々のパターンを捉えます。
- self-attention(自己注意): 画像や系列の中で、各部分が他の全ての部分を参照して「どこが自分に関係するか」を学習する仕組み。Transformer の核です。
- パッチ埋め込み(patch embedding): Vision Transformer が画像を小さな正方形パッチ(例: 4×4 ピクセル)に分割し、各パッチを1つのトークン(ベクトル)に変換する最初の層。
- 階層的特徴(hierarchical features): ネットワークの段(stage)が進むにつれ解像度を下げチャンネルを増やしていく構造。浅い段は細部、深い段は大域構造を捉えます。CNN や Swin が持つ性質です。
- プーリング(pooling): 特徴マップの空間解像度を縮める操作。max プーリングは領域内の最大値、average プーリングは平均値を取ります。

## 仕組みを詳しく

典型的な実装は、入力 `(B, 3, 256, 256)` の地図画像を受け取り、`(B, C, h, w)` の特徴マップを返します。256×256 固定なのは、よく使われるバックボーン(例: `swinv2_tiny_window8_256` のような Swin V2 Tiny の窓サイズ8・入力256版)のパッチ埋め込みが256入力を前提にしているためです。処理を段階的に追います。

### ステップ1: バックボーンで多段特徴を抽出

```python
backbone = timm.create_model(
    "swinv2_tiny_window8_256",
    pretrained=False, features_only=True, in_chans=3,
)
```

Swin Transformer V2 は画像を「窓(window)」に区切って、その中だけで self-attention を計算する Vision Transformer です。窓に区切ることで、画像全体で attention を計算する素の ViT(計算量が画像サイズの2乗)よりも軽くなります。`features_only=True` は最終分類層を捨て、各段(stage)の途中特徴をすべて出してもらう設定です。

`pretrained=False`(重みをランダム初期化する)選択がよく取られるのは、地図画像が自然写真と見た目が大きく違うためです。ImageNet などの自然画像で事前学習した重みは「猫や犬や木」のパターンを学んでおり、単色の道路や青いルート線で構成された地図とは統計が合いません。地図でゼロから学ぶ方が素直という判断です。ただしこれはトレードオフで、ランダム初期化は学習データ量への依存を高めます(後述)。

Swin-Tiny は典型的に4段あり、各段のチャンネル数はおおむね `[96, 192, 384, 768]`。段が進むほど解像度が下がりチャンネルが増えます(これが「階層的特徴」です)。各段のチャンネル数を読み、合計 `96+192+384+768 = 1440` を後で使います。浅い段は細かいテクスチャ(線の縁など)、深い段は大域的な構造(交差点の形など)を捉えます。

### ステップ2: 各段を共通解像度へプール

各段の特徴マップは解像度がバラバラ(64×64, 32×32, 16×16, 8×8 など)です。これを `nn.AdaptiveMaxPool2d((h, w))` で全部同じ `(h, w)` に縮めます。AdaptiveMaxPool は「入力サイズに関係なく、出力を指定サイズにする」プーリングで、各出力セルが対応する領域の最大値を取ります(max を取るので、その領域で最も強く反応した特徴が残ります)。

実装上の注意として、timm の Swin は出力テンソルがチャンネル最後の `(B, H, W, C)` で返ることがあります。これを検出したら `permute(0, 3, 1, 2)` で `(B, C, H, W)` に直してからプールする必要があります。permute はテンソルの軸の順番を入れ替える操作です。

```python
pooled = [pool(f) for f in features]   # 各段 (B, C_i, h, w)
x = torch.cat(pooled, dim=1)           # (B, 1440, h, w)  チャンネル方向に連結
```

### ステップ3: 1×1 畳み込みでチャンネルを C にそろえる

```python
channel_proj = nn.Sequential(
    nn.Conv2d(1440, C, kernel_size=1),   # 例: C=256
    nn.GELU(),
)
out = channel_proj(x)   # (B, C, h, w)
```

1×1 畳み込みは「各位置のチャンネル方向だけを混ぜて次元を変える全結合層」です(空間的には近傍を見ず、その1点のチャンネルベクトルを線形変換するだけ)。1440 チャンネルを目的の C(典型的には256)に圧縮し、GELU(なめらかな活性化関数。ReLU の滑らか版で、負の値を完全には捨てない)を通します。

### 形状の流れ(具体例: h=w=8 の場合)

```
入力 map_image: (B, 3, 256, 256)
  → Swin 4段: [(B,96,64,64), (B,192,32,32), (B,384,16,16), (B,768,8,8)]
  → AdaptiveMaxPool((8,8)): 各段 (B, C_i, 8, 8)
  → concat(dim=1): (B, 1440, 8, 8)
  → 1x1 Conv → GELU: (B, 256, 8, 8)
出力: (B, 256, 8, 8)  ← カメラ BEV 特徴と同形状
```

出力解像度 `(h, w)` は呼び出し側が、カメラ側の特徴解像度に合わせて指定します。これにより両者が確実に同形状にそろい、後段の融合が成立します。

### なぜ「多段を連結」するのか

1段だけ(たとえば最深段 768ch)を使う設計もあり得ます。しかし多段を連結すると、解像度の異なる情報を1つの特徴に同居させられます。浅い段は8×8に強く縮めても「線がどの向きに走っているか」のような細部の手がかりを残し、深い段は「ここが交差点」という大域パターンを持ちます。これは物体検出の Feature Pyramid Network(Lin et al., CVPR 2017)が示した「多スケール特徴を混ぜると小物体も大物体も拾える」原理の、地図版の応用です。地図には太い道路と細いルート線が混在するので、多スケールを残す価値があります。

## 手法の系譜と主要論文

ナビ地図やルート情報を別ブランチで符号化して運転モデルに与える発想は、End-to-End 自動運転の中で発展してきました。

- Codevilla et al., "End-to-end Driving via Conditional Imitation Learning"(ICRA 2018, arXiv:1710.02410): ナビ指示(直進/右折/左折)を「コマンド」として与え、コマンドごとに別々の制御ヘッドを切り替える条件付き模倣学習を提案。これが「外部のナビ意図を運転モデルに注入する」初期の代表例です。動機は、同じ景色でもナビ意図が違えば取るべき行動が違う、という多義性の解消でした。コマンドという離散信号だった意図表現が、後にラスタ地図のような連続的・空間的な表現へと発展していきます。

- Cui et al., "Multimodal Trajectory Predictions ... using Deep Convolutional Networks"(ICRA 2019, arXiv:1809.10732): 周囲の HD マップと他車をラスタ画像に描いて CNN に通し、複数候補の将来軌跡を予測。ラスタ地図 + 画像バックボーンという、本トピックと同じパイプラインを軌跡予測で確立した代表例です。

- Wu et al., "TCP: Trajectory-guided Control Prediction"(NeurIPS 2022, arXiv:2206.08129): 目標地点やルート情報を別経路で取り込み、軌跡予測と制御予測を統合します。CARLA(自動運転シミュレータ)のリーダーボードで上位に立ち、軌跡枝と制御枝を相互補完させる設計が効いたと報告されました。

- Hu et al., "Planning-oriented Autonomous Driving (UniAD)"(CVPR 2023 Best Paper, arXiv:2212.10156): 知覚・予測・計画を1つの Transformer で統合し、最終目的(計画)に向けて各タスクを協調させる枠組み。ルートや目標情報を含む文脈を運転意思決定に効かせる流れの到達点の一つです。

地図表現そのものについては、ラスタライズド対ベクトルの論争があります。

- Gao et al., "VectorNet"(CVPR 2020, arXiv:2005.04259): 軌跡予測の文脈で「ラスタライズドはピクセルにすると幾何情報が薄まり、計算も重い」と批判し、車線や物体を折れ線(polyline)のベクトルとして符号化する手法を提案しました。Argoverse の軌跡予測でラスタ手法を上回り、計算コストも削減できることを示しました。一方でラスタライズドは「実装が単純でカメラ画像と同じ画像処理に乗せやすい」利点があり、両者はトレードオフの関係にあります。

- バックボーンの Swin Transformer V2 は Liu et al., "Swin Transformer V2: Scaling Up Capacity and Resolution"(CVPR 2022, arXiv:2111.09883)。窓ベースの self-attention で計算量を画像サイズに線形にしつつ、scaled cosine attention(注意スコアを正規化して大規模化時の不安定を防ぐ)と log-spaced continuous position bias(解像度を変えても位置情報が破綻しない相対位置バイアス)を導入しました。これにより事前学習と微調整で解像度が変わっても安定します。

なお、カメラから BEV 特徴を作る側の代表手法として Philion & Fidler, "Lift-Splat-Shoot"(ECCV 2020, arXiv:2008.05711)があり、各カメラ画素に深度分布を予測して3次元に持ち上げ(lift)、BEV 平面に貼り付け(splat)ます。地図エンコーダはこのカメラ BEV と同じ座標系・同じ形に出力をそろえることで融合を可能にします。

## 論文の実験結果(定量データ)

ラスタ対ベクトルの代表的な比較として VectorNet(CVPR 2020)の報告があります。Argoverse Forecasting ベンチマーク(車両の3秒後までの軌跡を予測する課題)で、評価指標は DE(Displacement Error: 予測軌跡と正解軌跡の位置ずれをメートルで測る。小さいほど良い)と minADE/minFDE(複数候補のうち最良のものの平均誤差/最終位置誤差)です。VectorNet はラスタ化 + ConvNet のベースラインに対して予測誤差を下げつつ、モデルの計算量(FLOPs)を大幅に削減できたと報告しました(報告では同等以上の精度を約1桁少ない計算量で達成)。これは「幾何が重要な軌跡予測では、ピクセルにせず座標のまま扱う方が情報損失が少ない」ことを示す典型例です。

逆に、End-to-End 運転で「ルート/地図情報を入れる効果」については CARLA リーダーボードでの一連の報告があります。TCP(NeurIPS 2022)は、ルート情報を活用した軌跡誘導により、Driving Score(走破率 Route Completion と違反ペナルティ Infraction を掛け合わせた総合指標。100点満点に近いほど良い)で当時の最上位級を達成しました。ナビ意図を抜くと交差点での進路選択が定まらず Route Completion が落ちる、という効果は条件付き模倣学習(ICRA 2018)以来繰り返し確認されています。ICRA 2018 の原論文では、ナビ条件を与えない素朴な模倣学習だと交差点での進路判断が不安定になり、条件付けによって成功率が大きく上がったと報告されています。

Swin V2 の貢献は ImageNet-V2 などでの分類精度と、何より「高解像度への転移の安定性」です。論文では最大30億パラメータのモデルを 1536×1536 解像度まで安定して学習できることを示し、scaled cosine attention を外すと大規模化時に attention が一部のトークンに偏って学習が崩れる、というアブレーション(構成要素を抜いて効果を確かめる実験)を報告しています。地図エンコーダで Swin V2 を選ぶ実利は、出力解像度を後段に合わせて変えても表現が破綻しにくい点にあります。

定量の意味を初心者向けに補足します。minADE が 1.0m から 0.8m に下がる、というのは「3秒後の予測位置が平均20cm 正解に近づいた」ことを意味し、車線幅が約3.5m であることを思えば、車線内に収まるか隣車線にはみ出すかを分ける実用的な差です。Driving Score が10点上がるのは、同じ100ルートを走らせて衝突や信号無視が目に見えて減ることを意味します。

## メリット・トレードオフ・限界

メリット
- 地図画像をカメラ BEV 特徴と同形状の特徴マップに変換でき、足し算や注意機構で直接融合できる。
- 多段(階層的)特徴を結合するので、細かい線から大域的な交差点構造まで多スケールで拾える。
- ラスタライズドなのでカメラ画像と同じ画像処理パイプラインに素直に乗る。実装と保守が容易。
- バックボーンを差し替えやすく、レジストリ設計と組み合わせれば別の地図エンコーダへ簡単に切り替えられる([マップフュージョンレジストリ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0980_map-fusion-registry))。

トレードオフ・限界
- 入力解像度がバックボーンに縛られる(パッチ埋め込みの制約で 256×256 固定など)。任意解像度を入れたい場合はリサイズや別バックボーンが要る。
- ラスタライズドは VectorNet が指摘したとおり、車線中心線などの厳密な幾何情報が量子化で薄まる。1ピクセルが実空間で何メートルかという解像度の壁があり、細い車線や近接した分岐が同じピクセルに潰れる。
- バックボーンをランダム初期化すると、性能が学習データ量に強く依存する。地図ドメインの大規模事前学習は自然画像ほど整備されていない。データが少ないとバックボーンが十分に学習されず、地図が役に立たないこともある。
- AdaptiveMaxPool で解像度を強く落とすため、細かい位置情報の一部が失われる。max プーリングは平均より局所のピークを保つが、空間的な精密さは犠牲になる。8×8 まで縮めると1セルが広い実空間領域に対応し、細かい進路の曲がりは表現できない。

研究上の未解決課題として、ラスタとベクトルのハイブリッド(ラスタの扱いやすさとベクトルの幾何精度を両立する表現)や、地図ドメイン専用の自己教師あり事前学習(ラベルなし地図画像から有用な初期重みを得る)、そして「地図のどの色チャンネルがどの意味か」という事前知識をネットワークにどう注入するか、があります。

## 発展トピック・研究の最前線

オンライン地図構築の流れが近年活発です。HDMapNet(Li et al., ICRA 2022, arXiv:2107.06307)や MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)は、事前作成の HD マップに頼らず、車載カメラからその場でベクトル地図(車線・縁石)を推定します。これにより「ラスタ地図を前段で用意する」必要自体が薄れ、地図エンコーダの役割が「推定された地図ベクトルを符号化する」方向に移りつつあります。

もう一つの最前線は、地図とカメラを共通の BEV クエリで統合する Transformer 系(BEVFormer など)で、地図側を明示的な幾何グリッドとして扱い、カメラ BEV と座標を厳密に合わせる方向です。地図エンコーダの出力をどの座標系・どの解像度に合わせるかが、後段の融合の成否を決めます。

さらに、地図情報を画像ではなくトークン列(車線ポリラインを点列のトークンで表す)として与え、Transformer のクロスアテンションで運転意思決定に効かせる方向(VectorNet 系から TNT, DenseTNT への流れ)も主流化しています。ラスタ地図エンコーダは「簡単で頑健」という立ち位置で、ベクトル系は「幾何精度が高い」という立ち位置で、両者は使い分けられています。
