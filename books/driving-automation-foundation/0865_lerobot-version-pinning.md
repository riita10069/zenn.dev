# LeRobot バージョン固定問題

## ひとことで言うと

機械学習のデータ読み込みでは、外部ライブラリの新しい版で内部のフォルダ構成（モジュール配置）が変わり、古い書き方のコードが突然動かなくなることがあります。たとえば、あるロボット学習ライブラリでは新版で `lerobot.common` という置き場所が廃止され、そこを参照していたコードが `No module named 'lerobot.common'`（そんなモジュールは無い）というエラーで壊れます。これを防ぐには「どの版を使うか」を明示的に固定（version pinning）し、必要ならコード側を複数版に対応させます。このノートは、なぜ依存ライブラリの版がズレると壊れるのか、その仕組みと対策、そして機械学習の再現性問題との関係を、ゼロから研究レベルまで掘り下げます。

## 直感的な理解

電化製品とコンセントの関係を思い浮かべてください。あなたの家電（コード）は、ある形のコンセント（ライブラリの内部構造）に合うプラグを持っています。ところがコンセント側が新しい規格に変わり、差し込み口の形が変わってしまうと、同じプラグはもう刺さりません。家電は壊れていないのに、つなぎ先が変わったせいで動かなくなる——これがライブラリの破壊的変更で起きることです。

ソフトウェアでは、「最新版を自動で取ってくる」設定にしていると、いつの間にかコンセントの規格が変わり、昨日まで動いていたコードが今日突然落ちます。しかも厄介なのは、これが「自分の PC では動くのに、他の人の環境では落ちる」という形で現れることです。原因は、人それぞれインストールした版が違い、コンセントの形が違うから。だから「全員が同じ版を使う」と約束すること（バージョン固定）が、再現性の土台になります。一人で開発しているときは「自分の環境では動く」ので問題に気づきにくく、共同作業や数か月後の自分が同じ環境を再構築しようとした瞬間に表面化する、時限爆弾のような性質を持ちます。

## 基礎: 前提となる概念

用語を噛み砕きます。

- 「ライブラリ（library）」とは、他人が作った便利な機能の詰め合わせで、`pip install` で自分のプロジェクトに取り込んで使うものです。自分で全部書かずに済む反面、その中身の変更に振り回される宿命があります。

- 「モジュール（module）」とは、Python のコードを置く部品（ファイルやフォルダ）の単位です。`import lerobot.common.datasets` と書くと「lerobot の中の common の中の datasets を持ってくる」という意味になります。この import 経路は、ライブラリ内部のフォルダ構成に直接対応しています。だからフォルダ構成が変わると import が壊れます。

- 「公開 API（public API）」と「内部実装（internal）」の区別が重要です。ライブラリが「これは使ってよい」と約束した入り口が公開 API、約束していない内部構造が internal です。`lerobot.common` のような中間階層を直接 import するのは internal に踏み込む行為で、作者は予告なく変えてよい部分です。だから internal に依存すると壊れやすい、という一般則があります。

- 「バージョン（version）」とは、ライブラリの版番号です。`5.0.1` のように付き、版が上がると機能追加だけでなく内部構造が変わることもあります。

- 「バージョン固定（version pinning）」とは、`lerobot==5.0.1` のように「この版を使う」と釘で留める（pin する）指定です。`==` は完全一致、`>=` は「これ以上」を意味します。

- 「破壊的変更（breaking change）」とは、新版で過去のコードが動かなくなる変更のことです。import 経路の変更、引数の廃止、関数名の変更、デフォルト挙動の変更などが典型です。

- 「依存（dependency）」とは、自分のプロジェクトが動くために必要な外部ライブラリのことです。`requirements.txt` や `pyproject.toml` といったファイルに一覧で書きます。

## 仕組みを詳しく

まず、何がどう変わって壊れるのかを before/after で示します。多くのライブラリは成長の過程で内部を再編成し、import 経路を変えます。

import 経路の変化の例:

```
旧版:  from lerobot.common.datasets.lerobot_dataset import LeRobotDataset
新版:  from lerobot.datasets.lerobot_dataset import LeRobotDataset
        （common という中間階層が消えた）
```

古いコードは消えた `common` を探しに行くため、利用者の環境にたまたま新しい版が入っていると `ModuleNotFoundError: No module named 'lerobot.common'` で即座に落ちます。重要なのは、これが連鎖するという点です。データ読み込み（データローダ）が落ちると、それを使う学習も評価も可視化も全部止まります。データローダは多くの処理の入り口（依存グラフの根に近い位置）なので、ここが壊れると下流が芋づる式に停止します。一箇所の import エラーが、システム全体を起動不能にするわけです。

なぜ「版を明記する必要がある」のか。依存一覧に単に `lerobot` とだけ書くと、`pip install` が「その時点の最新版」を勝手に入れます。すると人によって入る版がバラバラになり、ある人の環境では動き別の人では壊れる、という再現性の崩壊が起きます。これが "works on my machine（自分の環境では動く）" 問題の典型です。版を明記すれば、全員が同じ版・同じ内部構成を前提にできます。

この問題への現実的な対処は2層あります。

1つ目は「版を固定する」こと。`requirements.txt` などに `lerobot==X.Y.Z` と書き、コードが想定する構成と一致する版に揃えます。効果は再現性の確保。トレードオフは、固定したまま放置すると新版の改善や脆弱性修正を受け取れず、いつか上げる作業（しかもそのとき破壊的変更がまとめて降ってくる）が必要になることです。

2つ目は「コード側を複数版に対応させる」こと。代表的なのが try/except によるフォールバック（失敗したら別経路を試す）です。

```python
try:
    from lerobot.datasets.lerobot_dataset import LeRobotDataset
except ModuleNotFoundError:
    # 旧版や派生パッケージが同じクラスを別の名前空間で公開している場合に備える
    from lerobot.common.datasets.lerobot_dataset import LeRobotDataset
```

これは「まず新版の正規の場所を試し、見つからなければ旧版・別パッケージの同名クラスを使う」という二段構えです。`try/except ModuleNotFoundError` は「import してみて、モジュールが無ければ例外を捕まえて別の手を打つ」という Python の定石です。新旧どちらの環境でも動くので、移行期に有効です。

破壊的変更は import 経路だけでなく、関数の引数にも及びます。たとえば、ローカルのファイルだけを使いネットに取りに行かないことを指示する `local_files_only=True` のような引数が、ある版で廃止され、デフォルトでキャッシュ（一度落としたデータの保存場所）から読む挙動に変わる、ということが起きます。後方互換のために引数を受け取りつつ実際には何もしない no-op（ノーオペ、無効な引数）として吸収する、という互換コードがしばしば書かれます。これも「版が変わると引数すら変わる」ことの一例で、版固定が必要な理由を補強します。

before/after の効果まとめ:

- before: 依存が `lerobot` だけ → 最新版が入り、消えた `lerobot.common` を探して `ModuleNotFoundError`。データローダが止まり、学習・評価・可視化まで連鎖的に停止。
- after: 版を明記し、かつコードを新 import + フォールバックに修正 → 想定どおり読み込め、下流も復旧。

## 手法の系譜と主要論文

これはアルゴリズム論文ではなく、ソフトウェア工学と機械学習の再現性をめぐる知見に属します。系譜を整理します。

- 「セマンティックバージョニング（SemVer）」（Tom Preston-Werner, 2013, semver.org）は、版番号を `MAJOR.MINOR.PATCH` の3つに分け、後方互換を壊す変更のときだけ MAJOR を上げる、という約束事です。import 経路が変わる変更は本来 MAJOR の破壊的変更にあたります。教訓は、SemVer を信じて `lerobot>=X` と緩く指定していると、MAJOR が上がった瞬間に壊れる、ということ。一方で、すべてのライブラリが SemVer を厳密に守るわけではなく、MINOR でも実質破壊的変更が混じることがある点も実務上の現実です。

- 「再現可能なビルド／依存固定」の流れ。Python では PEP 440（バージョン指定の文法、`==` や `>=` の意味を定義）が標準で、`pip freeze` で全依存の版を書き出して固定する手法や、`pip-tools`、`poetry`、`uv` のロックファイル（厳密な版とハッシュを記録するファイル）が広く使われます。核心は「依存の版を明示して同一環境を再現する」こと。動機は、暗黙の最新版取得は時間とともに環境を変え、過去に動いたコードが将来動かなくなる（いわゆる bit rot、ビット腐敗）から。トレードオフは固定した版が古びるメンテ負荷です。ハッシュ付きロックは、同名同版でも中身がすり替わる供給網攻撃まで防ぐ点で、`==` 固定より一段強い保証になります。

- 「機械学習の再現性危機（reproducibility crisis）」の議論。Joelle Pineau ら "Improving Reproducibility in Machine Learning Research"（NeurIPS 2019 reproducibility program を経て JMLR 2021, arXiv:2003.12206）は、ML 研究で結果が再現できない大きな原因の一つに、ソフトウェア環境（ライブラリ版）の不一致を挙げました。処方箋は環境とコードを明記・公開すること（ML Reproducibility Checklist の提唱）。本件はその実例で、データローダのライブラリ版がズレるだけで実験が一切回らなくなることを示します。

## 論文の実験結果（定量データ）

依存の不一致がどれほど深刻かを示す経験的データがあります。

論文の再現性を、付随コードを実際に動かして検証した研究があります。Christian Collberg と Todd Proebsting の "Repeatability in Computer Systems Research"（CACM 2016）は、システム系の論文 600 件超を対象に、付属コードを入手し実際にビルド・実行できるかを試みました。結果、相当な割合（彼らの基準では半数前後）が「コードが手に入らない、またはビルドできない」ことが報告されました。ビルドできない原因の上位に「依存ライブラリの版が不明、または最新版で API が変わって壊れる」が必ず入ります。これは「依存固定の欠如が再現失敗の主要因である」という主張を数量的に裏付けます。

破壊的変更の頻度については、Steven Raemaekers らの "Semantic Versioning versus Breaking Changes: A Study of the Maven Repository"（ICSME 2014）が代表的です。Maven（Java のパッケージ基盤）の大量のライブラリ版を解析し、API シグネチャの変化を機械的に検出したところ、後方互換を壊す変更が MAJOR だけでなく MINOR や PATCH のリリースにも無視できない割合で混入していることを示しました。具体的には、本来後方互換を保つべきはずの MINOR リリースでも一定割合で破壊的変更が観測された、と報告しています。つまり「MINOR だから安全」という前提は経験的に裏切られることがあり、だからこそ厳密な `==` 固定やロックファイルが推奨される、という結論につながります。

これらの数値が示す教訓は明快です。再現性は「測れる量」で語れます。すなわち「同じ環境を再構築できる確率」「コードが別環境で初回に動く確率（first-run success rate）」です。版を緩く指定するとこの確率が時間とともに下がり、`==` で固定するとほぼ一定に保てます。アブレーション的に言えば、依存固定という一要素を抜くだけで、再現成功率が大きく落ちる、という関係です。

## メリット・トレードオフ・限界

メリット:

- 版を固定すれば全員が同じ内部構成を前提にでき、「自分の環境では動く」問題を防げる。再現性が上がる。
- コード側のフォールバック（try/except）で、複数版や派生パッケージにまたがって動かせる。移行期をまたげる。
- データローダの復旧により、連鎖的に止まっていた下流（学習・評価・可視化）をまとめて復旧できる。

トレードオフ・限界:

- 版を固定したまま放置すると、新版の改善・修正を受け取れず、いずれ上げる作業が必要。しかもそのとき破壊的変更がまとめて降ってきて、移行コストが一度に膨らむ（依存負債の蓄積）。
- フォールバックを増やすほどコードが複雑になり、どの経路が実際に動いているか追いにくくなる。テストで両経路をカバーしないと、片方の経路だけ壊れていても気づけない（テストが片肺飛行を見逃す）。
- 廃止された引数を no-op で吸収する互換コードは、利用者に「効いていない」ことが見えにくく、誤解を招きうる（指定したつもりの挙動が無視される、サイレントな振る舞い変化）。
- そもそも internal（公開 API でない内部構造）に依存していること自体が脆弱性の根。公開 API だけを使えば破壊的変更の頻度は下がるが、ライブラリが必要な入り口を公開していないと internal に手を出さざるを得ない。
- 根本的には上流ライブラリの破壊的変更に振り回される構造であり、下流（利用側）の努力には限界がある。上流の安定性が低いほど、この問題は再発する。

研究上の論点として、「破壊的変更を機械的に検出し、利用側に自動移行を促す」方向があります。API の差分を解析して壊れた呼び出しを検出する研究や、自動でコードを書き換える codemod ツール（Facebook の jscodeshift など）がありますが、import 経路の変更のような構造的変化を完全自動で安全に追従するのは難しく、人手の確認が残ります。

## 発展トピック・研究の最前線

依存管理の最前線では、ロックファイルとコンテナ（Docker など）を組み合わせ、ライブラリだけでなく OS レベルまで含めた環境全体を固定する手法が標準化しつつあります。さらに Nix のようなツールは、依存の完全なハッシュベース固定で「ビット単位で再現できるビルド（bit-reproducible build）」を目指しています。機械学習では、データ・コード・環境・乱数シードまで含めて固定する「完全再現パッケージ」を論文に添付する文化が広がっており、NeurIPS などの主要会議が再現性チェックリストの提出を求めるようになりました。

また、依存の脆弱性や破壊的変更を継続監視し、安全なものだけ自動で上げるツール（Dependabot, Renovate など）が普及しています。これらは「固定すると古びる」というトレードオフに対し、「固定しつつ、テストが通る範囲で小刻みに上げる」という折衷を自動化する試みです。テストスイートが厚いほどこの自動更新は安全になり、薄いほど破壊的変更を見逃す——ここでも再現性とテストは表裏一体です。上流の破壊的変更にどう追従するかは、いまも実務と研究の両方で進行中のテーマです。
