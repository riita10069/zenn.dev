# MobileNet (V1/V2/V3)

## ひとことで言うと
MobileNet は「スマホや車載コンピュータのような計算力の小さい機械でも動くように、わざと軽く作った画像認識ニューラルネットワーク」です。画像から特徴を取り出す「バックボーン(背骨)」と呼ばれる土台部分を、計算量を大幅に減らす工夫で作り直したものです。V1・V2・V3 と世代を重ねるごとに、同じ精度をより少ない計算で出せるよう改良されてきました。今日でもエッジ(端末側)推論の標準的な選択肢の1つで、検出・領域分割の軽量バックボーンとしても広く使われます。

## 直感的な理解
画像認識の高精度モデル(ResNet や VGG など)は、画像 1 枚を処理するのに数十億回から数百億回の掛け算を必要とします。データセンターの強力な GPU なら一瞬ですが、スマホや自動運転車に載る小さな組込みコンピュータ(エッジデバイス。「クラウドの端=端=エッジで計算する機械」の意味)では遅すぎて使い物になりません。

なぜエッジで動かす必要があるのか。クラウドへ送って計算してもらう手もありますが、通信の往復に時間がかかり、電波が届かない場所では止まり、プライバシー上の懸念もあります。自動運転ではカメラ映像をミリ秒単位でリアルタイムに処理し続ける必要があり、たとえば 1 フレームに 500 ミリ秒かかると、時速 60km(秒速約 17m)の車は処理を待つ間に 8.5m も進んでしまいます。だからこそ「精度を保ったまま計算量を 1/8〜1/10 に減らす」技術が強く求められました。MobileNet はその代表格で、「少ない掛け算で同じ仕事をする部品」を発明したことが核心です。

## 基礎: 前提となる概念
用語を平易に言い換えます。

CNN(畳み込みニューラルネットワーク)とは、画像を小さな窓(フィルタ)でスキャンしながら、低レベルの特徴(縦線・エッジ)から高レベルの特徴(タイヤ・顔)へと段階的に拾い上げる仕組みです。

畳み込み(convolution)とは、小さなフィルタ(例えば 3×3 の数値の表)を画像上で滑らせ、重なった部分で掛けて足す計算を繰り返す操作です。CNN の計算量の大半を占めます。

チャネル(channel)とは、特徴マップの「層の枚数」です。入力画像なら赤緑青の 3 チャネル、中間層では数十〜数百チャネルあります。

バックボーンとは、認識システムの中で「画像から汎用的な特徴を取り出す土台」のことです。この上に「物体を検出する頭」「車線を引く頭」など用途別の部品(head)を載せて使います。

FLOPs / MACs という指標も押さえます。MAC は「掛けて足す演算 1 回(Multiply-Accumulate)」で、論文では MAdds とも書きます。FLOPs はおおむねその約 2 倍(掛け算と足し算を別に数える)です。どちらも理論計算量の指標で、モデルの重さを測る共通のものさしになります。ただし後述するように、理論計算量と実機速度は必ずしも一致しません。実機速度はメモリアクセスのコスト(MAC、Memory Access Cost)や並列度にも左右されます。

## 仕組みを詳しく

### V1 の核心: 深さ方向分離畳み込み (Depthwise Separable Convolution)
これが MobileNet 最大の発明です。普通の畳み込みを、2 つの安い操作に分解します。

普通の畳み込み(標準畳み込み)を具体例で見ます。入力が「縦 112 × 横 112 × チャネル 32」の特徴マップだとします。これを「出力チャネル 64」へ変換するのに 3×3 フィルタを使うと、必要な掛け算回数は次のとおりです。

```
標準畳み込み: 3 × 3 × 32(入力ch) × 64(出力ch) × 112 × 112(出力の各位置) ≈ 2.31 億回
```

深さ方向分離畳み込みは、これを 2 段に割ります。

1. Depthwise(深さ方向)畳み込み: 各チャネルを、それぞれ専用の 3×3 フィルタで別々に処理します。チャネルをまたいで混ぜません(赤の層は赤だけ、緑は緑だけ処理するイメージ)。空間方向の特徴抽出だけを担当します。
   ```
   3 × 3 × 32 × 112 × 112 ≈ 0.036 億回
   ```
2. Pointwise(点ごと)畳み込み: 1×1 フィルタで、各位置のチャネル同士を混ぜ合わせ、32 チャネルから 64 チャネルへ変換します。チャネル方向の混合だけを担当します。
   ```
   1 × 1 × 32 × 64 × 112 × 112 ≈ 0.257 億回
   ```

合計は約 0.293 億回で、標準畳み込みの 2.31 億回に対して約 1/8 になります。一般に出力チャネル数を N、フィルタを 3×3 とすると、計算量はおよそ「1/N + 1/9」倍に減ります。N が大きいほど 1/9(約 0.11 倍)に近づきます。つまり「空間方向に特徴を見る仕事(depthwise)」と「チャネルを混ぜる仕事(pointwise)」を分業させることで、ほぼ同じ表現力をはるかに安く実現したのが V1 です。

V1 にはさらに 2 つのつまみ(ハイパーパラメータ)があります。幅倍率 α(width multiplier)は全層のチャネル数を一律 α 倍(例 0.75)に減らすつまみで、計算量はおよそ α² に比例して減ります。解像度倍率 ρ(resolution multiplier)は入力画像を小さく(例 224→160)するつまみで、計算量はおよそ ρ² に比例して減ります。これで「精度は少し落ちていいから速く」という調整を 1 ダイヤルで行えます。

### V2 の核心: 逆ボトルネックと線形ボトルネック
V2 は 2 つの改良を加えました。

Inverted Residual(逆残差)構造。通常の ResNet のボトルネックは「太い → 細い → 太い」とチャネルを絞ってから戻しますが、V2 は逆に「細い → いったん太く広げる(expand)→ depthwise で処理 → また細く戻す(project)」という「細い → 太い → 細い」にしました。なぜ逆かというと、計算が重い depthwise はチャネルを広げた空間でやったほうが表現力が出る一方、層をまたいで運ぶデータ(残差接続でつなぐ部分)は細いほうがメモリと計算が安いからです。広げる倍率(expansion ratio)は通常 6 倍で、たとえば入力 24ch を 144ch に広げてから depthwise し、また 24ch に戻します。残差接続は細い 24ch どうしを結ぶので、推論時のメモリピークが小さく抑えられます。

Linear Bottleneck(線形ボトルネック)。チャネルを細く絞った最後の出力に、ReLU(マイナスを 0 に潰す活性化関数)を「かけない」ようにしました。ReLU は情報を一部捨てる性質があり、すでに細く圧縮された低次元空間でそれを使うと貴重な情報が壊れて精度が落ちる、という理論的観察(低次元の多様体に ReLU をかけると情報が崩れやすい)に基づきます。論文はトイ実験で、低次元での ReLU が入力の構造を壊す様子を可視化しています。

### V3 の核心: 探索 + SE + 新活性化
V3 は人間の手設計をやめ、NAS(Neural Architecture Search、構造の自動探索)で層の構成を機械的に最適化しました。具体的には MnasNet 系の探索(実機レイテンシを報酬に含める)で大枠(ブロックの種類と数)を決め、NetAdapt(Yang et al., ECCV 2018)でレイヤー単位にチャネル数を微調整しています。加えて、SE モジュール(Squeeze-and-Excitation、チャネルごとの重要度を動的に調整。詳細は [Squeeze-and-Excitation (SENet)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1085_squeeze-and-excitation))を組み込み、h-swish という軽い活性化関数を採用しました。h-swish は、性能の良い swish(x·sigmoid(x))の重い sigmoid 計算を、ReLU6 を使った区分線形な近似 x·ReLU6(x+3)/6 に置き換えてモバイルでも速くしたものです。さらに、計算が重い最終ステージの構造を手で簡素化(高コストな層を削減)し、Large と Small の 2 サイズを用意しました。

代表的な計算量と精度の目安は次のとおりです(ImageNet、入力 224)。

```
MobileNetV1 (1.0, 224)  ≈ 569 M MACs, ImageNet top-1 約 70.6%
MobileNetV2 (1.0, 224)  ≈ 300 M MACs, top-1 約 72.0%
MobileNetV3-Large       ≈ 219 M MACs, top-1 約 75.2%
MobileNetV3-Small       ≈ 56 M MACs,  top-1 約 67.5%
(参考) ResNet-50        ≈ 4100 M MACs, top-1 約 76%台
```

世代を追うごとに、より少ない計算で ResNet-50 に迫る精度を出せるようになっているのが分かります。V3-Large は V1 の半分以下の計算量で約 4.6 ポイント高い精度に達しています。

## 手法の系譜と主要論文
MobileNets(Howard et al., 2017, Google, arXiv:1704.04861)が出発点です。深さ方向分離畳み込みをバックボーン全体に徹底適用し、幅倍率・解像度倍率というシンプルなつまみで精度と速度のトレードオフを連続的に選べることを示しました。動機は「モバイル/組込みでの実用化」。効果は計算量 8〜9 分の 1 です。なお depthwise separable のアイデア自体は Sifre のテーゼや Xception(Chollet, 2017)が先行していますが、MobileNet はそれをモバイル設計として体系化しました。

MobileNetV2(Sandler et al., CVPR 2018, Google, arXiv:1801.04381)は逆残差と線形ボトルネックを提案。「細い層でつなぎ、太い層で処理する」ことでメモリ効率と精度を両立し、ReLU を低次元で外す根拠を理論的観察で説明しました。効果は同精度で V1 より計算量・パラメータが減ったことです。この逆残差ブロック(MBConv)は、後の EfficientNet 系の基本部品になります。

Searching for MobileNetV3(Howard et al., ICCV 2019, Google, arXiv:1905.02244)は、NAS(MnasNet 系探索 + NetAdapt)で構造を自動最適化し、SE と h-swish を組み合わせました。基盤となった MnasNet(Tan et al., CVPR 2019, arXiv:1807.11626)は、探索の報酬に実機レイテンシ(実際のスマホ Pixel での処理時間)を直接組み込んだ点で重要で、「FLOPs ではなく実速度を最適化する」流れの代表です。SE は Hu らの Squeeze-and-Excitation Networks(CVPR 2018、arXiv:1709.01507)に由来します。

この系譜は、その後 EfficientNet(MBConv = 逆残差 + SE を基本部品に複合スケーリング)や、後発の軽量モデル([ShuffleNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0406_shufflenet)、GhostNet など)に強く影響しました。MobileNet の MBConv ブロックは、現在も多くの軽量・効率モデルの標準部品になっています。

## 論文の実験結果(定量データ)
評価は ImageNet(1000 クラス、約 128 万枚)の top-1 accuracy と、計算量(MACs)、実機レイテンシ(スマホ上の処理時間)で行われました。

V1 は、α と ρ を変えることで精度と計算量を連続的に動かせることを実証しました。例えば α=1.0 で top-1 約 70.6%、α=0.5 へ落とすと計算量が約 1/4 になり精度は約 63.7% へ下がる、というトレードオフ曲線が示されています。同じ計算予算なら、層を薄くするより幅を細くした方が精度が高い、という設計上の知見も報告されました。

V2 は同じ計算量帯で V1 を上回りました。300M MACs で top-1 約 72.0% は、V1 の同等計算量より明確に高い値です。物体検出(COCO データセット、物体の位置と種類を当てるタスク。指標は mAP = mean Average Precision、検出の正確さの総合点)でも、V2 を SSDLite の特徴抽出器に使うと、VGG ベースの SSD より約 1/10 のパラメータ・計算量で同等の mAP を達成したと報告されました。領域分割(セマンティックセグメンテーション、画素ごとにクラスを当てるタスク)でも DeepLabv3 のバックボーンとして軽量化に貢献しています。

V3 のアブレーションでは、h-swish と SE の寄与が定量化されています。h-swish を導入すると、ReLU 比でわずかな計算増で精度が上がり、特に深い層(解像度が小さく適用コストが低い箇所)で効果が大きいこと、SE を加えると計算をほとんど増やさず精度が向上することが示されました。NAS で得た骨格に、SE と h-swish を載せ、最終ステージを手で簡素化することで、V3-Large は V2 比でレイテンシを約 20% 下げつつ top-1 を約 3 ポイント引き上げました。V3-Small は約 56M MACs という極めて軽い領域で top-1 約 67.5% を達成し、超低リソース帯の選択肢を広げました。さらに、セグメンテーション向けに Lite R-ASPP という軽量デコーダも提案し、Cityscapes(都市の走行シーンの画素分類)で速度と精度のバランスを改善しています。

## メリット・トレードオフ・限界
メリットは、計算量・パラメータが ResNet 系の 1/10 前後でエッジ/組込みでもリアルタイム推論しやすいこと、幅倍率・解像度倍率・V3 のサイズ選択で速度と精度を細かく調整できること、多くの推論フレームワーク(TensorRT、TFLite、CoreML、ONNX Runtime 等)で最適化済みで車載 GPU やモバイル NPU でも回しやすいことです。逆残差ブロックは検出・分割のバックボーンに差し替えやすく、転移学習の起点としても定番です。

トレードオフと限界もあります。第1に、同じ計算量でも絶対精度の上限は重いモデルに及ばず、難しいシーン(小さい物体、悪天候、まれな物体)を取りこぼしやすいことです。第2に、depthwise 畳み込みは理論 FLOPs が小さくても、メモリアクセス比率が高くハードウェアによっては期待ほど速くなりません([ShuffleNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0406_shufflenet) V2 が指摘した問題)。計算密度が低く GPU の演算ユニットを使い切れないためです。第3に、V3 は NAS で得た構造のため設計意図が人間に追いにくく、改造や移植がしづらい面があります。h-swish や SE は一部のハードウェア(特定の NPU やコンパイラ)で効率的にサポートされず、期待した高速化が出ないこともあります。第4に、ImageNet 系の自然画像で最適化されており、自動運転の俯瞰(鳥瞰)特徴のような分布の異なるドメインでそのまま最適とは限りません。

## 発展トピック・研究の最前線
MobileNet の逆残差ブロックは、その後の効率モデル設計の共通言語になりました。EfficientNet / [EfficientNetV2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0382_efficientnet-v2) はこれを土台に複合スケーリングや Fused-MBConv を加え、GhostNet は「特徴マップの冗長性を安価な線形変換で生成する」ことでさらに計算を削りました。

近年はモバイル向けに CNN と Transformer を融合する流れが強まっています。MobileViT(Mehta & Rastegari, ICLR 2022)は、MobileNet の畳み込みブロックと軽量な自己注意を組み合わせ、軽量ながら大域的な文脈を扱えるようにしました。EfficientFormer や MobileOne(Apple, CVPR 2023)は「実機レイテンシを直接最適化する」設計をさらに推し進め、MobileOne は学習時の多分岐構造を推論時に単純な一本道へ畳み込む再パラメータ化(re-parameterization)で iPhone 上 1ms 級の推論を実現しています。共通する教訓は、[ShuffleNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0406_shufflenet) V2 が示した「FLOPs ではなく実機速度で設計せよ」という指針が、エッジ向けモデルの研究全体を貫いていることです。量子化(8 ビット整数化)や知識蒸留(大きいモデルの知識を小さいモデルへ移す。[知識蒸留損失](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0633_knowledge-distillation-loss))と組み合わせて、さらに軽くする研究も活発で、MobileNet は量子化フレンドリーな構造の代表例としても扱われます。
