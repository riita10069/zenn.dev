# ベクトル地図のポリライン符号化

## ひとことで言うと
高精度地図(HD 地図)に含まれる車線・境界線・横断歩道などを、ニューラルネットワークが扱いやすい「1本の地図要素 = 等間隔に並んだ固定個数の点の列(ポリライン)」という決まった形に整える方法です。1要素あたり点を20個、地図1枚あたりの要素数を固定する、という形式が、近年のベクトル地図研究で事実上の標準になっています。地図を画素の絵(ラスタ)に潰さず、幾何構造を保ったまま機械学習に乗せるための「入力規格」だと考えると理解しやすいです。

## 直感的な理解
ニューラルネットに何かを入力するとき、最大のハードルは「形 (shape) が毎回同じであること」をネットが要求する点です。画像なら全部を同じ解像度にリサイズすれば済みますが、地図は厄介です。ある交差点には車線が3本、別の場所には7本ある。ある車線は短くて点が10個、別の車線は長くて点が200個で記録されている。要素数も点数もバラバラなのです。

バラバラのものは、そのままバッチ(複数サンプルをまとめて並列処理する単位)にできません。ノートの行数がページごとに違うと、きれいに束ねて一括で機械にかけられないのと同じです。そこで「地図1枚を、必ず固定数の要素 × 固定数の点という決まった箱に詰め直す」標準化を行います。これがポリライン符号化です。形をそろえることで GPU で大量サンプルを並列処理でき、学習が安定し、地図予測を「決まった数の枠を埋める問題」として綺麗に定式化できるようになります。

身近な比喩を1つ。住所をデータベースに入れるとき、人によって書き方がバラバラ(「東京都〜」「Tokyo〜」郵便番号あり/なし)だと検索も集計もできません。だから「郵便番号7桁・都道府県・市区町村・番地」という固定フィールドに正規化します。ポリライン符号化は、地図の幾何に対する同じ意味での正規化です。

## 基礎: 前提となる概念
ベクトル地図は、車線などを点と線の幾何データで持つ地図です([ラスタ地図とベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0433_rasterized-vs-vectorized-maps) 参照)。ラスタ地図(画素の絵)と対になる概念で、「車線中心線は座標 (x1,y1),(x2,y2),... を結んだ折れ線」という形で構造を明示的に持ちます。

ポリライン (polyline) は複数の点をつないだ折れ線です。1本の車線中心線は実世界では連続曲線ですが、コンピュータでは点の列で近似します。点を密に取れば曲線に忠実、疎にすればデータが軽くなりますが、点数が要素ごとに違うと扱いにくくなります。

リサンプリング (resampling) は、点列を別の点数・別の間隔で取り直す操作です。等間隔リサンプリング(equidistant resampling)は、ポリラインの全長 L を測り、L/(N-1) 間隔で N 個の点を打ち直すことを指します。元の点が不均一に並んでいても、出力は等間隔の N 点にそろいます。

集合予測 (set prediction) は、「決まった数のスロットを用意し、それぞれに1つの要素(または空)を割り当てる」形でものを予測する枠組みです。DETR (Carion et al., ECCV 2020) が物体検出で導入した考え方で、出力の順序に依存せず、ハンガリアン法(二部グラフの最小コスト完全マッチングを多項式時間で解くアルゴリズム)で予測と正解を1対1対応させます。ベクトル地図の「固定インスタンス数」もこの枠組みに乗ります。

順列同変・点順序の曖昧さ(permutation ambiguity)。同じ車線でも、点列を始点から終点へ並べるか逆順に並べるかで2通りの表現があり得ます。形は同じなのに数値列としては別物に見えるため、学習時にこの曖昧さを吸収する工夫(両方向の並びを許すマッチング)が要ります。横断歩道のような多角形だと、どの頂点から始めるか・時計回りか反時計回りかでさらに多くの等価表現があり、扱いがより難しくなります。

## 仕組みを詳しく
標準形式は次の手順で作ります。具体的な数値を入れて説明します。

ステップ1: 地図要素をクラスごとに集める。車線分割線 (divider)、道路境界 (boundary)、横断歩道 (pedestrian crossing) など、扱うクラスを決めます。一般的なベンチマークではこの3クラスがよく使われます。

ステップ2: 各要素を等間隔に N 点でリサンプリングする。N=20 がよく使われます。短い要素も長い要素も必ず20点で表します。例えば元が8点で記録された横断歩道は20点に補間して引き伸ばし、300点の長い車線は20点に間引きます。結果、すべての要素が `(20, 2)` の形になります(2 は x,y 座標)。等間隔にする理由は、点の密度を要素間でそろえ、ネットワークが「点間隔」というアーティファクトに依存せず形そのものを見られるようにするためです。

ステップ3: 要素数を固定数 M でそろえる。1枚の地図で扱える要素数を上限 M(例えば 50 や 100)に固定します。要素が M より少なければダミー(パディング)で埋め、「有効/無効」を示すマスクを別途持ちます。多すぎれば自車に近いものを優先して M 個に絞ります。

結果として、地図1枚は次の形のテンソルになります(B はバッチサイズ)。

```
points : (B, M, N, 2)   例 (B, 100, 20, 2)   各要素の点列(x,y、メートル単位の自車中心座標)
classes: (B, M)         各要素のクラスID
mask   : (B, M)         有効要素か否か(パディング判定)
```

座標系は通常、自車を原点とする BEV 平面のメートル座標(例えば前方 ±30m、左右 ±15m の範囲)に正規化します。範囲外の点はクリップするか要素ごと除外します。

ステップ4: エンコーダに通す。そろった形を専用エンコーダに入れます。VectorNet 系ならポリライン内をサブグラフ、ポリライン間をグローバルグラフで集約します。MapTR 系なら、各ポリライン要素を「インスタンスクエリ × ポイントクエリ」の階層的クエリとして Transformer デコーダで扱い、自己注意(self-attention)で点間・要素間の関係を学び、画像由来の BEV 特徴へ交差注意(cross-attention)して位置を refine します。最終的に要素ごとの特徴ベクトル、あるいは BEV 特徴 `(B, embed_dim, h, w)` を得ます。

before / after は次の通りです。

```
before: 車線3本(各10〜200点)、境界2本、横断歩道1個 … 要素数も点数もバラバラの生HD地図
after : (B, 100, 20, 2) という常に同じ形のテンソル + クラス + マスク。バッチ化でき、そのまま流せる
```

なぜ「20点・固定インスタンス数」が標準になったのか。理由は3つあります。(1) 形が固定でバッチ処理・GPU 並列が効く。(2) 20点もあれば実用上の車線形状を十分表現できる(直線や緩いカーブなら数点で足り、急カーブでも20点あれば滑らかに近似できる)。(3) 固定インスタンス数にすると地図予測を「決まった数のスロットを埋める」集合予測(DETR 的)として定式化でき、ハンガリアン・マッチングで学習が安定する。

損失の組み立て方も見ておきます。MapTR 系の代表的な損失は、(a) 分類損失(各クエリがどのクラスか、空スロットも含む)、(b) 点回帰損失(予測点列と正解点列のマンハッタン距離やチャンファー距離)、(c) 方向損失(隣接点を結ぶ辺のベクトル方向をそろえる)の和です。マッチングの段階で、正解1要素に対し「正順・逆順」の両方の点並びでコストを計算し、安いほうを採用することで点順序の曖昧さを吸収します。

## 手法の系譜と主要論文
ポリライン符号化の系譜をたどります。

VectorNet (Gao et al., CVPR 2020, Waymo) が元祖です。地図と他車・自車の軌跡をポリライン集合として表し、ポリライン内をサブグラフ(点とその近傍を集約)、ポリライン間をグローバルグラフ(要素どうしの相互作用を attention で集約)で処理しました。「ラスタ化して CNN に通すより、もともと構造を持つ地図はベクトルのまま扱う方が情報を失わず効率的」という主張で、軌跡予測タスクで計算効率と精度を両立させました。ほぼ同時期の LaneGCN (Liang et al., ECCV 2020, Uber) も、車線をノードとするグラフ畳み込みで地図構造を符号化し、ベクトル系の有効性を裏付けました。

MapTR (Liao et al., ICLR 2023, Horizon Robotics) と MapTRv2 (IJCV 2024) が、地図を「固定数のポリライン要素」として end-to-end に検出する枠組みを確立しました。各要素を等間隔点で表し、点順序の曖昧さを扱う順列同変なポイント集合マッチングを導入したのが新規点です。同じ車線を始点・終点どちらから並べても等価に扱えるよう、複数の点順序を許して最良の対応を選びます。MapTRv2 では、デカップルド自己注意(点レベルと要素レベルの注意を分離して計算量を削減)、補助的な深度監督や PV(透視図)セグメンテーション監督の追加で精度と速度をさらに引き上げました。「1ポリライン20点・固定インスタンス数」という標準形はこの系譜から来ています。

VAD (Jiang et al., ICCV 2023, Horizon Robotics) が、MapTR 流のベクトル地図表現を地図予測だけでなく軌跡計画まで一体で使いました。ベクトル化した地図要素を計画の制約・参照(車線を踏み越えない、境界に近づきすぎない等のインスタンスレベル制約)として用い、地図とプランニングの相乗効果と計算効率を両立させました。

集合予測の枠組み自体は DETR (Carion et al., ECCV 2020, Facebook AI) に由来します。物体検出を「固定数のクエリへの集合割り当て」として解いたこの論文が、地図要素を固定インスタンスのスロットに割り当てる発想の土台です。

時間方向への発展として StreamMapNet (Yuan et al., WACV 2024) があり、過去フレームのベクトル地図クエリを次フレームに引き継ぐ(ストリーミングする)ことで、一瞬の遮蔽や検出ノイズに強い時間的に安定した地図を作る方向が示されました。固定ポリライン表現は、こうした時間統合の「持ち越す単位」としても都合がよいことが分かります。

## 論文の実験結果(定量データ)
ベクトル地図構築の精度は mAP (mean Average Precision, 平均適合率) で測られます。推定したポリラインが正解とどれだけ一致するか(点列間のチャンファー距離が閾値以下なら正解とみなす)を、車線分割線・境界・横断歩道などクラス平均で表します。値が大きいほど良い指標です。チャンファー距離(Chamfer distance)とは、2つの点集合について「各点から相手集合の最近傍点までの距離」を双方向に平均したもので、点列どうしの形の近さを測ります。

MapTR の報告では、nuScenes のオンライン地図構築で、ラスタ化してセグメンテーションで地図を作る旧来手法を上回る mAP を、リアルタイムに近い速度(報告により十数〜数十 FPS)で達成しました。報告される mAP はおよそ 50 台で、計算量の重いセグメンテーション系を速度・精度の両面で上回りました。MapTRv2 はさらに改良され、報告によれば nuScenes でおよそ 60 台後半の mAP に到達し、推論速度も実用水準を保ちました。10 FPS 前後で動く点は、毎フレーム地図を作り直す「マップレス運転」を現実的にする上で重要です。

アブレーションが示す要点を、初心者向けに噛み砕きます。点数 N を 20 から減らす(例えば 10 や 5 に)と、急カーブの車線が直線的に潰れて mAP が落ちます。これは「20点は急カーブを表すぎりぎりの解像度」という設計上の根拠を裏付けます。逆に N を増やしても、ある点から先は mAP がほぼ頭打ちになり、計算量だけが増えます。インスタンス数 M を絞りすぎると、要素の多い交差点で取りこぼし(切り捨て)が起きて精度が下がります。最も劇的なのは点順序マッチングを外したときで、同じ車線が逆順で出てくるたびに損失が暴れて学習が不安定になり、mAP が明確に低下することが報告されています。これは「順列同変マッチングがベクトル地図学習の安定性に効いている」ことの定量的な証拠です。

計画への波及効果としては、VAD がこのベクトル表現を使い、nuScenes 計画タスクで平均 L2 誤差およそ 0.7m 前後・低い衝突率を、報告によりベースラインの数倍の推論速度で達成したと報告されています。L2 誤差は予測軌跡と人間の走行のずれ(メートル)で、小さいほど良い指標です。地図ブランチを外すと計画の L2・衝突率がともに悪化することも報告され、ベクトル地図が計画に効く入力であることが裏付けられます。

## メリット・トレードオフ・限界
メリット。形が固定されるためバッチ処理・GPU 並列が効き、学習が安定します。ラスタ化せず幾何構造を保つので、車線の正確な形を失いません。集合予測として定式化でき、地図のオンライン生成と計画への利用がしやすくなります。固定単位なので時間方向の持ち越し(ストリーミング)やトラッキングとも相性が良いです。

トレードオフと限界。要素数 M・点数 N の上限で表現力が頭打ちになり、複雑な交差点で要素が溢れると切り捨てが起きます。等間隔リサンプリングで、もとの点の非一様な配置情報(カーブの入口だけ点を密に打ちたい、といった意図)は失われます。点列はあくまで幾何であり、車線どうしの接続関係(トポロジー)を点座標だけからは表せません。パディングとマスクの管理、点順序の曖昧さ処理など実装も地味に煩雑です。そもそもベクトル HD 地図データ(Lanelet2/OpenDRIVE など)か、それを学習する正解ラベルが手元にないと使えず、入手・整備のコストがかかります。

未解決の課題としては、固定 N・M を可変にしつつ並列効率を保つ表現(可変長を許す疎な Transformer)、点列のトポロジー(車線のつながり・分岐)を点座標と別に明示的に符号化する方法、地域やフォーマットをまたいでも汎化する正規化、長距離(数百メートル先まで)の地図を破綻なく表す表現、が挙げられます。

## 発展トピック・研究の最前線
最近は、ポリラインを離散的な点列でなく連続曲線(ベジエ曲線やクロソイドのパラメータ)で表し、点数固定の制約から解放する研究が進んでいます。車線中心線のトポロジー(どの車線がどの車線につながるか)をグラフとして同時に予測する研究(TopoNet, TopoMLP など)も活発で、これにより「点の形」だけでなく「走行可能な経路網」まで出力できます。さらに、ベクトル地図要素をトークン化し、軌跡予測・計画・対話を1つの大規模 Transformer に統合する流れ(地図も他の要素もトークンとして同列に扱う world model 的アプローチ)も進み、ポリライン符号化は「地図モダリティのトークナイザ」として位置づけ直されつつあります。時間統合(StreamMapNet 系)と組み合わせ、瞬間ではなく時系列として安定した地図を出す方向も主流化しています。
