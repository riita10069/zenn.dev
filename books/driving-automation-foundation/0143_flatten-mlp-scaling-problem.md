# BEVフラット化MLPのスケール問題

## ひとことで言うと
BEV(鳥瞰図、真上から見下ろした地図)を「1本の長いベクトルに引き伸ばして(フラット化して)MLPに入れる」という素朴なやり方は、高解像度のBEVだと入力が数千万次元になり、最初の全結合層だけでパラメータ数が数十億に爆発して破綻します。だから高解像度BEVから軌跡を出すには、フラット化MLP以外の仕組み(クエリ + クロスアテンション)が必要だ、という構造的な問題の解説です。

## 直感的な理解

ニューラルネットワークで「地図を見て軌跡を出す」とき、一番素朴に思いつくのは「地図全部を1本のベクトルにして、それを全結合のネットワーク(MLP)に通して軌跡を出す」やり方です。地図が小さければこれで動きます。問題は、地図を細かくした瞬間に起きます。

直感的な例えで言えば、これは「全国の地図を1ピクセルも漏らさず暗記してから、目的地への経路を一発で答えよ」と要求するようなものです。地図が粗いうちは暗記できますが、地図を高精細にすると暗記すべき情報量が爆発し、暗記係(=ネットワークの最初の層)が破綻します。賢いやり方は、地図全体を丸暗記せず「今いる場所の周りだけ見て、必要に応じて地図の別の部分を参照しに行く」ことです。これが後述するクロスアテンション方式です。

この問題は単なる実装上の不便ではなく、「BEVを高解像度にして安全性を上げる」という方向と「フラット化MLPで計画する」という設計が、原理的に両立しないことを意味します。だから多くの end-to-end 自動運転研究は、最初からフラット化MLPを避けています。これは「全結合層は入出力次元の積だけ重みを持つ」という、ニューラルネットワークの最も基礎的な性質が、空間データの規模と衝突して生まれる必然的な問題です。

## 基礎: 前提となる概念

「MLP(エムエルピー、Multi-Layer Perceptron、多層パーセプトロン)」は最も基本的なニューラルネットワークで、入力ベクトルを全結合層(全部の入力を全部の出力につなぐ層)で順に変換します。全結合層の重み(パラメータ)の数は「入力次元 × 出力次元」です。ここが問題の核心になります。入力次元が10倍になれば、重みも10倍になります。

「フラット化(flatten、平坦化)」は、縦横に並んだ多次元データを1列に並べ直して1本のベクトルにすることです。例えば[256, 3, 3]の特徴マップ(3×3の各位置に256次元)をフラット化すると、256 × 3 × 3 = 2304次元のベクトルになります。フラット化すると、もともと隣り合っていたマスの「空間的な近さ」の情報が失われ、ネットワークは「どの次元がどのマスか」をゼロから学習し直すことになります。

「パラメータ数と過学習」の関係も基礎として重要です。パラメータが多いほどモデルは複雑な関数を表せますが、訓練データを丸暗記してしまい、見たことのない状況で破綻する「過学習(overfitting)」が起きやすくなります。経験則として、パラメータ数が訓練サンプル数を桁違いに上回ると汎化(未知データでの性能)が困難になります。

「軌跡計画(trajectory planning)」は周囲の状況を入力に未来の軌跡を出力する処理です。素朴に作るなら、BEV地図をフラット化して1本のベクトルにし、それをMLPに食わせて軌跡(例: 64点 × 2値 = 128次元)を出す構成が思いつきます。低解像度なら問題ありません。BEVが7×7×256なら、フラット化しても 7 × 7 × 256 = 12544次元で、MLPの入力として十分扱える大きさです。

## 仕組みを詳しく

問題は、安全のためBEVを高解像度にした瞬間に起きます。前方を遠くまで(例: 120m)見つつ、隣り合う車を別マスに分離できる細かさにすると、BEVは数百×数百マスが必要になります([BEV知覚レンジ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0263_bev-perception-range))。高解像度BEVをフラット化するとどうなるか、具体的に数えます。

```
BEV: bev_h × bev_w × embed_dim = 450 × 300 × 256
フラット化後の次元数 = 450 × 300 × 256 = 34,560,000 ≈ 3450万次元
```

MLPの入力が3450万次元になります。なぜこれが破綻か。MLPの最初の全結合層は「入力次元 × 出力次元」個の重みを持ちます。出力を控えめに256次元としても、

```
最初の層のパラメータ数 = 入力次元 × 出力次元 = 34,560,000 × 256 ≈ 88億パラメータ
```

88億パラメータです。比較のため、標準的な画像認識ネットワーク ResNet-50(He et al., CVPR 2016, arXiv:1512.03385)は全体で約2500万パラメータ、中規模言語モデル GPT-2 で約15億パラメータです。つまり「軌跡を出すための最初の1層」だけで、それらをはるかに超えるパラメータを抱えます。これは1層分の話で、これに匹敵する規模の重み行列を作っただけで設計は破綻します。

実害を具体的に:

- メモリ: 88億パラメータを fp32(1個4バイト)で持つだけで約35GB。重みだけで大容量GPUが埋まる。学習時は勾配(同じく35GB)とオプティマイザ状態(Adam なら運動量と分散で重みの2倍)も要るので、実際にはこの数倍、合計100GB超になり1枚のGPUには載らない。
- 計算量: 1回の推論で88億回規模の積和。車載リアルタイム(10Hz = 0.1秒に1回)など到底無理。
- 過学習: パラメータが多すぎて訓練データを丸暗記し、未知の道で破綻する。88億パラメータに対し訓練シーンは多くて数十万であり、汎化は望めない。
- 解像度を変えられない: BEVを200×200に変えると入力次元が変わり、MLPを丸ごと作り直す羽目になる。実験の柔軟性がゼロ。

before / after:

- before(フラット化MLP): 入力次元 = BEVマス数 × 256 に比例して増える。7×7なら平気でも450×300では3450万次元で破綻。BEV解像度を上げられない構造的な壁。
- after(クロスアテンション方式): BEVを直接フラット化せず、固定サイズ(256次元)のクエリ(問い合わせ票)がBEVの必要な場所だけを参照する。入力サイズがBEV解像度に依存しなくなる。

解決の鍵は「BEV全体を一度に1本のベクトルに潰さない」ことです。固定サイズの小さなクエリを用意し、それがクロスアテンション([Deformable Cross-Attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0259_deformable-cross-attention))でBEVの必要な部分だけを参照します。クエリのサイズはBEV解像度と無関係なので、450×300でも200×200でもクエリは256次元のまま。これが[TrajectoryPlanner (ego query + GRU)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0139_trajectory-planner-ego-query-gru)のエゴクエリ方式です。

補足として、なぜ全結合(MLP)だけがこの問題を起こすのか。CNN(畳み込み)は重みを空間全体で共有する(同じ小さなフィルタを全マスに適用する)ので、入力が大きくてもパラメータは増えません。プーリングや適応プーリングで固定サイズに潰す手もあります。しかし畳み込みやグローバルプーリングは「BEVのどこを重視するかを状況に応じて変える」ことができず(重みが固定)、運転状況依存の注意配分には不向きです。だから単純なCNN+プーリングではなく、状況依存に「どこを見るか」を選べるクロスアテンションが好まれます。

## 手法の系譜と主要論文

フラット化MLPを避ける設計は、この分野でほぼ標準化しています。

UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)は「エゴクエリ + GRU の自己回帰デコーダ」を採用しました。各未来時刻でGRUの隠れ状態が変形アテンションでBEVを参照し、MLPが1点ずつウェイポイントを出します。動機はまさに、フラット化MLPだとBEV解像度に縛られ過学習するため。効果は、クエリが固定サイズなのでBEV解像度に依存せず、運転状況に応じてBEVのどこを見るか動的に変えられること。トレードオフは自己回帰ゆえの逐次処理です([TrajectoryPlanner (ego query + GRU)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0139_trajectory-planner-ego-query-gru))。

VAD(Jiang et al., ICCV 2023, arXiv:2303.12077)は、BEVをいったん「ベクトル化したエージェント・地図クエリ」というコンパクトな表現に圧縮し、エゴクエリがそれらに注意を向けます。BEVを直接フラット化せず、意味のある少数の要素(各エージェントの軌跡、各車線のポリライン)に縮めてから扱う点が工夫で、軽量・高速。トレードオフはベクトル化段での情報損失です。

BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)は、下流の計画ヘッドでもBEVをフラット化せず変形アテンションで参照する設計を取りました。非対称グリッド(前方を長く、後方を短く取るなど)でもこの方式なら破綻しません。

Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159)は、そもそも「全位置を見るのではなく数点だけサンプリングする」変形アテンションを提案し、計算量を入力サイズの2乗から線形に落としました。フラット化MLPのスケール問題を回避する理論的土台です([Deformable Cross-Attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0259_deformable-cross-attention))。

## 論文の実験結果(定量データ)

この問題は「フラット化MLPで何点まで上げられたか」という直接の数値で語られるより、「フラット化を避けた手法がどれだけ高解像度BEVと高い計画精度を両立できたか」で語られます。

UniAD は nuScenes の計画ベンチマークで、3秒先の平均L2誤差(予測軌跡と実走行軌跡のユークリッド距離、メートル、小さいほど良い)をモジュール分離型ベースラインより低くし(およそ1.6m級)、衝突率(%、小さいほど安全)も従来比でおよそ半減させました。これを高解像度BEV(典型 200×200)を保ったまま達成できた点が、フラット化MLPでは原理的に不可能だった成果です。フラット化MLPで同じ200×200のBEVを扱おうとすると、200×200×256 = 1024万次元 → 最初の層だけで約26億パラメータになり、学習も推論も成立しません。

VAD は、ベクトル化により計算量をさらに削り、UniAD と同等以上の計画精度を保ちつつ推論をおよそ数倍高速化したと報告されています。これは「BEVを潰さずに済ませる」だけでなく「そもそもBEV全体を持たずに済ませる」方向の効率化です。

参考として、フラット化を使う場合でも、グローバル平均プーリングでBEVを[256]の固定ベクトルに潰せば次元爆発は防げます。しかしこの「空間情報を完全に捨てる」圧縮は、計画に必要な「どこに何があるか」を消してしまうため、計画精度が大きく落ちることがアブレーション的に知られています。次元爆発(フラット化)と情報喪失(プーリング)の板挟みを同時に解くのがクロスアテンションだ、という構図です。クエリは固定サイズで次元爆発しないが、サンプリングで空間の必要箇所は保持する、という両取りを実現します。

## メリット・トレードオフ・限界

フラット化MLPのメリット(低解像度限定)
- 実装が単純で、低解像度BEV(7×7など)では問題なく動く。最初のプロトタイプや概念実証には十分。

フラット化MLPの致命的トレードオフ・限界
- 入力次元がBEVマス数に比例して増え、高解像度(450×300=3450万次元)で破綻する。
- 最初の層だけで数十億パラメータになり、メモリ・計算量・過学習が一斉に悪化。
- BEV解像度を変えるたびにMLPを作り直す必要があり、実験の柔軟性が無い。
- 「どこを重視するか」を状況依存に変えられない(重みが固定)。フラット化で空間構造も失う。

解決策(クロスアテンション方式)のトレードオフ
- 実装が複雑になる(クエリ・アテンション・自己回帰など)。
- 自己回帰デコーダ(GRU)は逐次処理で推論が並列化しにくい(拡散・フローマッチング・ベジェ等で回避する流れもある)。
- クロスアテンションは参照点(=カメラ校正やクエリ予測)の品質に依存する。

研究上の含意として、このスケール問題は「end-to-end 自動運転では、知覚表現(BEV)の解像度と計画ヘッドの設計を分離できる構造が必須」という設計原則を示しています。クエリ + アテンションは、その分離を実現する標準解になっています。

## 発展トピック・研究の最前線

「BEVをそもそも陽に作らない」方向の研究も出ています。スパースクエリ系(Sparse4D, Lin et al., 2022, arXiv:2211.10581 / SparseDrive, Sun et al., 2024, arXiv:2405.19620)は、密なBEVグリッドを経由せず、物体やエゴのクエリが直接マルチビュー画像特徴を参照します。これはフラット化問題を根本から回避するアプローチで、密なBEVを保持するメモリそのものを節約でき、SparseDrive は計画精度を保ちつつ推論を大きく高速化したと報告しています。

逆に、占有グリッド(Occupancy)では解像度がさらに上がり(2次元のBEVから3次元のボクセルへ)、フラット化の禁忌はより強くなります。例えば 200×200×16 のボクセルを256次元で持てば 1.6億次元で、フラット化は論外です。ここでもボクセルクエリ + 変形アテンションが標準です。

また、VLM(Vision-Language Model)を計画に持ち込む流れでは、BEVや画像特徴を「トークン列」に変換して言語モデルに渡しますが、トークン数(=系列長)がアテンションのコスト(O(N²))を決めるため、ここでも「いかに少ないトークンで空間情報を表すか」というフラット化問題の変奏が現れます。トークン圧縮・選択は、この文脈での最前線の研究テーマです。要するに、本ノートの教訓「高次元の空間表現を全結合で素朴に潰すな」は、表現形式が変わっても繰り返し立ち現れる普遍的な設計原則だと言えます。
