# PolarFormer

## ひとことで言うと
カメラ画像から「鳥瞰図」(上空から見下ろした地図のような表現)を作るとき、ふつうは縦横の四角いマス目(直交格子)で地図を区切ります。PolarFormer は、その地図を四角マスではなく「中心からの距離」と「角度」で区切る極座標(きょくざひょう)で組み立てる Transformer 手法です。カメラが世界を扇状に見ているという事実に、地図の座標の形そのものを合わせることで、画像から鳥瞰図への変換を素直に学習させます。多カメラ 3D 物体検出のために提案されました。

## 直感的な理解
人間が運転するとき、頭の中で「あの交差点は前方50mくらい、あの歩行者は右斜め前」というような、上から見た地図を無意識に作っています。自動運転でも同じことをしたい。カメラ画像はあくまで「自分の目線から見た平らな写真」なので、これを「上から見た地図」に変換する処理が必要になります。これを BEV(Bird's-Eye-View、バーズアイビュー、鳥瞰図)変換と呼びます。

ここで「地図をどんなマス目で区切るか」という設計の自由度があります。素朴には、前後120m × 左右120m の正方形を 0.5m 刻みの四角いマスに切る、という直交格子が思い浮かびます。地図帳のように直感的です。しかしカメラの見え方を考えると、これは少し不自然です。

カメラのレンズは1点から光を集めます。だから視野は扇状に外へ広がります。近くは細かく、遠くは粗く、左右には角度で均等に広がる。ところが四角い格子は、カメラの中心からの距離や角度とは無関係に、世界を一律の正方形で切ってしまう。このミスマッチが、特に画像の端や遠方で「同じ角度に並んだ物体が地図上ではバラバラのマスに散る」という不便を生みます。PolarFormer の発想は単純で、「カメラが扇状に見るなら、地図も扇状に区切ればいい」というものです。

## 基礎: 前提となる概念

### 直交座標と極座標
平面上の点を表す方法には2通りあります。直交座標(Cartesian coordinates、デカルト座標)は、点を「横にどれだけ(x)、縦にどれだけ(y)」で表します。極座標(polar coordinates)は、点を「中心からどれだけ離れているか(距離 r)、どの向きか(角度 θ)」で表します。

たとえば「自車の真正面 10m」の点は、直交座標なら (x, y) = (10, 0)、極座標なら (r, θ) = (10m, 0度) です。「右斜め前」の同じ距離の点は、直交なら (7, 7) のように両方の値が動きますが、極座標なら距離はそのまま (10m, 45度) と角度だけが変わります。極座標は「中心から放射状に観測する」状況を自然に表せるのが特徴です。両座標系の変換は x = r·cosθ、y = r·sinθ(直交へ)、r = √(x²+y²)、θ = atan2(y, x)(極へ)で行き来でき、どちらも同じ平面を表すので情報量は等価です。違うのは「どの方向に解像度を割くか」だけです。

### ピンホールカメラと「画像横位置 ≒ 角度」
カメラはピンホール(針穴)モデルで近似されます。光は1点(レンズ中心)を通って奥のセンサーに像を結ぶ、という単純化です。このモデルの重要な性質は、画像の横方向の位置(u 座標)が、おおむね「視線の水平角度」に対応するということです。より正確には、横画素 u と焦点距離 fx・主点 cx を使って tan(方位角) = (u − cx) / fx の関係になり、画像の左端は左方向の角度、中央は正面、右端は右方向の角度に相当します(レンズ歪みを無視した近似)。つまり画像の横位置は、極座標の角度 θ と単調に対応します。この対応こそが PolarFormer が極座標を選ぶ幾何的な根拠です。直交 BEV ではこの「u ↔ θ」の関係が x, y の両方に絡んでしまい、ネットワークが暗黙に学び直さねばなりません。

### BEV クエリとクロスアテンション
近年の BEV 手法の多くは Transformer の注意機構(attention)を使います。地図のマス目それぞれを「BEV クエリ」と呼ばれる学習可能なベクトルとして用意し、各クエリが画像特徴に対して「自分に関係する場所はどこか」を問い合わせ、その特徴を集めて自分の中身を埋めます。この「片方(クエリ)がもう片方(画像特徴)に注目して情報を集める」処理がクロスアテンション(cross-attention)です。注意は「クエリ Q とキー K の内積で重みを作り、その重みでバリュー V を加重平均する」演算 softmax(QKᵀ/√d)·V で表されます。PolarFormer ではこのクエリを極座標のマス目に割り当てます。

## 仕組みを詳しく

### 極座標 BEV の構成
直交 BEV が (x, y) のマスの集まりなのに対し、極座標 BEV は (r, θ) のマスの集まりです。

- r: 自車中心からの距離。0m, 2m, 4m, ... のように同心円状に区切る。距離方向のビン数を N_r とする。
- θ: 角度。真正面を0度として、左右に扇状に等間隔で分割する。角度方向のビン数を N_θ とする。

イメージとしてはダーツの的です。中心から放射状の線と同心円で区切った、扇形の小片(セル)の集合です。各セル(全部で N_r × N_θ 個)が1つの BEV クエリになります。たとえば N_θ=360(1度刻み)、N_r=128 なら 46080 個のクエリで、各クエリが C=256 次元のベクトルだとすると BEV テンソルの shape は [N_θ, N_r, C] = [360, 128, 256] です。

```
直交格子(従来)            極座標格子(PolarFormer)
+--+--+--+--+              \   |   /
+--+--+--+--+               \  |  /
+--+--+--+--+        ・・・・・○・・・・・   ← 同心円(r 一定)
+--+--+--+--+               /  |  \
+--+--+--+--+              /   |   \    ← 放射線(θ 一定)
   ↑ ego                     ↑ ego(中心)
一律の正方形              中心から扇状に区切る
```

### カメラ視錐との整合がもたらす効果
具体例で違いを見ます。同じ角度(真正面やや右、θ=10度)に、前方10m と前方50m の2台の車が並んでいるとします。

- 極座標 BEV: 両者は「θ=10度の同じ角度列、違う r」のセルに入ります。画像上では同じ横位置(同じ角度方向)に写るので、両セルとも画像の同じ列付近から特徴を引けます。「角度が同じものは画像の同じ横位置」という構造的な手がかりが座標系に最初から組み込まれている。
- 直交 BEV: 同じ車たちが (x, y) = (49.2, 8.7) と (9.8, 1.7) という離れたマスに入ります。それぞれ別々に「画像のどこを見ればいいか」をネットワークが学ぶ必要があります。

つまり極座標は、画像の幾何的構造(横位置=角度)を BEV の座標構造に直接写し込んでいるため、変換が学びやすくなる、というのが核心です。

### Transformer パイプライン全体
PolarFormer のおおまかな流れは次の通りです。

1. 多視点カメラ画像をバックボーン(画像から特徴を抽出する CNN や Transformer、たとえば ResNet-50/101 や VoVNet-99)と特徴ピラミッド(FPN)に通し、各カメラの多解像度特徴マップを得る。
2. 極座標 BEV クエリを用意し、各クエリ(角度 θ・距離 r)に対応する画像領域へ可変形クロスアテンション(deformable cross-attention、注目点の周りの少数の標本点だけを見て計算を節約する注意)で問い合わせる。画像横位置との角度対応を使って、どのカメラのどの列を見るかを幾何的に絞る。
3. 複数の高さ(z 方向)に沿ってサンプリングし、柱状(pillar)に特徴を集約する。地面付近から空近くまでを束ねることで、単眼画像が本質的に持つ「奥行きが分からない=同じ画素が複数の3D点に対応する」曖昧さに対応する。
4. 時間方向(複数フレーム)を統合する拡張も提案され、過去フレームの BEV を自車運動で現在座標に揃えてから融合することで、動く物体の速度推定や遮蔽への頑健性を高める。
5. 最終的に、極座標で作った BEV 特徴を直交座標へ変換し直して検出ヘッドに渡す。検出ヘッドや評価指標が直交前提のものが多いためで、ここで極座標↔直交の双線形補間(bilinear interpolation)が入る。

この「極座標で集める → 直交で出す」という二段構えが PolarFormer の実装上の特徴です。論文では、極座標で予測してから直交へ戻す Polar 検出ヘッドの設計が、座標系のミスマッチを最小化するうえで重要だと論じられています。

## 手法の系譜と主要論文

PolarFormer は「BEV を Transformer で作る」流れと「極座標センサ表現」の流れが合流した地点に位置します。

- PolarNet (Zhang et al., CVPR 2020, "PolarNet: An Improved Grid Representation for Online LiDAR Point Cloud Semantic Segmentation", arXiv:2003.14032): LiDAR の点群セグメンテーションで極座標格子を導入した先行研究。「LiDAR はセンサ中心から放射状に点を取得するので、点が極座標格子に均等に分布する。直交格子だと近傍に点が集中し遠方は疎になって不均衡になる」と主張しました。極座標が「放射状に観測するセンサ」と相性がよいことを示した系譜の出発点です。
- PolarStream (Chen et al., NeurIPS 2021): 極座標を LiDAR のストリーミング検出(扇形のスキャンが届くたびに逐次処理する)へ広げ、極座標が放射状観測のオンライン処理に自然である点を補強しました。
- DETR3D (Wang et al., CoRL 2021, arXiv:2110.06922): BEV の密な格子を持たず、疎な 3D オブジェクトクエリを画像へ投影して特徴を引き、検出する手法。集合予測(set prediction)の枠組みを 3D に持ち込みました。PolarFormer は「密な極座標格子を持つ」点で対照的で、密格子は周辺文脈を取りやすい反面、計算コストが高いというトレードオフがあります。
- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270): 直交 BEV を空間クロスアテンション(spatial cross-attention)と時間自己アテンション(temporal self-attention)で構築する代表手法。PolarFormer はこの「BEV クエリが画像へ注意する」枠組みをほぼ踏襲しつつ、BEV の座標系だけを直交から極座標へ置き換えた、と理解すると関係が明確です。実験でも主要な比較対象になります。
- PolarFormer (Jiang, Zhang, Huang, Li, Zhao, Wen, Tang et al., AAAI 2023, "PolarFormer: Multi-camera 3D Object Detection with Polar Transformer", arXiv:2206.15398): 本トピックの出典。極座標 BEV を多カメラ 3D 検出に適用しました。新規性は (1) カメラ視錐に整合した極座標 BEV 表現、(2) 異なる角度・距離のスケールを扱う Polar Detection Head、(3) cross-attention に基づく多スケール画像特徴の集約と時間統合、の3点です。

## 論文の実験結果(定量データ)

評価は主に nuScenes という大規模自動運転データセットで行われました。nuScenes は6台のカメラ、LiDAR、レーダーを搭載した車で都市部を走行し、1000シーン(各20秒)を収録したものです。3D 物体検出の標準ベンチマークとして広く使われます。

主要な評価指標を噛み砕きます。

- mAP(mean Average Precision): 検出の正確さの総合点。物体を正しく見つけ、位置を正しく当てられたかを 0〜1(または%)で表します。高いほど良い。nuScenes では IoU(重なり率)ではなく、予測ボックスと正解ボックスの中心距離が 0.5/1/2/4m 以内かで正解判定し、その平均を取る方式です。
- NDS(nuScenes Detection Score): nuScenes 独自の総合スコア。mAP に加えて、位置誤差(mATE)・サイズ誤差(mASE)・向き誤差(mAOE)・速度誤差(mAVE)・属性誤差(mAAE)をまとめた指標。実用に近い総合力を測るため、nuScenes ではこちらが最重視されます。高いほど良い。

報告によると、PolarFormer は同等のバックボーン・入力解像度の条件で、BEVFormer など直交 BEV ベースの手法を NDS / mAP の両方で上回りました。ResNet-101-DCN バックボーンを使った設定で、nuScenes 検証(validation)セットにおいて NDS でおよそ 0.52〜0.53、mAP でおよそ 0.43 前後が報告されており、これは同条件の BEVFormer(NDS およそ 0.51)を数ポイント上回る水準です(具体値は入力解像度・学習スケジュールで変動するため「およそ」と添えます)。テスト(test)セットでも、VoVNet-99 等の大きなバックボーンと時間統合を用いた設定で NDS およそ 0.57 前後と、当時の上位手法に匹敵する数値を達成しました。特に向き(orientation)の推定誤差 mAOE で改善が報告されており、これは「同じ角度の物体を同じ角度列に集約できる」極座標の構造が方位推定に効いている、という主張と整合します。

アブレーション(どの要素を抜くと性能がどう変わるかを調べる実験)では、次のことが報告されています。

- 直交 BEV を極座標 BEV に置き換えると検出性能(NDS/mAP)が向上する。これは座標系の選択そのものが効いている証拠です。
- 時間統合(複数フレームの利用)を抜くと、速度推定の誤差(mAVE)が悪化する。動く物体には時間情報が効きます。
- 多スケール画像特徴(FPN の複数解像度)を1スケールに簡略化すると、遠近で大きさの異なる物体への対応力が落ちる。

これらは「極座標化」「時間」「多スケール集約」がそれぞれ独立に貢献していることを示します。なお改善幅は数ポイント規模で、劇的というより着実な向上で、座標系設計という比較的「安価」な(追加の学習コストや教師信号が要らない)変更でこれを得られる点が評価されました。

## メリット・トレードオフ・限界

メリット:
- カメラの放射状の視錐と極座標が幾何的に整合し、画像横位置と BEV 角度の対応が素直になる。学習が容易になる。
- 同じ角度の物体を同じ角度列に集約でき、方位推定や遠方検出で有利になりやすい。
- LiDAR・カメラとも「中心から放射状に観測するセンサ」であるため、極座標表現は両モダリティと親和性が高い。

トレードオフ・限界:
- 中心から遠いほど、隣り合う放射線の間隔(弧の長さ r·Δθ)が広がる。つまり遠方では同じ角度刻みでも空間解像度が粗くなり、遠方の小さい物体が1セルに潰れやすい。距離方向と角度方向の解像度バランス(N_r と N_θ の配分)を設計するパラメータが、直交格子より増える。
- 検出ヘッドや評価ツール、後段の経路計画は直交座標前提のものが圧倒的に多い。極座標↔直交の変換・補間が必ず追加で必要になり、補間誤差や実装の複雑さが生じる。
- マルチタスク(検出と同時にセグメンテーションや占有グリッド予測をするなど)では、タスクごとに好ましい座標系が異なることがあり、極座標一択が最適とは限らない。
- 中心(自車)付近では逆にセルが密になりすぎ、近傍の解像度配分が過剰になる場合がある。

研究上の未解決課題としては、極座標と直交を場面に応じて使い分ける、あるいは両者を融合するハイブリッド表現の設計、極座標のまま下流タスク(経路計画・占有予測)まで一貫処理する枠組みの確立、距離ビンを等間隔でなく対数スケールにして遠方解像度を確保する設計などが挙げられます。

## 発展トピック・研究の最前線
近年は BEV から一歩進んで、空間を 3D ボクセルで占有/非占有として表す占有グリッド予測(occupancy prediction、SurroundOcc や OccFormer など)が盛んです。ここでも放射状サンプリングや極座標的な発想が議論されます。また、検出のための密な BEV 格子を作らず、疎なクエリだけで効率化する Sparse 系手法(Sparse4D, PETR 系)との対比で、「密 vs 疎」「直交 vs 極座標」という2軸の設計空間が整理されつつあります。座標系の選択は、計算コスト・遠近の解像度配分・下流タスクとの相性という複数のトレードオフの結節点であり、単一の正解はまだありません。視錐と座標系の整合という発想は、レンジビュー(LiDAR の球面投影表現)やシリンドリカル(円柱)座標の研究とも通底しており、「センサの観測幾何に表現を合わせる」という大きな研究潮流の一翼を担っています。
