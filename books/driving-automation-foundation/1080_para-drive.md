# PARA-Drive (並列タスクE2E)

## ひとことで言うと
PARA-Drive は、自動運転の「知覚・予測・計画」という3つの仕事を、これまでのように一列に順番(逐次)でつなぐのではなく、横に並べて同時に処理する(並列に配置する)End-to-End(入力から出力までを1ネットワークで結ぶ方式)の設計です。これにより推論が速くなり、しかも本番の走行時には軌道計画に不要な補助タスク(検出・地図・予測・占有など)を丸ごと取り外してさらに高速化できます。2024 年の CVPR(コンピュータビジョンの主要国際会議)で発表されました。

## 直感的な理解
工場の組み立てラインを思い浮かべてください。直列(ライン式)だと、A 工程が終わらないと B が始められず、B が終わらないと C が始められません。最後の製品(ここでは計画)を得るには、全工程を順番に通すしかなく、合計時間が各工程の足し算になります。

一方、複数の作業を別々の作業台で同時に進める(並列)なら、互いを待たずに済みます。全体の所要時間は最も遅い作業台の時間に近づきます。PARA-Drive のアイデアはまさにこれで、知覚・地図・予測・占有・計画という作業を、共通の材料(BEV 特徴)から同時に進めます。

さらに賢い点があります。本番で本当に欲しい成果物は「計画(軌道)」だけです。知覚や予測は、学習中はネットワークを賢く育てるのに役立ちますが、走行中の最終出力としては必須ではありません。直列ラインだと途中工程を抜くと後工程が動きませんが、並列なら不要な作業台を畳んでも計画だけ取り出せます。「学習時は手厚く、本番は身軽に」を実現する設計です。これは、たくさんの先生から学んで賢くなった生徒が、試験本番では参考書を持ち込まず一人で解くようなものです。

## 基礎: 前提となる概念
BEV(Bird's-Eye-View、鳥瞰図): 複数カメラの画像を真上から見下ろした座標系へ変換した中間表現。`[H, W, C]` 形(例 `[200, 200, 256]`)で、他車との距離や車線の形を扱いやすくします。多くの近年手法でこれが全タスク共通の土台になります。代表技術は BEVFormer(Li ら、ECCV 2022)。

タスクヘッド(task head): 共通の特徴から、特定の出力(検出結果、地図、予測軌道、占有マップ、計画軌道など)を作る、ネットワークの末端部分。胴体(バックボーン+BEV エンコーダ)が共通で、頭(ヘッド)がタスクごとに分かれるイメージです。

マルチタスク学習(multi-task learning): 1つのネットワークに複数の仕事を同時に学ばせると、互いの学習信号(勾配)が補い合って共有表現が豊かになり、各タスク単独より良くなることがある、という学習法。補助タスク(auxiliary task)とは、最終目的そのものではないが共有表現を鍛えるために併走させる仕事のこと。たとえば「他車を検出する」課題を学ばせると、BEV 特徴のなかに「物体らしさ」「速度」の情報が自然に蓄えられ、それが計画にも効きます。

planning-oriented(計画指向): 「最終目的は計画なのだから、上流のすべてのタスクは計画に役立つように設計・整列させよう」という設計思想。UniAD が打ち出しました。

トポロジー(topology): タスク同士の「つなぎ方・配線の形」。どのタスクがどのタスクの出力を入力に使うか、という構造のことです。直列(逐次依存あり)か並列(依存なし)かはトポロジーの違いです。

推論レイテンシ(inference latency): 入力が来てから出力が出るまでの所要時間。自動運転は数十ミリ秒ごとに判断を更新する必要があり、レイテンシは安全に直結します。直列構造ではレイテンシが各段の和になりますが、並列では理屈上もっとも重い段に近づきます。

## 仕組みを詳しく
PARA-Drive の核心は2つ、「並列配置」と「推論時の補助タスク削減」です。

ステップ1: 共有 BEV 特徴を作る。複数カメラ画像から BEV 特徴表現を作ります。これが全タスク共通の土台 `[H, W, C]` です。

ステップ2: 各タスクを並列に接続する。ここが UniAD との最大の違いです。UniAD はおおよそ次のように直列でした。

```
画像 → 知覚(検出・追跡) → 動き予測 → 占有予測 → 計画
        (各段の出力クエリが次段の入力に流れる)
```

UniAD では計画モジュールが上流のすべての出力(追跡された他車、予測された軌道、占有マップ)を待ってから走り出すため、計画を得るには全段を順に通す必要がありました。これに対し PARA-Drive は、各タスクヘッドを共有 BEV から同時にぶら下げます。

```
              ┌→ 知覚(検出・追跡)
              ├→ オンライン地図構築
共有 BEV 特徴 ┼→ 動き予測
              ├→ 占有予測(occupancy)
              └→ 計画(planning)  ← 本番で欲しい出力
```

各タスクは互いに直列依存せず、同時に計算できます。論文の重要な貢献は、考えられる接続パターン(どのタスクをどう並べ、どこに依存を入れるか)を体系的に探索し、ほぼ並列な配置が速度と精度のバランスで最も優れることを実証した点です。「知覚→予測→計画の直列が当然」と思われていた設計を、設計空間の網羅的な実験でひっくり返したわけです。

ステップ3: 補助タスクは学習時だけ使い、推論時は省く。実用上いちばん効くアイデアです。学習中は、知覚・地図・予測・占有といった補助タスクが共有 BEV 特徴を「計画に役立つ豊かな表現」へ育てます。学習の勾配が各ヘッドから共有 BEV へ逆伝播し、「ここに車がいる」「この先がふさがる」という知識を BEV 特徴に染み込ませます。しかし本番走行で実際に欲しいのは計画(軌道)だけ。並列構造なら計画ヘッドは他タスクの出力に依存しないので、推論時に補助タスクヘッドをまるごと取り外せます。直列構造ではこれが不可能です(下流が上流の出力を必要とするため)。

結果として、学習時はリッチなマルチタスクで賢く育て、推論時は計画ヘッド(+共有バックボーン)だけの軽い構成で走る、という「いいとこ取り」が成立します。

なぜ並列でも、しかも補助タスクを外しても精度が落ちないのか。直感的には、補助タスクの役割は「最終的に良い計画を出すための表現を作ること」であって、推論時にその補助出力そのものを計画モジュールが消費する必要はない、という洞察です。学習の勾配を通じて共有 BEV に知識が染み込んでいれば、本番でヘッドを外しても表現は残ります。これは知識蒸留(大きな教師の知識を生徒に移す)の自己版とも言え、補助タスクが「教師」として共有表現を整えます。

## 手法の系譜と主要論文
UniAD(Hu ら、上海 AI Lab など、CVPR 2023 best paper、arXiv:2212.10156)。知覚→予測→計画をクエリで直列につなぎ、すべてを計画に役立つよう整列させる planning-oriented な統合 End-to-End を提案。各タスクを独立に積む素朴な多段構成や、一部のタスクしか持たない構成と比べて計画性能で大きく前進しましたが、直列ゆえの推論コストと、トポロジーが固定であるという弱点を残しました。PARA-Drive はこの「つなぎ方は本当に直列が最適か」という問いから出発します。

VAD(Jiang ら、ICCV 2023、arXiv:2303.12077)。世界をベクトル化することで表現を軽くし高速化する方向。PARA-Drive はアーキテクチャのトポロジー(並列化)で高速化する方向で、両者は直交します。表現の軽量化(VAD)と配線の最適化(PARA-Drive)は同時に適用でき、組み合わせも考えられます。両者は「UniAD は重い」という同じ問題意識から、異なる切り口で解こうとした兄弟と言えます。

PARA-Drive(Weng, Ivanovic, Wang, Pavone ら、NVIDIA・トロント大など、CVPR 2024)。タスクを並列配置し、推論時に補助タスクヘッドを削除できる End-to-End 設計を提案。動機は UniAD の直列構造による速度の限界を破ること。設計空間(直列・並列・依存の有無の組み合わせ)を系統的に調べ、並列配置が速度と精度を両立することを示した点が新規性です。研究の貢献は新しいモジュールの発明というより、「アーキテクチャのトポロジーそのものを設計変数として最適化する」という方法論の提示にあります。

## 論文の実験結果(定量データ)
評価は nuScenes(6カメラ等を積んだ車で市街地 1000 シーンを記録した公開データセット)で行われました。End-to-End 計画では主に次の指標を使います。

L2 誤差(L2 error、メートル): 予測した自車軌道と人間が実際に通った軌道との平均距離。1秒後・2秒後・3秒後で測り、小さいほど良い(お手本への近さを測る開ループ指標)。衝突率(collision rate、パーセント): 予測どおり進んだ場合に他物体とぶつかる割合、小さいほど安全。そして推論速度(FPS、毎秒処理フレーム数、高いほど速い)とレイテンシ(ミリ秒、小さいほど速い)。

報告の要点。PARA-Drive は nuScenes で UniAD と同等以上の計画性能(平均 L2 誤差と衝突率)を保ちつつ、推論速度を大幅に向上させたとされています。論文では、補助ヘッドを外した推論構成で UniAD 比でおよそ3倍規模(報告ではおおむね 3.x 倍)のスループット改善が示されました(絶対値は GPU・版に依存)。要するに「精度は落とさず、むしろわずかに上げて速くする」を実現しています。知覚・地図・予測・占有といった補助タスク自体の精度(検出 mAP、地図 IoU 等)も UniAD と遜色ない水準を維持したと報告されており、並列化が補助タスクを犠牲にしていないことが示されています。

アブレーションが本手法の肝です。タスクの接続トポロジーを直列・並列・部分依存で系統的に比較すると、並列に近い配置が最良の速度・精度バランスを与えました。直列に依存を入れていくと、わずかな精度の利得に対して速度が大きく犠牲になることが定量的に示されています。さらに、補助タスク(検出・地図・予測・占有)を学習から外すと計画性能が低下することが示され、これらが共有表現を鍛える役割を果たしていること、すなわち「学習時には必要・推論時には不要」という非対称性が裏づけられました。占有予測や地図構築を学習に含めることが計画の安全指標(衝突率)に効くことも報告されています。逆に言えば、計画ヘッドだけを単独で学習させると性能が落ち、補助タスクという「足場」が学習の本質であることが分かります。

注意点として、評価は開ループが中心です。後年の研究(AD-MLP、BEV-Planner など)は nuScenes が直進偏重で、自車運動だけを入れた単純な MLP でも低い L2 が出てしまうため、L2 が楽観的に出やすいと指摘しています。速度の優位は明確でも、計画の質は閉ループ評価で改めて確認する必要があります。

## メリット・トレードオフ・限界
メリット。タスクが並列なので推論が速く、リアルタイム車載向きです。学習時はマルチタスクで表現を強化し、推論時は計画だけに絞って軽量化できます。タスク間に直列の誤差伝播がないため、上流の一時的な誤り(誤検出など)が下流を連鎖的に汚しにくい構造でもあります。トポロジーを設計変数として扱う方法論は、他のタスク構成にも一般化できる汎用性があります。

トレードオフと限界。推論時に補助出力(検出・占有・地図)を切ると、それらが本番で得られず、安全監視・解釈・後段のフォールバック(計画が怪しいときにルールベースへ切り替える等)に使える情報が減ります。完全なブラックボックスに近づくと、誤動作時の原因究明や安全保証が難しくなる、という解釈性とのトレードオフが残ります。実運用では補助ヘッドを部分的に残す折衷も考えられますが、その分だけ速度の利得は薄れます。並列にした分、共有 BEV 特徴が全タスクの要求を同時に満たす必要があり、タスク間でどの程度の表現容量を割くかのバランス調整が難しい。各タスク損失の重み付けが性能に効くため、チューニングコストもかかります。最後に開ループ評価中心という分野共通の限界を抱え、閉ループ・実車での再現は今後の検証課題です。

## 発展トピック・研究の最前線
PARA-Drive が示した「トポロジーは設計変数である」という視点は、End-to-End 自動運転の設計を「固定のパイプライン」から「探索すべき空間」へと開きました。表現の軽量化(ベクトル化)と配線の最適化(並列化)は直交するため、両者を統合してさらに速く・正確にする方向が自然です。また「補助タスクは学習にだけ使い推論で外す」という非対称設計は、近年の大規模基盤モデルや知識蒸留(大きな教師モデルの知識を小さな生徒へ移す)とも相性がよく、学習時に豊富な監督を与え推論時に絞る運用の理論的な後ろ盾になっています。

評価面では閉ループ(CARLA、Bench2Drive)・疑似閉ループ(NAVSIM)・多様シーンへの移行が進み、速度と安全性を同時に測る枠組みでの再評価が進行中です。並列構造で補助タスクを外す設計が、こうした厳しい閉ループ評価でも頑健さを保てるかは、今後の重要な検証点です。さらに、補助タスクを「いつ外しても安全か」を自信度に応じて動的に切り替える、適応的なヘッド管理といった発展方向も考えられます。
