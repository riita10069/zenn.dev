# 軌道バウンディングボックス取得とコンテインメント再利用キャッシュ

## ひとことで言うと

地図データ(OpenStreetMap など)をネットワーク越しに取得して機械学習の前処理に使うとき、2つの工夫でダウンロードを賢く減らす設計の話です。第一に、車が走った経路(軌道)全体を囲む四角形(バウンディングボックス、bbox)で地図を一括取得します。第二に、新しいリクエストの範囲が、すでにキャッシュ済みのもっと大きな四角にすっぽり収まるなら、再ダウンロードせずそれを使い回します。bbox は bounding box の略で、点群を囲む軸に平行な最小の長方形のことです。

## 直感的な理解

地図サーバからデータを取るのは、図書館で本を借りるのに似ています。素朴なやり方は「自分の家の周り半径3kmの本を借りる」ですが、もし通勤経路が一直線に10kmも伸びていたら、経路の端の本(地図)が借りられず、そこだけ手元に無い状態になります。これが「中心+半径」取得の落とし穴です。

代わりに「自分が今日歩いたルート全体をすっぽり囲む長方形ぶんの本をまとめて借りる」ことにすれば、ルートが長かろうが短かろうが、経路上のどの地点の地図も確実に手元に来ます。これが「軌道bbox取得」の発想です。

さらに、隣の席の人がすでに「もっと広い範囲の本」を借りていたら、自分が必要な狭い範囲はその中に含まれているので、改めて借り直す必要はありません。借りてある大きい束をそのまま参照すればよい。これが「コンテインメント(内包)再利用キャッシュ」です。地図サーバは多くの場合ボランティアが運営する公共リソースなので、無駄なダウンロードを避けて「礼儀正しく(politely)」使うことは、技術的な効率だけでなくコミュニティへの配慮としても重要です。実際、公共 Overpass インスタンスはレート制限とフェアユース方針を持ち、過剰なクエリは一時的な遮断(429/503 応答)を招きます。

## 基礎: 前提となる概念

**OpenStreetMap(OSM)** は、世界中のボランティアが編集する自由な地図データベースです。道路・建物・歩道・交差点などが、点(node)・線(way)・関係(relation)という3種類の幾何要素で記録されています。node は緯度経度を持つ一点、way は node を順に並べた線や面(道路や建物の輪郭)、relation は複数の way/node をまとめた論理グループ(交差点やバスルートなど)です。各要素は key=value 形式の「タグ」(`highway=residential` のような意味づけ)を持ち、これが地図の意味情報になります。

**Overpass API** は、OSM データに対して空間的な問い合わせ(「この四角の中にある道路を全部ください」)を投げられる公開APIです。クエリ言語 Overpass QL で範囲・要素種別・タグ条件を指定します。公共インフラであるため、適切な User-Agent の設定やレート制限の遵守といった利用マナーが求められます。OSM の生データを直接ダンプする方法(planet.osm や地域抽出ファイル)と違い、Overpass は「必要な範囲だけ動的に切り出せる」のが利点ですが、サーバ負荷が高いクエリは弾かれます。

**バッチ前処理とキャッシュ**。機械学習の学習では同じデータを何エポックも繰り返し読みます(1エポック=学習データを一巡すること)。地図取得やパースのような重い処理を毎回やり直すのは無駄なので、一度処理した結果をディスクに保存(キャッシュ)し、2回目以降はそれを読むだけにするのが定石です。ネットワーク往復はミリ秒〜秒、ディスク読み込みはマイクロ秒〜ミリ秒と、桁が違うため効果が大きい。

**空間インデックスと矩形包含**。「ある矩形が別の矩形に含まれるか」を高速判定する問題は、空間データベースの基礎です。R-tree(Guttman, 1984, SIGMOD)に代表される空間インデックスは、bbox の入れ子関係(MBR: Minimum Bounding Rectangle, 最小外接矩形)を木構造で管理し、範囲検索を対数時間に近づけます。本トピックの内包再利用は、その最も素朴で確実な一例(線形走査版)です。

## 仕組みを詳しく

### 軌道bbox取得(中心+半径ではなく)

軌道上の全点(各フレームの自車位置)の緯度経度を集め、それらを囲む最小の矩形に各辺へ一定のマージン(余白)を足した bbox を計算します。

```
lat_min, lat_max = min(lats), max(lats)
lon_min, lon_max = min(lons), max(lons)
# 各辺に margin_m メートルを度へ換算して足す(経度は cos(緯度) 補正)
dlat = margin_m / deg_to_m
dlon = margin_m / (deg_to_m * cos(radians(lat_center)))
bbox = (lat_min - dlat, lon_min - dlon, lat_max + dlat, lon_max + dlon)
```

これにより「長い軌道は大きく、短い軌道は小さく」自動で範囲が決まり、経路の端を取りこぼしません。マージン `margin_m` は、その地図の用途が軌道の外側にどれだけの周辺地図を必要とするかで決めます。ベクトル地図を使う系では「各フレームで自車中心から切り出す関心範囲(perception range)の角までの距離+バッファ」が最小マージンになり、ラスタ地図を描画する系では描画半径(たとえば 800 m)に合わせます。複数の地図表現を併用するパイプラインでは、両者のマージンをそろえておくと、後述の共有キャッシュで1回の取得に集約できます。

### Overpass クエリの典型形

範囲内の要素とその構成メンバーを過不足なく取るクエリは、おおむね次の形を取ります。

```
(node(S,W,N,E); rel(bn)->.x; way(S,W,N,E); node(w)->.x; rel(bw);); out meta;
```

意味は「指定 bbox(南S・西W・北N・東E)内のノード、それらを参照するリレーション、bbox 内のウェイ、ウェイを構成するノード、それらを参照するリレーションを取得し、メタ情報付きで出力」です。way は構成 node を別途取らないと「線はあるが頂点座標が無い」状態になるため、`node(w)` で補完するのが要点です。`out meta` を付けると各要素のバージョン・タイムスタンプ等が付き、再現性が上がります。無関係な要素を引きずらず、必要な node/way/relation とそのメンバーだけをきっちり取るのがポイントです。

### コンテインメント再利用キャッシュ

取得した生 OSM データ(XML)を gzip 圧縮し、bbox を名前に埋め込んでキャッシュします。

```
osm_bbox_{S:.4f}_{W:.4f}_{N:.4f}_{E:.4f}.xml.gz
```

緯度経度を小数4桁で丸めるのは、4桁=約11mの分解能で、これより細かい差は同じ地図とみなしてよいという割り切りです(丸めすぎると別の場所が衝突し、細かすぎるとキャッシュヒットしにくくなる、というトレードオフのバランス点)。新しい要求が来たときの探索順は次の通りです。

1. 完全一致する bbox のキャッシュがあれば、それを読む。
2. なければ、キャッシュ済みの中から「要求 bbox を完全に包含する(covers)」ものを探し、あればそれを使う。判定は単純な4辺の比較です。

   ```
   # 既存(cs,cw,cn,ce)が要求(rs,rw,rn,re)を内包するか
   covers = (cs <= rs) and (cw <= rw) and (cn >= rn) and (ce >= re)
   ```

3. それでもなければ Overpass から取得し、キャッシュに保存する。

この「内包なら再利用」が肝です。広めのマージンで取得した XML があれば、同じ場所のより狭い要求はその中に収まるので再ダウンロード不要です。生 XML を共有することで、同じ1回の取得を複数の下流(ラスタ枝はグラフ化、ベクトル枝はトークン化)が別解釈で使い回せます。注意点として、内包元の XML をそのまま下流に渡すと余分な要素(要求範囲外)が混ざるため、下流側で要求 bbox による空間クリップを必ずかける必要があります。これを怠ると、内包再利用したサンプルだけ余計な地図が見えてしまい、結果が不一致になります。

### エピソード単位の二段キャッシュ

さらに上位で、連続走行区間(エピソード)ごとに「重い一連の処理(Overpass 取得 → OSM パース → トークン化)を1回だけ」やり、結果を `{frame_index: map_data}` のような辞書にまとめてファイル保存する設計が一般的です。生 OSM パースはエピソードで1回(地図は共有)、フレームごとのパッチ抽出とトークン化はフレーム数ぶん(各フレームで自車位置が違うため)走ります。学習時の DataLoader はこのファイルを読むだけで済み、ネットワークアクセスも重い解析も発生しません。これにより、初回前処理は重いが、2エポック目以降は劇的に速くなります。二段にする理由は、外側(生 OSM 取得)はエピソード全体で共有でき、内側(自車中心のパッチ抽出)はフレーム固有で共有できない、という再利用粒度の違いを分けて最適化するためです。

## 手法の系譜と主要論文

この領域は論文の「新規手法」というより、地理空間工学と機械学習データパイプラインの交点にある工学的最適化です。系譜としては次のように位置づけられます。

- R-tree(Guttman, 1984, SIGMOD)とその派生(R*-tree, Beckmann et al. 1990 など)は、bbox の入れ子で空間検索を効率化する基礎データ構造を確立しました。本トピックの内包判定は、この空間包含という概念の最も単純な応用です。
- Web マップタイル配信(Google Maps 等が2005年前後に普及させた slippy map / XYZ タイル方式)は、「固定グリッドのタイルをキャッシュし、要求範囲を覆うタイルを再利用する」という発想を一般化しました。本トピックの矩形キャッシュは、固定グリッドの代わりに可変 bbox を使う点が異なります。タイル方式は境界での重複が予測しやすい一方、可変 bbox 方式は軌道形状に密着して取得できるという違いがあります。
- OpenStreetMap(2004年開始)と Overpass API は、研究用に再現可能な地図取得を可能にしました。地図を使う自動運転研究の多くがこのエコシステムに依存します。
- SDTagNet(Immel et al., NeurIPS 2025)に代表される SD マップ符号化手法は、データセットごとに OSM を取得する前処理を持ちますが、それらは研究の再現用スクリプトであり、本トピックのようなキャッシュ最適化はそれを大規模学習で実用化するための工学層です。論文が報告する精度はモデルの貢献ですが、その手前のデータ供給を律速させない地味な工夫が、大規模学習を現実的な時間で回す前提になります。

## 論文の実験結果(定量データ)

このトピックは速度・効率の工学であり、評価は精度指標ではなく「取得回数」「前処理時間」「取りこぼしの有無」で測られます。代表的な効果は次の通りです。

- 取りこぼしバグの定量: 「中心+半径」取得では、軌道長が半径を超えるエピソードで地図の端が欠落します。経験的に、半径相当を超える(たとえば 1.5 km 程度を超える)長いエピソードで、エラーを出さずに地図が部分欠損する潜在バグ(latent bug、表に出にくいバグ)が発生し得ます。軌道bbox取得に切り替えると、定義上この欠損は 0 になります。これは精度ではなく「データの完全性」の改善で、欠損した地図領域での予測が静かに劣化するのを防ぎます。
- ダウンロード削減: 内包再利用により、隣接クリップや複数の地図枝が同一領域を重複取得する分が削減されます。マージンをそろえてラスタ枝とベクトル枝を1取得に集約すれば、その領域の取得回数は単純計算で半減します。
- アブレーション的観点: 内包再利用を切ると、毎回完全一致のみのキャッシュになり、わずかに範囲がずれた隣接クリップで再取得が多発します。逆にマージンを過大に取ると、内包再利用率は上がるが1取得あたりのデータ量(圧縮 XML サイズ)が増え、ディスクとパース時間が膨らむというトレードオフが生じます。最適なマージンは「再利用率 × 取得サイズ」のバランスで決まります。たとえばマージンを2倍にすると bbox 面積はおよそ4倍に増えうるため、再利用率の上昇がこのサイズ増を上回るかで採否が決まります。

## メリット・トレードオフ・限界

メリット

- 軌道全体を囲む取得により、長いエピソードでも地図を取りこぼさない(完全性の保証)。
- 内包再利用により、隣接クリップや複数の地図表現がダウンロードを共有でき、サーバ負荷と前処理時間を最小化できる。
- 生データを共有キャッシュにすることで、ラスタ・ベクトルなど複数の下流が1回の取得を使い回せる。
- エピソード単位のファイルキャッシュにより、学習時はファイル読み込みだけになり高速。データ供給が GPU を律速しなくなる。

トレードオフ・限界

- bbox は軸平行の長方形なので、斜めや曲がりくねった経路では実際の走行帯よりかなり広めに取り、無駄なデータを含む(対角線方向の長い軌道で顕著。45度方向に伸びる軌道では有効率が半分以下に落ちうる)。
- キャッシュ済み bbox が増えると、内包判定のためにディレクトリ全体を走査するコストが線形に乗る(R-tree 等の空間インデックスを使えば対数に近づくが、実装は複雑になる)。
- 生データは圧縮しても容量を食う。広域 bbox ほど大きく、ストレージ圧迫の要因になる。
- 公開サーバ依存のため、レート制限・障害・利用規約変更に左右される。大規模利用ではセルフホスト Overpass や事前一括ダウンロード(地域抽出ファイル)が望ましい。

未解決の課題として、最適なマージン自動調整(軌道形状に応じて動的に決める)、矩形でなく走行帯に沿った凸包やバッファ多角形で取得して無駄を減らす、キャッシュ無効化(地図が更新されたら古いキャッシュをどう扱うか。OSM はボランティア編集で日々変わる)などが実務上の論点です。

## 発展トピック・研究の最前線

- 走行帯バッファによる取得: bbox の代わりに軌道周りの一定距離バッファ(緩衝多角形)で取得し、対角長軌道の無駄を削る。空間演算ライブラリ(GEOS / Shapely)で実装可能。
- 空間インデックス化キャッシュ: 内包判定を R-tree で高速化し、数万件規模のキャッシュでもサブミリ秒で再利用候補を見つける。
- データバージョニングとの統合: 地図データセットをデータレイク上でバージョン管理(LakeFS や DVC 等)し、前処理キャッシュと地図のバージョンを紐づけて再現性を担保する流れ。OSM の編集日時(`out meta` のタイムスタンプ)をキャッシュキーに含める発展形もある。
- 地図とセンサの非同期取得: 地図前処理と画像/点群前処理を分離し、地図は安価なCPUで先に焼いておき、GPU学習はそれを読むだけにするパイプライン設計が、データ供給律速の解消に効く。
