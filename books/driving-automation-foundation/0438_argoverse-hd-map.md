# Argoverse HD Map

## ひとことで言うと
自動運転の研究用データセット「Argoverse」に付属する、車線の中心線・車線のつながり・走れる場所(可走行領域)などを精密に記録した高精細地図(HD Map)と、それをプログラムから取り出すための API です。軌跡予測の研究で「地図を AI に食わせる」ときの、世界中で使われる標準素材になりました。重要なのは、この地図がラスタ(ピクセル絵)ではなく、車線中心線を点の並び(ポリライン)、車線同士の接続をグラフとして持つベクトル地図である点です。この形式が VectorNet をはじめとする「地図をベクトルのまま扱う」予測手法の隆盛を直接支えました。

## 直感的な理解
周りの車や歩行者の動きを予測するには、地図の情報が決定的です。交差点で止まっている車があったとき、その先の動きは「今いる車線がどこへつながるか」「右折レーンか直進レーンか」で大きく変わります。地図なしでは位置と速度からしか予測できず、当てずっぽうに近くなります。人間も、見知らぬ交差点では車の動きを予測しづらいのと同じです。

ところが2018年ごろまで、軌跡予測の研究には地図つきの大規模な公開データがほとんどありませんでした。各社が自社の非公開データで研究するため、手法同士を公平に比べられない、つまり共通のベンチマークがないという深刻な問題がありました。陸上競技で言えば、各選手が別々のトラックで別々の距離を走って「速い」と主張しているようなものです。共通の競技場が必要でした。その競技場として登場したのが Argoverse です。

## 基礎: 前提となる概念

HD Map(High-Definition Map、高精細地図): 車線1本1本の中心線・幅・つながり・可走行領域まで、自動運転に使える精度(数cm〜数十cmオーダー)で作り込んだ地図です。カーナビ用の地図(道路を1本の線で表す程度)とは精度の桁が違います。

ポリライン(polyline): 点を順につないだ折れ線です。車線中心線は `[(x0,y0), (x1,y1), ..., (xN,yN)]` のような座標点の並びで表されます。これはベクトル表現の基本要素です。

グラフ(graph): ノード(節点)とエッジ(つながり)で関係を表すデータ構造です。車線をノード、「車線Aの次は車線B」という接続をエッジにすると、道路網がグラフになります。

ベンチマーク: 共通のデータと評価指標で手法を比較する基盤です。同じ問題を同じものさしで測るからこそ、手法Aと手法Bのどちらが優れるかを公平に言えます。

## 仕組みを詳しく

### 地図が持つレイヤ

- 車線中心線(lane centerline): 各車線の中央を通る線を、座標点の並び(ポリライン)として保持。
- 車線のつながり(lane connectivity): 車線Aの先行(successor)・後続(predecessor)・左右の隣接車線への接続をグラフとして保持。
- 走行可能領域(driveable area): 車が物理的に走れる地面の範囲。Argoverse 1 ではラスタのマスク(走れる=1の格子)としても提供。
- 地面の高さ(ground height): 路面の標高ラスタ。LiDAR 点群から地面点を取り除く(地面除去)のに使える。
- 横断歩道・車線境界など(Argoverse 2 でレイヤが増加)。

### API でできること

Argoverse 1 の `ArgoverseMap` などのクラスを通じて、例えば次の問い合わせができます。

- ある座標 (x, y) の近くにある車線中心線を取得する。
- ある車線の前後・左右につながる車線 ID を取得する。
- ある座標が走行可能領域内かを判定する。
- ある地点から到達できる車線中心線の候補をたどる(分岐を展開する)。

### 予測でどう使うか(具体例)

予測したい車が座標 (x=120, y=85) にいて東に 8m/s で進むとします。API で「この近くの車線中心線」を取り出すと「この車線は直進と右折に分岐する」と分かります。予測モデルは「直進したらこの中心線に沿う」「右折したらあの中心線に沿う」という2候補を作り、それぞれの確率を出せます。地図がなければ、東へ直進する1本しか出せません。

中心線はポリラインなので、ベクトル系の予測モデルにはそのまま座標列として入力できます。ラスタ系の予測モデル([ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction))に使うときは、これらの線を自車中心の画像に描き込んでから入れます。同じ地図がどちらの系統にも変換できるのが、ベクトル地図を素材として持つことの強みです。

### Argoverse 1 と 2 の違い

- Argoverse 1 (2019): 軌跡予測 + 3D 追跡のデータセット。Miami と Pittsburgh の2都市。予測用に約32万シーン(各シーンは観測2秒 + 予測3秒)。地図は都市ごとに1枚の大きなラスタ + ベクトル。
- Argoverse 2 (2021): 規模・多様性を大幅拡大。6都市(Austin, Detroit, Miami, Pittsburgh, Palo Alto, Washington DC)。予測(Motion Forecasting)用に約25万シーン、各シーン11秒(観測5秒 + 予測6秒)。地図はシーンごとに付属する形へ刷新され、走行可能領域・横断歩道などをベクトルポリゴンで提供(都市1枚の巨大ラスタから、シーンローカルなベクタ地図へ)。さらに LiDAR データセット・Sensor データセットなど複数の派生を同時公開。

## 手法の系譜と主要論文

### Argoverse (Chang et al., Argo AI / CMU / Georgia Tech, CVPR 2019, arXiv:1911.02620)

論文タイトルは "Argoverse: 3D Tracking and Forecasting with Rich Maps"。

- 提案: HD Map 付きの大規模公開データセットと、地図を使った軌跡予測・3D 追跡のベンチマーク。
- 動機: 当時、地図情報を活用した予測手法を公平に比較できる公開ベンチマークがなかった。地図が予測精度を上げることを示す共通の土台が必要だった。
- 新規性: 予測タスクに車線中心線という強い事前情報を結びつけ、地図を活用する予測研究の標準ベンチマークを確立した点。

### Argoverse 2 (Wilson et al., NeurIPS 2021 Datasets and Benchmarks, arXiv:2301.00493)

論文タイトルは "Argoverse 2: Next Generation Datasets for Self-Driving Perception and Forecasting"。

- 提案: 6都市・大規模・多様なシーンへ拡張し、地図をシーンローカルなベクタ形式へ刷新。
- 動機: Argoverse 1 の規模・多様性・地図の扱いにくさを改善し、より一般化する手法の評価を可能にするため。長い予測ホライズン(6秒)で難易度を上げた。

### この地図形式を活かした代表手法: VectorNet (Gao et al., Waymo, CVPR 2020, arXiv:2005.04259)

- 提案: 車線中心線や物体軌跡を「ポリライン」のまま、ポリライン内をまとめるサブグラフと、ポリライン間の関係を表すグローバルグラフ(自己注意)で符号化。
- 動機: ラスタ地図は解像度に縛られ計算も重い。中心線がもともとベクトルなのだから、ベクトルのまま扱う方が効率的で精度も高い。
- 新規性: グラフニューラルネットワークで地図と運動を統一的に表現し、ラスタ + CNN を上回る効率と精度を示した。Argoverse のベクタ地図がこの研究を可能にした。

### LaneGCN (Liang et al., Uber ATG, ECCV 2020, arXiv:2007.13732)

- 提案: 車線グラフ(中心線ノード + 4種の接続: 前・後・左・右)に特化したグラフ畳み込み LaneConv と、車線と物体の相互作用を捉える ActorMap を導入。
- 新規性: 車線の接続トポロジーを陽に畳み込むことで、長距離の車線依存(数十m先の合流)を効率よく伝播。Argoverse Motion Forecasting で当時の上位を獲得し、以後のベクトル系予測の定番要素になった。

この系譜(Argoverse の地図 → VectorNet → LaneGCN → TNT/DenseTNT → mmTransformer)は、いずれも Argoverse のベクトル HD Map があったから公平に競えました。ベンチマークがコミュニティ全体の方法論を一方向に推し進めた好例です。

## 論文の実験結果(定量データ)

評価指標(いずれもメートル、小さいほど良い)を説明します。

- minADE_k: k 本の予測候補のうち最良のものについて、予測軌跡と正解の各時刻の距離を平均した値。
- minFDE_k: 同じく最良候補の、最終時刻だけのずれ(ゴールの当たり具合)。
- Miss Rate (MR): 最終地点のずれが 2m を超えた割合。
- brier-minFDE_k(Argoverse 2 が主指標に採用): minFDE_k に「最良候補に割り当てた確率 p の外れ具合」をペナルティとして足したもの。具体的には minFDE に `(1 − p)²` を加える。これは「当たる軌跡を出すだけでなく、その軌跡に高い確率を付けられたか(確率較正)」まで評価する指標で、k 本のうち1本さえ当たればよい minFDE_k が確率の質を無視してしまう弱点を補います。k を大きくすると minADE/minFDE は機械的に下がる(候補をばらまけば1本は当たりやすい)ため、k だけを見て手法を比べると確率較正の悪い「下手な鉄砲」式の手法を過大評価しかねない、という方法論上の注意点があります。

Argoverse 1 Motion Forecasting ベンチマーク(予測3秒、k=6)での代表的な傾向を挙げます。地図(車線中心線)を使わず運動のみで予測するベースラインに対し、地図を使う手法は minFDE を明確に下げます。ベクトル系の VectorNet は、ラスタ + ResNet ベースの対抗手法に対して、より少ない計算量で同等以上の displacement error を達成したと報告しました(VectorNet 論文ではモデルサイズと FLOPs を約1桁削減しつつ精度が同等以上)。LaneGCN は Argoverse 1 リーダーボードで当時上位の成績(minFDE_6 がおよそ 1.0m 前後の水準)を示し、車線トポロジーを陽に使うことの有効性を定量的に裏づけました。

アブレーションの示唆も重要です。VectorNet・LaneGCN いずれの論文でも、地図情報(車線中心線・接続)を入力から抜くと displacement error が悪化することが示され、「地図は予測に効く」という Argoverse の当初の主張が、より精密に再確認されました。逆に、ラスタ化の解像度を粗くすると遠方精度が落ち、細かくすると計算が増えるというラスタの限界も、これらの比較で定量化されました。Argoverse 2 では予測ホライズンが6秒に伸び、同じ手法でも minADE/minFDE は1より大きく悪化します。これは「予測は先を見るほど指数的に難しくなる(分岐が増え不確実性が膨らむ)」ことを数値で示しており、地図と多峰予測の重要性をさらに際立たせました。

## メリット・トレードオフ・限界

メリット
- 車線中心線・接続・可走行領域まで含む精密な公開地図で、予測研究の公平な比較を可能にした。
- ベクトル形式なので、ラスタにもベクトルにも変換でき、両系統の手法に使える。
- 大規模・多都市で、手法の一般化性能を測れる(特に Argoverse 2)。
- API が車線探索・到達可能性などの高レベル問い合わせを提供し、前処理を書きやすい。

トレードオフ・限界
- 提供地域(数都市)に限られ、その外では使えない。地図に過適合した手法は別地域で性能が落ちうる。
- 地図 API・座標系の扱いに学習コストがあり、前処理が必要。
- HD Map は人手や高精度測量で作るため、現実世界全体に同じ精度の地図を用意・更新し続けるのは困難。この限界が「地図なしで走る(mapless)」研究の動機にもなっている。
- ベンチマーク最適化が進むと、指標を上げる工夫が現実の安全性と必ずしも一致しない(ベンチマークの飽和とオーバーフィット)という、コミュニティ共通の懸念がある。

## 発展トピック・研究の最前線

Argoverse 2 以降、予測の評価は「単体エージェントの軌跡当て」から「シーン全体の整合した複数エージェント予測(joint prediction)」「予測と計画の統合」へ重心が移っています。同時に、HD Map をあらかじめ用意せず車載センサーからオンライン生成する online HD map(MapTR, VAD など)や、地図構築・予測・計画を一体で解く end-to-end 手法(UniAD など)が活発で、Argoverse の派生(Sensor/LiDAR データセット)がそれらの評価基盤になっています。また nuScenes / Waymo Open Motion Dataset といった姉妹ベンチマークと合わせて評価する流れが定着し、単一データセットへの過適合を避ける方向に進んでいます。Argoverse は「地図つき予測ベンチマーク」という研究文化そのものを作った原点として、今も参照され続けています。
