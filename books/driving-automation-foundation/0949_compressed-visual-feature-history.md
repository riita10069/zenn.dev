# 圧縮視覚特徴ベクトルと視覚履歴バッファ

## ひとことで言うと
いま目の前のカメラに映っている場面を、ごく短い数値のベクトル(たとえば長さ14程度)にギュッと圧縮します。そして毎フレームこの小さなベクトルを「過去の場面を並べた履歴バッファ」に積んでいきます。こうするとモデルは「数秒前まで何が見えていたか」という時間的な文脈を、ごく軽い情報量で覚えておけます。画像を丸ごと貯めるのではなく要約だけを貯めるのがポイントで、これにより時間方向の手がかり(速度や変化)を安価に扱えるようになります。

## 直感的な理解
運転の判断は「いまこの瞬間」だけでは決まりません。直前までの流れが効きます。たとえば、信号が今まさに黄から赤に変わったところなのか、それとも赤になってしばらく経つのか。前の車がさっきブレーキを踏んだのか、ずっと一定速度なのか。横断歩道で人が立ち止まっているのか、踏み出そうとしているのか。これらはどれも1枚の静止画だけでは判別できません。連続した数フレームを見て初めて「動き」や「変化」が分かります。物理的に言えば、位置は1枚から読めても速度や加速度は時間微分なので最低でも2枚、滑らかに推定するならもっと多くのフレームが要るということです。

素朴に考えれば「過去の画像を全部モデルに入れればいい」となります。しかしこれは重すぎます。カメラ画像1枚を畳み込みネットワークに通して得る特徴は、たとえば `[256, 8, 8]`(256チャンネル × 縦8 × 横8 = 16384個の数値)のような大きな塊です。これを何十フレームも保持して毎フレーム全部ネットワークに通すと、メモリも計算量も爆発します。1秒あたり10フレーム、6秒ぶんを貯めれば60枚 × 16384 ≈ 約98万次元です。

そこで「現在の場面を、判断に効くエッセンスだけ残して極端に小さく圧縮し、その小さなものだけを履歴として貯める」という発想が出てきます。たとえば各フレームを長さ14に圧縮し、それを64フレームぶん並べれば 14 × 64 = 896 次元です。98万次元と比べれば三桁近く小さい。これが圧縮視覚特徴ベクトルと視覚履歴バッファのアイデアです。要約だけを時系列に並べることで、メモリ予算を守りながら「最近の流れ」をモデルに渡します。

## 基礎: 前提となる概念

理解のために、いくつかの言葉をほどいておきます。

特徴ベクトル(feature vector)とは、画像や場面を数値の並びで表したものです。生のピクセルではなく、ネットワークが「この場面の意味を要約した」中間表現です。長さ256のベクトルなら、256個の実数で1つの場面を表します。

時系列(time series)とは、時間順に並んだデータのことです。運転の入力はカメラの動画なので、本質的に時系列です。1フレームごとに新しい場面が来て、過去のフレームが蓄積されていきます。

文脈(context)あるいは状態(state)とは、「過去に何が起きたかを要約した、現在の判断に使う情報」です。人間でいえば短期記憶に近い。運転では「さっき何が見えたか」を覚えておくことが文脈にあたります。

リングバッファ / ローリングバッファ(rolling buffer)とは、固定された長さの箱に新しいデータを末尾から押し込み、いっぱいになったら一番古いデータを先頭から押し出していく仕組みです。電車のドア付近で人が乗り降りするのに似ています。容量が決まっているので、いつでも「直近N個」だけを保持します。FIFO(First-In-First-Out、先に入れたものから出ていく)キューの固定長版だと考えるとよいでしょう。

次元削減 / 圧縮(dimensionality reduction)とは、大きなベクトルを小さなベクトルに写すことです。情報の一部は必ず捨てますが、うまく学習すれば「大事な手がかり」だけを残せます。もっとも単純な圧縮は全結合層(線形変換) `y = Wx + b` です。ここで `x` が入力の大ベクトル、`W` が学習される行列、`b` がバイアス、`y` が小さくなった出力です。`x` が192次元、`y` が14次元なら、`W` は `14 × 192` の行列で、合計14×192+14=2702個のパラメータをデータから学びます。

自己回帰(autoregressive)とは、自分の過去の出力を次の時刻の入力として戻す方式です。文章生成で前に出した単語を見て次の単語を決めるのと同じ構造で、本機構では「前フレームで自分が作った圧縮ベクトルを、今フレームの入力として参照する」点がこれにあたります。

## 仕組みを詳しく

典型的な実装では、次の3段階で時間文脈を作ります。

第1段階は、現在フレームの視覚特徴を1本のベクトルに均(なら)すことです。マルチカメラを融合した特徴マップ(たとえば `[B, C, H, W]`、B=バッチ、C=チャンネル、H・Wは空間)を、チャンネル方向に圧縮し空間方向に平坦化して、`[B, 192]` のような中サイズのベクトルにします。192 はたとえば 3チャンネル × 8 × 8 を平らにした数です。空間平坦化の前に 1×1 畳み込みやプーリングでチャンネルを落とすのが普通で、ここで「どの空間位置のどのチャンネルを残すか」がある程度決まります。

第2段階は、その中サイズのベクトルを極端に小さく圧縮することです。1枚の全結合層 `Linear(192 → 14)` を通して `[B, 14]` を得ます。192個の数を14個に押し込むので、情報の大半は捨てられます。しかし「全体としてどういう場面か」という大づかみな手がかりは残ります。なぜ14なのかに深い理論的根拠はなく、「運転判断に効く最小限の要約だけ残す」という設計上の選択値です。小さすぎれば手がかりが失われ、大きすぎれば履歴が重くなる、というトレードオフの中で選ばれます。これはちょうど、情報ボトルネック(information bottleneck、入力から出力に必要な情報だけを通す狭い通路を意図的に作る考え方)を狭く設定する操作に相当します。

第3段階は、その14次元を履歴バッファへ積むことです。履歴は固定長 `[B, 896]` のベクトルで、896 = 14 × 64、つまり「14次元の圧縮シーンを64フレームぶん」並べたものです。新しいフレームが来るたびに、一番古い14次元を先頭から捨て、新しい14次元を末尾に足す。これがローリング更新です。常に直近64フレームぶんの場面要約が保たれます。

形(tensor shape)で時系列を追うと次のようになります。

```
現在フレームの融合特徴   [B, C, H, W]
   └ チャンネル圧縮 + 平坦化
中サイズベクトル         [B, 192]
   └ Linear(192 → 14)
圧縮シーンベクトル       [B, 14]
   └ 履歴バッファ末尾に push、先頭を pop(rolling)
視覚履歴バッファ         [B, 896]   (= 14 × 64 フレーム)
```

数値例で更新を追います。いま履歴が `[v_{t-63}, v_{t-62}, ..., v_{t-1}]`(各 `v` は14次元)だとします。新しいフレーム t で圧縮ベクトル `v_t` が出来たら、バッファは `[v_{t-62}, ..., v_{t-1}, v_t]` になります。先頭の `v_{t-63}` は捨てられます。こうして窓が1フレームぶん前進します。撮影レートが10Hzなら64フレームは6.4秒ぶんの記憶に相当します。

この `[B, 896]` の履歴は、次フレームの方策(policy、現在の状態から運転行動を出す部分)の入力に戻されます。現在フレームの特徴、過去の自車運動(egomotion)履歴と一緒に連結され、軌道予測に使われます。注目すべきは「自分が前フレームで圧縮した要約を、次フレームで自分が参照する」という自己回帰的なループになっている点です。これは時間方向に状態を持ち回るという意味で、RNN(再帰型ニューラルネット)の隠れ状態に似ています。違いは、隠れ状態を学習で暗黙に作るのではなく「明示的に過去N枚の圧縮を並べる」点です。

なぜ全結合の連結という素朴な方式を使うのか。RNN(LSTM/GRU)を使えば可変長の過去を1個の固定隠れ状態に畳み込めますが、学習が時間方向に逐次的で並列化しにくく、長期依存で勾配消失も起きやすい。対して「固定長の窓を平らに並べて連結する」方式は、実装が単純で並列化しやすく、何フレーム前を見ているかが明示的で解釈しやすい。代わりに窓の外の長期文脈は完全に忘れます。どちらを選ぶかは、必要な記憶の長さと計算予算のトレードオフです。

実装上の落とし穴も触れておきます。ローリング更新では先頭を捨て末尾に足すので、バッファ内の位置が「何フレーム前か」という意味を持ちます。もしモデルが「先頭が最古・末尾が最新」という順序を前提に学習しているなら、推論時に詰め方を逆にすると意味が完全に壊れます。また学習データの系列が短い場合、最初のうちはバッファをゼロ埋め(zero-padding)するため、エピソード冒頭では履歴が情報を持ちません。この冷えた状態(cold start)をどう扱うかは設計判断です。

## 手法の系譜と主要論文

時間文脈を固定長の表現に圧縮して持ち回るという考え方には長い歴史があります。

Hochreiter & Schmidhuber, "Long Short-Term Memory"(Neural Computation 9(8), 1997)は、時系列の文脈を固定長の隠れ状態(cell state)に圧縮して持ち回り、ゲート機構で「何を覚え何を忘れるか」を学習で制御する仕組みを提案しました。視覚履歴バッファは、この隠れ状態を学習で作るのではなく「明示的に過去N枚を並べる」素朴版に当たりますが、過去を要約して持ち越すという動機は同じです。Cho et al., "Learning Phrase Representations using RNN Encoder-Decoder for Statistical Machine Translation"(EMNLP 2014)が提案した GRU は LSTM を簡略化した亜種で、ゲートを2つに減らしてパラメータを節約しつつ同等の性能を出すことが多く、運転の時間文脈モジュールでもよく使われます。

Mnih et al., "Human-level control through deep reinforcement learning"(Nature 518, 2015、DQN)は、Atari ゲームで「直近4フレームを積み重ねて入力する」frame stacking を採用しました。1枚ではボールの速度や進行方向が分からないため複数フレームを束ねる、という発想は、圧縮シーンを多数フレーム並べる本機構とまったく同じ問題意識です。frame stacking は「生フレームをそのまま積む」のに対し、本機構は「圧縮してから積む」点で軽量化されています。

Kim & Canny, "Interpretable Learning for Self-Driving Cars by Visualizing Causal Attention"(ICCV 2017)は、運転に効く視覚情報が実際には画像のごく一部に集中することを attention の可視化で示しました。これは「視覚特徴を強く圧縮しても運転判断に必要な情報は残せる」という、本機構の前提を支持する結果です。

運転分野では、過去フレームの特徴を時間方向に蓄えて連結する発想は鳥瞰図(BEV)系でも一般化しています。Huang & Huang, "BEVDet4D: Exploit Temporal Cues in Multi-camera 3D Object Detection"(arXiv:2203.17054, 2022)は、過去のBEV特徴をキューに保持し、自車運動で位置合わせしてから現在のBEVと連結することで速度推定を可能にしました(空間版の時系列メモリ)。Hu et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving"(ICCV 2023)などのエンドツーエンド計画手法でも、時間情報の保持は性能の鍵とされています。

## 論文の実験結果(定量データ)

時間情報を入れることの効果は、複数の分野で定量的に確認されています。

DQN(Mnih et al., 2015)では、49本の Atari ゲームで人間のスコアを正規化した指標(human-normalized score、人間=100%、ランダム方策=0%)で評価しました。frame stacking を含む構成は29本で人間以上(100%超)のスコアを達成しました。論文中の議論では、入力を単一フレームにすると Breakout や Pong のように「動きの速度・方向」が判断に効くゲームで性能が大きく落ちることが示されています。これは「1枚では動きが取れない」ことの直接の証拠です。指標の意味は単純で、100%なら人間のテスターと互角、200%なら倍のスコアということです。

BEVDet4D(Huang & Huang, 2022)は、nuScenes という大規模自動運転データセットで評価しました。nuScenes の主要指標 NDS(nuScenes Detection Score、検出精度に加え速度・向き・属性の誤差をまとめた0〜1のスコア、高いほど良い)で、時間情報を加えることで速度推定誤差 mAVE(mean Average Velocity Error、単位 m/s、小さいほど良い)がおよそ半減し、報告によると mAVE が約0.9 m/s 級から0.5 m/s 前後へ下がり、NDS が数ポイント向上したとされます。とくに「他車がどれだけ速く動いているか」という速度の推定は、単一フレームでは原理的に不可能で、時間文脈があって初めて精度が出ます。NDS は0〜1のスコアで、1ポイント(0.01)の差でもベンチマーク順位が入れ替わる程度には敏感な指標です。

これらに共通する読み方は、「速度・加速度・変化の方向」という時間微分的な量こそが時間文脈の最大の価値だということです。位置(いまどこにあるか)は1枚で取れますが、速度(どこへ向かっているか)は最低2枚要ります。圧縮履歴バッファは、その速度・変化の手がかりを軽い情報量で持ち込むための仕組みだと理解できます。

## メリット・トレードオフ・限界

メリットの第1は軽さです。過去数十フレームぶんの文脈を、わずか数百〜千次元程度で保持でき、メモリと計算が安い。生フレームを積む frame stacking よりはるかに軽量です。前述のとおり、生特徴を64枚積むと約98万次元のところを896次元に抑えられます。

メリットの第2は、時間的手がかりを扱えること。信号変化、発進・停止、他車の接近速度など、静止画では取れない情報を方策に渡せます。

トレードオフの第1は情報損失です。192→14 のような強い圧縮は情報の大半を捨てるため、細かい場面の違いが履歴上で潰れます。何を残し何を捨てるかは学習任せで、明示的に制御しにくい。圧縮幅(14という数)を変えると性能が変わるので、ここはチューニング対象です。圧縮が強すぎると、似て非なる2つの場面(例: 赤信号と消えかけの黄信号)が同じ14次元に丸められ、履歴上で区別がつかなくなる恐れがあります。

トレードオフの第2はエラーの蓄積です。自己回帰で自分の圧縮結果を持ち越すため、ある時点の圧縮が悪いとその誤りが履歴に残り、後続フレームへ伝播します。これは模倣学習一般の共変量シフト(covariate shift、訓練時と実行時で入力分布がずれる問題)と同種の弱さで、DAgger のようなオンライン補正が研究されてきた背景でもあります。

トレードオフの第3は固定窓の限界です。たとえば64フレームの窓なら、それより前の長期文脈は完全に忘れます。交差点に入る30秒前に見た標識のような長期記憶は保持できません。長期記憶が要るなら LSTM/GRU のような可変長を畳み込む方式や、attention で過去全体を参照する方式が必要になります。

未解決の課題としては、「何次元に圧縮するのが最適か」「ローリング窓の長さをどう決めるか」「圧縮ベクトルが時間的に滑らかであるべきか(急に内容が飛ぶと履歴が不連続になる)」といった設計選択に、まだ理論的な指針が乏しい点が挙げられます。多くは経験的に決められています。また、圧縮ベクトルが「何を符号化しているか」の解釈性が低く、デバッグや安全性検証の観点では不透明という弱点もあります。

## 発展トピック・研究の最前線

時間文脈の持ち方は、近年さらに洗練されています。第1の方向は、固定長連結から attention ベースの時間融合への移行です。Transformer の自己注意で過去の全フレームを参照し、どのフレームを重視するかを学習で決める方式は、固定窓の制約を緩めます。BEVFormer の時間自己アテンションはこの代表例で、過去BEVを attention で参照します。

第2の方向は、世界モデル(world model)との統合です。過去を要約するだけでなく「この後どうなるか」を予測する表現を持たせると、計画の質が上がるという流れがあります(関連: 将来特徴予測)。過去の圧縮履歴と未来予測を対にして持つことで、時間軸の前後両方を扱う設計が増えています。

第3の方向は、明示的なメモリ機構です。Neural Turing Machine や Memory Networks の系譜で、読み書き可能な外部メモリを持たせ、長期と短期を分けて扱う研究があります。運転では「直近の数秒」と「このルートで何分前に通った場所」のように時間スケールの異なる記憶を分離する試みにつながります。さらに、状態空間モデル(SSM、Mamba など)が長い系列を線形計算量で扱える新潮流として注目され、固定窓に頼らず長期文脈を効率的に保持する候補になっています。
