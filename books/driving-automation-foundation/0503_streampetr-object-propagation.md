# StreamPETR型 オブジェクト中心伝播

## ひとことで言うと

カメラ映像から周囲の車や歩行者を3次元で検出するとき、過去の情報をどう引き継ぐかは大きな問題です。素直にやると「鳥瞰図(真上から見た地図のような特徴マップ)を丸ごと」覚えますが、これは巨大でメモリを食います。オブジェクト中心伝播とは、過去から引き継ぐものを「検出した物体ごとの軽いベクトル(オブジェクトクエリ)」だけに絞る発想です。覚えるものが「地図全体の数千万個の数字」から「数百個の物体ベクトル」へ激減します。引き継ぐときには、自車や物体の動きで位置がずれているので、運動情報を使って整える特別な正規化を挟みます。これにより、何フレーム覚えてもメモリがほぼ一定のまま、長期の時間文脈が使えるようになります。

## 直感的な理解

運転中に脳が本当に追いかけているのは、シーンの全ピクセルではありません。「左前の白い車」「右から横断しそうな歩行者」といった、数えられる個数の物体です。背景の路面やビルは「だいたい変わらない静的な舞台」として大ざっぱに扱い、注意の大半は動く物体に向けます。

ところが初期の時系列BEV手法は、この人間的な省略をせず、毎フレーム「地図全体」を記憶として保存していました。地図全体を覚える方式は表現力こそ高いものの、フレーム数ぶん巨大テンソルが積み上がり、長い時間を覚えようとするとすぐメモリが破綻します。例えば 200×200×256 の密なBEVマップは1フレームで約1000万要素あり、これを8フレーム並べると8000万要素、勾配やオプティマイザの状態まで含めると数GBに膨らみます。

オブジェクト中心伝播は「追いたいのは物体だ。ならば物体だけ覚えればいい」という割り切りです。各物体を1本のベクトル(クエリ)で表し、それだけをフレーム間で持ち回ります。物体クエリは1個あたり256次元程度なので、数百個でも数十万個の数字で済み、密なBEVマップの100分の1以下になります。これが「メモリ効率的な時系列手法」と呼ばれる理由です。動画を1フレームずつ流れてくる順に処理し、フレームが来るたびに記憶を逐次更新するので「ストリーミング(streaming)」とも呼ばれます。

## 基礎: 前提となる概念

このトピックを理解するには、いくつかの用語を先に噛み砕いておく必要があります。

- 3D物体検出(3D object detection): 画像から「物体が3次元空間のどこ(x, y, z)に、どのくらいの大きさ(幅・奥行き・高さ)で、どの向きで、どんな速度で存在するか」を当てるタスクです。2次元の四角(バウンディングボックス)を当てる普通の物体検出を、奥行き・向き・速度まで含めて3次元に拡張したものです。自動運転では速度まで当てることが特に重要で、これは1枚の画像では原理的に不可能なので時間情報が必須になります。

- オブジェクトクエリ(object query): 「ある物体について問い合わせるベクトル」です。Transformer型検出器(DETR系)では、あらかじめ N 本の学習可能なベクトルを用意し、各ベクトルが画像特徴と相互作用しながら「自分はどの物体を表すか」を獲得していきます。出力として各クエリが (位置, サイズ, 向き, クラス, 速度) を吐きます。1本のクエリ ≒ 1個の物体スロット、と考えてよいです。

- アテンション(attention): 「どの情報にどれだけ注目するか」を学習で決める仕組みです。クエリ(問い合わせ)・キー(目印)・バリュー(中身)の3つを使い、クエリとキーの内積で注目度(重み)を計算し、その重みでバリューを加重平均します。クロスアテンションは「クエリ側(物体)が、別のもの(画像特徴)を参照する」形を指します。

- レイヤー正規化(Layer Normalization, LN): ベクトルの各要素を「平均0・分散1」に揃えてから、学習可能なスケール係数 γ とシフト係数 β を掛け足す処理です。式は y = γ ⊙ (x − μ) / σ + β で、μ と σ はそのベクトルから計算した平均と標準偏差です。値の大きさを毎回そろえることで学習を安定させます。

- 位置埋め込み(positional embedding): 「この特徴は空間のどこにあるか」をベクトルとして特徴に足し込む情報です。3D検出では、各特徴が3次元空間のどの方向・どの深度に対応するかを符号化します。アテンションそのものは要素の順番や位置を区別しないので、位置情報を別途注入する必要があります。

- ego-motion(自車運動): 自車そのものの移動(前進量・横移動量・回転角)です。GPS/IMU/オドメトリから得ます。過去の情報を現在の座標系へ揃えるのに使います。詳細は [自車運動補償（ego-motion alignment）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0266_ego-motion-compensation)。

## 仕組みを詳しく

オブジェクト中心伝播の処理を、ストリーミング型の代表的な実装を例に段階的に分解します。

ステップ1: メモリキューに物体クエリを溜める。前フレームで検出・追跡した物体クエリ群を保持します。例として最大256個の物体、各256次元なら形は `[256, 256]` です。これを密なBEVマップ `[200, 200, 256]`(約1000万要素)と比べると桁違いに小さい(約6.5万要素、約150分の1)ことが分かります。多くの実装ではメモリキューに「上位 K 個の信頼度が高いクエリ」を数フレームぶん保持します。

ステップ2: 運動による位置更新。物体は動いているので、前フレームのクエリが指す位置はもう古い情報です。各物体の推定速度 v と経過時間 Δt から、現在いるであろう位置を x_now ≈ x_prev + v · Δt のように予測して更新します(等速モデル)。さらに自車も動いているので、ego-motion ぶんの座標変換も施し、前フレームの物体位置を現フレームの自車中心座標系に移します。

ステップ3: 運動考慮レイヤー正規化(Motion-aware Layer Normalization, MLN)。ここがこの系統の核心です。物体クエリには位置埋め込みが含まれますが、時間が経って位置がずれると、その位置埋め込みも古くなります。MLNは、運動情報(自車運動 + 物体運動 + 経過時間 Δt)を小さなネットワークに通して変換係数を計算し、レイヤー正規化のスケール γ とシフト β をその係数で動的に作り替えます。

```
通常のLN:        y = γ        ⊙ normalize(x) + β
Motion-aware LN: y = γ(motion) ⊙ normalize(x) + β(motion)
                       └─ ego運動・物体運動・Δt から計算 ─┘
```

つまり「どれだけ時間が経ち、自車と物体がどれだけ動いたか」に応じて、クエリの正規化の仕方を変えるのです。これにより、時間でずれた物体クエリを現在フレームの座標系に合った状態へ整えます。可変フレームレート(不規則な時間間隔)にも、Δt を明示的に入力に取るため自然に対応できます。実装上は ego-pose の差分(回転・並進)と Δt を結合したベクトルを MLP に通し、γ と β を出力します。

ステップ4: 現フレームとのクロスアテンション。整合済みの過去クエリと、現フレームのマルチビュー画像特徴をクロスアテンションで相互作用させ、各物体の状態(位置・速度・クラス)を更新します。3D位置埋め込みを画像特徴側に注入する(PETR系の流儀)ことで、明示的なBEVグリッドを介さず画像特徴から直接3D情報を引き出せます。新しく現れた物体には新規クエリを割り当て、画面外へ消えた物体のクエリは捨てます。

ステップ5: ストリーミング更新。更新後のクエリを次フレームのメモリへ渡し、フレームごとに逐次回します。学習時は「propagation training」と呼ばれる工夫を使い、数フレームを通してクエリを伝播させながら勾配を流すことで、推論時のストリーミング挙動を学習でも再現します。1フレームあたりの計算量は「メモリ内のクエリ数 × 画像特徴数」のオーダーで一定に保たれ、何フレーム覚えてもメモリは爆発しません。

数値例で効果を実感しましょう。8フレームぶんの過去を持ちたいとします。密マップ方式なら 1000万 × 8 ≈ 8000万要素ですが、物体クエリ方式なら 6.5万 × 8 ≈ 52万要素で、密マップ1枚(1000万)の20分の1にすら届きません。この差が「長期の時間モデリングを安価に回せる」直接の理由です。

## 手法の系譜と主要論文

オブジェクト中心の3D検出は、密なBEVを使う系統とは別の流れとして発展してきました。

- DETR (Carion et al., ECCV 2020, arXiv:2005.12872): すべての出発点。学習可能なオブジェクトクエリ集合とアテンションで、2D物体検出を「集合予測(set prediction)」として解いた最初の手法。NMS(重複した検出を後処理で消す手続き)もアンカーも不要にしました。後続のクエリベース検出器はすべてこの子孫です。

- Deformable DETR (Zhu et al., ICLR 2021, arXiv:2010.04159): DETRの収束の遅さ(500エポック級)を、各クエリが少数のサンプリング点だけを参照する変形アテンションで解消し、計算量を大幅に削減しました。3D化された後続手法の多くがこのサンプリング機構を継承します。

- DETR3D (Wang et al., CoRL 2021, arXiv:2110.06922): 3Dの参照点を画像平面に逆投影して特徴をサンプリングし、マルチビュー画像から直接3D検出する先駆け。明示的な密深度推定もBEVグリッドも作らず、クエリ起点で3Dを解く発想を確立しました。

- PETR (Liu et al., ECCV 2022, arXiv:2203.05625): マルチビューカメラ特徴に「3D位置埋め込み(3D Position Embedding)」を注入し、明示的なBEV特徴を作らずに3Dオブジェクトクエリで直接検出します。「Position Embedding TRansformer」が名前の由来です。BEVグリッドを介さない点が後の効率化の伏線になりました。

- PETRv2 (Liu et al., ICCV 2023, arXiv:2206.01256): PETRに時間方向を最初に足したバージョン。前フレームの3D座標を現フレームへ ego-motion で揃えて2フレーム融合します。ただし融合は基本2フレームで、長期ストリーミングではありません。

- Sparse4D / Sparse4Dv2 (Lin et al., arXiv:2211.10581, 2305.14018): スパースなアンカー(箱の候補)を、複数フレーム・複数視点・複数スケールにわたって特徴サンプリングして洗練する系統。v2では「instance のリカレント伝播」を取り入れ、StreamPETRと同時期に「スパースに時間融合する」問題意識を競い合いました。

- StreamPETR (Wang et al., ICCV 2023, arXiv:2303.11926): 本トピックの中心。論文タイトルは "Exploring Object-Centric Temporal Modeling for Efficient Multi-View 3D Object Detection"。提案の新規性は2点。(1)密な特徴マップではなく物体クエリそのものをフレーム間でストリーミング伝播する「object-centric temporal modeling」。(2)時間で位置がずれたクエリを運動情報で整える Motion-aware LN。動機は、密マップ保存のメモリ・計算コストを避け、長期の時間モデリングを安価にすることです。

- SparseBEV (Liu et al., ICCV 2023, arXiv:2308.09244): 同じく密BEVを避けるスパース系の代表。クエリ適応的な時空間サンプリングで高性能を出します。StreamPETRと問題意識を共有し、同時期に「スパース・クエリベース時系列」を競い合いました。

- MOTR (Zeng et al., ECCV 2022, arXiv:2105.03247): クエリを時間方向に伝播して追跡を解く 2D の代表。物体クエリを「時間を通したアイデンティティの担い手」として扱う発想は、3D検出のクエリ伝播と地続きで、後述の知覚・予測一体化の源流です。

この系譜の論点は一貫して「密なBEVマップの表現力 vs スパースなクエリの効率」のトレードオフです。

## 論文の実験結果(定量データ)

評価は主に nuScenes(自動運転の標準ベンチマーク。6カメラ・360度・約140万カメラ画像、1000シーン、車/歩行者/自転車など10クラス)で行われます。指標を先に説明します。

- mAP(mean Average Precision): 検出の正確さ。物体を取りこぼさず、かつ誤検出を出さないほど高くなります。0〜1で大きいほど良い。nuScenes版は中心距離(0.5/1/2/4m)でマッチングします。

- NDS(nuScenes Detection Score): nuScenes独自の総合スコア。mAPに加えて、位置誤差・サイズ誤差・向き誤差・速度誤差・属性誤差を統合した指標。0〜1で大きいほど良い。「検出できるか」だけでなく「位置や速度まで正しいか」を含むのが特徴です。

- mAVE(mean Average Velocity Error): 速度推定の平均誤差(m/s)。小さいほど良い。時系列手法が最も効く指標で、「物体がどっちにどれだけ速く動くか」の精度を測ります。

報告されている代表的な結果(StreamPETR論文より、おおよその値):

- nuScenes val、ResNet50バックボーン・428×800入力で NDS 約0.55、mAP 約0.45 前後。単一フレームのベースライン(NDS 約0.43〜0.44前後)から、時系列化で NDS が10ポイント以上向上したと報告されています。NDSの1ポイントは体感としてかなり大きく、10ポイント差は「別物のレベルで強くなった」と言える差です。

- 速度誤差 mAVE が大きく低下。単一フレームでは速度をほぼ当てられない(画像1枚に動きの情報がないため、mAVE が 1.0 m/s 近い)のに対し、時系列伝播で過去位置との差分から速度が推定でき、報告では mAVE がおよそ 0.3 m/s 前後まで下がるケースがあります。これが NDS 向上の主要因です。

- 推論速度。ストリーミング設計のおかげで、ResNet50 設定でリアルタイム級(報告でおよそ 30 FPS 前後)の推論が可能とされ、密マップを多フレーム積む方式より高速です。「精度と速度の両立」が論文の売りの一つです。

- nuScenes test の大型バックボーン設定(V2-99 等)では NDS 0.63前後・mAP 0.55前後と、当時のカメラのみ手法でトップクラスを記録しました。

- 効率・要素のアブレーション: メモリキューのクエリ数(覚える物体数)やメモリ長(何フレーム遡るか)を変える実験で、密マップ方式に比べてメモリがほぼ一定に保たれること、Motion-aware LN を外すと特に速度誤差が悪化することが示されています。MLN は「あると速度推定が安定し、抜くと劣化する」要素として位置づけられます。propagation training(数フレームを通して学習)を外すと、学習と推論のギャップで性能が落ちることも報告されています。

注意として、これらの数値は入力解像度・バックボーン・学習エポック数で大きく変動します。論文間の比較では設定を揃えて見る必要があります。

## メリット・トレードオフ・限界

メリット:

- メモリ効率が極めて高い。密マップの数十〜数百分の1で過去を保持でき、何フレーム覚えてもほぼ一定メモリ。
- 1フレームあたりの計算量が一定なので、長期の時間モデリングが安価で、リアルタイム推論と両立する。
- 可変フレームレートに自然対応(Δt を明示入力に取るため)。
- 速度推定の精度が高い。時系列手法の主目的の一つを直接的に満たす。

トレードオフ・限界:

- 物体中心クエリが前提。空間グリッド型(BEVクエリ。格子の各マスに1クエリを置く方式)とは構造が根本的に異なり、グリッド型システムへ後付けで足すには大きな改修が要ります。
- 検出されなかった物体や背景・路面・障害物の時間情報は持ち回りません。密マップなら背景も記憶できる点と対照的です。BEVセグメンテーション(道路・車線・歩道を「面」として塗るタスク)とは相性が悪い。
- 物体の対応付け(同じ物を追い続けること)が崩れると時間情報も崩れます。混雑シーンでIDが入れ替わると、過去状態が別物に紐づく失敗が起きます。
- Motion-aware LN が暗黙に仮定する等速直線運動は、急加速・急ハンドル・歩行者の方向転換では破綻し、整合誤差が残ります。

未解決の研究課題としては、(1)物体クエリと密マップのハイブリッド(動く物体はクエリ、静的な舞台は粗いマップ)、(2)等速モデルを超えた運動予測でのクエリ整合、(3)長時間オクルージョン(長く隠れた物体)のクエリをいつ捨て、いつ復活させるかのメモリ管理ポリシー、などが挙げられます。

## 発展トピック・研究の最前線

オブジェクト中心の時系列モデリングは、検出にとどまらず「クエリを下流タスクへ流す」方向へ発展しています。クエリベースの時系列追跡(MOTR系)、クエリをそのまま軌道予測や行動計画へ渡すエンドツーエンド自動運転(UniAD, VAD など)では、物体クエリが「知覚から予測・計画まで一貫して運ばれる共通の通貨」になりつつあります。StreamPETR型の伝播はこの流れの知覚側の基盤技術です。

また、密BEVの長期保持側(temporal embedding で長期文脈を保つ再帰融合)との対比は研究上の重要な軸です。スパース・クエリ系が効率で勝つ一方、密マップ系は「面」のタスクや背景情報で勝ち、どちらが優れるかはタスクとシーンの動的さに依存します。両者の統合は活発な研究テーマで、近年は「静的背景は粗い長期BEV、動的物体は精密なクエリ」と役割分担するハイブリッドが有望視されています。さらに、クエリ伝播をオンライン地図構築(HDマップのベクトル要素を時間方向に伝播する手法)へ拡張する流れも出ています。
