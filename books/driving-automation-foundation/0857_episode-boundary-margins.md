# エピソード境界マージンとサンプルインデックス

## ひとことで言うと
学習に使う時系列データは、「1回の連続した記録 (エピソード)」がいくつも連なってできています。あるフレームの「過去数秒」と「未来数秒」をまとめて取り出したいとき、その窓 (まど、連続したフレームの区間) が別のエピソードにはみ出してはいけません。はみ出さないように、各エピソードの端から十分な余白 (マージン) を空けてサンプルを作る設計を、初歩から研究レベルまで掘り下げます。複数種類の窓を同時に扱うときに「マージンの最大値を取る」のがなぜ正しいのか、境界違反をなぜ静かに無視せずエラーで止めるのかまで踏み込みます。

## 直感的な理解
動画の中から「今この瞬間」を1つ選び、その前後を切り出して「過去を見て未来を当てる」訓練をしたいとします。ところがデータは複数の別々の走行・録画がつなげて並んでいることが多い。エピソードA (昨日、ある街での走行) の最後のフレームを「今」に選ぶと、「1秒後・2秒後」を取りに行った先がエピソードB (今日、まったく別の街) になってしまう。これは「昨日の動画の続きが、なぜか今日の別の場所の動画」という、時間的につながっていない偽の系列です。

こんなデータで未来予測を学ばせると、モデルは存在しない因果関係を覚えてしまい、性能が壊れます。たとえば「Aの最後で右折していたら、Bの最初で別の街の風景が現れる」といった、物理的に意味のない対応を学習信号として吸収してしまう。これを防ぐには、エピソードの端から「窓がはみ出さないだけの余白」を空けて、その内側だけを「今」として選べばよい。この余白の幅を、窓の構造から計算するのが本ノートの主題です。

## 基礎: 前提となる概念

用語を噛み砕きます。

- エピソード (episode): 1回の連続した記録の単位。ある日のA地点からB地点までの走行動画一本、あるいは強化学習なら1回の試行 (リセットから終了まで)。データセット内部では、複数エピソードが1本の長いフレーム列として連結され、各エピソードの開始位置・長さが別途記録されているのが一般的です (例: 全フレームを一列に並べたうえで、各エピソードの開始インデックスと長さのテーブルを別に持つ)。

- 窓 (window): 連続したフレームの区間。「過去窓 (history window)」は現在より前のフレーム列、「未来窓 (future window)」は現在より後のフレーム列。

- マージン (margin): エピソードの端から、窓が収まるために必要な余白フレーム数。

- データリーク (data leakage): 本来モデルが知りえない情報が、訓練データに紛れ込むこと。ここでは「別エピソードのフレームが、時間的につながっているかのように混入する」クロスエピソードリークが該当します。リークがあると、検証時の性能が見かけ上良くても実運用で崩れます。リークの怖さは、評価指標が「良い」と言っているのに本番でだめ、という最も発見しにくい失敗を生む点にあります。

- fail fast (早く失敗する): バグがあったとき、間違ったまま処理を続けるのではなく、その場でエラーを出して止める設計思想。静かに壊れたデータで学習し続けるより、すぐ気づけるほうが安全です。

複数の窓を同時に扱う状況も前提として押さえます。実応用では、1つのサンプルから性質の異なる複数の系列を取り出すことがよくあります。例えば、

1. 自車の動き (egomotion = エゴモーション) を学ぶための高密度な窓 (例 過去64フレーム・未来64フレーム、生周波数のまま)。
2. 世界モデルの未来予測のための、粗く間引いた窓 (例 stride 10 で過去4個・未来4個、[Hz リサンプリングと stride 計算](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0860_hz-resampling-stride) 参照)。

この2つは必要とする余白が違います。両方を同時に満たすには「より大きいほうの余白」を確保すればよい、というのが「最大値を取る」設計の核心です。

## 仕組みを詳しく

### ステップ1: 窓のオフセットを決める

現在フレームの位置を 0 として、過去・未来の各フレームが「現在から何フレームずれた位置にあるか」を計算します。

```python
def window_offsets(num_frames, stride):
    history = [-(num_frames - 1 - i) * stride for i in range(num_frames)]
    future  = [(i + 1) * stride for i in range(num_frames)]
    return history, future
```

`num_frames=4`, `stride=10` のとき:

- history = `[-30, -20, -10, 0]` (古い順→新しい順。現在 0 が最後に入る)
- future = `[+10, +20, +30, +40]` (1ステップ先から4ステップ先)

つまり現在が `t=100` なら、過去は 70, 80, 90, 100、未来は 110, 120, 130, 140 を取ります。過去窓には「現在 (0)」が含まれ、未来窓は「現在の次 (+stride)」から始まる、という非対称が後で効いてきます。この非対称は「現在は既知 (過去側に属する)、未来は次の時刻から」という自然な約束に対応します。

### ステップ2: 必要な余白を計算する

窓を作るのに、現在フレームの前に何個、後ろに何個あればよいかを返します。

```python
def required_margins(num_frames, stride):
    return (num_frames - 1) * stride, num_frames * stride
```

`num_frames=4`, `stride=10` のとき `((4-1)*10, 4*10) = (30, 40)`。

- 前に 30 フレーム必要 (history が -30 まで届く)
- 後ろに 40 フレーム必要 (future が +40 まで届く)

未来側の余白が大きいのは、過去窓には現在 (0) が含まれるのに対し、未来窓は +stride から始まり、1ステップ分先まで届くからです。`(num_frames-1)*stride` と `num_frames*stride` の差がちょうど stride 1 ぶんです。

### ステップ3: 複数種類の窓のマージンを最大値で統合する

これが設計の肝です。窓が2種類あるとき、それぞれの (前余白, 後余白) を計算し、要素ごとに最大値を取ります。

```python
past_margin   = HIST_TIMESTEPS    # 例: 64 (高密度な egomotion 窓)
future_margin = FUTURE_TIMESTEPS  # 例: 64
if world_model_enabled:
    wm_past, wm_future = required_margins(wm_num_frames, wm_stride)  # 例: (30, 40)
    past_margin   = max(past_margin,   wm_past)    # max(64, 30) = 64
    future_margin = max(future_margin, wm_future)  # max(64, 40) = 64
min_len = max(MIN_FRAMES, past_margin + future_margin + 1)
```

なぜ最大値で正しいのか。前余白を `max(64, 30) = 64` にすれば、「64 個の余白がある」状態は当然「30 個の余白がある」状態も含みます。大きいほうを確保すれば小さいほうも自動的に満たされる、という単純な包含関係です。この例では高密度窓 (64) が勝ちますが、もし世界モデル窓のほうが大きい設定 (例 8 個・stride 10 → 後余白 80) なら、そちらが採用されます。どちらの窓も必ず収まることが、最大値によって保証されます。これは「複数の制約をすべて満たす最小の安全マージンは、各制約の上限を取ったもの」という、制約充足の素直な帰結です。

`+1` は現在フレーム自身のぶんです。前余白 + 現在 + 後余白 がエピソードに収まる最小長 `min_len` になります。

### ステップ4: 有効なフレームだけをサンプル化する

各エピソードについて、「現在」に選べるフレーム範囲を決めます。

```python
min_frame = past_margin                  # 先頭から past_margin だけ空ける
max_frame = ep_len - future_margin - 1   # 末尾から future_margin だけ空ける
for frame_idx in range(min_frame, max_frame + 1):
    samples.append((ep_idx, ep_start + frame_idx))
```

エピソード長が短くて窓が入らない (`ep_len < min_len`) 場合は、そのエピソードを丸ごとスキップします。こうして集めた `samples` リストが「このデータセットで学習に使える全サンプル」になり、データセットの `__getitem__(idx)` はこのリストの idx 番目を引きます。サンプルインデックスをこの段階で一度だけ作っておくのは、各エポックで毎回境界判定をやり直す無駄を省く意味もあります (前計算によるインデックス化)。データローダから見れば、サンプル数 = `len(samples)` であり、各サンプルは「どのエピソードの、どの絶対フレーム番号が現在か」のタプルとして表現されます。

### ステップ5: 念のための二重チェック (fail fast)

サンプルインデックスが正しければ窓は必ず収まりますが、窓を実際に組み立てる関数の内部でもう一度確認します。

```python
if t + hist_off[0] < ep_start or t + fut_off[-1] >= ep_end:
    raise IndexError("window exceeds episode boundary")
```

`hist_off[0]` は最も古いオフセット (-30)、`fut_off[-1]` は最も先のオフセット (+40)。これらがエピソード範囲 `[ep_start, ep_end)` の外に出たら、黙って間違ったフレームを返すのではなく、わざとエラーで止めます。これが fail fast です。仮にステップ4のインデックス作成にバグがあっても、ステップ5が誤データの混入を即座に検出します。静かに壊れたデータで何時間も学習し、後で「なぜか性能が出ない」と悩むより、はるかに安全です。一般に、配列の添字でデータ破損が起きると例外ではなく「たまたま隣の有効な値」を拾ってしまい、エラーにすらならないことが多い。だからこそ、論理的には冗長に見えるチェックを意図的に残すのが堅牢なデータパイプラインの作法です。

## 手法の系譜と主要論文

「窓のマージン計算」そのものを論じる論文は多くありませんが、その背後にあるデータリーク防止・エピソード尊重の原則は、時系列学習・強化学習・自己教師あり学習で広く確立しています。

- 時系列予測の標準的教科書 (Hyndman & Athanasopoulos, "Forecasting: Principles and Practice", 第3版オンライン無料公開) は、訓練と検証を時間順に分け、未来の情報が過去に漏れる「リーク」を防ぐことを鉄則として繰り返し強調します。時系列のクロスバリデーション (交差検証) では、ランダム分割ではなく時間方向の前進的分割 (time-series cross-validation、訓練窓を時間順にスライドさせる) を用いるべきだと述べています。エピソード境界をまたがない窓設計は、この原則をデータ生成時点で守る実装に相当します。

- 強化学習・世界モデル系 (DQN の経験再生 Mnih et al., Nature 2015、DreamerV3 Hafner et al. 2023) では、学習はエピソード単位のリプレイバッファから行い、エピソードをまたいだ遷移を作らないのが標準です。各エピソードの終端 (done フラグ) を尊重し、終端を越える系列は学習対象にしません。終端をまたぐ偽の遷移を混ぜると、ブートストラップ (次状態の価値を使って現状態の価値を更新する手続き) が「実際には続かない未来」の価値を取り込んでしまい、価値推定が破綻します。動画フレームの窓に対する境界チェックは、この思想を観測系列に適用したものです。

- 自己教師あり動画学習 (V-JEPA, Bardes et al. 2024 など) では、時空間のクリップを切り出して学習しますが、クリップが意味的に連続した区間に収まることが暗黙の前提です。シーン切り替えをまたぐクリップは学習を乱すため、データ前処理でショット境界 (shot boundary) を検出して分割する手法も併用されます。エピソード境界マージンは、走行データにおける「ショット境界尊重」と見なせます。

系譜としては、(1) 時系列予測でのリーク防止という古典原則 → (2) 強化学習でのエピソード終端尊重 → (3) 動画自己教師あり学習でのクリップ境界尊重、という流れの中に位置づけられます。

## 論文の実験結果(定量データ)

境界・リークの扱いが性能に与える影響を示した報告を挙げます。

- データリークの一般的影響: 機械学習の再現性研究 (Kapoor & Narayanan, "Leakage and the Reproducibility Crisis in ML-based Science", Patterns, 2023) は、時間的・集団的リークによって報告精度が過大評価される事例を多数の分野横断で指摘しています。彼らのレビューでは、十数分野・数百本規模の論文でリーク起因の過大評価が確認され、リークありの設定では検証精度が見かけ上数ポイント〜十数ポイント高く出るのに、リークを正しく除くと大幅に低下する、という逆転が複数分野で報告されています。これは「境界を守らないと、評価が信用できなくなる」ことの定量的裏づけです。

- 時系列クロスバリデーションの効果: 時系列予測の実証研究では、ランダム分割 (リークを含む) と前進的分割 (リークなし) で、テスト誤差の見積もりが大きく食い違うことが繰り返し示されています。ランダム分割だと未来のサンプルが訓練に混ざり、汎化誤差を楽観的に見積もり、本番で誤差が膨らみます。

- 世界モデルのサンプル効率: DreamerV3 (Hafner et al., 2023) は、150 以上の環境で単一ハイパーパラメータ設定により高い性能を達成しました。エピソード終端を正しく扱うリプレイ設計は、こうした安定した学習の前提条件であり、終端をまたぐ偽遷移を混ぜると価値推定が破綻することが、関連研究で示されています。

直接「マージン幅 N でどれだけ精度が変わるか」を測った公開ベンチマークは稀ですが、上記はいずれも「境界・リークの扱いが、最終性能と評価の信頼性を左右する」ことを定量的に支持しています。境界違反は「壊れたら気づく」種類のバグではなく「静かに評価を汚す」種類のバグである、という点が、これらの研究の共通メッセージです。

## メリット・トレードオフ・限界

メリット
- エピソードをまたぐ偽の系列を、データ生成段階で根本から排除できる (クロスエピソードリーク防止)。
- 複数種類の窓のマージンを最大値で統合するので、片方の窓設定を変えても、もう片方が壊れない (疎結合)。
- 窓組み立て時の境界チェック (fail fast) により、インデックス生成のバグを早期発見できる。
- オフセット計算とマージン計算が副作用のない純粋関数なので、合成データだけで単体テストでき、本物のデータを用意せずに検証できる。

トレードオフ・限界
- 各エピソードの先頭・末尾の数十フレームは「現在」に選べず、学習に使えない。データの端を捨てることになる。
- マージンが大きい (例 64) ほど、短いエピソードからは取れるサンプルが激減し、極端に短いエピソードは丸ごと捨てられる。短い記録が多いデータセットでは利用効率が落ちる。例えばマージン前後64・最小長129のとき、長さ130のエピソードからは1サンプルしか取れず、長さ128のエピソードは0サンプルになる。
- 最大値で大きいほうに合わせるため、本来は小さい余白でよい窓でも、共存する大きい窓に引きずられて広いマージンを確保してしまう。
- 未解決の課題として、捨てる端のデータをどう活かすかがある。端のフレームでは窓を片側だけにする (片側パディング)、可変長窓を許す、といった工夫は実装が複雑になり、境界保証とのトレードオフになります。

## 発展トピック・研究の最前線
- 可変長・適応的窓: 固定マージンではなく、シーンの複雑さに応じて窓長を変える設計。境界保証を保ちつつ端のデータも活かす試み。
- ストリーミング学習: エピソードを逐次受け取りながら、リークなしで窓を生成するオンライン前処理。記録が終わる前のエピソードについては未来窓が未確定なので、確定したフレームだけからサンプルを生成する工夫が要ります。
- ショット境界検出との統合: 1エピソード内でもシーンが大きく切り替わる箇所 (トンネル出入り、急な明暗変化) を検出し、より細かい意味的境界を尊重する方向。エピソードの記録単位と意味的な連続性は必ずしも一致しないため、両者を区別する研究があります。
