# モジュール内lossカプセル化

## ひとことで言うと
それぞれの生成モジュール（軌跡を作るプランナなど）が、自分の学習に使う誤差（loss）の計算を自分の内部で行い、最終的に1つの数値（スカラ）として返す設計です。学習ループ側に「この方式ならこう誤差を計算する」という分岐や、方式固有の一時テンソルを書かずに済み、方式ごとの計算の取り違えを構造的に防げます。「学習する量」と「出力する量」が一致しない生成モデルでは、この分離が安全策として効きます。

## 直感的な理解
レストランの注文を想像してください。客（学習ループ）は「料理をください、できたら味の評価点も教えて」と頼むだけで、厨房（モジュール）の中でどんな調理をしているかを知る必要はありません。和食でも中華でも、客は同じ頼み方をし、同じ形の「評価点（スカラ）」を受け取ります。厨房の中の包丁さばきや火加減（方式固有の中間計算）は厨房の外に漏れません。客が厨房の中の半製品（下ごしらえ中の食材）を勝手に持ち出して別の皿と混ぜる、ということが起こり得ないのが要点です。

機械学習でこれが特に重要になるのは、生成モデルの方式ごとに「正しい loss の計算方法」がまったく違うからです。

- 模倣学習（imitation learning）系: 予測した軌跡と、お手本（人間の運転など）の軌跡との差を直接測る。
- フローマッチング系: 軌跡そのものではなく、ノイズから目標へ向かう「速度場」を予測し、それが理論上の正解速度にどれだけ近いかを測る。
- 拡散モデル系: 加えたノイズ（あるいはスコア）を予測し、その予測精度を測る。

ここに致命的な落とし穴があります。フローマッチングや拡散の「中間出力」は軌跡ではありません。それなのに学習ループ側が「とりあえず予測と教師軌跡の二乗誤差を取る」という共通処理を書いてしまうと、中間出力（速度場やノイズ）を軌跡だと勘違いして誤差計算し、学習が静かに壊れます。テンソルの形は一致しているのでエラーは出ず、loss の値もそれらしく下がっていきます。エラーは出ないのに正しく学習されない、という発見の難しいバグです。これを防ぐ最も確実な方法が「loss の計算を、その方式を一番よく知っているモジュール自身に任せる」ことです。これがモジュール内 loss カプセル化です。

## 基礎: 前提となる概念
loss（損失）とは、モデルの予測が正解からどれだけずれているかを表す1つの数値です。学習はこの数を小さくする方向に重みを少しずつ調整する作業で、その「調整の方向」を loss から逆算する操作を backward（逆伝播）と呼びます。backward を実行するには、loss が最終的に1つのスカラに集約されている必要があります（複数の数の集まりでは「どちらへ調整すべきか」が一意に決まらない）。複数の項を足し合わせて1つのスカラにする、というのが loss 設計の基本形です。

カプセル化（encapsulation）とは、オブジェクト指向設計の基本原則で、中身の複雑な事情とデータを箱の中に隠し、外からは単純な窓口（メソッド）だけ見せる考え方です。箱の内部状態に外から直接触れさせないことで、不正な操作や取り違えを防ぎます。

速度場（velocity field）とは、フローマッチングで学習する量で、「各位置・各時刻で、サンプルがどちらへどれだけ動くべきか」を表すベクトル場です。ノイズから目標データへ向かう連続的な変形（フロー）を定義し、これを時間方向に積分すると最終的なサンプル（ここでは軌跡）が得られます。重要なのは、速度場と最終出力の軌跡は数学的に別物だということです。前者は「変化率」、後者は「変化を積み上げた結果」です。

時刻サンプリング（timestep sampling）とは、フローマッチングや拡散の学習で、ノイズと目標の間のどの時刻 t（0 が完全ノイズ、1 が完全な目標、などの規約）の中間点を使って学習するかを確率的に選ぶことです。どの t を多く引くかで、学習が安定する領域が変わります。

## 仕組みを詳しく
典型的な設計では、各モジュールが `compute_loss()` のようなメソッドを持ち、その中で自分の目的関数を計算してスカラ loss を返します。抽象基底クラスがこのメソッドを契約として強制します（詳細は [差し替え可能Plannerインターフェース](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1014_swappable-planner-interface)）。

フローマッチングのプランナを例に、何が箱の中に隠れているかを段階的に見ます。学習時の処理は次の通りです。

1. ノイズ `x_0 ~ N(0, I)` を標準正規分布から引く。形は教師軌跡と同じ `[B, T*S]`。
2. 教師軌跡 `x_1` との間を、時刻 `t` で線形に混ぜた中間点 `u_t = (1 - t) * x_0 + t * x_1` を作る。
3. その中間点での「正解の速度」は、線形補間パスの場合 Lipman et al. (2023) の結果より単純に `x_1 - x_0`（時刻によらない定数ベクトル）になる。これがフローマッチングの理論的な美点で、複雑なスコア関数を計算せずに正解の回帰目標が閉じた形で得られる。
4. ネットワークが予測した速度 `v_theta(u_t, t, c)`（c は文脈、たとえば BEV 特徴や履歴）と、正解速度 `x_1 - x_0` との二乗誤差（サンプルごとの MSE）を loss とする。数式で書けば `L = E_{t, x_0, x_1} [ || v_theta(u_t, t, c) - (x_1 - x_0) ||^2 ]`。各記号は、E が期待値（多数のサンプルでの平均）、|| · ||^2 がベクトルの二乗の長さ。
5. 学習時刻 `t` は単純な一様分布ではなく、しばしば偏った分布から引く。たとえば「シフトした Beta(1.5, 1) 分布」や対数正規（logit-normal）分布から引いて、速度場が学びにくい領域にサンプルを寄せる工夫が、近年の大規模運転・ロボット基盤モデルで採られている。

これら（ノイズ、中間点、正解速度、時刻サンプリング）はすべて `compute_loss()` の内部に閉じており、外部スコープには一切漏れません。これが核心です。これらの一時テンソルが外に漏れて呼び出し側のスコープに置かれると、誤って別の相手（たとえば教師軌跡）と組み合わせて誤差計算する事故が起き得ます。箱に閉じ込めることで、その事故をコードレベルで不可能にします。

ベジエ曲線のプランナなら、まったく別の loss が箱の中に入ります。少数の制御点を出力し、その制御点からなめらかな軌跡を構成し、軌跡そのものと教師軌跡の差を測ります。模倣学習系なので速度場やノイズは登場せず、軌跡の直接比較で済みます。同じ `compute_loss()` という窓口を持ちながら、中身は方式に応じて完全に異なります。外から見ると、どちらも「文脈と教師軌跡を渡すとスカラを返す」関数に見えます。

学習ループ側から見た姿はこうです（疑似コード）。

```
loss = planner.compute_loss(context, target_trajectory)
loss.backward()      # 方式を一切 if 分岐していない
optimizer.step()
```

推論用の `forward()` と学習用の `compute_loss()` を分けるのも本質的です。`forward()` は常に完成した軌跡 `[B, T*S]` だけを返します。学習時は速度場を、推論時は軌跡を、という別物を同じ窓口から取り違える事故を、入口を分けることで根絶します。

役割の二層構造に注意します。実務では「方式固有の loss はモジュール内」「全方式に共通の loss は外側」と分けることがあります。全プランナが同一形式の軌跡を出すなら、軌跡に対する共通の評価（平滑性ペナルティ、衝突回避ペナルティ、進行方向の整合性など）は外側に一括で置けます。一方、速度場 MSE のような方式固有の目的関数はモジュール内に残します。「方式固有は内、共通は外」という分担を理解しておくと、どこを直すべきか迷いません。逆にこの境界が曖昧だと、内側と外側で同じ軌跡誤差を二重に計算してしまう、といったミスが起こります。

## 手法の系譜と主要論文
モジュール内 loss カプセル化は、生成モデルが「学習する量」と「出力する量」を分離したことの自然な帰結です。

Flow Matching for Generative Modeling（Lipman et al., ICLR 2023, arXiv:2210.02747）は、条件付き速度場を単純な MSE で学習する枠組み（Conditional Flow Matching）を提案しました。動機は、拡散モデルのスコアマッチングが扱う複雑な目的関数を避け、より単純で安定な回帰目標で生成を学ぶことです。これにより、loss 計算が「速度場という方式固有の量に対する MSE」という形で確立し、軌跡や画像そのものとは切り離されました。loss をその方式の内側に閉じる必然性が、ここで明確になりました。

Diffusion Policy（Chi et al., RSS 2023, arXiv:2303.04137）は、拡散でロボット行動を生成し、loss を「加えたノイズの予測誤差」に対して計算します。行動そのものではない量で loss を取るため、ここでも計算を方式の内側に隠す設計が安全です。

Flow Matching Guide and Code（Lipman et al., 2024, arXiv:2412.06264）は、フローマッチングの実装上の細部（時刻サンプリングの選び方、確率パスの設計、数値積分の方法）を体系的にまとめた実践ガイドで、「これらの細部を呼び出し側に露出させず、生成モジュールの内部に閉じるべき」という設計思想を支えています。時刻サンプリングを一様分布から偏った分布へ変えるだけで学習の安定性が変わるため、こうした調整こそモジュール内に隠して一括管理すべき対象です。

複数モジュールの loss を束ねる外側の設計には、別の系譜があります。Kendall et al.（CVPR 2018, arXiv:1705.07115）は、各タスクの不確実性に基づいて loss の重みを自動調整する手法を提案しました。PCGrad（Yu et al., NeurIPS 2020, arXiv:2001.06782）は、複数タスクの勾配が互いに干渉して打ち消し合うとき、片方の勾配を他方に直交する成分だけ残す「勾配手術」を提案しました。これらは「各モジュールが自分の loss を返す」というカプセル化を前提に、その外側でどう足し合わせるかを扱う発展形です。

## 論文の実験結果（定量データ）
カプセル化そのものの数値ではなく、その内部に隠れる loss 設計が性能に与える影響が、各論文で定量化されています。

Lipman et al. (2023) は ImageNet 等の画像生成で、フローマッチングの loss（速度場 MSE）が拡散のスコアマッチング loss と同等以上の生成品質（FID、生成画像が本物の分布にどれだけ近いかの指標で、小さいほど良い）を、より少ない関数評価回数（NFE）で達成したと報告しています。たとえば同等の FID を、拡散より大幅に少ない積分ステップで到達できたとされ、これは loss の選び方が推論コストにも直結することを示します。loss を方式の内側に隠してあれば、同じ外側コードのまま loss だけ差し替えてこの恩恵を比較できます。

時刻サンプリング分布のアブレーションも重要です。フローマッチング・拡散の研究では、学習時に t を一様に引くか、特定の領域に寄せるかで最終品質が変わることが繰り返し報告されています。たとえば中間付近の t を厚く引く（logit-normal 等）と、速度場が暴れやすい難所を重点的に学習でき、生成の安定性が向上するという結果があります。逆に偏りを付けないと、難所の学習が不足して軌跡がぶれる、という失敗例が知られています。これらの調整は loss メソッドの内部で完結すべきで、外に漏らすと再現性が崩れます。

Diffusion Policy (2023) のアブレーションでは、行動の表現（絶対 vs 相対）や予測ホライズン（何ステップ先まで一度に生成するか）を変えると成功率が変わることが示されています。これらの選択は loss 計算と密結合しており、モジュール内に閉じて一貫管理することで、外側のループに影響を与えずに方式ごとの最適設定を保てます。

マルチタスクの loss 重み付けについては、Kendall et al. (2018) が、手動でグリッドサーチした固定重みに匹敵かそれを上回る性能を、不確実性ベースの自動重み付けで達成したと報告しています。PCGrad (2020) は、複数タスクの勾配干渉が強い設定で、素朴な和に比べて成功率や精度を改善したと報告しています。これらは「各モジュールが返すスカラ loss のスケールが揃っていないと、片方が支配的になって他方が学習されない」という実務上の問題への直接の処方箋です。

## メリット・トレードオフ・限界
メリット。学習ループから方式ごとの分岐が消え、コードが単純で読みやすくなります。速度場や予測ノイズと軌跡の取り違えのような、無言で学習が壊れるバグを構造的に防ぎます。新方式の追加時、その方式の loss を自分の内部に書くだけで済み、外側を壊しません。時刻サンプリングなどの方式固有の繊細な調整を一箇所に集約でき、再現性が高まります。

トレードオフと限界。loss がモジュール内に隠れるため、外から loss の内訳（速度場誤差の項別、ステップごとの誤差分布など）を覗いて分析・可視化しにくくなります。デバッグや論文用の可視化のためには、内部状態を返す別の窓口（診断用メソッドやフック）を意図的に用意する必要があります。また「方式固有は内、共通は外」という二層構造を理解していないと、どの loss をどこで直すべきか迷いやすく、二重に loss をかけてしまう（内側と外側で同じ軌跡誤差を重複計算する）といったミスも起こり得ます。さらに、複数モジュールの loss を最終的に足し合わせて学習する場合、各モジュールが返すスカラ loss のスケール（大きさの桁）が揃っていないと、片方の loss が支配的になって他方が学習されない問題が生じます。スケールの重み付けをどこで行うか（各モジュール内か、足し合わせる外側か）は、カプセル化の境界をどこに引くかという設計判断として残ります。

## 発展トピック・研究の最前線
近年は、複数の生成タスク（軌跡生成、占有予測、将来フレーム予測など）を1つのモデルで同時に学習するマルチタスク設定が増え、各タスクモジュールが自分の loss を返し、外側で重み付き和を取る構成が一般的になっています。ここで loss の自動重み付け（Kendall et al. の不確実性重み付け）や、勾配の干渉を抑える手法（PCGrad、GradNorm など）が研究されています。これらはモジュール内 loss カプセル化を前提に、その外側でどう束ねるかを扱う発展形です。また、大規模基盤モデルでは loss の内部に強化学習的な報酬や人間の選好（preference）に基づく目的（RLHF/DPO 系）も組み込まれ始め、カプセル化の対象がますます複雑化しています。loss の中身が「単純な MSE」から「報酬最大化」へ拡張されても、外から見た契約（スカラを返す）が変わらない限り、学習ループは無改変で済む、という設計の堅牢性が改めて評価されています。
