# L2D LeRobotデータセットパーサ

## ひとことで言うと

実走行データセットを読み込み、運転モデルがそのまま受け取れる形(テンソル)に変換する部品をデータセットパーサと呼びます。本ノートでは大規模マルチカメラ運転データセット「L2D」(Learning to Drive、yaak-ai 公開、7 カメラ・10Hz)を題材に、動画と数値ログをどうやってモデルの入力(画像群と過去の運動)と正解(これからの軌跡)に切り分けるかを、パーサ設計の一般論として解説します。地味ですが、ここを誤ると下流の学習がすべて狂う土台の工程です。

## 直感的な理解

機械学習モデルを訓練するには「お手本のデータ」が大量に必要です。しかし生のデータセットは、人間や標準ツールが扱いやすい形(MP4 動画ファイル、CSV ライクな数値テーブル、タイムスタンプ付きログ)で保存されており、ニューラルネットがそのまま食べられる形ではありません。ニューラルネットが欲しいのは、決まった形・決まった正規化を施した数値テンソルだけです。

パーサの仕事は、この「人間に優しい保存形式」と「モデルに優しい入力形式」の間の翻訳です。料理にたとえると、データセットは市場で買ってきた生の食材(皮も骨もついている)で、パーサはそれを下ごしらえして、調理(=学習)にすぐ使える状態の切り身にする工程に当たります。下ごしらえを誤る(サイズが違う、正規化がずれる、時刻が揃っていない)と、その先の調理がすべて台無しになるため、地味ですが極めて重要な工程です。

## 基礎: 前提となる概念

Dataset と DataLoader。PyTorch では、データ供給を `Dataset` と `DataLoader` の 2 段で設計します。`Dataset` は「i 番目のサンプルを 1 つ返す」責務だけを持ち、`__len__`(総数)と `__getitem__(i)`(i 番目)の 2 メソッドを実装します。`DataLoader` はその上に乗り、複数サンプルをまとめてバッチ化し、シャッフルし、複数プロセス(`num_workers`)で並列に先読みします。パーサが作るのは通常この `Dataset` です。学習の GPU を遊ばせないために、データ供給が GPU 計算より速いことが理想で、ここがボトルネックになると高価な GPU が待ち時間で空回りします。

テンソルの形(shape)。テンソルは多次元配列で、各次元のサイズを並べたものが形です。たとえば画像 1 枚は `(3, 256, 256)`(RGB の 3 チャンネル × 縦 256 × 横 256 ピクセル)。これを複数カメラ束ねると `(num_views, 3, 256, 256)` になります。形がモデルの期待と 1 つでも違うと実行時エラーになるため、パーサは出力テンソルの形を厳密に設計します。

正規化(normalization)。画像の各ピクセルは通常 0〜255 の整数ですが、ニューラルネットには平均 0・分散 1 付近にそろえた値を与えると学習が安定します。ImageNet(画像分類の大規模データセット)で事前学習された画像エンコーダ(backbone、特徴抽出の土台ネットワーク)を使う場合は、その事前学習時とまったく同じ平均・標準偏差で正規化しないと性能が出ません。だからパーサは backbone の設定から正規化パラメータを取得して合わせます。これを間違えると、見た目には動くのに精度だけ静かに落ちる、発見の難しいバグになります。

BEV(Bird's Eye View、鳥瞰図)。真上から見下ろした地図のような視点のことです。複数の前方・側方カメラの画像を統合して BEV を作るのが近年の自動運転の主流で、L2D のようなデータセットは BEV マップビューを 1 つのカメラ的チャンネルとして含むことがあります。

LeRobot フォーマット。HuggingFace が提唱したロボット学習向けの統一データ形式です。カメラ映像は MP4 として、ロボット/車両の状態(state)と行動(action)は数値テーブル(parquet、列指向で高速に読める形式)として格納され、`LeRobotDataset` という Python API で透過的にアクセスできます。自前で MP4 をデコードし時刻を揃える手間を省ける標準化の恩恵があります。

## 仕組みを詳しく

典型的なマルチカメラ運転パーサは、次のような責務分割で実装されます。

- データセット本体(PyTorch `Dataset` を継承): サンプルの列挙と取り出し
- egomotion 導出: 車両状態から運動信号を計算([egomotion信号の導出（速度・加速度・ヨーレート・曲率）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0813_egomotion-signal-derivation))
- カメラ定義: カメラ名・読み込み・キャリブレーション(投影行列)
- スモークテスト: 1 サンプルを実際にモデルに通して形が合うか検証

`dataset[i]` を呼ぶと辞書が返り、中身は概ね次のテンソル群です(ホライズン 64 ステップ、7 カメラ、4 運動信号、2 出力信号の例)。

```
visual_tiles      (7, 3, 256, 256)  7ビュー × RGB3ch × 縦256 × 横256
egomotion_history (256,)            過去64ステップ × 4信号 を平坦化
visual_history    (896,)            視覚履歴(初期はプレースホルダになりがち)
trajectory_target (128,)            未来64ステップ × 2信号(正解)
episode_index     int
frame_index       int
```

処理の流れをステップで分解します。

ステップ 1: 有効サンプルの列挙(コンストラクタ)。各エピソード(連続した 1 回の走行記録)の中で、「過去 H フレームと未来 H フレームの両方が同じエピソード内に存在する」フレームだけが有効サンプルです。たとえば過去 64・未来 64・現在 1 なら最低 129 フレーム必要。エピソード境界をまたいで過去や未来を取ると、別の走行の運動が混ざって正解が壊れるため、境界内に収まるフレームだけを `(エピソード番号, ローカル行番号)` の組として全列挙します。これにより `__getitem__` はメタデータ走査をせず O(1)(一定時間)で目的フレームへ飛べます。長さの違うエピソードが多数あるデータセットで、毎回境界判定を走らせると遅くなるため、列挙はコンストラクタで一度だけ行うのが定石です。

ステップ 2: 画像変換の準備。画像エンコーダ(例: Swin Transformer V2 の tiny、階層的な Vision Transformer の小型版)のデータ設定を取り出し、入力サイズ(256×256)と ImageNet 正規化の平均・標準偏差を取得します。backbone の設定から取るのは、後段エンコーダが期待する前処理と必ず一致させるためです。

ステップ 3: `__getitem__` 本体。
1. 対象フレームの行番号と所属エピソードを引く。
2. 車両状態(数値ベクトルの時系列)をエピソード分まとめて読む。ここが性能上の最重要ポイント。
3. その車両状態から egomotion を導出して `egomotion_history` と `trajectory_target` を作る。
4. 現フレームのカメラ画像を取得(ここで初めて動画デコード)。各カメラを 256×256 にリサイズ、ImageNet 正規化し、7 枚を `torch.stack` で `(7,3,256,256)` に束ねる。
5. `visual_history` は実装段階によりゼロ埋めのプレースホルダのこともある。

ステップ 2 と 3 の性能上の工夫(最重要)。LeRobot 系 API で 1 フレームを丸ごと取ると、そのフレームに紐づく全カメラの MP4 を毎回デコードしてしまい、運動信号(数値ベクトル)を読むだけなのに数十秒/サンプルかかることが報告されています(典型的には 7 動画デコードで 30〜40 秒オーダー)。運動信号に動画デコードは不要なので、`select_columns(["observation.state.vehicle"])` のように数値列だけを直接読むことで、この巨大なオーバーヘッドを回避します。動画デコードは「今この瞬間の画像」を取る最後の 1 回だけに限定するのが定石です。これは「不要なデータを触らない」という、データパイプライン高速化の基本原則の具体例で、桁違い(数十倍)の高速化につながります。

カメラ名は固定リストで定義します(例: 前方左・左前・右前・左後・後方・右後・BEV マップの 7 つ)。カメラ数や配置をモデル側にハードコードせず、データセット側のパーサが吸収する設計にすると、モデルがデータセット非依存(dataset-agnostic)になります。各カメラの位置姿勢は extrinsics(外部パラメータ、カメラが車のどこにどの向きで付いているかを表す回転と並進)として渡し、モデルはそれを使って視点を統合します。内部パラメータ intrinsics(焦点距離や光学中心、ピクセルと光線の対応)と合わせて、画像のピクセルを 3 次元空間へ正しく投影するのに使います。

## 手法の系譜と主要論文

カメラ映像から運転を学ぶアプローチの系譜を追います。

End to End Learning for Self-Driving Cars(Bojarski ら、NVIDIA、2016、arXiv:1604.07316)。1 台の前方カメラ画像から直接ステアリング角を回帰する CNN(PilotNet)を提案。動機は「人手で特徴量を設計せず、生映像から運転を end-to-end で学べば設計コストが下がる」。効果として高速道路や一般道での車線維持を実車で実証しました。限界は単カメラ・単出力(ステアのみ)で、複雑な市街地や周囲全方位の把握に弱いこと。L2D のような多カメラ・長期軌跡データはこの限界を埋める方向です。

nuScenes(Caesar ら、Motional/nuTonomy、CVPR 2020、arXiv:1903.11027)。6 カメラ・5 レーダ・1 LiDAR を同期し、フルキャリブレーション(intrinsics/extrinsics)と物体軌跡 GT(ground truth、正解アノテーション)を備えた標準データセット。1000 シーン・各 20 秒・キーフレーム 2Hz(センサ生は高頻度)。BEV 知覚や end-to-end 運転研究のデファクト標準になりました。ライセンスは非商用(CC BY-NC-SA)。L2D は同様に多カメラ + キャリブレーションを持つ点で系譜が近いが、スケールとライセンス条件が異なります。

Lift-Splat-Shoot(Philion & Fidler, NVIDIA/Toronto, ECCV 2020, arXiv:2008.05711)。複数カメラの画像特徴を、各ピクセルの深度分布を予測して 3 次元へ持ち上げ(lift)、BEV 平面へ散らし(splat)、統合する方式を提案。多カメラを extrinsics/intrinsics で幾何的に融合する後続研究(BEVFormer, BEVFusion 等)の土台になりました。これがあるからこそ、パーサがキャリブレーションを正しく渡す意味が生まれます。

LeRobot(Cadene ら、HuggingFace、2024)。ロボット学習データを統一フォーマットで扱う枠組みで、模倣学習・強化学習のデータと方策を再現性高く共有することを狙います。L2D はこの上に構築され、運転を「ロボット操作タスクの一種」として扱う流れに位置づきます。

L2D(yaak-ai、HuggingFace 2024、ブログ "LeRobot goes to driving school")。報告によると大規模(エピソード数で数万〜十万規模)・7 カメラ・10FPS の実走行データで、運転を模倣学習タスクとして扱うために LeRobot フォーマットで公開されています。10FPS は多くの運転モデルが想定する自車運動の頻度と一致し、フレーム間引きが不要という実務上の利点があります。

## 論文の実験結果(定量データ)

データセットそのものの「実験結果」は性能スコアというより規模と多様性で評価されますが、その上で動いた手法の数値を通して価値が分かります。

nuScenes のインパクト。公開後、BEV セグメンテーションや 3D 物体検出のベンチマーク(nuScenes Detection Score, NDS)が標準指標となり、Lift-Splat-Shoot(ECCV 2020)、BEVFormer(Li ら, ECCV 2022, arXiv:2203.17270)、BEVFusion(Liu ら, ICRA 2023)など主要 BEV 手法がこの上で競われました。たとえば BEVFormer は nuScenes の NDS で約 0.56(LiDAR を使わずカメラのみで)を達成し、当時のカメラ専用手法の到達点を示しました。NDS は検出精度(mAP、平均適合率)に位置・速度・属性・向きの誤差を統合した 0〜1 の合成指標で、高いほど良い。

end-to-end 運転の到達点。UniAD(CVPR 2023)は nuScenes 計画タスクで平均 L2 誤差を約 1.0m 前後、衝突率を先行手法比で大幅に低減し、知覚から計画までの統合学習が有効であることを示しました(詳細は [軌跡損失の時間減衰重み付け](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0568_temporal-decay-weighting))。これらの結果は「多カメラ + キャリブレーション + 軌跡 GT を備えた高品質データセット」が研究を駆動した好例で、L2D が目指す方向性の根拠になっています。

データ規模の効果。模倣学習一般に、データ量とシーン多様性が性能を大きく左右することが知られます。報告によれば、走行ログを数千時間規模に増やすとレアケース(割り込み、雨天、夜間)のカバレッジが上がり、クローズドループでの介入頻度が下がる傾向があります。L2D の大規模性はこの「多様性によるロングテール(まれだが重要な事象の裾)対応」を狙ったものです。ただしデータ量を増やしても性能向上は対数的に鈍化する(スケーリング則)ことも知られ、量だけでなく多様性とラベル品質が効きます。

## メリット・トレードオフ・限界

メリット:
- 大規模実データを学習に使えるため、合成データやシミュレータでは得にくいリアルな分布(歩行者の挙動、路面状況の多様性)を学べる。
- 10FPS が運転モデルの想定運動頻度と一致し、ダウンサンプリング不要。
- 統一フォーマット(LeRobot)に乗ることで、データ読み込みが堅牢になり、数値列だけ先読みするなどの最適化で動画デコードのボトルネックを回避できる。

トレードオフ・限界:
- キャリブレーション(extrinsics/intrinsics)が未整備またはプレースホルダだと、BEV 統合のような視点を幾何的に重ねるモードが機能しない。意味のある俯瞰投影には実キャリブレーションが必須。
- 視覚履歴(過去フレームの画像特徴)を使う設計では、その読み込みと特徴抽出のコストが大きく、初期実装ではゼロ埋めのプレースホルダにとどまることが多い。
- 公開データセットはライセンス・配布条件・地域バイアス(撮影された都市や国・交通規則に偏る)を抱える。商用利用や他地域(左側通行/右側通行の違いなど)への汎化には注意が要る。
- 模倣学習はデータに現れない状況(専門家が起こさない異常状態)に弱く、データセットを増やすだけでは長い裾を完全には埋められない。これは行動クローニングの分布シフト問題そのもので([軌跡損失の時間減衰重み付け](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0568_temporal-decay-weighting) の DAgger 参照)、データ量だけでは原理的に解けない未解決課題です。

## 発展トピック・研究の最前線

データバージョニングと再現性。大規模データは更新・分割・サブセット抽出が頻繁に起きるため、データのバージョン管理(LakeFS、DVC など)と、シャード化(WebDataset 形式で多数の小ファイルに固めて並列読み込みを高速化)が実務の鍵になっています。どの版のデータでどのモデルを学習したかを追跡できないと、実験の再現や性能劣化の原因究明ができません。

オンライン/オフラインの前処理分離。動画デコードや画像特徴抽出は重いので、学習のたびに行わず、事前に特徴量へ変換して保存(pre-extraction、前段抽出)し、学習時は軽い特徴テンソルだけ読む構成が広く使われます。これはパーサの責務を「読み込み」と「特徴抽出」に分けるアーキテクチャ判断で、GPU を計算に専念させデータ供給律速を解消する標準手段です。

データセット非依存モデル。カメラ数・配置・座標系をデータ側のパーサで吸収し、モデルには統一インターフェース(画像群 + extrinsics + 過去運動)だけを渡す設計が、複数データセットを横断学習する研究の前提になっています。これにより L2D・nuScenes・自社データを混ぜて訓練でき、単一データセットの地域バイアスを薄められます。近年の基盤モデル的アプローチ(多データセット同時事前学習)はこの設計を強く要求します。
