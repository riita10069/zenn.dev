# DiLu

## ひとことで言うと
DiLu(ディルー)は、大規模言語モデル(LLM。大量の文章で学習し、言葉で論理を述べるのが得意な AI)を運転手の脳に見立て、「記憶(Memory)」「推論(Reasoning)」「反省(Reflection)」という3つの働きを組み合わせて運転させるフレームワークです。最大の特徴は、データから統計パターンを学ぶ(=データ駆動)のではなく、人間のような常識と論理(=知識駆動, knowledge-driven)で運転を判断し、しかも運転を重ねるたびに経験を貯めて賢くなっていく点です。

## 直感的な理解
人間のドライバーを思い浮かべてください。免許を取りたての人でも「歩行者が飛び出しそうなら速度を落とす」という常識を最初から持っています。これは何百万キロも運転データを見たからではなく、世界についての一般的な知識から導いた判断です。そして運転を重ねるうちに「あの交差点は見通しが悪い」といった経験を蓄え、ヒヤリとした失敗からは「次はこうしよう」と反省して改善します。

従来の自動運転 AI の多くは、この人間的な学び方とは正反対でした。膨大な走行データを見せて統計パターンを丸暗記させる方式(データ駆動)で、データにない珍しい場面に弱く、いったん学習が終わると運転中に賢くなる仕組みもありませんでした。たとえば強化学習(報酬を頼りに試行錯誤で方策を学ぶ手法)は、ある交通密度・車線数の環境で何百万回も試行して最適な振る舞いを覚えますが、その環境設定を少し変えただけで性能が崩れます。DiLu は「常識を持ち、経験を貯め、失敗から反省する」という人間の運転の学び方そのものを LLM で再現しようとした試みで、論文タイトルが "Knowledge-Driven Approach"(知識駆動アプローチ)を掲げているのはこのためです。

## 基礎: 前提となる概念
DiLu を理解するには、データ駆動と知識駆動の対比、そして LLM エージェントの周辺技術を押さえる必要があります。

データ駆動(data-driven)の限界。大量の運転データから統計的パターンを学ぶ方式には3つの弱点があります。第一に汎化の弱さ。学習データにないレアケース(道路に落ちた家具を避ける、特殊な工事規制など)では、見たことがないので適切に振る舞えません。学習データの分布から外れた入力(out-of-distribution, OOD)に脆いのです。第二に因果の理解不足。統計的な相関は覚えても「なぜそうすべきか」という因果や常識は持たず、人間には当たり前の判断を外すことがあります。第三に経験を積み重ねられないこと。学習が終わると重みは固定され、運転中に得た新しい経験を即座に取り込んで賢くなる仕組みがありません。

few-shot プロンプティングとは。LLM に「数件のお手本(例)」を入力文に添えて見せると、それを参考に類似の問題を解けるようになる性質です。追加の重み更新なしで、入力に例を入れるだけで振る舞いを誘導できます。in-context learning(文脈内学習)とも呼ばれ、これが DiLu の「重みを更新せずに賢くなる」仕組みの中核です。

RAG(Retrieval-Augmented Generation, 検索拡張生成)とは。LLM に答えさせる前に、外部のデータベースから関連情報を検索して取り出し、その情報を入力に足してから答えさせる枠組みです(Lewis et al., NeurIPS 2020)。LLM 自身の固定知識だけに頼らず、必要なときに外部知識を引いてくることで、最新性・正確性・専門性を補えます。検索は通常、文章を数値ベクトル(埋め込み, embedding)に変換し、ベクトル同士の近さ(コサイン類似度)で「意味的に似たもの」を探す方式です。DiLu の記憶モジュールはこの運転版です。

自己反省(self-reflection)とは。LLM エージェントに自分の行動の結果を振り返らせ、失敗の原因を言葉で分析させて次に活かす技術です。Reflexion(Shinn et al., NeurIPS 2023)が代表で、強化学習のように重みを更新せずとも、言語による反省を記憶に残すことでエージェントが改善することを示しました。DiLu の反省モジュールはこの発想を運転に適用したものです。記憶を持つ LLM エージェントという発想自体は、町をシミュレートする Generative Agents(Park et al., UIST 2023)など 2023 年に一気に広がった「LLM エージェント」研究の潮流の一部でもあります。

## 仕組みを詳しく
DiLu は LLM を中心に、3つのモジュールを循環させて運転します。

推論モジュール(Reasoning Module)。現在のシーン(他車の位置・速度、車線、自車状態)を言葉で記述し、それを LLM に渡して「次にどんな行動を取るべきか」を Chain-of-Thought(答えの前に途中の推論ステップを書かせる手法)で考えさせます。出力は離散的な高レベル行動(例: 加速する、減速する、左に車線変更する、右に車線変更する、現状維持する)です。ただし LLM 単独で考えさせるのではなく、次の記憶モジュールから取り出したお手本を一緒に渡すのが要点です。さらに出力した行動が安全制約を満たすかを軽くチェックし、危険なら却下するセーフティ層を挟みます。

記憶モジュール(Memory Module)。過去の運転シーンと、そのとき取った良い判断のペアを貯めておく外部メモリ(ベクトルデータベース)です。新しいシーンに出会うと、現在のシーンを数値ベクトル(埋め込み)に変換し、メモリ内で意味的に近い過去シーンを検索して数件(典型的には数件規模)取り出し、推論モジュールのプロンプトに few-shot のお手本として添えます。これが前述の RAG + few-shot の運転版です。LLM は「過去に似た場面でこう判断してうまくいった」という実例を見ながら判断するので、出力が安定し、メモリに良い経験が増えるほど判断が洗練されます。重要なのは、これがニューラルネットの重みを更新しない学習であること。経験は重みでなくメモリに蓄積されるので、走りながらリアルタイムに「賢くなる」ことができます。

反省モジュール(Reflection Module)。取った行動の結果を振り返り、良し悪しを評価する仕組みです。安全でない判断(例: 危険な車間で車線変更した、衝突した)を検出すると、LLM 自身に「なぜそれが危なかったか、どう判断すべきだったか」を言葉で分析させ、その修正された安全な判断を新たな経験としてメモリに書き戻します。良い判断もお手本としてメモリに追加します。これにより、同じ間違いを繰り返さない自己改善ループが回ります。

一周分の処理を例で追うと:
```
ステップ1 (記憶検索): 現在「右車線に減速車両、後方に接近車」
  → シーンを埋め込みに変換 → メモリから意味的に近い過去シーンを 5 件取り出す
ステップ2 (推論): 取り出した実例を few-shot で見ながら LLM が CoT で
  「後方車が速いので車線変更は危険。減速して様子を見る」と判断 → 行動 = 減速
ステップ3 (実行): シミュレータ上で減速を実行
ステップ4 (反省): 結果が安全 → 良い経験としてメモリに追加
              もし危険だった → LLM に原因を分析させ、修正版をメモリに追加
```
このサイクルにより、DiLu はデータで一括再学習しなくても、運転経験を積みながら判断を徐々に洗練させます。データ駆動が「学習フェーズと推論フェーズが分離」しているのに対し、DiLu は「運転しながら学ぶ」のが質的な違いです。さらに、あるシミュレータで貯めたメモリを別の環境設定に持ち込んで使う(memory transfer)ことで、知識の移植も可能であることが示されています。

## 手法の系譜と主要論文
DiLu は、LLM エージェント研究の2つの潮流を運転に合流させた位置にあります。

一つは RAG(Lewis, Perez, Piktus et al., NeurIPS 2020, arXiv:2005.11401)に始まる「外部記憶からの検索で LLM を補強する」流れ。もう一つは Reflexion(Shinn, Cassano, Gopinath, Narasimhan, Yao, NeurIPS 2023, arXiv:2303.11366)に代表される「言語による自己反省でエージェントを改善する」流れです。Generative Agents(Park et al., UIST 2023, arXiv:2304.03442)が「記憶・検索・反省を持つ LLM エージェント」を社会シミュレーションで示したのとも同時代で、DiLu はこの2つを「記憶検索による経験の参照」と「反省による経験の修正・蓄積」として運転判断に組み込みました。

DiLu(Wen, Fu, Lin, Wang, Shi, Liu, Yang, Wang, ICLR 2024, arXiv:2309.16292, "DiLu: A Knowledge-Driven Approach to Autonomous Driving with Large Language Models")の新規性は、これらを運転という連続的意思決定タスクに統合し、「データ駆動の強化学習(報酬で試行錯誤させる学習)よりも、知識駆動 LLM のほうが汎化に強く、サンプル効率も高い」ことを実証した点にあります。

同時期の運転 LLM 研究の中では、座標を直接出す GPT-Driver(Mao et al., 2023)や、LLM の判断を MPC に橋渡しする [LanguageMPC](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0540_languagempc) が「単発の判断や制御」に焦点を当てたのに対し、DiLu は「経験を蓄積して継続的に改善する」点が際立っています。ツール使用型エージェント([Agent-Driver](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0526_agent-driver))とも記憶活用の発想で近縁です。

## 論文の実験結果(定量データ)
DiLu は主に運転シミュレータ(highway-env。複数車線での車線変更・合流・追従を扱う離散行動環境)で評価されました。中心的な指標は次の通りです。

- 成功率(success rate / 平均走行ステップ数): 衝突せず指定ステップ数を走り切れた割合、または衝突までの平均走行長。これが高いほど安全に運転を継続できることを意味します。
- 必要経験数(サンプル効率): どれだけの試行・どれだけのメモリ件数で高性能に達したか。少ない経験で性能が出るほど効率的。
- 汎化性能: 学習に使った環境設定と異なる条件(車線数・交通密度などを変えた環境)でどれだけ成功率を保てるか。これが知識駆動の真価を測る最重要指標です。

報告によれば、DiLu は強化学習(RL)ベースの手法と比べて、(1) はるかに少ない経験(数十件規模のメモリ)で高い成功率に到達し、(2) 学習環境と異なる環境への汎化で RL 系を大きく上回ったとされます。論文では、わずか数十件の経験を貯めた DiLu が、桁違いに多い学習量を要する RL ベースラインに匹敵または凌駕する成功率を、しかも環境設定を変えても保つ、という結果が示されています。RL は学習した環境に過適合しやすく、環境が変わると性能が崩れがちなのに対し、DiLu は常識ベースの判断と経験メモリのおかげで、未知設定でも判断が崩れにくいのが効きました。さらに、経験メモリのサイズを増やすほど成功率が伸びる「継続的改善」のカーブが示され、使う基盤 LLM を強いもの(GPT-3.5 → GPT-4)に替えるほど性能が上がることも報告されています。

アブレーション(構成要素を抜く分析)では、(1) 記憶モジュールを外して LLM 単独で判断させると成功率が大きく落ちること(=過去の実例を参照する効果が大きい)、(2) 反省モジュールを外すと、危険な判断を修正できないため失敗が繰り返され改善が頭打ちになること、が示されています。これは記憶(経験の参照)と反省(経験の修正)の両輪が揃って初めて継続的改善が成立することを意味します。なお、絶対数値は環境設定や使用した基盤 LLM に依存するため、「RL 比で汎化が大きく改善」「メモリ増加で性能が伸びる」という傾向として理解するのが妥当です。

## メリット・トレードオフ・限界
メリット:
- 常識と論理で判断する知識駆動なので、レアケース・未知環境への汎化に強い。
- 経験メモリが増えるほど賢くなる継続的な自己改善ができる(重み再学習が不要)。
- 反省モジュールで失敗を次回に活かせる(同じ間違いを繰り返しにくい)。
- 強化学習より圧倒的に少ない経験で高性能に到達しやすい(サンプル効率が高い)。
- 判断が言葉で説明されるため解釈性・デバッグ性が高い。メモリの中身を見れば「何を根拠に判断したか」が分かる。

トレードオフ・限界(研究上の未解決課題):
- シミュレータ上の高レベル離散行動が中心で、生のセンサ入力や連続的な低レベル制御は扱っていない。実車のフルスタックへの橋渡しは未解決。
- LLM を毎判断で呼ぶため遅く、計算・API コストが高い。車載リアルタイム(数十 ms 周期)には程遠い。
- 記憶と反省の質に性能が依存する。悪い経験を貯めると判断が悪化しうる(garbage-in の問題)。反省の正しさを誰が保証するかは未解決。
- メモリが大きくなると検索コストが増え、無関係な経験を引いて判断を乱すリスクもある。
- 幻覚(hallucination)による誤判断が安全に直結する。

## 発展トピック・研究の最前線
DiLu が示した「記憶・推論・反省の循環で知識駆動運転を行う」枠組みは、LLM エージェントを運転に使う研究の基本テンプレートの一つになりました。発展方向としては、まず生センサや 3D 空間理解との接続が挙げられます。DiLu はシーンをテキスト化した抽象表現を入力としますが、これを VLM(画像も言葉も扱えるモデル)による 3D 認識([OmniDrive](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0542_omnidrive) のような 3D クエリ表現)と組み合わせれば、テキスト化で失われる細かな空間情報を保てます。

第二に、離散行動から連続制御への橋渡し。DiLu の高レベル判断を [LanguageMPC](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0540_languagempc) のように MPC のパラメータへ翻訳すれば、滑らかで物理的に妥当な制御まで一貫させられます。第三に、記憶・反省の質の保証。悪い経験の混入を防ぐためのフィルタリングや、複数エージェントによる相互検証、人間のフィードバックによる経験の選別などが研究の焦点です。メモリの肥大化に対しては、重要な経験だけを残す要約・忘却の仕組み(memory consolidation)も検討されています。実車適用にはリアルタイム性・安全保証・幻覚対策という共通の壁が残っており、これらは LLM 運転全般の最前線の課題です。
