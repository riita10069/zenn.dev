# ShuffleNet

## ひとことで言うと
ShuffleNet は「スマホよりさらに非力な機械でも動くように、極限まで計算量を削った画像認識ニューラルネットワーク」です。計算を減らすために「グループ畳み込み」という分業のしくみを使うのですが、それだけだと特徴の流れが分断されてしまうので、「チャネルシャッフル」という、特徴の層を混ぜ直す操作で情報を行き来させるのが最大の特徴です。続く V2 では「理論計算量(FLOPs)ではなく実機の速度で設計せよ」という、軽量モデル研究全体の考え方を変える指針を打ち出しました。

## 直感的な理解
[MobileNet (V1/V2/V3)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0397_mobilenet) のような軽量モデルでは、計算を減らす鍵は 1×1 畳み込み(pointwise、チャネル同士を混ぜる操作)でした。ところがモデルを極端に小さくしていくと、皮肉にもこの 1×1 畳み込みが計算量の大半(MobileNetV1 では約 94%)を占めるようになります。1×1 畳み込みは全チャネルを全チャネルへ繋ぐので、チャネル数の 2 乗で計算が膨らむからです。depthwise 畳み込みで空間方向は十分安くなったのに、チャネル混合のコストが残った、という状況です。

そこで「1×1 畳み込みもグループ化して安くしよう」と考えるのが自然です。グループ化とは、チャネルをいくつかの班に分け、班の中だけで計算する分業です。班を 10 個に分ければ計算は約 1/10 になります。しかし落とし穴があります。グループ畳み込みを重ねると、各班が自分のチャネルだけを延々と処理し続け、班どうしで情報が一切交わらなくなります。「右半分の特徴と左半分の特徴が永遠に会話しない」状態で、表現力が大きく落ちます。ShuffleNet はこの分断を、ほぼゼロコストの「チャネルシャッフル」という並べ替えで解消しました。班をまたいでカードをシャッフルし直すように、特徴の層を混ぜ替えるのです。

## 基礎: 前提となる概念
用語を平易にします。

チャネルとは、特徴マップ(画像から取り出した中間データ)の「層の枚数」です。中間層では数十〜数百枚あります。

グループ畳み込み(group convolution)とは、チャネルを g 個のグループに分け、グループの中だけで畳み込みをする方式です。例えば 100 チャネルを 10 グループ(各 10 チャネル)に分けると、各グループは自分の 10 チャネルしか見ないので、計算量がおよそ 1/g(この例で 1/10)に減ります。もともと AlexNet(2012)が 2 枚の GPU にモデルを分割してメモリを節約するために使った古い技法で、ResNeXt(Xie et al., CVPR 2017, arXiv:1611.05431)が表現力強化の手段(cardinality という新しい次元)として再評価しました。

depthwise 畳み込みとは、グループ畳み込みの極端な場合、すなわちグループ数をチャネル数と同じにして「1 チャネル = 1 グループ」とした畳み込みです。MobileNet の中核部品です。

FLOPs と実機速度の違いは、ShuffleNet を理解する最重要の前提です。FLOPs は理論上の掛け算回数(紙の上の計算量)です。一方、実機の速度(レイテンシ)は計算量だけでなく、メモリの読み書き回数や GPU の並列度にも強く左右されます。この「メモリアクセスのコスト」を MAC(Memory Access Cost)と呼びます。FLOPs を下げても MAC が増えれば、実機はかえって遅くなりえます。V2 はこの食い違いを正面から扱いました。

## 仕組みを詳しく

### チャネルシャッフル
具体例で説明します。チャネルが 9 枚あり、3 グループ(各 3 枚)に分かれているとします。番号で書くと:

```
グループ分け後:  [1 2 3] [4 5 6] [7 8 9]
```

グループ畳み込みのあと、これを「各グループから 1 枚ずつ取って並べ替える」と:

```
シャッフル後:    [1 4 7] [2 5 8] [3 6 9]
```

こうすると、次のグループ畳み込みでは、もとは別グループだったチャネルが同じグループに混ざります。これを繰り返すと、グループ畳み込みの安さを保ったまま、全チャネルが時間をかけて互いに情報をやり取りできるようになります。

実装は、チャネルを (グループ数 g, グループ内チャネル数 n) という形に reshape(形を変える)し、軸を入れ替え(transpose)、また元の 1 次元に戻すだけです。新しい重みパラメータは不要で、追加の掛け算もゼロ、しかも微分可能なので学習をそのまま通せる、という極めて安い操作です。

```
[N, g*n, H, W] → reshape [N, g, n, H, W] → transpose [N, n, g, H, W] → reshape [N, g*n, H, W]
```

(N=バッチ数、H/W=縦横、g=グループ数、n=グループ内チャネル数)

### ShuffleNet V1 ユニット
ShuffleNet V1 の 1 ブロックは「1×1 グループ畳み込み → チャネルシャッフル → 3×3 depthwise 畳み込み → 1×1 グループ畳み込み」で構成され、残差接続(入力をそのまま出力へ足す近道)で繋がれます。MobileNet の depthwise separable を、1×1 部分までグループ化してさらに安くしたもの、と理解できます。ダウンサンプリング(解像度を半分に落とす)時は、近道側に 3×3 の平均プーリングを置き、加算ではなく結合(concat)でチャネルを倍増させます(解像度が半分になる代わりにチャネルを増やすことで情報量を保つ常套手段)。

### V2 の核心: FLOPs ではなく実速度で設計する
ここが ShuffleNet で最も重要な貢献です。V2 の論文(Ma et al.)は「FLOPs が小さくても、実際の速度が速いとは限らない」と指摘し、実機計測(GPU と ARM CPU の両方)から 4 つの設計指針(Guideline、G)を導きました。

G1: 入力チャネルと出力チャネルが等しいとき、メモリアクセスが最小になる。FLOPs を固定したとき、入出力チャネル比が 1:1 から離れるほど MAC が増えて遅くなる。したがって極端な「太い→細い」変換は避ける。

G2: グループ畳み込みを増やしすぎるとメモリアクセスが増える。V1 の売りだったグループ化が、実は遅さの原因になりうる。グループ数 g を上げると FLOPs は下がるが MAC が増える。

G3: ネットワークを枝分かれ(fragmentation)させすぎると、GPU の並列性が落ちて遅くなる。小さなカーネルや分岐を多用する設計(NAS が好んで生む断片化した構造)は、FLOPs が同じでも実機で遅い。

G4: 要素ごとの演算(elementwise、ReLU や残差加算、テンソル加算、depthwise なども)も無視できないコストがある。FLOPs にはほとんど計上されないが、メモリ帯域を食う。

これらに従い、V2 は Channel Split に基づく新ユニットを採用しました。入力チャネルを 2 つに分割(split)し、片方はそのまま素通り(identity)、もう片方だけを「1×1 → 3×3 depthwise → 1×1」(すべて入出力チャネルが等しい、G1 を満たす設計)で処理し、最後に両者を結合(concat)してチャネルシャッフルします。グループ畳み込みを実質やめ(G2 対応)、分岐を 2 本に抑え(G3 対応)、残差加算を結合に置き換える(G4 対応)ことで、4 指針すべてに沿わせています。

```
ShuffleNetV2 ユニット:
  入力 ─ split ─┬─ そのまま素通り ───────────────┐
               └─ 1x1 → 3x3 dw → 1x1 (入出力ch等しい) ─┴─ concat → channel shuffle → 出力
```

なお Channel Split + concat + shuffle の組み合わせは、半分を素通りさせることで実質的に DenseNet 的な特徴再利用(feature reuse)を生み出しており、残差加算を使わずとも勾配が流れやすくなっています。結果、同じ理論計算量でも V1 より実機で速く、かつ精度も高くなりました。

計算量と精度の目安(ImageNet、入力 224):

```
ShuffleNetV1 (1.0x, g=3)  ≈ 140 M MACs, ImageNet top-1 約 67.4%
ShuffleNetV2 (1.0x)       ≈ 146 M MACs, top-1 約 69.4%
ShuffleNetV2 (1.5x)       ≈ 299 M MACs, top-1 約 72.6%
ShuffleNetV2 (2.0x)       ≈ 591 M MACs, top-1 約 74.9%
(参考) MobileNetV2 (1.0)  ≈ 300 M MACs, top-1 約 72.0%
```

V1 が約 140M MACs と MobileNet よりさらに軽い領域を狙ったこと、V2 が同計算量でより高精度になったことが読み取れます。

## 手法の系譜と主要論文
ShuffleNet(Zhang et al., CVPR 2018, Megvii/Face++, arXiv:1707.01083)は、1×1 畳み込みもグループ化して計算量を削り、その副作用(グループ間の情報分断)をチャネルシャッフルで解消することを提案しました。動機は超低計算量領域(数十〜百 M MACs)で精度を保つこと。効果は同計算量で MobileNetV1 を上回る精度です。トレードオフは、グループ畳み込みとシャッフルが実機のメモリアクセスを増やし理論ほど速くないことがある点で、これを V2 が自己批判的に修正します。

ShuffleNet V2(Ma et al., ECCV 2018, Megvii, arXiv:1807.11164)は「FLOPs だけを指標にするのは誤り、実機速度(latency)で評価せよ」という、軽量モデル設計の考え方そのものを変えた重要論文です。4 つの実用指針を実測から導出し、Channel Split に基づく新ユニットを提案しました。効果は同精度・同 FLOPs で V1 や MobileNetV2 より実速度が速いことです。

系譜としては、グループ畳み込みは AlexNet(2012)に始まり ResNeXt(Xie et al., CVPR 2017, arXiv:1611.05431)が表現力強化の次元として再評価、depthwise separable は MobileNet(2017)が確立しました。ShuffleNet はこの両者を極限まで安くしつつ情報混合を補い、V2 で「実機速度第一」という設計哲学を打ち立てました。この哲学は MobileNetV3、MnasNet、後の MobileOne や RepVGG など「実機レイテンシを直接最適化する」研究へ広く継承されています。

## 論文の実験結果(定量データ)
評価は ImageNet(1000 クラス、約 128 万枚)の top-1 accuracy、計算量(MACs)、そして実機の推論時間(GPU・ARM CPU の両方)で行われました。top-1 は画像 1 枚に対しモデルの第1候補が正解と一致する割合で、軽量モデルの比較ではこの 1〜2 ポイントの差が実用上重要になります。

V1 のアブレーションでは、チャネルシャッフルの有無を比較し、シャッフルを外すとグループ数が多いほど精度が大きく落ちる(情報分断が顕在化し、報告では数ポイント単位で悪化する)こと、シャッフルを入れると同じ計算量で明確に精度が上がることが示されました。これがシャッフルの効果を直接裏づける実験です。グループ数 g を増やすと、同じ FLOPs 予算でより多くのチャネルを使えるため精度が上がる傾向も示されました(ただし V2 で見るように実速度では逆効果になりうる)。

V2 の実験は、4 指針それぞれを実機計測で検証した点が白眉です。G1 では、FLOPs を固定して入出力チャネル比を 1:1、1:2、1:6 と変えると、比が偏るほど実測速度が遅くなることを示しました。G2 では、グループ数 g を 1, 2, 4, 8 と増やすと、FLOPs は同じなのに実測時間が増えることを示しました。G3 では、分岐数(直列・並列の断片)を増やした構造が GPU 上で顕著に遅くなることを示しました。G4 では、ReLU や残差加算(shortcut)を取り除くだけで実速度が数パーセント改善することを示しました。これらは「FLOPs では見えないコストが実在する」ことの定量的証拠です。総合性能では、同じ実機レイテンシのもとで V2 が V1・MobileNetV2・Xception より高い精度を達成したと報告されました。物体検出(COCO)のバックボーンとしても、同レイテンシで MobileNetV2 を上回る mAP を示しています。

## メリット・トレードオフ・限界
メリットは、超低計算量領域(数十〜百 M MACs)でも比較的高い精度を出せること、チャネルシャッフルが追加パラメータ・追加計算ほぼゼロという極めて安い情報混合手段であること、V2 が「実機速度で設計する」という後続研究に広く影響した普遍的指針を提供したことです。

トレードオフと限界もあります。第1に、V1 のグループ畳み込み多用は、理論 FLOPs を下げても実機メモリアクセスを増やし遅くなることがあります(V2 自身が指摘・修正)。第2に、設計が特定ハードウェア(GPU の並列性、ARM CPU のキャッシュ特性など)に強く依存し、別のチップでは最適でない場合があります。V2 の 4 指針も、計測したハードを前提とする経験則であり、たとえば最近の NPU や Tensor Core を前提とすると重みづけが変わりえます。第3に、絶対精度はやはり重いモデルに及ばず、難しいシーンでの取りこぼしリスクは残ります。第4に、Channel Split + concat 中心の設計は実装が素直な反面、残差接続による恒等写像の学習しやすさという ResNet 系の利点を明示的な加算では持たないため(素通りパスで一部代替してはいる)、極端に深くした場合の学習安定性には別途配慮が要ります。

## 発展トピック・研究の最前線
ShuffleNet V2 が打ち立てた「実機速度第一」の哲学は、その後のエッジ向けモデル研究を方向づけました。MobileNetV3 と MnasNet は実機レイテンシを NAS の報酬に直接組み込み、より新しい MobileOne(Apple, CVPR 2023)や RepVGG(Ding et al., CVPR 2021)は「学習時は複雑な多分岐、推論時は単純な一本道へ畳み込みを再パラメータ化(re-parameterization)する」ことで、G3(分岐は遅い)の教訓を活かしつつ表現力を確保しました。学習時と推論時で構造を切り替える、という発想は ShuffleNet V2 の問題意識の延長線上にあります。

チャネルシャッフルのアイデア自体も発展しています。GhostNet は安価な線形変換で冗長な特徴を生成する別アプローチを示し、ESPNet/ESPNetv2 などは異なる接続・混合(階層的な空洞畳み込み)で低レイテンシを追求しました。研究の最前線では、軽量 CNN と軽量自己注意を融合する MobileViT 系や、量子化(整数化)・知識蒸留([知識蒸留損失](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0633_knowledge-distillation-loss))・構造の再パラメータ化を組み合わせて、同じ実機予算でさらに精度を引き出す方向が活発です。ShuffleNet の本質的な教訓、すなわち「紙の上の FLOPs ではなく動かす機械での速度を測れ」は、リアルタイム性が命となるシステム(自動運転やロボティクスなど)でバックボーンを選ぶ際の普遍的な指針として今も生きています。
