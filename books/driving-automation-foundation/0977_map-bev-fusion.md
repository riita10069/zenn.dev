# 地図融合 (Map-BEV Fusion) — ナビ地図を鳥瞰特徴へ重ねる

## ひとことで言うと
カメラ画像から作った「鳥瞰図(上から見下ろした地図のような特徴マップ)」に、ナビ地図(進路や車線が描かれた地図画像)の情報を重ね合わせるモジュールです。これにより「カメラに写っている今の状況」に加えて「この先どこを曲がるか」という地図の知識をモデルに与えられます。融合のやり方には、同じ位置同士を足す残差ゲート方式と、位置をまたいで柔軟に対応づけるクロスアテンション方式があります。

## 直感的な理解
自動運転モデルにカメラ画像だけを入力すると、「目の前に何があるか」は分かりますが「次の交差点を右折すべきかどうか」は分かりません。これはカメラには写らない情報だからです。人間のドライバーがカーナビを見るのと同じで、機械にも進路情報を渡す必要があります。実際、同じ交差点の同じ見え方でも「直進」と「右折」では正解の運転がまったく違います。カメラ入力だけでは原理的にこの曖昧さを解けません。

ここで鍵になるのが BEV(Bird's-Eye-View、鳥瞰図)という考え方です。これは、複数のカメラ画像を「上空から地面を見下ろした1枚の地図」のような特徴表現に変換したものです。なぜわざわざ上から見た形にするのか。運転の判断(どこを走る、どこで曲がる)は地面の平面上で考えるのが自然だからです。前・横・後ろのカメラ情報をバラバラに持つより、1枚の地面マップにまとめたほうが「自車の周囲半径30mに何があるか」を一望できます。

ナビ地図も同じ鳥瞰図の座標系で表現できます。だとすれば「カメラから作った鳥瞰特徴」と「地図から作った鳥瞰特徴」は同じ平面・同じ解像度に揃えられるので、ピクセル単位で重ねられます。この重ね合わせを担うのが、地図を特徴に変換する地図エンコーダと、2つの鳥瞰特徴を融合する融合モジュールです。鍵は「同じ座標系に揃える」ことで、これさえできれば融合は足し算という単純な操作で済みます。

## 基礎: 前提となる概念
特徴マップ(feature map)とは、画像をニューラルネットに通した後の中間表現で、`(C, H, W)` という形を持ちます。C はチャンネル数(各位置が持つ数値の個数)、H と W は縦横の解像度です。元の画像のピクセルが RGB の3チャンネルだったのに対し、中間特徴マップは各位置が256個の数値を持つ、といった具合に「意味の濃い」表現になります。BEV 特徴マップは、この H×W が地面の格子(グリッド)に対応します。たとえば 1セルが現実の 0.5m 四方に対応する、というように物理的な意味を持たせます。

残差接続(residual connection)とは、ある処理の入力をそのまま出力に足す `output = x + f(x)` という構造で、深いネットワークの学習を安定させる定番技法です(ResNet, He et al. 2016 で確立)。ここに学習可能な係数を掛けて `output = x + α·f(x)` とし、α を 0 から学習で立ち上げると、追加した経路(f)の寄与を「最初はゼロ、徐々に増やす」と制御できます。最初がゼロなら、追加経路がまだ役に立たない学習初期でも本体を一切乱しません。

クロスアテンション(cross-attention)とは、ある系列(query=質問する側)が別の系列(key/value=参照される側)から情報を取り込むアテンション機構です。各 query 位置が「どの key 位置を見るべきか」を内積で測り、その重みで value を加重平均して取り込みます。同じ位置同士に限らず、離れた位置からも情報を引ける柔軟さが特徴です。自己アテンション(self-attention)が同じ系列内で情報を混ぜるのに対し、クロスは異なる2つの情報源を橋渡しします。

ラスタ地図(raster map)とは、車線や進路を画素で描いた画像形式の地図です。対してベクトル地図(vector map)は、車線中心線を点列、道路境界を多角形といった幾何データで表します。ラスタは画像なのでそのまま画像バックボーンに通せる手軽さがあり、ベクトルは情報密度が高く解像度に縛られない利点があります。

## 仕組みを詳しく
処理は2段階です。

ステップ1: 地図エンコーダが地図画像を特徴マップに変換します。典型的には入力は `(B, 3, 256, 256)` の RGB 地図画像です。B はバッチサイズ、3 は RGB の3チャンネル、256×256 はピクセル数です。これを画像バックボーン(画像から特徴を抽出するネットワーク、たとえば Swin Transformer 系)に通し、`(B, 256, 8, 8)` という特徴マップにします。256 はチャンネル数、8×8 は粗い鳥瞰グリッドの解像度です。重要なのは、この出力の形状をカメラ由来の BEV 特徴と完全に一致させる点です。形が同じだから次のステップで重ねられます。詳細は [ラスタ地図エンコーダ (Raster Map Encoder) — 地図画像を鳥瞰特徴へ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1005_raster-map-encoder) を参照してください。

ステップ2: 融合モジュールが2つを融合します。入力はカメラ由来の `image_bev` `(B, 256, H, W)` と地図由来の `map_bev` `(B, 256, H, W)` の2つで、出力は融合後の `(B, 256, H, W)`(形は変わらない)です。方式が2つあります。

残差ゲート(residual)方式:
```
output = image_bev + alpha * map_bev
```
alpha はチャンネルごとに1つずつある学習可能なパラメータ(長さ256のベクトル)で、最初はすべてゼロに初期化されます。alpha=0 のとき output は image_bev そのものになり、地図は何も寄与しません。学習が進むにつれ、役に立つチャンネルの alpha だけが少しずつ大きくなり、地図情報が混ざっていきます。なぜゼロ初期化が重要か。学習初期に、まだ役に立たない地図特徴が一気に混ざってカメラ特徴を壊すのを防ぐためです。「最初は地図なしと完全に同じ挙動」から滑らかに地図ありへ移行できます。チャンネルごとに alpha を持つのは、地図が有用なチャンネル(進路に関わる方向情報など)と無関係なチャンネルを分けて扱えるようにするためです。

クロスアテンション(cross_attn)方式:
カメラ BEV の各位置を query、地図 BEV の各位置を key/value として、各カメラ位置が「地図のどの位置を見るべきか」を学習で決めて情報を取り込みます。`(B, 256, H, W)` を `(B, H·W, 256)` という系列に平坦化し、マルチヘッドアテンション(複数のヘッドで並行にアテンションを取る機構、たとえば8ヘッド)を通し、残差接続と FFN(全結合の小さなネット)を経て元の `(B, 256, H, W)` に戻します。残差ゲートが「同じ位置同士を足す」固定的な融合なのに対し、クロスアテンションは「位置をまたいで柔軟に対応づける」ぶん表現力が高い反面、計算量とパラメータが増えます。計算量は query 位置数 × key 位置数、すなわち (H·W)² に比例し、解像度の2乗で増えます。

つなぎ方は概念的に次のようになります。
```
map_bev        = MapEncoder(map_input)              # 地図 → 鳥瞰特徴 (B,256,8,8)
fused_features = MapBEVFusion(image_bev, map_bev)   # カメラ鳥瞰 + 地図鳥瞰 (B,256,H,W)
```

なぜこの設計なのか。BEV という共通座標系に揃えることで、異なる情報源(カメラと地図)を素直に重ねられます。残差ゲートのゼロ初期化は、新しい入力経路を既存の安定した挙動を壊さずに足し込むための安全装置です。クロスアテンションは、地図とカメラの空間的なずれ(GPS 誤差や地図の古さで、地図と実際の見え方が少し食い違うとき)を学習で吸収できる柔軟さを与えます。残差ゲートが「位置がぴったり合っている前提」なのに対し、クロスは「多少ずれていても対応点を探せる」と理解すると違いが明確です。

## 手法の系譜と主要論文
鳥瞰図表現とその上での情報融合は、近年の自動運転知覚研究の中心テーマです。

LSS(Lift, Splat, Shoot; Philion & Fidler, ECCV 2020, arXiv:2008.05711)は、カメラ画像の各ピクセルに奥行きの確率分布を推定して3D空間へ持ち上げ(lift)、それを鳥瞰グリッドへ投げ落とす(splat)ことで、複数カメラを1枚の BEV 特徴へ統合する方法を提案しました。動機は、複数カメラの情報を後段のプランニングが扱いやすい統一座標系へ落とし込むこと。効果は複数カメラの一貫した統合で、トレードオフは奥行き推定の誤差が BEV 特徴の歪みに直結する点です。

BEVFusion(Liu et al., ICRA 2023, arXiv:2205.13542)は、カメラと LiDAR という異なるセンサーをそれぞれ独立に BEV 特徴へ変換し、同じ BEV 空間で統合する汎用フレームワークを提案しました。地図融合は「センサー同士」ではなく「カメラと地図」を同じ BEV 空間で重ねる点で発想が同じで、異種情報を同一座標系に揃えてから足し合わせる戦略の有効性を引き継いでいます。トレードオフは各ブランチの座標系・解像度を厳密に揃える前処理コストです。

地図を明示的に入力するアプローチとしては、VectorNet(Gao et al., CVPR 2020, arXiv:2005.04259)や HD 地図(高精度地図)を条件として軌跡を予測する一連の予測研究があります。これらは地図が進路の事前知識を与えると予測精度が上がることを示しました。VectorNet はベクトル地図(点列・多角形)をグラフとして符号化しますが、ラスタ地図を使うアプローチは画像バックボーンをそのまま使える実装上の単純さを優先します。事前の HD 地図に頼らずカメラから車線をその場で予測する MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)はこの流れを進め、地図を「入力」から「予測対象」へ移す転換点になりました。さらに UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)は、知覚・地図・予測・プランニングを単一の BEV 表現の上で一気通貫に最適化し、地図を含む中間表現を共有する設計が最終的なプランニング性能に効くことを示しました。

残差ゲートでゼロ初期化して新ブランチを安全に足し込む手法は、ReZero(Bachlechner et al., 2020, arXiv:2003.04887)や Flamingo(Alayrac et al., NeurIPS 2022, arXiv:2204.14198)の gated cross-attention に通じます。ReZero は各残差ブロックに学習可能なスカラーをゼロ初期化で掛けることで、非常に深いネットワーク(数百層の Transformer)を正規化なしで安定に学習できることを示しました。Flamingo は、事前学習済み言語モデルに視覚を後付けする際、新規の cross-attention 層を tanh ゲートでゼロから立ち上げ、既存の言語能力を壊さずに視覚を統合しました。これらは「新しく追加した経路の寄与を学習可能な係数でゼロから立ち上げると、既存の安定した挙動を壊さず学習が安定する」ことの実証で、地図融合のゼロ初期化ゲートはまさにこの原理の応用です。

## 論文の実験結果(定量データ)
LSS(2020)は nuScenes の BEV セマンティックセグメンテーション(鳥瞰図上で車・道路・車線などを画素分類する課題)で評価され、当時の単眼ベースの手法を上回る IoU(Intersection over Union、予測領域と正解領域の重なり率。1に近いほど良い)を報告しました。複数カメラを統一座標系で扱う設計が、後段タスクにとって有効であることを定量的に示した点が重要です。

BEVFusion(2023)は nuScenes の3D物体検出で、当時の最先端を更新する mAP(mean Average Precision、検出の総合精度。高いほど良い)と NDS を達成し、報告では検出で mAP 約 70.2%、NDS 約 72.9% に達しました。カメラ単独・LiDAR 単独に対し、両者を BEV で融合した版が明確に上回り、特に LiDAR が苦手な遠距離・小物体でカメラ情報が補完的に効くことを示しました。さらに BEV セグメンテーションでも同時に高性能を出し、単一の BEV 表現が複数タスクを支えられることを実証しました。アブレーションでは、融合を BEV 空間でなく特徴レベルで雑に行うと性能が落ちることが示され、共通座標系への厳密な整列が効いていることが分かります。

ReZero(2020)は、12層 Transformer の言語モデリングで収束を約56%高速化し、128層という極端に深い Transformer を追加の正規化なしで安定に学習できることを示しました。ゲートをゼロ初期化することの実用的な威力を端的に表す結果です。Flamingo(2022)は、少数のサンプルだけで多様な視覚言語タスクに適応する few-shot 性能で当時の最先端を更新し、ゲート付き cross-attention によって既存の言語能力を保ったまま視覚を統合できることを示しました。地図融合のゼロ初期化ゲートは、これらの知見を「カメラ BEV に地図を後付けする」場面に応用したものと位置づけられます。UniAD(2023)は、エンドツーエンドでプランニングまで最適化したことで nuScenes のプランニング指標(平均 L2 誤差・衝突率)を当時の分割パイプラインから明確に改善し、地図を含む中間表現を共有・整列させる価値を最終タスクで実証しました。

アブレーションの観点では、地図ブランチを抜くと進路依存の判断(分岐での進路選択、車線変更のタイミング)が劣化する一方、目の前の障害物回避のような近距離反応は地図がなくても保たれる、というのが期待される典型です。また残差ゲートをゼロ初期化せず通常初期化にすると、学習初期にカメラ特徴が乱されて収束が遅くなる、あるいは不安定になることが観察されます。

## メリット・トレードオフ・限界
メリットは、カメラだけでは得られない進路・車線情報をモデルに与えられる点です。BEV 座標系を共有するので画素単位で素直に重ねられ、残差ゲートのゼロ初期化により「地図なしと同じ挙動」から安全に学習を始められます。融合方式を残差ゲートとクロスアテンションから選べるので、計算予算と表現力のバランスを実験で決められます。チャンネルごとのゲートは、どの特徴次元で地図を効かせるかを学習で選べる柔軟さを持ちます。

トレードオフと限界もあります。地図側に別途バックボーンが必要で、パラメータと計算が増えます。ラスタ地図はベクトル地図より情報密度が低く、解像度に縛られます。クロスアテンションは表現力が高い反面、H·W 個の位置間でアテンションを取るため計算量が解像度の2乗で増えます。地図とカメラの座標系・解像度を厳密に揃える前処理が前提で、ここがずれると融合がかえって害になります。さらに、地図情報の正確さ・最新さに依存するため、地図が古い・誤っている場合にモデルがそれを過信する危険があります(工事で車線構成が変わった、地図に無い新しい道など)。ゼロ初期化ゲートは初期に地図側へ勾配がほとんど流れないため、地図エンコーダの学習が始まるまでに時間がかかる、あるいは局所解で地図を使わないまま固定される(地図を無視する楽な解に落ち着く)可能性も指摘されます。これを避けるため、地図ブランチに別途の補助損失を与える、ゲートに小さな正のバイアスを入れるといった工夫が検討されます。

## 発展トピック・研究の最前線
さらに深く学ぶ方向として、第一に地図をモデル自身に作らせる「オンライン地図構築」(MapTR; Liao et al., 2023 など)があり、事前の HD 地図に頼らずカメラから車線をベクトルとして直接予測します。これにより地図の古さ問題を回避できますが、予測誤差という新たな不確実性が入ります。第二に、ベクトル地図とラスタ地図の利点を併せ持つハイブリッド表現や、地図を query として軌道生成に直接条件づける手法があります。第三に、地図の不確実性(古さ・誤り)を明示的にモデル化し、信頼度に応じて地図への依存度を変える研究が進んでいます。第四に、BEV 表現そのものを占有グリッド(occupancy、各3Dボクセルが埋まっているかを予測する表現)へ拡張し、地図・知覚・予測を1つの空間表現に統合する流れ(Occupancy Network)が最前線です。第五に、UniAD のように地図融合を含む全モジュールをプランニングという最終目的から逆算して同時最適化する end-to-end 設計が主流になりつつあり、地図融合を単独で最適化するのでなく下流タスクと一体で評価する流れが強まっています。
