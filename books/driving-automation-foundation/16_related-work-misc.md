## bf16 混合精度学習 (GradScaler 不要)

### ひとことで言うと
学習を速く・省メモリにするために「数値を 32bit でなく 16bit で計算する」のが混合精度学習です。16bit には fp16 と bf16 の2種類があり、bf16 を使うと、表せる数値の範囲が 32bit と同じくらい広いので、fp16 で必要だった「桁あふれ防止の補正装置(GradScaler)」を入れずにそのまま学習を回せます。これが「bf16 なら GradScaler 不要」という方針の核心です。

### 直感的な理解
数値を「定規の目盛り」にたとえます。fp32(32bit)はとても長く目盛りも細かい定規で、極小から極大まで何でも精密に測れますが、持ち運び(メモリ)が重く扱いが遅い。16bit はその半分の軽い定規です。ただし軽くする削り方に2通りあります。fp16 は「測れる範囲を狭めて、その範囲内の細かさは残す」削り方。bf16 は「測れる範囲は fp32 並みに保ったまま、細かさを犠牲にする」削り方です。

学習で困るのは「測れる範囲が狭い」ことの方です。学習中に出てくる勾配(直す方向の大きさ)は時に極端に小さくなり、fp16 の狭い範囲だと小さすぎて 0 に丸められてしまいます。0 では何も直せません。bf16 は範囲が広いので小さな勾配も 0 に潰れず、そのまま学習できます。だから fp16 で必要だった「小さい値を一時的に底上げする補正装置」が要らないのです。

### 基礎: 前提となる概念

まず「浮動小数点(floating point)」の精度を噛み砕きます。コンピュータは実数を「符号 + 指数部(どれくらいの大きさか) + 仮数部(細かい桁)」のビットで表します。指数部が「測れる範囲の広さ」を、仮数部が「目盛りの細かさ(精度)」を決めます。よく使う形式の内訳は次のとおりです。

| 形式 | 全体 | 指数部 | 仮数部 | 表せる範囲 | 細かさ |
|---|---|---|---|---|---|
| fp32(単精度, 32bit) | 32 | 8 | 23 | とても広い(約1e-38〜3e38) | 細かい |
| fp16(半精度, 16bit) | 16 | 5 | 10 | 狭い(約6e-5〜65504) | やや粗い |
| bf16(bfloat16, 16bit) | 16 | 8 | 7 | fp32 と同じくらい広い | 粗い |

ここが核心です。fp16 と bf16 はどちらも 16bit で「メモリ半分・計算速い」点は同じですが、ビットの配分が違います。fp16 は指数部 5bit で範囲が狭い一方、仮数部 10bit で細かさはそこそこ。bf16 は指数部 8bit(fp32 と同じ)で範囲は fp32 並みに広い一方、仮数部 7bit で細かさは粗い。

「混合精度学習(mixed precision, AMP = Automatic Mixed Precision)」とは、計算の大部分を 16bit で行い(速くて省メモリ)、桁落ちに弱い一部の計算(損失の合計、正規化層の統計量など)だけ 32bit に保つ、という良いとこ取りの手法です。重みの更新に使う「マスターコピー」も fp32 で保持するのが標準で、こうしないと小さな更新量が重みに反映されず学習が止まります。画像を多視点ぶん扱う自動運転モデルは非常にメモリを食うため、混合精度の恩恵が特に大きい領域です。

問題は fp16 を使うときに起きます。学習中の勾配はしばしば非常に小さくなり、fp16 の狭い範囲だと小さすぎて 0 に潰れます(アンダーフロー)。これを防ぐため fp16 では「loss scaling(損失スケーリング)」を使います。PyTorch ではこれを担う部品が `GradScaler` です。GradScaler は、損失を一時的に大きな数(例: 1024 倍)に掛けてから backward し、小さな勾配を fp16 の表せる範囲まで底上げし、重み更新の直前に同じ倍率で割り戻します。さらに掛けすぎて今度はオーバーフロー(範囲超え、結果が inf になる)したら倍率を自動で下げる、という動的調整も必要です。実装が複雑で、倍率調整が原因で学習が不安定になることもあります。

### 仕組みを詳しく

bf16 はこの面倒を丸ごと無くします。bf16 の指数部は fp32 と同じ 8bit なので、表せる範囲が fp32 並みに広く、小さな勾配が 0 に潰れにくいのです。そのため GradScaler なしでそのまま backward・step できます。代償は「仮数部 7bit で細かさが粗い」点ですが、深層学習の学習では多少の細かさの粗さより「範囲の広さ(潰れない・あふれない)」のほうが安定に効くため、bf16 は学習にとても向いています。

典型的な bf16 AMP の実装は次のようになります。

```python
use_amp = args.amp and device.type == "cuda"   # GPU のときだけ有効

with torch.autocast(device_type=device.type, dtype=torch.bfloat16, enabled=use_amp):
    pred = model(...)                # 内部の行列積・畳み込みが bf16 で走る
    loss = loss_fn(pred, target)
loss.backward()                      # GradScaler 不要
torch.nn.utils.clip_grad_norm_(model.parameters(), grad_clip)
optimizer.step()                     # scaler.step(...) ではなく素直な step
```

`torch.autocast(dtype=torch.bfloat16)` は「このブロック内の対象演算を自動で bf16 にする」という指示です。すべてを bf16 にするわけではなく、PyTorch が「bf16 で安全」と判断する演算(畳み込みや行列積など)だけを 16bit にし、損失の合計のような精度が要る箇所は 32bit に保ちます。これが「混合」精度たるゆえんです。

fp16 で必要だった典型コードと比較すると差が明確です。

fp16(参考):
```python
scaler = torch.cuda.amp.GradScaler()
with torch.autocast(dtype=torch.float16):
    loss = ...
scaler.scale(loss).backward()   # loss を底上げして backward
scaler.unscale_(optimizer)      # クリップ前に元の倍率へ戻す
scaler.step(optimizer)          # inf/nan があればこのステップをスキップ
scaler.update()                 # 倍率を自動調整
```

bf16 ではこの 4 行ぶんの補正がまるごと消え、`loss.backward()` → `optimizer.step()` の素直な形になります。コードが単純になり、倍率調整による不安定さも無くなります。

ただし bf16 は GPU のハードウェアが対応している必要があります。NVIDIA では Ampere 世代(A100, A10G など)以降、Ada Lovelace 世代(L40S, RTX 40 系)や Hopper 世代(H100)でもネイティブ対応しています。これらの GPU は bf16 専用の演算ユニット(tensor core)を持ち、fp32 の数倍のスループットで行列積を処理できます。逆に Turing 世代(T4 など)以前は bf16 のハードウェア対応が弱く、fp16 中心になります。bf16 ネイティブ対応の GPU を前提にできるからこそ「bf16 でよい → GradScaler 不要」と言い切れます。

### 手法の系譜と主要論文

- Micikevicius et al. (2018), "Mixed Precision Training"(ICLR, NVIDIA, arXiv:1710.03740): 混合精度学習を確立した論文。fp16 で計算を 16bit 化しつつ重みの「マスターコピー」は fp32 に保ち、小さい勾配の潰れを防ぐために loss scaling(GradScaler の原型)を提案しました。これが現代の AMP の出発点です。
- Wang & Kanwar (2019, Google AI Blog "BFloat16: The secret to high performance on Cloud TPUs"): bf16 の設計意図(範囲を犠牲にせず精度ビットを削る)を解説。bf16 はもともと Google の TPU 向けに設計され、TPU 上の大規模学習で標準フォーマットになりました。
- Kalamkar et al. (2019), "A Study of BFLOAT16 for Deep Learning Training"(Intel/Google, arXiv:1905.12322): bf16 を学習に使う体系的検証。指数部を fp32 と同じ 8bit にすることで loss scaling なしでも fp32 並みの収束が得られると報告しました。「GradScaler 不要」方針の理論的・実験的裏付けです。

系譜としては、fp16 + loss scaling(2018, NVIDIA)→ bf16 on TPU(2019, Google)→ bf16 on GPU(Ampere 世代以降)→ 現在は大規模言語モデルから視覚モデルまで bf16 が学習のデフォルトに、という流れです。fp16 は推論や一部の高精度が要る学習で今も使われますが、学習の安定性を重視する場面では bf16 が主流になりました。

### 論文の実験結果(定量データ)

Micikevicius et al. (2018) は ImageNet 画像分類(1000 クラス・約 128 万枚)や機械翻訳・音声認識など幅広いタスクで、fp16 混合精度が fp32 とほぼ同等の精度を達成しつつメモリをおよそ半減できることを示しました。たとえば ResNet-50 の ImageNet 学習で、fp32 のトップ1精度に対し混合精度がほぼ差のない精度(誤差 0.1〜0.2 ポイント程度)に収束しています。ただし loss scaling を外すと一部のネットワークで精度が大きく劣化(あるいは発散)し、fp16 では loss scaling が事実上必須であることも同時に示されました。トップ1精度とは「最も確信したクラスが正解と一致した割合」で、ImageNet では 1 ポイントの差でも有意とされます。

Kalamkar et al. (2019) は、bf16 を ResNet-50・DenseNet・各種言語モデルなどに適用し、loss scaling なしでも fp32 と同等の収束曲線(同じエポック数でほぼ同じ精度)が得られることを示しました。fp16 では loss scaling が必要だった同じネットワークが、bf16 では補正なしで素直に収束した、という対比が報告されています。これが「bf16 なら GradScaler を外せる」根拠です。仮数部が fp16 より 3bit 少ない(7 対 10)にもかかわらず最終精度がほぼ落ちないのは、学習の安定性には目盛りの細かさよりも測れる範囲の広さが効く、という主張を定量的に裏づけています。

実務上のメモリ・速度の効果も具体的です。bf16/fp16 化により活性化と勾配のメモリがおよそ半分になり、同じ GPU により大きなバッチや高解像度入力を載せられます。tensor core を使う行列積は fp32 比でしばしば 2〜数倍のスループットになり、学習時間が大きく短縮されます。たとえば多視点画像を扱う重いモデルでは、bf16 化でバッチサイズを倍にできることが珍しくありません。

### メリット・トレードオフ・限界

メリット
- 活性化・勾配のメモリがおよそ半減し、より大きなバッチや高解像度入力を載せられる。
- 16bit 演算で計算が速い(対応 GPU の tensor core を使う)。
- bf16 なら GradScaler が不要で、コードが単純・学習が安定(倍率調整の不安定さが無い)。
- マスターコピーを fp32 に保つだけで、fp32 とほぼ同等の最終精度に到達できる。

トレードオフ・限界
- bf16 は仮数部 7bit と粗いので、微小な数値差が重要な計算(損失の総和、正規化の統計量、累積和など)では精度が落ちうる。だからこれらは 32bit に残す。
- bf16 ネイティブ対応の GPU(Ampere 世代以降, L40S/A10G/H100 等)が前提。Turing 以前の GPU では恩恵が限定的か非対応で、その場合は fp16 + GradScaler に戻る必要がある。
- 細かさを犠牲にしているため、科学計算的に高精度が要る一部タスクや、極端に値域の狭い数値を扱う処理には不向きな場合がある。
- fp16 と bf16 の選択は GPU 世代・タスク・ライブラリ実装に依存し、「常に bf16 が最適」とは限らない(推論では fp16/int8 が有利なことも多い)。

### 発展トピック・研究の最前線

精度をさらに下げる流れは続いており、学習でも fp8(8bit 浮動小数点)を使う研究が進んでいます。Hopper 世代以降の GPU は fp8 の tensor core を持ち、行列積を fp8 で、累積を高精度で行う「混合精度の極限」が探求されています。fp8 には E4M3 / E5M2 という指数部・仮数部の配分違いがあり、forward と backward で使い分ける手法が提案されています。

また、NVIDIA Ampere 世代が導入した TF32(指数部 8bit・仮数部 10bit の 19bit 相当フォーマット)は、bf16 と fp32 の中間として行列積を高速化しつつ精度を保つ仕掛けで、デフォルトで有効になることが多いです。混合精度の設計は「どの演算をどの精度で、累積をどこまで高精度に保つか」という細かな配分の問題に発展しており、フレームワークが自動で最適配分を選ぶ方向(自動混合精度のさらなる自動化)が研究・実装の最前線です。

## 最小の学習ループ (forward→loss→backward→step)

### ひとことで言うと
ニューラルネットワークに「データを見せて、正解をマネさせる」一番小さな学習プログラムのことです。モデル・データ読み込み・損失計算という3つの部品をつなぎ、「予測する → どれだけ間違えたか測る → 直す方向を計算する → 重みを少し更新する」を繰り返します。自動運転の文脈では「カメラ画像などを入力し、人間が実際に走った軌跡を正解として真似る」回帰問題として実装されることが多く、これが実際に重みが更新される最小単位です。装飾的な機能を一切持たないこの骨格を最初に確実に動かすことが、後の複雑な学習システムすべての土台になります。

### 直感的な理解
楽器を独学で練習する場面を想像してください。お手本の演奏を聴き(正解)、自分で弾いてみて(予測)、お手本とのズレを耳で感じ取り(損失)、「ここの指づかいをこう直そう」と判断し(勾配)、少しだけ弾き方を変える(更新)。これを何百回も繰り返すと、だんだんお手本に近づきます。ニューラルネットワークの学習はこれと同じ構造です。違うのは、ズレの感じ方が数式で定義され、直す方向が微分で自動計算され、更新が数千万個のパラメータに対して一度に行われる点だけです。

なぜ「最小の」ループをわざわざ意識するかというと、深層学習の研究では、学習率スケジューラ・分散学習・チェックポイント・評価ループ・データ拡張など装飾的な機能が無数にあり、最初からすべて盛り込むとどこが原因で学習が失敗したのか分からなくなるからです。まず「データを1つ流して損失が下がる」ことを確実に動かす骨格を作り、そこから機能を足していくのが定石です。これはデバッグ哲学そのもので、「変数を一度に1つだけ動かす」という実験科学の基本に通じます。

### 基礎: 前提となる概念

ニューラルネットワークの「学習」とは何かを噛み砕きます。ネットワークは大量の数値パラメータ(「重み」と呼びます。現代の視覚モデルでは数百万〜数億個オーダー)を持ち、入力(カメラ画像など)から出力(進むべき軌跡など)を計算します。最初、重みはランダムなので出力はデタラメです。学習とは、「正解」と「モデルの出力」のズレを測り、そのズレが小さくなる方向へ重みを少しずつ調整していく作業です。

このズレを数値1個にまとめたものを「損失(loss)」と呼びます。学習の目標は損失をできるだけ小さくすることです。重みをどちらに動かせば損失が減るかは「勾配(gradient、損失を各重みで偏微分したもの)」が教えてくれます。勾配は「この重みを少し増やすと損失がどれだけ増えるか」を表す傾きで、損失を減らしたければ勾配と逆向きに重みを動かせばよい、というのが勾配降下法(gradient descent)の発想です。数式で書くと、重み w の更新は w ← w − η·∇w L です。ここで L は損失、∇w L は損失の w に関する勾配(各重みごとの傾きを並べたベクトル)、η(イータ)は学習率(歩幅)を表します。

勾配を効率よく計算する仕組みが「逆伝播(backpropagation, backward)」です。これは連鎖律(合成関数の微分の法則)を使い、出力側から入力側へ向かって勾配を伝えていく計算手順で、現代の深層学習フレームワーク(PyTorch など)が自動微分(autograd)として提供しています。私たちは forward の計算式さえ書けば、backward は自動で導出されます。autograd は forward の各演算を記録した「計算グラフ」を裏で組み立て、backward 時にそのグラフを逆向きにたどって連鎖律を適用します。

ここまでをコードに落とすと、最小限でも次の4ステップが要ります。

1. forward(前向き計算): 入力をモデルに通して出力を得る。
2. loss(損失計算): 出力と正解を比べてズレを1つの数値にする。
3. backward(逆伝播): 損失を各重みで微分して勾配を求める。
4. step(更新): 勾配の向きに重みを少し動かす(最適化器が担当)。

「最適化器(optimizer)」とは、勾配を受け取って実際に重みをどう動かすかを決める部品です。最も単純なのは SGD(確率的勾配降下法)で、重みから「学習率 × 勾配」を引くだけです。現代では Adam(Kingma & Ba 2015)/ AdamW(Loshchilov & Hutter 2019)が広く使われ、過去の勾配の一次モーメント(平均)と二次モーメント(分散)の移動平均を使って各重みごとに歩幅を自動調整するため、SGD より収束が安定・高速なことが多いです。「確率的」とは、データ全体ではなく一部(ミニバッチ)だけで勾配を近似する、という意味です。全データで毎回勾配を計算するのは重すぎるため、ランダムに選んだ小さなバッチで近似します。このノイズが逆に局所解からの脱出を助ける効果もあります。

### 仕組みを詳しく

自動運転の end-to-end 模倣学習を例に、1ステップの中身を入出力の形(tensor shape)つきで追います。`B` はバッチサイズ(一度に処理するサンプル数)、`V` はカメラ視点数(例として 7 視点)とします。

入力バッチ(辞書形式)の典型的な中身:
- `visual_tiles`: `[B, V, 3, 256, 256]` — V 視点ぶんの RGB 画像(3ch, 256x256)。
- `map_input`: `[B, 3, 256, 256]` — レンダリングしたナビ地図画像。
- `visual_history`: `[B, 896]` — 過去の視覚特徴をまとめたベクトル。
- `egomotion_history`: `[B, 256]` — 過去の自車運動(速度・舵角など)をまとめたベクトル。
- `trajectory_target`: `[B, T*S]` — 正解軌跡。例として `T=64` タイムステップ × `S=2` 信号(x と y)で `128`。

1ステップの流れを擬似コードで示します。

```python
optimizer.zero_grad(set_to_none=True)        # 前ステップの勾配をクリア
with torch.autocast(device_type="cuda", dtype=torch.bfloat16, enabled=use_amp):
    pred = model(visual_tiles, map_input,
                 visual_history, egomotion_history)   # forward: 出力 [B, 128]
    loss = loss_fn(pred, trajectory_target)           # loss: スカラー1個
loss.backward()                               # backward: 全重みの勾配を計算
torch.nn.utils.clip_grad_norm_(model.parameters(), grad_clip)  # 勾配の暴れを抑制
optimizer.step()                              # step: AdamW で重み更新
```

各行を初心者向けに補足します。

- `zero_grad(set_to_none=True)`: PyTorch は backward を呼ぶたびに勾配を `.grad` 属性へ「加算」していくので、毎ステップの最初にリセットしないと前回の勾配が混ざります。`set_to_none=True` はメモリ上の勾配バッファを 0 で埋め直すのではなく None に戻す省メモリ・高速な初期化です(次回 backward 時に新しく確保される)。
- 最適化器は `AdamW`(典型的に lr=1e-4, weight_decay=1e-2)。lr(学習率)は「1回の更新でどれだけ重みを動かすか」の歩幅です。weight_decay は重みが大きくなりすぎないよう少し縮める正則化(過学習を抑える仕掛け)です。AdamW は Adam の weight_decay を「損失への L2 正則化項として勾配に混ぜる」のではなく「重みを直接 (1−η·λ) 倍に縮める操作」として分離したもので、Loshchilov & Hutter (2019) が提案しました。Adam では正則化項が二次モーメントの分母に巻き込まれて意図通りに効かない問題があり、AdamW はそれを切り離すことで汎化性能を改善しました。
- `grad_clip`(典型的に 1.0): 勾配ベクトルの長さ(L2 ノルム)が閾値を超えたら、その長さに収まるよう全体を一律スケールして縮める処理です。学習初期や外れ値で勾配が急に巨大化して重みが吹き飛ぶ(発散する)のを防ぎます。Transformer 系のモデルでは勾配スパイクが起きやすく、clip はほぼ必須の安定化策とされています。
- `torch.autocast(dtype=torch.bfloat16)`: 混合精度学習(計算の一部を 16bit にして速く・省メモリにする)を有効にするコンテキストです。詳しくは [bf16 混合精度学習 (GradScaler 不要)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/16_related-work-misc)。

損失関数 `loss_fn` は、軌跡回帰では予測軌跡と正解軌跡の差を smooth-L1(Huber 損失)または MSE(平均二乗誤差)で測るのが一般的です。MSE は差を二乗するため外れ値に過敏で、smooth-L1 は誤差が小さい範囲(|x|<δ)では二乗・大きい範囲では一次関数に切り替わるため外れ値にロバストです。軌跡では稀に教師データに大きなノイズ(GPS のジャンプなど)が混ざるため、smooth-L1 が好まれます。詳しくは [軌道模倣損失（Trajectory Imitation Loss）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/10_losses)。

「imitation(模倣)」とは、専門的には「imitation learning(模倣学習)」または「behavior cloning(行動クローニング)」と呼ばれる手法で、「専門家(ここでは人間ドライバー)の行動データを正解として、そのまま真似るように学習する」やり方です。報酬を設計して試行錯誤する強化学習とは違い、「正解の軌跡にできるだけ一致させる」教師あり回帰問題として解きます。実装が単純で正解データさえあれば学べる反面、後述する分布シフトという本質的な弱点を抱えます。

### 手法の系譜と主要論文

end-to-end 運転の模倣学習は、深層学習以前から続く長い系譜を持ちます。

- Pomerleau (1989), "ALVINN: An Autonomous Land Vehicle in a Neural Network"(NeurIPS): カメラ画像と距離センサから人間の操舵を直接学習した古典で、入力 30x32 のビデオ + 8x32 のレンジ、隠れ層 29 ユニット、出力 45 ユニットというごく小さな3層ネットワークでした。end-to-end 運転学習の源流であり、「正解操作を真似る」発想そのものを提示しました。
- Bojarski et al. (2016), "End to End Learning for Self-Driving Cars"(NVIDIA, PilotNet, arXiv:1604.07316): 前方カメラ1枚から操舵角を回帰する CNN。約 25 万パラメータの 9 層(5 畳み込み + 3 全結合)ネットワークで、「画像 → 操作」の現代的原型を示しました。提案は単純な behavior cloning、効果は実車での車線追従ですが、トレードオフは「専門家が遭遇しない状況に弱い(分布シフト)」点です。
- Ross et al. (2011), "DAgger: A Reduction of Imitation Learning and Structured Prediction"(AISTATS): behavior cloning の弱点を理論的に整理し、その緩和策を提案しました。ナイーブな behavior cloning では1ステップあたりの誤り率を ε とすると累積コストが時間ステップ T に対して二次的(O(ε·T²))に膨らみうるのに対し、DAgger は線形(O(ε·T))に抑えられることを示しています。最小ループの「次」に来る改善方向を示す重要論文です。
- Codevilla et al. (2018), "End-to-end Driving via Conditional Imitation Learning"(ICRA): ナビ指示(直進・右折など)で出力を条件づけて模倣する手法。交差点で「どちらに曲がるか」が画像だけからは決まらない曖昧性を、追加の指示入力(コマンド)で解消しました。地図入力やナビ意図を扱う現代的な end-to-end モデルの土台です。
- Codevilla et al. (2019), "Exploring the Limitations of Behavior Cloning for Autonomous Driving"(ICCV, arXiv:1904.08980): behavior cloning が「データを増やしても性能が頭打ちになる」「学習ごとに結果が大きくばらつく(high variance)」「データセットのバイアス(ほとんど直進)を増幅する」といった構造的限界を体系的に示し、最小ループの素朴な拡張だけでは越えられない壁を明確にしました。

これらが共通して示すトレードオフは「模倣学習は実装が簡単で正解データさえあれば学べるが、専門家データの分布から外れた状況(事故寸前など)に弱い」という点です。最小ループはまずこの「簡単で確実な骨格」を作る位置づけにあります。

### 論文の実験結果(定量データ)

模倣学習の弱点と改善効果を定量的に見ると、最小ループの位置づけがはっきりします。

DAgger 論文(Ross et al. 2011)の代表的実験は、3D レーシングゲーム(Super Tux Kart)とアタリ風ゲームでの運転・操作タスクです。ナイーブな behavior cloning(教師データで一度だけ学習)はコース上を走るうちに少しずつ理想軌道からずれ、一度外れると教師データに無い状態に入って復帰できず、誤差が雪だるま式に増えました。これが compounding error(誤差累積)です。DAgger は学習したエージェント自身を走らせ、訪れた状態に対して専門家の正解を追加収集して再学習することを繰り返し、同じ正解データ量でも走行成功率を大きく改善しました。指標は「コースから外れるまでの距離」や「衝突回数」で、これらが BC より明確に少なくなった、という形で報告されています。

PilotNet(Bojarski et al. 2016)では、約 72 時間ぶんの走行データを集め、評価指標として「自動運転が有効だった時間の割合(autonomy)」を定義しました。autonomy は「(総経過時間 − 介入回数 × 6 秒)/ 総経過時間」で計算され(1回の介入を 6 秒の復帰時間と仮定)、実路テストでおよそ 98% という値が報告されています。これは「ほぼ自動で走れるが、たまに人間の介入が要る」水準で、模倣学習が実車で機能することを示した一方、残り数%が分布シフトの難しさを物語っています。

Codevilla et al. (2019) は CARLA(運転シミュレータ)上で、学習データ量を 2 時間から 50 時間まで増やしたときの「ルート完走率」を測りました。直感に反して、データを増やしても完走率が単調には伸びず、ある点で飽和あるいは悪化することがあると示しました。さらに同じ設定・同じデータで初期値だけ変えて学習すると、最終性能が大きくばらつく(成功率で数十ポイントの分散)ことを報告し、behavior cloning の「再現性の低さ」を定量化しました。

数値の意味を補足します。autonomy 98% は一見高く見えますが、「2% の時間は人間が直さないと危険」を意味し、これは数分に一度の介入に相当します。完全自動運転には桁違いの信頼性(介入間隔が数千〜数万 km)が必要で、模倣学習単体ではこのギャップを埋めきれない、というのが研究上の共通認識です。

### メリット・トレードオフ・限界

メリット
- 既存部品(モデル・データ・損失)をつなぐだけの最小構成で、確実に「損失が下がる」ことを検証できる起点になる。バグの切り分けが容易。
- 教師あり回帰なので学習が安定しやすく、強化学習のような報酬設計やシミュレータが不要。データさえあれば即学習できる。
- bf16 混合精度・勾配クリップなど安定学習の最低限を組み込めば、素直なループのまま発散しにくい。

トレードオフ・限界
- 模倣学習は分布シフト(専門家が経験しない状況)に弱く、誤差が累積するという本質的限界を抱える(DAgger が緩和を示したが完全解決ではない)。
- 「正解の真似」しかしないため、専門家を超える性能や、専門家が示さない安全マージン(危険回避の積極操作)の確保は原理的に難しい。
- 最小ループは学習率スケジューラ・評価ループ・早期終了・本格的なチェックポイント戦略を持たないため、長時間学習や本番運用には機能追加が必須。
- 軌跡が本質的にマルチモーダル(交差点で右折も左折も正解)な場合、L2/L1 回帰は平均をとってしまい「どちらでもない直進」を出す mode averaging が起きうる。これを避けるには分布を出力する手法(混合密度ネットワーク、拡散モデル等)が必要。
- データセットのバイアス(直進が大多数)をそのまま増幅し、稀だが重要な操作(急ブレーキ等)を学びにくい。

### 発展トピック・研究の最前線

最小ループから先へ進む研究の流れは大きく3方向あります。第一は分布シフト対策で、DAgger 系のオンライン学習、訪問しそうな状態への摂動を加えたデータ拡張(steering の左右ずらし画像を合成して復帰挙動を教える PilotNet のトリック)、敵対的に難しい状況を生成する手法などがあります。第二は出力のマルチモーダル化で、単一軌跡ではなく複数候補や確率分布を出す方向です。近年は拡散モデル(Diffusion Policy, Chi et al. 2023)で軌跡分布をサンプリングする手法が注目を集めています。第三は世界モデルとの統合で、未来の観測を予測する補助タスクを足して表現学習を強化する流れです(補助損失をどう本体損失に接続するかは依然として実装上の課題)。

また、模倣学習と強化学習を組み合わせる手法(模倣で初期化し RL で微調整する、逆強化学習で報酬を推定する等)も活発で、純粋な behavior cloning の限界を超えようとする研究が続いています。近年は大規模な走行ログから自己教師あり的に表現を事前学習し、その上で少量データの模倣を載せる「事前学習 + 模倣」の構成も主流になりつつあり、最小ループはその最終段(微調整)としての役割を担うことが増えています。

## レジストリ駆動の CLI 選択肢 (プラグイン的な部品差し替え)

### ひとことで言うと
学習スクリプトを `--backbone swin_v2_tiny` のようにコマンドラインから設定するとき、選べる値の一覧(choices)を、ソースコードに直接ベタ書きするのではなく「部品の登録表(registry)」から自動で引っ張ってくる設計のことです。新しい部品を登録表に1行足すだけで、スクリプト本体を一切いじらずにコマンドから選べるようになります。深層学習の実験管理で広く使われるプラグイン方式です。

### 直感的な理解
レストランのメニューと厨房を考えてください。お客さんが注文できる料理の一覧(メニュー)と、実際に作れる料理(厨房のレシピ集)が別々に管理されていると、新しい料理を作れるようになってもメニューに載せ忘れたり、逆にメニューにあるのに作れなかったりします。これを防ぐには、「厨房のレシピ集をそのままメニューとして表示する」のが確実です。レシピを1つ追加すれば自動でメニューに載り、削除すれば自動で消えます。

レジストリ駆動の CLI 選択肢はこれと同じ発想です。「実際に作れる部品の一覧(レジストリ)」を唯一の正とし、コマンドの選択肢(メニュー)はそこから自動生成します。深層学習では backbone やデータ拡張や損失関数を頻繁に差し替えて実験するため、この「メニューとレシピを自動で同期させる」仕組みが効きます。

### 基礎: 前提となる概念

まず「CLI」と「choices」を噛み砕きます。CLI(Command Line Interface)とは、`python train.py --backbone swin_v2_tiny` のように、端末(黒い画面)でコマンドに引数を付けてプログラムを動かす操作方法です。`--backbone` のような引数を「オプション」、それに渡せる値の許可リストを「choices(選択肢)」と呼びます。choices を指定しておくと、許可外の値(タイプミスなど)を渡したときに、プログラムが走り出す前に「その値は選べません。選べるのは A, B, C です」と即エラーで教えてくれます。Python の標準ライブラリ `argparse` では `add_argument("--backbone", choices=[...])` のように書きます。

次に「backbone(バックボーン)」とは、入力画像から特徴を抽出する土台のネットワークのことです。物体検出や軌跡予測などのタスクでは、まず汎用の画像認識ネットワーク(Swin Transformer、ConvNeXt、ResNet など)で画像を特徴ベクトルに変換し、その上にタスク固有の頭(head)を載せる、という構成が一般的です。この土台部分が backbone です。研究では backbone を差し替えて精度・速度・メモリを比較することが日常的に行われます。

「fusion-mode(視点融合方式)」のように、複数の特徴をどう統合するかを選ぶ引数も同様です。複数カメラの特徴を単純連結(concat)するか、注意機構(cross attention)で統合するか、俯瞰グリッド(BEV)に投影するか、といった選択肢が考えられます。

ここで問題になるのが「choices をどこに書くか」です。素朴にやると、スクリプト本体に次のようにベタ書きします。

```python
p.add_argument("--backbone", choices=["swin_v2_tiny", "conv_next_v2_tiny", "res_net_50"])
```

これだと、新しいバックボーン(たとえば EfficientNet)を実装したとき、実装ファイルに加えてこの行も忘れずに直す必要があります。2か所を手で同期させる必要があり、片方を直し忘れると「実装はあるのにコマンドから選べない」「choices に書いたのに実装が無い」というズレ(バグ)が起きます。部品が増えるほどこの同期コストとミスのリスクが膨らみます。

### 仕組みを詳しく

「registry(レジストリ、登録表)」を使うと、この同期問題が消えます。registry は「名前 → 部品を作る関数」を1か所にまとめた辞書(dict)です。典型的にはこう書きます。

```python
BACKBONE_REGISTRY = {
    "swin_v2_tiny":      lambda **kw: timm.create_model("swinv2_tiny_window8_256", **kw),
    "conv_next_v2_tiny": lambda **kw: timm.create_model("convnextv2_tiny", **kw),
    "res_net_50":        lambda **kw: timm.create_model("resnet50", **kw),
}
```

キー(`"swin_v2_tiny"` など)が「人が指定する名前」、値が「その名前のモデルを作る関数」です。`lambda` は無名の小さな関数で、ここでは「呼ばれたら対応するモデルを生成する」役割を持ちます。スクリプト側では、このキー集合をそのまま choices に流し込みます。

```python
p.add_argument("--backbone", default="swin_v2_tiny",
               choices=sorted(BACKBONE_REGISTRY))
```

`sorted(BACKBONE_REGISTRY)` は辞書を「キーのリスト」に変換します(Python では辞書を `sorted()` に渡すとキーがアルファベット順に並んだリストになる)。それをそのまま許可リストにしています。つまり choices は登録表の中身そのもので、登録表が唯一の正(single source of truth, 真実は1か所だけ)になります。

before / after で効果を見ます。新しいバックボーン `efficient_net_v2` を足したいとき、

- before(ベタ書き): 実装を追加 + スクリプトの choices にも `"efficient_net_v2"` を手で追記。2か所、同期忘れの危険あり。
- after(registry 駆動): レジストリに1行 `"efficient_net_v2": lambda ...` を足すだけ。スクリプトは無改変で `--backbone efficient_net_v2` が選べる。

部品を実際に組み立てる側(ビルド関数)にも、選択肢にない名前を渡したときの親切なエラーを入れておくのが定石です。

```python
def build_backbone(name, **kw):
    if name not in BACKBONE_REGISTRY:
        raise ValueError(f"Unknown backbone '{name}'. Available: {list(BACKBONE_REGISTRY)}")
    return BACKBONE_REGISTRY[name](**kw)
```

これで「どれが選べるか」を常に登録表が答える形になり、CLI の choices と実体が食い違いません。

このパターンは一般に「registry pattern」または「plugin(プラグイン)方式」と呼ばれます。本体を触らずに、登録表へ部品を差し込むだけで機能拡張できる構造で、ソフトウェア設計の Open-Closed 原則(Bertrand Meyer 1988 が定式化。「拡張には開き、修正には閉じる」)に沿っています。デコレータを使って登録を自動化する変種(`@register("name")` を関数定義の上に付けるだけで辞書に登録される)も広く使われます。

### 手法の系譜と主要論文

これは学術論文の手法というより、深層学習フレームワークで広く使われる設計パターンです。代表的な実装の系譜を挙げます。

- timm(PyTorch Image Models, Ross Wightman, 2019-): `timm.create_model("名前", pretrained=True)` という「名前文字列でモデルを作る」レジストリ機構を持ち、登場以来 1000 を超える画像モデルを文字列1つで切り替えられるようにしました。バックボーン比較実験のデファクト基盤です。
- Detectron2(Wu et al., Facebook AI Research, 2019): `@REGISTRY.register()` というデコレータでモデル・データ拡張・評価器を登録し、設定ファイルの文字列から組み立てます。物体検出研究で「実験設定をコードでなく設定で切り替える」標準形を確立しました。
- MMDetection / OpenMMLab(Chen et al., 2019, arXiv:1906.07155): `Registry` クラスと設定システムを徹底的に体系化し、backbone・neck・head・loss を独立した登録表として管理。論文の再現実験で「どの部品を組み合わせたか」を設定ファイルだけで完全に記述できるようにしました。
- Hydra / OmegaConf(Omry Yadan, Facebook, 2019): 設定とコードを階層的に分離し、CLI から構成要素を差し替える思想。レジストリと組み合わせると「実験の組み合わせ爆発」を設定ファイルとコマンドだけで扱えます。

これらが共通して示す効果は「実験の組み合わせ爆発を、コード改変なしに扱える」点で、トレードオフは「文字列キーのタイプミスや登録漏れが実行時まで分かりにくい(静的型では追いきれない)」点です。choices と build 関数の両方で `Unknown ...` を即エラーにすることで、この弱点を実務的に補います。

### 論文の実験結果(定量データ)

レジストリ自体は精度を上げる手法ではないため「ベースライン比 X% 改善」という形の数値は持ちませんが、その価値は再現性と実験効率という形で現れます。

OpenMMLab(MMDetection)のモデルズーは、レジストリと設定システムのおかげで数百種類の検出モデルを統一インタフェースで提供しています。たとえば同じ COCO データセット(物体検出の標準ベンチマーク、80 カテゴリ・約 12 万枚)上で、backbone を ResNet-50 から Swin-Tiny に差し替えるだけで box mAP(mean Average Precision、検出精度の主要指標で 0〜100%、高いほど良い)が変わる比較が、設定ファイル1行の変更で再現できます。具体的には Mask R-CNN の COCO box AP は ResNet-50 backbone でおよそ 38〜39、Swin-Tiny backbone でおよそ 43〜46 と報告されており、backbone を差し替えるだけで 5〜7 ポイントの差が出ます。mAP の意味は「IoU(予測枠と正解枠の重なり率)のしきい値 0.5〜0.95 を 0.05 刻みで動かしながら計算した Average Precision の平均」で、1〜2 ポイントの差でも検出研究では有意とみなされます。レジストリ駆動でこの比較が手作業のコード書き換えなしに回せることが、実験の信頼性(同じコードパスで条件だけ変える)に直結します。逆にもしコードを手で書き換えていたら、性能差が backbone の真の差なのか書き換え時の取り違えなのか切り分けられなくなります。

timm を使ったバックボーン比較の研究でも同様で、「コードを変えず文字列だけ変えて 50 種類以上の backbone を同条件で評価する」といったアブレーションが容易になりました。実際 timm の作者らによる ImageNet ベンチマーク表は、同一の学習・評価レシピで数百モデルの top-1 精度・スループット・パラメータ数を並べており、「同じ精度なら ConvNeXt と Swin のどちらが速いか」「計算予算 X GFLOPs で最良の精度は何か」を一目で比較できます。この体系的な知見の蓄積は、選択肢を文字列1つで切り替えられる registry 機構がなければ現実的な工数で集められませんでした。

### メリット・トレードオフ・限界

メリット
- 部品の追加が登録表への1行追記で済み、スクリプト本体を触らない(拡張に開き、修正に閉じる)。
- choices と実体が常に一致し、選択肢の同期忘れによるバグが起きない。
- 許可外の値を渡すと実行前に親切なエラーが出る。
- 同一コードパスで条件だけ変えられるため、アブレーション実験の信頼性が上がる。

トレードオフ・限界
- 文字列キーで指定するため、キーのタイプミスや登録漏れは静的(実行前)には完全に防げず、実行時のエラーに頼る面がある。IDE の補完や型チェックの恩恵を受けにくい。
- 「名前から関数を引く」間接化により、コードを追うとき登録表をいちいち見に行く必要があり、初見の読みやすさはベタ書きより落ちる。
- 過度に汎用化すると、実際には1択しか無い場合に仕組みだけ残って空回りする(縮退状態)。設計の複雑さに見合う拡張性が本当にあるかは見極めが要る。
- 動的登録(import 時にデコレータで辞書に積む)はモジュールが import されないと登録されないため、「実装はあるのにレジストリに現れない」という落とし穴がある。

### 発展トピック・研究の最前線

レジストリ思想は、実験管理ツール(MLflow、Weights & Biases 等)と組み合わさることで「設定ハッシュから実験を一意に再現する」仕組みへ発展しています。設定ファイル(どの backbone・どの loss・どのハイパーパラメータか)をそのまま実験 ID として記録し、後から完全に再現できるようにする流れです。

また、ハイパーパラメータ探索(sweep)との相性も良く、Hydra のマルチラン機能や Optuna のような最適化ライブラリと組み合わせると、「レジストリのキーの全組み合わせ × 学習率の格子」を自動で回す、といった大規模探索が設定だけで記述できます。プラグイン方式は今後も、コードと実験設定を分離して再現性を高める基盤として深層学習の標準であり続けると考えられます。

## smoke-test によるピーク VRAM プロファイリング

### ひとことで言うと
本物のデータを用意せず、形だけ合わせたランダムな数値のかたまり(テンソル)を使って学習を数ステップ回し、「GPU メモリを最大どれくらい使うか(ピーク VRAM)」を素早く測る手法です。とくに一番メモリを食う設定が手持ちの GPU に載るかを、データのダウンロードもライブラリの準備もなしに確認できます。深層学習の実務でハードウェア選定やバッチサイズ決定に使う定番の調査です。

### 直感的な理解
引っ越し前に「この家具は新しい部屋に入るか」を確かめたいとします。家具を実際に運び込んでから入らないと分かったら大損です。だから先に寸法だけ測って、部屋のサイズと比べます。家具の中身(木製か金属製か)は寸法に関係なく、入るかどうかは形(縦横高さ)だけで決まるからです。

GPU メモリも同じです。学習が GPU に載るかどうかは、流すデータの中身(本物の道路画像かランダムノイズか)ではなく、テンソルの形(shape)と計算グラフの構造で決まります。だったら、形だけ正しいランダムテンソルを流せば、本物のデータを用意せずに「最悪ケースのメモリ」を正確に測れます。これが smoke-test によるメモリプロファイリングの発想です。

### 基礎: 前提となる概念

まず「VRAM」と「OOM」を噛み砕きます。VRAM(Video RAM)は GPU が持つ専用メモリで、モデルの重み・計算途中の値(活性化)・勾配・最適化器の状態などをすべてここに置きます。VRAM が足りなくなると「OOM(Out Of Memory, メモリ不足)」というエラーで学習が落ちます。GPU の VRAM 容量は機種で大きく異なり、例として T4 は約 16GB、A10G は約 24GB、L40S は約 45GB、A100/H100 は 40〜80GB です。

学習で VRAM を消費する内訳は主に4つです。

1. モデルの重み(パラメータ): モデルのサイズで決まる固定費。
2. 勾配: 各重みに対応する勾配で、重みとほぼ同じ量。
3. 最適化器の状態: Adam/AdamW は各重みについて移動平均を2つ持つので、重みのおよそ2倍。
4. 活性化(activation): forward の途中結果で、backward で勾配を計算するために取っておく必要がある。これがバッチサイズ・入力解像度・層数に比例して膨らみ、大規模モデルでは支配的になる。

自動運転モデルが VRAM を大量に使うのは主に活性化が原因です。(1) 多視点ぶんの画像を同時に処理する、(2) BEV(Bird's-Eye View, 俯瞰の鳥瞰図)という空間グリッドを高解像度で持つと中間データが巨大になる、(3) バッチサイズを上げるほど比例して増える、という3つが重なります。「この構成は VRAM に収まるのか、それとも OOM するのか」を事前に知りたい、という強い需要があります。

ここで素朴に本物のデータで確認しようとすると、数十 GB のデータセットをダウンロードし、データ読み込みライブラリを入れ、デコード処理を通す、という重い準備が要ります。VRAM の上限確認をしたいだけなのにこれは時間の無駄ですし、環境が整う前には試せません。

「smoke test(スモークテスト)」という言葉は、電子機器に電源を入れて「煙が出ないか(壊れて発火しないか)」を最初に見る工学の慣習から来ています。ソフトウェアでは「細かい正しさは置いておき、まず動くか・致命的に壊れないかをざっと確認する最小テスト」を指します。

### 仕組みを詳しく

smoke-test では、本物のデータ読み込みをスキップして、データセットが出すのと同じ形のランダムテンソルでバッチを作ります。

```python
B, V = batch_size, num_views      # 例: B=4, V=7
batch = {
    "visual_tiles":      torch.randn(B, V, 3, 256, 256, device=device),
    "map_input":         torch.randn(B, 3, 256, 256, device=device),
    "visual_history":    torch.randn(B, 896, device=device),
    "egomotion_history": torch.randn(B, 256, device=device),
    "trajectory_target": torch.randn(B, num_timesteps * num_signals, device=device),
}
```

`torch.randn(...)` は指定した形の、標準正規分布に従うランダム値を作る関数です。中身はデタラメですが、形は本物のバッチと一致させてあります。これを数ステップ(例: 5 回)用意して回します。

メモリ計測の心臓部は、計測区間の開始時にピークをリセットし、終了時に最大値を読む2行です。

```python
torch.cuda.reset_peak_memory_stats(device)             # 計測開始(ピークを0に戻す)
...                                                    # forward/backward/step を回す
peak = torch.cuda.max_memory_allocated(device) / 1e9   # GB に直して出力
```

`max_memory_allocated` は「その区間で PyTorch が確保した GPU メモリの最高到達点」を返します。学習のメモリは forward 後・backward 中に最大化する(forward で活性化が全層ぶん積み上がり、backward でそれを消費しながら勾配を作る)ので、この値が「この設定でこの GPU に必要な最低限の VRAM」の目安になります。

最悪ケースを測る工夫として、補助的なブランチ(損失には未接続でもメモリは食う計算経路)を明示的に有効化するフラグを用意することがあります。これを付けると「フル機能を有効にしたときの最悪メモリ」を測れ、外すと軽い構成を測れます。

数値例として、ある多視点 + BEV モデルを bf16 AMP で smoke-test した場合の典型的な傾向を挙げます。

| 構成 | peak VRAM |
|---|---|
| 単純連結 (concat) バッチ8 | およそ 7 GB |
| BEV 高解像度 バッチ4(補助ブランチ off) | およそ 20 GB |
| BEV 高解像度 バッチ4(補助ブランチ on) | およそ 23 GB |
| BEV 高解像度 バッチ8(補助ブランチ on) | OOM |

この表から、「BEV 高解像度はバッチ4までが安全圏、バッチ8 + 補助ブランチは 45GB クラスの GPU でも OOM する」という設計判断が、データを一切落とさずに分かります。こうした実測値が、GPU を 16GB クラスから 45GB クラスへ切り替える根拠や、バッチサイズ・解像度の上限を決める根拠に直結します。

1つ注意があります。ランダム入力では損失値そのものは意味を持たず、計算経路によっては nan(非数)になることもあります。これはコードの欠陥ではなく、ランダム入力を特定の数値経路(投影変換など)に通したときの数値的副作用です。smoke-test の目的は損失の正しさではなく VRAM 計測なので問題になりませんが、実データでの数値安定性は別途確認が必要です。

### 手法の系譜と主要論文

メモリプロファイリングそのものは論文の手法ではなく実務のプラクティスですが、背景にある「学習メモリの理解と削減」の研究は豊富です。

- Sohoni et al. (2019), "Low-Memory Neural Network Training: A Technical Report"(arXiv:1904.10631): 学習メモリの内訳(重み・活性化・最適化器状態・勾配)を体系的に分解し、活性化が大規模モデルで支配的であることを定量的に示しました。smoke-test が測ろうとしている「forward〜backward のピーク」が何に支配されるかを理解する基礎です。
- Chen et al. (2016), "Training Deep Nets with Sublinear Memory Cost"(arXiv:1604.06174, gradient checkpointing の原典): 学習時メモリの主因が「backward 用に取っておく中間活性化」であることを示し、計算を一部やり直してメモリを削る手法(gradient checkpointing)を提案しました。「メモリと計算のトレードオフ」を意識する出発点です。
- Rajbhandari et al. (2020), "ZeRO: Memory Optimizations Toward Training Trillion Parameter Models"(SC20, Microsoft, arXiv:1910.02054): 大規模学習でメモリが律速になることと、その内訳を整理し、最適化器状態・勾配・パラメータを複数 GPU に分散して保持する手法を提案。なぜ「事前にメモリを見積もる」ことが重要かを大規模スケールで裏づけます。
- PyTorch 公式ドキュメント "CUDA semantics > Memory management": `reset_peak_memory_stats` / `max_memory_allocated` などの標準 API を解説。smoke-test の計測はこの標準 API に素直に従います。

これらが共通して示すのは「学習メモリは活性化が支配的で、解像度・バッチ・モデル構造で大きく変わる」という点で、だからこそ構成ごとに実測する smoke-test の価値が出てきます。

### 論文の実験結果(定量データ)

メモリ削減手法の効果を数値で見ると、smoke-test で何を測っているかがより鮮明になります。

Chen et al. (2016) の gradient checkpointing は、n 層のネットワークの活性化メモリを O(n) から O(√n) に削減できることを示しました。具体例として、当時の深い CNN で活性化メモリをおよそ数分の1に圧縮し、その代わり forward の一部を再計算するために学習時間がおよそ 30% 程度増える、というトレードオフが報告されています。これは「メモリは活性化が支配的で、計算を犠牲にすれば削れる」ことの定量的証拠で、smoke-test で測ったピーク VRAM が大きすぎた場合の次の一手(checkpointing を入れる)を選ぶ判断材料になります。

ZeRO(Rajbhandari et al. 2020)は、Adam の最適化器状態が fp32 で重みの何倍ものメモリを食うことを定量化しました。混合精度学習では、fp16 の重み・勾配に加えて fp32 のマスター重み・運動量・分散を持つため、1 パラメータあたりおよそ 16 バイト(2+2+4+4+4 の内訳)が必要になり、これが大規模モデルで支配的になることを示しています。ZeRO はこれを複数 GPU に分散することで、たとえば数十億〜数千億パラメータ級のモデルを現実的な GPU 台数で学習可能にしました。smoke-test が「重み + 勾配 + 最適化器状態 + 活性化」の合計を測っているという事実は、この内訳の理解があって初めて正しく解釈できます。

数値の意味を補足します。smoke-test で得られる peak VRAM は「最低限これだけは要る」という下限の目安であり、実運用ではデータローダのプリフェッチバッファや CUDA のメモリ断片化(フラグメンテーション)で実際の使用量がやや上振れします。したがって OOM ギリギリ(GPU 容量の 95% など)で設計するのは危険で、ある程度の余裕を見込むのが定石です。

### メリット・トレードオフ・限界

メリット
- データのダウンロードやライブラリ準備なしに、数秒〜数十秒で peak VRAM を測れる。
- 形さえ合っていればメモリ量は本物と同等なので、最悪ケースの見積もりとして信頼できる。
- OOM するかどうかを事前に判定でき、GPU 選定・バッチ・解像度の設計に直結する。
- backbone や融合方式を差し替えながら測れば、構成ごとのメモリ特性を体系的に比較できる。

トレードオフ・限界
- 入力がランダムなので学習の正しさ(損失が下がるか、数値が安定か)は一切検証しない。損失が nan になることもある。
- 測れるのは「メモリと、forward/backward が落ちずに通るか」だけで、精度・収束は別途実データで確認が必要。
- 実データ特有のメモリ要因(可変長シーケンス、極端な入力値、データローダのバッファ)は再現しないので、実運用の使用量とは完全には一致しない(やや低めに出がち)。
- CUDA のメモリ断片化や cuDNN のアルゴリズム選択により、同じ shape でも実行ごとにピークが数%揺れることがある。

### 発展トピック・研究の最前線

メモリ削減技術は smoke-test と組み合わせて使うことが多く、代表的なものに gradient checkpointing(活性化を捨てて再計算)、activation offloading(活性化を CPU メモリへ退避)、ZeRO 系の状態分散、そして FlashAttention のような「注意機構の中間行列を実体化しない」省メモリ実装があります。FlashAttention は注意の計算をブロック単位で行い、巨大な注意行列を VRAM に書き出さないことで、系列長に対する活性化メモリを大幅に削減しました。

近年は学習前にメモリと計算量を解析的・実測的に見積もるプロファイラ(PyTorch の memory profiler、各種 FLOPs カウンタ)が整備され、smoke-test 的な実測と解析的見積もりを突き合わせて「どの層がどれだけメモリを食うか」を可視化する流れが進んでいます。大規模モデル時代には、学習を始める前にメモリ収支を正確に予測することが計算資源の効率的利用に直結するため、この方向の研究・ツール整備は今後も重要であり続けます。

## 凍結 backbone によるターゲット生成

### ひとことで言うと
JEPA 風に「未来や欠損部分の特徴」を予測して学習させるには、予測の正解(ターゲット)が必要です。その正解を、対象フレームの画像を「凍結した(重みを更新しない)画像エンコーダ(backbone)」に通して作る方式が、凍結 backbone ターゲット生成です。ターゲット側を動かさず勾配も流さないことで、正解が安定し、表現崩壊(representation collapse、全部が同じ無意味な特徴に潰れる失敗)を避けやすくなります。これは自己教師あり学習(self-supervised learning, ラベルなしで表現を学ぶ枠組み)における中心的な技術課題への、最もシンプルな解の一つです。

### 直感的な理解
特徴空間で未来や欠損部分を予測する仕組みは、「予測 vs 正解」の損失で学習します。ところがこの正解は、画像分類のような外から与えられる人手ラベルではありません。正解もまたネットワークが画像から作る特徴です。「自分で出題して自分で答え合わせ」をする構造になっています。

ここに自己教師あり学習特有の落とし穴があります。もし正解を作るエンコーダ(ターゲットエンコーダ)と予測器を一緒に勾配で学習させると、両者が共謀して「正解も予測も常に同じ定数(たとえば全部ゼロ)」にしてしまえます。そうすれば損失はゼロになりますが、特徴は何の情報も持たない、まったく無意味なものになります。テストで「全問の答えを『3』にしておけば、採点基準も『3が正解』にすればいつでも満点」というズルと同じです。これが表現崩壊です。損失だけ見ていると「うまく学習できている」ように見えるのに、できた特徴はゴミ、という最も厄介な失敗様式です。

崩壊を防ぐ最も素直な方法が「ターゲット側を動かさない」ことです。正解を作るエンコーダの重みを固定(凍結, frozen)し、その経路には勾配を流しません(stop-gradient、誤差逆伝播をそこで止める)。予測器だけが学習するので、予測器が正解を都合よく定数に引きずることができず、崩壊しません。さらに、凍結 backbone を ImageNet などで事前学習済みの強いものにすれば、最初から意味のある特徴を正解として使えます。「採点基準(正解)が固定されていて、生徒(予測器)がそれを書き換えられない」状態にするわけです。

### 基礎: 前提となる概念
backbone(バックボーン): 画像から特徴を取り出す土台ネットワークです。Swin Transformer、ConvNeXt、ResNet、Vision Transformer (ViT) などが代表で、しばしば ImageNet で事前学習されています。下流タスク(検出・予測など)はこの上に小さなヘッドを載せて作ります。

凍結(freeze): パラメータを更新対象から外すことです。学習中も重みが変わりません。対義は fine-tune(微調整)で、こちらは下流タスクに合わせて重みを動かします。実装上は requires_grad=False と eval() の併用で実現します。

stop-gradient: 計算グラフ上で、ある経路の勾配計算を止める操作です(PyTorch なら .detach())。ターゲットの値は使うが、その値を作った経路は学習信号を受け取らない、という非対称性を作ります。崩壊回避の核心です。

EMA(指数移動平均, Exponential Moving Average): あるネットワーク(コンテキスト/オンライン側)の重みを、別のネットワーク(ターゲット側)へゆっくりコピーする方式です。ターゲット = momentum × ターゲット + (1 − momentum) × オンライン、のように毎ステップ少しずつ追従させます(momentum は 0.99〜0.9999 程度)。完全凍結より「ターゲットも少しずつ良くなる」が、予測器に即座に引きずられないので崩壊しにくい、という中間策です。

表現崩壊(representation collapse): 上述のとおり、エンコーダの出力が入力によらず一定(または極端に低次元)になり、情報を失う失敗です。自己教師あり学習の最大の敵で、各手法はこれをどう防ぐかで分類できます。検出には、特徴の分散がほぼゼロになっていないか、線形評価精度がランダム水準に落ちていないか、を見ます。

### 仕組みを詳しく
未来特徴を予測するワールドモデルにこの方式を組み込む場合の典型的な配線を、ステップで示します。

1. 学習データには「現在フレーム」と「未来フレーム(+1.6/3.2/4.8/6.4 秒など)」が含まれる。
2. 現在フレームを学習対象のメインモデル(backbone + 融合 + プランナー + 未来予測ヘッド)に通し、未来各ステップの予測特徴 [B, C, H, W] × 4 を出す。
3. 未来フレームを凍結 backbone(勾配なし、stop-gradient)に通し、各未来時刻の正解特徴を作る。
4. 損失 = 各時刻について「予測特徴」と「凍結 backbone が出した正解特徴」の距離(例: L2、要素ごとの差の二乗和、あるいはコサイン距離)。これを最小化する。

```
現在フレーム ─[学習する backbone+融合+planner]─ 未来予測ヘッド ─▶ 予測特徴 ×4 ─┐
                                                                              ├─ loss
未来フレーム ─[凍結 backbone(勾配なし / stop-grad)]──────────▶ 正解特徴 ×4 ─┘
```

ここに実務的な落とし穴があります。予測特徴と正解特徴の形(チャンネル数 C、空間サイズ H×W)が一致しないと損失が計算できません。未来予測ヘッドの出力が [B, 256, H, W] でも、凍結 backbone の特定ステージ出力は [B, 768, H', W'] のように異なるのが普通です。多くの backbone は features_only(中間の特徴マップを取り出すモード)で複数解像度を返し、そのチャンネル数・空間サイズは予測側と揃いません。たとえば階層型 backbone の各ステージは入力解像度の 1/4, 1/8, 1/16, 1/32 と段々に小さくなり、チャンネルは逆に増えます。そこで両者の次元を合わせる射影層(projection layer)を間に挟む必要があります。

数値例: 未来予測ヘッドの1枚が [B, 256, H, W]、凍結 backbone の該当ステージ出力が [B, 768, H', W'] だとすると、(a) チャンネルを 768→256 にする 1×1 畳み込み、(b) 空間を H'×W' → H×W に合わせる補間(interpolation、画素を補って解像度を変える)、を入れて初めて L2 損失が取れます。射影層を「予測側に付ける」か「ターゲット側に付けるが勾配は流さない」かで設計が分かれますが、ターゲット経路に学習可能な部品を増やすと崩壊リスクが戻る(その射影が正解を都合よく潰せてしまう)ため、射影は予測側に置き、ターゲットは完全に凍結したまま、とするのが安全です。

凍結と EMA の違いも整理します。完全凍結は「事前学習済みの backbone を一切動かさない」最も単純な実装で、ハイパーパラメータ調整が不要な反面、ターゲット特徴が学習中に改善しません。EMA は「ターゲットがゆっくり良くなる」ため最終的な表現品質で有利になりやすい反面、momentum という追従速度のハイパーパラメータ調整が要ります。完全凍結は、EMA の momentum を 1 に固定した極端な特殊ケース(まったく追従しない)と見ることもできます。実装の手軽さを取るなら凍結、最終品質を狙うなら EMA、という棲み分けです。

### 手法の系譜と主要論文
- Jean-Bastien Grill et al., "BYOL: Bootstrap Your Own Latent" (NeurIPS 2020, arXiv:2006.07733)。JEPA ではないが、崩壊回避の発想の源流です。ターゲットネットを EMA + stop-gradient にすることで、ネガティブサンプル(対比用の別画像、押し離す相手)なしでも崩壊しないと示しました。提案動機は対比学習のネガティブ収集コスト(大きなバッチや巨大なメモリバンクが要る)を省くこと。後続研究で「予測器(predictor)と stop-gradient の組み合わせが崩壊回避の本質」と整理され、ターゲット側を非対称に扱う設計の正当化になりました。
- Xinlei Chen & Kaiming He, "SimSiam" (CVPR 2021, arXiv:2011.10566)。BYOL から EMA すら外し、「予測器 + stop-gradient だけ」で崩壊しないことを示しました。これにより「ターゲットを勾配から切る非対称性こそが崩壊回避の鍵で、EMA は必須ではない」ことが明確になり、完全凍結ターゲットの妥当性を理論的に後押しします。論文中の「stop-gradient あり/なし」の対比実験が、この分野で最も明快な崩壊デモになっています。
- Mathilde Caron et al., "DINO" (ICCV 2021, arXiv:2104.14294)。ViT を自己蒸留(self-distillation、teacher は student の EMA、teacher 出力を stop-gradient)で学習し、centering(出力の平均を引いて一点集中を防ぐ)と sharpening(温度を下げて鋭くする)で崩壊を防ぎました。ターゲット(teacher)を勾配から切る設計の代表例で、教師なしでも物体の輪郭が attention に現れることで有名になりました。
- Mahmoud Assran et al., "I-JEPA" (CVPR 2023, arXiv:2301.08243)。ターゲットエンコーダをコンテキストエンコーダの EMA で更新し stop-gradient を併用、特徴空間でマスク領域を予測しました。「ピクセルでなく特徴を正解にし、ターゲットは勾配から切る」という、凍結 backbone ターゲットの直接の手本です。
- Adrien Bardes et al., "V-JEPA" (2024)。動画版で同じターゲット設計を時間方向に拡張し、未来予測ヘッドのターゲット生成に最も近い構図(未来フレームの特徴を正解にする)を与えます。
- 対照群として Kaiming He et al., "MAE: Masked Autoencoders" (CVPR 2022, arXiv:2111.06377)。正解を「生のピクセル」にするため、そもそもターゲットエンコーダが不要で崩壊問題も起きません(正解が固定の画素値だから潰しようがない)。代わりに予測不能なピクセル細部を当てる無駄を抱えます。凍結 backbone ターゲットは「ピクセルでなく意味的特徴を正解にしつつ、凍結で安定させる」中間策と位置づけられます。

### 論文の実験結果(定量データ)
- BYOL (NeurIPS 2020): ImageNet 線形評価で ResNet-50 を用い top-1 約 74.3% を達成し、当時の対比学習 SimCLR(約 69.3%)を約 5 ポイント上回りました。アブレーションで「stop-gradient を外す(ターゲットにも勾配を流す)と表現が崩壊し精度が激減する」ことが示され、非対称性の必須性が定量化されました。
- SimSiam (CVPR 2021): stop-gradient を外すと損失は急速にゼロへ向かう一方で表現は崩壊し、線形評価精度がほぼランダム水準(数%)まで落ちることを明示しました。逆に stop-gradient を入れると ResNet-50 で top-1 約 70% 前後の良好な表現が得られ、「崩壊か健全かは stop-gradient 一つで切り替わる」という劇的な対比を示しました。これは凍結ターゲット設計の最も直接的な実験的根拠です。損失が下がっても特徴がゴミ、という前述の罠を一枚のグラフで突きつけた研究として教育的価値が高いです。
- I-JEPA (CVPR 2023): ImageNet 線形評価で ViT-Huge を用い約 72–73% top-1。ターゲットを EMA + stop-gradient にしないと崩壊し性能が大きく低下すること、予測ブロック(当てる領域)を十分大きくすると下流性能が上がることがアブレーションで確認されました。
- DINO (ICCV 2021): teacher を student の EMA + centering/sharpening にすることで、凍結特徴の k-NN 分類(学習なしで最近傍だけで分類)でも ImageNet 約 78%(ViT-S/8)に達し、ターゲットを勾配から切る設計でも非常に強い表現が得られることを示しました。線形評価でなく k-NN でこの精度というのは、特徴がそのまま意味的に整っている強い証拠です。

これらに共通する定量メッセージは明快です。「ターゲット経路を勾配から切る(凍結 or EMA + stop-gradient)」というたった一つの非対称性が、崩壊(精度がランダム水準)と健全(精度 70% 超)を分ける、ということです。指標としての線形評価精度は「凍結した特徴の上に線形分類器だけを載せたときの精度」で、特徴がどれだけ意味情報を線形に取り出せる形で持つかを直接測るため、崩壊の検出に最適です。崩壊すれば線形分類器は何も学べずランダム水準に落ちるので、一発で異常が分かります。

### メリット・トレードオフ・限界
- メリット: ターゲットを凍結 + stop-gradient にするだけで表現崩壊を確実に防げる。実装が単純で EMA の momentum 調整も不要。
- メリット: 事前学習済み backbone を使えば、最初から意味のある正解特徴が得られ、学習初期から信号が安定する。
- トレードオフ: 凍結ゆえに正解特徴が学習中に改善しない。当該タスク(運転シーンの未来予測)に最適化された特徴にはならず、EMA 方式に表現品質で劣ることがある。
- トレードオフ: 予測側出力と backbone 出力の次元不一致を埋める射影層が別途必要で、その設計(どちらに付けるか、解像度合わせ)が崩壊リスクや学習しやすさに影響する。
- トレードオフ: 正解の質が凍結 backbone の事前学習に依存し、事前学習ドメイン(自然画像、ImageNet)と対象ドメイン(運転シーン、特に夜間・悪天候・逆光)がずれていると正解特徴が弱く、予測学習も伸びない。これをドメインギャップと呼びます。
- 未解決課題: 凍結 vs EMA vs 学習可能ターゲットのどれが運転の未来予測で最適かはタスク依存で定説がない。崩壊を事前に検知する確実な指標、ドメインギャップを埋める凍結 backbone の選び方も研究途上。

### 発展トピック・研究の最前線
発展方向の一つは、完全凍結から EMA ターゲットへ移行し、ターゲットも徐々に対象ドメインへ適応させることです(I-JEPA / V-JEPA の方式)。もう一つは、ターゲットを単一スケールでなく複数スケール(backbone の複数ステージ)で取り、未来予測ヘッドにも複数解像度の予測を出させてマルチスケールで損失を取ること。これは細部と大局の両方を正解に含められ、検出・セグメンテーションのように多解像度が効くタスクと相性がよいです。

さらに、運転特化の自己教師あり事前学習(大量の無ラベル走行映像で凍結 backbone を先に鍛える)により、ドメインギャップを縮める研究や、ターゲットエンコーダにマルチモーダル(画像 + LiDAR)を使い、より幾何的に正しい正解特徴を作る方向も注目されています。研究者として押さえるべき核心は、SimSiam / BYOL が示した「崩壊回避は stop-gradient による非対称性が本質であり、対比のためのネガティブも必須ではない」という洞察で、凍結 backbone ターゲットはその最も実装コストの低い実現形だ、という点です。新しい予測タスクを設計するときは、まず「ターゲット経路は勾配から切れているか」を確認するのが、崩壊を避ける第一の防衛線になります。

## カメラ数が異なる多データセット学習

### ひとことで言うと
自動運転の公開データセットは、車に付いているカメラの台数がそれぞれ違います。これらを全部使って1つのモデルを学習したいのですが、台数の違うデータを同じバッチに混ぜることはできません。そこで「1つのミニバッチの中ではカメラ台数を揃え、バッチごとにどのデータセットから取るかを切り替える」という戦略をとります。これが均一バッチ(homogeneous batch)による多データセット学習です。1バッチ内は均一、バッチ間は多様、というのがポイントです。

### 直感的な理解
料理教室で、ある班は「7種類の食材セット」、別の班は「8種類の食材セット」を使うとします。先生が一度に全班へ同じ指示を出すには、混ぜたセットだと指示が通りません。そこで「このターンは7食材セットの班だけ」「次のターンは8食材セットの班だけ」と分けて指導すれば、各ターン内は均一なので同じ手順で教えられます。そしてターンを切り替えていけば、結局すべての班(=すべてのデータセット)を教えられます。

機械学習は基本的にデータが多く多様なほど賢くなります。1つのデータセットだけだと、そのデータの偏り(撮影地域・天候・車種・カメラ配置)をそのまま学んでしまい、初見の状況に弱くなります。複数データセットをまとめて学べば、より多様な状況を経験でき、汎化(見たことのない場面でもうまく動くこと)が期待できます。だから台数の違いという技術的な壁を、均一バッチという工夫で乗り越えるわけです。研究コミュニティでは、画像分類・物体検出・自然言語処理のいずれでも「複数の出所のデータを混ぜて1つの大きなモデルを鍛える」のが標準的な強化策になっており、自動運転もその流れに乗っています。

### 基礎: 前提となる概念

基本用語から。

- **データセット**: 学習に使うデータの集まり。自動運転では「走行中のカメラ映像 + そのときの正解の軌道や物体位置」のセットが大量に入っています。

- **ミニバッチ(mini-batch)**: 一度の学習ステップでまとめて処理するデータのかたまり。バッチサイズ8なら8シーンぶんを同時に流します。GPU は同じ計算を大量に並列で行うのが得意なので、まとめて処理します。

- **V(num_views)**: カメラ台数。データセットごとに異なります。

- **`torch.stack`**: 複数のテンソルを新しい次元で積み重ねて1つにする操作。形が完全に一致していないと使えません。7ビューのサンプル `[7,3,H,W]` と8ビューのサンプル `[8,3,H,W]` は最初の次元が違うので積めません。これが「1バッチ内 V 均一」が必要な技術的理由です。テンソルは「形がそろった数の塊」なので、形が違うものは1つのテンソルに入れられない、という GPU 計算の根本制約から来ています。

- **汎化(generalization)**: 学習に使っていない新しいデータでも正しく動くこと。機械学習の最終目標です。学習データに過剰に適合(過学習、overfitting)すると、訓練データでは高得点でも初見データで失敗します。データの多様性を増やすことは、この過学習を抑える最も効く手段のひとつです。

- **ドメインギャップ(domain gap)**: データセット間の系統的な違い。色味(カメラのセンサ特性)、解像度、撮影地域、天候、交通ルール(左側通行/右側通行)、カメラ配置などが該当します。混ぜるだけで汎化が上がる一方、ギャップが大きすぎると「水と油」で学習が不安定にもなります。

データセットごとの台数は構成によりさまざまで、たとえば「6カメラ」「7カメラ」「5カメラ」「8カメラ」のように手元の複数データセットでバラつくのが普通です。なぜ全部使いたいかは前述のとおり、多様性が汎化を生むからです。物体検出や3D知覚の世界では、複数の公開データセットを混ぜて基盤モデル(foundation model)を作る流れが定着しています。

技術的な壁は `torch.stack` の制約です。だから「1バッチ内は台数を揃える」必要があります。

### 仕組みを詳しく

戦略は2段構えです。

#### 段1: モデルを台数可変にする

そもそもモデルが V=7 でも V=8 でも動かないと話になりません。BEV融合モジュールを実行時の V に追従させる改修が前提になります。詳細は [BEV融合の num_views ランタイム動的化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/14_model-architecture-design)。これができると、同じモデルインスタンスに7ビューバッチも8ビューバッチも流せます。出力は V によらず `[B, embed_dim, bev_h, bev_w]` で揃うので、その先の軌道計画器や損失計算は、どのデータセット由来かを気にせず同じコードで動きます。ここが効いていて、台数の違いはモデルの入り口(BEVエンコーダ)で吸収され、それ以降のネットワークは一切カメラ台数を意識しません。

#### 段2: バッチを均一に組む

データローダ側で、1つのミニバッチには同じデータセット(=同じ V)のサンプルだけを入れます。そしてステップごとにどのデータセットから取るかを切り替えます。

```
ステップ1: データセットA から 8シーン → バッチ形 [8, 7, 3, H, W]  (V=7)
ステップ2: データセットB から 8シーン → バッチ形 [8, 8, 3, H, W]  (V=8)
ステップ3: データセットC から 8シーン → バッチ形 [8, 5, 3, H, W]  (V=5)
ステップ4: データセットA から 8シーン → バッチ形 [8, 7, 3, H, W]
...
```

各バッチの中では V が揃っているので `torch.stack` できます。モデルは V動的なので、どのバッチも処理し、出力は常に同じ形で揃います。

#### 切り替えの具体的なやり方(実装の選択肢)

- **重み付きサンプリング**: 各データセットからどれくらいの頻度でバッチを取るかを決めます。データ量に比例させると大きいデータセットが支配的になり、わざと均等にすると小さいデータセットが埋もれません。実務では温度付き(temperature)サンプリングがよく使われ、データ量 `n_i` に対し確率を `p_i ∝ n_i^(1/T)` とします。`T=1` なら量に比例、`T→∞` なら均等。多言語翻訳の研究(Arivazhagan et al., 2019)で `T=5` 前後がよく使われたように、自動運転でも `T=2〜5` 相当で「大きすぎるデータの支配」と「小さいデータの無視」の中間を取るのが定石です。たとえばデータ量の0.5乗(`1/T=0.5`)に比例させる設定がよく引かれます。

- **複数 DataLoader の束ね**: 各データセットを別々の `DataLoader` にし、どのローダから次のバッチを取るかを毎ステップ確率的に選ぶサンプラを書きます。各ローダが内部で自分のシャッフルを持つので実装が素直です。小さいデータセットのローダが先に尽きないよう、無限イテレータ化(尽きたら再シャッフルして繰り返す)するのが定番です。

- **校正情報を一緒に渡す**: データセットごとにカメラの校正(キャリブレーション)が違うので、各バッチに対応する `camera_params` を必ず付けます。BEV射影はこの校正に依存するため、付け忘れると座標がずれて性能が落ちます。詳細は [カメラパラメータ(キャリブレーション)入力](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/14_model-architecture-design)。

#### 注意点: 勾配のばらつきとバランス調整

データセットごとにシーンの難しさや正解軌道の分布が違うと、バッチごとに損失の大きさや勾配の向きがばらつきます。これが学習を不安定にします。具体的には、難しいデータセットのバッチで勾配が急に大きくなり、せっかく学んだ重みを壊すことがあります(loss spike)。対策:

- **データセットごとの損失スケーリング**: 各データセットの損失に係数をかけ、スケールを揃える。たとえば軌道の単位(m と cm)やサンプリング周期が違うと損失の絶対値がずれるので、それを正規化します。
- **勾配クリッピング**: 勾配ノルムが閾値を超えたら頭打ちにし、暴れを抑える。`torch.nn.utils.clip_grad_norm_` が定番です。
- **GradNorm 的なバランス**: 複数のタスク/データの勾配の大きさを動的に揃える手法(GradNorm, Chen et al., ICML 2018)。本質的には「学習が遅いほうに重みを寄せる」発想です。各損失の重みを、学習進度(loss の下がり方)が遅いものほど大きくなるよう自動調整します。

#### 均一バッチ以外の選択肢

均一バッチは実装が素直ですが唯一解ではありません。**パディング+マスク**方式では、足りないカメラをダミー画像で埋めて全サンプルを最大Vに揃え、ダミー部分を可視マスクで無効化します。これなら混合バッチが組めますが、ダミーぶんの無駄計算が発生します(V=5 のサンプルを V=8 に詰めると 3/8 が無駄)。どちらを取るかは「実装の単純さ・GPU効率(均一バッチは無駄計算ゼロ)」対「サンプリングの自由度(混合バッチは1ステップで複数データセットを混ぜられ、勾配が滑らかになりうる)」のトレードオフです。

### 手法の系譜と主要論文

「多データセットを混ぜて1モデルを鍛える」という思想の系譜です。

- **共有表現の有効性: UniAD (Hu et al., CVPR 2023, arXiv:2212.10156)**。複数タスク(検出・追跡・地図・予測・計画)を1つのBEV表現で統合学習する手法。データセット混合とは別軸ですが、「共有表現を多様な信号で鍛えると最終性能が上がる」という、多データセット学習の根拠と同じ思想です。トレードオフは「タスク間/データ間のバランス調整が難しい」。

- **多データセット物体検出: Simple Multi-Dataset Detection (Zhou et al., CVPR 2022, arXiv:2102.13086)**。COCO・Objects365・OpenImages など、ラベル体系の違う複数の検出データセットを統一して1つのモデルで学習する手法。各データセットの分類器を分けつつ共有のバックボーンを鍛え、統一ラベル空間を自動で学習します。「データセットごとにラベル定義が衝突する問題をどう吸収するか」が中心課題でした。

- **語彙拡張: Detic (Zhou et al., ECCV 2022, arXiv:2201.02605)**。画像分類データ(ラベルだけある弱い教師)も検出学習に取り込み、扱える物体クラスを大幅に拡張。多種多様なデータを混ぜて汎化を広げる流れの代表例です。

- **台数非依存BEVの基盤: BEVFormer / PETR (arXiv:2203.17270 / arXiv:2203.05625)**。カメラ台数や配置に依存しない3D表現を作る研究群。均一バッチ戦略が成立する前提(モデルが台数非依存であること、[BEV融合の num_views ランタイム動的化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/14_model-architecture-design))を支えます。

- **勾配バランス: GradNorm (Chen et al., ICML 2018, arXiv:1711.02257)**。複数の損失の勾配スケールを動的に揃える手法。多データセット/多タスク学習の「勾配のばらつき」問題への一般的処方箋です。

- **温度付きサンプリングの出自: 多言語機械翻訳 (Arivazhagan et al., 2019, arXiv:1907.05019)**。データ量の偏った多言語コーパスを混ぜる際、`p_i ∝ n_i^(1/T)` の温度付きサンプリングで高頻度言語の支配を抑える手法。自動運転の多データセット混合でも、この比率制御の発想がそのまま使われています。

自動運転で出力が軌道(連続値)の場合、物体検出のようなラベル体系の衝突は起きにくく、主な障壁はカメラ台数と校正の違いに絞られます。だから検出分野の「統一ラベル空間」問題ほど難しくなく、台数動的化と均一バッチで大半が解けます。逆に検出ヘッドも持たせる場合は、Simple Multi-Dataset Detection が扱ったラベル衝突問題が再び顔を出します。

### 論文の実験結果(定量データ)

- **Simple Multi-Dataset Detection (arXiv:2102.13086)**。COCO・Objects365・OpenImages を混ぜて学習したモデルが、各データセットを個別に学習したモデルと同等以上の精度(各データセットの mAP、検出の正確さを測る0〜1のスコア)を1つのモデルで達成し、さらに学習に使っていない検出ベンチマーク(たとえば未学習のデータセット)へのゼロショット転移でも単一データセット学習を上回ったと報告されました。混合学習の汎化利得を定量的に示した代表例です。1つのモデルが3つのデータセットを「個別最適に近い精度」で同時にこなせる点が、保守コストの面でも大きな意味を持ちます。

- **Detic (arXiv:2201.02605)**。LVIS の希少クラス(出現回数の少ない物体)の検出 AP(平均精度)を、画像分類データの併用で大きく改善(希少クラス AP で約数ポイント〜十数ポイント、報告による)。データを混ぜて多様性を足すほど、データの少ない領域の性能が上がるという典型的な傾向です。希少クラスほど混合の恩恵が大きい、という非対称性は自動運転でも当てはまり、出現の少ない状況(緊急車両、特殊な交差点)の汎化に効くと期待されます。

- **多データセット混合の一般傾向**。自動運転の知覚・計画でも、複数データセット混合は初見ドメイン(別都市・別天候)での性能低下を抑えることが繰り返し報告されています。改善幅は混合比率・データの近さ・指標により数ポイント〜十数ポイントと幅があり(報告による)、「データ量と多様性が増えるほど汎化が上がるが、ドメイン差をどう吸収するかで利得が変わる」という共通理解があります。

数値の読み方として大事なのは、(1) 平均値だけ見ると小さいデータセットの低性能が隠れるので、データセット別に評価を分けて見ること、(2) 真の利得は「学習に使っていないデータでのゼロショット性能」に現れること、の2点です。均一バッチ学習を評価するときも、データセット別 + クロスデータセットの両方を報告するのが研究上の作法です。さらに、混合比(温度 T)を動かしたときの「小データセット性能 vs 大データセット性能」のトレードオフ曲線を示すと、混合戦略の良し悪しが明確になります。

### メリット・トレードオフ・限界

- **メリット**: 複数データセットを使えるのでデータ量が増え、汎化性能が上がります。地域・天候・車種の多様性を取り込めます。
- **メリット**: モデルが1つで済むので、保守・推論デプロイが単純になります。
- **トレードオフ**: 1バッチ内 V 均一の制約があるため、データローダの実装が複雑になります(データセット単位のバッチ構成・サンプリング戦略が必要)。
- **トレードオフ**: データセット間でシーン難易度・軌道分布が違うと勾配がばらつき、学習が不安定になりえます。サンプリング比・損失スケールのバランス調整に手間がかかります。
- **トレードオフ**: 各データセットの校正情報を正しく渡さないと、BEV射影がずれて性能が落ちます。
- **限界・未解決課題**: 評価のとき平均値に小さいデータセットの低性能が隠れる問題。データセット間で正解の定義(軌道の長さ・サンプリング周期・座標規約・左右どちら通行か)が微妙に違うときの統一。均一バッチはGPUの並列効率を「同一データセット連続」に縛るため、分散学習で各ワーカーが別データセットのバッチを引くと勾配同期時に意味が混ざりうる(全ワーカーで同じデータセットを引かせる工夫が要る)点も実務上の課題です。

### 発展トピック・研究の最前線

- **温度付きサンプリングの自動調整**: データセット混合比を固定せず、学習中の各データセットの損失低下速度を見て動的に配分する手法。バリデーション損失をオンラインで監視し、伸び悩むデータセットの比率を上げます。
- **ドメイン適応との融合**: 単に混ぜるだけでなく、データセット間のドメイン差(色味・解像度・カメラ配置)を明示的に縮める(特徴空間でのアライメント、敵対的学習、スタイル正規化)研究。
- **基盤モデル化**: 大量のデータセットを混合して大規模に事前学習し、下流の特定環境へ少量データで適応(fine-tuning)する流れ。自動運転でも「運転の基盤モデル」を目指す研究が増えています。
- **センサ構成のカリキュラム**: 学習初期は単純な構成(少ないカメラ)から始め、徐々に複雑な構成を混ぜる学習スケジュール。安定性を上げる工夫として検討されています。
- **負の転移(negative transfer)の検出**: 混ぜると逆に性能が下がるデータセットの組み合わせを見つけ、自動で重みを下げる/外す研究。すべての混合が善とは限らない、という現実的な視点です。

## BEVFormer（論文）

### ひとことで言うと

BEVFormer は、車のまわりに取り付けた複数台のカメラ映像を、「車を真上から見下ろした1枚の地図」（これを BEV、Bird's-Eye-View と呼びます。鳥が空から見下ろした視点という意味です）の形に統合する手法です（Li ら、ECCV 2022）。最大の特徴は、各ピクセルの奥行き（深度）を直接予測しないことです。代わりに「地図上のこのマスは、各カメラ画像のどこに映るはずか」をカメラの幾何計算で求め、その投影先から画像特徴を集めてきます。深度予測の不安定さを回避しつつ、複数カメラと過去フレームを1つの統一表現にまとめる、というのが核心です。

### 直感的な理解

人間が運転するとき、頭の中には自然と「自分の車を中心とした周囲の見取り図」があります。前に車が1台、左車線に1台、後ろから1台近づいてくる、というように、視点を空から見下ろす形に変換して状況を把握しています。ところがカメラは斜め前方や横を向いて景色を撮っているだけで、この「見取り図」を直接は与えてくれません。

カメラ1台の視野は水平方向に60〜120度程度しかなく、360度をカバーするには6台前後を並べる必要があります。さらに、1枚の画像は3次元世界を2次元に押しつぶしたものなので、「前の車まで何メートルか」「自分は車線のどこにいるか」を画像から直接読み取れません。

BEVFormer が解く問題はまさにここです。複数の斜め視点の画像から、運転判断に使いやすい「真上から見た見取り図（BEV 特徴マップ）」を機械的に組み立てる。これができれば、その上に物体検出・車線認識・軌跡計画といった下流タスクを載せられます。BEVFormer の登場以前にも BEV を作る試みはありましたが、それらは「ピクセルごとに奥行きを当てる」という難しい問題を中心に据えていました。BEVFormer はその難所を回避する設計を提案した点に大きな意味があります。

### 基礎: 前提となる概念

理解に必要な道具を順に噛み砕きます。

カメラの内部・外部パラメータ（キャリブレーション）。カメラがどこにどの向きで付いているか（外部パラメータ、回転 R と並進 t）と、レンズの焦点距離や画像中心（内部パラメータ K）を表す行列です。これがあれば「3次元空間のある点 (X, Y, Z) は、このカメラの画像の (u, v) ピクセルに映る」という対応を計算できます。式で書くと、ある3次元点 p をカメラ画像上の点 (u, v) に写す操作は、おおまかに `[u, v, 1]ᵀ ∝ K · [R | t] · [X, Y, Z, 1]ᵀ` という行列の掛け算で表せます。ここで「∝」は定数倍の自由度（奥行きスケール）が残ることを意味します。この計算を「投影（projection）」と呼びます。BEVFormer はこの投影を使い、地図上の点が画像のどこに映るかを求めます。

アテンション（attention）。Transformer という仕組みの中心部品で、「ある問い合わせ（クエリ）が、たくさんの候補（キーとバリュー）のうちどれをどれだけ重視するか」を重み付き平均で計算する操作です。式では `Attention(Q, K, V) = softmax(QKᵀ / √d) · V` と書きます。Q がクエリ、K がキー、V がバリュー、d は次元数で、√d は内積が大きくなりすぎないための正規化です。重みは「クエリとキーの相性（内積 QKᵀ）」から決まります。BEVFormer では「地図のマス（クエリ）が、画像特徴（バリュー）をどれだけ集めるか」にこれを使います。

変形可能アテンション（deformable attention）。通常のアテンションは全候補との相性を計算するため、候補が多いと計算量が爆発します（候補 N 個に対して O(N²)）。Deformable DETR（Zhu ら、ICLR 2021、arXiv:2010.04159）が提案したこの仕組みは、各クエリが「基準点のまわりの少数点（例えば4〜8点）」だけを見るように制限します。具体的には、クエリから「基準点に対するオフセット（ずれ）」を MLP で予測し、そのずれた位置から `grid_sample`（補間サンプリング）で特徴を取り出して重み付き和を取ります。これで計算量を O(N×K)（K は少数のサンプル点数）まで下げます。BEVFormer の空間アテンションはこれを多視点・3次元に拡張したものです。

BEV グリッド。地図を格子状のマスに区切ったものです。例えば 200×200 のマスで前後左右約 100m をカバーすると、1マスは約0.51m四方になります。各マスに特徴ベクトルを持たせたものが BEV 特徴マップです。

### 仕組みを詳しく

BEVFormer は「BEV クエリ」「空間クロスアテンション（Spatial Cross-Attention, SCA）」「時間自己アテンション（Temporal Self-Attention, TSA）」の3本柱で構成されます。

#### 1. BEV クエリ

BEV 地図を H×W（論文では 200×200）のマス目に区切り、各マスに C 次元（論文では256次元）の学習可能なベクトルを割り当てます。テンソルの形で書くと `[H*W, C] = [40000, 256]` です。各クエリは「私の担当するマスの場所では、世界はどうなっている?」と画像に問いかける主体だと考えてください。学習可能なので、最初はランダムな値ですが、学習を通じて「どんな特徴を集めればよいか」を覚えます。各クエリには対応する BEV 平面上の物理座標 (x, y) も紐づいており、これが後の投影計算の出発点になります。

#### 2. 空間クロスアテンション（SCA）

各 BEV クエリが画像特徴から情報を吸い上げる中心処理です。

```
1. BEV マスごとに3D基準点を作る（高さ方向に複数=ピラー）
   例: 1マスにつき高さ違いの4点 (Z = -5m, -3m, 0m, +1m など)
2. その3D点をカメラ行列（内部×外部）で各カメラ画像へ投影
   → 「このマスは前カメラの (u,v) に、右カメラには映らない」を得る
3. 投影先の周辺から少数点だけ特徴をサンプル（deformable attention）
4. そのマスを物理的に見られるカメラだけで集約して平均
   output = (1/|見えるカメラ数|) * Σ_view  サンプルした特徴
```

ここで肝になるのが「ピラー（pillar）」です。BEV のマスは平面上の位置 (X, Y) しか決まっておらず、高さ Z が分かりません。地面の高さの物体（道路標示）と背の高い物体（トラックの屋根）では、同じ (X, Y) でも画像上の投影位置が違います。そこで複数の高さの点を用意し、それぞれを投影してサンプルし、「どの高さが効くか」をアテンションの重みに学ばせます。これにより、明示的な深度推定をしなくても奥行きの曖昧さを吸収できます。これが「深度を予測しない」の正体です。深度の確率分布を当てる代わりに、あらかじめ用意した複数の高さ候補のどれを重視するかをデータから学ぶ、と言い換えられます。

可視性マスクも重要です。車の後ろのマスは前カメラには映りません。投影した (u, v) が画像の範囲外なら、そのカメラからは集約しない。物理的に正しい対応だけを使うことで、無関係な特徴の混入を防ぎます。1つの BEV マスが平均すると1〜2台程度のカメラにしか映らないため、この絞り込みは計算効率にも効きます。

#### 3. 時間自己アテンション（TSA）

1フレーム前の BEV 特徴を、自車がその間に動いた量（ego-motion、自車運動）で位置合わせします。例えば車が前に2m進んだなら、過去の BEV を2m後ろにずらして現在の座標系に合わせる、という操作です。位置合わせした過去 BEV を、現在の BEV クエリが（やはり変形可能アテンションで）参照します。これにより、動いている物体の速度推定や、一瞬隠れた（オクルージョンした）物体の補完ができます。BEVFormer は過去の BEV を再帰的に引き継ぐ（RNN のように1フレームずつ蓄積する）方式を取るため、多数の過去フレームを毎回まとめて処理する方式より計算とメモリが軽くなります。

#### 全体構造と数値感

SCA と TSA を1セットとしたエンコーダ層を6層スタックします。各層で BEV クエリは少しずつ精緻化されます。

- BEV 解像度: 200×200（約 102.4m × 102.4m、1マス約0.51m）
- 埋め込み次元: 256
- エンコーダ層数: 6
- カメラ台数: nuScenes 標準で6台
- ピラー点数: 4
- バックボーン: ResNet-101-DCN もしくは VoVNet（V2-99）

得られた `[200, 200, 256]` の BEV 特徴マップの上に、検出ヘッド（Deformable DETR ベース）やセグメンテーションヘッドを載せます。BEVFormer 自体は「BEV エンコーダ」であり、その出力が下流の様々なタスクの共通土台になる、という構造です。

### 手法の系譜と主要論文

多視点カメラからの BEV 表現構築には大きく2系統がありました。

深度ベース（Lift-Splat-Shoot, LSS）。Philion & Fidler、ECCV 2020、arXiv:2008.05711。各ピクセルについて奥行きの確率分布を予測し、その分布に従って画像特徴を3次元空間に「持ち上げ（lift）」、BEV 平面に「ばらまき（splat）」ます。直感的で強力ですが、深度予測が外れると BEV が大きく崩れます。また深度の確率ボリュームがメモリを食います。BEVDet（Huang ら、2021）がこれを3D検出に応用し、BEVDepth（Li ら、AAAI 2023、arXiv:2206.10092）は LiDAR を使って深度を教師あり学習することで精度を底上げしました。

クエリベース・幾何投影（BEVFormer）。Li, Wang, Li ら、ECCV 2022、arXiv:2203.17270。深度を予測する代わりに、BEV クエリから画像へ幾何投影してサンプルする「第3の道」を示しました。土台になったのは Deformable DETR（Zhu ら、ICLR 2021）の変形可能アテンションと、DETR（Carion ら、ECCV 2020、arXiv:2005.12872）のクエリベース検出の発想です。BEVFormer の新規性は、これを多視点・3次元基準点・時間方向に拡張し、深度教師なしで動く統一 BEV エンコーダを作った点にあります。

その後の展開。BEVFormer v2（Yang ら、CVPR 2023、arXiv:2211.10439）は、最新の画像バックボーン（ImageNet 事前学習だけでなく検出向け事前学習を生かしたもの）との相性を改善し、「perspective 3D 検出」を補助タスクとして加えることで、純粋な ImageNet 分類で訓練した画像エンコーダでも高精度を引き出せるようにしました。UniAD（Hu ら、CVPR 2023、Best Paper、arXiv:2212.10156）は BEVFormer を BEV エンコーダとして採用し、検出・追跡・予測・計画まで一気通貫にしました。これは BEVFormer の BEV 表現が下流タスクの土台として有効であることを示した代表例です。

### 論文の実験結果（定量データ）

評価は主に nuScenes データセット（6カメラ・360度・約1000シーン・40,000フレーム）で行われました。指標を噛み砕きます。

mAP（mean Average Precision）。検出した3Dボックスが正解とどれだけ一致するかの平均適合度です。一致の判定には3次元の中心点距離（0.5/1/2/4m の4閾値）を使い、その平均を取ります。0〜1で高いほど良い。

NDS（nuScenes Detection Score）。nuScenes 専用の総合指標で、mAP に加えて、位置誤差(mATE)・大きさ誤差(mASE)・向き誤差(mAOE)・速度誤差(mAVE)・属性誤差(mAAE)の5種の誤差をまとめた0〜1のスコアです。式は `NDS = (1/10) · [5·mAP + Σ(1 - min(1, 誤差))]` の形で、検出の「総合点」と考えてください。

報告値（テストセット、VoVNet-V2-99 バックボーン）として、BEVFormer は NDS 約0.569、mAP 約0.481 を達成しました。当時の多視点カメラ手法を上回る水準です。比較として、純粋な単眼検出手法 FCOS3D（NDS 約0.428、mAP 約0.343）や、深度ベースの初期手法 DETR3D（NDS 約0.479）より明確に高いスコアでした。検証セット（ResNet-101-DCN）では NDS 約0.517、mAP 約0.416 が報告されています。

時間融合（TSA）の効果はアブレーションで示されています。時間自己アテンションを入れると、特に速度推定の誤差（mAVE）が大きく下がります。報告では、TSA を入れると mAVE がおよそ 0.7m/s 台から 0.4m/s 台へと顕著に改善し、NDS も数ポイント上がりました。これは「1フレームだけでは静止物と動体の区別がつきにくいが、過去フレームを参照すると動きが見える」という直感に合致します。さらに時間融合はオクルージョン下（一時的に隠れた物体）でのリコール（取りこぼしの少なさ）も改善したと報告されています。

空間クロスアテンションの設計でも、ピラー点数（高さ方向のサンプル点）を増やすと精度が向上する一方、計算コストが増えるトレードオフが観察されています。BEV 解像度を上げると遠方・小物体に強くなりますが、メモリと計算が増えます。BEV セグメンテーション（走行可能領域・車線の真上視点での塗り分け）でも、当時の手法（LSS 系）を上回る IoU（領域の重なり率）が報告され、同じ BEV 特徴が検出とセグメンテーションの両方に使えることが示されました。

### メリット・トレードオフ・限界

メリット。深度を明示的に予測しないため、深度教師（LiDAR による奥行きラベル）が不要で、深度誤りによる BEV 崩壊のリスクが小さい。幾何投影に基づくので「どのマスがどのカメラに映るか」が物理的に正しく、可視性マスクで筋の悪い対応を排除できる。時間自己アテンションを再帰的に積むことで、メモリを抑えつつ動画情報を取り込める。BEV という解釈しやすい中間表現が得られ、下流の多タスク（検出・セグメンテーション・予測・計画）に横展開しやすい。

トレードオフと限界。原論文の変形可能アテンションは効率的な実装に専用 CUDA カーネルを要し、移植性が下がります（後続研究では `F.grid_sample` ベースの近似で代替する例もあります）。カメラの内部・外部パラメータが正確であることが前提で、キャリブレーションがずれると投影位置がずれて精度が落ちます。6層・マルチヘッドのスタックは計算とメモリが重く、車載のリアルタイム推論には最適化が要ります（密な 200×200 グリッドを毎フレーム更新するため）。さらに、純粋なカメラ手法ゆえ、濃霧・逆光・夜間など視覚的に厳しい条件では LiDAR 併用手法に劣る場面があります。

未解決の課題として、BEV 解像度と計算コストのトレードオフ、遠方の小物体の検出精度（遠方ほど1マスあたりの画素情報が乏しい）、複数フレーム時間融合をどこまで長くするか（メモリと精度のバランス）、そして再帰的時間融合が長期の誤差蓄積に弱い点などが、現在も研究が続くテーマです。

### 発展トピック・研究の最前線

BEVFormer 以降、BEV を中心に据えた知覚は急速に標準化しました。BEVFusion（Liu ら、ICRA 2023）はカメラ BEV と LiDAR BEV を同じ平面で融合し、センサ間の相補性を活かします。スパース化の流れ（SparseBEV, Sparse4D など）は、密な BEV グリッドをやめてスパースなクエリで直接検出することで計算を減らします。占有予測（occupancy prediction）への拡張では、BEV を3次元のボクセル占有グリッドに格上げし、任意形状の障害物を表現する研究（OccNet, TPVFormer など）が活発です。TPVFormer は BEV の1平面を3つの直交平面に拡張し、高さ方向の表現力を補いました。

研究の最前線では、BEV エンコーダをそのまま end-to-end 自動運転（知覚から計画まで一体学習）の土台に使う方向と、視覚言語モデルと組み合わせてシーンを言語で説明・推論させる方向が並走しています。BEVFormer が確立した「幾何投影でカメラ特徴を統一座標系に集める」という設計思想は、どちらの方向でも基礎部品として生き続けています。

## GAIA-1(生成的世界モデル)

### ひとことで言うと
GAIA-1 は、大量の運転動画を見て「次の瞬間、世界がどう変化するか」を予測できるように学習した AI モデルです。こうした「世界の変化を予測するモデル」を「世界モデル(world model)」と呼びます。世界モデルは、ただ動画を作れるだけでなく、運転シーンの仕組み(車が止まれば後続車も止まる、信号が赤なら誰も進まない、歩行者は車道の端から歩道へ向かう、など)を内部に表現として持つようになります。その内部表現は、レアシーンの生成、運転方策の学習、そして「いつもと違う異常なシーンの検知」など、別の用途に流用できることが分かってきています。

### 直感的な理解
人間のドライバーは、運転中に絶え間なく「この後どうなるか」を頭の中でシミュレーションしています。前の車のブレーキランプが光れば減速を予期し、横断歩道に近づけば歩行者が出てくるかもしれないと身構え、ボールが道路に転がってくれば「子どもが追ってくるかも」と速度を落とします。この「次に何が起きるかの内部モデル」こそが、安全運転の本質です。

問題は、従来の自動運転 AI の多くが「今のカメラ画像 → ハンドル/アクセル操作」という直接の対応だけを学んでいて、この未来予測の常識を持っていなかった点です。学習データに無い状況に出会うと、なぜそう動くべきかの「世界の仕組み」を理解していないために脆く崩れます。

世界モデルのアイデアは、まずこの「世界がどう動くか」を AI に学ばせてしまおう、というものです。一度シーンの力学を理解すれば、(1)危険なレアシーンを安全に大量生成して学習データを増やせる、(2)「もしブレーキを踏んだら」を頭の中で試せるので方策学習が効率化する、(3)予測から外れた観測を「異常」と検知できる、という複数の恩恵が同時に得られます。GAIA-1 は、この世界モデルを運転ドメインで、当時としては極めて大規模に作り上げた代表例です。GAIA-1 という名前は開発元 Wayve(英国の自動運転スタートアップ)が「Generative AI for Autonomy」を意図して付けたものです。

### 基礎: 前提となる概念

#### 世界モデルとは何か
世界モデル(world model)とは、「現在の状態 s_t と自分の行動 a_t を入力すると、次の状態 s_{t+1} を予測する」モデルのことです。数式で書けば、状態遷移 p(s_{t+1} | s_t, a_t) を学習で近似することにあたります。ここで「状態」は生のカメラ画像そのものでもよいですし、画像を圧縮した潜在ベクトル(latent vector、シーンの本質を表す低次元の数値列)でもかまいません。後者を「潜在状態」と呼びます。

#### 自己回帰とトークン
GAIA-1 の心臓部は、大規模言語モデル(GPT 系)と同じ「次トークン予測(next-token prediction)」です。トークン(token)とは、文章を単語に区切るように、データを離散的な小さな部品に分けたものです。「自己回帰(autoregressive)」とは、これまでに生成した列を見て次の1個を予測し、その予測を列に追加してまた次を予測する、という逐次生成のやり方です。GPT が単語を1つずつ紡いで文章を作るのと同じ要領で、GAIA-1 は「動画フレームのトークン」を1つずつ紡いで未来の映像を作ります。

#### 拡散モデル
拡散モデル(diffusion model)は、ノイズだらけの画像から少しずつノイズを取り除いて、きれいな画像を生成する手法です。GAIA-1 では、世界モデルが予測した抽象的なトークン列を、人間が見られる動画ピクセルに「戻す」デコーダとして使われます。

#### 表現学習という観点
世界モデルが優れているのは、「未来を予測する」という難しい課題を解かせると、その副産物として「シーンを理解した良い内部表現(representation)」が得られる点です。これは表現学習(representation learning)の中心的な仮説で、「予測こそ理解の証」という考え方に立脚しています。

### 仕組みを詳しく

GAIA-1 の処理は大きく3段階に分かれます。順を追って見ます。

#### ステップ1: 画像・行動・テキストを「トークン」に変換する
入力(動画フレーム、車の操作、文章による指示)をすべてトークンの列に変換し、共通フォーマットに揃えます。

- 画像トークン化: 1枚の運転フレーム(GAIA-1 では概ね 288×512 ピクセルの RGB 画像)を、画像専用の圧縮器に通して数百個の整数トークンに圧縮します。GAIA-1 は離散オートエンコーダ(DALL-E や VQ-GAN 系の発想)を使い、1フレームを 576 個のトークンに変換します。各トークンは「あらかじめ用意した辞書(コードブック)の 8192 種類のうちどれか」を表す整数です。ピクセル数で言えば 288×512×3 ≈ 44 万個の数値だったものが、576 個の整数に圧縮されます。重要なのは、この圧縮器がただ縮めるだけでなく「運転シーンとして意味のある特徴」を残すように学習されている点です。
- 行動トークン: ハンドル角・速度などの連続値も、量子化(離散化)してトークンにします。
- テキストトークン: 「右に曲がれ」「もっと混雑させて」のような言語指示も単語トークンに変換します。

これで、動画・行動・言葉という異なる種類の情報が、すべて同じ「トークンの列」になり、1つの Transformer で一緒に扱えるようになります。

#### ステップ2: 世界モデル本体(次トークン予測)
ここが中核です。GAIA-1 の世界モデルは、巨大な自己回帰 Transformer で、「これまでのトークン列を見て、次に来るトークンを当てる」よう学習します。

具体的なイメージ: 過去 N フレーム分の画像トークン列と行動トークンを入力し、「次のフレームの 576 トークンはこれだろう」と1個ずつ予測していきます。系列(コンテキスト)は数千トークンに及び、各位置で「次は 8192 種類のどれか」という確率分布を出力します。

```
入力コンテキスト(時刻 t まで):
[txt][txt] [act_t-2] [img_t-2(576トークン)] [act_t-1] [img_t-1(576)] [act_t] ...
                                                            │
                                                            ▼
       次トークンの確率分布 (語彙8192次元) → サンプリング → 次トークン
                                                            │
                                              576回繰り返して img_{t+1} を生成
```

学習の規模感として、GAIA-1 の世界モデル本体は約 65 億(報告により世界モデル単体で約 65 億、構成全体で約 90 億規模とも)パラメータで、英国の都市運転動画 約 4700 時間で訓練されたと報告されています。これは「自動運転版の GPT」と言える規模です。

確率分布を出すという設計が効いてきます。同じ「現在」から、サンプリングのたびに「直進する未来」「車線変更する未来」「前車が急ブレーキする未来」と分岐したシナリオを複数生成できます。これは決定論的な予測では得られない性質で、レアな危険シナリオの生成に直結します。

#### ステップ3: 動画デコーダ(見える形に戻す)
予測したトークン列を、人間が見られる動画に戻すために、拡散モデルベースの動画デコーダを通します。トークンは抽象的な記号なので、これをデコードして滑らかで時間的に一貫した動画フレームを再構成します。GAIA-1 はここで時間的整合性(フレーム間でちらつかない)を重視した設計を採っています。

#### なぜこの3段なのか
ピクセルを直接 Transformer で予測しようとすると、系列が途方もなく長くなり計算が破綻します。そこで「トークンに圧縮 → トークン列で予測 → ピクセルに復元」と分けることで、予測の負担を抽象空間に移しています。これは「予測は意味のレベルで、見た目の復元は別のデコーダで」という役割分担で、後述する JEPA や MILE と共通する設計思想です。

### 手法の系譜と主要論文

#### World Models(Ha & Schmidhuber, 2018, arXiv:1803.10122)
世界モデルの原点的論文です。提案内容: ゲーム環境の観測を VAE(変分オートエンコーダ)で潜在ベクトルに圧縮し、RNN(再帰型ニューラルネット)で潜在空間の未来を予測する世界モデルを作り、その内部表現の上で小さな方策(コントローラ)を進化戦略で学習する。動機: 環境のシミュレータを AI の頭の中に持てば、実環境を動かさず「夢の中(内部シミュレーション)」で方策を訓練できる。新規性: 「予測モデルの内部表現が、方策学習に有用な状態抽象を与える」ことを示した点。

#### Dreamer / DreamerV3(Hafner et al., 2020–2023)
潜在世界モデルを使った強化学習を体系化した系譜です。潜在空間で「想像上のロールアウト」を展開し、その中で方策を学習することで、実環境とのやり取りを大幅に節約できることを示しました。DreamerV3 は単一のハイパーパラメータ設定で多様なタスクを解き、世界モデル系強化学習の実用性を高めました。

#### MILE: Model-Based Imitation Learning for Urban Driving(Hu et al., Wayve, NeurIPS 2022, arXiv:2210.07729)
GAIA-1 の直接の前身です。提案内容: カメラ画像から運転シーンの潜在状態を推定し、その潜在空間で未来を予測する世界モデルを学習しつつ、同時に運転方策も学ぶ。動機: ピクセルを直接予測するのは影や質感まで再現させられて非効率なので、潜在空間で予測したほうが運転に必要な意味だけを捉えられる。新規性: 実世界の人間運転データから、世界モデルと方策を end-to-end で同時学習した点。CARLA シミュレータの閉ループ評価で当時の模倣学習を上回りました。

#### I-JEPA / JEPA(Assran et al., Meta, CVPR 2023, arXiv:2301.08243)
LeCun が提唱する設計思想を画像で具体化したものです。提案内容: ピクセルを再構成するのではなく、学習済みの特徴空間(feature space)で「隠した部分の特徴」を予測する。動機: ピクセル再構成は無駄な細部(テクスチャ、ノイズ)まで当てさせられるが、抽象的な特徴の予測なら本質的な構造だけを学べる。新規性: 自己教師あり学習で、生成を伴わずに高品質な表現を得る枠組み。GAIA-1 がピクセル空間で巨大に予測するのとは対照的な「特徴空間予測」の流れで、軽量な世界モデルの設計指針として重要です。

#### GAIA-1(Hu et al., Wayve, 2023, arXiv:2309.17080)
提案内容: 画像・行動・テキストを共通トークンに変換し、自己回帰 Transformer で次トークンを予測する世界モデル + 拡散デコーダ。動機: 言語モデルで成功した「次トークン予測でスケールさせる」レシピを運転動画に適用すれば、明示的に教えなくても運転の常識(交通ルール、車両の動き、歩行者の振る舞い)が創発するという仮説。

#### GAIA-2(Wayve, 2025)
GAIA-1 の後継で、潜在拡散ベースのアーキテクチャに移行し、マルチカメラ・条件付き生成(天候・時刻・他車配置などの細かい制御)・実時間に近い生成効率を強化したと報告されています。「自己回帰 Transformer から潜在拡散へ」という、近年の動画生成全体の潮流(SVD、Sora 系)とも合致する進化です。

### 論文の実験結果(定量データ)

GAIA-1 は純粋な動画生成器であり、検出精度のような単一の数値で評価しにくい性質があります。論文と報告で示された主な観察は次の通りです。

- スケーリング則の観測: モデルパラメータと学習データを増やすほど、世界モデルの学習損失(次トークン予測の交差エントロピー)が予測可能なべき乗則で下がり続けることが示されました。これは「言語モデルで効いたスケーリングが運転動画でも効く」ことを意味し、GAIA-1 の最大の主張の一つです。べき乗則とは「データ量を10倍にすると損失が一定割合で下がる」という規則性で、これが成り立つと「どれだけ投資すればどれだけ良くなるか」が事前に見積もれます。
- 生成の多様性と一貫性: 数秒の現実映像を与えると、その後に続く数十秒の運転シナリオを複数パターン生成でき、車線維持・分岐・他車の割り込みなど、交通として妥当な振る舞いが創発したと報告されています。これは明示的に交通ルールを教えていないにもかかわらず起きる点が重要です。
- 条件付き制御: テキスト(「夜にして」「歩行者を増やして」)や行動(自車の操作)で生成を条件付けでき、入力指示に沿った映像が得られたと報告されています。
- 創発的理解の例: 信号・他車・歩行者の挙動が物理的・社会的に妥当に振る舞う場面が観察され、「常識の創発」の定性的証拠とされています。

評価指標について補足します。動画生成の品質は FVD(Fréchet Video Distance、生成動画と実動画の特徴分布の近さを測る指標。低いほど良い)などで測るのが一般的ですが、GAIA-1 の主眼はベンチマーク数値より「スケーリングする世界モデルが運転の常識を獲得しうる」ことの実証にありました。後続の世界モデル研究(DriveDreamer、Vista、GenAD など)では、生成品質や下流タスクへの転移を FVD・検出精度・計画品質で定量比較する流れが強まっています。

### メリット・トレードオフ・限界

メリット
- レアで危険なシーンを安全に大量生成でき、学習データ不足を補える。
- 「次を予測する」だけで交通の常識的な振る舞いが創発し、明示的なルール記述が不要。
- 得られた内部表現を、軌跡予測・異常検知・方策学習など複数タスクに転用できる。
- スケーリング則が効き、モデルとデータを増やすほど品質が上がる見通しが立つ。

トレードオフと限界
- 計算コストが非常に大きい。数十億パラメータの自己回帰 Transformer に拡散デコーダが乗り、学習にも推論にも大量の GPU を要する。自己回帰生成はトークンを1個ずつ作るため、リアルタイム運転には遅すぎる。
- 生成された未来は確率的で、物理的・法的に必ずしも正しいとは限らない(車が壁をすり抜ける、物体が消えるなどの破綻が起こりうる)。安全用途には別途検証が要る。
- 動画を作るモデルであって、行動方策(どう操作すべきか)を直接出すわけではない。下流に別のプランナが必要。
- ピクセル予測は無駄な細部まで再現させられるため、運転に必要な意味だけを学ぶには非効率という批判がある。これが JEPA/MILE 系の「特徴空間で予測する」設計の動機になっている。
- 評価が難しい。「良い世界モデルとは何か」を測る統一指標がまだ確立しておらず、定量比較が困難なのが研究上の未解決課題。

### 発展トピック・研究の最前線

- 特徴空間世界モデル: ピクセルではなく、凍結した(学習済みで固定した)エンコーダの特徴空間で未来を予測する設計が広がっています。これは表現崩壊(representation collapse、全部同じ値になって損失だけ下がる退化現象)を避けるため、予測対象側に固定エンコーダ(BYOL/JEPA 系のいわゆる target network)を置くのが定石です。軽量で、運転の意思決定に必要な意味だけを学べる利点があります。
- 潜在拡散世界モデル: 自己回帰から潜在拡散へ移行する流れ(GAIA-2、Vista、GenAD など)。生成品質と効率の両立を狙います。
- 異常検知への応用: 世界モデルが「ふつうの運転ならこう変化するはず」という予測を持つと、実際の観測がその予測から大きく外れたとき(予測誤差が異常に大きいとき)、それを「いつもと違う異常なシーン」として検出できます。世界モデルがシーンの力学を内部に持つことの自然な副産物です。閉ループでの安全監視やコーナーケース発掘に使われ始めています。
- 閉ループシミュレーションへの展開: 生成した世界モデルを「ニューラルシミュレータ」として使い、その中で運転方策を訓練・評価する研究(MILE、Dreamer 系の発想の運転版)。実車を動かさずに反実仮想(「もしあの時ブレーキを踏んでいたら」)を試せる点が魅力です。

## MILE(運転向け潜在世界モデル)

### ひとことで言うと

MILE は「画素(ピクセル)を予測する世界モデルより、意味を凝縮した潜在空間で未来を予測する世界モデルの方が運転には強い」ことを、実際の都市運転データで実証した研究です。世界モデルを運転に持ち込むとき「未来の何を予測させるべきか」という問いに、「ピクセルではなく潜在を」という明確な答えを与えた点で、後続の運転向け世界モデル設計の重要な拠り所になっています。

### 直感的な理解

自動運転モデルに「未来予測」を学ばせると賢くなる、という発想は古くからあります。問題は「未来の何を予測させるか」です。

素朴には「未来の映像(ピクセル)」を予測させたくなります。しかしピクセル予測には深刻な無駄があります。未来の正確なピクセルを当てるには、影の位置、路面のテクスチャ、木の葉の揺れ、画像圧縮のノイズといった、運転判断にまったく関係ない細部まで再現しなければなりません。モデルの限られた学習能力がこうした「どうでもいい細部」に吸い取られ、肝心の「車や人がどう動くか」という本質を学べなくなります。

ここで「潜在(latent)」という考え方が効きます。潜在空間とは、生のデータ(ピクセル)を、意味が凝縮された少数の数値ベクトルに変換した「内部表現の世界」です。潜在世界モデル(latent world model)は、このコンパクトな潜在空間の中で未来を予測します。ピクセルではなく潜在で予測すれば、無関係な細部を捨て、シーンの意味的な進み方だけを学べます。MILE が答えたのは「この潜在世界モデルは、運転というシビアな実タスクで、本当にピクセル予測ベースラインより良いのか」という実証的な問いでした。当時は理屈としては期待されていたものの、運転ドメインでの明確な実証が不足していました。

### 基礎: 前提となる概念

模倣学習(imitation learning): お手本(ここでは人間の運転ログ)を真似るように学習する枠組みです。「この状況ではこう操作した」という人間の行動を正解として、同じ状況で同じ操作を出すよう学びます。報酬を試行錯誤で探す強化学習と違い、良質なお手本があれば効率よく学べます。

潜在状態 z: 各時刻の世界の状態を表す、意味の凝縮されたベクトルです。生のピクセルが数十万〜数百万次元なのに対し、潜在状態は数百〜数千次元で済みます。

遷移モデル(transition model): 「いまの潜在状態 z_t と行動 a_t から、次の潜在状態 z_{t+1} を予測する」ネットワークです。これがあると、実際の映像を見なくても潜在の中で時間を先に進められます。

方策(policy): 状態から行動を決める関数です。MILE では潜在状態 z_t を受け取って運転操作 a_t を出します。

imagination(想像ロールアウト): 遷移モデルで潜在状態を先へ先へと進めながら方策を回し、実際の観測なしに「頭の中で運転をシミュレートする」ことです。Dreamer 系の強化学習で確立された概念で、MILE はこれを運転・模倣学習に持ち込みました。

model-based(モデルベース): 「世界の進み方を予測するモデル(世界モデル)を内部に持つ」という意味です。MILE は Model-based Imitation LEarning の略で、世界モデルを持つ模倣学習を指します。

### 仕組みを詳しく

MILE は、人間の運転データから、世界の進み方(世界モデル)と運転方策を同時に学びます。大まかな構成は次の通りです。

```
カメラ画像列        → エンコーダ → 潜在状態 z_t(意味の凝縮ベクトル)
潜在状態 z_t + 行動 a_t → 遷移モデル → 次の潜在状態 z_{t+1} を予測
潜在状態 z_t        → 方策(policy) → 行動 a_t(運転操作)
```

ポイントは2つです。

第一に、未来予測を潜在状態 `z` の空間で行います(`z_{t+1}` を予測)。ピクセルそのものは予測しません。これにより、影やテクスチャではなく「シーンがどう展開するか」という意味的ダイナミクスに学習を集中させます。MILE は学習時に補助的な BEV セグメンテーション(鳥瞰図での意味分割。各地点が道路・車・歩行者などどのクラスかを塗り分ける)を再構成する設計を取り、潜在に運転へ有用な情報が確実に乗るよう誘導します。ここがポイントで、純粋に「次の z」を当てるだけだと潜在が運転に無関係な方向へ流れかねないので、BEV セグメンテーションという「運転に意味のある足場」を補助タスクで与えています。

第二に、世界モデルと方策を一緒に学ぶため、学習後に「潜在空間の中だけで運転を想像」できます。実際の映像なしに、潜在状態を遷移モデルで先に進めながら方策を回すことで、頭の中でドライブをシミュレートできます。これは Dreamer 系の「潜在の中で夢を見る」発想を運転に持ち込んだものです。

数値感覚として、ピクセル予測モデルは出力が「画像サイズ × 3チャンネル(数十万〜数百万次元)」と巨大ですが、潜在世界モデルの予測対象は「数百〜数千次元の潜在ベクトル」で済みます。予測対象が桁違いに小さく、本質に集中できる、というのが核心です。

MILE は確率的潜在変数モデル(変分推論)として定式化されており、未来の不確実性(同じ現在から複数の未来があり得ること)を、潜在状態を確率分布として扱うことで表現します。これは決定論的な MSE 予測が「複数未来の平均」を当てて出力をぼやかしてしまう問題への対処でもあります。

### 手法の系譜と主要論文

MILE は2つの系譜の合流点にあります。

一つは世界モデルと imagination の系譜です。Ha & Schmidhuber の "World Models"(2018, arXiv:1803.10122)が「観測を潜在に圧縮し、潜在の中で先読みして方策を学ぶ」枠組みを示しました。Hafner らの Dreamer(2020, arXiv:1912.01603)、DreamerV2(arXiv:2010.02193)、DreamerV3(2023, arXiv:2301.04104)は、潜在世界モデルの中で想像ロールアウトして強化学習する手法を洗練させ、Atari からロボット制御まで幅広いタスクで強力な性能を示しました。MILE の潜在内 imagination はこの系統を運転・模倣学習へ移植したものです。

もう一つは運転向けエンドツーエンド学習の系譜です。MILE(Hu et al., Wayve, NeurIPS 2022, arXiv:2210.07729, "Model-Based Imitation Learning for Urban Driving")は、都市運転を「潜在世界モデルと方策を同時学習する model-based 模倣学習」として定式化しました。後続の生成的世界モデル(同じ Wayve の GAIA-1 など)や、表現空間で未来を予測する自己教師あり学習(I-JEPA, Assran et al. 2023, arXiv:2301.08243)も、「潜在/表現空間で予測する」という同じ哲学を共有します。違いは、MILE が完全な model-based 模倣学習(潜在内 imagination で方策まで学ぶ)であるのに対し、JEPA 系は未来特徴予測を表現学習の自己教師あり信号として使う点です。

### 論文の実験結果(定量データ)

MILE(arXiv:2210.07729)の中心的な結果は CARLA(運転シミュレータ)での評価です。CARLA の標準的な評価では、運転スコア(route completion = ルートの完走割合と、infraction = 信号無視・衝突などの違反ペナルティを統合した総合指標、大きいほど良い)が使われます。MILE は当時の最先端手法を上回る運転スコアを報告し、特に新しい街(学習に使っていない town)への汎化で差を広げました。学習に使っていない環境でも崩れにくいことは、潜在に本質的なダイナミクスを学べている証拠とされます。

このデータセットで、どの指標が、どれだけ改善したかを噛み砕くと、route completion(どこまで走り切れたか)と infraction penalty(どれだけ違反せず走れたか)の両面でベースラインを上回り、その積に近い総合運転スコアで明確な差を示した、ということです。運転スコアは「ただ走るだけ」でなく「安全に走り切る」を同時に測るため、両方を上げないと伸びません。

最も重要なのはアブレーションです。MILE は「潜在世界モデルを使う/使わない(ピクセル予測や単純な模倣に置き換える)」を比較し、潜在世界モデルを入れた構成が運転性能で上回ることを示しました。これが「未来をピクセルではなく潜在で予測すべき」という主張の直接の証拠です。さらに、学習後に潜在空間内だけで運転をロールアウト(imagination)させても妥当な運転挙動を生成できることを示し、世界モデルが単なる補助ではなく、実際に「先読み」として機能していることを定性的・定量的に裏付けました。

ただし注意点として、これらの結果は主に CARLA という(実世界より単純化された)シミュレータ上のもので、実車・実世界の複雑さへの一般化は別途検証が必要です。論文自身もこの限界に触れています。

### メリット・トレードオフ・限界

メリット(潜在世界モデル一般、MILE が運転で実証):
- 無関係な細部(影・テクスチャ・ノイズ)を捨て、運転に本質的なダイナミクスを学べる。
- 予測対象が小さく、計算とメモリが軽い(ピクセルの数十万〜数百万次元に対し潜在は数百〜数千次元)。
- 潜在空間内で「想像ドライブ」ができ、計画やデータ拡張、稀なシーンの先読みに使える。

トレードオフ・限界:
- 潜在空間の質に全体性能が依存し、潜在が崩れると予測も方策も同時に劣化する。
- 潜在は人間に解釈しにくく、何を根拠に予測・判断したかが見えにくい(BEV セグメンテーション補助はこの緩和策)。
- 実証が主に CARLA シミュレータで、実車・実世界の複雑さでの一般化は別途検証が要る。
- model-based 模倣学習は構成が複雑で、未来特徴予測を補助タスクとして使う軽い使い方より、実装・学習が重い。

未解決の課題は、シミュレータと実世界のギャップ(sim-to-real)と、潜在世界モデルのスケール(より大規模な実走行データで MILE 流の同時学習がどこまで伸びるか)です。後続の大規模生成世界モデルはこの方向の探索でもあります。

### 発展トピック・研究の最前線

潜在世界モデルから生成世界モデルへ: MILE の「潜在で予測」を、より大規模・生成的に推し進めたのが GAIA 系の運転動画生成世界モデルです。行動やテキストで条件づけて長尺の未来を生成し、合成データやシミュレータ代替に使う方向が活発です。

補助タスクの使い分け: MILE のように世界モデルと方策を同時に学ぶ重い構成と、未来予測を表現学習の補助信号として軽く使う構成は、目的に応じて選び分けられます。前者は「潜在内で運転を想像する」能力まで得られる一方、後者は既存パイプラインに足しやすく、軌道計画の特徴を底上げする目的に向きます。

不確実性のモデリング: MILE が確率的潜在で扱った「未来の多義性」は、世界モデル全般の核心課題です。決定論予測のぼやけを避けるため、確率的潜在・拡散モデル・自己回帰生成など、複数未来を表現できる定式化が研究の最前線になっています。

## PETR（論文）

### ひとことで言うと

PETR は、複数台のカメラ画像から3次元の物体を検出する手法です（Liu ら、ECCV 2022）。核心のアイデアは「画像の特徴に、その特徴が3次元空間のどのあたりに対応するかという位置情報を埋め込む（染み込ませる）」ことです。BEVFormer のように真上から見た地図（BEV）を明示的に組み立てるのではなく、画像特徴そのものに3次元座標の手がかりを足し込み、物体を表すクエリがその特徴に注目して3Dボックスを出力します。「BEV 地図を作らずに、画像特徴に3次元の意味を持たせる」シンプルさが魅力です。

### 直感的な理解

3次元物体検出を Transformer で解く流れを思い出してください。DETR（Carion ら、ECCV 2020）という手法は、画像を Transformer に通し、「物体クエリ」と呼ぶ学習可能なベクトル（例えば100個や900個）が画像特徴に注目して、それぞれが1つの物体（位置・大きさ・クラス）を出力します。これは2次元の物体検出ではうまくいきました。

問題は「これを3次元・多視点にどう拡張するか」です。素朴に考えると、BEVFormer のように地図を作って3次元基準点を画像に投影してサンプルする、という幾何的に複雑な処理が必要になります。あるいは DETR3D（後述）のように、クエリの3次元参照点を毎回画像に投影して、その1点まわりの特徴を引いてくる、という方法もあります。どちらも「投影してサンプルする」という局所的な操作が中心です。

PETR の発想はもっと単純で、しかも逆向きです。DETR では画像特徴に「2次元の位置エンコーディング（このピクセルは画像の左上か右下か、という位置情報）」を足してクエリが注目していました。ならば「3次元の位置エンコーディング（この特徴は3次元空間のどの直線上にあるか）」を足せば、同じ枠組みのまま3次元検出ができるはずだ、というわけです。地図を作る代わりに、座標を特徴に足し算で混ぜ込む。一度埋め込んでしまえば、あとはクエリが全画像特徴を自由に見渡せる（グローバルアテンション）。これが PETR（Position Embedding Transformation、位置埋め込み変換）の名前の由来です。

### 基礎: 前提となる概念

位置エンコーディング（positional encoding）。Transformer はもともと入力の順序や位置を区別しません（アテンションは集合演算で、並べ替えても結果が変わらない）。そこで「これは何番目の要素か」「画像のどの位置か」を表すベクトルを特徴に足して、位置の手がかりを与えます。Vaswani ら（NeurIPS 2017, arXiv:1706.03762）が言語処理で導入し、画像にも転用されています。PETR の3D位置エンコーディングはこの考えを3次元座標に拡張したものです。

クロスアテンション（cross-attention）。アテンションのうち、クエリ（問い合わせる側）とキー・バリュー（参照される側）が別物の場合を指します。DETR や PETR では「物体クエリ」が「画像特徴」にクロスアテンションします。クエリが「私が担当する物体はどこ?」と問い、画像特徴のどこを重視するかを重みで決め、その重み付き平均を取り込みます。

視錐台（frustum）。1台のカメラが見ている空間は、レンズから手前は狭く奥は広がる四角錐（の一部）の形をしています。これを視錐台と呼びます。画像の1ピクセルは「カメラから伸びる1本の視線（直線）」に対応し、その直線上のどこに物体があるかは1枚の画像だけでは決まりません。PETR はこの直線上に複数の奥行きの点を並べて3次元化します。

カメラの外部パラメータ。カメラが車のどこにどの向きで付いているかを表す行列（回転と並進）です。視錐台の点をカメラ座標から自車座標（車を基準とした3次元座標系）に変換するのに使います。

二部マッチング損失（bipartite matching loss）。DETR 系の中心的な学習方法です。クエリの出力（予測ボックス集合）と正解ボックス集合を、最もコストが小さくなるように1対1で対応づけ（ハンガリアン法による最適割当）、対応がついたペアだけで損失を計算します。これにより「どのクエリがどの物体を担当するか」を後処理（NMS、重複除去）なしで学習できます。

### 仕組みを詳しく

PETR の処理を段階的に分解します。

#### 1. カメラ視錐台から3D点を作る

各カメラ画像の各特徴位置に対し、奥行きを D 段階に並べた点の集まり（視錐台メッシュ）を作ります。例えば特徴マップが H×W、奥行きを D 段階（論文では 64）とすると、`[H, W, D, 3]` という3次元点群ができます（最後の3は X, Y, Z 座標）。これらをカメラの内部・外部パラメータで自車座標系の3次元点に変換します。つまり「画像のこのピクセルは、自車から見て3次元空間のこの直線上のこれらの点に対応する」という情報を持ちます。

#### 2. 3D点を位置エンコーディングに変換

その3次元座標（1つの特徴位置あたり D×3 個の数値）を小さな MLP（多層パーセプトロン、数層の全結合ネットワーク）に通して、画像特徴と同じチャンネル数（例えば256）のベクトルに変換します。これが「3D 位置エンコーディング（3D Position Embedding, 3D PE）」です。1本の視線上の複数点の座標をまとめて1ベクトルに圧縮するので、各特徴は「自分がどの視線（3次元の直線）の上にあるか」を表す符号を持つことになります。

#### 3. 画像特徴に足す

画像をバックボーン（ResNet 等）に通して得た2次元特徴に、この3D位置エンコーディングを足し合わせます。

```
画像特徴 (2D)  [B, V, C, H, W]
        +
3D位置エンコーディング (frustum → MLP)  [B, V, C, H, W]
        =
3D を意識した特徴（これに物体クエリが注目する）
```

ここで B はバッチ、V はカメラ台数、C はチャンネル数です。足し算した後の各特徴ベクトルは「自分が3次元空間のどの直線上にあるか」という情報を内部に持つようになります。重要なのは、これが各カメラの特徴に対して一度だけ行う前処理だという点です。投影サンプリングのようにクエリごとに毎回計算し直す必要がありません。

#### 4. 物体クエリがクロスアテンション

学習可能な物体クエリ（論文では900個前後）を用意します。各クエリには3次元の参照点（アンカー）が紐づいており、その点も同じエンコーディングを通して3D位置情報を持ちます。クエリは全カメラの「3D意識つき特徴」にクロスアテンションし、「自分の担当する3次元位置の物体はどこ?」と問い合わせて特徴を集めます。クエリと特徴の双方に同じ3次元座標系の符号が乗っているため、アテンションの内積が自然と「3次元的に近いものほど高い相性」を反映します。最後に各クエリが3Dボックス（中心位置・サイズ・向き・速度・クラス）を出力し、二部マッチング損失で学習します。

BEVFormer との決定的な違いは、BEVFormer が BEV グリッドという明示的な中間表現を作るのに対し、PETR は中間 BEV を作らず、3次元情報を画像特徴に埋め込んだまま直接検出する点です。明示的なサンプリング処理がいらないぶん、構造がすっきりします。DETR3D との違いは、DETR3D が「クエリの1参照点を画像に投影して、その近傍だけ特徴を引く」のに対し、PETR は「クエリが全画像特徴をグローバルに見る」点で、投影の数値誤差の影響を受けにくくなります。

#### 続編 PETRv2

PETRv2（Liu ら、ICCV 2023、arXiv:2206.01256）は2点を拡張しました。第1に時間融合。過去フレームの特徴を自車運動で位置合わせし、現在のクエリが過去も参照できるようにしました。具体的には、過去フレームの3D位置エンコーディングを ego-motion で現在座標系に変換してから足し込む、という座標整合を入れます。第2に多タスク化。BEV セグメンテーション（車線や走行可能領域を真上視点で塗り分ける）などを同じ枠組みで解けるようにしました。

### 手法の系譜と主要論文

土台。DETR（Carion ら、ECCV 2020、arXiv:2005.12872）が「物体検出をクエリで解く（NMS 不要の集合予測）」枠組みを確立し、Vaswani ら（NeurIPS 2017）のアテンションと位置エンコーディングがその基盤です。3次元検出では FCOS3D（Wang ら、2021）のような単眼検出が先行し、続いて DETR3D（Wang ら、CoRL 2021、arXiv:2110.06922）が「クエリの3次元参照点を画像に投影して特徴をサンプルする」方式でクエリベース多視点検出の先鞭をつけました。DETR3D は PETR の直接の比較対象です。

PETR の新規性。Liu, Wang, Zhang ら、ECCV 2022、arXiv:2203.05625。DETR3D が「クエリを画像へ投影してサンプルする（局所的・スパース）」のに対し、PETR は逆に「画像特徴に3次元位置を埋め込む（前処理・グローバル）」ことで、明示的な投影サンプリングを不要にしました。これによりグローバルなアテンション（全特徴を見る）が可能になり、投影の数値誤差の影響を受けにくくなりました。設計が DETR にほぼ忠実なので、検出ヘッドや二部マッチング損失をそのまま流用できる手軽さも利点です。

その後。PETRv2（ICCV 2023）で時間融合と多タスク化。StreamPETR（Wang ら、ICCV 2023、arXiv:2303.11926）は、時間情報を「object-centric」なクエリに乗せてフレーム間で伝播させる効率的な時系列モデルで、過去の全フレームを毎回処理せず、検出した物体クエリだけを繰り越すことで長時間の文脈を軽量に扱えるようにしました。これらクエリベース・スパース系は、密な BEV グリッドを持たないぶん計算が軽く、車載向けに有利という流れを作りました。

### 論文の実験結果（定量データ）

評価は nuScenes データセットで行われました。指標は mAP（検出の平均適合度、3次元中心点距離による一致判定、高いほど良い、0〜1）と NDS（位置・サイズ・向き・速度・属性の5誤差を mAP と総合した nuScenes 専用スコア、0〜1）です。

報告値として、PETR はテストセットで NDS 約0.504、mAP 約0.441（VoVNet-V2-99 バックボーン、大解像度入力）を達成しました。先行の DETR3D（NDS 約0.479、mAP 約0.412）を上回り、BEVFormer と同等水準に並びました。重要なのは、PETR がこれを BEV グリッド構築なしのシンプルな構造で達成した点です。ResNet-50 など軽量バックボーンでの設定でも DETR3D を一貫して上回る傾向が報告されています。

PETRv2 では時間融合の追加により、特に速度推定の誤差（mAVE）が大きく下がり、NDS が単フレーム版から数ポイント向上したと報告されています（テストセットで NDS 約0.58 前後の水準に到達）。これは「1フレームでは速度が分からないが、過去を参照すると動きが見える」という時間融合共通の効果です。BEV セグメンテーションでも当時の手法と競合する IoU を示しました。

アブレーションでは、3D位置エンコーディングを外す（2D PE のみにする）と精度が大きく低下することが示されました。これは PETR の中心アイデアが効いている証拠です。また視錐台の奥行きサンプル数 D を増やすと精度が上がる一方で計算コストが増えるトレードオフ、座標を正規化する範囲（3D 空間のどこまでをカバーするか）の取り方が精度に影響することなどが報告されています。

### メリット・トレードオフ・限界

メリット。BEV グリッドを明示的に組み立てないため構造がシンプルで実装しやすい。DETR の枠組みをほぼそのまま使えるので、検出ヘッドや損失（ハンガリアンマッチングによる二部マッチング損失）を流用できる。3次元位置を特徴に埋め込むだけなので、複雑なサンプリング処理がいらず、投影の数値誤差にも比較的頑健です。グローバルアテンションにより、複数カメラにまたがって映る物体の手がかりを自然に統合できます。

トレードオフと限界。3次元位置を足し算で暗黙に埋め込むため、BEVFormer のような幾何的に明確な BEV 地図表現は直接は得られません。車線地図や占有グリッドのように「真上視点の地図そのもの」が欲しい下流タスクには別途工夫が要ります（PETRv2 が BEV セグメンテーションを足したのはこのため）。視錐台の3次元化にはカメラの外部・内部パラメータが必要で、キャリブレーションの精度に依存します。また初期の PETR は単フレームで、動体の速度推定が弱い（時間融合は PETRv2 以降）という限界がありました。グローバルアテンションは全特徴を見るため、画像解像度やカメラ台数が増えるとメモリと計算が膨らみ、ここが車載リアルタイム化の壁になります（StreamPETR の object-centric 化はこの緩和策です）。

### 発展トピック・研究の最前線

クエリベース3D検出は、BEVFormer 系の「密な BEV」とは別の系統として発展を続けています。Sparse4D（Lin ら）や StreamPETR は時間情報をクエリに蓄積し、長い時間文脈を効率的に扱う方向です。これらスパース系は計算量がクエリ数に比例し、画像解像度に対して線形以下でしか伸びないため、車載のリアルタイム推論で有利とされます。StreamPETR は ResNet-50 という軽量設定でも当時の重い BEV 手法に匹敵する精度を、はるかに高いフレームレートで達成したと報告され、「軽量・高速でも高精度」という流れを決定づけました。

研究の最前線では、PETR 型の位置埋め込みの発想を、検出だけでなく end-to-end 計画や占有予測に取り込む試みが進んでいます。「明示的な BEV を作るか、特徴に座標を埋め込むか」という設計の二項対立は今も続いており、密な地図表現が必要なタスク（計画・占有）では BEV 系を、検出主体で計算予算が厳しい車載では PETR/Sparse 系を、というようにタスクや計算予算に応じて使い分ける、というのが現在の共通理解です。

## UniAD（論文）

### ひとことで言うと

UniAD は、自動運転の「認識（まわりに何があるか）」「予測（他車がどう動くか）」「計画（自車がどう走るか）」を、別々のモジュールに分けず1つのネットワークで一気通貫に解いた手法です（Hu ら、CVPR 2023、Best Paper 受賞）。各段階を「クエリ」と呼ぶ特徴ベクトルでつなぎ、最終目的である「計画」が良くなるようにネットワーク全体を同時に学習します。この「計画指向（planning-oriented）」という設計思想が核です。

### 直感的な理解

従来の自動運転ソフトは、機能ごとに独立したモジュールを直列につなぐ構成が一般的でした。

```
カメラ → 物体検出 → 追跡 → 地図認識 → 動き予測 → 占有予測 → 計画 → 制御
```

これは各段を個別に開発・検証できる利点があります。検出担当チーム、予測担当チーム、と分業できますし、どこで間違えたかも追いやすい。ところが2つの根深い問題があります。

第1に情報の損失です。各モジュールは前段の「最終出力」だけを受け取ります。例えば検出は「車が座標 (x, y) に1台」というボックスのリストを出し、予測はそのリストだけを見ます。しかし検出が「ここに何かいる気がするが確信が持てない」と感じた微妙な手がかりは、ボックスに確定する段階で捨てられ、下流には伝わりません。確率や特徴の豊かさが、離散的なボックス座標に潰された瞬間に失われるのです。

第2に目的のずれ（累積誤差）です。検出は「検出精度」を、予測は「予測精度」を、それぞれ局所的に最適化します。しかし最終的に欲しいのは「安全な運転計画」です。検出の小さな誤差が予測へ、予測の誤差が計画へと積み重なり、各段が頑張っても全体として最適にならない、ということが起きます。例えば「検出精度を0.1%上げる改良」が、計画の安全性には何も寄与しない、ということが普通に起こります。

逆の極端として「画像から直接ハンドル操作を出す」完全 end-to-end もありますが、中間で何が起きているか見えず、安全保証やデバッグが困難です。UniAD はこの中間を取りました。各タスクのモジュールは残して中間表現を人間に見えるようにしつつ、それらを微分可能なクエリでつなぎ、計画目的に向けて全体を同時学習する。解釈性と end-to-end 最適化の両取りを狙ったわけです。

### 基礎: 前提となる概念

クエリ（query）でつなぐとは何か。Transformer の文脈で、クエリは「情報を集めるための学習可能なベクトル」です。UniAD では、検出した各物体が1本のクエリ（track クエリ）になり、そのクエリがそのまま予測モジュールへ渡されます。ボックス座標のような確定した離散データに潰さず、豊かな特徴ベクトル（例えば256次元）のまま渡すのが要点で、これが情報損失を抑えます。さらにクエリは微分可能なので、計画段階で生じた損失の勾配が、クエリを通って検出段階まで遡って流れます。これが「計画のために認識を学ぶ」を可能にします。

BEV エンコーダ。UniAD は前段に BEVFormer（[BEVFormer（論文）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/16_related-work-misc)）を使い、多視点カメラ画像を「真上から見た特徴マップ（BEV 特徴）」に変換します。以降の全モジュールはこの共通の BEV 特徴の上で動きます。

占有グリッド（occupancy grid）予測。空間をマス目に区切り、「将来この時刻にこのマスは何かに占有されるか」を予測するものです。ボックスで表しにくい不定形の障害物（曲がった連結車両、はみ出した荷物など）も表現できます。

open-loop 評価と closed-loop 評価。open-loop は記録済みデータに対して「人間と同じ軌跡をどれだけ正確に再現できたか」を測ります（実際には車を動かさない。録画を再生するだけ）。closed-loop はシミュレータ内で実際に車を動かし、計画の結果が次の観測を変える状況で評価します（自分の行動が未来の入力を変える、現実に近い設定）。UniAD は主に open-loop で評価されました。後述するようにこの評価方法には議論があります。

### 仕組みを詳しく

UniAD は BEVFormer の上に5つのタスクモジュールをクエリで連結します。

```
多視点カメラ画像
   │
   ▼ BEVFormer（BEV エンコーダ）
BEV 特徴
   │
   ├─► TrackFormer  : 物体の検出と追跡（track クエリ）
   ├─► MapFormer    : 車線・地図要素の認識（map クエリ）
   │       │ track クエリと map クエリを渡す
   │       ▼
   ├─► MotionFormer : 他エージェントの未来軌跡を予測（agent クエリ）
   ├─► OccFormer    : 将来の占有グリッド予測（誰がどこを占めるか）
   │       │
   │       ▼
   └─► Planner      : 自車の軌跡を計画（ego クエリ）
```

TrackFormer は検出と追跡を同時に行い、各物体を track クエリとして表現します。新しく現れた物体には検出クエリ、既存物体には追跡クエリが対応し、フレーム間で同一物体を追い続けます（MOTR 系の end-to-end 追跡の発想）。

MapFormer は車線・横断歩道・走行可能領域などの地図要素を map クエリで表します（パノプティックセグメンテーション風の枠組み）。

MotionFormer は track クエリ（動く物体たち）と map クエリ（道路構造）の両方を受け取り、各エージェントの未来軌跡を複数候補（multi-modal、右折するか直進するか等の複数の可能性）で予測します。物体同士の相互作用（agent-agent）、物体と道路の相互作用（agent-map）、物体とゴールの相互作用をアテンションでモデル化するのが特徴です。

OccFormer は将来の占有グリッドを予測し、不定形の障害物や混雑をマス単位で、複数の未来時刻について表します。

Planner は自車を表す1本の ego クエリが、BEV 特徴と他モジュールの出力に注目して、自車の未来軌跡を出力します。さらに OccFormer の占有予測を使い、占有されるマスを避けるように軌跡を後処理（最適化ベースのコリジョン回避）で調整します。

計画指向（planning-oriented）の意味。5つのモジュールはそれぞれ補助的な学習信号（検出損失、予測損失など）を持ちますが、設計思想として全モジュールが「最終的な計画を良くするため」に配置・連結されています。論文は「どのタスクをどう連結すれば計画が良くなるか」を実験で検証し、認識・予測を計画の前に正しくつなぐと計画が改善することを示しました。逆に言えば、計画に寄与しないモジュールは外す、という取捨選択をデータで決めたのが planning-oriented の実体です。

学習戦略。2段階で学習します。第1段階で認識系（BEVFormer + TrackFormer + MapFormer）を先に数エポック学習し、第2段階でその上に予測・計画を載せて全体を end-to-end 学習します。いきなり全部を同時学習すると勾配が不安定になり収束しないため、土台を固めてから積み上げる方式です。学習には大量の計算資源（多数の GPU で数日規模）が必要でした。

### 手法の系譜と主要論文

前史。モジュール直列のパイプラインが長く主流でした。一方で完全 end-to-end の試み（1989年の ALVINN まで遡り、近年では NVIDIA の PilotNet や条件付き模倣学習 CIL）もありましたが、解釈性の欠如が課題でした。クエリベースの検出（DETR 系）と BEV 表現（BEVFormer）、そして end-to-end 追跡（MOTR）が出揃ったことで、「クエリで全タスクをつなぐ」発想が技術的に可能になりました。

UniAD。Hu, Yang, Chen ら、CVPR 2023 Best Paper、arXiv:2212.10156。多数のタスク（検出・追跡・地図・動き予測・占有予測・計画）を微分可能クエリで連結し、計画目的に向けて統合学習する初の体系的な枠組みを示しました。BEVFormer を BEV エンコーダに採用しています。CVPR の Best Paper に選ばれたのは、個別タスクの最適化という長年の常識に対し「最終目的から逆算してアーキテクチャを組む」という思想の転換を、実験で説得力を持って示したためです。

後継・関連。VAD（Jiang ら、ICCV 2023、arXiv:2303.12077）は UniAD の重い占有グリッド（密なマス予測）の代わりに、シーンをベクトル（車線や物体を点列で表す）で表現し、大幅に軽量・高速化しました。VAD は UniAD よりおよそ数倍速い推論を、同等以上の計画精度で達成したと報告されています。VADv2 はさらに確率的な計画（軌跡の確率分布を出す）を導入しました。批判的研究として AD-MLP（Zhai ら、arXiv:2305.10430）と「Is Ego Status All You Need?」（Li ら、CVPR 2024、arXiv:2312.03031）があります。これらは「自車の過去の運動状態（速度・加速度・ハンドル角）だけを入力した単純な MLP でも、nuScenes の open-loop 計画でかなり高いスコアが出てしまう」と指摘し、UniAD 系の評価方法に一石を投じました。

### 論文の実験結果（定量データ）

評価は nuScenes で、認識・予測・計画の各タスクについて行われました。

計画の主要指標。L2 誤差（予測した自車軌跡と人間の実軌跡の平均ずれ、メートル、小さいほど良い。通常1秒・2秒・3秒先の3点で測り平均する）と衝突率（collision rate、計画軌跡が他物体と衝突する割合、パーセント、小さいほど良い）です。

報告値として、UniAD は計画の平均 L2 誤差を約1.03m、平均衝突率を約0.31% に抑え、当時の手法を上回りました。特に衝突率の低減が大きく（先行手法はしばしば 0.6〜0.7% 台）、これは「予測・占有を計画の前に正しくつないだ効果」と説明されています。アブレーションで MotionFormer を外すと L2 と衝突率がともに悪化し、OccFormer を外すと特に衝突率が悪化することが示され、各モジュールが計画に貢献していることが裏付けられました。逆に、必要なつながりを欠いた構成（例えば動き予測を計画に渡さない）では計画が改善しないことも示され、「つなぎ方」が効くという主張を支えました。

認識・予測でも、検出（NDS/AMOTA）・地図（IoU）・動き予測（minADE/minFDE、複数候補のうち最良の軌跡誤差）の各指標で、当時の個別最適化手法と同等以上を達成しました。1つのネットワークが複数タスクをまとめて解いてこの水準に達した点が評価され、Best Paper につながりました。

評価方法への注意。AD-MLP と「Is Ego Status All You Need?」が示した通り、nuScenes の open-loop 計画は自車運動の慣性が強く（運転は数秒スケールでは滑らかに続く）、過去の運動だけからでも未来軌跡をかなり当てられてしまいます。実際、自車状態だけの MLP が UniAD と近い L2 誤差を出したと報告されています。つまり open-loop の L2 誤差が小さいからといって、本当に外界（他車・地図）を理解して計画しているとは限りません。後者の論文は、評価指標の見直し（衝突だけでなく走行可能領域からの逸脱なども測る）や closed-loop 評価の重要性を訴えました。この問題意識は NAVSIM（Dauner ら、NeurIPS 2024、arXiv:2406.15349）のような、closed-loop に近い評価ベンチマークの整備につながっています。

### メリット・トレードオフ・限界

メリット。認識から計画までを微分可能クエリでつなぐので、情報を捨てずに最終目的（計画）へ最適化できる。中間表現（検出・予測・占有）が人間に見えるため、ブラックボックスな完全 end-to-end より解釈・デバッグしやすい。各タスクが補助信号として学習を助け合うマルチタスク学習の利点も得られ、認識タスク単体でも競合水準を保ちます。

トレードオフと限界。モジュールが多く巨大で、学習が重く不安定になりやすい（2段階学習が必須）。各タスクの教師データ（検出・追跡・地図・予測ラベル）が大量に必要で、アノテーションコストが高い。推論が重くリアルタイム化が難しく、後継の VAD はここを軽量化しました。そして前述の通り、open-loop 評価では「自車状態だけでも高得点が出る」問題があり、外界理解の真の評価は別途必要です。さらに、純カメラ入力ゆえ悪天候・夜間に弱く、closed-loop での頑健性（分布外状況、つまり訓練データにない珍しい状況での挙動）も未解決の課題です。模倣学習が土台なので、人間のデモにない危険回避（事故寸前の挙動）を学べないという、模倣学習共通の限界も抱えます。

### 発展トピック・研究の最前線

UniAD は「end-to-end かつ解釈可能」という設計の出発点になりました。以降の流れは大きく3方向です。第1に軽量化（VAD 系のベクトル表現、スパースクエリ化、SparseDrive など）で車載リアルタイムを目指す方向。第2に評価の厳密化（closed-loop シミュレータ、CARLA や nuPlan、NAVSIM など）で、open-loop の限界を埋める方向。第3に視覚言語モデル（VLM）や大規模言語モデルと統合し、シーンを言語で推論させたり運転判断を説明させたりする方向（DriveGPT, LMDrive など）です。

研究の最前線では、世界モデル（将来の観測そのものを予測するモデル。GAIA-1 など）を計画の土台に使う試みや、模倣学習だけでなく強化学習で closed-loop の安全性を直接最適化する試みが進んでいます。UniAD が示した「計画指向で全体を統合する」という思想は、これらすべての共通の基盤になっています。一方で「真に外界を理解した計画」を測る評価設計は依然として open question であり、ベンチマークそのものの研究が活発化しています。

## UniAD(プランニング指向の統合設計)

### ひとことで言うと

UniAD は、自動運転の「途中の仕事」(物体検出・物体追跡・地図認識・他車の動き予測・将来占有予測)を 1 つのニューラルネットワークの中に順番に積み上げ、それらすべてを「最終目的＝自車が安全に走る経路を作ること(プランニング)」のために働かせる統合モデルです。途中の仕事をバラバラに最適化するのではなく、最終目的のためにネットワーク全体を一気通貫で鍛えるのが核心です。この「すべての中間タスクをプランニングという最終ゴールに奉仕させる」という設計思想を planning-oriented(プランニング指向)と呼びます。2023 年の CVPR でベストペーパー(その年最も優れた論文)に選ばれ、エンドツーエンド自動運転の方向性を決定づけた研究です。

### 直感的な理解

料理を例にします。「野菜を切る人」「炒める人」「味付けする人」「盛り付ける人」を完全に分業させると、各人は自分の作業だけを最高にしようとします。切る人は見た目のきれいさを追求するかもしれません。しかし最終的に美味しい一皿を作るには、切る段階で「この後どう炒めるか」を意識しておく必要があります。分業の境目で「次の人に必要な情報」が落ちると、全体の質が下がります。

自動運転も同じです。「物体を検出する」「物体を追跡する」「地図を読む」「他車の動きを予測する」「自車の経路を決める」という工程があります。従来は各工程を別々のモジュールとして作り、出力を次に手渡していました。検出器が物体を 1 つ見逃すと、その先の予測も計画もその見逃しに気づけません。各工程の最適化目標(検出精度、予測精度)も、最終ゴール(安全な走行)とは微妙にずれています。

UniAD は「全工程を 1 つのネットワークにまとめ、しかも全工程をプランニングのために学習する」ことでこの問題を解こうとしました。各工程はまだ存在しますが(だから途中の結果を観察できて解釈性がある)、工程間で情報をリッチなまま受け渡し、全体をプランニング性能で評価して鍛えるのです。

### 基礎: 前提となる概念

UniAD を理解するには、いくつかの前提概念を押さえる必要があります。

第一に、自動運転を機械学習で解く方法には大きく 3 つの設計思想があります。

1. モジュール分離型(modular pipeline)。「検出係」「追跡係」「予測係」「計画係」を独立のモジュールとして作り、出力を順番に手渡します。各モジュールを個別に最適化でき、人間が中身を理解しやすい。しかしモジュール間の境界(インターフェース)で情報が圧縮され、上流の誤りが下流に伝播し、各モジュールの目標が最終ゴールとずれます。

2. 完全エンドツーエンド型(end-to-end)。画像から直接ハンドル角・アクセル量を出す 1 個の巨大ネットワークです。Bojarski らの NVIDIA の研究(2016 年)が代表で、境界での情報落ちはゼロですが、中身がブラックボックスで「なぜそう判断したか」が分からず、デバッグも安全保証も難しい。

3. 統合だが中間表現を残す型。1 個のネットワークに検出・追跡・地図・予測のモジュールを積みますが、それらを「プランニングのための補助タスク」と位置づけ、全体を一緒に学習します。中間タスクの結果も観察できるので解釈性を保ちつつ、境界の情報落ちを減らせます。UniAD はこの 3 番目の代表例です。

第二に、BEV(Bird's-Eye View、鳥瞰図)表現です。複数のカメラ画像を「真上から見下ろした 2 次元のグリッド地図」に変換した特徴のことです。各グリッドのマスに「そこに何があるか」を表す特徴ベクトルが入っています。BEV は複数カメラの情報を 1 枚の共通座標系にまとめられるので、自動運転知覚の標準表現になりました(詳しくは [BEVFormer風の暗黙的投影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/04_bev-view-transform))。

第三に、クエリ(query)という概念です。これは「これについて知りたい」という問い合わせを表すベクトルです。Transformer ベースの検出器(DETR、Carion et al. 2020)が広めた考え方で、「100 個の物体スロット」のような固定数のクエリを用意し、各クエリが画像のどこかに対応する物体を見つけて、その物体の情報を吸い上げます。クエリは単なる座標や確率ではなく、高次元(典型的には 256 次元)のベクトルなので、リッチな文脈を保持できます。

第四に、マルチタスク学習(multi-task learning)です。1 つのネットワークで複数のタスク(検出、追跡、予測など)を同時に学習することです。各タスクの損失(誤差)を重み付きで足し合わせ、その合計を最小化するよう全体を学習します。共有部分(UniAD では BEV 特徴)が複数タスクの勾配で同時に鍛えられるため、1 つの表現が複数の能力を獲得します。

### 仕組みを詳しく

UniAD は共有 BEV 特徴を土台に、クエリベースのモジュールを段階的に積みます。全体の流れは次のとおりです。

```
多視点カメラ(6台) → backbone(ResNet-101-DCN) → BEVエンコーダ(BEVFormer) → 共有BEV特徴 [200,200,256]
   │
   ├─ TrackFormer : 物体の検出と追跡(フレーム間で同じ物体に同じID)
   │       └ track クエリ群を次へ渡す
   ├─ MapFormer   : 車線・歩道境界・横断歩道など地図要素のパノプティック分割
   │       └ map クエリ群を次へ渡す
   ├─ MotionFormer: 各エージェント(他車・歩行者)の将来軌跡を予測
   │       └ track と map のクエリを使い、エージェント間の相互作用も考慮
   ├─ OccFormer   : 将来の占有グリッド予測(どのマスがいつ塞がるか)
   │
   └─ Planner     : 自車の走行経路を生成。上のクエリ群と占有予測を条件にする
```

各モジュールを順に見ます。

TrackFormer は、検出と追跡を 1 つにまとめたモジュールです。従来は「検出器で物体を見つけてから、別の追跡器でフレーム間を対応づける」という 2 段構えでしたが、TrackFormer は「track クエリ」を時間方向に持ち越すことで、検出と追跡を同時に行います。各 track クエリが 1 つの物体に対応し、フレームをまたいで同じ ID を保ちます。

MapFormer は、車線・歩道境界・横断歩道などの地図要素をオンラインで認識します(事前に用意した高精細地図に頼らず、その場のカメラ映像から地図を作るという発想)。map クエリが各地図要素に対応します。

MotionFormer は、各エージェント(他車・歩行者)の将来軌跡を予測します。ここが UniAD の設計の妙で、MotionFormer は TrackFormer の track クエリと MapFormer の map クエリを直接受け取ります。つまり「この車は今この車線にいて、こう動いている」という文脈が、確率や座標の数値に圧縮されることなく、256 次元のリッチなベクトルのまま伝わります。さらにエージェント同士の相互作用(agent-agent)、エージェントと地図の関係(agent-map)、エージェントとゴールの関係を注意機構で考慮します。

OccFormer は、将来の占有グリッド(occupancy)を予測します。これは「時刻 t に、BEV 上のどのマスが何かに塞がれているか」を表す予測です。軌跡予測(MotionFormer)が「物体ごとの代表的な動き」を出すのに対し、占有予測は「シーン全体の空間がどう埋まるか」を密に表現します。

Planner は、これらすべてを条件として自車の走行経路を生成します。学習時は全タスクの損失を重み付き合計して一括で最適化します(マルチタスク学習)。検出損失・追跡損失・地図損失・予測損失・占有損失・プランニング損失の勾配が、すべて共有 BEV 特徴に流れ込みます。結果として BEV 特徴は「プランニングに効く形」に育ちます。これが「補助タスクで共有 BEV を鍛えてプランニング性能を上げる」という設計の本質です。

最後に重要なのは、Planner が OccFormer の占有予測を安全フィルタとして使う点です。生成した経路が「将来塞がるマス」を通るなら、それを避けるよう経路を後処理で最適化します。学習で得た判断に、衝突回避の幾何的な歯止めをかける二重の安全設計です。

なぜ「クエリで連結する」設計にこだわるのか。もし MotionFormer が TrackFormer から「物体の中心座標と速度」という数値だけを受け取ったら、その物体の見た目・姿勢・周囲との関係といった豊かな情報は失われます。クエリ(高次元ベクトル)のまま渡せば、後段が必要な情報を自分で取り出せます。これがモジュール分離型の「境界での情報痩せ」を緩和する仕掛けです。

### 手法の系譜と主要論文

UniAD は突然現れたわけではなく、いくつかの研究の積み重ねの上に立っています。

- Bojarski et al., "End to End Learning for Self-Driving Cars" (NVIDIA, 2016, arXiv:1604.07316)。画像から直接操舵角を出す完全エンドツーエンドの先駆け。境界での情報落ちはないが解釈性ゼロ。UniAD は「エンドツーエンドの利点(境界なし)」と「モジュール型の利点(解釈性)」の折衷を狙う、という対比で語られます。

- Carion et al., "End-to-End Object Detection with Transformers" (DETR, ECCV 2020, arXiv:2005.12872)。物体検出を「クエリによる集合予測」として定式化し、クエリベース知覚の基礎を作りました。UniAD のクエリ受け渡し設計はこの系譜にあります。

- Li et al., "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers" (ECCV 2022, arXiv:2203.17270)。UniAD の土台となる共有 BEV 特徴の作り方を提供しました。UniAD はこの BEV 表現の上に成り立ちます。詳細は [BEVFormer風の暗黙的投影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/04_bev-view-transform)。

- Hu et al., "Planning-oriented Autonomous Driving" (UniAD, CVPR 2023 best paper, arXiv:2212.10156)。本トピックの主役。検出・追跡・地図・動き予測・占有予測をクエリで連結し、すべてをプランニングに奉仕させる統合設計を提案しました。バックボーンは ResNet-101-DCN(変形可能畳み込みつき ResNet-101)で、BEVFormer の事前学習重みで初期化します。

- Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving" (ICCV 2023, arXiv:2303.12077)。UniAD の「重さ」への応答として、シーンを密な占有グリッドではなくベクトル(点や線分の集合)で表現し、大幅に高速化しました。UniAD と並んで「統合だが軽量」を狙う系譜です。トレードオフはベクトル表現の表現力の制約。

この流れをまとめると、(1) 完全エンドツーエンド(2016)→(2) クエリ知覚の確立(DETR 2020)→(3) BEV 表現の確立(BEVFormer 2022)→(4) 統合かつ解釈可能なプランニング指向設計(UniAD 2023)→(5) その軽量化(VAD 2023)という系譜が見えます。

### 論文の実験結果(定量データ)

UniAD は nuScenes データセット(6 台のカメラを積んだ車で集めた大規模な自動運転データセット。波の Boston と Singapore の市街地で 1000 シーンを収録)で評価されました。各タスクの代表的な指標と数値を、初心者にも分かるよう意味から説明します。

物体追跡(multi-object tracking)では AMOTA(Average Multi-Object Tracking Accuracy、追跡の総合精度、高いほど良い)で評価します。UniAD は約 0.359 を達成し、前段の検出・追跡を分離して扱う手法を上回りました。AMOTA は「正しく追えた割合から、見逃し・誤検出・ID の入れ替わりを差し引いた総合点」だと思ってください。

オンライン地図認識では IoU(Intersection over Union、予測領域と正解領域の重なり具合、高いほど良い)で評価し、車線分割などで競合手法を上回りました。

動き予測では minADE(minimum Average Displacement Error、複数予測した軌跡のうち最も正解に近いものの平均位置誤差、メートル単位、低いほど良い)で評価します。UniAD は約 0.71 m を達成し、追跡や地図の文脈をクエリで受け取らない構成より誤差が小さくなりました。

最も重要なのがプランニング性能です。これは 2 つの指標で測ります。

- L2 誤差(L2 displacement error、計画した経路と人間ドライバーの実際の経路との距離、メートル単位、低いほど良い)。UniAD は 1 秒・2 秒・3 秒先の平均で約 1.03 m を報告しました。
- 衝突率(collision rate、計画経路が他の物体に衝突する割合、パーセント、低いほど良い)。UniAD は平均で約 0.31% を達成し、当時の競合手法(例えば ST-P3 などのプランナー)より衝突率を大きく下げました。衝突率は安全性に直結する指標なので、ここの改善が論文の最大の主張です。

アブレーション(どの要素を抜くと性能がどう変わるかを調べる実験)も丁寧に行われています。論文の核心的な主張は「補助タスクを足すほどプランニングが良くなる」ことです。具体的には、追跡・地図・動き予測・占有予測のモジュールを 1 つずつ取り除くと、最終的なプランニングの衝突率や L2 誤差が悪化することが示されました。特に動き予測(MotionFormer)と占有予測(OccFormer)を外すと衝突率が顕著に上がります。これは「他車がどう動くか、空間がどう塞がるかを知らないと、安全な経路を引けない」という直感と一致します。逆に言えば、これらの補助タスクは飾りではなく、プランニング性能に直接寄与する必須部品だということを実験で裏づけたわけです。これが planning-oriented という設計思想の実証です。

### メリット・トレードオフ・限界

メリット。

- 補助タスクの勾配で共有 BEV 表現が「プランニングに効く形」に育ち、最終性能(特に衝突率)が向上します。アブレーションでこの効果が定量的に示されました。
- 中間タスクの出力を観察できるため、完全エンドツーエンドより解釈性・デバッグ性が高い。「なぜこの経路を選んだか」を、検出・予測・占有の中間結果から追跡できます。
- クエリ受け渡しでモジュール間の情報落ちを抑え、モジュール分離型の弱点を緩和します。

トレードオフと限界。

- モデルが大きく学習が非常に重い。多数のタスクを同時に回すため計算資源・時間が膨大で、論文の学習には複数の高性能 GPU で長時間を要します。推論も遅く、リアルタイム性が課題です(VAD はこれへの応答です)。
- 全補助タスクの教師ラベル(検出・追跡・地図・予測など全部)が必要で、データ整備のコストが極めて高い。すべてのラベルが揃った大規模データセット(nuScenes など)が前提になります。
- マルチタスク損失の重み調整が難しい。あるタスクの損失が大きすぎると学習を支配し、他のタスクが崩れることがあります。損失重みのチューニングが研究上の手間です。
- nuScenes のオープンループ評価(録画データに対して「もしこう計画していたら」を測る評価)には批判もあります。実際の閉ループ(計画した行動が次の観測を変える状況)での性能とは乖離しうるため、近年は CARLA などのシミュレータでの閉ループ評価が重視されつつあります。
- 安全フィルタとしての占有予測は有用ですが、占有予測自体が誤れば安全保証も崩れます。学習ベースの安全性をどう形式的に保証するかは未解決です。

### 発展トピック・研究の最前線

UniAD 以後、エンドツーエンド自動運転は活発に発展しています。

- 軽量化の流れ。VAD(ICCV 2023)に続き、ベクトル表現や疎なクエリで計算量を削る研究が増えています。リアルタイム動作と解釈性を両立させる方向です。
- 言語モデルとの統合。大規模言語モデル(LLM)や視覚言語モデル(VLM)を運転判断に組み込み、「なぜこの行動を取るか」を自然言語で説明させたり、常識的な推論を取り込んだりする研究(DriveGPT、LMDrive などの系譜)が登場しています。
- 生成モデルによる世界モデル(world model)。将来の観測を生成的に予測し、その予測を使って計画を立てる方向(GAIA-1 など)。UniAD の OccFormer が占有を予測する発想を、より一般的な「未来予測」へ拡張するものです。
- 閉ループ評価の重視。nuScenes のオープンループ指標の限界が指摘され、CARLA や nuPlan のような閉ループ・大規模プランニングベンチマークでの評価が標準になりつつあります。
- 補助自己教師あり信号。検出・追跡のような人手ラベルが必要な補助タスクの代わりに、「将来の視覚特徴を予測する」といったラベル不要の自己教師あり補助タスクで共有表現を鍛える研究も進んでいます。UniAD が示した「補助タスクで共有表現を鍛える」原理を、ラベルコストを下げつつ実現しようとする流れです。
