# MapTRv2

## ひとことで言うと
車に積んだカメラの映像（必要なら LiDAR も）から、「車線の境界線はここ」「横断歩道はここ」といった地図要素を、その場で（オンラインで）点をつないだ線（ベクトル）として描き出すニューラルネットワークです。MapTRv2 は前作 MapTR を改良し、より速く正確に学習できるようにした版です。あらかじめ作っておいた精密地図（HD地図）に頼らず、走りながら必要な範囲の地図を作るのが目的です。本質は「地図構築を、後処理なしの end-to-end なベクトル予測問題として一気に解く」点にあります。

## 直感的な理解
自動運転には「自分がどの車線にいて、隣の車線がどこで、横断歩道がどこか」という地図的な理解が欠かせません。従来はこれを HD地図として人手であらかじめ作り、車に積んでいました。しかし専用計測車で世界中を回って車線を引くのはコストが高く、工事で車線が変われば地図はすぐ古くなり、GPS が弱い場所では地図と現在地の照合（ローカライズ）もずれます。

そこで「カメラ映像から、その場で必要な範囲だけ地図を作ればよい」という発想が生まれました。これがオンラインHDマップ構築です。MapTR / MapTRv2 はその代表で、地図を「点の列でできた線（ベクトル）」として直接出力します。ベクトルとは、たとえば1本の車線境界線を「(x1,y1),(x2,y2),...,(x20,y20)」という20個の点で表したものです。画像のようにピクセルで塗るのではなく折れ線で表すので、後段の経路計画（どう走るか決める処理）がそのまま線として扱えて便利です。

## 基礎: 前提となる概念
言葉を順に噛み砕きます。

- BEV（Bird's Eye View、鳥瞰図）: 真上から見下ろした平面図。カメラは斜め前を見ますが、地図は上から見た平面で持ちたいので、いったん BEV に変換します。
- バックボーン（backbone）: 画像から特徴を取り出す土台のネットワーク（ResNet など）。
- Transformer デコーダ: 「クエリ（query、問い合わせのためのベクトル）」を使って出力を生成する仕組み。物体検出の DETR（Carion ら, ECCV 2020）が有名で、MapTR 系はその地図版です。
- クエリ（query）: 「この1本の車線を表す座標を出して」とネットワークに問いかける学習可能なベクトル。
- 二部マッチング（bipartite matching, ハンガリアンマッチング）: DETR 系で使う、予測の集合と正解の集合を1対1で最適に対応づける手続き。これがあるから「N個の予測のどれがどの正解か」を手作りで決めずに学習できます。
- mAP（mean Average Precision）: 検出・地図構築の精度指標。値が大きいほど良い。後述します。
- end-to-end: 入力（画像）から最終出力（地図ベクトル）まで、途中で手作りの後処理を挟まず一つのネットワークで学習・推論すること。

HD地図を事前に全部用意するのが現実的でないため、「センサからその瞬間の周囲の地図を作る」オンライン手法が研究されてきました。MapTR 系はその中で、ラスタ（ピクセル地図）を経由せず最初からベクトルを出すことで、後処理を不要にした流れの中心にいます。

## 仕組みを詳しく
全体の流れをステップで分解します。

ステップ1: 入力
車の周囲を撮る複数台のカメラ画像（たとえば6台、前後左右斜め）。形状の例は `[B, 6, 3, H, W]`（B=バッチ数、6=カメラ台数、3=RGB、H×W=画像の縦横ピクセル）。

ステップ2: 画像特徴抽出
各画像をバックボーン（ResNet50 など）に通して特徴マップにします。

ステップ3: BEV 変換
カメラ視点の特徴を「真上から見た平面」=BEV に並べ替えます。BEV 特徴の形状の例は `[B, C, 200, 100]`（C=チャンネル数、200×100=地面を上から見たグリッドのマス目数）。視点変換には、深度を推定して持ち上げる方式（LSS、Lift-Splat-Shoot 系）や、BEVクエリでカメラ特徴を引きにいく方式（BEVFormer 系）が使われます。MapTRv2 では GKT（Geometry-guided Kernel Transformer）など効率的な視点変換も採用されました。

ステップ4: クエリで地図要素を予測
「地図インスタンスクエリ（どの地図要素か）」と「点クエリ（その要素内のどの点か）」を組み合わせた学習可能ベクトルを使い、Transformer デコーダで「この地図要素（1本の車線境界）は、これらの点の座標でできている」と予測します。インスタンス数 × 点数のクエリ（例: 50インスタンス × 20点）で、複数の地図要素を並列に一度に出力します。学習では予測インスタンス集合と正解集合を二部マッチングで1対1に対応づけ、対応した予測に対して分類損失（クラスが合っているか）と点の位置損失を課します。

ここで MapTR の中心的アイデアが「順列同値モデリング（permutation-equivalent modeling）」です。たとえば横断歩道を表す4点の長方形は、「点1→2→3→4」と回っても「点2→3→4→1」と回っても、あるいは逆回りでも、同じ形を表します。素朴に「点の順番まで完全一致させろ」と学習させると、本当は正しい形なのに「順番が違う」と誤って罰してしまい、学習が不安定になります。そこで MapTR は、形が同じになる複数の点並び順（等価な並べ替え集合）を全部「正解候補」とみなし、点マッチング時にその中で予測に一番近い並びを選んで損失を計算します。これにより学習が安定し、形の予測精度が上がりました。

MapTRv2 が前作に加えた主な改良は次の3つです。

1. 補助的な one-to-many 割り当て（auxiliary one-to-many assignment）
学習時、1つの正解地図要素に対して複数の予測クエリを一時的に対応づける補助タスクを足します。本番（推論時）は1対1のままですが、学習中だけ「正解1つにつき予測を複数当てる」ことで、各反復で得られる正例（学習の手がかり）が増え、収束が速くなります。DETR 系の収束の遅さ（1対1マッチングは正例が少ない）を補う、物体検出の Group DETR / H-DETR と同じ発想です。

2. decoupled self-attention（分離した自己注意）
クエリ同士が情報をやり取りする self-attention を、「どの地図インスタンスか」という軸と「インスタンス内のどの点か」という軸に分けて計算します。素朴に全クエリ（インスタンス数×点数）を一度に attention させると計算量がその二乗（`(50×20)^2`）に膨らみますが、軸を分けることで計算を `(インスタンス数)^2 + (点数)^2`（`50^2 + 20^2`）規模に抑えつつ表現力を保ちます。

3. 補助的な密な予測（auxiliary dense prediction）
BEV 上で「ここは車線/路面境界/横断歩道」というセグメンテーション（各マスを分類する）を補助タスクとして同時に学ばせ、共有する BEV 特徴の質を上げます。透視図側（カメラ画像上）でも補助予測（PV segmentation）を足し、特徴を強化します。これらは推論時には捨てる「学習を助けるための足場」です。

ベクトル予測のイメージを図にすると次のとおりです。

```
HDMapNet 系（旧）:  画像 → BEV → セグメンテーション(ラスタ) → 後処理でベクトル化
                                               ↑ここが重く誤りも入る

MapTR / MapTRv2:   画像 → BEV → クエリで点座標を直接予測（ベクトル） → 完成
                                  ↑後処理なし・end-to-end・並列で速い
```

なぜこの設計なのか。後処理ベクトル化はクラスタリングやしきい値に依存し、セグメンテーションの小さなノイズが線の途切れになって誤りが伝播します。MapTR 系は「最初からベクトルを出す」ことでこの脆さを断ち、推論も並列予測で速くしました。MapTRv2 の3改良は、その枠組みの「学習の速さ・特徴の質・計算効率」をそれぞれ独立に底上げするものです。

## 手法の系譜と主要論文
- Li, Q., Wang, Y., Wang, Y., Zhao, H. "HDMapNet" (ICRA 2022, arXiv:2107.06307)。BEV でセグメンテーションし後処理でベクトル化する二段構えと、ベクトル地図の評価指標を整備しました（[HDMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0442_hdmapnet)）。オンライン地図構築の出発点です。
- Liu, Y. et al. "VectorMapNet" (ICML 2023, arXiv:2206.08920)。地図を polyline として自己回帰的に1点ずつ生成する手法。end-to-end でベクトルを出す方向を示しましたが、逐次生成のため遅いのが弱点でした。
- Liao, B., Chen, S., Wang, X. et al. "MapTR: Structured Modeling and Learning for Online Vectorized HD Map Construction" (ICLR 2023, arXiv:2208.14437)。地図要素を「順列に頼らない点集合」として表現し、全点を並列予測する枠組みを提案。VectorMapNet の逐次生成を一括並列に変えて大幅に高速化しました。提案理由は、従来手法がラスタ出力＋後処理で遅かったこと。トレードオフは点数を固定するため極端に複雑な形状に上限があること。
- Liao, B. et al. "MapTRv2: An End-to-End Framework for Online Vectorized HD Map Construction" (IJCV 2024, arXiv:2308.05736)。上記の one-to-many 割り当て・decoupled self-attention・補助密予測を追加し、収束速度と精度を改善。さらに中心線（centerline、車線の中心を表す線で経路計画に直接使える）の予測にも対応を広げました。トレードオフは補助タスクのぶん学習の実装と計算がやや重くなること、恩恵の多くは学習時に限られること（推論時は1対1に戻すため）。

この系譜から読み取れる一般則は、「ラスタ＋後処理 → 自己回帰ベクトル生成 → 並列ベクトル予測」と進むにつれて、手作りの後処理を消し、推論を速くし、学習を安定化させてきたという点です。物体検出の DETR 系で起きた進化（集合予測・補助割り当て・効率的注意）が、地図構築でほぼそのまま再演されています。MapTR 系の後は VectorMapNet/MapTR の発想を引き継ぎつつ、StreamMapNet（時系列）、BeMapNet（曲線表現）、TopoNet/TopoMLP（トポロジー）など複数方向へ枝分かれします。

## 論文の実験結果（定量データ）
評価は nuScenes（6カメラ＋LiDAR、ボストン/シンガポール収集の代表的データセット）の地図構築ベンチマークで行うのが標準です。指標は mAP で、地図要素3種（lane divider=車線境界、ped crossing=横断歩道、boundary=路面境界）について、予測 polyline と正解 polyline の Chamfer 距離（点列同士の互いに最も近い点の平均距離）がしきい値（0.5m / 1.0m / 1.5m）以内かで AP を判定し平均します。値が大きいほど良い指標です。学習量は「エポック数（データを何周したか）」で測り、推論速度は FPS（フレーム毎秒）で測ります。

報告されている代表的な数値は次のとおりです（およその報告値で、バックボーン・解像度・エポック数で変動します）。
- MapTR は、カメラのみ入力で、HDMapNet 等の旧手法を大きく上回る mAP を、はるかに速い推論速度で達成しました。HDMapNet の後処理ベクトル化が数 FPS 台に対し、MapTR はおよそ十数 FPS（ResNet50, 24 エポックでおよそ 50% 前後の mAP、より長い 110 エポックで 58% 程度）のリアルタイム級です。
- MapTRv2 はカメラのみ・nuScenes・ResNet50 で、同じ 24 エポックなら MapTR より明確に高い mAP（およそ 61% 前後）を出し、110 エポックではおよそ 68% 前後に達すると報告されています。同等精度に少ないエポックで到達する（収束が速い）のも大きな利点です。LiDAR を加える、あるいはカメラ＋LiDAR を融合するとさらに数ポイント上がります。
- アブレーション（改良要素を一つずつ抜く実験）では、one-to-many 割り当てを外すと収束が遅くなり最終 mAP も下がる、decoupled self-attention を素朴な全結合 attention に戻すと計算が重くなる（メモリ・速度が悪化）、補助密予測（BEV/PV segmentation）を外すと mAP が数ポイント落ちる、と各要素の寄与が示されています。3つの改良はそれぞれ独立に効き、合わせると最大の改善になります。報告では補助密予測の寄与が特に大きいとされています。

指標の意味を補足します。mAP が「68%」というのは、地図要素を Chamfer 距離の厳しめのしきい値（0.5mなど）も含めて平均したときの正解率に相当します。MapTR の 50% 前後から MapTRv2 の 68% 前後へという十数ポイントの改善は、細かい車線位置の精度向上を意味し、車線変更や交差点通過のような細部依存の判断に直結するため重要です。FPS の向上は、車載のリアルタイム処理（数十ミリ秒以内に1フレームを処理）に乗るかどうかを左右する実用上の差です。

## メリット・トレードオフ・限界
メリット側:
- 事前のHD地図が不要で、地図の鮮度問題（工事・新設で古くなる）に強い。
- 出力が点列ベクトルなので、後処理なしで経路計画にそのまま渡せる。
- 全点を並列予測するため推論が速く、車載リアルタイムに乗る。
- MapTRv2 の改良で学習が速く収束し、少ないエポックで高精度になる。中心線対応で計画への接続も良い。

トレードオフ・限界:
- 1要素あたりの点数を固定するため、極端に長い・複雑な形状（大きく蛇行する車線など）の表現に上限があります。
- カメラ視界の外や遠方は精度が落ちます。センサで見えない範囲は推定にすぎず、事前HD地図のように「見えない先の車線」を保証できません。
- 学習に大量のラベル付き地図データと計算資源が要ります。地図のアノテーションは高コストです。
- 時間方向の整合（フレーム間で同じ車線が一貫して出るか）は基本フレームワークでは保証されません。単フレーム処理なので、フレームごとに線がちらつく問題は時系列を入れる発展研究（StreamMapNet）で扱われます。
- 評価のリーク問題が指摘されています。nuScenes は同じ地理エリアが学習・検証に混在するため、地図を「記憶」してしまい数値が過大評価される懸念があり、地理的に分割した評価（geographically disjoint split）で見ると数値が落ちる、という後続研究の指摘があります。
- 研究上の未解決課題として、地図のトポロジー（車線同士の接続・分岐合流の関係）まで含めて正しく出すこと、未知都市への汎化（学習していない国・道路様式での性能維持）が残された課題です。

## 発展トピック・研究の最前線
MapTR 系の後、研究は「形だけでなく関係（トポロジー）」「単フレームから時系列」「点列から滑らかな曲線」へ進んでいます。

- トポロジー: 車線中心線とその接続グラフを同時に出す研究（TopoNet、TopoMLP など）。経路計画には「どの車線からどの車線へ行けるか」が要るため、線分の検出だけでなく接続推定が重要になっています。MapTRv2 の中心線対応はその橋渡しです。
- 時系列・ストリーミング: 過去フレームの BEV や地図クエリを引き継ぎ、走行に沿って地図を更新し続ける StreamMapNet 系。単フレームより遠方・遮蔽にロバストになり、時間方向のちらつきも抑えます。
- 曲線表現: 点列の代わりにベジエ曲線などの滑らかな媒介変数表現で車線を出す BeMapNet 系。固定点数の弱点（細部の取りこぼし）を緩和します。
- prior の併用: SD地図やナビ地図を prior として注入し、遠方や遮蔽を補う方向（[SDTagNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0434_sdtagnet)、SMERF、P-MapNet など）。MapTR 系の純カメラ構築と相補的です。

オンライン地図構築は、知覚（perception）から計画（planning）へ向かう end-to-end 自動運転パイプラインの中核部品になりつつあり、地図出力を後段の予測・計画と微分可能につなぐ統合研究が最前線です。特に「地図をモジュールとして陽に出すのか、それとも end-to-end 運転モデルの中で暗黙の特徴として持つのか」という設計論が活発な論点になっています。
