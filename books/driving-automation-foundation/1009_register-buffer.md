# register_buffer(PyTorch)

## ひとことで言うと

ディープラーニングのフレームワーク PyTorch では、モデルを `nn.Module` というクラスを継承した形で書きます。そのモデルの中で `self.register_buffer("name", tensor)` と書くと、「学習(訓練)では更新されないが、モデルの状態の一部として一緒に持ち運びたい固定テンソル」を登録できます。ここでバッファ(buffer)とは「学習対象ではないがモデルに付随させたいデータ」のことです。登録したテンソルは、モデルを保存するときに一緒に保存され、`model.to("cuda")` でモデルを GPU に移すときに一緒に GPU へ移動します。学習で勝手に書き換わることはありません。一見すると地味な機能ですが、これを正しく使えるかどうかが「動くモデル」と「読み込んだ瞬間に精度が壊れるモデル」の分かれ目になります。

## 直感的な理解

ニューラルネットワークを「料理のレシピと道具のセット」にたとえてみます。レシピのうち「火加減や調味料の分量」にあたるのが学習で変わる重み(パラメータ)です。一方、「使う計量カップの容量」や「オーブンの定格出力」のように、調理のたびに使うけれど勝手に変えてはいけない固定値もあります。この固定値をレシピ本体と別々の引き出しにしまっておくと、引っ越し(GPU への移動)のときに置き忘れたり、レシピを誰かに渡すとき(保存)に同梱し忘れたりします。register_buffer は「この固定値もレシピと同じ箱に入れておくので、移動も配布も一緒にやってください」とフレームワークに宣言する仕組みです。

なぜこれが要るのか。現代のモデルには「学習しないが計算に必須の固定テンソル」がたくさん登場します。たとえば、推論時に使う正規化の統計量、決まったルールで生成される位置エンコーディング、座標変換に使う定数行列、注意計算で未来を隠すためのマスクなどです。これらをきちんと「モデルの状態」として登録しておかないと、後述する三種類のバグが静かに発生します。しかも厄介なのは、これらのバグはモデルが小さいうちは「たまたま CPU で動いていた」「たまたま保存し直さなかった」ために露見せず、GPU に移して大規模学習を始めた途端、あるいは別環境でチェックポイントを読み込んだ途端に初めて表面化する点です。register_buffer はそのバグを構造的に防ぐための、地味だが極めて重要な機能です。

## 基礎: 前提となる概念

まず PyTorch のテンソル(tensor)とは、数値を格子状に並べた多次元配列のことです。画像なら「縦×横×チャネル」のように複数の軸を持つ数の塊で、GPU 上に置いて高速に計算できます。テンソルには `device`(CPU にあるか、どの GPU にあるか)と `dtype`(float32 か bf16 かといった数値の型)という属性があり、計算する二つのテンソルはこれらが一致していないとエラーになります。GPU は数千個の演算ユニットを持ち行列計算を並列に処理できる装置で、深層学習はほぼ必ず GPU 上で行うため、「テンソルが正しい GPU 上にあること」は計算が成立する最低条件です。

次に `nn.Module` とは、PyTorch でモデルを表現する基底クラスです。自作のモデルはこれを継承し、`__init__` で部品(層)を定義し、`forward` で順伝播(入力から出力を計算する処理)を書きます。`nn.Module` は内部に三種類の状態を持ちます。

- パラメータ(`nn.Parameter`): 学習で更新される重み。`requires_grad=True`(勾配を計算する)が付き、`model.parameters()` が返す集合に含まれ、オプティマイザ(最適化器)が毎ステップ更新します。畳み込みの重みや全結合層のバイアスがこれです。
- バッファ(buffer): 学習では更新されないが、モデルの状態として持ち運びたいテンソル。`model.buffers()` で取得でき、`state_dict()`(モデルの全状態の辞書)に含まれます。
- 普通の Python 属性(`self.x = tensor`): 上のどちらにも登録されていない、ただのインスタンス変数。

ここで `state_dict()` とは、モデルが持つ全パラメータとバッファを名前付きの辞書としてまとめたものです。`torch.save(model.state_dict(), path)` で保存し、`model.load_state_dict(torch.load(path))` で読み込みます。チェックポイント(学習途中の保存)はこれを使います。実は `nn.Module` は内部で `_parameters`、`_buffers`、`_modules` という三つの順序付き辞書を持っていて、`__setattr__`(属性代入をフックする特殊メソッド)が代入された値の型を見て自動的に振り分けています。`nn.Parameter` を代入すれば `_parameters` へ、`register_buffer` を呼べば `_buffers` へ登録される、という仕掛けです。普通のテンソルをそのまま `self.x =` で代入すると、このどの辞書にも入らず、ただのインスタンス変数になります。これが後で問題を起こします。

最後に勾配(gradient)とは、「あるパラメータを少し動かすと損失がどれだけ変わるか」という傾きの値です。学習はこの勾配を使って重みを少しずつ修正します。バッファには通常この勾配を流さない(`requires_grad=False`)ので、学習で更新されません。

## 仕組みを詳しく

`register_buffer("name", tensor)` を呼ぶと、登録したテンソルは次の四つの性質を持ちます。

1. デバイス移動・型変換に追従する。`model.to("cuda")` や `model.half()` を呼ぶと、パラメータと一緒にバッファも GPU へ移り、型も変換されます。内部的には `_apply` というメソッドが `_parameters` と `_buffers` の両方を走査して変換をかけるため、両者がそろって移動します。
2. `state_dict()` に含まれる(デフォルトの persistent=True の場合)。したがって保存・読み込みの対象になります。
3. `model.buffers()` や `model.named_buffers()` で列挙できる。モデルの状態として一元管理されます。
4. オプティマイザの更新対象にならない。`optimizer` が更新するのは `model.parameters()` が返すものだけなので、バッファは学習で変化しません。

`nn.Parameter` との違いを整理します。次の三行はそれぞれ挙動が違います。

```
self.w = nn.Parameter(torch.randn(256))     # 学習される / to(device)で移動 / 保存される
self.register_buffer("c", torch.ones(256))  # 学習されない / to(device)で移動 / 保存される
self.x = torch.ones(256)                     # 学習されない / to(device)で移動しない / 保存されない
```

問題になるのは三番目です。普通の属性として固定テンソルを持つと、次の三つの困りごとが起きます。

一つ目はデバイスの取り残しです。`model.to("cuda")` を呼んでも `self.x` は CPU に残るため、forward の途中で「CPU のテンソルと GPU のテンソルを掛け算しようとした」というエラー(`RuntimeError: Expected all tensors to be on the same device`、いわゆる device mismatch)で落ちます。初学者が最もよく踏むバグの一つで、しかも CPU だけで動かしている間は決して再現しないため、GPU 移行や分散学習の段になって初めて顕在化します。

二つ目は保存漏れです。`state_dict()` に `self.x` が含まれないので、保存→読み込みすると値が失われます。たとえば推論時に必要な正規化統計量を普通の属性で持っていると、学習済みモデルを別環境で読み込んだとき統計量が初期値に戻り、精度が大きく崩れます。学習スクリプトと推論スクリプトが別プロセスのとき、学習側では正しく動いていたのに推論側だけ精度が出ない、という再現困難な不具合になります。

三つ目は管理からの脱落です。`parameters()` にもバッファにも現れないので、モデルの状態として認識されず、混合精度(複数の数値型を混ぜる高速化)やデバイス整合性チェックの仕組み、`DistributedDataParallel`(複数 GPU でモデルを複製する分散学習ラッパ)による状態同期からも漏れます。

register_buffer はこれらをまとめて解決します。

### persistent=False の使いどころ

`register_buffer("name", tensor, persistent=False)` と指定すると、「デバイス移動はされるが state_dict には保存されない」という挙動になります。読み込みのたびに同じ値を決定的に再計算できる定数(規則的に決まる基底行列や座標グリッドなど)は、これを non-persistent にすると保存ファイルを膨らませずに済みます。たとえば形状 [10000, 4, 3] の座標グリッドは float32 で約 480 KB あり、これが persistent だとチェックポイントごとに毎回保存されますが、`__init__` で必ず再生成されるなら保存する意味はありません。さらに重要なのは、persistent=False にしておくと、後でグリッドの解像度やマスクの長さを変えた新しいバージョンのモデルで古いチェックポイントを読み込んでも、形状不一致による `load_state_dict` の失敗を避けられる点です。固定の定数は「コードで決まる」ものなので、データ(チェックポイント)側に焼き込まないほうが保守しやすい、という設計判断が背景にあります。

### 数値例: 形状とサイズの感覚

| 用途 | 例の形状 | 学習する? | 推奨登録 |
|---|---|---|---|
| 畳み込み重み | [64, 3, 7, 7] | する | nn.Parameter |
| BatchNorm の running_mean | [64] | しない(移動平均で更新) | buffer (persistent) |
| BatchNorm の num_batches_tracked | スカラ | しない(カウントアップ) | buffer (persistent) |
| 正弦波の位置エンコーディング | [1, 512, 256] | しない(規則的に再計算可) | buffer (persistent=False も可) |
| 因果マスク(下三角の真偽値) | [512, 512] | しない | buffer (persistent=False) |
| 学習可能な位置埋め込み(ViT) | [1, 197, 768] | する | nn.Parameter |

ここで「学習しないが更新はされる」という微妙なケースが BatchNorm の running_mean です。これは勾配では更新されませんが、forward のたびにバッチ統計で移動平均が書き換わります。バッファは「勾配で更新されない」だけで「一切変化しない」とは限らない、という点に注意してください。だからこそバッファは保存の対象になっている必要があります(推論時にこの統計を使うため)。表の最後の行が示すように、「位置情報」という同じ役割でも、固定式で計算するなら buffer、データから学ばせるなら Parameter と、設計意図によって登録先が真逆になります。

## 手法の系譜と主要論文

register_buffer 自体は論文発の概念ではなく、PyTorch の `torch.nn.Module` が提供するフレームワーク機能です。出典は PyTorch 公式ドキュメントの register_buffer の項になります。ただし「学習しない統計量をモデルに同梱して持ち運ぶ」という発想そのものは、いくつかの代表的な研究で本質的に使われています。系譜を追うと、なぜこの機能が必要になったかが見えてきます。

最も典型的な例が Batch Normalization(Ioffe と Szegedy、ICML 2015、arXiv:1502.03167)です。BatchNorm は学習時にはミニバッチ(同時に処理する数十枚のデータ)の平均と分散で各特徴を正規化しますが、推論時には一枚ずつ処理することもあるためバッチ統計が使えません。そこで学習中に running_mean と running_var という移動平均を蓄積し、推論時にはこの固定統計量を使います。この running_mean / running_var はまさにバッファとして保持される代表例で、勾配では更新されないがモデルの状態として保存・移動される必要があります。BatchNorm を保存したのに統計量が抜け落ちていたら推論結果が壊れる、という事実が「バッファを state_dict に含める」設計の動機を与えています。BatchNorm が持つ `num_batches_tracked`(何バッチ処理したかのカウンタ)も buffer であり、これは勾配でもバッチ統計でもなく単純なインクリメントで更新される、という buffer の多様な使われ方を示す例です。

次に Transformer(Vaswani ら、NeurIPS 2017、arXiv:1706.03762)の位置エンコーディングです。元論文は正弦波(sin/cos)で位置を符号化する固定の行列を入力に足します。この行列は学習されず、決まった式で計算できる定数なので、バッファ(しばしば persistent=False)として保持するのが定石です。また Transformer のデコーダで使う因果マスク(未来のトークンを見ないようにする下三角行列)も、学習しない固定テンソルなのでバッファに置きます。マスクを普通の属性で持つと、GPU 移行時に取り残されて device mismatch を起こす典型例になります。

さらに視覚分野の Vision Transformer(Dosovitskiy ら、ICLR 2021、arXiv:2010.11929)では学習可能な位置埋め込みを使うため、こちらは Parameter になります。同じ「位置情報を入れる」目的でも、Transformer の正弦波版は buffer、ViT の学習版は Parameter という対比は、「固定なら buffer、学習させたいなら Parameter」という分岐が研究の文脈ごとにどう選ばれているかをよく示しています。どちらが優れているかはタスク依存で、ViT 論文自身も学習可能版と固定正弦波版でほとんど精度差がなかったと報告しており、選択の自由度が高い箇所です。

このように、register_buffer は単独の発明ではなく、「学習しない状態をどう正しく持ち運ぶか」という問題に対するフレームワーク側の標準解であり、上記の主要モデルすべてが暗黙にこの仕組みに依存しています。

## 論文の実験結果(定量データ)

register_buffer はフレームワーク機能なので、それ自体の精度比較実験はありません。しかし「学習しない統計量を正しく扱うことがどれだけ効くか」は、関連する BatchNorm の実験から定量的に読み取れます。

Ioffe と Szegedy(2015)は、ImageNet(約 128 万枚、1000 クラスの画像分類データセット)の Inception ネットワークに BatchNorm を導入する実験を報告しています。評価指標は分類誤り率(top-5 で正解候補上位 5 個に正解が入らない割合、低いほど良い)と、ある精度に到達するまでの学習ステップ数です。報告によると、BatchNorm を入れた版はベースラインの精度に到達するまでの学習ステップが約 14 分の 1(7%)に短縮され、さらに学習率を 5 倍から 30 倍程度まで高く設定できるようになって最終精度も改善し、ImageNet の top-5 検証誤り率で当時の最高水準(4.9% 程度のアンサンブル)を更新しました。この「推論時に使う running 統計量を保存・移動できなければならない」という要請が、まさにバッファという仕組みを必要としています。もし統計量を普通の属性で持って保存し損なえば、学習で得たこの高速化と精度向上は読み込み時にすべて失われます。

実装上の失敗例を定量的に言うと、BatchNorm 層を含むモデルでバッファの保存を誤ると、推論時の正規化が初期値(平均 0、分散 1)で行われ、特徴のスケールがずれて精度が壊滅的に落ちます。報告される例では top-1 精度が数十パーセントポイント単位で低下し、ランダム予測に近いところまで崩れることもあり、これは「バッファを正しく扱う」ことの実効性を裏側から示しています。逆に言えば、バッファを正しく登録するだけでこの崩壊は構造的にゼロにできる、という点に register_buffer の価値があります。

## メリット・トレードオフ・限界

メリットは三つです。第一に、固定テンソルのデバイス移動と型変換が自動で正しく行われ、device mismatch エラーを構造的に防げること。第二に、state_dict に含まれるため保存・読み込みで値が失われないこと。第三に、モデルの状態として一元管理され、混合精度や分散学習の仕組みとも整合すること。

トレードオフと限界もあります。persistent なバッファは state_dict に含まれるため、大きな定数テンソルを多数登録するとチェックポイントのファイルサイズが膨らみます(再計算可能な定数は persistent=False で回避します)。また「学習させたい値を誤ってバッファにしてしまう」と、そのテンソルは永遠に更新されず、原因の分かりにくいバグになります。逆に「固定したい統計量を Parameter にしてしまう」と、本来固定すべき値が勾配で動いて学習が不安定になります。つまり register_buffer は便利な反面、「学習したいか/固定したいか」という設計者の意図を正確に反映する責任を要求します。バッファか普通の属性かは「forward 中に GPU 上のテンソルと演算するか」「保存して別環境で再現したいか」の二問で判断するとほぼ間違いません。

研究上の未解決寄りの論点としては、分散学習(複数 GPU でモデルを分割・複製する)におけるバッファの同期があります。BatchNorm の running 統計量は各 GPU でローカルに計算されるため、複製間でずれます。`DistributedDataParallel` は学習開始時に rank 0 のバッファを全 GPU へブロードキャストしますが、その後の forward での更新は各 GPU 独立になるため、これを毎ステップ揃える SyncBatchNorm のような工夫が必要です。バッファの「いつ・どこで同期するか」は実装ごとに判断が分かれる難しい設計点で、small batch per-GPU の学習では SyncBatchNorm の有無で数ポイント精度が変わることが知られています。

## 発展トピック・研究の最前線

バッファの概念は、より高度な状態管理へと発展しています。一つは EMA(指数移動平均)モデルです。学習中の重みの移動平均を別途保持し、推論にはこの平滑化された重みを使うと精度が安定することが知られており(拡散モデルや自己教師あり学習で標準的)、この EMA 重みもバッファあるいは別モジュールとして持ち運ばれます。たとえば拡散モデルでは EMA 重みの有無でサンプル品質(FID スコア)が体感できるほど変わるため、EMA 重みを state_dict に正しく含めることが必須になります。

もう一つは量子化(quantization、重みを低ビットで表現して軽量化する技術)で、量子化のスケール係数やゼロ点といった固定パラメータがバッファとして保持されます。これらは勾配で学ぶ場合も固定する場合もあり、手法ごとに Parameter か buffer かが分かれます。さらに、KV キャッシュ(Transformer 推論で過去の Key/Value を貯める)のような「学習しないが状態として持つテンソル」をどう効率的に管理するかは、大規模言語モデルの推論最適化で活発に研究されている領域です。長文推論では KV キャッシュがメモリの大半を占めるため、これを buffer 的に保持しつつ量子化・圧縮する手法が次々提案されています。バッファという素朴な機能の延長線上に、現代の効率化技術の多くが乗っていると理解すると見通しが良くなります。
