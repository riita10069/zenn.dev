# ラスタ地図 vs ベクトル地図

## ひとことで言うと
地図をニューラルネットに渡すとき、「ピクセルで塗った画像(ラスタ)」として渡すか、「点の列やグラフ(ベクトル)」として渡すか、という表現の選び方の話です。ラスタは画像系ネットワーク(CNN)と相性が良く扱いやすい一方、ベクトルは元の幾何構造を丸めずに、必要な点だけを効率良く渡せます。どちらが絶対的に正しいというより、何を表したいかによるトレードオフです。線的・構造的なものはベクトル、面的なものはラスタ、というのが大まかな指針です。

## 直感的な理解
地図には「車線の境界線」「横断歩道」「他車の過去の軌跡」など、もともと線や点でできた情報がたくさんあります。これをコンピュータに食わせるには数値にしなければなりません。やり方は大きく 2 つあります。

一つは、地図を真上から見た写真のように「塗り絵」にすること(ラスタ化)。各ピクセルが「ここは車線/横断歩道/何もない」を色や数値で表します。写真と同じ形なので、画像処理のネットワーク(CNN)がそのまま使えます。

もう一つは、線を「点をつないだ折れ線」として、道路網を「点と辺のグラフ」として、座標そのままで渡すこと(ベクトル化)。地図帳をなぞって座標を書き写すイメージです。元の形を丸めずに、必要な点だけを持てます。

なぜこの選択が重要かというと、表現を変えると「失われる情報」「計算量」「後段の使いやすさ」が大きく変わるからです。細い車線境界をラスタにすると、解像度が粗いと線が潰れたりギザギザになったりします。これを量子化誤差(連続的な座標をマス目に丸めることで生じる誤差)と言います。ベクトルなら座標をそのまま持てるので潰れません。

## 基礎: 前提となる概念
- ラスタ化(rasterize)。連続的な図形を、決まった解像度のグリッド(マス目)に塗り分けて画像にすることです。各マスは「だいたいの位置」しか持てず、マスより細かい位置情報は失われます。
- ベクトル化(vectorize)。図形を点・線・面の座標データとして持つことです。座標は実数なので、マス目に丸まりません。
- CNN(畳み込みニューラルネットワーク)。画像を小さなフィルタで畳み込みながら処理するネットワークです。画像=ラスタを前提とするので、ラスタ地図と素直に組み合わさります。
- 受容野(receptive field)。CNN の出力の 1 点が、入力のどれだけ広い範囲の情報を見て決まっているか、を指します。受容野が狭いと、遠く離れた 2 つの要素の関係(交差点の反対側の車線と自車線の関係など)を捉えにくくなります。CNN は層を重ねるほど受容野が広がりますが、それでもグローバルな関係を直接見るのは苦手です。
- polyline(ポリライン=折れ線)。点を順につないだ線で、車線境界などをベクトルで表す基本単位です。
- グラフニューラルネット / Transformer。点やノードの集合と、その間の関係を直接扱えるネットワークです。Transformer の self-attention は、全要素ペアの関係を一度に見るので、受容野の制約がありません。ベクトル地図(点とグラフ)を処理するのに向きます。
- PointNet。順序を持たない点の集合を、各点を独立に処理してから集約(max pooling など)する形で扱うネットワークの古典(Qi et al., CVPR 2017)で、ベクトル要素のエンコードによく使われる考え方です。「点の集合は順番を変えても同じものなので、順序に依存しない処理(置換不変な処理)が要る」という洞察が肝です。

## 仕組みを詳しく
両者の中身を、同じ「車線境界 1 本を渡す」例で具体的に比べます。

ラスタの場合:

1. 地図範囲を決める(例: 前方 50m × 左右各 25m)。
2. それをグリッドに刻む(例: 0.25m ごとなら 200×100 マス)。
3. 車線境界の線が通るマスを塗る。出力は画像 `[B, チャンネル数, 200, 100]`(B はバッチ、チャンネルは要素種別など)。
4. これを CNN や画像バックボーンに通す。各マスは 0.25m 単位に丸まった「だいたいの位置」しか持てません。

ベクトルの場合(同じ車線境界):

1. 車線境界を等間隔の点にサンプリング(例: 20 点)。各点は `(x, y)` の実数座標。
2. 形状は `[B, 要素数, 20, 2]` のように、点の座標をそのまま持つ。
3. これを Transformer や PointNet/VectorNet のようなネットワークに通す。座標は丸まらず、必要な点だけなので軽い。

ベクトルの強みは、要素同士の関係を直接モデル化できることです。VectorNet のグラフ表現では、1 本の折れ線を「ノードの集合」とみなし、まず折れ線内で情報を集約し(局所サブグラフ)、次に折れ線同士の関係を集約します(グローバルグラフ)。これにより「この車線とこの軌跡は隣接している」「この車線はこの車線へつながる」といった構造的関係を、受容野の制約なしに直接扱えます。CNN だと、地図の端と端にある関連要素の関係を捉えるには深い層が必要ですが、グラフ/attention なら 1 ステップで結べます。

数値感覚で比べます。ラスタは範囲とマス数に比例してメモリを食います。200×100=2 万マスは、たとえ大半が空マス(何も描かれていないマス)でも、全部メモリと計算を消費します。ベクトルは要素数 × 点数だけ、たとえば車線 20 本 × 20 点 = 400 点しか持たないので、空きスペースが多い地図ほどベクトルが圧倒的に有利です。地図は本質的に疎(道路以外は空白)なので、この差は実用上きわめて大きく効きます。

```
ラスタ: ■=線が通るマス, ・=空マス
  ・・・■・・・・・・
  ・・■・・・・・・・   ← 2万マス全部を保持(大半は空)
  ・■・・・・・・・・   ← 線がガタガタに量子化される

ベクトル: (12.3, 4.1)-(11.8, 5.0)-(11.2, 5.8)... 20点だけ
  → 座標は丸まらない・軽い・関係を辺で持てる
```

逆に、走行可能領域の面のように「びっしり塗りつぶされた領域」はラスタが自然です。面をベクトルの点列で表そうとすると、輪郭は表せても内部の充填を別途扱う必要があり、かえって面倒になります。だから「線的・疎・構造的ならベクトル、面的・密ならラスタ」という使い分けが基本指針になります。

もう一つ、ベクトルには「並べ替え不変性(permutation invariance)と方向の曖昧さ」という固有の難しさがあります。車線境界を点列で表すとき、「左から右へ並べた点列」と「右から左へ並べた点列」は同じ車線を指しますが、ナイーブに教師ラベルと比較すると順序違いで大きな誤差に見えてしまいます。この問題は後述する MapTR が「点集合のあらゆる等価な並べ替えを許す損失(permutation-equivalent loss)」で正面から解決しました。ラスタにはこの曖昧さがない代わりに量子化誤差がある、という対照的なトレードオフです。

## 手法の系譜と主要論文
- ラスタ baseline(ChauffeurNet, Bansal et al., 2018, "ChauffeurNet: Learning to Drive by Imitating the Best and Synthesizing the Worst")。地図と物体を多チャンネル BEV 画像にレンダリングし、CNN で処理して運転を模倣学習しました。動機は実装が単純で画像技術をそのまま使えることです。トレードオフは、量子化誤差・受容野の制約・空マスへの無駄な計算という非効率でした。これがベクトル化への布石になります。
- VectorNet(Jiyang Gao, Chen Sun, Hang Zhao, Yi Shen, Dragomir Anguelov, Congcong Li, Cordelia Schmid, CVPR 2020, "VectorNet: Encoding HD Maps and Agent Dynamics from Vectorized Representation")。ラスタ化 + CNN の弱点を正面から突きました。動機は、(1) 解像度の量子化誤差で精密さを失う、(2) 受容野の制約で遠く離れた要素の関係を捉えにくい、(3) 空マスにも計算を使い非効率、という 3 点です。新規性は、地図と他車軌跡をベクトル(折れ線グラフ)で表し、サブグラフ → グローバルグラフと階層的に集約することで、これらを一挙に解消したことです。
- LaneGCN(Liang et al., ECCV 2020, "Learning Lane Graph Representations for Motion Forecasting")。車線をグラフとして表し、車線同士・車線とエージェント間の関係をグラフ畳み込みで扱いました。車線の接続(前後・隣接)を明示的にグラフ構造として持ち込み、ベクトル/グラフ表現が軌道予測に効くことをさらに推し進めた重要な一歩です。
- 地図構築側での同じ対比。HDMapNet(Li et al., ICRA 2022, "HDMapNet: An Online HD Map Construction and Evaluation Framework")はラスタ(BEV セマンティックセグメンテーション)を予測してから後処理でベクトル化する設計で、ラスタの扱いやすさを残しつつベクトル出力を得ます。MapTR(Liao et al., ICLR 2023, "MapTR: Structured Modeling and Learning for Online Vectorized HD Map Construction")は中間ラスタを介さず、地図要素を直接ベクトル(点列)として、学習可能なクエリでエンドツーエンドに予測します。この「ラスタ経由 → 直接ベクトル予測」という進化は、まさに本章のトレードオフの反映です。詳しくは [HDMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0442_hdmapnet) と [MapTRv2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0452_maptrv2)。

系譜を一言でまとめると、「ラスタは単純で面に強いが精密さと効率に難あり」→「VectorNet がベクトル/グラフで精密さ・関係・効率を獲得」→「予測も構築もベクトルを直接出す方向へ収束」という流れです。

## 論文の実験結果(定量データ)
評価の主戦場は軌道予測のベンチマーク(Argoverse など)と HD 地図構築(nuScenes)です。指標を先に説明します。軌道予測では ADE(Average Displacement Error=予測軌道と正解の各時刻のずれの平均、低いほど良い)と FDE(Final Displacement Error=最終時刻のずれ、低いほど良い)が代表です。地図構築では mAP(平均適合率、高いほど良い)を使い、予測ベクトルと正解の Chamfer 距離が一定しきい値以内かで正解判定します。

- VectorNet の効果。論文は、ラスタ化 + CNN のベースラインに対し、ベクトル表現で軌道予測の誤差(ADE/FDE)を下げつつ、モデルの計算量(FLOPs)とパラメータを大幅に削減したと報告しています。論文では、同等のラスタ ConvNet ベースラインに対し計算量を 1 桁規模で削減しながら予測誤差を下げた、という主旨の結果が示されました。「ベクトルが軽くて精度も良い」ことの実証です。
- アブレーションの知見(VectorNet)。グローバルな相互作用グラフ(折れ線同士の関係を集約する部分)を抜くと精度が落ちることが示されており、「遠く離れた要素の関係を直接モデル化する」ことの寄与が確認されています。これはラスタの受容野の制約をベクトル/グラフが埋めている、という主張の裏づけです。さらに、補助タスクとして「隠したノードの座標を復元させる(node completion)」自己教師を加えると精度が上がることも報告され、地図構造の事前学習が効くことを示しました。
- 地図構築側の対比。MapTR 系は、中間ラスタを介す HDMapNet 系に比べて、直接ベクトル予測により高い mAP と速い推論を達成したと報告しています([MapTRv2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0452_maptrv2))。nuScenes 上で、HDMapNet が mAP 数十%台前半だったのに対し、MapTR 系は同条件でそれを大きく上回り(報告では十数〜数十ポイント差)、かつリアルタイム級の推論速度を達成しました。中間ラスタの量子化と後処理(ヒューリスティックなベクトル化)のコストを省けることが効いています。
- 並べ替え不変損失の効果(MapTR)。点列の方向・順序の曖昧さを吸収する permutation-equivalent な損失を導入したことが精度向上の主因の一つと報告されており、これを外すと学習が不安定になり mAP が落ちます。ベクトル表現特有の難しさ(順序の曖昧さ)を損失設計で解いた好例です。

指標の意味を補足すると、ADE/FDE が数十 cm 縮むというのは、予測した他車の軌道が実際の動きにより近づくということで、その分だけ衝突回避や計画の安全余裕が増えます。地図構築の mAP が十数ポイント上がるのは、車線をより確実に描けるということで、その上を走る計画の信頼性に直結します。計算量の削減は、車載の限られた計算資源でリアルタイム動作させる上で死活的に重要です。

## メリット・トレードオフ・限界
- ラスタのメリット。CNN/画像バックボーンがそのまま使えます。面的な領域(走行可能エリアなど)の表現が自然で、実装が単純です。点列の順序の曖昧さがありません。
- ラスタのトレードオフ。量子化誤差で精密さを失い、空マスにも計算を使うので非効率です。受容野の制約で遠距離の関係を捉えにくい。
- ベクトルのメリット。座標が丸まらず精密で、必要な点だけなので軽い。要素同士の関係(グラフ)を直接扱え、後段の経路計画にそのまま渡せます。
- ベクトルのトレードオフ。前処理(サンプリング・グラフ化)が必要です。点数を固定する手法が多く、極端に複雑な形状には点数の上限が制約になります。面的な情報(塗りつぶし領域)は苦手です。点列の順序・方向の曖昧さに対処する損失設計が要ります。
- 限界。両表現の良いとこ取りをどう設計するか(面はラスタ、線はベクトル、という混在をどう統合するか)は依然工夫の余地があります。また、ベクトル化で「何を点として残し、何を捨てるか」の前処理が予測性能を左右し、この設計を自動化・学習化することは未解決の研究課題です。曲がりの強い車線では固定点数だと表現が粗くなる(適応的な点配置が要る)という問題も残ります。

## 発展トピック・研究の最前線
最前線では、ラスタとベクトルの境界自体が溶けつつあります。query-based なベクトル予測(地図要素を学習可能なクエリで直接デコードする MapTR 系の流れ)が主流化し、中間ラスタを介さない設計が標準になりました。一方で、BEV 特徴自体は密なグリッド(ラスタ的)で持ちつつ、そこから疎なベクトル要素をデコードする、というハイブリッドが定番です。つまり「内部表現は密、出力は疎なベクトル」という形に収束しつつあります。さらに、ベクトル要素を大きな運転 Transformer のトークンとして扱い、地図・他車・自車を同じ系列の中で attention させる統合表現が、知覚から予測・計画までを一気通貫で扱う方向として研究されています。地図構築のオンライン化を時間方向に拡張し、複数フレームの観測を貯めて地図を時系列で安定化させる stream/temporal な手法(StreamMapNet など)も登場しています。表現選択の問いは「どちらか」ではなく「どこを密に、どこを疎に持つか・いつ密からいつ疎へ落とすか」という設計問題へと深化しています。
