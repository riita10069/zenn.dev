# BEV融合の num_views ランタイム動的化

## ひとことで言うと
車に付いているカメラの台数は、データセットや車種ごとに異なります(あるデータは7台、別のデータは8台、また別のは6台)。モデルを作るとき(コードを `__init__` で組み立てる瞬間)にカメラ台数を定数として埋め込んでしまうと、その1つの台数にしか使えません。これを「実行時(データを流す瞬間)に渡されたカメラ台数に合わせて柔軟に動く」ように設計し直すのが num_views のランタイム動的化です。これができると、1つのモデルでカメラ台数の違う複数のデータセットを学習・推論できるようになります。

## 直感的な理解
本棚を作る大工さんを想像してください。設計図に「棚板は7枚」と書き込んで組み立てると、その本棚は7段専用です。8段欲しくなったら作り直すしかありません。一方、レールだけ通しておいて「棚板は後から好きな枚数を差し込める」設計にしておけば、同じ本棚で7段にも8段にもできます。

ニューラルネットのカメラ融合も同じです。構築時に「カメラは7台」と前提した重み行列を作ってしまうと、8台のデータが来たとき形が合わずクラッシュします。レール(計算のしくみ)はカメラ台数に依存しないように作り、棚板の枚数(台数)は実行時に決める——これがランタイム動的化の発想です。空間クロスアテンション(BEVの各マスが見えているカメラから情報を吸い上げる仕組み)は、もともと「見えているカメラの集合にわたって平均する」形をしているので、原理的にはカメラ台数に依存しません。実装の細部に残った「7」という定数を取り除く作業が本題です。「原理は台数非依存なのに実装が台数を固定してしまう」という、設計思想と実装の乖離をどう埋めるか、という話だと捉えてください。

## 基礎: 前提となる概念

用語を順に噛み砕きます。

- **num_views(ビュー数、V)** とは、入力するカメラ画像の枚数のことです。前方・後方・左右など7方向のカメラがあれば V=7。記号 `V` で書きます。

- **構築時(construction time)** とは、Python でモデルのクラスを `__init__` で組み立てる瞬間のことです。ここで `nn.Linear(...)` のような層を作ると、その重み行列の大きさが固定されます。一度作った重みの形は、学習や推論の途中で勝手には変わりません。重みの形が固定されるのは、最適化(勾配降下)がパラメータの形を前提に動くからで、これ自体は正しい設計です。問題はそこに「カメラ台数」という、本来データ側の量を混ぜてしまうことです。

- **実行時(runtime / forward)** とは、組み立てたモデルに実際のデータを流す `forward()` の瞬間のことです。ここで初めて「今回のバッチはカメラ何台か」が分かります。だからカメラ台数のような「データに付随する量」は、構築時の定数ではなく実行時にテンソルの形から読み取るのが筋です。

- **reshape** とは、テンソル(多次元配列)の中身の数値はそのままに、形(各次元のサイズ)を並べ替える操作です。たとえば28個の数を `[7, 4]` にも `[4, 7]` にもできますが、`[8, 4]`(=32個必要)には並べ替えられません。要素数が合わないとエラー(`RuntimeError: shape '[...]' is invalid for input of size ...`)になります。

- **空間クロスアテンション(SCA)**。BEVの各格子マスが、自分の場所を映しているカメラを射影で特定し、そのカメラ画像から特徴を吸い上げる仕組み。詳細は [BEVFormer 風 空間クロスアテンション(SCA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0935_bevformer-spatial-cross-attention)。

問題の核心はこうです。BEV融合モジュールを構築時に「num_views=7」で作ると、内部のいくつかのパラメータや計算がこの「7」に固定されます。この状態で8台のデータ(V=8)を流すと、形が合わずクラッシュします。

数字で見ましょう。各格子マスの高さ方向のサンプル点数(後述のピラー点数)を num_points_in_pillar=4 とします。

- 構築時に num_views=7 で注意重み(attention weights)を出す全結合層を作ると、出力サイズは `7 × 4 = 28` に固定されます。
- forward で V=8 のデータが来ると、その28個を `8 × 4 = 32` の形に reshape しようとして、要素数が合わずエラーになります(28 ≠ 32)。

この「台数固定」のせいで、カメラ構成ごとに別々のモデルを作るしかなくなります。公開データセットは台数がバラバラ(7台、8台、5台、6台など)なので、これらをまとめて1つのモデルで学習できません。

## 仕組みを詳しく

BEV融合の中心的な考え方は、もともと「カメラ台数に依存しない」ように設計できます。

- 学習可能な BEVクエリは、固定の `bev_h × bev_w` 格子(たとえば 200×200)の上に置かれます。これはカメラ台数とは無関係です。
- 各 BEVクエリが「見えているカメラ」だけに注目し、特徴を集めます。
- 出力は V が何であろうと常に `[B, embed_dim, bev_h, bev_w]`。

つまり「何台のカメラからでもBEV特徴を作り、同じ形で出す」ことが原理的に可能です。問題は実装の細部に固定値 `num_views` が残ることだけ。典型的な実装では、次の3か所が修正対象になります。

### 1. 射影結果の reshape

3次元参照点をカメラに射影した結果を、ビュー次元で並べ替える箇所:

```python
# before(壊れる): 構築時の固定値を使っている
ref_2d      = ref_2d.reshape(B, num_views, N, num_z, 2)
valid_depth = valid_depth.reshape(B, num_views, N, num_z)

# after(動的): 射影行列の実際のビュー次元から V を取る
V_runtime = proj.shape[1]           # 今回のバッチの実際の台数
ref_2d      = ref_2d.reshape(B, V_runtime, N, num_z, 2)
valid_depth = valid_depth.reshape(B, V_runtime, N, num_z)
```

ここで構築時の固定値(=7)を使っていると V=8 で壊れます。修正は「固定値ではなく、射影行列 `proj.shape[1]` から実際の V を取って reshape する」こと。こうすれば V=8 でも正しく `[B, 8, N, num_z, 2]` に並びます。上流の射影計算(典型的には einsum や行列積)は、渡された `camera_params` のビュー次元から V を取っていれば、すでに動的です。

### 2. attention_weights の固定幅

```python
# before: 出力幅が num_views に固定(=28)
self.attention_weights = nn.Linear(embed_dim, num_views * num_points_in_pillar)

# after: 1カメラ分だけの幅にして、forward で各カメラに同じ層を適用
self.attention_weights = nn.Linear(embed_dim, num_points_in_pillar)
# forward 内: for v in range(V): w_v = self.attention_weights(query)  # 重み共有
```

これが「28個固定」の正体です。修正は、出力を `num_points_in_pillar` 個だけ(つまり1カメラぶん)にして、forward の `range(V)` ループの中で各カメラに同じ層を適用すること。カメラ間の重み付けは別途「可視カメラ数で平均する(1/|V_hit|)」方式(SCA と同じ)で行うので、台数ぶんの固定ヘッドは不要になります。これは設計の美しい点で、SCA がカメラ間を学習重みではなく可視数の平均で扱うからこそ、台数依存のパラメータが消せるのです。逆に言えば、もしカメラごとに別々の重みを持たせる設計(per-camera embedding など)を採ると、その瞬間に台数非依存性が崩れます。

### 3. 校正なしフォールバックの固定形

```python
# before: 形が num_views に固定された擬似射影パラメータ
self.pseudo_projection = nn.Parameter(torch.randn(num_views, 3, 4) * 0.01)
```

これは本物の校正情報(camera_params)が無いときの代替で、形が `[num_views, 3, 4]` に固定されています。多データセット学習ではどのみち各データセットの本物の校正が必要なので、修正方針は2つ。(a) V が構築時の num_views と違うときは camera_params を必須にしてフォールバックを使わせない、(b) フォールバック自体を台数に依存しない生成(たとえばビューごとに同じテンプレートを複製して V 個に増やす)に書き換える。一般には(a)が安全です。擬似射影はそもそも幾何的に正しくない代替なので、本番の多データセット学習で常用するものではありません。

### バッチ内の制約は残る

重要な注意点として、1つのミニバッチの中では V を揃える必要があります。理由は単純で、7ビューのサンプル `[7,3,H,W]` と8ビューのサンプル `[8,3,H,W]` を `torch.stack` で1つのテンソルにまとめられないからです(最初の次元が違う)。だから多データセット学習は「バッチごとに V を揃え、データセット単位で V を切り替える(均一バッチを交互に流す)」やり方になります。V動的なモデルならこれを自然に処理できますが、V固定のモデルではそもそも8ビューバッチを受け取れません。この戦略の詳細は [カメラ数が異なる多データセット学習](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1212_multi-dataset-training-homogeneous-batch) を参照。

### before / after の要約

```
[before] num_views=7 固定モデル
  V=7 バッチ → OK
  V=8 バッチ → reshape 28→32 で RuntimeError(形不一致)

[after] V 動的モデル
  V=7 バッチ → OK  出力 [B,256,bev_h,bev_w]
  V=8 バッチ → OK  出力 [B,256,bev_h,bev_w]
  V=6 バッチ → OK  出力 [B,256,bev_h,bev_w]
  ※ ただし 1 バッチ内は V 均一が前提
```

## 手法の系譜と主要論文

カメラ台数に依存しない融合という発想の系譜です。

- **クエリ集約の原型: DETR3D (Wang et al., CoRL 2021, arXiv:2110.06922)**。3Dの物体クエリを各カメラに射影して特徴を引き、可視カメラ分を集約する手法。クエリ数とカメラ台数を別々の量として扱う設計の先駆けで、「クエリは固定、カメラは可変」という分離思想の源流のひとつです。

- **BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)**。BEVクエリが各カメラに空間クロスアテンションで注目する設計の出典。論文中の式は「可視カメラ集合 V_hit にわたる平均」で書かれており、もともとカメラ台数を変数として扱っています。動機は「複数カメラを統一BEV座標に集約したい」、効果は「カメラ台数や配置に原理的に依存しない融合」、トレードオフは「各カメラの校正行列が正確に必要」。num_views 動的化は、この論文の設計を実装でも忠実に再現する作業だと言えます。

- **Deformable DETR (Zhu et al., ICLR 2021, arXiv:2010.04159)**。BEVFormer の deformable attention の基礎。クエリの周辺の少数のサンプル点だけに注目することで計算を軽くする発想で、サンプル点数や注目先の数を可変に扱えます。可変長の入力に強いアテンション設計の源流です。

- **PETR / PETRv2 (Liu et al., ECCV 2022 / ICCV 2023, arXiv:2203.05625, arXiv:2206.01256)**。明示的な射影の代わりに、カメラの3D位置エンコーディングを画像特徴に足し、アテンションで暗黙に対応づける手法。クエリ側がカメラ台数に依存しない設計を取れることを示し、台数可変のBEV系の根拠のひとつになっています。位置エンコーディングはカメラごとに計算するだけなので、台数が増えてもクエリ側の構造は変わりません。

- **UniAD (Hu et al., CVPR 2023, arXiv:2212.10156)**。BEVFormer の台数非依存BEVをエンコーダとして使い、検出・追跡・地図・予測・計画を積み上げる統合モデル。下流タスクが「台数に依存しない統一BEV」の上に乗ることで、入力カメラ構成の違いを下流に波及させない設計になっています。

近年の自動運転基盤モデル(foundation model)研究全般が「カメラ配置・台数に依存しない表現」を志向しており、台数固定はその思想に反する実装上の制約と位置づけられます。

## 論文の実験結果(定量データ)

num_views 動的化そのものを評価したベンチマークは多くありませんが、関連する定量的根拠を挙げます。

- **BEVFormer の台数・カメラ依存性アブレーション(nuScenes, arXiv:2203.17270)**。BEVFormer は6カメラ構成の nuScenes で NDS 約0.56、mAP 約0.42を報告しました(NDS は位置・大きさ・向き・速度の誤差を総合した0〜1のスコア、高いほど良い)。論文のアブレーションでは、空間クロスアテンションを単純な特徴貼り合わせに置き換えると性能が落ちることが示され、可視カメラを射影で正しく選ぶこと(=台数を変数として扱う設計)が効いていると分かります。逆に言えば、貼り合わせ(台数を固定的に concat する素朴な方法)は精度面でも劣るので、台数動的化は「保守性」と「精度」の両方の理由から望ましいわけです。

- **多データセット混合学習の一般的知見**。複数のカメラ構成のデータを統合して学習すると、単一データセット学習に比べ初見ドメインでの汎化が向上することが、自動運転基盤モデル研究で繰り返し報告されています(具体的な改善幅は混合の仕方・指標により数ポイント〜十数ポイントと幅があり、報告による)。台数動的化はこの混合学習を可能にする前提技術であり、それ自体の性能利得は「混合学習を回せること」を通じて間接的に現れます。

定量データの読み方として重要なのは、num_views 動的化は新しい精度を出す手法ではなく「より多くのデータで学習する道を開く実装上の解放」だという点です。利得は下流の汎化指標(初見データでの L2誤差や衝突率の低下)に現れます。だから動的化単体を A/B 評価しても差は見えにくく、評価は「混合学習を回した最終モデルの汎化」で行うのが正しい設計です。

## メリット・トレードオフ・限界

- **メリット**: 1つのモデルでカメラ台数の異なる複数データセットを学習でき、データ量が増えて汎化しやすくなります。カメラ構成ごとにモデルを量産する必要がなくなり、保守・デプロイが単純になります。
- **メリット(推論時)**: 車種ごとにカメラ台数が違っても同じモデルを使える可能性が出ます。
- **トレードオフ**: ミニバッチ内は V を揃える制約が残るため、データローダ側でデータセット単位のバッチ構成・サンプリングが必要になります。実装の複雑さがモデルから上流(データパイプライン)に移ります。
- **トレードオフ**: 台数が変わると可視カメラ数 |V_hit| も変わるため、1/|V_hit| 平均の挙動がデータセット間で微妙に変わりえます。学習の安定性に注意が必要です。
- **トレードオフ**: 校正なしフォールバックの扱いを決め切る必要があります(校正なしパスをどこまで残すか)。
- **限界・未解決課題**: カメラの台数だけでなく、解像度・画角・配置(取り付け位置)もデータセット間で違います。台数を動的にしても、画角や配置の違いは校正行列(camera_params)に押し込むことになり、極端に異なる配置をまたいだ汎化がどこまで効くかは未解明です。動的バッチサイズと組み合わせると、GPUメモリのピークが V に応じて変動し、メモリ管理が難しくなる点も実務上の課題です(V が大きいデータセットのバッチで OOM が出やすい)。

## 発展トピック・研究の最前線

- **完全に配置非依存な表現**。カメラの台数も配置も問わず、各カメラの校正をトークン化してアテンションに食わせる(カメラを「位置情報つきの観測者」として一般化する)方向。PETR系の3D位置エンコーディングはその一歩です。
- **可変長バッチの効率化**。V が異なるサンプルを padding + mask で同一バッチに詰める(ダミーカメラを足して見えないマスクで無効化する)アプローチ。`torch.stack` の制約を回避できますが、無駄計算とのトレードオフがあります。
- **センサ脱落への頑健性**。学習時にランダムにカメラを落とす(view dropout)ことで、推論時にカメラ故障が起きても動くモデルを作る研究。台数動的化はこの種の頑健性訓練の実装基盤になります。
- **クロスデータセット評価**。あるデータセットで学習し別のデータセットでゼロショット評価する設定が、台数・配置非依存性の真の検証として重視されつつあります。
- **可変解像度との両立**。台数だけでなく入力解像度も実行時に変える設計。位置エンコーディングや格子サイズを解像度から動的に生成する工夫が研究されています。
