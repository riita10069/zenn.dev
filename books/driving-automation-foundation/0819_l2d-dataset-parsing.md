# L2D LeRobotデータセットのパース

## ひとことで言うと
運転データセットには、複数台のカメラ映像と、一定間隔で記録された車の状態(速度・加速度・進行方向など)が入っています。これを機械学習モデルに渡すには、提供元ごとにバラバラな生データを「モデルの入力口の形」にそろえる必要があります。データセットのパース(parsing、解析・変換)とは、この生データを読み込み、固定形状のテンソル(数値の多次元配列)に変換する一連の処理を指します。L2D はその具体例として、6台の周囲カメラ + 1枚の俯瞰マップ画像と、10Hz(1秒間に10回)で記録された車両状態をセットにした実車データセットです。

## 直感的な理解
料理にたとえると、データセットは「産地ごとに袋詰めもサイズもバラバラな野菜」です。レシピ(モデル)は「皮をむいて 5cm 角に切った状態」で材料が届くことを前提にしています。パーサは、この下ごしらえ(皮むき・カット・計量)を一手に引き受ける係です。下ごしらえさえ統一しておけば、産地(データセット)が変わってもレシピ本体は一切変えずに済みます。

具体的に何が「バラバラ」かというと、(1)カメラ映像は MP4 などの圧縮動画で、1フレームずつ取り出してデコードしないと画像にならない、(2)解像度がデータセットごとに違うのでリサイズと正規化が要る、(3)記録されている車両信号も「速度はあるが曲率はない」のように欠けがある、(4)学習では「過去を見て未来を予測する」ので、ある瞬間を中心に過去と未来を切り出す必要がある、といった点です。これらを毎回モデル側で処理していたら、データセットを増やすたびにモデルを書き換える羽目になります。だからパーサに閉じ込めます。

## 基礎: 前提となる概念

### LeRobot 形式
LeRobot は Cadene ら(Hugging Face, 2024)が公開した、ロボット学習向けの標準データ/モデル形式です。観測(observation、カメラ画像やセンサ値)と行動(action、関節角やハンドル操作)を時系列で統一的に扱える点が特徴で、カメラ映像は MP4 動画、低次元の状態量は表形式(Parquet など)のカラムで格納されます。エピソード(episode、一続きの記録)単位で区切られ、各エピソードはタイムスタンプ付きのフレーム列になっています。L2D はこの LeRobot 形式で配布されている実車運転データセットです。

### PyTorch の Dataset クラス
パーサは多くの場合 PyTorch の `Dataset` クラスとして実装されます。これは `dataset[i]` と書くと i 番目のサンプルを返してくれる仕組みで、`__len__`(総数)と `__getitem__`(i 番目の取り出し)の2つを実装します。学習ループは中身の形式を一切気にせず、ただ番号でアクセスするだけで済みます。これと `DataLoader` を組み合わせると、バッチ化・シャッフル・並列読み込みが自動で行われます。

### テンソルの形状(shape)
テンソルの「形状」は次元ごとの大きさの組です。たとえば `(7, 3, 256, 256)` は「7視点 × RGB3チャンネル × 高さ256 × 幅256」を意味します。モデルは固定形状を前提に作られるので、すべてのサンプルを同じ形にそろえることが下ごしらえの目的です。

## 仕組みを詳しく

典型的な L2D パーサが返す1サンプルは、次のような固定形状のテンソル群です(視点数を V、過去・未来の各長さを 64 ステップとした例)。

```
visual_tiles        (7, 3, 256, 256)  7視点 × RGB3ch × 高さ256 × 幅256
egomotion_history   (256,)            過去64ステップ × 4信号 = 256
visual_history      (896,)            過去フレームの視覚要約ベクトル
trajectory_target   (128,)            未来64ステップ × 2信号 = 128
episode_index       int               どのエピソードか
frame_index         int               その中の何フレーム目か
```

### カメラ: 複数視点を1つのテンソルに
L2D の視点構成は、6台の周囲カメラ(front_left, left_forward, right_forward, left_backward, rear, right_backward)と、1枚の俯瞰(BEV, Bird's-Eye-View)マップ画像です。

各カメラの1フレームは uint8(0〜255 の整数)の HWC 配列(高さ×幅×チャンネル)で取り出されます。これを PIL 画像に変換し、リサイズ + 正規化を行う前処理を通して `(3, 256, 256)` のテンソルにします。7視点を重ねて `(7, 3, 256, 256)` にします。

設計上の要点が一つあります。俯瞰マップ画像も、他のカメラとまったく同じ前処理を通し、同じ形状にそろえる方式がよく採られます。こうするとモデルから見ると「視点が7つある」のと区別がつかず、入力契約がシンプルになります。マップを特別扱いせず一般の視点として流すわけです。

正規化(normalization)とは、画素値をモデルが学習しやすい範囲・分布に直す処理です。一般には ImageNet で計算された平均と標準偏差で `(x - mean) / std` を行います。画像分類で事前学習されたバックボーン(backbone、特徴抽出を担う中核ネット)は、この分布で学習されているため、入力もそれに合わせる必要があります。前処理の解像度・平均・標準偏差は、使うバックボーンの標準設定から自動取得する実装が一般的で(timm の `resolve_model_data_config` 等)、こうしておくとバックボーンを差し替えても前処理が自動で追従します。

### 車両状態: 生センサーから必要な信号を導出
L2D の生の車両状態は、おおむね次のような列(10Hz)で記録されています。

```
observation.state.vehicle (T, 8):
  [speed, heading, heading_error, lat, lon, alt, accel_x, accel_y]
```

ここから、モデルが扱う4信号を計算で導出するのが典型です。

- speed(速度): そのまま使用。
- accel_x(前後加速度): そのまま使用。
- yaw_rate(ヨーレート、向きの変化速度): `heading`(進行方向の角度)を `np.unwrap`(±π をまたぐときの段差をならす処理)してから、隣り合うフレームの差を時間刻み `dt = 0.1` 秒で割って求める。`yaw_rate[i] = (heading[i] - heading[i-1]) / 0.1`。
- curvature(曲率、カーブの曲がり具合): `curvature = yaw_rate / speed`。停車中(speed ≈ 0)は0除算になるため、speed が極小のときは曲率を0にする保護を入れる。

この導出にはノイズ増幅という落とし穴があり、詳細は [自車運動と曲率の導出](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0808_egomotion-curvature-derivation) で扱います。

### 時系列の切り出し: 過去64 + 未来64
ある1フレームを「現在」と決め、その前 64 ステップ(6.4秒)を history、後ろ 64 ステップ(6.4秒)を future として切り出します。10Hz なので 64 ステップ = 6.4 秒です。

- egomotion_history: history の4信号すべてを平坦化 → `64 × 4 = 256` 次元。
- trajectory_target: future からは `acceleration_x` と `curvature` の2信号だけを選び平坦化 → `64 × 2 = 128` 次元。これがモデルに当てさせる「未来の軌跡」の正解(教師信号)です。

サンプルとして有効なのは「前後に十分な文脈があるフレーム」だけなので、1エピソードあたり最低 `64 + 64 + 1 = 129` フレーム必要です。エピソードの境界をまたいで切り出すと別の走行の未来を正解にしてしまうため、エピソード単位で有効サンプルだけをインデックス化するのが重要です。ここで、部分ロード時のオフセットずれを避けるために、メタ情報ではなく実データの `episode_index` カラムを境界の正とする、といった実装上の配慮が要ります。

### デコードコストとシャード化
MP4 のフレームデコードは計算コストが高く、学習のたびに毎回デコードすると I/O とデコードが律速になって GPU が遊びます。実務では、一度デコードして前処理した結果を WebDataset などのシャード(shard、tar でまとめた連番ファイル)に事前抽出しておき、学習時は軽い読み込みだけで済ませる方式が一般的です。WebDataset(Aizman ら)はクラウドストレージから順次ストリーミングで読める形式で、大規模学習で広く使われます。

## 手法の系譜と主要論文

データセットパースそのものは論文の主役になりにくい裏方ですが、「どんな入出力をモデルに与えるべきか」という設計はエンドツーエンド運転の定式化に直結します。

- UniAD(Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156): 知覚・予測・プランニングを1つのネットに統合し、最終的に自車軌跡を出力する。マルチカメラ画像 → BEV 特徴 → 各種タスククエリ → 軌跡、というパイプラインを示し、「複数カメラ + 自車状態 → 未来軌跡」という入出力契約の代表形を確立した。
- VAD(Jiang et al., ICCV 2023, arXiv:2303.12077): シーンをベクトル化して扱い、軌跡の滑らかさ・実行可能性を重視。低次元の運動学的表現で軌跡を出す流れを推し進めた。
- LeRobot(Cadene et al., 2024): 観測と行動を統一フォーマットで配布する標準を作り、データセット横断の学習を容易にした。L2D はこの上に乗る。

「過去の運動と画像を見て未来の軌跡を出す」という入出力設計は、これらの定式化に沿っています。軌跡を生の (x,y) ではなく加速度・曲率の系列で表すのは、運動学的に扱いやすく破綻しにくいからです(理由は [自車運動と曲率の導出](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0808_egomotion-curvature-derivation))。

## 論文の実験結果(定量データ)

パース処理自体に固有のベンチマーク数値はありませんが、関連して重要な定量的事実を挙げます。

- データ読み込みの律速: 大規模マルチカメラ学習では、MP4 デコードと前処理が GPU 計算を上回ることがしばしば起きます。事前抽出シャード化により、エポックあたりの学習時間が数倍改善する例が報告されています。GPU 利用率(GPU が遊ばず計算している時間の割合)が事前抽出前は数十%にとどまるのに対し、シャード化後は 90% 超まで上がるといった改善が典型です。
- 軌跡表現の選択: UniAD は nuScenes ベンチマークで、計画指標である L2 誤差(予測軌跡と正解軌跡の平均距離、メートル)と衝突率(collision rate、予測軌跡が他物体と衝突する割合)で当時の最先端を更新しました。報告では平均 L2 誤差・平均衝突率ともに先行手法を下回り、知覚から計画までを一体学習する効果を定量的に示しています。これは「どんなテンソルをモデルに与え、何を正解にするか」という入出力設計が最終性能に直結することの裏付けです。
- 視点を統一形状にする効果: マップやレーダなど異種の入力も「視点」として同形テンソルに揃える設計は、データセット混在学習を可能にし、データ量と多様性の増加を通じて汎化性能を底上げします(混在学習の効果はマルチタスク・マルチドメイン学習の一般的知見)。

## メリット・トレードオフ・限界

メリット
- 複数視点と車両状態を固定形状テンソルにそろえるので、学習ループはデータ形式を気にしなくてよい。
- マップ画像をカメラと同一前処理で扱うため、入力契約がシンプル(全部「視点」)になる。
- 過去/未来の切り出しと信号導出を1か所に集約し、有効サンプルだけを自動で抽出できる。
- バックボーンの前処理を自動取得すれば、バックボーン差し替えに強い。

トレードオフと限界
- カメラ外部パラメータ(取り付け位置・向き)が無い、または暫定の場合、BEV 融合のように幾何を使う処理では実キャリブレーションが必須になる(詳細は [カメラ外部パラメータとデータセット非依存設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0939_camera-extrinsics-dataset-agnostic))。連結や注意ベースの融合では不要。
- MP4 のフレームデコードはコストが高く、毎回デコードすると学習が遅い。事前抽出シャードで回避するのが定石だが、ストレージと前処理の手間が増える。
- 曲率は有限差分で導出するためノイズに弱い(詳細は [自車運動と曲率の導出](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0808_egomotion-curvature-derivation))。
- history/future を固定長(各 6.4 秒など)にすると、極端に短いエピソードは丸ごと捨てられ、データが減る。
- 固定解像度へのリサイズ・クロップは、遠方の小さな物体の情報を失わせる可能性がある。解像度とメモリのトレードオフが常につきまとう。

## 発展トピック・研究の最前線

- 可変視点数への対応: データセットごとにカメラ台数が違うとき、視点数 V を可変にしてパディングやマスクで吸収する設計。これにより複数データセットを真に混ぜて学習できます([カメラ外部パラメータとデータセット非依存設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0939_camera-extrinsics-dataset-agnostic))。
- 効率的な動画デコード: NVDEC などのハードウェアデコーダや PyAV による高速デコード、フレーム間引きやキーフレーム活用で I/O を削る研究。
- データバージョニング: LakeFS や DVC のようなツールで、データセットの版を Git のように管理し、再現性を担保する流れ。大規模 MLOps では前処理パイプラインの版管理が再現性の要になります。
- 自己教師あり前処理: 生データから疑似ラベル(深度、オプティカルフロー、未来フレーム)を自動生成し、追加のアノテーションなしで豊かな教師信号を作る方向。世界モデル([GAIA-1(生成的世界モデル)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1214_gaia-1-generative-world-model))とも接続します。
