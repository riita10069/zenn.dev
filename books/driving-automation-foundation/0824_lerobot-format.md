# LeRobot データフォーマット (v2.1/v3.0)

## ひとことで言うと

LeRobot フォーマットとは、ロボットや自動運転車が記録したデータ(カメラ映像・速度・ハンドル操作など)を、誰でも同じやり方で読み書きできるように決めた「データの並べ方のルール」です。映像は動画ファイル(mp4)に、数値は表形式ファイル(parquet)に分けて保存し、データセット全体の説明を JSON のメタデータで持ちます。HuggingFace が公開したロボット学習用の統一フォーマット兼ライブラリで、運転や操作の大規模データセットが続々とこの形式で配布されています。

## 直感的な理解

機械学習でモデルを訓練するには大量のデータが要ります。自動運転なら「このカメラ映像のとき、人間はこうハンドルを切った」という記録が何百万フレームも必要です。

ところが従来は、データセットごとに保存形式がバラバラでした。あるデータセットは映像を画像ファイル数百万枚のフォルダで配り、別のは独自バイナリ、また別のは CSV と動画が中途半端に混ざっている、という状況です。研究者はデータセットを使うたびに専用の読み込みコードをゼロから書く必要があり、貴重な時間の多くが「データを読むだけのコード」に消えていました。さらに、各自が独自に前処理を書くため、同じデータセットでも研究によって微妙に違う読み方になり、結果の再現や公平な比較が難しくなっていました。これはソフトウェア工学でいう「車輪の再発明」が分野全体で繰り返されている状態で、研究の進歩を確実に遅らせます。

LeRobot の発想は「どんなロボット・どんな車のデータも、同じ並べ方で保存し、同じ1行のコードで読めるようにしよう」というものです。これにより、データを読むコードを毎回書き直す手間が消え、研究間の比較も揃います。画像処理で ImageNet という共通データセットが分野を加速させたように、ロボット学習でも「共通のデータの土台」を作ろうという思想です。

ここで用語を噛み砕きます。
- データセット(dataset): 学習に使うデータのまとまり。例: 運転の記録が1万エピソード分。
- エピソード(episode): 1回の連続した記録のかたまり。例: 1回のドライブ(信号で止まり、また走り出す数分間)。
- フレーム(frame): ある一瞬のデータ。例: 0.1秒ごとに撮った1枚のカメラ画像と、その瞬間の速度・ハンドル角。
- 特徴(feature): 1種類のデータ。例: 前カメラ映像、車両状態の数値ベクトル、運転操作。

## 基礎: 前提となる概念

parquet(パーケイ): 「列ごとにまとめて圧縮して保存する表形式ファイル」です。普通の CSV は1行ずつ(行指向)保存しますが、parquet は列指向で、同じ列の値をまとめて格納します。これには2つの利点があります。第一に、同じ種類の数値が並ぶので圧縮がよく効きます(似た値が連続するとデータ圧縮アルゴリズムが効率良く縮められる)。第二に、「速度の列だけ全部欲しい」というとき、映像など他の列を一切読まずに済むので高速です。これを列プルーニング(column pruning)と呼びます。Apache が定めたオープン規格で、データ分析の世界では標準的に使われています。

動画エンコード: カメラ映像を画像1枚ずつ(PNG/JPEG)で保存するとファイル数が爆発し、容量も食います。連続したフレームは隣り合うフレーム同士がよく似ているので、その差分だけを記録する動画圧縮(H.264 / H.265 / AV1 など)を使うと、容量を劇的に削減できます。代償として、特定の1フレームを取り出すにはデコード(動画を画像に展開する計算)が必要で、ここに後述するコストが生じます。動画は数フレームおきに「完全な画像(キーフレーム / I フレーム)」を置き、その間は差分で表すため、任意の1フレームを得るには直前のキーフレームまで遡ってデコードする必要があります。

模倣学習(imitation learning): 人間のお手本(観測と、そのときの行動の組)を真似るようにモデルを学習する枠組み。「この画像のとき人間はこう操作した」という対応を大量に見せ、似た画像で似た操作を出せるようにします。LeRobot はこの模倣学習用データを統一的に配るために設計されています。

## 仕組みを詳しく

LeRobot フォーマットは、1つのデータセットを大きく3種類のファイルに分けます。

1. メタデータ(JSON ファイル群):
   - データセット全体の説明。エピソード数、各特徴のデータ型と shape、フレームレート(Hz)、総フレーム数などが記録されています。
   - 各特徴のキー名は階層的に決まっています。観測は `observation.` 接頭辞、行動は `action`、画像は `observation.images.<カメラ名>`、状態ベクトルは `observation.state.<名前>` のように名付けます。これにより、コードがキー名から「これは画像か数値か」を判別できます。
   - 統計情報(各特徴の平均・分散・最小・最大)もメタデータに持ちます。学習時の正規化(値を平均0・分散1に揃える前処理。これをしないと、桁の大きい特徴が学習を支配してしまう)にそのまま使えます。

2. 数値データ(parquet ファイル):
   - 1行が1フレームに対応し、列が `timestamp`(時刻)、`episode_index`(どのエピソードか)、`frame_index`(エピソード内で何フレーム目か)、`index`(データセット全体での通し番号)、各種センサー値です。
   - 列指向なので、必要な列だけ高速に読めます。

3. 映像データ(mp4 などの動画ファイル):
   - カメラ映像は動画として圧縮します。1台のカメラ・1エピソード(またはチャンク)が1本の動画になるイメージ。
   - 読むときは「このエピソードの何フレーム目」を指定し、その時刻の1フレームだけをデコードします。

具体的な数値例として、運転データセットの典型的な1フレームを示します。車両状態が8次元、運転操作が3次元の場合:

```
observation.state.vehicle  float32  shape [8]:
  [speed, heading, heading_error,
   loc_latitude, loc_longitude, loc_altitude,
   acceleration_x, acceleration_y]

action.continuous  float32  shape [3]:
  [gas, brake, steering]

observation.images.front   uint8   shape [H, W, 3]  (動画から復号)
... (カメラが複数台ぶん)
```

カメラが7台で、フレームレートが 10 Hz(1秒に10フレーム)なら、1秒の運転ごとに parquet 表に10行が増え、各カメラの動画に10フレームが追加される構造です。1時間運転すれば 36,000 行、7本の動画にそれぞれ 36,000 フレームが入ります。

バージョンの違い:
LeRobot は v2.0 → v2.1 → v3.0 と進化しています。最大の違いはファイル分割の粒度です。
- v2.x: 「1エピソード = 1 parquet + カメラごとに1 mp4」という細かい分け方。エピソード数が数万になるとファイルが数万〜数十万個になり、ファイルシステムやクラウドストレージへの大量アクセスが重くなります(小さいファイル問題)。
- v3.0: 複数エピソードを1ファイルにまとめる(チャンク化)方向に変更。ファイル数を減らし、転送と読み込みを効率化しました。これは [WebDataset シャーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0850_webdataset-sharding) が解決しようとする「小ファイル問題」と同じ動機で、コミュニティが大規模化に合わせて辿った自然な進化です。v2.1 では正規化用の統計の持ち方が改善されました。

利用側はライブラリの `LeRobotDataset` クラス越しに読むのが普通で、ライブラリがバージョン差を吸収します。フレーム取得時に「現在フレームの前後 N ステップ」をまとめて返す時間窓(delta timestamps)指定もでき、模倣学習で過去・未来の系列が必要な場面(過去数フレームの動きを見て、未来数フレームの行動をまとめて予測する)に対応します。

## 手法の系譜と主要論文

LeRobot は厳密には論文というよりオープンソースのライブラリ兼フォーマット仕様です(Cadene, Wolf ほか, HuggingFace, 2024)。提案内容は「実機ロボットの模倣学習・強化学習のためのデータ・学習コード・事前学習モデルを、PyTorch エコシステム上で統一的に扱えるようにする」こと。動機は、ロボット学習研究が再現性に乏しく、データ形式が乱立して比較が難しかったことです。効果として、代表的な方策学習手法と、それを学習するデータが同じ枠組みで配布・再利用できるようになりました。

LeRobot が統一的に扱えるようにした代表的な方策学習手法:
- ACT(Action Chunking with Transformers、Zhao, Kumar, Levine, Finn, RSS 2023, arXiv:2304.13705)。1ステップずつ行動を出すのでなく、未来の行動を「ひとかたまり(chunk、例: 100ステップ分)」でまとめて予測することで、模倣学習の誤差蓄積を抑える手法。チャンクで先を見越して行動するので、1ステップごとに判断がブレて発散するのを防げます。CVAE(条件付き変分オートエンコーダ)で人間デモのばらつき(同じ場面でも人ごとに微妙に違う動き)を吸収するのも特徴です。
- Diffusion Policy(Chi, Feng, Du, Xu, Cousineau, Burchfiel, Song, RSS 2023, arXiv:2303.04137)。画像生成で有名な拡散モデル(ノイズから徐々に綺麗な出力を作る生成モデル)を「行動の生成」に転用し、複数の妥当な行動が混在する状況(マルチモーダルな行動分布。例: 障害物を左右どちらから避けてもよい)を上手く表現する手法。
- π0(pi-zero, Black et al., Physical Intelligence, 2024, arXiv:2410.24164)などの大規模 Vision-Language-Action モデルも、LeRobot 形式のデータと相性が良く、ロボット基盤モデルの学習データ供給に使われています。

これらの手法が同じデータ形式・同じローダーで動くことが LeRobot の実用的価値で、研究の再現と比較を一気に容易にしました。

フォーマットの比較対象:
- nuScenes(Caesar et al., CVPR 2020)や Waymo Open Dataset(Sun et al., CVPR 2020)は、それぞれ独自フォーマットで専用 devkit(開発キット)を使って読みます。特定データセット専用です。
- RLDS(Reinforcement Learning Datasets, Google)は TensorFlow 寄りのロボットデータ形式で、Open X-Embodiment という巨大なロボットデータ統合プロジェクトで使われました。LeRobot は PyTorch エコシステム寄りという棲み分けで、RLDS データを LeRobot 形式へ変換する流れも一般的です。
- LeRobot は「特定データセット専用ではない汎用フォーマット」という点が決定的に異なります。一度ローダーを実装すれば、同じフォーマットで配られた別データセットもそのまま読めます。

## 論文の実験結果(定量データ)

LeRobot 自体はベンチマークスコアを競う論文ではないため、ここでは「このフォーマットが支える手法」の代表的な定量結果を、初心者にも分かる形で示します。

ACT(RSS 2023)は、実機の双腕ロボットによる細かい操作(電池の差し込み、カップの設置など)で、1タスクあたり数十回程度の人間デモンストレーションだけから高い成功率(タスクにより 80〜90% 台)を達成しました。アブレーションで、行動を1ステップずつ出す(チャンク長=1)に戻すと成功率が大きく下がる(報告ではタスクによっては数十%まで落ちる)ことが示され、行動チャンク化が誤差蓄積の抑制に効くことが定量的に裏付けられました。これは [CARLAシミュレータ合成データ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0880_carla-synthetic-data) で述べた「模倣学習の誤差蓄積」問題への一つの回答です。また CVAE をなくす(人間のばらつきを吸収しない)と成功率が下がることも示され、デモの多様性を確率モデルで扱う重要性が確認されています。

Diffusion Policy(RSS 2023)は、15 個の操作タスクにまたがるベンチマークで従来の模倣学習手法に対し成功率を平均で相対 46% 程度改善したと報告されています。特に「同じ状況で複数の正解行動がありうる」マルチモーダルな場面で、単一の平均行動を出してしまう従来手法(平均をとると、左に避ける動きと右に避ける動きの中間=障害物に直進、という最悪の行動になる)より優れることが示されています。

データフォーマット性能の観点では、動画エンコードの効果が定量的に効きます。画像を1枚ずつ保存する素朴な方式に比べ、動画圧縮はディスク使用量を概ね1桁以上削減できます(映像内容と圧縮設定に依存)。一方で、ランダムな1フレームのデコードには相応の計算が必要で、多カメラを同時にランダムアクセスするとデコードが律速になります。実務では、ランダムアクセスが頻発する本番学習では、一度デコードして別形式(例: [WebDataset シャーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0850_webdataset-sharding) の tar)に焼き直すことで、このデコードコストを学習の前段に1回だけ閉じ込める設計がよく取られます。

## メリット・トレードオフ・限界

メリット:
- 統一 API(`LeRobotDataset`)でデータセットが読め、専用ローダーを書かなくてよい。研究間の比較が揃う。
- 映像を動画圧縮するので、画像を1枚ずつ保存するよりディスク容量とファイル数が激減する。
- 数値は parquet なので、必要な列だけを高速に読める(列プルーニング)。状態ベクトルだけ欲しいときに映像をデコードせずに済む。
- メタデータに統計を持つので、正規化がすぐできる。delta timestamps で過去・未来の系列も簡単に取れる。
- HuggingFace Hub 経由でダウンロード・キャッシュ・バージョン管理が自動化される。

トレードオフと限界:
- ランダムアクセスのデコードコスト: フレームをランダムに取ると動画デコードのコストが乗る。多カメラだと顕著で、状態数値だけ欲しいのに全カメラを復号してしまう素朴な実装は非常に遅くなる。回避には、状態が入った内部の表から該当列だけを直接読み、映像が必要なフレームだけを復号する、という列単位の最適化が要ります。
- バージョン依存: v2.1/v3.0 でファイル構造が変わるため、ライブラリのバージョンに依存します。配布データのバージョンとライブラリのバージョンが食い違うと読めないことがあります。
- 非可逆圧縮: 動画圧縮は通常非可逆で、画質がわずかに劣化します。厳密なピクセル精度が要る用途(微細な計測、デプス推定の教師など)には不向きな場合があります。劣化を嫌う場合は可逆コーデックやフレーム個別保存に切り替える判断が要ります。
- ロボット中心の設計: もともと実機ロボット操作向けに設計されているため、自動運転特有のメタ情報(地図・他車のラベル・LiDAR 点群など)を素直に表現しきれない場面があり、運転データではキー設計の工夫が必要になることがあります。
- 時間同期の前提: 複数センサーが同じ timestamp に揃っている前提で扱いやすい一方、現実の車載センサーはカメラ・LiDAR・GPS で取得周期がバラバラなことが多く、ingest 時に再サンプリングや補間が必要になります。

## 発展トピック・研究の最前線

- ロボット基盤モデル(Robot Foundation Model): 多種多様なロボット・タスクのデータを LeRobot のような統一形式に集約し、一つの大きなモデル(VLA: Vision-Language-Action)を学習する流れ。Open X-Embodiment(複数研究機関のデータを統合)はその代表で、統一フォーマットの存在がこうした大規模統合を可能にしています。
- データバージョニングとの統合: データセットも「コードのようにバージョン管理」する動き。どのバージョンのデータでどのモデルを学習したかを再現可能にするため、データ用のバージョン管理システムと組み合わせます。
- ストリーミング学習: データを全部ダウンロードせず、Hub からストリームで読みながら学習する方式。データが TB 級になると全ダウンロードが非現実的なため重要になります。
- 自動運転向け拡張: 地図・3D ボックス・点群を LeRobot のキー設計に自然に載せる試みや、車載 ROS ログ([ROS bag / MCAP コンテナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0901_rosbag-mcap))から LeRobot 形式への変換パイプラインの整備が進んでいます。
