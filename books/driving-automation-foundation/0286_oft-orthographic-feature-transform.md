# OFT (Orthographic Feature Transform)

## ひとことで言うと

OFT は、カメラ画像から「真上から見た直交格子(BEV: Bird's-Eye-View、鳥の目線の地図)」の特徴を作る、初期(2019年)の幾何ベース変換手法です。やり方は「BEV のマスを画像に投影し、そのマスに対応する画像領域の特徴をまとめて拾う」というもので、奥行き(カメラからの距離)を確率で予測せず、カメラ校正の幾何だけで特徴を割り当てます。後に続く LSS や M2BEV、Simple-BEV といった BEV 系手法の原型のひとつであり、「3次元物体検出を画像視点ではなく BEV 空間で解く」という現代の主流の流れを切り開いた重要な研究です。

## 直感的な理解

カメラで撮った普通の画像は「前から見た光景」で、近くのものは大きく、遠くのものは小さく写ります(透視投影)。この見え方は人間の目には自然ですが、3次元の位置や大きさを正確に推定するには厄介です。同じ大きさの車でも、20m先と40m先では画像上の見かけのサイズがまるで違うため、「画像のこのサイズ=実際のこの距離・この大きさ」という対応が距離ごとにバラバラになります。畳み込みニューラルネットは「画像のどこでも同じフィルタを使う(平行移動不変)」ことを前提に作られていますが、透視投影ではこの前提が崩れ、同じ物体が距離によって違う見え方をするため学習が難しくなります。

一方、真上から見た地図(BEV、正射影)では、車の大きさは距離によらず一定で、隣り合う車の位置関係も素直に表現できます。3次元の推論をするなら、こちらの空間の方がずっと自然です。Roddick らの着想は明快で、「3次元の推論は、遠近のゆがみがない BEV 空間に移してからやろう」というもの。残る問題は「透視投影の画像特徴を、どうやって正射影の BEV 格子に移すか」で、その変換器こそが OFT です。地図づくりに例えると、斜めから撮った航空写真(画像)を、真上から見た平面図(BEV)に引き直す作業を、カメラの幾何を使って機械的に行う、というイメージです。

## 基礎: 前提となる概念

- 単眼3次元物体検出 (monocular 3D detection): 1台のカメラ画像だけから、物体の3次元的な位置・大きさ・向き(3D バウンディングボックス)を当てるタスク。LiDAR を積まない安価な構成で3次元認識をしたい動機がある。
- 正射影 / 直交投影 (orthographic projection): 遠近を無視して、真上から平行光線で見たような投影。BEV はこの正射影で表現される。距離によって見かけの大きさが変わらないのが特徴。
- 透視投影 (perspective projection): 普通のカメラ画像。近くが大きく、遠くが小さく写る。
- ボクセル (voxel): 3次元空間を区切った小さな立方体のマス。2次元のピクセルの3次元版(volume + pixel)。
- integral image (積分画像、サマリエリアテーブル): 画像の左上から各点までの値の累積和を前計算した表。これがあれば、任意の矩形領域の合計を、4点の値の足し引きだけ(定数時間)で求められる。OFT が領域平均を高速化するのに使う古典的なコンピュータビジョン技法。顔検出の Viola-Jones 法などでも使われた由緒ある技法です。
- カメラ校正 (calibration): カメラの内部・外部パラメータ。これで「3次元点が画像のどこに写るか」を計算できる。
- 帰納バイアス (inductive bias): モデルにあらかじめ組み込んだ「世界はこうなっているはず」という前提。OFT は「カメラ幾何で投影できる」という強い帰納バイアスを持つため、少ないデータでも妥当な対応を作れる。

2019年当時、画像からの3次元検出は「画像平面で2次元検出をして、そこから3次元位置を回帰する」という、画像視点に縛られたやり方が主流でした。上で述べた遠近のゆがみのため、3次元のスケールや位置を正しく扱いにくいという根本的な難しさがありました。同年には、奥行き推定で得た擬似点群を LiDAR のように扱う Pseudo-LiDAR(Wang ら, CVPR 2019)も登場しており、「画像視点のままでは限界がある」という問題意識が分野で共有されつつありました。OFT はこの土俵そのものを BEV に移すことで問題を作り直しました。

## 仕組みを詳しく

### 中心アイデア: BEVボクセルを画像に投影し、対応領域を積分する

OFT は次の手順で画像特徴を BEV(正射影格子)に移します。

1. ボクセル格子の定義: 車の前方に3次元のボクセル格子(直交格子)を置く。各ボクセルは実世界の3次元座標を持つ。

2. ボクセルを画像へ投影: 各ボクセルの3次元的な広がり(立方体)を、カメラ校正を使って画像平面に投影する。投影すると画像上ではある「矩形領域」になる。遠いボクセルは小さい矩形、近いボクセルは大きい矩形になる。

3. 領域内の特徴を集約: その矩形領域に含まれる画像特徴をすべて平均(積分)して、そのボクセルの特徴とする。OFT は計算を速くするために integral image を使い、どんな大きさの矩形でも定数時間で平均を取れるようにした。これにより、奥行きを予測しなくても「このボクセルに対応する画像領域はここ」という幾何的対応だけで特徴を割り当てられる。視線方向に同じ画像領域が複数ボクセルに割り当たる(にじむ)点は、後段の処理で解く。

4. 高さ方向の集約 → BEV: 3次元ボクセル特徴の高さ方向を、学習可能な重みで畳み込んでまとめ、2次元の BEV 特徴(正射影格子の特徴マップ)にする。単純に最大値や平均で潰すのではなく、高さごとの重みを学習することで「地面付近と物体の上部のどちらを重視するか」をデータから決められる。

5. BEV 上での検出: 得られた BEV 特徴に topdown network と呼ぶ2次元 CNN をかけ、BEV 上で3次元物体検出(位置・大きさ・向き)を行う。BEV 上で畳み込むことで、物体同士の配置(駐車車両の並びなど)という空間的文脈を捉えられる。

tensor の形でイメージすると次の通りです。

```
画像特徴:      [C, H, W]                (単眼なのでカメラ1台)
ボクセル格子:   [X, Y, Z] 個の3次元立方体
投影+積分:     各ボクセル → 画像上の矩形 → integral image で平均 → [C, X, Y, Z]
高さ集約:      Z を畳み込んでまとめ → BEV 特徴 [C', X, Y]
topdown CNN:   [C', X, Y] → 3D 検出結果
```

### なぜ integral image なのか(設計の理由)

近いボクセルは大きな矩形、遠いボクセルは小さな矩形に投影されます。素朴に各矩形内のピクセルを足すと、大きな矩形では足す回数が多く、ボクセル数×矩形サイズに比例して非常に遅くなります。integral image を使えば、矩形の大きさによらず4点の足し引きで合計が出るので、全ボクセルの集約が一定コストで済みます。具体的には、矩形の右下・右上・左下・左上の4点の累積和を S_rb − S_rt − S_lb + S_lt のように組み合わせるだけで、何ピクセルを含む矩形でも O(1) で合計が求まります。2019年当時、GPU メモリと速度の制約の中で密なボクセル集約を現実的にするための賢い工夫でした。

### LSS との違い(初心者向けの対比)

- LSS([Lift-Splat-Shoot (LSS)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0282_lift-splat-shoot))は「画像ピクセルごとに奥行き確率を予測して点を3次元にばらまく(画像 → 3次元へ push する)」方向。
- OFT は「BEV ボクセルから画像を見て、対応領域を拾う(3次元 → 画像から pull する)」方向。

どちらも幾何ベースですが、OFT は奥行きを確率で当てず、ボクセル側から画像をサンプリングする「pull 型」の発想で、Simple-BEV や M2BEV、FastBEV の均一サンプリング系の祖先にあたります。pull 型は「BEV のどのマスを埋めたいか」が起点なので出力 BEV のサイズを制御しやすく、push 型は「画像の各点をどこへ飛ばすか」が起点なので奥行きの表現力が高い、という相補的な性質があります。実装面でも、pull 型は出力側のループで完結するため scatter のような書き込み衝突が起きず、並列化しやすいという利点があります。

## 手法の系譜と主要論文

- Roddick, Kendall, Cipolla, "Orthographic Feature Transform for Monocular 3D Object Detection" (BMVC 2019, arXiv:1811.08188)。透視投影の画像特徴を、ボクセル投影 + integral image による領域積分で正射影 BEV 格子へ移す Orthographic Feature Transform を提案。動機は「3次元推論は遠近ゆがみのない BEV 空間の方が自然」。pull 型幾何変換の出発点。
- Wang et al., "Pseudo-LiDAR" (CVPR 2019)。奥行き推定で擬似点群を作り LiDAR 検出器に流す別アプローチ。同時期に「画像視点を脱する」問題意識を共有。
- Philion & Fidler, "Lift, Splat, Shoot" (ECCV 2020, arXiv:2008.05711)。翌年に登場し、逆に奥行き確率を明示予測する push 型を提案。OFT と対で読むと「pull 型 vs push 型」の設計対比が理解できる。[Lift-Splat-Shoot (LSS)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0282_lift-splat-shoot)。
- Harley, Fang, Li, Ambrus, Fragkiadaki, "Simple-BEV" (ICRA 2023, arXiv:2206.07959)。OFT の「ボクセルから画像をサンプリングする」発想を最小限に単純化したベースライン。OFT の積分の代わりに1点の双線形サンプリングを使う。[Simple-BEV](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0293_simple-bev)。
- Xie et al., "M2BEV" (2022, arXiv:2204.05088) / Li et al., "Fast-BEV" (2023, arXiv:2301.12511)。OFT 系の幾何投影をマルチカメラ・大規模・高速化に発展させた後継。M2BEV は均一深度仮定、Fast-BEV は LUT 化で車載リアルタイムへ。[M2BEV](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0284_m2bev_note) / [FastBEV](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0628_fast-bev)。

系譜を一言で言うと、OFT(2019)が「BEV 空間で3次元推論する」という枠組みと「ボクセルから画像を pull する」幾何変換の原型を作り、そこから push 型の LSS、それを単純化した Simple-BEV、マルチカメラ・高速化に進めた M2BEV/Fast-BEV、そして学習で対応を取る BEVFormer 系へと枝分かれしていきました。OFT 自体の精度は現代の手法に及びませんが、「この土俵で解く」という枠組みを最初に示した概念的な貢献は非常に大きいです。

## 論文の実験結果(定量データ)

OFT は主に KITTI(自動運転の古典的ベンチマーク。前方カメラ1台と LiDAR で集めたデータ。約7500フレームの学習・検証用画像)の単眼3次元物体検出・BEV 検出で評価されました。

- AP (Average Precision): 検出精度。正解との重なり(IoU)がしきい値(例: 0.7 や 0.5)以上の検出を正解とみなし、適合率-再現率曲線の下の面積で測る。大きいほど良い。
- BEV AP / 3D AP: それぞれ BEV 平面上のボックス重なりで測る AP と、3次元ボックスの重なりで測る AP。3D AP の方が向きや高さも問われるため一般に厳しい。
- Easy / Moderate / Hard: KITTI は物体の見やすさ(オクルージョン・切れ・大きさ)で難易度を3段階に分ける。Moderate がランキングの基準に使われる。

報告によると、OFT は当時の単眼3次元検出手法と比べて、特に BEV 検出(BEV AP)で競争力のある精度を示しました。値はクラス(Car など)や IoU しきい値で変わりますが、Car の Moderate 設定で BEV AP は数%〜十数%台という、当時の単眼手法としては有力な水準でした(IoU しきい値 0.7 という厳しい条件では単眼検出は全般に難しく、当時の最先端でも一桁〜十数%にとどまっていました)。重要なのは絶対値そのものより、「画像平面で3次元を回帰する従来路線に対し、BEV 空間に移して解く OFT の路線が有効だ」と定量的に示したことです。これがその後の BEV 中心の3次元検出研究の方向性を裏付けました。

アブレーションで示された設計の妥当性として、(1)高さ方向のボクセルを学習的に畳み込んで BEV にまとめる処理が、単純に潰すより精度に寄与すること、(2)topdown CNN を BEV 上にかけることで BEV 平面の文脈(物体同士の配置)を捉えられること、が報告されています。一方、奥行きを当てない均一集約に由来する視線方向のにじみは精度上の制約として残りました。なお KITTI 時代の数値は現代の大規模ベンチマーク(nuScenes 等)とは指標も難易度も異なるため、現在の手法と直接比較する数字としては扱えません。当時の単眼3次元検出はそもそも非常に難しいタスクで、絶対値の低さはタスクの難しさを反映しています。

## メリット・トレードオフ・限界

メリット:
- 3次元推論を遠近ゆがみのない BEV 空間で行えるので、物体のスケール・位置関係を素直に扱える。
- 奥行き確率の予測が不要で、幾何 + integral image による集約はシンプルかつ(矩形サイズによらず)高速。
- 後続の BEV 系手法の土台を作った、概念的に極めて重要な枠組み。
- カメラ幾何という強い帰納バイアスを持つため、学習データが限られていても妥当な対応を作れる。

トレードオフ・限界:
- 奥行きを当てないため視線方向に特徴がにじむ。領域積分(平均)は粗い近似で、細かい奥行き弁別は苦手。視線上のどこに物体表面があるかは後段に委ねられる。
- 単眼前提の元論文では、絶対距離の曖昧さ(スケール不定性。1枚の画像だけからは大きさと距離が一意に決まらない)が残る。
- 2019年の手法で、精度は現代の Transformer ベース BEV(BEVFormer 等)や丁寧な奥行き手法に及ばない。
- カメラ校正が必要(ボクセル投影に使うため)。
- integral image による平均は線形和なので、領域内の支配的な特徴を埋もれさせる(平均で薄まる)恐れがある。

研究上の未解決課題としては、当時から続く「奥行きのにじみをどう解くか」が中心です。これに対し、後続の手法は (a) 奥行きを確率で予測する(LSS)、(b) LiDAR で奥行きを監督する(BEVDepth)、(c) アテンションで対応を学習する(BEVFormer)、といった異なる答えを出してきました。OFT はこれらの分岐の根にある「BEV 空間で解く」という共通前提を確立した点で、系譜の理解に欠かせません。

## 発展トピック・研究の最前線

OFT が示した「BEV 空間に特徴を集約してから認識する」という発想は、その後マルチカメラ・大規模化の波に乗って自動運転知覚の標準アーキテクチャになりました。pull 型(ボクセルから画像を見る)の流れは Simple-BEV による最小化、M2BEV による均一深度仮定とマルチカメラ化、Fast-BEV による LUT 化と車載リアルタイム化へと発展します。一方、奥行きのにじみという OFT の弱点を正面から解こうとする流れは push 型(LSS)に、対応を学習で取る流れは Transformer 型(BEVFormer)につながりました。

なお OFT の著者 Roddick らは続く研究 "Predicting Semantic Map Representations from Images using Pyramid Occupancy Networks"(CVPR 2020)で、BEV セマンティック地図生成へと発想を展開しており、OFT が単眼検出にとどまらない一般的な視点変換の枠組みであることを示しました。現在の研究の最前線では、これらの BEV 表現を3次元検出だけでなく、地図要素のオンライン構築(HD マップ生成)、占有グリッド予測(BEV を超えた3次元占有 Occupancy)、そして end-to-end の運転計画まで一気通貫で扱う方向へ拡張しています。OFT を読むと、これら現代手法が共通して立つ「BEV という土俵」がどこから来たのかが分かります。
