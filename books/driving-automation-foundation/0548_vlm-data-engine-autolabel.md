# VLM オートラベリング・データエンジン

## ひとことで言うと
VLM(Vision-Language Model、画像も文章も理解できる AI モデル)に運転シーンの画像を見せて「この場面で何が起きているか」を文章やタグで自動的に書かせ、それを学習データの正解ラベルとして大量に作る仕組みです。人間が手で書くラベル作業(アノテーション)のコストを桁違いに下げ、データを「増殖」させる工場のように回すので「データエンジン」と呼ばれます。

## 直感的な理解
機械学習モデルを賢くするには「入力(画像)」と「正解(ラベル)」のペアが大量に要ります。自動運転では、たとえば「この画像には歩行者が3人、信号は赤、前の車が急ブレーキ」といった情報が正解にあたります。問題は、この正解を人間が手で付けるのが非常に高コストなことです。

具体的な数字感を持ちましょう。1枚の画像に物体の枠(bounding box)を付けるだけで数十秒〜数分、「なぜ車が止まったか」という因果関係まで文章で書かせると1シーンに数分〜十数分かかります。自動運転の学習には数百万シーンが必要なので、人手では費用も時間も現実的ではありません。さらに、珍しい状況(long-tail、ロングテール=めったに起きないが危険な場面、たとえば横転したトラックや道路上の動物)は、そもそもデータが少なくラベルを集めにくい。

ここで VLM が登場します。VLM は画像を見て質問に文章で答えられます。大規模 VLM に運転画像を見せ、「注意すべき物体を列挙して」「自車が減速すべき理由を述べて」と指示すれば、人間が書くようなラベルを自動生成できます。これが VLM オートラベリング・データエンジンです。ただし VLM の出力は人間の正解ではなく疑似ラベル(pseudo-label)で、間違いを含むので、そのまま信じてはいけません。

## 基礎: 前提となる概念
- アノテーション(annotation): 生データに正解ラベルを付ける作業。教師あり学習の前提で、品質と量がモデル性能を左右します。
- 疑似ラベル(pseudo-label): 人間ではなくモデルが生成したラベル。安価に大量に作れる一方、誤り(ノイズ)を含みます。人間の正解(ground truth)と区別して扱います。
- ロングテール(long-tail): 出現頻度の分布で、頻出する事象(直進、停止)が頭、稀な事象(緊急車両、落下物)が長い尾を作る形。安全上は尾の部分が致命的なのに、データが少なく学習しにくい。
- 幻覚(hallucination): VLM が画像に存在しないものをもっともらしく述べる現象。オートラベリングの最大の品質リスクです。
- VQA(Visual Question Answering): 画像を見て自然文の質問に答える課題。VLM の代表的な評価形式で、運転では「前方の信号は何色か」「自車は何をすべきか」といった問いになります。
- provenance(出所)と confidence(信頼度): 疑似ラベルを管理するためのメタ情報。どのモデルがいつ付けたか(provenance)、どれくらい確からしいか(confidence)を記録し、学習時の重み付けやフィルタに使います。
- アクティブラーニング(active learning): モデルが「自信のないデータ・苦手なデータ」を優先的に選び、そこだけ人間にラベルを付けてもらう枠組み。限られた人手予算を、効果の大きい箇所に集中させる考え方で、データエンジンの「どこを検品するか」と直結します。

## 仕組みを詳しく
典型的なパイプライン(処理の流れ)をステップで分解します。

1. シーン収集: 走行ログから画像/動画フレームを取り出す。たとえば nuScenes(自動運転の公開データセット)の1シーンは6台のカメラ画像と LiDAR(レーザー距離センサー)から成る。
2. プロンプト設計: VLM に何を答えさせるか指示文を作る。例「画像内の交通参加者(車・歩行者・自転車)を列挙し、それぞれの位置と、自車の運転に与える影響を述べよ」。出力形式(JSON のキー)を縛ると下流処理が安定します。
3. VLM 推論: 画像とプロンプトを VLM に入れ、文章や JSON で回答を得る。出力例:
   ```json
   {"objects": [{"type": "pedestrian", "position": "front-left", "risk": "high",
                  "reason": "crossing the road"}],
    "ego_action": "decelerate",
    "scene_tag": "intersection_crossing"}
   ```
4. 構造化・後処理: 自由文の回答を、決まった形式(タグ・クラス ID・座標)に変換する。たとえば `scene_tag` を `intersection / pedestrian / traffic_light / obstacle / clear` のような有限クラスへ写像する。
5. 品質フィルタ: 明らかにおかしい回答(存在しない物体、矛盾した記述)を弾く。VLM 自身に確信度を出させる、複数 VLM の多数決を取る、物体検出器の出力と突き合わせる(グラウンディング)などの手段を組み合わせる。
6. 疑似ラベルとして保存: 出力を confidence と provenance 付きで記録し、学習に使う。

before/after でコスト効果を見ます。

- before(人手): 100万シーンに因果ラベル。1シーン5分なら約8万時間。アノテータ100人をフル稼働させても800時間(約4ヶ月、休みなし換算)。しかも品質はばらつく。
- after(VLM): 1シーン数秒の推論で同じ規模を数日〜数週間で生成可能。人間は「サンプル検品」だけ担当する。

ただし VLM の出力はノイズを含むので、学習では主目的の損失(模倣損失など)より小さい重みの補助損失として使うのが定石です。また label smoothing(正解ラベルを少しだけ滑らかにして過信を防ぐ手法。one-hot の `[0,1,0,0]` を `[0.02,0.94,0.02,0.02]` のように緩める)を併用すると、誤った疑似ラベルへの過剰適合を抑えられます。

データエンジンの本質は「ループを回す」ことにあります。Tesla が AI Day で示した自動ラベリング(オフラインで重いモデルや複数フレーム統合を使い、高品質ラベルを生成してから車載モデルを学習する)の考え方に近く、(1) モデルで疑似ラベルを生成し、(2) モデルを学習し、(3) モデルが苦手なシーン(不一致・低信頼度・稀なケース)を見つけて優先的に集め直し、(4) そこにラベルを付け直す、という反復で、データの質と量を段階的に育てます。ここで VLM は「最初の大量ラベル付け」と「苦手シーンの言語的な説明づけ」の両方を担います。(3) の「苦手シーンの選別」がアクティブラーニングそのものです。

## 手法の系譜と主要論文
- BDD-X (Kim et al., ECCV 2018, arXiv:1807.11546): 運転動画に対し人手で「行動の説明(textual explanation)」を付けたデータセット。VLM 登場前から「運転判断を言語で説明する」課題が定式化されていたことを示し、後のオートラベリングが置き換えようとした人手作業の原点です。
- DriveGPT4 (Xu et al., RA-L 2024, arXiv:2310.01412): 動画を入力に、運転行動とその根拠を言語で出力する VLM を学習。学習データの一部を ChatGPT/GPT-4 で自動生成。「強い基盤モデルで疑似ラベルを作り、専用 VLM を育てる」流れの初期の代表で、論文自身が自動生成ラベルの幻覚とフィルタの重要性を指摘しています。
- NuScenes-QA (Qian et al., AAAI 2024, arXiv:2305.14836): nuScenes の3Dシーンに対して、物体の存在・数・状態・関係を問う VQA データセットとベンチマークを構築。多くの QA を3Dアノテーションから半自動生成しており、「既存ラベルとテンプレートを使って QA を増殖する」オートラベリングの一形態です。評価用の標準的な物差しにもなりました。
- DriveLM (Sima et al., ECCV 2024, arXiv:2312.14150): 運転シーンを「知覚 → 予測 → 計画」の QA グラフ(質問応答のノードを因果でつないだ構造)として表現し、各ノードに VLM 由来の QA を自動・半自動で付与する枠組みを提案。動機は、単発のキャプションでは運転の論理的連鎖(なぜ止まる → 前に歩行者 → だから減速)を表せないこと。グラフ構造で因果の連鎖を持たせ、推論を段階的に学習できるようにし、説明可能性を上げました。トレードオフはグラフ構造のラベル設計が複雑で、自動生成だけでは品質が安定せず人間検証を要すること。
- OmniDrive (Wang et al., CVPR 2025, arXiv:2405.01533): 3D空間に対応した運転用 VLM データセットとその自動アノテーションを提案。動機は、2D画像の説明だけでは「物体が3D空間のどこにあるか」を扱えず意思決定に不十分なこと。GPT-4 等を使い、反実仮想(counterfactual、「もし右に避けていたら」)を含む推論ラベルを自動生成し、3D知覚と言語推論を橋渡ししました。トレードオフは強力な商用 VLM への依存・コストと、生成ラベルの誤り混入。
- 共通の知見: いずれも「VLM で大規模ラベルを作れるが、品質はばらつくので疑似ラベル扱いし、人間検品やフィルタと組み合わせる」点で一致します。系譜としては、人手説明(BDD-X)→ 単発キャプションの自動化(DriveGPT4)→ 構造化 QA(NuScenes-QA)→ 因果グラフ QA(DriveLM)→ 3D・反実仮想つき(OmniDrive)と、ラベルの構造が年々リッチになっています。

## 論文の実験結果(定量データ)
オートラベリングの効果は「自動生成ラベルで学習・評価したモデルの性能」と「ラベルの正確さ・カバレッジ」の二面で測られます。指標の意味とあわせて報告例を示します。

- DriveLM: 提案した DriveLM-Data(QA グラフ)で学習したモデルが、知覚・予測・計画にまたがる QA で従来手法を上回ることを示しました。評価には言語生成スコア(BLEU、CIDEr、ROUGE など、生成文が参照文にどれだけ近いかを測る指標。BLEU と ROUGE は n-gram の精度・再現率、CIDEr は語の重要度重みづけ)と、計画タスクでは軌道誤差が使われます。とくに「未知のフレーム/未知の質問」への汎化(zero-shot 性)で、グラフ構造の段階推論が単発キャプション学習より頑健だと報告されています。これは「因果の連鎖を構造として与えると、丸暗記でなく推論が身につく」ことを示唆します。
- NuScenes-QA: 大規模 VQA ベンチマークとして、回答正解率(accuracy)を質問タイプ別(存在 exist、数え上げ count、物体 object、状態 status、比較 comparison)に報告。とくに count(数え上げ)系で多くのモデルが苦戦することが定量的に示され、VLM が個数や正確な空間関係に弱いという一般的傾向を裏付けます。これはオートラベリングで「個数・座標は信頼度を下げる」運用の根拠になります。
- OmniDrive: 3D グラウンディングを伴う VQA と計画で、2D のみのベースラインに対する優位を示しました。とくに反実仮想推論を含むラベルで学習すると、計画の安全関連サブ指標が改善する傾向が報告されています。3D 位置情報を言語推論に結びつけることで、「どこに何があるか」を踏まえた意思決定ラベルが作れる点が新しさです。

アブレーション的に重要なのは「フィルタ・検証を抜くとどうなるか」です。複数の研究で、VLM の生出力を無検証で教師に使うと、幻覚由来の誤ラベル(存在しない歩行者ゆえの減速など)が下流モデルに伝播し、計画性能が劣化することが示されています。逆に、多数決・検出器とのグラウンディング・信頼度フィルタを入れると、ラベル量を保ったまま誤り率を下げられます。つまり「データエンジンの価値はフィルタとセットで初めて引き出せる」のが共通の結論です。また、疑似ラベルを主損失と同等の重みで使うと過信が害になり、低重み + label smoothing で扱うと安定する、という実務的知見も広く共有されています。

## メリット・トレードオフ・限界
- メリット: アノテーションコスト(費用・時間)を桁違いに削減できる。
- メリット: ロングテールの珍しいシーンにもラベルを付けやすい(VLM は一般知識を持つため)。
- メリット: 因果・理由といった「文章でしか表せない高次のラベル」を作れる。
- トレードオフ: VLM は幻覚を起こすので、ラベルに誤りが混入する。検証・フィルタが必須。
- トレードオフ: 強力な VLM(クローズドな商用 API)は推論コストが高く、データ規模に比例して費用がかさむ。秘匿走行映像を外部に送る制約も生じる。
- 限界: 疑似ラベルは人間の正解と同等に信じてはいけない。confidence/provenance での管理と低重み学習が前提。
- 限界: VLM の偏り(bias)がそのままラベルに乗り、下流モデルへ伝播する危険がある。さらに、自分のモデルが付けたラベルで自分を学習し続けると誤りが固定化・増幅する(self-confirmation / モデル崩壊の一種)危険もある。
- 未解決の課題: ラベルの信頼度を自動でよく較正する方法、どのシーンを人間が検品すべきかの最適選別(active learning との接続)は依然オープンです。

## 発展トピック・研究の最前線
- データフライホイール(data flywheel): 「モデルで疑似ラベル生成 → 学習 → 苦手シーン発見 → 収集・再ラベル」のループを自動化し、データの質と量を雪だるま式に育てる運用。
- グラウンディング強化: VLM 出力を3D検出・占有グリッドと突き合わせて空間的に根拠づけ、幻覚を構造的に減らす方向。
- マルチモデル合議: 複数 VLM の出力を合議・相互検証して信頼度を較正する手法。
- 反実仮想・連鎖推論ラベル: 「もし右に避けていたら」「知覚 → 予測 → 計画」の段階推論を生成させ、計画の理由づけを豊かにする流れ。
- 蒸留への接続: 作った疑似ラベルで軽量モデルを学習する流れは、そのまま [VLM→Planner 知識蒸留](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1178_knowledge-distillation-vlm-to-planner) につながります。
