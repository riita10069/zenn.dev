# 大規模自動運転データセットの PyTorch Dataset 化

## ひとことで言うと
自動運転モデルを学習させるには「カメラ映像」と「正解の運転データ」を大量に読み込む仕組みが必要です。このノートは、生の動画ファイルやセンサーログの山を、PyTorch の学習ループにそのまま渡せる数値の塊 (テンソル) に変換する「データセットクラス」をどう設計するか、という一般論を扱います。データセットの種類 (NVIDIA PhysicalAI、nuScenes、Waymo など) が違っても考え方は共通です。ファイルの山を「学習プログラムが噛める形」に整える翻訳機を作る、と捉えてください。

## 直感的な理解
学習ループから見ると、データセットは「番号 i を渡すと i 番目の練習問題 (入力と正解のペア) を返してくれる箱」であってほしいわけです。中で動画をデコードしようがデータベースを引こうが、外から見れば `dataset[i]` で 1 サンプルが取れる。この「番号で引ける箱」という抽象を作るのが Dataset クラスの役割です。

なぜそんな抽象が嬉しいのか。学習は「ランダムな順番でサンプルを少しずつ取り出し、まとめて (バッチにして) モデルに食わせる」ことの繰り返しです。順番のシャッフル、並列読み込み、バッチ化といった面倒は全部 PyTorch 側のローダーが引き受けてくれます。私たちが書くのは「i 番目を取り出す方法」だけ。だから設計の勝負どころは、(1) 何を 1 サンプルと数えるか、(2) i 番目を取り出すときに何を計算し何を前もって済ませておくか、の 2 点に集約されます。

## 基礎: 前提となる概念

機械学習モデルは、生のファイル (動画 `.mp4` や数値テーブル `.parquet`) をそのまま食べられません。モデルが受け取れるのは「テンソル」という、決まった形 (shape) を持つ数値の多次元配列だけです。

- テンソル (tensor): 数値が格子状に並んだ箱です。たとえば `(8, 3, 256, 256)` という形のテンソルは「8 枚の画像 × 3 色チャンネル (RGB) × 縦 256 ピクセル × 横 256 ピクセル」を意味します。先頭の軸 (この例なら 8) を batch 次元や枚数の次元と呼びます。
- PyTorch: Python でニューラルネットワークを書くためのライブラリ (道具箱) です。
- Dataset (データセットクラス): PyTorch が定めた約束ごとで、「`__len__` (全部で何個のサンプルがあるか) と `__getitem__(i)` (i 番目のサンプルを取り出す) の 2 つのメソッドを持つクラス」を指します。この 2 つさえ実装すれば、PyTorch の `DataLoader` が自動で複数サンプルをまとめて (バッチ化して) 学習ループに流し込んでくれます。`DataLoader` は `num_workers` を指定すると複数のプロセスで並列に `__getitem__` を呼んでくれるので、データ供給がボトルネックになりにくくなります。
- parquet: 表形式のデータを列ごとに圧縮して保存するファイル形式です。「必要な列だけ読む」ことが速くできるのが特徴で、センサーログのように行数が膨大なデータに向きます。

自動運転データセットの多くは、1 回の走行を「クリップ (clip)」や「シーン (scene)」という単位に区切って配布します。1 クリップはだいたい 20 秒くらいの走行記録で、`fd1d1b6b-59bf-4292-8295-5028aa6aa5e3` のような UUID (世界で重複しない長い ID 文字列) で識別されることが多いです。1 クリップの中には典型的に次のものが入っています。

- 複数台のカメラそれぞれの動画 `.mp4` (前方・後方・左右など。データセットによって 6〜7 台が一般的)
- 各カメラのフレーム撮影時刻を記録したタイムスタンプの表
- 車の動き (速度・加速度・姿勢など) を記録した自車運動 (egomotion) のログ

このバラバラな素材を、モデルの `forward()` (順伝播。入力を受け取って出力を計算する処理) が要求する形のテンソルに揃えるのがパーサ (Dataset クラス) の役目です。

## 仕組みを詳しく

典型的な実装では、`__getitem__(idx)` が返す 1 サンプルを次のような辞書にまとめます。形はあくまで一例ですが、考え方を掴むのに役立ちます。

```
visual_tiles      (N, 3, 256, 256)  N 台のカメラ画像 (+ 地図タイルなどの追加面)
visual_history    (D_v,)            過去の圧縮済みシーン記憶 (任意)
egomotion_history (D_e,)            過去 K ステップ × M 信号 = 自車運動の履歴
trajectory_target (D_t,)            未来 K ステップ × 2 信号 = 学習の正解 (軌道)
clip_uuid         str               どのクリップか
sample_idx        int               クリップ内のどの時刻か
```

入力 (visual_tiles, egomotion_history) と出力 (trajectory_target) のペアを返す、という構造がポイントです。

### 設計の核心: 「サンプル = (クリップ, クリップ内の瞬間) のペア」

最も重要な設計判断は「1 サンプルを何と数えるか」です。広く使われるやり方は「1 サンプル = (どのクリップか, クリップ内のどの瞬間か) のペア」にすることです。

`sample_idx` は、一定レート (たとえば 10Hz = 1 秒に 10 回) に間引いた後の時系列における行番号です。たとえば `sample_idx = 70` は「このクリップの 7.0 秒地点を今この瞬間とみなす」という意味です。その瞬間を中心に、過去 K ステップ (例: 64 ステップ = 6.4 秒) を履歴として、未来 K ステップ (6.4 秒) を正解として切り出します。

ここで効いてくるのが「1 クリップから複数のサンプルが取れる」点です。20 秒のクリップは 10Hz で約 200 行になります。先頭の 64 行は「過去 64 ステップ」を確保するために使えず、末尾の 64 行は「未来 64 ステップ」のために使えません。つまり有効な `sample_idx` はおよそ 64 番目から 135 番目までの約 72 個。クリップ 1 本から約 72 サンプルが生まれます。これは「時間方向のスライディングウィンドウ」によるデータ増幅で、限られた走行データを大きく水増しできる嬉しい性質です。

ただし水増しには裏があります。同じクリップから切り出した隣り合うサンプルは、画像も運動もほとんど同じです。後述するように、これは訓練・検証の分割で配慮しないと評価を甘くします。

### 構築時に全サンプルを列挙し、`__getitem__` では I/O だけ

性能上の定石は「`__init__` (生成時) で全クリップを走査し、有効な `(clip_uuid, sample_idx, timestamp)` を平坦なリストに並べておく」ことです。

```python
self._samples = []                          # 平坦なサンプル一覧
for clip_uuid in clips:
    samples = self._valid_samples_for_clip(clip_uuid)
    if samples:
        self._samples.extend(samples)
        self._load_camera_timestamps(clip_uuid)   # 時刻表もキャッシュ
```

こうしておく理由は、`__getitem__` が呼ばれるたびに表計算 (どの行が有効かの判定) をやり直すのを避けるためです。`__getitem__` の中では「動画から 1 フレーム読む」「ログから窓を切り出す」という入出力 (I/O) だけを行います。学習中は `__getitem__` が何百万回も呼ばれるので、ここを軽くしておくのは速度面で大きく効きます。これは PyTorch の Dataset 抽象を前提にした典型的なパフォーマンスチューニングです。

### 妥当性チェック (壊れたクリップを弾く)

大規模データセットは必ず壊れたクリップを含みます。サンプルを登録する前に検査を入れるのが堅牢な設計です。代表的な検査は次の通りです。

1. 必要なファイルがそろっているか。全カメラの動画とタイムスタンプ表が存在するか。1 つでも欠けたらそのクリップを丸ごとスキップ。
2. 自車運動ログに必要な列がそろっているか (列指定読み込みが `KeyError` を出さないか)。
3. 間引いた後の行数が下限 (例: `MIN_ROWS = 129` = 64 + 64 + 1) 以上あるか。短すぎるクリップは 1 サンプルも取れないので除外。
4. 間引いた後のタイムスタンプが厳密に単調増加しているか (`np.all(np.diff(timestamps) > 0)`)。時刻が逆行したり重複したりするデータは時系列として壊れているので除外。

「読めないものは静かにスキップしてログ警告を出す」設計にすると、データに数件壊れたクリップが混ざっても学習全体が止まりません。

### 画像前処理はバックボーンの設定から自動導出する

画像をモデルに入れる前のリサイズ・正規化 (前処理) は、使う画像認識モデル (backbone) に合わせて決める必要があります。モデルの設定から前処理を自動導出するのが安全です。

```python
backbone = timm.create_model(backbone_name, pretrained=False)
data_config = timm.data.resolve_model_data_config(backbone)
transform = timm.data.create_transform(**data_config, is_training=False)
```

この狙いは [モデル設定から前処理を自動導出する (timm の resolve_model_data_config)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0840_timm-resolve-model-data-config) を参照してください。要点は「前処理をモデル本体に合わせて自動で決めることで、リサイズ寸法や正規化の値がモデルと食い違う事故を防ぐ」ことです。

### プレースホルダで先に shape を通す

すべての入力を最初から本物で埋める必要はありません。たとえば描画済み地図タイルや「過去フレームの圧縮シーン記憶」がまだ用意できないときは、正しい shape のゼロ埋めテンソルを入れておきます。ゼロ埋めでも形が正しければ `forward()` は問題なく動き、パイプライン全体の疎通確認 (forward pass test) ができます。あとから本物のエンコーダがその枠を埋めれば、下流のコードを変えずに段階的に機能を足せます。これはインターフェースを先に固定してから中身を実装する漸進的開発の常套手段です。

### 複数データセットを同じ形に揃える

データセットごとにディレクトリ構造もファイル形式も違いますが、すべてのパーサが同じ辞書 (同じキー、同じ shape) を返すように設計しておくと、下流のモデルはデータセットの違いを意識せずに済みます。これがマルチデータセット対応の鍵で、「アダプタパターン」(異なる入力源を共通インターフェースに変換する設計) の素直な適用です。

## 手法の系譜と主要論文

このような Dataset 化は単独の論文テーマというより、複数の研究系譜が積み重なって標準化された実務知識です。

- KITTI (Geiger et al., CVPR 2012, arXiv なし当時): 自動運転向けに「複数センサーを時刻同期して配布する」ベンチマークの先駆けです。カメラ・LiDAR・GPS/IMU を揃え、評価指標まで定義したことで、研究の比較土俵を作りました。
- nuScenes (Caesar et al., CVPR 2020, arXiv:1903.11027): 自動運転データを「シーン = 約 20 秒のクリップ」という単位で配布する慣習を広めた大規模データセットです。6 カメラ + LiDAR + レーダー + 自車運動を時刻同期し、1000 シーンを提供しました。「1 クリップ約 20 秒、クリップ内の時刻をスライドして複数サンプルを作る」設計はこの流れを汲んでいます。クリップ単位の利点は「連続した時系列を保ったまま訓練・検証を分けやすい」点で、トレードオフは「同一クリップ内のサンプルが互いに似ているため、ナイーブにシャッフルすると訓練と検証で情報が漏れやすい」点です。
- PyTorch (Paszke et al., NeurIPS 2019, arXiv:1912.01703): `Dataset`/`DataLoader` という「`__getitem__` だけ書けば並列読み込みとバッチ化を任せられる」抽象を提供しました。「`__getitem__` を I/O だけに絞り、重い計算は構築時に前倒しする」最適化は、この抽象を前提にした典型的なチューニングです。
- end-to-end 自動運転の系譜 (UniAD: Hu et al., CVPR 2023 best paper, arXiv:2212.10156): 「複数カメラ + 自車運動の履歴」から「未来の軌道」を直接予測する枠組みを定式化しました。検出・追跡・地図・予測・計画を 1 つのネットワークに統合し、それぞれを別々に最適化するより良い計画性能を出せることを示しました。入力 (画像群 + 運動履歴) と出力 (軌道) のペアという Dataset の構造は、この end-to-end 予測の素直な反映です。
- 大規模 WebDataset 化の流れ (近年): クリップを `.tar` シャードにパックしてストリーミング読み込みする手法 (WebDataset 形式) が、クラウド上の数 TB 級データで使われるようになりました。ランダムアクセスを諦める代わりにシーケンシャル読み込みの帯域を最大化する、というトレードオフを取ります。

## 論文の実験結果 (定量データ)

Dataset 化そのものは「指標で測る対象」ではありませんが、設計判断の良し悪しは下流タスクの数値に表れます。指標の意味から説明します。

- nuScenes は検出指標として mAP (mean Average Precision, 検出の正確さの平均。高いほど良い、0〜1) と独自の NDS (nuScenes Detection Score, mAP に距離・速度・向き誤差を加味した総合指標) を定義しました。データを時刻同期して配布したことで、後続研究が同じ土俵で比較できるようになり、3D 検出の mAP は登場時の 30% 前後から数年で 60% 超へと急速に伸びました。これは「揃ったデータセット」がベンチマーク競争を駆動した好例です。
- 軌道予測・計画の指標は ADE (Average Displacement Error, 予測軌道と正解軌道の各時刻の距離の平均、メートル) と FDE (Final Displacement Error, 最終時刻だけの距離) が標準です。値が小さいほど良い予測です。UniAD は nuScenes 上で、計画の L2 誤差 (予測経路と実走経路のずれ) を従来手法より下げ、衝突率も削減したと報告しています (たとえば平均 L2 をベースラインの数分の一に、衝突率も大幅減と報告)。これは「複数カメラ + 運動履歴 → 軌道」という Dataset の構造に沿ってモデルを組むことの有効性を示します。
- データ増幅の効果については、時間方向のスライディングウィンドウで実効サンプル数を約 70 倍に増やせる (20 秒クリップから約 72 サンプル) ことが直接の利点です。一般に、相関の強い水増しサンプルは「独立な新規データ 1 件」ほどの価値はないため、見かけのサンプル数ほど性能は伸びませんが、それでも過学習を抑える正則化効果が観測されます。

アブレーション (どの要素を抜くとどうなるか) の観点では、自車運動履歴を入力から抜くと軌道予測の ADE が悪化することが各種 end-to-end 研究で繰り返し報告されています。逆に運動履歴を入れすぎると後述の「過去なぞり」近道学習に陥ります ([自車運動の入力信号設計 [speed, accel, yaw_angle, curvature]](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0809_egomotion-curvature-signals) 参照)。

## メリット・トレードオフ・限界

メリット
- 1 サンプル = (クリップ, 瞬間) という単純な表現で、時間方向のデータ増幅 (1 クリップから約 72 サンプル) を自然に得られる。
- 構築時に有効サンプルを全列挙し `__getitem__` を I/O だけにすることで、学習時の CPU 負荷を下げられる。
- 壊れたクリップを静かにスキップするので、データ品質のばらつきに強い。
- 前処理を backbone 設定から自動導出し、複数データセットを同じ辞書形に揃えることで、モデルと前処理の不整合事故とデータセット差異の両方を吸収できる。

トレードオフ・限界
- 素朴な実装では `__getitem__` のたびに `.mp4` 全体をメモリに読み込んでから 1 フレームをデコードしがちで、学習時の主要なボトルネックになる。事前にフレームを抽出してキャッシュする別工程が望ましい ([動画 (.mp4) からの効率的なフレームデコードとセンサー時刻同期](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0832_physical-ai-av-sdk-video-decode))。
- プレースホルダ (ゼロ埋め) の入力からはモデルが情報を得られない。疎通確認はできても、その入力が効くかは本物を入れてから検証する必要がある。
- 同一クリップ内サンプルが強く相関するため、訓練・検証分割はクリップ単位で行わないと評価が甘くなる。サンプル単位でランダム分割すると、検証セットに「訓練セットとほぼ同じ瞬間」が混ざり、汎化性能を過大評価してしまう。これはこの設計の最大の落とし穴です。
- 未解決の課題として、相関の強いサンプルをどう重み付け・間引きすれば「実効的に独立なデータ量」を最大化できるか、最適なスライド幅やストライドの選び方には定説がありません。

## 発展トピック・研究の最前線
- ストリーミングデータローディング (WebDataset, FFCV など): 数 TB〜PB 級のクラウド保管データを、ランダムアクセスを諦めつつ高帯域でストリームする手法。学習の I/O ボトルネックを根本的に解消する方向です。
- 事前抽出・キャッシュパイプライン: 動画から学習に使うフレームだけを先に抜き出し、テンソルや圧縮画像として保存しておくことで `__getitem__` を軽くする。学習スループットが数倍変わることもあります。
- 自動データキュレーション: 大量クリップから「学習に効く難しいシーン」を選別する研究 (active learning, ハードサンプルマイニング)。相関の強い冗長サンプルを賢く間引く方向と表裏一体です。
