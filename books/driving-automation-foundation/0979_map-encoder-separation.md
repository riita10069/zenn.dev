# Map Encoder分離設計

## ひとことで言うと
Map Encoder 分離設計とは、自動運転モデルにおいて「カメラ画像を処理する部品」と「地図(ナビ地図や HD 地図)を処理する部品」を、最初からきっちり別々のニューラルネットワークとして作る設計のことです。ここで「エンコーダ(encoder)」とは入力から特徴(後段の判断に使う数値表現)を抜き出す部品を指します。両者を最初から1つの網にごちゃ混ぜに入れるのではなく、それぞれが得意な形で特徴を取り出し、最後に俯瞰図(鳥瞰図、BEV)という共通の座標空間で合流させます。別レーンで処理して後で合流させる、というのが要点です。

## 直感的な理解
自動運転モデルには、性質がまったく違う2種類の入力が同時にやってきます。

- カメラ画像: 実世界の生の写真。光の当たり方、影、天候、他車、歩行者など、複雑で多様で予測不能。何が写っているかをゼロから理解しなければなりません。
- 地図(ナビ地図 / HD 地図): 人間や地図会社が事前に作った「ほぼ正解」の情報。車線がどこにあるか、進むべき経路はどれか、停止線や信号がどこか、が記号的・整理された形で描かれています。

この2つは、たとえ見た目はどちらも画像でも、中身の性質が根本的に違います。カメラは「ノイズだらけの生の観測」、地図は「整理済みの事前知識」です。これを1つのエンコーダに同じように突っ込むと不都合が起きます。カメラ用に作った重い backbone を地図にも使えば、単純な地図に対して計算が無駄に重い。逆に地図用の軽い処理をカメラに使えば、複雑な画像を理解しきれない。さらに「どちらの情報がどれだけ効いたか」が混ざって追えなくなり、デバッグや改良が難しくなります。

そこで「それぞれの性質に合った専用エンコーダを別々に持ち、共通の空間(BEV)に揃えてから合流させる」分離設計が生まれます。BEV(Bird's-Eye-View、鳥の目で見た俯瞰図)とは、カメラのような斜め視点ではなく、真上から地面を見下ろした2次元グリッドのことです。カメラ画像も地図も、最終的にこの同じ俯瞰グリッドに変換すれば、同一座標系の上で素直に足し合わせたり、参照し合ったりできます。地図はもともと俯瞰の座標を持つので、BEV を共通土台にするのは地図融合と特に相性が良い、という事情もあります。

## 基礎: 前提となる概念
- 特徴抽出 / backbone: 画像から「角・テクスチャ・物体らしさ」といった抽象的な手がかりを段階的に抜き出すネットワーク本体。CNN(畳み込みネット)や Vision Transformer がよく使われます。ImageNet などで事前学習(pretraining、大量の自然画像であらかじめ賢くしておくこと)した重みを使うのが一般的です。
- ラスタ表現とベクトル表現: 地図の表し方の2系統。ラスタ(raster)はピクセルの絵として地図を描く方式(画像として扱える)。ベクトル(vector)は車線や道路境界を折れ線・点の座標列として表す方式(構造を直接持つ)。
- HD 地図(High-Definition map): 車線中心線・境界・標識・信号などを高精度(数cm〜十数cm)に記録した自動運転専用の詳細地図。普通のナビ地図より格段に細かい。
- モダリティ(modality): 入力の種類。カメラ画像、LiDAR 点群、地図、レーダーなどがそれぞれ別のモダリティです。
- マルチモーダル融合(multimodal fusion): 性質の異なる複数の入力(モダリティ)を統合すること。ここではカメラと地図という2モダリティを統合します。融合の位置で early fusion(入力段で混ぜる)/ mid fusion / late fusion(出力段で混ぜる)に分類され、分離設計は「各モダリティを専用エンコーダで処理してから BEV で混ぜる」mid〜late 寄りの設計です。
- チャネルと空間サイズ: 特徴マップのテンソル形 `(B, C, H, W)` の各次元。B はバッチ、C はチャネル数(特徴の種類数)、H×W は空間の縦横。2つの特徴を足し合わせるには、この C と H と W がそろっている必要があります。
- 1×1 畳み込み(pointwise convolution): カーネルサイズ1の畳み込み。空間方向には混ぜず、各位置でチャネルだけを線形変換する。チャネル数を別の値に「射影(projection)」するのに使います。

## 仕組みを詳しく
分離設計のモデルでは、forward(順伝播、入力から出力までの計算)が明確に2レーンに分かれます。擬似コードで示すと:

```
# --- Camera branch(カメラレーン) ---
x = camera_tiles.reshape(B * V, C, H, W)   # V 台のカメラ画像をまとめる
features = backbone(x)                      # カメラ用 backbone で特徴抽出
image_bev = feature_fusion(...)             # 俯瞰図(image BEV)に変換

# --- Map branch(地図レーン) ---
map_bev = map_encoder(map_input)            # 地図専用エンコーダで地図 BEV を作る

# --- Fuse(合流) ---
fused_features = map_bev_fusion(image_bev, map_bev)
```

地図エンコーダの中身(ラスタ地図エンコーダの典型例)は次の処理をします。

1. 入力。BEV 空間のラスタ地図画像 `(B, 3, 256, 256)`。B はバッチ、3 は RGB チャネル。256×256 などに固定するのは、内部で使う Swin Transformer 系 backbone の入力サイズ制約に合わせるためです。

2. backbone で特徴抽出。たとえば SwinV2-Tiny を `features_only=True`(分類の最終層を付けず、途中の複数解像度の特徴マップだけを取り出すモード)で使います。ここで重要な設計判断が `pretrained=False`、つまりランダム初期化から学習する点です。カメラ backbone は通常 ImageNet 事前学習を使いますが、地図画像は自然写真と統計的性質がまったく違う(線と塗りつぶしの記号的な図)ため、自然画像の事前学習が必ずしも有利でないとの判断です。Swin の各ステージ(解像度の異なる複数の特徴マップ)をすべて受け取ります。

3. 解像度を揃える。`AdaptiveMaxPool2d((output_h, output_w))` などで、各ステージの特徴マップを同じ空間サイズ `output_h × output_w` にプーリング(領域ごとに代表値を取って縮小)します。Swin はステージごとに解像度が違う(例: 1/4, 1/8, 1/16, 1/32)ので、足し合わせる前に空間サイズをそろえる必要があります。

4. チャネル数を揃える。全ステージを連結したあと `Conv2d(..., embed_dim, kernel_size=1)`(1×1 畳み込みでチャネル射影)+ GELU(活性化関数)で、出力チャネルを `embed_dim`(例: 256)にします。

   出力は `(B, embed_dim, output_h, output_w)`。ここが肝心です。モデル本体の初期化で `output_h / output_w` を画像 BEV と同じ `bev_h × bev_w`(例: 450×300)に合わせて地図エンコーダを作ります。こうすることで image_bev と map_bev が「同じチャネル数・同じ空間サイズ」になり、後段の融合([Map-BEV融合戦略（residual等）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0978_map-bev-fusion-strategies))でそのまま重ね合わせられます。

つまり分離設計の本質は2点に集約されます。第一に「2つの専用エンコーダを持つこと」、第二に「出力の形(チャネル数と空間サイズ)を厳密にそろえること」です。形をそろえるからこそ、別々に処理しても最後にきれいに合流できます。なぜわざわざ別エンコーダにするのかと言えば、各レーンを独立に設計・調整・テストでき、片方を差し替えても他方に影響しない疎結合な構造になるからです。たとえば地図レーンだけを「ラスタ → ベクトル」に作り替えても、出力形さえ守れば融合段とカメラレーンは無改変で動きます。

## 手法の系譜と主要論文
地図情報を画像とは別系統で扱い、後段で融合する考え方は、自動運転の認識・予測・計画の各分野で長く発展してきました。

- ChauffeurNet(Bansal ら, 2019, RSS, Waymo)。地図・経路・周辺物体・信号などを複数のラスタ画像チャネルとして与え、模倣学習で運転方策を学習しました。地図を独立した入力レーンとして明示的に与える発想で、ラスタ地図エンコーダの直接の祖先です。動機は「経路や車線という事前知識を与えると運転が安定する」こと。効果は安定性向上。トレードオフは「地図への過度な依存(地図が間違うと運転も間違う)」リスクで、論文自身がこの脆弱性に対処するためデータ拡張と摂動注入を導入しました。

- VectorNet(Gao ら, 2020, CVPR, Waymo)。HD 地図を画像のラスタではなく折れ線(ベクトル=ポリライン)として別エンコーダで処理し、エージェント軌跡の特徴と後段でグラフ的に統合しました。動機は「地図はもともと記号的な構造を持つので、画像化して畳み込むより、構造をそのまま符号化する方が効率的で正確」というもの。効果は計算量削減と精度向上。トレードオフは、ベクトル化された地図データを用意する前処理が増える点です。ラスタ方式とベクトル方式は対照的ですが、「地図を別レーンで処理する」という分離の思想は共通です。

- BEV 系のセンサ融合。BEVFormer(Li ら, 2022, ECCV)と BEVFusion(Liu ら, 2023, ICRA, MIT)は、カメラや LiDAR といった異なるセンサを共通の BEV グリッドに射影してから融合します。「異なる入力を同じ俯瞰座標系にそろえてから合流させる」という設計は、画像も地図も同じ BEV グリッドにそろえる分離設計と同じ流れに属します。効果は「異なるモダリティを同じ座標系で素直に統合できる」こと。トレードオフは「BEV への射影自体に誤差やコストが乗る」点です。

- 地図なし運転への展開。MapTR(Liao ら, 2023, ICLR)はオンラインで車線地図をベクトルとして推定し、事前 HD 地図への依存を減らす方向を示しました。これは「地図レーン」の中身を、固定の事前地図から走行時推定地図へ置き換える発展で、分離設計の疎結合性がこうした差し替えを容易にします。

これらに共通する教訓は「性質の異なる入力は専用エンコーダで処理し、共通空間にそろえてから融合すると、効率・精度・解釈性のいずれもが向上する」ということです。

## 論文の実験結果(定量データ)
地図を別レーンで扱う設計の効果を、代表論文の定量結果で確認します(指標と数値はおおよそ)。

- VectorNet。Argoverse 軌跡予測ベンチマークと Waymo 社内データで評価。評価指標は ADE / FDE です。ADE(Average Displacement Error、平均変位誤差)は予測軌跡と正解軌跡の各時刻の距離の平均、FDE(Final Displacement Error)は終端時刻の距離。どちらも小さいほど良い(単位はメートル)。VectorNet はラスタ + CNN ベースラインに対して、約70%少ない計算量(モデルの FLOPs を大幅削減)で、同等以上の予測精度を達成したと報告されました。「地図を構造のまま符号化する別エンコーダ」が効率の点で効く、という定量証拠です。アブレーション(ある要素を抜く実験)では、ポリライン間のグローバル相互作用を担うグラフ部分を外すと予測誤差が悪化し、地図と軌跡を相互参照する仕組みの寄与が確認されました。

- ChauffeurNet。シミュレーションと実車で評価。純粋な模倣学習(専門家の運転を真似るだけ)では、わずかな誤差が積み重なって軌道が逸脱する「分布シフト(共変量シフト)」に弱いことが課題でした。論文は地図ラスタ入力に加えて、軌道に摂動を与える拡張や補助損失(車線逸脱・衝突を罰する項)を導入することで、衝突や逸脱の発生を大幅に減らせることを示しました。アブレーションでは、これらの補助項を外すと閉ループ走行(自分の出力が次の入力になる実運転に近い設定)での失敗が顕著に増え、地図情報単体ではなく「地図 + 安全制約の組み合わせ」が安定運転に効くことが分かりました。

- BEVFusion。nuScenes ベンチマーク(自動運転の標準データセット)で評価。指標は3次元物体検出の NDS(nuScenes Detection Score、検出精度と各種誤差を総合した指標、高いほど良い)と mAP(mean Average Precision)、BEV セグメンテーションの mIoU。BEVFusion はカメラと LiDAR を共通 BEV で融合し、当時の最先端を上回る NDS(検出 mAP で従来比 +1〜2 ポイント、セグメンテーション mIoU で +6 ポイント超の報告)を達成しつつ、効率の良い BEV プーリングにより従来手法より高速(BEV プーリング部分で約40倍の高速化、全体で約1.9倍速いと報告)でした。要点は「別々に BEV を作ってから共通空間で融合する」設計が、精度と速度を同時に押し上げたことです。アブレーションでは、カメラ枝と LiDAR 枝のどちらか一方だけにすると、特に悪天候や遠距離・小物体で精度が落ち、両モダリティを BEV で統合する価値が定量的に示されました。

これらが共通して示すのは、「専用エンコーダで別々に処理し、共通空間で統合する」設計が、効率(VectorNet)、安定性(ChauffeurNet)、精度と速度の両立(BEVFusion)という異なる軸でそれぞれ利得を生む、という点です。

## メリット・トレードオフ・限界
メリット:
- カメラと地図それぞれの性質に合った処理を独立に設計・調整できる。
- 出力形をそろえることで後段融合がシンプルになり、融合方式([Map-BEV融合戦略（residual等）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0978_map-bev-fusion-strategies))を差し替えやすい。
- 各レーンを単体でテスト・デバッグでき、問題の切り分けがしやすい。
- 地図レーンをオフにする・別実装に差し替える改造が、カメラレーンに影響せず行える。

トレードオフ・限界:
- 地図専用にもう1つ backbone を持つので、その分パラメータと計算が増える。地図は構造が単純なので、カメラ並みに重い backbone は過剰になりやすく、軽量化の余地がある。
- 出力サイズを厳密にそろえる制約があり、片方を変えるともう片方も合わせる必要がある。
- 入力サイズが固定(例: 256×256)など、backbone 由来の制約がある。
- 地図に依存するため、地図が古い・間違っている場面では誤った事前知識を与えるリスクがある(ChauffeurNet でも指摘された一般的弱点)。地図への過信を抑える仕組み(後述のゼロ初期化残差や、地図信頼度の推定)が研究課題です。
- 自己位置推定(localization)の誤差で地図と画像の座標がずれると、BEV で重ねたときに位置がずれ、融合がかえって害になりうる。

研究上の未解決課題として、(1) ラスタ vs ベクトルのどちらが最終的に優れるか(両者を併用するハイブリッドも提案されている)、(2) 地図が無い・劣化した地域での頑健性(map-less driving、地図なし運転の研究が活発)、(3) 地図とカメラの座標ずれ(位置推定誤差)に対する耐性、などがあります。

## 発展トピック・研究の最前線
- 地図なし運転(map-less / online mapping)。HD 地図の整備・更新コストを避けるため、カメラから走行時にオンラインで車線地図を推定する研究(MapTR, Liao ら 2023, ICLR など)が進んでいます。これは「事前地図への依存」を減らす方向です。
- ベクトル化されたオンライン地図表現。ラスタではなくベクトル地図を end-to-end に予測し、下流の予測・計画に渡す流れ(VAD, UniAD など)。記号的な構造を保ったまま下流に渡せるのが利点です。
- 地図信頼度の明示的モデリング。地図が間違っている可能性を不確実性として推定し、信頼できないときはカメラ観測を優先する重み付け融合。これは次のノートで扱う融合戦略と直結します。
- BEV 表現学習の統一。カメラ・LiDAR・地図・レーダーをすべて同一 BEV に集約する統一基盤(BEVFusion 系の発展)が、認識から予測・計画までを貫く共通インターフェースになりつつあります。
