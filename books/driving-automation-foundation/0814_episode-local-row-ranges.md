# エピソードのローカル行レンジ算出 (添字空間の整合)

## ひとことで言うと
大きな時系列データの中から「このエピソード (1回の走行) は何行目から何行目までか」を正しく求める処理です。データの一部だけを読み込むと行番号が振り直されてズレる落とし穴があり、それを回避するために「各行がどのエピソードに属するか」を記録した列を直接見て範囲を導出します。一見ささいに見えますが、これを誤ると学習がエラーも出さずに静かに壊れる、データパイプラインの基礎中の基礎です。

## 直感的な理解
本棚に何冊もの日記帳が並んでいるとします。1冊が1回の走行 (エピソード) で、各ページが1コマ (フレーム) です。全部の日記帳を通しでページ番号を振ると「1冊目は1〜100ページ、2冊目は101〜200ページ」となります。これが「グローバル (全体での) ページ番号」です。

ところがあなたが2冊目だけを机に持ってきたとします。机の上では、その日記帳は当然1ページ目から始まります。つまり手元では「1〜100ページ」に振り直されます。これが「ローカル (手元での) ページ番号」です。

ここで、もし「2冊目は101ページから始まる」という全体の目次 (メタ情報) を信じて、机の上の日記帳の101ページ目を開こうとしたらどうなるでしょう。手元には100ページしかないので、はみ出して別の場所を読んでしまうか、エラーになります。グローバル番号とローカル番号を取り違えると、まったく違うデータを読んでしまうのです。

機械学習のデータローダでこの取り違えが起きると、たちが悪いことに「エラーすら出ずに、ただ間違ったデータで学習が進む」ことがあります。これを silent failure (静かな失敗) と呼びます。本ノートは、この取り違えを構造的に防ぐ方法を解説します。

## 基礎: 前提となる概念

用語を順に噛み砕きます。

エピソード (episode) とは、1回のまとまった走行の記録です。「ある車が10分間走った1ドライブ分」が1エピソードです。大規模データセットには数千〜数万のエピソードが含まれます。

フレーム (frame) とは、エピソードを時間で区切った1コマです。たとえば 10Hz (1秒に10コマ) で記録されていれば、1秒の走行は10フレームになります。

行 (row) とは、表形式データの1行です。時系列データは「1フレーム = 表の1行」として保存されるのが一般的で、列には速度・操作量・画像へのポインタ・そして後述の `episode_index` などが並びます。

なぜエピソードの境界を正確に知る必要があるのでしょうか。模倣学習では、各フレームについて「過去の何フレームか」と「未来の何フレームか」をまとめて1つの学習サンプルにすることがよくあります。過去のフレームから「自車がどう動いてきたか (エゴモーション履歴)」を作り、未来のフレームから「これからどう動くべきか (軌跡の正解)」を作るためです。たとえば過去64・未来64フレームを使うなら、1サンプルを作るのに前後 128 フレーム + 現在の1フレーム = 129 フレームの連続区間が必要です。

ここで決定的に重要なのが「エピソードの境界をまたいではいけない」という制約です。あるエピソードの末尾フレームから「未来64フレーム」を取ろうとすると、次のエピソード (別の走行) のデータに食い込みます。別の車・別の道の動きが混ざれば、軌跡の正解 (教師信号) がでたらめになります。だから「各エピソードがどこからどこまでか」を正確に知り、その内側でだけサンプルを作る必要があるのです。エピソード長が 129 フレーム未満なら、前後の文脈が確保できないのでそのエピソードはまるごとスキップします。

サブセット (subset) とは、データセット全体のうち一部のエピソードだけを切り出したものです。テスト時や小規模実験では、全体を読むのは重いので「エピソード0と1だけ」のように一部を読み込みます。このときに前述のローカル/グローバルの取り違えが起こります。

添字空間 (index space) という言い方も覚えておくと便利です。「どの番号体系で番号を数えているか」という意味で、グローバル添字空間とローカル添字空間という2つの世界が存在し、両者を取り違えてはいけない、という問題設定として整理できます。

## 仕組みを詳しく

問題の構造を図で整理します。

```
データセット全体 (グローバル行番号):
  エピソード0 : 行 0   - 99
  エピソード1 : 行 100 - 199
  エピソード2 : 行 200 - 299
  ...

ここで「エピソード1だけ」読み込むと、手元の表は:
  ローカル行 0 - 99   (中身は元のエピソード1の100行)

もしメタ情報の「エピソード1は行100から」を信じると:
  手元の表の行100を読もうとする → 存在しない / 別データ → 破綻
```

多くのデータセットライブラリは、過去のバージョンでは「各エピソードの開始・終了行を持つ便利な属性 (たとえば `episode_data_index` のような名前のテーブル)」を提供していました。ところがバージョンアップでこれが別のメタ情報構造に置き換わり、しかもそこに入っている開始行が「グローバル行番号のまま」というケースがあります。サブセットを読んだときにこの値を信じると、前述の破綻が起こります。これは [ロボット学習データセットライブラリの API 移行](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0823_lerobot-0-5-x-migration) で述べた「メタ情報構造の置き換え」パターンの典型的な落とし穴です。

頑健な解決策は「メタ情報を信じず、実際に読み込んだ表に必ず存在する真実を使う」ことです。その真実とは、各行に付いている `episode_index` (この行はどのエピソードに属するか) という列です。この列は手元に読み込んだ表そのものに含まれるので、必ずローカル行番号と一致します。メタ情報という「目次」ではなく、データ本体という「実物」を数えるのがポイントです。

```python
import numpy as np

def episode_local_ranges(hf_dataset):
    ep_col = np.asarray(hf_dataset["episode_index"])  # 各行のエピソード番号 (ローカル)
    ranges = {}
    for ep_idx in np.unique(ep_col):           # 含まれるエピソード番号を列挙
        rows = np.nonzero(ep_col == ep_idx)[0] # その番号を持つ行位置 (ローカル)
        ranges[int(ep_idx)] = (int(rows[0]), int(rows[-1]) + 1)  # [start, end)
    return ranges
```

ステップに分解します。

1. 読み込んだ表の `episode_index` 列を取り出す。これはローカル行に対応するので、必ず手元の表と一致する。
2. `np.unique` で、表に含まれるエピソード番号の一覧を得る (重複を除いてソートして返す)。
3. 各エピソード番号について、`np.nonzero(ep_col == ep_idx)` でその番号を持つ行の位置 (ローカル行番号) を全部取り出す。`ep_col == ep_idx` は各行が真/偽の配列を作り、`np.nonzero` がその真の位置を返す。
4. その最初の行を start、最後の行 + 1 を end として `[start, end)` の半開区間を作る。

`[start, end)` は「start を含み end を含まない」という区間の書き方です。Python のスライス `list[start:end]` と同じで、`end - start` がそのまま件数になります。たとえば `(0, 100)` なら 100 件です。この記法は off-by-one (1個ずれ) のバグを減らす定石です。

数値例 (エピソード1だけ読み込んだ場合):

```
ep_col = [1, 1, 1, ..., 1]    (100行ぶん、すべて 1)
np.unique          → [1]
np.nonzero(ep_col==1)[0] → [0, 1, 2, ..., 99]
ranges = {1: (0, 100)}        ← ローカル行で正しい

もしメタ情報を信じていたら:
ranges = {1: (100, 200)}      ← グローバル行。手元は0-99しかなく破綻
```

複数エピソードを読んだ例も見ておきます。

```
エピソード0 (60行) と エピソード2 (40行) を読み込んだ場合:
ep_col = [0]*60 + [2]*40   (合計100行、手元ではローカル0-99)
np.unique          → [0, 2]
ranges = {0: (0, 60), 2: (60, 100)}   ← ローカル行で連続・正しい
```

注目すべきは、元のグローバルではエピソード2は遠い位置 (たとえば行200台) にあったのに、手元では行60から始まっている点です。`episode_index` 列から導けば、この振り直しを自動で正しく拾えます。

この `ranges` を使って、後段の処理が「過去64・未来64フレームが確保できる有効フレーム」だけを列挙します。具体的には各エピソード `(start, end)` に対し、有効な中心フレームは `[start + past, end - future)` の範囲だけです。エピソード長 `end - start` が `past + future + 1` 未満ならこの範囲が空になるので、そのエピソードはスキップします。

## 手法の系譜と主要論文

この処理は特定の論文が提案した「手法」ではなく、時系列を扱う学習すべてに共通する基礎技術です。背景となる考え方の系譜を描きます。

時系列からのサンプル切り出しという発想は、模倣学習の源流である Pomerleau の ALVINN (NeurIPS 1989) の時代から本質的に存在します。連続するセンサ観測から運転操作を予測する以上、サンプルは必ず時間的に連続した区間から作られ、別の走行の混入は学習を壊します。この「区間の境界を正確に守る」要請は、自動運転に限らず、音声・動画・センサログなどあらゆる時系列学習に共通します。

添字空間 (index space) を明示的に管理するという発想は、PyTorch の `torch.utils.data.Subset` に明快に現れています。`Subset` は「元データセットの添字のリスト `indices`」を保持し、`subset[i]` が呼ばれると `original[indices[i]]` に変換します。つまりローカル添字 `i` をグローバル添字 `indices[i]` へ写す変換を内部で明示的に持つわけです。これは「添字の世界 (ローカル) と元データの世界 (グローバル) を取り違えない」という設計原則の教科書的な実装です。本ノートの手法は、この原則を「メタ情報を信じずデータ自身から導く」形で徹底したものと言えます。

列指向データへの効率的なアクセスという側面では、Apache Arrow / Parquet の系譜が関係します。`episode_index` という1列だけを取り出して処理できるのは、データが列指向 (columnar) フォーマットで保持されているからです。列指向では「特定の1列だけを高速に読む」ことができ、他の重い列 (画像へのポインタなど) を読まずに済みます。この特性は [状態量の高速読み出し (動画デコード回避)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0847_vehicle-state-fast-path) の高速読み出しと同じ基盤に立っています。

## 論文の実験結果(定量データ)

この種の「添字整合」処理に固有のベンチマーク論文はありませんが、その重要性を定量的に裏づける文脈を示します。

データパイプラインのバグが学習結果に与える影響は、再現性研究で繰り返し指摘されています。最も体系的な指摘が Kapoor & Narayanan "Leakage and the Reproducibility Crisis in ML-based Science" (Patterns, 2023) です。この論文は、データリーク (data leakage、学習に使ってはいけない情報がこっそり混入すること) を含む論文が17の科学分野・合計329本にわたって蔓延し、報告精度が系統的に過大評価されていたことを示しました。リークを修正すると、論文が主張していた性能が崩れる事例が多数報告されています。エピソード境界の取り違えは、過去/未来フレームに別走行を混ぜるという意味で、これと同種の汚染を引き起こします。

アブレーション的に言えば、「境界チェックを省く」と何が起きるかは明確です。エピソードをまたいだサンプルが混入し、軌跡の正解 (教師信号) に別走行の動きが入る。学習損失は下がるように見えても、検証データでの軌跡予測誤差 (たとえば ADE: Average Displacement Error、予測軌跡と正解軌跡の各時刻での距離の平均。単位はメートル。小さいほど良い。あるいは FDE: Final Displacement Error、終端時刻だけの距離) が悪化します。さらに厄介なのは、この劣化が「コードのどこにも明らかなバグが見えない」状態で起こることです。エラーログには何も出ず、ただ性能だけが説明不能に落ちます。だからこそ、データを自身の真実 (`episode_index` 列) から導く堅牢な設計が、定量的な信頼性に直結します。

## メリット・トレードオフ・限界

メリット
- 一部エピソードだけ読み込んでも行番号がズレず、正しいフレームを読める
- データに必ず存在する `episode_index` 列を使うので、メタ情報の仕様変更に強い (メタ情報がグローバル/ローカルどちらでも結果は正しい)
- ローカル行で統一されるため、後段のサンプル列挙・状態読み出しの添字も一貫する

トレードオフ・限界
- 全行の `episode_index` 列を一度メモリに読み込んで `np.unique` する。エピソード数・行数が極端に多いと初期化がやや重い (ただし数値1列だけなので実害は小さい)
- メタ情報の便利属性を使わず自前で範囲を作るため、ライブラリが将来ローカル行ベースの正しい API を提供しても自動では乗り換わらない (手動の見直しが必要)
- `episode_index` 列が必ず存在し、かつ行順がエピソードごとに連続している、という前提に依存する。シャッフル済みのデータでは `np.nonzero` の結果が非連続になり、`(rows[0], rows[-1]+1)` 方式が「途中に別エピソードを含む範囲」を作って破綻する。整列前提を満たすか確認が要る (満たさないなら範囲ではなく行インデックスのリストそのものを保持すべき)

未解決の課題として、「添字空間の整合を型システムやテストでどう保証するか」があります。現状はレビュー時の注意と単体テスト頼みです。ローカル添字とグローバル添字を別の型 (newtype) として扱い、混同したら静的解析で弾く、といった設計はまだ一般的ではありません。実務では「読み込んだサンプルの `episode_index` が期待どおりか」を少数件アサートするテストを入れるのが最も費用対効果の高い防御策です。

## 発展トピック・研究の最前線

サンプリング設計はデータローダの中心的な研究対象です。境界を守ったうえで、さらに「どのフレームをどの確率でサンプリングするか」が学習効率と公平性を左右します。たとえばカーブや急ブレーキのような稀少だが重要な状況を重点的にサンプリングする importance sampling や、長いエピソードと短いエピソードのバランスを取る重み付けは、自動運転の模倣学習で実際に効くことが知られています。単純な一様サンプリングだと、直進ばかりのデータに引っ張られてカーブが下手なモデルになりがちです。

オフライン強化学習 (offline RL、過去に集めたデータだけから方策を学ぶ枠組み) の文脈では、エピソード境界の扱いがさらに重要になります。価値関数のブートストラップ (次状態の価値を使って現状態の価値を更新する) がエピソード境界をまたぐと、終端状態の扱いを誤り、学習が発散します。境界を示すフラグ (done/terminal) を正確に管理することは、D4RL (Fu et al., 2020, arXiv:2004.07219) のような標準ベンチマークでも明示的に設計されています。D4RL はオフライン RL の事実上の標準ベンチマークで、各遷移に終端フラグを正しく付けたデータを配布しています。

研究の最前線では、可変長のエピソードを効率よくバッチ化する手法 (パディングとマスキング、あるいは長さでソートしてミニバッチ内のパディングを減らす bucketing) や、ストリーミングデータセットで境界をオンザフライに検出する手法が議論されています。いずれも「境界を正確に知る」という本ノートの基礎の上に積み上がっています。
