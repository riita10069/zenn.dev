# backbone特徴サイズの抽象化

## ひとことで言うと
画像認識の土台ネットワーク(バックボーン)を別のものに差し替えると、出てくる特徴のチャンネル数が変わります(階層型 Transformer/現代 CNN では連結 1440、ResNet50 では連結 3840 など)。バックボーンを変えても下流のモジュールを書き換えずに済むようにする設計が「特徴サイズの抽象化」です。鍵は2つで、(1)連結後に 1×1 畳み込みで必ず固定の埋め込み次元(よく使われるのは 256)へそろえること、(2)チャンネル数を人が数えてハードコードせず、コードで自動検出することです。これは精度を上げる手法ではなく、研究の生産性(差し替え・公平比較)を担保するソフトウェア設計上の規律です。

## 直感的な理解
家電のコンセントを思い浮かべてください。掃除機・冷蔵庫・スマホ充電器は、内部の電圧も消費電力もバラバラですが、差込口の形(インタフェース)は統一されています。だから新しい家電を買っても壁の配線を作り直す必要がありません。バックボーンの差し替えも同じで、内部(チャンネル数や出力レイアウト)が違っても、下流から見える「差込口の形(次元)」を一定に保てば、下流を作り直さずに済みます。

なぜこれが研究で重要か。研究では「どのバックボーンが速くて精度が出るか」を比較するために頻繁に差し替えます。ところがバックボーンごとに出力チャンネル数が違います。

```
階層型 Transformer Tiny:  96 + 192 + 384 + 768   = 1440
現代型 CNN Tiny:          96 + 192 + 384 + 768   = 1440
ResNet50:                256 + 512 + 1024 + 2048 = 3840
ResNet18/34:              64 + 128 + 256 +  512  =  960
```

もし下流モジュール(複数視点の融合、運転方策、未来予測など)がこの数字をハードコードしていたら、ResNet50 を試すたびに下流を全部書き換える羽目になります。数十箇所を毎回直すのはバグの温床で、比較実験のコストが跳ね上がります。これを根絶するのが特徴サイズの抽象化です。研究の本質は「公平な比較」にあり、抽象化はその前提条件を提供します。

## 基礎: 前提となる概念
- バックボーン(backbone): 画像から特徴を抽出する土台ネットワーク。差し替え可能な部品として扱います。
- 下流モジュール(downstream): バックボーンの出力を受け取って次の処理をする部分。検出ヘッド、融合モジュール、方策ヘッドなど。
- 埋め込み次元(embed_dim): 特徴ベクトルの長さ。下流が前提とする「差込口の太さ」に相当します。なぜ次元を小さく固定すると都合がよいかは [埋め込み次元と注意機構の計算コスト](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0914_attention-cost-embed-dim) を参照。
- 1×1 畳み込み(Conv1×1): 空間サイズ(H, W)はそのままに、チャンネル数だけを変換する畳み込みです。各位置で「全入力チャンネルから全出力チャンネルへの線形変換」を行うので、チャンネルの次元合わせ(射影)に使われます。1440→256 のように圧縮できます。空間方向には情報を混ぜず、純粋にチャンネルの線形変換である点が特徴です。
- 抽象化(abstraction): 中身(バックボーンの種類や次元)が変わっても、外から見たインタフェース(下流が受け取る次元)を一定に保つ設計のことです。ソフトウェア工学でいう「実装の隠蔽」と同じ考え方です。
- channels-first / channels-last: テンソルの軸の並び順です。channels-first は [B, C, H, W](CNN・PyTorch の慣習)、channels-last は [B, H, W, C](一部の Transformer 実装の出力)です。両者を混在させると形が合わずバグになるため、どこかでそろえる必要があります。

## 仕組みを詳しく
抽象化は2本柱で実現します。

柱1: 連結直後に 1×1 畳み込みで固定次元へ射影する。

```python
self.channel_proj = nn.Sequential(
    nn.Conv2d(backbone_channels, embed_dim, kernel_size=1),  # 例: 1440 or 3840 -> 256
    nn.GELU(),
)
```

ここで `backbone_channels` はバックボーン依存(1440 や 3840)ですが、出力 `embed_dim` は常に固定(例 256)です。

```
階層型:   [B*V, 1440, 8, 8]  --Conv1x1-->  [B*V, 256, 8, 8]
ResNet50: [B*V, 3840, 8, 8]  --Conv1x1-->  [B*V, 256, 8, 8]
```

下流から見ると、どんなバックボーンでも入力は必ず 256 次元。だから融合以降はバックボーンを一切意識せずに書けます。変わるのは 1×1 畳み込みの入力側チャンネル数だけで、それも構築時に自動で決まります。GELU などの活性化を挟むのは、射影に非線形性を持たせて単なる線形圧縮より表現力を確保するためです。

柱2: チャンネル数を手書きせず自動検出する。新しいバックボーンを足すたびに 1440 や 3840 を人が数えてハードコードすると間違えます。そこで2段構えで自動取得します。

1. バックボーンライブラリ(timm など)が `feature_info` のような「各段のチャンネル数を返すメタ情報」を提供していれば、そこから読む(forward 不要で速い)。
   ```python
   channels = [stage["num_chs"] for stage in info]
   ```
2. そうしたメタ情報を持たないカスタムバックボーンは、ダミー入力 `torch.zeros(1, 3, 256, 256)` を1回流し、返ってきた各特徴マップの形からチャンネル数を読み取る。
   ```python
   return [f.shape[1] for f in out]   # 各特徴マップの C を拾う
   ```

合計 `backbone_channels = sum(feature_channels)` が求まり、そのまま 1×1 畳み込みの入力次元に渡ります。さらに forward では、出力レイアウト(channels-first か channels-last か)がバックボーンごとに違う問題にも対応し、構築時に検出したチャンネル数と各特徴マップの形を突き合わせて、必要なら `permute(0, 3, 1, 2)` で channels-first にそろえます。この突き合わせがあるおかげで、「最後の軸がチャンネルなのか高さ・幅なのか」を取り違える典型バグを構築時に検出できます。

この結果、新しいバックボーンをレジストリ(名前→クラスの対応表)に登録するだけで、チャンネル数の検出と固定次元への射影が自動で働き、下流は無改変で動きます。「レジストリに足せば設定の文字列だけで切り替えられる」という運用上の利便はこの抽象化が下支えしています。設定ファイルに `backbone: resnet50` と書くだけでアーキテクチャ全体が組み変わる、というのが理想形です。

## 手法の系譜と主要論文
これは特定の1論文の手法というより、複数の系譜が合流した共通設計です。

- He et al., "Deep Residual Learning for Image Recognition (ResNet)", CVPR 2016(arXiv:1512.03385)。ResNet50 の4段が 256/512/1024/2048 チャンネル(連結 3840)であることの出典です。残差接続(skip connection、入力を出力に足し戻す経路)で深いネットを学習可能にし、長く検出・分割の標準バックボーンであり続けました。Swin と段数(4段)は同じでもチャンネル数が倍以上違うため、抽象化が無いと差し替えが破綻する好例です。

- Liu et al., "Swin Transformer", ICCV 2021(arXiv:2103.14030)。96/192/384/768 の4段階層を持つ Transformer です。CNN と同じ「4段の階層特徴」というインタフェースを採用したことで、検出・分割の下流をそのまま流用できるようにしました。Swin の最大の貢献の1つは、性能だけでなく「既存の CNN 用下流をそのまま使える階層インタフェースを Transformer に持ち込んだ」ことだと言えます。

- Liu et al., "A ConvNet for the 2020s (ConvNeXt)", CVPR 2022(arXiv:2201.03545)。Swin と同じ 96/192/384/768 の段構成を持つ現代型 CNN です。これにより「同じ連結インタフェース」で CNN と Transformer をシームレスに差し替えられる土壌が固まりました。

- Carion et al., "DETR", ECCV 2020 / Li et al., "BEVFormer", ECCV 2022 / Hu et al., "UniAD", CVPR 2023。いずれもバックボーンの出力を共通次元(256)のトークンに射影してから Transformer で処理します。検出ヘッドや BEV エンコーダをバックボーンから切り離し、自由に差し替えられるようにするためで、効果はモジュール性と再利用性です。固定次元への射影という規律は、これらの系譜から来ています。

- Wightman, "PyTorch Image Models (timm)", 2019-。`features_only=True` と `feature_info` という統一インタフェースを提供し、どのバックボーンでも「各段のチャンネル数」を同じ呼び出しで取れるようにしました。自動検出はこの規約に乗るのが定石で、未対応のカスタムバックボーンにはダミー forward でフォールバックするのが堅牢な実装パターンです。timm は数百のバックボーンを同一 API で提供しており、研究のアブレーション基盤として事実上の標準になっています。

- Oquab et al., "DINOv2", 2023(arXiv:2304.07193)。大規模自己教師あり学習で得た汎用画像エンコーダで、固定次元インタフェースがあると下流へそのまま接続できる例です。基盤モデルをバックボーン化する流れの代表で、抽象化の価値が「新しい強力なバックボーンが出るたびに下流を変えずに乗り換えられる」点に拡張されています。

## 論文の実験結果(定量データ)
この抽象化自体は精度を上げる手法ではなく「差し替え可能性」を担保する設計なので、定量評価はバックボーン比較の文脈で現れます。

- ConvNeXt(CVPR 2022)は ImageNet-1k で、同規模の Swin に対して同等以上の Top-1 精度(ConvNeXt-T で約 82.1%、Swin-T で約 81.3% 前後と報告)を、近い計算量(GFLOPs、ともに約 4.5 GFLOPs 級)で達成しました。重要なのは、両者が同じ4段階層インタフェースを持つため、下流を一切変えずに差し替えて比較できた点です。抽象化が無ければこの公平比較自体が困難になります。Top-1 精度とは「最も確信度の高い1クラスが正解と一致する割合」で、ImageNet では1ポイントの差が有意とされる厳しい指標です。

- DETR(ECCV 2020)のアブレーションでは、バックボーンを ResNet50 から ResNet101 へ替えると COCO mAP が数ポイント上がることが示されています(おおよそ 42→44 mAP 級)。256 への射影で下流を固定しているため、バックボーンだけを差し替えた純粋な比較ができ、向上分がバックボーンに帰属できます。これが「下流を固定する」設計の実験的な価値です。もし下流が ResNet50 の 2048ch を直接前提にしていたら、ResNet101 へ替えるたびに下流の入力次元も変わり、向上がバックボーンによるのか下流の再設計によるのか切り分けられません。

- 一般に、ResNet50(連結 3840)を 256 へ射影する圧縮率(約 1/15)は、階層型 Tiny(1440→256、約 1/5.6)より大きく、情報圧縮の度合いが違います。報告されるバックボーン比較では、チャンネルの多いバックボーンが必ず勝つわけではなく、射影後の 256 次元がボトルネックになって差が縮むことがあります。これは「バックボーンの生のチャンネル数の差は、固定次元への射影でかなり吸収される」という観察につながります。チャンネル数の多さは表現力の上限を上げますが、256 ボトルネックを通すと上限の差が実効差に直結しなくなる、という現象です。

定量データの読み方として、抽象化の貢献は「比較が公平にできること」そのものです。下流を固定したからこそ、mAP や精度の差をバックボーンの違いに正しく帰属できます。逆に、抽象化を入れていない実装でのバックボーン比較は、下流の再設計が交絡(confound)している可能性を疑う必要があります。

## メリット・トレードオフ・限界
メリット
- バックボーンを差し替えても下流を書き換えなくてよい。比較実験が安価かつ公平になる。
- チャンネル数を人が数えてハードコードしないので、登録ミス由来のバグが減る。
- channels-last / channels-first のレイアウト差も構築時に自動吸収できる。
- 新しい基盤モデル(DINOv2 等)が登場しても、レジストリ登録だけで既存パイプラインに接続できる。

トレードオフ・限界
- どのバックボーンでも同じ固定次元(256)へ圧縮するため、チャンネルの多いバックボーン(ResNet50 の 3840)では情報損失が相対的に大きい。バックボーン本来の表現力を活かしきれない場合があり、上記の「差が縮む」現象として現れる。
- ダミー forward によるフォールバック検出は、構築時に1回とはいえ余分な計算が発生する。動的形状のバックボーンでは想定外の段数を返すことがあり、検出ロジックが前提を置きすぎると壊れる。
- 「4段の特徴を返す」ことを暗黙に前提にしがちで、5段以上を露出するバックボーンや、段ごとに空間サイズの落ち方が違うバックボーンではプール・連結側の整合に追加配慮が要る。テスト用のモックバックボーンが本物と段数・空間サイズが違うと、テストが現実を反映しない罠もある(モックは本物の総ストライドと段数を再現すべき)。
- 単純な 1×1 畳み込み射影は段ごとの空間構造を無視してチャンネルだけを混ぜる。FPN のような段間の空間的伝播は起きないため、射影の質は単純さの代償を伴う。

未解決の課題として、「固定 256 が常に最適か」「バックボーンごとに最適な射影次元を変えるべきか」は実験依存で、抽象化の便利さと表現力のトレードオフは残ります。バックボーンに応じて射影次元を可変にすると比較の公平性が崩れる、というジレンマもあります。

## 発展トピック・研究の最前線
- 統一バックボーンインタフェース: timm のようなライブラリが事実上の標準 API を提供し、数百のバックボーンを同じ呼び出しで使えるようにしています。研究の再現性とアブレーションの容易さに直結します。
- アダプタ/射影の工夫: 単純な 1×1 畳み込みではなく、段ごとに別々の射影をかけてから統合する、ボトルネック構造を入れる、といった圧縮の質を上げる工夫が検討されています。FPN の横方向接続も、段ごとに別射影をかける一種の高度な抽象化と見なせます。
- 基盤モデルのバックボーン化: DINOv2 や SAM の画像エンコーダなど、大規模自己教師あり学習で得たバックボーンを差し替えて使う流れがあり、固定次元インタフェースがあると既存の下流へそのまま接続しやすくなります。これらは ViT 系で単一スケールしか出さないことも多く、階層特徴を前提にした抽象化との整合(単一スケールから疑似多スケールを作る、など)が新たな設計課題になっています。
- パラメータ効率的微調整(LoRA/Adapter): 大きなバックボーンを凍結し、小さなアダプタだけ学習する流れも、固定インタフェースの上で成り立ちます。
