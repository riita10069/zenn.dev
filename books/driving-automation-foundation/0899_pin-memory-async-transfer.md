# Pinned Memory + 非同期転送

## ひとことで言うと
CPU 側のメモリを「OS が絶対に動かさない・追い出さない」特別な固定領域(pinned memory)に置くことで、そこから GPU へのデータコピーを速くし、しかも「コピーしながら同時に計算する」ことを可能にするテクニックです。データを GPU に渡す段のボトルネックを和らげ、高価な GPU を遊ばせないようにします。

## 直感的な理解
荷物を倉庫からトラック(GPU)に積む場面を考えます。普通の倉庫では荷物の置き場所がしょっちゅう動かされるので、積み込み係(DMA エンジン)は「今この荷物はどこ?」を毎回確認し、いったん決まった搬出口まで運び直してから積みます。この「運び直し」が余計な手間です。pinned memory は「この荷物は絶対この場所から動かさない」と決めた専用搬出口に荷物を置くようなもので、積み込み係が直接そこから積めるので速くなります。さらに、積み込み係に「積んでおいて、終わったら教えて」と頼んで、その間に運転手(GPU)は別の作業(前のバッチの計算)を進められます。これが非同期転送です。

## 基礎: 前提となる概念
学習では、CPU 側で準備したデータ(テンソル)を GPU のメモリ(VRAM、GPU 上の高速メモリ)へ転送してから計算します。この CPU→GPU 転送は地味に時間を食います。なぜ遅いのか、メモリの仕組みから説明します。

OS はメモリを「ページ」という単位(よく 4KB)で管理し、物理メモリが足りなくなると一部ページをディスクへ退避(スワップ)したり、物理上の置き場所を動かしたりします。これを仮想記憶/ページング(paging)と呼びます。プログラムが見る「アドレス(仮想アドレス)」と物理メモリ上の実際の位置はずれていて、OS が対応表(ページテーブル)で対応付けます。この「いつ動くか分からない」性質が GPU 転送には都合が悪いのです。

GPU への高速転送は DMA(Direct Memory Access、CPU を介さずデバイスが直接メモリを読み書きする仕組み)で行います。DMA は物理アドレスを直接たどるため、物理的な置き場所が固定されたメモリしか安全に読めません。普通の(ページング対象の)メモリ、いわゆるページャブルメモリ(pageable memory)から転送する場合、CUDA は内部でいったん固定領域(staging buffer)へコピーしてから DMA する、という二度手間をします。この余計なコピー(CPU が1バイトずつ運ぶ)が遅さの主因です。

加えて、ページャブルメモリからの転送は同期的になりがちです。「転送が終わるまで CPU は次へ進めない」ので、転送中は GPU 計算と重ねられません。これらを解決するのが pinned memory(ピンドメモリ、page-locked memory とも)です。

物理的なデータの通り道も押さえておきます。CPU 側 RAM と GPU の VRAM は PCIe というバスでつながっており、ここを通る帯域(1秒に運べるデータ量)が転送速度の上限です。pinned memory はこの PCIe の実効帯域を上限近くまで引き出すための工夫だ、と理解できます。

## 仕組みを詳しく
pinned memory とは、OS に「この領域は絶対に動かさない・ディスクに追い出さない」と固定(pin / lock)してもらった CPU メモリです。物理位置が固定なので GPU の DMA が直接読め、中間コピーが不要になり転送が速くなります。

さらに pinned memory からの転送は非同期(asynchronous)にできます。`non_blocking=True` 相当を指定すると、CPU は「転送を依頼」した瞬間に次へ進め、転送は裏で進みます。これにより前のバッチの GPU 計算と、次のバッチの CPU→GPU 転送を時間的に重ねられます。CUDA の概念では、これらの操作は「ストリーム(stream、GPU 上の作業の順序付きキュー)」に積まれ、同一ストリーム内は順序保証され、別ストリームに置けば転送と計算が並行できます(GPU には計算コアとは別に転送専用の DMA エンジン=copy engine があるため、文字通り同時に動けます)。

多くのフレームワークでは2つの設定を組み合わせます(擬似コード)。

```
loader = DataLoader(dataset, batch_size=8, num_workers=4,
                    pin_memory=True)           # (1) ワーカーがバッチを pinned memory に置く

for batch in loader:
    x = batch["image"].to(device, non_blocking=True)  # (2) 非同期で GPU へ転送
    out = model(x)                                     # 転送と計算が重なりうる
```

- pin_memory: DataLoader のワーカーが用意したバッチを pinned memory に載せて渡す。
- non_blocking: そこから GPU への転送を非同期で行う。

重要なのは、この2つはセットで初めて効くことです。non_blocking だけ指定しても、元が普通のメモリなら非同期転送は安全に成立せず(物理位置が動きうるため)、フレームワークは安全のため同期に倒します。pin 済みのメモリだからこそ、非同期かつ高速な転送ができます。

時間軸のイメージ。

```
通常(同期):    [計算 b0][転送 b1][計算 b1][転送 b2]...   ← 転送中は計算が止まる
pinned+非同期:  [計算 b0          ][計算 b1          ]...
                       [転送 b1(裏で同時進行)][転送 b2 ...]   ← 転送が計算に隠れる
```

数値感覚。pinned memory にすると CPU→GPU 転送帯域が環境次第でおおむね 1.5〜2 倍程度に上がり(報告による。PCIe Gen3 経由の実測でページャブルが数 GB/s、pinned が十数 GB/s、PCIe Gen4 ではさらに高い値に届く例がある)、非同期化で転送時間を計算の裏に隠せます。1バッチの転送が 10ms かかっていて、それが計算の裏に隠れれば、その 10ms がまるごと節約できる、という効き方です。

```
例: 計算 100ms/バッチ、転送 10ms/バッチ
 同期:        110ms/バッチ
 非同期+pinned: ≈100ms/バッチ(転送が計算に隠れる)
 さらに転送帯域 2 倍なら転送自体も 10ms→5ms で、隠せる余裕がさらに増える
```

注意点。pinned memory はディスクへ追い出せない固定領域なので、確保しすぎると物理メモリ(RAM)を圧迫し、OS が他プロセスに割り当てる余地を奪ってシステム全体が不安定化したり、確保に失敗したりします。だから「全データを pin する」のではなく「DataLoader が渡す直前のバッチだけ pin する」のが定石で、これがメモリ消費を抑えつつ効果を得るバランスです。確保自体にも小さなコスト(OS にページロックを依頼するシステムコール)があるため、極端に小さいデータでは割に合わないこともあります。また非同期転送は「転送完了前に結果を使うと壊れる」危険があり、同期の取り方(イベントやストリーム同期)を誤るとデータ競合や不正値の原因になります(フレームワークは通常うまく扱うが、低レベルで自前実装すると注意)。

## 手法の系譜と主要論文
pinned memory と非同期転送は論文の手法ではなく、NVIDIA の CUDA プログラミングモデルが提供する標準機能です。一次資料と関連文脈を挙げます。

- NVIDIA CUDA C Programming Guide。page-locked(pinned)host memory と非同期コピー、CUDA streams による転送と計算のオーバーラップを規定する一次資料。cudaHostAlloc / cudaMemcpyAsync などの API がここで定義される。
- "How to Optimize Data Transfers in CUDA C/C++"(Mark Harris, NVIDIA, 2012)。pinned memory による高速・非同期転送と、ストリームで転送と計算を重ねる実践的解説。pinned で帯域がどれだけ上がるかの計測例が示される。
- tf.data(Murray et al., VLDB 2021, Google)。アクセラレータが高速化するほどデータ供給・転送が相対的ボトルネックになると指摘。pinned memory はその転送段の最適化に当たる。prefetch でホスト側準備を、転送最適化でデバイスへの受け渡しを、それぞれ計算に隠す。
- NVIDIA DALI(Data Loading Library)。デコードや前処理自体を GPU 側で行い、CPU→GPU 転送量そのものを減らすアプローチ。pinned memory が「転送を速くする」のに対し、DALI は「そもそも転送を減らす/GPU で前処理する」発想。目的(データ供給を詰まらせない)は同じでも手段が異なる。
- GPUDirect Storage(NVIDIA)。ストレージから GPU の VRAM へ CPU・システム RAM を経由せず直接 DMA する技術。pinned memory が「CPU RAM→GPU の段」を最適化するのに対し、こちらは「ストレージ→GPU」をショートカットして CPU バウンスバッファ自体を消す、さらに上流の最適化。

これらに共通する原則は「高価な GPU を遊ばせるな。読み込み・前処理・転送を計算の裏に隠せ」です。pinned memory はこの原則の「転送」部分を担います。

## 論文の実験結果(定量データ)
本テクニックは標準機能のため、論文では「手法」ではなく「基盤最適化」として現れます。定量的に言えること。

- NVIDIA の計測(技術ブログ・プログラミングガイド)では、ページャブルメモリからの転送帯域に比べ pinned memory からの転送帯域が概ね 1.5〜2 倍程度に向上する例が報告されています。指標は「転送帯域(GB/s、1秒に運べるデータ量)」。これは PCIe バスの実効効率がページャブルでは中間コピーで落ちるのに対し、pinned では DMA が直接動くため上がる、という説明です。
- tf.data(VLDB 2021)や DNN 入力パイプラインの研究は、転送と準備を計算に隠すことでアクセラレータ稼働率(GPU が実計算していた時間の割合)を上げられると示しています。転送が隠れる分だけ「1ステップ所要時間」が短縮します。
- アブレーション的観点: non_blocking を単独で使う(pin しない)と非同期化が無効化され効果なし。pin_memory のみで non_blocking を使わないと、転送は速くなるが計算と重ならず谷待ちが残る。両方そろえて初めて「速い+隠れる」が成立する、というのが切り分けの結論です。num_workers(並列準備)と併用すると効果が積み上がります。逆に、計算が極端に重い(転送が計算の 1% 程度)モデルでは、隠せる量が小さく効果も小さい、という当然の傾向も観測されます。

指標の意味。転送帯域は「1秒に何 GB 運べるか」で、高解像度の多カメラ画像のように1バッチが大きいほど効きます。例えば 6 カメラ × 高解像度 × バッチを毎ステップ送るような自動運転データでは1バッチが数百 MB に達し、転送の絶対時間が無視できなくなるため、pinned + 非同期の価値が高まります。アクセラレータ稼働率は「GPU が遊ばず計算していた割合」で、これが上がるほど同じ GPU 時間で多く学習でき、コスト効率が直接改善します。

## メリット・トレードオフ・限界
メリット
- CPU→GPU 転送が速くなる(中間コピーが消え、DMA が直接動くため)。
- 非同期転送で転送時間を GPU 計算の裏に隠せ、スループットが上がる。
- 多くのフレームワークでは pin_memory と non_blocking を足すだけで導入でき、コードの大改修が不要。
- num_workers による並列先読みと組み合わせると、データ供給パイプライン全体が効率化する。

トレードオフ・限界
- pinned memory はディスクへ追い出せない固定領域のため、確保しすぎると RAM を圧迫し、システムが不安定化・確保失敗する。確保量に上限がある環境もある。
- non_blocking を単独で使っても効かない(pin でないと同期に倒れる)。セットで使う必要がある。
- 非同期転送は「転送完了前に結果を使うと壊れる」危険があり、同期の取り方を誤るとデータ競合や不正値の原因になる(低レベル自前実装では特に注意)。CPU 側の元バッファを転送完了前に書き換えるのも同様に危険。
- 転送がそもそもボトルネックでない場合(計算が支配的なら)効果は小さい。
- pin する処理自体に小さなコストがあり、極端に小さいデータでは割に合わないことがある。

## 発展トピック・研究の最前線
- CUDA streams と多重バッファリング(double/triple buffering)。複数ストリームで「転送中の次バッチ」「計算中の現バッチ」を多重化し、転送と計算のオーバーラップを最大化する。GPU の copy engine が計算コアと独立に動くことを最大限活かす設計。
- GPU 前処理(DALI)で転送量そのものを削減。エンコード済み(JPEG など圧縮された)データを GPU に送って GPU でデコードすれば、PCIe を渡るデータが圧縮率ぶん小さくなる。
- Unified Memory / GPUDirect Storage。ストレージから GPU へ CPU を経由せず直接データを流す技術で、転送段のボトルネックをさらに削る方向。CPU バウンスバッファを完全に省く。
- 大きなバッチ・高解像度・多カメラの時代の重要性。1バッチのテンソルが大きいほど転送コストの絶対値が増えるため、pinned + 非同期 + GPU 前処理の合わせ技が効く。
- 分散学習との関係。複数 GPU/ノードでは、ホスト→デバイス転送(H2D)に加え、勾配同期などのノード間通信(NCCL の all-reduce 等)も律速になり得る。各段(I/O、前処理、H2D 転送、計算、ノード間通信)を重ねてアイドルを消す設計思想は共通で、pinned memory はそのうち H2D 段を担う1ピースです。
