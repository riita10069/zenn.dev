# KServe (Model Serving)

## ひとことで言うと
KServe は「学習し終えたモデルを、Kubernetes 上で『推論できるサービス』として立ち上げる」ための標準化された仕組みです。`InferenceService` という1枚の宣言 (Kubernetes のカスタムリソース) を書くだけで、外から問い合わせを受け付ける窓口 (エンドポイント) が立ち上がり、負荷に応じて自動でサーバ台数を増減し、必要なら0台にまで縮める (scale-to-zero) ことができます。モデルの種類 (PyTorch、TensorFlow、ONNX、…) を問わず同じ作法で扱えるのが特徴で、機械学習モデルを「本番サービス」に乗せる工程を大幅に省力化します。

## 直感的な理解
モデルを学習し終えたあと、それを実際に「使える」状態にするには、新しい入力を渡して出力を返すサーバが要ります。素朴には Flask や FastAPI で Web サーバを書き、その中でモデルをロードして `model(x)` を呼べば済みそうです。実験や個人利用ならそれで十分です。

ところが本番運用に近づけると、自前で書くべきことが一気に増えます。問い合わせが増えたらサーバを増やし、減ったら減らす (オートスケール)。GPU は高価なので、誰も使わない時間帯は台数を0に縮めたい。新しいモデルにこっそり一部のトラフィックだけ流して様子を見たい (カナリアデプロイ)。前処理・後処理を付け足したい。監視・ログ・リトライ・ヘルスチェックも要る。そして入出力フォーマットがモデルごとにバラバラだと、それを呼ぶクライアント側のコードを毎回書き直すことになる。

これらを毎回モデルごとに手で書くのは無駄が多く、間違いも起きます。KServe は「モデルの種類を問わず、`InferenceService` という統一された宣言を1枚書けば、これらをまとめて面倒見る」ことを目指したプラットフォームです。料理に例えるなら、自前サーバが「皿洗いから配膳まで一人で全部やる」のに対し、KServe は「注文を受ける窓口・厨房への振り分け・混雑時の応援呼び出し・客が引いたら店じまい」を自動でやってくれる店舗運営システムのようなものです。料理 (= モデルの計算そのもの) は別の調理器具 (= 後述するバックエンド推論エンジン) に任せ、KServe は店舗運営に徹します。

## 基礎: 前提となる概念
まず機械学習側の用語を整理します。

- **学習 (training)**: データからモデル (ニューラルネットの重み) を作る工程。時間とお金をかけて一度だけ、または定期的に行う。
- **推論 (inference)**: できたモデルに新しい入力を与え、出力を計算させる工程。実際にモデルを「使う」のはこちら。
- **サービング (serving)**: 推論を「いつでも問い合わせを受けて返せるサービス」として、常時またはオンデマンドで動かしておくこと。学習が「バッチ処理 (一括計算)」なら、サービングは「オンライン処理 (来たら即返す)」に近い。

次に Kubernetes 関連の前提です。Kubernetes を知らない人向けに噛み砕きます。

- **コンテナ**: アプリ本体・ライブラリ・設定をひとまとめに固めた「持ち運べる実行環境」。どこに置いても同じように動く。
- **Kubernetes (K8s)**: 大量のコンテナを多数のサーバ上で動かし、配置・再起動・スケールを自動でやる基盤 (オーケストレータ)。
- **Pod**: K8s が動かす最小単位。1つ以上のコンテナをまとめたもの。推論プロセス1つぶんは1つの Pod として動くと考えてよい。
- **CRD (Custom Resource Definition)**: K8s に「新しい種類のリソース」を追加する仕組み。標準の Pod や Service に加えて、`InferenceService` のような独自の宣言を K8s に理解させられる。KServe はこの CRD として推論サービスを表現する。
- **宣言的 (declarative)**: 「この手順を実行しろ」ではなく「最終的にこうあってほしい」という望ましい状態を書く流儀。利用者は望ましい状態を YAML で宣言し、K8s と KServe (コントローラと呼ばれる常駐プログラム) が現状と望ましい状態の差分を埋め続ける。これを reconcile (調整) ループと呼ぶ。

最後にサーバレスの前提です。

- **サーバレス / scale-to-zero**: 問い合わせが来ないときは稼働インスタンスを0にし、来た瞬間に起動する運用。常時確保による無駄を省けるが、0からの起動には遅延 (コールドスタート) が伴う。KServe は背後で Knative (K8s 上のサーバレス基盤) を使ってこれを実現できる。
- **SLO (Service Level Objective)**: サービスが守るべき目標値。推論サービングでは「99%の応答を 50ms 以内に返す」のような形で定める。後で出てくるレイテンシ議論は、この SLO を守れるかどうかが軸になる。

## 仕組みを詳しく

### InferenceService — 中心となる宣言
KServe の心臓部は `InferenceService` という CRD です。利用者は「どのモデルを、どのフレームワークで、どれくらいの資源 (GPU 等) で、どんなスケール条件でサーブするか」を1枚の宣言で書きます。概念的には次のような内容です。

```yaml
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: example-policy
spec:
  predictor:
    model:
      modelFormat: { name: triton }     # 推論エンジン (バックエンド) に Triton を使う例
      storageUri: s3://bucket/models/example-policy/v3
      resources:
        limits: { nvidia.com/gpu: "1" } # GPU を1枚割り当て
    minReplicas: 0     # 誰も使わなければ 0 台に縮小 (scale-to-zero)
    maxReplicas: 3     # 負荷が高ければ最大3台
```

この宣言から KServe のコントローラが自動で行うこと:

1. **モデルの取得**: `storageUri` (S3 / GCS / PVC など) からモデルファイルを取ってくる。多くの構成では、モデルレジストリ (学習済みモデルにバージョンを付けて管理する仕組み) に登録されたチェックポイントの場所を指す。初期化用のサイドカーコンテナ (storage-initializer) がダウンロードを担う。
2. **推論サーバの起動**: 指定したバックエンドでモデルをロードし、推論できる状態にする。
3. **エンドポイントの公開**: HTTP / gRPC の窓口を立て、標準化された入出力プロトコルで問い合わせを受け付ける。
4. **オートスケール**: 問い合わせ量に応じて Pod を増減する。`minReplicas: 0` なら未使用時は0台にして GPU コストを節約し、最初の問い合わせが来たら起動する。

ここで重要なのは、利用者は「何枚 GPU を、何台、どこから」を宣言するだけで、Pod の生成・ヘルスチェック・ルーティング・スケールという「運用の手続き」を一切書かない点です。これが Sculley らの言う「ML の周辺で発生する技術的負債 (glue code)」を標準コンポーネントへ吸収する、という発想です。

### 3つの役割 — Predictor / Transformer / Explainer
KServe は推論を役割で分けられます。

- **Predictor (予測器)**: モデル本体の推論を行う中核。必須。
- **Transformer (変換器)**: 推論の前処理 (画像のリサイズ・正規化、テキストのトークナイズ等) と後処理 (出力の整形、デコード等) を担う任意の段。学習時と同じ前処理を本番でもそろえる役割を持つ。学習と推論で前処理がズレると静かに精度が落ちる (training-serving skew と呼ばれる典型的な事故) ため、ここを明示的な段として持てるのは実務上大きい。
- **Explainer (説明器)**: なぜその出力になったかの説明 (特徴量の寄与など) を返す任意の段。SHAP や統合勾配などの説明手法をここに差せる。

これらを分けると、前処理だけ別コンテナにして CPU で回し、Predictor だけ GPU を使う、といった資源の使い分けができます。GPU は高価なので、軽い前処理に GPU を占有させないこの分離は効率上重要です。内部的には Transformer → Predictor の順にデータが流れ、各段が独立にスケールします。

### バックエンド (中身の推論エンジン) は差し替えられる
KServe 自体は「推論サービスの枠組み・宣言・オートスケール・ルーティング」を提供し、実際にモデルを高速に動かす中身 (推論エンジン = serving runtime) は差し替えられます。簡単なケースでは KServe 同梱のモデルサーバ (sklearn server、torchserve、ONNX runtime など) を使い、高性能が要るケースでは [Triton Inference Server](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0525_triton-inference-server) を指定します。つまり「KServe = 入れ物と運用の枠」「Triton 等 = 中の高性能エンジン」という明確な役割分担です。この分離のおかげで、新しいランタイム (LLM 向けの vLLM 等) が出てきても KServe の上に乗せ替えられます。

### Open Inference Protocol (v2 推論プロトコル)
KServe は推論の入出力に標準プロトコルを定めています。当初は v1 (シンプルな JSON)、後に Triton と共同で v2 (Open Inference Protocol、OIP) を策定しました。v2 では tensor を「名前・データ型・shape・値の配列」で表現し、HTTP/REST と gRPC の両方で同じ意味論を持ちます。これにより、フレームワークが違っても、Triton 等のバックエンドが違っても、クライアントは同じ作法で問い合わせられます。

例えば1枚の画像 (shape `[1,3,224,224]` = 1枚・3チャネルRGB・縦224・横224) を送ると、出力 tensor (例: shape `[1,1000]` の1000クラス分のスコア) が返る、という統一形式です。リクエストの概念は次のようになります。

```
POST /v2/models/example-policy/infer
{
  "inputs": [
    { "name": "images", "datatype": "FP32", "shape": [1,3,224,224], "data": [...] }
  ]
}
→ レスポンス
{
  "outputs": [
    { "name": "scores", "datatype": "FP32", "shape": [1,1000], "data": [...] }
  ]
}
```

このプロトコルが共通化されたことで、クライアント側のコードがバックエンド非依存になり、Triton でも他のランタイムでも同じクライアントが使い回せます。

### scale-to-zero とコールドスタート
KServe は背後で Knative を使い、問い合わせが一定時間来ないと Pod を0台に縮め、最初の問い合わせが来た瞬間に起動できます。GPU は高価なので「評価のときだけ立ち上げ、ふだんは0台」というコスト効率の良い運用ができます。ただし0台からの起動には、Pod のスケジューリング・コンテナイメージ取得・モデルロードを含めて数秒〜数分の遅延 (コールドスタート) が伴います。常に即応が要る用途では `minReplicas: 1` 以上にして温存する、というトレードオフを取ります。

```
[scale-to-zero の挙動イメージ]
 トラフィック ▁▁▁▁██████▁▁▁▁▁▁▁██▁▁
 Pod 台数     0   1→2→3    →1→0    0→1(コールドスタート)
                  ↑負荷で増   ↑無い間0で節約  ↑最初の1件は遅い
```

Knative のオートスケーラ (KPA、Knative Pod Autoscaler) は既定で「Pod あたりの同時実行数 (concurrency)」を指標にスケールします。例えば「1 Pod あたり同時10リクエストまで」と決めておくと、同時35リクエストなら 4 Pod に増える、という具合です。CPU 利用率ベースの標準オートスケーラより、推論のような「1件が重い処理」に向いた指標です。

## 手法の系譜と主要論文
KServe は単一の論文ではなく、複数の研究潮流が実務に結晶したものです。系譜を追うと位置づけが見えます。

- **2015年 — ML システムの技術的負債の指摘**: Sculley ら "Hidden Technical Debt in Machine Learning Systems" (NeurIPS) は、ML のコードそのものより、その周辺 (配信・設定・データ依存・glue code) に巨大な保守コストが潜むと警告しました。論文は「ML コードは全体のごく一部に過ぎず、周囲の配管 (plumbing) が支配的」という有名な図でこれを示しています。「モデルを作ること」と「モデルを本番で配信し続けること」は別物で、後者の標準化が要る、という問題意識が KServe の存在理由につながります。
- **2017年 — 推論サービングの体系化**: Crankshaw ら "Clipper: A Low-Latency Online Prediction Serving System" (NSDI、UC Berkeley) は、推論サービングにおけるレイテンシ (応答の速さ) とスループット (単位時間あたりの処理量) のトレードオフ、適応的バッチング、複数モデルからの選択を体系化しました。KServe / バックエンドのバッチングやスケールはこの系譜の実務的実装です。同年 Olston ら "TensorFlow-Serving" (arXiv:1712.06139) が、本番でモデルを安全に差し替えながら配信する仕組み (バージョニング、ホットスワップ、複数モデル同居) を示し、推論サーバの実務要件を整理しました。
- **2018-2019年 — Kubernetes 上の ML 基盤**: Google 主導の Kubeflow プロジェクトの一部として KFServing が登場。後に独立し、名称を KServe に変更、CNCF / LF AI & Data の傘下になりました。CRD ベースで宣言的に推論サービスを表す、という Kubernetes ネイティブな設計がここで固まりました。
- **2019年 — サーバレスの整理**: Jonas ら "Cloud Programming Simplified: A Berkeley View on Serverless Computing" (arXiv:1902.03383) は「資源を常時確保するのは無駄、来たら起動する」という FaaS (Function as a Service) の思想を整理しました。KServe の scale-to-zero はこの思想を GPU 推論に持ち込んだものです。
- **2020年 — 予測可能なレイテンシの追求**: Gujarati ら "Serving DNNs like Clockwork" (OSDI) は、推論の実行時間が決定的に予測できる性質を使い、テールレイテンシ (遅い側の応答時間) を厳しく抑える集中スケジューラを提案しました。「平均は速いが時々ものすごく遅い」というサービングの厄介な性質に対する研究で、SLO 厳守型サービングの理論的背景になっています。
- **現在**: KServe は ModelMesh (多数の小モデルを少数のサーバに同居させ、メモリに乗り切らない数のモデルを動的にロード/アンロードする方式) を統合し、大規模言語モデル (LLM) 向けの専用ランタイム (vLLM 等との連携) も整備しています。Open Inference Protocol は Triton と共通化され、デファクト標準になりました。

## 論文の実験結果 (定量データ)
KServe 自体はプロダクト/OSS であり論文の定量ベンチマークを持ちませんが、その中核技術には研究の数値的裏付けがあります。

- **適応的バッチングの効果 (Clipper, NSDI 2017)**: Clipper は適応的バッチングにより、レイテンシ上限 (例: 20ms) を満たしたままスループットを大幅に向上させられることを示しました。論文では、画像分類モデルでバッチングなしと比べてスループットが数倍に伸びる一方、99 パーセンタイルレイテンシ (遅い側から1%の応答時間) を SLO 内に保てることを報告しています。「スループットが数倍」とは、同じ GPU で単位時間あたりさばける問い合わせ数が数倍になるという意味で、必要 GPU 枚数 = コストに直結します。
- **テールレイテンシの制御 (Clockwork, OSDI 2020)**: Clockwork は、推論を「予測可能な実行時間」として扱うことで、同時に多数のモデルをホストしながら 99.9 パーセンタイルレイテンシ (1000件に1件の最悪応答) を厳しいSLO内に収められることを示しました。論文では数千モデル規模でも SLO 違反率を極めて低く抑えられたと報告しています。意味: サービングの難しさは「平均」ではなく「たまに出る極端に遅い応答 (テール)」にあり、それを設計で潰せるという定量的証拠です。
- **scale-to-zero のコスト/遅延トレードオフ**: サーバレス推論の一般的な報告では、scale-to-zero により低トラフィック時のコストを大きく削減できる (使われない時間帯のインスタンス費がゼロ) 一方、コールドスタートが数百 ms〜数分加わります。モデルが大きい (数 GB〜数十 GB のロードが必要な LLM 等) ほどコールドスタートは長くなる傾向があり、「即応性 vs コスト」の調整パラメータ (`minReplicas`、スケールダウンの猶予時間) で運用者が決めます。
- **ModelMesh のメモリ効率**: 多数の小モデルを抱えるケースで、ModelMesh は使われていないモデルをメモリから退避し、要求が来たらロードする LRU 的キャッシュ (最近使われていないものから捨てる方式) を使います。これにより、全モデルを常駐させる方式と比べて必要メモリ (= GPU 枚数 = コスト) を桁で削減できると報告されています。代償はキャッシュミス時のロード遅延です。

これらの「数値の意味」: スループットは GPU 1枚で何件さばけるか (= 何枚 GPU が要るか = いくらかかるか) を決め、レイテンシは利用者の体感速度を決めます。両者はバッチングで綱引きの関係 (バッチを大きくするとスループット↑だがレイテンシ↑) にあり、KServe/バックエンドはこの綱引きを設定で制御します。

## メリット・トレードオフ・限界
メリット:
- `InferenceService` の宣言1枚で、推論サービスの起動・公開・オートスケール・ルーティングをまとめて行える。手続き的な運用コードを書かずに済む。
- フレームワーク非依存で、Open Inference Protocol という統一プロトコルを使える。クライアントがバックエンドに縛られない。
- scale-to-zero で、未使用時は GPU を0台にしてコストを大きく削減できる。
- Predictor / Transformer / Explainer の分離で前処理と推論の資源を使い分けられ、training-serving skew を防ぎやすい。
- カナリアデプロイ (新モデルに一部トラフィックだけ流し、指標を見てから全面切替) や A/B テストなど本番運用機能を備える。

トレードオフ・限界:
- 背後に Knative / Istio (サービスメッシュ) など追加コンポーネントが必要で、構成が重く運用の学習コストが高い。小規模用途には過剰になりがち (近年はこれを軽くする RawDeployment モードもある)。
- scale-to-zero はコールドスタート遅延を生むため、即応が必須の用途では `minReplicas≥1` にせざるを得ず、節約効果が薄れる。
- KServe 単体では高速化 (動的バッチング等) は弱く、性能はバックエンド ([Triton Inference Server](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0525_triton-inference-server) 等) に依存する。
- GPU の割り当て粒度が「1枚単位」になりがちで、小さなモデルでは GPU を持て余す。ModelMesh や MIG (GPU 分割) で緩和するが設定は複雑。
- 未解決の課題として、LLM のような巨大モデルのコールドスタート短縮、GPU の細粒度共有、マルチモデルのスケジューリング最適化、テールレイテンシの保証などが研究・開発の途上にある。

## 発展トピック・研究の最前線
- **LLM 推論サービング**: vLLM の PagedAttention (KV キャッシュをページ単位で管理しメモリ断片化を減らす)、連続バッチング (continuous batching、生成中のシーケンスを動的に入れ替える) など、自己回帰生成特有の最適化が活発。KServe もこれらランタイムを取り込みつつある。
- **コールドスタート削減**: モデルの遅延ロード、コンテナイメージのストリーミング起動、メモリスナップショット復元などで「0台から最初の応答まで」を縮める研究。
- **GPU 共有 (multi-tenancy)**: MIG (Multi-Instance GPU、1枚を複数の独立スライスに分割) や MPS (Multi-Process Service) を使い、1枚を複数モデルで安全に共有する方向。隔離 (あるモデルの負荷が別のモデルのレイテンシを乱さない) が研究課題。
- **オートスケールの指標**: 同時実行数やリクエストレートだけでなく、GPU 利用率・キュー長・レイテンシ SLO を直接ターゲットにするスケーラの研究。Clockwork 的な「予測可能性」を活かす方向も含む。
