# バックボーンのステージ別チャネル数の動的発見

## ひとことで言うと
画像認識の土台部分(バックボーン)は種類によって「途中で何チャンネルの特徴を出すか」が違います。それを人間がいちいち書くのではなく、モデル自身に問い合わせて自動で調べる仕組みの話です。これにより、バックボーンを差し替えたり新規追加したりしても、後段の入力チャンネル数を手で直さずに済みます。さらに、Transformer 系と畳み込み系でテンソルの次元の並び順が違う問題も、発見した情報を使って吸収します。

## 直感的な理解
電源タップを思い浮かべてください。差し込む機器(バックボーン)によってプラグの形(出力チャンネル数)が違うとします。後段の機器は「何ピンの差し込み口を用意すればいいか」を知らないと作れません。毎回プラグの形を人間が定規で測って後段に書き写すのは面倒で間違えやすい。賢いタップなら、機器を差した瞬間に自動で「これは何ピンだ」と認識して、後段の差し込み口をその場で合わせてくれるでしょう。

バックボーンのチャンネル数も同じです。ResNet50 と Swin Transformer では、各段階で出すチャンネル数も段数も違います。後段の融合層(複数の特徴をまとめる層)は「合計何チャンネル入ってくるか」を知らないと作れません。これを固定値で直書きすると、バックボーンを変えた瞬間に壊れます。そこで「モデルに聞いて自動で合計を出す」のが動的発見です。

## 基礎: 前提となる概念

用語を一つずつ平易に説明します。

- チャンネル (channel): 特徴マップ(画像から抽出した中間データ)の「奥行きの数」。カラー画像は赤・緑・青の3チャンネルですが、ネット内部では数十から数千チャンネルになります。1チャンネルは「ある種類の模様(縦線・丸み・質感など)にどれだけ反応したか」を表す1枚の地図だと思ってください。
- ステージ (stage): バックボーンが画像を段階的に処理する各段。浅いステージは細かい模様(エッジ)を、深いステージは大きな意味(物体らしさ)を捉えます。多くのバックボーンは4ステージですが、ResNet 系は stem を含め5段相当の特徴を出します。
- 特徴マップ (feature map): 各ステージが出力する `(B, C, H, W)` という4次元テンソル。B はバッチ数、C はチャンネル数、H/W は縦横。深いステージほど H/W は小さく(解像度が落ち)、C は大きく(チャンネルが増え)なります。
- マルチスケール特徴: 浅い(高解像度・低チャンネル)から深い(低解像度・高チャンネル)まで複数ステージの特徴を併用すること。小さい物体は浅い層、大きい意味は深い層が得意なので、両方使うと認識が頑健になります。
- timm: 多数の画像認識モデルを統一インターフェースで使える Python ライブラリ(PyTorch Image Models)。`features_only=True` で各ステージの特徴マップを返すモードにできます。
- channels-first / channels-last: テンソルの次元の並び順。`(B, C, H, W)` はチャンネルが2番目(channels-first)、`(B, H, W, C)` はチャンネルが最後(channels-last)。畳み込み系は前者、Transformer 系は後者を返すことが多く、混在すると後段が壊れます。

なぜ動的発見が必要かを具体例で示します。複数ステージの特徴マップをチャンネル方向に連結(concatenate)して使う設計を考えます。後段の融合層は「入力チャンネルが合計いくつか」を知らないと、最初の畳み込み層 `nn.Conv2d(in_channels, ...)` の `in_channels` を決められません。バックボーンごとの合計チャンネル数は大きく異なります。

| バックボーン | ステージ数 | 各ステージのチャンネル | 合計 |
|---|---|---|---|
| swin_v2_tiny | 4 | 96, 192, 384, 768 | 1440 |
| conv_next_v2_tiny | 4 | 96, 192, 384, 768 | 1440 |
| res_net_50 | 5 | 64, 256, 512, 1024, 2048 | 3904 |

もし合計を `in_channels=1440` と直書きしたら、ResNet50(3904)に切り替えた瞬間に次元が合わず壊れます。新しいバックボーンを足すたびに人間が正しい数値を調べて書き換えるのは間違いやすく面倒です。だから「モデルに聞いて自動で合計を出す」仕組みが要ります。

## 仕組みを詳しく

典型的な実装は、バックボーン構築時に各ステージのチャンネル数を発見し、合計を `backbone_channels` として保持します。

```python
self.feature_channels = self._infer_feature_channels()   # 例: [64, 256, 512, 1024, 2048]
self.backbone_channels = sum(self.feature_channels)        # 例: 3904
```

発見は2段構えにすると堅牢です。

ステップ1(優先・速い): ライブラリのメタデータを読む。timm のモデルを `features_only=True` で作ると `feature_info` という属性が付きます。これは各ステージの情報のリストで、各要素が `num_chs`(そのステージのチャンネル数)を持ちます。フォワード(実際の計算)を一切走らせずにチャンネル数が分かるので、速くて安全です。

```python
info = getattr(self.backbone, "feature_info", None)
if info is not None:
    try:
        channels = [stage["num_chs"] for stage in info]
    except (TypeError, KeyError, IndexError):
        channels = None
    if channels:
        return channels
```

ResNet50 なら `[64, 256, 512, 1024, 2048]`(合計 3904)、SwinV2-Tiny なら `[96, 192, 384, 768]`(合計 1440)が得られます。

ステップ2(保険・遅い): `feature_info` を持たない自作バックボーンへのフォールバック。timm 製でないカスタムバックボーンはこの属性を持たないかもしれません。その場合は、ゼロ埋めのダミー画像 `(1, 3, 256, 256)` を一度だけ流し、返ってきた各特徴マップの形からチャンネル数を読み取ります。

```python
with torch.no_grad():
    dummy = torch.zeros(1, 3, 256, 256, device=device, dtype=dtype)
    out = self.backbone(dummy)
return [f.shape[1] for f in out]   # 各特徴マップの C 次元
```

`torch.no_grad()` は「勾配(学習用の微分情報)を計算しない」指定で、ただ形を知りたいだけのこの場面では無駄なメモリと計算を省きます。`f.shape[1]` は4次元テンソル `(B, C, H, W)` の2番目、すなわちチャンネル数 C です。出力が4次元テンソルのリストでない場合は分かりやすいエラーを投げ、原因不明の失敗を防ぎます。

さらに、フォワード時にレイアウト差(channels-first / channels-last)を吸収する処理を入れます。timm の Swin 系は `(B, H, W, C)`(チャンネルが最後)で返し、ResNet/ConvNeXt 系は `(B, C, H, W)`(チャンネルが2番目)で返すことがあります。発見時に分かったチャンネル数 `expected_c` と実際のテンソルの形を比べ、最後の次元が `expected_c` と一致していれば「チャンネルが最後にある」と判断し、`permute`(次元の並べ替え)で `(B, C, H, W)` に揃えます。

```python
for f, expected_c in zip(features, self.feature_channels):
    if f.dim() == 4 and f.shape[1] != expected_c and f.shape[-1] == expected_c:
        permuted.append(f.permute(0, 3, 1, 2))   # (B,H,W,C) -> (B,C,H,W)
    else:
        permuted.append(f)
```

これで後段は常に `(B, C, H, W)` を前提でき、バックボーンの種類差を意識せずに済みます。発見した `expected_c` を「どちらの次元がチャンネルか」の判定基準に流用しているのがうまい点で、形だけ見て曖昧に推測するより安全です(たとえば正方形の特徴マップで H==W のときでも、C が分かっていれば取り違えません)。

設計の理由をまとめると、(1) チャンネル数は後段の層の形を決める「契約」であり、契約を一箇所(モデル自身)から取得すれば二重管理によるずれを防げる、(2) メタデータ優先・フォワードフォールバックの二段構えで、速度(メタデータ)と汎用性(フォワード)を両立する、(3) レイアウト補正に発見済みチャンネル数を使うことで、Transformer/畳み込みの混在を頑健に吸収する、という三点です。

## 手法の系譜と主要論文

特定の論文が提案した単一技法ではなく、各バックボーン論文の構造設計と、それらを統一的に扱う実装慣行の組み合わせです。

- ResNet (He et al., "Deep Residual Learning for Image Recognition", CVPR 2016, arXiv:1512.03385): 残差接続(出力に入力を足し戻す)で非常に深いネットワークの学習を可能にしました。ResNet50 は stem(64ch) と4つの残差ステージ(256/512/1024/2048ch)を持ち、特徴抽出として5段相当のチャンネルを出します。合計 3904 の根拠です。
- Feature Pyramid Networks (Lin et al., CVPR 2017, arXiv:1612.03144): 複数ステージのマルチスケール特徴を組み合わせて検出精度を上げる枠組みを確立しました。「各ステージのチャンネル数を後段が知る必要がある」という本トピックの動機そのものを一般化した古典です。横方向接続(lateral connection)で深い層の意味を浅い層の高解像度に伝え、各レベルを共通チャンネル数(典型的には256)に揃える、という設計が後の検出器の標準になりました。
- Swin Transformer / SwinV2 (Liu et al., ICCV 2021 arXiv:2103.14030 / CVPR 2022 arXiv:2111.09883): ウィンドウ単位の自己注意で計算量を画像サイズに線形へ抑えた階層型 Transformer。4ステージで 96/192/384/768ch と段階的にチャンネルを倍増させます。出力が channels-last 寄りなのも Transformer 系の慣習で、レイアウト補正が必要になる典型です。
- ConvNeXt / ConvNeXtV2 (Liu et al., CVPR 2022 arXiv:2201.03545 / Woo et al., CVPR 2023 arXiv:2301.00808): Transformer の設計思想(大きなカーネル、LayerNorm、GELU など)を畳み込みに取り込んだモデル。Tiny は Swin-Tiny と同じ 96/192/384/768ch 構成で、両者を交換しやすくしています。ConvNeXtV2 は Masked Autoencoder 流の自己教師あり事前学習(FCMAE)を組み合わせました。
- timm の `feature_info` (Ross Wightman, timm ライブラリ): フォワードなしでステージ別チャンネルを公開する事実上の標準インターフェース。`features_only=True` と組み合わせ、本トピックの第一手段になります。

## 論文の実験結果(定量データ)

各バックボーンの ImageNet-1K(120万枚・1000クラスの画像分類ベンチマーク)での代表的な数値を挙げます。Top-1 精度は「最も確信したクラスが正解だった割合」で、1ポイントの差でも当時の最先端競争では大きな意味を持ちます。

- ResNet50: ImageNet-1K Top-1 約76パーセント(原論文の標準学習)。残差接続の導入で、152層という当時前例のない深さを安定して学習でき、ILSVRC 2015 の分類・検出・セグメンテーションで優勝しました。残差なしの同等深さネットワーク(plain net)は層を深くすると学習誤差すら劣化したのに対し、残差ありは深くするほど改善した、というアブレーションが残差接続の効果を直接示しています。
- Swin-Tiny: ImageNet-1K Top-1 約81.3パーセント、計算量 約4.5 GFLOPs。同等計算量の DeiT-S(約79.8パーセント)を上回りました。シフトウィンドウを外して通常のウィンドウ注意にするアブレーションでは精度が約1ポイント以上低下し、ウィンドウ間の情報交換が効いていることが示されています。SwinV2 はさらに大解像度・大モデルへスケールさせる安定化(post-norm、scaled cosine attention、log-spaced continuous position bias)を加えました。
- ConvNeXt-Tiny: ImageNet-1K Top-1 約82.1パーセント、計算量 約4.5 GFLOPs。同規模の Swin-Tiny(約81.3パーセント)を約0.8ポイント上回り、「純畳み込みでも適切に近代化すれば Transformer に匹敵・凌駕する」ことを示しました。論文はパッチ化ステム・大カーネル(7x7 depthwise)・LayerNorm 化・GELU 化・stage 比率変更などの改良を ResNet50 から一つずつ積み上げるアブレーション(モダナイズの roadmap、76.1 → 82.0 程度まで段階的に上昇)で各要素の寄与を定量化しています。
- ConvNeXtV2-Tiny: FCMAE 事前学習と Global Response Normalization(GRN)の追加で、ImageNet-1K Top-1 が ConvNeXtV1-Tiny からおよそ1から2ポイント改善したと報告されています(設定により値は変動)。GRN を抜くアブレーションでは FCMAE 事前学習との相性が悪化し、特徴の冗長性(feature collapse)が増えることが示されています。

これらの数値が本トピックに意味するのは、「同じ規模(約4.5 GFLOPs)でも、選ぶバックボーンで精度は数ポイント動き、各ステージのチャンネル構成も段数も異なる」という事実です。だからこそ手で固定値を書くのは脆く、差し替えを安全にする動的発見が要ります。Swin/ConvNeXt-Tiny が同じ 96/192/384/768 構成なのは偶然ではなく、互換性を意識した設計で、合計 1440 が一致するため後段の差し替えが容易になっています。

## メリット・トレードオフ・限界

- 利点: 新しいバックボーンを追加しても、チャンネル数を手で書き換える必要がない。設定ミスによる次元不整合のバグを構造的に防げる。
- 利点: `feature_info` 経由ならフォワードなしで即座に分かり、起動が速い。
- 利点: レイアウト差(channels-last / channels-first)をフォワードで吸収するので、後段はバックボーン種別に依存しない。
- トレードオフ: フォールバックのダミーフォワードは一度だけとはいえ計算とメモリを使う。`feature_info` を持つ timm モデルなら回避される。
- 限界: フォールバックは「4次元・channels-first 出力」を仮定する。3次元出力(トークン列を返す ViT 系など)や非標準レイアウトのバックボーンには対応せず、その場合は明示エラーになる。
- 限界: レイアウト補正は「最後の次元がチャンネル数と一致するか」で判定するため、まれに H や W がたまたまチャンネル数と同じ値だと誤判定の余地がある(実務上は稀だが、保証ではない)。
- 未解決の課題: ViT のようにステージ概念が曖昧で単一スケールのトークン列を返すバックボーンや、可変解像度・動的トークン数のモデルでは、「ステージ別チャンネル」という枠組み自体が当てはまらない。マルチスケール特徴をどう統一インターフェースで扱うかは継続的な設計課題です。

## 発展トピック・研究の最前線

- 統一バックボーンインターフェース: timm の `features_only` / `feature_info` のように、検出・セグメンテーション・3D 認識のフレームワーク(MMDetection など)はバックボーンの出力契約を抽象化しています。動的発見はこの抽象化を実装側で完結させる手段です。
- ViT のマルチスケール化: 元の ViT(Dosovitskiy et al., "An Image is Worth 16x16 Words", ICLR 2021)は単一スケールのトークン列を返すだけでしたが、PVT・Swin・MViT などが階層型に拡張し、検出・セグメンテーションでマルチスケール特徴を取り出せるようにしました。本トピックの「ステージ別チャンネル」前提が成立する条件を広げる流れです。
- BEV・マルチモーダルでの応用: 自動運転では複数カメラやラスタ化した地図など複数の入力ソースから特徴を取り、チャンネル方向に連結したり俯瞰視点(BEV)へ射影したりします。各ソースのチャンネル数を動的に取得して融合層を構築する発想は、本トピックと同型です。
- ニューラルアーキテクチャ探索 (NAS): バックボーン構造を自動探索すると、ステージ数やチャンネル数が探索のたびに変わります。動的発見は、こうした構造可変な設定で後段を自動整合させる前提技術になります。
