# ROS bag / MCAP コンテナ

## ひとことで言うと

ROS bag(ロスバッグ)と MCAP(エムキャップ)は、ロボットや自動運転車が走行中に出す大量のセンサーデータ(カメラ・LiDAR・GPS・速度など)を、すべて時刻つきで1つのファイルに束ねて記録するための「ログの入れ物(コンテナ)」です。動画における mp4 が映像と音声を1つのファイルにまとめるのと同じ役割を、ロボットが吐き出す何十種類ものデータの流れに対して果たします。目的は「実車で起きたことを、後からビット単位で完全に再現できる」ようにすること。自動運転やロボティクスの生ログの実体は、多くの場合この種のコンテナフォーマットです。

## 直感的な理解

自動運転車を1台思い浮かべてください。前後左右に複数のカメラ、屋根に LiDAR(レーザーで周囲の距離を測る装置)、車体に IMU(慣性計測装置、加速度と角速度を測る)、GPS、車輪速センサー、ハンドル角センサーが付いています。これらは互いに無関係なタイミングで、別々の頻度でデータを吐き出します。カメラは秒10〜30枚、LiDAR は秒10回転、IMU は秒100〜400回。これらが同時並行で流れ続けます。

ここで「あるテスト走行で起きた不具合(例えば交差点で突然ブレーキした)を、研究室の机の上で何度も再現したい」と思ったとします。各センサーを別々のファイルに保存したらどうなるでしょうか。再生のたびに「このカメラ画像はあの LiDAR スキャンの何ミリ秒後だったか」という時刻合わせを手作業でやる羽目になり、さらに「どのセンサーがどの瞬間に何を出したか」という順序関係(因果の手がかり)が失われてしまいます。ブレーキの直前にカメラが何を見ていたかを再現できなければ、原因究明はできません。

そこで発想を変えます。すべてのデータを、出てきた時刻順に、1本のタイムラインに混ぜて記録するのです。各データ片には「自分はいつ・どのセンサーの・何のデータか」というラベルが付いています。これがあれば、再生時に元のタイミングと順序を寸分違わず復元できます。録画(record)して、後で再生(replay)する。テレビ番組を録画して見返すのと同じ発想を、ロボットの全センサーに適用したものが ROS bag であり MCAP です。

## 基礎: 前提となる概念

まず ROS(Robot Operating System、ロボット用オペレーティングシステム)を理解する必要があります。名前に OS とありますが、Windows のような OS ではなく、ロボットソフトウェアを部品(ノード)に分けて、部品同士をメッセージで通信させるための共通基盤(ミドルウェア)です。各ノードは「トピック」という名前付きの通信路にメッセージを publish(発行)し、別のノードがそれを subscribe(購読)します。例えばカメラドライバのノードが `/camera/front/image` というトピックに画像を流し、物体検出ノードがそれを受け取る、という具合です。この疎結合な設計のおかげで、部品を差し替えても他に影響しません。

次に押さえるべき用語を噛み砕きます。
- メッセージ: トピックに流れるデータ1個。型(スキーマ、データの構造定義)が決まっている。例えば画像なら `sensor_msgs/Image` 型、3D点群なら `sensor_msgs/PointCloud2` 型。型が決まっているとは「このメッセージは幅・高さ・画素のバイト列を持つ」のような構造が事前に約束されているということ。
- シリアライズ(serialization): メモリ上のデータ構造を、保存・送信できる連続したバイト列に変換すること。逆に読み戻すのがデシリアライズ。ファイルに記録するときは必ずシリアライズが要る。なぜなら、メモリ上のオブジェクトはポインタなどで散らばっており、そのままディスクに書けないから。
- タイムスタンプ: そのメッセージが生成された時刻。ロボットでは通常ナノ秒(10億分の1秒)精度で持つ。センサー同士の時刻合わせはミリ秒以下の精度が要るため、これだけの分解能が必要になる。例えば LiDAR が回転している間に車が動くと、点群とカメラのずれは数十ミリ秒でも問題になる。
- コンテナフォーマット: 中に異なる種類のデータを時系列で詰め込める「箱」の規格。箱自体は中身が何であるかを問わない。mp4(映像+音声)や zip(任意のファイル群)が身近な例。

ROS には ROS 1 と ROS 2 があります。ROS 1(2007年〜)は研究用途で広く普及しましたが、単一のマスタープロセスに依存する(これが落ちると通信が止まる)・リアルタイム性が弱い・セキュリティ機構が無いといった限界があり、ROS 2(2017年〜、Macenski et al., Science Robotics 2022)では DDS(Data Distribution Service、産業用の分散通信規格)を採用し、産業利用に耐える設計へ作り直されました。現代の自動運転スタックは ROS 2 ベースが主流です。

## 仕組みを詳しく

ROS bag / MCAP の中身の基本構造は次の3要素です。

1. チャンネル(ROS で言うトピック)の登録。データの流れ1本ごとに、名前とそのメッセージ型(スキーマ)を登録する。例: `/camera/front` は `sensor_msgs/Image` 型、`/lidar/top` は `sensor_msgs/PointCloud2` 型。
2. メッセージ本体。実データ1個。タイムスタンプ + どのチャンネルか + シリアライズされたバイナリ本体(エンコードされた画像や点群)を持つ。
3. インデックス / チャンク。後から「この時刻範囲のこのトピックだけ取り出す」を高速に行えるよう、時刻順の索引(インデックス)と、複数メッセージをまとめて圧縮した塊(チャンク)を持つ。

時系列レイアウトのイメージ(ASCII):

```
時刻 →
t=0.000  /lidar/top      (3D点群, ~2MB)
t=0.005  /imu            (加速度・角速度, 数十バイト)
t=0.010  /imu
t=0.012  /camera/front   (JPEG画像, ~200KB)
t=0.012  /camera/rear    (JPEG画像)
t=0.015  /vehicle/speed  (浮動小数1個)
t=0.020  /imu
...
```

1つのファイルにすべてが時刻順で混在し、各メッセージが自己記述的(自分が何者かを内包する)です。これで再生時に元の順序とタイミングを完全に復元でき、しかもインデックスのおかげで「t=10秒から12秒の LiDAR だけ」のような部分読み出しがファイル全体を舐めずに済みます。インデックスとは、要するに「どのトピックの・どの時刻のメッセージが・ファイルの何バイト目にあるか」という目次です。

ROS bag と MCAP の系譜上の違いを整理します。
- ROS 1 rosbag(`.bag`): ROS 1 専用。ROS 1 のメッセージ型に強く結合しており、ファイル末尾にまとめてインデックスを書く構造のため、記録の途中で異常終了するとインデックスが書かれず壊れやすい弱点がありました。
- ROS 2 初期の `.db3`: SQLite データベースをそのままログに流用した形式。汎用 DB ゆえに行のランダム挿入・トランザクション管理のオーバーヘッドがあり、毎秒数百 MB を吐く多センサー連続録画では書き込み性能とファイルサイズの効率が課題でした。
- MCAP(Foxglove が2022年に公開したオープン仕様、mcap.dev): これらを解決する新世代コンテナ。設計上の核心は4つ。(a) シリアライズ非依存 — ROS 1 メッセージでも ROS 2 でも Protobuf でも JSON でも FlatBuffers でも詰められる。(b) 追記(append)に強い — 末尾に書き足していく構造で、途中で電源が落ちてもそこまでが読める(クラッシュ耐性)。(c) チャンク単位の圧縮(zstd や lz4)— 複数メッセージをまとめて圧縮するので高圧縮率と高速性を両立。(d) 自己記述スキーマの埋め込み — ファイル内にメッセージ型定義を同梱するので、その型を知らないツールでも構造を解釈できる。

MCAP が「シリアライズ非依存」であることの意味は重要です。容れ物(コンテナ)と中身(メッセージのエンコード方式)を分離したため、ROS に依存しない独自システムのログにも MCAP を使えます。これは ROS 1 bag が ROS 1 型と一体だったのとは対照的な、汎用化の方向です。実務では、メーカーごとに独自の ROS 型を使っていても、MCAP に入っていれば同じビューアで開けるという互換性のメリットが大きく効きます。

## 手法の系譜と主要論文

- ROS の登場(Quigley, Gerkey, Conley ほか, "ROS: an open-source Robot Operating System", ICRA workshop 2009)。ロボットソフトウェアをノードとトピックの疎結合通信で組む基盤を提案。動機は、研究のたびにセンサードライバ・通信・記録機構が再発明される非効率の解消。bag による記録/再生は、この中で「実機の状況を後からオフラインで何度も再現する」ために生まれました。効果は明快で、ログさえあればアルゴリズムを実車なしで反復検証できます。限界は、ROS 1 bag が ROS 1 型に密結合で、長時間ログだと巨大化し可搬性が低いこと。
- ROS 2 への刷新(Macenski, Foote, Gerkey ほか, "Robot Operating System 2: Design, architecture, and uses in the wild", Science Robotics 2022)。DDS を通信層に据え、分散性・リアルタイム性・セキュリティを強化。記録形式も rosbag2 として再設計され、ストレージバックエンドを差し替え可能にしました(初期は SQLite/`.db3`、後に MCAP プラグイン)。論文はマルチホスト・多トピックでの実測スケーラビリティを示しています。
- MCAP の標準化(Foxglove, 2022, オープン仕様 mcap.dev)。シリアライズ非依存・追記耐性・自己記述スキーマを備えた汎用ロボットログコンテナ。動機は `.db3` の書き込み性能問題と ROS 専用であることの不便さの解消。現在は rosbag2 の推奨ストレージプラグインの1つになり、可視化ツール Foxglove Studio などと密に連携します。
- 派生としての公開データセット。Waymo Open Dataset(Sun ほか, CVPR 2020)、nuScenes(Caesar ほか, CVPR 2020)、Argoverse(Chang ほか, CVPR 2019)はいずれも実車の生ログを後処理し、学習しやすい形に整えて配布したもの。MCAP/rosbag は「その元になる生ログの容れ物」という位置づけで、これらデータセットの上流に位置します。生ログ → 整形 → 公開データセットという流れの最初の段にあるのがコンテナフォーマットです。

## 論文の実験結果(定量データ)

ROS bag / MCAP は学術論文の主題というよりエンジニアリング規格なので、評価は性能ベンチマークの形を取ります。値は仕様・コミュニティ報告に基づくおよその目安です。

- Macenski ほか(Science Robotics 2022)は ROS 2 の通信レイテンシやスループットを実測で示し、DDS ベースの設計が ROS 1 比でマルチホスト・大量トピック下でスケールすることを報告しています。記録系も、SQLite バックエンドから MCAP へ切り替えることで連続書き込みのスループットが改善することがコミュニティのベンチマークで報告されています(報告では同一ハード上で書き込み速度がおおむね数倍、CPU 使用率が低下、という桁感)。
- MCAP のチャンク圧縮(zstd)は、JPEG カメラ画像や点群の混在ログで、無圧縮比でおおむね2〜4倍のサイズ削減が一般に観測されます。圧縮率は中身のエントロピー次第で、すでに圧縮済みの JPEG が多いと効きにくく、密な点群やテキスト的メッセージ(数値の羅列)が多いとよく効きます。
- 部分読み出しの効果は定性的にも明らかで、インデックスを持つ MCAP では「全長1時間・50GB のログから特定の30秒のトピックだけ」を取り出すのに、ファイル全体を走査する必要がなくシーク一発で済みます。インデックスを持たない素朴な連結ログでは線形走査(50GB すべてを読む)が要るため、ここが桁違いに効きます。

数値の意味を補足します。書き込みスループットは「毎秒何 MB 記録できるか」で、多カメラ車両は毎秒数百 MB を生むため、ここが足りないとフレーム取りこぼし(データ欠損)が起きます。欠損は後の学習データの質に直結する(欠けたフレームでは時系列が不連続になる)ので、記録系の性能はそのまま研究の土台の信頼性になります。

## メリット・トレードオフ・限界

メリット
- 多センサー・多頻度のデータを時刻つきで1ファイルに束ね、走行をビット単位で完全再現できる。
- 再生(replay)できるので、同じ場面でアルゴリズムを何度でも、実車なしで検証できる。研究の反復速度を決定的に上げる。
- MCAP はシリアライズ非依存で、ROS 以外のシステムにも使える汎用性がある。自己記述スキーマにより将来のツールでも読める耐久性を持つ。

トレードオフ・限界
- ファイルが巨大になりやすい。多カメラ・LiDAR を長時間記録すると1走行で数十 GB〜数百 GB に達し、ストレージと転送がボトルネックになる。
- そのままでは機械学習に直接食わせにくい。学習用にはフレーム抽出・センサー間の時刻同期・教師信号生成といった後処理(整形)が必須で、生ログと学習用データセットの間には常に変換工程が挟まる。
- ROS 1 bag は ROS 1 型に密結合で可搬性が低く、MCAP への移行や型変換が要る場面がある。
- 未解決寄りの課題として、巨大ログのクラウドストレージ上での効率的な検索・サンプリング(ログ全体をダウンロードせずに必要部分だけ取り出す)や、複数ベンダー・複数フォーマットのログを横断的に扱う標準化が、実運用では依然として重い課題です。

## 発展トピック・研究の最前線

生ログから学習用データへ橋渡しする工程は、データセット構築の研究テーマそのものです。実ログの多様な入力形式(ベンダーごとに異なる時刻基準・座標系・センサー配置)を共通の学習形式へ取り込む「取り込みアダプタ(ingest adapter)」の抽象化や、ログを直接ストリーム読みして学習する(中間ファイルを作らない)パイプラインが研究・実務で広がっています。また、クラウド上の MCAP を部分読みしながら分散学習に供給する仕組みは、ペタバイト級のフリート(車両群)データを扱う上で重要度を増しています。可視化側では Foxglove Studio が MCAP を直接開いて時系列・3D・画像を同期再生でき、デバッグとアノテーションの基盤になっています。学習データの観点では、コンテナフォーマットを WebDataset のようなシャード形式へ変換する工程([WebDataset シャーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0850_webdataset-sharding))が、ストリーミング学習との接点になります。
