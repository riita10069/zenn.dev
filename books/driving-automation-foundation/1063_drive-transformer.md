# DriveTransformer (統一Transformer E2E)

## ひとことで言うと
DriveTransformer は、自動運転の知覚・予測・計画という処理を、タスクごとに違う専用モジュールを直列でつなぐのではなく、1種類の Transformer ブロックを大量に積み重ねた「統一(unified)」構造で実現する End-to-End 手法です。GPT が同じブロックを積むだけで巨大化できるのと同じ発想を運転に持ち込み、3つの設計原則(タスク並列・スパースクエリ・ストリーミング)で大規模化(スケール)しやすく、かつ学習を安定させたのが売りです。

## 直感的な理解
これまでの統合 E2E は、いわば専門部署を縦に並べた会社です。知覚部が報告を上げ終えてから予測部が動き、予測部が終わってから計画部が動く。各部署はそれぞれ独自の様式・独自の机(アーキテクチャ)を持っています。これは性能は出ますが、(1) 部署ごとに作りが違うので全体の改造・拡張が大変、(2) 前の部署が終わるまで次が待つので遅い、(3) 前段が間違えるとその誤りが後段に一方向に流れ込み修正できない(error propagation)、という問題を抱えます。

DriveTransformer の発想は「全部署を同じ作りの机にして、横に並べて同時に喋らせる」ことです。深層学習の経験則として、構造が均質でシンプルなほど(同じブロックの繰り返しほど)、データと計算を増やしたときに素直に性能が伸びる、というスケール則が知られています。言語モデルが Transformer ブロックの単純な積み重ねで巨大化に成功したように、運転でも「同じ Transformer ブロックを積むだけ」にすれば、スケールの恩恵を取りに行けるはずだ、というのが動機です。専用モジュールが入り組んだ従来構造は、各部品の相性合わせが難しく、大きくしようとすると学習が不安定になりがちでした。

## 基礎: 前提となる概念
- Transformer / attention: 系列やトークン集合を処理する基本構造。attention は「各要素が他のどの要素にどれだけ注目するか」を学習で重み付けし、関連の強い情報を集める仕組みです。self-attention は同じ集合内、cross-attention は別の集合(例: クエリ→センサ特徴)への参照です。計算量はトークン数 N に対して O(N²) で増えます。
- クエリ(query): 「何を知りたいか」を表すベクトル。物体検出クエリ=「この物体は何でどこ?」、地図クエリ=「車線はどこ?」、計画クエリ=「自車はどう進む?」のように、出力したい対象を最初から問いとして持たせます。DETR(Carion et al., ECCV 2020)が物体検出で広めた考え方で、後処理(NMS など)なしに集合を直接出力できるのが特徴です。
- 密 BEV(dense BEV)とスパースクエリ(sparse query): 密 BEV はシーン全体を格子(例 200×200=4万セル)で持ち、全セルを毎フレーム計算します。情報は豊富ですが重い。スパースクエリは「気になる対象」だけを数百個のクエリで持ち、必要な箇所だけセンサ特徴を参照します。軽いが、何もない自由空間の細部は表しにくい。
- ストリーミング処理(streaming / temporal propagation): 毎フレームを一から計算せず、前フレームの処理結果(クエリ=記憶)を引き継いで差分だけ更新する方式。時系列情報を安く使えます。動画を1コマずつ独立に解くのではなく、前コマの理解を持ち越すイメージです。
- スケール則(scaling laws, Kaplan, McCandlish et al., 2020): モデルサイズ・データ量・計算量を増やすと、テスト損失(test loss)がべき乗則(両対数グラフで直線)で滑らかに改善するという、言語モデルで定式化された経験則。均質で素直な構造ほどこの恩恵を受けやすいとされます。

## 仕組みを詳しく
DriveTransformer は全タスクをクエリの集合として扱い、同じ構造の Transformer ブロックの中で3種類の attention を回します。この3つが3つの設計原則に対応します。

クエリ集合の例: 物体検出クエリ群(数百)+ 地図クエリ群(数百)+ 自車計画クエリ(少数)。これらを1つの集合にまとめてブロックへ流します。

各ブロック内の3 attention:

1. タスク自己注意(task self-attention / タスク並列): クエリ同士を attention で関連付けます。計画クエリが検出クエリ・地図クエリを直接参照でき、知覚→計画の情報が「直列の段」ではなく「並列の参照」で流れます。前段の完了を待つ直列依存が減り、計算が並列化しやすくなります。さらに、後段の誤りが前段に逆流して修正をかけることもでき、一方向の error propagation が緩和されます。

2. センサ交差注意(sensor cross-attention / スパースクエリ): 各クエリがセンサ側の画像特徴を参照し情報を取り込みます。密 BEV 格子を全部作るのではなく、クエリが必要な箇所だけを attention で拾います。これが計算節約の肝です。

3. 時間交差注意(temporal cross-attention / ストリーミング): 過去フレームから引き継いだクエリ(記憶バンク)を参照します。全フレーム再計算ではなく差分更新で時系列を扱い、コストを抑えます。

この3 attention を備えた均質ブロックを N 段積み、最終的に物体・地図・自車軌跡を同時出力します。BEV を中間表現として明示的に持たずクエリ中心で処理するため、認識範囲・解像度を上げても密格子のようには計算が爆発しません。論文では、こうした統一構造が学習を安定させ、収束を速める点も強調されています(専用モジュールの寄せ集めだと各部の勾配バランスが崩れやすい)。

### 数値の感覚
密 BEV 系が「200×200=4万セルを毎段更新」するのに対し、クエリ系は「物体クエリ数百 + 地図クエリ数百 + 計画クエリ少数」程度、合わせて千個前後しか持ちません。attention の計算量はトークン数の2乗にほぼ比例するため、対象が4万から千に減ることは、空間に関わる計算量を桁で削れることを意味します(具体数は設定依存)。例えば 40000² と 1000² では 1600 倍の差です。これがスケール余地を生みます。同じ計算予算で、より大きなモデルや高解像度の認識を載せられるわけです。

## 手法の系譜と主要論文
- UniAD(Hu et al., "Planning-oriented Autonomous Driving", CVPR 2023, Best Paper)が統合 E2E の起点。知覚・予測・計画を1ネットワークに統合し計画に向けて全体最適化しましたが、タスクごとに専用モジュール(TrackFormer, MapFormer, MotionFormer, OccFormer, Planner)を直列接続し、密 BEV を使うため重く複雑でした。性能は当時最高水準でしたが「重さ・直列性・拡張のしにくさ」が課題として残りました。
- VAD(Jiang et al., "Vectorized Scene Representation for Efficient AD", ICCV 2023, [VAD (Vectorized Scene Representation)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1094_vad-vectorized-scene))が、シーンをベクトル(点列・ポリライン)で表現して密ラスタを避け、軽量化と高速化を実現しました。UniAD 比で数倍速いと報告。
- SparseDrive(Sun et al., 2024, [SparseDrive (スパース表現E2E)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1084_sparse-drive))が、密 BEV を排してスパースなインスタンス表現で対称的に検出・予測・計画を解く方向を推し進めました。
- DriveTransformer(Jia, Sun, Lu, Yang, Li et al., ICLR 2025, arXiv:2503.07656)は、これら「軽量・統一・スケール」志向を、タスク並列・スパースクエリ・ストリーミングの3原則と均質 Transformer ブロックという1つの構造原理に統合した到達点です。背景理論としてスケール則(Kaplan et al. 2020)を運転アーキテクチャ設計の指導原理に据え、「言語モデルが均質構造でスケールしたなら運転もそうすべき」という哲学を明示した点が新規性です。

## 論文の実験結果(定量データ)
評価は nuScenes(6 カメラ実車データの E2E 標準ベンチ、open-loop)を中心に、closed-loop 評価(Bench2Drive など CARLA 系の閉ループ)も併用されます。指標の意味から。

- 計画 L2 誤差(メートル): 予測自車軌道と人間実走の平均距離。1/2/3 秒先、小さいほど良い。
- 衝突率(%): 計画軌道が他物体とぶつかる割合。小さいほど良い。
- closed-loop の Driving Score / Success Rate: シミュレータで実際に走らせたときの総合運転スコアや完走率。open-loop の当てっこより実運転に近い評価で、高いほど良い。自分の行動が次の状況を生むため、誤りが累積するかどうかまで測れます。
- 学習効率・スループット: 同じ性能に達するまでの学習コストや推論速度(FPS)。

DriveTransformer は nuScenes の open-loop 計画指標で UniAD に匹敵または上回る平均 L2 と衝突率を、より単純な構造で達成したと報告しています。さらに closed-loop(Bench2Drive)で、密 BEV 系を上回る運転スコアを示し、学習効率の良さ(同性能までの学習が速く、収束が安定)とスケール性(モデル/データを増やすと性能が素直に伸びる)を主張します。要点は、性能で同等以上を保ちつつ「単純さ・効率・スケール余地」で勝る、という構図です。論文では、密 BEV を作らないことによる速度向上と、統一構造による収束安定性が定量的に示されています。

アブレーションの方向は次の通りです。(1) 3 attention のどれを外しても劣化し、とくにタスク自己注意(タスク並列)を抜くと知覚→計画の情報流が細りスコアが落ちる。(2) ストリーミング(時間交差注意)を抜くと時系列が使えず予測・計画が悪化、とくに速度推定や追跡が崩れる。(3) スパースクエリを密 BEV に戻すと計算が増えスケール性が損なわれる。(4) 統一構造を専用モジュールの直列に戻すと学習が不安定化・低速化する。3原則が相補的に効くことが示されます。なお比較的新しい手法のため、多様な実車・長期運用での検証はこれからです。

## メリット・トレードオフ・限界
メリット
- 均質 Transformer ブロックの積み重ねで設計が単純。理解・改造・スケールアップが容易で、学習も安定しやすい。
- スパースクエリで密 BEV を避け、計算・メモリを節約しつつ認識範囲・解像度を上げられる。
- タスク並列で直列依存・一方向の誤り伝播が減り、ストリーミングで時系列を安く扱える。スケール則の恩恵を取りに行ける。

トレードオフ・限界
- クエリ中心設計は学習の安定化やハイパーパラメータ調整に工夫が要り、初期化やクエリ数の設計が性能に効く。クエリが少なすぎると見逃しが増え、多すぎると無駄が増える。
- 密 BEV が得意な稠密空間情報(自由空間の細かい形状、走行可能領域の境界、占有グリッドなど)を表しにくい場面がある。クエリは「物体」を捉えるのは得意だが「何もない空間の形」は苦手。
- 比較的新しく、実車・多様データ・長期運用での検証が不足。closed-loop 評価も特定シミュレータ依存で sim-to-real ギャップが残る。
- 未解決課題として、クエリ表現の解釈性(なぜその計画になったかの説明)、スケール則が運転タスクでどこまで本当に成立するか(言語ほどデータが無限ではなく、ライセンス・収集・アノテーションのコストが厳しい)が挙げられます。

## 発展トピック・研究の最前線
- スケール則の運転への適用検証: 運転データはライセンス・収集コストの制約が強く、言語のように無尽蔵ではありません。データ・モデルを増やしたとき本当にべき乗則で伸びるか、どこで頭打ちになるかの実証が進行中です。これは E2E 運転研究の中心的な問いの一つです。
- 統一トークン化のさらなる徹底: 知覚・予測・計画に加え、占有・地図・信号・自然言語指示まで全部クエリ/トークンで扱う統一化の流れ。VLA(Vision-Language-Action)モデルとの接続も視野に入ります。
- 世界モデル・生成との融合: 未来観測を生成する world model や生成的計画([GenAD (生成的軌道計画E2E)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1070_genad_note))を、統一 Transformer に組み込む方向。
- 実車リアルタイム化: スパース化で軽くなったとはいえ車載 GPU 制約は厳しく、量子化・蒸留・KV キャッシュ等の最適化が研究テーマです。
