# LaneGCN

## ひとことで言うと
車線を「点列」ではなく「つながりの情報を持つグラフ(網の目)」として表し、車線どうし・車と車線のあいだで情報をやりとりさせることで、周りの車が次にどう動くかを予測しやすくする地図エンコーダです。道路には「この車線の先はあの車線」「隣はこの車線」という決まった接続があり、それを正面から使うのが最大の特徴です。

## 直感的な理解
自動運転で周囲の車の未来の動きを予測するには、その車が今どの車線にいて、その車線がこの先どこへつながっているかを知ることが決定的に重要です。右折専用レーンにいる車は右折する確率が高い。直進レーンにいて前方で車線が分岐しているなら、どちらへ行くかを分岐構造を踏まえて考える必要があります。逆に言えば、車線の接続構造を知らずに「過去の動きの延長線」だけで予測すると、交差点やカーブで簡単に外れます。等速直線運動の外挿では曲がり角を曲がれません。

人間のドライバーは、前の車が右折レーンに入ったのを見ただけで「あ、右折するな」と分かります。これは道路の構造(右折レーンの存在、その先のつながり)を頭の中で把握しているからです。LaneGCN がやろうとしているのは、この「道路構造を踏まえて他車の意図を読む」という人間の推論を、車線のつながりをグラフとして明示的にネットワークへ与えることで実現することです。

[ポリラインエンコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0460_polyline-encoding)(VectorNet)は車線を点の列として表し、ポリライン全体を1本のベクトルに要約しました。これは強力ですが、車線の「接続関係」を陽には持っていません。VectorNet では全要素を attention で相互作用させますが、「この車線の直接の次はこの車線」という道路特有のトポロジー(つながり方の構造)をネットワークに明示的に教えてはいません。attention は「全部の要素を見て関係を学習で見つけてくれ」という方針なのに対し、LaneGCN の問題意識は「道路の接続構造はすでに分かっているのだから、それをグラフとして直接与えてその上で計算すれば、もっと効率よく自然に予測できるのでは」というものです。既知の構造を学習に任せず、人手で与える(帰納バイアスを入れる)発想です。

## 基礎: 前提となる概念
理解の土台を噛み砕きます。

グラフとは「ノード(点)」と「エッジ(ノードどうしをつなぐ線)」から成るデータ構造です。SNS の友達関係や路線図のように、「何と何がつながっているか」を表すのに使います。LaneGCN では車線の小区間がノード、車線のつながり方がエッジになります。エッジには種類(前後・左右など)を持たせられます。

GCN(Graph Convolutional Network、グラフ畳み込みネットワーク)とは、グラフの上で各ノードが「自分の隣のノードから情報を集めて自分を更新する」操作(message passing、メッセージ伝播)を繰り返すネットワークです。画像の CNN が「隣の画素から情報を集める」のに対し、GCN は「グラフ上で隣接するノードから情報を集める」と考えると分かりやすい。この操作を複数回(層を重ねて)繰り返すと、1ホップ先→2ホップ先と、遠くのノードの情報まで間接的に伝わってきます。

アクター(actor / agent)とは、予測対象である周囲の交通参加者、つまり他車・歩行者・自転車などのことです。各アクターは過去の数秒間の軌跡(時刻ごとの位置)を持ち、これを 1D 畳み込み(時間方向の CNN)などで特徴ベクトルに変換します。

軌跡予測(motion forecasting / trajectory prediction)とは、各アクターの未来数秒の位置の列を予測する課題です。未来は一つに決まらない(右折も直進もありうる)ため、通常は複数の候補軌跡と、それぞれの確率を出します。これを multi-modal(多峰性、複数の可能性=モードを持つ)予測と呼びます。1本だけ出すと「右折と直進の平均=車線をはみ出す非現実的な軌跡」になりがちで、複数モードを出すことが本質的に重要です。

## 仕組みを詳しく
LaneGCN は大きく「車線グラフを作る」「車線グラフ上で畳み込む」「車と車線を相互作用させる」「軌跡を予測する」の4段から成ります。

ステップ1: 車線グラフを構築する。
各車線(センターライン=車線中心線)を短い区間に区切り、それぞれを「車線ノード(lane node)」とします。道路全体が例えば500個の車線ノードになるとします。各ノードには位置と向き(その区間の座標と方向)の特徴を持たせます。重要なのは、ノードどうしを4種類の接続(エッジ)で結ぶことです。
- successor(後継): このノードの直後に続くノード(前方、進める先)
- predecessor(先行): 直前のノード(後方)
- left(左): 左隣の車線のノード(車線変更できる先)
- right(右): 右隣の車線のノード

```
   左車線  ─right─  自車線ノード  ─left─  右車線
                       │  successor(前方)
                       ▼
                    次のノード
                       ▲
                       │  predecessor(後方)
                    前のノード
```

この4種類で「進める方向」「車線変更できる方向」という道路構造を陽に表します。普通の格子画像には無いこの接続情報こそ LaneGCN の財産です。

ステップ2: 車線グラフ上でグラフ畳み込み(LaneConv)。
GCN を上の4種類のエッジそれぞれに対して行います。各車線ノードは、successor 方向・predecessor 方向・left・right の隣ノードから特徴を集めて自分を更新します。これを複数回繰り返すと、各ノードの特徴に「自分の遠く先・遠く後ろ・隣の車線の状況」までが染み込みます。ポイントは、接続の種類ごとに別々の重み行列を使うことです。前方からの情報と隣からの情報を区別して扱えるので、「この先で合流する」「隣に移れる」といった構造を反映できます。さらに dilated(拡張)接続、すなわち2ノード先・4ノード先・……と直接つなぐエッジも入れ、長い車線の遠い先まで少ない層数で情報を届けます(畳み込みの dilation と同じ発想で、計算を増やさず受容野=情報を集める範囲を広げる)。これにより、高速道路のように1本の車線が長く伸びる場面でも遠方の構造が効率よく伝わります。

ステップ3: 車(アクター)と車線を相互作用させる(4種の融合、FusionNet)。
各アクターの過去軌跡を 1D 畳み込みで特徴ベクトルにした上で、次の4方向の情報のやりとりを行います。
- Actor → Lane(A2L): 各アクターの状況を近くの車線ノードに伝える(「この車線にこういう速度の車がいる」)
- Lane → Lane(L2L): 車線グラフ上でさらに情報を回す(上記 LaneConv)
- Lane → Actor(L2A): 車線の状況をアクターに伝える(「お前がいる車線の先はこうつながっている」)
- Actor → Actor(A2A): アクターどうしの相互作用(「前の車が減速したら自分も」)

この4つを通すと、各アクターの特徴は「自分がどの車線にいて、その車線がどうつながり、周りの車がどう動いているか」を全部含んだものになります。A2L/L2A は空間的な近さ(アクターと車線ノードの距離)で attention を張り、A2A もアクター間の距離を考慮します。距離に応じて重みを付ける spatial attention です。

ステップ4: 軌跡予測。
仕上げられたアクター特徴から、未来の軌跡(複数候補と各候補の確率)を出力します。出力は回帰ヘッド(座標列を出す)と分類ヘッド(各候補の確率を出す)に分かれ、車線構造を踏まえているので「車線に沿った」現実的な予測が出やすくなります。

具体例で追います。交差点手前で右折レーンにいる車を考えます。LaneGCN では、その車のいる車線ノードが「successor として右折先の車線につながり、left として直進レーンに接する」構造を持ちます。L2A の融合でこの構造が車の特徴に入るので、ネットワークは「右折する未来」を高い確率で予測しつつ「直進レーンに移って直進する未来」も候補に残せます。格子画像ベースより、道路の選択肢を構造として正しく扱えるわけです。

## 手法の系譜と主要論文
軌跡予測の地図エンコードという流れの中に LaneGCN を置きます。

前提として Argoverse(Chang, Lambert, Sangkloy, Singh, Bak, Hartnett, et al., CVPR 2019, "Argoverse: 3D Tracking and Forecasting with Rich Maps", arXiv:1911.02620)が、車線中心線とその接続を持つ HD 地図つきの軌跡予測ベンチマークを公開し、「地図構造を予測に使う」研究を可能にしました。過去2秒(20フレーム)から未来3秒(30フレーム)を予測する設定で、これが LaneGCN の評価基盤です。

LaneGCN(Liang, Yang, Hu, Chen, Casas, Urtasun, ECCV 2020, "Learning Lane Graph Representations for Motion Forecasting", arXiv:2007.13732, Uber ATG/Waabi の Urtasun らによる)が本体です。車線を4種接続のグラフで表し、LaneConv でグラフ畳み込み、A2L/L2L/L2A/A2A の4融合(FusionNet)でアクターと車線を相互作用させて軌跡予測する枠組みを提案しました。動機は、道路の接続トポロジーを陽に使えば車線に沿った現実的な予測ができる、ラスタ画像では構造が失われる、という点です。

同時期の VectorNet(Gao et al., CVPR 2020)は、地図と軌跡をポリラインで統一して attention で相互作用させました。LaneGCN が「接続を陽に持つグラフ畳み込み」なのに対し、VectorNet は「順序不変の要約 + attention」で、設計思想の対比として重要です。詳細は [ポリラインエンコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0460_polyline-encoding) を参照してください。

以後、車線グラフの考え方は LaneRCNN(Zeng, Liang, Hu, Urtasun, IROS 2021, "LaneRCNN: Distributed Representations for Graph-Centric Motion Forecasting"、各アクターに車線グラフ表現をひも付けて分散表現で扱う)、TNT / DenseTNT(Zhao et al., CoRL 2020 / Gu et al., ICCV 2021、車線上の目標点を予測する)、そして近年の Transformer ベース予測(車線をトークンとして扱う系統。SceneTransformer, mmTransformer, HiVT, QCNet など。LaneGCN 的なグラフ事前知識を attention のマスクや相対位置エンコードに織り込む)へと受け継がれています。現在の主流は「グラフ構造の明示」と「Transformer の柔軟な attention」を融合する方向で、LaneGCN はその「構造を明示する」極の原典として位置づけられます。

## 論文の実験結果(定量データ)
LaneGCN は Argoverse Motion Forecasting ベンチマークで評価されました。過去2秒の軌跡から未来3秒(0.1秒刻みで30点)を予測する課題です。主要指標は次の通りで、いずれも小さいほど良い。
- minADE(minimum Average Displacement Error): 複数候補のうち最良の候補について、全時刻の位置誤差の平均(メートル)。「出した K 本のうち一番当たった1本」で測るので、多峰予測の質を見る。
- minFDE(minimum Final Displacement Error): 最良候補の最終時刻(3秒後)の位置誤差(メートル)。最終地点がどれだけ当たったか。
- MR(Miss Rate): 最終位置が正解から一定距離(通常2m)以上外れた候補しか出せなかった割合。低いほど「少なくとも1本は当てた」場面が多い。

報告によれば、LaneGCN は当時(2020年)の Argoverse リーダーボードで上位の成績を収め、minFDE などでラスタ地図ベースや車線構造を使わない手法を明確に上回りました。多峰予測(K=6 候補)設定で、minADE がおよそ 0.71m、minFDE がおよそ 1.08m、MR がおよそ 0.16 前後の水準に達したと報告されており、当時の最先端を更新しました。minFDE が約1mということは、3秒後の予測地点が平均してほぼ1m以内に収まることを意味し、車線幅(約3.5m)を踏まえれば「正しい車線・正しい挙動」を当てられている水準です。

アブレーションが示唆に富みます。第一に、4種類のエッジ(successor/predecessor/left/right)を区別せず単一の接続にまとめると精度が落ち、接続種別ごとに重みを分けることが効いていると分かりました。前方への伝播と横方向(車線変更)への伝播は意味が違うので、混ぜると損をします。第二に、A2L/L2L/L2A/A2A の4融合のいずれを抜いても性能が下がり、特に L2A(車線→アクター、車線構造を車に伝える経路)を抜くと顕著に悪化しました。これは「車に車線構造を伝える経路」が予測精度の核心であることを定量的に示しています。第三に、dilated 接続を入れると長距離の車線情報が効率よく伝わり、特に高速・長距離移動のシーンで誤差が改善したと報告されています。これらは「道路の接続構造を陽にモデル化することが、予測精度に直接効く」という主張を裏づけます。

## メリット・トレードオフ・限界
メリットを整理します。道路の接続トポロジー(前後・左右)を陽に扱えるので、車線に沿った現実的な予測ができます。A2L/L2L/L2A/A2A の4融合で、車線と車の相互作用を構造的に捉えられます。車線が前後左右の接続を持つ標準的なベクトル地図形式(Argoverse map や Lanelet2 など)と素直に対応するため、既存のベクトル地図資産を活かしやすい。構造を人手で与える分、学習データが少なくても妥当な予測が出やすい(帰納バイアスがデータ効率を上げる)。

トレードオフと限界も率直に挙げます。接続関係を持つベクトル地図(どの車線がどこへつながるか)を事前に用意する必要があり、生センサーから直接は得られません。グラフ構築と4融合でパイプラインがやや複雑になり、実装と学習の手間が増えます。地図の接続が間違っていると、予測も誤った構造に引きずられます(古い地図で消えた車線が残っていると、そこへ進む候補を誤って出す)。また、グラフ畳み込みは固定の接続に沿って情報を流すため、地図に載っていない動き(ショートカット、車線無視の割り込み、緊急回避)の表現は苦手です。グラフという離散構造ゆえ、構造が壊れた(地図が不完全な)場面で脆くなります。

未解決の課題としては、地図誤差に頑健な予測、地図に縛られすぎない柔軟性との両立、車線形状だけでなく信号や標識・横断歩道といった他の地図要素との統合、そしてグラフ事前知識を大規模 Transformer にどう最も効率よく注入するか、などが研究の焦点です。

## 発展トピック・研究の最前線
LaneGCN 以降、車線グラフの考え方は Transformer ベースの予測・計画モデルに吸収されつつあります。車線ノードをトークンとして扱い、接続関係を attention のバイアス(つながっているノードどうしを注目しやすくする)や相対位置エンコードとして与えることで、グラフ畳み込みの「構造の明示」と Transformer の「柔軟な長距離相互作用」を両立する手法(HiVT, QCNet, GoRela など)が主流です。query-centric な座標系の取り方と組み合わせると、シーンの並進・回転に不変な効率的予測ができます。

また、知覚から予測・計画までを一つのネットワークで end-to-end に解く統合モデル(UniAD, VAD など)でも、内部の地図表現として車線グラフ的な接続情報が使われています。地図側では、オフラインの整備済み地図に頼らず、オンライン地図構築([InstaGraM / インスタンスマスク地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0450_maplt-tr-instance-mask))で推定した車線とその接続を直接予測に流す研究(地図と予測の結合、map-free / online-map prediction)が進んでおり、LaneGCN の「接続を使う」発想は地図の作り方が変わっても生き続けています。推定地図の不確実性を予測に伝える(誤った車線を信じすぎない)研究も活発で、これは LaneGCN の限界として挙げた「地図誤差への脆さ」を直接埋める方向です。
