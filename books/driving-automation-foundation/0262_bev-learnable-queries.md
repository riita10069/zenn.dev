# 学習可能BEVクエリ

## ひとことで言うと
BEV(Bird's-Eye-View、鳥瞰図、車を中心に真上から見下ろした地図)の各マス目に「最初は空っぽの問い合わせ票(クエリ)」を1枚ずつ用意し、それを学習で育てる手法です。各マスのクエリが「自分の場所には複数カメラ画像のどこを見ればよいか」を学習し、クロスアテンション(後述)でカメラ特徴を取りに行ってマスを埋めます。最大の利点は、地図の解像度を、画像から特徴を抜くネットワーク(バックボーン)の出力サイズから完全に切り離せることです。地図を細かくしたければクエリを増やすだけで済みます。

## 直感的な理解

自動運転の知覚では、複数のカメラが撮った画像を「車を中心に置いた1枚の俯瞰地図」に統合したいという要求が常にあります。なぜ俯瞰地図(BEV)が欲しいかというと、運転の意思決定は本質的に「平面上のどこに何があるか」の問題だからです。前の車との車間、車線の位置、歩行者の立ち位置 ― これらはすべて地面に投影した2次元平面で考えるのが自然で、計画(プランニング)モジュールもこの平面表現を入力にすると素直に書けます。さらに、6台のカメラがそれぞれ別々の座標系で物体を見ているのを、1つの共通座標系(自車中心のBEV)に統合できれば、「右前カメラに写った車」と「正面カメラに写った車」が同一物体かどうかも自然に判定できます。

問題は「カメラ画像(遠近法で歪んだ斜め視点)」から「俯瞰地図(真上視点)」への変換です。素朴な発想は「バックボーンが出した画像特徴マップを、幾何学計算でそのまま地面に貼り付ける」ことですが、これだと地図のマス目の細かさがバックボーンの出力解像度に縛られてしまいます。例えばバックボーンの最終出力が7×7の特徴マップなら、地図も7×7にしかなりません。7×7のマスでは、隣り合う2台の車を別々のマスに置くことすらできず、知覚として使い物になりません。

ここで発想を逆転させます。「バックボーンの出力をBEVに変換する」のではなく、「先にBEVグリッド(例: 200×200マス)を自分で決めて、各マスが必要な情報をカメラ画像から能動的に取りに行く」設計にします。このとき各マスに持たせる「情報を取りに行くための問い合わせ票」が「BEVクエリ(BEV query)」です。地図の細かさはクエリの個数で決まり、それは設計者が自由に決められます。バックボーンが何であろうと、入力画像が何ピクセルであろうと、地図は望みの解像度を保てます。これは「データを押し出す(push)」発想から「欲しい側が引き寄せる(pull)」発想への転換だと言えます。

## 基礎: 前提となる概念

理解に必要な用語をひとつずつ噛み砕きます。

「特徴マップ(feature map)」とは、ニューラルネットワークが画像を処理して出す中間出力で、各位置に「その位置の画像が持つ意味」を数百次元のベクトルで表したものです。例えば[256, 7, 7]という特徴マップは、縦7×横7=49個の位置それぞれに256次元のベクトルがある、という意味です。バックボーン(画像を入力に受け取り特徴マップを出す中核ネットワーク、ResNet や Swin Transformer など)はこの特徴マップを出力します。

「クエリ(query)」とは、アテンション機構(後述)で「私はこういう情報が欲しい」と問い合わせる側のベクトルです。日本語で「問い合わせ票」と言うと直感的です。アテンションでは、クエリが多数の「候補(キーとバリュー)」の中から関連するものを選んで情報を集めます。

「学習可能(learnable)」とは、その値を人が手で決めるのではなく、ニューラルネットワークのパラメータとして持ち、誤差逆伝播(バックプロパゲーション、出力の誤差を入力側へ伝えて各重みをどう動かせば誤差が減るか計算する手続き)による最適化で自動的に良い値に育てる、という意味です。実装上は `nn.Parameter` や `nn.Embedding` という、勾配が流れて更新される重みテーブルとして持ちます。最初はランダムな数値ですが、学習が進むと各クエリが意味のある役割(「私は前方路面担当」「私は左隣車線担当」)を勝手に獲得します。

「クロスアテンション(cross-attention)」とは、別々の2つのデータの間で行うアテンションです。ここでは「BEVクエリ(欲しい側)」と「カメラ画像特徴(供給する側)」の間で行います。各BEVクエリが、カメラ特徴のどこをどれだけ重視するかを計算し、重み付き平均で情報を吸い上げます。これに対し、同じデータ同士で行うのがセルフアテンション(self-attention)で、BEVクエリ同士が互いを参照して空間的な文脈をならす際に使います。

「カメラ校正(camera calibration)」も鍵です。各カメラには、車のどこにどの向きで付いているか(外部パラメータ、回転行列と並進ベクトル)と、レンズの焦点距離・主点・歪み(内部パラメータ、行列K)があります。これらが分かっていれば、3次元の実世界座標を「このカメラのこの画素」へ投影する計算ができます。BEVクエリ方式は、この投影でクエリの「だいたいの見るべき位置(参照点)」を決めます。

## 仕組みを詳しく

学習可能BEVクエリは、典型的には次のように定義されます(BEVの縦をbev_h、横をbev_w、特徴次元をembed_dimとします)。

```python
# 例: bev_h=200, bev_w=200, embed_dim=256
self.bev_queries = nn.Parameter(torch.randn(bev_h * bev_w, embed_dim))
# = [40000, 256] の学習可能テーブル。等価に nn.Embedding(40000, 256) でも書ける
```

これは「40000個(=200×200マス)それぞれに256次元のベクトルを1本ずつ持つ、学習可能な表」です。合計 40000 × 256 ≈ 1024万個のパラメータが学習対象になります(解像度を450×300に上げると 135000 × 256 ≈ 3456万個に膨らむ)。

処理の流れをステップで追います。テンソルの形(shape)も併記します。

1. クエリを取り出してバッチ分に複製する。
   ```python
   queries = self.bev_queries.unsqueeze(0).expand(B, -1, -1)
   # [B, 40000, 256]  B はバッチサイズ
   ```
   この時点ではまだ「空っぽの問い合わせ票」で、画像の中身は入っていません。各クエリには別途、自分がBEVグリッドのどの位置を担当するかを示す位置エンコーディング(positional encoding)が足されます。これがないと全クエリが区別できず、すべて同じ場所を見てしまいます。

2. 各BEVマスに対応する3D実世界座標を、カメラの校正情報を使って各カメラの画像平面に投影する。これにより「このマスはこのカメラのこのあたりに写っているはず」という基準位置(参照点)が得られます。深度方向の曖昧さ(地図上の1点は、地面の上の柱のように高さ方向に幅を持つ)は、地面から数段の高さ(ピラー、pillar)に点を散らすことで吸収します。例えば各BEVマスにつき高さ方向に4点を取り、それぞれを投影すると、参照点は[B, 40000, 4, 2]のような形になります。

3. 参照点の周りの画像特徴を、クエリが計算したアテンション重みで集める。フルアテンション(全画素を見る)では計算量が爆発するため、参照点周りの数点だけをサンプリングする変形アテンション([Deformable Cross-Attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0259_deformable-cross-attention))を使うのが標準です。あるBEVマスがどのカメラにも写っていなければ、そのカメラは寄与しません。

4. 取ってきた特徴を重みで合算し、そのマスが複数カメラに写っている場合は可視カメラ数で正規化して、マスの値を確定する。

5. 残差接続(residual、入力をそのまま足し戻して勾配を流れやすくする)・LayerNorm(層ごとの正規化)・FFN(Feed-Forward Network、各位置に独立に効く小さな2層MLP)を通し、これを複数層(BEVFormer では6層)積む。最後にBEVグリッドの形に並べ直す。
   ```python
   bev_features = output.reshape(B, 200, 200, 256).permute(0, 3, 1, 2)
   # [B, 256, 200, 200]
   ```

決定的に重要なのは、クエリの個数(40000)とバックボーンの出力サイズ(7×7など)が完全に独立している点です。

before / after で言うと、

- before(素朴な貼り付け): BEVサイズ = バックボーン出力サイズ(7×7に固定)。地図がバックボーンの都合に縛られる。
- after(学習可能クエリ): BEVサイズ = クエリの個数(設計者が自由に200×200)。バックボーンとは無関係。

なぜ「学習可能」にするのかという設計上の理由も押さえておきます。各マスがカメラのどこをどう見るべきかは、カメラ配置・レンズ歪み・地面の傾き・物体の高さ分布など多くの要因に依存し、解析的に最適な見方を書き下すのは困難です。これをデータから学習させることで、人手では設計しきれない見方(状況依存の注意配分)を獲得できます。さらに、過去フレームのBEVを今のBEVクエリが参照する「時間セルフアテンション」を足せば、1枚の静止画では分からない速度・加速度の情報も取り込めます。

## 手法の系譜と主要論文

学習可能クエリの考え方は、まず物体検出で生まれ、次にBEV知覚へ展開されました。

DETR(Carion et al., ECCV 2020, arXiv:2005.12872)が起点です。物体検出を「学習可能なオブジェクトクエリ100個が、それぞれ1個の物体を担当して画像特徴に注意を向け、box とクラスを出す」という集合予測問題に定式化しました。手作業のアンカーやNMS(非最大抑制、重複検出を消す後処理)を排した点が革命的でした。ただしフルアテンションゆえ収束が極端に遅く(500エポック規模)、高解像度特徴を扱えないという弱点がありました。

Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159)がその遅さを解決しました。「全位置を見るのではなく、各クエリの参照点周りの少数点(典型的に4点)だけサンプリングする」変形アテンションを導入し、計算量を線形化、収束を約10倍速くしました。BEVクエリのクロスアテンションはこの仕組みを直接受け継いでいます([Deformable Cross-Attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0259_deformable-cross-attention))。

BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)が、この学習可能クエリをBEV知覚に持ち込んだ代表的提案です。グリッド状に並べた学習可能BEVクエリが、空間クロスアテンション(Spatial Cross-Attention、各BEVクエリが対応する3D点を各カメラに投影し周辺をサンプリング)で複数カメラ特徴を集め、時間セルフアテンション(Temporal Self-Attention、過去のBEVを参照)で時系列情報を取り込みます。当時、もう一方の主流だった「カメラごとに深度を推定して点群を作る」方式(LSS系、下記)は深度推定の誤差に弱かったため、深度を明示的に予測せずクエリが自分で「どこを見るか」を学習する方式に切り替えた、というのが動機です。

対照的な系譜として LSS(Lift-Splat-Shoot、Philion & Fidler, ECCV 2020, arXiv:2008.05711)があります。これは各ピクセルに離散的な深度分布を予測し、特徴を3D空間に「持ち上げて(lift)」BEVに「叩きつける(splat)」方式です。学習可能クエリ方式とは逆方向(ボトムアップに画像から地図へ)で、深度予測の質に性能が左右されます。BEVDet/BEVDepth など多くの後継があり、現在もクエリ方式と二大潮流をなしています。なお BEVDepth は LiDAR を弱教師として深度予測を明示的に監督することで、純カメラながら大きく精度を伸ばしました。

StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926)など PETR 系は、密なBEVグリッドを作らず、3D位置エンコーディングを画像特徴に加えてオブジェクトクエリが直接マルチビューを参照する「スパースクエリ」方向で、グリッド型BEVと並ぶ潮流を形成しています。

UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)は、BEVFormer の学習可能BEVクエリをそのままBEVエンコーダに使い、その上に検出・追跡・地図・動き予測・占有予測・計画をすべてクエリ伝播でつなぐ end-to-end 構成を作りました。学習可能クエリが知覚から計画までを貫く「共通言語」として機能する好例です。

## 論文の実験結果(定量データ)

BEVFormer の評価は主に nuScenes データセット(米ボストン・シンガポールの都市走行、6カメラ360度 + LiDAR、1000シーン)で行われました。指標は主に2つです。

NDS(nuScenes Detection Score)は検出の総合スコアで、0〜1の範囲、高いほど良い。物体の位置・大きさ・向き・速度・属性の誤差を総合した指標です。mAP(mean Average Precision)は中心距離しきい値(0.5/1/2/4m)で測った平均適合率で、こちらも高いほど良い。

報告によると、BEVFormer はカメラのみ(LiDARなし)で NDS 0.517、mAP 0.416 を達成しました(ResNet-101-DCN バックボーン)。これは当時のカメラ専用手法(例えば DETR3D の NDS 0.42 級)を大きく上回りました。アブレーション(ある要素を抜いて効果を測る実験)では、時間セルフアテンション(過去BEV参照)を抜くと NDS がおよそ2〜3ポイント下がり、特に速度推定の誤差指標 mAVE が顕著に悪化することが示されています。これは「1フレームだけでは速度が分からないが、過去BEVと付き合わせれば分かる」という直感と一致します。

空間クロスアテンションについてのアブレーションでは、全カメラ・全位置を見るフルアテンションと、参照点周りだけ見る変形アテンションを比べると、精度はほぼ同等のまま計算量とメモリが大幅に削減できることが報告されています。つまり「全部見なくても、関係ある数点だけ見れば十分」という変形アテンションの前提が、BEV知覚でも成り立つことが実証されています。

UniAD は CVPR 2023 のベストペーパーを受賞しました。各サブタスク(検出・追跡・地図・予測・計画)を個別に最適化した従来パイプラインに対し、共同学習することで計画性能を改善しました。報告では、3秒先の平均L2誤差(予測軌跡と正解の距離)でおよそ1.6m級、平均衝突率を従来比でおよそ半減させ、いずれも当時のモジュール分離型ベースラインを上回りました。

## メリット・トレードオフ・限界

メリット
- BEV解像度をバックボーン出力から切り離し、設計者が自由に決められる(7×7の呪縛からの解放)。
- クエリが「どこを見るか」を学習で獲得するので、別途の深度推定を必須としない(LSS系の深度誤差問題を回避)。
- 複数カメラを自然に統合でき、時間方向(過去BEV参照)にも素直に拡張できる。
- 下流の検出・予測・計画まで同じクエリ表現で貫けるため、end-to-end 学習と相性が良い。

トレードオフ・限界
- クエリ数が多い(数万)とパラメータとアテンション計算が増える。変形アテンションでの線形化がほぼ必須。
- 学習可能クエリは初期がランダムなので、収束が遅く学習が不安定になりがち。十分なデータと学習時間を要する。画像バックボーンへ十分な勾配が流れにくいという「勾配が薄まる」課題もある(BEVFormer v2 はこれを補助監督で緩和)。
- 性能がカメラ校正の精度に依存する。校正がずれると参照点がずれ、見る場所そのものが間違う。
- 深度を明示しないため、遠方の小物体や激しい遮蔽下では依然として難しい。LSS系とのハイブリッドや、深度を弱教師として併用する研究が続いている。

未解決の研究課題としては、クエリ初期化の良し悪し(良い初期値で収束を速められないか)、動的シーンでの時間融合の最適な設計、そして異なるカメラ構成・車種間での転移(あるセンサ配置で学んだクエリが別配置で使えるか)などが挙げられます。

## 発展トピック・研究の最前線

BEVFormer v2(Yang et al., CVPR 2023, arXiv:2211.10439)は、BEVクエリに加えて画像側の検出ヘッド(perspective supervision、画像視点での補助監督)を併用し、バックボーンの学習を助けることで精度を伸ばしました。学習可能クエリ単独では画像バックボーンに十分な勾配が流れにくい、という課題への対処です。

近年は、占有グリッド予測(Occupancy Network)へのクエリ拡張も盛んです。BEVは平面ですが、3Dボクセルごとに「占有/空き/意味クラス」を予測する占有ネットワーク(例: TPVFormer, SurroundOcc)では、ボクセル単位の学習可能クエリが用いられます。これにより、明示的なカテゴリに当てはまらない未知障害物(落下物など)も「占有されている」として扱えるようになり、安全性で注目されています。

また、世界モデル(world model)や拡散モデルと組み合わせ、BEVクエリ表現の上で未来のシーンを生成・予測する研究(GAIA系、DriveDreamer系)も進んでいます。学習可能クエリは、こうした生成・予測タスクの土台となる「シーンの圧縮表現」としての役割も担いつつあります。学習可能クエリは、もはや知覚だけの部品ではなく、知覚・予測・計画・生成をつなぐ普遍的なインタフェースになりつつある、というのが現在の見取り図です。
