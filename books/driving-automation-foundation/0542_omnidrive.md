# OmniDrive

## ひとことで言うと
OmniDrive は、車に取り付けた複数カメラの映像から 3D の立体的な空間を理解し、その上で「もし右に車線変更していたら危なかったか?」のような "もしも(if)" を問う反事実的推論(counterfactual reasoning)まで含めて、知覚(見る)・推論(考える)・計画(走る道を決める)を一気通貫でつなぐ視覚言語モデル(VLM)エージェントと、その能力を測る専用ベンチマーク(評価用のデータと指標のセット)を提案した研究です。

## 直感的な理解
熟練ドライバーが交差点で右車線へ移ろうとする瞬間、頭の中では無意識に「もし今ここで移ったら、右後ろから来ているあの車とぶつかるかもしれない」という "もしも" のシミュレーションが走っています。実際には起こさなかった行動の結果を想像して、危険なら思いとどまる。この「取らなかった選択肢の結末を想像する力」が反事実的推論であり、安全運転の核心です。

ところが運転に LLM や VLM を使う研究の多くは、この力を持ちませんでした。さらに、画像を平面(2D)のまま扱うため、運転に本質的な「物体が自分から何メートル先のどの高さにあるか」という 3D の立体把握が弱かった。たとえば「左のカメラに車が映っている」とは言えても「その車が自車の左 3m・前方 10m を時速 50km で接近している」という幾何的に正確な理解は不得意だったのです。OmniDrive は「3D を本気で理解し、反事実的推論で行動の安全性を吟味し、最終的に計画まで結ぶ」VLM エージェントを作り、加えて「そういう深い能力を公平に測るものさし(ベンチマーク)」まで整備した、というのが要点です。

## 基礎: 前提となる概念
理解に必要な概念を順に噛み砕きます。

VLM(Vision-Language Model, 視覚言語モデル)とは。画像と言葉を両方入力でき、画像について言葉で質問に答えたり説明したりできるモデルです。画像を特徴ベクトル列に変換し、それを言語モデル(LLM)が扱うトークン(=言語モデルが処理する最小単位)として受け取る構成が一般的です。画像特徴を LLM の語彙空間へつなぐ橋渡し役として、BLIP-2(Li et al., ICML 2023)が提案した Q-Former(少数の学習可能なクエリで画像特徴を要約しトークン化する仕組み)がよく使われます。

3D 知覚とは。カメラ画像は本来 2 次元(平面)ですが、運転では「物体が自分から何メートル先、どの高さ、どの方向にあるか」という 3 次元の立体配置が必要です。複数カメラ(サラウンドビュー)を組み合わせて周囲 360 度の立体空間を復元する処理を 3D 知覚と呼びます。前後左右の立体把握ができないと、車の周囲全体の安全判断ができません。

反事実的推論(counterfactual reasoning)とは。「実際には起きなかったが、もし別の選択をしていたらどうなったか」を考えることです。「もし今アクセルを踏んでいたら右から来た車とぶつかっていた」のように、取らなかった行動の結果を想像する力です。運転 AI にこれを明示的に学ばせると、「これからやろうとする行動が安全か」を実行前に自己検証できるようになります。安全クリティカルな計画では、正解の行動を当てるだけでなく「危険な行動を危険と認識できる」ことが同じくらい重要です。

クエリベースの 3D 知覚とは。膨大な画像特徴を全部扱う代わりに、少数の「クエリ(query, 問い合わせ点)」を用意し、各クエリが 3D 空間の特定の場所や物体に対応するように学習させる手法です。DETR3D や BEVFormer が代表で、「3D 空間の各所が何を意味するか」を少数のクエリベクトルに濃縮します。OmniDrive はこの仕組みを「画像特徴を LLM に渡すための圧縮器」として使います。

ベンチマークとは。複数の手法を同じ条件・同じ指標で比較できるよう整えた、評価用データセットと評価方法のセットです。共通のものさしがないと、各手法が本当に強いのか比較できません。

## 仕組みを詳しく
OmniDrive は「3D 対応の VLM アーキテクチャ(OmniDrive-Agent)」と「ベンチマーク(OmniDrive-nuScenes)」の2本柱です。

3D を扱う視覚言語アーキテクチャ。複数カメラ画像から 3D 空間表現を作り、それを LLM が言葉と一緒に扱えるようにします。鍵は、画像特徴を 3D 空間のクエリを介して圧縮・整理する仕組みで、Q-Former を 3D 位置クエリに拡張した設計(Q-Former3D と呼ばれる)を用います。
```
入力: 6 台のサラウンドカメラ画像
  各カメラ画像 → 視覚エンコーダ → 大量の 2D 画像特徴
3D クエリによる集約 (Q-Former3D):
  少数の 3D 位置クエリ (例: 数百個) が、画像特徴から
  「その 3D 位置に何があるか」を吸い上げて濃縮
  各クエリに 3D 位置エンコーディングを付与し、幾何を保つ
  → 3D 認識付きの少数トークン
LLM への入力:
  この少数の 3D トークン + テキスト (質問・指示) を LLM に渡す
  → LLM は「自車の前方 10m, 右 3m に車がいる」を言語文脈で理解
```
なぜクエリで圧縮するのか。6 カメラの生の画像特徴をすべて LLM に渡すとトークン数が膨大になり、計算が爆発します(自己注意の計算量はトークン数の2乗で増える)。3D クエリで「意味のある場所」だけに絞り込んでから渡すことで、3D 情報を保ちつつ計算を現実的に収める工夫です。さらに重要なのは、これらのクエリが 3D 物体検出や車線検出といった知覚タスクの教師信号でも学習される点です。これにより「言語を喋るためだけの曖昧な特徴」ではなく「幾何的に正確な 3D 表現」がクエリに宿り、LLM の言語推論能力と 3D 幾何の正確さを両立させる橋渡しになります。

知覚・推論・計画をつなぐタスク群。OmniDrive は単一の質問応答にとどまらず、1つのエージェントで次を扱います。
- シーン記述(今、周囲に何があるか)
- 注意すべき物体の特定(どの車・歩行者に注目すべきか)
- 反事実的推論(「もしこの行動を取ったら安全か/衝突するか/交通規則に違反するか」を判断)
- 意思決定と計画(最終的にどう走るか、軌道や高レベル行動を決める)

特に反事実的推論を明示的なタスクとして組み込んだのが特徴で、「取ろうとする行動が安全かを、結果を想像して検証する」能力を学習・評価します。

ベンチマーク OmniDrive-nuScenes。広く使われる nuScenes データセット(6 カメラ + LiDAR を備えた都市運転データ。ボストンとシンガポールの 1000 シーン、各 20 秒の実走行を含む)を土台に、上記タスクを問う質問応答と評価指標を整備しました。反事実的推論を問う QA(「この軌道を取ると衝突するか/車線を逸脱するか/赤信号を無視するか」など)を含むのが特徴です。これにより各手法が「見る・考える・走る」のどこが強くどこが弱いかを共通のものさしで比較できます。

処理の流れの例:
```
入力: 6 台のサラウンドカメラ画像
3D 圧縮: 画像特徴を 3D 位置クエリに集約 → 周囲の物体配置を 3D で把握
反事実問い: 「右車線へ変更したら安全か?」
  → LLM「右後方に高速接近車あり。変更すると衝突リスク。よって避けるべき」
計画出力: 「現車線を維持し緩やかに減速」+ 対応する未来軌道
```

## 手法の系譜と主要論文
OmniDrive は、クエリベース 3D 知覚と運転 VLM という2つの系譜が交わる地点にあります。

クエリベース 3D 知覚の系譜。DETR3D(Wang, Guizilini, Zhou, Li, Ambrus, Gaidon, CoRL 2021, arXiv:2110.06922)は、3D の物体クエリを画像へ投影して特徴を集める「クエリで 3D 物体検出する」枠組みを示しました。BEVFormer(Li, Wang, Lu et al., ECCV 2022, arXiv:2203.17270)は、複数カメラから BEV(Bird's-Eye-View, 鳥瞰図)クエリを介して時空間の鳥瞰図特徴を作る手法で、サラウンドカメラからの 3D 認識を大きく前進させました。StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926)は、3D 位置エンコーディングを付けたクエリを時系列に伝播させる効率的な設計を示しました。OmniDrive の「画像特徴を少数 3D クエリへ集約する」発想は、この系譜の延長です。

視覚言語の橋渡しの系譜。BLIP-2(Li, Li, Savarese, Hoi, ICML 2023, arXiv:2301.12597)の Q-Former が「少数クエリで画像を要約し LLM につなぐ」標準パターンを確立し、OmniDrive の Q-Former3D はこれを 3D 化したものです。

運転 VLM の系譜。DriveGPT4(Xu, Zhang, Tian et al., 2023, arXiv:2310.01412)は、動画と言語で運転を説明・制御する VLM を示し、「運転を言葉で説明できる」方向を切り開きました。ただし 2D 中心で、深い 3D 推論や反事実的推論、統一的な評価は弱かった。

OmniDrive(Wang, Huang, Zhang ら(NVIDIA ほか), CVPR 2025, arXiv:2405.01533, "OmniDrive: A Holistic Vision-Language Dataset for Autonomous Driving with Counterfactual Reasoning")の新規性は、(1) クエリベース 3D 知覚を VLM の入力圧縮機構(Q-Former3D)として組み込み、3D 幾何と言語推論を両立させたこと、(2) 反事実的推論を明示タスクとして導入したこと、(3) それらを公平に測る包括ベンチマーク OmniDrive-nuScenes を整備したこと、の3点です。言語整合特徴で汎化を狙う [DriveAnywhere](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0532_drive-anywhere) や、経験を蓄積する [DiLu](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0531_dilu_note)、LLM の判断を MPC に橋渡しする [LanguageMPC](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0540_languagempc) と比べ、OmniDrive は「3D 理解 + 反事実推論 + 統一評価」に焦点を当てた点で位置づけが異なります。

## 論文の実験結果(定量データ)
OmniDrive は OmniDrive-nuScenes 上で、シーン理解・注意物体特定・反事実的推論・計画という幅広いタスクを評価しました。主な評価軸は次の通りです。

- 質問応答の正確さ: シーン記述や注意物体などの QA に対する正答の質(言語生成の評価指標 CIDEr/BLEU や正答率)。
- 反事実的推論の正確さ: 「この行動を取ると衝突するか/安全か/規則違反か」の判断がどれだけ正しいか。これが OmniDrive の独自の核心指標です。
- 計画の質: 出力した軌道の衝突率(collision rate)と、軌道予測誤差(L2 距離。予測軌道と正解軌道のずれをメートルで測る指標で、小さいほど正確)。nuScenes 計画評価では 1秒・2秒・3秒先の平均 L2 と平均衝突率が定番の数値です。

報告によれば、3D クエリで画像特徴を LLM に橋渡しする設計により、2D ベースの VLM と比べて 3D 空間理解を要するタスク(物体の位置関係の把握、計画の衝突回避)で明確に優れ、反事実的推論を学習させた結果、「危険な行動を事前に検出する」能力で従来手法を上回ったとされます。計画では、反事実的推論を組み込むことで衝突率の低減と L2 誤差の改善が見られ、UniAD など従来の運転計画手法と競合する、あるいは上回る計画性能(低い平均 L2 と低い平均衝突率)を報告しています。衝突率は安全性を直接測る指標で、これが下がるほど実際の事故リスクが下がることを意味し、L2 誤差は計画の正確さを測るため、両者が同時に改善するのは「安全かつ正確な計画」を意味します。

アブレーションの観点では、(1) 3D クエリ圧縮を外して 2D 特徴を直接渡すと、空間理解タスクと計画の性能が落ちること(=3D 表現の寄与)、(2) 反事実的推論の学習タスクを外すと、危険行動の検出能力と計画の安全性が低下すること(=反事実推論の寄与)、(3) 知覚タスク(3D 検出・車線検出)による監督を外すとクエリの幾何精度が落ち、空間 QA が悪化すること、が示されています。これは「3D 表現」「反事実推論」「知覚監督」がそれぞれ独立に性能へ貢献していることを意味します。なお、絶対数値は使用した基盤 LLM やベンチマークのサブセットに依存するため、「3D 圧縮で空間理解が向上」「反事実推論で安全な計画になる」という傾向として理解するのが妥当です。

## メリット・トレードオフ・限界
メリット:
- 複数カメラから 3D 空間を理解でき、周囲 360 度の安全判断に向く。
- 反事実的推論で「取ろうとする行動が安全か」を実行前に吟味でき、衝突率を下げられる。
- 知覚・推論・計画を1つのエージェントで一貫して扱え、説明可能性が高い(なぜその計画かを言葉で言える)。
- 統一ベンチマーク OmniDrive-nuScenes により手法を公平に比較・評価でき、反事実 QA という新しい評価軸をコミュニティに提供した。
- 3D クエリ圧縮で、3D 情報を保ちつつ LLM のトークン数を現実的に抑える。

トレードオフ・限界(研究上の未解決課題):
- 複数カメラ + 3D + LLM の構成は計算が重く、車載でのリアルタイム運用(低レイテンシ)が難しい。
- ベンチマークが nuScenes 由来で、地域(ボストン/シンガポール)・天候・センサ構成に偏りがある。学習・評価データの多様性が課題。
- 反事実ラベル(「この行動は安全/危険」)の作成にコストがかかり、自動生成に頼ると質のばらつきがある。大規模化が難しい。
- LLM 推論を含むため幻覚(hallucination)のリスクがあり、安全判断の誤りに直結しうる。
- 3D クエリの数で表現できる細かさに上限があり、密集シーンでの取りこぼしの可能性。
- nuScenes の open-loop 計画評価は「正解軌道との一致」を測るため、自車挙動が周囲に与える影響を考慮しない既知の限界がある(low L2 が必ずしも実走行の安全を意味しない)。

## 発展トピック・研究の最前線
OmniDrive が示した「3D 認識を LLM に橋渡しし、反事実的推論で行動を検証する」枠組みは、運転 VLM エージェント研究の到達点の一つです。発展方向としては、まず時間方向の拡張が挙げられます。OmniDrive は主に瞬間のシーンを扱いますが、これを動画・時系列(StreamPETR 的なクエリ伝播)に拡張し、過去の動きから未来を予測したうえで反事実推論を行えば、より人間に近い先読みができます。

第二に、評価の刷新。open-loop の L2/衝突率は前述の限界があるため、実際にシミュレータで走らせて反応を見る closed-loop 評価(CARLA や nuPlan 系)への移行が進んでいます。第三に、効率化。3D クエリ圧縮はトークン削減に効きますが、それでも LLM 推論は重く、知識蒸留(大きいモデルの知識を小さいモデルに移す)、量子化、低頻度の言語推論と高頻度の制御の階層化([LanguageMPC](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0540_languagempc) の発想)などで車載レイテンシに近づける研究が進んでいます。第四に、言語整合特徴([DriveAnywhere](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0532_drive-anywhere))や経験蓄積([DiLu](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0531_dilu_note))との統合で、「3D で見て、経験を参照し、反事実で検証し、安全に計画する」完成度の高いエージェントを目指す方向です。データの偏り、反事実ラベルのコスト、幻覚対策は、運転 VLM 全般の最前線の課題として残っています。
