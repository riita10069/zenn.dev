# ポリラインエンコーディング

## ひとことで言うと
車線や道路境界、横断歩道などの「線状の地図要素」や、車・歩行者の「過去の動き」を、順番の付いた点の列(ポリライン=折れ線)として表し、それをニューラルネットが扱える1本の特徴ベクトル(数値の並び)に変換する技術です。ベクトル地図を使う知覚・予測・計画のほぼすべてで使われる、共通の「言葉(表現方法)」です。地図も動く物体も同じ形式に翻訳することで、一つのネットワークでまとめて扱えるようにするのが狙いです。

## 直感的な理解
道路の情報をコンピュータに渡すには、まず「形をどう表現するか」を決めなければなりません。やり方は大きく2通りあります。

一つはラスタ表現で、地図を画像(塗り絵)にして渡します。「この画素は車線、この画素は背景」という格子状の画像です。画像なので CNN(Convolutional Neural Network、畳み込みニューラルネット、画像処理が得意なネットワーク)がそのまま使えて手軽です。しかし弱点があります。線は本来細いのに、格子にすると太く滲んで正確な位置がぼやけます。広い範囲を高解像度で表すと画素数が爆発し、計算とメモリが重くなります(範囲を2倍にすると画素は4倍)。そして致命的なのが、「1本の車線」という個体(インスタンス)の情報が画素の集まりに溶けてしまうことです。隣り合う2本の車線が画素上でくっつくと、どこからどこまでが1本かが曖昧になります。

もう一つがベクトル表現で、形を点の座標の列で表します。「車線A = (0,0),(2,1),(4,3),...」という折れ線、これがポリラインです。細い線を細いまま、正確な座標で、コンパクトに表せます。点の数だけ持てばよいので広い範囲でも軽く(範囲を2倍にしても点数が線形にしか増えない)、「これは1本の車線」という個体性が自然に保たれます。

VectorNet(Gao et al., CVPR 2020)が「地図も周りの車の軌跡も、全部ポリラインで統一して表そう」と提案して以来、ベクトル表現が予測・計画分野の標準になりました。ポリラインエンコーディングとは、この「点列を、ネットワークが計算に使える1本の特徴ベクトルに変える」手続きを指します。地図と動く物体を同じ数値表現に翻訳して、一つのネットワークでまとめて扱えるようにする変換器、と理解してください。

## 基礎: 前提となる概念
理解の土台をいくつか噛み砕きます。

テンソルとは多次元の数値配列のことです。1次元なら数値の並び(ベクトル)、2次元なら行列(表)、それ以上もあります。本稿で `[9, 8]` と書けば「9行8列の表」を意味し、たとえば「9本のセグメントそれぞれが8個の数値で表される」状態を指します。shape(形)を追うことで、データがどう変形されていくかを正確に把握できます。

MLP(Multi-Layer Perceptron、多層パーセプトロン)とは、数値ベクトルを別の数値ベクトルへ変換する最も基本的なニューラルネットです。入力に重み行列を掛けて足し、非線形関数(ReLU など、負の値を0にする関数)を通す、を数層重ねたもので、「8次元の生入力を64次元の表現力ある特徴に持ち上げる」といった役割を担います。

max プーリングとは、複数のベクトルから「各次元ごとの最大値」を取って1本のベクトルにまとめる集約操作です。重要な性質として、入力の順番を入れ替えても結果が変わりません(順序不変、permutation invariant)。点列を要約するとき、点をどの順で並べたかに左右されない安定した要約が得られます。これは PointNet(Qi et al., CVPR 2017)が点群処理で確立した考え方で、VectorNet はこれをポリラインに応用しました。

attention(注意機構)とは、複数の要素があるとき「どの要素がどの要素に関係するか」を学習で重み付けする仕組みです。各要素が他の要素との類似度を計算し、その重みで他の要素の情報を集めます。ポリライン特徴どうしを相互作用させるとき、「この車はこの車線に関係が深い」という関係性をデータから学ばせるのに使います。

## 仕組みを詳しく
ポリラインを特徴ベクトルにエンコードする基本手順を、具体的な数値で追います。例として、ある車線を10個の点で表すとします。各点には座標 (x, y) があります。エンコードでは、点そのものより「点と点をつなぐ線分(セグメント、subgraph node)」を基本単位にすることが多いです。これは VectorNet の流儀で、各セグメントが向きの情報を自然に持てる利点があります。

ステップ1: 各セグメントの特徴を作る。
連続する2点 (xi, yi) と (xi+1, yi+1) から1本のセグメントを作ります。このセグメントを表す特徴として典型的に並べるのは次です。
- 始点 (xi, yi)
- 終点 (xi+1, yi+1)
- 種類ラベル(車線/境界/横断歩道などを表す番号やワンホットベクトル=該当要素だけ1で他は0のベクトル)
- 属性(その地図要素の ID、信号の状態など)

1セグメントを「始点2次元 + 終点2次元 + 属性4次元」で表すなら、10点から成る車線は9本のセグメント、それぞれ `[8]` 次元、合わせて `[9, 8]` の行列になります。これがエンコード前の生入力です。

```
  車線(点10個) → セグメント9本
  seg0: [x0,y0, x1,y1, type, attr...]  shape [8]
  seg1: [x1,y1, x2,y2, type, attr...]
   ...
  seg8: [x8,y8, x9,y9, type, attr...]
  まとめて [9, 8]
```

ステップ2: 各セグメントを MLP に通す。
各セグメント `[8]` を MLP に通し、表現力のある特徴 `[D]`(例 D=64)に持ち上げます。結果 `[9, 64]`。この時点では各セグメントはまだ自分の情報しか持っていません。

ステップ3: セグメント間で情報を集約する(ここが「ポリラインを1本にまとめる」核)。
VectorNet では、同じポリラインに属するセグメントどうしで情報を共有します。具体的には「各セグメント特徴に、そのポリライン全体の代表特徴(全セグメントの max プーリング = 各次元の最大値)を結合(concat)して、また MLP に通す」操作を数回繰り返します。これをサブグラフ(subgraph)層と呼びます。各セグメントが自分だけでなく全体の情報を持つように更新されていきます。最後にポリライン全体を max プーリングで1本のベクトル `[D]` にまとめます。これが「この車線」を表す1個のポリライン特徴です。max プーリングを使うので、点の並び順に左右されない安定した要約になります。

ステップ4: ポリライン特徴どうしを相互作用させる。
こうして「車線A=`[D]`」「車線B=`[D]`」「周りの車の過去軌跡C=`[D]`」のように、地図要素も物体も同じ `[D]` 次元のポリライン特徴に統一できました。VectorNet はこれら全部をグラフのノードとみなし、グローバルグラフ(global interaction graph)で attention を張って、どのノードがどのノードに関係するかを学習しながら相互作用させます。これで「この車はこの車線に乗っている」「この境界はあの車線の隣」という関係が捉えられ、未来の軌跡予測などに使えます。

正規化の注意も実務上重要です。座標はそのまま使うと値が大きくバラつくので、通常は自車(または予測対象のエージェント)を原点にした相対座標にし、進行方向を基準に回転させて正規化します(agent-centric / ego-centric 正規化)。「自車から見て前方5m・右1m」のように表すと、ネットワークが場所によらず一貫して学べ、汎化(学習していない場所でも正しく動く性質)が良くなります。近年はエージェントごとに座標系を取り直す query-centric な正規化も使われます。

なぜポリラインが「共通言語」なのか。上の手順で分かるように、車線も、道路境界も、横断歩道も、さらには周りの車や歩行者の過去の動きも、すべて「順序付き点列 → 特徴ベクトル」という同じ枠で表せます。表現が統一されていると、地図と物体を同じネットワークでまとめて扱え、相互作用を素直にモデル化できます。これが、ベクトル地図と軌跡予測で共通の基本単位として定着した理由です。

## 手法の系譜と主要論文
ポリライン表現の流れを系譜で追います。

VectorNet(Gao, Sun, Zhao, Anguelov, Li, Schmid, CVPR 2020, "VectorNet: Encoding HD Maps and Agent Dynamics from Vectorized Representation", arXiv:2005.04259)が出発点です。地図と物体軌跡を統一的にポリラインで表し、ポリライン内は MLP + max プーリングで集約(サブグラフ)、ポリライン間は attention で相互作用(グローバルグラフ)させる階層構造を提案しました。動機は、ラスタ(画像)表現のレンダリングコストと情報損失を避け、ベクトルで軽く正確に、構造的関係を保ったまま扱うことです。

LaneGCN(Liang, Yang, Hu, et al., ECCV 2020)が同時期に、車線を点列の要約ではなく「車線ノードのグラフ」として表し、前後左右の接続(predecessor/successor/left/right)を明示してグラフ畳み込みする路線を示しました。VectorNet が「順序不変の要約 + attention」なのに対し、LaneGCN は「接続を陽に持つグラフ畳み込み」で、設計思想の対比として重要です。詳細は [LaneGCN](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0443_lanegcn) を参照してください。

その後、TNT(Zhao et al., CoRL 2020)と DenseTNT(Gu, Sun, Zhao, ICCV 2021)が VectorNet 上に目標点(target/goal)ベースの予測を載せ、ポリライン表現が予測の主流基盤になりました。DenseTNT は車線上の密な候補点から目標を選ぶことで、アンカー不要の高精度予測を実現し、Argoverse リーダーボードで上位を取りました。HiVT(Zhou et al., CVPR 2022)や QCNet(Zhou et al., CVPR 2023)はポリライン化したシーンを階層的・query-centric な Transformer で処理し、大規模予測の標準を更新しました。

地図構築側では MapTR(Liao et al., ICLR 2023)が、ポリラインを学習で出力するときの最大の難所、点順序の曖昧さ(始点と終点を逆に書いても同じ車線、閉曲線ならどこから書き始めても同じ)を、permutation-equivalent(順序の入れ替えに対して等価)な点集合のモデリングと損失設計で吸収し、オンライン地図構築でポリラインを安定して直接生成できるようにしました。詳細は [InstaGraM / インスタンスマスク地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0450_maplt-tr-instance-mask) を参照してください。近年の Transformer ベースの予測・地図モデル(車線をトークンとして扱う系統)は、いずれもこのポリライン表現を内部の基本単位として受け継いでいます。

## 論文の実験結果(定量データ)
VectorNet は Argoverse(アルゴバース、車載センサーで集めた軌跡予測の標準ベンチマークで、過去2秒の軌跡から未来3秒を予測する課題)と、Waymo の社内データで評価されました。指標は ADE(Average Displacement Error、予測軌跡と正解軌跡の各時刻の位置誤差の平均、単位メートル、小さいほど良い)と DE@3s(3秒後時点の位置誤差)が代表です。

報告によれば、VectorNet はラスタ画像 + CNN(ResNet)のベースラインに対して、より少ない計算量で同等以上の精度を達成しました。具体的には、同程度の予測誤差を、モデルパラメータと計算量(FLOPs)を大幅に削減して実現したとされ、論文では VectorNet のモデルサイズと計算量がラスタ CNN ベースラインの数分の一でありながら DE@3s が同等以下だったと報告されています。これは「ベクトルで直接扱う方がラスタより軽くて正確」という主張を裏づけました。地図とエージェントを統合したことで、地図無しの場合より予測誤差が明確に下がったというアブレーションも示され、地図文脈が予測に効くことが定量化されています。DenseTNT はその上で Argoverse の minADE/minFDE/Miss Rate をさらに改善し、目標点ベースのポリライン活用が有効だと示しました。

MapTR 側の報告では、ポリラインの点順序を素朴に固定した場合に対して、permutation-equivalent な損失を導入すると学習が安定し、オンライン地図構築の mAP(地図要素の平均適合率、高いほど良い)が改善したとされます。順序の曖昧さを放置すると、同じ形でも点の書き順が違うだけで「不正解」と罰せられ学習が荒れるため、ここを吸収したことが効きました。MapTR は nuScenes で当時の手法を上回る mAP と高い推論速度(リアルタイムに近い FPS)を両立したと報告されています。これらの結果は「ポリラインは強力だが、出力する側では点順序と点数の扱いを丁寧に設計しないと学習が崩れる」という、この表現特有の注意点を示しています。

## メリット・トレードオフ・限界
メリットを整理します。線状の地図要素を細く正確に、コンパクトに表せるので、ラスタの滲みと計算爆発を回避できます。地図要素も物体軌跡も同じ点列→特徴ベクトルの枠で統一でき、相互作用を素直に扱えます。インスタンス(1本の車線)という個体性が自然に保たれます。座標ベースなので相対座標化・回転正規化との相性も良く、汎化しやすい。

トレードオフと限界も挙げます。点列という構造化データを用意する前処理が必要で、生のセンサーから直接は出ません(ベクトル地図か、ベクトル化処理が要る)。点順序の曖昧さや点数の扱いに工夫が要り、MapTR の permutation-equivalent などの対処を入れないと学習が不安定になります。max プーリング等の順序不変な集約は、点列の細かい順序情報(どちらが進行方向の前か、といった向き)を一部捨ててしまいます。また、点数を固定する設計が多く(車線を必ず20点で表す、など)、非常に長い車線や複雑な交差点を一定数の点で表しきれない場合に表現力が頭打ちになります。曲率の高いカーブを少ない点で表すと折れ線が実形状からずれます。

未解決の課題としては、可変長・可変点数のポリラインを破綻なく扱う表現、曲率の高い形状を少ない点で正確に表す方法(連続曲線表現)、ラスタとベクトルの長所を併せ持つハイブリッド表現などが研究されています。

## 発展トピック・研究の最前線
ポリライン表現の発展として、点列をそのまま Transformer のトークン列として扱い、自己回帰的に点を生成する系統(言語モデル的に座標トークンを順に吐く)や、ベジエ曲線・スプラインなど連続パラメトリック曲線で車線を表してポリラインの離散性を緩める研究があります。BeMapNet(Qiao et al., CVPR 2023)はベジエ曲線で地図要素を表し、少ない制御点で滑らかな形状を表現しました。地図構築では、形状のポリラインに加えて車線中心線の接続(トポロジー)を同時に出す方向(TopoNet など)が強まり、ポリラインが「形」だけでなく「つながり」も運ぶ表現へ拡張されています。軌跡予測では、ポリライン化した地図とエージェントを統一トークンとして大規模 Transformer(query-centric な QCNet、シーン全体を一括予測する MTR など)に流す手法が主流化しており、本トピックは地図エンコード系統([LaneGCN](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0443_lanegcn)、[InstaGraM / インスタンスマスク地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0450_maplt-tr-instance-mask))を学ぶ上で最も基礎的な出発点です。
