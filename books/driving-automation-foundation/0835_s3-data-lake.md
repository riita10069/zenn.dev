# オブジェクトストレージによるデータレイク (datasets/checkpoints/artifacts)

## ひとことで言うと
機械学習で扱う「学習データ」「学習途中・完成したモデルの重み(checkpoint)」「学習結果の成果物(ログやグラフなど)」を、クラウド上の大容量ストレージであるオブジェクトストレージ(Amazon S3 など)に置いて一元管理する仕組みです。計算するマシンとデータを置く場所を切り離し、用途ごとにバケット(入れ物)を分け、checkpoint には変更履歴を残す versioning を有効にするのが定石です。これにより、計算ノードを使い捨てにしてもデータは失われず、複数のジョブや複数の人で同じデータを共有できます。

## 直感的な理解
深層学習の学習を回すとき、最初に直面する現実的な壁は「データもモデルも大きすぎてマシンに乗り切らない」ことです。具体的にイメージしてみましょう。

自動運転の学習では、カメラを複数台(たとえば7台)積んだ車両が走り回って映像を集めます。これを何時間ぶんも集めると、データは簡単に数百GBから数TBになります。一方で、学習の途中で保存するモデルの重み(checkpoint)は1個あたり数百MBから数GBあり、それを何百個も生成します。

これらのファイルを、学習を実行する GPU マシンのローカルディスクに置こうとすると、すぐに行き詰まります。GPU マシンのディスクは数百GB程度で、データセット全部は入りません。さらにクラウドの GPU ノードは「使い終わったら廃棄」される運用が一般的で(コスト削減のため、あるいはオートスケーラがノードを入れ替えるため)、ローカルに置いたファイルはノード廃棄でまるごと消えます。そして学習ジョブA・評価ジョブB・別の研究者Cが同じデータセットを参照したいのに、各マシンにコピーするのは時間と容量の無駄です。

解決策はシンプルで、「計算するマシン」と「データを保管する場所」を分離します。データはクラウドストレージに一元的に置き、計算マシンは必要なときにそこから読み出す。この「形式を問わず大量の生データから加工済みデータまでまとめて貯めておく大きな保管庫」を、一般にデータレイク(data lake)と呼びます。湖(lake)のように、構造化・非構造化を問わずあらゆるデータを一か所に流し込むイメージです。

## 基礎: 前提となる概念
ここで前提となる概念をいくつか噛み砕いておきます。

オブジェクトストレージとは、ファイルを「フォルダの階層」ではなく「バケット(bucket、大きな入れ物)の中にキー(key、ファイル名のような一意の文字列)で1個ずつ置く」方式の保管庫です。これは「これはオブジェクト(object)という単位で、データ本体・メタデータ・一意の識別子をひとまとめにしたものを指します」。容量は実質無制限で、置いたぶんだけ課金されます。普通のファイルシステムのように OS にマウントして `open()` で読むのではなく、HTTP API(あるいは `aws s3 cp` のようなコマンド)で読み書きします。Amazon S3(Simple Storage Service)、Google Cloud Storage、Azure Blob Storage などが代表例です。

ここで重要な特性が2つあります。第一に、オブジェクトストレージは「大きなファイルを通しで読む(シーケンシャルリード)」のは非常に速いが、「小さなファイルを1個ずつ大量に開く(ランダムアクセス、多数の小オブジェクトへの個別リクエスト)」のは遅く、しかもリクエスト回数ぶんの課金がかかります。第二に、ローカルディスクと違ってネットワーク越しのアクセスなので、1回ごとのレイテンシ(遅延、リクエストを出して応答が返るまでの時間)が大きい。この2つが、後で「データをどう並べておくか」を決める制約になります。

checkpoint(チェックポイント)とは、「学習の途中や完了時点で、モデルの全パラメータ(重み)と、場合によっては optimizer の内部状態や学習ステップ数までをファイルに保存したもの」を指します。なぜ保存するか。学習は何時間から何日もかかるので、途中でマシンが落ちても再開できるよう定期的に状態をスナップショットしておく必要があります。また完成した重みは推論(実際に使う段階)に必須です。

## 仕組みを詳しく
典型的な構成では、用途ごとに役割を分けた3つのバケットを使います。

```
s3://<prefix>-datasets/      ← 学習データ(入力)
s3://<prefix>-checkpoints/   ← モデルの重み(versioning 有効)
s3://<prefix>-artifacts/     ← 成果物(ログ・評価結果・可視化など)
```

なぜ分けるのか。3者は書き換え頻度・サイズ・保護したい度合いがまったく違うからです。バケットを分けると、それぞれに別々のライフサイクルポリシー(古いデータを安いストレージ階層へ移す、一定期間で消すといった自動ルール)・別々の versioning 設定・別々のアクセス権限を付けられます。

### datasets バケット(学習データ)

学習に使う入力データを置きます。ここで重要なのが、生のセンサーデータをそのまま大量の小ファイルとして置くのではなく、前処理で WebDataset のような「shard(シャード、データを一定サイズで束ねた塊)」形式に変換してから置くことです。WebDataset とは「たくさんの小さなサンプルを `.tar` ファイルに固めて連番で並べた形式」で、学習時に順番に大量読み出しするのに向いています。

具体例で考えます。複数カメラの映像を集めたデータを ingest(取り込み)するとき、映像デコードライブラリ(PyAV など)でフレームに分解し、サンプルごとにまとめて `shard-000000.tar`, `shard-000001.tar` … と連番のオブジェクトにします。学習時は GPU マシンがこの shard をストリーミング(少しずつ読みながら処理)します。全データをローカルにダウンロードし切る必要がない点が利点です。

ここで先述のオブジェクトストレージの特性が効いてきます。仮に1サンプルを1ファイルにして100万サンプル置くと、100万回の個別リクエストが必要で、レイテンシも課金も膨らみます。一方で1000サンプルを1つの `.tar` shard に束ね、それを1000個置けば、リクエストは1000回で済み、各 shard は通しで高速に読めます。

```
悪い例: 1,000,000 個の小オブジェクト → 1,000,000 リクエスト, レイテンシ支配
良い例: 1,000 個の .tar shard (各 1000 サンプル) → 1,000 リクエスト, シーケンシャル読みで高速
```

### checkpoints バケット(モデルの重み、versioning 有効)

このバケットだけ versioning(バージョニング、版管理)を有効にするのが定石です。versioning とは「同じキー(ファイル名)に上書き保存しても古い版を消さず、別バージョンとして残すオブジェクトストレージの機能」です。なぜ checkpoints だけ有効にするか。

- 同じパス(たとえば `model/latest.pt`)に最新版を上書きしていく運用をしたいが、間違って壊れた重みで上書きしても過去の正常な版に戻したい。
- 「3エポック目の重みで評価したら一番良かった」と後から分かったとき、その版を取り出したい。

versioning は古い版もストレージ課金されます。だからデータセットのように巨大かつ何度も書き換えるものには付けません(コストが膨らむため)。役割で分けるのは、このコストと安全性のトレードオフを最適化するためです。

### artifacts バケット(成果物)

学習・評価の結果として出てくるファイルを置きます。学習曲線のログ、評価指標の表、可視化画像など。実験管理ツール(MLflow など)が実験のアーティファクトをここに保存し、後から「どの実験が何を出したか」を追えるようにします。

### アクセス制御の仕組み

GPU マシンがストレージを読み書きするための認証は、アクセスキーをコンテナに直接埋め込まず、実行主体(Kubernetes の ServiceAccount や EC2 のインスタンスプロファイル)に IAM ロール(権限のかたまり)を紐づける方式が安全です。これは「短命の一時クレデンシャルが自動で配られ、漏洩リスクと管理コストを下げる」ためです。最小権限の原則(必要なバケットへ必要な操作だけ許可する)を守ると、誤って本番データを消す事故も防げます。

## 手法の系譜と主要論文
これはインフラ設計のトピックなので、特定の単一論文というより、分野で確立した設計原則と代表的なシステム論文の系譜で理解します。

「計算とストレージの分離(disaggregation / separation of compute and storage)」は、クラウド時代のデータ基盤の基本原則として広く知られています。古典的なデータウェアハウスは、ストレージと計算が同じノードに密結合していて(shared-nothing 構成)、片方だけを増やせませんでした。これに対し Dageville らの「The Snowflake Elastic Data Warehouse」(SIGMOD 2016)は、永続データをオブジェクトストレージ(S3)に置き、計算ノードを完全に状態を持たない使い捨てクラスタとして独立にスケールさせる設計を示し、商用システムとして大きく成功しました。要点は「計算資源とデータを別々にスケールできる」「ノード故障がデータ損失に直結しない」こと。トレードオフは「ローカルディスクより読み出しレイテンシが大きいため、キャッシュやプリフェッチの工夫が必須」になることです。

機械学習のデータ供給の効率化という文脈では、WebDataset の設計(tar ベースのシーケンシャル読み出し)が、ランダムアクセスの多い従来形式よりオブジェクトストレージと相性が良いことが知られています。これは PyTorch のデータローディング、NVIDIA DALI(GPU 上で前処理を行うライブラリ)、TensorFlow の TFRecord などにも共通する思想で、いずれも「小オブジェクトの個別アクセスを避け、大きな連続ブロックを高速に流し込む」ことを狙っています。

データのバージョン管理という観点では、git のコミットのように「データセットのスナップショットに immutable な版を付け、いつの版で学習したかを再現できる」仕組み(LakeFS、DVC、Delta Lake など)が近年広く使われるようになりました。これは再現性(同じ入力で同じ結果を得られること)の確保という、機械学習の信頼性に直結する課題への回答です。

## 論文の実験結果(定量データ)
インフラ系の主張は、論文の単一指標というより運用上の数値で語られます。代表的な定量論点を挙げます。

Snowflake の論文では、計算とストレージを分離したことで、同じデータに対して計算クラスタを需要に応じて秒〜分単位で増減でき、ピーク時とアイドル時のコスト差を大きく圧縮できることが示されました。ここで重要な指標は「ストレージ容量あたりのコスト」と「計算稼働時間あたりのコスト」を独立に最適化できる点です。両者を密結合した従来構成では、データを増やすために計算ノードを増やす(=使わない計算資源にも課金される)無駄が避けられませんでした。

機械学習のデータローディングについては、ベンチマーク上、小ファイルの個別読み出しと比べて shard 化したシーケンシャル読み出しでスループット(単位時間あたりに供給できるサンプル数)が桁違いに向上することが繰り返し報告されています。GPU を遊ばせない(GPU utilization を高く保つ)ためにはデータ供給がボトルネックにならないことが必須で、ここでデータ供給が詰まると、たとえ高価な GPU を使っていても実効性能が大きく落ちます。経験則として、データ供給を shard + プリフェッチ + 並列ワーカーで最適化すると、GPU 利用率が 50% 台から 90% 台まで改善する事例が珍しくありません。

## メリット・トレードオフ・限界
メリット
- 計算マシンを使い捨てにできる。ノードが消えてもデータ・checkpoint はストレージに残る。
- 容量が実質無制限で、置いたぶんだけ課金。事前にディスクサイズを見積もる必要がない。
- 複数ジョブ・複数人で同じデータを共有でき、コピーの無駄がない。
- checkpoints の versioning で「壊れた上書き」から復旧でき、過去の重みも取り出せる。

トレードオフ・限界
- ローカルディスクより読み出しが遅い(ネットワーク越し)。学習データはストリーミング前提の形式にする工夫が要る。
- versioning は古い版も課金対象。全バケットに付けるとコストが膨らむため用途で使い分けが必要。
- アクセス制御を正しく設計しないと、読めない/誤って書ける事故が起きる。最小権限の徹底が前提。
- 小さなファイルを大量に置くと、操作回数ぶんの課金とレイテンシが効く。事前に shard へまとめる前処理が必須。
- 強整合性(書いた直後に必ず最新が読める保証)の扱いはサービスごとに違う。古くは結果整合性で、書いた直後に古い版が読めることがあり、パイプライン設計で注意が要った。

未解決・研究上の課題としては、データセットのバージョン管理と実験のトレーサビリティ(どの版・どの前処理で学習したか)を完全に自動で担保する仕組みは、まだ運用の手間が残ること。また超大規模学習ではストレージ帯域そのものがボトルネックになり、ローカル NVMe キャッシュ階層を挟む multi-tier 設計が研究・実用の最前線です。

## 発展トピック・研究の最前線
- 階層化ストレージ(tiering): ホットなデータをローカル NVMe にキャッシュ、コールドなデータをオブジェクトストレージに置き、アクセスパターンに応じて自動で移す。Alluxio のような分散キャッシュ層が代表例。
- データバージョニング基盤(LakeFS / DVC / Delta Lake / lakehouse アーキテクチャ): オブジェクトストレージの上に git ライクなブランチ・コミット・ロールバックを載せ、データと実験の再現性を担保する流れ。
- フォーマットの進化: Parquet / Arrow / Lance など、列指向で圧縮効率とランダムアクセスを両立する形式が、機械学習データの保管でも採用されつつある。
- ゼロコピーなデータ供給: GPU Direct Storage のように、ストレージから GPU メモリへ CPU を介さず直接転送し、データローディングの CPU ボトルネックを外す研究・実装。
