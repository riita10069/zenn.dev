# ストリーミング時系列融合(クエリ繰り越し型)

## ひとことで言うと
過去のフレームで作った「物体を表すクエリ (問い合わせベクトル)」を捨てずに次のフレームへ繰り越し、毎フレーム少しずつ更新していく時間融合のやり方です。動画を毎回先頭から見直すのではなく、前回の理解を引き継いで「差分だけ」更新するイメージで、計算が軽く遅延が小さいのが最大の特徴です。マルチカメラの 3D 物体検出で、リアルタイム性と精度を両立する標準的なアプローチになりました。

## 直感的な理解
カメラだけで物体の 3D 位置や速度を当てるのは、本来かなり難しいタスクです。1 枚の画像には奥行きの情報が直接写っておらず、「その車が近づいているのか止まっているのか」は 1 枚では判別しにくい。動きや距離を読むには、複数フレームにまたがる時間的な手がかりが要ります。

素朴な時間融合は「過去 N フレームの画像特徴をまとめて並べ、一括で処理する」方式です。ところがこれを多視点 (6〜7 台のカメラ)・複数フレームでやると、扱う特徴の量が `フレーム数 × カメラ数 × 1 フレームの特徴量` と膨れ上がります。しかも新しいフレームが来るたびに過去 N 枚を再処理するので、フレーム数を増やすほど 1 回あたりの処理時間 (遅延) が伸びます。自動運転は秒間何十回も判断するリアルタイム処理なので、この遅延の増大は致命的です。

ここで発想を変えます。「ストリーミング」とは、データを一度に全部読み込むのではなく、流れてくるそばから少しずつ処理することです。動画を毎回頭から見直すのではなく、川の流れのように来たフレームを順次取り込み、前回までの理解だけを引き継ぐ。引き継ぐものを「重い画像特徴」ではなく「軽い物体クエリ」にするのが、この手法の核心的なアイデアです。物体はピクセルより圧倒的に数が少ないので、繰り越すデータ量が劇的に減ります。比喩を避けて言えば、過去の情報を「シーン全体の画素表現」ではなく「検出済みの物体リスト」として保持する、という表現の選び方の問題です。

## 基礎: 前提となる概念
クエリ (query): DETR 系の Transformer 検出器が使う仕組みで、「シーンの中の物体候補それぞれを表す、学習可能なベクトルの問い合わせ」です。各クエリが「ここにこういう物体がいそうだ」という仮説を担い、画像特徴に注意 (attention) を当てて中身 (クラス・位置・大きさ) を埋めていきます。1 フレームあたり数百個のクエリを使うのが典型です。

DETR (DEtection TRansformer; Carion et al., ECCV 2020): 物体検出を「集合予測」として定式化し、固定数のクエリと画像特徴の間で attention を繰り返して各クエリに物体を割り当てる枠組みです。正解との対応付けはハンガリアンマッチング (1 対 1 の最適割り当て) で行い、従来の検出器が必要としたアンカーや後処理 (非最大抑制 NMS) を不要にしました。クエリという「物体の器」を持つことが、後述のクエリ繰り越しを可能にします。

BEV (Bird's-Eye View): 俯瞰図のこと。複数カメラの特徴を地面を見下ろす格子に投影した表現で、複数視点・複数時刻の融合を一つの座標系で扱えるため運転知覚で多用されます。BEVFormer は時間融合をこの BEV 格子の上で「密」に行う代表例です。

疎 (sparse) と密 (dense): シーンを「物体がいる少数の点」だけで表すのが疎、空間全体を格子で埋め尽くして表すのが密。クエリ繰り越しは疎、BEV 融合は密の代表です。疎は軽いが背景情報を落とし、密は重いが情報を取りこぼしにくい、という対照があります。

ego motion (自車運動): 自分の車自身の移動。過去フレームで「ここに車がいた」と認識した 3D 位置は、自車が前進した現在のフレームでは座標がずれます。過去の情報を現在の座標系に正しく持ち越すには、経過時間と自車の移動量で位置を補正する必要があります。

## 仕組みを詳しく
中心となるのが「メモリキュー (memory queue)」と「クエリの伝播 (propagation)」です。代表例の StreamPETR を例に追います。

1. 各フレームで、検出された物体に対応するクエリ (物体の見え方・3D 位置・速度などをエンコードしたベクトル) を、過去数フレームぶんキューに溜めておきます。
2. 次のフレームでは、新しく初期化したクエリ群に加えて、キューに溜めた過去のクエリも入力に使います。つまり「今の検出を、過去の検出仮説で条件付ける」わけです。過去に確信を持って検出した物体は、今フレームでも見つけやすくなります。
3. ここで効くのが ego motion の補正です。過去クエリが指していた 3D 位置は、自車が動いた今のフレームでは座標がずれています。StreamPETR は動き対応の正規化 (motion-aware layer normalization, MLN) で、経過時間と自車運動からクエリの位置情報を現在の座標系へ更新します。これにより「過去の物体が今どこにいるはずか」を正しく持ち越せます。MLN は、経過時間と自車の運動行列を埋め込んでクエリの正規化のスケール・シフトを変調する仕組みで、単なる座標変換ではなく特徴側にも時間・運動の情報を注入する点が巧妙です。

形のイメージ:

```
従来(マルチフレーム一括):
  フレームt の入力 = [画像特徴(t), 画像特徴(t-1), ..., 画像特徴(t-N+1)]
  → 毎回 N フレーム分の重い画像特徴を再処理(計算重い、遅延大)
  コスト ~ N × カメラ数 × (H×W×C)

ストリーミング(クエリ繰り越し):
  フレームt の入力 = [今フレームのクエリ] + [メモリキュー: 過去クエリ(運動補正済み)]
  → 過去は「クエリ」という小さなベクトルとして繰り越すだけ(軽い、遅延小)
  コスト ~ (今のクエリ数 + キューのクエリ数) × クエリ次元
```

物体は画素より桁違いに少ない (数百個 vs 数十万画素) ので、繰り越すデータ量が劇的に減ります。結果として、参照する時間文脈を長くしても 1 フレームあたりの計算がほぼ一定に保て、長い履歴を低コストで活かせます。速度推定 (複数時刻の位置差から速度を読む) や遮蔽 (物体が一時的に隠れても過去クエリで追える) への頑健性が上がるのも、この繰り越しの恩恵です。

なぜクエリ繰り越しが密な BEV 融合より速いのか。BEVFormer のような密な方式は、時間融合を BEV 格子の全マス (数万マス) に対して行うため表現は豊かでも計算が重い。クエリ繰り越しは「物体がいる場所」という疎な点だけを持ち回るので、無駄な背景領域の再計算を避けられます。これが疎な物体中心モデリングの効率の源です。

学習時の注意点も触れておきます。繰り越しは前フレームの状態に依存する再帰的な構造なので、学習も複数フレームを連ねて行います。実装上は、メモリキューの勾配を全フレームに逆伝播させると重いため、過去クエリは勾配を切って (detach して) 入力として扱うのが一般的です。これは「過去は確定した観測として条件付けに使い、学習の勾配は今フレームに集中させる」という割り切りで、効率と安定性を両立させます。

## 手法の系譜と主要論文
- DETR (Carion et al., ECCV 2020, arXiv:2005.12872): クエリで集合予測する枠組みの原点。クエリという「物体の器」がのちの繰り越しの土台になった。
- PETR (Liu et al., ECCV 2022, arXiv:2203.05625): 多視点画像に 3D 位置埋め込みを与え、DETR 式のクエリで多視点 3D 検出する単フレーム枠組みを作った。3D 空間の位置情報を画像特徴に直接埋め込む点が新しく、後続のクエリベース手法の土台になった。
- StreamPETR (Wang et al., "Exploring Object-Centric Temporal Modeling for Efficient Multi-View 3D Object Detection", ICCV 2023, arXiv:2303.11926): PETR の単フレーム枠組みに「物体中心の時間ストリーミング」を足した。過去クエリをメモリキューで繰り越し、MLN で自車運動に応じて位置更新。動機は「画像特徴を全フレーム持ち回るのは冗長で重いが、物体は数が少なくクエリで表せば軽い」こと。
- Sparse4D / Sparse4Dv2 (Lin et al., 2022–2023, arXiv:2211.10581 / 2305.14018): 過去の物体特徴・サンプリング点を繰り越す再帰的時間融合 (recurrent temporal fusion) を提案。v2 では繰り越しを効率化し、ストリーミング方式が効率と精度を両立できることを補強した。
- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270): 対照的に、時間情報を BEV 特徴マップ上で融合する「密 (dense)」な方式。表現は豊かだがクエリ繰り越しより計算は重い、というトレードオフ関係にある。疎 (クエリ) 対密 (BEV) の代表的な対比として押さえておくとよい。研究コミュニティでは、この疎・密の対比が時間融合設計の中心軸になっています。

## 論文の実験結果 (定量データ)
評価ベンチマークの説明: nuScenes はマルチカメラ自動運転の標準データセットで、6 台のカメラ・LiDAR・レーダーを備えた約 1000 シーンから成ります。3D 検出の主指標は mAP (平均適合率; 検出位置・クラスの正確さ) と NDS (nuScenes Detection Score; mAP に加えて位置・大きさ・向き・速度・属性の誤差を総合した独自指標) です。NDS は速度推定の良し悪しも反映するため、時間融合の効果が現れやすい指標です。

- StreamPETR は nuScenes カメラ 3D 検出で、当時の高精度・高効率を達成しました。報告では、ResNet-50 バックボーンのオンライン設定で NDS が概ね 0.55 前後、mAP が 0.45 前後に達し、同等バックボーンの単フレーム PETR (NDS 0.44 前後) や一括方式に対して、計算をほとんど増やさずに NDS・mAP を有意に改善したとされています。大型バックボーン (V2-99 など) では NDS 0.6 を超える報告もあります。特に mAVE (平均速度誤差; 値が小さいほど良い) が大きく下がり、時間情報の活用で速度推定が改善することを示しました。
- アブレーション (どの要素を抜くとどうなるか) が示唆的です。MLN (運動補正) を外すと、自車が動く区間で過去クエリの位置がずれ、精度が明確に低下します。これは「過去を繰り越すだけでは不十分で、自車運動の補正が不可欠」という設計の核心を裏づけます。また、メモリキューに溜める過去フレーム数を増やすほど (ある範囲まで) 精度が上がり、長い時間文脈が効くことが確認されています。一方で長くしすぎると古いクエリの位置補正誤差が累積し、頭打ちまたは微減に転じます。
- 効率面では、参照フレーム数を増やしても 1 フレームの推論時間がほぼ横ばいに保たれることが報告されており、これがマルチフレーム一括方式に対する最大の優位点です。一括方式は N に比例して遅延が伸びるのに対し、ストリーミングはほぼ定数時間で済みます。報告では、StreamPETR は推論速度 (FPS) を保ったまま多フレームの恩恵を得られるとされ、リアルタイム要件のある車載応用に適合します。

数値の意味: NDS の 0.01〜0.02 の差は、nuScenes のリーダーボードでは順位を左右する有意な差です。mAVE の改善は「車の速度をより正確に当てられる」ことを意味し、追従・割り込み判断の安全性に直結します。速度を正確に当てられないと、前方車の減速の見積もりがずれ、ブレーキ判断が遅れるからです。

## メリット・トレードオフ・限界
- メリット: 過去を軽いクエリで繰り越すため、参照する時間文脈を長くしても 1 フレームあたりの計算がほぼ一定。リアルタイム性に強い。
- メリット: 長い時間文脈を低コストで使えるので、速度推定や遮蔽への頑健性が上がる。物体中心なので検出・追跡と相性がよく、物体 ID の一貫性も保ちやすい (繰り越したクエリがそのまま追跡の対応付けになる)。
- トレードオフ: クエリに圧縮した瞬間、物体に結びつかない背景や細部の情報は落ちやすい。新規に出現した物体 (過去クエリに対応がない) の初検出は、繰り越しの恩恵を受けにくい。
- トレードオフ: 過去クエリの位置更新が自車運動の推定精度に依存する。運動推定が荒い、あるいはセンサ時刻同期がずれると、過去情報が誤った位置に乗り、かえって有害になりうる。
- 限界: 実装が再帰的 (前回状態に依存) になるため、学習時の系列展開 (BPTT 的な扱い) や並列化の設計が、密な一括方式よりやや複雑。学習と推論で系列長や状態の扱いを一致させないと、train/test の不一致が精度を蝕む。最初のフレーム (まだ過去がない) の扱いをどう初期化するかも設計判断が要る。

## 発展トピック・研究の最前線
- 疎 (クエリ) 対密 (BEV) の融合: クエリ繰り越しの効率と BEV の表現力を両取りするハイブリッド (疎クエリで物体、密 BEV で地図・占有) が探られています。占有予測 (occupancy prediction; シーンを 3D ボクセルの占有として表す) との統合もこの流れです。
- 物体中心の世界モデルへ: 過去クエリの繰り越しは、シーンを「物体の集合とその状態遷移」として持つ世界モデル的発想と親和的で、検出にとどまらず将来予測・計画まで一気通貫で物体中心に扱う研究へつながっています。
- 低レート文脈との相補性: 物体クエリの繰り越し (疎・物体中心) と、密な特徴の時間圧縮 ([時系列メモリの低レート(1Hz)エンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0472_temporal-memory-1hz-encoder) のような低レートエンコーダ) は、捉える情報が異なる相補的な選択肢です。物体の動きはクエリで、シーン全体の雰囲気や背景変化は密特徴で、と使い分ける設計が考えられます。
- センサ同期と時刻補正: ストリーミングの精度はクエリの位置補正の質に強く依存するため、マルチセンサの時刻同期や ego motion 推定の高精度化が、知覚精度を底上げする研究テーマとして重要度を増しています。クエリ繰り越しは追跡 (tracking) と本質的に近いため、検出と追跡を一体で学習する end-to-end tracking への発展も進んでいます。
