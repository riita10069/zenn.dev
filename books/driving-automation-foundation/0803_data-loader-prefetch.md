# DataLoader Prefetch / num_workers(データの並列先読み)

## ひとことで言うと
学習中、GPU が今のバッチ(まとめて処理するデータの束)を計算している裏で、複数の作業員(ワーカー)に次のバッチを準備させておく仕組みです。これにより GPU が「次のデータがまだ来ない」と待つ時間を減らし、学習スループット(単位時間に処理できるデータ量)を上げます。コンピュータ工学で古くからあるパイプライン化(ある工程の出力を待たずに次工程を並行進行させる)と、プリフェッチ(必要になる前にデータを取り寄せる)を、データ供給に適用したものです。

## 直感的な理解
回転寿司を思い浮かべてください。客(GPU)が皿を食べている間に、厨房(CPU ワーカー)が次の皿を握ってレーンに流しておけば、客は食べ終わってすぐ次の皿を取れます。もし握り手が1人で「1皿食べ終わってから次を握り始める」と、客は毎回待たされます。握り手を増やし、しかも数皿先まで作り置きしておけば、客の口は止まりません。GPU は1台数百万円する高級客なので、絶対に待たせたくないのです。学習全体の所要時間は「最も遅い工程」で決まるので、データ準備が遅ければ GPU がどれだけ速くても全体は速くなりません。

## 基礎: 前提となる概念
学習の1ステップは、ざっくり2種類の作業からなります。

1. データ準備(CPU の仕事): ディスク/ネットワークから画像や動画を読み、デコード(JPEG や H.264 を画素配列に展開)し、リサイズ・正規化・データ拡張(augmentation、左右反転や色変えで水増し)をして、テンソル(数値の多次元配列)に整える。
2. 計算(GPU の仕事): 整ったテンソルをモデルに通して forward → loss → backward → 重み更新。

ここで問題が起きます。1人(1プロセス)で順番にやると次のようになります。

```
[CPUがデータ準備]→[GPUが計算]→[CPUがデータ準備]→[GPUが計算]→...
      (GPUは待ち)        (CPUは待ち)      (GPUは待ち)
```

GPU はデータ準備の間遊び、CPU は計算の間遊びます。特に自動運転データは「複数カメラの高解像度画像や動画フレームのデコード」が重く、データ準備が計算より長くかかることがよくあります。すると高価な GPU がデータ待ちで空回りします。これを「データローディングがボトルネック」あるいは「data stall(データ供給の停滞)」と呼びます。

用語整理。プロセスはOSが管理する独立した実行単位で、それぞれ別のメモリ空間を持ちます(スレッドより独立性が高い)。Python は GIL(Global Interpreter Lock、同時に1スレッドしか Python バイトコードを実行できない仕組み)があるため、CPU 重い処理の並列化はスレッドでなくプロセスで行うのが定石です。これがワーカーを「別プロセス」で立てる理由です。逆に I/O 待ち(ディスク読み込みなど)は GIL を解放するのでスレッドでも並列化できますが、デコードのような CPU 計算は別プロセスでないと並列化できません。

## 仕組みを詳しく
深層学習フレームワークの DataLoader に `num_workers=N` を渡すと、N 個の別プロセス(ワーカー)が立ち上がります。各ワーカーは独立にデータ準備(読み込み・デコード・前処理)を行い、できたバッチをキュー(待ち行列)に積みます。メインプロセス(GPU を回す側)は、キューから出来上がったバッチを取り出すだけで済みます。

時間軸で見ると、GPU が計算する裏でワーカーたちが次のバッチを並列に作るので、こうなります。

```
GPU:    [計算 b0][計算 b1][計算 b2][計算 b3]...
worker: [作る b1][作る b2][作る b3][作る b4]...  (GPU の計算と重なって進む)
        → GPU は待たずに次々バッチを受け取れる
```

prefetch(先読み)は「ワーカーが、今要求されているより先のバッチまで前もって作っておく」ことです。多くの実装は `prefetch_factor`(各ワーカーが前もって用意するバッチ数、よく既定値 2)で制御します。`num_workers=4, prefetch_factor=2` なら最大 4×2=8 バッチ分が先回りで準備されます。これにより、たまたまデータ準備が重いバッチが来ても、貯めておいた在庫で GPU を止めずに済みます。先読みは「準備時間のばらつき(分散)」を吸収する緩衝在庫の役割です。平均では足りていても、ときどき遅いバッチがあると GPU が谷待ちするので、在庫で平準化します。

`num_workers` の選び方の目安。

- `num_workers=0`: ワーカーを使わず、メインプロセスがデータ準備もやる。最も遅いがデバッグは簡単(エラーが追いやすく、スタックトレースが素直)。テストコードでよく使う。
- `num_workers=4` 前後: 一般的な出発点。CPU コア数やデータの重さに応じて調整。
- 多すぎると逆効果: ワーカーはメモリを食い、プロセス間でデータをやり取りするオーバーヘッド(シリアライズ・キュー転送)もある。物理 CPU コア数を超えて増やしても速くならず、メモリ不足や起動の遅さで悪化することがある。

数値例。1バッチのデータ準備に CPU で 80ms、GPU 計算に 100ms かかるとします。

```
num_workers=0: 1ステップ = 80 + 100 = 180ms
num_workers≥2: 準備(80ms)が計算(100ms)の裏に完全に隠れる
            → 1ステップ ≈ 100ms(GPU 計算律速)
=> 約 1.8 倍の高速化
```

逆に、データ準備が 200ms で計算が 100ms の場合、ワーカー1人では準備が追いつかず GPU が待ちます。ワーカーを2人にして並列に作れば、平均で 100ms ごとに1バッチ供給でき GPU を止めずに済みます。つまり「準備時間 ÷ 計算時間」を切り上げた数以上のワーカーが要る、というのが目安です(この例では 200/100 = 2)。これを超えてワーカーを増やしても、GPU 律速になっているので速度は頭打ちです。

注意点。ワーカーは別プロセスなので、起動コスト(最初の数秒、特に毎 epoch で再起動する設定では効く)、データセットの状態共有の難しさ(各ワーカーがデータセットのコピーを持つ)があります。乱数の扱いにも罠があり、ワーカーごとにシードを設定しないと全ワーカーが同じ乱数(同じ data augmentation)を出してしまうため、ワーカー初期化関数でシードをずらす必要があります(古い実装でこの罠は実際に多くの論文・コードで報告されました)。永続ワーカー(persistent_workers)を有効にすると epoch ごとの再起動コストを避けられますが、ワーカーがデータセットの状態を抱え続ける点に注意が要ります。

プロセス起動方式の罠もあります。fork で子プロセスを作る場合は親のメモリをコピーオンライトで共有でき軽いですが、Python オブジェクトの参照カウントが書き込まれるたびにページがコピーされ、巨大なリスト(ファイルパス一覧など)を Dataset が持っているとワーカー数ぶんメモリが膨らむ現象が起きます。対策として、ファイル一覧を Python リストでなく NumPy 配列やテンソルで持つとコピーオンライトが効きやすい、という実務知が知られています。

データ形式との関係。多数の小さなファイルをランダムに読むより、データを多数の tar アーカイブ(shard)にまとめてストリーミングで読む WebDataset 形式の方が、ディスク/ネットワーク効率が良い場面が多いです。小さなファイルのランダムアクセスはシーク(ディスクヘッドの移動)が多発して遅く、連続した大きなファイルの順次読みは帯域を活かせるからです。shard を順次読みつつ num_workers で並列化し、シャッフルはバッファ内で行う(buffer shuffle)、という設計が大規模学習で広く使われます。

## 手法の系譜と主要論文
DataLoader の prefetch / num_workers 自体は論文の「手法」ではなく、深層学習フレームワークの標準実装機能です。背景の考え方はコンピュータ工学のパイプライン化とプリフェッチです。ただしデータ供給がボトルネックになる問題は、近年の論文で正面から論じられています。

- PyTorch: An Imperative Style, High-Performance Deep Learning Library(Paszke et al., NeurIPS 2019)。マルチプロセス DataLoader を含む PyTorch の設計を記述。命令的(define-by-run)スタイルと性能の両立を目指した。
- tf.data: A Machine Learning Data Processing Framework(Murray et al., VLDB 2021, Google)。学習の入力パイプラインが GPU/TPU を遊ばせる主要因になりうると指摘し、並列読み込み・プリフェッチ・キャッシュ・自動チューニング(AUTOTUNE)を備えた設計を提示。動機は「アクセラレータが速くなるほどデータ供給が相対的ボトルネックになるから」。
- Analyzing and Mitigating Data Stalls in DNN Training(Mohan, Phanishayee, Raniwala, Chidambaram, VLDB 2021, Microsoft)。実際の学習で「データ供給の停滞(data stall)」がどれだけ時間を食うかを大規模に計測した研究。多くのワークロードで学習時間の相当割合がデータ待ちだと示し、キャッシュ・前処理オフロードの効果を定量化した。
- FFCV: Accelerating Training by Removing Data Bottlenecks(Leclerc et al., CVPR 2023, MIT)。独自バイナリフォーマットと高速デコード・JIT コンパイルされたパイプラインで、データ供給を大幅に高速化したと報告。num_workers の延長線上にある「準備工程そのものを速くする」アプローチ。

## 論文の実験結果(定量データ)
- Mohan et al.(VLDB 2021)は、複数の画像分類モデル(ResNet 等)の学習で、ストレージや前処理が律速になると GPU 稼働率が下がり、ある構成では学習時間の数十%(報告で最大 65% 程度に達するケースもある)がデータ準備待ち(data stall)に費やされると報告しました。指標は「GPU 稼働率」と「1 epoch 所要時間」。前処理を並列化しキャッシュ(MinIO ベースの分散キャッシュ等)を効かせると、待ち時間を大きく削減できることを示しています。これは「num_workers を増やす/キャッシュする」効果を定量化したものと読めます。
- tf.data(VLDB 2021)は、prefetch とパイプラインの並列化・キャッシュを組み合わせることで、入力パイプライン由来のアイドルを減らしアクセラレータ稼働率を上げられると報告。AUTOTUNE で並列度を動的調整すると、専門家による手動チューニングに近い性能を自動で得られることも示しました。指標は「スループット(steps/sec)」と「アクセラレータ稼働率」。
- FFCV(CVPR 2023)は、ImageNet の ResNet 学習などで、標準的な入力パイプラインに比べ学習時間を大幅に短縮(複数 GPU 構成で数倍規模の高速化)したと報告。データ供給が律速だった部分を解消した分が、そのまま総時間の短縮になることを示しています。
- アブレーション的観点(一般に観測される傾向): num_workers を 0→数個に増やすとスループットが大きく上がり、ある点(おおむね物理コア数や準備/計算比で決まる飽和点)を超えると頭打ちか悪化する。prefetch_factor を増やすとバッチごとの準備時間のばらつきを吸収でき GPU の谷待ちが減るが、メモリ消費が増える。pin_memory を併用すると CPU→GPU 転送が計算に隠れ、さらに稼働率が上がる。

指標の意味。GPU 稼働率は「GPU が実際に計算していた時間の割合」で、低いと高価な GPU を遊ばせている。スループットは「1秒あたり何ステップ/サンプル処理できたか」で、学習の総所要時間に直結します。データローディングはこれらを底上げするための土台で、モデル本体を1行も変えずに学習を速くできる「ただ飯」に近い最適化です。

## メリット・トレードオフ・限界
メリット
- データ準備を GPU 計算の裏に隠せるので、GPU の待ち時間が減り学習スループットが上がる。
- 設定が簡単(`num_workers` を渡すだけ)で、モデル本体を書き換えずに高速化できる。
- 重いデコード処理(動画・多カメラ画像)を並列化でき、自動運転のような重いデータで効果が大きい。
- prefetch によりバッチごとの準備時間のばらつきを吸収できる。

トレードオフ・限界
- ワーカーを増やすほどメモリ消費が増える(各ワーカーがデータセットのコピーやバッファを持つ。fork のコピーオンライト崩壊にも注意)。
- 物理 CPU コア数を超えて増やしても速くならず、起動コストやプロセス間通信のオーバーヘッドでむしろ遅くなることがある。
- 別プロセスのためエラーが追いにくく、デバッグ時は `num_workers=0` に戻す必要がある。ワーカー内の例外がメインに伝わりにくい。
- ワーカーごとの乱数シード設定を怠ると、全ワーカーが同じ拡張をかける不具合が出る(データ多様性の低下、再現性の罠)。
- データ準備が十分速い場合は効果が小さい(ボトルネックが計算側なら意味が薄い)。
- ストレージ I/O 自体が遅いと、ワーカーをいくら増やしてもディスク帯域で頭打ちになる(その場合はキャッシュやシャード化が必要)。並列化は「CPU 計算が律速」のときに効き、「I/O が律速」のときは別の打ち手が要る。

## 発展トピック・研究の最前線
- GPU 前処理(NVIDIA DALI)。デコードや前処理自体を GPU で行い、CPU の負担と CPU→GPU 転送量を減らす。CPU が律速のときに有効。
- 自動チューニング。tf.data の AUTOTUNE のように並列度・バッファ量を実行時に自動最適化する。手動で num_workers を当てる作業を機械化する方向。
- データのキャッシュ/事前抽出。重いデコードを毎 epoch 繰り返さず、一度デコードしてテンソルやシャードで保存しておく(事前抽出, pre-extraction)。学習を CPU 律速から I/O 律速へ移し、num_workers の効きを良くする。
- 分散学習でのデータ分割(sharding)。複数 GPU/ノードで重複なくデータを配る DistributedSampler 的な仕組みと、ノード間 I/O の最適化。各ランクが別の shard を読むことで I/O も分散させる。
- ストリーミングデータセット(WebDataset, FFCV, MosaicML Streaming 等)。大規模データをクラウドストレージから直接ストリーミングし、ローカルディスクに収まらない規模でも num_workers で並列供給する。
- pinned memory + 非同期転送との併用。num_workers で「作る」を並列化したうえで、転送段も計算の裏に隠すと供給パイプライン全体が GPU 計算と重なる。I/O・デコード・前処理・H2D 転送の各段をすべて計算に重ねるのが理想形。
