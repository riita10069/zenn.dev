# SwinV2特徴マップの次元並べ替え

## ひとことで言うと

画像から特徴を取り出す部品(バックボーン)は、種類によって出力データの「軸の並び順」が違います。Transformer をベースにした Swin 系のバックボーンは、特徴を `[枚数, 高さ, 幅, チャンネル]` の順で出すことが多く、畳み込みをベースにした ConvNeXt 系は `[枚数, チャンネル, 高さ, 幅]` の順で出します。後段のモジュールはたいてい後者の並び順を期待するので、Swin 系の出力だけは軸を並べ替え(permutation、ペアミュテーション)してから渡す必要があります。この「並び順の違いをどう吸収するか」が本トピックの主題です。一見すると些細なデータ整形の話ですが、複数のバックボーンを差し替え可能にする設計や、メモリレイアウトに起因する性能問題まで関わる、奥の深いテーマです。

## 直感的な理解

身近なたとえで考えます。引っ越しのとき、家具を「部屋ごと」にまとめる人もいれば、「種類ごと(本は本、服は服)」にまとめる人もいます。どちらも同じ家具を運んでいますが、箱の詰め方が違います。受け取る側(新居の片付け係)がどちらかの詰め方を前提に動いていると、別の詰め方で届いた箱は一度開けて詰め直さなければなりません。

ニューラルネットの特徴マップもまったく同じです。中身の数値はどちらのバックボーンでも「画像のこの位置に、こういう特徴がある」という同じ情報を表していますが、その数値をメモリに並べる順序が違います。受け取る後段のモジュール(複数カメラ画像を統合する融合モジュールや、鳥瞰図に投影するモジュールなど)が片方の順序を前提に書かれていると、もう片方の順序で来たデータはそのままでは使えません。だから「詰め直す」=並べ替える必要があるのです。

ここで重要なのは、なぜそんな面倒が起きるのかという点です。バックボーンを1種類に固定すれば並べ替えは一切不要です。しかし研究・開発では「Swin と ConvNeXt と ResNet のどれが一番性能が出るか」を比較したいことがよくあります。バックボーンを差し替えられるようにすると、どうしても出力フォーマットの違いに直面します。この違いをスマートに吸収する仕組みがあれば、後段のコードを一切変えずにバックボーンだけ取り替えて実験できます。

## 基礎: 前提となる概念

用語を一つずつ噛み砕きます。

- バックボーン (backbone): 入力画像から「特徴」を取り出す部品です。生の画像は情報量が多すぎて直接扱いにくいので、まず「ここに車らしきものがある」「ここは道路だ」といった抽象的な手がかりに変換します。この変換器をバックボーンと呼びます。背骨のようにモデル全体の土台になるからこの名前です。代表例が ResNet(畳み込み)、Swin Transformer(注意機構)、ConvNeXt(注意機構の設計を取り入れた畳み込み)です。

- 特徴マップ (feature map): バックボーンが出力する、特徴を縦横のグリッド状に並べたデータです。たとえると、元の画像を粗いマス目に区切り、各マスに「そのマスに何があるか」を表す数値の束を詰めたものです。

- テンソル (tensor): 多次元の数値の箱です。1枚のカラー画像は「高さ × 幅 × 色(RGB)」の3次元、複数枚をまとめて処理する(バッチ処理)ときはさらに「枚数」の軸が加わって4次元になります。

- チャンネル (channel): 各マスに何種類の特徴が詰まっているかの本数です。たとえば C=96 なら、各マスに96個の数値が入っています。最初の入力画像なら C=3(赤・緑・青)ですが、バックボーンを通ると C は数百に増えます。

ここで本題の「軸の並び順」です。4次元の特徴テンソルには代表的に2つの並べ方があります。

- channels-first (NCHW): `[バッチ数 B, チャンネル数 C, 高さ H, 幅 W]` の順。PyTorch の畳み込み演算 Conv2d が標準で期待する並び順です。N は number(枚数)の頭文字です。
- channels-last (NHWC): `[バッチ数 B, 高さ H, 幅 W, チャンネル数 C]` の順。チャンネルが最後に来ます。

なぜバックボーンによって並び順が違うのか。これは内部構造の違いに由来します。

- Swin 系(Swin Transformer / SwinV2 など): 内部が Transformer です。Transformer は画像を小さなパッチ(断片)に分割し、それを「トークンの系列」として扱います。系列の各要素はベクトル(チャンネルの束)なので、データを `[B, トークン数, C]` の形で持つのが自然で、空間に戻すと `[B, H, W, C]` すなわち channels-last になりがちです。

- ConvNeXt 系・ResNet 系: 内部が畳み込みです。PyTorch の畳み込みは入力を `[B, C, H, W]` の channels-first で扱う設計なので、出力も自然に channels-first になります。

後段の融合モジュールが channels-first を前提にしているなら、Swin の channels-last 出力はそのままでは渡せません。この軸の並べ替えを permutation(順列・置換)と呼びます。

## 仕組みを詳しく

素朴な実装は「バックボーンが Swin なら並べ替える、ConvNeXt ならそのまま」とバックボーン名で分岐することです。しかしこれは脆い設計です。新しいバックボーンを足すたびに条件分岐を書き足す必要があり、ライブラリのバージョンが上がって出力フォーマットが変わると壊れます。

より堅牢な設計は、バックボーン名ではなく実際のテンソル形状を見て自動判定することです。考え方はこうです。各ステージ(段階)の「正しいチャンネル数」を事前に知っておき、forward(順伝播)のときに「そのチャンネル数が軸のどこにあるか」を見て、最後の軸にあれば channels-last だと判断して並べ替える、というものです。

### ステップ1: 構築時に正しいチャンネル数を記録する

バックボーンを初期化する時点で、各ステージのチャンネル数を記録します。多くの画像モデルライブラリ(例: timm)は、各ステージのチャンネル数を教えてくれるメタ情報(feature_info のような属性)を持っています。読めない場合は、ダミー入力(たとえば `torch.zeros(1, 3, 256, 256)`)を一度通し、出力テンソルの形から推測します。

例えば SwinV2 Tiny なら4ステージで、チャンネル数は `[96, 192, 384, 768]` のようになります(ステージが進むほど空間は粗くチャンネルは多くなる、という階層的な構造です)。

### ステップ2: forward 時に形状を見て並べ替える

擬似コードで書くと次のようになります。

```python
for f, expected_c in zip(features, feature_channels):
    if f.dim() == 4 and f.shape[1] != expected_c and f.shape[-1] == expected_c:
        out.append(f.permute(0, 3, 1, 2))   # NHWC -> NCHW
    else:
        out.append(f)                       # 既に NCHW なのでそのまま
```

判定ロジックを噛み砕きます。「期待チャンネル数 `expected_c` が2番目の軸(`f.shape[1]`)に無くて、最後の軸(`f.shape[-1]`)にあるなら、それは channels-last なので並べ替える」という意味です。

具体例で見ます。第1ステージの出力で `expected_c = 96` の場合:

- SwinV2 の出力: `[B, 64, 64, 96]`(channels-last)
  → `f.shape[1] = 64 != 96`、かつ `f.shape[-1] = 96 == 96` なので条件成立
  → `permute(0, 3, 1, 2)` で `[B, 96, 64, 64]` に並べ替え
- ConvNeXtV2 の出力: `[B, 96, 64, 64]`(channels-first)
  → `f.shape[1] = 96 == 96` なので条件不成立
  → そのまま `[B, 96, 64, 64]`

`permute(0, 3, 1, 2)` は「0番目の軸はそのまま、3番目の軸を1番目の位置に、元の1番目を2番目に、元の2番目を3番目に持ってくる」という軸の並べ替え命令です。

ここで深い注意点があります。permute は論理的には軸の見方を変えるだけで、メモリ上の数値を実際に動かしはしません(strides という「各軸を1つ進むのに何要素飛ぶか」の情報だけを書き換えます)。つまり permute 自体は速いです。しかし、その直後に「メモリが連続していること」を要求する演算(多くの畳み込みや view など)が来ると、PyTorch は内部的に `.contiguous()` を呼んでメモリを物理的に並べ替えてコピーします。このコピーが実際のコストです。大きな特徴マップ(高解像度のステージ)では、このコピーがメモリ帯域を消費し、わずかに速度低下を招きます。

### 図でのイメージ

```
SwinV2:     [B, H, W, C]  --permute(0,3,1,2)-->  [B, C, H, W]  --> 後段モジュール
ConvNeXtV2: [B, C, H, W]  --(そのまま)-------->  [B, C, H, W]  --> 後段モジュール
```

数値で並べると、256×256 入力での SwinV2 Tiny の各ステージは概ね次のようになります。

```
ステージ1: [B, 64, 64, 96]  -> [B,  96, 64, 64]
ステージ2: [B, 32, 32, 192] -> [B, 192, 32, 32]
ステージ3: [B, 16, 16, 384] -> [B, 384, 16, 16]
ステージ4: [B,  8,  8, 768] -> [B, 768,  8,  8]
```

## 手法の系譜と主要論文

このトピックの背景には、Vision Transformer と畳み込みネットワークの2系統の進化があります。

- ViT (Dosovitskiy et al., ICLR 2021, arXiv:2010.11929): 「An Image is Worth 16x16 Words」。画像を16×16画素のパッチに分割し、各パッチを1つのトークンとして Transformer に入れる発想を確立しました。これにより画像も自然言語と同じように「トークンの系列」として扱えるようになり、出力が `[B, トークン数, C]` 形式になる流れができました。

- Swin Transformer (Liu et al., ICCV 2021, arXiv:2103.14030, Marr Prize 受賞): ViT は画像全体で注意を計算するため計算量が画素数の2乗で爆発する問題がありました。Swin は画像を小さなウィンドウに区切ってウィンドウ内だけで注意を計算し、ウィンドウを少しずつずらす(shifted window)ことで階層的な特徴を作ります。これで計算量を画素数に比例まで抑えつつ、4段階の階層特徴を得られ、検出やセグメンテーションで強さを発揮しました。出力は Transformer 由来で系列状(channels-last)になりがちです。

- Swin Transformer V2 (Liu et al., CVPR 2022, arXiv:2111.09883): V1 を大規模化・高解像度化できるよう改良しました。学習時より高い解像度で使うと位置の表現が壊れる問題を、log間隔の連続位置バイアス(log-spaced continuous position bias)で解決し、低解像度で学習した位置の知識を高解像度に転移できるようにしました。さらに残差後正規化(post-norm)とスケールドコサイン注意で大規模学習を安定化させました。

- ConvNeXt (Liu et al., CVPR 2022, arXiv:2201.03545): 「Transformer の良い設計(大きなカーネル、LayerNorm、GELU、逆ボトルネックなど)を畳み込みネットに移植すれば、注意機構なしでも Swin 並みの性能が出る」ことを示しました。畳み込みは実装が単純で高速、かつ channels-first のまま素直に扱えます。

- ConvNeXt V2 (Woo et al., CVPR 2023, arXiv:2301.00808): マスク自己教師あり学習(FCMAE)に適合させ、Global Response Normalization(GRN)というチャンネル間競合を促す正規化を追加しました。マスク再構成と相性の良い畳み込みネットとして完成度を高めました。

この系譜から分かるのは、channels-last か channels-first かは性能の優劣ではなく、内部構造(Transformer か畳み込みか)に由来する出力フォーマットの違いだということです。

## 論文の実験結果(定量データ)

各バックボーンの強さは ImageNet-1K(1000クラス・約128万枚の画像分類ベンチマーク)での Top-1 精度(最も確信したクラスが正解と一致する割合)でよく比較されます。報告値の代表例は次のとおりです。

- Swin-T(Tiny、約28Mパラメータ、224解像度): ImageNet-1K Top-1 で約81.3%。同規模の ResNet-50(約76%)を大きく上回りました。
- ConvNeXt-T(約28Mパラメータ、224解像度): ImageNet-1K Top-1 で約82.1%。同規模の Swin-T をわずかに上回り、「畳み込みでも注意機構に勝てる」ことを示しました。
- SwinV2: 巨大モデル SwinV2-G(約30億パラメータ)を高解像度で学習し、ImageNet-V2 などで当時の最高精度を更新。重要なのは絶対精度より「低解像度(192など)で事前学習し高解像度(384〜512など)へ転移しても劣化が小さい」という解像度転移の安定性です。
- ConvNeXt V2: GRN を入れた ConvNeXt V2-Huge は FCMAE 事前学習併用で ImageNet-1K Top-1 約88.9%に到達。Atto(約3.7M)という極小モデルまで一貫したスケーリングを示しました。

これらの数値は「どのバックボーンを差し替え候補にすべきか」を判断する材料になります。同規模なら ConvNeXt が Swin にわずかに勝つ報告が多い一方、Swin V2 は高解像度への転移しやすさという別の強みを持ちます。並べ替えの観点では、Swin を候補に含めるかぎり channels-last 対応は避けられません。

定量的なオーバーヘッドの感覚として、permute 後に contiguous コピーが発生する場合、そのコストは特徴マップの要素数(B×C×H×W)に比例します。高解像度の浅いステージほど H×W が大きくコピーが重くなるため、ボトルネックは深いステージより浅いステージ側に出やすい点も覚えておくと有用です。

## メリット・トレードオフ・限界

メリット

- バックボーン名のハードコードを避けられ、新しいバックボーンを足すだけで使えます。後段モジュールは一切変更不要です。
- channels-first / channels-last のどちらのバックボーンでも、後段は常に `[B, C, H, W]` を受け取れます。
- 構築時に1回だけチャンネル数を確定するので、forward のたびに重い判定をする必要がありません。

トレードオフ・限界

- permute 自体は軽いものの、後続演算が連続メモリを要求すると暗黙の `.contiguous()` でコピーが発生し、メモリ帯域と時間を消費します。高解像度ステージで顕著です。
- 判定ロジックは「期待チャンネル数が `shape[1]` か `shape[-1]` のどちらにあるか」に頼るため、H や W がたまたまチャンネル数と同じ値になる極端な入力では誤判定の余地があります(正方形かつチャンネル数と同じ辺長など)。実用上ほぼ起きませんが、限界として理解しておくべきです。
- 4次元以外の特徴(系列のまま `[B, N, C]` を返すバックボーン)には、この4D前提の判定はそのまま使えません。
- 研究上の未解決テーマとして、channels-last レイアウトを最後まで貫いて Tensor Core を最大活用する最適化(NHWC のまま畳み込みを回す)と、後段が channels-first を前提にする既存資産との折り合いがあります。理論上は全体を channels-last に統一する方が高速ですが、エコシステム全体の対応が必要で、現実には混在しがちです。

## 発展トピック・研究の最前線

- メモリレイアウト最適化: PyTorch には `to(memory_format=torch.channels_last)` があり、畳み込みを NHWC のまま実行して NVIDIA Tensor Core の利用効率を上げられます。Swin の自然な出力フォーマットと相性が良く、混合精度学習と組み合わせると速度向上が見込めます。バックボーン横断のパイプラインでは「どこで一度だけレイアウトを揃えるか」が設計上の論点になります。
- バックボーン統一の動き: 近年は SwinV2 や ConvNeXt のような畳み込み・注意のハイブリッド、あるいは Hiera(Ryali et al., 2023)のように階層を簡素化した ViT 系が登場し、「階層特徴を出すが内部はシンプル」という方向に進んでいます。出力フォーマットの多様性は今後も残るため、形状ベースの吸収層は引き続き有用です。
- 自己教師あり事前学習との接続: ConvNeXt V2 の FCMAE や I-JEPA のように、バックボーンの事前学習方法が下流性能を大きく左右します。どのバックボーンを選ぶかは、出力フォーマットだけでなく「どの事前学習重みが使えるか」も含めて検討する研究領域です。
