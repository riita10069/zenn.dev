# ViewAttentionPool (マルチビュー集約)

## ひとことで言うと
車のまわりを囲む複数のカメラ(典型的には前後左右で6〜7台)それぞれから取り出した画像特徴を、1個の「学習可能な質問ベクトル(クエリ)」が cross-attention(交差注意、後述)で見比べて、1本の「いまこの瞬間のシーン要約ベクトル」に圧縮する軽量なモジュールです。どのカメラを今どれくらい重視するかを、固定の平均ではなく、状況に応じて学習で変えられるのが核心です。集合(順序のないデータ)を学習可能な注意で要約するという、機械学習の一般的な道具をマルチカメラ融合に当てはめたものと言えます。

## 直感的な理解
人間が運転中に車線変更するとき、頭は360度を均等に見ているわけではありません。右に寄せたい瞬間は右サイドミラーと右後方に注意が集中し、前方は流し見になります。合流が終われば注意は前方へ戻ります。つまり「どの方向をどれくらい見るか」は固定ではなく、刻々と変わる重み付けです。

機械に同じことをさせたいとします。各カメラから出てくる特徴を、毎回ぜんぶ単純平均してしまうと、右後方の重要な情報が前方の平凡な情報に薄められてしまいます。逆に全部を横に連結して保持すると、情報は残りますが入力が肥大化し、しかも「右カメラは何番目」という順番に縛られて、カメラ台数が変わると使えなくなります。

ここで欲しいのは「平均より賢く、連結より軽く、台数変化に強い」集約です。その素直な道具が、注意機構(attention)です。1本のクエリベクトルが「あなたたちカメラ群よ、今この判断に必要な情報を持っているのは誰?」と問いかけ、各カメラとの相性スコアを計算して、重要なカメラに大きな重みを置いた加重平均をとる。重みは入力ごとに変わるので「いまは右後方、次の瞬間は前方」と動的に切り替わります。これがマルチビューを注意で集約するという発想です。

## 基礎: 前提となる概念
前提知識ゼロの人向けに、必要な道具を順に噛み砕きます。

特徴マップ (feature map)。画像をCNNやTransformerの画像エンコーダ(backbone、骨格ネットワーク。これは画像を数値表現に変換する大きなネットワークのことです)に通すと、出てくるのは1枚の画像ではなく `[C, h, w]` という立体的な数値の塊です。これを特徴マップと呼びます。`h×w` は空間グリッド(たとえば 8×8 の64マス)で、各マスに `C` 次元(たとえば256個)の特徴ベクトルが乗っています。要は「画像のどの位置に、どんな模様・物体らしさがあるか」を数値化したものです。

埋め込み (embedding)。複雑なデータ(画像など)を、固定長の数値の並び(ベクトル)に圧縮した表現を埋め込みと呼びます。256次元の埋め込みなら、256個の数字で1つの対象を表します。

注意機構 (attention)、とくに cross-attention。注意機構は「問い合わせ役(query, クエリ)」が「情報源(key と value、キーとバリュー)」を見て、どこにどれだけ注目するかを重みで表す計算です。数式で書くと `Attention(Q,K,V) = softmax(QKᵀ/√d) V` です。これは「クエリ Q と各キー K の内積(相性スコア)をとり、次元 d の平方根で割ってスケールを整え(値が大きくなりすぎて softmax が飽和するのを防ぐため)、softmax で全部足すと1になる0〜1の重みに正規化し、その重みでバリュー V を加重平均する」という意味です。クエリとキー/バリューが別物のとき、これを cross-attention(交差注意)と呼び、自分自身に注意する self-attention と区別します。

マルチヘッド (multi-head)。注意を1種類だけでなく複数の「見方(ヘッド)」に分けて並行に計算し、最後に結合する仕組みです。たとえば4ヘッドなら、あるヘッドは色、別のヘッドは動き、というふうに異なる観点で同時に注意できます。各ヘッドは `C/h` 次元の部分空間で独立に注意を計算します。

順序不変性 / 集合 (permutation invariance / set)。カメラ群は本質的に「集合」です。順番に意味はありません(前カメラが内部的に1番目でも7番目でも同じ情報)。注意機構は順番を持たない演算なので、集合をそのまま扱えます。ただし「どのカメラか」を区別したい場合は、カメラ識別の埋め込みを別途足す必要があります。これを足さないと、注意は前カメラと後カメラを交換可能なものとして混ぜてしまいます。

## 仕組みを詳しく
典型的な軽量実装を見ます。記号は B=バッチ(同時処理する事例数)、V=カメラ台数、C=特徴次元、h/w=各カメラ特徴マップの縦横。

```python
class ViewAttentionPool(nn.Module):          # 軽量版: 1層・少ヘッド
    def __init__(self, embed_dim=256, num_views=7, num_heads=4):
        super().__init__()
        self.view_embed = nn.Parameter(torch.randn(1, num_views, embed_dim))  # カメラ識別埋め込み
        self.query      = nn.Parameter(torch.randn(1, 1, embed_dim))          # 学習可能な1クエリ
        self.attn       = nn.MultiheadAttention(embed_dim, num_heads, batch_first=True)
        self.norm       = nn.LayerNorm(embed_dim)

    def forward(self, per_view_feat, B, V):   # per_view_feat: [B*V, C, h, w]
        C   = per_view_feat.shape[1]
        tok = per_view_feat.mean(dim=(2, 3)).reshape(B, V, C)  # 各ビューを空間平均 → [B, V, C]
        tok = tok + self.view_embed                            # カメラごとの埋め込みを足す
        q   = self.query.expand(B, -1, -1)                     # [B, 1, C]
        out, _ = self.attn(q, tok, tok)                        # V個のキー(=Vカメラ)に注意
        return self.norm(out.squeeze(1))                       # [B, C] フレーム埋め込み
```

形(shape)を数値で追います。B=2, V=7, C=256, h=w=8 とします。

1. `per_view_feat` は `[B*V, C, h, w] = [14, 256, 8, 8]`。各カメラの特徴マップが14枚分(2事例×7カメラ)積まれています。
2. `mean(dim=(2,3))` で空間方向(h, w)の64マスを平均し `[14, 256]`、`reshape` で `[2, 7, 256]`。各カメラが1本の256次元トークンになりました。
3. `+ self.view_embed` で「これは何番カメラか」を区別する学習可能な埋め込みを足します。これがないと注意機構はカメラを区別できず、台数が交換可能になってしまいます。
4. クエリ `q` は `[2, 1, 256]`。1個のクエリが7個のキー/バリュー(7カメラ)に cross-attention をかけます。
5. 出力 `[2, 1, 256]` を `squeeze` して `[2, 256]`、LayerNorm で正規化(各ベクトルの平均と分散を整え、学習を安定させる)。

cross-attention の中身を一段ほどきます。クエリ q と各カメラトークンの内積で7個の相性スコアを計算し、softmax で合計1の重み(たとえば前0.4・右後0.3・残り0.3)にし、その重みでカメラトークンを加重平均します。

なぜこの設計なのか。クエリを1個にしているのは「出力が1ベクトルでよい」からです。物体検出のように複数の出力が欲しい場合はクエリを100個などにしますが(後述の DETR)、シーン要約は1本で足ります。1層・少ヘッドにしているのは、重い計算は backbone 側で済んでおり、この集約層は「数個のトークンへの注意」だけなので極めて軽いからです。注意の計算量はトークン数 V の2乗 `O(V²)` に比例しますが、V=7 なら 49 回の内積で済み、無視できる規模です。

空間情報の扱いに注意。`mean(dim=(2,3))` で各カメラ内の `h×w` を平均で潰しているため、カメラ内の「どこに何があるか」という細かい空間配置は失われます。BEV(俯瞰図)融合のような幾何的な空間理解はしません。これは「各カメラを1トークンに圧縮して軽く集約する」という割り切りの代償です。空間情報を残したい場合は、平均せず `h×w` の各マスをトークンにして V×h×w 個のキーに注意する設計もありえますが、トークン数が爆発し計算が重くなります。

同じ部品を時間軸へ転用できる。この「1クエリ + cross-attention でプール」というパターンは、カメラ軸(V個)の代わりに時間軸(N個のフレーム)にそのまま適用できます。カメラ識別埋め込みを時間位置埋め込みに置き換えれば、N個の過去フレームを1本の「視覚履歴」に集約する [temporal attention-pool (時間方向集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0477_temporal-attention-pool) になります。空間集約と時間集約に同じ基本演算を使い回せるのは設計上の美点です。

## 手法の系譜と主要論文
平均/最大プーリングの時代。古典的には特徴マップの集約は単純平均(Global Average Pooling)や最大値で行われていました。安価ですが、要素の重要度を選べません。

Set Transformer (Lee et al., ICML 2019, arXiv:1810.00825)。「集合(順序のないデータ)をどう集約するか」を真正面から扱い、PMA (Pooling by Multihead Attention) を提案しました。学習可能なシード(seed)ベクトル、すなわちクエリが集合要素に cross-attention して要約する仕組みです。「学習クエリで集合をプールする」考え方はこの直系です。動機は「平均や最大では集合内の重要要素を選べない」こと。新規性は「順序不変なまま、学習で要素の重要度を決められ、出力サイズが要素数に依存しない集約」を確立した点です。

DETR (Carion et al., ECCV 2020, arXiv:2005.12872)。物体検出を「集合予測」として定式化し、学習可能なクエリ(object queries)が画像特徴に cross-attention して物体を引き出す枠組みを作りました。クエリを1個にすればまさにマルチビュー集約と同じ形です。DETR は NMS(Non-Maximum Suppression、重複検出を除く後処理)を不要にした点で画期的でした。

Perceiver (Jaegle et al., ICML 2021, arXiv:2103.03206)。少数の潜在クエリ(latent array)で、画像・音声・点群などの大量入力を cross-attention で圧縮しました。入力サイズに対して計算量がほぼ線形になるよう、少数クエリへ情報を集約する有効性を示しました。attention-pool の一般化と見なせます。

BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)。自動運転で、学習可能な BEV クエリがマルチカメラ特徴に空間 cross-attention して俯瞰特徴を作りました。1クエリ・非幾何のマルチビュー集約は「BEVFormer の学習クエリから幾何投影(カメラの内部・外部パラメータを使った3D→2D射影)を取り除き、クエリを1個に潰した軽量版」と位置づけられます。BEV 幾何が不要で「1フレームのシーン要約」だけ欲しい場面に向きます。

PETR (Liu et al., ECCV 2022, arXiv:2203.05625)。3D座標の位置エンコーディングを画像特徴に注入し、明示的な幾何投影なしにマルチカメラの3D検出を行いました。「カメラ識別埋め込み」を「3D位置エンコーディング」に置き換えることで、軽量な注意集約に幾何情報を取り戻す方向の代表例です。

## 論文の実験結果(定量データ)
Set Transformer。集合回帰や点群分類などで、PMA は単純平均プーリングを安定して上回りました。代表的には、混合ガウス分布のクラスタ中心推定タスクや、ModelNet40 の点群分類で、平均・最大プーリングのベースラインに対して誤差・精度の改善が報告されています。要点は「集約器を学習可能にするだけで、同じエンコーダでも下流性能が上がる」こと。これが attention pooling を使う一般的な動機になっています。

DETR。COCO 物体検出で、当時の標準だった Faster R-CNN と同等の AP(Average Precision、検出精度の総合指標で高いほど良い)を達成しました。論文では ResNet-50 バックボーンで AP 42前後を報告し、大きな物体では Faster R-CNN を上回りました(クエリが画像全体を見るため、文脈が効く)。一方で小さい物体は苦手でした(後の Deformable DETR が改善)。「学習可能クエリ+cross-attention で集合を引き出す」設計が、検出という難しいタスクで通用することを実証した意義は大きいです。

Perceiver。ImageNet 分類で、画像をピクセル列として(畳み込みの強い帰納バイアスなしで)入力しても、ViT に迫る精度(報告でおよそ top-1 78%前後)を出しました。少数の潜在クエリへ集約しても情報が大きく失われないことを示しました。

BEVFormer。nuScenes データセットで、当時のカメラのみ手法の最高水準を更新しました。代表値として NDS(nuScenes Detection Score、検出と速度・属性推定を総合した指標、高いほど良い)でおよそ0.56、mAP でおよそ0.42前後が報告されています。とくに時間的注意を加えると速度推定誤差 mAVE が大きく改善し、単一フレーム比で NDS が数ポイント向上しました。アブレーションでは、空間 cross-attention(マルチカメラ融合)と時間 self-attention(過去フレーム融合)のどちらを抜いても性能が落ち、両方が寄与することを示しました。1クエリのマルチビュー集約はこの空間 cross-attention 部分を、幾何を捨てて極限まで軽くしたものに相当します。

これらの数値が伝えるのは「学習可能な注意プーリングは、単純な平均/連結より下流性能を上げうるが、その効き目はタスク・データ量に依存する」という研究上の経験則です。

## メリット・トレードオフ・限界
メリット
- カメラごとの重要度を状況に応じて学習でき、単純平均より賢い集約ができる。
- V個のトークンへの注意だけなので計算が極めて軽い(クエリ1個・1層なら無視できるコスト)。
- カメラ識別埋め込みにより、順番に縛られず台数変化にも比較的柔軟。
- 同じ attention-pool を時間軸にも転用でき、空間・時間で設計が統一される。

トレードオフ・限界・未解決課題
- 空間平均(mean over h,w)で各カメラを1トークンに潰すため、カメラ内の細かい空間情報は失われる。BEV のような幾何的な空間理解(メートル単位の物体配置の復元)はできない。
- 学習クエリ・カメラ埋め込みはランダム初期化なので、データが少ないと意味のある重みが学習されにくい。最悪、クエリが入力を無視して定数を出す「崩壊(collapse)」が起きうる。
- 1クエリ・1層・少ヘッドの軽量版は、表現力が幾何ベースの重い融合より低い。安さと表現力はトレードオフ。
- 注意の重みが妥当か(本当に右折時に右カメラを見ているか)は自動では保証されず、可視化や定量評価で別途確かめる必要がある。これは注意機構全般の解釈性の課題です。

## 発展トピック・研究の最前線
- 複数クエリへの拡張。1クエリを数個〜数十個に増やすと、シーンを複数の側面(前方の流れ・周囲の障害物・路面など)に分けて要約できます。DETR や Perceiver の設計がそのまま参考になります。
- 幾何との融合。非幾何の集約にカメラの内部・外部パラメータ(キャリブレーション)を位置埋め込みとして注入すると、軽さを保ちつつ幾何情報を一部取り戻せます。PETR 系(arXiv:2203.05625)の3D位置エンコーディングがこの方向です。
- 崩壊対策。注意プーリングの学習安定化には、補助損失(再構成や対比学習)、温度パラメータの調整、クエリの正則化、ウォームアップなどが研究されています。
- 効率化。トークン数が大きい場合(全空間マップを保持する場合)は、線形注意(linear attention)や Perceiver 式のボトルネックで計算量を抑える流れがあります。
- センサー融合への拡張。カメラだけでなく LiDAR や Radar のトークンも同じクエリに食わせる、マルチモーダル注意集約が研究されています。各センサーに識別埋め込みを与えれば、同じ枠組みで異種センサーを統合できます。
