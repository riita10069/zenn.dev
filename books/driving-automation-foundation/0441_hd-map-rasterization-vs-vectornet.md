# 地図表現: ラスタ化 vs ベクトル(VectorNet)

## ひとことで言うと

自動運転で「車線がどこにあるか」「横断歩道がどこか」といった地図情報をニューラルネットに食わせるとき、地図を画像(ピクセルの集まり)に描いて渡す「ラスタ化(rasterization)」と、地図を折れ線(点をつないだ線の集合)としてそのまま渡す「ベクトル表現(vectorized representation)」という2つの大きな流派があります。VectorNet は後者を代表する手法で、地図を画像に変換せず、線の幾何構造をそのままグラフとして扱うことで、効率と精度を大きく改善しました。

## 直感的な理解

地図には「車線中心線」「停止線」「横断歩道」「交差点の境界」といった要素があり、これらはほとんどが**折れ線(polyline)**、つまり点を順につないだ線で表せます。自動運転車は、自分や周囲の車が「これからどう動くか」を予測するために、この地図と過去の動きの履歴を手がかりにします。問題は「この地図情報をどうやってニューラルネットに入力するか」です。

**ラスタ化のやり方**: 地図を1枚の鳥瞰図(BEV, Bird's-Eye-View、真上から見下ろした図)の画像として描きます。車線は緑、横断歩道は青、他車は赤い箱、というように色やチャンネルで塗り分け、たとえば 400×400 ピクセルの多チャンネル画像にします。そしてこの画像を CNN(畳み込みニューラルネット、画像認識の定番)に通します。人間が地図を「絵」として見るのに近い、直感的なやり方です。ChauffeurNet (Waymo, 2018) がこの代表です。

**ベクトル表現のやり方**: 地図をわざわざ絵にせず、「車線A = (x1,y1)→(x2,y2)→(x3,y3) という点の列」という幾何データのまま扱います。各折れ線を小さなベクトルの断片に分け、それをグラフのノードとしてニューラルネットに渡します。VectorNet (Waymo, CVPR 2020) がこれを確立しました。

たとえるなら、ラスタ化は「住所を地図に印を付けて渡す」、ベクトルは「住所を緯度経度の数値で渡す」やり方です。前者は人間に分かりやすいが解像度の限界があり広い範囲を描くと粗くなる、後者は精密でコンパクトだが扱うアルゴリズムが少し高度になります。

## 基礎: 前提となる概念

**HD地図(High-Definition Map)**: センチメートル精度の高精細地図で、車線の中心線・境界線・信号・標識・横断歩道などを正確な幾何情報として持ちます。自動運転の予測・計画はこのHD地図を強く頼りにします。

**ポリライン(polyline)**: 点を順番につないだ折れ線。車線中心線は「等間隔にサンプリングした座標点の列」として表せます。たとえば1本の車線を `[(0,0), (1,0.1), (2,0.3), (3,0.6)]` の4点で表す、というイメージです。

**CNN と畳み込み**: 畳み込みニューラルネットは、画像の小さな窓を少しずつずらしながら同じフィルタを適用して特徴を抽出します。画像のような「格子状に整然と並んだデータ」には強いですが、入力が画像であることを前提とします。

**GNN(グラフニューラルネット)**: ノード(点)とエッジ(つながり)からなるグラフを処理するネットワーク。「隣接するノードの情報を集めて自分を更新する」ことを繰り返します。折れ線や物体どうしの関係のような、格子状でないデータに向いています。

**軌道予測(trajectory prediction)/モーション予測**: 周囲の車・歩行者が数秒後にどこにいるかを予測するタスク。これがラスタ vs ベクトル論争の主戦場です。

## 仕組みを詳しく

### ラスタ化アプローチの中身

地図と各エージェント(車・歩行者)を BEV 画像のチャンネルに描きます。典型的には次のようなテンソルになります。

```
入力テンソル: [C, H, W] = [チャンネル数, 400, 400]
  チャンネル0: 車線中心線を描いた2値画像
  チャンネル1: 道路境界
  チャンネル2: 横断歩道
  チャンネル3: 信号の状態(色で表現)
  チャンネル4〜: 過去Tフレーム分の他車の位置(時刻ごとに別チャンネル)
```

1ピクセルが現実の何メートルに対応するか(解像度)を決め、たとえば 0.5m/pixel なら 400px で 200m 四方をカバーします。これを CNN(ResNet など)に通し、出てきた特徴マップから各車の未来軌道を予測します。

利点は、CNN という成熟した道具がそのまま使え、地図もエージェントも統一的に「画像」として扱える点です。欠点は次の通りです。第一に**解像度と範囲のトレードオフ**で、遠くまで見たい(範囲を広げる)と1ピクセルあたりの距離が粗くなり、細い車線が潰れます。第二に**計算の無駄**で、画像の大半は空白(道路でない場所)なのに、CNN は全ピクセルを律儀に計算します。第三に**情報の損失**で、連続的な座標をピクセルに丸める(離散化する)ため、センチ精度のHD地図の精密さが失われます。

### VectorNet の中身(ベクトルアプローチ)

VectorNet は「全部ベクトルのまま扱う」と決めました。流れは2段階の階層グラフです。

**第1段(ポリラインサブグラフ)**: 各折れ線(1本の車線、1台の車の軌跡など)を、小さなベクトル断片の列に分けます。1つのベクトル断片は

```
v_i = [始点(xs, ys), 終点(xe, ye), 属性(線種など), このポリラインのID]
```

という特徴で表されます。同じポリラインに属するベクトル断片どうしをノードとする小さなグラフを作り、MLP(多層パーセプトロン)で各ノードを符号化し、max-pooling(各次元の最大値を取る集約)で1本のポリライン全体を1つの特徴ベクトル(ノード)にまとめます。

**第2段(グローバルインタラクショングラフ)**: こうしてできた「車線=1ノード」「各エージェント=1ノード」を、全結合のグラフ(全ノードが互いにつながる)にして、Self-Attention(自己注意、各ノードが他の全ノードとの関係の強さを学習する仕組み)で相互作用を計算します。これにより「この車はこの車線に沿って走りそう」「あの歩行者はこの横断歩道に向かいそう」といった関係が捉えられます。

最後に、対象エージェントのノード特徴から未来軌道をデコードします。

VectorNet のもう1つの工夫が**補助タスク(auxiliary task)**です。学習中にランダムにノード(ポリライン)を隠し、それを復元させる「masked map completion」を解かせます。これにより地図の構造的な理解が深まります。

ベクトル表現の利点を整理すると、(1) 座標をそのまま使うので精度が落ちない、(2) 空白を計算しないので効率的、(3) 広範囲を扱ってもノード数しか増えず解像度が落ちない、(4) 物体間の関係をグラフで自然に表現できる、です。

```
ラスタ化:  地図 → BEV画像[C,400,400] → CNN → 予測
ベクトル:  地図 → ポリライン集合 → MLP+maxpool → ノード群
                → 全結合グラフ + Attention → 予測
```

## 手法の系譜と主要論文

- **ChauffeurNet** (Bansal et al., Waymo, 2018, arXiv:1812.03079): ラスタ化BEV + CNN による模倣学習の代表。地図と周囲をすべて画像に描いて運転方策を学習。合成的な摂動(perturbation)で堅牢性を上げる工夫が有名。
- **IntentNet / Rules of the Road / MultiPath** (2018〜2019): いずれもラスタ化BEVベースの予測。MultiPath はアンカー軌道を使い多峰性(複数の可能な未来)を表現。
- **VectorNet** (Gao et al., Waymo, CVPR 2020, arXiv:2005.04259): ベクトル表現 + 階層GNN を確立。ラスタ化の効率・精度の限界を打破。
- **LaneGCN** (Liang et al., Uber ATG, ECCV 2020): 車線をグラフとして畳み込む LaneConv を提案。Argoverse でSOTA級。
- **TNT / DenseTNT** (Zhao et al., 2020/2021): VectorNet をベースに目標点(target)サンプリングで多峰予測を強化。
- **VectorMapNet** (Liu et al., 2022, arXiv:2206.08920) / **MapTR** (Liao et al., ICLR 2023): 地図を「与えられるもの」ではなく、センサからオンラインで**ベクトル地図として推定する**方向への発展。HD地図構築自体をベクトル出力で行う。

系譜の流れは「ラスタ化(2018前後)→ ベクトル表現(2020 VectorNet)→ ベクトル地図のオンライン推定(2022以降)」と、地図を画像から幾何構造へ、そして与件から推定対象へと移してきた歴史として読めます。

## 論文の実験結果(定量データ)

**VectorNet** (CVPR 2020) の核心的な結果:

- 評価指標は予測軌道の誤差です。**ADE(Average Displacement Error、予測点と正解点の平均ユークリッド距離、小さいほど良い)** と **DE@3s(3秒後の最終位置の誤差)** を使います。
- 自社の大規模データセットで、同等の計算量のラスタ化+CNNベースラインに対し、VectorNet は予測誤差を改善しつつ、**計算量(FLOPs)を約70%削減、モデルパラメータも大幅に削減**したと報告。同じ予測性能をはるかに軽い計算で達成できることを示した点が重要です。
- 公開ベンチマーク **Argoverse Forecasting** では、minADE(複数予測のうち最良のものの平均誤差)で当時の上位、論文時点で K=1 の DE@3s が他のベンチマーク手法と競合する水準を達成。
- アブレーション(構成要素を1つずつ外して効果を測る実験)で、(a) 階層グラフ構造、(b) 補助タスク(masked map completion)の両方が精度に寄与することを確認。補助タスクを加えると予測誤差がさらに数%改善した。

**ラスタ化との直接比較の意味**: VectorNet が示したのは「ラスタ化は地図のスパース(疎)な構造を画像という密な形式に無理やり押し込むため、計算が無駄で、かつ離散化で精度も落とす」という点です。同じ精度を出すのにラスタ化は約3倍以上のFLOPsを要した、というのが定量的なメッセージです。

**LaneGCN** (ECCV 2020) は Argoverse で minADE(K=6)0.71m、minFDE(K=6)1.08m 級を報告し、ベクトル/グラフ系がラスタ系を予測精度でも上回ることを補強しました(数値はバージョンにより前後します)。

**MapTR** (ICLR 2023) では、オンラインHD地図構築で nuScenes 上の mAP(平均適合率、地図要素の検出精度)を、従来のラスタ化ベースのセグメンテーション手法より大幅に高め、かつリアルタイム(数十FPS)で動くことを示しました。これはベクトル表現が「地図を読む」だけでなく「地図を作る」段階にまで有効であることの定量的証拠です。

## メリット・トレードオフ・限界

**ラスタ化のメリット**: CNNという枯れた技術を流用でき実装が容易。地図・エージェント・信号などを統一フォーマット(画像)で扱えるので、異種情報の融合が素直。占有グリッドや意味セグメンテーションなど画像出力タスクとも相性が良い。

**ラスタ化の限界**: 解像度と範囲のトレードオフ、空白領域の計算の無駄、離散化による精度損失、長距離の関係(遠い車線と遠いエージェントの関係)を畳み込みの局所性では捉えにくい。

**ベクトル表現のメリット**: 高精度(連続座標を保持)、計算効率(疎な構造を疎なまま扱う)、広範囲でも破綻しない、物体間の長距離関係をAttention/GNNで自然に表現できる。

**ベクトル表現の限界・トレードオフ**:

- **入力の前処理が必要**: 折れ線への分割やID付けなど、地図をきれいなベクトルに整える前処理が要る。生のセンサからは直接得られず、HD地図やオンライン地図推定に依存する。
- **画像的な情報を捨てる**: 路面の見え方、舗装の質感、看板の絵柄など「絵としての情報」はベクトル化で失われる。横断歩道の縞模様や工事の様子など非構造的な手がかりは扱いにくい。
- **グラフ構築の設計依存**: どの単位でノードを切るか、どうエッジを張るかの設計が性能に効き、ノウハウが要る。
- **可変長・順序の扱い**: 折れ線ごとに点数が違い、ノード数が動的に変わるため、バッチ処理やパディングの工夫が必要。

現実のシステムでは、両者は排他ではありません。占有予測やフリースペース推定はラスタ的に、車線追従や軌道予測はベクトル的に、と**タスクに応じて使い分ける/併用する**のが近年の主流です。

## 発展トピック・研究の最前線

- **オンライン・ベクトル地図構築**: VectorMapNet・MapTR・MapTRv2 など、事前のHD地図に頼らずカメラ/LiDARからその場でベクトル地図を生成する研究。HD地図の整備・更新コストを下げる狙い。
- **End-to-End への統合**: UniAD など、知覚→地図→予測→計画を1つのネットワークで貫く設計では、ベクトル(クエリ/トークン)表現が中間表現として自然に使われる。地図はクエリの集合として表現される。
- **SDマップ(標準精度地図)の活用**: HD地図がない場所向けに、ナビ用の粗い地図(OpenStreetMapなど)をベクトルの事前情報として注入する研究。HD地図の有無に頑健なモデルを目指す。
- **ベクトルとラスタのハイブリッド**: 占有・フリースペースはラスタ、車線・軌道はベクトル、と両表現を1モデルで併用するマルチ表現アプローチ。
- **未解決課題**: ベクトル地図のトポロジー(車線のつながり・分岐・合流関係)を正確に推定すること、地図の不確実性(オンライン推定が間違っている可能性)を下流の予測・計画に伝播させること、地図がない/古い状況での頑健性。
