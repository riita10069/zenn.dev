# DETR型マップquery decoder

## ひとことで言うと
「学習可能なquery(問い合わせ用の数値ベクトル)」を何本も用意し、それぞれに「地図の要素を1つ見つけて来い」と命じる仕組みです。これにより、走行中にカメラ映像から車線や横断歩道などの地図要素を、まとめて(集合として)一気に予測できます。物体検出の DETR が発明した枠組みを地図構築に応用したもので、MapTR などのオンライン地図構築モデルで共通して使われています。

## 直感的な理解
走行中にセンサ(カメラやLiDAR)から、その場でリアルタイムに地図を作ることを「オンライン地図構築」と言います([オンライン vs オフライン地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0456_online-vs-offline-mapping))。出したい出力は「車線がここからここまで」「横断歩道がこの位置」という、複数の地図要素のリストです。

ここで根本的な難しさがあります。シーンによって要素の数が違うのです。あるシーンでは車線が3本、別のシーンでは横断歩道込みで10要素、というふうに出力の個数が一定しません。ニューラルネットは普通「決まった長さの出力」を出すのが得意で、「個数が毎回変わるリスト」は苦手です。

従来の物体検出は、画像中のたくさんの格子点ごとに候補を出し、最後に重複候補を消す後処理(NMS、後述)で個数を絞っていました。この後処理は手作業で閾値を調整する必要があり、学習の外側にある「継ぎ目」でした。継ぎ目があると、その部分はデータから最適化されず、人間の勘で決めることになります。DETR はこの継ぎ目をなくし、「個数の決まらない要素集合を、モデルが直接ひとまとめに予測する」枠組みを提案しました。地図構築はまさに「個数不定の要素集合の予測」なので、この枠組みがそのまま当てはまります。レストランで「空いている席の数だけ料理を運ぶウェイター」を、最初から多めに待機させておき、客がいない席のウェイターは「配膳なし」と振る舞う、というイメージです。

## 基礎: 前提となる概念

query(クエリ)。モデルが持つ学習可能な数値ベクトル(たとえば長さ256の数の並び)で、「1つの要素を担当する空のスロット」だと考えてください。最初はランダムな値で、学習を通じて「自分はどんな要素を探すべきか」という役割を獲得します。学習後の query を可視化すると、「画面の左寄りの要素を探す係」「中央の要素を探す係」のように、おおまかな担当領域に分かれることが知られています。

アテンション(注意機構)。「特徴のどの場所をどれくらい重視して情報を引っ張ってくるか」を数値の重みで決める仕組みです。各 query が特徴マップ全体を見渡し、自分が担当する要素がありそうな場所から重点的に情報を吸い上げます。これを何層も重ねたのが Transformer デコーダです。デコーダには2種類のアテンションがあり、query 同士が情報交換する「自己アテンション(self-attention)」と、query が画像/BEV特徴から情報を取る「交差アテンション(cross-attention)」が交互に入ります。前者は query 同士が「自分はこの要素を担当するから君は別のを担当して」と役割を調整するのに効き、これが重複出力の抑制(NMS不要)につながります。

NMS(Non-Maximum Suppression、非最大値抑制)。近接した重複検出を1つに間引く後処理です。たとえば同じ車を5個の枠が囲んでいたら、最もスコアの高い1個だけ残す、という操作。閾値(どれだけ重なったら同一とみなすか)を人手で決める必要があり、学習とは切り離されています。

二部マッチング(bipartite matching)。2つの集合の要素を1対1で最適に対応づける問題です。「予測した50個」と「正解の3個」を、最もコストが小さくなるように組み合わせます。最適な組み合わせを求めるアルゴリズムがハンガリアン法(Hungarian algorithm)で、要素数 n に対して n の3乗程度の計算で厳密最適解を出せます。

ベクトル化地図要素。車線や横断歩道を「順序を持つ点の並び(折れ線)」で表したもの。たとえば1本の車線を等間隔20点の座標列 `(20, 2)` で表します。RGBで塗ったラスタ地図と違い、座標そのものを持つので解像度の制約を受けず、経路計画にそのまま渡せるのが利点です([地図チャンネル符号化方式](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0419_map-channel-encoding-scheme) のラスタ方式との対比)。

チャンファー距離(Chamfer distance)。2つの点列がどれだけ似た形かを測る尺度。一方の各点から相手側の最も近い点までの距離を平均し、それを両方向で足します。点の順番が違っても形が同じなら小さくなるので、形状比較に向いています。

## 仕組みを詳しく

処理は次のステップで進みます。

1. 出力したい要素の最大個数を決め、その本数だけ query を用意する。たとえば最大50要素なら query は50本、形状 `(50, 256)`。最初はランダムで、学習で意味を獲得する。地図構築では、これを階層化して「インスタンス query(要素そのもの)× 点 query(その要素を構成する点)」の二段にする設計もよく使われます。たとえば50要素 × 各20点なら、実質 `(50, 20, 256)` の query 群になります。

2. センサから特徴マップを作る。複数カメラの画像を処理し、BEV(真上視点)空間の特徴 `(256, H, W)` に変換しておく。BEV化の方法は手法ごとに違い、深度推定で持ち上げる方式(Lift-Splat 系、[ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction))や、BEVクエリで引き寄せる方式(BEVFormer 系)があります。

3. 各 query が Transformer デコーダのアテンションで特徴マップ全体を見渡し、「自分が担当すべき要素はどこか」を探して情報を吸い上げる。これを6層ほど重ねるのが典型です。層を重ねるごとに query の指す位置が正解へ近づいていきます。多くの実装は各層の出力にも損失をかける「補助損失(auxiliary loss / deep supervision)」を使い、途中層からも正解に近づくよう促して収束を速めます。

4. 情報を吸い上げた各 query を、小さな予測ヘッド(全結合層)に通す。地図構築なら、各 query から「クラス(車線か横断歩道か境界か)」と「形を表す点列の座標 `(点数, 2)`」が出ます。

5. 50本すべてが本物を出すわけではない。実際の要素が3個なら、3本だけが本物を出し、残り47本は「該当なし(no object)」という特別なクラスを出すよう学習する。これで個数不定に対応する。

集合予測と二部マッチング。学習時の核心は「予測50個と正解3個をどう対応づけて損失を計算するか」です。DETR は二部マッチングで、予測と正解を最もコストの小さくなる1対1対応に割り当てます。コストは「クラスが合っているか」と「点列の位置が近いか」を合わせたものです。具体例で言えば、正解が「車線A・車線B・横断歩道C」なら、50本のうち最もAに近い1本をAに、Bに近い1本をBに、Cに近い1本をCに割り当て、残り47本は「該当なし」を出すべきと判定します。この1対1対応のおかげで、同じ要素を複数の query が重複して出す必要がなくなり、NMS が不要になります。マッチングはまず「コスト行列」(予測i × 正解j のコストを並べた表)を作り、ハンガリアン法で総コスト最小の割り当てを求めます。この割り当ては各イテレーションで動的に変わるため、学習序盤はマッチングが不安定になりやすく、これが DETR の収束の遅さの一因です。

なぜこの設計か。地図要素は「順序のない集合」です(車線3本のどれを先に並べても同じ地図)。集合を直接予測し、正解と最適マッチングで突き合わせることで、出力順序に依存しない学習ができます。これが「画像から地図集合への直接写像」という end-to-end の本質です。一方で、要素「内部」の点列には順序があります(折れ線の点は順番に意味がある)。この「要素間は順不同、要素内は順序あり」という二重構造をどう扱うかが、地図への応用での工夫どころになります。

## 手法の系譜と主要論文

DETR(Carion, Massa, Synnaeve, Usunier, Kirillov, Zagoruyko; Facebook AI, ECCV 2020, "End-to-End Object Detection with Transformers", arXiv 2005.12872)が、query + 集合予測 + 二部マッチングの枠組みを物体検出で初めて提案しました。理由は、当時の検出器が依存していたアンカー(あらかじめ置く候補枠)や NMS という手作業の部品を全廃し、検出を end-to-end 化したかったから。効果として、人手の後処理なしで当時最先端の Faster R-CNN に匹敵する精度を達成しました。トレードオフは、学習の収束が非常に遅い(標準の検出器が約12〜36エポックのところ、DETR は500エポックを要した)ことと、小さい物体の検出が弱いことでした。

Deformable DETR(Zhu, Su, Lu, Li, Wang, Dai; SenseTime, ICLR 2021, "Deformable DETR: Deformable Transformers for End-to-End Object Detection", arXiv 2010.04159)が、その弱点を大幅に改善しました。各 query が特徴マップ全体ではなく、学習した少数の要点(典型的には各層・各スケールで数点)だけに注目する「変形可能アテンション(deformable attention)」を導入し、計算量を抑えつつ収束を約10倍速め(50エポック程度)、マルチスケール特徴を扱えるようにして小物体精度も上げました。地図構築の実装の多くは、この変形可能アテンションを内部で使っています。なぜ地図で重要かというと、BEV特徴は解像度が大きく(200×200など)、全画素にアテンションすると計算が爆発するため、要点だけ見る変形アテンションが事実上必須だからです。

HDMapNet(Li, Wang, Li, Xie, Zhou; ICRA 2022, arXiv 2107.06307)が、オンライン地図構築を定量評価する枠組みを最初に整えましたが、出力はラスタ的なセグメンテーション + 後処理でベクトル化する方式で、純粋な集合予測ではありませんでした。塗り分けた後にヒューリスティックな後処理で線をつなぐため、ここに「継ぎ目」が残っていました。

VectorMapNet(Liu, Wang, Chen, Zhao; ICML 2023, "VectorMapNet: End-to-end Vectorized HD Map Learning", arXiv 2206.08920)が、地図要素を点列として end-to-end に出す方向を本格化しました。要素を検出してから点列を自己回帰的(autoregressive、1点ずつ順に生成)に生成する2段構成です。自己回帰のため点を増やすほど推論が逐次的に遅くなる、という弱点がありました。

MapTR(Liao, Chen, Wang, Cheng, Zhang, Liu, Huang; Horizon Robotics, ICLR 2023, "MapTR: Structured Modeling and Learning of Online Vectorized HD Map Construction", arXiv 2208.14437)が、DETR型デコーダを地図構築に綺麗に持ち込み、決定版に近い形を示しました。地図要素を固定点数(たとえば20点)の順序つき点列で表し、各 query が1要素を担当します。点列を一度に並列で出す(自己回帰でない)ので高速です。核心は permutation-equivalent(順列等価)モデリングです。横断歩道のような閉じた多角形は始点をどこに取っても同じ形なので、正解の点列を1通りに固定せず、複数の等価な並び(始点をずらした巡回置換、向きを反転した並び)を許して二部マッチングします。これにより「始点の取り方や向きが違うだけ」の無意味なペナルティが消え、学習が安定します。

MapTRv2(Liao et al., 2023, arXiv 2308.05736)はこれを補助損失や深度監督で強化し、精度と速度をさらに上げました。具体的には、ワン・ツー・メニーのマッチング(1つの正解に複数 query を割り当てて学習信号を増やす)、補助的な one-to-one/dense 予測、深度の補助監督などを足しています。曲線で出す系統([ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation))もあり、各 query が点列の代わりにベジエ曲線の制御点を出すことで、少ない数値で滑らかな曲線を表します。

## 論文の実験結果(定量データ)

オンライン地図構築の主指標は mAP(mean Average Precision、平均適合率の平均)です。予測した地図要素と正解要素が「十分近い」と言える距離しきい値(たとえば 0.5m, 1.0m, 1.5m のチャンファー距離)で一致をカウントし、適合率と再現率を総合します。0〜100% で、高いほど良い指標です。3つのしきい値での AP を平均して総合 mAP とするのが慣例で、しきい値が厳しい(0.5m)ほど精密さが、緩い(1.5m)ほど大まかな検出力が問われます。

DETR は COCO 物体検出で、Faster R-CNN と同等の AP(報告では 42〜44 前後)を、NMS なしで達成しました。ただし当時の標準より桁で多い学習エポック(500)を要し、小物体 AP は劣りました。Deformable DETR はこれを、約1/10の学習回数(50エポック)で同等以上の AP に引き上げ、マルチスケール対応で小物体も改善しました。

MapTR は nuScenes のオンライン地図構築(車線境界・横断歩道・車線分離線の3クラス)で、カメラのみ入力で当時最高水準の mAP を、リアルタイム級の速度で達成したと報告しています。報告によれば、ResNet50 バックボーンのカメラのみ設定で mAP は概ね50前後に達し、HDMapNet 系のセグメンテーション+後処理方式(報告では mAP 20台〜30台)を二桁ポイント上回りました。かつ推論速度はおおむね10〜25 FPS 級(設定による)で、後処理を挟む方式より高速でした。これは「後処理を挟む方式」より「集合予測で end-to-end に出す方式」が、精度と速度の両面で優れることを示した重要な結果です。MapTRv2 ではさらに mAP が一段(報告ではおおむね60台へ)上がったとされます。

アブレーションでは、permutation-equivalent モデリングを外して点列の並びを1通りに固定すると mAP が明確に(報告では数ポイント〜十数ポイント)下がること、デコーダ層を浅くすると精度が落ちることが示されています。前者は「閉じた形状の始点・向きの自由度を学習に正しく織り込むこと」が効いている証拠で、後者は「query を反復的に正解へ寄せる多層構造(deep supervision)」が精度に効いていることを意味します。MapTRv2 はさらに、深度の補助監督(BEV化の精度を上げる学習信号)や one-to-many マッチングを加えると mAP がもう一段上がり、収束も速まると報告しました。これらアブレーションから、地図構築での精度の鍵は「マッチングをいかに学習信号として濃く・整合的にするか」にあることが読み取れます。

## メリット・トレードオフ・限界

メリット。NMS やアンカーといった手作業の後処理が不要で、学習の中ですべて完結します(end-to-end)。出力個数が不定でも自然に扱え、地図要素を集合として一気に出せます。出力が座標(ベクトル)なので、ラスタ方式と違い解像度の制約を受けず、経路計画にそのまま渡せます。query は学習で役割分担を獲得するため、要素タイプごとに query 群を割り当てるなどの発展もしやすい設計です。

トレードオフと限界。第一に、素の DETR は収束が遅く計算資源と時間を多く要します(変形可能アテンションや one-to-many マッチングで緩和)。第二に、query の本数(出力上限)を事前に決める必要があり、要素数がそれを超えると取りこぼします(密な都市交差点では取りこぼしが起きうる)。第三に、形状を固定点数の点列で表すため、非常に複雑な曲線は近似になり、点数を増やすと計算が増えます(これがベジエ曲線表現の動機、[ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation))。第四に、二部マッチングはバッチ内で要素数が少ないと学習信号がまばらになり、稀なクラス(たとえば滅多に出ない地図要素)の学習が不安定になりがちです。第五に、評価指標がチャンファー距離ベースのため「形は近いが向き(進行方向)が逆」のような誤りを十分に罰しない場合があり、指標と実用価値の乖離も残ります。

研究上の未解決課題としては、地図要素どうしの接続関係(どの車線からどの車線へ進めるか、というトポロジー)を集合予測だけでは表しにくいことが挙げられます。点列の位置は当てられても「つながり」が抜けると、経路計画には使えません。近年はこのトポロジー推定を query デコーダに統合する研究(車線中心線グラフを同時に出し、要素間の接続を query 同士の関係として予測する)が活発で、ベクトル地図構築の最前線になっています。また、時間方向の安定性(連続フレームで同じ車線が出たり消えたりせず一貫するか)も実用上の課題で、過去フレームの query を引き継ぐ時系列拡張が研究されています。

## 発展トピック・研究の最前線

DETR型デコーダは、地図構築だけでなく、軌跡予測ヘッドやBEVクエリにも同じ思想が広がっています。地図と物体と運動を1つの query 集合で同時に解く統合知覚(検出・地図・予測を共有デコーダで出す)や、地図要素の点列を曲線(ベジエ)で出して滑らかさと点数効率を両立する方向([ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation))、さらに車線トポロジーまで一括で出す研究へと発展しています。最近は「車線中心線グラフ + 交通要素(信号・標識)とのひもづけ」を同時に出して運転に直接使える地図を作る方向(いわゆる lane-topology / SD-map prior 併用)も盛んです。根底にあるのは「個数不定の構造化出力を、手作業の後処理なしに学習で直接出す」という DETR の発想で、これがオンライン地図・end-to-end 運転の共通基盤になりつつあります。end-to-end 運転モデル全体の中では、この地図 query デコーダが「知覚と計画をつなぐ中間表現」を担い、地図・物体・自車の query を共有することで、誤差が後段に伝播する前に一貫して最適化できる利点が注目されています。
