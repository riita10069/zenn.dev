# 抽象基底クラスによる契約定義

## ひとことで言うと
プログラムの部品（経路計画器、履歴処理モジュール、特徴抽出器など）を後から自由に差し替えられるようにするために、「入力と出力のルール（契約）」を抽象基底クラス（Abstract Base Class、略してABC）で先に決めておく設計です。契約を守らない実装はそもそも作れない状態にすることで、差し替えても呼び出し側のコードを変えずに済み、かつ「形は合っているのに意味が違う」種類のバグを構造的に防ぎます。

## 直感的な理解
電源コンセントを思い浮かべてください。家のコンセントは形と電圧が規格で決まっているので、メーカーの違う掃除機でも扇風機でも、同じ穴に挿せば動きます。コンセント側は「中身が掃除機か扇風機か」を一切気にしません。挿せること（契約を満たすこと）だけが保証されていれば良いのです。

ソフトウェアでもこれをやりたい。機械学習モデルを研究するとき「経路計画器をベジェ版にしたり拡散モデル版にしたり」「履歴処理を無しにしたり時系列エンコーダにしたり」と、部品をどんどん入れ替えて比較します。このとき部品を変えるたびに、それを呼ぶ学習ループのコードまで書き換えるはめになると、実験のたびに広範囲を直すことになりバグの温床です。コンセントのように「形さえ合えば挿し替えるだけ」にしたい。その「形の規格」を定めるのが抽象基底クラスです。

さらに厄介なのは、形（テンソルの shape）が同じでも意味が違う場合です。あるコンセントは100Vだが別のは200Vで形だけ同じ、という状況を想像してください。挿せてしまうが機器が壊れます。機械学習の部品でもこれが起きます。次の節で詳しく見ます。

## 基礎: 前提となる概念

### 情報隠蔽とモジュール分割（なぜそもそも分けるのか）
この設計の出発点は、David Parnas が1972年の論文 "On the Criteria To Be Used in Decomposing Systems into Modules"（CACM）で示した情報隠蔽（information hiding）の原則です。要点は「モジュールは変更されやすい設計判断を内側に隠し、外には安定したインターフェースだけを見せるべき」ということ。たとえば「経路計画にどのアルゴリズムを使うか」は研究中ころころ変わる設計判断なので、これは部品の内側に隠し、外には「特徴を入れたら軌跡が出る」という変わらない約束だけを見せる。これが契約です。半世紀前の知見ですが、機械学習の実験コードにこそ強く効きます。

### 契約がないと起きる「静かなバグ」
ルールを決めず、各計画器が好き勝手な戻り値を返す状況を考えます。計画器Aの forward() は完成した軌跡 `[B, 128]`（B はバッチサイズ）を返し、計画器Bの forward() は学習用の中間量である「速度場（velocity field）」を返すとします。速度場というのは拡散モデルやフローマッチングで使う作業用テンソルで、「ノイズから本物へ向かう各点の進む方向」を表します。形はたまたま `[B, 128]` と同じになりえます。

呼び出し側が「forward() の戻り値は軌跡だ」と思い込んで、それと正解軌跡との二乗誤差（MSE 損失）を計算したとします。計画器Aでは正しい。しかし計画器Bでは「速度場を軌跡と誤解して損失を計算する」ことになり、しかもテンソルの形は合っているのでエラーは一切出ません。学習は回るが、最適化している対象が間違っている。これが silent bug（静かなバグ）、別名 footgun（自分の足を撃つ罠）です。エラーが出るバグは見つけて直せますが、エラーが出ないバグは「結果がなぜか悪い」という形でしか現れず、原因究明に膨大な時間を奪われます。Sculley らの "Hidden Technical Debt in Machine Learning Systems"（NeurIPS 2015）は、こうした「境界が曖昧なことに起因する負債」が機械学習システムのコストの大半を占めると論じており、契約はその対策の一つです。

### 抽象基底クラス（ABC）とは
ABC は「それ自体は中身が空で、継承する子クラスに『このメソッドは必ず実装しなさい』と強制する親クラス」です。Python では abc モジュールの ABC を継承し、必須メソッドに @abstractmethod デコレータ（メソッドに付ける目印）を付けます。子クラスがそのメソッドを実装しないと、インスタンス化（オブジェクトを作ること）した瞬間に TypeError になります。つまり「契約を守らない実装はそもそも動かせない」状態を、実行の途中ではなくオブジェクト生成の段階で強制できます。これは PEP 3119（Python の言語仕様提案文書）で Python に正式導入された仕組みです。

### リスコフの置換原則（LSP）
この設計の理論的背景がリスコフの置換原則（Liskov Substitution Principle）です。Barbara Liskov が1987年の OOPSLA 基調講演「Data Abstraction and Hierarchy」で示し、Liskov & Wing が1994年の論文 "A Behavioral Notion of Subtyping"（ACM TOPLAS）で定式化しました。原則は「型 S が型 T の派生型なら、T を期待するプログラムのどこでも S を代入してプログラムの正しさが保たれねばならない」というものです。噛み砕くと「子クラスは親クラスの代わりに、何の予告もなく使えるべき」。コンセントの例でいえば「同じ穴に挿せる機器は同じ電圧で安全に動くべき」です。抽象基底クラスは、この原則を「契約を満たさない実装は作れない」という形でコードに落とし込む手段です。

ただし LSP は shape の一致だけでは不十分で、振る舞い（behavior）の一致まで要求する点が重要です。「形は軌跡だが中身は速度場」は LSP 違反です。だから契約は「forward は何を返すか」だけでなく「何を返してはならないか」まで明文化する必要があります。

## 仕組みを詳しく

差し替え可能な計画器の基底クラス（一般に BasePlanner のように呼ばれる）の典型的な設計を示します。これは nn.Module（PyTorch の層の基底）と ABC を多重継承し、契約を2本のメソッドに分けて定義します。

メソッド1: forward（推論専用、@abstractmethod で必須）。

```
forward(scene_features, visual_history, egomotion_history, **kwargs) -> Tensor
  返り値: 必ず完成した軌跡 [B, num_timesteps * num_signals]
  禁止:   モード依存の中間量（拡散の速度場、ノイズ予測など）を返してはならない
```

これにより呼び出し側は「forward の返り値は常に完成軌跡」と安心して依存できます。実装が決定的回帰でも生成的サンプリングでも、外から見える契約は同一です。

メソッド2: compute_planner_loss（学習目的の計算、契約上の役割）。

```
compute_planner_loss(predictions, targets, **kwargs) -> scalar loss
  役割: 学習に必要な中間量（ノイズサンプル、目標速度場など）を
        このメソッドの内側に閉じ込め、呼び出し側のスコープに漏らさない
```

なぜ2本に分けるのか。前述の silent bug の根本原因は「学習で扱う量（速度場）と推論で扱う量（軌跡）が違うのに、同じ forward から両方を出そうとした」ことです。これを「polymorphic forward（モードで意味が変わる多態の forward）は罠だ」と捉え、推論の出力（forward）と学習の損失（compute_planner_loss）を別メソッドに分離します。速度場のような作業用テンソルは compute_planner_loss の内側だけに存在し、外に出ないので、「速度を間違ったターゲットと組み合わせる」事故が構造的に起こせなくなります。これは拡散モデル系（Diffusion Policy; Chi et al., RSS 2023）や Flow Matching（Lipman et al., ICLR 2023）の実装の流儀でもあり、「学習と推論で扱う量が違うインターフェースは分離する」という実務的教訓の結晶です。これらの手法では、学習時にネットワークが予測するのはノイズや速度場であって最終サンプルではなく、サンプルは複数ステップの反復（denoising / ODE 積分）の結果として初めて得られます。学習対象と推論出力が原理的に別物なので、混同を防ぐインターフェース設計が一層重要になります。

時系列メモリの基底クラス（一般に BaseTemporalMemory のように呼ばれる）も同様に契約を定めます。役割は「過去の履歴をまとめて文脈ベクトルにする」ことです。

```
forward(visual_history, egomotion_history, **kwargs) -> (visual_context, egomotion_context)
  入力: visual_history    [B, T, Dv]  または [B, Dv]    （T は時間ステップ数）
        egomotion_history [B, T, De]  または [B, De]
  出力: visual_context    [B, Dv]      （系列を1ベクトルに圧縮）
        egomotion_context [B, De]
```

つまり「`[B, T, feat]` の系列履歴を `[B, feat]` の文脈に圧縮して下流に渡す」役割が契約として固定されます。実装は「履歴を使わない版（恒等的に最新フレームを返す）」「時系列エンコーダ版（RNN や Transformer で集約する版）」など何種類あってもよく、どれを使っても下流の計画器のコードは無変更で動きます。これは Template Method パターン（処理の骨格を親が決め、可変部を子に実装させる）の典型です。

### **kwargs による前方互換
両 forward が **kwargs（余分なキーワード引数を受け流す仕組み）を取る点も契約の一部です。ある計画器だけが必要とする追加入力（例: サンプリング用の乱数生成器 generator）を呼び出し側が一律に渡しても、それを必要としない部品は黙って無視できます。これにより呼び出し側を「部品の種類に依存しない（planner-agnostic）」書き方に保てます。代償は後述します。

### レジストリとの組み合わせ
契約が固定されていると、文字列キーで実装を選ぶレジストリ（registry、名前から実装を引く辞書）パターンが成立します。

```
PLANNER_REGISTRY = {"flow_matching": FlowMatchingPlanner, "bezier": BezierPlanner, ...}
planner = PLANNER_REGISTRY[name](**kwargs)   # 名前1つで差し替え
```

新しい計画器を作って契約を満たし、レジストリに1行足すだけで、学習ループを無改変で新手法を試せます。これが「足せば選べる」拡張性の正体で、抽象基底クラスはその土台です。設定ファイル（YAML など）に文字列で手法名を書くだけで構成が切り替わるので、実験管理ツールとも自然につながります。

## 手法の系譜と主要論文

これはソフトウェア工学の設計パターンであり、特定の機械学習論文に紐づくものではありませんが、背景には明確な系譜があります。

- モジュール分割: Parnas の1972年論文「On the Criteria To Be Used in Decomposing Systems into Modules」（CACM）が情報隠蔽の原則を確立。変わりやすい設計判断をインターフェースの裏に隠す、という発想の源流。
- 抽象データ型と型階層: Liskov の1987年 OOPSLA 基調講演「Data Abstraction and Hierarchy」。1994年の Liskov & Wing "A Behavioral Notion of Subtyping"（ACM TOPLAS）で振る舞い的部分型として定式化。
- デザインパターン: Gamma, Helm, Johnson, Vlissides（通称 Gang of Four）の "Design Patterns"（1994）。本設計は Strategy パターン（アルゴリズム本体を差し替え可能な部品として外出しする）と Template Method パターン（処理の骨格を親が決め、可変部を子に実装させる）の融合です。
- Python での実装手段: PEP 3119（2007）が abc モジュールと @abstractmethod を導入。PEP 544（2017）が Protocol（構造的部分型）を導入。
- 機械学習文脈での教訓: 拡散モデル系（Diffusion Policy; Chi et al., RSS 2023, arXiv:2303.04137）や Flow Matching（Lipman et al., ICLR 2023, arXiv:2210.02747）では学習対象（ノイズ予測・速度場）と推論出力（生成サンプル）が異なるため、それらを混同しないインターフェース分離が実務的に重要だという経験が、契約を分ける動機になっています。Sculley らの NeurIPS 2015 論文は、こうした境界設計の不備が機械学習システムの技術的負債の主因だと指摘しています。

## 論文の実験結果（定量データ）

設計パターンそのものに minADE のような評価指標はありませんが、その効果は「実験回転率」と「バグ混入率」という開発上の量で測れます。一般論として観察される効果を挙げます。

- 差し替えコストの削減: 契約が固定されていると、新手法の試行に必要なコード変更が「新クラス定義＋レジストリ1行」に限定されます。契約がない場合、呼び出し側の損失計算・前処理・後処理まで毎回触る必要があり、変更行数が数十倍に膨らみます。これは ablation study（どの部品を抜くと性能がどう変わるかの系統的比較）を回す研究では決定的で、契約があるからこそ「計画器だけを A→B に変えて他を完全固定」という公平な比較が成立します。逆に契約がないと、比較のたびに周辺コードも変わり「何が効いたのか分からない」交絡（confounding、複数の要因が同時に変わって因果が分離できないこと）が生じます。研究の主張の信頼性そのものが契約の有無に左右されます。
- silent bug の除去: forward と損失計算の分離により、前述の「速度場を軌跡と誤解する」種類のバグは、設計上発生する経路がなくなります。これは定量化しにくいですが、デバッグに費やす時間という形で実開発コストに直結します。
- 早期失敗（fail fast）: 契約違反の実装はインスタンス化の瞬間に TypeError で落ちます。実行を何時間も回した末に「なぜか loss が下がらない」と気づくのではなく、起動直後に失敗するため、フィードバックループが短くなります。GPU を何時間も無駄に回さずに済む、という形でコストに跳ね返ります。

これらは Brooks の "Mythical Man-Month"（1975）以来繰り返し論じられてきた「インターフェース設計が大規模開発の生産性を支配する」という経験則の、機械学習研究版の現れです。

## メリット・トレードオフ・限界

メリット:
- 実装を差し替えても呼び出し側を変えずに済み、ablation や手法比較が容易。
- 契約違反の実装はインスタンス化時にエラーになり、間違いを早期検出できる。
- forward と損失計算の分離など、契約設計で silent bug を構造的に排除できる。
- レジストリと組み合わせ、名前1つで構成を選べる拡張性が得られる。

トレードオフ・限界:
- 契約（抽象メソッドの引数・戻り値）を最初に正しく設計する必要があり、後から契約を変えると全実装に波及する。契約は資産であると同時に負債で、設計ミスのコストが大きい。早すぎる抽象化（premature abstraction）は、まだ要件が固まらないうちに誤った契約を固定してしまい、かえって足かせになる。
- 抽象の層が1枚増えるので、コードを初めて読む人には「本体がどこにあるのか」が一手間分かりにくい（間接性の代償）。
- **kwargs で引数を受け流す柔軟さは、タイプミスした引数名が黙って無視される危険と表裏一体。契約の緩さが両刃であり、Python の動的型付けでは型チェッカ（mypy 等）や Protocol を併用しないと shape・型の不一致が実行時まで露見しないことがある。
- LSP は shape ではなく振る舞いの一致を要求するが、それをコードで完全に強制する手段はなく、最終的には docstring と規律に頼る部分が残る。テンソルの shape は型システムでは表現しきれないため、近年は shape を型注釈に持ち込む試み（型レベルの次元検査）も研究されているが、まだ標準化されていない。

## 発展トピック・研究の最前線
静的な抽象基底クラスに加え、Python では typing.Protocol による構造的部分型（structural subtyping、明示的な継承なしに「メソッドが揃っていれば互換」とみなす）が PEP 544 で導入され、より緩い契約として使われます。型チェッカ（mypy, pyright）と組み合わせれば、契約違反を実行前に静的検出できます。機械学習フレームワークでは、契約を Pydantic のような実行時バリデーションや、JAX/PyTorch の関数変換と組み合わせ、入出力 shape まで契約に含める設計が進んでいます。さらに jaxtyping のようなライブラリは「`Float[Tensor, "B T D"]` のように shape を型注釈で書き、実行時にチェックする」ことを可能にし、shape の取り違えを契約として捕捉します。研究上の関心は「人間が書く docstring の契約」から「機械が検証可能な契約（型・shape・物理単位）」への移行にあります。
