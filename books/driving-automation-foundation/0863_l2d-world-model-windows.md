# World Model用シーケンシャルウィンドウ

## ひとことで言うと
動画のような連続フレームのデータセットから、「過去の数フレーム(履歴)」と「未来の数フレーム(予測の正解)」をまとめて切り出すための、純粋な関数群です。未来を予測するモデル(World Model)の学習には、入力となる過去の窓と、答え合わせ用の未来の窓が必要です。これらをデータセット本体の都合から切り離し、副作用のない関数として実装すると、本物のデータが無くても単体テストできます。さらに、データの記録レート(例 10 Hz)とモデルが動かしたいレート(例 1 Hz)が違う場合の「間引き(ストライド)」も明示的に扱えます。

## 直感的な理解
運転モデルには大きく2つの考え方があります。

反応的(reactive)モデルは「今この瞬間」の画像を見て、すぐ次の操作(ハンドル・アクセル)を決めます。反射神経のようなもので、高い頻度(例 1秒に10回)で現在のフレームを処理します。

世界モデル(World Model)は「過去の数フレーム」を見て「この先どうなるか」を頭の中で予測します。人間の運転で言えば、「前の車がさっきからじわじわ減速している、だからこの先止まりそうだ」と先を読む能力です。未来を内部で思い描けると、より先回りした判断ができます。世界モデルは反応的処理より粗い頻度(例 1秒に1回)で、過去の流れから未来を予測する、という設計がよく取られます。

ここで問題が起きます。世界モデルを学習させるには「過去の数フレーム(入力)」と「実際に起きた未来の数フレーム(正解)」のペアが要ります。ところが、現在フレームしか読まないように作られたデータ読み込みでは、過去も未来も手元にありません。これを供給するのが、本トピックの「シーケンシャルウィンドウ(連続フレームの窓)」です。動画から、ある時刻を中心に「過去◯枚・未来◯枚」を切り出す道具、と考えてください。

もう一つの論点がレートの違いです。データが 10 Hz(1秒に10フレーム)で記録されているのに、世界モデルは 1 Hz(1秒に1回)で動かしたいとします。すると、10フレームに1枚だけ拾えばよい(=ストライド10で間引く)ことになります。「秒間レートの違いを、フレーム間隔の間引きに翻訳する」のもこの道具の仕事です。

## 基礎: 前提となる概念
用語を噛み砕きます。

フレームとシーケンス: 動画は静止画(フレーム)の連なりです。連続したフレームの並びをシーケンスと呼びます。運転データは複数カメラ(前後左右など)を同時に記録するため、1時刻ぶんが「V 視点 × RGB 画像」になります。あるフレームのテンソル形は `[V, 3, H, W]`(V=視点数、3=RGB、H/W=縦横)です。

レートとストライド: レート(Hz、ヘルツ)は1秒あたりのフレーム数です。ストライド(stride)は「何フレームおきに拾うか」の間隔です。10 Hz のデータを 1 Hz 相当で扱うには、ストライド 10 で間引きます。一般に `stride ≈ source_hz / target_hz` です。

エピソード (episode): 1回の連続した走行(あるいは録画セッション)を指します。データセットは多数のエピソードを連結して保持することが多く、エピソード i の終わりと i+1 の始まりは「別の場所・別の状況」なので、時間的に隣り合って見えても連続していません。

データリーク (data leakage): 学習で「使ってはいけない情報」がこっそり混ざってしまうことです。時系列では、ある時刻の窓が別のエピソードのフレームを含んでしまうと、関係ない走行シーンが履歴や未来に紛れ込み、モデルが偽の相関を学んだり、評価が不当に甘くなったりします。

純粋関数 (pure function): 同じ入力に対して常に同じ出力を返し、外部状態を読み書きしない(副作用がない)関数です。純粋関数はテストが容易で、フレーム読み込みのような外部依存をコールバック(後述)として外から渡す設計にすると、データ本体無しで挙動を検証できます。

コールバック (callback): 「処理の一部を、呼び出し側から渡された関数に任せる」やり方です。ここでは `load_frame(row) -> [V, 3, H, W]`(行番号を渡すとその時刻のマルチビュー画像を返す関数)を外から受け取り、窓を組む関数自身はフレームの中身を知らずに済ませます。テストでは「行番号で埋めたダミーテンソルを返す合成ローダ」を渡せるので、実データ無しで検証できます。

## 仕組みを詳しく
典型的な実装は、4つの純粋関数に分かれます。すべて副作用がなく、フレーム読み込みは外から渡される `load_frame` コールバックに委ねます。

1. ストライドの計算

```python
def stride_for_hz(source_hz, world_model_hz):
    if source_hz <= 0 or world_model_hz <= 0:
        raise ValueError(...)
    return max(1, round(source_hz / world_model_hz))
```

10 Hz → 1 Hz なら 10、30 Hz → 1 Hz なら 30。`max(1, ...)` で最低でも 1 を保証し、0 以下にしません。レートが非正なら `ValueError` で早期に弾きます。

2. 相対オフセットの算出

「現在の行」を基準に、履歴と未来がそれぞれどの行を指すかを相対値で返します。

```python
def window_offsets(num_frames, stride):
    history = [-(num_frames - 1 - i) * stride for i in range(num_frames)]  # 古い順→現在
    future  = [(i + 1) * stride for i in range(num_frames)]                # 次→先
    return history, future
```

`num_frames=4, stride=10` なら、履歴 = `[-30, -20, -10, 0]`、未来 = `[10, 20, 30, 40]`。履歴の最後が必ず `0`(現在フレーム)になるのが設計の要点です。「現在を含む過去の流れ」を入力に、「これから起きること」を正解にする、という構造を表しています。

3. 必要な前後余白

完全な窓を取るには、現在行の前後にどれだけフレームが必要かを返します。

```python
def required_margins(num_frames, stride):
    return ((num_frames - 1) * stride, num_frames * stride)  # (before, after)
```

`num_frames=4, stride=10` なら `(30, 40)`。現在行の前に 30 行、後ろに 40 行ないと窓が完成しません。データセット側が「有効なサンプル(窓を取れる現在行)」を列挙するとき、この余白で端を除外します。

4. 実際に窓を組む

```python
def build_windows(load_frame, row, ep_start, ep_end, num_frames=4, stride=10):
    hist_off, fut_off = window_offsets(num_frames, stride)
    if row + hist_off[0] < ep_start or row + fut_off[-1] >= ep_end:
        raise IndexError(...)  # 窓がエピソード境界をまたぐなら拒否
    history = torch.stack([load_frame(row + o) for o in hist_off], dim=0)
    future  = torch.stack([load_frame(row + o) for o in fut_off], dim=0)
    return history, future
```

返り値はそれぞれ `[num_frames, V, 3, H, W]`(古い順→新しい順)。`ep_start, ep_end` は現在行が属するエピソードの `[start, end)` 範囲です。境界チェックが要るのは、別のエピソード(別の走行シーン)のフレームが履歴や未来に紛れ込むデータリークを防ぐためです。窓がエピソードからはみ出す場合は `IndexError` を投げ、呼び出し側はそのサンプルを無効として捨てます。

数値例で全体像を確認します。`row=50, num_frames=4, stride=10`、そのエピソードが `[0, 200)` なら:

```
履歴行: 50 + [-30,-20,-10, 0] = [20, 30, 40, 50]   (50 が現在)
未来行: 50 + [ 10, 20, 30, 40] = [60, 70, 80, 90]
→ history.shape = [4, V, 3, H, W], future.shape = [4, V, 3, H, W]
```

これに対し `row=20`(エピソード先頭近く)では履歴の先頭が `20 + (-30) = -10 < ep_start(0)` となり、`IndexError` が出て無効サンプル扱いになります。

このような関数群は、合成ローダ(`load_frame(row)` が「row の値で埋めたテンソル」を返す)で容易に単体テストできます。返ってきたテンソルの中身から「どの行を読んだか」を逆算し、`[20,30,40,50]` / `[60,70,80,90]` のように期待どおりかを assert します。境界をまたぐケースで `IndexError` が出ることも確認します。データセット本体(動画デコードや外部ライブラリ)に一切依存せずにロジックを保証できるのが、純粋関数 + コールバック設計の最大の利点です。

データセットへの組み込みはオプトイン(既定 OFF)にするのが安全な作法です。OFF のときは現在フレームのみ返す従来挙動と完全に同一にし、ON にしたときだけ `__getitem__` が履歴・未来を追加で返します。有効サンプルの余白計算は、他の用途で使う窓(例えば自車運動を見るためのエゴモーション窓)と World Model 窓の大きいほうを採用し、どちらの窓も完成する行だけを有効とします。

## 手法の系譜と主要論文
World Model と特徴空間での未来予測は、自己教師あり学習と自動運転研究の主要テーマです。系譜を辿ります。

World Models (David Ha & Jürgen Schmidhuber, NeurIPS 2018, arXiv:1803.10122) が現代的な起点です。彼らは、エージェントが環境のダイナミクス(状態がどう遷移するか)を内部モデルとして学習し、その「想像」の中で方策を訓練できることを示しました。構成は、観測を低次元に圧縮する VAE(変分オートエンコーダ)、時間遷移を予測する RNN(MDN-RNN)、それらの潜在表現上で行動を決める小さなコントローラ、の3点セットです。動機は「現実で試行錯誤すると高コストだが、学習した世界モデルの中なら安全・高速に練習できる」こと。ここで「過去の観測列から未来を予測する」という構造が定式化され、本トピックの窓構造はその入力(過去列)と教師(未来列)を供給する役割に当たります。

自動運転への展開として MILE (Anthony Hu et al., Wayve, NeurIPS 2022, arXiv:2210.07729) があります。MILE はカメラ映像から鳥瞰図(BEV)状態を予測する世界モデルを模倣学習で訓練し、CARLA シミュレータでオフラインデータのみから運転方策を学べることを示しました。続く GAIA-1 (Hu et al., Wayve, 2023, arXiv:2309.17080) は、映像・テキスト・行動を入力に未来の運転映像を生成する大規模な生成的世界モデルで、約 90 億パラメータ規模、数千時間の運転映像で訓練されたと報告されています。これらは「過去フレーム列を入力に未来を予測する」窓構造を前提とします。

予測の表現をめぐる系譜も重要です。初期の世界モデルは画素やそれに近い表現で未来を予測しましたが、「画素の細部(葉の揺れ、影のちらつき)まで当てるのは無駄が多く、意味のある抽象特徴を予測するほうが効率的で頑健」という発想から、特徴空間で予測する手法が現れます。JEPA / I-JEPA (Mahmoud Assran et al., Meta AI, CVPR 2023, arXiv:2301.08243) は画像の一部を隠し、画素ではなく埋め込み(特徴ベクトル)空間で隠れた部分を予測する自己教師あり学習を提案しました。V-JEPA (Adrien Bardes et al., Meta AI, 2024, arXiv:2404.08471) はこれを動画に拡張し、時空間的にマスクした領域の特徴を予測します。これらは「未来フレームの特徴を予測して正解特徴と比べる」という損失設計の直接の理論的基盤で、本トピックの未来窓 `future_frames` がそのターゲット(正解)になります。JEPA 系の共通課題は、ターゲット特徴を出すエンコーダが「常に同じ値を返す自明解(表現の崩壊、collapse)」に陥るのを防ぐ必要があることで、stop-gradient(ターゲット側に勾配を流さない)や EMA(指数移動平均でターゲットエンコーダをゆっくり更新する)といった工夫が併用されます。

時系列窓でデータリークを避ける重要性は、時系列予測・強化学習・オフライン RL のデータ設計全般で繰り返し指摘されてきました。エピソード境界をまたがせない、という本実装の制約はその実務的反映です。

## 論文の実験結果(定量データ)
代表的な定量結果を、指標の意味とともに挙げます。

World Models(Ha & Schmidhuber, 2018)は CarRacing-v0 という運転風タスクで、学習した世界モデルの「夢(内部シミュレーション)」の中だけでコントローラを訓練し、それを実環境に戻して平均スコア約 906(100エピソード平均)を達成したと報告しました。これは当時このタスクで「解けた(solved)」とされる基準(おおむね 900)を初めて安定して超えた結果で、「現実で一切練習せず、内部モデルの中だけで学んだ方策が通用する」ことを定量的に示した点が重要です。

MILE(Hu et al., 2022)は CARLA の標準評価で、オフラインの専門家データのみから学習したにもかかわらず、当時の最良手法を上回る運転スコア(route completion と違反ペナルティを組み合わせた driving score)を報告しました。特に「明示的な世界モデル(未来 BEV の予測)を学習に組み込むと、それを外した版より運転性能が向上する」というアブレーションを示し、未来予測を表現学習に組み込む価値を裏付けました。指標の driving score は「ルートをどれだけ完走したか × 信号無視や衝突などの違反でどれだけ減点されたか」を掛け合わせた総合指標で、高いほど安全に目的地へ到達できることを意味します。

I-JEPA(Assran et al., 2023)は ImageNet の線形評価(凍結した特徴の上に線形分類器だけを載せて精度を測る、表現の良さを測る標準手法)で、画素再構成型の手法(例 MAE)に匹敵〜上回る精度を、より少ない計算量で達成したと報告しました。特に「画素ではなく特徴空間で予測する」設計が、事前学習の計算効率(同じ精度に到達するまでの GPU 時間)を大きく改善する点を強調しています。V-JEPA はこれを動画に拡張し、動作認識(Kinetics-400 など)や時間推論を要するタスクで、凍結特徴のまま強い性能を示しました。これらの結果は「未来・欠損を特徴空間で予測する」損失が、画素予測より効率的で汎化しやすいという主張の定量的根拠です。

これらに共通するアブレーションの教訓は明快です。未来予測(世界モデル/JEPA 損失)を抜くと表現学習の効率や下流タスク性能が下がり、予測対象を画素から特徴へ移すと効率と頑健性が上がる、という方向性です。本トピックの窓供給は、こうした未来予測損失を実データ上で初めて回せるようにする前提インフラに当たります。

## メリット・トレードオフ・限界
メリット:
- World Model / JEPA に必要な「過去窓(入力)」と「未来窓(正解)」を供給でき、特徴予測損失を実データで回せるようになる。
- 純粋関数 + コールバック設計により、外部データやデコードライブラリ無しで単体テストできる。ロジック(オフセット計算・境界判定)を高い確度で保証できる。
- レート変換(例 10 Hz → 1 Hz、ストライド)を明示的に扱え、他レート(30 Hz など)にも一般化できる。
- エピソード境界チェックで別走行の混入(データリーク)を防ぎ、学習・評価の妥当性を守る。
- 既定 OFF のオプトインにすれば、既存のデータ読み込み挙動を一切壊さずに段階導入できる。

トレードオフ・限界:
- 1サンプルあたり「過去 N 枚 + 未来 N 枚 × 視点数」のフレームをデコードするため、I/O と計算が重い。本番学習では事前抽出(特徴をあらかじめ計算してディスクに保存)やキャッシュが推奨される。
- 窓に必要な前後余白ぶん、各エピソードの端のフレームは有効サンプルにならず使えない。エピソードが短いとサンプル数が大きく減る。
- ストライドで間引くと、その間のフレームの情報は捨てられる。世界モデルのレートを粗くするほど、速い動的事象(急な割り込みなど)を取りこぼすリスクがある。
- カメラの幾何(キャリブレーション)を伴う BEV 投影などは、窓の供給とは別問題として残る。窓は時間方向の切り出しのみを担い、空間方向の整合は別の仕組みに委ねる。

未解決の課題として、「最適な窓長 N とストライドをどう選ぶか」「未来予測の損失をどの特徴階層で取るか」「ターゲットエンコーダの崩壊をどう確実に防ぐか」は、JEPA 系・世界モデル系の研究で今も活発に検討されています。窓供給はその実験を可能にする土台であり、最適値そのものは下流のモデル設計と一体で探索する対象です。

## 発展トピック・研究の最前線
さらに学ぶ方向:

潜在空間の世界モデル: Dreamer 系列(Hafner et al., DreamerV1〜V3, 2019〜2023)は、潜在空間で未来を予測しながら方策をその「想像」の中で学習する強化学習手法で、世界モデルの代表的発展形です。Atari から実ロボット制御まで幅広いタスクで高い性能を示し、「世界モデルの中で計画・学習する」枠組みを成熟させました。

生成的世界モデルと運転シミュレーション: GAIA-1 の後、運転映像を生成する大規模世界モデル(GAIA-2、各種 driving video diffusion など)が相次ぎ、収集困難な危険シーンを合成してデータ拡張や閉ループ評価に使う研究が進んでいます。窓構造はこれらの「条件付け(過去)」と「生成対象(未来)」の境界として機能します。

自己教師あり表現学習の統合: V-JEPA 2(2024〜2025)など、動画から行動可能な表現を学ぶ研究は、世界モデルと表現学習の融合を進めています。「特徴空間で未来を予測する」という JEPA の原理と、「未来を想像して計画する」という世界モデルの原理が一つの枠組みに収束しつつあり、本トピックの過去窓・未来窓はその学習信号を作る共通部品です。

データエンジニアリングの最前線: 大規模な連続フレーム学習では、オンザフライのデコードがボトルネックになるため、WebDataset などのシャード形式での事前パッキング、特徴の事前抽出、効率的なデコーダ(ハードウェアデコード)の活用が研究・実務の焦点です。窓の余白計算とエピソード境界管理は、これら効率化と整合させながら実装する必要があります。
