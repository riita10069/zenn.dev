# MLPビュー変換 (VPN/PON)

## ひとことで言うと

カメラ画像を真上から見た地図(BEV, 鳥瞰図)に変換するとき、逆透視変換のような幾何の計算式を使うのではなく、全結合層(MLP = Multi-Layer Perceptron, 多層パーセプトロン。入力の全要素を出力の全要素につなぐ最も基本的なニューラルネット)に「画像のこの並びは BEV のこの並びに対応する」という対応関係そのものを学習で覚えさせる方式です。カメラの取り付け情報を式に入れる代わりに、データから暗黙に(陰に)幾何を学ばせます。

## 直感的な理解

カメラ画像から BEV を作る古典手法 IPM([逆透視変換 (IPM)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0280_inverse-perspective-mapping))には「世界は平らな地面」という強い仮定があり、立体物(車・歩行者)が倒れて歪む弱点がありました。深度を確率分布で扱う Lift-Splat-Shoot([深度分布スプラッティング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0277_depth-distribution-splatting))は深度推定が必要でした。

そこで「いっそ、幾何の式も深度推定も使わず、画像から BEV への対応づけ全体をニューラルネットに丸ごと学習させたらどうか」という発想が出ます。これが MLP ビュー変換です。動機の核心はこうです。

- カメラの透視投影では、画像の縦方向(行, 上下)が「地面上の前後方向の距離(奥行き)」に、横方向(列, 左右)が「地面上の左右方向」に、おおむね対応する。つまり画像座標と地面座標の間には強い規則性がある。
- ただしその対応は非線形で複雑。手で式を書くより、全結合層に大量の(画像, BEV)ペアを見せて学習させたほうが、平坦仮定のような無理な前提を置かずに済むかもしれない。

全結合層は入力の全要素と出力の全要素を結ぶので、原理的には「画像のどの位置の情報も、BEV のどの位置にでも運べる」柔軟さを持ちます。この柔軟さで、IPM が苦手な立体物の再配置も学習で吸収しよう、というのが狙いです。

## 基礎: 前提となる概念

全結合層(fully-connected / linear layer): 入力ベクトルの各要素に重みを掛けて足し合わせ、出力ベクトルの各要素を作る層です。入力 M 次元・出力 N 次元なら M×N 個の重みを持ちます。任意の入力位置を任意の出力位置に結べる反面、重みの数が入出力サイズの積で増えるのが特徴です。

平坦化(flatten): 2次元・3次元のテンソルを1本の長いベクトルに並べ直す操作。空間的な並びの情報を全結合層に渡すために使いますが、平坦化すると「隣のピクセル」という局所構造が暗黙になり、層がそれを重みで学び直す必要があります。

畳み込みとの違い: 畳み込み(CNN)は「近くのピクセルだけを見て、同じフィルタを全位置で使い回す(重み共有)」ので、平行移動に強くパラメータが少ない。一方、ビュー変換は画像座標→BEV座標という「位置によって変換のしかたが違う」処理なので、重み共有が前提の畳み込みでは表しにくい。だから位置ごとに別の重みを持てる全結合が使われた、という背景があります。

暗黙学習(implicit learning): カメラ幾何の式を明示的に与えず、データから対応関係を間接的に学ぶこと。柔軟だが、学習データの分布(カメラ配置・構図)に強く依存し、外れた状況への汎化が弱くなりがちです。

## 仕組みを詳しく

代表的な2手法で、やり方が少し違います。

VPN(View Parsing Network, Pan et al., 2020): 画像特徴を1本の長いベクトルに平坦化し、全結合層を通して BEV 特徴の並びに変換します。形状で追います。

```
画像特徴 [C, H, W]
 → 空間を平坦化 [C, H*W]
 → 全結合層で空間次元を変換 [C, H'*W']    (ここが「ビュー変換」)
 → BEV特徴 [C, H', W'] に整形
```

チャネル C はそのまま保ち、空間次元 H×W を別の全結合層で BEV の H'×W' に写像します。この空間変換の全結合の重みが「画像のどの位置が BEV のどの位置に効くか」という対応表そのものを学習で獲得します。カメラ幾何の式は一切登場しません。VPN は複数カメラ・複数モダリティをこの方式でそれぞれ BEV に変換してから融合し、その上で意味セグメンテーションを解きました。

PON(Pyramid Occupancy Network, Roddick & Cipolla, CVPR 2020): もう少し幾何に寄り添います。「画像の縦方向(行)は、地面上の奥行き(カメラからの距離)に対応する」という観察を使い、画像の各列(縦の1本のスライス)を、全結合層で BEV の奥行き方向の1本のレイ(光線)に変換します。これを各列について行い、扇状に並べて BEV を作ります。さらに近距離は細かく・遠距離は粗くという解像度の違いを、ピラミッド構造(複数スケールの特徴を使い分ける。バックボーンの各階層の特徴を、対応する距離帯の BEV に割り当てる)で扱います。なぜ列単位にしたか。画像の横位置(列)は地面の左右方向にほぼ対応するので、列ごとに独立に奥行きへ展開するのが幾何的に自然で、しかも全結合のパラメータを大幅に減らせるからです。

数値例で VPN の重さを実感します。画像空間が H×W = 32×32 = 1024、BEV が H'×W' = 32×32 = 1024 だと、空間変換の全結合だけで 1024×1024 ≈ 100万個の重みが要ります。解像度を2倍(64×64=4096)にすると 4096×4096 ≈ 1670万個へ急増。全結合は柔軟な反面、解像度の2乗でパラメータが膨らむのが弱点です。PON が列単位に分解したのは、この爆発を抑える工夫でもあります(列ごとなら、横方向は触らず、縦 H 次元 × 奥行きビン数だけで済む)。

```
VPN: 画像全体 ──全結合(密)── BEV全体     パラメータ ∝ (HW)×(H'W')
PON: 各列    ──全結合(列ごと)── 奥行きレイ   パラメータ ∝ H × depth
```

重要な構造的事実: 空間全結合の重み行列は、原理的には「BEV の各位置が、画像の全位置をどれだけ参照するか」という固定の重み表です。これは後述する通り、Transformer の密なクロスアテンションを「学習時に固定した版」と見ることができます。アテンションが入力依存で重みを動的に決めるのに対し、MLP は全位置対応を静的な重みとして焼き込みます。ここが汎化の弱さの根本でもあります。

## 手法の系譜と主要論文

VPN(Bowen Pan et al., 2020, "Cross-view Semantic Segmentation for Sensing Surroundings", RA-L / ICRA, arXiv:1906.03560): ビュー変換を学習可能なモジュール(View Transformer Module)として実装し、複数の視点・センサを跨いだセマンティックマップ生成を提案しました。なぜ全結合にしたか。視点間の対応は複雑で、明示的な幾何モデルに頼らずデータ駆動で学ぶほうが多様なセンサ配置に対応しやすいと考えたからです。シミュレーションで学んだ変換を実世界へ転移する工夫(sim-to-real。合成データで対応関係を学び、少量の実データで微調整)も提案しました。

PON(Thomas Roddick & Roberto Cipolla, CVPR 2020, "Predicting Semantic Map Representations from Images using Pyramid Occupancy Networks", arXiv:2003.13402): 画像の鉛直方向と地面の奥行きの対応に着目し、列ごとの変換 + ピラミッド構造で効率と多スケール対応を両立させました。単眼カメラから占有グリッド的な BEV セマンティックマップを生成できることを示しました。幾何の規則性を一部使うことで、純粋な全結合(VPN)より効率と汎化を改善しています。

HDMapNet(Li et al., ICRA 2022, arXiv:2107.06307)は、MLP ベースのビュー変換(各カメラ特徴を MLP で BEV に写してから複数カメラを融合)を地図要素のベクトル化に使い、MLP 変換が局所地図構築で実用になることを示した後発例です。

その後、MLP 系は主流の座を、深度推定の LSS([深度分布スプラッティング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0277_depth-distribution-splatting))やアテンション + 幾何投影の BEVFormer([学習可能BEVクエリ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0974_learnable-bev-queries))に譲りました。理由は、全結合の暗黙変換よりも、カメラ幾何を明示的に使うほうがキャリブレーションの知識を活かせて精度と汎化が良かったからです。とはいえ MLP ビュー変換は「ビュー変換を学習対象にする」という重要な転換点であり、深度推定も幾何式も使わない最もシンプルな学習ベース手法として、系譜の起点に位置づけられます。

## 論文の実験結果(定量データ)

評価指標は地図セグメンテーションの mIoU(予測領域と正解領域の重なり率、クラスごとに計算して平均、高いほど良い)が中心です。データセットは nuScenes、Argoverse、シミュレータ(CARLA)などが使われます。

PON は nuScenes と Argoverse で、単眼カメラから BEV のセマンティック占有マップ(道路・車・歩行者などのクラス別領域)を生成し、当時の比較手法に対して競争力ある mIoU を報告しました。特に IPM ベースのベースラインと比べ、立体物クラスで歪みが少なく、占有の予測がよりもっともらしいことを示しました。クラス別に見ると、道路や車線など面的なクラスで高く(road は数十%台)、小さく動的な歩行者・自転車では IoU が一桁%台にとどまる、という傾向が共通して見られます。これは「面的で頻出するものは学習しやすく、小さく稀なものは難しい」という、当時の BEV セグメンテーション全般の傾向を反映しています。

VPN は複数カメラ・複数モダリティから統合的な周囲セマンティックマップを作れることを示し、シミュレーションで学習した変換を実データへ転移する効果も報告しました。合成データのみで学んだ場合より、実データで微調整した場合に明確に mIoU が向上することが示され、暗黙変換が分布シフトに敏感であることが裏返しに見えます。

後発の LSS や BEVFormer との比較では、MLP 系は精度で劣ることが多いと報告されています。決定的な要因は2つ。第一に、空間全結合がカメラ配置に強く依存し、配置や構図が変わると汎化しにくいこと(学習時のカメラ姿勢を暗黙に重みへ焼き込むため)。第二に、解像度を上げるとパラメータが2乗で増えるため、高解像度の BEV を扱いにくいことです。これらの「アブレーション的な観察」が、その後の手法がカメラ幾何を明示的に使う方向へ舵を切る動機になりました。

数値の意味: mIoU が数ポイント違うというのは、地図上で「車や歩行者の領域をどれだけ正しく塗れたか」の差で、自動運転の状況認識の信頼性に直結します。MLP 系がここで伸び悩んだことが、「幾何という強い事前情報を捨てるのはもったいない」という分野共通の教訓になりました。

## メリット・トレードオフ・限界

メリット

- 幾何の式も深度推定も不要。最もシンプルな学習ベースのビュー変換。
- 全結合は柔軟で、原理的には画像の任意位置を BEV の任意位置に運べる(立体物の歪みも学習で吸収しうる)。
- カメラ校正の正確な値がなくても、データさえあれば対応を学べる(キャリブレーションが不確かな状況での選択肢になる)。

トレードオフ・限界

- 空間全結合のパラメータが解像度の2乗で増え、高解像度 BEV では非現実的(PON が列分解で緩和したのはこのため)。
- 学習したカメラ配置・構図に強く依存し、配置が変わると作り直しが必要(汎化が弱い)。新しいカメラリグへのゼロショット転移はほぼ不可能。
- 幾何を明示的に使わないため、キャリブレーション情報という強力な手がかりを活かせない。
- 後発の幾何 + アテンション手法・深度推定手法に精度で劣ることが多い。

研究上の未解決課題: 「幾何の事前情報をどこまで使い、どこから学習に任せるか」という配分の問題は、MLP 系が極端に「全部学習」に振った結果として明確になりました。現代の手法はこの中間(幾何で大枠を決め、学習で残差を埋める)を探っており、MLP 系はその振り子の片端として理解すると位置づけが鮮明になります。

## 発展トピック・研究の最前線

幾何と学習のハイブリッド: PON の「幾何の規則性を一部使う」発想は、後の手法に受け継がれました。カメラ幾何で投影位置を決め、学習でオフセットやアテンション重みを補正する BEVFormer 系は、MLP の「全部学習」と IPM の「全部幾何」の中間と見なせます。

Transformer による空間変換: 全結合の空間変換は、見方を変えると「各 BEV 位置が画像全体を見る密なアテンションを、学習時に固定重みへ焼き込んだもの」に近い。これを入力依存・効率化したのが変形可能アテンションで、MLP ビュー変換からアテンション系への橋渡しになっています。アテンションは「重みを動的に・スパースに決める」ことで、MLP の汎化の弱さとパラメータ爆発の両方を緩和しました。

軽量・組み込み向けの再評価: 全結合や列単位 MLP は計算が単純で実装が軽いため、計算資源の限られた組み込み環境では、解像度を抑えた MLP 系が今も実用的な選択肢になる場面があります。固定カメラリグで運用が閉じている用途(特定車種の量産 ADAS など)では、汎化の弱さが問題になりにくく、推論の軽さが活きます。
