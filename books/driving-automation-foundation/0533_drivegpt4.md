# DriveGPT4

## ひとことで言うと
DriveGPT4 は、車載カメラの映像を入力すると、「いま何が起きていて、なぜそう運転するのか」を人間の言葉で説明しながら、同時にアクセルやハンドルの操作量(低レベル制御)まで出力する運転AIです。大規模言語モデル(LLM、大量の文章で学習し言葉を扱うAI)を運転に本格的に使った初期の代表例で、後の VLM(Vision-Language Model、画像も言葉も扱うAI)運転研究の出発点になりました。「運転を言葉で説明する」と「実際に操作する」を一つのモデルで一気通貫にやる、という発想が核心です。

## 直感的な理解

教習所の指導員を思い浮かべてください。良い指導員は、運転しながら「いま前に子どもが見えたから速度を落とすよ」と理由を声に出してくれます。生徒は操作だけでなく、その背後にある判断の根拠まで学べます。従来の自動運転AIは、操作はできても、この「声に出して理由を言う」部分が完全に欠けていました。DriveGPT4 は、まさにこの指導員のように、運転操作と同時にその理由を言葉で語らせよう、という試みです。

なぜそれが必要なのでしょうか。従来の end-to-end 運転モデル(カメラ画像を入れたら直接ハンドル・アクセルを出すニューラルネット)には大きな弱点が2つありました。

1つ目は「ブラックボックス問題」です。モデルがなぜその操作を選んだのかが、出力された数値(ハンドル角度など)からは一切わかりません。急ブレーキをかけたとき、「前に歩行者がいたから」なのか「学習データの偏りで誤作動しただけ」なのかを人間が判断できません。自動運転は人命に関わるため、判断の理由を説明できること(解釈可能性、interpretability)が強く求められます。事故が起きたときの責任追及や、開発中のデバッグにも、理由が見えることは決定的に重要です。

2つ目は「常識・推論の欠如」です。従来モデルは「この画像のときはこう操作する」という対応を大量に暗記しているだけで、見たことのない状況に「人間ならこう考えるはず」という推論ができませんでした。「ボールが転がってきたら子どもが追ってくるかもしれない」といった常識的な先読みができないのです。

一方、ChatGPT に代表される LLM は、文章の中の常識や因果関係を大量に学習しており、質問に対して理由を言葉で説明できます。DriveGPT4 は「この LLM の推論力と説明力を、運転にそのまま持ち込めないか」という発想から生まれました。

## 基礎: 前提となる概念

DriveGPT4 を理解するには、いくつかの前提知識が要ります。順に噛み砕きます。

トークン(token): LLM が文章を処理する最小単位です。単語そのものではなく「単語の断片」のことが多く、たとえば "slowing" は "slow"+"ing" のように分割されます。LLM は文章をトークン列に変換し、「次に来るトークンは何か」を1つずつ予測することで文章を生成します。

自己回帰生成(autoregressive generation): LLM が文章を作るときの仕組みです。これまでに生成したトークン列を入力に、次の1トークンを予測し、それをまた入力に加えて次を予測する……という逐次的な繰り返しです。1トークンずつしか出せないため、長い出力ほど時間がかかる、という遅さの原因にもなります。

マルチモーダル LLM / 視覚的指示チューニング(visual instruction tuning): もともと LLM は文章しか扱えません。そこに画像を入れるには、画像を「視覚トークン」という、言葉のトークンと同じ土俵に乗る数値列に変換する必要があります。LLaVA(Liu et al., NeurIPS 2023)はこの方式を確立した代表的な研究で、画像エンコーダの出力を小さな変換層(projection)で LLM の入力空間に写し、「画像+質問→回答」の形式で指示チューニング(instruction tuning、指示に従う形で微調整すること)を行います。DriveGPT4 のマルチモーダル構造はこの LLaVA 系の系譜にあります。

BDD-X データセット(Kim et al., ECCV 2018): 運転映像に、行動(braking、turning など)と「なぜその行動をしたか」のテキスト説明が人手で付与されたデータセットです。「映像→行動+理由」の教師データとして使えるため、運転を言葉で説明させる研究の土台になっています。

これらを踏まえると、DriveGPT4 は「LLaVA 系のマルチモーダル LLM を、BDD-X をもとに運転用に指示チューニングしたもの」と要約できます。

## 仕組みを詳しく

処理の流れを段階で分解します。

ステップ1: 入力。連続する複数フレームの車載カメラ映像(動画)と、ユーザーからのテキスト質問(例: "What is the current action and why?")を受け取ります。動画を使うのは、1枚の静止画では「車が止まっているのか走っているのか」が判別できないからです。速度や加減速は時間方向の変化からしか読み取れません。

ステップ2: 映像のトークン化。映像を視覚エンコーダ(画像を数値ベクトルに変換するネット、CLIP の画像側などがよく使われる)に通し、さらに動画の時間情報をまとめる仕組みを使って、画像を視覚トークンに変換します。これで LLM は画像と言葉を区別なく一連の系列として扱えます。

ステップ3: LLM による生成。視覚トークンとテキストの質問トークンをつなげて LLM に入れると、LLM は普通の文章生成と同じ自己回帰生成で、トークンを1つずつ予測して出力します。DriveGPT4 の特徴は、出力が2種類混在することです。

- 自然言語の説明: 「前方に歩行者がいるため減速しています」のような文章。
- 低レベル制御信号: 次の時刻の車速(speed)と操舵(turning angle)の予測値。これらの数値もテキストのトークンとして生成します(数字を文字として吐き出すイメージ)。

つまり「説明」と「操作」を1つの言語モデルから一気通貫で出すのが核心です。出力イメージ:

```
Q: 今の行動とその理由は?
A: I am slowing down because a pedestrian is crossing
   ahead. Predicted speed: 3.2 m/s, turning angle: 0.0 deg.
```

ステップ4: 学習データ作り。LLM を運転用に学習させるには「映像+質問+(説明と制御を含む)回答」のペアが大量に要ります。DriveGPT4 は BDD-X(約2万6千クリップ、行動と理由のテキスト付き)をベースに、ChatGPT を使って質問応答ペアを大量に自動生成しました(報告によるとおよそ5万6千件規模の指示チューニング用対話データ)。具体的には、BDD-X の行動ラベルと理由テキスト、加えて検出済みの物体情報を ChatGPT にプロンプトとして与え、「このシーンについて人間が尋ねそうな質問とその回答」を多数生成させます。人手では作りきれない量の対話形式の運転教師データを、既存の説明注釈から増幅したわけです。これにより、固定された質問だけでなく、多様な問いかけに答えられるようになります。

学習は2段階で行われます。第1段階では、画像エンコーダ(CLIP 系)の出力を LLM の入力空間へ写す projection 層を、一般的な画像テキストペア(画像キャプションなど)で学習し、視覚トークンと言語トークンの土俵を合わせます。第2段階では、上記の運転用指示チューニングデータで LLM 本体を微調整し、運転に特化した説明と制御を出せるようにします。この「汎用マルチモーダル整合 → タスク特化微調整」の二段構えは LLaVA 系の標準的な学習レシピを運転に持ち込んだものです。

制御信号のトークン化について補足します。速度や操舵角は連続値ですが、DriveGPT4 はこれを数字列(例 "3.2")として文字トークンに分解し、言語と同じ語彙で生成します。専用の回帰ヘッド(連続値を1本のニューロンで直接出す層)を使わないのは、説明文と制御を完全に同一の自己回帰系列として扱えるようにするためです。代償として数値の分解能はトークン化の粒度に縛られますが、説明と制御の生成が一貫した1つの過程になるという設計上の利点があります。

## 手法の系譜と主要論文

DriveGPT4 (Xu, Zhang, Liu, Hong, Yang, Li ら, "DriveGPT4: Interpretable End-to-end Autonomous Driving via Large Language Model", arXiv 2023、のち IEEE Robotics and Automation Letters (RA-L) に採録)の位置づけを、前後の研究とともに整理します。

- 土台となった研究: LLaVA(Liu et al., NeurIPS 2023)が視覚的指示チューニングを確立し、画像を扱える LLM の作り方を一般化しました。BDD-X(Kim et al., ECCV 2018)が「運転を言葉で説明する」教師データを提供しました。DriveGPT4 はこの2つを掛け合わせ、ChatGPT によるデータ増幅を加えた構成です。

- DriveGPT4 の新規性: 映像を入力に、自然言語の説明と低レベル車両制御(速度・操舵)を同一の LLM から end-to-end に生成する初の枠組みであること。それまでの説明系研究(BDD-X 上の captioning など)は「説明」だけ、制御系研究は「操作」だけで、両者を一つのモデルに統合した点が新しい。

- 後続研究との関係:
  - LMDrive(Shao et al., CVPR 2024): DriveGPT4 が苦手だった「閉ループ・実時間制御」を CARLA シミュレータ上で実現した後続研究。カメラ+LiDAR と言語指示で走り続ける。詳細は [LMDrive](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0541_lmdrive)。
  - DriveVLM(Tian et al., CoRL 2024): 遅い VLM と速い従来プランナを併用する System1/System2 構成で、実時間性を確保しつつ深い推論を狙う。詳細は [DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm)。
  - EMMA(Hwang et al., 2024、Google/Waymo): マルチモーダル基盤モデルをそのまま運転に使い、知覚・予測・計画を言語空間で統一する大規模な発展形。詳細は [EMMA (End-to-End Multimodal Model for Autonomous Driving)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0536_emma_note)。
  - DriveLM(Sima et al., ECCV 2024): 運転シーンに対する Graph-of-Thought 形式の質問応答ベンチマークで、説明可能運転の評価基盤を整えた。詳細は [DriveLM (Graph VQA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0534_drivelm)。

この系譜は、DriveGPT4 が「LLM を運転に持ち込み、説明と制御を統合する」という旗を立て、その後の研究が「閉ループ化」「実時間化」「大規模化」「評価基盤整備」という各方向に展開していった、という流れを示します。

## 論文の実験結果(定量データ)

DriveGPT4 の評価は主に BDD-X 上で、(a) 行動説明の質、(b) 制御信号の予測精度、の2軸で行われました。指標を初心者向けに説明します。

説明文の質を測る指標:
- BLEU(B4)/ CIDEr / METEOR / ROUGE: いずれも「モデルが生成した文章が、人間の正解説明とどれだけ言葉として一致するか」を測る、機械翻訳・キャプション生成由来の自動評価指標です。値が高いほど正解に近い文章という意味です。BLEU4 は4単語連続(4-gram)の一致を、CIDEr は人間にとって重要な語に重みを付けた一致を測ります。DriveGPT4 は、BDD-X 上の行動説明(action description)・行動根拠(justification)の生成で、従来手法(ADAPT などの注意機構ベースのキャプション生成モデル)を上回るスコアを報告しました。報告値では、行動説明で CIDEr がおよそ 300 前後、行動根拠の生成でも従来の captioning ベースラインを CIDEr・BLEU4 ともに上回り、特に説明の説得力に効く CIDEr で明確な差を示しています(数値は論文・後続比較による)。
- ChatGPT スコア / 人間評価: 自動指標だけでは「もっともらしいが的外れな説明」を見抜けないため、ChatGPT を審査員にして説明の妥当性を採点する、あるいは人間が評価する方式も併用しました。ここでも従来手法より高い評価を得たと報告されています。

制御の精度を測る指標:
- RMSE(二乗平均平方根誤差): 予測した速度・操舵が、実際の人間の操作からどれだけ外れているかの誤差です。誤差を二乗して平均し平方根を取るので、大きく外したサンプルを強く罰します。小さいほど良い。DriveGPT4 は速度・操舵の予測で、当時のベースライン(ADAPT 等)と同等かそれ以上の RMSE を達成したと報告しました。「数値をテキストで吐く」方式でも、専用回帰ヘッドに対して致命的に劣るわけではない、と示した点が意義です。
- しきい値内一致率(A_τ など): 予測が実際の操作から一定の許容範囲 τ 内に収まった割合。RMSE が外れ値に引っ張られるのを補う指標で、実用上「だいたい合っている」割合を見ます。DriveGPT4 は速度・操舵の両方で高い一致率を報告しています。

汎化性の検証: DriveGPT4 は、学習時になかった種類の質問(zero-shot な問いかけ)にもある程度答えられることを定性的に示しました。これは LLM 由来の汎化能力で、従来の固定タスクモデルにはない性質です。

アブレーション的な知見: 制御信号をテキストトークンとして生成する設計が、説明文との同時生成と矛盾しないことを示しています。ただし論文自身が認めるとおり、評価はあくまで BDD-X 由来のオフライン(録画データに対する)予測が中心で、車を実際に走らせ続ける閉ループ性能や、実車での実時間制御は検証範囲外です。

## メリット・トレードオフ・限界

メリット

- 運転判断の理由を自然言語で説明でき、解釈可能性が高い。デバッグや事故分析に直結する価値がある。
- 質問応答形式で、人間が対話的に「なぜ?」と確認できる。
- LLM の常識・推論力により、学習データにない状況へある程度汎化できる。
- 説明と制御を1モデルから出すため、説明と実際の行動が乖離しにくい(別々のモデルだと「言っていることとやっていることが違う」が起きうる)。

トレードオフ・限界

- LLM の自己回帰生成は遅く(説明文を出すには数十〜数百トークンを逐次生成する)、運転に必要な毎秒10回以上の判断には間に合わない。実時間の閉ループ運転には重すぎる。
- 制御信号をテキストで出すため、専用の制御回帰ヘッドより数値の精度・分解能で不利になりがち。
- 学習データを ChatGPT で自動生成しているため、その質に依存し、もっともらしいが誤った説明(hallucination、幻覚)を出すリスクがある。生成された説明が実際の運転判断と無関係な「後付けの作文」になっている可能性も否定できない。
- 主にオフライン評価で、実車・実時間制御での安全性検証は今後の課題。

研究上の未解決課題: 「モデルが出した説明は、本当にモデルの内部判断を反映しているのか(faithfulness、忠実性)」という問題が根深いです。説明と制御が同じモデルから出ても、説明が制御の本当の原因とは限りません。また、説明の質を測る BLEU/CIDEr といった自動指標は「言葉の一致」しか見ておらず、「説明が運転判断として正しいか」を直接測れていません。説明可能運転の評価方法そのものが、いまだ発展途上です。

## 発展トピック・研究の最前線

- 閉ループ化: DriveGPT4 のオフライン中心という限界を、シミュレータ上で実際に走らせ続けることで克服する流れ([LMDrive](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0541_lmdrive))。
- fast-slow / System1-System2 アーキテクチャ: 重い VLM を常時使うのではなく、難所だけ呼び出して速いプランナと協調させることで実時間性を確保する設計([DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm))。
- 蒸留(distillation): 重い VLM/LLM にシーンの判断や説明を生成させ、それを軽量モデルへ蒸留して、推論コストを払わずに「言葉で考える能力」の一部を移す方向。自動運転の実用化に向けた有力路線です。詳細は [知識蒸留損失](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0633_knowledge-distillation-loss)。
- 大規模基盤モデル化: Waymo の EMMA のように、巨大なマルチモーダル基盤モデルを運転タスク全体に適用する方向([EMMA (End-to-End Multimodal Model for Autonomous Driving)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0536_emma_note))。
- 評価ベンチマークの整備: DriveLM、nuScenes-QA など、運転シーン理解と説明を体系的に評価するベンチマークが整いつつあります([DriveLM (Graph VQA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0534_drivelm))。
