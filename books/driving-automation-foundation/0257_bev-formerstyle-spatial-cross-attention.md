# BEVFormer 型 空間クロスアテンション

## ひとことで言うと

BEV (鳥瞰図) の地面のマス目 1 つ 1 つに「質問役のベクトル (BEV クエリ)」を置き、そのマス目が指す 3 次元地点を各カメラ画像に幾何で射影して、射影先の画像特徴を拾い集めて足し合わせる仕組みです。これが BEVFormer の中核をなす「空間クロスアテンション (spatial cross-attention)」で、アテンションベース視点変換の代表的な実装です。深度を明示的に予測せず、幾何が示す「見るべき場所」へ注意を集中させることで、深度ラベルなしに複数カメラを 1 つの BEV に融合します。

## 直感的な理解

複数のカメラで撮った画像を、真上から見下ろした 1 枚の地図 (BEV) にまとめたい、という状況を考えます ([BEV知覚 概論](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0928_bev-perception-overview))。素朴な発想は「画像から BEV を作る関数」をネットワークに丸ごと学ばせることですが、それでは校正という既知の幾何を捨てることになり、学習が重く汎化も弱くなります ([幾何ベース vs 学習ベースの視点変換](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0966_geometric-vs-learned-view-transform))。

そこで BEVFormer は発想を逆転させます。「画像から BEV を作る」のではなく「BEV 側から画像に問い合わせる」。BEV の各マス目が「私の場所には何が写っている?」と各カメラに聞きに行き、幾何 (校正) が「あなたの場所なら前カメラのこの辺を見なさい」と射影先を教える。クエリは射影先の周辺だけを見ればよいので、画像全体を総当たりするより圧倒的に軽く、しかも深度ラベルなしで動きます。

レストランで料理を探す比喩で言えば、深度ベースは「全料理を客席に運んでから選ぶ (push)」、このアテンションベースは「客が『私の席から見える範囲のおすすめは?』と店員に聞き、店員が席の位置 (幾何) から該当する厨房の一角だけを案内する (pull)」イメージです。聞く側が場所を指定するので、無駄な探索が要りません。

## 基礎: 前提となる概念

クロスアテンション (cross-attention): アテンションとは「クエリ (質問する側) が、キー・バリュー (答えを持つ側) のうち関連の強いものに重みをかけて情報を集める」仕組みです。クエリとキー・バリューが同じ系列から来るものをセルフアテンション、別物から来るものをクロスアテンションと呼びます。ここではクエリ = BEV マス目、キー・バリュー = カメラ画像特徴という別物なので、クロスアテンションです。「空間 (spatial)」は、参照する場所を 3 次元空間の幾何で決める、という意味です。

BEV クエリ (BEV query): BEV グリッドの各マス目に 1 個ずつ割り当てる学習可能なベクトルです。`nn.Embedding(bev_h * bev_w, embed_dim)` のように、マス目数 × 次元の埋め込みとして持ちます。学習を通じて「このマス目はこういう特徴を集めたい」という性質を獲得します。

変形可能アテンション (deformable attention): 通常のアテンションが全位置を総当たりするのに対し、各クエリが「参照点の周辺の少数の点だけ」をサンプリングするアテンションです。サンプル位置を参照点からのオフセット (ずれ) で表し、そのオフセットと重みをクエリから直接予測します。計算量を大幅に削減し、収束も速くなります。

双線形補間サンプリング (bilinear sampling): 画像特徴マップの「ピクセルとピクセルの間」の任意座標で特徴を取り出す操作です。周囲 4 ピクセルを座標の近さで重み付け平均します。射影先が整数ピクセルにきれいに乗らないので必須で、PyTorch では `F.grid_sample` がこれを担います ([affine_grid と grid_sample による微分可能ワープ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0912_affine-grid-grid-sample-warp))。

ピラー (pillar): BEV の 1 マスが表す地面の点を、高さ違いで縦に数点並べたものです。地面付近から空中まで何点か取ることで、車・歩行者・標識など高さの違う物体を捉えられます。深度を推定する代わりに、複数の高さの射影先をアテンションが重み付けすることで、暗黙の深度推定を実現します ([3D参照点とピラー（高さ方向）サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0909_3d-reference-points-pillar-sampling))。

## 仕組みを詳しく

代表的な実装の処理を段階分解します。記号で表すと、入力は各カメラの画像特徴 `[B*V, C, H, W]` (B = バッチ、V = カメラ台数、C = チャンネル、H/W = 特徴マップの高さ・幅)、出力は BEV 特徴 `[B, C, bev_h, bev_w]` です。以下のコードは特定の実装ではなく、この方式を組むときに典型的に現れる処理の骨格です。

### 1. BEV クエリを用意する

```python
bev_queries = nn.Embedding(bev_h * bev_w, embed_dim)         # 例: 200*200 個 × 256 次元
queries = bev_queries.weight.unsqueeze(0).expand(B, -1, -1)  # [B, N, C], N = bev_h*bev_w
```

各マス目が学習可能な C 次元クエリを 1 個ずつ持ちます。BEV 解像度が高いほど N が大きくなり、このクエリ群が巨大なパラメータ塊になります。例えば 200×200×256 で約 1,000 万パラメータ、解像度を 450×300 まで上げると 450×300×256 ≈ 3,460 万パラメータと、解像度の二乗オーダーで膨らみます。

### 2. 画像特徴をバリューに射影する

```python
feat = image_feats.reshape(B, V, C, H*W).permute(0, 1, 3, 2)  # [B, V, H*W, C]
values = value_projection(feat)                               # [B, V, H*W, C]
```

アテンションが集めてくる「中身 (バリュー)」を線形層で整えます。

### 3. 3D 参照点を作って画像へ射影する

各 BEV セルに高さ違いの 3D 点 (ピラー、例えば 4 点) を割り当て、知覚範囲 (pc_range、例: 前後左右 ±50m、高さ -5〜3m) で実座標にスケールし、カメラ射影行列で各カメラ画像に飛ばします (詳細は [3D参照点とピラー（高さ方向）サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0909_3d-reference-points-pillar-sampling) と [幾何ベース vs 学習ベースの視点変換](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0966_geometric-vs-learned-view-transform))。

```python
ref_2d, mask = project_to_2d(reference_points_3d, camera_params)
# ref_2d: [B, V, N, num_z, 2] (正規化済み [0,1]), mask: [B, V, N, num_z] (可視か)
```

射影で深度がゼロ以下や画像外に出た点は、可視マスクで除外します。後方の BEV セルが前カメラを誤参照しないための物理的な拘束です。

### 4. クエリからサンプリングオフセットとアテンション重みを予測する

```python
offsets = sampling_offsets(queries).reshape(B, N, num_z, 2) * offset_scale  # 例: 0.1 倍
attn_weights = attention_weights(queries).reshape(B, N, V, num_z).softmax(dim=-1)
```

オフセットは「幾何が示した射影点から少しずらして見る」微調整で、変形可能アテンションの肝です。`offset_scale` を小さく取って動きを抑え、学習を安定化させるのが定石です。アテンション重みはピラーの高さ方向にソフトマックスを掛け、「どの高さが効くか」を学びます。これが暗黙の深度推定にあたります。深度ベースが確率分布の山として奥行きを表すのに対し、ここでは「複数の高さに飛ばした射影点への注意の重み」が奥行きの役目を担う、という対応関係になっています。

### 5. カメラごとにサンプリングして足し合わせる

```python
for v_idx in range(V):
    sample_locs = ref_2d[:, v_idx] + offsets        # 射影点 + オフセット
    # オフセット後にもう一度画像内かを判定し、元の可視マスクと AND を取る
    sample_grid = sample_locs * 2 - 1               # [0,1] → grid_sample の [-1,1]
    sampled = F.grid_sample(feat_v, sample_grid, mode='bilinear',
                            padding_mode='zeros', align_corners=False)
    # 可視点だけに重みを掛け、可視点で重みを正規化し直して加重和
output = output / visible_count.clamp(min=1.0)      # 可視カメラ数で平均 (BEVFormer の 1/|V_hit|)
```

BEVFormer 本家の変形可能アテンションは専用 CUDA カーネル (MSDeformAttn) で実装され高速ですが、移植性を重視する実装では `F.grid_sample` で代替するのが一般的です。原理は同じ「参照点 + 予測オフセットで疎にサンプリングし、予測重みで足す」です。`grid_sample` 版は CUDA カーネルより遅い反面、フレームワーク標準だけで動き、任意のデバイスや精度設定 (bf16 など) に乗せやすい利点があります。

### 6. 残差 + LayerNorm + FFN

```python
output = norm1(queries + output_projection(output))   # 残差接続 + 層正規化
output = norm2(output + ffn(output))                  # FFN + 残差 + 層正規化
output = output * (visible_count > 0).float()         # 観測ゼロのセルは 0 に
bev_features = output.reshape(B, bev_h, bev_w, C).permute(0, 3, 1, 2)  # [B, C, bev_h, bev_w]
```

どのカメラからも見えないマス目は、層正規化や FFN のバイアスで非ゼロになってしまわないよう、最後に明示的にゼロでマスクします。「見えていない場所は空白」という物理的な整合を保つ工夫です。本家 BEVFormer はこのエンコーダ層を 6 段重ね、各層でマルチヘッドアテンションと時間融合を行いますが、軽量な実装では単層・シングルヘッド・時間融合なしまで簡略化することもあります。簡略版は計算が軽い反面、表現力は本家より落ちます。

## 手法の系譜と主要論文

この仕組みの源流は 2 つの流れの合流点にあります。

第一の流れは DETR (Carion ら, ECCV 2020, arXiv:2005.12872) に始まる「クエリベースの物体検出」です。学習可能なクエリが画像全体にアテンションを張る発想が、後の BEV クエリの原型になりました。ただし DETR のアテンションは全位置総当たりで計算量が O(N²) と重く、収束も遅い (500 エポック規模) という問題がありました。

これを解決したのが Zhu ら "Deformable DETR" (ICLR 2021, arXiv:2010.04159) です。各クエリが参照点周辺の少数 K 点だけをサンプリングし、重みも Q-K 内積ではなくクエリからの線形層で直接予測する変形可能アテンションを提案しました。動機は計算削減と収束高速化で、効果は O(N×K) への計算削減でした。トレードオフは専用 CUDA カーネルが必要なこと。BEVFormer の空間クロスアテンションは、この「参照点 + 予測オフセット + 予測重み」の原則をそのまま借用しています。

第二の流れは 3D 検出のクエリ化です。Wang ら "DETR3D" (CoRL 2021, arXiv:2110.06922) は、3D の物体クエリを画像に射影して特徴をサンプリングする枠組みを示しました。これが「3D 参照点を画像に飛ばす」という空間クロスアテンションの核心アイデアを提供しました。

両者を統合し BEV 表現として完成させたのが Li ら "BEVFormer" (ECCV 2022, 上海 AI Lab ほか, arXiv:2203.17270) です。提案は「学習可能 BEV クエリ + 空間クロスアテンション (各カメラへ参照点を射影してサンプル) + 時間的セルフアテンション (過去 BEV を参照)」からなる時空間 Transformer です。明示的な深度予測を避けつつ幾何を参照点として活用し、深度ラベルなしで頑健に複数カメラを融合することを狙いました。

その後 Hu ら "UniAD" (CVPR 2023, arXiv:2212.10156) は BEVFormer を BEV エンコーダとして採用し、検出・追跡・地図・予測・計画までを 1 本のネットワークに統合しました。これにより BEV 表現が「知覚の出力」から「知覚から計画までを貫く共通基盤」へと格上げされ、空間クロスアテンションは end-to-end 自動運転の中核部品として広く使われるようになりました。UniAD は CVPR 2023 のベストペーパーに選ばれ、BEV を計画まで貫く設計の影響力を象徴しています。

## 論文の実験結果 (定量データ)

評価は主に nuScenes (6 カメラ + LiDAR の都市運転データセット、約 1,000 シーン) で行われます。mAP は検出の正確さ (高いほど良い)、NDS は位置・大きさ・向き・速度・属性の誤差を統合した nuScenes 総合指標 (0〜1、高いほど良い)、IoU は地図セグメンテーションの重なり度合いです。

BEVFormer は nuScenes の 3D 物体検出と地図セグメンテーションで、当時の SOTA に到達したと報告されました。nuScenes test 分割で NDS およそ 0.56、mAP およそ 0.48 と報告され、カメラのみの手法でありながら、複数フレームの時間融合を加えることで LiDAR を使う一部手法に迫る精度を示した点が注目されました。

アブレーションが手法理解に重要です。論文では時間的セルフアテンション (過去フレームの BEV を参照する仕組み) を外すと、速度推定の誤差 (mAVE、値が小さいほど良い) が顕著に悪化することが報告されています。これは「物体が動いているかどうかは、1 フレームだけでは分からず、時間情報が必要」という直感を定量的に裏づけます。また、空間クロスアテンションの参照点サンプリングを素朴な全結合写像 (幾何を使わない学習ベース) に置き換えると検出精度が落ちることも示され、「幾何で見る場所を絞る」設計の有効性を支えています。

Deformable DETR のアブレーションも系譜の理解に役立ちます。報告では、変形可能アテンションは通常の DETR に比べて収束エポック数を約 10 分の 1 程度 (500 → 50 エポック規模) に短縮し、とくに小物体の検出精度を改善しました。サンプリング点数 K を増やすと精度は緩やかに上がるが計算も増える、というトレードオフも示されており、BEV 側で「ピラーの点数をいくつにするか」を決める際の指針になっています。

## メリット・トレードオフ・限界

メリットは次の通りです。幾何 (参照点) が「見るべき場所」を与えるので、画像全体を総当たりするより軽く、深度ラベルも不要です。可視マスクにより、物理的に見えないカメラからは情報を取りません (後方のマス目が前カメラを参照しない)。`grid_sample` 実装にすれば専用 CUDA が不要で移植性が高い。学習可能オフセットで射影点の周辺を柔軟に探索でき、校正の小さな誤差を吸収しうる。

トレードオフと限界は次の通りです。BEV クエリは解像度の二乗 × 次元でパラメータが膨らみ、高解像度では巨大になります (例えば 450×300×256 で約 3,460 万パラメータ)。単層・シングルヘッド・時間融合なしの簡略版にすると、本家ほどの表現力は出ません (本家は 6 層エンコーダ + マルチヘッド + 時間融合)。正しい射影には本物のカメラ校正が必須で、校正がない場合の疑似射影フォールバックは数値が不安定 (ランダムに近い入力で NaN に至りやすい) です。高解像度 BEV は下流のモジュールでメモリを圧迫し、bf16 (16 ビット浮動小数の混合精度) や解像度削減が必要になります。研究上の未解決課題として、校正ドリフトへの頑健性、超高解像度 BEV のメモリ効率、時間融合の長期化 (多数フレームを安価に統合する) が挙げられます。

## 発展トピック・研究の最前線

BEVFormer 以降、空間クロスアテンションは複数の方向に発展しました。BEVFormer v2 はバックボーンの近代化と 2D 補助監督で精度を底上げし、視点変換が強力なバックボーンとどう協調するかを示しました。時間融合の効率化では、多数の過去フレームを再帰的に蓄積する手法 (long-term fusion) が研究され、メモリを抑えつつ長い時間文脈を取り込む工夫が進んでいます。

また、BEV を平面に潰さず高さ方向も保持する 3D 占有 (occupancy) 予測へ空間クロスアテンションを拡張する流れ (TPVFormer, OccFormer 系) があり、地面平面仮定の限界を超えようとしています。センサ融合では、同じ BEV 空間で LiDAR・radar の特徴と統合する設計が一般化し、視点変換は単一センサの前処理から多センサ統合の土台へと役割を広げています。end-to-end 自動運転 (UniAD やその後継 VAD など) では、この BEV 表現を計画まで一気通貫で使う設計が標準になりつつあり、空間クロスアテンションの出力品質がスタック全体の性能を左右する中核要素として研究が続いています。
