# 事前抽出特徴キャッシュ設計

## ひとことで言うと
学習のたびに毎回やる「動画やデータを読み込んで→デコードして→画像にして→特徴を作る」という重い前処理を、学習の前に1回だけ済ませて結果をファイルに保存(キャッシュ)しておき、学習時はその保存済みデータをただ読むだけにする設計です。同じ前処理を何百回も繰り返す無駄をなくし、高価なGPUを「データの準備待ち」で遊ばせないようにします。深層学習における「データ供給がボトルネックになる(data stall)」問題への、最も直接的な対処法のひとつです。

## 直感的な理解
毎朝コーヒーを淹れるのに、その都度コーヒー豆を畑から収穫し、乾燥させ、焙煎し、挽いていたら、1杯飲むのに何日もかかります。普通は、収穫から焙煎・粉砕までを一度に済ませて「挽いた粉」の状態で保存しておき、毎朝はお湯を注ぐだけにします。豆の状態は飲むたびに変わらないのだから、毎回ゼロからやり直すのは純粋な無駄です。

深層学習の学習でも同じことが起きています。学習は同じデータを何十回も繰り返し使います(これを何十「エポック」と呼びます)。もしそのデータが「圧縮された動画」のような重い形式で、毎回それをデコード(展開)して画像にし、さらに固定された特徴抽出器に通しているなら、まったく同じ計算を何十回も繰り返していることになります。一度だけやって保存しておけば済むはずです。これが事前抽出キャッシュの発想です。

## 基礎: 前提となる概念

用語を順にかみ砕きます。

- エポック(epoch): 学習データ全体を1回ぶん通すこと。学習では同じデータを何十エポックも繰り返し通し、少しずつ重みを調整します。
- データローダ(DataLoader): 学習中に「次のバッチ(まとめて処理するデータの束)を用意してGPUに渡す」係。主にCPU側で動きます。GPUが計算している間に、CPUが次のバッチを準備しておくのが理想です。
- デコード(decode): 圧縮された動画ファイル(mp4 など)を、1枚ずつの画像(ピクセルの配列)に展開する処理。動画は時間方向に圧縮されているため(前後フレームの差分だけ記録するなど)、任意の時刻のフレームを取り出すにはCPUで重い展開計算が要ります。
- シーク(seek): 動画ファイルの中で、目的の時刻のフレームの位置まで飛ぶこと。圧縮形式では「キーフレームまで戻ってそこから順にデコードし直す」必要があり、ランダムな位置へのシークは高コストです。
- 特徴抽出(feature extraction): backbone(特徴抽出ネットワーク)で画像を特徴ベクトルに変換する処理。GPUで行うのが普通だが、これも計算量が大きい。
- frozen(凍結) backbone: 学習中に重みを更新しない特徴抽出器。凍結されていれば、同じ画像からは毎回まったく同じ特徴が出ます。だから特徴を保存して使い回せます。

学習ループの基本動作はこうです。GPUが「forward(予測) → 損失計算 → backward(重み更新)」をひたすら回します。GPUを止めないためには、CPU側のデータローダが「次のバッチ」を間に合わせて用意し続けなければなりません。ところが自動運転やマルチカメラのデータは非常に重い。例えば7台のカメラの動画を扱うと、毎ステップ次が起きます。

1. 動画ファイルから該当フレームの位置を探す(seek)
2. 動画をデコードして7枚の画像にする
3. リサイズ・正規化する(画素値を 0〜1 や標準正規分布に整える)
4. (素朴な構成なら)backbone に通して特徴にする

これがバッチごと、しかも毎エポック繰り返されます。問題は2つに整理できます。

問題1: デコードが遅くてGPUが待つ。動画デコードやシークはCPU負荷が高く、GPUの計算速度に追いつけません。するとGPUは「次のデータまだ?」と待つ時間が増え、高価なGPUが遊びます。学術的には「data stall(データ滞留)」や「input bottleneck」と呼ばれ、Mohan らの VLDB 2021 論文では、現実の DNN 学習で総時間の大きな割合がデータ待ちに費やされる実態が定量化されています。

問題2: 同じ計算を何度もやる。学習中、(データ拡張をしないなら)画像そのものは変わりません。なのに毎エポック同じ動画を同じようにデコードし、凍結 backbone なら同じ特徴を作り直す。30エポック学習するなら、同一のデコード・特徴抽出を30回やっていることになります。

そこで発想を変えます。「変わらない前処理は、学習の前に1回だけやって保存しておけばいい」。これが事前抽出(pre-extraction)とキャッシュ(cache)です。

## 仕組みを詳しく

中心となるのは、(1)オフラインで前処理を済ませてファイルに固める段階と、(2)学習時はそのファイルを順番に読むだけの段階、という二段構えです。保存形式には WebDataset の「シャード(shard)」がよく使われます。

ステップA: 事前に処理してシャードに固める(オフラインで1回)

動画をデコードして得た画像と、付随する数値データ(自動運転なら自車の運動履歴 egomotion など)を、`.tar` ファイル(これを shard と呼ぶ)に詰めます。1サンプルは例えば次のキーを持ちます。

```
"cam_0.jpg", "cam_1.jpg", ..., "cam_6.jpg"   ← 7カメラの画像(jpg にエンコードして保存)
"ego.npy"                                     ← 走行履歴 + 未来の数値配列
"meta.json", "__key__"                        ← メタ情報とサンプル識別子
```

これらを多数まとめて `shard-000.tar`, `shard-001.tar`, ... のように分割して保存します。分割するのは、1ファイルが巨大になりすぎず、かつ多数の小ファイルにもならない「中庸のサイズ」(数百MB〜数GB)に保つためです。クラウド学習では、このシャードをオブジェクトストレージ(S3 など)に置き、学習を回すマシンのローカルディスクに同期してから読みます。

ステップB: 学習時はシャードを順番に読むだけ

学習時のデータセットは、シャードを集めて各サンプルを学習用テンソルに変換します。やることは「jpg を開いて正規化」「ego.npy を履歴と未来に分割」だけで、動画のシークやデコードは一切ありません。

```python
loader = make_pre_extracted_loader("/data/shards", batch_size=8)
for batch in loader:
    # batch["visual_tiles"]       (B, 7, 3, 256, 256)   7カメラ画像
    # batch["egomotion_history"]  (B, 256)              履歴(例: 64ステップ×4信号)
    # batch["trajectory_target"]  (B, 128)              未来の正解軌道(例: 64×2)
    ...
```

なぜ速いのか、その鍵が「シーケンシャル読み(sequential read)」です。WebDataset の `.tar` はサンプルが先頭から順番に並んでいるので、ディスクを頭から順に読むだけで次々サンプルが取れます。動画の「あちこちに飛んで該当フレームを探す(ランダムアクセス)」より、順番に読むほうがディスクは圧倒的に速い(HDD なら数十倍、SSD でも数倍の差が出ます)。だから「ディスク帯域の限界に近い速さ」で供給でき、GPUが待たなくて済みます。

```
素朴な構成(毎エポック繰り返す):
  動画 seek → デコード → リサイズ正規化 → (backbone) → GPU
  ↑ ここがCPUボトルネック。同じ計算を毎エポック繰り返す

事前抽出キャッシュ:
  [事前1回] 動画デコード → shard(.tar) に固める → クラウド → ローカルへ同期
  [学習時]  .tar を順番読み → jpg 開いて正規化 → GPU
  ↑ デコードも seek も無い。順次読みでGPUを満たし続ける
```

シャード化の補助的な効果として、シャッフルバッファ(`shuffle`)で順番をランダム化しても順次読みの利点を大きくは損なわない、複数のCPUワーカーにシャードを分けて並列に読める(`split_by_worker`)、といった点もあります。完全なランダムサンプリングではなく「シャッフルバッファによる近似」になりますが、多くの学習では十分です。

キャッシュの「深さ」には段階があります。

- 画像段でキャッシュ: 動画をデコードした jpg を保存する。最も重いデコード・シークを省ける。backbone が学習可能(凍結しない)構成でも安全に使える。
- 特徴段でキャッシュ: backbone を通したあとの特徴ベクトルを保存する。さらに学習時の backbone forward すら省けるが、これは backbone が凍結されている前提でのみ正しい。backbone を学習すると重みが変わり、保存済みの特徴は古くなって(stale)使えなくなるからです。

どちらの段でキャッシュするかは「backbone を凍結するか学習するか」で決まります。凍結なら特徴段まで、学習可能なら画像段まで、が安全な選択です。

## 手法の系譜と主要論文

これは特定の自動運転モデルの手法ではなく、大規模学習のデータ供給(data loading / I/O)に関する工学の系譜です。

1. WebDataset(Aizman, Maltby & Breuel, "High Performance I/O for Large Scale Deep Learning", IEEE Big Data 2019): データを何百万個もの小ファイルとして置くのではなく `.tar` のシャードにまとめ、順次ストリーム読みすることで、クラウド/分散ストレージから高スループットで供給する方式を提案しました。動機は、小ファイルが膨大だとファイルシステムのメタデータ操作(open/seek/close)が律速になり、ストレージの帯域を全く活かせないこと。効果はストレージ帯域に近い供給速度。トレードオフは、シャード内はシーケンシャルなので完全なランダムサンプリングはできず、シャッフルバッファで近似する点です。

2. data stall の定量化(Mohan et al., "Analyzing and Mitigating Data Stalls in DNN Training", VLDB 2021): 現実の DNN 学習で、データ前処理・I/O がGPU計算に追いつかず、GPUがアイドルになる「data stall」が広範に起きていることを大規模に測定しました。前処理キャッシュ、ページキャッシュの活用、前処理の並列化といった緩和策を体系化し、「事前抽出キャッシュ」のような対策が理論的・実証的に正当化されました。

3. Data Echoing(Choi et al., "Faster Neural Network Training with Data Echoing", NeurIPS 2019, Google): 前処理が律速でGPUが余るとき、前処理済みデータを複数回使い回して(echo して)GPUを埋める手法を提案しました。動機は本ノートと同じ「前処理ボトルネック」ですが、解法は逆向きで、事前抽出キャッシュが「前処理を最初から1回に減らす」のに対し、data echoing は「同じ前処理結果を学習ループ内で再利用してGPUの空きを埋める」。両者は組み合わせ可能で、前処理ボトルネックへの相補的なアプローチです。

4. 凍結特徴のキャッシュ(linear probing の定番、Kornblith et al. ICML 2019 ほか自己教師あり学習の評価): 凍結した backbone の特徴を一度だけ抽出して保存し、その上の軽い分類層・回帰層だけを繰り返し学習する手法(linear probing)が広く使われます。動機は、凍結特徴は毎回同じなので再計算が純粋な無駄であること。効果は学習の劇的な高速化で、巨大な自己教師ありモデルの評価が現実的な時間で回せるようになりました。トレードオフは、特徴を保存するディスク容量と、データ拡張(毎回違う加工)が使えないことです。

## 論文の実験結果(定量データ)

- WebDataset(IEEE Big Data 2019)は、小ファイル方式と比べてシャード方式が、ネットワークストレージからの読み出しスループットを大幅に向上させることを示しました。ファイル数が数百万規模になると、小ファイル方式はメタデータ操作のオーバーヘッドで帯域の数分の一しか使えませんが、シャード方式はネットワーク/ディスク帯域をほぼ飽和させられます。指標は「秒あたりに読めるサンプル数(samples/sec)」で、これが GPU の処理能力を上回って初めて GPU を遊ばせずに済みます。

- Mohan らの VLDB 2021 論文は、複数の実モデル(画像分類など)で学習総時間のうちデータ滞留が占める割合を測定し、緩和前には総時間の相当割合(モデルとストレージ構成によっては大きな割合)がデータ待ちだったものを、キャッシュと前処理並列化で大きく削減できることを報告しました。アブレーションとして「キャッシュなし/ありで GPU 利用率がどう変わるか」を示し、キャッシュ導入で GPU 利用率が顕著に上がることを定量化しています。GPU 利用率(GPU が実際に計算している時間の割合)は、高価なハードウェアを無駄なく使えているかの直接の指標です。

- Data Echoing(NeurIPS 2019)は、前処理が律速の設定で、echo を入れることで「目標精度に到達するまでの実時間(wall-clock time)」を短縮できることを示しました。重要なのは、echo によってサンプルを使い回しても、必要な学習ステップ数(あるいは到達精度)がさほど悪化しないこと、つまり「同じデータを使い回しても学習がそれほど劣化しない」ことを実験で確認した点です。これは「データの新鮮さ」と「速度」のトレードオフを定量的に検討した点で価値があります。

これらを総合すると、定量データが一貫して示すのは「前処理・I/O の最適化は、モデルの精度を一切変えずに学習の実時間を短縮する」という、純粋に効率の利得だという点です。

## メリット・トレードオフ・限界

メリット
- 動画デコード・シークという最も重いCPU処理を学習ループから完全に外せる。GPUがデータ待ちで遊ばなくなり、GPU利用率が上がる。
- 同じ前処理を毎エポック繰り返す無駄が消える(30エポックなら最大30倍ぶんのデコードを1回に集約できる)。
- `.tar` の順次読みでディスク/ネットワーク帯域の限界速度に近い供給ができ、複数ワーカーでの並列化も容易。
- オブジェクトストレージにデータバージョンを置きやすく(データのバージョン管理ツールと相性がよい)、クラウド学習・分散学習に向く。

トレードオフ・限界
- 事前抽出のための前処理ジョブと、抽出済みデータを置くストレージ容量が別途必要。画像や特徴をそのまま保存するため、保存形式によっては元データより大きくなることもある(特に未圧縮の特徴ベクトルは巨大)。
- データ拡張を「毎回ランダムに」かけたい場合、キャッシュした時点の加工に固定されてしまう。多様な拡張を入れたい学習にはそのままでは不向き(拡張前の段でキャッシュし、学習時に軽い拡張だけ足す、などの工夫が要る)。これは効率と汎化性能のトレードオフで、強い拡張が精度に効く設定ではキャッシュの利点が削がれます。
- backbone を学習可能(凍結しない)にして特徴段までキャッシュすると、重みが変わるたびにキャッシュが無効(stale)になる。だから特徴段キャッシュは凍結前提でのみ正しく、学習可能な構成では画像段にとどめる規律が要る。
- シャード化したデータと、学習側が期待するテンソル形・キー名が食い違うと静かに壊れる(エラーにならず誤った学習が進む)。デコード規約(キー名、配列の分割方法)を抽出側と学習側で一致させる契約管理が要ります。
- 研究上の未解決課題として、「強いデータ拡張」と「前処理キャッシュ」を両立する方法があります。拡張をGPU上で軽く行う(GPU augmentation)、拡張のパラメータだけをキャッシュして決定論的に再現する、といったアプローチが研究されていますが、万能の解はまだありません。

## 発展トピック・研究の最前線
- GPU上でのデータ拡張(NVIDIA DALI など): デコードと拡張をGPUにオフロードし、CPUボトルネック自体を解消するアプローチ。キャッシュと組み合わせて使われます。
- ストリーミングデータセット(MosaicML Streaming, Hugging Face datasets のストリーミング): クラウドストレージから直接ストリーム読みし、ローカルへの全同期を不要にする設計。シャード方式の発展形です。
- 特徴キャッシュと基盤モデル: 巨大な凍結基盤モデル(画像エンコーダ、VLM)の特徴を一度抽出してキャッシュし、その上に軽いヘッドだけを学習する「凍結基盤モデル + 軽量アダプタ」の潮流。事前抽出キャッシュが学習効率の前提技術になっています。
- データバージョニングとの統合: LakeFS や DVC のようなデータバージョン管理ツールとシャードを組み合わせ、「どのバージョンのデータで学習したか」を再現可能にする MLOps の流れ。
