# BEVFormer v2

## ひとことで言うと
BEVFormer は、車の周りに付いた複数のカメラ画像から「上空から見下ろした地図(BEV)」を作り、その地図の上で物体検出をする手法です。BEVFormer v2 は、その改良版で、最大の工夫は「普通のカメラ視点(透視ビュー)でも物体検出を学習させる補助タスク」を足したことです。これにより、画像認識の世界で日進月歩に強くなっている最新の画像バックボーンをそのまま流用でき、検出精度が大きく上がりました。「補助タスク」とは、本来解きたい問題(BEV 検出)とは別に、学習を助けるために併走させる第2の問題のことです。

## 直感的な理解
画像認識の分野では、ImageNet などの巨大データセットで事前学習された強力なネットワーク(バックボーン)が次々と登場します。直感的には「強い画像認識器を持ってくれば、BEV 検出も自動的に強くなるはず」と思えます。ところが BEVFormer をそのまま使うと、最新の大型バックボーンに差し替えても期待ほど性能が伸びませんでした。

なぜか。BEVFormer はバックボーンの出力をすぐに BEV 空間へ変換し、BEV 側の損失(学習の正解との誤差)だけで全体を学習します。すると、バックボーンが透視ビューで持っていた強い能力が、BEV 変換という長いパイプラインを通すうちにうまく引き出されません。学習信号(勾配)がバックボーンまで届くころには薄まってしまうのです。「勾配とは、各パラメータをどちらにどれだけ動かせば誤差が減るかを示す数値で、層を遡るたびに小さくなりがちです(勾配消失)」。

身近なたとえで言えば、優秀な料理人(バックボーン)を雇っても、最終的な客の評価(BEV 損失)しか伝えないと、料理人は自分のどの工程を直せばいいか分かりません。途中で「この下ごしらえは良かった/悪かった」と直接フィードバックすれば、能力を発揮しやすくなります。BEVFormer v2 はこの「途中段階での直接フィードバック」を透視ビューに置いた手法です。

## 基礎: 前提となる概念

理解の土台として、いくつかの用語を噛み砕いておきます。

- BEV (Bird's-Eye-View、鳥瞰図): 車を真上から見下ろした 2 次元の地図表現です。各マスが地面上の一定範囲(たとえば 0.5m 四方)に対応し、距離や位置が座標として直接読めます。前方のカメラで「画面下に大きく写る車」と言うより「自車の 3m 前にいる車」と言えるほうが、進路計画にそのまま使えます。
- 透視ビュー (perspective view): カメラがそのまま撮った普通の画像、つまり遠近感のある正面からの見え方です。一般的な画像認識バックボーンは、ほぼすべてこの透視ビューの画像で事前学習されています。
- バックボーン (backbone): 画像を入力し、抽象的な特徴ベクトルの集まり(特徴マップ)を出力する土台ネットワークです。ResNet、Swin Transformer、ConvNeXt、ViT などが代表例で、ImageNet 等で事前学習したものを転用するのが定石です。
- 補助監督 (auxiliary supervision): メインの出力だけでなく、ネットワークの途中にも「正解」を与えて損失を計算する手法です。深いネットワークの途中に補助の損失を足して勾配を届きやすくする発想は古く、GoogLeNet(Szegedy ら, CVPR 2015)の補助分類器が代表例です。BEVFormer v2 はこの古典的な発想を「透視ビュー→BEV」という構造に合わせて適用したものといえます。
- DETR 系の物体クエリ (object query): 「ここに物体があるか?」を問い合わせる学習可能なベクトルです。デコーダがこのクエリを更新しながら、最終的に各クエリが 1 個の物体(ボックスとクラス)を予測します。

## 仕組みを詳しく

BEVFormer v2 の主要な変更点は 3 つあります。それぞれを段階的に見ていきます。

### 1. 透視ビューの補助監督 (perspective supervision)

これが最大の工夫です。BEV 側の検出ヘッドに加えて、バックボーンの出力(透視ビューの特徴マップ)に直接つながる「透視 3 次元物体検出ヘッド (perspective 3D detection head)」を足します。

- 透視ヘッドは、各カメラ画像の上で直接「ここに車がある、距離はこれくらい、向きはこれ」と 3 次元物体を検出するよう学習します。実装としては、画像座標系で動く 3D 検出器(たとえば FCOS3D 系のヘッド)が使われます。
- この補助タスクの損失がバックボーンに直接かかるので、勾配がバックボーンまでしっかり届きます。結果、最新の画像認識バックボーン(大規模事前学習された ViT 系や、後述する InternImage のような大型 CNN)の能力を引き出せるようになります。

噛み砕くと、「BEV の最終成績だけで採点していたのを、途中の透視ビュー段階でも採点するようにして、バックボーンがサボらず学習するようにした」ということです。学習が終われば透視ヘッドは推論時には外すことも、補助情報として残すこともできます。

### 2. 二段階デコーダ (two-stage decoder)

透視ヘッドが出した検出結果を「最初の当たり(プロポーザル、候補)」として使い、それを BEV のデコーダに渡して精緻化します。

- 第 1 段: 透視ヘッドが各カメラ画像から物体候補をざっくり出す。この候補は 3D 空間の位置を持つので、すべてのカメラの候補を BEV 座標系に集約できる。
- 第 2 段: その候補を BEV 検出デコーダの物体クエリの初期値として与え、空間クロスアテンション([空間クロスアテンション (SCA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0316_spatial-cross-attention))で BEV 空間の特徴を引きながら精密化する。

ゼロから BEV クエリを学ぶより、透視段階の良い当たりから始めるほうが収束が速く精度も上がります。具体的には、第 1 段の透視プロポーザルと、ランダム初期化された学習可能クエリの両方を結合して第 2 段に渡す「ハイブリッドな物体クエリ」を使います。これにより、幾何的に妥当な初期位置(透視由来)と、データから自由に学ぶ柔軟さ(学習クエリ由来)の両方を得ます。

### 3. 時間融合の拡張 (temporal encoder)

初代 BEVFormer は直前 1 フレームの BEV を参照する時間自己アテンション([時間自己アテンション (TSA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0343_temporal-self-attention))を持っていましたが、v2 ではより長い時間スパン(複数フレーム)の BEV 特徴を取り込めるよう時間融合を拡張しました。

- 過去複数フレームの BEV 特徴を、自車運動(ego-motion)で現在座標系に整列し、現在 BEV と結合してエンコードします。
- 長い履歴を見ることで、動きの推定(速度)や一時的な遮蔽(手前の車に隠れる)への対処が強化されます。論文では長時間の時間情報がとくに速度推定の指標(mAVE)に効くことが示されています。

形状感のある例で言うと、入力は 6 台のサラウンドカメラの画像列、バックボーンは最新の大型ネット、出力は 3 次元物体検出(車・歩行者などのボックス)です。透視ヘッドと BEV ヘッドの 2 系統の損失で同時に学習します。

```
              [カメラ6枚の画像列]
                     |
              [最新の大型バックボーン]
                     |
        +------------+-------------+
        |                          |
 [透視3D検出ヘッド]          [BEVエンコーダ(SCA)]
   (補助監督・第1段)              |
        |                  [時間エンコーダ(複数フレーム整列)]
   プロポーザル ----------> [BEVデコーダ(第2段, ハイブリッドクエリ)]
                                  |
                            [3D検出出力]
```

## 手法の系譜と主要論文

- 標準アテンションの起点: Vaswani ら "Attention Is All You Need" (NeurIPS 2017, arXiv:1706.03762)。クエリ・キー・バリューによる注意機構の元祖で、以下すべての前提です。
- DETR (Carion ら, ECCV 2020): 物体クエリとアテンションによる end-to-end 検出。後継の BEVFormer デコーダの祖先。
- Deformable DETR (Zhu ら, ICLR 2021, arXiv:2010.04159): 参照点周りの少数点だけにアテンションする効率化。BEVFormer の空間/時間アテンションの計算基盤。詳細は [Deformable Attention (BEV適用)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0330_deformable-attention-bev)。
- BEVFormer (Li ら, ECCV 2022, arXiv:2203.17270): BEV クエリが多視点画像へ空間クロスアテンションで集約し、直前 BEV へ時間自己アテンションで融合する枠組みを確立。v2 の直接の土台です。
- GoogLeNet (Szegedy ら, CVPR 2015): 補助分類器による中間監督。v2 の透視監督の思想的ルーツ。
- InternImage (Wang ら, CVPR 2023, arXiv:2211.05778): deformable convolution v3 をベースにした大型 CNN バックボーン。v2 が「活かしたい最新バックボーン」の代表例として採用。
- BEVFormer v2 (Yang, Chen, Tian ら, CVPR 2023, arXiv:2211.10439): 上記の透視監督・二段階デコーダ・拡張時間融合を導入した本トピック。「Adapting Modern Image Backbones to Bird's-Eye-View Recognition via Perspective Supervision」というタイトルが示すとおり、目的は「最新バックボーンを BEV 認識に適応させること」そのものです。

系譜としては「DETR が end-to-end 検出を確立 → Deformable DETR が効率化 → BEVFormer がそれを BEV に持ち込み時間も統合 → v2 が監督の与え方を変えて最新バックボーンを活かせるようにした」という流れです。

## 論文の実験結果(定量データ)

評価は主に nuScenes(自動運転の標準ベンチマーク。6 カメラ・LiDAR・レーダ付きの大規模都市走行データ)の 3D 物体検出で行われます。主要指標を初心者向けに説明します。

- mAP (mean Average Precision): 検出の正確さの総合指標。値が高いほど良い。0〜1 で、0.5 なら「半分くらい当てている」感覚。
- NDS (nuScenes Detection Score): nuScenes 独自の総合スコア。mAP に加えて、位置誤差・サイズ誤差・向き誤差・速度誤差・属性誤差を統合したもの。検出だけでなく「ちゃんと正しい位置・速度・向きで当てているか」まで含む。高いほど良い。
- mAVE (mean Average Velocity Error): 推定した物体速度の誤差(m/s)。低いほど良い。時間融合の効果がここに強く出ます。

報告されている主な結果(およその値):

- 初代 BEVFormer は ResNet-101 系バックボーンで nuScenes test セット NDS 約 0.569 を達成していました。
- BEVFormer v2 は、大型バックボーン(InternImage-XL など)と透視監督を組み合わせ、nuScenes test セットで NDS 約 0.634、mAP 約 0.556 という当時の最高水準を報告しました。これは初代に対し NDS で 6 ポイント以上の改善で、3D 検出ベンチマークでは非常に大きな差です。NDS は 1 ポイントの上積みでも難しい指標であり、5〜6 ポイントの差は世代交代に相当する大きさだと考えてよいです。
- 検証セット(val)での比較でも、同じバックボーンを使ったとき透視監督ありはなしより明確に高い NDS を示しています。論文では、何の工夫もせず ImageNet 事前学習済みの大型バックボーンを初代 BEVFormer にそのまま差し込むと、ResNet-50 のような小型バックボーンとの差がほとんど出ない、つまり「大きくしても得をしない」現象が報告されています。透視監督を足してはじめて、バックボーンを大きくした分だけ NDS が単調に伸びるようになります。この「監督を変えるとスケーリング則が復活する」という観察こそが論文の核心的な発見です。

アブレーション(どの要素を抜くとどうなるか)で報告されている要点:

- 透視監督を外すと: 大型バックボーンを使っても精度が伸び悩み、NDS が数ポイント低下します。とくに ImageNet 事前学習しただけの大型バックボーンを BEV 損失だけで学習すると、十分に適応できないことが示されています。これが「透視監督こそが鍵」という論文の主張の根拠です。
- 二段階デコーダ(透視プロポーザル)を外すと: 収束が遅くなり、最終精度もやや下がります。プロポーザルの「良い初期当たり」がデコーダの学習を助けていることが分かります。
- 時間融合のフレーム数を減らすと: mAP はあまり変わらないが mAVE(速度誤差)が悪化します。これは速度推定が時間情報に強く依存することと整合します。

数値の意味の補足: NDS が 0.57 から 0.63 に上がるというのは、誤検出が減り、位置・速度・向きの推定が全体的に正確になったことを意味します。自動運転では「相手の速度を 1〜2m/s 単位で正しく見積もれるか」が衝突回避の判断に直結するため、mAVE の改善は実用上きわめて重要です。

## メリット・トレードオフ・限界

メリット:
- 透視監督でバックボーンに勾配が直接届き、最新の強力な画像バックボーンを有効活用できる。これが最大の価値です。
- 二段階デコーダ(透視候補→BEV 精緻化)で収束が速く精度も向上。
- 拡張時間融合で速度推定(mAVE)と遮蔽への対処が強化される。
- 初代 BEVFormer の枠組みを保ちつつ、純粋に性能を底上げできる。

トレードオフ・限界:
- 補助ヘッド+二段階構造で学習パイプラインが複雑化し、計算コストとメモリが増える。大型バックボーンは学習時間も長い。
- 透視 3 次元検出のための各画像 3 次元アノテーションが必要(あるいは BEV の 3D ボックスを各カメラに投影して教師を作る前処理が必要)。ラベル作成・前処理コストがかかる。
- 初代より実装・ハイパーパラメータ調整の手間が大きい(透視損失と BEV 損失の重みバランス、プロポーザル数など)。
- カメラパラメータ(校正)と時間融合(ego-motion)への依存は初代と同様に残る。校正がずれれば投影がずれ、透視・BEV 双方の精度が落ちる。

研究上の未解決課題:
- 透視監督は LiDAR の蒸留(LiDAR の正確な深度を教師にする)とどう組み合わせるのが最適か、まだ探究の余地がある。
- 大型バックボーンの計算コストを保ったまま、車載のリアルタイム制約(数十 ms)にどう収めるかは依然として難しい。
- カメラのみ vs カメラ+LiDAR で透視監督の効き方がどう変わるかの体系的理解は発展途上。

## 発展トピック・研究の最前線

BEVFormer v2 が示した「中間表現に直接監督を置いてバックボーンを活かす」考え方は、その後のマルチビュー 3D 認識に広く影響しました。関連する流れとして、深度を陽に教師するアプローチ(BEVDepth: Li ら, AAAI 2023 など。LiDAR 由来の深度で BEV 持ち上げを監督する)、長期時間融合の効率化(SOLOFusion: Park ら, ICLR 2023)、そして検出にとどまらず追跡・地図・予測・計画まで一気通貫で学ぶ統合運転モデル(UniAD: Hu ら, CVPR 2023, ベスト論文)があります。UniAD は BEVFormer を BEV エンコーダとして採用しており、v2 流の「強いバックボーン+補助監督」の思想は、こうした end-to-end 自動運転の土台として今も生きています。さらに近年は、視覚言語モデルや生成モデルを用いた占有予測・世界モデルとの接続が研究の最前線で、中間表現への監督設計はその中でも繰り返し問われるテーマです。
