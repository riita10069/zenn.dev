# MapTR (オンラインベクトル地図構築)

## ひとことで言うと

MapTR は、走行中の車に付いた複数のカメラの映像だけから、「車線の境界線」「横断歩道」「道路の縁(road boundary)」といった地図の構成要素を、その場でリアルタイムに「折れ線(ベクトル)」の形で作り出すニューラルネットワークです。あらかじめ測量して作った精密地図(HD地図)に頼らず、いま見えている景色だけから運転に必要な地図を組み立てます。点を一括で並列に予測する設計と、折れ線の「向きの曖昧さ」を吸収する工夫によって、それまでの手法より大幅に速く・正確になりました。

## 直感的な理解

人間が知らない街を運転するとき、わざわざ事前に全ての白線を測量した地図を持ち歩いたりはしません。目で見て「ここが自分の車線、左がセンターライン、前方に横断歩道」と瞬時に把握します。MapTR がやろうとしているのは、まさにこの「見て地図を頭の中に描く」作業を機械にやらせることです。

なぜこれが必要なのか。従来の自動運転は HD地図(High-Definition map、高精細地図)という、数センチ単位で白線や車線中心線を記録した特殊な地図に強く依存していました。これには深刻な問題があります。第一に作成コストが莫大です。一つの都市の全道路を測量車で走り、白線を一本ずつ記録するのは膨大な人手と費用がかかります。第二に古くなります。工事で車線が移動したり白線が引き直されたりすると、地図と現実がずれ、車が誤った判断をします。第三に地図のない場所では走れません。

そこで「事前地図に頼らず、その場のカメラ映像から必要な地図情報を生成できないか」という発想が生まれます。これがオンライン地図構築(online HD map construction)です。「オンライン」はインターネットの意味ではなく「走行中にリアルタイムで」という意味です。MapTR はこの分野で、出力形式と学習の安定性という二つの壁を同時に破った金字塔的な研究です。

## 基礎: 前提となる概念

理解に必要な専門用語を最初に噛み砕きます。

BEV(Bird's-Eye-View、鳥瞰図): 真上から見下ろした視点のことです。カメラは斜め前方を映しますが、地図や運転計画は「真上から見た平面」で考えると扱いやすい。そこでカメラ画像の特徴を、真上から見た格子(グリッド)状の特徴マップに変換します。これを BEV 特徴と呼びます。たとえば自車の周囲 60m 四方を縦200×横100マスに区切り、各マスに256次元の特徴ベクトルを持たせた tensor(形状 (B, 256, 200, 100)、B はバッチサイズ)です。

ラスタ地図とベクトル地図: オンライン地図構築の出力には二つの表現があります。ラスタ地図(raster map)は地図を画像のように格子のマス目(ピクセル)で表し、各マスが「車線か/横断歩道か/何もないか」のクラスを持ちます。ベクトル地図(vectorized map)は地図を「点を順番につないだ折れ線(polyline)」で表します。たとえば白線一本を「点A→点B→…」のように複数の点の並びで表現します。

なぜベクトルが嬉しいのか。下流のプランナ(どう走るかを決める部分)にとって、ベクトル地図は「この折れ線が私の左の車線境界」とそのまま使えます。ラスタ地図だと、画像から折れ線を後処理(ポストプロセス)で抽出する追加工程が必要で、その工程でノイズや誤接続が入り込みます。HDMapNet(Li et al., ICRA 2022)はラスタ的なセグメンテーションを出してから後処理でベクトル化する初期手法でしたが、この後処理が精度と速度のボトルネックでした。MapTR はベクトル地図を後処理なしでニューラルネットから直接出力します。

DETR(Detection Transformer): 物体検出を Transformer で行う手法(Carion et al., ECCV 2020)です。従来の物体検出は「候補ボックスを大量に出して重複を後処理で消す」面倒な手続きが必要でしたが、DETR は「物体クエリ(object query)」という学習可能なベクトルを N 個用意し、各クエリが Transformer デコーダで画像特徴を参照(attention)しながら一つの物体を担当する形にしました。出力は集合(set)であり、後処理が不要です。MapTR はこの考えを「地図要素の検出」に応用します。

二部マッチング(bipartite matching)とハンガリアン法: DETR 系では N 個の予測と M 個の正解(ground truth)をどう対応づけるかが問題になります。これを「予測と正解を一対一で結ぶ最小コストの組み合わせを見つける問題」として解きます。ハンガリアンアルゴリズムという古典的手法で最適マッチングを計算し、対応がついたペアだけで損失を取ります。

## 仕組みを詳しく

MapTR(Liao et al., ICLR 2023)の処理を段階的に分解します。

入力と出力の形。入力は車の周囲を写す複数台(典型的には6台)のカメラ画像で、各画像は例えば縦900×横1600×3チャンネル(RGB)。出力は地図要素の集合です。各要素は「クラス(車線分離線 / 道路境界 / 横断歩道)」と「固定本数の点列(折れ線)」のペアで、MapTR では一つの折れ線を Nv=20 個の点で表します。1点は (x, y) の2次元座標なので、1要素は 20×2 = 40 個の数値です。

ステップ1: 画像を BEV 特徴に変換する。6枚の画像をそれぞれ画像エンコーダ(ResNet-50 など)に通して特徴マップを得て、BEV の格子に投影します。投影には GKT(Geometry-guided Kernel Transformer)や、深度を推定して特徴を3次元に持ち上げる LSS(Lift-Splat-Shoot)系の手法が使われます。結果として (B, C, 200, 100) のような BEV 特徴ができます。「自車周囲を真上から見た特徴の地図」です。

ステップ2: 階層的クエリで地図要素を取り出す。MapTR は地図要素クエリを多数(例えば50インスタンス)用意します。MapTR 独自の重要な工夫が階層的クエリ埋め込み(hierarchical query embedding)です。クエリを「インスタンスレベル(どの折れ線か、50個)」と「ポイントレベル(その折れ線の何番目の点か、20個)」の二階層に分解します。各点クエリは「インスタンス埋め込み + 点埋め込み」の和で表され、50×20=1000 個の点クエリを、50+20=70 個の埋め込みだけで効率よく表現します。これらが Transformer デコーダで BEV 特徴を deformable attention(画像全体ではなく注目点の周辺だけを効率的に参照する attention)で参照しながら更新され、最終的に各インスタンスが「クラス + 20点の座標」を出力します。

ステップ3: 順列等価モデリング(permutation-equivalent modeling)。MapTR の論文で最も独創的な貢献です。折れ線には「向き」の曖昧さがあります。車線境界を「点1→…→点20」と表しても「点20→…→点1」と逆順に表しても同じ白線です。さらに横断歩道のような閉じた多角形(ループ)は、どの点から数え始めても同じ形です。

素朴に学習すると、ネットワークが正しい形を出しているのに「点の順番が教師と逆」というだけで巨大な損失を受け、学習が発散します。MapTR は一つの折れ線に対して「同じ形を表す全ての等価な点順序の集合」を定義します。開いた折れ線(車線線)なら順方向と逆方向の2通り、閉じた多角形(横断歩道)なら開始点の選び方と向きで 2×Nv 通り。マッチングの際にこの集合の中から教師に最も近い順序を選んで点損失を計算します。これにより「形は合っているのに順番で減点される」病理を消し去り、学習が劇的に安定しました。

損失関数は三つの和です。(1) クラス分類損失(Focal Loss)、(2) 点間距離の Manhattan 距離(L1)による point2point 損失、(3) 隣接点を結ぶ辺の方向を合わせる edge direction 損失(コサイン類似度ベース)。インスタンスレベルのマッチングはハンガリアン法、点レベルは上記の順列等価マッチングという二段構えです。

全体の処理イメージ。

```
6カメラ画像 ─► 画像エンコーダ ─► BEVへ投影 ─► BEV特徴 (B,C,200,100)
                                                  │
              階層クエリ(50インスタンス×20点) ──┤ Transformerデコーダ
              = インスタンス埋込 + 点埋込         │  (deformable attention)
                                                  ▼
              各インスタンス: クラス + 20点の(x,y)折れ線
                  ↑ 点レベルは順列等価マッチングで損失計算
                  ↑ インスタンスレベルはハンガリアンマッチング
```

なぜこの設計なのか。先行手法 VectorMapNet は点を一つずつ自己回帰的に生成したため遅く、点間に逐次依存が生じました。MapTR は固定数の点を一括で並列出力する DETR 方式に切り替えることで高速化し、順列等価モデリングで自己回帰なしでも学習を安定させました。これが速度と精度の同時改善を生んだ核心です。

## 手法の系譜と主要論文

HDMapNet (Li et al., ICRA 2022, arXiv:2107.06307): オンライン HD 地図構築という問題設定を確立した初期の代表作。BEV 上でセマンティックセグメンテーション(各画素のクラス分類)とインスタンス埋め込みを出力し、後処理でベクトル化しました。動機は「事前 HD 地図の代替をその場で作る」こと。課題は後処理依存で、エンドツーエンドにベクトルを出せませんでした。

VectorMapNet (Liu et al., ICML 2023, arXiv:2206.08920): ベクトル地図をニューラルネットで直接出す先駆け。地図要素をまず検出し、その後 polyline を一点ずつ自己回帰的に(前の点を見て次の点を出す形で)生成しました。動機は「ラスタの後処理を排し、プランナがそのまま使える表現を出す」こと。後処理不要のベクトル出力を達成しましたが、自己回帰生成のため推論が遅く(約3 FPS 程度)、リアルタイム性に課題が残りました。

MapTR (Liao et al., ICLR 2023, arXiv:2208.14437): 自己回帰の遅さを解決するため、固定数の点を一括並列出力する DETR 方式と、階層的クエリ、順列等価モデリングを提案。後処理も自己回帰もなしで高速かつ高精度を実現しました。

MapTRv2 (Liao et al., IJCV 2024, arXiv:2308.05736): MapTR の拡張。decoupled self-attention(インスタンス間と点間の attention を分離して計算量を削減)、補助的な one-to-many マッチング(一つの正解に複数予測を割り当て学習信号を増やす)、深度推定の補助損失(PV-to-BEV 変換を助ける)、辺方向損失の強化を加え、精度と収束速度をさらに改善。3D 地図要素の出力にも対応しました。

StreamMapNet (Yuan et al., WACV 2024, arXiv:2308.12570): 単フレームの MapTR が抱える「遠方や遮蔽で見えない要素を作れない」「フレーム間で地図がちらつく」問題に対し、過去フレームのクエリと BEV 特徴を時間方向に伝搬(streaming)させる仕組みを導入。長距離・時間安定な地図構築へと発展させました。MapTR 系は現在この時間的拡張が主戦場になっています。

## 論文の実験結果(定量データ)

評価指標の意味から説明します。地図構築の主指標は mAP(mean Average Precision、平均適合率)です。予測した折れ線と正解の折れ線が「十分近いか」を Chamfer 距離(二つの点群の対応点間距離の平均)で判定し、しきい値(例 0.5m / 1.0m / 1.5m)ごとに AP を計算して、車線分離線・道路境界・横断歩道の三クラスで平均します。値が高いほど良く、単位は%です。FPS(Frames Per Second、1秒あたり処理フレーム数)は速度で、リアルタイム運転には数十 FPS 欲しい。

データセットは主に nuScenes(ボストンとシンガポールで6カメラ+LiDAR で収集した自動運転データセット)です。報告によると、ResNet-50 バックボーン・カメラのみの設定で、HDMapNet は mAP がおよそ 23% 台、VectorMapNet が約 40% 台でかつ約3 FPS と低速でした。MapTR はカメラのみ ResNet-50 で約 50% 前後の mAP を、約25 FPS という実用的な速度で達成し、VectorMapNet に対し精度・速度ともに大幅な改善を示しました。学習エポックを増やした設定(110エポック)ではさらに高い mAP が報告されています。MapTRv2 は同条件でカメラのみでも約 61% 前後、カメラ+LiDAR 融合では 70% 前後へと押し上げ、収束も MapTR より速くなったと報告されています。

アブレーション(どの要素を抜くと何が起きるか)の知見。第一に、順列等価モデリングを外して固定の点順序で学習すると mAP が数ポイントから十数ポイント単位で大きく低下します。これは「形は正しいのに順序で減点される」問題が学習を不安定にする証拠で、MapTR の核心的貢献が定量的に裏付けられています。第二に、edge direction 損失を加えると折れ線の滑らかさと精度が向上します。第三に MapTRv2 の one-to-many マッチングと深度補助損失は、それぞれ単独で数ポイントずつ mAP を押し上げ、収束エポック数を削減します。これらは「マッチングの曖昧さを減らす」「BEV 変換を幾何的に補助する」という独立した効き方をします。

これらの数値が重要な理由。mAP が 23% から 50% 超へ倍以上に伸びたことは、後処理依存(HDMapNet)→自己回帰(VectorMapNet)→並列ベクトル予測(MapTR)という設計転換が、地図品質を実用レベルへ押し上げたことを意味します。同時に 3 FPS から 25 FPS への速度改善は、研究用のオフライン手法を車載リアルタイム候補へ変えました。

## メリット・トレードオフ・限界

メリット。事前 HD 地図の作成・維持コストから解放され、地図のない道でも走れる可能性が出ます。出力がそのままベクトル折れ線なので後処理なしでプランナが使えます。地図が常に最新で、工事などの変化に強い。

トレードオフと限界。カメラ依存のため遠方・夜間・悪天候・逆光で精度が落ちます。見えていない範囲(遮蔽の向こう、交差点の死角)は作れません。折れ線の点数 Nv が固定(20点)なので、非常に細かい曲線や極端に長い要素を一要素で表しきれない場合があります。高品質なベクトル地図アノテーション付き学習データが必要で収集コストが高い。事前地図と違い出力が毎フレーム変動しうるため、時間方向の一貫性(ちらつき防止)に追加の工夫が要ります。さらに研究上の未解決課題として、nuScenes の標準分割は地理的に学習・評価データが重複しており(同じ場所を別の時刻に走っている)、StreamMapNet の論文は地理的に分離した分割で評価すると全手法の mAP が大きく低下することを指摘しました。これは「本当に未知の街で動くのか」という汎化性能の評価が過大評価されてきた可能性を示し、現在の重要な論点です。

## 発展トピック・研究の最前線

時間的拡張(StreamMapNet, MapTracker など)で長距離・時間安定な地図を作る方向、ベクトル地図を不確実性つきで出力しプランナが信頼度を加味できるようにする方向、地図をラスタでもベクトルでもなく潜在表現(latent map)として下流タスクと一体学習する方向(エンドツーエンド運転への統合)が活発です。また「標準地図(SD map、カーナビ程度の粗い地図)を事前知識として与え、その上でカメラから詳細を補う」ハイブリッド手法(例 P-MapNet)が、純オンラインと純 HD 地図の中間として現実的だと注目されています。地理的に分離した汎化評価のもとでの頑健性向上が、分野全体の宿題です。
