# 入力解像度256×256への変更

## ひとことで言うと

ニューラルネットに入れる画像のサイズ(入力解像度)は、好きな値にできるわけではありません。特に Swin Transformer のように画像を「ウィンドウ」という固定サイズのマス目で区切るモデルでは、画像サイズがウィンドウサイズや縮小率で割り切れる必要があります。割り切れないと、端に半端なマスが余って計算が破綻したり、無理にパディング(余白の水増し)してごまかすことになり性能が落ちたりします。多くの実装が 224 ではなく 256 を選ぶのは、256 が 2 の8乗で、4・8・16・32 という典型的な縮小率やウィンドウサイズで素直に割り切れるからです。本トピックは「なぜ解像度の選択がそれほど重要か」「解像度を変えると何が連動して変わるか」を、初心者から研究者レベルまで掘り下げます。

## 直感的な理解

折り紙で考えます。1枚の正方形の紙を半分に折り続けると、辺の長さが偶数なら何度でもきれいに半分にできますが、奇数だと途中で半端が出ます。Swin 系のバックボーンは画像を段階的に半分(あるいは 1/4)に縮めながら処理するので、辺の長さがずっと割り切れ続ける必要があります。256 は 256 → 64 → 32 → 16 → 8 とすべて整数で割れますが、224 は 224 → 56 → 28 → 14 → 7 と最後が 7(奇数)になり、ウィンドウサイズによってはここで割り切れず破綻します。

もう一つの直感として、解像度は「モデルがどれだけ細かく世界を見られるか」を決めます。解像度を上げれば遠くの小さな歩行者や標識まで見えますが、その分メモリと計算が増えます。だから解像度は「見える細かさ」と「コスト」のトレードオフであり、適当に決められない重要な設計値なのです。

## 基礎: 前提となる概念

用語を噛み砕きます。

- 入力解像度 (input resolution): モデルに渡す画像の縦横のピクセル数です。256×256 なら縦256画素・横256画素です。
- ウィンドウ (window): Swin が注意機構(画像のどの部分に注目するかを計算する仕組み)を計算するときに、画像を区切る小さな正方形の単位です。ウィンドウサイズ8なら、特徴マップを 8×8 のマス目に区切り、その中だけで注意を計算します。
- stride(ストライド、縮小率): バックボーンが各段階で空間をどれだけ縮めるか。stride 2 なら縦横が 1/2 になります。
- パディング (padding): テンソルのサイズを合わせるために端に詰める余白(多くはゼロ)。割り切れないサイズを無理に通すときに使われますが、本来無い情報を足すので歪みの原因になります。

なぜ画像全体ではなくウィンドウ単位で注意を計算するのか。画像全体で注意を計算すると、計算量が画素数の2乗に比例して爆発します。たとえば 224×224 = 50176 画素なら、その2乗で約25億の組み合わせを計算することになります。ウィンドウ内に限定すれば計算量を画素数に比例する程度まで抑えられます。これが Swin 系の核心的な工夫です(詳しくは [SwinV2特徴マップの次元並べ替え](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1015_swin-v2-feature-map-permutation) 参照)。

ここで割り切れ問題が起きます。Swin は画像を段階的に縮小しながら各段階でウィンドウに区切ります。特徴マップの縦横がウィンドウサイズで割り切れないと、端に半端なマスが余り、ウィンドウ分割が成り立ちません。224 という数字はもともと Swin V1 のウィンドウ7用の設定 `swin_tiny_patch4_window7_224` 向けに選ばれたもので、別のウィンドウサイズ(例えば8や16)とは整合しないことがあります。

そこで多くの実装は、4・8・16・32 のどれでも割り切れる 256 を採用します。256 は 2 の8乗なので、典型的なウィンドウ・stride 設定すべてで最後まで安全に割り切れます。

## 仕組みを詳しく

### 解像度を変えると特徴マップの形も変わる

バックボーンは画像を段階的に縮小します。典型的な4ステージのバックボーンでは、各ステージで縦横が連動して変わります。256×256 入力での典型的な特徴マップ形状を、SwinV2 Tiny のチャンネル数で示します。

```
入力           : [B, 3, 256, 256]   (3 は RGB の色チャンネル)
ステージ1 (/4) : [B,  96,  64,  64]
ステージ2 (/8) : [B, 192,  32,  32]
ステージ3 (/16): [B, 384,  16,  16]
ステージ4 (/32): [B, 768,   8,   8]
```

`/4` は「縦横が 1/4 になる」の意味です。256÷4=64、÷8=32、÷16=16、÷32=8 とすべて整数で割り切れています。もし 224 なら、224÷32=7 で一応割れますが、最後のステージの特徴マップが 7×7 になり、ウィンドウサイズが8なら 7 は 8 で割れず破綻します。256 なら最終段が 8×8 になり、ウィンドウ8で 1×1 のウィンドウ1個にきれいに収まります。

### 下流モジュールの更新が必要になる

入力解像度を変えると、上の表のように後段モジュールが受け取るテンソルの縦横(H, W)が変わります。空間サイズを前提にしている部品はすべて影響を受けます。代表例:

- 複数視点を統合する融合モジュール(空間サイズに依存した平坦化や結合をするもの)
- 鳥瞰図(BEV)グリッドへの射影(画像の各位置を地面座標に対応づける処理は、入力の H×W に依存します)
- 学習可能な位置エンコーディング(位置ごとのベクトルを持つので、位置の数が変わると次元が合わなくなります)

before/after を簡単に書くと:

```
before: 入力 224x224、Swin V1 window7 前提、下流が H,W=7 等を仮定
after : 入力 256x256、SwinV2 等でも割り切れる、下流が H,W=8 等に更新
```

つまり「入力解像度を変える」という一見小さな変更が、パイプライン全体の形状の整合を取り直す作業に波及します。これが解像度変更を慎重に行うべき理由です。

### 解像度とメモリ・計算の関係(数値で)

注意機構の計算量は画素数の2乗(ウィンドウ化すれば画素数に比例)、畳み込みやメモリは概ね画素数に比例します。256² = 65536 は 224² = 50176 の約1.31倍です。したがって 224 から 256 へ上げると、おおまかにメモリと計算が約3割増えます。さらに 256 から 384 へ上げると 384²/256² = 2.25 倍、512 なら 4 倍になります。解像度は性能に効きますが、コストは面積(辺長の2乗)で効くので、上げすぎるとすぐにメモリ上限(VRAM)にぶつかります。

## 手法の系譜と主要論文

- ViT (Dosovitskiy et al., ICLR 2021, arXiv:2010.11929): 画像を固定サイズのパッチ(16×16画素など)に分割して系列として扱う発想を導入しました。これにより「入力解像度はパッチサイズで割り切れる必要がある」という制約が一般化しました。本トピックのウィンドウ整合性も、この「固定単位で区切る」系統の制約の一種です。ViT は位置埋め込みも固定長で持つため、解像度を変えると位置埋め込みの補間が必要になるという別の課題も生みました。

- Swin Transformer (Liu et al., ICCV 2021, arXiv:2103.14030): ウィンドウ注意で計算量を抑え階層特徴を得る手法。ウィンドウサイズという新たな割り切れ制約を持ち込みました。

- Swin Transformer V2 (Liu et al., CVPR 2022, arXiv:2111.09883): 解像度を変えても性能が保たれるよう、log間隔の連続位置バイアス(log-spaced continuous position bias)を提案しました。学習時に低い解像度で得た位置の知識を、推論・微調整時の高解像度へ滑らかに転移できます。低解像度(192)で事前学習し高解像度(384〜512)へ微調整しても劣化が小さいことを示しました。これは「まず 256 で学習し、必要なら後で 448 などへ上げる」といった段階的な高解像度化を現実的にします。トレードオフとして位置バイアスを毎回計算する小さな追加コストがあります。

- EfficientNet (Tan & Le, ICML 2019, arXiv:1905.11946): 解像度・深さ・幅をバランスよく同時に上げる compound scaling を提案しました。解像度は単独で上げるより、深さ・幅と一緒に上げた方が効率が良いという知見です。「256 に統一しつつ、上げるなら他の要素とのバランスを見る」という判断の根拠になります。

- FlexiViT (Beyer et al., CVPR 2023, arXiv:2212.08013): パッチサイズ(=実質的に解像度の細かさ)を学習中にランダムに変え、1つのモデルで複数の解像度・パッチサイズに対応させる手法。固定解像度の制約を緩める研究方向の一例で、将来「解像度を後から自由に変えられる」モデルへの布石です。

## 論文の実験結果(定量データ)

解像度が性能に効くことを示す代表的な数値をいくつか挙げます。

- EfficientNet の compound scaling では、解像度だけを上げる(B0 の 224 から段階的に拡大)ことで ImageNet Top-1 が一貫して向上し、最終的に B7(600解像度)で約84.3%に達しました。同じパラメータ予算なら、解像度・深さ・幅を均等に上げた方が、解像度だけ上げるより精度効率が良いと報告されています。

- SwinV2 の解像度転移実験では、低解像度事前学習から高解像度微調整への適応で、従来の双三次補間による位置バイアス調整に比べ、log間隔連続位置バイアスが精度劣化を有意に抑えました。具体的には大解像度(例: 学習192→評価384)でのギャップを縮め、巨大モデル SwinV2-G が複数ベンチマークで当時の最高精度を更新しました。

- 物体検出・セグメンテーションでは解像度の影響がさらに大きく、COCO 検出や ADE20K セグメンテーションでは入力を大きくするほど小さな物体の検出 AP(平均適合率)が改善する一方、計算量が二乗で増える、というトレードオフが一貫して観測されます。

指標の意味を補足します。Top-1 精度は「最も確信したクラスが正解と一致した割合」で、1ポイントの差でも大規模ベンチマークでは統計的に意味のある改善とされます。AP(Average Precision)は検出の正確さと網羅性を両立できているかを 0〜1 で測る指標で、解像度を上げると小物体の AP が伸びやすいのは「小さな対象が画素上で大きく写るようになる」ためです。

## メリット・トレードオフ・限界

メリット

- 256 は 2 のべき乗なので、多くのバックボーン(Swin / ConvNeXt / ResNet)とウィンドウ・stride 設定で安全に割り切れます。
- 縦横が常に整数で縮小されるため、半端なマスによる注意計算の破綻やパディングによるごまかしが起きません。
- SwinV2 の位置バイアスと組み合わせれば、将来の高解像度化(448 など)への移行がやりやすくなります。

トレードオフ・限界

- 224 で事前学習された重みを 256 で使うと、位置に関する表現が完全には一致せず、微調整が望ましいです(SwinV2 はこれを緩和する設計)。位置埋め込みの補間が必要なモデルでは、補間による細かな精度劣化が起こり得ます。
- 解像度を上げるとメモリと計算が面積比で増えます(256² は 224² の約1.3倍、384 なら 2.25 倍)。バッチサイズや VRAM とのトレードオフになります。
- 公開ベンチマークの多くは特定解像度で報告されるため、厳密に比較したいときはその解像度に合わせる必要があり、256 統一が比較に最適とは限りません。
- 研究上の課題として、「動的解像度」(入力ごとに最適な解像度を選ぶ、近距離は低解像度・遠距離は高解像度のように領域別に解像度を変える)は計算効率と精度の両立を狙う活発な研究領域ですが、固定解像度前提のパイプラインには素直に組み込みにくいという未解決の難しさがあります。

## 発展トピック・研究の最前線

- 多解像度・動的解像度: NaViT(Dehghani et al., 2023)はアスペクト比と解像度を保ったまま可変長で画像を処理し、無駄なパディングを排除しました。FlexiViT と合わせ、「固定解像度の制約からの解放」が一つの潮流です。
- トークン削減: 高解像度のコストを下げるため、重要でないトークン(空・路面など情報の薄い領域)を間引く Token Merging / Token Pruning が研究されています。実質的に「必要な所だけ高解像度」を実現するアプローチです。
- 解像度とスケール則: 大規模事前学習では、解像度もモデルサイズやデータ量と並ぶスケーリング軸として扱われます。どの解像度でどれだけ学習するのが計算効率最良か、というスケール則の研究が続いています。
