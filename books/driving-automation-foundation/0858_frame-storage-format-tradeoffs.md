# フレーム保存フォーマットの選択

## ひとことで言うと
動画から取り出したフレーム(画像)を、学習で速く読めるようにディスクへ保存しておくとき、どの形式で保存するか——JPEG、PNG、tar アーカイブ、HDF5、LMDB、メモリマップしたテンソル——で、容量・読み込み速度・画質が大きく変わります。多くの場合、容量は形式によって一桁以上違います。本ノートはその選択肢と、容量がどれくらいになるかの見積もり方、そして「どの場面でどれを選ぶか」を整理します。

## 直感的な理解
引っ越しを想像してください。本を運ぶとき、一冊ずつ手で運ぶ(個別ファイル)と、何百回も玄関とトラックを往復することになり、運ぶ時間より往復の時間の方が長くなります。段ボールにまとめて詰める(アーカイブ)と、往復回数が激減して速い。一方、本を圧縮袋に入れて潰す(JPEG 圧縮)と容量は減りますが、開くときに袋を破る手間(デコード)が増えます。

学習データの保存も同じ構図です。「ファイル1個ずつのオーバーヘッド(往復)」「圧縮による容量削減と展開コストのトレードオフ(圧縮袋)」「ランダムに1冊取り出せるか、箱を順に開けるしかないか(アクセスパターン)」——この3つの綱引きで最適解が決まります。

特に見落としがちなのが「小さなファイルが何百万個もできる」問題です。画像1枚を1ファイルにすると、データセット規模で数百万〜数千万ファイルになります。ファイル1個ずつの open/close コスト、メタデータ管理、クラウドストレージ(S3 など)への大量アップロードが、画像本体の読み書きより支配的になります。S3 は1リクエストあたりの課金とレイテンシ(数十ミリ秒)があるため、500万個の小ファイルを1個ずつ取りに行くと、純粋なリクエスト往復だけで膨大な時間とコストになり、現実的に扱えません。

## 基礎: 前提となる概念

保存形式を語る前に、コストの内訳を理解します。

- ランダムアクセス vs 順次(シーケンシャル)アクセス: ランダムアクセスは「バラバラの位置を飛び飛びに読む」。順次アクセスは「先頭から連続して読む」。ディスク(特に HDD やネットワークストレージ)は順次の方が圧倒的に速い。SSD でも、小さなランダム読み込み(4KB など)は帯域を活かしきれず、大きな連続読み込みの数分の一の実効速度になります。
- 圧縮(可逆 vs 非可逆): 可逆(lossless)は元データに完全に戻せる(PNG, gzip)。非可逆(lossy)は人間の目に分かりにくい情報を捨てて容量を減らす(JPEG, Wallace 1992)。学習では多くの場合 JPEG の劣化は許容範囲ですが、微小な物体検出やセグメンテーションでは劣化が効くことがあります。
- bit 深度とチャンネル: 通常の画像は1画素あたり R/G/B の3チャンネル、各8ビット(0〜255)。これを uint8 と呼びます。学習で内部的に float32(32ビット浮動小数点)に変換することもありますが、ディスク保存は uint8 で十分です(float32 で保存すると無駄に4倍になります)。
- ファイルシステム/オブジェクトストレージのメタデータコスト: ファイル1個ごとに inode(ファイル管理情報)や S3 オブジェクトのキーが必要。数が増えると、リスト操作や同期だけで膨大な時間がかかります。多くのファイルシステムは数百万 inode で `ls` すら重くなります。

まず、生(無圧縮)の1フレームのサイズ感をつかみます。学習入力を 256×256・3チャンネル(RGB)にリサイズした場合:

- 画素数 = 256 × 256 × 3 = 196,608 個の数値
- float32(1個4バイト)なら: 196,608 × 4 ≈ 786 KB/フレーム
- uint8(1個1バイト)なら: 196,608 × 1 ≈ 192 KB/フレーム
- JPEG(圧縮あり)なら、内容にもよるが概ね 15〜40 KB/フレーム

この時点で、float32 と JPEG では1フレームあたり約20〜50倍の容量差があります。

## 仕組みを詳しく

データセット規模を仮定して総容量を見積もります。仮に有効サンプル点が「1クリップ72点 × カメラ7台」で、クリップが10,000本あるとすると、

- 総フレーム数 = 72 × 7 × 10,000 = 約 504万フレーム
- JPEG(平均25KB)なら: 504万 × 25KB ≈ 約 126 GB
- uint8 生(192KB)なら: 504万 × 192KB ≈ 約 968 GB(約1TB)
- float32 生(786KB)なら: 504万 × 786KB ≈ 約 3.96 TB

この一桁以上の差が、形式選択の核心です。クラウドストレージの月額課金は容量に比例するので、3.96TB と 126GB では運用コストが30倍以上違います。以下、形式別に。

個別 JPEG ファイル

- 非可逆圧縮。容量が小さい(上の例で約126GB)。
- 欠点: ファイル数が爆発(上の例で504万個)。S3 アップロードやファイルシステムが悲鳴を上げる。微小な画質劣化。

個別 PNG ファイル

- 可逆圧縮(元に完全に戻る)。JPEG より容量大(写真的な画像で JPEG の数倍)、ファイル数問題は同じ。
- 画質が要る場合に。ただし学習では通常 JPEG の劣化は許容範囲。

tar アーカイブ(WebDataset 形式)

- 多数の画像を1本の tar に束ねる(これを shard と呼ぶ)。読み込みは「順次」になり、ディスク/ネットワーク帯域いっぱいで速い。
- ファイル数問題を解消(数百万ファイル → 数百〜数千本の shard)。S3 との相性も良い(大きなオブジェクトを順次ストリーム)。1 shard を数百MB〜数GB にするのが定石です。
- 欠点: tar 内の任意1枚へのランダムアクセスは苦手(順に流す前提)。だからシャッフルは shard 単位 + バッファ(buffered shuffle)で行う。完全なランダムシャッフルは諦め、近似シャッフルで妥協する設計です。

TFRecord

- TensorFlow 由来のシーケンシャルレコード形式。WebDataset と同じく「小ファイルを束ねて順次読む」発想。protocol buffer でシリアライズされ、`tf.data` と密結合。TensorFlow 圏では事実上の標準。
- 欠点: 中身を確認するのに専用ツールが要る(tar のように `tar tf` で覗けない)、PyTorch からは扱いづらい。

HDF5

- 1ファイルに多次元配列やメタデータをまとめて格納できる科学計算用フォーマット。チャンク(塊)単位の読み込みや圧縮フィルタに対応し、配列の一部だけを効率よく読める。
- 欠点: 並列書き込みや複数ワーカー読み込みでロック・設定の難しさがある。fork されたプロセスでファイルハンドルを共有すると壊れることがあり、深層学習では「各ワーカーでファイルを開き直す」などの作法が要る。

LMDB

- メモリマップ型の Key-Value ストア。キー(フレームID)から値(エンコード済み画像バイト)を高速にランダム取得できる。Caffe 時代から画像分類学習でよく使われました。
- 欠点: 単一ファイルが巨大化し、書き込み時にマップサイズを事前確保する必要。分散ファイルシステム(NFS など)上での挙動に注意。

メモリマップ(memory-mapped)テンソル

- 1個の巨大ファイルに全フレームのテンソルを連結し、OS の仕組みで「ディスク上のファイルをメモリのように」読む方式(`numpy.memmap` など)。OS のページキャッシュが効くので、2回目以降のアクセスはメモリ速度になります。
- インデックスでのランダムアクセスが非常に速い。デコード不要(生テンソルを直接読む)。FFCV はこの系統を高度化したものです。
- 欠点: 無圧縮なので容量が巨大(上の例で TB 級)。事前にリサイズ解像度を固定するので後から変えにくい。

WebDataset の1サンプルは、関連する複数ファイルを共通の接頭辞(キー)で束ねる慣習があります。たとえばマルチカメラのサンプルは概念的にこうなります。

```
__key__: sample000123
cam_0.jpg, cam_1.jpg, ..., cam_6.jpg   # 各カメラ画像(JPEG)
ego.npy                                # 自車状態など数値(numpy 生バイト)
meta.json                              # メタ情報
```

読み込み側は JPEG をデコードしてテンソル化、`.npy` を numpy 復元、という流れで1サンプルを組み立てます。「画像は JPEG、数値配列は npy、メタは JSON」と種類ごとに最適な形式を混在させられるのが WebDataset の柔軟さです。拡張子で型を見分けるので、デコード処理を拡張子に紐づけて自動化できます。

## 手法の系譜と主要論文

- JPEG(Wallace, "The JPEG Still Picture Compression Standard", IEEE TCE 1992): 非可逆画像圧縮の標準。DCT(離散コサイン変換)で空間周波数に分解し、人間が知覚しにくい高周波成分を粗く量子化して容量を減らす。学習データ保存で容量を一桁減らす土台技術。
- LMDB / Caffe 時代(2014年頃): 画像分類の標準パイプラインは「全画像を LMDB に詰める」ことでした。個別ファイルの遅さを Key-Value ストアで回避する初期の答え。
- TFRecord(TensorFlow, 2015年〜): プロトコルバッファのレコードを連結したシーケンシャル形式。Google 内部の大規模学習で標準化。順次読み込み + シャーディングという発想を広めました。
- WebDataset / AIStore(Aizman, Maltby, Breuel, "High Performance I/O For Large Scale Deep Learning", IEEE Big Data 2019, arXiv:2001.01858): POSIX の tar をそのまま shard として使う発想。専用フォーマットでなく標準の tar を使うことで、既存ツールでの中身確認・S3 ストリーミング・言語非依存を実現。「小ファイル爆発」と「ランダムアクセスの遅さ」への明快な回答として、PyTorch コミュニティで広く採用されました。
- FFCV(Leclerc et al., MIT, "FFCV: Accelerating Training by Removing Data Bottlenecks", CVPR 2023): 独自バイナリ形式(.beton)にデータを変換し、メモリマップ + JIT コンパイルされた前処理 + GPU 転送パイプラインで、ImageNet 学習を劇的に高速化。「保存形式を変えるだけで学習が速くなる」ことを定量的に示した代表例。
- MosaicML Streaming / WebDataset の発展(2022年〜): オブジェクトストレージから shard を直接ストリームし、ローカル完全同期を待たずに学習開始できる形式。クラウドネイティブ学習の標準形式へ。

設計上の一般的な結論は「画質劣化を許容できるなら JPEG を tar(WebDataset)に束ねる」が容量・速度・運用のバランスで最も無難、というものです。容量を切り詰めたい大規模学習や、デコードすら省きたい超高速学習では memmap 系(FFCV など)が選ばれます。

## 論文の実験結果(定量データ)

- WebDataset/AIStore(IEEE Big Data 2019)の報告: tar shard を用いた順次読み込みで、小ファイルを個別に読む場合に比べてストレージ帯域をほぼ理論限界まで引き出せることを示しました。とくにネットワークストレージ(オブジェクトストレージ)からの学習で、個別ファイルアクセスのレイテンシ累積を排除できる効果が大きい。S3 のような高レイテンシ・高帯域なストレージで、shard 化が劇的に効きます(レイテンシ数十ミリ秒 × 数百万回が消える)。
- FFCV(CVPR 2023)の報告: ImageNet で ResNet を学習する際、データ供給を FFCV の memmap + JIT パイプラインに置き換えるだけで、PyTorch 標準の JPEG ディスク読み込み構成に比べ学習スループットが大きく向上し、ResNet-18 を1 A100 で約30分以下に到達させる構成を公開。モデルもハイパーパラメータも同一で、保存形式とデータ経路だけが差、という点が重要です。FFCV はさらに「画像を最大辺でリサイズしてから保存」「一部だけ生、残りは JPEG」といった保存時の工夫で容量とデコード負荷のバランスを取れることも示しています。
- 容量と画質のトレードオフ(一般的観測): JPEG の品質パラメータ(quality 75〜95)を下げると容量は減りますが、ある閾値を下回ると分類・検出精度がわずかに低下する報告があります。多くの画像分類タスクでは quality 90 前後で精度劣化はほぼ無視できる一方、容量は PNG の数分の一に収まる、というのが実務の落としどころです。一方、細かいテクスチャやエッジが重要なセグメンテーション・小物体検出では、quality を下げると mAP(物体検出の平均精度)や mIoU(セグメンテーションの平均重なり率)が無視できないほど落ちることがあり、タスク依存で検証が必要です。

数値の意味を補足します。スループット向上は学習時間の短縮に直結し、クラウド GPU コストの削減になります。ストレージ容量の削減(TB → GB)は月額ストレージ課金と転送コスト(egress、ストレージから出る通信の課金)の削減に直結します。形式選択は精度をほとんど変えずにコストと時間を大きく動かせる、コストパフォーマンスの高い最適化ポイントです。

## メリット・トレードオフ・限界

- 個別 JPEG: 容量小・直感的だが、ファイル数爆発で S3/FS 負荷大、微小な画質劣化。
- 個別 PNG: 無劣化だが容量大、ファイル数問題は同じ。
- tar(WebDataset): ファイル数問題を解消し順次読み込みが高速、S3 相性が良い。代償として tar 内ランダムアクセスは不得手(shard 単位 + バッファのシャッフルで近似)。
- TFRecord: 順次読みが速く TensorFlow と密結合。中身確認に専用ツールが要り、PyTorch から扱いづらい。
- LMDB: キーでのランダムアクセスが速い。単一巨大ファイルの扱いと分散環境での注意。
- HDF5: 1ファイルに集約でき圧縮も可、配列の部分読みが得意。並列 I/O・fork 安全性の運用が煩雑。
- memmap / FFCV: ランダムアクセス最速・デコード不要。代償に容量が TB 級、解像度を後から変えにくい。

共通の綱引きと限界:

- 圧縮(JPEG)を選ぶと容量と転送速度を得る代わりにデコードコストを払う。無圧縮(memmap)を選ぶとアクセス速度を得る代わりに容量を失う。この「容量 ↔ デコード負荷」のトレードオフは原理的に避けられません。
- シャッフルとの相性: 完全ランダムシャッフルが要る研究設定では、shard ベースの近似シャッフルが学習収束に影響しないか検証が要ります(一般には大きな問題にならないが、データに強い順序相関——たとえば時系列で連続するフレーム——があると効く)。
- 解像度の固定: 抽出時に解像度・色空間・正規化を固定すると、後から「高解像度で学習し直したい」「別の前処理を試したい」が効きにくい。研究の探索段階では柔軟性を、本番では速度を、というジレンマがあります。妥協案として「やや大きめの解像度で保存し、学習時にランダムクロップ」する手があります。
- 再現性: JPEG 圧縮は実装やパラメータ(quality、サブサンプリング設定)で微差が出るため、論文の数値を完全再現するには圧縮設定まで合わせる必要がある場合があります。

## 発展トピック・研究の最前線

- 学習時オンザフライ拡張との両立: データ拡張(ランダムクロップ、色変換)はランダム性が要るのでキャッシュしにくい。保存は素の画像にしておき、拡張は GPU で実行する(DALI / FFCV)ハイブリッドが主流です。保存形式は「拡張前の素材」を持ち、拡張は実行時に乗せる、という役割分担です。
- 多階層キャッシュ: コールドデータはオブジェクトストレージ、ホットデータはローカル NVMe、という階層化。大規模分散学習でストレージ帯域がボトルネックになるのを防ぎます。
- 圧縮の進化: AVIF や WebP など新しい画像圧縮で、JPEG より高画質・低容量を狙う動き。ただしデコードライブラリの対応状況とデコード速度がトレードオフになります([動画フレームデコードライブラリ比較](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0902_video-frame-decoding-libraries))。デコードが遅ければ、容量を減らしてもデータ律速が悪化しうるので、容量とデコード速度を両にらみで選びます。
- データバージョニング: 抽出済みデータセットを LakeFS や DVC でバージョン管理し、どの前処理・どの解像度で作ったかを再現可能にする MLOps の流れ。保存形式そのものに「いつ・どの設定で作ったか」のメタデータを埋め込む実践も広がっています。
- 圧縮ドメイン・潜在表現の保存: 生画像でなく、エンコーダで圧縮した潜在ベクトルやトークン列を保存しておき、学習時はそれを読むだけにする発想(自己教師あり事前学習や生成モデルで増加中)。保存容量とデコード負荷の両方を削れる一方、保存した表現がモデルに固定されてしまう制約があります。
