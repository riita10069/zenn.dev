# egomotionのダウンサンプリング

## ひとことで言うと
自車(自分の車)がどう動いたかの記録(egomotion=エゴモーション)は、多くのセンサーログで1秒間に100回(100Hz)というとても細かい間隔で残ります。一方、学習する運転モデルが扱うのは1秒10回(10Hz)程度です。この「細かい記録を粗い時間刻みに間引く」操作をダウンサンプリングと呼びます。やり方には大きく2つの流派があり、(1)単純に10行ごとに1行抜き出す行ストライド方式と、(2)タイムスタンプ(時刻)を基準に目標時刻へ揃えるリサンプル方式です。実データには欠損やレートのゆらぎがあるため、後者のほうが頑健です。本ノートはこの違いと、どちらをいつ選ぶべきかを、信号処理の原理まで掘り下げて説明します。

## 直感的な理解
ビデオを早送りすることを想像してください。10倍速で見たいとき、素朴には「10コマに1コマだけ表示する」ことが思いつきます。録画が完全に等間隔(毎秒きっちり100コマ)なら、これで正確に10倍速になります。ところがもし録画機が時々コマを取りこぼしていたら、「10コマ進む」あいだに実際の時間が一定でなくなり、早送りの速度がガタつきます。

自動運転のegomotionも同じです。本当に取りたいのは「100msごとの時刻の値」なのに、行番号で間引くと、ログに欠けがあったり記録レートが揺れたりした瞬間に、取り出した時系列の実時間間隔が想定とズレます。運転判断は「何秒先に何が起きるか」という時間軸そのものに依存するので、このズレは静かに精度を蝕みます。たとえば「未来6.4秒先の位置を予測する」というタスクで、間引きが時間軸をずらすと、モデルは実際には5秒先や7秒先を予測していることになり、しかもそのズレは検証データ上では見えにくい。だからこそ「行番号で間引くか、時刻で揃えるか」という一見些末な選択が、データパイプラインの頑健性を左右します。

## 基礎: 前提となる概念

egomotionとは、自車自身の運動状態の時系列です。典型的には次のような信号が一定レートで記録されます。

```
timestamp, qx, qy, qz, qw, x, y, z, vx, vy, vz, ax, ay, az, curvature
```

各記号の意味を噛み砕くと、timestamp は記録時刻(普通はマイクロ秒やナノ秒の整数)、qx〜qw は姿勢(車がどちらを向いているか)を表すクォータニオン(回転を4つの数で表す数学的道具。後述するが、ジンバルロックという特異点を避けられるため姿勢表現に好まれる)、x/y/z は位置、vx〜vz は速度、ax〜az は加速度、curvature は曲率(進路の曲がり具合で、ハンドルの切れ角に対応する量。半径 R の円弧を描いているなら曲率は 1/R)です。

ここで押さえるべき用語を4つ。

- サンプリングレート(Hz、ヘルツ): 1秒あたり何回値を記録するか。100Hzなら1秒に約100行。Hzは「1秒あたりの回数」という意味の単位です。
- サンプリング周期: レートの逆数で、1サンプル間の時間。100Hzなら周期は1/100秒=10ms(ミリ秒、千分の1秒)。
- ダウンサンプリング(間引き、decimation): 高いレートを低いレートに落とすこと。100Hz→10Hzなら10分の1に間引く。
- エイリアシング(折り返し雑音): 間引きで起こりうる現象で、本来は高い周波数の成分が、低い周波数の偽の信号に化けて混入してしまうこと。後述するが、本来は間引く前にローパスフィルタ(高周波を落とす処理)をかけて防ぐべきものです。

### ナイキスト周波数とサンプリング定理(なぜ「半分」が境界なのか)

ダウンサンプリングを正しく理解するには、シャノン=ナイキストのサンプリング定理(Shannon, 1949)を知っておくと見通しが良くなります。これは「ある信号に含まれる最高周波数を f とすると、それを失わずにデジタル化するには最低でも 2f のレートでサンプリングしなければならない」という定理です。逆に言えば、レート fs でサンプリングしたデータが正しく表せる周波数の上限は fs/2 で、これをナイキスト周波数と呼びます。10Hzにダウンサンプルするなら、表せる上限は5Hzです。元の信号に5Hzより速い変動(例えば路面の細かい振動が伝わる10Hzの加速度ゆれ)が含まれていると、それは5Hz以下の偽の遅い信号に「折り返って」混入します。これがエイリアシングの正体です。本来は間引く前に5Hzより上を削るローパスフィルタを通すべき、というのが信号処理の教科書的な答えです。

### なぜ10Hzへ落とすのか

理由は2つあります。第一に、運転の文脈を表すには十分長い時間幅が要るからです。100Hzのまま例えば64ステップ取っても0.64秒ぶんしか見られず、車線変更や合流の判断には短すぎます。10Hzなら64ステップで6.4秒ぶんの過去・未来を表現でき、運転の意思決定スパンに合います。第二に、隣り合う100Hzのサンプルはほぼ同じ値で冗長です。100分の1秒で車の状態はほとんど変わらないので、10倍に間引いても運転判断に効く情報量はほぼ落ちません。実際、軌道予測の主要データセット(Argoverse 2、Waymo Open Motion など)はおおむね10Hz前後でアノテーションを提供しており、10Hzは「運転の意思決定に必要十分な時間分解能」として業界に定着した値です。

## 仕組みを詳しく

### 方式1: 行ストライド(位置ベース)

最も単純な実装は、行番号だけを頼りに等間隔で抜き出す方法です。ソースが100Hz、目標が10Hzなら、間引きステップは 100 / 10 = 10 です。pandas なら次のように書けます。

```python
SOURCE_HZ = 100.0
TARGET_HZ = 10.0
STEP = int(SOURCE_HZ / TARGET_HZ)  # = 10
df_down = df.iloc[::STEP].reset_index(drop=True)
```

`df.iloc[::10]` は「先頭から10行ごとに1行だけ拾う」という意味で、0, 10, 20, 30, ... 行目を取り出します。行番号(整数インデックス)だけを見て機械的に間引いている点が肝心です。タイムスタンプ列は一度も参照しません。

before/after を具体例で見ます。元データが理想的な100Hz(時刻が 0, 10, 20, ... ms と等間隔)なら、

- before: 0, 10, 20, 30, ..., 990 ms(100行/秒)
- after(`iloc[::10]`): 0, 100, 200, ..., 900 ms(10行/秒) → きれいに10Hzになる

つまり「サンプリングが完全に等間隔で欠損もない」という前提では、行ストライドは正しく動きます。計算量も O(N) で、列のスライスだけなので極めて高速です。

### 方式2: タイムスタンプ基準リサンプル(時刻ベース)

もう一方は、行番号ではなく `timestamp` 列の実際の時刻を見て、「100ms間隔の目標時刻に最も近い行(最近傍)」または「前後の値から補間した値」を選ぶ方法です。pandas の `resample("100ms")` や、目標時刻列を生成して `np.searchsorted` / `argmin` で最近傍を引く実装がこれにあたります。最近傍方式の擬似コードは次のようになります。

```python
import numpy as np
t0, t1 = df["timestamp"].iloc[0], df["timestamp"].iloc[-1]
targets = np.arange(t0, t1, 100_000)        # 100ms = 100,000 us 刻みの目標時刻
idx = np.searchsorted(df["timestamp"].values, targets)  # 各目標時刻の挿入位置
idx = np.clip(idx, 0, len(df) - 1)
df_down = df.iloc[idx].reset_index(drop=True)
```

なぜこちらが頑健か。実データでは次の3つが頻繁に起こるからです。

- 行の欠損(missing rows): センサーやログ書き込みの取りこぼしで、本来あるべき行が抜けている。すると「10行ごと」が時刻として等間隔でなくなる。例えば途中で9行欠けていれば、10行進んでも実時間は本来の倍進んでいるかもしれない。
- レートのゆらぎ(jitter、ジッタ): 実際のレートが厳密な100Hzでなく、98Hzだったり102Hzだったりする。書き込みタイミングがOSのスケジューリングに左右されることは珍しくない。行ストライドはこのズレを完全に無視する。
- ギャップ(間欠): ログが一時的に途切れた区間。行ストライドはギャップをまたいでも気づかず、不連続な過去・未来を地続きのものとしてつないでしまう。これは「数秒間記録が飛んだのに、その前後をなめらかな連続データと誤認する」最悪のケースを生む。

タイムスタンプ基準なら、毎回「目標時刻に最も近い行」を選ぶので、欠損やゆらぎがあっても狙った10Hzの時間軸を保てます。さらにギャップ検出(隣接時刻の差が想定より大きい区間を弾く、例えば `diff > 1.5 × 期待周期` の区間を境界とする)と組み合わせれば、不連続区間を学習データから除外できます。

### 最近傍か、補間か

時刻ベースにも2系統あります。最近傍(nearest)は「目標時刻に一番近い既存サンプルをそのまま使う」方式で、元データにない値を作らない安全性がありますが、目標時刻と実サンプルが最大で周期の半分(100Hzなら5ms)ずれます。補間(interpolation)は「前後2点から目標時刻ちょうどの値を合成する」方式で、時刻はぴったり合いますが、元になかった値を作るため信号の種類ごとに正しい補間法を選ぶ必要があります。位置 x/y/z は線形補間でよく、速度・加速度も近似的には線形で構いませんが、姿勢クォータニオンは線形補間すると単位ノルムが崩れて無効な回転になるため、球面線形補間 Slerp(Shoemake, SIGGRAPH 1985)を使うのが正しい流儀です。Slerp は2つの回転を球面上の最短弧に沿って一定角速度で補間する手法で、回転の補間における標準解です。

### 窓の切り出し

ダウンサンプリング後は、選んだ「現在時刻」を中心に、過去Nステップを履歴(history)、未来Nステップを目標(target)として切り出すのが一般的です。例えば10Hzで64ステップなら、過去6.4秒・未来6.4秒の文脈が得られます。テンソル形状で言えば、特徴次元 D の信号を取るなら履歴は `[64, D]`、目標は `[64, D_target]` のような形になります。履歴側には速度・加速度・ヨー角・曲率といった信号、目標側には予測したい量(加速度・曲率・将来位置など)を並べる、といった設計がよく使われます。ここで時間軸が狂っていると「6.4秒先を予測しているつもりが実は5秒先だった」ような系統誤差が入るため、間引き方式の頑健性が直接効いてきます。

### カメラとの時刻合わせとの整合

多くのパイプラインでは、カメラ画像とegomotionの対応付けはすでに時刻ベースで行われます。典型的には `np.argmin(np.abs(camera_timestamps - egomotion_timestamp))` のように、egomotion の時刻に最も近いカメラフレームを選びます。ところが egomotion 自身の 100→10Hz 間引きだけは行ストライドのまま、という非対称な実装になっていることがしばしばあります。片方は時刻ベースで頑健なのに、もう片方は位置ベースで脆い、という不整合は、パイプラインのレビューで見落とされやすい典型例です。整合を取るなら間引き側も時刻ベースに寄せるのが筋で、こうした「同じパイプライン内で時刻基準と行基準が混在する」状態はバグの温床になります。

## 手法の系譜と主要論文

これは特定の自動運転論文というより、信号処理と時系列前処理の一般原則に根ざした論点です。系譜を追うと、原理の確立→ツール化→ベンチマークでの標準化、という流れが見えます。

- サンプリング定理(Shannon, "Communication in the Presence of Noise", Proc. IRE, 1949): デジタル化で失わずに表せる周波数の上限がレートの半分(ナイキスト周波数)であることを確立しました。ダウンサンプリングが情報を壊しうるのはこの定理が根拠です。
- 信号処理の古典(Oppenheim & Schafer, "Discrete-Time Signal Processing", 第3版, 2009): ダウンサンプリング(decimation)は本来、エイリアシングを避けるために間引く前にローパスフィルタをかけるべき、と説きます。単純な行間引き(filtering なしの decimation)は、急な加速度変化のような高周波成分を取りこぼす、あるいは偽の低周波成分に化けさせるリスクがあります。理想的にはカットオフ周波数を目標ナイキスト周波数(10Hz なら5Hz)に設定したフィルタを通してから間引きます。
- 回転補間(Shoemake, "Animating Rotation with Quaternion Curves", SIGGRAPH 1985): クォータニオンの球面線形補間 Slerp を提案。姿勢を時刻基準で補間する際の標準手法で、egomotion の姿勢リサンプルに直接効きます。
- 時系列ライブラリ(pandas の `resample()`、一次ドキュメント): 時系列インデックス(時刻)に基づいて固定間隔へ揃える機能を提供し、`iloc[::n]`(位置ベース)と `resample("100ms")`(時刻ベース)の違いを明確にしました。実データの不等間隔・欠損に強いのは後者です。
- 軌道・運動予測のベンチマーク(nuScenes, Caesar et al., CVPR 2020, arXiv:1903.11027; Argoverse 2, Wilson et al., NeurIPS 2021; Waymo Open Motion, Ettinger et al., ICCV 2021, arXiv:2104.10133): egomotion や軌道を一定Hz(nuScenes のアノテーションは2Hz、軌道予測系は10Hz)へ「時刻基準で」リサンプルするのが通例です。ログの欠損・ゆらぎを前提にした実装が標準的で、行ストライドをそのまま信頼することは稀です。

系譜としては「サンプリング定理が理論限界を定め → 古典的信号処理がフィルタ付き decimation の必要性を確立 → 時系列ライブラリが時刻ベース resample を一次機能として提供 → 自動運転データセットが時刻基準リサンプルを前処理の標準とした」という流れで、行ストライドは「等間隔が保証される限定的な場面のショートカット」という位置づけに収まっています。

## 論文の実験結果(定量データ)

この論点は単独の手法論文として大規模なベンチマークが取られることは少なく、定量評価は主に2つの形で現れます。

第一に、サンプリングレート自体がモデル性能に与える影響です。軌道予測の研究では、入力履歴の時間刻みを変えてアブレーション(ある要素を抜いたり変えたりして効果を測る実験)することがあり、一般に「10Hzより粗くすると急な動き(急ブレーキ・急ハンドル)を捉えられず予測誤差が増える」「100Hzのまま入れても冗長で計算が増えるだけで精度はほぼ頭打ち」という傾向が報告されます。ここで使う指標が ADE(Average Displacement Error, 平均変位誤差)と FDE(Final Displacement Error, 最終変位誤差)です。ADE は予測軌道と正解軌道の各時刻の距離(メートル)を全時刻で平均した値、FDE は最終時刻だけの距離で、いずれもメートル単位で小さいほど良い指標です。たとえば ADE が 1.0m から 1.5m に増えるというのは「平均で50cm余計に外す」ことを意味し、車線幅(約3.5m)に対して無視できない誤差です。10Hz前後が「情報量と計算量のスイートスポット」とされる根拠がここにあります。

第二に、欠損・ジッタへの頑健性です。これは正面から数値化されることは少ないものの、欠損を人工的に注入したアブレーションでは、行ストライド方式が時間軸のズレに比例して系統誤差を持ち込むのに対し、時刻ベースリサンプルはほぼ影響を受けないことが示せます。たとえば「平均5%の行が欠損したログ」を行ストライドで処理すると、64ステップ窓の実時間長が想定の6.4秒から約6.7秒へ伸び、未来予測の対応時刻が系統的にズレます。秒速30mで走っていれば、0.3秒のズレは約9mの位置ズレに相当します。この種のズレは検証データ上では見えにくく(検証データも同じ欠損特性を持つため誤差が相殺される)、実車でのドメインギャップとして表面化するため、定量化しにくいが実務上は重大です。

## メリット・トレードオフ・限界

- 行ストライド `iloc[::n]`
  - 利点: 実装が極めて単純・高速。等間隔・無欠損データでは正しく目標レートになる。外部依存も最小。計算量 O(N)。
  - 欠点: 行欠損・レートゆらぎ・ギャップに脆い。時刻が等間隔である保証がないと、過去/未来窓の実時間長が想定とズレる。間引き前フィルタがなければエイリアシングのリスクも残る。ギャップを地続きにつないでしまう危険がある。
- タイムスタンプ基準リサンプル
  - 利点: 欠損・ゆらぎに頑健で、狙った時間軸を保てる。ギャップ検出と組み合わせれば不連続区間を弾ける。カメラ・LiDAR との時刻合わせと方式が揃う。
  - 欠点: 実装がやや複雑(目標時刻の生成、最近傍選択または補間、欠損判定が必要)。補間を入れると元データにない値を合成するため、姿勢には Slerp を使うなど信号ごとに正しい補間法を選ぶ必要がある。最近傍方式でも周期の半分のズレが残る。

研究上の未解決の論点としては、(1)どの信号にどの補間法が適切か(位置は線形でよくても回転はSlerp、加速度はそもそも急変するので補間すべきでない場合もある)、(2)欠損・ギャップをどこまで許容して学習に含めるか(過度に弾くとデータが減り、緩いと不連続が混入する)、(3)間引き前のアンチエイリアスフィルタを入れるべきか(運転信号は急変が多く、フィルタが本物の急加速を鈍らせる懸念もあり、保存したい高周波成分そのものがフィルタで削られるジレンマがある)といったトレードオフが残ります。これらは「どの程度の頑健性が必要か」というタスク依存の判断で、万能の正解はありません。

## 発展トピック・研究の最前線

時刻ベース前処理の延長線上には、複数センサーの非同期な時刻を統一する時刻同期(temporal alignment、テンポラルアライメント)の問題があります。カメラ・LiDAR・IMU・GPS はそれぞれ異なるレートと位相で記録され、これらを一つの時間軸に揃えるのは知覚パイプラインの基盤工程です。たとえばLiDARは回転式だと1スキャンに時間がかかり、スキャン中に車が動くため点群そのものに時刻のズレ(motion distortion)が生じます。これを補正する deskewing も、結局は egomotion を時刻基準で扱えることが前提です。

近年の end-to-end 学習では、固定レートにリサンプルせず、不規則な時刻そのものを入力に与える連続時間モデルも研究されています。Neural ODE 系(微分方程式としてダイナミクスを学習し、任意時刻の状態を求める)や、時刻を位置エンコーディングとして注入する Transformer は、「間引かずに不等間隔のまま扱う」方向の探求です。これは行ストライド vs 時刻ベースという二択を、より根本から解消しようとする動きです。さらに、イベントカメラのように非同期に発火するセンサーが普及すれば、「固定レートへのリサンプル」という前提自体が問い直され、時刻を一級市民として扱うアーキテクチャがより重要になると考えられます。
