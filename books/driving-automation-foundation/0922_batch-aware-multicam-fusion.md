# バッチ対応マルチカメラ融合

## ひとことで言うと
自動運転車には複数のカメラ（典型的に6〜7台）が付いていて、それぞれ別の方向を映しています。これらの画像を特徴抽出器（backbone）に通すとき、1台ずつ処理すると無駄が多いので、複数シーン（バッチ）ぶんの全カメラ画像をいったん「ただの画像の山」として平らに並べて一気に処理し、あとで「どのシーンのどのカメラか」に正しく戻す、という設計です。5次元テンソルを一時的に4次元に畳んで共有 backbone に通し、元に戻すのが核心です。

## 直感的な理解
工場のベルトコンベアを想像してください。6種類の部品（前方カメラ画像、左カメラ画像、右カメラ画像…）を別々のラインで1つずつ加工するより、全部を1本の大きなラインにまとめて流したほうが速い。加工機（backbone）は1台で済み、稼働率も上がります。加工が終わったら「どれが前方カメラの部品で、どれが左カメラの部品か」を仕分けて元の箱に戻すだけです。

ここで「箱に戻すときの順番」が決定的に重要です。コンベアに乗せた順番を記録しておけば、出てきた順に正しく仕分けられます。もし順番をいい加減に管理すると、シーンAの前方カメラ特徴とシーンBの左カメラ特徴が混ざってしまい、学習が静かに壊れます。エラーは出ません。ただ精度が上がらないだけ。だから「畳む順序と戻す順序を厳密に一致させる」ことが、この設計の生命線です。

なぜ共有 backbone なのか。どのカメラも「画像から特徴を取る」という同じ仕事をします。カメラごとに別の backbone を6個用意すると、パラメータが6倍になり、各 backbone は自分のカメラのデータしか見ないので学習データが分散して効率が落ちます。1つの backbone を全カメラで共有すれば、前方も左も右も全部のデータが同じ重みの学習に貢献し、データ効率が最大化されます。これは「重み共有（weight sharing）」と呼ばれる深層学習の基本戦略で、畳み込み層が画像のどの位置でも同じフィルタを使い回すのと同じ発想を、カメラという軸に拡張したものです。

## 基礎: 前提となる概念

### テンソルと shape
テンソル（tensor）は多次元の数値配列です。画像1枚はチャンネル・高さ・幅の3次元 `(C, H, W)` = 例 `(3, 256, 256)` で表されます（チャンネル3は RGB の赤緑青）。shape（形）はその各次元のサイズの並びです。深層学習の計算は突き詰めると「正しい shape のテンソルを正しい関数に渡す」ことの連続で、shape の取り違えが最も多いバグ源です。

### バッチ（batch）
機械学習ではデータを1個ずつでなく何個かまとめて処理します。このまとまりがバッチで、その個数がバッチサイズです。GPU は大量のデータを並列に同時計算するのが得意なので、まとめたほうが圧倒的に速い。バッチサイズ8なら8シーンを同時に流します。GPU の演算器（数千コア）を遊ばせず埋めるには、1枚ずつ流すのではなくまとめて流して並列度を上げる必要がある、というハードウェアの都合がこの設計の動機の一つです。

### backbone（バックボーン）
画像から特徴（feature）を取り出すニューラルネットワークの本体です。Swin Transformer V2、ConvNeXt V2、ResNet-50 などが代表例。画像をそのまま使うのではなく「ここに車がいそう」「ここは白線」といった抽象的な情報の格子（特徴マップ）に変換します。多くの backbone は複数解像度の特徴（マルチスケール特徴、浅い層は細かく位置を、深い層は広い文脈を捉える）をリストで返します。

### メモリレイアウトと reshape
テンソルはメモリ上では1次元の連続した数値の並びとして格納されます。shape は「その並びをどう読むか」のメタ情報にすぎません。reshape（形の変更）は、データの並び自体は一切動かさず、読み方だけを変える操作です。だから連続（contiguous）なテンソルなら reshape はコピーすら起こさず、ほぼコストゼロです。逆に言えば「並びがどうなっているか」を理解せずに reshape すると、意図しない混線が起きます。これが後述の順序問題の本質です。

### マルチカメラ入力の shape
自動運転のマルチカメラ入力は5次元テンソルになります。

```
camera_tiles: (B, V, C, H, W)
              (バッチ, カメラ台数, チャンネル, 高さ, 幅)
例: (8, 7, 3, 256, 256)   8シーン × 7カメラ × RGB × 256×256
```

問題は、画像用 backbone が4次元 `(N, C, H, W)`（N は画像枚数）しか受け取れないことです。5次元のまま渡すと形が合わずエラーになります。backbone は「画像が何枚あるか」しか気にせず、「その画像がどのシーンのどのカメラか」は知らないし知る必要もない、という前提で作られているためです。

## 仕組みを詳しく

「バッチ対応（batch-aware）」設計は、5次元を一時的に4次元へ畳み（reshape）、共有 backbone に通し、元の `(B, V, ...)` に戻します。

ステップ1: 形を覚える。

```python
B, V, C, H, W = camera_tiles.shape   # 例: 8, 7, 3, 256, 256
```

B と V を変数に取り出して保持します。後で戻すために必須です。

ステップ2: バッチとカメラ次元を畳む。

```python
x = camera_tiles.reshape(B * V, C, H, W)   # (56, 3, 256, 256)
features = backbone(x)
```

`(8, 7, 3, 256, 256)` を `(56, 3, 256, 256)` にします。8シーン × 7カメラ = 56枚が「56枚の画像の山」として1つの4次元テンソルになり、backbone に渡せます。重要なのは reshape はメモリ上のデータ並びを変えず「見え方（形）」だけ変える操作なので、ほぼコストゼロだということ。

backbone はマルチスケール特徴をリストで返します。Swin/ConvNeXt 系では4段階で、先頭次元は全部 `B*V=56`:

```
stage0: (56, 96,  64, 64)
stage1: (56, 192, 32, 32)
stage2: (56, 384, 16, 16)
stage3: (56, 768, 8,  8)
```

ステップ3: 融合に B と V を渡し、戻せるようにする。

```python
image_bev = feature_fusion(features, B, V, camera_params=camera_params)
```

融合モジュールは受け取った B と V を使い、`(B*V, C', H', W')` を `(B, V, C', H', W')` に戻し、カメラ間で情報を統合します。BEV 融合では典型的に次のように戻します。

```python
# fused_per_view: [B*V, C, H, W]
feat = fused_per_view.reshape(B, V, C, H * W).permute(0, 1, 3, 2)  # [B, V, H*W, C]
```

ここが肝です。`B*V` を `B` と `V` に割り戻すには「最初の V 個が0番目のシーン、次の V 個が1番目のシーン…」という並び順が保たれていなければなりません。ステップ2で `(B, V, ...)` の順（B が外側、V が内側）に畳んだので、戻すときも `reshape(B, V, ...)` で同じ順に割れば正しく対応します。「外側の軸が遅く回り、内側の軸が速く回る（row-major / C order）」という規則を、畳むときと戻すときで一貫させるのが要点です。

もし `(V, B, ...)` の順で畳んでしまうと、戻したときにシーンとカメラの対応が入れ替わります。具体的には:

```
正: reshape(B*V) の並び = [s0c0, s0c1, ..., s0c6, s1c0, s1c1, ...]
    → reshape(B, V) で [[s0c0..s0c6], [s1c0..s1c6], ...] と正しく割れる
誤: reshape(V*B) の並び = [c0s0, c0s1, ..., c0s7, c1s0, ...]
    → reshape(B, V) で割ると別物に混線。エラーは出ず精度だけ崩れる
```

往復の全体像:

```
(B, V, C, H, W)
   │ reshape(B*V, C, H, W)   ← カメラ次元を畳む（コストゼロ）
   ▼
(B*V, C, H, W)  →  共有 backbone  →  各スケール (B*V, C', H', W')
   │ reshape(B, V, ...)       ← カメラ次元を戻す（同じ順序で）
   ▼
(B, V, ...)  →  カメラ間融合（BEV 等）  →  (B, C, bev_h, bev_w)
```

この混線が怖いのは、勾配は普通に流れ、loss も下がっていくように見え、ただ最終精度が頭打ちになるだけ、という形でしか現れない点です。検出のために、テスト時に「区別できる既知のパターン（例: カメラごとに違う定数で埋めたダミー画像）を入れ、往復後に元の対応が復元されるか」を assert で確認するのが定石です。

## 手法の系譜と主要論文

マルチカメラを1つのネットワークで扱い俯瞰表現（BEV、Bird's-Eye-View、車を真上から見下ろした表現）に統合する発想は、近年のカメラ系3D認識論文で共通します。

- LSS（Lift, Splat, Shoot; Philion & Fidler, ECCV 2020, arXiv:2008.05711）。各カメラ画像を共有エンコーダで処理し、各ピクセルに離散深度分布を推定して3D空間に持ち上げ（lift）、俯瞰グリッドに貼り付け（splat）、最後にカメラ間で1つの BEV に統合します。共有エンコーダにした動機は、カメラごとに別モデルだと学習データが分散して効率が悪く、共有なら全カメラの画像が同じ重みに貢献するから。効果はカメラ台数の増減に同じネットワークでスケールできること。トレードオフは深度推定誤差が BEV 誤差に直結すること。

- BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）。学習可能な BEV クエリが spatial cross-attention で各カメラ特徴から情報を集める方式。深度を陽に推定せず「どのカメラのどこを見るか」をデータから学べます。実装上、複数カメラ・複数バッチを一括で扱うため本ノートの `(B, V, ...)` 畳み込み・展開を多用します。トレードオフは attention の計算量。

- DETR3D（Wang et al., CoRL 2021, arXiv:2110.06922）と PETR（Liu et al., ECCV 2022, arXiv:2203.05625）。いずれも複数カメラ特徴を共有 backbone で抽出し、3Dの問い合わせ（object query / 3D position embedding）で統合します。DETR3D は3Dクエリを各カメラに逆投影して特徴をサンプリングし、PETR はその逆投影を「2D特徴に3D位置埋め込みを足す」形で暗黙化して投影計算を不要にした点が新規です。

- BEVDet（Huang et al., 2021, arXiv:2112.11790）。LSS 系の認識を物体検出に最適化し、共有 backbone + BEV エンコーダの実用構成を確立。後続の BEVDet4D で時系列も取り込みました。

- BEVFusion（Liu et al., ICRA 2023, arXiv:2205.13542）。カメラ枝（共有 backbone + BEV 化）と LiDAR 枝を統一 BEV 空間で融合。カメラ枝は本ノートのバッチ対応共有処理を前提とします。

共通する設計思想は「カメラの台数や配置はデータごとに変わりうるが、画像から特徴を取る処理は共通化する」こと。これがバッチ対応設計と同じ系譜です。

## 論文の実験結果（定量データ）

標準ベンチマークは nuScenes（6カメラ360度、約1.4M枚の注釈画像、1000シーン）。評価指標:

- mAP（mean Average Precision）: 物体検出の平均適合率。BEV 上で予測と正解の中心距離が閾値（0.5/1/2/4m）内なら正解とみなし、クラスごとの適合率を平均。大きいほど良い。
- NDS（nuScenes Detection Score）: mAP に加え、位置・サイズ・向き・速度・属性の誤差を統合した nuScenes 独自の総合スコア。大きいほど良い。

報告されている代表的な数値:

- LSS 系の BEVDet は nuScenes で当時のカメラのみ手法として mAP 30%台前半、NDS 40%前後を達成し、共有エンコーダ + BEV 統合の有効性を示しました。
- BEVFormer は nuScenes test で NDS およそ 0.56、mAP およそ 0.48 と報告され、当時のカメラのみ手法のトップ級を記録。時系列の BEV を attention で集約することで、特に速度推定（mAVE、平均速度誤差）が単フレーム手法より大きく改善しました。アブレーションでは temporal self-attention（過去 BEV を参照する機構）を外すと速度誤差が顕著に悪化することが示され、マルチフレーム統合の寄与が定量化されています。
- DETR3D / PETR も nuScenes で NDS 0.4〜0.45 前後を達成。PETR は3D位置埋め込みを外すと mAP が大きく落ちることをアブレーションで示し、投影情報の埋め込みが性能の核であることを実証しました。
- BEVFusion はカメラ＋LiDAR で nuScenes test の NDS を 0.70 超まで押し上げ、統一 BEV 融合の強さを示しました。

数値の意味を補足します。NDS が 0.05 上がるというのは、検出の見逃し・位置ずれ・速度誤差を総合した品質が約5ポイント改善するということで、自動運転の知覚では「隣車線の車を正しい速度で捉えられるか」に直結する重要な差です。共有 backbone のバッチ対応処理自体は精度を上げる機構ではなく、これらの手法を現実的な計算量・パラメータ数で実装可能にする土台として効きます。仮にカメラごとに別 backbone を持てば、パラメータが V 倍に膨れ、同じ計算予算ではより小さな backbone しか使えず、結果として精度が落ちます。

## メリット・トレードオフ・限界

メリット:
- backbone を1つだけ持てばよく、パラメータ数とメモリがカメラ台数ぶん増えない。
- GPU が `B*V` 枚を並列に一気に処理でき、台数ぶんループするより速い（カーネル起動回数も減る）。
- カメラ台数 V を変えてもコードを書き換えずに動く（設定値を変えるだけ）。
- 全カメラの画像が同じ重みの学習に寄与し、データ効率がよい。

トレードオフ・限界:
- reshape の順序を間違えると、シーンとカメラの対応が崩れて学習が静かに壊れる（エラーが出ないので発見が難しい）。畳む順と戻す順を必ず一致させる規律が要る。テストで「既知の並びを入れて戻りが一致するか」を検証するのが定石。
- `B*V` が大きくなると一時的なメモリピークが上がる（例 8シーン×7カメラ=56枚を同時に GPU に載せる）。バッチサイズと解像度の上限が GPU メモリで決まり、高解像度 BEV では特に効く。マルチフレーム（過去数時刻）も入れると `B*V*T` 枚になり、メモリ圧がさらに増す。
- 全カメラで同じ backbone を使うため、特定カメラ専用の最適化（後方カメラだけ別処理、魚眼カメラ専用補正など）はやりにくい。カメラごとの光学特性差（焦点距離・歪み・露出）を backbone 自体では吸収できない。
- 未解決の課題として、カメラ間で重みは共有しつつカメラ固有の較正・歪みを軽量に注入する手法（カメラ ID 埋め込みや較正条件付け）、台数が動的に変わる場合の効率的なパディング処理などが研究途上です。

## 発展トピック・研究の最前線
共有 backbone + BEV 統合は、時系列方向への拡張（複数時刻の BEV を attention や warping で集約する BEVFormer v2, SOLOFusion など）と、深度の明示的教師あり（BEVDepth）に発展しています。また Transformer の attention 計算量を下げる sparse query（Sparse4D など、全 BEV マスではなく疎な点だけ問い合わせる）や、LiDAR との fusion（BEVFusion）でも、カメラ枝は本ノートのバッチ対応共有処理を前提とします。エンドツーエンド自動運転（UniAD, Hu et al., CVPR 2023 best paper など）でも、知覚の最前段はこのマルチカメラ共有 backbone であり、その上に検出・追跡・地図・計画のモジュールが積まれます。
