# ナビ地図 vs HD地図

## ひとことで言うと
自動運転モデルに「地図」を入れるとき、ナビ地図(SD 地図=粗い経路案内用の地図)と HD 地図(車線レベルまで精密な地図)という 2 種類の地図があり、その性質の違いと、それぞれをモデルにどう入れるかという入力設計の話です。粗いナビ地図は広く安く手に入る代わりに不正確、精密な HD 地図は正確だが高価で範囲が狭い、というのが要点です。両者は用途も最適なデータ表現も違うので、別々に扱うのが筋が良い、というのがこの章の主張です。

## 直感的な理解
カーナビの「次の交差点を右」という案内と、熟練ドライバーが「この交差点は右折レーンが 2 本あって、自分は左から 2 番目に入る」と判断するのとでは、必要な地図の細かさがまるで違います。前者は道路網の大まかなつながりが分かれば十分で、後者は車線 1 本単位の精密な形が要ります。

自動運転モデルも同じで、「だいたいどっちへ向かうか(経路の意図)」を伝えるにはナビ地図で足り、「どの車線のどこを走るか(精密な制御)」には HD 地図が要ります。ここで素朴に「地図を 1 種類のカメラ画像のように扱う」と、この性質の違いを無視してしまい、もったいないことになります。粗い地図の安さ・広域性という長所も、精密地図の正確さという長所も、どちらも中途半端にしか活かせません。そこで「ナビ地図用のエンコーダ」と「HD 地図用のエンコーダ」を分けて持ち、それぞれの長所を活かしてから統合する、という設計が生まれます。

別の言い方をすると、ナビ地図は「どこへ行きたいか(intent)」を伝える役割、HD 地図は「いまどう走れるか(constraint)」を伝える役割だと整理できます。役割が違うものを一つの入力経路に押し込めるより、役割ごとに最適なエンコーダを与えて後で合流させる方が自然です。

## 基礎: 前提となる概念
- ナビ地図(SD 地図、Standard Definition map)。カーナビが経路案内に使う地図です。道路を中心線 1 本くらいで表し、車線の本数や正確な幅は持たないことが多い。世界中をカバーし、OpenStreetMap(OSM、無料の世界地図)なら無料で手に入ります。
- HD 地図(High Definition map)。車線の境界線・横断歩道・停止線・信号との対応まで数十 cm 精度で持つ地図です。専用計測車で事前に作るので高価で、範囲が限られ、時間とともに古くなります。意味と関係を構造的に持つフォーマットとして [Lanelet2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0445_lanelet2) や OpenDRIVE があります。
- BEV(Bird's-Eye-View=鳥瞰図)。情報を真上から見た平面のグリッドで表す表現です。地図はもともと真上から見た平面情報なので、BEV と非常に相性が良い。運転モデルの多くは、カメラ画像を BEV へ変換した「画像 BEV 特徴」を中心に処理します。
- エンコーダ。入力(画像や点列)を、後段で使いやすい特徴ベクトルへ変換するニューラルネットの部分です。
- ラスタとベクトル。地図をピクセル画像(ラスタ)で渡すか、点列やグラフ(ベクトル)で渡すかの表現選択です。詳しくは [ラスタ地図 vs ベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0426_rasterized-vs-vectorized-map)。ナビ地図は面的に塗ったラスタが、HD 地図は精密な線を保つベクトルが、それぞれ自然です。
- cross-attention。ある側の各要素が、別の側の情報を選んで取り込む Transformer の機構です。「画像 BEV の各マスが、対応する地図 BEV のマスの情報を吸収する」融合に使います。
- ルート/intent(経路の意図)。「次に左折する」「この先の高架へ上がる」といった、運転者がこれから取る大まかな進路です。ナビ地図はこの intent の供給源になります。intent が無いと、分岐点で「どっちへ行くべきか」がモデルから決められません。

## 仕組みを詳しく
地図をモデルに入れる設計の核心は「地図はすでに真上から見た平面(BEV)の情報なのだから、カメラ画像と同じパイプラインに無理やり通すのは不適切。地図専用のエンコーダで BEV 特徴にして、画像由来の BEV 特徴と融合すべき」という考え方です。これを段階的に分解します。

1. ナビ地図(ラスタ=ピクセル画像として渡す)を画像系バックボーン(Swin Transformer や ResNet など)で BEV 特徴にする。ラスタなので CNN や画像 Transformer と相性が良い。面的に塗った地図は画像処理がそのまま効きます。
2. HD 地図(ベクトル=点列/グラフとして渡す)を、ベクトル地図用エンコーダ(点列を Transformer/PointNet 系で処理する仕組み)で BEV 特徴にする。Lanelet2 や OpenDRIVE のような構造化フォーマットを入力に想定します。座標を丸めず精密に保てます。
3. 地図 BEV 特徴と画像 BEV 特徴を cross-attention で融合する。片方の各マスが、もう片方の対応する特徴を参照して取り込みます。
4. 決定的に重要な条件として、地図 BEV のカバー範囲(縦横の実距離)を画像 BEV と同じに作る。たとえば両方とも「前方 50m × 左右各 25m」を同じグリッド解像度で表せば、同じインデックスのマス同士が地面の同じ地点を指すので、ズレなく重ね合わせられます。

形状イメージで言うと、画像 BEV が `[B, C, 200, 100]`(B はバッチ、C はチャンネル=特徴次元、200×100 はマス目の数)なら、地図 BEV も同じ `[B, C, 200, 100]` の範囲・解像度に揃えます。こうすると「この画像マスとこの地図マスは同じ場所」と素直に対応づき、融合が幾何的に正しくなります。範囲がずれていると、地図の情報が間違った場所に貼り付き、性能が悪化します。

```
ナビ地図(ラスタ) → 画像系バックボーン → 地図BEV [B,C,200,100] ┐
                                                            ├ cross-attn → 統合BEV
カメラ画像 → BEV変換 →            画像BEV [B,C,200,100] ┘
HD地図(ベクトル) → ベクトルエンコーダ → 地図BEV [B,C,200,100] ┘(同じ範囲に揃える)
```

なぜ 2 つを分けるのか、を一言で言えば「ナビ地図はラスタが自然・HD 地図はベクトルが自然」だからです。形式が違うものを同じエンコーダに無理やり通すより、それぞれ得意な処理をさせてから BEV で合流させる方が、情報を保ったまま扱えます。さらに、片方の地図が無い状況(HD 地図が整備されていない地域など)でも、もう片方だけで動かせる柔軟さも生まれます。

ここで一つ研究上の重要論点を補足します。「2 系統のエンコーダ」は表現の最適性の点では筋が良いのですが、可用性(availability)の観点でも重要です。HD 地図は世界の一部地域しか整備されていません。HD 地図に強く依存した設計だと、地図のない地域では性能が崩壊します。逆にナビ地図は世界をほぼカバーするので、HD 地図経路を「あれば使う・なければ落とす(graceful degradation=なだらかな劣化)」設計にできれば、地図整備に左右されない自動運転に近づきます。この「地図の有無で性能が崖から落ちないようにする」ことが、入力設計の隠れた目標です。

## 手法の系譜と主要論文
この区別自体は特定の論文の発明というより業界共通の概念ですが、関連する代表的な研究の流れを描けます。

- 地図を BEV ラスタで描いて CNN に渡す系。ChauffeurNet(Bansal et al., 2018)などの初期 end-to-end 手法は、地図と物体を多チャンネル BEV 画像にレンダリングして CNN に通しました。実装が単純で画像技術をそのまま使える反面、量子化誤差(連続座標をマス目に丸める誤差)や非効率の問題を抱えていました。
- ベクトル表現で地図を扱う系。VAD(Jiang et al., ICCV 2023, "VAD: Vectorized Scene Representation for Efficient Autonomous Driving")は、地図や周囲の物体をベクトルで表し、それをもとに end-to-end で運転計画する手法です。HD 地図エンコーダの候補としてよく挙げられる「ベクトル地図モジュール」はこの系統にあたります。ベクトル表現は軽く、計画に直接使え、精度も保てる点が動機です。さらに後継の VADv2 では、計画を確率分布として出力する確率的計画(probabilistic planning)へ発展しました。
- ナビ地図(SD 地図)を prior として使う系。P-MapNet(Jiang et al., RA-L 2024)や SMERF(Luo et al., ICRA 2024)は、SD 地図を弱い事前情報としてオンライン HD 地図構築・車線知覚に注入しました。詳しくは [OpenStreetMap SD地図事前情報](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0432_openstreetmap-sd-map-prior)。これらは「粗い地図でも経路の意図や道路接続の事前情報として役立つ」ことを定量的に示しました。
- HD 地図をオンラインで作って事前作成地図を代替する系。詳しくは [HDMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0442_hdmapnet) と [MapTRv2](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0452_maptrv2)。ナビ地図と HD 地図の関係を考える上で、「HD 地図は事前に作るのか、走りながら作るのか」という軸も絡んできます。事前作成 HD 地図(Lanelet2 など)の鮮度問題が、オンライン構築への動機になっています。

この系譜から見える設計思想の収束点は、「粗い情報(ナビ/SD)は安く広く、精密な情報(HD)は高価で局所的。だから両方を別表現で持ち、BEV という共通土俵で統合する」という二段構えです。

## 論文の実験結果(定量データ)
ナビ地図と HD 地図を分けて扱う設計の効果は、関連研究の数値から間接的に読み取れます。評価の中心は nuScenes(自動運転の標準ベンチマーク、計 1000 シーン)で、指標は mAP(mean Average Precision、地図要素の予測精度。高いほど良い)や、計画の指標としての L2 誤差(予測軌道と正解軌道の平均ずれ。低いほど良い)、衝突率(計画軌道が他者と衝突する割合。低いほど良い)です。

- ベクトル地図を計画に直結させる効果。VAD は、ベクトル化されたシーン表現を用いることで、ラスタ的な中間表現を経る手法に比べて、計画の L2 誤差と衝突率を下げつつ、推論速度を大幅に上げたことを報告しています。論文では、当時の代表的な end-to-end 手法(UniAD など)に対し、推論速度が数倍規模で速く(報告では十数 FPS 級に到達)、計画精度も同等以上という結果が示されました。「ベクトルで持つと軽くて速くて精度も良い」ことの実証です。
- SD 地図(ナビ地図)を足す効果。P-MapNet や SMERF は、SD 地図 prior を加えることで HD 地図構築の mAP、特に遠方領域で十数ポイント規模の改善を報告しています([OpenStreetMap SD地図事前情報](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0432_openstreetmap-sd-map-prior))。これは「ナビ地図は経路の意図や道路接続の事前情報として確かに効く」ことの定量的裏付けです。
- アブレーションの示唆。地図情報そのものを抜くと計画・知覚の性能が落ちること、地図 BEV と画像 BEV の範囲・解像度を揃えないと融合の効果が減ることが、関連研究で一貫して観察されます。後者は本章の「範囲を揃える」という設計条件の重要性を裏づけます。
- 注意すべき方法論的論点。近年、nuScenes の open-loop 計画評価(録画された正解軌道との L2 誤差で測る評価)は、自車の過去軌道だけでカンニング的に当ててしまえる(直進が多いデータ分布のため)という批判があります(AD-MLP / BEV-Planner などの分析)。このため「地図を入れた効果」を open-loop の L2 だけで測ると過小・過大評価しがちで、衝突率や閉ループ(closed-loop、シミュレータ上で実際に走らせる)評価との併用が研究者には推奨されます。地図の効果は「正解軌道に近いか」より「実際に安全に走れるか」で測るべき、という認識が広がっています。

指標の意味を補足すると、計画の L2 誤差が数十 cm 縮む、衝突率が数%下がる、というのは、自動運転の安全性に直結する大きさです。地図を正しい表現とエンコーダで入れることが、これらの数値を動かす主要因の一つになっています。

## メリット・トレードオフ・限界
- ナビ地図のメリット。安い・広域・更新が比較的容易。経路の意図(intent)を伝えるのに十分です。
- ナビ地図のトレードオフ。粗く、車線レベルの制御には足りません。位置合わせ誤差でズレます。
- HD 地図のメリット。精密で、車線レベルの計画・制御に使え、見えない先の車線も保証できます。
- HD 地図のトレードオフ。高価・範囲限定・古くなります。位置合わせと保守が必要です。
- 分けて処理するメリット。それぞれ得意な表現(ラスタ/ベクトル)で扱え、情報を失いません。BEV を共通範囲に揃えれば融合が幾何的に正しくなります。片方が無い状況でも動かせます。
- 分けて処理するトレードオフ。エンコーダが 2 系統になり、実装と計算が増えます。両地図の座標・範囲を厳密に揃える手間があります。
- 限界。地図と現実のズレ(位置合わせ誤差、地図の陳腐化)への頑健性は未解決課題です。特に HD 地図に強く依存する設計は、地図が間違っていると危険な挙動につながりかねず、地図無し・地図少でも安全に劣化する設計(graceful degradation)が研究上の焦点です。学習データに HD 地図が常にある状態で訓練すると、モデルが地図に過度に依存し、地図のない実環境で脆くなる(over-reliance)危険も指摘されています。

## 発展トピック・研究の最前線
最前線では「地図への依存度をどう設計するか」が中心テーマです。一方には、HD 地図に強く依存して高精度な計画を実現する流れがあり、もう一方には、HD 地図を捨ててオンライン地図構築 + SD prior で代替し、地図整備されていない地域でも走れるようにする流れがあります。実務的にはこの中間、すなわち「HD 地図があれば使い、無ければ SD prior とオンライン構築で劣化を抑える」というハイブリッド設計が現実解とされつつあります。また、ナビ地図と HD 地図を別々のトークン集合として大きな運転 Transformer に渡し、attention に「どちらをどれだけ信じるか」を学習させる統合的な設計も登場しています。地図を固定の入力ではなく、信頼度つきの可変な事前情報として扱う、という見方が深まっているのが現状です。さらに最近は、訓練時にランダムで地図入力をドロップする(map dropout)ことで地図依存を意図的に弱め、地図の有無に頑健なモデルを作る手法も提案されており、入力設計の問いは「どう入れるか」から「どれだけ依存させるか」へと深化しています。
