# 時間文脈の低次元ボトルネック問題

## ひとことで言うと
自動運転やロボットの認識モデルが「少し前まで何が見えていたか」を覚えておこうとするとき、過去のシーンを少数の数字(たとえば十数次元のベクトル)にまで圧縮して持ち越す設計をよく見かけます。ところが、混んだ交差点や広い駐車場のような豊かな視覚情報を十数個の数字に潰すのは無理があり、過去の情報の大半が失われます。この「時間方向の情報が細い管を通って大きく失われる」現象を、ここでは時間文脈の低次元ボトルネック問題と呼びます。結果として、モデルは「記憶しているつもりで、実はほとんど何も覚えていない」状態に陥ります。

## 直感的な理解
人間は運転中、目の前の一瞬だけでなく直近の数秒も覚えています。たとえば「左から割り込んできた車が、今は自分の死角に入って見えない」とき、その車がまだそこにいると推測できるのは過去を覚えているからです。歩行者が駐車車両の陰に隠れた瞬間も、消えたとは思わず「まだいるはず」と扱えます。横断歩道に立っていた人が一瞬トラックに隠れても、人は速度を緩めたままにします。これらはすべて「直前の数秒のシーンの中身」を覚えているからこそできる判断です。

機械にこの記憶を持たせるには、過去フレームの情報をなんらかの形で次のステップへ持ち越す必要があります。素朴な実装では「過去の各時刻を1本の小さなベクトルに要約してバッファに積む」ことをします。問題は、その要約をどれだけ細くするか、です。要約を細くしすぎると、覚えているつもりで実はほとんど何も覚えていない、という状態に陥ります。これがボトルネック問題の核心です。たとえるなら、1日の出来事を毎日たった1単語でメモするようなもので、後から読み返しても「楽しかった」程度しか分からず、誰と何をしたかは消えています。

## 基礎: 前提となる概念
まず押さえるべき用語を噛み砕きます。

特徴(feature)とは、生の画像ピクセルではなく、ニューラルネットワーク(画像から抽象的なパターンを抜き出す多層の関数)が中間で生み出す数値の塊です。たとえば1枚の画像をネットワークに通すと、各位置を256個の数字で表した「特徴マップ」が得られます。形で書くと `[256, H, W]`(チャネル256、縦H、横W)のようなテンソル(多次元配列)です。各位置の256個の数字は「この場所には車のような何かがあり、向きはこちらで、テクスチャはこう」といった意味を符号化しています。

時間文脈(temporal context)とは、過去のフレームから引き継いだ情報を、現在の判断に使えるようにしたものです。これを持たせる方法は大きく2系統あります。

- 特徴レベル(feature-level)で持つ: 過去フレームの特徴マップそのもの、あるいはそれに近い高次元の表現を保持する。情報を捨てずに持てるが、メモリと計算が重い。
- 集約レベル(summary-level)で持つ: 過去フレームを少数の数字に要約して持つ。軽いが、要約で多くを捨てる。

ロッシー(lossy)とは「情報を失う」という意味です。写真をJPEGで強く圧縮すると細部がぼやけるのと同じで、元に戻せない圧縮を指します。逆にロスレス(lossless)は完全に元へ戻せる圧縮です。十数次元への要約は極端にロッシーな圧縮で、いったん潰すと外界の構造は二度と復元できません。

ボトルネック(bottleneck)とは、瓶の首のように、情報の流れが急に細くなって大半が通れなくなる箇所のことです。情報理論でいうと、通せる情報量(チャネル容量)が極端に小さい区間で、いくら上流が豊かでも下流へはこの細さ以上の情報は流れません。

遮蔽(occlusion)とは、手前の物体に隠れて奥の物体が見えなくなる現象です。遮蔽された物体を「まだそこにいる」と扱うには、隠れる前の履歴を覚えている必要があります。これは認識の頑健性に直結し、遮蔽追従ができないモデルは、隠れた瞬間に物体を「消失」とみなして急ブレーキや見落としを起こします。

## 仕組みを詳しく
ボトルネックの大きさをテンソルの形で具体的に見ます。仮に過去の各時刻を14次元のベクトルに要約し、64ステップ分をバッファに積む設計を考えます。

- 1時刻あたりの過去情報: `[14]`(14個の数字)
- 64ステップ分: `[64, 14]` → flatten すると `[896]`(896個の数字)

一方、特徴レベルで持つ代表例として、鳥瞰図(BEV: Bird's-Eye-View、上空から見下ろした座標系の特徴マップ)を用いるモデルでは、1フレームのBEV特徴が `[256, 200, 200]` ≒ 約1024万要素になります。

両者を並べると差は歴然です。

```
特徴レベルの1フレーム:  256 x 200 x 200 = 10,240,000 個の数字
要約レベルの1フレーム:                       14 個の数字
                       ----------------------------------
圧縮率:                約 731,000 分の 1
```

73万分の1まで潰せば、どこに何があったかという空間構造はほぼ消えます。残せるのはせいぜい「自車が全体としてどう動いてきたか」程度です。情報理論的に言えば、14次元の実数ベクトルが運べる情報量は、混雑した交差点のシーン全体を符号化するのに必要な情報量よりはるかに小さく、原理的に外界の構造が入りません。

実際、こうした十数次元のベクトルに詰め込まれるのは、典型的には自車の状態量(ego state)です。速度、加速度、ヨーレート(yaw rate: 旋回の角速度、車がどれだけ速く向きを変えているか)、ハンドル角、直近の軌跡点の座標などです。これらは低次元で表せる量なので残せます。しかし「周囲に何台の車がどう並び、歩行者がどこにいたか」という外界のシーン構造は、低次元ベクトルには入りません。言い換えると、低次元バッファは「自分の運動の記憶」は持てても「世界の記憶」は持てません。

処理の流れを追うと、ボトルネックがどこで生じるかが見えます。

1. 各時刻、複数視点のカメラ画像を取り、backbone(特徴抽出ネットワーク)に通し、必要なら多視点を融合して豊かな特徴を作る。ここまでは情報が豊富(`[256, H, W]` クラス)。
2. ところがその時刻が過去になると、この豊かな特徴は破棄され、十数次元の集約量だけがバッファに積まれる。ここで大半が捨てられる。
3. 次の時刻の意思決定モジュールは「現在の豊かな特徴 + 過去の貧弱な要約」を見る。
4. 結果として、過去のシーンの中身はモデルにほぼ届かない。

before / after で言えば、理想(after)は「過去フレームの特徴マップ相当を時間融合に使う」、現実(before)は「十数次元の要約だけ」です。

さらに悪いことに、こうしたバッファは設計だけ存在して中身が実質ゼロ埋めのプレースホルダ(形だけ確保され値が使われていない箱)になっていることがあります。この場合、時間文脈は形式上は存在するのに、実際には何の情報も供給していません。バッファの存在が「時間文脈はある」という誤った安心感を生み、問題を見えにくくします。テストでは形が合っているので通ってしまい、性能が頭打ちになる原因として最後まで見落とされがちです。研究現場では、こうした「在るのに効いていない」要素を切り分けるために、その経路を意図的に切ってアブレーションし、性能が変わらないことを確認する手順が重要になります。

## 手法の系譜と主要論文
時間方向の情報損失をどう減らすかは、認識・予測研究の中心的なテーマであり続けてきました。

- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270) — 直前フレームのBEV特徴マップ(`200 x 200 x 256`)をまるごと保持し、temporal self-attention(時間方向の自己注意。現在のBEVクエリが過去のBEV特徴を参照して情報を引き出す仕組み)で融合します。低次元に潰さず特徴レベルで時間文脈を持つことの有効性を明確に示した代表作で、後続の多くのモデルの基準点になりました。

- BEVDet4D (Huang & Huang, 2022, arXiv:2203.17054) — 単フレームのBEVDetに、直前フレームのBEV特徴を自車運動で位置合わせして連結する「4D」拡張を加えました。たった1フレームの時間情報を足すだけで速度推定が劇的に改善することを示し、時間融合が速度という量に本質的だと印象づけた研究です。

- VideoBEV / Exploring Object-Centric Temporal Modeling (Han et al., ICCV 2023, arXiv:2303.05970) — 直前1フレームだけを再帰的に融合する方式では、時間が経つと情報がだんだん薄まって失われる(recurrent decay)問題を指摘し、長期の文脈を多くのフレームにわたって保持する設計を提案しました。これは「時間方向のボトルネックを別の形で防ぐ」試みです。

- SOLOFusion (Park et al., ICLR 2023, arXiv:2210.02443) — 長期(low-resolution・多フレーム)と短期(high-resolution・少フレーム)の時間情報を組み合わせ、効率と精度を両立させました。どのくらい過去を、どの粒度で持つべきかという、ボトルネック設計のトレードオフに正面から取り組んだ研究です。

- I-JEPA / V-JEPA (Assran et al., CVPR 2023, arXiv:2301.08243; Bardes et al., 2024, arXiv:2404.08471) — ピクセルではなく特徴(埋め込み)空間で隠れた部分や未来を予測する自己教師あり学習です。履歴を低次元の人手要約ではなく、学習された特徴空間で持つという方向性を支える理論的裏付けになっています。後発の世界モデル型の設計は、この「特徴空間で履歴を持つ」考えを採り、低次元ボトルネックを回避しようとします。

系譜としては、(1) 素朴な低次元要約 → (2) BEVFormer/BEVDet4Dによる特徴レベル時間融合の確立 → (3) VideoBEV/SOLOFusionによる長期保持と効率の両立 → (4) JEPA系による特徴空間での履歴・予測、という流れで「時間文脈をいかに損失なく、かつ現実的なコストで持つか」が深化してきました。

## 論文の実験結果(定量データ)
時間融合の効果は、nuScenes(自動運転の標準ベンチマーク。6視点カメラとLiDARを備えた大規模データセットで、約1000シーン・各20秒)で繰り返し定量化されています。主要な指標は2つです。

- mAP(mean Average Precision): 物体検出の正確さ。値が高いほど良い。0から1(または%)。
- NDS(nuScenes Detection Score): nuScenes独自の総合指標で、検出精度(mAP)に加え、位置誤差(mATE)・サイズ誤差(mASE)・向き誤差(mAOE)・速度誤差(mAVE)・属性誤差(mAAE)をまとめたもの。高いほど良い。

報告されている代表的な数値は次の通りです。

- BEVFormer は、時間融合を入れることで単フレーム版に対しNDSがおよそ数ポイント改善したと報告されています。特に速度推定(velocity)の誤差(mAVE)が大きく下がりました。これは直感的で、速度は1フレームでは原理的に測りにくく(止まっている車と動いている車は1枚の絵では区別しにくい)、過去フレームとの差分があって初めて精度よく出せるからです。低次元要約ではこの速度履歴が潰れてしまい、この改善が得られません。

- BEVDet4D は、直前1フレームを足すだけで速度誤差(mAVE)が単フレーム比でおよそ半分以下(報告では0.9超の水準から0.3〜0.4台へ)まで下がり、NDSが大きく伸びたと報告されています。「1フレームの時間情報」というごく薄い文脈でも、速度のように差分で決まる量には絶大な効果があることの直接的な証拠です。逆に言えば、その薄い文脈すら持たない低次元要約は、この最も基本的な改善を取りこぼします。

- SOLOFusion は、長期+短期の時間融合により、nuScenes test で当時のカメラのみ手法として最高水準(NDSがおよそ0.5台後半)を達成し、特に多フレームを使うほど検出と速度推定が継続的に改善することを示しました。アブレーション(ある要素を抜いて効果を測る実験)では、参照フレーム数を増やすほど性能が上がり、ある点で頭打ちになる傾向が報告されています。これは「過去をどれだけ持てば足りるか」の実証的な目安になります。

- 逆に、過去を低次元要約に潰す素朴な方式では、こうした速度・遮蔽に関わる改善が得られにくいことが、特徴レベル手法との比較から間接的に裏付けられます。要約に残るのは自車運動程度で、外界の動的構造が消えるためです。

数値の意味を補足すると、NDSやmAPで数ポイント(たとえば3〜5%)の差は、このベンチマークでは大きな差で、論文1本分の貢献に相当することが珍しくありません。特にmAVE(速度誤差)の改善は、下流の軌道予測や計画の安全性に直結するため、定量値以上に実務的な重みを持ちます。速度を取り違えれば、追従距離や合流判断が崩れ、衝突リスクに直結するからです。

## メリット・トレードオフ・限界
低次元ボトルネック方式にも、当初採用される理由がありました。

メリット(なぜ最初にこうなりがちか):
- 極めて軽い。過去情報が数百次元しかないので、メモリも計算もほとんど増えない。
- 実装が単純。可変長の履歴も固定長の小さなベクトルに丸め込める。
- 自車の運動履歴(速度・操舵など)だけが必要なタスクなら、これで十分なこともある。
- 低次元ゆえに過学習しにくく、小さなデータセットでも安定して学習が回る。

トレードオフ・限界:
- 過去シーンの視覚構造がほぼ失われる(極端にロッシー)。
- 遮蔽履歴、他車の運動履歴といった、運転判断に重要な外界情報が運べない。
- 速度推定や軌道予測など、時間差分が本質的なタスクで性能が頭打ちになる。
- バッファが形だけ存在しゼロ埋めになっていると、問題が表面化せず発見が遅れる。

未解決の課題として、「特徴レベルで持つと重いが、低次元だと足りない」という根本的な緊張関係があります。どの粒度で、どれだけの長さの履歴を、どのモジュールに持たせるのが最適かは、タスク・計算予算・対象シーンの動的さに依存し、汎用の答えはまだありません。研究の最前線は、この緊張を緩める表現(特徴空間での圧縮、選択的記憶、世界モデルへの隔離など)を探る方向にあります。

## 発展トピック・研究の最前線
近年は、時間文脈を「速い反応経路」から切り離し、別の遅い予測経路(世界モデル)に隔離する設計が注目されています。反射的な障害物回避は現在フレーム中心で軽く保ち、過去の記憶は未来予測のために遅い頻度(たとえば1Hz)で別系統に持たせる、という分業です。これにより、低次元に潰さず特徴レベルの履歴を持ちつつ、リアルタイム性も維持しようとします。

また、JEPA系の特徴空間予測は、履歴を人手の要約ではなく学習された潜在空間で保持する道を開きました。これは「何を覚えるべきか」をモデル自身に学ばせる方向で、固定の十数次元という設計上の上限を取り払う試みと言えます。表現の崩壊(collapse: すべてが同じベクトルに潰れる手抜き解)をどう防ぐかが、この系統の中心的な技術課題です。さらに、すべての過去を等しく持つのではなく、重要なフレームだけを選んで残す選択的記憶(メモリトークンやslot attention 的な仕組み)や、長い系列を効率よく畳み込む状態空間モデル(SSM、Mamba 系)を時間融合に使う研究も現れており、「容量の細い管を太くしつつコストを抑える」方向が活発です。
