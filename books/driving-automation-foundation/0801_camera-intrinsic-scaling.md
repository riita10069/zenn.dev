# 前処理（Resize/Crop）に合わせたカメラ内部行列のスケーリング

## ひとことで言うと

画像をニューラルネットに入れる前には、縮小（Resize）や中央切り抜き（CenterCrop）といった前処理をします。すると、そのカメラの「内部行列 K」（3 次元の点が画像のどこに写るかを決める 3×3 の数表）が、加工後の画像に対しては間違った値になってしまいます。このズレを、画像に施した幾何変形とまったく同じだけ K にも適用して直し、加工後の画像に正しく対応する K を作り直す処理です。これを怠ると、3 次元 → 画像の射影が静かに壊れ、BEV 融合などの幾何処理が破綻します。論文の新規手法ではなく多視点幾何の教科書的事実ですが、実装で最も間違えやすく、しかも間違えてもエラーが出ない「サイレントバグ」の温床です。

## 直感的な理解

カメラのレンズには「3 次元のこの点は、撮った写真のここに写る」という決まった対応関係があります。この対応を表す数表が内部行列 K です。K は「ある特定の画像サイズで測ったときの値」であることが肝心です。

たとえば 1920×1200 で撮った写真の K は「1920×1200 の画像で測った値」です。この写真を 256×256 に縮めると、同じ 3 次元の点が写るピクセル位置も当然変わります（画像が小さくなったぶん、すべての位置が比例して縮む）。なのに元の K（1920×1200 基準）をそのまま使うと、「縮める前の座標系」で計算してしまい、点がまったく違う場所に投影されます。

地図にたとえると、縮尺 1:25000 の地図に書かれた距離の目盛りを、コピー機で 50% 縮小した地図にそのまま当てると距離が 2 倍ずれるのと同じです。地図を縮小したら目盛りも縮小しなければならない。画像を加工したら K も同じだけ加工する、というのがこの処理の本質です。

厄介なのは、間違えてもエラーが出ない点です。プログラムは平然と動き、ただ投影結果が静かにずれるだけ。BEV 上で物体が数メートルずれて配置され、モデルの精度がじわじわ落ちる。損失も普通に下がるので「学習は回っている」ように見え、精度が出ない原因がここだと気づくのに何日も溶かす、というのが典型的な失敗例です。

## 基礎: 前提となる概念

内部行列 K（intrinsic matrix）: カメラ固有の、3 次元の点を画像ピクセルに対応づける 3×3 行列。

```
K = [[ fx,  0, cx],
     [  0, fy, cy],
     [  0,  0,  1]]

  fx, fy : 焦点距離をピクセル単位にした値（拡大率のようなもの）
  cx, cy : 主点（画像中心、光軸が画像に当たる点）のピクセル座標
```

ピンホールカメラモデルでは、カメラ座標系の 3 次元点 `(X, Y, Z)` は `u = fx * X/Z + cx`, `v = fy * Y/Z + cy` で画像座標 `(u, v)` に写ります。`X/Z` は「奥行きで割る」操作で、遠くのものほど画像中心に集まる遠近法（透視投影）を表します。fx, fy が大きいほど望遠（拡大）、cx, cy は画像のどこが光軸の中心かを表します。同次座標で書くと `[u·w, v·w, w]ᵀ = K [X, Y, Z]ᵀ`、最後に w で割って `(u, v)` を得ます。

外部行列（extrinsic）: カメラが世界のどこに、どの向きで置かれているか（回転 R と並進 t）。K（内部）と組み合わせて投影行列 `P = K [R | t]`（3×4 行列）を作ると、世界座標の 3 次元点を画像ピクセルに一気に写せます。K が「カメラ内部の見え方」、外部行列が「カメラの置き場所」を担当する、と分けて覚えると整理しやすいです。

前処理（transform）: backbone に入れる前に画像にかける加工の列。torchvision の Compose で「Resize → CenterCrop → ToTensor → Normalize」のように並べます。このうち Resize と CenterCrop はピクセルの座標を動かす（幾何変形）ので K に影響しますが、Normalize（色の正規化）や ColorJitter（色ゆらし）は座標を動かさないので K に影響しません。「座標を動かすか否か」が、K を直すか無視するかの判定基準です。

BEV 融合（Bird's-Eye-View fusion）: 複数カメラの情報を、上から見下ろした俯瞰図に統合する処理。各カメラのピクセルを、その K と外部行列を使って 3 次元・俯瞰平面に投影します。だから K が狂うと俯瞰図上の位置が狂い、複数カメラの像が一致しなくなります。

## 仕組みを詳しく

内部行列補正の処理は、「前処理パイプラインを 1 ステップずつ歩いて、画像の幅・高さを追跡しながら、Resize と CenterCrop の効果だけを K に反映する」ことです。Normalize や ToTensor は座標を動かさないので無視します。現在の画像サイズを `(cur_w, cur_h)` として追います。

ステップ A: Resize の場合。

短辺合わせ（shortest-edge）モード（多くのライブラリの標準）: 目標サイズ `size` を短い辺に合わせる倍率を計算し、fx, fy, cx, cy すべてに掛けます。

```python
scale = size / min(cur_h, cur_w)
K_out[0, 0] *= scale  # fx
K_out[1, 1] *= scale  # fy
K_out[0, 2] *= scale  # cx
K_out[1, 2] *= scale  # cy
# 画像サイズも更新: cur_w, cur_h を scale 倍
```

縦横を別倍率で指定する場合（`(h, w)` を直接指定）: 横倍率 `scale_x = w/cur_w`、縦倍率 `scale_y = h/cur_h` を別々に求め、fx と cx に scale_x、fy と cy に scale_y を掛けます。Resize はすべての座標を一様に拡縮するので、焦点距離も主点も同じ倍率で動く、と覚えれば理解しやすいです。なぜ cx, cy にも掛けるのか迷いやすいですが、主点も「ピクセル単位の位置」なので、画像全体が縮めば主点の座標も同じ比率で縮みます。

ステップ B: CenterCrop の場合。

切り抜きは画像の左と上を削るので、原点が右下にずれ、主点（cx, cy）の位置がその削った量だけずれます。

```python
offset_x = (cur_w - crop_w) / 2.0
offset_y = (cur_h - crop_h) / 2.0
K_out[0, 2] -= offset_x  # cx
K_out[1, 2] -= offset_y  # cy
# 画像サイズも更新: crop_w, crop_h
```

焦点距離 fx, fy は切り抜きでは変わりません。クロップは拡大率を変えず、画角の端を捨てるだけだからです。主点だけが動く、と覚えます。直感的には、画像の左上を原点にしているので、左を 77 px 削れば「これまで x=77 にあった点が新しい x=0 になる」、つまり主点の x 座標も 77 だけ小さくなります。

数値例（before/after）。元 1920×1200、fx=fy=1000、cx=960、cy=600。前処理が「短辺を 256 に Resize → 256×256 を CenterCrop」だとします。

- Resize: `scale = 256 / min(1200, 1920) = 256/1200 ≒ 0.2133`。
  - fx=fy ≒ 213.3、cx ≒ 204.8、cy = 128.0。サイズは横 `1920×0.2133 ≒ 410`、縦 `1200×0.2133 = 256`（短辺が 256 に）。
- CenterCrop 256×256: 横 410 なので `offset_x = (410-256)/2 = 77`。cx = 204.8 − 77 = 127.8。縦は既に 256 なので offset_y = 0、cy = 128.0 のまま。
- 結果: K_scaled ≒ fx=fy=213.3、cx ≒ 127.8、cy = 128.0（256×256 画像のほぼ中央）。

加工後の小さい画像に幾何的に正しく対応する K が得られました。検算として、元画像で主点 (960, 600) に写っていた光軸上の点は、補正後 K では (127.8, 128.0) に写るべきで、これは 256×256 のほぼ中央です。元画像でも主点は中央付近だったので、整合します。

なぜ「transform を自動で歩く」のか。backbone を変えると前処理の解像度や手順が変わります。前処理を backbone から動的に解決する設計（モデル設定から入力解像度・正規化統計を取り出す）にしておけば、K の補正もその transform を読んで自動追従させられます。backbone を差し替えても、手で数字を直す必要がありません。逆に手で K を書き換える設計だと、前処理を変えるたびに人間が同期させねばならず、ミスの温床になります。一般化すると、各幾何変形を「2×3 のアフィン行列」で表し、それらを順に左から K に掛ける（合成する）と、Resize・Crop・Flip・回転を統一的に扱えます。`K' = A · K` の形です。

## 手法の系譜と主要論文

この補正は「論文の新規手法」というより、3 次元ビジョンの基礎（教科書的事実）であり、近年の BEV 認識手法では「データ拡張と内部行列の同期更新」として実装上の必須事項になっています。系譜をたどります。

Hartley & Zisserman, "Multiple View Geometry in Computer Vision"（第 2 版、2004、Cambridge University Press）。多視点幾何の定番教科書。内部行列 K と射影の基礎を与え、「画像をスケールすると K の対応成分が同じ倍率でスケールする」「クロップは主点をオフセットする」ことが幾何として説明されます。本ノートの補正はこの教科書事実の素直な実装で、画像変換を表すアフィン行列を K に左から掛ける、という一般形が示されています。

Lift-Splat-Shoot（LSS、Philion & Fidler、NVIDIA、2020、ECCV、arXiv:2008.05711）。複数カメラ画像を BEV に「持ち上げる（lift）」代表手法。各ピクセルに深度分布を推定し、K と外部行列で 3 次元に逆投影してから俯瞰平面に「ばらまく（splat）」。入力画像をネットワーク用にリサイズ・クロップするので、その変形に合わせて K を更新しないと BEV 上の位置がずれます。LSS の実装はこの同期更新を明示的に行っています。深度を伴うので、K の誤りは「どの方向に逆投影するか」を直接狂わせ、影響が大きいのが特徴です。

BEVDet（Huang ら、2021、arXiv:2112.11790）。LSS 系を物体検出に発展させた手法で、「画像空間のデータ拡張（リサイズ・クロップ・回転・反転）」と「BEV 空間のデータ拡張」を両方使います。ここで重要な指摘が、画像拡張を施したぶんだけカメラ行列（K や、回転を含む変換）を同期更新しないと、拡張が逆に性能を下げる、という点です。BEVDet 論文は、この同期更新を正しく行うことで画像拡張が初めて有効になると報告しました。BEVDet は画像拡張を 3×3 の変換行列としてまとめ、視点変換モジュールにその行列を渡して逆補正する実装になっています。

BEVFormer（Li ら、2022、ECCV、arXiv:2203.17270）。BEV クエリが各カメラに射影して特徴を集める Transformer ベース手法。BEV 平面上の各点を、各カメラの投影行列（K と外部行列の合成）で画像に射影し、その位置の特徴を spatial cross-attention で集めます。ここでもカメラの投影行列が正確であることが前提で、前処理に伴う K のズレは射影点の誤りに直結し、間違った画素から特徴を引いてきてしまいます。

BEVerse（Zhang ら、2022、arXiv:2205.09743）など後続のマルチタスク BEV 手法も、画像拡張とカメラパラメータの同期を標準装備しています。これらに共通するのは「画像側の加工と内部行列を必ず同期させる」という実務原則です。本ノートの補正処理は、まさにこの実務を担います。

## 論文の実験結果（定量データ）

この補正単体の論文はありませんが、「画像拡張 + 行列同期更新」の効果は BEV 検出論文で定量化されています。指標は nuScenes の mAP（mean Average Precision、検出の正確さ、0〜1、高いほど良い）と NDS（nuScenes Detection Score、mAP に位置・速度・属性誤差を統合した総合指標、高いほど良い）です。

BEVDet は、画像空間のデータ拡張を導入し、それに合わせてカメラ行列を同期更新することで、nuScenes 検出の mAP と NDS が明確に改善すると報告しました。論文のアブレーションでは、画像拡張と BEV 拡張を組み合わせると過学習が抑えられ、mAP が数ポイント単位で向上しています（報告では画像 + BEV 拡張の併用で mAP が数 % 改善、NDS も同様に向上）。ここで肝心なのは、行列同期を「正しく」やった場合に拡張が効く、という点です。同期を誤ると、拡張で画像をいじっても幾何ラベル（BEV 上の正解位置）とズレるため、学習が逆に悪化します。つまり「拡張が効かない」という症状の裏に「K 同期ミス」が隠れていることがあり、デバッグの勘所になります。

LSS は BEV セマンティックセグメンテーション（俯瞰図上で「ここは道路」「ここは車」と塗り分けるタスク）で、単眼ベースラインを上回る IoU（Intersection over Union、予測領域と正解領域の重なり比、0〜1、高いほど良い）を達成しました。複数カメラを正しい K・外部行列で統合できることが前提で、入力解像度を変えた実験では、内部行列をそれに合わせて更新することで幾何整合が保たれることが示されています。

定量化しにくいが重要なのが「失敗のサイレントさ」です。補正を忘れた場合、訓練は普通に進み損失も下がるように見えますが、BEV 上の物体位置が一定方向にずれ、検出 mAP が静かに数ポイント低下します。エラーログには何も出ません。だから「投影が正しいか」を確認する単体テスト（既知の 3 次元点を投影して期待ピクセルと一致するか、複数カメラの重なり領域で同一物体が一致するか）を別途持つことが、研究上の定石になっています。可視化（点群を画像に重ねて描く、BEV に複数カメラを投影して整合を目視する）も、サイレントバグを早期に捕まえる強力な手段です。

## メリット・トレードオフ・限界

メリット。加工後の画像に対して幾何的に正しい K が得られ、3 次元 → 画像の射影が壊れません。backbone 駆動の前処理に自動追従するので、backbone を変えても手作業の修正が不要です。Normalize 等の非幾何ステップを正しく無視するので、必要な操作だけを反映できます。アフィン行列で表せば、データ拡張（毎サンプル違うクロップ・スケール）にも一貫して同期できます。

トレードオフと限界。第一に、対応しているのは Resize と CenterCrop のみ、という典型実装が多いこと。回転（rotate）・ランダムクロップ・アスペクト比変更・水平反転（flip）といった拡張を加えるなら、その幾何効果も K（や外部行列）に反映するよう拡張せねばなりません。たとえば水平反転は cx を `W − cx` に直す必要があり（左右が反転するので主点も鏡映する）、回転は K だけでなく座標系の回転として扱う必要があります。これを忘れると、まさに「拡張が逆効果」になります。第二に、「短辺合わせ Resize」のサイズ追跡には丸め（整数化、`floor` か `round` かはライブラリ依存）があり、ライブラリ実装の丸めと厳密一致しないわずかな誤差が乗り得ます（通常は無視できる範囲ですが、サブピクセル精度が要る場面では注意）。第三に、本処理はピンホール（透視）モデルを前提とするため、魚眼・広角の強い歪み（レンズ歪み係数）には対応しません。歪み補正（undistort）は K のスケーリングとは別の処理で、先に歪みを除いてから K をスケールする順序が要ります。第四に、前述の通り間違えてもエラーが出ず静かに精度が落ちるため、射影の正しさを別途テストする必要があります。

## 発展トピック・研究の最前線

最前線では、データ拡張と幾何整合をより一般的に扱う枠組みが整理されています。BEV 系の多くの実装は、画像拡張を「3×3（または 2×3 を拡張した）のアフィン変換行列」として表し、それを内部行列に左から掛ける形で一括同期します。これにより Resize・Crop・Flip・回転を統一的に扱え、BEVDet 系では拡張行列を視点変換に明示的に渡します。また、深度推定を伴う手法（LSS 系）では、内部行列の誤りが深度の逆投影誤差に直結するため、キャリブレーションの不確かさをモデルに取り込む研究（確率的な深度・キャリブレーション補正、カメラパラメータをネットワークに条件付けする camera-aware な設計）も進んでいます。

さらに、自己教師あり深度推定（monodepth 系）や NeRF・3D Gaussian Splatting 系の 3 次元再構成では、内部行列自体を学習で微調整する（self-calibration、bundle adjustment 的に K を最適化する）方向もあり、「固定の K を補正する」段階から「K を推定・最適化する」段階へと研究が広がっています。近年のエンドツーエンド運転や占有予測（occupancy prediction）でも、カメラパラメータをトークンとしてモデルに与え、異なるセンサ構成（車種ごとにカメラ配置が違う）に汎化させる試みが現れており、「正しい K を渡す」ことの重要性はむしろ増しています。学習の出発点としては、まず既知の 3 次元点を手で投影して補正前後の差を確認し、可視化で整合を体感するのが、幾何の感覚を養う近道です。
