# ランタイム形状検証（ValueError による早期失敗）

## ひとことで言うと

ニューラルネットワークのある部品（モジュール）に、想定と違う形（次元）のテンソルが渡されたとき、何層も先の深いところで意味不明なエラーが出るのを待つのではなく、入口で「この形はおかしい」と気づいて即座にエラー（Python では `ValueError`）を投げて止める、という堅牢化のテクニックです。エラーメッセージに「期待した形」と「実際に来た形」の両方を書いておくことで、原因がひと目で分かるようになります。深層学習のコードは計算グラフが深く、バグの発生箇所と原因箇所が離れがちなので、この「入口で守る」工夫がデバッグ時間を劇的に縮めます。

## 直感的な理解

工場の組み立てラインを想像してください。あるロボットアームは「直径 5cm のボルト」を受け取って締める設計になっています。ここに間違って直径 3cm のボルトが流れてきたとします。

- 何もチェックしないと、アームは無理やり締めようとし、3 工程くらい先で製品が歪んで初めて「何かおかしい」と気づきます。しかもそのときには、どの工程で間違ったボルトが混入したのか分からなくなっています。
- もしアームの手前に「直径を測って 5cm でなければ即停止し『5cm を期待したが 3cm が来た』と表示する」検査ゲートがあれば、混入した瞬間に、その場所で、原因つきで止まります。

ニューラルネットワークの形状検証はこの検査ゲートそのものです。「直径」がテンソルの形（shape）に、「即停止」が `ValueError` に対応します。さらに深層学習には、製造ラインにはない厄介な性質があります。間違った形のテンソルでも、たまたま計算が成立してしまい、エラーすら出ずに「静かに間違った結果」を出し続けることがあるのです。これが最も発見しにくいバグであり、形状検証が特に効く理由です。

## 基礎: 前提となる概念

いくつかの用語を、知らない人向けに丁寧に噛み砕きます。

- テンソル（tensor）: これは「多次元の数の配列」という意味です。1 次元なら数のリスト（ベクトル）、2 次元なら表（行列）、3 次元以上は箱が積み重なったイメージ。深層学習のデータはほぼすべてテンソルで表されます。
- 形状（shape）: テンソルの各軸（次元）の大きさを並べたもの。例えば `[4, 256, 8, 8]` は「バッチ 4・チャンネル 256・高さ 8・幅 8」を意味します。慣習として `B`=バッチサイズ（同時に処理するサンプル数）、`C`=チャンネル数、`H`=高さ、`W`=幅、`D` や `L`=特徴次元や系列長を表します。
- 次元不一致（dimension mismatch）: あるモジュールが「最後の次元は 256 のはず」という前提で作られているのに、実際には 1440 のテンソルが来る、といった食い違い。
- ブロードキャスト（broadcasting）: 形の違うテンソル同士の演算を、片方を自動で「引き伸ばして」形を合わせる仕組み。便利な反面、本来エラーにすべき食い違いを「合わせられてしまう」ことがあり、静かな誤計算の温床になります。例えば `[B, 256]` と `[B, 1]` を足すと、後者が 256 列に複製されて足されます。これが意図通りなら良いのですが、`[B, 256]` と `[256]` のような偶然合うケースで意図しない結合が起きると、エラーにならないまま間違った値が伝播します。
- 例外（exception）と `ValueError`: Python では異常が起きたときに「例外」を投げてプログラムを止めます。`ValueError` はその一種で「値そのものがおかしい」ときに使う標準的なもの。`raise ValueError("メッセージ")` と書くと、メッセージつきでその場で停止します。

なぜ PyTorch のような動的フレームワークでこれが特に必要かを補足します。PyTorch は「define-by-run」（実行しながら計算グラフを組む）方式で、形のチェックを事前に強制しません。柔軟で書きやすい反面、形が少しずれていても「とりあえず計算してしまう」ことがあります。静的に形を検査するコンパイル型のシステムと違い、形の前提は開発者が明示的に守らないと誰も守ってくれない、というのが背景です。

## 仕組みを詳しく

中核となる考え方は「契約（contract）を入口でアサート（断言・検査）する」ことです。モジュールが受け取るべきテンソルの形を、計算を始める前に確認し、違えば説明つきで `ValueError` を投げます。典型的なパターンは次の通りです。

```python
def forward(self, x):
    # x は (B, expected_dim) の 2 次元テンソルであるべき、という契約
    if x.dim() != 2 or x.shape[-1] != self.expected_dim:
        raise ValueError(
            f"expected input of shape (B, {self.expected_dim}), "
            f"got {tuple(x.shape)}."
        )
    ...
```

ここで決定的に重要なのは、エラーメッセージに「期待」と「実際」の両方を入れることです。`got (4, 1440)` のように実際の形が見えれば、「ああ、256 に射影し忘れたな」と一瞬で原因が特定できます。メッセージが「shape error」だけだと、結局どこをどう間違えたのか調べ直す羽目になり、検証の価値が半減します。

検証できる対象は形状だけではありません。同じ「入口で早期失敗」の思想は、次のような幅広いチェックに展開できます。

- 構造の検証: バックボーン（特徴抽出器）の返り値がリスト/タプルでない、空である、各特徴マップが想定の次元数（例: 4 次元 `[B,C,H,W]`）でない、といったケースを弾く。
- パラメータ値の検証: 設定値が数値でない、`inf`（無限大）や `nan`（非数。0/0 のような不定値の結果）になっている、負であってはいけない値が負である、といったケースを弾く。形ではなく値の妥当性のチェックですが、思想は同じです。
- 整合性の制約: 例えば曲線をパラメトリックに表現するプランナ（軌道生成器）では「制御点の数が 2 未満では曲線を定義できない」「制御点数が時刻数を超えてはならない」といったドメイン固有の前提を `ValueError` で弾く。

具体例で before/after を示します。あるモジュールが 256 次元の隠れ状態を受け取る設計のところに、リファクタリング漏れで 14 次元のテンソルが渡ったとします。

```
before（検証なし）:
  入力 14 次元 → 数層先の nn.Linear で
  RuntimeError: mat1 and mat2 shapes cannot be multiplied (4x14 and 256x512)
  → どの「入力」が悪いのか、スタックトレースを遡らないと分からない

after（入口で検証）:
  forward 冒頭で
  ValueError: expected last dim 256, got (4, 14)
  → 原因が一目瞭然。射影層を入れ忘れたと即断できる
```

なぜこの設計が有効かを一段深く言うと、「バグの検出点を原因点に近づける」からです。深層学習の計算は何十層も連なるため、検出点（エラーが顕在化する場所）と原因点（実際に間違いが入り込んだ場所）が遠いと、その間を遡る作業に時間が溶けます。入口検証は両者を一致させます。

## 手法の系譜と主要論文

これは機械学習の研究テーマというより、ソフトウェア工学の古典的な堅牢化（defensive programming, 防御的プログラミング）に属します。引用すべきは ML 論文ではなく、ソフトウェア工学の系譜です。

- 契約による設計（Design by Contract）: Bertrand Meyer が著書「Object-Oriented Software Construction」（1988、第2版 1997）と Eiffel 言語で体系化した考え方。関数・メソッドは「事前条件（precondition, 入力が満たすべき条件）」「事後条件（postcondition, 出力が満たすべき条件）」「不変条件（invariant）」を持つべきで、事前条件が満たされなければ即座に失敗すべき、というもの。形状検証はまさに事前条件のアサートです。
- フェイルファスト（fail-fast）原則: Jim Shore らが提唱・普及させた経験則で、「異常はできるだけ早く、原因に近い場所で検出して止めるほうが、放置して伝播させるよりデバッグコストが小さい」というもの。Andrew Hunt と David Thomas の「The Pragmatic Programmer」（1999）でも "crash early"（早く落とせ）として強く推奨されています。
- アサーション文化: Steve McConnell の「Code Complete」（1993、第2版 2004）が、開発時に前提を assert で守る習慣を体系的に論じています。

機械学習特有の文脈としては、近年「shape typing」（形状を型として扱う）という研究的試みも出ています。代表的なものに Python の型注釈で軸名つきの形を表現するライブラリ群（例: jaxtyping, torchtyping、いずれも 2021 年前後）があり、`Float[Tensor, "batch 256"]` のように静的・動的に形を検査します。これは「実行時に手書きで if 文を書く」アプローチを、型システムに昇格させて自動化しようという発展形と位置づけられます。

## 論文の実験結果（定量データ）

形状検証はアルゴリズムではなく工学的プラクティスのため、ベンチマークデータセット上の精度向上のような「定量実験」は存在しません。代わりに、ソフトウェア工学研究で測られている「早期検出の効果」に関する経験的知見を、意味とともに紹介します。

- バグの修正コストは、混入から発見までの距離（時間・工程）に対しておおむね指数的に増大する、という観察が古くから報告されています（Barry Boehm のソフトウェア工学経済学、1981 など）。設計・実装段階で見つけたバグの修正コストを 1 とすると、テスト段階では数倍、運用後では数十倍になる、という桁感です。形状検証は「実装段階で混入したバグをその場で顕在化させる」ので、この曲線の最も安いところで捕まえます。
- 静かな誤計算（silent failure）の危険性は、機械学習で特に深刻です。学習が「動いてはいるが精度が伸びない」とき、その原因が形のずれによるブロードキャストの誤結合なのか、ハイパーパラメータなのか、データなのかを切り分けるのは非常に困難で、数日を溶かす例が珍しくありません。形状検証は、この切り分けに費やす時間をゼロに近づける（そもそも誤結合が起きた瞬間に止まる）という意味で、定量化しにくいが大きな効果を持ちます。

アブレーション的に言えば、「検証を抜くと何が起きるか」は明確です。検証なしのコードでリファクタリング（例: 特徴次元を 1440 から 256 へ統一する大規模変更）を行うと、更新し忘れた一部のモジュールに古い次元が流れ込み、運が悪ければエラーすら出ずに学習が静かに劣化します。検証ありなら、その瞬間に原因つきで停止します。

## メリット・トレードオフ・限界

メリット

- バグの発生箇所と検出箇所が一致するため、デバッグ時間が大幅に短縮される。
- ブロードキャストによる「静かな誤計算」を防げる。これは最も発見しにくい種類のバグ。
- 自己文書化になる。検証コードを読めば「このモジュールは何を期待しているか」がコメントを読まずとも分かる。
- 自動テストと組み合わせれば「不正入力を確かに弾けること」を回帰（一度直したものが再び壊れること）から守れる。不正な形を渡すと意図通り `ValueError` が出ることをテストで保証できる。

トレードオフ・限界

- 検証コードのぶんだけコード量が増える。可読性とのバランスが必要で、書きすぎると本筋の計算が埋もれる。
- `forward` のたびに数回の比較が走るが、テンソル演算（行列積など）のコストに比べれば無視できるオーバーヘッドである。
- 検証は「想定したケース」しか守れない。検証条件自体が間違っていれば守れないので、テストとセットで初めて意味を持つ。
- 形状が動的に変わる正当なケース（可変視点数、可変系列長など）では、検証条件を緩く書く必要がある。緩くしすぎると本来弾くべき入力を通し、厳しくしすぎると正当な入力を弾く。この線引きがドメイン知識を要する。
- 研究上の未解決課題として、「どこまでを実行時 if 文で守り、どこからを型システム（shape typing）に任せるか」のバランスは確立されていない。型注釈は静的に守れて美しいが、動的に形が決まるケース（実行時のバッチサイズ依存など）を完全には表現しきれず、結局実行時チェックと併用する場面が残る。

## 発展トピック・研究の最前線

- shape typing / named tensors: テンソルの各軸に名前をつけ、形を型レベルで検査しようという流れ。PyTorch の named tensors（実験的機能）、jaxtyping、torchtyping などが該当。手書きの if 検証を、より宣言的で再利用可能な形に置き換える方向。
- 契約の自動生成と検証: 関数の実行トレースから事前条件を推論して提案するツール（例: property-based testing と組み合わせた契約推論）の研究。Hypothesis のようなプロパティベーステストと組み合わせると、「ありうる入力空間」を自動探索して検証条件の穴を突いてくれる。
- 数値安定性の検証への拡張: 形だけでなく、`nan`/`inf` の伝播を検知する仕組み（PyTorch の `torch.autograd.detect_anomaly()` など）。これは「値の契約」を勾配計算にまで広げたもの。学習が突然発散する原因を、発生した演算の場所まで遡って特定できる。
