# オンライン vs オフライン地図

## ひとことで言うと
自動運転で使う地図を「事前に作っておいて読み込むか(オフライン地図)」「走りながらセンサからその場で作るか(オンライン地図)」という、設計の大きな分かれ道です。どちらを選ぶかで、システムの作り・コスト・対応できる地域・故障の仕方がまるごと変わります。E2E(end-to-end、入力から運転操作まで一気通貫)自動運転では特に重要な論点になります。

## 直感的な理解
ドライブに出るとき、二つのやり方があります。ひとつは「事前に詳細な地図帳を用意し、それを見ながら走る」。地図帳は精密で、曲がり角の奥や見えない先まで載っていますが、誰かが街を測量して作り、変化があれば更新し続けなければなりません。もうひとつは「地図帳を持たず、目の前の景色だけを見て、その場で道を読み取りながら走る」。どこへでも行けますが、見える範囲しか分からず、霧や障害物に弱い。

自動運転の地図もこの二択に対応します。前者がオフライン地図(事前構築 HD地図)、後者がオンライン地図(走行時生成)です。両者は単なる実装の違いではなく、「世界をどこまで事前に知っておくか」という哲学の違いであり、それぞれに固有の長所と弱点があります。

## 基礎: 前提となる概念

- HD地図(High-Definition map、高精細地図): 車線・停止線・信号を数センチ精度で記録した地図。オフライン地図の典型。
- SD地図(Standard-Definition map): 道路レベルの粗い地図。カーナビや OpenStreetMap がこれにあたり、車線単位の精度はないが「道のつながり」は分かります。
- ベクタ地図 / ラスタ地図: ベクタは車線を点列や曲線、つながり関係として記述する形式(数式的、軽い)。ラスタは地図を画像(ピクセルの格子)として描く形式(扱いやすいが重い)。
- BEV(Bird's-Eye-View): 真上から見た視点。カメラの斜め視点をこの平面図に変換して地図を扱うのが定番です。
- ローカリゼーション: 自車を地図上の正しい位置に合わせる処理([HD地図ローカリゼーション事前情報](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0440_hd-map-localization-prior))。オフライン地図には必須です。
- 集合予測(set prediction): これは「画像から N 個の地図要素を、順不同の集合として一度に出す」予測の形という意味です。DETR 型 Transformer がこれを得意とします([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))。

## 仕組みを詳しく

### オフライン地図の流れ

1. 測量車両(高精度 LiDAR・GNSS・IMU 搭載)で対象エリアを走り、点群と画像を集める。
2. オフラインで時間をかけ、車線の中心線・境界・停止線・信号位置・交通規則などを高精度に作図する。出力は Lanelet2 のようなベクタ地図フォーマット(車線を多角形や折れ線で、つながり関係込みで記述)。半自動化は進んでいるが、最終的な品質確認に人手が残るのが普通。
3. 走行時はこの地図を読み込み、ローカリゼーションで自車を地図上に正確に合わせ、自車周辺を切り出してモデルへ渡す。

ポイントは「重い作業をすべて走行前に済ませ、走行時は読み込みと照合だけ」という設計です。

### オンライン地図の流れ

1. 走行中、複数カメラ(と必要なら LiDAR)の映像をモデルに入力する。
2. モデルが画像を BEV 空間の特徴に変換する(視点変換。LSS のような深度持ち上げ方式や、BEVFormer のようなアテンション方式が使われる)。
3. その BEV 特徴から、車線・道路境界・横断歩道などの地図要素を、その場で集合予測する。ここで [DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr) の DETR 型クエリデコーダがよく使われ、出力は点列や [ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation) の曲線になる。
4. できた地図(自車中心)をそのまま運転判断に使う。事前地図もローカリゼーションも不要。

ポイントは「すべてを走行時のリアルタイム推論で済ませる」という設計です。

### 比較表

```
              オフライン地図                   オンライン地図
作成タイミング  走行前に専用車で測量              走行中にセンサからその場で生成
精度          非常に高い（数cm）               センサ性能と推定精度しだい（数十cm〜）
維持コスト     高い（常に作り直し・更新が必要）     不要（地図を持たない）
対応地域       測量済みエリアのみ                どこでも（未知の道でも）
工事・変化     地図が古いと危険                  その場で見るので強い
遠方の見通し    地図にあるので曲がり角の奥も分かる   センサの届く範囲・遮蔽に弱い
計算負荷       読み込みは軽い（照合は別途）         推定をリアルタイムで回すので重い
時間的一貫性    安定（同じ地図を読む）             フレーム間でちらつきやすい
```

この表が示すのは、両者がほぼ正反対の特性だということです。オフラインは「事前コストを払って走行時を楽にし、精度と先読みを得る」。オンラインは「事前コストをゼロにする代わりに走行時に重い推論を背負い、見える範囲に限られる」。実用システムは、この両極のどこに位置取りするかを選びます。

## 手法の系譜と主要論文

HDMapNet (Li, Ge, Liu, et al.; Tsinghua / MIT, ICRA 2022, "HDMapNet: An Online HD Map Construction and Evaluation Framework", arXiv:2107.06307) が、オンライン地図構築を体系化した初期の代表例です。複数カメラ(と LiDAR)から BEV 空間の意味地図(車線・境界・横断歩道)をセマンティックセグメンテーションで出し、後処理でベクトル化しました。動機は、HD地図の事前構築と維持があまりに高コストでスケールしないこと。新規性は、オンライン地図構築を「タスク」として定式化し、評価指標(後述の mAP)まで含めて整備した点です。トレードオフは、セグメンテーション後にベクトル化する段が継ぎ目になり、精度が頭打ちになることでした。

VectorMapNet (Liu, Wang, Zhang, et al.; ICML 2023, "VectorMapNet: End-to-end Vectorized HD Map Learning") は、後処理を介さず地図を直接ベクトル(折れ線の系列)として生成する初期の end-to-end 手法で、HDMapNet と MapTR の橋渡しにあたります。

MapTR / MapTRv2 (Liao et al., Horizon Robotics / 華中科技大学, ICLR 2023, arXiv:2208.14437) は、その後処理をなくし、DETR 型クエリで地図要素を直接ベクトル予測しました([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))。順序付き点集合の置換等価な対応づけ(permutation-equivalent matching、点列の始点・並び順の任意性を吸収する工夫)で、オンライン地図構築の精度と速度を大きく前進させました。これが「オンライン地図でも実用品質に近づいた」という流れを決定づけました。

オフライン側の代表は Lanelet2 (Poggenhans, Pauls, Janosovits, et al.; KIT, ITSC 2018, "Lanelet2: A High-Definition Map Framework for the Future of Automated Driving") です。ベクタ HD地図のフォーマット兼ライブラリで、車線・規制・つながりを構造的に記述し、ルーティングや交通規則の解釈まで扱えます。多くのオープンな自動運転スタックで採用されています。トレードオフは、当然ながら事前にこの地図を作って維持しなければならない点です。

E2E 自動運転の文脈では、UniAD (Hu et al., Shanghai AI Lab, CVPR 2023 best paper) や VAD (Jiang et al., Horizon Robotics, ICCV 2023) が、オンラインで地図(と物体・占有・軌跡)を推定して運転計画まで一気通貫でつなぎ、「オフライン HD地図への依存を減らす(map-light)」方向を後押ししました([マップフリー運転](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0429_map-free-driving))。

## 論文の実験結果(定量データ)

オンライン地図構築は nuScenes(ボストンとシンガポールの多センサ走行データセット)で評価されるのが標準です。指標は mAP(mean Average Precision)。これは予測した地図要素(車線=divider、境界=boundary、横断歩道=ped crossing)が正解とどれだけ重なるかを、チャンファー距離(2本の線の点同士の最近傍距離を双方向に測った平均)のしきい値(0.5m, 1.0m, 1.5m)で判定し、適合率を平均したものです。0〜100% で、高いほど良い指標です。なぜチャンファー距離かというと、車線のような「線」の予測精度は、点単位の一致ではなく「線同士がどれだけ近いか」で測るのが自然だからです。

代表的な数値の流れ:

- HDMapNet: nuScenes でおよそ mAP 20〜30% 台。オンライン地図が「できる」ことを示したが、後処理ベクトル化が精度を制約した。
- VectorMapNet: end-to-end 化で mAP 40% 前後へ。後処理を外す効果を示した。
- MapTR: 置換等価マッチングと並列クエリで mAP 50% 前後へ。リアルタイムに近い推論速度も達成。
- MapTRv2: 補助タスク(深度推定や BEV と PV の整合)の追加で条件により mAP 60% 台を報告。

この「20%台 → 50〜60%台」という倍以上の改善は、「後処理を挟む方式」から「クエリで直接ベクトルを出す方式」への移行が効いたことを示します。アブレーションでは、点集合の対応づけ方(順序を固定するか、置換不変にするか)や、BEV への視点変換方式(深度持ち上げかアテンションか)が精度に大きく効くこと、補助の深度監督を外すと数 mAP 落ちることなどが報告されています。

オフライン地図側は「精度」というより「カバレッジと維持コスト」で評価される性質のもので、mAP のような単一指標では測りません。むしろ E2E 全体の運転性能で間接的に評価されます。UniAD は nuScenes の計画タスクで、軌跡の予測誤差(L2 distance、予測軌跡と実軌跡の平均距離)を 1〜3 秒先で平均およそ 1.0m 前後、衝突率(collision rate)を 0.3% 程度まで下げたと報告し、「オンライン地図を含む統合学習」が計画性能に効くことを定量的に示しました。

## メリット・トレードオフ・限界

- オフライン地図メリット: 数センチ精度で、見通しの悪い先や遮蔽下でも地図情報が使える。交通規則(右折禁止等)も地図に書ける。先読みが効く。
- オフライン地図トレードオフ: 測量・作図・維持のコストが巨大。地図のない場所では走れない。工事や変化で地図が古くなると、むしろ危険な誤情報になる。高精度ローカリゼーションが別途必要。
- オンライン地図メリット: 事前地図・ローカリゼーション・地図維持が不要。未知の道や変化に強く、スケールしやすい。
- オンライン地図トレードオフ: 精度がセンサと推定モデルしだいで、遠方や遮蔽、悪天候に弱い。リアルタイム推論が計算的に重い。フレーム間で地図がちらつく(時間的一貫性の欠如)問題もある。
- 限界・未解決課題: 両者をどう組み合わせるか(オフライン地図を事前分布、オンライン推定を観測として融合する)が活発な研究テーマ。また、オンライン地図の「正解」をどう大量に用意するか(HD地図がない地域では教師データも作りにくい)という鶏と卵の問題があります。さらに、mAP は「地図がどれだけ正確か」を測るが「その地図で安全に運転できるか」は測らないため、地図精度と運転性能のギャップも未解決です。

## 発展トピック・研究の最前線

近年は中間的な立場として、世界中で既に整備済みの SD地図(OpenStreetMap など)を弱い事前情報として与え、オンライン地図構築を補助する研究が増えています(SD-map-aided online mapping)。SMERF や P-MapNet などが代表例で、SD地図の道路つながりを Transformer のトークンや BEV の追加チャンネルとして与えることで、遠方や遮蔽でオンライン推定が苦手な部分を補い、mAP を数ポイント押し上げると報告されています。SD地図は精度こそ粗いが「道のつながり」を無料で与えるため、コスト対効果が高い事前情報です。

さらに、地図を陽に出さず運転計画に必要な潜在特徴だけを持つ end-to-end 手法([マップフリー運転](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0429_map-free-driving))、フレーム間で地図を時間的に蓄積・更新して時間的一貫性を高める手法、地図の不確実性を陽に推定して計画に渡す手法、複数走行のデータをクラウドで束ねて地図を自動構築・更新するクラウドソース地図などが進んでいます。「事前にどこまで知り、走行時にどこまで作るか」の最適な配分は、コスト・安全・スケールのバランスで決まる、分野の中心的な問いです。
