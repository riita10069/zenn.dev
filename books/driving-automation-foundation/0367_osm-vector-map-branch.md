# OSM ベクトル地図ブランチ

## ひとことで言うと

自動運転モデルに「ここは道路です」「この先に交差点があります」といった地図情報を渡したいとき、地図を画像(絵)に描いてから渡す方法と、地図を「点と線と意味タグの列」というデータの構造のまま渡す方法があります。OSM ベクトル地図ブランチとは後者の方式で、OpenStreetMap(OSM、世界中の地図を誰でも編集できる公開地図)の地図要素を、ピクセル画像にせずに「トークン(モデルが処理する最小単位の数値ベクトル)の列」としてニューラルネットへ供給する仕組みです。ここでの「ブランチ」はモデル内部の処理経路の枝のことで、地図情報だけを担当する専用の枝を指します。

## 直感的な理解

カメラ画像だけを見て運転を学ぶモデルは、見えている範囲のことしか分かりません。たとえば「この先 50m で道が右にカーブする」「ここは T 字路だ」といった情報は、カメラに映る前から地図には書いてあります。人間のドライバーがカーナビや記憶の地図を頼りに先読みするのと同じで、モデルにも地図を渡すと先読みができて運転判断が安定します。地図は「未来のセンサー観測の事前分布」のような役割を果たすわけです。

問題は「どうやって地図をモデルに食べさせるか」です。一番素朴なやり方は、地図を真上から見た一枚の絵(ラスタ画像)にして CNN(画像を畳み込みで処理するネットワーク)で読ませる方法です。これは実装が楽ですが、絵に変換した時点で情報がかなり落ちます。たとえば「この道は制限速度 50km/h で一方通行」という文字情報は、絵にすると単なる1本の線になってしまいます。線の色を変えて種別を表すこともできますが、表現できる種類には限りがあり、細かいニュアンスは消えます。さらに、ラスタ化は「どの解像度で描くか」を決めた瞬間にそれより細かい構造が量子化されてしまい、車線中心線のような連続的な幾何が階段状にギザつきます。

ベクトル方式は、この「絵にする前の生データ」をそのまま扱おうという発想です。地図はもともと幾何(点や線の位置)と意味(タグ)の構造を持ったデータなので、その構造を保ったままモデルに渡せば情報を捨てずに済みます。連続座標を連続のまま、文字タグを文字のまま運べるのがベクトル方式の本質です。

## 基礎: 前提となる概念

まず地図の種類です。自動運転で使う地図には大きく2つあります。HD マップ(High-Definition map、高精度地図)は車線1本1本をセンチメートル単位で記録した非常に詳細な地図です。精度は高いですが、作成・維持に莫大なコストがかかり(専用計測車での走行・人手アノテーション・継続更新)、世界の道路すべてをカバーするのは現実的ではありません。一方 SD マップ(Standard-Definition map、標準精度地図)はカーナビ程度のざっくりした道路ネットワークの地図で、OSM がその代表です。位置精度はメートル級と粗く車線数も持たないことが多いですが、無料で世界中をカバーしており、誰でも入手できます。研究の大きな流れは「HD マップへの依存を減らし、安く手に入る SD マップ(OSM)を事前情報として最大限活かす」方向にあります。

次に OSM のデータ構造です。OSM は3種類の基本要素でできています。ノード(node)は緯度経度を持つ1つの点で、信号機や標識のような点状の物体や、線を構成する頂点を表します。ウェイ(way)はノードを順番に並べてできた線で、道路や建物の輪郭、川などを表します。リレーション(relation)は複数のウェイやノードをまとめた関係で、たとえば1つの大きな交差点や、複数の線分からなる長い道路、あるいは曲がり禁止のような交通規則を1つのまとまりとして扱います。そして各要素には「タグ(tag)」という意味ラベルが文字列で付きます。タグは `キー=値` の形で、たとえば `highway=residential`(住宅街の道路)、`maxspeed=50`(制限速度50)、`oneway=yes`(一方通行)のように書かれます。タグは自由形式で、誰でも新しい種類を作れるため世界中で何千種類も存在します。

最後にトークン化(tokenization)です。Transformer(系列データを attention で処理するネットワーク)に何かを入れるには、対象を「トークンの列」に変換する必要があります。文章なら単語1つ1つがトークンですが、地図の場合は地図要素1つ(あるいは要素上の代表点1つ)を1トークンとして扱います。各トークンは固定長の数値ベクトルで、その要素の「位置」「意味」「所属」などをまとめて表します。トークン化の粒度には主に2系統あり、要素全体を1トークンにする element-level と、線を等間隔の点に刻んで点ごとにトークンを持つ point-level があります。後者のほうが幾何の表現力が高く、地図構築タスクではよく使われます。

## 仕組みを詳しく

ベクトル地図ブランチの処理は、オフライン前処理とモデル内処理に分かれます。代表的な構成(SDTagNet 系の point-level トークン化)を例に段階的に説明します。

第1段階はデータ取得です。車両が走る経路(軌道)全体を囲む矩形範囲を決め、その範囲の OSM データを取得します。OSM データは Overpass API(OSM のデータベースに対してクエリでデータを取り出すサーバ)から取るのが一般的で、矩形(バウンディングボックス)を指定して中の全要素を取得します。取得は学習・推論のたびに行うとコストが高いので、軌道単位でキャッシュするのが普通です。

第2段階は座標変換です。OSM の点は緯度経度(WGS84 という地理座標系)で記録されています。これをそのままモデルに渡しても扱いにくいので、自車を中心とするローカル座標系(たとえば X=前方、Y=左、Z=上、単位はメートル)に変換します。これで「自車から見て前方20m、左3m」のように、運転に直結した相対位置で地図要素を表せます。緯度経度はそのままだと「経度1度が緯度によって何メートルか変わる」非等方な座標なので、メートル単位の局所平面に落とすことが幾何処理の前提になります。

第3段階はトークン化です。ウェイのような線は、そのままだと長さが要素ごとにバラバラなので、弧長等間隔(線に沿って等しい間隔)で固定個数(たとえば10点)にリサンプリングします。各点の幾何情報(位置)に加えて、その要素のタグ文字列を数値ベクトルに埋め込みます。タグを言語モデルで埋め込む手法は [NLP タグ埋め込み(OSMタグの言語埋め込み)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0366_nlp-tag-embeddings-osm) で詳しく扱います。

第4段階はエンコードです。各トークンについて、位置エンコーディング(座標を高次元のベクトルに変換したもの。連続的な正弦波エンコーディングを使うことが多い)、タグの意味埋め込み、そして「どの地図要素・どのグラフ構造に属すか」を示す識別子を結合し、線形層(`nn.Linear`、入力に行列を掛けて次元を変える層)で共通の埋め込み次元(たとえば256)に射影します。これを複数層(たとえば6層)の Transformer Encoder に通すと、要素同士の関係(この道とこの道は交差点でつながっている、など)を考慮した地図トークンが得られます。出力の形を tensor shape で書くと次のようになります。

```
tokens:           (B, N_max, 256)
key_padding_mask: (B, N_max)
# B      = バッチサイズ(同時に処理するサンプル数)
# N_max  = そのバッチで最も地図要素が多いサンプルに合わせたトークン数
# 256    = 各トークンの次元
# key_padding_mask が True の位置は「実体がない・パディング」を意味する
```

サンプルごとに地図要素の数は大きく違います(交差点の多い市街地と田舎では桁が違う)。Transformer はバッチ内で長さをそろえる必要があるので、一番長いサンプルに合わせて N_max を決め、足りない分はゼロなどで水増し(パディング)し、その位置を `key_padding_mask` で記録しておきます。後段の attention はこのマスクを見て、パディング位置を計算から除外します。具体例として、市街地のサンプルが 800 要素、田舎のサンプルが 30 要素なら、N_max=800 にそろえ、田舎側は 770 個がパディングになります。パディング率が高いとメモリが無駄になるため、極端に長いサンプルを上限で切る(crop する)実装も多いです。

第5段階は融合です。こうして作った地図トークンを、カメラ画像から作った BEV 特徴(Bird's-Eye-View、鳥の目線で真上から見た特徴マップ)へ cross-attention で混ぜ込みます。これは [BEV×OSM クロスアテンション融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0365_bev-osm-cross-attention-fusion) で扱います。

ラスタ方式とベクトル方式を before/after で比べると違いが明確です。

```
ラスタ方式: 地図 → 450×300×3 の画像 → CNN → 450×300×256 の BEV
           情報は「色」に圧縮されている。空セルが大半で無駄が多い。
           解像度を決めた時点で細かい幾何は量子化される。
ベクトル方式: 地図 → N 個のトークン(数百〜数千)→ Transformer → N×256
           タグの意味と連続座標がそのまま残る。要素数ぶんしか持たない。
```

450×300 のグリッドは 135,000 セルもあり、その大半は「何もない地面」です。ベクトル方式は実際に存在する要素の数(たとえば数百)だけトークンを持てばよいので、空白だらけのラスタ画像よりメモリ効率が良くなる場合があります。一方で、ベクトルは「順序のない可変長集合」を扱うためパディングとマスクの管理が必要で、ラスタの「固定形状の画像」より実装が複雑になるトレードオフがあります。

## 手法の系譜と主要論文

ベクトル地図を扱う研究は、まず「HD マップをベクトルで扱う」流れから始まり、その後「SD マップ(OSM)を事前情報として使う」流れへ広がりました。

VectorMapNet(Liu et al., ICML 2023, arXiv:2206.08920)は、HD マップの車線などを「折れ線(ポリライン)」として直接予測・表現する先駆けです。動機は、地図を一度ラスタ画像にしてしまうと、後段の経路計画で扱う際にまた点列に戻す手間と誤差が出ることでした。地図をベクトルのまま end-to-end(入力から出力まで一気通貫)で扱える点が新しさでした。要素を「キーポイント列を自己回帰的に生成する」という DETR 系の集合予測として定式化したのも特徴です。ただし HD マップの構築が前提で、SD マップのような粗くタグ付きのデータは想定外でした。

MapTR と MapTRv2(Liao et al., ICLR 2023 / 2024, arXiv:2208.14437 / 2308.05736)は、車線などのベクトル要素を permutation-equivalent(頂点の順序を入れ替えても同じ形と見なす)に学習することで、オンライン地図構築(走行中にセンサーから地図を作る)の精度と速度を大きく改善しました。VectorMapNet が自己回帰生成で遅かったのに対し、MapTR は全要素を並列にデコードするため大幅に高速で、リアルタイム性を実用域に引き上げました。これも HD レベルのオンライン地図構築が主目的で、外部の SD マップ事前情報を取り込む話とは目的が異なります。

SMERF(Luo et al., CoRL 2023, arXiv:2311.07843)は、SD マップ(OSM)を事前情報としてオンライン地図構築モデルに与える方向を確立しました。SD マップの道路を Transformer に入れて HD マップ予測を助けます。ただし道路の種別は one-hot(該当クラスだけ1にしたベクトル)で表現しており、タグの細かい意味は活かしきれません。P-MapNet(Jiang et al., RA-L 2024, arXiv:2403.10521)も SD マップを事前情報として使う流れにあり、SD マップとともに HD マップの事前分布をニューラルネットで記憶しておく工夫を加えています。

SDTagNet(Immel et al., 2025, arXiv:2506.08997)は、この SD マップ活用をさらに進め、タグを one-hot ではなく言語モデルの埋め込みで表現しました。OSM タグは自由形式で種類が無数にあり、固定クラスに押し込むと情報が落ちる、という問題意識からです。タグを「人間の言葉」と見なして言語モデルで埋め込むことで、未知タグや珍しいタグにも対応でき、SD マップだけで HD マップに近い地図理解を実現しました。「ベクトル地図ブランチ」として運転モデルに組み込まれるのは、多くがこの point-level トークン化と言語タグ埋め込みを組み合わせた SDTagNet 系の構成です。

## 論文の実験結果(定量データ)

SD マップ事前情報を扱う研究は、主に nuScenes(自動運転の標準ベンチマークデータセット。ボストンとシンガポールで収集した1000シーン)と Argoverse 2 でオンライン地図構築タスクを評価します。指標は mAP(mean Average Precision、複数クラスの平均精度。地図要素をどれだけ正しく検出・配置できたかを0から1で測り、高いほど良い)です。地図要素は通常、車線分割線(divider)・横断歩道(pedestrian crossing)・道路境界(boundary)の3クラスに分け、それぞれの AP を平均します。

報告によると、SD マップを一切使わないベースライン(カメラのみで HD マップを推定)に対し、SD マップを事前情報として加えると mAP が大きく向上します。SMERF は one-hot の SD マップ事前情報でも nuScenes でベースライン比おおよそ +5〜+6 mAP 程度の向上を報告しています。SDTagNet はタグの言語埋め込みによってさらに上積みし、SMERF 流の one-hot ベースラインを上回ったと報告されています。特に、学習データに少ししか現れない珍しいタグや、評価時に初めて現れる未知タグを含むシーンで差が出やすいとされます。one-hot 表現はそうしたタグを「未知=無視」とするしかないのに対し、言語埋め込みは文字列の意味から「だいたいこういう道路だ」と汎化できるためです。

アブレーション(どの要素を抜くと性能がどう変わるかを調べる実験)では、(1)言語タグ埋め込みを one-hot クラスに置き換えると性能が下がる、(2)幾何情報(point-level の位置)だけでタグを抜くとさらに下がる、(3)グラフ構造の識別子(要素の所属関係)を抜くと交差点まわりの精度が落ちる、といった傾向が報告されています。これは「幾何」「意味タグ」「構造」の3要素がそれぞれ独立に効いていることを示します。数値の意味としては、mAP が数ポイント違うだけでも、地図要素を1割多く正しく置けるかどうかの差であり、下流の経路計画の安全性に直結するため重要です。たとえば横断歩道を取りこぼせば「そこに歩行者が現れうる」という前提自体が消え、減速判断が遅れます。

## メリット・トレードオフ・限界

メリットは第一に、タグ(意味)情報を落とさずモデルに渡せることです。一方通行・道路種別・制限速度などの情報が、絵に潰されずそのまま活きます。第二に、実在する要素数ぶんのトークンしか持たないため、空白の多いラスタ画像よりメモリ効率が良い場合があります。第三に、HD マップが要らず、無料で世界中をカバーする OSM だけで地図先読みができることです。これはサービスを未整備地域へ展開する際のコストを劇的に下げます。

トレードオフと限界もあります。タグを言語モデルで埋め込む場合、その言語モデル(BERT 系など)を別途用意・ロードする必要があり、依存ライブラリと計算量が増えます。また Overpass API への通信とオフライン前処理が必要で、データパイプラインが複雑になります。ベクトルは「順序のない可変長集合」なので、パディングとマスクの取り回しが避けられず、ラスタより実装が面倒です。さらに OSM の主要タグは英語前提のため、地域固有の言語で書かれたタグには弱い可能性があります。OSM そのものの精度や更新頻度に品質が左右される点も SD マップ全般の限界で、現地と地図が食い違う(工事で道が変わった等)場合に誤った先読みをするリスクがあります。研究上の未解決課題としては、自車測位の誤差で地図がずれたときの頑健性(地図を絶対正と信じすぎない確率的な扱い)、そしてリアルタイム推論での前処理コスト削減が挙げられます。

## 発展トピック・研究の最前線

ベクトル地図表現は、地図構築だけでなく「地図を条件として軌道を予測・生成する」方向へ広がっています。SD マップトークンを直接 motion forecasting(他車の動き予測)や planning(自車の経路計画)の条件に使う研究が増えており、地図を中間ラスタ表現を経ずに end-to-end な運転モデルへ統合する流れにあります。また、言語モデルによるタグ埋め込みは、近年の VLM(Vision-Language Model、画像と言語を同時に扱うモデル)を運転に取り込む潮流とも親和性が高く、地図・画像・自然言語指示を共通の埋め込み空間で扱う研究につながっています。位置合わせ誤差への対処としては、SD マップを「だいたいの事前分布」として確率的に扱い、観測とずれたら信頼度を下げる研究も進んでいます。さらに、OSM の道路グラフを GNN(Graph Neural Network、グラフ構造を扱うネットワーク)で先にエンコードしてから Transformer に渡すハイブリッド設計も検討されています。
