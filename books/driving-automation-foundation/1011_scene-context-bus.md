# SceneContext 型付きバス

## ひとことで言うと

モデルが得た「高レベルなシーン理解 (この場面では今、何が起きているのか)」を、あらかじめ形が決まったデータの箱に詰めて、軌跡を決める部分 (プランナー) へ渡すための共通インターフェースです。箱の中には、推論を表す潜在ベクトル、分類のロジット、その推論をどれくらい信じてよいかを表す確信度、そしてその推論がどこから来たのかという出どころ (provenance) が入ります。プランナーを「ある1つの決まった出力形式」に縛り付けないことが狙いで、後からシーン理解の中身を入れ替えても、プランナー側のコードを書き直さずに済むようにする、ゆるく疎結合な設計です。型を定義するだけで効果が出るわけではなく、あくまで「進化の速いシーン理解」と「安定が求められるプランナー」を分離するための足場です。

## 直感的な理解

ニューラルネットワークを部品の集まりとして考えてみます。一方に「場面を理解する部品 (ここでは何が運転判断を左右しているかを推論する)」があり、もう一方に「その理解を使って実際にハンドルやアクセルに相当する軌跡を出す部品 (プランナー)」があります。この2つをどうつなぐかが問題です。

素朴なつなぎ方は、理解する部品の生の出力をそのままプランナーの引数に流し込むことです。たとえば「5種類の運転理由を表すスコアの並び (ロジット) `[B, 5]`」をプランナーが直接受け取る、という具合です。一見シンプルですが、これには困った副作用があります。プランナーの入力仕様が「5次元のロジット」という1つの形に固定されてしまうのです。

ここで現実の研究開発を想像してみてください。最初は「5種類の運転理由」を渡していたが、途中で「高精細地図 (HD マップ) の情報も渡したい」「ナビの経路指示も渡したい」「もっと表現力のある大規模言語モデルの出力に差し替えたい」と要望が出てきます。プランナーの引数が `[B, 5]` 固定だと、そのたびにプランナー本体の関数シグネチャ (引数の定義) を書き換え、プランナーを使っている他の全部分も連鎖的に直すことになります。研究コードでは、シーン理解の試作は週単位で作り替わる一方、プランナーは一度動いたら安定運用したい、という非対称があり、この連鎖的な書き換えコストが開発速度の足かせになります。

身近な比喩で言えば、これは「コンセントの形を決めずに機器を直接はんだ付けする」状態です。一度はんだ付けしてしまうと、機器を取り替えるたびに配線をやり直さねばなりません。対して、規格化されたコンセント (=型付きの箱) を間に挟んでおけば、コンセントの形 (インターフェース) さえ守れば、どんな機器でも差し替え自由になります。SceneContext はこの「規格化されたコンセント」に相当します。

## 基礎: 前提となる概念

いくつか用語を噛み砕いておきます。

- プランナー (planner) とは、エンドツーエンド (入力から出力まで1つのネットワークで貫く) 運転モデルの中で「最終的にどんな軌跡 (車が通る進路の予測) を出すか」を決める部分です。
- バス (bus) は、もともとコンピュータの世界で「複数の部品がデータをやり取りする共通の通り道」を指す言葉です (アドレスバス、データバスなど)。ここでは「シーン理解の情報をプランナーへ運ぶ共通の通り道」という比喩で使われます。
- 型付き (typed) とは、データの形 — どんな名前のフィールドが、どんな種類 (テンソルか、文字列か) で入っているか — が事前に決まっていること。Python では `@dataclass` という仕組みで、フィールドと型を宣言した小さな構造体として表現するのが一般的です。型があると、誤ったフィールド名や形のデータを渡したときに、実行のずっと前に静的解析やレビューで気づけます。
- ロジット (logits) とは、softmax (合計が1の確率に変換する関数) を通す前の生のスコアのこと。`[2.1, -0.3, 5.0, 0.0, 1.2]` のような実数の並びで、大きいほどそのクラスらしいことを表します。
- 確信度 (confidence) とは、モデルがその予測をどれくらい確からしいと考えているか。よくロジットを softmax にかけた確率の最大値で測ります。
- 出どころ (provenance) とは、そのデータがどのモジュール・どのバージョン・どのデータソースから来たのかという来歴情報です。データ工学で品質管理・追跡 (トレーサビリティ) のために重視されます。
- 疎結合 (loose coupling) とは、部品どうしが互いの内部実装を知らずに、約束されたインターフェースだけを介してやり取りする状態。逆が密結合 (tight coupling) で、片方を変えると相手も壊れます。

## 仕組みを詳しく

型付きバスの典型的な実装は、ネスト (入れ子) になった2段の構造体です。下位の構造体が「ある1種類のシーン理解の中身」を、上位の構造体が「シーン理解全体をまとめる容器」を表します。

下位 (推論1種類分):

```
ReasoningContext:
  reasoning_latent : Tensor [B, latent_dim]   推論を表す連続値の潜在ベクトル
  class_logits     : Tensor [B, num_classes]  分類の生スコア
  confidence       : Tensor [B]               0〜1 のスカラー確率
  provenance       : str                       例 "scene_reasoning_head_v1"
```

上位 (容器):

```
SceneContext:
  reasoning : Optional[ReasoningContext] = None   無ければ None でよい
  # 将来の拡張に備えたフィールド例 (どれも Optional):
  hd_map        : Optional[...] = None
  route_command : Optional[...] = None
  traffic_state : Optional[...] = None
```

各フィールドの役割を、形 (tensor shape) と数値例で見ます。バッチサイズ B=4 とします。

- `reasoning_latent` `[4, latent_dim]`。たとえば `latent_dim=256`。これはシーン理解を要約した連続ベクトルで、プランナーが文脈として取り込み、軌跡生成を条件づけられます。ロジットと違い「離散クラス」に潰れていないので、微妙なニュアンスを保持できます。たとえば「歩行者がいるが、まだ遠く、減速の必要は弱い」といった連続的な程度の情報は、5択のロジットには載りませんが、256 次元の潜在には埋め込めます。
- `class_logits` `[4, 5]`。5クラス分類なら、各行が5つの生スコア。
- `confidence` `[4]`。各サンプル1個ずつの確信度。典型的には `confidence = softmax(logits, dim=-1).max(dim=-1).values` で計算します。たとえばある行のロジットが `[0.1, 0.2, 3.4, 0.1, 0.0]` なら、softmax 後の最大確率はおよそ 0.90 になり、「この推論はかなり確からしい」とプランナーに伝わります。逆に `[0.5, 0.6, 0.7, 0.55, 0.5]` のように横並びだと最大確率は 0.22 程度で、「迷っている、あまり信じるな」という信号になります。
- `provenance`。文字列で来歴を記録。ログ追跡や、信頼できないソースの除外に使えます。実運用では「どのバージョンの理解器が出した推論か」を後から監査でき、不具合の切り分けに直結します。

上位 `SceneContext` のフィールドがすべて `Optional`(= `None` を許す) であることが、この設計の核心です。シーン理解を使わない構成では全部 `None` にしておけばよく、新しい種類の理解を足したいときはフィールドを1つ追加するだけで、既存のフィールドや、それを読まないプランナーには一切影響しません。

プランナーへの渡し方も工夫されます。よくあるのは、プランナーの基底クラス (共通インターフェース) の `forward` を `**kwargs` (任意のキーワード引数を受け取る仕組み) にしておき、`scene_context=...` を渡すやり方です。すると、

- SceneContext を使いたいプランナーは `kwargs.get("scene_context")` で受け取り、軌跡生成を条件づける。
- 使わない既存プランナーは `kwargs` を無視するだけで、引数定義を1文字も変えずに動き続ける。

つまり SceneContext は完全にオプショナルで、後方互換 (古い実装を壊さない) を保ったまま機能を足せます。なぜこの設計にするのか、を一言で言えば「シーン理解の進化スピードと、プランナーの安定性を両立させたいから」です。研究段階のシーン理解は頻繁に作り替えられますが、そのたびにプランナーの契約を壊していては開発が止まります。

確信度をわざわざ箱に同梱するのも理由があります。確信度が無いと、プランナーは「不確かな推論」と「自信のある推論」を区別できず、どちらも同じ重みで信じてしまいます。確信度があれば、たとえば「confidence < 0.5 のときは推論を無視して安全側に倒す」「confidence でゲートをかけて latent の混ぜ込み量を調整する (`ego_hidden + confidence * latent`)」といった、不確実性に応じた振る舞いが書けます。これは下流の安全機構を設計する上での重要な接点です。

## 手法の系譜と主要論文

この型付きバスは、特定の1論文の手法というより、自動運転アーキテクチャ研究とソフトウェア工学の両方の知見が合流した設計パターンです。系譜を追います。

- モジュラー型の古典 (〜2020 年頃) では、知覚 → 予測 → 計画を別々のモデルに分け、人間が決めた中間表現 (検出ボックス、占有グリッドなど) でつなぐのが主流でした。利点は各モジュールを個別に検証・デバッグできること。欠点は、中間表現が計画にとって最適とは限らず、誤差が段階的に蓄積すること (error accumulation) です。歩行者検出が外れると、その後の予測も計画も連鎖的に崩れます。

- Hu et al., "Planning-oriented Autonomous Driving (UniAD)" (CVPR 2023, Best Paper, arXiv:2212.10156) が転機になりました。知覚・予測・計画を1つのネットワークに統合し、各モジュールの中間表現を「クエリ (query)」という学習可能なベクトルの形で明示的に次段へ受け渡す設計を提案しました。動機は「タスクは分けたいが、表現は共有し、すべてを最終的な計画品質に向けて最適化したい」こと。SceneContext の「中間表現を構造的に次段へ渡す」発想は、この系譜に位置づけられます。違いは、UniAD が知覚・予測・計画のフルスタックを統合する重厚な設計なのに対し、型付きバスはもっと軽量で、シーン理解を1つの optional な部品として後付けできる点です。

- Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving" (ICCV 2023, arXiv:2303.12077) は、UniAD の重厚な高密度マップ表現を、ベクトル化した軽量な中間表現に置き換え、計画性能をほぼ維持したまま推論を大幅高速化しました。これは「中間表現の形 (インターフェース) は差し替え可能であるべき」という思想を実証した例で、型付きバスが目指す「中身を差し替えても下流を壊さない」発想と通じます。

- 不確実性を下流へ伝える重要性は、Kendall and Gal, "What Uncertainties Do We Need in Bayesian Deep Learning for Computer Vision?" (NeurIPS 2017, arXiv:1703.04977) が体系化しました。彼らは不確実性を「偶然性 (aleatoric、データそのもののノイズ)」と「認識論的 (epistemic、モデルの知識不足)」に分け、両方を扱うと安全性とロバスト性が上がることを示しました。SceneContext が `confidence` を必ず同梱するのは、この「不確実性を捨てずに下流へ運ぶ」思想の素直な実装です。

- ソフトウェア工学側では、依存性逆転の原則 (Dependency Inversion Principle, Robert C. Martin が SOLID 原則の一つとして整理) が背骨です。「上位モジュールは下位モジュールの具体に依存せず、両者が抽象 (インターフェース) に依存すべき」という原則で、プランナー (上位) をシーン理解の具体実装 (下位) から切り離す SceneContext は、まさにこれを ML パイプラインに適用したものです。

## 論文の実験結果(定量データ)

型付きバスそのものは設計パターンなので直接のベンチマーク数値は持ちませんが、その背景にある「中間表現を構造的に渡す統合設計」の効果は UniAD/VAD の結果で読めます。両者は nuScenes データセット (大規模な実走行マルチセンサデータセット、約 1000 シーン、6カメラ + LiDAR) で評価されました。主要な指標と報告値:

- 計画 (planning) の指標である L2 誤差 (予測軌跡と実際の軌跡のユークリッド距離、メートル単位、小さいほど良い)。UniAD は平均 L2 を当時の代表手法より大きく下げ、1秒/2秒/3秒先の平均でおよそ 1.0 m 前後だったベースラインに対し、報告では平均でおよそ 0.5〜0.7 m 程度まで改善したとされます。L2 が小さいほど、モデルが人間ドライバーの実際の進路に近い軌跡を出せていることを意味します。
- 衝突率 (collision rate、予測軌跡が他物体と衝突する割合、%、小さいほど安全)。UniAD は 3秒平均でおよそ 0.3 % 前後と報告し、当時の手法より低い水準を達成しました。衝突率が 1 % から 0.3 % へ下がるとは、危険な予測軌跡の頻度が約 1/3 になることを意味します。
- VAD は同等の計画品質 (L2・衝突率) を保ちつつ、推論速度を UniAD 比でおよそ数倍 (報告で 9 FPS 前後 vs 2 FPS 前後規模) に引き上げました。これは「中間表現を軽量な形に差し替えても下流性能は守れる」ことの実証です。

UniAD の重要なアブレーション (どの要素を抜くとどうなるかの検証) は「中間タスクの表現を計画へ明示的に渡すか否か」でした。知覚・予測の表現を計画へ渡すクエリ経路を切ると計画性能が落ちる、という結果は、「上流の理解を構造化して下流へ運ぶこと自体に価値がある」という SceneContext の前提を支持します。

不確実性側では、Kendall & Gal が示したのは、出力に不確実性推定を加えるとセグメンテーションや深度推定で精度が向上し (報告では数 % 規模)、かつ予測確率の信頼性 (キャリブレーション) が改善する、という点でした。確信度を下流へ運ぶ SceneContext の `confidence` フィールドは、この「不確実性は捨てるとコストが高い」という実証に裏打ちされています。

## メリット・トレードオフ・限界

メリット
- プランナーをシーン理解の具体実装に縛らず、後から中身を差し替え・追加しやすい (疎結合)。
- `confidence` と `provenance` を持つことで、信頼度や来歴に基づく判断・フィルタ・ログ追跡ができる。
- `Optional` 設計により、シーン理解を使わない構成や旧来のプランナーを壊さず後方互換を保てる。
- フィールド追加だけで将来の拡張 (HD マップ、ルート指示、別の推論器) に対応できる。

トレードオフ・限界
- 抽象化を1段増やすぶん、最小限の実装よりコードが複雑になり、初学者には流れが追いにくい。
- 箱を用意してもプランナー側が中身を実際に使わなければ意味がない。「型を定義した」ことと「効果を出した」ことは別問題で、消費側の統合と検証が別途必要。
- 構造体のフィールドを決め打ちするため、想定外の新しい種類の情報を入れたいときは型定義の更新が要る (Optional で緩和はできるが、完全に自由ではない)。可変長・異種混合のモダリティを扱いたいときは、固定フィールドより辞書やクエリ集合のほうが柔軟、という設計上の対立がある。
- 研究上の未解決課題として、確信度をプランナーがどう使うのが最適か (ゲーティング、重み付け、安全側フォールバックなど) の定石はまだ確立していません。確信度のキャリブレーションが甘いと、せっかくの不確実性情報が誤誘導になりうる点も注意が必要です。

## 発展トピック・研究の最前線

- クエリベースの統合運転 (UniAD, VAD など) は、中間表現を「学習可能なクエリの集合」として渡す方向へ進化しています。型付きバスを固定フィールドの構造体ではなく、可変長のクエリ集合に拡張する発想と接続できます。
- 大規模言語モデル / 視覚言語モデル (VLM) を運転に組み込む研究 (DriveGPT4, Xu et al. 2023; LMDrive, Shao et al. CVPR 2024) では、言語による推論結果を低レベル制御へつなぐ「インターフェースの設計」がまさに論点で、型付きバスはその素朴な一形態です。言語トークン列をどう連続制御に橋渡しするか、という課題は活発に研究されています。
- 不確実性のキャリブレーション (Guo et al., "On Calibration of Modern Neural Networks", ICML 2017, arXiv:1706.04599) を確信度フィールドに適用すれば、確信度の数値が実際の正解率と一致するようになり、下流での利用が安全になります。temperature scaling のような後処理だけで ECE を大きく下げられることが知られています。
