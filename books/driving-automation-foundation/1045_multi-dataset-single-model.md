# 多データセット単一モデル学習

## ひとことで言うと
カメラの台数も、走った国も、記録の形式もバラバラな複数の運転データセットを、別々のモデルにせず1つのモデルでまとめて学習しよう、という全体設計の目標です。学習データは多く多様なほど性能が伸びるため、データ源を統合して量を最大化したい。これを実現する鍵は「カメラ台数の動的化」と「座標系の統一」、そして「校正情報による位置差の吸収」と「フォーマットの正規化」です。

## 直感的な理解
料理学校を想像してください。フランス校・北米校・別校があり、それぞれ厨房の道具立て（カメラ台数）も食材の単位（座標系）もレシピの書式（フォーマット）も違います。素朴にやると「フランス校専用シェフ」「北米校専用シェフ」を別々に育てるしかなく、各シェフは自分の校の経験しか積めません。しかし「火を通す」「味を見る」といった料理の本質はどの校でも共通です。多データセット単一モデルは、全校の経験を1人のシェフに集約し、共通の本質を最大限の事例から学ばせる、という発想です。

そのためには、道具・単位・書式の違いを「シェフの頭」ではなく「入口の変換係」で吸収する必要があります。どの校のデータも、シェフが扱う共通の形に翻訳してから渡せば、シェフは1人で全校に対応できます。この「変換係」が、後述する台数動的化・校正・座標統一・パーサの4点セットです。

## 基礎: 前提となる概念
- データセット（dataset）: 実際に車を走らせて記録した、画像・速度・操作量などのまとまった運転ログ。
- ドメイン（domain）: データの「出どころの性質」。国・気候・時間帯・運転習慣・センサ構成などが違えばドメインが違う、と言います。
- ドメインギャップ（domain gap）: 学習に使ったデータと、実際に使いたい状況のデータの分布の差。差が大きいと、学習で良くても本番で性能が落ちます。
- 校正（calibration）: 各カメラの位置・向き（外部パラメータ、extrinsics）と焦点距離・レンズ歪み（内部パラメータ、intrinsics）を測った情報。3次元と画像2次元を正しく対応づけるのに必須です。
- スケーリング則（scaling law）: モデルやデータの規模を増やすと性能がべき乗的に伸びる経験則。言語モデルで定式化され、運転モデルでも「データ量を稼ぐ」動機の根拠になっています。
- 負の転移（negative transfer）: 別ドメインのデータを混ぜたことで、かえって互いの性能が落ちる現象。混ぜれば必ず良くなるわけではないことを示す重要概念です。

なぜ単一モデルか。現実の運転データセットは提供元ごとに、(1)カメラ台数が違う、(2)カメラの取り付け位置・向きが違う、(3)座標系や単位が違う、(4)記録フォーマットが違う、という4つの壁を持ちます。素朴に作るとこれらがモデルの形に焼き付き、データセット専用モデルを別々に作るしかなくなります。これは2つの理由でもったいない。第一に、大量データを合算できず各モデルが少量しか見られない。第二に、運転の本質（信号で止まる、車線を保つ、歩行者を避ける）はデータセットをまたいで共通なのに、その共通知識を1モデルに蓄積できない。

## 仕組みを詳しく
4つの壁を、どう乗り越えるかを順に説明します。

1. カメラ台数の違い → num_views の動的化

中核の融合（複数カメラを鳥瞰図 BEV に統合する処理）を、実行のたびにカメラ台数 `V` を見て対応する形にします。BEV は固定サイズのマス目（例 450×300）が「今あるカメラ全部」に問い合わせるので、出力は台数に関係なくいつも `[B, embed_dim, bev_h, bev_w]`（例 embed_dim=256）という同じ形になります。これにより7台バッチも8台バッチも同じモデルインスタンスが処理できます。実装上の制約として、1ミニバッチ内ではカメラ台数を揃える必要があるため、データセットごとに均一バッチを作り交互に流します。詳細は [num_views動的化（多データセット単一モデル）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1047_num-views-dynamic-bev)。

2. カメラ位置・向きの違い → 校正情報で吸収

各データセットの外部・内部パラメータを投影行列としてモデルに渡すと、BEV のマス目を各カメラ画像の正しい場所に投影できます。つまり「カメラがどこに付いているか」をモデルの重みではなくデータ側の情報として扱うので、搭載位置が違う車でもシームレスに統合できます。これは BEVFormer 系がカメラ構成に依存しない BEV を作れる理由そのものです。重みに焼き付けず外部入力として渡す、という発想自体が「構成依存性をデータ側に押し出す」一般原則の実例です。

3. 座標系・単位の統一

出力する軌跡（trajectory、車がこれからたどる経路）を、データセットごとの生の座標ではなく共通表現に揃えます。一般的な工夫として、軌跡を絶対座標 `(x, y)` ではなく、簡易車両運動モデル（unicycle / 自転車モデル）の制御信号（例えば加速度と曲率の系列）として表す方法があります。こうすると車体基準の相対表現になり、データセット間の座標系の取り方の違い（原点・軸方向・進行方向の規約差）に左右されにくく、損失（誤差の指標）を統一的に計算できます。加速度・曲率は物理的に解釈でき、運動学的に妥当な滑らかな軌跡を出しやすいという利点もあります。

4. フォーマットの違い → パーサで内部形式へ正規化

各データセットに専用のパーサ（読み込み・変換プログラム）を用意し、画像・自車運動・操作量を共通の内部テンソル形式に揃えます。ここで時刻同期（複数カメラと車両ログのタイムスタンプ合わせ）、座標系の符号・軸方向の統一、単位変換（km/h と m/s など）といった地道な正規化を行います。地味ですが、ここの取りこぼし（例: 軸の符号を1つ間違える）は学習を静かに汚染するため、データ統合の品質を左右する要です。

ASCII でまとめると（データセット名は例示）:

```
 DatasetA(7cam,欧州) DatasetB(8cam) DatasetC  Waymo(5cam,北米)  nuScenes(6cam)
      │              │            │            │                │
   [パーサ]       [パーサ]     [パーサ]      [パーサ]         [パーサ]   ← (4)書式の正規化
      └──────────────┴────────────┴────────────┴────────────────┘
                                 ▼  共通内部形式 (画像 + 校正 + 制御信号)
                      [ 台数動的 BEV 融合 ]   ← (1)台数動的化 + (2)校正で位置差吸収
                                 ▼  常に [B,256,bev_h,bev_w]
                      [ プランナ ] → 共通の制御信号軌跡   ← (3)座標統一
                                 ▼
                       1つの損失で統一的に学習
```

まとめると、多データセット単一モデルとは「(1)台数を動的化し、(2)校正で位置差を吸収し、(3)出力を共通の制御信号に統一し、(4)入力をパーサで正規化する」4点セットの設計目標です。融合出力もプランナ入出力も全データセットで同一形状に揃うため、学習ループ自体はデータセットを切り替えても変更不要、というのが理想形です。

## 手法の系譜と主要論文
- nuScenes（Caesar et al., CVPR 2020, arXiv:1903.11027）と Waymo Open Dataset（Sun et al., CVPR 2020, arXiv:1912.04838）。大規模で校正情報の整った公開マルチカメラ運転データセットを提供し、研究を一気に加速させました。しかし両者はカメラ台数（nuScenes 6台・Waymo 5台）も座標規約も異なり、「混ぜて使う」ための統一表現の必要性を浮き彫りにしました。
- BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）。カメラ台数・搭載位置に依存しない BEV 表現を作る手法。異なるセンサ構成を共通の鳥瞰図特徴に落とし込めるため、多データセット統合の技術的な土台になりました。トレードオフは正確な校正と重い投影計算です。
- UniAD（Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156）。検出・追跡・地図・予測・計画を1つのネットワークに統合する end-to-end 設計。タスクを分割すると情報が失われるため、計画に向けて全体を一気通貫で学習するのが動機で、計画性能でモジュール寄せ集めを上回りました。多データセット単一モデルは、この「1モデルに集約する」流れをさらにデータ源をまたいで広げる発想です。トレードオフは巨大で学習が重く、データの統一表現が前提になる点です。
- スケーリング則（Kaplan et al., 2020, arXiv:2001.08361）。言語モデルで「データ・モデル・計算を増やすほど損失がべき乗で下がる」ことを定式化。運転モデルでも「データ源を統合して量を稼ぐ」動機を理論的に後押しします。ただし、ただ混ぜるだけでなくドメイン差を吸収する設計が伴って初めて効きます。

系譜は「大規模だが規約バラバラな公開データの登場（nuScenes/Waymo）」→「構成非依存の BEV 表現で統合の土台ができる（BEVFormer）」→「1モデルに全タスクを集約する潮流（UniAD）」→「データ源もまたいで集約しスケールさせる（多データセット単一モデル）」と進みます。

## 論文の実験結果(定量データ)
- UniAD（CVPR 2023）: nuScenes で、計画タスクの安全性指標である衝突率（collision rate、予測軌跡が他物体とぶつかる割合、低いほど良い）を、モジュールを分割した従来手法に対して明確に低減したと報告されています。具体的には平均衝突率を従来比でおよそ半分以下に下げたケースが示され、L2 誤差（予測軌跡と正解軌跡の平均距離、低いほど良い、単位はメートル）も改善しました。これは「タスクを統合して情報を共有すると計画が良くなる」ことの定量的証拠で、統合の方向性を支持します。
- UniAD のアブレーション: 上流タスク（追跡・地図・予測）を段階的に外すと計画性能が悪化することが示され、各モジュールが計画に寄与していることが定量的に確認されました。これは「統合は単なる足し算でなく相互補完がある」ことを示します。
- BEVFormer（ECCV 2022）: nuScenes 検出で NDS（総合検出スコア、高いほど良い）およそ 0.51〜0.56 を達成。台数非依存の BEV 表現が高性能と両立することを示し、これを共通土台にすれば構成の違うデータも同じ枠組みで扱えるという見通しを与えました。
- データスケーリングの一般的知見: 言語・視覚いずれの分野でも、学習データ量を増やすと性能が伸びる傾向が繰り返し確認されています。一方で、ドメインが大きく異なるデータを無造作に混ぜるとドメイン干渉（negative transfer）で性能が落ちる例も報告されており、サンプリング比率の調整やドメイン条件付けが必要だと示されています。この「混ぜれば伸びるが、混ぜ方が重要」という両面が、多データセット学習の実務上の核心です。

注意: 「複数の運転データセットを混ぜると単一データセット学習に対して何ポイント改善するか」という直接の定量比較は、構成・指標がデータセット間で揃わないため公開ベンチマークでも確立されていません。したがってここでの数値は、統合の方向性を支持する周辺証拠（タスク統合の効果・台数非依存表現の性能・一般的スケーリング則）として読むのが正確で、データ統合そのものの効果量は今後の実証課題です。

## メリット・トレードオフ・限界
- メリット: 複数データセットを合算でき、学習データ量と多様性が最大化される。
- メリット: 運転の共通知識を1モデルに集約でき、新しい車種・地域への転用（転移学習の素地）が容易になる。
- メリット: モデルを構成ごとに分けずに済み、実験・運用が単純化する。
- トレードオフ: 各データセットの校正情報の整備と、共通表現への正規化（パーサ）整備のコストがかかる。
- トレードオフ: ミニバッチ内は台数均一の制約があり、データセットごとに均一バッチを組むサンプリング設計が要る。
- 限界: ドメイン差（国・天候・運転習慣・カメラ視野）が大きいと、そのまま混ぜるとドメイン干渉で逆効果になりうる。サンプリング比率調整やドメイン条件付けが必要。
- 限界: 校正が不正確・欠損なデータセットでは、BEV 投影が崩れて学習を汚染する。校正品質の検査が前提になる。
- 限界: 評価が難しい。データセットごとに正解の作り方・指標が違うため、統一モデルを公平に評価する共通指標の設計自体が未解決の研究課題。

## 発展トピック・研究の最前線
多データセット単一モデルの先には、いくつかの活発な研究方向があります。第一に、データセットを区別する識別子をモデルに与えるドメイン条件付け（dataset-aware な正規化やプロンプト）で、共通知識を共有しつつデータ源固有の癖も学ぶ手法。第二に、サンプリング比率を学習中に動的に調整して負の転移を抑えるバランシング戦略。第三に、自己教師あり事前学習や大規模基盤モデルを土台に、少量の各データセットでファインチューンする転移学習の枠組み。第四に、画像・LiDAR・地図・言語など異種モダリティをまたいで統合するマルチモーダル基盤モデルへの拡張で、運転版の「あらゆる入力を1モデルで」を目指す流れです。さらに、合成データ（シミュレータ生成）と実データを混ぜる sim-to-real の文脈でも、構成の異なるデータを統一表現に揃える本トピックの考え方が直接効いてきます。実証面では、統合の効果量を公平に測る評価プロトコルの確立が、この分野の最大の宿題として残っています。
