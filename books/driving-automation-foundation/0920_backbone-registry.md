# バックボーンのレジストリ構成

## ひとことで言うと

画像から特徴を取り出す「バックボーン」という心臓部の部品を、設定ファイルの文字列ひとつで Swin Transformer V2 / ConvNeXt V2 / ResNet などのどれにでも差し替えられるようにする設計手法です。コード本体を書き換えずに、名前を変えるだけでバックボーンを入れ替えられます。実現の鍵は「名前 → 作り方」の対応表(レジストリ)と、バックボーンごとに違う出力チャネル数を自動で調べる仕組みです。

## 直感的な理解

レストランの厨房を想像してください。メインの調理器具(オーブン、フライパン、グリル)を日替わりで使い分けたいとします。もし「どの器具を使うか」がレシピの本文の至るところに直接書き込まれていたら、器具を変えるたびにレシピ全体を書き直すことになり、ミスが起きやすい。

そこで「器具の注文票」を 1 枚作ります。注文票には「オーブン → このメーカーのこの型番」「グリル → あのメーカーのあの型番」という対応表が書いてあります。レシピ本文は「メイン器具を使う」としか書かず、当日の注文票で器具を 1 つ選ぶだけ。新しい器具を増やすときは注文票に 1 行足すだけで、レシピ本文はいじりません。これがレジストリパターンの発想です。さらに、器具ごとに「火力(出力の太さ)」が違っても、それを自動で読み取って後続の調理を合わせる仕組みがあれば、本当に注文票だけで切り替えが完結します。

## 基礎: 前提となる概念

まずバックボーン(backbone)とは何かを説明します。カメラ画像(例えば 256×256 ピクセルの写真)は、そのままでは「色のついた点の集まり」でしかありません。ここから「ここに車がいる」「これは道路の白線だ」といった意味のある情報を取り出す必要があります。この「画像から意味のある特徴(feature)を抽出する部品」がバックボーンです。背骨(backbone)が体を支えるように、知覚モデル全体を支える土台なのでこう呼ばれます。CNN(畳み込みニューラルネットワーク。画像を小さな窓で走査して模様を捉える方式)や Transformer(注意機構で離れた箇所同士の関係を捉える方式)が使われます。

問題は、「どのバックボーンが一番良いか」は一概に決まらないことです。タスク・データ量・計算予算・必要な解像度によって最適解が変わります。代表的な候補の特徴を挙げます。

- ResNet(特に ResNet-50)は研究論文で最もよく使われる定番です。残差接続という仕組みで深いネットワークを安定して学習でき、他の論文の数値と直接比べやすい(BEVFormer、UniAD、VAD などが ResNet 系を使用)。
- Swin Transformer V2 は、入力解像度を変えても性能が落ちにくい設計で、後で画像を大きくする(例えば 256→448)計画があるとき有利です。
- ConvNeXt V2 は「現代化された ResNet」で、Transformer 並みの性能を純粋な畳み込みだけで出します。注意機構を持たないぶん実装と高速化が素直です。

研究フェーズでは、これらを何度も切り替えて比較したくなります。もしバックボーンの種類がコードの奥深くにハードコード(直接書き込み)されていると、切り替えるたびに複数のファイルを書き換えることになり、間違いが起きやすく、実験の記録も残しにくくなります。

そこで「レジストリパターン」を使います。レジストリ(registry)とは「登録台帳」という意味で、ここでは「名前 → 作り方(生成関数)」の対応表のことです。多くの深層学習フレームワーク(MMDetection や Detectron2 など)が、モデル部品を切り替える標準手法としてこのパターンを採用しています。

## 仕組みを詳しく

レジストリの典型は、次のような辞書(対応表)です。

```python
BACKBONE_REGISTRY = {
    "swin_v2_tiny":      lambda pretrained=True, **kw: timm.create_model("swinv2_tiny_window8_256", pretrained=pretrained, features_only=True, **kw),
    "conv_next_v2_tiny": lambda pretrained=True, **kw: timm.create_model("convnextv2_tiny",        pretrained=pretrained, features_only=True, **kw),
    "res_net_50":        lambda pretrained=True, **kw: timm.create_model("resnet50",               pretrained=pretrained, features_only=True, **kw),
}
```

キー(左側の文字列)が「設定で指定する名前」、値(右側)が「実際にそのモデルを作る関数」です。`lambda` は「呼ばれたときに初めてモデルを作る、その場限りの小さな関数」だと思ってください。これにより、起動時に 3 つすべてを作るのではなく、選ばれた 1 つだけを作ります(遅延生成)。

ここで使われている `timm`(PyTorch Image Models、Ross Wightman 作)は PyTorch 用の有名な画像モデル集ライブラリで、`create_model("名前", pretrained=True)` と書くだけで、ImageNet などで事前学習済みの重み込みのモデルが手に入ります。`pretrained=True` は「ゼロから学習し直すのではなく、世界中の画像で先に賢くしてある重みを引き継ぐ」という意味で、学習を大幅に短縮し、少ないデータでも安定させられます。これを転移学習(transfer learning)と呼びます。

`features_only=True` が重要です。通常の分類モデルは最後に「犬か猫か」のような 1 個の答えを出しますが、ここでは途中の 4 段階の特徴マップ(解像度の違う 4 枚の中間結果)をリストで受け取りたい。`features_only=True` はその「途中段階のマルチスケール特徴を返す」モードです(この 4 段階の使い道は [FPNネックとチャネル射影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1039_fpn-neck-channel-projection))。

呼び出し側は次のような関数を使います。

```python
def build_backbone(backbone, pretrained=True, **kwargs):
    if backbone not in BACKBONE_REGISTRY:
        raise ValueError(f"Unknown backbone '{backbone}'. Available: {list(BACKBONE_REGISTRY.keys())}")
    return BACKBONE_REGISTRY[backbone](pretrained=pretrained, **kwargs)
```

ポイントは、知らない名前が来たら「利用可能な名前一覧」付きでエラーを出すことです。タイプミスがすぐ分かり、デバッグが楽になります。

### チャネル数の自動発見

バックボーンごとに、出力される特徴のチャネル数(特徴の太さ)は違います。

- Swin V2 Tiny: 4 段階で 96, 192, 384, 768(合計 1440)
- ConvNeXt V2 Tiny: 96, 192, 384, 768(合計 1440)
- ResNet-50: 256, 512, 1024, 2048(合計 3840)

下流の部品はこのチャネル数を知らないと配線できません。そこで「チャネル数を自動で調べる」仕組みを用意します。優先するのは timm が提供する `feature_info` というメタ情報です。各ステージの `num_chs`(チャネル数)を読めるので、一度も推論を走らせずにチャネル数が分かります。

```python
# feature_info からチャネル数を読む(推論不要)
channels = [info["num_chs"] for info in model.feature_info]
```

もし `feature_info` を持たない独自バックボーンの場合は、ダミー入力 `torch.zeros(1, 3, 256, 256)` を 1 回だけ流して、返ってきた特徴マップの形 `[1, C, H, W]` から `C` を読み取るフォールバック(代替経路)を持たせます。

この「自動発見」のおかげで、ResNet-50(合計 3840)に切り替えても下流のチャネル設定を手で書き直す必要がありません。後段の射影層([FPNネックとチャネル射影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1039_fpn-neck-channel-projection) と [統一埋め込み次元(embed_dim=256)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1051_unified-embed-dim-256))が、合計チャネルを一律 256 に揃えてくれるからです。「レジストリで生成 → チャネル自動発見 → 256 へ射影」の 3 点セットが揃って初めて「どのバックボーンでも下流共通」が完成します。

### 学習スクリプトとの連携

このパターンの最大の恩恵は、学習スクリプトの引数とレジストリを連動させることです。例えば学習スクリプトの `--backbone` 引数の選択肢(choices)をレジストリのキーから動的に取得するようにしておけば、レジストリに新しい行を 1 つ足すだけで、学習スクリプトを一切書き換えずに新しいバックボーンを選べるようになります。実験設定(どのバックボーンで回したか)も文字列として記録に残り、再現性が高まります。

## 手法の系譜と主要論文

レジストリパターン自体は工学的な設計手法で論文化されるものではありませんが、「バックボーンを差し替えて比較する」という営みは知覚・運転研究の標準です。各バックボーンの系譜を押さえます。

- He et al., "Deep Residual Learning for Image Recognition" (ResNet, CVPR 2016, arXiv:1512.03385)。層を深くすると逆に精度が落ちる劣化問題を「残差接続(入力をショートカットして足し戻す)」で解決し、50 層・101 層といった深いネットワークの安定学習を可能にしました。今でも知覚タスクのデファクト標準で、多くの論文がここを基準に数値を報告します。

- Liu et al., "Swin Transformer V2" (CVPR 2022, arXiv:2111.09883)。Vision Transformer に階層構造とウィンドウ注意を導入した Swin の改良版。解像度を変えても安定するよう、対数間隔の連続位置バイアスとスケールドコサイン注意を導入し、低解像度で学習して高解像度で使う転移を改善しました。デメリットは Transformer ゆえの実装の複雑さと、小データでの学習の難しさです。

- Woo et al., "ConvNeXt V2" (CVPR 2023, arXiv:2301.00808)。マスク自己教師あり学習(画像の一部を隠して復元させる事前学習、FCMAE と呼ぶ)と、グローバル応答正規化(GRN)という新しい正規化を組み合わせ、純粋な畳み込みで Transformer 級の性能を達成しました。注意機構が無いぶん実装と高速化が素直です。

時代の流れとしては、(1) CNN の深層化(ResNet 2016)→(2) Transformer の画像への導入と階層化(ViT 2020、Swin 2021)→(3) CNN の現代化による Transformer への対抗(ConvNeXt 2022、ConvNeXt V2 2023)、という「CNN と Transformer の競争と相互模倣」が続いています。レジストリパターンは、この多様な候補を公平に比較・切替するための実務的な土台です。

なお BEV 知覚・運転研究では、より大型・高性能な VoVNet-99(UniAD の大型版で使用、高性能だが重い)なども候補になります。バックボーンの大きさは精度と計算量を直接トレードオフするため、研究では「軽量 Tiny で素早く反復 → 有望なら大型で本番」という使い分けがよく行われます。

## 論文の実験結果(定量データ)

各バックボーンの実力は ImageNet-1K(1000 カテゴリ、約 128 万枚の画像分類ベンチマーク)の Top-1 精度(最も確率が高いと予測したカテゴリが正解と一致した割合、高いほど良い)で比較されます。代表的な報告値を挙げます(およその値)。

- ResNet-50: Top-1 約 76〜80%(学習手法の改良版で 80% 前後まで到達)。約 2560 万パラメータ。長年の基準点です。
- Swin Transformer V2 Tiny: Top-1 約 81〜82%。約 2800 万パラメータ。同規模の ResNet を上回ります。
- ConvNeXt V2 Tiny: Top-1 約 83%(FCMAE 事前学習つき)。約 2800 万パラメータ。Tiny クラスでは最高水準の精度を、純畳み込みで達成しました。

Tiny クラス(パラメータ約 2800 万)という同じ土俵で比べると、新しいバックボーンほど高精度になる傾向が見えます。ConvNeXt V2 の論文は、FCMAE 自己教師あり事前学習と GRN を抜くアブレーションを行い、両方が精度に寄与すること、特に GRN がマスク事前学習との相性で効くことを示しました。

ただし注意点があります。ImageNet 分類の Top-1 精度が高いバックボーンが、自動運転の下流タスク(BEV 検出、軌跡計画)で必ず最良とは限りません。下流での性能は、解像度耐性・マルチスケール特徴の質・事前学習データとの相性などに左右されます。例えば Swin V2 は解像度を上げる転移に強いので、高解像度入力を使う運転知覚では分類精度以上に有利になることがあります。だからこそ「設定ひとつで切り替えて、実際の下流タスクで比較する」レジストリの仕組みが研究上重要になります。比較を支える基盤であって、それ自体が精度を上げる手法ではない、という位置づけです。

## メリット・トレードオフ・限界

メリット。

- 設定の文字列ひとつでバックボーンを切替可能。実験の記録(どの設定で回したか)が残しやすく、再現性が高まります。
- 新バックボーンの追加はレジストリに 1 行足すだけ。学習スクリプトなどの改変が不要です。
- 知らない名前には利用可能一覧付きエラーが出るので、タイプミスに早く気づけます。
- チャネル数の自動発見により、チャネル数の違うバックボーンでも下流を書き換えずに済みます。

トレードオフと限界。

- 「どのバックボーンが今動いているか」が文字列で間接化されるため、コードを追う初心者には一段分かりにくくなります。
- `lambda` での遅延生成や `**kwargs` の素通しは柔軟ですが、引数の取り違えが起きてもエラーが遅れて出ることがあり、原因の特定が難しくなる場合があります。
- フォールバックのダミー forward は入力サイズ(例えば 256×256)やチャネルファースト前提を仮定しており、特殊な独自バックボーンでは破綻し得ます。
- レジストリは比較を楽にするだけで、どれが最適かは教えてくれません。バックボーン選択そのものは依然として実験で決める必要があります。

## 発展トピック・研究の最前線

- 大規模事前学習バックボーン。CLIP や DINOv2 のように、巨大データで自己教師あり/言語対応で事前学習したバックボーンを知覚に転用する研究が増えています。レジストリにこれらを足せば、最新の表現を運転タスクで試せます。
- マルチモーダルバックボーン。カメラだけでなく LiDAR・レーダーの特徴抽出器もレジストリ化し、センサー構成ごと差し替える設計が BEV 融合研究で広がっています。
- ニューラルアーキテクチャ探索(NAS)。バックボーン自体を自動探索する研究もあり、探索結果をレジストリに登録して既存手法と公平に比較する、といった使い方がされます。
- 効率重視のバックボーン。EfficientNet、MobileNet、あるいは蒸留で軽量化したモデルなど、車載のリアルタイム制約に合わせた選択肢が重要です。レジストリは「精度重視の大型」と「速度重視の軽量」を同じインターフェースで切り替える受け皿になります。
