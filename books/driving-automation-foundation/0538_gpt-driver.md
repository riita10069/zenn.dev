# GPT-Driver

## ひとことで言うと
GPT-Driver は、自動運転の「次にどこをどう走るか」を決める処理(これを運動計画、プランニング、あるいは経路計画と呼びます)を、まるで文章を書くタスクのように扱う研究です。具体的には、大規模言語モデル GPT-3.5 に「これから車がたどる座標の列」を文章として出力させます。普通なら数式や専用ニューラルネットで計算するところを、「言葉の続きを予測する AI」に肩代わりさせた点が新しいのです。プランニングを言語モデリング問題として定式化した、という一文に本質が詰まっています。

## 直感的な理解
ベテランの運転教官に助手席で「いま何を考えてどう操作する?」と聞くと、「前の車が止まっているね。右の車線は空いているから、ゆっくり右に寄って追い越すよ」と言葉で説明し、同時にハンドルを切ります。GPT-Driver の発想はこれに近く、「状況を言葉で説明し、その流れで未来の走行ルートも言葉(座標の列)で書き出す」のを、文章生成が得意な LLM にやらせます。

なぜ LLM なのか。従来のプランナには2つの流派があり、どちらも弱点がありました。ルールベース(人間が「赤信号なら止まる」と手書き)は挙動が読みやすく安全ですが、想定外のシーン(工事現場の変則誘導など)に弱い。学習ベース(運転データから機械が学ぶ)は複雑な状況に対応できますが、なぜその軌道を選んだのか説明できないブラックボックスになりがちで、学習に使っていない状況(未知シーン)への汎化が弱い。

ここで著者らが注目したのが LLM です。GPT のような大規模言語モデルは「歩行者が横断歩道にいるなら車は止まるべき」という常識や、それを筋道立てて説明する力を、運転データから学び直さなくても、もともと言葉として持っています。「この常識と説明力をプランニングに使えば、ブラックボックス性も未知シーンへの弱さも和らげられるのでは」という狙いです。

## 基礎: 前提となる概念
用語を順に噛み砕きます。

プランニング(運動計画)とは、周囲の状況(他車、歩行者、信号、自車速度など)を入力に、これから数秒で自車がどの位置を通るべきか、という未来の座標列(軌道、trajectory)を出力する処理です。たとえば「0.5秒後に前へ約5m、1秒後に前へ約10m、わずかに左へ」という点の並びになります。

大規模言語モデル(Large Language Model, LLM)とは、インターネット規模の膨大な文章で訓練され、「これまでの単語列を見て次の単語を確率的に予測する」のが得意な AI です。GPT-3.5 はその代表で、文章生成・常識・論理的説明を得意とします。

言語モデリング(language modeling)とは、まさにこの「これまでの系列を見て次のトークンを予測する」タスクのことです。GPT-Driver の核心は、プランニングをこの言語モデリングの形に書き換えた点にあります。形式的には、軌道を出すという問題を「条件付き言語生成 p(軌道トークン列 | 状況テキスト)」として表現します。

トークン化(tokenization)とは、文章を AI が扱う最小単位(トークン)に区切ることです。重要なのは、座標という数値も文字として扱う点です。座標 `5.27` は `5`,`.`,`2`,`7` のような文字に分解され、LLM はこれを1個ずつ予測して綴ります。普通のニューラルネットが数値を回帰(連続値を直接出力)で出すのに対し、GPT-Driver は数値も言葉として綴ります。

ファインチューニング(fine-tuning)とは、既存の学習済みモデルを、特定タスク向けに追加学習することです。GPT-Driver は GPT-3.5 を、人間の運転ログを正解にして「このプロンプトを見たらこの座標列を出す」よう追加学習します。OpenAI の API 経由で行うため、巨大な GPU を自前で用意せずに済みます。

Chain-of-Thought(CoT、思考の連鎖)とは、答えの前に途中の考えを文章で書かせる手法です(Wei et al., NeurIPS 2022)。中間推論を挟むと多段の判断の質が上がり、理由も追える利点があります。

## 仕組みを詳しく
GPT-Driver の処理は、プランニングを言語モデリングに対応づけることで成り立ちます。

入力(プロンプト = AI への指示文)を作る: 知覚システム(カメラや LiDAR で周囲を認識する部分)が出した結果を、すべて言葉に変換します。

- 自車の状態: 現在の速度、加速度、進行方向(ヘディング)。
- 周囲の物体: 「前方 12.3m、横 -2.1m に車があり、速度は秒速 8m」のように、物体ごとの位置・速度を列挙。物体の過去軌跡も含めます。
- 地図情報: 車線の境界、信号の有無など。

LLM に出力させる: GPT-3.5 に「この状況で安全な運転計画を立て、まず状況を分析し、次に未来の軌道座標を出力せよ」と頼みます。出力は2部構成です。

1. CoT による推論文: 「前方に車がいて減速している。右側に歩行者はいない。よって緩やかに減速しながら右に寄って追い越すべきだ」のような説明。
2. 軌道座標: たとえば `(x1, y1), ..., (x6, y6)` の6点(将来3秒ぶんを0.5秒刻みで6点。これは nuScenes 評価の慣習)を、テキストとして出力。

```
入力プロンプト(知覚結果をテキスト化):
  "自車速度 6.0 m/s。前方 10m に静止車両。右車線は空き。"
        │
   GPT-3.5(言語モデリングとして次トークンを予測)
        │
   CoT 推論文を生成:
     "前方車が静止。このまま進むと衝突。右車線が空いているので右に寄る。"
        │
   続けて軌道座標を文字として生成:
     "(0.0,6.0),(1.5,6.1),(3.0,6.2),..."  ← 右へ寄りつつ前進する6点
```

数値を文字トークンで綴るという設計が肝です。回帰で直接数値を出すのではなく、桁を1文字ずつ予測する。この統一により、推論文も座標も同じ言語生成として扱え、CoT の恩恵(考えてから書く)を軌道生成に直結できます。座標の小数点や桁を文字として再利用するため、LLM が学習済みに持っている「数の表記」の知識をそのまま活かせます。

学習では、人間の運転ログ(実際に走った軌道)を正解とし、「このプロンプト → この推論文 + 座標列」を出すよう GPT-3.5 をファインチューニングします。LLM がもともと持つ言語知識の上に運転を載せるため、比較的少ないデータでも素早く適応できる(データ効率がよい)とされます。報告では、全データの 1〜10% 程度の少量でも、ゼロから学ぶ専用モデルに迫る計画性能を出せたとされています。

## 手法の系譜と主要論文
GPT-Driver は、LLM を運転計画に持ち込む初期の代表例として位置づけられます。

直接の比較対象は UniAD (Hu et al., CVPR 2023, Best Paper, arXiv:2212.10156) です。UniAD は知覚・予測・計画を1つのニューラルネットに統合した end-to-end 手法で、当時の planning の強力なベースラインでした。同系統で、ベクトル化シーン表現で高速化した VAD (Jiang et al., ICCV 2023) も主要な比較対象です。GPT-Driver はこうした専用設計の強敵に対し「LLM でも肩を並べられる」ことを示すために比較しました。[UniAD (Planning-oriented統合E2E)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1090_uniad_note)。

Chain-of-Thought (Wei et al., NeurIPS 2022) を、軌道生成の前段に組み込んだ点も系譜上重要です。汎用の中間推論テクニックを運転計画に適用し、推論文 → 座標という2段出力にしました。

GPT-Driver の発想は、後続研究に引き継がれます。同じ著者グループの Agent-Driver (Mao et al., 2023, arXiv:2311.10813) は、GPT-Driver の「全情報を一括テキスト化」「経験を蓄えられない」という限界を、ツール使用・記憶・多段推論を持つエージェントへ拡張して克服しました([Agent-Driver](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0526_agent-driver))。さらに EMMA (Hwang et al., 2024) は、軌道をテキストで出すという発想を、知覚まで含めて巨大な基盤モデルに統合する方向へ推し進めました([EMMA (End-to-End Multimodal Model for Autonomous Driving)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0536_emma_note))。「軌道を言葉で綴る」という GPT-Driver の核は、これらに共通の遺伝子になっています。

GPT-Driver (Mao, Qian, Wang, Ye, Zhao, Wang, 2023, arXiv:2310.01415) の新規性をまとめると、(1) motion planning を言語モデリングとして定式化し、(2) GPT-3.5 をプランナとしてファインチューニングで使い、(3) CoT 推論を軌道生成の前段に組み込んで解釈性を持たせた点にあります。

## 論文の実験結果(定量データ)
評価は nuScenes(都市走行の標準ベンチマーク)の planning タスクで行われました。指標の意味から説明します。

L2 誤差は、計画した軌道と人間が実際に走った軌道との、各時刻でのユークリッド距離(メートル単位のずれ)の平均です。小さいほど人間の運転に近いことを意味します。通常は将来1秒・2秒・3秒先での値とその平均で報告されます。1秒先は小さく、3秒先は予測が難しく大きくなる傾向があります。

衝突率(collision rate)は、計画した軌道が周囲の他物体(他車・歩行者など)とぶつかる割合です。パーセントで表され、小さいほど安全な計画ということになります。

GPT-Driver は、これらの指標で、当時の代表的な学習ベース手法(UniAD を含む)と同等かそれ以上の性能を達成したと報告しています。報告された平均 L2 誤差はおよそ 0.4m 前後、平均衝突率はおよそ 0.1〜0.2% 程度の水準で、専用設計の end-to-end ニューラルネット UniAD に匹敵またはそれ以下に抑えたとされます。すなわち、汎用 LLM のファインチューニングだけで、専用パイプラインに並ぶ計画性能を出せたことになり、これが「言語モデルでもプランニングが解ける」という主張の定量的裏づけです。

加えて重要なのが、データ効率と汎化に関する知見です。LLM がもともと持つ言語・常識の事前知識を活用するため、訓練データのごく一部(数%)でファインチューニングしても妥当な計画が出せる(few-shot 的に適応しやすい)こと、そして学習時に見ていない状況に対しても、常識に基づいた説明とともに妥当な軌道を出しやすいことが示されました。

CoT を入れることの効果も、定性的な事例とともに示されています。推論文を先に出させると、「なぜその軌道にしたか」を自然言語で説明でき、人間が計画の妥当性を検証しやすくなります。一方で、座標を文字で綴るがゆえに、まれに桁の取り違えなど数値出力ミスが起こりうることも、この方式の固有の弱点として認識されています。なお、nuScenes の open-loop planning 指標は自車の過去状態を入れるだけで L2 が小さく見える「ショートカット」が存在することが後年指摘されており(直進が多いデータでは等速直進予測が強い)、GPT-Driver の数値もこの指標特性の上で読む必要があります。

## メリット・トレードオフ・限界
メリット

- LLM の常識・推論力を流用でき、未知シーンへの汎化が期待できる。
- 「なぜこの軌道か」を自然言語で説明でき、解釈性が高い。
- API のファインチューニングで済み、巨大な自前学習基盤が不要。
- 言語の事前知識のおかげで、少量データ(数%)でも素早く適応しやすい(データ効率がよい)。
- プランニングと言語生成を統一でき、CoT の恩恵を軌道生成に直接持ち込める。

トレードオフ・限界

- 座標を文字トークンで綴るため出力が長く、推論が遅い。高頻度制御が要る車載リアルタイムには厳しい。
- 知覚結果をテキストに変換する段階で情報が落ちる。生のピクセルや点群の細部、微妙な手がかり(歩行者の視線、ブレーキランプ)は伝わらない。知覚モジュールの誤りもそのまま入力に混入する。
- クラウド API への依存があり、再現性やモデルのバージョン変更のリスクを抱える(API モデルが更新されると挙動が変わりうる)。
- 数値を文字で綴るため、桁の取り違えなど言語モデル特有の数値出力ミスが起こりうる。専用回帰ヘッドより数値精度で劣る場面がある。
- 評価は主にオフライン(記録データでの open-loop 予測)で、しかもその指標自体にショートカットがあるため、実車の閉ループ性能・長期安全性の実証は別途必要。

研究上の未解決課題としては、知覚のテキスト化で失われる情報をどう補うか(VLM 化)、座標の数値精度と生成速度をどう上げるか、open-loop 指標のショートカットをどう避けるか、そして閉ループでの安全性をどう保証するかが挙げられます。

## 発展トピック・研究の最前線
GPT-Driver が開いた「LLM をプランナにする」流れは、2つの方向に発展しました。1つは、LLM を1問1答の生成器から、ツール・記憶・多段推論を持つエージェントへ拡張する方向([Agent-Driver](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0526_agent-driver) [DiLu](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0531_dilu_note))。もう1つは、知覚も含めて巨大な基盤モデルに統合する方向([EMMA (End-to-End Multimodal Model for Autonomous Driving)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0536_emma_note))です。

また、知覚をテキストに落とす情報損失を避けるため、画像特徴を LLM に直接食わせる VLM ベースの手法([DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm))や、推論を構造化して評価・学習するベンチマーク([DriveLM (Graph VQA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0534_drivelm))が整備されてきました。実用面では、賢いが遅い LLM の出力を軽量モデルに蒸留し、車載で動く速さにする研究が活発で、「賢さ」と「速さ」の両立が当面の主戦場になっています。さらに、open-loop の L2 偏重への反省から、閉ループ・反事実シナリオでの評価へと軸足が移りつつあります。
