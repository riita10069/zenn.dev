# 履歴・未来ウィンドウによるサンプル列挙

## ひとことで言うと
時系列の予測モデルが学ぶ「1サンプル」は、ある瞬間だけでなく「その前の過去 H フレーム」と「その後の未来 F フレーム」をセットで必要とします。だから過去も未来も十分に揃っている瞬間だけを有効なサンプルとし、それをデータセット作成時にすべて数え上げておく(全列挙する)設計です。これにより、データ取得が必ず成功し、データ件数も即座に分かるようになります。

## 直感的な理解
動画から「この人が次にどう動くか」を予測する学習を考えます。予測には「直前の数秒の動き(履歴)」を見せ、正解として「直後の数秒の動き(未来)」を与えます。ところが動画の最初の数秒には「直前の履歴」が存在せず、最後の数秒には「直後の未来」が存在しません。これらの端のフレームを無理に使おうとすると、配列の範囲外にアクセスしてクラッシュするか、存在しないデータをゼロで埋めて嘘の入力をモデルに食わせることになります。

そこで「過去 H フレームと未来 F フレームが両方そろう瞬間だけ」を有効サンプルと定義します。さらに賢いのは、それを学習のたびに毎回判定するのではなく、データセットを作る瞬間(構築時、construction time)に「どの動画のどのフレームが有効か」を全部リストアップしておくことです。こうすれば、データ件数(`len(dataset)`)が起動直後に確定し、i 番目のサンプルを取りに行けば必ず有効なものが返ってくる。学習基盤(PyTorch の DataLoader など、データをバッチにまとめて供給する仕組み)が素直に動きます。

## 基礎: 前提となる概念

時系列とフレームレート。連続したセンサーデータ(映像や走行ログ)は、一定間隔で記録された「フレーム」の列です。フレームレートが 10Hz(1秒間に10フレーム)なら、64フレームはちょうど 6.4 秒に相当します。フレーム数で窓を切ると、フレームレートを通じて秒数に換算できます。

履歴ウィンドウと未来ウィンドウ(history / future window)。予測モデルへの入力として使う過去の区間が履歴ウィンドウ(長さ H フレーム)、正解として与える未来の区間が未来ウィンドウ(長さ F フレーム)です。たとえば軌跡予測なら、履歴に「過去の速度・加速度・向き・曲率の列」、未来に「これから取るべき加速度・曲率の列」を入れます。

スライディングウィンドウ(sliding window、ずらし窓)。連続系列を「入力窓 → 出力窓」の対に切り出して教師あり学習のサンプルにする古典的な前処理です。窓を1フレームずつずらすことで、1本の長い系列から多数の学習サンプルを作れます。

PyTorch Dataset の契約。学習用データセットは普通 `__len__`(全件数)と `__getitem__(i)`(i 番目のサンプルを返す)の2つを実装します。DataLoader はこの2つを使ってシャッフル・分割・並列読み込みを行います。`__getitem__` が範囲外で落ちたり、件数が不定だったりすると、この仕組み全体が壊れます。だから「件数が確定し、どの i も必ず有効」という保証が重要になります。

## 仕組みを詳しく

まず窓長と最小フレーム数を定数で決めます。10Hz で過去・未来とも 6.4 秒を取りたいなら、次のようになります。

```
H (history)  = 64    # 過去 6.4 秒
F (future)   = 64    # 未来 6.4 秒
MIN_ROWS = H + F + 1 = 129
```

`MIN_ROWS = 129` は「過去64 + 現在1 + 未来64」で、たった1つでも有効サンプルを作るのに最低限必要なフレーム数です。129フレーム未満の短い系列は1サンプルも作れないので、丸ごとスキップします。

次に、1つの系列(シーン、走行区間)の中で有効なフレームの範囲を計算します。

```python
usable  = min(len(poses), num_frames)   # ポーズと映像が両方ある区間に制限
min_idx = H                              # = 64。これより前は過去が足りない
max_idx = usable - F - 1                 # これより後は未来が足りない
# 有効サンプル = frame_idx in [min_idx, max_idx]
samples = [(scene_id, f) for f in range(min_idx, max_idx + 1)]
```

数値例。ある系列に 200 フレームある(usable = 200)とします。

- min_idx = 64。フレーム 0〜63 は過去64フレームが取れないので除外。
- max_idx = 200 - 64 - 1 = 135。フレーム 136〜199 は未来64フレームが取れないので除外。
- 有効サンプルは frame_idx = 64, 65, ..., 135 の 72 個。

```
フレーム:  0 ......... 63 | 64 .......... 135 | 136 ........ 199
            └─ 過去不足 ─┘   └── 有効サンプル ──┘  └─ 未来不足 ─┘
                              (各点の前64・後64が揃う)
```

構築時には、各系列に対して妥当性チェックもまとめて行います。(1) 必要なセンサー(カメラ等)が全部そろっているか。欠けていれば系列ごとスキップ。(2) ポーズ(自車の位置・向きの列)が MIN_ROWS 以上あるか。足りなければスキップ。(3) センサーのフレーム数とポーズ数は末尾で食い違うことがあるので、両方がカバーする区間 usable = min(...) に切り詰める。

そして全系列をループして、有効な `(scene_id, frame_idx)` のペアを1本のフラットなリストに積み上げます。`__len__` はこのリストの長さ、`__getitem__(i)` はリストの i 番目を取り出して読み込むだけ、という形になります。

防御的チェックの二重化。窓を切り出す関数の側でも、渡された frame_idx が有効範囲内かを再チェックし、外れていればエラーを投げる実装がよく見られます。一見冗長ですが、「うっかり端のフレームを渡す」バグが静かに通り抜けて、ゼロ埋めの嘘データで学習が進んでしまう事故を防ぐための安全策です。

入出力テンソルの形。履歴に4種類の量(速度・加速度・yaw・曲率)を 64 フレーム分入れると、平坦化して 256 次元(64 x 4)になります。未来に2種類の量(加速度・曲率)を 64 フレーム分入れると 128 次元(64 x 2)になります。窓長を変えるとこの次元も変わるため、モデルの入力・出力層の形と窓長は一貫している必要があります。

## 手法の系譜と主要論文

「履歴ウィンドウ + 未来ウィンドウ」という時系列の切り出しは、特定論文の新規手法というより、軌跡予測・行動予測の分野で共有される標準的な実験設定です。

Alahi et al., "Social LSTM: Human Trajectory Prediction in Crowded Spaces" (CVPR 2016)。歩行者の軌跡予測で「過去の観測フレーム列を入力し、未来のフレーム列を予測する」枠組みを LSTM(再帰型ニューラルネット)で定式化しました。観測長と予測長を固定する考え方の代表例です。

Chang et al., "Argoverse: 3D Tracking and Forecasting with Rich Maps" (CVPR 2019, arXiv:1911.02620)。自動運転の動き予測ベンチマークで、過去2秒を観測し未来3秒を予測する、という固定窓で問題を定義しました。固定窓にする理由は、入出力の形を揃え、手法どうしを公平に比較できるようにするためです。これにより評価が再現可能になります。

Caesar et al., "nuScenes: A Multimodal Dataset for Autonomous Driving" (CVPR 2020, arXiv:1903.11027)。付随する予測タスク(nuScenes prediction)でも過去・未来を秒単位で固定する設計を採り、業界標準のひとつになりました。

Lee et al., "DESIRE" (CVPR 2017) や Gupta et al., "Social GAN" (CVPR 2018) も、観測8フレーム(過去3.2秒)・予測12フレーム(未来4.8秒)という固定窓(ETH/UCY 歩行者ベンチマーク、2.5Hz)を採用しており、軌跡予測分野では「固定の観測長・予測長で問題を切る」のが完全に標準化されています。この 8/12 という設定は今も多くの論文が比較のために踏襲しています。

これらに共通する設計思想を一般化したものが本トピックです。窓を1フレームずつずらして全列挙すれば、隣接サンプルは H + F - 1 フレーム(上の例なら 127 フレーム、現在を中心とすると 63 フレーム以上)重複します。スライディングウィンドウのトレードオフがそのまま当てはまります。ベンチマークによっては重複を避けるため、固定間隔(stride)でサンプルを間引いて切り出す(たとえば過去2秒・未来3秒の窓を 1 秒ごとに1サンプルだけ取る)設計も使われ、これは過学習を抑える一方でサンプル数を減らすトレードオフがあります。

## 論文の実験結果(定量データ)

窓長そのものはモデルの精度に直接効きます。軌跡予測の標準指標を先に説明します。ADE(Average Displacement Error、平均変位誤差)は予測軌跡と正解軌跡の各時刻の位置の差を全時刻で平均した値(メートル単位)で、小さいほど良い。FDE(Final Displacement Error、最終変位誤差)は予測の最終時刻だけの位置誤差で、こちらも小さいほど良い。

予測ホライズン(未来窓の長さ)を伸ばすと FDE は単調に増加します。Argoverse 系の手法では、未来3秒予測で minADE/minFDE(複数候補のうち最良の軌跡で測る指標)がおよそ minFDE 1〜2 メートル前後(上位手法)に収まる一方、予測を長くするほど誤差が累積的に増えることが一貫して報告されています。具体例として、ETH/UCY ベンチマーク(観測8/予測12)では、Social GAN が未来4.8秒で ADE 約 0.58m・FDE 約 1.18m(平均、報告による)を示し、後続の Trajectron++(Salzmann et al., ECCV 2020)はさらに ADE 約 0.19m 前後まで下げています。同じ固定窓だからこそこうした手法間の直接比較が成立します。予測を長くするほど誤差が累積的に増えるのは「遠い未来ほど予測が難しい」という直感を定量的に裏づけます。逆に履歴(観測窓)を短くしすぎると、加速・減速の傾向や曲がりの文脈が読めず精度が落ちます。Social LSTM 系の研究では、観測長を短くすると ADE/FDE が悪化する傾向が示されています。

アブレーション的な観点として、窓を1フレームずつずらして重複サンプルを大量に作る方式は学習データ量を稼げる一方、隣接サンプルがほぼ同一なので、実効的なデータの多様性は見かけの件数ほど増えません。重複の強い系列で過学習(同じような場面を覚えすぎる)が起きやすいことが、データ拡張やサンプリング間隔を空ける研究の動機になっています。

## メリット・トレードオフ・限界

メリット。第一に、i 番目のサンプルが必ず有効なものを返すので、範囲外アクセスやゼロ埋めの嘘データが混入しない。第二に、全件数が起動直後に確定し、DataLoader のシャッフル・学習/検証分割・進捗管理が素直になる。第三に、系列の妥当性チェック(センサー欠損・ポーズ不足)を構築時に1回で済ませられ、学習ループ中の例外が減る。

トレードオフ・限界。各系列の先頭 H フレームと末尾 F フレームは必ず捨てます。短い系列ほど捨てる割合が大きく、MIN_ROWS 未満の系列は丸ごと捨てます。窓長が固定なので、より長い文脈やより短い反応を見たい場合は窓長を変える必要があり、変えると有効サンプル数も入出力次元も変わります。隣接サンプルが大きく重複するため情報が冗長です。全列挙は系列数が膨大だと、構築時の前処理(ポーズ読み込みと妥当性チェック)に時間とメモリがかかります。

研究上の論点として、固定窓の代わりに可変長の文脈を扱えるアーキテクチャ(Transformer で任意長の履歴を入れる、など)や、重複を間引いて多様性を保つサンプリング戦略、長期予測のための階層的な窓設計などが検討されています。

## 発展トピック・研究の最前線

近年の動き予測では、固定の数値窓ではなく「シーングラフ」や「ベクトル化表現」(VectorNet, Gao et al., CVPR 2020)で文脈を可変長に扱い、Transformer 系(WayFormer, Nayakanti et al., ICRA 2023 など)で長い履歴と複数エージェントを同時に処理する流れが主流です。それでも、生データから学習サンプルを切り出す最初の段階では、本トピックのような「有効区間の全列挙」が信頼できる土台として使われ続けています。
