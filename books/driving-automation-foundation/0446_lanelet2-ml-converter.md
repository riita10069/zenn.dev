# HD地図のML向けポリライン変換 (lanelet2_ml_converter)

## ひとことで言うと
高精細地図(HD地図)を、機械学習モデルがそのまま入力できる形に整える前処理ツールです。具体的には、地図の車線境界線などを「自車を中心にした座標」「いつも同じ点数」「自車の周りだけ切り出す(ROIクロップ)」という3つのルールでポリライン(折れ線)に変換します。`lanelet2_ml_converter` はその一例で、オープンソースの自動運転スタックが提供する、検証済みの既製パイプラインです。自前で書くと面倒でバグも入りやすいこの定型処理を、まるごと任せられるのが価値です。

## 直感的な理解
HD地図の生データは、人間が地図ソフトで見るためや、厳密な経路計算のために作られていて、ニューラルネットに食べさせる形にはなっていません。生のままモデルに渡そうとすると、3つの困りごとが起きます。

第一に、地図はワールド座標(地図全体の絶対的な座標)で書かれていて、自車がどこにいるかに依存しません。しかしモデルが知りたいのは「自車から見て、この境界線がどこにあるか」です。絶対座標のままでは、同じ交差点でも自車の位置や向きが違うと数値がまるごと変わり、学習が安定しません。

第二に、各車線の境界線を構成する点の数がバラバラです。ある車線は10点、別の車線は53点、というように不揃いだと、固定サイズの入力を期待するニューラルネットにそろえて入れられません。

第三に、地図全体は広大で、自車周辺の数十メートルだけが運転に関係します。地図全体を渡すのは計算もメモリも無駄です。

これらを毎回自前で実装するのは手間で、座標変換の符号ミスや補間のバグが入りやすい。そこで「HD地図を ML 入力用に整える」処理を、検証済みの既製パイプラインとして用意したものが本トピックのツールです。既製であること自体が価値で、定型処理を信頼できるコードに任せて、研究はモデル本体に集中できます。

## 基礎: 前提となる概念
HD地図(High-Definition map、高精細地図)は、車線の境界線・中心線、停止線、横断歩道などをセンチ単位で記録した地図です。Lanelet2(Poggenhans et al., ITSC 2018)はその代表的なフォーマットで、多くのオープン自動運転スタックが標準採用しています。Lanelet2 では地図を「lanelet(レーンレット、左右の境界線で囲まれた車線の小片)」の集まりとして表します。各 lanelet は左右の境界線(点列)とその間の通行ルールを持ちます。

「ポリライン(polyline)」は、点を順につないだ折れ線です。車線境界線や中心線は本質的に折れ線なので、ポリラインで表すのが自然です。

「ベクトル地図(vectorized map)」は、地図要素を点・線・多角形といった幾何プリミティブの集合で表す方式です。これに対し「ラスタ地図(rasterized map)」は地図を画像(ピクセルの格子)に描く方式です([ラスタ地図とベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0433_rasterized-vs-vectorized-maps))。本トピックの変換は、地図をベクトル(ポリライン)として整えるものです。

「自車中心(ego-centric)座標」は、自車を原点、進行方向を一つの軸にとった相対座標系です。「ROI(Region of Interest、関心領域)」は、自車周辺の決まった範囲を指します。

## 仕組みを詳しく
主な変換は3つです。順に、なぜそうするかの理由とともに見ます。

第1の変換、自車中心への座標変換。ワールド座標の境界線を、自車を原点とする座標系に移します。自車の位置と向き(姿勢、pose)を使って、すべての地図点を「自車から見た相対位置」に回転・並進します。

```
before(ワールド座標): 点が (139.7651, 35.6812) のような絶対座標(あるいは UTM メートル)
after (自車中心)    : 自車を原点 (0, 0)、前方が X 軸、左が Y 軸として
                      (12.4, -3.2) = 前方 12.4 m、右 3.2 m のような相対座標
```

これにより、地図が世界のどこにあっても、モデルは常に「自車基準」の一貫した見え方で学習・推論できます。世界の絶対位置に対する不変性(translation/rotation invariance)を入力段で確保する、という設計です。

第2の変換、固定点数へのリサンプリング。不揃いな点数の境界線を、決まった点数(たとえば各ポリライン20点)に揃えます。点が少ない線は補間して増やし、多い線は間引いて減らします。

```
before: 境界線 A = 10 点, 境界線 B = 53 点(不揃い)
after : 境界線 A = 20 点, 境界線 B = 20 点(揃った)
```

これでバッチ処理(複数サンプルをまとめて一つのテンソルにする)が可能になります。形でいえば `[ポリライン数, 点数, 2]`(末尾の 2 は x, y)のような揃った形になります。リサンプリングは弧長(線に沿った距離)に対して等間隔に行うのが一般的で、曲率の高い区間で点が足りなくならないよう工夫されることもあります。

第3の変換、ROIクロップ。自車周辺の決まった範囲(たとえば前後左右数十メートルの矩形)だけを残し、その外の地図要素を捨てます。

```
before: 地図全体の数千本の境界線
after : 自車周辺 ROI 内の数十本だけ
```

無駄な計算とメモリを削り、運転に関係する範囲に集中できます。ROI の大きさは、後段の BEV 範囲(`pc_range`)と合わせるのが筋で、これが合っていると地図 BEV と画像 BEV の座標整合([地図BEVと画像BEVの座標整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0430_map-image-bev-alignment))が取りやすくなります。

これらを通すと、HD地図が「自車中心・固定点数・ROIクロップされたポリライン集合」という、ベクトル地図エンコーダ(VectorNet 系など)にそのまま入る形になります。後段のエンコーダは、各ポリラインを点列のトークンとして受け取り、点内・ポリライン間の関係をグラフや attention で集約していきます([ベクトル地図のポリライン符号化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0465_vectorized-map-polyline-encoding))。

## 手法の系譜と主要論文
この前処理が前提とするベクトル地図エンコーダの系譜を追います。

起点は VectorNet(Gao et al., CVPR 2020, arXiv:2005.04259)です。地図や物体を「ポリラインの集合」として表し、各ポリラインをサブグラフで符号化し、ポリライン間をグローバルグラフで関係付けます。ラスタ画像にせずベクトルのまま扱う動機は、車線などは元々折れ線であり、画像化すると解像度の無駄と情報損失が出るからです。効率と精度を両立し、以後のベクトル地図手法の基礎になりました。自車中心・固定点数という入力規約も、この系統で標準化されました。

VectorMapNet(Liu et al., 2022, arXiv:2206.08920)は、地図要素をポリラインとして「オンラインで推定する」(地図を持たずカメラから車線を直接ベクトルで出力する)方向へ進めました。地図をベクトルで出す・入れる、という表現の統一が進みます。

MapTR と MapTRv2(Liao et al., ICLR 2023 / 2024, arXiv:2208.14437)は、地図要素を「点の順序の曖昧さに頑健な集合」として表現し、Transformer で並列にオンライン地図を構築します。固定点数のポリラインで地図要素を表す、という入力・出力規約をさらに洗練させました。

VAD(Vectorized Autonomous Driving、Jiang et al., ICCV 2023, arXiv:2303.12077)は、地図と物体をベクトル表現で統一し、エンドツーエンドで計画まで行います。ベクトル地図を自車中心の整った形で入れる前提を共有し、本トピックの前処理が出力する形がそのまま使われます。

これらに共通するのは、自車中心・固定点数・ROIクロップという入力規約が、特定の論文の発明というより、ベクトル地図を扱う分野の標準作法だという点です。本トピックのツールは、その標準作法を Lanelet2 形式向けに既製化したものです。Lanelet2 自体は Poggenhans et al.(ITSC 2018, "Lanelet2: A high-definition map framework for the future of automated driving")が提案した地図フレームワークで、車線・規制・関係を一貫したデータモデルで表します。

## 論文の実験結果(定量データ)
ベクトル地図を整った形で使うことの効果は、上記系譜の論文で定量的に示されています。指標の意味を添えて見ます。

オンラインベクトル地図構築では、mAP(mean Average Precision、地図要素を正しい位置・形で当てられたかの平均精度。高いほど良い)が主指標です。MapTR は nuScenes のオンライン地図構築で、従来のラスタベースやポリライン逐次生成の手法を大きく上回る mAP を、推論速度も保ったまま達成したと報告されています。ポリラインの点の順序を集合として扱う設計(順序の曖昧さに罰を与えない)が効いており、アブレーションでこの設計を外すと mAP が落ちることが示されています。これは「地図要素をどう表すか」が精度に直結する例です。

VAD は、計画(自車の将来軌跡の予測)の評価で、ベクトル地図表現を使うことで、ラスタ地図を使う重い手法に匹敵する計画精度を、はるかに速い推論速度で達成したと報告しています。計画の指標としては、予測軌跡と正解軌跡のずれ(L2 誤差、メートル単位。小さいほど良い)や衝突率(collision rate、計画した軌跡が他物体とぶつかる割合。小さいほど良い)が使われます。VAD はこれらをラスタ手法と同等以下に抑えつつ、ベクトル表現の軽さで実時間性を稼ぎました。前処理段で地図を自車中心・固定点数に整えておくことが、この効率の前提になっています。

これらの数値の意味を初心者向けに補うと、地図構築の mAP が高いとは「車線や境界を正しい場所と形で認識できる」こと、計画の L2 誤差や衝突率が低いとは「安全で人間らしい運転計画を出せる」ことです。入力の地図がきれいに整っているほど、これらの後段タスクは安定して高い性能を出します。逆に、座標がバラバラ・点数が不揃いのまま入れると、学習が不安定になり、これらの指標は劣化します。前処理は地味ですが、後段性能の土台です。

## メリット・トレードオフ・限界
メリットは明快です。HD地図を、ベクトル地図エンコーダ向けの整った形(自車中心・固定点数・ROIクロップ)に一発で変換できます。車線抽出・座標変換・点数正規化・切り出しを自前実装せずに済み、バグと工数を減らせます。Lanelet2 を使う既存の自動運転スタックと統合しやすく、自車中心化により地図の絶対位置に依存しない一貫した入力が得られます。

トレードオフと限界もあります。第一に、Lanelet2 形式が前提で、OpenDRIVE など他形式には別の変換が要ります。第二に、固定点数へのリサンプリングは、点を間引く際に細かい曲率情報を失う可能性があります。急カーブを少ない点で表すと、形が直線的になまってしまう、という典型的な失敗です。第三に、ROIクロップで範囲外を捨てるため、遠方の地図情報(この先の合流や分岐)は見えなくなります。第四に、そもそも HD地図が整備された地域でしか使えず、HD地図の整備コストという根本問題は解消しません。だからこそ、無料で全世界にある SD地図を使う [SDTagNet SD地図エンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0462_sdtagnet-sd-map-encoder) と役割分担になります。研究上は、点数を固定しつつ曲率を保つ適応的サンプリングや、HD地図と SD地図をシームレスに統合する前処理が、未解決の方向です。

## 発展トピック・研究の最前線
前処理段の発展として、点数を一律固定するのではなく、要素の複雑さに応じて点を割り当てる適応的リサンプリングや、ベジエ曲線やスプラインで境界を表して少ない制御点で滑らかさを保つ表現が研究されています。また、HD地図を毎回前処理して入れるのではなく、走行ログからオンラインで地図要素を推定し、既存の HD地図と融合する「地図の自己更新」も活発です。地図を持たない地域への一般化、地図と知覚の不一致検出、そして地図前処理の ROI と後段 BEV の `pc_range` を厳密に一致させて座標整合を保証する設計が、実用面での重要テーマとして残っています。
