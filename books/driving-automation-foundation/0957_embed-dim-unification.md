# 隠れ次元の統一（embed_dim=256 という設計判断）

## ひとことで言うと
深層学習モデルの中をデータが流れていくとき、各部品が扱う「特徴ベクトルの太さ(次元数)」を全部同じ値(よく使われるのは256)に揃える、という設計上の決め事です。太さが揃っていると、部品Aの出力をそのまま部品Bの入力につなげられます。揃っていないと、つなぐ場所ごとに変換器を挟む必要が出てきて、コードが複雑になり壊れやすくなります。知覚・予測・計画を連結するような自動運転モデルのようなマルチモジュール構成では、この「共通次元」が部品どうしを接続する前提条件になります。

## 直感的な理解
水道管を想像してください。直径の違うパイプをつなごうとすると、つなぎ目ごとに「太さを変えるアダプタ」が必要になります。アダプタが増えるほど水漏れ(バグ)の箇所が増え、しかもどのアダプタがどの直径だったかを全部覚えておかないといけません。最初から全部のパイプを同じ直径で作れば、アダプタは入口で1個あれば足りて、あとはまっすぐつながります。

ニューラルネットの「特徴ベクトルの太さ」がパイプの直径に相当します。embed_dim(embedding dimension、埋め込み次元)とは、特徴を表現するベクトルの長さのことです。これを全部品で同じ値に固定すれば、つなぎ目のアダプタ(変換層)が要らなくなります。

## 基礎: 前提となる概念

次元(dimension)とベクトル。ニューラルネットは画像やセンサ入力を「数字の並び(ベクトル)」に変換して処理します。このベクトルの要素数を次元と呼びます。長さ256のベクトルは256次元で、各位置に1つの実数(例 0.37, -1.2, ...)が入っています。直感的には「特徴を表す座標の数」で、次元が多いほど多くの情報を詰め込めますが、その分だけ計算とメモリが重くなります。

層の入出力次元は固定。深層学習の層(全結合層, 畳み込み層など)は、入力ベクトルの長さと出力ベクトルの長さがあらかじめ決まっています。「長さ256の入力を受け取り長さ512を出す層」に、間違って長さ384のベクトルを入れると、形状(shape)が合わず実行時エラーで止まります。この「形が合わないと動かない」性質が、次元を揃える動機の根本です。

残差接続(residual connection)。多くの現代的なネットワークは、層の出力に層の入力をそのまま足し込みます(`out = layer(x) + x`)。これは勾配が深い層まで届きやすくなる重要な工夫ですが、足し算をするには入力と出力の次元が一致していなければなりません。残差接続を使う限り、その経路の次元は最初から最後まで同じである必要があります。これが「全体で一定の次元を保つ」ことを技術的に要請します。

tensor shape の表記。本稿では特徴マップを `(バッチ, チャネル, 高さ, 幅)` の4次元で書きます。バッチは一度に処理する画像枚数、チャネルが特徴ベクトルの太さ(=embed_dim)に相当します。Transformer 系ではトークン列を `(バッチ, トークン数, 特徴次元)` の3次元で書き、最後の次元が embed_dim です。

## 仕組みを詳しく

「揃っていない世界」と「揃った世界」を具体的なtensor shapeで比べます。

揃っていない世界(悪い例):
```
backbone 出力     : (8, 768, 8, 8)
  -> 変換層が必要  : 768 -> 512
view fusion 入力  : (8, 512, 8, 8)
  -> 変換層が必要  : 512 -> 256
planner 入力      : (8, 256, ...)
```
つなぎ目が2か所あり、それぞれに専用の変換層と「768」「512」という個別の数を覚えておく必要があります。backbone を別の種類に替えると 768 が 1024 になり、変換層の定義を全部直す羽目になります。

揃った世界(共通次元 256):
```
backbone 出力(連結) : (8, 1440, 8, 8)   ← マルチスケール連結後の生チャネル数
  -> 入口の射影      : 1440 -> 256      ← 入口で1度だけ 256 に射影
view fusion 入力    : (8, 256, 8, 8)
view fusion 出力    : (8, 256, H, W)
planner 入力        : 256
```
backbone の生出力を入口で1度だけ 256 に落とせば、その後は最後まで 256 のまま流れます。つなぎ目に変換層を挟む必要がありません。backbone を替えても「入口の射影で 256 に落とす」だけなので、下流は無影響です。

実装上は、各部品のコンストラクタのデフォルト引数として 256 を置きます。

```python
def build_view_fusion(fusion_mode, num_views, embed_dim=256, **kwargs): ...
def __init__(self, num_views=8, backbone_channels=1440, embed_dim=256, ...): ...
```

256 という値の選び方。これは「表現力(大きいほど多くの情報を保持できる)」と「計算量・メモリ(大きいほど重い)」のトレードオフで決まる中間的な大きさです。Transformer 系では 256 / 512 / 768 / 1024 がよく使われます。Self-Attention の計算量は次元 d に対しておおむね線形(投影は d^2、注意行列は系列長 N に対し N^2 で効く)、メモリは特徴を保持する箇所すべてで d に比例します。GPU メモリと速度に厳しい制約がある場合(高解像度の特徴を多数のカメラ分保持する自動運転など)、512 や 768 では収まらず、256 が現実的な選択になることがよくあります。次元を半分にすると、その特徴を保持するメモリも線形演算のFLOPsもおおむね半分になります。

数値例。1台のカメラ特徴を `(8, 256, 64, 64)` で保持すると、float32 で 8 x 256 x 64 x 64 x 4 バイト ≈ 134 MB です。これを 512 次元にすると倍の約 268 MB。7台のカメラ分を同時に保持すると、256 で約 0.9 GB、512 で約 1.9 GB と、次元の選択が VRAM 消費を直接左右します。これが「256 を選ぶ」現実的理由です。

## 手法の系譜と主要論文

一定の隠れ次元を保つ設計の系譜を追います。

Transformer(Vaswani et al., "Attention Is All You Need", NeurIPS 2017, arXiv:1706.03762)。この原論文が d_model(モデル次元)を 512 に固定し、Self-Attention・Feed-Forward・埋め込みなど内部のほぼ全ての層で同じ次元を保つ設計を確立しました。理由は前述の残差接続で、各サブ層の出力を入力に足し込むには次元一致が必須だからです。効果は実装の単純化と層の積み重ねやすさ、トレードオフは「次元を1か所変えると全体に波及する」剛直さです。base モデルが d_model=512、big モデルが d_model=1024 と、モデルサイズを次元で制御する流儀もここで生まれました。

BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)。複数カメラ画像を鳥瞰図(BEV, Bird's-Eye-View)特徴に変換する自動運転知覚モデルで、BEV クエリの次元を 256 に統一し、空間・時間の Attention を通じて一貫した次元で特徴を処理しました。複数カメラの特徴を1つの BEV 平面に集約するには、全カメラ・全クエリが同じ次元であることが前提になります。

UniAD(Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156)。知覚(検出・追跡・地図)・予測(動き予測・占有予測)・計画(自車軌道)という複数タスクを1つのネットワークに統合した記念碑的研究です。タスクモジュール間で特徴(クエリ)を受け渡すために、全モジュールで共通の特徴次元(256)を保つことが設計の根幹でした。共通次元にする効果はモジュール間接続の単純化、トレードオフは「あるタスクだけ次元を増やして表現力を上げる」個別最適がやりにくくなることです。UniAD のように「前段タスクの出力クエリを後段タスクの入力にする」連鎖構造では、共通次元がなければそもそも繋がりません。

## 論文の実験結果(定量データ)

次元の選択がどれだけ効くかは、複数の研究で系統的に測られています。指標の意味から説明します。

UniAD は nuScenes(自動運転の標準ベンチマーク。約1000シーン、6カメラ + LiDAR + レーダ)上で評価されます。代表指標は、検出の NDS(nuScenes Detection Score、検出の総合点)、動き予測の minADE/minFDE(予測軌道と正解の平均/最終距離誤差、メートル単位で小さいほど良い)、計画の L2 誤差(予測した自車軌道と正解の距離、メートル単位)と衝突率(Collision Rate、%で小さいほど良い)です。UniAD は計画の平均 L2 誤差を約1.03m、平均衝突率を約0.31%まで下げ、当時の代表的なベースライン(ST-P3 など、L2 約2.1m・衝突率約1%前後)を大きく上回りました。重要なのは、これらの全タスクが共通次元256でクエリを受け渡す統合設計のもとで達成された点です。アブレーションでは「前段タスクのクエリを後段に渡す経路を切る」と後段タスクの精度が悪化することが示され、共通次元による接続が性能に寄与していることが裏付けられています。例えば動き予測モジュールから占有予測・計画への情報伝播を切ると、計画 L2 と衝突率が悪化します。

Transformer 原論文のアブレーションでは、d_model を含む各ハイパーパラメータを振った表が示され、次元やヘッド数を変えると WMT 2014 英独翻訳の BLEU スコア(翻訳品質、高いほど良い)が変動することが報告されています。base(d_model=512)で BLEU 約27.3、big(d_model=1024)で約28.4と、次元を上げると精度は上がる一方、学習コストが大きく増えることが定量的に示されました。これは「次元は大きいほど表現力は上がるが、計算・メモリと引き換え」という一般則の原典的な実証です。

スケーリング則の観点(Kaplan et al., arXiv:2001.08361, 2020)。言語モデルでは、テスト損失がパラメータ数・データ量・計算量に対しておおむね冪則(power law)で滑らかに下がることが報告されました。モデル幅(=隠れ次元)はパラメータ数を増やす主要な軸の1つで、「次元を上げれば損失が下がる」傾向には予測可能な法則性があります。固定次元の設計はこの探索軸を意図的に1点に固定するもので、開発の単純さと引き換えにスケーリングの余地を制約します。

実装面の定量効果として、共通次元の採用は「接続のためのアダプタ層」を排除します。仮にモジュール境界が3か所あり、各境界に 768->256 のような全結合射影を置くと、それぞれ約 768x256 ≈ 20万個のパラメータと対応する演算が無駄に積み上がります。共通次元化はこれを入口の1箇所に集約し、無駄なパラメータと中間バッファのメモリを削減します。

## メリット・トレードオフ・限界

メリット。第一に、部品をつなぐとき次元が常に一致するので変換層を挟む必要がなく、接続が単純になる。第二に、融合(連結・加算・Attention)が「全入力が同じ256次元」という前提で安全に書ける。複数カメラ特徴を足し合わせる/Attention で混ぜる処理は、入力次元が全カメラで同じであることが大前提です。第三に、新しい部品を足すときも「入出力は256」と決まっているので設計判断が減り、開発が速くなる。

トレードオフと限界。第一に、256 という1つの値に全体が依存するため、本当に表現力が足りない部分だけ次元を増やす、という局所的なチューニングがやりにくい。あるタスクは128で十分でも、別のタスクは512欲しい、という最適化が共通次元では難しくなります。第二に、backbone の生出力は 256 より大きい(例 1440)ので、入口で256へ落とす射影の段階で情報が一部圧縮される。第三に、固定次元は「次元のスケーリング則(次元を増やすと精度がどう伸びるか)」の探索を制約します。大規模化が精度を稼ぐ近年の流れでは、固定の256が天井になる場面があります。

研究上の未解決課題として、「タスクごとに最適な次元は本来異なるはずだが、共通次元の利便性とのバランスをどう取るか」があります。一部の研究はモジュール境界だけ次元を変える射影(まさにアダプタ)を許容し、内部は共通次元に保つハイブリッドを採ります。また、Mixture-of-Experts のように「同じ次元のまま容量だけ増やす」アプローチが、固定次元の表現力上限を回避する方向として注目されています。

## 発展トピック・研究の最前線

スケーリング則(Kaplan et al., 2020; Hoffmann et al. "Chinchilla", 2022)は、モデル次元・層数・データ量・計算量と性能の関係を冪則で記述する研究で、「次元をどこまで上げるべきか」を理論的に議論する土台です。固定次元の設計はこの探索を制約する一方、実装の単純さで開発速度を稼ぎます。Chinchilla は「同じ計算予算なら、モデルを大きくしすぎずデータを増やす方が良い」という補正を加え、次元の上げ方にも最適点があることを示しました。

アダプタとボトルネック設計。共通次元を保ちつつ、必要な箇所だけ一時的に次元を絞る/広げる(ボトルネック)ことで表現力と効率を両立する手法が広く使われます。LoRA(低ランク適応、Hu et al., 2021)のように「共通次元のモデルに低次元の追加経路を差し込む」発展も、固定次元を前提にした拡張です。

自動運転の最前線では、UniAD 以降 VAD(ベクトル化シーン表現)や占有ネットワーク(Occupancy Network)など、共通クエリ次元でタスクを連結する設計が標準になっています。各社のエンドツーエンド計画モデルも、内部の特徴次元を一定に保つことでマルチタスク接続を成立させています。
