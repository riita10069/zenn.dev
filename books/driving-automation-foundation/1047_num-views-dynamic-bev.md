# num_views動的化（多データセット単一モデル）

## ひとことで言うと
車に載っているカメラの台数は、データセットごとに違います（あるものは7台、別のものは8台、5台、6台）。これらを別々のモデルにせず1つのモデルで学習したいとき、カメラ台数を融合処理（複数カメラの情報を1つにまとめる処理）に「実行のたびに見て柔軟に対応させる」ようにする設計が要ります。鳥瞰図（BEV）に統合する方式は本来カメラ台数に依存しない設計なのに、実装の一部に台数が固定で焼き付いてしまうと、その柔軟さが失われます。それを実行時の台数 V に完全対応させるのが num_views 動的化です。

## 直感的な理解
受付に「来客全員の意見を聞いて、決まった大きさのホワイトボード1枚にまとめる」係がいると想像してください。来客が7人でも8人でも、ホワイトボードの大きさは変わりません。係は「今日来ている人全員」に順に話を聞き、聞けた人数で平均してボードに書きます。これが BEV 融合の発想です。ホワイトボード（BEVグリッド）はカメラ台数と無関係に固定サイズなので、出力の形はいつも同じになります。

ところが、もし受付の台帳に「来客はいつも7人」と印刷された固定の集計欄があったらどうでしょう。8人来た日に台帳の欄が足りず作業が破綻します。num_views 動的化は、この「7人前提で刷られた欄」を「来た人数ぶんだけ動的に作る欄」に置き換える作業です。係の本来の手順（全員に聞いて平均する）は台数に依存していないので、固定欄さえ直せば何人でも回せます。

## 基礎: 前提となる概念
- ビュー（view）: 1台のカメラから見た1枚の映像。views=7 なら、ある瞬間に7枚の画像が同時に入ってくる、という意味です。
- マルチビュー（multi-view）: 複数カメラで車の周囲を同時に撮ること。死角を減らし360度を見るために使います。
- BEV（Bird's-Eye-View、鳥瞰図）: 上空から見下ろした地図のような平面表現。複数カメラの情報を共通の地面座標に集約するのに便利で、検出・予測・計画の後段を統一できます。
- BEVグリッド: BEV 平面を格子状のマス目に区切ったもの（例 450×300 マス）。各マスが「自分の位置が映っているカメラ」に情報を問い合わせます。
- 投影（projection）: 3次元空間の点を、各カメラの内部・外部パラメータ（焦点距離・レンズ歪み・車に対する位置と向き）を使って画像上の2次元座標に写す計算。
- attention（注意機構）: 複数の入力のうちどれをどれだけ重視するかを重みで決める仕組み。BEV では各マスがカメラ画像のどの位置の特徴を取り込むかをこれで決めます。

記号を整理します。

- `B`: バッチ数（一度に処理するサンプル数）
- `V`: 実行時のカメラ台数（runtime の view 数）
- `num_views`: モデルを作ったときに固定された「想定カメラ台数」
- `N`: BEVグリッドのマス目の総数（bev_h × bev_w）
- `num_z` / `num_points_in_pillar`: 各マスについて高さ方向にサンプリングする点の数（1マスを縦に何段か見る。地面だけでなく高さのある物体も拾うため）

普通にモデルを作ると、「カメラは7台」という前提がパラメータ（重み）の形に固定で焼き付きます。すると7台用モデルに8台分を流すと形が合わず止まります。逆も同じで、結果「構成ごとに別モデル」となり、複数データセットを1モデルでまとめて学習できません。これは単なる不便ではなく、学習に使えるデータ量を構成ごとに分断してしまうという、性能上の損失でもあります。

## 仕組みを詳しく
BEV 融合は本来カメラ台数に依存しません。固定サイズのグリッドが「今あるカメラ全部」に問い合わせ、出力は常に `[B, embed_dim, bev_h, bev_w]`（embed_dim は特徴次元、例 256）という同じ形になるからです。問題は、forward の多くの場所はすでに実行時 `V` を正しく使っているのに、一部だけが固定値 `num_views` に依存して残ってしまう典型パターンです。よくある「固定が残りやすい3か所」を分解します。

1. 投影結果の reshape が固定値

```python
ref_2d      = ref_2d.reshape(B, num_views, N, num_z, 2)
valid_depth = valid_depth.reshape(B, num_views, N, num_z)
```

`reshape` はテンソルの形を組み替える操作です。直前のカメラ行列による3D→2D投影（典型的には einsum で書かれる行列演算）はすでに実行時 `V` で動いているのに、ここで固定 `num_views=7` を使うと、実際のカメラが8台のとき総要素数が合わずクラッシュします。直し方は単純で、`V` を投影テンソルの view 次元（例 `proj.shape[1]`）から動的に取ることです。固定値を1か所でもハードコードすると、他がすべて動的でも全体が壊れる――この「鎖の弱い環」を潰すのが本作業の本質です。

2. attention 重みが固定幅の Linear

```python
self.attention_weights = nn.Linear(embed_dim, num_views * num_points_in_pillar)
```

`nn.Linear` は「入力次元→出力次元」の全結合層で、出力幅は作った瞬間に固定されます。`num_views=7` で作ると `7 * num_z` 個の値を出しますが、実行時 `V=8` で reshape しようとして形が合わず壊れます。直し方は「出力幅を `num_points_in_pillar` だけにして、カメラ1台ごとに `range(V)` ループで適用する」こと。カメラ間の重み付けは、すでに「見えているカメラだけで平均する（`1/|V_hit|`、実際に投影が画像内に収まったカメラ台数で割る）」という可視性ベースの平均で実現できるので、カメラ台数ぶんの固定ヘッドはそもそも不要です。これは後述する BEVFormer の Spatial Cross-Attention と同じ考え方です。

3. 校正なし時の固定フォールバック

校正情報（各カメラの位置・向き・レンズ特性を表す投影行列）が無いとき、`[num_views, 3, 4]` のような固定形の学習パラメータを代用する実装があります。これは入力をランダムに作る動作確認（スモークテスト）には便利ですが、台数が固定値に縛られます。複数データセット学習では各データセットが固有の校正値を持つので、本物の校正を渡すのが前提です。直し方は「`V != num_views` のときは校正情報を必須にする」か「フォールバック自体を台数非依存にする」のどちらかで、固定フォールバックは校正なしの簡易動作確認専用に限定します。

直す前と後のイメージ:

```
[修正前]  num_views=7 で作ったモデル
   7枚バッチ → OK
   8枚バッチ → reshape / Linear幅が合わずクラッシュ

[修正後]  同じ1つのモデルインスタンス
   7枚バッチ → [B, 256, 450, 300]
   8枚バッチ → [B, 256, 450, 300]   (再生成なしで連続実行できる)
```

重要な制約: 1つのミニバッチ（`torch.stack` で1つのテンソルに束ねる単位）の中ではカメラ台数を揃える必要があります。7枚のサンプルと8枚のサンプルは1つのテンソルに積めないからです。したがって多データセット学習は「同じ `V` のサンプルだけで作ったバッチを、データセットごとに切り替えながら回す（homogeneous-V batches を交互に流す）」形になります。台数が動的なモデルならこれを自然に処理できますが、固定モデルではできません。データ供給側（DataLoader / サンプラ）には、データセットごとに均一バッチを組むサンプリング設計が必要です。

検証の観点では、典型的な受け入れ条件として次のようなものが置かれます。同一インスタンスが7枚バッチと8枚バッチを再生成なしで連続実行できること、両方で学習可能パラメータ（BEVクエリや投影層・出力層）に勾配が流れること、既存の BEV テストが壊れないこと、そして `V` が交互に変わるバッチを1つの学習ループで回す新規テストが通ること。勾配が流れることを確認する点が重要なのは、形だけ合っても一部パラメータが計算グラフから切れていれば学習されず、バグが「精度が出ない」という遅い形で現れるからです。

## 手法の系譜と主要論文
「BEV やクエリ表現はそもそもカメラ台数に依存しない」という性質は、複数の論文が積み上げてきたものです。num_views 動的化は、その理論的性質を実装でも最後まで貫こうという話です。

- DETR3D（Wang et al., CoRL 2021, arXiv:2110.06922）。3D の参照点を各カメラ画像に投影して特徴を集める、クエリベースのマルチカメラ3D検出を提案。明示的な深度推定なしに複数カメラを統合できる点が新しく、台数非依存にマルチカメラを扱う素地を作りました。トレードオフは、投影が画像外に外れる点の扱いや、可視点が少ないクエリの不安定さです。
- Lift-Splat-Shoot（Philion & Fidler, ECCV 2020, arXiv:2008.05711）。各カメラ画素に深度の確率分布を推定し、特徴を3D空間に「持ち上げ（lift）」共通 BEV 平面に「貼り付ける（splat）」枠組みを提案。任意台数のカメラを共通 BEV に集約する典型例で、出力 BEV は台数に依存しません。
- BEVDet（Huang et al., 2021, arXiv:2112.11790）。Lift-Splat-Shoot 系の高速・高精度な検出パイプライン化。これも出力 BEV はカメラ台数に依存しません。トレードオフは深度推定の精度に性能が左右される点です。
- PETR（Liu et al., ECCV 2022, arXiv:2203.05625）。3D 位置エンコーディングを画像特徴に埋め込み、明示投影なしにクエリと対応付ける手法。DETR3D 系を簡潔化しつつ台数非依存を保ちました。
- BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）。学習可能な「BEVクエリ」が、自分の位置が映っているカメラにだけ問い合わせる Spatial Cross-Attention（SCA）を提案。各クエリは複数の高さ点（pillar 内の点）を各カメラに投影し、画像内に収まったカメラの特徴だけを集めて「見えた台数で平均」します。これによりカメラ台数や搭載位置に依存せず固定サイズの BEV 特徴を作れます。num_views 動的化が参照する「可視性ベースの平均」は、まさにこの SCA の考え方です。トレードオフは投影とアテンションの計算が重く、正確な校正情報が必須な点です。

系譜は「明示投影でマルチカメラを統合（DETR3D）」→「深度持ち上げ（Lift-Splat-Shoot / BEVDet）・位置埋め込み（PETR）で多様化」→「BEVクエリ＋可視性平均で台数非依存を洗練（BEVFormer）」と進みました。num_views 動的化は、この成熟した台数非依存設計を「実装の固定欄を消して」完成させる実務作業です。

## 論文の実験結果(定量データ)
- BEVFormer（ECCV 2022）: nuScenes（6カメラ・市街地走行の大規模データセット）の3D物体検出で、当時のカメラのみ手法として最先端級の NDS を達成しました。NDS（nuScenes Detection Score）は検出精度・位置・速度・属性などを総合した指標で、0〜1（百分率表示も多い）で高いほど良い指標です。BEVFormer は NDS でおよそ 0.51〜0.56、mAP（平均適合率、検出枠の当たり具合の平均）でおよそ 0.41〜0.48 程度を報告し、単フレームのカメラ手法を明確に上回りました。重要なのは、この性能が「カメラ台数に依存しない固定 BEV 表現」の上で得られている点です。
- BEVFormer のアブレーション: 時間方向の Temporal Self-Attention（過去フレームの BEV を参照する仕組み）を外すと、特に速度推定や遮蔽下の検出が悪化し、NDS が数ポイント低下すると報告されています。これは「BEV 表現は静的な1枚でも台数非依存だが、時系列を足すとさらに強い」ことを示します。num_views 動的化の文脈では、まず空間方向（マルチビュー集約）の台数非依存を担保することが前提になります。
- DETR3D（CoRL 2021）: nuScenes で、明示的な深度や複雑な後処理なしに当時の競合 LiDAR なしカメラ手法と肩を並べる性能（NDS 約 0.42、mAP 約 0.34 級）を示し、クエリベース・マルチカメラ統合の有効性を定量的に裏づけました。
- Lift-Splat-Shoot（ECCV 2020）: 任意のカメラ集合を共通 BEV に集約でき、カメラを意図的に欠落させても破綻しない頑健性（カメラ脱落への耐性）を示しました。これは「台数非依存集約」が単なる便宜でなく、欠損ビュー耐性という実用上の頑健性ももたらすことの早い実例です。
- これらの結果が共通して示すのは「出力 BEV/クエリ表現はカメラ台数で形が変わらない」という事実であり、num_views 動的化の主張「設計は台数非依存なのに実装が縛っている」の根拠になります。なお、台数が異なるデータを混ぜて学習する効果そのもの（混ぜると精度がどれだけ上がるか）は、後段の [多データセット単一モデル学習](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1045_multi-dataset-single-model) の論点で、ここでは「形を合わせる前提条件を整える」ところまでが射程です。

## メリット・トレードオフ・限界
- メリット: カメラ台数が違う複数データセットを1モデルで学習でき、データ量を最大化できる。
- メリット: 構成ごとにモデルを何個も持たずに済み、運用・実験管理が単純になる。
- メリット: BEV 本来の台数非依存設計を、実装でも完全に活かせる。
- トレードオフ: ミニバッチ内では台数を揃える制約が残り、DataLoader/サンプラ側でデータセットごとにバッチを組む工夫が必要。
- トレードオフ: 多データセットでは各データセットの本物の校正情報が必須になり、校正なしの簡易経路は動作確認用に限定される。
- 限界: この動的化が効くのは投影・可視性平均で台数非依存に書ける BEV/クエリ系の融合だけ。チャネル連結（各カメラ特徴をチャネル方向に並べる）のように台数が重みの形に直結する単一構成前提の融合方式には適用できない。
- 限界: 台数だけ合わせても、カメラの視野・解像度・搭載位置がデータセット間で大きく違うと、特徴分布のドメイン差が残る。これは別途のドメイン適応の課題として残る。

## 発展トピック・研究の最前線
台数動的化のさらに先には、センサ種別自体の動的化（カメラのみ／カメラ+LiDAR の混在を1モデルで扱う）や、欠損ビュー（一部カメラが故障・遮蔽）に頑健な BEV 集約といった研究があります。BEVFusion（Liu et al., 2022）のようなマルチモーダル融合は、異なるセンサを共通 BEV 空間に集約する流れの代表例です。また、可視性平均をそのまま使うとカメラ重複領域の重み付けが単純すぎるという指摘があり、学習可能な信頼度重みや不確実性推定を導入してビューごとの寄与を適応的に調整する研究も進んでいます。実務面では、台数が交互に変わるバッチを安定して回すためのサンプラ設計、データセット間のサンプリング比率の調整、勾配のスケール差の補正などが、多データセット学習を機能させる地道だが重要な要素になります。
