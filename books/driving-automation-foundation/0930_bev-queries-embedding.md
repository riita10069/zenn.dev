# BEVクエリ（学習可能なグリッドクエリ）

## ひとことで言うと

BEV（鳥瞰図、車を真上から見下ろした地図のような表現）の各マス目に、「この場所には何があるのだろう」と画像へ問い合わせるための小さなベクトル（数字の並び）を1本ずつ用意し、その問い合わせ方そのものを学習で賢く育てていく仕組みです。地図の格子上に並ぶ学習可能なベクトル群を BEVクエリ（BEV queries）と呼びます。これは Transformer の「クエリ」という概念を、物体ごとではなく空間のマス目ごとに割り当て直したものです。深さの推定を正面から解かずに済ませながら、複数カメラの情報を1枚の地図に統合できる点が画期的でした。

## 直感的な理解

自動運転では、車の周囲に取り付けた複数のカメラ（前・後・左右など、典型的には6台）の映像を、1枚の「真上から見た地図」にまとめたくなります。これを BEV（Bird's-Eye-View、バードアイビュー＝鳥瞰図）と呼びます。なぜ地図にしたいかというと、運転の判断（前の車に近づきすぎていないか、左に曲がれるか、歩行者がどこにいるか）は「上から見た位置関係」で考えるのが自然だからです。カメラ画像はフロントガラスから見た景色なので、そのままでは「自分から見て何メートル前の右側」という位置がわかりにくいのです。

ここで根本の問題は「どうやってカメラ画像から地図を作るか」です。素朴な方法は、画像の各ピクセルがどれくらい遠いか（深度）を推定し、その深度を使って3D空間に点を打ち、上から見下ろす、というものです。しかしこの深度推定はとても難しく、誤ると地図がぐちゃぐちゃになります。1ピクセルの深度を1m間違えるだけで、地図上で物体が1m先にずれて配置されます。

そこで発想を逆転させます。「深度を当てにいくのではなく、地図の各マスのほうから画像に問い合わせにいく」という考え方です。地図の各マス（たとえば前方5m・右2mの地点）が「私はこの位置だけれど、カメラ画像のどこを見ればいい? そこには何が写っている?」と尋ねるのです。この尋ねる側のベクトルが BEVクエリです。クエリ（query）とは「問い合わせ」という意味で、検索エンジンに打ち込む検索ワードのようなものだと考えてください。検索ワードを賢くすればよい結果が返るのと同じで、問い合わせベクトルを学習で賢くすると、画像から有用な特徴が返ってくるようになります。

## 基礎: 前提となる概念

埋め込みテーブル（embedding table）。深層学習では「番号（離散的なID）」を「ベクトル（連続値の並び）」に変換する表をよく使います。たとえば自然言語処理では単語IDをベクトルに変換します。この表は単なる学習可能な行列で、形状は `[項目数, ベクトルの次元]` です。番号 i を渡すと、その行 i のベクトルが返ります。BEVクエリでは「マス目の番号」を「問い合わせベクトル」に変換する表として使います。プログラム上では `nn.Embedding(H*W, C)` のような形で宣言され、テーブルの中身（重み）が誤差逆伝播で更新されます。

アテンション（attention、注意機構）。Transformer の中核となる仕組みで、「クエリ（問い合わせ）」が「キー（手がかり）」との一致度を計算し、その重みに従って「バリュー（値）」を加重平均する処理です。式で書くと `Attention(Q,K,V) = softmax(QKᵀ/√d) V` で、Q がクエリ、K がキー、V がバリュー、d は次元数、√d は内積が大きくなりすぎるのを抑える正規化です。直感的には「自分が知りたいことに関係する場所を選んで、そこの情報を重点的に集める」操作です。BEVクエリは、画像特徴に対してアテンションをかけるときの「クエリ」の役割を担います。

残差接続（residual connection）。`出力 = 入力 + f(入力)` という形で、変換後の値に元の入力を足し戻す設計です。なぜこうするかというと、深いネットワークでも勾配（学習の手がかりとなる微分値）が消えにくく、学習が安定するからです。BEVクエリでは「問い合わせベクトル + 画像から拾った特徴」という形で残差を作り、画像から何も拾えなかったマスでもクエリ自身の情報が残るようにします。

テンソルの形状（tensor shape）。深層学習のデータは多次元配列（テンソル）で表します。本稿では `[B, N, C]` のような表記を多用します。B はバッチサイズ（同時に処理する場面の数）、N はクエリ数（マス目の総数）、C は各ベクトルの次元数です。

## 仕組みを詳しく

典型的な BEVクエリの設計を、段階を追って具体的に見ていきます。

まず地図を格子に分割します。たとえば縦方向に H マス、横方向に W マスに分けると、合計 N = H × W 個のマスができます。BEVFormer の標準設定では 200 × 200 = 4万マス、応用によっては数十万マスにすることもあります。各マスは「自車から前方 x メートル・左 y メートル」という平面上の領域に対応します。たとえば 200×200 のマスで前後左右 ±51.2m をカバーするなら、1マスは約 0.512m 四方を表します。

各マスに1本ずつ、長さ C のクエリベクトルを割り当てます。これをまとめたものが学習可能な埋め込みテーブルで、形状は `[N, C]` です。たとえば N = 40,000、C = 256 なら、`[40000, 256]` の行列になります。0番のマスには 256個の数、1番のマスにも 256個の数、というように各マスが固有のベクトルを持ちます。

```
BEVクエリのテーブル（形状 [N, C]）
マス0  : [ 0.12, -0.03,  0.51, ... ]  (長さ C=256)
マス1  : [-0.22,  0.40,  0.09, ... ]
...
マスN-1: [ 0.05,  0.31, -0.18, ... ]
```

このテーブルの中身が学習で更新されることが本質です。最初はランダムな数（ゴミ）が入っていますが、学習を進めると「前方のマスは遠くの車や信号を見るのが得意な問い合わせ方」「自車の真横のマスは隣の車線を見る問い合わせ方」といった具合に、各マスが自分の役割に合った問い合わせベクトルへ育っていきます。位置ごとに役割が違うことを、モデルが自分で覚えていくわけです。なお BEVクエリには、各マスがどの座標に対応するかを表す位置埋め込み（positional embedding）を加えるのが普通で、これがないとアテンションがマス同士の位置関係を区別できません。

forward（順伝播、推論の1回ぶんの計算）では、まずテーブルをバッチ方向に複製します。

```
queries = テーブル.weight を [1, N, C] に変形し、[B, N, C] に複製
```

たとえば B=4 なら、形状は `[4, 40000, 256]` です。バッチ内のどの場面でも同じ問い合わせ方の表を使う点がポイントです。問い合わせ方そのものは場面に依存せず、得られる答え（拾われる画像特徴）が画像ごとに変わります。これは「同じ質問票を全員に配るが、回答は人によって違う」のと同じ構図です。

このクエリは forward の中で典型的に次の役割を担います。

1. サンプリングのずらし量（オフセット）を予測する材料になる。各マスが画像のどこを見るか、その基準点から少しずらす量をクエリから線形層で予測します（Deformable Attention の考え方、[F.grid_sample による可搬な変形アテンション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0968_grid-sample-deformable-attention) 参照）。
2. アテンションの重み（どの高さ・どの参照点・どのカメラを重視するか）を予測する材料になる。複数候補からどれを信じるかをクエリから決めます。
3. アテンション後の残差接続の基準になる。`出力 = queries + 射影(集めた特徴)` という形で、クエリ自身が最終出力の土台になります。

before / after で言うと、forward に入る前のクエリは「位置ごとの問い合わせ方を表す抽象ベクトル」、forward を抜けた後の出力は「その位置に実際に何があるかを表す BEV 特徴ベクトル」です。形状は同じ `[B, N, C]` ですが、意味が「質問」から「答え入りの記述」へ変わります。この BEV 特徴をさらに別タスク（物体検出、地図セグメンテーション、経路計画）の入力に渡していきます。

BEVFormer のエンコーダは、この問い合わせを1回ではなく複数層（標準で6層）繰り返します。各層は (1) 過去フレームの BEV を参照する時間的自己注意（temporal self-attention）、(2) 画像へ問い合わせる空間的相互注意（spatial cross-attention）、(3) 全結合（feed-forward）の3つを順に通し、毎回残差接続で元のクエリへ足し戻します。層を重ねるほど、最初は曖昧だったクエリが「答えの詰まった BEV」へと精緻化されていきます。

なぜ「物体ごと」ではなく「マスごと」にクエリを置くのか。物体検出だけが目的なら物体ごとのクエリ（後述の DETR 流）で十分ですが、自動運転では「空間そのものを密に記述したい」という要求があります。マスごとにクエリを置くと、空間全体を漏れなく覆う密な地図表現が得られ、検出・地図化・占有予測など複数タスクで共有できる土台になります。これが UniAD のような「1つの BEV 表現を全タスクで使い回す」設計を可能にしました。

## 手法の系譜と主要論文

学習可能なクエリで問い合わせるという発想の起源は DETR（Carion et al., ECCV 2020, "End-to-End Object Detection with Transformers", arXiv:2005.12872）です。DETR は「object query（物体クエリ）」という固定数（標準では100本）の学習可能ベクトルを用意し、各クエリが画像全体に注意を向けて「1つの物体」を担当する設計を示しました。動機は、従来の物体検出にあった煩雑な後処理（重複枠の除去＝Non-Maximum Suppression、アンカーの設計）を、クエリ同士が役割分担する形で不要にすることでした。新規性は、検出を「集合予測（set prediction）」として、二部マッチング損失（Hungarian matching）で end-to-end に解いた点にあります。BEVクエリはこの object query を「物体ごと」から「空間のマスごと」へ割り当て直したもの、と理解すると見通しが良くなります。

DETR の弱点（収束が遅い、小物体に弱い）を克服したのが Deformable DETR（Zhu et al., ICLR 2021, arXiv:2010.04159）で、クエリが画像全体ではなく少数の参照点周辺だけにアテンションを向ける deformable attention を導入しました。これが計算量を激減させ、後の BEVFormer の空間的相互注意の直接の土台になります。

これを BEV 構築に持ち込んだのが BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）です。BEVFormer は「grid-shaped BEV queries」と呼ぶ格子状のクエリを用意し、各クエリが対応する3D空間の点を複数カメラの画像に投影して、その投影先の特徴を deformable attention で集める方式（spatial cross-attention）を提案しました。動機は、当時主流だった「深度を明示的に推定してから3Dに持ち上げる」方式が深度誤差に弱かったことです。さらに時間方向の自己注意（temporal self-attention）で過去フレームの BEV を参照し、動いている物体の速度推定や一時的なオクルージョン（遮蔽）に強くしました。

ほぼ同時期に、クエリに位置情報を埋め込む別系統として PETR（Liu et al., ECCV 2022, arXiv:2203.05625）が登場しました。PETR は明示的な投影サンプリングを避け、3D座標を位置埋め込み（3D position embedding）に変換して画像特徴に注入し、クエリがグローバルアテンションで直接拾う設計です。BEVFormer が「局所的にサンプルする」のに対し、PETR は「位置情報を画像側に埋め込んで全体から拾う」点で対照的です。両者はその後の研究で頻繁に比較されます。

BEVFormer の BEV エンコーダを丸ごと使い、検出・追跡・地図・予測・計画までを1本につなげたのが UniAD（Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156）です。UniAD は「最終的にやりたいのは運転計画なのだから、BEVクエリで作った中間表現を全タスクで共有し、計画に向けて最適化すべき」と主張しました（planning-oriented design）。BEVクエリを起点に、各タスク用のクエリ（trackクエリ、mapクエリ、agentクエリ）へと情報を伝播させていく構造です。さらに VAD（Jiang et al., ICCV 2023, arXiv:2303.12077）は、密なラスタ表現ではなくベクトル化された地図・エージェント表現に置き換えて高速化し、BEVクエリ系の流れを実用的な推論速度へ近づけました。

## 論文の実験結果（定量データ）

BEVFormer は nuScenes という自動運転データセット（米国・シンガポールで収集した1000シーン、6カメラ・LiDAR付き）で評価されました。評価指標の中心は NDS（nuScenes Detection Score）で、これは検出の正確さ（mAP、mean Average Precision＝平均適合率の平均）に加えて、位置・大きさ・向き・速度・属性の5種の誤差を総合した0〜1のスコアです。高いほど良く、位置や速度まで正しく当てられて初めて伸びます。BEVFormer はテストセットでカメラのみの手法として NDS 0.569、mAP 0.481 を報告し、当時のカメラベース手法（DETR3D の NDS 0.479 など）を明確に上回りました。特に LiDAR を使わずにこの水準に達した点が注目されました。

アブレーション（どの要素を抜くと性能がどう変わるかの実験）では、時間方向の自己注意を外すと速度推定の誤差（mAVE、mean Average Velocity Error、単位は m/s）が大きく悪化することが示されました。報告によると、temporal self-attention を入れることで mAVE がおよそ 0.5 前後から 0.4 前後へ改善し、速度を当てる能力が時間情報に強く依存することがわかりました。これは「1フレームだけ見ても物が動いているか止まっているかわからない」という直感と一致します。空間方向のクロスアテンション（BEVクエリが画像へ問い合わせる部分）をフルの deformable attention から簡略なグローバルアテンションに置き換えると mAP が下がることも示され、クエリ設計そのものが精度に効くことが裏付けられました。

UniAD は同じ nuScenes で、計画タスクの衝突率（planning collision rate）を主要指標に評価しました。報告によると、タスクを別々に学習する従来手法に比べ、BEVクエリ表現を全タスクで共有して end-to-end に学習することで、3秒後の平均衝突率がおよそ半分以下（報告では 0.31% 前後）に下がりました。これは「中間表現を計画に向けて最適化する」という主張を定量的に支える結果です。VAD はこの精度水準を保ちつつ、推論速度を UniAD 比でおよそ数倍に高め、計画 L2 誤差でも同等以上を報告しました。

DETR は COCO（一般物体検出のベンチマーク）で、ResNet-50 バックボーンで AP 42.0 程度を達成し、当時の Faster R-CNN と同等の精度を、後処理なしの end-to-end で出せることを示しました。一方で小さい物体の検出が苦手で収束が遅い（500エポック必要）という限界も明らかになり、これが Deformable DETR（約50エポックで AP 43〜46）による高速化研究につながります。クエリ設計の効率化がそのまま BEV 構築の実用性を左右したことが、この系譜からわかります。

## メリット・トレードオフ・限界

メリット。深度を明示的に予測しなくても、各マスが画像に問い合わせる形で BEV を構築できるため、深度誤差に振り回されにくくなります。位置ごとに異なる問い合わせ方を学習で獲得できるので、固定ルールより柔軟です。出力が「上から見た地図」なので、後段の経路計画や物体配置の理解と相性が良いことも大きな利点です。残差接続のおかげで、画像から拾えなかったマスでもクエリ自身の情報が残ります。

トレードオフと限界。クエリ数（H × W）が増えると計算量とメモリが急増します。たとえば 4万マス × 256 次元のテーブルだけでも大きく、アテンション計算はクエリ数にほぼ比例して重くなります。学習初期はクエリがランダムなので、十分なデータと学習時間がないと役割が育ちません。BEVFormer 級のモデルは収束に数十エポック以上を要し、バックボーンを含めると学習コストは小さくありません。固定格子のクエリは遠方の疎な領域にも近傍の密な領域にも同じ解像度を割り当てるため、計算資源の配分が最適でないという根本的な課題も残っています。また、平面の格子に固定するため、高さ方向の構造（橋の下、立体交差）は表現できず、これが占有予測（3D voxel）への発展を促しました。

## 発展トピック・研究の最前線

固定格子のクエリの非効率を解消する方向として、疎な BEV 表現（sparse BEV）の研究が進んでいます。たとえば SparseBEV や Sparse4D 系は、密な格子を持たずにクエリを疎に配置したり、必要な領域だけ高解像度にする手法を探っています。

もう一つの大きな潮流が、BEV を平面の地図ではなく3Dのボクセル（立方体の格子）として持つ占有予測（occupancy prediction）です。SurroundOcc や TPVFormer は、BEVクエリを3次元に拡張し、「各立方体が埋まっているか空いているか」を予測します。これにより、平面 BEV では潰れてしまう高さ方向の情報（橋の下、はみ出した荷物、不定形の障害物）を扱えるようになります。

時間方向の活用も深化しています。BEVFormer v2 や StreamPETR は、過去の BEV を逐次的に伝播させる効率的な仕組みを導入し、長い時間文脈を低コストで扱えるようにしました。クエリ表現を起点に、検出・予測・計画を統合する end-to-end 設計（UniAD、VAD、その後の言語モデルと統合する流れ）も活発に研究されており、BEVクエリは現代の多視点認識・運転モデルの共通言語になりつつあります。
