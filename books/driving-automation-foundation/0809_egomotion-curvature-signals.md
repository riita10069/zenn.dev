# 自車運動の入力信号設計 [speed, accel, yaw_angle, curvature]

## ひとことで言うと
自動運転モデルには「カメラ映像」だけでなく「自分の車がどう動いてきたか」の履歴も入れます。その履歴をどんな数値で表すかが本ノートのテーマです。代表例として、速度・加速度・向きの角度・曲率という 4 つの数値で各時刻を表す設計を扱います。特に 4 つ目の「曲率 (curvature)」を、よくある代理計算ではなくセンサーに記録された実測値から直接使うことの利点に焦点を当てます。

## 直感的な理解
カメラの 1 枚の絵からは「今そこに何があるか」は分かっても、「自分が今どんな運動状態か」は読み取りにくいものです。同じ前方風景でも、減速中なら次はブレーキ継続、加速中ならアクセル継続が自然です。停止寸前なのか巡航中なのか、直進中なのか旋回の途中なのか。これらは画像だけでは曖昧で、過去の運動履歴を与えて初めてモデルが「運動の連続性」を踏まえた予測を出せます。

もう 1 つの直感は「道の曲がり方をどう数値化するか」です。同じカーブを、ゆっくり走っても速く走っても、道の形は同じです。この「道の形」を速度に左右されずに表せるのが曲率です。一方、よく使われるヨーレート (今どれくらいの速さで旋回しているか) は速度に依存してしまう。だから「道の形」を素直に表したいなら曲率の方が筋が良い、という話です。

## 基礎: 前提となる概念

- egomotion (エゴモーション / 自車運動): 「ego = 自分」の動きという意味で、自分の車の速度・加速度・姿勢 (向き) などの時系列です。GPS/IMU (慣性計測装置) やホイールオドメトリから得られます。
- yaw (ヨー): 車を真上から見たときの、左右の回転 (鉛直軸まわりの回転) です。車が左に曲がればヨー角が増え、右に曲がれば減ります。船首が左右に振れるのと同じ向きの回転です。航空機・船舶由来の用語で、他に pitch (前後の傾き) と roll (左右の傾き) があります。
- yaw_rate (ヨーレート): ヨー角が 1 秒あたりどれだけ変化するか (角速度, 単位 rad/s)。「今どれくらいの速さで旋回しているか」を表します。
- curvature (曲率): 車が描く軌跡の曲がり具合です。単位は rad/m (進んだ距離 1m あたり何ラジアン向きが変わるか)。直進なら曲率 0、急カーブなら大きな値。半径 r の円を走るとき曲率は `κ = 1/r` です。半径 100m のカーブなら曲率 0.01 rad/m。
- クォータニオン (quaternion): 3 次元の回転を 4 つの数 (qx, qy, qz, qw) で表す方式。ジンバルロック (特定姿勢で自由度が縮退する問題) を避けられるため、姿勢記録に広く使われます。人間に読みやすいオイラー角 (各軸まわりの角度) には変換が必要です。

3 者の関係は車両運動学で `yaw_rate = speed × curvature` です。同じ曲率の道でも、速く走るほどヨーレートは大きくなります。逆に言うと `curvature = yaw_rate / speed` で曲率を代理計算できますが、ここに落とし穴があります。

### なぜ curvature を「直接」使うのか

旋回の度合いを表す量として、多くの実装は `yaw_rate` を使い、それを速度で割って曲率の代理 `curvature ≈ yaw_rate / speed` を計算します。しかしこの代理には弱点があります。停車中や低速時は `speed` が 0 に近づくため、`yaw_rate / speed` がゼロ割りに近づいて値が暴れます (発散したり nan になったり)。信号待ちで微妙に車体が揺れただけでも、speed がほぼ 0 なら曲率代理が巨大な値に跳ねる、という事故が起きます。

データセットによっては自車運動ログに `curvature` 列がそのまま入っています。そうしたデータでは「代理計算をせず、記録された curvature を直接使う」のが安全な設計判断です。これにより低速時の数値不安定を避けられ、しかも軌道予測の出力単位 (曲率) とも一致します ([軌道教師信号 [加速度, 曲率]×ホライズン](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0842_trajectory-supervision-target))。出力と入力で単位系を揃えると、モデルが学ぶべき写像が単純になります。

## 仕組みを詳しく

履歴信号は、1 ステップあたり 4 つの数値を並べる関数で作ります。

```python
def to_history_signals(state) -> np.ndarray:
    vx = state.velocity[:, 0]
    vy = state.velocity[:, 1]
    speed = np.sqrt(vx ** 2 + vy ** 2)                       # [0] 速度 m/s
    acceleration = state.acceleration[:, 0]                  # [1] 加速度 m/s^2 (ax)
    yaw_angle = state.pose.rotation.as_euler("ZYX")[:, 0]    # [2] ヨー角 rad
    curvature = state.curvature[:, 0]                        # [3] 曲率 rad/m
    return np.stack([speed, acceleration, yaw_angle, curvature], axis=1).astype(np.float32)
```

各信号の意味と作り方を分解します。

- `[0]` 速度 (speed): 前後方向の速度 `vx` と横方向 `vy` から `sqrt(vx^2 + vy^2)` で大きさ (スカラー) を求めます。向きを捨てて「どれくらい速いか」だけにしています。
- `[1]` 加速度 (acceleration): `ax` (進行方向の加速度) をそのまま使います。正なら加速、負なら減速。
- `[2]` ヨー角 (yaw_angle): 車の姿勢はクォータニオンで記録されています。これを `as_euler("ZYX")` でオイラー角 (Z 軸・Y 軸・X 軸まわりの角度) に変換し、先頭の Z 軸まわり成分 = ヨー角を取り出します。`"ZYX"` は回転を Z→Y→X の順に適用する規約 (intrinsic rotation) の指定です。
- `[3]` 曲率 (curvature): ログの列を直接読みます。代理計算なし。

`state.acceleration[:, 0]` の `[:, 0]` は、配列の各時刻について最初の成分 (進行方向成分) だけを取り出す操作です。

これらを `np.stack(..., axis=1)` で時刻方向に並べると、形は `(T, 4)` になります (T はステップ数)。過去の窓が 64 ステップなら `(64, 4)`。これを上流で `.flatten()` して 1 本のベクトル `(256,)` にし、モデルの入力 `egomotion_history` にします。256 = 64 ステップ × 4 信号です。

### 形 (shape) のまとめ

```
EgomotionState (64 ステップ分)
   → to_history_signals → (64, 4)   行=時刻, 列=[speed, accel, yaw_angle, curvature]
   → flatten            → (256,)    モデル入力 egomotion_history
```

### 設計上の注意

flatten で `(64, 4)` を `(256,)` にすると、時刻の構造 (どの 4 つが同じ時刻か) は MLP にとって暗黙の並びになります。MLP は全結合なので並びを学習できますが、時系列構造を陽に活かしたいなら `(64, 4)` のまま 1D 畳み込みや RNN/Transformer に通す設計もあり得ます。flatten + MLP は最も単純な選択で、まず疎通させるには十分です。

## 手法の系譜と主要論文

自車運動を入力に使う設計は、模倣学習 (imitation learning, 人間の運転を真似て学ぶ) ベースの end-to-end 運転で広く使われます。

- ALVINN (Pomerleau, NeurIPS 1989): ニューラルネットで人間の操舵を真似た最初期の研究。画像から操舵角を直接出力する end-to-end の原型ですが、運動履歴は使っていません。
- PilotNet (Bojarski et al., NVIDIA, 2016, arXiv:1604.07316): CNN で画像から操舵を回帰。やはり単一フレーム中心で、運動の連続性は陽には扱いませんでした。
- ChauffeurNet (Bansal et al., RSS 2019, arXiv:1812.03079, Waymo): 過去の自車軌跡や速度を入力に含めて将来軌跡を予測しました。狙いは「運動の連続性をモデルに教える」ことです。重要なのは、この論文が「過去の運動を強く入れると、過去をなぞるだけ (慣性的に直進し続ける) 近道学習に陥る」問題を明確に指摘し、合成摂動 (trajectory perturbation, わざと軌道を乱して復帰を学ばせる) と補助損失でこれを緩和したことです。運動履歴は諸刃の剣だ、という教訓を与えました。
- 車両運動学による横方向表現 (kinematic bicycle model, 古典的): 操舵を曲率で表すのが運動学的に自然です。曲率は速度に依存しない幾何量なので、速度が変わっても同じカーブは同じ曲率になります。yaw_rate は速度依存 (同じカーブでも速いほど大きい) なので、曲率の方が「道の形」を素直に表します。曲率を直接使うのはこの利点を活かす選択です。
- end-to-end 統合の現在 (UniAD: Hu et al., CVPR 2023 best paper, arXiv:2212.10156): 知覚から計画までを統合したネットワークでも、自車状態 (速度・運動履歴) は計画ヘッドの重要な条件入力として使われ続けています。

## 論文の実験結果 (定量データ)

指標は軌道予測・計画の ADE (Average Displacement Error, 予測軌道と正解の各時刻の距離の平均、メートル) と FDE (Final Displacement Error, 最終時刻の距離)、それに collision rate (衝突率、%) や intervention (人間の介入回数) です。いずれも小さいほど良い。

- ChauffeurNet (2019) のアブレーションが最も示唆的です。過去の運動を素直に与えただけのモデルは閉ループ評価 (実際に走らせる評価) で「過去をなぞって直進し続け、カーブで離脱する」失敗を起こしました。著者らが軌道摂動と補助損失 (車線逸脱・衝突を罰する項) を加えると、閉ループでの離脱・衝突が大幅に減ったと報告しています。これは「運動履歴を入れるだけでは不十分で、近道学習を抑える仕掛けが要る」ことの定量的証拠です。
- 一般に、自車運動履歴を入力から除くと開ループ ADE/FDE は悪化します (運動の連続性が使えないため)。一方、入れすぎると開ループの数値は良くても閉ループで破綻する、という開ループ・閉ループの乖離が広く観測されています。開ループ指標 (記録データに対する 1 ステップ予測の誤差) だけ見ていると、この危険を見落とします。
- 曲率を直接使うか代理計算するかの比較は、低速・停車を多く含むデータで差が出ます。`yaw_rate / speed` 代理は速度に下限クリップ (たとえば speed < 0.5 m/s で 0 に固定) を入れないと nan/発散で学習が不安定になることが知られ、記録済み curvature を使えばこの前処理自体が不要になります。これは「精度」というより「学習の安定性とコード簡潔さ」の改善です。

## メリット・トレードオフ・限界

メリット
- 曲率を直接使うことで、低速・停車時の数値不安定 (ゼロ割り) を回避できる。
- 曲率は速度非依存の幾何量なので「道の曲がり方」を素直に表現でき、出力の軌道表現とも単位が揃う (学習写像が単純になる)。
- 速度・加速度・ヨー角・曲率の 4 信号で、縦方向 (前後) と横方向 (旋回) の運動状態をコンパクトに表せる。256 次元という小さな入力で済む。

トレードオフ・限界
- 過去の運動を強く入れると「過去をなぞるだけ」の近道学習に陥りやすい (ChauffeurNet の指摘)。軌道摂動・補助損失・履歴のドロップアウトなどの対策が別途要る。これは入力設計だけでは解けない、学習設計と一体の課題です。
- curvature の品質はデータ提供側の計算に依存する。値が荒い (微分由来のノイズが乗っている) と履歴ノイズが増え、モデルがノイズを学習してしまう恐れがある。
- 速度を `sqrt(vx^2+vy^2)` のスカラーにする際に進行方向の符号 (前進/後退) 情報を捨てている。後退 (バック) を扱うシーンでは速度の符号が必要になり、この設計では不足する。
- flatten + MLP は時系列の順序構造を陽に保持しないため、長い履歴の時間依存を捉えるには 1D 畳み込みや Transformer の方が適することがある。

## 発展トピック・研究の最前線
- 近道学習 (causal confusion / copycat problem) の研究: 運動履歴を使う模倣学習が「過去の出力をコピーするだけ」に陥る現象を理論・実験で解析し、履歴のマスキングや因果的介入で緩和する研究が続いています。
- 運動状態のより豊かな表現: 4 信号に加え、横加速度・操舵角・ジャークなどを足す、あるいは生のクォータニオンを保持する設計。表現力と過学習・近道のトレードオフがあります。
- 入力としての運動と出力としての運動の単位統一: 曲率で入出力を揃えるように、表現の整合をとることで学習を安定させる設計思想は、軌道計画全般に通じます ([軌道教師信号 [加速度, 曲率]×ホライズン](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0842_trajectory-supervision-target))。
