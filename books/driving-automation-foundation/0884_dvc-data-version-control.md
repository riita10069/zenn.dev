# DVC (Data Version Control)

## ひとことで言うと
DVC(Data Version Control、データ・バージョン・コントロール)は、Git の使い勝手のまま「何ギガバイトもある大きなデータファイルや学習済みモデルにもバージョンを付けて管理する」ためのコマンドラインツールです。Git はソースコードのような小さなテキストファイルの履歴管理は得意ですが、巨大な画像データセットやモデルの重みファイルは苦手です。DVC はその弱点をちょうど埋めるように設計されています。「同じコミットをチェックアウトすれば、同じコード・同じデータ・同じモデルが揃う」という機械学習の再現性を、Git を中心に据えて実現する代表的な手段です。

## 直感的な理解
機械学習の実験では、こういう困りごとが頻繁に起きます。「3 週間前はうまくいっていたモデルが、データを更新したら急に精度が落ちた。何が原因かわからない」。原因の候補は、コードの変更かもしれないし、データセットの差し替えかもしれないし、前処理のパラメータかもしれません。これを切り分けるには「あのときのコード」と「あのときのデータ」を両方とも正確に再現できる必要があります。

コードの方は Git があるので、過去のコミットに戻せば確実に再現できます。問題はデータです。「dataset_final_v2_本当に最終.zip」のようなファイル名で管理していては、どのファイルがどの実験に対応するのか、すぐにわからなくなります。DVC は、この「データにもコードと同じレベルの厳密なバージョン管理を与えたい」という欲求を満たします。Git のコミットに戻すと、それにひも付くデータも自動でそのときの状態に戻る。これが DVC の提供する直感です。

身近な比喩で言えば、Git は「文章の編集履歴」、DVC は「文章に添付された巨大な動画ファイルの版管理」を担い、両者が一つのタイムラインで同期している状態を作る、と考えるとわかりやすいです。重要なのは「同期している」という点です。コードだけ、データだけが正しくても再現はできません。両方が一本の時間軸の上で対応づいて初めて、過去の任意の時点をまるごと復元できます。

## 基礎: 前提となる概念
DVC を理解するには、いくつかの土台となる概念を押さえる必要があります。知らない前提で順番に説明します。

Git とは、ソースコードの「スナップショット(ある時点の全体の状態)」を記録し、いつでも過去に戻れるツールです。誰がいつ何を変えたかがすべて残るため、ソフトウェア開発に欠かせません。Git は内部では、ファイルの中身から計算した SHA-1 ハッシュ値(後述)でオブジェクトを管理しています。

ハッシュ値とは、ファイルの中身を入力すると固定長の短い文字列(指紋のようなもの)を返す関数の出力です。中身が 1 バイトでも違えば、出力はまったく違う値になります。たとえば SHA-256 なら、どんな大きさのファイルでも 64 文字の 16 進数(`a1b2c3...`)に変換されます。この性質から、ハッシュ値が一致すれば「中身が同じ」と高い確からしさで断定でき、ファイルが壊れていないかの検証(完全性チェック)にも使えます。DVC は既定では MD5 を、設定によってはより衝突に強いハッシュを使います。MD5 は暗号用途では弱いとされますが、悪意のない一般データの同一性判定には十分実用的です。

コンテンツアドレス指定ストレージ(content-addressable storage、CAS)とは、ファイルを「置き場所の名前」ではなく「中身のハッシュ値」で名前付けして保存する方式です。同じ中身のファイルは同じハッシュになるので、自動的に重複排除(同じものを二重に保存しない)が効きます。Git の内部構造もこれです。DVC はこの考え方をデータファイルへ適用します。

リモートストレージとは、巨大なファイルの実体を置いておく外部の保管庫のことです。S3(Amazon の object storage)、GCS、Azure Blob、あるいは SSH 越しのサーバー、HDFS など、さまざまな種類が選べます。DVC はこの抽象化により、ストレージの種類を意識せず同じコマンドで push/pull できます。

object storage(オブジェクトストレージ)とは、ファイルを「キー(名前)と中身の対」として大量に保管するクラウドの保管庫です。階層的なディレクトリではなく巨大な平坦な名前空間で、ほぼ無限に近い容量を低コストで持てます。DVC のリモートとして最もよく使われます。

## 仕組みを詳しく
DVC の核心的なアイデアは「巨大ファイルそのものは Git に入れず、別の保管庫に置く。Git には『そのファイルの指紋(ハッシュ値)を書いた小さなテキスト』だけを入れる」というものです。なぜこの設計かというと、Git はファイルの全バージョンを履歴として保持するため、巨大ファイルを直接入れるとリポジトリが肥大化し、クローン(取得)に何時間もかかったり、サーバー容量を食いつぶしたりするからです。Git の delta 圧縮はテキストの差分には強いですが、すでに圧縮済みのバイナリ(JPEG、動画、モデルの重み)には効きにくく、肥大化が顕著になります。

具体的な流れをステップで見ます。

1. データを追跡対象にする。
   ```
   dvc add data/dataset.zip
   ```
   このとき DVC は次の 2 つをします。
   - `data/dataset.zip` の中身からハッシュ値(指紋。例 `a1b2c3...`)を計算する。
   - `data/dataset.zip.dvc` という小さなテキストファイル(数十バイト)を作り、そこにハッシュ値とファイルサイズを書く。実体ファイルは `.gitignore` に追加され、Git の追跡から外れる。

   `.dvc` ファイルの中身は次のように人間が読める YAML です。
   ```yaml
   outs:
   - md5: a1b2c3d4e5f6...
     size: 209715200
     path: dataset.zip
   ```

2. Git にはこの `.dvc` ファイルと `.gitignore` だけをコミットする。
   ```
   git add data/dataset.zip.dvc .gitignore
   git commit -m "track dataset v1"
   ```
   Git に入るのは数十バイトのテキストだけなので、リポジトリは軽いままです。

3. 実体は別の保管庫(例として S3)へ送る。
   ```
   dvc remote add -d storage s3://my-bucket/dvc-store
   dvc push
   ```
   実体ファイルはハッシュ値で名前付けされて保存されます。たとえば `a1/b2c3...` のように、ハッシュの先頭 2 文字をディレクトリにする CAS 形式です。先頭 2 文字でディレクトリを切るのは、一つのディレクトリにファイルが集中しすぎないための古典的な工夫(Git も同じ)です。

4. 別の人やマシンで取得するときは次のようにします。
   ```
   git clone <repo>      # コード + .dvc ファイル(軽い)を取得
   dvc pull              # .dvc に書かれたハッシュを見て、保管庫から実体を取得
   ```

ここで決定的に重要なのは、Git のコミットを `git checkout <過去のコミット>` で過去に戻すと `.dvc` ファイルの中身も過去のハッシュに戻り、続けて `dvc checkout` を実行すると、そのハッシュに対応する過去のデータ実体が作業ディレクトリに復元される点です。つまり「コードを 3 週間前に戻したら、データも自動でそのときのバージョンに揃う」が実現します。コードとデータが一本のタイムラインで同期するわけです。なお作業ディレクトリへの復元では、DVC は可能ならファイルのコピーではなくリフリンク・ハードリンク・シンボリックリンクを使い、ローカルキャッシュから瞬時に実体を出します。これにより数十 GB の「チェックアウト」が一瞬で終わります。

DVC にはもう 1 つ「パイプライン機能」があります。`dvc.yaml` に「前処理 → 学習 → 評価」の各ステージと、その入力(deps)・出力(outs)・実行コマンドを書いておくと、`dvc repro` を実行したときに「入力やコードのハッシュが変わったステージだけ」を賢く再実行します。

```yaml
stages:
  preprocess:
    cmd: python preprocess.py data/raw data/clean
    deps: [data/raw, preprocess.py]
    outs: [data/clean]
  train:
    cmd: python train.py data/clean model.pt
    deps: [data/clean, train.py]
    params: [lr, batch_size]
    outs: [model.pt]
    metrics: [metrics.json]
```

変わっていないステージは過去の結果(run-cache)を使い回すため、無駄な再計算を避けられます。これは Make の「変更があったターゲットだけビルドする」発想をデータ・モデルにまで拡張したものです。`params` で囲ったハイパーパラメータは別ファイル(`params.yaml`)に書き、その値が変わったステージだけ再実行される、という細やかな依存追跡もできます。

数値例で重複排除の効果を見ます。100 GB のデータセットのうち 1 ファイル(50 MB)だけ差し替えた場合、変わった 1 ファイルのハッシュだけが更新され、残り 99.95 GB は既存の実体を再利用します。新しく保存されるのは 50 MB だけです。これがコンテンツアドレス指定ストレージの威力です。ディレクトリを追跡する場合、DVC は中の各ファイルのハッシュを並べた一覧ファイル(`.dir` オブジェクト)を作り、その一覧自体のハッシュでディレクトリ全体を表すため、ファイル単位の細かい重複排除が効きます。

## 手法の系譜と主要論文
DVC は Iterative.ai 社が 2017 年ごろから開発するオープンソースツールであり、特定の論文が一度に提案した手法ではありません。しかし背後にある考え方は、確立された技術の系譜の上にあります。系譜として整理します。

第一に、コンテンツアドレス指定と Merkle ツリー(各ノードが子のハッシュを持つ木構造で、全体の完全性を一つのルートハッシュで検証できる)の考え方は、古くからの分散システムの基礎技術です。これは Linus Torvalds が 2005 年に Git を設計したときの内部表現に結実し、DVC はその発想をデータファイルへ持ち込みました。

第二に、巨大ファイルを Git ポインタで扱う直接の先行者が Git LFS(Large File Storage、2015 年に GitHub が公開)です。Git LFS は「実体は別保管庫、Git にはポインタ」という点で DVC とよく似ていますが、ML パイプラインや run-cache、データ依存の差分実行のような機能はなく、純粋にファイル保管に特化しています。また Git LFS はサーバー側に専用の LFS サーバーを必要とするのに対し、DVC は汎用の object storage をそのまま使える点が運用上の差です。DVC は Git LFS のアイデアを取り込みつつ、ML 実験の再現性とパイプライン機能まで広げた、という関係になります。

第三に、再現性を「データのバージョン固定」の観点から論じた研究として、Pineau ら "Improving Reproducibility in Machine Learning Research"(JMLR 2021、NeurIPS 2019 の再現性チェックリストを発展させたもの)があります。ここでは「コードだけでなく、データのバージョン・前処理・環境を固定しなければ実験は再現できない」と主張されており、DVC はまさにこの主張をツールとして体現します。

第四に、より大規模なデータレイク全体をバージョン管理する系譜として lakeFS(Treeverse 社、2020 年〜)があります。これは object storage(S3 など)そのものに Git ライクなブランチ・コミット・マージを被せる方式で、DVC が「Git リポジトリにひも付くファイル群」を管理するのに対し、lakeFS は「バケット全体」を管理する、というスケールの違いがあります。さらに古くは Pachyderm(2016)が、データのバージョン管理と Kubernetes 上のパイプラインを統合する方式を提示しました。系譜としては「個人〜小チームの再現性(DVC)」と「組織規模のデータレイク(lakeFS / Pachyderm)」が並走している状態です。

第五に、データのカタログ化・来歴追跡の研究として Halevy ら "Goods: Organizing Google's Datasets"(SIGMOD 2016)があります。これは社内に散在する数十億のデータセットを自動で発見・索引付けし、来歴(どこから来てどう使われたか)をたどる試みで、「データを資産として体系的に管理する」という問題意識の先駆けです。DVC が個々のプロジェクトで担う再現性を、組織全体で扱うとどうなるか、を考える上での参照点になります。

## 論文の実験結果(定量データ)
DVC は学術的なベンチマーク論文を持つ手法ではないため、論文上の精度向上の数値という形のデータは存在しません。代わりに、その価値は「ストレージ効率」「取得時間」「再現性」という運用上の定量指標で測られます。これらは初心者にもわかる具体的な数字で説明できます。

ストレージと取得時間の例。仮に 200 GB のデータセットを直接 Git に入れた場合、毎回の変更で履歴が積み上がり、リポジトリは数百 GB から TB 級に膨らみ、`git clone` は実用的でなくなります。DVC を使えば Git リポジトリは数百 KB(.dvc ファイルとコードのみ)に保たれ、`git clone` は数秒で終わります。実体が必要になったときだけ `dvc pull` で取得します。この「リポジトリサイズが TB から KB へ」という差が、DVC の最も直接的な定量効果です。クローン時間で言えば、巨大バイナリ込みのリポジトリで数十分から数時間かかっていたものが、メタデータのみなら数秒というオーダーの違いになります。

重複排除の効果。前述のとおり、100 GB のうち 50 MB だけ変えた更新では、追加ストレージは 50 MB で済みます。直接保管なら 100 GB を丸ごとコピーするため、約 2000 倍の差になります。10 回更新すれば、素朴な保管は 1 TB を消費するのに対し、DVC は 100 GB + 50 MB × 10 = 約 100.5 GB に収まります。

パイプラインの再実行削減。`dvc repro` は変更のないステージをスキップします。前処理に 2 時間、学習に 6 時間かかるパイプラインで、学習のハイパーパラメータだけを変えた場合、前処理 2 時間分は run-cache から復元され、再計算は学習の 6 時間だけになります。前処理を毎回やり直す素朴な運用と比べ、この例では実行時間が 8 時間から 6 時間へ短縮されます。前処理ステージが重い(例 24 時間)ほど、この節約は劇的になります。

再現性の検証は「アブレーション」に近い実験として理解できます。「データのバージョン固定を外す」とどうなるか。Pineau らの議論にあるとおり、データを固定しないと同じコードでも結果が再現できず、報告された精度を第三者が確認できなくなります。DVC でデータバージョンを固定する効果は、この「再現の成否」という二値の指標で測られ、これは ML 研究の信頼性に直結する重要な差です。彼らの再現性チェックリストは多くの主要会議(NeurIPS など)で投稿要件に採り入れられ、データとコードの公開・固定が事実上の標準になりました。

## メリット・トレードオフ・限界
メリットを整理します。Git の使い勝手そのままで巨大なデータ・モデルにバージョンを付けられること。コードのコミットとデータのバージョンが厳密にひも付き、再現性が高いこと。ハッシュベースの重複排除で変更分だけを効率よく保存できること。`dvc repro` のパイプラインキャッシュで無駄な再計算を避けられること。バックエンドに S3 / GCS / Azure / SSH / HDFS など多様なストレージを選べること。ローカルキャッシュからのリンクで巨大ファイルのチェックアウトが瞬時に終わること。

トレードオフと限界も率直に挙げます。Git と DVC の二つを併用するメンタルモデルが必要で、初心者は `git checkout` と `dvc checkout` の二段操作に最初戸惑います(`dvc` を Git フックに仕込んで自動化する手はあります)。巨大ファイルの push/pull はネットワーク帯域に律速され、数百 GB の転送には相応の時間がかかります。データの実体は別ストレージ任せなので、保管庫側の容量・権限・ライフサイクルの運用は別途必要です。多人数で同じファイルを同時に書き換えると、Git のマージと違ってバイナリの自動マージはできず、どちらか一方を選ぶしかありません。そして根本的な限界として、DVC は「Git リポジトリにひも付くファイル群」を単位とするため、複数チームが同じデータレイクへ並行に読み書きする組織規模の運用では、ブランチ・マージの境界をストレージ側に置ける lakeFS のような方式の方が向く場面があります。

研究上の未解決課題としては、「データのバージョン管理」と「データの来歴(どの生データから、どの変換を経て作られたか、というリネージ)」を統合的にたどる仕組みがまだ発展途上である点が挙げられます。ハッシュで同一性は保証できても、「なぜそのデータになったか」「個人情報を含む行を後から削除したい(忘れられる権利)」といった意味的・規制的な要求への対応は、メタデータ管理ツールや専用のデータガバナンス基盤との連携に委ねられているのが現状です。

## 発展トピック・研究の最前線
さらに深く学ぶ方向として、まず DVC と実験管理ツールの統合があります。DVCLive のように、学習中の指標を `.dvc` のパイプラインと結び付けて記録する仕組みが整備されつつあり、「データバージョン × ハイパーパラメータ × 結果」を一つの軸でたどる試みが進んでいます。`dvc exp`(実験機能)では、Git のブランチを汚さずに多数のハイパーパラメータ試行を管理し、後から良かったものだけを本流にコミットできます。

次に、データバージョン管理とデータレイクの融合という流れがあります。lakeFS や Pachyderm のように、object storage そのものにバージョン制御を持たせる方式と、DVC のように Git 中心で軽量に始める方式は、それぞれの利点を取り込み合いながら発展しています。組織が成長すると前者へ移行する、という現実的な選択もしばしば見られます。テーブル形式データでは Apache Iceberg や Delta Lake の「タイムトラベル(過去の時点のテーブルを読む)」機能が、構造化データに対する同種の再現性をもたらしています。

最前線としては、データセントリック AI(モデルではなくデータの質の改善に焦点を当てる潮流。Andrew Ng らが提唱)の文脈で、データのバージョン管理が再注目されています。「どのデータを足し引きすると性能がどう変わるか」を系統的に実験するには、データの各バージョンを厳密に固定・再現できる基盤が不可欠であり、DVC 的なツールはその土台として位置づけられます。大規模言語モデルの学習では、学習コーパスの構成(どのデータソースをどの比率で混ぜたか)が性能を大きく左右することが知られており、コーパスのバージョン管理と来歴追跡は研究上ますます重要なテーマになっています。
