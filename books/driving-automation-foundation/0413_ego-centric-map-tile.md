# Ego中心マップタイルのレンダリング

## ひとことで言うと

地図タイルを「自車(ego)を画面の中央に固定し、車の進行方向がいつも真上を向く」ように描く方法です。地球上の緯度経度をいったんメートル単位の平面に直し、車の向きの分だけ回転させてから描画します。これによりモデルは「常に自分目線・前方が上」の安定した地図を見られるようになります。カーナビの「進行方向アップ」表示と同じ発想を、機械学習の入力作りに持ち込んだものだと考えてください。

## 直感的な理解

地図をモデルに見せるとき、向きの基準をどう取るかが最初の設計判断になります。素朴に「北が上(north-up)」で描くと、車が南向きに走っているときは進行方向が画面の下、東向きなら右、というふうに毎回バラバラになります。モデルが地図から読み取りたいのは「自分がこれからどっちへ進むべきか」なのに、向きが毎回変わると、同じ「直進」という状況でも画面上では全然違う見え方になり、学習が難しくなります。

ここで ego 中心(ego-centric)・heading 上向きという基準を使います。これは「常に自車を画面中央に置き、車の前方を画面の上に固定する」描き方です。こうすると、どんな方角に走っていても「いま自分が向いている方向を上にした地図」に統一されます。直進中はいつも経路がまっすぐ上に伸び、右折時はいつも経路が右に折れる、という具合に、状況と見え方が一対一で対応します。これは入力の正規化(normalization、入力のばらつきを減らして揃える操作)の一種で、モデルが学ぶべきパターンの種類を減らし、データ効率を上げます。

なぜ正規化が効くかを直感で言うと、モデルは「世界の絶対的な向き」ではなく「自分から見た相対的な配置」で運転判断をすればよいからです。北がどっちかは運転には関係なく、関係するのは「自分の前方に道がどう続いているか」です。だから入力の段階で絶対方位を捨て、自車相対の座標系に揃えてしまうのが理にかなっています。これは画像認識でデータ拡張として回転を加えるのとは逆向きの発想で、「不要な変動をあえて消す」ことで学習を楽にします。

## 基礎: 前提となる概念

緯度経度と地図投影: 地球はほぼ回転楕円体なので、その表面の位置は緯度と経度という角度で表されます。これを平らな紙(=画像)に描くには「地図投影(map projection)」という変換が要ります。地球を平面に押し付けると必ずどこかが歪むため、用途に応じて投影法を選びます。狭い範囲なら「接平面近似」という、地球表面を局所的に平らとみなす簡易な投影で十分です(Snyder, 1987, USGS の地図投影マニュアルが古典的な参考書です)。

経度の縮み: 経度1度あたりの実距離は緯度によって変わります。赤道では経度1度が約111kmですが、北緯60度では地球の輪切りの円(緯線)の半径が小さくなるため約半分の約56kmになります。緯度1度はどこでも約111kmでほぼ一定です。この「経度の縮み」を補正しないと、高緯度ほど東西方向が間延びした歪んだ地図になります。

ヘディング(heading): 車の進行方向を方位として表した角度です。慣習によって基準が異なりますが、ここでは「北を0、東を正の方向(時計回り)」で測るものとします。GPS の連続する2点の差分から計算できます。

回転行列: 2次元平面上で点を原点まわりに角度θだけ回す操作は、2×2の行列の掛け算で表せます。これを使って「世界を車の向きと逆に回す」ことで、車の前方を画面の上に持ってきます。

ラジアン: 角度の単位で、180度がπ(約3.14159)に対応します。三角関数(sin, cos)はラジアンを引数に取ります。度をラジアンに直すには π/180 を掛けます。

## 仕組みを詳しく

座標変換の核心は「平面化 → 回転 → 固定窓での描画」の3段階です。順に、数式と数値例で追います。

### ステップ1: 緯度経度をメートル平面に直す(等距円筒近似)

緯度経度をそのまま XY 座標として描くと、前述の経度の縮みで歪みます。そこで自車の周りの狭い範囲だけを平らとみなす「等距円筒近似(equirectangular approximation)」で、東方向 east と北方向 north のメートル座標に変換します。

```
x_east  = (lon - ego_lon) * cos(ego_lat) * R * (pi / 180)
y_north = (lat - ego_lat)                 * R * (pi / 180)
```

ここで R は地球半径(約6,378,137m)、lon・lat は変換したい点の経度・緯度、ego_lon・ego_lat は自車の経度・緯度です。`(pi/180)` は度をラジアンに直す係数、`cos(ego_lat)` が経度の縮みを補正する係数です。

数値例で確認します。自車が北緯35度にいるとして、cos(35°)≈0.819 です。つまり経度方向の距離を約82%に縮めて正しいメートルに直します。緯度0.001度の差は、`0.001 × R × (pi/180) ≈ 111m` になります。経度0.001度の差は、`0.001 × 0.819 × R × (pi/180) ≈ 91m` です。同じ0.001度でも、経度方向のほうが東西に縮んでいるのがわかります。この補正によって、緯度経度の歪みが消え、メートル単位の平らな (east, north) 平面が得られます。

### ステップ2: 自車の向きだけ回転させる(heading 上向き)

平面化しただけだと「北が上」の状態です。これを「車の前方が上」にするため、自車の向き heading(ラジアン)の分だけ逆回転、つまり -heading 回します。標準的な2次元回転行列を使います。

```
x_local = x_east * cos(-h) - y_north * sin(-h)
y_local = x_east * sin(-h) + y_north * cos(-h)
```

ここで h = ego_heading です。マイナスを付けて回すのがポイントで、「世界を車の向きと逆に回す」ことにより、結果として車の前方が +y(画面の上)に来ます。h = 0(北向き)なら回転はゼロで、北が上のままです。

数値で確かめます。車が真東(heading = 90° = π/2)を向いて走っているとします。自車の真東(=前方)にある点は east 軸上、つまり (x_east, y_north) = (d, 0)(d は距離)です。これを -90° 回すと、

```
x_local = d * cos(-90°) - 0 * sin(-90°) = d * 0  = 0
y_local = d * sin(-90°) + 0 * cos(-90°) = d * (-1) = -d
```

符号の扱いは座標系の取り方次第で、描画側で +y を画面上に対応させるか反転させるかを合わせる必要がありますが、要は「前方の点が x=0 の縦軸上に移動する」ことが保証されます。前方が縦軸に乗り、横方向(左右)が x 軸に乗る、という自車相対の安定した配置が得られます。直進中の経路はいつも縦軸に沿って伸び、右左折はいつも左右の折れとして現れます。

### ステップ3: メトリック窓を固定して等アスペクトで描画する

描画範囲は自車を中心に ±R メートルの正方形(または用途に応じた長方形)に固定します。縦横の縮尺を等しくする(等アスペクト、equal aspect)ことが重要で、これをしないとメートル平面の比率が崩れて道路の形が歪みます。

```
xlim = (-R, R)
ylim = (-R, R)
set_aspect("equal")    # メートル平面 — 緯度経度の歪みなし
set_axis_off()         # 軸・目盛りを消して余白ゼロにする
```

この窓の中に、背景・道路・経路・GPS 点を所定のパレットで重ねて描き、軸を消して余白なしの画像に書き出します。固定窓にすることで、出力タイルの「1ピクセルが何メートルに相当するか」が全クリップで一定になり、モデルがスケールを学びやすくなります。たとえば窓が ±800m で画像が 640px 幅なら、解像度は 1600m / 640px = 2.5m/px です。この値が全クリップで揃っていれば、CNN は「画面上で何ピクセル先=実世界で何メートル先」という対応を1つ覚えればよくなります。

こうして「自車中央・前方上・等縮尺・既存タイルと同配色」のタイルが完成します。

## 手法の系譜と主要論文

自車中心のラスタ地図表現を運転方策学習に持ち込んだ代表が ChauffeurNet(Bansal et al., 2018, arXiv:1812.03079, Waymo)です。自車を中央・進行方向上向きに固定したトップダウン画像に、道路・経路・周辺エージェントを別チャンネルで描き、その積み重ねを CNN に入れます。論文は「自車を原点・進行方向で正規化したトップダウン表現にすると、模倣学習が安定し、長時間の閉ループ走行でも破綻しにくい」ことを主張しています。この「ego 正規化」の考え方が、その後の多くの地図入力設計の基盤になりました。

ベクトル地図側でも同じ正規化思想が見られます。VectorNet(Gao et al., 2020, CVPR)や LaneGCN(Liang et al., 2020, ECCV)は、車線や軌跡を点列(ベクトル)として扱いますが、いずれも「自車を原点・進行方向で正規化した局所座標系」で表現します。違いは表現の媒体だけで、ラスタ地図は画像(ピクセル)、ベクトル地図は点列で同じ正規化を実現しています。つまり「自車相対に正規化する」という発想は表現方式を超えた共通原理です。

座標変換そのものは測地学(geodesy)の標準で、等距円筒近似や局所接平面近似は Snyder(1987, USGS "Map Projections — A Working Manual")などで体系化されています。狭い範囲(数km以内)では十分高精度で、広域になると ENU(East-North-Up)局所直交座標系や UTM(Universal Transverse Mercator)投影が使われます。地図タイルの半径が800m程度なら、等距円筒近似で実用上問題ありません。

さらにセンサ由来の BEV 表現に目を向けると、Lift-Splat-Shoot(Philion & Fidler, 2020, ECCV)や BEVFormer(Li et al., 2022, ECCV)は、複数カメラ画像から自車中心の BEV 特徴を学習で構築します。ここでも「自車を原点・進行方向上」という正規化基準は受け継がれており、地図を外から与えるか、センサから立ち上げるかの違いだけで、座標系の取り方は共通です。

## 論文の実験結果(定量データ)

ChauffeurNet(Bansal et al., 2018)は、自車正規化したトップダウン地図入力の効果を閉ループ評価で示しました。約26百万例の運転データで学習し、純粋な行動模倣だけのベースラインと比較しています。論文が報告する主要な知見は、ベースラインのモデルは閉ループで走らせると、わずかな誤差が時間とともに積み重なって停止位置をはみ出したり障害物に接近したりする失敗を頻発したのに対し、自車正規化した表現と「最悪ケースの合成(perturbation で軌跡をわざと崩して回復を学ばせる)」を組み合わせると、それらの失敗が大幅に抑えられた、という点です。アブレーションでは、perturbation を除く・補助損失(衝突や脱輪を罰する項)を除くと、それぞれ閉ループでの失敗率が悪化することが示され、自車正規化表現を土台に各要素が積み上がって効いていることがわかります。

座標変換の精度については、等距円筒近似の誤差は範囲が広がるほど増えることが測地学的に知られています。半径800m程度では、緯度経度から平面への変換誤差はせいぜい数十センチ〜1m程度のオーダーで、2.5m/px の地図タイル解像度に対して無視できます。一方、半径を数十kmに広げると地球の曲率の影響が無視できなくなり、周縁部で誤差がメートル〜十メートル級に増えます。これが「狭域なら等距円筒近似で十分、広域なら UTM や ENU が必要」というトレードオフの数値的な裏づけです。

これらの数値の意味は二つあります。第一に、自車正規化という入力設計が、開ループの一発予測ではなく実際に走らせる閉ループでこそ差を生むということ。誤差が積み重なる厳しい評価で初めて、安定した入力表現の価値が表れます。第二に、座標近似の選択は地図タイルのサイズに依存し、運転制御で使う近距離地図(数百m)ならば最も単純な近似で十分だということです。

## メリット・トレードオフ・限界

メリットは、まず地図の見え方が常に「自車中央・前方上」で統一され、モデルが一貫した基準で経路を読めることです。同じ運転状況がいつも同じ画像配置になるので、学習に必要なデータ量が減ります。次に、等距円筒近似で緯度経度の歪みを除き、等アスペクト描画で縦横比を保つので、道路の形が正しく保たれます。さらに、固定窓により1ピクセルあたりのメートル数が一定になり、スケールが安定します。既存の地図入力と同じフレーミングに揃えれば、生成タイルをそのまま差し替えられます。

トレードオフと限界としては、まず heading の精度に強く依存することが挙げられます。heading が不正確だとタイル全体の向きがずれ、モデルに誤った進路を見せてしまいます。GPS の2点差分から heading を推定する場合、停車中や低速時には2点がほぼ同じ位置になり、heading が不定になる(ノイズで暴れる)という弱点があります。実装上は速度がしきい値未満なら直前の安定した heading を保持する、点が足りなければ北上にフォールバックする、といった保険が要ります。次に、等距円筒近似は狭域前提なので、半径を大きく取りすぎると周縁部で誤差が増えます。回転・再描画のコストがクリップごとにかかる点も、オフライン前処理で吸収する前提です。そして、自車中心ゆえ絶対方位(北はどっち)を画像からは読み取れないので、北方位を必要とする下流処理には heading を別途渡す必要があります。

研究上の未解決課題としては、heading が不定になる低速・停車シーンでの安定した向き推定、そして地図の正規化基準(進行方向アップか、目的地方向アップか、固定北上か)が下流タスクの性能にどう影響するかの体系的な比較が挙げられます。軌跡予測では進行方向アップが定番ですが、長距離ナビでは目的地方向アップのほうが意図を表しやすい、という直感があり、どれが最適かはタスク依存で未決着です。

## 発展トピック・研究の最前線

近年の運転モデルは、自車正規化されたトップダウン表現を入力ではなく中間表現として「学習で立ち上げる」方向に進んでいます。BEVFormer(Li et al., 2022, ECCV)や LSS(Lift-Splat-Shoot, Philion & Fidler, 2020, ECCV)系の手法は、複数のカメラ画像から自車中心の BEV 特徴を直接構築します。ここでも自車中心・進行方向基準という正規化思想は受け継がれていますが、地図を外から与えるのではなく、センサからその場で BEV を生成する点が異なります。BEVFormer は nuScenes の3次元物体検出で、当時の単眼・マルチビュー手法を上回る NDS(nuScenes Detection Score、検出精度と位置・速度誤差を総合した指標)を報告し、自車中心 BEV を中間表現に据える設計の有効性を示しました。

また、地図の向き基準そのものを研究対象にする流れもあります。進行方向アップではなく、目的地の方向を上にする・固定方位にするなど、どの基準がどのタスク(軌跡予測、行動方策、地図生成)に最適かは、データセットとタスクに依存することが少しずつわかってきています。座標変換は一見すると枯れた技術ですが、「どの基準系でモデルに世界を見せるか」という問いは、今も設計上の重要な選択肢として残っています。
