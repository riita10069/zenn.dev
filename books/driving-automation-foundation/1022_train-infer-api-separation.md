# 学習/推論API分離（construct_training_data と external loss）

## ひとことで言うと
学習/推論 API 分離とは、生成モデル（軌跡を作るプランナや Diffusion / Flow Matching デコーダ）に対して、「学習のときに呼ぶ入口」と「推論（実運用）のときに呼ぶ入口」を、別々の名前のメソッドにきっぱり分ける設計原則です。`forward()` は必ず「完成した出力（軌跡・画像など）」を返すと約束し（推論専用）、損失計算はそれとは別のメソッドに閉じ込めます。さらに、学習に必要な中間データ（ノイズ・補間・時刻・目標速度）を作る処理を `construct_training_data()` のような専用関数に切り出します。こうすることで「学習モードと推論モードで forward の返り値の意味が入れ替わり、呼ぶ側が取り違える」というバグを、構造的に発生不可能にします。

## 直感的な理解
1つの蛇口から、ひねり方によって「水」と「お湯」が出る設計を想像してください。便利そうですが、急いでいると間違えてやけどします。これに対して「水の蛇口」と「お湯の蛇口」を別々に分けておけば、間違いようがありません。学習/推論 API 分離は、生成モデルにおけるこの「蛇口を分ける」考え方です。

なぜこれが特に生成モデルで重要かというと、Diffusion や Flow Matching では学習と推論でやることが根本的に違うからです。学習時に欲しいのは「ノイズや速度の予測（中間量）」、推論時に欲しいのは「完成したサンプル」。同じ `forward()` がモードによって違うものを返すと、呼ぶ側が「いまどっちが返ってくる設計だったか」を常に覚えていなければなりません。覚え間違えても、多くの場合エラーは出ず、形（tensor の shape）だけは合った無意味な値で学習が静かに進んでしまいます。この「静かに間違える（silent bug, 例外も出ず、ただ学習が収束しないだけ）」のが一番たちが悪いのです。形が合っているせいでアサーションにも引っかからず、デバッグに何日も溶かすことになります。

## 基礎: 前提となる概念
まず、生成モデルにおける「学習目標」と「推論手続き」が別物であることを理解する必要があります。

Flow Matching を例に取ります。正解の軌跡（あるいは画像）を x_1、純粋なノイズを x_0 とします。

- 学習時にやること: ノイズ x_0 と正解 x_1 を時刻 t（0 から 1 の値）で線形補間した中間状態 u_t を作り、その u_t と t をネットに入れて「いまどちらへ動くべきか」という速度 v を予測させ、正解の速度（x_1 - x_0）との二乗誤差を損失にします。つまり学習で必要なのは「速度の予測値と目標速度」です。
- 推論時にやること: 純粋ノイズ x_0 から出発し、ネットが予測した速度場を時間 t について何度も積分（Euler 法など）し、最終的に完成サンプル x_1 を得ます。必要なのは「完成サンプル」です。

学習は1回の forward（速度を1回予測すれば損失が計算できる）で済むのに対し、推論は同じネットを何回も（10〜100 ステップ）呼んで積分する、という非対称性に注目してください。同じネットを使うのに、呼び方も返り値の意味もまったく違うのです。

「多態的（ポリモーフィック）な forward」という言葉も押さえておきます。これは「同じメソッドが、引数（mode など）によって違う型・違う意味の値を返す」設計のことです。柔軟に見えますが、返り値の意味が文脈依存になるため、本トピックが避けたい設計の典型です。

「external loss（外部損失）」という用語も重要です。これは「損失関数の計算をモデルの forward の中に埋め込まず、モデルの外側（学習スクリプトや専用メソッド）で行う」スタイルを指します。対義語は「forward が損失まで計算して返す」内部損失スタイルです。深層学習フレームワーク（PyTorch など）の慣習でも、`nn.Module.forward` は「予測（推論）」を返し、損失は `criterion(pred, target)` のように外で計算するのが標準的で、本トピックはこの慣習を生成モデルにも徹底する話だと言えます。

## 仕組みを詳しく
この設計は、典型的には2〜3個の安定したインターフェース（契約）を定義します。

### 1. forward() は常に推論を行い、完成した出力を返す
mode に依存する中間量（Flow Matching の速度場、Diffusion のノイズ予測）を返してはいけない、という約束です。呼ぶ側は「forward の返り値は必ず完成した形状の出力だ」と無条件に信頼できます。

軌跡プランナの例で、出力形状は `[B, num_timesteps * num_signals]` のように固定されます。たとえば `num_timesteps=64`, `num_signals=2`（x, y の2成分）なら `[B, 128]` です（B はバッチサイズ）。この形状が学習でも推論でも変わらないことが、契約の核心です。

### 2. 損失計算を別メソッド（external loss）に閉じ込める
`compute_loss()` のようなメソッドが、訓練の目的（損失）を計算して返します。ここに、ノイズ・目標速度・時刻といったデコーダ固有の一時テンソルを閉じ込め、呼ぶ側のスコープには漏らしません。漏らさないことで、別のサンプルの目標と取り違える事故が物理的に起きなくなります（変数がそもそもその場所に存在しないから）。

### 3. construct_training_data() で学習用中間データを作る
Flow Matching の学習データ構築は、次のような処理です。

```
def construct_training_data(trajectory_target):   # trajectory_target: [B, 128]
    B   = trajectory_target.shape[0]
    x_0 = randn_like(trajectory_target)              # ノイズ [B, 128]
    t   = sample_timesteps(B, ...)                   # 時刻 [B]（例: Beta(1.5,1) で後半に偏らせる）
    u_t = (1 - t) * x_0 + t * trajectory_target      # 線形補間 [B, 128]
    target_velocity = trajectory_target - x_0         # 正解速度 [B, 128]
    return u_t, t, target_velocity
```

このメソッドは「ノイズ補間 u_t」「時刻 t」「目標速度」をまとめて作ります。損失計算はこの3つを使って:

```
v_pred = v_theta(u_t, t, 条件)        # ネットの速度予測 [B, 128]
loss   = mse_loss(v_pred, target_velocity)
```

t のサンプリングを一様分布ではなく Beta 分布などで偏らせるのは、学習で重視したい時刻領域（たとえば x_1 に近い後半、細部が決まる領域）にサンプルを集めるためのテクニックです。逆に Diffusion では SNR（信号対雑音比）に応じた loss 重み付けで似た効果を出します。どちらも「学習時にしか存在しない調整」であり、推論の forward には現れません。だからこそ分離する意味があります。

`construct_training_data` を「外から呼べる public」にしておく設計には理由があります。高度な呼び出し側が、同じ (u_t, t) を複数の損失項で共有したいとき（たとえば速度損失とは別に正則化項を同じ補間点で測りたいとき）、毎回ノイズを引き直さずに済むようにするためです。ただし正規の経路はあくまで `compute_loss` で、そこは生の速度を外に出しません。

### before / after の対比
```
before（危険な設計）:
    out = model(x, mode="train")     # out は速度? 軌跡? 呼ぶ側が覚えていないといけない
    loss = mse_loss(out, target)     # 取り違えると silent bug（エラーなく無意味な学習）

after（API 分離）:
    loss = model.compute_loss(...)   # 学習はこれ。中間量は内部に隠蔽
    traj = model(...)                # 推論はこれ。返り値は必ず完成軌跡 [B, 128]
```

返り値の「意味」が固定されるため、取り違えようがありません。型システム（mypy など）の助けがなくても、人間がコードを読んだ瞬間に「forward は軌跡」と分かるのが利点です。

## 手法の系譜と主要論文
これは「論文の新手法」というより、生成モデル実装で繰り返し再発見されてきた設計原則（エンジニアリングの作法）です。源流をたどると次のようになります。

Ho et al., 2020, "Denoising Diffusion Probabilistic Models (DDPM)"（NeurIPS, arXiv:2006.11239）。Diffusion を実用化した記念碑的論文。学習目標は「各時刻のノイズ予測の MSE」という単純な形（simplified objective, L_simple）に整理され、サンプリング（推論）は逆過程の反復という別手続きでした。この論文の ε 予測の二乗誤差が、「学習目標」と「推論手続き」が別物であることを明確に示した初期の代表例です。

Lipman et al., 2023, "Flow Matching for Generative Modeling"（ICLR, arXiv:2210.02747）と、その実用実装である Conditional Flow Matching（Tong et al., 2024, TMLR, arXiv:2302.00482, torchcfm ライブラリ）。Flow Matching は構造的に「補間 u_t と目標速度を構築する部分」「速度場を学習する部分」「速度場を積分してサンプルを得る推論部分」が明確に分かれています。`construct_training_data` という関数分割は、この数学的構造をそのままコードに落としたものです。torchcfm の API も「補間サンプルと目標速度を返す関数」と「ODE を解くサンプラー」をはっきり分けており、本トピックの実装上の手本になっています。

Rombach et al., 2022, "Latent Diffusion Models / Stable Diffusion"（CVPR, arXiv:2112.10752）。画像を VAE で潜在空間に落としてから Diffusion する構成で、学習（潜在へのノイズ付与と予測）と推論（潜在のサンプリング＋デコード）の経路が明確に分かれています。大規模実装でこの分離が標準化された代表例です。

Chi et al., 2023, "Diffusion Policy"（RSS, arXiv:2303.04137）。ロボット制御（visuomotor policy）に Diffusion を使い、画像観測から行動列を生成する研究です。ここでも訓練（ノイズ予測）と推論（行動列のサンプリング）の API を分けて実装するのが定石で、行動を「条件付き Diffusion でサンプリングする」枠組みを確立しました。実車・実ロボットの制御に生成モデルを使う際の API 設計のリファレンスになっています。

自動運転の motion planning でも、損失計算をモデル外に出す external loss スタイルが採られることが増えています。プランナの内部実装（GRU / Diffusion / Flow / 回帰）が何であれ、外側の学習ループは「forward は軌跡、損失は専用メソッド」という同じ契約で扱えるため、複数方式の比較実験がしやすくなります。この点でレジストリパターン（[プランナ レジストリパターン（BasePlanner + PLANNER_REGISTRY）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1000_planner-registry-pattern)）と強く結びつきます。

これらに共通する「学習と推論で返り値の意味を変えない」「中間テンソルを外に漏らさない」という方針は、深層学習フレームワーク全般（PyTorch の nn.Module の forward を推論意味で使い、損失は外で計算する慣習など）とも整合します。

## 論文の実験結果（定量データ）
この設計原則自体は定量評価の対象ではありませんが、それを採用した代表的手法の数値を挙げます。

Diffusion Policy（Chi et al., 2023）は、複数のロボット操作ベンチマーク（Robomimic, Push-T, 多数の実機タスクを含む 15 タスク程度）で、当時の代表的な行動クローン手法（LSTM-GMM, IBC, BET など）に対して平均で大幅な成功率向上を報告しました。論文では「平均で約 46.9% の相対的な成功率改善」という主張がなされています。学習（ノイズ予測の回帰）と推論（receding-horizon でのサンプリング, 一度に複数ステップ予測して一部だけ実行し再計画する方式）を分離した API 設計が、この安定した学習・推論を支えています。ここで重要なアブレーション的知見は、行動を「単一の決定論的回帰」で出すのではなく「条件付き生成（サンプリング）」で出すことが、マルチモーダルな行動（複数の正解がありうる状況、たとえば障害物を左右どちらでも避けられる）を扱う鍵だった、という点です。単純回帰だと複数正解の平均（=どちらでもない危険な中間）を出してしまう「mode averaging」が起きますが、生成サンプリングはそれを避けられます。学習目標（ノイズ予測 MSE）と推論手続き（反復サンプリング）が別物だからこそ、この表現力が得られます。

Flow Matching（Lipman et al., 2023）は、ImageNet など画像生成で Diffusion と同等以上の品質を、より少ない推論ステップ・より安定した学習で達成できることを示しました（直線的な経路を学ぶため積分が少なくて済む）。学習が「目標速度への回帰」という単純な形に閉じ込められること（construct_training_data に相当する部分が明快なこと）が、実装の単純さと安定性に寄与しています。Conditional Flow Matching（Tong et al., 2024）は、最適輸送に基づく補間（OT-CFM）を使うとさらに経路がまっすぐになり、少ステップ推論での品質が上がることを示しました。これも「補間の作り方（学習データ構築）」と「推論サンプラー」が分離されているからこそ差し替えて比較できた成果です。

これらに共通する設計上のトレードオフは、「メソッド/関数がやや増える（forward 1 個では済まない）」一方で、「取り違えバグを構造的に防げる安全性」と「方式差し替えの容易さ」が得られる、という点です。

## メリット・トレードオフ・限界

メリット:
- 「学習モードで速度を軌跡と取り違える」類の silent bug を構造的に排除できる。
- 一時テンソル（ノイズ・目標速度・時刻）がモデル外に漏れず、別サンプルの目標と取り違える事故も防げる。
- 損失をモデル外に出す（external loss）ことで、損失項の追加・組み替え（補助損失、正則化、複数項の重み付け）が柔軟になる。
- プランナ/デコーダを差し替えても呼ぶ側の契約が変わらず、レジストリパターン（[プランナ レジストリパターン（BasePlanner + PLANNER_REGISTRY）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1000_planner-registry-pattern)）と相性が良い。
- 推論サンプラー（Euler / Heun / DPM-Solver / consistency 蒸留）を後から差し替えられる。学習済み重みはそのままに推論手続きだけ交換できる。

トレードオフ・限界:
- メソッドが増え、最初は「forward 1 個で済ませたい」より冗長に見える。
- `construct_training_data` を public にすると、誤って生の中間量（速度など）を外で損失に使う余地が残る（docstring や正規経路の明示で緩和する）。
- GRU や多項式（Bézier 等）のように学習と推論で振る舞いが大きく変わらないモデルでは、この分離の恩恵は相対的に小さい（それでも契約を揃える価値はある）。
- 「forward は推論専用」という規約を破る誘惑（学習中にちょっと内部値を覗きたい等）が常にあり、チーム内で規約を共有・徹底する必要がある。規約はコードでは強制しにくく、レビューと文書に依存する弱さがある。

## 発展トピック・研究の最前線
- 損失関数の外部化が進むと、複数の補助タスク（軌跡予測に加えて占有予測、衝突回避損失、不確実性推定など）を同じ forward 出力に対して柔軟に足し引きできるようになります。マルチタスク end-to-end 運転の実装で重要です。
- 推論手続きの差し替え（同じ学習済みネットに対して、Euler / Heun / 適応ステップ / consistency 蒸留など異なるサンプラーを後から差し込む）も、forward と損失を分離しておくと容易になります。蒸留で少ステップ化（数十ステップ→1〜4ステップ）する研究はリアルタイム制御で特に重要です。
- 「学習時にだけ存在する教師信号（teacher forcing, ノイズスケジュール、データ拡張）」と「推論時の決定論性」をきれいに分離することは、sim-to-real や分布シフトのデバッグでも効いてきます。どこで学習と推論の振る舞いがずれているかを切り分けやすくなるからです。
