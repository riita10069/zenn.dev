# 統一埋め込み次元(embed_dim=256)

## ひとことで言うと

ニューラルネットワークの中を流れる特徴には「太さ」(チャネル数、特徴次元)があります。画像から特徴を取り出すバックボーンの出力は、バックボーンの種類ごとに太さがバラバラで、しかも下流で扱うには太すぎることがあります。これを 1×1 畳み込みという小さな変換部品で一律に 256 次元へ揃えることで、後ろに続く全モジュールを「太さ 256 が来る前提」で共通設計できるようにする、という設計上の作法です。256 という値は、性能と計算量のバランスが良い慣習値として、Transformer ベースの知覚モデルの多くが採用しています。

## 直感的な理解

工場のベルトコンベアを想像してください。前の工程から流れてくる部品の「幅」が、使う機械によって 1440 mm だったり 3840 mm だったりまちまちだと、後ろの全ての機械をその都度作り直さなければなりません。さらに幅が広いほど、後ろの機械の処理コストが幅の 2 乗で跳ね上がる、という厄介な性質があるとします。

そこで、前の工程の直後に「どんな幅でも 256 mm に揃える変換機」を 1 台置きます。これ以降、後続の全機械は「幅 256 mm が来る」とだけ決め打ちで設計できます。バックボーン(前工程の機械)を別のものに交換しても、変換機の入力設定を変えるだけで、後続は一切いじらずに済みます。これが「統一埋め込み次元」の発想です。256 という幅は「広すぎず狭すぎず、処理が軽くて表現力も十分」という経験的な落とし所です。

## 基礎: 前提となる概念

ニューラルネットワークの中を流れるデータは、典型的に `[B, C, H, W]` という 4 次元の塊(テンソル)です。

- B = バッチ数(同時に処理する画像の枚数)
- C = チャネル数(各空間位置に何個の数値=特徴が並んでいるか。これが「特徴の太さ」)
- H, W = 縦と横の解像度(マス目の数)

このうち C(チャネル数、別名 埋め込み次元 embed_dim)が本トピックの主役です。「埋め込み(embedding)」とは「各位置の情報を C 個の数値の並びで表現したもの」という意味で、その個数 C が embed_dim です。例えば C=256 なら、画像の各位置(あるいは各クエリ、各トークン)が 256 個の数値で表現されている、ということです。

なぜ次元を揃える必要があるのか。理由は 2 つあります。

第一に、バックボーン依存からの解放です。画像から特徴を抽出するバックボーン(ResNet、Swin Transformer、ConvNeXt など)は、通常 4 段階の階層的な特徴を出します。これらをそのまま連結(チャネル方向に積み上げ)すると、合計チャネル数はバックボーンごとに違います。

```
Swin Transformer V2 Tiny: 96 + 192 + 384 + 768 = 1440 チャネル
ConvNeXt V2 Tiny       : 96 + 192 + 384 + 768 = 1440 チャネル
ResNet-50              : 256 + 512 + 1024 + 2048 = 3840 チャネル
```

もし下流の全モジュールがこの合計チャネル数を直接前提にしていると、バックボーンを変えるたびに下流の全モジュールの入力サイズを書き換える必要が生じます。研究では何度もバックボーンを切り替えて比較したいので、これは大きな手間とバグの源になります。

第二に、計算量の抑制です。下流でよく使う Multi-Head Attention(注意機構。各位置が他の全位置を見て関係を測る仕組み)の計算量は、おおむねチャネル数 C の 2 乗に比例します。1440 のまま流すと、256 に比べておよそ (1440/256)² ≈ 31.6 倍の計算量になります。つまり次元を落とさなければ約 31 倍重い。3840(ResNet-50)ならさらに重く、(3840/256)² = 225 倍です。次元を最初に絞ることで、この爆発を入口で断ち切れます。

## 仕組みを詳しく

考え方はシンプルです。バックボーンが出すマルチスケール特徴をプール(解像度を揃える)して連結したあと、1×1 畳み込みを 1 枚だけ通して、チャネル数を「合計値」から 256 に圧縮します。

1×1 畳み込み(Conv1x1)とは何か。普通の 3×3 畳み込みは「3×3 の窓で周囲の空間も一緒に見て」特徴を作りますが、1×1 畳み込みは「各ピクセル位置で、C 個の入力数値を重み付き和して、出力 256 個の数値を作る」だけの操作です。空間方向(H, W)には一切混ぜず、純粋にチャネル方向の線形変換(行列のかけ算)を、全ての空間位置で同じ重みを共有して行います。だからどんな入力チャネル数でも、出力を 256 に揃える「変換アダプタ」として使えます。重み行列のサイズは `(入力チャネル数 × 256)` です。

典型的な実装は次のような形になります(疑似コード)。

```python
self.channel_proj = nn.Sequential(
    nn.Conv2d(backbone_channels, embed_dim, kernel_size=1),  # 例: 1440 -> 256
    nn.GELU()
)
```

`nn.GELU()` は活性化関数(非線形性を加えて表現力を上げる関数)です。線形変換(行列かけ算)だけを重ねても、数学的には 1 つの大きな線形変換に潰れてしまい表現力が頭打ちになります。そこで軽い非線形を挟みます。GELU は ReLU(負を 0 にする関数)を滑らかにしたもので、Transformer 系で広く使われます。

形の変化を追うと(Swin V2 Tiny、1 画像あたり、8×8 にプール後の例)。

```
backbone 出力(4段階)  [BV, 96, 8, 8], [BV,192,8,8], [BV,384,8,8], [BV,768,8,8]
   └ プールで 8x8 に揃える
連結(channel方向)     [BV, 1440, 8, 8]
   └ channel_proj: Conv1x1(1440 -> 256) + GELU
射影後                  [BV,  256, 8, 8]   ← ここから先は常に 256
```

`BV` は「バッチ B とカメラ台数 V を一時的に掛け合わせた数」です(例: B=4, V=6 なら BV=24)。複数カメラをまとめてバックボーンに通すための一時的なまとめ方で、この後の複数カメラ統合で B と V に戻して扱います。

ResNet-50 を選んだ場合は連結が 3840 チャネルになりますが、`channel_proj` の入力チャネルを 3840 に設定するだけで、出力はやはり 256。だから「射影後」から先の全モジュール(複数カメラ統合、軌跡計画など)は一切変更不要です。これが「統一埋め込み次元」の狙いです。

256 という値の意味も補足します。256 自体に数学的な魔法はありません。「注意機構の計算量を抑えつつ、表現に必要な情報量を確保できる」経験的なバランス点として、研究コミュニティで広く採用されているにすぎません。タスクが複雑なら 512 を使う研究もありますし、軽量化を極めるなら 128 にする例もあります。256 は最も標準的なデフォルト値です。

主要な知覚モデルの埋め込み次元を並べると、256 がいかに標準かが分かります。

| モデル | embed_dim | 出典 |
|---|---|---|
| DETR / Deformable DETR | 256 | Carion et al. ECCV 2020 / Zhu et al. ICLR 2021 |
| BEVFormer | 256 | Li et al. ECCV 2022 |
| PETR / PETRv2 | 256 | Liu et al. ECCV 2022 / ICCV 2023 |
| UniAD | 256 | Hu et al. CVPR 2023 |
| VAD | 256 | Jiang et al. ICCV 2023 |

256 に合わせておけば、これらの論文の知見・事前学習重み・実装を取り込みやすく、性能比較もしやすくなります。

## 手法の系譜と主要論文

「特徴次元を 256 に統一する」という慣習の源流は、物体検出を Transformer で解く研究にあります。

- Carion et al., "End-to-End Object Detection with Transformers" (DETR, ECCV 2020, arXiv:2005.12872)。物体検出を「集合予測(オブジェクトの集合を直接出す問題)」として定式化し、Transformer のトークン次元を 256 に設定しました。以後の派生研究が広くこの 256 を踏襲します。DETR のトレードオフは収束が遅いこと(学習に長いエポックを要する)。

- Zhu et al., "Deformable DETR" (ICLR 2021, arXiv:2010.04159)。DETR の遅い収束と高解像度の重さを、参照点まわりの少数のサンプル点だけに注意を向ける「deformable attention(変形可能注意)」で解決しました。embed_dim はやはり 256。詳細は [BEVFormer風の暗黙的投影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0273_bevformer-deformable-attention)。

- Li et al., "BEVFormer" (ECCV 2022, arXiv:2203.17270) と Liu et al., "PETR" (ECCV 2022, arXiv:2203.05625)。多視点カメラからの 3 次元知覚で、BEV クエリや 3 次元位置埋め込みの次元を 256 に統一しました。PETR の系譜は [PETR / PETRv2 の3D位置埋め込み](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0288_petr-3d-position-embedding)。

- Hu et al., "UniAD" (CVPR 2023, arXiv:2212.10156)。検出・追跡・地図・予測・プランニングの全モジュール間で同じ 256 次元の表現(クエリ)を流すことで、モジュールの付け外しを容易にしました。詳細は [UniAD(プランニング指向の統合設計)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1218_uniad-planning-oriented)。

これらに共通する理由は、(1) 注意機構の計算量が次元の 2 乗で効くため小さめに抑えたい、(2) モジュール間で次元を合わせておくと組み替えが楽、という工学的な合理性です。256 が一種の業界標準(de facto standard)になったことで、研究間のコード移植と比較が格段にやりやすくなりました。

## 論文の実験結果(定量データ)

埋め込み次元の選択そのものを主題にした論文は少ないですが、関連する定量的知見を挙げます。

第一に、次元削減の計算効果です。注意機構の計算量は次元の 2 乗に比例するので、1440 → 256 への射影は注意機構コストを約 31.6 倍(= (1440/256)²)減らします。さらに、複数カメラの特徴を連結して 256 に落とす射影層のパラメータ数も劇的に減ります。例えば入力 1440・出力 256 の 1×1 畳み込みのパラメータは 1440×256 ≈ 約 37 万。仮に出力も 1440 のまま流す設計だと 1440×1440 ≈ 約 207 万になり、約 5.6 倍重い。さらに大きな射影(例えば連結特徴を空間ごと潰して全結合する設計)では、256 にすることで数百万から数十万へとパラメータが 1 桁減る例もあります。

第二に、Transformer 検出器のスケーリング知見です。DETR 系の研究では、トークン次元を 256 から 512 へ増やすと表現力は上がるものの、計算量とメモリが増え、小規模データでは過学習しやすくなることが報告されています。逆に 128 まで落とすと軽くなる反面、複雑なシーンで精度が頭打ちになる傾向があります。多くの研究が 256 を選ぶのは、この中間で精度・計算量・汎化性能のバランスが良いという経験則によります。

第三に、次元統一の移植性への寄与です。BEVFormer・UniAD・VAD・PETR がすべて 256 を採用したことで、これらの間で事前学習重みやモジュールを流用する研究が容易になりました。例えば UniAD は BEVFormer の 256 次元 BEV エンコーダの重みでそのまま初期化できます。次元が揃っていなければ、こうした流用には次元合わせの追加層と再学習が必要で、研究のスピードが落ちます。これは数値で測りにくい「研究生産性」への寄与ですが、コミュニティ全体に効いている重要な効果です。

## メリット・トレードオフ・限界

メリット。

- 下流モジュールがバックボーン非依存になります。バックボーンを変えても射影層の入力チャネルを直すだけで、後続は無改変です。
- 注意機構の計算量が大幅に削減されます(1440 起点なら約 31.6 倍、3840 起点なら約 225 倍の削減)。
- 主要論文(DETR/BEVFormer/PETR/UniAD/VAD)と次元が揃い、知見・事前学習重み・実装の移植がしやすくなります。

トレードオフと限界。

- 1440(や 3840)→ 256 は情報圧縮なので、理屈の上では細かな特徴が一部失われ得ます。実際には深層特徴に冗長性が高く影響は小さいことが多いですが、ゼロではありません。
- 何でも 256 に揃えると、本当に高次元が要るタスク(非常に多くの物体カテゴリや細粒度の区別が必要な場合)で表現力不足になる可能性があります。256 はあくまで経験的な落とし所であって、最適値の保証はありません。
- 射影層という部品が 1 つ増えます(学習対象パラメータと推論コストが増える)。ただし削減できる計算量のほうが圧倒的に大きいので、ネットの効果はほぼ常にプラスです。
- 次元統一は「全モジュールが同じ次元を共有する」前提を強制するため、ある特定モジュールだけ別の次元を使いたい場合に追加の変換層が必要になり、設計の自由度を少し下げます。

## 発展トピック・研究の最前線

- 次元のスケーリング法則。大規模化が進む知覚・運転モデルでは、データ量・計算量に応じて最適な次元が変わります。小型モデルは 256、大型モデルは 512 や 1024 を使う、といったスケーリングの指針が経験的に蓄積されつつあります。
- トークン圧縮との関係。最近は「次元を絞る」だけでなく「トークン数(クエリ数や BEV グリッド数)を絞る」ことで計算量を下げる研究も盛んです。次元 256 を保ったままトークンを間引く、あるいは可変解像度にする、といった工夫です。
- 異種モダリティの統一表現。カメラ・LiDAR・レーダーといった異なるセンサーの特徴を、共通の 256 次元空間に射影してから融合する設計(BEVFusion など)が広まっています。次元統一の発想が、バックボーン非依存からセンサー非依存へと拡張されているわけです。
