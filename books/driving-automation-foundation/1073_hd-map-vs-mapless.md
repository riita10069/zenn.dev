# HDマップ依存 vs マップレス

## ひとことで言うと
自動運転車が走るとき、あらかじめ精密に作っておいた高精度地図(HDマップ)を頼りにするのか、それともその場のセンサーだけを使って走行中に地図を生成するのか、という設計上の大きな分岐の話です。前者は正確だが地図の整備・更新が大変、後者は地図整備が不要だが走行中にリアルタイムで地図を作る難しさがあります。どちらを選ぶかで、必要なデータ・学習タスク・評価指標までが変わります。

## 直感的な理解
人間の運転を考えてみます。何度も通った道なら「この先で右車線が消える」「あの交差点は左折専用」と覚えていて、それを前提に運転します。初めての道なら、目の前の車線や標識をその場で読み取って判断します。

自動運転のHDマップ依存は前者、マップレスは後者に対応します。HDマップ依存は「記憶(事前地図)に頼る」、マップレスは「今見えているもの(センサー)だけで判断する」。前者は記憶が正しければ強いが、記憶が古いと危険。後者は記憶がいらない代わりに、その場の認識精度に運命を委ねることになります。この二択が、自動運転システムの設計思想を大きく分けます。実際、地図整備の運用コストが「自動運転の事業が新しい都市に広がる速度」を律速するため、これは技術論であると同時に事業戦略の論点でもあります。

## 基礎: 前提となる概念
- HDマップ (High-Definition Map、高精度地図): カーナビ地図とは桁違いに精密な地図です。車線の境界線が1本ずつ、横断歩道・停止線・道路標示・縁石の位置が、数センチ〜十数センチ精度で記録されます。普通のカーナビ地図が「この道を右」レベルなのに対し、HDマップは「今いる車線の右端は自車中心から1.8m、その先30mで車線が消える」レベルの情報を持ちます。
- SDマップ (Standard-Definition Map): カーナビ並みの粗い地図。道路のつながり(トポロジ)は分かるが、車線1本ずつのジオメトリは持たない。OpenStreetMap が代表で、無償・広域・更新が容易という性質を持ちます。HDマップとマップレスの中間の「弱いヒント」として近年注目されます。
- ローカリゼーション (localization、自己位置推定): 車が「自分は地図上のどこにいるか」を、センサーと地図を照合して数センチ精度で割り出すこと。HDマップ依存の要です。地図があっても、その上で自分の位置を正確に置けなければ意味がありません。
- 鳥瞰図 (BEV, Bird's-Eye View): 空から真下を見下ろした地図のような視点。前向きカメラの斜めの映像を、頭上から見た平面図に変換したものです。オンライン地図構築はこのBEV上に地図を描きます。
- ベクトル表現とラスタ表現: 地図要素を「点や線の並び(ベクトル)」で表すか、「画素ごとのラベル(ラスタ、画像)」で表すか。計画にそのまま渡せるのは前者、認識タスクとして解きやすいのは後者です。
- セマンティックセグメンテーション: 画像の画素1つ1つに「ここは車線/横断歩道/背景」とクラスを割り当てる認識タスク。初期のオンライン地図はこの形で地図を出していました。

なぜHDマップが要るのか。自動運転車は「自分が今どの車線のどこにいて、車線がこの先どう曲がり、どこで止まるべきか」を知る必要があります。カメラだけでその場で全部を正確に読むのは難しいので、あらかじめ地図に書いておけば、車はローカリゼーションするだけで済みます。これがHDマップ依存の利点であり、同時に「地図を作って更新し続けねばならない」という弱点の源でもあります。米国の一般道は数百万キロあり、工事や標示の引き直しで地図は日々古くなる。これを鮮度を保って維持するコストが、HDマップ依存の最大の弱点です。

## 仕組みを詳しく

### HDマップ依存の流れ
```
[事前] 計測車で走行 → LiDAR点群・画像収集 → 人手+半自動で車線/標示を引く
       → HDマップ完成 → 車両へ配信
[走行時] センサーで自己位置推定(ローカリゼーション) → HDマップを照合
       → 「自分は地図上のここ」と確定 → その地図情報を前提に計画
```
走行時の負荷は軽い反面、事前作業と更新の負荷がすべて地図側に乗ります。地図のカバー範囲外には出られず、地図が古ければ古い情報を信じてしまいます。「地図には車線があると書いてあるが、実際は工事で封鎖されている」という乖離が、HDマップ依存の典型的な失敗モードです。

### マップレス(オンライン地図構築)の流れ
```
[走行時] 複数カメラ画像 → ニューラルネットでBEV特徴に変換(view transform)
       → 車線・横断歩道・境界線をその場でベクトル(点の並び)として出力
       → そのまま計画に使う
```
ここで「複数カメラの斜め画像 → BEV」への変換が技術的な核です。Lift-Splat-Shoot (Philion & Fidler, ECCV 2020) は、各画素に奥行きの確率分布を持たせて3Dに「持ち上げ(lift)」、BEVグリッドに「貼り付ける(splat)」方式を提案し、この分野の基礎になりました。各カメラ画素 (u,v) について離散化した奥行き候補 D 個ごとに確率を出し、その重みで特徴を3D空間にばらまく、という発想です。

### ベクトル化された地図とは何か、なぜ重要か
初期のオンライン地図はラスタ(画素ごとのセグメンテーション)で地図を出していました。しかしこれだと「車線が1本の線としてどこからどこへ伸びているか」という、計画に直接使える形にするための後処理(線追跡・スケルトン化・連結成分解析など)が必要でした。後処理はヒューリスティックで壊れやすく、誤差が乗ります。

そこで地図を最初からベクトルで出す方向が生まれます。ベクトルとは、例えば「ある車線境界線 = 点 (x1,y1),(x2,y2),…,(x20,y20) を結んだ折れ線」のように、線を点の並びで表したものです。形状をテンソルで書くと、典型的には「最大 N 本の地図要素 × 各要素 P 点 × 2次元座標」、つまり (N, P, 2) の固定形になります(例: N=50, P=20 なら (50, 20, 2))。これにより、人が後処理で線をつなぐ手間なく、計画にそのまま渡せる地図が一発で得られます。

### 順序の曖昧さという落とし穴
ベクトル地図には独特の難しさがあります。例えば横断歩道の4隅を点で表すとき、「左上→右上→右下→左下」でも「右下→左下→左上→右上」でも同じ形なのに、学習時に「点の順番が違う」とペナルティを与えると、モデルが混乱します(順序の曖昧さ permutation ambiguity)。MapTR はこれを「同値な並べ方の集合を作り、その中で予測に最も近い順序を選んでマッチングする(hierarchical matching)」ことで解き、安定して高速にベクトル地図を出せるようにしました。要素どうしの対応付け(どの予測がどの正解か)はハンガリアン法による集合マッチングで、点どうしの対応付けは順序不変なマッチングで、という二段構えです。

## 手法の系譜と主要論文

### HDMapNet (Li et al., ICRA 2022)
提案: 複数カメラ(とLiDAR)からBEV上のセマンティック地図(車線・横断歩道・境界)を生成する初期の代表手法。動機は「事前HDマップなしで走行中に地図を作れないか」。効果は、オンライン地図構築というタスクを定式化し、評価指標(地図要素のIoUやAP)を整備して後続研究の出発点になったこと。トレードオフは、出力がラスタなので車線を線として使うには後処理が必要で、計画に直結しにくい点でした。

### VectorMapNet (Liu et al., ICML 2023)
提案: 地図をベクトル(点列)で直接出力する初期手法。動機はHDMapNetの「後処理が要る」課題の解消。効果は、地図要素を計画に近い表現で得たこと。トレードオフは、要素を逐次的(autoregressive、1点ずつ次の点を予測)に生成するため遅く、生成順序に敏感だった点です。

### MapTR / MapTRv2 (Liao et al., ICLR 2023 / IJCV 2024)
提案: 地図要素を順序不変な点集合マッチングで並列にベクトル出力する手法。動機は VectorMapNet の遅さと順序敏感性の解消。効果は、精度・速度の両面でオンライン地図構築を大きく前進させたこと。MapTRv2 では補助的なBEVセグメンテーション監督や奥行き監督、PV(透視図)補助監督を足してさらに改善しました。トレードオフは、要素数 N・点数 P を固定する設計のため、想定外に複雑な交差点では表現力が足りないこと、遠方・遮蔽領域は依然苦手なこと。

### StreamMapNet (Yuan et al., WACV 2024)
提案: 過去フレームの地図表現(BEV特徴とクエリ)を記憶として引き継ぎ、時間方向に地図を「ストリーム」する手法。動機は、単一時刻だけでは遠方や一時的な遮蔽に弱いこと。効果は、時間融合により遠距離・遮蔽下の地図安定性と一貫性を改善し、より長距離(例 100m級)の地図を扱えるようにしたこと。トレードオフは、記憶の引き継ぎ設計が複雑になり、誤った地図が時間方向に伝播しうる点です。

共通する論点は「事前地図のコスト・鮮度問題への回答」であり、効果は地図整備からの解放です。一方で、走行中にその場で地図を作る以上、作り込んだHDマップより精度・安定性で劣りうること、悪天候や複雑交差点で崩れやすいことが共通のトレードオフです。

## 論文の実験結果(定量データ)
評価指標の意味から説明します。オンライン地図構築は主に mAP(平均適合率)で測ります。ここでの「正解」は、予測したベクトル要素と正解要素がどれだけ近いか(Chamfer 距離、2つの点列の各点から相手の最近点までの距離の平均が閾値以内か)で判定し、車線境界(divider)・歩道境界(boundary)・横断歩道(ped crossing)といったクラスごとに AP を計算して平均します。0〜100% で大きいほど良い。Chamfer 距離の閾値は通常 {0.5, 1.0, 1.5} m の3段階で平均します。速度は FPS(1秒あたり処理フレーム数)で測り、リアルタイム性の目安になります。

nuScenes(6カメラを備えた代表的な自動運転データセット)上での報告される傾向は次の通りです。

- HDMapNet: ラスタ地図ベースで mAP は 30%前後にとどまり、ベクトル化手法に明確な改善余地を残しました。後処理を含めて遅く、リアルタイムには程遠い。
- VectorMapNet: ベクトル直接出力により mAP で HDMapNet を上回り、40%台に到達。ただし逐次生成のため推論が遅い(数 FPS 程度)。
- MapTR: VectorMapNet を大きく上回り mAP 50%台を報告し、かつ並列出力で速い(十数 FPS、リアルタイムに近い)。MapTRv2 はさらに mAP を 60%超へ押し上げ、当時の最高水準を更新しました。論文では世代ごとに十数ポイント規模の mAP 改善が積み上がっています。
- StreamMapNet: 時間融合により、特に遠距離(自車から離れた領域)や一時的遮蔽での地図 mAP を改善。長距離評価設定(従来より広い知覚範囲)で従来手法を上回ることが報告されています。

アブレーションの知見:
- 順序不変マッチングを外して固定順序にすると、MapTR の精度が大きく落ち学習が不安定化する。順序の曖昧さへの対処が効いていることの証拠です。
- 時間融合(StreamMapNet)を外して単一フレームに戻すと、遠方・遮蔽領域の精度が顕著に低下する。これは「事前HDマップなら当然持っている遠方情報を、マップレスは時間方向の蓄積で補おうとしている」構図を示します。
- view transform に奥行き監督(各画素の奥行き正解を補助損失に足す)を加えると、BEVへの投影が正確になり地図精度が上がる(MapTRv2 の報告)。

重要な注意点として、Lilja et al. (CVPR 2024) は「同じ場所が学習と評価に分かれて入り込む(地理的データリーク)」と数字が楽観的に出ると指摘し、nuScenes の標準分割では学習路と評価路が地理的に重なっていることを示しました。地理的に分離した分割(geographically disjoint split)で評価し直すと mAP が大きく(報告では半減級に)下がることが示されており、「既存手法は新地図を作っているのではなく、見覚えのある場所を思い出しているだけ(localization に近い)」という研究上の警鐘になっています。マップレス手法の真の汎化はまだ難しい、という重要な含意です。

## メリット・トレードオフ・限界

HDマップ依存
- メリット: 走行時の地図精度が高く安定、走行時計算が軽い(照合中心)、見通しの悪い場所・遮蔽でも地図が補う。検証済みの「真実」を持てるので安全論証がしやすい。
- 限界: 地図の作成・更新コストが莫大で鮮度を保てない、地図のない地域では走れない(展開速度が地図整備で律速)、地図が古いと誤情報を信じて危険。

マップレス
- メリット: 事前地図不要で未整備地域でも走れる、工事などの変化に走行中追従できる、地図整備の運用コストから解放、新都市展開が速い。
- 限界: 事前HDマップより精度・安定性で劣りうる、悪天候・遮蔽・複雑交差点で崩れやすい、リアルタイム生成の計算が要る、要素数・点数固定だと複雑な道路の表現力が不足、地理的汎化が難しい(上記リーク問題)。

現実には二者択一ではなく、両者の中間が研究されています。粗い事前地図(SDマップ)を弱いヒントとして与え、細部はオンラインで補う「SDマップ事前(SD-map prior)」アプローチが代表で、地図整備コストを抑えつつオンライン地図の精度を底上げできることが報告されています(SDマップのトポロジを BEV エンコーダに条件付けると、特に遠方や交差点で mAP が改善する)。

## 発展トピック・研究の最前線
- SDマップ/粗い事前地図の活用: 安価で更新しやすい粗地図(OpenStreetMap 等)をヒントにオンライン地図を強化する。地図整備ゼロではなく「軽い地図」を許容する現実解。
- End-to-End 運転への統合: ベクトル地図を明示的に出さず、BEV特徴の中に地図情報を暗黙に持たせて運転軌跡を直接出す方向(明示地図のいらない極端なマップレス)。地図を中間タスクとして補助的に学ばせると運転性能が上がるという報告もある(UniAD 系の補助タスク設計)。
- 長距離・時間一貫地図: ストリーム化や記憶機構で、遠方や交差点まで一貫した地図を保つ。地図要素の追跡(map element tracking)まで踏み込む研究も。
- ベクトル化の表現力向上: ベジェ曲線やグラフ構造(車線のつながり・分岐合流をグラフで表す lane graph)で、固定点数の制約を超えて複雑な道路形状を表す。
- 評価の厳密化: 地理的分離分割や時間的分離分割で、見たことのない場所への汎化を正直に測る。リーク問題の発覚以降、評価プロトコル自体が研究対象になっている。
