# WebDatasetシャードへのフレーム事前抽出

## ひとことで言うと
深層学習の学習ループでは、ディスクやネットワークから読み込んだ生データ(動画・音声・画像)を、その場でモデルが食べられる数値テンソルに変換します。動画から特定のコマ(フレーム)を取り出す「デコード」はこの変換のなかでも特に重く、放っておくと高価な GPU が「次のデータまだ?」と遊んでしまいます。WebDataset シャードへのフレーム事前抽出とは、学習で実際に使うフレームだけをあらかじめ画像として抜き出し、tar 形式の大きなまとまり(シャード)に詰めておき、学習時はそれを前から順番に流し読みするだけにする、という入出力(I/O)最適化の手法です。GPU を遊ばせないための、データパイプライン側の代表的な工夫です。

## 直感的な理解
高級レストランの厨房を想像してください。注文が入るたびに牛を一頭さばいて、骨を外して、必要な部位だけ切り出していたら、料理が出てくるのは何時間も先になります。実際の厨房は仕込みをします。営業前に肉を切り分け、下味をつけ、すぐ焼ける状態でバットに並べておく。注文が来たら焼くだけです。

深層学習の学習も同じです。GPU(料理人)は 1 秒間に膨大な計算をこなせるのに、データの「仕込み」(動画デコード)が間に合わないと手持ち無沙汰になります。1 枚数百万円する調理器具を遊ばせるのと同じで、これは大きな損失です。事前抽出とは「学習が始まる前に仕込みを済ませておく」発想であり、WebDataset シャードとは「すぐ取り出せるよう整然と並べたバット」に相当します。

なぜわざわざ tar に詰めて順番に読むのか。これがもう一つの肝です。ストレージ(特にハードディスクやクラウドのオブジェクトストレージ)は、小さなファイルをあちこちランダムに開くのが極端に苦手で、連続した大きなデータを流し読みするのが得意です。冷蔵庫から食材を一つずつ取りに何度も往復するより、台車にまとめて一度に運ぶほうが速い、という直感に対応します。

## 基礎: 前提となる概念

専門用語を一つずつ噛み砕きます。

- フレームとは、動画を構成する 1 枚 1 枚の静止画です。動画は秒間に数十枚(30fps なら 30 枚)のフレームをパラパラ漫画のように並べたものです。
- デコードとは、圧縮された動画(H.264 / H.265 などの形式)から、特定の時刻のフレームを 1 枚の画像(画素値の配列)に復元する計算です。動画圧縮は「前のフレームとの差分だけ記録する」方式(フレーム間予測)を多用するため、任意の 1 枚を取り出すには、その手前にある基準フレーム(キーフレーム、I フレーム)まで戻ってから順に再構成する必要があります。だから 1 枚抜くだけでも地味に重く、しかもランダムな時刻へのアクセスほど遅くなります。
- DataLoader とは、学習中にデータを少しずつ読み込んでモデルに渡す部品です。PyTorch の `torch.utils.data.DataLoader` が代表で、複数の worker(並列で動く子プロセス)を立てて、GPU が今のバッチを計算している裏で次のバッチを先読みします。理想は「GPU が 1 バッチ計算し終える前に、次のバッチの準備が必ず終わっている」状態で、これを I/O が計算に隠れている(オーバーラップしている)と言います。
- ランダムアクセスとシーケンシャルアクセス。ランダムアクセスは「好きな場所を直接読む」、シーケンシャルアクセスは「先頭から順に読む」です。回転式ハードディスクは物理的にヘッドを動かすためランダムが遅く、SSD でも小ファイルを大量に開くとファイルシステムのオーバーヘッドが効きます。クラウドのオブジェクトストレージ(Amazon S3 など)では、1 オブジェクトの取得ごとに HTTP リクエストが発生し、レイテンシ(往復遅延、典型的に数十ミリ秒)が積み重なります。小ファイルを 100 万個読むと、それだけで数時間レベルの遅延が積み上がる計算になります。
- tar とは、複数のファイルを 1 本のアーカイブにまとめる、Unix 由来の古典的な形式です。中身は「ファイル名 + サイズ + データ」を順番に連結しただけの単純な構造で、圧縮はせず連結するだけのため、先頭から順に読むのが非常に速い。WebDataset はこの tar をデータ供給の単位として使います。

GPU 利用率(GPU utilization)という指標も覚えておくと理解が進みます。これは GPU が実際に計算していた時間の割合で、データ供給が間に合っていれば 90% 以上に張り付き、間に合わなければ 30〜50% まで落ち込みます。事前抽出の目的は端的にこの数字を上げることです。GPU 利用率が 50% ということは、買った計算資源の半分をデータ待ちで捨てている、という意味になります。

## 仕組みを詳しく

従来のよくある作り方(オンデマンドデコード方式)では、DataLoader の `__getitem__`(「i 番目のサンプルをください」と呼ばれたときに 1 件作る関数)のなかで、その都度こうします。

```
__getitem__(i):
    video = open(動画ファイル)         # ファイルを開く
    video.seek(時刻 t)                  # 必要な時刻にシーク(キーフレームまで巻き戻し)
    frame = video.decode()              # デコードして画素配列に
    return transform(frame)             # リサイズ・正規化
```

マルチカメラの自動運転では 1 サンプルが複数視点のフレームを束ねるため、1 サンプルあたり視点数 V 回(例えば 7 カメラなら 7 回)のシーク+デコードが必要になり、コストはさらに膨らみます。問題は三つあります。

1. 遅い。デコードが重く GPU が待たされる。
2. ランダムアクセスが苦手。学習はサンプル順をシャッフルしたいが、動画は前から順読みが得意でランダムシークは遅い。
3. worker と相性が悪い。動画デコーダのオブジェクト(ファイルハンドルや内部状態を持つ)を複数 worker で共有すると、クラッシュやデッドロックを起こしやすい。

事前抽出方式はこれを「前処理」と「学習」の二段階に分離します。

前処理ステージ(学習とは別ジョブ、CPU で並列実行できる):

1. データセットから「有効なサンプル点」を 1 件ずつ取り出す。マルチカメラ運転データなら、各サンプルは複数視点の画像(形状 (V, 3, H, W))、過去の自車運動の履歴、未来の軌道(教師信号)などを持つ。
2. 各カメラ画像を学習解像度(例: 256x256)にリサイズして JPEG(品質 90 程度)で再圧縮し、`s00000001.cam_0.jpg` のような規則的なファイル名で tar に追記する。
3. 数値配列(自車運動履歴・未来軌道)は `s00000001.ego.npy`、メタ情報は `s00000001.meta.json` として同じ tar に追記する。WebDataset はファイル名の頭(`s00000001`)が一致するファイル群を「1 サンプル」として自動的に束ねる規約を持つ。拡張子(`.jpg` / `.npy` / `.json`)を見て、デコーダが自動的に適切なデコードを掛ける。
4. 一定数(例: 1000 サンプル)ごとに新しいシャード(`train-000000.tar`, `train-000001.tar`, ...)へ切り替える。シャードを分けるのは、並列ダウンロードとシャッフルの単位を作るため。1 シャードが 100MB〜1GB 程度になるサイズに調整するのが定石。
5. 最後に `manifest.json`(総サンプル数、シャード数、画像サイズ、視点数 V など)を書き出す。

学習ステージ(GPU、シャードを流し読みするだけ):

```python
import webdataset as wds
dataset = (
    wds.WebDataset("train-{000000..000099}.tar", shardshuffle=True)
       .shuffle(2000)                  # シャッフルバッファ
       .decode("rgb")                  # JPEG → テンソル(軽い)
       .to_tuple("cam_0.jpg", "ego.npy", "meta.json")
)
loader = wds.WebLoader(dataset, batch_size=8, num_workers=4)
for visual, ego, meta in loader:
    # visual  (B, 3, 256, 256)
    # ego     (B, 256)
    ...
```

ここで起きるのは JPEG デコード(これは軽い。1 枚あたりマイクロ秒〜ミリ秒オーダー)と正規化だけで、動画デコードは一切ありません。before/after をデータの流れで対比すると次の通りです。

```
従来:  動画ファイル ──[重いシーク+動画デコード ×V]──→ テンソル   (毎エポック毎回)
事前:  動画ファイル ──[前処理1回だけ]──→ tarシャード ──[流し読み+軽いJPEGデコード]──→ テンソル
```

シャッフルは「シャッフルバッファ」で近似します。これは一定数(例: 数千件)のサンプルをメモリに溜めてから、そこからランダムに 1 件取り出して補充する方式です。tar のシーケンシャル読みの速さを保ったまま、十分なランダム性を確保できます。さらにシャード自体の読み込み順もシャッフルする(shardshuffle)ことで、二段階のランダム化が掛かります。worker 分割は WebDataset がシャード単位で割り当てるため、worker 同士が同じファイルを奪い合わず衝突しません。

なぜ JPEG で保存して、生のテンソル(.npy)で保存しないのか。これは設計上のトレードオフで、JPEG は非可逆圧縮(画素を多少劣化させて小さくする)のためストレージとネットワーク転送量を 1/10 程度に削れますが、デコードのコストが乗ります。生テンソルはデコード不要ですが容量が膨大になります。256x256x3 の 8bit 画像は生だと約 192KB ですが、JPEG 品質 90 なら 10〜20KB 程度に収まります。視覚タスクでは JPEG 品質 90 程度の劣化は精度にほぼ影響しないことが経験的に知られており、容量と転送のメリットが勝つため JPEG が選ばれます。

## 手法の系譜と主要論文

「データ I/O を学習のボトルネックにしない」という問題意識は、データ並列学習が大規模化した 2010 年代後半から明確に研究テーマになりました。

- WebDataset の基礎は Aizman, Maltby, Breuel(NVIDIA)「High Performance I/O For Large Scale Deep Learning」(IEEE Big Data 2019, arXiv:2001.01858)です。大規模学習でファイルを 1 件ずつランダムに読むとストレージのランダムアクセスがボトルネックになる、という観察から出発し、多数の小ファイルを tar シャードにまとめてシーケンシャルに流し、シャッフルはシャード順とバッファで近似する設計を提案しました。POSIX ファイルシステムに依存せず、ローカルディスクでもオブジェクトストレージ(S3 等)でも同じインタフェースで高スループットが出る点が新しい。この設計は後に PyTorch エコシステムの `torchdata` / `webdataset` パッケージとして広く普及しました。

- NVIDIA DALI(Data Loading Library、2018 年頃〜)は別アプローチで、デコードやリサイズなどの前処理を GPU 上で実行することで CPU ボトルネックを解消します。JPEG デコードを GPU の専用ハードウェア(nvJPEG)で行えるため、CPU が貧弱な構成で特に効きます。事前抽出が「重い処理を学習前にオフロードする」のに対し、DALI は「重い処理を GPU に肩代わりさせる」点が対照的です。

- Leclerc, Ilyas, Engstrom, Park, Salman, Madry(MIT)「FFCV: Accelerating Training by Removing Data Bottlenecks」(CVPR 2023、arXiv:2306.12517)は、独自のバイナリフォーマット(.beton)に前変換し、メモリマップ・準ランダムアクセス・JIT コンパイルされた前処理パイプラインを組み合わせて、データローディングを極限まで高速化しました。ImageNet で ResNet-50 を 1 台の 8GPU マシンで 20 分台で学習できると報告し、コミュニティに「データローダの遅さは設計で消せる」ことを強く印象づけました。本質的なアイデアは事前抽出と共通で、「学習時の前処理を最小化するために前もって最適なフォーマットへ変換する」ことです。

- 関連して、TensorFlow 陣営の TFRecord(複数サンプルを Protocol Buffer で連結したシーケンシャル形式)も同じ思想の先行例で、tar の代わりに独自バイナリを使う点を除けば WebDataset と発想を共有します。

系譜をまとめると、オンデマンド処理(素朴な DataLoader)→ シーケンシャル化された連結フォーマット(TFRecord / WebDataset)/ 前処理の GPU オフロード(DALI)→ 専用フォーマットによる総合最適化(FFCV)という流れで、いずれも「GPU を遊ばせない」という共通目標に向かっています。

## 論文の実験結果(定量データ)

数値で効果を見ます。指標の意味から説明します。

- スループット(images/sec, samples/sec): 1 秒間に何サンプル処理できるか。大きいほど学習が速い。
- GPU 利用率(%): 前述の通り、データ供給が間に合っているかの目安。

WebDataset の原論文(2019)では、ローカルファイルを 1 件ずつ読む素朴な方式に対し、tar シャードのシーケンシャル読みでストレージ帯域をほぼ飽和させられること、ネットワーク越し(リモートストレージ)でも数 GB/s 級のスループットを維持できることを示しました。特に、ファイル数が数百万〜数千万になるデータセットで、ファイルシステムのメタデータ操作(inode 探索)がランダムアクセス方式のボトルネックになるのを、シャード化が解消すると報告しています。

FFCV の論文(CVPR 2023)はより定量的です。ImageNet 分類で ResNet-50 を学習する際、標準的な PyTorch DataLoader 構成に比べ、同じハードウェアで学習時間を数分の一に短縮できると報告しました。具体的には、8x A100 で ResNet-50 を上位 1 精度(top-1 accuracy、正解クラスが最も高い確率になった割合)約 75% まで到達させるのに、数十分のオーダーで済むとしています。アブレーション(どの工夫を抜くとどうなるか)では、(1)専用フォーマットへの事前変換、(2)前処理パイプラインのコンパイル、(3)準ランダムアクセスによる I/O 削減、の三つがそれぞれ独立に効いており、特に I/O 削減を外すと CPU 律速に戻ってスループットが大きく落ちることを示しました。これは「事前抽出のうち I/O のシーケンシャル化が効果の主因」という、本トピックの主張を裏づけます。

事前抽出方式全般について経験的に言えるのは、オンデマンド動画デコード方式で GPU 利用率が 40〜60% に留まっていた構成が、フレーム事前抽出により 90% 超まで改善し、結果として 1 エポックあたりの実時間が短縮される、というパターンです。マルチカメラのように 1 サンプルあたりのデコード回数が多いほど、事前抽出の相対的な効果は大きくなります。視点数 V が 7 なら、オンデマンド方式のデコード負荷は単眼の 7 倍になるため、事前抽出で消せる無駄も 7 倍大きい、という関係です。

## メリット・トレードオフ・限界

メリット

- 学習時の動画デコードがゼロになり、GPU の遊休が減って学習が速くなる。マルチカメラほど効果が大きい。
- tar のシーケンシャル読みはディスク/ネットワーク帯域を最大限活かせ、worker 並列でも安全(状態を持つデコーダの共有問題が起きない)。
- 前処理が学習から切り離されるので、同じシャードを何度も再利用でき、実験間の再現性が上がる(全実験が同一の入力を見る)。
- オブジェクトストレージ(S3 等)と相性が良く、クラウド学習にそのまま乗せられる。tar を直接 HTTP ストリームで読めるため、全データをローカルにダウンロードしてから始める必要がない。

トレードオフと限界

- 前処理に一度コストと時間がかかり、抽出したフレーム分のストレージを消費する(JPEG でも、元動画が高圧縮なら展開後のほうが容量が増えることがある)。
- 完全なランダムシャッフルではなくシャッフルバッファによる近似になる。バッファが小さいと、同一シャード内のサンプルが固まって出やすく、ミニバッチの統計が偏るリスクがある(バッファサイズと精度のトレードオフ)。極端な例として、バッファサイズ 1 だと全くシャッフルされない。
- 抽出時に決めた画像サイズ・視点構成・サンプル点に固定される。解像度やデータ拡張(augmentation)の方針を変えるとシャードを作り直す必要があり、探索的な実験では足かせになる。データ拡張を学習時に動的に行いたい場合は、JPEG をデコードした後に適用する設計にする必要がある。
- 未解決の課題として、可変長シーケンスや時系列の動的サンプリング(エポックごとに違う時刻窓を切り出したい)を事前抽出と両立させるのは難しい。固定化した瞬間に時系列の自由度が失われるため、ここは GPU デコードやハイブリッド方式が研究されている。

## 発展トピック・研究の最前線

- ハイブリッド方式: よく使うフレームだけ事前抽出してキャッシュし、稀なアクセスはオンデマンドデコードにフォールバックする設計。ストレージと柔軟性のバランスを取る。
- GPU デコード(NVIDIA DALI / nvJPEG / NVDEC): 動画デコード自体を GPU の専用ハードウェア(NVDEC)で行い、事前抽出なしでも高スループットを狙う。事前抽出と組み合わせることも可能。
- 潜在表現の事前計算(latent caching): 近年の大規模モデルでは、画像そのものではなく事前学習済みエンコーダ(VAE や視覚バックボーン)に通した後の潜在ベクトルを事前計算してキャッシュする手法が一般化している。デコードどころか前段のニューラルネットの推論まで前倒しする発想で、拡散モデルの学習などで広く使われる。
- 分散シャッフルの理論: シャッフルバッファがどの程度の偏りを生むか、収束にどう影響するかは確率的勾配降下法(SGD)の収束理論と絡む論点で、シャードサイズ・バッファサイズの設計指針として研究が続いている。
- ストレージ階層の最適化: ローカル NVMe にホットなシャードをキャッシュし、コールドなものはオブジェクトストレージに置く、といった階層化が大規模学習のコスト最適化として実務的なテーマになっている。
