# マルチビュー融合アーキテクチャ設計

## ひとことで言うと

マルチビュー融合（multi-view fusion）とは、車のまわりに付けた複数台のカメラ（や地図タイル）の映像を、1つの統一されたシーン表現にまとめる処理のことです。この表現の上に物体検出や軌跡計画を載せます。融合の戦略には大きく「単純連結（concat）」「アテンションによる融合（cross-attention）」「幾何投影による BEV 構築（BEV）」の3系統があり、それぞれ計算コスト・精度・必要な前提（カメラキャリブレーションの有無）が異なります。本ノートは、これら融合戦略の動機・仕組み・比較・選び方を、研究設計の観点から体系的にまとめます。

## 直感的な理解

なぜ「融合」がそもそも必要なのか。カメラ1台の特徴をそれぞれ独立に処理しただけでは、自動運転の計画はできません。理由は3つあります。

第1にカメラをまたいだ対応付けです。前カメラと右前カメラの両方に半分ずつ映る1台の車を「同じ1台」だと認識する必要があります。各カメラを独立に見ていては、2台と数えてしまいます。隣り合うカメラの視野は通常少し重なるよう設計されますが、その重複領域での同一性判断は融合の役目です。

第2に深度の曖昧さの解消です。1枚の画像だけでは奥行きが定まりません（カメラから伸びる視線上のどこに物体があるか分からない）。複数視点の幾何（同じ物体が別々のカメラのどこに映るか）が、距離の手がかりを与えます。これは人間が両目の視差で距離を測るのと同じ原理を、広く配置したカメラ群で行うものです。

第3に統一表現の構築です。下流の計画モジュールは「カメラごとの特徴」ではなく「1つにまとまった場面表現」を前提にします。自車を中心とした統一座標系で「どこに何があるか」が分かって初めて、安全な軌跡を引けます。

つまり融合は、ばらばらの斜め視点を、運転判断に使える1枚の統一表現に翻訳する工程です。問題は「どう翻訳するか」で、ここに複数の設計流派が生まれます。流派の違いは突き詰めると「カメラの幾何（投影行列）をどれだけ明示的に使うか」という1本の軸に整理できます。

## 基礎: 前提となる概念

特徴マップとテンソル形状。各カメラ画像をバックボーン（ResNet や Swin Transformer などの画像特徴抽出器）に通すと、空間的に縮小された特徴マップが得られます。バッチ B、カメラ台数 V、チャンネル数 C、高さ H、幅 W として、`[B, V, C, H, W]` あるいは効率のためカメラとバッチをまとめた `[B*V, C, H, W]` という形で扱います。融合はこの多視点テンソルを入力に取り、`[B, C, H_out, W_out]` という単一の統一特徴を出力します。

レジストリパターンと差し替え可能性。研究では複数の手法を公平に比較（アブレーション、ある要素を変えて効果を測る実験）したいので、融合戦略を「同じ入出力インターフェース」で実装し、文字列キーで切り替えられるようにする設計がよく使われます。下流モジュールを変えずに融合だけを差し替えられると、比較が容易になります。一般的なパターンは次のような形です。

```python
def forward(self, per_view_features, B, V, camera_params=None) -> Tensor:
    # per_view_features: [B*V, C, H, W]
    # 戻り値: [B, C, H_out, W_out]
    ...

FUSION = {"concat": ConcatFusion, "cross_attn": CrossAttnFusion, "bev": BEVFusion}
```

カメラキャリブレーション。各カメラの取り付け位置・向き（外部パラメータ R, t）と焦点距離・画像中心（内部パラメータ K）です。幾何投影による融合（BEV）にはこれが必須ですが、データセットによっては与えられないこともあります。その場合に備えて「キャリブレーションがなくても動く近似（疑似投影）」を用意する設計判断がしばしば取られます。

アテンションの計算量。アテンションは候補数 N に対して O(N²) で増えます。多視点・高解像度では N が大きくなりすぎるため、変形可能アテンション（Deformable DETR、Zhu ら、ICLR 2021）のように「少数の点だけ見る」近似が重要になります。融合戦略の選択は、この計算量とのにらみ合いでもあります。

## 仕組みを詳しく

3つの融合戦略を、難易度・前提の少ない順に並べます。

### 戦略1: 単純連結（concat）

最も単純な融合です。全カメラの特徴をチャンネル方向に並べ（連結し）、1×1 畳み込みで元のチャンネル数に圧縮します。

```
[B*V, C, H, W] → reshape → [B, V*C, H, W]
            → Conv2d(V*C → C, kernel=1) → [B, C, H, W]
```

幾何情報を一切使わず、キャリブレーション不要、計算も軽い。ただしカメラ間の空間的対応を明示的にはモデル化できません（畳み込みの重みが暗黙に学ぶだけ）。前提として、各カメラの特徴が同じ空間グリッドに整列している必要があり、視野の異なるカメラを単純に重ねると意味的に噛み合わないため、精度は限定的です。プロトタイピングや、他戦略の精度を測る基準線（ベースライン）として価値があります。

### 戦略2: アテンション融合（cross-attention）

各空間位置で、V 個のカメラから来たベクトルが互いにアテンション（自己アテンションまたはクロスアテンション）し合います。各カメラには学習可能な識別埋め込み（view embedding、「これは前カメラ」「これは右カメラ」という識別子）を付けます。アテンション後、視点方向に集約（平均プーリングなど）して1つにまとめます。

PETR（[PETR（論文）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1216_petr-paper)）の「カメラ間で3次元情報をやり取りする」発想や、UniAD（[UniAD（論文）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1217_uniad-paper)）のクエリ連結に着想を得た系統です。3次元幾何を明示的に使わずとも、カメラ間の対応をデータから学べるのが利点です。キャリブレーションがあれば、view embedding を3次元投影座標で強化して PETR 型（特徴に3次元位置を足し込む）に近づけられます。concat より表現力が高い一方、アテンションの計算量が増えるため、見る範囲を絞る工夫（局所アテンション、変形可能アテンション）が併用されます。

### 戦略3: 幾何投影による BEV 構築（BEV）

最も高度で、最も強力です。学習可能な BEV クエリ（真上から見た地図の各マスに対応するベクトル）が、3次元基準点を各カメラ画像へ投影した位置から特徴をサンプルします。BEVFormer（[BEVFormer（論文）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1213_bevformer-paper)）の空間クロスアテンションそのものです。

```
BEV マスごとに3D基準点（高さ違いの複数点=ピラー）を作る
→ カメラ行列で各画像へ投影
→ 投影先から特徴をサンプル（変形可能アテンション or grid_sample）
→ 見えるカメラだけで集約（可視性マスク）
→ 出力: [B, C, H_bev, W_bev]
```

幾何的に正しい対応が得られ、可視性マスクで「見えないカメラから集めない」を保証できます。BEV という解釈しやすい地図表現が直接得られるため、計画タスクと相性が良い。一方で、原論文の変形可能アテンションは効率実装に専用 CUDA カーネルを要し移植性が下がるため、`F.grid_sample`（標準の双線形補間サンプリング）で近似して CUDA 依存を避ける実装上の工夫がよく取られます。キャリブレーションがない段階では、投影行列を学習可能パラメータで代用する「疑似投影（pseudo-projection）」のフォールバックを用意します。なお BEV を作る別系統として、深度を予測して特徴を3次元に持ち上げる LSS 型（[BEVFormer（論文）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1213_bevformer-paper) の系譜で触れる）もあり、こちらは深度教師がある場合に強みを持ちます。

### 設計要件の整理

研究用の融合アーキテクチャに共通して求められる要件を挙げます。

実行時に融合戦略を選べること（アブレーションと段階的高度化のため）。バッチ次元の正しさ（複数サンプルを同時に学習できること。素朴に `flatten` でバッチ次元まで潰すと、異なるサンプルの特徴が混ざってミニバッチ学習が壊れる、というのは実装で頻発する落とし穴です）。移植性（できれば専用 CUDA カーネルに依存しない）。キャリブレーションなしでも動くフォールバック（データセット未確定でも開発を進められる）。全融合モードで統一インターフェース（下流を変えずに済む）。これらは正しさと再現性のための要件であり、性能そのものとは別軸の品質基準です。

## 手法の系譜と主要論文

深度ベースの BEV 構築。LSS（Philion & Fidler、ECCV 2020、arXiv:2008.05711）が、ピクセルごとの深度分布を予測して画像特徴を3次元に持ち上げる方式を確立しました。BEVDet, BEVDepth などが発展させました。深度教師があると強いが、深度誤りに弱くメモリが重い。

幾何投影ベースの BEV 構築。BEVFormer（Li ら、ECCV 2022、arXiv:2203.17270）が、深度を予測せず BEV クエリから画像へ投影してサンプルする方式を示しました。土台は Deformable DETR（Zhu ら、ICLR 2021、arXiv:2010.04159）の変形可能アテンションと、Vaswani ら（NeurIPS 2017、arXiv:1706.03762）のアテンション機構です。

明示的 BEV を作らないクエリベース。PETR（Liu ら、ECCV 2022、arXiv:2203.05625）は画像特徴に3次元位置を埋め込み、BEV グリッドを作らずに検出します。これは「特徴に座標を足す」型の融合で、アテンション融合戦略の幾何強化版とも見なせます。連結（concat）系はこれらより古典的で、特徴量を単に並べる素朴な融合に相当します。

センサをまたぐ融合。BEVFusion（Liu ら、ICRA 2023、arXiv:2205.13542）は、カメラ由来の BEV と LiDAR 由来の BEV を同一の BEV 平面で統合し、片方のセンサが弱い状況をもう片方で補う相補性を実証しました。融合の対象が「複数カメラ」から「異種センサ」へ拡張された例です。

統合アーキテクチャでの位置づけ。UniAD（Hu ら、CVPR 2023、arXiv:2212.10156）は BEVFormer を融合層として採用し、その BEV 特徴の上に認識・予測・計画を載せました。これは「融合の出力（BEV 特徴）がそのまま下流の土台になる」という設計の代表例です。

## 論文の実験結果（定量データ）

融合戦略の優劣は、共通のデータセット・指標で比較してはじめて意味を持ちます。多視点3D検出では nuScenes が標準で、指標は mAP（検出の平均適合度、3次元中心点距離による一致判定、0〜1、高いほど良い）と NDS（位置・サイズ・向き・速度・属性の誤差を mAP と総合した nuScenes 専用スコア、0〜1）です。

戦略間の傾向として、文献の報告を総合すると次の関係が見られます。単純連結のような幾何を使わない融合は実装が軽い一方、多視点の対応付けが弱く、検出精度は幾何を使う手法に劣ります。BEVFormer 型の幾何投影は nuScenes テストで NDS 約0.569 を達成し、深度ベースの初期手法（BEVDet 系）や単眼検出（FCOS3D, NDS 約0.428）を明確に上回りました。PETR 型のクエリ融合は NDS 約0.504 で、BEV を作らないシンプルさのわりに高い水準でした。センサ融合の BEVFusion はカメラ単体を大きく上回り（LiDAR を加えることで NDS が 0.7 台に到達）、複数センサの相補性が定量的に確認されました。

アブレーションから分かる重要な知見。第1に、幾何情報（投影や3次元位置エンコーディング）を外すと精度が大きく落ちる。PETR では3D位置エンコーディングを抜くと mAP が大幅低下し、BEVFormer では投影に基づくサンプリングが性能の中核であることが示されました。つまり多視点融合では幾何の活用が効きます。第2に、時間融合（過去フレームの参照）を足すと、特に速度推定の誤差（mAVE）が顕著に下がる（BEVFormer/PETRv2 とも報告）。第3に、BEV 解像度を上げると遠方・小物体に強くなるが計算とメモリが増える。これらは「精度を上げる要素は計算コストを伴う」というトレードオフを定量的に示しています。

下流タスクでの効果。UniAD では、融合（BEVFormer）の上に予測・占有を正しくつなぐと計画の衝突率が下がる（平均約0.31%、先行手法の 0.6〜0.7% 台から改善）ことが示されました。融合の質が最終的な計画性能に波及することの証拠です。

## メリット・トレードオフ・限界

複数戦略を同一インターフェースで持つ設計のメリット。3戦略を公平に比較（アブレーション）でき、プロジェクトの成熟度に合わせて単純（concat）から高度（BEV）へ段階的に移行できる。「なぜその設計にしたか」を論文引用つきで記録すれば、後から経緯を追える。バッチ次元の正しさ・CUDA 不要・キャリブレーションなし動作という移植性要件を明文化できる。

トレードオフと限界。簡易化（1層・シングルヘッド・grid_sample・疑似投影）を選ぶと、原論文（BEVFormer/UniAD）ほどの性能は出ない前提になります。あくまで実験基盤・土台としての位置づけです。3戦略を維持するには相応のテスト・保守コストがかかります。そして根本的に、幾何投影系はカメラキャリブレーションの精度に依存し、疑似投影フォールバックは精度を犠牲にします。融合戦略の選択は「計算予算・キャリブレーションの有無・下流タスクが BEV 地図を必要とするか」の三者で決まる、というのが研究上の共通理解です。例えば検出だけが目的で計算予算が厳しいならスパースなクエリ融合、計画や占有予測まで視野に入れるなら密な BEV、というように設計判断が分かれます。

未解決の課題として、融合の計算コストと精度の最適点、時間融合をどこまで長くするか、キャリブレーションがずれた場合の頑健性（オンラインで外部パラメータを補正する研究もある）、そして異種センサ（LiDAR・レーダ）を同じ融合層に統合する方法などが、現在も活発に研究されています。

## 発展トピック・研究の最前線

融合アーキテクチャは「密な BEV か、スパースなクエリか」という軸で多様化しています。密な BEV（BEVFormer 系）は地図表現が直接得られ計画に向くが計算が重い。スパースクエリ（PETR/Sparse4D/StreamPETR 系）は軽量で車載向きだが明示的な地図は持たない。両者を使い分ける、あるいは融合する研究が進んでいます。

センサ融合の方向では、BEVFusion がカメラ BEV と LiDAR BEV を同一平面で統合し、センサの相補性を活かしました。表現の方向では、2次元の BEV を3次元の占有グリッドに格上げする占有予測（TPVFormer, OccNet など）が、不定形障害物への対応として注目されています。TPVFormer は BEV の1平面を3つの直交平面に拡張し、高さ方向の情報損失を補いました。さらに、融合の出力を end-to-end 計画や世界モデルの入力として直接使う統合設計が、現在の最前線のテーマです。融合は単独の前処理から、知覚・予測・計画を貫く共通基盤へと位置づけが移りつつあります。
