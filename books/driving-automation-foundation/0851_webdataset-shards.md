# WebDataset (Shard Packing)

## ひとことで言うと
WebDataset とは、学習用のデータ（画像・音声・数値・ラベル）を「tar」という1つの箱（アーカイブファイル）にまとめて固め、それを先頭から順に読み流すためのデータ形式と、その読み書きを行うライブラリのことです。何百万個もの小さなファイルを1つずつ開く代わりに、数百MB〜数GBの大きな箱を頭から一直線に読むので、ローカルディスク・ネットワークストレージ・クラウドのオブジェクトストレージのいずれでも入出力（I/O）が極めて速くなります。大規模な深層学習のデータ供給で広く使われている標準的なやり方の1つです。

## 直感的な理解
本棚から1冊ずつ本を取りに行く場面を想像してください。1冊ごとに「どの棚の何段目か」を探して歩いて取りに行くと、本そのものを読む時間より「探して歩く時間」のほうが長くなります。これがファイルを1つずつ開くときに起きていることで、OS は毎回「そのファイルがディスクのどこにあるか」を調べに行きます（メタデータ参照とシーク）。

WebDataset の発想は「あらかじめ必要な本を箱に順番に詰めておき、箱ごと運んで上から順に読む」ことです。探して歩く時間が消え、純粋に読む速度だけで律速されるようになります。深層学習では同じデータを何十エポックも繰り返し読むので、この差が学習全体の時間を大きく左右します。

機械学習の学習では、何十万〜何百万枚もの画像と、それに対応するラベルや数値を繰り返し読み込みます。素朴に「画像1枚=ファイル1個」と保存すると、たとえば1サンプルあたりカメラ複数枚の画像 + 数値データ + メタ情報があり、10万サンプルなら数十万〜百万個の小ファイルになります。GPU は1枚あたり高価な計算資源なので、データが間に合わず GPU が遊ぶ状態（I/O ボトルネック）は大きな損失です。WebDataset はこの問題を「小ファイルを大きな tar に固める」shard packing（シャード化）で解決します。

## 基礎: 前提となる概念

### I/O ボトルネックとは
深層学習の1ステップは「データを読む → GPU で計算する → 重みを更新する」の繰り返しです。GPU の計算が速くても、データの読み込みが追いつかなければ GPU はデータを待って遊びます。この「データ供給が計算より遅い状態」を I/O ボトルネックと呼びます。GPU 利用率（`nvidia-smi` の Volatile GPU-Util の値）が低いのに学習が遅いときは、たいていデータ供給が原因です。近年は GPU 演算性能（FLOPS）が世代ごとに大きく伸びる一方、ストレージの帯域やレイテンシの改善は緩やかなので、相対的に I/O が律速になりやすい状況が続いています。

### ランダムアクセスとシーケンシャルアクセス
- ランダムアクセス: バラバラの場所のデータを順不同で読む。小ファイルを1つずつ開くのはこれにあたる。回転式ハードディスク（HDD）では物理ヘッドが移動するため遅く、クラウドのオブジェクトストレージ（Amazon S3 など）では1ファイル=1 HTTP リクエストになりコストとレイテンシが大きい。
- シーケンシャルアクセス: データを先頭から末尾まで一直線に読む。ストレージが最も得意とする読み方で、HDD はヘッドが動かず、SSD は先読み（プリフェッチ）が効き、S3 は数十MBを1リクエストでまとめて取れる。

WebDataset は前者を後者に置き換えることで速くしています。1個あたりのレイテンシは多少あっても、後述のプリフェッチで隠せるため問題になりません。

### スループットとレイテンシ
- レイテンシ: 1個のデータを取り出すのにかかる待ち時間。
- スループット: 単位時間あたりに供給できるデータ量（例: 何サンプル/秒、何GB/秒）。

学習で効くのは主にスループットです。多少のレイテンシがあっても、パイプラインで先読み（prefetch）しておけば、GPU には途切れなくデータが流れます。WebDataset はシーケンシャル読みでストレージの帯域（GB/秒）を使い切ることでスループットを最大化します。

## 仕組みを詳しく

### tar とは何か
tar（ターと読む。tape archive の略で、もとはテープにファイルを連結して書く形式）は「複数のファイルを1つに連結する」古くからある形式です。圧縮ではなく連結が本質で、内部は次のように並びます。

```
[ヘッダ: 名前 s00000000.left.jpg, サイズ N1][中身 N1 バイト]
[ヘッダ: 名前 s00000000.right.jpg, サイズ N2][中身 N2 バイト]
[ヘッダ: 名前 s00000000.label.cls,  サイズ N3][中身 N3 バイト]
[ヘッダ: 名前 s00000001.left.jpg, ...][中身 ...]
...
```

中身が先頭から順番に並んでいるので、先頭から末尾まで一直線に読むだけで全データが取れます。これがシーケンシャル I/O であり、ストレージが最も得意とする読み方です。tar には各メンバの先頭に固定512バイトのヘッダがあり、ファイル名・サイズ・更新時刻などが入ります。データ本体も512バイト境界にパディングされるため、極端に小さいファイルを大量に入れるとパディング分のオーバーヘッドが出ますが、画像のように数KB〜数百KBのファイルでは無視できる量です。

### サンプルのまとめ方（同じ接頭辞でグループ化）
WebDataset の中心ルールは「ファイル名の最初のドットより前（接頭辞、`__key__` と呼ぶ）が同じファイル群を、1つのサンプルとしてまとめる」というものです。たとえば次のように並べておくと、

```
s00000000.left.jpg    ← サンプル0、左カメラ画像
s00000000.right.jpg   ← サンプル0、右カメラ画像
s00000000.label.cls   ← サンプル0、ラベル
s00000001.left.jpg    ← サンプル1...
```

接頭辞 `s00000000` が同じものを自動でまとめ、Python の辞書（dict）として渡してくれます。拡張子（jpg, cls, json, npy など）が辞書のキーになります。

```python
sample = {
    "__key__": "s00000000",
    "left.jpg":  b"...バイト列...",
    "right.jpg": b"...",
    "label.cls": b"7",
}
```

この設計の利点は、データ形式が完全に自由なことです。画像は JPEG、数値は NumPy の `.npy`、メタは JSON と、それぞれ自然な形式のまま入れられ、デコードはあとで好きな関数に任せられます。重要なのは「同じサンプルに属するファイルは tar 内で隣り合って連続している必要がある」点で、これがあるからこそ1度のシーケンシャル走査でサンプルを順に組み立てられます。

### シャード化（packing）
シャード（shard）は「破片・分割片」の意味で、データセット全体を一定サンプル数ごと（例: 1000サンプル）に区切った tar 1個を指します。10万サンプルを1000サンプル/シャードで詰めると100個の tar になり、ファイル数が百万個から100個へ激減します。書き出し側の擬似コードは次の通りです。

```python
samples_per_shard = 1000
shard_idx = 0
tar = tarfile.open(f"train-{shard_idx:06d}.tar", "w")
for i, sample in enumerate(dataset):
    write_member(tar, f"s{i:08d}.jpg",  encode_jpeg(sample.image, quality=90))
    write_member(tar, f"s{i:08d}.cls",  str(sample.label).encode())
    if (i + 1) % samples_per_shard == 0:
        tar.close()
        shard_idx += 1
        tar = tarfile.open(f"train-{shard_idx:06d}.tar", "w")
```

動画や生信号からフレームを切り出す前処理を伴う場合、ここで1度だけデコードして JPEG として固めておくと、学習時には重い動画デコードをせずに済みます。前処理で重い処理を済ませ、学習ループは「読んで GPU に流す」だけにするのが定石です。シャードのサイズは「数百MB〜数GB」に収めるのが経験則で、小さすぎるとシャード切り替えのオーバーヘッドとリクエスト数が増え、大きすぎると分散学習で負荷が偏る・途中再開の粒度が粗くなります。

### 読み込み側のパイプライン
読み込みは「tar の列挙 → シャッフル → デコード → バッチ化」のパイプラインで書きます。

```python
import webdataset as wds
urls = "train-{000000..000099}.tar"          # ブレース展開で100個指定
ds = (wds.WebDataset(urls, shardshuffle=True, nodesplitter=wds.split_by_worker)
        .shuffle(1000)                         # バッファ1000でサンプルシャッフル
        .decode("pil")                         # jpg→PIL, npy→ndarray などへ自動デコード
        .to_tuple("jpg", "cls")                # 必要なキーをタプルに
        .map_tuple(transform_img, int))
loader = wds.WebLoader(ds, batch_size=64, num_workers=8)
```

`decode` はバイト列を拡張子に応じてデコードし、`to_tuple` でモデルが欲しいキーだけ取り出します。出力のバッチ形状（tensor shape）はたとえば `(B, 3, 224, 224)` の画像テンソルと `(B,)` のラベルになります（B はバッチサイズ、3 は RGB チャネル、224×224 は画素）。S3 から読む場合は URL を `pipe:aws s3 cp s3://bucket/train-{000000..000099}.tar -` のようにストリームコマンドにするだけで、ローカルと同じコードのまま透過的に扱えます。

### シャッフルの工夫（buffer shuffle）
通常のデータセットは全データをメモリに載せてランダムに並べ替えますが、シャードは順次ストリーミングが前提なのでそれができません。WebDataset はバッファシャッフルで近似します。

仕組み: 大きさ N（例 1000）のバッファを用意し、tar から読んだサンプルを入れる。満タンになったらバッファからランダムに1個取り出して学習に渡し、空いた分を tar から補充する。これを繰り返すと、完全ランダムではないが「ほどよく混ざった」順序になります。加えてシャードファイル自体の順番も `shardshuffle=True` で混ぜると、相関がさらに減ります。バッファを小さくしすぎると、近接サンプル（同じ動画の連続フレームなど）が固まって出てきてバイアスになる点に注意が要ります。逆にバッファを大きくするほど真のランダムに近づきますが、その分メモリを食います。

```
tar の流れ:  s0 s1 s2 s3 s4 s5 s6 s7 ...
              ↓ バッファ(容量4)に充填
バッファ:    [s0 s1 s2 s3] → ランダムに s2 を出力、s4 を補充
              [s0 s1 s4 s3] → ランダムに s0 を出力、s5 を補充 ...
出力順:      s2 s0 s3 s1 s4 ...  (局所的にシャッフル)
```

二段シャッフル（シャード順をシャッフルし、さらにバッファ内をシャッフルする）にすると、「どのシャードから読むか」と「シャード内のどのサンプルを出すか」の両方がランダム化され、エポック間で異なる順序になります。

### 分散学習での分け方（nodesplitter / split_by_worker）
複数 GPU・複数 DataLoader worker で読むとき、同じシャードを2回読むと無駄です。`nodesplitter`（ノード=GPUプロセス間の分割）と `split_by_worker`（プロセス内の worker 間の分割）で、シャードを重複なく割り当てます。シャード100個を4プロセス×2worker で読むなら、各 worker が約12〜13個ずつ担当し、全体をちょうど1周します。シャード数が worker 総数の倍数でないと末尾に偏りが出るため、シャード数は worker 総数より十分多くしておくのが実務上の鉄則です。倍数でない場合は、エポック境界でドロップするか、シャードを循環させて長さを揃えるかの選択になり、前者は一部データを毎エポック捨て、後者は一部を多く見るというトレードオフが生じます。

## 手法の系譜と主要論文

データ I/O を高速化する「小ファイルをまとめて順次読む」発想には複数の系譜があります。

- TFRecord（Google, TensorFlow, 2015 頃〜）: Protocol Buffers でシリアライズしたサンプルを連結したバイナリ形式。TensorFlow エコシステムの標準。中身は専用形式なので汎用ツールでは覗きにくいが、TensorFlow の `tf.data` と密に統合され前処理パイプラインを構築しやすい。
- WebDataset（Aizman, Maltby, Breuel, NVIDIA。"High Performance I/O For Large Scale Deep Learning", IEEE BigData 2019, arXiv:2001.01858 およびオープンソース実装）: 提案の核心は「深層学習の I/O を POSIX のランダムファイルアクセスから tar のシーケンシャルストリーミングへ移す」こと。当時すでにデータセットが巨大化し、数百万の小ファイルのランダム読みがストレージのボトルネックになっていた。特にクラウドのオブジェクトストレージはランダムアクセス前提にできない。tar に固めて順次読むことでストレージ帯域を使い切る。中身が普通の tar なので `tar tf shard.tar` で人間が覗け、JPEG や JSON をそのまま入れられる汎用性が特徴。ローカル・NFS・HTTP・S3 を同じURL記法（パイプ経由のストリーム）で透過的に扱える点も実用上大きい。
- MosaicML Streaming / MDS（2022〜）: クラウド学習に最適化した形式。サンプル単位の決定的シャッフルと、途中から再開できる resumption（エポック途中で中断しても同じ位置から続けられる）、複数ノードでの効率的なシャード分配を強化。WebDataset の弱点だった「厳密なシャッフルと確実な再開」に踏み込んだ後継的位置づけ。
- FFCV（Leclerc et al., MIT, 2022, arXiv:2306.12517）: 独自バイナリ + JIT デコード + メモリマップで ImageNet 学習を極端に高速化。データローディングを徹底的に最適化した研究例で、CPU 側の前処理がボトルネックにならないよう設計されている。
- NVIDIA DALI: JPEG デコードや画像変換そのものを GPU で行い、CPU を解放する。WebDataset と組み合わせて使うこともできる。

大規模言語モデルや CLIP（Radford et al., OpenAI, 2021, arXiv:2103.00020）など、Web 規模のデータを扱う学習では、tar ベースのストリーミングが事実上の標準になっています。LAION-5B（Schuhmann et al., 2022, arXiv:2210.08402）のような数十億規模の画像テキストデータセットも WebDataset 形式で配布されています。

## 論文の実験結果(定量データ)

WebDataset 系の評価で測られる主要指標は次の通りです。意味とともに示します。

- スループット（samples/sec または MB/sec）: 単位時間に供給できるデータ量。これが GPU の処理速度を上回れば GPU は遊ばない。WebDataset の原論文（IEEE BigData 2019）は、小ファイルのランダム読みに比べ tar シーケンシャル読みでローカル SSD でもネットワークストレージでも供給スループットが大きく向上し、回線・ディスク帯域の上限近くまで到達できることを示した。具体的な数字は環境（ストレージ種別・帯域）依存だが、報告では複数の並列ストリームで読むことでネットワーク帯域を飽和させられるとされる。
- GPU 利用率: I/O が律速だと数十%に落ち込むが、シャード化で先読みが効くと90%以上に保てる。これは「同じ GPU を買っても学習が1.5〜2倍速く終わる」ことに直結し、計算コスト削減として重要。
- S3 など API 課金とリクエスト数: 1ファイル=1リクエストだと百万サンプルで百万リクエスト課金になるが、1000サンプル/シャードなら千リクエストに減り、リクエストコストがおよそ3桁下がる。レイテンシ面でも、リクエスト確立のオーバーヘッドが千分の1になる。

FFCV（Leclerc et al., 2022）の論文では、最適化したデータローディングにより ResNet-50 の ImageNet 学習を従来パイプライン比で数倍高速化したと報告されている。報告では1台のマシンで ResNet-18 を ImageNet 上で十数分規模、ResNet-50 でも従来の数分の一の壁時計時間で学習できたとされる。ボトルネックがデータ供給にあったことの裏返しで、I/O 形式の選択が学習全体の壁時計時間（wall-clock time）を支配しうることを定量的に示した例といえる。

アブレーション的に整理すると、(1) シャード化をやめて小ファイルに戻すとスループットが大きく落ちる、(2) buffer shuffle のバッファを極端に小さくすると、近接サンプルの相関で検証精度がわずかに劣化する、(3) シャードを1個だけ巨大にすると分散時の負荷が偏る、(4) `num_workers` を増やすとプリフェッチが効いて GPU 待ちが減るが、増やしすぎると CPU・メモリを食って逆効果になる、という依存関係が一般に観察されます。

## メリット・トレードオフ・限界

メリット
- 小ファイル大量問題を解消し、ストレージ帯域を使い切る（GPU が遊ばない）。
- クラウドのオブジェクトストレージと相性がよい（1シャード=数リクエスト、API 課金が激減）。
- 中身がただの tar なので `tar tf shard.tar` で人間が確認でき、デバッグが容易。形式が自由（JPEG, JSON, npy 混在可）。
- ローカル・NFS・HTTP・S3 を同じコードで透過に扱える。分散学習で worker/node 単位の分割が簡単。
- 前処理段階で重いデコード（動画→フレーム等）を済ませ、学習ループを軽くできる。

トレードオフ・限界
- ランダムアクセスができない。「サンプル番号で1個だけ取り出す」用途（インデックス参照、ランダムな評価サブセット抽出）には不向き。
- シャッフルがバッファ近似で完全ランダムではない。バッファが小さいとサンプル相関が残り、学習にバイアスが出うる。
- 分散学習での厳密な再現性・途中再開（resumption）が素の WebDataset では弱い。MosaicML Streaming はこの点を改善した。
- 前処理コストが先払い。後からデータ項目を追加・修正するには再パックが必要で、大規模だと作り直しの手間が大きい。
- 圧縮ではなく連結なので、容量削減効果はない（元の小ファイル合計とほぼ同じ）。圧縮したい場合は中身のファイルを個別に圧縮するか tar 全体を gzip するが、後者はランダムなシャード途中再開と相性が悪い。
- シャードのサンプル数が中途半端だと末尾シャードが小さくなり、分散時に偏りが生じる。

未解決・研究課題: 「巨大データでの厳密シャッフルと効率的なランダムアクセスの両立」「途中再開時のデータ位置の正確な復元」「異種データ（テキスト・画像・動画）を1ストリームで効率的に混ぜる」は、データローダ設計で今も改良が続く領域です。

## 発展トピック・研究の最前線
- 決定的シャッフルと再開可能性: MosaicML Streaming のように、エポック途中で停止しても同じ順序で続行でき、かつ各サンプルが各エポックでちょうど1回出る性質を保証する設計。大規模 LLM 事前学習で重要で、数週間に及ぶ学習が途中で落ちても無駄なくやり直せる。
- オンザフライ変換と GPU デコード: NVIDIA DALI のように JPEG デコードや画像変換を GPU で行い、CPU をさらに解放する流れ。WebDataset とも組み合わせられる。
- データの重み付き混合: 複数ドメイン（言語・分野）のシャードを比率を変えて混ぜる data mixing は、LLM の性能を左右する研究テーマ。混合比そのものを学習対象にする研究（DoReMi など）も登場している。
- ストリーミング下のデータ重複除去・品質フィルタ: Web 規模データでは重複や低品質サンプルが性能を下げるため、シャード生成時のフィルタリング戦略が研究対象。
- ストレージ階層との統合: ローカル SSD をキャッシュとして使い、初回はリモートから読み2回目以降はローカルから読む二層構成で、リモート帯域とローカル速度の両取りをする実装が広がっている。
