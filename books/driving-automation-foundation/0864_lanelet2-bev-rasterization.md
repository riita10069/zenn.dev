# ベクタHDマップのBEVラスタライズ

## ひとことで言うと
HD マップ(車線レベルの精密地図)が「線や多角形の集まり」(ベクタ形式)で与えられるとき、それを上空から見下ろした 1 枚の色付き画像(BEV 画像、Bird's-Eye-View)に描き起こすことを「ラスタライズ」と言います。こうすると地図を「もう 1 台のカメラ画像」のように扱え、画像用のニューラルネットワーク(CNN)にそのまま入れられます。ラスタ化は実装が簡単で動かしやすい反面、地図が本来持つ「つながりの情報」(トポロジー)を捨ててしまうという代償があります。

## 直感的な理解
人間がカーナビを見るとき、画面に描かれた道路の絵を見ます。それは地図データそのものではなく、地図データを「描画」した結果です。コンピュータビジョンのモデルも同じで、画像処理が得意な CNN にとっては「道路の絵」のほうが「道路の座標データ」より扱いやすいのです。

地図には大きく 2 つの表し方があります。

- ベクタ形式(vector): 道路の境界線・車線中央線・停止線などを「点を結んだ折れ線」(polyline、ポリライン)や「多角形」として、座標と属性(これは車線境界、これは停止線、など)で表す。Lanelet2 がこれにあたります。データは小さく構造化されていますが、画像ではないので CNN にそのまま入りません。
- ラスタ形式(raster): 同じ地図を、ピクセルの格子に色を塗った画像として表す。画像なので CNN に入りますが、元の構造(どの線がどの車線につながっているか、というつながり=トポロジー)は色の並びの中に埋もれて失われます。

カメラ画像を CNN で処理するモデルにベクタ地図を渡したいなら、ベクタをラスタに変換する必要があります。これがラスタライズで、これをやれば地図は「追加のカメラ視点」としてカメラと完全に同じ経路で扱え、モデルの入力形式をシンプルに保てます。なお、もともと地図を画像として持っているデータセットではこの変換は不要で、ラスタライズは主にベクタ地図を持つデータセットで必要になります。

## 基礎: 前提となる概念
いくつかの言葉を噛み砕きます。

BEV(Bird's-Eye-View、鳥瞰図)とは、真上から見下ろした 2 次元の俯瞰表現です。カメラは斜めから前を見ますが、運転の判断(どの車線にいて、どこで曲がるか)は俯瞰で考えるほうが自然なので、BEV はプランニング(経路計画)で多用されます。

ROI(Region of Interest、関心領域)とは、地図全体のうち自車周辺の必要な範囲だけを切り出した領域です。世界中の地図を描く必要はなく、たとえば自車前後 50m × 左右 30m のような窓だけを描きます。

エゴ中心・進行方向上向き(ego-centric, heading-up)とは、自車を画像中央に置き、自車の進行方向が常に画像の真上を向くように地図を回転させて描く流儀です。こうすると、自車がどの方角を向いて走っていても、モデルから見た地図の見え方が一貫します(常に「自分は上を向いて前進している」絵になる)。これは方角に依存しない学習(rotation invariance に近い効果)のための定石です。

トポロジー(topology)とは、要素どうしの「つながり方」です。地図では、ある車線が次のどの車線につながるか、隣にどの車線があるか、どの停止線がどの交差点に属するか、といった関係を指します。ラスタ化するとこの関係は明示的には残らず、ピクセルの配置から間接的に推測するしかなくなります。

## 仕組みを詳しく
ベクタ地図を BEV ラスタタイルに変換する典型的な処理を、段階的に分解します。

### ステップ1: エゴ周辺の地図要素を取り出す
自車の現在位置を中心に、ROI 内のレーンレットや境界線を地図データベースから問い合わせます。座標系は地図ローカル(地図原点からのメートル、たとえば UTM 投影に整列)で、自車姿勢も境界線も同じ座標系なので直接重ねられます。

### ステップ2: エゴ中心・進行方向を上に向ける
タイルは自車中心で作ります。前方(+X)が画像の上、左(+Y)が画像の左になるよう、自車のヘディング(進行方向)が常に真上を向くように地図全体を回転させて描きます。具体的には、地図ローカル座標の点を、自車位置を引いてから自車ヨー角の逆回転をかけ、画像座標に投影します。

```
点(地図座標) → 平行移動(自車位置を原点に) → 回転(ヨーを打ち消す) → スケール(m→px) → 画像座標
```

### ステップ3: 要素ごとに色を塗る
各地図要素を、種類に応じた色で描画ライブラリ(OpenCV など)を使って画像に描きます。意味ごとに色を分けることで、モデルに「ここは道路境界」「ここは停止線」という属性を明示的に与えます。色テーブルの一例:

```
道路境界(road border)        → 緑
縁石(curbstone)              → 濃い緑/ライム
柵・壁(fence/guard rail/wall)→ 茶系
実線車線境界(solid lane line) → 青
中央線(centerline)           → 暗赤(破線)
停止線(stop line)            → 赤
横断歩道(pedestrian marking) → 黄
```

車線境界を「太いグレーの下地ストローク + その上に細い色付きストローク」と二重に描いて見やすくしたり、中央線を破線で描いたりといった工夫をします。結果は `(H, W, 3)` の uint8 RGB 画像になります。

### ステップ4: カメラと同じ前処理を通す
できあがった RGB タイルを、カメラフレームと同じバックボーン前処理(リサイズ + ImageNet 平均・分散での正規化)に通し、`(3, H, W)` のテンソルにして視点スタックの末尾に積みます。これでモデルから見ると地図も他のカメラ画像と区別がつきません。

### キャッシュ
同じシーンの地図ファイルを毎フレーム再パースすると遅いので、プロセスごとに一度だけ読んでメモリに保持する(LRU キャッシュなど)のが定石です。地図はフレーム間で不変だからです。

### 設計上の選択肢: ゼロプレースホルダ vs ラスタ化
地図入力チャネルの扱いには段階的な選択肢があります。

- ゼロプレースホルダ: 地図が無い、またはまだ未対応のデータセットでは、地図スロットを全ゼロのダミー画像で埋める。モデルの入力形は保てるが、地図情報はゼロ(=何も無い)として扱われる。最小実装で経路だけ通したいときの一手。
- ラスタ化: 上記のように実際にベクタ地図を描画して意味のある地図タイルを入れる。情報量は増えるが描画コストと情報損失(トポロジー喪失)を伴う。

研究実装では、まずゼロプレースホルダで全体の経路を通し、地図が効くことを確認してからラスタ化に進む、という段階的アプローチがよく取られます。

## 手法の系譜と主要論文
地図表現をどうモデルに与えるかは、ラスタ vs ベクタの綱引きの歴史です。

ChauffeurNet(Bansal et al., RSS 2019, arXiv:1812.03079)は、HD マップ・経路・他車・交通灯などをすべて俯瞰のラスタ画像チャネルに描き、それを CNN に入れて運転方策を学びました。動機は、多様な情報を「画像チャネル」という統一形式で CNN に流したかったことです。この設計は「中間表現としてラスタ地図を使う」流儀を普及させました。トレードオフは、ラスタ化で地図のグラフ構造が失われ、情報量が画像解像度に縛られる点です。本トピックのラスタライズはこの古典的アプローチの系譜にあります。

VectorNet(Gao et al., CVPR 2020, arXiv:2005.04259)は、その反動として、地図と軌跡をラスタ化せずベクタ(ポリライン)のままグラフニューラルネットワーク(GNN)で符号化しました。動機は「ラスタ化はトポロジーを壊し、画像化で解像度と計算が無駄になる」という指摘です。少ない計算で高精度な軌跡予測を達成し、ベクタ地図エンコーダの流れを作りました。トレードオフは、可変長のグラフ(要素数がシーンごとに違う)を扱う実装が複雑になる点です。

MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)は、オンラインで(走行中にカメラから)ベクタ HD マップそのものを構築する手法で、地図要素を順序不変な点集合としてモデル化し、Transformer で end-to-end に予測します。VAD(Jiang et al., ICCV 2023, arXiv:2303.12077)は、ベクタ化された地図表現の上で直接プランニングを行い、ラスタ化の情報損失を避けつつ高速化を達成しました。これらは「ベクタのまま扱うほうが情報が豊か」という主張で、ラスタライズのトレードオフを浮き彫りにします。

地図フォーマット側では、Lanelet2(Poggenhans et al., ITSC 2018)がベクタ地図の代表的標準であり、ラスタ化のたびに「この豊かな構造を捨ててよいのか」という問いを突きつけます。

## 論文の実験結果(定量データ)
ラスタ vs ベクタの差は、軌跡予測ベンチマークで定量化されています。代表的な指標は次の通りです。ADE(Average Displacement Error、予測軌跡と正解軌跡の各時刻の位置誤差をメートルで平均したもの、小さいほど良い)と FDE(Final Displacement Error、最終時刻の位置誤差、小さいほど良い)、そして miss rate(予測が一定距離以上外れた割合、小さいほど良い)です。

VectorNet は Argoverse 軌跡予測ベンチマークで、同等の計算量のラスタ化ベースライン(ConvNet で地図画像を符号化する方式)に対し、予測誤差を明確に下げたと報告しました。論文では、ラスタ表現と比べてベクタ表現が DE(変位誤差)を相対的に改善し、かつモデルの計算量(FLOPs)を桁で削減できることを示しています。これは「ラスタ化はトポロジーを失うだけでなく計算も無駄」という主張の定量的裏付けです。

MapTR は nuScenes のオンライン地図構築タスクで、地図要素(車線分割線・道路境界・横断歩道)の mAP を、ラスタ化セグメンテーションをベクタ化する先行手法 HDMapNet(Li et al., ICRA 2022)の約 23 mAP に対しておよそ 50 mAP 前後へと倍以上に引き上げ、同時に HDMapNet が 0.x FPS 級だったのに対し 20FPS 超のリアルタイムに近い推論を達成しました。これは「地図をラスタの中間表現を経由して復元する」より「最初からベクタの点集合として end-to-end に予測する」ほうが精度・速度の両面で優れることの直接的な証拠です。VAD は nuScenes のオープンループ・プランニング評価で、平均 L2 軌跡誤差(1/2/3 秒先の計画軌跡と人間の運転の差、メートル、小さいほど良い)をラスタ中間表現を使う UniAD の約 1.03m から約 0.72m へ、平均衝突率(collision rate、計画が障害物とぶつかる割合、小さいほど良い)を 0.31% から 0.22% 前後へ低減し、しかも推論速度を UniAD 比でおよそ 9 倍弱に高速化したと報告しています。ベクタ表現が「精度」と「速度」を同時に改善した点が重要です。

これらの数値が示すのは、地図のトポロジーを保つベクタ表現が、予測・計画の精度で一貫して有利であり、ラスタ化は「動かしやすさ」と「精度・効率」を引き換えにしている、という構図です。一方で、ラスタ化は CNN バックボーンをそのまま流用できる単純さがあり、まず動かす段階や、地図が補助的な役割にとどまる設計では依然として現実的な選択肢です。

## メリット・トレードオフ・限界
ラスタライズの利点は明快です。地図を「もう一つのカメラ画像」として扱えるので、カメラ用バックボーンの処理経路をそのまま再利用でき、入力形式が一本化されます。実装がシンプルで、まず動かす(経路を通す)のに向きます。エゴ中心・進行方向上向きに正規化すれば、向きに依存しない一貫した地図表現になります。色分けで道路境界・車線・停止線・横断歩道などの意味を明示的に与えられます。

トレードオフと限界は深刻です。ラスタ化でベクタ地図のトポロジー(車線のつながり・隣接関係)が失われ、上で見たように予測・計画精度で不利になります。描画コストがフレームごとにかかり、大規模学習では I/O・CPU のボトルネックになりえます。画像解像度に情報量が縛られ、細い線や遠方の要素は潰れます。色テーブルの設計が恣意的で、属性の取りこぼしや色の衝突が起こりえます。

代替のベクタエンコーダ(GNN や Transformer)は情報を保てますが、可変長グラフを固定サイズの出力にする実装が複雑で、学習も不安定になりやすいという課題があります。どちらを選ぶかは、地図がモデルにとって主役か脇役か、計算予算がどれだけあるか、で決まります。

未解決の課題として、ラスタとベクタの良いとこ取り(ラスタの扱いやすさとベクタのトポロジー保持を両立する表現)はまだ確立されていません。グラフを画像チャネルに埋め込みつつ隣接情報を補助的に与えるハイブリッド表現などが模索されています。

## 発展トピック・研究の最前線
最近の End-to-End 運転モデルは、地図を「事前に与えるラスタ入力」ではなく「カメラからオンラインで推定するベクタ出力」として扱う方向に進んでいます(MapTR, VAD の流れ)。これにより、HD マップが整備されていない地域でも走れる「mapless(地図なし)」運転が現実味を帯びています。

また、地図要素を言語モデルのトークンのように系列化して Transformer で扱う研究や、地図のトポロジー(車線グラフ)を明示的にグラフ構造で予測する研究(TopoNet, Li et al., 2023, arXiv:2304.10440 など)も活発で、「つながりを保ったまま学習する」方向が主流になりつつあります。OpenLane-V2 ベンチマークでは、車線中央線どうしの接続関係や、車線と交通標識・信号の対応関係を「トポロジー mAP(つながりの正しさを測る指標)」として明示的に評価しており、ラスタ表現ではそもそもこの指標を測る対象(つながり)が表現できないことが、ベクタ・グラフ表現へ移行する強い動機になっています。ラスタライズはこの文脈では「枯れた、動かしやすいベースライン」という位置づけになり、新規性を競う場面では避けられる傾向があります。一方で、地図が運転方策の主役ではなく、カメラ知覚を補助する弱い事前情報にとどまる設計や、計算予算が厳しく CNN バックボーンを共有したい設計では、ラスタ化は今なお合理的な第一選択であり続けます。重要なのは「ラスタかベクタか」を絶対視せず、地図がそのシステムでどれだけの役割を担うかに応じて選ぶことです。
