# features_only マルチスケール特徴抽出

## ひとことで言うと
分類用に作られたCNN(たとえばResNet50)から、「最後の分類結果」ではなく「途中の複数段(ステージ)の特徴マップ」をまとめて取り出す使い方です。多くのライブラリでは `features_only=True` のようなフラグ一つで実現できます。こうすると、細かい模様を捉えた浅い段の特徴と、意味のある物体を捉えた深い段の特徴の両方を同時に使えます。物体検出やセグメンテーション、自動運転の知覚など、「画像のどこに何があるか」を扱うタスクで標準的に用いられる手法です。バックボーンを差し替えても同じインターフェースで多段特徴を取り出せるので、実験の公平な比較や再現にも欠かせません。

## 直感的な理解
ResNet50のようなCNNは、もともと画像分類(この写真は犬か猫か、など)のために作られています([ResNet (残差ネットワーク)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0359_resnet_note) 参照)。だから最後には「1000クラスのどれか」を1つ出すように設計されています。途中で計算した豊かな特徴マップは、最後に大域平均プーリング(空間全体を1点に潰す処理)と全結合層を通って、1個のラベルへと圧縮されて捨てられます。せっかく作った「どこに何があるか」の空間情報が、最後に消えてしまうのです。

ところが、自動運転の知覚やプランニングで欲しいのは「最後の1個のラベル」ではありません。「画像のどこに、どんな物体が、どのくらいの大きさであるか」を表す、空間的に広がった豊かな特徴です。そこで、分類ヘッド(最後の判定部分)を取り外し、途中の特徴マップを取り出して使います。

ここで「特徴のスケール(解像度)」が決定的に効いてきます。ネットワークの浅い段は解像度が高く(たとえば56×56)、エッジや細かいテクスチャを捉えます。深い段は解像度が低く(たとえば7×7)、その代わり特徴マップの1マスが画像の広い範囲を見ていて(受容野が広い)、車・人といった意味のある物体を捉えます。

物体は、近くにあれば画像上で大きく、遠くにあれば小さく写ります。小さい物体は浅い高解像度の特徴で、大きい物体は深い低解像度の特徴で捉えるのが得意です。なぜなら、深い段では小さい物体が解像度の低下で潰れて消えてしまい、浅い段では大きい物体が受容野に収まりきらないからです。だから「1段だけ」使うと、どちらかを必ず取りこぼします。複数段(マルチスケール、multi-scale)を同時に使えば、小さい物体も大きい物体も拾えます。`features_only` は、この「複数段の特徴マップを取り出す」を、自分でネットワークを切り貼りせず一行で実現する仕組みです。

## 基礎: 前提となる概念
いくつかの言葉を噛み砕きます。

特徴マップ(feature map)とは、CNNが画像から抽出した中間表現で、(チャンネル, 高さ, 幅) の3次元テンソルです。高さ・幅は空間的な位置に対応し、チャンネルは「その位置で検出した特徴の種類」に対応します。浅い段ほど高さ・幅が大きく(高解像度)チャンネルが少なく、深い段ほど小さく(低解像度)チャンネルが多い、という傾向があります。チャンネルが増えるのは、抽象度が上がるほど「区別すべき特徴の種類」が増えるからです。

ステージ(stage)とは、CNNを解像度ごとに区切ったまとまりです。ResNet50なら、stem(最初の処理)と4つのstageの計5段に分かれ、ステージが進むたびに解像度が半分、チャンネルが倍になります。

ダウンサンプリング率(downsampling ratio、または stride)とは、入力画像に対して特徴マップがどれだけ縮んでいるかの比です。1/4なら入力の4分の1の解像度、という意味です。深い段ほどこの率が大きく(より縮んで)なります。FPNの文脈では P2〜P5(それぞれ 1/4〜1/32)のように、ダウンサンプリング率で段を呼ぶ慣習があります。

受容野(receptive field)とは、出力の1マスが入力画像のどれだけの範囲の情報を集約しているかの広さです。深い段ほど受容野が広く、画像の大域的な文脈を見ています。マルチスケールが効くのは、段ごとに受容野が異なり、見ている対象の大きさが違うからです。

ライブラリの統一API。多くの事前学習済みモデルを集めたライブラリ(timm = PyTorch Image Models, Ross Wightman 製が代表)は、`features_only=True` のような共通フラグと、各段の情報を返す `feature_info` のようなメタデータを、多くのモデルに横断的に用意しています。これにより、ResNet50・Swin Transformer・ConvNeXtといった異なるバックボーンを、同じインターフェースで差し替えられます。

## 仕組みを詳しく
`features_only=True` を指定すると、ライブラリはモデルの振る舞いを次のように変えます。

- 最後の全結合層(分類ヘッド)と大域平均プーリングを取り外す。
- forward(順伝播)の戻り値を「途中の各ステージの特徴マップのリスト」にする。
- どの段を返すか、各段のチャンネル数はいくつか、ダウンサンプリング率はいくつか、というメタ情報を `feature_info` として持たせる。
- どの段を出すかを `out_indices` で選べる(全段でなく必要な段だけ取り出してメモリを節約できる)。

ResNet50で `features_only=True` のときに返るのは通常5段です。各段のチャンネル数は 64, 256, 512, 1024, 2048 です。入力が256×256のとき、おおよそ次の形(tensor shape は (チャンネル, 高さ, 幅))になります。

```
段0 : (  64, 128, 128)  ← stem、1/2 解像度、低レベルなエッジ
段1 : ( 256,  64,  64)  ← stage1、1/4
段2 : ( 512,  32,  32)  ← stage2、1/8
段3 : (1024,  16,  16)  ← stage3、1/16
段4 : (2048,   8,   8)  ← stage4、1/32、高レベルな意味的特徴
```

これらをリストで受け取り、後段の処理(特徴の融合、地図のエンコード、検出ヘッドなど)が、それぞれの解像度を用途に応じて使い分けます。たとえば物体検出では、浅い段で小さい物体を、深い段で大きい物体を検出します。バッチ次元を含めれば段4は (B, 2048, 8, 8) のような4次元テンソルで、B はバッチサイズ(同時処理するサンプル数)です。

実装上の重要な工夫が「チャンネル数の自動取得」です。下流の層をバックボーンの出力に正しく配線するには、各段が何チャンネルかを知る必要があります。これを手で書くとバックボーンを差し替えるたびに修正が要りますが、`feature_info` を見れば各段のチャンネル数(timmなら `num_chs`)を機械的に読めます。典型的な実装では、まず `feature_info` があればそこから読み、無いカスタムモデルの場合はダミー入力を1回流してみて返ってきた特徴マップのチャンネル次元を数える、というフォールバック(保険)を用意します。これにより、ライブラリ製モデルでも自作モデルでも同じインターフェースで扱えます。この「設定をコードに焼き込まず実物から取得する」流儀は、バックボーンの総当たり比較実験を支える地味だが重要な工夫です。

before/after で整理すると、

- before(普通の分類モデル): forwardの出力は (B, 1000) のクラススコア1本。空間情報は失われている。
- after(features_only): forwardの出力は5段の特徴マップのリスト。空間情報と複数スケールが保たれる。

なお、Swin Transformer のような階層型 Transformer も timm では features_only に対応し、CNN と同じ4段(1/4, 1/8, 1/16, 1/32)の特徴を返します。ただし内部のトークン列を空間グリッドに整形し直す処理が挟まる点が CNN と異なります。インターフェースが揃っているおかげで、下流コードはバックボーンの中身を意識せず使えます。

## 手法の系譜と主要論文
マルチスケール特徴の有用性そのものは、物体検出の歴史とともに確立されてきました。画像ピラミッド(画像を何段階にも縮小して各サイズで検出する)という古典的手法が出発点です。これは効果はあるものの、サイズごとに何度も推論するので計算が重いという欠点がありました。

決定的な進展が Lin, Dollár, Girshick, He, Hariharan, Belongie, "Feature Pyramid Networks for Object Detection", CVPR 2017 (arXiv:1612.03144、FPN) です。FPNは、バックボーンが自然に作る複数段の特徴マップ(まさに features_only で取り出すもの)を、トップダウン経路(深い段から浅い段へ意味情報を流す)と横方向の結合(lateral connection)で統合し、各スケールで検出する仕組みを提案しました。動機は「1段だけでは大小の物体を同時に捉えられない、しかし画像ピラミッドは重い」という二律背反の解消です。深い段は意味は豊かだが解像度が粗く、浅い段は解像度は細かいが意味が薄い。FPNは深い段の意味を浅い段へ流し込むことで「細かくて意味も豊か」な特徴を全スケールで作りました。効果は、追加計算をほとんど増やさずに小さい物体の検出精度を大きく向上させたことでした。FPNはその後の検出・セグメンテーションのほぼ標準部品になり、「バックボーンの複数段をマルチスケール特徴として使う」という発想を定着させました。features_only でマルチスケール特徴を取り出すのは、この知見を活かすための実装上の入口です。

FPN以降も特徴融合の改良が続きました。PANet (Liu et al., "Path Aggregation Network for Instance Segmentation", CVPR 2018) は、FPNのトップダウンに加えてボトムアップ経路を足し、浅い段の正確な位置情報を深い段へも伝えました。BiFPN (Tan et al., EfficientDet, CVPR 2020) は、双方向の融合を重み付きで何度も繰り返し、効率と精度を両立しました。これらはいずれも「features_only で取り出した多段特徴をどう統合するか」を競う系譜です。

ResNet自体は He et al., CVPR 2016(残差学習、[ResNet (残差ネットワーク)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0359_resnet_note) 参照)で、features_only はその「途中表現の取り出し方」であり、別の手法というよりライブラリの機能です。

ライブラリ面では、timm (PyTorch Image Models, Ross Wightman, 2019〜) が数百の事前学習済みモデルを統一APIで提供し、`features_only=True` と `feature_info` という共通契約を多くのモデルに整備したことが大きな貢献です。これにより、研究者は数行コードを変えるだけでバックボーンを総当たりで比較でき、再現性と公平な比較が一気に進みました。論文の実験表で「同じ下流タスクをバックボーンだけ変えて比較」できるのは、この統一APIの恩恵が大きいです。

## 論文の実験結果(定量データ)
features_only という機能自体に固有の数値はありませんが、その背後にある「マルチスケール特徴が効く」ことの定量的根拠はFPNが明快に示しました。

Lin et al. (2017) は、COCO物体検出データセット(80クラス、小さい物体を多く含む難しいベンチマーク)で、Faster R-CNN(当時の標準検出器)にFPNを組み込むと、単一スケール特徴を使う構成に比べて AP(Average Precision、適合率を様々な条件で平均した総合検出精度の指標、高いほど良い)が大きく向上することを報告しました。報告された全体APはおおよそ 33% 前後から 36% 台へと数ポイント改善し、検出タスクではこれは大きな差です。特に小さい物体に対する AP_small が顕著に改善し(数ポイント規模)、マルチスケールが小物体検出に効くことを実証しました。AP は IoU(予測枠と正解枠の重なり度合い)を 0.5 から 0.95 まで変えながら平均する厳しい指標で、わずか数ポイントの差でも検出器の優劣を分ける重みがあります。

アブレーション的に重要なのは、「深い段だけ(最も意味的な1段)」「浅い段だけ」を使う構成より「複数段を統合」する構成のほうが一貫して良い、という結果です。論文のアブレーションでは、トップダウン経路を外す、横結合を外す、といった要素を1つずつ削ると AP が落ちることが示され、各部品の寄与が定量化されました。深い段だけでは小物体を、浅い段だけでは大物体や意味的判断を取りこぼします。これが「1段では足りない」ことの直接的な証拠です。

バックボーン選択の観点では、timmの統一APIのおかげで、同じ下流タスクに対してResNet50・Swin・ConvNeXtなどを features_only で差し替えて比較する実験が容易になりました。一般に、同程度の計算量ならTransformer系やConvNeXt系がResNet50を数ポイント上回ることが多いものの、ResNet50は軽量・安定で、強いベースラインとして残ります。具体的な優劣は下流タスクとデータ規模に依存し、「常にこれが最良」という単純な結論はない、というのが実務的な知見です。だからこそ、差し替えて測れる仕組み自体に価値があります。

## メリット・トレードオフ・限界
メリット。

- 一行で複数段のマルチスケール特徴が手に入り、自分でネットワークを切り貼りする必要がありません。
- `feature_info` でチャンネル数・ダウンサンプリング率を機械的に取得でき、下流の配線を自動化できます。バックボーンの差し替えがコード変更ほぼ無しで可能です。
- 浅い段(細部)と深い段(意味)の両方を使えるので、大小の物体・近遠の対象を取りこぼしにくくなります。
- 統一APIにより、複数バックボーンの公平な比較実験が容易で、再現性が高まります。

トレードオフ・限界。

- 複数段を保持・処理する分、メモリと計算が単段より増えます。
- 浅い段は解像度が高く特徴マップが大きい(たとえば256チャンネル×64×64)ため、メモリを特に食いやすいです。高解像度入力では浅い段がメモリのボトルネックになります。要素数で見ると段1(256×64×64=約105万)は段4(2048×8×8=約13万)の8倍にもなり、浅い段がメモリを支配します。
- どの段をどう使うかは下流側の設計次第で、全段を漫然と使えば必ず良いわけではありません。タスクに合わせて取り出す段を選ぶ必要があります(`out_indices` で絞る)。
- バックボーンごとに段数・ダウンサンプリング率・チャンネル数が異なるため、下流が特定の構成を前提にしていると、差し替え時に整合を取る作業が要ります(`feature_info` がこれを緩和しますが、ゼロにはなりません)。特に Transformer 系は段の解像度や正規化の流儀が CNN と異なり、そのまま挿し替えると微妙な不整合が出ることがあります。

## 発展トピック・研究の最前線
マルチスケール特徴の使い方そのものが、今も活発な研究テーマです。FPNを拡張したPANet(ボトムアップ経路を追加)、BiFPN(双方向に何度も特徴を融合、EfficientDetで採用)などが、特徴融合の効率と精度を改良しました。NAS-FPN のように融合構造そのものを自動探索する研究もあります。

近年は、CNNだけでなくTransformer系バックボーン(Swin Transformerなど)も features_only でマルチスケール特徴を出すのが標準で、これらをDETR系の検出器やBEV(鳥瞰図)知覚に組み込む流れが主流です。特に自動運転では、複数カメラの features_only 特徴を3次元のBEV空間へ投影して統合する手法(BEVFormer, Li et al., ECCV 2022, arXiv:2203.17270 など)が広く使われており、マルチスケール特徴抽出はその最初の一歩を担います。BEVFormer は多段の画像特徴を空間クロスアテンションで BEV クエリへ集約し、検出・地図認識を同時に行います。features_only はこうした下流の複雑な処理に「素材」を供給する土台です。

研究者として深掘りするなら、「どの段の特徴がどのタスクにどれだけ寄与するか」を定量化するアブレーションや、段ごとの特徴を可視化して何を捉えているかを調べる解析が出発点になります。また、高解像度の浅い段のメモリコストをどう抑えるか(特徴を間引く、低精度で持つ、勾配チェックポイントで再計算する、など)は、実用上の重要な未解決課題です。さらに、事前学習(ImageNet分類)で得た特徴が下流タスクの段ごとにどれだけ転移するか、どの段を凍結しどの段をファインチューニングすべきか、という転移学習の最適配分も継続的な研究テーマです。
