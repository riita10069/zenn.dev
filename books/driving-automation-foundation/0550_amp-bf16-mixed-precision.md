# AMP / bf16混合精度

## ひとことで言うと

ニューラルネットの中の数値は、ふつう32ビット(fp32)の細かい精度で計算します。混合精度学習(AMP, Automatic Mixed Precision)とは、計算の一部をわざと16ビットの粗い精度で行い、メモリ使用量を減らし計算を速くする技術です。16ビット形式のうち bf16(bfloat16)は fp32 と同じ「表せる数の範囲」を持つため、扱いが簡単で安定しています。比較的新しい世代以降の GPU/TPU では bf16 がほぼ標準の学習精度になっています。

## 直感的な理解

定規で長さを測る場面を想像してください。ミリ単位の細かい目盛り(fp32)は正確ですが、目盛りを読むのに時間がかかり、記録する桁数も多くなります。多くの場面では1センチ単位の目盛り(bf16)でも十分で、読むのも速く記録も短い。一方、ごく細かい差が結果を左右する場面(loss の合計など)だけは細かい目盛りを使う。「ほとんどは粗く、要所だけ細かく」という使い分けが混合精度です。

なぜそうしたいのか。ニューラルネットの学習では GPU のメモリ(VRAM)が常に不足しがちで、計算時間も長い。数値を16ビットにすればメモリは半分になり、GPU には16ビット計算専用の高速回路(Tensor Core)があるので計算も速くなります。粗くしても学習結果がほとんど変わらないなら、得しかありません。実際、メモリが半分になれば同じ GPU でバッチサイズを倍にできたり、これまで OOM(メモリ不足)で回らなかった高解像度入力が回せるようになったりします。

## 基礎: 前提となる概念

コンピュータは小数を「浮動小数点数(floating point)」で表します。浮動小数点数は「符号」「指数部(exponent、数の大きさのスケールを表す)」「仮数部(mantissa、細かさを表す)」の3つのビットからできています。指数部が広いほど表せる数の範囲が広く、仮数部が多いほど細かく正確に表せます。

- fp32(単精度、32ビット): 指数部8ビット、仮数部23ビット。範囲も細かさも十分。1数あたり4バイト。表せる範囲はおおよそ 10^-38 〜 10^38。
- fp16(半精度、16ビット): 指数部5ビット、仮数部10ビット。1数あたり2バイト。仮数部は多めだが、指数部が狭いため「表せる数の範囲」が狭く(おおよそ 6×10^-5 〜 65504)、大きい数や小さい数で破綻しやすい。
- bf16(bfloat16、16ビット): 指数部8ビット(fp32 と同じ)、仮数部7ビット。1数あたり2バイト。範囲は fp32 と同じだが、仮数部が少ないので細かさは fp16 より粗い。

```
            符号  指数部   仮数部     表せる範囲      細かさ
  fp32       1     8       23         広い(10^±38)   細かい
  fp16       1     5       10         狭い(±65504)   やや細かい
  bf16       1     8        7         広い(fp32同)    粗い
```

この表が混合精度のすべての挙動を説明します。bf16 が「範囲は広いが粗い」、fp16 が「範囲は狭いが少し細かい」という対比が、後述する GradScaler の要否を決めます。仮数部7ビットの bf16 は、有効数字でいうとおおよそ2〜3桁しか表せませんが、ニューラルネットの学習はこの程度の粗さに耐えるよう設計されています(そもそも確率的勾配降下自体が大きなノイズを含むため)。

## 仕組みを詳しく

「混合」とは、どの計算を16ビットにし、どの計算を32ビットに保つかを使い分けることです。一般に次のように分けます。

- 行列積や畳み込みなど、量が多く多少粗くても影響が小さい計算 → 16ビット(bf16)で高速・省メモリに。
- loss の合計、ソフトマックス、正規化(layer norm / batch norm)、指数・対数を含む計算など、細かい精度や広いダイナミックレンジが結果を左右する計算 → fp32 のまま。
- モデルの重み本体(マスターコピー) → fp32 で保持し、更新の正確さを守る。小さな重み更新が16ビットの粗さで丸められて消えるのを防ぐため。

多くのフレームワークの autocast 機能がこの使い分けを自動でやってくれます(だから Automatic Mixed Precision)。どの演算をどの精度にするかは内部のホワイトリスト/ブラックリストで決まっています。

```python
with autocast(device_type="cuda", dtype=bfloat16, enabled=use_amp):
    output = model(inputs)     # この中の重い演算は自動で bf16 になる
    loss = loss_fn(output, target)   # loss 計算は fp32 に保たれる
loss.backward()
optimizer.step()
```

### fp16 と bf16 の決定的な違い: GradScaler が要るかどうか

fp16 は「表せる数の範囲」が狭いという弱点があります。学習中の勾配([勾配クリッピング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0576_gradient-clipping))はとても小さい値になることがあり、fp16 の下限(約 6×10^-5)を下回るとその小さな値が表現できずに 0 に潰れます(アンダーフロー)。勾配が 0 になればその重みは更新されず、学習が進みません。これを防ぐため、fp16 では「GradScaler(loss scaling)」という仕組みで loss をいったん大きな数(例: 1024倍)に拡大してから backward し(連鎖律により勾配も同じ倍率で拡大される)、更新の直前に元のスケールへ戻す手順が必要でした。拡大係数が大きすぎると今度はオーバーフロー(inf)するため、inf が出たらスケールを半分にし、しばらく安定したら倍にする、という動的調整も要ります。さらに勾配クリッピングと併用する際は「スケールを戻してからクリップ」という順序制約も生まれます([勾配クリッピング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0576_gradient-clipping))。

bf16 は範囲が fp32 と同じなので、小さな勾配が 0 に潰れる問題が起きません。したがって GradScaler は不要で、コードがシンプルかつ安定します。「bf16 has fp32 dynamic range, so no loss scaling is needed」というのがこの意味です。bf16 のデメリットは仮数部の細かさが fp16 より粗いことですが、ニューラルネットの学習は多少の数値ノイズに強く、実用上ほとんど問題になりません。この「実装の単純さ」と「数値の安定性」こそが、大規模学習で bf16 が fp16 を置き換えた最大の理由です。

### メモリの数値イメージ

ある中間テンソルの形状が `(B=8, C=1440, H=7, W=7)` だとすると、要素数は 8×1440×7×7 = 564,480 個です。

```
fp32 で持つと: 564,480 × 4バイト ≈ 2.26 MB
bf16 で持つと: 564,480 × 2バイト ≈ 1.13 MB
```

ネットワーク中の多数の中間結果(活性化、activation)すべてでこれが効くため、合計のメモリ節約は数GB規模になります。その結果、「fp32 では OOM で回らなかったバッチサイズや高解像度が、bf16 なら回る」ことが起きます。注意すべきは、節約されるのは主に活性化(順伝播の中間結果で、逆伝播のために保持されるもの)のメモリであり、fp32 マスター重みとオプティマイザ状態は依然 fp32 で保持されるため、削減効果はモデルとバッチの構成に依存する点です。バッチが大きく活性化が支配的なモデルほど bf16 の恩恵が大きく、パラメータが支配的でバッチが小さいモデルでは効果が限定的です。

## 手法の系譜と主要論文

- Micikevicius et al., "Mixed Precision Training", ICLR 2018(NVIDIA/Baidu, arXiv:1710.03740)。混合精度学習を確立した代表論文です。重い演算を fp16 で行いつつ、(1) 重みのマスターコピーを fp32 で保持、(2) 小さい勾配が潰れないよう loss を拡大する loss scaling、(3) 一部の累積を fp32 で行う、という3つの工夫を提案しました。狙いは fp16 のメモリ・速度メリットを取りつつ精度を落とさないこと。loss scaling 係数の調整という追加の複雑さが、後の bf16 採用の動機になりました。

- Wang & Kanwar, "BFloat16: The secret to high performance on Cloud TPUs", Google AI, 2019。Google が TPU 向けに広めた bf16 形式の設計思想を解説しました。fp32 と同じ指数部を持たせることで、既存の fp32 前提のコードや学習レシピをほぼそのまま動かせる、という実務的利点を強調しています。

- Kalamkar et al., "A Study of BFLOAT16 for Deep Learning Training", 2019(Intel/Google/NVIDIA など, arXiv:1905.12322)。bf16 を学習に使う有効性を体系的に調べました。bf16 は fp32 と同じ指数部を持つため loss scaling なしでも学習が安定し、精度は fp32 とほぼ同等に保てることを示しました。狙いは混合精度をより簡単・安全に使えるようにすること。効果は実装の単純化(GradScaler 不要)です。

- NVIDIA, TensorFloat-32(TF32), Ampere アーキテクチャ whitepaper, 2020。行列積の内部で仮数部を切り詰めた中間形式 TF32(指数部8ビット、仮数部10ビット相当)を使い、コード変更なしに fp32 の行列積を高速化する仕組みを導入しました。bf16 ほど積極的ではないが「黙っていても速くなる」中間段階です。

- Micikevicius et al., "FP8 Formats for Deep Learning", 2022(NVIDIA/Arm/Intel, arXiv:2209.05433)。さらに低い8ビット(fp8、E4M3 と E5M2 の2形式)を学習・推論に使う形式を提案し、bf16 の次の段階を切り開きました。混合精度の流れが16ビットから8ビットへ進みつつあることを示します。

## 論文の実験結果(定量データ)

- Mixed Precision Training(Micikevicius et al., 2018)は、画像分類(ImageNet 上の AlexNet/VGG/ResNet/Inception 等)、物体検出、音声認識、機械翻訳、言語モデリングと幅広いタスクで検証されました。いずれも fp16 + loss scaling + fp32 マスター重みの混合精度が、fp32 単独とほぼ同等の精度(差は誤差範囲)を保ったまま、メモリ使用量をおおよそ半減できることを示しました。アブレーションとして、fp32 マスター重みを省くと精度が落ち、loss scaling を省くと小さい勾配が消えて学習が悪化することを示し、各要素がなぜ必要かを明確にしました。たとえば一部のタスクでは、loss scaling なしの素朴な fp16 学習は勾配の大半がアンダーフローして発散・精度劣化したことが報告されています。

- Kalamkar et al.(2019)は画像・言語・推薦など複数タスクで bf16 を評価し、bf16 が loss scaling なしで fp32 とほぼ同等の最終精度に達することを示しました。fp16 で必要だった loss scaling の手間を省けることが、bf16 の実務的な核心的利点として定量化されています。

- 速度面では、Tensor Core を用いた行列積・畳み込みで、fp32 比でおおよそ 2〜3 倍前後のスループット向上が一般に報告されます(モデルや層構成に依存し、メモリ帯域律速の層では恩恵が小さい)。大規模言語モデルの学習報告では、メモリと計算の制約から bf16 が事実上の標準になり、大規模になるほど fp16 の数値不安定(loss スパイク)が顕在化しやすいため範囲の広い bf16 が好まれる、という実務知見が広く共有されています。

- FP8(2022 以降)では、大規模 Transformer の学習で bf16 比でさらにメモリ削減と高速化(報告では学習スループットで 1.3〜2 倍前後)が示されつつありますが、E4M3/E5M2 のどちらをどの経路に使うか、テンソルごとのスケーリングをどう管理するかが安定性の鍵で、bf16 より調整が難しいことも報告されています。

数値の意味を補足します。「精度が fp32 とほぼ同等」とは、ImageNet の top-1 精度などで差が 0.1〜0.3 ポイント程度の誤差範囲に収まる、という意味です。この小さな代償でメモリ半減と数倍の高速化が得られるため、混合精度は「ほぼ無料の高速化」として広く採用されています。

## メリット・トレードオフ・限界

メリット
- GPU メモリ使用量(主に活性化)が減り、より大きなバッチ・高解像度を同じ GPU で扱える。
- Tensor Core により行列積・畳み込みが高速化し、学習時間が短くなる。
- bf16 は fp32 と同じ範囲を持つため、GradScaler 不要でコードがシンプルかつ安定。大規模学習での loss スパイクも起きにくい。

トレードオフ・限界
- bf16 は仮数部(細かさ)が fp32 より粗い。通常の学習では問題にならないが、極端に精度が要る計算(一部の正規化や累積和、長い系列の累積)では誤差が出るため fp32 に残す設計が要る。
- 古い GPU は bf16 に対応しておらず使えない(対応は比較的新しい世代以降)。fp16 しか速くない GPU もあり、その場合は GradScaler が必要になる。
- すべての演算を闇雲に16ビットにすると loss 計算などで精度が落ちる。何を fp32 に残すか(autocast の挙動)の理解が必要。
- 数値が不安定な経路や不適切な入力では nan が出ることがあり、混合精度の問題か別の問題かの切り分けが要る。bf16 でも、入力が正規化されていない・loss 設計が悪いといった根本問題までは救えない。
- 未解決の課題として、fp8 など8ビット学習の安定化は活発な研究領域で、どの層をどの精度にするかの自動最適化や、低ビットでの数値安定化手法(per-tensor/per-block スケーリング)が探求されています。

## 発展トピック・研究の最前線

- fp8 学習: 8ビット形式(E4M3 / E5M2)を用いた学習で、bf16 比でさらにメモリと速度を改善する研究が進行中。テンソルごとのスケーリング管理がより難しくなる。
- TF32(TensorFloat-32): Ampere 以降が行列積の内部で使う形式で、コード変更なしに fp32 計算を高速化する中間形式。精度は fp32 と bf16 の中間。
- 量子化推論(int8 / int4)とは別物だが、低精度の系譜として関連が深い。学習は混合精度、推論はさらに低ビット量子化、という役割分担が一般的です。
- 混合精度は勾配の大きさや数値スケールに影響するため、勾配クリッピングのしきい値や学習率スケジュールと相互作用します。チェックポイントでは fp16 の loss スケーリング係数も保存対象になります([勾配クリッピング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0576_gradient-clipping) [学習率スケジューラ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0578_lr-scheduler) [チェックポイント(保存・再開)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1161_checkpointing-save-resume))。
