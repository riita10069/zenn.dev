# VectorMapNet

## ひとことで言うと
カメラや LiDAR の観測から、その場で「車線・道路境界・横断歩道」などの地図要素を、ピクセルの塗り分けではなく「折れ線(ポリライン)の座標列」として直接出力する手法です。地図を「点の並び」というベクトル形式でエンドツーエンドに作る、初めての本格的な手法として知られ、オンライン HD マップ構築という研究分野の起点になりました。

## 直感的な理解
自動運転で使う地図要素は、本来「線」です。車線中心線や道路境界は「ここからここへ伸びる1本の線」、横断歩道は「四角い領域の輪郭」。経路計画や制御はこうした線・ポリゴンの座標を直接使いたい。地図を画像のように塗り分けても、結局そこから線を取り出す手間が残ります。

ところが [BEVマップセグメンテーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0269_bev-map-segmentation) のような BEV セグメンテーションが出すのは「各マスが車線か否か」というピクセル塗り分け(ラスタ)です。これをそのまま使うと、線として使うには「細線化(skeletonize)→連結→ベクトル化」という壊れやすい後処理が必要で、1本1本の線を区別(インスタンス分離)できず、隣り合う2本の車線がくっついて1かたまりに見え、出力が画像解像度に縛られて滑らかな曲線を表現しにくい、という問題が出ます。たとえば 0.5m グリッドのラスタからは、それより細かい車線の曲率は復元できません。

そこで「最初から地図要素をベクトル(座標列)として、ニューラルネットに直接出させてしまえばいい」という発想が生まれました。これが VectorMapNet です。エンドツーエンド(end-to-end)とは、途中に手作りの後処理を挟まず、入力から最終出力まで1つの学習可能なネットワークで一気通貫に行うことを指します。後処理を学習対象に含めることで、地図品質を最終損失で直接最適化できます。

## 基礎: 前提となる概念

ベクトル地図とは、地図要素を「クラス(車線境界か、道路境界か、横断歩道か)」と「順序づいた点の列(ポリライン)」で表したものです。たとえば 1 本の車線境界は `[(x1,y1), (x2,y2), ..., (xn,yn)]` のような座標列で表されます。これに対しラスタ地図はピクセルの塗り分けです。ベクトルは解像度に縛られず、少ない点で滑らかな線を表せるのが利点です。

DETR(DEtection TRansformer、Carion et al., ECCV 2020)という物体検出の枠組みを理解しておくと、この分野の手法がすっと入ります。DETR は「学習可能なクエリ(query)」を多数(例 100 個)用意し、Transformer デコーダがそれぞれのクエリに「ここに、このクラスの物体がある(または無い)」を1個ずつ割り当てます。重複検出を抑える後処理(NMS = Non-Maximum Suppression)が不要で、予測と正解をハンガリアンマッチング(二部グラフの最小コスト1対1割り当てを解く手法)で対応づけて学習します。VectorMapNet の第1段はこの考え方を地図要素の検出に応用したものです。

自己回帰生成(autoregressive)も鍵概念です。これは「直前までの出力を見て次の1個を出す」生成方式で、言語モデルが単語を1つずつ吐くのと同じです。座標を1点ずつ順に生成すれば可変長のポリラインを表現できますが、逐次なので遅く、途中の誤りが後に伝わりやすい(exposure bias / 誤差蓄積)欠点があります。

評価指標は mAP(mean Average Precision)が中心です。予測したポリラインと正解ポリラインの一致を、チャンファー距離(Chamfer distance、2つの点列で互いに最近傍点までの距離を平均した非対称量を双方向で取ったもの)が閾値以下なら正解、として AP を計算し、クラスとしきい値(典型的に 0.5/1.0/1.5m)で平均します。1 に近いほど良く、HD マップ要素を「線として」どれだけ正しく出せたかを測ります。

## 仕組みを詳しく

VectorMapNet は地図構築を「2段階(coarse-to-fine、粗→細)」に分けて解きます。

前段: BEV 特徴の構築。複数カメラ(必要なら LiDAR も)の特徴を、[BEVマップセグメンテーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0269_bev-map-segmentation) と同じく自車中心の BEV 特徴マップ `(C, X, Y)` に変換します。これが共通の土台になります。

第1段: 要素検出(Map Element Detector)。DETR に似た仕組みで「地図要素の候補」をまとめて検出します。学習可能な element query(要素クエリ)を N 本(例 100 本)用意し、Transformer デコーダが BEV 特徴を参照しながら各クエリに対して、その要素のクラス(車線境界 / 道路境界 / 横断歩道)と、大まかな位置を表す少数のキーポイント(key points、例: 数個の代表点)を予測します。ここで出るのは「だいたいこのあたりに、このクラスの線がある」という粗い表現です。クエリ方式なので、要素ごとに別インスタンスとして自然に分離され、セグメンテーションのインスタンス分離問題が解消されます。

第2段: ポリライン生成(Polyline Generator)。第1段が出した各要素について、その粗いキーポイントを条件(condition)として、要素の詳細な形を表すポリライン(連続した点列)を生成します。ここが VectorMapNet の特徴で、点の座標を離散化(量子化、たとえば BEV 範囲を細かいビンに分割)してトークンのように扱い、Transformer が `(x1,y1), (x2,y2), ...` と自己回帰的に順に吐いていきます。出力は1要素あたり可変長の点列、たとえば `(num_points, 2)` 形状の座標列です。可変長を扱える点が、固定点数の手法に対する自己回帰の理論的な強みでした。

イメージ図(ラスタ vs ベクトル):

```
セグメンテーション(ラスタ)         VectorMapNet(ベクトル)
  ┌───────────────┐                要素1(車線): (12,3)->(12,40)->(11,78)
  │ ░░░██░░░░██░░ │   →           要素2(車線): (60,5)->(61,44)->(63,80)
  │ ░░░██░░░░██░░ │                要素3(横断歩道輪郭): (...)
  └───────────────┘                ※線が個別に、座標として出る
   どのピクセルが線か             どの点をどう結ぶか
```

学習は、第1段の検出をハンガリアンマッチングで正解の地図要素に1対1対応づけ(コストはクラス確率と位置の組み合わせ)、第2段のポリラインを正解点列との損失で学習します。なぜ2段に分けるか。位置と存在(粗)を先に決めてから形(細)を埋めると、いきなり長い座標列を直接出すより学習が安定するからです。これは物体検出で「まず物体の有無と大まかな箱→次に細かい形」と段階を踏むのと同じ発想です。

## 手法の系譜と主要論文

- Qi Li et al., "HDMapNet: An Online HD Map Construction and Evaluation Framework" (ICRA 2022)。BEV セグメンテーション結果を後処理でベクトル化する半エンドツーエンド手法で、オンライン HD マップという問題設定と評価指標(チャンファー距離ベースの AP)を整備した先駆です。後処理が複雑で誤差が乗りやすいのが課題で、VectorMapNet はこの後処理依存を取り除いた点が新規でした。

- Yicheng Liu, Tianyuan Yuan, Yue Wang, Yilun Wang, Hang Zhao, "VectorMapNet: End-to-end Vectorized HD Map Learning" (ICML 2023、arXiv:2206.08920)。センサ入力から地図要素をベクトルで直接出す初のエンドツーエンド手法。動機は、ラスタセグメンテーション+後処理の壊れやすさを排し、地図を本来の形(ベクトル)で学ぶこと。新規性は、DETR 風の要素検出とポリラインの自己回帰生成を組み合わせた2段構成です。

- Bencheng Liao et al., "MapTR: Structured Modeling and Learning for Online Vectorized HD Map Construction" (ICLR 2023、arXiv:2208.14437)。各地図要素を「固定数の点(例 20 点)+並び順の等価性(permutation-equivalence、同じ形を表す点の並び順の曖昧さを学習で吸収する)」で表し、VectorMapNet の自己回帰生成をやめて全点を並列に1回で予測しました。動機は VectorMapNet の遅さと誤差蓄積の克服で、効果は大幅な高速化と精度向上。これが現在の主流(並列予測)になりました。

- Bencheng Liao et al., "MapTRv2: An End-to-End Framework for Online Vectorized HD Map Construction" (arXiv:2308.05736, 2023)。補助的な one-to-many マッチング、PV(透視ビュー)・BEV 双方での補助セグメンテーション損失、中心線(centerline)の追加などで MapTR をさらに精度・速度の両面で伸ばしました。

- 点の表現をさらに改良したのが [PivotNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0459_pivotnet)(要素を少数の「ピボット点」で表し点数を動的にする)、時間方向に拡張して連続フレームの一貫性を担保したのが [StreamMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0463_streammapnet) です。系譜をまとめると、HDMapNet(後処理ベクトル化)→ VectorMapNet(初の end-to-end・自己回帰)→ MapTR/MapTRv2(並列予測で高速化)→ PivotNet / StreamMapNet(表現と時間の改良)という流れになります。

## 論文の実験結果(定量データ)

評価は nuScenes のオンライン HD マップ構築ベンチマークで、車線境界(lane divider)・道路境界(boundary)・横断歩道(pedestrian crossing)の3クラスを mAP(チャンファー距離ベース、しきい値 0.5/1.0/1.5m で平均)で測るのが標準です。

VectorMapNet (Liu et al., 2023) は、当時の強いベースライン(HDMapNet を含む後処理ベクトル化手法)を mAP で明確に上回りました。報告では、カメラのみの設定で全体 mAP がおよそ 40 台に達し、後処理ベクトル化手法に対して十数ポイント規模の改善を示しています。LiDAR を加える、あるいはカメラと融合するとさらに向上します。論文は、ラスタ→後処理という経路に比べ、ベクトルを直接学ぶ方が地図要素の品質(特に細い線の連続性とインスタンス分離)で優れることを示しました。アブレーション的には、ポリライン生成段を外して粗いキーポイントだけにすると形状の精度が落ち、2段構成(粗→細)の有効性が確認されています。

MapTR (Liao et al., 2023) は、VectorMapNet を上回る精度と、決定的に大きな速度差を報告しました。VectorMapNet が自己回帰生成のため推論が遅い(報告でおよそ 数 FPS)のに対し、MapTR は全点並列予測でリアルタイムに近い速度(報告でおよそ 25 FPS 前後、軽量設定)を達成し、同時に mAP も VectorMapNet より十数ポイント高い水準(カメラのみで 50 台前後)に引き上げました。MapTRv2 はさらに mAP を伸ばし 60 台に達しています。これは「自己回帰をやめて並列化する」設計変更が、速度と精度の両方を改善した好例で、以後の手法はほぼ並列予測を採用しています。アブレーションでは、permutation-equivalent な点表現を外すと、同じ形の線でも点の並び順の違いが損失を不安定にし、精度が落ちることが示されています。

数値の意味を補足すると、mAP が数ポイント上がるのは「しきい値(例 1.0m)内に正しく入る地図要素が増えた」ことを意味します。地図要素はそのまま経路計画の入力になるため、横断歩道や車線境界の検出漏れ・誤検出が減ることは、下流の安全性に直結する改善です。一方で、後述するように、この nuScenes の標準分割は train と val が地理的に近接しており、「同じ街での内挿」を測っている面があるため、額面どおりの mAP が未知の街でも出るとは限りません。

## メリット・トレードオフ・限界

メリット
- 地図要素を本来のベクトル(座標列)で直接出力。後処理(細線化・連結)が不要。
- 要素を個別インスタンスとして分離して出せる(隣接車線がくっつかない)。
- 出力が画像解像度に縛られず、少ない点で滑らかな曲線を表現できる。

トレードオフ・限界
- ポリラインを点ごとに自己回帰生成するため推論が遅く、リアルタイム性に課題(MapTR の並列予測が直接の解決策)。
- 自己回帰生成は学習時(教師強制)と推論時(自前の出力を入力)のズレ(exposure bias)で、途中の誤りが後続に伝播(誤差蓄積)しやすい。
- 1フレーム単独の予測で、時間的な一貫性(連続フレーム間で同じ車線がブレない)は保証しない。これを解決するのが [StreamMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0463_streammapnet)。
- 学習に正解ベクトル地図(HD マップ由来)が必要で、データ作成コストが高い。さらに、ベンチマークの train/val 分割が地理的に近いため「未知の街でどれだけ汎化するか」は過大評価されやすい、という指摘がある(Lilja et al. などが geographically disjoint split で再評価し、mAP が大きく落ちることを報告)。

研究上の未解決課題としては、未知地域への汎化、遠方・交差点での精度、地図要素間の関係(車線の接続関係、合流・分岐)をグラフとして出力する lane graph の生成、そして検出した地図の不確実性をどう下流(計画)に伝えるかがあります。

## 発展トピック・研究の最前線
- MapTR / MapTRv2: 並列予測で高速・高精度。現在のベースライン。
- [PivotNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0459_pivotnet): 要素を少数のピボット点で表現し、点数を動的にして表現効率を上げる。
- [StreamMapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0463_streammapnet): BEV 特徴とクエリを時間方向に伝播させ、連続フレームの地図のちらつきを抑える。
- 車線グラフ生成(lane graph / centerline topology): LaneGAP, TopoNet, TopoMLP などが、地図要素の塗り分け・線だけでなく「どの車線がどの車線につながるか」の接続関係まで出力する方向へ進めている。
- 事前地図(SD マップ)併用: P-MapNet などが OSM 相当の粗い道路地図を弱い事前情報として注入し、遠方・オクルージョンを補う。
- 汎化評価の見直し: 地理的に重ならない分割での再評価が進み、見かけの mAP より未知地域での頑健性が重視されつつある。
