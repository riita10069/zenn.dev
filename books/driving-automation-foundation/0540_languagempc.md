# LanguageMPC

## ひとことで言うと
LanguageMPC は、大規模言語モデル(LLM。大量の文章で学習し、言葉で論理を述べたり判断を説明したりするのが得意な AI)に「どう振る舞うべきか」という高レベルの方針だけを決めさせ、実際にハンドルやアクセルをどう動かすかという細かい制御は、制御工学で数十年使われてきた数学的手法 MPC(Model Predictive Control, モデル予測制御)に任せる手法です。「考える役(LLM)」と「動かす役(MPC)」を意図的に分離(decouple)したのが本質で、両者の長所を組み合わせつつ短所を打ち消すことを狙います。

## 直感的な理解
自動運転で一番難しいのは、実は「ハンドルを何度切るか」という低レベルの操作そのものではなく、「いま誰に道を譲るべきか」「合流のタイミングをどう取るか」という高レベルの状況判断です。後者は人間の常識・経験・文脈理解がものを言う領域で、ここを純粋な数式やルールで書き切るのは非常に難しい。たとえば無信号の交差点では、相手の意図(行きたいのか待っているのか)を読み、社会的なルール(先に来た方が優先、目を合わせて譲り合うなど)を踏まえて判断する必要があります。これを if-else のルールで網羅すると、例外の例外が無限に増えて破綻します。

一方で、いったん方針が決まれば「その方針を満たす滑らかで物理的に無理のないペダル・ハンドル操作を求める」のは、最適化計算が得意とする領域です。LanguageMPC の発想は、この2つの難しさが質的に違うことに注目し、それぞれを最も得意なツールに割り当てることです。人間に例えるなら、ベテラン教官(LLM)が「右から来る車に道を譲って、ゆっくり合流しなさい」と方針を言い、熟練した運転技術(MPC)がその方針を滑らかなペダル・ハンドル操作へ翻訳する、という役割分担です。教官は具体的なハンドル角度までは言いません。それは運転技術側の仕事だからです。

## 基礎: 前提となる概念
理解に必要な概念を順に噛み砕きます。

MPC(モデル予測制御)とは何か。MPC は「システムのモデル(=入力を与えると未来の状態がどうなるかを表す数式)」を使い、未来数秒ぶんの動きを頭の中でシミュレーションしながら、最も望ましい操作系列を最適化計算で求める制御手法です。車であれば「アクセルをこれだけ踏むと数秒後にこの速度・位置になる」という運動モデルを持ち、未来 N ステップ(=予測ホライズン、たとえば 3 秒先まで)の軌道を予測し、コスト関数(=どれだけ望ましくないかを測る数値)を最小にする操作を解きます。コスト関数はふつう「目標速度からのズレ」「急なハンドル操作」「他車への接近」といった複数の項を、それぞれ重み(weight)を付けて足し合わせた形をしています。数式で書くと、状態 x、入力 u、未来 k ステップに対して

```
J = Σ_{k=0}^{N-1} [ w_track ・(目標速度ズレ)² + w_smooth ・(操作量の変化)² + w_safe ・(他車接近ペナルティ) ] + 制約
```

の J を最小にする入力系列 u を解きます。w_track, w_smooth, w_safe が各項の重みで、これが「何を重視するか」を決めます。求めた操作系列のうち最初の1ステップだけを実行し、次の時刻でまた解き直す(これを receding horizon, 後退ホライズンと呼びます)。航空機や化学プラントでも使われる枯れた技術で、出力が安定し、物理的に無理のない滑らかな動きを保証できるのが強みです。

MPC の弱点。上の重み w_* や、安全のための制約(constraint。たとえば「この車との距離は最低 5m 空ける」)は、人間が状況に応じて設定する必要があります。「前の車にどれだけ車間を空けるか」「車線変更を許すか」「どの車との関係を最重視するか」といった方針は、複雑で曖昧な状況(渋滞の合流、譲り合いが必要な無信号交差点)では機械的に決めるのが難しい。MPC は与えられた重みと制約の下では最適ですが、その重みと制約を誰が、どんな状況判断に基づいて決めるのか、という上流の問題が残るのです。従来はこれを有限状態機械(状況に応じてモードを切り替えるルールの集合)や手書きのヒューリスティクスで埋めていましたが、シナリオが増えるほど保守が破綻しました。

純 LLM 運転の弱点。逆に LLM だけで運転させる手法(座標列をそのまま LLM に出力させる方式)は、状況理解や常識的判断は得意な一方、出力する座標がガタつく(時間方向に滑らかでない)、物理的に実現不可能な軌道を出す、推論が遅くリアルタイム性に欠ける、といった問題を抱えます。LLM は「言葉で考える」のは得意でも「物理法則に従った滑らかな数値列を吐く」のは本質的に不得意なのです。

Chain-of-Thought(CoT)とは。LLM に答えを出させる前に、途中の推論ステップを言葉で書かせる手法です(Wei et al., NeurIPS 2022)。「まず A だから B、よって C」と段階的に書かせると、いきなり答えを出させるより複雑な判断の精度が上がることが知られています。LanguageMPC はこの CoT を運転の状況分析に使います。

LanguageMPC は、この「LLM の方針決め」と「MPC の制御」の境界面をうまく設計することで両者を接続します。

## 仕組みを詳しく
処理は「LLM が方針を決める → その方針を MPC のパラメータに翻訳する → MPC が制御量を計算する」の3段です。

第1段: シーンのテキスト化と LLM への問い合わせ。周囲の状況(交差点の形状、他車の位置と速度、信号、自車状態)を構造化されたテキストで記述し、LLM に「どう振る舞うべきか」を CoT で尋ねます。論文では判断を次の3段に分けます。

- 注目すべき相手の特定(identify vehicles to attend to): 周囲に多数の車がいても、いま重要なのはどの車か(=どの車との相互作用が判断を左右するか)を選ぶ。多数の他車を全部同等に扱うと判断が発散し、プロンプトも長大になるため、LLM の常識で「関係ある相手」に絞るのがポイント。
- 状況分析(situational reasoning): 選んだ相手に対して、譲るべきか、先に行くべきか、待つべきか、という関係性を判断する。
- 行動の決定(action guidance): 「減速する」「停止して待つ」「ゆっくり進む」などの離散的な(=有限の選択肢から選ぶ形の)高レベル行動を選ぶ。

第2段: LLM の言語的判断を MPC のパラメータへ変換。ここが設計の核心です。LLM の言葉の判断をそのまま座標にするのではなく、MPC が解釈できる「コスト関数の重み」や「制約の強さ」という数値に翻訳します。たとえば:
- 「あの車に道を譲る」→ その車との衝突回避制約を厳しくし(安全距離を広げ)、速度維持の重み w_track を下げる
- 「先に行ってよい」→ 速度維持の重み w_track を上げ、過度な減速を避ける

この翻訳テーブル(言語的判断とパラメータの対応関係)を人間が設計するのが LanguageMPC の前提であり、後述するボトルネックでもあります。論文では LLM の出力を、観測行列・重み行列という MPC の構成要素に対応づける形で、判断を数値パラメータへ落とし込みます。

第3段: MPC による低レベル制御。受け取った重み・制約の下で、MPC が運動モデルに基づき未来数秒の最適な操作(加速度、操舵角)を解きます。これにより最終的なペダル・ハンドルの動きは物理的に無理がなく滑らかになります。

数値例でイメージすると:
```
状況: 無信号交差点。右から車 X が接近 (速度 10 m/s)、左後方に車 Y。
LLM の CoT 判断:
  「重要なのは右の X。X は速く先に通過しそう → X に譲るべき。行動 = 減速して待つ」
パラメータ変換:
  X との安全距離制約: 5m → 10m に強化
  目標速度の重み w_track を下げ、安全項の重み w_safe を上げる
MPC 出力 (予測ホライズン 3s, 0.1s 刻み):
  滑らかな減速指令 a = -1.5 m/s² (急ブレーキでなく緩やかに)、操舵 ≈ 0
```
この分離のおかげで、LLM は「重要な相手は誰か」「譲るべきか」という人間的判断に集中でき、数値の安定性・物理的妥当性は MPC が保証します。LLM が多少おかしな方針を出しても、MPC の制約層が「物理的にあり得ない操作」や「衝突する軌道」は弾くため、純 LLM 運転より安全側に倒れやすいという副次的な利点もあります。階層の周期も非対称で、LLM は数百 ms〜秒オーダーで方針を更新し、MPC はその間 10〜100Hz で高速に制御を回す、という二層構成が想定されます。

## 手法の系譜と主要論文
この分野は「LLM をどの抽象度で運転に組み込むか」という軸で整理できます。

MPC 自体は制御工学の古典で、Mayne らのサーベイ(Mayne, Rawlings, Rao, Scokaert, Automatica 2000, "Constrained model predictive control: Stability and optimality")が安定性・最適性の理論的基盤を与え、自動運転の軌道追従・運動計画に広く使われてきました。CoT(Wei et al., NeurIPS 2022)が「LLM に段階的推論をさせると複雑タスクの精度が上がる」ことを示したのも、LanguageMPC の状況分析の土台です。

LLM を運転に直接使う流れは 2023 年に集中して現れました。GPT-Driver(Mao, Qian, Zhao, Wang, 2023, arXiv:2310.01415)は、運転を「言語モデリング問題」として定式化し、座標トークンを LLM に直接生成させる端的なアプローチを示しました。これは LLM の表現力を最大限使う一方、前述の「座標がガタつく/物理的に無理」という問題を抱えます。

LanguageMPC(Sha, Mu, Jiang, Chen, Xu, Luo, Li, Tomizuka, Zhan, Ding, 2023, arXiv:2310.03026, "LanguageMPC: Large Language Models as Decision Makers for Autonomous Driving")は、これと対照的に「LLM は高レベル決定者に徹し、低レベルは MPC に任せる」分離設計を提案しました。新規性は、LLM の定性的判断を MPC の行列パラメータ(コスト重み・制約)へ系統的にマッピングする橋渡し機構を初めて明確に定式化した点にあります。

同時期の知識駆動・エージェント系([DiLu](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0531_dilu_note) の記憶・反省、[Agent-Driver](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0526_agent-driver) のツール使用型エージェント)とは、LanguageMPC が「制御理論との接続」に焦点を当てた点で位置づけが異なります。LanguageMPC は「LLM × 古典制御のハイブリッド」という設計パターンの代表例として、後続の階層型運転エージェント研究に繰り返し参照されます。

## 論文の実験結果(定量データ)
LanguageMPC は、複雑な多車両インタラクションを含む運転シナリオ(無信号交差点、合流、複数車線でのインタラクションなど、idsim 系のシミュレータ環境)で評価されました。評価軸は主に次の3つです。

- 安全性(衝突率・事故の有無): 危険なインタラクション場面でどれだけ衝突を避けられたか。
- 効率性(タスク完了時間・平均速度): いたずらに止まりすぎず、滑らかに目的地へ到達できたか。
- 快適性・滑らかさ(加加速度 jerk = 加速度の変化率): 急なペダル操作がないか。

報告によれば、単純なルールベースの方針切り替えや、座標を直接出す純 LLM 方式と比べて、LanguageMPC は複雑なインタラクション場面での意思決定の質(適切に譲る/先行するの判断)が高く、ベースラインに対して衝突回数・違反回数を顕著に減らしつつ、MPC によって制御の滑らかさ(低 jerk)と物理的妥当性を維持できたとされます。とくに「注目すべき相手を LLM に絞らせる」工程を入れることで、周囲車両が多い場面での判断の安定性が改善したと報告されています。jerk(加加速度)は乗り心地の指標で、これが小さいほど乗員が感じる揺れが少なく、急操作による制御破綻のリスクも下がるため、自動運転では安全性と快適性の両面で重要な数値です。

アブレーション(構成要素を抜いて効果を測る分析)の観点では、(1) CoT による段階的推論を外すと判断の妥当性が落ちること、(2)「注目相手の特定」工程を省くと多車両場面で判断が不安定になること、(3) MPC 層を外して LLM に直接制御させると jerk が増え滑らかさが失われ衝突も増えること、が示唆されています。これらはそれぞれ「言語推論の段階化」「注意の絞り込み」「制御の分離」という3つの設計要素が、いずれも性能に寄与していることを意味します。なお、具体的な数値は環境設定や使用した基盤 LLM(GPT-3.5/GPT-4 など)に依存するため、絶対値より「ベースライン比でどの方向に改善したか」の傾向として理解するのが妥当です。

## メリット・トレードオフ・限界
メリット:
- LLM の状況理解と MPC の制御安定性を分業で両立できる。
- 最終制御が運動モデルに基づくため、滑らかで物理的に無理がない(低 jerk)。
- 複雑な多車両インタラクション(交差点・合流)での意思決定が向上する。
- LLM の判断が言葉で出るため「なぜその方針か」を説明しやすく、デバッグや安全監査がしやすい。
- MPC の制約層が物理的に不可能な操作を弾くため、LLM の誤りに対する安全マージンがある。

トレードオフ・限界(研究上の未解決課題):
- 言語的判断を MPC パラメータへ翻訳する対応表を人間が設計する必要があり、ここがスケールのボトルネック。状況の種類が増えるほどマッピングの保守が難しくなる。この翻訳を学習で自動獲得できないかは未解決課題。
- LLM 推論を毎判断で挟むため、車載リアルタイム制御(数十 ms 周期)には依然ハードルがある。LLM を低頻度(数秒に1回方針更新)で呼び、MPC を高頻度で回す階層構成が現実的だが、方針更新の遅れが急変場面で問題になりうる。
- MPC の運動モデルが不正確だと制御が破綻する。タイヤ滑りや路面変化など、モデル化しにくい現象に弱い(MPC 由来の前提)。
- 高レベル行動が離散的(選択肢から選ぶ)なので、連続的で微妙なニュアンスの方針(「ほんの少しだけ譲る」など)を表しにくい。
- LLM の判断が常に正しい保証はなく、幻覚(hallucination, 事実でない出力)が安全に直結する。

## 発展トピック・研究の最前線
LanguageMPC が示した「言語による高レベル決定 × 古典制御による低レベル実行」という階層分離は、その後の研究でさまざまに発展しています。一つの方向は、言語からパラメータへの翻訳を手書きの対応表でなく学習可能にする試みで、報酬・コスト関数自体を LLM に生成させる「reward/cost design by LLM」(ロボティクスでの Eureka 系の流れ)と合流しつつあります。もう一つは、MPC の代わりに学習済みの制御方策や微分可能な最適化層(differentiable MPC)を置き、LLM の出力から制御までを勾配で一貫して学習する方向です。

また、3D 空間理解を持つ VLM(画像も言葉も扱えるモデル)を上位に置く流れ([OmniDrive](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0542_omnidrive) など)や、経験を蓄積して方針を改善する知識駆動エージェント([DiLu](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0531_dilu_note))と組み合わせれば、「3D で状況を把握 → 経験を参照して方針決定 → MPC で安全に実行」というより完成度の高い階層が描けます。LLM を方針層、MPC を実行層とする二層分離は、安全保証(MPC の制約による下限保証)と柔軟性(LLM の常識)を両立させる枠組みとして、形式検証やシールド(safety filter)研究とも接点を持ち始めています。一方で、リアルタイム性・幻覚への対処・物理モデルの精度向上は、いずれも実車適用に向けた未解決の大きな課題として残っています。
