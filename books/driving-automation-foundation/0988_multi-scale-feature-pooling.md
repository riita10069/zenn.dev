# マルチスケール特徴のプーリング連結

## ひとことで言うと
最近の画像認識ネットワーク(backbone)は、解像度の違う複数段の特徴マップを出します。粗いもの(意味的)・細かいもの(空間的)を全部活かしたいので、いったん全段を同じ解像度に縮めて揃え、チャネル方向に重ねて1つの分厚い特徴にまとめる処理が「マルチスケール特徴のプーリング連結」です。揃える縮め方には AdaptiveMaxPool(適応的最大プーリング)のような、出力サイズを指定するプーリングをよく使います。複雑な統合(FPNなど)に比べ非常に単純で軽量な一方、細かい空間情報は強い縮小で失われます。

## 直感的な理解
地図を思い浮かべてください。縮尺の違う地図(都市全体の俯瞰図、街区の詳細図、建物の間取り図)を全部使いたいとき、そのままでは紙のサイズが違って重ねられません。全部を同じ大きさの紙にコピーし直し、重ねて透かして見れば、大局も細部も同時に把握できます。マルチスケール特徴のプーリング連結は、まさにこの「縮尺の違う特徴を同じサイズに揃えて重ねる」操作です。

なぜ複数の縮尺が要るのか。自動運転では、遠くの小さな信号機(細かい特徴が必要)も、近くの大きなトラック(大きな意味的特徴で十分)も、どちらも見落とせません。浅い段は解像度が高く細かいエッジや質感を捉え、深い段は解像度が低い代わりに「車らしい」「人らしい」といった大きな意味的特徴を捉えます。だから「細かい段も粗い段も両方使う」ことが望まれます。

## 基礎: 前提となる概念

ステージ(stage)とマルチスケール。現代の backbone(ResNet, Swin Transformer, ConvNeXt など)は「ステージ」と呼ばれる段を重ねた構造です。入力に近い浅いステージは解像度が高く(例 64x64)、深いステージは解像度が低い(例 8x8)。各ステージの出力をまとめて「マルチスケール特徴(multi-scale features)」と呼びます。スケール(scale)とは解像度・縮尺のことです。多くのライブラリでは「特徴のみ返すモード(features_only)」を指定すると、全ステージの特徴マップをリストで取得できます。ステージが進むごとに解像度は半分、チャネル数はおおむね倍になるのが典型です。

プーリング(pooling)。特徴マップを縮小する操作です。Max プーリングは窓の中の最大値を採り、「その領域で最も強く反応した特徴」を残します。Average プーリングは平均を採ります。プーリングには「局所的な位置ずれに頑健になる」「解像度を下げて計算を軽くする」効果があります。

連結(concatenation)。`torch.cat(..., dim=1)` のように、複数のテンソルをチャネル方向に積み重ねる操作です。積み重ねるには、チャネル以外の次元(バッチ・高さ・幅)が全部一致している必要があります。だから連結の前に解像度を揃えるのです。連結はパラメータを持たない(学習しない)操作で、単に並べるだけです。

## 仕組みを詳しく

典型的には2つの道具で実現します。解像度揃えのプーリングと、チャネル方向の連結です。

```python
self.pool = nn.AdaptiveMaxPool2d(image_feature_size)  # 出力解像度を固定
...
for i in range(len(features)):
    features[i] = self.pool(features[i])     # 全段を image_feature_size に縮める
fused = torch.cat(features, dim=1)           # チャネル方向(dim=1)に連結
fused = self.channel_proj(fused)             # 1440 -> 256 へ射影(別トピック)
```

AdaptiveMaxPool2d について。普通のプーリングは「2x2 ごとに最大値を取る」のように縮小率(カーネルサイズ)を指定しますが、AdaptiveMaxPool は逆に「出力サイズ」を指定します。`AdaptiveMaxPool2d(8)` は「入力が何であろうと出力を 8x8 にしろ」という意味で、入力サイズに応じて窓の大きさを自動調整します(adaptive = 適応的)。これにより入力解像度が段ごとにバラバラでも、出力を共通の 8x8 に揃えられます。

各段の入力解像度はバラバラです(数値は説明用の例)。

```
ステージ0 : (B, 192, 64, 64)   ← 浅い、高解像度、細かい特徴
ステージ1 : (B, 384, 32, 32)
ステージ2 : (B, 768, 16, 16)
ステージ3 : (B, 768,  8,  8)   ← 深い、低解像度、意味的特徴
```

AdaptiveMaxPool2d(8) を通すと全段の高さ・幅が 8x8 に揃います。

```
ステージ0 : (B, 192, 64, 64) -> (B, 192, 8, 8)
ステージ1 : (B, 384, 32, 32) -> (B, 384, 8, 8)
ステージ2 : (B, 768, 16, 16) -> (B, 768, 8, 8)
ステージ3 : (B, 768,  8,  8) -> (B, 768, 8, 8)
```

次に `torch.cat(features, dim=1)`。`dim=1`(チャネル次元)に沿って重ねます。高さ・幅が同じなので、チャネル数だけが足し算されます。

```
(B, 192, 8, 8)
(B, 384, 8, 8)
(B, 768, 8, 8)   をチャネル方向に積む
(B, 768, 8, 8)
--------------------------------
(B, 192+384+768+768 = 2112, 8, 8)   ← 例。構成によってチャネル合計は変わる(1440 など)
```

こうして「全段の情報を含んだ分厚い特徴マップ」が1つできます。チャネル合計が大きく半端な数になるので、この後 1x1 Conv で固定次元(256)に落とします([Channel Projection（1x1 Conv によるチャネル次元射影）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0947_channel-projection-layer))。

なぜ Max プーリングで縮めるのか。最も単純で軽く、「強く反応した特徴を残す」性質が、物体検出のように「目立つものを見つける」タスクと相性が良いためです。Average なら平均化で目立つ反応が薄まりますが、Max は最も強い反応を保ちます。一方、Average プーリングは領域全体の文脈(平均的な明るさ・密度など)を保つので、シーン全体の意味を捉えたい場合は Average が向くこともあり、選択はタスク依存です。なお、この縮小先の解像度は、後段で BEV(鳥瞰図)グリッドへ再投影する場合のグリッドサイズとは独立に扱われます。特徴の中間解像度と最終的な出力グリッドは別物です。

## 手法の系譜と主要論文

マルチスケール特徴を扱う研究の系譜です。

プーリングの起源。Max プーリングは LeCun らの初期 CNN や AlexNet(Krizhevsky et al., NeurIPS 2012)で広く使われ、「局所的な位置ずれに頑健にしつつ解像度を下げる」効果が確立しました。

Spatial Pyramid Pooling(He et al., arXiv:1406.4729, 2014)は、可変サイズの入力から固定長の特徴を作るために、複数の解像度でプーリングして連結する SPP 層を提案しました。Adaptive Pooling の発想に直結します。「入力解像度に依存せず固定サイズ出力を作る」点が、本稿のプーリング連結と同じ動機です。

SSD(Liu et al., ECCV 2016, arXiv:1512.02325)は、複数解像度の特徴マップそれぞれで独立に検出を行う「マルチスケール検出」を提案しました。浅い段で小物体、深い段で大物体を担当させる発想です。

FPN(Feature Pyramid Network, Lin et al., CVPR 2017, arXiv:1612.03144)は、マルチスケール統合の決定版です。「浅い高解像度の段と深い低解像度の段を、上から下へ情報を流しながら(top-down pathway + lateral connection)統合する」ピラミッド構造を提案しました。狙いは大小さまざまな物体を1つのネットワークで検出すること、効果は小物体の検出精度向上、トレードオフは段ごとの結合に追加の畳み込みが必要で計算とメモリが増えることです。本稿の「縮めて連結する」方式は FPN ほど凝った結合はせず、はるかに単純な代わりに、各段の細かい空間情報は強い縮小で大きく失われます。FPN が「解像度を保ったまま意味を伝播」するのに対し、プーリング連結は「解像度を犠牲にして全段を1つに圧縮」する、という設計思想の違いがあります。

PANet(Liu et al., CVPR 2018, arXiv:1803.01534)と BiFPN(Tan et al., EfficientDet, CVPR 2020, arXiv:1911.09070)。FPN に bottom-up 経路を足したり(PANet)、双方向のスケール間結合を学習可能な重み付きで繰り返したり(BiFPN)して、「どの段をどれだけ重視するか」を学習で決める方向に発展しました。単純連結が捨てた「学習可能な重み付け」を取り戻す系譜です。

## 論文の実験結果(定量データ)

FPN の効果が、マルチスケール統合の価値を最もよく定量化しています。指標から説明します。COCO(物体検出の標準ベンチマーク、約12万枚・80カテゴリ)の AP(Average Precision、検出精度。0-100で高いほど良い)と、特に小物体に対する AP_small が重要です。

FPN を Faster R-CNN に組み込むと、COCO 上で box AP が、FPNなしのベースライン(単一スケール特徴)に対しておよそ 8 ポイント前後改善したと報告されています(ResNet backbone で AP 約36前後)。特に小物体の AP_small が大きく伸び、マルチスケール統合が小物体検出に効くことが明確に示されました。これは「浅い高解像度の段を活かすと小さい対象が見える」という直感の定量的裏付けです。アブレーションでは、top-down pathway や lateral connection を外すと AP が低下し、各構成要素が寄与していることが示されています。

SSD は、複数解像度の特徴マップを使うことで、当時の単一スケール検出器より高速かつ高精度を達成し、PASCAL VOC 2007(検出ベンチマーク)で mAP 約74-77%(入力解像度による)を、リアルタイムに近い速度(数十FPS)で記録しました。マルチスケール特徴を「使うこと自体」が精度に効くことを示した初期の例です。

BiFPN の効果。EfficientDet は BiFPN による重み付き双方向統合と複合スケーリングで、COCO 上で従来の検出器より少ない計算量(FLOPs)で高い AP を達成し、D7 構成で当時の最高精度(box AP 約53)を、はるかに少ないパラメータで記録しました。「統合の仕方を学習可能にする」ことの価値を定量化した例です。

プーリング連結という単純方式の限界も定量的に語れます。FPN のような丁寧な統合と比べ、強い縮小(例 64x64 -> 8x8)で空間情報を捨てる単純連結は、小物体検出や精密な位置決めのタスクで FPN に劣ることが一般に知られています。一方で、計算量とメモリははるかに小さく、追加の畳み込み・アップサンプリングが不要で、追加パラメータがほぼゼロです。タスクが「精密な位置」より「シーン全体の意味の集約」を求める場合(自車の進路を1本決める計画タスクなど)は、この単純方式でも実用上十分なことがあります。

## メリット・トレードオフ・限界

メリット。第一に、浅い段(細かい特徴)と深い段(意味的特徴)の両方を1つの特徴にまとめられる。第二に、AdaptiveMaxPool は入力解像度に依存しないので、画像サイズや backbone が多少変わっても固定の出力解像度を作れる。第三に、実装が pool と連結だけで非常に単純・軽量で、追加パラメータがほぼゼロ。

トレードオフと限界。第一に、高解像度の段を強く縮める(例 64x64 -> 8x8)ので、浅い段の細かい空間情報が大きく失われる。FPN のように解像度を保ったまま意味を伝播する丁寧さがない。第二に、単純連結なので「どの段を重視するか」をネットワークが明示的に学ぶ仕組みを持たない(連結後の 1x1 Conv に暗黙的に委ねる)。第三に、Max プーリングは最大値以外を捨てるため、平均的な文脈情報が失われうる。第四に、全段を同じ解像度に潰すため、本来は段ごとに最適な解像度が違うという情報を活かせません。

研究上の未解決課題として、「どの程度の解像度まで縮めてよいか」「Max と Average のどちらが、どのタスクで良いか」はタスク依存で、理論的な指針が確立していません。また、単純連結 vs FPN vs Attention ベースの統合(後述)のどれが最良かは、精度と計算予算のトレードオフで決まり、一律の答えはありません。

## 発展トピック・研究の最前線

Attention ベースの統合。Transformer 系では、マルチスケール特徴を Deformable Attention(Deformable DETR, Zhu et al., ICLR 2021)などで統合する手法が主流になりつつあります。固定のプーリングではなく、「どの位置のどのスケールを見るか」をデータから学習する点が新しい。Deformable Attention は全画素を見る代わりに少数のサンプリング点だけを参照するため、マルチスケールでも計算量を抑えられます。

自動運転の文脈では、マルチスケールの画像特徴を BEV(鳥瞰図)空間へ持ち上げる Lift-Splat-Shoot(Philion & Fidler, ECCV 2020)や、Deformable Attention で BEV を構成する BEVFormer(Li et al., ECCV 2022)が、本稿の「プーリングして潰す」より洗練されたマルチスケール集約を実現しています。これらは深度分布の推定や3D空間への投影を伴い、平面的な連結より幾何的に正確な集約を行います。単純なプーリング連結はこれらの軽量な代替・前段として位置づけられます。

NAS による統合構造探索。NAS-FPN(Ghiasi et al., CVPR 2019)は、マルチスケール特徴の結合パターン自体を自動探索し、人手で設計した FPN を上回る結合トポロジを発見しました。「縮めて連結する」固定設計から、「どう結合するかを機械が決める」方向への発展です。
