# BEVFormer 風 時系列セルフアテンション

## ひとことで言うと
過去数フレーム分の「シーンを要約したベクトル」を並べたキュー（待ち行列）に対して self-attention（自分自身の中で重要な部分を選んで重み付けする仕組み）をかけ、運転判断に必要な過去情報を1本にまとめる時系列メモリの方式です。自動運転の代表的手法 BEVFormer が広めた「過去を attention で融合する」というアイデアを核としています。再帰型（RNN/GRU）が苦手とする「遠い過去を忘れずに、必要なフレームへ直接アクセスする」ことを、attention の仕組みで実現するのが狙いです。

## 直感的な理解
過去5秒間の運転映像があるとします。いまブレーキを踏むべきか判断するとき、その5秒すべてを均等に重視するでしょうか。普通はしません。「2秒前に前の車のブレーキランプが光った瞬間」のような、決定的なフレームに自然と注目するはずです。残りのフレームは背景として軽く扱います。

self-attention はまさにこの「どのフレームのどの部分に注目すべきか」を、データから学習して決める仕組みです。再帰型のメモリ（GRU など）は過去を1ステップずつ順番に畳み込んでいくので、遠い過去ほど情報が薄れて「忘れて」しまいます。これに対し attention 型は、全フレームを横一列に並べて一望し、必要なフレームへ距離に関係なく直接アクセスできます。「2秒前のあの瞬間」を、間に挟まる無関係な49フレームに邪魔されずに参照できる、というのが本質的な強みです。経路（パス）の長さで言えば、再帰型では遠い過去への経路が系列長に比例して長くなり勾配が消えやすいのに対し、attention では任意の2フレーム間の経路長が常に1で、勾配が届きやすいという理論的な裏付けがあります。

## 基礎: 前提となる概念
まず self-attention を噛み砕きます。入力が「ベクトルの列」だとします。self-attention は、列の各要素について三つの役割のベクトルを作ります。クエリ（query、「私は何を探しているか」）、キー（key、「私は何を提供できるか」）、バリュー（value、「私が持っている中身」）です。ある要素のクエリと、すべての要素のキーとの内積をとり、関連度（attention 重み）を計算します。関連の強い要素のバリューを多めに、弱い要素を少なめに足し合わせて、その要素の新しい表現を作ります。「自分（self）の列の中で互いに関連度を計算する」から self-attention と呼びます。Transformer の中核技術です。式で書けば `Attention(Q,K,V) = softmax(QKᵀ/√d) V` で、`√d` で割るのは内積が次元数 d とともに大きくなりすぎて softmax が飽和するのを防ぐためのスケーリングです。

次に重要な性質として、self-attention は順序を区別できません。入力列を「集合」として扱うため、フレームを並べ替えても出力が変わりません（置換同変性）。しかし時系列では「古い→新しい」の順序こそが速度や加速の手がかりです。そこで各要素に「何ステップ目か」を表す位置エンコーディングを足して、順序を明示的に教えます（詳細は [学習可能クエリトークンと正弦波位置エンコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1043_learnable-query-token-sinusoidal-pos)）。

もう一つ、self-attention は出力が入力と同じ本数になります。T本のフレームを入れれば T本の出力が返ります。しかし時系列メモリが下流に渡したいのは時間をたたんだ1本の文脈ベクトルなので、T本を1本にまとめる仕掛け（学習可能なクエリトークン）が要ります。これも [学習可能クエリトークンと正弦波位置エンコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1043_learnable-query-token-sinusoidal-pos) で扱います。

時系列メモリのインターフェースは、入力 `[B, T, D]`（B=バッチ、T=時間ステップ数、D=特徴次元）から出力 `[B, D]`（時間をたたんだ要約）への変換です。下流のプランナはこの `[B, D]` だけを受け取るので、内部がどんな方式でも下流は同じです。

## 仕組みを詳しく
本家 BEVFormer は「BEV グリッド（鳥瞰図を格子状に区切った2次元の特徴マップ。たとえば 200×200 のセル）」上で時系列 attention を行います。具体的には、現在フレームの各 BEV セルが、直前フレームの BEV 特徴のうち「自車の動きで補正した対応位置」を参照する Temporal Self-Attention を使います。自車が前に進んだ分だけ過去の BEV を平行移動・回転させて位置合わせし、同じ実世界の場所どうしを比べられるようにするのが肝です。さらに計算を抑えるため、全セル総当たりではなく Deformable Attention（各クエリが少数の学習された参照点だけを見る疎な attention）を使います。

一方、シーンを時刻ごとに1本のベクトルへ要約した `[B, T, D]` という列を扱う軽量な構成では、空間グリッドではなくベクトル列に対して同じ思想を適用します。直近 `queue_len` ステップ分の履歴ベクトルに self-attention をかける Transformer エンコーダ、という形です。

処理の流れを、形を追いながら分解します。視覚特徴 `[B, T, 896]` と自車運動特徴 `[B, T, 256]` の二つのストリームがあるとします。

1. パススルー判定: 入力が2次元（時間軸なし `[B, D]`）なら、何もせずそのまま返す。NoMemory と同じフォールバックで、履歴が無い状況でも壊れないようにする。
2. 二つのストリームを連結: 視覚と自車運動を特徴次元方向で結合し `joint = [B, T, 896+256] = [B, T, 1152]`。
3. キュー長で切り詰め（任意）: `queue_len` が指定されていれば `joint[:, -queue_len:, :]` で直近 `queue_len` ステップだけ残す。履歴は古い→新しいの順なので末尾を取る。これが BEVFormer の固定長キューに対応する（詳細は [固定長履歴キュー(queue_len)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1037_fixed-length-history-queue)）。
4. 入力射影: 線形層で `[B, T, 1152]` を attention 用の幅 `d_model`（たとえば512）へ写像し `[B, T, 512]`。
5. 位置エンコーディング加算: 順序を区別させるため、sinusoidal positional encoding で「何ステップ目か」を足す。
6. クエリトークンを先頭に連結: 学習可能な `query_token`（`[1,1,512]`）をバッチ分に複製して列の先頭に付け `[B, T+1, 512]`。このトークンの出力が「キュー全体の要約」になる。
7. Transformer エンコーダ: 複数層（たとえば2層）の self-attention をかけ `[B, T+1, 512]`。
8. クエリ位置を取り出して逆射影: 先頭（`x[:, 0]`）を線形層で元の `1152` 次元に戻し、視覚896次元と自車256次元に切り分けて `(visual_context [B,896], egomotion_context [B,256])` を返す。

ASCII 図で要点を示します。

```
履歴  [ f0 f1 ... f63 ]   → 切り詰め → 射影+位置PE → [Q f? f? ... f?]
                                                       │
                                          self-attention 複数層
                                                       │
                                                先頭 Q だけ取り出す → [B, D]
```

出力は時間軸 `T` が消えた `[B, D]` で、NoMemory とまったく同じ契約です。だから下流のプランナは変更不要で、構成キーを差し替えるだけでこの方式に切り替えられます。このベクトル列版は、空間グリッド上の attention を行う本家 BEVFormer よりはるかに軽量です。なぜなら本家は 200×200=4万セル × 過去フレーム数のスケールで attention するのに対し、ベクトル列版は1時刻=1トークンなので系列長がせいぜい数十だからです。計測例では、ベクトル列版のパラメータ数が GRU 圧縮型のおよそ3分の1、遅延は同程度と報告されることがあります。

## 手法の系譜と主要論文
- Vaswani et al. (NeurIPS 2017, "Attention Is All You Need", arXiv:1706.03762): self-attention と sinusoidal positional encoding の原典。順序情報を attention 自体は持てないため位置エンコーディングが必須、という設計もここが出典です。時系列 attention のすべての土台です。任意の2要素間を1ホップで結ぶ「経路長 O(1)」が、長距離依存の学習で再帰型に勝る理由として示されました。

- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270): 過去フレームの BEV 特徴を時系列キューとして保持し、現在フレームの各 BEV 位置が過去の対応位置を temporal self-attention で参照する Temporal Self-Attention を提案しました。動機は「1枚では速度が測れず遮蔽に弱いから」。効果は速度推定の改善と遮蔽下での頑健性向上。トレードオフは過去 BEV を保持・位置合わせ・attention する分の計算とメモリ増、そしてキューを長くするほどコストが増える点です。ベクトル列版はこのうち「過去をキューに溜めて attention で融合する」中核思想を簡約して採用したものです。

- SOLOFusion (Park et al., ICLR 2023, arXiv:2210.02443): 「履歴は長いほど効くが計算が増える」というトレードオフを明示し、短期高解像度＋長期低解像度の併用で折り合いをつけました。`queue_len`（どれだけ過去を見るか）の設計判断に直結する知見です。

- Sparse4D (Lin et al., 2022, arXiv:2211.10581): 3次元の疎な anchor を時間方向にも展開し、複数フレーム・複数視点の特徴を anchor 単位で集約する手法です。BEV グリッド全体ではなく物体周辺だけを見る疎な時系列融合の系譜で、効率と精度の両立を進めました。

- StreamPETR (Wang et al., ICCV 2023, arXiv:2303.11926): 過去の物体クエリを次フレームへ伝播させる「物体中心の」時系列融合を提案しました。BEV グリッド全体を毎回 attention するより安く長期履歴を扱える点が貢献で、トレードオフは物体クエリの設計依存性です。「最近フレームのキューに attention する」考え方と近縁で、効率の良い後継系統です。

系譜を一言でまとめると、Transformer（2017）→ BEV 空間での時系列 attention（BEVFormer、2022）→ 物体／クエリ単位の効率的伝播（Sparse4D・StreamPETR、2022–2023）と進み、軽量なベクトル列版はこの流れの核を最小構成で取り込んだもの、と位置づけられます。

## 論文の実験結果（定量データ）
評価は主に nuScenes データセット（6カメラ＋LiDAR＋レーダの大規模実走行データ）で行われ、総合指標 NDS（nuScenes Detection Score、各種誤差をまとめた総合点。高いほど良い）と mAP（mean Average Precision、検出精度。高いほど良い）、速度誤差 mAVE（mean Average Velocity Error、m/s。低いほど良い）が報告されます。

BEVFormer は nuScenes test で NDS 0.569、mAP 0.481 を報告しました。論文のアブレーションで Temporal Self-Attention を外すと、NDS が数ポイント低下し、特に速度推定（mAVE）と遮蔽下・低速領域の検出が悪化することを示しています。これは「過去を attention で融合する」ことが速度・遮蔽に効くという直接の証拠です。具体的には、temporal を入れることで mAVE が大きく下がり、見えにくい物体の再現率（recall）が上がることが報告されています。StreamPETR は BEVFormer より少ない計算量（高いフレームレート）で同等以上の NDS（0.55〜0.6 級）を達成し、attention 型時系列融合が効率面でも改善し続けていることを示しました。SOLOFusion は履歴長を伸ばすほど NDS が単調に上がり、長期履歴版で当時の最高水準に達したと報告しています。

これらの数値が示すのは、attention 型時系列融合が「速度推定の正確さ」と「総合検出精度」を、単フレームベースライン比で着実に押し上げる、という点です。なぜ重要かというと、速度推定の精度は前方車の追従や衝突回避の信頼性に直結するからです。なお、ベクトル列へ簡約した軽量版については、3D 検出ベンチマークのような確立した数値が出そろっているわけではなく、遅延・パラメータ数といった計算特性の比較が先行し、軌道予測精度（ADE/FDE など。それぞれ予測軌道と正解軌道の平均誤差・終点誤差）への寄与は学習済みモデルでの評価を要する段階にあります。

## メリット・トレードオフ・限界
メリット

- 全履歴フレームへ直接アクセスでき、どのフレームが重要かを学習で選べる。再帰型より遠い過去を忘れにくい（経路長 O(1)）。
- 順序を位置エンコーディングで明示するため、時間の前後関係（速度・加速の向き）を考慮できる。
- 並列計算しやすく、軽量なベクトル列版では GRU 圧縮型と同程度の遅延でより少ないパラメータという計測例がある。
- 共通インターフェースに準拠するため、下流を変えずに差し替え可能。

トレードオフ・限界

- self-attention は系列長 T の2乗に比例して計算が増えるため、`queue_len` を長くするとコストが急増する。
- 学習可能パラメータが増え（単フレームベースラインは0）、十分なデータが無いと過学習しやすい。
- ベクトル列への簡約版は、本家 BEVFormer の「BEV 空間グリッド上の attention」が持つ空間的な位置合わせ（自車運動補正つきの場所どうしの対応）の恩恵を受けていない。シーンを1本のベクトルに潰す段階で空間情報が失われている可能性がある。
- 運転性能（軌道精度・閉ループ安定性）への寄与は、計算特性の比較だけでは確認できず、end-to-end 評価が必要。

研究上の未解決課題として、「ベクトルに潰した時系列要約が、空間グリッド上の時系列融合とどこまで等価か」という問いがあります。空間情報を保ったまま安価に時系列融合する設計（StreamPETR / Sparse4D 系の物体クエリ伝播など）が、この簡約版の上位互換になり得ます。

## 発展トピック・研究の最前線
attention 型時系列融合の最前線は「効率化」と「長期化」の二方向です。効率化では、全フレーム×全位置の総当たり attention をやめ、物体やトークン単位で過去を伝播させる手法（StreamPETR、Sparse4D 系）が主流になりつつあります。これは attention の2乗コストを回避しつつ長期履歴を扱う狙いです。長期化では、SOLOFusion のように短期と長期を解像度で分けて持つ多時間スケール設計が研究されています。さらに近年は、線形 attention や状態空間モデル（Mamba 系）で系列長に対し線形コストにする試みも進んでいます。

もう一つの流れは、エンドツーエンド運転（UniAD、VAD など知覚から計画まで一気通貫で学習する系統）への統合です。ここでは時系列メモリが軌道予測そのものの精度を左右するため、attention 型メモリの設計が計画性能に直結します。ベクトル列版の軽量メモリは、こうした計画寄りのパイプラインに組み込みやすいという実用上の利点があります。
