# 履歴ウィンドウと有効サンプル数

## ひとことで言うと
自動運転モデルに「過去の連続した状況」を見せて未来の運転を予測させたいのですが、過去 N フレームを使うルールだと、録画時間が短いシーンほど「学習に使える有効サンプル」が一気に減ってしまいます。さらに、未来 M フレームを正解として使うなら両端がさらに削られます。このノートでは、なぜそうなるのか、どのくらい減るのか、短すぎるシーンをどう扱うか(フィルタリング)、そして「履歴は長ければよいわけではない」という学習上の落とし穴を、具体的な数値で説明します。

## 直感的な理解
自動運転の End-to-End(E2E)モデルは、「いまこの瞬間に、これからどう走るか」を出力します。ただし「いまの 1 枚の画像」だけ見て判断するのは危険です。前の車がさっきからブレーキを踏み続けているのか、たった今踏んだのかで、自分が取るべき行動は変わります。だから過去の連続した状況(履歴)をまとめて入力します。

ここで一つの動画(シーン、scene、clip とも呼ぶ)には始まりと終わりがある、という当たり前の事実が効いてきます。動画の最初のほうのフレームには「その前の N フレーム」が存在しません。動画の最後のほうのフレームには「その後の M フレーム」が存在しません。つまり両端のフレームは学習サンプルとして使えず、シーンが短いほどこの「使えない両端」の割合が大きくなります。極端に短いシーンは、有効サンプルがゼロになって丸ごと無駄になることすらあります。

## 基礎: 前提となる概念
言葉を噛み砕いておきます。

フレーム(frame)とは、一定間隔で記録された 1 時点ぶんのデータです。10Hz(1 秒 10 フレーム)のデータなら、1 フレームは 0.1 秒に対応します。

履歴ウィンドウ(history window)とは、現在時刻からさかのぼって何フレーム分を入力に使うか、という長さです。たとえば過去 64 フレーム(10Hz なら 6.4 秒分)を履歴に使う、という設計です。

予測ホライズン(prediction horizon)とは、未来の何フレーム分を「正解の運転軌跡(trajectory)」として学習に使うか、です。たとえば未来 64 フレーム(6.4 秒分)を正解にする、という設計です。

ダウンサンプリング(downsampling、間引き)とは、高い記録レート(たとえば 100Hz)を低いレート(10Hz)に減らすことです。10 フレームに 1 つだけ残す、といった処理で、これをすると総フレーム数が減るので有効サンプル数の計算にも効きます。

模倣学習(imitation learning)とは、人間の運転というお手本をまねるようにモデルを学習させる枠組みです。中でも behavior cloning(行動クローニング)は、状態を入力・人間の行動を正解とする教師あり学習で、E2E 運転の基本的な学習法です。

## 仕組みを詳しく
履歴 N フレーム・未来 M フレームを使うとき、1 本のシーンから何個の学習サンプルが取れるかを計算します。

### 「有効サンプル」とは何か
1 本のシーンの中で「現在時刻」として選べるフレームを「サンプル点」と呼びます。あるサンプル点が学習に使えるための条件は二つです。

- そのフレームの前に、履歴ぶんの N フレームがそろっている
- そのフレームの後に、未来予測ぶんの M フレームがそろっている

ダウンサンプル後のフレーム列の長さを `L` とすると、有効なサンプル点の範囲は次のようになります(履歴 N、未来 M、現在フレーム自体は未来に含めないとする)。

```
有効サンプルの最初のindex = N
有効サンプルの最後のindex = L - M - 1
有効サンプル数            = (L - M - 1) - N + 1 = L - N - M
```

具体例として N = M = 64 とすると、必要な最小フレーム数は `N + M + 1 = 129` で、`L < 129` のシーンは有効サンプルが 1 個も作れず丸ごと捨てられます。有効サンプル数は `L - 128`(= L - N - M、上式で N=M=64)になります。

### 数値例(なぜ短いシーンで激減するか)
シーン長が 10〜22 秒、10Hz でダウンサンプルなしと仮定して、N = M = 64 のときの有効サンプル数を計算します。

| シーン長 | フレーム数 L | 有効サンプル数 = L − 128 |
|---|---|---|
| 10秒 | 100 | 0(129未満なので丸ごと捨てられる) |
| 13秒 | 130 | 2 |
| 15秒 | 150 | 22 |
| 19秒 | 193 | 65 |
| 22秒 | 220 | 92 |

ここで重要なのは「線形に減る」のではなく「閾値(129 フレーム)を境にゼロに落ちる」点です。13 秒のシーンは 2 サンプルしか生み出さないのに、22 秒のシーンは 92 サンプルを生み出します。シーン長がほんの少し違うだけで、学習への貢献が桁違いになります。

```
有効サンプル数
  ^
92|                                    ___---
  |                              ___---
22|                  ___---
 2|        _-
 0|________|________________________________ シーン長
       129フレーム(=12.9秒)
```

### before / after で見るフィルタリング
データセット構築時に、シーンごとに有効サンプルを列挙し、`L < N + M + 1` のシーンは警告を出して空リストを返す(=スキップ)のが定石です。

- before: 全シーンを無条件に入れる。短いシーンが「サンプル 0 個」で混ざり、データセット構築時に無駄な I/O が発生し、最悪 `__getitem__`(添字アクセス)で範囲外参照のバグになる。
- after: 構築時に短いシーンを除外し、残ったシーンの有効サンプル `(scene_id, sample_idx, timestamp)` を平らなリストに展開しておく。添字アクセスはリストを引くだけで、毎回の範囲計算が不要になり高速かつ安全。

この「有効サンプルを事前に平坦化しておく」設計は、PyTorch の `Dataset` を実装する際の一般的なパターンで、`__getitem__` を純粋な I/O に保ち学習を速くします。

### ダウンサンプリングとの相互作用
記録 100Hz を 10Hz にダウンサンプル(10 フレームに 1 つ残す)すると、`L` が 1/10 になります。すると同じ N・M でも、より長い実時間のシーンが必要になります。たとえば N=M=64 を 10Hz で使うなら 12.9 秒以上のシーンが必要ですが、これは元の 100Hz 換算では 129 秒ではなく 12.9 秒(間引き後のフレームで数える)です。Hz 設定と履歴長はセットで決める必要があります。

## 手法の系譜と主要論文
履歴ウィンドウの長さは、単なるデータ整形の都合ではなく、模倣学習の根本的な落とし穴と結びついています。

Bojarski et al.(NVIDIA, "End to End Learning for Self-Driving Cars", 2016, arXiv:1604.07316)は、単一フレームの前方画像から操舵角を直接回帰する E2E の原型を示しました。ここでは履歴を使わず、多様で大量のサンプルを集めることが性能の鍵だと示されました。短いシーンを混ぜると有効サンプルが偏る、という問題はこの「データ量・多様性が命」という前提から生じます。

Codevilla et al.("Exploring the Limitations of Behavior Cloning for Autonomous Driving", ICCV 2019, arXiv:1904.08980)は、模倣学習で過去情報や自車速度を多く与えると、モデルが「直前の自分の速度をそのままコピーすれば損失が下がる」ことを学んでしまい、信号や歩行者を見なくなる「慣性問題(inertia problem)」を報告しました。赤信号で止まったあと青になっても動き出さない、という失敗が典型例です。履歴は長ければよいわけではない、という重要な指摘です。

de Haan et al.("Causal Confusion in Imitation Learning", NeurIPS 2019, arXiv:1905.11979)は、この現象を「因果混同(causal confusion)」として一般化しました。入力情報を増やすほど、モデルは「真の原因(前方の障害物)」ではなく「相関するショートカット(過去の自分の行動)」を頼りにしやすくなり、しかも訓練時の損失はむしろ下がるため見逃されやすい、という逆説を理論と実験で示しました。履歴ウィンドウを伸ばすことは、まさにこのショートカットの材料を増やす行為です。

近年の BEV 系 E2E(例: UniAD, Hu et al., CVPR 2023, arXiv:2212.10156)では、数フレーム〜十数フレームの短い時系列を BEV 特徴上で集約します。長すぎる履歴はメモリと計算量を急増させ、古い情報の有用性も下がるため、実用上は数秒オーダーに収めるのが一般的です。6.4 秒(64 フレーム)という設定は、文脈を取りつつ計算量と因果混同のリスクを抑えた中間的な選択といえます。

## 論文の実験結果(定量データ)
履歴の扱いがどれだけ性能に効くかは、模倣学習の研究で定量化されています。

Codevilla et al. は CARLA(運転シミュレータ)の NoCrash ベンチマークで、自車速度を入力に含めるモデルが、信号や停止状況で「動き出さない」失敗を起こし、成功率(走破できたエピソードの割合、高いほど良い)を下げることを示しました。指標は走破成功率で、過去・自車状態を素朴に与えた設定では、与えない設定に比べて特定の状況(停止からの発進)で成功率が顕著に低下しました。とりわけ「inertia problem(慣性問題)」のシナリオでは、速度入力ありのモデルが赤信号後の発進に失敗して成功率がほぼ底を打つ一方、速度入力を抜くと同じシナリオの成功率が大きく回復することが示され、文脈情報の追加が二桁ポイント規模で性能を悪化させうることを定量化しました。彼らはまた、同じデータ・同じアーキテクチャでも乱数シード(初期化の違い)だけで最終成功率が大きくばらつくことを報告し、behavior cloning の評価が見かけより不安定であることも明らかにしました。これは「文脈を増やす」ことが必ずしも性能を上げないことの直接的な証拠です。

de Haan et al. は、入力に過去の行動を含めた模倣学習エージェントが、訓練損失(予測誤差)は低いのに実環境での報酬(タスク達成度)が低い、という乖離を複数環境で示しました。彼らの提案手法(どの入力が真の原因かを介入実験で特定する)を使うと、因果混同を起こしたベースラインに対して、必要な環境とのやり取り回数を桁で削減して正しい方策に到達できると報告しています。指標は「正しい方策に到達するまでの介入回数」で、少ないほど良く、因果混同を放置すると探索コストが爆発することを示しています。

データ量の観点では、E2E 運転は一般に「サンプル数が多いほど汎化する」という経験則が成り立ちますが、サンプルがシーン長で偏ると、特定のシーン長に学習が引っ張られます。フィルタリングで「サンプル 0 個」のシーンを除けば計算の無駄は減りますが、短いシーンを完全に捨てると、急な割り込みのようなレアで重要な状況が失われ、汎化に悪影響が出る可能性があります。これは「I/O 効率」と「データ多様性」のトレードオフとして残る、未解決の実務的課題です。

## メリット・トレードオフ・限界
メリットとしては、長い履歴(数秒)で前後の文脈をモデルに与えられること、フィルタリングで「サンプル 0 個」の無駄なシーンを除けてデータセット構築とアクセスが安全・高速になること、有効サンプルを事前に平坦化することで添字アクセスが軽くなり学習が速くなることが挙げられます。

トレードオフと限界は複数あります。履歴が長いほど短いシーンが捨てられ、データセット全体のサンプル数が減ります。特に閾値付近では急減します。短いシーンに固有のレアな状況が捨てられ、多様性が下がる可能性があります。履歴を長くしすぎると、論文が指摘する「過去の自己状態への過依存(causal confusion / inertia)」のリスクが上がり、訓練損失が下がっているのに実走性能が悪化するという、最も気づきにくい失敗を招きます。同じ秒数でもダウンサンプル率次第で必要フレーム数が変わるため、Hz 設定と履歴長はセットで決める必要があります。

研究上の未解決課題として、「どのくらいの履歴が最適か」はタスク・データセット依存で、原理的な決定法がありません。また因果混同を避けつつ時系列文脈を活かす入力設計(過去の行動を直接入れず、観測の変化だけを与えるなど)は研究が続いていますが、決定版はありません。

## 発展トピック・研究の最前線
因果混同への対策として、データ拡張(過去の行動をノイズで撹乱して相関を断つ)、入力選択(介入実験で真の原因入力を特定する de Haan らの方向)、補助タスク(未来の観測を予測させて「世界を見る」よう仕向ける)などが研究されています。

時系列の扱いそのものも進化しており、生のフレーム列を直接 Transformer に入れるのではなく、BEV 特徴空間で時間方向に集約する(UniAD, VAD の流れ)ことでメモリ効率と性能を両立する設計が主流になっています。UniAD は過去数フレームの BEV 特徴を再帰的に融合する設計で、nuScenes のプランニングで先行の単一フレーム手法より平均 L2 誤差・衝突率を下げており、「適度な時系列文脈は効くが、それは生フレームの素朴な連結ではなく圧縮された特徴上での集約による」ことを示しています。さらに、world model(世界モデル、未来の観測を生成的に予測するモデル)を使って履歴から未来を想像させ、それをプランニングに活かす研究(GAIA-1, MILE など)も活発で、「履歴をどう圧縮して未来予測に使うか」が最前線の論点です。なお、E2E プランニングのオープンループ評価(L2 誤差・衝突率)自体が ego status(自車の速度・加速度)に強く引っ張られ、知覚をほとんど無視しても良いスコアが出てしまうという指摘(AD-MLP, Zhai et al., 2023, arXiv:2305.10430)もあり、これはまさに本ノートの因果混同・慣性問題が評価指標のレベルでも顔を出している例で、履歴・自車状態の与え方は今も慎重な設計を要する未解決領域です。
