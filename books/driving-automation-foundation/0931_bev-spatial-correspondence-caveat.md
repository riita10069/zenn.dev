# BEV空間対応の前提と制約

## ひとことで言うと

複数のセンサや地図を「真上から見た同じマス目の地図」に揃えてから足し合わせたり比べたりする処理(BEV融合)は、両方の地図のマス目 (i, j) が現実世界の同じ場所を指しているときだけ意味を持ちます。この「空間対応(spatial correspondence)」が成り立つかどうかは、その特徴マップをどう作ったかで決まります。明示的に距離付きのグリッドを座標変換で作った場合は成り立ち、ただプーリングで作った抽象的な特徴マップでは成り立ちません。これは見落とすと静かにモデルを壊す、幾何の落とし穴です。

## 直感的な理解

2枚の透明シートに地図を描いて重ね合わせる場面を想像してください。1枚目はカメラ映像から推定した「ここに車、ここに車線」という地図、2枚目はカーナビの地図です。この2枚を重ねて「同じ場所に何があるか」を読み取りたい。ここで決定的なのは、2枚のシートが同じ縮尺・同じ原点・同じ向きで描かれていることです。もし1枚目が自車から前方50メートルを縦450マスで描き、2枚目が全然違う縮尺やズレた原点で描かれていたら、重ねた瞬間に「自車の真正面のマス目」と「カーナビ上の自車の真正面」がズレ、足し算した結果は意味のない混合物になります。

自動運転のモデルでも全く同じことが起きます。複数カメラの映像とHD地図を一つのBEV(Bird's-Eye View、鳥瞰図。真上から見下ろした平面)表現に統合するとき、各入力を独立にBEV特徴マップへ変換します。それらを `output = image_bev + α · map_bev` のように足し合わせたり、片方が片方に注目(cross-attention)して情報を引いてくる融合をしたりします。このとき暗黙の前提が「両方のBEVのマス目 (i, j) が同じego座標(自車を原点とした座標系)を指す」ことです。この前提が崩れると、計算は数値的には通るのに、幾何的な意味は失われます。問題はエラーにならず、ただ性能が出ない・期待した地図情報が効かないという形で表れるため、気づきにくいのです。

研究コミュニティでこの問題が繰り返し意識されてきたのは、BEV表現が「複数の入力を共通の座標系で混ぜる器」として急速に普及したからです。器が共通であることは、足し算の前に必ず座標が一致していることを要求します。逆に言えば、BEVという枠組みの強みは「座標を揃えてから混ぜる」規律を守ったときに初めて発揮されます。

## 基礎: 前提となる概念

順を追って用語をほどきます。

BEV(鳥瞰図特徴): カメラは斜め前を向いて世界を撮ります。画像のピクセル座標 (u, v) は「カメラから見た方向」を表すだけで、地面上の距離は直接持ちません。これに対しBEV特徴は「地面を真上から見た平面を格子(グリッド)で区切り、各マスに特徴ベクトルを詰めたもの」です。形状はテンソルで `(B, C, H, W)` と書き、B はバッチサイズ(同時に処理するサンプル数)、C は各マスの特徴次元(チャネル数)、H と W はグリッドの縦横のマス数です。例えば `(4, 256, 200, 200)` なら、4サンプル分、各マスが256次元、200×200マスの鳥瞰地図です。

ego座標系: 自車の中心を原点として、前方をx、左をy(あるいは右、流儀による)に取った座標系です。「前方20メートル、左3メートル」のように現実の距離で位置を表せます。BEVの (i, j) を現実の (X, Y) に対応づける基準は、常にこのego座標系です。

pc_range(point cloud range): BEVグリッドが現実のどの範囲をカバーするかを定義するパラメータです。例えば `[-50, -50, -5, 50, 50, 3]` なら、x方向 -50〜+50m、y方向 -50〜+50m、z方向 -5〜+3m を表します。グリッドのマス数 H×W とこの範囲が決まると、「マス (i, j) は現実のここ」という対応が一意に決まります。これが空間対応の土台です。具体的には、x方向の分解能は (xmax - xmin) / W、y方向は (ymax - ymin) / H で、例えば100m を 200マスで割れば1マスあたり0.5mです。

射影行列(projection / camera matrix): 3次元空間の点 (X, Y, Z) が、あるカメラの画像のどのピクセル (u, v) に写るかを決める行列です。カメラの内部パラメータ(焦点距離・主点など、intrinsics)と外部パラメータ(カメラが車体のどこにどの向きで付いているか、extrinsics)から組み立てます。式で書けば、同次座標で `[u, v, 1]^T ∝ K · [R | t] · [X, Y, Z, 1]^T` の形です。ここで K が intrinsics、[R | t] が extrinsics。これが正確であって初めて、グリッド点を画像へ正しく射影できます。

残差融合(residual fusion): `output = a + α · b` の形で、片方の特徴 a を主、もう片方 b を補正項として足す方式です。α は学習可能なゲート(門)で、しばしば0で初期化します。0初期化だと学習開始時は `output = a`、つまり「補正なし」と同じ挙動から始められ、安全に学習を立ち上げられます。これは ResNet の恒等写像近傍からの学習や、Transformer の層スケール(LayerScale)初期化と同じ思想です。

クロスアテンション融合(cross-attention fusion): 片方のマス目(クエリ側)が、もう片方のマス目群(キー・バリュー側)に重みをつけて注目し、必要な情報を引いてくる方式です。これも「クエリのマス目 (i, j) が、対応する場所の情報を引く」という前提に立ちます。

## 仕組みを詳しく

具体的な数式と形状で見ます。残差融合の典型的な実装はこうです。

```python
self.alpha = nn.Parameter(torch.zeros(embed_dim))   # チャネルごとのゲート、ゼロ初期化

def forward(self, image_bev, map_bev):
    gate = self.alpha.view(1, -1, 1, 1)             # (1, C, 1, 1) にして放送(broadcast)
    return image_bev + gate * map_bev
```

両入力は `(B, C, H, W)`、例えば `(4, 256, 200, 200)`。`image_bev + gate * map_bev` は要素ごとの足し算なので、`image_bev[b, :, i, j]` と `map_bev[b, :, i, j]` という、添字 (b, i, j) が完全に一致する位置どうしを混ぜます。式で書けば、

```
output[b, c, i, j] = image_bev[b, c, i, j] + α[c] · map_bev[b, c, i, j]
```

ここで (i, j) は両者で共通の添字です。この演算が幾何的に意味を持つ条件は、写像 (i, j) → ego座標 が両者で一致していること。つまり同じ pc_range と同じ H, W で両方のBEVを張ること。これが満たされれば、「自車前方20m・左3mのマス」同士を正しく足していることになります。

なぜ「明示的に距離付きでグリッドを張った場合だけ」成り立つのか。BEVFormer流の作り方を例に取ると、

1. ego座標系上に pc_range と H×W から決まる距離付きグリッドを張る。マス (i, j) の中心の現実座標 (X, Y) が数式で確定する。
2. 各グリッド点を複数の高さ(z)に持ち上げ、各カメラの射影行列で画像座標 (u, v) へ投影する。
3. 投影先の画像特徴を双線形補間でサンプリングし、そのマスに集約する。

この手順では「マス (i, j) は現実の (X, Y)」が定義によって保証されます。地図側も同じ pc_range・同じ H×W でラスタライズ(ベクトル地図を画素地図に焼く)すれば、両BEVの (i, j) は同じ (X, Y) に対応します。よって足し算もクロスアテンションも幾何的に正当です。

対照的に、各カメラビューの特徴を空間的にプーリング(平均などで縮約)して並べただけの「抽象的な特徴マップ」を image_bev として使う場合を考えます。このマップの (i, j) には「現実のどこ」という距離の意味がありません。単に何らかの内部表現が並んでいるだけです。ここに同形状の map_bev を足しても、足し合わせている (i, j) 同士が同じ場所を指す保証はどこにもありません。数値計算は通り、学習によって「ほどよい混ぜ方」を獲得することすらありますが、それはもはや幾何的融合ではなく、形状がたまたま合う2つのテンソルの学習されたブレンドにすぎません。

before/after を整理します。

| 特徴マップの作り方 | (i, j) の意味 | 地図BEVと同じ場所か | 融合の正当性 |
|---|---|---|---|
| 距離付きグリッド + 射影サンプリング(BEVFormer/LSS流) | ego座標に固定された距離付きセル | 同一 pc_range・同一 H×W なら一致 | 幾何的に正当 |
| 各ビュー特徴を空間プーリングして整形 | 距離の意味を持たない抽象表現 | 保証なし | 学習されたブレンドにとどまる |

数値例で破綻を体感します。仮に image_bev の (100, 100) が「自車前方0m・横0m(自車位置)」を指し、map_bev の (100, 100) が「前方25m・左方25m」を指していたとします。残差融合は何の警告も出さずこの2マスを足します。学習はこの定常的なズレに対して「では map_bev は無視しよう」と α を小さく学習するかもしれません。結果として地図情報は事実上効かず、しかしロスは下がり、テストも通る。問題は「地図を入れたのに性能が伸びない」という鈍い形でしか現れません。これがこの落とし穴の厄介さです。

ASCII図で「揃っている場合」と「揃っていない場合」を対比します。

```
揃っている (bevモード):              揃っていない (concat/抽象特徴):
image_bev[i,j] → ego(20,3)           image_bev[i,j] → 抽象表現 #k
map_bev  [i,j] → ego(20,3)           map_bev  [i,j] → ego(20,3)
  ↓ 足すと                              ↓ 足すと
ego(20,3) の情報が正しく重なる         無関係な2つを混ぜているだけ
```

このように、問題の本質は「形状(shape)が合うこと」と「幾何(geometry)が合うこと」が別物だという点です。PyTorch のようなフレームワークは形状さえ合えば加算を許します。幾何が合っているかどうかはフレームワークには分からず、設計者が保証しなければなりません。

## 手法の系譜と主要論文

BEV表現と幾何整合の系譜をたどります。

OFT / Orthographic Feature Transform (Roddick et al., BMVC 2019, arXiv:1811.08188)。単眼画像特徴を直交格子のBEVへ投影する初期の試み。画像特徴を地面平面へ落とすという発想の源流の一つです。voxel ごとに画像特徴を集める単純な逆投影で、深度を明示的に推定しない点が後の手法との差です。

Lift-Splat-Shoot (Philion & Fidler, ECCV 2020, arXiv:2008.05711)。各画素に深度の確率分布を予測し(lift)、その分布に沿って画素特徴を3D空間へばらまき(splat)、BEV平面へ落とす(shoot)。明示的に距離付きBEVを作る代表手法で、空間対応が定義上成り立ちます。トレードオフは深度予測の誤差がそのままBEVのズレになること。深度がぼやければBEV特徴も滲みます。

HDMapNet (Li et al., ICRA 2022, arXiv:2107.06307)。カメラ(と任意でLiDAR)から、車線・横断歩道などの地図要素をBEVのセマンティックマップとして推定。地図とセンサを同じBEVグリッドで扱う発想を明確化しました。

BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)。ego座標に張ったBEVクエリを、deformableな空間クロスアテンションで各カメラの射影位置周辺からサンプリングして埋める。BEVグリッドが明示的に距離付きである点が、「距離付きグリッドなら空間対応が成り立つ」根拠そのものです。さらに時間方向にも過去BEVを ego-motion で位置合わせ(warp)してから融合する。この「足す前に必ず座標を合わせる」設計が空間対応の好例です。

BEVFusion (Liu et al., ICRA 2023, arXiv:2205.13542 / および Liang et al. の同名研究 NeurIPS 2022, arXiv:2205.13790)。カメラとLiDARをそれぞれ独立にBEV特徴へ変換し、同一BEVグリッド上で連結・融合する。複数モダリティを「共通のBEV空間」で足し合わせる枠組みを一般化しました。ここでも前提は両モダリティが同じBEVグリッドに揃っていることです。

地図とセンサの統合という観点では、UniAD (Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156) のように、知覚・予測・計画を共通のBEV特徴の上で連結する研究が、すべて「同じego BEV空間に揃える」ことを暗黙の基盤にしています。系譜全体を貫く教訓は、足し合わせる前に必ず座標系を一致させる(warp / project して揃える)という規律です。研究コミュニティでは、この規律を「BEV を共通の特徴インターフェースとして使うための暗黙の契約」とみなしています。

## 論文の実験結果(定量データ)

空間対応の重要性は、位置合わせを「する/しない」のアブレーションで定量的に現れます。

BEVFormer は nuScenes(自動運転の標準ベンチマーク。約1000シーン、6カメラ)で評価されます。3D物体検出の主指標は NDS(nuScenes Detection Score。検出精度に加え位置・速度・属性の誤差を統合した0〜1の総合指標、高いほど良い)と mAP(mean Average Precision、平均適合率)です。報告によると BEVFormer は NDS おおよそ 0.517、mAP おおよそ 0.416 を達成し、当時のカメラのみ手法を上回りました。重要なのは時間融合のアブレーションで、過去BEVを ego-motion で位置合わせしてから融合する設計が、速度推定の誤差指標 mAVE(mean Average Velocity Error、低いほど良い)を大きく下げています。位置合わせをやめて素朴に過去BEVを重ねると速度推定が劣化する、という形で「足す前に座標を合わせる」ことの効果が出ています。mAVE は m/s 単位で測られ、過去フレームの整合が取れていれば物体の移動量を正しく見積もれる、という直感に対応します。

Lift-Splat-Shoot は nuScenes のBEVセマンティックセグメンテーション(鳥瞰図上で車線や車両領域を塗り分けるタスク。指標はIoU、Intersection over Union、予測領域と正解領域の重なり率)で評価されます。明示的な深度持ち上げによりBEVを作る手法として、当時の単純な射影ベース手法を上回るIoUを報告しました。深度分布の品質がIoUに直結することがアブレーションで示され、これは「空間対応が深度精度に依存する」ことの定量的裏付けです。深度がデルタ関数的に鋭いほどBEVの点が正しい位置に落ち、ぼやけるほど特徴が周囲に滲んでIoUが下がります。

BEVFusion(ICRA 2023)は nuScenes 物体検出で NDS おおよそ 0.729、mAP おおよそ 0.702(LiDAR+カメラ)を報告し、BEVセグメンテーションでも当時最高水準を示しました。カメラとLiDARを共通BEVで融合する効果が、単一モダリティ比で大きな mAP 向上として現れます(LiDAR 単独に対してカメラ情報を加えると、特に遠方の物体や小物体で AP が伸びます)。ここでも前提は両者が同一BEVグリッドに整合していることで、整合が崩れれば融合の利得は消えます。

一般化すると、これらの実験が共通して示すのは次の点です。第一に、複数の表現を同じBEV空間に正しく揃えてから融合すると、揃えない(あるいは抽象特徴を素朴に混ぜる)場合より一貫して指標が改善する。第二に、その改善幅は座標整合の正確さ(深度精度、キャリブレーション、ego-motion 補正)に比例する。第三に、整合が崩れてもロスは下がりうるので、検出・分割の指標まで見ないと退化に気づけない。

## メリット・トレードオフ・限界

メリット。距離付きグリッドで揃えたBEV融合は、地図情報やマルチセンサ情報を幾何的に正しい位置へ注入でき、知覚精度を素直に押し上げます。残差融合をα=0初期化にすれば「最初は補助入力なしと同じ」から安全に学習を立ち上げられ、補助入力が有害な場合は学習が自然に α を小さく保ちます。さらに、共通BEV空間を採用すると、新しいモダリティ(別のセンサや地図)を「同じグリッドに変換して足すだけ」で追加でき、拡張性が高いという設計上の利点もあります。

トレードオフと限界。第一に、空間対応は正確なキャリブレーション(射影行列)と一致した pc_range・解像度に全面的に依存します。これらがズレても演算はエラーにならず、性能が静かに落ちるだけなので、検出・分割の指標やオフセット検証ツールでの監視が要ります。第二に、抽象特徴を素朴に足す方式は、空間対応の保証がないまま「学習されたブレンド」として動いてしまい、本来期待した幾何的整合の利得が得られません。形状が合うこと(コンパイルが通ること)と幾何が合うこと(意味があること)は別物だ、というのがこの落とし穴の核心です。第三に、深度ベースの持ち上げ(LSS流)は深度誤差がBEVのにじみになり、射影ベース(BEVFormer流)はキャリブレーション誤差がサンプリング位置のズレになる。どちらも誤差の入り口が違うだけで、整合が完璧になることはありません。

未解決の課題として、キャリブレーションの経時ドリフト(走行による微小なカメラ位置変化)に頑健なオンライン補正、地図とセンサの時間ズレ(地図は静的、センサは現在時刻)の扱い、整合崩れを自動検知する仕組みなどが、研究上のオープンな問いとして残っています。特に「整合が崩れたことを学習中に検知する」のは難しく、検出やセグメンテーションといった下流指標が劣化して初めて気づく、という後追いになりがちです。

## 発展トピック・研究の最前線

近年は「明示的なBEVグリッドを介さず、3Dクエリで直接ego空間を表現する」方向(PETR系、Liu et al. ECCV 2022, arXiv:2203.05625)や、疎なクエリで効率化する Sparse4D 系も登場しています。これらも内部的にはクエリの3D参照点をego座標で持つことで空間対応を担保しており、「足す前に座標を合わせる」原則は形を変えて生き続けています。地図融合の側では、ベクトル地図を直接クエリ化して同一ego空間で扱う流れ([ベクトル化HDマップエンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0436_vectorized-hd-map-encoder))が主流化しつつあります。さらに、占有グリッド(occupancy)やNeRF的な連続表現でego空間を表す研究も進み、「どの表現でも、融合の単位はego座標で一致していなければならない」という幾何の規律は普遍です。占有予測(occupancy prediction)では座標の一致がボクセル単位まで要求され、空間対応の規律はむしろ厳しくなる方向にあります。
