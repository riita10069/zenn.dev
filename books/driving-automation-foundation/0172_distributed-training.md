# 分散学習

## ひとことで言うと

分散学習(distributed training)とは、1つのニューラルネットワークの学習を複数のGPU、あるいは複数のマシン(ノード)に分担させて、より速く、あるいはより大きなモデルを学習する技術です。「GPUが1枚で足りないなら8枚使えばいい」という単純な発想から始まりますが、複数のGPUが同じモデルを学習するには、それぞれが計算した結果(勾配)を互いにやり取りして同期する必要があり、ここに技術の中心があります。

## 直感的な理解

100万枚の画像を1枚のGPUで学習すると1日かかるとします。もし8枚のGPUがあれば、画像を8等分して各GPUに12.5万枚ずつ配り、同時に学習させれば理屈の上では約3時間で終わります。これが最も基本的な「データ並列(data parallelism)」の考え方です。

ただし、ここに落とし穴があります。8枚のGPUがそれぞれ別の画像で学習すると、8つの異なるモデルができてしまいます。これでは意味がありません。そこで、各GPUは自分の担当した画像から「モデルをどう更新すべきか」という方向(勾配, gradient)だけを計算し、その勾配を全GPUで平均してから、全員が同じ更新を適用します。こうすれば、8枚のGPUは常に同一のモデルを保ったまま、8倍のデータを同時に消化できます。

別の問題もあります。モデルが巨大で、そもそも1枚のGPUのメモリに乗り切らない場合です。たとえばパラメータが100億個あると、float32(1個4バイト)で40GBになり、L40Sのような45GBのGPUでもオプティマイザの状態などを足すと溢れます。このときはモデル自体を切り分けて複数GPUに分散させる「モデル並列(model parallelism)」が必要になります。

## 基礎: 前提となる概念

学習の1ステップは次の4段階で進みます。

1. 順伝播(forward): 入力データをモデルに通して予測を出す
2. 損失計算(loss): 予測と正解のズレを数値にする
3. 逆伝播(backward): 損失を各パラメータで微分し、勾配を求める
4. 更新(optimizer step): 勾配を使ってパラメータを少し動かす

ここで重要なメモリ消費の内訳を押さえます。学習中にGPUメモリを食うのは主に次の4つです。

- パラメータ(weights): モデルの重み本体。100億パラメータなら float32 で 40GB
- 勾配(gradients): パラメータと同じ数だけ存在。同じく 40GB
- オプティマイザ状態(optimizer states): Adamなら1パラメータあたり「移動平均(momentum)」と「分散(variance)」の2つを持つので、float32 で 80GB
- アクティベーション(activations): 順伝播の途中結果。逆伝播で必要なため保持する。バッチサイズと層数に比例

つまり100億パラメータのモデルを Adam で学習すると、パラメータ40 + 勾配40 + オプティマイザ80 = 160GB がアクティベーションを除いても必要になります。これは単一GPUにはとても乗りません。分散学習が「速度」だけでなく「そもそも学習を可能にするため」に必要となる理由です。

専門用語を1つ。「集団通信(collective communication)」とは、複数のGPUが協調してデータを交換する操作の総称です。代表的なのが all-reduce で、これは「全GPUが持つ値を合計(または平均)し、その結果を全GPUに配り直す」操作です。勾配の同期はこの all-reduce で行われます。

## 仕組みを詳しく

### データ並列 (DDP: Distributed Data Parallel)

PyTorch の DDP は最も標準的な手法です。各GPUにモデルの完全なコピー(replica)を置き、ミニバッチを GPU 数で分割します。

```
GPU0: モデル全体 + データの1/4 → 勾配g0
GPU1: モデル全体 + データの1/4 → 勾配g1
GPU2: モデル全体 + データの1/4 → 勾配g2
GPU3: モデル全体 + データの1/4 → 勾配g3
          ↓ all-reduce で勾配を平均
   全GPU が (g0+g1+g2+g3)/4 を受け取る
          ↓
   全GPU が同じ更新を適用 → モデルは同一に保たれる
```

DDP の巧妙な点は、逆伝播の途中で勾配が出来上がった層から順に通信を始める「通信と計算のオーバーラップ」です。最後の層の逆伝播が終わるのを待たず、計算と通信を同時進行させることで通信コストを隠します。

DDP の限界は明確で、各GPUがモデル全体・勾配全体・オプティマイザ状態全体を持つため、メモリ削減には一切ならない点です。速くはなりますが、大きなモデルは載りません。

### ZeRO とメモリの分割

ここで登場するのが ZeRO (Zero Redundancy Optimizer) です。Microsoft の Rajbhandari らが 2020年(SC'20, arXiv:1910.02054)に提案しました。「DDP はオプティマイザ状態・勾配・パラメータを全GPUで重複して持っている。この冗長(redundancy)をゼロにしよう」という発想です。

ZeRO は段階的に冗長を削ります。

- ZeRO-1: オプティマイザ状態だけをGPU間で分割保持する。各GPUは全体の 1/N だけ持つ
- ZeRO-2: 加えて勾配も分割する
- ZeRO-3: さらにパラメータ本体も分割する

ZeRO-3 では、各GPUは普段はパラメータの 1/N しか持ちません。順伝播である層を計算する直前に、その層のパラメータを all-gather(全GPUのかけらを集めて完全な層を一時的に復元する操作)で集め、計算が終わったら捨てます。こうしてピークメモリを抑えつつ、論理的には全パラメータで学習できます。

### FSDP (Fully Sharded Data Parallel)

PyTorch がこの ZeRO-3 の考え方を公式実装したものが FSDP です。「Sharded(細かく断片化された)」という名前のとおり、パラメータ・勾配・オプティマイザ状態をすべて GPU 間で断片(shard)に分けます。

FSDP の1層分の動作を時系列で書くと次のようになります。

```
[順伝播]
  all-gather でこの層の完全なパラメータを復元
  → 計算
  → 自分の担当 shard 以外のパラメータを解放(メモリを返す)
[逆伝播]
  再び all-gather でパラメータを復元
  → 勾配を計算
  → reduce-scatter で勾配を平均しつつ各GPUに 1/N ずつ配る
  → パラメータを解放
```

reduce-scatter は all-reduce を分解した操作で、「合計(または平均)した結果を、全GPUに丸ごと配るのではなく 1/N ずつ分けて配る」ものです。これにより各GPUは自分の担当分の勾配だけを持てばよく、メモリが節約されます。

### モデル並列・パイプライン並列・テンソル並列

データ並列でもメモリが足りないほど巨大なモデルでは、モデルそのものを切ります。

- パイプライン並列(pipeline parallelism): 層をグループに分け、GPU0が層1-8、GPU1が層9-16…と縦に分担。ミニバッチをさらに細かい「マイクロバッチ」に割り、流れ作業のように各GPUを埋める。GPipe(Huang et al., NeurIPS 2019, arXiv:1811.06965)が代表
- テンソル並列(tensor parallelism): 1つの行列演算そのものを複数GPUで分担。たとえば大きな全結合層の重み行列を列方向に切り、各GPUが部分積を計算して結合する。Megatron-LM(Shoeybi et al., 2019, arXiv:1909.08053)が代表

現実の大規模学習ではこれらを組み合わせた「3D並列(データ × パイプライン × テンソル)」が使われます。

## 手法の系譜と主要論文

- GPipe (Huang et al., NeurIPS 2019, arXiv:1811.06965): パイプライン並列を実用化。マイクロバッチで「バブル(GPUが遊ぶ空き時間)」を削減
- Megatron-LM (Shoeybi et al., 2019, arXiv:1909.08053): Transformer に特化したテンソル並列。8.3B パラメータの GPT を学習
- ZeRO (Rajbhandari et al., SC 2020, arXiv:1910.02054): メモリ冗長の排除。後の DeepSpeed の中核
- PyTorch Distributed / DDP (Li et al., VLDB 2020, arXiv:2006.15704): 通信と計算のオーバーラップによる高効率データ並列の標準実装
- ZeRO-Offload (Ren et al., 2021, arXiv:2101.06840): オプティマイザ状態をCPUメモリに退避し、単一GPUで巨大モデルを学習
- FSDP (Zhao et al., 2023, arXiv:2304.11277): ZeRO-3 を PyTorch ネイティブに統合

自動運転の文脈では、BEV(俯瞰視点)知覚モデルや大規模な軌道予測モデルが高解像度・多カメラ入力でメモリを食うため、DDP やマルチノード学習が日常的に使われます。

## 論文の実験結果(定量データ)

ZeRO の原論文(SC'20)では、メモリ削減効果が定量的に示されています。64個のGPU(V100, 各32GB)を使い、ZeRO-1/2/3 を段階適用することで、従来 1.4B パラメータが上限だったところを 100B パラメータ超まで学習可能にしました。ZeRO-2 の段階で 1台あたりのメモリ使用を約 8分の1 に削減し、スループットも線形に近いスケーリング(GPU数を増やすほど比例して速くなる)を達成したと報告されています。論文の代表的なデモでは 400個のGPU で 100B パラメータモデルを学習し、1GPUあたり 38 TFLOPS を超える効率を示しました。

ZeRO-Offload (2021) はさらに極端で、たった1枚の V100(32GB)で 13B パラメータのモデルを学習できることを示しました。これは当時 PyTorch 単体で載せられる上限(約1.4B)の約10倍です。オプティマイザの計算と状態をCPUに逃がすことで実現しています。

FSDP の論文(2023)では、175B パラメータの GPT-3 級モデルを 128〜512 個の A100 GPU で学習し、モデルサイズに対してほぼ線形のメモリスケーリングと、A100あたり 84 TFLOPS 級のスループットを報告しています。DDP では原理的に載らないサイズが FSDP では学習できる、というのが核心的な結果です。

数値の意味を補足します。TFLOPS は1秒あたり何兆回の浮動小数点演算ができたかで、GPUの理論ピーク(A100 で bf16 約312 TFLOPS)に対する実効値です。84/312 ≒ 27% という値は、通信や同期のオーバーヘッドを差し引いた現実的な効率で、大規模分散学習としては良好な水準です。

## メリット・トレードオフ・限界

メリットは明快です。学習時間の短縮(データ並列)と、単一GPUに載らないモデルの学習(モデル並列・FSDP)の2点に集約されます。

トレードオフと限界は次のとおりです。

- 通信コスト: GPU数を増やすほど all-reduce のデータ量と同期回数が増える。ネットワーク帯域(ノード間の場合は InfiniBand や NVLink の有無)がボトルネックになり、GPU数を倍にしても速度が倍にならない「スケーリング効率の低下」が起きる
- メモリと速度のトレードオフ: FSDP/ZeRO-3 はメモリを節約する代わりに、層ごとに all-gather する通信が増えるため、DDP より遅くなりうる。メモリに余裕があれば DDP の方が速い
- 実効バッチサイズの肥大化: GPU数を増やすと全体のバッチサイズが大きくなり、学習率の調整(linear scaling rule など)を誤ると収束が悪化したり精度が落ちる
- デバッグの難しさ: 複数プロセス・複数ノードにまたがるため、1台だけ落ちる、デッドロック(全GPUが互いの通信を待ち合って止まる)などの障害切り分けが難しい
- 再現性: 勾配の合計順序が変わると浮動小数点の丸め誤差で結果がわずかに変動しうる

## 発展トピック・研究の最前線

- 勾配チェックポイント(gradient/activation checkpointing): アクティベーションを保存せず、逆伝播時に再計算する。メモリと計算のトレードオフで、長いコンテキストや大バッチを可能にする
- 混合精度(mixed precision)との併用: bf16/fp16 でパラメータと計算を半分のメモリにし、分散学習のメモリ圧をさらに下げる。L40S や A100 では bf16 が標準
- 通信圧縮: 勾配を量子化・スパース化して all-reduce の通信量を減らす研究
- 非同期・ローカル更新: 毎ステップ同期せず、数ステップごとに同期する手法(Local SGD)でノード間通信を減らす
- MoE(Mixture of Experts)のためのエキスパート並列: 巨大モデルの一部だけを各入力で活性化させ、専門家(expert)を GPU 間に分散する

未解決の課題として、数千〜数万 GPU 規模での通信効率の維持、故障に強い学習(1台落ちても止まらない fault tolerance)、メモリ・通信・計算の最適な並列戦略を自動探索する問題などが活発に研究されています。
