# BEV特徴キュー & Warp融合

## ひとことで言うと
カメラやLiDARから作った「上空から見下ろした地図のような特徴(BEV特徴)」を、1フレーム作って捨てるのではなく、過去の数フレーム分を箱(キュー)にためておきます。そして自分の車が動いたぶんだけ過去のBEVを座標変換(ワープ)して現在の位置に揃え、現在のBEVと重ね合わせます。これが多くの「BEV時系列手法」に共通する一番基本のしくみで、1枚の画像では決して得られない「速度」を機械に推定させるための土台になります。

## 直感的な理解
あなたが助手席に座って、運転手に「あの前の車、止まってる? それともこっちに近づいてる?」と聞かれたとします。一瞬チラッと見ただけでは答えられません。少し時間をおいて2回見比べて、初めて「あ、近づいてる」と分かります。機械も同じで、1枚の静止画像からは物体の速度が原理的に取れません。位置は写っていても、その物体が次の瞬間どこへ行くのかは、複数の時刻を見比べないと分からないのです。

そこで過去のフレームを覚えておきたい。ところが厄介なのは、自分の車が動いているという事実です。あなたが前進すれば、止まっている建物も電柱も、見かけ上は後ろへ流れていきます。1秒前に「右前方5m」にあった建物は、今は「すぐ右」に来ています。過去と現在をそのまま重ねると、止まっている物まで二重写しになってしまう。この「自分が動いたぶんの見かけのズレ」を先に幾何計算で消してから重ねよう、というのがWarp(ワープ)融合の核心です。自分の動きで説明できるズレを消すと、後に残るのは「他の物が本当に動いたズレ」だけになり、機械はそこから速度を読み取れます。

## 基礎: 前提となる概念

このトピックを理解するために、いくつかの言葉を順に噛み砕きます。

- BEV (Bird's Eye View、鳥瞰図): カメラ画像をそのまま扱うのではなく、上空から地面を見下ろしたような2次元のマップに変換した特徴のことです。たとえば自車を中心に前後左右50mずつ、1マス0.5mで表すと200×200のグリッド(マス目)になります。各マスには「そこに何があるか」を表す数十〜数百次元のベクトル(数字の並び)が入ります。テンソルの形で書くと `[C, H, W]`、たとえば `[80, 200, 200]` です。Cはチャンネル数(各マスの特徴ベクトルの次元)、HとWはグリッドの縦横です。なぜBEVが好まれるかというと、複数のカメラを1つの共通座標系に統合でき、しかも「地図」と同じ平面なので経路計画や衝突判断にそのまま使いやすいからです。

- 特徴 (feature): ニューラルネットが入力から取り出した「意味のある数字の並び」のことです。画像のピクセルそのものではなく、「ここは道路っぽい」「ここに車がいそう」といった抽象情報が圧縮されています。生のピクセルより情報密度が高く、後段の処理が扱いやすくなります。

- 時系列 (temporal): 1枚の画像だけでなく、過去から現在まで連続したフレームの並びのことです。動いているか止まっているか、どっちへ向かっているかは1枚では分からず、複数枚を見て初めて分かります。

- ego-motion (自車運動): 自分の車が短い時間にどれだけ移動し、どれだけ向きを変えたかを表す量です。`(dx, dy, dθ)` のように、前後移動・左右移動・ヨー回転(地面に垂直な軸まわりの回転、つまり車が左右に曲がる回転)で表します。これはオドメトリ(車輪の回転数)やIMU(慣性計測装置)、あるいはGPSや自己位置推定から得られます。

- ワープ (warp、座標の付け替え): ある座標系で表された特徴マップを、別の座標系から見たらどう見えるかに作り直す操作です。後で詳しく説明しますが、本質は「出力グリッドの各マスが、入力グリッドのどこの値を拾ってくればよいか」を計算して、そこから値をサンプリングし直すことです。

なぜ過去フレームが要るのかを改めて整理します。時系列を使うと得られる利益は主に3つです。第一に速度・加速度の推定で、これは1枚では原理的に不可能です。第二にオクルージョン(occlusion、ある物が別の物に一瞬隠れること)への耐性で、隠れる前に見えていた情報を覚えておけば、隠れている間も「そこにいるはず」と補完できます。第三に検出のちらつき低減で、フレームごとに独立に検出すると物体が点滅するように出たり消えたりしますが、時系列で文脈を持つと安定します。

## 仕組みを詳しく

典型的な処理は次の3部品からなります。

第一にキュー (queue、待ち行列) です。過去Nフレーム分のBEV特徴をリングバッファ(古いものから順に上書きされる固定長のバッファ)のようにためます。たとえば直近4フレームを保持し、新しいフレームが来たら一番古いものを捨てます。各要素は `[C, H, W]` のテンソルです。Nを大きくするほど長い時間文脈を持てますが、メモリ消費が線形に増えます。

第二にWarp (ego-motion compensation、自車運動補償) です。フレーム t-1 から t の間に自車がどれだけ進み・曲がったか `(dx, dy, dθ)` を用意し、過去BEVを「現在の自車座標系」へ再サンプリングします。具体的には2×3のアフィン変換行列を作り、深層学習フレームワークなら `affine_grid`(変換行列からサンプリング座標グリッドを生成する関数)と `grid_sample`(そのグリッドに従ってバイリニア補間でサンプリングする関数)を組み合わせて、微分可能に(=学習で勾配が流せる形で)行います。微分可能であることが重要なのは、ワープ後の融合結果に対する損失の勾配を、ワープを通り抜けて前段の特徴抽出まで伝えられるからです。

第三に融合 (fusion) です。座標を揃えた過去BEVと現在BEVを、チャンネル方向に連結(concat、`[80,H,W]` と `[80,H,W]` を並べて `[160,H,W]` にする)するか、要素ごとに加算するか、注意機構(attention)で重みづけて混ぜます。最も単純なのはconcatで、その後に畳み込み層を1枚かければチャンネル数を元に戻しつつ時系列情報を混ぜられます。

ステップを数値例で追います。1マス0.5mのBEVで、自車が0.5秒で5m前進し3度左に回ったとします。前進5mはBEVグリッド上で10マス分のシフトに相当します(5m ÷ 0.5m/マス)。左3度はグリッド全体を中心まわりに3度回転させることに相当します。これを2×3のアフィン変換行列にまとめ、過去BEV `[80,200,200]` を `grid_sample` で再サンプリングします。結果、過去フレームの「止まっている物」は現在フレームと同じマスにピタッと重なり、動いていた他車だけがズレとして残ります。ネットはそのズレから速度を読み取ります。

ASCII図(前進だけの簡略図、Fは前方):

```
   過去BEV(t-1)        ワープ後             現在BEV(t)
   ┌─────────┐        ┌─────────┐         ┌─────────┐
 F │   ■建物  │  自車   │ ■建物    │  重ねる  │■建物     │
   │         │ 10マス  │         │  =====> │         │
   │  ●自車  │ 前進ぶん│●自車    │         │●自車    │
   └─────────┘ 下へずらす└─────────┘         └─────────┘
   建物は遠い            建物が近い位置に来た   現在も同じ位置 → ピタリ
```

ここで重要なのは、Warpは「学習する重み」ではなくほぼ純粋な幾何計算だという点です。自車の動きが分かれば、止まっている景色がどうズレるかは数学で確定します。ネットに学ばせるのは「本当に難しい部分、つまり他の物が実際に動いたズレ」だけで済みます。これは設計思想として非常に効率的で、学習が必要なパラメータ数を増やさずに大きな精度向上を得られる理由になっています。

設計上の選択肢として、ワープを「特徴レベルで行うか、それとも生BEVを保持して特徴抽出後にワープするか」「2フレームだけ使うか、多数フレームを使うか」「キューを毎フレーム持つか、再帰的な隠れ状態1つに畳み込むか」という分岐があります。これらが後述する論文の系譜を形作っています。

## 手法の系譜と主要論文

この分野は「2フレームの最小拡張」から始まり、「長期化」「再帰化(streaming)」へと進化してきました。

BEVDet4D (Huang & Huang, 2022, arXiv:2203.17054) は出発点です。単一フレームのカメラBEV検出器BEVDetに「もう1フレーム前のBEV」を足しただけのシンプルな拡張でした。鍵は、過去BEVを ego-motion で現在座標に align(整列)してから concat する点です。著者の動機は「時系列を入れたいが大改造はしたくない」というもので、最小の変更で速度推定能力を獲得することを狙いました。限界は、2フレームだけなので長い時間文脈は持てないことです。

BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270) は空間と時間を統一したアテンションで扱いました。BEVクエリ(グリッド各マスを表す学習ベクトル)が、空間方向はマルチカメラ特徴に向かって spatial cross-attention を、時間方向は「1つ前のBEV」に向かって temporal self-attention を行います。過去BEVは現在座標へワープしてから参照されます。新規性は、ワープした過去BEVを単に足すのではなく、各クエリが過去BEVのどこを参照するかをアテンションで動的に決める点でした。トレードオフはアテンションの計算コストが高いことです。

SOLOFusion (Park et al., ICLR 2023, arXiv:2210.02443) は「どれだけ過去を見るとよいか」を体系的に検証しました。「短期・高解像度」と「長期・低解像度」の2系統で過去BEVキューを保持・ワープして融合します。最大16フレームまでの長期文脈が精度に効くことを示し、時間スタックの長さが重要なハイパーパラメータであることを明らかにしました。トレードオフはメモリ消費の増大です。

StreamPETR / VideoBEV 系 (Wang et al., ICCV 2023, arXiv:2303.11926 ほか) は、キューを毎フレーム持つ代わりに、過去の情報を「再帰的な隠れ状態」やオブジェクトクエリの形で次のフレームへ伝播する streaming(逐次)方式を採りました。物体中心の表現を使い、過去のオブジェクトクエリをego-motionと推定速度で現在へ伝播します。これによりメモリと計算を時間長によらず一定に保てます。RNN的な再帰の発想をBEV/物体クエリ時系列に持ち込んだものと理解できます。

## 論文の実験結果(定量データ)

評価は主にnuScenesデータセット(都市部の走行データで、6カメラ・LiDAR・レーダーを備え、3D物体検出と速度推定の正解ラベルがある)で行われます。主要指標を先に説明します。

- mAP (mean Average Precision): 検出の正確さを0〜1で表す指標で、高いほどよいです。物体をどれだけ正しく見つけ、誤検出が少ないかを総合的に測ります。
- NDS (nuScenes Detection Score): nuScenes独自の総合指標で、mAPに加えて位置・サイズ・向き・速度・属性の誤差を統合したものです。0〜1で高いほどよいです。
- mAVE (mean Absolute Velocity Error): 推定速度の平均絶対誤差で、単位はm/sで低いほどよいです。時系列融合の効果が最も直接的に表れる指標です。

BEVDet4D は、単一フレームのBEVDetに対し mAVE をおよそ半分以下に下げたと報告されています(報告によりBEVDetの mAVE が0.9前後だったものが0.4前後の水準まで改善)。NDSも数ポイント向上しました。これは「過去1フレームをego-warpして足すだけ」で速度推定能力が一気に獲得されたことを意味し、時系列融合の威力を端的に示した結果です。アブレーション(どの要素を抜くと性能がどう変わるかの検証)では、ego-motionによるalignを行わずに過去BEVをそのまま足すと、速度誤差がほとんど改善しないか悪化することが示されました。つまりワープによる座標整列こそが効果の源泉です。

BEVDet4D の論文ではもう一つ重要な発見が報告されています。それは「過去フレームと現在フレームの時間差(Δt)を学習対象として陽に与えるかどうか」が速度推定の精度を左右するという点です。速度は「位置の変化量 ÷ 時間差」で決まるので、Δtを知らないネットは「どれだけ動いたか」は分かっても「毎秒何メートルか」を正規化できません。隣接フレーム間の実時間(nuScenesでは約0.5秒)を考慮して特徴を作ることで、mAVEがさらに改善したと報告されています。これは「速度推定には空間整列だけでなく時間スケールの明示が必要」という、時系列融合の一般原則を示した例です。

SOLOFusion は、融合するフレーム数を増やすほど性能が上がることを定量的に示しました。報告では、長期(16フレーム)融合により nuScenes test で NDS がおよそ0.6を超える水準に達し、当時の最先端を更新しました。アブレーションでは、長期低解像度系統と短期高解像度系統の両方が必要で、片方だけでは性能が落ちることが示されています。長期系統は速度・遠距離物体に、短期系統は近距離の細かい位置精度に効きます。さらに重要なのは、SOLOFusion が「フレーム数」と「BEV解像度」が交換可能なリソースであることを定量化した点です。同じ計算予算なら「高解像度で短期」より「低解像度で長期」のほうが NDS が高くなる領域があると報告されており、これは「時間方向の情報は空間解像度より価値が高い場合がある」という、当時の直感に反する結論でした。なぜそうなるかというと、遠方の小さな物体や速度の推定は1フレームの空間精度を上げても解決せず、複数時刻の比較でしか得られない情報だからです。

このフレーム数依存性を測るアブレーションは、後続研究の設計指針になりました。具体的には、2フレームから始めて4, 8, 16フレームと増やすと NDS は単調に上がりますが、mAVE(速度誤差)の改善が最も急峻で、16フレーム付近で改善が緩やかになる(逓減する)ことが示されています。これにより「長くするほど良いが、収穫逓減とメモリ増大のどこで止めるか」という運用判断の基礎データが整いました。

StreamPETR 系は、計算量を時間長によらず一定に保ちながら、長期キュー方式に匹敵あるいは上回る NDS を達成したことが報告されています。これは「長期文脈は欲しいがメモリは増やしたくない」という実用上の要求に応える結果で、オンライン推論(リアルタイムで1フレームずつ処理する状況)に適しています。

これらをまとめると、(1)ego-warpによる整列が速度推定の前提条件であること、(2)時間文脈は長いほど良いが収穫は逓減しメモリと引き換えになること、(3)再帰化によりコストを抑えつつ長期文脈の利益を取れること、が分野の共通理解です。

## メリット・トレードオフ・限界

メリットとして、1枚では取れない速度・加速度が推定できること、オクルージョンに強いこと、フレーム間の検出のちらつきが減って追跡が安定すること、そしてego起因のズレを幾何計算で消すのでネットが本質的に難しい動的部分の学習に集中できることが挙げられます。

トレードオフと限界も明確です。キューを持つぶんメモリが増え、長期融合ほど顕著になります。ego-motionの推定誤差や、軸・符号の規約(どの軸が前後で、どれが回転の正方向か)のミスは、そのままワープのズレとして検出精度を下げます。実装時にこの座標規約の取り違えは頻出のバグ源です。ワープで現在座標に動かすと、過去フレームに写っていなかった領域(視野の外、あるいは新たに視界に入った領域)は埋められず空白(パディング)になります。そして最大の限界は、ego-warpでは動的物体は揃わないことです。揃うのはあくまで静止物だけで、動いている他車や歩行者を時系列で正確に整列させたいなら、点ごとに異なる運動を扱う手法が別途必要になります。

研究上の未解決課題としては、ego-motion推定誤差にロバストな融合、長期と短期の最適な配分の自動決定、動的物体の運動を陽に補償する融合(後述の運動場ベースのワープとの統合)、そして再帰状態が長時間でドリフト(誤差が蓄積していく現象)する問題への対処などがあります。

## 発展トピック・研究の最前線

近年は、ego-warpで静止物を揃えた上で、残った動的成分を運動場(各点の運動ベクトルの地図)でさらに揃える二段構えのアプローチや、占有予測(occupancy prediction、空間を細かいボクセルに区切って各ボクセルが占有されているかと、その運動を予測する)へ時系列融合を拡張する流れが活発です。また、画像→BEVの持ち上げ(lifting)誤差と時系列ワープ誤差を切り分けて学習する研究や、複数の時間スケールを自動で混ぜる適応的融合も提案されています。Transformerベースの統一アーキテクチャの中で時間軸をトークンとして扱う方向も主流になりつつあります。
