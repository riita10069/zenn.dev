# 双方向RNN

## ひとことで言うと
双方向 RNN は、系列を「過去から未来へ」読む RNN と「未来から過去へ」読む RNN の2本を並べて走らせ、各時刻でその前後両方の文脈を同時に使えるようにした仕組みです。系列全体がすでに手元に揃っている場面(オフライン処理)では精度を大きく上げますが、未来をまだ知らないリアルタイム処理には原理的に使えない、という明確な得意・不得意を持ちます。

## 直感的な理解
普通の RNN は系列を時刻順に1方向だけ読むため、時刻 `t` の理解には「`t` までの過去」しか使えません。しかし人間が文章を読むとき、私たちはしばしば「先を読んでから前に戻って解釈し直す」ことをします。

たとえば「銀行」という単語にラベルを付ける場面を考えます。「川の銀行(=土手)」なのか「お金を預ける銀行」なのかは、その単語だけ見ても決まりません。後ろに「で泳いだ」と続けば土手、後ろに「で口座を開いた」と続けば金融機関です。つまり正しい解釈には未来側(後ろ)の文脈が要ります。音声認識でも、ある瞬間の音素は前後の音の流れで決まります。これらは「系列が録音済み・記録済みで全部手元にある」状況なので、未来側を覗いて構いません。1方向 RNN ではこの未来の手がかりを使えず、もったいないのです。双方向 RNN は、前向きと後ろ向きの2本を走らせることで、各時刻に前後両方の文脈を渡します。

別の例として、英語の品詞タグ付けで "I will book a flight"(私は航空券を予約する)の "book" を考えます。"book" は名詞(本)にも動詞(予約する)にもなります。前だけ見ると "I will" の後ろなので動詞らしいと推測できますが、"This is a book"(これは本です)では前の "a" を見ても、後ろがどうなるかで確定する場面が多々あります。前後を同時に見れば曖昧性がぐっと減る、というのが双方向化の御利益です。

## 基礎: 前提となる概念
前提として、再帰ネットワーク(RNN)は系列を1ステップずつ処理し、隠れ状態 `h_t`(時刻 t までの要約メモリ)を更新していくネットワークです。素朴な RNN は勾配消失で長期依存が苦手で、これを改良したのが [LSTM (Long Short-Term Memory)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0490_lstm_note) と [GRU (Gated Recurrent Unit)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0486_gru_note) です。双方向化はこれら任意の RNN セルに適用できる「使い方の工夫」であり、新しいセルではありません。実際に双方向と組み合わせる場合は LSTM や GRU を使うのが普通で、BiLSTM・BiGRU と呼ばれます。

オンライン処理とオフライン処理の区別も重要です。オンライン処理は、データが時々刻々と到着し、その都度すぐ出力を出さねばならない設定(走行中のセンサ処理、ライブ字幕など)です。未来のデータはまだ存在しません。オフライン処理は、系列全体がすでに記録済みで一括して処理できる設定(録音済み音声の書き起こし、保存済みログの解析、収集済みの走行ログの後処理など)です。双方向 RNN が使えるのはオフライン処理に限られます。

系列ラベリング(sequence labeling)とは、系列の各要素にラベルを付けるタスクの総称です。品詞タグ付け(各単語に名詞・動詞などを割り当て)、固有表現抽出(人名・地名などの範囲を特定)、音素分類などが代表で、双方向 RNN が最も力を発揮する応用分野です。「各時刻に1つ出力を出す(入力長と出力長が一致する)」タスクであることがポイントで、これは後述のように生成タスクと相性が悪い理由とも関わります。

## 仕組みを詳しく
入力系列を `x_1, x_2, ..., x_T` とします。双方向 RNN は独立した2つの RNN を持ちます。

前向き RNN は `x_1 → x_2 → ... → x_T` の順に読み、各時刻で前向き隠れ状態 `h→_t` を作ります。これは「時刻 t までの過去」を要約したものです。後ろ向き RNN は `x_T → x_{T-1} → ... → x_1` と逆順に読み、各時刻で後ろ向き隠れ状態 `h←_t` を作ります。これは「時刻 t から先の未来」を要約したものです。各時刻 `t` の最終的な表現は、両者を連結(concatenate、横に並べる)します。

```
h_t = [ h→_t ; h←_t ]
```

tensor shape の数値例を挙げます。隠れ次元 H=128 の前向き・後ろ向きをそれぞれ用意すると、`h→_t` も `h←_t` も `[B, 128]`、連結した `h_t` は `[B, 256]` です。系列全体では `[B, T, 256]` になります。ライブラリでは LSTM/GRU 層に「双方向」のフラグを1つ立てるだけで実現でき、出力次元が `2 × hidden_size` になります。後段の層(分類器など)の入力次元をこの2倍に合わせる必要があり、ここが実装でよく忘れる落とし穴です。多層にする場合は、各層の双方向出力(2倍次元)が次の層の入力になる点にも注意します。

連結のほかに、和 `h→_t + h←_t` や平均で結合する変種もありますが、連結が最も一般的です。連結だと前後の情報を独立に保てるためです(和や平均だと前向き・後ろ向きが混ざって区別がつかなくなる)。

ASCII 図で全体像を示します。

```
前向き:  h0→─►[RNN]─►h1→─►[RNN]─►h2→─►[RNN]─►h3→
                x1        x2        x3
後ろ向き:      [RNN]◄─h2←◄─[RNN]◄─h3←◄─[RNN]◄─ h4←(末尾から)
                x1        x2        x3
各時刻 t の出力 = [ ht→ ; ht← ] を連結
```

学習は前後それぞれの RNN を BPTT(時間方向の誤差逆伝播)で同時に学びます。2本の RNN は重みを共有せず、独立に学習されます。ここで決定的に重要な制約があります。後ろ向き RNN が時刻 `t` の表現 `h←_t` を作るには、系列の末尾 `x_T` から読み始めて `t` まで戻ってこなければなりません。つまり末尾 `x_T` が確定するまで `h←_t` は計算できません。したがって、各時刻の最終表現 `h_t` は「系列が全部揃ってから」しか得られません。これがリアルタイム処理に使えない理由であり、なぜそういう設計なのか(=未来文脈を使うという目的そのものの代償)を端的に示しています。設計の御利益と制約が同じ1つの原理(末尾まで読む)から出ている、という構造を理解すると、後述の限界も腑に落ちます。

## 手法の系譜と主要論文
Schuster と Paliwal「Bidirectional Recurrent Neural Networks」(1997, IEEE Transactions on Signal Processing 45(11))が出典です。前向き・後ろ向きの2つの RNN を組み合わせ、各時刻で過去と未来の両文脈を使う枠組みを定式化し、BPTT による同時学習の方法を示しました。トレードオフとして、系列全体が必要でオンライン処理に使えない点を最初から明記しています。当時はまだ LSTM が普及する前で、素朴な RNN ベースの双方向化でした。

Graves と Schmidhuber「Framewise Phoneme Classification with Bidirectional LSTM and Other Neural Network Architectures」(2005, Neural Networks)が、双方向化と LSTM の長期依存を組み合わせた BiLSTM を提案・実証しました。これにより「長期記憶(LSTM)」と「前後文脈(双方向)」の両取りが可能になり、音声系タスクで一気に実用水準へ近づきました。

Huang, Xu, Yu「Bidirectional LSTM-CRF Models for Sequence Tagging」(2015, arXiv:1508.01991)は、BiLSTM の上に CRF(Conditional Random Field、系列全体のラベルの整合をとる層。例えば「固有表現の途中ラベル(I-)は開始ラベル(B-)の後にしか来ない」といった遷移の制約を学ぶ)を載せ、品詞タグ付け・チャンキング・固有表現抽出で当時の最高水準に並ぶ結果を出しました。BiLSTM-CRF はその後、系列ラベリングの定番アーキテクチャになりました。

系譜として、双方向に文脈を見るという発想は Transformer の双方向版である BERT(Devlin, Chang, Lee, Toutanova, 2019, NAACL)へと受け継がれます。BERT は「両側の文脈を同時に見る」を自己注意(self-attention)で実現し、マスク言語モデル(文中の単語を隠して両側から当てる学習)で大量データから事前学習しました。双方向 RNN は BERT の精神的な先祖と言えます。一方、生成方向に特化した GPT 系は前向き(因果的)のみを使い、双方向は使いません。これは「双方向はラベリングに向き、生成には向かない」という双方向 RNN 時代からの原理が、Transformer の世代でもそのまま継承されていることを示します。

## 論文の実験結果(定量データ)
Schuster と Paliwal (1997) は音素分類で、双方向 RNN が1方向 RNN(前向きのみ・後ろ向きのみ)より誤り率を下げることを示しました。直感どおり、前後両方の文脈を使える方が各時刻の判定が正確になる、という結果です。後ろ向き単独でも前向き単独と同程度の性能が出たこと、そして両者を組み合わせると単独のどちらよりも良くなったことが報告され、前後の文脈が相補的であることが定量的に示されました。

Graves と Schmidhuber (2005) は TIMIT 音声コーパスのフレーム単位音素分類で、各構成を比較しました。報告された傾向として、双方向は1方向を上回り、LSTM は素朴な RNN を上回り、その両方を兼ねた BiLSTM が最良でした。フレーム単位の正解率で、BiLSTM が単方向 LSTM や双方向の素朴 RNN より明確に高いことが示され、「長期記憶」と「双方向文脈」が相補的に効くことが定量的に裏づけられました。これは「2つの良い工夫を足すと、それぞれの効果がほぼ独立に加算される」という、アーキテクチャ設計上の心強い知見です。

Huang et al. (2015) は、固有表現抽出(CoNLL-2003 英語、評価は F1 スコア。F1 は適合率(予測した固有表現のうち正しかった割合)と再現率(本当の固有表現のうち拾えた割合)の調和平均で、高いほど良い)で BiLSTM-CRF が約90前後の F1 を達成し、当時の手法と同等以上であることを報告しました。重要なアブレーション的知見として、CRF を外して BiLSTM 単体にすると F1 が低下し(各時刻を独立に分類するとラベル列の整合が崩れ、例えば B-PER の直後に I-LOC のような不正な遷移が起きる)、逆に双方向を外して前向き LSTM-CRF にしても低下する、と示されました。つまり「双方向で前後文脈を取る」ことと「CRF でラベル列の整合をとる」ことの両方が、それぞれ独立に効いているわけです。この相補性が BiLSTM-CRF が長く定番であり続けた理由です。また、この論文は手作りの素性(綴りの特徴など)への依存が少なく、頑健性が高いことも利点として挙げています。

## メリット・トレードオフ・限界
メリットは、各時刻で過去と未来の両方の文脈を使えるため、系列ラベリング(品詞タグ付け、固有表現抽出、音素分類)の精度が大きく上がる点です。既存の LSTM/GRU に双方向フラグを足すだけで使え、機械翻訳のエンコーダ側など「入力全体を一度に符号化する」用途でも標準的に有効です([Seq2Seq Encoder-Decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0497_seq2seq-encoder-decoder) のエンコーダは双方向にするのが定石でした)。

トレードオフと限界は明確です。最大の制約は、系列全体が揃わないと各時刻の最終表現を確定できず、リアルタイム(オンライン)推論に使えないことです。走行中のセンサ処理やライブ音声認識のように、まだ来ていない未来の入力を知り得ない設定では原理的に適用できません。また、前向き・後ろ向きの2本を持つため、パラメータと計算量がおよそ2倍になります。出力次元も2倍になり、後段の入力次元設計に注意が要ります。さらに、双方向は「ラベリング(各時刻に出力を出す)」には向きますが、「生成(系列を1つずつ作る)」のデコーダには使えません。生成中は未来がまだ存在しない(これから自分で作る)からです。

研究上の課題として、オンライン性とのトレードオフを緩める工夫が音声認識分野で研究されてきました。具体的には、短い未来窓だけ先読みする lookahead 付き RNN(数フレーム分だけ未来を待ってから出力する、わずかな遅延を許す)、系列を小さな塊に区切ってその塊内だけ双方向にする chunk-based / latency-controlled BiLSTM、後ろ向き情報を別途蒸留する手法などです。いずれも「許せる遅延」と「双方向で得られる精度」を交換するもので、完全な双方向の精度を遅延ゼロで得るのは本質的に不可能なままです。この「遅延 vs 精度」のトレードオフを設計でどう刻むかが、ストリーミング系タスクの腕の見せ所になります。

## 発展トピック・研究の最前線
双方向の発想は、注意機構を介して [Seq2Seq Encoder-Decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0497_seq2seq-encoder-decoder) のエンコーダに組み込まれ(双方向エンコーダ+注意デコーダ)、最終的に Transformer の自己注意へと一般化されました。自己注意は系列内の任意の2位置を直接結べるため、双方向性を自然に内包します(注意マスクをかけなければ全位置が全位置を見られる)。BERT 系の双方向事前学習モデルはこの延長線上にあり、ELMo(Peters et al., 2018)のように双方向 LSTM で文脈付き単語表現を作る研究が、その後の事前学習言語モデルの隆盛への橋渡しになりました。

一方、走行系の知覚や音声のストリーミングといったオンライン制約の強い領域では、双方向を使わず前向きのみの再帰や、未来を覗かない因果的(causal)な注意マスクを用いるのが定石です。「いつ双方向を使えて、いつ使えないか」「どれだけの遅延なら未来を覗いてよいか」を見極める設計判断そのものが、実務上の勘所になります。双方向 RNN は古典的な手法ですが、その「前後文脈を使う御利益」と「未来を待つ代償」という原理は、最新のストリーミング Transformer の設計にもそのまま受け継がれており、学ぶ価値は色褪せていません。
