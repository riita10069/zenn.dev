# アンカー軌道ボキャブラリ計画

## ひとことで言うと
「これから走り得る代表的な道筋」を何千本もあらかじめ用意しておき(これを軌道ボキャブラリ、軌道の単語帳と呼びます)、その中から「今の状況に一番合う 1 本」を選ぶ計画方式です。軌道を一から数値で書き出す回帰問題としてではなく、用意した候補のどれが良いかを当てる分類問題として運転計画を解きます。この発想は、連続値を直接出す手法が抱える「複数の正解を平均してしまう」問題を、離散化によって根本から回避するためのものです。

「アンカー(anchor)」とは「あらかじめ固定した基準となる候補」という意味です。「ボキャブラリ(vocabulary)」は言語処理から来た言葉で「選べる単語の一覧」のことです。ここでは軌道一本一本が単語にあたり、計画とは「いま発する単語を選ぶ」作業になります。

## 直感的な理解
レストランのメニューを思い浮かべてください。お客さんに「好きな料理を自由に作って注文してください」と言うと、ありえない組み合わせや調理不能なものが出てきます。代わりにメニュー(=ボキャブラリ)を用意し「この中から選んでください」とすれば、どれを選んでも必ず作れる料理が出てきます。アンカー軌道ボキャブラリはまさにこのメニュー方式です。あらかじめ「右に避ける」「左に避ける」「直進」「減速停止」といった実走行可能な軌道を何千本も用意し、ネットワークはその中から選ぶだけにします。

なぜこれが必要かは、連続回帰の弱点を見ると分かります。運転軌道をネットワークに出させる最も直接的な方法は回帰(regression)、つまり 64 ステップ分の (x, y) や (加速度, 曲率) を直接吐かせることです。しかしここに多峰性(multimodality)の問題が立ちはだかります。前方に障害物があれば右回避も左回避も正解です。回帰モデルを平均二乗誤差で学習すると、右(+2m)と左(-2m)の平均、すなわち「まっすぐ突っ込む(0m)」を出してしまいます。これが危険な mode averaging(モード平均化)です。メニュー方式なら、右と左の両方が高スコアになっても、選ぶのはどちらか 1 本なので、中間の危険な軌道は決して出てきません。

## 基礎: 前提となる概念
この手法を理解するには「回帰と分類の違い」が要です。回帰は連続値を当てる問題で、例えば「明日の気温は何度か」を 23.4 のように出します。分類は有限の選択肢から当てる問題で、「明日は晴れ・曇り・雨のどれか」を確率で出します。アンカー軌道ボキャブラリは、本来連続値である軌道を、あえて有限の選択肢に変換して分類問題として解きます。

「クラスタリング(clustering)」も前提です。これは似たものをグループにまとめる教師なし学習の手法で、代表例が k-means(データを k 個の重心に割り当てる)です。膨大な人間運転データから「だいたいこういう動きのグループ」を抽出し、各グループの代表軌道を 1 本ずつ取り出せば、ボキャブラリができあがります。

「交差エントロピー損失(cross-entropy loss)」は分類学習の標準的な目的関数です。正解クラスの予測確率を上げ、それ以外を下げるよう働きます。$\mathcal{L} = -\sum_i y_i \log p_i$ と書け、$y_i$ は正解なら 1・それ以外 0、$p_i$ はネットワークが出した候補 $i$ の確率です。軌道計画では「人間が実際に取った軌道に最も近い候補」を正解クラスとみなして学習します。

## 仕組みを詳しく
処理を段階的に追います。

第 1 段階(軌道ボキャブラリを作る)。大量の人間運転データから、実際に走られた軌道を集めます。各軌道は「将来 T 秒間の (x, y) 列」として表現します。これをクラスタリング(例えば k-means や最遠点サンプリング)で代表的な N 本に絞ります。N は数千本規模(典型的には 4096 本程度)が使われます。これが固定の単語帳で、学習中も推論中も変わりません。

第 2 段階(状況を特徴に変える)。カメラやセンサから認識した周囲の状況、すなわち他車・歩行者・地図・信号・自車運動状態を、ニューラルネットワークで特徴ベクトルにまとめます。ベクトル化(vectorized)表現を使う手法では、各エージェントや車線を点列・ポリラインとして符号化し、それらの関係をアテンションで集約して 1 つの状況特徴にします。

第 3 段階(各候補に確率を割り当てる = 分類)。ネットワークは、N 本の各候補軌道について「この状況でこの軌道を取る確率(スコア)」を出します。出力は長さ N の分布で、softmax を通せば全部足して 1 になります。これは「N クラスの分類」とまったく同じ形です。学習時は、人間が実際に取った軌道に最も近い候補を正解とみなし、その確率が高くなるよう交差エントロピーで学習します。候補ごとにスコアを出すので「確率的(probabilistic)計画」と呼ばれます。

第 4 段階(選ぶ)。推論時は、確率が最も高い候補を選ぶ、あるいは上位数本を取って衝突チェック・車線遵守チェックといった安全フィルタで絞り込む、という使い方をします。候補が有限なので、各候補に対して安全評価を事前計算しておくこともでき、出力の暴走を構造的に防げます。

形状で追うと、ボキャブラリは [N=4096, T, 2] のテンソル(固定)、状況特徴が [B, D]、出力スコアが [B, 4096] の分布です。この [B, 4096] の argmax(最大の位置)を取り、対応する固定軌道を引いてくれば、最終軌道 [B, T, 2] が得られます。計画の出力が「インデックスを 1 つ選ぶ」だけになるのが、この方式の最大の特徴です。

```
状況特徴 [B, D] ──┐
                  ├─→ スコア [B, 4096] ─→ argmax ─→ 固定軌道を引く ─→ [B, T, 2]
ボキャブラリ      ┘                                     ↑
[4096, T, 2] ───────────────────────────────────────────┘
```

## 手法の系譜と主要論文
アンカーの発想は予測の分野で生まれ、計画に移植されました。

源流は MultiPath(Yuning Chai, Benjamin Sapp, Mayank Bansal, Dragomir Anguelov; Waymo; CoRL 2019; arXiv:1910.05449)です。歩行者・車の将来軌道予測で「アンカー軌道集合 + 各アンカーへのオフセット回帰 + 各アンカーの確率」という三点セットを提案しました。動機は、単一の軌道回帰では多峰な未来を表せないこと。アンカーで大まかなモードを離散的に押さえ、そこからの微調整(残差回帰)で細部を表す、という二段構えです。後続の MultiPath++(Varadarajan et al., 2021)はアンカーの学習可能化などで予測精度をさらに高めました。

この予測側のアンカー発想を「自車の計画」に正面から持ち込んだのが VADv2(Shaoyu Chen, Bo Jiang, Hao Gao, Bencheng Liao, Qing Xu, Qian Zhang, Chang Huang, Wenyu Liu, Xinggang Wang; 2024; arXiv:2402.13243)です。提案は「人間運転データから作った数千本の軌道ボキャブラリに対し、各候補の選択確率を出す確率的計画」。動機は連続回帰の mode averaging を避け、不確実で多峰な運転意図を確率分布として明示的に表すこと。VAD(ベクトル化シーン表現の end-to-end 計画)の後継として、計画ヘッドを回帰から確率的分類へ置き換えた点が新規性です。

その後 Hydra-MDP(Zhenxin Li ら; 2024; arXiv:2406.06978)は、固定軌道ボキャブラリに対して複数の評価指標(衝突・走行可能領域・進行など)を教師としてマルチヘッドで蒸留し、各候補を多面的に採点する方向へ発展させました。NAVSIM 系のベンチマークで上位を占め、「ボキャブラリ + 多目的スコアリング」が強力な構成であることを示しました。

拡散計画との接続も重要です。[Diffusion Planner (拡散モデル計画)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0169_diffusion-planner) の DiffusionDrive は、純粋なノイズからではなくアンカー軌道の周りから拡散を始めます。これは「アンカーで大まかな多峰性を押さえ、拡散で連続的に微調整する」ハイブリッドで、離散ボキャブラリと連続生成の良いとこ取りを狙ったものです。アンカーボキャブラリは単独手法であると同時に、生成計画の良い初期化としても機能します。

## 論文の実験結果(定量データ)
VADv2 は CARLA の closed-loop ベンチマーク、特に Town05 で評価されました。closed-loop とは、planner の出力で実際に車を動かし、その結果生まれた新しい状況にまた planner が反応する、という連鎖をシミュレータ上で回す評価です(一度きりの予測を採点する open-loop より実運用に近く厳しい)。主要指標は Driving Score で、ルート完走率(進めたか)と違反ペナルティ(衝突・信号無視など)を掛け合わせた 0〜100 のスコアです。

報告によると VADv2 は Town05 の closed-loop 設定で当時最高水準の Driving Score を達成し、回帰ベースの先行手法を明確に上回りました。重要なのは、改善が単なる平均性能ではなく、衝突回避と運転の安定性に効いている点です。確率的分類により mode averaging が消え、危険な中間軌道が出なくなったことが、closed-loop での違反減少として現れたと解釈されています。

アブレーションでは、ボキャブラリの本数 N を変える実験が示唆的です。N を小さくしすぎると、表現できる軌道の解像度が荒くなり(離散化誤差が増え)、細かい操作を要する場面で性能が落ちます。逆に N を増やすと表現力は上がりますが、スコアリングの計算とメモリが N に比例して増え、収穫逓減も生じます。VADv2 系では 4096 本規模が実用上のバランス点として報告され、これより本数を減らすと Driving Score が明確に低下する一方、さらに増やしても伸びは緩やかになる、という典型的な飽和曲線が観測されています。また、計画ヘッドを回帰から確率的分類に変えるだけで closed-loop 性能が向上することが示され、mode averaging 回避という設計動機が定量的に裏づけられました。

ボキャブラリの作り方そのものもアブレーションの対象です。一様サンプリングや単純な k-means で作ったボキャブラリより、人間運転データの分布を忠実に反映した(走行頻度の高い領域を密に、稀な領域も取りこぼさない)ボキャブラリの方が、同じ N でも closed-loop での違反が少ない、という報告があります。これは「単語帳に含まれない軌道は原理的に出せない」という離散化の宿命を踏まえると自然な結果で、ボキャブラリ設計が学習やネットワーク構造と同じくらい性能を左右することを意味します。スコア最大の 1 本を取る argmax 選択ではなく、上位 k 本に対して衝突・走行可能領域チェックを掛けてから最終決定する事後フィルタを入れると、まれに最上位候補が安全違反を含む場面で救済が効き、Driving Score の分散が小さくなる(極端に悪いケースが減る)ことも示されています。

Hydra-MDP 系の研究は NAVSIM で PDMS(衝突・走行可能領域・進行・快適性を統合した 0〜100 のスコア)を評価指標に用い、固定ボキャブラリに多目的スコアリングを組み合わせることで 80 台後半の高い PDMS を報告しています。これは、ボキャブラリ方式が単に多峰性を扱えるだけでなく、各候補を安全指標で明示的に採点できる「説明可能性」と「安全フィルタの組み込みやすさ」を持つことの実証でもあります。

## メリット・トレードオフ・限界
メリット。多峰性を分類で素直に表現でき、危険な mode averaging を構造的に回避できます。候補が有限なので、各候補に衝突・車線遵守などの安全評価を事前計算でき、安全フィルタや事後選択をかけやすい。出力が暴走しにくく、なぜその軌道を選んだか(各候補のスコア)を見られるため説明性も高い。学習が分類問題になるので目的関数が安定しやすく、収束も比較的素直です。

トレードオフと限界。ボキャブラリが固定なので、単語帳にない細かい軌道は離散化誤差として原理的に出せません(N を増やせば緩和しますが計算・メモリが増えます)。ボキャブラリの作り方(クラスタリング手法、元データの分布)が性能を大きく左右し、訓練データに偏りがあればボキャブラリも偏ります。新しい運転環境や車両ダイナミクスに合わせて単語帳を作り直す手間もあります。研究上の未解決課題として、離散化誤差を抑えつつ計算量を抑える方法(アンカー + 残差回帰やアンカー + 拡散のハイブリッドが一つの答え)、シーンに応じてボキャブラリを動的に適応させる方法などが挙げられます。

## 発展トピック・研究の最前線
最も自然な発展は「アンカー + 連続微調整」のハイブリッドです。MultiPath が予測で行った残差回帰、DiffusionDrive が計画で行ったアンカー周りの拡散は、いずれも離散の粗さを連続生成で補う方向です。これにより、離散化誤差を抑えつつ多峰性と安全フィルタの利点を保てます。

もう一つは多目的スコアリングの精緻化です。Hydra-MDP のように衝突・進行・快適性を別々のヘッドで採点し、シミュレータの評価指標そのものを教師にする蒸留は、closed-loop 性能と評価指標の整合を高めます。さらに、言語指示や運転スタイルを条件にボキャブラリのスコアリングを変える指示追従計画、ボキャブラリを学習可能なクエリ集合として持ち端から端まで微分可能にする方向も研究が進んでいます。
