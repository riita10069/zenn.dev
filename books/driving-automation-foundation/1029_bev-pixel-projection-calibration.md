# BEV画素投影のキャリブレーション

## ひとことで言うと

自動運転では、軌道の各地点(自車から見たメートル単位の座標 (x, y))を、地図画像の画素位置 (row, col) に変換する必要があります。この変換には「1m が何画素か(pixels_per_meter)」「自車が画像のどの画素にいるか(ego_row, ego_col)」「行・列が前後・左右どちら向きで符号はどちらが正か」といった取り決めが必要です。これらを地図を描いた側(マップレンダラ)の規約と正確に一致させないと、変換がずれて下流の判定がすべて無意味になります。キャリブレーションとは、この数値合わせのことです。

## 直感的な理解

地図の上に自分の現在地と進路を重ねて表示する場面を考えてください。地図画像と現実の座標を正しく重ねるには、「縮尺(地図の 1cm が現実の何 m か)」と「自分が地図のどこにいるか(中心点)」が分かっていないといけません。縮尺を取り違えれば進路は地図上で短すぎたり長すぎたりし、中心点を間違えれば進路が全体に平行移動してしまいます。

BEV(真上から見下ろした表現)の画素投影も全く同じです。縮尺にあたるのが pixels_per_meter、中心点にあたるのが ego_row / ego_col。さらに「画像の上はどっちが前か」「右に進むと列番号は増えるか減るか」という向きの取り決めも要ります。これらは一見ささいですが、1つでも地図を描いた側とずれていると、軌道地点が現実とは全く違う画素に落ち、そこの色や意味を読んでも嘘の答えしか返りません。しかも返ってくるのはもっともらしい数字なので、間違っていることに気づきにくい。だからこそ厳密な合わせ込み(キャリブレーション)が決定的に重要なのです。

## 基礎: 前提となる概念

用語を噛み砕きます。

- BEV(Bird's-Eye View, 鳥瞰図): 真上から見下ろした 2D 平面の表現。地図やセンサ情報を上から見た格子(grid)として持ちます。
- 画素投影(pixel projection): 現実のメートル座標を、画像の画素座標へ写す変換。カメラの世界座標→画像座標の変換に似ていますが、BEV は上から見た平面なので、透視投影のような複雑さはなく、単純な拡大縮小+平行移動(アフィン変換)で書けます。
- pixels_per_meter: 1 メートルが画像上で何画素分かを表す解像度。0.4m/画素なら 1 画素 = 0.4m、逆に pixels_per_meter = 2.5。
- ego_row / ego_col: 自車(ego vehicle)が BEV 画像のどの画素位置にいるか。自車を原点 (0,0) とした座標系を画像に重ねるとき、原点がどの画素に来るか。
- 自車座標系(ego frame): 自車を原点とした座標系。通常 x が右(または前)、y が前(または左)など、プロジェクトごとに約束があります。この「どちらが x でどちらが前か」の約束が合っていないと符号がずれます。

なぜ厳密な一致が要るのか。BEV 画像から軌道地点の色や意味を読む処理(走行可能領域の判定など、[ラスタマップの色ルックアップによるDAC代理](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1034_drivable-area-color-lookup))は、まず軌道地点を正しい画素に写せて初めて意味を持ちます。投影パラメータが地図レンダラの規約とずれると、本当は「前方 12m の道路上」の点を、画像の全然違う画素(たとえば常に路外の隅)に写してしまいます。

## 仕組みを詳しく

変換の中身は概念的に次の通りです。自車中心の座標 (x_m, y_m)[m] を画素 (row, col) に写します。

```
col = ego_col + x_m * pixels_per_meter      (横方向)
row = ego_row - y_m * pixels_per_meter      (前後方向。上が前なら符号は -)
```

数値例で具体化します。よくある BEV 設計として「前方 120m / 後方 60m / 左右各 60m、解像度 0.4m/画素」を取ります。

- pixels_per_meter = 1 / 0.4 = 2.5 画素/m
- 縦は前方 120m + 後方 60m = 180m → 180 / 0.4 = 450 画素
- 横は左右各 60m = 120m → 120 / 0.4 = 300 画素
- よって地図画像は概ね 450 行 × 300 列
- 自車は前後非対称(前方 120m / 後方 60m)なので縦の中央ではなく後方寄り。後方 60m 分が下にあるなら ego_row = 60 / 0.4 = 150(下端からの数え方による)。横は中央なので ego_col = 300 / 2 = 150

ここで ego_row を誤って中央(225)と勘違いすると、自車位置が 75 画素 = 30m もずれ、すべての軌道地点が実際より 30m 後ろの画素を指してしまいます。たった1つの定数の取り違えで判定全体が壊れる、という具合です。

```
前方120m
   ↑
   │  ← ここに軌道地点が落ちるべき
 [自車] ego_row, ego_col
   │
   ↓ 後方60m
←60m─ 0 ─60m→   (横)
```

「合わせるべき相手」は、地図を画像として描く処理(マップレンダラ)の取り決めです。具体的に確認すべきは次の4点。(1) 解像度: 何 m/画素で描いたか。(2) 原点: 自車が画像のどの画素か。(3) 軸: 行と列がそれぞれ前後・左右のどちら向きで、符号はどちらが正か。(4) 向き: 画像の上が前方か後方か。これらをコード側の既定値と一つずつ突き合わせます。

なぜ符号や向きのミスが特に危険か。解像度の取り違えは「軌道が縮尺だけずれる」ので比較的気づきやすい一方、前後反転・左右反転は「形は正しいが鏡像・上下逆」になるため、テンソルの形状チェック(shape check)では絶対に検出できません。形状(450×300 など)は合っているのに中身が裏返っている、という最も発見しにくい種類のバグです。目視確認や実データでの検証が不可欠になります。

符号ミスを数値で追ってみます。「前方 12m、右 2m」の点を、正しくは `row = ego_row − y·ppm`(上が前なので前方は row が減る)で写すべきところ、誤って `row = ego_row + y·ppm` と前後の符号を逆にしたとします。ego_row=300、ppm=2.5 なら、正しい row = 300 − 12·2.5 = 270(画像の上の方=前方)。誤った row = 300 + 30 = 330(画像の下の方=後方)。前方の点が後方の画素に落ちます。地図は前方が道路・後方が建物の壁、という場面だと、本当は「道路上=適合」なのに「壁の上=非適合」と判定され、安全な軌道を不当に棄却します。逆の場面なら危険な軌道を見逃します。どちらも数字としてはもっともらしい (row=330 は画像内の妥当な画素) ので、例外も警告も出ません。これが silent failure(静かな失敗)の典型です。形状チェック・型チェックをすべて通過してしまう点が厄介さの核心です。

## 手法の系譜と主要論文

これは特定論文の手法というより、BEV 表現を扱うあらゆる研究で前提になる「座標系の取り決め」の問題です。BEV の解像度・範囲・原点をどう設計するかは、各論文が明示的に決めています。

- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270): BEV グリッド 200×200、知覚範囲 51.2m、セル約 0.256m。BEV クエリ(grid 上の各セルに対応する学習可能ベクトル)を明示的なグリッドとして定義し、カメラ特徴を投影で埋めます。投影には各カメラの内部・外部パラメータ(キャリブレーション)が必須で、ずれると BEV 全体が崩れる、という点で本トピックと同じ難しさを共有します。
- LSS: Lift-Splat-Shoot (Philion & Fidler, ECCV 2020, arXiv:2008.05711): カメラ画像を深度推定で 3D に持ち上げ、BEV に splat(撒く)手法。ここでも「どの 3D 点がどの BEV セルに落ちるか」の対応(=投影と解像度の取り決め)が中核で、対応が狂えば特徴が誤った位置に積まれます。
- BEVDet / BEVDepth (Huang et al., 2022 / Li et al., AAAI 2023, arXiv:2206.10092): 128×128、範囲 51.2m、セル 0.4m が代表的設定。0.4m/画素という解像度はこの系譜と整合します。BEVDepth は深度推定の精度が BEV 投影の正しさを左右することを定量的に示しました。
- VAD (Jiang et al., ICCV 2023, arXiv:2303.12077): 200×200、前後 60m / 左右 30m と非対称な範囲を採用。前後非対称な範囲では ego が画像中央に来ないため、原点画素の取り扱いに注意が要ることを示す例です。前方をより広く見る設計は計画タスクで自然ですが、その分 ego_row の誤りが起きやすくなります。

共通の教訓は、BEV 系手法は「解像度・範囲・原点・軸の向き」という座標規約が合っていて初めて意味を持ち、ここがずれると下流(知覚でも計画でもスコアリングでも)がすべて静かに壊れる、という点です。これらの論文がいずれもこれらの値を明記しているのは、再現性とキャリブレーションの正しさのためです。

## 論文の実験結果(定量データ)

キャリブレーションや解像度・範囲の選び方が性能にどう効くかを、関連研究の数値で見ます。

- BEV 解像度と性能のトレードオフ: BEVFormer 系の研究では、BEV グリッドを細かくする(セルサイズを小さくする)ほど物体検出の精度(nuScenes の mAP, mean Average Precision や NDS, nuScenes Detection Score)が上がる傾向が報告されています。たとえばグリッドを 100×100 から 200×200 に上げると NDS が数ポイント改善する一方、計算量とメモリは概ねセル数に比例して増えます。これは「解像度を上げれば投影の量子化誤差(連続座標を離散画素に丸める誤差)が減るが、コストが増す」という直接のトレードオフを表します。
- 深度・投影の誤差の影響: BEVDepth (arXiv:2206.10092) は、明示的な深度教師信号を加えてカメラの深度推定を改善すると、nuScenes の NDS が顕著に向上する(報告では数ポイント規模)ことを示しました。深度がずれれば 3D 点が誤った BEV セルに落ちる、つまり投影のキャリブレーションがそのまま下流精度を決める、という関係を定量的に裏づけています。
- キャリブレーションずれの影響(ロバスト性研究): センサキャリブレーション(外部パラメータ)に意図的に誤差を入れると BEV 検出性能が大きく劣化することが、複数のロバスト性ベンチマークで報告されています。たとえば外部回転に数度の誤差を入れるだけで mAP が数十パーセント低下する、という結果があり、「投影パラメータの小さなずれが大きな性能劣化を生む」ことを実証しています。本トピックの ego_row / 符号の取り違えも、これと同じメカニズムで下流を壊します。

数値の意味を補足します。mAP / NDS は物体検出の正確さを 0〜1(または %)で測る指標で、差が数ポイントでも順位やランキングが大きく入れ替わるほど競争が激しい領域です。そこで「キャリブレーション誤差で数十ポイント落ちる」というのは、事実上システムが使い物にならなくなるレベルの劣化を意味します。BEV 投影の正しさは、性能の前提条件であってチューニング項目ではない、ということです。

## メリット・トレードオフ・限界

メリット(正しく合わせられたとき)

- メートル↔画素の変換が単純な線形式(アフィン変換)で済み、計算が極めて軽い。
- 地図解像度(0.4m など)と原点・軸が決まれば一意に定まる、理解しやすい変換。
- 透視投影と違い深度の曖昧さがないので、平面内では誤差要因が少ない。

トレードオフ・リスク・限界

- パラメータ1つの取り違えで全判定が壊れる、しかも出力がもっともらしいため気づきにくい(silent failure)。
- レンダラ規約への依存が強く、レンダラ側を変更すると連動して直す必要がある。仕様の単一情報源(single source of truth)を持たないと食い違いが起きやすい。
- 前後非対称な範囲(前方 120m / 後方 60m など)では ego が画像中央に来ず、原点画素の誤りが起きやすい。
- 軸の向き・符号の取り決めミス(前後反転・左右反転)は形状チェックでは検出できず、目視・実データ検証が必須。
- 量子化誤差: 連続座標を離散画素に丸めるため、解像度が粗いと境界付近で判定がぶれる。

研究・実装上の未解決課題として、(1) レンダラとコンシューマで投影規約を共有する型・スキーマを設けて食い違いを構造的に防ぐ、(2) 既知の地物(車線中心など)を使って投影の正しさを自動検証(セルフチェック)する、(3) 量子化誤差を減らすため画素をまたぐ補間(bilinear sampling)でルックアップを連続化する、といった方向があります。

## 発展トピック・研究の最前線

オンライン地図生成(mapless)の流れでは、MapTR (Liao et al., ICLR 2023) や VectorMapNet (Liu et al., ICML 2023) がセンサからベクタ地図を直接推定します。この場合、地図はセンサと同じ座標系・キャリブレーションで生成されるため、事前 HD マップを別パイプラインでラスタ化するときに生じる「レンダラ規約とのずれ」問題が原理的に減ります。投影キャリブレーションのリスクをパイプライン設計で回避する一例です。

また、微分可能なサンプリング(differentiable grid sampling)を使い、軌道地点から地図特徴を読む処理を学習可能にする研究もあります。これにより投影の小さなずれを学習で吸収できる余地が生まれますが、一方で「学習がずれを隠してしまい、根本のキャリブレーションバグに気づけなくなる」という新たな落とし穴も指摘されています。キャリブレーションは自動運転の最も地味で、しかし最も事故に直結しうる基礎で、近年は安全性検証(verification)の文脈でも重視が高まっています。
