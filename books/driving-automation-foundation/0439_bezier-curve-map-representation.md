# ベジエ曲線地図表現

## ひとことで言うと
車線や道路境界のように緩やかに曲がった線を、たくさんの点を並べて表すのではなく、「ベジエ曲線」という滑らかな曲線の数式で、ほんの数個の制御点だけで表す方法です。少ないパラメータで自然なカーブを表現でき、しかも滑らかさが数式で保証されるのが利点です。地図を機械学習で出力するとき、「モデルに何を吐き出させるか」という出力表現の選択肢の一つで、点列(たくさんの座標を並べる方式)の有力な代替になります。

## 直感的な理解
あなたが手描きで道路の車線を地図に写すことを想像してください。素朴なやり方は、車線の上にたくさん点を打って、その点を順につなぐことです。点を細かく打てば滑らかに見えますが、点の数だけ座標を記録しなければなりません。一方、製図用のテンプレート(雲形定規)を当てて「ここからここへ、こう曲げて」と数個の指示で曲線を引く方法もあります。後者がベジエ曲線の発想です。

機械学習モデルに地図を出力させる場面でも同じ選択が起きます。モデルが車線1本を「20個の座標点」として吐き出すか(点列表現)、「4個の制御点」として吐き出すか(ベジエ表現)です。点列は柔軟ですが冗長で、点と点の間が折れ線になりがちです。ベジエは簡潔で、出力した瞬間に滑らかな曲線が確定します。この「簡潔さと滑らかさ」がなぜ嬉しいのか、そしてどこで点列に負けるのかが、このノートの中心テーマです。

## 基礎: 前提となる概念

まず用語をいくつか噛み砕きます。

- 車線(lane)・道路境界: 道路の上に引かれた白線や、走れる領域の縁。これらは直線ではなく、たいてい緩やかに曲がっています。
- BEV(Bird's-Eye-View、鳥瞰図): これは「真上から見下ろした視点」という意味です。自動運転では、カメラの斜めから見た映像を「真上から見た平面図」に変換して地図を扱うのが定番です。地図はこの平面上の2次元の線として表されます。
- 回帰(regression): これは「連続した実数値を予測すること」という意味です。たとえば「この制御点の x 座標は 3.2 メートル」のように、クラスを当てる分類(classification)ではなく、実数を当てるタスクを指します。
- 媒介変数(parameter)t: 曲線上の位置を表す 0 から 1 の数です。t=0 が曲線の始点、t=1 が終点。t を 0 から 1 へ動かすと、曲線上を端から端までなぞります。
- 制御点(control point): 曲線の形を決める少数の点です。曲線そのものは制御点を必ず通るとは限らず、制御点に「引っ張られて」形が決まります(始点と終点だけは必ず通ります)。
- 凸包(convex hull)性: ベジエ曲線は必ず制御点が作る凸多角形の内側に収まる、という性質です。曲線が制御点から飛び出して暴れないことが保証されるため、安全性を気にする自動運転で都合がよい性質です。

ベジエ曲線は 1960 年前後に、フランスの自動車メーカー Renault の Pierre Bézier と Citroën の Paul de Casteljau が、車体の曲面を数式で設計するために独立に考案したものです(de Casteljau のほうが先で 1959 年頃)。今では CAD やフォントの輪郭、画像編集ソフトのペンツールなど、滑らかな曲線が要るあらゆる場所で使われています。

## 仕組みを詳しく

3次ベジエ曲線(制御点4個、cubic Bézier)を例にします。曲線上の点は次の式で決まります。

```
B(t) = (1-t)^3 P0 + 3(1-t)^2 t P1 + 3(1-t) t^2 P2 + t^3 P3   （t は 0〜1）
```

ここで P0, P1, P2, P3 が制御点で、それぞれ (x, y) の2次元座標です。P0 が始点、P3 が終点、P1 と P2 が途中の曲がり具合を決めます。各係数 (1-t)^3 などは「バーンスタイン基底関数(Bernstein basis)」と呼ばれ、t に応じて4つの制御点の影響の混ざり方を決めます。この4つの係数は常に足すと1になり(重みの合計が1)、すべて0以上です。だから曲線は「4つの制御点を、場所ごとに違う割合で混ぜた加重平均」として書けます。これが先ほどの凸包性の正体です。t が小さいうちは P0 の影響が強く、t=1 に近づくと P3 の影響が強くなります。

具体的に数値を入れてみます。P0=(0,0)、P1=(1,2)、P2=(3,2)、P3=(4,0) とすると、

```
t=0.0 → B = (0, 0)        始点
t=0.25→ B = (0.86, 1.12)  立ち上がり
t=0.5 → B = (2, 1.5)      中間あたり、上に膨らむ
t=0.75→ B = (3.14, 1.12)  下りはじめ
t=1.0 → B = (4, 0)        終点
```

このように、4点 × 2次元 = わずか 8個の数値で、1本の滑らかな車線を表せます。次数 n のベジエは制御点 n+1 個で、n が大きいほど複雑な形を表せますが、出力すべき数値も増えます。

### 点列表現との比較

```
点列表現:    1本の車線 = 20点 × 2次元 = 40個の数値、折れ線、点数を事前固定
ベジエ表現:  1本の車線 = 4制御点 × 2次元 = 8個の数値、滑らか、密度は後から自由
```

点列表現の弱点を整理します。

- 滑らかさが保証されない: 点と点を直線でつなぐので、点が少ないとカクカクの折れ線になる。滑らかにするには点を増やすしかなく、出力すべき数値が増える。
- 点の個数を事前に固定する必要がある: 短い車線も長い車線も同じ20点で表そうとすると、短い車線では点が無駄に密集し、長い車線では粗くなる。
- 連続性をモデルが自力で学ぶ必要がある: 点はバラバラの数値なので、「本来一本につながっている」という制約が表現に組み込まれていない。隣り合う点が大きくばらつく失敗が起きうる。

ベジエはこれらを解決します。少ない制御点から数式で連続曲線が確定するので、出力数値が少なく、必ず滑らかです。さらに、出力後に t を細かく刻めば、好きな密度の点列を後から生成できます。表示用には密に、衝突判定用には粗く、と用途ごとに解像度を変えられます。

### モデル出力としての組み込み

地図を出力するモデルでは、Transformer の「クエリ(query)」という仕組みと組み合わせるのが定番です([DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr))。各クエリが地図要素1本を担当し、点列20点を出す代わりに制御点8個を出すようにします。出力テンソルの形でいえば、N 本の車線を予測するなら [N, 4, 2](N本 × 制御点4 × 座標2)という小さなテンソルで済み、点列の [N, 20, 2] より大幅に小さくなります。各クエリにはさらに「これは車線か、横断歩道か、それとも何もないか」という分類ヘッドが付き、地図要素の有無と種類を同時に当てます。

### 教師データの作り方

注意すべきは「正解の制御点」の作り方です。データセットには車線が点列(アノテーションされた座標列)で入っていることが多いので、それにベジエ曲線を最小二乗で当てはめて(curve fitting)、制御点に変換する前処理が要ります。点列 → ベジエの当てはめは、各点に対応する t を弧長で割り当て、バーンスタイン基底行列を作れば、最小二乗法で「点列と曲線の距離の二乗和を最小にする制御点」を解析的に(線形方程式を解くだけで)求められます。

損失関数の設計には2通りあります。一つは制御点そのものを直接比べる方式で、計算は速いですが「制御点が少しずれただけで形は大して変わらない」場合に過剰なペナルティを与えることがあります。もう一つは、予測した制御点から t を刻んで点列を生成し、それと正解点列のチャンファー距離(後述)を測る方式で、形の一致を直接最適化できますが計算がやや重くなります。どちらを使うかで学習の挙動が変わるため、これ自体が設計上の論点です。

## 手法の系譜と主要論文

STSU (Can, Liniger, Paudel, Chhatkuli, Van Gool; ETH Zürich, ICCV 2021, "Structured Bird's-Eye-View Traffic Scene Understanding from Onboard Images", arXiv:2110.01997) が、車線をベジエ曲線として回帰した代表的な研究です。1枚の車載カメラ画像から、BEV での車線のジオメトリと、車線どうしのつながり方(どの車線がどの車線に合流・分岐するか、という有向グラフ=連結性)を同時に推定しました。各車線は DETR 型 Transformer のクエリから「制御点列」として出力されます。動機は、車線は本質的に滑らかな曲線であり、ベジエなら少パラメータで滑らかさを保証でき、集合予測(set prediction、N個の要素を順不同の集合として一度に出すこと)と相性が良いからです。新規性は、車線のジオメトリと連結性(交差点でのつながり)をひとつのネットワークで構造的に出した点にあります。

続く TPLR (Can et al., CVPR 2022, "Topology Preserving Local Road Network Estimation from Single Onboard Camera Image") は STSU を発展させ、車線の交差・合流の整合性(あるべきつながりが切れない、ないつながりが生まれない)をより明示的に保つ損失と評価を導入しました。これはベジエで「形」を出すだけでなく「トポロジー(つながりの構造)」を保つ方向の研究です。

BeMapNet (Qiao, Ding, Cao, et al.; CVPR 2023, "End-to-End Vectorized HD-map Construction with Piecewise Bezier Curve") は、地図要素を「区分的(piecewise)なベジエ曲線」、つまり複数の3次ベジエをつないだ曲線として直接出力する end-to-end 手法です。1本の低次ベジエでは表しきれない複雑な形を、短いベジエを連結することで滑らかさを保ったまま表現します。各曲線の「区分の数」も予測対象にし、形の複雑さに応じて柔軟に表現力を割り当てます。これは「ベジエ表現を本格的にオンライン地図構築へ持ち込んだ」研究と位置づけられます。

一方、MapTR (Liao, Chen, Wang, et al.; Horizon Robotics / 華中科技大学, ICLR 2023, arXiv:2208.14437) とその拡張 MapTRv2 (arXiv:2308.05736) は、地図要素を「等間隔の点列」として DETR 型クエリで直接ベクトル予測しました。MapTR はベジエではなく点列を採用しましたが、「順序付き点集合の置換等価な対応づけ(permutation-equivalent point-set matching)」、つまり点列の始点・向き・並び順の任意性を吸収する工夫で、点列の冗長さと曖昧さを抑えつつ高速・高精度を達成しました。これは「ベジエ vs 点列」の論点で点列側を強力に推し進めた研究で、ベジエ表現はこれと対比されることが多いです。

LaneGAP (Liao et al., 2023, "Lane Graph as Path: Continuity-preserving Path-wise Modeling for Online Lane Graph Construction", arXiv:2303.08815) は、車線を「つながった経路(path)」として扱い、点列の柔軟さとパラメトリック曲線の連続性のバランスを取る方向を示しました。

道路設計の文脈では、クロソイド曲線(clothoid、オイラー螺旋)も研究されています。クロソイドは曲率が弧長に比例して一定の割合で変化する曲線で、実際の道路や鉄道の緩和曲線に使われます。車両の運動学(ハンドルを一定速度で切ると軌跡はクロソイドに近い)に合うため物理的に自然ですが、ベジエより数式が複雑で機械学習に組み込みにくい面があります。ベジエは「物理的厳密さは劣るが計算が単純」という位置づけです。

## 論文の実験結果(定量データ)

地図構築の評価には主に mAP(mean Average Precision、平均適合率)が使われます。これは「予測した地図要素が正解とどれだけ重なるか」を、しきい値を変えながら測った精度の平均です。車線(divider)・道路境界(boundary)・横断歩道(pedestrian crossing)のそれぞれで計算し、平均します。値が高いほど良く、0〜100% で表します。重なりの判定には「予測曲線と正解曲線のチャンファー距離(Chamfer distance、2本の線の点同士の最近傍距離を双方向に測った平均)」がしきい値(0.5m, 1.0m, 1.5m)以内かどうかを使います。なぜチャンファー距離かというと、線の予測は点単位の一致ではなく「線同士がどれだけ近いか」で測るのが自然だからです。

代表的な数値の流れを nuScenes(ボストンとシンガポールで集めた大規模多センサ走行データセット)上で追うと、初期の HDMapNet 系がおよそ mAP 20〜30% 台だったのに対し、MapTR(カメラのみ)はおよそ 50% 前後、MapTRv2 は補助タスクや学習の改良で条件により 60% 台へと改善しました。これは「セグメンテーション後にベクトル化する後処理方式」から「曲線・点列をクエリで直接出す方式」への移行で精度が跳ね上がったことを示します。

ベジエ表現そのものの効果として、BeMapNet は nuScenes 上で、点列ベースの強力なベースライン(MapTR 系)と同等以上の mAP を、より少ないパラメータ数の出力で達成したと報告しています。報告によれば、区分ベジエの「区分数」を変えるアブレーションで、区分を増やすほど複雑形状(横断歩道のコーナーや急カーブ)の精度が上がる一方、単純な直線車線では少ない区分で十分で、過剰な区分はかえって学習を難しくする、という傾向が見られます。STSU 系でも「点列に比べて出力次元が減り、滑らかさが保証される」ことを示しましたが、1本の低次ベジエでは複雑形状で点列に劣る場面が報告されています。

アブレーション的に整理すると、ベジエの次数(滑らかさ)を上げる、あるいは複数本を連結すると表現力は上がるものの、出力次元が増え、点列に対する簡潔さの利点が薄れていきます。ここに「次数(滑らかさ)と表現力のトレードオフ」が定量的に現れます。実務では「3次ベジエの区分連結」あたりが、滑らかさ・表現力・学習しやすさのバランス点としてよく選ばれます。

## メリット・トレードオフ・限界

- メリット: 少ないパラメータ(制御点数個)で滑らかな曲線を表現でき、出力次元が小さい。学習も収束しやすい傾向。
- メリット: 滑らかさが数式で保証される。点列のようにカクカクにならず、凸包性で曲線が暴れない。
- メリット: 出力後に t を刻む密度を自由に変えられ、任意解像度の点列を後から生成できる。長さの違う車線を統一的に扱える。
- トレードオフ: 1本の3次ベジエでは表現力に上限があり、急なヘアピンカーブや複雑形状は近似が粗くなる。区分連結が必要になり、区分数の予測という新たな難しさが生じる。
- トレードオフ: 制御点の意味が点列より直感的でなく、教師データ(正解の制御点)を作る際にベジエ当てはめの前処理が要る。当てはめ時の t の割り当て方しだいで制御点がぶれる。
- 限界・未解決課題: ベジエは車両の運動学を直接は反映しない(クロソイドのほうが物理的)。また、車線の交差・分岐(連結性=トポロジー)はベジエ単体では表せず、別途グラフ構造を出す必要がある。地図の「形」と「つながり」を統一的に滑らかに表す表現は今も研究途上です。

## 発展トピック・研究の最前線

近年は「車線をグラフ(節点と辺)として出す」研究(lane graph / lane topology)が活発で、ベジエやスプラインで辺の形を表しつつ、ノード間の接続を Transformer で推定する流れがあります。TopoNet や LaneSegNet など、形とトポロジーを同時に出す手法が登場し、評価指標も mAP に加えてトポロジー一致を測る指標(TOP score など)が併用されるようになっています。

表現としては、ベジエの上位互換である B-スプライン(B-spline、区分的に多項式をつなぐ曲線で、局所的な制御点の変更が曲線全体に波及しない)や NURBS を使う研究もあり、より複雑な形を局所的な制御で表せます。また、地図を陽に出さず、運転計画に必要な特徴だけを潜在表現として持つ end-to-end 手法も増えており([オンライン vs オフライン地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0456_online-vs-offline-mapping)、[マップフリー運転](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0429_map-free-driving))、「そもそも幾何曲線を明示的に出力すべきか」という問い自体が議論されています。「どの曲線族が、滑らかさ・表現力・学習しやすさ・下流タスクへの渡しやすさのバランスで最良か」は、分野の継続的な問いです。
