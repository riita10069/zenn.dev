# Overpass API / OSMnx

## ひとことで言うと
世界中の地図データが集まっている「OpenStreetMap (OSM)」という無料の地図データベースから、指定した場所まわりの道路網をプログラムで取ってくるための道具立てです。Overpass API は OSM のデータを問い合わせるサーバ窓口で、OSMnx はそれを Python から簡単に扱い、道路を「点と線のネットワーク(グラフ)」として手元に取り込むライブラリです。自動運転や交通解析で「ナビ用の地図画像」や「経路探索の土台」を作るときの、いちばん最初の「データ取得」段階を担います。

## 直感的な理解
自動運転モデルにカメラ画像だけを見せても、「次の交差点を右折」のような目的地までの経路情報は分かりません。人間が運転するときにカーナビを見るように、機械にも「今どの道を走っていて、これからどの道を進むのか」を伝える追加の入力が要ります。この入力をよく「自車を中心にした俯瞰図(BEV = Bird's Eye View、鳥が真上から見下ろしたような視点)のナビ地図画像」として与えます。

その地図画像を作るには、まず「その場所にどんな道路があるか」という生データが要ります。これを自分で測量するのは非現実的なので、世界中のボランティアが作っている無料の地図データベース OpenStreetMap を使うのが定石です。問題は、OSM の生データが膨大で(全世界の `planet.osm` は圧縮済みでも 100GB を超えるオーダー、展開すると数 TB)、しかも独特の XML / PBF 形式で配布されている点です。「この緯度経度から半径 800m の車道だけ欲しい」という用途に対して、全データをダウンロードするのは無駄が大きすぎます。

そこで2つの道具が登場します。Overpass API は「この範囲のこの種類の地物だけください」という絞り込み問い合わせを受け付ける、OSM 専用の検索サーバです。必要な部分だけをサーバ側で切り出して返してくれるので、手元には小さなデータだけが届きます。OSMnx は、その問い合わせ・受け取ったデータのグラフ化・描画・経路探索までを数行で書けるようにまとめた Python ライブラリです。両者は「データの貯蔵庫と検索窓口(Overpass)」と「それを扱う台所道具(OSMnx)」の関係にあります。

## 基礎: 前提となる概念

ここで言う「API」とは Application Programming Interface の略で、「プログラムから機能を呼び出すための窓口」のことです。Overpass API は HTTP でクエリ文字列を送ると、該当データを返してくれる Web サービスです。クエリは Overpass QL という独自の問い合わせ言語で書きます。たとえば「ある矩形範囲(bounding box)の中の、`highway` タグを持つ全ての way(線)を返せ」という指示を1行で表現できます。

OSM のデータモデルは3種類の要素でできています。node(緯度経度を持つ単一の点)、way(node を順に並べた線または閉じた多角形。道路・建物の輪郭など)、relation(複数の way/node をまとめた論理的なグループ。バス路線・行政区画など)です。さらに各要素には key=value 形式の「タグ」が付きます。道路で重要なのは `highway` タグで、`highway=motorway`(高速)、`highway=residential`(生活道路)、`highway=footway`(歩道)のように道路の種別を表します。OSMnx の `network_type` 引数は、内部的にこの `highway` タグの値をどう絞るかに対応しています。

「グラフ」とは、数学でいう「ノード(node = 点)」と「エッジ(edge = 点と点をつなぐ線)」の集まりのことです。道路網に当てはめると、ノードは交差点や道の曲がり角(緯度経度を属性として持つ)、エッジはノード間をつなぐ道路区間(長さなどの属性を持つ)になります。道路網を点と線のネットワークとして表すと、後で「A 地点から B 地点までの最短経路」のような計算が、グラフ理論のアルゴリズム(後述のダイクストラ法など)でそのまま解けるようになります。

OSMnx の名前は「OpenStreetMap + NetworkX」から来ています。NetworkX は Python の代表的なグラフ計算ライブラリで、OSMnx が返す道路網はこの NetworkX のグラフオブジェクトとして表現されます。具体的には `MultiDiGraph` という型がよく使われ、「Di」は有向(Directed、一方通行を表現できる)、「Multi」は同じ2点間に複数の道(例: 並走する側道、往復路)を持てる、という意味です。

OSM のデータは VGI(Volunteered Geographic Information、ボランティアが作る地理情報)です。誰でも編集できるため、都市部は詳細ですが、田舎やマッピングが進んでいない地域では道が欠けていたり古かったりします。この品質の地域差は使う上で常に意識すべき前提です。

## 仕組みを詳しく

典型的な使い方は、中心点と半径を指定して道路網グラフを取得する1行です。

```python
import osmnx as ox
graph = ox.graph_from_point(
    (center_lat, center_lon),  # 中心の緯度・経度
    dist=800,                  # 半径800メートル
    network_type="drive",      # 車が走れる道だけ
)
```

ここで起きていることをステップで分解します。

1. `graph_from_point` が、中心点(例: 緯度 48.8566, 経度 2.3522)から半径 800m の四角い範囲(バウンディングボックス)を計算する。緯度1度はおよそ 111km、経度1度は緯度によって縮むので、内部で球面上の距離を考慮して bbox を緯度経度に変換する。
2. その範囲の「車道(drive)」だけを取るための Overpass クエリ文字列を自動生成し、Overpass API サーバへ HTTP で送る。`network_type` には他に `walk`(歩道込み)、`bike`(自転車)、`drive_service`(私道・駐車場通路込み)、`all`(全部)などがある。
3. サーバが該当する道路データ(道の形を表す node 列、交差点、way の属性)を返す。
4. OSMnx がそれを `MultiDiGraph` に変換する。OSM の way は「交差点でない中間 node」も多数含むため、OSMnx は標準で「次数2の中間ノードを潰してエッジに畳み込む」簡略化(topology simplification)を行い、交差点だけをノードに残す。これにより、何百もの形状点を持つ長い道路が1本のエッジ(形状は `geometry` 属性に保持)になり、経路計算が軽くなる。一方通行はエッジの向きで、ロータリーや側道は多重エッジで表現される。

取得後のグラフは、半径 800m 程度なら数百〜数千ノード規模になります。各ノードは経度を `x`、緯度を `y` として属性に持ち、各エッジは区間長 `length`(メートル)、道路種別 `highway`、制限速度 `maxspeed`、一方通行フラグ `oneway` などの属性を持ちます。このグラフから、典型的には次の3つの使い道があります。

- 道路の線を地図画像に描く(全エッジを線分・曲線として描画)。`ox.plot_graph` がそのまま描画関数を提供する。
- GPS 軌跡をこのグラフ上の経路に対応付ける(マップマッチング。詳細は [マップマッチング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0420_map-matching))。
- 2 ノード間の経路を `networkx.shortest_path(graph, src, dst, weight="length")` で求めて線として描く。内部はダイクストラ法。

ナビ地図画像を作る場合は、最終的に `(3, H, W)` 形状(チャンネル = RGB の 3、高さ H、幅 W)の画像テンソルに落とします。自車を画像中心に固定し、進行方向を上に向けて(ego-centric, heading-up)描くと、モデルにとって扱いやすい一定の座標系になります。緯度経度のままだと地球は球面なので、ローカルな平面に投影する必要があります。近距離なら UTM(Universal Transverse Mercator、地域ごとに地球を平面に切り取る投影法)に変換し、メートル単位の x-y 座標で扱うのが定石です。OSMnx には `ox.project_graph` という投影関数があります。

```
緯度経度のグラフ          UTM投影(メートル)        自車中心・heading-up 画像
  node: (lat, lon)   →    node: (easting, northing) →  ┌──────────┐
  edge: way の形状点        edge: メートル長            │    ↑進行   │
                                                       │  ──┼──道路 │  (H, W)
                                                       │    ●自車   │
                                                       └──────────┘
```

注意点として、ネットワーク取得は秒オーダーで遅く、インターネット接続が必須です。そのため学習中にリアルタイムで毎回 Overpass を叩くのではなく、事前にまとめて取得・キャッシュして使うオフライン前処理として組むのが定石です。OSMnx はローカルにレスポンスをキャッシュする仕組みを内蔵しており(`ox.settings.use_cache`)、同じ範囲の再取得を避けられます。大量取得時は、公共の Overpass サーバへの負荷を避けるため、自前の Overpass インスタンス(`wiktorn/overpass-api` のような Docker イメージや、OSM の地域抽出ファイル `.osm.pbf` から構築)を立てるのが実務的です。

## 手法の系譜と主要論文

OSMnx 自体が査読付き論文として発表されています。

- Geoff Boeing, "OSMnx: New methods for acquiring, constructing, analyzing, and visualizing complex street networks" (Computers, Environment and Urban Systems, vol. 65, 2017)。この論文は「OSM から都市の道路網を一貫した手順でグラフとして取得し、トポロジー(つながり方)や形状を解析・可視化する方法」を提案しました。動機は、それまで研究者ごとにバラバラだった道路網データの取得・整形作業を、再現可能な1つのツールに統一することでした。効果として、誰でも数行で世界中の任意都市の道路網を同じ品質で取得できるようになり、都市解析(街路の方位分布、交差点密度、回り道率など)の研究を一気に標準化しました。トレードオフは、データ品質が OSM の整備状況に依存する点と、Overpass サーバへの負荷・速度です。

- OpenStreetMap / Overpass の背景については、Mordechai Haklay & Patrick Weber, "OpenStreetMap: User-Generated Street Maps" (IEEE Pervasive Computing, vol. 7(4), 2008) が代表的な紹介です。「ボランティアが作る地図(VGI)」という考え方と、その品質が地域によって大きく違うことを論じました。Haklay は別論文 "How good is volunteered geographical information? A comparative study of OpenStreetMap and Ordnance Survey datasets" (Environment and Planning B, vol. 37(4), 2010) で、イギリスの OSM 道路網を公式地図(Ordnance Survey)と重ね合わせ、ロンドン都心では位置誤差が概ね 6m 程度で公式地図とよく一致するが、地方ではカバレッジが大きく落ちることを定量的に示しました。

- 自動運転の文脈では、OSM のような既存(オフライン)地図と、車載センサからその場で作る地図を組み合わせる研究が多くあります。Teddy Ort, Liam Paull, Daniela Rus, "Autonomous Vehicle Navigation in Rural Environments without Detailed Prior Maps" (MapLite, ICRA 2018) は「詳細な HD マップを使わず、OSM のような粗いトポロジー地図と LiDAR だけで田舎道を自動運転する」ことを示し、舗装路で 100km 超を介入なしで走破しました。重い事前地図に頼りすぎない方向性を打ち出した代表例で、OSM をナビ用の粗い経路地図として使い、詳細な周辺認識はセンサに任せるという発想と相性が良いものです。

## 論文の実験結果(定量データ)

OSMnx の評価は、画像認識のような単一スコアではなく「再現性と網羅性」で測られます。Boeing (2017) は、ツールを使ってアメリカの全都市(2万7千を超える自治体)の街路網を一括取得し、街路の総延長・ノード数・平均ノード次数(1交差点に集まる道の本数)・回り道率(circuity、直線距離に対する実際の道のり距離の比)といった指標を一貫した方法で算出できることを示しました。たとえば回り道率が 1.0 に近い都市は格子状に整然とした街路、1.0 から離れるほど曲がりくねった街路を意味します。重要なのは、これらの指標が「誰が計算しても同じ手順で同じ値になる」ようになった点で、それまで各研究室が独自に手作業で前処理していた状況を一掃しました。

データ品質の定量例としては、Haklay (2010) が、ロンドンでの OSM と公式地図の位置ずれが平均 6m 程度であること、ただし OSM のカバレッジ(道路総延長の網羅率)が都心では 9 割超なのに対し、整備の遅れた地域では数割に落ちることを報告しています。この「都市は良いが地方は薄い」という偏りは、OSM をデータソースに使うあらゆる下流タスクの精度に直接効いてきます。たとえば、自動運転のナビ地図を OSM から作るとき、都市部では問題なくても郊外で道路が欠けていれば、経路線そのものが現実とずれます。

MapLite (Ort et al., 2018) は、OSM の粗い地図と LiDAR だけで田舎の未舗装路を含むコースを自律走行し、舗装路区間で 100km を超える距離を人間の介入なしで走り切ったと報告しています。これは「詳細な HD マップが無くても、粗い OSM 地図+リアルタイム認識で走れる」ことの実証で、OSM の使い方として「精密な周辺地図」ではなく「目的地への経路ヒント・トポロジー」に役割を限定すれば十分機能する、という設計判断を裏づけます。逆に言えば、OSM 単体で車線レベルの精密制御まで行うのは無理がある、という限界の確認でもあります。

## メリット・トレードオフ・限界

メリット
- 無料で全世界の道路網が手に入り、自前の地図整備コストがゼロ。
- OSMnx により取得・グラフ化・投影・描画・経路探索が数行で完結し、再現性が高い。
- 範囲・道路種別を指定して必要分だけ取れるので軽量。簡略化(topology simplification)で交差点グラフが小さくなり、経路計算が速い。

トレードオフ・限界
- ネットワーク取得が遅く(秒オーダー)、インターネット必須。学習中のリアルタイム取得には不向きで、オフライン前処理+キャッシュが前提。
- データ品質が地域依存。都市部は詳細だが、田舎やマッピングが進んでいない地域は道が欠けていたり古かったりする。鮮度も保証されず、新設道路・工事の反映には遅れがある。
- Overpass は公共の共有サーバなので、過度な連続アクセスはレート制限される。大量取得時はキャッシュやローカルサーバ運用が必要。
- 車線数・信号・横断歩道などの詳細属性は OSM に必ずしも入っておらず、入っていても精度・タグ付けの一貫性が一定しない。HD マップ(高精度地図、車線形状を cm オーダーで持つ)の代替にはならず、あくまで「粗いナビ地図・経路トポロジー」用途。

研究上の未解決課題としては、OSM のような VGI の品質を自動で評価・補完する手法(衛星画像や走行ログからの自動マッピング、欠損道路の補完)が活発に研究されています。OSM をナビ地図として使う場合、地図と現実のずれ(工事・新設道路)をオンライン認識でどう吸収するかが、地図依存を減らす自動運転研究の中心課題の一つです。

## 発展トピック・研究の最前線
- 衛星・航空画像からの自動道路抽出(RoadTracer, Bastani et al., CVPR 2018 / Sat2Graph, He et al., ECCV 2020 など)で OSM の欠損を補い、グラフ構造を直接生成する研究。
- OSM の粗い地図と車載センサのオンライン地図を融合する「prior-aided online mapping」。SD マップ(OSM 相当の Standard Definition map)を弱い事前知識としてオンライン HD マップ推定に注入する手法(P-MapNet など)が増え、HD マップへの依存を減らしつつ精度を保つ方向に進んでいる。
- 大規模走行ログからの自動 HD マップ生成・更新。OSM はその初期トポロジーの種(seed)として使われることが多い。
