# Concat Fusion(連結融合)

## ひとことで言うと
車に付いた複数のカメラ(例えば前後左右の数台〜8台)それぞれが作った「特徴」を、ただ束ねて一本につなげるだけの、いちばん素朴な合わせ方です。とても軽くて単純ですが、カメラがどこを向いているかという幾何情報を一切考えません。そのため、性能の「下限の物差し(ベースライン)」として研究で広く使われます。「concat」は英語の concatenate(つなげる)の略です。

## 直感的な理解
人間が複数の防犯カメラの映像を見て状況を判断する場面を想像してください。一番単純なやり方は、4 つのモニタを横一列に並べて全部一度に眺めることです。これは「とりあえず全部の情報を横に並べる」やり方で、楽ですが、別々のモニタに同じ人が映っていても「これは同じ人だ」と自動で結びつけてはくれません。Concat Fusion はこれと同じで、複数カメラの特徴を機械的に横並びにするだけです。賢い対応づけは一切しませんが、確実に「全部の情報がそこにある」状態は作れます。

なぜこんな単純な手法が大事なのでしょうか。後でもっと賢い合わせ方(クロスアテンションや BEV 融合)を試すとき、「その工夫が本当に効いたのか」を測る基準が必要だからです。新しく提案した複雑な手法が、この素朴な連結に勝てなければ、その複雑さは無駄だったことになります。研究では、強い主張をするほど強いベースラインを置く必要があり、連結はその最も基本的な土台です。

## 基礎: 前提となる概念
- 特徴(feature): ニューラルネットワークは画像をそのまま扱わず、まず「特徴」という数値のテンソル(多次元配列)に変換します。特徴は「画像のどこにどんなパターン(エッジ、色、物体らしさ)があるか」を数値で表したものです。
- テンソルとチャネル: 画像特徴はふつう [C, H, W] の 3 次元テンソルで表されます。C はチャネル(特徴の種類の数)、H・W は空間の縦横の解像度です。RGB 画像なら C=3(赤・緑・青)ですが、ネットワーク内部では C が数百に増えます。
- バックボーン(backbone): 画像から特徴を抜き出す土台ネットワーク(ResNet, Swin Transformer など)。各カメラ画像をこれに通して特徴を得ます。
- 融合(fusion): 複数の特徴を 1 つにまとめる工程。複数カメラを使うとカメラ台数ぶんの特徴ができるので、最終判断(どこに何があるか、どう運転するか)につなげるには融合が必要です。マルチモーダル学習のサーベイ(Baltrušaitis ら 2019)では、融合を「どの段階で合わせるか」で early fusion(生に近い早い段階で混ぜる)/ intermediate(feature-level)fusion(中間特徴を束ねる)/ late fusion(各々が出した判断を最後に合わせる)に分類します。Concat は intermediate fusion の最も単純な実装です。
- 1×1 畳み込み(pointwise convolution): 空間位置(H, W)はそのままに、各位置のチャネル方向だけを線形変換する小さな畳み込み。チャネル数の圧縮・拡張に使います。連結後の次元削減に頻出します。
- ベースライン(baseline): 性能比較の基準。手の込んだ手法が単純な手法に勝てなければ、その工夫は無駄だったと判断できます。研究では必ずベースラインを置きます。

## 仕組みを詳しく
核心は「チャネル方向に連結する」ことです。具体的なテンソル形状で追います。バックボーンが各カメラ画像から次の特徴を出したとします。
- 1台のカメラの特徴: 形状 [C, H, W] = [256, 8, 8]
  - C=256: チャネル数(特徴の種類が 256 個)
  - H=8, W=8: 空間解像度(8×8 マスに圧縮)

カメラが 8 台(V=8)あると特徴が 8 個できます。
```
カメラ0の特徴: [256, 8, 8]
カメラ1の特徴: [256, 8, 8]
   ...
カメラ7の特徴: [256, 8, 8]
```
Concat はこれらをチャネル軸(C)に積み重ねます(PyTorch なら torch.cat(features, dim=チャネル軸))。
```
連結前: [256, 8, 8] が 8 個
連結後: [256 × 8, 8, 8] = [2048, 8, 8]
```
8 台ぶんの 256 チャネルがそのまま横並びになり、合計 2048 チャネルの 1 つの大きな特徴になります。空間の縦横(8×8)は変わりません。その後、1×1 畳み込みで扱いやすい次元(例: 256)に圧縮し、後段の判断ネットワークに渡すのが定石です。

ポイントは、連結作業そのものに「学習するパラメータ」が一切要らないことです(続く 1×1 畳み込みには少量のパラメータがあります)。計算もメモリも軽く(チャネルを並べるだけ)、バグも入りにくいので、ベースラインとして信頼できます。連結は微分可能で勾配がそのまま各ブランチへ分配されるため、end-to-end 学習にもそのまま組み込めます。

設計上の注意点:
- 連結はカメラの並び順を固定で扱います。「チャネル 0〜255 がカメラ0、256〜511 がカメラ1」という対応をネットワークが暗黙に覚えるため、カメラの取り付け位置・向きが変わると意味が崩れ、再学習が必要になります。順序に対する不変性(permutation invariance)を持ちません。
- 別カメラに同じ物体が写っていても、Concat はそれが「同じ物体」だと教えません。あくまで別々の塊として横に並べるだけです。隣り合うカメラの視野が重なる領域(overlap)の冗長情報を統合できず、二重カウントや矛盾を解決できません。
- チャネル数が台数倍に膨らむため、後段の畳み込み・全結合層のパラメータと計算が増えます。

なお「マルチスケール特徴の連結」という別の使い方もあります。1 台のカメラでも、バックボーンは複数の解像度の特徴(細かい特徴と粗い特徴)を出します。これらを同じ大きさにそろえてからチャネル方向に連結し、多重解像度の情報を 1 つにまとめる、という使い方です。ビュー間融合(カメラ同士)とスケール間融合(解像度同士)で、どちらも「連結」という同じ操作を使います。U-Net のスキップ接続や DenseNet も、内部的にはこの連結を多用しています。

## 手法の系譜と主要論文
Concat は特定の論文が提案した手法ではなく、深層学習のあらゆる場面で使われる基本操作です。系譜としては「融合をどう設計するか」という大きな流れの中の出発点に位置します。
- MV3D (Chen et al., CVPR 2017, arXiv:1611.07759): LiDAR の前面図(front view)・BEV とカメラ画像の特徴を融合する初期のマルチビュー 3D 検出。ROI(関心領域)ごとに各ビューの特徴を結合(deep fusion)する設計で、特徴連結が 3D 検出に使われた代表的初期例。
- AVOD (Ku et al., IROS 2018, arXiv:1712.02294): MV3D を発展させ、画像と BEV の特徴を領域提案段階で融合。やはり中間特徴の結合が骨格で、連結ベース融合の有効性と限界を示しました。
- マルチモーダル学習サーベイ(Baltrušaitis et al., TPAMI 2019, arXiv:1705.09406): early / intermediate / late fusion の整理を定番化した論文。Concat を融合設計の地図の上に位置づける際の参照点です。
- BEVFusion (Liu et al., ICRA 2023, arXiv:2205.13542): カメラ BEV と LiDAR BEV を同じ BEV 空間にそろえてから連結し、畳み込みで混ぜる手法。ここでも最終段の融合は「連結 + 畳み込み」です。重要なのは、論文の主眼が「連結する前に各特徴を共通座標系(BEV)へそろえること」にある点です。連結そのものは賢さの源ではなく、座標合わせが効いています。
- Attention Is All You Need (Vaswani et al., NeurIPS 2017, arXiv:1706.03762): 連結ではなくアテンションで「関係する部分を選んで混ぜる」枠組み([Cross-Attention Fusion(クロスアテンション融合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0318_cross-attention-fusion))。Concat の限界(対応づけができない)を乗り越える対比手法として位置づけられます。

この系譜が一貫して示すのは「単純連結だけでは幾何の不整合(カメラ位置・重なり)を吸収できないので、座標合わせ(BEV)やアテンションを足すと精度が上がる」というトレードオフです。

## 論文の実験結果(定量データ)
Concat 単体のベンチマーク論文はありませんが、融合手法の比較研究では連結が下限の基準として頻繁に登場します。一般的に報告される傾向は次の通りです。指標は主に nuScenes 3D 検出の mAP と NDS です。
- mAP(mean Average Precision): 検出と正解の中心距離しきい値での一致度の平均(高いほど良い)。NDS(nuScenes Detection Score): mAP に位置・サイズ・向き・速度・属性の各誤差を統合した総合スコア(高いほど良い)。
- 同じバックボーン・同じデータで、ビュー融合を「単純連結」から「クロスアテンション」や「BEV 空間融合」に置き換えると、3D 検出の mAP・NDS が改善します。改善幅は構成によりますが、数ポイント規模の向上が報告されることが多いです。例えば BEVFusion 系の比較では、座標を BEV にそろえてから連結する設計が、画像段で素朴に連結する設計を明確に上回ります。
- ただし連結は計算が軽く、レイテンシ(推論時間)では有利です。アテンションは位置数 N の 2 乗に比例する計算量のため、連結より明確に重くなります。つまり「精度では負けるが速度では勝つ」というトレードオフが、定量比較で繰り返し観察されます。
- アブレーション的に言えば、賢い融合から「座標合わせ」や「アテンション」を取り除いて連結に戻すと精度が落ちる、という形でその効果が示されます。これは Concat が「何を足すと効くか」を測るための土台として機能していることを意味します。

この差が重要なのは、自動運転では遠方や遮蔽下での検出精度が安全に直結するためで、数ポイントの mAP/NDS 差でも見逃しや位置ずれの頻度に効きます。逆に、連結の「速くて単純」という性質は、計算資源が厳しい組込み環境では依然として実用的価値を持ちます。

## メリット・トレードオフ・限界
メリット
- 実装が極めて単純で、連結自体に学習パラメータが一切要らない。微分可能で end-to-end 学習にそのまま組み込める。
- 計算もメモリも軽く、推論が速い(チャネルを並べるだけ)。
- バグが入りにくく、ベースラインとして信頼できる。
- マルチスケール特徴の統合やスキップ接続など、汎用の前処理部品としても広く使える。

トレードオフ・限界
- カメラの幾何(取り付け位置・向き・視野の重なり)を一切考えない。
- 別カメラに写る同一物体を対応づけられない(重なり領域の冗長情報を統合できない)。
- カメラ台数や並び順に依存し、構成が変わると意味が崩れて再学習が要る(順序不変でない)。
- チャネル数が台数倍に膨らみ、後段の負担(パラメータ・計算)が増える。

研究上の含意として、Concat は「賢い融合の効果を測る基準」として価値があり、新手法の必須の比較対象です。また、計算資源が極端に限られる組込み環境では、精度を多少犠牲にしても連結を採る判断があり得ます。連結が負ける主因は「幾何を無視すること」なので、改良の方向は明確に「連結の前に座標をそろえる」「連結の代わりに対応関係を学ぶ」の二択に収束します。

## 発展トピック・研究の最前線
- 座標合わせ後の連結: BEV 空間に各カメラ特徴をそろえてから連結すると、幾何の不整合が減り、単純連結でも質が上がります([Lift-Splat-Shoot (LSS)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0282_lift-splat-shoot) や BEV 系手法)。連結は「そろえた後の最終段」として今も現役です。
- アテンションへの発展: 連結の「対応づけできない」限界を、クエリ・キー・バリューで関係を学ぶ [Cross-Attention Fusion(クロスアテンション融合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0318_cross-attention-fusion) が乗り越えます。さらに参照範囲を絞る [Deformable Attention(変形可能アテンション)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0329_deformable-attention) で計算量を抑えます。
- 学習可能な重み付き融合: 単純連結と全結合の中間として、各ビューやチャネルに学習可能な重み・ゲートを与える gated fusion / SE(Squeeze-and-Excitation)的なチャネル注意 / 残差的な融合(例: [Residual Map Fusion(残差ゲート地図融合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0314_residual-map-fusion))があり、軽さと表現力のバランスを取ります。
