# HDマップのセマンティックRGBラスタライズ

## ひとことで言うと

ベクトル形式の HD マップ（道路の境界線や停止線などを点と線で表した精密地図）を、自車を中心に置き・進行方向を上向きにそろえた 1 枚の RGB 画像に描き起こす技術です。道路境界・車線区分線・中央線・停止線・横断歩道などを「意味ごとに色分け」して描き、カメラ画像とまったく同じ前処理・同じ画像エンコーダに流し込めるようにします。地図を「もう 1 枚のカメラ画像」として扱う発想で、実装が単純なため、地図入力を初めてモデルに組み込むときの定番の出発点になっています。

## 直感的な理解

自動運転モデルには「自車の周りの道路がどうなっているか」を教えたい。地図はその情報源ですが、地図とカメラ画像は形式がまったく違います。カメラ画像はピクセルの格子（縦×横×色）ですが、HD マップは「ここからここまでが車線の左境界線で、座標は (12.3, 45.6), (12.4, 47.0), ... という点の列」というように、図形を数値で持っています。これがベクトル表現です。

画像エンコーダ（CNN や Transformer）はピクセルの格子しか受け取れません。そこで、地図をいったん「絵」に描いてしまえば、カメラ画像と同じ箱に入れられる、という発想が生まれます。これがラスタライズ（rasterize、ベクトル図形をピクセルの格子に塗って画像化する操作）です。お絵かきソフトで「線ツール」を使って白いキャンバスに道路を描くのをプログラムで自動化する、とイメージすれば近いです。

ポイントは色に意味を持たせること（セマンティック）です。緑なら道路境界、赤なら停止線、黄なら横断歩道、と決めておけば、モデルは色を手がかりに道路構造の意味を読み取れます。これは人間が地図記号の色で情報を読むのと同じ発想です。なぜわざわざ「絵に描く」のかという疑問には、「既存の強力な画像エンコーダ（ImageNet 事前学習済みの ResNet/Swin など）をそのまま流用できる」という実利が答えになります。地図専用のネットワークをゼロから設計せずに済むのです。

## 基礎: 前提となる概念

HD マップ（High-Definition map、高精細地図）: 車線の形・境界・停止線・横断歩道などをセンチメートル級の精度で記録した自動運転用の精密地図。普通のカーナビ地図（道路を 1 本の線で表す程度）よりはるかに詳細です。

Lanelet2: HD マップのオープンな表現形式（Poggenhans ら、KIT、2018、ITSC）。道路を「レーンレット（lanelet、左右の境界線で挟まれた走行可能な区間）」というベクトル要素で表します。各レーンレットは左右の境界ポリライン（点を結んだ折れ線）を持ち、さらに「このレーンレットの次はあのレーンレット」という接続関係をグラフで持ちます。停止線・横断歩道・信号などは「規制要素（regulatory element）」として表現されます。Lanelet2 は OpenStreetMap のデータ構造（点 = node、線 = way、要素 = relation）の上に構築されており、テキスト（OSM XML）で持てるのも特徴です。

ラスタ vs ベクトル: 「ラスタ」はピクセルの格子で表す表現（写真、ビットマップ）。「ベクトル」は座標と数式で表す表現（線・点・図形）。ベクトルは拡大しても粗くならず構造を保ちますが、そのままでは CNN に入りません。両者の変換がラスタライズ（ベクトル→ラスタ）です。逆向き（ラスタ→ベクトル）はベクトル化と呼ばれ、近年のオンライン地図構築はこちらを学習で行います。

BEV（Bird's-Eye View、鳥瞰図）: 上空から真下を見下ろした視点。複数カメラやセンサの情報を 1 枚の俯瞰図に統合するのに使われ、地図タイルもこの BEV で描きます。自車を中心に置くのが普通です。BEV を採る理由は、運動・地図・障害物がすべて「地面の上の 2 次元配置」として一貫して扱え、距離がメートルで保たれるからです。

回転行列とヨー角: ヨー角（yaw）は車が水平面内でどちらを向いているかの角度（ラジアン）。2 次元の点を角度 θ だけ回す操作は `[cosθ, -sinθ; sinθ, cosθ]` の行列を掛けることで表せます。地図を自車向きにそろえるとき、この回転を逆向き（-yaw）に掛けます。

主点・焦点距離など投影の話は [前処理（Resize/Crop）に合わせたカメラ内部行列のスケーリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0801_camera-intrinsic-scaling) に、ベクトルのまま処理する代替案は [ベクトル化マップエンコーダ（GNN/Transformer）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0437_vectorized-map-encoder) に譲ります。

## 仕組みを詳しく

地図ラスタライズ関数の典型的な処理を、入力から段階的に追います。出力は `(H, W, 3)` の uint8 RGB 画像です。

入力。地図データへのパス（または地図オブジェクト）、自車の地図座標 `(ego_x, ego_y)`、自車のヨー角 `ego_yaw`（進行方向の向き、ラジアン）、出力サイズ（たとえば 256）、観測半径（たとえば 60 メートル）。

ステップ 1: 関心領域（ROI、Region Of Interest）の抽出。自車から半径 60 m 以内のレーンレットだけを取り出します。地図全体を毎回描くのは重く、遠くの道路は不要なので、自車周辺だけに絞ります。半径は「モデルにどこまで先を見せたいか」と計算コストのトレードオフで決めます。高速道路では先を長く見たいので半径を大きく、市街地では小さくする、といった調整があり得ます。空間インデックス（k-d tree など）で「半径内の要素」を高速に引けるよう、地図を一度だけ前処理しておくのが定石です。

ステップ 2: 座標変換（地図座標 → ピクセル座標）。これが核心です。地図のメートル座標を、画像のピクセル座標に変換します。自車を画像中心に、進行方向を画像の上に向けます。

```
rel    = 点の座標 - 自車位置              # 自車を原点に平行移動
x_rot  = cos(-yaw)*rel_x - sin(-yaw)*rel_y  # -yaw だけ回転して
y_rot  = sin(-yaw)*rel_x + cos(-yaw)*rel_y  # 進行方向を真上に向ける
col    = 中心 - y_rot * scale             # 左(+Y方向)を画像の左へ
row    = 中心 - x_rot * scale             # 前方(+X方向)を画像の上へ
```

`-yaw` で回す理由を噛み砕きます。自車が北を向いていても東を向いていても、地図上の絶対方位で描くと、同じ「右折」という状況でも画像の見え方がバラバラになります。そこで自車の向きを打ち消す（`-yaw` 回す）ことで、常に「画面の上 = これから進む先」という自車中心・進行方向上向きの一貫した見え方にそろえます。この正規化により、モデルは「絶対方位ごとに別々のパターン」を覚える必要がなくなり、データ効率（少ない例で学べる）が上がります。`scale` は `内部描画解像度 / (半径 × 2)` で、60 m 四方の窓を画素数に対応づける倍率です。たとえば半径 60 m を 256 px に描くなら `scale = 256 / 120 ≒ 2.13 px/m`、つまり 1 m が約 2 ピクセルです。

ステップ 3: 要素を色分けして描画。意味ごとに色を割り当てるテーブル（セマンティックカラーマップ）を用意し、OpenCV の `polylines`（折れ線描画）でポリラインを描きます。色分けの一例:

```
道路境界 (road border)        : 緑
縁石 (curbstone)              : 濃緑/ライム
柵・ガードレール・壁          : 茶系
車線区分線 (lane line)        : 青系（灰色の太い下地 + 細い色線）
中央線 (centerline)           : 暗赤の破線
停止線 (stop line)            : 赤
横断歩道 (pedestrian marking) : 黄
```

「セマンティック」とは「色が見た目でなく意味を符号化している」こと。車線区分線は灰色の太い下地に細い色線を重ねて視認性を上げ、中央線は破線で描く、といった工夫をします。描画は重なり順を制御するために複数パスに分けます（境界線→中央線→横断歩道→停止線の順など）。重要な要素（停止線）を最後に描けば、他の線に隠れず確実に見えます。色の選び方には設計の自由度があり、互いに区別しやすい色（色空間で離れた色）を割り当てると、モデルが意味を分離しやすくなります。

ステップ 4: 解像度調整（アンチエイリアス）。いきなり 256×256 に描くと線がギザギザ（ジャギー）になります。そこで内部では高解像度（たとえば 2048×2048）で描いてから、`INTER_AREA` 補間で出力サイズに縮小します。高解像で描いて縮める「スーパーサンプリング」は、線をなめらかに見せるアンチエイリアスの定石です。細い車線がサブピクセル幅になって消えてしまう（ピクセルに乗らない）問題も、高解像で描いてから縮めることで軽減されます。代償は描画コストで、2048×2048 は 256×256 の 64 倍の画素数なので、ここが訓練の律速になりやすい部分です。

ステップ 5: エンコーダへの接続。このタイルをカメラ画像とまったく同じ前処理（リサイズ + 正規化）に通します。

```
bev_map  = ラスタライズ(...)                  # (H,W,3) uint8
map_tile = transform(Image.fromarray(bev_map)) # (3,H,W) float 正規化済み
```

これで地図タイルはカメラタイルと同じ形・同じ正規化になり、追加の「視点」として同じ backbone に入れられます。

自車中心・進行方向上向きのイメージ:

```
       ↑ 進行方向（前方）
   ┌─────────────┐
   │   │     │   │   緑線 = 道路境界
   │   │  ▲  │   │   ▲ = 自車（中心）
   │   │     │   │   破線 = 中央線
   │ ──┼─────┼── │   赤 = 停止線（前方）
   └─────────────┘
   半径60mの正方タイルを256×256pxに
```

## 手法の系譜と主要論文

「地図をどう表現してモデルに渡すか」は、ラスタ派とベクトル派の長い綱引きの歴史があります。

ラスタ派の代表が ChauffeurNet（Bansal ら、Waymo、2019、RSS、arXiv:1812.03079）。道路・経路・他車・信号・自車の過去軌跡などを、それぞれ別チャンネルのトップダウン（俯瞰）ラスタ画像に描き、それを CNN（FeatureNet）に入れて運転方針を学ぶ模倣学習の枠組みを提案しました。動機は「多様な異種情報を、一様な画像表現に統合すれば CNN で扱いやすい」こと。1 つの工夫として、情報の種類ごとにチャンネルを分ける（道路は 1 チャンネル、他車は別チャンネル）方式と、本ノートのように 1 枚の RGB に色分けで詰め込む方式があり、前者は情報が混ざらない代わりにチャンネル数が増え、後者はチャンネルを節約できる代わりに色が衝突し得ます。ChauffeurNet は本ノートのラスタライズ手法の思想的な源流です。

同時期、軌跡予測でもラスタ表現が広く使われました。Cui ら（Uber ATG、2019、ICRA、arXiv:1809.10732）は HD マップと周辺車両をラスタ画像に描いて CNN で多峰性（複数の起こり得る未来）の軌跡を予測しました。これらはラスタ表現が実装容易で堅実だったことを示します。

ラスタ派への反論が VectorNet（Gao Jiyang ら、Waymo、2020、CVPR、arXiv:2005.04259）。「ラスタライズは計算が重く、解像度を上げないと細部が潰れ、しかも地図の構造（車線の接続）を捨ててしまう」と批判し、地図と軌跡をベクトル（ポリライン）のままグラフで符号化する手法を提案しました。これがベクトル派の出発点で、本ノートのラスタライズに対する代替案（[ベクトル化マップエンコーダ（GNN/Transformer）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0437_vectorized-map-encoder)）の論拠です。

地図形式そのものの標準が Lanelet2（Poggenhans ら、KIT、2018、ITSC）。レーン・規制要素・トポロジーをベクトルで表す HD マップ標準で、本ノートで扱うラスタライズはこの Lanelet2 の豊かな構造を、あえて画像に落として使っています。落としてしまうトポロジーをどう惜しむか、が次節以降の論点です。

並べると、ラスタ（ChauffeurNet 型、実装が単純で堅実）とベクトル（VectorNet 型、構造を保ち効率的）という根本的トレードオフが見えます。研究コミュニティでは「まずラスタで動かし、描画がボトルネックと実測で分かったらベクトルに移行する」という段階的判断がよく採られます。プロトタイピング段階ではラスタの単純さが圧倒的に有利だからです。

## 論文の実験結果（定量データ）

ラスタ vs ベクトルの差は、VectorNet の論文で定量的に示されています。指標は前述の minADE / minFDE（予測軌跡と正解の平均/終端の距離、メートル、小さいほど良い）です。

VectorNet は、自社の大規模予測データセットと Argoverse Forecasting ベンチマークで評価しました。論文の主張によれば、ラスタライズ + CNN（ResNet ベース）のベースラインに対し、ベクトル表現 + グラフネットの VectorNet は、計算量（FLOPs、浮動小数点演算回数）をおよそ 1/10 に削減しながら予測精度（DE 系指標）を改善した、と報告されています。Argoverse では minADE がベースライン比で数 % 改善し、計算は大幅に軽くなった、という結果です。「同等以上の精度を、はるかに少ない計算で出した」点がインパクトでした。なぜラスタが重いのかというと、地図の大半は「何も描かれていない空白ピクセル」で、CNN はそこにも一様に畳み込み計算を費やすからです。ベクトルは要素（線・点）の数だけ計算すればよく、空白に計算を払いません。

アブレーション（どの要素を抜くと性能がどう落ちるかの分析）も示唆的です。VectorNet は「ポリライン間のグローバルなグラフ相互作用（地図と軌跡の関係を学ぶ部分）」を外すと予測精度が明確に劣化することを示し、ベクトル表現の利点が「構造を保ったまま要素間の関係を学べる」点にあると裏づけました。逆に言えば、ラスタライズはこの構造をピクセルに潰してしまうので、構造的な関係をモデルが暗黙に再学習しなければならず、その分だけ不利になり得ます。さらに VectorNet はラスタの解像度を上げる実験も行い、解像度を上げると精度はわずかに上がるが計算コストが二乗で増える（256→400 で FLOPs が大幅増）ことを示し、ラスタのスケーリングの悪さを定量化しました。

一方でラスタ表現も健在です。ChauffeurNet は閉ループ（実際に運転させる）評価で、ラスタ + 模倣学習に「合成した危険シナリオ（衝突しかけ・逸脱）」を加えると、素の模倣学習より大幅に堅牢になることを示しました。これは「表現の良し悪し」だけでなく「学習データの設計（データ拡張・摂動）」が効くという別の教訓です。指標の意味としては、閉ループでの逸脱・介入が大幅に減ったことが報告されています。模倣学習が陥りがちな分布シフト（学習時に見ない状態に入ると破綻する問題）を、ラスタ表現の上で摂動データを足して緩和した、という流れは現在のエンドツーエンド学習にも受け継がれています。

## メリット・トレードオフ・限界

メリット。ベクトルマップをカメラと同じパイプラインに統一でき、実装が単純で堅牢です。自車中心・進行方向上向きの一貫表現でモデルが道路構造を学びやすく、色によるセマンティック符号化で停止線・横断歩道・境界などの意味を保持できます。高解像描画 → 縮小でアンチエイリアスが効き、線がきれいに出ます。既存の画像エンコーダと事前学習済み重みを流用できるのも実利です。

トレードオフと限界。第一に、ラスタライズで車線の接続・トポロジーといったグラフ構造が失われます（VectorNet が指摘した本質的弱点）。第二に、描画コストが訓練のボトルネックになり得ます（解像度・要素数に比例、高解像描画はさらに重い）。第三に、ImageNet で事前学習した backbone 用の正規化統計が、人工的なラスタ地図に最適とは限りません。地図はカメラ写真と色分布が大きく違う（自然画像のような連続的なテクスチャがなく、均一色の線が並ぶ）ので、ヒストグラム均等化や CLAHE（局所コントラスト強調）といった前処理、あるいは地図専用の正規化統計を検討する余地があります。第四に、BEV 投影の正しさは自車位置・ヨー角の精度に依存し、ポーズ誤差がそのままタイルのズレ（地図と実世界の不一致）になります。第五に、情報の量子化です。色の数（チャンネル）は有限なので、表現したい意味の種類が増えると色が足りなくなります。

研究上の未解決課題として、ラスタとベクトルを併用する（ラスタの実装容易さとベクトルの構造保持を両取りする）ハイブリッド表現の最適設計は、まだ決着がついていません。また、ラスタ地図に最適な事前学習（自然画像の ImageNet ではなく、地図的な画像で事前学習する）も十分には探索されていません。

## 発展トピック・研究の最前線

地図表現の最前線は、ベクトル派が「オンライン地図構築（online mapping）」へと進んでいます。事前作成した HD マップに頼らず、カメラ画像から走行中にベクトル地図を推定する MapTR（Liao ら、ICLR 2023）や VectorMapNet（Liu ら、ICML 2023）が代表で、地図の鮮度・整備コストの問題（HD マップは作るのも更新するのも高価で、工事や道路変更にすぐ追従できない）を回避しようとしています。一方、ラスタ派は BEV セグメンテーション（Lift-Splat-Shoot、Philion & Fidler、ECCV 2020、[前処理（Resize/Crop）に合わせたカメラ内部行列のスケーリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0801_camera-intrinsic-scaling) も参照）として、カメラから直接 BEV のラスタ地図を生成する流れに発展しました。

エンドツーエンド運転（UniAD、VAD など）では、地図を独立した入力としてではなく、モデル内部の中間表現として扱い、知覚から最終的な計画（プランニング）まで一気通貫にする設計が主流になりつつあります。この文脈では、外部から与えるラスタ地図は「事前情報として補助的に使う」位置づけになり、地図そのものをモデルが推定・更新する方向に重心が移っています。とはいえ、事前 HD マップが利用可能な環境（整備された運行設計領域）では、本ノートのラスタ地図入力は今も実用的で、実装の単純さと堅牢さゆえにベースラインとして根強く使われ続けています。
