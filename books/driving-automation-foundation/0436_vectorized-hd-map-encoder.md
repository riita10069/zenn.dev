# ベクトル化HDマップエンコーダ

## ひとことで言うと

地図には「画像として描いたもの(ラスター)」と「線や点の座標データとして持つもの(ベクトル)」の2系統があります。後者の高精度地図(HDマップ)を、ニューラルネットが扱える特徴ベクトルへ変換する部品が「ベクトル化HDマップエンコーダ」です。車線中心線や停止線を点の列(ポリライン)として正確な座標と意味のラベル付きで扱えるため、ラスター化で失われる精度と意味を保ったまま、運転モデルに地図知識を注入できます。

## 直感的な理解

カーナビが「次の交差点を右」と言えるのは、地図を画像としてではなく構造として持っているからです。「この線は車線境界、ここからここまでが右折レーン、この点が停止線」という、意味と座標が結びついたデータです。これがベクトル地図の世界観です。

一方、地図を画像に焼いてしまう(ラスタライズする)と、解像度の細かさで精度が頭打ちになり、「白い線が車線なのか路肩なのか」を色から推測し直すことになります。曲率の緩い長い車線を細い線で描けば、遠くのマスでは1ピクセルに潰れて消えるかもしれません。ベクトルならどんなに長い車線も座標の列で誤差なく表せ、「これは車線」という型(タイプ)も最初から付いています。

運転を計画するモデルにとって、地図は「これから通れる道の骨格」です。骨格を正確な座標と意味で渡せれば、モデルは「車線に沿って曲がる」「停止線で止まる」といった判断を、ピクセルから推測するより素直に学べます。ベクトル化HDマップエンコーダは、この骨格データをモデルの内部表現(特徴)へ翻訳する通訳係です。

## 基礎: 前提となる概念

ラスター地図(rasterized map): 地図を画素の集まり、つまり画像として描いたもの。道路を白、歩道を灰色で塗った 256×256 の画像など。画像なので通常の画像エンコーダ(CNNやSwin Transformer)でそのまま処理できる手軽さが利点です。

ベクトル地図(vectorized map): 道路要素を点の列(ポリライン、polyline)として座標で持つもの。例えば「車線Aは (10.2, 3.1) → (12.0, 3.0) → (13.9, 2.8) という点列」。各ポリラインに「車線境界」「横断歩道」「停止線」といった型が付きます。

HDマップ(High-Definition map): センチメートル級の精度を持つ自動運転用の地図。車線形状・接続関係・信号や標識の位置などを含み、ベクトル形式が主流です。代表的なフォーマットに次の2つがあります。Lanelet2 は車線(lanelet)を左右の境界線(linestring)と接続関係で表す、ロボティクスや自動運転の研究で広く使われる形式。OpenDRIVE(.xodr)は道路ネットワークを参照線(reference line)とその上のオフセットで記述する業界標準フォーマットで、シミュレータや量産開発で広く採用されています。両者に対応できることが、地図エンコーダの汎用性の鍵になります。

ポリライン(polyline): 連続した線分でできた折れ線。点列 (p1, p2, ..., pk) で表し、各点は (x, y) あるいは方向や型を付けた高次元ベクトル。曲線も十分な点数で近似できます。

レジストリパターン(registry pattern): 「名前 → 実装クラス」の対応表を持ち、文字列キーで実装を選んで生成する設計。`build_encoder("vectorized")` のように呼び出し側はキーだけ指定し、実装本体を後から差し替えられる。新しいエンコーダを足すときに呼び出し側のコードを変えずに済む拡張性が利点です。地図のように「ラスター/ベクトルを切り替えたい」状況に向きます。

自己注意(self-attention): 入力要素どうしが互いに重み付きで参照し合い、文脈を取り込む仕組み。地図では「この車線はあの停止線の手前」のような要素間の関係を捉えるのに使います。

置換等価(permutation equivariance / invariance): 入力要素の並び順を変えても出力が本質的に変わらない性質。ベクトル地図の要素集合(100本の車線)は「並び順に意味がない集合」なので、順序に依存しないモデリングが望ましい、という考え方の前提になります。

## 仕組みを詳しく

ベクトル化マップエンコーダの典型的な処理を、超初心者向けにステップ分解します。最終目標は、ベクトル地図を画像BEVと同じ形状 `(B, C, H, W)` の特徴へ変換し、同じego座標・同じ pc_range に揃えることです。ここを揃えないと [BEV空間対応の前提と制約](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0931_bev-spatial-correspondence-caveat) で述べた空間対応が崩れます。

ステップ1: 入力の整形。地図要素の集合をテンソルにします。例えば「最大100本の要素、各要素は最大20点、各点は (x, y) の2次元」なら `(B, 100, 20, 2)`。要素数や点数は地図によって変わるので、足りない分はパディング(0埋め)し、どこが本物かを示すマスクを別に持ちます。可変長を固定形状に押し込むこの前処理が、ラスター地図より手間のかかる点です。座標はego座標系へ変換し、pc_range で正規化しておくと学習が安定します。

ステップ2: ポリライン内の集約。各ポリライン(20点)を1本の特徴ベクトルにまとめます。小さなMLP(多層パーセプトロン)や1次元畳み込みで各点を変換し、最大値プーリングなどで1本に縮約します。`(B, 100, 20, 2)` → `(B, 100, C)`。点ごとに「自分の座標」「線分の方向」「前後の点との相対位置」などを特徴に含めると形を保ちやすい。これはVectorNetが「サブグラフ(subgraph)」と呼んだ操作に相当します。

ステップ3: ポリライン間の関係づけ。100本の要素ベクトルをTransformerの自己注意で相互参照させ、「この車線はあの交差点に属する」といった地図全体の文脈を持たせます。`(B, 100, C)` → `(B, 100, C)`(形状は変わらず内容が文脈化される)。VectorNetの「グローバルグラフ(global graph)」に相当します。

ステップ4: BEVグリッドへの配置。文脈化した要素特徴を、ego座標上のBEVグリッド `(B, C, H, W)` へ並べ直します。やり方は流儀によります。各要素の座標から対応するセルへ散布(scatter)する方法、BEVクエリが要素群にクロスアテンションして引き込む方法(MapTR/VAD系に近い)などです。いずれにせよ最後に「マス (i, j) は現実の (X, Y)」という対応を、画像BEVと同じ pc_range・同じ H×W で確定させます。

なぜこの設計か。ステップ2でポリライン内をまず畳むのは、点の数が要素ごとに違っても固定長ベクトルに揃えられ、かつ「線としての形」を保てるからです。ステップ3で要素間を注意で結ぶのは、地図の意味が要素単体ではなく関係(接続・隣接・前後)に宿るからです。ステップ4で画像BEVと形状・座標を揃えるのは、後段の融合(残差/クロスアテンション)が空間対応を要求するからで、ここがベクトルエンコーダ設計の最大の制約になります。

レジストリ設計の例を示します。

```python
MAP_ENCODER_REGISTRY = {
    "rasterized": RasterizedMapEncoder,
    # "vectorized": VectorizedMapEncoder,  # 将来この1行を足すだけで切替可能
}

def build_map_encoder(map_type, **kwargs):
    if map_type not in MAP_ENCODER_REGISTRY:
        raise ValueError(f"unknown map encoder: {map_type}")
    return MAP_ENCODER_REGISTRY[map_type](**kwargs)
```

このように入り口だけ先に用意しておけば、ベクトルエンコーダ本体の実装が後から入っても、学習ループや呼び出し側を一切変えずに `"vectorized"` へ切り替えられます。これは大規模な研究コードベースで、未完成の部品の置き場所を確保しつつ既存経路を壊さないための定石です。地図表現のように「ラスターで素早く立ち上げ、後でベクトルに差し替える」段階的開発と相性が良い設計です。

ラスターとベクトルの違いを数値で対比します。

| | ラスター地図 | ベクトル地図 |
|---|---|---|
| 表現 | 画素 `(B, C, H, W)` | ポリライン集合 `(B, N, P, D)` |
| 精度 | 解像度で頭打ち(遠方が潰れる) | 座標精度そのまま |
| 意味の型 | 色から推測し直す | 最初から付与済み |
| 前処理 | 画像エンコーダで直接処理、簡単 | パディング・マスク・正規化が必要 |
| 計算量 | 解像度に比例(密) | 要素数に比例(疎) |

## 手法の系譜と主要論文

ベクトル地図表現の系譜は、軌道予測から知覚・計画へと広がってきました。

VectorNet (Gao et al., CVPR 2020, arXiv:2005.04259)。地図要素とエージェント軌道をすべて「ポリラインのグラフ」として統一的に扱う先駆け。まずポリライン内をサブグラフで集約し、次にポリライン間をグローバルグラフ(完全グラフ上の注意)で関係づける、という2段構えを確立しました。ラスター化を捨てることで、解像度に縛られない正確な幾何表現が可能になりました。

LaneGCN (Liang et al., ECCV 2020, arXiv:2007.13732)。車線グラフの接続(前後・左右・交差)を畳み込みで明示的に伝播させ、地図の連結構造を強く活かしました。VectorNetが完全グラフで関係を学ぶのに対し、道路トポロジーを構造として与える方向です。両者は「関係を学習で見つける」対「関係を構造として与える」という設計思想の対比をなします。

VAD (Jiang et al., ICCV 2023, arXiv:2303.12077)。シーン全体(地図要素もエージェントも)をベクトル化し、Transformerでエンドツーエンドに計画まで行う。ラスターの密な計算を避けるため軽量・高速で、地図をベクトルクエリとして計画に直結させる代表例です。後継のVADv2は確率的な計画(行動分布の出力)へ拡張しました。

MapTR (Liao et al., ICLR 2023, arXiv:2208.14437)、MapTRv2 (Liao et al., IJCV 2024, arXiv:2308.05736)。オンラインでHDマップ自体をベクトル要素として推定する手法。地図要素を「順序を入れ替えても同じ形状を表す点集合」として扱う置換等価なモデリング(permutation-equivalent point set)を導入し、点の対応付けの曖昧さを階層的マッチングで解消して学習を安定化しました。地図エンコーダの逆方向(地図生成)ですが、ベクトル地図のニューラル表現の標準的な扱いを定めた重要研究で、エンコーダ設計の参考になります。

StreamMapNet (Yuan et al., WACV 2024, arXiv:2308.12570) は時間方向にベクトル地図を伝播し、遮蔽や遠方での頑健性を高めました。過去フレームの地図要素を ego-motion で位置合わせして引き継ぐことで、一瞬隠れた車線も推定を維持できます。Lanelet2 (Poggenhans et al., ITSC 2018) は地図フォーマットそのものを定義した論文で、研究と実車をつなぐ標準として位置づけられます。

系譜を一言でまとめると、ラスター(VectorNet以前)→ ベクトルグラフ(VectorNet, LaneGCN)→ ベクトルをエンドツーエンドの知覚・計画に統合(VAD, MapTR, 統合スタック系)という流れです。

## 論文の実験結果(定量データ)

VectorNet は Argoverse(軌道予測の標準ベンチマーク)で評価されます。主指標は ADE(Average Displacement Error、予測軌道と正解の各時刻の平均ズレ、単位メートル、低いほど良い)と FDE(Final Displacement Error、最終時刻のズレ)です。報告によると、同等の計算量でラスター化+CNNより小さい誤差を達成し、かつパラメータ数と計算量を大きく削減しました。要点は「ベクトル表現はラスターより少ない計算でより正確」という定量的な裏付けです。アブレーションでは、ポリライン間のグローバルグラフ(要素間の注意)を外すと予測誤差が悪化することが示され、「要素間の関係を明示的に捉える」ことの効果が確認されています。

VAD は nuScenes のオープンループ計画評価で報告されています。主指標は L2誤差(予測軌道と人間の運転軌道の平均ズレ、メートル)と衝突率(collision rate、予測軌道が他物体とぶつかる割合、%)です。報告では、当時のラスターベースのエンドツーエンド計画(例えばST-P3など)に比べてL2誤差と衝突率をともに下げつつ、推論速度を数倍高速化しました。具体的には平均L2が約1m前後の水準まで縮み、推論は当時の代表手法より大幅に速いと報告されています。ベクトル化が「精度と速度を同時に改善する」ことを示した点が重要です。L2誤差が1mというのは、3秒先の予測位置が人間の運転線からおよそ1m以内に収まる水準で、車線幅(約3.5m)に対して許容範囲に入る目安です。

MapTR / MapTRv2 はオンラインHDマップ構築を nuScenes で評価し、指標は mAP(車線・境界・横断歩道など要素種別ごとのAverage Precisionの平均)です。MapTR は当時の手法を上回るmAPをリアルタイムに近い速度(報告で数十FPS級)で達成し、MapTRv2 でさらにmAPを大きく押し上げました(報告では従来比で十数ポイント以上の改善)。アブレーションでは、点集合を置換等価に扱うマッチング設計を外すとmAPが顕著に低下することが示され、「ベクトル要素を順序非依存に扱う」ことの効果が定量的に確認されています。順序を固定して学習させると、本来は同じ形を表す点列の並べ替えに対してモデルが過剰に敏感になり、学習が不安定化するためです。

これらを横断して読み取れるのは、第一に、ベクトル表現はラスター比で計算効率と精度を両立しやすいこと。第二に、その利得は「要素間の関係を注意やグラフで明示的に捉える」設計と「要素を順序非依存に扱う」設計から来ること。第三に、地図品質(整備度・鮮度)に性能が依存するため、地図が古いと逆に誤誘導になりうること、です。

## メリット・トレードオフ・限界

メリット。ベクトル地図は解像度の限界と意味のあいまいさを同時に回避し、車線・停止線を正確な座標と型で扱えます。計算効率も高く、軌道予測・計画で精度と速度を両立しやすい。レジストリ設計と組み合わせれば、ラスター経路を壊さずに後から差し込めます。

トレードオフと限界。第一に、入力が可変長のポリライン集合なので、パディング・マスク・正規化などラスターより前処理が複雑で、バグの温床になりやすい。マスクを取り違えるとパディングのゼロ点が本物の車線として扱われ、静かに精度が落ちます。第二に、性能がベクトル地図そのものの品質と鮮度に強く依存します。地図が現実とズレていれば、正確な座標であるがゆえにかえって自信を持って誤誘導します。第三に、画像BEVと同一ego空間・同一 pc_range へ揃える制約が常につきまとい、ここを誤ると空間対応が崩れて地図情報が効きません。第四に、Lanelet2 と OpenDRIVE では座標系・要素粒度・接続表現が異なるため(Lanelet2 は左右境界の対、OpenDRIVE は参照線+オフセット)、両対応するパーサとエンコーダの正規化層を作るコストがかかります。

未解決の課題として、オフライン地図(整備済みHDマップ)とオンライン地図(走行中に推定する地図)の使い分けと融合、地図の鮮度不一致への頑健性、Lanelet2/OpenDRIVE をまたぐ統一表現の設計などが、実用化に向けたオープンな論点として残っています。特に「地図が間違っているかもしれない」という不確実性を計画にどう伝えるかは、安全性に直結する難問です。

## 発展トピック・研究の最前線

最近は「オフラインHDマップに頼らず、走行中にベクトル地図をオンライン推定して即座に計画へ使う」流れ(MapTR系、StreamMapNet)が加速しています。これにより地図整備コストと鮮度問題を緩和できますが、推定誤差をどう計画に伝えるか(不確実性の伝播)が新たな課題です。また、ベクトル地図とセンサBEVを単一の統合空間で扱うエンドツーエンド計画(統合スタック系、VAD系)では、地図エンコーダは独立部品ではなく大きな知覚・計画スタックの一構成要素になりつつあります。さらに、地図要素をトークン列として言語モデル的に扱い、トポロジー(接続関係)まで生成する研究(TopoNet 等)も登場し、ベクトル地図の表現は「点列」から「関係を含む構造化トークン」へ進化しています。車線中心線(centerline)とその接続を有向グラフとして出力する流れは、計画が「次にどの車線へ行けるか」を直接読み取れる点で実用上の意義が大きいと考えられています。
