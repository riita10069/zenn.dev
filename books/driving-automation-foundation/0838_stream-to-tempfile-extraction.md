# 一時ファイルへのストリーム展開でメモリ節約

## ひとことで言うと
数GBもある圧縮ファイル (zip など) を、いったんメモリ (RAM) に丸ごと載せてから展開すると、その瞬間だけメモリ使用量が急増 (スパイク) してプログラムが落ちることがあります。これを避けるため、まず一時ファイルへ少しずつコピー (ストリーム) してから展開する、という設計です。RAM の圧迫を一時的なディスク使用に置き換えることで、ファイルサイズに依存しない安定したメモリ消費を実現します。これは特定のソフトウェアの工夫ではなく、有限メモリで巨大データを扱う際の普遍的な定石です。

## 直感的な理解
大きな段ボール箱 (圧縮ファイル) の中身を取り出したいとします。素朴な方法は「箱の中身を全部いったん机の上 (RAM) に広げてから、必要なものを探す」です。箱が小さければ問題ありませんが、箱が机より大きければ広げきれず物が床に散らばる、つまりプログラムがクラッシュします。

賢い方法は「箱を床 (ディスク) の隅に置いたまま、必要なものだけ箱から手を伸ばして取り出す」です。机の上には常に一度に1つの物しか乗らないので、どんなに大きな箱でも机が足りなくなることはありません。一時ファイルへのストリーム展開は、まさにこの「箱を床に置いたまま中身を取り出す」やり方です。

機械学習のデータパイプラインでは、この問題が頻繁に起きます。学習用データセットはしばしば数百GB〜数TB規模で、1ファイルあたり数GBの圧縮アーカイブとして配布されることが珍しくありません。とくにマルチセンサー (複数カメラ・LiDAR) の運転ログは、1区間ぶんだけで複数のカメラ映像をまとめた数GB級のアーカイブになります。これを複数同時に、しかもGPU上のモデルやバッチデータと並行してメモリに展開しようとすると、RAM が容易に枯渇します。データの前処理段階で OOM (Out Of Memory) によってプロセスが落ちると、せっかく確保したGPU時間が無駄になります。「データを全部RAMに載せない」ことは、大規模学習基盤を安定運用するための土台です。

## 基礎: 前提となる概念
理解に必要な用語を、知らない人向けに噛み砕きます。

- メモリ (RAM): CPU が高速に読み書きできる作業領域。高速だが容量が限られる (一般的なサーバで数十GB〜数百GB)。プログラムが「いま処理中のデータ」を置く場所です。
- ディスク (ストレージ): SSD や HDD。RAM より桁違いに大容量だが、アクセスは遅い。電源を切っても消えません。
- メモリスパイク: ある瞬間だけメモリ使用量が急に跳ね上がる現象。平均的には収まっていても、ピークで容量を超えるとそこで落ちます。
- OOM (Out Of Memory): メモリが枯渇した状態。Linux では OOM killer という仕組みが、メモリを食っているプロセスを強制終了します。学習ジョブがこれで突然死すると、終了コードが 137 (= 128 + SIGKILL の 9) になるだけで原因メッセージが残らず、特定が難しい厄介なバグになります。
- ストリーム (stream): データを「全部まとめて」ではなく「流れてくるそばから少しずつ」扱う処理の仕方。動画のストリーミング再生と同じ発想で、全体をダウンロードし終える前から処理を始められます。
- バッファ / チャンク: ストリーム処理で一度に扱う小さなかたまり。たとえば「1MB ずつ読む」の 1MB が 1 チャンクです。
- zip (圧縮アーカイブ): 複数ファイルを 1 つにまとめて圧縮した形式。内部に「セントラルディレクトリ (中央目次)」という索引がファイル末尾近くにあり、どのエントリがどの位置 (オフセット) にあるかを記録しています。この目次のおかげで、必要なファイルだけを狙って取り出せます (ランダムアクセス可能)。
- ランダムアクセス vs シーケンシャルアクセス: 前者は「任意の位置へ飛んで読む」、後者は「先頭から順に読む」。zip はセントラルディレクトリ方式なので前者ができます。
- ガベージコレクション (GC): Python など多くの言語が持つ、不要になったメモリを自動回収する仕組み。数GBのオブジェクトを確保・解放すると、この GC 自体にも負荷がかかり、処理が一時停止 (stop-the-world) することがあります。
- 計算量の O 記法: 入力サイズ N に対する資源 (時間やメモリ) の増え方を表す表記。O(N) は「N に比例して増える」、O(1) は「N によらず一定 (定数)」を意味します。本トピックの核心は、ピーク RAM を O(N) から O(1) に落とすことです。

## 仕組みを詳しく
問題のあるコードと、改善したコードを対比します。まず、メモリスパイクを起こす素朴な書き方です。

```python
import io, zipfile

with open_file(chunk_file) as f:
    data = f.read()                          # zip全体をRAMに載せる (数GB)
    zf = zipfile.ZipFile(io.BytesIO(data))   # そのRAM上のバイト列を展開
```

ここで `f.read()` は「ファイル全体を 1 つのバイト列としてメモリに読み込む」命令です。`io.BytesIO(data)` は「メモリ上のバイト列を、ファイルのように扱えるようにする」ラッパーです。問題は、`f.read()` の瞬間に zip 全体ぶんのメモリ (たとえば 3GB) を一気に確保してしまう点です。複数のセンサーぶん、複数クリップを順に処理するとき、各 zip でこの数GBの確保が起き、ピークで他の処理ぶんと重なるとメモリが足りなくなります。さらに `BytesIO` に渡す時点でコピーが生じる実装もあり、瞬間的に 2 倍のメモリを踏む場合すらあります。

改善版は「ストリームコピー」です。読み込み元から少しずつ (たとえば 1MB ずつ) 読んでは一時ファイルへ書く、を末尾まで繰り返します。

```python
import tempfile, shutil, zipfile

with open_file(chunk_file) as src:
    with tempfile.NamedTemporaryFile(suffix=".zip", delete=True) as tmp:
        # 少しずつ読んで一時ファイルに書く (RAMには1チャンクぶんしか乗らない)
        shutil.copyfileobj(src, tmp, length=1024 * 1024)  # 1MBずつ
        tmp.flush()
        # 一時ファイルを zip として開く。zipfile は必要な部分だけシーク読みする
        with zipfile.ZipFile(tmp.name, "r") as zf:
            for name in zf.namelist():
                with zf.open(name) as member:
                    ...  # 各エントリをストリームで取り出す
```

ポイントを分解します。

1. `tempfile.NamedTemporaryFile` で OS の一時領域 (Linux なら通常 `/tmp`、ただし環境変数 `TMPDIR` で変更可能) にファイルを作ります。`delete=True` で `with` ブロックを抜けると自動削除されるため、後始末が要りません。
2. `shutil.copyfileobj(src, tmp, length=1MB)` が肝心です。これは「読み込み元 `src` から `length` バイトずつ読んで `tmp` に書く」を末尾まで繰り返す関数です。1MB ずつなので、メモリ常駐は約 1MB に固定されます。
3. 一時ファイルが完成したら、`zipfile.ZipFile` でそのパスを開きます。`zipfile` はセントラルディレクトリ (目次) を読み、必要なエントリだけシークして取り出すので、これも全体をメモリに載せません。

メモリ使用量の before / after を概念図で示します。

```
before (BytesIO 方式):
  RAM |####################| ← zip全体 3GB が常駐
      └ 複数 zip 同時処理でピークが重なると OOM

after (一時ファイル方式):
  RAM |#|                    ← 常に1MB チャンク + 展開中の1エントリ
  Disk|####################| ← zip全体 3GB は一時的にディスクへ
      └ ピーク RAM はファイルサイズに依存しない (定数)
```

数値で言えば、3GB の zip を 10 個順に処理する場合、BytesIO 方式は最悪 3GB × 同時処理数のピークになりますが、一時ファイル方式ではピーク RAM は常に数MB程度に抑えられます。トレードオフは明確で、「RAM の圧迫」を「一時的なディスク使用」に置き換えています。ディスクは RAM より桁違いに潤沢で、足りなくなっても直ちにプロセスが死ぬわけではないぶん安全です。

なぜこの設計が正しいのか。鍵は「zip が目次を持つランダムアクセス可能なフォーマットである」点です。zip を展開するには、ファイル末尾近くにあるセントラルディレクトリを読む必要があります。BytesIO でメモリに載せればもちろん末尾も読めますが、それは全体を載せる代償を払っています。一時ファイルにしておけば、`zipfile` はファイルシステムのシーク (`seek`) 機能を使って末尾の目次へ直接ジャンプし、必要なエントリのデータ部分だけを順に読めます。「目次があるからこそ全体を載せずに済む」のです。逆に言えば、目次を持たない逐次圧縮形式 (後述の gzip 単体) ではこの利点が消えます。

補足として、`NamedTemporaryFile` でなく `SpooledTemporaryFile` を使う選択肢もあります。これは「小さいうちは RAM、閾値を超えたらディスクへ自動退避」というハイブリッドで、小ファイルが多数あるときに RAM の高速性とディスクの安全性を両取りできます。閾値の設計が新たなパラメータになる点だけ注意が要ります。

## 手法の系譜と主要論文
これはアルゴリズムの論文ではなく、システム設計の普遍的な原則です。背景となる系譜を挙げます。

- UNIX パイプ哲学 (Ritchie & Thompson, 1974, "The UNIX Time-Sharing System", Communications of the ACM, vol.17): 「小さなプログラムを、データを流しながら連結する」という設計思想。`cat file | grep x | sort` のように、各段はデータ全体を保持せず流れてくるそばから処理します。メモリ有限のマシンで無限長・巨大データを扱う標準手段の源流であり、「全部を一度に持たない」という本トピックの精神そのものです。
- out-of-core (external memory) アルゴリズム: データがメモリに収まらない場合に、ディスクを作業領域として明示的に使う技法の総称。Aggarwal & Vitter (1988, "The Input/Output Complexity of Sorting and Related Problems", CACM, vol.31) が、メモリサイズ M とブロック転送サイズ B をパラメータに I/O 計算量を理論化しました。「メモリに収まらない問題はブロック単位の I/O 回数で計る」という枠組みで、一時ファイル経由の展開はその実務的な一形態です。
- ZIP フォーマットの設計 (PKWARE APPNOTE, 1989 以降): セントラルディレクトリを末尾に置く設計により、アーカイブ全体を読まずにエントリ一覧とオフセットを取得できます。tar との対比が分かりやすく、tar は索引を持たず先頭から舐めるため部分取り出しに向きません。この違いが「zip なら一時ファイル + シークで効率的、tar なら結局シーケンシャル」という挙動差を生みます。
- 大規模学習データの分野: WebDataset (Aizman, Maltby & Breuel, 2020, "High Performance I/O For Large Scale Deep Learning") は、データを tar シャードにまとめ「シーケンシャルに少しずつ」読む設計を提案しました。「全データを一度に RAM へ載せない」「ランダムアクセスよりシーケンシャル読みを優先して I/O 効率を上げる」という同じ動機に基づきます。FFCV (Leclerc et al., 2023, "FFCV: Accelerating Training by Removing Data Bottlenecks", CVPR) や NVIDIA DALI も同様に、データ供給段でメモリと I/O のボトルネックを解消する設計を採ります。

## 論文の実験結果(定量データ)
この手法は単独で論文化されるものではないため、関連する大規模 I/O 研究の定量的知見を引きます。

- WebDataset / tar シャード方式 (Aizman et al., 2020) は、多数の小ファイルをランダムアクセスする従来方式に対し、シーケンシャル読みでディスクおよびネットワークのスループットを大幅に改善することを報告しています。回転ディスクや分散ストレージで小ファイルランダムアクセスがスループットを 1 桁以上低下させる一方、シャード化したシーケンシャル読みはストレージの最大帯域に近い速度を引き出せるとされます。指標である「スループット (単位時間あたり処理バイト数)」は、データ供給が GPU 計算に追いつくかを決めます。GPU が遊ぶ (data starvation) と学習全体が遅くなるため、この差が学習基盤の実効効率に直結します。
- FFCV (Leclerc et al., 2023) は、ImageNet 学習でデータローダのボトルネックを取り除くことにより、同一精度に到達する学習時間を従来パイプライン比で大幅短縮 (条件により数倍) できると報告しました。論文の主張は「モデル計算ではなくデータ供給が律速になっている学習が多い」というもので、メモリと I/O の設計が精度ではなく速度・コストを直接左右することを定量的に示しています。
- メモリ消費の観点では、BytesIO 全読み込み方式のピーク RAM はファイルサイズに線形 (O(N)) で増えるのに対し、ストリーム + 一時ファイル方式は O(チャンクサイズ) = 定数です。たとえばチャンク 1MB なら、1GB ファイルでも 10GB ファイルでも常駐は約 1MB。アブレーション的に言えば、「一時ファイル経由」を外して BytesIO に戻すと、ファイルが大きいほどピーク RAM が比例して増え、ある閾値で OOM する、という明確な退行が起きます。
- `shutil.copyfileobj` の `length` (チャンクサイズ) は性能パラメータです。小さすぎる (例: 4KB) とシステムコール回数が増えて CPU オーバーヘッドが乗り、大きすぎる (例: 256MB) とその分だけ RAM 常駐が増えます。実務では数十KB〜数MBが定番で、Python の `shutil.copyfileobj` のデフォルトバッファは環境により 64KB 前後です。Aggarwal-Vitter の枠組みで言えば、これはブロック転送サイズ B のチューニングに対応し、I/O 回数 (N/B に比例) と 1 回あたりの常駐量のトレードオフを決めます。

## メリット・トレードオフ・限界
メリット

- ピーク RAM をファイルサイズ非依存 (1 チャンクぶん) に抑えられ、数GB〜数十GBのアーカイブでも OOM しにくい。
- 一時ファイルは処理後に自動削除でき、後始末が要らない (`delete=True`)。
- zip がファイル経由ならセントラルディレクトリ経由で必要エントリだけシーク読みでき、無駄な展開も減らせる。
- 大きなオブジェクトを RAM に確保・解放しないため、GC の停止 (stop-the-world) や、確保・解放を繰り返すことによるヒープの断片化も避けられる。

トレードオフ・限界

- 一時的にディスクをアーカイブサイズぶん消費する。ディスク残量が乏しい環境 (小さい `/tmp`、コンテナの薄い書き込みレイヤ、tmpfs が RAM 上にマウントされている環境) では別の枯渇要因になる。tmpfs の `/tmp` だと「ディスクに逃がしたつもりが実は RAM」という落とし穴があり、`TMPDIR` を実ディスクの大容量ボリュームに向ける運用が要ることがあります。
- ディスク I/O が一往復増えるため、データが RAM に余裕で収まる小さいファイルでは、BytesIO 直接展開よりわずかに遅くなることがある。安定性の利得が効くのは巨大ファイルのときに限られます。
- 一時ファイルのパス・権限・後始末を正しく扱う必要があり、素朴な BytesIO 方式よりコードがやや複雑になります。マルチプロセス (DataLoader の複数ワーカー等) で同じ一時ディレクトリを使うと衝突しないよう、ファイル名のユニーク性に注意が要ります (`NamedTemporaryFile` は内部でユニーク名を生成するので基本は安全)。
- gzip 単体 (.gz) のような目次を持たない逐次圧縮形式では「必要部分だけシーク」が効かず、結局は先頭から順に展開するしかありません。ランダムアクセスの利点はフォーマット依存です。
- プロセスが SIGKILL で突然死すると `with` の後始末が走らず、一時ファイルが残留しうる。長時間運用では `/tmp` のゴミ掃除 (systemd-tmpfiles 等) を併用するのが堅実です。

未解決・発展的な課題として、根本的にアーカイブ展開を介さずに済ませる設計 (データを最初からシャード化された読みやすい形式で保存しておく) のほうが、I/O もメモリも有利になります。一時ファイル方式は「既存の巨大 zip を安全に扱う」対症療法であり、データレイアウトそのものを学習向けに最適化する方向 (WebDataset 化、列指向フォーマット化) が本質的な解決に近いと言えます。

## 発展トピック・研究の最前線
- メモリマップ (mmap): ファイルを仮想メモリ空間に対応づけ、アクセスしたページだけ OS が遅延ロードする技法。一時ファイル方式と組み合わせると、展開後のデータへのアクセスもメモリ効率よく行え、ページキャッシュを介して複数プロセス間で共有もできます。
- ストリーミングデコード: 動画 (mp4 等) を全フレーム展開せず、必要なフレームだけデコードする (PyAV / decord / torchcodec など)。これも「全部メモリに載せない」の動画版で、運転ログのように長尺マルチカメラ映像を扱う際に効きます。
- データ供給最適化全般: DataLoader の `num_workers`、`prefetch_factor`、`pin_memory`、そして WebDataset / FFCV / DALI といった I/O フレームワークは、いずれも「GPU を飢えさせない」ことを目標に、メモリと I/O のトレードオフを設計します。本トピックはその最小単位の実践例です。
- クラウドストレージからの直接ストリーミング: S3 などのオブジェクトストレージから、ローカルに全ダウンロードせず HTTP Range request で部分取得しながら処理します。zip のセントラルディレクトリを Range で先に取得し、必要エントリのバイト範囲だけ追加取得すれば、一時ファイルすら作らずにメモリ定数を保てます。`zipfile` が seekable なファイルオブジェクトを要求する点を、Range request を返すアダプタで満たすのが実装上のコツです。
- 圧縮形式の選択: zstd や lz4 はデコード速度が gzip より速く、学習データの再圧縮で I/O 律速を緩和できます。「圧縮率を少し犠牲にしてデコード速度を取る」判断は、データ供給がボトルネックの学習で合理的です。
