# SparseDrive (スパース表現E2E)

## ひとことで言うと
SparseDrive(スパースドライブ)は、多くの自動運転モデルが使う「密(dense)な BEV(鳥瞰図)」=真上から見た世界全体を埋め尽くす格子地図を毎フレーム計算する重い方式をやめ、代わりに「物体や車線ごとに1個ずつのクエリ(問い合わせベクトル)」だけを持つスパース(疎、まばら)な表現で知覚・予測・計画をまとめて行うエンドツーエンド(E2E)手法です。計算する対象が数万セルから数百クエリへ激減するので、性能をほぼ保ったまま大幅に速くて軽いのが特徴です。

## 直感的な理解
密な BEV は「方眼紙を世界に重ね、全マスに今そこに何があるかを書き込む」やり方です。分かりやすいのですが、道路の真ん中の何も無い空き地にも全マスぶんの計算をしてしまう無駄があります。運転で本当に大事なのは「どこに車・歩行者・車線があるか」という世界のごく一部だけです。

SparseDrive は方眼紙を捨て、「重要な物体や車線にだけ付箋を貼り、その付箋(クエリ)同士で情報をやりとりする」やり方です。付箋は数百枚で済むので、方眼紙4万マスを毎回更新するより圧倒的に軽くなります。これは物体検出の分野で DETR が「物体ごとのクエリ」で格子(アンカー)を不要にした流れを、知覚だけでなく予測・計画まで含むE2E全体に押し広げたものです。

加えて SparseDrive はもう一つ、見落とされがちな問題を解こうとしています。従来の統合E2E(UniAD など)は「知覚→予測→計画」を直列に並べていました。直列だと、上流(予測)の誤りが下流(計画)に一方向に流れ込み、しかも計画は予測の結果を「与えられた事実」として受け取るだけで、計画側の意図を予測に反映できません。実際の運転では「自分がここで右折すると決めたら、隣の車はそれを見て減速するだろう」という相互作用があるはずです。SparseDrive は自車も周囲エージェントも同じクエリで対等に扱い、予測と計画を並列・相互作用させることで、この一方通行を緩和します。

## 基礎: 前提となる概念
- BEV(Bird's-Eye-View): 世界を真上から見た格子表現。カメラ画像をこの俯瞰格子に投影して扱います。多くの統合E2Eは BEV を中間表現の土台にします。密 BEV を作る代表が BEVFormer で、空間クロスアテンションで多視点画像を固定格子に集約します。
- 密(dense)とスパース(sparse): 密は「全格子セルを常に計算・保持する」こと。スパースは「意味のある少数の要素だけを保持する」こと。SparseDrive は後者です。密格子の計算量は格子辺の二乗で増える(200×200を400×400にすると4倍)のに対し、スパースは要素数に比例するだけです。
- クエリ(query): Transformer に「これについて教えて」と投げる学習可能なベクトル。SparseDrive では1つのクエリが1つの物体または1本の車線に対応し、繰り返し更新されながら位置や属性を表していきます。学習開始時はランダムな初期ベクトルですが、学習を通じて「典型的な物体らしさ」を覚えていきます。
- アンカー(anchor): クエリが「3D空間のどのあたりを指しているか」の手がかりとなる初期位置・初期ボックス。クエリにアンカー(中心座標・大きさ・向き・速度など)を持たせると、画像のどこをサンプルすべきかが定まります。アンカーは層を通るたびに更新されます。
- deformable attention(変形可能アテンション): 通常の attention は画像全画素との関連を計算して重いですが、deformable attention はクエリが「画像の必要な数点だけ(参照点まわりの少数のサンプル点)」を学習で選んでサンプルする軽い attention です。Deformable DETR で提案され、スパース手法の計算効率の鍵になっています。アンカーを多視点カメラ画像に投影し、その近傍だけを見るので、3D→2Dの射影を attention に組み込めます。
- 知覚・予測・計画: 知覚=周りに何があるか検出、予測=他車がこれからどう動くか、計画=自車がどう動くか。従来は別モジュールで直列につなぎますが、E2Eでは一体で学習します。
- マルチモーダル予測(multi-modal prediction): 「未来は一通りではない」ことを表すため、1エージェントにつき複数本の候補軌跡を出すこと。交差点で「直進も右折もありうる」状況を1本に潰さないための工夫です。

## 仕組みを詳しく
SparseDrive はシーンを「インスタンスクエリ(instance query)」の集合で表します。インスタンスとは個々の物体や地図要素のことです。

主な部品:
1. 画像エンコーダ: 複数カメラ画像から多スケールの特徴マップ(解像度の異なる複数の特徴ピラミッド)を抽出します。ここで密 BEV は作りません。生の画像特徴をそのままスパースなクエリが参照します。多スケールにするのは、近くの大きな物体と遠くの小さな物体を別の解像度で捉えるためです。
2. 対称スパース知覚(symmetric sparse perception): 物体用クエリ(動的物体)と地図用クエリ(車線・道路境界)を並列・対称に扱います。各クエリは3D空間のアンカーを持ち、deformable attention で画像特徴のごく少数の点だけをサンプルして、物体の位置・大きさ・速度・向き、あるいは車線の形を推定します。物体と地図を別々の重い枝にせず対称構造にすることで、設計が単純になり相互の情報も流れます(物体クエリと地図クエリの間でも attention を取る)。時間方向には、過去フレームのクエリを記憶バンクとして引き継ぎ、速度推定やトラッキングに使います(Sparse4D 系の時間集約)。
3. 並列運動プランナ(parallel motion planner): 「他車がどう動くか(motion prediction)」と「自車がどう動くか(planning)」を、別々の直列処理ではなく同じ枠組みで同時に解きます。自車も1つのエージェントとみなし、周囲エージェントのクエリ・地図クエリと相互作用(attention)させながら、複数の候補軌跡(multi-modal、複数の起こりうる未来)を出します。自車については、ナビゲーション指令(直進/左折/右折)ごとに別のモード(motion mode)を用意し、衝突回避や快適性・経路追従などのコストで最良の軌跡を選びます。直列だと「予測の誤りが計画に一方向に伝わる」問題がありますが、並列・相互作用型はこれを緩和します。

数値イメージ: 密 BEV 系が 200×200=4万セルを毎フレーム更新するのに対し、SparseDrive は物体クエリ数百個 + 地図クエリ数百個程度しか持ちません。tensor で言えば、スパース表現は `(B, N_query, C)`(B=バッチ、N_query=数百、C=特徴次元、例256)という細い形で、密格子の `(B, C, 200, 200)` より桁違いに小さいテンソルです。4万セル × 256ch ≈ 1000万要素に対し、数百クエリ × 256ch ≈ 十数万要素で、約2桁の差になります。計算量・メモリがほぼ N_query に比例し、空き空間にコストを払いません。

学習は知覚・予測・計画をまとめた1段のE2Eで、各タスクの損失を同時に最適化します。出力は周囲物体の検出・予測軌跡と、自車の計画軌跡(waypoint 列)です。クエリは複数の Transformer 層を通るたびにアンカーを更新する反復改善(iterative refinement、各層で位置を少しずつ修正する)で精度を高めます。学習は実務上、まずスパース知覚だけを学習し、次に予測・計画を足す段階学習(staged training)が安定すると報告されています。これはクエリの初期化と二部マッチング(後述)が不安定になりやすいためです。

## 手法の系譜と主要論文
- DETR(Carion et al., "End-to-End Object Detection with Transformers", ECCV 2020): 物体検出を「物体ごとのクエリ」と二部マッチング損失(予測と正解を1対1に最適割り当てしてから損失を取る、ハンガリアンマッチング)で解き、アンカーやNMS(重複除去の後処理)を不要にした原点。スパース表現の思想的起点です。
- Deformable DETR(Zhu et al., ICLR 2021): クエリが画像の数点だけをサンプルする変形可能アテンションで、DETR の収束の遅さ(学習に500エポック必要だった)と計算コストを大幅改善。SparseDrive の画像参照機構の直接の祖先です。
- Sparse4D / Sparse4Dv2 / v3(Lin et al., 2022/2023, arXiv:2211.10581 ほか): 多視点カメラからの3D物体検出を、密 BEV を作らずスパースなアンカー+時間方向の特徴集約で行う。v2 で再帰的な時間融合(前フレームのインスタンスを引き継ぐ)、v3 で end-to-end トラッキングを実現し、検出だけでなく時系列を扱う点で SparseDrive の知覚部の基盤です。
- UniAD(Hu et al., CVPR 2023, Best Paper): 知覚・予測・計画をクエリで連結した統合E2Eの代表。ただし内部に密 BEV を使い計算が重い。SparseDrive はこの「計画を頂点に据える(planning-oriented)」思想を引き継ぎつつ密 BEV を外しました。
- VAD(Jiang et al., ICCV 2023): シーンをベクトル化して密 BEV 計算を減らし軽量化を進めた先行。SparseDrive はさらに踏み込み密 BEV を完全に排除しました。詳細は [VAD (Vectorized Scene Representation)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1094_vad-vectorized-scene)。
- SparseDrive 本体(Wenchao Sun, Xuewu Lin, Yang Zhang, Yongqi Fan, Shifeng Zhang, Xiaowei Zhao, Wei Wei, Sifa Zheng, "SparseDrive: End-to-End Autonomous Driving via Sparse Scene Representation", 2024, arXiv:2405.19620): 対称スパース知覚 + 並列運動プランナでシーンをインスタンスクエリ表現する統合E2E。動機は「密 BEV が計算・メモリ非効率で、予測と計画の直列接続が情報を制約していた」こと。

背景として、BEVFormer(Li et al., ECCV 2022)など密 BEV の代表に対するアンチテーゼとして読むと位置づけが鮮明になります。「世界を全部地図化してから考える」密派と、「重要なものだけ拾って考える」スパース派の対立構図です。

## 論文の実験結果(定量データ)
評価は主に nuScenes(都市部走行のマルチカメラ・大規模データセット、1000シーン・6カメラ・約140万枚の画像)で行われ、検出・予測・計画の各指標を一度に見ます。指標の意味を添えます。

- 検出: mAP(mean Average Precision、検出の正確さ。高いほど良い)と NDS(nuScenes Detection Score、検出と速度・属性・向きの推定を総合した nuScenes 独自の指標)。
- 予測: minADE / minFDE(複数候補軌跡のうち真値に最も近いものと真値との、全時刻の平均/最終位置の誤差。低いほど良い)、EPA(End-to-end Prediction Accuracy、検出を含めた予測の総合精度)など。
- 計画: L2 誤差(planning の予測軌跡と人間の実走行との平均距離、メートル。1秒/2秒/3秒先の平均で報告。低いほど良い)と Collision Rate(衝突率、計画軌跡が他物体とぶつかる割合。低いほど良い)。

報告によれば、SparseDrive は nuScenes で UniAD や VAD と同等以上の計画性能(L2 誤差・衝突率)を保ちながら、推論速度を大きく上げました。論文では UniAD に対しておよそ4倍前後の推論高速化(FPS で大きく上回る)と、学習時間・メモリの削減を達成したとされています。計画の平均 L2 誤差はおよそ0.6m前後、平均衝突率は0.1%前後のオーダーで、UniAD/VAD と比較して同等もしくはわずかに改善という報告で、「軽くしても性能を落とさない、むしろ速さで圧勝する」という主張の核です。検出 mAP・NDS でもスパースながら密 BEV 系に匹敵する水準を保ったとされます。

L2 誤差0.6mという数字の意味を補足します。これは「3秒後までの自車予定位置が、人間の実走行から平均0.6mずれる」ことを表します。車線幅が約3.5mなので、0.6mのずれは車線内に収まる範囲で、人間に近い運転と言えます。衝突率0.1%は「1000回の計画軌跡のうち約1回が他物体と幾何的に重なる」水準です。

アブレーションでは次の傾向が報告されています。

- 並列運動プランナを直列構成に戻すと、予測と計画の整合が崩れて計画性能(特に衝突率)が悪化する。予測と計画を相互作用させる並列設計が効いているという結果で、SparseDrive の中心主張を支えます。
- 対称スパース知覚で物体と地図を対称に扱う構成が、片方を簡略化した非対称構成より知覚・計画ともに安定する。地図情報が計画の文脈として効くためです。
- 時間方向の特徴集約(過去フレームのクエリを引き継ぐ)を外すと、速度推定や予測が悪化する。スパースでも時系列の情報は重要だという結果です。静止画1枚では物体の速度は本質的に推定困難なので、これは理にかなっています。

これらの数値の意味は「密 BEV を捨ててスパースなクエリ表現にしても、計画の正確さ・安全さ(L2・衝突率)を犠牲にせず、むしろ計算効率を桁で改善できる」という点に集約されます。なお絶対値は nuScenes の評価プロトコル(open-loop、過去ログ上での再生評価)に依存し、open-loop の L2 誤差は実際の運転能力を完全には反映しないという批判(自車の過去軌跡だけで未来をほぼ当てられてしまう「ショートカット」問題)もあるため、研究上は「ベースライン比の改善方向」と「速度・メモリの優位」を併せて読むのが妥当です。

## メリット・トレードオフ・限界
メリット
- 密 BEV を作らないため計算・メモリが大幅に少なく、推論が速い。車載のリアルタイム実行(限られた車載GPUでの高頻度推論)に向きます。
- 運動予測と計画を並列・対称に扱い、エージェント間の相互作用を直接モデル化できる。予測誤差が計画へ一方向に蓄積する問題を緩和します。
- 重要な物体・車線だけを表すので、知覚範囲や解像度を上げても計算が二乗で爆発しにくい。遠方まで見る長距離知覚と相性が良いです。

トレードオフ・限界
- クエリ数が概ね固定で、想定を超える多数の物体(密集した交差点、渋滞)では取りこぼしの恐れがあります。クエリ数を増やせば対応できますが、その分コストが戻り、二部マッチングも不安定になりやすいです。
- 密な空間情報(自由空間の細かい形状、占有グリッド的な連続情報)を表しにくい。物体・車線という「点・線」では表しづらい障害物(工事帯、倒木、不定形物、はみ出した荷物)に弱い側面があります。検出カテゴリに無いものは存在しないことになってしまいます。
- deformable attention などスパース機構は学習の安定化・初期化・収束に工夫が要り(段階学習・クエリ初期化・マッチングの工夫)、ハイパーパラメータ調整の手間がかかります。
- 評価が nuScenes 中心(open-loop)で、地図統合の仕方や多様な実データ、閉ループ(closed-loop、実際に走らせて誤差が累積する状況)評価での汎化は別途検証が必要です。

## 発展トピック・研究の最前線
- スパース表現と占有予測の融合: スパースなクエリでは表しにくい自由空間を、スパース占有(sparse occupancy、占有のある所だけ格子を持つ)として効率的に併用する研究が進んでいます。点・線・面をハイブリッドに扱い、不定形障害物への弱点を補う方向です。
- 端から端までスパース化した知覚バックボーン: Sparse4D 系の進化で、検出・追跡・予測を一貫してスパースに保つ流れが強まっています。NMS や明示トラッカーのような非微分の後処理を消し、全部を学習で一体化します。
- 閉ループ評価への対応: open-loop の L2 誤差が運転能力を測りきれない問題に対し、CARLA や nuPlan、近年の Bench2Drive のような閉ループ/計画特化ベンチマークでスパースE2Eを評価する動きがあります。閉ループでは小さな誤差が累積して破綻するので、open-loop より厳しい試験です。
- 大規模化と基盤モデル: スパースクエリを言語・視覚言語モデルと接続し、シーン理解や説明可能性を高める研究も登場しています。DriveTransformer のようにスパースクエリを設計原則(タスク間・空間・時間の3種の attention)に据えるアーキテクチャとも合流しつつあります。
