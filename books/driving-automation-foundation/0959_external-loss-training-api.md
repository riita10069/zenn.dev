# 外部ロス方式の学習API

## ひとことで言うと

フローマッチングや拡散モデルの学習で、「ノイズ入りの途中状態」と「お手本となる答え(目標速度)」をモデルの外で先に作っておき、モデルには予測だけさせて、その差(誤差)を外側で計算する設計です。学習と推論で入出力の意味が混ざらないよう、データ生成・予測・損失計算の役割分担を明確にするのが要点です。生成モデルの学習が「推論とは別物」であることを API レベルで表現する設計パターンです。

## 直感的な理解

料理の練習にたとえます。本番(推論)は「材料を渡されて、完成した料理を作る」ことです。一方で練習(学習)は違います。先生が「この半分できた状態から、次にすべき一手は何か」を問題として用意し、生徒はその一手だけを答え、先生が模範解答と照らし合わせて採点します。生徒に毎回完成品を作らせるのではなく、「中間状態」と「正解の一手」を先生が準備するのです。

生成モデルの学習はこの練習に似ています。本番は「ノイズから完成データを作る」ですが、学習で毎回完成データを作らせて採点するのは非効率かつ不安定です。代わりに「ノイズと正解を混ぜた中間状態」を作り、「その地点で進むべき方向(速度)」だけをモデルに当てさせ、正解の方向との差を測ります。この「中間状態と正解の方向を外で作る」役割を担うのが、本トピックの外部ロス方式です。なぜわざわざ外で作るのか。それは、同じモデルの呼び出し口が「学習では方向を返す」「推論では完成データを返す」と意味を変えると、極めて見つけにくいバグの温床になるからです。

## 基礎: 前提となる概念

学習と推論の非対称性。生成モデルは、学習時と推論時でやることが大きく違います。推論は「ノイズからデータを作る」(出力=完成したサンプル)、学習は「ある途中状態での速度や、加えたノイズを当てる」(出力=速度ベクトルやノイズ)です。同じ関数がモードによって返すものの意味を変えると、間違えやすくなります。

具体的な事故例を挙げます。ある実装で `forward` が学習モードでは速度を返すのに、呼び出し側がそれを軌道だと思って `MSE(出力, 軌道の正解)` を計算してしまったとします。出力は速度なのに軌道の正解と比べてしまうので、まったく意味のない損失で学習が進みます。しかもエラーは出ません(両者の形状が同じ `[B, 128]` だから)。こうした「型は合うが意味が違う」バグは発見が極めて難しく、学習が収束しない原因を何日も探すことになります。外部ロス方式はこの種の事故を構造的に防ぎます。

MSE(Mean Squared Error、平均二乗誤差)とは、予測と正解の差を二乗して平均した損失です。`MSE(a,b) = mean((a-b)^2)`。差が小さいほど0に近づき、回帰問題(連続値を当てる問題)の標準的な損失です。フローマッチングの学習は、この MSE で速度を回帰する単純な形になります。

最適輸送パス(OT path)と直線パス。フローマッチングでは、ノイズ x_0 とデータ x_1 をどう結ぶかでパスが決まります。最も単純で扱いやすいのが直線パスで、中間状態を `u_t = (1-t)*x_0 + t*x_1` と定義します。これは「ノイズと正解を t:(1-t) の比で混ぜた点」です。このパスに沿った目標速度は時刻によらず一定で `target_velocity = x_1 - x_0`(ノイズから正解へまっすぐ向かう方向)になります。直線なので速度が一定、という性質が学習も推論も単純にします([Euler法によるODE推論サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0218_euler-ode-sampling-inference) の Rectified Flow と同じ発想)。

## 仕組みを詳しく

フローマッチングの学習1ステップは、数学的に次の手順です。軌道生成を例にします。

1. お手本のデータ x_1(`trajectory_target`、データセットの正解軌道)を用意。
2. ノイズ x_0 を標準正規分布からサンプリング(x_1 と同じ形状)。
3. 時刻 t を [0,1) からサンプリング(各サンプルごとに別の t)。
4. 直線パス上の中間状態 `u_t = (1-t)*x_0 + t*x_1` を作る。
5. その地点での目標速度 `target_velocity = x_1 - x_0` を計算。
6. モデルに u_t と t(と条件)を渡して速度予測 v_pred を得る。
7. 損失 `= MSE(v_pred, target_velocity)`。

このうち 1〜5(データ作り)を、学習データ構成用の明示的な関数(以下では仮に `build_training_batch` と呼びます)に切り出します。

```
def build_training_batch(trajectory_target):
    x_0 = randn_like(trajectory_target)                 # ノイズ
    t   = sample_timesteps(B)                           # 時刻 [B]
    u_t = (1 - t)[:,None] * x_0 + t[:,None] * trajectory_target
    target_velocity = trajectory_target - x_0           # x_1 - x_0
    validate_shapes(u_t, t)                             # 形状チェック
    return u_t, t, target_velocity
```

学習スクリプト側はこうなります。

```
u_t, t, target_velocity = build_training_batch(trajectory_target)
v_pred = model(noisy_input=u_t, timestep=t,
               condition=(vis_hist, ego), mode="train")
loss = mse_loss(v_pred, target_velocity)   # ← 損失はモデルの外で計算
```

損失計算がモデルの外にあるのが「外部ロス方式」の名の由来です。なぜこの分担が安全か。共通の基底インターフェースが2つの契約を定めます。1つ、推論用の呼び出し口(`forward()`)は常に完成したサンプルだけを返す。中間量(速度)は絶対に返さない。2つ、学習用の中間テンソル(ノイズや目標速度)は専用経路でのみ受け渡し、推論経路に漏らさない。この分離により「速度をデータと取り違える」事故が構造的に起きません。型(形状)だけでなく「どの入口から入れたか」で意味が区別されるからです。

時刻サンプリングの工夫。手順3の時刻 t は、単純な一様分布 U(0,1) でもよいのですが、分布を偏らせると学習が安定・高品質になることが知られています。代表的な選択肢が3つあります。一様 U(0,1) は教科書的でクセがない。Beta 分布や logit-normal 分布は中央や端に重みを寄せられる。たとえば「t が小さい側(ノイズに近い側)に偏らせる」と、速度場を当てるのが難しい領域を重点学習でき、最終的な生成品質が上がるという経験則があります。ロボット制御のフローマッチングポリシー(pi-0 系)や画像生成(Stable Diffusion 3 の logit-normal)で、この種のスケジュールが採用されています。

```
時刻分布の例:
 一様:        |■■■■■■■■■■|   0……1 を均等
 ノイズ寄り:  |■■■■■■□□□□|   t 小 を厚く(難所を重点学習)
 中央寄り:    |□□■■■■■■□□|   logit-normal、中間 t を厚く
```

なぜ中間状態を作って速度を当てるのか。完成データを毎回生成させて採点する(推論をそのまま学習に使う)と、推論は多ステップの逐次積分なので勾配計算が重く不安定です。代わりに「1点の中間状態での速度」を当てる回帰にすると、各ステップが独立した単純な MSE になり、ミニバッチで効率よく安定に学習できます。これがフローマッチング/拡散モデルが「単純な回帰目的」で学習できる理由であり、設計上の最大の利点です。

## 手法の系譜と主要論文

「中間状態を外で作り、予測の差を回帰する」学習目的は、拡散モデルとフローマッチングの研究で確立されました。

DDPM(Ho et al., 2020, NeurIPS)は、拡散モデルの学習を「加えたノイズを予測する単純な MSE」に簡略化できることを示しました。中間状態(ノイズを加えた x_t)を作り、加えたノイズ epsilon を当てる、という構図はまさに本トピックの原型です。学習がノイズ予測の回帰で済むことが拡散モデル普及の鍵でした。

Flow Matching for Generative Modeling(Lipman et al., 2023, ICLR, arXiv:2210.02747)は、「直線(最適輸送)パス上で目標速度 x_1 - x_0 を MSE で回帰する」という学習目的を定式化しました。上記の学習データ構成の数式はこれに一致します。拡散モデルより少ないステップで生成でき、学習が単純な回帰で済むのが効果です。

Rectified Flow(Liu et al., 2023, ICLR, arXiv:2209.03003)は直線パスの理論を整理し、reflow という再学習で軌跡をさらに直線化できることを示しました。直線パスでは目標速度が一定になり、外部ロスの実装が最も単純になります。

ロボット制御では、Diffusion Policy(Chi et al., 2023, RSS, arXiv:2303.04137)が「観測を条件に行動列の生成モデルを学習し、推論はサンプリング」という train/inference の非対称性を明確にし、これらを混同しない API 設計の重要性を実例で示しました。pi-0(Black et al., Physical Intelligence, 2024, arXiv:2410.24164)とその後継 pi-0.5 は、フローマッチングによる汎用ロボットポリシーで、学習データ(ノイズ・時刻・目標)を外で構成しモデルは速度予測に専念する、という外部ロス型インターフェースを採用しました。時刻サンプリングを一様でなく偏らせる工夫もここで実証されています。

画像生成では Stable Diffusion 3(Esser et al., 2024, ICML, arXiv:2403.03206)が、rectified flow の学習で時刻を logit-normal 分布からサンプリングすると一様より一貫して良いことを大規模に検証し、時刻スケジュールの重要性を定量的に裏づけました。

## 論文の実験結果(定量データ)

外部ロス型の学習目的と時刻スケジュールがもたらす効果は、これらの論文で定量化されています。

Flow Matching(Lipman et al.)は ImageNet で、同じ計算予算下で拡散(score matching)ベースより低い FID(生成画像と本物の特徴分布の距離。低いほど良い)と低い NLL(負の対数尤度、データの当てはまりの良さ。低いほど良い)を達成しました。特に少ステップ推論時の品質低下が拡散より緩やかで、「直線パス+回帰」の組み合わせの効率を示しました。

Stable Diffusion 3(Esser et al.)は、時刻サンプリング分布のアブレーションを系統的に行いました。一様サンプリング、logit-normal、mode サンプリングなどを比較し、logit-normal(中間 t を厚くする分布)が複数のモデルサイズで一貫して最良の検証損失と FID を出すことを報告しました。これは「どの時刻を重点的に学習するかが生成品質を左右する」ことの直接的な証拠です。数値の意味として、時刻スケジュールを変えるだけで FID が体感できる差で改善するため、モデル構造を変えずに性能を引き出せる安価な手段として重視されています。

pi-0(Black et al.)は実機ロボットの多様なタスク(洗濯物畳み、箱詰めなど)で、フローマッチングポリシーが従来手法より高いタスク成功率を達成したことを報告しました。アブレーションで、行動を連続フローで生成する設計が、行動を離散トークンに量子化する方式より滑らかで高精度な制御を生むことを示しています。

これらに共通するのは、学習目的を「中間状態での予測の回帰」に単純化しつつ、時刻分布という1つのハイパーパラメータで難所を重点学習することで、モデルを変えずに品質を底上げできるという知見です。

## メリット・トレードオフ・限界

メリット。学習と推論の入出力の意味が分離され、「速度をデータと取り違える」型の重大バグが構造的に防げます。損失が外部にあるため、複数の損失項(将来の補助損失や正則化)を組み合わせやすく、研究的な拡張に強いです。時刻スケジュール(Beta / logit-normal)で難所を重点学習でき、モデル構造を変えずに品質を上げられます。データ生成関数に形状検証を仕込めば、学習データの形ミスを早期に検出できます。

トレードオフと限界。呼び出し側がデータ生成関数の戻り値(u_t, t, target_velocity)を正しく扱う責任を負うため、完全自動の単一 `forward` よりは手数が増えます。時刻スケジュールはハイパーパラメータ(分布の形状パラメータ)が増え、調整対象が増えます。外部ロス方式は損失定義が学習スクリプトに散らばりやすく、チームで開発する場合は規約やレビューで縛る必要があります。

未解決の課題。最適な時刻分布はタスク・データ・モデルに依存し、理論だけでは決められず実験で探す必要があります。また、条件付き生成(BEV や指示で条件づける軌道生成)では、条件の強さをどう注入し学習時にどう扱うか(classifier-free guidance のドロップアウト率など)が品質に効き、外部ロス API にその制御をどう組み込むかは設計上の論点として残っています。

## 発展トピック・研究の最前線

時刻スケジュールの自動化(学習中に難所を検出して動的に重みを変える)、複数の損失項(速度回帰に加え、物理制約や安全制約を補助損失として足す)を外部ロス枠で統合する研究が進んでいます。自動運転の軌道生成では、複数の妥当な未来を出すための多様性確保と、生成軌道が運動学的・安全的に実現可能であることを学習で保証する制約付きフローマッチングが最前線です。また蒸留(consistency / rectified flow の reflow)と外部ロス学習を組み合わせ、少ステップ推論用のモデルを効率よく作る流れも活発です。
