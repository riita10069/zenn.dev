# モデルの forward インタフェース設計（入出力の契約と整理統合）

## ひとことで言うと

ニューラルネットワークのモデルを呼び出すときの「何を受け取り、何を返すか」の取り決めを、関数のシグネチャ（signature, 引数と戻り値の並び）と呼びます。エンドツーエンド自動運転のように内部構造を大きく作り替える局面では、このシグネチャも一緒に変わります。本ノートでは、内部アーキテクチャの刷新が外側のインタフェースにどう波及するか、そして「複数の細切れな入出力を 1 本の意味のある隠れ状態に整理統合する」という設計判断が、なぜ理にかなっているのかを、初心者にも分かるように解説します。具体例として「過去シーンの圧縮ベクトルを入力で渡し、別の圧縮ベクトルを出力で返す」古い設計を、「自車の意図を表す 1 本の隠れ状態 `ego_hidden` に統合する」新しい設計へ移す変更を扱います。

## 直感的な理解

ある会社の受付窓口を想像してください。最初は「申請書 A」「申請書 B」「身分証のコピー」を別々に出させ、窓口は「受理番号」「整理券」「控え」を別々に返していました。手続きが増えるたびに窓口の前後で大量の紙が行き交い、どの紙が何のためのものか、職員も来客も分からなくなります。

ここで業務を見直し、「必要な情報は窓口の奥にある共有データベースに直接アクセスして取る」方式に変えたとします。すると、来客は余計な紙（申請書 B）を持ってこなくてよくなり、窓口は「あなたの手続き状況を表す 1 枚の整理票」だけを返せばよくなります。複数の細切れな紙が、意味の通る 1 枚に統合されたわけです。

モデルのシグネチャ変更も同じです。内部が「奥のデータベース（空間特徴マップ）に直接アクセスする」方式に変わったので、外から渡していた余計な入力（圧縮済みの視覚ベクトル）が不要になり、返していた複数の細切れな出力が「自車の意図を表す 1 本の隠れ状態」に統合される。これがインタフェース整理の本質です。

## 基礎: 前提となる概念

用語を順に噛み砕きます。

- forward メソッド: PyTorch では、モデルは「関数のように」呼び出します。`model(x, ...)` と書くと、内部で `forward(x, ...)` が呼ばれ、これがモデルの入口です。「カメラ画像などを入れると軌道などを返す」窓口だと思ってください。
- シグネチャ（signature）: forward が受け取る引数の並びと、返す戻り値の並びの取り決め。`trajectory, ego_hidden, future = model(x, egomotion_history)` のように、位置と意味が決まっています。
- 契約（contract）: モデルを使う側のコード（学習スクリプト、推論スクリプト、評価、テスト、データセットの受け渡し）は、すべてこのシグネチャに依存しています。引数や戻り値を変えると、依存している全箇所を直さないと壊れます。だからシグネチャは「使う側と使われる側の契約」です。
- 隠れ状態（hidden state）: ネットワーク内部で計算される、意味を凝縮したベクトル。ここでの `ego_hidden (B, 256)` は「自車がこれからどう振る舞うつもりか（意図）」を 256 個の数で表したもの、と捉えてください（`B` はバッチサイズ）。
- 下流タスク（downstream task）: あるモジュールの出力を受け取って次の処理を行う部分。ここでは「未来のシーンを予測する部品」が下流にあたり、`ego_hidden` を受け取って自車意図を反映します。

なぜシグネチャ設計が重要かを一段深く言うと、インタフェースは「変更のコストが集中する場所」だからです。内部実装はカプセル化（外から隠す）できますが、シグネチャは外に露出しているため、変えると波及範囲が広い。逆に言えば、良いシグネチャは「内部をどう変えても外を壊しにくい」抽象を選んでいます。

## 仕組みを詳しく

### 変更前のシグネチャ（細切れな入出力）

```python
# 変更前
trajectory, compressed_visual, future = model(x, visual_history, egomotion_history, ...)
```

- 入力 `x`: カメラ画像（複数視点）などの主入力。形は `(B, N_views, C, H, W)` のような多視点画像テンソル。
- 入力 `visual_history`: 過去シーンを小さな次元に圧縮した記憶（例: `(B, 64, 14)`）。詳しくは [視覚履歴をプランナの初期隠れ状態 h_0 に注入する](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0212_visual-history-input)。
- 入力 `egomotion_history`: 自車がどう動いてきたかの履歴。
- 出力 `trajectory`: 予測軌道（例: `(B, 128)` = 複数ステップ × 各時刻の位置/速度信号）。
- 出力 `compressed_visual`: 圧縮した視覚特徴ベクトル（下流に渡していた）。
- 出力 `future`: 未来予測モジュールの出力。

ここでの設計上の弱点は、「入力の `visual_history`」と「出力の `compressed_visual`」が、どちらも"圧縮した視覚的文脈"という似た役割を、別々の場所・別々の名前で担っていたことです。概念が二重化し、呼び出し側は何を渡し何を受け取るのか直感的に分かりにくい。

### 変更後のシグネチャ（1 本に統合）

```python
# 変更後
trajectory, ego_hidden, future = model(x, egomotion_history, ...)
```

変更点を 1 つずつ説明します。

1. `visual_history` 入力を削除: 圧縮シーン記憶を渡す引数をやめた。新しい内部プランナは空間特徴マップ（BEV, 鳥瞰図特徴）に直接 attention するため、事前に潰したベクトルを外から渡す必要がなくなった。呼び出し側は引数を 1 つ減らせる。
2. `compressed_visual` 出力を削除: 圧縮視覚ベクトルの出力をやめた。
3. `ego_hidden` 出力を新設: 軌道生成の過程で得られる「自車の意図を表す 256 次元の隠れ状態」`ego_hidden (B, 256)` を返すようにした。これが旧 `compressed_visual` の役割（下流への文脈受け渡し）を引き継ぐ。具体的には、未来予測モジュールが、旧来の強く圧縮されたベクトルではなく、この情報量の多い 256 次元状態を受け取り、自車意図を空間方向のチャネルごとのバイアスとして注入する。

要するに「2 つの別物（入力の visual_history と出力の compressed_visual）」を「1 本の ego_hidden」に整理統合した、というのが本質です。

```
before:  入力  visual_history ┐                出力  compressed_visual ┐
                              ├ 似た役割が二重化              ├ 概念が分散
                              ┘                                       ┘

after:   入力から visual_history を撤去（内部で BEV に直接 attention）
         出力は ego_hidden (B,256) に一本化 → 下流へ引き回す
```

### なぜシグネチャ変更は波及するのか

シグネチャは契約なので、変えると依存箇所すべてを直す必要があります。典型的に波及する箇所は次の通りです。

- モデル本体: 内部プランナの差し替えとシグネチャ本体。
- 速度ベンチマーク・推論スクリプト: 新しい呼び出し並びへの更新。
- データセット側: 不要になった `visual_history` を渡す処理の削除。
- テスト: モックや期待値を新インタフェースに合わせて書き直し。位置引数の順序（`x` の次が `egomotion_history` であって `visual_history` ではない）が変わるため、テストでこの並びを固定して回帰を防ぐ。

これは「シグネチャ変更がいかに広く影響するか」を体現します。だからこそ、インタフェースは最初から「概念的に最小で、内部変更に強い」形を選ぶのが望ましく、変えるときは依存箇所を漏れなく直し、テストで契約を固定する必要があります。

## 手法の系譜と主要論文

シグネチャ変更そのものは実装の話ですが、「何を入出力に選ぶか」という設計判断は、その背後の手法選択を反映します。エンドツーエンド自動運転では「学習可能な自車クエリ／隠れ状態を下流に引き回す」設計が定着しており、`ego_hidden` を出力に昇格させる発想はこの系譜に沿います。

- BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）: 多視点カメラから BEV 特徴を構築し、空間・時間の attention でシーンを保持。これにより「圧縮した固定ベクトルを外から渡す」必要がなくなり、内部で空間特徴を直接参照できる、という流れの起点。
- UniAD（Hu et al., CVPR 2023, Best Paper, arXiv:2212.10156）: 知覚・予測・計画を 1 本のクエリベースのパイプラインに統合。自車意図を学習可能な ego query で表し、それを下流タスクに引き回す設計。`ego_hidden` を出力に昇格させた発想と一致する。クエリを共有して複数タスクをつなぐと一貫学習できる反面、インタフェースが密結合になり 1 箇所の変更が全体に波及しやすい、というトレードオフも共通する。
- VAD（Jiang et al., ICCV 2023, arXiv:2303.12077）: ベクトル化シーンを直接参照する方向で、圧縮した固定ベクトルを入力から外す設計の後ろ盾。
- リファクタリングの工学: Martin Fowler「Refactoring」（第2版, 2018）が、「外部の振る舞いを保ちつつ内部構造を改善する」体系を与えている。シグネチャ変更は外部振る舞いを変える破壊的変更なので、Fowler の言う"安全な小さなステップ"とテストによる保護網が特に重要になる。

## 論文の実験結果（定量データ）

シグネチャ設計はアルゴリズムではないため、それ自体の精度実験はありません。背後の設計判断（自車クエリ／隠れ状態を引き回し、空間特徴を直接参照する）の有効性は、上記論文の計画性能で測られます。指標は nuScenes 上の L2 変位誤差（予測軌道と実走行軌道の平均ずれ、メートル。小さいほど良い）と衝突率（予測軌道での衝突割合、%。小さいほど良い）です。

- UniAD: 平均 L2 誤差およそ 1.0m 前後、平均衝突率およそ 0.31% を報告。ego query を介して知覚・予測・計画を一貫学習する設計が、特に安全指標（衝突率）で従来比の改善をもたらすことを示した。アブレーションでは、上流タスク（物体追跡・地図・占有予測）のクエリを計画に渡す経路を削ると計画性能が劣化することが報告されており、「意味のある隠れ状態を引き回す」インタフェース設計の価値を裏づける。
- VAD: 同等以上の計画品質を保ちつつ推論を大幅高速化（報告ではおよそ数倍）。圧縮した固定ベクトルではなくベクトル化シーンを参照する設計が、効率と品質を両立できることの実証。

注意として、nuScenes の open-loop 計画評価は自車の過去運動だけでかなり当てられてしまう（ego status ショートカット）という指摘があり、上記の絶対値は「およその水準」「報告による」ものとして読むべきです。設計判断の優劣を厳密に論じるには closed-loop 評価が望ましい、というのが現在の認識です。

## メリット・トレードオフ・限界

メリット

- 入出力が `ego_hidden` に整理され、概念が 1 本化されて理解しやすい。
- 不要な圧縮ベクトルの受け渡しが消え、コードが簡潔になる。
- 下流モジュールが情報量の多い 256 次元状態を受け取れ、表現力が上がる。

トレードオフ・限界

- 後方互換性が壊れる。旧シグネチャで呼ぶ既存コードはすべて修正が必要で、波及範囲が広い。
- 位置引数の並びが変わると、意味を知らない利用者が順序を間違えやすい（`x` の次が `egomotion_history` に変わる、など）。キーワード引数の利用やテストでの契約固定で緩和する。
- インタフェースを密結合にすると（クエリ／隠れ状態を多段に引き回す）、一箇所の変更が全体へ波及しやすくなる。疎結合と一貫学習のメリットはトレードオフの関係にある。
- 研究上の未解決課題として、「どの隠れ状態を、どの粒度で、どこまで下流に引き回すのが最適か」は定まっていない。引き回しすぎると密結合に、絞りすぎると情報損失になる。

## 発展トピック・研究の最前線

- クエリベース設計の標準化: ego query や各種クエリトークンで知覚・予測・計画をつなぐ設計が、エンドツーエンド運転の事実上の標準になりつつある。インタフェースを「クエリの集合」として扱うことで、タスク追加時の波及を抑える試みが進む。
- 生成プランナへの拡張: 軌道を逐次生成する代わりに拡散・フローマッチングで分布をサンプリングする場合、forward シグネチャは学習時に追加引数（ノイズ付き軌道、フロー時刻など）を取り、推論時は基本並びを踏襲する、という二相設計になる（[Flow Matching 軌道プランナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0123_flow-matching-trajectory-planner) を参照）。
- インタフェースの安定化と契約テスト: 大規模モデルでシグネチャ変更の波及を抑えるため、契約テスト（consumer-driven contract testing）の考え方を ML パイプラインに持ち込む動き。形状検証（[ランタイム形状検証（ValueError による早期失敗）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1010_runtime-shape-validation)）と組み合わせ、インタフェース変更を安全に行う。
