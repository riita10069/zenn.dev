# BEVセグメンテーション補助損失

## ひとことで言うと

複数カメラから作った鳥瞰図（BEV、上空から見下ろした地図状の特徴）に、小さな分類器（セグメンテーションヘッド）を付け足し、「ここは走行可能な路面」「ここは車線」「ここは車」「ここは歩行者」とBEVグリッドの各マスを塗り分けさせて学習させる、補助的な学習信号です。最終的に欲しい出力（例えば軌道予測）とは別に追加で課す目標で、BEVが幾何的に意味のある表現になるよう後押しします。

## 直感的な理解

軌道予測のような最終タスクだけを学習信号にすると、モデルは「BEV特徴が実際に空間を正しく表しているか」を直接チェックされません。極端に言えば、最終出力さえ当たれば、途中のBEVが内部的にぐちゃぐちゃでも損失は下がってしまいます。これでは、複数カメラを投影して作ったBEVが本当に「上から見た空間配置」を表しているか保証できません。

そこで、分かりやすい中間タスクを追加で与えます。「BEVグリッドの各マスが路面か車か歩行者かを当てよ」というセグメンテーションです。このタスクを解くには、BEV特徴が「どこに何があるか」を空間的に正しく保持していなければなりません。結果として、最終タスクにも効く良いBEV表現が育ちます。試験勉強で言えば、最終問題（応用問題）だけ解かせるより、途中で基礎の小問（場所当て）も解かせたほうが、土台がしっかりして応用力も伸びる、という発想です。

## 基礎: 前提となる概念

補助損失（auxiliary loss）とは、最終的に欲しい出力そのものではないが、途中の表現を良くするために追加で課す学習目標のことです。本命の損失（主損失）に重み付きで足し、ネットワークの中間表現に「こういう性質を持て」という圧力をかけます。

セグメンテーション（segmentation）とは、画像（ここではBEVグリッド）の各ピクセル（マス）に「これは何クラスか」のラベルを割り当てる作業です。物体検出が「物体を四角で囲む」のに対し、セグメンテーションは「マス単位で塗り分ける」ので、より細かい空間情報を扱います。BEVでのクラス例は、走行可能領域・車線・車両・歩行者などです。

これらを当てる学習には、分類の標準損失であるクロスエントロピー（cross-entropy）を使います。クロスエントロピーは「正解クラスにどれだけ高い確率を割り当てたか」を測り、正解クラスの予測確率が低いほど大きな罰を与えます。各マスが複数クラスに同時に属しうる場合（走行可能かつ車線、など）は、クラスごとに独立した二値分類として BCE（Binary Cross-Entropy）で解くマルチラベル設計にすることもあります。

セグメンテーション評価の標準指標は IoU（Intersection over Union）です。予測した領域と正解領域の「重なり（積集合）」を「合計（和集合）」で割った値で、0〜1の範囲を取り、1に近いほど予測が正解とよく重なっています。複数クラスの平均を取った mIoU（mean IoU）もよく使われます。

## 仕組みを詳しく

実装の流れは次のようになります。

ステップ1: BEV特徴を得る。多視点画像特徴と射影行列（[カメラキャリブレーション抽出（BEV用）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0264_camera-calibration-extraction) で作る `[3,4]` 行列）から、BEV融合モジュールが `bev_features: [B, C, H, W]` を出します（B はバッチ、C はチャンネル数で例えば256、H×W はBEVグリッドの解像度で例えば 450×300）。これが入力です。

ステップ2: 軽量セグヘッドを付ける。数層の畳み込みで `C` チャンネルを `num_classes` チャンネルへ落とします。

```
bev_features [B, 256, 450, 300]
  → Conv 3x3 → ReLU → Conv 1x1
  → seg_logits [B, num_classes, 450, 300]   # 各マスのクラス別スコア
```

`num_classes` は、走行可能領域・車線・車両・歩行者＋背景で5前後にすることが多いです。マルチラベルなら各クラス1チャンネルの二値出力にします。

ステップ3: 正解ラベルと突き合わせる。BEVグリッドと同じ解像度の正解ラベルマップ `[B, H, W]`（各マスにクラス番号）を用意し、クロスエントロピーを取ります。

```
seg_loss = CrossEntropy(seg_logits[B, K, H, W], seg_target[B, H, W])
```

ステップ4: 主損失と重み付きで合算する。

```
total_loss = main_loss + λ_seg · seg_loss
```

`λ_seg` は補助損失の重みです。大きすぎるとBEVがセグ向けに最適化されすぎて主タスクがおろそかになり、小さすぎると効果が出ません。手調整するのが普通ですが、後述するように不確かさで自動調整する手法もあります。

ステップ5: ラベルがあるときだけ有効化する。BEVの正解塗り分けは、地図アノテーションを持つデータセットでしか得られません。素の運転ログにはこれが付属しないことが多いので、ラベルがあれば `seg_loss` を有効化し、なければ `seg_loss = 0` として主損失だけ流す設計が現実的です。

数値イメージ:

```
入力 BEV  : [8, 256, 450, 300]
seg_logits: [8, 5,   450, 300]   # 5クラス
seg_target: [8,      450, 300]   # 各マス 0..4
seg_loss  : スカラー（クロスエントロピー）
total     : main_loss + 0.5 · seg_loss
```

重要な実装上の配慮として、観測カメラが届かないBEVマス（死角や遠方の希薄領域）の扱いがあります。そういう「見えていないマス」にもセグ損失を課すと、見えない場所のクラスを当てよという無理な目標になり学習が不安定になります。これを避けるには、可視マスのみで損失を取る（無効マスを `ignore_index` 指定でクロスエントロピーから除外する）必要があります。さらに、BEVグリッドのクラス分布は極端に偏ります（路面が大多数で歩行者はごく一部）。この不均衡（class imbalance）に対しては、頻度の逆数で重み付けしたクロスエントロピーや、難しいサンプルを重視する Focal Loss を使うのが定石です。

## 手法の系譜と主要論文

BEVをカメラだけから作ってセグメンテーションする流れは LSS（Lift, Splat, Shoot; J. Philion & S. Fidler, ECCV 2020, arXiv:2008.05711）が確立しました。各カメラ特徴を深度分布つきで3Dへ持ち上げ、BEVへ撒き、BEVセグ（車・道路レイアウト）を学習します。多視点カメラだけからBEVセグが学べることを示し、本トピックの妥当性の土台になりました。トレードオフは深度推定の曖昧さがセグのにじみになる点です。

BEVFormer（Z. Li et al., ECCV 2022, arXiv:2203.17270）は、BEVセグおよびBEV検出を主タスクとして学習しました。空間クロスアテンションで作ったBEV特徴に検出・セグヘッドを付け、map segmentation の IoU などで評価します。多くの簡易BEV融合実装はこの縮小版なので、「BEVFormer が主信号にしている損失を、軌道予測のような別の主タスクには補助として足す」というのが本トピックの位置づけになります。

HDMapNet（Q. Li et al., ICRA 2022, arXiv:2107.06307）は、カメラやLiDARからBEVで車線・横断歩道・道路境界をセグメンテーションして地図化する手法です。BEVセグのクラス設計（車線・走行可能領域・道路境界）の実践例として参考になります。

補助損失で表現を鍛える一般論としては、UniAD（Y. Hu et al., CVPR 2023 best paper, arXiv:2212.10156）が代表です。検出・追跡・地図・占有・動き予測という複数タスクを1つのBEV表現の上に積み、各タスクが互いの表現を良くし合う「マルチタスクで表現を鍛える」効果を示しました。BEVセグを補助に足す狙いと同じ方向です。トレードオフはタスク間の損失バランス調整が難しく、競合すると主タスクを損なう点です。この損失重み調整を自動化する古典的根拠が、Kendall et al.（"Multi-Task Learning Using Uncertainty to Weigh Losses for Scene Geometry and Semantics", CVPR 2018, arXiv:1705.07115）で、各タスクの不確かさ（観測ノイズ）を学習しながら損失重みを自動調整する手法を提案しました。`λ_seg` の手調整を置き換える選択肢になります。

## 論文の実験結果（定量データ）

LSS は nuScenes（6カメラ、1000シーンの自動運転データセット）で、車・道路レイアウトのBEVセグを IoU で評価し、当時の単眼ベースのBEV手法を上回りました。複数カメラを正しい投影で統合し、深度分布を学ぶことで、単一カメラからは見えない領域を補完できることを定量的に示しています。

BEVFormer は nuScenes で、BEV map segmentation の IoU を高水準で報告し、同時に検出ベンチでも NDS（nuScenes Detection Score、検出の総合指標。大きいほど良い）でおよそ 0.56、mAP でおよそ 0.42 を達成しました。アブレーションで、複数フレームのBEVを統合する temporal self-attention を外すと検出・セグの両方が落ちることを示し、時系列統合の寄与を定量化しています。

UniAD は nuScenes で、複数タスクを統合した場合と単独で解いた場合を比較するアブレーションを丁寧に行いました。報告によると、地図・占有・動き予測といった中間タスクを共通BEV表現の上で同時に学習すると、最終の計画（planning）の指標（衝突率、L2誤差＝予測軌道と正解軌道のメートル単位のズレ）が、それらを抜いた構成より改善します。具体的には、補助的な認識タスクを足すことで計画の衝突率が下がることを示し、「中間タスクが最終タスクの表現を鍛える」という補助損失の核心を、エンドツーエンド自動運転の文脈で実証しました。これは best paper を取った理由の1つです。

マルチタスクの損失重みについては、Kendall et al. が深度推定・セマンティックセグ・インスタンスセグの3タスク同時学習で、固定重みの単純な和より、不確かさで自動重み付けしたほうが各タスクの精度（IoU や深度誤差）が向上することを報告しました。手で `λ` を探さなくても、競合するタスク間のバランスをデータから学べることを示した点が重要です。

## メリット・トレードオフ・限界

メリットは、BEV特徴に「空間配置を正しく持て」という明示的な圧力をかけ、最終タスクに効く良い表現を育てやすいことです。セグ出力は人間が見て解釈できるので、BEVが壊れていないかのデバッグ・可視化にそのまま使えます。さらに任意（optional）に有効化できる設計にすれば、ラベルのあるデータでだけ効かせ、ない場合は無害にできます。

トレードオフと限界も明確です。BEVの正解ラベル（地図アノテーション）が必要で、対応するデータセットが限られます。地図APIから正解を生成するにも、自車位置の正確な推定（localization）が前提になり、ずれると教師ラベル自体が汚れます。損失重み `λ_seg` の調整は難しく、誤るとセグに引っ張られて主タスクが劣化します。セグヘッドぶん計算とメモリが増え、高解像度BEV（例えば 450×300）はそれだけでVRAMを多く消費するため、補助損失を足すとさらに圧迫します。観測のないマスや遠方の希薄領域の `ignore` 指定を誤ると、無理な目標で学習が不安定になります。クラス不均衡への対処（重み付け、Focal Loss）も必須で、放置すると「全部を路面と予測する」退化に陥ります。研究上の未解決課題としては、地図アノテーションなしで補助信号を作る方法（自己教師あり、擬似ラベル）、複数補助タスクの自動バランス、補助損失が本当に主タスクに転移しているかの検証方法などがあります。

## 発展トピック・研究の最前線

補助損失の発展として、地図アノテーションに頼らず補助信号を作る方向が活発です。LiDARや時系列の一貫性から擬似ラベルを生成する自己教師あり学習、画像基盤モデル（SAM など）でセグマスクを自動生成して教師に使う試みがあります。BEVセグ自体も、平面的な意味地図から3Dの占有グリッド（occupancy prediction、空間を立方体マスに区切って各マスが埋まっているか・何クラスかを当てる）へ拡張する流れが主流になりつつあり、より細かい立体構造を補助信号にできます。マルチタスクのバランスでは、Kendall 系の不確かさ重み付けに加え、勾配の干渉を抑える GradNorm や PCGrad のような勾配操作手法が研究されています。研究を始めるなら、まず地図アノテーション付きデータ（nuScenes など）でBEVセグ単独を学習して IoU を出し、それを軌道予測の補助に足したときに最終指標がどう動くかをアブレーションで測ると、補助損失の効き方を直接観察できます。
