# GPSからBEVマップタイル生成

## ひとことで言うと

GPS で記録された緯度経度の点列(車が走った座標の並び)と、OpenStreetMap などの無料の地図データを組み合わせて、「真上から見た道路地図の小さな画像(タイル)」を後から作り出す技術です。地図画像をあらかじめ持っていないデータセットでも、ニューラルネットワークが期待する「地図ビュー」という入力枠を埋められます。緯度経度という数値だけから、人間が見ても道路と進路がわかる俯瞰図を合成する、データ準備の一工程だと考えてください。

## 直感的な理解

自動運転の意思決定モデルを学習させるとき、入力としてカメラ映像だけを与える方式と、それに加えて「この先どっちへ進むべきか」を示すナビ的な地図を与える方式があります。後者のほうが、交差点での右左折や複雑な分岐で正しい行動を選びやすくなることが経験的に知られています。なぜなら、カメラ映像だけでは「次の交差点で右に曲がる」という意図が画像のどこにも写っていないからです。人間のドライバーがカーナビの画面をチラ見して進路を決めるのと同じで、モデルにも進路情報を別チャンネルで渡してあげると判断が安定します。

ところが現実のデータセットは寄せ集めです。あるデータセットには進路を描いた地図タイルが最初から同梱されているのに、別のデータセットには GPS の生ログ(緯度・経度の時系列)しか入っていない、ということが普通に起こります。地図ビューの入力枠が空のままだと、地図入力を前提に学習したモデルにそのデータを食わせられず、せっかくのデータが使えません。

ここで解こうとしている問題は「GPS の点列しかないデータから、地図入力ありのデータと同じ形式の地図タイルを合成し、複数データセットの入力フォーマットを揃える」ことです。身近に例えるなら、写真しか撮っていなかった旅行の記録から、後で地図アプリ上に足取りを描き起こして「移動経路つき地図」を作るのに似ています。元データに地図画像はなくても、位置情報さえあれば地図は再構成できる、という発想です。これはマルチデータセット学習(複数の出所の異なるデータを混ぜて1つのモデルを学習すること)を成立させるための、地味だが不可欠なデータ整備工程です。

## 基礎: 前提となる概念

このトピックを理解するには、いくつかの言葉を押さえておく必要があります。どれも初出なので丁寧に噛み砕きます。

BEV(Bird's-Eye View、鳥瞰図): 真上から見下ろした視点の表現です。カメラの前方視点(perspective view)が「自分の目線」だとすると、BEV は「ドローンを真上に飛ばして見下ろした地図」です。道路の形・自車と他車の位置関係・進路が一枚の平面に収まるので、空間的な配置を扱う処理と相性が良い表現です。なお、ここで作る「地図タイル」は厳密にはセンサから作る BEV 特徴量とは別物で、地図情報を BEV と同じ俯瞰の枠に描いたラスタ画像(ピクセルの2次元配列)です。両者を後段で重ね合わせて使うこともあります。

GPS と緯度経度: GPS は人工衛星からの信号で地球上の位置を測る仕組みで、結果は緯度(latitude、南北方向の角度)と経度(longitude、東西方向の角度)で表されます。消費者向け GPS の精度はおおむね数メートルから十数メートルで、トンネルやビル街では衛星信号が遮蔽・反射(マルチパス)して一時的に大きくずれることもあります。だから生の点列はガタガタに揺れます。

OpenStreetMap(OSM): 世界中の道路・建物・地形を、ボランティアが誰でも編集できる形で集めた無料の地図データベースです(Haklay & Weber, 2008, IEEE Pervasive Computing で紹介)。利点は無料で全世界をカバーすること、欠点は地域によって精度・更新頻度に差があり、田舎では道が抜けていることがある点です。

道路グラフ: OSM の道路網は「グラフ」というデータ構造で表せます。グラフとはノード(点)とエッジ(点をつなぐ線)の集まりのことで、ここではノードが交差点やカーブの折れ点、エッジが道路区間に対応します。各ノードには経度 x と緯度 y が属性として入っています。

OSMnx: OSM から道路グラフを取得・解析・可視化する Python ライブラリです(Boeing, 2017)。内部で Overpass API(OSM データの問い合わせ窓口サーバ)に通信して、指定地点周辺の道路グラフを取ってきます。

マップマッチング: ガタガタした GPS 点列を、実在の道路の上に吸い付けて整える処理です。詳細は [osmnx による道路ネットワーク取得とマップマッチング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0423_osmnx-map-matching) で扱います。

テンソル: ニューラルネットワークが扱う多次元配列のことです。画像は典型的に (チャンネル数, 高さ, 幅) という3次元テンソルで表され、RGB 画像ならチャンネル数は 3 です。

## 仕組みを詳しく

GPS 座標から地図タイルのテンソルを作るパイプラインは、典型的には次の4段階で構成されます。各段の入出力(shape を含む)を追いながら見ていきます。

### ステップ1: 道路網を取得する

自車(ego)の現在位置を中心に、半径 R メートル(よく使われる値は 800m 前後)の範囲の道路網を OSMnx で取得します。疑似コードで書くと次のようになります。

```
graph = graph_from_point((center_lat, center_lon), dist=R, network_type="drive")
```

`network_type="drive"` は「車が通れる道路だけ」を意味し、歩道や自転車専用道を除外します。返り値は道路グラフ(有向多重グラフ)で、典型的には数百〜数千ノード規模です。この取得は Overpass API へのネットワーク通信なので、1回あたり秒オーダーかかります。だから学習中に毎回呼ぶのは禁物で、後述の通りオフライン前処理に隔離します。

### ステップ2: GPS をマップマッチングする

GPS の緯度列・経度列を受け取り、各点を最寄りの道路ノードに吸着させ、ノード間を最短経路でつないで連続したルートを作ります。

```
matched_nodes = nearest_nodes(graph, lons, lats)   # 各 GPS 点 → 最寄りノード
route_nodes   = 隣り合うノードを shortest_path で連結
```

マップマッチングは失敗しうる前提で書くのが定石です。半径内に道が一本もない、グラフが分断されて経路が引けない、GPS がグラフ範囲外に飛ぶ、といったケースで失敗します。その場合は route を空にして、呼び出し側が「生の GPS 点だけを描く」フォールバックに切り替えます。詳しくは [osmnx による道路ネットワーク取得とマップマッチング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0423_osmnx-map-matching) を参照してください。

### ステップ3: 地図タイルをレンダリングする

道路網とマッチした経路を、自車を中心にした俯瞰画像に描きます。色の割り当ては、合成タイルを既存の地図入力タイルと見分けがつかなくするために、参照データセットのパレットに揃えるのが定石です。典型的な配色は次のようなものです。

```
背景       暗いグレー   #111111
道路       グレー       #444444
進む経路   明るい青     #00CCFF
生 GPS 点  赤           #FF3333
```

なぜ色を既存タイルに合わせるのか。画像バックボーン(後述)の前処理は、リサイズと正規化を通じて色やコントラストに反応します。既存タイルと同じ見た目にしておけば、前処理はこの合成タイルを区別なく扱え、地図エンコーダを追加学習し直さなくても流用できます。これは「ドメインギャップ(訓練元と適用先のデータの見た目の差)を描画段階で埋める」という設計判断です。色チャンネルに意味を持たせる(道路は灰、経路は青、というふうに)ことで、CNN が色フィルタで道路と経路を分離学習しやすくなる効果もあります。

座標は緯度経度のままだと歪むので、いったんメートル単位の平面に変換し、さらに自車の向きが画面の上を向くよう回転させてから描きます。この座標変換の詳細は [Ego中心マップタイルのレンダリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0413_ego-centric-map-tile) で扱います。出力は固定解像度の RGB 画像(例として 640×360 ピクセル)です。

### ステップ4: テンソル化する

描いた画像をバックボーンの前処理に通し、(3, H, W) のテンソルにします。3 は RGB の3チャンネル、H と W は高さと幅です。ここでバックボーンとは、画像から特徴を抽出する事前学習済みのネットワーク本体のことです。`timm`(PyTorch Image Models)のような画像モデルライブラリでは、各モデルに対応した前処理(指定サイズへのリサイズ、ImageNet 統計での正規化)が用意されており、それをそのまま使います。正規化とは各チャンネルから平均を引き標準偏差で割る操作で、ImageNet 統計の平均 (0.485, 0.456, 0.406)・標準偏差 (0.229, 0.224, 0.225) がよく使われます。

```
tensor = transform(image)   # (3, H, W) — カメラ画像タイルと同じ前処理
```

これでカメラ画像タイルと寸分違わぬ形式のテンソルになるので、モデルの地図ビュー枠にそのまま差し込めます。複数の入力ビューを同じ shape・同じ前処理で揃えることが、後段の融合(fusion、複数の入力特徴を1つに統合する処理)を素直にする鍵です。

### なぜオフライン前処理なのか

ステップ1の Overpass 通信は遅く、ネットワーク接続を必須とします。これを DataLoader(学習中に CPU 側で次のバッチを準備する仕組み)のホットパスに入れると、GPU がデータ待ちで遊んでしまい学習が止まります。そこで、道路グラフを座標の量子化キーでキャッシュし、データセットの全タイルを学習前に一括で描いておきます。この「事前レンダリングとキャッシュ」の設計は [オフラインのタイル事前レンダリングとネットワークキャッシュ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0692_offline-tile-prerendering-cache) で詳述します。

## 手法の系譜と主要論文

地図をニューラルネットの入力にする流れは、大きく「ラスタ地図(画像化)」派と「ベクトル地図(点列・グラフ)」派に分かれます。系譜を時系列で追います。

OSMnx(Geoff Boeing, 2017, Computers, Environment and Urban Systems)は、OSM から道路グラフを取得・解析・可視化する手続きを一つのライブラリにまとめ、研究者が数行で街路ネットワークを扱えるようにしました。それ以前は Overpass のクエリや座標投影を自前で書く必要があり、参入障壁が高かったのです。地図タイル生成の道路網取得・最近傍ノード・最短経路は、この OSMnx が提供する機能の上に成り立っています。

マップマッチングの古典は Newson & Krumm(2009, ACM SIGSPATIAL)で、隠れマルコフモデル(HMM)を使って GPS 軌跡を道路網に確率的に対応づける定式化を示しました。各 GPS 点が各道路に乗る確率(観測確率)と、隣接点間の道路の移動しやすさ(遷移確率)を掛け合わせ、最も尤もらしい道路列を選びます。タイル生成で使われる「最近傍ノード+最短経路」方式はこれより素朴ですが、目的(GPS を道路へ乗せる)は同じです。

地図をモデル入力にする自動運転研究では、ラスタ地図方式の代表が ChauffeurNet(Mayank Bansal, Alex Krizhevsky, Abhijit Ogale, 2018, arXiv:1812.03079, Waymo)です。自車中央・進行方向上向きのトップダウン画像に、道路・経路・周辺車両を別チャンネルで描いて CNN に入れ、模倣学習で運転方策を学びます。一方ベクトル方式の代表は VectorNet(Jiyang Gao et al., 2020, CVPR, Waymo)で、車線や軌跡を折れ線(ポリライン)の点列として表し、グラフニューラルネットで関係を学びます。さらに LaneGCN(Ming Liang et al., 2020, ECCV, Uber ATG)は車線グラフ上で畳み込みを行います。GPS からの地図タイル生成はラスタ地図方式に属し、画像バックボーンと直接つながる手軽さが強みです。最近では MapTR(Bencheng Liao et al., 2023, ICLR)のように、そもそも事前地図に頼らずセンサからベクトル地図をオンライン生成する流れもあり、ラスタ前処理とは対極の発想として理解しておくと位置づけが明確になります。

## 論文の実験結果(定量データ)

ChauffeurNet(Bansal et al., 2018)は、ラスタ化したトップダウン地図入力が運転方策学習に有効であることを示した代表例です。サンフランシスコ近郊で収集した約60日相当・26百万例規模の運転データで学習し、純粋な行動模倣だけのベースラインに対し、衝突・脱輪・経路逸脱といった失敗事象を起こすシナリオを大幅に減らせることを報告しています。とくに「最悪のケースを合成して学習させる(synthesizing perturbations、軌跡をわざと崩して回復を学ばせる)」拡張を加えると、ベースラインでは閉ループ走行で停止位置をはみ出す・障害物に近づくといった失敗が頻発したのに対し、提案手法ではそれらが顕著に抑えられた、というアブレーションが示されています。ここでの「閉ループ」とは、モデルの出力を実際に車(またはシミュレータ)に反映し、その結果の新しい状況を再びモデルに入れる評価のことで、誤差が積み重なるため一発予測(開ループ)より厳しい評価になります。アブレーションでは perturbation を抜く・補助損失(衝突や脱輪を罰する項)を抜くと、いずれも閉ループ失敗が増えることが示され、地図入力を土台に各要素が積み上がって効いている構図が読み取れます。

VectorNet(Gao et al., 2020, CVPR)は、Argoverse という軌跡予測ベンチマークで評価されました。指標は ADE(Average Displacement Error、予測軌跡と正解軌跡の各時刻の距離の平均)と DE@3s(3秒後の最終位置誤差)で、いずれもメートル単位で小さいほど良い指標です。VectorNet はラスタ地図を CNN に通す方式に対し、同等以上の精度をより少ない計算量で達成したと報告しています。具体的には、ラスタ方式の ResNet ベースのモデルと比べ、モデルのパラメータ数と演算量(FLOPs)を大幅に削減しつつ、社内大規模データセットでの予測誤差をおよそ12%改善したと述べています。これは「画像化はピクセルに無駄が多く、ベクトルのほうが幾何を効率よく表せる」という主張の数値的裏づけです。

これらの数値が意味するのは、地図情報を入れること自体が運転・予測性能を底上げするということ、そして表現方式(ラスタかベクトルか)に効率と精度のトレードオフがあるということです。GPS からのタイル生成はラスタ側に立つので、ChauffeurNet 系の「画像で地図を扱う」利点を享受しつつ、VectorNet が指摘する「ピクセル表現の非効率」という限界も引き継ぎます。

## メリット・トレードオフ・限界

メリットは次の通りです。第一に、地図入力を持たないデータセットでも地図ビューを後付けでき、複数データセットの入力形式を統一できます。これは異種データを混ぜて学習する(マルチデータセット学習)上で決定的に重要です。第二に、参照データセットのパレットに色を揃えることで、地図エンコーダを追加学習せずに既存の前処理・モデルへ流用できます。第三に、マップマッチが失敗しても生 GPS 描画にフォールバックするので、難しいクリップでも空タイルにならず、堅牢です。第四に、重い通信・描画をオフライン前処理に隔離することで、学習ループの速度を落としません。

トレードオフと限界もあります。Overpass フェッチが遅くネットワーク必須なので、大規模データでは事前レンダリングの計算時間とタイル PNG のストレージが要ります。OSM のカバレッジと鮮度に依存するため、地図が古い・欠けている地域では道路が抜けます。簡易マップマッチング(最近傍+最短経路)は平行路や複雑な交差点で誤マッチを起こしうる、という精度限界があります。そしてラスタ地図ゆえ、ベクトル地図ほどの幾何精度は持たず、解像度を超える細部(車線の本数や正確な曲率)は表現できません。たとえば解像度 2.5m/px のタイルでは、幅3.5m の車線2本を分離して描くことすら困難で、車線レベルの判断には不向きです。

さらに見落とされがちな失敗例として、heading(進行方向)推定が停車中に不定になり、タイルの向きが乱れるケースがあります。GPS の連続2点が同じ位置に重なると方向が定義できず、ノイズで地図が回転してしまいます。この問題は [Ego中心マップタイルのレンダリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0413_ego-centric-map-tile) で詳述します。

未解決の研究課題としては、合成タイルと本物タイルの間にわずかでも残るドメインギャップ(描画スタイル・座標精度の差)がモデルにどう影響するか、その差を埋めるための学習側の工夫(ドメイン適応やデータ拡張)をどう設計するか、が挙げられます。また、GPS 精度が低い・更新頻度が疎なデータでは、そもそも合成元の品質が悪く、生成タイルの信頼性が下がるという根本的な制約も残ります。

## 発展トピック・研究の最前線

近年は、ラスタとベクトルの中間として、地図を「クエリ可能なトークン列」として扱う Transformer ベースの手法が主流になりつつあります。VectorMapNet(Liu et al., 2023)や MapTR(Liao et al., 2023, ICLR)は、オンラインで車線・縁石をベクトルとして検出し、地図そのものをモデル内部で生成する方向に進んでいます。これは「事前に地図タイルを描く」アプローチとは逆に、「センサ入力からその場で地図表現を立ち上げる」発想で、HD マップ(高精細地図)への依存を減らす狙いがあります。MapTR は nuScenes ベンチマークの地図要素検出(車線分離線・横断歩道・道路境界)で、リアルタイムに近い推論速度を保ちながら従来手法を上回る平均適合率(mAP)を報告し、オンライン地図生成が実用域に入りつつあることを示しました。

一方で、GPS からの地図タイル生成のような「事前地図の活用」は依然として価値があります。とくに進路の意図(どこへ向かうか)はセンサだけからは復元できないため、ナビ情報を別チャンネルで与える設計は今も広く使われています。研究の最前線では、ラスタ地図・ベクトル地図・言語による経路指示などを一つのモデルにまとめて融合する方向と、地図への依存自体を学習で減らす方向が、並行して探索されています。
