# ピンホール投影による3D→2D変換

## ひとことで言うと

「車を中心とした3D空間のある一点（たとえば前方20m・高さ1m）が、カメラ画像のどのピクセルに写るか」を計算する幾何（図形の数学）の処理です。カメラの位置や向きの情報（外部パラメータ）とレンズの情報（内部パラメータ）を1つの行列にまとめ、3D座標にかけ算してから「遠いものほど小さく写る」割り算（透視除算）を行うと、画像上の座標が出てきます。コンピュータビジョンで最も基礎的かつ重要な計算の一つで、多視点カメラから BEV を作るあらゆる手法の足場になっています。

## 直感的な理解

BEV（鳥瞰図）を作るとき、各マスは「車から見た3D空間の点」として定義されます（[3D参照点とピラー（高さ方向）サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0909_3d-reference-points-pillar-sampling) 参照）。しかし、答え（その場所に何があるか）が詰まっているのはカメラ画像のほうです。だから「この3Dの点は、画像のどこを見ればいいのか」を計算できないと、特徴を拾いに行けません。この「3Dの点を画像の2D座標に変換する」のがピンホール投影です。

ピンホール（pinhole＝針の穴）カメラモデルとは、カメラを「光が小さな穴を1点だけ通って、その奥のスクリーンに像を結ぶ箱」として理想化した数学モデルです。実際のレンズは複雑ですが、多くの場合この単純なモデルで十分よく近似できます。このモデルのおかげで、3D→2Dの変換が「行列のかけ算」という扱いやすい形で書けます。

身近な例で言えば、目の前に窓ガラスを立て、片目をつぶって遠くの景色を見ながら、見える物の位置をガラスにペンで描き写すのを想像してください。あなたの目（針の穴）と物を結ぶ直線がガラス（画像面）と交わる点が、その物の写る位置です。遠くの物ほどガラス上では小さく中央寄りに集まります。これが透視（perspective）であり、ピンホール投影が数式で表すのはまさにこの作図です。

## 基礎: 前提となる概念

内部パラメータ（intrinsic parameters）。カメラのレンズと撮像素子の特性を表す `[3, 3]` の行列 K です。中身は焦点距離（fx, fy＝どれくらいズームか、ピクセル単位）と主点（cx, cy＝画像中心がどこか）です。標準形は `K = [[fx, 0, cx], [0, fy, cy], [0, 0, 1]]` で、同じ3D点でもズームが違えば写る位置が変わるため、この情報が要ります。

外部パラメータ（extrinsic parameters）。「車中心の座標系」を「カメラ中心の座標系」に移す変換で、回転 R（カメラの向き、`[3,3]`）と並進 t（カメラの取り付け位置、`[3]`）からなります。`[3, 4]` の行列 `[R | t]` で表します。カメラがどこに付いていて、どちらを向いているかを表します。

同次座標（homogeneous coordinates）。3D点 `[x, y, z]` の末尾に1を足して `[x, y, z, 1]` の4要素にする数学的な工夫です。なぜ1を足すかというと、回転だけでなく「ずらし（並進）」も含めた変換を、すべて1つの行列のかけ算で表せるようになるからです。通常の座標では並進は足し算、回転はかけ算と別扱いですが、同次座標にすると両方をかけ算に統一できます。

透視除算（perspective division）。投影の結果得られる `[u', v', w]` のうち、最初の2つを3つ目（深度に相当）で割る操作です。これが「遠いものほど小さく写る」遠近法の数学的本体です。

## 仕組みを詳しく

まず、内部パラメータと外部パラメータをかけ合わせて1つの投影行列にまとめます。

```
proj = K @ [R | t]       (形状 [3,4])
   K     : 内部パラメータ [3,3]
  [R|t]  : 外部パラメータ（ego→camera）[3,4]
   @      : 行列のかけ算
```

なぜ1つにまとめるかというと、3つの変換（車中心座標→カメラ座標→画像座標）を別々にやると手間で誤差も増えるからです。あらかじめ合成しておけば、1回のかけ算で済みます。複数カメラがあれば、カメラごとにこの `[3,4]` 行列を持ち、形状 `[V, 3, 4]`（V はカメラ台数）にまとめておきます。座標系の約束は、ego frame で X=前方、Y=左、Z=上を取るのが一般的です。

処理の流れを段階で追います。

第一に、3D点を同次座標にします。`[x, y, z]` の末尾に1を足して `[x, y, z, 1]` にします。

```
ref_world = [x, y, z]      →   ref_homo = [x, y, z, 1]   (末尾に1)
```

第二に、投影行列をかけます。各カメラの `[3,4]` 行列を、各点の `[4]` ベクトルにかけ、`[3]` の結果を得ます。多数の点と複数カメラを一括処理するため、テンソルの一般化された積（einsum など）を使うのが普通です。たとえば `[H*W, num_points, 4]` の点群と `[V, 3, 4]` の投影行列から、`[V, H*W, num_points, 3]` の投影結果を一発で計算できます。

```
projected = proj @ ref_homo     →   [u', v', depth]
```

この時点で結果は `[u', v', depth]` の3要素です。depth はカメラの光軸方向の距離（正なら前方）を表します。

第三に、透視除算を行います。`[u', v']` を depth で割ります。

```
u = u' / depth ,   v = v' / depth
```

これが遠近法の本体です。同じ大きさの物体でも、深度が2倍なら画像上では半分の大きさになります。実装では depth がゼロや負になると無限大や NaN（非数）が出るため、`depth.clamp(min=1e-5)` のような下限ガードを入れます。深度が0以下（カメラの後ろや真横）の点は、この後の可視性マスクで除外されるので、ここでは計算が壊れないようにするだけです。

第四に、ピクセル座標 `[u, v]` を画像サイズで割って 0〜1 の範囲に正規化します。後段のサンプリング（多くは `grid_sample`、[F.grid_sample による可搬な変形アテンション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0968_grid-sample-deformable-attention) 参照）が正規化座標を必要とするためです。

```
ref_2d = [u, v] / image_size      →   [0..1, 0..1]
```

数値例で全体像を示します。前方20m・左0m・高さ0m の点 `[20, 0, 0]` を考えます。同次座標は `[20, 0, 0, 1]`。前向きカメラの投影行列をかけると、たとえば `[2560, 1280, 20]`（u'=2560, v'=1280, depth=20m）のような結果になります。透視除算で u=2560/20=128、v=1280/20=64。画像サイズ256で割って `[0.5, 0.25]`。つまり画像の横中央・やや上に写る、と計算できました。距離を40mに変えると u'・v' も比例して増えますが depth も2倍になるため、画像上の位置はほぼ同じで、物体の見かけの大きさだけが半分になります。これが透視の効果です。逆に言えば、遠方の点は深度が大きいため、外部パラメータのわずかな角度誤差が透視除算後に大きな画素ずれへ増幅されます。これが後述のキャリブ感度の正体です。

## 手法の系譜と主要論文

ピンホールカメラモデルと内部・外部パラメータの定式化は、コンピュータビジョンの古典的教科書 Hartley & Zisserman, "Multiple View Geometry in Computer Vision"（2nd ed., 2004）にまとまっています。これは深層学習以前から確立した幾何で、特定の論文の発明というより分野の共通基盤です。同次座標を使う理由は、回転・並進・投影をすべて行列積で表して合成できるからで、効果は計算の単純化、トレードオフはレンズ歪み（広角カメラの樽型歪みなど）を別途補正しないとピンホール近似が崩れる点です。広角・魚眼を扱うには、ピンホールを一般化した Kannala & Brandt のモデル（IEEE TPAMI 2006）のような歪みを含むカメラモデルが必要になります。

この投影を BEV 構築の核に据えたのが BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）です。BEVFormer は3D参照点をカメラの投影行列で画像に落とし、当たったカメラの特徴だけを集めます。投影方向（3D→2D）を選んだのは、逆方向（2D→3D、深度を持ち上げる手法）が深度推定の誤差に弱いからです。効果は深度予測モジュールが不要なこと、トレードオフはカメラキャリブレーション（投影行列）が正確に必要なことです。

LSS（Lift, Splat, Shoot; Philion & Fidler, ECCV 2020, arXiv:2008.05711）は逆方向のアプローチで、同じ内部・外部パラメータを使って画像の点を3Dに「持ち上げ（lift）」ます。各ピクセルに深度の確率分布を持たせて3D空間にばらまく（splat）方式です。比較すると、ピンホール投影（3D→2D）はキャリブの正確さに依存するが深度推定が要らない、LSS（2D→3D）は深度分布の推定が要る、という対照的なトレードオフがあります。

PETR（Liu et al., ECCV 2022, arXiv:2203.05625）はさらに別解で、明示的な投影サンプリングを避け、3D座標を位置埋め込みに変換して画像特徴へ注入します。ここでも投影行列（カメラパラメータ）は3D位置をエンコードするのに使われており、ピンホール幾何が形を変えて生き続けていることがわかります。これら3系統（投影サンプリング・深度持ち上げ・位置埋め込み）はいずれもピンホール投影の幾何を土台にしており、違いは「その幾何をどの方向に・どの段階で使うか」にあります。

## 論文の実験結果（定量データ）

投影幾何そのものは決定的な計算なので「精度」を持ちませんが、それを使う BEV 手法の性能を通して間接的に評価されます。BEVFormer は nuScenes テストセットでカメラのみ NDS 0.569、mAP 0.481 を達成しました。ここで重要なのは、キャリブレーションの質が結果に直結する点です。nuScenes は高精度なカメラキャリブを提供しており、それが投影サンプリングの正確さを支えています。

キャリブ誤差の影響を調べた研究では、外部パラメータ（カメラの取り付け角度）にわずかな誤差を注入すると、遠方の物体の位置誤差が急増することが報告されています。これは透視除算の性質によります。遠方ほど depth が大きく、角度のわずかなズレが横方向の大きなズレに増幅されるためです。報告によると、回転で数度の誤差でも遠距離での検出精度（mAP）が目に見えて低下し、自動運転で用いるにはキャリブの再較正（online calibration）が重要であることが示されています。

LSS 系（BEVDet, BEVDepth）の実験は、3D→2D 投影と 2D→3D 持ち上げの違いを定量的に浮かび上がらせます。2D→3D 系は深度監督を加えると nuScenes で mAP・NDS が数ポイント改善する一方、深度監督なしでは投影サンプリング系に対する優位が小さいことが示されました。これは「どちらの方向に投影を使うかで、必要な追加情報（深度の正解）が変わる」という設計上のトレードオフを定量的に裏付けます。

## メリット・トレードオフ・限界

メリット。確立した幾何なので挙動が予測可能で、誤差の原因を追いやすいです。行列1つにまとめてあるので計算が軽く、全点・全カメラを一括処理できます。深度推定モジュールが不要で、投影は決定的な計算です。微分可能なので、後段のサンプリングと合わせて end-to-end に学習できます。

トレードオフと限界。正確なカメラキャリブレーション（内部・外部パラメータ）が前提で、ずれると投影先が大きく狂います。ピンホール近似なので、広角・魚眼レンズの強い歪みはそのままでは扱えず、歪み補正（distortion correction）が別途必要です。透視除算で0割りや負の深度を扱う際にガード（clamp・マスク）が必須で、忘れると NaN になります。さらに、キャリブが取得できないデータセットや、走行中の振動・経年でキャリブがずれる現実環境では、この前提自体が崩れます。この制約を緩めるための代替が擬似投影です（[擬似投影（pseudo-projection）フォールバック](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1004_pseudo-projection-fallback) 参照）。

## 発展トピック・研究の最前線

レンズ歪みを扱う拡張として、魚眼カメラ向けの一般化カメラモデル（Kannala-Brandt モデルなど）を投影に組み込む研究が進んでいます。駐車支援などで使う広角カメラはピンホール近似では誤差が大きく、歪みパラメータまで含めた投影が必要です。

走行中にキャリブのズレを自動補正する online calibration / self-calibration も活発な領域です。LiDAR と画像の対応や、消失点・地平線などの幾何的手がかりから外部パラメータを推定し続け、振動や温度変化によるドリフトを吸収します。

キャリブそのものを学習に取り込む方向もあります。投影行列を固定で与えるのではなく、ネットワークの一部として推定したり、補正項を学習で足したりする手法です。これは次の擬似投影（[擬似投影（pseudo-projection）フォールバック](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1004_pseudo-projection-fallback)）の発想と地続きで、「幾何をどこまで陽に与え、どこから学習に委ねるか」という、現代の多視点認識における中心的な設計判断につながっています。
