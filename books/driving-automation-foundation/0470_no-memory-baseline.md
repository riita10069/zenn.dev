# メモリ無し (NoMemory) ベースライン

## ひとことで言うと
NoMemory とは「過去の履歴をまったく使わず、最新の 1 タイムステップだけで判断する」最もシンプルな時系列モジュールです。過去の系列が渡されても、その中から最新の 1 ステップを取り出すだけで、学習するパラメータを 1 つも持ちません (パラメータフリー)。だからこそ、時間圧縮や記憶を行う凝ったモジュール ([時間メモリ (Conv1d による時間圧縮 + GRU 要約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0473_temporal-memory-conv1d-gru)) が「本当に効果があるのか」を測る公平な比較基準 (ベースライン) になります。性能を語る上での「ゼロ点」を提供する道具です。

## 直感的な理解
新しい仕組みを足すとき、研究・開発で最も大事なのは「それが本当に効いているか」を公平に確かめることです。たとえば「過去履歴を 1Hz に圧縮して GRU で要約する時間メモリを入れたら運転が安定した」と主張したいとします。しかし、もし「履歴を一切使わなくても同じ性能が出る」なら、その複雑な仕組みは無駄だったことになります。複雑さにはコスト (計算量、バグの余地、解釈の難しさ) が伴うので、得られる利益がそれに見合うかを確かめねばなりません。

そこで必要なのが「何もしない素朴な基準線」です。これがベースラインです。ベースラインがあって初めて「履歴を使うモジュールはベースラインより X 点良い」と差分で語れます。基準線が無いと、絶対値だけ見ても「それが良い数字なのか悪い数字なのか」を判断できません。料理で言えば「味付けした料理」を評価するには「味付けしていない素材そのもの」と比べる必要があるのと同じです。テストの点数 80 点が良いか悪いかは、平均点を知らないと判断できないのと同じ理屈です。

## 基礎: 前提となる概念
ベースライン (baseline) とは、比較の土台になる最も単純な、あるいは既知の標準的なやり方のことです。機械学習研究では、提案手法の性能を語るとき必ずベースラインとの差で示します。「精度 92%」と言われても、ベースラインが 91% なのか 50% なのかで意味がまるで変わるからです。

アブレーション研究 (ablation study) とは、提案手法の構成要素を 1 つずつ外して、それぞれがどれだけ性能に寄与しているかを確かめる実験です。「ablation」は医学用語で「切除」を意味し、ここでは「機能を切り取って影響を見る」という比喩です。NoMemory は「時間メモリ機構を完全に切除した」アブレーション条件にちょうど対応します。論文の Table で「Ours」の隣に「w/o temporal」のような行が並ぶとき、その「w/o temporal」が NoMemory に相当します。

パラメータフリー (parameter-free) とは、学習で調整される重み (パラメータ) を一切持たないことです。これが重要なのは、ベースラインが学習可能な重みを持つと「その重みがたまたま良かった/悪かった」という偶然が性能差に混ざってしまうからです。パラメータが 0 なら、性能差を純粋に「履歴処理の有無」だけに帰属できます。乱数シードや初期値の影響も受けないので、何度走らせても挙動が完全に一致します。

テンソルの形 (shape) について。履歴は `[B, T, D]` の 3 次元テンソルで表されます。B はバッチサイズ、T は時間ステップ数、D は各ステップの特徴次元です。NoMemory は時間軸 T のうち最後の 1 ステップだけを残し、`[B, D]` を出力します。スライス記法 `[:, -1]` が「全バッチについて時間軸の最後の要素」を意味します。

## 仕組みを詳しく
実装は驚くほど短く、本質は数行です。共通のインターフェース (全モジュールに共通の呼び出し方を定める親クラス) を継承し、forward (順伝播の処理) だけを実装する、という形が典型です。

```python
class NoMemory(BaseTemporalMemory):
    def __init__(self, **kwargs):
        super().__init__()

    def forward(self, visual_history, egomotion_history, **kwargs):
        v_ctx = visual_history[:, -1] if visual_history.ndim == 3 else visual_history
        e_ctx = egomotion_history[:, -1] if egomotion_history.ndim == 3 else egomotion_history
        return v_ctx, e_ctx
```

処理を分解します。

`__init__` が `**kwargs` (任意のキーワード引数をまとめて受ける記法) を受け取って何も使わないのには理由があります。差し替え可能な部品を一覧表 (レジストリ) で管理する設計では、ビルダ関数はどの部品に対しても同じ引数 (visual_dim や egomotion_dim など) を一律に渡すのが普通です。NoMemory はそれらを必要としませんが、引数を受け取れないとビルド時にエラーになります。だから `**kwargs` で受けて無視する作りにしておく。これは「すべての部品が同じ呼び出し方で作れる」という設計上の都合 (インターフェースの統一) です。

`forward` は履歴の次元数で分岐します。
- 入力が 3 次元 `[B, T, D]` のとき: `[:, -1]` で時間軸の最後の要素 (最新タイムステップ) だけを取り出し `[B, D]` にする。
- 入力が 2 次元 `[B, D]` (すでに時間軸が無い) のとき: そのまま素通し (passthrough) する。

視覚と自己運動の両方に同じ処理を適用し、2 つの文脈ベクトルを返します。

before / after で対比すると分かりやすいです。
- 入力 (before): `[8, 64, 896]` — 64 ステップ (10Hz なら 6.4 秒分) の視覚履歴。
- NoMemory 出力 (after): `[8, 896]` — 最新の 1 ステップのみ。過去 63 ステップは捨てられる。
- 時間メモリ出力 (比較): `[8, 896]` — 同じ形状だが、こちらは 64 ステップ全体を圧縮して要約した「過去文脈入り」のベクトル。

ここが設計の妙です。NoMemory と時間メモリは出力の形が同じで、中身が「最新だけ」か「履歴の要約」かだけが違う。形が揃っているので、下流のプランナーを一切変えずにモジュールだけ差し替えて A/B 比較できます。設定上のモジュール名を切り替えるだけで実験できる、というのが差し替え可能な部品設計 (レジストリ方式) の利点です。比較の公平性 (下流が同一、入力が同一、違いは履歴処理だけ) が構造的に保証される点が重要です。

## 手法の系譜と主要論文
NoMemory 自体は実装上の道具ですが、その背後には機械学習研究の重要な規範と、運転 E2E (End-to-End、知覚から制御までを 1 つのネットで学ぶ) の歴史的設定があります。

運転 E2E の原点は Pomerleau の ALVINN (NeurIPS 1988) で、前方カメラ画像を 3 層ネットに入れて操舵を出しました。現代的な深層学習版を広めたのは Bojarski et al., "End to End Learning for Self-Driving Cars" (NVIDIA, 2016, arXiv:1604.07316) です。これは単一フレームの前方カメラ画像を CNN に入れ、操舵角を直接出力しました。明示的な車線検出や経路計画を挟まず、画像から操舵への写像を一気に学ぶ。NoMemory が体現する「最新フレームのみで判断する」入力形式は、まさにこの原初的な E2E 設定に対応します。時間メモリを使うモジュールは、この最小設定に対して「時間文脈を足すと何が改善するか」を問う発展形です。

ベースラインを必ず置くという規範は、再現性と説明責任の観点で広く共有されています。Codevilla et al., "On Offline Evaluation of Vision-based Driving Models" (ECCV 2018, arXiv:1809.04843) は、運転モデルのオフライン評価 (録画データ上での予測誤差) と実際の closed-loop 運転性能が必ずしも一致しないことを示し、評価設計の重要性を強調しました。この研究は「強い/素朴なベースラインと公平な条件で比較しないと、見かけの改善に騙される」という教訓を運転分野に突きつけた点で重要です。NoMemory はその規範をコードに組み込んだものと言えます。

履歴を足すことの落とし穴も明確に研究されています。Wen et al. (NeurIPS 2020, arXiv:2007.13731) は behavioral cloning (お手本の行動を真似る学習) において、過去の自分の行動を入力に含めると「直前の行動をそのままコピーするだけ」のショートカット (copycat / inertia 問題) を学んでしまうことを指摘し、その対処を提案しました。これは NoMemory のような「履歴を持たない条件」が、ときに履歴ありより堅牢になり得る理由を説明します。

## 論文の実験結果 (定量データ)
NoMemory 単体の論文実験はありませんが、「時間文脈を足すかどうか」の比較は多くの研究で定量化されています。

Codevilla et al. (2018) は、過去フレームを与える設定が必ずしもオフライン指標を改善せず、むしろ過去観測への過度な依存 (inertia problem、慣性問題と呼ばれる、止まっている履歴を見て止まり続けてしまう現象) を引き起こすことを報告しました。具体的には、複数フレームを入力に与えたモデルが、赤信号で停止した後に青になっても発進しにくくなる傾向を示し、closed-loop での走行成功率が低下する場合があると指摘しています。これは「履歴を足せば必ず良くなるわけではない」ことを示す重要なアブレーション的知見で、NoMemory のような最新フレームのみのベースラインを置く意義を裏付けます。Wen et al. (2020) も、履歴ありモデルがオフライン予測誤差は低いのに closed-loop では失敗する (copycat) という逆転現象を定量的に示しました。

一般に、時間メモリ機構の論文は次の形でベースライン比較を報告します。たとえば CARLA の closed-loop 評価では driving score (route completion と infraction penalty の積、高いほど安全に完走) を指標とし、「メモリ無し」条件に対して時間文脈を加えると driving score が数 point から十数 point 改善する、といった形です。同時に inertia problem が悪化していないか (青信号での発進遅れが増えていないか) を別途確認するのが定石です。

オフライン指標としては軌道予測の ADE (Average Displacement Error、予測点と正解点のユークリッド距離の全ステップ平均) や FDE (Final Displacement Error、最終点での距離) がよく使われます。単位はメートルで小さいほど良い。NoMemory ベースラインに対し時間メモリが ADE/FDE を下げれば、過去文脈が予測精度に寄与したと言えます。逆に下がらなければ、その複雑さは正当化できません。たとえば「メモリ無し ADE 1.20m → 時間メモリ ADE 0.95m」なら 21% の相対改善、というように差分で価値を語ります。

## メリット・トレードオフ・限界
メリット
- パラメータフリーで挙動が完全に予測可能。性能差を「履歴処理の有無」だけに帰属できる。
- 出力形状が他モジュールと同一なので、下流を変えずに差し替え比較 (A/B テスト) ができる。
- 既定モードとして使えば、追加の時間機構を入れていない素のモデルを常に動かせる (複雑さの最小起点)。
- 実装が数行でバグの混入余地がほぼなく、信頼できる基準になる。再現性が完璧 (乱数に依存しない)。
- inertia / copycat 問題に原理的にかからない。履歴を持たないので「過去の自分をコピーする」ショートカットが存在しない。

トレードオフと限界
- 過去文脈を完全に捨てるため、直前の出来事 (前車の急ブレーキ等) を考慮できず、判断がちらつきやすい。
- あくまで比較基準であり、実運用の性能を狙うモジュールではない。
- 「最新 1 フレーム」前提なので、時間情報が本質的なタスク (物体の速度・加速度の推定など) では原理的に不利。単一の静止画からは速度が原理的に読み取れない (止まっている車か低速で動く車かを 1 枚では区別できない)。
- 研究上の注意として、ベースラインが弱すぎると「提案手法が優れて見える」だけで実は当たり前という落とし穴がある。NoMemory はあえて最弱に振った基準なので、これに勝つだけでは不十分で、強いベースライン (単純な多フレーム連結など) との比較も併せて行うのが望ましい。

## 発展トピック・研究の最前線
ベースラインの選び方そのものが研究品質を左右します。近年は「弱いベースラインに勝った」だけでなく、強いベースラインや既存 state-of-the-art との公平比較、複数シードでの分散報告 (平均だけでなく標準偏差や信頼区間)、closed-loop とオフラインの両面評価が求められます。Codevilla et al. が示したオフライン指標と運転性能の乖離は、いまも自動運転の評価設計における中心的な論点です。

また inertia problem への対処として、過去観測の重みを下げる、ドロップアウト的に履歴をランダムに落とす (history dropout)、過去行動と現在出力の依存を切る因果的介入を加えるといった手法が研究されています。NoMemory はこれらの議論の「対照群」として常に有用で、「履歴を入れる工夫」の効果はすべて NoMemory との差で測られます。
