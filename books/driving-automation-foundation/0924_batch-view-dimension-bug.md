# マルチカメラモデルのバッチ次元バグと [B,V,C,H,W] へのパラメータ化

## ひとことで言うと

複数カメラを扱う深層学習モデルでよく起きる、地味だが致命的なバグの話です。カメラの台数を「バッチ(同時に処理するサンプル数)の次元」に混ぜて入力してしまうと、複数のシーンをまとめて学習する「ミニバッチ学習」が原理的にできなくなります。さらに、特徴をベクトルに潰す `torch.flatten` を引数なしで呼ぶと、バッチ次元まで一緒くたに潰してしまい、サンプルの区切りが消えます。これを直すには、入力を `[B, V, C, H, W]`(バッチとカメラを別々の次元に)へ作り直し、flatten がバッチを潰さないよう `start_dim=1` を指定します。

## 直感的な理解

学校の試験の採点を想像してください。30 人分の答案を採点して平均点を出すには、答案が「生徒ごとに分かれて」いなければなりません。もし 30 人分の答案を1枚に全部書き写してしまい、どこからどこまでが誰の答案か分からなくなったら、個別の点数も平均点も計算できません。

ニューラルネットの学習も同じです。複数のサンプル(シーン)をまとめて処理し、それぞれの「間違い具合(勾配)」を計算して平均し、一回モデルを更新します。これがミニバッチ学習です。この計算が成り立つ大前提は、「サンプルがバッチ次元できれいに分かれて並んでいること」です。サンプルの区切りが消えると、誰の勾配なのか分からなくなり、学習が破綻します。

マルチカメラモデルでは、ここに罠があります。1シーンに複数のカメラ(例えば 8 台)が付いているので、「8 台のカメラ」を「バッチサイズ 8」のフリをさせて処理してしまいたくなるのです。これは1シーンだけなら動きます。しかし「複数シーンをまとめて学習したい」と思った瞬間、バッチ次元がカメラで埋まっているので、シーンの区切りが表現できなくなります。30 人分の答案を1枚にまとめてしまうのと同じ過ちです。

## 基礎: 前提となる概念

順に噛み砕きます。

**バッチ次元** とは、テンソルの先頭の次元で、「同時に処理するサンプルの数」を表します。画像なら `[B, C, H, W]` の `B` がそれです。深層学習のほとんどの演算は、この先頭次元を「独立したサンプルの並び」として扱うことを前提に設計されています。

**ミニバッチ確率的勾配降下法(mini-batch SGD)** とは、深層学習の標準的な学習法です。1サンプルずつ更新するのでも、全データを一度に使うのでもなく、適度な数(ミニバッチ)をまとめて勾配を計算し平均して更新します。GPU の並列計算を活かせて速く、勾配のノイズが減って学習が安定する、という二つの利点があります。

**勾配(gradient)** とは、「モデルのパラメータをどちらにどれだけ動かせば誤差が減るか」を示す数値です。各サンプルが自分の勾配を持ち、それらを平均して一回更新します。

**`torch.flatten`** は、多次元テンソルを潰して次元を減らす関数です。引数なしで呼ぶと全次元を1本に潰し、`start_dim=1` を付けると「1番目の次元から後ろだけを潰し、0番目(バッチ)は残す」という挙動になります。

**`reshape` と `view` と `contiguous`** は、テンソルの形を変える操作です。`view` はメモリ上の並び(連続性)を保ったまま見方を変えるもので、メモリが連続でないと失敗します。`reshape` はメモリが連続なら `view` と同じ、そうでなければコピーを作ります。形を変える際、元のメモリ配置と意図した次元の対応が合っていないと、見た目は同じ shape でも中身が別物のテンソルになる、という落とし穴があります。

これらを踏まえると、このバグの核心は「バッチとカメラという論理的に別物の二つの軸を、一つの次元に押し込んでしまった」ことだと分かります。

## 仕組みを詳しく

### before(バグのあった状態)

バグのある実装では、複数カメラ(V 台)を扱うためにカメラをバッチ次元に積んでいました。8 台のカメラを `[8, C, H, W]` のように「バッチサイズ 8」のフリをさせて Backbone に流していたのです。これだと1シーン(8 カメラ)しか扱えません。8 シーンをまとめて学習するには本来 `[8 シーン × 8 カメラ, C, H, W]` = `[64, ...]` のようにしたいのに、バッチ次元がカメラで埋まっているので、どこまでが1シーンの境目か区別できません。

さらに悪いことに、特徴をベクトルに変換する flatten を引数なしで呼んでいました。

```python
# バッチ次元まで含めて全部1本に潰してしまう
visual_feature_vector = torch.flatten(feature_map)
feature_vector = torch.cat((visual_feature_vector,
                            visual_history, egomotion_history), dim=0)
```

`torch.flatten(feature_map)`(引数なし)は、`[B, C, H, W]` を `[B*C*H*W]` という完全な1次元に潰します。本来は `[B, C*H*W]` にして「サンプルごとに1本のベクトル」にしたかったのに、サンプルの区切りが消えてしまいます。加えて連結が `dim=0`(サンプルを縦に積む向き)になっており、これも次元の扱いが破綻していました。結果として、8 シーンぶんの特徴がシーンの区切りなくダンゴになり、ミニバッチ学習が原理的に不可能でした。これがバグの本質です。

### after(修正後)

入力を5次元 `[B, V, C, H, W]` に明示し、B と V を別々の次元として扱います。Backbone を通すときは計算の都合上いったん `[B*V, ...]` に畳みますが、その後 ViewFusion 段で「V を統合して B を保つ」処理に分離します。

```python
def forward(self, fused_per_view, B, V):
    C, H, W = fused_per_view.shape[1:]
    x = fused_per_view.reshape(B, V, C, H, W)   # B と V を明示的に分ける
    x = x.reshape(B, V * C, H, W)               # V をチャンネルに畳む (B は残る)
    return self.view_reduce(x)                  # [B, C, H, W]
```

そして flatten には `start_dim=1` を付けて、バッチ次元(0番目)を潰さないようにします。

```python
# 修正後: [B, C*H*W] のように、サンプルごとに1本のベクトルにする
visual_feature_vector = torch.flatten(feature_map, start_dim=1)
feature_vector = torch.cat((visual_feature_vector,
                            visual_history, egomotion_history), dim=1)  # dim=1 に修正
```

`start_dim=1` は「1番目の次元から後ろを潰し、0番目(バッチ)はそのまま残す」という指定です。連結も `dim=1`(特徴の方向)に直します。

### 数値例で before/after を比較

`[B=8, C=3, H=7, W=7]` のテンソルがあったとします。

- before: `torch.flatten(x)` → 形 `[8*3*7*7]` = `[1176]`。8 シーンがダンゴになり区別不能。
- after: `torch.flatten(x, start_dim=1)` → 形 `[8, 3*7*7]` = `[8, 147]`。8 シーンが別々の行として保たれる。

after なら、8 シーンぶんの勾配を別々に計算して平均できるので、ミニバッチ学習が成立します。before では、そもそも「8 シーン」という構造が消えているため、誤差を逆伝播させてもサンプルごとの勾配が定義できません。

### reshape の罠

`reshape(B, V, C, H, W)` で B と V を分離する際、元のテンソルが本当に「B が外側、V が内側」の順でメモリに並んでいなければ、別物のテンソルになります。例えば `[B*V, ...]` を作るときに `cat` や `stack` をした順序が「V が外側」だったのに、戻すとき「B が外側」と仮定して reshape すると、シーンとカメラが入れ替わったデタラメな対応になります。この種のバグは shape が合ってしまうため例外を吐かず、静かに間違った学習を続けるのが厄介です。並べる順序と戻す順序を一致させ、必要なら `contiguous()` を挟んでメモリ連続性を保証することが重要です。

### バグを再発させない仕組み

この種のバグは、テストで守るのが定石です。中でも重要なのが **バッチ独立性(batch independence)のテスト** です。これは「あるサンプルの入力を変えても、別のサンプルの出力は変わらない」ことを検証します。もしバッチ次元がカメラや他の軸と混ざると、あるサンプルの変更が別サンプルに漏れ出し、このテストが落ちて気づけます。具体的には、バッチの一部だけ入力を差し替えて forward し、差し替えていないサンプルの出力が一致するかを確認します。他にも、カメラ台数(num_views)を変えても動くか(view flexibility)、勾配が全パラメータに流れるか(gradient flow)、数値が NaN にならないか(numerical stability)などをセットで検査します。

## 手法の系譜と主要論文

これは特定の論文が提案した「手法」というより、PyTorch のようなテンソルライブラリでマルチカメラ・多軸データを扱うときの一般的な落とし穴と、その回避の定石です。背景となる考え方を二つ挙げます。

第一は、ミニバッチ学習の理論的基盤です。確率的近似の源流は Herbert Robbins と Sutton Monro の 1951 年の論文 "A Stochastic Approximation Method"(Annals of Mathematical Statistics)にさかのぼります。ここで定式化された「ノイズのある観測から少しずつ真の値へ近づく」という考え方が、現代のミニバッチ確率的勾配降下法の理論的土台です。複数サンプルの勾配を平均して一回更新すると、勾配のノイズが減って学習が安定し、GPU 並列で高速になります。この手法が成立する前提が「サンプルがバッチ次元で独立に並んでいること」であり、今回のバグはまさにこの前提を壊していました。

第二は、マルチカメラ知覚における次元設計の定石です。Zhiqi Li らの BEVFormer(ECCV 2022, arXiv:2203.17270)や Yue Wang らの DETR3D(CoRL 2021, arXiv:2110.06922)に代表されるマルチカメラ3次元検出の手法では、入力を `[B, V, C, H, W]` のように batch と view を明示的に分けるのが標準です。view をバッチに畳むのは、Backbone を通す計算の都合で一時的に行うことはあっても、論理的には batch と view を区別し続け、必要なときだけ畳んで必ず元に戻します。今回の修正は、この確立した定石に合わせたものと言えます。

## 論文の実験結果(定量データ)

このトピックは性能を競う手法ではなく実装上の正しさに関するものなので、「データセットで指標が何ポイント改善した」という形の数値はありません。代わりに、効果は「学習が成立するか / しないか」という二値で現れます。

before の状態では、バッチサイズを 2 以上にしても、実質的に学習が機能しません。サンプルの区切りが消えているため、勾配がサンプルごとに正しく定義されず、損失が下がらない、あるいは下がっても意味のある表現が学べない、という形で破綻します。after に直すと、バッチサイズを上げられるようになり、GPU の並列計算を活かして1ステップあたりの処理サンプル数が増え、学習が安定します。

定量的に語れる側面としては、バッチ独立性テストの挙動があります。正しい実装では、バッチ内のあるサンプルの入力を変えても、他のサンプルの出力との差はゼロ(浮動小数点の丸め誤差の範囲内、典型的には 1e-5 以下)に収まります。バグがあると、この差が有意な値(ゼロから大きく外れた値)になり、テストが失敗します。この「差がゼロかどうか」という単純な数値判定が、バグの有無を機械的に検出する確実な指標になります。

参考として、マルチカメラ手法の文脈では、ミニバッチを正しく扱えることが大規模学習の前提です。BEVFormer や DETR3D が nuScenes のような大規模データセットで高い NDS / mAP(検出性能の総合指標)を達成できたのは、batch と view を正しく分離した上で大きなバッチで安定に学習できたからであり、その逆(バッチ次元が壊れた状態)では、そもそもこれらの結果に到達できません。

## メリット・トレードオフ・限界

メリットは明確です。ミニバッチ学習が可能になり、GPU を活かした高速で安定な学習の前提が整います。B と V を明示的に分けたことで、カメラ台数を可変にでき、異なる車両構成(カメラ 6 台と 8 台など)に同じコードで対応できます。バッチ独立性テストにより、同種のバグの再発を自動検知できます。

トレードオフと注意点もあります。Backbone は画像を1枚ずつ処理する都合上、いったん `[B*V, ...]` に畳んで通す必要があり、その後で必ず `[B, V, ...]` に戻す「畳んで戻す」操作を正しく書く責任が残ります。reshape で B と V を分離する際、元のメモリ配置と順序が合っていないと、shape は同じでも中身が別物になる(view と reshape の罠)ため、ここはテストで守る必要があります。5次元入力は人間が直感的に追いにくく、各演算の入出力 shape をコメントで丁寧に書かないと、後続の開発者が混乱しやすくなります。

研究者視点での一般的な教訓としては、「shape が合っている」ことと「意味が正しい」ことは別物だ、という点が挙げられます。テンソルライブラリは shape の整合性しかチェックしないので、軸の意味の取り違え(バッチとカメラ、時間と視点など)は例外を出さずに通ってしまいます。こうした「サイレントな次元バグ」を防ぐには、(1) 入出力 shape を明示するアノテーションやコメント、(2) バッチ独立性のような不変条件をテスト化すること、(3) `einops` のような名前付き次元操作ライブラリで軸の意味を式に明示すること、が有効です。これは深層学習工学における未解決かつ実務的に重要な課題であり、近年は型レベルで shape を検査する研究(tensor shape typing)も進んでいます。

## 発展トピック・研究の最前線

この種の次元管理は、扱う軸が増えるほど深刻になります。マルチカメラに加えて時間(過去フレーム)、複数モダリティ(カメラ + LiDAR)、複数エージェント(周辺車両)を扱う近年のモデルでは、`[B, T, V, M, C, H, W]` のように6軸・7軸のテンソルが当たり前になり、軸の取り違えのリスクが跳ね上がります。これに対し、`einops`(Alex Rogozhnikov による軸操作ライブラリ、ICLR 2022 で発表)の `rearrange` のように軸を名前付きの式(`"b v c h w -> (b v) c h w"` など)で明示的に変形する手法が広く使われるようになっています。括弧で囲んだ `(b v)` は「b と v をこの順序で畳む」という意図を式そのものに刻み込むので、戻すときも `"(b v) c h w -> b v c h w"` と書けば順序の取り違えが起きません。これにより、どの軸をどう畳んで戻すかがコード上で読めるようになり、サイレントなバグを大幅に減らせます。本稿の `reshape` で問題になった「畳む順序と戻す順序の不一致」は、einops を使うと式レベルで明示されるため、そもそも起こりにくくなります。

さらに研究の最前線では、テンソルの shape を型システムで静的に検査する試み(named tensors、shape typing、依存型を使った次元検査など)が進んでいます。これらが成熟すれば、「バッチとカメラを取り違える」ような次元バグを、実行前にコンパイラやリンタが検出できるようになると期待されています。
