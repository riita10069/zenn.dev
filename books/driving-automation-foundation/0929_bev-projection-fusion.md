# BEV投影によるカメラ融合

## ひとことで言うと

複数のカメラそれぞれが撮った画像の特徴を、車を真上から見下ろした地図のような共通の座標系(鳥瞰図、Bird's-Eye-View、略して BEV)の上に投影してまとめる融合方式です。前カメラも右カメラも後カメラも、いったん「上から見た1枚のマップ」の上に貼り直すので、車のまわりの状況を一貫した地図として後段の処理(物体検出や経路計画)に渡せます。マルチカメラを使う自動運転の知覚で、いまや事実上の標準的な中間表現になっています。

## 直感的な理解

人間がカーナビを見るとき、私たちは「上から見た地図」の上で「自分はいまここ、目的地はあちら」と考えます。前を向いたフロントガラス越しの景色(これがカメラ画像に相当します)だけで運転計画を立てるのは大変です。なぜなら、フロントの景色には「自分のすぐ右隣に何があるか」「後ろから来る車との距離」が一切写っていないからです。

自動運転車には前後左右に複数のカメラが付いています。たとえば6台や7台です。それぞれが別々の方向の景色を撮ります。問題は、これらの画像をそのまま重ねても意味が通らないことです。前カメラの「画面の右端のピクセル」と右カメラの「画面の左端のピクセル」は、実世界では同じ場所(右斜め前)を指しているかもしれませんが、ピクセル座標だけ見ても対応がつきません。

BEV はこの問題を「共通の地図に貼り直す」ことで解決します。すべてのカメラの情報を「自車を中心にした真上からの地図」という1つの座標系にまとめてしまえば、カメラごとの見え方の違いが吸収され、実世界の位置関係(前方20m・右3mに車がいる、など)がそのまま地図上の位置として表現されます。これは後段の経路計画にとって極めて自然な形です。

## 基礎: 前提となる概念

まず、いくつかの言葉を噛み砕きます。

**特徴(feature)とは何か。** カメラ画像をそのままモデルに入れるのではなく、まず畳み込みニューラルネットワーク(CNN、画像から模様や形のパターンを抽出するネットワーク)や Vision Transformer などの「バックボーン(backbone、特徴抽出の土台になるネットワーク)」に通します。出てくるのは、画像を小さなマス目に区切って各マスに数百次元のベクトル(数の並び)を割り当てた「特徴マップ」です。たとえば元画像が高解像度でも、特徴マップは縦横が大きく縮小され(たとえば16分の1)、各マスに256個の数が入る、といった形になります。この256個の数が「そのあたりに何が写っているか」を圧縮して表しています。テンソルの形(shape)で書くと `[C, H, W]`、C がチャンネル数(=特徴の次元、たとえば256)、H と W が縦横のマス数です。

**カメラの校正(キャリブレーション)とは何か。** 各カメラが「自車のどこに、どの向きで取り付けられ、どんなレンズか」という情報です。これは2種類の行列で表されます。内部行列(intrinsic matrix)はレンズの焦点距離や画像中心など、カメラ内部の性質です。外部行列(extrinsic matrix)はカメラが自車座標系のどこにあって、どの方向を向いているかです。この2つを合わせると、「実世界の3D座標 (X, Y, Z)」を「そのカメラ画像上のピクセル座標 (u, v)」に変換する射影が計算できます。重要なのは、3D から 2D への変換は数学的に一意に決まる(曖昧さがない)ことです。逆に 2D から 3D を復元するのが難しい(奥行きが分からない)のです。

**BEV グリッドとは何か。** 自車を中心とした地面を、たとえば前方何メートルから後方何メートル、左右何メートルの範囲で、格子状のマス目に区切ったものです。たとえば 200×200 のマス、あるいはもっと細かく 450×300 のマスなどです。各マスが実世界の何メートル四方かは設計で決めます。このマス目1つ1つに特徴ベクトルを持たせたものが「BEV 特徴マップ」で、形は `[C, X, Y]` です。

## 仕組みを詳しく

カメラ特徴を BEV に投影する方式は、大きく2つの考え方に分かれます。両方とも「3D と 2D の対応」を使いますが、向きが逆です。

**方式A: 画像から BEV へ押し出す(forward projection、明示的深度方式)。** 各画像ピクセルについて「これは自車から何メートル先か」という深度を予測し、その深度を使ってピクセルを 3D 空間に持ち上げ、BEV マスに落とし込みます。Lift-Splat-Shoot(LSS)が代表で、詳細は [LSS(明示的深度)によるBEV化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0283_lift-splat-shoot-depth) を参照してください。

**方式B: BEV から画像へ問い合わせる(backward projection、暗黙的投影方式)。** こちらが本ノートの中心です。先に BEV グリッドを置き、各マスが「自分は各カメラ画像のどこに写っているはずか」を計算して、その画像位置の特徴を拾い集めます。深度を明示的に予測しないのが特徴です。手順を追います。

1. **BEV クエリを用意する。** BEV グリッドの各マスに、学習可能なベクトル(クエリ、query、「情報を集めに行く担当者」のようなもの)を割り当てます。形は `[X*Y, C]`、たとえば 450×300=135000 マスそれぞれに 256 次元、で `[135000, 256]`。最初はランダムな初期値で、学習を通じて「そのマスにどんな物があるか」を表すように育ちます。

2. **3D 参照点を作る。** 各 BEV マスは平面座標 (x, y) を持ちますが、高さ z が決まっていません。そこで高さ方向に何段か(たとえば4段)の点を立てた「ピラー(柱)」を作ります。1マスにつき (x, y, z₁), (x, y, z₂), ... と複数の 3D 点ができます。形は `[X*Y, 高さ段数, 3]`。実世界のメートル単位に直すための範囲(たとえば X 前方 -60〜120m、Y 左右 ±60m、Z 上下 -5〜3m)を設計で決めます。

3. **3D 点を各カメラ画像に投影する。** 各 3D 点に対し、カメラの校正行列(自車座標→ピクセル座標の射影、形 `[B, V, 3, 4]`。B はバッチ、V はカメラ台数)を掛けて、その点が各カメラのどのピクセル (u, v) に写るかを計算します。透視除算(奥行きで割る操作)でピクセル座標を出し、画像サイズで割って [0,1] に正規化します。カメラの後ろ(深度が負)や画像の外に出た点は「可視性マスク(visibility mask)」で除外します。

4. **特徴をサンプリングして集約する。** 各カメラの特徴マップから、投影先の位置の特徴を `grid_sample`(任意の小数位置の値を周囲4点から双線形補間で取り出す操作)で抜き取ります。ここで deformable attention(変形可能注意、後述)を使う実装が多く、参照点の周りで位置を少しずらしながら数点をサンプリングし、重み付けして混ぜます。ピラー内の高さ段ごとに集めた特徴を、そのマスが見えているカメラ台数で割って平均します(見えるカメラが0台のマスはゼロにする)。

5. **後処理。** 残差接続(入力をそのまま足して学習を安定させる)、LayerNorm(正規化)、FFN(全結合の小さなネットワーク)を通し、最終的に `[B, C, X, Y]` の BEV 特徴マップに整形します。

形の流れを数値例で示します。入力がバッチ B=4、カメラ V=7、各カメラ特徴が `[256, 8, 8]` だとすると、入力は `[4*7, 256, 8, 8]`。出力は BEV グリッド 450×300 として `[4, 256, 450, 300]`。出力の空間サイズが入力画像特徴の 8×8 ではなく BEV グリッドの 450×300 になっている点が、特徴空間で混ぜるだけの単純な融合(concat やカメラ間アテンション)との決定的な違いです。

```
カメラ画像 ×7 → backbone → 画像特徴 [7,256,8,8]
                                    │
BEV クエリ [450×300, 256] ──┐       │
   各マスに 3D 参照点 (x,y,z×4高さ)  │
        └→ カメラ校正で 2D 画像座標へ射影 (一意に計算可能)
              └→ 投影位置の近傍を deformable attention でサンプル
                    └→ 高さ段・カメラを集約 → BEV 特徴 [4,256,450,300]
```

**なぜ「深度を当てずに正しい奥行きの特徴が得られる」のか。** ここが暗黙投影の肝です。各 BEV マスは高さ方向に複数の参照点を投げています。あるマスの「正しい高さ」に置いた参照点は、画像上で「本当にその物体が写っているピクセル」に投影されます。逆に間違った高さの参照点は、関係のないピクセルに投影されます。注意機構の重み付けが、関係の強い(=正しい奥行きの)特徴を強調し、無関係なものを抑えます。結果として、深度を1つの値に決め打ちせずとも、複数候補の中から正しい対応が浮かび上がる仕組みです。

## 手法の系譜と主要論文

BEV 表現は段階的に発展してきました。

- **Lift, Splat, Shoot(Philion & Fidler, ECCV 2020)** が、カメラのみから BEV を作りエンドツーエンドで運動計画まで行う枠組みを提案しました。画素ごとの深度を確率分布として予測する明示的深度方式の出発点です。詳細は [LSS(明示的深度)によるBEV化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0283_lift-splat-shoot-depth)。

- **BEVDet(Huang et al., arXiv:2112.11790, 2021)** が LSS の lift-splat を 3D 物体検出に応用し、実用的な検出パイプラインとして広めました。

- **BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)** が、深度を明示的に予測しない暗黙投影方式を確立しました。学習可能な BEV クエリが 3D 参照点を各カメラに投影した位置の特徴に、deformable attention で注目する「空間クロスアテンション」と、過去フレームの BEV を再利用する「時間セルフアテンション」を組み合わせました。本ノートの方式Bの直接の代表です。詳細は [BEVFormer風の暗黙的投影](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0273_bevformer-deformable-attention)。

- **PETR / PETRv2(Liu et al., ECCV 2022 / ICCV 2023)** は、毎回投影する代わりに 3D 座標情報を画像特徴に位置埋め込みとして焼き込む別アプローチです。詳細は [PETR / PETRv2 の3D位置埋め込み](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0288_petr-3d-position-embedding)。

- **UniAD(Hu et al., CVPR 2023, 最優秀論文)** は、BEVFormer を BEV エンコーダに据え、検出・追跡・地図・予測・計画を1本のクエリの流れで統合しました。BEV 表現を End-to-End 自動運転の共通基盤に位置づけた重要な仕事です。詳細は [UniAD(プランニング指向の統合設計)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1218_uniad-planning-oriented)。

研究コミュニティでは、明示的深度方式(LSS 系)と暗黙投影方式(BEVFormer 系)を BEV 化の二大潮流として対比して語ります。

## 論文の実験結果(定量データ)

評価は主に nuScenes データセット(6カメラ・LiDAR・レーダーを積んだ車で都市部を走った大規模データセット。3D 物体検出の標準ベンチマーク)で行われます。主な指標は2つです。**mAP(mean Average Precision、平均適合率)** は「検出した物体がどれだけ正しく当たっているか」を 0〜1 で測り、高いほど良い。**NDS(nuScenes Detection Score)** は mAP に加えて位置・大きさ・向き・速度・属性の誤差を総合したスコアで、これも高いほど良い。

報告によると、BEVFormer は nuScenes テストセットで NDS およそ 0.569、mAP およそ 0.481 を達成し、発表当時のカメラのみ手法で最先端を更新しました(バックボーンは ResNet-101-DCN)。これは、それ以前の単純な手法に比べて NDS で数ポイント高い水準です。

明示的深度方式の改良である **BEVDepth(Li et al., AAAI 2023)** は、LSS の深度予測を LiDAR で教師あり学習する仕組みを足し、nuScenes で NDS を大きく押し上げました(報告では 0.60 前後の水準)。これは「明示的深度方式では深度予測の質が BEV の質を直接左右する」ことを定量的に示したアブレーションとして重要です。深度の教師信号を抜くと精度が落ちる、という関係がはっきり出ています。

暗黙投影方式の側でも、時間方向の情報(過去フレームの BEV)を使うか否かが大きく効きます。BEVFormer の論文では、時間セルフアテンションを外すと特に速度推定の誤差が悪化し、遮蔽された物体の検出が落ちることが報告されています。逆に時間を入れると、LiDAR を使わないカメラのみでも速度をある程度推定できるようになります。

## メリット・トレードオフ・限界

メリット。すべてのカメラを実世界の共通座標系にまとめるので、周囲の位置関係が一貫し、後段の経路計画が扱いやすくなります。複数カメラにまたがって写る物体が1か所にまとまり、二重計上を避けられます。暗黙投影方式は明示的深度予測が不要なので、深度推定誤差が BEV 上の位置ずれに直結する問題を避けられます。

トレードオフと限界。第一に計算とメモリが重いことです。BEV 解像度を上げるとマス目数が二乗で増えます(450×300 で 13.5 万マス)。実際の学習では高解像度 BEV はバッチサイズを小さくしても GPU メモリを 20GB 超消費する、という報告があり、解像度とバッチサイズのトレードオフが常に問題になります。第二に、正しい投影には正確なカメラ校正が不可欠です。校正がずれると投影先がずれ、間違った特徴を拾います。第三に、暗黙投影方式は中間に「深度」という解釈しやすい量を持たないため、デバッグや誤り分析がしにくい面があります。

未解決の課題として、グリッド表現そのものの限界があります。固定解像度のグリッドは、遠方を粗く、近傍を細かく扱う、といった適応的な解像度配分が苦手です。これに対し、グリッドではなく疎な点やクエリで表現する方向(後述)が研究されています。

## 発展トピック・研究の最前線

研究の流れはいくつかの方向に分岐しています。一つは BEV を経由しない「疎な(sparse)」表現です。SparseBEV や StreamPETR は、密なグリッドを作らず物体クエリだけを保持して計算を軽くします。もう一つは時間方向の長期化で、複数フレームを効率よく蓄積して動きの推定を強める方向です。さらに、占有グリッド予測(occupancy prediction、空間を細かいボクセルに区切って「そこに物があるか」を予測する)へ BEV を拡張し、高さ情報も含めて環境を 3D で表す流れもあります。Tesla が公開した Occupancy Network の発表以降、この方向が活発になりました。また LiDAR や HD マップを同じ BEV 座標系で融合するマルチモーダル化(BEVFusion など)も実用上重要です。
