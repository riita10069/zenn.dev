# modeによるforward分岐設計

## ひとことで言うと
ニューラルネットワークの「順方向計算(forward)」に `mode="train"` か `mode="infer"`(推論)という引数を1つ持たせ、その値によって「返り値の意味」と「補助モジュールを実行するかどうか」を切り替える設計の話です。学習用と推論用で別々の入口を作らず、1つの入口(エントリポイント)に処理を集約し、末尾だけを分岐させます。コードの重複を減らし、学習と推論で「同じ前処理」が確実に共有されることを保証する、大規模な深層学習コードベースで広く使われる設計パターンです。

## 直感的な理解
1台の機械に「製造モード」と「検査モード」のスイッチが付いている状況を想像してください。製造モードでは原材料から製品を作り、最後に「規格との誤差(どれだけズレているか)」を測ってその誤差を記録します。検査モードでは同じ機械で製品を作りますが、誤差測定の工程は飛ばし、製品そのものを出荷します。前半の「製品を作る」工程は両モードでまったく同じ。違うのは末尾の処理だけです。

もし製造ライン用と検査ライン用に機械を2台別々に作ってしまうと、前半の作り方を改良したとき、2台とも直さないと食い違いが起きます。「片方だけ直して、もう片方は古いまま」というバグは現実に頻発します。だから1台にスイッチを付け、前半を共有し、末尾だけ切り替えるほうが安全で保守しやすい。深層学習モデルの学習と推論の関係も、まさにこれと同じ構造です。

## 基礎: 前提となる概念

用語を順にかみ砕きます。

- forward(順伝播): ニューラルネットワークに入力を入れて出力を計算する処理。PyTorch では `model(x)` と書くと、内部で定義された `forward(self, x)` メソッドが呼ばれます(正確には `__call__` がフックを処理したうえで `forward` を呼びます)。
- 学習(train)と推論(inference): 学習は「モデルの予測と正解(教師データ)を見比べ、どれだけ間違えたかを表す損失(loss)を計算し、その損失が小さくなるように重み(モデル内部のパラメータ)を更新する」段階。推論は「学習を終えたモデルに新しい入力を与え、答えを出させる」段階。実環境で実際に使われるのは推論です。
- 損失(loss): 予測と正解のズレを1つの数値(スカラー)にまとめたもの。学習はこの値を最小化することです。自動運転の軌道予測なら「予測した進路と、人間ドライバーが実際に取った進路の差」のような量になります。
- 逆伝播(backward): 損失を各重みについて微分し、「どの重みをどちらに動かせば損失が減るか」を計算する処理。PyTorch では `loss.backward()` の一行で行われます。逆伝播を回すには「微分可能な計算グラフ」が必要で、推論時には不要です(`torch.no_grad()` でグラフ構築を省いてメモリと時間を節約します)。
- 補助モジュール: 学習を助けるために追加する部品。例えば「未来の状態を予測させ、その予測誤差を補助的な損失に足す」ような自己教師あり的な仕掛け。学習時には損失に寄与させたいが、推論時には答えを出すのに不要で、しかも計算とメモリを食うことがある。

ここで核心の問題です。学習と推論では「やること」も「欲しい返り値」も違います。

- 学習では、軌道そのものより「損失(スカラー)」が欲しい。さらに補助モジュールを使うなら、それを実行する必要がある。
- 推論では、損失は要らない。欲しいのは「軌道(出力そのもの)」だけ。補助モジュールは走らせるだけ無駄で、メモリと時間を食う。

この違いをどう実装に落とすか。素朴な選択肢は2つあります。

(A) `forward_train` と `forward_infer` の2つの関数に分ける。
(B) 1つの `forward` に `mode` 引数を持たせ、内部で末尾だけ分岐する。

## 仕組みを詳しく

多くの大規模コードベースは(B)を採ります。理由は、学習も推論も「入力→特徴抽出→融合→時系列処理→プランナー(出力生成)」という前半の流れがまったく同じだからです。(A)で2つに分けると同じ前半処理を2か所に書くことになり、片方だけ直して食い違うバグが起きやすい。(B)で1つにまとめて末尾だけ分岐すれば、前半は1か所で済みます。

典型的なシグネチャと契約はこうなります。

```python
def forward(self, inputs, mode="train", target=None, **kwargs):
    # --- ここから前半: train/infer 共通 ---
    features = self.backbone(inputs)
    fused    = self.fusion(features)
    hidden   = self.temporal(fused)
    # --- ここまで共通 ---

    if mode == "train":
        loss   = self.planner.compute_loss(hidden, target)   # 損失(スカラー)
        future = self.future_module(hidden)                  # 補助モジュールを実行
        return loss, hidden, future
    else:  # "infer" など
        trajectory = self.planner.predict(hidden)            # 軌道そのもの
        return trajectory, hidden, None                      # 補助モジュールはスキップ
```

この設計の核は「返り値タプルの1番目の意味が mode で切り替わる」点です。

```
forward(..., mode="train", target=正解)
   → (loss[スカラー], hidden, future)    ← 学習ループはこの loss を backward する

forward(..., mode="infer")
   → (trajectory[出力], hidden, None)    ← 実環境はこの軌道を使う。loss は計算しない
```

なぜ「損失計算をモデル内部に閉じ込める」のが嬉しいか。損失をモデルの中(プランナーの中)で計算して返すと、外側の学習ループは「どんなプランナーが使われているか」を知らなくてよくなります。例えばプランナーが単純な回帰(GRU で軌道を直接出す)でも、確率的な生成モデル(Flow Matching や拡散モデル)でも、損失の計算方法はまったく違いますが、外のループは常に「返ってきた1番目の値を `backward()` する」だけで済みます。プランナーを別方式に差し替えても、学習ループのコードは一文字も変えなくてよい。これは関心の分離(separation of concerns)という設計原則の実例で、ソフトウェア工学でいう「結合度(coupling)を下げる」典型です。

補助モジュールの実行可否を mode で切り替えるのも重要です。未来予測のような補助モジュールは、高解像度の中間表現(例えば大きな BEV 特徴マップ)を扱うと、1サンプルあたり数百MBものメモリを消費することがあります。具体的には 450×300×256 次元の特徴マップを fp32 で1サンプル持つと約 132 MB、これに backward 用の中間活性化が積み重なる。推論時にこれを走らせないだけで、メモリと推論レイテンシを大きく節約でき、実環境(車載の限られた計算資源)での実用性が上がります。

なぜ mode 1つに集約できるか、整理すると:
- 前半(特徴抽出→融合→時系列→プランナー本体)は train/infer 共通。
- 末尾だけが分岐する。学習なら「損失を計算 + 補助モジュールを実行」、推論なら「出力をそのまま返す + 補助モジュールをスキップ」。

実装上の注意点として、この設計は「契約(返り値の意味)」を docstring や型注釈で明示しないと、呼び出し側が事故を起こします。返り値が `Union[loss, trajectory]` のように mode 依存で変わるため、設計の意図(契約)が先に確立され、実装が段階的にそれへ追従していく、という開発過程もよく見られます。契約と実装を同期させ続ける規律が、この設計を安全に運用する鍵です。

## 手法の系譜と主要論文

これは特定の自動運転論文の発明ではなく、深層学習フレームワークの設計慣習(ソフトウェア設計パターン)の系譜です。

1. PyTorch の train/eval フラグ(Paszke et al., "PyTorch", NeurIPS 2019, arXiv:1912.01703): PyTorch は学習と評価で挙動が変わる層(Dropout=ランダムに一部のニューロンを無効化する層、BatchNorm=バッチ統計で正規化する層)のために、モジュール全体に「train か eval か」のブール状態を持たせます(`model.train()` / `model.eval()`)。動機は「同じ forward コードで、学習時と評価時の確率的挙動を切り替えたい」こと。本ノートの `mode` 引数は、この発想を「層の確率的挙動」だけでなく「返り値の意味」「補助モジュールの実行可否」まで拡張したものと位置づけられます。

2. 検出ライブラリの mode 引数(MMDetection: Chen et al., arXiv:1906.07155, 2019 / Detectron2: Wu et al., 2019): 大規模物体検出ライブラリは、`forward` に `return_loss=True/False` や `mode="loss"/"predict"/"tensor"` といった引数を持たせるのが定石です。学習では損失の辞書を、推論では検出結果を返します。動機は、学習パイプラインと推論パイプラインで「同じモデル定義」を共有しつつ、出力だけを切り替えること。MMDetection v2/v3 の `BaseDetector.forward` はまさにこの形で(v3 では `mode` が `"loss"/"predict"/"tensor"` の3値)、コミュニティに広く影響を与えました。損失計算をモデル内部に閉じ込める思想も、これらが普及させたものです。

3. Hugging Face Transformers の慣習(Wolf et al., EMNLP 2020): Transformers ライブラリでは、`forward` に `labels` を渡すと損失も計算して返し、渡さなければ出力だけ返す、という方式が定着しています。これも「1つの forward、引数の有無で挙動分岐」という同じ系譜です。`labels=None`(=推論)、`labels=正解`(=学習で損失も返す)という設計は、本ノートの `mode` + `target` の組み合わせと本質的に同じです。返り値は名前付きの `ModelOutput`(辞書ライクなオブジェクト)で、フィールドの有無が状況で変わる点もタプル分岐と同じ問題を抱えます。

これらの先行慣習が示すのは、「単一 forward + mode 分岐」が大規模な機械学習コードベースで広く採られた定石だということです。

## 論文の実験結果(定量データ)

このトピックは精度を競う手法ではなく設計パターンなので、ベンチマーク数値という形のデータはありませんが、定量的に語れる側面があります。

- コード重複の削減: (A)の二関数方式では前半処理が2か所に複製されます。前半が数百行に及ぶ大規模モデルでは、複製が保守コストと食い違いバグの温床になります。MMDetection や Detectron2 が mode 方式を採用した理由の一つは、数十種類の検出器を統一インターフェースで扱うためで、もし各検出器が train/infer の2関数を別々に持てば、メンテナンス対象の関数数が単純計算で倍増します。数十アーキテクチャ × 2関数の保守は現実的でなく、統一 forward が必須でした。

- 推論時メモリの節約: 補助モジュールを推論時にスキップする効果は、その補助モジュールが扱う中間表現のサイズに比例します。例えば 450×300 の BEV 特徴マップ(チャンネル次元込み)を fp32 で持つと1サンプルあたり百数十MB規模になり、これを推論で走らせないだけで、車載 GPU の限られたメモリ予算内に収められるか否かが変わります。アブレーション的に言えば「mode 分岐で補助モジュールをスキップする/しない」は、推論可能なバッチサイズや解像度の上限を直接左右します。

- 差し替えコストの定量化: 損失計算をプランナー内部に閉じ込める設計では、プランナーを別方式(回帰→生成モデル)に差し替えても学習ループのコード変更行数がゼロに保てます。これは「結合度(coupling)」を下げる典型例で、ソフトウェア工学では結合度の低下が変更コストの低下に直結することが知られています(モジュール間の依存が少ないほど、片方の変更が他方に波及しない)。

## メリット・トレードオフ・限界

メリット
- 学習と推論で共通の前半処理を1か所に書ける。コード重複が減り、片方だけ直して食い違うバグを防げる。
- 損失計算をプランナー内部に閉じ込められるので、学習ループはどのプランナーかを知らずに `loss.backward()` できる。プランナーの差し替えが容易。
- 推論時に重い補助モジュールを自動でスキップでき、メモリと時間を節約できる。車載などリソース制約の厳しい環境で効く。

トレードオフ・限界
- 返り値タプルの「1番目の意味」が mode で変わる。呼び出し側が mode を取り違えると、損失だと思って軌道を `backward()` する(あるいはその逆)といった事故が起きうる。Python の動的型付けではこのミスを型システムが防げないため、docstring と規律に依存する。型注釈で `Union[Loss, Trajectory]` のように書くしかなく、静的に守りきれないのが弱点です。
- フレームワーク標準の `model.train()/eval()` フラグと、独自の `mode` 引数という2系統の切り替えが併存しうる。例えば `model.eval()` を呼び忘れたまま `mode="infer"` で推論すると、Dropout が効いたままになり結果がぶれる、という不整合が起きうる。両系統を整合させる注意が要ります(推論側で `mode="infer"` を渡したら内部で `self.eval()` も呼ぶ、といった防御策がときに採られます)。
- 1つの関数に学習・推論の両方の責務が乗るため、分岐が増えるほど関数が肥大化し読みにくくなる。mode が3つ4つと増え、各 mode で大きく違う処理をするようになったら、`forward_train` / `forward_infer` への分割や、Strategy パターンへのリファクタを再検討する余地があります。
- 研究上の論点として、確率的生成モデル(拡散・Flow Matching)をプランナーに使う場合、学習時の損失(例えばノイズ予測誤差)と推論時のサンプリング手続き(反復的なデノイズ)は内部実装が大きく異なります。これを mode 一つで綺麗に閉じ込められるかは設計の腕の見せ所で、サンプリングのステップ数など推論固有のハイパーパラメータをどこに置くか、という新たな設計判断が生じます。

## 発展トピック・研究の最前線
- マルチタスク学習での mode 拡張: 検出・セグメンテーション・軌道予測を1つのモデルで同時に学ぶ場合、mode を「どのタスクの損失を返すか」まで拡張する設計が研究されています。複数の損失をどう重み付けして合算するか(不確実性に基づく重み付けなど)も合わせて設計判断になります。
- TorchScript / ONNX エクスポートとの相性: 推論用に計算グラフを書き出す(エクスポートする)際、mode 分岐(Python の if 文)がそのままでは追跡(trace)しにくい問題があります。`torch.jit.trace` は実行された一方の分岐しか記録しないため、推論パスだけを切り出す `torch.jit.script` 対応や、エクスポート時に分岐を固定する工夫が実務上の課題です。
- 生成的プランナーの統一インターフェース: 拡散モデルやFlow Matchingをプランナーに用いる End-to-End 自動運転(VAD, DiffusionDrive など)の流れの中で、学習(損失)と推論(サンプリング)を1つの forward 契約に収める設計が模索されています。
- 構造化出力での型安全化: 返り値タプルの曖昧さを解消するため、`TypedDict` や `dataclass` で「学習用出力」と「推論用出力」を別の型として明示し、mypy などの静的型チェッカで取り違えを検出させるアプローチも広まりつつあります。
