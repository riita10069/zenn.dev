# オフライン強化学習 (Offline RL)

## ひとことで言うと

オフライン強化学習(Offline Reinforcement Learning、データ駆動型強化学習とも呼ぶ)は、「もう集め終わった行動ログだけ」を使って、それ以上は環境とやり取りせずに方策(policy、状態を見て行動を決める関数)を改善する強化学習です。普通の強化学習が「環境で実際に動いてみて、その結果から学ぶ」のに対し、オフライン RL は一度も新しく試行せず、固定のデータセットの中だけで学ぶのが決定的な違いです。自動運転のように実車での試行錯誤が危険でコストの高い領域で、模倣学習の次の一手として注目されています。

## 直感的な理解

なぜわざわざ「環境とやり取りしない RL」が要るのでしょうか。運転を例に考えます。

通常の強化学習は「実際に走ってみて、うまく走れたらご褒美(報酬)、ぶつかったら罰、を受け取って学ぶ」やり方です。これは囲碁やゲームでは強力ですが、運転では致命的です。学習途中の下手な方策に実車を走らせれば事故になります。シミュレータで代替しても、模擬と現実の差(sim-to-real ギャップ)に悩まされます。

一方、模倣学習([模倣学習 (Behavior Cloning)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0560_imitation-learning))は安全です。集めた人間の運転ログを教師あり学習で真似るだけなので、新たに走る必要がありません。しかし模倣学習には「お手本を真似るだけで、行動の良し悪しを評価しない」という限界があります。データに混じった下手な運転も、上手い運転も区別せず真似てしまい、結果として「専門家の平均的性能」が上限になります。

オフライン RL はちょうど中間を狙います。発想はこうです。「すでに手元にある運転ログには、良い運転も平凡な運転も混じっている。新たに走らなくても、このデータの中で『どの行動がより高い報酬につながったか』を推定すれば、ただ真似る(模倣)より良い行動を選べるはずだ」。つまり追加の実車走行ゼロのまま、模倣より一歩賢い方策を作ろうという試みです。良い部分をつなぎ合わせる能力を「stitching(縫い合わせ)」と呼び、これがオフライン RL の核心的な売りです。たとえば「A 地点から B 地点へ行く良い軌跡」と「B 地点から C 地点へ行く良い軌跡」が別々のログに入っていれば、オフライン RL は両者を縫い合わせて「A から C への良い軌跡」を学べます。単純な模倣ではこれはできません。

## 基礎: 前提となる概念

オフライン RL を理解するには、強化学習の基本語彙を押さえる必要があります。知らない人向けに噛み砕きます。

- 状態 s(state): いまの状況。運転ならカメラ画像や周囲の物体配置。
- 行動 a(action): 状態を見て取る選択。運転なら加速度・ステア(操舵)など。
- 報酬 r(reward): 行動の結果もらえる「良さの点数」。前進したら+、衝突したら大きな−など。
- 方策 π(policy): 状態を入力して行動を出す関数。これを賢くするのが目標。
- 価値関数 V(s)(value function): その状態から将来もらえる報酬の合計見込み。「この状況はどれくらい有利か」を1つの数で表す。
- Q 関数 Q(s,a)(Q-function、行動価値関数): 「この状態でこの行動を取ったら、将来どれくらい報酬がもらえるか」。状態と行動のペアの良さ。
- 割引率 γ(gamma): 将来の報酬を「いまの価値」に換算するときの割引。0.99 なら「1ステップ先の報酬は 0.99 倍に値引きして数える」。遠い未来ほど軽く見る。
- ベルマン方程式(Bellman equation): 価値の自己整合条件。「いまの Q = いまの報酬 + γ × 次の状態の価値」。RL のほぼすべての手法がこの等式を満たすよう学習する。

オフライン RL に固有の最大の罠が distributional shift(分布シフト)と overestimation(過大評価)です。これは初心者には分かりにくいので具体的に説明します。ニューラルネットの Q 関数に「データに一度も出てこなかった行動 a の良さ」を尋ねると、根拠がないのに高い値を返しがちです(関数近似器の外挿は当てにならない)。普通の Q 学習は「次状態で Q が最大になる行動」を使って学習しますが、この max がまさにデータ外の行動を選びやすく、その「幻の良い行動」に方策が引き寄せられて崩壊します。オンライン RL なら「実際にやってみてダメだった」とフィードバックで訂正できますが、オフラインでは新しく試せないので、過大評価がいつまでも訂正されません。これを抑えることが、オフライン RL の全手法の中心テーマです。

## 仕組みを詳しく

オフライン RL の代表的かつ実装が比較的シンプルな手法 IQL(Implicit Q-Learning)を例に、典型的な処理を段階的に見ます。多くの実装は次の3つのネットワークを使います。

- 方策 π: 状態(例: カメラ画像から得た特徴ベクトル)から行動を出す。模倣学習で事前学習したものを出発点にすることが多い。
- 価値関数 V: 状態の良さを1つの数で出す MLP(多層パーセプトロン)。典型的には `feat_dim → 256 → 1`。
- Q 関数: 状態と行動のペアの良さを出す MLP。典型的には `(feat_dim + action_dim) → 256 → 1`。

### ステップ1: Q 関数の更新(ベルマン回帰)

```
q_target = reward + gamma * next_v       # 次状態の価値を使った教師信号
qf_loss  = MSE(q_pred, q_target)         # 予測 Q を教師に近づける
```

「いまの行動の良さ Q」が「いま得た報酬 + 割引した次状態の価値」に一致するよう学習します。これがベルマン方程式の実装です。

### ステップ2: 価値関数の更新(IQL の肝、expectile 回帰)

ここが IQL が「データに無い行動を一切評価しない」ための工夫です。普通の RL は次状態で「全行動の中で最大の Q」を取り、これがデータ外行動の過大評価を招きました。IQL はその max を取らず、代わりに価値 V を「Q の上側 expectile」として推定します。expectile(エクスパクタイル)は分位点(quantile)の2乗誤差版で、非対称な重み付き2乗誤差で定義します。

```
diff   = q_pred - v_pred
weight = expectile         if diff > 0  else  (1 - expectile)   # 例: 0.7 と 0.3
v_loss = mean(weight * diff^2)
```

`expectile = 0.7` なら「Q が V より大きい(diff > 0)場合は重み 0.7、小さい場合は 0.3」と非対称に重み付けします。これにより V は「Q 値分布のやや上の方」を狙い、データ内で実際に観測された良い行動の Q に V を近づけます。重要なのは、この計算が現在のデータ点の Q しか使わず、データ外の行動の Q を一度も問い合わせない点です。expectile を 0.5 にすると単なる平均(普通の回帰)になり、1 に近づけるほど「データ内の最良行動」に近い保守的な上限を狙います。max のリスクを取らずに「ほぼ max」を近似する巧妙な仕掛けです。

### ステップ3: 方策の更新(advantage-weighted regression)

```
advantage  = qf(s, a) - vf(s)                  # この行動は平均よりどれだけ良かったか
weights    = exp(beta * advantage).clamp(max=W) # 良い行動ほど重い。爆発防止に上限クリップ
policy_loss = weights * MSE(pred_action, a)     # 重み付きの模倣
```

アドバンテージ A = Q − V は「この行動が、その状態の平均的な良さよりどれだけ良かったか」を表します。A が大きい(平均より良かった)行動ほど大きな重み `exp(β·A)` をかけて強く真似ます。`β`(beta、典型的に 1〜10)がその強調の効き具合で、大きいほど良い行動だけを尖って真似ます。これは「重み付き模倣学習」と読めます。普通の模倣は全お手本を同じ重みで真似ますが、ここでは良かった行動を重く、平凡な行動を軽く真似る。だから模倣より賢くなれます。`clamp(max=W)` は重みが指数で爆発して学習が壊れるのを防ぐ安全装置です。この方策抽出は AWR / AWAC(後述)と同じ系譜です。

### 報酬の設計

オフライン RL は報酬ラベルを必要としますが、運転ログにはしばしば明示的な報酬がありません。そこで代理報酬(proxy reward)を設計します。よくある単純な例は次のようなものです。

```
jerk     = sum((next_action - action)^2)   # 操作の急変=ガタつき(乗り心地の悪さ)
progress = action の前進成分                # どれだけ前に進んだか
reward   = progress - lambda * jerk         # 前進は+、ガタつきは−
```

「滑らかに前進したら良い」という常識的な代理報酬です。ただしこれは運転の良し悪しを完全には表しません。安全性・信号順守・歩行者への配慮などが入っていないため、代理報酬の設計こそがオフライン RL の成否を左右します。報酬が雑だと「報酬を稼ぐが危険な方策」を学んでしまう(報酬ハッキング)危険があります。

### 全体像(ASCII図)

```
収集済みの行動ログ(走行不要・固定データ)
        │  (state, action, reward, next_state) を取り出す
        ▼
 (報酬が無ければ代理報酬を計算)
        │
        ├─► Q更新(ベルマン回帰: r + γ·V')
        ├─► V更新(expectile回帰=データ内に閉じる, max を避ける)
        └─► 方策更新(advantage重み付き模倣 exp(β·A))
                    │
                    ▼
      模倣より一歩賢い方策(環境との追加やり取りゼロ)
```

## 手法の系譜と主要論文

オフライン RL は「どうやってデータ外行動の過大評価を抑えるか」を巡って進化してきました。

- BCQ(Batch-Constrained Q-learning、Fujimoto, Meger & Precup, ICML 2019, arXiv:1812.02900): 過大評価の原因が「データに無い行動の選択」だと特定した先駆け。生成モデル(VAE)でデータに似た行動だけを候補に絞り、その範囲で Q を最大化しました。動機は「データ分布から離れすぎない(batch-constrained)」こと。これがオフライン RL を独立した研究テーマとして立ち上げた論文の一つです。

- CQL(Conservative Q-Learning、Kumar et al., NeurIPS 2020, arXiv:2006.04779): 過大評価を抑えるため、データに無い行動の Q 値を意図的に低く(保守的に)見積もる正則化項を Q 学習に足しました。直感的には「見たことのない行動は疑ってかかる」。効果は安定したオフライン学習で、当時の標準的ベースラインになりました。トレードオフは保守的すぎて性能が頭打ちになりうること、追加損失項の係数調整が要ること。

- AWAC(Advantage-Weighted Actor-Critic、Nair et al., 2020, arXiv:2006.09359): advantage-weighted regression による方策抽出を整理し、オフライン事前学習からオンライン微調整への橋渡しを示しました。IQL の方策更新はこの系譜です。

- TD3+BC(Fujimoto & Gu, NeurIPS 2021, arXiv:2106.06860): 「オフライン RL は複雑すぎる」という反省から、標準的なオンライン手法 TD3 の方策損失に行動クローニング(BC)項を一つ足し、さらに状態を正規化するだけ、という極めて単純な手法を提案。それでも D4RL で CQL に匹敵する性能を出し、「派手な保守化機構より、データから離れすぎないという最小限の制約があれば足りる」ことを示しました。オフライン RL のミニマリズム路線の代表で、実装・調整が容易なため強いベースラインとして定着しています。

- IQL(Implicit Q-Learning、Kostrikov, Nair & Levine, ICLR 2022, arXiv:2110.06169): CQL のように明示的な保守化をせず、「次状態で max Q を取る代わりに expectile で V を推定する」ことで、データに無い行動の Q を一度も問い合わせずに学習します。提案の動機は「データ外行動の評価を構造的に回避し、実装をシンプルかつ安定にする」こと。当時の D4RL ベンチマークで高性能かつ学習が安定・高速で、現在も強力なベースラインです。トレードオフは、データに含まれない大胆な改善は学べない(データ内の良い行動の組み合わせに留まる)こと、expectile の係数など調整パラメータが残ること。

- Offline RL Tutorial(Levine, Kumar, Tucker & Fu, 2020, arXiv:2005.01643, "Offline Reinforcement Learning: Tutorial, Review, and Perspectives"): 問題設定と中心課題を整理した総説。核心は分布シフトと過大評価で、これを抑える制約が全手法に必要だと示しました。自動運転のように試行が危険な領域でオフライン RL が有望だと位置づけました。

近年は系譜が2方向に広がっています。一つは Decision Transformer(Chen et al., NeurIPS 2021, arXiv:2106.01345)に代表される「RL を系列予測問題として解く」流れで、報酬を条件として与え、望む報酬につながる行動列を Transformer で生成します。もう一つは拡散モデルを方策に使う Diffuser / Diffusion Policy で、複雑な多峰性のある行動分布を表現します。

## 論文の実験結果(定量データ)

オフライン RL の標準ベンチマークは D4RL(Datasets for Deep Data-Driven RL、Fu et al., 2020, arXiv:2004.07219)です。MuJoCo の歩行ロボット(halfcheetah, hopper, walker2d)や迷路探索(antmaze)などのタスクで、データの質を変えた複数の設定(専門家だけの medium-expert、中級者の medium、ランダム混じりの medium-replay など)を用意し、学習した方策の正規化スコア(normalized score、ランダム方策を 0、専門家を 100 とした相対点)を測ります。

IQL 論文(Kostrikov et al., 2022)の報告では、Gym ロコモーション系の平均でおよそ 77 前後の正規化スコアを達成し、CQL(平均およそ 73 前後)や旧来の手法を上回るか同等でした。特筆すべきは antmaze(報酬がゴール到達時にしか得られない疎報酬で、stitching が決定的に効くタスク)です。難しい antmaze-large-play / antmaze-large-diverse で、多くのベースラインがほぼ 0% の成功率に沈む中、IQL は数十%(報告でおよそ 40〜60% 前後)の成功率を出しました。これは「別々の良い断片を縫い合わせる」というオフライン RL 本来の強みが、疎報酬かつ長い系列で顕著に現れることを示します。

CQL 論文(Kumar et al., 2020)は、データがランダム寄り(medium-replay 等)で品質が低い設定で、行動クローニング(単純な模倣)を大きく上回ることを示しました。模倣はデータの平均に張り付くのに対し、CQL はデータ内の良い部分を選び取れるためです。この「データの質が低いほど RL の優位が大きい」という傾向は、オフライン RL の価値を理解する鍵です。逆にデータが専門家だけで構成されていると、単純な模倣でも十分に高性能で、RL の上積みは小さくなります。

アブレーション(要素を抜く実験)の知見も重要です。IQL では expectile の係数 τ を 0.5(=ただの回帰、実質的に SARSA 的)に下げると性能が落ち、難しい疎報酬タスクほど τ を 0.7〜0.9 と高くする(より楽観的に良い行動を狙う)ことが効くと報告されています。一方 τ を 1 に近づけすぎると max に近づき、過大評価で不安定になります。つまり「どれだけ楽観的に良い行動を狙うか」のバランス点がタスクごとに存在することが、アブレーションから読み取れます。

## メリット・トレードオフ・限界

メリット
- 環境との追加やり取りゼロで方策を改善できる(安全・低コスト)。自動運転のように試行が危険な領域に適する。
- 模倣学習と違い「良い行動を重く真似る」ため、専門家の平均を超え、断片を縫い合わせる(stitching)余地がある。
- IQL のようにデータ外行動の評価を構造的に避ける手法は、オフラインでも学習が安定しやすく実装も比較的シンプル。
- すでに収集済みのログをそのまま再利用でき、模倣学習の自然な後処理(post-training / fine-tuning)として組み込める。

トレードオフ・限界
- 性能の天井がデータの質と多様性に縛られる。データに無い状況・行動には根本的に弱い。
- 報酬(運転では代理報酬になりがち)の設計に成否が大きく依存する。安全性・規範が入っていないと危険な方策を選びうる(報酬ハッキング)。
- ハイパーパラメータ(expectile の τ、advantage 温度 β、割引 γ、保守化係数)の調整が残り、オフラインでは「実際に試して選ぶ」ことができないため、ハイパラ選択・モデル選択(offline model selection)自体が未解決の難問。
- オフライン学習した方策の本当の良し悪しは、最終的にはオンライン(クローズドループ)で走らせないと分からない。オフラインの評価指標(推定価値など)は楽観バイアスを含みやすい。

研究上の未解決課題として、(1) 信頼できるオフライン評価・ハイパラ選択、(2) 報酬関数の自動推定(inverse RL との接続)、(3) 学習したシミュレータ(World Model)で生成した経験を混ぜる model-based offline RL、(4) オフライン事前学習→少量オンライン微調整の安全な接続、などが活発に研究されています。

## 発展トピック・研究の最前線

model-based offline RL(例: MOPO=Yu et al. NeurIPS 2020, COMBO=Yu et al. NeurIPS 2021)は、データから環境の遷移モデルを学び、その中で仮想的に試行を増やして方策を鍛えます。これは「学習したシミュレータ(World Model)で安全に試行錯誤する」発想と直結し、自動運転で特に有望視されています。固定ログだけでは届かない状況も、モデルが補間・外挿してくれる可能性があるためです。ただしモデルの誤差が方策を誤らせるリスクがあり、モデルの不確実性が高い領域では報酬にペナルティをかける(pessimism、悲観主義)などの工夫が要ります。MOPO はモデル予測の不確実性(複数モデルのばらつき)を推定し、その大きさに比例した量を報酬から差し引くことで「モデルが自信のない領域へ方策が逃げ込む」のを防ぎます。自動運転特有の難しさは、シーンが高次元(複数カメラ・多数の動的物体)で長期予測の誤差が累積しやすく、信頼できる遷移モデル(World Model)を学ぶこと自体が未解決の大問題である点です。

なお、運転には「報酬がそもそも与えられていない」という根本問題があり、これが他分野のオフライン RL と決定的に違います。代理報酬の手設計は報酬ハッキングを招きやすいため、専門家のデモから報酬関数を逆推定する逆強化学習(inverse RL)や、人間の選好(どちらの走行が良いか)から報酬を学ぶ手法との接続が研究されています。また、オフライン学習した方策をハイパラ選択する「offline policy selection / off-policy evaluation(OPE)」は、実走なしで方策の良し悪しを当てねばならず、推定価値が楽観バイアスを持つため依然として信頼性が低い、というのがこの分野全体のボトルネックになっています。

系列モデリング系(Decision Transformer など)は、RL を「望む報酬を条件にした行動列の生成」と捉え直し、大規模 Transformer の表現力を持ち込みました。運転のように長期の文脈が効くタスクと相性が良いと期待されています。
