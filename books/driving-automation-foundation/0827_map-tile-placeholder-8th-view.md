# マップタイルのプレースホルダ (追加ビューのゼロ埋め)

## ひとことで言うと
運転モデルには「複数のカメラ画像 + 1枚の地図画像」をひとまとめにして入力する設計がよく使われます。ところが学習データに地図を絵にしたもの (マップタイル) が含まれていないことがあります。そういうとき、地図用の枠を全部ゼロ (真っ黒な画像) で埋めて「形 (テンソルの shape) だけ合わせる」やり方をプレースホルダ方式と呼びます。将来ちゃんとした地図描画器 (レンダラ) ができたら、その1か所を差し替えるだけで本物の地図が流れ込むように設計しておきます。

## 直感的な理解
料理のレシピで「最後にハーブを散らす」と書いてあるのに、今日はハーブが手元にない。でもレシピの工程数や盛り付けの構成は変えたくない。そこで「ハーブの位置に何も置かない (= ゼロ)」状態で完成させ、ハーブが手に入ったら同じ位置に足すだけにしておく。これがプレースホルダの発想です。

運転モデルでこれが起きるのは、地図情報を「絵 (ラスタ画像)」としてカメラ画像と一緒に入れたいのに、その絵を作る仕組みがまだ無いときです。地図の枠を消してしまうと、モデルの入力数が変わり、画像を融合する部分や位置埋め込みの設計まで作り直すことになります。それは影響範囲が大きい。だから「枠は残す、中身は空 (ゼロ)」で進め、インターフェース (入出力の形) を凍結したまま中身だけ後から育てる、という段階的な開発を可能にします。

## 基礎: 前提となる概念

- ビュー (view): モデルに入れる1つの視点・画像のことです。前方カメラ、左カメラ、地図…それぞれが1ビュー。
- マルチビュー融合 (multi-view fusion): 複数の視点の画像をまとめて1つの判断にする処理です。各ビューを特徴量に変換し、それらを結合 (concat)、注意機構 (attention)、あるいは鳥瞰図 (BEV) 空間への投影などで統合します ([マルチビュー融合アーキテクチャ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1046_multi-view-fusion-design-doc))。
- HDマップ (High Definition Map, 高精度地図): 車線・標識・信号・横断歩道まで含む高精細地図です。センチメートル級の精度を持ち、自動運転の経路計画に使われます。普通のカーナビ地図が「道がある」程度なのに対し、HD マップは「右折専用レーンが2本」「停止線の正確な位置」まで持ちます。
- ラスタ化 (rasterization): 地図やベクトル情報を「ピクセルの絵 (ラスタ画像)」に描き出すことです。逆に、線や点の列のまま扱うのが「ベクトル表現」。
- トップダウンビュー / 鳥瞰図 (top-down / bird's-eye view, BEV): 真上から見下ろした視点。地図や周辺物体を重ねて描くのに自然な座標系です。
- プレースホルダ (placeholder): 本物が来るまでの「仮の埋め合わせ」。ここではゼロテンソル (全要素が0の多次元配列)。
- テンソルの shape (形): 多次元配列の各次元のサイズ。例: $(8, 3, 256, 256)$ は「8ビュー × 3チャンネル (RGB) × 縦256 × 横256」。

### なぜ地図を入れたいのか

カメラだけでは「この先がカーブだ」「この交差点は右折専用レーンがある」といった、今見えていない・見えにくい道路構造が分かりません。雨や逆光でカメラが見えにくいときも、地図は影響を受けません。HD マップをレンダリングして画像として渡せば、モデルは経路や道路構造の手がかりを得られます。とくに見通しの悪い交差点や、車線が複雑に分岐する場所で効きます。

### なぜゼロ埋めなのか

地図のレンダラが未整備だが、入力のビュー数を減らすとモデル側 (融合層・位置埋め込み = 各ビューがどの視点かをモデルに教える付加情報) の設計変更が必要になり影響が広がる。そこで「形は固定 (例: 8ビュー) のまま、地図ビューの中身だけゼロで埋める」プレースホルダを採ります。これにより、モデルの構造を一度決めたら、地図機能が完成するまで構造を変えずに済みます。

## 仕組みを詳しく

### ゼロタイルの作り方

参照テンソル (例: 1枚目のカメラ画像を前処理した後のテンソル、形 $(3, H, W)$) と「同じ形・同じ型 (dtype)・同じ置き場所 (CPU/GPU)」のゼロテンソルを作ります。

```
reference  = カメラ画像テンソル  shape (3, 256, 256), dtype float32, device cuda
map_tile   = ゼロテンソル        shape (3, 256, 256), dtype float32, device cuda
                                 中身は全画素 0.0
```

参照テンソルから形・型・デバイスを自動でコピーするので、解像度を変えても、CPU/GPU を切り替えても、地図ビューだけ整合がとれなくなる事故が起きません。もし形を直書きすると、カメラの解像度を変えたときに地図ビューだけ古い解像度のまま残り、stack 時に次元不一致でエラーになる、という典型的なバグを生みます。参照から継承する設計はそれを構造的に防ぎます。

### 差し替え点を1つにする設計

地図タイルを作る関数を1つだけ用意し、その関数を「唯一の差し替え点」にします。今はゼロを返すだけですが、本物のレンダラができたら、その関数の中身を「地図を描いて → 前処理 (transform) をかけて返す」に変えるだけで済みます。呼び出し側 (データ取り出し処理)、モデル、融合層は一切変えません。これはソフトウェア設計でいう「関心の分離」と「単一責任」の実践で、変更の影響範囲を1関数に閉じ込めています。

将来本物のレンダラに差し替えたときに API の形を変えずに済むよう、ゼロを返す今の段階でも、本来必要になる引数 (前処理パイプライン transform、自車位置、地図ハンドルなど) を受け取っておく、という配慮がよくなされます。今は使わなくても、引数の形を将来と揃えておくことで、差し替え時に呼び出し側の修正が不要になります。

### 組み立ての流れ

```
1. N台のカメラ → 最近傍フレーム ([カメラとegomotionのタイムスタンプ整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0800_camera-egomotion-timestamp-alignment)) → 前処理 → (3, 256, 256) を N個
2. map_tile = make_map_tile(transform, camera_tensors[0])   # ゼロの追加ビュー
3. リストに append → 長さ N+1
4. stack → (N+1, 3, 256, 256)  を返す
```

### before/after

```
今 (レンダラ無し):     追加ビュー = 全画素0 (真っ黒)
レンダラ完成後:        make_map_tile の中だけ実装変更
                       → 呼び出し側・モデルは無改修で本物の地図が流れ込む
```

## 手法の系譜と主要論文

地図をモデルに入れる方法は、大きく「ラスタ画像方式」と「ベクトル方式」に分かれます。プレースホルダが想定するのは前者 (画像として渡す) です。

### ラスタ画像方式

- IntentNet (Casas et al., CoRL 2018): LiDAR と HD マップをラスタ化して BEV 上に重ね、物体の検出と意図 (右折・直進・停止など) を同時予測しました。地図をチャンネルとして画像に重ねる初期の代表例です。地図を入れることで、車の見えだけからは分からない「この車は右折レーンにいるから右折しそう」という推論ができるようになりました。
- ChauffeurNet (Bansal et al., RSS 2019, Waymo): HD マップ・経路・他車・信号などを複数チャンネルのトップダウンラスタ画像に描いて入力する方式を採りました。理由は、地図と動的物体を同じ画像座標系に重ねれば CNN で一括処理でき、運転に必要な空間関係を素直に学べるからです。さらに「合成した危険シナリオ (わざと逸脱させた軌道や障害物を人工的に置いたデータ)」で学習を補強する工夫を導入しました。純粋な模倣だけでは経験できない失敗状況を人工的に作り、そこからの復帰を教えた点が新規性です。「地図をタイル画像として渡す」発想はこの系譜です。

### ベクトル方式 (ラスタの弱点への反論)

- VectorNet (Gao et al., CVPR 2020): ラスタ画像方式の弱点 (解像度に縛られ、細かい車線形状を表すのに大きな画像が要り計算も重い。さらに画像の大半が空白で無駄が多い) を指摘し、車線や経路を点の列 (ベクトル = ポリライン) として扱うグラフ表現を提案しました。各ポリラインをサブグラフとしてエンコードし、ポリライン間の関係をグローバルグラフで捉えます。効果は表現効率と精度の向上、トレードオフはデータ構造が複雑になること。
- LaneGCN (Liang et al., ECCV 2020): 車線をグラフとして扱い、グラフ畳み込みで車線間の接続関係 (前後・隣接・分岐・合流) を直接モデル化しました。ラスタ画像では表しにくい「トポロジー (つながり方)」を扱える点が新規性です。ピクセルの隣接ではなく「論理的に繋がった車線」を辿れます。

プレースホルダは「ラスタ画像方式の枠」を将来想定したものですが、ベクトル方式という有力な選択肢があることを知っておくと、レンダラ実装時の設計判断 (ラスタで描くか、いっそベクトルを別経路で入れるか) に役立ちます。

### より新しい流れ

- ST-P3 (Hu et al., ECCV 2022): カメラのみから BEV 特徴を作り、知覚・予測・計画を end-to-end でつなぎました。地図を外から与えるのではなくモデル内部で BEV 表現を作る方向で、「地図ビューを別途レンダリングして渡す」設計とは別の解法を示しています。
- UniAD (Hu et al., CVPR 2023, Best Paper): 検出・追跡・地図・予測・占有・計画を1つのネットワークに統合し、すべてを計画というゴールに向けて協調させました。地図はモデル内部のタスクとして扱われます。
- MapTR (Liao et al., ICLR 2023): 事前作成の HD マップに頼らず、車載カメラからその場でベクトル地図 (車線・境界線のポリライン) を生成するオンラインマッピングの代表手法です。

## 論文の実験結果(定量データ)

- 地図入力の効果 (ChauffeurNet, 2019): HD マップと経路をラスタ入力に加え、合成した逸脱シナリオでの学習と組み合わせると、地図なし・合成データなしのベースラインに比べてクローズドループ走行 (実際にモデルに運転させる評価) での失敗 (衝突・道路逸脱) が顕著に減ったと報告されました。論文の趣旨は「純粋な行動クローニングは数十マイル相当の走行で逸脱を起こすが、地図情報と合成摂動データを足すと安定して走れるようになる」というもので、地図が道路構造の事前知識を与えカメラだけでは得られない手がかりになることの定量的裏付けです。
- ベクトル vs ラスタ (VectorNet, 2020): Argoverse 軌道予測ベンチマークで、ベクトル表現は同等以下の計算量でラスタ CNN を上回る予測精度を示しました。論文では、ラスタ + ResNet のベースラインに対し、VectorNet が DE (Displacement Error, 予測位置と正解位置の距離。小さいほど良い) を改善しつつモデルのパラメータ数と計算量 (FLOPs) を大幅に削減したと報告しています (ラスタ方式は大きな画像と重い CNN を要するのに対し、ポリラインは疎で軽い)。これは「地図をどう表現するか」が性能と効率の両方を左右することを示します。
- ゼロ埋めプレースホルダの含意 (一般的観察): 入力をゼロで埋めると、モデルはそのビューを「常に情報ゼロ」として学習し、対応する重みをほぼ0に最適化してそのビューを無視するようになります。後で本物の地図を入れても、いったんゼロで学習した重みは地図ビューに注意を向けないため、再学習 (fine-tuning) が必要になります。これは特定の論文の結果というより、ニューラルネットが「無情報・無分散な入力には反応しなくなる」(勾配が流れず重みが育たない) という一般的性質から導かれる注意点です。アブレーション的に言えば「地図ビューを抜く (= ゼロ埋め) と、地図入力の効果分だけ性能上限が下がる」ことになり、ChauffeurNet 等が示した地図の寄与分がそのまま失われます。

## メリット・トレードオフ・限界

メリット

- 地図レンダラが未完成でも、固定ビュー数前提のモデル・融合・位置埋め込みを変えずに学習やテストを進められる。開発を地図機能と切り離して並行できる。
- 差し替え箇所が1関数に局所化されており、将来の作業範囲が明確 (単一責任の原則)。
- 参照テンソルから形・dtype・デバイスを自動継承するので、解像度や GPU/CPU を変えても壊れない。

トレードオフ・限界

- ゼロは「情報が無い」だけでなく「真っ黒という偽の入力」でもある。モデルが地図ビューを無視するよう学習すると、後で本物を入れても活かせず、再学習が要る。最悪、ゼロ正規化の影響で BatchNorm 等の統計を歪める副作用もありうる。
- 地図由来の道路構造・経路情報が一切使えないので、純粋なカメラ模倣に近い性能上限になる。
- 追加ビューの枠ぶんだけ計算とメモリを消費するのに、現状その情報量はゼロ (無駄な計算が乗る)。8ビュー中1ビューがゼロなら、融合層の計算の約1/8が空回り。
- レンダラ実装時に「ラスタかベクトルか」「どの座標系・解像度か」「自車中心か地図絶対座標か」を決め直す必要があり、プレースホルダはその設計判断を先送りしているだけ。

## 発展トピック・研究の最前線

- オンラインマッピング (online HD map): 事前作成の HD マップに頼らず、車載カメラ・LiDAR からその場で車線地図を生成する研究 (HDMapNet, MapTR (Liao et al., ICLR 2023), MapTRv2 など) が活発です。これが実用化すれば「外部の地図レンダラ」自体が不要になり、地図はモデル内部で生成されるので、プレースホルダの設計前提が変わります。
- BEV 統合: ST-P3 や UniAD (Hu et al., CVPR 2023) のように、地図・知覚・予測・計画を共通の BEV 空間で end-to-end に扱う流れが主流化しています。地図を「別ビューとして外挿入力する」より、BEV 内部表現に溶かし込む方が筋が良いとされつつあります ([End-to-End運転パラダイム](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1066_end-to-end-driving-paradigm))。
- ベクトル地図のトークン化: 近年は地図ポリラインを Transformer のトークンとして扱い、軌道予測 (例: Wayformer, MTR = Motion TRansformer) に直接食わせる方式が高精度を出しています。ラスタ画像ビューという枠組み自体が見直されている段階で、プレースホルダのゼロ画像が将来「ゼロトークン」になる可能性もあります。
