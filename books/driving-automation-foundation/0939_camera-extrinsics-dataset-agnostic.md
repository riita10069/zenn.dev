# カメラ外部パラメータとデータセット非依存設計

## ひとことで言うと
車によってカメラの台数も取り付け位置も違います。「カメラがどこにどう付いているか」を表す数値(外部パラメータ)をモデルに一緒に渡してあげれば、1つのモデルで台数や配置の違う複数のデータセット・複数の車を扱えるようになります。これが「データセット非依存(dataset-agnostic)設計」です。カメラの配置をモデルに暗記させるのではなく、その都度「今回のカメラはこう付いている」と数値で教える、という発想です。

## 直感的な理解
普通にモデルを作ると、「カメラはちょうど6台で、前カメラはこの位置、左カメラはこの位置」という前提を暗黙に学んでしまいます。すると別の車(たとえばカメラ9台で配置も違う)に持っていったとたんに使えません。データセットごとに別モデルを作るのは無駄ですし、せっかく集めた複数データセットを混ぜて学習することもできなくなります。

人間の感覚で言えば、「いつも同じ座席から同じ角度で外を見ている」前提で運転を覚えた人が、急にミラーの位置も座席の高さも違う車に乗ると戸惑うのに似ています。これを避けるには、「今日の車はミラーがここ、視点がこの高さ」という情報を毎回明示的に与えればよい。カメラパラメータをモデル入力として渡すのは、まさにこの「視点の付き方を毎回教える」ことに相当します。

## 基礎: 前提となる概念

カメラパラメータには2種類あります。古典的なコンピュータビジョン(Hartley & Zisserman の射影幾何が標準的な教科書)では次のように区別します。

### 内部パラメータ(intrinsics、内部行列 K)
そのカメラ単体の光学的性質です。「焦点距離(どれだけ拡大して写るか)」と「画像中心(光軸が画像のどこを通るか)」をまとめた 3×3 の行列で表します。

```
K = | fx   0  cx |
    |  0  fy  cy |
    |  0   0   1 |
```

`fx, fy` は焦点距離(ピクセル単位)、`cx, cy` は画像中心です。同じ物体でも、ズームの強いカメラ(fx 大)と広角カメラ(fx 小)では画像上の写り方が変わります。それを表す数値です。

### 外部パラメータ(extrinsics)
そのカメラが車のどこに、どの向きで付いているか。「並進(位置 x,y,z)」と「回転 R(向き)」をまとめたもので、ふつう 3×4 や 4×4 の行列 `[R | t]` で表します。前向きカメラと後ろ向きカメラでは、同じ世界の点でも見える方向が逆になります。それを表す数値です。

### 射影行列(projection matrix、3×4)
内部と外部をかけ合わせると射影行列 `P = K [R | t]` になります。これは「世界の中の 3D 点(X,Y,Z)を、このカメラの画像上の 2D 点(u,v)のどこに写すか」を計算するための行列です。同次座標で書くと:

```
[u·w]     [X]
[v·w] = P [Y]      (w で割って u, v を得る)
[ w ]     [Z]
                   [1]
```

射影行列があれば、複数カメラの画像を共通の世界座標系で結びつけられます。データセット非依存設計の核心は、この射影行列(camera_params)をモデルへの入力として外から渡すことです。

## 仕組みを詳しく

### 入力契約をそろえる
各データセットのパーサは、視点数を `V` として次の形をモデルに渡します。

```
visual_tiles  (V, 3, H, W)   V視点のRGB画像
camera_params (V, 3, 4)      V視点それぞれの射影行列
```

あるデータセットでは `V = 7`(6カメラ + 1マップ)、別のデータセットでも `V = 7`(高解像度前 + 6リングカメラ)というように、V が違っても同じ形の契約で渡せます。融合モジュール(複数視点の特徴を1つにまとめる部分)は、ハードコードした「6台」ではなく、渡ってきた V の数だけループして処理するので、台数が変わってもコード変更なしで動きます。これが「非依存」の実装上の肝です。

### 内部パラメータは前処理に合わせてスケールする
見落としがちな実装上の落とし穴があります。元の画像が 1920×1080 でも、バックボーンに入れる前にリサイズ・中央クロップして 256×256 などに縮めます。すると画像上のピクセル座標が変わるので、内部行列 K もそれに合わせて補正しないと、射影が現実とずれてしまいます。

たとえば横方向に `scale_x = 256/1920` 倍リサイズしたら、焦点距離成分 `K[0,0]=fx` と画像中心成分 `K[0,2]=cx` を `scale_x` 倍します。縦も同様に `scale_y` 倍。中央クロップで `cx_offset` だけ切り取ったら `cx -= cx_offset`。一方 `Normalize`(画素値の正規化)や `ToTensor` のようにピクセル座標を動かさない処理は K に影響しないので無視します。

before/after で言うと、補正前の K のままだと「世界の点が画像のどこに写るか」が縮小前の座標で計算されてしまい、補正後は縮小後の 256×256 上の正しい位置になります。前処理パイプラインを1ステップずつ調べ、Resize と CenterCrop の幾何的効果だけを K に反映する、という丁寧な実装が必要です。これを忘れると、値はそれっぽく出るのに射影が静かにずれる、デバッグしにくいバグになります。

### 外部パラメータが無い/暫定のときの扱い
外部パラメータ(実キャリブレーション)が手元に無いデータセットでは、単位行列ベースのプレースホルダ `(V, 3, 4)` を返しておく方式が取られます。これは連結融合や注意ベース融合では問題ありません。これらの融合は射影を使わず、視点ごとの特徴をそのまま連結したり、注意機構(attention、特徴同士の関連度に応じて重み付けして混ぜる仕組み)で混ぜたりするだけだからです。

一方 BEV 融合(各カメラ特徴を共通の俯瞰グリッドに投影してから混ぜる方式)では、正しい外部パラメータが無いと「どのカメラ画素が俯瞰グリッドのどのマスに対応するか」が計算できず、意味のある投影になりません。だから BEV を使うなら、キャリブレーションファイル(extrinsic 行列を記した YAML 等)から実外部パラメータを読む必要があります。

### 融合方式と幾何の依存度
```
融合方式        射影行列(camera_params)の要否
concat(連結)        不要(特徴を並べるだけ)
cross-attention      不要だが、位置エンコーディングに使うと有利
BEV(俯瞰投影)      必須(幾何で投影先を決める)
```

この対応を理解しておくと、「どの融合を選ぶか」と「どのデータセットがキャリブレーション付きか」が連動して設計判断になることが見えてきます。

## 手法の系譜と主要論文

### LSS: Lift, Splat, Shoot(Philion & Fidler, ECCV 2020, arXiv:2008.05711)
提案内容: 各カメラ画像の各ピクセルに「奥行きの確率分布」を予測させ(Lift)、内部・外部パラメータを使って 3D 点群として共通の俯瞰グリッドに撒く(Splat)。動機: 任意のカメラ台数・配置を、キャリブレーションを使って1つの BEV 表現に統合したかった。新規性: カメラ構成に依存しない BEV 認識の基礎を作った点。トレードオフ: 奥行き推定が不確実だと投影がぼやける。この「外部パラメータで複数カメラを共通空間に統合する」考え方が、外部パラメータ入力と BEV 融合の理論的土台です。

### DETR3D(Wang et al., CoRL 2021, arXiv:2110.06922)
提案内容: 3D の物体クエリを射影行列で各カメラ画像に投影し、対応する画像特徴を引いてくる。動機: 明示的に密な BEV を作らず、疎なクエリで効率的に多カメラ 3D 検出をしたかった。新規性: 射影を「特徴の引き先を決める鍵」として使う発想。LSS のような密な投影とは別系統の流れです。

### BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)
提案内容: 学習可能な BEV クエリが、各カメラの射影行列を使って画像特徴を空間的注意(spatial cross-attention)で参照し、時間方向にも履歴 BEV を参照する(temporal self-attention)。動機: カメラ配置の違いを射影で吸収しつつ、注意機構で柔軟に特徴を集めたかった。新規性: 空間 + 時間の両方向で注意を使う BEV 構築。トレードオフ: 射影とアテンションの計算が重い。射影行列を入力として明示的に使う点が、camera_params 入力と同じ思想です。

### PETR(Liu et al., ECCV 2022, arXiv:2203.05625)
提案内容: カメラパラメータから作った 3D 位置エンコーディング(3D position embedding)を画像特徴に足し込み、Transformer が暗黙に幾何を扱えるようにする。動機: 明示的な BEV グリッドを作らずに、カメラの 3D 幾何をモデルに教えたかった。新規性: 投影演算を陽に書かず、位置エンコーディングだけで幾何を注入する簡潔さ。トレードオフ: 位置エンコーディングの設計に性能が依存する。これは「外部パラメータをどう渡すか」の別解として重要で、後続の PETRv2・StreamPETR へ発展しました。

これらをまとめると、系譜は「(1)LSS が幾何投影で BEV を作る → (2)DETR3D/PETR が射影をクエリや位置符号として暗黙に使う → (3)BEVFormer が時空間注意で統合する」と進化してきました。共通するのは「カメラパラメータを入力として明示的に渡し、構成に依存しない認識を実現する」という思想です。

## 論文の実験結果(定量データ)

主要評価は nuScenes(6カメラのリング構成を持つ大規模運転データセット)上の 3D 物体検出で行われ、指標は NDS(nuScenes Detection Score、検出精度・位置・速度・属性誤差を総合した 0〜1 の指標。高いほど良い)と mAP(mean Average Precision、検出の平均適合率。高いほど良い)です。

- LSS は BEV セグメンテーションで、カメラ構成に依存しない統合の有効性を示しました。任意のカメラサブセットを抜いても破綻せず推論できる頑健性が報告されており、これが「構成非依存」の実証になっています。
- BEVFormer は nuScenes 3D 検出で NDS およそ 0.56(報告値)を達成し、当時のカメラのみ手法で最先端を更新しました。アブレーション(ある要素を抜く実験)では、時間方向の注意(過去 BEV の参照)を外すと速度推定の誤差が顕著に悪化することが示され、時間情報が動きのある物体の把握に効くことが定量化されています。
- PETR は nuScenes で NDS およそ 0.44(単フレーム設定の報告値)を達成し、明示的な BEV を作らずとも位置エンコーディングだけで競争力ある精度が出ることを示しました。後継の PETRv2 で時間統合とさらなる精度向上を報告しています。
- DETR3D は疎なクエリ方式で、密な BEV 構築より計算を抑えつつ妥当な mAP を達成し、効率と精度のトレードオフ上で別の動作点を示しました。

これらの数値が示す実用的な含意は、「射影行列を正しく与えれば、台数や配置の違いを吸収して頑健に動く」ことです。逆にアブレーションで外部パラメータを誤らせると、投影先がずれて精度が大きく落ちる(特に BEV 系)ことも各論文の頑健性分析で示唆されています。

## メリット・トレードオフ・限界

メリット
- 1つのモデルで台数・配置の違う複数データセット/複数車種を扱える。
- 新データセット追加はパーサ1本で済み、モデル本体を触らない。
- 複数データセットを混ぜて学習でき、データ量と多様性を増やせる。
- 射影行列を渡すので、BEV のような幾何ベースの融合に発展できる。

トレードオフと限界
- 正しい外部パラメータ(キャリブレーション)が無いと BEV 融合は意味をなさない。キャリブレーションの精度がそのまま投影の精度になる。
- 内部行列を前処理(リサイズ・クロップ)に合わせて補正し忘れると射影がずれ、値はもっともらしいのに間違うという発見しにくいバグになる。
- 各データセットでキャリブレーション形式・座標系の規約(右手系か左手系か、軸の向き、RDF/FLU など)が違うため、パーサ側の読み込み実装コストがかかる。座標系の取り違えは典型的な落とし穴。
- 視点数 V を可変にすると、バッチ内で V が揃わない場合(データセット混在)にパディングやマスクの追加処理が要る。
- キャリブレーションは経年やぶつけで実際とずれることがあり(キャリブレーションドリフト)、運用時のずれに弱い。オンラインキャリブレーション補正が研究課題。

## 発展トピック・研究の最前線

- 暗黙的幾何 vs 明示的幾何: PETR 系のように位置エンコーディングで暗黙に幾何を扱う流れと、LSS/BEVFormer のように明示的に投影する流れがあり、精度・効率・頑健性のトレードオフで選ばれます。
- キャリブレーションフリー/自己キャリブレーション: 外部パラメータを学習で推定したり、補正項を学習したりして、不正確なキャリブレーションに頑健にする研究。運用時のドリフトに効きます。
- センサ汎化: カメラだけでなく LiDAR・レーダも同じ「視点」枠組みで統合し、センサ構成が違う車両群を1モデルで扱う方向(BEVFusion 系)。
- スケール不変な前処理: リサイズに伴う K の補正を自動化し、解像度を動的に変えても破綻しないパイプライン設計。遠方の小物体の検出と計算コストの両立が論点です。
