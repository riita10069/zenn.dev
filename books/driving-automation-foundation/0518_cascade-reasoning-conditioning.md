# Cascade 条件付け (reasoning latent → planner)

## ひとことで言うと

「場面の理由をじっくり推論する部分 (熟考系)」が作った潜在ベクトルを、「素早く軌跡を出す部分 (プランナー)」へ流し込み、しかも両者の間で勾配 (学習の信号) を遮断しない設計です。勾配を止める命令 (`torch.no_grad` のようなもの) をどこにも入れないことで、推論という補助タスクの学習信号が共有の特徴抽出部まで遡り、共有表現を「場面の意味をよく捉えた」状態へ整える効果を狙います。つまり推論部品を「ただの観察者」ではなく「共有表現を能動的に磨く参加者」にする設計思想です。

## 直感的な理解

2つの仕事を1人でこなす人を想像してください。仕事 A は「とっさにハンドルを切る (軌跡を出す)」、仕事 B は「この場面で一番気をつけるべき要因は何かを言葉にする (理由を推論する)」です。

ここで2通りの教え方があります。1つ目は、仕事 B の練習を仕事 A とは完全に切り離して行う方法。仕事 B でいくら反省しても、その学びが仕事 A の感覚 (=共有された脳の使い方) には一切還元されません。仕事 B は脇に置かれた「観察者」にとどまります。2つ目は、仕事 B の反省を仕事 A の根っこにある共通の感覚にまでフィードバックする方法。仕事 B で「あ、ここは歩行者が支配的だった」と気づけば、その気づきが場面を見る目そのものを鍛え、結果として仕事 A の判断も良くなります。

機械学習でこの「共通の感覚」に当たるのが、ネットワークの下層にある共有表現 (バックボーンや特徴融合部) です。「反省 (補助タスクの学習信号) を共通の感覚まで届ける」のが cascade 条件付けで、これは勾配を遮断しないことで実現されます。逆に、補助タスクの勾配を `no_grad` で切ると、仕事 B は観察者のままになります。

cascade (カスケード) という語は「滝のように上流から下流へ段階的に流れる」イメージから来ています。ここでは推論の出力が滝のようにプランナーへ流れ込み、同時にその逆向き (誤差の逆伝播) も上流まで流れ続ける、という二重の流れを指します。

## 基礎: 前提となる概念

- System 1 / System 2 — 認知心理学者 Daniel Kahneman が一般向けに整理した二重過程理論。System 1 は速くて直感的・自動的な思考、System 2 は遅くて熟考的・論理的な思考です。自動運転 ML では「即座に軌跡を出す経路 (System 1 的)」と「場面の意味をじっくり推論する経路 (System 2 的)」を分ける比喩として頻繁に借用されます。
- 潜在ベクトル (latent vector) — ネットワークの途中で計算される連続値の特徴ベクトル。人間が直接読める意味はないが、ネットワークにとっての「場面の要約」です。
- 勾配 (gradient) と逆伝播 (backpropagation) — 学習では、出力の誤差 (損失) を入力側へ伝えて、各パラメータをどちらへ動かせば誤差が減るかを計算します。この計算が逆伝播で、各パラメータに付く「動かすべき方向と量」が勾配です。勾配が届かないパラメータは学習されません。
- `torch.no_grad` / detach — 勾配計算をその地点で止める操作。順方向の値は流れるが、逆伝播の信号はそこで遮断され、上流のパラメータは更新されません。
- マルチタスク学習 (multitask learning) — 1つのネットワークに複数のタスクを同時に学ばせ、共有部分の表現を豊かにする枠組み。
- 共有表現 (shared representation) — 複数のタスクが共通して使う下層の特徴。ここを良くすると全タスクが恩恵を受けます。
- 負の転移 (negative transfer) — 補助タスクが主タスクと相性が悪いと、共有表現がかえって悪化し、主タスク性能が下がる現象。マルチタスク学習の最大の落とし穴です。

## 仕組みを詳しく

中心となるのは、共有表現から作られた文脈ベクトルを受け取り、推論の潜在ベクトルを返す「接続口 (cascade hook)」です。典型的にはこういう形です。

```
def reason(context):                  # context: [B, embed_dim] 共有表現由来の文脈
    return reasoning_trunk(context)   # 小さな MLP。no_grad を使わない → 完全微分可能
```

`reasoning_trunk` は小さな多層パーセプトロン (MLP、全結合層を数枚重ねたもの) で、文脈ベクトルを推論の潜在ベクトルへ変換します。データの流れを B=4 で図示します。

```
[共有 backbone/fusion]
        |
        v
   context [4, 256]
        |
        +--> reason() --> reasoning_latent [4, 256] --+
        |                                              |
        |        (planner の自車埋め込み ego_hidden に  )
        |        ( 足し込む or 結合して条件づける       )
        v                                              v
   planner  <------------------------------------------+
        |
        v
   trajectory [4, T, 2]   (T ステップ分の (x,y) 軌跡)
```

設計上のポイントが2つあります。

第一に、推論の潜在ベクトルの次元を共有表現の次元 (例 `embed_dim=256`) とわざと一致させること。次元を揃えておくと、余計な射影層 (次元を変換する線形層) を挟まずに、プランナーの自車埋め込みへそのまま足したり結合したりできます。これは「条件づけを最小コストで挿入する」ための実務的な工夫です。

第二に、そして最も重要なのは「勾配を遮断する命令をどこにも置かない」という制約です。なぜこれが効くのかを勾配の流れで説明します。推論タスクの損失 (たとえば因果分類の交差エントロピー) を逆伝播すると、勾配は次の順で遡ります。

```
損失 → 分類ヘッド → reasoning_trunk → context → 共有 backbone/fusion
```

途中に `no_grad` があると、その地点で勾配が 0 に切られ、それより上流 (共有 backbone) は一切更新されません。遮断しなければ、勾配は最後まで遡り、共有 backbone のパラメータも「推論タスクがやりやすい特徴を出す」方向へ更新されます。これがマルチタスク学習の核心的なメカニズムで、補助タスクが共有表現に意味的な構造を注入するのです。

before / after を言葉にすると、

- before (no_grad で遮断): 推論ヘッドは共有表現を「読むだけ」。共有表現は主タスク (軌跡模倣) の信号だけで形作られる。推論ヘッドは観察者。
- after (遮断しない cascade): 推論ヘッドの誤差が共有表現まで届き、「場面の支配要因が判別しやすい」方向へ表現を整える。推論ヘッドは表現の共同設計者。

数値の直感も添えます。仮に主タスク損失の勾配が共有 backbone のあるパラメータに +0.8、推論タスク損失の勾配が同じパラメータに +0.2 付くとします。cascade では合計 +1.0 が反映され、表現は両タスクが満たされる方向へ動きます。もし両者が逆向き (主タスク +0.8、補助タスク −1.0) だと合計 −0.2 となり、補助タスクが主タスクの学習を打ち消してしまう。これが負の転移の正体で、後述の勾配手術 (PCGrad) はこの衝突成分を取り除こうとします。

この性質は、勾配が末端 (入力 context) まで届くかをテストで確認するのが定石です。具体的には context に `requires_grad=True` を付けて損失を逆伝播し、`context.grad` が None でないこと (勾配が末端まで到達したこと)、および全パラメータに有限の勾配が付くことを検証します。

## 手法の系譜と主要論文

- Kahneman, "Thinking, Fast and Slow" (2011)。System 1 / System 2 という二重過程の枠組みの一般向け出典。自動運転 ML では「速い反応系」と「遅い熟考系」を分ける命名の元ネタとして広く引用されます。

- Caruana, "Multitask Learning" (Machine Learning, 1997)。補助タスクの勾配が共有表現を整えるというマルチタスク学習の古典的根拠論文です。動機は「関連タスクを同時に学ぶと、各タスク単独より良い表現が得られる」こと。cascade で勾配を遮断しないのは、まさにこの「補助タスクで共有表現を磨く」効果を得るためです。トレードオフとして、補助タスクが主タスクと衝突 (負の転移) すると共有表現が悪化しうる点が当初から指摘されています。

- 対比として、知識蒸留や半教師あり学習では「教師側を意図的に固定する」設計が一般的です。Tarvainen and Valpola, "Mean teachers are better role models" (NeurIPS 2017, arXiv:1703.01780) は、教師モデル側へ勾配を流さず (重みを生徒の指数移動平均で更新)、安定した教師信号を作りました。これは安定性のために勾配を切る発想で、cascade とは逆方向の選択です。両者のトレードオフ — 安定性 (勾配を切る) vs 表現整形 (勾配を通す) — を対比すると、cascade の狙いがはっきりします。cascade が成立するのは、推論ヘッドが軽量で再学習可能だからです。巨大で凍結したい推論器 (大規模 VLM など) には no_grad 連結のほうが現実的です。

- Yu et al., "Gradient Surgery for Multi-Task Learning (PCGrad)" (NeurIPS 2020, arXiv:2001.06782)。複数タスクの勾配が衝突する (内積が負になる) とき、片方の勾配をもう片方の直交平面へ射影して衝突成分を除去する手法。cascade で負の転移が起きたときの直接的な処方箋であり、「勾配を通すが、衝突は調整する」という cascade の発展形を与えます。

- Hu et al., UniAD (CVPR 2023, arXiv:2212.10156) は、知覚・予測・計画を1つのネットワークで end-to-end に学習し、上流タスクの勾配が計画の質に向けて全体を整える設計を実証しました。cascade 条件付けは、この「補助的理解を主タスクへ勾配で接続する」思想を、軽量な推論ヘッド1つに絞った最小形と位置づけられます。

## 論文の実験結果(定量データ)

cascade 条件付けの効果を読むには、マルチタスク学習と統合運転の実証が手がかりになります。

- マルチタスク学習一般では、補助タスクを足すと主タスクの汎化が改善するという結果が多数報告されてきました。ただし効果はタスクの関連性に依存し、無関係・対立するタスクを足すと負の転移で主タスク性能が下がります。Yu et al. (PCGrad) は、複数のマルチタスクベンチマーク (MultiMNIST、CityScapes セグメンテーション + 深度、強化学習の Meta-World 等) で、衝突成分を除去すると素朴な勾配合算より一貫して成績が上がることを示しました。たとえば CityScapes の同時学習で、単純合算に比べセグメンテーション・深度の双方の指標が改善したと報告されています。

- 統合運転では UniAD が nuScenes で、上流タスクの表現を計画へ流す設計により、計画の L2 誤差 (予測軌跡と実軌跡のユークリッド距離、m、小さいほど良い) と衝突率 (collision rate、%、小さいほど安全) を同時に改善したと報告しています。アブレーションで「上流表現を計画へ渡す経路を切る」と計画性能が劣化する、という結果は、cascade が前提とする「勾配で上流と下流を結ぶことの価値」を裏づけます。

- 数値の意味の補足。L2 が 0.1 m 縮むとは、3秒先の予測位置が人間の実際の軌跡へ平均 10 cm 近づくこと。車線内の余裕がしばしば数十 cm 規模であることを考えると、この差は安全余裕に直結します。衝突率が 1 % から 0.5 % へ下がれば、危険な予測軌跡の頻度が半減することを意味します。

- 定量データの注意点として、cascade を「勾配を流す版」と「no_grad 版」で直接比較した統制実験は、補助タスクの種類・重み・データ品質に強く依存するため、汎用的な「常に何 % 改善」という数値は存在しません。効果はケースバイケースで、特に補助ラベルがノイジーな場合は逆効果になりうる、というのが研究上の正直な現状です。

## メリット・トレードオフ・限界

メリット
- 補助タスクの学習信号が共有表現まで届き、表現を意味的に整えられる (マルチタスク学習の恩恵)。
- 推論の潜在ベクトルをプランナーへ足すだけで条件づけでき、次元を揃えておけば追加の射影層が不要。
- 推論部品が観察者でなく、表現整形へ能動的に寄与する。

トレードオフ・限界
- 補助損失が主タスク (軌跡模倣) と衝突すると、負の転移で共有表現が悪化し、メイン性能を下げうる。補助損失を低い重みで運用するのが前提になりやすい。
- 勾配を遮断しないぶん、補助タスクのノイズ (たとえば自動生成されたノイジーな疑似ラベルの誤り) が共有表現にまで波及するリスクがある。ノイズ対策 (label smoothing 等) との併用が要る。
- 推論潜在をプランナーへ実際に注入してメイン軌跡を変える配線は、設計と検証が分かれており、「勾配が流れること」と「最終的に軌跡が良くなること」は別問題。
- 学習が一段重くなる (共有部分まで勾配を流すコスト、メモリと計算量の増加)。
- 未解決課題: 補助タスクと主タスクの勾配衝突をどう自動調整するか、確信度に応じて条件づけ量を動的に変えるべきか、などは活発な研究領域です。

## 発展トピック・研究の最前線

- 勾配衝突の緩和: PCGrad (Yu et al., NeurIPS 2020) や GradNorm (Chen et al., ICML 2018, arXiv:1711.02257) は、タスク間で勾配が打ち消し合う問題を、勾配の射影や損失重みの動的調整で和らげます。cascade で負の転移が起きるときの直接的な処方箋です。
- 凍結 vs 微調整の中間: 大規模推論器を完全凍結するのでも全微調整するのでもなく、LoRA (Hu et al. ICLR 2022, arXiv:2106.09685) やアダプタで一部だけ勾配を流す手法が VLM 連結で使われています。cascade の「勾配を通す」と蒸留の「凍結」の中間に位置します。
- 条件づけ機構の高度化: 単純な加算でなく、FiLM (Perez et al., AAAI 2018, arXiv:1709.07871) のように潜在ベクトルで特徴をスケール・シフトする条件づけや、cross-attention で文脈を注入する設計が、より表現力の高い条件づけとして研究されています。
- 確信度ゲート付き条件付け: `ego_hidden + confidence * latent` のように、推論の確信度で注入量を動的に変える設計。確信度が低い場面では補助情報を弱め、ノイズの波及を抑える狙いで、安全機構と結びついた研究方向です。
