# VectorNet シーンエンコーダ

## ひとことで言うと
VectorNet は、HD 地図 (高精度地図) と車・歩行者の移動軌跡を、画像 (ピクセルの格子) に描かず「折れ線 (ポリライン) を細かく分割した小さなベクトルの集まり」として扱い、グラフニューラルネットワーク (GNN) で符号化する手法です。地図の滑らかな形を解像度で潰さずに、しかも空白を計算せずに済むので、軽くて正確に「シーンの特徴」を作れます。この特徴が後段の軌道予測・計画モデルの入力になります。軌道予測の入力表現を「画像からベクトルへ」と転換させた起点的な研究です。

## 直感的な理解
紙の地図を思い浮かべてください。地図そのものは「この交差点からこの通りへ、こう曲がって繋がる」という線の集まりです。これをわざわざ写真に撮ってピクセルにしてしまうと、細い路地はぼやけ、どの道がどの道に繋がっているのかも一目では分からなくなります。せっかく線として持っている情報を、画像化で台無しにしているわけです。

VectorNet の発想は単純で力強いです。「地図はもともと線 (ベクトル) なのだから、線のまま扱えばいい」。線を短い区切りに分け、それぞれを「ここからここへ、こういう種類の線」というベクトルにして、線どうし・線とエージェントどうしの関係をグラフで結ぶ。これだけで、画像化で失われていた幾何の精度と接続構造の両方を保てます。「地図を写真ではなく路線図として扱う」と言い換えると分かりやすいです。

## 基礎: 前提となる概念
用語を噛み砕きます。

HD 地図 (High-Definition map): 自動運転用の高精細地図で、車線中心線・車線境界・横断歩道・停止線などが「座標の点列」で記録されています。普通のカーナビ地図よりはるかに細かく、センチメートル単位の幾何を持ちます。

ラスタ化 (rasterization): 地図や軌跡を上空から見た 1 枚の画像 (鳥瞰図) に描く方法。車線を青、横断歩道を緑、というように色分けしたピクセル画像を作り、CNN (畳み込みニューラルネットワーク。画像を小さな窓で走査して特徴を取る仕組み) に通します。VectorNet 以前の主流でした。詳しくは [ラスタライズドマップエンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0425_rasterized-map-encoder)。

ベクトル化表現 (vectorized representation): 地図や軌跡を、座標を持つ線分・点の集合として扱う方法。ピクセルではなく数値の座標そのものを使います。

ポリライン (polyline): 点を順に結んだ折れ線。1 本の車線中心線も、1 台の車の過去軌跡も、点列なのでポリラインで表せます。

グラフニューラルネットワーク (GNN): 「ノード (頂点) とエッジ (つながり) から成るグラフ」の上で、各ノードが隣のノードの情報を集めて自分の特徴を更新する仕組み。画像の CNN が「隣のピクセル」を見るのに対し、GNN は「エッジで繋がった隣ノード」を見ます。

self-attention: 各要素が他の全要素を見て、関係の強さ (重み) に応じて情報を取り込むアテンション演算。Transformer の中核です。要素数 N に対し O(N²) の計算量がかかります。

順序不変性 (permutation invariance): 入力の並び順を変えても出力が変わらない性質。車線を構成する点に「正しい順番」はあるが、ポリラインを集約するとき本数や並びに依存しない演算 (max-pooling など) を使うと、可変長・順不同のデータを安定して扱えます。

ラスタ化の弱点を整理すると次の通りです。第一に情報の劣化。連続的な車線の曲線をピクセルに焼くと、解像度の分だけ角張り、遠くの細い車線や微妙な角度が失われます。第二に計算の無駄。画像の大半は「何もない空白ピクセル」なのに CNN は一様に畳み込むため計算が無駄に重く、広範囲を見るには画像を巨大化させ層を深くする必要があります。第三に構造の喪失。「この車線とあの車線が繋がっている」「この軌跡はこの車のもの」というグラフ構造がピクセルからは読み取りにくい。VectorNet はこの 3 つを同時に解きます。

## 仕組みを詳しく

### 1. すべてをポリラインのベクトル列に
地図要素 (車線中心線、境界線、横断歩道など) と各エージェントの軌跡を、すべてポリラインとして表し、各ポリラインを連続する小さな線分 (ベクトル) に分割します。1 本のベクトルを次の特徴で表します。

```
v = [ 始点(xs, ys), 終点(xe, ye), 属性a (車線/横断歩道/車の軌跡など), ポリラインID j ]
```

例: ある車線が点列 P0→P1→P2→P3 なら、ベクトル (P0→P1), (P1→P2), (P2→P3) の 3 本に分割し、3 本とも同じポリライン ID を付けます。軌跡なら時刻ごとの位置を順に結びます。属性には「これは車線か横断歩道か軌跡か」「時刻」「信号状態」などのメタ情報を入れます。座標は通常、予測対象エージェントを原点・進行方向を基準にした相対座標に正規化します (回転・並進不変性のため。地図上のどこにいても同じシーン形状なら同じ特徴が出るように揃える)。

### 2. ローカルグラフ (polyline subgraph): 1 本の中をまとめる
同じポリラインに属するベクトルどうしを、小さなグラフとして処理します。1 ステップは次の通りです。

1. 各ベクトルを MLP (全結合層、全ベクトルで重みを共有) に通して特徴に変換する。
2. 同じポリライン内の全ベクトル特徴を max-pooling (要素ごとの最大値) で集約し、「このポリラインの代表」を作る。
3. 各ベクトル特徴に、その代表を連結 (concat) して情報を共有する。

これを数回 (論文では 3 層) 繰り返すと、各ポリラインが 1 本のノード特徴 `[D]` (例 D=128) に集約されます。max-pooling を使うのは、ベクトルの並び順や本数に依存しない (順序不変・可変長対応) 集約をしたいからです。N 本のポリラインがあれば `[N, D]` になります。これがポリラインサブグラフ (subgraph) です。「線分の集まりを 1 本の線の代表ベクトルにまとめる」段階だと考えてください。

### 3. グローバルグラフ (global interaction graph): ポリライン同士の関係
次に、N 本のポリライン特徴 `[N, D]` を「完全連結グラフ」とみなし self-attention にかけます。これで「この車線とあの車線」「この車とその前方の車線」といった要素間の相互作用を捉えます。

```
ポリライン特徴 [N, D]  → self-attention → 相互作用込みの特徴 [N, D]
```

予測したいエージェント (注目する車) に対応するノード特徴を取り出せば、それが「そのエージェント周辺のシーンを要約した特徴ベクトル」になり、後段のデコーダ (軌道予測ヘッド、例えば MLP で複数本の未来軌道を回帰) へ渡せます。

### 4. 補助タスク: マスク復元 (node completion)
VectorNet は学習を助けるため、入力ノードの一部を隠して (mask) 復元させる補助タスクを加えています。具体的には、あるポリラインノードの特徴を隠し、残りのシーンからそのノードの幾何属性を当てさせます。「文章の一部を隠して当てさせる」言語モデル (BERT) の学習に似た自己教師あり的な工夫で、シーンの構造理解 (見えない部分を周辺から推測する力) を促します。これがあると、限られた予測ラベルだけでは学べない地図構造の事前知識をモデルが獲得できます。

### 形状の流れ (具体例)
```
生データ: 車線/軌跡のポリライン群
  └ 各ポリラインを小ベクトルに分割 → ポリライン j に L_j 本のベクトル
ポリラインサブグラフ (MLP + maxpool ×3):
  ポリライン j → ノード特徴 [128]
  全 N 本 → [N, 128]
グローバルグラフ (self-attention):
  [N, 128] → [N, 128]
注目エージェントのノードを取得 → [128] → 軌道デコーダへ
```

ラスタ画像と違い空白ピクセルを処理しないので計算が軽く、車線の形も座標のまま正確に保たれます。

## 手法の系譜と主要論文
VectorNet は「シーン表現をラスタからベクトルへ」転換させた分水嶺です。

- ラスタ + CNN 時代 (〜2019): IntentNet (Casas et al., CoRL 2018)、ChauffeurNet (Bansal et al., RSS 2019)、MultiPath (Chai et al., CoRL 2019) など。地図を鳥瞰画像に焼いて CNN で処理する手法が主流だった。
- Gao et al., "VectorNet" (CVPR 2020, Waymo): 本ノートの主題。地図と動きをベクトル化し、ポリラインサブグラフ + グローバル interaction グラフの階層 GNN で符号化。ラスタ化の情報損失・計算非効率・構造喪失を一挙に解消。
- Liang et al., "LaneGCN" (ECCV 2020): 車線を有向グラフとして明示的に扱い、車線トポロジに沿った畳み込みを行う。VectorNet と同時期の「ベクトル/グラフでシーンを表す」流れの双璧。VectorNet が汎用ポリラインなのに対し、LaneGCN は前後左右の接続を明示する。[LaneGCN（車線グラフ畳み込みによる動き予測）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0444_lanegcn-planning)
- Zhao et al., "TNT" (CoRL 2020) / Gu et al., "DenseTNT" (ICCV 2021): VectorNet のシーンエンコーダを土台に、ゴール駆動で軌道を予測する系統へ発展。
- Liu et al., "mmTransformer" (CVPR 2021) / Shi et al., "MTR" (NeurIPS 2022, [MTR (Motion Transformer)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0194_mtr-motion-transformer)): ベクトル化シーン表現を入力に、Transformer デコーダで多峰予測する後継。VectorNet の「ベクトルで符号化する」考えが標準的な前処理として定着した。
- Liao et al., "MapTR" (ICLR 2023, [MapTR (ベクトルHDマップ)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0760_maptr-vectorized-hd-map)): 地図そのものをベクトル化して end-to-end に推定する方向へ展開。VectorNet が確立した「ベクトル化地図表現」を、生成側でも採用した系譜。

## 論文の実験結果(定量データ)
評価は Argoverse Motion Forecasting (米国の大規模運転データ、過去 2 秒から未来 3 秒を予測) と、Waymo の社内大規模データで行われました。指標は次の通りです。

- ADE (Average Displacement Error): 予測軌道と正解の、全時刻の位置誤差 (メートル) の平均。小さいほど良い。
- FDE (Final Displacement Error): 最終時刻の位置誤差。終点をどれだけ当てたか。
- DE@t: 特定時刻 t 秒での位置誤差。

論文の主要な報告は次の通りです。

- ラスタ + ResNet ベースのベースラインと比べ、VectorNet は同等以上の精度を、はるかに少ない計算量・パラメータで達成した。論文ではモデルのモデルサイズと計算量 (FLOPs) がラスタ系の約 1/10 以下に収まる一方で、予測精度 (DE@3s) は同等以上であることが示され、「ベクトル化は軽くて強い」ことを実証した。具体的には、同等精度のラスタ系 ResNet エンコーダの計算量と比べ大幅に小さい数字が報告されている。
- Argoverse の予測精度 (DE@3s など) で当時の上位に並ぶ結果を出し、ベクトル化アプローチの実用性を示した。
- アブレーションでは、補助タスク (node completion、隠したノードの復元) を加えると DE が改善することを報告。シーンの構造理解を促す自己教師あり信号が効いている。
- グローバル interaction グラフ (ポリライン間の self-attention) を外すと精度が落ちる。要素間の相互作用を明示的にモデル化することが効いている。
- 地図情報を入力から外すと精度が大きく落ち、HD 地図のベクトル特徴が予測に強く効いていることを確認した。

数値の意味を補足すると、FDE が数十センチ改善することは、3 秒後の他車の位置の読み違いがそれだけ減ることを意味し、自車の計画における安全マージンに直接効きます。また計算量が約 1 桁小さいことは、車載の限られた計算資源でリアルタイム動作させやすいという実用上の利点に直結します。予測モデルは知覚・計画と同じ計算予算を分け合うので、軽さは精度と同じくらい重要な評価軸です。

## メリット・トレードオフ・限界
メリット

- 地図の幾何 (車線の滑らかな形状) を解像度で潰さず正確に保てる。
- 空白を処理しないので計算が軽く、広範囲のシーンも扱いやすい。
- ポリライン ID やグラフ構造で「どの軌跡が誰のもの」「どの車線が繋がる」という関係を自然に表現できる。
- 多くの後続手法 (TNT, MTR など) の標準入力表現になり、研究エコシステムが厚い。

トレードオフ・限界

- 入力をベクトル化する前処理が必要。HD 地図のベクトル抽出やポリライン分割が要り、生のカメラ/LiDAR から直接は使えない。
- HD 地図への依存。地図が無い・古い地域では使いにくい (これが HD 地図不要の BEV 知覚や [MapTR (ベクトルHDマップ)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0760_maptr-vectorized-hd-map) が登場した動機の一つ)。
- ベクトル化の過程で、ラスタなら拾えた「見た目の質感」的な情報 (路面の状態、ペイントの摩耗など) は落ちる (用途次第で長所にも短所にもなる)。
- グローバルグラフの self-attention はポリライン数の 2 乗の計算量で、要素が非常に多い大規模シーンでは依然として重くなりうる。これが後の局所アテンションや query-centric 表現を生む動機になった。
- ポリライン分割の粒度 (1 本を何本のベクトルに割るか) や正規化基準が性能に効くハイパーパラメータとして残る。

## 発展トピック・研究の最前線
VectorNet の後、ベクトル化表現はさらに洗練されました。

- 階層化とスパース化: グローバル self-attention の 2 乗コストを避けるため、近傍だけに注意を絞る局所アテンション ([MTR (Motion Transformer)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0194_mtr-motion-transformer) の局所精緻化) や、k 近傍グラフでエッジを間引く手法が広がりました。
- Query-centric / agent-centric 表現: QCNet (Zhou et al., CVPR 2023) は各エージェント中心の相対座標でシーンを符号化し、座標系を揃えることで計算の再利用と高精度を両立しました。VectorNet の正規化を一般化した発想で、Argoverse 2 でも高い指標を出しています。
- HD 地図への依存からの脱却: HD 地図のメンテナンスコストが実運用の足かせになるため、カメラから地図要素をベクトルとして直接推定する [MapTR (ベクトルHDマップ)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0760_maptr-vectorized-hd-map) や、地図なしで走る研究が進んでいます。「ベクトル表現」は地図を作る側と使う側の双方で標準言語になりました。
- 予測と計画の統合: ベクトル化シーン特徴を入力に、周辺予測と自車計画を 1 つのネットワークで解く end-to-end 計画 ([ゴール条件付き計画（先に目標点を予測し、そこへ至る軌道を生成する2段構成）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0179_goal-conditioned-planning) など) が増えています。VectorNet 系のエンコーダは VAD や PARA-Drive など近年の end-to-end 運転モデルでも地図・エージェント符号化の土台として使われています。
