# VectorNet（ベクトル地図エンコード）

## ひとことで言うと
VectorNet(ベクトルネット)は、高精細地図(HDマップ)や周囲の車・人の動きを「画像(ラスタ)」として描くのではなく、「点をつないだ折れ線(ポリライン)」というベクトルの集合として表し、それをグラフ(点と線のつながり)としてニューラルネットに入れる設計です。地図を絵に描いて CNN に通す従来手法に代わる、幾何ベースのエンコード手法であり、その後の動作予測モデルの主流を形づくった起点です。

## 直感的な理解
カーナビの地図を思い浮かべてください。あの地図は本来「この道はここからここまで」「この交差点はこの形」という線と図形の情報です。それをわざわざ写真のように描き起こし、その写真を改めて画像認識で「ここに道がありますね」と読み取り直すのは、二度手間に見えます。最初から線として与えれば、読み取り直しの誤差も計算の無駄もないはずです。

VectorNet の発想はこの「最初から線として渡す」です。地図の車線も、横断歩道も、他車が通った軌跡も、すべて「線分をつないだ折れ線」として表し、線同士の関係を直接ニューラルネットに学ばせます。絵に描いて読み直す遠回りをやめる、というのが出発点です。地図情報はもともと座標値の集まりなのだから、座標値のまま扱うのが素直だ、という考え方です。

## 基礎: 前提となる概念
動作予測(motion forecasting / trajectory prediction)とは、「この先の数秒で周囲の車や歩行者がどう動くか」を予測するタスクです。自動運転では、自車がどう動くか決めるために、周囲がどう動くかを先読みする必要があります。例えば3秒先までの位置の系列を予測します。

HDマップ(High-Definition Map、高精細地図)は、車線中心線・車線境界・横断歩道・停止線・信号位置などを数センチ精度で持つ地図です。普通のカーナビ地図よりはるかに詳細で、各要素が幾何形状(点列)として記述されています。

ラスタ化(rasterization)とは、情報を画像(ピクセルの格子)として描くことです。「ラスタ」は格子状の画素配列を意味します。逆に「ベクトル」表現は、座標を持つ点や線として情報を保持します。地図を真上から見た1枚の画像として描けばラスタ、座標付きの線分集合として持てばベクトルです。

グラフ(graph)は、ノード(点、頂点)とエッジ(つながり、辺)からなる構造です。グラフニューラルネット(Graph Neural Network、GNN)は、このグラフ上で隣接するノード同士が情報を交換しながら各ノードの特徴を更新するニューラルネットです。地図のように「要素同士のつながり」が本質的な対象と相性が良いです。

MLP(Multi-Layer Perceptron、多層パーセプトロン)は、線形変換と非線形関数(ReLU など)を重ねた基本的なニューラルネットで、1つのベクトルを別のベクトルに変換します。

## 仕組みを詳しく

### ラスタ地図エンコードの何が問題か
従来の定番は地図と動きをラスタ化することでした。車線を青い線、横断歩道を白い帯、他車の軌跡を色付きの点列として、1枚のトップダウン(真上から見た)画像に描き、その画像を CNN で処理します。この方式には次の弱点があります。

情報損失。連続的な座標を画素グリッドに丸めるため、細かい位置精度が落ちます。例えば1ピクセル=0.5mの地図だと、それより細かい位置は表現できません。

計算の無駄。画像の大半は「何もない空きスペース(背景)」で、そこにも CNN は一様に計算を費やします。地図は本来スカスカ(疎)なのに、密な画像として扱う無駄が生じます。広い範囲を見ようとすると画像が大きくなり、計算量は面積(画素数)に比例して増えます。

受容野の制約。CNN が「遠く離れた2つの車線のつながり」を捉えるには、何層も重ねて受容野(1つの出力が見ている入力範囲)を広げる必要があり、長距離の関係を捉えにくい。100m 先の信号と自車の関係を1つの畳み込みで結ぶのは難しいわけです。

地図は本来「車線という線」「横断歩道という多角形」といった幾何オブジェクトの集まりです。それを無理に絵にしてから読み取り直すのは遠回りだ、というのが VectorNet の問題意識です。

### ステップ1: 何でもポリラインにする
VectorNet(2020、Waymo の Gao ら、arXiv:2005.04259)は地図と動きを最初からベクトルで表します。

- 車線: その中心線を、短い線分(ベクトル)の列として表す。1本の車線は複数のベクトルからなる「ポリライン(折れ線)」。
- 横断歩道・交差点の領域: 境界をたどるポリライン。
- 他車・歩行者の過去の軌跡: 過去の位置を時間順につないだポリライン。

各ベクトル(線分)は属性を持ちます。例えば `[始点x, 始点y, 終点x, 終点y, 属性(車線か軌跡か等), ポリラインID, 時刻]` のような特徴ベクトルです。形で言えば、1本のポリラインが `[線分数, 特徴次元]`、シーン全体が「ポリラインの集合」になります。すべての要素(地図も動きも)を同じ「ベクトル(線分)」という統一形式に落とし込むのが要点で、これにより地図と軌跡を区別なく同じネットワークで扱えます。

### ステップ2: 局所グラフ(ポリライン内サブグラフ)
まず各ポリラインの内部で、線分どうしの関係をまとめます。同じポリラインに属する線分をサブグラフ(部分グラフ)として扱い、線分の特徴を集約して「1本のポリライン = 1個のノード特徴ベクトル」に圧縮します。具体的には、各線分を共有 MLP で変換し、ポリライン内で最大プーリング(max pooling、要素ごとに最大値を取る集約)して得た「ポリライン全体の要約」を各線分に連結し、これを数回繰り返します。最後にもう一度プーリングして1本のベクトルにします。例えば10本の線分からなる車線が `[256]` のベクトル1個に要約される、というイメージです。

```
ポリライン(車線A): 線分1..線分10 [10, d]
   └ サブグラフ(MLP + maxpool を数回) ─> ノード特徴 [256]
```

このプーリングは順序に依存しない(permutation invariant)集約なので、線分をどの順で並べても同じ結果になり、点群を扱う PointNet と同じ思想です。

### ステップ3: 大域グラフ(ポリライン間)
次に、シーン内の全ポリライン(車線・横断歩道・各車の軌跡)をノードとする「完全グラフ(全ノードが互いにつながったグラフ)」を作り、その上でアテンション(セルフアテンション)をかけます。これにより「この車線」と「この歩行者の軌跡」のような、空間的に離れていても意味的に関係する要素どうしが、層を重ねずに一発で情報をやり取りできます。ラスタ+CNN では難しかった長距離関係が、グラフでは自然に扱えます。

```
全ポリラインをノードに: [車線A, 車線B, 横断歩道, 車X軌跡, 歩行者Y軌跡, ...]
   └ 完全グラフ上のアテンション ─> 各ノードが他の全要素から情報を集約
   └ 予測対象エージェントのノード特徴 ─> デコーダ ─> 未来軌跡
```

最後に、予測対象エージェントに対応するノードの特徴を取り出し、MLP デコーダで未来の座標系列(例えば3秒ぶんの位置)を出します。

### 補助タスク(自己教師あり的な穴埋め)
VectorNet は学習を助けるため、ノードの一部をわざと隠して「隠したポリラインの特徴を当てる」補助タスクを併用します。マスクされたノードを、その識別情報(どのポリラインかを示す ID 埋め込み)だけから他のノードの文脈を使って再構成させる課題で、後の BERT 的なマスク事前学習に通じる発想です。これにより地図構造の理解が深まると報告されています。

### 座標系の正規化
予測精度に効く実務的工夫として、座標を予測対象エージェント中心の座標系に正規化します。各エージェントを原点に置き、進行方向を基準に回転させると、絶対座標に依存しない表現になり汎化が効きます。同じ右折でも、世界座標では場所ごとに違う数値になりますが、エージェント中心に直せば似た数値になるため、学習が効率化します(この回転・並進不変の考え方は後継の HiVT で中心的設計になります)。

## 手法の系譜と主要論文

- Gao, Sun, Zhao, Shen, Anguelov, Li, Schmid, "VectorNet", CVPR 2020(Waymo、arXiv:2005.04259)。地図と動きをポリラインのベクトル集合で表し、局所(ポリライン内)→大域(ポリライン間)の階層グラフで符号化。ラスタ化の情報損失・計算の無駄・長距離関係の捉えにくさを避けるのが動機。ベクトル表現の起点。

- Liang et al., "LaneGCN", ECCV 2020(arXiv:2007.13732)。車線を専用のグラフ(lane graph)として構成し、グラフ畳み込みで車線と車の相互作用を、actor-to-lane(車→車線)、lane-to-lane(車線→車線)、lane-to-actor(車線→車)、actor-to-actor(車→車)の4種類の関係として明示的に表す。車線の接続(分岐・合流・前後関係)という構造を活かすのが狙い。

- Varadarajan et al., "MultiPath++", ICRA 2022(Waymo、arXiv:2111.14973)。ベクトル入力を集約しつつ、出力を複数モード(複数のありうる軌跡)として表し、各モードの確率も出す。混合ガウシアンで未来分布を表現し、動作予測の多峰性(行き先が複数あること)を明示的に扱う流れを強めた。

- Zhou et al., "HiVT: Hierarchical Vector Transformer for Multi-Agent Motion Prediction", CVPR 2022(arXiv:2207.04624)。ベクトル表現に Transformer と、各エージェント中心に座標系を取り直す回転・並進不変な設計を組み合わせる。多数の車を同時に、向きに依存せず効率よく予測する狙い。局所(エージェント近傍)→大域(エージェント間)の階層構造は VectorNet の精神を継承。

- Nayakanti et al., "Wayformer", ICRA 2023(Waymo、arXiv:2207.05844)。複雑な階層グラフ設計を整理し、ベクトル入力を素直にアテンションで融合する「シンプルだが強い」Transformer 構成を示した。入力の融合方法(early/late/hierarchical fusion)を体系的に比較し、設計の複雑さと性能のトレードオフを問い直した。

- Liao et al., "MapTR", ICLR 2023(arXiv:2208.14437)。HDマップそのものを生センサからベクトル化(オンラインで車線等のポリラインを検出)する方向。地図要素を順序のない点集合として表し、DETR 的なクエリと集合マッチングでベクトル地図を直接出力する。VectorNet が前提とした「整備済みベクトル地図」を、知覚側でその場で作り出す課題に取り組んだ。

これらに共通するのは「地図を画像ではなく幾何オブジェクト(ベクトル/グラフ)として扱う」潮流で、VectorNet はその起点です。

## 論文の実験結果(定量データ)

VectorNet と後継は主に Argoverse(動作予測の大規模ベンチマーク、約32万シナリオ)や、Waymo の社内データ・Waymo Open Motion Dataset で評価されました。動作予測の標準指標は次の通りです。ADE(Average Displacement Error、平均変位誤差。予測軌跡と正解軌跡の各時刻の距離を平均したもの、単位メートル、小さいほど良い)、FDE(Final Displacement Error、最終地点の誤差。数秒後の終端位置のずれ)、miss rate(予測が一定距離以内に当たらなかった割合)。複数モードを出す手法では minADE_k(k本予測してそのうち最良のものの ADE。例えば k=6 なら6本のうち一番当たったものを採点する)を使います。

VectorNet の論文は、同等以上の精度を達成しながらラスタ+CNN(ResNet ベース)に比べてモデルの計算量(FLOPs)とパラメータを大幅に削減できたことを報告しました。論文では、ラスタ化 ConvNet ベースラインと比べて約70%少ない計算量(モデルサイズも約29%)で同等以上の予測精度を達成した、と述べられています。これはベクトル表現の「疎な地図を疎なまま扱う」効率性を直接示す結果です。アブレーションでは、(1)大域グラフ(ポリライン間アテンション)を外すと長距離の文脈が失われ精度が落ちること、(2)補助の穴埋めタスクを加えると予測誤差がわずかに改善すること、(3)地図情報を入れる(車線文脈を与える)と入れない場合より明確に予測が当たること、が示されました。

後継研究はこの土台の上で精度を押し上げました。HiVT(2022)は Argoverse で当時の上位の minADE/minFDE を達成し、回転不変設計が向き依存のばらつきを減らすことを示しました。報告された Argoverse の minADE(k=6)はおよそ0.8m前後の水準です。Wayformer(2023)は Waymo Open Motion Dataset で、複雑な手作りグラフを使わずアテンションの素直な融合だけで競争力のある結果を出し、「設計の複雑さは必ずしも性能に必要ない」ことを定量的に示しました。MultiPath++ は複数モード出力により miss rate を下げ、行き先の多峰性を捉える重要性を裏づけました。指標の差はメートル単位の小ささに見えますが、自動運転では数十センチの予測ずれが「ぶつかるか避けられるか」を分けるため、これらの改善は安全性に直結します。

## メリット・トレードオフ・限界

メリット側。連続座標をそのまま使うため位置精度の損失が小さい。疎な地図を疎なまま扱うので、空きスペースに計算を浪費しない。グラフ+アテンションで、遠く離れた要素間の長距離関係を一発で捉えられる。座標正規化により向き・位置への不変性を持たせやすい。地図も軌跡も同じベクトル形式に統一できるので、入力の種類が増えても扱いやすい。

トレードオフと限界。正確なベクトル地図(HDマップ)とベクトル化済み軌跡の入力が前提で、生のカメラ/LiDAR からのベクトル化は別問題(これに取り組んだのが MapTR 等のオンライン地図構築)。グラフ構築のルール(何をノードにし、どうつなぐか、線分をどう区切るか)に設計判断が要り、設計者の事前知識に依存します。画像と違い人間が直感的に可視化・デバッグしにくい面もあります。さらに、HDマップへの依存はスケーラビリティの課題でもあります。世界中の道路を高精細地図で維持し続けるのは高コストで、地図がない・古い地域では破綻しうるため、地図に頼りすぎない設計が近年の論点です。ラスタ手法が画像系の豊富な事前学習資産(ImageNet で学んだ CNN など)を流用できたのに対し、ベクトル手法はそうした資産を直接は使いにくい、という実務上の不利もあります。

## 発展トピック・研究の最前線
ベクトル/グラフ表現は動作予測の事実上の標準になりましたが、最前線ではいくつかの方向に発展しています。第一に、知覚と予測の統合です。MapTR のようにセンサから直接ベクトル地図を作り、それを予測へ渡すことで、別途整備された HDマップへの依存を減らす流れがあります。第二に、表現の統一です。検出・地図・予測・計画を、固定本数の学習可能クエリの集合として一貫して扱う設計(クエリベースの統合アーキテクチャ)が広がり、ベクトル表現はその入力エンコードの一部として吸収されつつあります。第三に、HDマップなし・地図軽量(SD マップや車線トポロジーのみ)の予測です。高精細地図の維持コストを避け、その場で観測した道路構造から予測する研究が活発です。

地図エンコードを設計する際の本質的な選択は「ラスタ(画像として CNN で処理)かベクトル(幾何として GNN/Transformer で処理)か」というトレードオフで、ベクトル側は精度・効率・長距離関係で有利、ラスタ側は実装の素直さと既存の画像系資産の流用しやすさで有利、という構図はいまも有効な比較軸です。近年は大規模 Transformer の普及でベクトル側が主流ですが、エンドツーエンドの BEV(鳥瞰図)特徴に地図情報を融合する方式では、ベクトルとラスタの中間的な「BEV グリッド上の地図クエリ」のような折衷も現れています。
