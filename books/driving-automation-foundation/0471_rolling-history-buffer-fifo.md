# RollingHistoryBuffer (FIFO履歴)

## ひとことで言うと
モデルが「直近の数秒間に何を見てきたか」を覚えておくための、いちばん単純な仕組みです。各時刻のカメラ画像を1本の短いベクトルに圧縮し、それを「最新N個だけ」棚に並べておきます。新しいフレームが来たら一番古いものを捨てて押し出す、ところてん方式(FIFO = First In First Out、先入れ先出し)です。並べた中身を連結して時系列の入力にします。学習パラメータがゼロで、ただ並べて繋ぐだけ、という割り切りが特徴です。

## 直感的な理解
1枚の静止画だけを見て運転を決めると、分かるのは「物体が今どこにあるか」だけです。しかし運転で本当に必要なのは「その物体がどっちへ、どれくらいの速さで動いているか」です。前の車が止まっているのか減速中なのか、横断歩道の歩行者がこちらへ歩いてくるのか立ち止まっているのか。これは1枚の静止画では原理的に判別できません。動きは「複数の時刻の画像を比べる」ことで初めて分かります。

そこで「過去の見た目」を時系列として持っておく必要があります。最も単純なやり方は、付箋に各瞬間の要約を書いて、机に最新の数枚だけ並べておくことです。新しい付箋が来たら一番古い付箋を剥がして捨てる。並んだ付箋をまとめて読めば、「数秒前から今まで、シーンがどう変化したか」が分かります。RollingHistoryBuffer はこの付箋方式そのものです。学習も推論もせず、ただ並べて繋ぐだけ。これが「最初の一手はとにかく単純に」という設計判断です。

## 基礎: 前提となる概念
埋め込み (embedding)。画像のような複雑なデータを、固定長の数値の並び(ベクトル)に圧縮した表現です。たとえば「フレーム埋め込み224次元」とは、1時刻ぶんのシーンを224個の数字で表したもの。圧縮の作り方は [FrameEncoder (GAP+Linear埋め込み)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0354_frame-encoder-gap-linear) や [ViewAttentionPool (マルチビュー集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0319_view-attention-pool) を参照。

凍結 backbone (frozen backbone)。画像をエンコードする骨格ネットワークの重みを、学習中に更新しない(固定する)使い方です。大規模事前学習済みの強い特徴抽出器をそのまま流用し、その上の軽い部品だけを学習します。利点は学習が安定・高速で、過学習しにくいこと。さらに「同じ画像なら毎回同じ特徴が出る」ので、特徴を一度計算してキャッシュ(保存して再利用)でき、時系列処理が大幅に軽くなります。

FIFO キュー (先入れ先出しの待ち行列)。入った順に出ていくデータ構造です。容量N(たとえば4)で満杯のとき新しい要素が来ると、最も古い要素を1つ捨て(pop)、新しい要素を末尾に追加(push)します。常に「直近N個」だけを保持します。リングバッファ(ring buffer)で実装すれば、毎回ずらし直さず添字を回すだけで O(1) で更新できます。

連結 (concat)。複数のベクトルを横に繋いで1本の長いベクトルにする操作です。224次元を4本連結すると `224×4=896` 次元になります。情報は一切失われませんが、本数を増やすと次元が線形に増えます。

## 仕組みを詳しく
処理をステップで分解します。典型的な設定値を例にとります(カメラ台数 V=7、フレーム埋め込み次元 D=224、履歴長 N=4、履歴の周期 1Hz = 1秒に1フレーム、よって約4秒分の記憶)。

1. 1時刻ぶんの符号化。複数カメラ画像を凍結 backbone に通し、フレームエンコーダ(GAP+Linear、あるいは注意プール)が1本の `[B, 224]` 埋め込みを作る。B はバッチサイズ。
   - 形状の変化: V カメラの特徴マップ → 集約 → `[B, 224]`
2. push。この `[B, 224]` を棚(FIFO キュー、容量4)に追加する。
3. pop。4個埋まっている状態で5フレーム目が来たら、最も古い(4秒前の)ものを捨てる。これが FIFO。
4. 連結。棚の中身4個を最後の軸で連結し `[B, 896]` を作る。これが時系列入力(視覚履歴)になる。

ASCII図(時刻が進むときの棚の動き):

```
時刻 t=5 が来る直前の棚(古い→新しい):
  [ e_1 | e_2 | e_3 | e_4 ]        各 e は 224 次元
新フレーム e_5 が到着:
  e_1 を捨てる(pop)→ e_5 を末尾に追加(push)
  [ e_2 | e_3 | e_4 | e_5 ]
連結(concat)して下流へ渡す:
  visual_history = [ e_2 e_3 e_4 e_5 ] = 896 次元
```

重要な性質: 学習パラメータがゼロ。並べて繋ぐだけなので、4フレームすべてが完全に同じ重み(等重み)で扱われ、順序も固定(古い→新しいが必ず同じ位置)です。連結では「位置そのものが順序を表す」ので、後述する attention のような明示的な位置埋め込みは不要です。バグや学習不安定の余地がほとんどなく、挙動が完全に予測可能でデバッグしやすい。一方で「直近を重視する」「無関係なフレームを軽くする」といった選択はできません。これが後述の attention pool([temporal attention-pool (時間方向集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0477_temporal-attention-pool))で改善される点です。

なぜ N=4 か / なぜ 1Hz か。履歴長と周期は速度・加速度を捉えるのに十分かつ過大でない範囲で決めます。位置だけなら1フレーム、速度を1階差分で得るには2フレーム、加速度を2階差分で得るには3フレーム、というのが目安で、ノイズへの頑健性のために少し余分に持って4フレーム程度にすることが多いです。フレームが少なすぎると動きが読めず、多すぎると入力が肥大化し(連結だと次元が線形増加)、計算とメモリを食います。周期は対象の運動スケール(歩行者や車の数秒の動き)に合わせて選びます。1Hz で4フレームなら約4秒、これは前方車両の減速や歩行者の進入を捉えるのに妥当な時間窓です。

ドロップイン互換。出力次元を `D×N = 224×4 = 896` に設計しておくと、もともと896次元の「視覚履歴」入力を期待する下流モジュールに、改修なしでそのまま差し込めます。最初はこの入力が全ゼロのダミーで埋められていることがあり、それを「本物の過去の視覚特徴」で置き換えるのがこのバッファの役割です。学習初期にバッファがまだ満杯でない(過去フレームが揃わない)場合は、ゼロ埋めや最初のフレームの複製でパディングするのが一般的です。

## 手法の系譜と主要論文
フレームスタッキング(frame stacking)の古典: DQN (Mnih et al., Nature 2015, "Human-level control through deep reinforcement learning")。Atari ゲームを学習する深層強化学習で、直近4フレームの画像をチャンネル方向に積んでネットワークに入力しました。動機は「1枚ではボールの速度や向きが分からない」こと。直近4枚を積むだけで、ネットワークが速度・方向を暗に推定できるようになりました。「直近4枚を積む」という設計は RollingHistoryBuffer とほぼ同型です。トレードオフは等重みで「どのフレームが今の判断に効くか」を選べないこと。

固定長キューによる時間融合: BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)。過去の BEV(俯瞰)特徴を固定長のキューで保持し、現在フレームと時間的自己注意(temporal self-attention)で融合しました。動機は単一フレームでは速度や遮蔽(オクルージョン、物体が一時的に隠れること)に弱いこと。「固定長キュー」という発想はここに近いですが、BEVFormer が注意で賢く融合するのに対し、RollingHistoryBuffer は単純連結に割り切っています。

学習可能な集約への発展: Set Transformer (Lee et al., ICML 2019, arXiv:1810.00825)。集合を平均や連結ではなく、学習可能なクエリの attention で集約する道筋を示しました。連結を、この学習可能な集約に置き換えたものが [temporal attention-pool (時間方向集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0477_temporal-attention-pool) です。つまり「単純連結(等重み)→ 学習可能な加重(attention)」という改良の系譜上に、このバッファは出発点として位置します。

再帰の対照: GRU (Cho et al., EMNLP 2014, arXiv:1406.1078)。明示的にフレームを並べるのではなく、1本の隠れ状態に過去を畳み込む方式の代表です。任意長を一定サイズの状態に圧縮できる反面、遠い過去を忘れやすく、時系列を順に処理するため並列化しにくい。FIFO+連結はこの逆で、長さに上限がある代わりに並列化と解釈が容易です。

FIFO 履歴バッファ自体は「単純な実装テクニック」なので論文の主題になることは少ないですが、その背後には「複数時刻を束ねて動きを捉える」という、時系列認識・ビデオ理解・強化学習に共通する普遍的な発想があります。

## 論文の実験結果(定量データ)
DQN。Atari 2600 の49ゲームで、4フレームスタックを入力とする1つのネットワーク構成が、多くのゲームで人間のスコアを上回りました(報告では半数以上のゲームで人間のプロ水準の75%以上)。アブレーション的な観察として、フレームスタックを外して単一フレームにすると、ボールやキャラの速度が読めず性能が大きく落ちます。「複数フレームを束ねるだけで運動情報が入る」ことの実証です。

BEVFormer。nuScenes で、時間的注意(過去フレームのキューを使う融合)を加えると、加えない場合に比べて速度推定誤差 mAVE(mean Average Velocity Error、物体の速度推定の平均誤差、小さいほど良い)が大きく改善し、総合指標 NDS(nuScenes Detection Score、高いほど良い)が数ポイント向上しました。代表値として NDS およそ0.56、mAP およそ0.42前後が報告されています。これは「過去フレームを保持して融合すること」自体の効果を示しており、その融合を単純連結にするか注意にするかが次の設計選択になります。

要点は、(1) 複数時刻を保持するだけで速度・遮蔽への頑健性が上がる、(2) その保持の仕組みは固定長キュー(=RollingHistoryBuffer)で十分実用になる、(3) 融合を等重み連結から学習可能な注意に上げると、さらに改善の余地がある、という3段の経験則です。

## メリット・トレードオフ・限界
メリット
- 実装が極めて単純。学習パラメータがゼロなのでバグや学習不安定の余地が小さい。
- 出力次元が `D×N` で固定なので、既存の下流入力にそのまま合わせやすい(ドロップイン)。
- N個のフレームを等重み・固定順で扱うので挙動が予測可能でデバッグしやすい。
- 凍結 backbone と組み合わせると、フレーム特徴をキャッシュでき計算が軽い。

トレードオフ・限界・未解決課題
- N個のフレームを完全に等しく扱うため、「直近を重視する」「無関係なフレームを軽くする」といった選択ができない(学習で重み付けできない)。
- 履歴長が固定。データセットや対象の運動スケールごとに最適な履歴長が違っても、連結だと柔軟に変えにくい。
- 連結なのでフレーム数を増やすと次元が線形に増える(8フレームなら 224×8=1792)。下流の入力次元も変わってしまう。
- 1Hz のような粗い周期だと、速い運動(高速で横切る車両)を取りこぼす。周期を上げるとメモリ・計算が増える。
- 急な遮蔽やフレーム欠損があると、古い等重みのフレームが誤った文脈を与える危険がある。

## 発展トピック・研究の最前線
- 学習可能な時間集約への置換。連結を [temporal attention-pool (時間方向集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0477_temporal-attention-pool) に替えると、フレームごとの重要度を学習でき、出力次元も履歴長Nに依存しなくなります。両者を同一インターフェースにしておけば差し替え(swappable)できます。
- 再帰的状態(RNN/GRU)との比較。FIFO+連結は「直近N枚を明示的に並べる」のに対し、GRU や LSTM は「1本の隠れ状態に過去を畳み込む」方式です。再帰は任意長を1状態に圧縮できますが、遠い過去を忘れやすく、並列化しにくい。明示的バッファは並列化しやすく解釈しやすいが、長さに上限があります。
- 特徴キャッシュとの統合。凍結 backbone なら、各フレーム特徴を一度計算してキャッシュし、時系列学習で backbone を再実行しない最適化が効きます。
- 不等間隔・欠損への対応。実データではフレーム間隔が一定でなかったり欠損したりします。時間位置埋め込みで「いつのフレームか」を明示する方式([temporal attention-pool (時間方向集約)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0477_temporal-attention-pool))は、こうした不規則性に強くなります。
- マルチスケール時間窓。短い窓(高頻度・近過去)と長い窓(低頻度・遠過去)の複数バッファを併用し、速い運動と長期文脈の両方を捉える設計も研究されています。
