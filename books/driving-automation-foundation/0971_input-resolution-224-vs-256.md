# 入力解像度の統一（224 vs 256）

## ひとことで言うと
モデルに入れる画像の縦横サイズ(入力解像度)が、学習・評価・ベンチマーク・テストの間でバラバラ(たとえば 224 か 256 か)だと、特徴マップの形が食い違ってバグや評価のズレを生みます。階層型 Transformer を主役にする設計では 256×256 に統一するのが扱いやすいことが多いです。理由は、窓注意の窓サイズ(たとえば window_size=8)で 256 はきれいに割り切れる(256/32=8)のに対し、224 では端数(224/32=7)が出てパディングが要るからです。「解像度は単独で決めず、ネットワーク内部の区切り(総ストライドと窓サイズ)とセットで決める」というのが本質です。

## 直感的な理解
料理のレシピで「200g の小麦粉」と書いてあるのに、計量カップが 150ml 単位でしか測れなかったら、毎回端数を切ったり足したりして誤差が積もります。最初から「カップ1杯=ちょうど一区切り」になるよう材料の量を選べば、計量が正確で速くなります。入力解像度も同じで、ネットワークの内部処理の「区切り」にぴったり合う解像度を選ぶと、無駄なパディング(余白の水増し)が要らず、形のそろえ方もきれいになります。

なぜ統一が安全に直結するか。学習・評価・テストで解像度が違うと、次の問題が起きます。第一に形の不一致です。224 入力と 256 入力ではバックボーンを通った後の特徴マップの空間サイズが変わり、あるコードが 256 前提、別のコードが 224 前提だと、テンソルの形が合わず実行時エラーや、最悪は黙ったままの誤動作(ブロードキャストやリサイズが暗黙に走って間違った結果を返す)になります。第二に評価の不公平です。学習を 256、評価を 224 でやると、モデルが見たことのない解像度で測ることになり、性能を正しく測れません。第三にベンチマークの不整合で、速度計測を 224、本番を 256 でやると、測ったレイテンシが実運用と乖離します(画素数が約31%違うので無視できない差)。

## 基礎: 前提となる概念
- 入力解像度: モデルに入れる前にリサイズした画像の縦横ピクセル数。224×224 や 256×256 がよく使われます。
- ダウンサンプリング(downsampling): 特徴マップの空間サイズを縮める操作。階層型バックボーンは段階的にこれを繰り返します。「総ストライド 32」とは、入力に対し最終的に 1/32 の解像度になることを意味します。
- 窓注意(window attention): Swin Transformer が採用する仕組みで、画像全体ではなく小さな正方形の窓(たとえば 8×8)の中だけで注意を計算します。全画素間の注意は計算量が画素数の2乗で爆発するため(理由は [埋め込み次元と注意機構の計算コスト](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0914_attention-cost-embed-dim) の N² 項)、窓に区切って局所化することで現実的なコストに抑えます。窓サイズ(window_size)は、最深段の特徴マップの一辺がその倍数になっていると都合がよい設計です。
- シフト窓(shifted window): Swin は層ごとに窓の位置を半分ずらすことで、窓の境界をまたぐ情報も伝わるようにしています。窓に区切る局所性の欠点(窓をまたぐ関係が見えない)を補う工夫です。
- 割り切れる/端数: 特徴マップの一辺(画素数)を窓サイズで割ったときに余りが出るかどうか。余りが出ると、窓に合わせるためのパディングが要ります。
- パディング(padding): サイズを合わせるために特徴マップの周囲に余白(多くはゼロ)を足すこと。実際の画像情報ではないので、計算が一部無駄になり、段ごとのサイズも半端になります。
- 位置埋め込み(positional embedding): Transformer はトークンの順序・位置を本来知らないため、各位置に「ここは何番目・どこ」という情報をベクトルで足します。解像度を変えるとトークン数(位置の数)が変わるため、位置埋め込みの補間(interpolation)が必要になります。

## 仕組みを詳しく
なぜ 256 が窓注意にとって都合がよいのかを数値で見ます。階層型 Transformer は、最初に 4×4 のパッチに切って 1/4 にし、その後3回ダウンサンプリング(各 1/2)するので、最深段では入力の 1/32 の解像度になります(4 × 2 × 2 × 2 = 32)。

```
入力 256 -> 最深段の一辺 = 256 / 32 = 8
入力 224 -> 最深段の一辺 = 224 / 32 = 7
```

window_size = 8 の場合、最深段の一辺がちょうど 8 なら1窓に収まり、パディング不要です。256 はこれを満たします(256/32=8、8/8=1 でぴったり)。一方 224 では最深段が 7×7 になり、8×8 の窓に収まりません。実装上はパディングで 8 に水増しして処理しますが、

- パディングした余白は実際の画像情報ではないので計算が一部無駄になる。
- 段ごとのサイズが「7 → 余白付きで 8」のように半端になり、複数スケールをプールして連結する設計([マルチスケール特徴の連結](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0986_multi-scale-feature-concat))で形のそろえ方が複雑になる。

256 入力での各段の一辺を見ると、すべて2のべき乗にそろいます。

```
256 入力での各段の一辺:
  stage0: 64,  stage1: 32,  stage2: 16,  stage3: 8   (すべて 2 のべき乗)
```

すべて2のべき乗にそろうため、`AdaptiveMaxPool2d(8)` で 8×8 に落とす処理(連結の前段)もきれいに働きます。逆に 224 だと、

```
224 入力での各段の一辺:
  stage0: 56,  stage1: 28,  stage2: 14,  stage3: 7
```

最深段が 7 になり、窓サイズ 8 と割り切れません。window_size=7 を使う設計なら 224 がぴったりです(224/32=7)。つまり「解像度と窓サイズはセットで決める」のが本質で、片方だけを慣習で決めると不整合になります。Swin の標準構成が「224 入力 + window_size=7」なのはこの割り切れ条件を満たすためで、window_size=8 を使うなら対応する 256 を選ぶのが自然、という関係です。

実務では、学習・速度ベンチマーク・テスト用のモックを含めて入力を一律 256 に固定し、メタデータにも `input_resolution=(256, 256)` のように明記しておくと、後から「どの解像度で測ったか」が追えて再現性が保てます。モックバックボーンも 256 入力で 64/32/16/8 の段サイズを再現するよう作ると、テストが本番の形を正しく反映します([backbone特徴サイズの抽象化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0919_backbone-feature-size-abstraction) のテスト落とし穴と同根)。

なお、解像度を上げると計算は画素数の2乗で増えます。256² / 224² ≒ 1.31 なので、224 → 256 で計算とメモリが約31%増えます。窓注意自体は窓数に線形なので破滅的ではありませんが、無料ではありません。だから「窓サイズに割り切れる最小の解像度」を選ぶのがコスト効率の落としどころになります。

## 手法の系譜と主要論文
- He et al., "Deep Residual Learning (ResNet)", CVPR 2016(arXiv:1512.03385)/ Dosovitskiy et al., "ViT", ICLR 2021(arXiv:2010.11929)。これらが 224×224 を慣習として広めました。CNN や非階層 ViT はストライドやパッチ分割だけで縮小し、窓注意のような「窓サイズで割り切れる必要」が無いため、224 でも端数問題が起きにくく、224 で困りませんでした。この慣習が「とりあえず 224」という設定をコードに残し、窓注意中心の設計で不整合を生む背景になりました。

- Liu et al., "Swin Transformer", ICCV 2021(arXiv:2103.14030)。Swin の標準入力は 224 ですが、これは window_size=7 と組み合わせる設計です(224/32=7)。「解像度と窓サイズはセットで割り切れるよう選ぶ」のが前提だと示しています。window_size=8 を使うなら対応する 256 を選ぶのが自然、というのがこの論文からの含意です。Swin はシフト窓で窓境界の情報伝播を確保しており、窓サイズと解像度の整合がアーキテクチャの正しさに直結することを浮き彫りにしました。

- Touvron et al., "Training data-efficient image transformers (DeiT)", ICML 2021(arXiv:2012.12877)。学習解像度と評価解像度を一致させること、不一致がある場合は位置埋め込みの補間などの対処が要ることを実証的に論じました。「学習と評価で解像度をそろえる」原則の裏付けです。少ないデータで ViT を学習する蒸留手法も提案し、解像度の扱いと学習効率が密接であることを示しました。

- Touvron et al., "Fixing the train-test resolution discrepancy (FixRes)", NeurIPS 2019(arXiv:1906.06423)。学習時のデータ拡張(ランダムクロップ)と評価時の中央クロップで、物体の見かけ大きさがずれるため、学習解像度より少し大きい解像度で評価すると精度が上がる、という現象を定量化しました。解像度の不一致が無視できない効果を持つことの直接的な証拠です。

- Tan & Le, "EfficientNet", ICML 2019(arXiv:1905.11946)。深さ・幅・解像度を1つの複合係数(compound coefficient)で同時にスケールする方法を提案し、解像度を独立に上げるより効率がよいことを示しました。解像度はモデル設計の独立変数ではなく、深さ・幅と連動して決めるべき量だという視点を与えます。

系譜としては、ResNet/ViT が 224 を慣習化し、Swin が「窓サイズと解像度をセットで決める」必要を持ち込み、DeiT/FixRes が「解像度の不一致は性能に効く」ことを定量的に示し、EfficientNet が「解像度は他の次元と連動して決める」ことを示した、という流れです。これらが合わさり、「主役バックボーンの内部構造に合う解像度を1つ選んで全工程で統一する」という実務上の規律につながっています。

## 論文の実験結果(定量データ)
指標は ImageNet の Top-1 精度(分類正解率、大きいほど良い。最も確信度の高い1クラスが正解と一致する割合)が中心です。

- FixRes(NeurIPS 2019)は、ResNet50 を 224 で学習して 224 で評価する標準設定に対し、評価解像度を上げて(たとえば 320 など)分類器を微調整することで Top-1 を約 1〜2 ポイント改善できると報告しました。さらに大規模学習との組み合わせで、当時の ImageNet SOTA(86%級)を更新しています。ここで重要なのは「学習と評価の解像度のわずかなずれが、1ポイント規模で精度を動かす」という事実です。自動運転の文脈に置き換えれば、解像度の不整合は性能評価を 1〜2 ポイント単位で歪めうる、ということです。

- DeiT(ICML 2021)は、224 で学習したモデルを高解像度(384 など)へ微調整すると Top-1 が上がる一方、位置埋め込みを補間せずに解像度だけ変えると精度が落ちることを示しました。アブレーションとして「位置埋め込みの補間あり/なし」で差が出るため、解像度を変えるなら埋め込みの調整が必須、という教訓が得られます。位置埋め込みはトークン位置の数に紐づくため、トークン数が変わると整合が崩れるのが理由です。

- Swin(ICCV 2021)の報告では、224 入力・window_size=7 の構成で ImageNet Top-1 が Swin-T で約 81.3%、Swin-B で約 83.3〜83.5% です。解像度を上げる(384 入力)と精度はさらに上がりますが、計算量(GFLOPs)も解像度の2乗で増えます。256 と 224 の比では画素数が約 31% 多く(256²/224² ≒ 1.31)、その分計算とメモリが増えるという素直なトレードオフがあります。

- EfficientNet(ICML 2019)は、深さ・幅・解像度を別々に上げるより複合係数で同時に上げる方が、同じ FLOPs で高い精度に達することを示しました。解像度だけを単独で上げても効率の頂点には届かない、という定量的な裏付けです。

定量データの読み方として大事なのは2点です。第一に、解像度の不一致は無視できる誤差ではなく Top-1 を 1〜2 ポイント動かしうること。第二に、解像度を上げれば精度は上がりがちだが計算は2乗で増えるので、「窓サイズに合う最小の解像度を選ぶ」のがコスト効率の観点で合理的なこと。256 は window_size=8 に対する最小の割り切れる解像度として、この両方を満たす落としどころです。

## メリット・トレードオフ・限界
メリット
- 学習・評価・ベンチマーク・テストが同じ解像度で動き、形の不一致バグと評価のズレが消える。
- 256 は window_size=8 で割り切れ、窓注意がパディングなしで動くため計算が無駄にならない。
- 各段が2のべき乗サイズになり、プール+連結の前処理がきれいに働く。
- 解像度をメタデータに明記すれば、後から測定条件を追え、再現性が保てる。

トレードオフ・限界
- 256 は 224 より画素数が約 31% 多く、その分わずかに計算とメモリが増える。組み込みで予算が厳しいと無視できない。
- 224 で学習された既存の事前学習重みを使う場合、256 へ合わせると位置埋め込みの補間など追加の配慮が要る(DeiT が示した通り、補間を怠ると精度が落ちる)。窓注意の相対位置バイアスも再スケーリングが要る。
- 解像度を上げても遠方の小物体が必ず見えるわけではない。実効的な見え方はカメラ画質・レンズ・視野角とのバランスで決まり、入力解像度だけ上げても頭打ちになりうる。
- 窓サイズと解像度を一度固定すると、後からどちらかだけを変えると不整合が再発する。両者は常にセットで管理する必要がある。設定ファイルに両方を併記し、不整合を起動時に検証するのが堅牢。

未解決の課題として、「タスクごとに最適な解像度・窓サイズの組をどう決めるか」「可変解像度入力にどう頑健にするか」は研究途上で、固定 256 が全タスクで最適とは限りません。自動運転では遠方小物体のために高解像度が欲しい一方、車載計算予算が厳しく、解像度の最適点はタスクとハードウェアの両制約で決まります。

## 発展トピック・研究の最前線
- 解像度スケーリング: EfficientNet(Tan & Le, ICML 2019)は深さ・幅・解像度を同時にスケールする複合係数を提案し、解像度を独立に上げるより効率がよいことを示しました。
- 可変解像度・動的解像度: NaViT(Dehghani et al., NeurIPS 2023, arXiv:2307.06304)は、画像を固定解像度に揃えずパッチ列として可変長で扱い、複数の画像を1つの系列に詰め込む(packing)ことで、解像度統一の制約を緩めつつ計算効率を上げる方向を探っています。固定解像度前提を崩す試みとして注目されています。
- 位置埋め込みの補間: 解像度変更に伴う位置埋め込みの双線形補間や、Swin の相対位置バイアスの再スケーリングは、事前学習重みを別解像度で使う際の標準テクニックとして確立しつつあります。
- 自動運転特有の高解像度需要: 遠方小物体検出のために高解像度入力やタイル分割(画像を分割して各部を高解像度で処理)が研究されており、計算予算と検出距離のトレードオフが活発な論点です。
