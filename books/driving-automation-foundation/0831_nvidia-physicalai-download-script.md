# 大規模マルチカメラ走行データセットの取得と前処理

## ひとことで言うと
公開されている大規模な自動運転データセットは、配布されている形 (圧縮アーカイブの塊) と、学習コードが読みたい形 (決まったフォルダ構成) が一致しないことがほとんどです。両者の橋渡しをするのが「取得・展開・整形スクリプト」です。公式 SDK でクラウドからクリップを取得し、パーサが期待するディレクトリレイアウトに並べ替えます。本ノートは、研究で公開データを使うときに必ず直面する「データの前処理パイプライン」を一般論として解説します。

## 直感的な理解
通販で大型の組み立て家具を買ったとします。届くのは「平たく梱包された板の束」(配布形式) です。一方、組み立て説明書 (学習コード) は「天板はここ、脚はここ、ネジはこの順で」という決まった配置 (期待レイアウト) を前提にしています。梱包を開けて、板とネジを説明書どおりの場所に並べる作業が要ります。これを手でやると、部品が多いほど時間がかかり、間違えやすい。

データセットでもまったく同じです。配布側は「容量を小さく、転送しやすく」するために、カメラごと・特徴ごとに圧縮アーカイブ (zip / tar) にまとめて配布します。一方、学習コードのパーサは「カメラ動画はこのフォルダ、タイムスタンプはこの名前、自車の動きはあのフォルダ」という固定レイアウトを前提に動きます。配布形式のままではパーサは1ファイルも読めません。そこで「ダウンロード → 展開 → 期待レイアウトに配置」を自動化するスクリプトが必要になります。

なぜ自動化するかというと、マルチカメラのデータセットは「7台のカメラ × 複数クリップ × 複数の特徴 (動画・タイムスタンプ・自車の動き)」と組み合わせが膨大で、手作業ではミスが避けられないからです。前処理を自動化・再現可能にすることは、データセットを使う研究の信頼性の前提です。1回しか使わない使い捨て作業に見えても、データ追加・別環境での再現・バグ修正のたびに再実行するので、スクリプト化の投資はすぐ回収できます。

## 基礎: 前提となる概念

用語を噛み砕きます。

クリップ (clip) とは、1本の短い走行記録です。数十秒〜数分の運転シーンに、複数カメラの映像と各種センサ値が紐づいています。データセットは大量のクリップの集まりで、各クリップは UUID (Universally Unique Identifier、世界で重複しない128ビットの識別子。`3f2a...` のような長い文字列) で一意に区別されます。

特徴 (feature) とは、クリップを構成する情報の種類です。たとえば「前方広角カメラの動画」「後方望遠カメラの動画」「自車の動き (egomotion)」などがそれぞれ1つの特徴です。配布側はこの特徴ごとに別アーカイブを用意することが多く、利用者は必要な特徴だけを選んでダウンロードできます。LiDAR 点群が要らなければカメラと egomotion だけ落とす、といった選択ができ、転送量を節約できます。

egomotion (エゴモーション) とは、自車自身の動きのことです。位置・速度・進行方向の時系列で、「この車がいつどこをどう動いたか」を表します。模倣学習では、これが「自車の過去の動き (履歴)」と「これからの軌跡の正解 (教師信号)」の元データになります。

Parquet とは、表データを効率的に保存する列指向フォーマットです (詳細は [状態量の高速読み出し (動画デコード回避)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0847_vehicle-state-fast-path) 参照)。タイムスタンプや egomotion のような数値表は Parquet で配布されることが多く、必要な列だけを高速に読めます。タイムスタンプ (timestamps) は、各動画フレームが何時何分何秒のものかを記録した時刻表で、複数センサの時刻を揃える (同期する) のに使います。7台のカメラはシャッターのタイミングが完全には揃わないため、このタイムスタンプを使って「同じ瞬間のフレーム」を後から対応づけます。

公式 SDK (Software Development Kit) とは、データ提供元が用意した「データセットをプログラムから扱うためのライブラリ」です。生の URL を直接叩くのではなく SDK を介すと、認証 (アクセス権の確認)・キャッシュ (再ダウンロード回避)・索引管理 (どんなクリップがあるかの目録)・再試行 (ネットワーク失敗時のリトライ) を SDK に任せられます。これは HuggingFace の `huggingface_hub` / `datasets` が広めた「データセットを API でプログラム的に取得する」流儀に沿った設計です。

## 仕組みを詳しく

典型的な取得スクリプトが想定する「期待レイアウト」は次のような構造です。

```
<出力ルート>/
  camera/<カメラ名>/<uuid>.<カメラ名>.mp4                # カメラ動画
  camera/<カメラ名>/<uuid>.<カメラ名>.timestamps.parquet  # フレーム時刻表
  labels/egomotion/<uuid>.egomotion.parquet             # 自車の動き
```

ここで `<カメラ名>` は `camera_front_wide_120fov` (前方広角120度) や `camera_rear_tele_30fov` (後方望遠30度) のように、視野角 (FOV: Field Of View、カメラが見渡せる角度) と取り付け位置を名前で表します。7台構成なら、前方広角・前方望遠・左右・後方など、死角を減らす配置になっています。広角は近距離の広い範囲、望遠は遠方の狭い範囲を担い、互いを補完します。

スクリプトの処理の流れをステップで示します。

1. コマンドライン引数を解析する。出力先フォルダは必須。取得対象は「ランダムに N 本」か「特定の UUID を指定」のどちらか一方を排他的に指定するのが定石。

```
python download_dataset.py --out /data/av --clips 1
python download_dataset.py --out /data/av --clip-uuids <uuid1> <uuid2>
```

`--clips 1` のように1本だけ落とせると、いきなり全量 (数TB) を落とさずに動作確認 (smoke test) ができます。これは大規模データを扱ううえで非常に重要な実務習慣です。

2. SDK を初期化する。このときクラウドからデータセットのメタ情報 (どんなクリップがあるかの索引) をダウンロードする。非対話 (CI やバッチ実行) で回すために、容量確認ダイアログを抑止する設定を入れることが多い。

```python
ds = DatasetInterface(
    local_dir=str(out / ".hf_cache"),
    confirm_download_threshold_gb=float("inf"),  # 何GBでも確認なしで進む
)
```

3. どのクリップを落とすか決める。`--clips N` なら索引の先頭から N 本、UUID 指定ならその UUID をそのまま使う。

4. 各クリップについて、必要な特徴 (全カメラ + egomotion) を選んでダウンロードし、圧縮アーカイブを取得する。

```python
features_to_dl = CAMERAS + ["egomotion"]   # 7カメラ + 自車の動き
ds.download_clip_features(clip_id, features=features_to_dl)
```

5. カメラごとに、アーカイブを開いて該当クリップのファイルだけを取り出し、期待レイアウトに書き出す。ファイル名を見て分岐するのが一般的で、`.mp4` ならカメラ動画、`timestamp` を含む `.parquet` なら時刻表として、それぞれ正しい名前 (`<uuid>.<カメラ名>.mp4` など) で保存する。

6. egomotion のアーカイブも同様に開き、`.parquet` を `labels/egomotion/<uuid>.egomotion.parquet` として保存する。

アーカイブを開くときは、可能ならストリーム (少しずつ読む) で開くヒントを付けるのが望ましいです。マルチGBのファイルをいきなり全部メモリに載せるとメモリスパイク (一時的な急増) でプロセスが落ちる危険があるからです。32GB メモリの環境で 10GB のアーカイブを2つ同時に全読みすれば、それだけで OOM の危険水域に入ります。このメモリ管理の詳細は [一時ファイルへのストリーム展開でメモリ節約](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0838_stream-to-tempfile-extraction) で扱います。

整形が終われば、出力ルートをそのままパーサに渡せば学習に使えます。データが無い環境ではテストを自動スキップする、という設計にしておくと、データを持たない開発者でもリポジトリ全体のテストが通り、CI が落ちません。

## 手法の系譜と主要論文

これは「手法」ではなくデータエンジニアリングですが、その背景には大規模自動運転データセットの明確な系譜があります。

公開データセットの時代は KITTI (Geiger et al., "Are we ready for Autonomous Driving? The KITTI Vision Benchmark Suite", CVPR 2012) に始まります。ステレオカメラと Velodyne LiDAR を積んだ車でドイツ・カールスルーエ周辺を走り集めたデータに、物体検出・追跡・深度推定・オドメトリなどのベンチマークを付け、自動運転の知覚研究を一気に加速させました。ここで「公式の評価指標と決まったデータレイアウト」という発想が定着します。

次の世代が nuScenes (Caesar et al., CVPR 2020, arXiv:1903.11027) と Waymo Open Dataset (Sun et al., CVPR 2020) です。nuScenes は6カメラ・1 LiDAR・5 レーダの全周センサ、1000シーン (各20秒)、約140万枚のカメラ画像、約140万の3Dバウンディングボックス (23物体クラス) という規模で、マルチカメラ・マルチモーダルの全周知覚を標準化しました。Waymo Open Dataset はさらに大規模で、1150シーン・1200万超の3Dラベル・高解像度 LiDAR を提供します。これらは公式 SDK (devkit) を同梱し、「SDK でデータを読む」流儀を確立しました。本ノートのスクリプトが SDK を前提にするのは、この系譜の直接の延長です。

最新世代は、より大規模で「Physical AI」「world model」を志向するデータ群です。実車に多数カメラを積み、数千時間〜数万時間規模の走行を集めて、end-to-end 自動運転や世界モデルの学習を後押しすることを狙います (大規模 AV データセットや、それを用いた基盤モデル群)。これらは容量が巨大 (1クリップで数GB、全体で数TB〜) なため、配布側は圧縮アーカイブのチャンク化と SDK による選択的ダウンロードを前提にし、利用者側は本ノートのような整形パイプラインを書くことになります。

## 論文の実験結果(定量データ)

データセット取得スクリプトに固有のベンチマークはありませんが、関連する定量的事実を示します。

データ規模が手法性能に与える影響は、自動運転で繰り返し確認されています。nuScenes 論文は、6カメラ全周入力と LiDAR の融合により、単一センサ・小規模設定に比べて3D物体検出の主要指標である mAP (mean Average Precision、検出の正確さを0〜1で測る指標。物体を正しい位置・正しいクラスで当てるほど高い) と、同データセット独自の NDS (nuScenes Detection Score、mAP に加えて位置・サイズ・向き・速度・属性の誤差を総合した0〜1の指標) が大きく向上することを報告しました。一般に、学習データを増やすほど検出・予測の性能が改善する傾向 (スケーリング則) は広く観測されており、だからこそ大規模データセットの取得・前処理が研究の生命線になります。

前処理の効率も定量的に効きます。1クリップが数GBあるデータセットでは、ダウンロードと展開だけで数分〜数十分かかります。`--clips 1` のような最小取得で動作確認できる設計があると、全量取得 (数時間〜数日、数TB) に踏み込む前にパイプラインの正しさを検証でき、無駄な転送コスト・ストレージコストを大幅に削減できます。仮に全量取得に丸2日かかるとして、レイアウトのバグに最後の1%で気づけば2日が無駄になりますが、1クリップ smoke test なら数分で同じバグを発見できます。これは「実験を回せるかどうか」を左右する実務的な定量効果です。

アブレーション的に言えば、「整形を省いて配布形式のままパーサに渡す」と、パーサは1ファイルも認識できずに 0 サンプルを返すか、ファイルが見つからずエラーになります。「ストリーム展開を省いて全部メモリに読む」と、巨大ファイルで OOM (Out Of Memory、メモリ不足によるプロセス強制終了) が起き、取得が完了しません。どの工程を抜くとパイプラインが成立しなくなるかは明確で、効果は「データが学習に使える形になるか否か」というオール・オア・ナッシングの形で現れます。

## メリット・トレードオフ・限界

メリット
- 公式 SDK 経由なので、認証・索引・キャッシュ管理・再試行を自前で書かずに済む
- 配布形式 (圧縮アーカイブ) をパーサが読める固定レイアウトへ自動変換し、再現可能にする
- 最小取得 (1クリップ) での smoke test ができ、全量取得の前に正しさを検証できる

トレードオフ・限界
- 1クリップで数GBと大きく、ダウンロードに時間とディスクを要する
- 利用にライセンス同意とアクセストークン (HuggingFace のアクセストークンなど) が前提のことが多く、誰でもすぐ動かせるわけではない
- アーカイブ内のファイル命名規則 (`.mp4` / `timestamp` を含む `.parquet` など) に依存しており、配布側が命名を変えると展開ロジックの見直しが必要
- 素朴な実装はアーカイブを一度メモリに全読みするため、巨大ファイルでメモリスパイクの懸念が残る (ストリーム展開で緩和、[一時ファイルへのストリーム展開でメモリ節約](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0838_stream-to-tempfile-extraction) 参照)

未解決の課題として、「異種データセット間でレイアウトとセンサ仕様 (カメラ台数・FOV・座標系・サンプリングレート・タイムスタンプの基準時刻) がバラバラ」という問題があります。データセットごとに専用の整形スクリプトを書く現状は労力が大きく、共通の中間表現 (canonical format) に一度変換してから学習に流す、という設計が望ましいものの、まだ業界標準は存在しません。Open X-Embodiment (Padalkar et al., 2023, arXiv:2310.08864) のような統合の試みはあるものの、自動運転の多センサ・多座標系を完全に吸収する標準には至っていません。

## 発展トピック・研究の最前線

データバージョニングと再現性は近年の重要トピックです。LakeFS や DVC (Data Version Control) のようなツールは、「どのデータのどのバージョンで学習したか」を Git のように管理します。前処理スクリプトの出力をバージョン管理すれば、過去の実験を正確に再現できます。データセットを差し替えたのに結果が再現できない、という事故を防ぐ実務的な基盤です。

ストリーミングデータセットも最前線です。数TBのデータをローカルに全展開せず、クラウドストレージから学習時に直接ストリーミングする設計 (WebDataset、MosaicML の StreamingDataset など) が広がっています。これにより、巨大データでも「全量を手元に置く」必要がなくなり、本ノートの整形パイプラインも「ローカル展開」から「ストリーミング読み出し」へと進化しつつあります。

world model (世界モデル、環境の遷移を予測する生成モデル) の学習は、こうした大規模マルチカメラ走行データの最大の応用先のひとつです。GAIA-1 (Hu et al., Wayve, 2023, arXiv:2309.17080) は約4700時間の走行映像から、テキスト・行動・画像を条件に「次に何が起きるか」を予測する生成モデルを学習し、現実的な運転シーンを自己回帰的に生成しました。NVIDIA の Cosmos (2025) のような world foundation model も、大量の映像から物理世界のダイナミクスを学び、自動運転やロボティクスのシミュレーション・データ拡張に使おうとしています。これらは膨大な動画クリップを必要とするため、本ノートで扱う「効率的なデータ取得・前処理」が性能を支える土台になります。
