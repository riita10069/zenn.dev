# 地図タイル/ズームレベル

## ひとことで言うと
世界地図を、ズームの段階(z)ごとに正方形の小さな画像「タイル」へ階層的に切り分けて配信する仕組みです。Google マップや OpenStreetMap で拡大すると地図が段階的に差し替わって細かくなるのは、この仕組みが裏で動いているからです。`z/x/y` という3つの数字で世界中の任意の場所・任意の縮尺の正方形1枚が一意に決まります。自動運転でラスタ地図を作るとき、「自車の周り何メートルを・1ピクセル何メートルで描くか」という設計が、この仕組みのズームレベルと数学的に対応します。

## 直感的な理解
世界全体の地図を1枚の画像で持とうとすると、街路1本まで見えるほど細かくした場合のファイルサイズは天文学的になります。一方で利用者が一度に見たいのは普段「ある場所の周辺だけ」です。膨大な全体を作り込んでおきながら、実際に取り出すのはごく一部、という状況です。

ここで人類が地図でずっと使ってきた発想が効きます。世界地図・国の地図・市街図というように、見たい縮尺に応じて別の地図を用意し、必要なページだけ開く。デジタル地図はこれを徹底し、各縮尺の地図をあらかじめ正方形の断片(タイル)に切り分けておきます。画面に映る範囲に必要なタイルだけをサーバから取り寄せて並べれば、巨大な地図を扱いながら通信も表示も軽量に保てます。拡大・移動のたびに必要なタイルだけ差し替えるので、いわゆる「ぬるぬる動く(slippy)地図」が実現します。

## 基礎: 前提となる概念

ズームレベル(zoom level、略して z): 全体をどのくらい拡大して見るかの段階を表す整数です。z=0 が「世界全体を1枚に収めた状態」で、z が1増えるごとに縦も横も2倍ずつ細かくなります。

タイル: 各ズームレベルで地図を正方形(慣習的に256×256ピクセル、近年は512も)に格子状に切り分けた断片です。各タイルは縦位置 y と横位置 x で番地が振られます。

投影(projection): 丸い地球の表面を平らな地図に引き伸ばす方法です。地球は球(正確には回転楕円体)なので、平面に写すと必ずどこかが歪みます。何を犠牲にして何を保つか(面積・角度・距離のどれを正確にするか)で投影法が分かれます。

これらを組み合わせると、`z/x/y` を指定すれば「どのズームで・どの位置の・正方形1枚か」が一意に決まります。この方式は Slippy Map(スリッピーマップ)、または OSM(OpenStreetMap)タイルスキームと呼ばれ、Web 地図の事実上の標準です。

## 仕組みを詳しく

### タイルの枚数

ズームレベル z では、地図は縦横それぞれ `2^z` 枚に分割されます。全体のタイル枚数は `2^z × 2^z = 4^z` 枚です。

- z=0: 1×1 = 1枚(世界全体が1枚の256×256画像)
- z=1: 2×2 = 4枚
- z=2: 4×4 = 16枚
- z=10: 1024×1024 = 約100万枚
- z=18: 262144×262144 = 約690億枚(市街地レベルの詳細)

z が1上がるごとに縦も横も2倍細かくなり、1タイルが覆う実距離は半分になります。タイル枚数は4倍ずつ増えるため、高ズームの全世界タイルを事前生成・保存するのは膨大なコストになります(実運用では人がよく見る地域だけ高ズームを用意する)。

### 座標の決め方(Web メルカトル投影)

タイルの基盤は Web メルカトル投影(Web Mercator、EPSG:3857 というコードで参照される)です。メルカトル投影は角度(方位)を保つ正角図法で、世界をほぼ1つの正方形に収めます(緯度はおよそ ±85.05度で切る。これは正方形に収めるための都合の数字です)。緯度 lat(度)、経度 lon(度)から、ズーム z でのタイル番号 (x, y) を求める式は次の通りです。

```
x = floor( (lon + 180) / 360 * 2^z )
y = floor( (1 - ln( tan(lat_rad) + 1/cos(lat_rad) ) / pi) / 2 * 2^z )
```

`floor` は小数点以下切り捨て、`lat_rad` は緯度をラジアンにしたもの、`ln` は自然対数、`pi` は円周率です。式の全部を理解する必要はなく、要点は「緯度経度を入れると、そのズームでの正方形の番地が決まる」ことです。x は経度に線形対応し、y は緯度に対して非線形(高緯度ほど引き伸ばされる)になっているのがメルカトルの特徴です。

### 1ピクセルが何メートルか(ground resolution)

タイルが256×256ピクセルだとすると、緯度 φ での1ピクセルの実距離(ground resolution、地上解像度)はおよそ次式です。

```
解像度[m/px] ≒ 156543.03 / 2^z * cos(φ)
```

先頭の 156543.03 は「赤道全周(約 4007.5万m)÷ (256 × 2^0)」に相当する定数です。赤道上(cos φ = 1)で z ごとに計算すると、おおよそ次の通りです。

- z=10: 約153 m/px(街全体が見える)
- z=15: 約4.8 m/px(街区が見える)
- z=18: 約0.6 m/px(個々の建物・道路が見える)
- z=19: 約0.3 m/px
- z=20: 約0.15 m/px

自動運転のラスタBEV地図([ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction))で「1ピクセル=0.2〜0.3m」を使いたいなら、z=18〜19 相当の詳細さが必要、という対応関係が読み取れます。逆に言えば、ラスタBEVで使う解像度・範囲を決めることは、暗黙のうちにズームレベルを選んでいるのと同じです。

### z/x/y の使われ方

地図配信サーバは `https://tile.example.com/{z}/{x}/{y}.png` のような URL でタイル画像を返します。クライアントは画面に映る範囲に必要なタイルだけを計算して取りに行き、並べて表示します。一度取ったタイルはキャッシュ(再利用のため手元に保存)できるので、同じ場所を再訪したときの通信を省けます。

なお y 軸の向きには2系統あり、Google/OSM の標準(XYZ)は上が y=0 ですが、地理空間業界の TMS(Tile Map Service)規格は下が y=0 です。両者は `y_tms = 2^z - 1 - y_xyz` で変換でき、混同するとタイルが上下逆になるという実務上のはまりどころです。

### 階層構造としてのピラミッド(LOD)

タイルセット全体は「ミップマップ(mipmap)」と同じピラミッド構造をなします。ミップマップとはコンピュータグラフィックスで使う概念で、同じ画像を解像度の異なる版で何段も用意しておき、表示に必要な段だけ使う技法です。地図タイルでは z=0 が頂点(粗い1枚)、z が増えるほど底辺が広がる(細かく多数)四分木(quadtree)になっており、z レベルの1枚のタイルは z+1 レベルの2×2=4枚にちょうど分割されます。この性質を使うと「z レベルのタイルの番地 (x, y) は、子の4枚を (2x, 2y), (2x+1, 2y), (2x, 2y+1), (2x+1, 2y+1) として再帰的にたどれる」ことが分かります。Microsoft Bing Maps は各タイルにこの四分木の経路を base-4 の文字列で符号化した「クアッドキー(quadkey)」という1次元の番地を与え、`z/x/y` の3値を1つの文字列に畳み込んでキャッシュやデータベースのキーにしやすくしています。自動運転のラスタBEV地図でも、解像度を半分ずつ落とした複数版を持ってマルチスケール特徴を作る設計はこのピラミッドの発想と同型です。

## 手法の系譜と主要論文

タイル/ズームの仕組みは学術論文の提案手法というより、地図配信の業界標準(デファクトスタンダード)です。系譜は次の通りです。

- メルカトル投影(Gerardus Mercator, 1569): 大航海時代に、羅針盤の方位(等角航路)を直線で描けるよう作られた正角図法。角度を保つ代わりに高緯度の面積を著しく拡大する(グリーンランドがアフリカ並みに見えるのは有名な歪み)。
- Web メルカトル / EPSG:3857: Google マップが2005年に採用し、計算の単純さと正方形タイルへの収まりやすさから、OpenStreetMap など多くの Web 地図が追従。地図学的には球を球面メルカトルではなく楕円体パラメータで割り切るなど厳密でない点があり、当初は正式な EPSG コードが与えられず議論を呼んだが、結局事実上の標準になった。
- Slippy Map タイルスキーム: OpenStreetMap プロジェクトが整理・普及させた `z/x/y` の命名規則。OSM Wiki の "Slippy map tilenames" に仕様がまとまっている。
- ベクトルタイル(Mapbox Vector Tile, MVT, 2014頃〜): ピクセル画像ではなく、タイル単位でベクトルデータ(道路ポリラインなど)を配信する後継規格。クライアント側で任意の見た目に描画でき、回転・スタイル変更が自由。自動運転の地図前処理でベクトル道路網を取得する文脈で関係する。

自動運転・予測の論文では、地図データの取得元として OpenStreetMap を使う研究が多く、その背後でこのタイルスキームが動いています。論文本文でタイルの数式まで解説することは稀で、前処理の一部として扱われるのが普通です。

## 論文の実験結果(定量データ)

タイルスキーム自体に「精度何%向上」という実験結果はありません。代わりに、この仕組みが自動運転研究に効いてくる定量的な接点を挙げます。

ラスタBEVを使う予測・計画の論文では、入力地図の解像度(m/px)と範囲(m)がモデル性能とコストに直結します。一般に、解像度を細かくする(ズームを上げる)と遠方や細部の予測誤差(minADE/minFDE などの displacement error。小さいほど良い)は下がる傾向がありますが、画像サイズが二乗で増えるため計算量(FLOPs)とメモリが急増します。例えば範囲を一定にして 0.4 m/px から 0.2 m/px に上げると、画素数は4倍、畳み込みの計算量もおおむね4倍になります。これは z を1つ上げてタイルを4倍にするのと同じ構造のトレードオフです。ラスタBEVの代表的な実装研究では、この解像度依存性がアブレーションで報告され、「ある程度より細かくしても精度はサチるのに計算だけ増える」飽和点の存在が示されています([ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction) のベクトル系移行の動機にもなった)。

つまり、ズームレベルの数学は「どこまで細かくすれば精度が見合うか」というモデル設計の意思決定に、定量的なものさしを与えます。

## メリット・トレードオフ・限界

メリット
- 巨大な世界地図を、必要な部分だけ軽量に取り出せる(階層 LOD による帯域節約)。
- `z/x/y` でキャッシュが効き、再利用が容易。
- ズームレベルが、ラスタ地図の解像度・範囲設計とそのまま対応するので見通しが良い。
- 業界標準なので対応サーバ・ライブラリが豊富。

トレードオフ・限界
- Web メルカトルは高緯度ほど面積が歪む。広範囲を実距離で正確に扱う用途(計測・面積計算)には不向きで、ローカルなメートル系(UTM など)への変換が要る。
- 極付近(±85.05度の外)は表現できない。
- ズームを1つ上げるとタイルが4倍に増え、事前生成・保存コストが爆発する。
- 固定の正方形格子・北固定なので、自車中心・進行方向上向きのような任意の向き・中心の切り出しには、別途座標変換([自車座標系へのラスタ化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0415_ego-pose-rasterization-coord))が必要。
- ラスタタイルは見た目が焼き込み済みで、スタイル変更や回転に弱い(この弱点を解いたのがベクトルタイル)。

## 発展トピック・研究の最前線

近年は、ラスタタイルからベクトルタイル(MVT)・3Dタイル(Cesium 3D Tiles, glTF ベース)への移行が進み、自動運転 HD マップの配信でも階層化・タイル化の発想が使われています。研究の最前線では、HD マップそのものを車載センサーからオンラインで生成する online HD map construction(HDMapNet, MapTR など)が活発で、ここでもBEVグリッドの解像度・範囲という、本質的にタイル設計と同じパラメータが性能を左右します。「地図をどの縮尺・どの範囲で持つか」という問いは、配信の世界からニューラルネットの設計の世界へ場所を移しながら、形を変えて生き続けています。
