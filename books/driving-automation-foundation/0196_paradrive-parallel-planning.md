# ParaDrive / 並列パラダイム

## ひとことで言うと
UniAD のような End-to-End 自動運転は「認識 → 予測 → 計画」をクエリ(情報を取りに行く質問ベクトル)で一直線につなぎます(逐次=シーケンシャル)。ParaDrive(パラドライブ、PARA-Drive)は「本当にその順番でつなぐ必要があるのか?」という疑問から出発し、全モジュールを横並び(並列=パラレル)で動かす構成を含めて、End-to-End 設計の選び方(どこを繋ぎ、どこを切るか)を体系的に実験で比較した研究です。結論として「思っているほど逐次接続は計画性能に効いていない」こと、そして並列化すれば計画性能を保ったまま大幅に高速化できることを示し、計画モジュールをどう設計すべきかの実証的な指針を与えました。

## 直感的な理解
UniAD(→ [UniAD プランニングヘッド](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0209_uniad-planning-head))が「計画指向で全段をクエリで逐次に繋ぐ」設計で大成功し、その後の End-to-End 自動運転は「とにかく全部を一直線に繋いで共同学習するのが正解」という空気になりました。しかし、ここには検証されていない思い込みがありました。

「逐次接続(認識の出力を予測へ、予測の出力を計画へ、と順に渡す)が本当に性能に効いているのか?」「それとも、共有する BEV 特徴(上空視点の地図状の特徴)さえあれば、各モジュールは独立に動かしても計画は良いのか?」

UniAD の設計は複雑で計算も重い。しかも逐次だと、計画ヘッドが動くには認識・予測の全段が先に終わるのを待たねばならず、推論が遅くなります。もし逐次接続が実は効いていないなら、もっと単純で速い構成で同じ性能が出せるかもしれません。ParaDrive の問題意識は「End-to-End の設計選択を、思い込みではなく実験で一つずつ確かめよう」という、極めて実証的なものです。新しい派手な手法を出す論文ではなく、構成要素を一つずつ外して効果を測る アブレーション(ablation)中心の分析研究という点が特徴です。これは物理実験で「どの変数が結果を決めているか」を切り分ける作業に似ています。

## 基礎: 前提となる概念
逐次(sequential)接続: 認識の出力を予測の入力に、予測の出力を計画の入力に、と前段の結果を次段へ順に渡す方式です。情報が一本道で流れます。UniAD がこの代表です。利点は、後段が前段の整理済み情報を使えること。欠点は、前段が詰まると後段が待つこと(レイテンシが直列に積み上がる)、前段の誤りが後段へ伝播すること、構成が複雑になることです。

並列(parallel)接続: 認識・予測・計画の各モジュールが、共有する BEV 特徴を「それぞれ独立に」読みに行く方式です。モジュール間の直接の受け渡しを減らし、計画ヘッドも BEV 特徴から直接、自車軌道を出します。利点は、モジュールが互いに待たなくてよいので推論を並列に走らせて速くできること、依存が減って構成が単純になることです。さらに、推論時には必要なモジュール(計画ヘッド)だけ動かし、認識・予測のヘッドは省略するという選択も可能になります(学習時だけ補助として使う)。

教師信号(supervision): あるモジュールに「正解」を与えて学習させることです。たとえば認識モジュールに「実際の物体位置」を教師として与えれば、認識は物体を当てるよう学習します。ここで論点になるのは、認識・予測の教師信号が「計画性能」にどう効くか、です。計画に直接繋がっていなくても、共有 BEV 特徴を意味的に良くする副作用(補助タスクとしての効果)で計画を助ける可能性があります。

開ループ評価の偏り: 記録データに対して軌道の正解との近さを測る開ループ評価は、nuScenes のような直進主体のデータでは「ほぼ直進すれば良いスコアが出る」性質を持ちます。AD-MLP や BEV-Planner が示したとおり、自車状態(速度・加速度)だけで高得点が出てしまう。設計の優劣を論じる土台がこの偏った指標だと、結論の解釈に注意が要ります。

## 仕組みを詳しく
ParaDrive が比較した「設計の軸」を整理します。共通の土台は「複数カメラ → BEV 特徴」で、ここは各構成で共有します(UniAD と同じ BEVFormer 系のエンコーダを使う)。違うのは認識・予測・計画モジュールの「つなぎ方」です。

```
逐次(UniAD型):  BEV → 認識 → 予測 → 計画 → 自車軌道
                       (各段の出力が次段の入力。情報は一本道。計画は全段を待つ)

並列(ParaDrive): BEV ─┬→ 認識(教師信号で特徴を磨く)
                       ├→ 予測(教師信号で特徴を磨く)
                       ├→ 占有(教師信号で特徴を磨く)
                       └→ 計画 ← BEV を直接読む → 自車軌道
                       (全ヘッドが BEV から同時に。計画は他段を待たない)
```

軸1 モジュール間の接続(逐次 vs 並列)。逐次は前段出力を次段へ渡す。並列は各モジュールが共有 BEV を独立に読み、計画ヘッドも BEV から直接軌道を出す。並列ではすべてのヘッドが同じ BEV を入力に同時並行で動くため、レイテンシは「最も遅い1ヘッド分」で済みます。

軸2 計画ヘッドが何を入力にするか。認識・予測の結果(エージェントクエリ等)を入力にするか、それとも BEV 特徴だけを入力にするか。ここを切り替えて「計画にとって本当に必要な入力は何か」を測る。

軸3 各タスクの教師信号の有無。認識・予測・占有に教師ラベルを付けて学習させるか、付けないか。「計画性能に、認識・予測の教師信号がどれだけ寄与するか」を測る。

これらを組み合わせて多数の構成を作り、同じデータ(nuScenes)・同じ評価指標で計画性能を比べるのが ParaDrive の中身です。重要なのは、比較を公平にするため土台(BEV エンコーダ)と学習条件を揃え、変える軸だけを変える点です。これが「どの設計要素が効くか」を切り分ける鍵になります。

主要な発見。第一に、並列構成でも逐次構成(UniAD 型)に匹敵する計画性能が出ました。つまり「逐次に繋ぐこと自体」は計画にとって決定的ではありませんでした。第二に、計画ヘッドにとって最も効くのは、豊かな BEV 特徴そのものへの直接アクセスでした。認識・予測の中間出力を律儀に受け渡すことの寄与は、思われていたより小さかった。第三に、認識・予測・占有の教師信号(マルチタスク学習)は、BEV 特徴を意味的に良くする補助としては有効でした。「計画に直接つなぐ」のではなく「特徴を良くする副作用」として効く、という整理です。第四に、これらのヘッドは学習時にだけ必要で、推論時には計画ヘッドだけ動かせばよいため、並列構成は推論を大幅に高速化できました。

これらは「計画モジュールをどう設計すべきか」の具体的指針になります。たとえば「複雑な逐次パイプラインを組む前に、まず BEV 特徴を良くして計画ヘッドに直接渡す並列構成を試せ」「認識・予測ヘッドは学習時の補助と割り切り、推論では落として高速化せよ」というメッセージです。

## 手法の系譜と主要論文
ParaDrive(PARA-Drive: Parallelized Architecture for Real-time Autonomous Driving)は Xinshuo Weng らによる CVPR 2024 の研究で、NVIDIA の研究者が関わっています。接続方法(逐次/並列)・計画ヘッドの入力・各タスクの教師信号という設計軸を網羅的にアブレーションして比較し、並列パラダイムを掲げました。タイトルにある "Real-time"(実時間)が示すとおり、性能を保ったまま推論速度を上げることを実用上の主眼に置いています。

この研究の背景には、同時期に相次いだ「End-to-End の前提を疑う」研究群があります。AD-MLP / "Rethinking the Open-Loop Evaluation ..."(Zhai et al., 2023, arXiv:2305.10430)は、自車の状態(速度・加速度など)だけを入れた単純な MLP でも nuScenes 計画指標が高く出ると示し、開ループ評価の限界を突きました。BEV-Planner / "Is Ego Status All You Need for Open-Loop End-to-End Autonomous Driving?"(Li et al., 2024, arXiv:2312.03031)も同様に、自車状態への依存が計画スコアを支配していることを指摘しました。ParaDrive はこの「設計と評価を実験で問い直す」系譜に属し、特に「モジュールの接続トポロジ(つなぎ方の形)」という設計軸に光を当てた点に新規性があります。

源流の逐次・計画指向の代表は UniAD(→ [UniAD プランニングヘッド](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0209_uniad-planning-head))で、ParaDrive はこれを主要な比較対象としました。軽量化の流れ(VAD / VADv2)も「設計を単純にして速く・強くする」という点で問題意識を共有します。ParaDrive はそれらと並び、End-to-End 設計の「常識」を実験で再点検する重要な一里塚になりました。

## 論文の実験結果(定量データ)
評価は nuScenes の計画指標、すなわち L2 誤差(予測軌道と正解軌道の座標距離。メートル。小さいほど良い)と衝突率(予測軌道に沿うと他物体に衝突する割合。低いほど安全)で行われました。同時に、推論速度(FPS = 1秒あたりの処理フレーム数。大きいほど速い)も主要な評価軸です。

中心的な定量結果は二つです。一つ目は、並列構成が逐次構成(UniAD 型)と同等の L2・衝突率を、より単純な接続で達成したこと。つまり接続トポロジを逐次から並列に変えても計画性能はほぼ変わらず、逐次接続のために払う複雑さのコストが見合わないことが数値で示されました。二つ目は、その並列構成が UniAD に対して大幅に高い FPS を達成したことです(報告では UniAD 比で数倍の推論速度)。計画性能を保ったまま実時間性を大きく改善できた、というのが PARA-Drive の実用的な売りです。アブレーションでは、(a) 計画ヘッドの入力を認識・予測クエリから BEV 特徴へ切り替えても計画性能が落ちない(むしろ BEV 直アクセスが効く)、(b) 認識・予測・占有の教師信号を外すと計画性能がやや低下する(特徴の意味付けとして効いている)、(c) 推論時に補助ヘッドを落としても計画性能はほぼ保たれる、という分解結果が得られました。

ただし、これらの数値はすべて nuScenes の開ループ評価に基づきます。AD-MLP の指摘どおり、この指標は直進主体のシーンで飽和しやすく、絶対値の差が小さく出やすい性質があります。ParaDrive 自身もこの限界を認識しており、計画性能の結論は「開ループでは逐次接続の優位は確認できない」という慎重な表現になります。閉ループ(実際に走らせる)評価での再検証が今後の宿題として残されました。一方で「並列化による高速化」という結論は評価プロトコルの偏りに左右されにくく、実装上の堅い貢献として残ります。

## メリット・トレードオフ・限界
メリット。設計の思い込み(逐次接続が必須)を実験で否定し、より単純・高速な並列構成の余地を示した。「計画には BEV 特徴への直接アクセスが効く」「認識・予測の教師信号は特徴を磨く補助として効く」という具体的な設計指針を提供した。並列化と推論時のヘッド省略により、計画性能を保ったまま実時間性を大幅に改善できた。研究コミュニティに「設計の前提を実験で確かめる」姿勢の重要性を示した。

トレードオフと限界。第一に、計画性能の結論が主に nuScenes の開ループ評価に依存します。開ループ指標は直進で高得点になりがちで、計画の真価(危険回避や反応的判断)を測れません。閉ループ性能では結論が変わる可能性があります。第二に、「並列が有利」という結論は評価設定・データ・BEV エンコーダの選択に依存し、別ドメイン(都市部の密な交差点、悪天候、別センサー構成)で同じとは限りません。第三に、これは派手な新手法ではなく分析中心の研究なので、そのまま使える完成モデルというより「設計判断の指針」として使うものです。第四に、逐次接続が効かないという結論は「開ループでは見えない」だけで、反応的な閉ループ場面では前段からの明示的な物体情報が効く可能性を否定しきれません。第五に、推論時に認識・予測ヘッドを落とすと、それらの解釈可能な中間出力(誰がどこにいるか)も失われ、デバッグや安全検証の手がかりが減ります。

未解決課題としては、(1) 閉ループ・反応的ベンチマークでの接続トポロジの再評価、(2) 並列構成が有効/逐次が必要になる場面の切り分け(どんなシーンで明示的な物体受け渡しが効くか)、(3) 評価指標そのものの改善(開ループの飽和を避ける設計)、(4) 速度と解釈可能性のトレードオフ(補助ヘッドを落とす代償)の最適化、が挙げられます。

## 発展トピック・研究の最前線
ParaDrive と AD-MLP / BEV-Planner 系の「設計と評価を疑う」研究は、End-to-End 自動運転の評価パラダイムを変えました。nuScenes 開ループの限界が広く認識され、NAVSIM(非反応的だが衝突を正しく測る)や CARLA / nuPlan の閉ループ評価へ重心が移りつつあります。設計面では、計画ヘッドに何を入力すべきか(BEV 直接か、明示的な物体・占有か)という問いが、語彙ベース計画(VADv2)や多目標蒸留(Hydra-MDP)といった後続手法の設計に影響しています。ParaDrive が体系化した「接続トポロジ・入力・教師信号」という三軸のアブレーション設計は、新しい End-to-End アーキテクチャを提案する際の標準的な検証フレームとして参照されています。実時間性という観点では、並列化に加え、軌道を候補集合から選ぶことで重い回帰デコーダを省く語彙ベース計画とも相性がよく、実車適用に向けた効率化研究の流れに合流しています。
