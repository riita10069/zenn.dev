# ロボット学習データセットライブラリの API 移行

## ひとことで言うと
ロボットや自動運転の学習データを読み込むためのライブラリは、バージョンアップのたびに関数名・import (読み込み) パス・オプション引数の意味が変わることがあります。新しいバージョンに合わせて自分のコードを直す作業を「API 移行 (migration)」と呼びます。これを怠ると、昨日まで動いていたコードが「そんな関数はない」というエラーで突然動かなくなり、最悪の場合はエラーすら出ずに間違ったデータで学習してしまいます。本ノートは、研究を再現可能に保つために避けて通れないこの実務作業を、初心者にも分かる一般論として解説します。

## 直感的な理解
他人が書いた料理レシピ集 (ライブラリ) を使って料理 (データ読み込み) をしているとします。あるとき著者がレシピ集を改訂し、「砂糖」という材料名を「グラニュー糖」に変え、章立ても並べ替えたとします。あなたの台所メモ (自分のコード) には「砂糖を3章から取る」と書いてあります。改訂版を買ったのに古いメモのまま作業すると、「3章に砂糖が見つからない」と詰まります。さらに悪いケースとして、章立てだけ変わって「3章」には別の材料 (塩) が入っていたら、エラーも出ずに塩で料理を作ってしまい、できあがって初めて失敗に気づきます。

ソフトウェアでもまったく同じことが起きます。ライブラリの「呼び出し方の決まりごと」を API (Application Programming Interface) と呼びます。これは「ライブラリと自分のコードのあいだの約束」です。ライブラリ側がこの約束を変えると、約束に依存していた自分のコードは壊れます。API 移行とは、新しい約束に合わせてメモを書き直す作業です。

機械学習の世界では、データセットライブラリは特に進化が速い領域です。研究が日進月歩で、扱うデータ形式 (画像・動画・点群・センサ時系列) も増え続けるため、ライブラリ作者は頻繁に内部設計を作り直します。その結果、利用者は「論文の手法そのもの」とは別に、こうした追従コストを定期的に払うことになります。地味ですが、実験を再現可能に保つうえで本質的な作業です。

## 基礎: 前提となる概念

ここで使う用語を、知らない人向けにひとつずつ噛み砕きます。

ライブラリ (library) とは、誰かが作った「再利用できるコードの部品集」です。Python では `import` という文で「この部品を使いたい」と宣言します。たとえば `from foo.bar import Baz` は「foo の中の bar というファイルから Baz という部品を持ってくる」という意味です。この `foo.bar` の部分を import パス (どこから読むか) と呼びます。

クラス (class) とは、データと処理をひとまとめにした設計図です。データセットを表すクラスは典型的に `Dataset` という名前を持ち、「i 番目のサンプルをくれ」と頼むと観測 (画像やセンサ値) と行動 (操作) のペアを返してくれます。PyTorch では `torch.utils.data.Dataset` がその基底になっており、`__len__` (全部で何件か) と `__getitem__` (i 番目をくれ) という2つの約束 (メソッド) を満たすことが期待されます。この2つを実装したオブジェクトは `DataLoader` に渡してバッチ化・並列読み込みできます。

ロボット学習データセットライブラリ (代表例が HuggingFace の LeRobot、Cadene らが2024年に公開) は、「観測と行動のペアを時系列で保存・共有する標準フォーマット」と、それを読み書きするツールを提供します。狙いは、実機ロボットや自動運転車から集めた走行記録 (カメラ画像・速度・ステアリング操作など) を、誰でも同じ形式で再利用できるようにすることです。フォーマットを標準化すると、ある研究者が集めたデータを別の研究者がそのまま学習に使える、という再現性・再利用性の利点が生まれます。代わりに、フォーマットが進化するたびに本ノートのような追従作業が発生する、というトレードオフを抱えます。

セマンティックバージョニング (semantic versioning、SemVer) という慣習も押さえておきましょう。バージョン番号は `MAJOR.MINOR.PATCH` (例 `0.5.2`) の3つの数字でできています。SemVer の規約では、一番右 (PATCH) はバグ修正のみ、真ん中 (MINOR) は後方互換な機能追加、左 (MAJOR) は後方互換を壊す変更、と決められています。ところが一番左の数字が `0` のうちは「まだ安定版でない (初期開発段階)」という特別扱いで、SemVer 仕様自体が「`0.x` ではどんな変更が入ってもよい」と明記しています。つまり `0.4.x` から `0.5.x` のようにマイナーが上がるだけで破壊的変更 (breaking change、既存コードを壊す変更) が入ることが規約上も許容されています。研究用ライブラリの多くは長く `0.x` に留まるため、利用者は破壊的変更に常に身構える必要があります。

## 仕組みを詳しく

データセットライブラリのバージョンアップで起きる破壊的変更は、いくつかの典型パターンに分類できます。代表的な3つを順に見ます。

### パターン1: import パスの再編

ライブラリが成長して内部ファイルが増えると、作者は名前空間 (どのファイルにどの部品を置くか) を整理し直します。よくあるのが「中間の階層を1つ削る」変更です。研究系ライブラリでは、初期に置いた `common` のような曖昧な中間階層が成長後に不要になり、削除されることがよくあります。

```
before (旧バージョン):
  from somelib.common.datasets.dataset import RobotDataset
              ^^^^^^^ この common 階層が

after (新バージョン):
  from somelib.datasets.dataset import RobotDataset
              ^^^^^^^ 消えて短くなった
```

古い import 文を実行すると、Python は次のようなエラーを出して停止します。

```
ModuleNotFoundError: No module named 'somelib.common'
```

これは「`somelib.common` という場所が見つからない」という意味です。対処は単純で、import パスを新しいものに書き換えるだけです。ただし、環境によって複数のインストール形態 (本体パッケージか、依存を削った単体版か) が混在しうる場合は、両方を試す防御的な書き方が有効です。

```python
try:
    from somelib.datasets.dataset import RobotDataset
except ModuleNotFoundError:
    from somelib_lite.datasets.dataset import RobotDataset
```

これは「まず新パスから読み込み、無ければ代替パスから読み込む」という二段構えです。どちらのインストール形態でも動く頑健性が得られます。ただし副作用として、本物のタイポ (打ち間違い) によるエラーまで `except` 側で握りつぶしてしまう危険がわずかにあります。例外の捕捉は範囲を最小に絞り、`ModuleNotFoundError` のように具体的な例外型だけを捕まえるのが定石です (広い `except Exception:` は禁物)。

### パターン2: オプション引数の削除と意味の変化

ライブラリは「初期化時に渡すオプション」も作り変えます。たとえば旧バージョンに `local_files_only=True` (インターネットから取りに行かず手元のキャッシュだけ使う) というオプションがあったとします。キャッシュ (cache) とは、一度ダウンロードしたデータを手元に保存して再利用する仕組みです。

新バージョンでこのオプションが削除され、「デフォルトでキャッシュを使い、明示的に `force_sync=True` を指定したときだけ再ダウンロードする」という挙動に変わったとします。つまり「ローカル優先」が標準動作になり、わざわざオプションで指定する必要がなくなったわけです。これは挙動としてはむしろ改善ですが、引数の名前と存在自体が消えたので、古い呼び出しは壊れます。

このとき、呼び出し側のコードがまだ `local_files_only=...` を渡していると、新バージョンは「そんな引数は知らない」と次のエラーを出します。

```
TypeError: __init__() got an unexpected keyword argument 'local_files_only'
```

移行の選択肢は2つあります。

1. 呼び出し側からそのオプションを完全に削除する (最もクリーン)。
2. 後方互換のため、自分のラッパークラス (新ライブラリを内部で包む薄い自前クラス) でその引数を受け取りつつ、内部では新ライブラリに渡さない (no-op、何もしない) ようにする。

```python
def __init__(self, repo_id, episodes=None, local_files_only=False):
    # local_files_only は後方互換のダミー引数。新APIには渡さない。
    self.dataset = RobotDataset(repo_id=repo_id, episodes=episodes)
```

2の方式は、自分のラッパーを使う既存コード (テストや他のスクリプト) を壊さない利点があります。代わりに「`local_files_only=True` を渡したのに実際は何もしていない」という認知のズレを生むため、ドキュメントや docstring に「この引数は no-op です」と明記して緩和します。

```
before (旧API):
  RobotDataset(repo_id=..., episodes=..., local_files_only=True)
after (新API):
  RobotDataset(repo_id=..., episodes=...)   # local_files_only は廃止
```

### パターン3: メタ情報構造の置き換え

最も厄介なのが「便利だった属性が別の構造に置き換わり、しかも中身の意味も微妙に変わる」パターンです。たとえば「各エピソードの開始・終了行を持つ便利な属性 (`episode_data_index` のようなもの)」が削除され、別のメタ情報オブジェクト (`meta` のような器) に移動し、保持する行番号の基準 (全体での通し番号=グローバルか、手元での番号=ローカルか) まで変わる、というケースです。

これが恐ろしいのは、import は通り、`TypeError` も出ず、プログラムが最後まで走ってしまう点です。返ってくる行番号がグローバルのままなのに、一部だけ読み込んだ手元の表ではローカル番号でアクセスする必要がある、という食い違いに気づかないと、まったく別のフレームを読んで学習が静かに壊れます (silent failure)。単なる import 書き換えでは済まず、データの意味論を理解したうえで導出ロジックを自前で書き直す必要があります。この具体例は [エピソードのローカル行レンジ算出 (添字空間の整合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0814_episode-local-row-ranges) で詳しく扱います。

## 手法の系譜と主要論文

API 移行そのものは学術的新規性のある「手法」ではありません。しかし、その背景にある「データセットを標準フォーマットで共有する」という発想には明確な系譜があります。

模倣学習 (imitation learning、人間の操作を教師にネットに真似させる学習) と行動クローニング (behavior cloning) の源流は、Pomerleau による ALVINN (Autonomous Land Vehicle In a Neural Network, 1989, NeurIPS) にさかのぼります。これはカメラ画像を入力に、人間の運転操作を教師として、3層のニューラルネットに運転を真似させた先駆的研究です。当時はデータ共有の文化もフォーマットもなく、各研究室が独自にデータを集めていました。

時代が下り、深層学習が自動運転に本格適用されると (Bojarski et al. "End to End Learning for Self-Driving Cars", NVIDIA, 2016, arXiv:1604.07316)、大規模な走行データの重要性が認識されます。やがて KITTI (Geiger et al., CVPR 2012)、nuScenes (Caesar et al., CVPR 2020, arXiv:1903.11027)、Waymo Open Dataset (Sun et al., CVPR 2020) のような、公式パーサ (devkit) 付きの大規模公開データセットが登場し、「データセットには専用のロード API が付属する」のが当たり前になりました。

HuggingFace の LeRobot (Cadene et al., 2024) は、この流れをロボット・自動運転の模倣学習に持ち込んだものです。観測と行動の時系列を統一フォーマット (内部的には Apache Arrow による表 + mp4 動画ファイル + 統計メタ情報) で保存し、PyTorch から直接学習できるツールを提供します。さらに L2D (Learning to Drive、yaak-ai が2024-2025年に公開) のような、欧州の運転教習車から集めた大規模実走行データ (数千時間規模) がこのフォーマットで配布されるようになり、エコシステムが形成されました。LeRobot はこのエコシステムの中心として活発に開発が続いており、バージョンごとにフォーマット (v2.0、v2.1 など) と Python API の両方が更新されています。だからこそ利用者の API 移行が構造的に発生し続けます。

この系譜が示すのは、「データを誰でも同じ形式で再利用できるようにする」ことが、再現可能な研究の基盤として年々重視されてきた、という大きな流れです。

## 論文の実験結果(定量データ)

API 移行そのものに「精度がどれだけ上がった」という指標はありません。代わりに、移行を支える「標準データフォーマット」の効果を測った定量的な議論を紹介します。

データセット標準化の効果を測る代表的な切り口は「再現性 (reproducibility)」と「データ供給スループット (samples/sec)」です。

再現性については、機械学習の再現性危機を扱った研究 (Pineau et al. "Improving Reproducibility in Machine Learning Research", JMLR 2021) が、NeurIPS で導入された再現性チェックリスト (Reproducibility Checklist) によって、コードを公開した投稿の割合が大きく増えたことを報告しています。同チェックリスト導入後、論文に再現に必要な情報を添える文化が定着し、コード提供率は数年で大幅に上昇しました。データフォーマットが標準化されていれば、論文の数値を他者が追試できる確率が上がる、という定性的に重要な効果があります。

データ供給スループットについては、後続ノート [状態量の高速読み出し (動画デコード回避)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0847_vehicle-state-fast-path) で扱う「列指向フォーマットからの選択的読み出し」が、1サンプルあたりの読み出し時間を桁違いに短縮する例 (報告によっては数十秒から実質ゼロへ) を示します。標準フォーマットが Apache Arrow のような効率的な基盤の上に作られていることが、こうした最適化を可能にします。

アブレーション的な観点で言えば、「import パスだけ直してメタ情報の意味変化を見落とす」と、エラーは出ないのに読み込むデータが静かに壊れる (silent failure) という最悪のケースが起こります。これは [エピソードのローカル行レンジ算出 (添字空間の整合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0814_episode-local-row-ranges) の「グローバル行番号とローカル行番号の取り違え」がまさにそれで、「どの要素を抜くと何が壊れるか」を理解せず移行すると、学習結果が説明不能に悪化します。逆に言えば、移行作業のうち「import パス変更」「引数削除」は失敗すれば即エラーで気づける安全な変更で、「メタ情報の意味変化」だけが沈黙して壊れる危険な変更だ、と切り分けられます。移行後は必ず「読み込んだサンプルが期待どおりか」を少数件で目視・アサートする検証を入れるべきです。

## メリット・トレードオフ・限界

メリット
- 新しいライブラリ環境でデータを読み込めるようになる (古い API ではエラーで停止する)
- try/except による二段構えで、複数のインストール形態のどちらでも動く頑健性が得られる
- 後方互換のダミー引数を残せば、自分のラッパーを使う既存コードを壊さずに移行できる

トレードオフ・限界
- ライブラリのバージョンが上がるたびに追従作業が再発する。これは `0.x` 系のライブラリでは構造的に避けられない
- 後方互換のダミー引数は「指定したのに効かない」という認知のズレを生む。docstring やドキュメントでの明記が必須
- import の二段構えは、本物のエラーを代替パス側で握りつぶす危険がある。例外捕捉は最小範囲・具体的な例外型に絞るべき
- 最大の落とし穴は「動くが意味が壊れている」silent failure。import が通ってもメタ情報の意味変化を見落とすとデータが静かに壊れる

未解決の研究的課題として、「データセットフォーマットの後方互換性をどう保証するか」があります。現状は各ライブラリが場当たり的にバージョンを上げており、利用者が追従コストを払う構造です。スキーマのバージョン番号をデータ自体に埋め込み (実際 LeRobot のような実装は `codebase_version` をデータに記録する)、ローダが自動で適切な読み方を選ぶ、といった仕組みが理想ですが、まだ業界横断の標準化には至っていません。

## 発展トピック・研究の最前線

データパイプラインの堅牢性は、近年「データ中心 AI (data-centric AI)」という潮流のもとで再注目されています。Andrew Ng らが提唱するこの考え方は、「モデルを固定してデータの質を上げる方が、データを固定してモデルをいじるより効く場面が多い」というものです。その実践には、データの読み込みが正確で再現可能であることが大前提になります。

また、大規模学習では WebDataset (Aizman et al., IEEE Big Data 2019) や NVIDIA DALI のように「データセットフォーマットそのものを学習効率最優先で設計する」アプローチが広がっています。これらは標準データを学習時に読みやすい形 (シーケンシャルに読める shard など) へ変換する層を持ち、ライブラリ API の進化とは別レイヤーで I/O を最適化します。

研究の最前線では、複数の異種データセット (異なるカメラ構成・座標系・サンプリングレート) を統一インターフェースで扱う「データセットアダプタ層」の設計が課題になっています。各データセットの配布形式・API の差異を1箇所のアダプタで吸収し、学習コードには単一のインターフェースを見せる設計は、移行コストとデータセット差異の両方を一箇所に閉じ込める実務的な解です。Open X-Embodiment (Padalkar et al., 2023, arXiv:2310.08864) のように、数十種類のロボットデータセットを共通フォーマットに揃えて統合学習する取り組みは、この方向性の大規模な実例です。
