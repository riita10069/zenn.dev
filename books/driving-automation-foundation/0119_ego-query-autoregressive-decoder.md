# 単一ego queryによる自己回帰軌跡デコーダ

## ひとことで言うと

自動運転で「これからどう走るか(軌跡)」を決める部品の設計の話です。自車(自分の車)を表す「たった1個の問い合わせベクトル(ego query)」を用意し、それを1ステップずつ未来へ進めながら各時刻の運転操作を順番に生成します。物体検出のように多数のクエリを使うのではなく、計画では自車1個のクエリで足りる、という軽量な設計思想です。

## 直感的な理解

軌跡(きせき)とは、車がこれからたどる道筋のことです。たとえば「6.4秒先まで、0.1秒刻みで64個の点」として表し、各点を「加速度(どれだけ速く・遅くなるか)」と「曲率(どれだけ曲がるか)」の2つの数で記述すると、出力は 64点×2値 = 128個の数のベクトルになります。

軌跡を作るには、周囲の状況を表す特徴(鳥瞰図など)を見て「どこを通るのが安全で効率的か」を判断する必要があります。素朴には、周囲の特徴マップ全体を平らな1本のベクトルに潰して(flatten)、そこから多層パーセプトロン(MLP、単純な全結合ニューラルネット)で128個の数を出す方法が考えられます。

しかしこの素朴な方法には致命的な弱点があります。鳥瞰図が 450×300 マスで各マス256チャネルなら、平坦化すると 450×300×256 ≈ 3450万次元という巨大な入力になります。これに MLP をつなぐと重みの数が爆発し、メモリに乗らず、しかも「どこに何があるか」という空間構造が混ざって失われます。低解像度(7×7 など)のときだけ許される、その場しのぎです。

そこで発想を変えます。「マップ全体を潰す」のをやめ、「自車が今この瞬間に注目すべき場所だけを選んで見に行く」。この注目する主体が ego query です。自車という存在を1本のベクトルで表し、それが時々刻々と「次はどこを見るべきか」を判断しながら未来へ進んでいく。人間の運転手が一点を凝視せず、刻々と視線を移しながら運転するのに似ています。

## 基礎: 前提となる概念

**クエリ(query)とアテンション**: Transformer 系のモデルでは「問い合わせベクトル(query)」が「参照対象(key/value)」のどこに注目するかを計算します。物体検出の DETR では各物体に1個ずつクエリを割り当てますが、計画では予測対象が「自車1台」なので、原理的にクエリは1本で足ります。

**自己回帰(autoregressive)生成**: 系列を1要素ずつ生成し、各要素を生成するとき直前までの出力を入力に使う方式です。「前の点が決まらないと次の点が決まらない」軌跡生成と相性が良い一方、T 個の要素を順番に作るので並列化できず、推論が遅くなりがちです。

**GRU(Gated Recurrent Unit、ゲート付き回帰ユニット)**: 時系列を1ステップずつ処理する記憶つきの箱です。前ステップの内部メモリ(隠れ状態、hidden state)を受け取り、ゲート(0〜1で情報の出入りを調整する弁)で取捨選択して次のメモリを出します。詳しくは [軌跡デコーダにGRUを選ぶ理由](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0128_gru-vs-transformer-decoder)。

**deformable cross-attention**: 特徴マップ全体を見るのではなく、学習で決めた少数の点だけを座標指定で見に行く軽量なアテンションです。座標で注目先を指すため、特徴マップの解像度が変わってもコードが変わりません。詳しくは [grid_sampleによるdeformable cross-attention](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0303_deformable-cross-attention-grid-sample)。

**埋め込み(embedding)**: `nn.Embedding(1, 256)` は1個の256次元学習可能ベクトルを保持します。ego query はこれで作り、学習を通じて「自車という問い合わせ係」の役割を獲得します。

## 仕組みを詳しく

このデコーダの中核は3つの仕組みの組み合わせです。形は `B`=バッチ数, `C`=256, `H×W`=鳥瞰図の広さとします。

1. **ego query**: 学習で決まる256次元ベクトル1本。`nn.Embedding(1, 256)`。
2. **GRU**: 「1ステップ前の状態」から「次の状態」を吐く時間方向のメモリ。隠れ状態は256次元。
3. **deformable cross-attention**: 鳥瞰図のどこを見るかをモデル自身が選ぶ仕組み。

**初期化**: GRU の初期隠れ状態 `h_0` を作ります。典型的な実装では、過去の自車の運動履歴と過去フレームの視覚記憶という2系統を、それぞれ線形層(`Linear`、全結合層)で 256 次元へ射影してから足し合わせます。

```
h_0 = ego_state_proj(egomotion_history) + visual_history_proj(visual_history)
```

- `egomotion_history`: 過去の自車の動き(速度・加速度・ヨー角=車の向き・ヨー角速度=向きの変化率など)を並べたベクトル。`ego_state_proj` で 256 次元へ射影。
- `visual_history`: 過去フレームの圧縮された視覚記憶(たとえば「直近の数十フレームをそれぞれ低次元へ要約したもの」を連結したベクトル)。`visual_history_proj` で 256 次元へ射影。
- この2つを足して `h_0 [1, B, 256]` とします。過去の動き(運動の慣性)と過去の見た目(直前まで何が見えていたか)を初期記憶に込めることで、軌跡生成が「いきなり真っ白な状態」からではなく「これまでの文脈」から始まります。

鳥瞰図特徴は `value_proj: Linear(256, 256)` で変換して `values [B, 256, H, W]` を用意します。これが見に行く対象の棚です。

**各時刻 t(t=0〜63 の64回)のループ**:

1. 問い合わせを作る: `query = h_{t-1}.squeeze(0) + ego_query` → `[B, 256]`。
2. どこを見るか決める: `ref = sigmoid(reference_point(query))` → `[B, 2]`。`[0,1]` の基準座標。
3. 周りに何点ばらまくか: `off = offset_scale * sampling_offsets(query)` → `[B, 8, 2]`。`offset_scale=0.1` でばらつきを抑制。
4. 各点の重み: `attn_w = softmax(attention_weights(query))` → `[B, 8]`。
5. 実際に見に行く: `sample_locs = ref + off` を `[-1,1]` に直し `F.grid_sample` で双線形補間 → `[B, 256, 8]`。
6. 重み付き合計: `attended = output_proj(Σ attn_w · sampled)` → `[B, 256]`。
7. 時間を進める: `_, h_t = GRU(attended.unsqueeze(0), h_{t-1})` → `h_t [1, B, 256]`。
8. 操作を出す: `waypoints[t] = waypoint_head(h_t.squeeze(0))` → `[B, 2]`(加速度・曲率)。

**ループ後**:
- `trajectory = cat(waypoints)` → `[B, 128]`。出力する軌跡。
- `ego_hidden = h_T.squeeze(0)` → `[B, 256]`。最終隠れ状態。後段の世界モデル等へ渡せます([プランナの隠れ状態による「計画意図→世界モデル」の受け渡し](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0004_ego-hidden-world-model-handoff))。

**解像度非依存(resolution-agnostic)**: 注目場所を `[0,1]` の正規化座標で選ぶため、鳥瞰図が 8×8 でも 450×300 でもコードが一切変わりません。融合方式が低解像度の格子でも高解像度でも、同じデコーダがそのまま動きます。これは平坦化+MLP 設計には絶対にない性質で、設計上の最大の利点のひとつです。

**before/after**:

```
before (平坦化+MLP):
  BEV [B, 256, 450, 300] --flatten--> [B, 34,560,000] --MLP--> [B, 128]
  → 入力次元が巨大、空間構造が消失、解像度を変えると重みも作り直し

after (ego query + 自己回帰):
  ego_query [1個] が64回ループし、毎回 BEV の局所8点だけ見る
  → 入力次元は一定、空間構造を保持、解像度に依存しない
```

## 手法の系譜と主要論文

この設計は「物体検出から計画まで1本のクエリ機構でつなぐ」近年の自動運転研究の流れに沿っています。

- **DETR**(Carion et al., ECCV 2020, arXiv:2005.12872): 物体検出を「学習可能なクエリの集合」で解く枠組みを確立しました。各クエリが1つの物体を表し、画像特徴にアテンションして枠を出します。「タスクの出力要素をクエリで表す」という発想の源流で、計画における ego query もこの延長です。

- **UniAD / Planning-oriented Autonomous Driving**(Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156): 検出・追跡・地図・予測・計画をクエリを介して1つの Transformer パイプラインで貫く設計を提案しました。各物体に1個のクエリを割り当て、計画段階では自車を表す ego query が周囲の特徴に注目して軌跡を出します。動機は、タスクを別々のモジュールに分けると誤差が伝播し情報が落ちるため、クエリで全工程を微分可能につなぎ全体最適化したかったこと。単一 ego query で軌跡を出す発想の代表例です。

- **VAD / Vectorized Scene Representation**(Jiang et al., ICCV 2023, arXiv:2303.12077): シーンをベクトル化(物体や車線を点列で表す)し、ego query で計画する効率重視の設計です。UniAD の密なラスタ表現が重いので、軽く速くしたかったのが動機。ego query で計画するという骨格は共通で、表現を疎にした点が新規性です。

- **AD-MLP / Rethinking the Open-Loop Evaluation**(Zhai et al., 2023, arXiv:2305.10430): 極端な対照実験として、シーンを一切見ずに自車状態だけを MLP に入れるだけでも、nuScenes の開ループ評価ではある程度競争的な軌跡が出てしまうと示しました。これは「派手な知覚モジュールより、まず軽い設計でどこまで行けるか」を問う警鐘であり、同時に開ループ評価指標の偏り(過去の動きの外挿だけで当たってしまう)を暴いた重要な指摘です。

系譜としては「クエリで出力を表す(DETR) → 全タスクをクエリで貫く(UniAD) → 表現を疎にして軽量化(VAD) ⇔ そもそも知覚は要るのかという問い直し(AD-MLP)」という対話の中に、単一 ego query デコーダは位置づけられます。

## 論文の実験結果(定量データ)

計画性能は主に nuScenes の開ループ評価で測られ、代表指標は次の2つです。

- **L2 誤差(L2 displacement error)**: 予測軌跡と正解軌跡の各時刻の座標のずれ(ユークリッド距離)をメートルで表したもの。1秒後・2秒後・3秒後、およびその平均で報告します。小さいほど良い。
- **衝突率(collision rate)**: 予測軌跡が他の物体と衝突する割合(%)。小さいほど良く、安全性に直結します。

報告されているおおよその数値(設定により幅があります):
- UniAD は平均 L2 誤差 約 1.03m、平均衝突率 約 0.31% を達成し、当時の同条件手法(ST-P3 など)を上回りました。検出・追跡・予測を共同学習することで計画が改善する、つまりタスク間の相互作用が効くことを示しました。
- UniAD のアブレーションでは、上流タスク(追跡・地図・モーション予測)のヘッドを順に外していくと計画の L2 と衝突率が悪化し、特に占有予測やモーション予測を外すと衝突率が上がると報告されています。「計画は周囲の予測情報に支えられている」ことの定量的証拠です。
- VAD はベクトル表現により UniAD より高速(推論レイテンシで数倍速い設定が報告)で、L2・衝突率も同等以上を達成しました。効率と精度の両立を示しています。
- AD-MLP は自車の過去状態だけで平均 L2 約 0.29m(報告値)という、知覚を使う手法に匹敵あるいは下回る誤差を出しました。これは性能の証明ではなく、開ループ評価が「過去の動きの外挿」で当たってしまう欠陥を持つことの証拠で、以後の研究は閉ループ評価(シミュレータ内で実際に運転させる評価)の重要性を強調するようになりました。

指標の意味: L2 が 0.1m 縮むことは平均的な位置精度の向上ですが、衝突率の方が安全上は重要です。衝突率 0.1% の差でも、長距離走行の累積では事故有無を分けうるため重く見られます。

## メリット・トレードオフ・限界

メリット:
- 鳥瞰図解像度に依存しない。格子サイズが変わっても再学習もコード変更も不要(正規化座標で注目するため)。
- パラメータが小さい(この種の軽量デコーダは1M前後)。学習が軽く end-to-end で回しやすい。
- 「どこを見るか」をモデルが選ぶので、平坦化設計が捨てていた空間構造を保持できる。各軌跡点が地図上の根拠を持つ。

トレードオフ・限界:
- 64ステップを順番に回す自己回帰なので時間方向を並列化できず、推論レイテンシが大きい。
- ego query が1本なので予測は1通り(unimodal)。「右折も直進もありうる」といった複数仮説を同時に出せない([マルチモーダル軌跡予測](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0151_multimodal-trajectory-prediction))。
- 単一ヘッドのアテンションかつ `offset_scale` が小さいので、遠方を一度に広く見る能力は基準点 `ref` の移動に依存する。

未解決課題: 自己回帰の逐次性を捨てて並列に全軌跡を出す非自己回帰デコード([非自己回帰な軌跡デコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0152_non-autoregressive-trajectory-decoding))、複数モードを同時に扱うマルチモーダル化、そして開ループ評価の限界を超える閉ループ評価への移行が、研究の主要な方向です。

## 発展トピック・研究の最前線

- **拡散・フローベース計画**: 軌跡を1点ずつではなく、ノイズから一括で生成する拡散モデル(Diffusion Planner)やフローマッチングによる計画が登場し、マルチモーダルかつ並列生成を両立しようとしています。
- **言語条件付き計画(VLM プランナ)**: 大規模視覚言語モデルに「左折せよ」などの指示や交通規則を理解させ、ego query 的な計画と組み合わせる流れ(DriveGPT 系)。
- **閉ループ評価への移行**: CARLA や nuPlan のシミュレータで実際に運転させ、開ループ評価の外挿バイアスを排した評価が標準化しつつあります。
