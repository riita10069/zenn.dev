# BEV知覚レンジ設計

## ひとことで言うと
自動運転の車は、自分の周囲を「真上から見下ろした地図(BEV = Bird's-Eye View、鳥瞰図)」として把握します。この地図を「前方どこまで・後方どこまで・左右どこまで」カバーするか(知覚レンジ)、そして「1マスを何メートル四方にするか」(セルサイズ)をどう決めるか、という設計の話です。前後対称が必須なわけではなく、高速走行では前を厚く見る非対称レンジ(例えば前120m・後60m・左右60m、セル0.4m)のほうが、限られた地図サイズ(=計算予算)を安全に活かせます。レンジ設計は「限られた予算をどの方向にどれだけ配分するか」という資源配分の問題です。

## 直感的な理解
高速道路を時速120kmで走っているとします。これは秒速約33mです。前方を50mしか見ていなければ、わずか1.5秒先には地図の外に出てしまい、その先に何があるか分からないまま走ることになります。人間の運転でも「ずっと先を見て早めに備える」のが安全の基本で、機械も同じく前方を遠くまで見たいのです。停止距離の観点でも、時速120kmからの制動距離はおおむね100m前後(反応時間込み)になるため、前方100m以上を見られないと「止まりきれない領域」を知覚できないことになります。

一方、後ろはどうでしょう。後方は「追突を避ける」「車線変更でぶつからない」程度に見えれば十分なことが多く、120m後ろまで詳細に見る必要は普通ありません。つまり「前」と「後ろ」で必要な情報量が違う。それなら地図を前後対称(前50m・後50m)に作るのは無駄が多く、前を厚く後ろを薄くした非対称レンジのほうが、同じ地図サイズ(=同じ計算予算)を有効に使えます。知覚レンジ設計とは、限られた予算をどの方向に配分するかという「予算配分」の問題でもあります。

## 基礎: 前提となる概念

BEV(鳥瞰図)とは、カメラが横から撮った画像を、頭上から見下ろした2次元の地図に変換したものです。なぜ地図にするかというと、運転の判断(どこを走るか、いつ止まるか)は「自分の周りに何がどこにあるか」という空間情報を必要とするからです。カメラ画像のままだと遠くの物体は小さく写るなど見え方が距離でゆがむので、メートル単位で位置が分かる地図に直したほうが扱いやすいのです。

ここで2つの量を区別します。

- 知覚レンジ(perception range): BEV地図が現実世界の何メートル四方をカバーするか。例えば「前後左右それぞれ50m」なら100m × 100mの正方形。
- グリッド解像度: その地図を何マス×何マスで表すか(縦 H × 横 W)。詳細は[BEVグリッド解像度](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0261_bev-grid-resolution)。

この2つから、1マスの大きさ(セルサイズ)が決まります。

```
セルサイズ = 知覚レンジ(その方向の長さ) ÷ その方向のマス数
```

座標系の約束も押さえます。多くのBEV実装では自車(エゴ、ego=自分の車)の中心を原点とし、X=前方、Y=左方向、Z=上方向とします(右手系)。知覚レンジはしばしば `pc_range`(point cloud range の略、点群レンジ)という6要素で表されます。

```
pc_range = (x_min, y_min, z_min, x_max, y_max, z_max)  # 単位はメートル
```

例えば `(-60, -60, -5, 120, 60, 3)` なら、

- X(前後): x_min = -60m(後ろ60m)、x_max = +120m(前120m)。前後で2倍の差をつけた非対称設計。
- Y(左右): y_min = -60m、y_max = +60m。左右は対称で各60m。
- Z(高さ): z_min = -5m(地面より下)、z_max = +3m(車の屋根より上)。BEVは平面表現なのでZは投影時に高さ方向の何点をサンプルするかの範囲として使われる。

前後方向の合計距離は 60 + 120 = 180m、左右方向は 60 + 60 = 120m です。原点が幾何中心からずれる(前後非対称)点が、後述するように投影計算をやや複雑にします。

参考までに、主要ベンチマーク nuScenes の評価範囲は半径50m前後(検出評価は中心から最大50m)で、論文の多くがこの範囲に合わせて 51.2m 四方の対称レンジを採用してきました。これは「ベンチマークが対称・近距離中心だから論文も対称」という構造的な理由が大きく、必ずしも実車の安全要求から導かれた値ではない、という点は押さえておくべきです。

## 仕組みを詳しく

レンジとグリッド解像度からセルサイズが決まる関係を、具体例で計算します。前後180m・左右120mの範囲を、縦450マス・横300マスのグリッドで切ると、

- 前後: 180m ÷ 450マス = 0.4m/マス
- 左右: 120m ÷ 300マス = 0.4m/マス

きれいに1マス0.4m × 0.4mの正方形になります。レンジが非対称でも、マス数を比例配分すればセルサイズは等方(縦横同じ)に保てるわけです。逆にマス数を比例させないと、縦と横でセルサイズが食い違って物体が歪んで表現される(例: VADの 0.3m/0.15m のように意図的に変える設計もある)。

なぜ0.4mなのかが安全性の核心です。乗用車は全長約4.5m、全幅約1.8mです。1マスが0.4mなら、1台の車は「縦に約11マス × 横に約4.5マス」を占めます。これだけ細かければ、隣り合う2台の車のすき間や、歩行者(肩幅約0.5m)1人を、別々のマスとして区別できます。0.4mは「車・歩行者・車線を区別できる」実用上の目安として複数の論文が採用してきた値です。車線幅(約3.5m)は約9マス分なので、車線中央なのか端に寄っているのかも表現できます。

逆に、もし1マスが7m四方だったらどうなるか。これはBEVグリッドを意図して設計せず、バックボーン(画像から特徴を抜くCNN)の最終出力サイズ7×7をそのまま使ってしまった場合に陥る状態です。50m四方を7×7で割ると1マス約7m。乗用車1台がまるごと1マスに収まり、「2台並んでいるのか1台なのか」「車線のどちら側にいるのか」が原理的に判別不能になります。これは[BEVグリッド解像度](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0261_bev-grid-resolution)で扱う失敗パターンです。

設計手順をまとめると、

1. 知覚レンジ(`pc_range`)を決める。安全のため前を厚く: 前120m・後60m・左右60m。
2. グリッド数(H × W = 450 × 300)を決める。前後方向のマス数を多くして縦長にする。
3. セルサイズ = レンジ ÷ グリッド数 = 0.4m に揃える。
4. この0.4mが「車・歩行者・車線を区別できる細かさ」の根拠になる。
5. 前方120mを実際に写せる狭画角カメラ(例えば望遠の前方カメラ)が物理的に必要、という制約をカメラ構成と一体で検討する。

最後の点は見落とされがちです。地図上で前方120mまでマスを用意しても、そこを写すカメラがなければマスは空っぽです。遠方の物体は画像上で極端に小さく写る(120m先の車は通常の広角カメラでは数画素にしかならない)ため、遠方ほど狭い画角の高解像度カメラ(望遠)が要り、知覚レンジ設計はセンサー構成と不可分です。地図のマスを用意することと、そのマスを埋める画素情報があることは別問題だ、という点が実務上の落とし穴です。

## 手法の系譜と主要論文

主要なBEV知覚モデルがどんなレンジとセルサイズを採用してきたかを時系列で見ると、レンジ設計の考え方の変遷が読めます。

- BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270): 200×200グリッドで51.2m × 51.2mをカバー、セル0.256m。前後対称の正方レンジ。「明示的なBEVグリッドを定義し、学習可能なBEVクエリが空間クロスアテンションで複数カメラから情報を集める」基本形を確立。深度推定を別途行わずにマルチカメラを統合できる利点がある反面、グリッドが固定で前後対称なため、高速走行で前方をもっと見たい要求には素直に応えにくい。
- UniAD(Hu et al., CVPR 2023, arXiv:2212.10156): BEVFormerをBEVエンコーダに流用し、200×200・51.2m四方・0.256m。知覚から予測・計画までを1ネットワークに統合(end-to-end化)した点が新規。レンジ設計はBEVFormerを踏襲。
- BEVDet(Huang et al., 2022, arXiv:2112.11790)/ BEVDepth(Li et al., AAAI 2023, arXiv:2206.10092): どちらも128×128で51.2m四方、セル0.4m。BEVDepthは明示的な深度推定を加えて投影精度を上げた。0.4mという値はこのあたりが実績値で、リアルタイム性とのバランスを取った設定。
- VAD(Jiang et al., ICCV 2023, arXiv:2303.12077): 200×200で60m × 30mと、非正方形(前後60m・左右30m)のレンジを採用、セル0.3m/0.15m。「前後と左右を別々に取る非対称設計」の代表的先例で、計画に効く範囲を選んで予算を配分する発想。市街地の走行では横方向は車線数本ぶんで足り、縦方向を優先する設計。
- PETRv2(Liu et al., 2022)/ StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926): 100×100・51.2m四方でセル0.512mと、やや粗めにして計算を軽くする方向。
- PolarFormer(Jiang et al., AAAI 2023, arXiv:2206.15398): デカルト(正方マス)ではなく極座標(距離 × 角度)でBEVを切る。近くを密に、遠くを粗くする自然な配分ができ、カメラの放射状の見え方とも整合する。レンジ設計を「直交格子前提」から解放した代表例。

系譜の論点は「対称か非対称か」と「広く取るか細かく取るか」、さらに「直交格子か極座標か」の3軸です。初期のBEVFormer/UniADは対称・標準範囲を採用し、VADが非対称レンジを実用化、PolarFormerが座標系そのものを問い直しました。近年は走行ドメイン(高速か市街地か)に応じてレンジを変える設計思想が広がっています。一般的な研究方針としては「ベンチマークの多くが短〜中距離中心なので論文は短距離BEVが多いが、高速走行の安全には前方を厚く見る非対称グリッドが有用」という認識が共有されつつあります。具体例として前120m・後60m・左右60m → 450×300グリッド → セル0.4m、といった非対称・縦長設計が試みられています。

## 論文の実験結果(定量データ)

知覚レンジが性能に与える影響は、主に nuScenes(評価範囲は基本的に半径50m前後)上のアブレーションで間接的に観測されます。指標は NDS(nuScenes Detection Score、総合スコア、高いほど良い)と mAP、距離別では「遠距離の物体に対するAP」が鍵になります。

- 各論文のアブレーションで共通して見られるのは、知覚範囲を広げるほど遠距離物体のAPは改善する一方、同じグリッド解像度のまま範囲だけ広げるとセルサイズが粗くなり近距離の小物体APが落ちる、というトレードオフです。つまり「範囲を広げる」と「細かさを保つ」を両立するには解像度を上げるしかなく、それが計算量増に直結します。範囲・解像度・コストは三すくみの関係にあります。
- BEVFormer系の報告では、評価距離を遠方(例えば30〜50m帯)に限ると、近距離帯に比べてAPが大きく低下します。これは「遠くは画像上で小さく写り、投影誤差も大きい」ためで、前方を厚く見る設計の難しさ(範囲を広げるだけでは精度が伴わない)を定量的に裏づけています。遠距離の物体は数画素しか占めず、深度誤差も距離の2乗に比例して大きくなる(同じ画素ずれが遠方では大きなメートルずれになる)ため、本質的に難しい。
- VADの非対称レンジ(60m×30m)は、計画タスクの評価(L2軌道誤差や衝突率)で、計画に不要な横方向の範囲を削ることで効率を上げつつ性能を維持できることを示しました。L2誤差は予測軌道と正解のメートル単位のズレ(将来1s/2s/3s時点で測ることが多い)、衝突率(collision rate)は予測軌道が他車・障害物とぶつかる割合(%)で、いずれも小さいほど良い指標です。これは「レンジを賢く配分すれば、無駄を削っても安全性を損なわない」という非対称設計の正当化になっています。

数値の意味を補足すると、遠距離APの低下は実運転では「高速で前方の車を見落とす」リスクに直結し、前方レンジを厚くする動機そのものです。L2誤差が将来3秒で1mか2mかは、車線変更の安全マージンに直結します。ただし範囲を広げても投影精度が伴わなければ遠距離APは上がらないため、レンジ設計はカメラ構成・深度推定品質とセットで評価する必要があります。

## メリット・トレードオフ・限界

メリット
- 前方を厚く(例えば120m)見るので、高速走行時の長距離知覚に対応でき、安全側に振れる。停止距離をカバーできる。
- セル0.4mは車・歩行者・車線を区別できる細かさで、軌跡計画に十分な空間分解能を持つ。
- 前後左右を別々に指定できるので、地図サイズという限られた予算を「本当に見たい方向」に配分できる。

トレードオフ・限界
- レンジを広く・セルを細かくするほどグリッド数が増え、VRAMと計算量が増える(450×300は13.5万マス)。詳細は[BEV解像度とVRAMのトレードオフ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0260_bev-cell-size-vram-tradeoff)。
- 前方120mを写せる狭画角の前方カメラが物理的に必要。カメラ構成と一体で考えないと、地図上だけ遠くまで作っても中身が空っぽになる。
- 非対称レンジは座標系や投影の計算をやや複雑にする(前後で原点からの距離が違うため、グリッド原点のオフセットを正しく扱う必要がある。中心 = 幾何中心という暗黙の前提を置いたコードはバグる)。
- 遠距離の知覚は画像解像度・投影誤差・物体の小ささに律速され、レンジを広げただけでは精度が伴わない。

研究上の未解決課題として、走行シーンに応じてレンジを動的に変えるべきか(高速では前方を伸ばし、市街地では全方位を密に)、遠距離知覚の精度をどう底上げするか(高解像度カメラ、時系列での累積、深度の明示推定)、そして「検出に必要なレンジ」と「計画に必要なレンジ」をどう統合するかが残っています。レンジは固定パラメータとして与えられることが多いものの、本質的には走行コンテキスト依存の変数だという認識が広がりつつあります。

## 発展トピック・研究の最前線

レンジ設計の延長には、可変・適応的な知覚範囲の研究があります。固定グリッドではなく、自車速度や進行方向に応じて関心領域(前方の走行レーンなど)だけ高解像度に切る adaptive resolution、極座標で前方を密に取る polar BEV(PolarFormer など。距離が遠いほど角度あたりのメートルが広がるので、カメラの放射状の解像度低下と自然に整合する)、3Dへ拡張した占有(Occupancy)表現で高さ方向も含めて範囲を設計する流れがあります。

また、計画タスクに直結する end-to-end 系では「検出器が見るべき範囲」と「プランナが必要とする範囲」を別々に最適化する研究もあり、レンジを単一の固定値ではなくタスク・速度・シーンの関数として扱う方向が最前線です。さらに、レンジを伸ばすほど効いてくる「遠方の不確かさ」を明示的にモデル化する(遠方ほど予測の信頼度を下げる)アプローチや、複数フレームを累積して遠方を時系列で補強する手法も、長距離知覚の精度を底上げする研究として進んでいます。レンジ設計はもはや単なるハイパーパラメータ選択ではなく、センサー・座標系・時系列・不確かさを横断する設計問題になりつつあります。
