# バッチ次元の崩壊バグ

## ひとことで言うと

複数の場面をまとめて学習する「ミニバッチ学習」では、テンソルの先頭に「何個ぶんか」を表す次元(バッチ次元)を必ず残しておく必要があります。ところが特徴を組み替える処理のなかで `torch.flatten()` をうっかり引数なしで使うと、このバッチ次元まで一緒に潰してしまい、複数場面を同時に学習できなくなります。これは特定のソフトウェア固有の話ではなく、PyTorch/TensorFlow など多次元配列を扱う深層学習フレームワークで繰り返し起こる典型的な実装バグです。

## 直感的な理解

工場のベルトコンベアを思い浮かべてください。8個の部品を箱に入れて同時に検査ラインに流すとき、「どこからどこまでが1個ぶんか」という仕切りが消えてしまうと、8個の部品がぐちゃぐちゃに混ざって1つの塊になり、個別の検査結果が出せなくなります。バッチ次元はこの「仕切り」にあたります。

ニューラルネットワークの学習では、1つの場面だけを見せて少しずつ直すよりも、複数の場面(たとえば8場面)をまとめて見せて平均的な誤差で直すほうが、学習が安定して速くなります。この「まとめて処理する単位」をミニバッチ(mini-batch)、その個数をバッチサイズ(batch size)と呼びます。バッチ次元が崩壊すると、この「8場面ぶんを区別したまま並列処理する」という前提そのものが壊れます。

## 基礎: 前提となる概念

- テンソル(tensor): 数値の多次元配列。画像のミニバッチなら `[B, C, H, W]`(バッチ・チャンネル・高さ・幅)のように4次元が一般的。マルチカメラなら `[B, V, C, H, W]` の5次元。
- バッチ次元(batch dimension): PyTorch/TensorFlow では慣習として先頭(0番目)の次元をバッチとして扱います。後段の全結合層・畳み込み・損失関数・正規化層はすべて「先頭がバッチ」を暗黙の前提に書かれています。だからこの次元が消えると計算全体が静かに、あるいは派手に壊れます。
- ミニバッチ確率的勾配降下法(mini-batch SGD): パラメータ更新を1場面ごとではなく数〜数百場面の平均勾配でおこなう手法。勾配推定のノイズが減って学習が安定し、GPU の並列性も活かせます(Goodfellow et al., Deep Learning, 2016)。
- バッチ正規化(Batch Normalization; Ioffe & Szegedy, ICML 2015, arXiv:1502.03167)など多くの層は、バッチ次元に沿って統計量を計算します。バッチ次元が壊れるとこれらの統計も無意味になります。
- `flatten` と `reshape`: どちらもテンソルの形を変える操作。`flatten` は次元を平坦化(1次元化)し、`reshape` は指定した形に組み替えます。要素の総数が一致していれば形は自由に変えられます。

## 仕組みを詳しく

問題の核心は `torch.flatten()` の既定動作です。`torch.flatten(x)` を引数なしで呼ぶと、`start_dim=0` が既定なので、テンソルの全次元を1本に潰して1次元にしてしまいます。

具体例で示します。視点ごとの特徴が `[B, V, C, H, W] = [8, 7, 256, 8, 8]`(B=バッチ8、V=カメラ7、C=チャンネル256、H×W=8×8の空間)だったとします。

```
誤り:   torch.flatten(x)        -> [8*7*256*8*8] = [916480]   1次元の棒
本来:   x.reshape(8, 7*256, 8, 8) = [8, 1792, 8, 8]           先頭の8(バッチ)を保持
```

誤った方では、バッチ8もカメラ7も区別がつかなくなり、「どこからどこまでが1場面ぶんか」が失われます。本来やりたいのは「バッチは残したまま、カメラ特徴だけをチャンネルに混ぜる」ことです。

バッチ次元が崩壊すると起きることは2つあります。第一に、複数場面の情報が混ざり合って「8場面ぶんの平均的な学習」が成立しなくなる。第二に、後段の損失計算や層が「先頭=バッチ」を前提にしているため、形が合わずエラーになるか、エラーにならなくても意味のない値を計算してしまう。後者は厄介で、プログラムは動き続けるのに精度がまったく上がらない、という「サイレントな失敗」になります。結果として、バッチサイズ1でしか動かせない(=ミニバッチ学習が事実上不可能)状態に陥ります。

修正の方針は、全部を潰す `flatten()` をやめ、`reshape` で「どの次元を残し、どの次元を混ぜるか」を明示的に指定することです。

```
x = fused_per_view.reshape(B, V, C, H, W)   # バッチBとカメラVを明示的に分離
x = x.reshape(B, V * C, H, W)               # Bは残し、V*Cだけをチャンネルに統合
```

ここで `B` を第1引数に固定しているのが要点です。あるいは `flatten` を使うなら `torch.flatten(x, start_dim=1)` のようにバッチ次元を除外します。さらに、カメラ枚数 `V` を関数の外から受け取る引数(`num_views`)にしておくと、「7枚なのか8枚なのか」を確実に区別でき、間違った形への組み替えを防げます(詳細は [num_views のパラメータ化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0990_num-views-parameterization))。

before / after を整理します。

- before: `torch.flatten(x)` → 形 `[916480]`。バッチもカメラも消失。損失計算が破綻。バッチサイズ1運用に追い込まれる。
- after: `reshape(B, V, C, H, W)` → `reshape(B, V*C, H, W)`。バッチ8を保持したまま、カメラ次元だけをチャンネルに統合。ミニバッチ学習が成立。

## 手法の系譜と主要論文

これは研究テーマというより実装の落とし穴ですが、背景にある原理は深層学習の基礎に根ざしています。

- Goodfellow, Bengio & Courville「Deep Learning」(MIT Press, 2016) — ミニバッチ確率的勾配降下法が、計算効率と勾配推定の安定性のバランスを取る標準手法であることを解説。バッチ次元はこの仕組みの前提であり、崩壊させてはならない理由がここにあります。
- Ioffe & Szegedy「Batch Normalization」(ICML 2015, arXiv:1502.03167) — バッチ次元に沿って統計量を取る正規化を提案。バッチ次元が意味を持つことが前提なので、崩壊すると正規化も壊れます。後年、バッチに依存しない Layer Normalization(Ba et al., 2016)や Group Normalization(Wu & He, ECCV 2018)が提案されたのは、小バッチ・バッチ非依存の場面でこの前提を緩めるためでした。
- PyTorch 公式ドキュメント(`torch.flatten`, `torch.reshape`) — `flatten` は既定で `start_dim=0` から全次元を平坦化するため、`start_dim` を指定しないとバッチ次元まで潰れることが明記されています。

## 論文の実験結果(定量データ)

直接この「バグ」を測る論文はありませんが、関連する定量的事実があります。ミニバッチSGDの古典的研究(Bottou ら)では、バッチサイズを1から増やすと勾配推定の分散が概ね 1/B に比例して下がり、1ステップあたりの収束が安定することが知られています。すなわちバッチ次元が崩壊してバッチサイズ実質1になると、勾配ノイズが最大化し、学習率を下げざるを得ず、収束が大幅に遅くなります。

また Batch Normalization の原論文では、BN を入れることで ImageNet 学習のステップ数を従来の約1/14に削減できたと報告されています。BN はバッチ統計に依存するため、バッチ次元が壊れたコードでは BN がまったく機能せず、この高速化効果も失われます。「形状バグ1つで学習速度が一桁変わりうる」ことの裏づけです。

## メリット・トレードオフ・限界

- 修正のメリット: バッチサイズ2以上のミニバッチ学習が可能になり、学習が安定・高速化する。BN など統計依存の層も正しく働く。
- メリット: `reshape` で次元を明示するスタイルは、後から読む人が「どの次元が何か」を追いやすく、同種のバグの再発を防ぐ。コードがドキュメントを兼ねる。
- トレードオフ: `reshape` は次元の積が一致している必要があり、`V` を間違えると別の形状エラーになる。だからこそカメラ枚数を信頼できる単一の値として管理することが重要(=[num_views のパラメータ化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0990_num-views-parameterization))。
- 限界・教訓: このバグはユニットテストでバッチサイズと視点数を複数パターン与え、出力形状を検査すれば検出できます。バッチサイズ1だけでテストすると見逃すのが落とし穴で、最低でもバッチサイズ2以上・視点数複数で検査するのが鉄則です。`torch.flatten()` を使うときは必ず `start_dim=1` のようにバッチ次元を除外する習慣が有効です。

## 発展トピック・研究の最前線

- 形状検査の自動化: 近年は `jaxtyping` や `torchtyping` のように、テンソルの次元名と形状を型注釈として記述し、実行時に検査するライブラリが普及しています。`Float[Tensor, "batch views channels h w"]` のように書くと、バッチ次元の崩壊を実行前後に検出できます。
- `einops` の `rearrange`: `rearrange(x, "b v c h w -> b (v c) h w")` のように、混ぜたい次元を明示的に名前で書く記法。`flatten`/`reshape` より意図が明確で、バッチ次元の取り違えを構造的に防ぎます。研究コミュニティでは可読性と安全性の観点から広く推奨されています。
- バッチ非依存の正規化: Group/Layer Normalization の採用は、バッチサイズが小さい・1にならざるを得ない状況(高解像度3D、検出ヘッド等)での安定化策として確立しています。
