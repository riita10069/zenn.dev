# マップクロスアテンションフュージョン

## ひとことで言うと

カメラから作った俯瞰(BEV)特徴を「質問する側(query)」、地図から作った BEV 特徴を「答えを持つ側(key/value)」として、アテンション(注意機構)で混ぜ合わせる融合方式です。各位置のカメラ特徴が「地図のどこを参照すれば自分の意味がはっきりするか」を学習で選び、地図の文脈を自分に取り込みます。単純な足し算ではなく、内容に応じて柔軟に参照先を変えられるのが特徴です。位置の対応が固定されている残差融合に対し、こちらは「どこを見るか」自体を学習します。

## 直感的な理解

自動運転モデルは、カメラ映像だけ見ても「この交差点で右に行くのか左に行くのか」が分かりません。そこでナビ地図の情報を別の枝(ブランチ)として入れ、カメラ由来の特徴に合流させます。この「2つの情報源を1つにまとめる」処理を融合(fusion)と呼びます。

融合の最も単純な方式は残差融合([マップBEV残差フュージョン（per-channel alpha）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0309_map-bev-residual-fusion))で、`image_bev + alpha * map_bev` という重み付き足し算です。これは速くて安全ですが、「カメラ特徴の(i,j)番地」と「地図特徴の(i,j)番地」がぴったり同じ実空間地点を表していることを前提にしています。位置がずれていると、見当違いの地図情報を足してしまいます。たとえば地図側の進路線がカメラ BEV より1セルずれた位置に描かれていると、足し算では「隣の場所の地図情報」を混ぜてしまうのです。

クロスアテンションは、この「同じ番地どうしを足す」という固定の対応を捨て、「カメラ側の各位置が、地図側のどの位置を見るべきか」をモデル自身に決めさせます。たとえばカメラ側の「前方やや右」の位置が、地図側で「青いルート線が通っている位置」を強く参照する、といった柔軟な対応を学習できます。これにより、多少の位置ずれや、地図と画像の意味の対応が単純でない場合にも、対処できる可能性があります。

## 基礎: 前提となる概念

- アテンション(注意機構): たくさんの候補の中から「今の自分にとって関係が深いものほど強く参照する」仕組み。機械翻訳で「この単語を訳すには元の文のどの単語を見ればいいか」を選ぶのと同じ発想です。
- query / key / value(クエリ・キー・バリュー): query は「探しに行く側(質問)」、key は「照合に使う見出し」、value は「実際に取り込む中身」。query と各 key の似ている度合い(内積)で重みを決め、その重みで value を加重平均します。数式では `Attention(Q,K,V) = softmax(QKᵀ/√d) V`(d は次元数。√d で割るのは内積が大きくなりすぎて softmax が飽和するのを防ぐため)。
- クロスアテンション: query と key/value が別々の情報源から来るアテンション(ここでは query=カメラ、key/value=地図)。同じ情報源どうしなら self-attention と呼びます。
- マルチヘッド(multi-head): 特徴次元を複数のヘッドに分割し、それぞれ別の観点で並列にアテンションを計算してから結合する方式。1つの観点に縛られず、複数種類の対応を同時に学べます。
- pre-norm(プレノーム): アテンションや全結合層に入れる前に LayerNorm(各トークンの特徴を平均0分散1に正規化)をかける設計。post-norm(後に正規化)より深いネットで学習が安定することが知られます。
- 残差接続(residual connection): `出力 = 入力 + 変換(入力)` の形。元の情報を保ったまま変換結果を上乗せできます。
- FFN(Feed-Forward Network): 各トークンを独立に通す小さな2層全結合。アテンションが「位置間のやり取り」を担うのに対し、FFN は「各位置内での特徴の作り直し」を担います。

## 仕組みを詳しく

典型的な実装を、形(tensor shape)を追いながら説明します。`C=256`、空間サイズ `H=W=8` を例にします。入力は2つとも `(B, C, H, W) = (B, 256, 8, 8)`。`image_bev` が query に、`map_bev` が key/value になります。

### ステップ1: 空間を1列に並べ替える(flatten)

アテンションは「並んだトークン列」に対して動くので、8×8 の格子を 64 個のトークンに展開します。

```python
q  = image_bev.permute(0, 2, 3, 1).reshape(B, H * W, C)   # (B, 64, 256)
kv = map_bev.permute(0, 2, 3, 1).reshape(B, H * W, C)     # (B, 64, 256)
```

`permute(0,2,3,1)` でチャンネルを最後に持ってきて `(B, H, W, C)` にし、`reshape` で `(B, 64, 256)` にします。これで「64個の位置、それぞれ256次元の特徴ベクトル」という列ができます。

### ステップ2: pre-norm クロスアテンション + 残差

```python
kv_norm = self.norm_kv(kv)
attn_out, _ = self.cross_attn(self.norm_query(q), kv_norm, kv_norm)
q = q + attn_out                                             # 残差接続
```

`self.cross_attn` は `nn.MultiheadAttention(embed_dim=256, num_heads=8, batch_first=True)` のような実装です。`num_heads=8` は256次元を8等分(各32次元)して並列にアテンションを計算します。直感的には、query 側の各位置(64個)について「地図側の64個の位置それぞれとどれくらい関係が深いか」を内積で測り、softmax で重みにして、地図側の value を加重平均で取り込みます。`q = q + attn_out` の残差接続により、元のカメラ特徴を保ったまま地図から取り込んだ情報を上乗せします。地図がノイズでも元のカメラ特徴が壊れにくいのは、この残差のおかげです。

アテンションが内部で何をしているかを、ごく小さな数値例で見ます。あるカメラ位置の query ベクトルを q、地図側の3つの位置の key を k1,k2,k3 とすると、まず内積 `q·k1, q·k2, q·k3` を計算します。仮にそれぞれ 8.0, 0.5, 0.1 だったとし、次元 d=32 なので √d≈5.66 で割ると 1.41, 0.09, 0.02。softmax をかけると重みはおよそ `[0.76, 0.13, 0.11]` になり、地図側の value を `0.76*v1 + 0.13*v2 + 0.11*v3` と加重平均します。つまり「このカメラ位置は地図の1番目の位置を主に参照する」と学習で決まったわけです。√d で割らないと内積が大きくなりすぎて softmax がほぼ `[1,0,0]` に飽和し、勾配が消えて学習が進みません。これがスケーリングの理由です。

```
カメラ位置(query)        地図側の各位置(key/value)
   q ──内積→ k1,k2,k3 ──/√d→ softmax重み[0.76,0.13,0.11]
                                       │
   出力 = q + Σ(重み × value)  ←───────┘  (残差で元の q に上乗せ)
```

### ステップ3: FFN + 残差

```python
q = q + self.ffn(self.norm_ffn(q))
```

`ffn` は `Linear(256→512) → GELU → Dropout → Linear(512→256) → Dropout` のような、各位置の特徴を一度広げてから戻す Transformer 標準の処理です。これも pre-norm + 残差で構成します。

### ステップ4: 格子の形に戻す

```python
return q.reshape(B, H, W, C).permute(0, 3, 1, 2).contiguous()   # (B, 256, 8, 8)
```

出力は入力と同じ `(B, C, H, W)`。つまり「地図の文脈を取り込んで更新されたカメラ BEV 特徴」が、形を変えずに返ります。後段の軌跡プランナは融合前後で形が同じなので、融合方式を意識せずに使えます。

### 空間対応に関する重要な注意

クロスアテンションも残差融合も、「`image_bev` と `map_bev` の各位置が同じ実空間地点を表す」ことを暗黙に期待しています。この前提が厳密に成り立つのは、カメラの射影行列と距離範囲(pc_range: 自車中心の何メートル四方を BEV グリッドにするか)から明示的に計量グリッドを作るモードのときだけです。このとき両特徴が同じ自車座標系に固定されます。

一方、各カメラ視点を単にプールして抽象的な特徴マップを作るだけのモードでは、特徴の位置と実空間の対応がありません。この場合でも融合は「学習で重みを決める混合」として機能はしますが、セル単位の空間的位置合わせは存在しません。クロスアテンションは固定の番地対応に縛られない分、残差融合よりこのズレに強い可能性がありますが、根本的な解決ではありません(地図側に誤った位置の情報しか無ければ、どこを見ても正しい対応は得られない)。

## 手法の系譜と主要論文

- Vaswani et al., "Attention Is All You Need"(NeurIPS 2017, arXiv:1706.03762): アテンション機構と Transformer を提案した起点。query/key/value の内積アテンション、マルチヘッド、pre/post-norm、FFN という構成要素はすべてここに由来します。動機は、系列処理で遠く離れた要素どうしを直接結びつけたい(RNN や CNN は遠距離の関係を捉えにくい)というもの。効果は長距離依存の獲得と並列計算。トレードオフは、トークン数 N に対し計算量が O(N²) で要素数が増えると重くなること。BEV を 8×8 の N=64 に保てば軽量です。

- Carion et al., "End-to-End Object Detection with Transformers (DETR)"(ECCV 2020, arXiv:2005.12872): 画像特徴を key/value、学習可能なオブジェクトクエリを query として、クロスアテンションで物体を集合予測する枠組み。動機は、検出を手作りのアンカーや後処理(NMS: 重複検出の抑制)なしにすること。「片方を query、片方を key/value にして異種情報を結びつける」という本トピックの設計はこの DETR 系の使い方に近いです。

- Li et al., "BEVFormer"(ECCV 2022, arXiv:2203.17270): BEV 空間に並べた学習可能なクエリが複数カメラ画像へ(deformable)クロスアテンションして俯瞰特徴を作る手法。動機は、複数視点を共通の BEV 空間で幾何的に整合させて統合すること。地図・カメラを共通空間で結びつける本トピックの直接の系譜上にあります。トレードオフは、正しい射影(カメラパラメータ)が必要で、それがズレると性能が落ちること。

- Liang et al., "BEVFusion"(NeurIPS 2022, arXiv:2205.13542): カメラと LiDAR をそれぞれ BEV 空間に持ち込み、共通の BEV 表現上で融合する枠組み。モダリティを共通空間に揃えてから混ぜるという発想は、カメラ BEV と地図 BEV の融合と同型です。

クロスアテンション融合は、ゼロ初期化の残差ゲート([マップBEV残差フュージョン（per-channel alpha）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0309_map-bev-residual-fusion))に対する「より表現力の高い代替案」という位置づけで併設されることが多く、両者を切り替えて比較できるよう設計されます([マップフュージョンレジストリ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0980_map-fusion-registry))。

## 論文の実験結果(定量データ)

DETR(ECCV 2020)は COCO 物体検出で、最適化済みの Faster R-CNN ベースラインと同等の AP(Average Precision: 検出の精度を再現率にわたって平均した総合指標、高いほど良い)約 42 を達成し、手作りのアンカーや NMS を不要にしました。一方で「学習が遅い(500エポック規模)」「小物体に弱い(small AP が低い)」というトレードオフを報告しており、これがクロスアテンション系の典型的な弱点です。後続の Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159)は、全位置ではなく各 query につき少数(典型的には4点)の参照点だけをサンプリングして注目する変形アテンション(deformable attention)でこれを改善し、学習を約10分の1のエポック(500→50 程度)に短縮しつつ、小物体 AP を含めて精度を上げました。計算量も O(N²) から参照点数に比例する形へ落ちます。高解像度 BEV でアテンション融合を行う際、この疎化が事実上必須になる理由がここにあります。

pre-norm を使う理由も実験で裏付けられています。Xiong et al.(ICML 2020, arXiv:2002.04745)は、LayerNorm を残差枝の前に置く pre-norm 構成では学習開始時の勾配が安定し、学習率の warmup なしでも収束することを理論と実験の両面で示しました。post-norm(後に正規化)は深いネットで初期勾配が爆発しやすく warmup が必須になります。融合モジュールを安定して学習させたい本トピックで pre-norm を選ぶのは、この知見に沿っています。

BEVFormer(ECCV 2022)は nuScenes 検出ベンチマークで、NDS(nuScenes Detection Score: 検出精度と位置・速度・属性の誤差を合成した総合指標、高いほど良い)を当時のカメラのみ手法の中で最上位級に押し上げました。空間 cross-attention(複数カメラへの参照)と temporal self-attention(過去フレームの BEV の参照)を組み合わせ、temporal を抜くアブレーションでは速度推定の誤差(mAVE)が悪化することを示しています。これは「クロスアテンションで適切な参照先を選べることが、融合の質を直接左右する」証拠です。

なお本融合方式そのものの一般的なアブレーションとしては、(a) 残差接続を外すと地図ノイズで序盤の表現が崩れやすい、(b) マルチヘッドを1ヘッドにすると複数種類の対応を同時に捉えられず精度が伸び悩む、(c) ゼロ初期化の安全装置がないため、残差融合に比べ学習初期の損失が高めに出やすい、という傾向が予想されます(具体的数値はタスク・データセット依存で、報告により異なります)。

## メリット・トレードオフ・限界

メリット
- カメラ側の各位置が地図側のどこを参照するかを学習で決めるため、固定の番地対応に縛られず柔軟。位置ずれに残差融合より強い可能性がある。
- 残差・FFN の二段構成で、地図情報を取り込みつつ元のカメラ特徴を壊しにくい。
- マルチヘッドで複数の観点から同時に地図を参照できる。

トレードオフ・限界
- 残差融合(`image_bev + alpha*map_bev`)より重い。アテンションと FFN の計算・パラメータが増える。
- 残差融合のような「学習開始時は地図枝が無寄与」というゼロ初期化の安全装置が標準では無く、最初から地図が混ざるため、地図がノイズだと学習初期に悪影響が出る可能性がある(クロスアテンション出力枝にもゼロ初期化ゲートを足して両者の長所を併せる設計もあり得る)。
- 計算量はトークン数 N に対し O(N²)。BEV 解像度を上げると急に重くなる。高解像度では Deformable のような疎なアテンションが要る。
- 空間対応が幾何的に保証されるのは、計量 BEV グリッドを作るモードのみ。抽象特徴プールのモードでは位置整合の前提が崩れる。

研究上の未解決課題として、高解像度 BEV でのアテンションの計算効率(疎化・線形アテンション)、地図とカメラの座標系を学習で整合させる仕組み(明示的射影に頼らない暗黙の位置合わせ)、そして残差ゲートとクロスアテンションを使い分ける/組み合わせる最適な融合戦略があります。

## 発展トピック・研究の最前線

cross-attention によるモダリティ融合は、BEV 統合の主流になりつつあります。LiDAR・カメラ・地図・レーダーを共通 BEV クエリで統合する一連の研究(BEVFusion 系、UniAD など)が代表です。計算効率の面では Deformable Attention が定番化し、全位置への二乗計算を少数の参照点へのサンプリングに置き換えます。

別の最前線は、地図を「画像」ではなく「ベクトル要素のトークン列」として与え、それを key/value にするオンライン地図構築との接続です(MapTR 系)。この場合、地図トークンは車線ポリラインそのものを表すため、クロスアテンションが幾何的に意味のある参照を学びやすくなります。ラスタ地図のセル対応の曖昧さを、ベクトル地図側で根本的に解消する方向と言えます。
