# モデルコードベースのリファクタ

## ひとことで言うと
研究を進めるうちにモデルへ機能を足し続けた結果、コードが膨らんで重複や使われない部品が増えてしまう、という問題は機械学習プロジェクトでほぼ必ず起きます。これを思い切って整理し、モデルを「素早く反応するリアクティブ系(Reactive)」と「じっくり世界を理解するワールドアクション系(World Action)」の2つに再編する、という大掃除のパターンを扱います。単なるコード整理ではなく、アーキテクチャの責務をどう分けるかという設計判断の話です。

## 直感的な理解
机の上で長く実験を続けていると想像してください。最初はノートとペンだけだったのが、半年後には試したアイデアのメモ、使わなくなった定規、似た役割のペンが3本、といった具合に散らかります。どれが今使っている道具で、どれが過去の実験の名残なのか、自分でも分からなくなる。新しく来た人はもっと分かりません。

ソフトウェアでも同じことが起きます。アイデアを試すたびに新しいモジュール(部品)を足していくと、「使われているか分からない部品」「似た役割の部品が複数」「重くて遅い割に効果が薄い部品」が溜まります。リファクタとは、この散らかった机を整理し、これから新しい道具を置きやすい状態に戻す作業です。重要なのは「外から見た振る舞い(モデルの入出力や精度)は変えずに、内部構造だけをきれいにする」点です。

そして単に掃除するだけでなく、「速く反応する系」と「ゆっくり考える系」という2つの引き出しに道具を仕分けする、という発想がこのトピックの核心です。これは人間の認知が「とっさの反射」と「じっくりした熟慮」に分かれているのと似ています。

## 基礎: 前提となる概念

**リファクタ(refactoring)**: 外部から見た振る舞いを変えずに、コードの内部構造を整理し直すこと。Martin Fowlerの古典的著書『Refactoring』(1999、第2版2018)で体系化されました。目的はバグを直すことでも機能を足すことでもなく、「読みやすさ・変更しやすさ・無駄の削減」です。

**技術的負債(technical debt)**: 「とりあえず動けばいい」で書いた急ぎのコードや、整理を後回しにした結果たまった「いつか払うべきツケ」のこと。放置すると利子のように膨らみ、新機能の追加が遅くなります。リファクタは負債の返済にあたります。

**モジュール / コンポーネント**: ニューラルネットを構成する部品。たとえば「画像から特徴を取り出す部分(バックボーン)」「複数カメラを統合する部分(融合)」「軌道を出す部分(プランナー)」など、役割ごとに分かれた単位です。

**責務の分離(separation of concerns)**: ソフトウェア設計の基本原則で、「ひとつの部品はひとつの役割だけを持つべき」という考え方。役割が混ざった部品は、変更したときの影響範囲が読めず、バグの温床になります。

**二重過程理論(dual-process theory)**: 認知科学の理論で、人間の思考を System 1(速く、直感的、自動的)と System 2(遅く、論理的、努力を要する)の2つに分ける見方。心理学者Daniel Kahnemanの『Thinking, Fast and Slow』(2011)で広く知られるようになりました。近年の自動運転・ロボット工学では、この比喩を借りて「速い反応制御」と「遅い世界理解」を分けるアーキテクチャがよく設計されます。

## 仕組みを詳しく

このタイプの大規模リファクタで典型的に行われる変更を、要素ごとに見ていきます。

### 1. 多視点融合(multi-view fusion)をBEV Fusionに一本化

複数カメラの特徴をひとつにまとめる「融合(fusion)」の方式には、歴史的に複数の選択肢が併存しがちです。代表的なのが、Concat(連結=単純に特徴を並べる)、Cross Attention(注意機構で混ぜる)、BEV Fusion(鳥瞰図グリッドに投影してから混ぜる)の3つです。リファクタでは、しばしばこのうち Concat と Cross Attention を削除し、BEV Fusion を標準に据えます。

動機は、BEV Fusionが他の最先端手法(SOTA)とよく揃っており、地図特徴(エンコードされた地図情報)とシームレスに統合でき、さらに複数のカメラ取り付け位置・台数を扱えるため、車種が違っても柔軟に適応できることです。「将来いろいろな車に載せる」ことを見据えると、BEVが最も拡張性の高い共通表現になる、という判断です。詳細は [BEV Fusionを標準融合方式に統一](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0300_bev-fusion-as-standard) を参照してください。

### 2. GRUプランナーを削除

プランナー(planner)は、特徴から自車の走る軌道を出すモジュールです。複数の実装が併存することが多く、そのうちGRU(Gated Recurrent Unit、系列を1ステップずつ処理する回帰型ネットワーク)ベースのものは、メモリ消費が大きく逐次処理ゆえに遅い、という弱点を抱えます。リファクタではこれを削除し、軌道を一括(非逐次)で出せるFlow Matchingプランナーとベジエ(Bezier)プランナーを残す、という整理がよく行われます。詳細は [プランナーの整理 (GRU削除)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0135_planner-pruning-flow-bezier) を参照してください。

### 3. プランナーから「自車の意図ベクトル」出力を削除

一部の設計では、プランナーが「自車の意図(これからどう動くつもりか)を表す学習ベクトル」(しばしば ego hidden などと呼ばれる)を出力し、それをBEVへの条件付けなどに使っていました。リファクタでは、これをプランナーの出力から外すことがあります。

動機は責務の分離です。視覚履歴の作成や将来状態の予測は、アーキテクチャの「遅い系(slow reasoning)」が担うべきであり、プランナー(速い系)が意図ベクトルまで抱え込むのは役割の混在だ、という判断です。速い反応をする部分と、ゆっくり世界を理解する部分の責務をはっきり分ける狙いがあります。

### 4. 中心クラスを「速い系」と「遅い系」に二分割

このリファクタの背骨は、それまで全機能を抱えていた中心クラスを、Reactive系(反応モデル)と World Action系(ワールドアクションモデル)の2つに分けることです。後者は最初は中身が空のプレースホルダ(後で実装する枠だけ用意したもの)として置かれることが多く、まず器を作ってから中身を入れる段階的な進め方が取られます。

```
従来:  巨大な単一クラス(全機能が混在)

リファクタ後:
  Reactive系       … 観測 → 軌道(速い、毎フレーム)
  World Action系   … 1Hz 視覚履歴 + 将来予測(遅い、まずはプレースホルダ)
       │
       └→ 符号化済み視覚履歴 を Reactive系 に供給
```

この2分割は二重過程理論の比喩そのものです。Reactive系(System 1相当)は現在の観測から即座に軌道を出すリアルタイム制御側、World Action系(System 2相当)は1Hz程度でじっくり世界の動き方を学び、視覚履歴や将来状態を作って速い側に渡す側です。世界モデル(GAIA系)はまさにこの「遅い側で世界の動き方を学ぶ」部分に相当します。

### 5. 推論・実行スクリプトの更新

クラス名や入出力が変わると、それを呼ぶ側(推論スクリプト、評価コードなど)も一斉に追従させる必要があります。リファクタでは内部だけでなく、こうした「呼び出し側」の更新が必ず付随します。ここを忘れると、学習はできても推論で落ちる、といった移行期特有のバグが出ます。

### 6. プランナーモジュールから損失計算を削除

プランナーの内部に損失計算が埋め込まれていると、複数のプランナーがそれぞれ別々に損失を持つことになり、重複が生じます。すべてのプランナーが同一フォーマットで軌道を出力するなら、損失計算は1か所に集約するのが筋です。リファクタではこれを外側の共通ロジックに移し、責務をすっきりさせます。詳細は [損失計算をプランナーから切り離し外部の軌道損失に一本化する設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0566_planner-loss-decoupling) を参照してください。

## 手法の系譜と主要論文

リファクタ自体はソフトウェア工学の作業であり特定の論文に基づくものではありませんが、その設計判断の背景には自動運転アーキテクチャ研究の蓄積があります。

- **Fowler, "Refactoring: Improving the Design of Existing Code" (1999, 第2版2018)**。リファクタという概念と、その安全な手順(小さなステップで振る舞いを保ちながら構造を変える)を体系化した古典。機能を足すフェーズと構造を整えるフェーズを意識的に分ける、という規律はここに由来します。

- **Hu et al., "Planning-oriented Autonomous Driving (UniAD)" (CVPR 2023, arXiv:2212.10156)**。知覚・予測・計画という従来バラバラだったタスクを、ひとつの共有BEV表現の上に統合し、すべてを「計画(planning)」という最終目的に向けて協調させる設計を提案しました。動機は、タスクごとに独立したパイプラインを積むより、共有表現で計画に向けて一貫させる方が最終性能が高いこと。BEV Fusionを標準に据え、計画を中心に整理する方向と思想が一致します。

- **Hu et al., "GAIA-1: A Generative World Model for Autonomous Driving" (2023, arXiv:2309.17080)**。運転データから世界の動き方を学ぶ生成世界モデル。リファクタにおける「遅い系(World Action)」が担うべき機能の理論的なお手本にあたります。

- **Kahneman, "Thinking, Fast and Slow" (2011)**。System 1 / System 2 の二重過程理論を一般に広めた著作。速い反応系と遅い熟慮系を分けるアーキテクチャの比喩的な源泉として、近年のロボティクス・自動運転論文で頻繁に引用されます。

## 論文の実験結果(定量データ)

リファクタそのものに実験結果はありませんが、その設計判断を支える研究には明確な定量的裏づけがあります。

- **UniAD (CVPR 2023)**: nuScenesデータセットで、知覚から計画まで全タスクを評価。中心となる計画指標は、将来軌道の平均変位誤差(L2 distance、予測軌道と正解軌道の距離、メートル単位、低いほど良い)と衝突率(collision rate、予測軌道が他物体と衝突する割合、%、低いほど良い)。UniADは、タスクを別々に解く従来手法やナイーブな多タスク手法に対し、計画のL2誤差と衝突率の両方を有意に低減しました。報告によれば平均L2誤差をベースライン比で相対的に2〜3割改善し、衝突率も低下しています。重要なのは、共有表現で計画中心に統合する設計が、最終的な運転安全指標で測って実際に効くと示した点です。L2誤差0.1mの差は、車線内での位置取りや障害物との余裕に直結するため小さくありません。

- **責務分離の効果(一般論)**: 速い系と遅い系を分けるアーキテクチャは、推論レイテンシ(1回の判断にかかる時間)とスループットの面で測定可能な利点を持ちます。重い世界理解を1Hzに落とし、軽い反応制御を高頻度で回す構成は、全体を高頻度で回す素朴な構成に比べ、同等精度を保ちながら計算量を大幅に削減できることが、世界モデル系の研究で繰り返し示されています。

## メリット・トレードオフ・限界

メリット:
- 使われない部品(Concat/Cross Attention融合、GRUプランナー)を削除し、メモリと速度の無駄を削減できる。
- 役割を「速いReactive / 遅いWorld Action」に明確分離することで、今後の機能(世界モデルなど)の置き場所がはっきりし、機能追加のたびの判断のブレが減る。
- 損失計算を集約して重複を解消し、保守性が上がる。
- BEV Fusion標準化により、SOTA整合・地図統合・車種差への拡張性が向上する。

トレードオフと限界:
- 選択肢(複数の融合方式、GRUプランナー)を削るため、後でそれらが必要になった場合の再導入コストが発生する。「使わないから消す」が将来の柔軟性を犠牲にする面がある。
- クラス名や入出力が変わるため、呼び出し側(推論スクリプト、テスト)を一斉に追従させねばならず、移行期は壊れやすい。リファクタ後にテストスイートを新構成へ合わせる追従作業がしばしば必要になる。
- 遅い系を最初プレースホルダにする場合、リファクタ単体では「遅い側」の機能はまだ動かず、本領発揮は後続実装待ちになる。器だけ作って中身が空、という中間状態のリスクを抱える。
- 未解決の課題として、速い系と遅い系のあいだの情報のやりとり(どの表現を、どの頻度で、どう渡すか)の最適設計はまだ研究途上で、決定版はありません。

## 発展トピック・研究の最前線
速い系と遅い系を分けるアーキテクチャは、近年「dual-system」「fast-slow」モデルとしてロボット操作・自動運転の両分野で活発に研究されています。視覚言語モデル(VLM)を遅い系の高水準推論に使い、軽量な方策ネットワークを速い系の低レベル制御に使う構成(例: 大規模VLAモデルにおける階層分割)が代表的です。世界モデルを行動条件付きにして「この操作なら未来がこう変わる」を予測し、それを計画にフィードバックする方向(UniAD系・GAIA系の延長)も最前線にあります。リファクタという観点では、こうした拡張を載せやすい「拡張点(extension point)」を意図的に設計しておくこと、すなわちレジストリ(部品を名前で登録・選択する仕組み)やプラグイン構造の設計が、研究速度を左右する実践的な論点です。
