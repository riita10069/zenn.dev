# WGS84→自車ローカル座標変換

## ひとことで言うと

地図データの点は、ふつう「緯度・経度」という地球全体を表す座標(WGS84)で記録されています。一方で自動運転の認識・予測モデルは「自分の車から見て前に何メートル、左に何メートル」という、自車を原点とした座標で世界を捉えたいです。WGS84→自車ローカル座標変換は、この緯度経度を「X=前方、Y=左、Z=上(単位はメートル)」という自車中心の座標へ直す前処理です。これをやって初めて、地図上の道路や交差点の位置と、カメラやLiDARから作った自車視点の特徴を同じ土俵で重ね合わせられます。

## 直感的な理解

スマートフォンの地図アプリを思い浮かべてください。アプリ内部の地図データは「東京駅は北緯35.681度・東経139.767度」のように、地球上の絶対位置で記録されています。これは世界中どこでも一意に場所を指せる便利な表現です。しかし、あなたが運転中に知りたいのは「次の交差点まであと何メートルか」「左の歩道はどこか」という、自分から見た相対的な位置です。

人間は無意識にこの変換をしています。地図を見て「今いる場所はここ、進行方向はこっち」と頭の中で地図を自分の向きに合わせて回転させ、「では右に曲がる道はあそこ」と読み替えています。WGS84→ローカル変換は、この「頭の中の地図合わせ」を計算式にしたものです。コンピュータに「地球座標で書かれた地図を、今の自車の位置と向きに合わせて読み替えなさい」と指示する処理だと考えてください。

なぜわざわざ変換が要るのか。緯度経度のまま使えない理由は2つあります。第一に、単位が「度」であってメートルではありません。しかも経度1度の実距離は緯度によって変わります(後述)。第二に、原点が地球の決まった場所(赤道と本初子午線の交点)にあり、自車とは無関係です。モデルは「自分から見て」の位置を扱いたいので、原点を自車に、軸を自車の向きに合わせ直す必要があります。深層学習の観点で言えば、ニューラルネットワークは入力のスケールと並進・回転に敏感です。同じ「前方50mの停止線」が、ある場面では緯度経度の絶対値が大きく、別の場面では小さい値で表されると、ネットワークは同じ意味の入力を別物として学習させられ、汎化が著しく難しくなります。自車中心座標へ正規化することは、画像で言えば「常に同じ向き・同じ縮尺に揃えてから入力する」ことに相当し、学習の効率と汎化を大きく助けます。

## 基礎: 前提となる概念

**WGS84(World Geodetic System 1984)** とは、地球の形と座標の基準を定めた世界標準です。1984年に米国防総省系の機関(現NGA)が策定し、仕様は NIMA Technical Report 8350.2 にまとめられています。GPS が位置を返すときの座標系がこれで、地球を完全な球ではなく、わずかに扁平な回転楕円体(赤道半径 a = 6,378,137 m、極半径 b ≈ 6,356,752 m、扁平率 f ≈ 1/298.257223563)として近似します。緯度(latitude)は赤道からの南北の角度、経度(longitude)は本初子午線からの東西の角度です。「度」という角度の単位である点が重要です。

**座標系の右手系規約**。自動運転やロボティクスでは、自車ローカル座標を「X=前方、Y=左、Z=上」と取るのが一般的です(ROS の REP-103 もこの右手系を推奨)。これは右手の親指をX(前)、人差し指をY(左)に向けると中指がZ(上)を向く関係で、回転の符号や外積の向きを一貫させるための約束事です。「右手系(right-handed coordinate system)」とは、この3軸の向きの取り方の一族を指します。なお自動車工学では Y を右、Z を下に取る「右手系だが向きが逆」の規約(車両座標系の ISO 8855 など)もあり、規約の混在が後述の「静かなずれ」を生む典型的な原因になります。

**ヘディング(heading)とヨー(yaw)**。ヘディングとは、自車が今どちらを向いているかを表す角度です。測量・航法の世界ではコンパス式で「真北を0として時計回り(北→東→南→西)に増える」ラジアン/度で表すことが多いです。一方、数学の標準は「東を0として反時計回り」です。さらに姿勢推定の文脈では Z 軸まわりの回転角を「ヨー(yaw)」と呼び、ピッチ(縦の傾き)・ロール(横の傾き)と区別します。どの規約を採るかで変換式の三角関数の置き方が変わるため、実装では必ず明示する必要があります。

**地図投影(map projection)** とは、丸い地球(曲面)を平らな座標(平面)に対応づける方法です。曲面を平面に潰す以上、距離・角度・面積のすべてを同時に正しく保つことは原理的に不可能で(これはガウスの驚異の定理、Theorema Egregium に由来します。曲面の内在的な曲率は等長変換で消せないため、平面へ歪みなく展開できません)、どれを優先するかで多数の投影法が存在します。

## 仕組みを詳しく

自車ローカル変換の典型的な関数は、入力として「変換したい点群の緯度・経度の配列」「自車の緯度・経度」「自車のヘディング」を受け取り、(N, 2) または (N, 3) の `(x_forward, y_left[, z_up])` メートル配列を返します。中身は2段階です。

### ステップ1: 緯度経度の差をメートルに直す(正距円筒近似)

地球を局所的に平らな板とみなす「正距円筒近似(equirectangular approximation)」を使います。自車の近傍(数百m〜数km)に限れば、地球の曲率は無視できるほど小さく、緯度経度の差をそのままメートルに比例変換できます。

```
deg_to_m = a * pi / 180.0          # 緯度1度あたりのメートル(約111,320 m)
cos_lat  = cos(radians(ego_lat))   # 自車緯度の余弦
east  = (lon - ego_lon) * cos_lat * deg_to_m   # 自車から見た東向き距離(m)
north = (lat - ego_lat) * deg_to_m             # 自車から見た北向き距離(m)
```

ここで余弦 `cos_lat` を掛けるのが核心です。子午線(南北)方向は緯度がどこでもほぼ同じ間隔ですが、緯線(東西)方向は極に近づくほど縮みます。赤道では経度1度が約111kmですが、緯度60度では約半分(cos 60° = 0.5 倍)、緯度80度では約 0.17 倍になります。これは地球を地軸に垂直に輪切りにすると、高緯度ほど輪(緯線円)の半径が小さくなる、という幾何から来ます。この縮みを `cos(緯度)` で補正することで、地図上の各点が「自車から東に何m、北に何m」という正しいメートル量になります。`a = 6,378,137 m` は WGS84 の赤道半径で、同じパイプライン内の他の地図処理(ラスタ地図のレンダリング等)と値をそろえておくことが、2つの地図表現の座標感覚を一致させる上で重要です。厳密には子午線方向の「度あたりメートル」も緯度で微妙に変わりますが(楕円体のため)、その変動は 0.7% 程度であり、近傍用途では定数で十分です。

### ステップ2: 自車の向きに合わせて回転

ステップ1で得た east/north は、まだ「地球の方角(東・北)」です。これを「自車の前・左」へ回転します。ヘディングがコンパス式(0=北、東向きに増加)のラジアンであるとき、自車の前方単位ベクトルは east-north 座標系で `(sin h, cos h)`、左方向はそれを反時計回りに90度回した `(-cos h, sin h)` になります。

```
sin_h, cos_h = sin(ego_heading), cos(ego_heading)
x_forward =  east * sin_h + north * cos_h     # 前方成分への射影
y_left    = -east * cos_h + north * sin_h     # 左方向成分への射影
```

これは east/north で表された各点を、自車の前方軸・左軸へ内積で射影しているだけです。行列で書けば 2×2 の回転行列の適用に等しく、結果は X=前方、Y=左 のメートル座標になります。地図が平面(高さ情報なし)なら Z は 0 のまま扱い、`pts_dim=2` なら xy だけ、`pts_dim=3` なら z を加えます。

### 数値例で確認

自車が真北(heading=0)を向いているとします。`sin_h=0, cos_h=1` なので `x_forward = north`, `y_left = -east`。つまり「北=前」「西=左(東の逆)」となり、北を向いて立ったときの直感と一致します。次に自車が真東(heading=π/2、`sin_h=1, cos_h=0`)を向いているとき、`x_forward = east`(東=前)、`y_left = north`(北=左)。東を向けば北は左手側に来る、これも直感どおりです。

before/after で言えば、入力 `(lat=35.6815, lon=139.7675)` という地球座標が、自車 `(35.6810, 139.7670, heading=0)` を基準に `(x_forward ≈ +55.6 m, y_left ≈ -45.2 m)` という「前方56m・右45m」の自車相対メートル座標になる、というイメージです。tensor shape で言えば、入力が緯度経度の `(N, 2)` 配列で、出力も同じ `(N, 2)` のメートル配列。バッチ処理する場合は `(B, N, 2)` を一括でブロードキャスト演算でき、ループを書かずに済むのが実装上の利点です。

## 手法の系譜と主要論文

座標変換そのものは測地学・地図投影の古典的な基礎であり、特定の論文の「提案」というより数百年積み上げられた標準技術です。系譜としては次のように整理できます。

- 正距円筒図法(equirectangular / plate carrée)は紀元1世紀ごろのマリヌス・ティルスにさかのぼる最古級の投影で、緯度経度を等間隔格子にそのまま割り当てます。USGS の標準教科書である Snyder "Map Projections — A Working Manual"(USGS Professional Paper 1395, 1987)に各投影の歪み特性が体系的にまとめられています。狭い範囲では平面近似として十分精度が出て、計算が四則演算と三角関数だけと軽量なので、車載のリアルタイムなローカル変換に向きます。
- WGS84(NIMA TR8350.2, 第3版 1997)は、GPS が採用する地球楕円体の基準系を定めました。これにより世界中のセンサが共通の絶対座標を出せるようになり、後段で自車相対へ落とす変換が普遍的に必要になりました。
- より高精度が要る測量・建図用途では UTM(Universal Transverse Mercator、地球を6度幅のゾーンに分け各ゾーンを横メルカトル投影でメートル化)や、局所接平面 ENU(East-North-Up、基準点での接平面に投影)が使われます。ENU は本質的にステップ1と同じ発想を楕円体上で厳密化したもので、ECEF(地球中心地球固定直交座標)を経由して基準点の接平面へ回す行列演算で得られます。
- 自動運転の地図系では、SDTagNet(Immel et al., NeurIPS 2025)のように、OSM 等の標準定義地図(SD マップ、Standard-Definition map: 車線数や形状の細かい幾何は持たないが道路ネットワークの位相を持つ軽量地図)をベクトルトークンとして符号化する手法が登場しました。この種の手法では、地図要素を自車中心座標へ落とす前処理が不可欠で、データセット固有のシティ座標変換ではなく、緯度経度から直接落とす汎用 equirectangular 変換へ一般化する流れがあります。HD マップ(High-Definition map、車線レベルの精密幾何を持つ)を学習に用いる nuScenes 系・Argoverse 系のベンチマークでも、地図と知覚を結ぶ前段にこの種の座標統一が暗黙に入っています。

## 論文の実験結果(定量データ)

座標変換単体のベンチマークというものは存在しませんが、「どの近似がどの範囲でどれだけ誤差を出すか」という定量感覚は研究上重要です。

正距円筒近似の誤差は、基準点からの距離の二乗に比例して増えることが知られています(地球の曲率の二次項が効くため)。経験的な目安として、基準緯度から数km以内なら水平誤差はおよそ数十cm〜1m未満に収まり、SD マップ(車線中心線や交差点形状のおおまかな幾何)の用途では無視できます。一方で 50〜100km スケールになると誤差は数十m〜数百mに達し、近似は破綻します。これが「small-window(小さな窓)近似」と呼ばれる所以です。具体的には、曲率による誤差はおよそ `d^2 / (2R)`(d は基準点からの距離、R は地球半径約6371km)のオーダーで効き、d=1km なら約 0.08m、d=10km なら約 7.8m、d=100km なら約 785m と二次で膨らみます。

比較として、UTM は1ゾーン内(東西約数百km)でスケール誤差を 0.0004(0.04%)以内に抑える設計で、測量級の精度を出します。ENU は基準点近傍で厳密にメートル正確です。アブレーション的に言えば、ステップ1の `cos(緯度)` 補正を外すと、緯度の高い地域ほど東西方向が系統的に伸び、たとえば緯度60度では東西距離が約2倍に膨れて地図が左右に引き伸ばされます。ステップ2のヘディング回転を誤ると、地図全体が自車の向きから一定角だけ回転してずれ、後段の地図特徴との対応づけ(クロスアテンション等)が系統的に崩れます。ヘディング推定の誤差1度は、前方50mの点で約0.9m(50m × sin1° ≈ 0.87m)の横ずれを生む計算になり、これが認識・予測性能に直結します。地図と知覚の整合誤差が数十cm〜1mのオーダーで増えると、車線中心線への寄せや交差点での進路予測の精度が目に見えて落ちる、というのが地図融合研究で共有された経験則です。

## メリット・トレードオフ・限界

メリット

- 緯度経度という扱いにくい角度座標を、モデルが直接使える自車中心メートル座標へ統一できる。入力の正規化として学習を安定・汎化させる。
- 正距円筒近似は四則+三角関数だけで軽量。フレームごとに数千〜数万点を高速に変換でき、リアルタイム前処理に向く。ベクトル化(行列演算)でき GPU/CPU いずれでも安価。
- GPS さえあればデータセットに依存せず使える。データセット固有のシティ座標系に縛られない。

トレードオフ・限界

- 正距円筒近似は広域で歪む。あくまで自車近傍(数km以内)の局所近似であり、それを超えると誤差が二次的に増大する。
- ヘディング(自車の向き)の推定誤差がそのまま地図全体の回転ずれになる。GPS/IMU/オドメトリのヘディング品質が地図整合の上限を決める。市街地の GPS マルチパス(ビルでの反射による位置跳ね)はこの誤差を増幅する。
- 地球を局所平面とみなす単純化のため、測量級の精度は出ない(SD マップ用途では問題にならないが、HD マップや測量では UTM/ENU/ECEF を使うべき)。
- 高さ(Z)を 0 とみなす実装が多く、勾配のある地形や立体交差では高さ方向の整合が取れない。

未解決・実務的な課題として、ヘディングの定義(コンパス式か数学式か、ラジアンか度か)、原点緯度の取り方(自車か固定基準点か)、楕円体補正の有無といった規約の不一致は、複数のコンポーネントを連結する際に「静かなずれ(silent misalignment)」を生む典型的なバグ源です。エラーは出ないのに地図がじわっとずれ、精度だけが下がるため発見が難しい。実装間でこれらの規約を文書化して固定し、合成データ(既知の真値を入れて出力を検算する単体テスト)で検証することが重要です。

## 発展トピック・研究の最前線

- 高さを考慮した3D整合: 勾配・立体交差を扱うため、Z を地形DEM(数値標高モデル)やLiDAR地面推定から補う研究。
- 楕円体厳密変換: ECEF を経由して ENU へ落とす厳密変換は、広域SLAMや複数都市にまたがるデータ統合で使われる。equirectangular との誤差を距離関数として定量比較し、用途ごとに使い分ける評価が重要になる。
- 学習可能な座標整合: 地図と知覚特徴のずれを、固定式の変換ではなくネットワークが学習で吸収する(deformable attention で参照点位置を微調整する等)アプローチが、地図ノイズ・ローカリゼーション誤差に頑健とされる。BEVFormer 系の空間クロスアテンションがこの思想に近い。
- ローカリゼーション誤差への頑健性: GPS マルチパスやヘディングドリフトを前提に、地図特徴の取り込みを誤差に頑健化する研究は、地図を使う E2E 自動運転の信頼性で活発な論点になっている。地図を「正解」ではなく「ノイズ付き事前情報」として扱う確率的融合が一つの方向。
