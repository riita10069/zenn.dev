# ラスタマップの色ルックアップによるDAC代理

## ひとことで言うと

自動運転の計画モデルは、これから車がたどる軌道(経路の候補)を何本も生成し、その中から一番良いものを選びます。選ぶときに「その軌道は道の上を走っているか、歩道や路外にはみ出していないか」を測りたい。これを走行可能領域適合(DAC)と呼びます。本来 DAC は、画素ごとに「ここは道路か歩道か」を当てる学習済み分類器で計算します。色ルックアップによる DAC 代理とは、その学習をいったん省き、すでに色分けして描かれている地図画像(ラスタマップ)の該当画素の色を直接読んで「ここは道路色か?」を見るだけにする近似手法です。学習ゼロ、計算は画素を1点読むだけで、組込み向けに極めて安上がりです。

## 直感的な理解

地図アプリの画面を思い浮かべてください。道路はグレー、建物はベージュ、公園は緑、と色分けされています。人間がそれを見て「この点は道路の上だ」と判断するとき、わざわざ「これは道路という意味カテゴリだ」と推論しているのではなく、単に色を見ています。色ルックアップはこの素朴な発想をそのままアルゴリズム化したものです。

なぜわざわざ学習済み分類器を使わずに済ませたいのか。学習済みの意味セグメンテーションヘッド(画素ごとに道路・歩道などのカテゴリを推定するモデル)を一から作るには、(1) 画素ごとの意味ラベルが付いた大量のデータ、(2) 学習のための GPU 時間、(3) 新しい損失関数の設計、が必要です。プロジェクトの初期段階や、確実な地図がすでに手元にある状況では、この投資は過剰になりがちです。地図が正しく色分けされて描かれているなら、その色を読むだけで「道路かどうか」は判定できる。学習する必要がない情報を、わざわざ学習し直すのは無駄だ、というのが動機です。

## 基礎: 前提となる概念

用語を一つずつ噛み砕きます。

- DAC(Drivable Area Compliance, 走行可能領域適合): 軌道が「走ってよい場所(車道)」にどれだけ収まっているかの度合い。歩道に乗り上げたり中央分離帯を越えたりしていれば低くなります。自動運転の標準ベンチマーク(後述の nuScenes ベースの評価など)で、安全性を測る代表指標の一つです。
- ラスタマップ / ラスタ化(rasterized map): 道路・車線・横断歩道などの地図情報を、ピクセルが格子状に並んだ画像(写真のような形式)として描いたもの。対義語のベクタ地図(vector map)は「線分や多角形の座標リスト」で地図を持つ形式で、ラスタ化はそのベクタ地図を画像に焼き付ける作業を指します。なぜ画像にするかというと、畳み込みニューラルネットワーク(CNN, 画像処理が得意なネットワーク)がそのまま食えるからです。
- 分類器(classifier): 入力を見てカテゴリを当てる学習済みモデル。DAC 分類器なら「この軌道地点は走行可能領域か否か」を当てます。
- BEV(Bird's-Eye View, 鳥瞰図): 真上から見下ろした 2D 平面の表現。地図やセンサ情報を上から見た格子として持つと、車の位置や軌道との対応が取りやすくなります。

DAC を計算する正攻法は、BEV 上で「各画素が走行可能領域か」を推定する学習済みセグメンテーションヘッドを持ち、そのマスクと軌道地点を照合することです。色ルックアップは、このセグメンテーションヘッドの代わりに「すでに色で意味が描き込まれた地図画像」を流用します。地図に描かれた色そのものが、人間が用意した正解ラベルの役割を果たすわけです。

## 仕組みを詳しく

軌道は T ステップ(たとえば T=6、つまり 6 点)からなり、各点が自車座標系での (x, y) 座標(単位はメートル)を持つとします。

### ステップ1: 軌道地点を BEV 画素座標に変換

各 waypoint(軌道上の点)の自車座標 (x, y)[m] を、地図画像の画素位置 (row, col) に変換します。この変換には次の3つのパラメータが要ります。

- pixels_per_meter: 1 メートルが画像上で何画素分か(解像度)。たとえば 1 画素 = 0.4m なら pixels_per_meter = 2.5。
- ego_row, ego_col: 自車(原点)が画像のどの画素に位置するか。

概念的には単純な拡大縮小+平行移動です。

```
col = ego_col + x_m * pixels_per_meter   (横方向)
row = ego_row - y_m * pixels_per_meter   (前後方向。上が前なら符号は -)
```

たとえば 0.4m/画素、自車が画像中央付近にいるとして「前方 12m, 右 2m」の点は、ego から右に 2*2.5=5 画素、前に 12*2.5=30 画素の (row, col) に落ちます。この変換の正しさが全体の生命線です。1つのパラメータがレンダラの取り決めとずれただけで、全軌道地点が誤った画素を指し、判定が無意味になります(詳細は [BEV画素投影のキャリブレーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1029_bev-pixel-projection-calibration))。

### ステップ2: その画素の色を読む

地図画像のその (row, col) 画素の値(色チャンネル)を読み、道路を表す色に近いかどうかを判定します。

- 画素が道路色 → その waypoint は「適合」
- 画素が路外色・歩道色 → 「非適合」
- 画素が画像範囲外(地図に描かれていない遠方) → 領域外として扱う(境界処理を明示的に決めておく必要がある)

色の照合の仕方は、レンダラ(地図を画像に焼く処理)が意味をどう色に割り当てたかで2通りに大別できます。一つは RGB 連続色での照合で、道路を表す基準色(たとえば中間グレー (128,128,128))との距離が閾値以内かを見ます。式で書くと、画素の色を c=(r,g,b)、基準色を c_road としたとき `‖c − c_road‖ < τ` なら適合、というユークリッド距離の閾値判定です(τ は許容色差)。地図に陰影やアンチエイリアス(境界をなめらかにするため中間色を混ぜる処理)が入っていると、境界画素が「半分だけ道路色」になり、この距離判定が不安定になります。

もう一つは、意味カテゴリごとに別チャンネルを割り当てる one-hot(ワンホット)エンコーディングです。たとえば 8 チャンネルの地図テンソル [8, H, W] を用意し、チャンネル0=道路、チャンネル1=歩道、チャンネル2=横断歩道…と決め、各画素ではその意味のチャンネルだけが 1、他は 0 になっています。この場合、判定は「(row,col) 画素のチャンネル0の値が 1 か」を見るだけで、距離計算も閾値も不要です。アンチエイリアスの問題も起きにくく、ChauffeurNet 以来この多チャンネル方式が地図ラスタ化の主流です。色ルックアップという呼び名は前者の連続色を連想させますが、実装としては後者のチャンネル参照の方が頑健で、両者を同じ枠組みで扱えます。

数値例で具体化します。waypoint が画素 (row=120, col=152) に落ち、地図テンソルが [8,450,300] の one-hot だとします。`map[:, 120, 152]` を読むと `[1,0,0,0,0,0,0,0]` なら道路、`[0,1,0,...]` なら歩道。前者を適合(1)、後者を非適合(0)とカウントします。RGB 方式なら同じ画素から (130,126,131) が読め、基準グレー (128,128,128) との距離が約 4.2 で τ=20 以下なので適合、と判定します。

### ステップ3: 軌道全体のスコアにまとめる

T 点のうち何点が道路色だったか、あるいは適合画素の割合を 0〜1 のスコアにします。6 点中 6 点が道路上なら 1.0、4 点なら約 0.67。これを軌道ごとに計算し、複数候補の再ランク([複数サンプル軌道の再ランク（コンプライアンス・スコアリング）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1050_trajectory-compliance-scorer))に渡します。

```
軌道 [T,2] (x,y in m)
   │ メートル→画素のアフィン変換(投影)
   ▼
画素 (row,col) × T
   │ ラスタ地図画像から該当画素の色を読む
   ▼
[道路色? 道路色? 路外? ...]  → 適合スコア(例 5/6 = 0.83)
```

なぜこの設計が成り立つのか。ポイントは「地図が信頼できる」という前提に全面的に乗っていることです。地図のラスタ化のときに人間(または高精度な地図作成パイプライン)がすでに道路と非道路を正しく塗り分けてくれている。だから推論時に分類器で推定し直す必要がない。逆に言えば、地図が無い場所・誤っている場所では何もできない、という限界がここに埋め込まれています。

## 手法の系譜と主要論文

色ルックアップは特定の1論文が名付けた手法ではなく、「ラスタ地図を画像として直接扱う」という長い系譜の応用です。系譜を追います。

- ChauffeurNet (Bansal et al., RSS 2019, arXiv:1812.03079): Waymo の研究で、道路・車線境界・信号・他車・自車の経路などを複数チャンネルのラスタ画像に描き、それを CNN に入れて運転を学習しました。地図を画像として表現する発想を運転計画に広めた代表例です。ここで「地図=画像」という考え方が確立しました。
- CoverNet (Phan-Minh et al., CVPR 2020) と MultiPath (Chai et al., CoRL 2019): ラスタ地図を入力に、軌道予測を「あらかじめ用意した軌道候補集合からの分類問題」として解きました。軌道がどこを通るかを画素単位の地図文脈と照合する流れを定着させました。
- LSS: Lift-Splat-Shoot (Philion & Fidler, ECCV 2020, arXiv:2008.05711) や BEV セグメンテーション系: 地図が無い場所でも、カメラセンサから走行可能領域を BEV 上に推定できることを示しました。色ルックアップが「正しい地図がある前提」に乗るのに対し、これらは「センサから地図的情報を作り出す」立場で、地図のない領域でも機能する強みがあります。
- GoalFlow (Xu et al., arXiv:2503.05689, 2025): 軌道生成にフローマッチングを使い、生成した複数候補を DAC を含むスコアで再ランクします。GoalFlow の DAC は学習済みの BEV セグメンテーションから計算されます。色ルックアップは、この学習済み DAC を「学習なしで近似する代理(proxy)」と位置づけられます。確実なラスタ地図がある状況では、学習済みセグメンテーションを省いても近い判定が得られる、という割り切りです。

- nuPlan / NAVSIM のシミュレーション評価系(2023-2024): 計画モデルの良し悪しを、地図上での走行可能領域逸脱・衝突・進行度などのルールベース指標で測る枠組みが整いました。ここで DAC は「地図のどのポリゴンの内側に軌道点が入っているか」という幾何判定で計算されます。色ルックアップはこの幾何判定(点がポリゴン内かを毎回計算する)を、あらかじめポリゴンをラスタ化して画素を引くだけに置き換えた、計算量を O(点×ポリゴン辺) から O(点) に落とす最適化とも見なせます。

整理すると、ChauffeurNet が「地図を画像にする」を、CoverNet 系が「軌道を地図文脈で評価する」を確立し、nuPlan/NAVSIM が「ルールベースで軌道を採点する」を標準化し、GoalFlow が「生成候補をスコアで選ぶ」を持ち込んだ。色ルックアップはこれらの交点で、評価部分を学習なしの色(チャンネル)照合に置き換えた最も軽量な実装、と言えます。手作りスコアという意味では ChauffeurNet が導入した補助損失(車線逸脱・衝突を罰する手作りペナルティ)とも思想を共有します。ChauffeurNet はそうした手作りの幾何ロスを学習に組み込むことで、模倣学習が陥りがちな「データ分布からの逸脱で破綻する」問題を緩和できると報告しました。

## 論文の実験結果(定量データ)

DAC という指標がベンチマークでどう効くかを、関連研究の数値で見ます。

- nuScenes ベースの計画ベンチマーク(NAVSIM など)では、軌道の良し悪しを PDM Score(Predictive Driver Model Score)という複合指標で測ります。その構成要素に DAC が入っており、走行可能領域からの逸脱はスコアを直接押し下げます。GoalFlow (arXiv:2503.05689) は NAVSIM の navtest 分割で PDMS 約 90.3 を報告し、当時の上位成績を示しました。この複合スコアは衝突なし(NC)、走行可能領域適合(DAC)、自車進行(EP)、快適性(Comfort)、到達時間(TTC)などの積で構成され、どれか一つが低いと全体が大きく下がる設計です。つまり DAC を正しく測り、DAC の低い軌道を弾けるかどうかが、最終スコアに直結します。
- アブレーション(要素を抜いて効果を測る実験)の観点では、再ランク(複数候補から良い軌道を選ぶ後処理)を外すと、生成された最良候補をそのまま使う場合より計画品質が落ちることが、GoalFlow 系および軌道スコアリング全般の報告で共通して見られます。再ランクの中で DAC が「物理的にあり得ない・道路外の軌道」を排除する役割を担うため、DAC 項を抜くと衝突や逸脱が増える方向に効きます。
- 学習済みセグメンテーションの精度自体は、LSS や BEVFormer 系で BEV の走行可能領域 IoU(Intersection over Union, 予測領域と正解領域の重なり率。1 に近いほど良い)が報告され、おおよそ数十パーセント台です。これは「センサから走行可能領域を推定するのは依然として難しい」ことを意味します。色ルックアップが地図に頼るのは、この推定の難しさを丸ごと回避するためでもあります。地図が正しければ実質 IoU 100% 相当の走行可能領域マスクが手に入る、という見方ができます。

数値の意味を補足します。PDMS のような複合スコアは「掛け算」で組むことが多く、DAC が 0.5 なら他の項がどれだけ良くても全体が半減する。だから DAC を正確に測ること自体が、最終性能のレバーになります。一方 IoU は「重なり具合」なので、推定マスクが正解からずれるほど下がる。地図ルックアップはこのずれを地図品質に肩代わりさせている、と理解できます。

## メリット・トレードオフ・限界

メリット

- 学習・新規ヘッド・新規損失が一切不要で、すぐ使える。
- 計算が極めて軽い(画素を1点読むだけ)。組込み・量産向け。
- 地図が正しく描かれている限り、走行可能/不可の境界を画素精度で判定できる。

トレードオフ・限界

- 座標キャリブレーション依存。投影パラメータがずれると全判定が無意味になり、しかも出力はもっともらしい数字なので誤りに静かに気づけない(silent failure)。これが最大のリスクです([BEV画素投影のキャリブレーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1029_bev-pixel-projection-calibration))。
- 色は意味が単純。バス専用レーン・一時的な工事規制・道路種別など、細かい区別ができない。学習済みセグメンテーションなら文脈から推定できる微妙な領域を取りこぼします。
- 動的障害物(他車・歩行者)は地図に描かれないので、このルックアップでは避けられない。地図ベースの DAC は静的な「道かどうか」しか見ません([カメラのみ構成とLiDARのトレードオフ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1032_camera-only-vs-lidar-tradeoff))。
- 地図が無い/誤っている領域では完全に無力。センサから推定する手法が機能する場面でも何も返せません。
- 連続値ではなく色の閾値判定になりやすく、道路境界の真上などでは判定が不安定になりうる。

研究上の未解決課題としては、(1) 色閾値判定を、境界付近で連続的に変化する soft なスコアにして安定化できないか、(2) 地図品質が場所ごとに異なる現実で、地図の信頼度に応じてルックアップと学習済み推定をブレンドできないか、(3) 動的物体を別チャンネルとして地図に焼き込み、静的・動的を統合的にスコアリングできないか、といった方向があります。

## 発展トピック・研究の最前線

近年は「地図に頼らない(mapless)」運転が大きな流れです。MapTR (Liao et al., ICLR 2023) や VectorMapNet (Liu et al., ICML 2023) はセンサからベクタ地図そのものをオンラインで生成し、事前 HD マップへの依存を減らそうとします。これが進めば、色ルックアップの前提である「事前に正しく塗られた地図」自体をセンサから作り、その上で同じ色照合を回す、というハイブリッドが考えられます。

また、軌道評価を学習ベースのスコアラ(value function や reward model)で行う流れも強く、Hydra-MDP (Li et al., 2024) のように複数の評価指標を蒸留して一つのスコアリングヘッドにまとめる研究があります。色ルックアップのような手作りルールは解釈しやすく軽い反面、こうした学習スコアラに比べると表現力で劣ります。両者をどう使い分けるか(安全に関わる硬い制約は手作り、微妙な好ましさは学習)が実務的な論点です。
