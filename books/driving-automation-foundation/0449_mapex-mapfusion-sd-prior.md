# SD-map prior融合 (MapEX等)

## ひとことで言うと
すでに分かっている地図情報(カーナビ的な粗い SD 地図や、一部だけ作ってある HD 地図)を、ニューラルネットへの「ヒント」として与え、足りない部分・まだ分かっていない部分だけをセンサーで補って地図を完成させる手法です。MapEX(Map EXisting priors、既存地図の活用)はその代表で、「ゼロから地図を作り直す」のではなく「既存地図を出発点にして埋める」発想を取ります。地図づくりを「白紙から」ではなく「下書きに加筆する」作業に変える、と言い換えてもよいでしょう。

## 直感的な理解
オンライン HD マップ構築(走りながらその場で地図を作る方式)の主流は、毎回まっさらな状態から地図を生成します。つまり「すでに正確な地図がある場所」でも、毎フレーム最初から作り直しています。これは計算効率が悪いだけでなく、せっかく手元にある正確な情報を捨てる行為でもあります。同じ交差点を何百回通っても、毎回ゼロから車線を引き直しているわけです。

現実の道路には、ほぼ必ず何らかの事前情報があります。カーナビ用の SD 地図(Standard Definition map、道のつながりや車線本数の大雑把な情報を持つ地図。OpenStreetMap のような無償の公開地図もこれに含まれる)は世界の大半の道路に存在します。SD 地図は形状の精度は低い(車線の正確な位置までは持たない)代わりに、道路の接続関係(この道は次の交差点であの道につながる)を広域に持っています。また過去に測量した HD 地図(High Definition map、車線の正確な形状まで持つ高精度地図)が、一部区間だけ残っていることも多い。

人間に例えると、初めての街でも手元に地図アプリがあれば「この通りには車線が2本ある」と知った上で運転を始め、実際に走って「あ、工事で1本増えている」と差分だけ修正します。地図ゼロから記憶を起こすより遥かに楽で正確です。MapEX 系がやろうとしているのは、まさにこの「既存地図を出発点にして、変わった分だけセンサーで上書きする」プロセスの機械化です。鍵は、既存地図という外部知識を、どうやって地図生成ネットワークの内部に自然に注入するか、という設計問題に帰着します。注入の仕方には大きく「ラスタ(画像)として混ぜる」流派と「クエリ(問い合わせ票)として混ぜる」流派があり、両者は得意領域が異なります。

## 基礎: 前提となる概念
この手法を理解するには、まず MapTR 系のオンライン地図構築の「クエリ」という仕組みを知る必要があります。

Transformer のクエリ(query)とは、ざっくり言えば「これから探すものの問い合わせ票」です。地図生成では「1本の地図要素(車線や境界)を担当する予約席」だと考えると分かりやすい。たとえば「クエリを100個用意する」とは「最大100本の地図要素を出せる枠を用意する」という意味です。各クエリは最初はランダムな初期値の数値ベクトルで、Transformer デコーダの中で BEV 特徴(鳥瞰図に投影したマルチカメラ特徴)を cross-attention で参照しながら、「自分が担当する車線はここで、形はこの点列だ」と中身を徐々に埋めていきます。最終的に各クエリから、順序付き点列(ポリライン)としての地図要素が1本出力されます。ポリライン表現については [ポリラインエンコーディング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0460_polyline-encoding) を参照してください。

ポリラインとは、車線や境界を順番の付いた点の列(折れ線)で表したものです。「車線A = (0,0),(2,1),(4,3),...」のように、形を座標の列でコンパクトかつ正確に表せます。MapTR 系ではこの点列をクエリの内部に埋め込み、デコーダで点の座標を回帰(数値を予測)します。

二部マッチング(bipartite matching、二部グラフマッチング)とは、出力したクエリと正解の地図要素を一対一で対応づける割り当て問題です。MapTR 系は「どのクエリがどの正解を担当したか」を学習中に最適割り当て(ハンガリアン法という、総コスト最小の割り当てを求めるアルゴリズム)で決め、その対応に基づいて損失(予測と正解のズレ)を計算します。これにより「クエリに順番を付けて教師を割り当てる」必要がなくなり、出力の集合をそのまま正解の集合と照合できます。MapEX で「既存地図をどのクエリに割り当てるか」を考えるとき、この対応づけの考え方が基礎になります。

## 仕組みを詳しく
MapEX の核心は、クエリの初期値に既存地図を埋め込むことです。普通の MapTR は全クエリがランダムスタートですが、MapEX は次のようにします。

ステップ1: 既存地図要素をクエリに変換する。
手元の既存地図(SD 地図から得た「ここに車線が1本ある」という情報や、過去 HD 地図の正確な折れ線)を、それぞれ1つのクエリに変換します。このクエリは「私はこの場所のこの車線を担当する。形は既存地図のこれ。あとはセンサーで微調整して」という、中身が半分埋まった予約席になります。これを既知クエリ(known query / prior query)と呼びます。実装としては、既存地図のポリラインを点列として埋め込み、それをクエリの位置成分(reference points、参照点)・内容成分(content embedding)の初期値に流し込む形が典型です。種類ラベル(車線か境界か)が分かっていればそれもクエリに付与します。

ステップ2: 足りない分はランダムクエリで補う。
既存地図に載っていない要素(新設された車線、SD 地図には無い横断歩道など)を拾うため、従来どおりランダム初期値のクエリも一定数混ぜます。これを未知クエリ(unknown query)と呼びます。

```
  クエリ集合 = [ 既知クエリ ×k ] + [ 未知クエリ ×u ]
                既存地図から生成        ランダム初期値
  例: 既存地図に車線3本 → 既知3 + 未知(100-3)=97
      合計100クエリでデコーダへ
```

ステップ3: Transformer で全クエリを更新する。
既知クエリと未知クエリをまとめて Transformer デコーダに通し、各層で BEV 特徴を cross-attention で参照させます。既知クエリは「既存地図の位置をセンサーに合わせて微修正する」軽い仕事をし、未知クエリは「ゼロから新しい要素を見つける」重い仕事をします。役割分担が自然に起き、全体として効率が良く、既存地図の正確さが最終出力に活きます。学習時の二部マッチングでは、既知クエリと未知クエリを区別したマッチング規則を設けることもあり、既知クエリが対応する既存要素を担当しやすくする工夫が入ります。

ステップ4: 出力。
全クエリから最終的な地図要素(各クエリ → ポリラインとクラス確率)を出します。既存地図が正しかった部分はほぼそのまま保たれ、変わった/新しい部分だけが更新された地図になります。

具体例で追います。ある区間の HD 地図が過去に作られていて 3本の車線が記録されているとします。ところが工事で1本増えて4本になった。MapEX は既知クエリ3つで元の3本を保持しつつセンサーで微修正し、未知クエリのどれかが新しい4本目を検出します。結果、ゼロから4本作るより速く正確に、変化分だけ反映した地図が得られます。

設計上の最重要ポイントは「既存地図が間違っていても壊れにくい」ことです。既知クエリはあくまで初期値・ヒントであって固定ではありません。Transformer デコーダの各層で BEV 特徴を参照して内容が書き換わるため、センサーが「ここに車線はもう無い」と強く示せば、ネットワークはその既知クエリの出力を抑制(分類スコアを下げて棄却)できます。古い・不完全・ノイズ入りの prior でも安全に使えるよう、論文では prior の完全性をいくつかの水準(完全に正確/一部の要素だけ与える部分 prior/座標が少しずれた prior/種類ラベルが欠けた prior など)に分けて体系的に扱っています。この「prior の不完全さを段階的に想定して頑健性を検証する」姿勢が MapEX の方法論上の貢献でもあります。

## 手法の系譜と主要論文
SD/HD prior の活用という流れを系譜で追います。

MapTR(Liao, Chen, Wang, Cheng, Zhang, Liu, Huang, ICLR 2023, "MapTR: Structured Modeling and Learning for Online Vectorized HD Map Construction", arXiv:2208.14437)が土台です。地図要素を「順序を入れ替えても同じ形を表す点集合」として扱い、Transformer のクエリで直接ベクトル出力する枠組みを確立しました。続く MapTRv2(Liao et al., IJCV 2024, arXiv:2308.05736)が補助損失(深度監督、PV と BEV の補助セグメンテーション)やデコーダ改良で精度と収束速度を改善しました。これらは「ゼロから生成する」純粋なオンライン地図構築です。詳細は [InstaGraM / インスタンスマスク地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0450_maplt-tr-instance-mask) を参照してください。

ここから「事前情報を使おう」という二つの流れが分かれます。一つは外部 SD 地図をラスタ(画像チャンネル、BEV グリッドの追加の層)として注入する系統で、P-MapNet(Jiang, Mei, Hou, et al., RA-L 2024, "P-MapNet: Far-seeing Map Generator Enhanced by Both SDMap and HDMap Priors", arXiv:2403.10521)が代表です。SD 地図を BEV グリッドの追加チャンネルとして与えて attention で融合し、さらに過去 HD 地図を MAE(Masked Autoencoder、入力の一部を隠して残りから元を復元する事前学習)で補完に使います。詳細は [P-MapNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0458_p-mapnet) を参照してください。同系統で SD 地図を BEV にラスタライズして特徴に連結する研究は複数あります。

もう一つが本トピックの MapEX(Sun, Bartoccioni, Bursuc, Boulch, Vu, Cord, et al., 2024, "Mind the Map! Accounting for Existing Maps in the Online Vectorized HD-Map Construction", arXiv:2311.10517)で、既存地図をクエリ初期値として与えるアプローチです。同系統の発展として、SD 地図の道路グラフ構造を Transformer でエンコードして注入する SMERF(Luo, Zheng, et al., 2024, "Augmenting Lane Perception and Topology Understanding with Standard Definition Navigation Maps")や、トポロジー推定に SD prior を効かせる研究群があります。注入の仕方が「ラスタ融合」(P-MapNet)か「クエリ注入」(MapEX)か「グラフエンコード」(SMERF)かで設計思想が分かれ、いずれも「既存地図を捨てない」という共通の問題意識から出発しています。

## 論文の実験結果(定量データ)
MapEX 系は主に nuScenes(6カメラ + LiDAR を備えた大規模自動運転データセット)で評価されます。地図要素は車線分割線・道路境界・横断歩道の3クラスで、推定と正解のチェンファー距離(点列どうしの平均最近傍距離)が一定しきい値(0.5/1.0/1.5m)以内なら当たりとし、mAP(平均適合率の平均、precision と recall を総合した 0〜100% の値、高いほど良い)で測ります。

報告される傾向は明快です。第一に、既存地図(prior)を与えるほど精度が上がり、prior が完全に近いほど mAP が大きく改善します。何も与えない素の MapTR ベース(nuScenes で mAP おおむね 50 前後)に対して、正確な prior を与えると mAP が大きく跳ね上がり、prior がほぼ完全な構成では mAP が 80 を超える水準まで達したと報告される例もあります。これは「正しい地図がある場所では作り直す必要がない」という直感を定量的に裏づけます。第二に、prior にノイズ(位置ずれや要素欠落)を加えても精度の劣化が緩やかで、prior が完全でなくても恩恵が残ります。これは既知クエリが固定でなくセンサーで上書き可能な設計が効いている証拠です。

アブレーションで重要なのは、prior の完全性を段階的に落とす実験です。「位置は正しいが種類ラベルが欠けた prior」「一部の要素だけ与えた部分 prior」「座標が少しずれた prior」などを比較し、いずれの不完全さに対しても破綻せず段階的に性能が変化することが示されました。たとえば prior に与える要素の割合を 100% から減らしていくと mAP も滑らかに下がり、急激に崩壊しない(catastrophic な失敗をしない)ことが確認されています。P-MapNet 側の報告でも、SD prior を加えると遠方(自車から離れた 60m 以遠など、カメラ解像度が落ちて単独では見えにくい領域)の地図精度が特に改善し、MAE 補完を入れると穴の多い領域で mAP が追加で向上したとされます。これらは「prior をどう注入しても、与えれば与えるほど効く。ただし注入方式で得意な領域(近傍か遠方か、形状か接続か)が変わる」という分野の知見を形作っています。

## メリット・トレードオフ・限界
メリットを整理します。既存の SD/部分 HD 地図を出発点にでき、ゼロ生成より効率的で精度が高い。変化分(新設・撤去)だけをセンサーで補えばよいので、計算と学習の負担が「既知の微修正」と「未知の発見」に分かれて軽くなる。prior が不完全・古くても、クエリは固定でなくセンサーで上書きできるため壊れにくい。SD 地図は世界中ほぼどこにでもあるため、適用範囲が広いという実用上の強みもあります。

トレードオフと限界も率直に挙げます。既存地図とセンサーの位置合わせ(GPS や自己位置推定の精度)が前提で、ずれが大きいと既知クエリが間違った場所を指して足を引っ張ります。位置合わせ誤差への頑健性は MapEX が重点的に検証した点ですが、許容範囲を超えれば破綻します。prior の質に最終精度が引っ張られるため、極端に古い・大幅に変わった区間では恩恵が薄れ、場合によっては素のゼロ生成より悪くなる危険もあります(古い車線が消えたのに既知クエリが残し続ける誤検出)。さらに、どの既存要素をどのクエリに割り当てるか(マッチング設計)や、SD と HD で粒度が違う情報をどう揃えるか、といった追加の作り込みが必要です。SD 地図はそもそも形状の精度が低く接続情報が主なので、形状の事前知識としては弱い、という素材側の限界もあります。

未解決の課題としては、prior の信頼度をネットワークが自覚して融合重みに反映する仕組み、地図変化の能動検出(prior と観測の不一致を「変化があった」と判定する)、複数の prior(SD と部分 HD と過去走行)を同時に矛盾なく統合する方法などが研究の対象です。

## 発展トピック・研究の最前線
近年は、地図要素の形状だけでなくトポロジー(車線中心線どうしの接続、車線と信号・標識の対応)まで一体推定する流れが強まっており、SD 地図の最大の強み(広域の接続情報)を活かす方向と相性が良いとされます。SD 地図をグラフとしてエンコードして注入する SMERF 系、時間方向の融合(連続フレームの蓄積、StreamMapNet 的なストリーミング)と prior 融合を組み合わせる研究、そして外部地図に頼らず走行ログから自前の prior を育てる [Neural Map Prior (NMP)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0453_neural-map-prior) との統合などが最前線です。これらは「prior をどこから得るか(外部 SD / 外部 HD / 自己走行)」という軸の上で互いに補完的で、複数 prior の階層的統合が実運用に近い構成として研究されています。

実運用の観点では、部分的に整備されたベクトル地図(車線を点列と接続で持つ標準形式、Lanelet2 などのフォーマット)を最大活用するという要請が、ゼロ生成より prior 注入を選ぶ強い動機になっています。地図ベンダーが持つ既存資産を捨てずに、センサーで差分だけ更新する、という運用は地図の鮮度維持コストを直接下げます。
