# Ingest Adapter (データソース取込みの共通インタフェース)

## ひとことで言うと
形式がバラバラな複数のデータセットを、共通の「窓口(インタフェース)」を通して同じ手順で取り込めるようにする設計です。後段の学習用形式への変換(WebDataset 化など)の前に挟む「形式の違いを吸収する層」だと考えてください。データソースを1つ増やしたいときは、その窓口の約束を満たす小さな変換器を1個書くだけで済み、学習側のコードはまったく変えずに使い回せます。

## 直感的な理解
旅行先で電化製品を使う場面を想像してください。国ごとにコンセントの形が違うので、本体を作り変える代わりに「変換プラグ(アダプタ)」を1個用意します。本体側は常に同じ差し込み口だけを見ていればよく、各国の差はプラグが吸収します。データ取り込みでも同じことをします。

機械学習のデータセットは、提供元ごとに中身の並び方・ファイル形式・項目名がまったく違います。具体的に困る場面を挙げます。あるデータセットは複数台のカメラ映像・速度・ハンドル角を独自のバイナリ形式で持ち、別のデータセットは同じような情報をまったく違うディレクトリ構造と別のキー名で持ち、さらに別のものは独自の圧縮形式を使う、といった具合です。

これらを学習に使うには、最終的に同じ形(たとえば「画像テンソル + 自車の運動 + 目標軌道」というサンプル)に揃えないといけません。もし揃える処理を「データセットA専用の前処理」「データセットB専用の前処理」と完全に別々に書くと、後段の処理(学習用形式への変換、学習ループへの供給)もデータセットの数だけ作り直すことになり、データセットを1つ増やすたびに後段まで含めて手を入れる羽目になります。

この「データソースごとの違い」を1か所に閉じ込め、後段からは常に同じ顔をして見えるようにするのが Ingest Adapter(取り込みアダプタ)です。adapter(アダプタ)とは「形の合わない2つのものをつなぐ変換器」のことです。

## 基礎: 前提となる概念
ここで前提となる用語を噛み砕きます。

インタフェース(interface)とは「あるものを使うために外から見える約束ごと(どんなメソッドがあり、何を受け取り何を返すか)」を指します。中身の実装がどうであれ、約束さえ守っていれば呼ぶ側は同じように扱えます。テレビのリモコンの「電源ボタン」がインタフェースで、内部の電子回路が実装、という関係です。

protocol(プロトコル)は Python での用語で、「このメソッドを持っていれば、どのクラスでも同じ役割として扱ってよい」という共通の型を、継承を強制せずに定義する仕組みです(`typing.Protocol`)。これは「ダックタイピングを型で表現したもの」です。ダックタイピングとは「アヒルのように鳴いてアヒルのように歩くなら、それをアヒルとみなす」という考え方で、つまり「`class X(Base)` のように親クラスを継承していなくても、必要なメソッド(形)さえ持っていれば仲間とみなす」ことです。これにより各 adapter は互いに完全に独立して書け、共通の基底クラスへの依存を持たずに済みます。

iterator / generator とは「サンプルを1件ずつ順番に取り出す仕組み」です。全データを一度にメモリへ展開せず、必要なぶんだけ逐次生成できるので、巨大データセットを扱うのに向きます。PyTorch の `IterableDataset` がこの考え方を採用しています。

## 仕組みを詳しく
中心にあるのは「共通の取り込みプロトコル」です。仮に `IngestAdapter` という protocol を考え、おおよそ次の約束を持つとします。

```
IngestAdapter (protocol)
  - サンプルを1件ずつ列挙する (iterate / __iter__)
  - 各サンプルを「共通サンプル」の形に変換して返す
  - 必要ならソース固有のメタ情報も付ける
```

共通サンプルの形を具体的にイメージすると、たとえば次のような辞書になります(形状は例。`H` は画像の縦、`W` は横、`T` は未来の時間ステップ数)。

```
sample = {
    "images":     Tensor[N_cam, 3, H, W],  # カメラ N 台 × RGB3チャンネル × 縦 × 横
    "egomotion":  Tensor[D_ego],            # 自車の速度・ヨーレートなど D_ego 次元
    "trajectory": Tensor[T, 2],             # 未来 T ステップの (x, y) 目標軌道
    ...
}
```

処理の流れをステップで分解します。

1. データソースを選ぶ。ソースAなら `AdapterA`、ソースBなら `AdapterB` を使う。どちらも同じ protocol を満たす。
2. adapter がそのソース固有のファイルを読む。たとえば映像なら、各カメラのストリームをデコードライブラリ(PyAV など)でフレームに分解する。
3. adapter が「共通サンプルの形」に詰め替えて1件ずつ吐き出す。ここでキー名・順序・テンソルの形・正規化のルールをすべて統一する。
4. 後段(学習用形式への shard packing や DataLoader)は、どの adapter から来たかを一切気にせず、常に同じ形のサンプルを受け取って処理する。

before / after で価値を示します。

```
before (adapter なし):
  ソースA 用前処理 → A 用 shard 化 → 学習
  ソースB 用前処理 → B 用 shard 化 → 学習     ← 後段が2系統に分裂

after (adapter あり):
  AdapterA ┐
           ├→ (共通サンプル) → 共通の shard 化 → ストレージ → 学習
  AdapterB ┘                  ← 後段は1系統に統一
```

データセットを追加したいときは、新しい adapter を1個書いて protocol を満たすだけでよく、shard 化や学習側のコードはそのまま使い回せます。これが抽象化層を1枚はさむことの効果です。設計の用語で言えば、変わりやすい部分(データソースごとの差)を一点に局所化し(関心の分離)、安定している部分(後段)を変更から守る、ということです。

なぜ「共通サンプルの形」を最初にきちんと決めるのが重要かというと、それが事実上のスキーマ(データの取り決め)になるからです。後段のモデルやロス関数はこの形を前提に書かれます。形を後から大きく変えると全 adapter と後段に波及するため、設計初期に「何を必須項目とし、何をオプションとするか」を慎重に決めます。

## 手法の系譜と主要論文
これはソフトウェア設計のトピックなので、学術論文というより設計原則と実装規約の系譜で理解します。

Adapter パターンは、いわゆる GoF(Gang of Four)の書籍「Design Patterns: Elements of Reusable Object-Oriented Software」(Gamma, Helm, Johnson, Vlissides, 1994)で定式化された古典的設計パターンの1つです。提案内容は「既存クラスのインタフェースを、クライアントが期待する別のインタフェースに変換する仲介役を置く」こと。動機は、変更したくない(あるいはできない)既存コードと、新しい呼び出し側を疎結合に保つこと。効果は再利用性と差し替えやすさ。トレードオフは「間接層が1枚増えてコードの追跡がやや煩雑になる」ことです。

データ供給の標準化という観点では、PyTorch の `Dataset` / `IterableDataset` 抽象が、機械学習における同じ思想の代表例です。学習ループ側は `Dataset` のインタフェース(`__getitem__` または `__iter__`)だけを知っていればよく、その裏にどんなデータソースがあるかを意識しません。HuggingFace の `datasets` ライブラリは、何百ものデータセットを共通 API で読めるようにしており、ロボット学習向けの LeRobot のデータセット規約も、多様なロボットのログを共通スキーマに揃える同じ発想を採っています。

歴史的に見ると、ETL(Extract-Transform-Load、抽出・変換・ロード)というデータ処理の古典的枠組みも本質的に同じです。多様なソースから抽出(Extract)し、共通形式へ変換(Transform)し、保管先へロード(Load)する。Ingest Adapter は、この Extract と Transform を「ソースごとに差し替え可能な部品」として実装したものと位置づけられます。

## 論文の実験結果(定量データ)
設計パターンは「何 % 改善した」という実験指標で測るものではありませんが、その効果は保守性・拡張性として定量的に語れます。

ソフトウェア工学の経験則として、機能追加のたびに既存コードを修正する必要がある設計(後段がソースの数だけ分裂する設計)では、コード量と修正箇所がデータソース数 N に対しておおむね線形〜それ以上に膨らみます。一方 adapter で後段を1系統に統一すると、ソース追加時の新規コードは「adapter 1個ぶん(定数)」に抑えられ、後段は不変です。これは「開放閉鎖原則(Open-Closed Principle)」— 拡張に対して開いており(adapter を足せる)、修正に対して閉じている(既存後段を触らない)— の具体的な効果です。

実務上の定量効果として、共通スキーマに揃えておくと、新データセットの取り込みから学習開始までのリードタイムが大幅に短縮されることが繰り返し観察されます。複数カメラ映像を扱うパイプラインでは、ソース固有のデコード処理(映像コーデックの違いなど)を adapter に閉じ込めることで、後段の学習・評価コードの再利用率を実質 100% に保てた、という運用報告が一般的です。逆に adapter 層を持たない場合、データセットを N 個サポートするだけで、前処理・shard 化・学習供給の3段が N 倍に増殖し、バグの温床になります。

## メリット・トレードオフ・限界
メリット
- データソースを追加しても、adapter を1個書くだけで後段(shard 化・学習)を変えずに済む。
- 後段は常に同じ形のサンプルを受け取るので、コードが1系統に統一され保守が楽。
- protocol(継承不要のダックタイピング)なので、各 adapter は互いに独立して実装・テストできる。
- ソース固有の難しさ(コーデックの違い、座標系の違い、欠損値の扱い)を adapter に局所化できる。

トレードオフ・限界
- 抽象層が1枚増えるぶん、「実際にどこで何を変換しているか」を追うのにファイルを行き来する必要がある。
- 共通サンプルの形を一度決めると、それが事実上の規約になる。後から大きく変えると全 adapter に波及する。
- ソースによっては「共通の形」に綺麗に収まらない固有メタ情報があり、取りこぼすか共通形に無理に押し込む判断が要る。
- 共通形を最大公約数にしすぎると、リッチなソースの情報量を捨ててしまう。逆に欲張りすぎると、貧弱なソースで埋められない必須項目が増える。この粒度設計が腕の見せ所。

研究上の課題としては、「異なるセンサー構成・座標系・キャリブレーションを持つデータを、どこまで共通スキーマに正規化すべきか」という問いがあります。正規化しすぎると元データの情報が失われ、しなさすぎると後段が複雑になる。マルチデータセット学習やドメイン横断の転移学習では、このスキーマ設計そのものが性能に効く重要因子になります。

## 発展トピック・研究の最前線
- マルチデータセット共同学習(co-training): 形式の違う複数データセットを共通スキーマに揃えて同時に学習し、ドメイン汎化を高める。adapter 設計がそのまま学習設計に直結する。
- データのスキーマ進化(schema evolution): 共通サンプルの形を後方互換に進化させる仕組み(Protocol Buffers / Avro のようなスキーマ管理)を取り込み、規約変更のコストを下げる。
- ストリーミング ETL とオンライン学習: 静的なファイル取り込みだけでなく、走行中の車両やシミュレータからリアルタイムにサンプルを生成し、同じ adapter インタフェースで流す設計。
- データ品質ゲート: adapter の出口に検証(欠損・外れ値・キャリブレーション異常の検出)を挟み、汚れたデータが後段へ流れるのを防ぐ。データ中心AI(data-centric AI)の流れと結びつく。
