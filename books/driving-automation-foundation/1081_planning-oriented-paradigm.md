# Planning-oriented設計思想

## ひとことで言うと

自動運転モデルの中にある「検出・追跡・予測」といった途中の仕事(中間タスク)を、それ単体の精度を上げるためではなく、最終目的である「安全な走行経路を作ること(プランニング、planning)」に役立つように選び・つなぎ・一緒に学習する、という設計の考え方です。UniAD という論文がこの思想を体系化し、CVPR 2023 の最優秀論文(best paper)になりました。

## 直感的な理解

ニューラルネットで自動運転を作るとき、長年の悩みは「中間タスクをどう扱うか」でした。中間タスクとは、最終的な運転判断にたどり着くまでの途中の作業、たとえば「周りの物体を検出する」「他車の将来を予測する」といったものです。

たとえ話で考えます。会社で「売上を上げる」という最終目標があるとき、各部署が「自分の部署の数字(残業時間の削減、書類の処理件数など)」だけを最適化しても、売上は上がるとは限りません。むしろ部署間の連携が壊れて逆効果になることすらあります。本当に必要なのは「各部署の活動を、最終目標である売上に資するかどうかで評価・調整する」ことです。planning-oriented(プランニング指向)は、これを自動運転モデルの中でやろうという思想です。「検出器がいくら精度を上げても、それが安全運転につながらなければ意味がない。すべての中間タスクを、最終目的(安全な計画)に従属させて統合しよう」というわけです。

## 基礎: 前提となる概念

前提として、自動運転モデルの設計には両極端があります([モジュラー vs E2E アーキテクチャ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1078_modular-vs-e2e) も参照)。

一つの極端は完全 E2E(エンドツーエンド)で、中間タスクを一切持たず、画像から操舵を直接出します。つなぎ目はありませんが、内部が何を見ているか分からず、危険な判断の原因を追えません(ブラックボックス)。

もう一つの極端はモジュラー型で、検出・追跡・予測を別々のネットで作り、個別に最適化します。すると各モジュールは「自分の指標」だけを追います。ここで指標の語彙を噛み砕きます。

- mAP(mean Average Precision、平均適合率): 物体検出のうまさを測る指標。検出した物体がどれだけ正確か・取りこぼしが少ないかを総合した点数。
- ADE / FDE(Average / Final Displacement Error、平均・最終変位誤差): 軌跡予測のうまさを測る指標。予測した将来位置と実際の位置のずれ(メートル)。ADE は全時刻の平均、FDE は最終時刻のずれ。
- 衝突率(collision rate): プランニングの安全性を測る代理指標。出した経路で走ると他物体にぶつかる割合(%)。

ここに落とし穴があります。「検出 mAP が上がる=運転が安全になる」とは限らないのです。たとえば進路に無関係な遠くの停車車両を1台多く検出できて mAP が上がっても、運転は 1mm も良くなりません。逆に、進路上の歩行者の速度をほんの少し読み違えるだけで衝突します。つまり各中間モジュールの最適化目標が、最終目的(安全運転)と方向がずれているのです。この「目標のずれ(objective misalignment)」を正面から解消するのが planning-oriented です。中間タスクを捨てる(完全 E2E)のでも独立に最適化する(モジュラー)のでもなく、最終目的に従属させて統合するのが核心です。

## 仕組みを詳しく

planning-oriented 設計の代表である UniAD を例に、何が「プランニング指向」なのかを4点に分解します。

1. 中間タスクを「捨てない」。検出・追跡・地図・動き予測・占有予測をすべて1つのネットに含めます。これらは解釈性(中身が見える)を保ち、かつ計画に効く文脈(誰がどこにいてどう動くか)を提供します。完全 E2E のようにブラックボックス化しません。

2. 中間タスクを「独立に最適化しない」。モジュラー型なら検出器を検出損失だけで鍛えますが、UniAD は全タスクの損失を足し合わせ、共有された特徴(BEV 特徴、Bird's-Eye-View=上空俯瞰の特徴地図 [H, W, C])を全部の勾配で同時に更新します。すると共有特徴は「プランニングにも検出にも予測にも効く形」へ育ちます。これは多タスク学習で各タスクの勾配が互いに正則化として働く効果でもあります。

3. 情報を「数値に潰さず」次へ渡す。モジュラー型は「ボックス座標とスコア」のような細い数値で受け渡しますが、UniAD はクエリ(query、知りたいことを表す高次元ベクトル。Transformer の attention で文脈を集める単位)のまま渡します。たとえば動き予測モジュールは、追跡モジュールの物体クエリと地図モジュールの地図クエリを直接受け取り、「この車は今この車線にいてこう動く」という文脈をリッチに保ったまま使います。これにより、検出段で取り落とした情報が後段で復元不能になる、というモジュラー型の弱点を緩和します。

4. 何を入れるかを「プランニング基準」で検証する。UniAD は ablation study(要素を1つずつ抜いて効果を確かめる実験)で、各中間タスクが本当にプランニング性能(衝突率の低下など)に寄与するかを確認しました。寄与しないなら入れない、という判断基準そのものが planning-oriented です。

数値の感覚をつかむため、処理の流れを図にします。

```
複数カメラ画像
   │  backbone + BEV エンコーダ
   ▼
 共有 BEV 特徴 [H,W,C] ───────────────┐
   │            │           │        │  (全タスクの勾配で同時更新)
   ▼            ▼           ▼        ▼
[追跡クエリ]→[地図クエリ]→[動き予測]→[占有予測]
                                  │  (クエリのまま文脈を渡す)
                                  ▼
                            [プランニング] ← 最終目的
                                  │
                                  ▼
                            自車の走行経路
```

## 手法の系譜と主要論文

- Hu et al., "ST-P3"(2022, ECCV, arXiv:2207.07601): UniAD に先行し、知覚・予測・計画を時空間 BEV 特徴で一気通貫に学ぶ。中間タスクを共有特徴で結ぶという planning-oriented の前段にあたる作品。

- Hu et al., "Planning-oriented Autonomous Driving"(UniAD, 2023, CVPR best paper, arXiv:2212.10156): planning-oriented という言葉を掲げ、検出・追跡・地図・動き予測・占有予測をクエリで連結し、全てを計画に奉仕させる統合設計を提案。動機は上記の「中間タスクの目標と最終目的のずれ」の解消。効果は nuScenes での各中間タスク指標の維持と衝突率の低下を同時に達成したこと。トレードオフはモデルが巨大で学習が重く、全中間タスクの教師ラベルが必要なこと。

- Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving"(2023, ICCV, arXiv:2303.12077): planning-oriented の思想を継ぎつつ、UniAD の重さを問題視。シーンを密な占有グリッドではなくベクトル(点・線の集合、たとえば車線を折れ線で、他車を境界点列で表す)で表現し、軽く速くした。効果は大幅な高速化と同等以上のプランニング性能。トレードオフはベクトル表現の表現力の限界(密な空間情報が落ちうる)。

- 対比として Zhai et al., "Rethinking the Open-Loop Evaluation of E2E Autonomous Driving"(AD-MLP, 2023, arXiv:2305.10430): planning-oriented なモデルの「良さ」をどう測るかという問題提起。自車の過去状態だけを入力にした単純な MLP でも、オープンループの L2 指標が既存の高度なモデルに匹敵してしまうことを示し、planning 指向の真価は適切な評価(クローズドループ)でこそ測れると示唆した([オープンループ vs クローズドループ評価](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0725_open-loop-vs-closed-loop-eval))。

## 論文の実験結果(定量データ)

UniAD は nuScenes(都市運転の公開大規模データセット)で評価され、planning-oriented 設計の有効性を定量的に示しました。中心となる主張は「各中間タスクを足し、共有特徴を一緒に鍛えると、最終目的であるプランニングが改善する」という ablation です。

プランニングの主指標は 1/2/3 秒先の L2 誤差(予測軌跡と人間軌跡のずれ、メートル)と衝突率(%)です。UniAD は当時の有力な E2E 手法(ST-P3 など)と比べ、平均衝突率を大きく下げました。報告では平均衝突率が ST-P3 の 1% 前後に対し UniAD は 0.3% 前後という水準で、相対的におおむね半分以下です(評価条件により数値は変動)。同時に平均 L2 誤差も改善しました。

planning-oriented の真骨頂は ablation study に表れます。UniAD の論文は、フルモデルから中間タスクを抜いていくと最終性能が下がることを示しました。具体的には、追跡(tracking)モジュールを外す、動き予測(motion forecasting)を外す、占有予測(occupancy)を外すと、いずれもプランニングの衝突率が悪化する方向に動くと報告されています。論文の ablation では、追跡や地図といった知覚系のタスクを段階的に足していくと動き予測の精度(minADE など)が改善し、さらに動き予測・占有予測を足すとプランニングの平均衝突率が単調に下がる、という積み上げが示されています。報告された平均衝突率はおおむね 0.3% 前後の水準で、これは過去状態だけに頼らず周囲の動的物体を考慮できているサインだと解釈されました。逆に言えば、各中間タスクが確かに最終目的に寄与していることを実験で確認したわけで、これが「中間タスクを最終目的の基準で選ぶ」という思想の実証です。一方、すべてのタスクが等しく効くわけではなく、計画に近い動き予測・占有予測の寄与が相対的に大きい、という傾向も読み取れます。設計上は「物体クエリをそのまま動き予測へ渡す(数値に潰さない)」ことが効いており、検出結果を座標へ離散化してから渡す素朴な接続より、クエリ連結のほうが下流タスクで安定して優れる、という比較も報告されています。

VAD は、この性能をほぼ保ちながら推論速度を大幅に上げたことを示しました。報告では UniAD 比でおおむね数倍の高速化を達成しつつ、プランニング指標は同等以上でした。密な BEV グリッドを使わずベクトル表現にしたことで計算量を削った成果で、planning-oriented の思想は維持したまま実用性を高めた例です。

ただし注意すべきは、これらの数値の多くがオープンループ評価(録画と比較するだけで、自分の判断が次の状況を変えない)であることです。AD-MLP の指摘どおり、L2 や衝突率のオープンループ値は過去の自車運動への過適合で良く見えうるため、planning-oriented 設計の本当の優劣はクローズドループ(NAVSIM や CARLA など、自分の判断が次状況を変える評価)で測り直す必要がある、というのが現在のコンセンサスです。

## メリット・トレードオフ・限界

メリット
- 中間タスクの目標と最終目的のずれを解消し、最終性能(衝突率など)が直接改善しやすい。
- 中間表現を残すので完全 E2E より解釈性・デバッグ性が高い。事故時に「どの中間タスクが何を見ていたか」を追える。
- 共有表現を多タスクの勾配で鍛えるため、各タスクが互いに正則化として働き、データを効率よく使える。

トレードオフ・限界
- 全中間タスクの教師ラベル(検出ボックス・追跡 ID・地図・予測軌跡など)が必要で、データ整備コストが非常に高い。
- モデルが大きく学習が重い(VAD のような軽量化が後続の主要課題になった)。
- マルチタスク損失の重み調整が難しく、ある一つのタスクが学習を支配すると他タスクが崩れる(損失バランス問題)。
- 「どの中間タスクが計画に効くか」の検証(ablation)に多大な手間がかかる。
- 評価がオープンループ指標に依存しがちで、真の運転性能を測りにくい。

研究上の未解決課題として、(1) ラベルコストを下げる自己教師あり・弱教師あり化、(2) 損失バランスの自動化、(3) クローズドループでの再評価と頑健性の確保、(4) どの中間タスクが因果的に計画に効くのかの理解、が残ります。

## 発展トピック・研究の最前線

planning-oriented の思想は、その後の E2E 運転研究の主流になりました。VAD の軽量化に続き、占有予測を高解像度・時系列に拡張する流れ、ベクトル表現と密表現を組み合わせるハイブリッド、そして大規模な事前学習や視覚言語モデル(VLM)を統合して「言語で運転判断を説明させる」方向が活発です。一方で、AD-MLP の指摘を受けて「中間タスクを全部入れる重厚な統合より、評価の妥当性とクローズドループ性能こそが本丸だ」とする揺り戻しもあります。Hydra-MDP(Li et al., 2024、NAVSIM チャレンジ優勝)はルールベース評価者を教師にした multi-target distillation でクローズドループ寄りの指標を直接最適化し、PARA-Drive(Weng et al., CVPR 2024)は UniAD の逐次的なタスク連結を見直して中間タスクを並列化し、推論時には不要なタスクヘッドを切り離して大幅な高速化を達成しました。これらは planning-oriented の精神(中間タスクは計画のための手段)を保ちつつ、「逐次接続でなければならない」という UniAD の構造的前提を緩めた点に新規性があります。また、明示的な中間タスク群を持たずとも、World Model(環境の遷移を学習した内部シミュレータ)を補助タスクとして共有特徴を鍛える、という「中間タスクを自己教師あり化する」アプローチも planning-oriented の延長線上にあります。共通する精神は一貫しており、「途中の仕事はすべて最終目的(安全な計画)のための手段である」という基準で設計全体を律することにあります。
