# マップマッチング

## ひとことで言うと
GPS で測った位置は数メートルずれることがあり、点をそのままつなぐと「道の上をきれいに走っている線」になりません。マップマッチング(map matching)とは、そのノイズ混じりの GPS 点列を、地図上の実際の道路にスナップ(吸着)させ、「実際にはこの道をこう走ったはず」というもっともらしい経路に直す処理です。カーナビ、配車サービスの軌跡解析、自動運転モデルへ渡すナビ地図の経路線整形など、位置データを使うほぼ全ての場面で前処理として登場します。

## 直感的な理解
GPS(全地球測位システム)で得られる位置は、ビルの反射(マルチパス)、衛星配置、受信環境、電離層の遅延などの影響で数メートルから十数メートル単位で揺らぎます。都市部のビル街(アーバンキャニオン)では誤差がさらに増えます。たとえば実際には1本の直線道路を走っていても、記録された GPS 点を順につなぐと、道路の脇にはみ出したり、ジグザグに振れたりします。

この GPS 点列をそのまま「走った経路」として使うと、次の問題が起きます。経路の線が道路から外れて建物の中や歩道の上を通っているように見える。ジグザグやワープ(瞬間移動したような飛び)が混じり、下流の解析やモデルが「経路の形」を扱いにくい。並走する道路(本線と側道、上り線と下り線)のどちらを走ったのか区別できない。

そこで、GPS 点を「実際にあり得る道路上の経路」に補正するのがマップマッチングです。「測定された軌跡」を「地図と矛盾しない軌跡」に翻訳する作業、と言えます。身近な例では、カーナビが交差点を曲がった瞬間に自車マークをカクッと道路上に乗せ直すのが、まさにマップマッチングの結果です。

## 基礎: 前提となる概念

道路網は「点と線のグラフ」で表せます(詳細は [Overpass API / OSMnx](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0424_overpass-api-osmnx))。ノードが交差点や曲がり角、エッジが道路区間です。マップマッチングは「GPS 点の列」を「このグラフ上のエッジ(またはノード)の列」に対応づける問題、と定式化できます。出力は「どのエッジをどの順に通ったか」というエッジ列(path)であり、単なる点ごとの最寄り道路ではなく、つながった一筆書きであることが本質です。

サンプリング間隔(sampling rate)という用語が重要です。これは GPS 点が何秒ごと・何メートルごとに記録されるかのことです。1 秒ごとの密なログなら隣の点はすぐ近くにあり、対応づけは比較的やさしい。一方、30 秒〜数分おきの疎なログ(タクシーの配車ログなどに多い)では、2 点の間に交差点が何個も挟まり、どの道を通ったか自明でなくなります。疎なログを正しく扱えるかが手法の難所です。

マップマッチング手法は大きく3系統に分けられます。幾何的手法(geometric)は単純に最も近い道路へ点を貼る。トポロジー的手法(topological)は道路のつながり(連結性)を考慮し、前の点から到達可能な道に限定して選ぶ。確率的手法(probabilistic)は「観測のもっともらしさ」と「移動のもっともらしさ」を確率で重みづけ、列全体として最尤の経路を選ぶ。後ろほど頑健ですが計算が重くなります。

隠れマルコフモデル(HMM = Hidden Markov Model)という枠組みも押さえておくと、現代の標準手法の理解が速くなります。HMM は「直接見えない真の状態(ここでは『今いる道路区間』)が時間とともに遷移し、各状態から観測(ここでは『GPS 点』)がノイズ込みで出る」という確率モデルです。観測列から最も尤もらしい状態列を逆算する問題は、Viterbi アルゴリズムという動的計画法で効率よく解けます。

## 仕組みを詳しく

最も実装が軽い「幾何+トポロジー」の近似マッチングは、次の2段で書けます。入力は道路網グラフと GPS 点の緯度経度列です。

ステップ1: 各 GPS 点を最近傍ノードへスナップ

```python
matched_nodes = ox.distance.nearest_nodes(graph, longitudes, latitudes)
```

各 GPS 点に対し、道路網グラフの中で最も近いノードを返します。GPS 点が3点 P0, P1, P2 あるとき、P0→ノード A、P1→ノード C、P2→ノード F のように対応します。これで GPS 点が道路上の点に吸着されますが、この段階では「A, C, F」という飛び飛びの点で、間がつながっていません。

ステップ2: 連続するスナップ点の間を最短経路で埋める

```python
route = []
for src, dst in zip(matched_nodes[:-1], matched_nodes[1:]):
    segment = nx.shortest_path(graph, src, dst, weight="length")
    route.extend(segment[1:])   # 端点の重複を避ける
```

A から C へ、C から F へ、それぞれ道路網上の最短経路(`weight="length"` = 距離が最短の道筋)を求め、間の中間ノード(例: A→B→C)を補います。これで「A, B, C, D, E, F」のように、道路上を切れ目なくたどる連続経路ができます。

ステップ3: 失敗時のフォールバック

マップマッチングは失敗しうることを前提に組むべきです。半径内にエッジが無い、グラフが分断されている(disconnected components)、GPS がグラフ範囲外、といった状況で破綻します。頑健な実装では、最近傍探索が例外を投げたら空の経路を返して「生 GPS だけ使え」と上位に伝え、2 点間に経路が存在しない場合はそのセグメントをスキップし、どの経路もつなげなければ最初の 1 点だけを返す、といったフォールバックを用意します。

ステップ4: 描画・下流への受け渡し

得られた連続経路は、各ノードの緯度経度を自車中心のローカル座標へ投影したうえで、ナビ地図画像に経路線として描く、あるいは下流の解析に渡します。元の生 GPS 点を別色で重ねておくと、補正前後を同じ画像で比較できます。

```
生 GPS(ノイズ)              マップマッチング後
  ・    ・                       ●─────●
    ・ ・  ・  (道から外れて       │     (道路に吸着し、間も
  ・     ・  ジグザグ)        ●──┘      連続した経路に整形)
```

この軽量実装は「最近傍ノード+最短経路で接続」という近似であり、後述する確率的(HMM ベース)なマップマッチングほど厳密ではありません。しかし依存が少なく高速で、サンプリングが密でナビ地図タイルを描く程度の用途には十分です。逆に、疎なログや並走道路が多い高速道路では誤りが目立つため、用途に応じて確率的手法へ切り替えます。

確率的手法(HMM)の中身を数式で示すと、各 GPS 点 $z_t$ に対し「実際に居た道路区間 $r_t$」を隠れ状態とし、次の積を最大化する状態列を求めます。

$$P(r_1,\dots,r_T \mid z_1,\dots,z_T) \propto \prod_t \underbrace{P(z_t \mid r_t)}_{\text{観測確率}} \cdot \prod_t \underbrace{P(r_t \mid r_{t-1})}_{\text{遷移確率}}$$

ここで観測確率(emission probability)は「GPS 点 $z_t$ と道路区間 $r_t$ の垂直距離 $d$」が小さいほど高くなる。典型的にはガウス分布 $\frac{1}{\sqrt{2\pi}\sigma}\exp(-d^2/2\sigma^2)$ を使い、$\sigma$ は GPS 誤差の標準偏差(数 m〜十数 m)。遷移確率(transition probability)は「区間 $r_{t-1}$ から $r_t$ への道路網上の最短移動距離 $d_{\text{route}}$」と「GPS 点間の直線距離 $d_{\text{gc}}$(great-circle distance)」の差 $|d_{\text{route}} - d_{\text{gc}}|$ が小さいほど高くなる(指数分布 $\frac{1}{\beta}\exp(-|d_{\text{route}}-d_{\text{gc}}|/\beta)$ でモデル化されることが多い)。直感的には「GPS 上は 30m しか進んでいないのに、道路をたどると 500m 回らないと届かない候補」は遷移確率が低くなり、捨てられる。この最尤状態列を Viterbi アルゴリズムで求めると、各時刻で生き残った候補だけを動的計画法で前向きにつなぐので、候補数を K、点数を T として O(T·K²) で解けます。

## 手法の系譜と主要論文

- Mohammed Quddus, Washington Ochieng, Robert Noland, "Current map-matching algorithms for transport applications: State-of-the-art and future research directions" (Transportation Research Part C, vol. 15(5), 2007)。幾何的・トポロジー的・確率的・先進的手法を体系的に整理した代表的サーベイです。前述の3系統の分類はこの論文に基づきます。単純な幾何マッチングが交差点付近や並走道路で誤吸着しやすいことを指摘し、より高度な手法の必要性を整理しました。

- Paul Newson & John Krumm, "Hidden Markov Map Matching Through Noise and Sparseness" (ACM SIGSPATIAL 2009)。Microsoft Research の研究で、上の HMM 定式化を確立し、以後の事実上の標準にしました。動機は、点を1つずつ最近傍に貼るだけだと「近いが実際には通っていない道」に誤吸着する問題を避けること。新規性は、観測確率と遷移確率の具体的なモデル化と、サンプリングが疎でも頑健に動く設計です。論文には実装の数値(典型的な $\sigma \approx 4.07$ m、$\beta \approx 5$ など)も示され、再現の指針になりました。

- Yin Lou et al., "Map-Matching for Low-Sampling-Rate GPS Trajectories" (ST-Matching, ACM SIGSPATIAL 2009)。Newson & Krumm と同年に、疎なログ(2〜5 分間隔)に焦点を当て、空間的近さ(spatial)に加えて道路の制限速度と実際に必要な速度の整合(temporal)も考慮する手法を提案。候補をグラフ(candidate graph)として結び、最良経路を探索することで、低サンプリング率での精度を引き上げました。

- 近年は、深層学習でマッチングを学ぶ手法も出ています。Kai Zhao et al. の DeepMM (ACM SIGSPATIAL 2019) は、軌跡を seq2seq(系列から系列への変換)的に道路 ID 列へ翻訳する学習ベースのマッチングを提案し、極端にノイズの多い・疎な軌跡で HMM を上回る精度を報告しています。実用ツールとしては Valhalla(Mapbox)、OSRM、fast-map-matching (FMM) が HMM ベースの高速実装を提供しています。

## 論文の実験結果(定量データ)

マップマッチングの評価指標は、主に「正しい道路区間をどれだけ当てたか」を測ります。代表的なのが route mismatch fraction(誤った道のりの割合。正解経路と推定経路で「片方にしか含まれない長さ」の合計を正解経路長で割った比率。0 に近いほど良い)と、区間単位の正解率(accuracy、推定区間が正解と一致した割合)です。route mismatch fraction はさらに $d_- / d_0 + d_+ / d_0$ のように、誤って外した長さ($d_-$)と誤って足した長さ($d_+$)を正解長 $d_0$ で割って分けて報告されることもあります。

Newson & Krumm (2009) は、シアトルで収集した正解付き軌跡(約 50 マイル / 約 80km、高密度の真値経路)を使い、人工的に GPS をダウンサンプリング(間引き)・ノイズ付加して頑健性を測りました。報告のポイントは、サンプリング間隔が 30 秒程度まで粗くなっても、また GPS 誤差を意図的に増やしても、route mismatch fraction が低く保たれることです。たとえばサンプリングが密(数秒)なら誤マッチはほぼゼロに近く、30 秒間隔・大きめのノイズでも実用に耐える低さに留まりました。これに対し単純な最近傍貼り付けは、間隔が粗くなると誤マッチが急増します。この差が「確率的整合性チェックを入れる価値」を定量的に示しています。

ST-Matching (Lou et al., 2009) は、低サンプリング率(2〜5 分間隔)の実タクシー軌跡で評価し、間隔が粗いほど従来手法に対する精度差が広がること、つまり疎なログほど時空間的なもっともらしさの考慮が効くことを示しました。学習ベースの DeepMM (Zhao et al., 2019) は、中国・成都などのタクシー軌跡で、ノイズや欠損が大きい条件で HMM ベースより accuracy を数ポイント改善したと報告しています。

アブレーション的に言えば、確率的手法から「遷移確率(道路上の移動の妥当性)」を抜いて観測確率だけにすると、ほぼ最近傍貼り付けに退化し、並走道路や交差点での誤りが増えます。逆に観測確率を抜くと、地図上を滑らかに動くが実際の GPS から離れた経路を選びやすくなります。両者の積で初めて「道に乗っていて、かつ滑らかに移動する」経路が選ばれる、というのがこの枠組みの核心です。実装上は遷移確率の $\beta$ を大きくしすぎると遠回りを許容しすぎ、小さすぎると正しい遠回りを弾くため、$\sigma$ と $\beta$ のチューニングが精度を左右します。

## メリット・トレードオフ・限界

メリット
- ノイズ混じり GPS を道路上のもっともらしい経路に補正でき、軌跡・経路線がきれいになる。
- 軽量実装(最近傍+最短経路)はシンプルで高速、依存もグラフライブラリだけ。サンプリングが密ならこれで十分。
- 確率的手法は疎なログ・大きなノイズでも頑健で、誤吸着を確率で抑え込める。

トレードオフ・限界
- 軽量な幾何マッチングは、並走道路や本線と側道のように近接する道で誤吸着しうる。確率的整合性チェックを行わないため。
- 道路網グラフの解像度(ノード密度)に経路の滑らかさが依存する。ノードが粗いとカクカクした経路になる。
- 最短経路で間を埋める実装は、実際にドライバーが遠回りしても「最短の道を通った」と補正してしまう可能性がある(これは確率的手法でも、遷移確率の設計次第で起こりうる)。
- 地図(基盤の道路網)が古い・欠けている場所では、どの手法でも正しくマッチングできない。地図品質が上限を決める。off-map(地図にない道を走った)の検出は本質的に難しい。

研究上の未解決課題としては、新設道路や工事で地図にない道を走ったときの扱い(off-map detection)、極端に疎・ノイジーなログでのオンライン(逐次)マッチング、複数の経路仮説を保持して後から確定する遅延決定型マッチングなどがあります。

## 発展トピック・研究の最前線
- オンライン(逐次)マップマッチング: 全軌跡が揃う前にリアルタイムで対応づける。Viterbi を逐次更新する手法や、複数仮説を保持するパーティクルフィルタ系。確定遅延(latency)と精度のトレードオフが研究テーマ。
- 学習ベースのマッチング(DeepMM など): seq2seq で軌跡を道路 ID 列へ翻訳。極端なノイズ・欠損に強いが、学習データと地図への依存・未知地域への汎化が課題。
- 自動運転での高精度版: GPS だけでなく IMU・車輪オドメトリ・カメラ/LiDAR を融合し、HD マップ上で車線レベル(どの車線か)まで対応づける lane-level map matching。これは [BEVマップセグメンテーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0269_bev-map-segmentation) のようなオンライン地図認識と組み合わさり、事前地図とのアラインメント(位置合わせ)に発展する。
