# BEVDepth

## ひとことで言うと
BEVDepth は、カメラ画像だけから「鳥が真上から見たような地図(BEV)」を作って3次元物体を検出する手法群において、最大の弱点が「深度(奥行き)推定が驚くほど不正確であること」だと正面から突き止め、それを治した研究です。具体的には、LiDAR(レーザー距離計)から作った正解の奥行きで深度推定を直接教え込み(明示的深度監督)、さらにカメラの設定情報(焦点距離や取り付け角度)を深度推定器に注入(カメラ認識深度)することで、それまで暗黙の学習に任せられていた深度を「信頼できる量」に変えました。深度の質を上げただけで検出精度が大きく底上げされることを実証した点が最大の貢献で、以後この発想は多視点カメラ3次元検出のほぼ標準装備になりました。

## 直感的な理解
人間が片目だけで遠くの物体までの距離を当てるのは難しいですが、ある程度はできます。経験から「このくらいの大きさに見える車はだいたい何メートル先」と推測しているからです。カメラ1枚から奥行きを当てるニューラルネットも同じことをしています。問題は、その推測の正確さを誰も直接チェックしていなかったことです。

カメラから3次元の物体を検出する初期の手法では、深度を「物体検出が当たればそれでよい」という間接的な目標だけで学習させていました。これは、生徒に「テストで良い点を取れ」とだけ言って、途中式が正しいかは一切見ない指導に似ています。生徒は答えだけ合わせる癖がつき、途中の計算(=深度)はデタラメでも構わなくなります。BEVDepth の著者たちは、この「途中式」にあたる深度をわざわざ取り出して正解(LiDAR)と突き合わせたところ、深度がひどく狂っていたことを発見しました。論文の言葉を借りれば、学習済みモデルの予測深度を LiDAR の真値で置き換えるだけで検出精度が劇的に跳ね上がった——これは「深度さえ正しければもっと当てられる、つまり深度がボトルネックだ」という決定的な証拠でした。そして「途中式も採点する」(深度を直接教える)ようにしただけで、最終的な答え(検出)が大きく良くなったのです。

## 基礎: 前提となる概念
このノートを読むために、いくつかの言葉を噛み砕いておきます。

BEV (Bird's-Eye-View) とは、鳥が真上から見下ろしたような俯瞰の視点です。自動運転では前後左右に取り付けた複数のカメラ画像を、最終的に「自車を中心にした真上からの地図」に統合します。この俯瞰地図の上で「ここに車がいる」「ここが車線だ」と判断するほうが、複数カメラの情報を一枚に重ね合わせやすく、距離関係も素直に扱えるからです。

深度 (depth) とは、画像の各ピクセルが「カメラからどのくらい遠いか」を表すメートル単位の量です。カメラ画像は3次元世界を平面に潰したものなので、各ピクセルに深度を割り当てれば、そのピクセルを3次元空間の点に戻せます。これがカメラから BEV を作るときの心臓部です。1枚のカメラ画像から深度を当てる問題は「単眼深度推定 (monocular depth estimation)」と呼ばれ、本質的に曖昧(同じ見た目でも大小の物体が遠近どちらにもあり得る)で難しいことが古くから知られています。

LSS (Lift-Splat-Shoot) という手法 (Philion & Fidler, ECCV 2020、別ノート [Lift-Splat-Shoot (LSS)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0282_lift-splat-shoot)) は、各ピクセルについて「深度の確率分布」を予測します。1点の深度を1つの値で決め打ちするのではなく、「2m先の確率20%、3m先30%、4m先50%」のように離散的な深度候補それぞれに確率を割り当てるのです。この「深度をカテゴリ(離散ビン)の確率分布として扱う」考え方は CaDDN (Reading et al., CVPR 2021、[CaDDN (Categorical Depth Distribution Network)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0275_caddn_note)、[カテゴリ深度分布 (depth bins)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0276_categorical-depth-distribution)) が体系化しました。各ピクセルの特徴ベクトルにこの確率を掛けながら、視線方向に伸びる多数の3次元点を生成します。これを Lift(持ち上げる)と呼び、その点群を BEV グリッドのマスに振り分けて合計するのが Splat です。式で書くと、ある視線上の深度ビン d の3次元点が持つ特徴は `f_d = α_d · c` です。ここで `c` はピクセルの文脈特徴ベクトル(C 次元)、`α_d` はそのピクセルの深度分布のうち深度 d の確率(全ビンで足すと 1)です。深度分布 `α` が外れていれば、特徴は間違った場所にばらまかれます。

カメラの内部パラメータ (intrinsics) とは、焦点距離や画像中心など、そのカメラのレンズ・センサー固有の値です。外部パラメータ (extrinsics) とは、そのカメラが車のどこに、どの向きで付いているかという位置・姿勢の値です。両者をまとめてキャリブレーション(校正)情報と呼びます。これらは「画像上のこのピクセルが、3次元のどの方向の光線に対応するか」を決めるのに不可欠で、深度を3次元座標に直すときにも使われます。

## 仕組みを詳しく
BEVDepth の改良は大きく4つです。順に、なぜそれが必要かと一緒に見ていきます。

### 1. 明示的な深度監督 (Explicit Depth Supervision)
学習時に LiDAR の点群を各カメラ画像に投影し、「このピクセルの正解深度は何メートル」という教師ラベルを作ります。LiDAR は地面や物体に当たって跳ね返った点の3次元座標を直接測れるので、深度の正解として信頼できます。

予測した深度分布が、この正解深度に一致するように損失関数で直接罰を与えます。深度を離散ビンの分類問題として扱い、正解深度が属するビンを当てるクロスエントロピー損失(あるいは binary cross-entropy)を使うのが一般的です。深度ビンの取り方には2通りあり、等間隔の UD (uniform discretization) と、遠方ほどビン幅を広げる LID (linear-increasing discretization) があります。遠方は1ピクセルの誤差が大きな距離誤差につながるため、CaDDN 由来の LID/SID 系のビン割りが好まれます。全体の損失は次の形になります。

```
全体の損失 L = L_detection + λ * L_depth
  L_detection : 3D box の位置・サイズ・向き・速度・クラスを当てる損失
  L_depth     : 予測した深度分布が LiDAR 正解深度のビンを当てる分類損失(BCE/CE)
  λ           : 深度損失の重み(ハイパーパラメータ。3前後がよく使われる)
```

これにより、間接学習でぼやけていた深度がシャープで正確になります。重要なのは、LiDAR が必要なのは学習のときだけで、推論時はカメラだけで動く点です。LiDAR は「先生」として学習中に答え合わせをするために使い、本番ではカメラ単独で走れます。これは知識蒸留の一種(モダリティ間の蒸留)とも見なせます。

### 2. カメラ認識の深度推定 (Camera-aware Depth Estimation)
複数のカメラはそれぞれ焦点距離や取り付け角度が違います。ところが深度推定器がそのカメラ固有の情報を知らないと、同じ「見かけの大きさ」でもカメラによって正しい深度が違う、という事実に対応できません。

具体例で説明します。広角レンズ(短い焦点距離)のカメラと望遠寄り(長い焦点距離)のカメラで同じ車を撮ると、画像上での見かけのサイズが変わります。物体の実サイズ S、焦点距離 f、深度 Z の間には `画像上のサイズ ≈ f · S / Z` という関係があるため、同じ見かけサイズでも f が違えば Z は変わります。カメラ情報なしのネットワークは「見かけサイズ」だけから深度を当てようとして混乱します。そこで BEVDepth は、内部・外部パラメータをまとめたベクトルを小さな MLP(多層パーセプトロン、数層の全結合ネット)に通して特徴変換用のスケール係数を生成し、画像特徴を SE (Squeeze-and-Excitation) 風にチャンネル方向で再重み付けしてから深度を予測します。「カメラパラメータで条件づけたチャンネルアテンション」と言えます。これで「このカメラでこの見かけなら深度はこれくらい」と正しく対応づけられます。

### 3. 深度精緻化 (Depth Refinement)
予測した深度をさらに洗練するための畳み込みモジュール(深度方向の3次元的な集約)を加え、深度の空間的な滑らかさや整合性を改善します。隣り合うピクセルの深度が極端に飛ばないようにし、視線方向(深度方向)の集約も行って、ノイズの多い生の深度予測を整えます。

### 全体の流れ

```
複数カメラ画像 + カメラパラメータ(intrinsics/extrinsics)
   │
   ▼ 画像エンコーダ(ResNet-50 / V2-99 など)
画像特徴 [N, C, H, W]   N=カメラ台数, C=チャンネル, H/W=特徴マップの縦横
   │
   ▼ カメラ認識 深度ネット(DepthNet)─── 学習時のみ LiDAR 正解深度で明示監督
深度分布 [N, D, H, W]   D=離散深度ビン数(例 59〜112 ビン)。シャープで正確
   │  + 深度精緻化(畳み込み)
   ▼ LSS 視点変換(Lift で点群化 → Splat で BEV に集約)
BEV 特徴 [B, C, X, Y]
   │
   ▼ BEV エンコーダ → CenterPoint 系検出ヘッド
3D 物体 box(位置・サイズ・向き・速度・クラス)
```

DepthNet と文脈特徴(context)は同じ画像特徴から分岐します。`[N,C,H,W]` の画像特徴を2つに分け、片方を深度分布 `[N,D,H,W]` に、もう片方を文脈特徴 `[N,C',H,W]` にして、外積(`α_d · c`)で `[N,D,C',H,W]` の Lift 後特徴を作る、という流れです。

### 計算効率の工夫 (Voxel Pooling の高速化)
LSS の Splat(膨大な3次元点を BEV グリッドに集める処理)は素直に書くと遅く、学習のボトルネックになります。BEVDepth はこの集約のための効率的な CUDA 実装(Voxel Pooling)を提供し、大きな解像度でも現実的な時間で学習できるようにしました。この方向の最適化は後に [BEVPool / BEVPoolv2 (高速voxel pooling)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0620_bevpoolv2) でさらに洗練されます。

### 4. 多フレーム融合 (任意)
BEVDepth は単フレーム版に加え、BEVDet4D ([BEVDet4D](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0272_bevdet4d)) 流に直前フレームの BEV 特徴を自車運動でアラインして結合する多フレーム版も提供します。これにより速度推定(mAVE)が改善し、NDS が上がります。深度の質改善(本論文の主眼)と時系列融合は直交する改善で、両方入れると最も強くなります。

## 手法の系譜と主要論文
深度ベースのカメラ BEV 変換は、ひとつの明確な系譜を描きます。

最初の土台が LSS (Philion & Fidler, ECCV 2020, arXiv:2008.05711) です。各ピクセルの深度を確率分布として予測し、特徴を3次元に持ち上げて BEV に集約するという枠組みを作りました。ただし深度は検出損失からの間接学習で、精度は二の次でした。

次に CaDDN (Reading, Harakeh, Chae, Waslander; CVPR 2021, arXiv:2103.01100) が、単眼3次元検出の文脈で「深度を LiDAR で直接教える(categorical depth distribution の明示的監督)」という重要な発想を導入しました。深度をカテゴリ分布として丁寧に学習させると検出が良くなることを示し、後続の理論的下地になりました。同時期に DD3D (Park et al., ICCV 2021) も大規模深度事前学習が単眼3次元検出を大きく助けることを示し、「深度こそ鍵」という機運を作りました。

BEVDet (Huang, Huang, Zhu, Du; 2021, arXiv:2112.11790、[BEVDet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0271_bevdet_note)) は LSS の枠組みを多カメラの大規模3次元検出に実用化したパイプライン(画像エンコーダ → LSS → BEV エンコーダ → CenterPoint ヘッド)で、BEVDepth はこの BEVDet 系の構成を踏襲しています。BEVDet4D (Huang & Huang, 2022, arXiv:2203.17054) が時系列融合で速度推定を可能にしました。

そして BEVDepth (Li, Ge, Yu, Yang, Wang, Shi, Sun, Li; AAAI 2023, arXiv:2206.10092「BEVDepth: Acquisition of Reliable Depth for Multi-view 3D Object Detection」) が、CaDDN の明示的深度監督を多カメラ大規模設定に体系的に持ち込み、さらにカメラ認識の深度推定・深度精緻化・高速 Voxel Pooling を組み合わせました。最大の新規性は、提案手法そのものよりも「LSS 系の深度が実は壊れている」という診断を実験で示し、深度の質が検出精度の主要なボトルネックであると突き止めた点にあります。

この系譜のもう一方の枝が [BEVStereo](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0274_bevstereo) (同じ AAAI 2023) で、LiDAR 監督に頼らず時系列ステレオ(幾何)で深度を上げるアプローチを取りました。BEVDepth と BEVStereo は「深度の質を上げる」という目的を共有しつつ、手段が対照的です。後継の SOLOFusion / VideoBEV などは「長期時系列 + 高品質深度」を組み合わせ、明示的深度監督が標準前提となった上での次の改善を追求しています。

## 論文の実験結果(定量データ)
評価は主に nuScenes ベンチマークで行われました。nuScenes は6台のカメラ・1台の32ビーム LiDAR・5台のレーダーを積んだ車で集めた大規模自動運転データセット(約1000シーン、各20秒、計約140万枚の注釈付き画像)で、カメラのみ3次元検出の標準的な評価舞台です。

指標の意味を先に説明します。

mAP (mean Average Precision) は検出の正確さの指標で、「物体を正しい位置・クラスで見つけられたか」を平均したものです。nuScenes では3次元中心位置の BEV 上での距離しきい値(0.5/1/2/4m の4段階)で TP 判定し、10クラス平均します。IoU ではなく中心距離で判定するのが nuScenes の特徴です。高いほど良い(0〜1)。

NDS (nuScenes Detection Score) は nuScenes 独自の総合指標で、mAP に加えて、位置誤差(mATE)、サイズ誤差(mASE)、向き誤差(mAOE)、速度誤差(mAVE)、属性誤差(mAAE)を統合したものです。`NDS = (5·mAP + Σ(1 - min(1, errorᵢ))) / 10` という重み付き和で、検出が「当たったか」だけでなく「どれだけ精密に当たったか」まで含むため、実用的な総合点として重視されます。高いほど良い。

論文の報告によれば、BEVDepth は当時のカメラのみ3次元検出で最高水準を達成しました。代表的な数値を挙げます。

- ResNet-50 バックボーン・256x704 入力・単フレームの中規模設定で、nuScenes val で mAP およそ 0.33、NDS およそ 0.42 前後。多フレーム(4フレーム)を加えると NDS がおよそ 0.475 前後まで上がります(速度誤差 mAVE が大きく下がるため)。
- 大きなバックボーン(ConvNeXt や V2-99)・高解像度・長い時系列・test-time augmentation を組み合わせた設定では、nuScenes test split で NDS が約 0.60、mAP が約 0.52 に到達したと報告されています。これは当時のカメラ単独 leaderboard でトップクラスでした。

これは同程度の計算条件のベースライン(明示的深度監督なしの BEVDet 系)に対して数ポイントの NDS 改善にあたり、カメラ単独手法としては当時非常に強い結果でした。

最も示唆的なのはアブレーション(構成要素を抜き差しして寄与を測る実験)です。

- 動機の核となった診断実験: 学習済みモデルの予測深度を「LiDAR から作った正解深度」に差し替えて検出ヘッドに通すと、mAP が劇的に(報告では十数ポイント規模、おおむね 0.28 付近から 0.45 超へ)跳ね上がりました。これは「深度さえ正しければもっと当てられる」ことを直接示す、本研究の出発点です。
- 明示的深度監督を加えると、加えない場合に比べて NDS が数ポイント(報告でおよそ +1.5〜+3 NDS)改善します。これが BEVDepth の中心的な貢献です。
- カメラ認識の深度推定(カメラパラメータの注入)を外すと精度が落ち、複数カメラの違いに対応することの効果が確認されます(報告でおよそ +1 NDS 程度の寄与)。
- 深度精緻化モジュールも積み増しの改善をもたらします。
- 効率面では、提案 Voxel Pooling により view transformation が大幅に高速化し、学習スループットが上がったと報告されています。

これらを総合すると、「深度の質 → 検出精度」という因果が定量的に裏づけられ、深度に投資する価値があると示されました。

## メリット・トレードオフ・限界
メリット

- 深度推定が大幅に正確になり、BEV 3次元検出の精度(mAP / NDS)が底上げされる。診断実験で因果まで示した説得力。
- カメラパラメータの注入により、焦点距離や取り付けが異なる複数カメラに正しく対応できる。
- 効率的な Voxel Pooling で学習が高速化し、大きな解像度でも現実的に学習できる。
- 推論時はカメラのみで動く(LiDAR は学習時の教師としてのみ必要)。多フレーム版で速度推定も強い。

トレードオフ・限界

- 学習に LiDAR 由来の深度ラベルが必要で、データ収集・校正のコストが上がる。LiDAR のない環境では明示的深度監督を直接適用できない。
- カメラ認識・深度精緻化でモジュール構成がやや複雑になる。
- 単フレームの深度監督が中心で、時系列の幾何的手がかり(動きから奥行きを測る)は活用しない。それは [BEVStereo](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0274_bevstereo) の領域。
- LiDAR 点群は画像に比べて疎(まばら)なので、画像の全ピクセルに正解深度が付くわけではない(投影後の有効ピクセルは数%程度のことも)。教師信号が疎であることが残る課題。
- 深度を離散ビンの分類として解くため、ビン境界での量子化誤差が残る。遠方ほど影響が大きい。
- 遠方や夜間・悪天候など、もともと深度推定が苦手な領域での信頼性は依然として未解決。

研究上の未解決課題としては、LiDAR なしで同等の深度信頼性をどう得るか(自己教師あり深度や時系列ステレオとの統合)、疎な教師信号をどう密に活かすか(深度補完との併用)、深度の不確かさ(uncertainty)を下流の検出にどう伝えるか、回帰と分類のハイブリッド深度表現、といった方向が残っています。

## 発展トピック・研究の最前線
BEVDepth 以降、深度監督は多視点カメラ3次元検出のほぼ標準装備になりました。発展の主な流れは次の通りです。

時系列ステレオとの統合。BEVStereo は時系列ステレオで深度を補強しますが、これと LiDAR 監督を組み合わせて両方の長所を取る研究が進みました。長期時系列を効率よく積む SOLOFusion (Park et al., 2022) は、短期の高解像度深度と長期の低解像度履歴を併用し、明示的深度監督を前提に NDS をさらに押し上げました。

深度の不確かさの活用。単に深度を当てるだけでなく、「この深度はどれくらい自信があるか」をモデルに出させ、自信の低い領域では集約の重みを下げる、といった工夫で頑健性を高める方向。

クエリベース手法との対比。BEVDepth は「深度を当てて特徴を持ち上げる」明示的な前向き投影(forward projection)ですが、これと対照的に、3次元の問い合わせ点から画像にサンプリングしに行く後ろ向き(backward / query-based)な手法群(DETR3D、PETR、BEVFormer の空間クロスアテンション)も並行して発展しました。両者の比較と統合は今も活発な研究領域です。

蒸留による LiDAR 依存の緩和。LiDAR を持つ強い teacher から camera-only の student へ BEV 特徴を蒸留する研究(BEVDistill など)で、推論はカメラのみのまま精度を上げる方向。

長距離・高解像度化。検出範囲を遠方まで広げ、BEV 解像度を上げるとメモリと計算が急増するため、深度の質を保ちながら効率化する研究が続いています。占有予測(occupancy)への発展でも、信頼できる深度は土台になっています。
