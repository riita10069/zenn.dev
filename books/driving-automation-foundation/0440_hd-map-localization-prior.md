# HD地図ローカリゼーション事前情報

## ひとことで言うと
あらかじめ用意した高精度地図(HD地図)に対して、自車が「今この地図上のどこに、どの向きで居るか」を数十センチ以下の精度でぴったり合わせる(localize する)技術です。これが合って初めて、地図に描かれた車線や信号の情報を、自車の周りの正しい場所に引き当てて使えます。地図をモデルに渡す処理の、さらに上流にあたる前提技術です。

## 直感的な理解
精巧な街の模型(これが HD地図)を持っていても、その模型のどこに自分が立っているかが分からなければ役に立ちません。「自分は模型のこの交差点の北東角に、北向きで立っている」と正確に分かって初めて、「模型によれば、自分の右手3メートルに信号がある」と現実世界に対応づけられます。ローカリゼーションとは、現実の自分を模型(地図)の上の正しい一点に重ねる作業です。

自動運転でこれが致命的に重要なのは、誤差がそのまま命に関わるからです。自車位置が地図上で1メートル横にずれていれば、隣の車線を自分の車線だと誤認します。車線幅は3〜3.5メートルしかないので、数メートルの誤差では車線を取り違え、対向車線にはみ出すような判断をしかねません。だから「だいたいここ」では足りず、「数十センチ以内」が要求されます。

## 基礎: 前提となる概念

- HD地図(High-Definition map、高精細地図): これは「車線1本ずつの形・停止線・信号の位置・縁石などを、数センチ〜数十センチの精度で記録した地図」という意味です。カーナビの地図(道路レベル)よりはるかに精密です。
- ローカリゼーション(localization、自己位置推定): 自車が世界のどこに、どの向きで居るかを推定すること。位置 (x, y) と向き(ヨー角)の最低3自由度、立体的には6自由度(位置3+姿勢3)を求めます。
- GNSS / GPS: 衛星測位。単独だと誤差が数メートルあり、ビルの谷間(マルチパス=電波の反射で測位が乱れる現象)やトンネルではさらに悪化します。RTK(Real-Time Kinematic、基準局との差分で誤差を補正する方式)を使えばセンチ精度も出ますが、基準局・通信・開けた空が要ります。
- 推測航法(dead reckoning): 直前の位置から、車輪の回転量(オドメトリ)や IMU(Inertial Measurement Unit、慣性計測装置=加速度と角速度を測るセンサ)で移動量を積算し、現在位置を推定する方法。短時間は正確ですが、誤差が時間とともに積み上がります(これをドリフトと呼びます)。
- 事前情報(prior): これは統計・推定の用語で「観測する前から分かっている前提知識」という意味です。HD地図は走行前に作ってあるので、自車にとって強力な事前情報です。ベイズ推定の枠組みでは、地図が事前分布、センサ観測が尤度(likelihood、ある位置だと仮定したとき今の観測がどれだけ起こりやすいか)にあたり、両者を掛け合わせて事後分布(最も確からしい位置)を得る、という形で定式化できます。

## 仕組みを詳しく

地図ベースのローカリゼーション(map-based localization)は、おおむね次のステップで動きます。

1. 大まかな初期位置を得る。GPS や推測航法で、誤差数メートル程度の当たりをつける。ここで全くの当てずっぽうから始めると照合が局所解にはまって収束しにくいので、初期値は重要です。

2. 車載センサで周囲を観測する。LiDAR(レーザーで周囲の3次元形状を点群=点の集まりとして測る装置)なら点群が、カメラなら画像が得られます。

3. 観測を地図と照合する(マッチング)。LiDAR の点群を、HD地図にあらかじめ記録された点群や特徴(縁石、標識、建物の壁、地面の反射強度パターンなど)と重ね合わせ、「どれだけ平行移動・回転させると最もよく一致するか」を探します。代表的アルゴリズムは次の2つです。

   - ICP (Iterative Closest Point): 観測点群の各点について地図側の最近傍点を対応づけ、その対応の距離の二乗和を最小にする剛体変換(回転+平行移動)を求め、これを収束するまで繰り返します。直感的には「2つの点群がいちばん重なる位置にじわじわ寄せていく」処理です。最近傍の対応づけが間違うと局所解にはまるため、初期値の良さに敏感です。
   - NDT (Normal Distributions Transform): 地図を小さなグリッドに区切り、各セル内の点群を「平均と共分散を持つガウス分布」で表します。観測点がそのガウス分布の山のどこに落ちるかでスコアを計算し、スコアが最大になる位置を勾配法で探します。点同士の一対一対応づけが不要なぶん、ICP より外れ値や粗い点群に強い傾向があります。

   カメラの場合は、地図上の車線・標識・ランドマークの見え方を画像平面へ投影し、画像から検出した同じ要素と照合します。

4. 最も一致する位置・姿勢を高精度位置とする。さらに時間方向で滑らかにつなぐため、カルマンフィルタ(線形・ガウス前提で予測と観測を最適に統合する逐次推定。非線形には拡張カルマンフィルタ EKF を使う)や、パーティクルフィルタ(多数の位置候補=粒子を撒いて、観測との一致度で重み付け・再サンプリングしながら追跡する手法)で、GPS・推測航法・地図照合を融合します。これをセンサフュージョンと呼びます。

結果として、横方向・縦方向ともに数十センチ以下、向き(ヨー角)も1度以下といった精度で自車位置が決まります。この精度があって初めて、HD地図の車線・信号情報を自車周りの正しい座標へ重ねられます。

### 地図をモデル入力にする処理との関係

ローカリゼーションが済むと、HD地図から「自車周辺の地図要素」を切り出し、自車中心(ego-centric、自車を原点・進行方向を上にした座標系)のラスタ画像やベクタにして、運転モデルへ渡せます(地図の符号化方法は [地図チャンネル符号化方式](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0419_map-channel-encoding-scheme))。逆に言えば、ローカリゼーションがずれていれば、どんなに優れた地図符号化をしても、モデルには「ずれた地図」が入ります。つまり地図エンコードの品質は、上流のローカリゼーション精度に決定的に依存します。誤差が下流へ伝播するこの構造が、モジュール分割型スタックの本質的な弱点でもあります。

## 手法の系譜と主要論文

地図ベースのローカリゼーションは特定の1論文というより一分野です。系譜をたどります。

ICP (Besl and McKay, IEEE TPAMI 1992, "A Method for Registration of 3-D Shapes") が点群位置合わせの古典で、ロボティクス全般の基礎部品になりました。

NDT (Biber and Straßer, IROS 2003, "The Normal Distributions Transform: A New Approach to Laser Scan Matching") が、点群を確率分布で表す照合法を提案し、現在も自動運転の自己位置推定で広く使われています。計算が軽く、外れ値に頑健なため、オープンな自動運転スタックの定番ローカライザになっています。

Levinson and Thrun (Stanford, ICRA 2010, "Robust Vehicle Localization in Urban Environments Using Probabilistic Maps") は、地面の反射強度(路面の白線やアスファルトの違いがレーザー反射に出る)をあらかじめ確率地図(各セルが反射強度の平均と分散を持つ)として作っておき、走行中の LiDAR 反射強度と確率的に照合して数十センチ精度の位置を得ました。動機は GPS 単独では都市部で精度が足りないこと、新規性は反射強度を確率分布として地図化し、雨や経年変化に頑健にした点です。これが DARPA Urban Challenge 以降の自動運転ローカリゼーションの礎になりました。

Wolcott and Eustice (IROS 2014, "Visual Localization within LIDAR Maps for Automated Urban Driving") は、LiDAR で作った3次元の反射強度地図に対して、安価なカメラ画像だけで位置を合わせる方向を示し、「高価な LiDAR で地図を作り、安いカメラで走る」という実用的な分業の先駆けになりました。

近年は学習ベースのローカリゼーションが登場しています。BEV-Locator (Zhang et al., 2022, "BEV-Locator: An End-to-end Visual Semantic Localization Network Using Multi-View Images") は、複数カメラから作った BEV 特徴と、ベクタ地図(車線・境界)を Transformer のクロスアテンションで突き合わせ、両者のずれ(位置・姿勢のオフセット)を直接回帰します。これは「地図エンコードとローカリゼーションを同じニューラルネットで一気にやる」方向で、古典的な幾何ベースのローカリゼーションと、地図をモデル入力にする話([地図チャンネル符号化方式](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0419_map-channel-encoding-scheme))の橋渡しにあたります。同系統に DA4AD や、HD地図のベクタ要素を画像と対応づける各種手法があります。

## 論文の実験結果(定量データ)

ローカリゼーションの評価指標は主に、横方向・縦方向の位置誤差(lateral / longitudinal error、単位はメートルやセンチ)と、ヨー角誤差(度)です。位置誤差は「推定位置と真値(高精度 RTK-GNSS や手動アノテーションで得た正解)との距離」で、小さいほど良い指標です。なぜ横と縦を分けるかというと、自動運転では横方向(車線内のどこに居るか)の誤差が車線逸脱に直結し、縦方向(進行方向の前後)より厳しく問われるからです。横誤差は車線中心からのずれ、縦誤差は停止線や交差点までの距離の正確さに効きます。

報告されている数値感として、Levinson らの反射強度マッチングは都市環境で数十センチ(報告では概ね 10cm 前後の RMS 横方向誤差)を達成し、GPS 単独の数メートルから一桁以上改善しました。LiDAR + NDT/ICP ベースの実用システムでも、横方向で 10〜30cm、ヨー角で 0.1〜1 度程度が一般的に報告される水準です。Wolcott らのカメラ照合は LiDAR ほどではないものの、地図に対しサブメートル級の精度を低コストで実現できることを示しました。

学習ベースの BEV-Locator は、nuScenes で横方向誤差およそ 0.1m 台、縦方向 0.2〜0.3m 台、ヨー角 1 度未満を報告し、カメラのみ(LiDAR なし)でも実用的な精度に到達できることを示しました。これは「高価な LiDAR を使わずカメラ+地図で位置を出す」道を開いた点で重要です。アブレーションでは、入力から地図のベクタ情報を外すと誤差が大きく悪化することが示され、地図(事前情報)の寄与が定量的に確認されています。また、利用するカメラ台数を減らすと精度が落ちることも報告され、周囲全方位の情報が照合に効くことが裏づけられています。

## メリット・トレードオフ・限界

- メリット: GPS 単独より桁違いに高精度(数十センチ・1度以下)。車線レベルの判断が可能になる。
- メリット: HD地図という強い事前情報により、センサが一部隠れても(遮蔽)位置を保てる。トンネルや GPS 不感地帯でも、地図照合と推測航法で位置を維持できる。
- トレードオフ: HD地図を事前に作り、維持・更新し続けるコストが大きい。地図のない場所では使えない。
- トレードオフ: 地図と現実が食い違うとき(工事・新設道路・季節変化・降雪での見え方変化)に精度が落ち、誤った前提を与えるリスクがある。これは事前情報を使う手法の宿命的な弱点です。
- トレードオフ: 高精度照合には LiDAR 等のセンサや計算資源が要り、システムが重くなる。
- 限界・未解決課題: 動的物体(他車・歩行者)が観測を汚すと照合が乱れる(これを除去するために動的点群フィルタリングが必要)。長期間の環境変化に地図をどう自動更新するか(life-long mapping / change detection)は研究途上です。学習ベースは未知環境への汎化(訓練と違う街での精度低下)が課題です。

## 発展トピック・研究の最前線

近年は HD地図への依存自体を減らす方向(SD地図やオンライン地図構築)が強まっており([オンライン vs オフライン地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0456_online-vs-offline-mapping)、[マップフリー運転](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0429_map-free-driving))、その文脈で「精密ローカリゼーションをどこまで必要とするか」が問われています。一方で、ローカリゼーションと地図構築を同時に行う SLAM(Simultaneous Localization and Mapping)、複数走行のデータを束ねて地図を自動更新するクラウドソース地図、ニューラル表現(NeRF や暗黙的地図)に対するローカリゼーション、HD地図ではなく粗い SD地図(OpenStreetMap)に対して大まかに位置を合わせる軽量ローカリゼーションなど、地図とローカリゼーションの境界を溶かす研究が進んでいます。「地図をどこまで陽に持ち、どこまで精密に合わせるか」は、システム全体のコストと頑健性を左右する設計判断です。
