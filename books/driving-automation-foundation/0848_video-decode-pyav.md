# 動画デコード前処理 (PyAV)

## ひとことで言うと
カメラの走行映像は、容量を小さくするために圧縮された動画ファイル(MP4 など)で保存されています。学習で使うには、その圧縮動画から必要な瞬間の1枚1枚の静止画(フレーム)を取り出さねばなりません。この取り出し作業をデコード (decode) と呼び、PyAV(動画処理エンジン FFmpeg の Python ラッパー)が広く使われます。さらに実務では、学習のたびに動画をデコードすると遅いので、事前にフレームを抜き出して画像シャードに焼き直す前処理を行うのが定石です。

## 直感的な理解
深層学習の学習は、データを何周も(エポックを重ねて)繰り返し読みます。1エポックに数十万〜数百万サンプル、それを数十エポック。つまり同じ画像を何十回も読み込みます。

ここで「圧縮動画のまま、毎回その場でデコードする」と何が起きるか。GPU(高価で速い計算装置)が、CPUが画像をデコードし終えるのをじっと待つ状態になります。GPU 使用率が 100% に張り付かず、50% や 30% でうろうろする。これは「速い料理人(GPU)が、下ごしらえ係(CPU のデコード)が間に合わずに手を止めている」状態で、最も避けたい無駄です。最新の GPU は1枚数百万円する一方、それを遊ばせるのは丸ごと損失なので、データ供給がボトルネックにならない設計は学習コストに直結します。

だから前処理として「重いデコードは学習の前に1回だけ済ませ、学習中は展開済みの軽い形式を読むだけ」にします。これがデコード前処理(フレーム抽出+シャード化)の本質です。1回の重さを払って、その後の何十周を軽くする取引です。

## 基礎: 前提となる概念

### そもそもなぜ動画は「そのまま」使えないのか
カメラは1秒間に10〜30枚のペースで画像を撮ります。これを全部 PNG や JPEG の静止画で保存すると、すぐに膨大な容量になります。例えば7台のカメラで1時間走ると、画像は数十万〜数百万枚。だから現場では、連続フレームをまとめて H.264 / H.265 などの方式で圧縮し、1本の MP4 動画にして保存します。圧縮率が高く、容量を1桁以上小さくできます。

しかしニューラルネットワークが学習で食べるのは、圧縮された動画のバイト列ではなく、生の画像、つまり「高さ H × 幅 W × 3色(RGB)」の数値の格子(テンソル)です。圧縮動画からこの生画像を復元する処理がデコードです。

### 動画圧縮の「飛べない」問題(GOP 構造)
ここが初心者のつまずきどころです。動画は全フレームを独立に保存していません。

- I フレーム (Intra-coded、キーフレーム): それ単独で完全な1枚の画像。JPEG 1枚と似た自己完結した符号化。
- P フレーム (Predicted): 直前のフレームとの「差分」だけを保存。単独では絵にならず、前のフレームに差分(動きベクトル+残差)を足して初めて絵になる。
- B フレーム (Bi-directional): 前後両方のフレームを参照して差分を保存。だから「未来のフレーム」を先にデコードしておかないと作れず、デコード順と表示順がずれる。

この I/P/B の塊を GOP (Group of Pictures) と呼びます。典型的には数十フレームに1枚だけ I フレームがあり、残りは差分です。つまり「100フレーム目だけ欲しい」と思っても、直前の I フレームから順に差分を積み上げて復元しないと、100フレーム目の絵は作れません。だから「狙ったフレームだけをピンポイントで取り出す(シーク seek)」のは、想像より重い処理です。

ASCII で GOP を描くと以下です。

```
I  P  P  P  P  B  P  P  ... I  P  P ...
↑単独で絵になる    ↑前のフレーム+差分で初めて絵になる   ↑次のキーフレーム
[ 欲しいのが5番目のPでも、先頭のIから順に積み上げて復元が要る ]
```

### なぜ「学習時にデコード」だと困るのか
上述の通り、毎ステップで MP4 を開き、目的フレームまでシークし、GOP を展開してデコードしていると、GPU が画像の到着を待ってアイドル(手待ち)になります。CPU 側のデコードがデータ供給の律速になり、せっかくの GPU が遊びます。これがデータパイプラインのボトルネックです。CPU 1コアでの H.264 デコードは解像度にもよりますが毎秒数十〜数百フレーム程度で、複数 GPU で大バッチを回すと供給が追いつかなくなります。

### フレームとタイムスタンプの整合
カメラ映像と他のセンサー(自車運動 egomotion、LiDAR など)は別々のクロックで記録されることが多く、「カメラの何フレーム目が、自車運動のどの時刻に対応するか」を時刻で突き合わせる必要があります。動画の各フレームにはタイムスタンプ(PTS, Presentation Time Stamp と DTS, Decode Time Stamp)が付いており、正確に扱うにはこれを読める必要があります。B フレームがあると表示順とデコード順がずれるため、PTS を正しく扱わないと時刻が前後します。簡易なデコーダだと PTS の扱いが不正確で時刻ズレが起きるため、PyAV のような低レベルアクセスが好まれます。

## 仕組みを詳しく
典型的なデータパイプラインは2段階です。

### 第1段階: 生データのデコード(PyAV)
複数カメラの映像がそれぞれ MP4 で保存されているとします。前処理は次のステップを踏みます。

1. 基準センサー(例: 自車運動)のタイムスタンプ(マイクロ秒)を読む。
2. カメラ側のフレームごとのタイムスタンプ表(例: parquet)と突き合わせ、一番近いフレーム番号を求める。これでカメラ画像と他センサーを時刻で揃える(時刻同期 temporal alignment)。
3. その動画を開き、目的フレーム番号までシークしてデコードし、RGB の配列(高さ × 幅 × 3)を得る。
4. PIL 画像などにして、リサイズ・正規化などの前処理 transform をかけ、テンソル (3, H, W) にする。
5. 複数カメラ分を積み重ね、(V, 3, H, W) という1サンプルの視覚入力にする(V はカメラ台数)。

このデコードの土台が FFmpeg であり、Python から扱うために PyAV(`import av`)を使います。FFmpeg 本体と PyAV を両方インストールするのが前提です。PyAV はコンテナ(MP4 という入れ物)→ ストリーム(その中の映像トラック)→ パケット(圧縮された塊)→ フレーム(デコード後の生画像)という動画の内部構造に1段ずつアクセスできるため、「キーフレームへシークしてから N フレーム進める」「PTS で位置を指定する」といった正確な操作ができます。

### 第2段階: フレームを抜き出してシャードへ焼き直す
学習のたびにデコードしないよう、デコード済みフレームを WebDataset 形式の tar シャードに事前に書き出します。シャード (shard) とは、たくさんの小さなサンプルを1つの大きな tar アーカイブにまとめたファイルで、学習時はこれを順番にストリームで読むだけにする入れ物です。

処理の中身(ステップ):
1. データセットを1サンプルずつ取り出す。各サンプルは「視覚入力」「センサー履歴」「予測目標」などを共通スキーマに揃える。
2. 各カメラのフレームを目標解像度(例 256×256)にリサイズし、JPEG(品質90など)にエンコードして tar に格納(`s00000001.cam_0.jpg` のような名前)。
3. 数値配列(センサー履歴・目標など)を `.npy` として格納。
4. メタ情報を `.json` として格納。
5. 1シャードあたり一定数(例 1000)たまったら次のシャード(`train-000000.tar`, `train-000001.tar`, …)に切り替える。
6. 出来上がったシャード群をオブジェクトストレージ(S3 など)に置き、学習時はそれを読む。

これにより学習側は「圧縮動画を開いてシークしてデコードする」重い処理を一切せず、展開済みの JPEG を読むだけになります。tar の中身は格納順に並ぶので、シーケンシャル(順次)読み出しになり、HDD やネットワークストレージでも高いスループットが出ます。

### before / after の効果
- before(学習時デコード): 毎ステップで MP4 を開く → I フレームから差分を積んで目的フレームを復元 → GPU が待つ。
- after(事前抽出シャード): 学習前に1回だけデコードして JPEG 化 → 学習時は JPEG を読むだけ → GPU が待たない。

「1回だけ重い処理をして、あとは軽くする」典型的なトレードオフで、学習を何周も回すほど効果が出ます。

### 数値例
7カメラ・30分・10Hz・学習用に2Hzサンプリングとすると、1カメラあたり 30分×60秒×2 = 3600 フレーム、7カメラで約25000フレーム。これを毎エポック学習時にデコードすると、シークと GOP 展開のコストが毎周乗ります。事前抽出ならこのデコードは生涯1回。仮に1エポックのデコードに10分かかるなら、50エポックで約8時間の節約になります。逆に言えば、エポック数が少ない(1〜数回しか回さない)使い捨て実験では、事前抽出のオーバーヘッドが回収できないこともあり、構成は学習回数で決まります。

## 手法の系譜と主要論文

### FFmpeg / PyAV
FFmpeg は動画・音声を扱うデファクト標準のオープンソースエンジンで、ほぼあらゆる圧縮方式をデコード/エンコードできます。PyAV はその C ライブラリ(libavcodec / libavformat など)を Python から直接叩けるようにしたバインディングです。動機は「OpenCV の `VideoCapture` のような簡易インターフェースだと、フレーム単位の正確なシークやタイムスタンプ取得が不安定になりがちで、PyAV ならコンテナ・ストリーム・パケット・フレームという動画の内部構造に正確にアクセスできる」こと。効果は正確なフレーム指定と PTS 整合。トレードオフは、API が低レベルで学習コストがあり、I/P/B フレームの内部構造を理解していないと使いこなしにくい点です。

### GPU デコード(NVDEC / DALI / torchvision の GPU パス)
近年は、デコード自体を GPU の専用ハードウェア(NVIDIA の NVDEC、各 GPU に載る固定機能デコーダ)で行い、CPU ボトルネックを回避する選択肢があります。NVIDIA DALI は「読み込み・デコード・拡張」を GPU 上でパイプライン化するライブラリで、CPU デコードが律速になる大規模学習で使われます。動機は CPU デコードの限界突破。NVDEC は CUDA コアとは別の専用回路なので、デコードしながら GPU 本体の学習計算を邪魔しにくいのが利点。トレードオフは、ハードウェアと対応コーデック(H.264/H.265/AV1 など世代依存)に縛られ、セットアップが複雑になる点、NVDEC のセッション数や同時デコード本数に上限がある点です。事前抽出と GPU デコードは「重さをどこで払うか(前か、走らせながらか)」という別解で、状況により選びます。

### WebDataset (Aizman et al., IEEE BigData 2019)
データを tar シャードに固め、順番にストリームで読むことで、大規模データの I/O を高速化するライブラリ。動機は、何百万もの小さなファイルを個別に開くとファイルシステム(やオブジェクトストレージ)が遅くなる(1ファイルごとに open/seek/close のメタデータ操作が走る)ため、まとめ読みで速くすること。効果は学習時のデータ供給の高速化と、S3 等オブジェクトストレージとの相性の良さ(HTTP range で tar を順に流せる)。トレードオフは、ランダムアクセス(任意サンプルへ即ジャンプ)が苦手で、シャッフルがシャード単位+バッファ内に限られる点です。抽出済みフレームをこの形式で吐き出すのが定番です。

## 論文の実験結果(定量データ)
このトピックは単一の論文成果ではなく工学的定石ですが、I/O が学習速度を律速する事実は広く定量化されています。WebDataset の文献では、数百万ファイルを個別に開くファイルシステムアクセスに比べ、シャード化したシーケンシャル読み出しが I/O スループットを大幅(報告により数倍規模、ストレージ種別によっては桁規模)に改善し、ネットワークストレージ越しでも GPU を飽和させやすいことが示されています。指標は「秒あたりサンプル数(samples/sec)」や「GPU 使用率(%)」で、前者は大きいほど、後者は高く安定しているほど良いデータパイプラインです。

実務上の典型的な観測:
- 学習時デコード構成では GPU 使用率が 30〜60% で頭打ちになりやすい(CPU デコード律速)。`nvidia-smi` の利用率がのこぎり状に上下するのが手待ちの兆候。
- 事前抽出+シャード化に切り替えると GPU 使用率が 90%以上 に張り付き、エポック時間が短縮される。
- JPEG 品質を下げる(例 90→75)とシャード容量と読み込み量が減り I/O はさらに軽くなるが、画質劣化が精度に響く境目を探る必要がある。アブレーションとして JPEG 品質を振り、検証精度がほぼ変わらない最低品質を選ぶのが定石です(多くの認識タスクでは品質75〜85程度まで精度がほぼ落ちないと経験的に知られる)。
- DALI / NVDEC を使った GPU デコードの文献でも、CPU デコード比でデータローダのスループットが数倍に伸び、CPU コア数の制約から解放される報告がある。

## メリット・トレードオフ・限界

メリット
- 圧縮動画のままなら保存容量が小さく、生データの保管・転送が現実的。
- PyAV/FFmpeg で正確なフレーム指定と PTS 整合ができ、カメラと他センサーを正しく時刻合わせできる。
- 事前抽出+シャード化で学習時の I/O ボトルネックを解消し、GPU を遊ばせない。

トレードオフ・限界
- 事前抽出には1回分のデコードコスト(時間・計算)がかかり、抽出後の JPEG は元動画より容量が増える(展開するため)。
- JPEG 再圧縮で画質が劣化する。元が H.264 圧縮ですでに劣化しているところに JPEG をかける「二重圧縮」になり、微細なテクスチャ(遠くの標識文字、細い車線)が失われうる。可逆形式(PNG)にすると容量が跳ね上がるトレードオフ。
- 抽出時にフレームレート・解像度を固定するので、後から別解像度・別サンプリングが欲しくなると抽出し直しが必要(柔軟性の喪失)。
- WebDataset はランダムアクセスが弱く、厳密な全体シャッフルはしにくい。シャード分割とバッファサイズ(shuffle buffer)の設計で擬似的にシャッフルする。時系列の塊が固まりすぎるとバッチの多様性が落ち学習が不安定になりうる。
- 事前抽出は元動画と抽出物の二重保管が要り、両者の対応(どのシャードがどの動画コミット由来か)を管理しないと再現性が崩れる。ここで [データバージョニング (LakeFS)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1197_lakefs-data-versioning) のようなデータ版管理が効いてきます。

## 発展トピック・研究の最前線
- GPU デコードパイプライン(DALI / NVDEC)で「学習しながらデコード」しても GPU を律速させない構成。事前抽出の保管二重化を避けつつ供給を間に合わせる方向。
- メモリ/ローカル NVMe へのキャッシュ階層: 初回デコードをキャッシュし、2周目以降を高速化する中間策。クラウドでは「S3 → ローカル NVMe キャッシュ → GPU」の階層設計が定番。
- オンザフライ拡張との両立: 事前抽出すると幾何拡張(ランダムクロップ・回転等)の自由度が減るため、抽出は緩め(やや高解像度・低圧縮)に行い学習時に拡張する設計。
- 圧縮ドメイン学習 (compressed-domain learning): デコードせず圧縮表現(動きベクトルや残差など、コーデックが既に計算している運動情報)を直接モデルに食わせる研究方向。デコードコスト自体を消す試みで、動作認識などで先行研究がある。
- フレーム精度シークの落とし穴対策: シーク後に最寄りキーフレームから正しく目的フレームへ進める実装(seek + decode-to-target)を堅牢化し、オフバイワン(1フレームずれ)のラベル時刻ずれを防ぐ。
