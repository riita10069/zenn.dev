# 状態量の高速読み出し (動画デコード回避)

## ひとことで言うと
速度や加速度といった「数値だけ」が欲しいのに、データを1件読むたびに何台ぶんものカメラ動画まで毎回デコード (展開) してしまい、1サンプルあたり数十秒もかかる、という落とし穴があります。数値が入っている列を直接読むようにすれば、これを実質ゼロまで短縮できます。本ノートは「学習速度はモデルだけでなくデータ供給の効率で決まる」という、実務上きわめて重要な経験則を、具体的な最適化を通して解説します。

## 直感的な理解
分厚い百科事典 (DVD 付き) があるとします。各項目には文章 (数値データ) と、付属 DVD の動画 (カメラ映像) が紐づいています。あなたが知りたいのは、ある項目の「人口の数字」だけだとします。

ところが手元の検索ツールは、項目を1つ開くたびに「文章も DVD の動画も全部再生して」から渡す仕様だったとします。人口の数字 (たった数バイト) が欲しいだけなのに、毎回 DVD プレーヤーが立ち上がって動画を再生し終えるまで待たされる。これは明らかに無駄です。

賢いやり方は、「数字が欲しいときは文章のページだけ見て、動画は本当に映像が要るときだけ再生する」ことです。データ読み込みでもこれと同じで、数値が欲しいだけのときに動画デコードを通る経路を避けるだけで、桁違いに速くなります。

なぜこれが研究上も重要かというと、GPU は非常に高速なので、データの供給が追いつかないと GPU が遊んでしまうからです。高価な GPU が「次のデータ待ち」でアイドルしている状態を data starvation (データ枯渇) と呼びます。モデルをいくら速くしても、データローダが遅ければ全体の学習速度はデータローダで頭打ちになります。これはアムダールの法則 (全体の高速化は最も遅い部分で律速される) の典型例です。

## 基礎: 前提となる概念

用語を噛み砕きます。

デコード (decode) とは、mp4 などに圧縮された動画を、計算に使える生のピクセル配列に展開する処理です。動画は容量を小さくするため、全フレームを丸ごと保存するのではなく、ときどき完全なフレーム (キーフレーム、I フレーム) を置き、その間は「前のフレームからの差分」だけを記録します (時間方向の圧縮、フレーム間予測)。そのため、ある1フレームを取り出すだけでも、直前のキーフレームまで遡ってそこから差分を順に適用する必要があり、思いのほか時間がかかります。マルチカメラのデータセットは1フレームを得るのに複数本の動画を同時にデコードするので、コストはおおよそカメラ台数に比例します。

状態量 (state) とは、車両の速度・進行方向 (heading)・加速度・位置 (緯度経度高度) といった数値の集まりです。これらは画像ではなくただの浮動小数点数なので、本来は一瞬で読めます。模倣学習では、過去フレームの状態量から「自車がどう動いてきたか (エゴモーション履歴)」を、未来フレームの状態量から「これからどう動くべきか (軌跡の正解)」を作ります。つまり状態量は「エピソード全体ぶん (数百フレーム)」をまとめて欲しいことが多いのです。

列指向フォーマット (columnar format) とは、表データを「行ごと」ではなく「列ごと」に並べて保存する方式です。Apache Arrow や Parquet が代表で、HuggingFace の `datasets` ライブラリも内部で Arrow を使います。

```
行指向 (row-major) のディスク上の並び:
  [行0: 速度,heading,画像ptr][行1: 速度,heading,画像ptr][行2: ...]
  → 速度だけ欲しくても、各行の全列を飛び飛びに舐める必要がある

列指向 (columnar) のディスク上の並び:
  [速度列: v0,v1,v2,...][heading列: h0,h1,...][画像ptr列: ...]
  → 速度列だけを連続して一気に読める。画像ptr列には触れない
```

列指向の利点は「特定の1列だけを、他の列を一切読まずに高速に取り出せる」ことです。連続したメモリ領域をまとめて読むので、CPU キャッシュやディスク I/O とも相性が良く、SIMD (1命令で複数データを処理) 最適化も効きます。

二層構造の理解が鍵です。多くのロボット学習データセットは、内部的に次の2層を持ちます。

- 下層 (表): 数値・文字列を行・列で保持する列指向の表 (Arrow 形式)。状態量はここにある。読み出しは速い。
- 上層 (高レベル API): 上記の表に加えて、動画ファイルから画像をデコードして合体させた「1フレーム完全版」を返す便利な関数。`dataset[i]` のように添字でアクセスでき、画像も状態量もまとめて返してくれる。便利だが、画像デコードを伴うので遅い。

## 仕組みを詳しく

問題の核心は、状態量という「下層の表にある数値」を、わざわざ上層の高レベル API 経由で1フレームずつ取りに行っていた点です。

```
遅い経路 (アンチパターン):
  for i in range(ep_start, ep_end):       # エピソード全体ぶん繰り返す
      frame = dataset[i]                   # i フレーム目を高レベルAPIで取得
      states.append(frame["state"])        # ← 欲しいのはこの数値だけ
  # dataset[i] は毎回、全カメラの動画をデコードする!
  # 数値8個のために、エピソード長 × カメラ台数 ぶんのデコードが走る
```

たとえば7カメラ構成で1フレームの取得に約35秒かかるとします。20ステップ × バッチ4 = 80サンプルでこれを繰り返すと、状態量を読むだけで `80 × 35 ≒ 2800秒` (約47分)。学習どころか短いテストすら現実的に回りません。GPU はその間ずっと next batch を待ってアイドルし、GPU 利用率は数パーセントまで落ちます。

解決策は、「数値が欲しいときは動画デコードの経路を通らず、下層の表を直接スライスする」ことです。

```python
import numpy as np

def get_state_window(hf_dataset, ep_start, ep_end):
    col = hf_dataset.select_columns(["observation.state.vehicle"])  # 数値列だけ選ぶ
    states = np.asarray(
        col[ep_start:ep_end]["observation.state.vehicle"], dtype=np.float32
    )
    return states  # shape: (フレーム数, 状態次元)
```

ステップに分解します。

1. 下層の表 (`hf_dataset`) を直接参照する。高レベルの `dataset[i]` ではなく、Arrow 表そのものに触る。
2. `select_columns([...])` で、欲しい数値列だけに絞る。画像へのポインタなど他の列は読み込まない。
3. `col[ep_start:ep_end]` でエピソードの行範囲をスライスし、状態量だけを一括取得する。動画には一切触れない。
4. `np.asarray(..., dtype=np.float32)` で NumPy 配列にする。形状は `(エピソードのフレーム数, 状態次元)`。

状態次元が8の典型例は `[speed, heading, heading_error, latitude, longitude, altitude, acceleration_x, acceleration_y]` のような並びで、形状はたとえば `(300, 8)` (300フレーム × 8次元) になります。ここから後段の処理がエゴモーション履歴 (たとえば過去64フレームを平坦化した数百次元のベクトル) や軌跡の正解 (たとえば未来64フレームの (x, y) を並べたベクトル) を導出します。

before/after の対比:

```
before: states = [dataset[i]["state"] for i in range(ep_start, ep_end)]
        → 各 i で全カメラの動画をデコード。約35秒/サンプル

after:  states = hf_dataset.select_columns(["...state..."])[ep_start:ep_end]
        → 列指向の一括スライス。動画デコードゼロ。約0ms
```

ここで重要な設計判断は、画像と数値で経路を分けることです。画像が本当に必要な「現在の1フレーム」だけは、これまで通り高レベル API でデコードします。つまり「数値は表から速く (過去・未来の数百フレームぶん)、画像は必要な1フレームだけデコード」という役割分担です。この分担が成立する前提は、数値スライスの添字と画像取得の添字が同じフレームを指していることです。だからこそ [エピソードのローカル行レンジ算出 (添字空間の整合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0814_episode-local-row-ranges) でローカル行に統一しておくことが、この最適化の安全性を支えます。添字がズレると、数値と画像が別フレームのものになり、これも silent failure を招きます。

## 手法の系譜と主要論文

これは論文の「手法」ではなく、データエンジニアリングの定石です。背景となる技術の系譜を示します。

列指向ストレージの系譜は、Google の Dremel (Melnik et al., VLDB 2010) にさかのぼります。Dremel は「必要な列だけを読む」ことで、ペタバイト級のデータに対する集計を秒オーダで返す対話的クエリを実現しました。この思想がオープンソースの Apache Parquet (2013、ディスク保存形式) と Apache Arrow (2016、メモリ内表現) に受け継がれ、「分析ワークロードでは列指向が圧倒的に速い」ことが定着しました。HuggingFace `datasets` が Arrow を採用しているため、`select_columns` での列選択がそのまま高速化につながるのは、この系譜の直接の恩恵です。

データローダのボトルネック問題は、深層学習が大規模化した2018年前後から明確に意識されるようになりました。NVIDIA DALI (Data Loading Library, 2018) は、JPEG デコードや画像前処理 (リサイズ・正規化・データ拡張) を GPU 上で並列実行することで「CPU 側のデータローダが GPU を待たせる」状況を解消しようとしたツールです。同じく WebDataset (Aizman et al. "High Performance I/O For Large Scale Deep Learning", IEEE Big Data 2019) は、データを shard (数百MB単位の tar アーカイブのまとまり) にしてシーケンシャルに読むことで、無数の小ファイルへのランダムアクセスがもたらすディスクシークの遅延を回避しました。本ノートの「不要なデコードを避ける」工夫は、これらと同じ問題意識の上にあります。

遅延読み込み (lazy loading) と必要列のみの読み出し (projection pushdown、述語下げと並ぶ列指向クエリ最適化の基本) は、Apache Arrow / Parquet が設計思想として掲げる中核機能です。「データ全体をメモリに載せず、必要になった列・行だけをその場で読む」ことで、メモリ使用量と I/O を同時に削減します。

## 論文の実験結果(定量データ)

データローダ最適化の効果は、スループット (samples/sec または images/sec) と GPU 利用率 (GPU utilization、GPU が実際に計算している時間の割合) で測られます。

NVIDIA DALI の公式ベンチマークでは、ResNet-50 の学習で、CPU ベースの標準データローダ (torchvision の JPEG デコード) を DALI に置き換えると、データ供給がボトルネックだった構成で学習スループットが大きく向上したと報告されています。とりわけ高速な GPU (V100 や A100) を複数枚使い多数枚を学習するとき、CPU デコードでは GPU 利用率が大幅に落ちるのに対し、DALI ではほぼ飽和近くまで上げられます。これは「GPU が速いほどデータ供給の遅さが顕在化する」という関係を定量的に示しています。FFCV (Leclerc et al., CVPR 2023) はさらに踏み込み、ImageNet の ResNet-18/50 学習を、適切なハードウェアで1モデルあたり数十分〜数時間という、従来比で桁違いに短い時間で回せることを示しました。

列指向アクセスの効果は、本ノートの事例そのものが端的です。状態量の読み出しが1サンプルあたり約35秒から実質0msへ、という報告は、桁にして数万倍以上の短縮にあたります (なお保守的な記録では 35s → 1s と控えめに見積もる場合もありますが、それでも数十倍です)。差の本質は「数値8個を読むために全カメラの動画をデコードしていた無駄」を完全に除いたことにあります。

アブレーション的に整理すると、「どの要素を抜くと何が起きるか」は明快です。
- 列選択 (`select_columns`) を抜いて全列を読むと、画像ポインタ列などの処理が混じり遅くなる。
- 一括スライスをやめて1フレームずつループすると、Python のループオーバーヘッドと関数呼び出しコストが行数ぶん加算される。
- 最も致命的なのは「高レベル API で取る」ことを抜けない場合で、これは動画デコードを毎回起こし、上記の数十秒を生む。

これらの差が「実データでテストすら回らない」か「現実的な時間で学習が回る」かを分けるため、最適化の効果は精度指標ではなく「実験が成立するかどうか」という形で現れます。地味な I/O 最適化が研究の生死を分ける典型例です。

## メリット・トレードオフ・限界

メリット
- 状態量の読み出しが数十秒から実質ゼロへ。実データでの学習・テストが現実的な時間で回る
- 不要な動画デコードを完全に避けるので CPU 負荷も大きく下がり、GPU 枯渇 (data starvation) が解消する
- 列指向の一括スライスなので、エピソードが長くても効率的 (件数に対してほぼ線形、しかも係数が小さい)

トレードオフ・限界
- 特定の列名・状態次元レイアウトに依存する。データセット側でスキーマが変わると追従が必要
- 「数値は表から、画像は高レベル API から」という二経路を持つため、両者が同じフレームを指しているか (添字の整合) を実装者が保証しなければならない。これは [エピソードのローカル行レンジ算出 (添字空間の整合)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0814_episode-local-row-ranges) でローカル行に統一して担保する
- 下層の表が Arrow である、という内部実装に踏み込むため、ライブラリが内部構造を変えると壊れる可能性がある (公開 API ではなく実装詳細への依存)
- 画像はやはりデコードが必要なので、画像も大量に欲しい場合 (たとえば動画クリップ全体を入力にする world model) には、この最適化だけでは不十分で、DALI / WebDataset / GPU デコード的な別の対策が要る

## 発展トピック・研究の最前線

データローダの最適化は、大規模学習では今なお活発な研究領域です。FFCV (Leclerc et al., CVPR 2023) は、データを専用バイナリ形式に事前変換し、メモリマップ・JIT コンパイル・非同期 GPU 転送を組み合わせて、ImageNet 学習を桁違いに高速化しました。発想は本ノートと地続きで、「学習時の I/O を限界まで削る」ことを徹底しています。

ストレージ階層を意識した設計も最前線です。クラウドでの大規模学習では、データが S3 のようなオブジェクトストレージにあり、ネットワーク経由で読みます。このとき「シーケンシャルに大きなチャンクで読む」ことが帯域を活かす鍵で、WebDataset の shard 設計や、ランダムアクセスを擬似的に実現する shuffle buffer (一定数のサンプルをバッファに溜めてその中でシャッフルする) が使われます。MosaicML の StreamingDataset はこの方向の代表的な実装です。

ビデオ理解・world model の学習では、本ノートとは逆に「動画を大量にデコードすること自体」が主作業になります。ここでは GPU デコード (NVIDIA NVDEC、ハードウェア動画デコーダ) を使って動画展開を GPU にオフロードしたり、デコード済みフレームをキャッシュしたりする手法が研究されています。つまり「何を速くすべきか」はタスク依存であり、状態量だけ欲しいなら本ノートの列読み出し、画像系列が主役ならデコード自体の高速化、と使い分けるのが研究者の判断になります。最適化はつねに「自分のボトルネックがどこか」をプロファイルで特定してから着手するのが鉄則です。
