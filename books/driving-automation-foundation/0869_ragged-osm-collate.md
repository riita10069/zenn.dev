# 可変長フィールドの collate(ragged バッチ化)

## ひとことで言うと

ディープラーニングの学習では、データを1件ずつではなく「バッチ」というまとまりにして GPU に流します。ところが、サンプルごとに長さの違うデータ(たとえば周囲の地図要素の集合)は、画像のようにきれいに積み重ねられません。この問題を「長さが可変のものは Python のリストにまとめ、形がそろっているものだけ通常どおり積み重ねる」と役割分担して解く collate 関数の話です。collate は「照合して束ねる」という意味です。

## 直感的な理解

引っ越しの荷造りを思い浮かべてください。同じ規格のダンボール(画像のように形がそろったデータ)は、きれいに積み上げて1つの山にできます。一方で、傘・観葉植物・自転車のように形も長さもバラバラなもの(可変長データ)は、無理に重ねようとすると崩れます。こういうものは「とりあえずカゴに並べて入れておき、運ぶ先で個別に扱う」のが現実的です。

ragged バッチ化の発想はこれと同じです。形がそろったフィールドは「積み重ねて1つの大きなテンソルにする」、形がバラバラなフィールドは「リストに並べておき、後段(モデルの中)で必要になったときに揃える」と分けます。なぜ後段で揃えるのかというと、バッチを作る段階で全部を最大長に合わせて穴埋めすると、巨大な無駄テンソルを毎回作ることになり、メモリと転送が無駄になるからです。揃える作業は、本当に必要になるGPU上の処理の直前まで遅らせるのが効率的です。

## 基礎: 前提となる概念

**バッチ(batch)** とは、複数のサンプルを1つの大きなテンソル(多次元配列)に積み重ねたものです。たとえばカラー画像 `[3, 256, 256]`(チャンネル3、縦256、横256)を8件まとめると `[8, 3, 256, 256]` という1つのテンソルになります。GPU は同じ計算を大量に並列実行するのが得意なので、1件ずつより8件まとめた方がはるかに高速です。先頭の次元 B(バッチサイズ)が「何件まとめたか」を表します。

**collate 関数 / collate_fn**。PyTorch の DataLoader は、データセットから取り出した個々のサンプルを「束ねて」バッチにします。この束ねる処理を担うのが collate 関数で、指定しないと標準の `default_collate` が使われます。`default_collate` は、すべてのサンプルで形(shape)がそろっている場合だけ積み重ねられます。画像のように常に `[3, 256, 256]` なら問題ありませんが、形が違うと `stack expects each tensor to be equal size` のようなエラーになります。

**ragged data(ギザギザのデータ)**。サンプルごとに長さが違うデータを ragged data と呼びます(ragged は「端がそろっていない、ギザギザの」の意。jagged とも言う)。たとえば地図要素のトークン列を考えると、市街地の交差点では周囲に何十もの道路・歩道要素があってトークン数が多く、郊外の一本道では数個しかありません。サンプルAは「地図要素40個 → `[40, C]`」、サンプルBは「12個 → `[12, C]`」のように要素数 N がバラバラです(C は1要素あたりの特徴次元)。自然言語でも、文ごとに単語数が違うので同じ問題が起きます。点群(LiDAR)も同様に点数がフレームごとに変わります。

**トークン(token)とトークン化**。連続的なデータや構造化データを、モデルが扱える「1つの単位ベクトル」の列に分解することをトークン化と呼びます。地図なら「1要素=1トークン」、文なら「1単語(やサブワード)=1トークン」です。トークン数がサンプルで変わることが、ragged 問題の根本原因です。

**パディングとマスク**。可変長を揃えるには、短いものに「穴埋め(padding)」を足してバッチ内最大長にそろえます。ただし、どこが本物でどこが穴埋めかをモデルに教えないと、存在しない要素を見て予測が歪みます。これを示すのが key padding mask で、「ここは本物=False、ここは穴埋め=True」を表すブール配列 `[B, N]` です(B はバッチサイズ、N は最大長)。Transformer の Attention はこのマスクを受け取り、True の位置のスコアを −∞ に置き換えてから softmax を取ることで、穴埋めへの注意の重みを実質ゼロにします。

## 仕組みを詳しく

ragged 対応の collate 関数は、1バッチ分のサンプル(各サンプルは辞書 `dict`)を受け取り、キー名で2種類に振り分けます。典型的な実装は、可変長フィールドを特定の接頭辞(たとえば地図系なら `osm_map_` のような命名規約)で見分け、それ以外(画像、自車運動、正解軌跡など)を通常データとみなします。

処理のステップは次の通りです。

1. バッチが空なら空辞書 `{}` を返す(防御的な早期リターン)。
2. 先頭サンプルのキー一覧を取り、可変長フィールドのキー群と、それ以外のキー群に分ける。
3. それ以外のキーだけを取り出した小さな辞書のリストを作り、`default_collate` に渡して普通に積み重ねる。画像なら `[8, 3, 256, 256]` のようにテンソル化される。
4. 可変長フィールドの各キーは、積み重ねずに「長さ B(バッチサイズ)の Python リスト」にまとめる。`out[k] = [s[k] for s in batch]` のように、サンプルごとの可変長テンソルをそのままリストの要素として並べる。
5. 両者を1つの辞書に統合して返す。

before/after を具体例で示します。入力バッチ(B=2)が

```
[
  {"image": tensor[3,256,256], "map_tokens": tensor[40, C], "map_relations": tensor[40,40]},
  {"image": tensor[3,256,256], "map_tokens": tensor[12, C], "map_relations": tensor[12,12]},
]
```

のとき、出力は

```
{
  "image":         tensor[2, 3, 256, 256],       # default_collate で stack
  "map_tokens":    [tensor[40,C], tensor[12,C]],  # リストのまま(積まない)
  "map_relations": [tensor[40,40], tensor[12,12]],
}
```

になります。重要なのは「画像など既存フィールドの挙動はまったく変わらない」点です。可変長フィールドを持たないサンプル(接頭辞付きキーが無いだけ)は実質 `default_collate` と同じ扱いになり、既存パイプラインとの後方互換が保たれます。`map_relations` のような `[N, N]` の隣接行列(要素間の関係をグラフで表したもの)も、N がサンプルごとに違うのでリスト化対象になる点に注意してください。

### なぜ collate でパディングせず後段でやるのか

この collate 自体はパディングをしません。リストのまま下流に渡すだけで、実際に長さを揃えるのは地図(系列)エンコーダの中です。エンコーダがトークン列をバッチ最大長 N にパディングし、同時に key padding mask `[B, N]` を作ります。Attention(注目機構)はこのマスクを見て、穴埋め部分には注意を向けないようにします。穴埋めに注意が漏れると、存在しない地図要素や単語を見ているのと同じになり、予測が歪むためです。

collate 段階でパディングすると、全サンプルをバッチ最大長にそろえた巨大テンソルを毎回作ることになり、メモリと CPU→GPU 転送が無駄になります(短いサンプルが多いバッチほど穴埋めの割合が増える)。数値例で言えば、N が `[12, 12, 12, 40]` のバッチをすべて 40 にそろえると、実データ 76 トークンに対し総枠は 160 トークンとなり、半分以上が穴埋めです。リストのまま渡し、GPU 上のエンコーダで必要なときだけ揃える方が効率的で、マスク生成もエンコーダ内で一貫管理できます。これは「責務をエンコーダに寄せる」設計判断であり、パディングとマスクが必ずペアで生成されるため「マスクを付け忘れる」種類のバグが構造的に起きにくくなる利点もあります。

## 手法の系譜と主要論文

可変長系列のバッチ化とパディングマスクは、自然言語処理(NLP)で確立された定石であり、地図・軌跡をトークン集合として扱う流れと合流して現在の形になりました。

- Vaswani et al. "Attention Is All You Need"(NeurIPS 2017, arXiv:1706.03762, Google)。Transformer の原論文で、文ごとに長さの違う単語列を扱うため padding mask の考え方を整理しました。提案は「Self-Attention で系列内のどの要素にも直接注目できる構造」で、その副作用として「揃えるための穴埋めに注意が漏れないようマスクで遮断する」必要が生じました。効果は可変長系列を1バッチで並列処理できること、トレードオフはマスク管理のミスが静かなバグ(精度劣化)になりやすい点です。
- Gao et al. "VectorNet"(CVPR 2020, arXiv:2005.04259, Waymo)。地図や軌跡を「ベクトル化されたポリラインの集合」としてグラフ符号化する流れの起点です。提案の動機は、地図をラスタ画像にすると解像度と計算量のトレードオフが厳しく、ベクトル表現の方が幾何を正確かつ疎に表せるから。効果は表現効率の向上ですが、トレードオフとして「要素数がシーンごとに変わる=ragged になる」性質が生まれ、まさに本トピックの collate が対処する対象です。
- VectorNet 以降、軌跡予測の TNT・DenseTNT(Zhao et al., CoRL 2020 / Gu et al., ICCV 2021)、地図ベクトル化の VectorMapNet(Liu et al., ICML 2023)・MapTR(Liao et al., ICLR 2023, arXiv:2208.14437)など、トークン集合を Transformer に通す手法が主流化し、いずれも可変長バッチ化を前提とします。MapTR は地図要素を順序不変な点集合として表現し、要素数の可変性とマッチングを正面から扱った点で代表的です。
- SDTagNet(Immel et al., NeurIPS 2025)は OSM のタグをテキストとして埋め込み SD マップを符号化する手法で、点レベルのトークン列を Transformer に通すため、サンプルごとにトークン数が変わる可変長バッチ化を前提とします。

## 論文の実験結果(定量データ)

collate 関数そのものに精度ベンチマークはありませんが、可変長処理の設計が性能・速度に与える影響は定量的に語られています。

- マスクの正しさが精度に直結: Transformer 系では、padding mask を正しく適用しないと、穴埋めトークンにも注意が分配され、系列が長くゼロパディングが多いバッチほど劣化が顕著になります。NLP の実装事例では、マスク欠落が BLEU(機械翻訳の精度指標)や分類精度で数ポイント単位の劣化を生むことが知られ、「静かなバグ」の典型例として頻繁に報告されます。穴埋めの割合が大きいバッチほど影響が大きい、という距離依存も観測されます。
- パディング戦略と計算量: Attention の計算量は系列長 N に対して O(N^2) です。バッチ全体を最大長に揃えると、短いサンプルが混じるバッチでは無駄な計算が二乗で効きます。長さの近いサンプルをまとめる length bucketing(長さ別の箱に仕分けてからバッチを組む)や、可変長を保ったまま処理する手法は、無駄パディングを減らして実効スループットを改善します。NLP の学習では bucketing で学習スループットが数十%改善する報告があり、長系列タスクほど効果が大きい。
- アブレーション的観点: collate でリスト化する設計を外して全部 `default_collate` に通すと、長さの異なる地図/系列フィールドで即座に stack エラーになり学習が始まりません。逆に collate 段階で最大長パディングする設計にすると動きはしますが、バッチ内に1つでも極端に長いサンプル(交差点が密な都心など)があると、その長さに全サンプルが引き伸ばされ、メモリピークと転送量が跳ね上がり、まれに OOM(メモリ不足で異常終了)を起こします。

## メリット・トレードオフ・限界

メリット

- 可変長フィールドを、既存データの挙動を一切変えずにバッチへ同居させられる(後方互換)。
- パディングを下流(エンコーダ)へ遅延させるため、collate 段階で巨大な穴埋めテンソルを作らずに済む。
- 実装が単純(キー名で振り分けるだけ)で読みやすく、壊れにくい。パディングとマスク生成を1か所(エンコーダ)に集約でき、整合性を保ちやすい。

トレードオフ・限界

- 振り分けがキー名の接頭辞に依存するため、命名規約を守らないとリスト化されず `default_collate` でエラーになる(暗黙の契約に依存)。
- 可変長フィールドが Python リストのまま下流へ渡るので、GPU へ載せる・パディングする責任がエンコーダ側に移る(責務の分散)。
- 先頭サンプルのキーで判定する実装だと、サンプルごとにキー構成が異なる場合に想定外の挙動になりうる。
- リストのまま渡すと、`pin_memory`(ページ固定メモリで CPU→GPU 転送を高速化する仕組み)や非同期転送の最適化が効きにくくなることがある(まとまったテンソルでないため)。

未解決・実務的論点として、極端に長いサンプルによるメモリスパイクの抑制(動的バッチサイズや length bucketing、トークン数上限でのサブサンプリング)、nested tensor / jagged tensor(PyTorch が可変長を一級市民として扱う新機能)の活用、マスク生成を collate とエンコーダのどちらに置くかの責務設計が挙げられます。

## 発展トピック・研究の最前線

- Nested/Jagged Tensor: PyTorch が可変長テンソルをネイティブ表現として扱う仕組み。手書きのリスト+マスク管理を置き換え、可変長 Attention を効率実装できる方向。API として offsets を持ち、内部で packed 表現を扱う。
- FlashAttention の可変長カーネル(varlen): パディングを物理的に詰めた状態(packed sequence、各サンプルの境界を cu_seqlens で管理)で Attention を計算し、無駄計算を消す。長系列・大バッチで顕著に高速で、メモリも線形に近づく。
- Length bucketing / sorted batching: 似た長さのサンプルをまとめてパディング率を下げる古典的だが効果的な手法。バッチ内分散を下げるほど無駄が減る。
- グラフ・集合の collate: 地図や軌跡を集合・グラフとして扱う場合、PyTorch Geometric の Batch のように「全サンプルを1つの大きなグラフに連結し batch index で区切る」方式があり、ragged を別の角度から解決する。N×N の隣接行列を疎な辺リスト(edge_index)で持つことで二乗メモリを避けられる。
