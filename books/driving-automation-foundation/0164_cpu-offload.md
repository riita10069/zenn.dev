# CPU/NVMe Offload

## ひとことで言うと
GPU のメモリ(VRAM、ビデオメモリ)に入り切らないほど大きなニューラルネットを学習するために、いま使っていないデータ(主に最適化器の内部状態やパラメータ)を、容量の大きい CPU 側のメモリ(RAM)や、さらに大容量の NVMe ストレージ(高速な SSD)に一時的に逃がしておく技術です。必要になったぶんだけ GPU に運び込み、使い終わったらまた逃がします。代表的な手法が ZeRO-Offload と ZeRO-Infinity です。

## 直感的な理解
机の上(GPU メモリ)が狭くて、いま開いている本を全部は広げられない状況を想像してください。よく使う本だけ机の上に置き、すぐには使わない本は近くの本棚(CPU メモリ)、めったに使わない本は別室の倉庫(NVMe)にしまう。必要になったら取りに行く。こうすれば狭い机でもたくさんの本を「扱える」ようになります。代わりに、本棚や倉庫まで取りに行く往復の手間(データ転送)が増えます。これがオフロードの本質です。

なぜこれが要るのか。深層学習のモデルは年々巨大化していて、パラメータが数十億から数千億に達するものも珍しくありません。一方 GPU の VRAM は 1 枚あたり 24GB や 48GB、高価なデータセンタ向けでも 80GB 程度で、簡単には増やせません。GPU を増やせばよいのですが、GPU は高価で、多数を揃えられる組織は限られます。「手元の少ない GPU で、できるだけ大きなモデルを学習したい」という現実的な要求に応えるのがオフロードです。ZeRO-Offload の論文タイトルにある「Democratizing(民主化)」は、まさに「多 GPU を持たない人でも大規模学習を」という思想を表しています。

## 基礎: 前提となる概念
オフロードを理解するには、まず「学習中に GPU メモリを食うものは何か」を知る必要があります。大きく分けて 4 種類あります。

1. パラメータ(モデルの重み本体)。ネットワークが持つ学習対象の数値そのものです。
2. 勾配(gradient)。各パラメータを「どちらへどれだけ動かせば誤差が減るか」を表す値で、パラメータと同じ個数だけ存在します。
3. 最適化器(optimizer)の状態。最適化器とはパラメータの更新ルールのことです。広く使われる Adam という最適化器は、各パラメータごとに「移動平均(momentum、過去の勾配の平らな平均)」と「分散の推定(variance、勾配の大きさのばらつき)」という 2 つの補助変数を持ちます。さらに混合精度学習では、更新精度を保つためパラメータの高精度コピー(fp32)も持ちます。
4. 活性化(activation)。順伝播(forward、入力から出力を計算する過程)の途中結果で、逆伝播(backward、勾配を計算する過程)で再利用するため保持します。

混合精度学習(fp16 や bf16 という 16 ビットの低精度で計算しつつ、要所は fp32 という 32 ビットの高精度を保つ手法)で Adam を使う場合のメモリを、パラメータ数を P として概算すると次のようになります。

```
パラメータ(fp16):          2P バイト
勾配(fp16):                2P バイト
Adam: 重みの fp32 コピー:   4P バイト
Adam: momentum (fp32):      4P バイト
Adam: variance (fp32):      4P バイト
合計 ≈ 16P バイト(活性化は別計算)
```

ここで衝撃的なのは、純粋な「モデル本体」である fp16 パラメータは 2P バイトに過ぎないのに、最適化器まわり(fp32 コピー + momentum + variance、合わせて 12P バイト)が全体の 4 分の 3 を占めることです。10 億(1B)パラメータのモデルなら、最適化器関連だけで約 12GB、合計 16GB に達します。これに活性化を足すと、48GB の GPU でもすぐに足りなくなります。この「12P をどう逃がすか」が ZeRO-Offload の主戦場です。

重要な観察は、「最適化器の状態は forward と backward の最中には一切使われず、パラメータ更新(optimizer step)のときだけ使われる」という点です。つまり、forward/backward を回している間、最適化器の状態は GPU 上で場所を取っているだけで遊んでいます。これが「逃がす候補」として最適な理由です。

## 仕組みを詳しく
ZeRO-Offload(DeepSpeed という分散学習ライブラリの機能、Ren ら 2021)を例に、何をどこに置き、どこで計算するかを整理します。

### 役割分担: どの計算をどこでやるか
ZeRO-Offload の設計思想は「GPU と CPU で仕事を最適に分担する」ことです。重い計算は GPU、メモリを食うが計算は軽い処理は CPU に割り当てます。論文ではこれを「データフローグラフを GPU 側と CPU 側に分割する最適な切り方」として定式化し、ある条件下で通信量を最小にする分割が一意に定まることを示しています。

```
GPU 側で実行: forward と backward(行列積など計算量の多い処理。GPU が圧倒的に得意)
CPU 側で実行: パラメータ更新(optimizer step。要素ごとの単純な算術で計算量は軽い)
CPU 側に保管: 最適化器の状態(momentum, variance, fp32 重みコピー)
```

なぜこの分担が合理的かというと、計算量の観点で見るとわかります。forward と backward の計算量はおおよそ「パラメータ数 × バッチ内のトークン数(またはサンプル数)」に比例し、非常に大きいです。一方 optimizer step の計算量は「パラメータ数」に比例するだけで、バッチサイズには依存しません。つまり大きなバッチで学習するほど、forward/backward に対して optimizer step は相対的に軽くなり、CPU で実行しても全体のボトルネックになりにくいのです。

1 ステップの流れをイメージで追います。

```
1. GPU が forward を計算(パラメータは GPU 上の fp16)
2. GPU が backward を計算 → fp16 の勾配ができる
3. 勾配を GPU → CPU へ転送
4. CPU が optimizer step を実行:
   CPU 上の momentum / variance / fp32 重みを使ってパラメータを更新
5. 更新後のパラメータ(fp16)を CPU → GPU へ転送
6. 次のステップへ
```

こうすると、最もメモリを食う最適化器状態(前述の 16P のうち 12P 相当)が GPU からなくなります。GPU には forward/backward に最低限必要なもの(fp16 パラメータ、勾配、活性化)だけが残るので、同じ GPU でずっと大きなモデルが学習できます。

### CPU を遅くしない 2 つの工夫
単純に optimizer step を CPU でやると、CPU の Adam が遅くて全体のボトルネックになりかねません。ZeRO-Offload は 2 つの工夫でこれを抑えます。

第一に「高速 CPU Adam 実装」。SIMD(1 命令で複数データを並列計算する CPU の機能)やキャッシュを意識した手書き最適化で、素朴な実装より約 6 倍速い CPU Adam を提供します。これにより CPU での更新が forward/backward と隠れる程度に収まります。

第二に「one-step delayed parameter update(1 ステップ遅延更新、オプション)」。CPU の更新計算を、次のステップの GPU の forward/backward と時間的に重ねる(オーバーラップさせる)ことで、CPU の計算待ちで GPU が遊ぶのを防ぎます。代償として更新が 1 ステップ古いパラメータに基づくことになりますが、論文では収束への悪影響はほぼ無視できると報告しています。

### NVMe へのさらなるオフロード
CPU の RAM でも足りないほど巨大なモデルでは、最適化器状態を NVMe SSD にまで逃がします(ZeRO-Infinity、Rajbhandari ら 2021)。階層は次の 3 段になります。

```
GPU メモリ(最速・最小): 数十 GB、いま計算に使うものだけ
CPU メモリ(中速・中容量): 数百 GB〜TB
NVMe SSD(低速・最大容量): TB 級、安価
```

使う直前に上の階層へ運び、使い終わったら下へ戻す、というデータの出し入れを自動化します。ZeRO-Infinity は単一ノードで 1 兆(trillion)パラメータ規模のモデルのメモリを収容できることを示しました。

### コストはデータ転送と帯域のボトルネック
このやり方の代償は「GPU と CPU(や NVMe)の間のデータ転送」です。GPU と CPU は PCIe というバスでつながっており、その帯域(単位時間に運べるデータ量)は、GPU 内部のメモリ帯域より一桁以上遅いのが普通です。たとえば GPU 内部のメモリ帯域が毎秒 1〜3TB あるのに対し、PCIe Gen4 x16 は片方向で毎秒約 32GB、NVMe SSD はさらに遅く毎秒数 GB です。

毎ステップでパラメータや勾配を行き来させると、転送待ちで GPU が遊ぶ時間が生まれます。ZeRO-Offload はこの転送を計算と重ねる(オーバーラップ、計算しながら裏で転送する)工夫や、CPU 側の Adam を高速化した実装で影響を抑えていますが、それでも純粋に GPU メモリだけで回す場合より学習スループットは下がるのが一般的です。NVMe オフロードまで使うと、低下はさらに顕著になります。

## 手法の系譜と主要論文
- Rajbhandari ら、「ZeRO: Memory Optimizations Toward Training Trillion Parameter Models」(SC 2020、arXiv:1910.02054)。系列の出発点です。複数 GPU でデータ並列学習する際、従来は各 GPU が「パラメータ・勾配・最適化器状態」をすべて重複して持っていました。ZeRO はこれらを GPU 間で分割して持つ(各 GPU は全体の一部だけ保持し、必要なときに集める)ことでメモリを削減しました。分割の度合いに応じて ZeRO Stage 1(最適化器状態を分割)、Stage 2(勾配も分割)、Stage 3(パラメータも分割)があります。オフロードは「分割の延長で、GPU の外、つまり CPU へ逃がす」発想として自然に出てきます。

- Ren ら、「ZeRO-Offload: Democratizing Billion-Scale Model Training」(USENIX ATC 2021、arXiv:2101.06840)。提案は前述の「最適化器状態を CPU に置き、optimizer step を CPU で実行、forward/backward は GPU」という GPU/CPU 分担です。動機は題名どおり「高価な多 GPU を持たない研究者や開発者でも大規模モデルを学習できるようにする(民主化)」こと。新規性は、単純に全部 CPU に置くのではなく、計算量と通信量の解析に基づいて「どの計算をどちらで実行するのが通信最小かつ効率最大か」を理論的に導いた点と、高速 CPU Adam・遅延更新にあります。

- Rajbhandari ら、「ZeRO-Infinity: Breaking the GPU Memory Wall for Extreme Scale Deep Learning」(SC 2021、arXiv:2104.07857)。CPU だけでなく NVMe まで階層的にオフロードし、限られた GPU で兆パラメータ級を狙う手法です。さらに「帯域中心の設計」として、転送と計算を細粒度で重ね、帯域を使い切る工夫(infinity offload engine、memory-centric tiling)を導入しました。トレードオフは、階層が増えるほど転送が増え、設定とチューニング(何をどこに逃がすか、転送と計算の重ね方)が複雑になることです。

- Zhao ら、「PyTorch FSDP: Experiences on Scaling Fully Sharded Data Parallel」(VLDB 2023)。FSDP(Fully Sharded Data Parallel)は ZeRO Stage 3 相当のシャーディングに CPU オフロードのオプションを備えた PyTorch 公式の実装で、研究コミュニティでは ZeRO/DeepSpeed とともに大規模学習の標準的な選択肢になっています。フレームワークに統合され、設定だけで使える点が普及を後押ししました。

## 論文の実験結果(定量データ)
ZeRO-Offload 論文の代表的な結果を、指標の意味とともに説明します。

- 学習可能なモデルサイズ。オフロードなしの素直なデータ並列では、32GB の V100 1 枚で学習できるのはおおよそ 1.3B パラメータが上限とされます。ZeRO-Offload を使うと、同じ 1 枚で最大 13B パラメータ規模を学習できたと報告しています。これは約 10 倍で、「同じハードウェアでどれだけ大きいモデルを扱えるか」という意味で決定的な差です。

- スループット(処理速度の指標)。論文は 1 枚の V100 上で 10B モデルを学習する際、毎秒 40 TFLOPS を超える演算スループットを達成したと報告しています。TFLOPS は 1 秒あたり何兆回の浮動小数点演算をしたかで、GPU をどれだけ効率よく使えているかの目安です。最適化器を CPU に逃がしているにもかかわらず、計算と転送の重ね合わせによって GPU の遊び時間を抑え、PyTorch 単体の小さいモデルに迫る実用的な効率を保てたことを示します。

- スケーラビリティ。ZeRO-Offload は複数 GPU(最大 128 枚規模)へも拡張でき、GPU 数にほぼ比例して総スループットが伸びる(ほぼ線形スケール)ことを示しました。これは、各 GPU が独立に CPU へオフロードできるため、GPU を増やしても CPU 側のボトルネックが致命的にならないことを意味します。Megatron など既存のモデル並列と組み合わせると、さらに大きなモデルへ拡張できることも示されています。

- アブレーション(どの要素を抜くとどうなるか)の観点。最適化器状態だけでなく勾配や活性化まで CPU に逃がすと、メモリはさらに減りますが転送量が増えてスループットが大きく落ちます。ZeRO-Offload が「最適化器状態と fp32 更新だけを CPU に置く」という配分を選んだのは、メモリ削減効果が最大の部分だけを逃がし、転送量を抑えるための設計上のバランスです。また高速 CPU Adam を素朴実装に置き換えると CPU 更新がボトルネック化し、遅延更新を切るとオーバーラップが効かず GPU の空き時間が増えます。逆に何も逃がさなければ高速ですが大きなモデルは入りません。この「メモリ削減 vs 速度」のトレードオフ曲線上で、ちょうど良い点を選ぶのが手法の肝です。

ZeRO-Infinity 論文はさらに、単一 DGX-2 ノード(16 GPU)で 1 兆パラメータのモデルを収容・学習できること、512 GPU で 32 兆パラメータ規模のメモリを扱えることを報告し、「GPU メモリの壁」を超えられることを定量的に示しました。GPU あたり実効 40 TFLOPS 超を維持しつつ兆級を回せた点が要点です。

## メリット・トレードオフ・限界
メリット
- GPU メモリに入り切らない大きなモデルを、少ない(または 1 枚の)GPU で学習できる。
- 最適化器状態という「更新時しか使わない大物」を逃がすため、削減効果が大きい(全体の約 4 分の 3 を占める部分を外せる)。
- DeepSpeed や FSDP などのライブラリで設定だけで使え、モデルのコードを大きく変えなくてよい。

トレードオフと限界
- GPU と CPU(さらに NVMe)の間の転送が増え、学習スループットが落ちる。PCIe や NVMe の帯域がボトルネックになる。
- CPU 側で optimizer step を実行するぶん、CPU の計算力と RAM 容量も必要になる。CPU が貧弱だとここが新たなボトルネックになる。NVMe オフロードでは SSD の帯域・寿命(書き込み回数)も考慮が要る。
- 設定・チューニング(何をどこに逃がすか、転送と計算の重ね方、バッチサイズの選び方)が複雑になりがち。
- メモリ不足の主因が活性化側にある場合(高解像度入力や長い系列など)には直接効かない。その場合は活性化の再計算など別の手法が適切。何がメモリを食っているかを先に切り分ける必要がある。
- 遅延更新は収束に微妙な影響を与えうるため、収束が敏感なタスクでは検証が要る。
- 未解決の課題として、転送帯域が依然として根本のボトルネックであり、PCIe や NVMe の速度がモデル成長に追いつかない限り、オフロードで隠せる範囲には限界があります。圧縮転送や、より賢いスケジューリングの研究が続いています。

## 発展トピック・研究の最前線
- 量子化オフロード。逃がすデータ自体を低ビット(8 ビットなど)に圧縮して転送量を減らす研究があり、転送ボトルネックの緩和を狙います。8-bit optimizer(bitsandbytes)はそもそも最適化器状態を小さくする補完的な手法です。
- 活性化のオフロード。最適化器状態だけでなく、長系列の活性化を CPU に逃がしつつ再計算と組み合わせる手法も登場しています([Activation Recomputation (selective gradient checkpointing)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0155_activation-recomputation) と併用)。
- 推論時のオフロード。学習だけでなく、巨大モデルの推論を限られた GPU で動かすため、レイヤ単位で重みを GPU に流し込む手法(FlexGen のようなストリーミング)も実用化されています。
- オフロードと、シャーディング(ZeRO/FSDP)、テンソル並列、パイプライン並列をどう組み合わせるかという「並列化戦略の自動最適化」(Alpa など)は、大規模学習の中心的な研究テーマであり続けています。
