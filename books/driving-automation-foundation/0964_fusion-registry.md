# マルチビュー融合のレジストリ機構 (Fusion Registry)

## ひとことで言うと
複数のカメラ画像から取り出した特徴を「どうやって1つにまとめるか」という方式を、名前(文字列キー)で登録しておく辞書(レジストリ)です。モデル本体のコードを書き換えずに、設定値を変えるだけで融合方式を差し替えられるようにする設計手法です。研究コードでは「どの融合方式が一番良いか」を実験で比べる場面が多いため、こうした切り替え機構が広く使われています。手法そのものではなく、手法を比較・再現可能にするための「足場」だと理解してください。

## 直感的な理解
自動運転の車には複数のカメラが付いています。前方・側方・後方をそれぞれ撮るので、ある瞬間に得られる画像は1枚ではなく、6枚や7枚といった複数枚になります。ニューラルネットワーク(脳の神経細胞のつながりを真似た計算モデル)で運転判断をするには、このバラバラの情報を最終的に1つの「車の周りはこうなっている」という統合表現にまとめる必要があります。この「まとめ方」を、この分野では「ビュー融合 (view fusion)」または「マルチビュー融合 (multi-view fusion)」と呼びます。ビュー(view)はカメラ1台ぶんの視点(視界)という意味です。

問題は、ビュー融合のやり方が1通りではないことです。研究では何通りもの方式が提案されてきました。たとえば、

- 各カメラの特徴ベクトルを単純に横につなげる(concatenation, 連結)
- 各ビューをクロスアテンションで互いに参照させてまとめる
- いったん「真上から見下ろした地図 (Bird's-Eye-View, BEV)」の座標系に投影してまとめる

どれが良いかは、データセットと指標を決めて実験してみないと分かりません。そこで、方式を切り替えるたびにモデルの本体コードを書き換えていたら、ミスも増えるし、過去の実験設定も再現しにくくなります。「あの論文の Table 3 の B 行はどの融合方式だったか」を後から追えなくなるのは、研究では致命的です。再現性(同じ条件で同じ結果が出ること)は科学的主張の根拠そのものだからです。

そこで使われるのが「レジストリパターン」という設計手法です。レジストリ(registry)は「登録簿」という意味で、ここではプログラミング言語の辞書(キーと値のペアを持つデータ構造)に「方式の名前 → その方式を実装したクラス」を登録しておきます。利用側は名前の文字列(例: `"bev"`)を渡すだけで、対応する実装が組み立てられます。実験を回す人は「どの方式を使うか」だけ宣言すればよく、「その方式がどう作られるか」を知らなくて済みます。これが「関心の分離 (separation of concerns)」です。

## 基礎: 前提となる概念
レジストリを理解するには、いくつかの基本概念を押さえておく必要があります。

クラスとインスタンス。オブジェクト指向プログラミングでは、「クラス」は設計図、「インスタンス」はその設計図から作られた実体です。たとえば「自動車」というクラスから「この赤い車」というインスタンスを作る、という関係です。レジストリでは値としてインスタンスではなくクラス(設計図)そのものを登録しておくのがポイントです。なぜなら、組み立てるタイミングで初めて引数(カメラ台数や特徴の次元など)を渡したいからです。設計図を持っておけば、実体化は後回しにできます。これを「遅延生成 (lazy instantiation)」と呼びます。

ファクトリ関数。「組み立て係」のことです。名前を受け取り、対応するクラスを取り出して引数を渡し、インスタンスを作って返します。利用側は「どのクラスをどう組み立てるか」を知らなくてよくなり、名前だけ知っていれば済みます。これを「生成の詳細を隠蔽する」と言います。デザインパターンの古典(Gamma らの GoF 本, 1994)で言う Factory Method / Abstract Factory に相当します。

特徴マップ (feature map)。画像をニューラルネットワークの前段(バックボーン、backbone と呼ばれる画像から特徴を取り出す部分)に通すと、出てくるのは画素そのものではなく「特徴マップ」です。これは縦 H・横 W の格子の各点に C 個の数値(チャネル, channel)が並んだ、形 `[C, H, W]` のテンソルです。テンソル(tensor)は多次元の数の配列のことです。たとえば 256 チャネル・解像度 16x16 の特徴マップは `[256, 16, 16]` という形を持ちます。複数のカメラ・複数の場面を同時に扱うと、先頭にバッチ次元 B(同時処理する場面数)とビュー次元 V(カメラ台数)が付いて `[B, V, C, H, W]` のようになります。融合とは、この V 個ぶんの特徴を 1 つにまとめる操作のことです。

## 仕組みを詳しく
典型的なレジストリの実装は驚くほど小さいものです。中核は次の2つだけです。

```python
FUSION_REGISTRY = {
    "bev":        BEVViewFusion,
    "concat":     ConcatFusion,         # 例: 連結方式
    "cross_attn": CrossAttentionFusion, # 例: クロスアテンション方式
}

def build_view_fusion(fusion_mode, num_views, embed_dim=256, **kwargs):
    if fusion_mode not in FUSION_REGISTRY:
        raise ValueError(
            f"Unknown fusion_mode '{fusion_mode}'. "
            f"Available: {list(FUSION_REGISTRY.keys())}"
        )
    return FUSION_REGISTRY[fusion_mode](
        num_views=num_views, embed_dim=embed_dim, **kwargs
    )
```

ステップで分解すると、

1. `FUSION_REGISTRY` は辞書です。キーは文字列(`"bev"` など)、値はクラス(設計図)そのものです。インスタンスではなくクラスを入れている点が肝心で、あとから引数を渡して組み立てられるようにするためです。

2. `build_view_fusion(...)` がファクトリ関数です。`fusion_mode` という文字列を受け取り、それが辞書のキーに存在するかをチェックします。

3. 存在しなければ `ValueError`(値が不正というエラー)を投げ、メッセージで「使えるキーの一覧」を表示します。タイプミスをしてもすぐ気づけます。`build_view_fusion("bevv", ...)` のような打ち間違いが、深いところで分かりにくいエラーになるのを防ぎます。これを「フェイルファスト (fail fast)」と言い、不正な状態を最も早い段階で検出して原因究明を容易にします。

4. 存在すれば `FUSION_REGISTRY[fusion_mode]` でクラスを取り出し、`(num_views=..., embed_dim=..., **kwargs)` を渡して実体化します。`**kwargs` は「残りのキーワード引数をそのまま転送する」という意味で、ある方式に特有の設定(BEV なら `bev_h`, `bev_w` といったグリッド解像度)を、ファクトリが個別に知らなくても素通しできます。

具体例で言うと、`build_view_fusion("bev", num_views=7, embed_dim=256, bev_h=200, bev_w=200)` を呼ぶと、7カメラぶんの特徴を 200x200 の BEV グリッドにまとめる融合モジュールが返ってきます。`build_view_fusion("nonexistent", ...)` を呼ぶと、辞書にないので即エラーになり、何が使えるかが一覧表示されます。

より洗練された実装では、デコレータ(関数やクラスに付ける `@register("名前")` のような印)を使って、各方式のクラス定義の直上に登録処理を書く形を取ります。こうすると、新しいファイルを追加してインポートするだけで自動的にレジストリへ登録され、中央の辞書を手で編集する必要がなくなります。これを「自己登録 (self-registration)」と呼びます。PyTorch のモデルライブラリ(`timm` など)や物体検出フレームワーク(MMDetection など)はこの方式を多用しています。自己登録には1つ注意があり、登録はそのクラスを定義したモジュールが「インポートされた時点」で実行されるため、一度も import されないファイルは登録されません。フレームワークはこの問題を避けるため、起動時に対象ディレクトリを走査して全モジュールを強制 import する仕掛けを持つことが多いです。

レジストリと密接に関係するのが「設定ファイル駆動 (config-driven)」の実験運用です。融合方式・バックボーン・損失関数といった構成要素をすべて文字列キーで指定できるようにしておき、YAML や JSON の設定ファイルに `fusion_mode: bev` のように書く。すると、実験ごとに設定ファイルを1つ残しておけば、後からその設定ファイルを読むだけで「どの方式で回した実験か」が完全に再現できます。論文の Table の各行を設定ファイルに対応づけられるのが、研究運用上の大きな価値です。設定ファイルをバージョン管理(git など)に入れておけば、実験の系譜を差分として追えます。

ASCII で関係を図示すると次のようになります。

```
   設定ファイル                ファクトリ                  レジストリ(辞書)
  fusion_mode: bev  ──→  build_view_fusion("bev")  ──→  {"bev": BEVViewFusion,
  bev_h: 200                       │                      "concat": ConcatFusion, ...}
  bev_w: 200                       ▼                              │
                          BEVViewFusion(num_views=7,  ◄───────────┘ クラスを取り出す
                                        embed_dim=256,
                                        bev_h=200, bev_w=200)
                                       │
                                       ▼
                              融合モジュールの実体
```

## 手法の系譜と主要論文
レジストリという仕組み自体はソフトウェア工学の定番パターンであり特定の論文の手法ではありませんが、「複数のビュー融合方式を比較する」という発想は数多くのマルチカメラ知覚論文に通底しています。なぜ切り替え機構が要るのかは、融合方式の系譜を追うと腑に落ちます。

連結 (concatenation)。最も古典的なベースラインです。各カメラの特徴ベクトルを単に並べて1本の長いベクトルにし、後段の全結合層に渡します。実装は最も簡単(ベクトルを並べるだけ)ですが、カメラ間の幾何的対応(同じ物体が複数カメラの重なる視野に写る関係)を陽に扱えないため、空間的な一貫性の学習はすべてネットワークに丸投げになります。カメラ台数が変わると入力次元が変わってしまう弱点もあります。

LSS (Lift, Splat, Shoot; Philion & Fidler, ECCV 2020, arXiv:2008.05711)。各カメラのピクセルごとに「奥行きの確率分布」を予測し(Lift)、その確率に従って特徴を3D空間に点群として撒き(Splat)、BEV グリッドに集約します(Shoot)。深度を明示的に推定するのが特徴です。効果は複数カメラを幾何的に正しく BEV へ統合できること、トレードオフは深度推定の誤差がそのまま BEV 上の位置ずれになることです。

BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)。学習可能な BEV クエリが各カメラ画像に空間クロスアテンションで問い合わせる方式で、深度の明示推定を避けます。LSS の「深度を当てにいく」路線とは対照的に、「BEV の各マスがカメラに問い合わせる」路線を確立しました。詳細は [BEV (鳥瞰図) によるマルチビュー融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0933_bev-view-fusion) を参照してください。

これらの比較を見ると、「方式を差し替えて同じ条件で測りたい」という需要が確かに存在し、レジストリのような切り替え機構が研究コードで重宝される理由が分かります。実際 LSS と BEVFormer は同じ nuScenes データセット上で比較され、後発手法が前任を上回ることで進歩が示されてきました。公平な比較のためには、融合方式だけを入れ替えて他の条件(バックボーン・学習率・データ拡張など)を固定できる仕組みが不可欠なのです。これを「制御変数法 (controlled comparison)」といい、レジストリはそれを技術的に支える道具です。

## 論文の実験結果(定量データ)
レジストリ自体には実験結果はありませんが、レジストリが切り替え対象とする融合方式の優劣は定量的に測られています。指標の意味から説明します。

nuScenes は自動運転の標準ベンチマークの1つで、6台のカメラ・LiDAR・レーダーを備えた車で都市部を走行した大規模データセットです(Caesar et al., CVPR 2020, arXiv:1903.11027)。1000シーン・各20秒・約140万枚のカメラ画像を含みます。3D 物体検出の主要指標が mAP(mean Average Precision、平均適合率の平均)と NDS(nuScenes Detection Score、検出の総合スコア)です。mAP は「検出したもののうち正しかった割合(適合率)と、見逃さなかった割合(再現率)のバランス」を表し、高いほど良い(最大1.0)。NDS は mAP に加えて位置・大きさ・向き・速度・属性の誤差も総合した nuScenes 独自の指標で、これも高いほど良い指標です。NDS が約0.5を超えると当時は最高水準とされました。

報告されている代表値として、BEVFormer は nuScenes 検出で NDS およそ 0.517(base 構成、バックボーン R101-DCN)を達成し、当時のカメラのみ手法の最高水準を更新しました。mAP はおよそ 0.416 です。LSS 系の後続(BEVDet など)はおおむね NDS 0.4 前後の水準から出発しており、BEVFormer の時間情報を使う設計が大きく押し上げたと報告されています。0.4 から 0.5 への 0.1 という差は、検出漏れ・誤検出が目に見えて減り、特に向きや速度の推定が安定したことを意味します。これらは「同じデータセット・同じ指標で、融合方式を変えると最終性能がどれだけ動くか」を示す具体例です。

アブレーション(ある要素を抜いて効果を測る実験)の観点では、BEVFormer の論文は「時間方向の自己アテンション(過去の BEV を参照する仕組み)を外すと、動いている物体の速度推定誤差(mAVE)が悪化し NDS が数ポイント下がる」と報告しています。融合方式そのものを連結に置き換えると幾何的整合が失われ、特に視野が重なる領域での検出が劣化します。こうした結果は、融合方式の選択がエンドツーエンドの性能に確かに効くことを示しており、だからこそレジストリで切り替えて測る価値があるわけです。レジストリがなければ、これらのアブレーションは「方式ごとに別々のコードを手で書き換える」ことになり、変更が混入して比較の公平性が崩れるリスクがあります。

## メリット・トレードオフ・限界
メリット。融合方式の差し替えが文字列1つで済み、学習スクリプトなど利用側コードを改変しなくてよい。設定ファイルだけで実験を切り替えられるので、過去実験の再現性が上がる。未知のキーを渡すと「使えるキー一覧つき」のエラーが即出るので、タイプミスや設定ミスの発見が早い。新方式の追加が辞書への登録1つで完結し、変更範囲が局所化されるためレビューしやすい。複数人で開発するときも、ある人が新方式を足しても既存方式のコードに触れずに済むため、衝突や副作用が起きにくい。

トレードオフと限界。登録されている方式が1つしかない状態では、「切り替えられる」というメリットは活きず、枠だけある状態になる(過剰な抽象化は「使われない一般性」というコストになる)。また、ファクトリが `**kwargs` を素通しするだけで引数の事前検証をしない実装だと、ある方式が受け取らない引数を渡したときに、エラーが出るのはレジストリではなく各クラスのコンストラクタの中になり、エラー箇所が一段奥になって原因究明しにくい。レジストリは型チェックを弱める(文字列で間接的に指定するため、エディタの補完や静的解析が効きにくい)という代償もあります。文字列キーの打ち間違いはコンパイル時には捕まらず実行時まで分からないのが本質的な弱点です。研究の最前線では、Hydra(Yadan, Meta 製の設定管理フレームワーク, 2019)のように、設定ファイルとレジストリを統合し、型情報も保ちながら構成を組み立てる仕組みが好まれる傾向があります。

## 発展トピック・研究の最前線
レジストリの考え方は、融合方式に限らずモデルのあらゆる構成要素(バックボーン、ネック、ヘッド、損失、最適化器、データ拡張)に拡張できます。MMDetection / MMDetection3D(OpenMMLab のフレームワーク, Chen et al., 2019)は、ほぼすべての構成要素をレジストリで管理し、巨大な設定ファイル1つでモデル全体を組み上げる設計の代表例です。この設計は、数百種類のモデルを統一的に扱い、論文間の公平な比較を可能にする基盤になっています。Detectron2(Meta)や Hugging Face Transformers の `AutoModel` も、文字列キーから実装を解決するという同じ思想に立っています。

近年は、レジストリと自動探索を組み合わせる流れもあります。融合方式やハイパーパラメータをキーで指定できるようにしておけば、ニューラルアーキテクチャ探索 (NAS, Neural Architecture Search) や自動ハイパーパラメータ最適化 (Optuna, Ray Tune など) が、その文字列キーの空間を自動的に探索できます。人手で1つずつ試す代わりに、機械が「どの融合方式・どの解像度が最良か」を探す、という研究運用につながります。さらに進んで、設定空間そのものをコードとして表現する Hydra の構造化コンフィグや、実験管理ツール(Weights & Biases, MLflow)のスイープ機能と組み合わせると、「レジストリのキー × ハイパーパラメータ」の格子を自動で網羅探索し、結果を一覧表として残す、という現代的な研究パイプラインが組めます。
