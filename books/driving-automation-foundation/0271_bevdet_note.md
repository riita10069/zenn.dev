# BEVDet

## ひとことで言うと
BEVDet は、車両周囲を覆う複数のカメラ画像から鳥瞰図(BEV、上空から見下ろした地図のような表現)を作り、その上で3次元の物体検出を行う実用的なパイプラインです。Lift-Splat-Shoot(LSS)という深度ベースの視点変換でカメラ画像を BEV に変換し、そこへ検出器を載せるという部品の組み合わせを明快に整理し、画像空間と BEV 空間でデータ拡張(学習データの水増し)を分離する工夫で過学習を抑えた点が特徴です。カメラのみによる BEV 3D検出を「実際に使える形」にまとめ上げた基盤的な仕事です。

## 直感的な理解
自動運転車には前後左右をカバーする複数のカメラ(典型的には6台)が付いています。これらをバラバラに処理するのではなく、すべてを共通の鳥瞰図に統合できれば、「自車の周囲360度のどこに、何が、どんな向きでいるか」を1枚の地図として扱えます。鳥瞰図はメートル単位の正確な位置を表せるので、経路計画や衝突回避にそのまま使えます。画像座標のままだと「画面の右上に小さく写る車」が実空間でどこにいるか直感的に分からないのに対し、BEV なら「自車の右前方 20m」と即座に読めるのです。

LSS は各カメラ画像を深度分布で3次元へ打ち上げ BEV に集約する基礎技術を提供しましたが([カテゴリ深度分布 (depth bins)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0276_categorical-depth-distribution) 参照)、もともと BEV セグメンテーション(地図を意味ごとに塗り分けるタスク)向けで、3次元物体検出(車などをボックスで囲い位置・サイズ・向きを出す)へそのまま使うには実用上の隙間がありました。検出ヘッドの載せ方が整理されていない、学習が過学習しやすい、画像と BEV の2つの空間でデータ拡張をどう一貫させるかが難しい、といった点です。BEVDet はこれらを埋め、再現性のある強いベースラインを示しました。後続のカメラ BEV 検出研究の多くが BEVDet を出発点・比較対象にしたという意味で、この分野の「基準点」になった仕事です。

## 基礎: 前提となる概念
BEV(Bird's-Eye View)は、世界を上空から見下ろした2次元グリッドで表す表現です。各セルが実空間の小さな矩形(例えば 0.5m × 0.5m)に対応し、そこに特徴ベクトルが入ります。深さ方向の曖昧さが消え、距離が直接読めるので計画に向きます。

視点変換(view transformation)は、カメラの透視投影で得た画像(遠近がついた像)を、距離が正しく読める BEV へ移す操作です。深度ベースの視点変換では、各ピクセルの深度を推定して3次元へ戻し、それを上から見下ろす形に並べ直します。これに対し Transformer 系は逆向き、すなわち BEV の各セルから画像へ問い合わせる方式を取りますが、BEVDet は前者(順方向の打ち上げ)です。

検出ヘッド(detection head)は、特徴マップから物体のボックスを出す出力部です。BEV では、各セルに物体中心がある確率(ヒートマップ)と、そこからの中心オフセット・サイズ・向き・速度などを回帰する CenterPoint 系のヘッド(Yin et al., CVPR 2021)がよく使われます。CenterPoint は LiDAR 検出で確立された手法で、アンカー(候補箱のテンプレート)を使わず中心点で物体を表すため、向きが自由な3D箱に向いています。

過学習(overfitting)は、訓練データに合わせすぎて未知データで性能が落ちる現象です。データ拡張(回転・反転・拡大縮小などで学習データを人工的に増やす技術)は、その抑制に使う基本手段です。

## 仕組みを詳しく
BEVDet は4つのモジュールを直列につなぎます。

```
複数カメラ画像(例:6枚)
   │
   ▼ ① Image-view Encoder(画像エンコーダ)
画像特徴 [N枚, C, H, W]
   │
   ▼ ② View Transformer(LSSによる視点変換)
BEV特徴 [B, C, X, Y]
   │
   ▼ ③ BEV Encoder(BEVエンコーダ)
洗練されたBEV特徴
   │
   ▼ ④ Detection Head(検出ヘッド)
3D物体box(位置・サイズ・向き・速度)
```

1. Image-view Encoder。各カメラ画像を CNN バックボーン(ResNet や Swin Transformer など)に通して画像特徴を抽出し、FPN(Feature Pyramid Network、異なる解像度の特徴を混ぜる仕組み)で多スケール情報をまとめます。N 枚のカメラは重みを共有する同じエンコーダで処理されます。

2. View Transformer(LSS)。各ピクセルで深度分布 [N, D, H, W] を予測し、画像特徴 [N, C, H, W] と外積を取って錐台特徴を作り、3次元へ打ち上げて BEV グリッドへ振りかけ合計します(splat、同じセルに落ちた点を加算する pooling)。出力は BEV 特徴 [B, C, X, Y](X, Y は BEV 平面のマス目、例えば 128×128)です。ここで N 枚のカメラ画像が1つの上から見た特徴地図に統合されます。

3. BEV Encoder。BEV 特徴をさらに畳み込みネットワークで処理し、物体の形や配置といった BEV 空間内のパターンを洗練させます。視点変換で生じた歪みや、カメラ境界での不連続を均す役割も担います。

4. Detection Head。洗練された BEV 特徴の上で物体検出を行い、各物体の中心位置・サイズ・向き(ヨー角)・カテゴリ・速度などを出力します。

BEVDet の最大の実用的貢献が、画像空間と BEV 空間でデータ拡張を分離する設計です。なぜ分離が要るのか。画像を拡大・回転すると見かけの物体サイズや位置が変わり、深度推定にズレが生じます。素朴に画像だけを拡張すると BEV も連動して歪み、ラベルとの対応が壊れます。そこで BEVDet は、画像空間の拡張(リサイズ・クロップ・反転・回転)を行ったら、その幾何変換をカメラの内部・外部パラメータ行列へ反映させ、視点変換が正しい BEV を返すようにします。さらに BEV 空間では独立に別の拡張(回転・スケール・反転)を追加できるようにしました。BEV 拡張は3D ボックスのラベルにも同じ変換を適用するので、特徴とラベルの整合が保たれます。

この BEV 空間の追加拡張が決定的でした。LSS の視点変換は深度推定にデータ量を要するため、画像拡張だけだと BEV エンコーダと検出ヘッドが訓練データの配置を丸暗記して激しく過学習します。BEV 空間で独立に拡張を効かせると、後段が多様な配置に晒され、過学習が大きく和らぎます。

数値で重さの感覚をつかみます。N=6, C=64, D=59, H=16, W=44 の錐台特徴は 6×64×59×16×44 ≈ 1.6 億要素で、これが「LSS の打ち上げ計算が重い」と言われる理由です。BEVDet はバックボーンと入力解像度を上げると素直に精度が伸びるスケーラブルな設計で、小さい設定(ResNet-50、低解像度)では高速、大きい設定(Swin-Base 等の大バックボーン、高解像度)では高精度と用途に応じて調整できます。

## 手法の系譜と主要論文
LSS (Philion & Fidler, ECCV 2020, arXiv:2008.05711) が深度分布による微分可能な視点変換を確立し、BEVDet の土台になりました。

CenterPoint (Yin et al., CVPR 2021, arXiv:2006.11275) が、中心点ベースの3D検出ヘッドを確立しました。BEVDet はこのヘッドを BEV 特徴の上に載せています。

BEVDet (Huang, Huang, Zhu, Du, 2021, arXiv:2112.11790)「BEVDet: High-Performance Multi-Camera 3D Object Detection in Bird-Eye-View」が、画像エンコーダ → LSS視点変換 → BEVエンコーダ → 検出ヘッドの4モジュール構成を整理し、画像空間と BEV 空間で分離したデータ拡張戦略を提案しました。動機は、LSS の BEV 表現を3D検出へ直結させ、過学習を抑える学習法を確立することでした。

BEVDet4D (Huang & Huang, 2022, arXiv:2203.17054) が前フレームの BEV を自車運動で位置合わせして融合し、速度推定と時間一貫性を加えました([BEVDet4D](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0272_bevdet4d))。BEVDepth (Li et al., AAAI 2023, arXiv:2206.10092) が深度を LiDAR で明示監督して深度品質を底上げしました([BEVDepth](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0270_bevdepth))。これらは BEVDet の弱点(時間情報なし・深度の暗黙学習)をそれぞれ補う直系の後継です。

## 論文の実験結果(定量データ)
nuScenes(車両周囲360度を6カメラ+LiDAR 等で記録した大規模データセット、約 1000 シーン)で評価します。代表指標は NDS(nuScenes Detection Score)と mAP(mean Average Precision)です。mAP は中心位置の距離(BEV 平面上の 2D 距離、0.5/1/2/4m の4閾値で平均)で正解判定する平均適合率、NDS は mAP に加えて並進誤差 mATE・スケール誤差 mASE・向き誤差 mAOE・速度誤差 mAVE・属性誤差 mAAE をまとめた総合点で、いずれも高いほど良い指標です。NDS は mAP の比重を半分、残りを5つの誤差から作るため、検出するだけでなく「正確に・正しい向きで・正しい速度で」当てたかを総合評価します。

BEVDet は、カメラのみ構成で当時の高い精度を達成しました。報告によれば、Swin 系の大きめバックボーン+高解像度の設定で validation の NDS が約 0.40 前後、mAP が約 0.33 前後に達し、当時の同条件のカメラのみ手法で先頭グループに入りました(数値は設定と報告に依存)。小さい設定(ResNet-50、低解像度)では高速・軽量(リアルタイムに近い推論速度)、大きい設定では高精度と、速度・精度のスケーラビリティが定量的に示されました。特筆すべきは、当時の FCOS3D 等の画像空間で完結する単眼検出系に対し、BEV 空間で検出する優位を実証した点です。

アブレーションで本質が見えます。BEVDet の研究では、BEV 空間の追加データ拡張を外すと検証性能が大きく崩れ、激しく過学習することが定量的に示されました(訓練精度は高いまま検証 NDS が大幅に落ちる)。画像空間の拡張だけでは不十分で、BEV 空間の独立拡張が後段の過学習抑制に効く、というのが核心の知見です。逆に拡張を入れると、訓練・検証のギャップが縮まり、より大きなモデル・高解像度でも安定して学習できるようになりました。さらに入力解像度を上げる、バックボーンを強くする、BEV グリッドを細かくする、といった各因子がそれぞれ単調に精度を押し上げることが示され、設計のスケーラビリティが裏づけられました。

## メリット・トレードオフ・限界
メリットは、画像エンコーダ・視点変換・BEVエンコーダ・検出ヘッドという明快なモジュール構成で部品の差し替えがしやすいこと(各モジュールを独立に研究・改良できる)、データ拡張の分離で過学習を抑え安定して学習できること、バックボーンや解像度を変えるだけで速度と精度を調整できスケーラブルなこと、カメラのみで動くため LiDAR を持たない車両でも使えることです。

トレードオフと限界は次のとおりです。単一フレームのみだと物体の速度を測れず、動く物体に弱い(時間軸を加えた [BEVDet4D](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0272_bevdet4d) で改善)。LSS の深度が暗黙学習中心で深度精度に限界がある(LiDAR 監督の [BEVDepth](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0270_bevdepth) で改善)。LSS の打ち上げ計算が重く、錐台特徴のメモリと計算量が大きい(高速な Voxel Pooling 実装が後に必要になった)。さらに、深度分布のビン設計や BEV グリッド解像度がメモリと精度を左右し、チューニングが要ります。BEV 空間 splat では、同じセルへ複数カメラ・複数深度の特徴が加算されるため、遠方の疎なピクセルが希釈されやすく、遠方・小物体が苦手という構造的な弱点もあります。

未解決の課題としては、巨大な錐台テンソルを避けつつ深度の明示性を保つ効率化、遠方・小物体の検出精度、そして昼夜・天候など分布外条件への頑健性が挙げられます。

## 発展トピック・研究の最前線
BEVDet 系の弱点それぞれが後続研究の主題になりました。時間融合([BEVDet4D](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0272_bevdet4d))、明示的深度監督([BEVDepth](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0270_bevdepth))、時系列ステレオによる深度精緻化([BEVStereo](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0274_bevstereo))がその系譜です。並走する別系統として、深度分布を陽に持たず学習可能クエリで3次元から画像へ逆サンプリングする Transformer 系(BEVFormer、PETR など)があり、錐台テンソルの巨大化を避けられる一方で幾何の明示性が下がるという別のトレードオフを持ちます。実装の効率化(高速 Voxel Pooling、CUDA カーネル最適化、深度を陽に展開しない加速)も、車載でのリアルタイム動作に向けた重要な研究方向です。さらに、検出に留まらず BEV 表現を地図構築(HD マップのオンライン推定)や占有予測(occupancy)、end-to-end の運転計画へ拡張する流れが進み、BEVDet が示した「画像→BEV→タスクヘッド」のモジュール構成は、カメラベース知覚の標準的な骨格として広く受け継がれています。
