# backbone処理のための B*V reshape

## ひとことで言うと

自動運転車やロボットは、周囲を一周見渡すために複数台のカメラを積んでいます。たとえば6台や7台です。これら複数枚の画像を1枚ずつ画像認識ネットワークに通すのではなく、「バッチ（一度にまとめて処理する枚数）」と「カメラ枚数」を一時的に1つの軸へ押し込み、まとめて1回で処理するためのテンソル整形のテクニックです。処理が終わったら元の「バッチ × カメラ」の形へ戻します。これはマルチカメラ・マルチフレームを扱うほぼ全ての深層学習モデルに共通する、地味だが避けて通れない定石です。

## 直感的な理解

「8人ぶんの集合写真を、7台のカメラで同時に撮った」状況を想像してください。あなたが画像認識の係で、合計 56 枚の画像を処理しなければなりません。ここで2つのやり方があります。

1. 1枚ずつ机に置いて、56回「これは何が写っているか」を判定する。
2. 56枚を1つのトレーに積んで、判定機に一気に流し込む。

GPU（画像処理に強い計算装置）は、後者のように「大量のデータを一度に流す」ほど効率が上がる設計になっています。1枚ずつ呼び出すと、呼び出すたびに固定の準備コスト（カーネル起動オーバーヘッド、後述）がかかり、しかも並列計算の能力が遊んでしまいます。だから「カメラ枚数ぶんをバッチに畳み込んで、一気に backbone へ流す」のが定石になります。

なぜわざわざ整形が要るのか。それは画像認識ネットワークが受け取れるテンソルの「形」が決まっているからです。カメラ入力は5次元（バッチ・カメラ・色・縦・横）ですが、ネットワークは4次元（枚数・色・縦・横）しか受け取れません。次元の数を合わせるために、5次元を4次元へ畳む必要があるのです。

## 基礎: 前提となる概念

いくつかの用語を、知らない人向けに噛み砕きます。

- backbone（バックボーン）。これは「画像から特徴を取り出す、事前学習済みの画像認識ネットワーク」という意味です。特徴（feature）とは「画像のどこに何があるか」を表す数値の集まりのことです。代表例は CNN 系の ResNet、Transformer 系の Swin Transformer や ViT などです。多くの場合、ImageNet という巨大な画像データセットで先に学習させた重みを流用します（事前学習、pre-training）。

- テンソル（tensor）。これは「多次元の数値の箱」という意味です。1次元なら数列、2次元なら表（行列）、3次元なら表を何枚も重ねた立体、というように軸の数が増えます。カラー画像1枚は「色チャネル × 縦 × 横」の3次元、それを何枚も束ねると4次元、さらにカメラの軸を足すと5次元になります。

- バッチ（batch、記号 B）。これは「同時にまとめて処理するサンプル数」という意味です。GPU は1サンプルずつ処理するより、たとえば8サンプルをまとめて処理した方が速くなります。このまとめた数が B です。

- カーネル起動オーバーヘッド（kernel launch overhead）。GPU で1回の計算（カーネル）を起動するたびに発生する、データ量に関係しない固定の準備時間のことです。小さな計算を何度も呼ぶと、この固定費が積み重なって無視できなくなります。

マルチカメラ入力のテンソルは `[B, V, C, H, W]` という5次元になります。各記号の意味は次のとおりです。

- B = バッチサイズ（同時に処理するシーンの数。例: 8）
- V = view（ビュー、カメラ台数。例: 6 や 7）
- C = チャネル数（カラー画像なので 3 = 赤・緑・青）
- H = 画像の高さ（ピクセル数。例: 256）
- W = 画像の幅（ピクセル数。例: 256）

ところが画像認識の backbone は、画像処理ライブラリ（PyTorch など）の慣例に従い「4次元テンソル `[N, C, H, W]`」しか受け取れません（N は処理する画像の枚数）。5次元をそのまま渡すと、次元が合わずにエラーになります。

素朴な解決策は「forループで V 回まわし、1台ずつ backbone に通す」ことです。しかしこれには2つの問題があります。第一に遅い。GPU の並列性を活かせず、カーネル起動オーバーヘッドが V 倍になります。第二にコードが煩雑。ループの中で結果をリストに溜め、あとで結合する処理が必要になります。そこで「B と V を1つの軸にまとめて `[B*V, C, H, W]` という4次元にしてから1回だけ backbone に通す」という定石が使われます。

## 仕組みを詳しく

典型的な実装では、forward 処理の冒頭で次のように書きます。

```python
B, V, C, H, W = camera_tiles.shape       # 例: 8, 7, 3, 256, 256
x = camera_tiles.reshape(B * V, C, H, W) # [8,7,3,256,256] → [56,3,256,256]
features = backbone(x)                   # 56枚をまとめて1回で処理
```

ステップごとに追います（B=8, V=7 の例）。

1. 入力 `camera_tiles` は `[8, 7, 3, 256, 256]`。総要素数は 8×7×3×256×256 個です。
2. `reshape(B*V, C, H, W)` で先頭2軸 (8, 7) を1軸 (56) に潰します。形は `[56, 3, 256, 256]`。中身の数値は1つも変わらず、並び順の解釈だけが変わります（reshape はメモリ上のデータを動かさず、形の見え方を変えるだけの操作だからです）。
3. backbone は「56枚のバラバラな画像」だと思って処理します。出力は特徴マップ（feature map、画像を縮小して情報を凝縮した格子状の数値）で、たとえば `[56, 96, 64, 64]` のような形になります。

ここで決定的に重要なのが「reshape は要素の順序を保つ」という性質です。`[B, V, ...]` を `[B*V, ...]` に潰すと、新しい先頭軸のインデックスは次の規則で並びます。

```
index = b * V + v
```

つまり具体的には次のように並びます。

```
行 0 : (バッチ0, カメラ0)
行 1 : (バッチ0, カメラ1)
 ...
行 6 : (バッチ0, カメラ6)
行 7 : (バッチ1, カメラ0)   ← ここでバッチが切り替わる
 ...
行55 : (バッチ7, カメラ6)
```

backbone を通したあとは、後段の融合処理に「どの行がどのバッチ・どのカメラか」を伝える必要があります。実務上は B と V を引数として明示的に持ち回り、融合モジュール側で `reshape(B, V, C, H, W)` と書いて view（カメラ軸）を復元します。復元の式は潰すときと逆で、行 `b*V+v` がちょうど `[b, v]` の位置へ戻ります。順序が保たれているので、これだけで安全に元へ戻せます。

注意点（落とし穴）。`reshape` による畳み込みと復元は「B と V をこの順序で潰し、この順序で戻す」前提で成り立ちます。途中に `permute`（軸の入れ替え）を挟んでメモリ上の並びが変わっていると、単純な reshape が意図しない混ぜ方をすることがあります（PyTorch では非連続メモリに対する reshape が暗黙にコピーを作る、あるいは `view` がエラーになる）。そのため backbone のように「軸を入れ替えず、枚数次元だけを処理する」操作にこの定石を使うのが安全です。時間軸 T を扱う動画モデルでも同様に `[B*T, C, H, W]` と畳む手口が使われ、これは "temporal-as-batch" などと呼ばれます。

## 手法の系譜と主要論文

この操作は独立した「手法」というより、マルチビュー・マルチフレームを扱うときに必ず現れる実装上の定石であり、論文では1行の式や図でさらりと書かれることが多いものです。系譜としては「共有 backbone でカメラ枚数をまとめて処理する」という設計思想の流れに位置づけられます。

- Philion and Fidler, "Lift, Splat, Shoot"（ECCV 2020, arXiv:2008.05711）。N 台のカメラ画像をまとめて共有 backbone（EfficientNet）に通し、各カメラの特徴を取り出してから3D空間（BEV、bird's-eye view = 真上から見た鳥瞰図）へ投影します。論文の本質は投影の仕組みですが、その前段としてカメラ枚数をまとめて処理する構成が当然のように使われています。効果は計算共有による効率化、トレードオフは「全カメラが同じ backbone の重みを共有する」点です（カメラごとの個性は後段で補います）。

- Li et al., "BEVFormer"（ECCV 2022, arXiv:2203.17270）。複数カメラ画像を共有 backbone（ResNet-101 など）に通して多視点特徴を得てから、学習可能な BEV クエリで集約します。過去フレームも扱うため、実装上は「カメラ × 時刻」をまとめて backbone に流す形になります。

- Liu et al., "PETR"（ECCV 2022, arXiv:2203.05625）。マルチビュー画像に3D位置エンコーディングを付与してから Transformer デコーダで検出します。ここでも複数カメラを共有 backbone でまとめて処理する前提は共通です。

- フレームワークの慣習。`nn.Conv2d` をはじめ画像系の層は4次元 `[N, C, H, W]` を前提とするため、「余分な軸（V や T）はバッチに畳んでから処理し、あとで戻す」が標準イディオムです。論文よりも実装コードやチュートリアルで定着したテクニックです。

## 論文の実験結果(定量データ)

この reshape 自体を主役にしたアブレーション論文は存在しませんが、「共有 backbone でマルチカメラをまとめて処理する」設計が成立することは、上記の手法群が大規模ベンチマークで成果を出したことで間接的に裏づけられています。

代表的な指標は nuScenes データセット（6カメラのマルチビュー自動運転データセット）の NDS と mAP です。NDS（nuScenes Detection Score）は検出精度に加えて、位置・大きさ・向き・速度・属性の誤差を総合した0〜1の指標で、高いほど良い。mAP（mean Average Precision）は検出の平均精度で、こちらも高いほど良い。報告によれば BEVFormer は nuScenes test で NDS 約0.569、mAP 約0.481 を達成し、当時の単一フレーム手法を上回りました。LSS 系・PETR 系もいずれも6カメラを共有 backbone でまとめて処理する構成で、NDS 0.4〜0.5 台の性能を示しています。これらは「カメラごとに別 backbone を用意しなくても、共有 backbone で十分な性能が出る」ことを実証しています。

効率面では、forループ方式に対する一括処理の優位は GPU の特性そのものです。56枚を1回で流すのと7回に分けるのとでは、後者はカーネル起動オーバーヘッドが約7倍になり、さらに各回のバッチが小さくなって GPU の演算ユニットが遊びます。厳密な数値は backbone と画像サイズに依存しますが、小バッチを多数回呼ぶ構成は実測でしばしば数十パーセント遅くなることが知られています。

## メリット・トレードオフ・限界

メリット。GPU を効率よく使えます。複数台ぶんを1回の呼び出しでまとめて処理でき、カーネル起動オーバーヘッドが減ります。コードも単純で、forループや結果の連結が不要になり、reshape 2回で表現できます。さらに backbone の重みを全カメラで共有できるため、パラメータ数が増えません。

トレードオフと限界。第一に、瞬間的なメモリ使用量が増えます。`B*V` 枚を同時に GPU へ載せるため、VRAM 使用量は枚数に比例します。カメラが多く解像度が高いと OOM（out of memory、メモリ不足）になりやすくなります。第二に、全カメラが同じ backbone を通るため、カメラごとの違い（前方カメラと後方カメラの役割の差など）は backbone 自体では区別されません。これは後段の view fusion や視点位置エンベディングで補う必要があります。第三に、reshape の順序（`b*V+v`）を取り違えると、別バッチ・別カメラの特徴を混ぜてしまうバグになります。`permute` と組み合わせる場合は特に注意が必要です。

未解決の課題としては、カメラ台数が可変なデータセット（車種ごとにカメラ数が違う）への対応や、解像度の異なるカメラ（広角と望遠の混在）を同一 backbone でどう扱うかといった点があり、研究上はパディングやマスク、カメラごとの軽量アダプタなどで対処が試みられています。

## 発展トピック・研究の最前線

時間軸を含むモデルでは `[B, T, V, C, H, W]` と6次元になり、`[B*T*V, C, H, W]` へ畳む拡張がよく使われます。ただし時間方向のメモリ爆発を避けるため、過去フレームの特徴はキャッシュして再利用する設計（recurrent BEV や memory queue）が主流になりつつあります。また、カメラごとに backbone を完全共有するのではなく、共有 backbone に「カメラ ID 埋め込み」や軽量な per-camera adapter を足して個性を持たせるハイブリッドも研究されています。大規模化の文脈では、複数カメラを1枚の大きな画像にタイル状に貼り付けて単一 ViT に通す手法も登場しており、reshape の畳み方自体が設計上の自由度として扱われ始めています。
