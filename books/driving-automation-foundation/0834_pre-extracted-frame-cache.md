# フレーム事前抽出・キャッシュ

## ひとことで言うと

学習のたびにMP4などの動画から必要なフレームをその場でデコード（圧縮された動画を画像に展開すること）するのはとても遅いです。そこで、必要なフレームを学習の前に一度だけ取り出して画像ファイルと数値ファイルにまとめて保存しておき、学習時はそれを高速に読むだけにする最適化を「フレーム事前抽出・キャッシュ」と呼びます。「毎回作るのをやめて、作り置きする」発想です。

## 直感的な理解

ニューラルネットの学習は、データを一巡（1エポック）するだけでなく、同じデータを何十回も繰り返し見ます。もし毎回、動画ファイルから「3分12秒のあたりの、左カメラのこのフレーム」を取り出すたびに動画を展開していたら、同じ展開作業を何度も繰り返すことになります。これは、毎食ごとに小麦を挽いて粉から作り直すようなもので、明らかに無駄です。

しかも動画の展開は単に遅いだけでなく、「飛び飛びのフレームを取り出す」のがとくに苦手です。MP4 のような動画は、前のフレームとの差分だけを保存して圧縮しているため、欲しいフレームを得るには手前のキーフレーム（完全な画像が保存された基準フレーム）まで戻って順に展開する必要があるからです。そこで、学習を始める前にいったん全部展開して素の画像として保存しておけば、あとは画像を読むだけで済みます。これが事前抽出です。GPUは1ステップを数ミリ秒で回せるのに、データ供給が遅いとGPUが大半の時間を遊んで待つ「入力律速（input-bound）」状態になります。事前抽出はこれを解消するための定石です。

## 基礎: 前提となる概念

学習パイプラインは「データを読む側（CPU、ディスク、ネットワーク）」と「計算する側（GPU）」の二人三脚です。理想は両者が常に同時に働き続けることですが、データ供給が追いつかないとGPUが手待ちになります。GPU稼働率（utilization、GPUが実際に計算している時間の割合）が低いまま学習が遅い場合、たいていデータ供給がボトルネックです。

動画デコードのコストは無視できません。多視点（複数カメラ）の動画から、過去と未来にまたがる複数フレームを毎サンプル取り出す用途では、1サンプルの供給に数十秒かかることも珍しくありません。報告例では、多視点ログから素朴にフレームを取り出すと1サンプルあたり数十秒規模になり、GPUが99%以上遊ぶ事態が起こります。

ここで重要なのが「ランダムアクセス vs シーケンシャルアクセス」の違いです。たくさんの小さなファイルをバラバラに開く（ランダムアクセス）と、1ファイルごとにファイルシステムやネットワークのオーバーヘッドがかかり、ディスク・ネットワークの帯域を活かせません。逆に、データを大きな塊にまとめて先頭から順に読む（シーケンシャルアクセス）と、帯域をフルに使えます。とくにクラウドのオブジェクトストレージ（S3 など）は、小ファイルの大量リクエストに弱く、大きなファイルの順次読みに強いので、この差は決定的です。

## 仕組みを詳しく

事前抽出の典型的なやり方は、学習前に1回だけ必要なフレームをデコードして素の画像（JPEG など）として保存し、車両状態などの数値は `.npy`（NumPy配列のバイナリ）にして一緒に固めることです。保存形式として広く使われるのが WebDataset です。

WebDataset は、たくさんの小さなファイルを `.tar`（複数ファイルを1つに束ねたアーカイブ）にまとめ、それを先頭から順に読む方式です。ランダムに小ファイルを開く代わりにシーケンシャルに読むので、ディスク・ネットワークの帯域をフルに使えます。1つの tar の中身は、同じキー（`__key__`、サンプルの識別子）を共有するファイル群が1サンプルになります。

```
sample000123.cam_0.jpg   # 視点0のフレーム
sample000123.cam_1.jpg
   ...
sample000123.cam_6.jpg   # 視点6
sample000123.ego.npy     # 車両状態（例: 過去64×4 + 未来64×2 = 384次元）
sample000123.meta.json   # メタ情報
```

読み込み時のデコードの流れは次のようになります。

1. `.jpg` で終わるキーを集めて視点順にソートし、画像を開いて `ToTensor`（0〜255の画素を0〜1のテンソルへ）と `Normalize`（平均・分散で正規化）を掛ける。
2. それらを `torch.stack` して視点方向に積む（7視点・256×256・RGBなら `(7, 3, 256, 256)`）。
3. `.npy` のバイト列を `np.frombuffer` で配列に戻し、過去ぶん・未来ぶんに分ける（例: 先頭256要素を文脈、残り128要素を教師信号）。

ローダ生成の典型コードはこうです。

```python
import webdataset as wds
dataset = wds.WebDataset(urls, shardshuffle=True, nodesplitter=wds.split_by_worker)
dataset = dataset.shuffle(1000)         # シャッフルバッファ内で混ぜる
dataset = dataset.map(decode_sample)    # 上のデコード関数
loader  = wds.WebLoader(dataset, batch_size=8, num_workers=4)
```

バッチ後の形:

```
batch["visual_tiles"]      (B, 7, 3, 256, 256)
batch["egomotion_history"] (B, 256)
batch["trajectory_target"] (B, 128)
```

シャッフルの工夫が重要です。WebDataset はシーケンシャル読みが速さの源なので、全データを完全ランダムアクセスはしません。代わりに「シャッフルバッファ」（例えば1000サンプル）を持ち、その中だけで混ぜます。これでシーケンシャル性を保ちつつランダム性を得ます。さらに tar（シャード）自体の順序も混ぜれば（`shardshuffle`）、十分なランダム性になります。バッファが小さいと近接サンプル同士の相関が残り、ミニバッチが偏るので、バッファサイズはトレードオフです。

before / after の比較:

```
[before] 学習中に毎回動画デコード:
  GPUステップ < 数ミリ秒、データ供給 ~数十秒/サンプル → GPUがほぼ遊休

[after] 事前抽出 + WebDataset:
  抽出は1回きり（tar をオブジェクトストレージへ保存）
  → 学習時はローカルディスクへ同期 → tar 順次読み（デコード不要）
  → GPU を飽和できる
```

クラウド学習では、抽出済み tar をオブジェクトストレージ（S3 など）に置き、学習ジョブの起動時にローカルの高速ディスク（EBS、NVMe など）へ同期してから読む構成がよく使われます。あるいは WebDataset の URL に直接 S3 を指定してストリーミングすることもできます。

## 手法の系譜と主要論文

事前抽出・キャッシュは、論文の新規アルゴリズムというより大規模学習で確立したエンジニアリング技法です。系譜をたどると、まずフレームワーク付属の直列化フォーマットがあります。TFRecord（TensorFlow）と RecordIO（MXNet）は、サンプルを連続したレコードに直列化して保存し、デコードと I/O を学習ループから切り離す先行例でした。動画・画像をデータ供給段階で前処理しておくのは、この頃から大規模学習の定石です。

WebDataset（A. Aizman, G. Maltby, T. Breuel, "High Performance I/O For Large Scale Deep Learning", IEEE Big Data 2019, arXiv:2001.01858）は、多数の小ファイルを tar に束ねてシーケンシャルに読むことで、ランダムな小ファイルアクセスのオーバーヘッドを避け、ローカルでもクラウドでも I/O 帯域を飽和させる方式を整理しました。PyTorch エコシステムで広く採用され、巨大データセットをオブジェクトストレージからストリーミングする標準的な手段になっています。

データ供給のボトルネックそのものを攻める系統として、NVIDIA DALI（Data Loading Library）は、デコードやデータ拡張をGPUへオフロードして前処理を高速化します。FFCV（G. Leclerc et al., "FFCV: Accelerating Training by Removing Data Bottlenecks", CVPR 2023, arXiv:2306.12517）は、専用の高速フォーマットへ事前変換し、メモリマップとキャッシュを駆使して読み込みを極限まで速くします。いずれも「事前変換 → 高速読込でGPUを遊ばせない」という本トピックと同じ構図です。

## 論文の実験結果（定量データ）

FFCV は ImageNet 学習で、データ供給を高速化することで全体の学習時間を大きく短縮したと報告しています。論文では、標準的な PyTorch の DataLoader を FFCV に置き換えるだけで、同じ精度に到達するまでの時間が数分の一になるケースを示しました。とくに ResNet 系の ImageNet 学習を、複数GPUで数十分規模まで縮められることを実証しています。これは新しいモデルではなく、ひとえに「データ供給がGPUの足を引っ張っていた」ぶんを取り除いた効果です。指標としては「同じ検証精度に達するまでの壁時計時間（wall-clock time）」が使われ、モデルの良し悪しではなくパイプライン効率を測っています。

WebDataset の文献やベンチマークでは、小ファイルをバラバラに読む方式に比べ、tar のシーケンシャル読みがディスク・ネットワークのスループット（単位時間あたりに読めるデータ量）を桁違いに引き上げることが示されています。報告によると、数百万ファイル規模のデータセットでは、ファイルシステムのメタデータ操作（ファイルを開く・閉じる）のオーバーヘッドが支配的になり、tar 化でこれをほぼ消せます。クラウドのオブジェクトストレージでは、リクエスト数が課金とレイテンシの両方に効くため、tar 化によるリクエスト数削減は速度とコストの両面で効きます。

入力律速の改善効果を一般化すると、GPU稼働率が例えば10〜30%（データ待ちが支配的)だった学習を、事前抽出と効率的ローダで80〜95%まで引き上げられることが各種報告で共通して見られます。同じGPUで実効スループットが数倍になるため、計算資源あたりのコスト効率が直接改善します。

## メリット・トレードオフ・限界

メリットは、学習時の動画デコードを完全に排除し、供給速度を「動画展開速度」から「ディスク帯域速度」へ引き上げられることです。GPU稼働率が大幅に上がり、tar のシーケンシャル読みでクラウドストリーミングとも相性が良くなります。重い依存ライブラリ（動画読み込みやデータセット固有のSDK）を学習コンテナから外せるので、コンテナが軽くなり再現性も上がります。

トレードオフも明確です。抽出済みデータのぶん、ディスク・オブジェクトストレージの容量が余分に要ります（元動画に加えて展開済み画像を持つため、JPEG圧縮を使ってもデータ量は増えます）。抽出時にウィンドウ長・解像度・視点を固定するので、後から「別の窓長で学習したい」「もっと高解像度で」となると再抽出が必要です。データ拡張（augmentation）の自由度も下がります。事前に固定の前処理を焼き込むと、ランダムクロップや色変換のようにエポックごとに変えたい拡張がやりにくくなるため、拡張は読み込み後にオンザフライで掛ける設計にする必要があります。ランダム性がシャッフルバッファに依存するため、バッファが小さいと相関が残ります。さらに、抽出パイプライン自体の実装・保守コストとバグ混入リスクが増えます。

未解決寄りの課題としては、保存容量と読み込み速度のバランス（圧縮率を上げると読み込み時のデコードが増える）、巨大データの再抽出コスト、動画の時間方向のランダムアクセスを保ちながら高速化する手法などがあります。動画特有の圧縮を活かしたまま高速ランダムアクセスを実現する研究（フレーム索引付きフォーマットなど）も進んでいます。

## 発展トピック・研究の最前線

近年は、抽出済みデータを使いつつ拡張の自由度を取り戻す工夫が進んでいます。読み込み後にGPU上で拡張を掛ける（DALI 系）、複数の前処理バリアントを確率的に切り替える、潜在表現（latent、画像をエンコーダで圧縮した特徴）の段階でキャッシュしてストレージと拡張のバランスを取る、といった方向です。マルチノード分散学習では、シャードをワーカー間で重複なく分配する（`split_by_worker` / `split_by_node`）設計や、各ノードのローカルキャッシュを使うハイブリッド構成が研究・実用されています。研究を始めるなら、まず手元のデータローダにプロファイラを当ててGPU稼働率とデータ待ち時間を測り、入力律速かどうかを切り分けるところから始めると、事前抽出が効くかどうかを数値で判断できます。
