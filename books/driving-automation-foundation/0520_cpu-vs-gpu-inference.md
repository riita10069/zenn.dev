# CPU推論とGPU推論の性能差

## ひとことで言うと

同じニューラルネットワークでも、普通の CPU(パソコンの汎用の頭脳)で動かすと 1 回の推論に数秒かかるのに、GPU(画像・並列計算が得意な専用チップ)で動かすとそれが数十ミリ秒で終わります。条件によっては数十倍から百倍以上の差になります。なぜここまで違うのか、その差がリアルタイムシステムにとって何を意味するのか、そして CPU でも間に合わせるための工夫(量子化・蒸留・専用ハード)は何かを、ハードウェアの仕組みと実測値から説明します。

## 直感的な理解

ある画像認識モデルを、6 年落ちのノート PC の CPU で動かしたら 1 回の推論に約 2.5 秒かかった。同じモデルを安価なノート向け GPU で動かしたら約 68 ミリ秒(0.068 秒)で終わった。およそ 36 倍の差です。さらに精度を少し落とす数値形式(FP16)を使うと約 34ms になり、CPU との差は約 73 倍に広がります。

この差はソフトウェアの工夫ではなく、ハードウェアの構造そのものから来ています。深層学習の計算の本質は「巨大な行列どうしの掛け算を、同じパターンで大量に繰り返す」ことです。CPU と GPU は、この「同じ計算を大量に並列で」という作業に対する得手不得手が正反対なのです。料理にたとえると、CPU は「腕利きの料理人が数人で、複雑な一品を順番に丁寧に作る」のが得意。GPU は「単純な作業なら、数千人のアルバイトが一斉に同じ工程をこなす」のが得意。行列演算は後者、つまり単純作業の大量並列なので、GPU が圧勝します。

## 基礎: 前提となる概念

### CPU と GPU のアーキテクチャの違い

CPU(Central Processing Unit)は、数個から数十個のコア(計算の実行単位)を持ちます。各コアは非常に賢く、分岐予測(if 文の結果を先読みする)・アウトオブオーダー実行(命令を効率の良い順に並べ替える)・大きなキャッシュ(よく使うデータを手元に置く)といった高度な仕組みを備え、複雑で逐次的(順番に進む)な処理を高速にこなします。しかし同時に走らせられる計算の本数(並列度)はコア数 × SIMD 幅(1 命令で同時に処理できるデータ数)で、実質数十〜数百本程度です。CPU はトランジスタの大半を「1 本のスレッドを賢く速く進める」ための制御回路に使っています。

GPU(Graphics Processing Unit)は、数千個の単純なコアを持ちます。1 個 1 個は分岐予測も持たず賢くありませんが、同じ命令を大量のデータに一斉に適用する SIMT(Single Instruction Multiple Threads)方式に最適化されています。GPU はトランジスタの大半を「計算そのもの(演算ユニット)」に振り向けています。さらに、行列積を専用に高速化する Tensor Core(NVIDIA)のような専用回路を持つ世代もあります(詳細は [FP16推論とTensor Core活用](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0521_fp16-inference-tensorcore))。

数字で比べると、ハイエンド CPU の理論演算性能が数 TFLOPS(1 秒あたり数兆回の浮動小数点演算)なのに対し、データセンター向け GPU は数百 TFLOPS から PFLOPS 級に達します。メモリ帯域も、CPU が数十〜百 GB/s なのに対し、GPU は数百 GB/s〜数 TB/s と桁違いです。深層学習は演算量とメモリ帯域の両方を大量に要求するので、この差がそのまま推論速度の差になります。

### なぜ行列演算は並列向きなのか

ニューラルネットワークの全結合層は数学的には行列とベクトルの積、畳み込み層も内部的には行列積(im2col 変換などで)に落ちます。出力行列の各要素は、入力行列の行と重み行列の列の内積で、互いに独立に計算できます。つまり「100 万個の要素を同時に計算してよい」という、並列化に理想的な構造です。GPU の数千コアはまさにこれを一斉にこなすために存在します。CPU では数十本ずつしか並列に処理できないので、同じ計算を何万回も順番に回すことになり、圧倒的に時間がかかります。

この性質を「演算強度(arithmetic intensity、メモリから読んだ 1 バイトあたり何回演算するか)」という言葉で整理することもできます。行列積は演算強度が高く、データを一度読めば大量に計算するので、演算ユニットを遊ばせない GPU 向きのワークロードです。

## 仕組みを詳しく

### 同一モデルの CPU/GPU 実測比較

「まったく同じモデル・同じ入力を、CPU と GPU でそれぞれ動かして測る」のが、ハードウェア差を切り出す最も誠実な方法です。ある実測報告では、入力テンソル形状 `[1, 8, 3, 256, 256]`(バッチ 1、8 カメラ、RGB 3 チャンネル、256×256 画素)のマルチカメラ運転モデルを、CPU と GPU で比較しています。

CPU 側。Intel Core i7-6600U @ 2.60GHz(2016 年発売の 2 コア / 4 スレッドのモバイル CPU)、RAM 約 8GB、深層学習フレームワークを CPU のみで実行。ウォームアップ 2 回・本計測 5 回。

| 指標 | 値 |
|---|---|
| 平均レイテンシ | 約 2466 ms(約 2.5 秒) |
| 中央値 | 約 2367 ms |
| 標準偏差(ジッタ) | 約 168 ms |
| p99 | 約 2764 ms |
| スループット | 約 0.41 推論/秒 |

GPU 側。NVIDIA GeForce RTX 3050 Laptop GPU 6GB(エントリークラスのノート向け GPU)。

- FP32(単精度、モデル単体ストレステスト): 平均 約 67.7 ms、約 14.8 FPS
- FP32(実データ込みパイプライン): 平均 約 81.6 ms、p99 約 104 ms
- FP16(半精度): 平均 約 33.8 ms、約 29.6 FPS

### 性能差の整理

```
CPU (i7-6600U):         約 2466 ms / 推論   →   0.41 FPS
GPU (RTX 3050, FP32):   約  67.7 ms / 推論   →  14.8  FPS
GPU (RTX 3050, FP16):   約  33.8 ms / 推論   →  29.6  FPS
```

CPU の 2466ms と GPU FP32 の 67.7ms を比べると約 36 倍、FP16 の 33.8ms と比べれば約 73 倍 GPU が速いことになります。重要なのは、これがエントリークラスのノート GPU での値だという点です。データセンター GPU を使えば差はさらに開きます。なお、これらの値を信頼するには CPU/GPU 双方で計測条件を揃え、GPU 側では非同期実行による過小評価を防ぐ必要があります(詳細は [推論レイテンシ計測の方法論](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0522_inference-latency-benchmark-methodology))。

### リアルタイム要件との関係

自動運転やロボティクスでは 10Hz(100ms 以内)が一つの目安です。GPU(FP32 で 68ms、FP16 で 34ms)はこの要件を満たしますが、CPU(2.5 秒)はまったく間に合いません。1 秒に 0.41 回しか処理できないということは、車が時速 36km(秒速 10m)で走っているとき、1 回の判断の間に 25m も進んでしまう計算です。これでは反応が致命的に遅れます。

したがって実務上の結論は明確です。CPU 推論は「コードが動くことの確認」や「GPU のない開発環境でのデバッグ」には使えますが、リアルタイム走行には使えません。CPU の数字は「最低でもこれだけかかる、つまり GPU が必須だ」という根拠として位置づけられます。

### 注意: 公平な比較のための条件統一

CPU と GPU を比べるとき、計測条件を揃えないと意味がありません。両方で同じウォームアップとサンプル数を取り、GPU 側は前後に `torch.cuda.synchronize()` を入れて非同期実行による過小評価を防ぎ、CPU 側はスレッド数(`torch.set_num_threads`)を明記します。CPU は MKL や OpenBLAS といった行列演算ライブラリの最適化、スレッド数、メモリ帯域、NUMA 構成に強く左右されるので、「最適化された CPU 実行」と「素朴な CPU 実行」でも数倍の差が出ることがあります。たとえば全コアを使い切る設定と 1 スレッドの設定では、コア数倍に近い差が出ることもあります。報告では条件を明記するのが必須で、明記しない CPU の数字は再現性がありません。

## 手法の系譜と主要論文

CPU と GPU の性能差は、深層学習の歴史を動かしてきた根本的な事実です。

Krizhevsky・Sutskever・Hinton(2012, NeurIPS, "ImageNet Classification with Deep Convolutional Neural Networks"、通称 AlexNet)は、大規模 CNN を 2 枚の GPU(GTX 580、各 3GB)で学習させ、ImageNet 画像認識コンペ(ILSVRC-2012)で 2 位以下を大きく引き離して優勝しました(top-5 エラー率 約 16.4%、2 位は約 26%)。それ以前は大規模ネットワークの CPU 学習が現実的でないほど遅く、GPU の並列性が深層学習を実用化した歴史的転換点です。ここで示されたのは「GPU でなければ学習も推論も現実的な時間で終わらない」という事実でした。

Jouppi ら(Google, 2017, ISCA, "In-Datacenter Performance Analysis of a Tensor Processing Unit")は、推論専用チップ TPU(Tensor Processing Unit)を提案し、当時の CPU(Haswell)や GPU(K80)に対して推論で平均的に高速化、電力効率では大幅な改善を報告しました。TPU の中核は行列積を一気に処理するシストリックアレイ(systolic array、データを波のように流して計算する格子状の演算回路)で、「行列積をハードで専用化すると汎用 CPU より桁違いに速い」ことを定量的に示しました。これは CPU が深層学習推論に向かない理由を、専用ハードの優位という裏側から説明する論文です。

Han ら(2016, ICLR, "Deep Compression")は、枝刈り(重要でない重みを 0 にする)・量子化(重みを少数の代表値に丸める)・ハフマン符号化を組み合わせてモデルを大幅に圧縮(AlexNet を約 35 倍、VGG-16 を約 49 倍に縮小、精度劣化ほぼなし)し、CPU やエッジでも推論を間に合わせる道を開きました。Jacob ら(2018, CVPR, "Quantization and Training of Neural Networks for Efficient Integer-Arithmetic-Only Inference")は、FP32 を INT8 整数に量子化して CPU・モバイル CPU 上での推論を整数演算だけで高速化する手法を示しました。Hinton ら(2015, "Distilling the Knowledge in a Neural Network")の知識蒸留は、大きな教師モデルの出力分布を小さな生徒モデルに学習させることで、小型モデルの精度を底上げします。これらは「GPU が使えない環境でどう間に合わせるか」という、まさに CPU の遅さを起点にした研究系譜です。

## 論文の実験結果(定量データ)

TPU 論文(2017)の代表的な数値を見ると、推論ワークロード(当時の本番モデル MLP/CNN/LSTM 群)で、TPU は同時代の CPU・GPU に対して大きなスループット向上を示しました。ここで効く指標はスループット(1 秒あたりの推論件数)と TOPS/Watt(電力あたりの演算性能)で、後者は「同じ電力でどれだけ計算できるか」を測ります。データセンターでは電力が運用コストの大半なので、この指標が桁違いに改善することが TPU 導入の決め手でした。汎用 CPU は制御回路に多くの電力を使うため、純粋な行列積では電力効率で専用ハードに大きく劣ります。

量子化(INT8)のアブレーション的な効果としては、Jacob ら(2018)が MobileNet を INT8 化して、モバイル CPU 上でレイテンシを大きく改善しつつ、ImageNet 精度の低下を 1〜2% 程度に抑えられることを示しました。「精度を少し犠牲にすれば CPU でも実用速度に近づける」という、CPU の遅さを部分的に埋める手段の定量的根拠です。冒頭の実測例でも、GPU 側で FP32(67.7ms)を FP16(33.8ms)にすると約 2 倍速くなっており、精度形式の選択がレイテンシに直結することが分かります。Deep Compression の圧縮率(35〜49 倍)は、モデルがいかに冗長なパラメータを持っているか、つまり「正しく削れば小さなハードでも動く余地が大きい」ことを示しています。

ただし注意すべきは、量子化や圧縮の効果はワークロード次第だという点です。INT8 化が速度に効くのは、ハードに整数演算の専用パスがあり、かつそのモデルが演算律速(compute-bound)である場合です。メモリ帯域律速(memory-bound)の小さなモデルでは、データ量が減るぶんメモリ転送で速くなることはあっても、CPU の根本的な並列度不足は埋められません。

## メリット・トレードオフ・限界

GPU 推論のメリットは、圧倒的な速度(CPU 比で数十倍)、リアルタイム要件を満たせること、FP16/INT8 でさらに高速・省メモリにできることです。トレードオフは、GPU ハードが高価で消費電力・発熱が大きいこと、VRAM(GPU 上のメモリ)の容量制約があること、そして物理的に GPU を積めない環境(小型エッジ機器、既設の組込みシステム)では使えないことです。さらに、CPU から GPU へのデータ転送(PCIe 経由)自体がオーバーヘッドになり、小さな入力では転送時間が計算時間を上回って GPU の優位が薄れることもあります。

CPU 推論のメリットは、どんなマシンでも動くこと、GPU 用のドライバや CUDA 環境の構築が不要なこと、開発時の手軽な動作確認やデバッグに使えることです。トレードオフは、非常に遅く(本件で約 2.5 秒)リアルタイム走行に使えないこと、バッチサイズや解像度を上げるとさらに遅くなることです。

未解決・難しい論点として、「どこで推論するか」の設計判断があります。クラウド GPU に投げればモデルは速いが、通信遅延とネットワーク途絶のリスクが乗ります。エッジ GPU を積めば自己完結するが、コスト・電力・熱・スペースの制約に縛られます。CPU しかないレガシー環境では、量子化や蒸留でモデルを軽くしても、大規模モデルはどうしても間に合わないことがあり、モデル設計とハードウェア選定を同時に最適化する必要があります。さらに、CPU の最適化は使う BLAS ライブラリ・スレッド数・NUMA 構成に強く依存し、「公平な CPU の数字」を出すこと自体が難しい、という計測上の課題も残ります。

## 発展トピック・研究の最前線

CPU の遅さを起点に発展した分野は広大です。モデル圧縮(枝刈り・量子化・蒸留)、専用アクセラレータ(TPU, NPU, FPGA, エッジ AI チップ)、ハードウェアを意識したアーキテクチャ探索(HW-aware NAS、レイテンシ実測を目的関数に組み込む)などがあります。近年は INT8 を超えて INT4・FP8・二値化(BNN)といった超低ビット量子化や、混合精度(層ごとに精度を変える)が研究されています。

LLM 時代に入ると、巨大モデルを限られたメモリで動かすための量子化(GPTQ, AWQ など)や、CPU でも LLM を動かす取り組み(llama.cpp など)が活発です。これは「GPU が足りない・使えない環境でどう推論するか」という古典的問題が、モデルの巨大化とともに再燃したものです。LLM 推論ではメモリ帯域律速の側面が強く、量子化が速度に効きやすいという特徴もあります。逆にデータセンターでは、GPU・TPU・専用 ASIC を使い分け、ワークロードごとに最もコスト効率の良いハードを選ぶ「ハードウェアの異種混合(heterogeneous computing)」が標準になりつつあります。自動運転の車載推論では、限られた電力・熱予算の中でリアルタイム性を満たすため、車載向け SoC(NVIDIA DRIVE シリーズなど)の専用アクセラレータ上で量子化済みモデルを動かす、という構成が実用化の主流です。
