# デュアルモードデータセット (ローカル直読み と 事前抽出シャードの統一)

## ひとことで言うと
深層学習モデルに大量のデータ(動画・画像・センサーログなど)を供給するとき、開発初期は手元の1台のマシンで小さく動かしたく、本番では何百万サンプルをクラウドのGPUクラスタで何時間も回したくなります。この2つは「データをどこから・どう取り出すか」がまったく違うのに、「1サンプルをモデルが食える形に組み立てる」中身は同じです。デュアルモードデータセットとは、その組み立てロジックを1本に統一し、データの入口(読み出し経路)だけを差し替えられるように設計する考え方です。これにより、同じ前処理コードがローカルでもクラウドでもそのまま動き、二重メンテナンスを避けられます。

## 直感的な理解
料理店を思い浮かべてください。試作の段階では、シェフが市場(オンライン上の元データ)へ自分で買い出しに行き、その場で野菜を洗って切って調理します。手軽ですが、注文が増えると「買い出しと下処理」に時間を取られて調理の手が止まります。本番営業では、あらかじめ下処理済みの食材(洗って切ってパックした状態)を冷蔵庫にストックしておき、注文が来たら取り出して焼くだけにします。こうすればコンロ(GPU)が常にフル稼働できます。

ここで重要なのは「焼く」という調理ロジック自体はどちらでも同じだということです。違うのは「食材をどこから・どんな状態で取り出すか」だけ。デュアルモードはまさにこの分離を行います。試作時は元データを直読みしてその場で下処理(動画デコードや特徴計算)を行い、本番時は下処理済みのデータをただ読み出すだけにする。下流のモデルや損失関数から見ると、出てくるデータの形は完全に同じです。

なぜこの分離が要るのか。深層学習の学習速度は、しばしばモデルの計算速度ではなく「データをどれだけ詰まらせずGPUに供給できるか」で決まります。GPUが1ステップを10ミリ秒で計算できても、次のバッチを用意するのに30ミリ秒かかれば、GPUの3分の2は遊んでいることになります。これを入力パイプラインのボトルネック(input-bound、入力律速)と呼びます。クラウドのGPUは1時間あたり数ドルかかるので、遊んでいるGPUは直接お金を捨てているのと同じです。デュアルモードの本番経路は、このボトルネックを潰すために設計されています。

## 基礎: 前提となる概念
このトピックを理解するために、いくつかの言葉を噛み砕いておきます。

Dataset と DataLoader。多くの深層学習フレームワークには、データを1件ずつ取り出す「Dataset(データの保管庫から1サンプルを整形して返す係)」と、それを複数まとめてGPUへ送る「DataLoader(複数サンプルを束ねてバッチにし、裏でCPUワーカーを使って先読みする係)」という2層構造があります。Dataset は `__getitem__(i)` という関数で「i番目のサンプルをくれ」に応える、と考えてください。DataLoader は `num_workers`(並列ワーカー数)を指定すると、複数のCPUプロセスが先回りして次のバッチを準備し、GPUが現在のバッチを計算している間に裏で次を仕込みます。これをプリフェッチ(prefetch、先読み)と呼びます。

tensor と shape。tensor とは「数値が格子状に並んだ多次元配列」です。shape はその各次元の大きさを表します。例えばカラー画像1枚は `(3, 256, 256)` という shape を持ちます。これは「3チャンネル(赤・緑・青)× 縦256ピクセル × 横256ピクセル」を意味します。複数枚を束ねると先頭に枚数の次元が付き `(B, 3, 256, 256)` になります。この B をバッチサイズ(batch size、一度にまとめて処理する枚数)と呼びます。

動画デコード。カメラ映像は H.264 / H.265 などの形式で圧縮されて保存されています。圧縮された動画から特定の1フレームを取り出すには、デコード(圧縮を解いて1枚1枚の画像に展開する処理)が必要です。動画圧縮は「キーフレーム(I フレーム、それ単独で画像になる完全な1枚)」と「差分フレーム(P/B フレーム、前後のフレームとの差だけを記録)」で構成されるため、任意の1フレームを取り出すには直前のキーフレームまで遡ってデコードし直す必要があります。これはCPUにとってかなり重い処理で、毎ステップ繰り返すと前述の input-bound の主犯になります。

LeRobot 形式。ロボット学習・模倣学習の分野で広まっている標準的なデータフォーマットで、複数カメラの動画・行動(action)・状態(state)・タイムスタンプを統一的に格納します。Hugging Face(機械学習データを共有するプラットフォーム。略してHF)上で多くの運転・操作データセットがこの形式で配布されています。模倣学習(imitation learning)とは「人間(あるいは熟練ポリシー)のデモンストレーションを真似してモデルを学習させる」枠組みで、自動運転では「人間ドライバーがこの状況でどう操作したか」を正解として学びます。

シャード(shard)。大量のデータを小分けにした塊のファイルです。後述の WebDataset 方式では tar(複数ファイルを1つにまとめた書庫)1個が1シャードになります。1シャードを数百MB〜数GB程度にするのが定石で、これはネットワーク転送効率(1リクエストで大きく読む)と並列性(複数シャードを別ワーカーが同時に読む)のバランスから来ます。

自車運動(egomotion)。自分の車両がどう動いたか、すなわち並進(前後左右の移動)と回転(向きの変化)を時系列で表したものです。自動運転では「過去どう動いてきたか」が将来の軌跡予測の重要な手がかりになり、サンプルの一部として前処理段階で計算しておきます。

## 仕組みを詳しく
デュアルモードは大きく2つの読み出し経路を持ち、両者の出力 shape を完全に一致させるのが設計の肝です。

### モード1: ローカル直読み(オンラインデコード)
HF などから元データ(LeRobot 形式など)をダウンロードし、`__getitem__` が呼ばれるたびにその場で必要なフレームを動画からデコードし、過去・未来の自車運動を計算します。1サンプルの中身は例えば次のような形になります。

```
sample["visual_tiles"]       (N_cam, 3, H, W)   N_cam台のカメラ画像 (各 RGB×H×W)。例: (7, 3, 256, 256)
sample["egomotion_history"]  (D_ego,)           過去の自車運動を D_ego 次元ベクトルに要約。例: (256,)
sample["visual_history"]     (D_vis,)           過去フレームの視覚特徴を要約。例: (896,)
sample["trajectory_target"]  (T*2,)             正解軌跡。T ステップ × 2成分(x,y)。例: 64*2=128
sample["episode_index"]      int                どの走行エピソードか
sample["frame_index"]        int                その中の何フレーム目か
```

この経路の長所はセットアップが簡単で、元データさえあれば1台で完結する点。データセットを切り出す解像度・カメラ選択・拡張(augmentation)をその場で自由に変えられる柔軟さもあります。短所は毎サンプルで動画デコードが走るため遅く、CPUを食う点です。開発初期の「とりあえずモデルが動くか」「損失が下がるか」の確認には最適です。

### モード2: 事前抽出シャード(オフライン前処理 + ストリーム読み)
本番では、学習を始める前に一度だけ重い前処理を済ませてしまいます。動画から必要フレームを画像(JPEG/PNG)へ展開し、自車運動も計算済みの数値ファイル(.npy など)にして、tar にまとめておきます。1シャード(tar)内のファイル構成は次のようになります。

```
000123.cam_0.jpg, ... 000123.cam_6.jpg   1サンプルぶんの N_cam 枚(デコード済み)
000123.ego.npy                            計算済みの自車運動
000123.meta.json                          メタ情報(episode, frame など)
000124.cam_0.jpg, ...                      次のサンプル...
```

WebDataset 方式では「同じ接頭辞(000123)を持つファイル群が1サンプル」という規約でグルーピングします。学習時はこの tar を先頭から順に読むだけで、動画デコードは一切起きません。tar を頭から順に読むのはディスク・ネットワークにとって最速のアクセスパターン(シーケンシャルリード)で、クラウドストレージ(S3 など)からでも帯域をフルに使えます。バッチを取り出すと:

```
batch["visual_tiles"]       (B, N_cam, 3, H, W)   例: (8, 7, 3, 256, 256)
batch["egomotion_history"]  (B, D_ego)            例: (8, 256)
batch["visual_history"]     (B, D_vis)            例: (8, 896)
batch["trajectory_target"]  (B, T*2)              例: (8, 128)
```

モード1の1サンプルに先頭の B 次元が付いただけで、各キーの意味と shape は完全に同じです。だからこの後ろにつながるモデルや損失関数は、データがどちらの経路から来たかを区別する必要がありません。これが「契約(contract)を固定する」という設計原則です。Dataset の出力スキーマ(キー名と shape)を契約として定義し、両モードがそれを満たすことを保証すれば、下流は契約だけに依存できます。

### 入口を吸収する薄い中間層(shim)
両モードをつなぐのが「shim(シム)」と呼ぶ薄い中間層です。shim とは本来「部品の隙間に挟んで高さや位置を合わせる薄板」のことで、ソフトウェアでは「呼び出し方の違いを吸収する薄いアダプタ層」を指します。ここでは「学習スクリプトは DataLoader を1つ受け取るだけ。その loader が直読み由来かシャード由来かは、生成関数の選択(あるいは import / loader の差し替え)で決まる」という構造を作ります。環境変数やフラグ1つでモードを切り替えられるようにしておけば、同じ学習コードがローカルとクラウドの両方で動きます。

```
            ┌─────────────────────────┐
  元データ ─▶│ モード1: 直読み Dataset   │─┐
  (HF/動画)  │  (オンラインデコード)     │  │
            └─────────────────────────┘  │   同じ shape の
                                          ├─▶ batch dict ─▶ model.forward() ─▶ loss
            ┌─────────────────────────┐  │   (経路を区別しない)
  シャード ─▶│ モード2: WebDataset loader│─┘
  (tar/S3)   │  (ストリーム読み)         │
            └─────────────────────────┘
```

学習ループ本体(forward → loss → backward → step)は loader の正体を知りません。before/after で言えば、before は「ローカル用学習コード」と「クラウド用学習コード」を別々に書いて前処理ロジックが二重化した状態。after は前処理ロジックが1本で、loader 生成だけが違う状態です。この一本化は、保守コストを下げるだけでなく「ローカルで確認したものとクラウドで走るものが本当に同じか」という再現性の不安も消します。

## 手法の系譜と主要論文
デュアルモードという呼称は分野共通の固有名詞ではありませんが、その背後の発想には明確な系譜があります。

入力パイプラインを学習の妨げにしない、という運用知見は ImageNet 規模の分散学習を扱う研究群から確立しました。代表が Priya Goyal らの "Accurate, Large Minibatch SGD: Training ImageNet in 1 Hour"(Facebook、2017年、arXiv:1706.02677)です。バッチサイズ8192で ResNet-50 を1時間で学習する際、256基のGPUを遊ばせないために前処理・データ供給を徹底的に先回りする必要があると示しました。論文の主題は学習率のリニアスケーリング則とウォームアップですが、付随して「データ供給がボトルネックになる」現実が広く共有されました。

その供給を高速化する具体策が WebDataset です。Alex Aizman, Gavin Maltby, Thomas Breuel(NVIDIA)の "High Performance I/O For Large Scale Deep Learning"(IEEE Big Data 2019、arXiv:2001.01858)が背景思想を述べています。動機は明快で、何百万もの小さなファイルをファイルシステムやオブジェクトストレージから個別に開くと、1ファイルごとの open/close オーバーヘッドが膨大になります。これを tar にまとめて連続ストリーム読みすることで、ローカルディスクでもクラウドストレージ(S3 など)でも高スループットを実現します。POSIX ファイルシステムへの依存を断ち、純粋なシーケンシャル I/O に落とし込むのが核心です。同系統の発想として TFRecord(TensorFlow の protobuf ベースのシャード形式)、NVIDIA DALI(GPU上でデコード・拡張まで行うデータローディングライブラリ)、FFCV(独自バイナリ形式)があります。

データの「フォーマット統一」の系譜では LeRobot(Remi Cadene 他、Hugging Face、2024年)が重要です。ロボット学習・自動運転のデータを統一スキーマで配布・読み出しできるようにし、HFハブを通じて多様なデータセットへ同じAPIでアクセスできるようにしました。デュアルモードのモード1はこの統一APIに乗ることで、複数の異種データセットを同じ Dataset クラス設計で扱えます。

これらに共通する主張は一貫しています。「学習速度はモデルだけでなく、データをどれだけ詰まらず供給できるかで決まる」。デュアルモードは、開発の手軽さ(モード1)とスループット(モード2)を両立させるための実務的な統合です。

## 論文の実験結果(定量データ)
入力パイプラインがどれだけ効くかを示す定量例を挙げます。

FFCV(Guillaume Leclerc, Andrew Ilyas, Logan Engstrom, Sung Min Park, Hadi Salman, Aleksander Madry ら、MIT、CVPR 2023)は、データローダの改良だけで ImageNet の ResNet-50 学習を、標準の PyTorch ImageFolder + JPEG デコード経路に比べて大幅に短縮できると報告しました。公開ベンチマークでは、1台の8基A100構成で ResNet-50 を1時間未満、ResNet-18 なら数十分で目標精度に到達する水準を示しています。ここでの指標は「images/sec(秒あたり何枚処理できるか)」と「目標精度に達するまでの総時間(time-to-accuracy)」です。前者はパイプラインの純粋なスループット、後者は実務上の総コストを測ります。重要なのは、モデルを一切変えずローダだけで total time が縮むという点で、これは「学習が input-bound だった」ことの直接的証拠です。論文はメモリへのデータ常駐(quasi-random なメモリマップ)、非同期プリフェッチ、JIT コンパイルされた拡張パイプラインといった要素のアブレーションを示し、どの要素が効くかを切り分けています。

WebDataset 系の報告では、オブジェクトストレージ(S3 など)からの学習で、小ファイル個別取得に対して tar シーケンシャル読みがスループットを桁で改善する事例が知られています。具体的には、何百万の小ファイルを個別 GET すると1リクエストあたりのレイテンシ(数十ミリ秒)が支配的になり、有効帯域がストレージの数%に落ちることがあります。tar 連続読みではこのオーバーヘッドが償却され、帯域をほぼ使い切れます。Aizman らはこのアーキテクチャがクラウドストレージ上で線形に近いスケーリングを示すことを報告しています。

アブレーション的な観点で言えば、「オンラインデコードを残したまま分散学習する」と、CPUワーカー数を増やしてもデコードが追いつかずGPU利用率(GPU util)が頭打ちになる、という劣化が観測されます。GPU util(GPUが計算に使われている時間の割合)が例えば50%付近で張り付くなら、残り50%はデータ待ちであり、事前抽出に切り替えるとこれが90%超まで改善する、というのが典型的な before/after です。指標としての GPU util は「投じたGPU費用のうち何割が実際の計算に使われたか」を測る、コスト効率の直接的な物差しです。たとえば time-to-accuracy が半分になれば、同じ実験を倍の回数試せる、あるいはクラウド費用を半減できる、という研究の回転速度に直結します。

## メリット・トレードオフ・限界
メリット。前処理ロジックが1本に統一され、二重メンテナンスとそれに伴うバグ混入を避けられます。本番経路では動画デコードが消えてGPUが待たされず、学習が速く省CPUになります。開発時は元データ直読みで手軽に動作確認でき、開発ループが速い。下流のモデル・損失は経路非依存なので、データ供給の最適化をモデルと独立に進められます。

トレードオフと限界。事前抽出には「あらかじめ全フレームを展開してシャード化する一度きりの重い処理」と、それを置くストレージ容量が要ります。画像を生(または軽圧縮)で持つため、元動画より総容量が増えがちです(動画は時間方向の差分圧縮が効くが、独立した JPEG にすると効かなくなる)。WebDataset のような tar ストリームはランダムアクセスが苦手で、エポックごとの厳密な全体シャッフルができません。実装上はシャードの並び替えと、読み出しバッファ内での部分シャッフル(shuffle buffer)で近似しますが、これが収束や汎化に微妙な影響を与える可能性は残ります(特にデータに強い時間的・空間的相関がある場合、同一エピソードの連続フレームが同じバッチに固まりやすく、勾配のばらつきが減って学習が偏ることがある)。さらに、前処理の切り出し方・正規化・解像度・カメラ選択を変えたくなったら、シャードを作り直す必要があり、実験のイテレーション速度を下げます。研究上の未解決課題としては、「オンラインデコードの柔軟さ(任意の拡張・解像度変更)」と「事前抽出のスループット」を両立させる方法(GPU上デコード、部分的事前計算、キャッシュ階層など)の最適設計が挙げられます。

## 発展トピック・研究の最前線
GPU上でデコードまで行う NVIDIA DALI は、CPUデコードのボトルネックそのものをGPUに移すアプローチで、オンライン経路の柔軟さを保ちつつ高速化します。データ拡張(augmentation)をGPUで行う流れとも合流しています。FFCV のように独自バイナリ形式でメモリマップとプリフェッチを極限まで最適化する系統、ストレージ階層(ローカルNVMe → ネットワークFS → オブジェクトストレージ)を意識したキャッシュ設計、そして大規模言語・マルチモーダル学習で主流の「ストリーミングデータセット」(MosaicML Streaming など、S3 から決定的シャッフルしつつストリーム)が現在の最前線です。MosaicML Streaming は「途中再開しても各サンプルがちょうど1回ずつ現れる決定性(deterministic resumption)」と「ノードをまたいだシャッフルの質」を両立させる工夫を導入し、WebDataset の弱点であるシャッフルの偏りに正面から取り組んでいます。自動運転の文脈では、マルチカメラ・LiDAR・地図など異種・大容量モダリティをどう効率良くシャード化し、時間窓(過去Nフレーム)を持つサンプルをどう構成するかが固有の難しさになります。時系列の窓を事前抽出する場合、隣接サンプルがフレームを大きく共有するため、同じフレームを重複して保存する冗長さと、ストライドを空けて重複を減らす設計のトレードオフが研究課題として残ります。
