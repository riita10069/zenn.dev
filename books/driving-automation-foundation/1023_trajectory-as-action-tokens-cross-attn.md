# 軌道のアクショントークン化とBEVへのクロスアテンション

## ひとことで言うと

自動運転モデルが「これからどう走るか」を決めるとき、その候補となる軌道(未来の動きの並び)を「単語の列」のように扱い、周囲の地図情報(BEV=真上から見た空間マップ)を「参照する辞書」として見に行く仕組みです。軌道の各時刻が辞書のどこを見るかを学習することで、空間の細かい情報を潰さずに軌道を生成します。これは Transformer のクロスアテンション(交差注意機構)を「軌道生成」という問題に当てはめた設計で、物体検出の DETR や生成モデルの DiT で確立された「クエリが情報源を選択的に読む」という考え方の系譜に位置します。

## 直感的な理解

人間が運転計画を立てるとき、頭の中には「いま自分は片側二車線の道の右レーンにいて、前方50mに信号、左前方に歩行者」といった周囲の地図イメージがあります。これから3秒後にどこにいるかを考えるとき、地図全体を均等に意識するわけではありません。「3秒後の位置の周辺」に注目し、そこに障害物や車線境界があるかを確認します。つまり「未来のこの時刻のことを考えるときは、地図のこの場所を見る」という対応づけを、無意識にやっています。

機械学習でこれを素朴にやろうとすると、よくある失敗は「地図全体を1つの平均的な数値ベクトルに圧縮してから軌道を作る」ことです。これだと「どこに何があるか」という位置情報が混ざって消えてしまい、せっかく細かく作った地図が台無しになります。クロスアテンションは「軌道の各時刻が、自分に関係する地図の場所だけを選んで読む」という人間の注目に近い操作を、微分可能(学習可能)な形で実現します。これが本トピックの核心です。

## 基礎: 前提となる概念

理解に必要な用語を順に噛み砕きます。

軌道(trajectory)とは、これから数秒間に車がどこを通るかを時刻ごとに並べたものです。表現の仕方は研究によりさまざまですが、よくあるのは「未来 T 個の時刻 × 各時刻 D 個の制御量」です。制御量の取り方には2系統あります。1つは位置そのもの(x, y 座標)を並べる方式。もう1つは「加速度 acc」と「曲率 curv」のような車の運動指令を並べ、それを積分して位置を復元する方式です。後者は車の運動学的に滑らかな軌道を出しやすい利点があります。たとえば T=64、D=2(acc と curv)とすると、軌道は `[B, 64, 2]` という3階テンソル(数値の多次元配列)で表せます。B はバッチサイズ(同時処理するサンプル数)で、これを平坦化すると `[B, 128]` になります。

BEV(Bird's-Eye View、鳥瞰図)とは、カメラ画像やLiDAR点群を「真上から見下ろした平面マップ」に変換した特徴表現です。前方カメラの斜めの映像をそのまま使うより、真上から見た座標系に揃えたほうが、複数センサの統合・距離の把握・経路計画がやりやすくなります。BEV 特徴は典型的に `[B, C, H, W]` という形で、C はマス目あたりの特徴量(チャネル)の長さ、H と W は縦横のマス目数です。たとえば `[B, 256, 200, 200]` なら、200×200=40,000 個のマスそれぞれに 256 次元のベクトルが乗っています。カメラからBEVへの変換は Lift-Splat-Shoot(Philion & Fidler, 2020, ECCV)が代表的な手法で、各画素に深度分布を推定して3次元に持ち上げ(lift)、BEV平面に投影(splat)します。

トークン(token)とは「処理の最小単位となるベクトル」のことです。自然言語処理では単語1個がトークンに相当します。Transformer はトークンの列を入力に取り、トークン同士の関係を注意機構で計算します。

アテンション(attention、注意機構)は Transformer(Vaswani et al., 2017, NeurIPS)の中核です。3種類のベクトルを使います。クエリ Q(query=「何を探したいか」)、キー K(key=「見出し語」)、バリュー V(value=「見出し語の中身」)です。Q と K の内積で「Q が各 K にどれだけ注目するか」のスコアを出し、softmax で正規化した重みで V を加重平均します。式で書くと `Attention(Q,K,V) = softmax(QK^T / sqrt(d)) V` で、`sqrt(d)` は内積が大きくなりすぎるのを抑えるスケーリングです。Q と K・V が同じ列なら「セルフアテンション(自分自身を見る)」、別の情報源なら「クロスアテンション(交差)」と呼びます。

## 仕組みを詳しく

軌道をアクショントークン列に変換し、BEV をクロスアテンションで参照する処理を、段階的に追います。ここでは T=64(未来64時刻)、D=2(acc, curv)、C=256(特徴次元)、BEV を `[B, 256, 200, 200]` とする数値例で説明します。

ステップ1: 軌道をアクショントークンの列にする。生成途中の軌道(後述する Flow Matching や拡散の途中状態)は `[B, 128]` という平坦なベクトルです。これを `[B, 64, 2]` に reshape(形を変える)します。64 個のトークン、各トークンは2次元(acc, curv)です。各2次元トークンを小さな線形層(`action_proj` のような名前で呼ばれることが多い)で 256 次元に持ち上げます。結果は `[B, 64, 256]`。これが「64個の時刻クエリ」になります。各クエリは「未来のこの時刻について、地図のどこを見たいか」を表すベクトルとして学習されます。

```
軌道 [B,128] → reshape [B,64,2] → 線形射影 → クエリ [B,64,256]
```

ステップ2: BEV を平坦化してキー・バリューにする。BEV `[B, 256, 200, 200]` を空間次元で平坦化し `[B, 256, 40000]`、転置して `[B, 40000, 256]` にします。40,000 個のマスがそれぞれ 256 次元、という「辞書の見出し語の列」になりました。これを線形層に通してキー・バリューに使います(K と V を別の射影で作る実装も、同じ列をそのまま K=V とする実装もあります)。

```
BEV [B,256,200,200] → flatten [B,256,40000] → transpose [B,40000,256] → 射影 → KV列
```

ステップ3: クロスアテンションで「軌道が地図を見に行く」。クエリ `[B,64,256]` を Q、BEV列 `[B,40000,256]` を K・V としてクロスアテンションを実行します。各時刻クエリは 40,000 マスのうちどこに注目するかを内積で計算し、注目度に応じてバリューを加重平均して取り込みます。出力は `[B,64,256]`。「軌道の各時刻が、自分に関係する BEV の場所を読み取った特徴」です。64×256 という構造が保たれているので、空間情報を1本に潰していない点が重要です。

マルチヘッド(multi-head)とは、注意の計算を複数の異なる部分空間で並行して行い、結果を結合する仕組みです。たとえば4ヘッドなら、あるヘッドは車線境界を、別のヘッドは交差点を、というように複数の手がかりを同時に拾えます。ヘッド数 h、各ヘッド次元 d/h で計算し最後に結合します。

```
クエリ[B,64,256]  ──┐
                    ├─ cross-attn (multi-head) ─→ attended [B,64,256]
BEV列[B,40000,256] ─┘
   (Q)                              (K, V)
```

ステップ4: 時刻条件と運転状況を AdaLN で注入する。生成モデルとして使う場合、「いま生成プロセスのどの段階か」という時刻 t と、「これまでの映像履歴・自車状態」という運転コンテキストも効かせる必要があります。これらは BEV のように空間的に広がっていないので、クロスアテンションではなく AdaLN という別経路で注入するのが定石です。

AdaLN(Adaptive Layer Normalization、適応的レイヤ正規化)は、画像生成の DiT(Peebles & Xie, 2023)で広まったテクニックです。条件ベクトル(t の埋め込みと運転コンテキストを足したもの)から (gamma, beta) という2本のベクトルを作り、正規化済みの特徴を `normed * (1 + gamma) + beta` でスケール・シフトします。空間的でない「全体に効く条件」を、特徴量の統計を変調することで注入するわけです。DiT では特に gamma と beta を初期値ゼロから始める AdaLN-Zero が学習安定化に効くと報告されています。

最後に各トークンを出力ヘッド(線形層)で D 次元に戻し、`[B,64,2]` を `[B,128]` に reshape して出力します。生成モデルの場合、この出力は「速度場(velocity field)」や「ノイズ予測」になり、軌道そのものではない点に注意します(詳細は [Euler法によるODE推論サンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0218_euler-ode-sampling-inference) と [外部ロス方式の学習API](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0959_external-loss-training-api))。

なぜこの設計なのか。空間的な条件(BEV)と非空間的な条件(時刻・履歴)を別経路に分けるのは、それぞれの役割が混ざらず学習が安定するためです。BEV は「どこを見るか」という位置選択が本質なのでアテンションが向き、時刻や履歴は「全体のトーンを決める」ものなので変調(AdaLN)が向きます。この分業は DiT が画像生成で確立し、行動生成にも転用された設計思想です。

## 手法の系譜と主要論文

クロスアテンションで「クエリが情報源を選択的に読む」という構図は、いくつかの研究が積み重なって定番化しました。

Transformer(Vaswani et al., 2017, NeurIPS, arXiv:1706.03762)が、アテンションだけで系列を処理する枠組みを提案し、Q/K/V とマルチヘッドという基本部品を確立しました。これがすべての出発点です。

DETR(Carion et al., 2020, ECCV, arXiv:2005.12872)は物体検出に Transformer を持ち込み、「学習可能な100個程度のオブジェクトクエリが、画像特徴をクロスアテンションで参照して各物体を出す」設計を広めました。「クエリ=出したいものの個数、キー・バリュー=見に行く情報源」という構図がここで明確になりました。本トピックは「クエリ=軌道の各時刻」という形でこの構図を踏襲しています。

DiT(Peebles & Xie, 2023, ICCV, arXiv:2212.09748)は画像生成の拡散モデルを Transformer で実装し、拡散の時刻条件を AdaLN(特に AdaLN-Zero)で注入すると、単純な加算やクロスアテンション注入より安定して品質が上がることを示しました。「時刻条件は変調、空間情報はアテンション」という分業の理論的・実験的な裏づけです。

Diffusion Policy(Chi et al., 2023, RSS, arXiv:2303.04137)は、ロボット制御で「行動列をトークン列として生成し、観測を条件として参照する」枠組みを示しました。行動(action)を生成対象のトークンとして扱う発想は、本トピックの「アクショントークン化」の直接の源流です。

自動運転側では、BEV を中間表現とする系譜(Lift-Splat-Shoot: Philion & Fidler 2020、BEVFormer: Li et al. 2022)と、軌道をクエリで生成する系譜(VAD: Jiang et al. 2023、UniAD: Hu et al. 2023, CVPR Best Paper)が合流し、「BEV 特徴を軌道クエリがアテンションで読む」エンドツーエンド設計が主流になりつつあります。これらはいずれも「広い空間や系列の情報を平均で潰さず、必要な場所だけアテンションで選択的に読む」という共通の動機を持ちます。

## 論文の実験結果(定量データ)

クロスアテンション型の選択的参照がもたらす効果は、関連する実験から読み取れます。指標の意味も添えて説明します。

DETR は COCO 物体検出で、当時主流だった Faster R-CNN と同等以上の AP(Average Precision、検出の正確さを再現率と適合率の曲線下面積で測る指標。高いほど良い、0-100)を達成しました。具体的には ResNet-50 バックボーンで AP 42.0 程度で、強化版 Faster R-CNN と互角でした。重要なのは、手作りのアンカーや NMS(非最大抑制という後処理)を使わずに「クエリのクロスアテンション」だけでこの精度に到達した点で、選択的参照の表現力を示しています。一方で小物体の AP は当時の CNN 系より劣り、これは「全画素をキーにする計算量」と「収束の遅さ」というトレードオフの表れでした(DETR は学習に500エポックを要しました)。

DiT は ImageNet 256×256 のクラス条件付き画像生成で、FID(Fréchet Inception Distance、生成画像と本物画像の特徴分布の距離。低いほど本物に近い)を当時の最良値である 2.27 まで下げました(DiT-XL/2)。アブレーションでは、条件注入の方式を比較し、AdaLN-Zero が in-context 条件付けやクロスアテンション注入より一貫して低い FID を出すことを示しました。これが「時刻・クラスのような全体条件は変調で注入するのが良い」という設計指針の根拠です。

自動運転のエンドツーエンドでは、UniAD(Hu et al., 2023, CVPR)が nuScenes データセットで、知覚・予測・計画をクエリのアテンションで連結する設計により、計画の衝突率(planning collision rate、生成軌道が他物体とぶつかる割合。低いほど安全)を従来手法比で大きく下げたことを報告しました。アブレーションでは、上流タスク(検出・追跡・地図・占有・動き予測)のクエリを下流の計画に渡すほど計画性能が上がることを示し、「中間特徴を潰さずクエリで受け渡す」設計の価値を定量化しました。

これらに共通するのは、「平均プーリングで空間や系列を1本に潰す」素朴な方式に対し、クロスアテンションで選択的に読むと精度が上がる一方、計算量(クエリ数×キー数の内積)とメモリが増えるというトレードオフです。

## メリット・トレードオフ・限界

メリット。BEV の空間構造(数万マス)を潰さず、軌道の各時刻が必要な場所だけを選んで読めます。時刻・履歴のような全体条件と空間条件を別経路(AdaLN とアテンション)に分けるので役割が明確で学習が安定しやすく、マルチヘッドにより複数の手がかりを同時に拾えます。クエリ数を未来時刻数に対応させる設計は、出力長を柔軟に変えられる利点もあります。

トレードオフと限界。計算量とメモリがクエリ数×キー数に比例し、BEV 解像度を上げるほど重くなります。200×200=40,000、450×300=135,000 とマス数が増えると、アテンション行列がボトルネックになります。これを緩和するのが Deformable Attention(Zhu et al., Deformable DETR, 2021, arXiv:2010.04159)で、全マスを見る代わりにクエリごとに少数の参照点だけをサンプリングして見ます。計算量を大幅に削減しつつ収束も速め、BEVFormer などの自動運転モデルで広く採用されています。本トピックの素朴な全結合クロスアテンションは、解像度が上がると Deformable 系への置き換えが課題になります。

もう1つの限界は、アテンションは学習が必要で、初期は「どこを見ればよいか」が定まらず収束に時間がかかる点です。DETR の500エポックはその典型例でした。さらに、キーが全マスに及ぶため、無関係なマスへの注目を抑える正則化や、参照位置を物理座標に紐づける幾何学的な事前知識(positional encoding を BEV 座標で与えるなど)が、性能を引き出す上で重要になります。未解決の研究課題としては、長い未来時刻(クエリが多い)と高解像度 BEV(キーが多い)を両立しつつリアルタイムで動かす効率化が挙げられます。

## 発展トピック・研究の最前線

最近の流れは3つあります。1つ目は効率化で、Deformable Attention や、参照を疎にする手法(sparse query)により、高解像度 BEV でも実用速度を出す研究が進んでいます。2つ目は時間方向の統合で、複数フレームの BEV を時系列にアテンションで参照し動的物体の動きを捉える BEVFormer 系の発展です。3つ目はマルチモーダル化で、テキスト指示(「次の交差点を右」)やナビゲーション目標をクエリ側の条件に加え、言語条件付き軌道生成へ広げる研究です。いずれも「クエリが選択的に情報源を読む」という本トピックの骨格を保ったまま、読む対象と効率を拡張する方向に進んでいます。
