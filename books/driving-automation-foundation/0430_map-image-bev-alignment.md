# 地図BEVと画像BEVの座標整合

## ひとことで言うと
地図の特徴マップ(map BEV)と画像の特徴マップ(image BEV)を「同じマス目には同じ場所の情報が入っている」という前提のもとで足し合わせたり注目し合ったりするのが、残差融合や cross-attention 融合です。ところがこの前提が成り立つのは、画像 BEV が実距離に紐づいた本物の鳥瞰グリッドになっているときだけです。特徴を小さな解像度に圧縮しただけの「抽象的な特徴マップ」では、マス目と現実の場所の対応が失われていて、同じインデックスのマスが同じ地点を指す保証がありません。これを見落とすと、融合の式は動くけれども意味のない地点どうしを混ぜてしまう、という落とし穴に陥ります。

## 直感的な理解
2枚の地図状の特徴を重ね合わせる、と想像してください。1枚目は画像から作った特徴、2枚目は地図から作った特徴です。重ね合わせが意味を持つのは、2枚が同じ縮尺・同じ向き・同じ原点でそろっているときだけです。1枚目の左上が「自車の前方10m・右5m」を指し、2枚目の左上も同じ「前方10m・右5m」を指していれば、左上どうしを足すのは正しい。しかし2枚目の左上が実はまったく別の地点を指していたら、足し算は「前方10mの画像情報」と「どこか別の場所の地図情報」を混ぜることになり、現実には筋の通らない合成になります。

具体例で考えます。道路の左側に歩行者がいるとします。画像 BEV ではその歩行者の特徴が左寄りのマスに乗ります。地図 BEV で同じ左寄りのマスが「左の歩道」を指していれば、「左の歩道に歩行者がいる」という整合した融合ができます。ところが地図 BEV の左寄りマスが実は右車線を指していたら、融合結果は「右車線に歩行者」という誤った合成になってしまいます。位置対応がない融合は、こうした取り違えを静かに起こします。エラーも出ず、形(テンソルの shape)も合っているので、気づきにくいのが厄介です。

## 基礎: 前提となる概念
「BEV 特徴マップ」は、縦横に並んだマス目(grid cell)それぞれに数値ベクトルが入った `(C, H, W)` のテンソルです。`C` はチャンネル数(各セルのベクトルの次元)、`H` と `W` は鳥瞰平面の縦横のマス数です。

「メトリック BEV(metric BEV)」とは、各マスが実距離に紐づいた本物の鳥瞰グリッドのことです。「メトリック」は「計量的、実寸に基づく」という意味です。たとえば前後 ±50m、左右 ±25m の範囲を `200 × 100` マスで切れば、1マスが 0.5m 四方に対応し、マス `[i, j]` が「自車基準で前方 x メートル・横 y メートル」と物理的に決まります。この範囲指定を `pc_range`(point cloud range、点群が覆う実距離範囲)と呼ぶことがあります。

「抽象特徴マップ(abstract feature map)」とは、画像をネットワークに通して空間方向に圧縮(プーリングや畳み込みの stride)した結果の、小さな `(C, h, w)` テンソルです。たとえば `8 × 8` まで縮めると、各マスは「元画像のおおまかなどこか」を表しますが、それが実空間の何メートル地点かという物理的な意味は持ちません。圧縮の過程で、カメラ投影の幾何が捨てられているからです。

「座標整合(spatial alignment)」とは、2つの特徴マップで、同じインデックス `[i, j]` のマスが現実の同じ場所を指している状態を意味します。本トピックの核心は、この整合がいつ成り立ち、いつ崩れるかを見極めることです。

## 仕組みを詳しく
画像 BEV を作る方式は複数あり、座標整合が成り立つかどうかは方式によって変わります。代表的に2つのケースを対比します。

ケース A: メトリック BEV を明示的に構築する方式(整合する)。

カメラの投影行列(projection matrix、3次元空間の点が画像のどこに写るかを表す行列)と `pc_range` を使って、実距離の鳥瞰グリッドを構築します。各マス `[i, j]` が「自車基準で前方 x メートル・横 y メートル」と物理的に確定します。代表的な実現法は次節で述べる LSS や BEVFormer です。

このとき地図エンコーダ側も、地図を同じ `pc_range`・同じ解像度 `(H, W)` の BEV として描けば、両者のマス `[i, j]` が同じ実座標を指します。だから残差融合 `image_bev[i,j] + α · map_bev[i,j]` も、cross-attention のマス対応も、幾何的に正しく機能します。

```
メトリック BEV (整合あり)
  image_bev[i,j] ─┐
                  ├─ 同じ実座標 (前方 x m, 横 y m) ── 融合は幾何的に正しい
  map_bev[i,j]  ─┘
```

ケース B: 特徴を圧縮して抽象特徴マップにする方式(整合しない)。

各カメラの特徴を空間方向にプーリングして、たとえば `8 × 8` の抽象特徴を作ります。ここで重要なのは、この `8 × 8` のマス `[i, j]` は「実距離で前方 x メートル」という意味を持たない、ただの圧縮特徴である点です。投影行列も `pc_range` も使っていないので、マスと実座標の対応がありません。

このモードで地図エンコーダ出力を同じ `8 × 8` にそろえても、地図側の `8 × 8` は実座標に紐づいた本物の BEV、画像側の `8 × 8` は実座標と無関係な抽象特徴です。両者のマス `[i, j]` が同じ場所を指す保証はありません。したがって残差融合の足し算も cross-attention のマス対応も、本来想定された「同一地点どうしの融合」にはなりません。

```
抽象特徴 8x8 vs メトリック 8x8 (整合なし)
  image_bev[i,j] = 実座標と無関係な圧縮特徴
  map_bev[i,j]   = 実座標に紐づく地図
   ──> 同じ [i,j] でも別々の場所 ──> 足し算は意味のない混合
```

ただし、融合「機構」自体は学習で動きます。ネットワークが「この抽象特徴とこの地図特徴を混ぜると損失が下がる」と学べば、空間対応がなくても一定の効果は出ることがあります。失われているのは「マス単位の物理的な位置対応」という、残差・cross-attention 融合の本来の動機(同一地点の情報を強め合う)です。融合をただの「学習可能なブレンド」として使うなら動くが、地図の位置情報を幾何的に活かすという旨味は出ない、という整理になります。

before / after でまとめます。

```
メトリック BEV モード:
  image_bev[i,j] <-> map_bev[i,j] が同じ実座標 ──> 融合は幾何的に正しい(整合あり)

圧縮 8x8 モード:
  image の [i,j] は実座標と無関係 / map の [i,j] は実座標
  ──> 位置対応なし(整合なし)。学習ブレンドとしては動くが本来の動機は喪失
```

この区別が実務上やっかいなのは、どちらのモードでも出力の shape が同じ(たとえば両方 `8 × 8`)になりうるため、コードを読んでもエラーが出ず、整合が崩れていることに気づきにくい点です。「shape が合うから足せる」は「幾何的に正しい」を意味しません。

## 手法の系譜と主要論文
メトリック BEV を作る方法論の系譜を追います。

出発点は LSS、すなわち Lift, Splat, Shoot(Philion & Fidler, ECCV 2020, arXiv:2008.05711)です。各カメラ画素について深さの確率分布を推定し、画素特徴をその深さに沿って3次元空間へ「持ち上げ(lift)」、鳥瞰グリッドへ「撒く(splat)」ことで、メトリックな BEV 特徴を作ります。これにより各 BEV セルが実空間の位置に対応します。本トピックでいう「実座標に紐づいた本物の BEV」を作る古典的な方法論で、メトリック BEV モードが整合を持てるのは、この種の明示的な投影に基づくからです。

次に BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)が、学習可能な BEV クエリと、カメラ投影に基づく spatial cross-attention で BEV を構築しました。BEV グリッドの各セルが実空間の位置に対応するよう設計されており、各セルが「自分の実座標が画像のどこに写るか」を投影で計算し、その画像位置の特徴を集めてきます。さらに時間方向の temporal self-attention で過去フレームの BEV も取り込みます。各セルが実座標に固定されているため、地図など他の BEV 情報と空間的に整合させやすいのが利点です。

BEVFusion(Liu et al., ICRA 2023, arXiv:2205.13542)は、カメラと LiDAR をそれぞれ BEV にしてから共通 BEV 空間で融合します。融合が成り立つのは、両者を同じ `pc_range`・同じ解像度のメトリック BEV にそろえてあるからです。これは本トピックの主張「整合がないと融合の意味が薄れる」の裏返しの好例で、整合をきちんと作り込んだから融合がうまくいく、という事例です。

残差ゲート自体の系譜は ReZero(Bachlechner et al., UAI 2021, arXiv:2003.04887)などゼロ初期化残差の研究につながります([ゼロ初期化残差によるマップ融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0464_umpe-residual-map-fusion) 参照)。重要なのは、ゲートで「混ぜる量」を安全に学べることと、「混ぜる相手が幾何的に正しい地点か」は別問題だということです。前者を解いても後者は自動では解けません。

これらの論文に共通する前提は「BEV 融合は、関与する特徴マップが共通の実座標系に固定されてはじめて幾何的に意味を持つ」というものです。本トピックは、その前提が圧縮特徴モードでは満たされない、という注意喚起です。

## 論文の実験結果(定量データ)
整合のある BEV 構築が性能に効くことは、上記の系譜の論文で定量的に示されています。

BEVFormer は nuScenes(自動運転の標準ベンチマーク、6カメラ + LiDAR の大規模データセット)の3次元物体検出で、報告によりおよそ 56.9 の NDS(nuScenes Detection Score、検出の正確さと位置・速度などの推定誤差を総合した指標、高いほど良い)を達成し、当時のカメラのみ手法の最高水準を更新しました。NDS が高いということは、物体を「正しい実座標に」置けているということで、これは BEV セルが実空間と整合している効果の現れです。アブレーションとして、投影に基づく spatial cross-attention を単純なグローバル attention に置き換えると性能が落ちることが示されており、「セルを実座標に固定する」設計の寄与が確認できます。

BEVFusion は nuScenes 検出で、報告によりおよそ 70 前後の NDS を達成し、カメラ単独・LiDAR 単独を大きく上回りました。これは「両モダリティを同じメトリック BEV にそろえてから融合した」効果です。融合前に座標系をそろえる手順を省くと、異なる座標の特徴が混ざって性能が劣化するため、整合が融合の前提であることが裏付けられます。

LSS は当初、複数カメラからのセマンティック BEV セグメンテーション(鳥瞰平面の各マスが道路・車両・歩道のどれかを分類する課題)で、画素を実空間へ正しく持ち上げる枠組みの有効性を示しました。深さ分布の推定品質が BEV の整合精度に直結することも報告されており、整合は「作ろうとして初めて得られる」ものであって、特徴を圧縮しただけでは得られないことを示唆します。

これらの数値の意味を初心者向けに補うと、NDS や mAP(mean Average Precision、検出の平均精度)が高いとは「物体や道路要素を、現実の正しい場所に正しく置けている」ということです。座標整合が崩れた特徴で融合すると、置き場所がずれるため、これらの指標は上がりません。逆に言えば、これらの指標が上がる手法はいずれも整合を丁寧に作っています。

## メリット・トレードオフ・限界
整合を保った場合のメリットは明快です。map BEV と image BEV を同一地点どうしで融合でき、地図の位置情報が正しく効きます。残差融合も cross-attention 融合も本来の幾何的意味を持ちます。結果として地図の事前情報が、正しい場所に正しく作用します。

トレードオフと限界もあります。整合を得るにはメトリック BEV の構築が必要で、投影行列や `pc_range` の準備、そして深さ推定や spatial cross-attention といった重い計算コストがかかります。一方、特徴を圧縮するだけの抽象特徴モードは軽量ですが座標整合がなく、地図融合の効果が空間的には保証されません。さらに厄介なのは、整合の有無が出力 shape に現れないため、見た目が同じでも内部で全く違う、という点です。`8 × 8` 同士だからと安易に足すと前提が崩れていることに気づけず、誤用しやすい。研究上は、整合がない軽量モードでも地図融合がどれだけ効くのか、あるいは効かないのかを定量的に切り分けること、そして軽量さと整合を両立する中間的な BEV 構築法を探ることが、未解決の課題として残ります。

## 発展トピック・研究の最前線
整合を保ちつつ軽量化する方向として、粗いメトリック BEV を低解像度で作り、必要な領域だけ高解像度化する適応的グリッドや、深さ推定を簡略化しつつ投影の幾何だけは保持する手法が研究されています。また、地図と画像を同じ座標系の BEV にそろえる前処理(地図のラスタ化やポリラインのレンダリングを `pc_range` に合わせる)の自動化も実務上の論点です。位置対応がない融合がそれでも効く理由を解析する研究は、attention が暗黙に空間対応を学習しうるかという問いにつながり、解釈可能性の研究領域とも接します。
