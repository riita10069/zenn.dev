# 自転車モデルによる軌道デコード

## ひとことで言うと

軌道生成モデルの中には、出力を「各時刻の (加速度, 曲率) のペア」という制御量の系列で表すものがあります。これは人間が見ても車がどこを通るのか分かりません。そこで、この制御量を時間に沿って 1 ステップずつ積み上げて(積分して)、速度 → 向き (heading) → 位置 (x, y) の順に計算し、「車が実際に通る地図上の点 (waypoint)」へ変換します。この変換に使うのが自転車モデルという単純化した車の運動方程式です。制御量表現と位置表現を橋渡しする、地味だが不可欠なデコード処理です。

## 直感的な理解

カーナビが「この先 50 m を右へ」と言うのは位置の言葉です。一方、運転者の手足は「アクセルをこれだけ踏む」「ハンドルをこれだけ切る」という制御の言葉で動きます。両者は別の言語で、間に「車という物理系」が挟まっています。アクセルとハンドルをどう操作すれば車がどこを通るかは、車の運動方程式を時間に沿ってたどらないと分かりません。

軌道デコードはまさにこの翻訳です。モデルが「アクセルとハンドルの指示列」を吐いたとき、それを車の運動方程式に通して「実際に通る道筋」へ変換します。なぜわざわざ制御の言葉で出力するのかというと、その方が「物理的に無理な動き(横にワープする、瞬間的に直角に曲がる)」を最初から排除できるからです。位置を直接吐くと、隣り合う点が物理的につながらない非現実的な軌道を平気で出してしまうことがあります。制御量で出して運動方程式を通せば、出てくる軌道は必ず「ある車が実際に走れる軌道」になります。

ただし翻訳には副作用があります。1 ステップずつ積み上げる方式なので、序盤の小さな誤差が後半に効いてきて、終端ほどズレが溜まります。これは積分という操作の宿命で、後で限界として詳しく触れます。

## 基礎: 前提となる概念

登場する物理量を、知らない人向けに定義します。

- 加速度 (acceleration, 単位 m/s²): 速度の変化率。正ならアクセル(加速)、負ならブレーキ(減速)。
- 曲率 (curvature, 単位 1/m あるいは rad/m): 軌道がどれだけ急に曲がるか。直進なら 0、半径 R の円弧を描くなら 1/R。半径が小さい(急カーブ)ほど曲率は大きい。
- heading(ヘディング、進行方向角、単位 rad): 車が向いている方向。x 軸からの角度。
- waypoint(ウェイポイント): 軌道上の通過点 (x, y)。「道筋を表す点列」のこと。

積分 (integration) の意味も噛み砕きます。加速度が分かっていて速度を知りたいとき、速度は「加速度を時間でだんだん足していったもの」です。連続的には速度 = ∫加速度 dt ですが、計算機では時間を細かい刻み dt(例 0.1 秒)に区切り、`速度 ← 速度 + 加速度·dt` と繰り返し足します。これが数値積分(オイラー法)です。同様に、向きは曲率から、位置は速度と向きから積み上げます。デコードはこの 3 段の積分を連鎖させる処理です。

自転車モデル (kinematic bicycle model) とは何か。四輪車は前後左右にタイヤがありモデルが複雑なので、これを「前輪 1 つ・後輪 1 つの自転車」とみなして単純化します。車の状態を「位置 (x, y)・向き (heading)・速度 (speed)」の 4 つだけで表し、タイヤの横滑り(スリップ)は無いと仮定します。通常の走行速度域では、この単純化でも実車にかなり近い軌道が得られることが知られています(後述の Kong 論文)。

なぜ位置でなく制御量で出力するのか、利点をもう一段。制御量(加速度・曲率)は車のスケール(現在速度)に依存しない相対的な量なので、速い場面でも遅い場面でも同じ表現で扱えます。また、加速度や曲率に上限を課せば「物理的に出せない急操作」を学習・生成の段階で抑え込めます。欠点は、評価も可視化も位置の上で行うため、必ずデコードが要ること、そして積分誤差が累積することです。

## 仕組みを詳しく

典型的なデコード関数は、入力 `[..., T*2]`(平らな (加速度, 曲率) ペア列、T はタイムステップ数)を `[..., T, 2]`(各時刻の (x, y) waypoint)に変換します。中身は自転車モデルの逐次積分です。dt を時間刻み(例 0.1 秒)、初期速度を v0(例 5.0 m/s)とすると:

```
状態を初期化:  x=0, y=0, heading=0, speed=v0
各タイムステップ t = 0..T-1 について:
  speed   = clamp(speed + accel[t] * dt, min=0)    # 速度を加速度で更新。負(バック)にしない
  heading = heading + curvature[t] * speed * dt     # 向きを曲率と速度で更新
  x       = x + cos(heading) * speed * dt           # 向いている方向へ前進
  y       = y + sin(heading) * speed * dt
  (x, y) を記録
```

ステップごとの意味:

1. 速度の更新: 前の速度に「加速度 × 時間」を足す。`clamp(..., min=0)` は速度が負にならないようにする処理(後退を許さない単純化)。
2. 向きの更新: 曲率 × 速度 × 時間。曲率が一定でも、速いほど同じ時間で大きく向きが変わる、という物理を表す(速い車ほど同じハンドル切れ角でも軌道が大きく曲がる、ではなく、ここは「曲率 = 1/半径」を弧長で積分しているため速度が入る)。
3. 位置の更新: 向いている方向 (heading) へ `速度 × 時間` だけ進む。前進量を cos と sin で x 成分・y 成分に分解。

数値例で正しさを確認します。v0=5.0 m/s、ある時刻で accel=0, curvature=0(まっすぐ等速)なら、heading は 0 のまま、x は `cos(0)·5·0.1 = 0.5 m` 増え、y は `sin(0)·5·0.1 = 0`。0.1 秒で 0.5 m 前進は 5 m/s = 時速 18 km と一致し、積分が正しいと分かります。

曲率を小さな正値(例 0.01)にすると、毎ステップ heading が `0.01·5·0.1 = 0.005 rad` ずつ増え、軌道が緩やかに左へカーブします。1 秒(10 ステップ)で約 0.05 rad ≒ 2.9 度向きが変わる計算です。曲率を大きくすれば急カーブになります。

tensor shape の流れ(T=64 の例):

```
入力 trajectory : [B, 128]      (B はバッチ数, 128 = 64*2)
  → reshape       [B, 64, 2]     (加速度と曲率に分離)
  → 逐次積分       [B, 64, 2]     (各時刻の x, y waypoint)
```

座標系の規約に注意が要ります。実装によって「x が前方・y が左」「y が前方・x が右」など取り方が異なります。同じ「制御量 → 位置」の積分でも、評価用と可視化用で座標規約が分かれることがあり、片方の感覚でもう片方を読むと符号や軸が食い違って混乱します。デコード関数を扱うときは、まずどの座標系かを最初に確認するのが鉄則です。

なぜこの設計か。自転車モデルは、状態 4 変数・更新 4 式というミニマルさで、微分可能(各演算が cos/sin/加減乗で滑らか)なため学習ループに組み込めます。生成器を制御量で学習し、デコードを微分可能にしておけば、位置ベースの損失(ADE など)を制御量側まで誤差逆伝播できます。これが「制御量出力 + 微分可能デコード」という近年の end-to-end 設計が好まれる理由です。

## 手法の系譜と主要論文

自転車モデル自体は古典的な車両運動学です。Kong, Pfeiffer, Schildbach & Borrelli ("Kinematic and Dynamic Vehicle Models for Autonomous Driving Control Design", IEEE Intelligent Vehicles Symposium 2015) は、運動学的自転車モデルと、タイヤの滑りまで考慮する動力学モデルを比較し、通常の走行速度域では単純な運動学モデルでも制御設計に十分な精度が得られることを示しました。計算が軽く実装が容易な点が利点で、トレードオフは高速・低摩擦(雪道など)でタイヤ横滑りを無視するため精度が落ちることです。

Polack, Altché, d'Andréa-Novel & de La Fortelle ("The Kinematic Bicycle Model: a Consistent Model for Planning Feasible Trajectories for Autonomous Vehicles?", IEEE IV 2017) は、運動学モデルがどの条件まで「実行可能な軌道」を保証するかを定量的に分析し、横加速度がおおむね 0.5g を超えない範囲なら運動学モデルで生成した軌道は実車でも追従可能だと示しました。計画段階で運動学モデルを使う妥当性に理論的根拠を与えた研究です。

軌道を「位置の系列」でなく「制御入力(加速度・曲率/操舵)の系列」で表し、運動学モデルで積分して位置に直す設計は、近年の end-to-end 自動運転で一般的になりました。UniAD (Hu et al., "Planning-oriented Autonomous Driving", CVPR 2023 Best Paper, arXiv:2212.10156) のような計画指向アーキテクチャや、GoalFlow (Xu et al., arXiv:2503.05689, 2025) のような生成的計画手法でも、出力をそのまま位置にするか制御量にするかが設計選択として議論されます。制御量表現の利点は運動学的に実行可能な軌道を生成しやすいこと、欠点は積分誤差が時間とともに累積し後半ほど位置ずれが大きくなることです。

## 論文の実験結果(定量データ)

評価指標の意味。ADE (Average Displacement Error) は予測軌道と正解 (Ground Truth) の各時刻の点間距離の平均(単位 m、小さいほど良い)、FDE (Final Displacement Error) は最終時刻のみの距離(同)です。FDE は終端のズレを直接測るので、積分誤差の累積に敏感な指標です。

運動学モデルの近似精度について、Polack ら (2017) の分析では、横加速度 0.5g 以下の通常走行では運動学自転車モデルと高精度な動力学モデルの軌道差は数十 cm オーダーに収まり、計画には十分とされます。一方、限界走行(高速での急旋回、低摩擦路)では差が m オーダーに拡大し、運動学モデルだけでは追従不能な軌道を出しうると報告されています。これが「通常域では運動学で十分、限界域では動力学が要る」という実務上の使い分けの根拠です。

制御量表現と位置表現の比較は、end-to-end 計画の論文でアブレーションされることがあります。一般的傾向として、制御量(加速度・曲率)を出して運動学デコードを通す方が、位置を直接回帰するより「物理的に滑らかで実行可能な軌道」になりやすく、急なジグザグや非現実的なジャンプが減ります。半面、ADE/FDE の数値だけ見ると、表現による差は小さいかタスク依存で、累積誤差により制御量表現の方が終端 (FDE) で不利になる場合も観察されます。どちらが良いかはデータと評価軸に依存し、一意の勝者はいないというのが現状です。

## メリット・トレードオフ・限界

メリット:

- 制御量(加速度, 曲率)という抽象表現を、評価・可視化できる位置 (x, y) に直せる。
- 自転車モデルは計算が軽く実装が単純で、通常走行域なら十分な精度。
- 演算が微分可能なので、位置ベースの損失を制御量まで誤差逆伝播でき、学習ループに組み込める。
- 位置ベースの正解・指標(ADE/FDE、Waymo/nuScenes など)と整合する。

トレードオフ・限界:

- 逐次積分なので時間方向は並列化しづらい。バッチ方向は並列だが、T ステップを for ループで回すと CPU では遅くなりうる。
- 積分誤差が時間とともに累積し、後半の waypoint ほど位置ずれが大きくなる(FDE に効く)。
- 運動学モデルなのでタイヤの横滑り(高速・低摩擦)を表現できず、限界走行では軌道が実車から乖離する。
- 初期速度や dt を固定値で仮定すると、実データの実際の初速・サンプリング周期と食い違い軌道がずれる。理想は egomotion 履歴から実測初速を与えること。
- 座標規約(どちらが前方か、符号の取り方)が実装ごとに異なり、共有しないと混同しやすい。

未解決・研究課題: 積分誤差の累積を抑えるため高次の積分(RK4 など)を使う、あるいは曲率だけでなく操舵角の変化率まで状態に入れたより忠実なモデルを使う、といった精度向上策がありますが、計算コストと微分可能性とのバランスが課題です。また、学習時に運動学制約をどこまで強く課すか(完全に運動学に縛るか、緩く正則化するか)も設計上の開かれた問いです。

## 発展トピック・研究の最前線

運動学自転車モデルの上位には、タイヤ力・荷重移動まで含む動力学モデル(dynamic bicycle model)があり、高速・限界走行のシミュレーションや制御に使われます。学習ベースでは、微分可能シミュレータの中に車両運動を埋め込み、軌道生成と運動学を一体で最適化する研究が進んでいます。また、ニューラルネットで車両ダイナミクスそのものを学習し(residual dynamics learning)、解析的な自転車モデルの誤差を補正するハイブリッドも登場しています。デコードの観点では、可微分・低コスト・実行可能性保証という 3 条件をどこまで同時に満たせるかが、計画系研究の継続的なテーマです。
