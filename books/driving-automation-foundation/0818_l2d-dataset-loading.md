# L2D データセットの読み込み

## ひとことで言うと

L2D（Learning to Drive）は、実際の運転動画と車の状態をまとめた大規模な運転データセットです。これを学習に使うには、生のデータ（複数カメラの動画と車両状態の数値表）を、モデルがそのまま受け取れる形（テンソル、数値の多次元配列）に変換する読み込み層が必要です。典型的には、ロボット学習ライブラリ（LeRobot など）を経由して、必要なエピソード（運転の1区間）だけを番号で指定して読み込み、画像のリサイズ・正規化や、過去・未来の状態系列の切り出しを行う専用の Dataset クラスを用意します。このノートは、運転データの読み込みがなぜ難しく、どう設計するのか、そして模倣学習データセットの系譜と評価上の限界までを、ゼロから掘り下げます。

## 直感的な理解

料理に例えます。スーパーで買ってきた食材（生の運転データ）は、泥のついた野菜や塊肉のままで、そのまま鍋に放り込めません。皮をむき、決まった大きさに切り、下味をつける——この「下ごしらえ」をして初めて調理（モデルの学習）に使えます。データの読み込み層は、まさにこの下ごしらえを担います。

運転データの下ごしらえが特に難しいのは、データが「重くて多種類」だからです。複数の車載カメラが回す動画は容量が大きく、デコード（圧縮された動画を画像に戻す処理）に時間がかかります。一方で、車両の速度や加速度といった状態は、ほんの数十バイトの数値にすぎません。「軽い数値だけ欲しいのに、毎回重い動画まで全部展開してしまう」という無駄が起きやすく、ここをどう避けるかが性能の鍵になります。さらに、未来の軌道を予測させたいので、「あるフレームの前後に十分な過去・未来がそろっているか」も気にしながらサンプルを切り出す必要があります。

## 基礎: 前提となる概念

用語を一つずつ噛み砕きます。

- 「データセット（dataset）」とは、学習に使うデータの集まりです。運転データセットには「運転中の複数カメラ映像＋車の速度・加速度などの状態」が大量に入っています。

- 「エピソード（episode）」とは、一回の連続した運転区間のことです。データセットは多数のエピソードに分かれており、L2D は欧州の実走行をおよそ10万エピソード規模で、10 FPS（1秒あたり10コマ）で記録していると報告されています。

- 「FPS（frames per second）」とは1秒あたりのコマ数です。10 FPS なら、64コマで6.4秒分になります。

- 「テンソル（tensor）」とは、数値を格子状に並べた多次元配列です。画像は (チャンネル, 高さ, 幅) の3次元テンソル、それを複数枚束ねると4次元、という具合に表します。深層学習フレームワーク（PyTorch など）が扱う基本データ型です。

- 「Dataset クラス」と「DataLoader」。PyTorch では、`dataset[i]` と書くと i 番目のサンプルが取れるよう振る舞いを定義したものを Dataset と呼びます。これを DataLoader に渡すと、複数サンプルをまとめて（バッチにして）並列に効率よく取り出せます。バッチ（batch）とは、まとめて処理するサンプルの束のことです。

- 「正規化（normalization）」とは、画像の画素値を、モデルが学習時に前提とした平均・標準偏差（mean / std）に合わせて変換することです。これをしないと、モデルが見慣れない明るさのデータを渡され性能が落ちます。

- 「模倣学習（imitation learning）／行動クローニング（behavior cloning）」とは、人間の運転（お手本）を真似るように、入力（カメラ画像や過去の状態）から出力（未来の軌道や操作）を学ぶやり方です。運転データセットはこの「お手本」の供給源になります。

## 仕組みを詳しく

読み込み層の使い方は、概念的には次のようになります。

```python
dataset = L2DDataset(
    repo_id="yaak-ai/L2D",     # どのデータセットか
    episodes=[0, 1],           # 読むエピソードを番号で指定（None なら全部）
    backbone_name="swinv2_tiny_window8_256",  # 画像前処理の基準にする骨格モデル名
)
loader = DataLoader(dataset, batch_size=8, shuffle=True, num_workers=4)
```

`episodes` 引数が実務上きわめて重要です。10万エピソード全部を落とすのは現実的でないため、`episodes=[0, 1]` のように「番号で必要な区間だけ」を指定できます。動作確認やテストでは1〜2エピソードだけ読みます。これは「巨大データセットを扱うときは、まず小さな部分集合で正しく回ることを確認してから全体に広げる」という鉄則の実装でもあります。

`dataset[i]` で取れる1サンプルは、典型的には辞書（キーと値の組）で、複数のテンソルとメタ情報を持ちます。代表的な構成例を挙げます。

| キー | 形状の例 | 中身 |
|---|---|---|
| `visual_tiles` | `(7, 3, H, W)` | 7視点（周囲6カメラ＋俯瞰マップ1枚）×（RGB3チャンネル, 高さH, 幅W） |
| `egomotion_history` | `(256,)` | 過去64ステップ × 4信号（速度・加速度・ヨーレート・曲率） |
| `visual_history` | `(896,)` | 逐次推論時に埋める枠（学習時はゼロ初期化のことが多い） |
| `trajectory_target` | `(128,)` | 未来64ステップ × 2信号（加速度・曲率）＝学習の正解 |
| `episode_index` / `frame_index` | int | どのエピソードの何フレーム目か |

数値の意味を補足します。`256 = 64 × 4` は「過去64ステップ（10 Hz なら6.4秒分）それぞれに4つの信号」を一列に並べたもの。`128 = 64 × 2` は「未来64ステップそれぞれに加速度と曲率の2値」で、これがモデルに当てさせたい正解（教師信号）です。`visual_tiles` の7は「6台の周囲カメラ＋1枚の俯瞰（BEV: Bird's-Eye View、鳥瞰図）マップ」を意味します。BEV とは、複数カメラの情報を真上から見下ろした座標系にまとめた表現で、自動運転で広く使われます。

内部処理をステップで分解します。

1. ロボット学習ライブラリで、指定したエピソードのデータセットオブジェクトを作る（版差への対応は [LeRobot バージョン固定問題](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0865_lerobot-version-pinning)）。
2. 画像モデルのライブラリ（timm など）から、指定した骨格モデル（backbone）の標準入力サイズと正規化パラメータ（mean / std）を取り出す。これで「どのサイズにリサイズし、どう色を正規化するか」を、後段のモデルに自動で合わせる。
3. 各エピソードの行範囲を求め、有効なフレームだけをサンプルとして列挙する。「有効」とは、その前に64ステップ分の過去、後ろに64ステップ分の未来がそろっているフレームのこと。端のフレームは過去または未来が足りないので使えない。
4. `dataset[i]` のとき、まず車両状態の表だけを読み、過去・未来の egomotion を計算する。
5. 続いて現在フレームの複数カメラ画像をデコードし、リサイズ・正規化して `(7,3,H,W)` に積む。

ここに性能上の重要な工夫があります。素朴に実装すると、車両状態（たった数十次元のベクトル）を1本取るだけのために、フレームに紐づく全カメラの動画を毎回デコードしてしまい、1サンプルあたり数十秒（報告される実測では7カメラで約35秒）もかかることがあります。これでは学習が始まりません。そこで、車両状態が欲しいだけの場面では動画を一切デコードせず、内部の数値テーブル（状態が列として格納されている部分）から直接読むようにします。動画デコードは、カメラ画像が本当に必要なときだけに限定します。

before/after で性能差を示すと:

- before: 状態1本のために全カメラ動画をデコード → 約35秒/サンプル。学習が事実上始まらないほど遅い。
- after: 数値テーブルから状態だけ抜く → 動画デコードは画像が要るときだけ。実用的な速度に。

この最適化の本質は「I/O の粒度を必要な分だけに絞る」という、データエンジニアリングの普遍的な原則です。重いリソース（動画）と軽いリソース（数値）を分離してアクセスする、という設計判断がボトルネックを解消します。

## 手法の系譜と主要論文

運転データセットと、それを使う模倣学習の系譜をたどります。

- 最初期は ALVINN（Dean Pomerleau, "ALVINN: An Autonomous Land Vehicle in a Neural Network", NeurIPS 1988）です。カメラ画像から直接ハンドル操作を学ぶニューラルネットで、行動クローニングの原点です。提案の核心は「人間の運転を入出力ペアとして学べば、明示的なルールを書かずに運転できる」こと。当時は計算機もデータも小規模で、汎化に限界がありました。

- 大規模実データによる行動クローニングの代表が、NVIDIA の PilotNet（Bojarski et al., "End to End Learning for Self-Driving Cars", arXiv:1604.07316, 2016）です。前方カメラ1枚からステアリングを回帰し、E2E（end-to-end、入力から出力まで一気通貫）学習の実用性を示しました。

- 近年は軌道・行動を直接学ぶ手法が主流になりました。VAD（Bo Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving", ICCV 2023, arXiv:2303.12077）はベクトル化したシーン表現で計画を効率化し、VADv2（Shaoyu Chen et al., "VADv2: End-to-End Vectorized Autonomous Driving via Probabilistic Planning", arXiv:2402.13243, 2024）は軌道を確率分布として扱い、語彙化した軌道集合上の分布を学ぶ方向に進めました。これらは大規模な実走行データを燃料とします。

- データ基盤の系譜として、LeRobot（Remi Cadene ら／Hugging Face, 2024〜）は、ロボット・運転データを統一フォーマットで扱い、動画と状態を効率よくストリーム読みする枠組みを提供しました。動機は、ロボット学習でデータ形式がバラバラだと再利用しにくいから。効果は相互運用性。トレードオフは版間で構成が変わりコードが壊れることがある点で、これが [LeRobot バージョン固定問題](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0865_lerobot-version-pinning) の問題そのものです。

- L2D（yaak-ai, Hugging Face 上で `yaak-ai/L2D` として配布, LeRobot v3.0 形式, 2024〜2025）は、欧州の多数都市・長期間の実走行を10万エピソード規模で集めた大規模運転データセットです。狙いは「実世界の多様な運転を、模倣学習に使える形で大量に提供する」こと。規模と地理的多様性が強みです。

## 論文の実験結果（定量データ）

運転モデルの評価指標と、データセット規模が結果に与える影響を、初心者にも分かる形で説明します。

行動クローニングの評価でよく使う指標が、計画した軌道と人間の実走行軌道の差を測る「L2 変位誤差（L2 displacement error）」と、自車が他の物体や境界にぶつかった割合を測る「衝突率（collision rate）」です。L2 誤差は「予測した位置と本当の位置が平均何メートルずれたか」、衝突率は「シミュレーションや評価区間で何割の場面でぶつかったか」を表します。両方とも小さいほど良い指標です。

オープンループ評価（モデルが出した軌道を、実際には車を動かさず、記録された正解と比べるだけの評価）の代表的なベンチマークである nuScenes 上で、近年の手法は L2 平均誤差をおおむね1メートル前後まで下げてきました。VAD は、それ以前の手法に比べて計画の L2 誤差を下げつつ、推論速度を大幅に向上させた（報告では従来手法の数倍速い）と示しています。VADv2 は軌道を確率分布として扱うことで、衝突率をさらに低減したと報告しています。これらの数値が示すのは、「より良いシーン表現と確率的な扱いが、安全性に直結する衝突率を下げる」という関係です。

データ規模の効果については、行動クローニング全般で「データを増やすほど誤差が下がるが、収穫逓減（増やすほど改善幅が小さくなる）」が観測されています。だからこそ L2D のような10万エピソード級の大規模データに価値がありますが、規模だけでは解けない問題も残ります。後述の評価指標の限界がそれです。

アブレーション的な観点も重要です。読み込み層の最適化（状態だけ数値テーブルから読む）は、モデルの精度には影響しませんが、学習が現実的な時間で回るかどうかを左右します。約35秒/サンプルのままでは、そもそも十分なステップ数を学習できず、上記のような精度に到達できません。つまりデータ読み込みの効率は「精度の指標には現れないが、その精度に到達できるかを決める隠れた前提」です。

## メリット・トレードオフ・限界

メリット:

- 生の運転データ（複数動画＋数値表）を、モデルが直接食えるテンソル辞書に一括変換できる。
- エピソード番号で必要な区間だけ読めるので、テストや動作確認が軽く、段階的にスケールできる。
- 状態を数値テーブルから直接読む最適化により、動画デコードを避け大幅に高速化できる（数十秒/サンプル → 実用速度）。
- 画像モデルライブラリの設定からリサイズ・正規化を自動導出し、骨格モデルに前処理を自動で合わせられる。

トレードオフ・限界:

- ロボット学習ライブラリに強く依存するため、その版変更で壊れやすい（[LeRobot バージョン固定問題](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0865_lerobot-version-pinning)）。
- カメラ画像が必要な場面では結局すべての動画デコードが走り、そこが重い。デコード並列化やキャッシュが別途要る。
- L2D には他車両・歩行者の詳細なアノテーション（注釈）が乏しく、衝突率のような他物体との関係を要する評価指標を直接計算できない。これは大きな限界で、安全性評価には別の手段（シミュレーションや追加アノテーション）が要る。
- `visual_history` のような逐次推論用の枠は、学習時はゼロ初期化のままで中身が埋まらず、実質未使用になりがち。学習と推論で入力の使われ方が食い違う点に注意。
- オープンループ評価そのものの限界として、記録された正解と比べるだけでは、自車の行動が周囲に与える影響（他車が避けてくれる/ぶつかる）を評価できない。クローズドループ（実際に車を動かす）評価が本来は必要。

研究上の未解決課題として、模倣学習の「分布シフト（distribution shift）」問題があります。学習はお手本（人間の運転）の分布で行われますが、推論時にモデルが少しずれた状態に入ると、お手本に無い状況に陥り誤差が雪だるま式に増える（compounding error）。DAgger（Ross et al., 2011）などの対策はありますが、実車での収集コストが高く、完全には解けていません。

## 発展トピック・研究の最前線

最前線では、大規模運転データを使って、行動クローニングを超えた「世界モデル（world model）」を学ぶ方向が活発です。これは未来の観測（次の画像やセンサ値）を予測できるモデルを学び、その内部表現を計画に使うアプローチで、データの自己教師的な活用が進んでいます。

評価面では、オープンループ評価の限界（前述）を補うため、収集データから運転シミュレータを再構成し、クローズドループで安全性を測る研究が増えています。L2D のような大規模実データは、こうしたシミュレータの素材としても価値が見直されています。データ読み込み層は、こうした多様な下流タスク（学習・評価・シミュレータ構築）の共通の入り口であり、その効率と正確さが研究全体の生産性を左右します。
