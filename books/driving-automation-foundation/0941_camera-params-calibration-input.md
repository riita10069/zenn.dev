# カメラパラメータ(キャリブレーション)入力

## ひとことで言うと
「車のどこに、どの向きで、どんなレンズのカメラが付いているか」をモデルに教える数値の束が、カメラパラメータ(キャリブレーション情報)です。これがあると、3次元空間のある点が各カメラ画像のどのピクセルに写るかを計算でき、複数カメラの映像を幾何的に正しく上空図(BEV、鳥瞰図)へまとめられます。校正が無い・間違っていると、複数カメラを統合した知覚は静かに破綻します。マルチカメラ3次元知覚を学ぶうえで最初に押さえるべき土台の概念です。

## 直感的な理解
人間は左右の目の間隔(約6cm)と、それぞれの目がどちらを向いているかを脳が知っているから、両目に映った映像を一つの立体世界へ融合できます。もし片目が斜めに付いていてその傾きを脳が知らなければ、二重に見えてしまうでしょう。

車載のマルチカメラも同じです。前・後・左右・斜めに付いた6〜8台のカメラそれぞれについて「車のどの位置に、どんな角度で、どんな画角で付いているか」を正確に知らない限り、各カメラ画像を一つの統一座標系(自車を中心とした地面のマップ)へ重ね合わせることはできません。カメラパラメータは、この「重ね合わせのための地図情報」です。逆に言えば、3次元知覚モデルが正しく機能するための前提条件であり、ここがずれると後段がいくら高性能でも結果は狂います。

## 基礎: 前提となる概念

用語を順に噛み砕きます。

- **キャリブレーション(calibration)** とは「校正」、つまりカメラの位置・向き・レンズ特性を正確に測って数値化することです。工場出荷時や設置時に、既知のパターン(チェッカーボードなど)を撮って逆算します。具体的には、サイズの分かった格子模様を様々な角度から数十枚撮り、その格子の交点が画像のどこに写ったかを観測値として、内部行列・歪み係数・各カメラの相対姿勢を最小二乗で同時推定します(Zhang の手法、Zhang ICCV 1999 が古典的な基盤)。「測る」と一言で言っても、実際にはこうした最適化問題を解いて得た推定値であり、観測ノイズや格子検出誤差ぶんの不確かさを必ず含みます。この不確かさが後段の射影誤差の源になります。
- カメラパラメータは大きく2種類に分かれます。
  - **内部パラメータ(intrinsics、内部行列 K)**: レンズの焦点距離 fx, fy(ピクセル単位)と画像中心 cx, cy など、そのカメラ単体の「写り方」を表す値。3×3 の行列で、3次元のカメラ座標 [X,Y,Z] を画像のピクセル [u,v] へ落とす役目を持ちます。レンズ歪み(distortion)係数も内部パラメータの一部ですが、行列 K とは別にもつのが普通です。
  - **外部パラメータ(extrinsics)**: そのカメラが世界(または自車)のどこに、どの向きで付いているか。回転 R(3×3)と並進 t(3×1)の組で、4×4 の剛体変換行列としてまとめます。「ワールド座標 → カメラ座標」の変換です。

- **同次座標(homogeneous coordinates)**: 3次元点 [X,Y,Z] を 4次元 [X,Y,Z,1] と書く表現。これにより回転・並進・射影を一つの行列積で表せ、最後に同次成分で割ると画像座標が出ます。この「割り算」が遠近法(遠いものほど小さく写る)を生みます。

射影の流れを式で書くと、ワールド点 Pw を画像ピクセル p へ写す変換は次のようになります。

```
s * [u, v, 1]^T = K @ [R | t] @ [Xw, Yw, Zw, 1]^T
```

ここで K は内部行列(3×3)、[R|t] は外部行列(3×4)、s はスケール(同次成分)です。K @ [R|t] をまとめた 3×4 行列を**射影行列(projection matrix)**と呼びます。BEV系のモデルでは、この射影行列を直接モデルへ渡すことが多くあります。

具体的に内部行列 K の中身を書くと次の形です。

```
      [ fx   0   cx ]
K  =  [  0  fy   cy ]
      [  0   0    1 ]
```

`fx, fy` は焦点距離をピクセル単位で表した値(レンズの焦点距離 mm をセンサーの画素ピッチで割ったもの)、`cx, cy` は光学中心が画像上のどのピクセルに当たるか(理想的には画像中央だが、製造誤差でずれる)。`fx` が大きいほど望遠(狭い画角で遠くを大きく写す)、小さいほど広角です。たとえば横 1600px のカメラで水平画角が約 60 度なら、`fx ≈ 800 / tan(30°) ≈ 1386px` 程度になります。この一本の数字が「画像上の何ピクセルが実世界の何度に対応するか」を決め、射影の倍率を支配します。

なぜ必要か。BEV融合([BEVFormer 風 空間クロスアテンション(SCA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0935_bevformer-spatial-cross-attention))では、BEVの格子マスが3次元空間のどこに当たるかを決め、その3次元点が各カメラ画像のどこに写るかを射影で計算し、そのピクセル位置から画像特徴を拾います。校正が無い・間違っていると、「車の真正面5m先」を計算しても、それが前カメラ画像のどこに写るか分からず、間違ったピクセルから特徴を拾います。エラーは出ず、ただ精度が落ちるので発見が難しい。

さらにデータセットごとにカメラの配置も台数も違います。あるデータセットは7台、別のは8台、取り付け位置も向きも別です。だから複数データセットをまたいで学習する場合([カメラ数が異なる多データセット学習](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1212_multi-dataset-training-homogeneous-batch))は、サンプルごとに異なる校正を渡すことが必須になります。共通の固定値では成立しません。

## 仕組みを詳しく

マルチカメラBEVモデルで校正情報を扱う典型的な設計を、テンソルの形(shape)つきで説明します。

### 渡される形

射影行列をまとめて渡す場合、形は次のようになります。

```
camera_params: [B, V, 3, 4]
```

B はバッチ(同時に処理する枚数)、V はカメラ台数、3×4 が各カメラの射影行列です。この 3×4 は内部行列 K(3×3)と外部行列 [R|t](3×4)を掛け合わせた「自車座標 → 画像ピクセル」の一括変換です。実装によっては K と extrinsics を別々に渡し、内部で掛け合わせる流儀もあります。どちらでも数学的には等価です。

### 伝播の経路

カメラパラメータはモデルの一番外側から BEV融合の射影計算まで透過的に渡されます。途中のモジュール(特徴抽出やビュー統合の前段)は中身を加工せず素通しします。BEV以外の融合方式(単純連結 concat、画像横断アテンション cross-attn)は射影を使わないため、校正は任意(Optional)引数とし、渡さなくても動く後方互換の設計が一般的です。

### 射影の計算

BEVの参照点 `[N, num_z, 3]`(N 個のBEVセル × 高さ方向のサンプル点 × 3次元座標)に射影行列を掛け、同次座標で割って2次元のピクセル位置を得ます。

```
pix_homogeneous = proj @ ref_point_homogeneous   # [.., 3]
pix = pix_homogeneous[:2] / pix_homogeneous[2]    # 同次成分で割る → [u, v]
```

本物の校正があるときは、射影結果が画像範囲内(0 ≤ u < W, 0 ≤ v < H)かをチェックし、範囲外のセル・ビューを除外する**可視マスク(visibility mask)**を作ります。あるBEVセルが前カメラには写るが後カメラには写らない、という事情をこのマスクで扱います。

### 数値例

1マス0.4mのBEVグリッドで、自車の真正面8m先(20マス先)の点を、焦点距離 fx=560px、画像中心 cx=480px の前カメラへ射影するとします。点がカメラ光軸上ならピクセルは中心 cx 付近に来ますが、横に2mずれていれば u ≈ cx + fx * (2 / 8) = 480 + 140 = 620px のように、深さ8mと横ずれ2mの比で決まります。深いほど(分母が大きいほど)横ずれの画素変位は小さくなる。これが遠近法です。

### 校正が無いときのフォールバック

校正データが手元に無い段階でもモジュールを動かして学習を回せるよう、学習可能な疑似射影行列(pseudo projection、形 `[V, 3, 4]`)を代わりに使う設計が取られることがあります。中身はランダム初期化された学習可能な値で、本物の幾何ではありません。

重要な注意: これは本物の校正の代わりにはなりません。疑似射影の出力は値域が定まらないため、sigmoid などで 0〜1 に押し込むフォールバックを挟む実装もありますが、あくまでテスト・アブレーション(機能を部分的に外して効果を測る実験)用です。実運用では本物のカメラパラメータを渡すべきです。ランダム入力 + 偽射影の組み合わせでは損失が NaN(非数)になることもあり、これはコードの欠陥ではなく数値的に当然の帰結です。

### 座標規約の落とし穴

校正の座標規約はデータセットごとに違います。X軸が前か右か、単位が m か pixel か、原点が後輪車軸中心か車体中心か、ヨー角の正の向きはどちらか。これらを揃えずに射影すると、結果が静かにずれます(例: 軸を取り違えると左右反転)。マップ系BEVのレンダラ規約(pixels_per_meter、原点セルの行列インデックス)と知覚系BEVの校正を一致させることは、地味だが重大な工程です。詳しくは [マップBEVと知覚BEVの違い](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1044_map-bev-vs-perception-bev)。

## 手法の系譜と主要論文

校正情報の使い方には大きく「明示射影派」と「暗黙学習派」があり、その対比が学びになります。

- **LSS (Lift-Splat-Shoot、Philion & Fidler, ECCV 2020, arXiv:2008.05711)**: 各ピクセルに深度の確率分布を予測させ、内部・外部パラメータでその点群を3次元へ「持ち上げ(lift)」、BEVへ「ばらまく(splat)」。校正を陽に使ってカメラ特徴を3次元配置する初期の代表例。明示射影派の源流。

- **DETR3D (Wang et al., CoRL 2021, arXiv:2110.06922)**: 3次元のオブジェクトクエリを各カメラへ射影して特徴をサンプリングするスパースな手法。校正行列が前提。クエリ駆動の3次元検出を切り開いた。

- **BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270)**: BEVクエリを各カメラへ射影し、空間クロスアテンションで特徴を集める。校正情報を陽に使う代表例で、時間方向の自己アテンションも加えた。提案理由は複数カメラの幾何整合を正確に取ること。トレードオフは校正への依存で、校正誤差がそのまま性能劣化につながる。

- **PETR (Liu et al., ECCV 2022, arXiv:2203.05625)**: 射影行列を陽に掛ける代わりに、各ピクセルにカメラ幾何由来の3D位置エンコーディング(3D Position Embedding)を足し、アテンションで暗黙に対応づける。明示射影の離散化誤差や複雑な座標変換を避けたい動機。実装が単純化される一方、幾何の解釈性はやや下がる。校正情報の使い方が BEVFormer と対照的で、暗黙学習派の代表。

- **キャリブレーション誤差のロバスト性研究**: 外部パラメータの微小なずれ(取り付け角の数度の誤差)がBEV系の性能を大きく下げることが各所で報告されており、車載では走行中の振動・温度変化による校正ドリフトに対し、オンライン校正や校正ノイズに頑健な学習(キャリブレーション拡張)が課題とされます。これは疑似射影が本番に使えない理由とも通じます。

## 論文の実験結果(定量データ)

3次元検出BEV系の標準ベンチマークは nuScenes で、主指標は **mAP**(mean Average Precision、検出の正確さ、高いほど良い)と **NDS**(nuScenes Detection Score、mAP に位置・大きさ・向き・速度・属性の誤差を統合した総合指標、高いほど良い)です。

- BEVFormer は nuScenes test で NDS およそ 0.569、mAP およそ 0.481 を報告し、当時の単フレームBEV手法を上回りました。時間方向の自己アテンション(過去BEVの利用)を抜くアブレーションでは、特に速度推定(mAVE、平均速度誤差、低いほど良い)が悪化し、NDS が数ポイント下がることが示されています。時間情報が速度推定に効くという定量的根拠です。

- PETR は nuScenes test で NDS およそ 0.504、mAP およそ 0.441(マルチスケール・大規模backbone構成ではさらに上)を報告。明示射影を使わず3D位置エンコーディングだけで競合する性能に達したことが、暗黙学習派の有効性を示しました。

- 校正ロバスト性の研究では、外部パラメータの回転に 1〜2度のノイズを注入しただけで、BEVセグメンテーションの IoU(Intersection over Union、重なり率、高いほど良い)が数ポイント〜十数ポイント低下する例が報告されており、明示射影派が校正誤差に敏感であることを定量的に裏づけます。暗黙学習派(PETR系)はこの面でやや頑健とされますが、根本的に校正情報そのものを使う以上、誤差から完全には自由ではありません。

- 深度教師あり派の **BEVDepth (Li et al., AAAI 2023, arXiv:2206.10092)** は、LSS 系の暗黙的な深度予測が実は不正確であることを定量的に暴き、LiDAR から作った正解深度で深度ヘッドを明示的に教師ありにすることで nuScenes val の NDS を当時の LSS 系から大きく引き上げました(報告では NDS 0.47 前後から 0.50 超へ)。ここで重要なのは、深度を正しく推定するには内部・外部パラメータを深度ネットの入力に明示的に与える(camera-awareness)ことが効いた点で、校正情報をネットへ「特徴として」流し込むことの価値を示しています。校正を射影計算だけに使うのではなく、ネットの条件付け入力にもするという発想です。

これらの数値は「校正が正しければ高精度、ずれると静かに崩れる」という本トピックの核心を、ベンチマーク上で具体的に示しています。指標の感覚をつかむため補足すると、nuScenes の mAP は中心距離(物体中心の位置誤差)を 0.5/1/2/4m の4閾値で判定した平均で、0.481 という値は「半分弱の物体を緩めの閾値まで含めて当てられている」水準です。NDS の 0.569 と mAP 0.481 の差は、位置・向き・速度などの属性誤差ぶんを反映しており、この差が小さいほど検出ボックスの質(向きや速度まで正確)が高いことを意味します。校正がずれると、まず位置・向きの誤差(mATE, mAOE)が増え、NDS が mAP より先に悪化するという形で症状が現れます。

## メリット・トレードオフ・限界

- メリット: 3次元点とカメラピクセルの正確な対応がとれ、複数カメラを幾何的に正しくBEVへ統合できる。
- メリット: サンプルごとに異なる校正を渡せるため、多カメラ構成が混在する複数データセット学習が成立する。
- メリット: 疑似射影フォールバックにより、校正データが無い初期段階でもモジュールを動かして実装を検証できる。
- トレードオフ: 校正が間違っていると射影が狂い、性能が静かに劣化する。エラーが出ないので発見が難しい。
- トレードオフ: 疑似射影は本物の幾何でないため実運用に使えず、ランダム入力では NaN を生じうる。
- トレードオフ: 座標規約(軸の向き・単位・原点・ヨー符号)がデータセットごとに違い、揃える前処理が不可欠。
- 限界・未解決課題: 走行中の校正ドリフトに対するオンライン校正、校正誤差に頑健な学習、固定形フォールバックと可変カメラ台数([BEV融合の num_views ランタイム動的化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1030_bev-runtime-dynamic-num-views))の両立は、いずれも研究上の課題として残っています。

## 発展トピック・研究の最前線

近年は、明示射影と暗黙学習の中間として、深度を明示的に教師あり学習する BEVDepth、射影を高速・正確にする BEVPoolv2、時間融合を加えた BEVFormer v2 / SOLOFusion など、校正幾何を効率よく使う方向の改良が続いています。また校正を学習で補正する(自己校正)、複数フレーム間で校正の一貫性を制約する研究もあります。Transformer 一辺倒からスパースクエリ(Sparse4D 系)への揺り戻しもあり、いずれも「校正幾何をどこまで信じ、どこまで学習で補うか」の設計選択に帰着します。
