# 運動快適度スコア

## ひとことで言うと
予測した軌道が「乗っていて快適か(急発進・急ブレーキ・急ハンドルがないか)」を測る指標です。前後方向の加速度と、カーブで横に振られる横加速度を計算し、それぞれが「人が不快に感じない境界」を超えた割合(違反率)から、0〜1の快適度を出します。1なら違反ゼロ(快適)、0なら全タイムステップで違反(不快)です。学習を必要としない軽量な物理計算で求められます。

## 直感的な理解
道路をはみ出さないこと([走行可能領域適合度 (DAC)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0142_drivable-area-compliance))は安全の最低条件ですが、それだけでは良い運転とは言えません。車道の内側にいても、急発進・急ブレーキ・急なハンドルを繰り返す運転は、乗員が車酔いしたり恐怖を感じたりします。タクシーに乗ったとき、安全運転でも荒い運転だと不快に感じた経験があるはずです。自動運転の評価では「安全(衝突・逸脱しない)」と並んで「快適(滑らかに走る)」が重要な評価軸になります。

快適さは主観的に思えますが、実は物理量でかなり表せます。人が不快を感じるのは、体に大きな力(加速度)がかかるとき、あるいは力が急に変化するとき(加加速度)です。発進や停止での前後方向の揺さぶり、カーブで横に押される力、これらを定量化して「ある境界を超えていないか」を見れば、軌道の滑らかさを数値で採点できます。

## 基礎: 前提となる概念
まず加速度の階層を整理します。位置を時間で1回微分すると速度、2回微分すると加速度、3回微分すると加加速度(jerk、ジャーク)になります。

- 速度(velocity): 単位時間あたりにどれだけ進むか(m/s)。
- 加速度(acceleration): 速度の時間変化(m/s^2)。前後方向のものを前後加速度(longitudinal acceleration)と呼び、大きいと急発進・急ブレーキを意味します。
- 横加速度(lateral acceleration): カーブで体が横に押される力(m/s^2)。物理的には「曲率 × 速度の2乗」(curvature × speed^2)で近似できます。曲率(curvature)はカーブの曲がり具合で、半径Rのカーブなら 1/R です。同じカーブでも速いほど横方向のGが大きくなる、という直感に合います。時速30kmで曲がれる急カーブも、時速80kmでは横Gが大きすぎて不快・危険になります。
- 加加速度(jerk): 加速度の時間変化(m/s^3)。加速度そのものが境界内でも、加速度が頻繁に切り替わるとガクガクした不快な動きになります。これを捉えるのが jerk です。

次に快適境界の目安です。乗用車の一般的な快適境界として、前後加速度はおよそ 2〜3 m/s^2、横加速度はおよそ 1.5〜2 m/s^2 程度とされます(報告や規格により幅があります)。これを超えると乗員が不快や危険を感じやすくなります。これらの値は ISO 2631(全身振動の人間暴露評価に関する国際規格)などの人間工学的な知見に基づきます。

最後に軌道の表現です。軌道を(加速度, 曲率)のペア列で持っている場合、加速度はそのまま前後加速度として読めます。横加速度は曲率と速度から計算します。速度そのものが軌道に含まれていなければ、初速を仮定して加速度を時間積分(累積和)して推定します。

## 仕組みを詳しく
(加速度, 曲率)ペア列の軌道を入力とする典型的な運動快適度スコアは、次のように計算されます。入力は軌道 `trajectory` [B, T*2](平坦化した(加速度, 曲率)ペア列)です。

ステップ1: ペアに戻し、加速度と曲率を取り出す。

```
pairs = trajectory.reshape(B, T, 2)
accels, curvatures = pairs[..., 0], pairs[..., 1]
```

ステップ2: 速度を推定する。初速 `initial_speed`(例 5.0 m/s)から、加速度を時間で累積積分して各時刻の速度を出します。`clamp(min=0)` で速度が負(後退)にならないようにします。

```
speed = clamp(initial_speed + cumsum(accels * dt, dim=1), min=0)   # dt 例 0.1s
```

ステップ3: 横加速度を計算する。

```
lateral_accel = curvatures * speed^2
```

ステップ4: 違反を判定する。

```
accel_violation   = (accels.abs()        > max_comfortable_accel)         # 例 3.0
lateral_violation = (lateral_accel.abs() > max_comfortable_lateral_accel) # 例 2.0
violation = maximum(accel_violation, lateral_violation)   # どちらか違反なら1
violation_rate = violation.float().mean(dim=1)            # 全ステップ中の違反割合 [B]
score = 1.0 - violation_rate
```

`maximum`(論理ORに相当)を使うのがポイントです。前後加速度と横加速度の「どちらか一方でも」境界を超えたタイムステップは違反(1)とカウントします。両者を足すのではなく、いずれかが不快なら不快、という扱いです。最後に違反率を全タイムステップで平均し、`1.0 - 違反率` を快適度として返します。

数値例。T=10 の軌道で、3ステップだけ加速度が境界 3.0 を超え(他は境界内)、横加速度はずっと境界内なら、違反ステップは3つ、違反率 = 3/10 = 0.3、快適度 = 1 - 0.3 = 0.7 です。全ステップ境界内なら快適度 1.0 です。

このスコアはマルチサンプル・スコアリングの合算で comfort 重み(例 0.5)を掛けて DAC と足されます([マルチサンプル軌道スコアリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0153_trajectory-scorer-multi-sample))。DAC が「道路に収まるか(安全)」、快適度が「滑らかか(快適)」を担当し、両方を満たす軌道が高得点になります。

注意点があります。第一に、速度推定は加速度の累積積分という近似なので、座標展開に使う積分規約と揃える必要があります。第二に、初速を固定値で仮定すると、実際の自車速度と乖離したときに横加速度の推定がずれます。横加速度は速度の2乗に効くため、速度が2倍ずれれば横加速度の推定は4倍ずれ、影響が大きいです。可能なら実測の初速を入力すべきです。

## 手法の系譜と主要論文
快適な軌道を最適化問題として扱う古典が Werling et al.("Optimal Trajectory Generation for Dynamic Street Scenarios in a Frenét Frame", ICRA 2010)です。Frenét 座標(道路に沿った縦方向 s と横方向 d で表す座標系)で軌道を多項式として表し、jerk(加加速度)の二乗積分を最小化する目的関数を立てました。動機は、jerk を抑えると数学的に最も滑らかな軌道が得られ、それが乗り心地の良さに直結することです。この「高階微分量を抑えると滑らかになる」という原理は、その後のプランニング研究の基礎になりました。

人間工学的な根拠としては ISO 2631 系の規格があり、全身振動と加速度が乗員の快適性・疲労に与える影響を定量化しています。横加速度・前後加速度・jerk が乗り心地に直結することはこれらの研究で確立しています。

評価指標としての快適性を planning ベンチマークに組み込んだのが nuPlan(Caesar et al., Motional, 2021)とその PDMS です。PDMS の Comfort 部分スコアは、加速度・jerk・ヨーレート(車体の回転速度)などが定められた上限を超えていないかで採点されます。NAVSIM(Dauner et al., NeurIPS 2024)もこれを引き継ぎ、Comfort を PDMS の構成要素として採用しています。NAVSIM の Comfort は複数の運動量(前後加速度・横加速度・jerk・ヨーレートなど)それぞれに閾値を設け、すべて境界内なら 1、どれか超えたら 0 とするバイナリ判定です。

GoalFlow(Xing et al., 上海交通大学/Bosch 他, arXiv:2503.05689, 2025)は、複数サンプルした軌道の再ランクスコアに運動制約・快適性の項を含め、急操作な候補の順位を下げます。ここで紹介する0パラメータ版の快適度スコアは、こうした選別項を学習なしの物理計算で再現したもので、jerk まで見ず加速度・横加速度の2量に絞った簡略版に相当します。

## 論文の実験結果(定量データ)
NAVSIM(Dauner et al., 2024)で報告される Comfort 部分スコアは、多くの上位手法で 0.99〜1.00 とほぼ満点に近い水準です。これは、現代のプランナーが滑らかさの面ではすでにかなり良く、Comfort は PDMS の中で差が付きにくい項であることを意味します。しかし、ここで快適性を軽視して急操作を許すと Comfort が落ち、PDMS が積の構造で下がるため、満点を維持すること自体が重要です。

GoalFlow(arXiv:2503.05689)のアブレーションでは、快適性・運動制約を考慮した再ランクが、無視した場合に比べて急操作軌道を減らし PDMS の Comfort 内訳を保つことが示されています。一方で、快適性だけを過度に重視すると進捗(EP, ego progress)が下がる傾向も報告されており、これは「あまりに慎重で遅すぎる運転」になることを意味します。安全・快適・進捗のバランスを採点でどう取るかが性能を左右します。

簡略版(加速度・横加速度のみ、バイナリ違反)の限界として、jerk を見ないことの影響が指摘できます。加速度が常に境界内でも、加速度が +3 と −3 を毎ステップ往復するような軌道は jerk が極端に大きく、実際には非常に不快です。こうした軌道を簡略版は快適と誤判定します。NAVSIM が jerk とヨーレートまで含めるのは、この見逃しを防ぐためです。

## メリット・トレードオフ・限界
メリットは、学習不要・0パラメータで、軌道の運動量だけから快適性を数値化できることです。計算は累積和と比較だけで軽量です。DAC と組み合わせれば「安全かつ滑らか」な軌道を選べます。

トレードオフと限界は次のとおりです。初速を固定値で仮定する実装では、実速度と乖離したとき横加速度推定が2乗で効いて大きくずれます。違反を「境界を超えたか 0/1」で数えるため、わずかな超過も大幅な超過も同じ重みになり、連続的な減点になりません。jerk を見ないため、加速度自体は境界内でも頻繁に符号反転するガクガクした軌道を快適と誤判定します。境界値はハイパーパラメータで、車種・路面・乗員の状態(立っているか座っているか)により最適値が変わります。未解決の課題としては、jerk やヨーレートまで含めた多次元の快適性を軽量に評価する方法や、乗員のフィードバックから快適境界を個別最適化(personalization)する方向が挙げられます。

## 発展トピック・研究の最前線
さらに深く学ぶ方向として、快適性を生成・最適化の段階に組み込むアプローチがあります。Werling 系のように jerk 最小化を目的関数に据えれば、事後採点ではなく軌道生成の時点で滑らかさを保証できます。モデル予測制御(MPC)では、加速度・jerk・ヨーレートに制約とコストを課して滑らかな制御列を直接最適化します。また、快適性の主観評価を機械学習で予測する研究(乗員の生体信号や主観アンケートから不快度を回帰する)も進んでおり、固定境界より個人差・状況差に適応した評価が可能になります。さらに、商用ライドシェアの実運用データから「人間が許容する加速度プロファイル」を学習し、それを快適性の基準にする data-driven な方向も注目されています。
