# num_views のパラメータ化

## ひとことで言うと

「車に付いているカメラの枚数」をプログラムの中に固定値として埋め込む(ハードコードする)のではなく、外から指定できる変数(慣習的に `num_views` と呼ぶ)にしておく設計です。こうすると、6カメラのデータセットでも7カメラ・8カメラのデータセットでも、コードを書き換えずに同じモデルを使い回せます。一見ささいな実装上の工夫ですが、複数のセンサ構成へ手法を一般化するための必須要件であり、形状バグの予防にも直結します。

## 直感的な理解

料理のレシピで「卵を3個割る」と固定で書いてあると、4人前に増やしたいとき毎回レシピを書き直す羽目になります。代わりに「卵を人数ぶん割る」と書いておけば、人数を変えるだけで同じレシピが使えます。`num_views` のパラメータ化はこれと同じで、「カメラ枚数ぶん処理する」と書いておくことで、構成の違うデータセットにそのまま対応できるようにする発想です。

自動運転データセットは、提供元によってカメラの台数や配置が違います。nuScenes は6カメラ(前・前左・前右・後・後左・後右)、Waymo は5カメラ(主に前方〜側方)、その他の運転データセットでは7〜8カメラというように、まちまちです。融合モジュールの中に「カメラは8枚」と決め打ちすると、6カメラのデータを入れた瞬間にテンソルの形が合わずエラーになります。

## 基礎: 前提となる概念

- ハードコード(hard-coding): プログラム中に具体的な数値を直接書き込むこと。`8` のような定数を埋め込むと、その前提が崩れた瞬間に壊れます。
- パラメータ化(parameterization): その値を外から渡せる引数・設定にすること。汎用性と保守性が上がります。
- 単一の信頼できる値(single source of truth): ある事実を1か所だけで管理し、各所はそれを参照する設計原則。カメラ枚数を1か所で持てば、矛盾や取り違えが起きにくくなります。
- 投影行列(projection matrix): 3D空間の点を、あるカメラの2D画像上のどこに写るかへ変換する行列。カメラの内部パラメータ(焦点距離・画像中心)と外部パラメータ(車体に対する位置・向き)から作られます。カメラごとに異なるので、カメラ枚数が変われば投影行列の数も変わります。

## 仕組みを詳しく

設計のポイントは「カメラ枚数を、モデルの初期化時と forward 実行時の両方で扱えるようにする」ことです。

初期化時(層の形を決める): カメラ枚数に依存したパラメータを持つ方式では、初期化引数で枚数を受け取ります。連結方式(concat)なら、入力チャンネル数が `num_views * embed_dim` で決まる 1×1 畳み込みを作ります。たとえば `embed_dim=256` で8カメラなら入力 `8*256=2048` チャンネル、6カメラなら `6*256=1536` チャンネルの畳み込みになります。カメラ間アテンション方式なら、カメラ位置埋め込み(形 `[1, num_views, embed_dim]`)の大きさが枚数で決まります。

forward 実行時(形を組み替える): `[B*V, C, H, W]` で来た特徴を `reshape(B, V, C, H, W)` で正しく分離するには、その場のバッチ `B` とビュー数 `V` の値が必要です。固定値で `reshape(8, 8, ...)` と書いてあると、6カメラのデータが来た瞬間に「要素数が合わない」と失敗します。

before / after:

```
before(ハードコード):  x = x.reshape(8, 8, C, H, W)   # 6カメラを入れると失敗
after(パラメータ化):    x = x.reshape(B, V, C, H, W)   # 6でも7でも8でも動く
```

さらに実務では、データセットを読んだ時点でカメラ枚数を自動検出し、その値をモデルに渡す仕組みを入れることが多いです。これにより利用者が手で枚数を指定し忘れても正しく動きます。ただし自動検出はデータ読み込みミスで誤った枚数を拾う危険もあるため、検証用テストで枚数を明示して確認するのが望ましい運用です。

方式ごとの「枚数依存の度合い」も重要です。

- concat 方式: 畳み込みの入力チャンネルが枚数に比例するため、枚数依存が強い。学習済み重みを別枚数に流用できない。
- カメラ間アテンション方式: カメラ位置埋め込みが枚数に依存する。埋め込みだけは作り直しになる。
- BEV 投影方式: 各カメラを `for v in range(V)` のようにループで投影してから BEV 上に集約するため、原理的に枚数非依存に近い。BEVFormer や PETR がこの性質を持ち、可変枚数に最も柔軟です。

## 手法の系譜と主要論文

「カメラ枚数を可変にする」こと自体は単独の論文テーマではなく、複数カメラ手法を異なるセンサ構成へ一般化するための実装上の必須要件です。関連する流れを挙げます。

- nuScenes(Caesar et al., CVPR 2020, arXiv:1903.11027) — 6カメラ・全周囲・LiDAR・レーダーを備えた大規模データセット。マルチカメラ3D知覚研究の事実上の標準ベンチマークになりました。
- Waymo Open Dataset(Sun et al., CVPR 2020, arXiv:1912.04838) — 5カメラ構成。nuScenes と枚数・配置が異なるため、両方で評価する研究が「枚数を固定しない」設計を後押ししました。
- BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)・PETR(Liu et al., ECCV 2022, arXiv:2203.05625) — カメラごとの幾何情報(投影や3D位置埋め込み)をクエリ側でループ処理する構造のため、カメラ枚数に対して比較的柔軟。各カメラを順に処理して BEV/クエリへ集約するので、原理的に可変枚数へ対応しやすい設計です。

複数データセットでの学習・評価が一般化したことが、「カメラ枚数を固定しない」設計を実務上の標準にしました。

## 論文の実験結果(定量データ)

枚数のパラメータ化そのものを直接評価した数値は少ないですが、関連する定量的事実があります。

- 複数データセットでの汎化研究では、あるデータセット(例: nuScenes 6カメラ)で学習したモデルを別構成(例: Waymo 5カメラ)へ適用すると、何の対応もしなければ大きく精度が落ちることが報告されています。カメラ配置・枚数・内部パラメータの違いが原因で、これがクロスデータセット汎化の主要な障壁の一つです。
- クエリベース手法(BEVFormer/PETR系)は、カメラ幾何をループで明示的に扱うため、枚数差に対する頑健性が連結型より高い傾向が示されています。一方 concat 型は枚数依存のパラメータを持つため、枚数が変わると当該層を作り直して再学習する必要があります。これは「設計選択が汎化性能とパラメータ再利用性に直結する」具体例です。

## メリット・トレードオフ・限界

- メリット: 異なるカメラ構成のデータセットを、コード変更なしで同じモデルに流せる。
- メリット: 枚数を1か所で管理することで [バッチ次元の崩壊バグ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0923_batch-dimension-collapse-bug) のような形状ミスを防ぎやすい。
- メリット: 将来カメラを増減するハードウェア変更にもコード側が追従しやすい。
- トレードオフ: concat / アテンション方式はカメラ枚数に依存したパラメータ(畳み込みチャンネル数や位置埋め込み)を持つため、学習済み重みを別枚数のデータに直接使い回せない(その層は作り直し)。
- トレードオフ: 枚数の自動検出に頼ると、データ読み込みミスで誤検出したとき気づきにくい。検証テストで枚数を明示して確認すべき。
- 限界・未解決課題: 枚数を可変にしても、カメラの内部・外部パラメータ(画角・配置)の違いまでは吸収できません。真に構成非依存なモデルを作るには、各カメラの校正情報を入力として与え、それに条件づけて処理する設計(カメラ条件づけ)が必要で、これはいまも研究途上です。

## 発展トピック・研究の最前線

- カメラ条件づけ(camera-aware modeling): 枚数だけでなく各カメラの内部・外部パラメータを埋め込みとしてモデルへ与え、構成差に明示的に適応させる流れ。クロスデータセット汎化の鍵とされています。
- センサ非依存表現: カメラ・LiDAR・レーダーを共通の BEV/3D 空間へ落とし込み、入力センサ数や種類が変わっても下流が同じ表現を受け取れるようにする設計(BEVFusion 系)。
- データセット横断学習: 複数データセットを混ぜて学習し、構成差に頑健なモデルを得る研究。枚数のパラメータ化はその最低条件です。
