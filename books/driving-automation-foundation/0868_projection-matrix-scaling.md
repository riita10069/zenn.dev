# 射影行列とリサイズ時の内部パラメータスケーリング

## ひとことで言うと
カメラが撮った画像の中で「3 次元世界のどの点が、画像の何ピクセル目に写るか」を計算する行列を射影行列(projection matrix)といいます。モデルに入れる前に画像をリサイズ(たとえば 256×256)すると、ピクセルの位置がずれます。このノートは、リサイズやクロップに合わせて内部パラメータ(行列 K)を正しくスケールし直さないと、複数カメラを鳥瞰図(BEV)に貼り合わせたときに地図が崩れる理由と、その正しい直し方を、具体的な数値で説明します。

## 直感的な理解
複数のカメラ映像を「真上から見た地図」(BEV、Bird's-Eye-View、鳥瞰図)のような共通の座標に貼り合わせて使う設計を考えます。前カメラが見た車も、左カメラが見た車も、同じ鳥瞰地図上の同じ位置に来てほしい。そのためには「このカメラのこのピクセルは、世界のこの方向を見ている」という対応関係が正確である必要があります。

ところがニューラルネットワークの特徴抽出器(backbone)は決まった入力サイズ(たとえば 256×256)を期待するので、元の高解像度画像(たとえば 1920×1080)をリサイズして渡します。このとき、同じ世界の点が写るピクセル座標が変わります。1920 幅のときに横 960 ピクセル目だった点は、455 幅にリサイズすれば横 227 ピクセル目あたりになります。元画像基準のカメラパラメータをそのまま使うと、貼り付け位置が系統的にずれ、複数カメラの BEV が二重像になったり回転したりしてぐちゃぐちゃになります。だから、リサイズに合わせてカメラの内部パラメータを直す必要があるのです。

## 基礎: 前提となる概念
カメラの幾何に必要な道具を順に説明します。

内部パラメータ K(intrinsic、3×3 行列)は、3 次元のカメラ座標を画像のピクセル座標に変換します。典型的な形は次の通りです。

```
      [ fx   0   cx ]
K =   [  0  fy   cy ]
      [  0   0    1 ]
```

`fx, fy` は焦点距離(ピクセル単位)で、レンズの画角に対応します。`cx, cy` は主点(principal point、画像中心)のピクセル座標です。

外部パラメータ T(extrinsic、4×4 の剛体変換行列)は、カメラが車体や共通の基準座標(reference frame)に対してどこにどんな向きで付いているかを表します。回転 R(3×3)と並進 t(3×1)からなります。

同次座標(homogeneous coordinates)とは、3 次元点 `[X, Y, Z]` に 1 を足して `[X, Y, Z, 1]` と書く流儀です。こうすると並進(平行移動)も行列のかけ算で表せて、射影が一つの行列演算にまとまります。

射影行列 P(3×4)は K と T を組み合わせて、基準座標の 3 次元点を一気にピクセルに落とす行列です。`P = K [R | t]` の形をとります。

BEV(Bird's-Eye-View、鳥瞰図)は真上から見下ろした俯瞰表現で、複数カメラを一つの座標に統合する先で使います。

## 仕組みを詳しく
射影行列を組み立て、リサイズに合わせて K をスケールする手順を段階的に見ます。

### 射影行列の組み立て
データセットはしばしば `T_camera_to_reference`(カメラ座標→基準座標の変換)を提供します。しかし射影で必要なのは逆向き「基準座標→カメラ座標」なので、行列を反転します。

```
K_scaled     = scale_intrinsic(K, original_image_size, transform)   # 3x3
T_ref_to_cam = inverse(T_camera_to_reference)                       # 4x4
P            = K_scaled @ T_ref_to_cam[:3, :]                       # (3, 4)
```

`T_ref_to_cam` の上 3 行(回転と並進)を取り出し、左から `K_scaled` を掛けて `P`(3×4)を得ます。`P` は同次座標の点 `[X, Y, Z, 1]` を受け取り `[u', v', w']` を返します。最後に `w'` で割って `u = u'/w', v = v'/w'` とすると、ピクセル座標 `(u, v)` になります(この割り算が遠近投影、遠くのものが小さく写る効果を表します)。

### リサイズが K に与える効果(直感)
画像を `s` 倍にリサイズすると、ピクセルが詰まる(または伸びる)ので、焦点距離も主点も同じ `s` 倍になります。

```
fx → fx*s,  fy → fy*s,  cx → cx*s,  cy → cy*s
```

これは K の上 2 行に `s` を掛けるのと同じです(3 行目の `[0,0,1]` はそのまま)。

### スケーリングの数値例(before / after)
元画像 1920×1080、`fx=fy=1000, cx=960, cy=540` とします。短辺基準で 256 にリサイズすると、短辺は高さ 1080 なので `s = 256/1080 ≈ 0.237` です。

| 量 | before(元画像) | after(リサイズ後) |
|---|---|---|
| fx | 1000 | 237.0 |
| fy | 1000 | 237.0 |
| cx | 960 | 227.6 |
| cy | 540 | 128.0 |
| 画像幅 | 1920 | 455 |
| 画像高さ | 1080 | 256 |

リサイズで横が 455 になるので、256×256 にするには中央クロップ(CenterCrop)で左右を切ります。クロップは主点をずらします。

```
offset_x = (455 - 256) / 2 = 99.5
cx → 227.6 - 99.5 = 128.1
```

最終的に `cx ≈ 128, cy ≈ 128`(256 の中央)に落ち着き、これでリサイズ後画像の射影が正しくなります。

### なぜ Resize と CenterCrop だけ K を変えるのか
前処理パイプライン(torchvision の `Compose` のように複数の変換を順に適用するもの)を 1 ステップずつ走査し、`Resize` と `CenterCrop` のときだけ K を更新するのが定石です。

```
for t in transform.transforms:
    if isinstance(t, Resize):       # 短辺リサイズ: s = size / min(h, w)
        K[0,0] *= s; K[1,1] *= s
        K[0,2] *= s; K[1,2] *= s
    elif isinstance(t, CenterCrop): # 主点を中央クロップぶん平行移動
        K[0,2] -= offset_x
        K[1,2] -= offset_y
    # Normalize / ToTensor は無視
```

理由は明快です。K は「世界の点 ↔ ピクセル位置」の対応です。位置を動かす幾何変換(リサイズ、クロップ、パディング)だけがこの対応関係を変えます。明るさ正規化(Normalize)やテンソル化(ToTensor)はピクセルの位置に無関係なので、K はそのままでよいのです。前処理を変えても、走査するだけで K の補正が自動で追従するのがこの実装の利点です。

## 手法の系譜と主要論文
射影行列の更新則は古典的な多視点幾何の教科書事項であり、それを深層学習の BEV 認識が前提として使う、という二段構えで理解できます。

Hartley & Zisserman の "Multiple View Geometry in Computer Vision"(2003、書籍)は、射影行列 `P = K [R | t]` の標準的な定式化と、画像のアフィン変換(スケール・平行移動・回転)に伴う K の更新則の理論的背景を与えます。画像変換 H を施したとき新しい内部行列が `K' = H K` になる、という関係がここで厳密に述べられており、上の Resize/CenterCrop の補正はこの特殊形(H が対角スケール + 平行移動)です。

Philion & Fidler の Lift-Splat-Shoot(LSS, ECCV 2020, arXiv:2008.05711)は、各カメラ画像を内部・外部パラメータを使って 3 次元に「持ち上げ(lift)」、BEV グリッドに「撒く(splat)」手法を提案しました。各ピクセルに深度の確率分布を予測させ、その分布に沿って特徴を 3 次元に配置します。ここで射影が画素単位で正確であることが、複数カメラを正しく重ねる前提です。画像をリサイズして学習するのが普通なので、内部パラメータのスケーリングは LSS 系の暗黙の必須処理です。

Li et al. の BEVFormer(ECCV 2022, arXiv:2203.17270)は、BEV 上のクエリ点から各カメラ画像へ射影してアテンション(注目機構)で特徴を引いてきます。ここでも `P = K @ T` の正確さがそのまま BEV 特徴の質に直結します。前処理でリサイズ・クロップした分だけ K を補正しないと、サンプリング位置が系統的にずれ、性能が落ちます。BEVDet(Huang et al., arXiv:2112.11790)も LSS をベースにしつつデータ拡張(画像のスケール・回転を変える)を多用しますが、拡張のたびに K を整合的に更新することで初めて拡張が有効に働く、と報告しています。

## 論文の実験結果(定量データ)
カメラ幾何の正確さと BEV 認識性能の関係は、nuScenes(6 カメラ周囲認識の標準ベンチマーク)の数値で示せます。主要指標は NDS(nuScenes Detection Score、検出の位置・速度・属性を総合した 0〜1 のスコア、高いほど良い)と mAP(平均適合率)です。

BEVFormer は nuScenes test で NDS 約 0.569、mAP 約 0.481 を報告し、カメラのみで LiDAR に迫る 3D 検出を達成しました。この性能は、各カメラの射影が正確であること(リサイズ後の K 補正を含む)を前提にしています。

BEVDet 系の研究では、データ拡張(画像のリサイズ・回転・フリップ)を入れる際に、対応するカメラ内部・外部パラメータを整合的に更新する「image-space + BEV-space augmentation」を導入することで、mAP と NDS が数ポイント規模で改善することが報告されています。BEVDet の論文では、画像空間の拡張だけでなく BEV グリッド側にもスケール・回転・フリップを掛け、その両方を内部・外部行列に反映する設計を入れることで、拡張なしのベースラインに対して mAP を明確に押し上げ、過学習(訓練データに過剰適合して未知データで性能が落ちる現象)を抑えられたとしています。逆に、画像だけを拡張して K を更新し忘れると、射影とのずれが学習を妨げ性能が下がります。これは「幾何変換と K 補正をセットにしないと壊れる」ことの定量的な裏付けです。BEVDepth(Li et al., AAAI 2023, arXiv:2206.10092)はさらに踏み込み、内部・外部パラメータの微小な校正誤差(取り付け角の数度ずれ)を意図的に注入するアブレーションを行い、誤差が大きいほど深度推定と mAP が単調に劣化することを示しました。これは「射影の正確さは精度の前提条件であり、校正がずれると上限そのものが下がる」ことを定量化した結果です。

アブレーション的に言えば、リサイズ時の K スケーリングを省くことは「サンプリング位置の系統的なバイアス」を入れることに等しく、BEV 特徴が一貫してずれた位置から引かれるため、深度・位置の誤差が増えます。報告されるのは主点ずれ(数十ピクセル規模)に比例した位置誤差の悪化で、特に遠方・周辺視野で顕著です。

## メリット・トレードオフ・限界
メリットは明快です。リサイズ・クロップ後でも射影が正確になり、複数カメラを正しい BEV 位置に貼れます。前処理パイプラインを走査して幾何ステップだけ拾う実装にすれば、前処理を変えても K の補正が自動で追従し、手で係数を直す必要がありません。データセットごとの実校正値を使えるので、学習可能な擬似射影(pseudo projection、校正値が無いときに学習で代用する射影)より物理的に正しい初期値になります。

トレードオフと限界もあります。前処理に未対応の幾何変換(任意角度の回転、一般のアフィン warp、レンズ歪み補正など)が入ると、単純なスケール + 平行移動の補正では扱えず破綻します。対応する変換の種類を増やすメンテナンスが要ります。校正の誤差(取り付け公差、センサ較正のずれ)はそのまま射影に乗るため、較正が悪いデータセットでは、いっそ擬似射影をモデルに学習させたほうが頑健なこともあります。外部パラメータの向き(camera→reference か reference→camera か)を取り違えると、見た目は動くのに地図が反転・回転するという、テストでも気づきにくいバグになります。

未解決の課題として、魚眼レンズなど強い歪みを持つカメラでは線形の K では表せず、歪みモデル込みの射影をどう深層学習に組み込むかが研究途上です。また、校正値が時間とともにずれる(温度変化や振動)問題に対し、学習中・推論中にオンラインで校正を補正する手法も模索されています。

## 発展トピック・研究の最前線
近年は、カメラの内部・外部パラメータをモデルに明示的に与える「camera-aware」な設計が広がっています。パラメータをエンコードして特徴に注入することで、異なる FOV・解像度のカメラを同じ重みで扱える視点不変性を狙います。これはデータセット非依存設計([カメラ外部パラメータとデータセット非依存設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0939_camera-extrinsics-dataset-agnostic))と直結します。

また、深度推定を明示的に行う LSS 系と、深度を陽に求めず BEV クエリで暗黙に引く BEVFormer 系の対比は依然として活発な論点で、どちらも射影の正確さに依存します。さらに、校正値そのものをデータから推定する self-calibration(自己校正)や、複数フレームの一貫性から外部パラメータを補正する手法など、「正確な P をどう保証し続けるか」が最前線のテーマです。
