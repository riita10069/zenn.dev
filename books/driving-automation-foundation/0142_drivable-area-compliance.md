# 走行可能領域適合度 (DAC)

## ひとことで言うと
予測した軌道(車がこれから通る点の列)が、ちゃんと「走ってよい場所(車道)」の上に乗っているかを測る指標です。DAC は Drivable Area Compliance の略で、日本語にすると「走行可能領域への適合度」です。最も軽量な実装は、上空から見た地図画像のピクセルの色を直接参照し、「この点は道路の色か?」を判定して、軌道全体のうち道路に収まった点の割合を [0, 1] で返します。学習を一切必要としません。

## 直感的な理解
自動運転で最低限守るべきことの1つが「車道から出ない」ことです。歩道に乗り上げる、対向車線にはみ出す、壁やガードレールに突っ込む、こうした逸脱は重大事故に直結します。プランナーが出した軌道がどれだけ安全かを数値で評価したいとき、一番直接的なのは「軌道の各点が走行可能領域(走ってよい場所)に入っているか」を1点ずつ確認することです。

ここでカーナビの地図を思い浮かべてください。地図には道路が一定の色で塗られています。もし自車の未来の位置をその地図の上に重ねられれば、「この点は道路の色の上にあるか、それとも歩道や建物の色の上にあるか」を見るだけで逸脱が分かります。DAC のもっとも素朴な実装は、まさにこの「地図に未来位置を重ねて色を見る」発想を計算機で行うものです。10点の軌道のうち8点が道路の上なら DAC = 0.8、全点が道路上なら 1.0、全点逸脱なら 0.0、という具合です。

## 基礎: 前提となる概念
最初に走行可能領域(drivable area)という言葉を押さえます。これは「自車が物理的・法的に走行してよい路面領域」を指し、車線・交差点内などを含み、歩道・路肩外・対向車線(進行方向によっては除外)などを含みません。

次に、走行可能領域をどう表現するかです。大きく2つあります。1つはベクトル表現で、道路や車線の境界を多角形(ポリゴン)や折れ線で持ちます。HDマップ(高精細地図)は典型的にこの形式です。もう1つがラスタ表現で、領域を1枚の画像(ピクセルの格子)に塗り分けます。ラスタライズ(rasterize)とは、ベクトルの地図を画素の画像に描き起こす操作です。

そして上空視点(Bird's-Eye View, BEV)という考え方です。自動運転では、カメラやLiDARの情報を「真上から見下ろした座標系」に変換した俯瞰図を作ることがよくあります。この BEV 上に地図をラスタライズすると、自車を中心とした俯瞰の道路画像が得られます。多くのエンドツーエンドモデルは、この BEV ラスタ地図をネットワークの入力として既に作っています。すると、軌道の各点をその BEV 画像のピクセル座標に変換するだけで、追加コストほぼゼロで走行可能領域判定ができます。これが「色ルックアップ型 DAC」の着想です。

最後に座標系の話です。自車相対座標(ego-centric coordinate)では、自車を原点とし、前方を x の正方向、左を y の正方向に取ることが多いです(右手系)。一方、画像の座標は左上が原点で、行(row)は下に向かって増え、列(col)は右に向かって増えます。メートル座標から画像のピクセル座標へ変換するには、原点の位置(自車が画像のどのピクセルに来るか)と、メートルあたり何ピクセルかの倍率(pixels per meter)、そして前方が画像の上か下かの向きを合わせる必要があります。この対応付け(較正)を間違えると、射影先のピクセルがずれて指標全体が無意味になります。

## 仕組みを詳しく
色ルックアップ型 DAC の典型的な処理は、入力として軌道のウェイポイント `xy` [B, T, 2](自車相対のメートル座標)とラスタ地図 `map` [B, 3, H, W](RGB画像)を取り、次のように進みます。

ステップ1: メートル座標をピクセル座標へ射影する。

```
row = ego_row + sign * x * pixels_per_meter   # 前方が画像上なら sign=-1
col = ego_col - y * pixels_per_meter          # +y(左)なら列は小さくなる
(row, col) を四捨五入して整数化
```

`pixels_per_meter` は解像度の逆数です。たとえば 0.4 m/ピクセルなら 2.5 ピクセル/m です。`ego_row` / `ego_col` は自車(x=0, y=0)が来るピクセル原点です。前方が画像の上か下かによって x の符号を反転させます。

ステップ2: 画像範囲内かを判定する。`in_bounds = (0 <= row < H) & (0 <= col < W)`。範囲外の点(視界の外に出た点)は不適合として扱います。ただし配列アクセスでエラーを出さないよう、座標は一度 clamp で画像内に丸めてから色を読み、不適合判定だけは元の in_bounds で行う、という実装が安全です。

ステップ3: 色で走行可能か判定する。

```
target = drivable_rgb          # 例: 道路色 (255,255,255)
sampled = map[b, :, row, col]  # 各点のRGB [T,3]
diff = (sampled - target).abs().sum(dim=-1)   # L1距離(3チャンネル合計)
compliant = diff <= 3 * tolerance             # 例: 各ch許容10 → 合計30以内
```

地図画像はJPEG圧縮やアンチエイリアス(輪郭をなめらかにする処理)で色が多少ぶれることがあるため、ぴったり一致ではなく許容差(tolerance)を持たせます。許容を広げすぎると別の領域の色まで道路と誤認し、狭すぎると本来道路の点を見逃します。

ステップ4: 適合率を返す。`(compliant & in_bounds).float().mean(dim=1)` で、T点のうち道路かつ範囲内だった割合 [B] を返します。

数値例。T=10点の軌道で、8点が道路色かつ範囲内、2点が歩道色なら DAC = 8/10 = 0.8 です。この値はマルチサンプル・スコアリングの主要項として使われます([マルチサンプル軌道スコアリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0153_trajectory-scorer-multi-sample))。

較正に関する重要な注意があります。`pixels_per_meter` / `ego_row` / `ego_col` / 前方の符号は、地図をレンダリングした側の規約と完全に一致していなければなりません。たとえば「前方120m・後方60m・左右60mを 0.4m 解像度で 450x300px に描く」という規約なら、それに対応する原点と倍率を設定する必要があります。ここがずれると射影先のピクセルが間違い、DAC が無意味になります。もう1つの落とし穴は正規化です。ネットワーク入力用に地図画像が ImageNet 正規化(平均を引いて標準偏差で割る標準化)されていると、生のRGB色との比較が成立しません。色比較は必ず正規化前の生ピクセルを持つ段階で行う必要があります。

## 手法の系譜と主要論文
地図整合を予測・評価に組み込む発想の源流の1つが Argoverse(Chang et al., Argo AI / CMU 他, CVPR 2019)です。大規模運転データセットを公開し、HDマップ(車線中心線・走行可能領域)を予測タスクのメタデータとして提供しました。走行可能領域をラスタとベクトルの両方で提供し、軌道予測の妥当性評価に地図整合を使う枠組みを広めました。動機は、地図を使えば「物理的にあり得ない予測(建物を貫通するなど)」を排除でき、予測精度が大きく上がることです。トレードオフはHDマップの作成・維持コストの高さです。

評価指標としての DAC を明確に定式化したのが nuPlan(Caesar et al., Motional, 2021)とその planning スコア PDMS です。PDMS は衝突回避・走行可能領域順守・進捗・快適性などの部分スコアの組み合わせで、走行可能領域から逸脱すると大きく減点されます。NAVSIM(Dauner et al., Tübingen 他, NeurIPS 2024)はこれをオープンループの非反応的シミュレーションで効率的に測れるようにし、DAC を PDMS の必須構成要素として採用しました。NAVSIM での DAC は、自車のフットプリント(車体の四隅)が走行可能領域ポリゴンの内側にあるかをベクトル幾何で判定する厳密版です。

GoalFlow(Xing et al., 上海交通大学/Bosch 他, arXiv:2503.05689, 2025)は、学習させた BEV 意味セグメンテーションヘッド(画素ごとに道路/非道路を分類するネットワーク)で DAC を計算し、軌道再ランクに使いました。動機は、色ルックアップより正確に道路/非道路を判定できることです。色ルックアップ型 DAC は、この学習版の「学習なし代替(proxy)」と位置づけられます。学習信号(意味ラベル付き地図)が用意できれば学習版へ置き換えるのが精度上は望ましいが、用意できない状況では色ルックアップが現実的な選択肢になります。

## 論文の実験結果(定量データ)
NAVSIM ベンチマーク(Dauner et al., 2024)では、各手法の PDMS とその内訳が報告されます。DAC スコアは典型的に 0.95〜0.99 程度の高い水準で、ここでの数%の差が逸脱頻度の大きな違いを意味します。たとえば DAC 0.98 と 0.93 の差は、軌道全体で逸脱する確率が数倍違うことに相当し、PDMS は積の形を含むため最終スコアに無視できない影響を与えます。

GoalFlow(arXiv:2503.05689)のアブレーションでは、学習済み DAC スコアによる再ランクを外すと走行可能領域逸脱が増え、PDMS が低下することが示されています。これは DAC ベースの選別が逸脱削減に直接効いていることの証拠です。逆に、純粋な回帰プランナーは地図整合を明示的に評価しないため DAC が相対的に低く出やすく、地図を使った再ランクや地図条件付け(map conditioning)を入れると改善する、という傾向が複数の手法で一貫して報告されています。

色ルックアップ型の近似については、厳密なベクトル判定と比べた誤差が問題になります。報告される傾向として、地図解像度が荒い(1ピクセルが大きい)ほど境界付近での判定誤差が増え、車線境界ぎりぎりを走る軌道で過小評価・過大評価が起きやすくなります。これは指標としての DAC を使う際に、解像度と較正精度がそのまま信頼性を左右することを意味します。

## メリット・トレードオフ・限界
メリットは、学習不要で既存のラスタ地図だけから道路逸脱を検出でき、追加データもモデルも要らないことです。計算もピクセル参照と色比較だけなので軽量で、複数候補の再ランクに直接使えます。

トレードオフと限界は次のとおりです。較正値(倍率・原点・前方符号)が地図レンダラと一致していないと射影先がずれ、指標が完全に壊れます。色ベースの判定は描画色や圧縮ノイズに敏感で、許容差の設定が誤検出と見逃しのトレードオフを生みます。色は「道路かどうか」しか表さないので、車線の種類・進行方向・信号・一時的な工事規制といった意味情報は分かりません。学習済みセグメンテーションより表現力が低いのが本質的な限界です。さらに、ウェイポイントを点として判定する場合、車体の幅を考慮しません。点としては道路上でも、車体の角がはみ出すケースを見逃します。厳密にはNAVSIM 流に車体フットプリントの四隅で判定すべきで、点判定は楽観的な近似です。未解決の課題としては、軽量さを保ちつつ意味情報や車体形状を取り込む中間的な指標の設計が挙げられます。

## 発展トピック・研究の最前線
さらに学ぶ方向として、占有グリッド(occupancy grid)や占有フロー(occupancy flow)を使った走行可能・占有判定があります。これらは静的な道路だけでなく、他車・歩行者が「今後どこを占有するか」も予測して時空間の安全性を評価します。また、DAC のような幾何指標を生成過程に直接組み込む手法(微分可能な地図損失で軌道を学習段階から道路内に押し込む)も研究されています。地図そのものを学習で推定する HDMapNet や MapTR 系の研究と組み合わせれば、事前地図がない場所でも DAC を計算できるようになり、地図整備コストの問題を緩和する方向につながります。
