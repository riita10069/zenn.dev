# ラスタ地図エンコーダ (Raster Map Encoder) — 地図画像を鳥瞰特徴へ

## ひとことで言うと
ナビ地図を描いた1枚の画像を、ニューラルネットが扱いやすい「特徴マップ」に変換するモジュールです。中身は既製の画像認識バックボーン(たとえば Swin Transformer 系)で、画像モデルのライブラリから取り出して再利用します。出力の形状はカメラ由来の鳥瞰特徴とぴったり一致するよう調整され、後段でそのまま重ねられるようにします。

## 直感的な理解
地図融合([地図融合 (Map-BEV Fusion) — ナビ地図を鳥瞰特徴へ重ねる](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0977_map-bev-fusion))を実現するには、まず「地図画像」を「数値の特徴」に変換する部品が要ります。地図はピクセルの集まり(RGB 画像)であって、そのままではニューラルネットの中間表現と足し合わせられません。地図に描かれた線・領域・色が「何を意味するか(車線・進路・交差点)」を、学習可能な特徴ベクトルへ翻訳する必要があります。これがラスタ地図エンコーダ(ラスタ=画素ベースの地図エンコーダ)の役割です。

ここで一から畳み込みネットを設計してもよいのですが、画像から良い特徴を取り出すネットワークはすでに数多く研究され、ライブラリに何百種類も用意されています。それを呼び出して再利用するのが現代的なやり方です。これは「車輪の再発明をしない」という実装上の合理性であり、よく検証された強力なバックボーンを部品として使い回すという思想でもあります。新しいバックボーンが出るたびに、コードを大きく書き換えずに差し替えて性能を比較できるという研究上の利点もあります。

## 基礎: 前提となる概念
バックボーン(backbone)とは、画像から段階的に特徴を抽出するネットワークの本体部分です。画像分類器は「バックボーン(特徴抽出)+ヘッド(分類)」に分かれており、ヘッドを外してバックボーンだけを特徴抽出器として使い回すのが定番です。物体検出・セグメンテーションなど多くのタスクがこの「バックボーン+タスク固有ヘッド」構造を共有します。

特徴ピラミッド(feature pyramid)とは、バックボーンの各段(ステージ)が出す異なる解像度の特徴マップの集まりです。浅い段は高解像度で細かい(エッジや小物体)、深い段は低解像度で意味的(大きな構造)になります。複数解像度を組み合わせると、小さいものも大きいものも捉えられます。この考えは Feature Pyramid Networks(Lin et al., CVPR 2017, arXiv:1612.03144)が物体検出で確立しました。地図でいえば、浅い段は「細い車線の輪郭」を、深い段は「交差点という大きな構造」を捉える、という役割分担に対応します。

画像モデルライブラリ(timm, PyTorch Image Models; Wightman 2019-)は、数百の画像モデルを統一 API で提供する実装基盤です。`features_only=True` という指定で分類ヘッドを外して中間特徴ピラミッドだけを取り出せ、`feature_info` というメタ情報で「各ステージが何チャンネルの特徴を出すか」を取得できます。これにより、どのバックボーンを選んでも同じやり方で中間特徴を扱えます。メタ情報を実行時に問い合わせる(動的に取得する)ので、バックボーンを変えても後段の射影層のチャンネル数が自動で合う、という汎用性が生まれます。

適応的プーリング(AdaptiveMaxPool2d)とは、入力解像度がいくつであっても出力解像度を指定したサイズ(たとえば 8×8)に固定する操作です。各出力セルが対応する入力領域の最大値を取ります。異なる解像度のステージ出力を同じ大きさに揃えるのに使います。最大値を取る(平均でなく)のは、線や輪郭といった目立つ特徴を残しやすいためです。

1×1 畳み込み(1×1 Conv)とは、空間方向に動かず各位置のチャンネルだけを線形変換する操作で、実質「チャンネル数の付け替え」を行います。たとえば 1440 チャンネルを 256 チャンネルに圧縮するのに使います。空間構造をそのままに、各位置の特徴ベクトルを別の次元に写す射影だと考えてください。

## 仕組みを詳しく
構成は次の流れです。

1. バックボーンを作る。画像モデルライブラリから Swin Transformer V2 系の小型版などを `pretrained=False`(重みをランダム初期化)、`features_only=True`(中間特徴ピラミッドを取り出す)、`in_chans=3`(入力チャンネル数、RGB なら3)で生成します。`features_only=True` が肝で、これにより最終分類層ではなく各ステージの中間特徴(複数解像度の特徴マップ)が返ります。

2. 各ステージのチャンネル数を取得する。`feature_info` から各ステージの出力チャンネル数を読み取ります。Swin V2 の小型版なら、4ステージで例えば 96, 192, 384, 768 と深くなるほど増えます。これらの合計(96+192+384+768=1440 など)が、後で連結したときの総チャンネル数になります。メタ情報を動的に読むことで、別のバックボーンに差し替えてもこの合計が自動で再計算されます。

3. プーリングと射影層を用意する。`AdaptiveMaxPool2d((8, 8))` で各ステージを 8×8 に揃え、`Conv2d(総チャンネル, 256, kernel=1) → GELU` でチャンネルを 256 に圧縮する射影を用意します。

forward の処理を数値例つきで追います。
- 入力 `map_image`: `(B, 3, 256, 256)`。256×256 は Swin 系のパッチ埋め込みが要求する固定サイズです(後述)。
- バックボーンに通すと4つの特徴マップ(解像度の異なるピラミッド)が返ります。たとえば 64×64, 32×32, 16×16, 8×8 のように半分ずつ縮みます。
- 各特徴を `AdaptiveMaxPool2d((8,8))` で 8×8 に揃え、チャンネル方向に連結して `(B, 総チャンネル, 8, 8)` にします。
- 最後に 1×1 Conv でチャンネルを 256 に射影して、出力 `(B, 256, 8, 8)` を得ます。

```
(B,3,256,256) ─backbone─> [(B,96,64,64),(B,192,32,32),(B,384,16,16),(B,768,8,8)]
              ─pool 8x8 + concat─> (B,1440,8,8) ─1x1 Conv─> (B,256,8,8)
```

なぜ最後に 256 にするのか。カメラ由来の BEV 特徴のチャンネル数(埋め込み次元)が 256 だからです。チャンネル数を揃えないと融合モジュールで足し合わせられません。出力形状をカメラ BEV と一致させてあるので、融合が単純な足し算やアテンションで済みます。全ステージを連結してから射影するのは、浅い高解像度特徴(細かい線)と深い意味的特徴(大きな構造)の両方を後段に渡すためで、これは FPN の発想そのものです。

実装上の注意が2つあります。第一に、Swin Transformer は特徴を channels-last `(B, H, W, C)` の並びで返すことがあるため、channels-first `(B, C, H, W)` に直す入れ替え(permute)が必要になる場合があります。期待チャンネル数が最後の次元に来ていたら入れ替える、という防御的なコードを置きます。ライブラリやバージョンによって並びが違うことがあるため、形状を仮定せず実行時に確認するのが安全です。第二に、複数ステージのチャンネル数をメタ情報から動的に取得する設計にすると、バックボーンを別のものに差し替えても射影層のチャンネル数が自動で合うので、汎用性が上がります。

## 手法の系譜と主要論文
Swin Transformer(Liu et al., ICCV 2021, arXiv:2103.14030)は、画像を小さなパッチに分け、ウィンドウ(局所領域)単位の self-attention を階層的に適用するアーキテクチャです。全画素間でアテンションを取ると計算量が画素数の2乗で爆発するため、局所ウィンドウに限定し、ウィンドウを段ごとにずらす(shifted window)ことでウィンドウ間の情報もやり取りします。これにより、計算量を画素数に対して線形に抑えつつ、CNN のような階層的な特徴ピラミッドを Transformer で実現しました。動機は「Transformer を検出・セグメンテーションのような密な予測タスクへ汎用バックボーンとして使えるようにする」ことです。ViT(Vision Transformer)が単一解像度・全域アテンションだったのに対し、Swin は階層構造を持ち込んだ点が転換でした。

Swin Transformer V2(Liu et al., CVPR 2022, arXiv:2111.09883)は V1 を大規模・高解像度へスケールさせる改良版です。学習の安定化として residual-post-norm(残差の後に正規化を置く)と scaled cosine attention(アテンションのスコアを内積でなくコサイン類似度ベースにしてスケールを安定させる)を導入し、低解像度で事前学習して高解像度へ適応する log-spaced continuous position bias を提案しました。動機は「Transformer を30億パラメータ級・1536×1536 解像度まで安定にスケールさせる」こと。効果は分類・検出・セグメンテーションでの高精度更新で、トレードオフは CNN より計算が重く、固定パッチサイズ由来の入力解像度制約(小型版が 256×256 固定なのもこのため。パッチ分割と階層的なパッチ統合の段数が解像度を割り切れる前提に縛られる)がある点です。

画像モデルライブラリ(timm)は論文というより実装基盤ですが、「事前学習済みの強力なバックボーンを部品として再利用する」現代的なやり方を支える役割を果たしています。`features_only` / `feature_info` という統一インターフェースは、Feature Pyramid Networks(Lin et al., 2017)が示した「複数解像度の特徴を組み合わせる」発想を、どのバックボーンに対しても同じやり方で使えるようにしたものです。バックボーンの選択肢としては、Swin 系のほか ResNet(He et al., CVPR 2016, arXiv:1512.03385)のような古典的 CNN、ConvNeXt V2(Woo et al., CVPR 2023, arXiv:2301.00808)のような近代化 CNN もあり、いずれも同じ `features_only` API で扱えます。

地図エンコーダで `pretrained=False`(ランダム初期化)を選ぶことには根拠があります。地図画像はラインや塗りつぶしで構成され、自然写真(ImageNet)とは画素統計(色の分布、テクスチャの細かさ、エッジの鋭さ)が大きく異なります。転移学習の研究、たとえば Yosinski et al. "How transferable are features in deep neural networks?"(NeurIPS 2014, arXiv:1411.1792)は、ソースとターゲットのドメインが離れるほど事前学習の恩恵が薄れることを層ごとに定量化しました。地図のような合成的な画像では、ゼロから学習するほうが妥当なことが多いという判断につながります。

## 論文の実験結果(定量データ)
Swin Transformer(2021)は ImageNet-1K 分類で Swin-B が約 84% の top-1 精度(正解が最上位予測に一致する割合)を達成し、当時の CNN/Transformer を上回りました。さらに COCO 物体検出で box AP(検出の総合精度。高いほど良い)を当時の最先端から約 +2.7 ポイント、ADE20K セマンティックセグメンテーションで mIoU を約 +3.2 ポイント改善しました。汎用バックボーンとして密な予測タスクで強いことを示した点が重要で、地図エンコーダのように「中間特徴ピラミッドを後段で使う」用途に向くことの裏づけになります。

Swin Transformer V2(2022)は、30億パラメータのモデル(SwinV2-G)を 1536×1536 解像度で学習し、ImageNet-V2 で約 84% の top-1、COCO 検出で約 63.1 の box AP、ADE20K で約 59.9 の mIoU と、複数ベンチマークで当時の最先端を更新しました。アブレーションでは、residual-post-norm と scaled cosine attention を外すと大規模モデルの学習が発散しやすくなることが示され、V2 の安定化技法が大規模化に不可欠であることが分かります。

Feature Pyramid Networks(Lin et al., 2017)は、単一解像度の特徴だけを使う検出器に対し、複数解像度のピラミッドを組み合わせると COCO 検出の AP が明確に向上することを示しました(報告では Faster R-CNN ベースで AP が数ポイント改善)。特に小物体の AP が大きく改善し、「浅い高解像度特徴と深い意味的特徴を両方使う」ことの効果を定量化しました。地図エンコーダで全ステージを連結してから射影する設計は、この「複数解像度を併用する」思想に沿っています。

転移学習に関して Yosinski et al.(2014)は、ImageNet で事前学習した特徴を別タスクへ転用する際、浅い層ほど汎用的(転移しやすく)、深い層ほどタスク特化(転移しにくい)であることを層ごとに定量化しました。ドメインが大きく異なる場合に事前学習の利得が縮むという知見は、地図という非自然画像でランダム初期化を選ぶ判断の根拠になります。一方で、データ量が乏しい場合は、たとえ遠いドメインでも事前学習が初期収束を助けることがあるため、ランダム初期化か事前学習かは実験で確かめるべき設計判断です。実務では、地図データが十分にあるならランダム初期化、少ないなら ImageNet 事前学習からの微調整、と使い分けます。

## メリット・トレードオフ・限界
メリットは、既製のバックボーンを再利用でき、地図用ネットを一から作らずに済む点です。`features_only` と適応的プーリング、1×1 Conv の組み合わせで、出力解像度とチャンネルを自由に合わせられます。出力形状をカメラ BEV と一致させてあるので融合が単純です。チャンネル数をメタ情報から動的に取得すれば、バックボーンの差し替えにも強くなり、Swin・ResNet・ConvNeXt などを1行の設定変更で比較できます。地図ドメイン向けにランダム初期化から学習でき、自然画像の事前学習の悪影響を避けられます。

トレードオフと限界もあります。Swin 系の小型版は入力が 256×256 固定(パッチ埋め込みの制約)で、地図解像度がここに縛られます。バックボーンぶんのパラメータと計算が常に乗ります(地図のためだけに数百万〜数千万パラメータを抱える)。channels-last/first の取り回しや、ライブラリ依存のメタ情報の扱いなど、実装上の細かい注意が必要です。ランダム初期化ゆえ学習初期は地図特徴が弱く、収束に時間がかかる可能性があります。さらに、適応的最大プーリングで 8×8 まで一気に縮めると、地図の細かい幾何(交差点の細かい形状、車線の本数、分岐角度)が潰れうるため、より高解像度を保つ設計とのトレードオフがあります。最大プーリング自体も、平均的な情報を捨てて目立つ点だけを残すので、面的な情報(広い走行可能領域など)の表現には不利になることがあります。

## 発展トピック・研究の最前線
さらに深く学ぶ方向として、第一にバックボーンの選択があります。地図のような合成画像では、Transformer より軽量な CNN(ConvNeXt V2, EfficientNet)や、地図専用に設計した軽いエンコーダのほうが速度・精度のバランスが良いこともあり、比較する価値があります。地図は自然画像ほどテクスチャが豊かでないため、巨大なバックボーンが過剰なこともあります。第二に、ラスタ地図でなくベクトル地図を直接符号化するアプローチ(VectorNet 系のグラフ符号化)があり、解像度の制約を受けず情報密度が高い特徴を得られます。第三に、地図をモデル自身に予測させるオンライン地図構築(MapTR など)では、そもそも入力としての地図画像が不要になり、エンコーダの役割が変わります。第四に、自己教師あり事前学習(masked image modeling、画像の一部を隠して復元させる学習)を地図画像に適用し、ラベルなしの大量の地図でバックボーンを事前学習してから微調整する、というドメイン適応の研究方向があります。これは「地図ドメインだから事前学習が効かない」という問題を、地図ドメイン自身での事前学習で回避する筋です。第五に、出力を 8×8 のような粗いグリッドでなく可変解像度・高解像度に保ち、細い車線や分岐の幾何を残す設計が、進路選択の精度を要する用途で検討されています。
