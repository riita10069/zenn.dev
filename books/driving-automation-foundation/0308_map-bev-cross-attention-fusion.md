# マップBEVクロスアテンション融合

## ひとことで言うと
カメラ画像から作った「上から見た特徴マップ (画像 BEV)」に、地図から作った「上から見た特徴マップ (地図 BEV)」の情報を、アテンション (注目) という仕組みで混ぜ込む部品です。画像だけでは分かりにくい「ここは交差点」「この先カーブ」「片側 2 車線」といった地図の手がかりを、画像特徴のマス目ごとに必要なぶんだけ取り込みます。画像 BEV を query (問い合わせ側)、地図 BEV を key/value (参照される側) とするクロスアテンションで実現します。

## 直感的な理解
まず BEV (Bird's-Eye-View、鳥瞰図) とは、カメラの斜め視点ではなく真上から見下ろした座標系のことです。自動運転では、複数カメラの映像を一度この真上視点へ変換した「画像 BEV 特徴」を作ると、車線や周囲の物体の位置関係を扱いやすくなります ([BEVマップセグメンテーション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0269_bev-map-segmentation) 参照)。地面を上空から眺めた格子状のマップに、各マスがその場所の意味を表すベクトルを持つ、と考えてください。

画像 BEV だけでも走れますが限界があります。前方の道が「この先で右に分岐する」ことや「片側 2 車線」であることは、遠方や遮蔽の裏側だと画像からは見えません。一方、地図 (OpenStreetMap など) にはそうした道路構造が事前情報として入っています。これを画像 BEV に注入できれば、知覚が補強されます。地図は「見えない先を教えてくれるカンニングペーパー」のような役割です。

素朴には「画像 BEV と地図 BEV を足し算する」「チャンネル方向に連結する」だけでも混ぜられます (後述の残差ゲート版がこれに近い)。しかし単純な足し算には弱点があります。「どのマス目で地図をどれだけ参照すべきか」を場所ごとに変えられないのです。交差点のマスでは地図を強く見たいが、何もない直線では地図に頼りたくない、という場所依存の重み付けができません。すべてのマスに一律で地図を足してしまいます。

そこでクロスアテンション (cross-attention) を使います。アテンションは「問い合わせ (query) が、候補 (key/value) のどれにどれだけ注目するか」を動的に計算する仕組みです。画像 BEV を query、地図 BEV を key/value にすれば、画像側の各マスが「自分にとって関連する地図情報」を選んで取り込めます。交差点のマスは地図の交差点情報を強く引き、空き地のマスはほとんど地図を引かない、という場所ごとの賢い取り込みが可能になります。

## 基礎: 前提となる概念
- query / key / value: アテンションの三役です。query は「何を探しているか」を表すベクトル、key は「各候補が何者か」を表す見出し、value は「実際に取り込む中身」です。query と各 key の類似度 (内積) を計算し、その大小で value を加重平均します。図書館で探したい本のテーマ (query) を、本の背表紙 (key) と照合し、最も関連する本の中身 (value) を読む、という比喩が分かりやすいです。
- スケール済みドット積アテンション: 数式では `Attention(Q,K,V) = softmax(Q·Kᵀ / √d) · V`。`Q·Kᵀ` が query と key の類似度行列、`√d` (d は特徴次元) で割るのは内積が大きくなりすぎて softmax が飽和するのを防ぐため、softmax で行ごとに合計 1 の重みに正規化し、その重みで V を加重平均します。
- セルフアテンションとクロスアテンションの違い: セルフアテンションは query も key/value も同じ系列から作ります (自分の中で関係を見る)。クロスアテンションは query を一方から、key/value を別の系列から作ります (二つの情報源を橋渡しする)。地図融合は後者で、画像と地図という別系列をつなぎます。
- マルチヘッドアテンション (multi-head attention): 注目の計算を複数の独立な「ヘッド」に分け、別々の関係を同時に捉える仕組みです。あるヘッドは「車線数」に注目し、別のヘッドは「分岐」に注目する、といった役割分担が起きます。
- 残差接続 (residual connection): 計算結果を元の入力に足し戻す配線です。`出力 = 入力 + 変換(入力)`。地図情報が役に立たないときに元の画像特徴をそのまま残せるので、学習が安定します。
- Pre-norm: 正規化 (LayerNorm) を変換の前にかける配置です。深いネットワークでの学習を安定させる定番です。
- FFN (Feed-Forward Network): 各トークン (マス) を独立に通す 2 層の全結合で、表現を非線形に豊かにします。
- 変形アテンション (deformable attention): 各 query が key 系列全体ではなく、学習で選んだ少数の参照点だけに注目する仕組み。計算量を劇的に下げます (後述)。

## 仕組みを詳しく
クロスアテンション融合の典型的な処理を段階で追います。入力はどちらも形 `(B, C, H, W)` の BEV 特徴とします。B = バッチ数、C = チャンネル (特徴次元、例 256)、H,W = BEV グリッドの縦横マス数です。

- `image_bev`: 画像 BEV 特徴 → query になる
- `map_bev`: 地図 BEV 特徴 → key と value になる (空間サイズ H,W は image_bev と一致が条件)

ステップ 1: 平坦化 (flatten)。`(B, C, H, W)` を `(B, H*W, C)` に並べ替えます。H×W 個のマス目を「トークン (系列の 1 要素)」とみなし、各トークンが C 次元ベクトルを持つ系列に変換します。たとえば H=200, W=200 なら 40,000 トークン、H=450, W=300 なら 135,000 トークンです。

ステップ 2: Pre-norm クロスアテンション + 残差。

```
q_n  = LayerNorm(q)                          # query (画像BEV) を正規化
kv_n = LayerNorm(kv)                          # key/value (地図BEV) を正規化
attn = MultiheadAttention(q_n, kv_n, kv_n)   # 例 8 ヘッド
q    = q + attn                              # 残差接続 (元の q に足し戻す)
```

各 query トークン (画像 BEV のマス) が、全 key トークン (地図 BEV のマス) との類似度を計算し、その重みで value を加重平均して取り込みます。残差接続は「地図が役に立たなければ元の画像特徴を残す」安全弁です。

ステップ 3: FFN + 残差。

```
q   = q + FFN(LayerNorm(q))
FFN = Linear(C -> 4C) -> GELU -> Dropout -> Linear(4C -> C) -> Dropout
```

各トークンを独立に変換して表現を豊かにします。Dropout は学習時にランダムに一部をゼロにして過学習を防ぎます。

ステップ 4: 元の形に戻す。`(B, H*W, C)` を `(B, C, H, W)` に戻して返します。出力は「地図文脈で補強された画像 BEV」です。

比較用の軽量版 (残差ゲート融合):

```
output = image_bev + alpha * map_bev    # alpha は (C,) のチャンネルごと学習パラメータ、初期値ゼロ
```

`alpha` はチャンネルごとの学習可能なゲートで、初期値ゼロから始めます。学習開始時は地図の寄与が完全にゼロ = 「地図なしと同じ挙動」から始まり、学習が進むにつれ役立つチャンネルだけ `alpha` が育ちます。これは「新機能を足しても最初は既存挙動を壊さない」ための安全な初期化 (ゼロ初期化ゲート、ResNet の残差ブロックや一部の正規化研究で有効性が示された手法) です。クロスアテンション版より計算は軽いですが、場所依存の選択はできません。

数値例で計算量の違いを実感します。クロスアテンションは query 数 × key 数の類似度行列を作るので、トークン数 N = H*W に対して計算量・メモリが O(N^2) です。N=40,000 なら行列は 16 億要素、N=135,000 なら 182 億要素となり、後者は単一 GPU のメモリを容易に超えます (fp16 で 182 億要素はおよそ 36GB)。一方ゲート融合は要素ごとの足し算なので O(N) です。これが「賢いが重い」アテンションと「素朴だが軽い」足し算のトレードオフの正体です。後述の変形アテンションは、各 query が固定数 K 個 (例 4 点) だけ参照することで計算量を O(N·K) に落とします。

座標合わせの重要性: 画像 BEV と地図 BEV は「同じグリッドの同じマスが同じ実世界位置」を指していないと融合が破綻します。自車を中心に、進行方向を上に固定し、メートル/マスの解像度を揃える、という座標規約 (ego-centric な BEV) を両者で一致させる前処理が前提です ([ego中心ラスタタイル](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0414_ego-centric-raster-tile) [GPS→BEV地図タイルレンダリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0417_gps-to-bev-map-tile-rendering))。ここがずれると、交差点のマスに別の場所の地図情報を注入してしまい逆効果になります。

## 手法の系譜と主要論文
- Vaswani, A. et al. "Attention Is All You Need" (NeurIPS 2017, arXiv:1706.03762)。アテンションとクロスアテンションの原典です。query が key/value に動的に注目する仕組みを提案し、再帰や畳み込みを使わずに系列の長距離依存を一度に捉えられることを示しました。トレードオフは計算量がトークン数の二乗 O(N^2) に増えることで、地図融合でも H*W が大きいと重くなります。

- Zhu, X. et al. "Deformable DETR" (ICLR 2021, arXiv:2010.04159)。各 query が特徴全体ではなく、学習で決めた少数の参照点 (例 4 点) だけに注目する変形アテンションを提案しました。計算量を O(N²) から実質 O(N·K) (K は参照点数) に下げ、高解像度の空間特徴へアテンションを適用可能にしました。後続の BEV 研究の多くがこの仕組みを採用しています。

- Li, Z. et al. "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers" (ECCV 2022, arXiv:2203.17270)。BEV グリッドの各セルを query とし、複数カメラの画像特徴に spatial cross-attention で問い合わせて BEV を構築しました。「BEV セルが query になって外部情報を引く」という発想は、地図 BEV を引く融合と同型です。計算量を抑えるため、各 BEV セルが画像全体ではなく対応する少数の点だけを参照する変形アテンション (Deformable DETR 由来) を使った点が工夫です。さらに過去フレームの BEV を参照する temporal self-attention で時間方向の情報も統合しました。

- Liu, Z. et al. "BEVFusion: Multi-Task Multi-Sensor Fusion with Unified Bird's-Eye View Representation" (ICRA 2023, arXiv:2205.13542) および Liang et al. の同名研究 (NeurIPS 2022)。カメラと LiDAR の特徴を共通の BEV 空間で融合する枠組みです。複数モダリティを BEV で足し合わせる思想は、ここで地図 BEV を足すのと同型です。前者は BEV へのビュー変換 (画像→BEV の投影) を高速化する最適化を施し、推論を大幅に高速化した点が実用上の貢献です。単純連結は実装が容易だが場所依存の重み付けができないというトレードオフを共有します。

- Gao, J. et al. "VectorNet" (CVPR 2020) や Li et al. "HDMapNet" (ICRA 2022) など、地図情報を運転モデルに注入する研究群。地図を事前情報として与えると分岐・車線数など見えない構造を補えますが、地図の誤り・古さがそのまま誤情報として入るトレードオフがあります。VectorNet は地図をラスタ画像でなくベクトル (折れ線の集合) として扱い、グラフニューラルネットで関係を捉える方向を開きました。HDMapNet は事前地図に頼らず車載センサから車線・道路境界をオンライン推定する方向を示しました。これらはラスタ BEV 融合と並ぶもう一つの地図注入の系譜です。

## 論文の実験結果 (定量データ)
- BEVFormer (Li et al. 2022) は nuScenes (自動運転の標準ベンチマーク、6 カメラ・LiDAR の大規模データ) の 3D 物体検出で、当時の純カメラ手法を上回り、テストセットでおよそ NDS 56.9% (nuScenes Detection Score、検出の総合指標、値が大きいほど良い) を達成しました。アブレーションでは、時間方向のアテンション (過去フレームの BEV を参照する temporal self-attention) を外すと、とくに速度推定や遮蔽下の検出が悪化することが示され、アテンションによる情報統合の寄与が定量的に裏づけられました。空間 cross-attention を変形アテンションにすることで、全画素参照に比べ計算量を大幅に削減しつつ精度を保てたことも報告されています。

- BEVFusion (2022/2023) は nuScenes の検出・セグメンテーションで、カメラ単独・LiDAR 単独より両者の BEV 融合が明確に高い性能 (検出で mAP・NDS の双方が向上、BEV セグメンテーションでも大幅改善) を示しました。アブレーションの要点は、融合を BEV 空間で行うことが、画像空間や点群空間で個別に処理するより頑健 (片方のセンサが劣化しても他方が補える) という点で、これは地図 BEV を足す設計の頑健性の根拠とも読めます。Liu らの版はビュー変換の最適化で推論を従来比で数倍高速化したことも報告しました。

- 地図注入の効果は軌跡予測ベンチ (nuScenes prediction、Argoverse など) で広く検証されており、地図 (車線中心線・接続関係) を入力に加えると、地図なしのベースラインに比べ予測軌跡の最終地点誤差 (FDE, Final Displacement Error、予測終点と正解終点の距離) や平均誤差 (ADE, Average Displacement Error) が大きく下がることが多数報告されています。直感どおり「道がどこに続くか」を知ると軌跡予測が当たりやすくなる、という効果が数値で確認されています。

指標の意味を補足します。mAP・NDS は検出の正確さ、ADE/FDE は予測軌跡が正解からどれだけ離れているか (小さいほど良い) を測ります。mAP (mean Average Precision) は検出の適合率を全クラスで平均したもの、NDS はそれに位置・速度・属性などの誤差項を加えた nuScenes 独自の総合点です。地図融合の価値は、これらの指標が地図なし比でどれだけ改善するか、特に遠方・遮蔽・分岐といった「画像だけでは不確実な領域」での改善幅に現れます。

## メリット・トレードオフ・限界
メリット:
- 画像 BEV の各マスが「自分に関連する地図情報」を場所ごとに選んで取り込める (足し算より賢い)。
- 残差 + pre-norm で学習が安定。ゼロ初期化ゲートの残差版なら既存挙動を壊さず段階的に地図を導入できる。
- 地図の道路構造・分岐・車線情報で、見えない/遮蔽された領域を補える。遠方の経路予測が安定する。
- マルチヘッドにより、車線数・分岐・接続関係など異なる地図属性を別ヘッドで同時に拾える。

トレードオフ・限界:
- アテンションは計算量・メモリが O((H*W)^2) で、高解像度 BEV (例 450x300 で 13.5 万トークン) だと単一 GPU を超えうる。変形アテンションや低解像度化で緩和するが、解像度と計算コストのトレードオフは残る。
- 地図が古い/誤っていると誤情報をそのまま注入してしまう。地図への過度な依存はかえって危険 (地図と現実が食い違う工事区間など) で、地図と知覚の整合性チェックが研究課題。
- 画像 BEV と地図 BEV の空間サイズ・座標系・自車向きを揃える前処理 (位置合わせ、レジストレーション) が必須。ここがずれると融合は逆効果。
- 残差ゲート版は軽いが場所依存の選択ができない (精度と速度の引き換え)。どちらを使うかは計算予算次第。
- アテンションは「ショートカット」も学びやすく、地図情報を本当に使っているのか、それとも画像だけで決めて地図は無視しているのかを切り分けるには注意重みの分析やアブレーションが要る。

## 発展トピック・研究の最前線
研究の最前線では、アテンションの二乗計算量を避ける工夫が中心課題です。BEVFormer の変形アテンション (各 query が少数の参照点だけ見る) のほか、地図を密なラスタでなく疎なベクトル (車線中心線のポリライン) として扱い、トークン数を数万から数百に圧縮する方向 ([SDTagNet](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0434_sdtagnet) のようなベクトル地図エンコーダ) が有力です。ベクトル化すると地図トークンが激減するので、画像 BEV セルがベクトル地図トークンに cross-attend しても計算が軽く済みます。

もう一つの流れは、地図をオンライン推定する方向です。事前地図を使わず、その場のカメラ・LiDAR から車線・道路構造をリアルタイムに推定 (オンラインマッピング、HDMapNet 系) し、推定地図を画像 BEV に融合します。事前地図の古さ・誤りという弱点を回避できますが、推定地図自体の誤りという新たな不確実性を抱えます。事前地図とオンライン推定をどう組み合わせ、どちらをどれだけ信頼するか (信頼度に応じた融合) が活発に研究されています。

加えて、融合した特徴を下流の何に使うか (検出・セグメンテーション・軌跡予測・行動計画) によって最適な融合方式が変わるため、マルチタスクで融合表現を共有する設計 ([補助ヘッド/マルチタスク学習](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0917_auxiliary-head-multitask)) との組み合わせも進んでいます。さらに、End-to-End 自動運転 (知覚から計画までを一つのネットで貫く) の文脈では、地図融合した BEV を計画モジュールへ直接渡す設計が主流になりつつあり、地図情報がどの段階でどれだけ計画に効くかの分析が新たな研究テーマになっています。
