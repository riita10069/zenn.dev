# 独立マップエンコーダ

## ひとことで言うと
地図を「カメラ画像の一種」として画像と一緒に処理するのではなく、地図専用のニューラルネットワーク(エンコーダ)で別々に処理して特徴に変え、そのあとでカメラ由来の特徴と融合する設計です。「地図は最初から上から見た図 (BEV) なので、わざわざカメラ扱いして座標変換するのは無駄」という考え方が根っこにあります。マルチモーダル(複数種類の入力)を扱う end-to-end 自動運転で広く採られるアーキテクチャ思想です。

## 直感的な理解
複数の入力源を1つのネットワークにまとめる方法は2つ考えられます。1つは全部を同じ入り口(同じエンコーダ)に通す「早期融合 (early fusion)」、もう1つは入力ごとに最適なエンコーダで別々に特徴化してから混ぜる「中期/後期融合 (mid/late fusion)」です。

地図とカメラを早期融合しようとすると不都合が起きます。カメラ画像は地面を横から斜めに見た映像なので、上から見た平面図 (BEV) に直すには大がかりな幾何変換が要ります。一方、ナビ地図や HD 地図は最初から真上から見下ろした図、すなわちすでに BEV です。すでに BEV のものを、わざわざカメラ画像と同じ列に並べて「カメラから BEV を作る」変換ネットに通すのは、出来上がっている料理をもう一度生の材料に戻して調理し直すようなものです。だから地図は地図専用のエンコーダで素直に BEV 特徴にし、カメラから苦労して作った BEV 特徴と「同じ BEV 空間どうし」で混ぜればよい、という発想に至ります。

もう1つの直感は「入力の性質が違えば、適した道具も違う」ことです。カメラ画像はテクスチャ豊富な稠密な絵で CNN/ViT が向き、ベクトル地図は疎で構造的なデータでグラフ/Transformer が向きます。無理に1つのエンコーダで両方を捌くより、それぞれ得意な処理器に分けたほうが性能も出やすく、保守もしやすいのです。

## 基礎: 前提となる概念
用語を平易にします。

エンコーダ (encoder) とは、生の入力(画像・地図・点群など)を、後段の計算で扱いやすい「特徴 (feature)」、すなわち意味を圧縮した数値の塊に変換するネットワーク部分です。

特徴マップ (feature map) は、画像のように縦横の格子を持ち、各格子(画素に相当する位置)に複数チャンネルの数値ベクトルが入ったテンソルです。形は `(C, H, W)`、バッチを入れると `(B, C, H, W)` と書きます。C はチャンネル数(各位置が持つ特徴の次元)です。

BEV (Bird's Eye View) は「鳥が真上から見下ろした図」、つまり地面を上から見た平面図です。自動運転では複数カメラ画像を変換して「上から見た特徴マップ」を作り、これを image BEV と呼びます。画素の位置が実世界の地面座標(メートル単位)に対応する点が重要で、これによって距離や占有を直接表現できます。「メトリックな BEV」とは、特徴マップの (i,j) 座標が実距離(例えば 1 ピクセル = 0.5m)に対応している BEV を指します。

融合 (fusion) とは、異なる入力源から得た特徴を1つにまとめる操作です。最も単純なのはチャンネル方向に並べる連結 (concat) や要素ごとの加算、より高度なものに attention(後述)があります。

cross-attention は、ある特徴 X(質問役 query)が別の特徴 Y(答えの素材 key/value)を参照して、X の各位置が Y のどこに注目すべきかを学習で決め、Y の情報を取り込む仕組みです。同じ情報源の中で注目し合う self-attention に対し、異なる情報源どうしを結ぶのが cross-attention です。

## 仕組みを詳しく
典型的な独立マップエンコーダの構成は、おおよそ3つの部品からなります。

部品1: マップエンコーダ本体。入力は BEV のナビ地図画像、例えば `(B, 3, 256, 256)` のラスタ画像、あるいはベクトル地図のポリラインテンソル `(B, M, 20, 2)` です([ラスタ地図とベクトル地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0433_rasterized-vs-vectorized-maps))。ラスタの場合、中身は画像用バックボーン(CNN や Vision Transformer)で、各段階の特徴マップを取り出し、適応プーリング (adaptive pooling) で同じ空間サイズにそろえ、チャンネル方向に連結し、`1x1` の畳み込みでチャンネル数を融合先と同じ埋め込み次元 (embed_dim, 例えば 256) にそろえます。出力は `(B, 256, h, w)` で、`(h, w)` は image BEV の特徴マップと同じ大きさに合わせます。形を合わせないと後段の融合で次元が衝突するためです。

数値例で見ると次の通りです。

```
地図画像  (B, 3, 256, 256)
   │  画像バックボーン + features_only(多段の中間特徴を取り出す)
   ▼
多段特徴  [(B,c1,64,64), (B,c2,32,32), (B,c3,16,16), ...]
   │  adaptive pool で (h,w) にそろえる + concat
   ▼
連結特徴  (B, c1+c2+c3+..., h, w)
   │  1x1 Conv + 活性化
   ▼
map BEV   (B, 256, h, w)
```

多段特徴を集めて束ねるのは、浅い層が捉える細かい模様(道の縁・線)と深い層が捉える大局構造(交差点全体の配置)の両方を地図特徴に取り込むためです。

部品2: 融合モジュール。image BEV と map BEV をどう混ぜるかには複数方式があります。代表的なのは次の2つです。

残差融合 (residual fusion): `fused = image_bev + alpha * map_bev`。`alpha` をゼロ初期化の学習パラメータにすると、学習初期は地図を無視(恒等写像)し、学習が進むにつれて必要なぶんだけ地図情報を足し込みます。既存のカメラ系を壊さずに地図を導入できるのが利点です。詳細は [ゼロ初期化残差によるマップ融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0464_umpe-residual-map-fusion)。

cross-attention 融合: image BEV を query、map BEV を key/value として cross-attention で混ぜます。地図のどの位置をどれだけ参照するかを学習で適応的に決められる柔軟さがありますが、計算量は残差加算より大きくなります(BEV の解像度を H×W とすると、素朴な実装で (HW)^2 オーダーの注意計算がかかるため、変形可能注意 (deformable attention) などで軽くするのが一般的)。

部品3: 全体への組み込み。順方向 (forward) の流れは次のようになります。

1. カメラ画像 → view transform(複数視点を統合し BEV を作る変換) → image BEV `(B, 256, h, w)`
2. 地図 `(B, 3, 256, 256)` または `(B, M, 20, 2)` → マップエンコーダ → map BEV `(B, 256, h, w)`
3. 2つを融合モジュールで混ぜる → 融合 BEV `(B, 256, h, w)`
4. 融合 BEV を下流のプランナ(走行軌跡を予測する部分)へ渡す

マップエンコーダの出力サイズは、カメラ側の BEV 表現のサイズに合わせて決めます。カメラ側がメトリックな BEV(画素が実距離に対応する)を作る方式なら、地図 BEV もそのグリッドに合わせるのが自然です。逆にカメラ側が抽象的な特徴トークンしか持たない方式だと、地図 BEV と空間的に整合させるのが難しくなります(後述の限界、[地図BEVと画像BEVの座標整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0430_map-image-bev-alignment))。座標整合がずれていると、地図上の「ここは車線」という情報がカメラ特徴の別の場所に足し込まれ、かえって性能を下げます。

学習上の工夫として、地図をランダムに落とす「地図ドロップアウト」(一定確率で map BEV をゼロにして学習する)を入れると、地図が無いシーンでも破綻せず、地図への過依存も抑えられます。これは融合を残差にしておくと特に自然に組み合わせられます。

## 手法の系譜と主要論文
「地図を専用ブランチで処理し、特徴空間で融合する」という考えは、近年の知覚・計画統合モデルで広く採られています。系譜をたどります。

ChauffeurNet (Bansal et al., RSS 2019, Waymo) は、地図・経路・他車・自車過去軌跡を複数のラスタ画像チャンネルとしてレンダリングし、CNN で処理して運転を模倣学習しました。地図を「環境を表す入力チャンネルの1つ」として扱う発想の原型で、地図が運転方策の有力な入力になることを示しました。この時点では地図とカメラを別ブランチに分ける構成ではありませんが、「地図を BEV ラスタとして与える」基礎を作りました。

BEVFusion (Liu et al., ICRA 2023, MIT) は、カメラと LiDAR をそれぞれ別エンコーダで BEV 特徴にしてから、共有 BEV 空間で融合する手法です。地図ではありませんが、「異なる入力源をそれぞれ最適なエンコーダで BEV 化し、共通 BEV で統合する」という本トピックの骨格そのものを示し、設計根拠としてよく引用されます。各モダリティを独立に BEV 化することで、片方が欠けても頑健に動く利点も示しました。

UniAD (Hu et al., CVPR 2023, Best Paper, 上海AI Lab) は、検出・追跡・地図・占有・予測・計画を1本のネットワークに統合し、地図は専用のマップヘッドが BEV 特徴から作ります。地図情報を「計画に役立つ形で」内部表現として持つ設計思想は、独立マップエンコーダと地続きです。

VAD (Vectorized Autonomous Driving, Jiang et al., ICCV 2023, Horizon Robotics) は、地図を「ベクトル化したインスタンス(車線などを点列で表したもの)」として専用の map モジュールで扱い、地図要素の予測と軌跡計画を一体で学習します。地図要素は画素ではなく幾何構造(線や領域)であり、構造を活かすには専用表現が有効だという考えに基づきます。詳細は [ベクトル地図のポリライン符号化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0465_vectorized-map-polyline-encoding)。

融合の細部に関しては、ゼロ初期化の残差で外部情報を緩やかに足し込むアイデア(条件付け信号を恒等写像から徐々に効かせる)が、ControlNet (Zhang et al., ICCV 2023) など他分野でも有効性が示されており、地図 BEV の残差融合の根拠として参照されます。ControlNet は画像生成で「ゼロ初期化の畳み込み (zero convolution) で条件を注入すると、事前学習済みモデルを壊さず制御を足せる」ことを示し、同じ原理が地図融合にも応用できます。詳細は [ゼロ初期化残差によるマップ融合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0464_umpe-residual-map-fusion)。

## 論文の実験結果(定量データ)
end-to-end 自動運転モデルの評価は、主に nuScenes データセットでの計画指標で行われます。指標の意味を先に説明します。L2 誤差 (L2 error) は、予測した将来軌跡と実際に人間が走った軌跡の、各時刻における位置のずれ(メートル)で、小さいほど良いです。通常 1秒後・2秒後・3秒後の平均で報告されます。衝突率 (collision rate) は、予測軌跡が他の物体とぶつかる割合(パーセント)で、安全性の代理指標です。

UniAD の報告では、地図・知覚・予測を統合した計画により、平均 L2 誤差が当時の代表的ベースラインに対しておよそ 1.0m 前後まで縮み、平均衝突率も 0.3% 前後と、個別タスクを別々に解く構成より明確に低減したことが示されました。これは「地図や他車の情報を計画に役立つ内部表現として共有すると、計画が改善する」ことの定量的証拠です。

VAD の報告では、ベクトル地図表現を用いることで nuScenes 計画タスクで UniAD に匹敵する精度(平均 L2 でおよそ 0.7m 前後、衝突率も低水準)を達成しつつ、ラスタや稠密な BEV を扱う構成に比べて推論が大幅に高速(報告によりベースラインの数倍のフレームレート)になることが示されました。これは「地図を構造のまま扱うと、精度を保ちつつ計算を節約できる」ことを意味します。

アブレーション的な観点では、地図入力(マップブランチ)を丸ごと外すと計画の L2 誤差・衝突率がともに悪化することが各論文で繰り返し報告されており、地図が計画に効く入力であることが裏付けられています。逆に、地図を与えても融合の仕方が悪い(座標が整合していない、強く足しすぎてカメラ特徴を壊す)と改善が出ない、あるいは悪化することもあり、融合設計が結果を左右します。なお nuScenes の計画指標は自車の過去軌跡だけでもある程度当たってしまう(エゴ運動の慣性が強い)ことが後続研究で指摘されており、地図の寄与を厳密に測るには衝突率や、より難しいシーンでの評価、closed-loop(実際に走らせる)評価が望ましい、という点も研究者として押さえておくべきです。

## メリット・トレードオフ・限界
メリット。地図はすでに BEV なので、カメラ扱いして無駄な座標変換をかけずに済みます。地図とカメラを「同じ BEV 空間」で融合でき、空間的な対応づけが自然です(ただし座標整合の前提が必要、[地図BEVと画像BEVの座標整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0430_map-image-bev-alignment))。地図の種類(ラスタ/ベクトル)ごとに最適なエンコーダを差し替えられる拡張性も得られます。片方の入力が欠けても残りで動く頑健性(BEVFusion が示した性質)も、地図ドロップアウト学習と組み合わせて期待できます。

トレードオフと限界。ブランチが増えるぶんパラメータと計算が増えます。融合が成立する前提として map BEV と image BEV の座標系が一致している必要があり、カメラ側が抽象トークンしか持たずメトリックな対応を欠く方式では前提が崩れます([地図BEVと画像BEVの座標整合](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0430_map-image-bev-alignment))。また地図入力そのものを各データセットで用意する必要があり、無ければ GPS からの合成([OSMマップマッチング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0431_map-matching-osm))などの前処理が要ります。地図とカメラの時間的ずれ(地図が古い、位置誤差で数メートルずれる)があると、独立に作った2つの BEV が物理的に整合せず、融合がノイズ源になります。

未解決の課題としては、地図とカメラの不整合や時間的ずれに頑健な融合をどう設計するか、地図を強く信頼しすぎて実際の障害物より地図に従ってしまう「過依存」をどう防ぐか、地図が無いシーンでも性能を落とさない設計をどう作るか、が研究的に重要です。

## 発展トピック・研究の最前線
最近は、HD 地図をあらかじめ用意せず、その場でカメラ・LiDAR から地図要素を推定する「オンライン地図構築」(MapTR 系)を計画と統合する流れが強まっています。これにより地図メンテナンスのコストを避けつつ独立マップ表現の利点を得ようとします。また、地図・知覚・計画を1つの大規模 Transformer やトークン列に統一し、地図も他の要素もトークンとして同列に扱う「トークン化された world model / 行動モデル」への融合も進んでいます。VLM(視覚言語モデル)に地図と画像を同時に与えて運転判断を言語的に説明させる研究も活発で、独立マップエンコーダの考え方はその中で「地図モダリティの専用トークナイザ」として再解釈されつつあります。標準ナビ地図 (SD map) を軽い事前情報としてブランチに与え、精密なオンライン地図推定を補強するハイブリッドも登場しています。
