# BEV Spatial Cross-Attention（鳥瞰図への空間クロスアテンション融合）

## ひとことで言うと
車の周りに付いた複数のカメラ画像を、車を真上から見下ろした地図のような格子(BEV、Bird's-Eye View=鳥瞰図)に貼り直す部品です。BEV の各マス目が「自分はどのカメラのどこを見ればいいか」を、3D空間の座標計算で求めて画像から特徴を吸い上げます。代表的には BEVFormer という手法がこれを「空間クロスアテンション」で実現し、その簡易版を特別なCUDAカーネルなしで作ることもできます。要は、バラバラな一人称視点の画像群を、運転計画に使いやすい1枚の俯瞰地図に統合する変換です。

## 直感的な理解
車には前後左右に複数のカメラが付いています(典型的には6〜7台)。それぞれ別の方向を向いた2D画像を出しますが、運転計画は「車の周囲の地面の上で、どこに何があるか」という統一された空間で考えたい。バラバラな画像のまま扱うと困ったことが起きます。たとえば右カメラに映った車と、後ろカメラに映った同じ車が、別物に見えてしまう。距離も「画像の中で小さく映っているから遠い」程度しか分からず、地面上の正確な位置が出ない。

そこで、すべてのカメラ画像を1枚の鳥瞰図に融合します。BEV は車を中心に上から見た格子で、各マスが地面の決まった範囲(たとえば数十cm四方)に対応します。一度 BEV にしてしまえば、距離・位置・他車の動きを地面座標で素直に扱え、軌道計画や地図との照合がぐっと楽になります。ゲームの俯瞰マップを思い浮かべると分かりやすい。バラバラな一人称視点を、神の視点の1枚絵にまとめる作業です。隣り合うカメラの視野が重なる領域では同じ物体が2つの画像に映りますが、BEV 上では同じマスに統合されるので「二重に数える」問題も自然に解けます。

難しいのは「BEV のあるマス目は、どのカメラのどのピクセルを見るべきか」という対応づけです。これはカメラの位置・向き・レンズの特性(カメラパラメータ)に依存します。空間クロスアテンションはこの対応づけを 3D 幾何で計算します。

## 基礎: 前提となる概念

カメラの内部パラメータ(intrinsic)とは、焦点距離や画像中心など、レンズと撮像素子の性質を表す行列(3×3)です。3D 点を画像のどのピクセルに写すかを決める要素のうち、カメラ自身の性質ぶんです。

外部パラメータ(extrinsic)とは、カメラが車のどこに、どの向きで付いているかを表す行列(回転と並進、4×4 または 3×4)です。車基準の座標を、そのカメラ基準の座標へ移す変換です。内部と外部をまとめて「カメラ行列」「射影行列」と呼びます。

透視投影(perspective projection)とは、3D の点を 2D 画像に写す計算で、現実のカメラの「遠くのものは小さく写る」性質を表します。3D 点を奥行き(depth)で割ることでこれを再現します。割り算が入るため、これは線形変換ではなく射影変換(同次座標を使う)です。

クエリ(query)とは、attention(注意機構、入力のどこに注目するかを重みで決める仕組み)で「私はこういう情報が欲しい」と表明するベクトルです。BEV の各マスがクエリを持ち、対応するカメラ特徴を引き寄せます。

deformable attention(変形可能注意)とは、画像全体を見るのではなく、少数の「注目すべき点」だけをサンプリングして特徴を集める軽量な注意機構です(Deformable DETR で提案)。普通の attention が全ピクセルとの関係を計算して重い(計算量がピクセル数の二乗)のに対し、数点だけ見るので線形に近い計算量で済みます。

grid_sample とは、特徴マップの任意の座標(整数でなくてよい)の値を、近傍ピクセルの加重平均(双線形補間、bilinear interpolation)で滑らかに取り出す標準操作です。専用 CUDA カーネルを書かずに「特徴を吸い上げる」処理を実装できます。座標は通常 [-1, 1] に正規化して渡します。

## 仕組みを詳しく

学習可能 BEV クエリと 3D 参照点を使う空間クロスアテンション融合を、形を追いながら分解します(設定例: BEV グリッド 450×300、特徴次元 256、カメラ8視点、柱あたりサンプル点4)。

第1に、学習可能な BEV クエリを用意します。`bev_queries = Embedding(bev_h * bev_w, embed_dim)`。BEV の全マス(450×300=135000個)それぞれに、長さ256の問い合わせベクトルを持たせます。各マスが「私はこの地面位置の情報が欲しい」と表明する役です。これらはデータから学習され、初期はランダムですが、訓練を通じて「道路上のこの位置はこういう特徴を集めるべき」という事前知識を蓄えます。

第2に、各 BEV マスに 3D 参照点(reference points)を立てます。地面の同じ位置でも高さによって映り方が違うので、高さ方向に複数点(柱、pillar)を取ります。たとえば高さ4点。これらは自車(ego)座標系の 3D 点で、知覚範囲 `pc_range`(例: 前方120m・左右±60m・上下 −5〜3m)を使って正規化座標を実空間座標へ変換します。形は `[N, num_z, 3]`(N=135000, num_z=4)。高さ方向に複数点を取るのは、車・歩行者・標識など高さの違う物体を1本の柱でまとめて拾えるようにするためです。

第3に、各 3D 点を各カメラの画像平面へ射影します(透視投影)。カメラパラメータ `[B, num_views, 3, 4]`(内部 @ 外部 をまとめた、3D ego 座標をピクセルへ写す行列)で一括計算し、得た `[u, v, depth]` のうち depth で割って透視投影を再現、ピクセル座標を画像サイズで割って [0,1] に正規化します。

```
einsum('bvij, nj -> bvni', proj, ref_flat)   # 全点・全カメラを一括投影
 → [u, v, depth]
 → u/depth, v/depth   (透視投影)
 → 画像サイズで割って [0,1] 正規化
```

第4に、可視性マスクを計算します。これが幾何的に重要です。たとえば「車の前方の点」は後ろ向きカメラには絶対に映りません。そこで `mask = (depth > 0) AND (0 <= u,v <= 1)`、つまり「カメラの前方にあり、かつ画像の枠内に入る」点だけを有効とします。映らないカメラから無理に特徴を吸わない、という物理的に正しい制約です。`depth > 0` はカメラの背後(負の奥行き)にある点を弾く条件で、これを忘れると後ろにある物体が前方カメラに「幽霊」として射影されてしまいます。

第5に、各 BEV クエリからサンプリングのずれ(sampling offsets、参照点をどれだけずらすか、小さく抑制する)と注意重み(attention weights、柱の各点・各カメラのどれを重視するか、softmax で正規化)を予測します。これが deformable な要素で、剛直な格子点だけでなく学習で微調整した位置を見ます。キャリブレーション誤差や物体の微妙な位置ずれを、このオフセットがある程度吸収します。

第6に、`grid_sample` で各カメラの特徴マップから「参照点+オフセット」の位置の特徴を双線形補間で取得します。BEVFormer 本家は deformable attention の専用 CUDA オペを使いますが、標準の grid_sample で代替すれば特別なカーネルが不要になり、CPU でも GPU でも動きます(代わりに本家のマルチヘッド deformable attention の表現力は完全には再現しません)。

第7に、各 BEV マスについて、可視だったカメラ群から吸い上げた特徴を、命中カメラ数で割って平均します。BEVFormer の空間クロスアテンション式 `output = (1/|V_hit|) Σ_{i ∈ V_hit}(...)` に対応します(`V_hit` は命中したカメラ集合)。どのカメラからも見えないマスはゼロにします。命中カメラ数で割るのは、重なり領域(2台に映る)と単独領域(1台のみ)でスケールが揃うようにするためです。

第8に、Pre-Norm Transformer 構成で整えます。`output = queries + output_proj(output)` の残差加算、LayerNorm、FFN(全結合の小ネット)+残差、LayerNorm。残差加算は勾配を流しやすくし、深い積み重ねを安定させます。

第9に、`[B, embed_dim, bev_h, bev_w]` の鳥瞰図特徴に整形して返します。

```
全体の流れ:
マルチカメラ特徴 [B, V, C, H, W]
  ＋ BEV クエリ [bev_h*bev_w, C]
  ＋ 3D 参照点 → 各カメラへ射影 → 可視性マスク
  → grid_sample で特徴吸い上げ → 可視カメラ平均
  → Pre-Norm Transformer
  → BEV 特徴 [B, C, bev_h, bev_w]
```

カメラパラメータが無いとき(キャリブレーション未取得時)は、学習可能なダミー射影行列(pseudo projection)で形だけ通すことがあります。これは本物のカメラ幾何ではないので意味のある BEV にはならず、データセット選定前にコードの配線確認のため学習を回すための仮置きにすぎません。本番には実キャリブレーションが必須です。

## 手法の系譜と主要論文

マルチカメラから BEV を作る研究には2つの大きな系譜があります。

ひとつは「深度を推定して持ち上げる」系です。Philion & Fidler, "Lift, Splat, Shoot: Encoding Images from Arbitrary Camera Rigs by Implicitly Unprojecting to 3D"(ECCV 2020)は、各ピクセルの深度を離散分布として推定し、画像特徴を 3D に持ち上げ(lift)、BEV 平面に落とし込む(splat)方式を提案しました。明示的に深度を扱うので直感的ですが、深度推定の誤差がそのまま BEV に乗ります。後の BEVDet/BEVDepth がこの系譜を高速・高精度化しました。

もうひとつが「クエリで吸い上げる」系で、これが空間クロスアテンションの直接の系譜です。Zhu et al., "Deformable DETR: Deformable Transformers for End-to-End Object Detection"(ICLR 2021)が、全ピクセルではなく少数の参照点だけサンプリングする deformable attention を提案し、計算を大幅に減らしました。Li et al., "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers"(ECCV 2022)は、これを使って「学習可能な BEV クエリが、3D 参照点を各カメラに射影し deformable attention で特徴を集める」空間クロスアテンションを提案しました。動機は、明示的な深度推定なしに 2D 画像から BEV を作りたかったこと。さらに過去 BEV を参照する時間自己アテンション(temporal self-attention)も備え、時空間 Transformer として速度推定も可能にしました。

Liu et al., "PETR: Position Embedding Transformation for Multi-View 3D Object Detection"(ECCV 2022)は、明示的な射影の代わりに 3D 位置情報を画像特徴へ埋め込み、クエリと暗黙的に対応づける別アプローチを示しました。射影を明示するか暗黙にするかという設計上の分岐です。Huang & Huang, "BEVDet4D"(arXiv:2203.17054, 2022)は過去 BEV を連結して時間情報を取り込みました。

Hu et al., "Planning-oriented Autonomous Driving"(UniAD, CVPR 2023, ベストペーパー)は、BEVFormer を BEV エンコーダとして使い、検出・追跡・地図・予測・計画を1本のネットに統合し、BEV 表現が計画指向のエンドツーエンド運転に有効であることを示しました。Jiang et al., "VAD: Vectorized Scene Representation for Efficient Autonomous Driving"(ICCV 2023)は BEV を経由しつつベクトル化表現で効率を上げました。

## 論文の実験結果(定量データ)

BEVFormer(Li et al., 2022)は nuScenes(大規模マルチカメラ自動運転データセット)で評価されました。主要指標 NDS(nuScenes Detection Score、検出精度に速度・向き・属性の誤差を統合した0〜1のスコア、高いほど良い)で、当時のカメラのみ手法の最高水準を更新し、NDS おおよそ0.56前後を達成したと報告されています。とくに、時間自己アテンション(過去 BEV の参照)を加えることで速度推定誤差 mAVE(mean Average Velocity Error、単位 m/s、小さいほど良い)が大きく下がり、これは「時間情報が速度推定に効く」という一般法則とも一致します。

アブレーションでは、空間クロスアテンションを単純な平均プーリングや全結合での融合に置き換えると mAP(mean Average Precision、検出の平均精度、高いほど良い)・NDS がともに低下し、「3D 幾何に基づく射影 + deformable サンプリング」が効いていることが確認されています。可視性に基づくカメラ選択(見えないカメラを除外)も、ノイズ混入を防いで精度に寄与します。指標の意味は、mAP が「どれだけ多くの物体を正しい位置で当てたか」、NDS が「それに加えて速度・向き・サイズ・属性の推定誤差まで総合した点」です。

Lift-Splat-Shoot(Philion & Fidler, 2020)は nuScenes の BEV セマンティックセグメンテーション(BEV 上で各マスが道路/車/歩行者などどれかを当てる課題)で、複数カメラを幾何的に統合する有効性を IoU(Intersection over Union、予測領域と正解領域の重なり率、高いほど良い)で示しました。深度推定型はクエリ型より深度誤差に敏感である一方、明示的な深度を出力できる利点があり、両系譜は用途で使い分けられます。

UniAD(Hu et al., 2023)は nuScenes で、検出・追跡・予測・占有・計画を統合した結果、計画の衝突率(collision rate、低いほど良い)と L2 誤差(予測軌道と正解軌道の距離、単位 m、低いほど良い)の両方で、タスクを別々に解くベースラインを上回りました。報告では平均 L2 誤差と3秒先の衝突率がともに改善し、これは BEV 表現を共有して各タスクを協調させることの有効性を示し、BEV 融合を運転の中核に据える動機を裏づけます。

数値の読み方として、NDS が数ポイント(0.01〜0.03)上がることは検出・追跡の質が実用上はっきり改善することを意味し、mAVE の低下は「他車がどれだけ速く動いているか」をより正確に当てられることを意味します。後者は衝突回避に直結する、安全上重要な量です。

## メリット・トレードオフ・限界

メリットの第1は、マルチカメラを1枚の鳥瞰図に統合でき、距離・位置・他車を地面座標で素直に扱えること。計画や地図照合に向きます。

メリットの第2は、可視性マスクで物理的に見えないカメラを除外するため、幾何的に正しい融合になること。明示的な深度推定を必要としないので、深度誤差の伝播を避けられます。

メリットの第3は、grid_sample で実装すれば deformable attention の専用 CUDA を使わずに済み、移植性が高く環境を選ばないこと。

トレードオフの第1は計算とメモリの重さです。BEV マス数(数万〜十数万)× カメラ数 × 柱点数のサンプリングが必要で、高解像度ほど VRAM を食います。グリッドを細かくすると遠方の小物体まで表現できる反面、メモリと計算がグリッド辺長の二乗で増えます。450×300 のような高解像度では、混合精度や勾配チェックポイントが事実上必須になります。

トレードオフの第2はキャリブレーション依存です。正しく機能するには実カメラの内部・外部パラメータが必須で、ダミー射影では意味のある BEV になりません。キャリブレーション誤差は BEV の位置ずれに直結し、特に遠方ほど誤差が拡大します。

トレードオフの第3は、deformable attention を grid_sample で簡略化した場合、本家のマルチヘッド表現力にやや劣ること。

トレードオフの第4は高さ方向の粗さです。深度を明示推定しない代わりに柱(複数高さ)のサンプリングに頼るので、高さ方向の表現は柱点数の粗さに制約されます。柱点を増やせば改善しますがコストが上がります。

未解決の課題としては、悪天候・逆光・キャリブレーションずれへの頑健性、遠方領域での解像度不足、動的な物体と静的な背景を BEV 上でどう区別するか、カメラ単独では原理的に曖昧な絶対スケール(大きな遠い物体と小さな近い物体の区別)をどう詰めるか、といった点が残っています。

## 発展トピック・研究の最前線

第1の最前線は、時間 BEV です。BEVFormer の時間自己アテンションや、過去 BEV を自車運動で位置合わせして連結する方式(BEVDet4D 系)で、速度・動きの推定を強化する流れです。複数フレームを統合することで、瞬間的なオクルージョン(他車に隠れる)にも頑健になります。

第2は、効率化です。BEVPoolv2 などの高速化された lift 演算や、スパースなクエリ(BEV 全マスではなく物体候補だけクエリ化する Sparse 系、SparseBEV など)で、計算とメモリを削減する研究が進んでいます。密なグリッドを捨てることで高解像度化のコスト爆発を避けます。

第3は、エンドツーエンド計画との統合です。UniAD や VAD のように BEV を共有表現として検出・予測・計画を1本に繋ぎ、勾配を計画まで通して全体最適化する方向が、自動運転の主流になりつつあります。

第4は、LiDAR や地図との融合です。カメラ BEV と LiDAR BEV を同じ格子上で統合し、互いの弱点(カメラの深度曖昧さ、LiDAR の意味情報不足)を補い合うマルチモーダル BEV(BEVFusion など)が活発に研究されています。同じ BEV 格子という共通土俵があるからこそ、異種センサの融合が素直に書ける点が BEV 表現の強みです。
