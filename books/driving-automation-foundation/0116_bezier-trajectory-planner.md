# ベジェ曲線トラジェクトリプランナー

## ひとことで言うと
車が次にどう動くか(各時刻の制御指令の並び)を、1時刻ずつ全部直接予測するのではなく、少数の「制御点」だけをニューラルネットに予測させ、それを数学的に滑らかな曲線(ベジェ曲線)に展開して全時系列を生成するプランナー(計画器)です。曲線そのものが滑らかなので、後処理の平滑化フィルタや滑らかさを罰する損失項を足さなくても、ガクガクしない低ジャークな運転指令が「作りによって(by construction)」必ず得られます。

## 直感的な理解
手で滑らかな曲線を描く場面を想像してください。フリーハンドで点を 64 個プロットしてつなぐと、手が震えてガタガタになりがちです。一方、デザインソフトで曲線を引くときは、数個の「ハンドル(制御点)」を置いてドラッグするだけで、ソフトが間を滑らかな曲線で埋めてくれます。どんなにハンドルを動かしても、出てくる線は滑らかなままです。

ベジェ軌道プランナーはこの後者の発想です。ネットには「ハンドルの位置(制御点)」だけを予測させ、間の埋め方は固定の数学(ベジェ曲線)に任せます。すると、ネットがどんな制御点を出しても、できあがる軌道は必ず滑らか。乗り心地が悪くなる急なカックン(高いジャーク)が、設計段階で原理的に排除されます。素朴に各時刻を直接予測する方式だと、隣り合う時刻がバラバラに暴れてギザギザになり、後から平滑化する手間と副作用(応答遅れ)が付きまといます。それを丸ごと回避するのが狙いです。

## 基礎: 前提となる概念
言葉を噛み砕きます。

- トラジェクトリ (trajectory, 軌道): 車がこれからたどる動きの計画。実装によって「各時刻の (x, y) 位置の列」だったり「各時刻の制御指令の列」だったりします。後者を採る設計が増えています(理由は後述)。
- 制御指令 (control signal): 車に出す物理的な命令。よくある選択は (加速度, 曲率) の2チャンネルです。曲率 (curvature) はカーブの曲がり具合で、ハンドルの切れ角に対応します。生の (x, y) 位置ではなく制御指令を出力すると、運動方程式を通して物理的に実現可能な動きしか生成できないという利点があります。
- ジャーク (jerk): 加速度の時間変化率(加速度の微分)。ジャークが大きいと「急に効き方が変わる」乗り心地の悪い・危険な動きになります。滑らかな運転とは低ジャークの運転です。
- プランナーヘッド (planner head): モデルの末端に付く、抽出済みの特徴から軌道を出力する部分。
- ベジェ曲線 (Bezier curve): 少数の制御点と固定の重み関数(Bernstein 基底、詳細は [Bernstein多項式基底](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0925_bernstein-polynomial-basis))から定義される滑らかな曲線。CAD やフォント設計で広く使われます。制御点の凸包(制御点を囲む最小の領域)の中に必ず収まり、暴れません。
- パラメトリック曲線 (parametric curve): パラメータ t を 0 から 1 まで動かすと点が連続的に動いて軌跡を描く曲線。ベジェ曲線はその代表です。

素朴な「全時刻直接予測」方式の欠陥を具体化します。仮に 64 時刻ぶんの (加速度, 曲率) を直接出すなら、ネットは 64×2 = 128 個の数字を出力します。ネットはこれらを互いにほぼ独立に出すため、隣り合う時刻の値が不連続に飛びやすく、軌道がギザギザになります。低ジャークの保証は一切なく、対策として (1) 出力後に平滑化フィルタをかける、(2) 学習時に「滑らかでないと損をする」追加損失項(smoothness penalty)を足す、のいずれかが必要です。どちらも調整が面倒で、フィルタは強くすると応答が遅れる副作用があり、ペナルティは重みのチューニングが難しい。これらを根本から不要にするのがベジェ方式です。

## 仕組みを詳しく
代表的な構成を、テンソルの形 (shape) を追いながら説明します。典型値として num_timesteps=64(時刻数)、num_signals=2(加速度・曲率)、num_controls=5(制御点数)、特徴次元 embed_dim=256 とします。バッチサイズを B と書きます。

ステップ1: 文脈をまとめる。複数の入力を受け取り、それぞれを共通の特徴次元に射影して足し合わせ、1本の文脈ベクトルにします。たとえば自車の過去の動き、視覚の履歴、俯瞰特徴(BEV, Bird's-Eye-View, 上空から見た特徴マップ)の3つを使う場合、

```
bev_context = bev_features.mean(dim=(2, 3))           # [B, 256, H, W] を空間平均 → [B, 256]
context = ego_proj(egomotion_history)                 # [B, 256]
        + visual_proj(visual_history)                 # [B, 256]
        + bev_proj(bev_context)                       # [B, 256]
bezier_feature = context_mlp(context)                 # [B, 256]
```

BEV を空間方向に平均して 1 ベクトルに潰すのは、計算を軽くする要約です(代わりに空間的にどこが重要かの細かい情報は落ちます)。

ステップ2: 制御点を予測する。文脈ベクトル(256次元)から num_controls × num_signals = 5×2 = 10 個の数字を出し、[B, 5, 2] に整形します。これが「5個の制御点、各点が (加速度, 曲率)」です。

```
control_points = control_head(bezier_feature).view(B, 5, 2)   # [B, 5, 2]
```

ステップ3: 固定のベジェ基底で全時刻に展開する。あらかじめ計算しておいた Bernstein 基底行列(形 [P=64, C=5])を使い、5個の制御点を64時刻ぶんの滑らかな曲線に「広げ」ます。einsum(添字を指定したテンソル積)1行で書けます。

```
# bernstein_basis: [P=64, C=5],  control_points: [B, C=5, S=2]
trajectory = einsum("pc,bcs->bps", bernstein_basis, control_points)   # [B, 64, 2]
```

ここがこの手法の核心です。出力された64時刻ぶんの曲線は、各時刻が「5個の制御点の重み付き平均」になっています。重み(Bernstein 基底)は時刻に対して滑らかに変化する固定値で、隣り合う時刻の重みが急にジャンプできません。したがってネットが制御点をどう動かしても、出てくる軌道は必ず滑らか(低ジャーク)です。これが「by construction(作りによって)」滑らかという意味で、後処理フィルタも滑らかさ損失項も不要になります。before/after で言えば、直接予測では「滑らかさは祈るしかない(後処理で繕う)」、ベジェ方式では「滑らかさは数学的に保証される」。

出力の意味を取り違えないことが重要です。この軌道は (x, y) 座標ではなく制御指令 (加速度, 曲率) のプロファイルである場合が多い。出力した加速度・曲率を unicycle(単輪車)などの運動モデルで時間積分して、初めて実際の位置になります。ベジェで滑らかにしているのは加速度・曲率の時間変化そのものなので、結果としてコントローラに優しく物理的に実現可能な指令になります。詳しくは [単輪車モデル(unicycle / 車両運動学)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0844_unicycle-dynamics) を参照してください。

入力検証も実装上は重要です。num_controls < 2(曲線が定義できない)、num_controls > num_timesteps(時刻より制御点が多い不自然な構成)、各入力履歴の次元不一致などは早期にエラーにすべきです。制御点を5個程度に絞ることには、出力次元128個を直接予測するより学習対象が少なく(10個)、過学習しにくく学習が安定しやすいという副次的な利点もあります。少ないパラメータで滑らかな軌道全体を表す、という低次元パラメータ化の発想です。

## 手法の系譜と主要論文
軌道を低次元のパラメトリック曲線で表す考え方は、運転計画で長い歴史があります。

- Werling et al. (2010), "Optimal Trajectory Generation for Dynamic Street Scenarios in a Frenet Frame" (ICRA 2010) は、Frenet 座標系(道路中心線に沿った座標)で軌道を quintic/quartic(5次/4次)多項式で表し、ジャーク最小の軌道を解析的に生成しました。提案は「軌道をジャーク最小の多項式で表現」、動機は乗り心地と物理的実現可能性、効果は滑らかで追従しやすい軌道、トレードオフは多項式次数が表現力の上限になること。ベジェ/多項式による軌道表現の古典です。
- 学習ベースのアンカー/係数表現が続きます。Chai et al. (2019), "MultiPath: Multiple Probabilistic Anchor Trajectory Hypotheses for Behavior Prediction" (CoRL 2019) は、あらかじめ用意したアンカー軌道集合からの確率と残差を予測し、出力空間を構造化しました。Phan-Minh et al. (2020), "CoverNet: Multimodal Behavior Prediction using Trajectory Sets" (CVPR 2020) は軌道予測を、固定された軌道集合からの分類問題として定式化しました。いずれも「出力の自由度を絞って学習を安定させる」流儀で、ベジェ制御点予測も同じ低次元パラメータ化の一種です。
- Bansal et al. (2018), "ChauffeurNet: Learning to Drive by Imitating the Best and Synthesizing the Worst" (arXiv:1812.03079) は、模倣学習だけだと滑らかさや実現可能性に課題が出ること、補助損失や摂動による頑健化が要ることを示しました。ベジェ方式は「補助損失で滑らかさを促す」のではなく「表現自体を滑らかにする」アプローチで、後処理も追加損失も要らないのが差別化点です。
- 近年の end-to-end 運転、Hu et al. (2023) UniAD (CVPR 2023) や Jiang et al. (2023) "VAD: Vectorized Scene Representation for Efficient Autonomous Driving" (ICCV 2023) は、ベクトル化された軌道を扱いますが、滑らかさの担保はモデルや後処理に依存します。基底で滑らかさを保証する設計は、これらと相補的な明快さを持ちます。

系譜として整理すると、「解析的な多項式最適化(Werling)」→「学習ベースの離散アンカー/分類(MultiPath, CoverNet)」→「end-to-end のベクトル軌道(UniAD, VAD)」と進む中で、ベジェ制御点予測は「学習で連続パラメータ(制御点)を出しつつ、基底で滑らかさを数学的に保証する」位置にあります。

## 論文の実験結果(定量データ)
軌道予測・計画でよく使う指標を先に説明します。

- ADE (Average Displacement Error, 平均変位誤差): 予測軌道と正解軌道の、全時刻にわたる位置誤差の平均。小さいほど良い。
- FDE (Final Displacement Error, 最終変位誤差): 予測の終点と正解の終点の距離。長期予測の精度を測る。
- minADE_k / minFDE_k: k 本の候補を出してそのうち最良のものの ADE/FDE。多様な未来を予測する設定で使う。
- 計画 L2 誤差と衝突率: 自車計画の正解軌道との L2 距離(小さいほど良い)と、計画が他物体と衝突する割合(小さいほど安全)。

定量データの要点。

- MultiPath (Chai et al., 2019): nuScenes と社内大規模データで、アンカー+残差表現により minADE/minFDE を当時の回帰系ベースラインより改善したと報告。出力構造化が予測精度に効くことを示しました。指標 minADE の改善は「複数の妥当な未来のうち少なくとも1本を正確に当てる能力」が上がったことを意味します。
- CoverNet (Phan-Minh et al., 2020): nuScenes 予測タスクで、軌道集合の粒度(集合サイズ)を変えるアブレーションを行い、集合を細かくするほど表現力(下限の誤差)が上がる一方で分類が難しくなるトレードオフを定量的に示しました。これは「制御点数を増やすと表現力は上がるが学習が難しくなる」というベジェ方式のトレードオフと同型の知見です。
- UniAD (Hu et al., 2023): nuScenes で計画の平均 L2 誤差と衝突率を、知覚・予測補助なしの構成に比べ改善。アブレーションで補助モジュールを外すと計画性能が悪化することを示しました。滑らかさそのものより「計画品質」を測る指標ですが、出力表現の良し悪しが最終性能に直結することの傍証です。

ベジェ/多項式表現の「滑らかさ」自体を直接定量化する場合は、軌道のジャーク(加速度微分)の二乗積分や最大ジャーク値を指標にし、直接予測+平滑化と比べてジャークが系統的に小さくなることを示すのが典型です。報告される傾向として、制御点数を増やすと位置追従誤差(ADE)は下がるがジャークは増える、という表現力と滑らかさのトレードオフが観測されます。

## メリット・トレードオフ・限界
メリット。

- 後処理フィルタも滑らかさ損失項も不要で、出力が必ず低ジャーク(滑らか)。設計段階で保証される安心感は大きい。
- 予測対象が制御点数個ぶん(例: 10個)と少なく、学習が安定しやすく過学習しにくい。
- 出力が制御指令 (加速度, 曲率) なら、運動モデルでそのまま積分でき、物理的実現可能性が保たれコントローラに優しい。
- 自己回帰(1点ずつ順に出す)ではなく一括行列展開なので、推論が速く、誤差が時刻方向に累積しない。

トレードオフ・限界。

- 制御点が少ないと、急な回避や鋭いコーナリングのような高周波な動きを表現しにくい。num_controls が表現力と滑らかさのつまみで、増やすと表現力は上がるが学習難度と数値不安定性(高次ベジェの両端での振る舞い)が増す。
- 滑らかすぎて「本来カクッと曲がるべき緊急回避」で鈍る恐れがある。安全クリティカルな場面で過度な平滑化はリスクになりうる。
- BEV を空間平均で1ベクトルに潰す要約を使う場合、空間的にどこが障害物かといった細かい情報が落ちる。計算は軽くなるが表現力とのトレードオフ。
- 単一の高次ベジェは局所制御が苦手(1つの制御点が曲線全体に影響する)。区間ごとに分けたい場合は区分多項式(B-spline)のほうが適する。
- マルチモーダルな未来(右に避けるか左に避けるか)を表現するには、ベジェ単体では不足で、複数の制御点集合や混合分布を出す拡張が要る。

## 発展トピック・研究の最前線
- 区分多項式・B-spline 軌道: 単一ベジェの局所制御の弱さを補うため、複数の低次ベジェを滑らかに連結する B-spline を使う流れ。局所的な急変に強く、制約付き最適化(衝突回避を凸制約で表す)とも相性が良い。
- 制約つき軌道生成: Bernstein 基底の凸包性を使い、「軌道全体を安全領域内に閉じ込める」ことを制御点の凸制約として課す研究(quadrotor やロボットアームの軌道計画で発展)。学習と最適化のハイブリッド。
- マルチモーダル計画: 1本の滑らかな軌道ではなく、複数の制御点集合(複数モード)を出し、確率や評価関数で選ぶ。MultiPath/CoverNet の発想とベジェ表現の融合。
- 微分可能シミュレーションとの結合: ベジェで生成した制御指令を微分可能な車両モデルに通し、模倣損失だけでなく物理制約違反(横すべり、加速度上限超過)も学習信号に取り込む方向。
- 生成モデル系プランナーとの比較: 拡散モデルやフローマッチングで軌道を生成する手法([Flow Matching トラジェクトリプランナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0122_flow-matching-planner))は多様性と表現力に優れる一方、推論コストや滑らかさ保証で課題があり、ベジェ方式の「決定論的・滑らか保証・高速」と対極を成す。両者の使い分けが実務上の論点です。
