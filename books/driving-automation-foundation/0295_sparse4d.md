# Sparse4D

## ひとことで言うと
Sparse4Dは、車の周囲を撮る複数のカメラ映像から、車や歩行者といった3D物体を直接見つけ出す手法のひとつです。多くの手法は、まず画像を「上から見下ろした地図(BEVグリッド)」という細かいマス目に変換してから物体を探しますが、Sparse4Dはその巨大な中間地図を作りません。代わりに「ここに物体がありそう」という候補(クエリ)を少数だけ用意し、その候補のまわりの空間と時間を少しずつ画像から読み取って、候補を何段階にも分けて磨き上げます。「4D」とは、空間の3次元(縦・横・高さ)に時間の1次元を加えた4次元という意味です。スパース(疎)とは「全部のマスではなく物体がいる場所だけを見る」ことを指します。

## 直感的な理解
広い駐車場で友達を探す場面を想像してください。ひとつの戦略は「駐車場全体を1メートル四方のマス目に区切り、全マスを順番に確認する」やり方です。漏れがない代わりに、誰もいない大半のマスにも労力を使います。これが密なBEVグリッド方式の発想です。

もうひとつの戦略は「あのへんにいそうだ」という予感を頼りに、いくつかの候補地点だけを集中的に見て、見当が外れたら少しずつ場所を修正していくやり方です。確認する場所が少ないので速く、しかも目的地点を高い解像度でじっくり見られます。これがSparse4Dのスパースクエリ方式の発想です。

自動運転では6〜8台のカメラから、半径数十メートル以内の車・歩行者・自転車などを3次元で当てる必要があります。物体がいる場所は空間のごく一部なので、「物体がいそうな場所だけを賢く見る」ほうが計算もメモリも節約できます。さらに、過去のフレーム(時間軸)を一緒に見ることで、止まっている車と動いている車を区別したり、一瞬電柱に隠れた歩行者を「さっきまでここにいた」という記憶で補ったりできます。動いている世界を理解するのに、1枚のスナップショットより連続した時間の流れのほうが圧倒的に情報が多い、という直感が「4D」の根底にあります。

## 基礎: 前提となる概念
Sparse4Dを理解するために、いくつかの言葉を順に噛み砕きます。

マルチビュー3D物体検出。マルチビューとは「複数の視点(ここでは複数のカメラ)」という意味です。各カメラは別々の方向を向いていて視野が一部だけ重なります。これらの画像から、自車を中心とした共通の3D座標系(ego座標系)で物体の位置・大きさ・向き・速度を当てるのがこのタスクです。1台のカメラでは見えない死角を、複数台で補い合います。

BEV(Bird's-Eye View、鳥瞰図)。地面を真上から見下ろした2次元の地図です。自動運転では物体がほぼ地面の上に並ぶので、上から見た地図は扱いやすい表現です。BEVグリッドとは、この地図を例えば200×200のマス目に区切ったものを指します。

クエリ(query)。Transformer(系列データを「どこに注目するか」の重み付けで処理する仕組み)で使われる用語で、「私はこういう情報が欲しい」という問い合わせベクトルのことです。Sparse4Dでは1つのクエリが1つの物体候補に対応します。クエリは特徴ベクトル(物体の見た目や種類を表す数値の列)と、その候補の3D位置を表すアンカーボックスを持ちます。Sparse4Dの論文用語では物体候補をインスタンス(instance)と呼びます。

アンカーボックス(anchor box)。候補が「いま自分はこのへんにある」と思っている3Dの直方体です。中心座標(x, y, z)、大きさ(幅・奥行き・高さ)、向き(yaw、地面に対する回転角。実装ではsin/cosで表すことが多い)、速度(vx, vy)などをまとめた11次元程度の数値で表します。

アテンションとサンプリング。アテンションは「関連の高い場所に大きな重みを置いて情報を集める」仕組みです。サンプリングは「画像のある特定の座標の特徴を取り出す」操作で、座標がぴったり画素の中心に乗らないことが多いため、近い4画素の重み付き平均(双線形補間、bilinear interpolation)で値を作ります。Sparse4Dは全画素に均等にアテンションする代わりに、少数の点をサンプリングするDeformable Attention(Zhu et al., Deformable DETR, ICLR 2021)の系譜に属します。

密(dense)と疎(sparse)。密なBEV系は画像特徴をいったんBEVグリッド全体に変換します。例えば200×200=40000セルのうち物体がいるのはわずか数十セルなのに、空きセルにも計算とメモリを使います。疎なクエリ系は物体候補だけを持ち、必要な場所だけ画像を見ます。Sparse4Dは後者で、計算量がBEV解像度ではなくクエリ数に比例する点が本質的な違いです。

## 仕組みを詳しく
Sparse4Dの心臓部は「変形可能な4次元特徴集約(Deformable 4D Aggregation)」と呼ばれる処理です。これを段階的に分解します。

前提として、各インスタンスは2つを持ちます。
- インスタンス特徴: 例えば256次元のベクトル。tensor shapeで書くと [num_query, 256]、たとえば [900, 256]。物体の見た目・種類を表す入れ物です。
- アンカーボックス: 3Dの箱を表す11次元程度のベクトル。shape [900, 11]。

処理の流れは次のとおりです。

第1段階、キーポイント生成。各アンカーボックスの内側と表面に、複数のキーポイント(注目点)を置きます。箱の8つの頂点や中心といった固定の点(fixed keypoints)に加えて、インスタンス特徴から学習的に「ずらし量(offset)」を予測し、点を物体の形に合わせて動かす可変点(learnable keypoints)を加えます。たとえば1物体あたり13点を置きます。shape [900, 13, 3](最後の3は各点のx,y,z)。これがDETR3Dとの最大の違いです。DETR3Dは1クエリにつき箱の中心1点だけを投影していたため、物体の広がりや、中心が遮蔽(occlusion、手前の物体に隠されること)されたときに弱いという欠点がありました。Sparse4Dは複数点を分散配置することでこれを克服します。1点だと中心が隠れたら情報がゼロになりますが、13点なら一部が隠れても残りで補えます。

第2段階、時間方向への展開。各キーポイントを過去の複数フレームの時刻にも展開します。自車の動き(自己運動、ego-motion。GPSやIMUから既知)と、各インスタンスが持つ速度(vx, vy)を考慮し、「そのキーポイントが過去フレームではどこにあったか」を等速運動の仮定で逆算します。たとえば過去4フレーム分なら shape は [900, 13, 4, 3] となり、時間軸が増えます。これが「4D」の意味です。止まっている物体は過去フレームでも同じ場所、動いている物体は過去フレームでは別の場所に現れるため、この展開で速度の手がかりが得られます。重要なのは、過去の画像特徴そのものではなく、過去時刻に投影したサンプリング点を通じて時間情報を取り込む点です。

第3段階、投影とサンプリング。各3Dキーポイントを、各カメラの画像平面へ投影します。投影にはカメラの校正情報、すなわち内部パラメータ(intrinsics、焦点距離やレンズ中心)と外部パラメータ(extrinsics、カメラの取り付け位置・向き)を使い、3D点 [x,y,z] を同次座標を経て画像上の画素座標 [u,v] に変換します。その画素位置の特徴を、画像特徴マップから双線形補間で取り出します。物体が複数カメラの視野にまたがる場合は、見えているカメラからのみ取ります(視野外への投影は無効としてマスク)。

第4段階、階層的な集約。取り出した特徴は膨大です。キーポイント数 × 時間フレーム数 × カメラ数 × 特徴ピラミッドの段数(FPN、解像度の異なる複数の特徴マップ)を掛け合わせた数だけあります。これらを学習した重みで段階的に重み付き平均し、1つのインスタンス特徴に畳み込みます。Sparse4Dはこの集約を「点ごと→時間ごと→カメラ・スケールごと」と階層化することで、単純な平均より表現力を高めています。数値例では、6カメラ・クエリ900・キーポイント13・時間4フレームなら、1クエリあたり最大 13×4×6 = 312か所をサンプリングして1本の256次元ベクトルにまとめます。BEVの40000セルを全部計算するより、はるかに的を絞った計算です。

第5段階、反復精緻化(iterative refinement)。集約した特徴を使って、インスタンス特徴とアンカーボックスを更新します。これを6段ほどのデコーダ層で繰り返します。各層は前層の箱を残差(delta)として修正します。1段目では大雑把だった箱が、段を重ねるごとに正解へ寄っていきます。ASCII図で表すと次のイメージです。

```
[初期クエリ900個]
   │  キーポイント生成 → 4D展開 → 投影サンプリング → 4D集約
   ▼
[デコーダ層1] → 箱を粗く修正
   ▼
[デコーダ層2] → さらに修正
   ▼   ...(6段)
[デコーダ層6] → 最終的な3D箱+クラス+速度
```

学習のラベル割り当てにはDETR系と同じく二部マッチング(Hungarian matching、予測と正解を1対1に最適対応させる)を使い、各正解に1つの予測だけを当てるため、後処理のNMS(重複検出を消す処理)が不要です。これがクエリ系の「エンドツーエンド」性の核です。

後継のSparse4Dv2では、過去全フレームを毎回展開し直すと時間が伸びるほど計算が増えるという問題を、「再帰的時間融合(recurrent temporal fusion)」で解決しました。前フレームで得たインスタンス(クエリ)を、自車の動きに合わせて座標変換しつつ次フレームへ1ステップだけ持ち越して再利用します(隠れ状態のように扱う)。これにより1フレームあたりの計算量を一定に保ちながら、実質的に長い時間文脈を扱えます。これはStreamPETRなどと同じ「ストリーミング型(逐次型)」の時間モデリングの流れに属します。v2ではさらに、画像特徴を毎フレームBEVに焼き直さずキャッシュする工夫や、効率的な変形集約のCUDA実装も導入し、推論を実用速度に乗せました。

Sparse4Dv3では学習を助ける補助タスクを2つ加えました。ひとつは時間的インスタンスデノイジング(temporal instance denoising)で、DN-DETR系の発想を3D・時間方向へ拡張したものです。学習時にわざとノイズを乗せた正解箱を追加クエリとして与え、元の正しい箱へ戻させる訓練を本タスクと並走させ、二部マッチングの不安定さを緩和して収束を速め安定させます。もうひとつは品質推定(quality estimation)で、各予測に位置の信頼度を出させてスコアを補正します。加えて、検出と同時に同一物体へID(追跡番号)を割り当てる仕組みにより、検出と追跡(トラッキング)を1つのモデルで一体に行えるようにしました(再帰的に持ち越すクエリにIDを付与し続けることで、明示的な追跡器なしに軌跡がつながる)。

## 手法の系譜と主要論文
スパースクエリ系の3D検出は、2D検出のDETR(Carion et al., ECCV 2020、物体検出をクエリ集合の予測として定式化したTransformer手法)と、その効率化版Deformable DETR(Zhu et al., ICLR 2021、全画素アテンションを少数のサンプリング点に置き換え収束を速めた)を3Dへ持ち込む流れから始まりました。

DETR3D(Wang et al., CoRL 2021, arXiv:2110.06922)。各クエリが3D参照点を1つ持ち、それを各カメラへ投影して1点ぶんの画像特徴をサンプリングし、クエリを更新する手法です。BEVグリッドを作らずに済む点が新しく、スパースクエリ系の出発点になりました。弱点は前述のとおり1点しか見ないことです。

PETR(Liu et al., ECCV 2022, arXiv:2203.05625)。投影サンプリングの代わりに、画像特徴へ3D位置エンコーディング(3D position embedding)を埋め込み、クエリがグローバルなアテンションで特徴を集める設計です。投影に頼らない柔軟さがありますが、全画素アテンションのため計算は重めです。続くPETRv2(ICCV 2023)は時間融合とBEVセグメンテーションを加えました。

Sparse4D(Lin, Tao, et al., 2022, arXiv:2211.10581)。「Sparse4D: Multi-view 3D Object Detection with Sparse Spatial-Temporal Fusion」。DETR3Dの1点投影を、複数キーポイント × 複数フレーム × 複数カメラ × 複数スケールのサンプリングに拡張し、Deformable 4D Aggregationを提案しました。動機は、1点投影が物体の広がりと遮蔽に弱いことを、点を分散させて補い、同時に時間方向のサンプリングで速度を当てることでした。

Sparse4Dv2(Lin et al., 2023, arXiv:2305.14018)。「Recurrent Temporal Fusion」を中心に、毎フレーム過去を展開する非効率を再帰化で解消しました。時間が伸びても計算量が一定になり、推論速度が大きく向上、長時間文脈の活用が現実的になりました。設計思想はStreamPETRと並走します。

Sparse4Dv3(Lin et al., 2023, arXiv:2311.11722)。「Sparse4Dv3: Advancing End-to-End 3D Detection and Tracking」。temporal instance denoising、quality estimation、検出・追跡の一体化を加え、当時のnuScenesリーダーボードで最上位水準に達しました。

並行する流れとして、StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926)は同じく「過去クエリ(object-centric temporal memory)を次フレームへ持ち越すストリーミング型時間融合」を提案しており、Sparse4Dv2/v3と思想を共有します。密なBEV側ではBEVFormer(Li et al., ECCV 2022)が時空間アテンションでBEVを作る代表で、Sparse4Dはこの「BEVを明示的に作る派」への対抗として「クエリだけで完結する派」を確立しました。これらは現在のマルチビュー時間モデリングの二大潮流です。

## 論文の実験結果(定量データ)
評価には主にnuScenesデータセット(Caesar et al., CVPR 2020)が使われます。これは6カメラ・LiDAR・レーダを搭載した車で集めた大規模データで、1000シーン・約140万の3D箱を含む3D検出の標準ベンチマークです。

主要指標を噛み砕きます。
- mAP(mean Average Precision、平均適合率)。検出の正確さの総合点で、0〜1(または%)で高いほど良い。nuScenesの3D検出では、画像のIoUではなく中心位置のBEV上の距離しきい値(0.5/1/2/4m)で正解判定し、10クラスで平均します。
- NDS(nuScenes Detection Score)。nuScenes独自の総合スコアで、mAPに加えてmATE(位置誤差)・mASE(大きさ誤差)・mAOE(向き誤差)・mAVE(速度誤差)・mAAE(属性誤差)をまとめた指標。0〜1で高いほど良く、検出の総合品質を1つの数字で表します。NDS ≈ (5·mAP + Σ(1−誤差項)) / 10 のような重み付き和です。
- mAVE(mean Average Velocity Error)。速度推定の平均誤差(m/s)。低いほど良い。時間融合の効果が最も顕著に出る指標です。

報告されている代表的な数値は次のとおりです(検証セット中心。画像解像度・バックボーン・学習スケジュールで変動するためおよその水準として読んでください)。

初代Sparse4Dは、ResNet101バックボーンを用いてnuScenes検証セットでNDSおよそ0.54前後、mAPおよそ0.44前後を報告し、当時の密なBEV系の強手法であるBEVFormerに匹敵する精度を、BEVグリッドを作らずに達成しました。重要なのは、時間融合(過去フレームの利用)を入れることで、時間を使わない単フレーム版よりmAVEが大きく下がった点です。速度を当てられると、止まっている車と動いている車の区別がつき、後段の経路計画に直結します。

Sparse4Dv2は再帰的時間融合で速度を保ちながら精度を向上させ、テストセットでNDSおよそ0.59前後を報告しました。同程度精度の他手法より1フレームあたりの計算が一定で軽く、推論FPSが改善して車載の実時間要求(10Hz前後)に近づきました。「精度を落とさず速くする」ことを定量的に示した点が貢献です。

Sparse4Dv3はテストセットでNDSおよそ0.71〜0.72前後(VoVNet/EVA系など強いバックボーンと長い時間文脈を用いた構成)を報告し、当時のカメラのみリーダーボード最上位に位置しました。さらに追跡指標AMOTA(Average Multi-Object Tracking Accuracy、複数物体追跡の正確さ。IDの取り違え=ID switchや見失いを総合評価する指標で高いほど良い)でもおよそ0.57前後と上位を示し、検出と追跡の一体化が機能することを実証しました。明示的な追跡アルゴリズムを後段に置かずにこの水準を出せたことが注目されました。

アブレーション(どの要素を抜くと性能がどう変わるかの分析)から読み取れる傾向は次のとおりです。キーポイントを1点だけ(DETR3D相当)に減らすと精度が落ち、複数キーポイント化と学習的offsetの効果が確認されます。時間融合を切るとmAVEが顕著に悪化します(報告では速度誤差がおよそ半分程度まで改善する例がある)。v3でtemporal instance denoisingを外すと収束が不安定になり最終精度も下がります。quality estimationを外すとmAPが下がります。これらは各構成要素が独立に効いていることを示します。

## メリット・トレードオフ・限界
メリット。巨大なBEV中間表現を作らないため計算量とメモリが小さく、計算がBEV解像度ではなくクエリ数に比例します。高解像度の画像特徴を量子化を経ずに直接サンプリングできるため、遠方の細部を保てます。時間融合により速度推定や遮蔽補完が得意で、検出と追跡を1つのモデルに統合できます(v3)。再帰的時間融合(v2以降)で長時間文脈をほぼ一定コストで扱えます。NMS不要のエンドツーエンド設計です。

トレードオフと限界。投影サンプリングが正確なカメラ校正(内部・外部パラメータ)に依存するため、校正がずれると取り出す特徴の位置が狂い、精度が落ちます。アテンションで柔らかく対応付けるカメラ-LiDAR融合手法(soft association型、[TransFusion](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0345_transfusion))ほどには校正ずれに頑健ではありません。キーポイント・時間・カメラ・スケールの多重サンプリングは実装が複雑で、効率的なCUDAカーネルを要します(これがなければ理論上の効率が出ません)。クエリ数が検出できる物体数の上限を決めるため、極端に混雑した場面ではクエリ不足になりえます。密なBEVのような「人が読める中間地図」が出力されないため、車線や走行可能領域といったBEVセグメンテーション系の下流タスクと組み合わせるには別途設計が要ります。

未解決の研究課題としては、再帰的時間融合での誤差の時間伝播(一度間違えると次フレームへ誤りが持ち越され、長時間で蓄積しうる)、クエリ初期化の最適化、校正ずれや時刻同期ずれへの頑健化、そして長距離・遠方小物体での精度がLiDAR系に劣りがちな点が挙げられます。カメラのみの宿命として、絶対的な距離(深度)の曖昧さが残ります。

## 発展トピック・研究の最前線
スパースクエリ系は現在、エンドツーエンド自動運転スタックの知覚モジュールとして取り込まれる方向にあります。検出・追跡・地図・予測・計画を1つのクエリ集合で貫くUniAD(Hu et al., CVPR 2023 best paper)やVAD/SparseDrive系の流れでは、Sparse4D流のスパースな時間融合が知覚の効率的な土台として相性が良いと考えられています。実際、SparseDriveのような手法はSparse4Dのスパース表現を計画まで延長しています。

また、スパースクエリ系とオンラインHDマップ生成(走行中に車線・縁石を地図として推定)を統合する動き、レーダや超音波など低コストセンサとのスパース融合、そして「BEVを作る派」と「クエリだけで完結する派」のハイブリッド設計が研究の前線です。効率(計算・メモリ)とロバスト性(校正ずれ・センサ欠損)の両立、そして長距離知覚(高速道路で必要な100m超の検出)が、今後の主要なテーマになっています。
