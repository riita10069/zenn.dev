# backbone出力チャネル数の動的取得

## ひとことで言うと

画像から特徴を取り出す「バックボーン」が出力する特徴マップのチャネル数を、コードに数字で直接書く(ハードコード)のをやめて、ライブラリのメタ情報や一度だけの試し実行から自動で読み取る、という設計の話です。これにより、別のバックボーンに差し替えても下流のネットワークコードを書き換えずに済むようになります。一見すると地味なエンジニアリングですが、複数のバックボーンを公平に比較したり、新しいバックボーンを次々と試したりする研究スタイルでは、ここがハードコードだと実験そのものが回らなくなる、という意味で重要な土台です。

## 直感的な理解

研究の現場では「ResNet-50 と Swin と ConvNeXt のどれが一番良いか比べたい」という場面が頻繁に起こります。このとき問題になるのが、バックボーンごとに出力する特徴の「太さ」(チャネル数)が違うことです。下流のモジュール(特徴を融合したり、軌道を予測したりする部分)は、この太さに合わせて入力サイズを決めなければなりません。

もし下流のコードに「入力は 1440 チャネル」と数字で書き込んでしまうと、ResNet-50(複数段を連結すると 3840 チャネル)に差し替えた瞬間に形状不一致のエラーになります。そのたびに人間がソースの数字を書き換えるのは、ミスの温床であり、実験のスピードを大きく落とします。さらに悪いのは、数字が運悪く一致してしまった場合です。たとえば「期待チャネルがたまたま整合してしまったが、軸の意味が違う」状況だと、エラーすら出ずに静かに誤った特徴を下流へ流し続け、学習が収束しない原因が何週間も特定できない、ということが起こり得ます。

直感的には「下流が必要とするのは『太さの数値』そのものではなく、『その時点のバックボーンが出す太さに自動的に合わせる』能力だ」という発想です。これを実現するのが、出力チャネル数の動的取得です。バックボーンに「あなたは何チャネル出すのか」を尋ね、その答えに合わせて下流を構築します。

## 基礎: 前提となる概念

いくつかの用語を、知らない前提で噛み砕きます。

- バックボーン(backbone): 入力画像を受け取り、「この画像のどこに何がありそうか」を表す数値のかたまり(特徴マップ)を作る、ニューラルネットワークの前半部分です。多くの自動運転・検出モデルは、ImageNet(約 128 万枚・1000 クラスの画像分類データセット)などで事前学習したバックボーンを使い回します。代表例は ResNet 系、Swin Transformer 系、ConvNeXt 系です。

- 特徴マップ(feature map): バックボーンの出力で、`[バッチ, チャネル, 高さ, 幅]` という4次元の数値配列(テンソル)です。これは「画像をぼかして抽象化した小さな画像が、チャネル数だけ重なっている」イメージです。

- チャネル数(channels): 特徴マップのうち「特徴の種類の数」を表す軸です。96 チャネルなら、各位置について 96 種類の特徴量(エッジの向き、色のパターン、テクスチャの種類など、学習で勝手に決まる抽象的な特徴)が並んでいます。

- 多段特徴 / マルチスケール(multi-stage / multi-scale): 最近のバックボーンは、画像を段階的に縮小しながら複数の解像度の特徴マップを出します(stage 0〜3 など)。浅い段は解像度が高くチャネルが薄い(細かい情報、小さい物体に強い)、深い段は解像度が低くチャネルが厚い(意味的な情報、大きい物体に強い)という性質があります。これらをまとめて使うのが現代の標準です。

ここで重要なのが、段ごとのチャネル数がバックボーンごとに違うという事実です。具体例を挙げます。

- Swin Transformer V2 Tiny: 各段 96, 192, 384, 768 チャネル。4段の合計は 96+192+384+768 = 1440。
- ConvNeXt V2 Tiny: 各段 96, 192, 384, 768(Swin Tiny と同じ設計思想なので合計も 1440)。
- ResNet-50(中間特徴を取り出すモード): 各段 256, 512, 1024, 2048。合計 3840。

なぜ Swin と ConvNeXt は同じなのに ResNet だけ太いのか。これは設計時の「初段チャネルを何にして、段が進むごとに何倍にするか」というポリシーの違いです。Swin / ConvNeXt Tiny は初段 96、各段で 2 倍。ResNet は初段(中間特徴の取り出し点)が 256 で、これも 2 倍ずつですが起点が太いので最終段が 2048 と大きくなります。同じ「4段・2倍」でも起点が違えば最終的な太さが桁で変わる、というのがチャネル設計の肝です。

以前のよくある書き方は、この合計値を `if backbone == "swin": channels = 1440 elif backbone == "resnet50": channels = 3840` のように文字列分岐でハードコードするものでした。これには次の弱点があります。

1. 新しいバックボーンを足すたびに、この if/elif を手で書き足さないと動かない。
2. ライブラリ側がアーキテクチャ定義を少し変えてチャネル数が変わると、ハードコードした数字が静かに食い違い、原因の分かりにくいバグになる。
3. ライブラリのメタ情報という「正解」がすでに存在するのに、それを使わず人間が転記しているので、転記ミスのリスクが常にある。

## 仕組みを詳しく

動的取得は、典型的には2段構えで実装されます。「速くて確実な経路」を優先し、それが使えないときだけ「重いが汎用的な経路」へ落ちる、という設計です。

ステップ1: ライブラリのメタ情報から読む(優先経路)

`timm`(PyTorch Image Models)のような主要ライブラリでは、`features_only=True` を指定して作ったモデルが `feature_info` という属性を持ち、各段の情報を取り出せます。各段の `num_chs`(number of channels)がそのままチャネル数、`reduction` が入力に対する縮小率(例 stage 0 で 4 なら、出力解像度は入力の 1/4)です。

```python
info = getattr(backbone, "feature_info", None)
if info is not None:
    channels = [stage["num_chs"] for stage in info]   # 例 [96, 192, 384, 768]
    strides  = [stage["reduction"] for stage in info]  # 例 [4, 8, 16, 32]
```

この経路は実際にモデルを動かさずに済むので速く、メモリも食わず、確実です。主要なバックボーンはほぼこの API に対応しているので、通常はここで完結します。`reduction` まで取れると、後段で「この特徴マップは入力の何分の1の解像度か」が分かるので、複数段を同じ空間サイズへ揃える処理(プーリング窓の決定)や、特徴位置を画像座標へ戻す逆変換にも使えます。

ステップ2: ダミー forward でプローブする(フォールバック経路)

自作バックボーンや、メタ情報を公開していないモデルには `feature_info` がないことがあります。その場合は「実際に1回だけ通してみて、出てきた特徴マップの形状を見る」ことでチャネル数を知ります。プローブ(probe)とは「探りを入れる」という意味で、ここでは試し撃ちの forward を指します。

```python
dummy = torch.zeros(1, 3, 256, 256)        # 1枚・3ch・256x256 の仮入力
was_training = backbone.training
backbone.eval()                             # BatchNorm の移動平均を汚さない
with torch.no_grad():                       # 勾配計算なし(学習に影響させない)
    out = backbone(dummy)                   # 複数の特徴マップのリストが返る想定
if was_training:
    backbone.train()                        # 元の状態に戻す
return [f.shape[1] for f in out]            # 各テンソルの dim=1 がチャネル
```

このフォールバックには、堅牢化のために次の防御を入れるのが定石です。

- 一時的に `eval()` モードにして実行し、終わったら元の `train/eval` 状態に戻す。これは BatchNorm(バッチ正規化、ミニバッチ内の平均と分散を使う層)が、ダミー入力の統計でうっかり内部の移動平均(running mean / running var)を汚さないようにするためです。汚すと、本番データの最初の数ステップで正規化が狂い、学習が不安定になります。
- 返り値がリスト/タプルでない、または空なら明示的に `ValueError` で失敗させる。
- 各特徴マップが4次元 `[B, C, H, W]` でなければ `ValueError`。曖昧なまま進めて後で意味不明なエラーにするより、その場で原因の分かるエラーを出すのが、デバッグ時間を最小化する原則です(fail fast の思想)。

before/after を一言で言うと、before は「`channels = 1440` と人が書く」、after は「`channels = sum(infer_feature_channels(backbone))` でコードが自分で数える」です。求めた各段チャネル数のリスト(例 `[96, 192, 384, 768]`)は、後段で「チャネルの軸がどこにあるか」を判定する物差し(後述の形状ベース permute 判定)にも再利用されます。つまり動的取得は単独で完結するのではなく、形状検証・軸正規化・射影層の構築といった一連の処理の根に位置します。

## 手法の系譜と主要論文

「チャネル数を動的に取る」こと自体は単独の研究テーマではなく、複数の研究潮流が合流した結果として自然に要求された実装上の解決策です。系譜を描くと次のようになります。

- マルチスケール特徴の確立: Lin らの Feature Pyramid Networks(FPN, CVPR 2017, arXiv:1612.03144)が、バックボーンの複数段の特徴を組み合わせると小さい物体も大きい物体も検出しやすくなることを示しました。FPN では各段のチャネル数が違うのが前提なので、それを共通次元(典型的には 256)へ揃える横方向結合(lateral connection、各段に 1x1 畳み込みを挟んで 256 に揃える)が必要になります。この「段ごとにチャネルが違う」という事実が、動的取得の必要性の出発点です。

- バックボーンの多様化: He らの ResNet(CVPR 2016, arXiv:1512.03385)を起点に、Liu らの Swin Transformer(ICCV 2021, arXiv:2103.14030)と Swin V2(CVPR 2022, arXiv:2111.09883)、Liu らの ConvNeXt(CVPR 2022, arXiv:2201.03545)と Woo らの ConvNeXt V2(CVPR 2023, arXiv:2301.00808)など、設計の異なるバックボーンが次々と提案されました。これらは段数こそ似ていてもチャネル設計や軸の並び(channels-first か channels-last か)が異なります。研究者がこれらを横断比較したいという需要が、ハードコードを排除する動機を強めました。

- 統一インターフェースの登場: Wightman の timm(PyTorch Image Models, 2019〜)が、数百種のバックボーンを統一 API で「中間特徴を出すモード」(`features_only`)にできる仕組みを提供し、`num_chs` や `reduction` をメタ情報として公開しました。動的取得の優先経路はこの API に乗ることで成立します。timm の存在によって「コードを変えずにバックボーンを差し替える」ことが現実的になり、研究コミュニティの実験回転速度を大きく押し上げました。

- 遅延構築という抽象化: さらに根底には、PyTorch の `LazyConv2d` / `LazyLinear`(初回 forward で入力チャネルを観測してから重みを確定する遅延層)に代表される「入力形状を見てから層を作る」という設計思想があります。動的取得はこれを「初期化フェーズで先に1回観測しておく」形に展開したものと位置づけられます。

## 論文の実験結果(定量データ)

動的取得そのものに固有のベンチマークはありませんが、その効果を測る土台となるバックボーン比較の数値を、指標の意味とともに示します。

FPN(CVPR 2017)は、ResNet をバックボーンとした検出器の COCO 物体検出で、単一スケール特徴のベースラインに対して AP(Average Precision、検出の正確さを 0〜100 で測る指標で、高いほど良い)を大きく改善しました。報告では、FPN を入れた Faster R-CNN(ResNet-101)が COCO test-dev で box AP 約 36.2 を達成し、とくに小さい物体の AP(APs、面積の小さい物体だけで測った AP)で顕著な向上を示しました。ここで重要なのは「複数段を束ねること自体が効く」という事実で、束ねるには段ごとに異なるチャネル数を扱う仕組みが不可欠だという点です。

バックボーンの差による性能差も、動的取得が支える比較で見えてきます。ImageNet-1K 分類のトップ1精度(画像を正しいクラスに分類できた割合、高いほど良い)で、ResNet-50 はおよそ 76〜80%(学習レシピ次第。近年の強化レシピでは 80% 超も報告)、Swin-T はおよそ 81.3%、ConvNeXt-T はおよそ 82.1%、ConvNeXt V2-T(自己教師あり事前学習 FCMAE 込み)はおよそ 83% 前後、と各原論文で報告されています。これらは出力チャネル数が違う(例: ResNet-50 の最終段 2048 に対し Swin-T は 768)モデル同士の比較であり、下流を動的に組み立てられて初めて同一の下流条件で公平に比べられます。

アブレーション的に言えば、「動的取得を外してハードコードに戻す」と、機能的な精度は変わらないものの、バックボーンを1つ差し替えるたびに数値の書き換えと再検証が必要になり、比較実験のコストが線形に増えます。研究の生産性という軸での「アブレーション」として、ここが効いていることになります。逆に言えば、動的取得は精度を上げる仕掛けではなく「正しく比較できる土台を保証する」仕掛けであり、その価値は得られた数値そのものではなく「その数値が信用できる」ことにあります。

## メリット・トレードオフ・限界

メリット

- 新バックボーン追加時に下流コードを書き換えなくてよい。バックボーンの登録(レジストリへの追加)だけで選択肢に増やせる。
- ライブラリのアーキテクチャ変更によるチャネル数の食い違いを自動で吸収する。
- 人手の転記ミスが原理的に起きない。
- フォールバックがあるので、メタ情報を持たない自作バックボーンも動く。

トレードオフ・限界

- フォールバックは「1回ダミー forward する」ぶん、初期化がわずかに遅く、メモリも一瞬使う。大きなバックボーンでは数百MB単位の一時確保が起きうる。CPU 上で初期化すればこの一時確保は GPU メモリを圧迫しないので、初期化を CPU で行う実装も多い。
- フォールバックは出力が `[B, C, H, W]`(channels-first)であることを暗黙に仮定しがちで、channels-last で返す自作バックボーンだと誤判定しうる(形状ベース判定と組み合わせて回避する)。
- メタ情報を持つが規約が微妙に違う実装だと `num_chs` 取得に失敗しうる。その場合は例外を捕まえてフォールバックへ落ちる設計が望ましい。
- 未解決の課題として、3次元トークン列 `[B, N, C]` を返す純 Transformer 系や、可変段数を返すバックボーンへの一般化は、単純な形状検査だけでは取りこぼしうる。ここは「段数・軸・空間構造をどう抽象化するか」という設計の余地が残る。

## 発展トピック・研究の最前線

- バックボーン非依存の設計は、近年の「基盤モデルを凍結して特徴だけ使う」流れ(DINOv2 などの自己教師あり事前学習特徴)とも相性が良い。凍結バックボーンを差し替えてもアダプタ層だけ再学習すれば済む構成では、チャネル数の動的取得がほぼ必須になる。
- マルチスケール特徴を「いつ・どう融合するか」は今も活発な研究対象で、BiFPN(Tan et al., EfficientDet, CVPR 2020, arXiv:1911.09070)のように段間の結合に学習可能な重みを入れる手法もある。これらはいずれも段ごとのチャネル情報を前提にする。
- メタ情報経由とプローブ経由のどちらでも、最終的には「下流を入力形状に対して遅延構築する(lazy initialization)」設計に行き着く。PyTorch の `LazyConv2d` のような遅延層はこの思想の標準化版と言える。型レベルで形状を注釈する(named tensor / jaxtyping)流れと組み合わせると、チャネル取り違えを静的に検出できる方向へ進む。
