# カメラ射影行列 P = K_scaled @ T_ref_to_cam

## ひとことで言うと
「世界のどこにある3次元の点が、あるカメラ画像のどのピクセルに写るか」を1回の行列の掛け算で計算できるようにした 3x4 の数表が、カメラ射影行列(camera projection matrix)P です。カメラの「レンズ特性(内部パラメータ)」と「取り付け位置・向き(外部パラメータ)」を1つにまとめたもので、複数カメラを上から見た地図(鳥瞰図、BEV)に統合する処理の土台になります。前処理のリサイズ・クロップに合わせて補正した内部行列 K_scaled と、基準座標系からカメラ座標系への変換 T_ref_to_cam を掛けて作るので、P = K_scaled @ T_ref_to_cam という形になります。

## 直感的な理解
スマホで写真を撮るとき、目の前の3次元の世界が2次元の写真に潰されます。このとき「あの木のてっぺん(3次元のある点)が、写真のどこ(ピクセル)に写るか」は、カメラの位置・向きとレンズの画角で一意に決まっています。射影行列 P は、まさにこの「3次元の点 → 写真上の点」という写しの規則を1つの行列として書き下したものです。

なぜこれが要るのか。自動運転車には複数のカメラが付いています(前方・側方・後方など、6〜7台が典型的)。それぞれ向きも位置も違います。これらの映像を「車を上空から見下ろした1枚の地図(BEV、Bird's-Eye View、鳥瞰図)」に統合したい。そのためには「前方10mの地面という3次元の点は、前方カメラのどのピクセルと、左前カメラのどのピクセルに、それぞれ写っているのか」を正確に対応づける必要があります。P があれば、この対応づけが各カメラについて行列の掛け算1回で求まります。逆に P がなければ、どのカメラのどのピクセルがどの3次元位置を見ているのか分からず、地図に正しく貼り込めません。複数カメラの視野は隣同士で少し重なっているので、同じ3次元点が2台に写ることもあり、P を使えばその重なりも整合的に扱えます。

## 基礎: 前提となる概念

座標系(coordinate frame)。3次元空間で点の位置を表すには「どこを原点に、どの向きを軸に取るか」という基準が要ります。これが座標系です。自動運転では複数の座標系が登場します。(1) 世界座標系または基準座標系(reference frame): 車の中心や自車原点(後輪車軸中心など)を基準にした共通の座標系。(2) カメラ座標系(camera frame): 各カメラのレンズを原点に、光軸(レンズが向いている方向)を Z 軸に取った座標系。同じ物理的な点でも、どの座標系で見るかで数値が変わります。座標系の変換を間違えると、点が全く別の場所に飛ぶので、まずどの座標系で何を語っているかを意識することが基礎の基礎です。

同次座標(homogeneous coordinates)。3次元の点 (X, Y, Z) を、わざと1次元増やして (X, Y, Z, 1) と4次元で書く表記法です。なぜこんな面倒なことをするか。回転・並進(平行移動)・投影という別々の操作を、すべて「行列の掛け算」という同じ形にまとめられるからです。並進は本来「足し算」ですが、同次座標を使うと掛け算に変換でき、複数の変換を1つの行列に合成できます。これがコンピュータビジョンで同次座標が標準になっている理由です。最後の成分(スケール成分)を 1 に正規化すると元の3次元座標に戻ります。

内部パラメータ行列 K(intrinsic matrix)。カメラ自身のレンズ特性を表す 3x3 の行列です。「カメラ座標系の点」が「画像のどのピクセル」に写るかを決めます。中身は焦点距離(画素単位の f_x, f_y)と主点(画像中心のピクセル座標 c_x, c_y)です。

```
      [ f_x   0    c_x ]
  K = [ 0    f_y   c_y ]
      [ 0     0     1  ]
```

f_x が大きいほど望遠(画角が狭く、遠くが大きく写る)、小さいほど広角です。f_x と f_y は通常ほぼ等しいですが、ピクセルが正方形でないセンサーではわずかに違います。注意: 画像をリサイズしたりクロップ(切り出し)したりすると、ピクセルの座標系が変わるので K も補正が必要です。たとえば画像を半分に縮めたら f_x, f_y, c_x, c_y もすべて半分にしないと、もう正しい対応になりません(→[前処理（Resize/Crop）に合わせたカメラ内部行列のスケーリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0801_camera-intrinsic-scaling))。補正後を K_scaled と呼びます。

外部パラメータ(extrinsic parameters)。カメラが基準座標系のどこに、どの向きで付いているかを表す情報です。回転行列 R(3x3、向き)と並進ベクトル t(3次元、位置)からなり、これを 4x4 の同次変換行列にまとめます。「基準座標系の点」を「そのカメラから見た座標」に移す変換を T_ref_to_cam と書きます。逆に「カメラ座標 → 基準座標」へ移す変換は T_cam_to_ref で、両者は互いに逆行列の関係 T_ref_to_cam = (T_cam_to_ref)^(-1) です。校正データがどちらの向きで保存されているかは規格(データセットやセンサーメーカ)によって異なり、ここを取り違えると全カメラの射影が破綻します。

## 仕組みを詳しく

射影行列は内部行列と外部変換を掛け合わせて作ります。標準的な形は次の通りです。

```
P = K_scaled @ T_ref_to_cam[:3, :]    # (3x3) @ (3x4) = (3x4)
```

`T_ref_to_cam[:3, :]` は 4x4 の同次変換行列の上3行(回転 R と並進 t を横に並べた `[R | t]` に相当)で、最終行は常に [0, 0, 0, 1] なので落として 3x4 にします。古典的教科書では P = K [R | t] と書きますが、`[R | t]` が T_ref_to_cam の上3行という意味では同じものです。

実装上の典型的な手順は次のようになります。

1. カメラの校正データを読む。内部行列 K(3x3)、外部変換(多くは T_cam_to_ref の形、4x4)、元画像サイズ(幅・高さ)を取得します。

2. K を前処理に合わせて補正する。学習に入れる画像は backbone の期待解像度にリサイズ・クロップされるので、その幾何変形と同じだけ K も補正して K_scaled を得ます(→[前処理（Resize/Crop）に合わせたカメラ内部行列のスケーリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0801_camera-intrinsic-scaling))。これを怠ると、加工後画像に対して K がずれ、射影が静かに狂います。前処理を backbone のメタデータから引く設計(→[timmによるbackbone駆動の前処理解決](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0841_timm-transform-resolution))とセットにすると、解像度と K の整合を1か所で保てます。

3. 外部変換の向きを揃える。校正データが T_cam_to_ref(カメラ→基準)で保存されている場合、射影に必要なのは逆向きの T_ref_to_cam(基準→カメラ)なので逆行列を取ります。

```python
T_ref_to_cam = np.linalg.inv(T_cam_to_ref)   # (4, 4)
```

4. 射影行列を組む。

```python
P = K_scaled @ T_ref_to_cam[:3, :]    # (3,3) @ (3,4) = (3,4)
```

全カメラ分を積めば `(V, 3, 4)`(V はカメラ台数)のテンソルになります。たとえば6カメラなら (6, 3, 4) です。

P の使い方(意味)。基準座標系の3次元点を同次座標 [X, Y, Z, 1] で表すと、

```
[u', v', w'] = P @ [X, Y, Z, 1]        # 3次元のベクトルが出る
ピクセル座標 (u, v) = (u'/w', v'/w')    # 最後の成分 w' で割る(同次座標を普通の座標に戻す)
```

最後に w' で割るのが投影の本質です。w' はカメラから見たその点の奥行き(深さ)に対応し、遠いものほど小さく写る(透視投影、perspective projection)効果を生みます。なお w' が負またはゼロのとき、その点はカメラの後ろ(または面上)にあり、画像には写りません。BEV 融合の実装ではこの「後ろにある点」を捨てるマスク処理が必要です。

数値例。前方カメラの P があり、基準座標で前方10m・地面高さの点 (X=0, Y=0, Z=10)(Z を前方とする座標系)を入れたとします。P を掛けて w' で割ると、たとえば加工後 256x256 画像の (u≒128, v≒170) のように画像内の具体的なピクセル位置が出ます。u≒128 は横方向にちょうど中央、v≒170 は縦方向にやや下(地面なので画像の下寄り)という妥当な値です。前方20mに離すと、地面の点はさらに画像の上(消失点に近づく)に来て v が小さくなり、奥行きが画像の縦位置に表れることが分かります。

```
基準座標系の点 (X,Y,Z)
        │  T_ref_to_cam(取り付け位置・向きで回す・移す)
        ▼
   カメラ座標系の点
        │  K_scaled(レンズ特性で2次元に投影)
        ▼
   加工後画像のピクセル (u,v)
   ──── この一連を1つの行列にまとめたものが P = K_scaled @ T_ref_to_cam ────
```

なぜ事前計算してキャッシュするか。カメラの取り付け位置・向きは走行中変わりません(リジッドに固定されている)。したがって P はフレームに依存せず、シーン(走行区間)ごとに1回計算すれば全フレームで使い回せます。学習でデータを1件取り出すたびに P を計算し直すのは無駄なので、構築時に計算してためておき、データ取得時はキャッシュを返すだけにします(→[__getitem__をI/Oのみにする派生配列キャッシュ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0815_getitem-io-only-caching))。

BEV 融合で P がどう使われるか。大きく2系統あります。(A) 画像から3次元へ「押し出す」系(LSS 系)では、各ピクセルに深度を予測させ、K と外部行列から逆算してそのピクセル特徴を3次元の錐台(frustum)に配り、BEV 平面に集約します。(B) BEV グリッドから画像へ「引きに行く」系(BEVFormer 系)では、BEV グリッド上の各点(の高さ違いの複数サンプル)を P で各カメラ画像へ射影し、その位置の特徴をアテンションで集めます。どちらも P=K_scaled@T_ref_to_cam という同じ幾何が中核にあります。

## 手法の系譜と主要論文

射影行列そのものは古典的な幾何ですが、「複数カメラの P をニューラルネットに渡して BEV 表現を作る」という使い方が、ここ数年の知覚研究の中核になっています。

Hartley & Zisserman, "Multiple View Geometry in Computer Vision" (2nd ed., 2004, Cambridge University Press)。射影行列の定義 P = K [R | t]、同次座標、ピンホールカメラモデル、エピポーラ幾何などを体系化した古典的教科書です。なぜ P = K [R | t] という形になるかの導出はここが原典で、現代の BEV 手法もこの定義の上に立っています。

Philion & Fidler, "Lift, Splat, Shoot: Encoding Images from Arbitrary Camera Rigs by Implicitly Unprojecting to 3D" (ECCV 2020, NVIDIA, arXiv:2008.05711)。各カメラの内部・外部行列を使って、画像の各ピクセル特徴を奥行き分布つきで3次元空間に「持ち上げ(lift)」、BEV 平面に「ばらまく(splat)」手法です。動機: 複数カメラを統一座標で扱いたいが、深度が未知。新規性: 各ピクセルに離散的な深度分布(カテゴリ分布)を学習で予測させ、特徴を錐台に展開してから BEV にプール(集約)する。K と外部行列が無いと、どのピクセルが3次元のどこに対応するか決まらないため、この手法の前提として P 系の校正情報が必須です。

Huang et al., "BEVDet: High-Performance Multi-Camera 3D Object Detection in Bird-Eye-View" (2021, arXiv:2112.11790)。LSS の lift-splat を物体検出に最適化し、データ拡張(画像側とBEV側で別々に拡張する)と効率的な BEV エンコーダで実用的な検出器に仕上げました。画像側の拡張(リサイズ・クロップ・回転)を行うと、その幾何変形を K に反映しないと射影が崩れる点を明確に扱っています。

Li et al., "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers" (ECCV 2022, arXiv:2203.17270)。LSS が「画像から3次元へ押し出す」のに対し、BEVFormer は逆に「BEV グリッドの各点から各カメラ画像へ射影して、その位置の特徴をアテンションで引きに行く」アプローチ(空間クロスアテンション)です。動機: グリッド側から引きに行くほうが複数カメラの重なりや時間方向の統合を扱いやすい。ここでも「基準座標 → 画像ピクセル」の射影、つまり本トピックの P と同じ変換を使います。

Li et al., "BEVDepth: Acquisition of Reliable Depth for Multi-view 3D Object Detection" (AAAI 2023, arXiv:2206.10092)。LSS 系の弱点である深度推定の質を、LiDAR の点群を地面真値として明示的に監督することで底上げしました。深度の質が BEV 検出の質を律速することを示し、ここでも LiDAR 点群を画像に投影するために P と同じ幾何を用います。

これらに共通するのは「正しい P を与えることが BEV 認識の前提条件」という点で、本トピックはその下ごしらえにあたります。

## 論文の実験結果(定量データ)

LSS と BEVFormer は主に nuScenes(Caesar et al., CVPR 2020、6カメラ・LiDAR・レーダーを備えた大規模自動運転データセット)で評価されます。指標の意味を先に説明します。mAP(mean Average Precision、平均適合率)は「検出した物体が正しく当たっている割合」を複数のしきい値で平均した値で、高いほど良い。NDS(nuScenes Detection Score)は mAP に加えて、位置(mATE)・大きさ(mASE)・向き(mAOE)・速度(mAVE)・属性(mAAE)の誤差をまとめた nuScenes 独自の総合スコアで、こちらも高いほど良い(0〜1)。

報告される代表値として、BEVDet 系はカメラのみで nuScenes 検出 NDS がおよそ 0.4 前後から、解像度や時間融合を足した改良版で 0.48〜0.52 程度まで達します。BEVFormer は nuScenes 検出で NDS 約 0.517(BEVFormer-base、報告による)を達成し、当時のカメラのみ手法で最高水準を示しました。BEVDepth は深度の明示監督により、同等規模の構成で NDS をおよそ数ポイント押し上げたと報告されています(構成依存)。これらの数字は「P による正しい射影」が成立していることを暗黙の前提にしています。

重要なアブレーション(ある要素を抜くと性能がどう変わるかの検証)として、BEVFormer は時間方向のアテンション(過去フレームの BEV 特徴を引き継ぐ機構)を抜くと、特に速度推定の誤差(mAVE)が悪化することを示しています。これは、複数フレームにわたる同一の幾何対応(P による射影)を時間方向に積み上げることで動きの推定が改善することを意味します。BEVDepth は深度監督を外すと NDS が明確に落ちることを示し、深度=射影の奥行き方向の精度が性能の鍵だと裏づけています。

校正精度の影響に関する実験的知見として、外部パラメータ(取り付け位置・向き)にノイズを加えると BEV 検出性能が顕著に劣化することが複数の研究で報告されています。たとえば回転に数度のずれを与えるだけで mAP が数ポイント落ちる、という報告があり、P の正確さが性能に直結することを定量的に裏づけています。並進(位置)より回転(向き)のずれのほうが影響が大きい傾向も指摘され、これは遠方の点ほど角度誤差が距離誤差に増幅されるという幾何的な理由で説明できます。

## メリット・トレードオフ・限界

メリット。3次元点から各カメラのピクセルへの写像を1つの行列の掛け算で計算でき、BEV 融合が実装しやすい。前処理に合わせた K_scaled を使えば加工後画像に対して幾何的に正しい。フレームに依存しないので1回計算してキャッシュでき、データ供給が速い。古典的で枯れた数学なので、デバッグ時に「投影した点を画像に重ねて目視確認」ができる(可視化で正しさを検証しやすい)。

トレードオフ・限界。第一に、校正(K と外部パラメータ)が正確であることが大前提です。校正がずれると P もずれ、BEV が静かに歪みます(例外は出ないので気づきにくい)。第二に、外部変換の向き(cam_to_ref か ref_to_cam か)を取り違えると全カメラが破綻します。第三に、標準的なピンホールモデルの P はレンズ歪み(広角・魚眼の樽型歪み)を表現できません。歪みの強いカメラでは、射影の前後で歪み補正(undistortion)を別途かけるか、歪みパラメータを含む拡張モデルが必要です。第四に、時間方向に車体やカメラがわずかに揺れる(サスペンションの沈み込み、加減速のピッチ変化等)場合、静的な校正だけでは誤差が残ります。

研究上の未解決課題として、校正の自己補正(走行データから外部パラメータのずれをオンラインで推定する)、歪みの大きい魚眼カメラを含むリグの統一的な扱い、校正ノイズに頑健な BEV 表現の学習などが活発に研究されています。校正フリーで暗黙にカメラ間関係を学習する手法も提案されていますが、明示的な P を使う手法の精度・解釈性にはまだ及ばないことが多いのが現状です。

## 発展トピック・研究の最前線

深度を明示的に与えず暗黙的に学習する LSS 系に対し、LiDAR から得た正確な深度で監督する BEVDepth(Li et al., AAAI 2023)は、深度推定の質が BEV 検出の質を律速することを示しました。また、射影をベースにした手法をさらに進めて、複数フレーム・複数モダリティ(カメラ+LiDAR+レーダー)を統一 BEV 上で融合する研究(BEVFusion, Liu et al., ICRA 2023)が主流になっています。スパースなクエリで物体を直接当てる DETR 系(DETR3D, Wang et al., CoRL 2021; PETR, Liu et al., ECCV 2022)では、3次元の参照点を P で画像へ射影して特徴をサンプリングする形で、やはり同じ幾何を使います。射影行列はこれら全ての土台として一貫して使われ続けています。
