# 埋め込み次元の256統一

## ひとことで言うと
画像から抽出した特徴を、成り行きで決まった大きく中途半端な太さ(たとえば1440次元)のまま下流へ流す代わりに、すべて固定の256次元にそろえる設計です。この分野の主要モデルはほぼ例外なく256次元を採用しており、それに合わせると計算量とパラメータが大きく減り、特徴抽出ネットワーク(backbone)を別物に差し替えても下流をいじらずに済みます。「全モジュールが同じ太さの配線でつながる」状態を作るのが目的です。

## 直感的な理解
複数の部品を組み合わせて1つのシステムを作るとき、部品どうしをつなぐ「端子の規格」がそろっていると、どの部品を入れ替えても配線をやり直さずに済みます。USB-Cのように1つの規格に統一しておけば、充電器もモニタもストレージも同じ口で挿せます。逆に部品ごとに端子の形がバラバラだと、1つ交換するたびに全部の変換アダプタを作り直す羽目になります。

ニューラルネットワークでも同じです。各モジュールが扱う特徴ベクトルの太さ(次元)を全モジュールで同じにそろえておくと、モジュールを自由に差し替え・接続できます。ところがbackboneの出力をそのまま使うと、backboneの種類ごとに太さが変わってしまい、下流が全部その太さに縛られます。これを断ち切るのが「埋め込み次元の統一」です。

## 基礎: 前提となる概念
埋め込み次元(embedding dimension、embed_dim、Transformer文脈では d_model とも呼ぶ)とは、「1つの特徴ベクトルが何個の数字でできているか」です。画像の各位置を256個の数字で表すなら embed_dim=256 です。この次元数が、後段の融合モジュールや注意機構(attention)の計算量とパラメータ数を直接決めます。

backbone(バックボーン)とは、画像から特徴を抜き出す土台のネットワークです。ImageNet等で事前学習され、Swin Transformer、ConvNeXt、ResNetなど多くの種類があります。

マルチスケール特徴(multi-scale features)とは、backboneの浅い層から深い層までの、解像度とチャネル数が異なる複数の特徴マップです。浅い層は高解像度・少チャネルで細部を、深い層は低解像度・多チャネルで意味を捉えます。これらをチャネル方向に連結(concat)して使うと、合計チャネル数はbackbone依存になります。

たとえば代表的なbackboneの4ステージのチャネル数を足すと、以下のようにバラバラです。

```
Swin-Tiny:   96 + 192 + 384 +  768 = 1440 チャネル
ConvNeXt-T:  96 + 192 + 384 +  768 = 1440 チャネル
ResNet-50:  256 + 512 + 1024 + 2048 = 3840 チャネル
```

注意機構(self-attention / cross-attention)とは、特徴どうしの関連を重み付きで計算し、関連の強いものから情報を集める仕組みです。Transformer(Vaswani et al., 2017)の中核です。トークン数 N、次元 d とすると、計算量はおおむね N の2乗かつ d に比例し、内部の線形射影(Q・K・V や出力射影)は d の2乗で効くので、次元を下げる効果が大きく現れます。

## 仕組みを詳しく
やることは1つです。マルチスケール特徴をconcatした直後に、Conv1x1(カーネルサイズ1×1の畳み込み。空間方向には何もせず各位置のチャネルだけを混ぜて次元を変える層)を1枚かませ、backbone依存の太いチャネルを固定256へ写します。詳細は [チャネル射影層（Conv1x1）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0946_channel-projection-conv1x1)。

テンソルの形でbefore / afterを見ます(B=バッチ, V=視点数, 7×7=プール後の1視点解像度)。

- backbone出力(プール後にチャネル方向concat): `[B*V, 1440, 7, 7]`
- Conv1x1 + GELU で射影: `[B*V, 256, 7, 7]`
- 以降の視点融合・方策・予測モジュールはすべて256次元で動く。

ポイントは「backboneごとに違う入力チャネルを、出口で必ず256にそろえる」ことです。これにより下流は常に256次元しか見ないので、backboneを差し替えても下流モジュールを一切変えずに済みます(backbone非依存)。

なぜ256かを、削減効果の数字で見ます。

計算量の削減: 注意機構の線形射影は次元の2乗におおむね比例します。1440から256へ落とすと、

```
(1440 / 256)^2 = 5.625^2 ≒ 31.6 倍
```

つまり次元由来の計算が約31倍軽くなります。

パラメータの削減: 連結後の射影に使う線形層を例に取ると、入力次元が1440のときと256のときで、重み行列の要素数は次元に比例(または視点数×次元の積で効く)します。具体的な構成にもよりますが、おおむね数十分の1(約33分の1程度)のオーダーでパラメータが減るのが典型です。下流のTransformer層が複数あれば、その全層で d の2乗の節約が積み重なります。

backbone差し替えの容易さ: 次元がbackbone依存だと、ResNet-50(concatで3840チャネル)に替えた瞬間、入力次元が1440から3840に変わり、視点融合・方策・予測など下流の入力次元定義を全部直す必要があります。出口で256に固定すれば、この連鎖的な改修が不要になります。実験で「backboneを5種類試す」ようなとき、下流を一切触らずに比較できるのは研究効率上きわめて大きい利点です。

## 手法の系譜と主要論文
この分野のTransformer系の知覚・計画モデルは、ほぼ一貫して embed_dim=256 を採用しています。理由は「全モジュールで同じトークン次元を共有する設計が扱いやすく、256が精度と計算量のバランスの良い実用値として定着した」ためです。元をたどると、機械翻訳のTransformer(Vaswani et al., 2017)の base 構成が d_model=512 で、視覚系ではその半分の256が「軽くて十分」な点として落ち着いた経緯があります。

- DETR (Carion et al., ECCV 2020, arXiv:2005.12872) — 物体検出をTransformerで解いた先駆け。embed_dim=256。物体クエリと画像特徴を同じ256次元のトークン空間で扱う設計が、後続のBEV・計画モデルの土台になりました。

- Deformable DETR (Zhu et al., ICLR 2021, arXiv:2010.04159) — DETRの収束の遅さと計算量を、変形可能な注意(参照点周辺の少数の位置だけを参照する注意)で改善。次元はDETRと同じ256を踏襲しました。

- BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270) — BEVクエリ・画像特徴ともに256。BEV知覚の事実上の標準を確立しました。

- PETR / PETRv2 (Liu et al., ECCV 2022, arXiv:2203.05625 ほか) — 3D位置エンコーディングを画像特徴に注入する方式。256。

- UniAD (Hu et al., CVPR 2023, arXiv:2212.10156) — 検出・追跡・地図・予測・計画を一気通貫で解くモデル。全段256で統一し、モジュール間でトークンを受け渡します。256統一が「end-to-endでモジュールを連結する」設計を可能にした好例です。

- VAD (Jiang et al., ICCV 2023, arXiv:2303.12077) — 地図とエージェントをベクトル化してend-to-end計画。256。

系譜としては、DETRが「画像とクエリを共通の256次元トークンで扱う」原型を作り、BEV系・end-to-end計画系がそれを継承して、256が分野の事実上の標準次元になりました。共通する効果は「モジュール間でトークン次元が一致するので接続が自由」「256で十分な表現力がありつつ計算が現実的」です。

## 論文の実験結果(定量データ)
256という次元選択そのものを単独で検証したアブレーションは多くありませんが、256統一が支える「end-to-endでのモジュール連結」の有効性は、UniADの結果に明確に表れています。

UniAD は nuScenes で、検出・追跡・オンライン地図・動作予測・占有予測・計画を1つのネットワークで解き、各サブタスクで個別最適化された既存手法を上回るか同等を達成しました。特に計画(planning)で、衝突率(collision rate)と軌道誤差(L2誤差: 予測軌道と実際の軌道のメートル単位のずれ)を、従来のend-to-end手法に対して大きく下げたと報告されています。報告では平均L2誤差をおよそ0.3m台まで、平均衝突率を0.3%前後まで下げ、当時最良を示しました。これが可能だったのは、全モジュールが共通の256次元トークンで情報をやり取りでき、上流の検出・予測の特徴が下流の計画へ滑らかに流れたからです。次元がバラバラなら、この一気通貫の特徴伝播は成り立ちません。UniADのアブレーションでは、上流タスク(追跡・地図・予測)を1つずつ外すと計画性能が悪化することが示され、共通トークン空間を介した特徴の受け渡しが効いていることが裏付けられています。

BEVFormer は nuScenes で、カメラのみでLiDARに迫る検出精度(NDSでおよそ0.5台、mAPで0.4台)を達成しました。256次元のBEVクエリに多視点・時間情報を集約する設計が、この精度を支えています。

計算量の観点では、次元を2乗で効かせる注意機構を256に抑えることが、これらのモデルを現実的なGPUメモリ・推論速度で動かす前提になっています。1440のまま全段を組めば、次元由来の計算が約31倍に膨れ、リアルタイム運転には載りません。つまり256統一は「精度を保ちつつ実機に載せる」ための、性能と実用性のトレードオフの最適点として選ばれています。

## メリット・トレードオフ・限界
メリット:
- 注意機構の次元由来計算が約31分の1、連結後の射影パラメータが数十分の1のオーダーで削減される。
- backboneを差し替えても下流モジュールが無改変(backbone非依存)。
- 主要な公開モデル(DETR/BEVFormer/UniAD/VAD等)と同じ256で、比較・設定・事前学習済み重みの流用がしやすい。
- メモリが節約でき、高解像度BEVや長い時間文脈を扱う余地が広がる。

トレードオフ・限界:
- 1440(や3840)から256へ圧縮する際、元のマルチスケール特徴の情報を一部捨てる(表現力をやや削る)。
- 射影層という小さなモジュールと、わずかなパラメータ・計算が新たに増える。
- 256が常に最適とは限らない。大規模モデルや情報量の多いタスクでは、より大きい次元(512や768)が有利な場合もある。256は「中小規模で実用的」という経験則に基づく既定値であり、理論的最適ではない。

未解決の課題として、最適な埋め込み次元はモデル規模・データ量・タスクの複雑さに依存し、スケーリング則(モデルとデータを大きくしたときにどう性能が伸びるか)の観点で256が常に正しいわけではありません。基盤モデル化が進む運転分野では、より大きい次元への移行も検討余地があります。

## 発展トピック・研究の最前線
近年は、知覚から計画までを単一の大規模Transformerで解くend-to-end化がさらに進み、その文脈で「全段で共通のトークン次元を持つ」ことの重要性が増しています。256統一はその基礎インフラです。一方、モデルを大規模化する流れ(運転基盤モデル、視覚言語モデルとの統合)では、トークン次元を512や768、あるいはそれ以上に上げる選択も現れています。ここでのトレードオフは、表現力の向上と、注意計算の2乗増大・メモリ消費のバランスです。

実務的には、次元削減を担うConv1x1射影層(後述)を入口に置いておくと、こうした次元の変更をbackboneや下流から切り離して局所的に行えるため、次元設計の実験がしやすくなります。さらに、複数モダリティ(カメラ・LiDAR・地図・言語)を1つのトークン空間に統合する流れでは、各モダリティのエンコーダ出口で共通次元へ射影してから混ぜるのが定石で、256統一はマルチモーダル基盤モデルの前提部品としても再評価されています。
