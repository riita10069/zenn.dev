# NVIDIA PhysicalAI-Autonomous-Vehicles データセット

## ひとことで言うと

NVIDIA が HuggingFace(機械学習のモデルやデータを共有するプラットフォーム)上で公開している、実世界の自動運転データセットです。車体まわりの複数カメラ映像(おおむね7台)が中心で、HD マップ(高精細地図)は付いておらず、自車の動き(egomotion、自車が時間とともにどれだけ移動・回転したか)は parquet という表形式ファイルで提供されます。正解の将来軌道(人間が実際に走った経路)を含まないため、主に「学習済みモデルが本物のカメラ画像で最後まで落ちずに動くか」を確かめる動作確認(forward pass の検証)に向いています。

## 直感的な理解

自動運転モデルを作るとき、いきなり精度を測る前に確かめたいことがあります。それは「ランダムな数値や合成画像ではなく、本物のカメラ映像を流したとき、前処理からモデルの出力まで一気通貫で壊れずに動くか」です。これを sanity check(健全性確認)と呼びます。

合成データやランダムテンソル(でたらめな数値で埋めた配列)での動作確認では、テンソルの形やメモリ使用量は分かりますが、「実画像特有の落とし穴」は見えません。たとえば解像度が想定と違う、色の正規化がずれている、カメラ台数がモデルの期待と合わない、egomotion のファイル形式が読めない、画像コーデック(動画の圧縮形式)が想定と違ってデコードできない、といった問題は実データを通して初めて表面化します。料理にたとえると、レシピ(コード)が机上で正しくても、実際の食材(実データ)を使うと「分量が違う」「器が足りない」と気づくのと同じです。

このデータセットが選ばれる理由は、(1) HuggingFace から手軽に入手でき、(2) 実世界の多様なシーン(NVIDIA の説明によれば多数の国・都市をカバーする大規模な走行映像)を含み、(3) egomotion が機械可読な parquet で付いているため自車状態の入力を実データで埋められるからです。一方で正解の将来軌道が無いという制約が、用途を「動作確認」に限定します。NVIDIA はこのデータを Physical AI(物理世界で動く AI。ロボットや自動運転車)の基盤モデル学習・評価のために整備しており、Cosmos(世界モデル、世界の動きを生成・予測するモデル)などの研究エコシステムと結びついています。

## 基礎: 前提となる概念

- forward pass(順伝播): 入力をモデルに与え、層を順に通して出力を得る一回の計算。正解との比較や勾配計算(backward pass、誤差を逆に伝えて重みを直す計算)を伴わない。動作確認では「クラッシュせず出力が出る」ことだけを見る。
- egomotion(自車運動): 自車が時間とともにどれだけ並進(平行移動)・回転したかの記録。位置・速度・姿勢(向き)などを時刻ごとに並べた時系列。将来軌道の予測には「今どう動いているか」という初期条件が要るため重要。
- HD マップ(High-Definition map): 車線・停止線・信号位置などをセンチメートル級の精度で記録した詳細地図。多くの自動運転モデルが入力に使うが、このデータセットには含まれない。
- parquet: Apache が定めた列指向(column-oriented)の表データ形式。CSV のように行と列のある表を、列ごとにまとめて圧縮して保存する。CSV より高速に読め、ファイルが小さく、各列の型(数値・文字列など)を保持できる。egomotion のような数値時系列の保存に適する。読み込みは pandas や pyarrow で数行で済む。
- ImageNet 正規化: 事前学習済みの画像バックボーン(画像から特徴を取り出す土台ネットワーク)が期待する入力分布にそろえるため、各色チャンネルから ImageNet データセットの平均を引き標準偏差で割る操作。これをしないと学習済みの重みが想定外の入力を受け取り性能が落ちる。
- ドメインシフト(domain shift): 訓練したデータと、実際に使うデータの分布が違うこと。地域・天候・カメラ機種・時間帯が訓練と異なると、見た目の統計がずれて性能が落ちる。多様な国・都市を含む実データはこのドメインシフトの検証に向く。

## 仕組みを詳しく

### データの中身と、典型的なモデル入力とのギャップ

多くのマルチカメラ E2E モデルは入力テンソルの「形」が固定されており、実データの形がそれと食い違うと動きません。このデータセットを使う際に典型的に発生するギャップと、その回避策を整理します。

| ギャップ | 典型的な回避策 |
|---|---|
| カメラが7台(モデルが8視点 = 7カメラ + マップタイルを期待) | 8番目の視点をゼロ埋め(zero-padding) |
| HD マップが含まれない | マップ用スロットにゼロテンソルを入れる |
| egomotion が parquet で提供 | 読み込んで固定次元のベクトルにエンコード |

順に意味を説明します。

1. カメラ台数の不一致。あるモデルが「8視点(うち1つはマップタイル用)」を前提に設計されている場合、7カメラしかないデータでは8番目の枠が空きます。足りない枠を全要素0の画像テンソルで埋めて形を `[B, 8, 3, H, W]` にそろえる、というのが zero-padding です。これは「情報は無いが形だけ合わせる」応急処置で、その限界は [カメラ数不一致のゼロパディング対処](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0799_camera-count-mismatch-padding) で詳述します。

2. HD マップ無し。マップ画像を入力に取れるモデルでも、地図が無ければマップスロットにゼロテンソルを入れて「地図情報なしでカメラだけで運転を試す」設定にします。地図に依存する判断(右折レーンの有無など)はできなくなります。

3. egomotion の parquet。各時刻の自車の位置・速度・姿勢などが行として並んだ parquet を読み込み、モデルが期待する固定次元のベクトル(たとえば 256 次元)にエンコードして自車状態履歴の入力に流します。後述の AD-MLP が示したように、この自車状態は軌道予測に対して非常に強い手がかりになるため、実データで egomotion を正しく扱えるかは評価の信頼性に直結します。注意点として、parquet に記録された座標がグローバル座標(地図上の絶対位置)なのか自車中心座標なのか、単位(m/s か km/h か)を確認しないと、エンコード後の数値スケールがずれてモデルが暴れます。

### 動作確認の流れ

実データでの forward pass 検証は、典型的には次の段階を踏みます。

```
[HuggingFace から小チャンク(数GB)取得]
        ↓
[MP4 クリップからフレーム抽出 → リサイズ(例 256×256)→ ImageNet正規化]
        ↓
[7カメラ + ゼロ埋め1枚 → 形 [B,8,3,256,256] / マップスロットもゼロ]
        ↓
[egomotion parquet を読み固定次元ベクトルへ]
        ↓
[モデルへ forward pass → 出力(予測軌道など)が出ることを確認]
```

正解軌道が無いので、この段階では ADE のような正確さは測れません。あくまで「ダウンロード → 前処理 → 推論」が実カメラデータで end-to-end に通り、クラッシュせず出力が出ることの確認です。正確さの計測は正解軌道を持つデータ([Waymo Open End-to-End Driving データセット](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0903_waymo-e2e-dataset))に進んでから行います。

### CPU でも回せる軽さ

小さなチャンク(数本のクリップ、数 GB 程度)を選べば、GPU が無い環境(ノート PC の CPU のみ)でも forward pass を回せます。これは初期検証データとして選ばれる実務的な理由の一つで、開発の最初期に「手元の貧弱な環境でもパイプラインが通るか」を確かめられます。バッチサイズ1・低解像度なら CPU でも数秒で1フレーム処理でき、CI(継続的インテグレーション、コミットのたびに自動でテストを回す仕組み)に組み込んで「実データが通り続けるか」を監視する用途にも使えます。

## 手法の系譜と主要論文

このデータセット自体は NVIDIA が公開した実データ資産であり、特定の手法を提案する論文ではありません。関連する評価上の背景を系譜で示します。

- AD-MLP(Zhai et al., 2023, arXiv:2305.10430)。自車の過去状態(ego status)だけを単純な MLP に入れるだけで、nuScenes の planning ADE が当時の最先端カメラ入力 E2E 手法に匹敵してしまうことを示した批判的研究。この知見から、egomotion が parquet で整っているデータは「ego-status だけのベースライン」を実データで容易に組めるため、カメラを本当に活かせているかの検証台として価値があります。

- 正解将来軌道を持つベンチ(nuScenes 系の UniAD / VAD、Waymo の E2E チャレンジ、[Waymo Open End-to-End Driving データセット](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0903_waymo-e2e-dataset))。本データセットはこれらの「本計測」に進む前段の forward pass 検証データという位置づけ。HD マップ・他車アノテーションが無いため、ADE/FDE や衝突率の本計測には向きません。

- sim-to-real とデータ生成の潮流。CARLA(Dosovitskiy et al., CoRL 2017, arXiv:1711.03938)に代表される運転シミュレータは、危険シーンを安全かつ無限に生成できる一方、見た目が現実と違う(ドメインギャップ)弱点があります。NVIDIA はこの実データ供給に加え、Cosmos のような世界モデルで現実的な走行シーンを生成し、実データと合成データを混ぜて学習する流れを推進しています。本データはその実データ側の供給源として位置づけられます。

- データバージョニングと大規模実データの基盤化。Physical AI 向けに、メーカーが大規模実走行データを HuggingFace のような共有基盤で公開する流れが強まっており、本データはその一例。WebDataset や LeRobot(ロボット学習用データ規格)などの標準フォーマットとの連携も進んでいます。

注: 正確なカメラ配置・台数、サンプリングレート、parquet の列定義、ライセンス条件は HuggingFace の公式データセットカードが最終的な正となります。本ノートは公開情報と実データ検証の一般的な慣行に基づきます。

## 論文の実験結果(定量データ)

このデータセット単体のベンチマーク数値は性格上存在しません(正解軌道が無いため)。代わりに、なぜ「egomotion が整っていること」が評価上重要かを、関連研究の数値で示します。

- AD-MLP(2023)のアブレーション: カメラ・LiDAR・地図を一切使わず ego status だけを MLP に入れた場合でも、nuScenes の平均 L2 誤差(1・2・3秒先の平均)がおよそ 0.3〜0.5 m 台と報告されました。ここで L2 誤差とは予測位置と正解位置のユークリッド距離(直線距離)で、小さいほど良い。カメラ入力を持つ当時の E2E 手法(平均 L2 がおよそ 1.0 m 前後)より良い値が出てしまった、という点が衝撃的でした。意味するところは「運転は慣性が支配的で、直前の速度・加速度を延長するだけで多くの場面が当たる」ということ。

この結果から得られる実務的教訓は、実データ検証で egomotion を入力に使うなら、必ず「ego-status だけのベースライン」を併走させ、カメラを足したときに ADE がどれだけ下がるか(知覚の純増分)を測るべき、ということです。本データセットは egomotion が parquet で機械可読に提供されるため、このベースライン構築が容易です。逆に egomotion を全ゼロにして検証した場合の数値は、自車状態という最強の手がかりを捨てた条件であり「モデル本来の能力を反映しない」ことに注意が必要です。

もう一つの実務上の数値感覚として、ドメインシフトの効き方があります。多くの研究で、ある地域・カメラ機種で訓練したモデルを別地域で評価すると、検出や軌道予測の指標が二桁パーセント単位で劣化することが報告されています。多様な国・都市を含む本データは、こうした「訓練分布の外」での頑健性を確かめる素材として価値があります。

## メリット・トレードオフ・限界

メリット

- HuggingFace から手軽に入手でき、小チャンクなら GPU 無しの CPU 環境でも回せる。開発初期の sanity check に最適で、CI への組み込みも容易。
- 実世界の多様なシーンをカバーし、合成データでは見えない前処理の落とし穴(解像度・正規化・台数差・コーデック)を早期に発見できる。
- egomotion が機械可読な parquet で付き、自車状態入力を実データで埋められ、ego-status ベースラインを組みやすい。
- ドメインシフトへの頑健性評価の素材として使える。

トレードオフ・限界

- 正解将来軌道が無く、ADE/FDE など正確さの指標を計算できない(forward pass 検証に限定)。
- HD マップ・他車アノテーション(3D ボックス)が無く、地図融合や衝突率評価に使えない。回避策はゼロ埋めで形だけ合わせること([カメラ数不一致のゼロパディング対処](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0799_camera-count-mismatch-padding))で、これは性能を劣化させうる応急処置。
- カメラ台数がモデルの期待とずれる場合があり、ゼロ埋めや入力レイアウトの調整が要る。
- egomotion の座標系・単位の解釈ミスが数値スケールの破綻を招きやすい。
- ライセンス・利用規約の確認が必要。

研究上の未解決課題として、実データ単体では「予測が正しいか」を測れないため、正解軌道つきデータや closed-loop シミュレータと組み合わせる必要があります。また、大規模実データを使う際のドメインシフトへの頑健性評価と、合成データとの最適な混合比の探索は継続的な研究テーマです。

## 発展トピック・研究の最前線

- WebDataset などのシャーディング形式。大規模実データを効率良く DataLoader(学習にデータを供給する仕組み)に流すため、tar アーカイブを分割(shard)してストリーミング読み込みする方式が普及しています。parquet の egomotion と画像シャードを組み合わせる前処理パイプラインが実務の標準です。
- sim-to-real とドメイン適応。実データ(本データセット等)と合成データ・世界モデル生成データを混ぜて学習し、現実への転移性能を高める研究。NVIDIA Cosmos のような生成基盤との組み合わせが最前線。
- 自己教師あり学習。正解軌道が無くても、未来フレームの予測や対照学習(contrastive learning、似たものを近づけ違うものを遠ざける表現学習)で表現を学ぶことで、ラベルの無い大量の実映像を活用する方向([自己回帰的 visual_history のローリング更新](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0916_autoregressive-visual-history))。
