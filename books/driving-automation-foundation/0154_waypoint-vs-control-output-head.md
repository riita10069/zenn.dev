# waypoint出力 vs 制御量出力ヘッド

## ひとことで言うと

運転モデルの一番最後の部分（出力ヘッド）が、「未来の位置 (x, y) の並び（waypoint）」を出すか、「加速度・曲率・操舵角といった制御量」を出すか、という設計上の二択です。どちらを選ぶかで、学習の安定性・累積誤差への強さ・評価指標との一致度・実装の手間が変わります。さらに現実には、学習に使うデータセットの教師軌道がどちらの形式で配られているかという制約も絡みます。

「出力ヘッド（output head）」とは、モデルの最後にある小さな層（多くは1〜数層の全結合層、fully-connected layer）で、内部で計算した特徴ベクトルを「最終的に欲しい形（ここでは軌道）」の数値へ変換する部分のことです。同じバックボーン（backbone、特徴抽出の本体。画像から意味のある特徴ベクトルを取り出す大きなネットワーク）に対して、ヘッドだけ差し替えれば出力形式を変えられます。

## 直感的な理解

ドライビングスクールで教官が生徒に指示を出す場面を想像してください。

教官Aは「次の角を曲がったら、あの電柱の手前に車を止めなさい」と、目標地点（位置）で言います。生徒はその位置に着くために、ハンドルとアクセルを自分で細かく調整し、ずれたら毎瞬補正します。これがwaypoint出力に相当します。位置という「目標」を出し、実際の操作は下流のコントローラに任せる発想です。

教官Bは「今すぐハンドルを15度右、アクセルはこのまま」と、操作そのもの（制御量）を言います。指示どおり動かせば一応進みますが、もし最初の指示がほんの少しずれていると、その誤差を補正する仕組みがないまま走り続け、数秒後には大きく道を外れます。これが制御量出力に相当し、「累積誤差」に弱いという弱点が直感的に分かります。

## 基礎: 前提となる概念

模倣学習（imitation learning / behavior cloning、行動クローニング）とは、人間の運転ログ（このときこの状況でこう操作した、という記録）を教師データにして、状況→行動の対応をそのまま真似るよう学習する方法です。教師あり学習として素直ですが、固有の弱点があります。

分布シフト（distribution shift）とは、学習時に見たデータの分布と、実際に動かすとき（推論時）に出会う分布がずれる現象です。模倣学習では、モデルが少しでも教師と違う行動をすると、車は教師が一度も訪れなかった状態へ入っていきます。そこではモデルは正しい行動を学んでいないので、さらに大きくずれ、雪だるま式に破綻します。これを compounding error（累積誤差）と呼びます。

開ループ（open-loop）と閉ループ（closed-loop）の違いも決定的です。開ループ評価は、ある1フレームの予測を人間の実走軌道と比べるだけで、予測どおりに車を動かしはしません。閉ループ評価は、モデルの出力を実際に車（やシミュレータ）に適用し、その結果として変化した次の状態をまた入力に戻す、というループを回します。累積誤差・分布シフトは閉ループでしか現れないので、出力ヘッドの良し悪しは開ループ数値だけでは判断しきれません。

waypoint表現と制御量表現の中身は [加速度・曲率による軌道表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0108_acceleration-curvature-trajectory) で詳しく扱いますが、ここでは次の2系統がある、とだけ押さえます。

1. waypoint表現: 未来の各時刻の位置 (x, y) を並べる。評価指標（位置誤差・衝突率）と直結する。
2. 制御量表現: 各時刻の加速度・曲率（あるいは操舵角・速度）を並べ、積分して位置に復元する。滑らかで値のスケールが安定する。

## 仕組みを詳しく

出力ヘッドの違いを形で書くと次のようになります。どちらも見た目のテンソル形状は同じ `[B, T, 2]`（バッチ B、時刻 T、各時刻2成分）になりうる点が、実務上の落とし穴です。

```
制御量ヘッド:   特徴 → Linear → [B, T, 2] = (加速度 a[t], 曲率 κ[t])  → 必要なら積分で位置へ
waypointヘッド: 特徴 → Linear → [B, T, 2] = (x[t], y[t])             を直接出力
```

### 形は同じでも意味が違う、という核心

教師軌道（ground truth, GT、正解データ）がwaypointで配られているのに、モデルが制御量を出していると、次の状況になります。

```
GT (waypoint):    [B, T, 2] = (x[t], y[t])
予測 (制御量):    [B, T, 2] = (a[t], κ[t])      ← 形は同じだが意味が違う
→ そのまま損失に入れると、位置と加速度を比べることになり無意味
```

形状チェックは通ってしまうのでコードは「動く」ように見えますが、学習はまったく進みません（損失が下がらない、あるいは無意味な値に収束する）。この種のバグは静かで発見が遅れるため、ヘッド・損失・GTの3者の「意味」を揃えることが決定的に重要です。一般則として、テンソルの形状（shape）が一致することと、テンソルの意味（semantics）が一致することは別物だ、と肝に銘じる必要があります。

### 食い違いを吸収する2つの方針

方針A: ヘッドをGTの形式に合わせる。GTがwaypointなら出力ヘッドをwaypoint出力に差し替える。損失計算の枠組み（時刻ごとに誤差を測る）はほぼ流用でき、実装差分は小さい。ただし学習挙動（滑らかさ・累積誤差への強さ）は変わる。

方針B: 出力はそのままにして変換でつなぐ。
- 経路B1: GT側を制御量へ変換する。waypointの一階差分から速度、二階差分から加速度、進路の角度差から曲率を逆算し、制御量空間で損失を測る。二階差分はノイズを増幅するため、逆算でノイズが乗りやすい（実走ログのGPS/IMU揺らぎが拡大される）。
- 経路B2: モデルの制御量出力を微分可能な積分層（前項で述べた運動学的自転車モデルの離散化を学習グラフに組み込んだもの）で位置に変換し、waypoint空間で損失を測る。評価指標（位置）に直結する一方、積分が学習グラフに入るので勾配の経路が長くなり、初期速度・初期向きの推定誤差が全時刻に伝播する。さらに勾配がステップ数 T に沿って連鎖するため、長いホライズンでは勾配消失・爆発に注意が要る。

どれを選ぶかは、データセットのGT形式・評価指標・許容できる実装複雑度で決まります。一般則として「学習空間」と「評価空間」をできるだけ一致させると、検証時の挙動が読みやすくなります。

## 手法の系譜と主要論文

制御量出力（古典）。ALVINN（Pomerleau, NeurIPS 1988）と DAVE-2（Bojarski et al., NVIDIA 2016, arXiv:1604.07316）は、カメラ画像から操舵角という制御量を直接回帰しました。実装は最小限で済む一方、補正機構がないため累積誤差に弱いことが後に問題化します。DAVE-2 は学習データに左右ずらしカメラの映像と補正操舵を人為的に加える（データ拡張で復帰行動を教える）ことで、累積誤差をある程度緩和しようとした点が示唆的です。

累積誤差・分布シフトの理論化。Ross et al. の DAgger（AISTATS 2011, arXiv:1011.0686）は、模倣学習の誤差がホライズン長 T に対して最悪 O(T²) のオーダーで増えうることを理論的に示し、学習中に自分が訪れた状態へ教師ラベルを足していく対策（DAgger, Dataset Aggregation）を提案しました。Codevilla et al. "Exploring the Limitations of Behavior Cloning"（ICCV 2019, arXiv:1904.08980）は、制御を直接出す behavior cloning の限界を実証的に分析し、データ量を増やしても性能が頭打ちになること、慣性問題（inertia problem、止まっていると止まり続けたがる）などの失敗モードを報告しました。

waypoint出力（近年の主流）。UniAD（Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156）と VAD（Jiang et al., ICCV 2023, arXiv:2303.12077）は、計画ヘッドが位置（waypoint）を出し、L1系の損失と衝突ペナルティで学習します。評価ベンチマーク（nuScenes の L2 誤差・衝突率）が位置で定義されるため、位置で出して位置で測るのが指標と一直線で結びつく、という動機です。

中間解。出力を運動学パラメータ（加速度・曲率）で出し、積分して軌道に直す方式は、制御量の「物理的妥当性」と軌道の「補正しやすさ・滑らかさ」を両立させようとするものです。位置を目標として出し、下流の追従コントローラ（pure pursuit や MPC, Model Predictive Control）で毎フレーム補正する2段構成も、累積誤差に強い実務的な定番です。pure pursuit は「目標点を見て、そこへ向かう円弧を毎瞬引き直す」幾何追従則、MPC は「短い未来のコストを毎瞬最小化して最初の1手だけ実行する」最適化ベースの追従則です。

## 論文の実験結果（定量データ）

waypoint出力の有効性。UniAD（CVPR 2023）は nuScenes で1〜3秒平均の L2 誤差約 1.03m、平均衝突率約 0.31% を達成し、ST-P3（Hu et al., ECCV 2022, 平均 L2 約 2.11m, 衝突率約 1.27%）など先行手法を上回りました。L2 誤差は予測軌道と人間の実走軌道の各時刻位置ずれ（m）の平均で、小さいほど人間に近い計画です。衝突率は予測どおり走ると他物体にぶつかる割合で、安全性の代理指標です。位置で出力した方が、これら位置ベース指標で素直に良い数字が出ます。

制御量直接出力の弱さ。Codevilla et al.（ICCV 2019）は CARLA シミュレータで、制御を直接回帰する behavior cloning の成功率（決められた区間を事故なく走り切る割合）が、訓練データを増やしても飽和し、初期条件のわずかな違いで結果が大きくばらつく（high variance, 同じ設定で乱数シードを変えるだけで成功率が大きく変わる）ことを示しました。これは「制御量を直接出す設計が累積誤差・分布シフトに弱い」ことの定量的裏づけです。

追従コントローラを挟む効果。waypoint を出して下流の幾何コントローラに追従させる2段構成は、毎フレーム位置を再計画して補正できるため、制御直接出力に比べてシミュレータ閉ループ（実際に車を走らせて評価する設定）での走破率が高くなる傾向が、Learning by Cheating（Chen et al., CoRL 2019）などで報告されています。なお nuScenes の標準計画評価は開ループであり、開ループの良さが必ずしも閉ループの良さを意味しないという点は重要です。AD-MLP（Zhai et al., 2023, arXiv:2305.10430）は、カメラ画像をいっさい使わず自車の過去状態（速度・加速度・操舵の履歴）だけを多層パーセプトロンに入れるだけで、nuScenes の開ループ計画指標で当時の最先端と同等以上の L2 誤差が出てしまうことを示しました。これは「nuScenes の自車運動には強い慣性（過去の動きを延長するだけでかなり当たる）があり、開ループ L2 は知覚や計画の良し悪しをほとんど測れていない」という痛烈な指摘で、出力ヘッドの議論を開ループ数値だけで結論づける危うさを定量的に突きつけました。

## メリット・トレードオフ・限界

waypoint出力ヘッド
- メリット: 多くのデータセットのGT形式と一致し変換不要。位置誤差・衝突率という評価指標に直結。下流の追従コントローラで毎フレーム補正でき、累積誤差に強い。
- トレードオフ・限界: 滑らかさ・運動学的実行可能性を別途担保しないと出力がガタつく。値のスケールが時刻で大きく変動し、損失が遠未来に偏りやすい（[軌道損失の時間重み付け](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0584_temporal-loss-weighting) で対処）。

制御量出力ヘッド
- メリット: 値のスケールが安定し、積分で必ず滑らかな軌道になる。車両制御へ直接渡しやすい。物理的に不可能な軌道を構造的に出しにくい。
- トレードオフ・限界: GTがwaypointだと変換が必要。学習空間（制御量）と評価空間（位置）がずれる。補正機構なしに開ループで使うと累積誤差・分布シフトに弱い。

共通の落とし穴
- 「形は同じ `[B, T, 2]` でも中身の意味（位置か制御量か）が違う」ため、ヘッド・損失・GTの意味を揃えないと、見かけ上は動くのに学習が壊れる。

研究上の未解決課題として、開ループ指標と閉ループ性能の乖離をどう埋めるか、また微分可能な運動学積分層を通して「制御量で学びつつ位置で評価する」両取りをどこまで安定して実現できるか、が挙げられます。

## 発展トピック・研究の最前線

近年は出力ヘッドを「1本の軌道」ではなく「複数候補の確率分布」にする設計が主流です。MultiPath++（Varadarajan et al., ICRA 2022）や Wayformer（Nayakanti et al., ICRA 2023, arXiv:2207.05844）は混合分布で複数モードを出し、各モードを waypoint でも運動学パラメータでも表せます。さらに軌道を離散トークンに量子化して言語モデル風に自己回帰生成する方式（trajectory tokenization）や、拡散モデルで軌道を生成する方式も登場し、「ヘッドは何を出すべきか」という問いは、生成モデル・大規模言語モデルとの接続を含む広い設計問題へ拡張しています。また、開ループ偏重への反省から、nuPlan や Bench2Drive など閉ループ評価を前提としたベンチマークへ研究の重心が移りつつあり、そこでは「追従しやすく実行可能な出力をどう作るか」というヘッド設計が改めて重要になっています。
