# 合成軌道による学習ループ検証

## ひとことで言うと

実データで本格的に学習を始める前に、わざと人工的に作った「答え(正解軌道)」を使ってモデルを一度だけ動かしてみる検証のやり方です。最終的な損失関数(モデルの「間違い具合」を測る計算式)をきちんと設計する前に、「データを入れて → 予測して → 誤差を計算して → パラメータを更新する」という学習の一連の流れが端から端まで(end-to-end で)正しく回るかどうかだけを先に確かめます。賢くなったかどうかは問わず、配線が壊れていないことの確認に専念するのが要点です。深層学習の実務では「sanity check(健全性確認)」と呼ばれる作業の中核に位置します。

## 直感的な理解

新しく組み立てた電気回路を、いきなり本番の高電圧で動かす人はいません。まずテスター電源やダミー負荷をつなぎ、配線がショートしていないか、電流が末端まで流れるかを低リスクで確かめます。合成軌道による検証は、これと同じ発想を深層学習に持ち込んだものです。

自動運転の end-to-end モデル(カメラ画像などの入力から運転操作・未来軌道までを 1 つのニューラルネットワークで一気に出す方式)は、入力も出力も巨大で、内部には複数のサブネットワーク(画像特徴抽出、複数カメラの融合、時系列の符号化、軌道のデコードなど)が連なります。これを実データと本番用の複雑な損失で同時に動かすと、もし学習が進まなかったときに「データが悪いのか、モデルの配線が間違っているのか、損失の設計が悪いのか」が切り分けられません。原因の候補が多すぎてデバッグが地獄になります。

そこで「正解データを人工的に作る」ことで変数を一つ消します。データの正しさをいったん脇に置けば、残るのは「モデルの配線」と「勾配が末端まで流れるか」だけ。最小コストでそこだけを確認できます。これは科学実験で「対照群を用意して一度に一変数だけ動かす」考え方と同じで、デバッグの探索空間を縮めるための工学的な工夫です。

## 基礎: 前提となる概念

このトピックを理解するために、深層学習の学習ループを構成する用語をかみ砕いておきます。

- テンソル(tensor): 数値を多次元に並べた配列のこと。画像なら `(チャンネル, 高さ, 幅)` のような形(shape)を持ちます。バッチ(一度にまとめて処理する複数サンプル)を先頭につけて `(B, ...)` と書くのが慣例です。
- forward(順伝播): 入力テンソルをモデルに通して予測を出す計算。
- loss(損失): 予測と正解の食い違いを 1 つの数値(スカラー)にまとめたもの。小さいほど良い。
- backward(逆伝播): loss を各パラメータで微分し、「このパラメータをどちらに動かせば loss が下がるか」(勾配, gradient)を計算する処理。連鎖律(合成関数の微分)を自動で適用する仕組み(autograd, 自動微分)で実現されます。
- optimizer step: 計算した勾配を使って実際にパラメータを少し動かす更新操作。SGD や Adam といった最適化アルゴリズムが担います。
- 計算グラフ(computational graph): forward の各演算をノード、テンソルをエッジとして表した有向グラフ。backward はこのグラフを逆向きにたどって勾配を伝えます。途中で `detach()` したり、NumPy など微分追跡外の処理が挟まったりするとグラフが「切れて」勾配が流れなくなります。

学習ループはこの 4 つ(データ読み込み → forward → loss → backward + step)を何万回も繰り返します。どこか 1 か所でもテンソルの形が合わなかったり、計算グラフが途中で切れていたりすると、学習はそもそも成立しません。形の不一致(shape mismatch)、勾配がパラメータまで届かない(`requires_grad` が立っていない、グラフから切れている)、GPU メモリ(VRAM)に乗り切らない(out-of-memory, OOM)、といった故障はどれも「賢さ」以前の問題で、合成データでも必ず露見します。逆に言えば、こうした故障は合成データで「最も安く」発見できます。

合成軌道(synthetic trajectory)とは、実際の運転ログから取った正解ではなく、乱数生成器などでその場で作ったダミーの正解軌道のことです。たとえば軌道を「64 タイムステップ × 2 信号(加速度・曲率)」を平らに並べた長さ 128 のベクトルで表すなら、合成軌道は `torch.randn(B, 128)` のような正規分布乱数で埋めたテンソルになります。中身に運転の意味はありませんが、形(shape)と型(dtype)は本物とそろっています。配線の検証に必要なのはこの「形と型の整合」だけで、意味は要りません。

## 仕組みを詳しく

典型的な検証は、ランダムな入力 1 バッチを作る関数を中核に据えて次の手順を取ります。

```
# B: バッチサイズ, V: カメラ視点数
visual_input      = randn(B, V, 3, 256, 256)   # 複数視点のダミー画像
egomotion_history = randn(B, 256)              # 過去の自車運動 (64ステップ×4信号)
trajectory_target = randn(B, 128)              # ← 合成軌道。64ステップ×2信号 (加速度・曲率)
```

学習ループの中身を分解すると次のようになります。

1. 上のようにランダムな入力 + ランダムな `trajectory_target` を 1 バッチ作る。
2. モデルに入力を通し予測 `trajectory` を得る。形は `(B, 128)`。
3. `loss = loss_fn(trajectory, trajectory_target)` で予測と合成正解の差を計算する。`loss_fn` は最も単純な MSE(平均二乗誤差)か Smooth L1 を使う。
4. `loss.backward()` で勾配を計算する。
5. `optimizer.step()` でパラメータを更新する。これを数 step 繰り返す。

ここで決定的に重要なのは「合成軌道だから loss の数値そのものには意味がない」という点です。ランダムな正解に対する誤差なので、loss が下がるか上がるかで賢さは測れません。確認したいのはあくまで次の 3 点だけです。

- 形が最後まで合っているか。`(B, 128)` のまま loss の計算まで届くか。
- `loss.backward()` がエラーを出さず、勾配が軌道デコーダや各サブネットワークのパラメータまで流れるか(勾配が `None` や全ゼロになっていないか)。
- VRAM に乗り切るか、step が最後まで回り切るか。

before(検証前): forward は通るかもしれないが、学習機構は一度も動かしたことがなく、パラメータは初期値のまま。
after(検証後): 合成軌道に対して loss → backward → step が数 step 走り、配線とメモリが健全だと確認できた状態。実データはこの後で安心して差し込める。

なぜ MSE から始めるかというと、最も単純で挙動が読みやすい損失だからです。`MSE = mean((予測 - 正解)^2)` で、差を二乗して平均するだけ。配線確認の段階では損失の良し悪しを議論する必要がなく、単純なほど切り分けが楽です。MSE は予測がどんな値でも有限の滑らかな勾配を返すので、勾配が流れているかどうかの観測にも向きます。

この検証には強力な拡張版があります。「1 バッチに意図的に過学習させる(overfit a single batch)」テストです。合成データを乱数で毎回作るのではなく、固定した 1 バッチを何百 step も繰り返し学習させ、loss がほぼ 0 まで下がるかを見ます。下がるなら「モデルは少なくとも与えられた答えを暗記できる表現力(容量)を持ち、勾配も正しく流れている」ことが保証されます。下がらなければ、それは賢さの問題ではなく確実なバグ(計算グラフが切れている、学習率が不適切、ラベルがずれている、入力と正解の対応が壊れている等)の証拠です。これは「機構が答えを覚えられるか」というモデル容量と勾配伝播の同時テストになっています。

数値例で直感をつかみましょう。固定した 1 バッチで MSE を観測したとき、step 0 で 1.20、step 50 で 0.04、step 200 で 0.0003 のように下がっていけば、配線と勾配は健全です。逆に 1.20 のまま全く動かなければ、勾配が届いていない疑いが濃厚です。途中で `NaN`(非数, Not a Number)になれば、数値が発散している(学習率が大きすぎる、入力が正規化されていない、log や除算でゼロを踏んだ等)サインです。

実務では段階を踏んだ多段の sanity check を設計します。粗い順に並べると次のようになります。

1. 形の検査: forward を 1 回通し、各テンソルの shape を print して期待どおりか確認する。
2. 勾配の検査: `loss.backward()` 後に各パラメータの `.grad` が `None` でないか、ノルムが 0 でないかを層ごとに確認する。
3. 学習率ゼロテスト: 学習率を 0 にして数 step 回し、loss が「動かない」ことを確認する。もし学習率 0 でも loss が変わるなら、入力に依存しない経路(バッチ正規化の統計更新、dropout の乱数など)に隠れたバグがある。
4. 初期 loss の理論値照合: 分類なら初期 loss が `-log(1/クラス数)` 付近になるはずで、ずれていれば出力スケールやラベル定義が疑わしい。回帰なら出力を 0 に固定したときの loss と一致するか確認する。
5. overfit-single-batch: 上述の固定 1 バッチ過学習で、loss がほぼ 0 まで落ちることを確認する。

## 手法の系譜と主要論文

合成データやごく小さなデータで学習機構を先に検証する考え方は、深層学習実務における sanity check として古くから定着しています。

- Andrej Karpathy, "A Recipe for Training Neural Networks"(2019, ブログ記事)は、ニューラルネットの訓練で最初にやるべきことの 1 つとして「ごく小さなデータに意図的に過学習させ、loss がほぼ 0 まで下がることを確認する。下がらなければバグがある」と明言しました。これは「正解が人工的でも構わないので、まず機構が答えを覚えられるかを見る」という本トピックそのものです。Karpathy はさらに「学習率を 0 にしても loss が動くなら入力依存性のバグ」「初期 loss の理論値と一致するか確認」「入力をシャッフルしても loss が下がるなら入力を無視している」といった一連の sanity check を体系化しました。
- 理論的背景は Goodfellow, Bengio, Courville の教科書 "Deep Learning"(MIT Press, 2016)第 11 章「実践手法」が整理しており、「まず単純なベースラインで配線を確認してから複雑化せよ」「学習データに過学習できないモデルは、汎化以前に最適化が失敗している」という原則を述べています。最適化の失敗(under-fitting)と汎化の失敗(over-fitting)は別物で、合成軌道検証は前者を切り分ける道具です。
- ソフトウェア工学の観点では、機械学習システムのテスト自動化コミュニティが「ML のためのスモークテスト/メタモルフィックテスト」を整理してきました。出力の絶対値ではなく、入出力の整合性・不変条件(invariant)を確認するという発想は、合成軌道検証と同根です。
- 軌道を (x, y) のウェイポイント列ではなく (加速度, 曲率) のような制御量で表す表現選択は、近年の end-to-end 運転研究で学習の数値安定性が上がると報告されています。ただし合成軌道検証の段階では中身が乱数なので、この知見は「最終的に何を予測対象にするか」に効くものであり、検証フェーズの本質ではありません。
- UniAD(Hu et al., "Planning-oriented Autonomous Driving", CVPR 2023, Best Paper)や VAD(Jiang et al., "VAD: Vectorized Scene Representation", ICCV 2023)は最終的な軌道損失や安全制約の設計を論じる論文で、合成軌道検証そのものは主題にしていません。しかし、こうした複雑な複合損失(模倣損失 + 衝突ペナルティ + 車線逸脱ペナルティ等)を組む前段として、単純損失で配線を確認するステップは現場では暗黙の前提になっています。複合損失をいきなり組むと、どの項がバグっているのか切り分けられないからです。

## 論文の実験結果(定量データ)

合成軌道検証は「モデルの精度を上げる手法」ではなく「開発プロセス上の検証手法」なので、論文のリーダーボード数値で語られることは多くありません。とはいえ定量的な効果は実務で繰り返し観測されています。

- overfit-single-batch テストの典型的な振る舞いとして、表現力のあるモデルなら固定 1 バッチの MSE を数百 step で 1.0 オーダーから 1e-4 オーダーへ、3〜4 桁下げられます。下がらないモデルはほぼ確実にバグを抱えています。3〜4 桁という落差は「与えられた少数サンプルを暗記する容量があるか」を測る明確な閾値になっており、桁が落ちない場合は最適化経路に欠陥がある強い証拠です。
- 計算コストの観点では明確な利点があります。実データでの 1 サンプル準備は、複数カメラの動画デコードを伴うと 1 サンプルあたり数十秒かかることが報告されています(7 カメラ動画を毎フレームデコードすると 1 サンプルおよそ 35 秒という実測例があります)。一方、合成バッチは乱数生成だけなのでミリ秒オーダーで作れます。つまりデバッグの反復速度が数千倍に上がる、というのが定量的なメリットです。バグ修正のたびに 35 秒待つか 1 ミリ秒で済むかは、1 日に試せる仮説の数を桁で変えます。
- ピーク VRAM の実測にもそのまま流用できます。たとえば複数カメラ + BEV(Bird's-Eye View, 真上視点)融合のモデルで、解像度 450×300 の BEV グリッド・バッチ 4 のとき約 20GB、未来状態予測モジュールを足すと約 23GB、バッチ 8 では OOM、というような構成ごとの上限を、実データを用意せずに測れます。これは「どのバッチサイズ・どの解像度なら本番学習が回るか」を事前に知るための定量データになります。

アブレーション的な視点で言えば、「この検証を省くとどうなるか」が逆向きの実証です。検証を飛ばして実データ + 複合損失を一気に投入したプロジェクトでは、shape mismatch や勾配未伝播といった初歩的バグを、長時間の実データ学習が失敗してから発見することになり、開発が大きく停滞します。長い学習を回してから「実は軌道デコーダに勾配が一度も流れていなかった」と気づく事故は、合成検証 1 回で防げたはずの典型例です。

## メリット・トレードオフ・限界

メリット

- 原因の切り分けが容易。データ・モデル・損失を同時にデバッグせず、配線と勾配だけを先に確認できる。
- 高速・低コスト。実データのダウンロードや動画デコードが不要で、乱数生成だけで回る。反復速度が桁違いに速い。
- VRAM やバッチサイズの実測にも流用できる。構成ごとのメモリ上限を実データなしで把握できる。

トレードオフ・限界

- loss の数値に意味がないため「賢くなったか」は一切分からない。確認できるのは配線の健全性まで。
- 合成データで通っても、実データ特有の問題(欠損値・スケールの不一致・正規化忘れ・数値不安定・ラベルノイズ)は別途出る。特に BEV のように複数カメラを幾何投影する経路では、ランダム入力だと幾何が破綻して loss が `NaN` になることがあり、これはコードの欠陥ではなく入力が非現実的なことに起因します。実データでの数値安定性は別途検証が必要です。
- 「動いた」ことに安心して実データ検証を省くと、本質的な不具合(分布のずれ、ラベル品質、ドメインギャップ)を見逃します。合成軌道検証はあくまで「最初の関門」であって「合格証」ではありません。

未解決の課題としては、合成データの分布をどこまで現実に近づけるべきかという問いがあります。完全な乱数では数値不安定を誘発しやすく、かといって現実的な分布を作り込むと検証の手軽さが失われます。現実とのギャップを段階的に埋める(まず乱数 → 次に正規化済み乱数 → 次に少量の実データ 1 バッチ overfit)という多段の sanity check 設計が、実務的な落としどころになっています。また、複数 GPU・複数ノードの分散学習では「単一プロセスでは通るが分散すると勾配同期で詰まる」種類のバグがあり、合成データでも分散構成のまま検証する必要が出てきます。

## 発展トピック・研究の最前線

- メタモルフィックテスト: 「入力をこう変えたら出力はこう変わるはず」という不変条件でモデルを検証する手法。たとえば「軌道入力を時間反転すれば出力も対応して変わるはず」といった性質を使う。合成データ検証を体系化した発展形で、自動運転モデルの安全性検証にも応用が進んでいます。
- 勾配の健全性診断: 勾配ノルムの層ごとの分布を可視化し、勾配消失(vanishing gradient)や勾配爆発(exploding gradient)を早期に検出する手法。合成バッチと組み合わせると、どの層で勾配が死んでいるかを特定できます。
- 微分可能シミュレーション: 合成データを乱数ではなく微分可能な物理シミュレータから生成し、配線確認だけでなく、シミュレータを通した end-to-end 学習の検証まで一気に行う研究方向。閉ループ評価へと地続きです。
- 自動化された sanity check: 学習フレームワークに「overfit テストを 1 コマンドで回す」機能を組み込み、CI(継続的インテグレーション)で配線退行(regression)を毎回検出する流れ。モデル改変のたびに人手で確認する負担を減らします。
