# DataLoader 実装（データ供給パイプライン）

## ひとことで言うと

DataLoader(データローダ)は、ディスク上に保存されたデータを「学習に使える形のかたまり(バッチ)」へ整え、モデルへ途切れなく供給する部品です。学習の主役は GPU での計算ですが、その GPU にデータが届かなければ計算は始まりません。DataLoader の役割は、読み込み・前処理・バッチ化を並列かつ高速に行い、GPU を待たせないことです。深層学習の性能は「モデルの賢さ」だけでなく「データを速く流せるか」にも大きく左右され、ここが詰まると高価な GPU が遊んでしまいます。

## 直感的な理解

工場のラインを想像してください。中央に超高性能の組み立てロボット(GPU)があり、部品(データ)が流れてくれば猛烈な速さで製品(学習結果)を作ります。ところが部品の供給が遅いと、ロボットは手を止めて待つしかありません。どれだけロボットが速くても、ライン全体の速度は「いちばん遅い工程(ボトルネック)」で決まってしまうのです。

深層学習でも同じことが起きます。1ステップの流れは「ディスクからデータを読む → CPU で前処理する(リサイズ・正規化など) → GPU へ転送 → GPU が計算する」です。もし読み込みと前処理が GPU 計算より遅ければ、GPU は毎ステップ待たされます。この現象を GPU starvation(GPU の飢餓)と呼びます。学習が「遅い」とき、犯人はモデルではなくデータパイプラインであることが驚くほど多いのです。プロファイラ(nvidia-smi で GPU 利用率を見る、PyTorch Profiler でステップ内訳を見る)を取ると、GPU 利用率が 100% から大きく下がっている=供給が追いついていない、と分かります。

DataLoader はこれを防ぐために、(1) データの読み込みと前処理を複数の CPU プロセス(ワーカー)で並列化し、(2) GPU が現在のバッチを計算している間に次のバッチを先読み(prefetch)します。つまり「ロボットが今の部品を組み立てている間に、別の作業員たちが次の部品を準備しておく」わけです。さらに大規模学習では、データそのものの読み方(無数の小ファイルをどう速く読むか)が支配的になり、ファイルの並べ方・まとめ方まで設計対象になります。

## 基礎: 前提となる概念

- Dataset(データセットクラス): 「i 番目のサンプルはこれ」という対応を定義する抽象。`__len__()` で総数を、`__getitem__(idx)` で 1 サンプルを返す map-style と、順次イテレートする `__iter__()` を実装する iterable-style がある。1 サンプルを読み、モデルが食べられるテンソルに変換するロジックをここに書く。
- 前処理(preprocessing): 生データをモデル入力に整える操作。画像なら resize(サイズ統一)、normalize(正規化、画素値を学習しやすい範囲・統計に整える。ImageNet 学習済みモデルなら平均 [0.485, 0.456, 0.406]・標準偏差 [0.229, 0.224, 0.225] で各チャンネルを引き割りするのが定番)、テンソル化(NumPy/PIL → torch.Tensor)など。
- バッチ(batch): 複数サンプルを一つのテンソルにまとめたもの。`(B, ...)` の B がバッチサイズ。GPU は並列計算(SIMD/SIMT)が得意なので、1サンプルずつより大きなバッチで処理する方がはるかに効率的。バッチ化は collate 関数が担う。
- ワーカー(worker): データ読み込み・前処理を担う並列プロセス。`num_workers=4` なら4プロセスが同時に次のバッチを準備する。プロセスゆえ GIL(Python のグローバルインタプリタロック、スレッド並列を阻む仕組み)を回避できる。
- シャッフル(shuffle): データの順序をランダムに混ぜること。同じ順序で学習し続けると勾配に偏りが出て(系統的バイアス)学習が不安定・収束が悪くなるため、毎エポック混ぜる。
- I/O(input/output): ディスクやネットワークからのデータ入出力。大量の小ファイルへのランダムアクセスは、ファイルシステムの inode 検索やシーク(読み取りヘッドの移動)のオーバーヘッドで非常に遅くなる。

## 仕組みを詳しく

実装には目的の異なる二系統がよく現れます。「実データを直接読む(開発・小規模向け)」と「事前に展開したデータを高速に読む(大規模向け)」です。

系統1: 実データ直読み

PyTorch の `Dataset` を継承し、`__getitem__` で生データから1サンプルを組み立てます。動画ベースの運転データセットでの典型的な流れ。

1. サンプルインデックスから (エピソード, 行) を引く。前後に十分なフレーム余裕がある有効点だけを登録しておく([逐次フレームウィンドウ（過去・未来）アクセス](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0837_sequential-frame-windows) 参照)。
2. 車両状態の数値列から運動履歴 `(256,)` と未来軌道の教師 `(128,)` を計算する。これは動画をデコードしない軽量パス。
3. 現在フレームのカメラ画像だけをデコードし、各画像を所定サイズへ resize、ImageNet 統計で normalize し、`(7, 3, 256, 256)`(7視点の例)へ stack する。

この系統の長所は実データをそのまま扱えること。短所は動画デコードが重いことです。素朴に毎サンプルで動画全体をデコードすると1サンプル数十秒かかることもあり、「現在フレームのみデコード・数値は別読み」といった最適化が要ります。それでも大規模学習では再デコードのコストが効いてきます。動画コンテナ(MP4 など)は時間圧縮(フレーム間予測、I/P/B フレーム)を使うため、任意フレームを開くには直前のキーフレームまで遡ってデコードする必要があり、ランダムアクセスが本質的に高コストなのです。

系統2: 事前抽出 + シャードストリーミング(WebDataset)

大規模学習では、あらかじめフレームを JPEG に、運動信号を `.npy` に展開し、それらを `.tar` アーカイブ(WebDataset の「シャード」=かたまり)へまとめておきます。学習時は動画デコードも重い依存ライブラリも不要で、ディスク/ネットワーク帯域いっぱいで tar を順次(sequential)読みます。典型的なローダ構築は次のような形です。

```python
dataset = wds.WebDataset(urls, shardshuffle=True, nodesplitter=wds.split_by_worker)
if shuffle > 0:
    dataset = dataset.shuffle(shuffle)        # 例: バッファ1000件でストリーミングシャッフル
dataset = dataset.map(decode_sample)          # 1サンプルをテンソルへ変換
loader  = wds.WebLoader(dataset, batch_size=B, num_workers=N)
```

`decode_sample` の中身(7視点の例):

1. `cam_0.jpg` 〜 `cam_6.jpg` を全部開き、transform(テンソル化 + ImageNet 正規化)して `torch.stack` で `(7, 3, H, W)` に。
2. `ego.npy` を読み、たとえば 384 次元を `256 + 128` に分割する。先頭 256 を運動履歴 `egomotion_history`、残り 128 を未来教師 `trajectory_target` に割り当てる。
3. まだ埋めない要素(過去の視覚特徴など)はゼロのプレースホルダにしておく。

両系統で返すバッチの形を揃えるのが設計の肝です(例):

```
visual_tiles      (B, 7, 3, 256, 256)
egomotion_history (B, 256)
trajectory_target (B, 128)
```

形を揃えれば、どのデータセット・どの読み込み方式でも、学習ループ側(forward → 損失 → backward → 最適化ステップ)を無改変で使い回せます。これは「データの抽象化境界」を引く設計で、ソフトウェア工学でいう依存性逆転(具体実装でなくインターフェースに依存させる)と同じ発想です。

`nodesplitter=wds.split_by_worker` は、複数ワーカーがシャードを重複なく分担するための指定です(これを忘れると各ワーカーが全シャードを読み、同じデータが何度も流れて学習が偏る)。`shuffle(1000)` は1000件のバッファ内でランダムに入れ替えるストリーミング・シャッフルで、データ全体をメモリに載せずに順序の偏りを減らします(完全ランダムではない近似)。シャード単位のシャッフル(`shardshuffle`)とサンプル単位のバッファシャッフルを二段で組み合わせるのが定石で、シャードを小さく切り分散させておくほど近似が良くなります。

before/after で効果を整理します。

```
before: モデル入力は torch.randn の乱数。学習しても賢くならない(配線確認にしかならない)。
after : 実カメラ画像と実運転軌道がバッチで流れ、forward → 損失 → backward → step が意味を持つ。
        大規模時は事前抽出パスで GPU を待たせず回る。
```

クラウド学習では、S3 等のオブジェクトストレージに置いたシャードを、計算ノード起動時にローカルディスク(NVMe SSD など)へ同期してから読む運用も一般的です。これによりネットワーク往復のレイテンシを避け、ローカル帯域で読めます。あるいは WebDataset の URL 機構で S3 から直接ストリームし、ノードのローカルディスク容量を超える規模を扱う運用もあります。

## 手法の系譜と主要論文

- PyTorch(Paszke et al., "PyTorch: An Imperative Style, High-Performance Deep Learning Library", NeurIPS 2019、arXiv:1912.01703)。`Dataset` / `DataLoader` という抽象を標準化しました。`__getitem__` で1サンプルを返し、複数ワーカーで先読みするという分業を、ユーザーが意識せず使える形にしたのが貢献です。`pin_memory`(ページ固定メモリ=OS にスワップさせない領域に置き、CPU→GPU 転送を DMA で高速・非同期化)や `prefetch_factor`(各ワーカーが何バッチ先読みするか)、`persistent_workers`(エポックごとにワーカーを再生成せず使い回す)といった調整つまみも提供します。

- WebDataset(Aizman, Maltby, Breuel, "High-Performance I/O For Large-Scale Deep Learning", IEEE Big Data 2019)。無数の小ファイルを `.tar` にまとめて順次読みする方式を提案。ランダムアクセスの遅さとファイルシステムのオーバーヘッドを避け、ローカルディスク・クラウドストレージのどちらでもシーケンシャル帯域を活かせます。動機は ImageNet 規模・それ以上のデータでの I/O ボトルネック解消で、トレードオフは「順次読みゆえ完全ランダムシャッフルができず、バッファ内シャッフルで近似する」点です。POSIX ファイルシステムを介さずオブジェクトストレージから直接ストリームできる点が、クラウド大規模学習で重宝されます。

- tf.data(Murray et al., "tf.data: A Machine Learning Data Processing Framework", VLDB 2021)。TensorFlow のデータパイプライン。`prefetch`・`map`・`interleave` を宣言的に組み、入出力と前処理と計算をオーバーラップさせる設計思想を体系化しました。AUTOTUNE(並列度を実行時に自動調整)という仕組みで、ワーカー数の手動チューニングを自動化しようとした点が示唆に富みます。

- FFCV(Leclerc et al., "FFCV: Accelerating Training by Removing Data Bottlenecks", CVPR 2023)。独自バイナリ形式(`.beton`)で前処理を極限まで軽量化し、JIT コンパイルした前処理パイプライン、メモリマップ、キャッシュ、GPU 側デコードを組み合わせて、データ供給を計算に追従させます。

- 関連する高速 I/O の系譜として、NVIDIA DALI(GPU 上で JPEG デコード・リサイズ・正規化を行い CPU ボトルネックを外す)、TFRecord/Parquet(列指向・シャード形式)、MosaicML Streaming(クラウドから直接ストリームしつつシャッフル品質を保つ)などがあります。いずれも「計算は速いのにデータ供給が追いつかない」という共通課題への異なる回答です。

## 論文の実験結果(定量データ)

データローダは「モデルの精度」ではなく「学習スループット(単位時間あたり処理サンプル数, samples/sec)」と「GPU 利用率」で評価されます。

- WebDataset の報告では、シーケンシャル読みにより、ローカルディスクでもネットワークストレージでも、小ファイルのランダムアクセスに比べて I/O スループットが大幅に向上し、複数 GPU・複数ノードへ線形に近くスケールすることが示されています。鍵は「ディスクのシーク回数を減らす」点で、HDD/ネットワークではランダムアクセスのシークが支配的だったのを、連続読みで解消します。回転ディスクではシーク1回が数ミリ秒かかり、小ファイル1万個のランダム読みはシークだけで数十秒に達しうるため、連続読みの効果は桁違いです。

- FFCV(CVPR 2023)の報告では、ImageNet の ResNet-50 学習を、最適化したデータパイプラインにより、複数 GPU を備えた単一マシンで従来比で大幅に短縮(報告では条件により1時間を切る規模)できたとされ、「同じモデル・同じ精度でも、データ供給を速くするだけで学習時間が何倍も変わる」ことを定量的に示しました。これはモデル改良と同等以上にパイプライン最適化が効くという研究上の教訓です。

- 一般に、`num_workers` を増やすと CPU が前処理を並列化でき、ある点まではスループットが向上しますが、物理 CPU コア数を超えると頭打ち・かえって悪化します(コンテキストスイッチやメモリ帯域の競合、プロセス間通信のオーバーヘッド)。`pin_memory=True` は CPU→GPU 転送を非同期化して数%〜十数%の改善が出ることがあります。これらは経験的チューニングの対象で、最適値はハードウェア(コア数・ディスク種別・GPU 速度)とデータ(画像サイズ・前処理の重さ)に依存します。

アブレーション的に見ると、「動画を毎サンプル全デコードする」素朴実装と「現在フレームのみデコード + 数値は別読み」最適化では、1サンプルの読み込み時間が数十秒から数十〜数百ミリ秒へ桁で改善することがあり、これが学習可能/不可能を分けます。さらに事前抽出に切り替えると、デコード自体が学習ループから消えるため、ボトルネックは純粋に I/O 帯域へ移ります。ここまで来ると、ストレージ帯域(NVMe で数 GB/s、ネットワークで数百 MB/s〜)が上限になり、画像サイズの縮小や JPEG 品質の調整がスループットの最後のつまみになります。

## メリット・トレードオフ・限界

メリット

- 出力形式を統一すれば、複数データセット・複数読み込み方式を学習コード無改変で切り替えられる。
- 事前抽出 + シャードストリーミングで、動画デコードと重い依存を学習ループから排除し、大規模学習で GPU を待たせない。
- 直読み系統は実データをそのまま扱え、開発・検証を手早く回せる。
- 有効サンプルのみ登録・ワーカー分担・バッファシャッフルで、データ漏れや順序偏りを抑える。

トレードオフ・限界

- 直読み系統は動画デコードが重く、最適化しても大規模には不向き。
- 事前抽出系統は「展開 → シャード化 → アップロード」の事前コストとストレージを要し、前処理を変えるたびに再生成が必要(データが固定スナップショットになる=オンザフライなデータ拡張の自由度が下がる)。
- シーケンシャル読みは完全ランダムシャッフルができず、バッファサイズで近似する。バッファが小さいと相関の強い隣接サンプルが固まりやすく、勾配のノイズが偏る。
- 過去視覚特徴などをゼロ埋めのままにすると、世界モデル系の損失には情報不足になる(別途ウィンドウ拡張で補完)。
- ワーカー数・prefetch・ストレージ帯域はハードウェア依存で、最適値は実測チューニングが必要。再現性のため乱数シードをワーカーごとに設定する配慮も要る(`worker_init_fn`)。
- iterable-style では `__len__` が定まらないことがあり、エポック境界や分散学習の同期(全ワーカーが同数バッチを出す)に注意が要る。

## 発展トピック・研究の最前線

- GPU 側前処理(NVIDIA DALI など)。JPEG デコード・リサイズ・正規化を GPU で行い、CPU ボトルネックを根本から外す。CPU が貧弱で GPU が強いクラウドインスタンスで効果が大きい。
- ストリーミング学習。データセット全体をローカルに置けない超大規模で、クラウドから直接ストリームしつつシャッフル品質と決定性(中断・再開時の再現性)を保つ手法(MosaicML Streaming など)。
- データ重複排除・カリキュラム学習。同じデータを何度も見るより、難しい/有益なサンプルを優先供給するデータ中心(data-centric)の最適化。重要度サンプリングや損失に応じた動的選択。
- 入出力と計算のオーバーラップ最適化。`prefetch_factor`、`persistent_workers`、`pin_memory`、CUDA ストリームによる非同期転送の組み合わせで GPU 利用率を 100% に近づけるプロファイリング駆動の調整。
- 分散データ並列との統合。複数ノードでシャードを重複なく分配し(`DistributedSampler` 相当)、各ランクが偏りなく学習できるようにするシャーディング設計。
