# Causal Reasoning Head (System 2 補助ヘッド)

## ひとことで言うと

これは、運転モデルが作ったシーン要約ベクトル(文脈ベクトル、context vector)を受け取り、(1)「今このシーンを支配している原因は何か」を少数のクラスに分類しつつ、(2) その推論を凝縮した連続ベクトル(reasoning latent、推論潜在表現)を軌道プランナへ渡す、軽量な補助ニューラルネットワークの設計パターンです。人間の思考のうち「速い直感(System 1)」ではなく「ゆっくり熟考(System 2)」に相当する部分を、巨大な推論器を毎フレーム回さずに学習で注入する狙いがあります。本体の運転モデルの処理を壊さない「任意・追加(optional)」のヘッドとして据えるのが定石です。

## 直感的な理解

人間の運転を観察すると、ほとんどの操作は無意識の反射です。前の車が止まればブレーキを踏む、車線が曲がればハンドルを切る。いちいち言語で考えていません。これが System 1(速い・直感的・無意識)です。一方、めったに起きない難しい場面では、人は一瞬立ち止まって「なぜ今これが危ないのか」を意識的に推し量ります。たとえば「信号が点滅していて、歩行者が縁石でためらっている。これは飛び出すかもしれない」。これが System 2(遅い・熟考・意識的)です。

運転 AI の多くは System 1 だけでできています。画像を入れたら即座に軌道を吐く反射機械です。普段はそれで十分ですが、「めったに起きないが重大な場面(ロングテール)」では、もう一段の理由づけがほしくなります。理想は大きな視覚言語モデル(VLM)を常に動かして「なぜ危ないか」を考えさせることですが、車載のリアルタイム処理にはコストが高すぎます。VLM の推論は1場面あたり数百ミリ秒から秒単位かかることもあり、10Hz 以上で制御を回したい車載系には乗りません。

そこでの発想が「重い推論器を毎回回す代わりに、その推論結果を真似する小さなヘッドを 1 つ抱える」ことです。事前に VLM などに難場面を見せて「支配的な原因は何?」を答えさせ、その答えを教師(疑似ラベル)として軽量ヘッドを訓練しておく。すると本番では、その小さなヘッドが「原因らしきもの」を即座に推定し、その推論を凝縮したベクトルをプランナに添えられます。重い思考を一度だけ行い、その成果を軽い形に蒸留して持ち歩く、というアイデアです。試験前に賢い友人(VLM)に難問の解き方を教わって要点をノートに写し、本番では自分のノート(軽量ヘッド)だけで即答する、というイメージに近いです。

## 基礎: 前提となる概念

いくつか用語を噛み砕きます。

- 文脈ベクトル(context vector): シーン全体を 1 本のベクトルに要約した表現です。たとえば BEV 特徴や時系列特徴を集約して得た 256 次元のベクトルで、「今この状況がどういう状況か」を数値で表したものと思ってください。プランナはこの文脈ベクトルを起点に軌道を決めます。

- ロジット(logits): 分類モデルが出す「softmax をかける前の生のスコア」です。各クラスに対する未正規化の点数で、softmax を通すと確率になります。softmax は各スコアを指数関数にかけて総和で割る操作で、大きいスコアほど高い確率になります。

- 微分可能・勾配の逆伝播(backpropagation): ニューラルネットワークは、出力の誤差(損失)を入力側へ「勾配(損失を減らすにはパラメータをどちらへ動かすか)」として伝えて学習します。途中の部品をすべて微分可能につないでおくと、後ろの誤差が前の部品まで届き、前の部品も一緒に学習されます。

- 知識蒸留(knowledge distillation): 大きな教師モデルの出力を、小さな生徒モデルに学ばせて、教師の知識を軽量モデルへ移す技術です(Hinton et al. 2015)。この設計は「VLM=教師、補助ヘッド=生徒」の蒸留とみなせます。

- 補助タスク/マルチタスク学習(multitask learning): 主タスク(運転)に関連する別タスク(原因分類)を同時に学習させると、共有する内部表現が良くなり主タスクの汎化も上がる、という枠組み(Caruana 1997)です。

これらが分かると、この種のヘッドは「VLM の因果判断を蒸留した小さな分類器であり、かつ補助タスクとして本体表現を磨くために勾配を共有部分まで流す」装置だと理解できます。

## 仕組みを詳しく

研究コミュニティで使われる典型的な実装は次の要素から成ります。

### 少数の因果クラス

支配的な原因要因を、扱いやすい少数のクラスに離散化します。運転では交差点・歩行者・信号・障害物・平穏(特に注意要因なし)といった 5 クラス程度がよく用いられます。

```
CAUSAL_CLASSES = (
    "intersection",   # 交差点
    "pedestrian",     # 歩行者
    "traffic_light",  # 信号
    "obstacle",       # 障害物
    "clear",          # 何もない(平穏)
)
```

クラスの順序はラベル番号(0〜4)と対応するため、一度決めたら固定するのが鉄則です(並べ替えると過去の学習済み重みと整合しなくなる)。少数クラスにする理由は、教師となる VLM の出力を安定して所定の語彙に縛れること、そして稀少クラスでも各クラスにある程度のサンプルが集まることです。

### モジュール本体

入力はシーンの文脈ベクトル(たとえばプランナ内部の自車隠れ状態、典型的に 256 次元)です。処理は概ね 2 段です。

1. 推論トランク(reasoning trunk): 線形層 → 活性化(GELU など、なめらかな非線形関数) → 線形層 → 正規化(LayerNorm)という小さな多層パーセプトロン(MLP)。次元の流れは [B, 256] → [B, 512] → [B, latent_dim] のように一旦広げて絞ります。中間で次元を広げるのは、非線形変換に「考える余地」を与えるためで、Transformer の FFN と同じ発想です。出力が reasoning latent。latent_dim を本体の埋め込み次元(例 256)に合わせておくと、プランナへ「足す/連結する」際に追加の射影層が要らず実装が簡潔になります。

2. 決定ヘッド(decision head): reasoning latent に線形層をかけて 5 クラスのロジットを出す。[B, latent_dim] → [B, 5]。

forward は (reasoning_latent, decision_logits) を返します。形状は [B, 256] と [B, 5]。重要なのは2つの出力が別物だという点で、reasoning latent は「プランナに渡す連続的な推論の中身」、decision_logits は「学習信号を受け取るための分類の窓口」です。後者があるおかげで前者に意味が宿ります。

### カスケード設計(勾配を共有表現まで流す)

最も重要なのは、reasoning latent を取り出す経路に勾配遮断(detach のような操作)を入れず、完全に微分可能にしておくことです。これを「カスケード設計(cascade design)」と呼びます。狙いは、原因分類の補助損失から流れる勾配が、文脈ベクトルを作った上流の共有トランクまで逆伝播することです。

```
[共有バックボーン/トランク] → context vector → [reasoning trunk] → latent → [decision head] → logits
        ↑___________________________ 勾配が遡る ___________________________|
```

これにより「原因を当てる」という補助タスクの学習が、運転モデル本体の特徴表現も「シーンの意味を捉える方向」に形作ります。これはマルチタスク学習の中心的な仮説そのものです。もし勾配を遮断すると、ヘッドは原因当てが上手になるだけで本体表現には何の影響も与えず、補助タスクとしての意義が失われます。逆に流しすぎる(重みを上げすぎる)と本体が原因分類に引っ張られて運転が劣化する、という綱引きが存在します(後述)。

### 構造化された出力

生のロジットをそのまま渡すのではなく、来歴と確信度を添えた構造体に包んで渡す設計がよく採られます。

```
@dataclass
class CausalReasoningContext:
    reasoning_latent: torch.Tensor      # [B, latent_dim]
    causal_class_logits: torch.Tensor   # [B, num_classes]
    confidence: torch.Tensor            # [B]  予測の確信度
    provenance: str                     # 例 "vlm_causal_head"  この情報の出所
```

confidence は softmax(logits).max()、つまり「最も自信のあるクラスの確率」です。これが低いとき(どのクラスも確率が割れているとき)、プランナはこの推論を割り引いて扱える、という運用ができます。provenance(由来)を持たせると、プランナ側は「この推論は VLM 由来の因果ヘッドが出したものだ」と来歴を区別でき、信頼度に応じて使い分けられます。このような構造体は、地図情報やルート指示なども将来差し込める「標準化された情報バス(SceneContext のような上位コンテナ)」に詰めて渡すことが多く、プランナを単一の固定ヘッドに縛らない拡張余地を残します。

### 学習信号

学習はクロスエントロピー(分類の標準損失)で行い、教師ラベルは VLM に難場面を見せて得た疑似ラベルです。実務上 2 つの配慮が入ります。ノイズの多い疑似ラベルに過度に従わないための label_smoothing(ラベル平滑化)、そしてクラス不均衡(平穏ばかり多い等)に対処する class_weights(クラス重み)です。詳細は [Causal Consistency Loss](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0553_causal-consistency-loss) と [VLM による疑似ラベリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0523_vlm-pseudo-labeling) を参照してください。この補助損失は主損失(模倣損失)に対して小さい重みで足すのが鉄則です。

## 手法の系譜と主要論文

- Kahneman, "Thinking, Fast and Slow"(2011)。人間の思考を System 1(速い直感)/System 2(遅い熟考)に分ける二重過程理論(dual-process theory)の枠組み。この種のヘッドの「System 2 を軽量に注入」という発想と命名の源泉。運転の文脈では「反射的な制御」と「熟慮的な状況判断」の分離として持ち込まれます。

- Hinton, Vinyals & Dean, "Distilling the Knowledge in a Neural Network"(NeurIPS Workshop 2015, arXiv:1503.02531)。大きな教師モデルの出力(soft target)を小さな生徒に学ばせる。このヘッドを「VLM 教師 → 軽量ヘッド生徒」の蒸留として位置づける理論的背景。

- Caruana, "Multitask Learning"(Machine Learning, 1997)。関連補助タスクを同時学習すると共有表現が改善し主タスクの汎化が上がる。カスケード設計の正当化。

- Xu et al., "Explainable Object-Induced Action Decision for Autonomous Vehicles"(CVPR 2020)。運転の行動決定と「その理由」を同時に学習し、説明性と精度を両立する先駆け。原因分類という補助信号の運転への適用例で、BDD-OIA というデータセットを整備した点でも重要。

- 近年の自動運転 VLM 研究: Tian et al., "DriveVLM"(CoRL 2024, arXiv:2402.12289)、Xu et al., "DriveGPT4"(RA-L 2024)など。大規模 VLM でシーンを高次に推論し計画を補助する流れ。このヘッドはこの重い推論を「疑似ラベル蒸留」でコスト回避する軽量版にあたる。

系譜としては、「人間の二重過程理論(Kahneman)」という発想を出発点に、「行動+理由の同時学習(Xu 2020)」「VLM による高次推論(DriveVLM 2024)」と発展し、それを車載コストに収めるための「蒸留(Hinton 2015)+補助タスク(Caruana 1997)」という古典技術で実装する、という合流点にこの設計が位置します。

## 論文の実験結果(定量データ)

このヘッドそのものは設計パターンであり単一論文ではないため、関連研究の定量結果でその有効性を裏づけます。

- 補助タスクの効果: Xu et al.(CVPR 2020)は BDD-OIA データセットで、行動決定(進む・止まる・左折可・右折可などの分類)に「その理由」を予測するタスクを併学習すると、行動分類の F1 スコア(適合率と再現率の調和平均、高いほど良い)が単独学習比で向上することを報告。理由づけ補助信号が運転判断の精度自体を底上げしうることを示しました。F1 を使うのは、行動クラスが不均衡(止まる場面は少ない等)で、単純な正解率では少数クラスの見逃しが見えないためです。

- 蒸留の効果: Hinton et al.(2015)は MNIST や音声認識で、教師の soft target を使うと小モデルのテスト誤りが減ることを示しました。たとえば大モデル(アンサンブル)から蒸留した小モデルが、ラベルのみで学習した同サイズモデルよりテスト誤り数を顕著に減らした例があります。これは「重い教師の知識を軽量モデルに移せる」というこのヘッドの前提を支えます。

- VLM 推論注入の効果: DriveVLM(CoRL 2024)は nuScenes 計画タスクで、言語推論を計画前段に挟むことで難場面の L2 誤差・衝突率を改善し、特にロングテール・コーナーケースで効くと報告。ただし VLM の推論遅延が大きく、論文自身が「軽量化(DriveVLM-Dual で高速 end-to-end 経路と併用)」を提案しており、これがこのヘッドの「重い推論を毎フレーム回さない」動機と一致します。

- アブレーションの観点で重要なのは「補助損失の重み」です。マルチタスク学習一般の知見として、補助損失の重みを上げすぎると主タスク性能が下がる(タスク衝突)ことが繰り返し報告されており(GradNorm, Chen et al. ICML 2018 など)、原因分類補助も「低重みで足す」ことが性能維持の条件になります。重みを 0 にすれば純粋な反射モデルに戻り、適切な小さい値で表現が磨かれる、という挙動が期待されます。実務では λ を 0、0.05、0.1、0.2 と振って計画の L2・衝突率が劣化しない範囲を探す、というアブレーションが典型です。

## メリット・トレードオフ・限界

メリット:
- System 2 的な因果推論を軽量に注入でき、重い VLM を毎フレーム回さずに済む(VLM は疑似ラベル生成に一度使うだけ)。推論コストはほぼ MLP 1個分。
- 任意・非破壊設計にすれば、本体の運転モデルを壊さず付け外しできる。ヘッドを外しても運転は動く。
- カスケード設計により、補助タスクの学習が共有表現を改善しうる(本体の汎化に寄与)。
- confidence と provenance を持つ構造化出力なら、プランナ側が信頼度に応じて使い分けでき、来歴も追える。デバッグ時に「なぜこの軌道を出したか」の手がかりになる。

トレードオフ・限界:
- 教師ラベルが VLM 由来の疑似ラベルでノイズを含む。誤った因果判断を学んでしまう恐れ(label_smoothing で緩和するが完全ではない)。
- 原因要因を少数クラスに離散化しており、複数要因が同時に効く場面(交差点で歩行者かつ信号)や、定義済みクラスに収まらない原因を取りこぼす。
- 補助損失の重み調整を誤ると、本体の模倣学習を阻害する(タスク衝突)。低重みでの慎重なバランスが必要。
- 「因果」と名乗っても本質は分類であり、真の介入的因果推論(do 演算や反実仮想)ではない。「原因のラベルを当てる」のと「原因を介入的に同定する」のは別物である点に研究上の注意が要る。分類器が学ぶのは相関であって、介入のもとでの不変性ではない。
- reasoning latent がプランナに渡っても、それが実際に計画を変えている保証はない。プランナが latent を無視する「ショートカット学習」が起きうるため、latent の寄与を別途検証する必要がある。

## 発展トピック・研究の最前線

この方向の発展はいくつかあります。第一に、離散クラスでなく自由形式の言語推論(chain-of-thought、段階的に理由を言語化する推論)を中間に持たせ、その潜在表現を計画に渡す手法(DriveVLM, LMDrive 系)。これは表現力が高い一方、学習も評価も難しくなります。第二に、疑似ラベルのノイズに対する頑健化で、不確実性推定や複数 VLM のアンサンブル、自己学習(self-training)による反復精製。第三に、因果性をより厳密に扱う方向で、介入や反実仮想を取り入れた causal representation learning を運転に持ち込む試み。第四に、reasoning latent が本当に計画に効いているかを測る介入実験(latent を入れ替えると軌道が変わるか)で、ショートカット学習を検出する方法。いずれも「速い反射と遅い推論をどう統合するか(System 1/System 2 統合)」という大きな問いに収束しており、このヘッドはその最小実装的な位置づけにあります。
