# フレーム事前抽出とキャッシュ

## ひとことで言うと
深層学習の学習中に、動画ファイルから必要な画像フレームを取り出す処理 (デコードと呼びます。圧縮された動画データを1枚1枚の画像に戻す計算のことです) は、想像以上に重い処理です。学習の前にあらかじめ必要なフレームを取り出して扱いやすい形で保存しておき (これを事前抽出と呼びます)、学習中はそれを読むだけにする、あるいは一度読んだものを覚えておいて再利用する (これをキャッシュと呼びます)。こうすることで、計算装置 (GPU) がデータの到着を待って遊んでしまう時間を減らし、学習全体を高速化できます。動画を大量に使う自動運転や World Model (世界モデル) の学習で特に効果が大きい技法です。

## 直感的な理解
料理にたとえると分かりやすいです。デコードは「冷凍された食材を解凍する」作業に似ています。注文が入るたびに毎回解凍していると、調理人 (GPU) は解凍が終わるまで何もできず手が空きます。事前抽出は「営業前に全部解凍しておく」こと、キャッシュは「一度解凍した食材を冷蔵庫に置いておき、また使うときにそのまま出す」ことです。

深層学習では、GPU は1秒間に何兆回もの演算ができる非常に高価な装置です。一方、動画から特定の時刻の画像を取り出す処理は CPU が担当し、しかも重い。GPU が「次の画像をくれ」と要求しても CPU のデコードが間に合わないと、GPU は計算するものがなく待機状態になります。これを「データストール (data stall、データ供給の停滞)」と呼びます。Mohan ら (VLDB 2021) の研究では、画像分類のような比較的軽いタスクですら、設定によっては学習時間の最大 65% がデータ供給待ちに費やされうると報告されています。動画を扱う場合はこの比率がさらに跳ね上がります。投資の観点で言えば、数百万円する GPU の半分以上が遊んでいるなら、その投資は半分しか回収できていないことになります。だからデータ供給の最適化は、モデルそのものを速くする工夫と同じくらい、いやしばしばそれ以上に費用対効果が高い作業です。

## 基礎: 前提となる概念

順番に専門用語をほぐしていきます。

動画の圧縮とデコード。MP4 (H.264/H.265 などの符号化を内部に持つコンテナ形式) のような動画ファイルは、ファイルサイズを小さくするため強く圧縮されています。圧縮の中心的な仕組みは「フレーム間予測」です。すべてのフレームを丸ごと保存するのではなく、ときどき完全な画像 (キーフレーム、I フレームと呼びます) を置き、その間のフレームは「直前または前後のフレームからの差分」だけを保存します (差分のみのフレームを P フレーム・B フレームと呼びます)。これにより容量は劇的に小さくなりますが、代償として「ある時刻のフレームだけを取り出す」のが難しくなります。中間のフレームを得るには、直前のキーフレームまで戻り、そこから差分を順に適用して再構成する必要があるからです。この「特定フレームに飛ぶ」操作を seek (シーク) と呼び、シークのたびにキーフレームからの再展開が走るため CPU を大量に消費します。キーフレーム同士の間隔 (GOP, Group of Pictures と呼ぶ) が広いほど圧縮率は高くなりますが、ランダムシークの代償は重くなります。これは圧縮効率とランダムアクセス性のトレードオフです。

エポックとデータローダ。学習はデータセット全体を何度も繰り返し舐めます。1周を 1 エポックと呼びます。各エポックで同じサンプルを再び読むため、毎回同じフレームをデコードし直すと、まったく同じ計算を何百回も繰り返すことになります。データを読み込んで GPU に供給する部品をデータローダ (data loader) と呼び、通常は複数の CPU プロセス (ワーカー) で並列にデコードしますが、フレーム数が多いと並列化しても追いつきません。データローダの並列度を上げても、CPU の物理コア数という天井があり、コア数を超えてワーカーを増やすと文脈切り替え (コンテキストスイッチ) のオーバーヘッドで逆に遅くなることもあります。

スループットとボトルネック。学習全体の速さは「最も遅い部分 (ボトルネック)」で決まります。GPU 演算が速くても、データ供給が遅ければ全体はデータ供給の速さに律速されます。理想は GPU 演算とデータ供給を時間的に重ね合わせる (オーバーラップさせる) ことで、これが完全に達成できれば「データ供給は GPU 計算の影に隠れて見えなくなる」状態になります。事前抽出とキャッシュは、このボトルネックをデータ供給側からどかし、オーバーラップを成立させるための技法です。

## 仕組みを詳しく

なぜ動画ベースの学習でデコードが特に重くなるのか、まず数で押さえます。

一般的な反応型 (reactive) の運転モデル、つまり「今この瞬間の画像を見て、今どう動くかを決める」モデルなら、1サンプルに必要なフレームは「現在時刻の各カメラ画像」だけです。たとえばカメラが 7 台なら 7 フレームです。

ところが未来予測を学習する World Model や、過去と未来の両方を窓 (window) として切り出す学習では話が変わります。1サンプルに次のフレームが必要になります。

```
現在フレーム : 7 カメラ × 1 時刻        =  7 フレーム
過去窓       : 7 カメラ × N 時刻 (N=4)  = 28 フレーム
未来窓       : 7 カメラ × N 時刻 (N=4)  = 28 フレーム
-----------------------------------------------------
合計                                    = 63 フレーム / サンプル
```

「カメラ台数 × (過去 N + 現在 + 未来 N) 時刻」のフレームを毎サンプル取り出す必要があり、反応型の数倍から十数倍のデコード量になります。これに「同じデータを 10 エポック舐める」が掛かると、

```
事前抽出なし : 63 フレーム × 10 エポック = 630 回のデコード / サンプル
事前抽出あり : 63 フレーム × 1 回 (学習前) =  63 回のデコード / サンプル
```

デコード回数が 1 桁減ります。これがそのまま GPU の待ち時間の削減になります。

事前抽出 (pre-extraction) の流れ。学習を始める前に、必要なフレームを動画から一度だけデコードし、すぐ読める形で保存します。保存形式にはいくつかの選択肢があり、それぞれに得手不得手があります。

- 個別の画像ファイル (PNG/JPEG) として書き出す。最も単純だが、何百万もの小さなファイルが生じ、ファイルシステムのメタデータ操作 (open/close、inode 探索) が新たなボトルネックになる。1ファイル開くのに数十マイクロ秒かかるとして、数百万ファイルを毎エポック開けばそれだけで分単位の時間を食う。
- テンソルを連続バイト列にまとめて保存する (memory-mapped array、メモリにファイルを直接対応づけてランダムアクセスする方式)。連続領域への一括書き込みでファイル数を抑えつつ、任意位置への高速ランダムアクセスが効く。
- tar アーカイブの shard (シャード、断片) にまとめる。後述の WebDataset 形式。クラウドストレージとの相性が良い。

保存するときに、リサイズや正規化といった前処理まで済ませてしまうこともできます。デコード済みかつ前処理済みのテンソルを保存しておけば、学習時は読み込んでそのまま GPU に渡すだけになります。ただし後述するように、これは前処理を固定してしまうという副作用を持ちます。

キャッシュ (caching) の流れ。事前抽出ほど大がかりにせず、「一度デコードしたフレームをメモリやローカルディスクに覚えておき、次に同じフレームが要求されたら再デコードせず再利用する」軽量な方法です。窓を使う学習では、隣り合うサンプル同士でフレームが重複します。

```
サンプル A (現在 row=100) の未来窓 : 110, 120, 130, 140
サンプル B (現在 row=110) の過去窓 :  80,  90, 100, 110
                                           重複 ↑↑↑↑ (100, 110 など)
```

一度読んだフレームをキャッシュしておけば、この重複デコードを省けます。キャッシュには上限 (容量) を設け、古いものから捨てる LRU (Least Recently Used、最も長く使われていないものを追い出す) などの戦略を組みます。ただしキャッシュのヒット率 (再利用できた割合) は、サンプルをどの順で取り出すか (アクセスパターン) に強く依存します。完全ランダムシャッフルではキャッシュが効きにくく、時間的に近いサンプルをまとめて取り出すとヒット率が上がる、というシャッフル品質との緊張関係があります。

階層性とどこにキャッシュするか。記憶装置には速度と容量のヒエラルキーがあります。速い順に GPU メモリ、CPU メモリ (RAM)、ローカル SSD、ネットワーク経由のオブジェクトストレージ (S3 など) で、速いものほど容量が小さく高価です。事前抽出データをどの階層に置くかで挙動が変わります。データセット全体が RAM に乗るなら、2 エポック目以降は OS のページキャッシュ (一度読んだファイル内容を OS が自動でメモリに保持する仕組み) が効いてディスク I/O が実質ゼロになります。乗り切らないなら、ローカル SSD に置いて RAM はホットなサンプルだけにする、といった階層設計が要ります。

設計上のきれいな分離点。フレームを取り出す関数 (たとえば `load_frame(動画, 時刻, カメラ)` のような抽象) を、窓を切り出すロジックから引数として注入する設計にしておくと、窓のロジックを一切変えずに「素直に動画からデコードする版」から「事前抽出済みデータを読む高速版」へ差し替えられます。これは関心の分離 (separation of concerns) と呼ばれる設計原則で、「正しく窓を切れること (correctness)」と「速く供給できること (performance)」を独立に開発・検証できる利点があります。研究プロトタイプではまず素朴なデコードで正しさを確認し、スケールさせる段階で供給側だけを最適化する、という進め方が定石です。

## 手法の系譜と主要論文

データ供給がボトルネックになる問題と、その解決策には独立した研究の系譜があります。

入力パイプラインのフレームワーク化。Murray ら (Google) の tf.data (VLDB 2021) は、データ前処理を宣言的なパイプラインとして記述し、プリフェッチ (prefetch、GPU が計算している裏で次のバッチを先読みする)、並列マッピング、キャッシュ、自動チューニング (AUTOTUNE、並列度やバッファ量を実行時に自動調整する) を提供しました。提案動機は、大規模学習で前処理が GPU に追いつかず GPU が遊ぶ問題を一般的な枠組みで解くことです。プリフェッチとキャッシュという二大概念をフレームワークの一級市民に据えた点が新規性です。

GPU 側でのデコード。NVIDIA DALI (Data Loading Library) は、JPEG デコードや動画デコード、リサイズといった前処理を GPU 上で実行し、CPU のボトルネックを解消する方向です。動機は「前処理を CPU に置く限り CPU コア数が天井になる」という観察。専用のハードウェアデコーダ (nvJPEG、NVDEC) を使うことで、CPU では届かない速度を出します。効果は学習スループットの向上ですが、トレードオフとして前処理にも GPU メモリと演算を食うため、本来モデルに使いたい資源と取り合いになります。

shard 形式によるストリーミング I/O。Aizman ら (NVIDIA) の WebDataset (IEEE BigData 2019) は、データを tar アーカイブの shard にまとめ、ストリーミングで順次読む形式を提案しました。動機は、何百万もの小さなファイルをランダムアクセスすると I/O が遅く、特にオブジェクトストレージ (S3 など) ではリクエスト単位の遅延 (1 リクエストあたり数十ミリ秒) が支配的になることです。まとめて順次読むことでスループットを最大化し、クラウド上の分散学習に向きます。トレードオフは、完全なランダムシャッフルがしにくいため、shuffle buffer (一定数を一時的に貯めてその中でシャッフルする) という近似が必要になる点です。

事前デコード済みフォーマット。Leclerc ら (MIT) の FFCV (CVPR 2023) は、データセットをあらかじめデコード済みかつメモリマップ可能な独自フォーマット (.beton) に変換し、学習時のデコードを排除する方向を突き詰めました。報告では、ImageNet 上の ResNet-50 学習で、一般的な PyTorch DataLoader 構成に比べて大幅な高速化を示し、適切なハードウェアでは ImageNet 1 エポックを数分のオーダーで回せると主張しています。これは事前抽出の発想そのものを製品化したものです。トレードオフは、前処理に時間がかかり、デコード済みデータが元の圧縮データより大きくディスクを食う点です。

データストールの体系的分析。Mohan ら (Microsoft Research) の "Analyzing and Mitigating Data Stalls in DNN Training" (VLDB 2021) は、複数のモデル・データセットでデータ供給待ちを定量化し、キャッシュ・並列デコード・MinIO のようなストレージ最適化でストールをどれだけ減らせるかを系統的に示しました。設定次第で学習時間の半分以上が供給待ちになりうること、キャッシュが効く局面と効かない局面 (データセットがメモリに乗るか否か) を切り分けた点が重要です。

ジョブをまたぐ共有。Kakaraparthy ら (HotCloud 2019) や、その発展である CoorDL、そして Graur ら (Google) の Cachew (USENIX ATC 2022) は、「同じデータセットで何本ものハイパーパラメータ探索ジョブが同時に走るなら、デコード結果をジョブ間で共有して一度きりにすればよい」という発想を制度化しました。Cachew はクラウド上で前処理を「サービス」として切り出し、入力処理の自動スケールとキャッシュ自動挿入を行います。動機は、研究の現場では1つのデータセットに対し何十回も学習を回すため、デコードの重複がクラスタ規模では巨大なコストになることです。

## 論文の実験結果 (定量データ)

各指標が何を測っているかを添えながら、報告値を並べます。ここでの主役の指標は「学習スループット (images/sec、1秒あたり処理画像数。大きいほど速い)」と「GPU 利用率 (GPU が遊ばず計算していた時間の割合。100% に近いほど良い)」、そして「エポックあたり時間」です。

FFCV の報告。ImageNet 分類で、データパイプラインを .beton 事前デコード形式に置き換えることで、標準的な DataLoader 構成に対して学習スループットが数倍に向上したと報告されています。特筆すべきは、データがディスクからメモリに乗り切る規模では、2 エポック目以降は OS のページキャッシュが効いてデコードもディスク I/O も実質ゼロに近づき、GPU 利用率がほぼ飽和する点です。指標の意味としては、GPU 利用率が 50% から 95% に上がれば、同じハードで学習が約 2 倍速くなることを意味します。FFCV はまた、データ拡張すら JIT コンパイルして高速化する設計を持ち、ResNet-18/50 を 1 GPU でも従来比で大幅に速く回せると示しました。

DALI の報告。NVIDIA は、JPEG デコードを CPU から GPU (nvJPEG) に移すことで、特に高解像度・多 GPU 構成で CPU がボトルネックになる状況での画像供給スループットが大きく改善することを示しています。8 GPU や 16 GPU といったマルチ GPU では、CPU コア数あたりの GPU 数が増えるほど CPU デコードが律速しやすく、GPU デコードの効果が顕著になります。逆に、CPU コアが潤沢で GPU が少ない構成では、GPU デコードが本来の学習計算と資源を取り合い、効果が薄れたり逆効果になったりすることもあります。これは「ボトルネックがどこにあるか次第」という普遍的な教訓です。

データストール分析の報告。Mohan らは、ResNet/VGG/Inception などで、ストレージとメモリの構成によってはエポック時間の 10% から 65% がデータ供給待ちであったと定量化し、キャッシュをデータセットサイズに合わせて設計するとこの待ち時間を大幅に削減できると示しました。アブレーション的な知見として、データセットがキャッシュに収まらない場合は素朴なメモリキャッシュの効果が急減し (キャッシュミスが頻発するため)、その場合は事前デコード形式の縮小 (解像度を落とす、ある程度の圧縮を残しておく) のほうが効くことが報告されています。これは「キャッシュ容量とデータ量の比」という一つの数で挙動が不連続に変わることを示す重要な観察です。

動画ドメインでの一般的な傾向。動画デコードは画像 JPEG デコードよりさらに重く、特にランダムな時刻へのシークを伴うと、フレーム単位のデコードコストがキーフレーム間隔 (GOP 長) に強く依存します。キーフレームから遠いフレームほど多くの差分を再生する必要があるため、最悪ケースのフレームはキーフレームの数倍から十数倍のコストになりえます。このため、動画ベースの学習では事前抽出による削減効果が画像タスクより大きく出る傾向があります。前掲の「63 フレーム × 10 エポック = 630 回 を 63 回に減らす」見積りはこの効果を端的に表しています。

## メリット・トレードオフ・限界

メリット。重いデコードを学習前の 1 回に集約し、毎エポックの再デコードを排除できるため、GPU の待ち時間が減り学習が速くなります。窓を使う学習では隣接サンプルで共有されるフレームの重複デコードもキャッシュで省けます。shard 化すればオブジェクトストレージと相性よく、分散学習にスケールしやすくなります。供給関数を差し替えるだけで済む設計にしておけば、窓のロジック (correctness) を変えずに高速化を後付けできます。

トレードオフ。事前抽出したデコード済みデータは、フレーム間予測による圧縮を解いた状態なので、元の圧縮動画よりはるかにストレージ容量を食います (数倍から十数倍になりうる)。たとえば 7 カメラ × 数百時間の運転ログを生フレームで保存すると、テラバイト級からペタバイト級に膨れることもあります。前処理工程そのものに時間がかかり、学習を始めるまでのリードタイムが伸びます。キャッシュはメモリを消費し、窓が大きい・データが多いとメモリ不足になります。さらに、前処理 (リサイズ、正規化、データ拡張) を抽出時に固定してしまうと、後で前処理を変えたくなったときに全データを作り直す必要があります。データ拡張 (augmentation、ランダムに切り抜く・色を変える等で多様性を水増しする操作) のように毎エポック異なる変換を当てたい場合は、デコードだけを事前化し拡張は学習時に行う、という分割が要ります。ここに「どこまで事前計算するか」というスペクトラムがあり、生フレームだけ事前化する/前処理まで事前化する/特徴 (バックボーン出力) まで事前化する、の各点で容量・柔軟性・速度のバランスが変わります。

限界・未解決の課題。事前抽出済みデータとモデルが要求する前処理仕様の整合を保つのは運用上やっかいで、解像度やカメラ構成を変えるたびに作り直しが必要になります。キャッシュとデータセットサイズの関係 (収まるか否か) で挙動が不連続に変わるため、汎用に効く一つの設定は存在せず、データ規模・ストレージ階層・GPU 数に応じた設計が要ります。また、ランダムシャッフルの質と I/O 効率はトレードオフ関係にあり、shard ストリーミングではシャッフルが弱まることで学習の収束や汎化に影響しうる点は、依然として実務的な調整事項です。理論的には、shuffle buffer のサイズと shard 内サンプル順序によって「どれだけ真のランダムに近いか」が決まり、これが小さすぎると同じ shard 内の相関したサンプルが固まって流れ、勾配のばらつき (=学習の探索性) が下がります。

## 発展トピック・研究の最前線

近年は、入力パイプラインそのものを「学習する」「自動最適化する」方向 (tf.data の AUTOTUNE、Cachew のようなパイプライン共有・キャッシュ共有システム) が進んでいます。また、複数の学習ジョブが同じデータセットを使う場合に、デコード結果をジョブ間で共有してデコードを一度きりにする「データエコシステム」的アプローチ (CoorDL など) も研究されています。動画基盤モデルや World Model のように 1 サンプルのフレーム数が爆発的に増える領域では、デコードコストが学習設計を左右する一級の要素となり、潜在表現 (latent) を事前に計算して保存する「特徴キャッシュ」(画素ではなく、画像エンコーダ=バックボーンの出力テンソルを保存する) が次の最適化点として注目されています。これは精度と容量・前処理柔軟性のトレードオフをさらに先鋭化させる方向です。特徴を固定保存すると、バックボーンを再学習 (fine-tune) できなくなる、データ拡張が当てられなくなるという制約が生じる一方、デコードも順伝播も省けるため最も高速になります。凍結バックボーン (frozen backbone) を前提とした自己教師あり学習や世界モデルでは、この特徴キャッシュが特に親和性が高く、画像の代わりに数十次元から数百次元の特徴ベクトルだけを保存すればよいため容量も劇的に小さくなります。
