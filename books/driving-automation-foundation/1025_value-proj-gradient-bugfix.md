# value_proj へ勾配を流すバグ修正（dead layer の検出と回避）

## ひとことで言うと

アテンション系のモジュールでは、入力特徴を「学習可能な線形変換」に通してから使うのが設計の基本です。ところが実装ミスで、変換を「計算はするが出力を使わない」状態にしてしまうと、その層には勾配（学習の手がかり）が一切流れず、初期値のまま永遠に学習されません。これを「層が死んでいる（dead layer）」と呼びます。本稿は、アテンションのバリュー変換（value projection）でこの種のバグが起きる典型例と、原因・影響・検出方法・直し方を、前提知識ゼロから掘り下げます。修正自体は数行ですが、「学習されない死んだ層」を「ちゃんと学習する層」に戻す効果は大きく、しかもエラーが出ないため見逃されやすい、研究実装で頻出の落とし穴です。

## 直感的な理解

ニューラルネットワークの学習は、各層の重み（パラメータ）を、誤差が小さくなる方向に少しずつ調整する作業です。「どちらにどれだけ動かせば誤差が減るか」を教えてくれる量が勾配（gradient）で、これは最終的な誤差（loss）から出発してネットワークを逆向きにたどり、各層へ伝わります。この伝播を誤差逆伝播（backpropagation）と言います。

ここに決定的な性質があります。「ある層の出力が、最終的な loss の計算に一切つながっていない」場合、その層には勾配が伝わりません。計算グラフ（どの値がどの値から作られたかの依存関係）の上で、loss から逆向きにその層へ到達する経路が無いからです。勾配が来なければ重みは初期値から1ミリも動かず、永遠に学習されません。これが dead layer です。

たとえるなら、会議で発言したつもりが実は誰にも聞こえておらず（出力が使われていない）、議事録（loss）にも反映されない状態です。本人（その層）は「自分の発言が良かったか悪かったか」のフィードバック（勾配）を一切受け取れないので、次回も改善できません。

厄介なのは、この状態でもプログラムはエラーを出さず、loss も下がっていくように見えることです。死んだ層は「最初から無かったのと同じ」になるだけで、目に見える失敗が出ません。だからこそ「全パラメータに本当に勾配が来ているか」を明示的に検査する価値があります。これは「サイレントな失敗（silent failure）」の代表例で、深層学習の実装では数値が出ない・落ちない・しかし最適性能に届かない、というタイプの不具合が最も発見しにくいのです。

## 基礎: 前提となる概念

**線形層（`nn.Linear`）**: 入力ベクトルに重み行列を掛けてバイアスを足す変換 `y = Wx + b` です。`nn.Linear(256, 256)` なら 256 次元入力を 256 次元出力に写し、パラメータは重み 256×256 とバイアス 256、合わせて約 6.6 万個です。

**アテンションの Query/Key/Value**: アテンションでは入力をそれぞれ別の線形変換に通して、Query（問い合わせ）、Key（見出し）、Value（中身）を作ります（Vaswani et al., 2017）。Value への変換（value projection、本稿の主役）は「集約される中身の表現」を作る本質的な部品です。これを通さずに生の特徴を使うと、「学習可能なバリュー表現を作る」という設計意図そのものが失われ、アテンションは固定の入力特徴をそのまま混ぜるだけの操作に退化します。

**計算グラフと `requires_grad`**: PyTorch は順伝播の最中に「どの計算でどのテンソルが作られたか」を記録し、計算グラフを動的に構築します。`loss.backward()` はこのグラフを逆向きにたどって各パラメータの `.grad` を埋めます。ある層の出力がグラフ上で loss につながっていなければ、その層のパラメータの `.grad` は `None` のまま、あるいはゼロになります。

**`F.grid_sample`**: 特徴マップ上の任意の小数座標から、近傍4点を双線形補間して値を取り出す関数です。BEV 融合や変形アテンション近似で「このマスは画像上のこの位置に映っている」という座標の特徴を抜き出すのに使います（[F.grid_sample による可搬な変形アテンション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0968_grid-sample-deformable-attention) 参照）。サンプリングの対象に何を渡すか（生特徴か変換後か）が、本稿のバグの分かれ目になります。

## 仕組みを詳しく

問題が起きる典型的なコード構造を、テンソル形状つきで追います。BEV 融合モジュールが、複数カメラ特徴 `fused_per_view`（形状 `[B*V, C, H, W]`、B=バッチ、V=カメラ台数、C=チャンネル例256、H, W=特徴マップ縦横）を入力に取り、`value_proj`（`nn.Linear(C, C)`）でバリューを作って `grid_sample` でサンプルする、という流れを想定します。

**正しい流れ（修正後）**

```
feat   = fused_per_view を [B, V, H*W, C] に並べ替え
values = value_proj(feat)                    # 学習可能な変換を通す [B, V, H*W, C]
                                             #  ← value_proj に勾配が来る経路の起点
values_spatial = values を [B*V, C, H, W] に戻す
for v_idx in range(V):
    feat_v  = values_spatial[..., v_idx]      # value_proj 済みの特徴
    sampled = F.grid_sample(feat_v, grid)     # この特徴をサンプル
    ...                                       # sampled が loss につながる
```

`grid_sample` に渡すのが `value_proj` の出力（`values_spatial`）なので、`value_proj` → `grid_sample` → 集約 → loss という経路ができ、勾配が `value_proj` まで遡れます。

**バグの流れ（修正前）**

```
values = value_proj(feat)             # 計算はする…が
sampled = grid_sample(生の特徴)        # ← value_proj の出力ではなく生特徴を渡す
                                      #    value_proj の出力はどこにも使われない
```

このとき `value_proj(feat)` の結果はどの後続計算にも使われず、loss につながりません。結果として、

- `value_proj` の出力は計算グラフ上で「行き止まり」になる
- `value_proj` には勾配が来ない（`.grad` が `None` または全ゼロ）
- `value_proj` は初期化されたランダム重みのまま一度も更新されない

という状態になります。設計上学習させたかった約 6.6 万パラメータが完全に無駄になり、しかもエラーは出ません。さらに無駄な順伝播計算（使われない `value_proj`）が毎ステップ走るため、ごくわずかですが計算も浪費します。

**修正の本質**: 直し方は「`grid_sample` に渡す特徴を、生特徴から `value_proj` 済みの特徴に付け替える」だけです。1経路の差し替えで、死んでいた層が生き返ります。

```
修正前: sampled = grid_sample(生特徴)        → value_proj に勾配ゼロ
修正後: sampled = grid_sample(value_proj済み)  → value_proj に勾配が流れる
```

**なぜ見逃されやすいか**: PyTorch は「ある層の出力を使わない」こと自体をエラーにしません。形状さえ合えば順伝播は通り、loss も他の生きている層が学習することで下がっていきます。死んだ層は性能をわずかに損なうだけで、明確な失敗信号を出しません。コードレビューでも「value_proj は呼ばれているし、その結果らしき変数もある」ため、出力が後段に渡っていないことに気づきにくいのです。だからこそ、勾配の有無を明示的にテストする必要があります。

**検出方法（テストの書き方）**: 1ステップ学習を回した後、対象パラメータの勾配が「`None` でない、かつ全要素ゼロでない」ことを確認します。

```
loss.backward()
for name, p in model.named_parameters():
    assert p.grad is not None, f"{name} に勾配が来ていない（dead layer 疑い）"
    assert p.grad.abs().sum() > 0, f"{name} の勾配が全ゼロ"
```

このテストは value_proj に限らず、クエリ埋め込みや投影層など「学習されるべき全パラメータ」に勾配が届くかをまとめて検証できる、汎用的な健全性チェックです。実装が大きくなるほど、こうした「全パラメータが生きているか」の自動テストの価値が高まります。

## 手法の系譜と主要論文

この修正は特定論文の手法ではなく、アテンション系アーキテクチャの実装上の一般的注意点です。背景となる設計思想を挙げます。

**Vaswani et al., "Attention Is All You Need", NeurIPS 2017 (arXiv:1706.03762)**。アテンション機構を定式化し、入力を Query・Key・Value の3つに線形変換してから使う構造を確立しました。Value への変換は、集約される中身の表現力を担う本質的な部品です。これを通さず生特徴を使うのは、論文の設計図に反する実装です。マルチヘッドアテンションでは、Value 変換が特徴空間を複数の部分空間に分割する役割も担っており、これが死ぬとヘッド間の表現の多様性も失われます。

**Zhu et al., "Deformable DETR", ICLR 2021 (arXiv:2010.04159)**。変形アテンションを提案し、ここでも value は線形変換 `W_value` を通してからサンプリングされます。`grid_sample` で変形アテンションを近似する実装（[F.grid_sample による可搬な変形アテンション](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0968_grid-sample-deformable-attention)）は、この構造を簡略再現するため、value 変換を経由させるのが正しい振る舞いです。サンプルする対象が「生特徴」か「value 変換後」かは、設計意図の根幹に関わります。

**Glorot & Bengio, AISTATS 2010「Understanding the difficulty of training deep feedforward neural networks」**。重み初期化と勾配フローの関係を体系的に論じた古典で、勾配が層をまたいで適切に流れることの重要性を確立しました。dead layer は「勾配がゼロ」という極端なケースですが、勾配消失（gradient vanishing、層を遡るほど勾配が指数的に小さくなる現象）と地続きの問題であり、いずれも「勾配フローの健全性」という同じ視点で捉えられます。

論文側に「value 変換をスキップするとどうなるか」という直接の議論はほぼありません。これは手法ではなく「実装が設計図通りか」という品質レイヤの問題だからです。ただし、より広い文脈では、dead layer や勾配消失の検出・回避は深層学習の実装健全性として長く議論されてきました。勾配フローの可視化（各層の勾配ノルムを監視する）、勾配チェック（数値微分との突き合わせ）、ユニットテストでの勾配存在確認などは、研究実装の品質を保つ標準的な手段です。

## 論文の実験結果（定量データ）

このバグ修正自体に固有のベンチマーク数値はありませんが、効果の大きさは関連事実から見積もれます。

**影響を受けるパラメータ規模**: 256→256 の value 変換なら 256×256 + 256 ≒ 6.6 万パラメータ。これがアテンション集約の中身表現を担う層であれば、死んでいる間はその表現力がゼロ（ランダム固定で、しかも出力が使われていないので実質的に恒等的なバイパスと同等）になります。アテンションが「どの特徴をどう混ぜるか」のうち「混ぜる中身を変換する」部分を学べないため、融合品質に直接響きます。

**勾配存在テストの価値**: 一般に、深層モデルでは「学習が進んでいるように見えても一部の層が動いていない」事例が報告されます。アブレーション的に言えば、value 変換を恒等固定にした場合と学習可能にした場合では、アテンションの表現力に差が出ます。Deformable DETR / BEVFormer 系の論文で value 変換を含む全投影層が学習対象として設計されている事実が、「これらを学習させることが精度に寄与する」ことの間接的な裏づけです。逆に言えば、value 変換が死んでいるモデルは、設計上期待される精度の上限に届きません。

**検出の確実性**: 勾配存在テスト（`p.grad` が非ゼロか）は、dead layer を 1 ステップの学習で確定的に検出できます。これは「精度が出ない原因が dead layer かどうか」を切り分ける、コストゼロに近い診断です。数日かかる学習を回す前に、1ステップのテストで構造的な欠陥を弾けるため、実験のサイクルを大きく短縮します。

## メリット・トレードオフ・限界

**メリット**

- 学習されるはずだったパラメータが実際に学習され、アテンション集約の表現力が設計通りに発揮される。
- 「全パラメータに勾配が来るか」のテストを併せて導入すれば、同種の dead layer バグを将来も自動検出できる。
- 計算コストの増加はほぼゼロ。value 変換はもともと計算自体はしていたので、出力を使うだけ。むしろ「使われない計算」が意味を持つようになる。

**トレードオフ・注意点**

- ほぼ無いが、「学習される層が増える」ことで学習初期の挙動がわずかに変わる。修正前は value 変換が事実上ランダム固定だったため、修正後はハイパーパラメータ（学習率など）の再調整が要る場合がある。これは正しい挙動に戻った副作用であり、デメリットではない。
- 勾配存在テストは「勾配が来ているか」は検出できるが、「勾配の大きさが妥当か（勾配消失していないか）」までは保証しない。極端に小さい勾配が長時間続く別種の問題は、勾配ノルムの監視など追加の検査が要る。
- 勾配が全ゼロでないことを確認するだけでは、「ごく一部の要素だけ非ゼロで残りは死んでいる」部分的な dead を見逃すことがあり、より厳密には勾配の分布まで見る必要がある。

**未解決の課題（研究視点）**: dead layer は「使われていない出力」という静的な誤りですが、より発見しにくいのは「動的に一部の経路だけ使われない」ケース（条件分岐で特定入力のときだけ層が無効になるなど）や、「勾配は来るが恒常的に消失している」ケースです。計算グラフを静的解析して「loss に到達しないパラメータ」を自動列挙するツールや、学習中に勾配ノルムを層ごとに常時監視する仕組みは、実装健全性を担保する研究・エンジニアリングの継続テーマです。

## 発展トピック・研究の最前線

勾配フローの健全性は、近年の大規模モデルでも重要性を増しています。混合精度学習（bf16/fp16）では小さな勾配がアンダーフローで消える問題があり、勾配スケーリング（fp16）や、ダイナミックレンジの広い bf16 の採用で対処します。また、`torch.compile` や計算グラフ最適化によって「未使用の計算が自動で枝刈り（dead code elimination）される」ことがあり、これは効率上は望ましい一方で、dead layer を「見えなくする」リスクもあります。最適化後も意図した層が学習されているかを検証する習慣が重要です。さらに、勾配チェックポインティング（メモリ節約のため順伝播を一部再計算する技法）や勾配累積（小バッチを足して大バッチ相当にする技法）を併用する複雑な学習ループでは、勾配が正しく蓄積・伝播しているかのテストが一段と価値を持ちます。研究の最前線では、層ごとの勾配ノルムや有効ランクを可視化して学習の健全性を監視するダッシュボード（Weights & Biases などでの勾配ヒストグラム）が標準的なプラクティスとなりつつあり、「サイレントな失敗」を可視化する方向に進んでいます。
