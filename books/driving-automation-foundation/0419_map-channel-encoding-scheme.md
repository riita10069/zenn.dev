# 地図チャンネル符号化方式

## ひとことで言うと
自動運転モデルに地図を画像として渡すとき、車線・道路境界・横断歩道・信号などの要素を「どの色」「どの層(チャンネル)」に描くかを決める設計の話です。この決め方しだいで、モデルが地図から読み取れる情報量と、計算の重さ、既存モデルとの相性が大きく変わります。地図表現の最初の分かれ道です。

## 直感的な理解
モデルは、カメラ映像だけでは「この先の道がどこへ続くか」「自分はどの車線を走るべきか」を判断しきれません。曲がり角の奥や建物の陰の道は映らないからです。そこで地図を追加で渡します。

ところがモデルは、人間のように「これは横断歩道、これは信号」と地図記号を意味として読めません。モデルが扱えるのは数値の並びだけです。だから地図を「数値の配列=テンソル」に翻訳して渡す必要があります。テンソルとは、数値を縦・横・奥行きなど複数方向に並べた多次元配列のことで、たとえばカラー画像は「縦のピクセル数 × 横のピクセル数 × 色チャンネル数(赤・緑・青の3つ)」という3次元テンソルです。

地図をテンソルにする最も素直なやり方は、真上から見た絵(BEV、鳥瞰図)に描くことです。これをラスタ地図と呼びます。ラスタとは情報をピクセルの格子で表す方式で、写真と同じ形式です(対義語のベクタ地図は座標で表す方式で、これは別ノート[オンライン vs オフライン地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0456_online-vs-offline-mapping)の主題)。

ラスタ地図を作る最大の設計判断が「どの要素を、どの色・どの層に描くか」です。これが本ノートのテーマ「チャンネル符号化方式」で、符号化(エンコード)とは情報を別の形式に変換することを指します。料理にたとえると、同じ食材(地図要素)を、何枚の皿に・どう盛り付けるか(チャンネル配置・配色)で、食べる側(モデル)の理解しやすさが変わる、というイメージです。盛り付けを間違えると、せっかくの食材も価値が伝わりません。

## 基礎: 前提となる概念

チャンネル。画像の「奥行き方向の層」のことです。白黒画像は1チャンネル、カラー画像は赤・緑・青の3チャンネル。ニューラルネットの入力は `(チャンネル数 C, 高さ H, 幅 W)` という形のテンソルで、C を自由に増やせます。最初の畳み込み層(画像を小さな窓で走査して特徴を抜き出す層)は、入力チャンネル数 C に応じた重みを持つので、C を変えるとこの層の形も変わります。

セマンティック(semantic)。「意味的な」という意味で、ここでは「要素の種類(車線・歩道・信号など)を表す」ことを指します。セマンティックレイヤとは、種類ごとに分けた層のことです。

one-hot 表現。「クラスを整数1個で持つ」代わりに「クラス数ぶんの0/1ベクトルで、該当クラスだけ1」にする表し方です。たとえば3クラスなら「車線=(1,0,0)、横断歩道=(0,1,0)、信号=(0,0,1)」。整数のまま渡すと「車線=0、横断歩道=1」のように大小関係がついてしまい、モデルが「横断歩道は車線より大きい」と誤解しかねないので、種類に順序がないものは one-hot にするのが定石です。セマンティックレイヤ方式は、空間方向に one-hot を展開したものとも言えます。

事前学習バックボーン。ImageNet という1000種類の物体写真の大規模データであらかじめ学習させた特徴抽出器(ResNet, Swin Transformer など)のことです。写真の特徴を捉える力をすでに持っているので、新しいタスクでもゼロから学ぶより速く・高精度に学べます。ただしこれは入力が「3チャンネルのカラー画像」であることを前提に作られています。ここが後述の方式選択に効いてきます。

正規化(normalization)。各ピクセル値を一定の範囲(たとえば平均0・分散1)にそろえる前処理。写真用バックボーンは ImageNet の統計値(チャンネルごとの平均と標準偏差)で正規化された入力を期待します。地図を同じ3チャンネル形式にすると、この前処理もそのまま使えます。

## 仕組みを詳しく

地図要素をテンソルへ割り当てる方式は、大きく2つに分かれます。

方式A: 要素ごとに別チャンネル(セマンティックレイヤ方式)。種類ごとに専用の1チャンネル(白黒1枚)を用意します。

```
チャンネル0: 走行可能領域   (道路があれば 1.0、なければ 0.0)
チャンネル1: 車線中心線
チャンネル2: 車線境界線
チャンネル3: 横断歩道
チャンネル4: 信号           (赤=1.0 など状態を数値で込みにできる)
チャンネル5: 自車の過去軌跡
...
```

要素が K 種類あれば地図テンソルは `(K, H, W)` になります。たとえば8種類を256×256で描けば `(8, 256, 256)`。強みは、各チャンネルが1種類の意味だけを持つので情報が混ざらないこと。横断歩道の層には横断歩道しか描かれないため、モデルは「チャンネル3を見れば横断歩道」と素直に学べます。信号の赤・青・黄のように時間で変わる動的情報も、専用チャンネルに数値として入れられます。さらに「過去の各時刻を別チャンネルにする」ことで、他車の動き(過去0.5秒前・1.0秒前…の位置を別レイヤに描く)まで表現でき、これは多くの予測モデルが採る形です。before/after で言えば、ある交差点に「横断歩道あり・信号は赤・右折レーンあり」という3つの事実が重なっていても、別々のチャンネルに分かれているので互いに干渉せず同時に表せます。

方式B: RGBの色で符号化(3チャンネル方式)。要素の種類を、赤・緑・青の3チャンネル(普通のカラー画像と同じ)の中に色で塗り分けます。

```
背景       → 暗いグレー
道路        → 中間グレー
走行ルート → 明るい青
GPS生データ → 赤
```

形状は `(3, H, W)` で固定。要素が何種類あっても色で区別するため層は3枚のまま増えません。強みは、普通のカラー画像とまったく同じ形式なので、写真用に作られた既存バックボーンをそのまま流用でき、カメラ画像処理と同じ部品で地図も処理できることです。事前学習の重みを最大限に活かせるので、学習データが少ないときに特に有利です。

情報量のトレードオフ。方式Aは K 個の独立した数値スロットを持つので、複数の事実を干渉なく同時表現できます。反面、要素種類を増やすほどチャンネル数が増え、メモリと計算が重くなります。たとえば K を8から20に増やせば、入力テンソルは2.5倍に膨らみ、最初の畳み込み層の計算も比例して増えます。加えて、事前学習バックボーンは3チャンネル入力を前提に重みが作られているため、入力チャンネルを K にすると最初の層だけは事前学習を捨ててゼロから学ぶ(あるいはRGB重みを複製して初期化する)必要があり、転移学習の恩恵が一部失われます。方式Bは3チャンネルしかないので、同じ場所に複数要素が重なると色が衝突します(横断歩道の白と車線の白が見分けられない、など)。情報量は方式Aより原理的に少ないものの、軽量で既存モデルとの相性が抜群です。

中間的な設計もあります。よく使う数種類だけを別チャンネルにし、残りは色で混ぜる、あるいは「クラスを整数で持つラベルマップ1枚」を入力直前に one-hot 展開して `(K,H,W)` に開く、といった折衷です。どこまで分けるかは、後段ネットワークの容量と、表現したい要素の重要度で決めます。実装上の注意として、重なりがある要素(横断歩道の上を車線が通るなど)は、別チャンネルなら自然に共存できますが、1枚のラベルマップ方式では「どちらを上に塗るか」の優先順位を決めねばならず、塗り順が情報の取捨選択になります。

数値例で重さを比較すると、256×256のタイルで方式B(3ch, float32)は 3×256×256×4 ≈ 786KB/枚、方式A(16ch)は約4.2MB/枚です。バッチ32なら前者25MB、後者134MB と、メモリ差は無視できません。学習では中間特徴も含めるとさらに差が開くため、GPU メモリが厳しい設定ではチャンネル数の節約が効きます。

## 手法の系譜と主要論文

IntentNet(Casas, Luo, Urtasun; Uber ATG, CoRL 2018, "IntentNet: Learning to Predict Intention from Raw Sensor Data")は、LiDAR の鳥瞰図と地図ラスタを複数チャンネルに重ねて入力し、他車の意図(直進・右左折など)と軌跡を同時予測しました。地図を「高さ方向に積んだ複数の意味チャンネル」として扱う、方式A 的な設計の早い例です。

ChauffeurNet(Bansal, Krishnan, Ogale; Waymo, 2018/RSS 2019, "ChauffeurNet: Learning to Drive by Imitating the Best and Synthesizing the Worst", arXiv 1812.03079)が方式A(セマンティックレイヤ)の代表例です。地図を複数の白黒チャンネルに分け、道路マスク・車線中心線・信号状態・速度制限・他車のバウンディングボックス・自車過去軌跡を、それぞれ別の層としてトップダウン(真上視点)ラスタに描きました。理由は、人間の運転を模倣して学習させる際、車線や信号といった構造化された地図情報を明示的に与えれば、生のカメラだけより安定して運転ルールを学べるからです。

ChauffeurNet の系譜から、ラスタ方式の弱点を突く流れが生まれます。VectorNet(Gao, Sun, Zhao, Shen, Anguelov, Li, Schmid; Waymo, CVPR 2020, "VectorNet: Encoding HD Maps and Agent Dynamics from Vectorized Representation", arXiv 2005.04259)は、ラスタの「解像度を上げるほど計算が爆発する」「細い車線が潰れる」という問題を指摘し、地図を点と線の座標(ベクタ表現)で扱う方向へ進みました。地図要素を折れ線として持ち、それをグラフニューラルネットワークで符号化することで、ラスタ化の情報損失を回避します。これは方式A/Bとは別系統で、ラスタ符号化に対する根本的な代替案です([オンライン vs オフライン地図](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0456_online-vs-offline-mapping), [ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation))。

公開データセットの側でも、nuScenes(Caesar et al., CVPR 2020, arXiv 1903.11027)のような地図は走行可能領域・歩道・横断歩道などのセマンティックレイヤとして提供され、方式Aが研究コミュニティの標準入力の一つとして定着しました([nuScenes Map API](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0454_nuscenes-map-api))。一方で、GPS とナビ経路しか手元にない実用設定では、軽量な方式B(RGBタイル)を採るシステムも多く、両方式は用途で住み分けています。

## 論文の実験結果(定量データ)

ChauffeurNet は、方式A のセマンティックレイヤを入力にすることで、地図情報を持たない素朴な模倣学習に比べ、閉ループ評価(モデルが運転し続ける評価)での車線逸脱・衝突・停止線無視といった違反が減ったと報告しました。とくに、わざと危険状況(車線からのはみ出しなど)を合成して学習に混ぜる工夫と、補助損失(車線内に留まる・衝突しない、を直接ペナルティにする)を組み合わせた条件で、違反が大きく減少しています。アブレーションとして、地図の意味レイヤや補助損失を外すと、複雑な交差点での挙動が崩れることが示され、「構造化された地図入力が安全な挙動の学習に寄与する」ことを定量的に裏づけました。

VectorNet は、地図をラスタで与える従来手法と、自分たちのベクタ表現を、Argoverse と社内データの軌跡予測で比較しました。報告によれば、ベクタ表現はラスタ手法と同等以上の予測精度(ADE/FDE で改善)を、計算量を桁で削減しながら達成しています。具体的には、同等精度を保ったまま演算量(FLOPs)を大幅に(報告では桁オーダーで)下げたとされ、これは「ラスタは解像度を上げないと細部が出ないが、ベクタは座標で正確に持てるので無駄が少ない」ことの表れです。ここで FLOPs は浮動小数点演算の回数で、計算コストの目安です。さらにアブレーションで、地図のベクタ情報をエージェントの相互作用グラフに加えると ADE/FDE が下がること(地図文脈が予測精度に効くこと)も示されました。この結果は、方式A/B というラスタ符号化そのものの限界(解像度と計算量のトレードオフ)を浮き彫りにしました。

指標の意味を補足すると、ADE/FDE は予測軌跡と正解軌跡の平均/最終時刻の距離(メートル)で、小さいほど良い指標です。地図の符号化が良いほど「この先の車線形状を踏まえた自然な軌跡」が出やすく、これらの誤差が下がります。たとえば FDE が 3.0m から 2.5m に下がると、6秒先の予測位置が平均で 0.5m 正確になる、ということで、これは交差点で隣車線にはみ出すか否かを分けうる差です。

## メリット・トレードオフ・限界

方式A(別チャンネル)のメリットは、情報が混ざらず要素ごとに独立表現でき、信号状態などの動的情報も扱いやすく、HD地図の豊かな情報を素直に渡せること。デメリットは、チャンネルが増えるとメモリ・計算が増大し、事前学習バックボーンの最初の層を捨てる必要があり、整備された HD 地図が前提なので地図のない場所では使えず、専用の地図描画パイプラインが要ること。

方式B(RGB)のメリットは、軽量(3チャンネル固定)で、普通のカラー画像と同じため写真用の学習済みモデルをそのまま丸ごと流用でき、カメラ処理と部品を共有できること。デメリットは、同じ場所に複数要素が重なると色が衝突して情報量が原理的に少なく、表現できる要素種類に限りがあること。

共通の限界として、ラスタである以上「解像度未満の細部は潰れる」問題から逃れられません。細い車線境界や近接した2本の線は、解像度が粗いと1ピクセルにまとまって区別不能になります。研究上の未解決課題は、要素の重要度に応じてチャンネルや解像度を適応的に割り当てる方法(重要な要素ほど高解像度に)が体系化されていないこと、そして「どの要素をモデルに与えると最終的な運転性能が最も上がるか」というチャンネル設計の最適化が、いまだ試行錯誤に頼っていることです。極端な話、入れたチャンネルがモデルにとってノイズになり性能を下げる場合すらあり、足せば足すほど良いとは限らない点も実務上の落とし穴です。

## 発展トピック・研究の最前線

ラスタ符号化の解像度・計算の壁を避けるべく、近年は地図を点と線の座標で扱うベクタ表現が主流化しつつあります。これを集合予測として出すのが[DETR型マップquery decoder](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0447_map-query-decoder-detr)、曲線で滑らかに表すのが[ベジエ曲線地図表現](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0439_bezier-curve-map-representation)です。一方、ラスタを完全に捨てるわけではなく、カメラ特徴を学習可能な投影でBEVラスタへ持ち上げ、その上でセマンティックに塗り分ける手法も発展しています([ラスタBEV地図(予測向け)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0461_raster-bev-rasterization-prediction))。地図をどう符号化するかは、最終的に「どの表現が最も少ない情報損失で、最も既存資産(事前学習・パイプライン)を活かせるか」という工学的判断であり、用途によって最適解が変わり続けているのが現状です。end-to-end 運転の文脈では、地図を入力として明示的に符号化するのではなく、生センサからBEV特徴を作る過程で地図的な情報を内部表現として暗黙的に獲得させる流れもあり、「人手で設計するチャンネル」から「学習で獲得する表現」へと、符号化の主体そのものが移りつつあります。
