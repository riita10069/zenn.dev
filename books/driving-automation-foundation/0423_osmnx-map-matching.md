# osmnx による道路ネットワーク取得とマップマッチング

## ひとことで言うと

GPS で記録された「車が走った緯度・経度の点列」を、OpenStreetMap という世界中の地図データから取ってきた「道路のつながり方の地図(グラフ)」の上にスナップ(吸い付け)して、どの道を走ったかをきれいな線にする処理です。GPS は誤差でガタガタしているので、それを実在の道路に沿わせて整えます。地図に道がうまく見つからなかったときは、無理に道路へ吸い付けず、生の GPS 点だけを描くフォールバック(代替手段)に切り替えます。

## 直感的な理解

自動運転モデルに「次にどう動けばいいか」を教えるとき、カメラ画像だけでなく「この先どっちへ進むべきか」というナビ情報(進路の地図)を入力として与えると性能が上がることが知られています。その地図を作る材料が GPS の記録、つまり各時刻の緯度・経度です。

ところがここで GPS の誤差が問題になります。消費者向け GPS は数メートルから十数メートルの誤差を持ちます。10Hz(1秒に10点)で記録された緯度・経度をそのまま線でつなぐと、車が実際には直進していても、画面上ではジグザグに揺れた線になります。トンネルやビル街では衛星の信号が反射・遮蔽されて一時的に大きく飛ぶこともあります。このノイズだらけの点列をそのままモデルに見せると、「揺れているのが正しい走り方」とモデルが誤解しかねません。

そこで「マップマッチング(map matching)」を行います。直感的には「GPS の各点を、最も近い実在の道路上の点に置き換える」処理です。道路は連続して滑らかにつながっているので、各点を道路に乗せ直すと、ジグザグだった線が道路に沿ったまっすぐな進路に整います。人間が手描きの地図に走行ルートを清書するとき、おおよその位置から「ああ、この道を通ったんだな」と道路に沿って線を引き直すのと同じ作業です。

## 基礎: 前提となる概念

グラフとノード・エッジ: グラフとは「ノード(点)とエッジ(点をつなぐ線)の集まり」というデータ構造です。道路網では、ノードが交差点やカーブの折れ点、エッジが道路区間に対応します。各ノードには経度 x と緯度 y が、各エッジには長さ length が属性として入っています。

有向多重グラフ: 道路は一方通行があるので向きを持ち(有向)、同じ2地点を複数の道がつなぐこともあるので多重(同じノード対に複数エッジ)です。これを表すのが有向多重グラフ(MultiDiGraph)です。

最近傍探索: ある点(GPS 点)に対して、グラフのノードの中で最も距離が近いものを探す処理です。k-d 木や R 木といった空間インデックス(点を空間的に区切って管理するデータ構造)を使えば、全ノードと総当たりで距離を測るより高速に探せます。

最短経路: グラフの2ノード間を、エッジの長さの合計が最小になるようにたどる経路です。古典的には Dijkstra のアルゴリズム(Dijkstra, 1959)で解けます。「次に進むべき最も近い未確定ノードを順に確定していく」貪欲な手続きで、エッジ重みが非負なら最適解を保証します。計算量はノード数 V、エッジ数 E に対しておよそ O((V+E) log V) です。

Overpass API: OpenStreetMap のデータをクエリで取り出す公開サーバの仕組みです。ある地点の周囲の道路だけを取ってくる、といった問い合わせができます。通信を伴うため遅く(秒オーダー)、ネットワーク接続が必須です。公開サーバには利用レート制限があるため、大量に叩くと弾かれることもあります。

OSMnx: 上記の取得・解析・可視化を一つにまとめた Python ライブラリ(Boeing, 2017)。指定地点周辺の道路グラフ取得、最近傍ノード探索、最短経路といった機能を数行で呼べます。

## 仕組みを詳しく

処理は大きく3段階です。順に見ます。

### 段階1: 道路ネットワークの取得

ある中心地点(緯度・経度)の周囲 R メートル(よく使う値は 800m)以内の道路を OSM から取得します。中身は Overpass API への通信です。

```
graph = graph_from_point((center_lat, center_lon), dist=R, network_type="drive")
```

`network_type="drive"` は「車が走れる道路だけ」を意味し、歩道や自転車専用道を除外します。返ってくるのは有向多重グラフで、ノードが交差点・折れ点、エッジが道路区間です。各ノードに x(経度)・y(緯度)、各エッジに length(メートル)が入っています。

### 段階2: マップマッチング

GPS の緯度列・経度列を受け取り、2つのものを作ります。

1. matched_nodes: 各 GPS 点に最も近いグラフ上のノード。最近傍探索で求めます。入力点と同じ個数になります。
2. route_nodes: それらをつないで「連続した1本の経路」にしたノード列。

ただ最近傍ノードを並べるだけでは不十分です。なぜなら、隣り合う2つのマッチ先が道路グラフ上では離れていることがあるからです。例えば1つ目が交差点A、次が交差点Cで、間に交差点Bを通らないと物理的に行けない、という状況です。そこで、隣り合うマッチ先の間を最短経路で埋めて、すべてつないで連続経路にします。

```
matched_nodes = nearest_nodes(graph, lons, lats)
route_nodes   = []
for (src, dst) in 隣り合うペア(matched_nodes):
    seg = shortest_path(graph, src, dst, weight="length")
    route_nodes += seg(先頭の重複は除く)
```

連結部分でノードが重複しないよう、各セグメントの先頭を切り落とす処理を入れます。具体例で流れを追います。GPS 5点があり、最近傍ノードが [A, A, B, D, E] だったとします。

```
A→A  同じノードなので1つだけ残す
A→B  最短経路 [A, X, B] で補間(間の交差点 X を通る)
B→D  最短経路 [B, C, D]
D→E  最短経路 [D, E]
```

重複を除いてつなぐと、route_nodes は [A, X, B, C, D, E] という連続列になります。元の GPS では飛んでいた A から B の間も、道路上で通ったはずの X が補われ、滑らかな経路になっているのがわかります。

### 段階3: フォールバック(空ルートへの対処)

マップマッチングは失敗しうる前提で書くのが堅牢な設計です。失敗の典型例は次の通りです。

```
- 半径内に道路エッジが1本もない(山道や私有地)
- グラフが分断されていて経路が引けない(NoPath)
- GPS がグラフの範囲外に飛んでいる
```

最近傍探索が例外を投げたら空の結果を返し、最短経路が失敗した区間はスキップ、それでもルートが空なら最初のマッチノード1つだけを入れる、といった段階的な保険をかけます。最終的に route_nodes が空であれば、呼び出し側(描画処理)は「道路に沿った青いルート線は描かず、生 GPS の赤い点だけを描く」よう振る舞います。これが「空ルートは生 GPS のみにフォールバック」の意味です。失敗で処理全体を止めず、最低限の情報(GPS 点)だけでも残す堅牢な作りです。多数のクリップを一括処理するとき、ごく一部の難しいクリップで全体を止めないためにこのフォールバックは必須です。

## 手法の系譜と主要論文

マップマッチングは自動運転より前から、カーナビゲーションやフリート(車両群)管理の分野で長く研究されてきた古典的問題です。系譜を追います。

OpenStreetMap 自体は Haklay & Weber(2008, IEEE Pervasive Computing)で紹介された、ボランティアが作る世界地図プロジェクトです。利点は無料で世界中をカバーすること、トレードオフは地域によって精度・更新頻度に差があり、田舎では道が欠けていることがある点です。この地域差が、後述するマップマッチング失敗の現実的な原因になります。

最短経路の理論的土台は Edsger Dijkstra(1959, Numerische Mathematik)です。グラフ上の2点間を最小コストでたどる手続きで、道路区間の長さをエッジ重みにすれば「最も短い道のり」を求められます。マップマッチングの連結補間はこのアルゴリズムを土台にしています。

マップマッチングの代表的な定式化が Newson & Krumm(2009, ACM SIGSPATIAL "Hidden Markov Map Matching Through Noise and Sparseness", Microsoft Research)です。GPS の各点が「どの道路上にあるか」を隠れマルコフモデル(HMM、各時刻の隠れた状態を確率的に推定する手法)で解きます。各 GPS 点が道路に乗る確率(観測確率、距離が近いほど高い)と、隣り合う点の道路間の移動しやすさ(遷移確率、道なりに行ける距離と GPS 点間の距離の差が小さいほど高い)を掛け合わせ、最も尤もらしい道路列を Viterbi アルゴリズム(系列の最尤経路を動的計画法で求める手続き)で選びます。提案の動機は「単純な最近傍だと、平行に走る2本の道や交差点付近で取り違えるから」。効果はノイズが大きく・サンプリングが疎な GPS でも頑健にマッチできること。トレードオフは計算量が増え、観測・遷移確率のパラメータ調整が要ることです。低サンプリングレートへの対処を別角度から扱った Lou et al.(2009, ACM SIGSPATIAL "ST-Matching")も同時期の重要な系譜で、空間的近さと経路の妥当性を組み合わせるアイデアを示しました。

これに対し「最近傍ノード+最短経路補間」方式は、HMM を使わない軽量な実装です。最近傍探索と最短経路を組み合わせるだけで、確率モデルは使いません。採用の動機は、オフライン前処理([オフラインのタイル事前レンダリングとネットワークキャッシュ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0692_offline-tile-prerendering-cache))で一度動けばよく、実装が単純で依存も少ないこと。効果は素早く書けて保守しやすいこと。トレードオフは、平行路の取り違えや、ノイズが大きいときに HMM ほど賢く復元できないことです。教育上の整理としては「軽量だが素朴なマップマッチング」と理解してください。多くの実用パイプラインでは、地図描画のように多少の誤マッチが致命的でない用途なら軽量方式で十分、車両の正確な走行レーン特定のように精度が要る用途なら HMM 方式、と使い分けます。

## 論文の実験結果(定量データ)

Newson & Krumm(2009)は、シアトル近郊で収集した実走行 GPS データ(高精度の真値経路つき)を使い、サンプリング間隔とノイズを人工的に変えながらマッチ精度を評価しました。指標は「正しい道路区間にマッチした割合」と「誤マッチの距離割合(route mismatch fraction、誤って選んだ道路の長さが全体に占める割合)」です。報告の核心は、サンプリング間隔が1〜2秒と密なら最近傍ベースの素朴な方法でも高精度だが、間隔が30秒〜数分と疎になる、あるいは GPS ノイズが大きくなると、素朴な方法のマッチ率が急落するのに対し、HMM 方式は劣化が緩やかで、疎・ノイジーな条件でも高いマッチ率を維持できた、という点です。具体的には、サンプリング間隔が長くなるほど両者の差が広がり、数十秒間隔の条件では HMM が素朴法を大きく引き離す結果が示されています。

この数値が意味するのは、データの「密さ」と「ノイズの大きさ」がマップマッチングの難しさを決めるということです。10Hz のように密で、地図描画という多少の誤差を許容する用途なら、素朴な最近傍+最短経路で実用上問題ありません。逆に、サンプリングが疎で精密なレーン特定が要る用途では、HMM のような確率的手法の投資が報われます。アブレーション的に言えば、「遷移確率(道なりの移動しやすさ)」を外して観測確率だけにすると、それは実質最近傍マッチに退化し、平行路や交差点での誤マッチが増える、という関係になります。つまり HMM の頑健さの源泉は遷移確率にある、と読み解けます。

OSMnx(Boeing, 2017)については、論文は世界中の都市の街路ネットワークを取得・解析する再現性のある手続きを提供したことが貢献で、特定の精度数値というよりは「数行のコードで任意都市の道路グラフを取得・指標化(交差点密度や circuity といった都市形態の指標)できる」という再現性・利便性が評価ポイントです。

## メリット・トレードオフ・限界

メリットは、GPS の誤差を実在の道路へ吸い付けて進路をきれいな線に整えられること、地図タイルを持たないデータセットでも地図入力ありのデータと同じ形式をそろえられること、失敗時に生 GPS へフォールバックするので処理が止まらないこと(堅牢)、そして実装が軽量(最近傍+最短経路)で依存も前処理時だけで済むことです。

トレードオフと限界は次の通りです。第一に、Overpass API への通信が必要で遅いため、学習のホットパス(DataLoader 内)では使えず、事前のバッチ処理が前提になります([オフラインのタイル事前レンダリングとネットワークキャッシュ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0692_offline-tile-prerendering-cache))。第二に、最近傍+最短経路は素朴なので、平行路の取り違えや大ノイズ時の誤マッチがありえます(HMM 方式ほど賢くない)。具体的な失敗例として、高速道路とその側道のように平行する2本の道があると、GPS 誤差で側道側のノードに吸着し、実際とは違う経路を描いてしまうことがあります。第三に、OSM の地域差により道が欠けてマッチが失敗することがあります。第四に、半径や最近傍の探索範囲などのパラメータが用途に合わないと精度が落ちます。半径が小さすぎると経路が範囲外に出てマッチが切れ、大きすぎると無関係な道まで候補に入り誤マッチが増えます。

研究上の未解決課題としては、計算コストを抑えつつ HMM 並みの頑健さを得る軽量近似(例えば候補を絞った簡易 Viterbi)、地図が欠けている地域での補完、そして GPS と地図の双方に誤差がある状況での不確実性の定量化が挙げられます。

## 発展トピック・研究の最前線

近年は、HD マップ(高精細地図)とレーンレベルのマップマッチングが研究の中心になっています。車線単位で「どのレーンを走っていたか」まで特定する手法は、自動運転の挙動評価や予測に直結します。また、地図を事前に取得せず、センサ(カメラ・LiDAR)からオンラインで車線地図を生成する MapTR や VectorMapNet(2022〜2023)のような手法が登場し、「地図取得とマッチング」というパイプライン自体を学習で置き換える流れも生まれています。

一方で、本トピックのような「公開地図データ + 軽量マッチング」によるオフライン前処理は、大規模な過去データから学習用ラベルや入力を安価に合成する手段として依然として有用です。研究の最前線では、確率的マップマッチングの高速化と、地図品質が低い地域への頑健化が、地味ながら継続的なテーマになっています。深層学習でマップマッチング自体を学習する試み(GPS 系列を入力に道路系列を出力するエンコーダ・デコーダ)も現れており、HMM のような手作りの確率モデルとの精度・汎化のトレードオフが議論されています。
