# プランナレジストリ（差し替え可能なプランナ設計）

## ひとことで言うと
「軌道を出す部品（プランナ）」を、文字列のキー（たとえば `"gru"` や `"bezier"`）で呼び出して差し替えられるようにする設計の考え方です。名前とクラスの対応表（レジストリ）にプランナを登録しておくと、設定の1行を変えるだけで別のプランナに切り替わり、まったく同じ条件で複数の手法を公平に比べられます。アルゴリズムを比較研究するときに「比較の土俵を一定に保つ」ための土台になる仕組みであり、ソフトウェア工学の Factory／Registry パターンを機械学習研究の文脈に持ち込んだものです。

## 直感的な理解
自動運転や軌道予測のモデルでは、最後の段に「プランナ（planner）」があります。これはカメラや地図などから得た状況をもとに「これからどう走るか」を出力する部品です。出力はたとえば加速度と曲率（ハンドルの曲がり具合）の時系列だったり、未来の座標点の列だったりします。

プランナの作り方には複数の流派があります。1ステップずつ過去を踏まえて出す再帰型（GRU など）、生成モデルとして滑らかな軌道をサンプルする拡散・フローマッチング系、少数の制御点から滑らかな曲線を引くベジエ・多項式系などです。どれが優れているかは「速いか」「滑らかか」「学習が安定か」「最終的な運転が安全か」といった複数の軸があり、机上では決まりません。実際に作って比べるしかないのです。

ここで研究上の最大の落とし穴が「公平な比較」です。たとえば手法Aと手法Bを比べたいとき、Aは埋め込み次元128・予測32ステップ、Bは埋め込み次元256・予測64ステップという具合に条件がずれていると、「Aが速いのは設計が優れているからなのか、単に小さく作っただけなのか」が切り分けられません。論文の世界では、この種の条件のズレが原因で「本当はベースラインの調整不足だったのに新手法の勝ちと報告してしまう」事故が繰り返し起きてきました。比較のたびに呼び出しコードを手で書き換えていると、こうした条件のズレが必ず混入します。

そこで「すべてのプランナを同じインターフェース（共通の入出力の約束）に従わせ、名前で呼び出せる対応表にまとめる」という設計が効いてきます。これがレジストリパターンです。比較したい手法を `["gru", "flow_matching", "bezier"]` のようにキーで並べ、同じ構成引数・同じ入力・同じハードウェアでループすれば、条件のズレを構造的に排除できます。

## 基礎: 前提となる概念
**プランナと軌道**。軌道（trajectory）は未来の運動を時間で区切った点の列です。たとえば「0.05秒後にここ、0.10秒後にここ」を64点並べたものです。プランナはこの列を出力する関数だと思ってください。出力の表現には、各時刻の座標 (x, y) を並べるもの、各時刻の制御信号（加速度・曲率）を並べるもの、複数候補とその確率を出すもの（多峰出力）などがあります。

**インターフェース（interface）**。「この部品はこういう引数を受け取り、こういう形の出力を返す」という約束ごとです。プログラミングでは、共通の親クラスや関数シグネチャとして表現します。すべてのプランナが同じインターフェースに従っていれば、利用する側はどの実装を呼んでいるか意識せずに使えます。

**レジストリ（registry）**。「名前 → 実体（ここではクラスや生成関数）」を対応づける辞書です。利用する側は実体の中身を知らなくても、名前さえ渡せば対応する部品を受け取れます。これにより「使う側」と「作る側」を疎結合（互いの内部実装に依存しない関係）にできます。ソフトウェア工学では Factory パターン（Gamma, Helm, Johnson, Vlissides, "Design Patterns", 1994）の一種で、生成（オブジェクトを作る処理）の責務を一箇所に集めて使う側から隠す考え方です。Registry はその発展形で、「どんなクラスがあるか」自体も実行時に登録・追加できるようにします。

**埋め込み次元（embedding dimension）**。ニューラルネットが扱う特徴ベクトルの長さです。たとえば 256 なら、状況を要約した 256 個の数の並びで表します。比較ではこれを揃えないと容量（モデルの表現力）が変わってしまうため、公平性の鍵になります。

**ホライズン（horizon）／タイムステップ数**。何点・何秒先まで予測するか。これも揃えないと「短く予測して速いだけ」が混ざります。たとえば 64 ステップを 0.05 秒刻みで出せば 3.2 秒先までの軌道になります。

**疎結合（loose coupling）**。部品どうしが互いの内部に立ち入らず、決められた約束（インターフェース）だけでやり取りすること。新しいプランナを足しても既存コードを書き換えずに済む、という保守性につながります。逆に、各プランナの呼び出しがコードのあちこちに直書きされていると、1つ足すたびに全箇所を直す必要が生じ（密結合）、比較実験のたびにバグが混入します。

## 仕組みを詳しく
典型的な実装は、辞書ひとつと生成関数ひとつだけのごく小さなものです。一般化した形で示します。

```python
PLANNER_REGISTRY = {
    "gru": GRUPlanner,
    "flow_matching": FlowMatchingPlanner,
    "bezier": BezierPlanner,
}

def build_planner(planner_mode, **kwargs):
    if planner_mode not in PLANNER_REGISTRY:
        raise ValueError(
            f"Unknown planner_mode {planner_mode!r}. "
            f"Available: {sorted(PLANNER_REGISTRY)}."
        )
    return PLANNER_REGISTRY[planner_mode](**kwargs)
```

使い方は `build_planner("bezier", embed_dim=256, num_timesteps=64, num_signals=2, ...)` のように、キーと共通の構成引数を渡すだけです。知らないキーを渡すと「利用可能なキー一覧」を添えてすぐ例外になるので、タイプミスが静かに通ってしまう事故を防げます。これは地味ですが研究では重要で、`"flow_maching"`（スペルミス）のようなキーが黙ってデフォルトにフォールバックすると、半日かけて回した比較実験が全部ゴミになりかねません。

`**kwargs`（キーワード引数のまとめ受け）はそのまま各プランナの `__init__` に転送されます。レジストリ側で引数を選り分けないのが普通なので、「選んだプランナが必要とする引数を、呼び出す側が正しく渡す」という約束になります。ここはトレードオフで、簡潔さと引き換えに、引数の整合チェックは呼び出し側の責任になります。より厳密にしたい場合は、各プランナに「自分が必要とする引数」を宣言させ、レジストリ側で検証する設計（dataclass や pydantic を使う）もあります。

**登録の自動化（デコレータ方式）**。手で辞書に書き足すかわりに、クラス定義にデコレータを付けて自動登録する流儀が広く使われます。

```python
PLANNER_REGISTRY = {}

def register_planner(name):
    def deco(cls):
        PLANNER_REGISTRY[name] = cls
        return cls
    return deco

@register_planner("gru")
class GRUPlanner(BasePlanner):
    ...
```

この方式の利点は、プランナの定義と登録が同じ場所にまとまり、別ファイルの辞書を更新し忘れる事故が減ることです。Detectron2（Wu et al., 2019）や MMDetection（Chen et al., 2019, arXiv:1906.07155）といった大規模な検出ライブラリは、まさにこのデコレータ式レジストリでバックボーン・ヘッド・損失などを差し替え可能にしています。

このレジストリの威力が最大化されるのが**ベンチマーク**です。比較対象を `PLANNERS = ["gru", "flow_matching", "bezier"]` と並べ、まったく同一の構成（たとえば embed_dim=256, num_timesteps=64, num_signals=2 など）・同一入力テンソル・同一デバイスでループします。

```
for name in PLANNERS:
    if name not in PLANNER_REGISTRY:   # 未登録キーは静かにスキップ
        continue
    planner = build_planner(name, **CONFIG)
    measure(planner, fixed_input)      # latency / params / smoothness ...
```

ここで重要なのは、ベンチマーク対象に名前を並べても**レジストリに登録されていなければ実際には測られない**という点です。比較表に名前が見えても登録漏れだと黙ってスキップされることがあるので、「比較対象＝レジストリ登録済み」を必ず確認する、というのが実務上の注意です。研究コミュニティでは、こうした「測ったつもりで測れていない」問題を避けるため、ベンチマークの冒頭で「登録済みキー一覧」を出力し、比較対象との集合差分を明示することが推奨されます。

測る項目の典型例は次のとおりです。

- 推論レイテンシ: p50（中央値）, p99（ほぼ最悪値）, jitter（p99 と p50 の差＝ばらつき）→ [推論レイテンシ指標（p50 / p99 / jitter）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0684_inference-latency-p50-p99-jitter)
- パラメータ数（モデルの大きさ）と FLOPs（演算量）
- 構造的な滑らかさ: 加速度差分の分散 `Var(Δaccel)`（ジャーク相当）と曲率差分の分散 `Var(Δcurvature)` → [軌道スムーズネス指標（Var(Δaccel) / Var(Δcurvature)）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0707_trajectory-smoothness-metrics)

代表的な実測例（あるGPU上、学習前のランダム重み、batch=1, 64ステップ）はおおむね次のようになります。

| planner | params | p50 (ms) | Var(Δaccel) | Var(Δcurv) |
|---|---|---|---|---|
| gru | 約829K | 約50.3 | 約1.4e-04 | 約3.8e-04 |
| flow_matching | 約988K | 約13.9 | 約4.2e+00 | 約2.0e+00 |
| bezier | 約495K | 約0.63 | 約5.0e-06 | 約1.2e-06 |

レジストリのおかげで、この3手法を「キーを並べるだけ」で同条件比較できています。ここで研究者として外してはならない但し書きがあります。**これは学習前のランダム重みでの比較**なので、軌道の正確さ（後述のADE/FDE）や衝突率・閉ループ安定性はまだ比較できていません。これらには学習済みの重み・正解データ・シミュレータが要ります。レジストリは「公平に比較する箱」を提供するだけで、何が良いかの最終結論は学習後に出ます。逆に言えば、レイテンシ・パラメータ数・構造的滑らかさは重みに依存しにくいので、開発の初期段階から測れる、というのがこの設計の現実的な価値です。

## 手法の系譜と主要論文
レジストリは設計パターン（ソフトウェア工学）であり、特定の論文の手法ではありません。Factory／Registry の考え方自体は Gamma, Helm, Johnson, Vlissides の "Design Patterns: Elements of Reusable Object-Oriented Software"（1994）に整理された生成パターンの一種です。機械学習の文脈では、Caffe の "layer registry"、Detectron2（Wu et al., 2019）の `Registry` クラス、MMDetection（Chen et al., 2019）の `Registry` がこの設計を大規模に普及させ、いまや実験フレームワークの標準装備になっています。ここでは、レジストリが束ねる「プランナの流派」の系譜を描きます。

- **再帰型／自己回帰デコーダ（RNN・GRU・LSTM 系）**。Cho et al. が GRU を提案（EMNLP 2014, arXiv:1406.1078）。軌道生成では ChauffeurNet（Bansal et al., RSS 2019, arXiv:1812.03079）が代表で、模倣学習で人間の運転を真似しつつ軌道を1点ずつ逐次生成します。利点は時間的整合の取りやすさ、欠点は逐次計算で遅く、序盤の誤差が後半に伝播しやすい点（誤差累積＝exposure bias）。詳細は [GRU 軌道プランナ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0127_gru-trajectory-planner)。

- **ベジエ／多項式によるパラメトリック軌道**。少数の制御点から連続で微分可能な滑らかな曲線を引く古典手法です。Werling et al.（ICRA 2010）のフレネ座標系での5次多項式（quintic polynomial）による最小ジャーク軌道生成や、Bernstein 基底を使った軌道計画がこの系譜です。利点は「滑らかさを構造で保証」できること（学習前から低ジャーク）、欠点は表現できる形が制御点数に縛られ、急な回避のような鋭い動きを出しにくい点。

- **拡散・フローマッチング系の生成プランナ**。Lipman et al. の Flow Matching（ICLR 2023, arXiv:2210.02747）や Ho et al. の拡散モデル（DDPM, NeurIPS 2020, arXiv:2006.11239）を軌道生成に応用したもの。Diffusion Planner（Zheng et al., 2024）や MotionDiffuser（Jiang et al., CVPR 2023）などがあります。利点は多峰な（複数の妥当解がある）軌道分布を表現できること、欠点は推論コストと、構造的な滑らかさ保証が無い点（上表で Var が大きいのと整合）。

これら互いに前提の異なる流派を「同じ土俵で」比べたい、という研究上の動機こそ、レジストリパターンが存在する理由です。レジストリは流派そのものを発明したわけではなく、流派を交換可能にして比較を再現可能にするインフラを提供します。

## 論文の実験結果（定量データ）
レジストリそのものに固有の実験はありませんが、束ねる手法群の代表的な数値を、指標の意味とともに挙げます。

**ADE / FDE（軌道の正確さ）**。ADE（Average Displacement Error, 平均変位誤差）は予測軌道と正解軌道の各時刻のずれ（ユークリッド距離、メートル）を全時刻で平均した値、FDE（Final Displacement Error, 最終変位誤差）は最終時刻だけのずれです。小さいほど正確です。運転の End-to-End モデルの代表である UniAD（Hu et al., CVPR 2023, Best Paper, arXiv:2212.10156）は nuScenes 上で平均 L2 誤差を従来比で下げ（報告では 1.03m 前後の平均 L2、衝突率 0.31% 前後と、当時の ST-P3 などのベースラインより改善）、衝突率（collision rate, 予測軌道が他物体とぶつかる割合）も低減したと報告しています。VAD（Jiang et al., ICCV 2023, arXiv:2303.12077）はベクトル表現を使い、UniAD と同等以上の精度を保ちつつ推論を数倍高速化（VAD-Base でおおむね 4.5 倍程度、9〜10 FPS 級）したと報告しています。

**滑らかさ系の数値**。上のレジストリ比較表のとおり、ベジエの `Var(Δaccel)≈5.0e-06` はフローマッチングの `≈4.2e+00` より約100万倍小さく、構造的に圧倒的に滑らかです。フローマッチングはノイズから少しずつ軌道を作る仕組み上、ランダム重み段階では出力が暴れて分散が大きく出ます。

**アブレーション的な読み**。同条件比較なので、「レジストリのキーを差し替える」こと自体がアブレーション（ある要素を抜き差しして効果を測る実験）になります。たとえば bezier から flow_matching に替えると、レイテンシが約0.63ms→約13.9msに増え、滑らかさの分散が約6桁悪化します。逆に多峰性（複数の妥当な進路を表現する能力）が要る場面では生成系が要るはずで、この比較は「速度・滑らかさ vs 表現力」のトレードオフを一枚の表で可視化していると言えます。研究文献でも、こうした「同一バックボーン・同一入力でヘッドだけ差し替える」アブレーションは、寄与の切り分けに最も信頼できる方法とされており、レジストリはその実装を1行に縮める道具です。

## メリット・トレードオフ・限界
メリット

- 設定1行（キー文字列）でプランナを差し替えられ、同条件比較が容易になる。比較研究の再現性が上がる。
- 「使う側」と「作る側」が疎結合になり、新プランナの追加が既存コードに波及しない（保守性）。
- 未知キーで分かりやすく即エラーにし、利用可能な選択肢を提示できる（タイプミス事故の防止）。
- ベンチマークと相性が良く、公平な比較基盤になる。アブレーションが構造的に「キーの集合操作」へ落ちる。

トレードオフ・限界

- レジストリは引数を選り分けないため、選んだプランナに合う `**kwargs` を呼び出し側が正しく渡す必要がある（整合チェックの責務が外に出る）。
- 「比較できる箱」を提供するだけで、何が最良かは結局、学習と閉ループ評価をしないと分からない。レイテンシ・滑らかさだけで結論を急ぐのは危険。
- 共通インターフェースに合わせる制約があり、特殊な入出力を持つプランナ（たとえば不確実性つき多峰出力や、可変長ホライズン）は無理に型を合わせると本来の表現力を削いでしまうことがある。インターフェースを最大公約数に絞りすぎると、新手法が「箱に入らない」。
- ベンチマーク対象に名前が並んでいても、レジストリ未登録だと黙ってスキップされ、実際には比較されない。「比較した気になる」事故に注意。
- 文字列キーは型安全でない。リファクタリングでクラス名を変えてもキー文字列は自動で追従しないため、IDE の補完やコンパイル時チェックの恩恵を受けにくい。

研究上の未解決課題としては、「公平な比較とは何か」の定義そのものがあります。パラメータ数を揃えるべきか、推論コスト（FLOPs/レイテンシ）を揃えるべきか、表現力を揃えるべきか――揃える軸によって結論が変わるため、レジストリは比較の道具を提供しても、比較設計の妥当性は人間が論じ続ける必要があります。近年は同一 FLOPs・同一レイテンシ予算での比較（iso-FLOP, iso-latency）が公平性の基準として重視されつつあります。

## 発展トピック・研究の最前線
- **設定駆動の実験管理との統合**。Hydra や OmegaConf のような設定管理ツールと組み合わせ、レジストリのキーを設定ファイルから注入すると、スイープ（多数の設定を自動で回す実験）と相性が良くなります。実験管理ツール（MLflow, W&B）にキーをタグとして記録すれば、後から「どの手法がどの条件で勝ったか」を機械的に集計できます。
- **プラグイン化**。Python の entry points 機構で外部パッケージのプランナを動的にレジストリへ登録すると、コア実装を触らずに新手法を試せます。Detectron2 や MMDetection のレジストリはこの発想を大規模化したもので、サードパーティが本体を fork せずに新コンポーネントを足せる生態系を作りました。
- **比較の公平性メトリクスの研究**。同一 FLOPs・同一レイテンシ予算での精度比較（iso-FLOP, iso-latency 比較）は、近年のニューラルアーキテクチャ研究で重視されており、レジストリはこの種の制約付き比較の実装基盤になります。NAS（ニューラルアーキテクチャ探索）でも、探索空間を「レジストリに登録された候補の集合」として定義する設計が一般的です。
