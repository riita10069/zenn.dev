# Loss Scaling / GradScaler

## ひとことで言うと
Loss Scaling(ロススケーリング)は、fp16(16ビット浮動小数点)で学習するときに、勾配(パラメータをどちらへどれだけ動かすかを示す値)が小さすぎて0に潰れる「アンダーフロー」を防ぐテクニックです。損失(loss、予測の誤差を表す数)をわざと大きな数(例: 1024倍)に膨らませてから誤差を逆向きに伝えると、連鎖律の働きで途中の勾配も一斉に膨らみ、fp16 で表せる範囲に持ち上がります。PyTorch ではこれを自動でやる `GradScaler` があります。bf16 を使う場合は範囲が広いため不要です。

## 直感的な理解
fp16 という「目盛りの粗い、しかも目盛りの下限が高い物差し」で勾配を測ると、本当はわずかに存在する小さな勾配が「0」と読まれてしまう。すると「動かすべき方向がない」と誤判定され、その層は学べなくなります。Loss Scaling の発想は単純で、「測る前に対象を拡大コピーする」こと。1e-7 のような極小の勾配でも、1024倍に拡大してから測れば 1e-4 になり、物差しの目盛りに乗る。測り終えたら同じ倍率で縮めて元に戻せば、最終的な更新量は正しいまま、途中で消えるのだけを防げます。

なぜ「損失」を膨らませるのか。勾配を個別に拡大するのは大変ですが、損失はネットワークの出口の1点。そこを `S` 倍するだけで、逆伝播の連鎖律によって全層の勾配が自動的に `S` 倍になります。1か所いじるだけで全体に効く、という効率の良さがこの手法の肝です。

## 基礎: 前提となる概念
- 誤差逆伝播(backpropagation): 損失を出発点に、連鎖律(合成関数の微分のルール)を使って各パラメータの勾配を出口側から入口側へ順に計算する手続き。
- 勾配(gradient): 損失をそのパラメータで微分した値。学習はこの値に沿って `重み ← 重み - 学習率 × 勾配` で更新します。
- アンダーフロー: 値が小さすぎてその数値型で表せず、0に丸まる現象。fp16 では正の最小がおおむね 6e-5 で、それより小さい正の値は0になる(さらに小さい subnormal 領域はあるが精度が極端に落ちる)。
- オーバーフロー: 逆に値が大きすぎて表せず、無限大(inf)や非数(NaN)になる現象。fp16 の最大は 65504。
- 連鎖律: `損失を S 倍する` と、出口から入口まで掛け算で伝わる勾配がすべて `S 倍` になる、という性質。Loss Scaling の数学的な土台。

なぜ学習で勾配が小さくなるのか。深いネットワークでは、入口に近い層の勾配は多数の小さな係数の積になりがちで、1e-7 のような極小値が普通に現れます。fp16 の下限(約6e-5)を下回ると0に丸まり、その層が止まります。学習データに完璧に当てはまっていても、肝心の特徴抽出層が学べないと汎化しません。

## 仕組みを詳しく
逆伝播の前に、損失全体へ大きな係数 `S`(スケール係数、例: 1024 や 65536)を掛けます。

1. 通常どおり forward(順伝播)で損失 `L` を計算。
2. スケール: `L_scaled = L × S`。
3. `L_scaled` から逆伝播。連鎖律により、得られる勾配はすべて元の `S` 倍。
   - 元が 1e-7 の勾配は `1e-7 × 1024 ≒ 1.0e-4` になり、fp16 の範囲(6e-5 超)に収まって生き残る。
4. オプティマイザ更新の前にアンスケール(unscale): 勾配を `S` で割って元の大きさへ戻す。
   - これを忘れると `S` 倍だけ大きすぎる更新になり、学習が発散する。実装ミスの定番。
5. 戻した勾配で重みを更新。

数値例。`S=1024`、入口層の勾配 1e-7 は逆伝播中は 1.024e-4 として扱われ fp16 で生き残る。更新直前に 1024 で割り戻して 1e-7 に復元してから反映するので、更新量は正しいまま、途中のアンダーフローだけを回避できます。

`S` の選び方が難しい。小さすぎると勾配が消えるのを防げず、大きすぎると今度は勾配がオーバーフローして inf/NaN になります。良い `S` はモデル・データ・学習段階によって動くので、固定値を手で当てるのは面倒です。

そこで PyTorch の `torch.cuda.amp.GradScaler`(現在は `torch.amp.GradScaler`)が動的ロススケーリング(dynamic loss scaling)で自動調整します。
- スケール後の勾配に inf/NaN が出たら「`S` が大きすぎた」と判断し、そのステップの更新をスキップして `S` を例えば半分にする。
- 一定回数(例: 2000ステップ)連続で問題が出なければ「もっと上げられる」と判断し `S` を倍にする。

こうして `S` は「inf を出さないギリギリ大きい値」に自動収束し、人手調整が要りません。実装上の既定値はおおむね初期 `S=65536`(=2^16)、inf 検出時の縮小率 0.5、増大インターバル 2000 ステップ、増大率 2.0 で、いずれも2のべき乗です。2のべき乗にする理由は、浮動小数点を2のべき乗で掛け割りすると仮数ビットがまったく変わらず指数だけがずれるため、スケーリング自体が新たな丸め誤差を一切持ち込まないからです(`x × 2^k` は完全に可逆)。

典型的な使い方。
```python
scaler = torch.amp.GradScaler()
with torch.autocast(device_type="cuda", dtype=torch.float16):
    loss = criterion(model(x), y)
scaler.scale(loss).backward()        # 勾配を S 倍して逆伝播
scaler.unscale_(optimizer)           # クリッピング前に元スケールへ戻す(任意)
clip_grad_norm_(model.parameters(), 1.0)
scaler.step(optimizer)               # inf/NaN がなければ更新、あればスキップ
scaler.update()                      # S を自動調整
```

なぜ bf16 では不要か。bf16 は fp32 と同じ広い指数範囲を持つので、1e-7 でもそのまま表せます。アンダーフローしないので持ち上げる必要がなく、Loss Scaling も GradScaler も使いません。bf16 対応GPUが手元にあるなら、fp16 + Loss Scaling より bf16 を選ぶのが定石です(理由は [Mixed Precision Training (AMP)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0133_mixed-precision-training) 参照)。

## 手法の系譜と主要論文
- Micikevicius ら「Mixed Precision Training」(2018, ICLR, arXiv:1710.03740, NVIDIA & Baidu)。混合精度学習の元論文で、Loss Scaling を中核の工夫の1つとして提案。最初は「全勾配の最大値を見て、オーバーフローしない範囲で最大の固定 `S` を選ぶ」やり方を示しました。
- 動的ロススケーリング(NVIDIA Apex / PyTorch AMP として実装が普及)。固定 `S` の調整の手間を、「inf が出たら下げ、しばらく出なければ上げる」自動制御に置き換えたもの。論文の手法を実務的に使いやすくした発展で、現在の標準です。
- Kalamkar ら「A Study of BFLOAT16 for Deep Learning Training」(2019, arXiv:1905.12322)。bf16 なら Loss Scaling 自体が要らないと示し、「Loss Scaling は fp16 の狭い範囲を補うための補助輪であって、本質ではない」という位置づけを明確にしました。bf16 普及の一因です。

## 論文の実験結果(定量データ)
- Micikevicius et al.(2018)。物体検出モデル SSD(Single Shot Detector)を fp16 で素朴に学習すると、Loss Scaling なしでは勾配の大半が fp16 の表現範囲の下限を下回って0に丸まり、学習がほぼ進まないことをヒストグラムで可視化しました。一方で損失をわずか 8倍するだけで fp32 と同等の mAP(平均適合率、検出の当たり具合の指標)まで回復したと報告。指標 mAP の意味は「検出ボックスとクラスがどれだけ正しいか」で、ここが fp32 並みに戻ったことが「Loss Scaling が機能した」証拠です。
- アブレーションの含意。同論文は、3本柱(fp32 マスターウェイト・Loss Scaling・一部 fp32 演算)のうち、特に勾配が小さくなりやすいタスク(検出・一部の言語モデル)で Loss Scaling を外すと精度が崩れることを示しました。逆に勾配が比較的大きいタスクでは Loss Scaling なしでも動く場合があり、必要性はモデル依存だと分かります。
- bf16 側の対照実験(Kalamkar et al., 2019)。同じ系統のタスクで bf16 は Loss Scaling なしに fp32 同等精度へ到達。これにより「Loss Scaling が必要かどうかは数値型の指数範囲で決まる」という因果が裏づけられました。

数値の意味。「8倍で回復」というのは、たった3ビットぶん(2^3=8)持ち上げるだけで、消えていた勾配が表現範囲に収まる、ということ。fp16 の範囲の狭さが、ほんの数ビットで明暗を分けるほどタイトだと示す具体例です。

## メリット・トレードオフ・限界
メリット
- fp16 のアンダーフローを防ぎ、小さい勾配を持つ層もきちんと学習できる。
- GradScaler を使えばスケール係数の調整が自動化され、人手チューニングが要らない。
- fp16 の高速・省メモリの恩恵を、精度をほぼ落とさず受けられる。

トレードオフ・限界
- bf16 が使えるなら、そもそも不要(余計な複雑さが増えるだけ)。
- 固定スケールを手で選ぶ場合は調整が難しく、大きすぎればオーバーフロー、小さすぎれば効果なし。
- inf/NaN が出たステップはスキップされるため、動的スケーリングが安定するまでの初期に学習がわずかに遅れる/最初の数ステップが捨てられることがある。
- アンスケールを忘れると更新量が `S` 倍ずれる実装ミスが起きやすい。クリッピングや勾配の検査をする場合は「アンスケール後の勾配」で行う必要がある。
- 単一の `S` を全パラメータに使うため、層ごとに勾配スケールが大きく異なるモデルでは、ある層に最適な `S` が別の層には不適切になる(層ごとスケーリングの研究はあるが標準ではない)。

## 発展トピック・研究の最前線
- per-tensor / per-layer スケーリング。fp8 学習では仮数2〜3ビットと範囲・精度がともに極端に狭いため、損失全体に1つの `S` を掛ける従来のグローバル方式では足りず、テンソルごと・層ごとに別々のスケールを動的に管理する手法が必須になります。実際には各テンソルの絶対値の最大(amax)の履歴を数ステップぶん記録し、それをもとに「次ステップのスケール」を先回りで決める delayed scaling が使われます。これは「inf が出てから下げる」事後対応より破綻しにくく、Loss Scaling の考え方をより細かい粒度・予測的な制御へ拡張したものです。
- スケール係数の履歴を使った予測制御。inf が出てから下げる「事後対応」ではなく、勾配統計から先回りして `S` を決める研究もあります。
- bf16 の普及により、汎用学習では Loss Scaling は「歴史的・教育的に重要だが日常的には bf16 で回避」という位置づけに移りつつあります。一方で fp8 等のさらなる低精度では、Loss Scaling 的なスケール管理が再び主役に戻っており、概念としての重要性は続いています。
