# 学習可能なカメラ位置埋め込み

## ひとことで言うと

カメラ間アテンション（[CrossAttentionViewFusion（カメラ間アテンション融合）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0952_cross-camera-attention-fusion)）では、複数台のカメラ特徴を「トークン（順序の区別がない要素）の集合」として扱います。このままだとモデルは「どれが前方カメラでどれが後方カメラか」を区別できません。そこで、カメラごとに用意した小さな学習可能なベクトルを各カメラ特徴に足し込み、「これは何番カメラの特徴だ」という識別情報を持たせます。これが学習可能なカメラ位置埋め込み（learnable view position embedding）です。

## 直感的な理解

トランプを7枚、裏向きに机に並べる場面を想像してください。裏向きのままでは、左から3枚目を入れ替えても誰も気づきません。アテンションが「順序不変」というのはまさにこの状態で、入力の集合に対して順番を入れ替えても同じ結果を返します。

ところが運転では、7台のカメラには明確に役割があります。前方カメラは進行方向の障害物、側方カメラは合流時の死角、後方カメラは車線変更時の追従車を捉えます。「前方カメラの情報を後方より重視する」ことを学習させたいのに、モデルが7台を「区別のつかない7個」としか見られないなら、役割に応じた使い分けは原理的に学べません。

そこで各カメラに「名札」を付けます。名札はカメラ0用、カメラ1用…と7種類あり、対応する特徴に足し込みます。名札の中身は最初ランダムですが、訓練を通じて「このカメラは前方を担う」「このカメラは死角を見る」といった役割を反映する値へと自動で育っていきます。この名札が学習可能なカメラ位置埋め込みです。

## 基礎: 前提となる概念

専門用語を順に噛み砕きます。

- 埋め込み（embedding）= 何か（単語・位置・カメラ番号など）を表す固定長の数値ベクトルです。ここでは「カメラ番号 → 256次元ベクトル」という対応関係です。
- 学習可能（learnable）= そのベクトルの中身を、最初はランダムに初期化し、訓練を通じて誤差逆伝播（出力の誤差から各パラメータをどう動かすかを逆算する仕組み）で最適な値に更新していくことです。
- 順序不変（permutation invariant）= 入力の順番を変えても出力が変わらない性質です。アテンションそのものはこの性質を持ち、だからこそ順序や位置の情報を別経路で注入する必要があります。
- 位置エンコーディング / 位置埋め込み（positional encoding / embedding）= 順序不変なアテンションに「どこの要素か」を教えるために足し込む情報です。固定式（数式で決める）と学習式（パラメータとして学ぶ）があります。
- ブロードキャスト（broadcasting）= 形の合わない次元を自動で繰り返して合わせるテンソル演算の仕組みです。`[B*H*W, 7, 256] + [1, 7, 256]` のように先頭次元が1のものは、相手に合わせて自動で複製されて加算されます。

なぜアテンションは順序不変なのか。アテンションは各トークンの query と全トークンの key の内積で重みを決め、その重みで value を合計します。この計算はトークンの並び順を一切参照しません。文章なら「単語の順番」が意味を左右するのに、アテンション自体は順番を見ない。だから元祖の Transformer も位置エンコーディングを足して順番情報を補いました。同じ問題がカメラ間アテンションでも起き、カメラの識別を別経路で与える必要があります。

## 仕組みを詳しく

典型的な実装は、カメラ台数 V ぶんの埋め込みベクトルを用意し、アテンション直前に足し込みます。V=7、埋め込み次元 C=256 の例で追います。

ステップ1: パラメータの用意。
```
view_embed = Parameter( randn(1, V, C) * 0.02 )   # 形 [1, 7, 256]
```
平均0・標準偏差およそ0.02の小さなランダム値で初期化した `[1, 7, 256]` のテンソルを、学習対象のパラメータとして登録します。`*0.02` と小さくするのは、初期の埋め込みが大きすぎて元の画像特徴を上書きしてしまわないようにするためです。位置情報は補助であって主役ではないので、最初は控えめにしておき、必要なぶんだけ学習で大きくしていく設計です。`view_embed[0,0]` が「カメラ0の識別ベクトル」、`view_embed[0,1]` が「カメラ1の識別ベクトル」…と、7台ぶんの固有ベクトルを保持します。

ステップ2: 加算。
```
x = x + view_embed   # x: [B*H*W, 7, 256], view_embed: [1, 7, 256]
```
アテンション直前のトークン `x`（各空間位置で7台のトークンを並べたもの）に `view_embed` を足します。ブロードキャストにより、すべての空間位置・全バッチに同じ7台ぶんの埋め込みが加算されます。加算後、各トークンは「もとの画像特徴 + そのカメラの識別ベクトル」になります。アテンションはこの合成ベクトルから query/key を作るので、「同じカメラ同士」や「特定カメラ」を識別できるようになります。

ステップ3: 学習。誤差逆伝播のとき `view_embed` も勾配を受け取り更新されます。訓練が進むと、各カメラの埋め込みは「そのカメラを他と区別し、役割に応じた扱いを引き出す」ように最適化されます。

なぜ連結ではなく「足す」のか。位置エンコーディングは伝統的にトークン特徴へ加算します。連結すると次元が増えて以降の層を作り直す必要がありますが、加算なら次元が変わらず、アテンション層をそのまま使えます。トレードオフとして、加算は元の特徴と位置情報を同じ次元空間で混ぜるため、強すぎると元情報を壊します（だから初期値を小さくします）。

before / after の効果イメージ：
- before（埋め込みなし）: アテンションは7トークンを「同じ7個」と見る。カメラ0と1を入れ替えても出力が不変。「前方を重視」が原理的に学べない。
- after（埋め込みあり）: 各トークンが固有の識別を持つ。アテンションが「3番カメラの key」を狙って重みを付けられる。カメラの役割を学習できる。

## 手法の系譜と主要論文

位置埋め込みは Transformer 系の中核技術で、固定式と学習式の両方が広く使われてきました。

- Vaswani et al., "Attention Is All You Need"（NeurIPS 2017、arXiv:1706.03762）。アテンションが順序不変であることを指摘し、sinusoidal（正弦波）による固定の位置エンコーディングを提案しました。動機は系列の順序情報をアテンションに与えること。固定式は学習データに依存せず、学習時より長い系列にも外挿しやすいのがメリットです。

- Devlin et al., "BERT"（NAACL 2019、arXiv:1810.04805）/ Dosovitskiy et al., "ViT"（ICLR 2021、arXiv:2010.11929）。位置埋め込みを学習可能パラメータにしました。動機はデータからより適した位置表現を学べること。トレードオフは、学習で見た位置数を超える長さへ外挿しにくいこと。カメラ位置埋め込みはこの「学習可能位置埋め込み」と同じ流儀を、カメラ番号という有限・固定の集合（例えば V=7）に適用したものです。カメラ台数は固定なので外挿の問題は起きにくいという利点があります。

- Carion et al., "DETR"（ECCV 2020、arXiv:2005.12872）。物体クエリやエンコーダ特徴に位置埋め込みを付与し、順序不変なアテンションに空間情報を与えました。「アテンション前に識別情報を足す」構図の代表例の一つです。

- Liu et al., "PETR"（ECCV 2022、arXiv:2203.05625）。3Dカメラ幾何（各画素に対応する3D点の座標）から位置埋め込みを生成し、多視点特徴に付与しました。動機は明示投影なしでカメラ間の3D関係をモデルに伝えること。効果は多視点3D検出の精度向上です。これは「どのカメラか」だけでなく「そのカメラが空間のどこを向いているか」までを埋め込みに織り込む、より高度な方式です。単純な学習可能ベクトルは前者（カメラ識別）までしか担えません。

これらに共通する論理は「アテンションは順序を見ないので、別経路で位置・識別情報を注入する必要がある」ことです。固定式 vs 学習式のトレードオフは、外挿性（固定式が有利）と適合性（学習式が有利）の綱引きとして整理できます。カメラ識別という用途では、要素数が少なく固定なので、シンプルな学習可能埋め込みが扱いやすい選択になります。

## 論文の実験結果（定量データ）

位置埋め込みの有無や種類が効くことは、複数の論文でアブレーション（ある要素を抜く実験）として報告されています。

- ViT（Dosovitskiy et al., 2021）のアブレーションでは、位置埋め込みを「全く付けない」場合、画像分類精度（ImageNet top-1 accuracy、上位1候補が正解の割合）が位置埋め込みありに比べて数ポイント低下すると報告されています。一方で、1次元の学習可能埋め込みと2次元のものの差はごく僅か（およそ0.1ポイント程度）で、「位置情報があること自体が支配的に重要で、その表現形式の違いは小さい」ことが示されました。これは「カメラ識別の有無は本質的に効くが、ベクトルの作り方の細部は二次的」という直感と整合します。

- Transformer 原論文では、sinusoidal な固定位置エンコーディングと学習可能位置埋め込みを機械翻訳タスク（WMT 英独翻訳、指標は BLEU スコア。BLEU は翻訳の n-gram 一致度で高いほど良い）で比較し、両者の BLEU はほぼ同等だったと報告しています。固定式を選んだ理由は学習時より長い系列に外挿できる利点でした。

- PETR の論文では、3D幾何由来の位置埋め込み（3D Position Embedding）を取り除くと nuScenes 検証セットの NDS（位置・向き・速度などを統合した総合指標、高いほど良い）が明確に低下することが報告されており、「カメラの識別だけでなく幾何的な位置情報まで埋め込むと、多視点3D検出で追加の利得がある」ことを示しています。逆に言えば、単純な学習可能識別埋め込みはこの幾何ぶんの利得は取り切れない、という限界の定量的裏付けにもなっています。

数値の意味を補足すると、accuracy は分類の正解率、BLEU は翻訳一致度、NDS は3D検出の総合スコアで、いずれも「位置・識別情報を与えると向上する」という一貫した傾向が、異なるタスクをまたいで観測されている点が重要です。

## メリット・トレードオフ・限界

メリット
- 順序不変なアテンションにカメラの識別情報を与え、カメラの役割に応じた学習を可能にする。
- 加算なので次元が変わらず、既存のアテンション層をそのまま使える（実装が軽い）。
- 学習可能なので、データから各カメラに最適な識別表現を獲得できる。
- カメラ台数は固定なので、学習式位置埋め込みの弱点である「外挿」の問題が起きにくい。

トレードオフ・限界
- 幾何は持たない。単なる学習ベクトルなので「カメラが空間のどこを向いているか」は直接表せません。PETR のような幾何由来の埋め込みや、投影を使う BEV 方式の方が3D整合は強くなります。
- 加算が強すぎると元の画像特徴を上書きしてしまうため、小さく初期化する配慮が必要です。
- カメラ構成（台数・配置）が変わると埋め込みを作り直して再学習が要ります。データセット間で台数が違うと使い回しにくい。
- 識別はできても「なぜそのカメラを重視するか」は解釈しにくい（学習で得た抽象ベクトルのため）。
- 未解決の課題として、カメラ台数が異なる車種間で位置埋め込みをどう共有・転移するか、幾何情報（外部パラメータ）を学習埋め込みにどこまで明示注入すべきか、といった点が研究上のオープンな論点です。

## 発展トピック・研究の最前線

位置埋め込みの研究はいくつかの方向に発展しています。1つは相対位置エンコーディングで、絶対的な位置ではなくトークン間の相対距離を埋め込む方式（Transformer-XL や T5 が採用）です。もう1つは回転位置埋め込み（RoPE, Rotary Position Embedding）で、query/key を位置に応じて回転させることで相対位置を内積に自然に織り込み、長系列への外挿性を高めます。マルチカメラ知覚の文脈では、PETR の幾何由来埋め込みをさらに発展させ、深度や ego 運動まで埋め込みに織り込む手法（PETRv2 など）が研究されています。「識別だけでなく幾何・時間まで位置表現に統合する」のが現在の流れです。
