# BEV×OSM クロスアテンション融合

## ひとことで言うと

カメラ画像から作った「真上から見た特徴マップ(BEV、Bird's-Eye-View)」の各マス目が、地図(OSM)から作ったトークン列を見にいって、関係する地図情報を吸い上げる仕組みです。このとき cross-attention(クロスアテンション、片方が自分に必要な情報をもう片方から取りに行く注意機構)を使います。BEV のマス目が数万個に及ぶ大きなグリッドでも GPU メモリを爆発させないよう、attention の中間行列を実体化しない省メモリ実装を組み合わせるのが要点です。

## 直感的な理解

BEV 特徴は画像のようなグリッドで、各マス目が「自車から見て前方何メートル・左何メートルの地点」に対応します。一方、地図トークンは「この交差点」「この道路」といった地図要素を表す可変長の集合です。両者は形も性質も違います。BEV は規則正しい格子、地図トークンは順序のないリスト。これを混ぜたいのですが、単純に足し算はできません。位置が一対一に対応していないからです(地図要素は数百個、BEV マス目は数万個で、個数も並びも違う)。

そこで cross-attention を使います。発想は「BEV の各マス目が、自分に関係しそうな地図トークンを選んで重み付きで取り込む」ことです。たとえば交差点の真上にあるマス目は、近くの交差点トークンを強く見て、遠くの道路トークンは弱く見る、という重み付けを学習で獲得します。人間が地図を見て「今いる場所のまわりだけ注目する」のと同じです。attention は「誰がどれを見るか」を固定の対応表ではなくデータから学ぶので、位置がずれていても意味的に近いものを結びつけられます。

ただし素朴に実装すると深刻なメモリ問題が起きます。後述するように、マス目の数とトークンの数の積の大きさの重み行列を一度メモリに作ってしまうからです。これを回避するのがこのモジュールの技術的な核心です。

## 基礎: 前提となる概念

attention(注意機構)は、query(問い合わせる側)・key(問い合わせられる側の見出し)・value(実際に取り出す中身)の3つで動きます。各 query について、すべての key との類似度(内積)を計算し、softmax(合計が1になるよう正規化する関数)で重みに変え、その重みで value を加重平均します。式で書くと次の通りです。

```
Attention(Q, K, V) = softmax(Q Kᵀ / √d) V
# Q: (query数, d)   各 query のベクトル
# K: (key数,   d)   各 key のベクトル
# V: (key数,   d)   各 value のベクトル
# d : ヘッドあたりの次元(√d で割るのは内積が大きくなりすぎて softmax が飽和するのを防ぐため)
```

self-attention は Q・K・V がすべて同じ系列から来る場合、cross-attention は Q が一方(ここでは BEV)、K・V がもう一方(ここでは地図トークン)から来る場合を指します。翻訳で「出力文の各単語が入力文の単語を見る」のと同じ構図で、ここでは「BEV マス目が地図要素を見る」関係です。

multi-head attention(多ヘッド注意)は、上の計算を複数の「ヘッド」に分けて並行に行い、結果を結合する方式です。ヘッドごとに違う観点(近さ、種別、方向など)で注目できるようになります。ヘッド数が8、全体次元が256なら、各ヘッドの次元は 256/8 = 32 です。各ヘッドが異なる部分空間に射影されるので、1つの attention で複数の関係を同時に捉えられます。

ここで問題なのが `Q Kᵀ` の大きさです。これは `(query数 × key数)` の行列で、BEV のマス目が 450×300 = 135,000 個、地図トークンが N 個なら `135,000 × N` のサイズになります。N が数百でもこれは数千万要素で、ヘッドとバッチを掛けるとすぐに GPU メモリが足りなくなります。たとえば N=512、8ヘッド、バッチ2、float32 なら 135,000 × 512 × 8 × 2 × 4 バイト ≈ 4.4 GB を中間スコア行列1つだけで消費し、softmax 後の行列でさらに同量、勾配計算用に保持する分も加わって簡単に out-of-memory になります。

この問題に対する答えが FlashAttention(Dao et al., NeurIPS 2022, arXiv:2205.14135)です。attention を小さなブロックに区切って(タイリング)、巨大な `Q Kᵀ` 行列を GPU の遅い主メモリ(HBM)に丸ごと書き出さずに、高速なオンチップメモリ(SRAM)上でブロックごとに計算しながら softmax を逐次更新(online softmax)します。結果として中間行列を実体化しないのでメモリ使用量が query 数に対して線形になり、メモリ往復が減るぶん同時に高速化もされます。PyTorch には `F.scaled_dot_product_attention`(略称 SDPA)という標準関数があり、状況に応じて FlashAttention などの省メモリ backend を内部で自動選択します。後継の FlashAttention-2(Dao, 2023, arXiv:2307.08691)は並列化と作業分割を改良し、さらに高速化しています。

## 仕組みを詳しく

典型的な BEV×OSM 融合モジュールの入出力は次の通りです。

```
入力:
  image_bev:        (B, 256, H, W)   カメラ由来 BEV(query になる)
  osm_tokens:       (B, N, 256)      地図トークン(key/value になる)
  key_padding_mask: (B, N)           True=パディング(実体がない)
出力:
  (B, 256, H, W)                     地図文脈を足し込んだ BEV(形は不変)
```

処理を段階的に分解します。

第1に query 化です。`image_bev` を `(B, H*W, 256)` に並べ替え(flatten し)、H*W 個のマス目それぞれを1つの query にします。450×300 なら 135,000 query です。

第2に null トークンの前置です。`osm_tokens` の先頭に、学習可能な「地図情報なし」を表すトークンを1個くっつけて `(B, N+1, 256)` にします。これは [ゼロ初期化ゲートと学習可能 null トークン](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1054_zero-init-gate-null-token) で詳述しますが、地図要素がゼロのサンプルでも attention が必ず1個は有効な key を持てるようにし、softmax の分母がゼロになって NaN(無効値)が出るのを防ぐためです。

第3にマスク作成です。`key_padding_mask` を反転して「True=参加してよい key」にし、null トークンの位置は常に有効として先頭に足し、ヘッド方向と query 方向へブロードキャスト(同じ値を全ヘッド・全 query に複製)して `(B, 1, 1, N+1)` の attention マスクを作ります。パディングされたトークンはこのマスクで attention 計算から外れます(該当スコアを −∞ にして softmax で重みゼロにする)。

第4にメモリ効率の良い attention です。query/key/value をヘッド(たとえば8個)に分け、SDPA を呼びます。

```
attn = F.scaled_dot_product_attention(qh, kh, vh, attn_mask=attn_mask)
# qh: (B, 8, H*W, 32)   8ヘッド, 各ヘッド head_dim = 256/8 = 32
# kh, vh: (B, 8, N+1, 32)
```

SDPA が FlashAttention 系の backend を選ぶことで、`135,000 × (N+1)` の重み行列を一度に全部メモリへ作らずに計算します。これが「数万 query の大グリッドを処理できる」理由の核心です。なお、attn_mask を渡すと backend 選択が制約される(マスク対応 backend に限られる)ことがあるため、可変長対応と省メモリを両立させる実装上の注意が要ります。

第5に出力射影と FFN です。attention 出力を出力射影層で元の次元に戻し、`h = attn + FFN(LayerNorm(attn))` を計算します。FFN(Feed-Forward Network、全結合2層で間に非線形を挟む)は各トークンを個別に非線形変換する小さなネットで、LayerNorm(層正規化)は値のスケールをそろえて学習を安定させます。

第6にゲート付き残差です。最後に次のように出力します。

```
out = q + alpha * h
# alpha: チャンネルごとの学習可能パラメータ。ゼロ初期化
```

`alpha` はチャンネル(256次元の各成分)ごとの学習可能パラメータで、ゼロで初期化されます。つまり学習開始時は `alpha=0` なので `out = q` となり、地図を足していないのと完全に同じ出力です。学習が進むにつれ、地図が役立つチャンネルだけ `alpha` が少しずつ持ち上がり、必要なぶんだけ地図情報を取り込んでいきます。これも [ゼロ初期化ゲートと学習可能 null トークン](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1054_zero-init-gate-null-token) で扱います。

before/after を数値で言うと、入力 BEV が `(2, 256, 450, 300)` なら出力も同じ `(2, 256, 450, 300)` で、形は一切変わりません。変わるのは中身で、学習が進めば各マス目に地図の文脈が足し込まれます。形が不変なので、このモジュールは既存パイプラインの BEV 特徴の途中に挿し込むだけで使え、後段を一切改造せずに済みます。

## 手法の系譜と主要論文

別ソースの情報を空間特徴へ attention で注入する発想は、Transformer の登場から一貫した流れの上にあります。

Attention Is All You Need(Vaswani et al., NeurIPS 2017, arXiv:1706.03762)は、query/key/value による attention だけで系列間の関係を学習する Transformer を提案しました。本モジュールの cross-attention と SDPA はこの定式化そのものです。再帰(RNN)や畳み込みを使わず attention で長距離の依存関係を一度に捉えられること、そして並列計算しやすいことが、後のあらゆる応用の基礎になりました。

BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)は、BEV のグリッド query が複数カメラ画像の特徴へ attention して BEV 表現を作る方式を確立しました。動機は、多視点画像という異なるソースを BEV という共通空間に集約することでした。BEV×OSM 融合は、この「グリッドを query にする(query-as-grid)」発想を引き継ぎ、見にいく相手を画像特徴から地図トークンに替えたものと位置づけられます。BEVFormer は deformable attention(参照点まわりの少数の位置だけ見る省計算な attention)を使って計算を抑えていますが、地図トークンのように位置が不定形な集合相手では、全トークンを見るフル attention を省メモリ backend で回す構成が選ばれることがあります。

Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159)は、各 query が全位置ではなく学習した少数のサンプリング点だけを見る deformable attention を提案し、計算量を大幅に削減しつつ収束を速めました。BEVFormer の効率はこの技術に支えられています。地図トークン側の位置が画像のように規則的でない場合はサンプリング点の定義が難しく、フル attention が選ばれる理由になります。

FlashAttention(Dao et al., NeurIPS 2022, arXiv:2205.14135)と FlashAttention-2(Dao, 2023, arXiv:2307.08691)は、前述の通り attention を tiling して巨大な中間行列を GPU 主メモリに書き出さずに計算する手法で、メモリを query 数に対して線形に抑えます。大グリッドの query を扱えるのはこの種の backend のおかげです。

SDTagNet(Immel et al., 2025, arXiv:2506.08997)は、融合される側の OSM トークンの作り手です。SDTagNet 自体は地図トークンを地図構築タスクに使いますが、「カメラ BEV に地図トークンを cross-attention で混ぜる」という統合方法は、運転モデル側で別途設計される部分で、上記の attention 系譜と BEVFormer 的な query-as-grid を組み合わせて実装されます。

## 論文の実験結果(定量データ)

attention の省メモリ実装の効果は FlashAttention の報告が分かりやすいです。標準実装に比べて学習を高速化し(報告では BERT-large の学習で約15%、GPT-2 で最大約3倍の壁時計時間短縮、長系列ではさらに大きい)、メモリ使用量を系列長に対して2次から線形へ削減しました。FlashAttention-2 はさらに GPU 利用効率を高め、A100 で理論ピーク性能の50〜70%前後に達したと報告されています。これにより従来 out-of-memory で扱えなかった長系列・大 query 数を扱えるようになりました。BEV のマス目を数万 query 扱えるのは、まさにこの線形メモリ特性の恩恵です。

BEVFormer は nuScenes の3次元物体検出で、当時のカメラのみ手法を上回る NDS(nuScenes Detection Score、検出の各種誤差を総合する指標。高いほど良い)を達成し、おおよそ 0.56 NDS 前後を報告して BEV query 方式の有効性を示しました。Deformable DETR は COCO 物体検出で、元の DETR が必要とした学習エポック数を約10分の1に短縮しつつ同等以上の AP を達成し、attention のサンプリング局所化が収束と効率に効くことを示しました。地図情報を cross-attention で融合する研究では、地図なしベースラインに対して地図ありが motion forecasting や planning の誤差(たとえば軌道の平均変位誤差 ADE / 最終地点誤差 FDE、いずれもメートル単位で小さいほど良い)を下げることが繰り返し報告されています。BEV×OSM 融合は地図枝を後付けする構成のため、定量効果はゲート(`alpha`)が学習で持ち上がって初めて現れる点に注意が必要です。

## メリット・トレードオフ・限界

メリットは、グリッド(BEV)と可変長トークン(地図)という異種の表現をきれいに融合できること、SDPA の省メモリ backend で 450×300 級の大グリッドでも巨大な重み行列を実体化せず処理できること、ゼロ初期化ゲートで「最初は地図なしと同一」から安全に学習を始められること、そしてパディングトークンをマスクで除外でき、サンプルごとの地図要素数の違いに対応できることです。形が不変なので既存パイプラインへの挿し込みが容易な点も実務上大きい利点です。

トレードオフと限界としては、cross-attention は query 数が多いほど(中間行列を実体化しなくても)計算量自体は依然 query×key に比例して高いこと、null トークンやゲートなどの安全装置のぶん実装が複雑になること、そして効果は学習がゲートを持ち上げて初めて出るため、地図が役に立たない状況ではゲートがほぼゼロのまま計算だけが無駄になりうることが挙げられます。研究上の課題としては、フル attention の計算コストを deformable attention のような局所化でさらに削れるか、地図と BEV の位置合わせ誤差が大きいときに attention が誤った対応を学ばないか(自車測位がずれると交差点トークンを別の場所に結びつけてしまう)、といった頑健性の検証が残ります。

## 発展トピック・研究の最前線

融合の効率化方向では、全トークンを見るフル attention の代わりに、各 BEV マス目の近傍トークンだけを見る局所 attention や deformable attention を使い、計算を query 数に依存しにくい形へ抑える研究があります。表現統合の方向では、地図・カメラ・LiDAR を1つの大きな Transformer の中で共通トークンとして扱い、ソースごとの個別融合モジュールを廃する end-to-end な設計(統一トークン空間)が進んでいます。また、ゲート付き残差で枝を後付けする設計は地図に限らず一般的な拡張パターンで、その安定化の理論的理解(なぜ恒等から始めると学習が安定するか)も研究の対象です。位置合わせ誤差への頑健性については、attention の参照位置自体を学習で補正する、あるいは地図を確率的な事前分布として扱い観測との整合度で重みを下げる、といった方向が探られています。
