# Waymo Open Perception v2 (modular)

## ひとことで言うと
Waymo Open Perception v2 は、自動運転企業 Waymo が公開している大規模な知覚(センサー認識)データセットの、新しい配布形式です。中身のセンサーデータやラベルは初代(v1)と同等の品質・規模ですが、データを「種類ごとに分けた列指向の部品ファイル(Apache Parquet 形式)」に再編成したことで、必要な部分だけをダウンロードして使えるようになりました。巨大な全部入りファイルを丸ごと落とす必要がなくなった、というのが v2 の本質的な改良です。

## 直感的な理解
百科事典を全巻まとめて買う場面を想像してください。「鳥について調べたいだけ」なのに、地学も歴史も化学も全部入った重い全集を買って家まで運ばないといけないとしたら不便です。初代の Waymo データはこれに近く、ある20秒の走行を扱いたいだけでも、カメラ・LiDAR・全ラベルが一塊になった巨大ファイル(全体で数テラバイト規模)を丸ごと取得して展開する必要がありました。研究の入り口が重く、ちょっと試すこともしづらかったのです。

v2 は、この全集を「鳥の巻」「地学の巻」のように主題ごとの薄い冊子に分け、欲しい巻だけ取り寄せられるようにしたものです。「カメラ画像だけ欲しい」「3次元ラベルだけ欲しい」という研究者が、その部品だけを落として手早く実験を始められます。データの中身そのものは変わっていないのに、配り方を変えるだけで研究の参入障壁が劇的に下がる、という発想の改良です。

## 基礎: 前提となる概念
前提となる言葉を、知らない人向けに噛み砕きます。

- LiDAR(ライダー): レーザー光を周囲に飛ばし、跳ね返りまでの時間から距離を測るセンサーです。出力は点群(空間に散らばった大量の3次元点の集合。各点は x, y, z 座標を持つ)で、物体の立体形状や周囲の距離を直接表します。Waymo は中距離用1台+近距離用4台の計5 LiDAR を積みます。
- レンジイメージ(range image): LiDAR の点群を、行=レーザーの仰角、列=方位角、画素値=距離、という2次元画像の形で表したもの。点をばらまく代わりに格子に整列させた表現で、Waymo の LiDAR データはこの形で保存されます。
- 3次元物体検出(3D object detection): 点群やカメラ画像から、周囲の物体を立体の箱(3D bounding box。中心位置・縦横高さの寸法・向き=heading を持つ箱)で囲み、種類(車・歩行者・自転車・標識)を当てるタスクです。安全運転には周囲を立体で正確に把握することが不可欠です。
- 物体追跡ID(tracking ID): 同じ物体を時間をまたいで同一だと認識するための番号です。フレームごとに別物として検出すると軌道が分からないため、同じ車にはずっと同じIDを振ります。
- segment(セグメント): Waymo データの基本単位で、約20秒の連続した走行シーンです。1セグメントの中に複数カメラの動画、複数 LiDAR の点群、それらに付けた3次元ラベル、追跡IDなどが詰まっています。10Hz で記録されるため1セグメントは約200フレームです。
- 行指向 / 列指向(row-oriented / column-oriented): データの並べ方の違いです。行指向は「1件のレコードに属する全項目をまとめて」並べ、列指向は「同じ項目の値だけを集めて」並べます。必要な項目だけ読みたいときは列指向が圧倒的に有利です。
- Parquet(パーケット): 列指向でデータを保存する代表的なファイル形式です。同じ種類の値が並ぶので圧縮が効きやすく、必要な列だけを選んで読み込めて、分散処理基盤(大量データを多数の計算機で並列処理する仕組み、Spark など)とも相性が良いという特徴があります。
- mAP / mAPH(mean Average Precision / + Heading): 3次元検出の代表的な評価指標です。AP は「検出の正確さ(取りこぼし=recall と誤検出=precision のバランス)」を1つの数にまとめた値で、それをクラス平均したのが mAP です。mAPH は予測した向き(heading)の正しさも重みとして掛け合わせた指標で、向きを間違えると値が下がります。どちらも高いほど良く、Waymo はさらに難易度で層別(LEVEL_1 は点が5個以上ある明瞭物体、LEVEL_2 は点が1個以上で遠方・遮蔽の難物体も含む)して報告します。

## 仕組みを詳しく
v2 の核心は、保存形式を列指向の Parquet に作り替え、データ種類(コンポーネント)ごとに分割した点です。

初代(v1)は「1セグメントまるごと1ファイル(TFRecord という Google の形式)」でした。発想としては行指向的で、あるセグメントに関する全情報が1つの塊になっています。これだと一部だけ欲しくても全体を読む羽目になります。

v2 は、データをコンポーネントごとの表に分解しました。たとえば次のように分かれます。

```
v1: segment_0001.tfrecord
    └─ カメラ画像 + LiDAR点群 + 2Dボックス + 3Dボックス + 車の姿勢 + ... が全部入り(巨大)

v2(コンポーネント別の Parquet 群):
  CameraImageComponent          … カメラ画像(JPEG)だけ
  LiDARComponent                … LiDAR 点群(レンジイメージ)だけ
  CameraBoxComponent            … 画像上の2D物体ボックスだけ
  LiDARBoxComponent             … 3D物体ボックスだけ
  CameraToLiDARBoxAssociation   … 2D箱と3D箱の対応付け
  VehiclePoseComponent          … 車の姿勢(位置・向き)だけ
  CameraCalibration / LiDARCalibration … センサー校正情報
  LiDARSegmentationLabel        … 点ごとの意味ラベル(3D semantic seg)
  ...(さらに細分化)
```

利用者は欲しいコンポーネントだけを取得し、共通のキー(セグメントの context.name、フレームの時刻 timestamp_micros、レーザー名 laser_name、カメラ名 camera_name など)を使って後から結合(join)します。各行が「どのフレームのどのセンサーか」を一意に特定するキーを持つので、表計算で VLOOKUP するように別コンポーネントを突き合わせられます。たとえば次のような部分利用ができます。

- カメラ画像だけで研究したい → CameraImageComponent だけ取得。
- 3次元検出をやりたい → LiDARComponent と LiDARBoxComponent だけ取得し、timestamp で結合して「点群とその正解箱」を組み立てる。
- カメラと LiDAR の融合をやりたい → 画像・点群・対応付け・校正情報を取得して結合する。

つまり「巨大な全部入りを丸ごと落とす」から「必要な部品だけ落として後で組み立てる」へ、配布の発想を転換したのが v2 です。中身のセンサー構成・ラベル品質は v1 と同等を保ち、規模感としては約1150セグメント前後・各約20秒・5カメラ・5 LiDAR・1200万を超える3次元ボックス・約1000万の2次元ボックスといった大規模さです。

列指向が効く理由を数値感覚で補足します。仮に1セグメントが、点群2GB、カメラ画像0.3GB、3次元ラベル数MBだとします。3次元ラベルだけ欲しい研究者は、v1 では2.3GB+を丸ごと取得・展開してから数MBを取り出していました。v2 なら LiDARBoxComponent の数MBだけを直接読めます。データ種類が10あれば、不要な9割を運ばずに済む、というのが部分ダウンロードの威力です。さらに Parquet は列ごとに圧縮するため、似た値が並ぶ列(たとえば全部「車」のクラスIDが並ぶ列)が小さく潰れ、転送量がさらに減ります。

## 手法の系譜と主要論文
Waymo Open Dataset 本体(v1)は Sun et al., CVPR 2020「Scalability in Perception for Autonomous Driving」(arXiv:1912.04838)で提案されました。タイトルが示すとおり主眼は scalability(規模)で、多数の高品質なセグメントと正確な3次元ラベルを公開し、3次元検出・追跡・予測研究の標準ベンチマークになりました。当時の主要データであった KITTI(前方中心・小規模)に対し、5カメラ・5 LiDAR の全周構成と1200万超のラベルで規模の壁を一気に押し上げた点が新規性です。同時期に登場した nuScenes(Caesar et al., CVPR 2020)が6カメラ+1 LiDAR+レーダーで全周認識を提供したのと並び、全周大規模データの双璧をなします。

v2 のモジュラー形式(2023)は、論文というよりデータセットのメジャー改訂・公式リリースです。中身を維持しつつ列指向 Parquet にコンポーネント分割し、部分ダウンロードと分散前処理を容易にしました。動機は v1 の「全部入りで重い」問題の解消で、研究の参入障壁を下げる狙いです。

このデータ上で発展した代表的な検出手法の系譜も押さえておきます。VoxelNet(Zhou & Tuzel, CVPR 2018)が点群を立体格子(voxel)に区切って学習する道を開き、PointPillars(Lang et al., CVPR 2019)が点群を縦の柱(pillar)状セルに集約して2次元畳み込みで処理する高速・軽量な手法を提案しました。CenterPoint(Yin et al., CVPR 2021、arXiv:2006.11275)は物体を「中心点」のヒートマップとして検出し、フレーム間の中心点速度を予測して追跡まで一体で行う方式で、Waymo 上で当時最高クラスの性能を示しました。さらにマルチモーダル(カメラ+LiDAR 融合の BEVFusion 系)や Transformer ベースの検出器(BEVFormer 系)が続き、Waymo は今もこれら手法の主要な評価舞台です。なお Waymo は知覚版とは別に、軌道予測用の Motion Dataset(約10万シーン、HDマップ付き)も提供しており、MultiPath++(Varadarajan et al., 2021)や Wayformer などの予測研究で使われています。

## 論文の実験結果(定量データ)
CVPR 2020 の本体論文では、当時の標準的な検出器(PointPillars など)を Waymo 上で評価し、ベースラインの 3次元検出 mAP / mAPH を距離レンジ別・難易度別に報告しました。重要な知見は、当時広く使われていた KITTI に比べ、Waymo は物体数が桁違いに多く、特に30m を超える遠方や歩行者・自転車のような小さな物体で精度が大きく落ちること、すなわち「規模を上げると簡単に飽和しない難しさが顕在化する」という点でした。報告では、車両検出の LEVEL_1 mAP がベースラインで概ね6〜7割台に届く一方、歩行者・自転車や遠方ではそれを大きく下回り、データ規模の意義を定量的に示しています。

CenterPoint(Yin et al., CVPR 2021)は Waymo の公式リーダーボードで、当時のアンカーベース手法(あらかじめ用意した基準箱に合わせて検出する方式)に対し、車両・歩行者・自転車のいずれでも mAPH を明確に向上させたと報告しました(報告では複数クラスでアンカーベース比 数ポイントの改善)。中心点ベースにすると、回転した物体に対して基準箱を多数用意する必要がなくなり向きの推定が安定するため、heading を加味する mAPH で特に効く、というのがアブレーション的な含意です。さらに2フレームの中心点を結ぶだけの軽量な追跡で、複雑な追跡器に匹敵する MOTA(多物体追跡精度)を出した点も、検出と追跡を同じ表現で扱う設計の強みを示しました。

数値の意味を補足します。mAPH が mAP より低く出るのは、向き(heading)を間違えた検出が罰せられるためで、両者の差は「位置は当たっても向きを外す度合い」を表します。自動運転では向きの誤りが将来軌道予測の誤りに直結し、衝突判断を誤らせるため、mAPH の改善は安全性の観点で重みがあります。また LEVEL_2 が LEVEL_1 より低いのは、点がわずかしか当たらない遠方・遮蔽物体まで含めて評価するためで、両者の差は「難物体への弱さ」を測ります。具体的な数値は手法・版・評価条件で変動するため、最新の公式リーダーボードと論文付表を確認してください(ここで述べた傾向は報告に基づくおよその要約です)。

## メリット・トレードオフ・限界
メリット。列指向 Parquet により、必要なデータ種類だけを部分ダウンロードでき、容量と時間を大きく節約できます。分散処理基盤(Spark/BigQuery 等)と相性が良く、大規模前処理を効率化できます。中身は全周5カメラ・5 LiDAR・正確な3次元ラベル・追跡IDを備えた高品質・大規模データで、3次元検出・追跡の主要ベンチマークです。コンポーネント単位なので、自分のタスクに不要なデータを持たずに済みます。点ごとの3次元意味ラベル(3D semantic segmentation)も含むため、点群セグメンテーション研究にも使えます。

トレードオフと限界。コンポーネントが分割されたぶん、利用者が共通キーで結合する処理を自分で書く必要があり、v1 向けのコードはそのまま使えず v2 形式への対応が要ります。それでも全体規模は依然大きく、保管・帯域コストはかかります。走行は北米(主にカリフォルニア・アリゾナ等)中心で、地域・気象・交通文化の多様性には限界があります(中国の稠密交通を含む [ApolloScape](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0875_apolloscape) や、地理的多様性を売りにする [Mapillary Vistas](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0889_mapillary-vistas) と補完関係にあります)。知覚版が主で、運転の意思決定そのもののラベルは別データ(Motion Dataset 等)に分かれています。ライセンスは非商用研究向けで、商用利用には制約があります。

未解決の課題として、遠方・低密度点群での小物体検出、雨・霧・夜間など劣悪条件でのロバスト性、ドメイン間転移(別都市・別センサーへの汎化、Waymo→nuScenes のような cross-dataset 評価)などが依然オープンであり、これらを測るうえでも部分利用しやすい v2 形式は実験の回しやすさに寄与しています。

## 発展トピック・研究の最前線
近年は、カメラ画像を俯瞰図(Bird's-Eye-View, BEV)平面へ持ち上げて LiDAR と同じ座標系で融合する手法(BEVFormer, BEVFusion など)が Waymo を含む大規模データで主流になっています。これは複数カメラと LiDAR の情報を上から見た平面に統合し、検出と予測を同じ表現で扱う流れです。また、大量ラベルのコストを下げるための半教師あり・自己教師あり学習(点群の masked autoencoder など)、合成データやドメイン適応による別地域への転移、そして検出・追跡・予測・計画を1つのネットワークで解くエンドツーエンド化(UniAD 系)が活発です。さらに大規模シーンを再構成する NeRF / 3D Gaussian Splatting によるシミュレーション素材としても Waymo データが使われ始めています。v2 のように「必要なモダリティだけを選んで取り出せる」配布形式は、こうした多様な実験設計を低コストで試す土台として機能しています。
