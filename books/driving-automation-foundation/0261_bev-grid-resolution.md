# BEVグリッド解像度

## ひとことで言うと
BEV(Bird's-Eye-View、車を真上から見下ろした地図のような表現)を、何マス×何マスの格子(グリッド)で表すか、という解像度の話です。粗すぎる7×7のような格子では、1マスに車1台が丸ごと収まってしまい物体を区別できません。100×100〜200×200に上げると1マスが0.25〜0.5mになり、車・歩行者・車線を見分けられるようになります。解像度はそのまま「BEVが何を表現できるか」の物理的な上限を決める基本パラメータで、どれだけ賢いネットワークを乗せても、グリッドが粗ければ細部は原理的に表現できません。

## 直感的な理解
方眼紙に駐車場の地図を描くことを想像してください。1マスを7m四方にすると、乗用車(全長4.5m)は1マスに収まりきり、隣に止まったもう1台と区別できません。「ここに2台いる」のか「1台だけ大きく描いた」のか、紙の上では判別不能です。逆に1マスを0.25mにすれば、車は縦18マス・横7マス分を占め、車と車のすき間も、歩行者1人も、白線1本も、別々のマスとして描けます。

BEVグリッドの解像度はこの「方眼の細かさ」そのものです。深層学習のモデルは最終的にこのグリッド上で「どのマスに何があるか」を判断するので、マスが粗ければ、どんなに賢いモデルでも原理的に細部を表現できません。これは情報理論で言う量子化(連続量を離散的なマスに丸める操作)の限界で、解像度は性能の天井を決める制約なのです。学習で頑張れば改善できる「精度」とは違い、解像度は「そもそも表現可能か」という別次元の制約だ、という点を最初に押さえてください。

## 基礎: 前提となる概念

BEVとは、複数のカメラ画像から車の周囲を真上から見下ろした平面マップを作る表現です。前後左右のカメラ映像をこの1枚の俯瞰図に統合すると、「どこに何があるか」を実世界のメートル単位で空間的に扱えます。カメラ画像のままだと遠くの物体は小さく写るなど見え方が距離でゆがみますが(これを射影変形と言います)、BEVに直せば位置がそのまま地図座標になるため、軌道計画や障害物判定が扱いやすくなります。複数カメラを共通の座標系に統合できるのも大きな利点で、前カメラと右カメラに半分ずつ写った車も、BEV上では1つの物体として扱えます。

このBEVを計算機で扱うには、平面を格子(グリッド)に区切ります。グリッドが H×W マスなら、各マスが知覚範囲(perception range、BEVがカバーする実世界の広さ)の一区画を担当します。1マスが担当する実距離を「セルサイズ(cell size)」または「グリッド分解能」と呼び、次式で決まります。

```
セルサイズ = 知覚範囲(1辺の長さ) ÷ グリッドの1辺のマス数
```

ここで重要なのは、解像度・知覚範囲・セルサイズの3つが連動することです。範囲を固定したまま解像度を上げればセルは細かくなり、解像度を固定したまま範囲を広げればセルは粗くなります。この3者の関係は[BEV知覚レンジ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0263_bev-perception-range)で詳しく扱います。

陥りやすい失敗が、BEVグリッドの解像度を意図して設計せず、画像特徴を抽出するバックボーン(backbone、CNNやTransformerの土台ネットワーク)の最終出力サイズをそのまま流用してしまうことです。例えば 256×256 の入力画像を5段ダウンサンプル(各段で縦横半分)すると 256 / 2^5 = 8、端数処理で7×7程度になります。これをBEVグリッドとして使うと、解像度は「設計判断」ではなく「ネットワーク構造の副産物」になってしまい、知覚に必要な細かさを満たす保証がありません。バックボーンの空間サイズは「画像認識に必要な特徴の粒度」で決まっており、「実世界のメートルあたり何マス必要か」とは無関係だからです。

なぜ7×7では駄目か。知覚範囲を片側50m程度と仮定すると、

- セルサイズ = 50m ÷ 7 ≒ 約 7m × 7m

普通の乗用車は全長4.5m程度なので、車1台が1マスに丸ごと収まってしまいます。これでは隣り合う2台の位置関係も、車線のどちら側にいるかも表現できません。物体を識別するには、サンプリング定理と同じ発想で、セルサイズを「区別したい最小物体」より十分小さく(目安として半分以下に)取る必要があります。歩行者を区別したいなら肩幅0.5mより細かく、つまりセル0.25m程度が要ります。

## 仕組みを詳しく

解決策は「BEVグリッドをバックボーン出力から切り離し、明示的に定義する」ことです。代表的な手法は2系統あります。

ひとつは Lift-Splat 系(各画素の深度を推定して3D点へ持ち上げ=Lift、BEVグリッドへ落とす=Splat)。もうひとつはアテンション系(学習可能な「BEVクエリ」を H×W 個用意し、クロスアテンションでマスを埋める)です。ここではアテンション系の処理を段階的に追います。クロスアテンションとは「各マスがカメラ画像の対応箇所に注目して特徴を吸い上げる仕組み」です。

1. グリッドを定義する。例えば 200×200 マス。各マスに学習可能なクエリベクトル(次元 embed_dim、例えば256)を割り当てる。テンソル形状は `bev_queries: [200*200, 256] = [40000, 256]`。
2. 各BEVマスの中心は実世界のある (x, y) 座標に対応する。高さ方向に複数点(pillar=柱、例えば z を4点サンプル)を取り、3D参照点を作る。テンソル形状は `ref_points: [40000, 4, 3]`(マス数 × 高さ点数 × xyz)。
3. これらの3D点を、カメラの内部・外部パラメータ(校正情報。内部=焦点距離など、外部=カメラの取り付け位置・向き)を使って各カメラ画像の画素座標へ投影する。投影行列は ego(自車)座標系から画素座標系への変換を表す。
4. 投影先の画素位置から、双線形補間(`grid_sample`、周囲4画素の重み付き平均で連続位置の値を取る)で画像特徴を拾い、そのマスを「見ている」カメラ全部から集めて平均する(可視カメラ数 |V| で割る)。画像外に落ちた投影点は無視する。
5. 集めた特徴でクエリを更新する。これを数層(BEVFormer なら6層)繰り返して、最終的に `bev_feature: [200, 200, 256]` を得る。

解像度ごとのセルサイズ(知覚範囲 片側50m と仮定)を並べると、

| グリッド | セルサイズ(50m範囲) | 1マスあたりの意味 |
|---|---|---|
| 7×7 | 約7.1m | 車1台が丸ごと入る(区別不能) |
| 50×50 | 1.0m | 車を数マスで表現(粗い、歩行者は1マス未満) |
| 100×100 | 0.5m | 車・車線・歩行者をかろうじて区別できる |
| 200×200 | 0.25m | 細部まで分解、レーンマーキングも個別に乗る |

計算量・メモリのトレードオフも無視できません。マス数は1辺を2倍にすると面積で4倍に効くので、200×200は100×100の約4倍重くなります。BEV特徴を保持するメモリだけでなく、各マスがアテンションする計算もマス数に比例して増えます(クロスアテンションのコストはクエリ数に線形)。この見積もりは[BEV解像度とVRAMのトレードオフ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0260_bev-cell-size-vram-tradeoff)で詳述します。

なお、明示グリッドを採用しても前後対称(正方形)である必然性はありません。高速走行では前方を遠くまで見たいので、前を厚く後ろを薄くした非対称グリッド(例えば縦長の H>W)を取ることもあります。詳細は[BEV知覚レンジ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0263_bev-perception-range)を参照。

## 手法の系譜と主要論文

明示的なBEVグリッドの考え方は、2020〜2022年にかけて複数の系統で確立しました。

- Lift-Splat-Shoot(Philion & Fidler, ECCV 2020, arXiv:2008.05711): 各画素について深度の確率分布を推定し、その分布で画像特徴を3D空間へ「持ち上げ」、BEVグリッドへ「落とす」枠組みを提案。明示的なBEVグリッド(典型的に 200×200、0.5m)上で表現を学ぶ系統の出発点で、後続のBEVDet/BEVDepth の基礎になりました。提案動機は「複数カメラを微分可能な形でBEVに統合し、エンドツーエンド学習する」こと。
- BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270): 200×200グリッド(51.2m四方、セル約0.256m)。学習可能なBEVクエリが、複数カメラ特徴へ「空間クロスアテンション(SCA, Spatial Cross-Attention)」で注目し、さらに時間方向の「テンポラルセルフアテンション」で過去フレームのBEVを参照する。深度を明示推定せずにBEVを構築する点が新規。提案動機は「多視点・時系列を統一的に1つのBEV表現へ畳み込む」こと。
- UniAD(Hu et al., CVPR 2023, arXiv:2212.10156): 200×200。BEVFormerをBEVエンコーダとして使い、知覚→予測→計画を1本のネットワークに統合(end-to-end化)。動機は「下流タスク(計画)に効く特徴を一気通貫で学習する」こと。CVPR 2023のベストペーパーに選ばれた代表的研究で、明示BEVグリッドが計画まで貫通する設計の到達点。
- BEVDet / BEVDepth(Huang et al., 2022, arXiv:2112.11790 / Li et al., AAAI 2023, arXiv:2206.10092): 128×128(セル0.4m)。Lift-Splat系の発展。BEVDepthは深度推定をLiDARで明示的に教師付けして投影精度を上げた。車載リアルタイム性を意識し、あえて解像度を抑える方向。
- PETRv2 / StreamPETR(Liu et al., 2022, arXiv:2206.01256 / Wang et al., ICCV 2023, arXiv:2303.11926): 100×100相当(セル0.512m)。3D位置エンコーディングを画像特徴に注入し、明示的BEVグリッドを軽量に扱う系。StreamPETRは時系列を効率的に伝播させて速度と精度を両立。
- VAD(Jiang et al., ICCV 2023, arXiv:2303.12077): 200×200、範囲60m×30m(セル0.3m/0.15m)。地図・エージェントをベクトル化して計画を効率化。
- SparseBEV(Liu et al., ICCV 2023, arXiv:2308.09244): 200×200相当(セル0.256m)。密なBEVを作らず疎なクエリで効率化する逆方向の探求。

系譜としては「Lift-Splat系(深度推定でBEVへ持ち上げる)とアテンション系(BEVFormerのクエリで吸い上げる)が並行して発展 → UniADがそれを計画まで統合 → 以降は密グリッドを保つか疎クエリで軽くするかで分岐」という流れです。共通する結論は「実用上 100×100〜200×200、セルサイズ 0.25〜0.5m」が、車・歩行者・車線を区別できる最低ラインだということです。

## 論文の実験結果(定量データ)

定量評価は主に nuScenes データセット(Caesar et al., CVPR 2020、6カメラ・LiDAR・レーダー、1000シーン)で行われ、指標は2つが中心です。NDS(nuScenes Detection Score、検出の総合スコアで0〜1、高いほど良い。位置・大きさ・向き・速度・属性の誤差を統合した指標)と mAP(mean Average Precision、平均適合率、検出が当たっているかを総合した値、高いほど良い)です。地図セグメンテーションでは IoU(Intersection over Union、予測領域と正解領域の重なり率、高いほど良い)を使います。

- BEVFormer は nuScenes 3D検出で、当時のカメラのみ手法を大きく上回り NDS 約0.517(base設定)を報告しました。論文のアブレーションでは、BEVグリッドの空間クロスアテンションを外すと検出精度が明確に落ち、時間方向のセルフアテンションを加えると速度推定の誤差(mAVE、mean Average Velocity Error、速度の平均誤差で小さいほど良い)が改善することが示されています。これは「明示グリッド+アテンション」という構造そのものが精度の源泉であることの証拠です。
- 解像度に直接踏み込んだアブレーションでは、グリッドを粗くすると小さい物体(歩行者・自転車)のAPが顕著に落ちることが各論文で繰り返し報告されます。直感どおり、セルサイズが物体サイズに近づくと、その物体は「1マスに溶けて」検出できなくなります。逆に解像度を上げると小物体APは改善するものの、200×200を超えると改善幅が頭打ちになりつつ計算量だけが増える、という飽和も観測されます。これは「物体が十分に分解された後はセルを細かくしても新しい情報が増えない」ためで、無闇な高解像度化が割に合わないことを示します。
- BEVDepth は深度の明示教師付けにより、同じ 128×128 解像度でも BEVDet から NDS を概ね +3〜4 ポイント規模改善しました(報告による)。これは「解像度が同じでも、投影の正確さ(=正しいマスに特徴が乗るか)が精度を左右する」ことを示し、解像度だけでなく投影品質も重要だという教訓を与えます。粗い深度推定だと、本来そこにない遠くのマスに特徴がにじんで乗り、解像度が高くても中身が崩れます。

指標の意味を補足すると、NDS が0.5前後というのは「カメラのみで、LiDARを使わずにここまで3D位置・速度・属性を当てられる」水準で、2022年時点では画期的でした(LiDARベースの手法は0.7前後)。mAP の数ポイント差は、実際の運転では「見落とす物体が数%減る」ことに対応し、安全性に直結します。

## メリット・トレードオフ・限界

高解像度化(7×7 → 100×100〜200×200)のメリット
- セルサイズが物体より小さくなり、車・歩行者・車線を区別できる。表現できる空間情報の天井が上がる。
- 下流の軌道予測・計画に効く細かい空間構造(車線のどちら寄りに他車がいるか等)が乗る。
- バックボーン出力に縛られず、知覚範囲とセルサイズを独立に設計できる。

トレードオフ・限界
- マス数は1辺の2乗で効くので、VRAMと計算量が急増する(200×200は100×100の約4倍)。詳細は[BEV解像度とVRAMのトレードオフ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0260_bev-cell-size-vram-tradeoff)。
- 高解像度ほどアテンションのサンプリング点が増え、学習・推論が遅くなる。リアルタイム車載推論(10Hz以上)を狙うなら上げすぎると間に合わない。
- 真の効果を得るには正確なカメラ校正(内部・外部パラメータ)が必須。校正がダミーだと幾何が学べず、解像度を上げても中身が伴わない。
- 解像度を上げても精度はどこかで飽和する。物体が十分分解された後はセルを細かくしても情報が増えないため、無闇な高解像度化は計算の無駄になる。

未解決の研究課題として、密グリッドの計算コストをどう下げるか(疎クエリ化、可変解像度、関心領域だけ高解像度にする adaptive grid)、そして「検出に最適な解像度」と「計画に最適な解像度」が一致しない可能性(計画は粗い占有でも足りるが検出は細かさが要る)などがあり、タスク横断で最適解像度を決める枠組みは確立していません。さらに、評価ベンチマークの多くが近〜中距離中心のため、遠距離での最適解像度はデータ不足で十分に検証されていない、という研究上の空白も残っています。

## 発展トピック・研究の最前線

近年は密なBEVグリッドそのものを置き換える動きが活発です。占有グリッド(Occupancy Grid)を3Dに拡張した Occupancy Network(各ボクセル=3Dの小さな立方体が占有か否かを予測。SurroundOcc / OccFormer / TPVFormer などの研究)は、BEVの2D平面を3Dボクセルに拡張して高さ情報も保持します。これは解像度の議論を縦方向にも広げるもので、ボクセル数 = H×W×Z とメモリがさらに膨らむため、疎ボクセルや階層的解像度(粗いグリッドで全体を見て、関心領域だけ細かくする)の工夫が研究の焦点になっています。

逆に SparseBEV / Sparse4D のように密グリッドを捨てて疎なクエリ集合で物体を直接表現する方向もあり、「明示グリッドの解像度」という枠組み自体を問い直す研究が並行して進んでいます。疎クエリは「物体がありそうな数百箇所だけを表現する」ので、グリッド全面を保持するコストを避けられますが、その代わり連続的な占有マップを直接は出せないという別のトレードオフを抱えます。密グリッド派と疎クエリ派のどちらが計画タスクに最終的に有利かは、まだ決着していない最前線の論点です。
