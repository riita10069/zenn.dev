# プランナ レジストリパターン（BasePlanner + PLANNER_REGISTRY）

## ひとことで言うと
プランナ レジストリパターンとは、軌跡を作る「プランナ」（GRU、Flow Matching、Bézier など中身の違う複数方式）を、共通の基底型 `BasePlanner` に揃え、名前の文字列からクラスを引ける辞書 `PLANNER_REGISTRY` に登録しておく設計です。`planner_mode="flow_matching"` のように設定の文字列を1つ変えるだけで、モデル本体のコードを書き換えずにプランナを差し替えられます。深層学習の研究・実験で「複数の手法を公平に比較したい」ときの定番の作法です。

## 直感的な理解
電源プラグとコンセントを思い出してください。世界中に色々な電化製品がありますが、プラグの形（インターフェース, 接続の約束事）が共通なら、どの製品でも同じコンセントに差すだけで動きます。レジストリパターンは、これを深層学習のモジュールでやる仕組みです。

軌跡を作る方式は1つではありません。リカレントネットで1ステップずつ吐く方式、ノイズから生成する方式、なめらかな曲線の制御点だけを出す方式。これらを「同じプラグの形（BasePlanner という共通インターフェース）」に揃え、「どのコンセントに差すか（registry の文字列キー）」を設定で選べるようにします。すると、新しい方式を試したいとき、本体（家の配線）を改造せず、新しい製品をプラグに合わせて作って差すだけで済みます。研究では「方式 A と方式 B のどちらが良いか」を比べる実験を何十回も回すので、この差し替えやすさが研究速度を直接左右します。

## 基礎: 前提となる概念
end-to-end の自動運転モデルは、カメラ画像などの観測から特徴を抽出し、最終段で「これからどう走るか」の軌跡を出します。この最終段が「プランナ（軌跡デコーダ）」です。プランナの作り方には代表的に次の系統があります。

- 自己回帰（リカレント）方式: GRU や LSTM などのリカレントネット（前の出力を次の入力に使うネット）で、軌跡を時刻ごとに1点ずつ順に吐く。古典的で安定だが逐次処理で遅め。
- 生成モデル方式: Diffusion / Flow Matching でノイズから軌跡をサンプリングする。多峰性（複数の正解がある状況。たとえば障害物を左右どちらでも避けられる）を表現できるが推論に反復が要る。
- パラメトリック曲線方式: Bézier 曲線やスプラインの制御点だけを予測し、軌跡を解析的に描く。超軽量・超なめらかだが表現力が制御点数に縛られる。

次に「デザインパターン」の用語を押さえます。デザインパターンとは、ソフトウェア設計で繰り返し現れる問題への定石（Gamma らの古典書, 1994 が起源）です。

- インターフェース（抽象基底クラス）: 複数の実装が共通に守るべきメソッドの約束（型・引数・返り値）を定めた雛形。中身は持たず「こういうメソッドを持て」とだけ宣言する。
- レジストリ（registry）: 名前（文字列）から実体（クラスや関数）を引くための辞書。
- ファクトリ関数（factory）: 名前を受け取って対応するインスタンスを作って返す関数。「どのクラスを new するか」の判断を1か所に集約する。

この3点セット（インターフェース＋レジストリ＋ファクトリ）がレジストリパターンの骨格です。

## 仕組みを詳しく
典型的なディレクトリ構成は次のようになります。

```
trajectory_planning/
├── __init__.py          ← PLANNER_REGISTRY + build_planner()
├── base.py              ← BasePlanner（共通インターフェース）
├── flow_matching_planner.py  ← Flow Matching デコーダ
├── gru_planner.py       ← 自己回帰デコーダ
└── bezier_planner.py    ← Bézier デコーダ
```

### 1. 共通インターフェース BasePlanner
全プランナが継承する抽象基底クラスです。`forward()` は必ず「完成した軌跡 `[B, num_timesteps * num_signals]`」を返すと契約します。学習と推論の入口を分ける思想（forward は軌跡、損失は別メソッド）は [学習/推論API分離（construct_training_data と external loss）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1022_train-infer-api-separation) に詳述されています。この契約があるから、本体は中身を知らずにどのプランナも同じ顔で扱えます。インターフェースが「軌跡を返す」だけでなく「何を返せるか（不確実性、複数候補など）」をどこまで含むかは、後で試せる手法の幅を決める重要な設計判断です（発展トピックで再論）。

### 2. レジストリ（辞書）PLANNER_REGISTRY
```python
PLANNER_REGISTRY = {
    "gru": GruPlanner,
    "flow_matching": FlowMatchingPlanner,
    "bezier": BezierPlanner,
}
```
文字列キーからプランナのクラスを引ける辞書です。新方式を足すには、ここに1行加えるだけ。デコレータ（`@register("xxx")`）で各クラス定義の直後に自動登録する流儀もあり、Detectron2 や timm がこの方式です。

### 3. ファクトリ関数 build_planner
```python
def build_planner(planner_mode, **kwargs):
    if planner_mode not in PLANNER_REGISTRY:
        raise ValueError(f"Unknown planner_mode {planner_mode!r}. "
                         f"Available: {sorted(PLANNER_REGISTRY)}.")
    return PLANNER_REGISTRY[planner_mode](**kwargs)
```
`planner_mode` を渡すと、対応するクラスを `**kwargs` で初期化して返します。知らない名前なら、選べる一覧つきのエラーを出すので、タイプミスに即座に気づけます。`**kwargs` はそのまま各プランナの `__init__` に渡され、各プランナは自分が理解する引数だけ受け取ります（レジストリ側は事前フィルタしないので、呼ぶ側が正しい引数を渡す責務を持ちます）。

### 使い方（モデル本体側）
```python
self.planner = build_planner(planner_mode, embed_dim=256, num_timesteps=64, ...)
```
本体は具体的なプランナクラスを知らず、`planner_mode` の文字列だけ受け取ります。実験で方式を変えるときは、設定の文字列を `"flow_matching"` から `"bezier"` に変えるだけ。本体コードは無改変です。同じ思想は backbone（特徴抽出ネットの選択）や view fusion（複数カメラ画像の融合方式）でも使われ、「registry に足せば本体無改変で選べる」という運用に統一できます。

### なぜこの設計が良いか
もし本体の中に特定プランナがべったり埋め込まれていると、別方式を試すたびに本体を書き換える羽目になります。すると (1) 実験のたびに if 文が増えて読みにくい、(2) 新方式追加で本体の複数箇所を直す必要がありバグの温床、(3) どの方式が選べるか一覧で分からない、という問題が生じます。レジストリパターンはこれらを一掃します。これはソフトウェア工学でいう「開放閉鎖原則（Open-Closed Principle, 拡張には開き・改変には閉じる）」の実践でもあります。新方式の追加（拡張）はできるが、既存の本体コード（改変対象）には触れない、という状態を作るのです。

## 手法の系譜と主要論文
レジストリパターン自体はソフトウェア設計（デザインパターン, Gamma et al., 1994 が体系化）であり、特定論文の機械学習手法ではありません。各プランナ方式の背景論文は次の通りです。

Cho et al., 2014, "Learning Phrase Representations using RNN Encoder-Decoder for Statistical Machine Translation"（EMNLP, arXiv:1406.1078）。GRU（Gated Recurrent Unit）を提案。系列を1ステップずつ処理する軽量なリカレント構造で、自己回帰で軌跡を吐くプランナの土台になります。トレードオフは、逐次処理ゆえ推論が遅めになりがちな点です。

Lipman et al., 2023, "Flow Matching for Generative Modeling"（ICLR, arXiv:2210.02747）。Flow Matching プランナの土台。ノイズから速度場を積分して軌跡を生成し、多峰性のある軌跡を表現できます。一方、推論に複数ステップの積分が要り、未学習・学習不十分だとなめらかさが悪くなりやすいです。Diffusion 系（Ho et al., 2020）や、行動生成への応用（Chi et al., 2023, Diffusion Policy, RSS, arXiv:2303.04137）とも連なります。Diffusion Policy は、生成方式のプランナが「複数正解の平均（mode averaging）」を避けてマルチモーダルな行動を出せることを実証し、生成プランナを運動制御に持ち込む流れを作りました。

Bézier 曲線は古典的な数学（Pierre Bézier、ルノーの技術者、1960年代に自動車ボディ設計のために体系化）。制御点だけを予測すればよいので超軽量・超なめらかで、自動運転の軌跡平滑化・後処理で広く使われます。トレードオフは表現力が制御点数に縛られ、急な機動や複雑な軌跡が苦手なことです。

「複数の実装を共通の型に揃えて差し替え可能にする」発想は、深層学習フレームワーク全般で標準的です。timm（PyTorch Image Models）のモデルレジストリ、Detectron2（Wu et al., 2019）の Registry、Hugging Face Transformers の Auto クラス（AutoModel など）、MMDetection の Registry が代表例で、いずれも「名前文字列 → 実装」を辞書で引いてインスタンス化します。これらは設定ファイル（YAML / dataclass）から実験全体を駆動する「config-driven」な研究基盤の中核です。

## 論文の実験結果（定量データ）
レジストリパターン自体に定量結果はありませんが、その最大の効用は「複数方式を同一ハーネスで公平に横並び比較できる」ことです。実際にプランナ方式を同条件でベンチマークすると、設計トレードオフがはっきり見えます。代表的に計測される量は次の3つです。

- パラメータ数: モデルの大きさ。小さいほど軽量・省メモリ。
- 推論レイテンシ（p50 = 中央値, ミリ秒）: 1回の推論にかかる時間。小さいほど速い。自動運転は 10〜20Hz（50〜100ms 以内）で軌跡を出し続ける必要があるため致命的に重要。
- 軌跡のなめらかさ（jerk = 加速度の変化量の分散, curvature = 曲率変化の分散）: 小さいほどなめらかで乗り心地が良い。乗員の快適性や車両の追従性に直結する。

3方式を同条件（同じ特徴次元・同じ出力長）で回した一例（未学習・ランダム重みでの計測のため、なめらかさの絶対値は学習後に大きく変わる点に注意）:

| planner | params | p50 推論(ms) | jerk Var(Δaccel) | curv Var(Δcurv) |
|---|---|---|---|---|
| gru | 約 829K | 約 50.3 | 1.4e-04 | 3.8e-04 |
| flow_matching | 約 988K | 約 13.9 | 4.2e+00 | 2.0e+00 |
| bezier | 約 495K | 約 0.63 | 5.0e-06 | 1.2e-06 |

この表の読み方:
- GRU は逐次処理（1ステップずつ）のため推論が最も遅い（約 50ms）。出力長が伸びるほど線形に遅くなる。
- Flow Matching は複数ステップ積分が要るが、各積分ステップが並列計算しやすく GRU より速い場合もある（約 14ms）。ただし未学習だとなめらかさ指標が極端に悪い（jerk 4.2、curv 2.0）。これはランダムな速度場を積分しているため当然で、学習後は大きく改善します。
- Bézier は制御点を1回出すだけなので圧倒的に速く（約 0.63ms）、構造的になめらか（jerk・curv が桁違いに小さい。曲線が数式で滑らかさを保証されるため）。

この種の表は「精度（ADE/FDE など）」を含めて初めて方式選択の結論が出せます。ADE（Average Displacement Error, 予測軌跡と正解軌跡の各時刻の平均距離誤差）や FDE（Final Displacement Error, 終端での距離誤差）は学習済みチェックポイントが必要で、未学習のベンチマークでは測れません。レジストリパターンの価値は「学習後に同じハーネスで ADE/FDE まで含めて公平比較できる土台を提供する」点にあります。実際、軌跡予測ベンチマーク（nuScenes, Argoverse, Waymo Open Motion など）では、同一の特徴抽出部に複数のデコーダを差し替えてアブレーションするのが標準的な実験設計で、レジストリパターンはそれを支える実装基盤です。

## メリット・トレードオフ・限界

メリット:
- 文字列1つでプランナを差し替えでき、モデル本体コードを触らずに実験できる。
- 新方式の追加が「クラスを書いて registry に1行足す」だけで済む（開放閉鎖原則）。
- 選べる方式が辞書として一覧でき、タイプミスは生成時にエラーで弾ける。
- 同条件ベンチマークが組みやすく、方式比較が公平になる。
- backbone / fusion など他のコンポーネント選択と設計が揃い、コードベース全体の一貫性が上がる。

トレードオフ・限界:
- ファクトリが `**kwargs` を素通しするため、プランナごとに必要な引数を呼ぶ側が正しく渡す責務がある（registry は事前フィルタしない）。引数の取り違えは実行時まで気づきにくい。
- 共通インターフェース `BasePlanner` に全プランナを合わせる必要があり、方式ごとに入出力が大きく違うと抽象化が窮屈になることがある（たとえば生成方式は確率的サンプリングで複数候補を出せるが、回帰方式は決定論的で1本だけ、と性質が違う）。
- レジストリは文字列キー依存なので、IDE の補完や静的型解析（mypy 等）の恩恵を受けにくい（存在しない名前でも import 時には通り、実行時にエラー）。
- 抽象化のために「共通でない便利機能」を捨てる場合がある。たとえば生成方式だけが持つ「複数候補軌跡のサンプリング」を共通 API に含めるか否かで設計が割れる。共通項に削ると生成方式の長所を活かせず、全部入りにすると回帰方式が使わない引数で太る。

## 発展トピック・研究の最前線
- 生成方式のプランナ（Diffusion / Flow）は近年の end-to-end 運転で活発で、マルチモーダルな運転意図（直進・右折・停止など複数の妥当な選択）を1つのネットで表現できる利点があります。複数候補をサンプリングして最良を選ぶ、あるいは安全フィルタを通す、といった運用が研究されています。
- レジストリパターンを「プランナ」だけでなく「損失関数」「データ拡張」「サンプラー（推論積分法）」にも広げると、実験設定を設定ファイル（YAML 等）だけで完全に切り替えられる「config-driven」な実験基盤になります。Detectron2 や MMDetection 系がこの思想を徹底しています。再現性（同じ config で同じ実験を再現できる）にも直結します。
- 共通インターフェースの設計が研究の質を左右します。たとえば「forward は軌跡を返す」だけでなく「不確実性も返せる」「複数候補（K本の軌跡＋各々の確率）を返せる」を契約に含めるかどうかは、その後に試せる手法の幅を決めます。インターフェース設計は単なる作法ではなく、研究の探索空間を規定する重要な意思決定です。最初に狭く設計すると、後でマルチモーダル予測を試したくなったときにインターフェース全体を作り直す羽目になります。
