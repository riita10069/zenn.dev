# 差し替え可能Plannerインターフェース

## ひとことで言うと
軌跡を生成する部品（プランナ）が複数あっても、すべて同じ形のテンソルを出力し、自分の学習誤差も自分で計算するように「契約（インターフェース）」を統一する設計です。これによって学習ループは「今どの種類のプランナを使っているか」を一切知らなくてよくなり、新しいプランナを追加してもループ側のコードを書き換える必要がなくなります。ソフトウェア工学でいう Strategy パターン（戦略の差し替え）を、運転・ロボットの軌跡生成モデルに適用したものと言えます。地味な設計ですが、研究の試行回数を増やし、再現性を高め、実運用への移行を滑らかにする「研究を加速させるインフラ」として機能します。

## 直感的な理解
電化製品のコンセント（プラグの形）を思い浮かべてください。掃除機でも扇風機でも、プラグの形が同じなら、壁のコンセント側は「中身が何か」を知らなくても電気を流せます。コンセント側は「形」という契約だけを信じればよいのです。中身が直流モーターか交流モーターかをコンセントが気にする必要はありません。

機械学習の軌跡生成でも同じことが起きます。「プランナ（planner）」とは、車やロボットがこれからどう動くべきか、その軌跡（trajectory、これから先の一連の状態の並び）を予測する部品です。軌跡の作り方には大きく異なる流派があります。

- 拡散モデル（Diffusion）系: ランダムなノイズから出発し、何度も少しずつノイズを除去して軌跡を生成する。
- フローマッチング（Flow Matching）系: ノイズから目標へ向かう「速度場（どちらへどれだけ動くか）」を学習し、それを積分して軌跡を得る。
- パラメトリック曲線系（ベジエ曲線など）: 少数の制御点となめらかな曲線の数式で軌跡を表す。
- 回帰型（GRU/LSTM）系: 1ステップずつ順に次の状態を予測していく。

これらは中身がまったく違います。素朴に実装すると、学習ループの中に「もし拡散ならこう、もしフローマッチングならこう、もしベジエならああ」という巨大な条件分岐（if/elif）が増殖します。分岐が増えると三重の害があります。第一に、新しいプランナを足すたびに学習ループを書き換える必要があり、変更が広範囲に波及します。第二に、分岐を書き間違えてもエラーにならず、学習だけが静かに壊れます（後述する「速度場を軌跡と取り違える」事故）。第三に、コードが読みにくくなり、どの部品が責任を持つのか曖昧になります。

差し替え可能（swappable）インターフェースは、この問題を「全プランナが外から見て同じ振る舞いをする」ことを契約で強制して解決します。誰がどんなプランナを書いても、外から見た見え方は完全に同じになる、という規律です。

## 基礎: 前提となる概念
理解に必要な前提をいくつか噛み砕きます。

抽象基底クラス（abstract base class）とは、「このクラスを継承するなら、必ずこれらのメソッドを実装しなさい」という約束だけを定めて、中身は子クラスに委ねる仕組みです。Python では `abc.ABC` を継承し `@abstractmethod` を付けることで、未実装のままインスタンス化しようとするとエラーになります。これが「契約を守らせる」技術的な仕掛けです。約束を破ったコードは、実行する前の段階で弾かれます。

テンソルの形（shape）とは、多次元配列の各軸の大きさを表すタプルです。たとえば `[8, 128]` は「8行128列」を意味します。深層学習では、部品同士をつなぐとき形が一致していないと計算できないため、形は部品間の「物理的なコネクタ形状」に相当します。逆に言うと、形さえ合っていれば中身の意味が違ってもつながってしまう、というのが落とし穴の根源でもあります。

forward と loss の分離。深層学習のモデルには2つの異なる使い方があります。推論（inference）では入力から出力（ここでは完成した軌跡）を計算します。学習（training）では、出力が正解とどれだけずれているかを表す数値（loss、損失）を計算し、その値を使って重みを更新します。多くのフレームワークでは推論を `forward()` メソッドで行いますが、生成モデルでは「学習時に最適化する量」と「推論時に出力する量」が別物になりがちです（次節で詳述）。この「別物になる」という性質こそが、入口を分ける動機になります。

レジストリ（registry）とは、「名前 → クラス」の対応表（辞書）です。文字列キーでクラスを引けるようにしておくと、設定ファイルやコマンドライン引数に書いた名前から動的にオブジェクトを生成でき、コードに種類をハードコードせずに済みます。これがプラグイン的な差し替えを実現する基盤です。たとえば `{"flow_matching": FlowMatchingPlanner, "bezier": BezierPlanner}` のような辞書を一箇所に置き、`build_planner("bezier")` で対応するクラスを取り出します。

## 仕組みを詳しく
統一インターフェースの典型的な設計は、抽象基底クラス（仮に `BasePlanner` と呼びます）に2つの入口を分けて定義することです。

入口その1: `forward()` は必ず推論を行い、軌跡だけを返す。

出力の形は常に `[B, T*S]` に固定します。記号の意味は次の通りです。
- `B`: バッチサイズ（同時に処理するサンプル数）
- `T`: 何ステップ先までの軌跡か（典型的には16〜64ステップ）
- `S`: 各ステップで出力する信号の数（典型的には2、たとえば加速度と曲率、あるいは前進量と横移動量）

たとえば `B=8, T=64, S=2` なら出力は `[8, 128]`（64×2=128）になります。重要なのは、`forward()` が決してモード依存の中間量（フローマッチングの速度場など）を返してはいけない、という規律です。呼ぶ側は「`forward()` の戻り値は常に完成した軌跡である」と信じてよい、と契約が保証します。なお、出力を `[B, T, S]` という3次元で持つか `[B, T*S]` と平坦化して持つかは実装の好みですが、いずれにせよ「全プランナで同一」であることが契約の本体です。

入口その2: `compute_loss()`（あるいは `compute_planner_loss()`）が学習誤差を計算してスカラ（1つの数）として返す。

学習に必要な内部の一時テンソル（拡散・フローマッチングならノイズサンプル、目標速度、サンプリングした時刻など）は、すべてこのメソッドの内部に閉じ込め、外に漏らしません。返り値は必ず requires_grad な単一スカラで、呼び出し側はそれに `.backward()` するだけです。

なぜ入口を2つに分けるのか。最大の理由は「無言で壊れるバグ」の構造的な防止です。具体例を挙げます。フローマッチングの内部では、ネットワークは軌跡そのものではなく速度場 v を出力します。もし学習ループが「とりあえず予測と教師軌跡の二乗誤差を取る」という共通処理を書いてしまうと、フローマッチング使用時には速度場 v を軌跡だと勘違いして誤差を計算してしまいます。テンソルの形は偶然一致している（どちらも `[B, T*S]`）ためエラーは出ません。しかし学習している量がまったく間違っているため、loss は下がるのにモデルは正しい軌跡を生成できない、という最悪の事態になります。入口を分け、loss 計算を各プランナの内部に閉じれば、このコードはそもそも書けなくなります。これは「型では防げないが、API 境界の設計で防げるバグ」の好例です。

統一されたことで、学習ループは次のように書けます（疑似コード）。

```
planner = build_planner(name)      # "flow_matching" でも "bezier" でも "diffusion" でも
...
for batch in dataloader:
    loss = planner.compute_loss(context, target_trajectory)
    loss.backward()                # プランナの種類で if 分岐していない
    optimizer.step()
```

プランナの種類による分岐がどこにもない点に注目してください。`build_planner(name)` はレジストリからクラスを引いてインスタンス化するだけです。新しいプランナを追加するときは、そのクラスを書いてレジストリに1行登録するだけで、学習ループは無改変で新方式を選べます。評価コードも同様で、`forward()` が常に同一形の軌跡を返すため、平均変位誤差（後述）などの指標を方式を問わず共通に当てられます。

before / after を図示します。

```
[統一前]                                   [統一後]
train loop:                                train loop:
  if mode == "flow_matching":                loss = planner.compute_loss(ctx, tgt)
    v = net(noisy); loss = mse(v, v_target)   loss.backward()
  elif mode == "bezier":
    pts = net(ctx); traj = bezier(pts)       planner クラスごとに forward()/compute_loss() を実装
    loss = mse(traj, tgt)                    レジストリで名前→クラスを解決
  elif mode == "gru": ...
  分岐が増殖、取り違えバグの温床           分岐ゼロ、追加はレジストリ1行
```

## 手法の系譜と主要論文
このインターフェース設計の背後には、ソフトウェア工学と生成モデル研究の両方の流れがあります。

ソフトウェア工学側の起源は、Gang of Four（Gamma, Helm, Johnson, Vlissides, 1994）の Strategy パターンです。「アルゴリズムの族を定義し、それぞれをカプセル化して交換可能にする」という考え方そのものです。差し替え可能プランナは、この古典的パターンを軌跡生成に適用したものと言えます。同じ書籍にある「条件分岐をポリモーフィズムで置き換える」リファクタリングも、ここでの設計動機と直結します。

生成モデル側では、行動・軌跡生成を「決定的な回帰」から「確率的な生成」へ移す流れが、入口分離の必要性を生みました。

Diffusion Policy（Chi et al., RSS 2023, arXiv:2303.04137）は、ロボットの行動（軌跡）を拡散モデルで生成するポリシーを提案しました。動機は、人間のデモには複数の正解があり得る（多峰性、multimodality）ため、単一の平均的な行動を回帰する従来手法では「複数の正解の中間」という存在しない行動を出してしまう問題を解くことです。拡散モデルは学習時にノイズ（あるいはスコア）を予測し、推論時に反復的に行動を生成するため、「学習で最適化する量」と「推論で出す量」が明確に異なります。これが forward と loss を分ける設計動機の出発点です。

Flow Matching（Lipman et al., ICLR 2023, arXiv:2210.02747）は、拡散モデルより単純な目的関数で生成を学ぶ枠組みです。ノイズから目標へ向かう確率の流れを定義する「速度場」を、単純な平均二乗誤差で回帰します。学習が安定し速い反面、学習時に扱う速度場と最終出力の軌跡が別物である点は拡散と共通で、ここでも入口分離が安全策になります。

π0（Black et al., 2024, arXiv:2410.24164）に代表される近年の大規模 Vision-Language-Action モデルや、自動運転向けの条件付き軌跡生成プランナ（2024〜2025）でも、推論経路と学習経路の契約を明確に分け、複数の生成ヘッド（フローマッチング・拡散・回帰）を差し替え可能に保つ設計が広く採られています。π0 はフローマッチングで連続的な行動チャンクを生成しますが、外から見た契約（観測と言語指示を受け取り、行動列を返す）は固定されており、内部の生成方式は差し替え可能な部品として扱われています。生成プランナを実運用に乗せるには、研究で新方式を試しつつ運用コードを壊さないことが必須であり、この統一契約がその土台になっています。

## 論文の実験結果（定量データ）
インターフェース設計そのものは「速度や精度の数値」を直接持ちませんが、その背後にある生成プランナの優位性は定量的に示されています。これらの数値が「なぜわざわざ複数方式を差し替え可能にしておきたいのか」を裏付けます。

Diffusion Policy（Chi et al., 2023）は、ロボット操作のベンチマーク群（Robomimic、Push-T、マルチタスク操作など、合計15タスク程度）で評価されました。報告によれば、当時の最良の行動クローニング手法（LSTM-GMM、IBC、BET など）に対し、平均成功率を相対でおよそ46.9%改善したとされています。成功率とは「タスクを最後までやり遂げた試行の割合」で、これが上がるほど実用に近づきます。改善の主因は多峰性の表現力で、たとえば Push-T（T 字ブロックを目標位置へ押すタスク）では、左右どちらから押すかという2つの正解を拡散モデルが両方表現できたのに対し、回帰型は中間の無効な行動を出して失敗しました。1つのモデルでも方式が違えばここまで結果が変わるため、複数方式を公平に比較できる統一基盤の価値が出てきます。

アブレーション（要素を抜く実験）も示唆的です。Diffusion Policy の論文では、行動を「絶対座標」で表すか「相対変位」で表すかなど出力表現を変えると性能が変わること、また推論時の反復ステップ数を減らすと速度は上がるが品質が落ちることが報告されています。これらは「forward が返すべきものは何か」「loss が最適化すべき量は何か」を明確に分けて管理する重要性を裏付けます。表現を取り違えれば、これらの調整自体が成立しません。

Flow Matching 系（Lipman et al., 2023）は画像生成（ImageNet 等）で、拡散モデルと同等以上の品質（FID、生成分布が本物にどれだけ近いかの指標で小さいほど良い）を、より少ない関数評価回数（NFE、Number of Function Evaluations、推論で何回ネットワークを呼ぶか）で達成したと報告されています。NFE が小さいほど推論が速いため、軌跡をリアルタイムに何度も生成し直す運転・ロボット用途で有利です。π0 系の実機評価でも、フローマッチングによる行動生成が高い制御周波数（数十 Hz）での実時間動作と両立することが示されており、方式選択が推論速度に直結することを物語っています。

## メリット・トレードオフ・限界
メリット。学習ループがプランナ種類を知らずに済み、条件分岐が消えます。新プランナの追加がレジストリ1行で済み、既存コードを壊しません。forward と loss の分離により「速度場を軌跡と取り違える」類の無言バグが構造的に防がれます。出力が共通形のため、外側の軌跡評価指標（平均変位誤差 ADE、最終変位誤差 FDE など）を種類を問わず共通に適用でき、方式 A と方式 B を厳密に同一条件で比較できます。

トレードオフと限界。抽象契約に縛られるため、あるプランナ固有の中間出力（たとえば拡散の各ステップの中間軌跡を可視化したい、不確実性の分散を外で使いたい）を外側で利用したい高度なケースでは、契約を回避する追加の窓口（診断用メソッドやフック）が必要になります。また「forward は必ず完成した推論結果を返す」という規律を一度破ると、取り違えバグの防波堤が崩れるため、開発チーム全員の規律が前提になります。さらに、すべてのプランナを同一出力形に押し込めることで、本来は異なる次元・異なる表現が自然な方式（たとえば占有グリッドを出すプランナと制御信号列を出すプランナ）を無理に統一すると、かえって表現力を削ぐ恐れがあります。どこまでを共通契約にし、どこからを各方式の自由にするかの線引きは、研究上の設計判断として残ります。契約を狭くしすぎると窮屈になり、広くしすぎると分岐が復活する、というバランスの問題です。

## 発展トピック・研究の最前線
近年は、単一のインターフェースの下に複数の生成ヘッドを束ねるだけでなく、推論時に方式を切り替える（速い方式で粗く生成し、遅い方式で精密化する）ハイブリッド設計が研究されています。Consistency Models や蒸留（distillation）によって拡散・フローを少ステップ化する流れも、推論コストと品質のトレードオフを契約の内側で吸収する例です。また、Vision-Language-Action モデルのように言語条件まで含めた巨大な文脈を受け取るプランナでも、出力契約（最終的な行動列の形）を固定することで、巨大モデルと軽量モデルを同じ評価基盤で比較できるようにする流れが進んでいます。インターフェース設計は地味ですが、研究の試行回数を増やし、再現性を高め、運用への移行を滑らかにする「研究を加速させるインフラ」として再評価されています。
