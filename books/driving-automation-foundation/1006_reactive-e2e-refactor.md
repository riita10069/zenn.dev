# 反応的ポリシーと世界モデルの分離(階層的 E2E 運転のリファクタリング)

## ひとことで言うと
自動運転の end-to-end(E2E、センサ入力から運転操作まで一気通貫で学習する)モデルを、設計し直して「高頻度で反応する部品」と「低頻度で未来を予測する部品」に分ける、という構造変更(リファクタリング)の話です。具体的には、これまで1つの大きなクラスが backbone(特徴抽出)・融合・プランナ(軌跡生成)を全部抱えていたのを、反応的処理一式を独立したクラスにまとめて切り出し、外側のクラスはそれを呼ぶだけの薄い包み(ラッパ)にします。これにより、後から「世界モデル」的な部品を並べて組み合わせられる土台ができます。あわせて、公開 API(外から呼ぶときの関数の形)を変えるとそれに依存するコード(特にテスト)が壊れる、というリファクタリングの典型的な副作用も学べます。

## 直感的な理解
人間の運転を観察すると、2つの異なる時間スケールの処理が同時に走っていることに気づきます。1つは「今この瞬間、前の車が急に止まったらブレーキを踏む」という反射的で高速な反応です。これは0.1秒オーダーで回ります。もう1つは「この先の交差点は混みそうだから車線を変えておこう」という、もう少し先を見越したゆっくりした計画です。これは数秒オーダーで考えます。

E2E 運転モデルもこの2層に分けると見通しが良くなります。高頻度(たとえば10Hz、1秒に10回)で多カメラ画像・地図・自車の動きを処理して、すぐ先の目標へ向かう軌跡を出す「反応的ポリシー」。そして低頻度(たとえば1Hz)で画像列を処理し、未来の特徴状態(future feature state)を予測して、学習時の補助的な手がかりに使う「世界モデル」的な部品です。

リファクタリングとは、外から見た振る舞いを保ったまま、内部のコード構造を整理し直すことです。この2層構造を作るには、まず「反応的処理一式」を1つのまとまった部品として切り出す必要があります。家を増築する前に、既存の部屋の壁をきれいに区切り直して、後から部屋を足せるようにするのに似ています。

## 基礎: 前提となる概念
end-to-end 運転とは、カメラやライダーなどのセンサ入力を受け取り、途中で「物体検出」「経路計画」などのモジュールに明示的に分けず、ニューラルネットが直接「どう動くか(軌跡やハンドル操作)」を出力する方式です。対義語はモジュラー方式(知覚→予測→計画を別々のモジュールで作る古典的な構成)です。

backbone(バックボーン)とは、画像などの生の入力から有用な特徴(feature)を抽出する、モデルの土台となるネットワークです。ResNet や Swin Transformer などが使われます。

融合(fusion)とは、複数の情報源(複数カメラの特徴、地図、自車の動き)を1つの表現にまとめる処理です。

プランナ(planner)とは、まとまった特徴から「これからどう動くか」の軌跡を生成する部分です。

公開 API(public API)とは、ある部品を外から使うときの「呼び出し方の約束」です。関数名、引数の名前と順序、戻り値の形などを指します。これが変わると、その部品を使っていたすべてのコードが影響を受けます。

state_dict とは、PyTorch でモデルの全パラメータを「名前→テンソル」の辞書として取り出したものです。チェックポイント(checkpoint、学習途中の重みを保存したファイル)はこの state_dict を保存します。重要なのは、パラメータの名前(キー)が、そのパラメータを保持する属性のパス名から自動的に決まる点です。

世界モデル(world model)とは、環境の状態がこの先どう変化するかを内部で予測するモデルです。Ha & Schmidhuber の同名研究(2018)以降、強化学習やロボティクスで「未来を想像して学習・計画する」枠組みとして注目されています。

## 仕組みを詳しく
この種のリファクタリングで起きる変更を、段階的に見ていきます。

第一はクラスの入れ子化です。外側のクラス(以下、外側クラス)は、内部に反応的部品(以下、反応クラス)を1つ持つだけの薄いラッパになります。外側クラスのコンストラクタは、受け取った引数をそのまま反応クラスに渡すだけです。これに伴い、backbone・融合・地図エンコーダ・時系列メモリ・プランナといったサブモジュールは、すべて反応クラスの下に移動します。

第二は、これが最も影響の大きい副作用ですが、パラメータ名のプレフィックス変化です。PyTorch では、あるモジュールを `self.reactive` という属性に入れると、その中の学習パラメータの名前(state_dict のキー)に `reactive.` という接頭辞が付きます。

```
before: Backbone.layer1.weight
after:  reactive.Backbone.layer1.weight
```

つまりチェックポイントの鍵が全部変わります。before に保存した重みファイルを after のモデルに `load_state_dict` で読み込もうとすると、鍵が一致せずエラーになります。移行(キー名の変換)が必要です。また「勾配が特定のサブモジュールに流れるか」を確認するテストも、参照するパス名の修正が必要になります。

第三は forward(前向き計算)の戻り値の単純化です。外側クラスの `forward` は、全モード共通で軌跡 `trajectory` だけを返すようになります。旧来あった「損失や中間表現を含む複数値の組」は廃止されます(詳細は別トピック)。

第四はプランナの整理です。プランナの種類が見直され、推論専用の `forward` は軌跡だけを返し、プランナごとに散らばっていた損失計算は削除されて、学習目的関数は学習ループ側に移ります。

第五は世界モデル部品の宙吊りです。未来特徴を予測する部品は、反応クラスの中で「構築はされる」ものの「forward から呼ばれない」状態になることがあります。すると出力を生まず勾配も受け取らない「宙吊り(dangling)」になります。これはリファクタ途中によくある負債で、後で損失に再接続する前提のプレースホルダです。

before / after を1行で言うと、before は `model.Backbone` で直接 backbone にアクセスでき forward が複数値を返す世界、after は `model.reactive.Backbone` でアクセスし forward が軌跡1本だけを返す世界です。

テストの追従も重要な作業です。公開 API を変えると、テストがモック(本物の代わりに差し込む偽物の部品)で差し替えていた対象の参照パスが、旧パス(たとえば「外側クラス直下のバックボーン」)から新パス(「外側クラス→反応クラス→バックボーン」)へ変わります。各テストが軌跡1本を受け取るよう期待値を直し、勾配テストは新しいパス名で参照する、といった修正が連鎖します。本番の挙動は変えていないのに、テストと型注釈だけで大量の修正が発生する、というのがこの種のリファクタの実感です。なぜモックのパスが壊れるかというと、Python ではモジュール内の名前は「どの名前空間に属するか」で参照されるため、クラスを別のクラスの内側へ移すと、その内部部品を指す完全修飾パス(ドット区切りの参照名)も一斉に変わるからです。

## 手法の系譜と主要論文
これはソフトウェア設計と運転モデル設計の両方にまたがるので、2つの系譜で説明します。

ソフトウェア設計の面では、リファクタリングは Martin Fowler "Refactoring: Improving the Design of Existing Code"(初版1999、第2版2018)に体系化されています。提案は「振る舞いを保ったまま内部構造を改善する、小さな変換の積み重ね」。利点は将来の拡張容易性、トレードオフは「公開 API を変えると依存コードが壊れ、テストや利用側の追従が必要」になる点です。テストが赤くなるのはまさにこの教科書的な現象で、Fowler が「リファクタリングはテストの裏付けがあって初めて安全にできる」と強調する理由でもあります。

運転モデル設計の面では、「高頻度の反応的ポリシー」と「低頻度の世界モデル/予測」を分ける発想は、複数の系譜が合流したものです。

World Models(Ha & Schmidhuber, NeurIPS 2018, arXiv:1803.10122)は、環境のダイナミクスを予測する世界モデルと、それを使って行動を決める軽量なコントローラを分離しました。提案は「観測を圧縮して未来を予測するモデルと、その上で動く方策を分ける」こと。利点は方策が軽くなり想像の中で学習できること、トレードオフは世界モデルの誤差が方策に伝播することです。

自己教師あり未来予測の流れでは、JEPA(Joint-Embedding Predictive Architecture、LeCun "A Path Towards Autonomous Machine Intelligence", 2022 で提唱)とその画像版 I-JEPA(Assran et al., CVPR 2023, arXiv:2301.08243)が、「ピクセルを再構成するのではなく、潜在表現(latent)空間で未来や欠損部分を予測して表現を学ぶ」ことを提案しました。利点はラベルなしデータからの強い表現学習と、無意味な細部の再構成を避けられること、トレードオフは表現が崩れる(collapse する)のを防ぐ設計が要ることです。

運転に特化した世界モデルとしては GAIA-1(Hu et al., arXiv:2309.17080, 2023)などが、運転シーンの未来を生成的に予測する大規模モデルを示しました。反応クラスへの切り出しは、こうした「反応部品」と「世界モデル部品」を別々に育てて後で組み合わせる準備にあたります。

## 論文の実験結果(定量データ)
リファクタリング自体は定量ベンチマークの対象ではないので、関連する設計上の知見を数値で押さえます。

リファクタリングのコストについては、ソフトウェア工学の実証研究で「公開 API 変更は影響範囲が広い」ことが繰り返し示されています。1つのインターフェース変更が数十から数百の呼び出し箇所・テストに波及するのは珍しくなく、これがテストスイートの追従コストとして現れます。テストの裏付けがあるコードでは、リファクタ後の回帰(意図しない挙動変化)を検出できる一方、テストが旧 API のままだと「壊れたのか、テストが古いだけか」の切り分けに余計な時間がかかります。

世界モデル/自己教師あり予測の効果は明確に数値化されています。World Models(2018)は、世界モデルの中で(実環境を使わず)方策を学習しても、CarRacing 環境で当時の最高スコアを更新したと報告しました。I-JEPA(2023)は ImageNet の線形評価(凍結した表現の上に線形分類器だけを載せて精度を測る方法)で、ピクセル再構成型の自己教師あり手法より少ない計算量で同等以上の精度を達成し、表現学習の効率が良いことを示しました。これらは「未来や欠損を潜在空間で予測する補助タスクが、強い表現を生む」というアブレーション的な裏付けになっています。逆に言えば、補助予測ブランチを外す(本トピックでいう世界モデル部品を宙吊りにしたまま使わない)と、その表現学習の恩恵は得られません。

## メリット・トレードオフ・限界
メリットは次の通りです。
- 反応的処理を独立クラスに凝集し、将来その外側に世界モデル部品を並べる拡張の土台ができる。
- forward の戻り値とプランナ/融合の選択肢が単純化され、学習ループ側がデコーダの種類を知らなくてよくなる。
- 公開 API が外側クラスに集約され、内部構造を変えても入り口を安定させやすい。

トレードオフと限界は次の通りです。
- パラメータ名に接頭辞が付き、旧チェックポイントとの鍵の互換が崩れる(移行が必要)。
- 公開 API 変更でテストが大量に赤くなり、追従コストが発生する。
- 世界モデル部品が構築されるのに forward から呼ばれない宙吊り状態が残るなど、リファクタ途中の小さな負債が生じやすい。古いデフォルト値が内部に残り、特定の経路でだけ例外になる、といった地雷も典型的。

研究上の未解決課題は、(1)反応的ポリシーと世界モデルをどう情報的に結合するか(単に並べるだけでなく、世界モデルの予測を反応側にどう還元するか)、(2)2つの時間スケールを1つの学習ループでどう協調させるか、(3)世界モデルの予測誤差が下流の運転に与える影響の制御、です。

## 発展トピック・研究の最前線
階層的・マルチタイムスケールな運転モデルは活発な領域です。低頻度で長期計画、高頻度で反応制御という階層は、ロボティクスの階層的強化学習とも通じます。

潜在空間での世界モデルは、運転に限らず Dreamer 系(潜在空間で想像して方策を学ぶ)などへ広がっています。運転では、未来のシーン表現を予測する補助タスクが知覚表現を強化する、という方向の研究が続いています。

API 設計とソフトウェア負債の観点では、宙吊りモジュールや陳腐化したデフォルト値のような「リファクタ途中の負債」をどう検出・返済するかが、大規模 ML コードベースの保守性として重要なテーマになっています。
