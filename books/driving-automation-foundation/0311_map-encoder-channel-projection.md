# 地図エンコーダのチャンネル射影

## ひとことで言うと
地図を読み込むニューラルネットワーク(地図エンコーダ)が吐き出す特徴量の「チャンネル数」を、カメラ画像から作った特徴量のチャンネル数(典型的には 256 や 512 など)にピッタリ揃えるための小さな変換層の話です。揃えておかないと、地図とカメラの情報を足し合わせたり結合したりできません。この役目を 1x1 の畳み込み(Conv)が軽量に担います。深層学習のあらゆる場所で使われる、地味だが本質的な「形を合わせる」道具です。

## 直感的な理解
2つの言語の通訳を考えてみてください。片方が日本語、もう片方が英語をしゃべっていたら、そのままでは会話になりません。どちらかを共通の言語に翻訳して初めて、意味のあるやり取りができます。ニューラルネットワークでも同じことが起きます。カメラから抽出した特徴と地図から抽出した特徴は、それぞれ別の「次元数(チャンネル数)」を持っていて、そのままでは足し合わせることも結合することもできません。チャンネル射影とは、地図側の特徴を「カメラ側と同じ言語(同じチャンネル数)」に翻訳する変換のことです。

なぜ揃える必要があるのか。自動運転の判断は、カメラに映る世界と地図が教える世界の両方を使いたいからです。交差点で右折するとき、カメラだけでは「この先が右折レーンか」は分かりにくいですが、地図には車線の形がはっきり描かれています。だから両者を融合(fusion、複数の情報源を一つにまとめること)したい。ところが融合の演算(足し算や結合、注意計算)は、特徴量の「形」が揃っていないと数学的に成立しません。そのための前処理が、このチャンネル射影です。

## 基礎: 前提となる概念

まず「チャンネル」とは何かを説明します。普通のカラー画像は (高さ H, 幅 W, 3) という形で、最後の 3 は R(赤)・G(緑)・B(青)の3チャンネルです。ニューラルネットワークの途中で生まれる特徴量(feature、ネットワークが画像から抽出した数値の塊)も同じ考え方で、(チャンネル C, 高さ H, 幅 W) という形を持ちます。ただし C はもう色ではなく「ネットワークが学習した抽象的な特徴の種類数」で、64・256・768 など、層によってバラバラの大きな数になります。「この位置に縦エッジがあるか」「この位置は道路っぽいか」といった意味を、それぞれ別のチャンネルが担っている、とイメージするとよいです。

次に「特徴マップ(feature map)」。これは (C, H, W) の3次元配列で、空間方向(H×W)に並んだ各位置に C 個の数値がぶら下がっている、と捉えます。CNN(畳み込みニューラルネットワーク)や Vision Transformer の各層は、この特徴マップを次々と変換していきます。

そして「畳み込み(convolution)」。これは小さな窓(カーネル、たとえば 3x3)を画像の上で滑らせ、窓の中の値に重みをかけて足し合わせる操作です。3x3 のカーネルなら「自分とその周囲8マス」を見て新しい値を作ります。つまり通常の畳み込みは空間的なご近所を見ます。ここで重要なのが「カーネルサイズ 1x1 の畳み込み」です。窓が 1x1 だと、空間的なご近所は一切見ず、その1点に立っている C 個の値だけを使って新しい値を作ります。これがチャンネル射影の主役です。

最後に「融合の演算」。2つの特徴マップを統合する代表的な方法は3つあります。要素ごとの加算(element-wise add、同じ位置・同じチャンネルの数を足す)、チャンネル方向の結合(concatenate、チャンネルを縦に積み重ねる)、そしてクロスアテンション(片方をクエリ、もう片方をキー/バリューにして注意計算する)です。加算は両者のチャンネル数が完全一致していないと不可能です。結合は一致していなくてもできますが、その後の処理を簡潔に保つため、揃えておくのが普通です。クロスアテンションも、内積を取るクエリとキーの次元が一致している必要があります。どの方式でも「揃える」前処理が効いてきます。

## 仕組みを詳しく

ラスタ地図(画像として描かれた地図)をエンコードして、カメラ由来の鳥瞰図(BEV、Bird's Eye View、真上から見下ろした視点)特徴と融合する典型的なパイプラインを例に、処理を順に追います。チャンネル数の具体値は実装によって変わりますが、ここでは「画像側の埋め込み次元(embed_dim)が 256」という代表的な設定で説明します。

入力は (B, 3, 256, 256) のラスタ地図画像とします。B はバッチサイズ(一度に処理する枚数)、3 は RGB、256x256 は画像の高さと幅です。

ステップ1 バックボーン通過。Swin Transformer のような階層型バックボーン(backbone、画像から特徴を抜き出す土台ネットワーク)に通します。階層型バックボーンは段階的に解像度を下げながらチャンネル数を増やすので、複数ステージそれぞれが特徴量を返します。各ステージのチャンネル数は典型的に 96, 192, 384, 768 のように倍々で増えます。「全ステージの特徴を取り出すモード(features_only と呼ばれることが多い)」を使うと、これら全部が手に入ります。浅いステージは細かい線(車線境界など)を、深いステージは大局的な配置(交差点全体の形など)を捉える傾向があり、両方を融合に使いたいので全段を取ります。

ステップ2 空間サイズの統一。各ステージは高さ・幅がバラバラ(64x64, 32x32, 16x16, 8x8 など)なので、適応的プーリング(adaptive pooling、出力サイズを指定するとそれに合わせて区切って代表値を取る操作)で全部を同じ空間サイズ、たとえば 8x8 に揃えます。最大値を取る max pooling を使えば「その領域で最も強く反応した特徴」を残します。

ステップ3 チャンネル方向の結合。統一した各ステージ特徴をチャンネル方向に積み重ねます。96+192+384+768 = 1440 チャンネルになり、形は (B, 1440, 8, 8) になります。これで全ステージの情報が1つの太い特徴マップに合流しました。

ステップ4 チャンネル射影(このノートの本体)。1x1 Conv で 1440 チャンネルを目標の 256 チャンネルへ変換します。典型的なコードは次のようになります。

```python
channel_proj = nn.Sequential(
    nn.Conv2d(in_channels=1440, out_channels=256, kernel_size=1),  # 1440 -> 256
    nn.GELU(),
)
```

`kernel_size=1` の畳み込みは、空間的にはご近所を一切見ず、各ピクセル位置のチャンネル方向だけを線形変換します。8x8 の 64 個の位置それぞれで、1440 個の数値を重み行列(形は 256×1440)で 256 個へ写します。重みは全位置で共有されます。最後に GELU(Gaussian Error Linear Unit、ReLU に似た滑らかな非線形活性化関数)を通して非線形性を足します。

before/after を整理します。

- before(バックボーン結合直後): (B, 1440, 8, 8)
- after(チャンネル射影後): (B, 256, 8, 8)

これでチャンネル数が 256 になり、カメラ由来の 256 チャンネル BEV 特徴と要素ごとに足したり結合したりできるようになります。

なぜ 1x1 Conv なのか。同じことを全結合層(fully-connected、全部の数値をつなぐ層)でやると、空間位置ごとに別々の重みを持つことになり、パラメータが膨大(8×8×1440×256 ≈ 2.4 億)になります。1x1 Conv は「どの位置でも同じチャンネル変換ルールを使う」ので重みは 1440×256 ≈ 37 万で済み、しかも空間構造(8x8 の配置)を壊さずに残せます。空間的な並進等価性(translation equivariance、入力が平行移動すれば出力も同じだけ平行移動する性質)も保たれます。これが軽量な整合機構として 1x1 Conv が選ばれる理由です。

数式で書くと、出力位置 (i, j) のチャンネル o の値は

  y[o, i, j] = GELU( Σ_c W[o, c] · x[c, i, j] + b[o] )

で、Σ_c は入力チャンネル c についての和、W[o, c] は出力チャンネル o と入力チャンネル c をつなぐ重み、b[o] はバイアスです。空間添字 (i, j) は和に入っていません。これが「空間を見ない」ことの数式的な意味です。ASCII で位置 (i,j) 一点の変換を描くと、次のようになります。

```
入力チャンネル(1440本)        重み行列 W (256×1440)        出力チャンネル(256本)
[x0, x1, ..., x1439]    ×    [256行の線形結合]      =     [y0, y1, ..., y255]
  ↑ 同じ空間位置(i,j)に立つベクトル              全位置でこの W を共有
```

## 手法の系譜と主要論文

1x1 Conv によるチャンネル射影は、特定の地図融合論文の発明ではなく、深層学習全般で広く使われる定番テクニックです。系譜をたどります。

Network in Network (Lin, Chen, Yan, ICLR 2014, arXiv:1312.4400)。1x1 Conv を最初に明示的に導入した論文です。当時は「cross-channel parametric pooling(チャンネル横断のパラメータ付きプーリング)」と呼ばれ、各空間位置でチャンネル方向の小さな多層パーセプトロン(mlpconv)を走らせる発想でした。動機は「チャンネル間の非線形な組み合わせを安価に表現する」こと。これにより表現力が上がり、後段の全結合層を大きなグローバル平均プーリングで置き換えられるようになりました。限界は、空間的なご近所情報をこの層自体は使わない点で、空間特徴抽出は別の層に任せる前提でした。

GoogLeNet / Inception (Szegedy et al., CVPR 2015, arXiv:1409.4842)。1x1 Conv を「次元削減(dimension reduction)」として体系的に使い、計算量を抑えつつ深いネットワークを組みました。たとえば 3x3 や 5x5 の高コストな畳み込みの前に 1x1 Conv でチャンネルを減らすことで、演算量を桁で削減しました。チャンネル射影の「安く揃える/減らす」用途はこの思想の延長です。

ResNet (He et al., CVPR 2016, arXiv:1512.03385)。ボトルネック構造で 1x1 Conv によりチャンネルを減らし、3x3 で処理し、また 1x1 で戻す形を使います。さらにスキップ接続でチャンネル数が合わないとき、1x1 Conv で次元を揃える「projection shortcut(射影ショートカット)」を導入しました。これはまさに本ノートの「形を合わせる」用途そのものです。論文の実験では、恒等ショートカットが使えない次元変化部分でのみ projection を使う方式が、計算コストと精度のバランスで良いと報告されています。

地図と知覚の融合という応用文脈では次が代表的です。VectorNet (Gao et al., CVPR 2020, arXiv:2005.04259) は地図と物体軌跡をベクトル(折れ線)として符号化し、各サブグラフをエンコードした後に共通次元のノード特徴へ揃えてグローバルな相互作用を計算します。HDMapNet (Li et al., ICRA 2022, arXiv:2107.06307) はカメラ/LiDAR の特徴を BEV 平面に投影してから統合します。これらはいずれも内部で「異なる出自・異なる次元の特徴を共通次元に揃えてから融合する」前処理を行っており、チャンネル射影はその最も基本的な構成要素という位置づけです。Transformer 系の検出・融合(DETR や BEVFormer など)でも、バックボーン出力チャンネルを Transformer の隠れ次元(d_model)に合わせる 1x1 Conv が入力アダプタとして必ず登場します。BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270) では、ResNet/ViT 系バックボーンの多段出力を 1x1 Conv で同一次元へ揃えてから、BEV クエリのキー/バリューに供給します。

## 論文の実験結果(定量データ)

1x1 Conv 単体の効果を定量で示した代表例として GoogLeNet の計算量削減があります。Inception モジュールに 1x1 Conv の次元削減を入れることで、5x5 畳み込みの入力チャンネルを大幅に減らし、モジュールあたりの乗算加算回数を数倍規模で削減しました。これにより GoogLeNet は VGG より演算量・パラメータが小さいまま(パラメータ約 6.8M、VGG-16 の約 138M の約 1/20)、ImageNet 分類で top-5 誤り率およそ 6.7%(2014 年 ILSVRC で当時の最良級)を達成しています。ここで top-5 誤り率とは「正解が予測上位5位以内に入らなかった割合」で、小さいほど良い指標です。1x1 Conv がなければ同じ深さ・幅は計算量的に現実的でなかった、というのが要点です。

ResNet の projection shortcut については、論文 Table 3 で3方式(A: ゼロパディングのショートカット、B: 次元変化部のみ projection、C: 全ショートカットを projection)が比較されています。ImageNet 検証セットの top-1 誤り率(34 層)で、A が約 28.7%、B が約 28.4%、C が約 28.1% と、A→B→C の順にわずかに改善します(差はおよそ 0.3〜0.6 ポイント)。一方 C は全ショートカットに 1x1 Conv を足すぶんパラメータと計算が増えます。論文は「効果はわずかで、メモリ・計算の節約を優先し B(必要な所だけ projection)を採用する」と結論しています。これは「チャンネル射影は便利だが、闇雲に増やすと計算コストだけ上がって割に合わない」という実務的な教訓を示すアブレーション(どの要素を抜くとどうなるかの比較実験)です。

地図融合の文脈では、VectorNet の論文がアブレーションで「地図情報を入れると軌道予測の誤差が下がる」ことを示しています。Argoverse 予測ベンチマークで、地図サブグラフを統合した構成が、地図なし構成に比べて minADE(最小平均変位誤差、予測した複数候補のうち最良のものと正解軌道の平均ずれ。単位はメートル、小さいほど良い)で改善し、当時の最良級の性能を報告しています。融合が効くためには「揃えてから足す」前処理が前提であり、チャンネル射影はその下支えです。なお、チャンネル射影そのものを取り去るアブレーションは普通行われません。揃えないと融合演算が定義できず学習が回らないため、これは「あるかないか」ではなく「融合の入口として必須」の部品だからです。

## メリット・トレードオフ・限界

メリット
- 異なる出自(地図バックボーン)の特徴を、画像側の埋め込み次元(例 256)に確実に揃えられ、融合層を単純な加算や結合、クロスアテンションで書けるようになる。
- 1x1 Conv なので軽量。重みは in_ch × out_ch のみで、空間サイズに依存しない。
- 空間構造を壊さずチャンネルだけ変換でき、並進等価性も保たれる。
- 非線形活性化(GELU など)を挟むことで、単なる線形射影以上の表現力を持てる。

トレードオフ・限界
- 大きなチャンネル数(例 1440)から 256 へ減らすと情報圧縮が起き、捨てられる情報がある。逆に増やす方向の射影は冗長を生みうる。
- 射影の重みは学習で決まるため、学習データに偏りがあると地図情報の使い方も偏る。
- チャンネルを揃えるだけでは「意味的に対応が取れている」保証はない。空間的な対応(地図上の位置とカメラ特徴上の位置が一致しているか)はプーリング解像度や後段の融合・アテンション層に依存する。チャンネル射影は次元を合わせるだけで、座標系の整合は別問題である点は研究上しばしば見落とされる。
- GELU のような非線形を挟むと、純粋な線形変換に比べて勾配の流れや初期化の感度がわずかに増す。射影層を深くしすぎると過学習のリスクもある。未解決の論点として、「どこで(どのステージで)、どの解像度でチャンネルを揃えるのが融合に最適か」は依然タスク依存で、自動探索(NAS など)の余地がある。

## 発展トピック・研究の最前線

チャンネル射影は近年、より洗練された「特徴アダプタ」へ発展しています。第一に、Squeeze-and-Excitation(Hu et al., CVPR 2018, arXiv:1709.01507)のように、1x1 Conv を使ってチャンネルごとの重要度を動的に再重み付けする手法。単に次元を変えるだけでなく、どのチャンネルを強調するかをデータ依存で決めます。SENet は ImageNet で ResNet-50 比 top-1 誤り率を約 1 ポイント以上下げ、わずかな計算増で効くことを示しました。第二に、Transformer ベースの融合における線形射影(linear projection)。BEVFormer や DETR では、画像特徴を Transformer の d_model 次元へ射影し、クロスアテンションで地図クエリと相互作用させます。第三に、マルチスケール融合(FPN: Feature Pyramid Networks, Lin et al., CVPR 2017, arXiv:1612.03144)。各解像度の特徴を 1x1 Conv で同一チャンネル(典型的に 256)へ揃えてから足し合わせる構造で、チャンネル射影が多段の橋渡しとして機能します。

研究の最前線では、異なるモダリティ(カメラ・LiDAR・地図・テキスト)を共通の潜在空間(latent space)へ射影して大規模に統合する流れが強まっています。ここでチャンネル射影は「モダリティ・アダプタ」として再解釈され、各モダリティ固有のエンコーダ出力を共通次元へ写す軽量な接続層になります。LLM とビジョンを橋渡しする視覚言語モデル(例 LLaVA の projection layer)も、本質的には同じ「次元を合わせて融合可能にする」1x1 相当の線形射影です。地味な部品が、マルチモーダル統合の中心的な接合部として再評価されている、というのが現在の位置づけです。
