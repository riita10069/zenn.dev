# モジュール剪定リファクタ

## ひとことで言うと
研究プロジェクトで「どの方式が良いか」を比較するために並行実装していた複数の部品（モジュール）のうち、勝ち残った方式が固まった段階で、もう使わない部品を思い切って削除し、コードベースを身軽にする整理作業です。融合方式やプランナの選択肢を1〜2個に絞ることで、学習ループの分岐を消し、保守性とテスト効率を高めます。研究の探索フェーズ（手を広げて多数の案を試す段階）から収束フェーズ（方針が定まる段階）への移行を、コードに反映させる行為と言えます。

## 直感的な理解
庭木の剪定（pruning、せんてい）を思い浮かべてください。木を健康に育てるには、枯れた枝や混み合った枝を切り落とし、養分を残す枝に集中させます。切らずに放置すると、栄養が分散し、見通しが悪くなり、病気の枝が他に害を及ぼします。

研究のコードベースでも同じことが起きます。プロジェクトの初期は「とりあえずいろいろな案を全部実装して比較する」探索フェーズです。複数カメラの統合方式を3種類、軌跡生成プランナを3種類実装し、どれが良いか実験で比べます。これは正しい進め方です。比較対象が手元にないと「どの方式が優れているか」を主張できないからです。しかし、実験を重ねて方針が定まると、負けた方式は「死んだコード（dead code、もう使われないのに残っているコード）」として残り続けます。死んだコードには明確な害があります。

- 新しく参加した研究者やエンジニアが「結局どれを本番に使うのか」分からず混乱する。
- 使わない部品の分までテストとメンテナンスが必要になり、CI（自動テスト）の時間と人手を食う。
- 学習ループに「どの方式か」を切り替える条件分岐が残り、コードが複雑になりバグの温床になる。
- メモリや計算を大量に食う重い旧方式が、設定ミスで誤って選ばれて事故を起こす余地が残る。

剪定リファクタは、勝ち残った方式が確定した段階でこれらの枝を切り落とし、コードベースを健康に保つ作業です。

## 基礎: 前提となる概念
リファクタリング（refactoring）とは、Martin Fowler が体系化した用語で、「外から見たプログラムの振る舞いを変えずに、内部構造を改善する」作業を指します。機能追加とは区別され、バグを増やさずに読みやすさ・保守性を高めるのが目的です。剪定（不要コードの削除）は、リファクタリングの一形態です。「振る舞いを変えない」がポイントで、残した方式の出力は剪定の前後で完全に同じであるべきです。

融合（fusion）とは、複数のセンサ（ここでは複数カメラ）からの情報を1つの表現にまとめる処理です。自動運転モデルは前後左右など複数のカメラ映像を同時に受け取るため、それらをどう統合するかが性能を左右します。

プランナ（planner）とは、車やロボットの将来の軌跡を生成する部品です（詳細は [差し替え可能Plannerインターフェース](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1014_swappable-planner-interface)）。

GRU（Gated Recurrent Unit）とは、時系列を1ステップずつ順番に処理する回帰型ニューラルネットワークの一種です。前のステップの計算結果を次のステップに渡すため、本質的に逐次的（並列化しにくい）で、長い系列ほど計算が遅くなる弱点があります。1ステップ目を計算し終えないと2ステップ目に進めない、という直列の依存があるためです。

レジストリ（registry）とは、「名前 → クラス」の対応表です。選択肢を1つに絞るとは、このレジストリのエントリを削ることに相当します。

## 仕組みを詳しく
典型的な大規模剪定リファクタが何を行うか、要素ごとに分解します。具体例として、ある運転モデルの整理を想定します。

第一に、融合方式の一本化。Concat（各カメラ特徴を単純に並べて結合）、Cross Attention（注意機構で混ぜる）、BEV（鳥瞰図にまとめる）の3案から、BEV を唯一の標準に決めて他2つを削除します。理由は、BEV が他の最先端手法と最もよく揃い、別途エンコードした地図特徴と同じ座標系で統合でき、カメラの取り付け位置の違いも投影で吸収できるためです（詳細は [BEV融合を標準融合に統一](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0926_bev-as-standard-fusion)）。結果として融合レジストリは1択になります。

第二に、重いプランナの削除。GRU で軌跡を逐次生成するプランナは、メモリ消費が大きく推論も遅いため削除し、並列計算しやすく多峰性を表現できる生成型プランナ（フローマッチング、ベジエ曲線など）を残します。GRU は1ステップずつ計算するため、長い軌跡ほど遅延が線形に積み上がるのが弱点でした。

第三に、責務の整理。プランナが返していた中間状態（自車の隠れ状態のような出力）を廃止し、「視覚的履歴の生成や将来状態の予測は別のモジュールの責任、軌跡生成はプランナの責任」と役割を明確に分けます。各部品が何をすべきかを単一責任（single responsibility、1つの部品は1つの理由でのみ変更されるべき、という設計原則）に絞ることで、依存関係が単純になります。

第四に、アーキテクチャの再編。モデル全体を「速い反応を担う部分」と「遅い熟考を担う部分」に分ける二段構えへ進む布石として、クラス構成を整理し、将来実装する部分のプレースホルダ（場所の確保）を置きます。

第五に、loss 経路の整理。全プランナが同一形式の軌跡を出すようになり、共通の軌跡 loss が外側に用意されたため、各プランナに重複していた古い loss 経路を削除します。方式固有の目的関数はモジュール内に残します（詳細は [モジュール内lossカプセル化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0976_loss-inside-module)）。

剪定の前後を図示します。

```
[剪定前]  融合: {concat, cross_attn, bev}     プランナ: {gru, flow_matching, bezier}
          学習ループに方式ごとの分岐が散在
          プランナが中間状態を外に出力
          loss 経路が各プランナに重複

[剪定後]  融合: {bev}                          プランナ: {flow_matching, bezier}
          分岐なし、責務分割、共通 loss は外側
          速い反応 / 遅い熟考 の二段構えへ準備
```

なぜ「とりあえず残しておく」ではなく削除するのか。バージョン管理（Git など）があれば、削除したコードはいつでも履歴から復元できます。一方、残し続けるコストは継続的に発生します。死んだコードは他の変更（共通インターフェースの改修など）に巻き込まれて壊れ、誰も気づかないまま動かなくなります。復元しても現在の API と噛み合わず、結局書き直しになることも多い。「いつでも戻せるなら、今は消す。本当に必要になったら履歴から起こして直す」が原則です。これは Fowler の言う「コードの臭い（code smell）を放置しない」姿勢の実践です。

## 手法の系譜と主要論文
剪定リファクタは、ソフトウェア工学の原則と、運転モデル研究の収束という2つの流れの交点にあります。

ソフトウェア工学側の起源は、Martin Fowler の Refactoring（初版1999、第2版2018）です。「死んだコードの除去（Remove Dead Code）」「条件分岐をポリモーフィズムで置き換える（Replace Conditional with Polymorphism）」といったリファクタリングのカタログが、剪定の理論的根拠です。条件分岐を消してレジストリ＋共通インターフェースに置き換えるのは、まさに後者の適用です。

研究側では、運転モデルの融合方式が BEV へ収束したことが、剪定の動機を与えました。

BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）は、学習可能な BEV クエリが空間交差注意で各カメラに問い合わせ、時間自己注意で過去フレームと結びつく BEV 表現を提案しました。カメラ構成に依存しない鳥瞰図特徴を作れるため、後段の検出・計画と統一しやすく、これ以降 BEV が事実上の標準になりました。Concat や Cross Attention を切って BEV を残す判断は、この収束を反映しています。

Flow Matching（Lipman et al., ICLR 2023, arXiv:2210.02747）と Diffusion Policy（Chi et al., RSS 2023）は、残されたプランナの理論的背景です。多峰的な行動分布を、GRU の逐次生成より効率よく並列に表現できるため、GRU を切ってこれらを残す選択は、効率と表現力の両取りでした。

UniAD（Hu et al., CVPR 2023, arXiv:2212.10156）は、BEV 表現を土台に検出から計画までを一気通貫で学習する計画指向アーキテクチャを示しました。BEV を共通基盤に据えると end-to-end 化しやすいという流れが、融合方式を BEV 一本に絞る判断を後押しします。

加えて、近年の運転・ロボットアーキテクチャでは、Kahneman の「速い思考・遅い思考」（Thinking, Fast and Slow, 2011）に着想を得た二重過程（System-1 / System-2）の発想が広がっています。反射的で速い反応モジュール（System-1）と、熟慮的で遅い基盤モデル（System-2）への分割は、この流れに沿った設計です。剪定でクラス構成を整理しておくことが、こうした二段構えへの移行を容易にします。

## 論文の実験結果（定量データ）
剪定リファクタ自体は機能を変えないため固有の数値を持ちませんが、剪定先・剪定元の各方式の性能差が、何を残し何を切るかの根拠になります。

BEVFormer（2022）は nuScenes（大規模自動運転データセット、約140万フレーム、6カメラ）で、3D 物体検出の NDS（nuScenes Detection Score、検出の総合指標、大きいほど良い）でおよそ0.56を達成し、当時のカメラのみ手法を上回りました。BEV を共通基盤にすると後段タスクと整合しやすいことが、この高スコアの背景です。Concat や単純な特徴結合では、カメラ間の幾何的対応を持てず、この水準には届きません。残すべき方式が定量的にはっきりしているからこそ、迷わず他を切れます。

UniAD（2023）は、検出・追跡・地図・動き予測・占有・計画を1つの BEV 基盤モデルに統合し、計画の衝突率（planning collision rate、生成した軌跡が他物体と衝突する割合、小さいほど安全）を従来のモジュール分離型に対して大幅に下げたと報告しています（先行手法の半分以下の水準）。これは「BEV 一本化＋責務の明確な分割」というアーキテクチャ方針の有効性を示し、剪定の方向性を裏付けます。

GRU 系プランナを切る判断は、計算効率の観点から定量化できます。逐次生成は系列長 T に対して推論レイテンシが概ね O(T) で線形に増える一方、生成型プランナ（拡散・フローマッチング）は全ステップを並列に扱えるため、GPU 上では系列長が伸びてもレイテンシの増加が緩やかです。長い軌跡（数十ステップ先）を高頻度に再生成する運転用途では、この差が実時間性を左右します。たとえば64ステップの軌跡を毎フレーム生成し直す場合、GRU では64回の直列計算が必要ですが、並列型なら1回のフォワードで全ステップを同時に扱えます（フローマッチングでも数値積分のステップ数は数回程度に抑えられる）。

## メリット・トレードオフ・限界
メリット。コードが大幅に単純化し、読みやすく保守しやすくなります。学習ループから方式分岐が消え、バグの余地が減ります。重く遅い旧方式を排し、計算資源を本命に集中できます。責務分割により、今後の二段構えアーキテクチャへ進みやすくなります。テスト対象が減るため CI が速くなり、開発のイテレーションも速くなります。

トレードオフと限界。削除した方式を後で比較・復活させたくなった場合、履歴から戻す手間がかかり、戻したコードが現在のインターフェースと噛み合わない可能性があります。融合方式を1つに絞る一本化は強力ですが、その方式が不利なケース（たとえばカメラが極端に少ない、校正情報が得られない環境での BEV）に対する逃げ道を持ちません。二段構えアーキテクチャは、片方がプレースホルダのままだと未完成で、2系統の協調設計が新たな研究課題として残ります。さらに、剪定の判断が早すぎると、本来有望だった方式を「まだ十分検証していないのに」切ってしまうリスクがあります。剪定はあくまで「方針が固まった」確信が得られてから行うべきで、探索が不十分な段階での剪定は研究の選択肢を狭めます。何をいつ切るかの見極めそのものが、研究マネジメント上の判断です。

## 発展トピック・研究の最前線
コードベースの剪定は、研究の再現性とスケーラビリティの観点から再評価されています。選択肢が爆発したコードベースは、新しい貢献者の参入障壁を高め、論文の追試を困難にします。近年は、設定で方式を切り替えられる柔軟性と、選択肢を絞った単純性のバランスを取る設計（少数の検証済み方式だけをレジストリに残し、実験的な方式は別ブランチや別パッケージに分離する）が議論されています。また、二段構えアーキテクチャ（速い反応系と遅い基盤モデル系の協調）は、Vision-Language-Action モデルの台頭とともに活発な研究領域になっており、どの計算をどちらの系に割り当て、両者をどう同期させるか（速い系が遅い系の出力を待つ間どう振る舞うか）が未解決の課題です。剪定は単なる片付けではなく、「次のアーキテクチャへ進むための地ならし」として戦略的に位置づけられるようになっています。
