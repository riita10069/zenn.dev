# マルチカメラ・センサ時刻同期

## ひとことで言うと
自動運転車には複数のセンサ(カメラ何台か、LiDAR、レーダ、IMU など)が付いており、それぞれが自分のペースで自分の時計に基づいてデータを出します。タイミングがズレたまま使うと「左カメラは0.1秒前、右カメラは今」というチグハグな組み合わせになり、世界の見え方が歪んで判断を誤ります。これを「共通の時計(タイムライン)」に揃える処理を時刻同期(time synchronization)と呼びます。多くのシステムは全センサを共通の基準周波数(たとえば10Hz=0.1秒ごと)に揃え、1つのフレーム番号(frame index)で全センサと自車姿勢を同時に指せるようにします。同期は「賢いモデルを作る前に、入力データそのものが物理的に正しいか」を保証する、最下層の前提です。

## 直感的な理解
カメラ7台で撮った写真をつなげて1枚の俯瞰図(鳥瞰図、BEV: Bird's-Eye-View)を作る場面を想像してください。7台のシャッターが完全に同じ瞬間に切れていれば、写った車や歩行者はぴったり1つの位置に収まります。ところが各カメラのシャッターが少しずつズレていると、同じ車が複数の写真で違う位置に写り、貼り合わせたときに二重に見えたり消えたりします。

数値で見ると怖さがよく分かります。自車が時速36km(=秒速10m)で走っているとき、左カメラと右カメラの撮影タイミングが0.05秒ズレていると、その0.05秒の間に車は0.5m進みます。2枚を「同じ瞬間のもの」として扱えば、本来0.5m離れた位置で撮ったものを同じ位置だと誤認します。前方の歩行者の位置を0.5m誤れば、止まるべきタイミングを誤り、ブレーキ判断に直結します。だからセンサ間のズレを揃えること、つまり同期が要るのです。

もう1つの動機は「下流の実装が劇的に単純になる」点です。全センサが同じタイムラインに乗っていれば、「いま欲しい瞬間」を1つの整数(frame index)で表せます。ズレたままだと「カメラAの何番目とカメラBの何番目とLiDARの何番目が同じ瞬間か」を毎回探索する羽目になり、コードが複雑でバグの温床になります。同期は性能の問題であると同時に、データ管理の問題でもあります。

## 基礎: 前提となる概念
理解の前提として、いくつかの言葉を噛み砕いておきます。

- 動作周波数(rate / frequency): センサが1秒間に何回データを出すか。単位はHz(ヘルツ)。カメラは10〜30Hz、回転式LiDARは10Hz、IMU(慣性計測装置、加速度と角速度を測るセンサ)は100〜200Hz、というように種類ごとに違います。レートが違うこと自体が同期を難しくする第一の原因です。
- タイムスタンプ(timestamp): 各サンプルに付く「いつ計測したか」の時刻。通常はナノ秒単位の整数で、全センサが共通の基準時計(後述)に同期したクロックで打刻するのが理想です。誰がいつ打刻したか(センサ内蔵か、受信したPCか)で意味がまったく変わるため、データセットのドキュメントで「タイムスタンプの定義」を必ず確認するのが研究の作法です。
- クロック同期(clock synchronization): そもそも各センサ・各計算機の時計自体を合わせる仕組み。代表が PTP(Precision Time Protocol, IEEE 1588)で、ネットワーク越しにマイクロ秒〜サブマイクロ秒の精度で時計を一致させます。古くは GPS の PPS(Pulse Per Second, 1秒ごとの精密パルス)を基準にする方式もよく使われます。クロックが合っていなければタイムスタンプ自体が信用できないので、これは同期の大前提です。NTP(Network Time Protocol)はミリ秒精度しか出ず、車載の要求には不足することが多い点も押さえておきます。
- 露光と取得遅延(exposure / latency): カメラは「シャッターを開けている時間(露光時間)」があり、その中心時刻(midpoint of exposure)と、画像がメモリに届く時刻はズレます。明るさが変わると露光時間が変わり、ズレも変動します。ローリングシャッター(行ごとに順に読み出す方式)だと画像の上端と下端で時刻が違うことすらあります。厳密な同期はこうした遅延までモデル化します。
- 自車姿勢(ego pose): 車自身の位置(x, y, z)と向き(姿勢)。多くは6自由度(3次元位置+3次元回転)で表され、これも別レートで記録されます。姿勢は連続量なので、後述の補間が素直に効く数少ない量です。

同期には大きく2段階あることを押さえてください。(1) ハードウェア同期: 計測する瞬間そのものを物理的に合わせる(トリガ信号で同時にシャッターを切る等)。(2) ソフトウェア同期: 各々が打刻したタイムスタンプを後処理で共通タイムラインに揃える。前者は精度が高いが専用機構が要り、後者は安価だが残差(消せないズレ)が残ります。実運用では両者を組み合わせ、ハードウェアで粗く揃え、ソフトウェアで残差を吸収するのが定石です。

## 仕組みを詳しく
ソフトウェア同期の典型的な手順を分解します。

1. 共通の基準周波数(reference rate)を決める。たとえば10Hz、つまり0.1秒ごとの基準時刻 t0, t1, t2, ... を並べた「基準タイムライン」を作ります。なぜ10Hzが定番かというと、回転式LiDARの多くが10回転/秒(10Hz)で、それに合わせると点群とのズレを最小化しやすいからです。逆に高速移動物体を細かく追うタスクでは20Hz以上を採ることもあり、基準周波数の選択そのものが設計判断です。
2. 各センサの生サンプルには、それぞれ正確なタイムスタンプが付いています(クロック同期が前提)。
3. 基準時刻ごとに、各センサから「その時刻に最も近いサンプル」を選ぶ(nearest matching、最近傍マッチング)、または前後2サンプルから補間(interpolation、間の値を計算で推定)します。

   数値例: 基準時刻 t=1.000秒に対し、左カメラの最近傍フレームが t=0.998秒、右カメラが t=1.003秒なら、両者を「t=1.000秒のフレーム」として組にします。残差はそれぞれ −2ms, +3ms。自車姿勢のように連続量で補間できるものは、t=0.99 と t=1.01 の姿勢を線形補間(回転は球面線形補間 slerp)して t=1.00 の姿勢を作るほうが正確です。画像は補間できない(2枚の画像の「中間の絵」を作るのは別問題)ので、画像系は最近傍、連続量は補間、と使い分けるのが普通です。
4. こうして基準時刻1つにつき「全カメラ各1枚 + LiDAR 1スキャン + その瞬間の自車姿勢」が揃った1フレームができます。

形状(shape)で見ると、before/after はこうなります。

```
before(同期前): センサごとにバラバラのレートとタイムスタンプ
  cam_front:  [t=0.000, 0.033, 0.066, ...] 30Hz
  cam_left:   [t=0.005, 0.038, 0.071, ...] 30Hz(位相もズレる)
  lidar:      [t=0.000, 0.100, 0.200, ...] 10Hz
  ego_pose:   [t=0.000, 0.010, 0.020, ...] 100Hz

after(同期後): frame_idx 一本で全部が揃う
  frame_idx=10 (t=1.000s) → {cam x7: (7,3,H,W), lidar: (N,4), ego_pose: (4,4)}
```

回転式LiDARとカメラの同期には特別な工夫があります。回転式LiDARは全周をスキャンするのに時間がかかる(10Hzなら1周100ms)ので、レーザービームがある方向を向いた瞬間に、その方向を見ているカメラのシャッターを切るとズレが最小になります。これを位相同期(phase-locked triggering)と呼び、ハードウェアトリガで実現します。「時刻を合わせる」だけでなく「見ている方向まで合わせる」のがポイントです。点群を画像に投影してマルチモーダル教師(2D-3D対応)を作るとき、この位相同期があるかどうかで投影誤差が桁違いに変わります。

LiDARのモーション補正(ego-motion compensation)も同期と一体です。1スキャン100msの間に車自身が動くので、スキャン開始時と終了時で点群の基準座標がズレます。各点のタイムスタンプと自車姿勢を使って、すべての点を「スキャン基準時刻の座標系」に引き戻すのが標準処理です。これを怠ると、まっすぐな壁が点群上で歪んで写ります。

端(録画開始・終了付近)の扱いも実務では重要です。同期済みデータでも、録画の終わり際にカメラのフレーム数と自車姿勢のサンプル数が数個ズレることがあり、片方の最大インデックスでもう片方を引くと範囲外アクセスで落ちます。典型的な安全策は、複数の系列がともにカバーする範囲の短い方に合わせる(usable = min(len_A, len_B) のように切り詰める)ことです。地味ですが「同期は端の処理が必ず厄介になる」ことの代表例で、長尺ログを大量に流すパイプラインほど顕在化します。

## 手法の系譜と主要論文
時刻同期は独立した手法論文というより、各データセット論文・各車両プラットフォームの中で語られる基盤技術です。系譜として押さえるべきは次の流れです。

- KITTI (Geiger, Lenz, Urtasun, CVPR 2012): カメラ・LiDAR融合の先駆け。GPS/IMU(OXTS)と Velodyne LiDAR、ステレオカメラを搭載し、LiDARの回転に合わせてカメラを撮るトリガを使いました。同期の重要性を実データで広く知らしめた最初期の例です。
- nuScenes (Caesar et al., nuTonomy/Motional, CVPR 2020): 都市運転をフルセンサ(6カメラ全周 + 1 LiDAR + 5レーダ)で記録。カメラがLiDARビームの向きを向いた瞬間に撮るハードウェアトリガ(位相同期)を採用し、点群と画像の投影誤差を最小化しました。動機は「マルチモーダルで都市運転を正しく評価する土台が欲しい」こと。専用同期機構のコストはかかるが、後続研究のデファクト標準になりました。
- Waymo Open Dataset (Sun et al., Waymo, CVPR 2020): 5カメラ + 5 LiDAR をより高頻度・高解像度で同期。同期精度の高さがアノテーション品質の前提になっており、3D検出・追跡・モーション予測ベンチマークの信頼性を支えています。
- 連続時間推定の系譜 (Furgale et al., ICRA 2012; Kalibr, IROS 2013): カメラ・IMU 等を「フレーム単位」でなく連続時間の軌道(B-スプライン基底)として推定し、センサ間の時間オフセットそのものをパラメータとして最適化する枠組みを確立しました。Kalibr ツールキットはカメラ-IMU の時間ずれと外部パラメータを同時に校正するデファクトとなり、「同期はデータから推定できる」という現代的視点を与えました。
- 産業界のクロック基盤: 車載・ロボティクスでは PTP(IEEE 1588)や GPS-PPS をネットワーク全体の時計基準にし、各センサのドライバがハードウェアタイムスタンプ(NICやキャプチャ基板が打刻)を付ける構成が主流になりました。ソフトウェア側(ROS 2 の message_filters の ApproximateTimeSynchronizer など)で最近傍マッチングを行う層と組み合わせます。

共通する教訓は「同期はモデルの賢さ以前の、データの正しさの問題」であり、ここが崩れると検出・追跡・予測・計画の下流すべてが崩れる、ということです。

## 論文の実験結果(定量データ)
同期そのものは「精度何%」という単独指標で測りにくいですが、関連する定量的事実を挙げます。

- nuScenes はカメラ・LiDAR間のタイムスタンプ差を抑える設計を採り、6カメラ全周で約140万枚の画像と約140万個の3D箱アノテーションを整合の取れた状態で提供しました。3D検出ベンチマークの主指標 NDS(nuScenes Detection Score、mAPと並進・スケール・向き・速度・属性の誤差を統合した0〜1のスコア。1に近いほど良い)や mAP(mean Average Precision、検出の適合率・再現率を統合した平均精度)は、点群と画像が同じ瞬間を指していることを暗黙の前提にしています。同期がズレれば、LiDAR点を画像に投影した教師(2D-3D対応)がズレ、mAP は系統的に劣化します。
- 一般に、同期残差 Δt とそれが生む位置誤差は Δx ≈ v · Δt で効きます。v=10m/s, Δt=10ms なら Δx=0.1m、v=30m/s(高速道路, 約108km/h), Δt=10ms なら Δx=0.3m。高速になるほど同じ時間ズレが大きな空間ズレになり、要求同期精度が厳しくなることが式から直接読み取れます。逆に言えば「許容できる同期精度」は走行速度から逆算でき、高速道路用システムはサブミリ秒級の同期を要求します。
- ローリングシャッターの影響: 行ごとの読み出し時間が数ms〜十数msあると、高速移動物体は画像内で歪み(skew)を生じます。たとえば読み出し時間20ms・横方向相対速度10m/sなら、画像の上端と下端で物体が最大0.2mズレて写ります。グローバルシャッター(全画素同時露光)カメラに替える、あるいは行ごとのタイムスタンプを補正することで、移動物体の3D投影誤差を数十cm規模で削減できると報告されています。
- 時間オフセット校正の効果(Kalibr 系の知見): カメラ-IMU の時間オフセットを未校正のまま使うと、VIO(視覚慣性オドメトリ)の軌道推定が系統的にドリフトします。オフセットをデータから推定して補正すると、推定軌道誤差が顕著に下がることが報告されており、「数msのオフセットでも累積すると大きい」ことの定量的裏付けです。
- アブレーション的な観察: 同期を意図的に乱したデータで学習・評価すると、近距離・高速の物体ほど検出・追跡 ID の安定性(追跡で同じ物体を同じIDで追えるか)が崩れます。これは「テールではなく日常的なシーンですら、同期が悪いと壊れる」ことを示し、同期が土台である根拠になっています。

## メリット・トレードオフ・限界
メリット
- 全センサが同じ瞬間を指すので、BEV合成・3D検出・軌跡予測の位置誤差が系統的に減る。
- 共通タイムライン(frame index 一本化)で、データセットのインデックス管理が劇的に単純になり、バグが減る。
- 下流タスク(認識・予測・計画)が「同じ瞬間のスナップショット」を前提に設計でき、モデル側で時間ずれを学習で吸収する負担が消える。

トレードオフ・限界
- 厳密な同期(位相同期トリガ、PTP、ハードウェアタイムスタンプ)には専用機構が要り、構築コストとキャリブレーション手間が大きい。
- 最近傍マッチングや補間には必ず微小な残差が残る。完全な同時計測は物理的に不可能で、露光時間・読み出し遅延・伝送遅延がゼロにはならない。
- 録画の端ではセンサごとにサンプル数がズレ、範囲を切り詰める追加処理が要る。長尺ログを大量に扱うほど、この端処理のバグが顕在化しやすい。
- 高速走行ほど許容ズレが小さくなる(Δx ≈ v·Δt)。同じデータセットでも高速シーンの方が同期要求が厳しい。
- 補間で異種レートを揃えると、補間誤差自体が学習・評価の系統誤差になる。とくに加速度のような高階の量や、画像のように補間できない量では破綻しやすい。
- 未解決寄りの課題: 異種レートのセンサ(100Hz IMU と 10Hz LiDAR と 30Hz カメラ)を「単一フレームに潰す」と本来の時間情報が失われる。補間に頼らず、各センサの素のタイムスタンプをモデルに入力して時間差を陽に扱う(continuous-time / asynchronous fusion)研究も進んでいるが、まだ標準化されていない。

## 発展トピック・研究の最前線
- 連続時間・非同期融合: 全センサを同一フレームに無理やり揃えず、各観測のタイムスタンプを保持したまま、連続時間の軌道や状態をスプライン/ガウス過程で表して融合する手法。補間で潰す情報を残せる点が利点。Furgale 系の連続時間推定がこの源流です。
- イベントカメラ(event camera): 画素ごとに輝度変化を非同期に出力するセンサ。そもそも「フレーム」という概念がなく、マイクロ秒解像度のため、従来のフレーム同期の枠組みを問い直す存在。高速移動・高ダイナミックレンジに強いが、データ表現が従来CNNと合わず、新しい処理系が必要です。
- 学習による時間ズレ補正(temporal calibration): カメラ・LiDARの時間オフセットを、データから自動推定する自己キャリブレーション。手動同期の残差をモデルが吸収する方向。online calibration として走行中に継続推定する研究もあります。
- マルチエージェント/V2X同期: 自車のセンサ間だけでなく、複数車両・路側機(infrastructure)のセンサを共通時刻に揃える協調認識。クロック基盤(PTP/GPS)の精度がそのまま協調の精度上限になり、通信遅延も加わるため、地上の単車より同期がさらに難しくなります。
- 時間ずれにロバストな融合モデル: 同期を完璧にする代わりに、ある程度の時間ずれを前提として、それでも崩れないようにモデル側で時間埋め込み(time embedding)やアテンションで吸収する設計も研究されています。
