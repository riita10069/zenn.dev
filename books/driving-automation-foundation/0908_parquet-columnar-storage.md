# Parquet列指向ストレージ

## ひとことで言うと

Parquet(パーケイ)は、大量の表形式データを「列ごとにまとめて圧縮して」ファイルに保存する形式です。必要な列だけを高速に読み出せて、しかもファイルサイズが小さくなるため、機械学習や自動運転のデータパイプライン(データを集めて整形し学習に渡すまでの一連の流れ)で広く使われています。

## 直感的な理解

まず「表形式データ」とは、Excel のような行と列の表のことです。たとえば自動運転の走行ログを思い浮かべてください。

```
timestamp      speed   steering   lidar_points   camera_path
1000           12.3    0.01       120000         img_0001.jpg
1001           12.5    0.02       119500         img_0002.jpg
1002           12.4    0.00       121000         img_0003.jpg
...(数百万行)...
```

このデータをファイルに保存する方法は大きく2つあります。

行指向(row-oriented): 1行ぶんのデータをまとめて書き、次の行を書き、と続ける方法です。CSV や一般的なデータベースがこれです。「ある1台の車のあるフレームの全情報」をまとめて取り出すのは得意ですが、「全フレームの speed 列だけ集計したい」というときは、欲しくない steering や lidar_points も含めて全部読み込まないといけません。

列指向(columnar): 同じ列の値を連続して書く方法です。speed の値を全部まとめて書き、次に steering を全部、と続けます。

```
行指向: [1000,12.3,0.01,...] [1001,12.5,0.02,...] [1002,...] ...
列指向: [1000,1001,1002,...] [12.3,12.5,12.4,...] [0.01,0.02,0.00,...] ...
          timestamp列          speed列              steering列
```

列指向の利点は2つです。第一に、speed 列だけ欲しいときは speed の塊だけ読めばよく、ディスクから読むデータ量が激減します(これを「列プルーニング」column pruning と呼びます)。第二に、同じ列の値は性質が似ている(全部が速度の数値、全部がタイムスタンプ)ので、圧縮がよく効きます。隣り合う値が似ていると圧縮アルゴリズムは小さく潰せるのです。Parquet はこの列指向を、Hadoop/Spark などの大規模分散処理を前提に標準化したファイル形式です。

## 基礎: 前提となる概念

Parquet を理解するために、いくつかの土台を押さえます。

圧縮(compression)とは、データの中の繰り返しや規則性を見つけて、より短い表現に置き換えてサイズを縮める処理です。例: `12.3, 12.3, 12.3, 12.4` を「12.3 が3回、そのあと 12.4」と書けば短くなります。元に戻せる圧縮を可逆圧縮と呼び、Parquet は可逆圧縮しか使いません(数値が変わったら困るため)。

エンコーディング(encoding)は圧縮の前段で、データの並びを圧縮しやすい形に変換することです。代表例を挙げます。

- ランレングス符号化(RLE, Run-Length Encoding): 同じ値の連続を「値 × 回数」で表す。`0,0,0,0,0` を `0×5` に。
- 辞書エンコーディング(dictionary encoding): 出てくる値に番号を振り、本体は番号の列にする。例えば camera_path のような文字列が `img_0001.jpg` など限られたパターンなら、辞書 `{0:img_0001.jpg, 1:img_0002.jpg, ...}` を作り、本体は `0,1,2,...` という小さな整数列にします。整数列はさらに RLE やビットパッキングで縮みます。

スキーマ(schema)とは、各列の名前と型(整数か小数か文字列か)を定めた設計図です。Parquet はファイル自身の中にスキーマを持つ「自己記述的(self-describing)」な形式です。つまりファイルを開けば、別途定義を渡さなくても列名と型が分かります。CSV はこれを持たないため、speed が文字列か数値か読む側が推測する必要があり、トラブルの元でした。

## 仕組みを詳しく

Parquet ファイルの内部は階層構造になっています。上から順に説明します。

```
Parquet ファイル
├─ Row Group 1 (例: 128MB ぶんの行の塊)
│   ├─ Column Chunk: timestamp 列のこの行範囲ぶん
│   │   ├─ Page (圧縮・エンコード単位、例 1MB)
│   │   ├─ Page
│   ├─ Column Chunk: speed 列
│   ├─ Column Chunk: steering 列
├─ Row Group 2
│   ├─ ...
├─ Footer(末尾)
    ├─ スキーマ定義
    ├─ 各 Row Group / Column Chunk の位置と統計(min, max, null数 など)
```

行グループ(Row Group): まず全体の行を、たとえば100万行ずつといった塊に分けます。これが行グループです。1つの行グループの中だけは列指向で並びます。なぜ全体を一気に列指向にしないかというと、分散処理で「この行グループはマシンA、次はマシンB」と並列に処理したいからです。行グループは並列処理の単位でもあります。

カラムチャンク(Column Chunk): 1つの行グループ内の、1つの列ぶんのデータです。上の図で「行グループ1の speed 列」が1カラムチャンクです。

ページ(Page): カラムチャンクをさらに小さく分けた、圧縮とエンコードの最小単位です。1ページごとに辞書化・RLE・圧縮が適用されます。

フッター(Footer): ファイルの末尾に置かれるメタデータの集まりです。ここに重要な仕掛けがあります。各カラムチャンクについて、その中の最小値・最大値・null の個数といった統計情報が記録されています。

この統計が「述語プッシュダウン」(predicate pushdown)を可能にします。たとえば `speed > 50 の行が欲しい` という問い合わせをするとします。読む側はまずフッターを見て、各行グループの speed 列の max を確認します。ある行グループの speed の max が 30 なら、その行グループには 50 超えの行は存在しないので、本体を1バイトも読まずにスキップできます。条件(述語 predicate)をデータの奥(下、push down)まで持っていって早期に枝刈りする、という意味です。

数値で効果を見ます。1億行のログがあり、speed > 100(急加速イベント)の行が全体の0.1%しかないとします。行グループが1万個あり、そのうち実際に max > 100 を含むのが50個だけなら、読むデータは全体の 50/10000 = 0.5% に減ります。データ転送量が200倍減るので、クラウドでの S3 読み出し料金や読み込み時間が大きく下がります。

ネストしたデータの扱い: Parquet は単純な表だけでなく、入れ子構造(配列や構造体)も列指向で保存できます。これは Google の Dremel という論文(Melnik et al., VLDB 2010)が提案した「繰り返しレベル(repetition level)」「定義レベル(definition level)」という仕組みに基づきます。配列の各要素がどの階層の何番目かを2つの整数で表現することで、入れ子データもフラットな列に展開して保存し、読むときに元の構造へ復元できます。自動運転データで「1フレームに可変個の検出物体がある」といったケースを、列指向の効率を保ったまま扱えるのが強みです。

## 手法の系譜と主要論文

列指向の考え自体は古く、データベース研究で長く議論されてきました。系譜を追います。

- C-Store(Stonebraker et al., VLDB 2005, arXiv なし): 列指向データベースの先駆的研究。列ごと保存・圧縮・読み出しの優位性を体系的に示し、後の商用列指向DB(Vertica など)につながりました。Parquet の思想的な源流の一つです。
- MonetDB/X100(Boncz et al., CIDR 2005): 列指向とベクトル化処理(値をまとめてCPUに流す)を組み合わせ、CPUキャッシュ効率を最大化する研究。「列で持つと現代CPUで速い」という主張を補強しました。
- Dremel(Melnik et al., VLDB 2010): Google 社内の大規模分析基盤。前述の繰り返し/定義レベルにより、ネスト構造を列指向で保存する手法を確立しました。これが Parquet のデータモデルの直接の元ネタです。
- Apache Parquet(2013年〜, Twitter と Cloudera が共同開発): Dremel の論文を実装し、Hadoop エコシステム向けのオープンな列指向ファイル形式として標準化。現在は Apache Software Foundation のプロジェクトです。
- Apache Arrow(Wes McKinney ら, 2016年〜): Parquet が「ディスク上の保存形式」なのに対し、Arrow は「メモリ上の列指向表現」を標準化したもの。Parquet を読んで Arrow 形式でメモリに展開し、そのまま Pandas や機械学習へ渡す、という連携が一般的です。Parquet と Arrow はセットで語られます。

機械学習・自動運転の文脈では、Parquet 単体というより、データレイク管理レイヤと組み合わせて使われます。Delta Lake(Armbrust et al., VLDB 2020)、Apache Iceberg、Apache Hudi はいずれも内部ストレージとして Parquet を使い、その上にトランザクション(複数の書き込みを安全にまとめる仕組み)やバージョン管理(過去のデータ状態に戻れる仕組み)を載せています。学習データの再現性(あの実験で使ったデータを正確に復元する)を担保するうえで重要です。

## 論文の実験結果(定量データ)

公開ベンチマークと一次資料の数値を挙げます。数値の意味も添えます。

C-Store 論文(VLDB 2005)では、列指向DBが行指向DBに比べ、分析系クエリ(集計・絞り込み)で平均164倍高速という結果を報告しています。これは極端な条件下の値ですが、「欲しい列だけ読む」効果の大きさを示します。一方で行を1件まるごと挿入・取得する処理は列指向が不利になることも同時に示され、ワークロード依存であることが当初から明確でした。

Dremel 論文(VLDB 2010)では、85億レコード・87TB のデータに対し、列指向(Parquet の元になった表現)が行指向よりも1桁速くスキャンできること、また数千台のサーバで数秒〜数十秒で集計が返ることを報告しています。列指向はディスクI/Oを減らすだけでなく、CPUでの展開コストも含めて優位だと示しました。

実務的な圧縮率の目安: 一般的な表形式データを CSV で保存した場合と Parquet(Snappy 圧縮)で保存した場合を比べると、Parquet はおおむね 1/4 から 1/10 のサイズになることが多いです(データの繰り返しの多さに依存)。重い圧縮アルゴリズム(Zstandard, Gzip)を使えばさらに縮みますが、読み書きのCPU時間が増えます。Snappy は「そこそこ縮んで非常に速い」ため機械学習パイプラインの既定値としてよく選ばれます。

述語プッシュダウンの効果(TPC-H など標準ベンチマークでの一般的傾向): 選択率(selectivity, 条件に合う行の割合)が低いクエリほど Parquet の優位が大きく、選択率が0.1%以下なら読み出しデータ量が2桁減ることも珍しくありません。逆に「全行・全列が必要」な処理では、行指向との差はほぼなくなります。

注意: これらの数値は比較条件(データの種類・クエリ・ハードウェア)に強く依存します。自分のデータで計測するのが原則です。

## メリット・トレードオフ・限界

メリットを整理します。第一に、必要な列だけ読めるためI/Oが減ります。第二に、列ごとに性質が揃うので高圧縮が効きます。第三に、フッターの統計で不要な行グループを早期スキップできます。第四に、自己記述的でスキーマを内包し、型が明確です。第五に、Spark・Pandas・PyArrow・各種クラウドサービスが標準対応しており相互運用性が高いです。

トレードオフと限界も率直に述べます。

書き込みの一括性: Parquet は「まとめて大量に書いて、何度も読む」用途に最適化されています。1行ずつ追記する、頻繁に1行だけ更新する、といった用途には向きません。1行更新したいと思っても、列ごとに散らばっているため実際にはファイル全体を書き直すことになりがちです。リアルタイムに細かく書き換わるデータ(オンライン処理の最新状態など)には不向きで、そこは行指向のデータベースや専用フォーマットの出番です。

小さいファイル問題(small files problem): 行グループは大きいほど効率が良い(目安128MB〜1GB)ため、小さなファイルを大量に作ると逆に遅くなります。フッターやメタデータの読み込みオーバーヘッドが相対的に大きくなるためです。自動運転で「1走行=1ファイル」と細かく分けすぎると、学習時に何万ものファイルを開くことになり、ファイルオープンのコストが支配的になります。適切に統合(compaction)する運用が必要です。

ランダムアクセスの弱さ: 「特定の1行を取り出す」のは列指向の苦手分野です。その行の値が複数の列チャンクに分散しているため、複数箇所を読んで組み立てる必要があります。

ストリーミングとの相性: フッターがファイル末尾にあるため、ファイル全体を受け取らないとメタデータが読めません。ネットワーク越しに少しずつ流しながら処理する(ストリーミング)用途には向かず、そこは Arrow IPC やメッセージ形式が使われます。

機械学習特有の課題: 学習では「データをランダムにシャッフルして少しずつ取り出す」ことが多いですが、Parquet のランダムアクセスの弱さと相性が悪い場合があります。このため、画像など重い生データは別形式(WebDataset の tar シャードなど)に固め、Parquet は「軽いメタデータと特徴量テーブル」に使い分けることが実務では一般的です。

## 発展トピック・研究の最前線

レイクハウス(lakehouse): データレイク(安価なオブジェクトストレージに生データを置く)とデータウェアハウス(構造化・高速集計)の利点を統合する考え方です。Delta Lake / Iceberg / Hudi が代表で、内部は Parquet を使いつつ、その上にトランザクション・スキーマ進化(列の追加変更を安全に行う)・タイムトラベル(過去版へのアクセス)を載せます。機械学習のデータバージョニングと再現性に直結します。

Parquet の世代更新: 近年、暗号化(列単位で暗号化し、権限のある列だけ復号)、より賢いインデックス(Bloom フィルタで「この列チャンクに値xが存在しうるか」を高速判定)、ページレベルの統計など、述語プッシュダウンをさらに細かくする機能拡張が進んでいます。Bloom フィルタは「絶対に無いことは確実に分かるが、有るかどうかは確率的」なフィルタで、等値検索 `id = 12345` のような問い合わせで無駄読みをさらに削ります。

ベクトル検索・特徴ストアとの融合: 機械学習の特徴量を Parquet で管理し、Arrow 経由でGPU学習へゼロコピー(コピーなしで)に渡す研究・実装が広がっています。データI/Oが学習のボトルネックになりやすい現代では、ストレージ形式の選択が学習スループットを左右します。

GPU での Parquet 読み込み: NVIDIA の cuDF などは Parquet を GPU上で直接デコードし、CPUを介さず展開する取り組みを進めています。データ前処理のCPUボトルネックを減らす方向です。
