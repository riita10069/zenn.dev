# nuScenes CAN bus expansion

## ひとことで言うと

CAN バス拡張とは、ある運転データセットに「車そのものが内部で発している電子信号」を追加した補助データのことです。具体的には、ハンドルの操舵角・アクセルやブレーキの踏み込み・車輪の回転速度・車体の傾きや揺れ(慣性)などを、自動車内部の通信網である CAN バスから取り出した高頻度の時系列です。これによって「自分の車(ego, エゴ)が実際にどう操作され、どう動いたか」を高精度・低ノイズに知ることができ、運転モデルを学習する際の「正解(教師信号)」として使えます。

ここで「ego(エゴ)」とは、センサーを積んでデータを記録している自分自身の車のことです。周囲の他車や歩行者と区別するための言葉で、日本語では「自車」と呼びます。「教師信号(supervision signal / target)」とは、モデルが出力すべき正しい答えのことで、学習中はこれとモデルの出力の差を縮めるように重みを調整します。

## 直感的な理解

カメラ映像を入力して「ハンドルをどれだけ切るか」「アクセルをどれだけ踏むか」を直接出力する運転モデルを学習したいとします。学習には「この映像のとき、人間は実際にどう操作したか」という正解が必要です。これは模倣学習(imitation learning, 人間のお手本をまねるように学ぶ枠組み)の根幹です。

ところが多くの運転データセットが提供する自車情報は ego_pose、つまり「地図上のどこに、どの向きで居たか」という位置・姿勢の記録が中心で、しかもサンプリング間隔が知覚ラベルに合わせて粗い(たとえば 2 Hz = 0.5 秒ごと)ことがよくあります。位置の時系列から速度や加速度を得ようとすると「数値微分(隣り合う時刻の差を時間で割る)」が必要ですが、これは次の致命的な問題を抱えます。

- 微分はノイズを増幅する。位置のわずかな測定誤差が、速度ではより大きな誤差に、加速度ではさらに大きな誤差に化ける。
- そもそも「ハンドルを何度切ったか」「ブレーキを何 % 踏んだか」という人間の操作そのものは、車の軌跡(位置)からは復元できない。ハンドルを切ってから車体が向きを変えるまでには遅れがあり、軌跡だけ見ても操作量は一意に決まらない。同じ右カーブの軌跡でも、低速でゆっくり切る場合と高速で小さく切る場合で操舵角は大きく違う。

CAN バス拡張は、この問題を「車の内部信号を直接記録する」ことで根本的に解決します。車のコンピュータ同士は CAN(Controller Area Network)という通信網でつながっており、操舵・ブレーキ・速度・姿勢の状態が常時 10〜100 Hz 程度で流れています。これをそのまま記録すれば、操作も運動も高頻度・低ノイズで手に入ります。

身近な比喩で言えば、ego_pose は「散歩中に 0.5 秒おきに撮った位置の写真」、CAN は「歩いている本人の足の運び・歩幅・体の傾きを直接記録した動作ログ」です。位置の写真から歩き方を推測するより、動作ログを直接見るほうが正確で滑らかなのは明らかです。

## 基礎: 前提となる概念

CAN バス(Controller Area Network)は、1980 年代に Bosch が自動車向けに開発したシリアル通信規格です。車には数十個の ECU(Electronic Control Unit, 電子制御ユニット)があり、エンジン制御・ブレーキ制御・ステアリング制御などを担います。これらが 2 本の差動信号線(CAN-High と CAN-Low の電圧差で 0/1 を表す)でつながり、「メッセージ ID + データ」の形で互いに状態を放送(ブロードキャスト)します。標準 CAN は最大 1 Mbps、各メッセージのペイロードは最大 8 バイトです。受信側は ID を見て自分に関係するメッセージだけ拾います。メッセージ ID は優先度も兼ねており、複数 ECU が同時送信したとき ID の小さいものが優先される(調停, arbitration)仕組みです。

重要なのは、CAN メッセージの中身(どのバイトが何を意味し、どんな単位・スケーリングか)はメーカーが定義する独自仕様だという点です。これを記述したファイルを DBC(CAN database)と呼びます。たとえば「ID 0x2B0 の上位 2 バイトは操舵角を 0.1 度単位で、符号は左旋回が正、オフセット 0」といった対応表です。つまり生の CAN ビット列を物理量(度、m/s)に直すには、車種ごとの DBC 解釈が必要になります。これが CAN データを扱う上での最初の壁です。同じ「操舵角」でも、ある車では時計回りが正、別の車では反時計回りが正だったり、単位が 0.1 度だったり 1/16 度だったりします。

もう一つの前提が IMU(Inertial Measurement Unit, 慣性計測装置)です。これは加速度センサー(3 軸の並進加速度)とジャイロセンサー(3 軸の角速度)を組み合わせた装置で、車体の傾き・揺れ・旋回の速さを測ります。GPS が使えないトンネル内でも、IMU の積分で短時間の運動を推定できます。ただし加速度を 2 回積分して位置を出すと、わずかなバイアス(ゼロ点のずれ)が時間の 2 乗で蓄積する「ドリフト」が起きるため、長時間の単独利用には向きません。実用では GPS や LiDAR オドメトリと融合(センサーフュージョン)して使います。

## 仕組みを詳しく

代表的な CAN バス拡張(Motional が nuScenes 向けに 2019 年に公開したもの)には、各運転シーン(20 秒程度)に対応する形で、次のようなメッセージ群が高頻度の時系列で収録されています。

- vehicle_monitor: 操舵角・スロットル・ブレーキ・選択ギア・車速など。人間の操作と車の即時応答を表す中核データ。
- pose: 高頻度の自車位置・向き・速度(vel)・加速度(accel)・角速度(rotation_rate)。位置記録(ego_pose, おおむね 2 Hz)より細かい時間刻みで運動を追える。
- IMU: 3 軸の線形加速度と 3 軸の角速度(ジャイロ)。
- steeranglefeedback: 実際の操舵角フィードバック信号。
- zoesensors / zoe_veh_info: 収集に使った特定車両(Renault Zoe)固有の車輪速(左右別)などの内部値。

tensor として扱うなら、ある 20 秒シーンの操舵角は次のような形になります。

```
steering = [
  (t=0.000s, angle=  +2.1°),
  (t=0.020s, angle=  +2.3°),    # 50 Hz なら 0.02 秒刻み
  (t=0.040s, angle=  +2.6°),
  ...
  (t=20.00s, angle= -15.7°)
]   # 数百〜千点規模、ラベル 2 Hz の 25〜50 倍の時間解像度
```

これを模倣学習の教師としてどう使うか、before/after で見てみます。

before(位置から微分する従来流儀):
```
ego_pose: 2 Hz の (x, y) 列  →  数値微分で速度  →  さらに微分で加速度
誤差: 位置 ±5cm が速度では ±0.1m/s、加速度では ±0.2m/s² に増幅 → 教師がガタガタ
操舵角: 復元不能
```

after(CAN バス拡張を使う):
```
速度・加速度: 50 Hz の低ノイズ実測値をそのまま教師に
操舵角: 実測値をそのまま教師に
→ end-to-end モデルの「次の 0.5 秒の操舵・速度」予測の正解 [未来ステップ数, 操作次元] が高品質
```

損失関数(loss function, モデルの出力と正解のズレを数値化する関数)の観点では、モデル出力 a_hat_t(予測操作)と CAN 由来の正解 a_t(実操作)の差を取って L1(絶対値の和)や L2(二乗和)で最小化します。ここで a_t がノイズだらけだと、モデルはノイズに引っ張られて学習が不安定になります。CAN の低ノイズ教師はこの問題を避けます。とくに L2 は外れ値(大きな誤差)を二乗で強く罰するため、微分由来のスパイクノイズに非常に敏感で、CAN 実測との差が学習安定性に直結します。

時刻同期も重要な設計ポイントです。カメラ・LiDAR・CAN・GPS はそれぞれ独立のクロックで動いており、共通のタイムスタンプ(多くは Unix epoch のマイクロ秒)で後から突き合わせます。CAN は高頻度なので、画像の各フレーム時刻に最も近い CAN サンプルを線形補間して対応づけるのが定石です。同期が 100 ミリ秒ずれると、時速 50 km(秒速約 14 m)では約 1.4 m の位置ずれに相当し、「曲がる映像なのに操舵が直進」のような教師の食い違いが起き、学習を悪化させます。

なぜ高頻度が効くのかをもう一段深掘りすると、車両ダイナミクス(操作 → 運動の物理応答)には時定数があります。ハンドルを切ってからヨーレート(旋回角速度)が立ち上がるまで数百ミリ秒の遅れがあり、2 Hz サンプリングではこの過渡応答を捉えきれません。50 Hz の CAN があれば、操作と運動の因果関係(どの操作がどの運動を生んだか)を時間方向に細かく観察でき、車両モデルの同定や応答遅れを含む制御の学習が可能になります。

## 手法の系譜と主要論文

自車の操作信号を教師にする発想は、深層学習以前から自動運転研究の中心にあります。

- ALVINN: Pomerleau, "ALVINN: An Autonomous Land Vehicle in a Neural Network", NeurIPS 1989。3 層パーセプトロンにカメラ(30×32 画素)と当時のレーザー距離計の画像を入れ、操舵指令を直接出力した最初期のニューラルネット運転。入力 960 + 距離 8×32 次元から隠れ層 29 ユニットを経て操舵 45 出力という小さなネットで、人間の運転中の操舵を正解にして学んだ。CAN 信号を教師にする発想の源流です。
- 模倣学習の理論的限界 DAgger: Ross, Gordon, Bagnell, "A Reduction of Imitation Learning and Structured Prediction to No-Regret Online Learning", AISTATS 2011。学んだモデルが少しずれた状態に入ると、それは訓練データに無い状況なので誤りが雪だるま式に増える「分布シフト(distribution shift)」を定式化。素朴な模倣学習の誤差は時間長 T に対し最悪 T² で増えうるのに対し、専門家にその場で正解を聞き直してデータを足す DAgger では誤差を T の線形に抑えられることを示した。CAN のように正確な操作正解があっても、模倣だけでは分布シフトに弱い、という背景知識として重要です。
- PilotNet / End-to-End Learning for Self-Driving Cars: Bojarski et al.(NVIDIA), arXiv 2016。実車のカメラ映像から CAN 由来の操舵角を直接学習する 9 層 CNN(約 27 万パラメータ)。中央カメラに加え左右にずらしたカメラ画像へ補正操舵を割り当てる(視点シフト augmentation)ことで、分布シフトに対するロバスト性を一部補った。CAN 操舵を教師にした大規模 CNN の代表例です。
- UniAD: Hu et al., "Planning-oriented Autonomous Driving", CVPR 2023(Best Paper)。検出・追跡・地図・動き予測・占有予測・計画を 1 つの Transformer ベースネットワークで end-to-end に解き、最終出力は自車の将来軌道です。学習・評価で「自車が実際にどう運動したか」の正確な記録が鍵になり、CAN バス由来の高精度な自車運動データがその品質を支えます。

系譜としては「人間操作を正解にする(ALVINN, 1989)」→「模倣だけでは分布シフトに弱いと判明(DAgger, 2011)」→「大規模 CNN で実車実証(PilotNet, 2016)」→「知覚から計画まで end-to-end で軌道を出す(UniAD, 2023)」という流れで、どの段でも CAN 的な自車運動・操作信号が教師信号の中核を担い続けています。

## 論文の実験結果(定量データ)

CAN バス拡張は独立の論文を持つ補助データのため、単体の数値ベンチマークというより「使うと教師の質がどう変わるか」を周辺研究の数値で見るのが適切です。

- PilotNet(NVIDIA, 2016)は CAN 操舵を教師に学習し、実車テストで「自動走行率(autonomy)」を報告。彼らの定義は autonomy = 1 −(人間の介入回数 × 6 秒)/ 走行時間 で、ある区間で約 98 % の自動走行率を達成したと報告しています。98 % とは「ほぼ人間の介入なしに走れた」ことを意味し、CAN 操舵教師による end-to-end 学習が実走で機能することを示した初期の定量的証拠です。
- 数値微分ノイズの実害は、軌道予測研究で繰り返し指摘されてきました。位置の測定誤差を σ_p とすると、サンプリング間隔 Δt での速度推定誤差はおよそ σ_p/Δt、加速度誤差は σ_p/Δt² に比例して増えます。たとえば σ_p = 5 cm、Δt = 0.5 秒なら速度誤差は約 0.1 m/s、加速度誤差は約 0.2 m/s² 程度に増幅され、これがそのまま教師ノイズになります。Δt を 0.02 秒(50 Hz)に縮めれば微分の分母は小さくなりますが、その細かさは CAN を使わない限り得られません。CAN の実測加速度はこの増幅を回避します。
- UniAD は nuScenes 上で計画(自車軌道予測)を評価し、3 秒先の L2 誤差(予測軌道と実軌道のメートル距離)を平均約 1.03 m、衝突率(planned trajectory が他物体と衝突する割合)を約 0.31 % まで下げたと報告しています(モジュール分離型の従来手法より誤差・衝突率ともに改善)。ここでの「実軌道」の品質が自車運動データの精度に依存し、CAN 由来の高精度運動がこの種の評価の信頼性を支えます。なお UniAD のこの open-loop 評価値自体は後続研究で「ego 状態をモデルに入れるだけで近い数値が出る」と批判されており、評価設計の議論にもつながっています。

アブレーション的な観点では、「速度・加速度を位置微分で作った場合 vs CAN 実測を使った場合」で教師の S/N(信号対雑音比)が桁で変わり、低速・微小操舵の区間ほど差が顕著になります。微小な操舵やわずかな減速は微分ノイズに埋もれやすく、CAN 実測でないと学習信号として機能しないためです。停止寸前のクリープ走行や緩いカーブのような「小さいが安全に重要な操作」が、位置微分では雑音に消えてしまうのが典型的な失敗例です。

## メリット・トレードオフ・限界

メリット:
- 操舵角・速度・加速度・IMU を高頻度(10〜100 Hz)・低ノイズで取得でき、模倣学習の教師品質が上がる。
- 位置の数値微分で生じる誤差増幅を回避できる。
- 人間の操作そのもの(操舵・スロットル・ブレーキ)を学習・解析でき、車両ダイナミクス(操作から運動への遅れ)を扱える。
- 高頻度なので操作と運動の因果関係や過渡応答を時間方向に観察できる。

トレードオフ・限界:
- 母体データセットを前提とする補助データで、単独では使えない。
- 信号は収集車両(特定の車種)固有で、DBC 解釈・単位・符号・座標系が車種ごとに異なる。別車種にそのまま当てはめられない値がある。
- 生 CAN を物理量に直すには車両仕様の理解が要り、扱いに手間がかかる。
- 全シーンに完全に揃うとは限らず、欠損や時刻同期のずれに注意が必要。
- CAN は人間ドライバの操作を記録したものなので、その人間の癖・地域の運転文化を含む(教師の偏り)。

研究上の未解決課題として、CAN 教師による模倣学習はあくまで open-loop(出力を実際の運転に反映しない)前提で誤差を測るため、DAgger が指摘した分布シフトには弱いままです。正確な操作教師を持っていても、closed-loop(出力で実際に車を動かしてループを回す)評価では破綻しうる、という課題は CAN データだけでは解けません。さらに、操舵角という量はステアリングギア比やタイヤ特性に依存する車両固有量なので、これを直接学習すると別車両への転移が難しく、近年は車両非依存な曲率(curvature)やヨーレートを中間表現にする設計も検討されています。

## 発展トピック・研究の最前線

最近の end-to-end 運転研究は、CAN 由来の自車運動を教師にしつつ、それを open-loop の模倣だけで使う限界を超えようとしています。微分可能シミュレータを通して closed-loop で学習する手法、強化学習を組み合わせる手法、あるいは将来の運動を確率分布として予測して不確実性を扱う手法(混合分布や拡散モデルによるマルチモーダル軌道生成)などです。

また、CAN の高頻度信号は「世界モデル(world model, 自車の行動が環境をどう変えるかを予測するモデル)」の学習にも使われ始めており、行動条件付きの未来予測(action-conditioned prediction)の教師として価値が高まっています。行動を入力に次の観測を予測するには、正確で高頻度な行動ログが不可欠だからです。

車両ダイナミクスの観点では、CAN の操舵・速度を使って車両モデル(自転車モデルなど、車を簡略化した運動方程式)のパラメータを同定し、予測軌道が物理的に実現可能か(運動学的制約を満たすか)を制約する研究も進んでいます。これは「ニューラルネットが物理的にあり得ない軌道を出す」失敗を防ぐ実用上の工夫として重要です。
