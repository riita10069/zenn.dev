# 予測空間の圧縮(forecast space を切り替える調整ノブ)

## ひとことで言うと

未来の特徴を予測する JEPA タスクで「何を予測対象にするか・どれだけ細かく予測するか」を切り替えられる圧縮モジュールの考え方です。フル解像度の未来 BEV(鳥瞰図)特徴はメモリを食いすぎるので、チャンネル方向(occupancy / projected / full)と空間方向(stride)の2軸で圧縮し、メモリ予算を守りつつ意味のある特徴空間を残します。「メモリ削減」と「表現の豊かさ」のあいだのダイヤルだと考えると分かりやすいです。

## 直感的な理解

数秒後の世界を潜在空間で予測する([JEPA 系の低速ワールドモデル分岐(予測表現を補助タスクにする設計)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1041_jepa-world-model-branch))とき、予測の対象は「数秒後の特徴マップ」です。問題は、この特徴マップが巨大なことです。

例として、鳥瞰図(BEV、Bird's Eye View、複数カメラ画像を真上から見下ろした地図状の表現)が `450 × 300` グリッドで各セルが `256` チャンネルの特徴を持つとします。これを複数の未来時刻(例 +1.6s, +3.2s, +4.8s, +6.4s の4本)分まとめて予測すると、`450 × 300 × 4 × 256` の出力になります。fp32(32 ビット浮動小数=4 バイト)では1サンプルあたり約 550MB。バッチを組めば即座に GPU メモリが尽きます(高解像度 BEV はバッチ4でも 20GB を超える領域です)。

素朴な解決策は「チャンネルを平均して `450 × 300 × 1` の1チャンネルにする」ことです。これはメモリを 256 分の1に減らせます。しかし重大な副作用があります。チャンネル平均は特徴を「占有度(occupancy、そのセルに何かあるかどうか)のようなスカラー1個」に潰してしまい、JEPA が本来学びたい意味のある特徴空間を捨ててしまうのです。

これは JEPA の根幹に反します。I-JEPA([I-JEPA / V-JEPA(自己教師あり予測表現学習)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1074_i-jepa-v-jepa))が示した通り、「抽象的な特徴を予測する」ことこそが、エンコーダに予測不能な低レベルの細部を捨てさせ、意味のある構造を残させます。チャンネルを1次元(rank-1)に潰すのは、その目的にとって「最も破壊的な縮約」です。占有度スカラーを予測するだけなら、もはや豊かな特徴空間の予測ではなくなります。そこで「固定の平均」ではなく「調整可能なノブ(tunable knob)」として圧縮を設計し、同じメモリ予算に到達しつつ本物の表現を残せるようにするのが、予測空間圧縮の発想です。

## 基礎: 前提となる概念

- 特徴マップ(feature map): ネットワークの途中で出てくる3次元テンソル `[C, H, W]`(C はチャンネル数=特徴の種類、H・W は空間的な縦横の解像度)。バッチを入れると `[B, C, H, W]`。
- チャンネル(channel): 各空間位置に並ぶ特徴の次元。256 チャンネルなら、各セルが 256 個の数で表される。
- rank(階数): ここでは「情報の次元数」のイメージ。チャンネル平均で1個に潰すと rank-1(ほぼ1次元の情報)になり、学習した射影で 16 チャンネルに落とすなら最大 16 次元の情報を保てる。
- 1×1 畳み込み(1×1 convolution): 空間方向には動かず、各セルのチャンネルベクトルだけを線形変換する操作。空間構造を一切壊さずにチャンネル次元だけを増減できる。Network in Network(Lin et al., 2014)や GoogLeNet(Szegedy et al., 2015)で確立された定番テクニック。
- average pooling(平均プーリング): 隣接する空間セルをまとめて平均し、解像度を下げる操作。`stride=4` なら縦横が 1/4 になり、空間の要素数は 1/16 になる。
- メモリ予算(memory budget): 使ってよい GPU メモリの上限。予測対象テンソルの要素数が直結する。
- forecast space(予測空間): 「未来予測の損失を計算する空間」。ピクセル空間・全チャンネル特徴空間・圧縮特徴空間・占有度スカラー空間など、どこで予測の正解と突き合わせるかを指す。

## 仕組みを詳しく

圧縮器は入力 `[B, embed_dim, H, W]` を2軸で圧縮し `[B, out_channels, H/s, W/s]` を返します(s は空間 stride)。

### 軸1: チャンネル圧縮(mode)

- `"full"`: 全 `embed_dim` チャンネルを保持。何も縮約しない(out_channels = embed_dim)。最大の情報、最大のメモリ。
- `"projected"`(推奨デフォルト): 学習可能な `1×1` 畳み込みで `embed_dim → compressed_dim` に射影(out_channels = compressed_dim)。rank が最大 `compressed_dim` の射影なので、同じ出力チャンネル数でも rank-1 の平均よりはるかに多くの情報を残せる。I-JEPA の精神(意味空間の保持)に沿う。
- `"occupancy"`: チャンネル方向に平均して `1` チャンネルに(out_channels = 1)。最も安いベースライン。比較対象として明示的に残す価値がある。

```python
if mode == "occupancy":
    x = x.mean(dim=1, keepdim=True)        # [B, 1, H, W]  rank-1
elif mode == "projected":
    x = self.proj(x)                       # Conv2d(embed_dim, compressed_dim, 1)
# full は何もしない
```

### 軸2: 空間圧縮(spatial_stride)

BEV グリッドを average pooling で縮小します。メモリコストは `H × W` に比例するので、空間方向の縮小がメモリ削減の最大のレバー(てこ)です。stride を2にすれば要素数は 1/4、4にすれば 1/16 になります。

```python
if spatial_stride > 1:
    x = F.avg_pool2d(x, kernel_size=spatial_stride, stride=spatial_stride)
```

### キモの数値例(同じメモリで情報量が違う)

入力を `256 × 32 × 32` とし、`mode="projected", compressed_dim=16, spatial_stride=4` を考えます。

- occupancy(平均、stride なし)の出力: `1 × 32 × 32` = 1024 要素
- projected(16ch, stride4)の出力: `16 × 8 × 8` = 1024 要素

要素数(=メモリ)は完全に同じ 1024 です。しかし occupancy は1チャンネル(rank-1)なのに対し、projected は 16 個の「学習された意味チャンネル」を持ちます。つまり「同じメモリ削減を達成しつつ、潰れた1チャンネルではなく 16 次元の意味空間を予測対象にできる」。before/after で言えば、

```
before: 256 × 32 × 32 = 262,144 要素(巨大)
after(occupancy):  1 × 32 × 32 =  1,024 要素 → だが情報は rank-1 に潰れる
after(projected): 16 ×  8 ×  8 =  1,024 要素 → 同じメモリで 16 次元の意味を保持
```

`out_channels` を外部に公開しておくのは、予測器と target エンコーダ([I-JEPA / V-JEPA(自己教師あり予測表現学習)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1074_i-jepa-v-jepa))のチャンネル数を揃え、両者を同じ空間に置いて距離を測れるようにするためです。予測側と正解側で次元が食い違うと損失が計算できないので、圧縮器の出力チャンネル数を「契約」として明示することが配線上重要になります。

### なぜ rank-1 が最悪なのか(占有度との対比)

占有度スカラーは「そのセルに何かあるか」だけを表します。これは「車なのか歩行者なのか、どちらを向いているか、動いているか」という運転に効く情報を全部捨てます。一方 projected の 16 チャンネルは、データから「残すと予測が当たりやすく、かつ下流に効く 16 軸」を学習で選びます。同じメモリでありながら、前者は1軸、後者は 16 軸。予測課題としての「意味の豊かさ」が桁違いです。

## 手法の系譜と主要論文

- Lin et al., "Network in Network" (2014, arXiv:1312.4400): 1×1 畳み込みで空間構造を保ったままチャンネル次元を学習的に変換する手法を導入。projected モードの直接の土台。
- Szegedy et al., "Going Deeper with Convolutions (GoogLeNet)" (CVPR 2015, arXiv:1409.4842): 1×1 畳み込みを「次元削減ボトルネック」として大規模に使い、計算量を抑えつつ表現力を保つ設計を確立。チャンネル圧縮の系譜。
- Assran et al., "I-JEPA" (CVPR 2023, arXiv:2301.08243): 「抽象特徴を予測する」ことが低レベル細部の切り捨てと意味構造の保持を可能にするという主張。rank-1 の平均が JEPA の目的に最も反する縮約だ、という判断はこの論文に立脚。
- Bardes et al., "V-JEPA" (2024, arXiv:2404.08471): 動画で特徴予測を扱い、何を予測対象にするかが性能を左右することを示す。予測空間の選択が重要、という前提を支える。
- Li et al., "BEVFormer" (ECCV 2022, arXiv:2203.17270): 高解像度 BEV 表現の構築とその計算・メモリコストを論じた代表的研究。BEV のメモリ問題が広く知られた課題であること、空間 stride による削減がその標準的緩和策であることの背景。
- Tian et al., "Occ3D" (NeurIPS 2023, arXiv:2304.14365): 占有度(occupancy)予測を3次元のベンチマークとして整備。occupancy モードが単なる最安縮約ではなく、占有予測という独立タスクへの橋渡しになることの背景。
- Bolya et al., "Token Merging (ToMe)" (ICLR 2023, arXiv:2210.09461): 一様な間引きの代わりに、似たトークンを併合して情報損失を抑えながら計算を削る。average pooling のより賢い代替の系譜。

## 論文の実験結果(定量データ)

- 予測空間の選択が下流性能を左右することの根拠: I-JEPA のアブレーションでは、予測対象をピクセルに戻す(=最も低レベルな空間で予測する)と下流の linear probing 精度が落ちることが示されています。逆に抽象特徴空間で予測すると効率も精度も向上します。これは「予測空間を意味的に豊かに保つほど良い」という設計方針を、ImageNet 分類精度という具体的指標で支持しています。占有度スカラー(rank-1)への圧縮は、この観点で「ピクセルとは逆方向だが、別の意味で情報を捨てすぎる」縮約に相当します。
- 1×1 畳み込みによる次元削減の効果: GoogLeNet は 1×1 畳み込みのボトルネックにより、ImageNet 分類で当時の VGG 等より大幅に少ないパラメータ・計算量(約 500〜700 万パラメータ、VGG-16 の約 1.38 億の十数分の1規模)で同等以上の精度を達成しました。これは「学習可能なチャンネル圧縮は、固定の縮約より効率的に情報を残せる」ことの古典的な定量証拠で、projected モードが occupancy(固定平均)より優れる根拠になります。
- BEV 解像度とメモリ: BEVFormer 系の研究では、BEV グリッド解像度を上げると小物体や遠方の検出精度(nuScenes の mAP / NDS。NDS は nuScenes Detection Score の略で、検出精度と速度・属性誤差を統合した総合指標)が上がる一方、計算量とメモリがおおむね二乗で増えることが報告されています。空間 stride を上げると逆に細かい空間構造の精度が落ちる、というトレードオフは、この BEV 解像度 vs 精度の関係と整合します。
- 検証の観点: 一般的な単体検証では「occupancy と projected を同じメモリ(同じ要素数)に揃えたとき、projected の方がチャンネル数(=保持情報次元)が多い」ことを確認します。具体的には `occ.numel() == proj.numel()` かつ `proj.shape[1] > 1` を満たすことを検証し、「同じメモリでより豊かな表現」という主張をコードで担保します。

## メリット・トレードオフ・限界

メリット:
- メモリ予算を1つの目標値に固定したまま、予測対象の意味的豊かさを選べる(占有度の1chから全256chまで連続的に)。
- projected モードは学習可能なので、何を残すかをデータから決められる(固定平均より賢い)。
- 空間 stride で最大のメモリ削減レバーを直接握れる。
- 最安の占有度ベースラインを捨てず明示オプションとして残すので、アブレーション比較ができる。

トレードオフ・限界:
- 圧縮すれば必ず情報は失われる。projected でも compressed_dim を小さくすれば表現力は落ちる。
- 空間 stride を上げると細かい空間構造(小さな物体・縁石など)の予測精度が下がる。遠方や小物体の予測が必要な応用では stride を上げにくい。
- どの設定が運転性能に最適かは実データ学習で確かめる必要があり、一般論としては未解決(タスク依存)。
- projected は学習パラメータを増やし、target エンコーダ側とチャンネルを揃える配線の手間が生じる。圧縮器を学習で動かすと target との「同じ空間」性が時間変化する点にも注意が要る。

## 発展トピック・研究の最前線

- 学習可能なトークン削減: 単純な average pooling の代わりに、重要な空間トークンだけ残す token merging / token pruning(ToMe, Bolya et al. 2023)を使えば、一様な間引きより情報損失を抑えてメモリを削れます。混雑領域は細かく、空き領域は粗く、という非一様な圧縮が可能になります。
- 予測空間としての占有グリッド: 近年の自動運転では「占有度(occupancy)予測」自体が重要タスク(Occ3D など)で、occupancy モードは単なる最安ベースラインではなく、占有予測タスクとの橋渡しにもなります。意味的豊かさと占有度予測のどちらを優先するかは応用次第です。
- 適応的圧縮: シーンの複雑さ(混雑した交差点 vs 空いた高速)に応じて compressed_dim や stride を動的に変える適応圧縮は、固定設定より効率的になりうる研究方向です。
- 量子化された予測空間: 連続特徴の代わりに VQ(ベクトル量子化)で離散コードを予測対象にすると、メモリを抑えつつ生成系世界モデルとも接続できます。
