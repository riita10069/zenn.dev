# FP16推論とTensor Core活用

## ひとことで言うと

モデル内部で使う数値の精度を「少し粗くする」ことで、計算が速くなりメモリも半分で済む、という話です。NVIDIA GPU に載っている Tensor Core という専用回路が、この粗い数値(16ビット浮動小数点)の行列演算を特別に高速処理できるため、推論(学習済みモデルに入力を与えて答えを出すこと)のスループットがおおむね2倍、VRAM(GPU のメモリ)使用量がほぼ半分になります。

## 直感的な理解

自動運転の推論では2つが死活問題です。1つは「1フレームをどれだけ速く処理できるか」(レイテンシ、遅延)。前方に飛び出した歩行者への反応が 100 ミリ秒遅れれば、時速 50 km なら約 1.4 メートル進んでしまいます。もう1つは「載せている GPU のメモリに収まるか」(VRAM)。安価な車載・ノート GPU だと VRAM が 6〜8 GB しかないこともあります。

ここで効くのが数値精度の落とし方です。コンピュータは数を「ビット(0/1 の桁)」で表します。深層学習の既定は FP32(32ビット浮動小数点)ですが、これを FP16(16ビット)に落とすと、数値1個あたりの容量が半分(4バイト → 2バイト)になり、メモリも、メモリ内のデータ運搬量も半減します。さらに NVIDIA GPU の Tensor Core が FP16 の行列積を専用ハードで一気に処理するため、速度も跳ね上がります。料理にたとえれば、レシピ(計算手順)はそのままに、計量を「グラム単位」から「ざっくり」に変えるだけで、調理が倍速になり鍋も小さくて済む、ようなものです。粗くしすぎると味(精度)が落ちるので、そこは賢く使い分けます。

## 基礎: 前提となる概念

- 浮動小数点数(floating point): 小数点の位置を動かして、とても大きい数も小さい数も表せる数値表現。符号部(プラスマイナス)・指数部(数の大きさの桁)・仮数部(数の細かさ)の3つで1つの数を表す。
  - FP32(単精度): 32ビット。符号1 + 指数8 + 仮数23。表せる範囲も精度も広い。深層学習の既定。
  - FP16(半精度、IEEE 754 half): 16ビット。符号1 + 指数5 + 仮数10。表せる範囲はおよそ ±65504 と狭く、精度も粗い。
  - BF16(bfloat16): 16ビット。符号1 + 指数8 + 仮数7。指数部を FP32 と同じ8ビットに保つので表せる範囲は FP32 並みに広い反面、仮数が7ビットと少なく細かさは FP16 にも劣る。
- Tensor Core: Volta 世代(2017)以降の NVIDIA GPU に載る、小さな行列の積和演算(掛けて足す)を1命令でまとめて処理する専用ハードウェア。深層学習の計算の大半は行列積なので、ここが速いと全体が速くなる。FP16 入力・FP32 累算(足し合わせは精度を保つ)が基本動作。世代が進むほど対応形式が増え、Hopper/Ada 世代では FP8 にも対応する。
- 推論(inference): 学習済みモデルに新しい入力を与えて出力を得ること。勾配計算が不要なぶん学習より軽いが、リアルタイム性が問われる。
- スループットとレイテンシ: スループットは単位時間あたりの処理量(FPS、1秒あたり処理フレーム数)。レイテンシは1件の処理にかかる時間(ミリ秒)。両者は関係するが別物。バッチをまとめるとスループットは上がるがレイテンシは悪化する、というトレードオフがある。
- メモリ帯域律速(memory-bound): 計算そのものより「メモリからデータを運ぶ速度」が遅くて全体が頭打ちになる状態。FP16 はデータ量を半減させるので、計算が速くなるだけでなくこの帯域も改善する。

## 仕組みを詳しく

PyTorch のような深層学習ライブラリで FP16 推論を行うには、典型的に2つの仕掛けを組み合わせます。

### 1. 重みを半精度に変換する

モデルが内部に持つ重み(学習で決まったパラメータ。掛け算に使う係数の集まり)をすべて FP32 から FP16 に変換します(PyTorch なら `model.half()`)。たとえば 3530 万(35.3M)パラメータを持つバックボーン(画像から特徴を取り出す土台ネットワーク)なら、1個4バイト → 2バイトになり、重みのメモリが約 141 MB → 約 71 MB に減ります。重みだけでなく、推論中に作られる中間特徴(activation、活性値)も半分になるため、実際の節約はこれより大きくなります。

### 2. 自動混合精度(AMP)で計算する

```python
with torch.autocast(device_type='cuda', dtype=torch.float16):
    output = model(input)
```

AMP(Automatic Mixed Precision、自動混合精度)は、計算の種類ごとに「ここは FP16 で速く、ここは FP32 で正確に」と自動で使い分ける仕組みです。行列積や畳み込みは FP16(速く、Tensor Core が効く)、合計を取る reduction(多数の値を足す処理。誤差がたまりやすい)や softmax(指数を取るため値が暴れやすい)、正規化層は FP32、というふうに賢く振り分け、速さと数値安定性を両立させます。入力テンソル(例 `[1, 8, 3, 256, 256]`)も半精度にそろえてモデルへ渡します。

`model.half()`(全部 FP16)と `autocast`(自動で混合)の違いは重要です。前者は最大限軽いが、softmax や正規化まで FP16 になり数値が崩れるリスクがあります。後者は崩れやすい演算だけ FP32 に保つので安定しますが、FP32 部分のメモリ・計算は残ります。実務では autocast を基本にし、安定性を確認したうえで必要なら全 half 化に進む、という順序が安全です。

### なぜ Tensor Core が効くのか

通常の演算ユニット(CUDA コア)は1要素ずつ掛け算しますが、Tensor Core は 4×4 程度の小行列の積和を1サイクルでまとめて処理します。深層学習の forward は実質「巨大な行列の掛け算の連鎖」なので、ここが専用ハードで加速されると全体が大きく速くなります。FP32 のままだと Tensor Core がフル活用されにくく、宝の持ち腐れになります。さらに FP16 はデータ量が半分なので、メモリからの読み出し量も半減し、メモリ帯域律速のワークロードでも効きます。

### 典型的な実測の感覚(エントリ級ノート GPU の例)

同一 GPU・同一モデル(軽量バックボーン + 単純融合、バッチ1、256×256、長めのストレステスト)で FP32 と FP16 を比較した報告例を示します。

| 指標 | FP32 | FP16 | 変化 |
|---|---|---|---|
| FPS(1秒あたり処理フレーム数) | 約 14.8 | 約 29.6 | およそ +100% |
| 平均レイテンシ | 約 67.7 ms | 約 33.8 ms | 約半分 |
| 最悪レイテンシ | 約 115 ms | 約 60 ms | 約半分 |
| ジッタ(ばらつき σ) | 約 2.0 ms | 約 1.8 ms | やや改善 |
| VRAM 割当(allocated) | 約 1074 MB | 約 546 MB | およそ −49% |

FPS が約2倍、VRAM 割当がほぼ半分。これは Tensor Core が実際に使われている証拠です。注意点として、VRAM の「reserved(予約)」値は FP16 のほうがやや大きくなることがあります(ライブラリがメモリを多めに確保する挙動による)。本質的な効果は実際に使われる「allocated(割当)」の半減です。さらに、下位 GPU を FP16 で回したほうが、1ランク上の GPU を FP32 で回すより速い、という逆転がエントリ帯ではしばしば観測されます。コスト効率の観点で重要な知見です。

ただし「約2倍」は理想に近い場合の値で、前処理(画像のデコード・リサイズ)や CPU↔GPU 間のデータ転送が支配的なワークロードでは効果が薄れます(後述のアムダールの法則)。最悪レイテンシ・ジッタ(処理時間のばらつき)も自動運転では重要で、平均が速くても稀に遅いフレームがあると安全マージンを食います。

## 手法の系譜と主要論文

- Micikevicius ら(NVIDIA・Baidu, 2018, ICLR, "Mixed Precision Training", arXiv:1710.03740)。混合精度学習を体系化した代表的論文。FP16 で計算しつつ、重みのマスターコピーを FP32 で保持し、損失をスケーリング(loss scaling、小さな勾配が FP16 で0に丸められるのを防ぐため損失を一時的に大きくする工夫)することで、精度をほぼ落とさずメモリと速度を改善できることを示しました。動機は「大規模モデルの学習をメモリと時間の両面で現実的にする」こと。推論でも同じ原理(行列積を FP16、累算を FP32)が効きます。

- NVIDIA Volta アーキテクチャ / Tensor Core のホワイトペーパー(2017 以降)。FP16 入力・FP32 累算の積和を1命令でまとめて行うことで、FP32 比で大幅なスループット向上(理論ピークで数倍)が得られると報告。動機は「行列演算をハードで一気に処理する」こと。トレードオフは FP16 の表現範囲の狭さに起因する精度劣化リスク。

- bfloat16(Google Brain 由来、TPU で普及)。指数部を FP32 と同じ8ビットに保ち仮数を削った16ビット形式。表せる範囲が広くオーバーフロー(大きすぎて表せない)/アンダーフロー(小さすぎて0になる)に強い反面、細かい精度は FP16 に劣ります。学習では数値安定性の高さから BF16 が好まれ(loss scaling が基本不要)、推論では精度範囲が許せば FP16 が選ばれる、という使い分けが一般化しました。どちらも「16ビットで速く省メモリに」という狙いは同じです。

- さらに下の精度。INT8(8ビット整数)量子化(Jacob ら, CVPR 2018, arXiv:1712.05877)や FP8(8ビット浮動小数点、Hopper/Ada 世代以降)が推論高速化の次の段として研究・実用化されています。INT8 量子化には校正(calibration、代表入力で値の範囲を測り丸め方を決める)が要り、精度劣化と速度のトレードオフがより厳しくなります。Jacob らはモバイル向けに整数演算だけで推論する枠組みを示し、量子化を意識した学習(QAT)の基礎を作りました。

## 論文の実験結果(定量データ)

- Mixed Precision Training(ICLR 2018)のアブレーション: ImageNet 分類(ResNet 系)・機械翻訳・音声認識など複数タスクで、FP16 + loss scaling を使っても FP32 とほぼ同等の精度(誤差 0.1 ポイント前後)を保ちつつ、メモリをおよそ半減、学習を大幅に高速化できることを示しました。重要な発見は、loss scaling を入れないと小さな勾配がアンダーフロー(小さすぎて0に丸められる)して精度が落ちる場合があること。つまり「ただ FP16 にする」だけでなく「丸めをどう守るか」が成否を分けます。たとえば SSD(物体検出)では loss scaling 無しだと収束しないが、有りなら FP32 と同等、という対比が示されました。

- Tensor Core の効果: 行列積中心のワークロードで FP32 比のスループット向上が報告され、実アプリ(推論)でもおよそ2倍前後の高速化が広く観測されます。前掲の表の「FPS が約2倍・VRAM が約半分」は、この理論効果がエントリ GPU の実推論でそのまま現れた典型例です。

- INT8 量子化(Jacob ら, 2018)のトレードオフ: MobileNet などで、INT8 化により推論を大きく高速化しつつ、QAT(quantization-aware training、量子化を織り込んだ学習)を併用すれば精度低下を1ポイント前後に抑えられると報告されました。一方、何も対策せず学習後にそのまま量子化(post-training quantization)すると、モデルによっては精度が数ポイント以上落ちることもあります。FP16 より下の精度ほど「ただ落とす」では済まない、という教訓です。

指標の意味を補足します。FPS が2倍とは「同じ時間に倍の枚数を処理できる」=リアルタイム処理(たとえば 30 FPS 要件)に近づけること。VRAM 半減とは「同じ GPU でより大きなバッチや高解像度を載せられる」「より安い小容量 GPU でも動く」こと。レイテンシ半減は安全に直結します(反応が速い)。一方、精度は FP32 と完全一致はせず、タスクによっては無視できない劣化が出るため、FP16 と FP32 の出力差(軌道予測なら ADE の差)を必ず検証すべきです。

## メリット・トレードオフ・限界

メリット

- スループットがおよそ2倍。リアルタイム推論に近づく。
- VRAM がほぼ半分。安価な小容量 GPU で動かせ、より大きなバッチ・解像度を載せられる。
- メモリ帯域も半減するので、帯域律速のワークロードでも効く。
- 下位 GPU + FP16 が上位 GPU + FP32 を上回ることがあり、コスト効率が良い。
- 実装が容易。重みの半精度化と autocast を加えるだけ。

トレードオフ・限界

- 数値精度が下がる。FP16 は表せる範囲がおよそ ±65504 と狭く、大きすぎる値でオーバーフロー、小さすぎる値でアンダーフロー(0に丸め)が起きうる。
- 出力がわずかにずれる。FP32 と数値が完全一致せず、タスクによっては劣化が無視できない。特に softmax・正規化・累積和を含む経路は崩れやすく、autocast でそこだけ FP32 に保つ配慮が要る。
- すべての演算が速くなるわけではない。Tensor Core が効くのは主に行列積・畳み込み。I/O やデータ整形・前処理は速くならず、そこがボトルネックなら効果が薄れる(アムダールの法則、全体の高速化は速くならない部分に頭打ちされる)。
- 安定性が問題なら BF16 など別形式の検討が要る。範囲の広い BF16 はオーバーフローに強いが細かさで劣る。

研究上の課題として、より低精度(INT8/FP8)へ進むほど校正や量子化対応の学習(QAT)が必要になり、精度・速度・実装コストのトレードオフが先鋭化します。安全クリティカルな自動運転では「速くなったが稀なケース(夜間・逆光・遠方の小さな物体)で精度が崩れる」リスクの検証が不可欠で、平均精度だけでなく最悪ケースの分布まで見る必要があります。

## 発展トピック・研究の最前線

- INT8 / FP8 量子化と QAT。推論をさらに軽くする次の段。TensorRT などの推論最適化エンジン(層の融合・カーネル自動選択・量子化を施して実行を最適化するツール)と組み合わせ、車載組み込み GPU での実時間推論を狙う。
- 混合精度の自動最適化。どの層をどの精度にするかを自動探索し、精度を保ちつつ最速の構成を見つける研究。感度の高い層だけ高精度に残す mixed-precision quantization が代表。
- メモリ帯域律速の見極め。FP16 で計算は速くなってもメモリ帯域や前処理が律速なら効果が出ない。プロファイリングでボトルネックを特定する手法([推論レイテンシ計測の方法論](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0522_inference-latency-benchmark-methodology))が実務で重要。Roofline 解析(計算量と帯域のどちらが律速かを図示する手法)などが使われる。
