# channels-first/last 自動判定

## ひとことで言うと
画像から取り出した特徴マップ（feature map、画像のどこに何があるかを数値の格子で表したもの）は、4つの軸「バッチ・チャネル・高さ・幅」を持つ4次元テンソルです。このうち「チャネルの軸を何番目に置くか」には2つの流儀があり、モデルによって異なります。ある特徴抽出器は `(B, H, W, C)`（最後がチャネル）を返し、別のものは `(B, C, H, W)`（前から2番目がチャネル）を返します。特徴抽出器を取り替えると流儀が変わって下流の処理が静かに壊れるため、抽出器の名前ではなく「実際に出てきたテンソルの形」を見て、必要なときだけ軸を並べ替える互換層を入れる、という設計パターンです。

## 直感的な理解
本棚を想像してください。「縦に並べる人」と「横に寝かせて積む人」がいて、どちらも同じ本（情報）を持っています。あなたが「3段目の本」を取りたいとき、相手がどちらの流儀かを知らないと、見当違いの本を掴んでしまいます。特徴マップのチャネルも同じで、「特徴の本数を表す軸」がどこにあるかを取り違えると、チャネルだと思って掴んだものが実は画像の幅だった、ということが起きます。

具体的な事故を挙げます。下流の処理がチャネル軸（前から2番目）に対して 1x1 の畳み込みやチャネル方向のプーリングをかけるとき、もしそこが本当は「幅」の軸だったら、計算自体はエラーを出さずに通ってしまうことがあります。形は4次元のままなので Python は文句を言いません。けれど中身は意味をなさず、学習が一向に進まない、損失が下がらない、という形で後から発覚します。こうした「黙って間違う」バグ（silent bug、例外を投げずに誤った結果を返す不具合）は最も厄介で、デバッグに何日も溶かすことが珍しくありません。だからこそ自動的に正しいレイアウトへ揃える仕組みが要ります。

## 基礎: 前提となる概念
まず用語を噛み砕きます。

- テンソル（tensor）: 数値を多次元の格子に並べたもの。1次元ならベクトル、2次元なら行列、それ以上もまとめてテンソルと呼びます。画像の特徴マップは4次元です。
- バッチ（batch, B）: 一度にまとめて処理するサンプルの数。GPU は1枚ずつより束で流したほうが速いので、B 枚をまとめます。
- チャネル（channel, C）: 「特徴の本数」。入力 RGB 画像なら C=3（赤・緑・青）。深い層では C=256, 512 のように増え、各チャネルが「エッジ」「特定の色」「車らしさ」など別々の特徴を表します。
- 高さ・幅（H, W）: 特徴マップの縦横の格子サイズ。深い層ほど小さくなります（例: 256x256 の画像が 8x8 の特徴マップになる）。

レイアウトの2流儀:

- channels-first（チャネル先）: `(B, C, H, W)`。PyTorch の畳み込み `nn.Conv2d` が標準で前提とする並び。多くの CNN（畳み込みニューラルネット、画像の局所領域に小さなフィルタを滑らせて特徴を取る方式）系がこの形で特徴を返します。
- channels-last（チャネル後）: `(B, H, W, C)`。空間位置 `(H, W)` ごとに「その地点の特徴ベクトル C 本」が連続して並ぶ形。Transformer 系（画像をパッチやトークンの並びとして扱うモデル）はこの並びが自然です。

なぜ2流儀あるのか。CNN は「チャネルごとの2次元平面に対して畳み込む」発想なので、平面 `(H, W)` をひとかたまりにする channels-first が素直です。一方 Transformer は「位置ごとの特徴ベクトル」を attention（自己注意、トークン同士の関連度で重み付けする仕組み。Vaswani et al., NeurIPS 2017 で言語向けに定式化された）でやり取りするので、位置を先に並べてチャネルを末尾に置く channels-last が素直です。Transformer の各トークンは「1つの C 次元ベクトル」であり、画像をトークン列として見るとき各空間位置がそのまま1トークンになる――この自然な対応が channels-last を生みます。

さらにハードウェア最適化の事情もあり、PyTorch は `torch.channels_last` という専用のメモリ形式を公式に提供しています。これは NVIDIA の Tensor Core（行列演算専用の演算ユニット）が channels-last 配置の畳み込みで高いスループットを出せるためで、混合精度（bf16/fp16、計算を低精度で行って速度とメモリを稼ぐ手法）と組み合わせると顕著に速くなることが報告されています。つまりレイアウトの差は「思想の違い」と「性能の事情」の両方から生まれた、現実的で避けられない問題です。

ここで強調すべき区別があります。「論理 shape の軸順」（テンソルが `.shape` で報告する次元の並び）と「物理メモリ配置」（メモリ上で要素が実際にどう並ぶか）は別物です。`torch.channels_last` は後者だけを変え、論理 shape は `(B, C, H, W)` のままに保ちます。本トピックが扱うのは前者、すなわち抽出器が返す論理 shape が `(B,C,H,W)` か `(B,H,W,C)` かの違いです。両者を混同すると判定ロジックを誤るので注意します。

## 仕組みを詳しく
最も素朴な対策は「抽出器の名前で分岐する」ことです。

```python
if backbone_name.startswith("swin"):
    feat = feat.permute(0, 3, 1, 2)   # channels-last を first に直す
```

しかしこれは脆い設計です。理由は3つあります。第一に、名前の文字列を1つでも間違えると無言で壊れます。第二に、新しい Transformer 系抽出器を追加するたびに分岐を足す保守作業が発生します。第三に、同じモデル内でもステージ（層の段階）によってレイアウトが混在する実装には対応できません。

そこで「名前ではなく実テンソルの形」で判定します。前提として、モデル構築時に各ステージの正しいチャネル数 `expected_c` を別途取得しておきます（この取得方法は [feature_infoによるチャネル自動発見](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0960_feature-info-channel-discovery) が担います）。forward（順伝播、入力から出力を計算する処理）では各特徴マップについて「どの軸が、構築時に分かっているチャネル数と一致するか」を調べます。

```python
def forward(self, image):
    features = self.backbone(image)        # ステージごとの特徴マップのリスト
    permuted = []
    for f, expected_c in zip(features, self.feature_channels):
        if f.dim() == 4 and f.shape[1] != expected_c and f.shape[-1] == expected_c:
            permuted.append(f.permute(0, 3, 1, 2))   # (B,H,W,C) -> (B,C,H,W)
        else:
            permuted.append(f)
    return permuted
```

判定の日本語訳: 「4次元で、かつ軸1（前から2番目）が期待チャネル数と違うのに、最後の軸が期待チャネル数と一致するなら、それは channels-last なので並べ替える」。

数値例。あるステージの期待チャネル数 `expected_c = 384` とします。

```
Transformer 系の出力 : (8, 8, 8, 384)
  shape[1]=8   != 384  かつ  shape[-1]=384 == 384  -> 条件成立 -> permute
  permute(0,3,1,2)     : (8, 384, 8, 8)   ← channels-first に直る

CNN 系の出力         : (8, 384, 8, 8)
  shape[1]=384 == 384  なので最初の条件が偽 -> そのまま
```

`permute(0, 3, 1, 2)` は「軸の順番を入れ替える」操作です。引数 `(0,3,1,2)` は「新しい0番目に元の0番目(B)、新しい1番目に元の3番目(C)、新しい2番目に元の1番目(H)、新しい3番目に元の2番目(W)」を置く指定で、結果 `(B,H,W,C) -> (B,C,H,W)` になります。permute はデータ本体をコピーせず「見え方（ストライド、各軸を1つ進むのに何要素分メモリを飛ぶか）」だけを書き換える軽い操作です。ただし permute 直後のテンソルはメモリ上で非連続になるため、後段で `view`（連続前提で形を変える操作）などを使う場合は `.contiguous()` で詰め直す必要が出る点には注意します。`.contiguous()` を呼ぶとそこで初めて実コピーが走り、メモリ帯域を消費します。

この方式の良いところは、ステージごとに独立して判定する点です。仮に抽出器がステージによってレイアウトを変える実装でも、各特徴マップを個別に見て必要なものだけ直せます。また `expected_c` を基準にしているので、たまたま H と C が同じ数（例えば両方 8）になる紛らわしいケースでも、`shape[1]`（C のはずの位置）と `shape[-1]`（末尾）の両方を確認することで誤判定を減らせます。

ASCII 図でまとめると:

```
抽出器の出力リスト ──┐
                      ▼
       [ レイアウト自動判定 ]  ← expected_c を基準に shape を照合
                      │ channels-last なら permute、first ならそのまま
                      ▼
       常に (B, C, H, W) に揃った特徴マップ
                      ▼
       下流(1x1 Conv / プーリング / 融合)は channels-first だけ想定すればよい
```

設計上の要点は「複雑さを1か所に閉じ込める」ことです。レイアウトの揺れを互換層で吸収してしまえば、下流のすべての層は channels-first だけを前提に書けます。下流が抽出器の種類を知らずに済むこの「関心の分離」は、複数抽出器を差し替え可能にするフレームワークの保守性を決定的に支えます。

## 手法の系譜と主要論文
これは単一論文の「手法」ではなく、複数モデルの設計思想の違いから生まれた実装上の課題です。系譜を追うと理解が深まります。

- ResNet（He et al., CVPR 2016, arXiv:1512.03385）。残差接続（入力をショートカットで足し、層が「差分」だけを学べばよいようにする工夫）で深いネットワークの学習を可能にした古典的 CNN。特徴は伝統的に channels-first `(B, C, H, W)` で扱われ、以後の検出・分割モデルの標準的な特徴抽出器になりました。
- Attention / Transformer（Vaswani et al., NeurIPS 2017, arXiv:1706.03762）。トークン列に対する自己注意を提案。各トークンが C 次元ベクトルであるという「位置ごとの特徴ベクトル」観が、後の画像 Transformer における channels-last 表現の源流です。
- Swin Transformer（Liu et al., ICCV 2021 Best Paper, arXiv:2103.14030）。画像を小さな window（窓）に区切り、その中だけで attention を計算し、window を少しずらしながら段階的に大域を見る（shifted window）手法。画像をトークン列として扱うため、内部表現は `(B, H, W, C)` の channels-last が自然です。
- ConvNeXt（Liu et al., CVPR 2022, arXiv:2201.03545）。「Transformer の設計上の工夫（大きなカーネル、LayerNorm、GELU、活性化を減らす）を純粋な CNN に逆輸入する」研究。内部に channels-last を使う層を持ちますが、特徴抽出インターフェースとしては channels-first に整えて出力されるのが通例です。
- PyTorch の `torch.channels_last` メモリフォーマット（公式ドキュメント）。論理的な軸順を変えずにメモリ上の物理配置だけを channels-last にする最適化機能で、上記の「論理的な軸順の違い」とは別概念です。

これらが共存する現代の画像認識では、「複数抽出器を差し替え可能にするフレームワーク」が一般的になりました。その文脈で、レイアウト差を吸収する互換層は事実上の必須部品になっています。

## 論文の実験結果(定量データ)
レイアウト自動判定そのものに固有のベンチマークはありませんが、関連する定量データを押さえておくと設計判断ができます。

- PyTorch の channels_last メモリ最適化: NVIDIA と PyTorch の報告では、ResNet-50 を Tensor Core 対応 GPU（V100/A100 等）で混合精度学習する際、channels_last メモリ形式を使うと channels-first（contiguous）比でおよそ 1.2〜1.8 倍程度のスループット向上が得られるとされます。ここでスループットとは「1秒あたり処理できる画像枚数」で、大きいほど学習が速いことを意味します。差が生じる理由は、Tensor Core が要求するメモリアクセスパターンと channels-last が合致し、無駄な並べ替えコピーが減るためです。これは「物理メモリ配置」の最適化であり、本トピックの「論理 shape の軸順」とは別ですが、なぜ Transformer 由来の channels-last 思想が性能面でも支持されるかの背景になります。
- Swin Transformer の認識性能（ICCV 2021）: ImageNet-1K 画像分類で Swin-B が top-1 精度 約83.5%、当時の同規模 CNN を上回りました。top-1 精度とは「最も確からしいと予測したクラスが正解と一致した割合」で、高いほど良い指標です。COCO 物体検出では当時の最先端を更新し、box AP（検出枠の正確さの平均、高いほど良い）でおよそ 58.7（Swin-L, HTC++ 系設定の報告値）に達するなど大きな改善を示しました。この強さゆえに Swin 系が抽出器の選択肢に入り、結果として channels-last 対応が必要になりました。
- ConvNeXt の認識性能（CVPR 2022）: 純 CNN でありながら ImageNet-1K で Swin と同等以上（ConvNeXt-B で top-1 約83.8%、ConvNeXt-XL で約87%超(22K事前学習)）を達成し、「Transformer 一辺倒ではなく CNN も依然有力」という流れを作りました。CNN 系抽出器を選びたい実務的な動機を与え、channels-first と channels-last の両方を1フレームワークで扱う必要性を一層高めました。

アブレーション的な観点では、「名前ベースの分岐」と「形ベースの自動判定」を比べると、新規抽出器を追加したときの破損率に差が出ます。名前ベースは追加のたびに分岐の更新漏れでバグが入りますが、形ベースは `expected_c` が正しければ追加コード不要で正しく動きます。ただし後述のとおり H==W==C の縮退ケースだけは形ベースでも判別不能で、ここだけは原理的な限界が残ります。

## メリット・トレードオフ・限界
- メリット: 抽出器名にハードコードした分岐が不要で、新しい抽出器を足しても forward を変えなくてよい。保守コストが下流に波及しない。
- メリット: 特徴マップごとに個別判定するので、ステージでレイアウトが混在する抽出器にも対応できる。
- メリット: `shape[1]` と `shape[-1]` の両方を確認するため、H と C がたまたま同値でも誤判定しにくい。
- トレードオフ: 構築時に正しいチャネル数 `expected_c` を知っている前提に依存する（[feature_infoによるチャネル自動発見](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0960_feature-info-channel-discovery) とセットで初めて成立する）。
- 限界: H==W==C のように全軸が同じ数になる極端なケースでは、形だけからレイアウトを原理的に判別できない。実用上はまれだが、小さな解像度・特定のチャネル数の組み合わせでは起こりうる。この場合は名前や明示フラグなど別の手掛かりに退避するしかない。
- 限界: permute はメモリ非連続を生むため、性能を突き詰める場面では明示的に `channels_last` メモリフォーマットを採用し、論理 permute を避ける設計のほうが速い場合がある。

## 発展トピック・研究の最前線
レイアウト互換の発想は、より一般的な「異種抽出器の統一インターフェース化」へ広がります。近年は基盤モデル（大量データで事前学習した汎用特徴抽出器）を凍結して下流に使う構成が増え、CNN・Transformer・state-space モデル（Mamba 系、系列を線形時間で処理する新方式）など出力形式の異なる抽出器を同じパイプラインに差し込む需要が高まっています。ここで重要なのは、レイアウトだけでなく「ステージ数」「各ステージのストライド（縮小率）」「正規化の有無」まで含めて契約化することです。さらに、性能を追う研究では論理 permute を排し、入力から出力まで一貫して channels_last メモリで流す「format-aware」な設計や、`torch.compile`（計算グラフを最適化コンパイルする機能）による自動最適化が進んでいます。レイアウト判定は地味ですが、複数抽出器を公平にベンチマークする実験基盤の信頼性を支える縁の下の力持ちです。
