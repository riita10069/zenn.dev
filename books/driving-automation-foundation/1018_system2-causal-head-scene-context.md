# System2 因果ヘッドとSceneContext

## ひとことで言うと

自動運転モデルに「今この場面で運転に効いている主な要因は何か(交差点なのか、歩行者なのか、信号なのか)」を考えさせる、軽量な推論ヘッド(System2 因果ヘッド)の話です。そしてその「考え」を、自由な文章のままプランナ(運転計画器)に流し込むのではなく、「信頼度(confidence)」と「出どころ(provenance)」を付けた型付きの構造体(SceneContext)としてきちんと包んで渡す、という設計の話です。要点は、推論結果を「強制的な指令」ではなく「信頼度付きの助言」としてアーキテクチャに統合することにあります。

## 直感的な理解

人間の思考には2つのモードがあるとされます。System1(速い・反射的・直感)と System2(遅い・熟考・論理)です(Kahneman の二重過程理論)。ボールが飛んできたら反射的によける(System1)が、住宅ローンを組むかどうかは時間をかけて論理的に考える(System2)。自動運転でも、ハンドルやアクセルの即時制御は System1 的な反射層が担い、「なぜそう運転するのか」という因果の判断は System2 的な熟考層が担う、という役割分担が考えられます。

純粋な模倣学習(ベテラン運転の真似)だけのモデルには弱点があります。「なぜブレーキを踏んだのか」という理由を内部に持たないため、見たことのない稀なケース(long-tail, ロングテール=出現頻度が低いが安全上重要な状況)で脆くなります。また、判断が場面ごとにブレて出力がちらつく(decision flicker, 同じような場面で前進と停止が交互に出るなど)ことがあります。そこで「この場面の支配的な因果要因」を明示的に推論させ、計画を安定させたい、という動機が生まれます。

ここで設計上の重要な問題があります。最近は VLM(Vision-Language Model, 画像を見て言葉で説明できるモデル)を使って「歩行者が飛び出しそうだから減速」のような自由文(free-form text)の理由を出させる流れがあります。しかし自由文をそのままプランナに突っ込むのは危険です。理由は3つ。

1. 自由文は型がない(どんな文字列でも来る)ので、プランナが安定して解釈できない。表現のゆれや言い回しに弱い。「歩行者注意」と「pedestrian ahead」と「人がいる」が同じ意味でも、別の文字列として扱われる。
2. その理由がどれくらい確からしいか(信頼度)が分からない。確信度95%の判断も、当てずっぽうも、同じ文字列として届く。
3. その理由がどこから来たか(VLM か、HD 地図か、ルート指示か)という出どころが分からず、信頼の重み付けができない。

この問題への一つの答えが「自由文を直接入れず、信頼度と provenance を付けた型付き SceneContext の一プロデューサ(producer)として因果ヘッドを扱う」設計です。文章という人間向けの表現を、機械が安全に扱える型付きデータに変換してから流す、という発想です。

## 基礎: 前提となる概念

「因果推論ヘッド(causal reasoning head)」とは、知覚特徴から「この場面で運転判断を支配している要因」を推定する小さなネットワークです。出力は連続的な潜在ベクトル(reasoning latent)であったり、離散的なクラス(交差点/歩行者/信号など)であったりします。ここでいう「因果」は厳密な因果推論(do 演算子による介入効果の推定)ではなく、「この行動を引き起こしている主因」という運転判断の根拠付けを指す、ゆるい意味です。

「型付き構造体(typed structure / dataclass)」とは、フィールド名と型が事前に決まったデータの入れ物です。自由文字列と違い、受け取る側は「どんな形のデータが来るか」を保証されているので、安全に処理できます。ソフトウェア工学でいうインターフェースの明確化です。コンパイラや型チェッカが「このフィールドは必ず存在し、float である」と保証してくれるため、下流のバグが減ります。

「provenance(出どころ・来歴)」は、ある情報がどの情報源から生成されたかを記録するメタデータです。データの信頼性管理で広く使われる概念で、ここでは「この因果判断は VLM ヘッド由来」「これは HD 地図由来」と区別し、下流が情報源に応じた重み付けをできるようにします。地図は確実だが VLM は幻覚しうる、といった信頼度の差を扱うために必要です。

「カスケード設計(cascade)」は、知覚 → 推論 → 計画のように段階を直列につなぎ、前段の出力を後段の入力にする構成です。各段の損失からの勾配が共有表現まで逆伝播すれば、補助タスク(因果推論)が主タスク(計画)の表現を底上げします。これはマルチタスク学習(複数の目標を同時に学ぶことで、共通表現が豊かになる)の一形態です。

## 仕組みを詳しく

典型的な実装は、型付き SceneContext と、それを生成する因果ヘッドの2つから成ります。

### 型付きの SceneContext(構造化されたバス)

SceneContext は、プランナに渡す高レベルの場面理解を運ぶ「標準化されたバス(共通の通り道)」です。複数の情報源の出力を、同じ型に揃えて1本のバスに乗せます。

```
SceneContext:
    causal_reasoning: Optional[CausalReasoningContext] = None
    # 将来は map_context, route_context など他プロデューサも同型で追加可能
```

中身の CausalReasoningContext も型付きで、典型的には次のフィールドを持ちます。

```
CausalReasoningContext:
    reasoning_latent     : [B, latent_dim]   推論の連続ベクトル
    causal_class_logits  : [B, num_classes]  因果クラスの生スコア
    confidence           : [B]               信頼度(確率の最大値など)
    provenance           : str               出どころ(例: "vlm_causal_head")
```

要点は、(1) 複数の情報源(VLM 出力・HD 地図・ルートコマンドなど)を同じ型のバスに乗せられること、(2) 各情報に必ず confidence と provenance が付くことです。プランナは「信頼度が低い助言は弱く効かせる」「出どころに応じて重みを変える」といった安全な使い方ができます。各フィールドが Optional(あってもなくてもよい)になっていれば、ある情報源が落ちても残りで動ける耐障害性も得られます。これにより、プランナを特定の1つの推論ヘッドに縛りつけない(ベンダーロックインならぬモジュールロックインを避ける)柔軟性が生まれます。

### System2 因果ヘッド(因果推論モジュール)

因果ヘッドは、コンテキストベクトル(例: プランナの再帰隠れ状態 形状 `[B, 256]`)を入力に取り、2つを出します。

1. `reasoning_latent` `[B, latent_dim]` — 連続的な推論潜在ベクトル。下流の計画に足す/連結する用。
2. `decision_logits` `[B, num_classes]` — 因果クラスの分類スコア。典型的には固定順の数クラス、例えば
   `("intersection", "pedestrian", "traffic_light", "obstacle", "clear")` = (交差点 / 歩行者 / 信号 / 障害物 / クリア)。

内部は小さな全結合トランク(例: `Linear(256→512) → GELU → Linear(512→latent) → LayerNorm`)と、その上の分類ヘッド(`Linear(latent→num_classes)`)です。パラメータ数は数十万程度で、知覚バックボーンや VLM に比べれば極めて軽量です。

カスケード設計が重要です。推論潜在だけを取り出してプランナに足したり連結したりすれば、「知覚 → 推論 → 軌跡」の段階的構成ができます。勾配を止めなければ、計画損失と因果損失の両方が共有表現まで流れ、因果という補助目的が表現の質を押し上げます。逆に、推論層を実行時に重くしたい(VLM を毎フレームは回せない)場合は、ここで勾配や情報の流れを切り、非同期に更新する設計も選べます。

### 学習信号

因果クラスの学習には、`decision_logits` と疑似ラベル(pseudo-labels)のクロスエントロピー(分類の標準損失)を使います。疑似ラベルは、ロングテールなシーンを VLM に「この場面の支配的な因果要因は?」と尋ねて得た答えを、少数クラスにマップしたものです。人手でラベル付けするのは高価なので、強力な VLM に自動で付けさせる「蒸留(distillation)に近い」発想です。VLM 生成なのでノイズが多く、次の工夫が併用されます。

- ラベル平滑化(label smoothing): 正解クラスに確率を集中させすぎず(例: 正解に 0.9、残りに均等配分)、間違った疑似ラベルへの過罰を和らげる。ノイズの多い教師に過適合するのを防ぐ。
- クラス重み(class weights): 稀なロングテールクラスを重く扱い、頻出クラス(多くは "clear")に埋もれさせない。出現頻度の逆数で重み付けるのが定番。

補助目的なので、主タスクである模倣損失に対して低い重みで足すのが定石です。重くしすぎると、ノイズの多い疑似ラベルが主タスクを引っ張ってしまいます。

### コンテキストの生成

forward の出力をまとめ、`confidence = softmax(logits).max()`(最も確からしいクラスの確率)を計算し、provenance を付けて SceneContext を返します。これが「型付き・信頼度付き・出どころ付きの一プロデューサ」の具体です。プランナ側は受け取った confidence を使って、たとえば「confidence が閾値未満なら無視する」「confidence をゲートにして reasoning_latent をスケールする(`latent * confidence`)」といった安全な統合を実装できます。後者は、自信のある推論ほど計画に強く効き、自信のない推論はほぼ無視される、という滑らかな振る舞いを与えます。

## 手法の系譜と主要論文

- Kahneman(2011)"Thinking, Fast and Slow" — System1/System2 の二重過程理論。自動運転の「反射層 vs 熟考層」分離の発想的下敷き。速く自動的な処理と、遅く意識的な処理を分けるという認知科学の枠組みが、そのままアーキテクチャの二層分離に写される。

- UniAD(Hu et al., CVPR 2023 Best Paper, arXiv:2212.10156)— 知覚・予測・計画を1本のネットで統合し、各段が次段に効くカスケード設計を確立。検出 → 追跡 → 地図 → 動き予測 → 占有予測 → 計画と、タスクをクエリ(Transformer のクエリトークン)で数珠つなぎにした。「補助タスクの勾配が共有表現を底上げする」という思想の代表例で、本トピックの「知覚→推論→軌跡」カスケードと勾配を流す設計に通じる。

- DriveVLM(Tian et al., 2024, arXiv:2402.12289)、LMDrive(Shao et al., CVPR 2024, arXiv:2312.07488)、DriveGPT4(Xu et al., 2023, arXiv:2310.01412)など VLM ベースの運転モデル群 — VLM に場面を説明させ、その自由文の理由付けや指示を運転に繋ぐ流れ。Chain-of-Thought(段階的な推論を言葉で展開する)で「シーン記述 → 重要物体の抽出 → 行動決定」と進める。表現力が高い反面、自由文をどう安全に計画へ接続するかが課題として残る。本トピックの設計は、あえて「自由文を直接入れない・型付き構造体で受ける」点でこれらと差別化し、安全な統合を狙う。

- 因果や説明可能性を行動に接地させる流れ(decision grounding, chain-of-causation)は、近年の VLA(Vision-Language-Action)モデルで活発です。行動を因果の説明に接地させて出力することで、判断の根拠が辿れる解釈性と安全性を得る一方、VLM の推論コストとラベル生成コストがトレードオフになります。本トピックの因果ヘッドはこの発想の軽量版という位置づけで、重い言語推論を毎フレーム回す代わりに、少数の因果クラスへの蒸留で代替します。

系譜としては「Kahneman の二重過程理論を発想の土台に、UniAD がカスケードと補助タスクで表現を底上げ、VLM 系が自由文の理由付けを導入、そして自由文の危うさを型付きコンテキストで包んで安全に統合する」という流れで理解できます。

## 論文の実験結果(定量データ)

このアーキテクチャの効果は、(1) 計画指標の改善、(2) 解釈性・因果接地の正しさ、の2軸で測られます。

- UniAD は nuScenes の open-loop 計画で、平均 L2 誤差(予測軌跡と正解の距離)をおよそ 1.03m、衝突率をおよそ 0.31% と報告しました。重要なアブレーションは「知覚・予測の補助タスクヘッド(検出、追跡、占有予測、動き予測)を順に外していくと、最終的な計画 L2 と衝突率が悪化する」というもので、補助タスクの勾配が計画表現を底上げすることを定量的に示しました。たとえば動き予測モジュールを外すと衝突率が目に見えて上がる、といった具合です。これは因果ヘッドのような補助プロデューサを足す設計の根拠になります。

- DriveVLM は nuScenes と自社データで、VLM による場面理解を計画に繋ぐと、とくにロングテールで複雑なシーン(工事帯、特殊車両、複雑な交差点)での計画品質が向上すると報告しました。同時に、推論レイテンシ(数百ミリ秒〜秒オーダー)が実時間制御(10〜20Hz 必要)には重く、速い反射層との二層構成(DriveVLM-Dual: 遅い VLM 推論と速い従来プランナを組み合わせる)が必要だと示しました。これは System1/System2 分離の実証的な裏づけで、熟考層を毎フレーム回さず低頻度で更新し、反射層が高頻度で制御する、という非同期協調の設計を示しています。

- 自由文 vs 型付き(構造化)出力の比較研究は標準データセットでの確立した代表値はまだ乏しいものの、報告される定性的傾向は一貫しています。自由文を直接プランナに入れると、言い回しの揺れや幻覚(hallucination, もっともらしいが誤った説明)に計画が引きずられやすい。対して、少数の因果クラス + 信頼度に落とすと、低信頼の助言を弱められるため、decision flicker が減り出力が安定する、という方向です。アブレーションとしては「confidence ゲートを外して全助言を等重みで効かせると、ノイズの多い疑似ラベルに引っ張られて計画がむしろ悪化する」ことが観察され、信頼度付きで受ける設計の意義を支えます。

指標の意味を補足すると、L2 誤差は「正解の道筋にどれだけ近いか(小さいほど良い、単位 m)」、衝突率は「予測軌跡が他者とぶつかる割合(小さいほど安全)」です。補助ヘッドを足して L2 が同等でも衝突率が下がるなら、それは「同じくらい正解に近いが、より安全側に倒した計画」を意味し、価値があります。逆に L2 だけ追うと「平均的に正解に近いが稀に危険」な計画を見逃すため、両指標を併せて見る必要があります(open-loop L2 の限界については [Bézier軌跡ヘッド](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0115_bezier-trajectory-head) も参照)。

## メリット・トレードオフ・限界

メリット
- 自由文を直接渡さず型付き構造体で受けるので、プランナが安定して解釈でき、特定の推論ヘッドへのロックインを避けられる。
- 各情報に信頼度と provenance が付き、安全な重み付け・取捨選択(低信頼は無視、情報源で重み変更)ができる。
- カスケードで勾配が共有表現まで流れ、因果という補助目的が表現の質を底上げする(UniAD で実証された効果)。
- 完全にオプションのプロデューサにすれば、デフォルトの計画挙動を壊さず追加できる。情報源の追加・削除がバスの規約だけで済む。

トレードオフ・限界
- 因果クラスが少数の固定集合に限定されると、現実の多様な因果を粗くしか表せない(連続 latent で補うが解釈性は落ちる)。固定クラスに当てはまらない新種の状況を表現できない。
- 学習ラベルが VLM 生成の疑似ラベルだとノイズが多く、品質は VLM の性能とラベル平滑化・クラス重み設定に依存する。VLM の偏りやエラーがそのまま蒸留される。
- 補助ヘッドであり、実際に decision flicker を減らす効果やロングテール頑健性の向上は、データセットと評価プロトコル次第で検証が必要。
- VLM による疑似ラベル生成自体に計算コストがかかる(オフラインで一度生成すればよいが、データが増えるたびに再生成が要る)。

未解決課題として、(1) 因果クラスを固定集合でなくオープン語彙に拡張しつつ型安全性を保つ方法、(2) 信頼度の校正(confidence が実際の正答率と一致するか, calibration。confidence 0.9 のとき本当に9割当たるか)、(3) 推論層(System2)と反射層(System1)の更新頻度・レイテンシをどう非同期に協調させるか、が挙げられます。

## 発展トピック・研究の最前線

VLM を運転に統合する研究は、自由文の説明を「中間表現」として扱い、そこから型付き・実行可能な指令に変換する方向(構造化デコーディング、関数呼び出し風の出力、JSON スキーマ強制デコード)へ進んでいます。言語の柔軟さは活かしつつ、最終的な出力は機械が確実に解釈できる型に落とす、という折衷です。また、推論をテスト時に複数ステップ展開する Chain-of-Thought 的な手法と、それを安全に計画へ接地させる decision grounding の研究が活発です。

信頼度付きの構造化バスという発想は、複数の異種プロデューサ(VLM、地図、ルータ、他車予測器)を1つのプランナに安全に束ねる「センサ融合の上位版」として、モジュラーかつ検証可能な自動運転スタックの設計に広がっています。各モジュールが独立に開発・検証でき、安全評価(どの情報源がどの判断に効いたかを provenance で追跡できる)もしやすくなります。速い反射層([Bézier軌跡ヘッド](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0115_bezier-trajectory-head))と遅い熟考層を役割分離し、後者を強制でなく助言として接続する構成は、その中心的な設計パターンです。世界モデル([FutureState + JEPA損失](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0014_futurestate-jepa-loss))が提供する未来予測を、この熟考層の追加プロデューサとしてバスに乗せる統合も自然な発展方向です。
