# 特徴融合前のembed_dim射影(256次元統一)

## ひとことで言うと

バックボーンが出す特徴は、複数段を連結すると 1440 チャネル(ResNet-50 なら 3840)と非常に太くなります。これを 1x1 畳み込みで一律 256 チャネルに細くしてから、複数カメラの融合や軌道予測など下流の処理に渡す、という設計の話です。これで下流の全モジュールが「バックボーンが何でも・入力解像度がいくつでも、内部は 256 次元」という一定の前提で動けます。256 という数字は思いつきではなく、DETR 以降の主要な検出・自動運転モデルが共有するデファクト標準です。

## 直感的な理解

なぜわざわざ太い特徴を細くするのか。3つの理由を直感で押さえます。

第一に、計算量です。下流で使う注意機構(attention)の計算コストは、特徴の太さ(次元 d)の概ね2乗で効きます。これは注意の中で「全位置 × 全位置」の関係を d 次元のベクトル同士の内積で測るためで、内積1回が d に比例し、それを多数の位置対について行うので、支配項がおおむね d の2乗で増えます。1440 次元のまま注意を計算すると、256 次元の場合に比べておよそ 31 倍(1440² ÷ 256² ≈ 31.6)重くなります。入口で細くするだけで、下流全体が一桁以上軽くなります。

第二に、相互運用性です。知覚から計画までを1本のネットワークでつなぐエンドツーエンドの自動運転モデルでは、モジュールからモジュールへベクトルを渡し続けます。途中で次元がバラバラだと、毎回つなぎ目で変換が要ります。最初に 256 で統一しておけば、どのモジュールも同じ「太さ」のベクトルをそのまま受け渡せます。

第三に、バックボーン非依存です。Swin(1440)でも ResNet-50(3840)でも、射影の出口を 256 に固定すれば、下流は入力が何チャネルだったかを気にしなくてよくなります。差し替えコストがほぼゼロになります。

たとえるなら、メーカーの違ういろいろな水道管(太さがバラバラ)を、家の入口で1種類の標準口径に変換してしまうようなものです。家の中の配管(下流)はすべて同じ口径だけ考えればよくなります。

## 基礎: 前提となる概念

- embed_dim(埋め込み次元): モデル内部で1つの「位置」や「トークン」を表す数値ベクトルの長さです。256 なら各位置を 256 個の数で表します。多くのモジュール(とくに Transformer 系の注意機構)は、この値を共通の「内部の太さ」として使います。embed は embedding(埋め込み)の略で、「対象を固定長ベクトルで表す」という意味です。

- 1x1 畳み込み(Conv 1x1): 各ピクセルを独立に、チャネル方向だけで線形変換する操作です。空間的な「ぼかし」(隣のピクセルとの混ぜ合わせ)は一切せず、チャネル数を `in → out` に変える「チャネルの組み替え・圧縮」として使います。1440→256 ならパラメータ数は `1440×256 ≈ 36.9 万`(プラス bias 256)です。これは全結合層(線形層)を空間の各位置に同じ重みで適用するのと数学的に等価です。

- マルチビュー融合(view fusion): 複数のカメラ(自動運転車では前後左右で 6〜8 視点が典型)の特徴を1つの表現にまとめる処理です。代表的な方式に、視点をチャネル方向に連結する concat 融合、視点をまたいだ注意で混ぜる cross-attention 融合、3D グリッドへ貼り直す BEV(鳥瞰図)融合があります。

- なぜ「融合の前」に射影するか: 融合は視点数 V 個ぶんの特徴を扱うので、太いまま融合すると計算量が `V × (太さ)` で膨らみます。たとえば連結融合で `Conv1x1(V × 1440, 1440)` を作ると、V=8 なら約 1660 万パラメータですが、256 にしてからなら約 50 万パラメータで済みます(約 1/33)。だから融合の手前で細くするのが効率的です。

## 仕組みを詳しく

典型的な実装は2段階です。

ステップ1: 多段特徴をプールして連結する

複数段の特徴マップは解像度がバラバラ(例 64x64, 32x32, 16x16, 8x8)なので、まず適応プーリング(`AdaptiveMaxPool2d` など、出力サイズを指定するとそれに合わせて自動でプール窓を決める操作)で全段を同じ空間サイズ(例 8x8)に揃えます。そのうえでチャネル方向に連結します。

```python
for i in range(len(features)):
    features[i] = pool(features[i])         # 全段を [B*V, ch_i, 8, 8] に
fused_per_view = torch.cat(features, dim=1) # [B*V, 1440, 8, 8] (Swin の場合)
```

`B` はバッチ、`V` は視点数です。8視点・バッチ4なら `B*V = 32`。ここで `[32, 1440, 8, 8]` というテンソルができます。

ステップ2: 1x1 畳み込みで 256 に射影し、非線形を通す

```python
channel_proj = nn.Sequential(
    nn.Conv2d(backbone_channels, embed_dim, kernel_size=1),  # 1440 → 256
    nn.GELU(),
)
fused_per_view = channel_proj(fused_per_view)   # [B*V, 256, 8, 8]
```

GELU(Gaussian Error Linear Unit、入力をなめらかにゲートする非線形関数で、Transformer 系で広く使われる)は、ただチャネルを線形圧縮するだけでなく「圧縮後にひとひねり加える」ことで表現力を保ちます。ここの `backbone_channels` は動的取得で求めた値(Swin 1440 / ResNet-50 3840)が入るので、どのバックボーンでも出力は必ず 256 になります。

before/after(Swin の例)

- before: `[B*V, 1440, 8, 8]` を視点融合へ。下流はすべて 1440 前提でハードコード。
- after: `[B*V, 256, 8, 8]` を視点融合へ。下流は 256 前提で固定。バックボーンを変えても 256 は不変。

その後、256 次元になった特徴を視点融合モジュールへ渡します。BEV 融合なら出力は `[B, 256, bev_h, bev_w]`(例 450x300)、concat / cross-attention なら `[B, 256, 8, 8]` です。いずれにせよチャネルは 256 で一定です。

なぜ 256 なのか、512 ではだめなのか、という設計判断にも触れておきます。256 は DETR 系の標準で、論文のハイパーパラメータや学習レシピがそのまま流用できる利点があります。512 にすれば表現の上限は上がりますが、注意の計算量は 4 倍(512²/256²)になり、メモリも増え、過学習のリスクも上がります。タスクの難しさとデータ量に応じて 256/384/512 から選ぶのが実務で、まず標準の 256 から始めるのが定石です。

設計上の注意点として、「いつプールするか」と「いつ射影するか」の順序があります。上の例は「プール → 連結 → 256射影」ですが、空間情報を重んじるなら「段ごとに 256射影 → 各段を保ったまま下流(BEV 等)へ」という FPN 流の順序もあります。前者は計算が軽い代わりに空間解像度がプール出力(8x8 など)に縛られ、後者は空間を保てる代わりに各段に射影層が必要でメモリが増えます。どちらが良いかはタスクの空間精度要求で決まります。

## 手法の系譜と主要論文

- DETR(Carion et al., ECCV 2020, arXiv:2005.12872): バックボーン特徴を 1x1 畳み込みで 256 次元へ落としてから Transformer に入れる構成を確立しました。理由は、Transformer の計算量を抑えつつ、物体クエリ(検出候補を表すベクトル)と画像特徴を同じ次元空間で扱うためです。DETR の hidden dimension は 256 で、これがこの系譜のデファクトになりました。

- Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159): DETR の遅い収束と高い計算量を、変形可能注意(少数の点だけ参照する注意)で改善しました。マルチスケール特徴をそれぞれ 256 へ射影して扱う点を継承し、256 次元で複数解像度をまたいだ注意を実現しました。

- PETR / PETRv2(Liu et al., ECCV 2022 / 2023, arXiv:2203.05625): 3D 位置エンコーディングを画像特徴に足して 3D 物体検出を行う設計でも 256 を採用。位置情報を特徴に注入する際も、共通次元 256 だからこそ要素ごとの足し算で混ぜられます(次元が違えば足せない)。

- BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270): BEV クエリと画像特徴を embed_dim=256 で統一し、空間 cross-attention で BEV を構築します。BEV クエリ自体が 256 次元のベクトルの集合で、画像側も 256 に揃えるからこそ注意が計算できます。

- UniAD(Hu et al., CVPR 2023, arXiv:2212.10156)/ VAD(Jiang et al., ICCV 2023, arXiv:2303.12077): 知覚・予測・計画を1本のネットワークにつなぐ際、全モジュールを 256 次元で貫くことでクエリの受け渡しを容易にしています。embed_dim を統一すること自体が「エンドツーエンド化」の前提になっています。UniAD は CVPR 2023 の Best Paper に選ばれ、この「クエリで全モジュールをつなぐ」設計を一気に主流化しました。

これらに共通する効果は「計算量の抑制」と「モジュール間でベクトルをそのまま渡せる相互運用性」、トレードオフは「次元を絞るぶん表現の上限が下がりうる(ただし 256 で実用上十分というのが各論文の経験則)」点です。

## 論文の実験結果(定量データ)

DETR(ECCV 2020)は COCO 物体検出で、同等の学習設定の Faster R-CNN ベースラインに対して box AP がおよそ同等以上(DETR-DC5 で約 43〜44 AP)を達成しました。ここで AP は検出の正確さを 0〜100 で測る指標で高いほど良いものです。重要なのは、この性能を 256 次元という比較的細い内部表現で達成した点で、太い特徴を素直に使う設計より大幅に軽量です。一方で DETR は学習に約 500 エポックを要し、小物体検出(APs)が弱いという弱点も同時に明らかにしました。

Deformable DETR(ICLR 2021)は、DETR が必要とした 500 エポックの学習を約 50 エポック(約 1/10)に短縮しつつ、COCO box AP を約 43.8(DETR 相当の設定)から約 46.2(マルチスケール変形可能注意込み)へ改善したと報告しています。エポックとはデータ全体を1周する学習単位で、これが少ないほど学習が速いことを意味します。256 次元のマルチスケール変形可能注意が、速さと精度の両立を可能にしました。小物体 APs も DETR より改善し、共通次元 256 を保ったまま複数解像度を扱えることの効果が示されました。

BEVFormer(ECCV 2022)は nuScenes(自動運転の標準ベンチマーク、6カメラ・LiDAR・レーダ付きの大規模データセット)で、3D 物体検出の総合指標 NDS(nuScenes Detection Score、検出精度と位置・速度・属性の正しさを総合した 0〜1 の指標で高いほど良い)が約 0.517、mAP(平均適合率)が約 0.416 を達成しました。BEV クエリ・画像特徴ともに 256 次元で統一されており、この共通次元設計が空間 cross-attention の計算を成立させています。

アブレーションの観点では、embed_dim を 256 から大きくしても、多くの論文では精度の伸びは緩やか(数値が頭打ちになりやすい)で、計算量だけが2乗で増えることが知られています。逆に極端に小さく(128 など)すると、複雑なシーンで表現が不足し精度が落ちる傾向が報告されています。256 は「精度と計算量のスイートスポット」として経験的に選ばれている、というのが系譜全体の含意です。

## メリット・トレードオフ・限界

メリット

- 計算量とパラメータ数が大幅に減る(注意機構で約 1/31、連結融合で約 1/33)。
- バックボーンを差し替えても下流が 256 で不変。差し替えコストが激減する。
- 業界標準の 256 なので、論文の設定・事前学習済み重み・ハイパーパラメータを流用しやすい。

トレードオフ・限界

- 1440→256 は情報を捨てる圧縮なので、原理的には表現の上限が下がる。難しいタスクでは 384 や 512 の方が良い場合もある。
- 射影層が1枚増えるので、わずかにパラメータと計算が足される(ただし下流の削減効果が圧倒的に大きいので差し引きは大きくプラス)。
- 先にプール(例 8x8 への圧縮)で空間解像度を落としてから射影する構成だと、細かい空間情報がプール解像度に制約される。空間情報が重要なタスクでは、プールせず空間を保ったまま射影する設計が望ましい。
- 未解決の課題として、「最適な共通次元はタスク・データ量・バックボーンの組み合わせで変わる」ため、256 固定が常に最善とは限らない。次元自体を学習やアーキテクチャ探索(NAS)で決める方向は研究の余地がある。

## 発展トピック・研究の最前線

- マルチスケール特徴を共通次元へ射影してから融合する考えは、FPN の横方向結合(全段を 256 へ)が源流で、検出から自動運転まで広く受け継がれている。
- 射影を単純な 1x1 ではなく、軽量な MLP やボトルネック(いったん絞ってまた広げる)にする変種もある。表現力と計算量のトレードオフを微調整できる。深さ方向に1層足すだけで非線形性が増えるが、過学習とのバランスに注意。
- 近年のクエリベースのエンドツーエンドモデル(UniAD, VAD, それ以降の世界モデル系)は、共通次元の維持を「設計原則」として明示しており、ここを崩すとモジュール接続が破綻するため、射影は単なる前処理ではなくアーキテクチャの背骨になっている。次元を統一したうえで、モジュール固有の情報(時刻・カメラ ID・物体種別など)を位置エンコーディングとして 256 次元の中に埋め込むのが現代的な作法。
