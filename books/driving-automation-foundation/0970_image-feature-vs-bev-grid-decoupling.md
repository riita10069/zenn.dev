# 画像特徴解像度とBEVグリッドの分離

## ひとことで言うと
カメラ画像から取り出した特徴の「マス目の細かさ」（画像特徴の解像度）と、車を真上から見下ろした地図（BEV）の「マス目の細かさ」（BEVグリッド解像度）を、別々の独立したパラメータとして持つ設計です。両者をわざと切り離し、間を「投影（projection）」という幾何計算でつなぎます。これにより、画像側は粗く・地図側は細かく、といった自由な組み合わせが計算予算と精度要求に応じて選べます。

## 直感的な理解
地図アプリを思い浮かべてください。画面に表示する地図の細かさ（ズームレベル）と、その地図を描くのに使う元データ（衛星写真や道路データ）の解像度は別物です。荒い元データから細かい地図を描くこともできるし、その逆もできます。両者を無理に一致させる理由はありません。

自動運転でも同じです。画像特徴の解像度は「カメラ画像のどれくらい細かいところを見るか」を決め、BEVグリッド解像度は「地面を何メートル刻みで表現するか」を決めます。この2つは全く別の都合で決まります。画像特徴は計算予算（GPU メモリと速度）で決まり、BEVの細かさは「どこまで遠くを、どれだけ正確に計画したいか」という安全要求で決まる。だから独立に持つのが自然です。

さらに本質的な理由があります。画像と地面は遠近法（perspective、遠いものほど小さく写る）で結ばれているので、対応が一対一ではありません。遠くのものは画像上で小さく写り、近くのものは大きく写ります。画像上の1ピクセルが、地面上では場所によって広かったり狭かったりします（地平線近くでは1ピクセルが何十メートルにもなる）。だから「画像のマス目」と「地面のマス目」を同じ数にしても意味がなく、両者は幾何変換（投影）でつなぐしかないのです。

## 基礎: 前提となる概念

### 特徴マップ（feature map）
backbone（画像から特徴を取るネットワーク本体）が画像から作る「マス目状の特徴の格子」です。`8×8` の特徴マップは、元画像を縦横8×8の領域に分け、各領域に例えば256次元の特徴ベクトルを置いたもの、というイメージ。マス目が細かい（解像度が高い）ほど位置情報が細かくなりますが、計算とメモリが増えます。多くの backbone は浅い層ほど細かく、深い層ほど粗い複数解像度の特徴を返します。

### BEV（Bird's-Eye-View）
車を真上から見下ろした俯瞰表現。地面を `bev_h × bev_w`（例 450×300）のマス目に区切り、各マスに「そこに何があるか」の特徴ベクトルを置きます。経路計画はこの俯瞰地図の上で行うと直感的で扱いやすく、自動運転の中間表現として広く使われます。BEVグリッドには物理的な意味があり、例えば前後120m・左右60mを 450×300 マスでカバーすると1マスはおよそ 0.4m 四方になります。これを「物理解像度（メートル/マス）」と呼びます。物理解像度が粗いと、近接した2台の車が同じマスに潰れて区別できなくなります。

### 投影行列とカメラ較正
3D空間の点を画像上の2Dピクセル位置に対応づける変換が投影（projection）です。カメラの内部パラメータ（焦点距離・主点＝光軸が画像と交わる点）と外部パラメータ（車両座標系に対するカメラの位置・姿勢）を合わせた `[3, 4]` の投影行列で表されます。マルチカメラなら `[B, V, 3, 4]`（B はバッチ、V はカメラ台数）。これらのパラメータを正確に測定する作業がカメラ較正（calibration）で、較正がずれると特徴の貼り付け位置がずれます。投影は同次座標（homogeneous coordinates、点に1を足して4次元で扱う表記）で行列積として書け、最後にスケールで割って2Dピクセルにします。

### grid_sample（双線形補間サンプリング）
特徴マップ上の任意の小数座標の値を取ってくる関数です。座標 (3.7, 5.2) のように整数でない位置を指定すると、周囲4マスの値を距離で重み付け平均（双線形補間, bilinear interpolation）して返します。Spatial Transformer Networks（Jaderberg et al., NeurIPS 2015, arXiv:1506.02025）で微分可能なサンプリングとして定式化され、PyTorch では F.grid_sample として提供されます。微分可能なので勾配が通り、投影で読み取った特徴をそのまま学習に使えます。座標は通常 [-1, 1] に正規化して渡すので、特徴マップが `8×8` でも `64×64` でも同じインターフェースで読めます。これが「解像度を一致させなくてよい」ことを実装レベルで支える鍵です。

## 仕組みを詳しく

2つの解像度を完全に別パラメータとして持ち、投影でつなぐ典型的な実装を示します。

画像特徴側: image_feature_size（例 8）。

backbone が返すマルチスケール特徴（`64×64`, `32×32`, `16×16`, `8×8` と解像度がバラバラ）を、適応的プーリング AdaptiveMaxPool2d(8) で全部 `8×8` に揃えてからチャンネル方向に連結します。AdaptiveMaxPool2d は「出力サイズを指定すると入力がどんな大きさでも自動でそのサイズに圧縮するプーリング」です。

```
backbone出力(per camera): (96,64,64), (192,32,32), (384,16,16), (768,8,8)
   │ AdaptiveMaxPool2d(8) で全部 8×8 に
   ▼
連結: (96+192+384+768 = 1440, 8, 8)
   │ 1x1 Conv で 256 次元へ
   ▼
fused_per_view: (B*V, 256, 8, 8)
```

BEVグリッド側: bev_h, bev_w（例 450×300）。BEVのマス目は画像特徴の `8×8` とは無関係に決められ、`bev_h ≠ bev_w` の非正方形も許されます（縦横で見たい距離が違うため。前後は遠くまで、左右はそこまで要らない、など）。

間をつなぐ投影と再マッピング。両者は数だけ別々に持つのではなく幾何的につながっています。BEV 融合は次を行います。

1. BEVの各マス（全部で `450×300 = 135,000` マス）に学習可能なクエリベクトルを持たせる。
2. 各BEVマスは地面上の3D座標に対応する。その3D点（必要なら複数の高さ z でサンプル）を投影行列 `[B, V, 3, 4]` で各カメラの画像上の2D位置に投影する。
3. その2D位置で、`8×8` の画像特徴を grid_sample（双線形補間）で読み取る。画像の外に落ちた投影は無効としてマスクする。
4. 読み取った値を attention で重み付けして集め、そのBEVマスの特徴にする（どのカメラのどの高さが有用かを学習で決める）。

```
BEVマス → 対応する3D点 → 投影行列で画像の2D位置 → 8×8 特徴を補間で読む → attention 集約
```

grid_sample が補間してくれるので、画像特徴が `8×8` と粗くても、BEVが `450×300` と細かくても、座標さえ計算できれば値を取り出せます。これが「両者の解像度を一致させる必要がない」理由です。

shape のまとめ:

```
入力画像特徴(値): (B, V, 8*8, 256)        ← image_feature_size=8 由来
BEVクエリ:        (B, 450*300, 256)        ← bev_h, bev_w 由来
投影 + grid_sample で再マッピング
出力BEV特徴:      (B, 256, 450, 300)
```

なぜ分離が得かを数値で見ます。仮に画像特徴とBEVを結合して同じ解像度にすると、BEVを 450×300 に細かくしたいときに画像特徴も 450×300 に持つはめになります。7カメラぶんなら `7 × 450 × 300 × 256` ≈ 2.4億要素の特徴を保持することになり、メモリが破綻します。分離していれば画像特徴は `7 × 8 × 8 × 256` ≈ 0.1M要素のまま、BEVだけ細かくできます。要素数で見て3桁の差で、これがそのまま GPU メモリと速度の差になります。

設計上の注意として、地面が平らだと仮定して z=0 の1点だけ投影する素朴な方式（IPM, Inverse Perspective Mapping）は、立体物（車や歩行者）の高さを無視するため、立体物の BEV 位置が歪みます。BEVFormer のように複数の高さ z をサンプルして集約する、あるいは LSS のように深度分布を推定する方式は、この高さの曖昧さに対処するための工夫です。

## 手法の系譜と主要論文

「画像の格子とBEVの格子は別物で、間は投影でつなぐ」という設計は、カメラ系BEV認識の各論文で共通します。

- LSS（Lift, Splat, Shoot; Philion & Fidler, ECCV 2020, arXiv:2008.05711）。画像特徴を離散深度ぶん持ち上げてからBEVグリッド（例 200×200、物理解像度 0.5m）に splat します。BEVグリッドの物理解像度（メートル/マス）は画像特徴の解像度とは独立に設定されます。投影の向きが「画像→BEV」（forward projection）である点が後述の attention 系と対照的。トレードオフは深度推定誤差がBEVに伝播すること。

- BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）。BEVクエリの解像度（例 200×200）と各カメラ特徴マップの解像度を独立に持ち、spatial cross-attention で投影してつなぐ設計を明確に打ち出しました。投影の向きは「BEV→画像」（BEVマスから画像を引く、backward projection）。分けた動機は、BEVの細かさは知覚範囲と精度要求で、画像特徴の細かさは計算予算で、それぞれ別の都合で決まるから。効果は両者を独立に調整できる柔軟性。トレードオフは投影に正確な較正が必要で、較正ずれが貼り付け位置のずれに直結すること。

- BEVDepth（Li et al., AAAI 2023, arXiv:2206.10092）。解像度を分離したうえで、深度推定を明示的に教師あり学習して投影精度を上げました。動機は「解像度を分離しても投影がずれると意味がない」ため、つなぎ目の精度向上が重要だと示したこと。LiDAR点群から深度の正解を作り、深度ヘッドを直接監督します。

- Simple-BEV（Harley et al., ICRA 2023, arXiv:2206.07959）。深度推定も attention も使わず、BEVマスを画像に投影してただ双線形サンプリングするだけの素朴な方式でも、解像度や入力（カメラ解像度・バッチサイズ）を上げれば競争力が出ることを示し、「凝った投影機構より解像度などの基本設定の影響が大きい」ことを実験的に明らかにしました。解像度を独立軸として扱う本ノートの主張を裏付ける研究です。

- BEVDet（Huang et al., 2021, arXiv:2112.11790）。画像エンコーダ解像度とBEVエンコーダ解像度を別々のハイパーパラメータとして扱い、それぞれの解像度を変えるアブレーションを系統的に行いました。

## 論文の実験結果（定量データ）

nuScenes（6カメラ、1000シーン）での mAP（平均適合率）と NDS（総合スコア）で評価されます。両指標とも大きいほど良い。

- BEVDepth は nuScenes で、深度を明示監督することで素の LSS 系（BEVDet）に対し mAP を約数ポイント、NDS を顕著に改善したと報告（NDS おおよそ 0.60 級まで押し上げ）。これは「解像度を分離したうえで投影精度を上げると効く」ことの直接的証拠です。アブレーションでは深度の明示監督を外すと mAP が大きく低下し、つなぎ目の精度が性能を支配することが示されました。
- BEVFormer のアブレーションでは、BEVクエリ解像度を上げると遠方・小物体の検出が改善する一方、計算量が(おおよそ)二乗で増えることが報告されています。画像特徴側の解像度（backbone 出力）を変えるアブレーションとは効果の出方が異なり、両者が独立な調整軸であることを裏付けます。
- Simple-BEV は、入力カメラ解像度を上げる・バッチサイズを上げるといった基本設定が、深度監督や複雑な投影機構の有無よりも検出スコアを大きく動かすことを示しました。これは「解像度を独立に調整できる設計」の価値を直接実証しています。
- 一般に、画像特徴を粗くしてもBEVを細かく保てば近距離の計画精度は維持できる一方、遠方の物体は元の画像情報が不足するため、いくらBEVを細かくしても補間で水増しされるだけで精度が頭打ちになる、という関係が観測されます。これが「下流の解像度は上流の情報量を超えられない」という原則の実証です。

数値の意味の補足。mAP が数ポイント上がるのは、見逃しと誤検出が減ることで、自動運転では「横断する歩行者を捉えられるか」に直結します。投影較正の誤差は致命的で、較正が数ピクセルずれるだけで遠方物体のBEV位置が数メートルずれ、mAP が大きく落ちることが各論文で報告されています。だからBEVDepth のような「つなぎ目を鍛える」方向が重視されます。

## メリット・トレードオフ・限界

メリット:
- 画像特徴を粗く・BEVを細かく（またはその逆）、用途に合わせて独立に調整できる。
- BEVを高解像度化してもカメラ枚数ぶんの画像特徴メモリは増えない（画像特徴は小さい解像度のまま）。
- テストでは BEV を小さく（例 8×8）落として高速化し、本番だけ高解像度（例 450×300）にする、といった切り替えが容易。これは解像度が独立パラメータだからこそ可能。
- 縦横で異なる解像度（前後は遠くまで細かく、左右は粗く）を自然に表せる。

トレードオフ・限界:
- 両者をつなぐのに正確なカメラ投影行列が必要。較正がない、または不正確だと、貼り付け位置がずれて意味のあるBEVにならない。較正が無い場合に学習可能なダミー投影で代用する実装もあるが、真の幾何ではないので幾何的に正しいBEVは得られない。
- 投影と grid_sample の計算が増える。解像度を別々にした代償としてつなぐ処理が複雑になり、計算グラフも深くなる。BEVマス×カメラ×高さ×時刻のぶんだけサンプリングが走るため、BEV解像度を上げると計算が(おおよそ)二乗で増える。
- 画像特徴が粗すぎる（例 8×8）と、いくらBEVを細かくしても元の情報がないため細部は補間で水増しされるだけ。下流の解像度は上流の情報量を超えられない。
- 平地仮定（z=0 投影）は立体物の高さで歪む。複数高さサンプルや深度推定で緩和するが、完全には消えない。
- 未解決の課題として、画像特徴解像度を領域ごとに適応的に変える（遠方を映す画像上部だけ高解像度にする）非一様サンプリングや、較正誤差にロバストな投影（オンライン較正補正）が研究途上です。

## 発展トピック・研究の最前線
解像度分離と投影の枠組みは、深度の明示監督（BEVDepth）、時系列BEVの集約（BEVFormer v2, SOLOFusion）、スパースな問い合わせで計算量を抑える方向（Sparse4D, PETRv2）へと発展しています。grid_sample による固定位置の微分可能サンプリングは Deformable Attention（Deformable DETR; Zhu et al., ICLR 2021, arXiv:2010.04159、学習でサンプリング位置自体をずらす機構）に一般化され、較正の小さな誤差を学習で吸収する試みにつながっています。研究の関心は「固定した投影でつなぐ」から「投影自体を学習・補正してつなぎ目の精度を上げる」方向、さらには明示的な BEV グリッドを持たずスパースな3Dクエリで直接タスクを解く方向（疎な表現でメモリと計算を節約する）へと移っています。
