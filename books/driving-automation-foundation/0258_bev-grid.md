# BEVグリッド

## ひとことで言うと
BEVグリッドとは、車を真上から見下ろした図(鳥瞰図、Bird's-Eye-View、これは「鳥が空から地面を見たような俯瞰の図」という意味です)を、方眼紙のように格子状のセル(マス目)に区切り、各マスに「そこに何があるか」を表す数のかたまり(特徴ベクトル、これは「対象の性質を数値の並びで表したもの」という意味です)を持たせる空間表現のことです。複数のカメラがバラバラに撮った映像を、この「自車を中心とした地面基準の格子」という1つの共通の土俵に集約することで、後段の地図照合や軌道計画を素直に書けるようにします。現代のカメラベース自動運転認識のほぼ全てが、この BEV グリッドを中間表現として採用しています。

## 直感的な理解
複数の防犯カメラの映像から、店内のどこに誰がいるかを把握したいとします。各カメラは別々の角度・別々の歪みで人を映すので、映像をそのまま見比べても「あの人とこの人は同じ人か」「2人の間は何メートルか」が分かりません。賢いやり方は、店の床の見取り図(真上から見た間取り図)を1枚用意し、各カメラの映像から「この人は床のここに立っている」と割り出して、見取り図の上に点を打っていくことです。一度この見取り図に集約してしまえば、距離も位置関係も一目で分かります。

自動運転も同じです。前・後・左・右に向いた複数のカメラは、それぞれ別の景色を撮ります。同じ歩行者でも前カメラと左カメラでは写る位置も大きさも違う。しかし運転の判断(「あと何メートルで止まるべきか」「左に寄れるか」)に必要なのは、カメラ視点の映像ではなく「自分の車を中心にして、周囲の何メートル先に何があるか」という地面基準の地図です。この地面基準・俯瞰の座標系が BEV であり、それを格子に区切ったものが BEV グリッドです。

## 基礎: 前提となる概念

用語を順にかみ砕きます。

- 視点(perspective)とBEV: カメラが撮るのは「透視投影された画像」で、遠くのものは小さく、奥行きが圧縮されています。BEV(鳥瞰図)は「真上から見た地面の地図」で、遠近の歪みがなく、距離がそのまま地図上の距離に対応します。運転判断には後者が向いています。
- 座標系: BEV は普通「自車中心座標系(ego-centric frame)」で定義します。自車の中心を原点とし、前後方向を X 軸、左右方向を Y 軸、高さを Z 軸に取るのが典型です。
- 特徴ベクトル(feature vector): ニューラルネットワークが扱う「対象の性質を表す数値の並び」。例えば 256 個の数で1つのマスの中身(そこに車があるか、道路か、何もないか、といった抽象的な情報)を表現します。
- query(クエリ): Transformer(注意機構を使うネットワーク)で「情報を取りに行く側」のベクトル。BEV グリッドの各マスに学習可能な query を置き、「このマスを埋めるには画像のどこを見ればよいか」を学習させます。
- アテンション(attention、注意機構): あるベクトル(query)が、複数の候補(画像特徴)の中から「関連の強いものを重み付けで選んで集約する」仕組み。Transformer の中核です。
- カメラの内部・外部パラメータ(intrinsic / extrinsic): カメラがどの焦点距離で(内部)、車のどこに、どの向きで付いているか(外部)を表す行列。3次元の点が画像のどの画素に写るかを計算するのに使います。これがあるからこそ「BEV のこのマスは、このカメラ画像のこの位置に写る」と幾何的に対応づけられます。

なぜ BEV グリッドという「中間表現」を挟むのか。複数カメラの画像を直接つないで判断しようとすると、各画像の座標系がバラバラで、後段が「どこがどこか」を毎回解釈し直す羽目になります。いったん BEV という共通座標系に集約しておけば、地図情報(車線、信号)との照合も、軌道計画(どこを通るか)も、すべて「車中心の1枚の地図」の上で完結します。これは「複雑な前段を、後段が扱いやすい標準形式に正規化する」という設計の典型です。

## 仕組みを詳しく

BEV グリッドの定義に必要なのは、(1)格子の解像度、(2)格子がカバーする実世界の範囲、(3)各マスをどう埋めるか、の3つです。

(1)解像度。縦 H マス × 横 W マスの格子を作ります。例えば H=450, W=300 なら 135,000 マス。各マスに次元 D(例えば 256)の特徴ベクトルを持たせます。実装では、各マスの query を埋め込みテーブルとして用意するのが典型です。

```python
self.bev_queries = nn.Embedding(H * W, D)   # 例: 450*300=135000 行、各 256 次元
```

`nn.Embedding(135000, 256)` は「135000 行・256 列の表」で、各行が1マス分の 256 次元ベクトルです。学習を通じて、各行が「このマスを埋めるにはカメラ画像のどこを見ればよいか」を表す内容に育っていきます。

(2)範囲。格子が実世界の何メートルをカバーするかを、点群範囲(point cloud range)として指定します。例えば `pc_range = (-60, -60, -5, 120, 60, 3)` は (x最小, y最小, z最小, x最大, y最大, z最大) で、単位はメートルです。

- X方向(前後): -60m 〜 120m、つまり 180m
- Y方向(左右): -60m 〜 60m、つまり 120m
- Z方向(高さ): -5m 〜 3m、つまり 8m

1マスの大きさ(解像度)はこれと格子数から決まります。X 方向を W=300 で割ると 180/300 = 0.6 m/マス、Y 方向を H=450 で割ると 120/450 ≈ 0.27 m/マス。つまり1マスは数十センチ四方の地面を表し、歩行者1人(横幅約 0.5m)が1〜2マスに収まる細かさです。解像度を上げる(マスを細かくする)ほど小さな物体を表現できますが、マス数が増えてメモリと計算量が急増します。

(3)各マスの埋め方。ここが手法によって最も分かれる部分です。代表的な2系統があります。

方式A(クエリ投影 + アテンション、BEVFormer 系): 各マスの query を「3次元空間の点」とみなし、カメラの内部・外部パラメータで各カメラ画像へ投影します。その投影先の周辺の画像特徴を、アテンションで重み付き集約してマスに吸い上げます(spatial cross-attention)。1つの BEV マスは複数の高さ(Z 方向に何点かサンプル)を持ち、それぞれを投影して集約することで、高さ方向の曖昧さに対処します。

```
BEVマス(x,y) → 数点の3D点(x,y,z1),(x,y,z2),... → 各カメラへ投影
   → 投影先の画像特徴をアテンションで集約 → マスの特徴
```

出力の形(tensor shape)は `[B, D, H, W]`(例 `[B, 256, 450, 300]`)です。B はバッチサイズ(一度に処理する事例数)。

方式B(深度推定 + 持ち上げ、Lift-Splat 系): 各カメラの画素について「奥行きの確率分布」を予測し、その確率で画素特徴を3次元空間に持ち上げ(Lift)、BEV 平面に投げ落として(Splat)足し合わせます。query を投影する方式Aと逆向きで、画素を3D に飛ばす方式です。

両者の本質的な違いは「深度を明示的に予測するか否か」です。方式Bは深度予測の精度に結果が左右されますが、方式Aはアテンションで「どの高さ・どのカメラを見るか」を学習に委ねるので、明示的な深度予測を回避できます。

## 手法の系譜と主要論文

BEV グリッドという中間表現は、この数年で爆発的に発展しました。系譜を追います。

1. Lift-Splat-Shoot(Philion & Fidler, ECCV 2020, arXiv:2008.05711): カメラベース BEV の出発点。各画素に深度の確率分布を推定し、特徴を3D に持ち上げて BEV に集約する枠組みを確立しました。動機は「複数カメラを微分可能な形で BEV に統合し、end-to-end で学習したい」こと。新規性は深度分布を陽に持つ点と、それが微分可能で全体を一気に学習できる点です。トレードオフは深度予測のずれが BEV のずれに直結すること。

2. BEVDet / BEVDepth(Huang et al. 2021, arXiv:2112.11790 ほか): Lift-Splat を3D 物体検出に本格適用し、深度推定を LiDAR で教師あり学習することで精度を底上げしました。深度の正確さが BEV の質を決める、という Lift-Splat 系の性質を逆手に取り、深度監督を強化したのが新規性です。

3. BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270): 学習可能な BEV query を各カメラへ投影し、spatial cross-attention で特徴を吸い上げる方式を提案。深度の陽な予測を不要にしました。さらに temporal self-attention で「過去の BEV」を参照し、時間情報(動いている物体の速度など)を取り込みます。新規性は、深度予測なしでアテンションだけで高精度な BEV を作れることと、時間方向の融合を BEV 上で行うことです。多くの後続研究が手本にする標準形になりました。

4. BEVFusion(Liu et al., ICRA 2023, arXiv:2205.13542): カメラの BEV 特徴と LiDAR の BEV 特徴を、同じ BEV グリッド上で融合する枠組み。BEV グリッドが「センサ種別を問わない共通の土俵」として機能することを示し、マルチモーダル融合の標準アーキテクチャになりました。

5. FIERY(Hu et al., ICCV 2021)や予測系: BEV グリッド上で未来の占有(occupancy)や運動を予測する流れ。BEV が認識だけでなく予測・計画の基盤にもなることを示しました。これが現在の end-to-end 自動運転(認識→予測→計画を BEV 上で一貫して行う)につながります。

## 論文の実験結果(定量データ)

評価は主に nuScenes(6カメラ + LiDAR の大規模自動運転データセット)で行われ、3D 物体検出と BEV セグメンテーションが代表的なタスクです。

主要な指標を説明します。
- mAP(mean Average Precision): 検出した物体が「正しい位置・正しいクラス」で当たっている割合の総合点。高いほど良い。
- NDS(nuScenes Detection Score): nuScenes 独自の総合指標で、mAP に加え、位置・大きさ・向き・速度・属性の誤差を統合したもの。0〜1 で高いほど良い。BEV 手法の良し悪しを測る事実上の標準です。
- IoU(Intersection over Union): セグメンテーションで「予測した領域と正解領域の重なり具合」。1 に近いほど良い。

具体的な数値の流れ(報告値、構成により変動するためおよその水準):
- Lift-Splat-Shoot(2020)は nuScenes の車両 BEV セグメンテーションで、当時のベースラインを上回る IoU を達成し、カメラのみでも実用的な BEV が作れることを初めて広く示しました。
- BEVFormer(2022)は nuScenes 検出で、カメラのみの構成として当時 NDS およそ 0.51〜0.56 水準(マルチスケール・大型 backbone 構成)を報告し、それ以前のカメラのみ手法を明確に上回りました。重要なアブレーションとして、temporal self-attention(過去 BEV の参照)を外すと、特に速度推定の誤差(mAVE)が悪化することを示しました。これは「BEV 上で時間情報を融合することが、動く物体の速度推定に効く」ことの定量的証拠です。さらに spatial cross-attention を単純な投影に置き換えると検出精度が落ちることも示し、アテンションによる吸い上げの有効性を裏付けました。
- BEVDepth は、深度を LiDAR で明示的に監督することで、カメラのみの検出 NDS をさらに押し上げ(報告で 0.6 前後の水準)、「Lift-Splat 系は深度の正確さがボトルネック」という仮説を実験で裏付けました。深度監督ありとなしのアブレーションで、mAP・並進誤差(mATE)が顕著に改善することを示しています。
- BEVFusion(2023)は、カメラ BEV と LiDAR BEV を融合することで、単一センサ構成を大きく上回る NDS(報告で 0.7 超の水準)を達成し、BEV グリッドが異種センサ融合の共通基盤として有効であることを定量的に示しました。

これらの数値が示す本質は、(1)BEV 中間表現はカメラのみでも実用水準に達したこと、(2)時間融合と深度の扱いが精度を大きく左右すること、(3)BEV はマルチモーダル融合の土俵として優れていること、の3点です。

## メリット・トレードオフ・限界

メリット
- 複数カメラ(さらには LiDAR・レーダー)の情報を「自車中心の地図」という後段が扱いやすい共通形式に統一できる。地図情報との融合や軌道計画がこの上で素直に書ける。
- 距離・位置関係が地図上の距離にそのまま対応するため、運転判断に直結する量(衝突余裕、車線内位置)を素直に計算できる。
- 認識・予測・計画を同じ BEV 表現の上で一貫して扱え、end-to-end 自動運転の自然な基盤になる。

トレードオフ・限界
- 解像度とメモリ・速度のせめぎ合い。マスを細かくすると小さな物体まで表現できますが、例えば 450×300=135,000 マスの query すべてを全カメラへ投影しサンプリングするため、計算量とメモリが急増します。高解像度 BEV + 補助モジュール併用では、バッチサイズを上げると容易に GPU メモリ不足(OOM)に陥ります。荒くすると軽くなりますが、近くの小物体がマスに埋もれます。
- 高さ方向(Z)の情報が潰れやすい。BEV は本質的に2次元の地図なので、高架下・立体交差・段差のような「同じ平面位置に高さ違いで物がある」状況の表現が苦手です。複数高さをサンプルする工夫はありますが、原理的な弱点です。
- 深度・幾何の誤差に敏感。カメラの外部パラメータ(取り付け位置・向き)の校正誤差や、Lift-Splat 系では深度予測の誤差が、BEV 上の位置ずれに直結します。
- 遠距離の希薄さ。遠方は1マスがカバーする実距離が画像上では小さな領域にしか対応せず、遠くの物体ほど特徴が希薄になり検出が難しくなります。
- 研究上の未解決課題として、(1)高解像度 BEV の計算コストをどう下げるか(疎な query、可変解像度 BEV)、(2)高さ情報を失わない 3D 表現(occupancy グリッド、tri-perspective view)への発展、(3)校正誤差にロバストな BEV 構築、が活発に研究されています。

## 発展トピック・研究の最前線
- Occupancy グリッド / 3D 占有予測: BEV の2次元の限界を超え、3次元の格子(ボクセル)で「各立体セルが占有されているか」を予測する流れ。CVPR の Occupancy 系コンペが牽引しています。高さ情報を保持できるのが利点です。
- TPV(Tri-Perspective View): 3つの直交する平面で3D を表現し、ボクセルより軽く BEV より表現力が高い中間表現。
- 疎な BEV / 可変解像度: 全マスを均一に計算する無駄を省き、物体がありそうな領域だけ細かく query を置くアプローチ(Sparse BEV など)。計算コスト問題への直接の回答です。
- End-to-end 自動運転での BEV: UniAD(CVPR 2023 best paper)や VAD のように、BEV を起点に検出・地図・予測・計画を1つのネットワークで貫通させる潮流。BEV グリッドが現代の自動運転スタックの背骨になっています。
