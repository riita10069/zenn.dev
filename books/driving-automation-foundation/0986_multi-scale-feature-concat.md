# マルチスケール特徴の連結

## ひとことで言うと
カメラ画像を画像認識ネットワーク(バックボーン)に通すと、粗いものから細かいものまで複数の解像度の「特徴マップ」が同時に取り出せます。それらをまとめて1つの大きな特徴として下流に渡す設計が「マルチスケール特徴の連結(multi-scale feature concatenation)」です。階層型のバックボーン(たとえば Swin Transformer Tiny)では4段階の特徴が出力され、各段のチャンネル数を空間サイズをそろえてからチャンネル方向に貼り合わせると、96+192+384+768=1440 チャンネルの統合特徴になります。これにより「細部の手がかり」と「意味的なまとまり」を1つのテンソルに同居させ、遠くの小さい物体と近くの大きい物体を同時に扱える土台が作れます。

## 直感的な理解
人間が運転するとき、視覚情報は2種類の「見え方」を同時に使っています。1つは細部です。「あの遠くに小さく動く点は歩行者の頭か、それとも電柱か」を見分けるには、画像の細かいテクスチャやエッジが要ります。もう1つは大局です。「目の前の大きな白い塊はトラックだ」と判断するには、細部より「形のまとまり・意味」が要ります。

ニューラルネットワークも同じです。畳み込みや段階的なダウンサンプリングを重ねるネットワークは、浅い層では高解像度で局所的な情報(エッジ、色、テクスチャ)を、深い層では低解像度だが意味的にまとまった情報(これは車だ、これは道路だ)を持ちます。もし深い層だけを使うと、遠くの小さな歩行者の手がかりは縮小の過程で消えてしまいます。逆に浅い層だけを使うと「これが何か」という意味理解が弱くなります。両方が欲しい。だから全段の特徴をまとめて下流に渡す、というのがマルチスケール特徴連結の動機です。自動運転のように「数十メートル先の小さな歩行者」と「目前の大型車」を同時に扱うタスクでは、この両立が安全性に直結します。

物体が画像に占める大きさと、それが鮮明に現れる段の対応関係を直感で押さえておくと理解が早いです。小さい物体は浅い段(高解像度)に、大きい物体は深い段(低解像度だが受容野が広い)に強く現れます。受容野(receptive field)とは「出力の1画素が、入力画像のどれだけ広い範囲を見ているか」のことで、深い段ほど広くなります。広い受容野は大きな物体の全体像を捉えるのに向き、狭い受容野は小さな物体の細部を捉えるのに向きます。1つの段だけでは全サイズに対応できない、という根本がここにあります。

## 基礎: 前提となる概念
いくつかの用語を、知らない前提でていねいに説明します。

- バックボーン(backbone): 画像から「特徴」を取り出す土台のネットワークです。ResNet、ConvNeXt、Swin Transformer などがあります。「土台」というのは、その上に物体検出ヘッドや経路予測ヘッドなど「目的別の部品」を載せて使うからです。

- 特徴マップ(feature map): 画像を畳み込みなどで処理した結果の3次元配列で、形は [チャンネル数 C, 高さ H, 幅 W] です。チャンネルとは「特徴の種類の数」です。たとえば「縦エッジに強く反応する検出器」「赤い領域に反応する検出器」のようなフィルタが C 個並んでいて、それぞれが H×W の反応マップを出している、というイメージです。C が大きいほど多種類の手がかりを表現できます。

- スケール(scale)/解像度の段階(stage): 階層型バックボーンは画像を段階的に縮小しながら処理します。浅い段(stage 0)は高解像度で細かく、深い段(stage 3)は低解像度だが意味的という性質があります。深い段ほどチャンネル数を増やして「意味の表現力」を補うのが定石です。各段で空間が半分・チャンネルが倍、という増減が階層型バックボーンの典型パターンです。

- 総ストライド(total stride): 入力に対して最終的にどれだけ縮小されたかの倍率です。総ストライド32なら、最深段は入力の1/32の解像度です。256入力なら最深段は8×8になります。

- プーリング(pooling): 小さな窓(たとえば2×2)の中の値を1つの値にまとめて、特徴マップを縮小する操作です。窓の中の最大値を残すのが Max プーリング、平均を取るのが Average プーリングです。Max は「最も強く反応した手がかり」を残すので、エッジや角のような目立つ特徴を保持しやすい性質があります。

- 連結(concatenation): 複数のテンソルを、ある軸に沿って並べてくっつける操作です。チャンネル方向(dim=1)で連結すると、空間サイズ H×W は同じまま、チャンネル数 C が足し算されます。情報を「混ぜる(足し合わせる)」のではなく「並べて全部残す」のが連結の特徴です。

## 仕組みを詳しく
処理は大きく3ステップです。複数カメラを扱う場合、バッチサイズ B とカメラ台数 V をまとめて B*V を1次元目に置く、という置き方をよくします(複数カメラを1枚ずつ独立にバックボーンへ通すため)。

ステップ1: バックボーンから複数段の特徴マップを受け取る。階層型バックボーンに 256×256 の画像を入れると、4段の特徴はおおよそ次の形になります。

```
stage 0: [B*V,  96, 64, 64]
stage 1: [B*V, 192, 32, 32]
stage 2: [B*V, 384, 16, 16]
stage 3: [B*V, 768,  8,  8]
```

ここで重要なのは、段ごとに空間サイズ(H, W)が違うことです。64×64 と 8×8 はそのままでは貼り合わせられません。チャンネル方向に連結するには、全段の H×W が一致している必要があります。

ステップ2: 空間サイズをそろえる。`AdaptiveMaxPool2d(target_size)` などで全段を同じ解像度(よく使われるのは最深段に合わせた 8×8)に落とします。「Adaptive(適応的)」というのは、出力サイズを指定すると窓サイズを自動計算して、64×64 でも 16×16 でも同じ 8×8 に縮めてくれる、という意味です。プール後はこうなります。

```
stage 0: [B*V,  96, 8, 8]
stage 1: [B*V, 192, 8, 8]
stage 2: [B*V, 384, 8, 8]
stage 3: [B*V, 768, 8, 8]
```

ステップ3: チャンネル方向に連結する。`torch.cat(features, dim=1)` で4段を積み重ねます。チャンネル数が 96+192+384+768 = 1440 に合算されます。

```
連結後: [B*V, 1440, 8, 8]
```

これで「1枚のカメラ画像の、全スケールの情報を 1440 チャンネルにまとめた特徴」ができます。多くの設計では、この直後に 1×1 畳み込み(Conv1×1)で次元を圧縮します。たとえば 1440→256 とすれば、下流のモジュール(複数視点の融合、運転方策、未来予測など)が固定次元で書けるようになります(次元統一の意義は [backbone特徴サイズの抽象化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0919_backbone-feature-size-abstraction) と [埋め込み次元と注意機構の計算コスト](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0914_attention-cost-embed-dim) を参照)。

設計選択を「なぜそうするのか」まで踏み込んで見ます。

なぜ「足し算」ではなく「連結」なのか。足し算(要素ごとの加算、element-wise sum)はチャンネル数が変わらず軽い反面、足し合わせる前に全段を同じチャンネル数へそろえる必要があり、さらに異なる段の情報を強制的に同じ次元へ重ね合わせてしまいます。連結は全段の情報をそのまま並べて残すので、後続の層が「どの段のどのチャンネルをどう使うか」を重みの学習で決められます。表現力は連結が上、メモリと後続計算は足し算が軽い、というトレードオフです。FPN 系が「足し算」を選ぶのは、段ごとに 256 チャンネルへそろえる射影を先に挟んで次元を一致させているからで、生のチャンネル数のまま足すわけではない点に注意してください。

なぜプール先を最深段(8×8)に合わせるのか。最も粗い段にそろえると連結後の H×W が最小(8×8=64画素)になり、下流のトークン数が少なく済みます。トークン数が少ないほど、後段の注意機構の N² コスト([埋め込み次元と注意機構の計算コスト](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0914_attention-cost-embed-dim) 参照)が軽くなります。逆に浅い段(64×64)へそろえると、深い段を拡大(アップサンプル)することになり、情報を増やせないのに計算だけ増えます。だから「最深段に合わせて落とす」のが計算効率の定石です。ただし後述するように、これは浅い段の高解像度情報を大量に捨てる副作用を持ちます。

なぜ Max プーリングなのか。経路予測や検出では「最も強い手がかり」を保持したい場面が多く、平均を取ると弱い背景に薄められがちです。8×8 という小さな窓へ 64×64 を一気に落とすとき、Average だと小さな物体の弱い反応が周囲の背景に埋もれます。Max は窓内で最も強い反応を1つ残すので、まばらな手がかりを拾いやすい。一方で Max はノイズの強い反応も拾うため、Average との使い分けは設計判断になります。

数値で「連結が何を捨て、何を残すか」を見ます。stage0 は 96ch × 64×64 = 約39万要素の情報を持ちますが、8×8 へ Max プールすると 96ch × 64 = 6144 要素まで圧縮されます。約64分の1への削減です。つまり連結後の1440チャンネルのうち、浅い段由来の成分は「64×64の広い空間を8×8に潰した残骸」であり、細部の位置情報の多くは失われています。これがマルチスケール連結の根本的な限界で、後述の FPN 系がトップダウン経路で「高解像度を維持したまま意味を注入する」方向へ進化した理由です。

## 手法の系譜と主要論文
マルチスケール特徴の統合は、物体検出・セグメンテーションで長く磨かれてきた基本技術です。系譜をたどります。

- Lin et al., "Feature Pyramid Networks for Object Detection (FPN)", CVPR 2017(arXiv:1612.03144)。FPN は「浅い段(高解像度・細部)」と「深い段(低解像度・高意味)」を、深い段を2倍に拡大して浅い段に足し合わせる「トップダウン経路」でつなぎました。動機は、小さい物体は浅い段に、大きい物体は深い段に強く現れるため、両方を1つのピラミッドに統合すれば1つの検出器で全サイズの物体を扱える、という点です。単純な連結(全段を1つに潰す)より一歩進んで「段の間で情報を伝播させ、各段を高解像度のまま意味的に強化する」設計です。各段は連結せず、それぞれが独立した予測層を持つのが特徴で、ここが「全段を1テンソルに連結する」素朴な方式との分岐点です。

- Liu et al., "Swin Transformer: Hierarchical Vision Transformer using Shifted Windows", ICCV 2021(arXiv:2103.14030)。CNN のように階層的に解像度を落とす Transformer で、4段の特徴を出力します。各段でチャンネルが倍々(96→192→384→768)に増える設計で、1440 という数はこの合計に由来します。動機は、Transformer の全画素間注意が画素数の2乗で爆発するため、窓に区切って局所注意にし、段を下げて画素数を減らすことで現実的なコストにすることです。「CNN と同じ4段の階層インタフェース」を Transformer に持ち込んだことが、後の差し替え可能性([backbone特徴サイズの抽象化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0919_backbone-feature-size-abstraction))を支えました。

- Tan et al., "EfficientDet: Scalable and Efficient Object Detection (BiFPN)", CVPR 2020(arXiv:1911.09070)。BiFPN は FPN を双方向(トップダウン+ボトムアップ)にし、さらに段ごとに学習可能なスカラ重みを付けて統合します。重みは正規化された非負係数で、「どの段をどれだけ信用するか」を学習で決められる点が単純連結や均等な足し算より優れます。同じ BiFPN ブロックを繰り返し積めるよう設計されており、モデルサイズに応じて繰り返し回数を変えるスケーリング戦略と組み合わせています。代償は構造の複雑さと、繰り返しによる計算増です。

- Ghiasi et al., "NAS-FPN: Learning Scalable Feature Pyramid Architecture", CVPR 2019(arXiv:1904.07392)。FPN の段間接続トポロジー(どの段からどの段へ情報を流すか)自体を Neural Architecture Search(NAS、ネットワーク構造の自動探索)で探索し、人手設計の FPN を上回る不規則な接続を発見しました。「最適な統合構造は人間の直感より複雑でありうる」ことを示した点が重要です。

- Liu et al., "A ConvNet for the 2020s (ConvNeXt)", CVPR 2022(arXiv:2201.03545)。CNN を Transformer 風の設計指針で現代化したもので、Swin と同じく 96/192/384/768 の4段階層を持ちます。これにより「同じ連結インタフェース」で CNN と Transformer を差し替えられる土壌ができました。

系譜としては、「単純連結(全段を潰して1つに)」→「FPN のトップダウン統合(各段を高解像度のまま強化)」→「BiFPN の双方向・重み付き統合」→「NAS による接続探索」と複雑化が進みました。一方で「とにかく全段をプールして連結する」軽量な選択も、計算予算が厳しい組み込み用途や、下流が固定次元トークンを要求する場面(Transformer 融合の前段)では現役で使われ続けています。どちらが優れるかは下流の構造とコスト予算に依存し、一律の正解はありません。

## 論文の実験結果(定量データ)
指標の意味から説明します。物体検出では mAP(mean Average Precision、平均適合率の平均)がよく使われます。これは「検出した枠がどれだけ正しく、見落としが少ないか」を 0〜100 の数値で表したもので、大きいほど良い指標です。適合率(precision)は「検出した枠のうち本当に物体だった割合」、再現率(recall)は「実在する物体のうち検出できた割合」で、その兼ね合いを物体サイズ別に平均したのが mAP です。とくに AP_S(small、小物体に対する AP)、AP_M(medium)、AP_L(large)はマルチスケール設計の効果がサイズ別に切り分けて見える指標です。

- FPN(CVPR 2017)は COCO データセットで、ResNet ベースの Faster R-CNN にマルチスケール統合を加えることで全体 mAP を大きく改善しました。論文では、単一の最深段特徴だけを使う構成に比べ、トップダウン統合を入れると mAP がおよそ数ポイント向上し、とりわけ AP_S(小物体)の伸びが顕著だったと報告されています。具体的には、FPN を入れた Faster R-CNN で COCO の AP がおよそ 36 前後に達し、ベースラインに対し小物体での改善が大きいことが示されました。これは「浅い段の高解像度情報を捨てないことが小物体検出に効く」ことの直接的な証拠です。アブレーション(構成要素を抜く実験)では、トップダウン経路を外す、横方向接続(lateral connection)を外すと小物体の AP が落ちる形で、各要素の寄与が確認されています。

- EfficientDet/BiFPN(CVPR 2020)は COCO で、当時の検出器と同等以上の mAP をはるかに少ない計算量(FLOPs)で達成したと報告しています。最大構成 EfficientDet-D7 で COCO mAP 53 級を、当時の同精度モデルより一桁少ない FLOPs で出しました。アブレーションでは、特徴統合を「素朴な FPN → 双方向 FPN → 重み付き双方向 FPN(BiFPN)」と進めるごとに mAP が段階的に上がり、「段ごとの学習可能重み」を外すと精度が下がる、という形でその寄与が示されています。重みを外すと精度が下がる、というのがアブレーションの読み方です。

- Swin Transformer(ICCV 2021)は、階層型の特徴を物体検出(COCO)やセグメンテーション(ADE20K)の下流に渡すことで、当時の最先端を更新しました。COCO 物体検出で box AP 58 級、ADE20K セグメンテーションで mIoU(平均 Intersection-over-Union、領域の重なり率)53 級を報告しています。ここでの含意は、「4段の階層特徴をそのまま下流の検出・分割ヘッドに渡せる」設計が、非階層の ViT(単一スケールしか出さない)より密な予測タスクで有利だという点です。

定量データの読み方として大事なのは、「小物体の指標(AP_S)が伸びるかどうか」です。マルチスケール統合の価値は平均 mAP より小物体で顕著に出るため、自動運転のように遠方小物体が安全に直結するタスクでは、この差が実利になります。逆に言えば、全段を最深段へ潰す素朴な連結は小物体情報を犠牲にしやすく、AP_S での弱さがその代償として現れます。

## メリット・トレードオフ・限界
メリット
- 細部(浅い段)と意味(深い段)を両方下流に渡せる。小物体と大物体を同時に扱いやすい。
- 単純連結は実装が軽い。プールと連結だけで済み、追加の畳み込み経路やトップダウン配線が要らない。
- 連結結果が固定の H×W・チャンネル数の単一テンソルになるため、Transformer 融合のように「固定長トークン列」を要求する下流へ素直に接続できる。

トレードオフ・限界
- 単純連結はスケール間の重み付けを学習しない。「どの段を重視するか」を調整できる FPN/BiFPN に対し、表現の柔軟性で劣る。後段の 1×1 畳み込みである程度の重み付けは学べるが、段の空間構造をまたぐ伝播は起きない。
- 連結チャンネル数がバックボーン依存になる(階層型 Tiny で 1440、ResNet50 で 3840 など)。下流次元がバックボーンに縛られるので、1×1 畳み込みで固定次元へ射影する対処が事実上必須になる([backbone特徴サイズの抽象化](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0919_backbone-feature-size-abstraction))。
- Max プーリングは外れ値に強く反応するため、ノイズの強い手がかりを拾うことがある。Average との使い分けや、学習可能なプーリングの導入が検討課題。
- 「最深段に合わせて 8×8 まで落とす」と、浅い段の高解像度情報の大半(前述のとおり stage0 で約64分の1)がプールで捨てられる。小物体に効くはずの細部を、プール段で失う本末転倒が起きうる。プール先解像度の選び方は精度を左右する重要な設計パラメータで、安易に最小解像度へ落とすと AP_S を犠牲にする。

未解決の課題として、「どの段をどう統合すべきか」をタスク依存で自動最適化する方向(NAS による FPN 構造探索など)があり、固定の連結が常に最適とは限りません。また、画像段で連結してから BEV へ射影するか、BEV へ射影してから多スケール統合するか、という統合の「順序」も未解決の設計問題です。

## 発展トピック・研究の最前線
- 学習可能な特徴ピラミッド: NAS-FPN(Ghiasi et al., CVPR 2019)は段の接続トポロジー自体を探索し、人手設計の FPN を上回る構造を見つけました。AutoML 系の流れで、統合構造そのものを最適化対象にする方向です。
- Transformer での多スケール統合: Deformable DETR(Zhu et al., ICLR 2021, arXiv:2010.04159)は、複数スケールの特徴に対して「変形可能な注意(deformable attention)」を使い、全画素を見ずに少数の参照点だけを参照することで多スケールの計算を現実的にしました。素朴な多スケール自己注意は画素数の2乗で爆発するため、参照点を絞ることでスケールをまたいだ注意を実用域に収めた点が新規性です。
- BEV(鳥瞰図、Bird's-Eye-View)表現への射影: 自動運転では、多スケールの画像特徴を BEV 平面に投影してから統合する流れが主流化しつつあり、画像段での連結は前処理として位置づけ直されています。BEVFormer などは多スケール画像特徴を空間クロスアテンションで BEV グリッドに集約します。
- 学習可能なプーリング・トークン削減: 固定の Max/Average プールではなく、注意で重要トークンを選ぶ・統合する手法(token merging など)が、情報損失を抑えつつトークン数を減らす方向として研究されています。
