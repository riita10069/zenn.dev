# ConcatViewFusion（チャネル連結融合）

## ひとことで言うと

複数カメラから得た特徴を統合する一番素朴なやり方です。各カメラの特徴を「チャネル（特徴の種類）方向」に単純に積み重ねて1つの分厚いテンソルにし、1×1の畳み込みで元のチャネル数へ圧縮します。賢い仕組みは何もありませんが、軽くて速く、より高度な融合手法と比べるときの「基準（ベースライン）」として広く使われます。

## 直感的な理解

7人の証人がそれぞれ別の角度から同じ事件を目撃したとします。捜査官（モデル）が全体像を1つにまとめるには、7人の証言を束ねて突き合わせる必要があります。最も単純なやり方は「7人ぶんの証言を全部1枚の紙に縦に書き写し、それを読んで総合判断する」ことです。証言の重み付け（誰の証言をどれだけ信じるか）は、読み手が経験で決めます。

ConcatViewFusion はまさにこれです。各カメラの特徴を縦に（チャネル方向に）並べて1つの分厚い束にし、その束を1×1 畳み込みという「読み手」に通します。読み手の重みは学習で決まり、「どのカメラのどの特徴をどれだけ混ぜるか」を獲得します。幾何（カメラがどの方向を向いているか）も時系列も一切使わない、最小構成の融合です。だからこそ、より複雑な手法が「本当にその複雑さに見合う価値があるのか」を測る物差しになります。

## 基礎: 前提となる概念

用語を噛み砕きます。

- view（ビュー）。カメラ1台ぶんの視点です。V=7 なら前方・側方・後方など7台ぶんを指します。

- 融合（fusion）。複数の view の特徴を1つにまとめることです。なぜ必要かというと、後段の経路計画（どこへ進むか）は「全カメラを総合した1つの状況認識」を必要とするからです。7台ぶんの特徴をバラバラに持っていても、最終的な行動は1つに決めなければなりません。

- チャネル（channel）。特徴の種類の本数です。各空間位置に C 本の数値が並んでおり、それぞれが別々の特徴（縦エッジ、赤色反応など）に対応します。

- 1×1 Conv（1×1 畳み込み）。各ピクセル位置のチャネル方向だけを線形に混ぜる操作です。空間的な広がりは見ず、「同じ位置の C 本の数値を別の本数へ変換する」のに使います。チャネル数を増減したり、チャネル間を線形結合したりするのに便利です。Lin et al.（2014）の Network in Network で提示されて以降、あらゆるアーキテクチャで使われています。

- ベースライン（baseline）。比較の基準となる単純な手法です。研究では、いきなり高度な手法から入ると「精度が出たのは手法が賢いからか、単に情報量が増えたからか」を切り分けられません。だから最も単純な統合を1つ用意し、これを下限の基準にします。ConcatViewFusion がその役割です。

## 仕組みを詳しく

典型的な実装は次のとおりです。

```python
class ConcatViewFusion(nn.Module):
    def __init__(self, num_views, embed_dim=256):
        super().__init__()
        self.view_reduce = nn.Sequential(
            nn.Conv2d(num_views * embed_dim, embed_dim, kernel_size=1),
            nn.GELU(),
        )

    def forward(self, fused_per_view, B, V, camera_params=None):
        C, H, W = fused_per_view.shape[1:]
        x = fused_per_view.reshape(B, V, C, H, W)   # view 軸を復元
        x = x.reshape(B, V * C, H, W)               # view をチャネルに畳む
        return self.view_reduce(x)                  # V*C → C に圧縮
```

入力 `fused_per_view` は、前段のマルチスケール連結（[マルチスケール特徴のプーリング連結](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0987_multi-scale-feature-fusion)）を経た `[B*V, C, H, W]` です。ここで C=embed_dim=256、H=W=格子サイズ（例 8）とします。B=8, V=7 の例で追います。

1. view 軸の復元。`reshape(B, V, C, H, W)` で `[56, 256, 8, 8]` を `[8, 7, 256, 8, 8]` へ戻します（[backbone処理のための B*V reshape](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0918_backbone-batch-view-reshape) の復元と同じ）。これで「どの行がどのカメラか」が軸として明示されます。

2. view をチャネルに畳む。`reshape(B, V*C, H, W)` で 7台 × 256ch を `7*256 = 1792` チャネルへ積み重ねます。形は `[8, 1792, 8, 8]`。

   - ここがこの手法の本質です。各空間位置 (h, w) に、7台ぶんの 256次元の特徴がただ縦に連結されます。3台目のカメラの特徴は 512〜767 番目のチャネルに、というように「カメラ順」で並びます。

3. 1×1 Conv で圧縮。`nn.Conv2d(V*C, C, 1)` で 1792ch を 256ch へ戻します。1×1 なので各位置のチャネル方向だけを線形結合し、続く GELU で非線形性を加えます。形は `[8, 256, 8, 8]`。

   - この 1×1 Conv の重み（形は `[256, 1792, 1, 1]`）が「どのカメラのどの特徴をどれだけ混ぜるか」を学習で決めます。つまり融合のロジックはこの1層に押し込まれています。

出力 `[8, 256, 8, 8]` は「7台を統合した1つの特徴マップ」で、後段（地図との融合、時系列メモリ、経路予測）へ渡されます。

なぜ「チャネル方向に積む」のか。空間的に貼り合わせる（横に並べる）方法もありますが、それだとカメラ間の重なりや視野の不連続を扱えません。チャネルに積めば「同じ格子位置に全カメラの情報が同居」し、1×1 Conv が位置ごとに全カメラを見て混ぜられます。

重要な性質（view 順依存）。`V*C` に畳むとき、カメラの並び順がチャネルの並び順に固定されます。1×1 Conv の重みは「1番目のカメラはこのチャネル」という前提で学習されるため、カメラの順番を入れ替えると意味が壊れます。これは [CrossAttentionViewFusion（カメラ間アテンション融合）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0952_cross-camera-attention-fusion) が順序不変（アテンションは集合に対する演算なので順番に依存しない）なのと対照的な弱点です。

## 手法の系譜と主要論文

特徴をチャネル方向に連結して 1×1 Conv で混ぜる、というのは深層学習で最も基本的な融合操作で、多くの論文がベースラインや構成要素として用います。

- Lin et al. "Network in Network"（arXiv:1312.4400, 2014）。1×1 畳み込みによるチャネル間の線形結合と次元削減を導入しました。以降、チャネル数調整とチャネル混合の標準部品になりました。本手法の最後の1×1 Conv はこの直系です。

- Szegedy et al. "GoogLeNet / Inception"（CVPR 2015, arXiv:1409.4842）。異なるカーネルサイズの枝の出力をチャネル方向に concat して次段へ渡す Inception モジュールを導入しました。「並列に得た複数の特徴をチャネル連結で束ねる」発想の代表例です。

- Ronneberger et al. "U-Net"（MICCAI 2015, arXiv:1505.04597）。エンコーダの特徴をデコーダ側にスキップ接続でチャネル連結します。「異なる情報源をチャネル連結して後続層に混ぜさせる」典型で、本手法のマルチビュー版と動機が近いです。

マルチカメラ融合の文脈では、concat はしばしば「単純すぎるが頑健なベースライン」として登場します。BEVFormer（Li et al., ECCV 2022, arXiv:2203.17270）や PETR のように幾何やアテンションを使う手法の論文でも、比較対象として単純連結系が引かれることがあります。提案手法が concat に対してどれだけ上回るかで、幾何・アテンションの価値を測る、という使われ方です。系譜としては「素朴な連結 → カメラ間アテンション（順序不変・適応的重み）→ 幾何を使う BEV 変換」という改良の流れがあり、concat はその出発点に位置します。

## 論文の実験結果(定量データ)

concat 自体を主役にした論文は少ないですが、ベースラインとしての挙動はよく報告されます。一般に、幾何やアテンションを使う融合手法は、単純連結ベースラインに対して nuScenes（6カメラのマルチビュー自動運転ベンチマーク）の NDS や mAP で数ポイント以上の改善を示します。NDS（nuScenes Detection Score）は位置・大きさ・向き・速度・属性の誤差を総合した0〜1の指標、mAP（mean Average Precision）は検出の平均精度で、いずれも高いほど良い指標です。BEVFormer は nuScenes test で NDS 約0.569 を報告しましたが、こうした数値は「3D空間で整合の取れた表現（BEV）」や「カメラ間アテンション」を入れたことの寄与を含みます。concat は幾何を使わないため、視野が重なる領域や3D位置の整合が要る課題で不利になり、ここで差がつくと解釈されます。

アブレーションの観点では、融合手法を concat に固定して他要素（時系列、マルチスケール、解像度）を変える対照実験により、「精度向上が融合の賢さによるのか、単なる情報量増加によるのか」を切り分けられます。concat がベースラインとして優秀なのは、計算が軽く（reshape 2回と 1×1 Conv 1層のみ）、再現が容易で、他要素の効果を観測する土台として安定しているからです。逆に言えば、concat 単独では「どのカメラを重視すべきか」を空間位置ごとに動的に変える適応ができず、その限界が高度手法との差として数値に表れます。

## メリット・トレードオフ・限界

メリット。とにかく単純で軽い。reshape 2回と 1×1 Conv 1層だけで、計算量・メモリともに小さい。アテンションのような全位置間の相互作用がないため推論が速い。比較の基準として優秀で、これを下限により高度な手法の上積みを測れます。全カメラの情報を同じ位置に同居させ、1×1 Conv が学習で混ぜ方を決められる点も利点です。

トレードオフと限界。第一に幾何を使いません。カメラの向きや視野の重なりを考慮しないため、3D 的に整合した統合にはなりません。第二に view 順に依存します。カメラの並び順がチャネル順に固定され、順番を変えると壊れます（順序不変でない）。第三にカメラ台数 V が固定です。`V*embed_dim` が 1×1 Conv の入力チャネルとして焼き込まれるため、台数の異なるデータセットへ柔軟に対応しにくい。第四に「どのカメラを重視すべきか」を空間位置ごとに動的に変える適応ができません（重みは学習後固定）。

研究上の含意として、concat の弱点（順序依存・台数固定・幾何非対応・非適応）は、それぞれカメラ間アテンション（順序不変・適応的）、BEV 変換（幾何を明示的に使う）、カメラ ID 埋め込み（台数や順序への頑健性）という後続手法の動機になっています。concat はその意味で「何が足りないか」を明確にするための原点として価値を持ち続けます。

## 発展トピック・研究の最前線

concat の限界を埋める方向は大きく2つあります。1つはアテンション化で、各カメラ特徴をトークンとして扱い、self-attention や cross-attention で順序不変かつ適応的に統合する流れです（[CrossAttentionViewFusion（カメラ間アテンション融合）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0952_cross-camera-attention-fusion)）。これにより「重要なカメラに動的に注目する」ことが可能になります。もう1つは幾何の導入で、各カメラのキャリブレーション（内部・外部パラメータ）を使って特徴を共通の3D空間や BEV へ投影してから統合する流れです。LSS 系の深度推定による splat や、BEVFormer 系の deformable cross-attention による BEV クエリ集約が代表です。最近は、concat のような軽量融合を浅い段で行い、アテンションや BEV を深い段で行うハイブリッドや、concat 出力をアテンションの初期値に使う段階的融合も研究されています。軽さと表現力のトレードオフをどこで折り合わせるかは、車載のリアルタイム制約も絡んで今なお実務的な研究テーマです。
