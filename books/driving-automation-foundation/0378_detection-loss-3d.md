# 3D物体検出損失

## ひとことで言うと

カメラ画像(やセンサー)から「どこに、何が(車・歩行者・自転車など)、どんな大きさと向きで存在するか」を3次元の箱(バウンディングボックス)として当てさせ、その当たり外れを測る損失です。自動運転モデルを「画像から直接ハンドル操作を出す」だけにせず、途中で「周りの物体を検出する」という人間にも分かる中間タスクを解かせることで、学習を安定させ、表現を意味のあるものにし、説明性を高めます。エンドツーエンド運転における検出損失は、最終目的(計画)を支える「足場」として働く代表的な補助損失です。

## 直感的な理解

入力(画像)と出力(運転操作)の間に巨大なネットワークを一枚置いて端から端まで学習する方式を End-to-End(エンドツーエンド、E2E)と呼びます。理屈の上では「画像を入れれば操作が出る」だけで済みますが、実際にこれだけで学習すると二つの困りごとが起きます。一つは、ネットワークが何を根拠に操作を決めたのか人間に分からない(ブラックボックス)こと。これは事故が起きたときに原因を切り分けられない致命的な問題です。もう一つは、運転操作という最終目的だけからの勾配は弱く遠回りで(運転データの大半は「直進」という退屈なラベルで、勾配の情報量が少ない)、内部の表現が育ちにくく学習が不安定になることです。

そこで「途中の階で、人間にも検証できる物体検出という仕事を解かせる」のが3D物体検出損失です。これは補助タスク(auxiliary task、本来の目的を助ける別課題)の代表例です。物体検出をきちんと解けるネットワークは、内部に「周りに何があるか」という意味のある中間表現を必ず持つことになります。その上に乗る計画(プランニング)は、その表現を再利用できるので賢くなります。検出という分かりやすい正解(箱のラベル)で内部を直接添削できるので、学習も速く安定します。

なぜ「3D」かというと、運転には地面に沿った左右位置だけでなく、奥行き・高さ・向きが要るからです。「前方20メートルに、長さ4.5メートルの車が、こちらに30度傾いて止まっている」という3次元情報がないと安全な軌道は引けません。画像上の平面の箱(2D検出)では、その車が車線に出てくるのか出ないのか、自車の進路と交差するのかを判断できません。

## 基礎: 前提となる概念

バウンディングボックス(bounding box、境界箱)は物体を囲む箱です。2Dなら画像上の長方形(中心 x, y と幅・高さ)、3Dなら空間中の直方体で、中心 (x, y, z)、寸法(幅 w・長さ l・高さ h)、向き(ヨー角 yaw、真上から見た回転)を持ちます。運転では加えて速度 (vx, vy) を出すこともあります。つまり1物体あたり9〜11個の連続値とクラスを当てます。

損失(loss)は「予測がどれだけ間違っているか」を数値化した関数で、これを小さくする方向にネットワークの重みを更新します。重みを更新する手続きを逆伝播(backpropagation)と呼びます。

アンカー(anchor)は「あらかじめ用意した検出候補の雛形(典型的な箱の位置・サイズ)」で、ネットワークはアンカーからのズレを予測します。アンカーを使わず各画素から直接箱を予測する方式をアンカーフリー(anchor-free)と呼びます。3Dではアンカーの設計(向き・サイズの組合せ)が爆発しやすいため、アンカーフリーが好まれます。

クエリ(query)は Transformer 系の検出器で使う「学習可能なベクトル」で、各クエリが一つの物体を担当します。Transformer の注意機構(attention)は、あるベクトル(クエリ)が入力(キー・バリュー)のどこに注目すべきかを学ぶ仕組みです。

IoU(Intersection over Union)は二つの箱の重なり具合(重なり体積÷和集合体積)で、0〜1で1なら完全一致。検出の当たり判定や評価に使います。nuScenes は IoU の代わりに中心間距離を当たり判定に使います。

## 仕組みを詳しく

3D物体検出損失は、おおむね二つの部品の足し算です。

分類損失(classification loss)。検出した箱が「車」か「歩行者」か「背景(何もない)」かを当てます。クラスが外れているほど大きくなる損失で、しばしば Focal Loss(Lin et al. ICCV 2017)が使われます。Focal Loss は、検出で大量に発生する「簡単な背景」の影響を弱め、難しい前景に学習を集中させる工夫です。標準の交差エントロピー CE = -log(p_t) に対し、Focal Loss は (1-p_t)^γ という重みを掛け、すでによく当たっている例(p_t が1に近い)の損失をほぼ0に押し下げます。γ(ガンマ、典型的に2)は集中の強さを決め、α(典型的に0.25)で前景・背景のバランスを取ります。検出では候補が数万・正例が数十という極端な不均衡があるため、この工夫がないと損失が背景に飲まれて学習が進みません。

ボックス回帰損失(box regression loss)。箱の中心 (x, y, z)、寸法 (w, l, h)、向き yaw を数値として当てます。予測値と正解値の差を L1 損失(絶対値の差)や Smooth L1(誤差が小さいと二乗、大きいと絶対値に切り替え、外れ値に頑健)で測ります。向きは周期性(0と2πが同じ)があるので、yaw を sin/cos に分けて回帰するか、bin 分類+残差にするなどの工夫をします。寸法は対数空間で回帰すると大小の物体を均等に扱えます。

数値例で示します。あるクエリについて、

- 正解: クラス=車, 中心=(20.0, 1.5, 0.0)m, サイズ=(1.9, 4.5, 1.6)m, yaw=0.5rad
- 予測: クラス=車(正解), 中心=(18.0, 1.5, 0.0)m, サイズ=(1.8, 4.6, 1.6)m, yaw=0.6rad

クラスは当たっているので分類損失は小さく、中心 x が2メートルずれているので回帰損失が立ち、yaw も 0.1rad ずれで少し損失が出ます。これらを足して逆伝播します。中心 x の2メートルずれは、20メートル先という距離では深度推定の誤差に由来することが多く、ここで [深度監督損失 (Depth Supervision)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0377_depth-supervision-loss) が効いてきます。

検出の枠組みには大きく二系統あります。

アンカーフリー/中心ベース(FCOS3D 系)。画像の特徴マップ各点から直接「ここに物体があり、距離・寸法・向きはこれ」と回帰します。出力は特徴マップ各点に「クラス確率 + 3Dボックス属性」を持つ密な形:

```
出力: [バッチ B, 属性数 A, 特徴マップ高さ H, 特徴マップ幅 W]
（各 (h,w) 位置が一つの候補。Aにクラス・中心オフセット・深度・寸法・yaw が並ぶ）
```

正解との対応は、各正解箱の中心付近の特徴位置を「正例」とする中心性(centerness)ベースの割り当てで決めます。中心から離れた予測ほど信頼度を下げる centerness 分岐を持ち、これで低品質な周辺予測を抑えます。

クエリベース(DETR 系、DETR3D, PETR, UniAD)。学習可能な物体クエリを N 個(例: 900個)用意し、Transformer の注意機構で画像特徴と対話させ、各クエリが一つの物体を出します。出力は疎な形:

```
出力: [バッチ B, クエリ数 N=900, 属性数 A]
```

ここで肝になるのが、予測 N 個と正解 M 個をどう対応付けるかです。DETR 系はハンガリアンマッチング(Hungarian matching、分類スコアと箱の一致度から作るコスト行列を最小化する一対一の最適割り当て)を使い、各正解にちょうど一つの予測を割り当てます。残りの予測は「背景(no object)」を当てさせます。この一対一マッチングのおかげで、従来検出が必要とした NMS(Non-Maximum Suppression、重複検出を後処理で消す手続き)が不要になります。割り当てが決まったら、上記の分類+回帰損失を計算します。マッチングコストと損失はほぼ同じ項(分類コスト + L1 + IoU/GIoU)で構成されます。

```
画像 ─→ [特徴抽出] ─→ 特徴マップ
                          │
        ┌─────────────────┴──────────────────┐
   (FCOS3D系: 密)                        (DETR3D/UniAD系: 疎)
各位置で直接回帰 [B,A,H,W]            物体クエリ×N と注意機構 [B,N,A]
        │                                     │
   中心性で正例割り当て                  ハンガリアンマッチングで一対一割り当て
        └──────────── 正解と対応付け ─────────┘
                          │
            分類損失(Focal) + 回帰損失(L1/SmoothL1) = 3D検出損失
```

## 手法の系譜と主要論文

2D検出から3D・E2Eへ向かう流れを描きます。

Focal Loss / RetinaNet(Lin, Goyal, Girshick, He, Dollár, ICCV 2017, arXiv:1708.02002)。前景と背景の極端な不均衡を損失の形で解く一段検出を提案し、以後の検出損失設計の土台になりました。一段検出器が二段検出器(Faster R-CNN)に精度で追いつけなかった主因が「クラス不均衡」だと突き止めた点が本質的貢献です。

DETR(Carion, Massa, Synnaeve, Usunier, Kirillov, Zagoruyko, ECCV 2020, arXiv:2005.12872)。検出を「集合予測」と捉え、クエリ・Transformer・ハンガリアンマッチングで NMS とアンカーを不要にした転換点です。手作りの後処理を学習に取り込んだことで、3D版・自動運転版の直接の母体になりました。弱点は収束が遅い(500エポック級)ことで、後の Deformable DETR がこれを大きく改善しました。

FCOS3D(Wang, Zhu, Wang, Lin, Yang, Yang, Loy, Lin, ICCV 2021 Workshop, arXiv:2104.10956)。2D検出器 FCOS を3Dに拡張し、アンカーを使わず画像各点から直接3Dボックスを回帰する単眼(カメラ1個)3D検出を提案しました。深度は対数空間で回帰し、向き・速度も同時に出します。動機は「3D用のアンカー設計は煩雑で点ベースの方が素直」。nuScenes 単眼検出チャレンジで当時の高水準を達成しました。

DETR3D(Wang, Vasudevan, Voigtlaender, Sun, Caine, Wang, et al. CoRL 2021, arXiv:2110.06922)。クエリベースの多視点(マルチカメラ)3D検出の先駆けです。3Dの参照点(クエリが持つ3D座標)を各カメラ画像に投影して特徴を集め、画像をまたいで一貫した3D検出を行います。明示的な深度予測なしに 2D→3D を結ぶ点が新しく、UniAD の検出部の直接の系譜です。

PETR / BEVFormer(2022)。PETR(Liu et al. ECCV 2022)は3D位置エンコーディングを画像特徴に注入し、参照点の射影を不要にしました。BEVFormer(Li et al. ECCV 2022)は BEV クエリと時空間の deformable attention で多視点3D検出の精度を押し上げました。いずれもクエリベース検出損失をそのまま使います。

UniAD(Hu, Yang, Chen, Li, Deng, Tao, Sima, Loy, Wang, Li, CVPR 2023, Best Paper, arXiv:2212.10156)。検出・追跡・地図・動き予測・占有予測・計画を一つのネットワークに統合した E2E 自動運転の代表作です。検出と追跡はクエリベースの TrackFormer が担い、その物体クエリが後段の動き予測(MotionFormer)・計画へそのまま流れます。論文の主張は「各タスクを別々に学ぶより、最終目的の計画に向けて全タスクをクエリで繋いで一緒に学ぶ方が良い」。検出損失は、後段に有用な物体表現を作る土台として機能します。

## 論文の実験結果(定量データ)

主な評価は nuScenes(ヌーシーンズ)で行われます。指標を初心者向けに説明します。mAP(mean Average Precision)は3D検出ボックスの当たり具合の総合スコアで高いほど良く、nuScenes は中心距離 0.5/1/2/4m のしきい値で当たり判定して平均します。NDS(nuScenes Detection Score)は mAP に位置・サイズ・向き・速度・属性の誤差を加味した総合点で、0〜1、高いほど良い指標です。

FCOS3D は nuScenes 単眼3D検出チャレンジで上位の成績を収め、画像のみの単眼検出としては当時最高水準の mAP・NDS を達成したと報告されています。DETR3D は多視点に拡張することで単眼を上回り、NMS なしで競争力ある NDS(報告ではテストセットで NDS 0.47 前後)を示しました。BEVFormer はさらに NDS を 0.56 前後まで引き上げ、時系列利用の有効性を示しました。

UniAD の実験は「統合して計画まで一緒に学ぶ」効果を多面的に示します。検出・追跡の指標(AMOTA など追跡精度)で個別最適化版に匹敵しつつ、最終目的の計画指標が大きく改善した点が核心です。具体的には、計画の平均変位誤差(将来軌道の予測点と正解点の平均距離、L2 誤差、メートル単位で小さいほど良い)と、衝突率(自車予測軌道が周囲物体とぶつかる割合、低いほど良い)で、当時の E2E 手法(ST-P3 など)を上回ったと報告されています。論文では平均 L2 誤差・衝突率がともに低下し、特に衝突率が相対的に大幅(報告によると数割規模)に下がったとされ、これは「中間タスクの表現が計画の安全性に直結する」ことの定量的証拠です。

アブレーション(要素を抜く検証)が重要です。UniAD は、検出・追跡・地図・動き予測・占有といった中間モジュール(=それぞれの補助損失)を段階的に外していくと、最終の計画指標(L2 誤差・衝突率)が悪化することを示しました。つまり検出損失は「あってもなくてもよい飾り」ではなく、計画性能を支える土台だと裏づけられています。一方で、全部を素朴に同じ重みで足すと特定タスクに最適化が偏るため、UniAD は段階的学習(まず知覚=検出・追跡を学び、次に予測・計画を足す二段階訓練)という学習スケジュールの工夫も併用しています。これは複数損失の重み調整([不確実性重み付けマルチタスク損失 (Kendall)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0601_uncertainty-weighting-kendall))や勾配競合の問題と直結します。

## メリット・トレードオフ・限界

メリット。E2E モデルに「周りに何があるか」という解釈可能な中間表現を持たせ、性能と説明性を同時に上げます。検出は箱のラベルが整備された大規模データセットが多く、教師信号が豊富で学習しやすい。さらに後段の動き予測・計画タスクと自然につながり、UniAD 型のクエリ受け渡しが可能になります。検出が出す箱は人間がそのまま検証・デバッグでき、安全性の議論にも使えます。

トレードオフと限界。3Dボックスのアノテーション(人手のラベル付け)は2Dより格段に高コストで、LiDAR を見ながら向き・寸法まで付ける必要があります。単眼カメラだけでは深度が曖昧で、遠方・小物体・遮蔽物体の検出精度に原理的限界があります(ここで [深度監督損失 (Depth Supervision)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0377_depth-supervision-loss) が効いてきます)。検出損失の重みを上げすぎると、本来の運転計画よりも検出に最適化が偏る恐れがあり、複数損失の重み調整(マルチタスク重み付け)が要ります。クエリベースは学習が遅く収束に多くのエポックを要する、ハンガリアンマッチングの一対一割り当てが学習中に不安定(同じ正解に付くクエリがエポックごとに入れ替わる)で密集シーンで取りこぼしやすい、といった既知の弱点もあります。

未解決の課題としては、補助タスクと最終目的のあいだの勾配の競合(検出を上げる更新が計画を下げる等、[PCGrad (勾配射影)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0598_pcgrad_note) が扱う問題)、長距離・希少クラスでの検出精度、ラベルなしデータの活用(自己教師・蒸留)が活発に研究されています。

## 発展トピック・研究の最前線

E2E 自動運転の後続では、UniAD のモジュール積み上げをスリム化して計画に直結させる VAD(ベクトル化シーン表現、Jiang et al. ICCV 2023)や、占有予測を中核に据える OccNet 系、さらに大規模言語モデルを意思決定に絡める研究(DriveGPT 系)が現れています。検出損失そのものも、ハンガリアンマッチングの不安定さを和らげる一対多割り当て(Group DETR / H-DETR、DETR の収束を速める研究)、IoU を直接最適化する回帰損失(GIoU/DIoU の3D拡張、回転 IoU)、点群とカメラを融合した検出(BEVFusion などのセンサーフュージョン)へと枝分かれしています。検出を「明示的な箱」ではなく潜在クエリのまま計画へ渡す「箱を出さない E2E」も議論されており、補助タスクとしての検出損失をどこまで残すべきか(明示的な中間表現は安全だが性能上限を縛るかもしれない)は研究の論点です。
