# 融合特徴レベルでのビュー寄与テスト

## ひとことで言うと

マルチカメラのモデルで「すべてのカメラがちゃんと使われているか」を確かめるテストの設計の話です。最終出力(軌道)で確認すると、下流の別部品の都合で誤って失敗することがあります。確認する場所を「融合直後の中間特徴マップ」に移すと、検証したい性質(ビュー融合の正しさ)だけを正確に測れます。テスト対象の責任範囲ちょうどで観測する、というテスト設計の一般原則の実例です。

## 直感的な理解

工場の検査を想像してください。「部品Aの組み立てが正しいか」を調べたいのに、完成品の最終動作テストで判定していると、組み立て後の塗装工程や梱包工程の不具合まで巻き込んで「不合格」が出てしまいます。部品Aの良し悪しを知りたいなら、Aを組み立てた直後にAだけを検査するのが筋です。観測点を下流に置きすぎると、無関係な工程のノイズが混ざるのです。

マルチカメラの運転モデルでも同じです。複数カメラの映像を一つにまとめる「ビュー融合(view fusion)」が正しいか確かめたい。自然な発想は「あるカメラの入力を強くいじったら出力が変わるはず。変わらないならそのカメラは使われていない」です。ところがこのテストを最終出力(軌道)で行うと、軌道を出す下流の計画部品(プランナ)の性質に引っかかります。プランナが内部で「広い鳥瞰地図のうちごく一部のマスだけを見る」設計(疎なサンプリング、詳細は [Deformableクロスアテンションのスパースサンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0304_deformable-cross-attention-sparse-sampling))だと、あるカメラがたまたま見られないマスにしか映らない場合、そのカメラを強くいじっても軌道は変わりません。すると「ビュー融合は正しいのに、プランナのサンプリングの都合でテストが落ちる」という事故が起きます。検査したいのは融合なのに、梱包工程(プランナ)のノイズで落ちているわけです。

## 基礎: 前提となる概念

ビュー融合(view fusion / multi-view fusion): 複数カメラ(例えば6〜8台で360度を覆う構成)の各映像から抽出した特徴を、一つの統合特徴へまとめる処理。各カメラは視野(field of view)が異なるので、重複や死角を考慮して融合します。

特徴マップ(feature map): モデルの途中段階で出てくる多次元配列。入力画像群を `(B, V, C, H, W)` で表すと、B はバッチサイズ、V はビュー(カメラ)数、C はチャネル(色や特徴の次元)、H・W は画像の縦横。融合後の特徴は典型的に `(B, C', H', W')` のように1枚に統合されます。

プランナ(planner / trajectory planner): 統合特徴から将来の走行軌道を出す下流の部品。BEV(鳥瞰図)上で動くものや、クエリベースのものなど形式は様々です。

疎なサンプリング(sparse sampling): 全部の位置を見ずに、学習で選んだ少数の位置だけを参照する仕組み。deformable attention が代表で、計算量を劇的に減らせる一方、選ばれない位置は「存在しないのと同じ」になります(これがテスト失敗の根本原因)。

偽陽性・偽陰性(false positive / false negative): テストの誤判定。ここでの偽陽性は「実際は正しいのに失敗と判定」、つまり「融合は正しいのにテストが落ちる」こと。テストが本来検証したい性質と違う要因で結論を出すと、こうした誤判定が起きます。

関心事の分離(separation of concerns): 異なる責任を持つ要素を切り分けて扱う設計原則。ここでは「ビュー融合の正しさ」と「プランナのサンプリング網羅性」という2つの別々の関心事を、別々に検証すべきだ、という話です。

メタモルフィックテスト(metamorphic testing): 「入力をこう変えたら出力はこう変わるはず」という関係(メタモルフィック関係)を検証するテスト手法。正解ラベルを直接持たなくても、入力変換と出力変化の関係だけで正しさを部分的に検証できる。本ノートのテストは「あるビューを変えたら融合特徴は変わるべき」という、この一種です。

## 仕組みを詳しく

問題の構造を分解します。素朴なテストは、各カメラビューを1台ずつ強くいじって(例: 値を 5.0 に書き換え)、最終出力(軌道)が変化するかを見るものでした。狙いは「ビューが出力に寄与しているか」の確認です。しかしこのテストは暗黙に2つの性質を同時に要求していました。

1. ビュー融合の正しさ: すべてのカメラが融合特徴マップに寄与している(本来検証したいこと)。
2. プランナのサンプリング網羅性: プランナがそのビューの担当領域を実際にサンプリングする(プランナ固有の別問題)。

ランダム初期化のモデルでは、プランナの疎サンプリングがあるカメラの担当領域をまるごと無視することがあり、そのとき 2 が偽になります。すると 1 が真でもテスト全体が落ちる。検証したいのは 1 だけなのに、2 が混ざってしまっているわけです。重要なのは、これは学習前のランダム初期状態で特に起きやすいという点です。学習が進めばサンプリング点は意味のある場所へ収束しますが、テストは初期化直後のモデルで走ることが多く、そこで偽陽性が出ます。

修正の核心は、観測点を最終出力(軌道)から融合特徴マップ(ビュー融合の直接の出力)へ移すことです。融合特徴はまさにビュー融合が責任を持つ出力なので、下流プランナのサンプリングの仕方に一切左右されません。典型的なテストコードはこうなります。

```python
def fused_features(x):
    # x: (B, V, C, H, W) を (B*V, C, H, W) にして backbone へ
    feats = model.backbone(x.reshape(B * V, C, H, W))
    return model.feature_fusion(feats, B, V)   # ビュー融合の直接出力

fused_base = fused_features(visual)            # 基準となる融合特徴
for view_idx in range(V):
    visual_mod = visual.clone()
    visual_mod[0, view_idx] = 5.0              # ビュー view_idx だけ強く書き換え
    fused_mod = fused_features(visual_mod)
    assert not torch.allclose(fused_base, fused_mod, atol=1e-5)
```

ステップで読みます。

1. 元の入力 `visual`(形状 `(B, V, C, H, W)`、例 `(1, 7, 3, 256, 256)`)を backbone → feature_fusion に通し、基準となる融合特徴 `fused_base` を作る。
2. ビューを1台ずつ選び、そのビューだけを 5.0 に書き換えた入力で融合特徴 `fused_mod` を作る。
3. `fused_base` と `fused_mod` が許容誤差(atol=1e-5)で完全一致して「いない」ことを確認する。もし一致してしまうなら、そのビューは融合特徴に寄与していない、つまりビュー融合のバグである。

ポイントは、プランナを通さないことです。これにより「プランナのサンプリング網羅性」という別の関心事が紛れ込まず、ビュー融合の正しさだけを直接測れます。一般に、異なる融合方式(連結、クロスアテンション、BEV など)のすべてで同じテストロジックが安定して通るようになります。

なぜ `not allclose` という形にするのか。これはメタモルフィック関係を「入力ビューを変える → 出力は変わるべき(不変であってはならない)」と定式化したものです。逆の関係、例えば「あるビューだけを変えたとき、そのビューに対応する領域の特徴だけが変わり、無関係な領域は変わらない」といった、より強い局所性のメタモルフィック関係を組むこともできます。緩い関係(全体が変わればOK)は通りやすく頑健で、強い関係(局所だけ変わるべき)は感度が高い代わりに実装依存になります。テスト設計では、検証したい性質の強さと頑健性のバランスを選びます。

before/after を表で整理します。

| | 修正前(出力で観測) | 修正後(融合特徴で観測) |
|---|---|---|
| 検証する出力 | 軌道(プランナ通過後) | 融合特徴マップ(融合直後) |
| 巻き込む関心事 | ビュー融合 + プランナサンプリング | ビュー融合のみ |
| 疎サンプリング系プランナでの結果 | 偽陽性で失敗しうる | 安定して合格 |
| 失敗時の原因切り分け | 融合かプランナか不明 | 融合の問題と確定 |

なぜこの設計が正しいのか。テスト設計の一般原則として、「テスト対象の責任範囲ちょうどで観測する(観測点を下流に置きすぎない)」というものがあります。観測点を最終出力に置くと、対象とは無関係な下流要因(ここではプランナの疎サンプリング)で偽陽性・偽陰性が生じます。観測点を対象の出力境界に合わせることで、テストの結論が対象の性質だけを反映するようになります。これはソフトウェア工学でいう「観測可能性(observability)を責任境界に合わせる」ことであり、ユニットテストの粒度設計の基本でもあります。

## 手法の系譜と主要論文

これはモデル手法ではなくテスト設計(ソフトウェア工学)の話なので、直接対応する単一の論文はありません。ただし、テスト失敗を引き起こした手法側の文脈と、テスト設計の原則の両方に系譜があります。

手法側の文脈として、DETR (Carion et al., ECCV 2020, arXiv:2005.12872) が密なアテンションでクエリベースの検出を確立し、Deformable DETR (Zhu et al., ICLR 2021, arXiv:2010.04159) が「各クエリが学習した少数の点だけを見る」疎なサンプリングを導入しました。BEVFormer (Li et al., ECCV 2022, arXiv:2203.17270) はこれを多カメラBEVに適用し、BEVクエリが各カメラの射影位置周辺の少数点だけを見るようにしました。この「全位置を必ず見るわけではない」性質こそが、出力ベースのビュー寄与テストを不安定にする根本原因です([Deformableクロスアテンションのスパースサンプリング](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0304_deformable-cross-attention-sparse-sampling) 参照)。

テスト設計の原則の系譜としては、ユニットテストにおける「観測可能性(observability)」と「関心事の分離」の議論が基盤です。機械学習システムのテストに特化した整理としては、Breck らの "The ML Test Score"(Breck et al., IEEE Big Data 2017, Google)が、データ・モデル・インフラの各層をどう検証すべきかを体系化しており、「中間表現を直接検証する」発想と通じます。また、ニューラルネットの「不変性テスト/メタモルフィックテスト」(入力を変えたとき出力がどう変わるべきかを規定するテスト)の文脈にも位置づけられます。NLP分野で行動テストを体系化した CheckList (Ribeiro et al., ACL 2020, arXiv:2005.04118) は、モデルに期待する不変性・方向性の関係をテストケースとして列挙する発想を広めました。本ノートのテストは「あるビューを変えたら融合特徴は変わるべき」という不変性/方向性テストの一種です。

## 論文の実験結果(定量データ)

テスト設計そのものに固有のベンチマーク数値はありませんが、関連する定量的事実を挙げます。

第一に、疎サンプリングの効果。Deformable DETR は COCO 物体検出(画像認識の標準ベンチマーク、指標は mAP)で、密なアテンションのDETRと同等以上の精度を達成しつつ、学習に必要なエポック数をおよそ10分の1(500エポック級から50エポック級へ)に短縮したと報告しています。この劇的な効率化の代償として「各クエリは少数点しか見ない」性質が導入され、それがビュー寄与テストの不安定化につながりました。効率化の利得と、テスト上の副作用は同じコインの裏表です。

第二に、メタモルフィックテストの一般的知見。ML システムのテスト研究では、「最終出力だけを観測するテストは、無関係な下流要因の影響でフレーキー(flaky、実行ごとに結果が揺れる)になりやすい」ことが繰り返し報告されています。観測点を中間表現に移すことでテストの安定性(同じ入力で同じ結果が出る再現性)が向上する、という定性的な改善は、実務上「修正前は疎サンプリング系の融合モードで一部のテストが偽陽性で落ち、修正後は全融合モードで安定通過する」という形で繰り返し観察されます。

数値の意味を補足します。mAP は検出の適合率(precision、検出したもののうち正しい割合)と再現率(recall、正解のうち検出できた割合)を統合した0〜100の指標で、高いほど良い。学習エポック数は収束の速さで、少ないほど計算コストが安い。フレーキーなテストは、コードを変えていないのに失敗したり成功したりするため、開発者の信頼を損ない、「またあのテストか」と無視されて本物のバグを見逃す原因になります。ソフトウェア工学の研究では、フレーキーテストが継続的インテグレーション(CI)の生産性を大きく損なうことが報告されており、観測点の移動はこの信頼性を取り戻す投資です。

## メリット・トレードオフ・限界

メリット。テストがビュー融合の正しさだけを検証するようになり、下流プランナの疎サンプリングに起因する偽陽性が消えます。融合方式(連結・クロスアテンション・BEV)に依存せず同一のテストロジックで検証できます。失敗時に原因(融合かプランナか)を切り分けやすくなり、デバッグが速くなります。テストが速くなる副次効果もあります。プランナまで通さないぶん、計算が軽く、CIでの実行時間が短縮されます。

トレードオフと限界。第一に、このテストは「最終出力(軌道)までビューの影響が伝わっているか」というエンドツーエンドの性質を保証しません。それはプランナを含む別のテスト(例えば学習済みモデルでの感度分析や、サンプリング網羅性の独立検証)で見る必要があります。テストピラミッドでいう「ユニットレベルの検証は安定するが、統合レベルの性質は別途必要」という分業です。第二に、中間出力に依存するため、融合部品の内部API(関数シグネチャや返り値の形状)が変わるとテストの修正が必要になります。テストが実装に密結合する代償です。第三に、「allclose で一致しない」という判定はビューが寄与する「ことの確認」であって、寄与の「正しさ(正しい量・正しい場所)」までは保証しません。寄与の質は別の評価が要ります。

未解決の論点として、中間表現テストとエンドツーエンドテストのどこに線を引くか、ランダム初期化でのテストと学習済みモデルでのテストをどう組み合わせるか、といった「ML特有のテスト戦略」の設計指針は、研究・実務の両面でなお発展途上です。

## 発展トピック・研究の最前線

ML システムのテストは近年急速に体系化されています。メタモルフィックテスト(入力変換と出力変化の関係を規定するテスト)、不変性テスト(あるべき不変性の検証)、行動テスト(CheckList, Ribeiro et al., ACL 2020 など、NLPで提唱されタスク横断に広がった発想)が代表です。本ノートの「中間特徴で観測する」発想は、解釈性研究のプロービング(probing、中間表現に何が符号化されているか線形分類器などで調べる手法)とも接続します。さらに、フレーキーテストの自動検出・原因特定はソフトウェア工学の活発な研究領域で、ML パイプラインのテスト安定化に応用が進んでいます。観測点をどこに置くべきかという問いは、モデルが大規模化・多段化するほど重要になり続けます。近年は知覚から計画までを一気通貫で学習するエンドツーエンドモデルが主流化しており、こうした多段スタックでは「どの段が壊れたか」を切り分ける中間レベルのテストの価値がいっそう高まっています。
