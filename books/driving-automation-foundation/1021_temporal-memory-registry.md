# テンポラルメモリ レジストリ

## ひとことで言うと

自動運転モデルが「過去のフレームの情報をどう覚えておくか」を、性質の異なる複数のやり方から自由に差し替えて比較できるようにする設計のことです。過去をほとんど覚えない素朴なやり方、過去を一定の頻度に間引いて圧縮するやり方、過去の特徴マップを時系列で並べてアテンションを掛けるやり方、軽量な物体クエリだけを伝播させるやり方などを、すべて同じ入口と出口のデータ形に揃えておきます。すると下流の処理を一切書き換えずに、記憶モジュールだけを名前1つで取り替えられます。この「同じ契約で差し替え可能にする」抽象化を、本稿では便宜的にテンポラルメモリ レジストリ(時間記憶モジュールの登録簿)と呼びます。

## 直感的な理解

人間は運転中、目の前の一瞬の景色だけで判断しているわけではありません。「さっきまであの車は減速していた」「歩行者が一瞬電柱の陰に隠れたが、まだそこにいるはずだ」「信号がついさっき青に変わった」といった、少し前の記憶を絶えず使っています。

機械にカメラ映像から運転判断をさせる場合も同じです。1フレーム(今この瞬間の画像)だけを見て判断すると、次のような壁にすぐぶつかります。

- 前の車が「今ちょうど止まっている」のか「減速中で止まりかけ」なのかは、静止画1枚では区別しづらい。数フレーム並べて初めて速度や加速度が読み取れる。
- 物体が一瞬だけ別の物体の陰に隠れる(これをオクルージョン、occlusion と呼びます。手前の物体が奥の物体を遮蔽すること)とき、直前まで見えていた記憶があれば「まだそこにいるはず」と推測できる。
- 信号や方向指示器の状態変化のような時間的な文脈は、過去なしには判定できない。

つまり時間方向の記憶(temporal memory、テンポラルメモリ=過去の情報を現在の判断に持ち越す仕組み)が必要です。問題は「どの覚え方が一番性能を上げるのか」が机上では決まらないことです。情報をたくさん覚えれば判断は良くなるかもしれませんが、メモリと計算が重くなり、車載のリアルタイム制約に乗らなくなります。だから複数の覚え方を公平な条件で差し替えて測れる土台が要ります。これがレジストリ型抽象化の存在理由です。

## 基礎: 前提となる概念

いくつかの用語を先に噛み砕きます。

- フレーム(frame): 動画の1コマ。10Hz(ヘルツ=1秒あたりの回数)なら1秒間に10枚撮影します。
- 特徴量(feature): 生のピクセルではなく、ニューラルネットが画像から抽出した「意味のある数値ベクトル」。たとえば「この領域に車がいそうだ」という情報を数百次元の数字に圧縮したものです。
- BEV(Bird's-Eye-View、鳥瞰図): カメラ画像を真上から見下ろした地図のような表現に変換したもの。複数カメラの情報を1枚の俯瞰格子(たとえば縦200×横200のセル、各セルが256次元の特徴を持つ)にまとめると、空間的な位置関係を扱いやすくなります。
- エゴモーション(ego-motion、自車運動): 自分の車が前のフレームから今のフレームまでにどれだけ動いたか(並進と回転)。過去の特徴を現在の座標系に合わせ直す(位置合わせ、alignment)のに使います。動いている車から見ると世界の方が逆向きに流れて見えるので、これを補正しないと過去と現在の特徴がズレます。
- セルフアテンション(self-attention): 系列の各要素が「他のどの要素にどれだけ注目すべきか」を学習で重み付けし、情報を混ぜ合わせる仕組み。Transformer の中核です。時間方向に掛ければ「今のフレームが過去のどのフレームを参照すべきか」を学べます。
- テンソルの形(tensor shape): 多次元配列のサイズ。`[B, T, feat]` のように書きます。`B` はバッチサイズ(同時処理するサンプル数)、`T` は時間ステップ数(過去フレームの本数)、`feat` は各時刻の特徴次元です。

時間記憶モジュールの仕事は、突き詰めると「過去の系列 `[B, T, feat]` を受け取り、時間軸 `T` を畳んで、今の判断に渡す1本の文脈ベクトル `[B, feat]` に要約する」ことだと理解しておくと、以降がすっきり読めます。

## 仕組みを詳しく

### 共通インターフェース: 時間軸を畳んで文脈ベクトルにする

差し替え可能性を成立させる核心は、すべての記憶方式に共通の入出力契約を定めることです。典型的には、抽象基底クラス(abstract base class=「このメソッドを必ず実装せよ」という約束だけを定めた親クラス)を用意し、順伝播処理の入出力形を固定します。

```
入力:  visual_history    [B, T, V]   過去の視覚特徴の系列
       egomotion_history [B, T, E]   過去の自車運動の系列
出力:  visual_context    [B, V]      今の判断に渡す視覚文脈
       egomotion_context [B, E]      今の判断に渡す運動文脈
```

具体的な数値例で読みます。`B=8`(8サンプル同時)、`T=64`(過去64フレーム。10Hz なら6.4秒分)、視覚側 `V` が数百〜千次元、運動側 `E` が数百次元、とします。入口は `[8, 64, V]` という「時間の系列」、出口は `[8, V]` という「時間を消した1本のベクトル」です。

出口の形を全方式で揃えてあるので、下流のトラジェクトリプランナー(trajectory planner=将来の運転軌道を出す部分)は、上流で何の記憶方式を使っているかを知らずに動けます。記憶方式を入れ替えてもプランナー側のコードに一切手を入れなくてよい。これが「差し替え可能」を成立させている要です。

### レジストリ: 名前で選ぶ

実装上は、文字列キーからクラスを引く辞書(レジストリ)を1つ用意するのが典型です。

```
TEMPORAL_MEMORY_REGISTRY = {
    "no_memory":  NoMemory,
    "compressed": CompressedHistoryEncoder,
    "bev_queue":  BEVQueueAttention,
    "object":     ObjectCentricMemory,
}

def build_temporal_memory(mode, **kwargs):
    return TEMPORAL_MEMORY_REGISTRY[mode](**kwargs)
```

`build_temporal_memory("compressed", ...)` のように呼ぶと対応するクラスが組み立てられます。新方式を足したいときは辞書に1行加えるだけ。設定ファイルや実験スクリプトはキー名を変えるだけで方式を切り替えられます。デフォルトを「過去を要約しない素朴な方式」にしておくと、機能を足しても既存挙動を壊さない(additive=追加的)という安全性が得られます。これはソフトウェア設計でいうストラテジーパターン(Strategy pattern=アルゴリズムを差し替え可能な部品として外から注入する設計)の一種であり、機械学習フレームワークで広く使われるコンポーネントレジストリ(部品の登録簿)と同じ発想です。

### 方式1: 過去を要約しないベースライン

```
v_ctx = visual_history[:, -1]   # 時間軸の最後 = 最新フレームだけ取る
```

`[:, -1]` は「時間軸の一番最後の要素 = 最も新しいフレーム」を取り出す操作です。`[B, 64, V]` → `[B, V]`。過去はほぼ捨てて最新フレームだけ渡します。学習パラメータをまったく持たない(parameter-free)パススルーです。性能比較の下限(baseline)として置きます。研究では「凝った手法が、この素朴な下限に本当に勝てているか」を必ず確認する必要があり、その基準線として重要です。後述のとおり、時間融合をうたう手法が実はこの素朴な下限とほとんど差がない、という落とし穴は珍しくありません。

### 方式2: 過去を一定頻度に圧縮する

「過去の表現が薄すぎる。時間の粒度は粗くしてよいから、その代わり1ステップあたりの特徴を厚くしたい」という要望に応える方式です。典型的には2段で処理します。

1. 時間圧縮(temporal compression): 1次元畳み込み `Conv1d(kernel=stride=R)` を使い、`R` ステップの窓を重なりなしで1ステップに畳み込みます。`R=10` なら 10Hz→1Hz。同時に特徴次元を厚く拡張します(粗い時間・厚い特徴)。`T` が `R` の倍数でないときは、一番古い余り `T % R` ステップを左から捨て(left-trim)、最新フレームが必ず窓に残るよう現在に揃えます。例: `[B, 64, V]` → `[B, 6, H]`(6.4秒分が6ステップ ≈ 1Hz、`H` は拡張後の隠れ次元)。
2. 系列要約(sequence summarisation): GRU(Gated Recurrent Unit=系列を順に読んで内部状態を更新する小型の再帰ネットワーク。LSTM を簡略化したもの)を6ステップに流し、最後の隠れ状態を文脈ベクトルにします。`[B, 6, H]` → `[B, H]`。

視覚特徴と自車運動を特徴軸で連結してまとめて圧縮し、最後に再分割して2本の文脈に戻す実装が多いです。狙いは判断のチラつき(decision flicker=フレームごとに出力がガタつく現象)を減らすこと。時間を粗くすると入力の高周波ノイズが平均化され、出力が安定します。

### 方式3: 履歴キューへの時間アテンション

直近Nフレームの履歴キュー(queue=待ち行列。古いものを押し出しながら新しいものを足す固定長バッファ)に対し、時間方向セルフアテンションを掛ける方式です。素朴方式(順序を見ない)や圧縮方式(再帰的に畳む)に対し、こちらは順序を意識した(order-aware)・アテンションベースの代替です。学習可能なクエリトークン、入出力の射影層、正弦波の位置埋め込み(sinusoidal positional encoding=各時刻に固有の波形ベクトルを足して順序情報を注入する古典的手法。Transformer の原論文由来)を持ちます。後述の BEVFormer の考え方を、特徴マップ全体ではなく履歴ベクトルのキュー上のアテンションへ簡略化して移植したもの、と位置づけられます。

### 方式4: 物体中心のクエリ伝播

各物体に対応する軽量なベクトル(物体クエリ)だけを次フレームへ持ち越し、密な特徴マップは保持しない方式です。記憶量が物体数に比例し、フレーム数に対して一定なので、長期履歴でもメモリが膨らまない利点があります。これは後述の StreamPETR 系の考え方を抽象インターフェースに載せたものです。

### ASCII図: 各方式の比較

```
no_memory:   [B,T,V] ──最新だけ取る──────────────▶ [B,V]      (学習なし、下限)
compressed:  [B,T,V] ─Conv1d圧縮─▶ [B,6,H] ─GRU─▶ [B,H]      (粗い時間・厚い特徴)
bev_queue:   [B,T,V] ─位置埋め込み+self-attn─────▶ [B,V]      (順序を見る、重め)
object:      過去クエリ ─伝播+motion補正──────────▶ 現在クエリ  (疎・長期に強い)
```

## 手法の系譜と主要論文

時間融合の研究は「密な特徴マップを覚える」系統と「軽いクエリだけ覚える」系統の2つに大きく分かれ、互いに改良し合ってきました。

- BEVFormer(Li et al., ECCV 2022, arXiv:2203.17270)。前フレームのBEV特徴マップ(およそ200×200×256)を丸ごと保持し、ego-motion で現在の座標系に位置合わせしてから時間セルフアテンションで融合します。動機は「細い情報ボトルネックを通さず、特徴マップの粒度で物体運動やオクルージョン履歴を扱いたい」。動的シーン理解を大きく前進させ、後続の多くの手法が採用しました。
- BEVDet4D(Huang & Huang, 2022, arXiv:2203.17054)。現在と前フレームのBEV特徴を ego-motion で揃えてからチャネル方向に連結し、畳み込みで融合します。動機は「アテンションより実装が単純で計算も軽くしたい」。速度推定を劇的に改善しました。隣接2フレームに限られ、長期履歴への自然な拡張が難しいのが弱点です。
- SOLOFusion(Park et al., ICLR 2023, arXiv:2210.02443)と VideoBEV(Han et al., CVPR 2023, arXiv:2303.05970)。隣接フレームだけでなく長期(十数フレーム)を融合します。SOLOFusion は「短期の高解像度+長期の低解像度」を組み合わせ、再帰融合で起きる情報損失と計算コストの両立を狙いました。長距離・低速物体の検出を改善する一方、メモリと複雑さが増します。VideoBEV は再帰的に履歴を畳み込み、長期情報を一定コストで持ち越すストリーミング融合を示しました。
- StreamPETR(Wang et al., ICCV 2023, arXiv:2303.11926)と Sparse4D v2(Lin et al., 2023, arXiv:2305.14018)。密な特徴マップを保存する代わりに、軽量な物体クエリ(object query)だけをフレーム間で伝播させ、motion-aware layer normalization で動きを補正します。動機はメモリ効率と可変フレームレートへの自然な対応。空間グリッド型の設計から物体クエリ型へ作り替える手間が要る点がトレードオフです。

この系譜は「密に覚える(BEVFormer/BEVDet4D/SOLOFusion/VideoBEV)」対「疎に覚える(StreamPETR/Sparse4D)」という設計軸の両端を示しています。レジストリ的な抽象化は、この軸上の複数点を同じ土俵で測るための実験基盤と言えます。

## 論文の実験結果(定量データ)

時間融合の効果は nuScenes(自動運転の標準的な大規模データセット。6カメラ・LiDAR付き、約1000シーン)での検出指標で測られることが多いです。指標の意味を先に説明します。

- mAP(mean Average Precision、平均適合率): 物体検出の正確さ。0〜1で高いほど良い。「検出した物体がどれだけ正しく、漏れがないか」を総合した値です。
- NDS(nuScenes Detection Score): nuScenes 独自の総合スコア。mAP に加え、位置・大きさ・向き・速度・属性の誤差をまとめたもの。0〜1で高いほど良い。速度誤差を含むため、時間融合の効果が出やすい指標です。
- mAVE(mean Average Velocity Error): 推定速度の平均誤差(m/s)。低いほど良い。時間融合の有無が最も鋭く効く指標です。

報告値(各論文より、いずれもおよその値):

- BEVFormer は時間アテンションありで NDS がおよそ0.51〜0.56に達し、単フレーム版に対して数ポイント改善しました。とくに mAVE が時間情報で大きく下がります。
- BEVDet4D は前フレーム連結により、単フレームの BEVDet 比で mAVE をおよそ半分近く(報告では 0.9 台 → 0.5 前後のオーダー)まで削減したとされます。速度を直接観測できない単眼系にとって、隣接フレーム差分は速度の最も自然な手掛かりです。
- SOLOFusion は長期融合により nuScenes で当時の最高水準 NDS およそ0.54前後を達成し、とくに遠方・低速物体の検出を改善しました。アブレーション(どの要素を抜くと性能がどう落ちるかを調べる実験)では「短期だけ」「長期だけ」より「短期+長期の併用」が一貫して優れることを示しています。
- StreamPETR は密マップを持たずに NDS およそ0.55〜0.62(バックボーン規模による)を達成し、メモリと計算を抑えつつ時間融合系と同等以上であることを示しました。クエリ伝播はフレームあたりの追加コストが小さく、リアルタイム性で有利です。

アブレーションで共通して見えるのは、(1)時間融合を完全に外すと速度系の誤差 mAVE が大きく悪化する、(2)ego-motion による位置合わせを外すと、動いている車から見た見かけの流れを補正できず効果が大きく減る、という2点です。つまり「過去を持つこと」と「正しく位置合わせすること」の両方が効いています。この2点は、レジストリ上のどの方式でも共通して確認すべき要点です。

なお、運転そのものの良し悪し(軌道の質)は検出指標とは別物で、時間記憶の比較は最終的には軌道評価で締める必要があります。詳しくは [軌跡品質メトリクス](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0730_trajectory-quality-metrics) を参照してください。

## メリット・トレードオフ・限界

メリット

- 記憶方式を1行で差し替えられ、同じ入出力契約のもとで公平に比較できる。
- 新方式の追加が辞書1行で済み、下流コードは無改変。
- デフォルトを素朴方式にしておけば、機能追加が既存性能やAPIを壊さない。
- 方式ごとに計算コスト・推論レイテンシ・パラメータ数を同条件で測れる。

トレードオフと限界

- 「どの方式が運転を良くするか」は、学習済みモデルと閉ループ評価が揃うまで結論が出ない。ランダム重みの段階では計算特性しか比べられない。
- BEVFormer 系は ego-motion による位置合わせと巨大マップの保存コストが大きい。車載のメモリ・レイテンシ制約に直結する。
- 入出力契約を `[B, feat]` ベクトルに固定すると、本来「空間マップそのもの」を渡したい方式は要約を強いられ、空間構造の一部を失う。表現の自由度と差し替え容易性はトレードオフの関係にある。物体中心方式のように出力が「クエリの集合」である方式は、ベクトル1本への要約と相性が悪く、この共通契約に無理なく載らないことがある。
- 長期履歴は誤差や古い情報の蓄積(stale memory)を招きうる。どこまで遡るかはデータセットと地平に依存する未解決の設計問題。

## 発展トピック・研究の最前線

最近は時間融合を検出だけでなく、エンドツーエンドの運転計画に直接組み込む流れが強まっています。占有予測(occupancy forecasting=各空間セルが将来どう埋まるかを予測する)や、世界モデル(world model=観測の未来を生成的に予測する内部モデル)に時間記憶を統合し、「過去の記憶から未来を想像して計画する」方向が活発です。また、可変フレームレートやセンサ脱落に頑健な記憶(クエリ伝播系)、長期と短期を階層的に持つメモリ、Transformer の長系列を効率化する状態空間モデル(state space model、例: Mamba 系)の応用も研究されています。レジストリ抽象化の価値は、こうした新方式が登場するたびに「既存の土台に1点追加して、確立した方式と同条件で測れる」ことにあります。
