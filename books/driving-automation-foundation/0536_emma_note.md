# EMMA (End-to-End Multimodal Model for Autonomous Driving)

## ひとことで言うと
EMMA は、大規模なマルチモーダル基盤モデル(画像と言葉を一緒に扱える巨大 AI)を土台にして、カメラ映像とテキストの指示・履歴を入力すると、走るべき軌道・周囲の物体・道路のレイアウトを「すべて言葉(テキスト)」として直接出力する自動運転モデルです。物体検出器・予測器・プランナという専用部品を別々に積み上げる従来のパイプラインを、1つの巨大な言語ベースモデルに溶かし込んだ点が最大の特徴です。「運転とは結局、見たものを理解して次の一手を言葉で説明できることだ」という発想を、基盤モデルの能力で極限まで推し進めた到達点と言えます。

## 直感的な理解
人間が運転教習で「次に何をするか言ってみて」と言われたら、「前の車が止まっているから、右に車線変更して追い越す」のように、見えているもの・予測・自分の行動を言葉でつなげて説明できます。EMMA がやろうとしているのは、まさにこれです。AI に対して「画像を見て、これからどう走るかを言葉で書き出して」と頼み、その「言葉」の中に未来の座標列(軌道)まで含めてしまいます。

なぜわざわざ全部を言葉にするのでしょうか。従来の自動運転は、専門の部品を縦に積み上げます。まず物体検出器が「ここに車、ここに歩行者」と箱(バウンディングボックス)を出し、次に予測器が「この車は左に動く」と予測し、最後にプランナが「では自車はこう走る」と軌道を決めます。各部品は別々のチームが別々に作り、間を「3次元の箱のリスト」のような決まった形式でつなぎます。

この方式には弱点があります。箱のリストにした瞬間、生の映像にあった微妙な手がかりが消えてしまうのです。たとえば「歩行者がスマホを見ながら歩いていて、こちらに気づいていない」「ブレーキランプが点いた」「警察官が手で誘導している」という情報は、ただの「歩行者:位置 (x,y)」という箱になると失われます。専用の箱フォーマットには「想定済みの物体カテゴリ」しか入らず、想定外のもの(道路に落ちた家具、変則的な工事誘導)は表現の枠そのものに収まりません。EMMA は「中間で情報を捨てるのをやめ、生の映像から直接、言葉で答えを出そう」と考えました。すべてを1つの基盤モデルに任せれば、部品間の情報損失も、別々に最適化する非効率も消える、という狙いです。

## 基礎: 前提となる概念
理解に必要な用語を順に噛み砕きます。

基盤モデル(foundation model)とは、インターネット規模の膨大なデータで一度だけ大規模に事前学習しておき、その後さまざまなタスクに使い回す巨大なニューラルネットのことです。GPT や Gemini がその代表で、文章の意味・世界の常識・論理的な説明力をあらかじめ獲得しています。「歩行者が横断歩道にいたら車は止まるべき」といった常識を、運転データから学び直さなくても、もともと言葉として知っているのが強みです。

マルチモーダル(multimodal)とは、複数の種類の入力(モダリティ)を同時に扱えること、という意味です。テキストだけでなく画像も同じモデルに入れて処理できるモデルを「マルチモーダル基盤モデル」と呼びます。EMMA の土台である Gemini はこのタイプで、画像トークン(画像を細かく区切って数値ベクトルにしたもの)とテキストトークンを区別なく1つの系列として扱えます。

トークン化(tokenization)とは、入力を AI が処理できる最小単位(トークン)に区切ることです。文章なら単語の断片に、画像なら小さなパッチに区切ります。EMMA で重要なのは、座標のような数値も「文字の並び」としてトークン化する点です。たとえば座標 `5.27` は `5`,`.`,`2`,`7` のように文字に分解され、モデルはこれを1文字ずつ予測していきます。この「数値も言葉として出す」設計は、画像中の物体を文字列で出力する Pix2Seq(Chen et al., 2021)に源流があり、回帰専用の出力層を捨てて生成に統一する流儀です。

自己回帰生成(autoregressive generation)とは、「これまでに出した出力を見て、次の1トークンを予測する」を繰り返して系列を作る方式です。言語モデルが文章を1単語ずつ綴るのと同じ仕組みで、EMMA は軌道の座標列もこの方式で1文字ずつ綴ります。系列が長いほど推論に時間がかかる、という性質はここから来ます。

Chain-of-Thought(CoT、思考の連鎖)とは、結論を出す前に途中の考えを言葉にしてから答えに至らせる手法です(Wei et al., NeurIPS 2022)。「前方車が減速 → 右車線は空いている → だから右に寄って追い越す」のように中間推論を明示させると、複雑な判断の精度が上がり、かつ「なぜその答えか」を人間が追えるようになります。

## 仕組みを詳しく
EMMA の核心は「運転に関わるあらゆる入出力を、基盤モデルが扱える自然言語のテキストとして表現する」ことです。専用の数値ヘッド(回帰用の出力層)をなるべく作らず、すべてを言語生成問題に帰着させます。

入力(すべてモデルに渡す):

- マルチビューのカメラ映像。複数方向のカメラ画像を画像トークン列に変換して入力します。
- 高レベルの運転意図やナビ指示(テキスト)。「直進してから右折」のような命令文(intent command)。
- 自車の過去の状態履歴(過去数秒の速度・位置)。これもテキストに書き起こして渡します。これにより、いま加速中か減速中かといった動的文脈をモデルが把握できます。

出力(すべてテキストとして自己回帰生成):

- 未来の運転軌道。自車を原点とする座標系(ego-centric BEV、自車中心の俯瞰座標)で、将来数秒ぶんの waypoint(通過点)の座標列を数字の文字列として出力。
- 周囲の物体。3次元の物体の位置・カテゴリをテキストで記述。
- 道路要素(road graph)。車線・標識などのレイアウトをテキストで記述。

```
入力: マルチビュー画像 + "go straight, then turn right" + 過去軌道(テキスト)
        │
   Gemini ベースの EMMA(画像トークン + テキストトークンを1系列で処理)
        │
   まず CoT で中間推論を言語生成:
     "前方に減速中の車。右車線は空き。右に寄りつつ前進すべき"
        │
   続けて答えを全部テキストで生成:
     軌道       → "(0.0,6.0),(1.5,6.1),(3.1,6.3),..."
     検出物体   → "car at (12.3,-2.1), pedestrian at (4.0,1.2)"
     道路要素   → "lane_boundary: ...; sign: stop"
```

軌道を数字の文字列で出すのは一見奇妙ですが、これにより検出・予測・計画という別々のタスクを1つの言語生成タスクに統一でき、複数タスクを1モデルで同時学習(マルチタスク学習)できます。同じ基盤モデルが、ある時は物体を記述し、ある時は軌道を綴る。重みを共有することで、知覚で学んだ表現が計画に効く、という相互作用(positive transfer、正の転移)が期待できます。

CoT 軌道生成という工夫も重要です。軌道座標をいきなり出すのではなく、その前に「重要物体は何か(critical objects)」「自車が取るべき高レベル行動は何か(減速・車線変更など)」「その行動の根拠(meta-driving decision)」という推論文をモデル自身に生成させ、それを条件として最終軌道を出します。中間推論を挟むことで計画品質が向上したと報告されています。

センサ構成も特徴的です。EMMA はカメラのみ(camera-only)で、LiDAR(レーザーで距離を直接測るセンサ)や HD マップ(事前に作り込んだ高精細地図)に依存しません。これはコストを大きく下げられ、また地図整備のいらない地域へ展開しやすい一方、3次元形状を直接測れないため、正確な距離推定や悪条件での頑健性で不利を抱えます。

## 手法の系譜と主要論文
言葉で運転を扱う研究は段階的に発展してきました。

Pix2Seq (Chen et al., ICLR 2022, arXiv:2109.10852) は、物体検出を「画像 → バウンディングボックスを記述するトークン列」の生成問題として解いた先駆けで、回帰ヘッドなしで検出が成立することを示しました。EMMA が座標を文字列で綴る発想の技術的祖先です。

DriveGPT4 (Xu et al., 2023, arXiv:2310.01412) は、ビデオ入力から運転の説明文と低レベル制御信号(アクセル・ハンドルなど)を出す、マルチモーダル LLM を運転に使った先駆けです。「説明できる運転 AI」の路線を開きました。[DriveGPT4](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0533_drivegpt4)。

GPT-Driver (Mao et al., 2023, arXiv:2310.01415) は、計画(軌道生成)を言語モデリング問題として定式化し、知覚結果をテキスト化して LLM に軌道を綴らせました。EMMA が採る「軌道をテキストで出力する」発想を共有する近縁研究です。ただし知覚をテキストに落とす段階で情報が失われる点が EMMA と決定的に異なります。[GPT-Driver](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0538_gpt-driver)。

DriveVLM (Tian et al., 2024, arXiv:2402.12289) は、遅いが賢い VLM と速い従来プランナを併用するハイブリッド(dual-system)を提案しました。基盤モデルの賢さと実時間性を両立させる現実解です。[DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm)。

EMMA (Hwang, Sharma, Greer, Refaat et al., Waymo 2024, arXiv:2410.23262, TMLR 2025) は、これらに対し「単一の基盤モデル(Gemini)に知覚から計画まで全部を統一表現で任せる」という最も統合度の高い立場を取りました。新規性は、(1) 物体検出・道路グラフ理解・軌道計画をすべてテキスト生成として1モデルに統合し、(2) CoT 軌道生成を組み込み、(3) 大規模実走データでこの統一表現が実際に成立することを実証した点にあります。

土台となる Gemini (Google, 2023) は EMMA のマルチモーダル能力の源泉です。CoT の原典 Wei et al. (NeurIPS 2022) は中間推論の有効性を示した汎用テクニックで、EMMA はこれを軌道生成の前段に組み込みました。比較の文脈では、知覚・予測・計画を1つのニューラルネットに統合した end-to-end 手法 UniAD (Hu et al., CVPR 2023 Best Paper) が、従来型統合パイプラインの代表として位置づけられます([UniAD (Planning-oriented統合E2E)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1090_uniad_note))。

## 論文の実験結果(定量データ)
EMMA は複数の運転タスクで評価されました。指標の意味から説明します。

軌道計画(motion planning)の主指標は L2 誤差です。これはモデルが出した未来軌道と、人間が実際に走った軌道との、各時刻でのユークリッド距離(メートル単位のずれ)を平均したものです。小さいほど人間の運転に近いことを意味します。EMMA は nuScenes(実走の標準ベンチマーク)の planning 評価で、当時の代表的 end-to-end 手法と同等以上の L2 誤差を達成したと報告されています。nuScenes の planning では、平均 L2 がおよそ 1m を切るかどうかが強い手法の目安になり、EMMA はこの水準に到達したと報告されています。とくに大規模な社内 Waymo データで学習・評価したとき、強い計画性能を示しました。

3次元物体検出の指標には、Waymo Open Dataset(WOD)で使われる検出スコアが用いられます。EMMA はカメラのみという不利な条件にもかかわらず、カメラベースの3次元検出で競争力のある精度を達成したと報告されています。LiDAR を使う手法には及ばない場面もありますが、「言語生成で物体を記述する」という変わった方式でも、生成した文字列を3次元ボックスとしてデコードして検出が成立することを示した意義は大きいです。

道路グラフ(road graph)理解では、車線中心線などの幾何構造をどれだけ正しく再現できるか(予測したポリラインと正解の幾何的一致)を評価し、こちらも有望な結果が示されました。

アブレーション(どの要素を抜くと性能がどう変わるかの分析)で重要な知見が複数あります。第1に、複数タスクを同時学習させると、それぞれを単独で学習させるより各タスクの性能が上がるという正の転移です。検出・道路グラフ理解・計画を1モデルで一緒に学ぶと相互に助け合う、という統一表現の強みを定量的に裏づけました。第2に、CoT(中間推論)を入れると計画品質が向上したことで、いきなり軌道を出すより「考えてから書く」方が良いという CoT の効果が運転でも確認されました。第3に、基盤モデルの規模(パラメータ数)を大きくするほど性能が上がるスケーリング傾向が観察され、性能が基盤モデルの能力に強く依存することが示されています。同時に、入力画像の枚数(マルチビュー)や ego 履歴を増やすと性能が上がる傾向も報告されています。

## メリット・トレードオフ・限界
メリット

- 知覚から計画まで1つの基盤モデルで統一でき、モジュール間の情報損失が小さい。
- 基盤モデルの世界知識・常識・推論力を運転に転用でき、学習データに乏しい希少シーン(レアな工事誘導、見慣れない障害物)にも一般常識で対応できる可能性がある。
- 複数タスクの同時学習で正の転移が起き、各タスクの性能が相互に高まる。
- CoT による中間推論で計画品質が上がり、判断の理由を言葉で追える解釈可能性が得られる。
- カメラのみで LiDAR/HD マップ不要なため、センサ・地図のコストを下げ、地図未整備地域へ展開しやすい。

トレードオフ・限界

- 基盤モデル級の計算は極めて重く、車載での実時間推論(高頻度の制御ループ)には現状不向き。座標を1文字ずつ自己回帰生成するため出力が長く、レイテンシが大きい。
- カメラのみで LiDAR/HD マップを使わないため、正確な距離計測や悪天候・夜間・逆光での頑健性に課題が残る。3次元の絶対距離は単眼系の弱点をそのまま引き継ぐ。
- 数値を文字で綴るため、桁の取り違えなど言語モデル特有の数値出力ミスが起こりうる。専用の回帰ヘッドに比べ数値精度で劣る場面がある。
- 入力フレーム数に上限があり、長い時系列(数十秒の文脈)を扱いにくい。テキスト系列の長さが時間ステップ数に比例して伸び、計算コストが膨らむ。
- 評価は主にオフライン(記録データに対する予測精度、open-loop)で、実車の閉ループ運用(自分の出力が次の入力を変える状況)での長期安全性は今後の課題。open-loop で L2 が小さくても、閉ループで安全とは限らないという指摘(直進バイアスで数値が良く見える問題)も知られている。

研究上の未解決課題としては、基盤モデルの賢さを保ったまま車載で動く速さにする蒸留・軽量化、カメラのみでの3次元理解の限界の克服、長文脈・時系列の扱い、そして閉ループ・長期安全性の保証が挙げられます。

## 発展トピック・研究の最前線
基盤モデルを運転に使う流れは、大きく2方向に分岐しています。1つは EMMA のように単一モデルへ統合する方向で、表現の美しさと正の転移が魅力ですが速度が課題です。もう1つは DriveVLM 的な dual-system(遅い賢いモデル + 速い実行系)で、実用性を優先します。実車適用では後者が現実的とされ、賢いモデルの出力を軽量モデルに蒸留する研究が活発です。

また、VLM の運転推論を測る・学ばせる基盤として Graph VQA([DriveLM (Graph VQA)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0534_drivelm))のようなベンチマークが整備され、EMMA 型モデルの推論能力を体系的に評価できるようになりつつあります。さらに、World Model(世界モデル。次の観測を予測する生成モデル)と組み合わせ、想像した未来で計画を検証する研究や、open-loop の L2 偏重を避けるための閉ループ・反事実評価の整備が進んでいます。EMMA 以後は、生成トークンとして軌道を出す方式と、拡散モデルでマルチモーダルな軌道分布を出す方式のどちらが優れるかという論点も活発に議論されています。
