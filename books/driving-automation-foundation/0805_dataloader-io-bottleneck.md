# 学習のIO/CPUボトルネック

## ひとことで言うと
深層学習の学習では、GPU(計算を担当する装置)に渡すデータを準備する作業と、GPU で実際に計算する作業が、流れ作業のように並行して進みます。ところがデータ準備が重すぎると、GPU は「次のデータがまだ来ない」と待たされ、ほとんど暇な状態(アイドル)になります。これがIO/CPUボトルネックです。動画から毎回1フレームをデコード(展開)するような重い前処理を学習ループの中で行うと、この問題が極端に出ます。学習全体の速度を決めてしまう最大の足かせ(律速段階)になり、高価な GPU を遊ばせる原因になります。

## 直感的な理解
工場のラインを想像してください。一方の作業員(GPU)はものすごく速く部品を組み立てられます。もう一方の作業員(CPU)は、部品を倉庫から運んできて梱包をほどき、組み立て可能な状態にして渡す係です。組み立てが0.1秒で終わるのに、部品の準備に1秒かかるなら、速い作業員は1秒のうち0.9秒を立ったまま待つことになります。

GPU は数十万円〜数百万円する高価な装置で、クラウドでは1時間あたり数百円〜数千円の課金がかかります。その GPU が学習時間の大半を「データ待ち」で遊んでいるなら、お金を捨てているのと同じです。だからこそ「GPU を絶対に遊ばせない」ことが、深層学習システム設計の中心的な目標になります。ここで重要なのは、データ供給と GPU 計算は本来「別の資源」なので同時に動かせる、という点です。同時に動かせるのに片方が遊んでいるなら、それは設計の失敗です。

動画を扱う自動運転やロボティクスの学習で、この問題は特に深刻です。理由は、動画ファイルが強く圧縮されていて、1フレームを取り出すだけでも展開計算が必要だからです。何百GBもの動画から、学習に本当に必要なごく一部のフレームだけを取り出そうとすると、毎回大量の無駄な読み込みとデコードが発生します。

## 基礎: 前提となる概念

まず登場人物を整理します。

- テンソル(tensor): 画像や数値を多次元の数の並びとして表したもの。たとえばカラー画像は「縦×横×3色」の3次元配列で、これがテンソルです。深層学習ライブラリ(PyTorch、TensorFlow など)が扱う基本データ型です。
- バッチ(batch): まとめて一度に処理するデータの一群。たとえば「画像8枚を1バッチ」として GPU に渡します。1枚ずつより、まとめた方が GPU の並列計算を活かせて効率的です。
- CPU と GPU: CPU は汎用の計算装置で、データの読み込みや前処理(画像のリサイズなど)が得意。GPU は同じ計算を大量に並列実行するのが得意で、深層学習の行列計算を担当します。役割分担しています。
- IO(Input/Output、入出力): ディスクやネットワークからデータを読み書きすること。ディスクは CPU や GPU の計算より桁違いに遅いので、ここが詰まりやすい。代表的な目安として、メインメモリへのアクセスは数十ナノ秒、ローカル SSD は数十マイクロ秒、ネットワークストレージは数ミリ秒以上と、桁が階段状に変わります。
- デコード(decode): 圧縮された動画(mp4 など)を、生のピクセル値(各画素の RGB の数値)に展開する処理。動画は容量を小さくするため強く圧縮されているので、1フレーム取り出すにも展開計算が必要です。
- スループット(throughput)とレイテンシ(latency): スループットは「1秒あたり何サンプル処理できるか」(images/sec)。レイテンシは「1サンプルを用意するのにかかる時間」。学習全体の速さを決めるのはスループットですが、その裏で1サンプルごとのレイテンシが積み上がっています。

学習の1ステップは、ざっくり次の繰り返しです。

1. データを用意する(画像をディスクから読み、デコードし、テンソル化し、リサイズなどの前処理をする)
2. GPU に渡して計算する(forward = 順伝播で予測を出す → loss = 誤差を計算 → backward = 逆伝播で勾配を計算 → パラメータ更新)
3. 1に戻る

ここで重要なのは、1の「データ用意」は主に CPU とディスクの仕事、2の「GPU 計算」は GPU の仕事なので、本来は同時に進められるという点です。GPU が今のバッチを計算している裏で、CPU が次のバッチを準備しておけば、GPU は止まらずに走り続けられます。これを「パイプライン化(オーバーラップ)」と呼びます。

問題は、データ用意(1)が GPU 計算(2)より時間がかかるときです。1ステップの所要時間はざっくり `max(T_data, T_compute)` で決まります(完全にオーバーラップできた理想ケース)。`T_data > T_compute` なら GPU は `T_data - T_compute` のあいだアイドルになり、これがそのまま無駄なコストです。逆に GPU 計算の方が長ければ、データ用意は裏で隠れて見えなくなり、ボトルネックにはなりません。つまり「どちらが長いか」が分岐点です。

## 仕組みを詳しく

動画を扱う典型的な学習で、何が無駄になるのかを数値で見ます。

仮に「20秒のクリップ、30FPS(1秒30フレーム)、カメラ7台」という構成を考えます。

- 1台あたりのフレーム数 = 20秒 × 30FPS = 約600フレーム
- 7台ぶん = 600 × 7 = 約4,200フレーム/クリップ

しかし学習で実際に使う有効なサンプル点(予測のタイミング)は、もっと少ないのが普通です。たとえば自車の動き(egomotion)を10Hz に落として、過去・未来の窓を確保すると、1クリップあたり数十点程度しか取れません。仮に72点とすると、

- 本当に必要なのは各サンプル点に対応する1フレームだけ(72点 × 7台 = 504フレーム)
- なのに、ナイーブな実装は1サンプル取り出すたびに動画ファイル全体(数百MB級)をメモリに読み込み、デコーダを作り、目的の1フレームだけ取り出して、残りを全部捨てている
- しかもカメラ7台 × サンプルごとに繰り返す

擬似コードで書くと、ナイーブな `__getitem__`(データセットから1サンプル取り出す関数)はこうなります。

```python
def __getitem__(self, idx):
    cam, frame_idx = self.samples[idx]
    raw = open(cam.video_path, "rb").read()   # 動画全体(数百MB)を毎回メモリに
    reader = VideoReader(raw)                  # デコーダを毎回作る
    frame = reader.get_frame(frame_idx)        # 必要な1枚だけ取り出す
    return preprocess(frame)                   # 残りのデコード結果は全部捨てる
```

ここで起きるコストを段階に分けます。

1. ファイル読み込み(IO): 数百MBのディスク読み込み。1フレームしか使わないのに全体を読む。
2. デコーダ初期化: コンテナ(動画の入れ物)を解析し、ストリーム情報を読む。
3. シーク + デコード: 目的フレームの手前のキーフレームまで戻り、そこから順に展開する(詳細は [動画フレームデコードライブラリ比較](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0902_video-frame-decoding-libraries))。
4. 破棄: 1枚を返した後、デコーダもバッファも捨てる。次のサンプルでまた1から。

before/after で整理すると、

```
before(ナイーブ):  CPU [====動画読み+デコード(遅い)====]  GPU [計算(速い)]待ち...待ち...
after(理想):       CPU [読]CPU[読]CPU[読]  ← 軽い読み込みが先行
                   GPU      [計算][計算][計算]  ← 止まらず連続
```

さらに悪いことに、PyTorch の DataLoader を `num_workers=4`(4つの並列プロセスでデータを用意)で動かすと、4プロセスがそれぞれ動画ファイルを丸ごとメモリに持つので、巨大な動画が同時に複数メモリに居座り、RAM(メインメモリ)も圧迫します。並列化したのに、メモリ不足(OOM)で逆に落ちる、という失敗がよく起きます。加えて、PyTorch のワーカーは `fork` で生成されるため、親プロセスが持つ大きな Python オブジェクト(リストや辞書)が copy-on-write で各ワーカーに共有されますが、Python の参照カウント更新がそのページを書き換えてしまい、結果的にコピーが発生してメモリが膨らむ、という落とし穴もあります。

この問題を診断する目安として、GPU 使用率(`nvidia-smi` の utilization)があります。学習中なのに GPU 使用率が10〜40%をうろうろしていれば、ほぼ確実にデータ供給が律速です。逆に90%以上で張り付いていれば、計算側が律速で、データパイプラインは間に合っています。より厳密には、PyTorch Profiler や TensorFlow の trace viewer で「各ステップの中で GPU が待っている時間(idle gap)」を可視化し、データ待ちかを切り分けます。経験則として、ステップ時間の内訳を `T_data` と `T_compute` に分解し、`T_data / (T_data + T_compute)` が大きいほどデータ律速、と判断します。

根本的な解決策は3方向あります。

- 前処理を軽くする/オフラインに追い出す: 重いデコードを学習前に1回だけ実行し、結果(抽出済みフレーム)を保存しておく。学習時はそれを読むだけにする(詳細は [raw動画とpre-extractedの二段構成](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0870_raw-vs-extracted-dual-mode)、[フレーム保存フォーマットの選択](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0858_frame-storage-format-tradeoffs))。
- 前処理を GPU にオフロードする: デコードやリサイズを CPU でなく GPU で行い、CPU の負担を下げる(NVIDIA DALI など)。
- 結果を再利用(キャッシュ)する: 一度処理したものを使い回し、2エポック目以降のデコードを省く。

## 手法の系譜と主要論文

このボトルネックは特定の論文の発見というより、深層学習システム工学で繰り返し再発見されてきた「input pipeline bottleneck(入力パイプラインの律速)」です。各陣営がどう取り組んできたかを系譜として並べます。

- TensorFlow の `tf.data`(Google, 2017年頃〜): 「prefetch(先読み)」「parallel map(並列前処理)」「cache(結果の再利用)」という3本柱でボトルネックを潰す設計を打ち出しました。Murray らの論文 "tf.data: A Machine Learning Data Processing Framework"(VLDB 2021, arXiv:2101.12127)は、入力パイプラインを独立した最適化対象として扱い、自動チューニング(AUTOTUNE)で並列度や先読み量を動的に決める仕組みを提案しています。
- NVIDIA DALI(Data Loading Library, 2018年〜): CPU 前処理が GPU 学習の律速になる問題に対し、JPEG デコードやリサイズ、データ拡張を GPU 上で実行することで CPU を解放する解決策を提示しました。動画デコードの GPU オフロード(nvJPEG, NVDEC 利用)も提供します。
- PyTorch の `DataLoader`(2017年〜): `num_workers`(並列ワーカー数)、`pin_memory`(CPU→GPU 転送を速くする固定メモリ確保)、`prefetch_factor`(各ワーカーの先読み量)、`persistent_workers`(ワーカーをエポック間で使い回す)といった調整パラメータを用意しました。裏を返せば、これらが必要になるほどデータ用意は重くなりがちだ、という事実の表れです。
- DeepIO / DS-Analyzer 系の分析研究(Kakaraparthy et al. 2019; Mohan et al., "Analyzing and Mitigating Data Stalls in DNN Training", VLDB 2021, いわゆる CoorDL): 学習が「データストール(data stall、データ待ちで GPU が止まる現象)」でどれだけ遅くなるかを体系的に測定し、ストレージ・キャッシュ・デコードのどこが律速かを切り分ける分析ツールと、複数ジョブでキャッシュを協調させる手法を提案しました。「データ供給は学習システムの一級市民である」という認識を定着させた研究群です。
- WebDataset / AIStore(Aizman, Maltby, Breuel, "High Performance I/O For Large Scale Deep Learning", IEEE Big Data 2019, arXiv:2001.01858): 「大量の小さなファイルを tar アーカイブ(shard)に束ね、順次読み込みでディスク/ネットワーク帯域をフル活用する」設計を提唱。ランダムアクセスの遅さと小ファイル爆発の両方を解消します(詳細は [フレーム保存フォーマットの選択](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0858_frame-storage-format-tradeoffs))。
- FFCV(Leclerc et al., MIT, 2023, "FFCV: Accelerating Training by Removing Data Bottlenecks", CVPR 2023): データを専用バイナリ形式に変換し、メモリマップ + JIT 前処理で ImageNet 学習を劇的に高速化。ResNet-50 を1 GPU で数十分、という結果を示し、「データローダの最適化だけでこれほど速くなる」ことを定量的に証明しました。

これらに共通する教訓は「重い前処理は学習ループの外(オフライン)で一度だけやって結果を保存しておく」「学習時はできるだけ軽い順次読み込みだけにする」という点に尽きます。

## 論文の実験結果(定量データ)

数値で効果を見ます。指標は主に「スループット(throughput、1秒あたり処理サンプル数 images/sec)」と「GPU 使用率(%)」、そして「総学習時間」です。

- FFCV(CVPR 2023)の報告: ImageNet(約128万枚の画像分類データセット)で、ResNet を学習する際に、PyTorch 標準の DataLoader + JPEG ディスク読み込みに比べ、FFCV のデータ経路に置き換えるだけで学習時間が大幅に短縮されたと報告。著者らは ResNet-18 を1枚の A100 GPU で約30分以下、ResNet-50 を約1時間程度で目標精度(おおむね top-1 75〜78%)に到達させた構成を公開しています。ここでの肝は「モデルもハイパーパラメータも同じで、データ供給を速くしただけ」で総時間が短縮した点です。つまり差は純粋にデータパイプラインに起因します。top-1 accuracy(上位1個の予測が正解と一致する割合)は ImageNet の標準指標で、ここが同じまま時間だけ縮むのが理想です。
- DALI のベンチマーク(NVIDIA 公式): 画像分類学習で CPU 前処理を GPU にオフロードすると、CPU が律速だったケースで GPU 使用率が数十%から90%超へ改善し、スループットが数倍になる例が示されています。とくに高解像度画像や強いデータ拡張を使う設定で効果が大きい。逆に、もともと GPU 計算が律速の小さいモデルでは効果が出にくく、「律速がどちらか」で恩恵が決まる、というのは重要な含意です。
- CoorDL / DS-Analyzer(VLDB 2021)のアブレーション: データストールを除去するとエポック時間が大幅に短縮されることを、ストレージ・デコード・キャッシュの各要素ごとに分解して示しました。とくに「データセットがメモリに乗り切らないとき、毎エポック再読み込みとデコードが発生して GPU が待つ」現象を定量化し、キャッシュ協調によってマルチジョブ環境のスループットが改善することを報告しています。
- tf.data 論文(VLDB 2021)のアブレーション的分析: prefetch を抜くと GPU が各ステップでデータを待つようになり、スループットが落ちる。parallel map の並列度を上げると CPU バウンドの前処理が隠れる。cache を効かせると2エポック目以降が劇的に速くなる、という形で、各要素の寄与が分解されています。「どれを抜くとどう悪化するか」が明確で、3本柱がそれぞれ独立に効くことが分かります。

数値の意味を補足します。スループットが2倍になれば、同じデータセットを同じエポック数学習する時間が半分になります。クラウド GPU が1時間1,000円なら、48時間の学習が24時間に縮めば、それだけで数万円の節約です。GPU 使用率が30%から90%になるというのは、3倍ぶんの計算を同じハードで回せるようになる、という意味です。

動画データに固有の注意として、動画デコードは画像の JPEG デコードよりさらに重い(キーフレームからの逐次展開が必要)ため、「学習ループ内での動画デコード」は上記の画像ベンチマークよりさらに律速になりやすい、という点があります。実務報告では、動画をその場でデコードする経路から事前抽出フレームの順次読み込みに変えるだけで、GPU 使用率が一桁%から80%超に跳ね上がる例が珍しくありません。これは「同じ GPU で8倍以上の計算を回せる」ことに等しく、データパイプラインの設計が学習コストを一桁動かしうることを示します。

## メリット・トレードオフ・限界

データ供給を「学習ループ内で重い前処理」のまま放置するか、「軽い経路に作り変える」かのトレードオフを整理します。

raw データをその場で処理する方式のメリット:

- 前処理ステップ不要。データさえあればすぐ学習を回せる(スモークテスト=最小動作確認に向く)
- 中間ファイルを作らないのでディスク容量を追加で使わない
- データの差し替えが即座に反映される(キャッシュの再生成が要らない)

raw 方式のデメリット(=ボトルネック):

- 1サンプルごとに重いデコードを繰り返すため極端に遅い
- 並列ワーカーを増やすとメモリ使用量が爆発する(各ワーカーがデコーダや動画を保持)
- GPU 使用率が低く、学習時間が何倍にも延びる

事前抽出/最適化キャッシュ方式のメリット:

- 学習時はデコード不要、必要な1フレームを直接読むだけなので GPU を遊ばせない
- 順次読み込み(シーケンシャル read)はランダムシークより圧倒的に速い

事前抽出方式のデメリット(トレードオフ):

- 学習前に1回、全フレームを抽出する前処理時間とディスク容量が要る
- データを追加・差し替えするたびに、その分の再抽出が必要(差分対応の設計が要る)
- 抽出時の解像度やフォーマットを固定するので、後から「やっぱり高解像度で」が効きにくい

研究上の未解決課題としては、(1)前処理とモデルが変わるたびにキャッシュを作り直すコストをどう抑えるか(差分キャッシュ、オンライン拡張との両立)、(2)データ拡張(augmentation)はランダム性が要るのでキャッシュしにくく、GPU オフロードとの併用が必要になる、(3)複数 GPU・複数ノードでデータを共有するときのストレージ帯域とネットワークの設計、などが挙げられます。とくに大規模分散学習では、データローダがノードをまたいでストレージを叩くため、単一ノードでは見えなかった帯域の壁が現れます。たとえば 8 ノード × 8 GPU で同一データセットを同時に読むと、ストレージへの読み込み要求が64倍に集中し、ストレージ帯域が新たな律速になります。

## 発展トピック・研究の最前線

- データ供給と計算のオーバーラップを自動最適化する研究(tf.data の AUTOTUNE、PyTorch の各種プロファイラ)。学習中にプロファイルを取り、ボトルネックがデータ側か計算側かを自動判定するツールが整ってきています。
- GPU デコード(NVDEC)を学習ループに直結する研究。動画フレームを GPU 上で直接デコードし、CPU↔GPU 転送すら省く方向。TorchCodec の GPU デコードや DALI の動画リーダーがこの系統です([動画フレームデコードライブラリ比較](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0902_video-frame-decoding-libraries))。
- ストレージ階層の活用: 高速なローカル NVMe SSD にホットなデータを置き、コールドなデータはオブジェクトストレージ(S3 など)から非同期にプリフェッチする多階層キャッシュ。
- データ前処理の分離・共有: 複数の学習ジョブが同じ前処理を重複実行しないよう、前処理を専用サービス化してキャッシュを共有する研究(Google の tf.data service、Cachew などの「データエコー」「分離型前処理」)。同じデータを何度も使う sweep(ハイパーパラメータ探索)で特に効きます。
- 大規模言語モデルや動画基盤モデルの学習で、データパイプラインがペタバイト級になり、専用のデータ前処理クラスタを学習クラスタと分離する設計が一般化しつつあります。学習用 GPU クラスタと前処理用 CPU クラスタを別々にスケールさせ、両者をストリーミングで結合する構成が主流になりつつあります。
