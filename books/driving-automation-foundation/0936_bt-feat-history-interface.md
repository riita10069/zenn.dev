# [B,T,feat] 履歴インターフェース

## ひとことで言うと

過去の何フレーム分かの特徴量を、「バッチ × 時間 × 特徴」という決まった形のテンソル（多次元配列）で受け渡しする共通ルールです。どの時系列メモリ手法もこの同じ形で入出力するので、手法を差し替えても周りのコードを変えずに比較できます。アーキテクチャ探索を効率よく進めるための、地味だが本質的な「契約（インターフェース規約）」です。

## 直感的な理解

電源タップを思い浮かべてください。掃除機でも扇風機でも充電器でも、プラグの形（コンセントの規格）が同じだから、どの家電も同じタップに挿して使えます。もしメーカーごとにプラグの形が違ったら、家電を買い替えるたびにコンセント工事が必要になります。

機械学習で「過去の情報をどう要約するか」を扱う部品（時系列メモリ）にも、これとまったく同じ問題があります。要約のやり方は何通りもあります。最新の 1 フレームだけ使う、過去を間引いて RNN で圧縮する、Transformer の注意機構で過去全体を見る、状態空間モデルで線形に流す、など。これらを実験で比較したいのですが、手法ごとに「入力の形」「出力の形」がバラバラだと、差し替えるたびに本体コードを書き直す羽目になります。家電を替えるたびにコンセント工事をするのと同じです。

そこで「過去の特徴はこの形で渡す、要約結果はこの形で返す」という共通のプラグ形状を一つ決めておきます。それが `[B, T, feat]` 履歴インターフェースです。この形さえ守れば、中身（要約アルゴリズム）を自由に差し替えられます。研究では「どの集約器が運転に効くか」を何十通りも試すので、この共通プラグの有無が実験速度を決定づけます。

## 基礎: 前提となる概念

まず「なぜ過去が要るのか」。自動運転では「今の 1 枚の画像」だけでは判断できない情報が多くあります。前の車が減速し始めているか、歩行者がこちらに近づいているか遠ざかっているか。こうした速度・意図は、複数フレームを時間方向に並べて初めて読めます。単一フレームからは「止まっている車」と「動き出した車」の区別がつきません。物理的に言えば、位置は 1 フレームで分かっても、速度は位置の時間差分なので最低 2 フレーム、加速度は 3 フレーム必要、という具合です。そこで過去フレームの特徴を溜め、それを要約してプランナに渡す部品、時系列メモリ（temporal memory）を入れます。

次にテンソルの各次元の意味です。テンソルとは数値を格子状に並べた多次元配列で、`[B, T, feat]` は 3 次元のテンソルです。

- B = batch（バッチ）: 同時に処理する場面の数。たとえば 8。GPU は複数を並列に処理する方が効率的なので、ひとまとめにします。
- T = time（時間）: 過去フレーム数。たとえば 10 Hz で 6.4 秒分なら 64。約束として「古い順 → 新しい順」に並べます。`[:, -1]` が最新フレームです。この並び順は契約の一部で、逆に並べると後述のとおり静かに壊れます。
- feat = feature（特徴）: 各フレームの特徴ベクトルの長さ。視覚特徴なら 896、自車運動特徴なら 256、というように系統ごとに違ってよいです。

抽象基底クラス（abstract base class）も前提です。これは「この種類の部品はこう呼べる」という型枠だけを定め、実装を持たないクラスです。すべての時系列メモリ実装がこの基底クラスを継承し、同じ forward シグネチャを共有することで、差し替え可能性（interchangeability）を担保します。プログラミングではこれを Liskov の置換原則（基底型を期待する場所に、どの派生型を入れても正しく動くべき）とも呼びます。

レジストリ（registry）パターンも登場します。これは「名前 → 実装」を引ける辞書で、`build("one_hz", ...)` のように文字列キーで部品を生成します。新手法は辞書に 1 行登録するだけで使えるようになり、設定ファイルから文字列で手法を選べるようになります。

## 仕組みを詳しく

契約を定義する基底クラスは、典型的にはこう書かれます。

```python
class BaseTemporalMemory(nn.Module, ABC):
    @abstractmethod
    def forward(self, visual_history, egomotion_history, **kwargs):
        """
        visual_history:    [B, T, visual_dim]  または [B, visual_dim]
        egomotion_history: [B, T, egomotion_dim] または [B, egomotion_dim]
        Returns:
            visual_context:    [B, visual_dim]
            egomotion_context: [B, egomotion_dim]
        """
```

入力は 2 系統です。視覚の履歴 `[B, T, 896]` と、自車運動の履歴 `[B, T, 256]`。出力は時間方向を畳んだ文脈ベクトルで、`visual_context` が `[B, 896]`、`egomotion_context` が `[B, 256]`。つまり「時間軸 T を要約して消す」のがメモリの仕事です。`[B, T, 896]` → `[B, 896]` という次元の縮約が起きます。視覚と自車運動を別々の系統で扱うのは、両者が別の意味（外界の見えと、自車の動き）を持ち、適切な次元数も異なるからです。

この契約には設計上の判断が二つ織り込まれています。

第一に、時間次元を持たない `[B, feat]` も受け付けるというフォールバック（代替動作）を契約に含めること。推論の最初など、まだ履歴が溜まっていない呼び出しでも壊れないようにするためです。実装側は入力テンソルの次元数（`ndim`）で分岐します。車が起動した直後の数フレームは過去が存在しないので、このフォールバックがないと実運用で即座に落ちます。

第二に、末尾の `**kwargs`。追加情報を後方互換に渡せる受け皿で、I/F を凍結したまま新しい文脈を流せます（[kwargs 経由のオプショナル入力でプランナ I/F を不変に保つ設計](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0993_optional-producer-kwargs-planner-interface) と同じ拡張規約）。

具体的な手法ごとの振る舞いを、契約に載せた例で示します。

最も単純な「最新フレームだけ使う」手法は、パラメータを持たない素通し（passthrough）です。

```python
v_ctx = visual_history[:, -1] if visual_history.ndim == 3 else visual_history
```

`[B, T, 896]` から `[:, -1]` で最後（最新）のフレームを取り、`[B, 896]` を返す。時間情報は捨てます。これがデフォルトのベースラインになります。新しい時系列メモリ手法は、このベースラインに勝てて初めて意味があります。

「間引いて RNN で要約する」手法（[1Hz 履歴エンコーダ](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0992_one-hz-history-encoder) 参照）は、視覚と自車運動を特徴次元で連結して `[B, T, 1152]` を作り、10 Hz → 1 Hz に圧縮して GRU で要約し、再び 896 と 256 に分けて返します。

「注意機構で履歴の列を見る」手法（BEVFormer 風）は、固定長の履歴キューに対して時間方向の self-attention をかけます。いずれも同じ `[B, T, feat]` 入力・`[B, feat]` 出力を守るので、レジストリのキーを差し替えるだけで交換できます。

```python
TEMPORAL_MEMORY_REGISTRY = {
    "no_memory": NoMemory,
    "one_hz":    OneHzHistoryEncoder,
    "attention": AttentionMemory,
}
mem = TEMPORAL_MEMORY_REGISTRY["one_hz"](...)  # 名前で生成
```

新手法を足すときは、基底クラスを継承して同じ契約を守り、レジストリに 1 行足すだけ。本体のモデルも学習コードも一切変えずに済みます。これが「契約を固定する」最大の見返りです。重要なのは、この契約が型システムで守れる部分（テンソルの次元数）と、守れない部分（「古い順 → 新しい順」「視覚は 896 次元」といった意味的約束）に分かれることです。後者を破ると例外は出ず、性能だけが静かに劣化する、という厄介な失敗が起きます。

## 手法の系譜と主要論文

「過去フレームの特徴系列を時間方向に集約してから判断する」枠組みは、運転モデルで広く使われてきました。系譜を辿ります。

Hu et al. "FIERY: Future Instance Prediction in Bird's-Eye View from Surround Monocular Cameras"（2021, ICCV, arXiv:2104.10490）は、複数時刻の BEV（鳥瞰）特徴を時間方向に集約し、未来のインスタンス（個々の車や人）の動きを予測しました。動機は、単一フレームでは速度や意図が読めないこと。新規性は、サラウンドカメラだけから（LiDAR なしで）時間集約による確率的な未来予測を行った点です。トレードオフは、メモリと計算が時間長 T に比例して増えることです。

Li et al. "BEVFormer: Learning Bird's-Eye-View Representation from Multi-Camera Images via Spatiotemporal Transformers"（2022, ECCV, arXiv:2203.17270）は、固定長の履歴キュー（recent history queue）に対して temporal self-attention をかけ、過去の BEV を現在のフレームに整列させて参照しました。動機は、順序を意識した注意で動きの手がかりを取り込むこと。新規性は、空間方向の deformable attention（参照点の周りの数点だけに注意を集中させる効率的な注意）と時間方向の self-attention を組み合わせた spatiotemporal transformer です。トレードオフは attention の計算コスト（系列長に対して敏感、原理的には二乗オーダー）です。注意機構ベースの時系列メモリは、この系譜の上に立ちます。

抽象的には、`[B, T, feat]` は系列モデリングの標準的なテンソル配置です。RNN/GRU（Cho et al. "Learning Phrase Representations using RNN Encoder-Decoder", 2014, EMNLP, arXiv:1406.1078）や Transformer（Vaswani et al. "Attention Is All You Need", 2017, NeurIPS, arXiv:1706.03762）が前提とする「バッチ・系列長・特徴」の並びそのものです。近年は状態空間モデル Mamba（Gu & Dao, "Mamba: Linear-Time Sequence Modeling with Selective State Spaces", 2023, arXiv:2312.00752）も同じ `[B, T, feat]` 配置を入出力に採り、線形計算量で長系列を扱う候補として加わりました。研究コミュニティがこの形を共通言語として採用しているおかげで、要約器（メモリ手法）を中身だけ差し替えられる、という恩恵が生まれます。この「共通テンソル契約による部品の差し替え可能性」は、PyTorch の `nn.Module` や HuggingFace Transformers の出力規約など、現代の深層学習エコシステム全体に通底する設計思想です。

## 論文の実験結果（定量データ）

FIERY（Hu et al. 2021）は nuScenes と Lyft データセットで評価しました。指標は将来のインスタンス予測の品質を測る IoU（Intersection over Union、予測領域と正解領域の重なりの割合。1 に近いほど良い）と、動きの一貫性を測る PQ（Panoptic Quality、検出と追跡を統合した品質指標）系の指標です。FIERY は、時間集約を行わない単一フレームのベースラインや、当時の LiDAR ベース手法と比較して、カメラのみで競争力ある未来予測 IoU を達成しました。アブレーションとして、過去フレームの時間集約を外すと未来予測の精度が明確に低下することが示され、「時間方向の情報が予測に効く」ことを定量的に裏付けました。

BEVFormer（Li et al. 2022）は nuScenes の 3D 物体検出で評価しました。主指標は NDS（nuScenes Detection Score、検出の総合スコア。位置・大きさ・向き・速度・属性の誤差を統合する。大きいほど良い）と mAP（mean Average Precision、平均適合率。大きいほど良い）です。BEVFormer は当時のカメラのみ手法で最高水準の NDS（およそ 0.56）を達成しました。最も重要なアブレーションは時間方向の self-attention の有無です。temporal self-attention を入れると、特に速度推定の指標 mAVE（mean Average Velocity Error、平均速度誤差。小さいほど良い）が大きく改善し、移動物体の速度誤差が顕著に下がりました。これは「単一フレームでは速度が読めず、過去を見て初めて分かる」という直感を数値で示したもので、時系列メモリの存在意義そのものを定量化しています。速度は位置の時間微分なので、原理的に複数フレームなしには推定不能であり、この改善は理論とも整合します。

これらが示す共通の教訓は、過去フレームの集約は特に「動き・速度・意図」の理解で効く、ということです。`[B, T, feat]` 契約は、そうした効果を持つ多様な集約手法を、公平に交換・比較できる土台を提供します。逆に言えば、静止物体の検出のように時間が効きにくいタスクでは集約の恩恵が小さく、契約の柔軟性（時間なしフォールバック）が活きます。

## メリット・トレードオフ・限界

メリット。入出力の形を固定したので、メモリ手法をレジストリのキー差し替えだけで交換・比較できます。アーキテクチャ探索（どの集約器が運転に効くか）を、本体を触らずに回せます。時間次元なし `[B, feat]` も受け付けるフォールバックにより、履歴がない初期フレームでも壊れません。視覚と自車運動を別々の系統で扱えるので、それぞれ適切な次元で文脈を返せます。

トレードオフと限界。第一に「文脈は `[B, feat]` の 1 本に畳む」という契約上、時間方向の細かい構造を出力に残せません。要約の過程で情報が落ちます。もし下流が時刻ごとの文脈を必要とするなら（たとえば各未来時刻に対応する過去文脈を使いたいなら）、この契約自体が制約になります。第二に、T が大きいほどメモリ・計算が増え、入力テンソルが重くなります。長い履歴を扱いたいほど、この契約をそのまま使うのが苦しくなります。第三に、「古い順 → 新しい順」「視覚 896・自車運動 256」といった暗黙の約束を破ると、エラーにならず静かに誤動作します。契約の意味的な部分（次元の並び順や各次元の中身）は型システムでは守れず、規律とドキュメントに依存します。

研究上の論点として、「要約して時間を消す」契約と「時刻ごとの文脈を保持する」契約のどちらを採るべきかがあります。前者は下流が単純になり計算が軽い一方、時間構造を捨てます。後者は表現力が高い一方、下流のプランナも時間軸を扱える必要があります。最適な粒度はタスク依存で、一般解はありません。実務では、まず最も単純な「最新フレームのみ」をベースラインとして固定し、より複雑な集約器がそれを有意に上回るかを closed-loop で検証する、という保守的な進め方が安全です。

## 発展トピック・研究の最前線

時間集約の手法は、RNN/GRU から attention、さらに状態空間モデル（State Space Model, 例: Mamba）へと広がっています。状態空間モデルは長系列を線形計算量で扱えるとされ、長い運転履歴の効率的な集約に向くと期待されています。Transformer が系列長の二乗に比例するコストを払うのに対し、Mamba 系は線形なので、数百フレームの長期履歴を現実的な計算量で集約できる可能性があります。また、契約の出力を「1 本のベクトル」ではなく「圧縮された短い系列」や「メモリスロットの集合」にする設計（例: Perceiver 系の latent array、固定数の潜在トークンに情報を蒸留する仕組み）も研究されており、これは「時間構造をどこまで残すか」のトレードオフを連続的に調整する試みと見なせます。実運用では、こうした多様な集約器を同一契約の下で公平にベンチマークし、open-loop（記録データでの予測誤差評価）と closed-loop（シミュレータで実際に走らせる評価）の両面で「運転に効くか」を確かめるのが定石です。open-loop で良くても closed-loop で悪化する例（記録データの軌跡を真似るだけで因果を学んでいない場合など）が知られており、契約の良し悪しは最終的に走行で判定する必要があります。
