# raw動画とpre-extractedの二段構成

## ひとことで言うと
データの読み込み経路を用途で使い分ける設計です。動作をちょっと確かめたいだけのスモークテスト(最小動作確認)では「raw 動画をその場でデコード」する手軽な経路を使い、本番の長時間学習では「事前に抽出して保存したフレーム(pre-extracted)」を読む高速な経路を使います。手軽さと速さという相反する要求を、片方だけ選ぶのでなく両方用意して切り替えることで満たします。鍵は「両経路が同じ形のデータを返す」ことです。

## 直感的な理解
料理に例えます。レシピが正しく動くか味見したいだけなら、その場で野菜を洗って切って(raw 動画をデコード)少量作るのが手軽です。下準備の作り置き(フレーム抽出)を毎回強制されたら、ちょっと試したいだけの人が面倒で離れてしまいます。一方、毎日大量に提供する本番営業では、その場で全部の野菜を切っていたら回りません。前日に切って冷蔵(事前抽出してディスクに保存)しておき、当日は盛り付けるだけ(読むだけ)にします。

ソフトウェア開発でも、この2つの立場の人がいます。新しく参加した貢献者が「自分の変更でコードが壊れていないか」を確かめたいとき、重い前処理(時間とディスクを食う)を強制されると、参入のハードルが上がります。一方、GPU クラスタで何日も学習を回す本番では、データ供給が律速になれば([学習のIO/CPUボトルネック](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0805_dataloader-io-bottleneck))高価な GPU を遊ばせることになります。求めるものが「正しさの確認」か「効率」かで、最適なデータ経路が逆向きなのです。

二段構成(dual-mode)は、この相反する要求を「同じデータセットに対して2つの読み込み経路を用意し、用途で切り替える」ことで両立させる設計パターンです。

## 基礎: 前提となる概念

- スモークテスト(smoke test): 装置の電源を入れて煙が出ないか見る、という語源の言葉。ソフトでは「最小限の動作確認(致命的な壊れがないか)」を指します。網羅的なテストではなく、「とりあえず起動して一周動くか」を見る軽いチェックです。CI(継続的インテグレーション、コミットのたびに自動でテストする仕組み)でよく使われます。
- 事前抽出(pre-extraction): 学習で使うフレームだけを、学習前に1回まとめて動画から抜き出し、ディスクに保存しておくこと。学習時はそれを読むだけにする([フレーム保存フォーマットの選択](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0858_frame-storage-format-tradeoffs))。
- インターフェース(interface): 「どんな形のデータを受け取るか」という約束ごと。2つの経路が同じ形のデータ(同じ shape のテンソル、同じキー名、同じ正規化)を返すようにすれば、学習コードはどちらの経路かを意識せずに使えます。これが二段構成を成立させる鍵です。
- スループット(throughput): 単位時間あたりの処理量。学習なら1秒あたり何サンプル処理できるか。本番ではこれを最大化したい。
- 技術的負債(technical debt): 「今は楽だが後で利子付きで返すことになる設計の借金」(Sculley et al., "Hidden Technical Debt in Machine Learning Systems", NeurIPS 2015 が機械学習特有の負債を整理)。二段構成は便利ですが「2経路を保守する」負債を背負う、というのが本質的なトレードオフです。

二段構成の根底にあるのは、機械学習システム工学で確立した原則です。「学習に必要なのは最終形のテンソルだけ。生データから最終形への変換は、再現可能なオフラインステップに分離し、学習はキャッシュを読むだけにする」。二段構成は、この原則に「開発時の手軽さ」を両立させた折衷案だと言えます。Sculley らが指摘した「データ依存の負債」「パイプラインの密結合」を避けつつ、開発体験(DX, developer experience)を守る工夫です。

## 仕組みを詳しく

2つの経路を具体的に対比します。

経路A: raw 動画モード(手軽・遅い)

- 流れ: 各サンプル取得時に動画ファイルを読み、デコーダを作り、目的のフレームを1枚だけデコードしてテンソル化する。
- 前処理: 不要(動画ファイルがあればそのまま使える)
- 速度: 遅い(動画読み込み + デコードを毎サンプル繰り返す)
- 用途: ローカルや CI でのスモークテスト、少数クリップだけ持っている開発者

経路B: pre-extracted モード(高速・要前処理)

- 前提: オフライン前処理で、有効サンプル点に対応するフレームだけを抽出し、JPEG として tar shard(WebDataset)に束ねてオブジェクトストレージに保存し、学習ノードのローカルディスクへ同期しておく。
- 流れ: shard を順次(シーケンシャル)に読み、JPEG をテンソル化、数値配列を numpy 復元する。動画デコードは一切しない。
- 速度: 速い(ディスク帯域いっぱいの順次読み込み)
- 用途: GPU クラスタでの本番長時間学習

ここで決定的に重要なのが「両経路の出力を同じ形に揃える」ことです。マルチカメラの軌道予測モデルを例にすると、バッチ(まとめて処理する一群、B はバッチサイズ)はおおむね次の形です。

```
batch["visual_tiles"]       (B, 7, 3, 256, 256)   # 7カメラ × RGB × 256×256
batch["egomotion_history"]  (B, 256)              # 過去の自車状態
batch["visual_history"]     (B, 896)              # 視覚特徴の履歴
batch["trajectory_target"]  (B, 128)              # 未来の目標(加速度・曲率など)
```

経路Aと経路Bが、このまったく同じ shape・同じキー名・同じ dtype・同じ正規化のバッチを返すように作れば、学習スクリプト側は「どちらの経路か」をほぼ意識せずに使えます。経路を切り替えるのは設定フラグ1つ、という形が理想です。共通のデータ整形ロジック(リサイズ、正規化、キー命名)を両経路から呼ぶ1つの関数に集約しておくと、ズレが生まれにくくなります。

ASCII 図でデータフローを描くと、

```
[動画ファイル]
     |
     +--(経路A: その場でデコード)----------------> [テンソル] --+
     |                                                          |
     +--(オフライン抽出)--> [tar shard on S3]                   +--> [モデル]
                                  |                             |
                                  +--(経路B: 同期して順次読み)--> [テンソル] --+
```

切り替えの考え方:

- 開発者がローカルや CI で「コードが通るか」を見たい → 経路A(前処理ゼロ)
- 本番の GPU クラスタで長時間学習する → 経路B(事前抽出)
- 一部クリップしか持っていない貢献者向けに、前処理を「差分(incremental)」対応にする論点もあります(まだ取り込んだクリップぶんだけ抽出する)。

数値整合性の担保: 経路Aと経路Bは、JPEG 圧縮の有無やリサイズ補間方式の差で、ピクセル値が完全一致しないことがあります。研究では「経路を変えたら精度が変わった」という再現性事故を避けたいので、同一サンプルを両経路で取り出してテンソルの差(最大絶対誤差など)を測る整合性テストを CI に組み込むのが望ましい運用です。許容誤差をどこに置くかは、タスクの感度しだいです。

## 手法の系譜と主要論文

これは特定論文の手法ではなく、機械学習システム工学で確立した「オフライン前処理 + キャッシュ」のパターンが、開発体験(DX)の要求と合流したものです。

- TensorFlow `tf.data` performance guide(Google, 2017年頃〜): `cache()`(一度処理した結果を保存して再利用)と、重い変換をループ外へ出す考え方を一次情報として推奨。二段構成は、この「キャッシュ」を学習前の独立ステップにまで昇格させたものと言えます。
- Sculley et al., "Hidden Technical Debt in Machine Learning Systems"(NeurIPS 2015): データ準備パイプラインが密結合・重複しがちで負債になることを警告。二段構成のような「経路の分離」は、この負債を意識した設計判断であり、同時に「2経路保守」という別の負債も生む、という両面を理解する土台です。
- Murray et al., "tf.data: A Machine Learning Data Processing Framework"(VLDB 2021, arXiv:2101.12127): 入力パイプラインを独立した最適化対象として体系化。前処理の重さに応じてキャッシュ・並列化・先読みを使い分ける思想を理論立てました。
- WebDataset(Aizman et al., IEEE Big Data 2019, arXiv:2001.01858): 「前処理済みデータを shard から順次読む」高速経路を提示。pre-extracted モードはこれに直接対応します。
- PyTorch DataLoader ドキュメント: `num_workers` や `prefetch_factor` による緩和策と、重い前処理はデータセット構築時に済ませる推奨。
- MLOps / 機械学習システム設計の一般原則(各種教科書・実務資料): 「学習はキャッシュを読むだけ、変換は再現可能なオフラインステップ」。データバージョニング(DVC, LakeFS)と組み合わせ、どの前処理で作ったデータかを追跡可能にする流れにつながります。

系譜の流れは「重い変換はループ外へ(tf.data の cache) → それを独立ステップに昇格(オフライン前処理パイプライン) → 高速読み込み形式と結合(WebDataset) → 開発時の手軽さも両立(二段構成)」と整理できます。

## 論文の実験結果(定量データ)

二段構成そのものを評価した論文はありませんが、構成要素の効果は定量化されています。

- キャッシュの効果(tf.data, VLDB 2021): 重い前処理をキャッシュすると、2エポック目以降のステップ時間が前処理コスト分だけ短縮されます。前処理がステップの大半を占めるケースでは、キャッシュ後のスループットが数倍になる例が示されています。これは「経路Bが経路Aより速い」根拠そのものです。
- 順次読み込みの効果(WebDataset, IEEE Big Data 2019): 個別ファイルアクセスを shard 順次読みに変えると、オブジェクトストレージからの学習でストレージ帯域をほぼ理論限界まで引き出せる。経路Bが本番でスループットを稼げる理由です。
- 動画デコード回避の実務報告: 動画をその場でデコードする経路(経路A相当)から事前抽出フレームの順次読み込み(経路B相当)に変えると、GPU 使用率が一桁%から80%超に跳ね上がる例が、自動運転・動画学習の実務で報告されています。GPU 使用率が10%から80%になるとは、同じハードで約8倍の計算を回せるという意味で、学習時間と GPU コストが大幅に削減されます。

数値の意味: 経路A(手軽だが遅い)は「正しさの確認」には十分で、ここでは速度を問いません。スモークテストは数サンプル通れば目的を達するので、GPU 使用率が低くても問題になりません。経路B(速いが要前処理)は「本番の効率」を担い、ここで GPU 使用率とスループットが効きます。役割を分けることで、どちらの数値も最適化できる、というのが二段構成の定量的な狙いです。1つの経路で両方を満たそうとすると「手軽さのために遅い」か「速さのために前処理必須」のどちらかに倒れ、片方の用途が犠牲になります。

## メリット・トレードオフ・限界

メリット:

- 開発者は前処理なしで即座に動作確認できる(参入障壁が低い)
- 本番は GPU を遊ばせない高速経路で回せる(コスト・時間を節約)
- 出力インターフェースを揃えることで、学習コードは経路を意識せず使える
- CI で経路Aを使えば、前処理パイプラインを動かさずに学習コードの回帰テストができる(テストが速く・軽くなる)

トレードオフと限界:

- 2つの経路を保守する必要があり、出力形状や前処理のズレを防ぐ努力が要る(片方だけ直して不整合、を避ける)。これが二段構成の最大の技術的負債です。Sculley らの言う「パイプラインの分岐」が増えるほど保守コストが上がります。
- 経路Aと経路Bでデータの数値が完全一致する保証を作るのは手間(JPEG 圧縮の劣化、リサイズ補間方式の差などで微差が出うる)。研究で「経路を変えたら精度が変わった」を避けるには、両経路の数値整合性テストが望ましい。
- 差分前処理(一部クリップだけ抽出)に対応するには、マニフェスト(どのサンプルがどの shard にあるかの対応表)の設計が要る。
- データ拡張のランダム性は経路Bのキャッシュと相性が悪い。素の画像をキャッシュし、拡張は学習時(できれば GPU)で行うハイブリッドが必要になります。
- 経路の選択ミスのリスク: 本番でうっかり経路A(遅い)を選ぶ、CI でうっかり経路B(前処理必須で重い)を選ぶ、といった設定ミスが起こりうる。デフォルトの安全側を決めておく設計判断が要ります。

研究上の未解決課題としては、(1)前処理パイプラインとモデル入力仕様が変わるたびに経路Bを作り直すコストの最小化(差分・増分キャッシュ)、(2)2経路の数値整合性を自動検証する仕組み、(3)経路の自動選択(データの所在やサイズを見て、抽出済みなら経路B、未抽出なら経路A、と透過的に切り替える)などがあります。とくに(3)は「保守する経路は2つでも、利用者からは1つに見える」ことを目指すもので、二段構成の理想形です。

## 発展トピック・研究の最前線

- パイプラインオーケストレーション: 前処理(抽出)→ 学習 → 評価を1つの DAG(有向非巡回グラフ)として記述し、データのハッシュが変わったときだけ再抽出する仕組み(Flyte, Kubeflow Pipelines, Airflow など)。二段構成の経路Bを自動再生成する基盤になります。入力(動画やコード)のハッシュをキーにしてキャッシュヒット判定するので、変わっていない部分は再計算しません。
- データバージョニング: 抽出済みデータセットをバージョン管理し、「どの生データ・どの前処理・どの解像度で作ったか」を再現可能にする(LakeFS, DVC)。経路Bのキャッシュの再現性を担保します。
- ストリーミングデータセット: オブジェクトストレージから直接 shard をストリームし、ローカルへの完全同期を待たずに学習を始める方式(WebDataset, MosaicML Streaming)。経路Bの「同期待ち」をなくす方向です。
- 統一データセット抽象化: 経路A/Bの違いを完全に隠蔽し、同じ API で「未抽出なら裏でデコード、抽出済みなら shard を読む」を透過的に切り替える設計。保守コストの根本的な低減を狙う流れです。さらに進むと、頻繁に使うサンプルだけを自動的に経路B化する「適応的キャッシュ」も考えられます。
