# WebDataset シャーディング

## ひとことで言うと

WebDataset とは、学習データを「tar アーカイブ」(複数ファイルを1つに束ねた箱)にまとめ、それを先頭から順番に読み流すことで、ディスクやネットワークの読み込み速度を最大限まで引き出すデータ形式です。たくさんの小さいファイルを1個ずつ開く遅さを避けるのが目的で、tar という枯れた標準形式をそのまま学習データのコンテナに使う点が特徴です。

## 直感的な理解

機械学習の学習ループは、毎ステップでデータを読み込みます。GPU の計算は非常に速いので、データ読み込みが遅いと GPU が「データ待ち」で遊んでしまい、せっかくの計算能力が無駄になります。これを I/O ボトルネックと呼びます(I/O は input/output、読み書きのこと)。学習が遅い原因が GPU 不足ではなくデータ供給だった、というのは大規模学習で頻発する落とし穴です。数百万円の GPU が「次の画像まだ?」と待ち続ける状況は、コスト効率の点でも深刻です。

データを小さいファイルで大量に持つと、この問題が深刻になります。理由は2つあります。

第一に、ファイルを開くたびのオーバーヘッドです。1枚の画像を開くたびに、OS は「そのファイルはどこにあるか」を探し(ファイルシステムのメタデータ参照、inode の探索)、開いて、読んで、閉じます。画像が数百万枚あると、この「開く・閉じる」の手間が積み上がり、肝心のデータ転送より管理コストが支配的になります。1枚あたりの中身が数十 KB しかないのに、ファイルを開く事務処理に同じくらいの時間がかかる、というイメージです。

第二に、ランダムアクセスの遅さです。ファイルがディスク上であちこちに散らばっていると、読み込みがあちこちに飛ぶ「ランダムアクセス」になり、連続して一気に読む「シーケンシャルアクセス」より大幅に遅くなります。HDD ではランダム読みは秒間100〜200回程度のアクセスが限界ですが、シーケンシャル読みなら100 MB/s 以上出ます。クラウドストレージ(S3 など)でも同じで、小さいオブジェクトを大量に GET するより、大きなファイルをまとめて1回で読むほうが圧倒的にスループットが出ます。

WebDataset は「だったら、たくさんの小ファイルを tar で束ね、先頭から順番になめるように読めばいい」という発想で生まれました。

## 基礎: 前提となる概念

tar(ターボール): 複数ファイルを1つのファイルに連結する古典的なアーカイブ形式。中身は「ファイルのヘッダ(名前・サイズ)→そのファイルの中身→次のヘッダ→…」と一列に並んでいるだけのシンプルな構造です。圧縮機能は本来別(gzip など)で、tar 自体はただ連結するだけ。この単純さゆえに、先頭から順番に読むのが極めて高速で、`tar tf shard.tar` で中身を一覧できるなど扱いやすさも高い。1979 年代からある枯れた規格なので、どの環境にもツールが揃っている点も実用上大きい。

シーケンシャル vs ランダムアクセス: シーケンシャルは「並んでいる順に連続で読む」、ランダムは「飛び飛びの位置を読む」。物理ディスクでもクラウドでも、シーケンシャルが速い。tar は中身が一列なので、束ねた小ファイルを順に読むだけでシーケンシャルアクセスになります。

map-style と iterable-style: PyTorch のデータセットには2種類あります。map-style は「i 番目のデータをください」とインデックスで取り出せる形(全長が分かり、任意の位置にジャンプできる)。iterable-style は「先頭から順に次々流れてくる」形(ジャンプはできないが順次読みが速い)。WebDataset は本質的に iterable-style です。この性質が後述の長所と短所を生みます。

シャッフル(shuffle): 学習ではデータの提示順をランダムにする必要があります。順番が固定だと、たとえば「最初は晴れ、後半は雨」のような偏った順序を学習が記憶してしまい、汎化(未知データへの対応力)が損なわれます。確率的勾配降下法は「各バッチが全体の偏りない縮図であること」を暗黙に仮定しているので、順序の偏りは学習を不安定にします。

DataLoader とワーカー: 深層学習フレームワークは、複数の並列プロセス(ワーカー)でデータを先読みして GPU に供給します。複数ワーカーが同じデータを重複して読まないよう、仕事を分担させる仕組みが要ります。

## 仕組みを詳しく

基本のアイデアはシンプルです。

1. データを「サンプル」単位で考える。1サンプル = 学習1件分(例: 複数枚のカメラ画像 + 数値ベクトル + メタ情報)。

2. 1サンプルを構成する複数ファイルに、同じ先頭名(キー)を付ける。WebDataset は「拡張子の前の部分(basename)が同じファイル群は1サンプル」とみなします。

```
000123.cam_0.jpg
000123.cam_1.jpg
...
000123.cam_6.jpg
000123.ego.npy      ← 数値ベクトル(例: 過去・未来の運動)
000123.meta.json    ← メタ情報
```

ここで `000123` が `__key__`(サンプル識別子)になり、`cam_0.jpg` などの拡張子が「データの種類」を表します。デコード時はこの拡張子を見て「jpg なら画像、npy なら numpy 配列、json なら辞書」と自動変換します。拡張子がそのまま型情報になるので、独自スキーマ定義ファイルが要らないのが特徴です。

3. これらを tar アーカイブにサンプルごとにまとめて格納する。重要なのは「同じサンプルのファイルを tar 内で連続して並べる」ことで、これにより1サンプルを読むときシークが起きません。1つの tar ファイルを「シャード(shard、破片)」と呼びます。「シャーディング」とは全データを複数の tar に分割することです。例: 全100万サンプルを1万サンプルずつ100個の tar に分割。命名規約は `shard-{000000..000099}.tar` のような連番が一般的で、ブレース展開で一括指定できます。

4. 学習時は tar を先頭から順番に読む。tar は中身が一列なので、ヘッダ→本体→次のヘッダ…と連続して読めます。これがシーケンシャルアクセスで、ディスク/ネットワーク帯域を使い切れます。S3 のようなクラウドストレージからも、シャードを1本のストリームとして読めるので高スループットです。

5. ランダム性は2段階で確保する。
   - シャードレベルのシャッフル: どのシャードから読むかをランダムにする。エポックごとにシャードの読む順を変える。
   - サンプルレベルのシャッフルバッファ: 一定個数(例: 1000サンプル)をメモリに貯め、そこからランダムに1個取り出す。取り出した穴には次のサンプルを流し込む。完全ランダムではないが、バッファが十分大きければ実用上問題ありません。これがシーケンシャル読みとシャッフルを両立させる肝です。完全シャッフル(全データを真にランダム)はランダムアクセスを要求するので、シーケンシャル読みの利点を捨てることになる。そこを「近似シャッフル」で折り合わせています。

6. 複数ワーカーへの分割: ワーカーごとにシャードを割り振り、同じ tar を二重に読まないようにします(split by worker)。分散学習では、さらにノード間でもシャードを分けます(split by node)。シャード数がワーカー数で割り切れないと、一部ワーカーが余分に働いて学習が同期待ちになるので、シャード数はワーカー総数の倍数に揃えるのが定石です。

擬似コードで流れを示すと:

```
dataset = WebDataset(urls, nodesplitter=split_by_worker)   # tar を順次ストリーム
dataset = dataset.shuffle(1000)                            # シャッフルバッファ
dataset = dataset.decode().map(to_tensor)                  # 拡張子で復号→テンソル化
loader  = WebLoader(dataset, batch_size=B, num_workers=W)  # 並列先読み
```

1バッチ出力の shape の例(カメラ7台、画像256×256、過去256次元・未来128次元の運動):

```
images       (B, 7, 3, 256, 256)
ego_history  (B, 256)
target       (B, 128)
```

設計上の重要点として、WebDataset 経路は「動画デコードや特定ライブラリへの依存を学習ループから排除できる」ことがしばしば狙いになります。たとえば [LeRobot データフォーマット (v2.1/v3.0)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0824_lerobot-format) のように映像を動画として持つ形式は、ランダムアクセス時のデコードが重い。そこで前処理段で一度だけ全フレームをデコードして JPEG などに展開し、tar(シャード)に焼いておけば、本番の学習ループは tar を逐次読むだけになり、デコードコストが消えます。デコードという「重い計算」を学習のたびに繰り返すのでなく、前処理で1回だけ払って結果を保存する、という典型的な計算とストレージのトレードオフです。クラウドでは init コンテナがシャードを S3 からノードローカルのディスク(EBS など)へ同期し、ローカルのフルバンド幅で読む構成がよく取られます。

## 手法の系譜と主要論文

WebDataset(Aizman, Maltby, Breuel; NVIDIA。背景論文 "High Performance I/O For Large Scale Deep Learning", IEEE Big Data 2019, arXiv:2001.01858)。提案内容は「POSIX tar という枯れた標準形式をそのまま学習データのコンテナに使い、シーケンシャル I/O でディスク/ネットワーク帯域を飽和させる」こと。動機は、ImageNet 級以上の大規模学習で、何百万もの小ファイルがファイルシステムとネットワークのボトルネックになっていたこと。効果は、ローカル SSD でもクラウドストレージでも理論帯域に近いスループットを出し、GPU の待ち時間を減らせること。設計思想として「データ形式は独自バイナリでなく標準 tar」「ストレージは POSIX でも S3 でも同じインターフェイス」という可搬性を重視しています。

比較対象となる形式:
- TFRecord(TensorFlow)。サンプルを Protocol Buffers でシリアライズして連続格納する形式で、「まとめて連続格納」という思想は WebDataset と近い。ただし TensorFlow 寄りで、独自スキーマ(Example の定義)が必要、中身を素のツールで覗きにくい、という違いがあります。
- FFCV(Leclerc, Ilyas, Engstrom, Park, Madry, 2022, arXiv:2306.12517)。さらに踏み込んでメモリレイアウト・JIT コンパイル・GPU 上デコードまで最適化し、ImageNet 学習を大幅に高速化しました。独自バイナリ形式で、性能は最強クラスだが汎用性・可搬性は WebDataset に劣ります。
- Parquet([LeRobot データフォーマット (v2.1/v3.0)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0824_lerobot-format) が数値に採用)。列指向で「特定の列だけ読む」のが得意な一方、画像のような大きなバイナリの逐次ストリーミングには WebDataset の方が向きます。両者は補完関係で、数値は parquet、画像系は WebDataset と使い分けることもあります。
- MosaicML Streaming(MDS)など後発の形式。WebDataset の弱点である「厳密な決定的シャッフル」「途中再開(チェックポイントからの resume)」「分散での均等割り」を作り込みで解決した、思想的な後継です。

系譜としては、WebDataset の思想は後に Hugging Face Datasets の WebDataset 対応や、大規模マルチモーダル学習(LAION-5B, Schuhmann et al., NeurIPS 2022, arXiv:2210.08402 など数十億画像規模のデータ)での標準的な配布形式採用へと広がりました。CLIP や Stable Diffusion といった基盤モデルの学習データはこの形式で配布・処理されており、「tar の連番シャードで配る」は今や大規模学習データ配布のデファクトの一つです。

## 論文の実験結果(定量データ)

WebDataset 系の評価は、精度指標ではなくスループット(秒間サンプル数)とストレージ帯域利用率で測られます。

- 背景論文と各種ベンチマークでは、小ファイル個別読み込みに対し、tar シャードのシーケンシャル読みはストレージの理論帯域に近い利用率(ローカル高速ストレージや並列ファイルシステムで GB/s 級)を達成できると報告されています。小ファイル方式では、ファイルオープンのオーバーヘッドで帯域の数分の一しか使えないことが多く、ここが直接の差になります。
- クラウド(S3 など)では、多数の小オブジェクトを GET する方式に比べ、シャード単位の大オブジェクトをストリーム取得する方式が、リクエスト数を桁で減らし、実効スループットを大きく改善します。クラウドストレージは「小さなリクエストを大量に出す」ほどレイテンシとリクエスト課金が効くため、ここでの差はコストにも直結します。100万枚を1枚ずつ GET すれば100万リクエスト、1万枚束のシャード100個なら100リクエストで、リクエスト課金が1万分の1になる計算です。
- FFCV の論文(2022)は、ImageNet の1エポックあたり学習時間を、標準的な PyTorch ImageFolder ローダーに比べて大幅短縮(設定により数倍)できることを示し、データローディングが学習全体の律速になりうること、そしてそれを潰すと総時間が大きく縮むことを定量的に裏付けました。WebDataset も同じ「I/O 律速を潰す」系譜の手法です。

アブレーション的に効く要素:
- シャッフルバッファのサイズ。小さすぎると提示順に局所的な偏りが残り(同じシャード=似た時間・場所のサンプルが固まって出る)、学習の安定性・最終精度がわずかに劣化することがあります。大きくするとメモリを食うので、メモリと擬似ランダム性の質のトレードオフです。動画から焼いたシャードでは「連続フレームが近接して出やすい」偏りが特に出やすく、バッファを大きめにとる配慮が要ります。
- シャードサイズ。小さすぎるとファイル数が増えて小ファイル問題が再燃し、大きすぎると分割の柔軟性(ワーカー・ノードへの均等割り)が失われます。経験的には1シャード数百 MB〜数 GB 程度が選ばれます。

## メリット・トレードオフ・限界

メリット:
- シーケンシャル読みでディスク/ネットワーク帯域を使い切り、I/O ボトルネックを解消できる。
- tar は標準形式なので可搬性が高く、中身の確認(`tar tf`)も容易。クラウドストレージとの相性も良い。
- 動画デコードや特定ライブラリ依存を学習ループから排除でき、学習段が軽く・速くなる。
- 分散学習でシャードをノード・ワーカーへ自然に分割でき、スケールしやすい。
- 拡張子=型なので独自スキーマ不要。新しい種類のデータ(別センサー)を足すのもファイルを1つ増やすだけ。

トレードオフと限界:
- ランダムアクセスが苦手。特定サンプルのピンポイント取得や、インデックスを指定した取得には本質的に向きません(map-style ではなく iterable-style)。エポックの厳密な反復回数管理や、バリデーションで「全データちょうど1回ずつ」を保証するのにも工夫が要ります。
- シャッフルが擬似的(バッファ近似)。バッファが小さいと並び順の偏りが学習に影響しうる。完全な統計的独立は保証されません。
- 前処理段でシャードを作る手間と、ストレージの二重持ち(元データ + シャード)が発生する。元データを更新するたびに焼き直しが必要です。
- サンプルの組み立てがファイル名(キー)の規約に依存するため、命名がずれると黙って壊れる(別サンプルと混ざる)危険があります。エラーを出さず静かに間違う種類のバグなので、検証(同じ key のファイルが揃っているか)を前処理に入れるのが安全です。
- 1サンプルだけ差し替えるといった部分編集が不向き。tar を作り直す必要があります。
- 途中再開が素朴には難しい。学習が落ちたとき「どこまで読んだか」を正確に復元するには追加の仕組み(決定論的なシャード順とサンプル位置の記録)が要ります。

研究・実務の最前線では、tar の弱点(インデックス不可・部分取得不可)を補うために、シャードにオフセット索引を付けて疑似ランダムアクセスを可能にする拡張(WebDataset の index ファイル)や、クラウドネイティブな配信(範囲 GET でのシャード部分読み)との組み合わせが進んでいます。大規模マルチモーダル学習では、WebDataset 形式での配布がほぼ標準となり、データ供給がモデル規模に追いつくための基盤技術として定着しています。

## 発展トピック・研究の最前線

- 決定的シャッフルと resume: 大規模学習は数日〜数週間かかり途中で必ず落ちるので、「どこまで学習したか」を正確に再現できることが重要になります。MosaicML Streaming など後発フォーマットは、決定的なサンプル順と途中再開を第一級でサポートし、WebDataset の弱点を補っています。
- データの重複排除・品質フィルタ: 数十億サンプル規模では、重複や低品質サンプルが学習を汚します。シャードを流しながらフィルタ(画像とテキストの整合度が低いものを捨てる等)をかける前処理が、基盤モデル学習の品質を左右します。
- GPU デコード: JPEG デコードを CPU でなく GPU で行う(NVIDIA DALI など)ことで、デコード律速をさらに潰す方向。WebDataset とも組み合わせられます。
- ストレージ階層の最適化: シャードをオブジェクトストレージ(S3)に置き、計算ノードのローカル NVMe にキャッシュする多段構成。初回はネットワーク、2回目以降はローカル帯域、という設計で大規模分散学習の I/O を支えます。
- フォーマット中立化: WebDataset・Parquet・MDS など複数形式を同じローダー抽象で扱い、後から最適形式に切り替えられるようにする設計が、長期運用のデータ基盤で重視されています。
