# GPS→マップ画像レンダリング（OpenStreetMap上にナビ経路を描画する入力生成）

## ひとことで言うと
GPS→マップ画像レンダリングとは、自車の GPS の点列（緯度・経度の時系列）を OpenStreetMap という無料の地図データの上に描き、「いまどの道をどちらへ進む予定か」を表した1枚のナビ地図画像を作る手法です。地図入力を最初から持っていない運転データセットでも、ナビ地図画像を後付けで生成し、end-to-end 運転モデルへの入力として与えられるようにするのが狙いです。

## 直感的な理解
人間は運転するとき、目（カメラ）だけでなくカーナビ（地図と経路）を見ます。「次の交差点を右」という情報は、目の前の映像だけからは分かりません。直進と右折で目の前の画素はほとんど同じなのに、取るべき行動は正反対です。end-to-end の運転モデルも同じで、カメラ映像に加えて「どこへ向かうか」を示すナビ地図を入力に与えると、ルートレベルの意図を伝えられます。

ところが、世の中の運転データセットの多くは、こうしたナビ地図画像を持っていません。持っているのは GPS の生データ（緯度・経度がずらっと並んだ時系列）だけ、というのがよくあります。そこで「GPS の点列から、後付けでナビ地図画像を描き起こす」ことを考えます。地図画像を持つ高級なデータセットと、GPS しか持たない素朴なデータセットを、同じ土俵（同じ入力形式）に揃えてモデルを学習・評価できるようにするのが目的です。

## 基礎: 前提となる概念
まず GPS の座標を理解します。GPS は緯度（latitude, 北緯/南緯）と経度（longitude, 東経/西経）で位置を表します。これは地球上の点を角度で示す座標系です。自車の走行ログには、この (緯度, 経度) のペアが時刻ごとに並んでいます。緯度経度は角度なので、距離（メートル）に直すには地球の半径を使った変換が必要で、緯度1度は約 111km、経度1度は緯度によって変わる（赤道で約 111km、高緯度ほど狭まる）点に注意します。

次に OpenStreetMap（OSM）。世界中のボランティアが整備している無料の地図データベースで、道路・交差点・建物などの地理情報を誰でも使えます。道路網は「グラフ」として表現できます。グラフとは、ノード（交差点や道路の端点）とエッジ（ノードをつなぐ道路区間）からなる構造です。一方通行や道路種別（高速・生活道路）などの属性もエッジに付きます。

OSMnx（Boeing, 2017）は、OSM から道路網グラフを取得し、解析・描画する Python ライブラリです。「ある地点の周辺 800m の車道ネットワークをグラフで取ってくる」「あるノードから別ノードへの最短経路を求める」「グラフを地図として描画する」といった処理を高レベル API で行えます。内部的には NetworkX のグラフとして道路網を保持します。

最後に「マップマッチング（map-matching）」。GPS は誤差で実際の道路から数メートル〜十数メートルずれることがあります（ビル街のマルチパスやトンネルで悪化）。マップマッチングとは、この誤差を含む GPS の生の点を、実際の道路網に対応づける（道路上に吸着させる）処理です。これをしないと「経路が建物の中を突っ切る」「並行する別の道に飛び移る」ような不自然な描画になってしまいます。

## 仕組みを詳しく
典型的なスクリプトは、地理空間ライブラリ（pandas で表データ、osmnx で OSM 取得・描画、matplotlib で描画、PIL で画像化）を使い、5ステップで地図画像を作ります。

### ステップ1: 生の GPS データを用意
```python
raw_gps_data = {
    'latitude':  [37.7749, 37.7752, 37.7759, 37.7765],
    'longitude': [-122.4194, -122.4185, -122.4173, -122.4162]
}
```
緯度・経度の点列です（例はサンフランシスコ付近の4点）。実際には走行ログから取り出した連続的な GPS waypoint（経路点）になります。

### ステップ2: 周辺の道路網を OSM からダウンロード
```python
center_point = (df['latitude'].mean(), df['longitude'].mean())   # 点列の中心
graph = ox.graph_from_point(center_point, dist=800, network_type='drive')
```
点列の中心から半径 800m の範囲の「車が走れる道路（drive）」ネットワークをグラフとして取得します（ノード=交差点、エッジ=道路）。半径は、自車の予定軌跡が収まる広さに合わせて選びます。狭すぎると経路がはみ出し、広すぎると描画が密になって意図が読み取りにくくなります。

### ステップ3: マップマッチング（GPS 点を道路に吸着）
```python
for each GPS point:
    nearest_node = ox.nearest_nodes(graph, X=lon, Y=lat)   # 最寄りの交差点ノード
# 隣り合うノード間を最短経路でつなぐ
path = ox.shortest_path(graph, node_i, node_{i+1}, weight='length')
```
各 GPS 点を最寄りの道路ノードに吸着（snapping）し、隣り合う点の間を道路に沿った最短経路で補間します。これで「ばらついた点の集まり」が「道路に沿ったなめらかな経路」になります。これは最も単純なマップマッチング近似です。後述するように、点がまばらだったり誤差が大きいと、ここで誤った道を選ぶことがあります。

### ステップ4: 描画して PIL 画像に変換
```python
fig, ax = ox.plot_graph(graph, bgcolor="#111111", edge_color="#444444", ...)  # 暗い地図
ax.scatter(lon, lat, color='#FF3333', ...)                  # 生 GPS を赤い点で重ねる
ox.plot_graph_route(graph, route_nodes, route_color="#00CCFF", ...)  # 経路を青線で
buf = io.BytesIO(); plt.savefig(buf, format='png', dpi=200)  # 図をメモリに保存
map_pil_image = Image.open(buf)
```
暗い背景に、道路網（灰）、生 GPS（赤い点）、道路に吸着した経路（青い線）を重ねた1枚の画像を作ります。色やコントラストは、モデルが経路を画像特徴として拾いやすいよう設計します（背景の道路と経路の色を強く分けるなど）。

### ステップ5: 画像を保存・利用
出力は標準的な PNG / PIL Image なので、そのままモデルの画像入力パイプラインに渡せます。tensor 化すれば `[3, H, W]` の RGB 画像として扱え、カメラ画像と同じ要領でモデルに入力できます。多くの end-to-end モデルは、このナビ地図を別ブランチのエンコーダに通し、カメラ特徴と融合（concat / cross-attention）してプランナの条件にします。

## 手法の系譜と主要論文
これは主にデータ前処理の実務的手法であり、単一の論文手法ではありませんが、背景となる技術系譜があります。

OpenStreetMap（2004年〜）は、商用地図データの代替として始まった協調型地図プロジェクトです。全世界をカバーし、無料・改変可能なため、研究のベースマップとして広く使われます。

Boeing, 2017, "OSMnx: New methods for acquiring, constructing, analyzing, and visualizing complex street networks"（Computers, Environment and Urban Systems, vol.65）。OSM から街路ネットワークをグラフとして取得・解析・可視化する手法とライブラリを提示しました。本手法のグラフ取得・最短経路・描画は、この OSMnx の API に依拠しています。

マップマッチングは ITS（高度道路交通システム）や GIS（地理情報システム）の古典的研究テーマです。Newson & Krumm, 2009, "Hidden Markov Map Matching Through Noise and Sparseness"（ACM SIGSPATIAL）が代表的で、隠れマルコフモデル（HMM, 観測=GPS 点、隠れ状態=どの道路区間にいるか、として確率的に最尤経路を推定するモデル）を使い「GPS 点の誤差」と「道路網の構造」を確率的に統合し、ノイズが大きく点がまばらでも頑健に経路を復元します。本手法の「最寄りノード＋最短経路」近似は、これよりずっと単純で実装は軽いものの、GPS ノイズが大きい・道が密な市街地では経路がぶれたり、実際とは違う道を選ぶ可能性があります。精度が必要なら HMM ベースのマップマッチングに置き換えるのが筋です。

ナビ地図やルート情報を end-to-end 運転モデルの条件入力に使う流儀は、近年の end-to-end 自動運転研究で一般的です。地図入力は「どこへ向かうか」というルートレベルの意図を与え、軌跡計画（[プランナ レジストリパターン（BasePlanner + PLANNER_REGISTRY）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/1000_planner-registry-pattern)）の条件になります。高精度地図（HD map, 車線レベルの精密な地図）を使う研究もありますが、HD map は整備コストが高く全世界をカバーしないため、OSM ベースの軽量なナビ地図で代替・補完する発想は実用的です。ST-P3（Hu et al., ECCV 2022, arXiv:2207.07601）のような end-to-end 視覚運転モデルは「high-level command（直進・左折・右折）」を条件に取りますが、これは画像化したナビ地図よりさらに粗い意図表現です。ナビ地図画像は、この離散コマンドと精密 HD map の中間的な粒度の意図を与える手段と位置づけられます。

## 論文の実験結果（定量データ）
本手法自体は前処理であり、それ単独の標準ベンチマークはありません。関連する定量的観点を挙げます。

マップマッチングの精度: Newson & Krumm（2009）は、HMM ベースのマップマッチングが、GPS のサンプリング間隔が長く（点がまばらで）誤差が大きい条件でも、単純な最近傍マッチングより大幅に正確に経路を復元できることを示しました。論文ではサンプリング間隔を 1 秒から 30 秒まで伸ばす実験を行い、間隔が長くなるほど単純法は急速に劣化するのに対し HMM 法は緩やかな劣化にとどまる、という頑健性を定量的に示しています。逆に言えば、本手法が採る最近傍＋最短経路の近似は、サンプリングが密で誤差が小さいときには十分実用的ですが、点がまばらだと誤った道を選ぶ確率が上がります。データのサンプリングレートと GPS 精度に応じて手法を選ぶべき、というのが定量的な教訓です。

地図表現の精度（参考）: 地図をオンラインで推定する研究では、HDMapNet（Li et al., ICRA 2022, arXiv:2107.06307）が nuScenes でセマンティック地図のセグメンテーション IoU を報告し、MapTR（Liao et al., ICLR 2023, arXiv:2208.14437）はベクタ化地図構築でリアルタイム（約 25 FPS 級）かつ高い mAP を達成しました。これらは「車線レベルの精密地図」を作る方向で、本手法の「ルートレベルの粗いナビ地図」とは目的も粒度も異なりますが、地図入力の品質が下流の運転性能に効くことを示す傍証です。

ナビ入力の効果: end-to-end 運転研究では、ルート/ナビ情報を条件に与えると、分岐点での意図（どちらへ曲がるか）の予測精度や、ルート遵守率が向上することが報告されています。地図入力なしのモデルは「目の前の道なりに走る」ことはできても「指定ルートへ向かう」のは苦手で、ナビ入力がこのギャップを埋めます。具体値はデータセット・指標に依存するため、適用先で測るのが前提です。

## メリット・トレードオフ・限界

メリット:
- 地図入力を持たないデータセットでも、ナビ地図を後付けで作れ、地図入力前提のモデルや評価の土俵を揃えられる。
- OpenStreetMap は無料・全世界カバーなので、追加の地図購入が不要。
- 出力が普通の PNG / PIL 画像なので、既存の画像入力パイプラインにそのまま載る。
- 実装が osmnx の高レベル API 中心で、コードが短く読みやすい。

トレードオフ・限界:
- マップマッチングが単純（最寄りノード＋最短経路）なので、GPS ノイズが大きい・道が密な市街地では経路がぶれたり、実際とは違う道を選ぶことがある。HMM ベースの精密なマップマッチングに劣る。
- OSM のデータ品質は地域差が大きく、整備の薄い地域では道路網が欠けることがある。欠けると吸着先のノードがなく経路が破綻する。
- `graph_from_point` は OSM サーバへ問い合わせる前提で、大量フレームをオフラインでバッチ処理するにはキャッシュやローカル OSM データの工夫が要る（素朴な実装だと遅い・ネット依存・サーバ側のレート制限に当たる）。
- matplotlib 経由で1枚ずつレンダリングするため、データセット全体に適用すると重く、学習用に大量生成するには高速化（並列化・ベクタ描画・GPU ラスタライズ等）が必要。
- 「地図入力を前提に学習されたモデル」と「後付け生成した地図画像」の見た目を完全一致させるのは難しく、色・縮尺・描画範囲・方位（北固定か進行方向固定か）などの差がドメインギャップ（学習時と評価時で入力分布がずれる現象）になりうる。ドメインギャップがあると、見た目の違いだけで性能が落ちるため、評価結果の解釈に注意が要る。
- 未来の GPS（これから通る点）を使って経路を描くと、推論時に「答えを先に見ている」ことになりリーク（leakage, 評価が不当に甘くなる）になる。学習・評価では「過去＋目的地」だけから経路を引くなど、情報の使い方を厳密に設計する必要がある。

## 発展トピック・研究の最前線
- ベクタ地図条件: 画像としてレンダリングする代わりに、車線・道路境界・経路を「ベクタ（点列・ポリライン）」のまま条件に与える手法が近年の end-to-end 運転で主流になりつつあります（MapTR 系の流れ）。画像化による情報損失とレンダリングコストを避けられ、座標として正確な幾何を渡せます。
- HD map / オンラインマッピング: 高精度地図を使う、あるいはカメラ・LiDAR からオンラインで車線地図を推定する（HDMapNet, MapTR 系の研究）方向もあります。OSM ベースのナビ地図は「ルートレベルの粗い意図」を担い、これらの「車線レベルの精密な地図」とは役割が異なり、両者を組み合わせる設計もあります。
- ドメイン適応: 後付け生成した地図と本物の地図入力の分布差を埋めるため、スタイル変換やドメイン適応を併用する研究があります。評価専用に揃えるか、学習にも使うかで必要な厳密さが変わります。
- バッチ前処理の高速化: ローカルにクリップした OSM データ（PBF ファイル, OSM のバイナリ配布形式）から直接グラフを構築し、ネットワーク往復を排して大規模データセットに適用する実務的工夫が重要になります。一度広域グラフを構築してキャッシュし、フレームごとに切り出すと往復を大幅に減らせます。
