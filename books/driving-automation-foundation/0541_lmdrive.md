# LMDrive

## ひとことで言うと
LMDrive は、人間が言葉で出した運転指示(「次の交差点を右折して」など)と、複数のカメラ・LiDAR からの周囲映像をもとに、シミュレータの中で実際に車を走らせ続ける運転AIです。「言葉で命令できる」ことと、「止まらずに連続して運転し続ける(閉ループ運転)」ことを大規模言語モデル(LLM)で両立させた点が新しい研究です。それまでの LLM 運転研究が「録画を見て答え合わせする」段階だったのに対し、LMDrive は「LLM に実際にハンドルを握らせて完走させる」ところまで踏み込みました。

## 直感的な理解

カーナビとの違いを考えると分かりやすいです。普通のカーナビは「200m先を右折」と決められた経路点を機械的に渡すだけです。一方、助手席の人間に「あの青い看板の手前で左に入って」「自転車に気をつけて」と言えば、相手はその言葉を理解して臨機応変に運転を調整してくれます。LMDrive が目指すのは後者で、自然言語の指示を理解して運転に反映できる、人間の助手のような運転AIです。

なぜこれが重要かを理解する鍵が「開ループ」と「閉ループ」の対比です。

開ループ評価(open-loop)とは、あらかじめ記録した運転データを使い、「この場面でモデルが出した操作」が「人間が実際にやった操作」とどれだけ近いかを測る方式です。モデルの出力は次の場面に影響しません(録画を見ながら答え合わせするだけ)。実装が簡単ですが、致命的な弱点があります。モデルが少しずれた操作をしても、次のフレームは人間の運転の続きなので、ずれが蓄積しないのです。現実には、小さなずれが積み重なって車線をはみ出し、見たことのない状況に陥る「分布シフト(distribution shift)」が起きます。開ループ評価はこれを見抜けません。

閉ループ評価(closed-loop)とは、シミュレータ(CARLA など)の中でモデルの出力が実際に車を動かし、その結果として次の場面が決まる方式です。モデルがミスすればその結果に自分で対処しなければならず、現実の運転に近い厳しい評価ができます。少しのずれが雪だるま式に悪化するか、自分で立て直せるか、までが問われます。

前身の [DriveGPT4](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0533_drivegpt4) は LLM で運転を説明・制御する道を開きましたが、評価は主に開ループで、LLM の遅さもあって閉ループ実時間運転は実証していませんでした。LMDrive(Shao et al., CVPR 2024)は「LLM を使って、言葉の指示に従いながら閉ループで走り続ける」という、より実用に近い難題に取り組みました。

## 基礎: 前提となる概念

LMDrive を支える前提知識を噛み砕きます。

CARLA(Dosovitskiy et al., CoRL 2017): オープンソースの自動運転シミュレータです。仮想の街・車・歩行者・天候を再現でき、モデルの出力で車を動かして結果を観測できるため、閉ループ評価のデファクト標準になっています。実車で危険な状況を何千回も試すのは不可能ですが、シミュレータなら安全に大量試行できます。

waypoint(ウェイポイント、経路点): 「この先どこを通るか」を表す、車の将来位置の点列です。end-to-end 運転モデルの多くは、ハンドル角を直接出すのではなく、まず waypoint を出し、それを低レベル制御器(PID 制御など、目標位置に近づくよう自動でハンドル・アクセルを調整する古典的な制御方式)が実際の操作に変換します。この2段構成は、出力が安定しやすく実装も簡単という利点があります。

BEV(Bird's-Eye-View、鳥瞰図表現): 周囲の状況を、上空から見下ろした地図のような2次元グリッドで表す表現です。複数カメラや LiDAR の情報を1枚の共通の地図に統合できるため、マルチセンサー融合と相性が良く、現代の運転知覚の中心的な表現です。詳細は [鳥瞰図特徴融合（BEVFormer風 空間クロスアテンション）](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0299_bev-fusion)。

センサー融合の系譜(TransFuser / InterFuser): カメラ(色・テクスチャに強いが距離に弱い)と LiDAR(正確な距離を測れるが色がない)を組み合わせると、互いの弱点を補えます。TransFuser(Chitta et al., TPAMI 2022)や InterFuser(Shao et al., CoRL 2022)は、Transformer でカメラと LiDAR を融合して CARLA を閉ループ運転する先行研究で、LMDrive の知覚側(ビジョンエンコーダ)はこの系譜にあります。

これらを踏まえると、LMDrive は「TransFuser/InterFuser 系の高性能な知覚エンコーダの上に、LLM による言語指示理解を乗せて、CARLA を閉ループで走らせる」構成だと理解できます。

## 仕組みを詳しく

LMDrive は大きく2つの部品からなります。

部品1: ビジョンエンコーダ(知覚側)。複数台のカメラ画像(前・左・右・後など)と LiDAR の点群を入力し、周囲の状況を表すトークン列に変換します。LiDAR を BEV に変換し、物体検出・信号認識・waypoint 予測などの補助タスクで事前学習することで、運転に必要な情報が詰まった「視覚トークン」を作ります。この事前学習が品質の要で、知覚が弱いと後段の LLM がいくら賢くても破綻します。

部品2: LLM(意思決定側)。言語指示(自然文)のトークンと、ビジョンエンコーダが出した視覚トークンをつなげて LLM(LLaMA 系などをベースに視覚トークンを受け付けるよう改造)に入力します。視覚トークンを LLM の入力空間へ写すために、Q-Former 風のアダプタ(少数の学習可能なクエリで視覚特徴を要約する小さな Transformer)を挟むのが典型で、これにより大量の視覚特徴を扱いやすい固定長のトークン列へ圧縮します。LLM はこれらを統合し、各時刻の waypoint と、「指示を完了したか」を表すフラグ(notice token などと呼ばれる完了判定の出力)を生成します。出力された waypoint を制御器(PID)が実際のハンドル・アクセルに変換し、CARLA 内の車を動かします。その結果次の場面が変わり、また入力に戻る、という閉ループを形成します。

学習時は、ビジョンエンコーダを先に補助タスクで事前学習して凍結(あるいはほぼ固定)し、LLM 側のアダプタと出力ヘッドを言語指示付きデータで学習する、という分担がとられます。知覚と意思決定を分けて学習することで、それぞれの役割が安定し、後段の言語推論が知覚の質に乗りやすくなります。

```
言語指示 "次の交差点で右折"
   │
マルチカメラ+LiDAR ──→ ビジョンエンコーダ ──→ 視覚トークン
   │                    (BEV化+補助タスク事前学習)    │
   └──────── テキストトークン ──────────────────────┘
                       │
                      LLM(視覚+言語を統合)
                       │
              waypoint + 完了フラグ
                       │
                  制御器(PID)→ CARLA で車を動かす → 次の場面へ(閉ループ)
```

「完了フラグ」が重要な工夫です。たとえば「次の交差点で右折」という指示は、右折を終えたら役目を終えます。LMDrive は指示ごとに「達成したか」を判定して次の指示へ進むことで、連続したナビゲーション指示の列を順にこなせます。

学習・評価のため、著者らは LangAuto という CARLA 上のベンチマークも作りました。これは多数の言語指示付き運転シナリオを含み、3つの観点を閉ループで測れます。

- ナビ指示への追従(LangAuto): 「右折」「直進」などの経路指示に従えるか。
- 短い指示区間(LangAuto-Short): より短い区間での追従性。
- 誤った・紛らわしい指示への頑健性、注意指示の遵守: 「(実際には曲がれない場所で)右折せよ」のような実行不能な指示や、安全に関わる注意喚起にどう振る舞うか。

## 手法の系譜と主要論文

LMDrive(Shao, Hu, Wang, Waslander, Liu, Li, "LMDrive: Closed-Loop End-to-End Driving with Large Language Models", CVPR 2024)の位置づけを系譜で整理します。

- 知覚側の源流: TransFuser(Chitta et al., TPAMI 2022)と InterFuser(Shao et al., CoRL 2022)が、Transformer でカメラ+LiDAR を融合し CARLA を閉ループ運転する基盤を作りました。InterFuser は LMDrive と著者が重なり、ビジョンエンコーダ設計の直接の前身です。

- 言語側の源流: DriveGPT4(Xu et al., 2023)が LLM で運転を説明・制御する道を開きましたが、開ループ・説明中心でした([DriveGPT4](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0533_drivegpt4))。

- LMDrive の新規性: この2つの系譜を統合し、「マルチモーダルセンサー+自然言語指示→ LLM → waypoint → CARLA 閉ループ」という、言語指示に従う閉ループ end-to-end 運転を初めて実現したこと。あわせて言語指示付き閉ループベンチマーク LangAuto を公開し、後続研究の比較基盤を提供したことも大きな貢献です。

- 並走・後続の研究:
  - DriveVLM(Tian et al., CoRL 2024): VLM の遅さを補うため速い従来プランナと併用する System1/System2 構成。LMDrive が「言語指示追従と閉ループ完走」に焦点を当てるのに対し、DriveVLM は「複雑シーンの深い推論品質」に焦点を当て、狙いが異なる。[DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm)。
  - 各種 CARLA 閉ループ運転モデル(ReasonNet ほか): LangAuto/CARLA 上で言語条件運転を競う研究が続いています。

この系譜は、「高性能な閉ループ知覚(InterFuser 系)」と「言語による柔軟な指示理解(DriveGPT4 系)」という別々に発展してきた2本の流れが、LMDrive で初めて合流した、という構図を示します。

## 論文の実験結果(定量データ)

LMDrive は CARLA の LangAuto ベンチマーク上で閉ループ評価されました。CARLA 系で標準的な指標を初心者向けに説明します。

- Route Completion(RC、経路完了率): 指定された経路のうち、車が実際に走破できた割合(%)。100% に近いほど、最後まで走り切れたという意味です。
- Infraction Score(IS、違反スコア): 衝突・信号無視・車線逸脱などの違反が少ないほど高くなるペナルティ係数(0〜1)。違反が多いと値が下がります。
- Driving Score(DS、運転スコア): RC と IS を掛け合わせた総合指標(およそ DS = RC × IS)。「どこまで走れたか」と「その間どれだけ安全だったか」の両方を1つにまとめた、CARLA 系の最重要指標です。値が高いほど良い運転です。

LMDrive は LangAuto 上で、言語指示に従いながら経路を完走できる Driving Score を達成したことを報告しました。報告値では、フルの LangAuto(長い経路)で Driving Score がおよそ 36 前後、Route Completion がおよそ 56% 前後、短い区間の LangAuto-Short ではいずれも大きく上がり Driving Score がおよそ 50 前後、Route Completion がおよそ 70% 台に達したとされます(数値は論文・後続比較による)。LangAuto は「言語指示に従う」という制約を課したうえでの閉ループ評価なので、指示なしの素の運転より難度が高く、この水準は言語条件付き閉ループ運転の最初期の到達点として意味があります。

特に、LLM のバックボーン(LLaVA ベースの視覚言語モデルや LLaMA-7B 系など)を変えると性能が変わること、より能力の高い・規模の大きい LLM ほど指示追従や難所での運転スコアが上がる傾向を示しました。これは「運転ループに載せる言語モデルの推論力が、そのまま運転品質に効く」という、LLM 運転の基本的な経験則を裏づける結果です。

アブレーション(どの要素を抜くとどうなるか)の代表的な知見:

- ビジョンエンコーダの事前学習を弱める(補助タスクを減らす)と、知覚情報の質が落ち、Driving Score が大きく低下します。これは「LLM が賢くても、入力の視覚トークンが貧弱なら運転は破綻する」ことを定量的に示すもので、LMDrive の性能が知覚側の事前学習品質に強く依存することを意味します。
- 誤った・紛らわしい指示(misleading instruction)を混ぜたシナリオでは、指示にむやみに従うと違反が増えます。LMDrive はこうした指示への頑健性も評価対象にしており、「実行不能・危険な指示をどう扱うか」が安全上の重要な評価軸であることを浮き彫りにしました。
- 言語指示なしの素の運転と、言語指示付きの運転を比較し、言語条件付けが経路追従にどう寄与するかを切り分けています。

これらの数値が伝える本質は、「LLM を運転ループに入れること自体」より、「知覚エンコーダの質」と「指示の検証・頑健性」が閉ループ性能を左右する、ということです。

## メリット・トレードオフ・限界

メリット

- 自然言語の指示に従って運転でき、柔軟なナビゲーションや注意喚起が可能。固定経路点しか渡せない従来手法にない柔軟さ。
- 閉ループでの実証なので、現実に近い厳しい条件で性能を確認できる(分布シフトに自分で対処できるかまで問える)。
- カメラ+LiDAR のマルチモーダル入力で、単眼カメラより周囲認識が頑健。
- ベンチマーク LangAuto を公開し、再現性と比較可能性が高い。後続研究が同じ土俵で競える。

トレードオフ・限界

- シミュレータ(CARLA)上の成果であり、実車への移行(sim-to-real ギャップ、シミュレータと現実の見た目・物理の差)は別の大きな課題。
- LLM を運転ループに入れるため計算が重く、車載の実時間制御にはハードウェア負荷が大きい。LLM の推論頻度がボトルネックになる。
- 性能がビジョンエンコーダの事前学習品質に依存し、知覚が弱いと LLM がいくら賢くても破綻する(アブレーションが裏付け)。
- 言語指示の解釈ミスや、誤った指示への過剰追従などの安全リスクがあり、指示の検証機構が別途必要。

研究上の未解決課題: 言語指示の「曖昧さ」をどう扱うかは未解決です。「あの店の前で停めて」の「あの店」が複数あったらどうするか、矛盾する指示が来たらどうするか、といった対話的な確認・拒否の仕組みは十分でありません。また CARLA で完走できても、実車の多様な天候・路面・他車の振る舞いに耐えるかは別問題で、sim-to-real は依然として大きな壁です。

## 発展トピック・研究の最前線

- sim-to-real 転移: シミュレータで学んだ運転を実車に移すための、ドメイン適応・データ拡張・現実データでの微調整などの研究。
- 言語指示の安全な解釈: 危険・実行不能な指示を検出して拒否する、曖昧な指示を人間に問い返す、といった対話的安全機構。
- 軽量化・蒸留: 重い LLM を運転ループから外し、その判断を軽量モデルへ蒸留して実時間性を確保する方向([知識蒸留損失](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0633_knowledge-distillation-loss))。
- VLM 推論との統合: DriveVLM のような System1/System2 構成と、LMDrive の言語指示閉ループ運転を組み合わせる研究([DriveVLM](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0535_drivevlm))。
- 閉ループ評価基盤の整備: LangAuto をはじめ、言語条件付き閉ループ運転を体系的に評価するベンチマークの拡充([CARLA(閉ループシミュレーション)](https://zenn.dev/riita10069/books/driving-automation-foundation/viewer/0715_closed-loop-simulation-carla))。
