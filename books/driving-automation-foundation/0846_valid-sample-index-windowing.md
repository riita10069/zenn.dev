# 有効sample_idxの判定（履歴/未来窓）

## ひとことで言うと
時系列データから機械学習のサンプルを作るとき、「ある瞬間」を基準点にして、その前の数十ステップ（過去の動き＝履歴）と後ろの数十ステップ（未来の動き＝予測の正解）を一緒に切り出します。この方法だと、データ列の先頭付近や末尾付近の瞬間は「前が足りない」または「後ろが足りない」のでサンプルにできません。この境界処理を「有効サンプルインデックス（valid sample index）の判定」と呼びます。どの瞬間なら前後の窓がきれいに揃うかを計算し、揃う瞬間だけを学習に使うという、地味だが極めて重要な前処理です。

## 直感的な理解
動画を1本想像してください。あなたは「今この瞬間の前6秒間の動き」を見て「次の6秒間でどう動くか」を当てる練習をしたいとします。すると、動画の最初の6秒間には「前6秒」が存在しないので、その区間を基準点にはできません。同様に、動画の最後の6秒間には「次の6秒」がないので、これも基準点にできません。使えるのは「前にも後ろにも6秒以上の余裕がある中央部分の瞬間」だけです。

この「前後に十分な余白があるか」を1フレームずつ判定して、OKな瞬間だけを集めたものが有効サンプルのリストです。自動運転の軌道予測、歩行者の動き予測、需要予測、音声認識など、過去から未来を予測するあらゆる時系列学習でこの考え方が登場します。設計が雑だと、データの端で配列の範囲外アクセス（index out of range）が起きてプログラムが落ちたり、足りない部分を勝手にゼロで埋めて意味のないサンプルを作って静かに精度を落としたりします。「サンプル数を最大化したい」気持ちと「窓を完全に埋めたい」気持ちのせめぎ合いを、整数の足し算引き算で正確にさばくのがこのトピックの核心です。

## 基礎: 前提となる概念
いくつかの用語を噛み砕いておきます。

- 時系列（time series）: 時間順に並んだデータの列です。各時刻に「自車の速度」「ハンドル角」「位置」などの値が記録されています。
- サンプリングレート / Hz: 1秒間に何個のデータ点を記録するかです。100Hzなら1秒に100個、つまり0.01秒ごとに1点。10Hzなら0.1秒ごとに1点です。
- ダウンサンプリング（downsampling）: 高い頻度で記録されたデータを間引いて頻度を下げることです。100Hzのデータを「10行に1行だけ残す」と10Hzになります。間引く理由は、隣り合うフレームはほとんど同じで情報がかぶっているうえ、データ量が10分の1になって計算が軽くなるからです。
- 窓（window）/ ウィンドウ化（windowing）: 長い時系列から固定長の区間を切り出すことです。「過去64ステップ」「未来64ステップ」のように長さを決めて切り取ります。
- 履歴（history）と目標（target）: モデルへの入力になる過去側の窓が履歴、モデルが当てるべき正解になる未来側の窓が目標（教師信号、ground truth）です。
- パディング（padding）とマスク（mask）: 窓の長さが足りないとき、足りない分をゼロなどで埋めるのがパディング。埋めた部分を「これは本物のデータではない」と区別するための印（0/1の真偽値の列）がマスクです。損失計算でマスク部分を除外しないと、ゼロという偽の正解を学習してしまいます。
- 基準点（reference point / anchor）: データ列の何番目を「現在時刻」とみなすか。その位置を表す整数が sample_idx（サンプルインデックス）です。sample_idx をどこに置けるかが、このノートの主題です。

## 仕組みを詳しく
具体的な数値で組み立てます。典型的な設定として、元データが約100Hzで記録され、それを10Hzにダウンサンプリングするとします。

```
DOWNSAMPLE_STEP = 100 / 10 = 10     # 10行に1行だけ残す (df.iloc[::10])
HISTORY_TIMESTEPS = 64              # 過去64ステップ = 6.4秒分
FUTURE_TIMESTEPS  = 64              # 未来64ステップ = 6.4秒分
```

1つのデータ列（運転クリップ1本など）がサンプルを1つでも生み出せるための最小行数は次のように決まります。

```
MIN_ROWS = HISTORY_TIMESTEPS + FUTURE_TIMESTEPS + 1 = 64 + 64 + 1 = 129
```

最後の「+1」は基準点そのもの1個分です。過去64＋基準点1＋未来64で合計129行が、最小1サンプルに必要な長さです。

有効な sample_idx の範囲は2つの境界で決まります。

```
min_idx = HISTORY_TIMESTEPS              = 64
          (これより前だと過去64ステップが取れない)

max_idx = len(df) - FUTURE_TIMESTEPS - 1
          (これより後だと未来64ステップが取れない)
```

数値例で確認します。ダウンサンプリング後の系列長 len(df) = 200 の場合、

```
min_idx = 64
max_idx = 200 - 64 - 1 = 135
有効な sample_idx = 64, 65, ..., 135  → 72個
捨てられる範囲: 先頭 0..63 (64個) と 末尾 136..199 (64個)
```

ASCII図で全体像を示します。

```
index:  0 ......... 63 | 64 ............... 135 | 136 ......... 199
        [履歴が足りない] [   有効な基準点 72個   ] [未来が足りない]
                          ↑min_idx          ↑max_idx
```

窓の切り出しは次のように行います。どちらもちょうど64行になります。

```python
history_df = df.iloc[sample_idx - 64 : sample_idx]            # 過去64行(基準点は含まない)
future_df  = df.iloc[sample_idx + 1 : sample_idx + 1 + 64]    # 未来64行(基準点の次から)
```

max_idx の式に「-1」が要る理由をもう一押しします。基準点の「次の行」から未来を取るので、未来側の最後の行は sample_idx + 64 です。これがデータ列の最後の有効インデックス len(df) - 1 を超えてはいけないので、

```
sample_idx + 64 <= len(df) - 1
sample_idx      <= len(df) - 65 = len(df) - 64 - 1
```

となり、max_idx = len(df) - FUTURE_TIMESTEPS - 1 と一致します。半開区間（始点を含み終点を含まない `[a, b)`）と閉区間の取り違えは、ここで1ずれ（off-by-one）バグを生む定番の落とし穴です。境界では必ず「最後の1サンプルが本当に取れるか」を具体値で手計算して確かめます。

窓化したデータがモデルの入力次元と対応する例を挙げます。1ステップあたり4信号（例: 速度・加速度・ヨー角速度・ハンドル角）の履歴なら 64×4 = 256次元、1ステップあたり2信号（例: 縦位置・横位置）の未来なら 64×2 = 128次元です。バッチ化すると履歴は (B, 256)、目標は (B, 128) というテンソル形状になります（Bはバッチサイズ＝まとめて処理する件数）。この次元はモデルの入力層・出力層と一致していなければならず、履歴長や未来長を変えると両方を直さないと壊れます。

実装上のよくある設計は、有効サンプルをデータセット構築時に一度だけ全列挙してキャッシュしておくことです。各データ列に対して range(min_idx, max_idx + 1) を列挙し、(列ID, sample_idx, 対応する時刻) の組を平らな1次元リストに展開します。こうすると、

- `__len__`（サンプル総数）は単にそのリストの長さを返すだけ
- `__getitem__`（i番目のサンプル取得）は添字でリストを引いて該当データをI/O（ファイル読み込み）するだけで、呼ばれるたびにインデックス計算やpandas操作をしなくて済む

という高速な構造になります。さらに前段で、行数がMIN_ROWS未満の列（短すぎてどのsample_idxも有効にならない）、必要な列が欠けたデータ、タイムスタンプが単調増加でない（時刻が逆行している＝壊れている）列は事前に弾きます。こうして「使える瞬間だけが、確実に揃った状態で並ぶ」状態を作ります。

学習時は、この有効範囲の中から基準点をランダムに選ぶことで、同じデータ列の中でも毎エポック違う瞬間を学べます。これは時間方向の一種のデータ拡張（data augmentation）になります。インデックスを指定しないときは安全策として範囲の中点 (min_idx + max_idx) // 2 を選ぶ、といった既定値の置き方も一般的です。

ここで重要な発展的設計が「ストライド（stride、基準点をいくつ飛ばしで列挙するか）」です。range(min_idx, max_idx + 1, stride) のように間引くと、隣接サンプル同士が情報をほぼ共有して冗長になる問題を緩和できます。たとえば stride=1 では sample_idx=64 と 65 の履歴窓は63ステップを共有し、ほとんど同じサンプルが大量に並びます。stride を大きくすると独立性は上がりますがサンプル数は減ります。研究では「実効的に何個の独立サンプルがあるか」を意識して stride を選びます。

## 手法の系譜と主要論文
固定長スライディングウィンドウで過去から未来を切り出す発想は、時系列学習の最も基本的な定石で、複数の分野・論文に通底しています。

- Sutskever, Vinyals & Le, "Sequence to Sequence Learning with Neural Networks" (NeurIPS 2014, arXiv:1409.3215): 入力系列と出力系列を対応づけて学習する枠組み（エンコーダ・デコーダ）を提示しました。可変長を扱うときはパディングとマスクで長さを揃えますが、これがバグや計算の無駄の温床になることも知られています。「完全な窓だけ使い、パディングしない」という割り切りは、このマスク処理を不要にする実務的な選択です。
- Alahi et al., "Social LSTM: Human Trajectory Prediction in Crowded Spaces" (CVPR 2016): 歩行者の軌道予測で「観測フレーム（過去）→予測フレーム（未来）」という固定窓を導入しました。代表的な設定は観測8フレーム（3.2秒）→予測12フレーム（4.8秒）で、この観測長・予測長を定数で決めるスタイルがその後の軌道予測研究のテンプレートになりました。
- Chang et al., "Argoverse: 3D Tracking and Forecasting with Rich Maps" (CVPR 2019, arXiv:1911.02620): 自動運転の動作予測ベンチマークとして「過去2秒（観測）→未来3秒（予測）」を 10Hz サンプリングで固定窓フォーマットとして標準化しました（観測20点→予測30点）。なぜ固定窓かというと、モデルの入出力サイズを一定にできてバッチ化と公平な比較が容易になるからです。代償として、窓に満たない端のデータは捨てられデータがやや減ります。
- Gao et al., "VectorNet: Encoding HD Maps and Agent Dynamics from Vectorized Representation" (CVPR 2020, arXiv:2005.04259): Argoverse の固定窓フォーマットを前提に、過去軌道と地図を「ベクトル集合」として符号化する系譜を開きました。窓化は変えず、窓の中身（軌道）をどう表現するかを進化させた例です。
- Caesar et al., "nuScenes: A Multimodal Dataset for Autonomous Driving" (CVPR 2020, arXiv:1903.11027) とその予測課題（nuScenes prediction、過去2秒→未来6秒）でも、過去履歴と未来軌道を固定窓で切り出すのが標準です。窓長（何秒の過去を見て何秒の未来を当てるか）はタスクごとに調整するハイパーパラメータになっています。

系譜としては、Seq2Seqで「窓を切って入出力を対応づける」という骨格ができ、Social LSTMで「軌道予測における観測窓・予測窓」という形に具体化し、Argoverse/nuScenes で「固定窓フォーマットをベンチマークの公式仕様にする」段階に進んだ、と整理できます。有効インデックス判定は、この固定窓フォーマットを実装するときに必ず付随する縁の下の作業です。

## 論文の実験結果(定量データ)
窓長そのものを変えると予測性能がどう変わるかは、多くの軌道予測論文がアブレーション（ある要素を変えて影響を測る実験）で報告しています。

- 軌道予測の代表指標は ADE（Average Displacement Error, 平均変位誤差）と FDE（Final Displacement Error, 最終変位誤差）です。ADEは予測した未来軌道の各時刻と正解の距離を全時刻で平均した値、FDEは予測の最終時刻だけの距離で、いずれも小さいほど良く、単位はメートルです。たとえば未来軌道予測でADEが2.0mから1.5mに下がれば「平均して50cm正解に近づいた」ことを意味し、運転判断では無視できない差です。
- 観測窓（履歴）を長くすると、過去の動きの傾向（加速中か減速中か、曲がろうとしているか）をより多く捉えられ、一般にADE/FDEが改善します。一方で、履歴を長くしすぎると古すぎる情報がノイズになり、また使える有効サンプルが減る（min_idx が大きくなり端の損失が増える）というトレードオフがあります。Argoverse 系の研究では、過去2秒前後・未来3秒前後が予測性能とデータ効率のバランス点としてよく採用されています。
- 未来窓（予測ホライズン）を長くすると、遠い未来ほど不確実性が増えてFDEが急激に悪化することが一貫して報告されています。nuScenes prediction（未来6秒）が Argoverse（未来3秒）より明確に難しく、各手法の FDE がおおむね数倍に膨らむのはこのためです。直感的にも、0.5秒先は当てやすく6秒先は当てにくい、という性質です。長い予測窓を要求するタスクほど、確率的・マルチモーダル予測（複数の可能性を出す）が必要になります。
- マルチモーダル予測の指標 minADE_K / minFDE_K（K個の候補のうち正解に最も近いものの誤差）では、K=1（1本だけ予測）から K=6 に増やすだけで FDE がしばしば半減します。Argoverse のリーダーボードでは、単峰予測の minADE が 1.7m 前後だったものが K=6 で 0.9m 前後まで下がる、といった報告が典型です。これは「未来は一意でない」という性質を、窓の取り方ではなく出力の持ち方で吸収した成果です。

実装上の境界処理の正しさは、精度の数値そのものより「学習が落ちずに回るか」「捨てるデータ量が許容範囲か」で評価されます。たとえば短いクリップが多いデータセットでは、各クリップから先頭64＋末尾64ステップを捨てる損失が相対的に大きくなり、有効サンプル数が想定より少なくなる、という形で実害が出ます。1クリップが平均150行しかなければ、129行未満で丸ごと捨てられる列が混じり、残った列も使えるのは中央の20点程度、というように利用率が一気に下がります。

## メリット・トレードオフ・限界
- メリット: 窓が常に完全に埋まるので、パディングやマスクが不要になり、損失計算や学習ループが単純で安全になる。範囲外アクセスによるクラッシュも構造的に防げる。
- メリット: 有効サンプルを構築時に全列挙してキャッシュすれば、`__getitem__` が添字アクセスとI/Oだけになり高速。学習中の前処理コストが下がる。
- メリット: 短すぎる列・列欠落・時刻が壊れた列を事前に弾くので、学習が長時間回った末に予期せぬエラーで止まる事故を減らせる。
- トレードオフ: 各列の先頭HISTORYステップと末尾FUTUREステップ（この例では計約12.8秒分）は常に捨てられる。短い列が多いほどデータ損失が相対的に大きい。
- トレードオフ: 最小行数（この例で129行）に満たない列は丸ごと使えない。希少なシーンが短い列にしか含まれない場合、貴重なデータを失う恐れがある。
- トレードオフ: stride=1 で密に列挙すると隣接サンプルが強く相関し、見かけのサンプル数ほど情報量がない。学習/検証の分割を列単位で行わないと、ほぼ同じサンプルが両側に漏れて性能を過大評価する（データリーク）。
- 限界・未解決: 履歴長・未来長がモデルの入出力次元と密結合するため、窓長を変えるとデータ側とモデル側の両方を改修する必要がある。可変長の予測ホライズンを扱いたい場合は、結局パディング＋マスクや系列を畳む工夫が必要になり、ここで避けた複雑さが戻ってくる。
- 限界: 「完全な窓だけ使う」方針は、端の希少イベント（クリップ冒頭の発進、末尾の停止など）を学習から排除してしまう。安全上重要な挙動が端に偏っている場合、データ分布に偏りを生む可能性がある。

## 発展トピック・研究の最前線
- マルチモーダル予測: 未来は1通りではない（直進も右折もあり得る）ため、複数の軌道候補を確率付きで出す手法が主流です。混合分布やアンカー（代表的な軌道のひな型）を使う設計が研究の中心になっています。
- 可変ホライズン・ストリーミング予測: 固定窓ではなく、来たデータから逐次予測する設定。境界処理が静的な列挙から動的なバッファ管理（リングバッファに直近Nステップを保持）に変わります。
- データ拡張としての窓ずらし・時間反転: 同じ列から複数の基準点を取ることで実質的にサンプル数を増やせます。隣接基準点の相関に注意しつつ、列単位の分割で過大評価を避けます。
- 履歴の重要度分析: アテンション機構を使って「過去のどの時刻が予測に効いているか」を可視化する研究があり、窓長の適切さを定量的に議論できます。直近フレームに重みが集中するなら長い履歴は無駄、という判断ができます。
- 不規則サンプリングへの一般化: 実データはセンサごとに時刻がずれ、欠損も多い。等間隔窓を前提にせず、連続時間モデル（Neural ODE 系）で「任意の時刻の状態」を扱う方向の研究があります。
