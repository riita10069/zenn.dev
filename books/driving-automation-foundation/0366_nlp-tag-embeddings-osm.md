# NLP タグ埋め込み(OSMタグの言語埋め込み)

## ひとことで言うと

OpenStreetMap(OSM)の地図要素には `highway=residential`(住宅街の道路)や `maxspeed=50`(制限速度50)のような「タグ」という意味ラベルが文字列で付いています。NLP タグ埋め込みとは、このタグ文字列を言語モデル(文章を数値ベクトルに変換する AI)で埋め込み、タグの意味をベクトルとしてモデルに渡す手法です。NLP は Natural Language Processing(自然言語処理)の略で、コンピュータに人間の言葉を扱わせる技術の総称です。「埋め込み(embedding)」とは、記号や単語や文を、意味の近さが距離に反映される数値ベクトルに変換することを指します。

## 直感的な理解

地図要素をモデルに渡すとき、その要素が「何なのか」を伝える必要があります。素朴なやり方は、タグの種類ごとに番号を振る one-hot 表現(該当する種類だけ1、ほかは0にしたベクトル)です。ところが OSM のタグは誰でも自由に作れるため、世界中で何千種類も存在し、しかも増え続けます。これを番号で管理しようとすると次の問題が起きます。

まず、種類が膨大なので one-hot ベクトルが超高次元になります。次に、学習時に見たことのないタグ(未知タグ、out-of-vocabulary)はそもそも番号が割り当てられておらず、無視するしかありません。そして最も本質的な問題として、タグ同士の意味の近さが完全に消えます。たとえば `highway=residential`(住宅街の道路)と `highway=living_street`(生活道路)は意味がとても近いのに、番号が3番と4001番ならベクトル上は何の関係もない他人として扱われます。one-hot ベクトルはどの2つを取っても距離が等しい(直交している)ので、「似ている」という情報を構造上まったく持てないのです。

ここで「タグも結局は人間の言葉(英語の短いフレーズ)である」と気づくのが転換点です。言葉なら言語モデルで埋め込めば、「意味の近いものは近いベクトルになる」という性質が手に入ります。`highway=residential` と `highway=living_street` を近いベクトルにでき、未知タグも文字として読めば「だいたいこういう意味だろう」と推測(汎化)できます。番号という恣意的なラベルではなく、意味そのものを数値化するわけです。これは単語埋め込みで「king − man + woman ≈ queen」のような意味演算が成り立つのと同じ原理を、タグに持ち込んだものと言えます。

## 基礎: 前提となる概念

言語モデル(language model)とは、大量の文章で事前学習され、単語や文の意味を数値ベクトルとして扱えるようにしたニューラルネットワークです。古くは GloVe(Pennington et al., EMNLP 2014)や word2vec のような「単語1つに固定の1ベクトル」を割り当てる手法がありました。これらは文脈を見ないので、同じ単語はいつも同じベクトルでした。代表的な発展形が BERT(Devlin et al., NAACL 2019, arXiv:1810.04805)で、文章中の単語の前後関係を双方向に見て各単語の文脈付きベクトルを作る Transformer 系のモデルです。BERT は大量のテキストを「一部の単語を隠して当てる(masked language modeling)」課題で事前学習することで、文脈に応じた豊かな意味表現を獲得しました。

トークナイザ(tokenizer)は、文字列を言語モデルが処理できる整数 ID の列に変換する前処理器です。たとえば "highway residential" という文字列を、サブワード(単語をさらに細かく分けた単位。未知語にも対応できる)ごとに ID に直し、`input_ids`(単語の整数 ID 列)、`attention_mask`(どの位置が本物の単語でどこがパディングか)、必要に応じて `token_type_ids`(文の区切り情報)を出力します。サブワード分割のおかげで、辞書にない新しいタグでも「既知の断片の組み合わせ」として読めるので、未知タグへの汎化の基礎になります。

文埋め込み(sentence embedding)とは、1つの文(ここではタグ文字列)を1本の固定長ベクトルにまとめたものです。BERT は単語ごとのベクトルを出しますが、タグ全体を1つのベクトルで表したいので、Sentence-BERT(Reimers & Gurevych, EMNLP 2019, arXiv:1908.10084)のような枠組みを使います。これは BERT を文ペアの類似度学習でファインチューン(追加学習)し、意味的に近い文が近いベクトルになるよう調整したもので、SentenceTransformer という実装で広く使われます。具体的には、単語ベクトルを平均する(mean pooling)などして1文を1ベクトルに圧縮し、それを意味検索やクラスタリングに使えるようにしたものです。

## 仕組みを詳しく

NLP タグ埋め込みの処理は、典型的にはオフラインのトークン化とモデル内の埋め込みの2段階に分かれます。

第1段階はオフラインのトークン化です。各地図要素のタグ文字列をトークナイザに通し、`input_ids`、`token_type_ids`、`attention_mask` の3つに変換します。1つの要素が複数のタグを持つことがある(`highway=residential` かつ `maxspeed=30` かつ `lit=yes` など)ので、要素内では長さをそろえる(padding する)のが普通です。ノード・ウェイ・リレーションそれぞれに対してこの変換を行い、リレーションのメンバー(構成要素)のタグも入れ子で保持します。タグ列は要素ごとに長さが異なるため、固定形状のテンソルにはまとめず、可変長のリストとして持つことが多いです。この段階は CPU で一度だけ行い、結果を保存(キャッシュ)しておけば学習・推論のたびに再計算せずに済みます。

第2段階はモデル内での埋め込みです。トークン ID 列を SentenceTransformer に通し、1要素あたり1本のタグ埋め込みベクトルにします。次元はモデルによりますが、OSM タグ専用に学習した軽量モデルでは比較的小さめ(たとえば144次元)に設計されることがあります。タグ埋め込みの次元を小さく抑えるのは、後段で点の位置エンコーディングやグラフ識別子と結合したときに全体の次元が膨らみすぎないようにするためと、タグの意味空間がそもそも数万次元も必要としない(道路の意味のバリエーションは有限)ためです。

このタグ埋め込みは、点の位置エンコーディングやグラフ識別子と結合され、最終的に共通の埋め込み次元(たとえば256)のトークンへ射影されます。エンコーダの入力次元は、結合する各要素の次元の和として決まります。具体例で書くと次のようになります。

```
input_dim = pts_dim * pos_num_feats + nlp_embed_dim + 2 * orf_dim
          = 3 * 16            + 144           + 2 * 64    = 320
# pts_dim       : 点の座標次元(x,y,z で 3)
# pos_num_feats : 各座標を正弦波で何本に展開するか(ここでは16)
# nlp_embed_dim : タグ埋め込みの次元(ここでは144) ← これがNLP埋め込みの取り分
# orf_dim       : グラフ識別子(所属関係)の次元
```

この 320 次元を線形層で 256 次元に射影し、Transformer Encoder に通します。NLP 埋め込みはこのうち 144 次元という大きな割合を占め、要素の「意味」を担います。つまり 320 次元の入力のおよそ半分弱がタグの意味で占められており、設計者がタグ意味を重視していることが次元配分から読み取れます。

OSM タグ専用の埋め込みモデルは、汎用の言語モデルをそのまま使うのではなく、OSM タグの語彙やフォーマットに合わせて学習し直したものを使うのが効果的です。汎用 BERT は一般的な英文で学習されているため、`highway=trunk` のような OSM 独特の `キー=値` 表記やドメイン特有の意味(trunk が「幹線道路」を指すこと)を必ずしも正しく捉えません。SDTagNet 系では OSM タグ専用に事前学習した BERT 系モデルを配布・利用し、この適合ギャップを埋めます。

実装上の重要な注意として、この NLP モデルは構築時に明示的に CPU 上に置くのが安全です。SentenceTransformer のような実装は、放っておくと内部で勝手に利用可能な GPU へ移動してしまうことがあり、エンコーダ本体は GPU、NLP モデルだけ別の場所、というデバイス不一致のバグを起こします。最初に CPU で作っておけば、あとでモデル全体を `to(device)` で動かしたときに NLP モデルも一緒に移動し、普通のサブモジュールと同じ振る舞いになります。さらに、タグ埋め込みを事前計算してデータに焼き込めば、学習・推論時に言語モデルを通す必要がなくなり、この問題自体が消えます。

## 手法の系譜と主要論文

タグや道路クラスをモデルに与える表現の進化は、固定ベクトル → 文脈埋め込み → ドメイン特化、そして one-hot → 言語埋め込みという2つの流れの交点として理解できます。

GloVe(Pennington et al., EMNLP 2014)や word2vec は、単語の共起統計から「意味の近い単語は近いベクトル」になる埋め込みを学んだ初期の代表で、one-hot の限界(意味の近さを表せない)を破った原点です。BERT(Devlin et al., NAACL 2019)はこれを文脈依存・双方向に拡張し、同じ単語でも文脈で意味が変わることを表現できるようにしました。

Sentence-BERT(Reimers & Gurevych, EMNLP 2019, arXiv:1908.10084)は、文を意味的に近いものが近くなるベクトルに埋め込む基盤技術です。元の BERT は文ペアの類似度を計算するのに両方を一緒に通す必要があり遅かった(1万文から最も似た2文を探すのに膨大な組み合わせ計算が要る)のですが、Sentence-BERT は各文を独立に1ベクトルへ落とせるようにし、意味検索やクラスタリングを劇的に高速化しました。OSM タグの埋め込みはこの SentenceTransformer の枠組みをタグ向けに学習し直したものに相当します。

SMERF(Luo et al., CoRL 2023, arXiv:2311.07843)は、SD マップの道路クラスを one-hot で Transformer に入れる方式の代表です。SD マップを事前情報として使う有効性を示しましたが、タグの意味を活かしきれない点が後続研究で課題とされました。one-hot は固定クラス集合を前提とするため、OSM の無数のタグを少数のクラスに丸めざるを得ず、丸めの過程で意味が失われます。

SDTagNet(Immel et al., 2025, arXiv:2506.08997)は、SD マップ(OSM)のタグを one-hot ではなく言語埋め込みで表す方式を提案しました。動機は、OSM タグが自由形式で種類が膨大、未知タグも頻出するため固定クラスでは取りこぼすことです。タグの意味を活かし、未知・希少タグにも汎化することで、SD マップだけで HD マップに近い性能を出しました。代償として、専用の言語モデルを事前学習・同梱する必要があり、推論時もこのモデルを通す分の計算と依存ライブラリが増えます(ただし前述のとおり事前計算で回避できます)。

## 論文の実験結果(定量データ)

SDTagNet 系の評価は、nuScenes や Argoverse 2 のオンライン地図構築タスクで mAP(mean Average Precision、地図要素をどれだけ正しく検出・配置できたかの平均精度。0から1で高いほど良い)を測ります。

報告によると、タグ表現を one-hot クラスから言語埋め込みに置き換えると mAP が向上します。SD マップなしのベースラインから SD マップ(one-hot)を足すだけでも数ポイント上がり、さらにそのタグを言語埋め込みに替えると追加で上積みされる、という二段の改善が示されています。特に効果が大きいのは、学習データにほとんど現れない希少タグや、評価時に初めて現れる未知タグを含むシーンです。one-hot ではこうしたタグを未知=無視とするしかないのに対し、言語埋め込みは文字列の意味から近い既知タグへ汎化できるためです。

アブレーションでは、(1)言語埋め込みを one-hot に戻すと性能が下がる、(2)タグ情報そのものを抜いて幾何だけにするとさらに下がる、という結果が示されており、意味タグが地図理解に実際に寄与していることが分かります。さらに、(3)汎用言語モデルを OSM タグ専用に学習し直したモデルへ替えると性能が改善するという報告もあり、ドメイン適合の重要性を裏づけます。Sentence-BERT 自体の評価では、意味的文類似度(STS、文の意味の近さを人手スコアと比べる課題)で、元の BERT を平文ベクトル化したものより大幅に高い相関を達成しており、「文を独立に良いベクトルへ落とす」枠組みの有効性が示されています。

数値の意味を補足すると、mAP の数ポイントの差は、地図要素を1割前後多く正しく置けるかどうかに相当します。地図要素の位置や種別を取り違えると下流の経路計画が誤った前提で動くため、この差は安全性に直結し、無視できない重要さがあります。

## メリット・トレードオフ・限界

メリットは、自由形式タグの意味を落とさず数値化できること、未知タグ・希少タグにも文字ベースで汎化できること、そして意味の近いタグが近いベクトルになることで学習効率が上がる(似たタグは似た振る舞いを共有できるのでデータが少なくても学べる)ことです。

トレードオフと限界としては、言語モデルのダウンロード・ロード・推論が必要で依存と計算が増えること、タグの言語が英語前提(OSM の主要タグは英語キーワード)のため地域固有言語のタグには弱い可能性があること、そして事前学習済みモデルの品質と配布元の維持に依存することが挙げられます。研究上の未解決課題としては、推論時の言語モデル通過コストをどう削るか(タグ埋め込みを事前計算してキャッシュする、軽量蒸留モデルを使う等)、多言語タグへの対応、そして複数タグが付いた要素の意味合成(`highway=residential` と `maxspeed=30` と `surface=cobblestone` を1ベクトルにどうまとめるか)をどう正確に表現するかが残っています。タグを単純連結して1文として埋め込むと順序や重みの情報が曖昧になり、タグごとに埋め込んで集約するとどう重み付けるかが新たな設計問題になります。

## 発展トピック・研究の最前線

タグの言語埋め込みは、近年の VLM(Vision-Language Model、画像と言語を同時に扱うモデル)を運転に取り込む流れと自然につながります。地図のタグ、カメラ画像、自然言語の運転指示を共通の埋め込み空間で扱えれば、「学校の前では徐行」のような言語ルールを地図要素に直接結びつけられます。また、タグ埋め込みを事前計算してデータセットに焼き込む最適化や、軽量な専用埋め込みモデルへの知識蒸留(大きいモデルの振る舞いを小さいモデルに移す技術)は、リアルタイム運転での実用化に向けた研究の焦点です。さらに、タグの埋め込み空間を可視化・解釈する研究は、モデルが地図の意味をどう内部表現しているかを理解する手がかりになり、安全性の検証にもつながります。
