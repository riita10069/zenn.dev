# 大規模マルチカメラ運転データセットのパースと前処理

## ひとことで言うと
自動運転のデータセットには、複数台のカメラで撮った長い動画と、その間に車がどう動いたか(位置・速度・姿勢など)の記録が、別々のファイル・別々の時刻管理で入っています。これをニューラルネットワークがそのまま食べられる形、つまり「決まった形(shape)を持つ数値の塊(テンソル)」に変換する処理が「パース・前処理」です。動画から必要な瞬間のフレームだけを取り出し、車の動きの記録から正解ラベルを切り出し、両者の時刻を合わせて、forward pass(順伝播、入力から出力を一度計算する流れ)に渡せる形にそろえます。地味な裏方ですが、ここが間違っているとモデルがどれだけ高性能でも学習は根本から成立しません。

## 直感的な理解
人間が運転教習を受ける場面を想像してください。教官は「あの瞬間、前方に歩行者がいたのに、あなたはアクセルを踏んでいた」と指摘します。このとき教官の頭の中では「あの瞬間の視界(カメラ画像)」と「そのときの操作(車の動き)」が時間的にぴったり対応づいています。

機械学習で運転を学ばせるときも、まったく同じ対応づけが必要です。「この映像を見ていたとき、正しい運転はこうだった」というペアが大量にないと、モデルは何を真似ればよいか分かりません。ところが現実のデータは、カメラは秒30枚の映像として、車の動きは秒100回の数値記録として、しかも互いにずれた時計で別々のファイルに保存されています。そのままでは「同じ瞬間」を取り出せません。パース・前処理とは、この散らばった記録から「同じ瞬間の視界と正解操作のペア」を機械的に大量生成する作業です。

ここでよくある誤解は「データの読み込みなんて単純作業だ」という思い込みです。実際には、時刻同期のずれ・ファイル欠損・座標系の取り方・前処理統計の不一致など、目に見えにくい落とし穴が多く、それらが「エラーは出ないのに精度だけ静かに落ちる」という最も発見しにくいバグを生みます。前処理は、研究の再現性と性能の両方を左右する第一級の設計問題です。

## 基礎: 前提となる概念

理解のために必要な用語を、知らない前提で噛み砕きます。

- テンソル(tensor): 数値が格子状に並んだ多次元配列のこと。画像は「縦×横×色チャンネル(RGBの3)」の3次元テンソル、それをまとめてバッチにすると4次元になります。形を `(N, C, H, W)`(バッチ数, チャンネル, 高さ, 幅)のように書きます。

- フレーム(frame): 動画は静止画(フレーム)を高速にパラパラめくったものです。秒30フレーム(30fps)なら、1秒の動画に30枚の画像が入っています。

- コーデックとキーフレーム: 動画は圧縮されて保存されます(H.264/H.265 など)。圧縮では「完全な画像(キーフレーム、I フレーム)」と「直前との差分だけを持つフレーム(P/B フレーム)」が混在します。任意の1枚を取り出すには、その手前のキーフレームまで遡ってから差分を順に適用する必要があり、これが「ランダムアクセスは案外重い」理由です。

- タイムスタンプ(timestamp): 「いつ起きたか」を表す時刻の数値。多くのデータセットはマイクロ秒(μs、100万分の1秒)単位の整数で記録します。例: `1699891234567890`。

- 自車運動(egomotion): 自分の車(ego vehicle)がどう動いたかの記録。位置 `(x, y, z)`、姿勢(向き)、速度 `(vx, vy, vz)`、加速度などを含みます。「ego」は「自分の」という意味で、他車(周囲の車)と区別します。

- クォータニオン(quaternion): 3次元空間での回転(向き)を4つの数 `(qx, qy, qz, qw)` で表す方法。オイラー角(ロール・ピッチ・ヨー)より、ジンバルロック(特定姿勢で自由度が失われる現象)が起きず計算が安定するため、姿勢の記録によく使われます。後で「ヨー角(進行方位)」だけ取り出すために、クォータニオンから角度へ変換します。

- 同期(synchronization): 別々の時計を持つセンサ同士で「同じ瞬間」を対応づけること。カメラと自車運動でタイムスタンプが完全一致することはまずないので、「最も時刻が近いもの」を選ぶのが基本戦略です。

- 較正(calibration): カメラの内部パラメータ(焦点距離・主点・歪み)と外部パラメータ(車体に対する取り付け位置・向き)。画像座標と3次元の世界座標を行き来するのに必要で、データセットには通常これらが添付されます。

- 前処理(transform): 生の画像を、モデルが期待するサイズ・明るさにそろえる加工。具体的にはリサイズ(例: 256×256 に縮小)と正規化(画素値を、事前学習時に使われた平均・分散で標準化する)を行います。

- Dataset / DataLoader: 機械学習フレームワークで、データを1サンプルずつ供給する標準インターフェースが `Dataset`、それをバッチにまとめて並列に取り出すのが `DataLoader` です。`Dataset` は「i番目のサンプルをくれ」と言われたら1個返す約束(`__getitem__`)を実装します。

## 仕組みを詳しく

代表的なマルチカメラ運転データセット(NVIDIA 系の Physical AI データセット、nuScenes、Waymo Open Dataset、Argoverse など)のパースは、おおむね次の段階を踏みます。

### 1. どのカメラを、何ビュー使うか決める

車の周囲を死角なく捉えるため、複数のカメラを異なる向き・画角で取り付けます。典型的な7カメラ構成は次のようになります。

```
前方広角  (front wide 120度視野)   交差点・近距離の歩行者
前方望遠  (front tele 30度視野)    遠方の信号・標識
左クロス  (cross left 120度)       左折時の死角
右クロス  (cross right 120度)      右折時の死角
後方左    (rear left 70度)          車線変更時の後方
後方右    (rear right 70度)          車線変更時の後方
後方望遠  (rear tele 30度)          高速での後方接近車
```

画角(視野角)を変えるのは、広角は近くを広く、望遠は遠くを狭く詳しく見るためです。モデルにはこれら全カメラを1つの軸にまとめて渡します。地図情報(俯瞰のマップタイル)を別ビューとして加える設計もあり、その場合「7カメラ + 1マップ = 8ビュー」のように扱います。ただし地図のレンダリング画像が用意できない場合は、8枚目をゼロ埋めのダミーにして、後で本物に差し替える設計がよく取られます。これは「インターフェース(テンソルの形)を先に固定し、中身の実装は後から差し替える」という、開発を段階的に進めるための定石です。

### 2. カメラフレームの取り出しと時刻合わせ

ある基準時刻(たとえば自車運動のあるサンプル時刻 `t_ego`、μs単位)が与えられたとき、各カメラについて「その時刻に最も近いフレーム」を選びます。

```python
frame_idx = int(np.argmin(np.abs(camera_timestamps_us - t_ego)))
```

`np.abs(camera_timestamps_us - t_ego)` は各フレーム時刻と基準時刻の差の絶対値の配列、`np.argmin(...)` はその最小値の位置(インデックス)を返します。つまり「いちばん時刻が近いフレーム番号」が得られます。

ここで時刻ずれの大きさを数値で感じておきましょう。30fps のカメラはフレーム間隔が約33ms です。最近傍を選んでも、最悪で半フレーム=約16ms のずれが残ります。時速60km(=約16.7 m/s)で走っていると、16ms は約27cm の移動に相当します。低速の市街地ではほぼ無視できますが、高速走行では「画像が示す位置」と「自車運動が示す位置」が数十cm ずれることになり、軌跡予測の精度に効いてきます。これが後述する同期精緻化の研究動機です。

そのフレーム番号だけを動画からデコード(圧縮された動画を画像に展開)します。動画全体を展開すると無駄が多いので、目的のフレームへ飛んで(シーク)1枚だけ取り出すのが効率的です。

```python
rgb = video_reader.decode_frame(frame_idx)   # H×W×3 のRGB画像
tile = transform(rgb)                         # (3, 256, 256) に整形
```

これを7カメラぶん集め、ゼロのマップタイルを足して1本のテンソルにまとめます。

```
tile_0 .. tile_6  各 (3, 256, 256)   7カメラ
tile_7            (3, 256, 256)       マップ(ゼロ)
stack →           (8, 3, 256, 256)    8ビュー × チャンネル × 高さ × 幅
```

ここで重要な実務上の落とし穴があります。学習ループの毎ステップで動画をバイト列から読み込んでデコードすると、ディスクI/OとCPUデコードが律速になり、高価なGPUが入力待ちで遊んでしまいます。このため、本番の大規模学習では「事前抽出(pre-extraction)」が定石です。必要フレームをあらかじめ画像やテンソルに展開し、`WebDataset` のようなシャード(tarアーカイブの連番)形式で保存しておき、学習時は逐次ストリーミングで読みます。オンザフライのデコードは小規模実験やデバッグ向き、事前抽出は本番向き、という役割分担です。前述のキーフレーム構造のせいでランダムアクセスのデコードは特に重く、事前抽出で「順次デコードして連番保存」しておくと、この重さを学習ループの外へ追い出せます。

### 3. 自車運動(egomotion)のパース

自車運動は列指向ファイル(parquet 形式が一般的。列ごとに連続配置されるため、必要列だけを高速に読める)に高頻度で記録されます。典型的な列は次の通りです。

```
timestamp, qx, qy, qz, qw, x, y, z, vx, vy, vz, ax, ay, az, curvature
```

100Hz(秒100行)で記録されていても、モデルが使いたいのは10Hz程度であることが多いので、間引き(ダウンサンプリング)します。100Hzから10Hzなら10行に1行だけ残します。

```python
DOWNSAMPLE_STEP = 100 // 10   # = 10
rows_10hz = rows_100hz[::DOWNSAMPLE_STEP]
```

ここから「過去(履歴)」と「未来(正解)」を切り出します。例として、ある時刻を中心に過去64フレーム・未来64フレームを取るとします(10Hz なら過去6.4秒・未来6.4秒に相当)。

- 履歴(過去64フレーム)から運転状態を表す信号を作る。たとえば `[速さ, 加速度, ヨー角, 曲率]` の4信号 → 形 `(64, 4)` → 平らにして `(256,)`。
- 未来(後ろ64フレーム)から予測すべき軌跡を作る。たとえば `[加速度, 曲率]` の2信号 → `(64, 2)` → 平らにして `(128,)`。

```
EGOMOTION_DIM  = 64 * 4 = 256   # モデルへの入力(過去の運転状態)
TRAJECTORY_DIM = 64 * 2 = 128   # モデルが当てるべき正解(未来軌跡)
```

各信号の作り方も具体的です。速さは水平方向の速度成分から `speed = sqrt(vx^2 + vy^2)` で求めます。ヨー角(進行方位、車がどちらを向いているか)は姿勢クォータニオン `(qx, qy, qz, qw)` をオイラー角(ZYX順)に変換したときのZ成分です。

未来の軌跡を「座標列 `(x, y)`」ではなく「加速度と曲率の時系列」で表す設計には理由があります。座標は基準フレームの取り方に依存し、わずかな較正誤差で大きくぶれます。一方、加速度と曲率は車の運動そのもの(どれだけ加減速し、どれだけ曲がるか)に近い量で、制御に直結し、座標系の取り方に頑健です。曲率 `κ`(カッパ)は「進行方向が単位距離あたりどれだけ回るか」で、半径 `R` の円弧を描くなら `κ = 1/R` です。直進なら `κ = 0`、急カーブほど大きくなります。ただし最終的に地図上の経路が必要な用途では、加速度・曲率から座標列を積分し直す変換が要り、その積分過程で誤差が累積しうる、という裏返しのコストもあります。

### 4. Dataset としての組み立てと有効サンプルの事前確定

`Dataset` を作るとき、全シーン(連続走行のひとまとまり、clip と呼ぶ)から「学習に使える有効なサンプル点」だけをあらかじめ平らなリストに展開しておくのが効率的です。

- シーン検証: 必要な全カメラの動画とタイムスタンプ記録がそろっているか。1つでも欠けていればそのシーンを丸ごと除外。
- サンプル検証: 自車運動ファイルに必要な列があるか。間引き後に最低限の行数(過去・未来を切り出すのに足りる長さ。たとえば過去64+現在1+未来64=129行以上)があるか。タイムスタンプが厳密に単調増加しているか(時刻の巻き戻りや重複は記録破損のサインなので除外)。
- 合格したサンプルを `(clip_id, sample_idx, timestamp)` の形でリストに貯める。

こうしておくと `__getitem__(i)` は「i番目のサンプル情報を引いてI/Oするだけ」になり、毎回ファイル全体を読み直す重い処理が消えます。返すサンプルの典型的な辞書は次の形です。

```
visual_tiles       (8, 3, 256, 256)  7カメラ + マップタイル
egomotion_history  (256,)            過去の運転状態
trajectory_target  (128,)            未来の正解軌跡
clip_id            str
sample_idx         int
```

長期的なシーン記憶(過去フレームの圧縮要約)を入力に加える設計もあり、その場合はメモリ用の追加テンソルを持ちますが、要約エンコーダが未実装の段階ではゼロ埋めのプレースホルダにしておき、後で接続するのが現実的な進め方です。

### 5. 前処理をモデル設定から自動決定する工夫

画像の前処理(リサイズ・正規化)は、使う画像エンコーダ(backbone)の事前学習設定から自動生成するのが堅実です。手で平均・分散の数値を書き写すと、モデルが期待する値と1つでもずれた瞬間に、エラーは出ないのに精度だけ静かに落ちる、という発見しにくいバグになります。たとえば ImageNet 事前学習の標準統計は平均 `(0.485, 0.456, 0.406)`、標準偏差 `(0.229, 0.224, 0.225)` ですが、これを 0〜1 スケールのまま正規化を忘れると入力分布が事前学習時と乖離し、転移性能が落ちます。

```python
_backbone = create_model(backbone_name, pretrained=False)  # 構成だけ取得
data_config = resolve_model_data_config(_backbone)          # 期待する前処理
transform   = create_transform(**data_config, is_training=False)
```

要点は「前処理の真実の源(source of truth)を1か所(backbone)に置く」ことです。これでモデル側とデータ側の前処理が原理的にずれません。

## 手法の系譜と主要論文

データのパース・前処理は、End-to-End 運転研究の歴史と一緒に発展してきました。

- Pomerleau, "ALVINN: An Autonomous Land Vehicle in a Neural Network"(NeurIPS 1989)が最初期の試みです。前方画像から操舵を出す小さな3層ネットワークで、入力をどう整形するか(画像を 30×32 に縮小して入力するなど)という前処理の問題はこのときから本質でした。

- Bojarski et al.(NVIDIA), "End to End Learning for Self-Driving Cars"(2016, arXiv:1604.07316)は、単一の前方カメラ画像から直接ステアリング角を回帰する現代的E2Eの原点です。実車で約98%の時間を自律走行できたと報告し、人手の特徴設計なしに画像→操作が学べることを示しました。ここで使われた「画像を切り出し、正規化し、操舵ラベルと対応づける」処理が、まさに本ノートのパースの原型です。彼らは直進バイアスを避けるため、左右のオフセットカメラ画像に補正した操舵ラベルを与えるという「データ拡張兼ラベル生成」も行っており、前処理がモデル性能に直結することを早くから示していました。

- Caesar et al., "nuScenes"(CVPR 2020, arXiv:1903.11027)と Sun et al., "Waymo Open Dataset"(CVPR 2020, arXiv:1912.04838)は、多カメラ + LiDAR + 自車運動を含む大規模データセットの「作法」を確立しました。具体的には、センサ間のタイムスタンプ管理、最近傍同期、較正(カメラの内部・外部パラメータ)の配布、キーフレーム(注釈を付ける基準時刻)の概念です。nuScenes は1000シーン・6カメラ・各20秒、Waymo は約1150シーン・5カメラを含み、その後のほぼ全てのマルチカメラ手法のベンチマーク基盤になりました。

- Wilson et al., "Argoverse 2"(NeurIPS 2021)は、運動予測に特化した大規模軌跡データセットで、「過去軌跡から未来軌跡を当てる」という本ノートの履歴/未来切り出しの枠組みを標準化しました。

- Hu et al., "Planning-oriented Autonomous Driving (UniAD)"(CVPR 2023, arXiv:2212.10156)は、知覚から計画までを1つのネットワークで貫く設計で、入力に複数カメラ画像列を、出力に未来軌跡を取る構成が広まりました。こうしたモデルの学習を支えるのが、本ノートの「マルチカメラ画像 + 自車運動を整合させたサンプル供給」です。

- 近年は Hugging Face の LeRobot(Cabon et al., 2024)のように、ロボット・運転の軌跡データを統一フォーマット(エピソード単位の parquet + 動画)で配布・読み込みする枠組みも登場しました。実装は様々ですが、「動画 + センサ時系列を、決まった形のテンソルに落とす」という設計思想は全て共通しています。

## 論文の実験結果(定量データ)

パース・前処理そのものは独立に数値評価されることは少ないですが、関連する定量的事実を挙げます。

- Bojarski et al.(2016)は、人間の介入なしに走行できた時間の割合(autonomy)を `autonomy = 1 - (介入回数 × 6秒) / 走行時間` と定義し、約98%を達成しました。これは「画像→操舵の直接学習が現実に機能する」という最初の定量的証拠で、データ対応づけの正しさがそのまま性能に効くことを示しています。6秒という定数は「介入から人が再び自律に戻すまでの平均時間」の見積もりで、指標の意味を理解せず数字だけ追うと比較を誤る一例です。

- データセット規模の比較: nuScenes は約140万枚のカメラ画像と約39万点の同期 LiDAR スイープ、140万件の3Dボックス注釈を含みます。Waymo Open は約100万フレーム規模で、より高密度の注釈を持ちます。規模が桁違いに増えたことで、知覚タスクの平均精度(mAP, mean Average Precision、検出の正しさを0〜1で測る指標。1に近いほど良い)が手法間で比較可能になり、研究が加速しました。

- 前処理の正しさが効くことを示す事実として、画像の正規化統計(平均・分散)を事前学習時とずらすと、ImageNet 事前学習モデルの転移性能が体感できるほど劣化することが広く知られています。具体的な低下幅はモデル依存ですが、数ポイントの精度低下が静かに起きるため、「前処理を backbone 設定から自動取得する」設計が推奨されます。

- I/O の重さに関する実務的事実: 動画をオンザフライでデコードする方式は、GPU 利用率(GPU が実際に計算している時間の割合)を大きく下げることがあります。報告例では、入力パイプラインが律速だと GPU 利用率が数十%にとどまり、事前抽出 + シャードストリーミングに切り替えると 90% 超まで改善する、という改善が観測されます。学習時間が同じハードで2〜3倍変わることもあり、これがマルチカメラ学習で事前抽出が標準になる理由です。

## メリット・トレードオフ・限界

- メリット: 大規模で雑多なデータを、形のそろったテンソルに統一供給でき、forward pass がそのまま回る。
- メリット: 前処理を backbone 設定から自動生成すれば、モデルとデータの前処理ずれという発見しにくいバグを構造的に防げる。
- メリット: 有効サンプルを構築時に確定すれば、`__getitem__` が軽くなり、学習中のCPUオーバーヘッドが小さい。
- メリット: シーン・サンプル検証(ファイル欠損・列欠損・時刻巻き戻り)で壊れたデータを自動除外でき、学習が途中で落ちにくい。
- トレードオフ: 最近傍タイムスタンプ同期は厳密な同時刻ではなく、カメラ間・カメラと自車運動の間に微小な時間ずれが残る。前述の通り高速走行ではこのずれが空間的なずれ(数十cm)に増幅される。
- トレードオフ: 軌跡を加速度・曲率で表すと座標系に頑健だが、地図上の経路が欲しい用途では積分による座標復元が必要で、誤差が累積しうる。
- トレードオフ: 事前抽出は学習を速くするが、ストレージ消費が増え、抽出設定(解像度・カメラ選択)を変えるたびに再生成が必要になる。
- 限界: 地図タイルやシーン記憶をゼロのプレースホルダで埋めている段階では、地図文脈や長期記憶を使う性能はまだ引き出せない。
- 未解決の課題: 同期の高度化(時間ずれを補間で補正する、運動ぼけや rolling shutter を考慮する)、複数センサ較正の自動検証、長距離依存を保つためのストリーミング設計、希少イベント(事故・割り込み)の偏りを補うサンプリングなどが研究対象として残っています。

## 発展トピック・研究の最前線

- ストリーミング/オンライン学習: 走行データは時間順に到来するため、ランダムシャッフルと相性が悪い面があります。シャードをまたいだバッファシャッフルや、時間相関を保ちつつ偏りを減らす供給法が研究されています。
- 自己教師あり前処理: 注釈の少ないデータから、未来フレーム予測や対比学習で表現を学ぶ手法(BEV 表現の自己教師ありなど)が広がっています。パースが供給する「過去・現在・未来の整合した系列」が、こうした学習の土台になります。
- マルチモーダル同期の精緻化: カメラ・LiDAR・レーダ・自車運動を、単なる最近傍ではなく時刻補間や運動補償で正確に揃える研究。とくに高速移動時の rolling shutter(画像が一瞬で撮られず行ごとにずれる現象)の補正は重要課題です。
- データバージョニング: 大規模データの再現性を保つため、データセット自体をコード同様にバージョン管理する仕組み(データレイク + バージョニング)が運用面で標準化しつつあります。同じコードでもデータが変われば結果が変わるため、「どのデータ版で学習したか」を固定できることが再現性の前提になります。
- 合成データとシミュレーション: 希少シーンを物理シミュレータ(CARLA など)やニューラルレンダリングで生成し、実データの偏りを補う研究。生成データもまた同じテンソル形式に落とす必要があり、パースの抽象化が効いてきます。
