## AWS における Solutions Architect

AWS の Solutions Architect とはどんな仕事なのか分からない、と言われることは少なくありません。これは、AWS における Solutions Architect のロール定義が「何をするか？」ではなく「何を得るか？」で表現されていることに由来していると考えています。

一緒に働いたことのある人から見ると、SA はセールスエンジニアにも、技術サポートエンジニアにも、プロトタイピングエンジニアにも、あるいは事業開発やマーケティング、エバンジェリストのような仕事にも見えると思います。実際、それらはある意味すべて正解です。

「何をするか？」という観点で言えば、お客様の技術相談にも乗りますし、お客様と一緒にプロトタイプを実装することもあります。自社製品の紹介や活用提案もしますし、AWS サービスチームへの機能リクエストや、新サービスや新機能の PR/FAQ の作成にも関わります。

これらは実際に私自身が取り組んでいることです。もちろん SA によって得意・不得意はありアプローチの仕方はさまざまかと思いますが、活動は多岐にわたります。

しかし、そうした活動を並べていくと「さすがに範囲が広すぎないか？」と感じるのではないでしょうか。

それは、AWS における Solutions Architect という職種を「何をするか？」という切り口で見ているからだと考えています。実際には、「何を得るか？」という切り口で定義されており、やるべきことについては基本的に手法が固定されているわけではありません。  
社内には一定の営業方針やアプローチに関する考え方は存在しますが、それらは教科書的なフレームワークであり、前述したようなさまざまな活動を肯定するための基盤に過ぎません。

では、AWS の SA は「何を得るためのロール」なのでしょうか。  
Solutions Architect のゴールは、自社製品をお客様に**長期的に**活用していただき、**利用額**を生み出すことです。

AWS では SA の KPI に売上金額は設定されていないため、「売上は気にしなくてよい」と考える人もいるかもしれません。確かに詳細な金額としての目標値は定められていないので、具体的な数字に執着する必要はありません。しかし、SA がセールスチームの一員である以上、利用額を増やすために雇われており、利用額に貢献するための職種であり、それ以上でもそれ以下でもありません。

では、なぜ AWS では SA の KPI に利用額が設定されていないのでしょうか。  
それは、SA が「技術者として中長期的な AWS の利用額に貢献すること」を求められているからです。

今年の KPI として 10 年後の利用額を設定することはできません。一方で、今年 1 年の売上だけを追いかけると、長期的なプラットフォームとしての成功につながらない活動が増えてしまう懸念があります。そのため AWS の SA は利用額を KPI にしていませんし、短期的な利用額のために長期的な利用額を損なうような活動は行いません。

## AWS の SA がやらない2種類のセールスプレイ - 「プロダクト営業」「短期的な利益」

### ❌ 製品の強みを説明するセールスプレイ

まず先に、プロダクト営業スタイルとはどういったセールスプレイでしょうか。
自社製品の強みや Value Proposition を掲げ、お客様に自社製品の強みを理解していただいて、購買/利用へ繋げるような方法をここでは指しています。
一般的な営業の手法だと思いますが、AWS ではほとんどこのような方法を取りません。なぜなら、そんなことしても AWS は売れないからです。

対面するお客様が一般的なビジネスパーソンや IT エンジニアであるとして、たとえば「Amazon S3 はイレブンナインのデータ耐久性を持つ優れたストレージです」とプロダクト営業をしている場面を想像してみてください。  
もちろん Amazon S3 自体は競争力のある良い製品ですし、ストレージのデータ耐久性に課題を抱えているインフラエンジニアであれば、その魅力を理解するかもしれません。

しかし、多くのビジネスパーソンにとっては、ストレージのデータ耐久性が高まったところで、自社のビジネスがどう好転するのかを直接イメージすることは難しいでしょう。それでは、利用にはつながりません。

これは、AWS の「売り物」がインフラであり、お客様のビジネス課題を直接解決するものではなく、その上で動く IT システムを支える存在であることに起因します。  
自社製品の魅力を語るだけでは利用につながりにくく、IT インフラという特殊な対象を売るうえで、プロダクト営業スタイルはセールスプレイとしての価値がほとんどないのです。

最近の AWS は、よりビジネスユーザーを対象としたプロダクト(QuickSuite, Connect etc..)も提供しているのは事実です。そのようなプロダクトが**本当の意味でお客様の課題を解決するものであれば**、製品の機能や強みを紹介するところから利用に繋げていくセールスプレイを行うこともあります。
ただしそれは、お客様の現状と課題を十分に理解し、「該当製品が本質的にお客様の課題を解決し、ビジネスの成功に貢献する」と判断できた場合に限られます。判断をせず、いきなりプロダクト営業スタイルでお客様と会話を始めることは絶対にありません。
あくまで、Issue Driven なセールスプレイの中のフェーズとして、製品の強みを紹介し、ご理解いただく場面が現れるだけです。
というのも、AWS は IT インフラを売り物にしているため、類似の課題を解くためにアプローチできる製品群、ソフトウェア群が多岐に渡ります。

例えば、
- 上記のようなビジネスユーザー向けの AWS サービスを利用する
- 完全スクラッチでアプリケーションを構築する
- OSS プロダクトを EC2 Marketplace や EKS を用いてホスティングする
- AWS を利用している 3rd Party の SaaS ベンダーが提供する製品を利用する

上記のような様々な選択肢から、お客様にとって最も価値があり、コスト効率の良いアプローチを SA はお客様と一緒に模索します。
どれを選択していただいても AWS は儲かるので、最もお客様にとって価値のあるソリューションを選択します。それによって長きにわたって多くの利用者にご利用いただく方が長期的な観点から AWS の成功につながるのです。

### ❌ 短期的な利益を追求したセールスプレイ

そもそも、短期的な利用額のためのセールス活動とは何でしょうか。いくつかのパターンが考えられますが、分かりやすい例として「コスト最適化」があります。

AWS の SA や TAM は、最もコスト効率の良いアーキテクチャや製品を提案し、継続的にコスト最適化の支援を行います。短期的なスパンで見れば、これらの活動によって AWS の利用額は下がります。そのため、「SA は利用額のために働く」と言っていた話と矛盾しているようにも見えるかもしれません。

しかし、これらの活動は長期的に AWS の利用額を増やすことに貢献します。ここには大きく二つの観点があります。一つ目は AWS 自体の競争力、二つ目は AWS ユーザーの競争力です。

AWS 自体の競争力については、比較的イメージしやすいと思います。端的に言えば、お客様の中で特定のシステムが生み出す価値に対して、より少ない金額でインフラを提供できることは AWS の競争力につながります。

もう少し具体的に言うと、それが業務管理システムだとしましょう。そのシステムがお客様のビジネスにもたらす価値自体は、インフラ費用にかかわらず一定です。このとき、インフラ費用が安ければ安いほど、長期的に見たときにお客様が別の SaaS や他のクラウドプロバイダーへ乗り換える理由は金額面から小さくなります。そのため、セールスチームは「同じ価値を出せるなら、できるだけ安い AWS 利用料金で提供する」ことに力を注ぎます。その結果、長期的に見たときの AWS の競争力を維持できます。

そして、コストを最適化して空いた予算で別のシステムに投資していただく、というのが基本的な考え方です。別システムへの投資によって、そのシステムでも AWS 利用料が発生する可能性がありますが、それ以上に重要なのは「AWS ユーザーの競争力が高まること」です。

AWS はグローバルなクラウドプロバイダーとして大きなシェアを握っています。システムを動かす前提として、インフラとしてオンプレミスを選ぶか、AWS を選ぶか、あるいは他社クラウドを選ぶかといった少数の選択肢から選ぶことが一般的になりつつあります。

このとき、AWS の利用額を増やすための手段として「世界中のシステムのうち、AWS を採択している割合を高める」という観点もあるでしょう。しかし、もっと効率的な方法は、「すでに AWS を IT インフラとして採用している企業のビジネスが成功すること」です。

営業として努力して AWS をご利用いただいても、その企業自体のビジネスがうまくいかなければ、利用額は徐々に減っていきます。逆に、その企業が大きく成功すれば、利用額は天文学的な数字まで跳ね上がります。

そのため、短期的な利益のみを追求するセールス手法ではなく、お客様のビジネス課題を本質的に解決し、お客様に競争力を高めていただくことが何より重要です。お客様のビジネスがうまくいけばいくほど、AWS にかかる費用も増えるのです。

このあたりが AWS のセールスプレイが Customer Obsession を重視していると言われる理由の一つだと思います。

## AWS の SA のセールスプレイはエンジニアリングの観点でお客様のビジネス課題を解決すること

前項で述べたことを合算すると AWS の SA が常に考えていることは、「お客様のビジネスを成功に導くためのソフトウェアを作っていただくこと」です。

実際に AWS の SA が提案するのは、お客様の事業を成長させ、業務を効率化し、何らかのメリットをもたらす「ソフトウェアの構築」です。最も近いイメージの営業スタイルがあるとすれば、IT コンサルかもしれません。
なぜなら、先ほど述べたように、AWS が提供する S3 や Aurora といったサービスは、それ自体ではお客様のビジネスに直接的な価値をもたらしません。その上で動くソフトウェアがお客様のビジネスに直接的に貢献するものです。
そのため、SA は「どのようなソフトウェアを作るべきか」をお客様へ提案します。お客様にとって価値のあるシステムを提案できれば、そのシステムが AWS 上で動き続けるかぎり、半永続的な利用額をもたらします。そのシステムが半永久的に稼働し続けるものであれば、単なる継続収入源になるだけでなく、お客様のビジネスの成長に応じて利用額も増えていきます。
したがって、SA の「売り物」はソフトウェアであればなんでも良いのです。特定の業務改善のための機械学習モデルの構築かもしれませんし、AIOps の提案やカオスエンジニアリングの導入かもしれません。どんなソフトウェアでもいいのです。
大切なのは、お客様のビジネスを前進させ、好転させることです。そして、その提案がお客様のビジネスを前進させられたのであれば、その分だけ AWS の利用額も増えます。

逆に言えば、それ以外の方法で長期的な AWS の利用額を増やすことはできません。本当に価値のあるシステムを AWS のインフラの上に乗せてもらうことだけが、長期的に AWS の利用額を大きく伸ばす方法であり、AWS の競争力を高める方法です。それ以外の方法を取った瞬間に、長期的な競争力を失う要因になりかねません。なぜならお客様は本当に価値のあるものにしか長期的にお金を払わないからです。これは一般的なサブスクリプションモデルのチャーンレートの話と同じだと思います。AWS ではここに従量課金が乗ってくるので、お客様のビジネスの成功はさらに重要になってくるわけですが。

そこから考えると、SA がやるべきセールスプレイはどのようなセールスプレイになるでしょうか。
お客様の役職や役割に応じて、あらゆる選択肢の中からお客様の現状を見つめ、事業提案をしていくことが SA が行うプリセールスフェーズの主な役割になります。ちょっとイメージが湧きづらいかもしれないですがあくまでも例ですがいくつか挙げてみます。

1. (As-Is) スキャンした資料からの文字の書き起こし業務を行なっており、それらに莫大な工数と委託費用がかかっているということがわかった。(To-Be) それを OCR や VLM を用いてなんとかできないかと考えて、簡単なデモを構築しご提示する。
2. (As-Is) お客様の複数組織において類似の BI 基盤を複数作っており、同じデータを複数のデータベースで管理していることを現場支援から確認したとする。現状のシステムの構成図からお客様組織内でのステークホルダーマッピングまで、データ基盤構築だけでなく利用者目線も含め調査を繰り返し資料にまとめる。(To-Be) 改善提案として改善後のアーキテクチャの例や Iceberg, ビジネスカタログなどを取り入れたレイクハウスアーキテクチャを提案し、長期的なデータストラテジーを打ち立てる。
3. (To-Be) お客様の業界における業界動向をまとめ、今後はサブスクリプション収益を上げる必要があることがわかったとする。具体的なソフトウェアにおける付加価値の例をいくつか示し、長期的なビジネスモデルを打ち立てる。
4. (To-Be) NeurIPS や CVPR など国際学会て提案されている最新手法を研究し、今後のモデル開発手法についてまとめて紹介。その中から次世代の ML モデル構築プロジェクトを打ち立てる。

これらの具体例をもとに抽象化して考えると SA はお客様の現場の現状 (As-Is) を AWS のセールスというポジションを活用し、縦にも横にも広く理解することで、お客様の立場では見えない View を確保できます。この View でしか見えない課題発見と改善提案を打ち立てます。普段私が作っている資料は、コンサルティングファームが作成するような概念図を用いて未来について語るわけですが、彼らとはちょっと立場が違うように思っています。彼らは本当にゼロから業界トレンドを作り出している側です。我々はその業界トレンドを参考にしていて、お客様の現場と直接の対話を繰り返しながら現状とのギャップを特定し、現実的な個社向けの解の一つをご提案させていただくような立場です。

そのため、NeurIPS や CVPR など国際学会や各社コンサルティングファームや Gartner の提供している業界動向を常にウォッチし次にやるべきことは何か (To-Be) を考えます。
考えた (As-Is) と (To-Be) をお客様にわかりやすく伝えられるよう、PowerPoint などで概念図を数多作成し我々の考えていることを、最大限具体的かつわかりやすくお伝えします。実際にプロジェクトとして始めていただけるかは、我々目線の広く浅い理解ではなく、お客様の各関係者を合わせた深い View によって決定されることになります。実際にプロジェクト化されるに値しない提案となってしまっていても、調査や提案のフェーズでクオリティの高い活動をすることでお客様から AWS のセールスチームは彼らのビジネスを少しだけでも前に進めてくれる存在だと認識いただけるように努めることが重要と考えています。
また、経営メンバー向けには、広い部署をまとめあげた5年後の長期的なシステム構成 Overview を打ち出す一方で、現場の方々には、半年後、一年後の具体的なシステムアーキテクチャを打ち出します。お相手のお客様の影響範囲やご関心を意識した上でご提案させていただいています。
これらの将来像を描くご提案の次に来るのは「構築」ではなく、「構想フェーズ」や「要件定義」のフェーズになります。この辺りをうまくまとめ上げて「具体化」することも SA らしいタスクになります。実際のところ SA をしていて最も多いフェーズはこちらになるように思います。

また、前述のようなプロアクティブな提案活動はもちろんしていくのですが、実際にプロジェクト化するということは成功事例に限られ、私の提案がそのままプロジェクトとして立ち上がったというのは少数です。
多くの場合は、お客様内ですでに重要プロジェクトとなっているビジョンについて、SA が深く入り込み「構想フェーズ」と「具体化フェーズ」をご支援させていただくという関わり方の方が私の場合は多いです。

結局のところプロジェクトの成功こそがチャーンを防止しお客様の組織、ビジネスに貢献できるため、このフェーズにおいても SA はソフトウェアエンジニアとしてプロジェクトを成功に導けるよう案件の具体化に全力を注ぎます。
この場面でも求められるタスクは、 (As-Is) と (To-Be) をお客様にわかりやすく伝えるというところで前述の場合と似たような形になってきます。

基本的には特にソフトウェアエンジニアリングの観点でお客様がご検討すべき事項を MECE に列挙し、それぞれ対策をご検討いただく、あるいは手法をご提案させていただくということがお客様にご期待いただく SA の役割になります。
つまり、プロジェクトの最終的な「あるべき姿を描く」→「現状の構想とのギャップを特定する」(→「ギャップを埋めるための具体的な解決策を提案する」) というところをソフトウェアエンジニアとして意見をまとめ、主張することが求められます。SA というと解決策(How)の提案をしているイメージがあるかもしれませんが、私の場合は、お客様もエンジニアですのでそこ解決策はお客様の方でご検討、実施いただくことが多いです。
それよりもプロジェクトを遂行するにあたってやるべきことの「全体像を描く」ようなことが私の活動としては多いです。

例えば、セキュリティ向上していきたいという長期的なプロジェクトがあったとして、具体的なアクションに「IAM の棚卸し」「振る舞い検知ソフトの導入」「定期的なバージョンアップ」などなど山のように「あるべき姿」が出てきますが、今のアクションアイテムって漏れなくダブりなくになっているのか？というのが結局のところお客様のお悩みで、SA の支援が求められる印象です。
NIST などの一般論としてのフレームワークはありますがお客様の既存のネットワークやサーバーの構成を前提としたときに、それらを当てはめて行ったときに、理想系の構成はどうなるのか(To-Be)、現在はどこまでできているのか(As-Is)を資料にまとめて提示します。
また多層防御の世界において全てを完璧に実施することは不可能ですので、優先順位をつけどこから着手すれば良いのか、 As-Is と To-Be のギャップを埋めるための具体的なアクションまで落とし込むことが求められます。
逆に言うと振る舞い検知ソフトの比較検討や導入自体は、お客様も技術に精通されているため、AWS からの支援は特に必要ないという場合が多いように思います。もちろん、ご要望に合わせて勉強会など実施させていただきます。

プリセールスフェーズで SA に求められている役割は、
**事業開発レイヤーから現場技術レイヤー、LoB レイヤーまで様々なレイヤーにおいて、(As-Is) と (To-Be) の分析とギャップを埋めるためのアクションの提案**
であり、「いわゆるプロダクト営業」とは全く異なる活動が求められます。

## AWS の SA はポストセールスももちろん行う

SA は、夢語りもしますが、ビジョンを語って終わるわけではありません。つまり、実際の構築や運用のフェーズでも、継続的な技術支援を行います。

たとえば、対面でホワイトボーディングを行いながら、機能要件・非機能要件・ステークホルダーの洗い出しを行い、プロジェクトを円滑に進められるようにサポートします。要件が固まったら、プロトタイプの提供、ワークショップの開催、SIer の紹介などを通じて、実際の開発フェーズでも技術的な支援を行います。

開発の途中で不明点や疑問点といった Technical Blocker が生じれば、技術的な Q&A に対応しますし、AWS サービスの機能不足が原因であれば、ワークアラウンドの検討やサービスチームへの対応依頼など、あらゆる手段を用いてプロジェクトを前に進めます。

SA は「プリセールス」というイメージを持たれることが多いかもしれませんが、構築フェーズにおいても、一人のソフトウェアエンジニアとしてお客様を支援します。  
そうする理由の背景には、いくつかのストーリーがあります。

一つは、お客様に SA が提示したビジョンを実現するための予算やケイパビリティが十分にないケースです。その場合、そもそも AWS の利用が開始されず、利用額にもつながりません。また、 AWS の機能面の不足によってプロジェクト自体が頓挫してしまうという可能性も考えられます。

また、構築自体はうまく進み、価値のあるシステムを構築することに合意をいただけたとしても、実際に出来上がったものが「当初の期待と少し違う」状態になってしまうと、（AWSの利用者ではなく）その構築したシステムの利用者が離れてしまう可能性があります。

重要なのは、AWS の利用者（システムを開発するエンジニア）だけでなく、そのシステムを利用するエンドユーザーが満足していることです。  
エンドユーザーが満足できなかった場合、そのプロジェクトは AWS にとっても失敗です。なぜなら AWS は従量課金かつオートスケーリングの世界なので、利用されなければ利用額は増えないからです。

そのため、AWS をご提案して採用いただくだけでなく、「利用者にとって価値のあるシステムを実際に構築するところまで」が SA に求められるタスクになります。結果的にお客様のビジネスを最優先に考えたソフトウェアエンジニアのように振る舞うことが求められます。

さらに、ポストセールスフェーズで開発に深く関わることで、お客様の開発現場の方々から上層部の方々まで、どのようなことを考え、どのようにプロジェクトや未来を見ているのか、運用上どのような課題やサイロが存在するのかが見えてくるようになります。

そして、一人のソフトウェアエンジニアとして解決できるサイロや課題が見つけられれば、それを新たな提案につなげることができます。その提案が採択されれば、また AWS の利用は加速していきます。一度構築案件や技術支援案件に入り込むと、「課題発見」→「解決策提案」→「ソリューション構築」→「課題発見」のフライホイールの輪の中に入りやすくなります。

パワポだけ渡してあとはよろしくではないというところが AWS の SA が realistic and tangible かつ個社最適化された提案を行えるところにつながってきていると思います。

## AWS は競合を意識しないが競合案件は存在する

「Amazon は顧客起点で考えるため、競合を意識しない」という言葉を聞いたことがある方も多いのではないのでしょうか。
これは間違いなく事実で前述のように基本的にはお客様の (As-Is) と (To-Be) を分析し提案すること、お客様だけと向き合い続けるので、競合を見る必要はないという考え方をしています。

とはいえ、Google Cloud や Microsoft Azure など、強力なクラウドプロバイダーは他にも存在します。そのため、コンペ（競合案件）になることは珍しくありません。

このときに重要なのは、「古典的な、情弱商法的なセールスプレイを絶対にしない」ことです。

自社製品 A と類似する他社製品 B を比較検討しているお客様に対して、古典的なプロダクト営業スタイルでは、B に比べて A が優れている点を説明し、製品同士を比較するようなアプローチを取るでしょう。
AWS では、このようなやり方は取りません。はっきり言って、最悪のやり方だと考えています。なぜなら一般論としてのカタログ上の機能比較はお客様の技術選定にほとんど影響を与えません。

重要なのは、お客様の状況を前提に「どちらを使うべきか」を SA が一人のソフトウェアエンジニアとして考えることです。  
お客様が本当に成し遂げたいことは何か、どのような課題を抱えているのかを認識し、それを解決できるソリューションとして、AWS の枠に囚われずどのような方法があるのかを真剣に検討します。SA である自分の立場は常にお客様側のソフトウェアエンジニアとしての立場で考えるべきです。

フェアに考察した時に他社製品 B のほうがお客様にとって適切であれば、その案件では B が選ばれるべきです。その場合、そもそも自社製品 A を無理に提案することは絶対にありません。「そのお客様にとって最適なのは B である」と理解できているのであれば、B を使っていただければそれで良いのです。なぜなら、情弱商法的に自社製品 A の強みを訴求したところで、お客様は決して愚かではありませんので、冷静に比較すれば、結局 B を選ぶでしょう。より適した製品が市場にある以上、製品特性自体が変わらない限りどんな営業トークをぶちかまそうと売れるはずがありません。

であれば、SA がやるべきことは別のところにあります。それが、社内で “Advancing Platform Capability” と呼ばれている役割です。

お客様の背景や状況を詳細に理解し、A と B の本質的な違いから敗因を分析します。そのうえで、A のプロダクトを開発しているサービスチームと議論を重ね、最終的には整理されたナラティブとして共有します。具体的に必要な機能や実現可能性を深掘りできていれば、Feature Request や新機能に関する Design Doc を同時にサービスチームへ提出します。

コンペで負けるというのは、多くの場合 SA のセールスプレイに起因するのではなく、AWS が持っている Capability の問題です。より多くのお客様に AWS を活用していただくために、どのような機能や特性が必要なのかをサービスチームに共有し、「顧客起点での情報を提供すること」が SA の仕事です。

一方で、SA の能力によってコンペの勝敗が変わり得るポイントはあります。それが、ソフトウェアアーキテクトとしての視点やソフトウェアエンジニアリングのスキルによってです。

前述の例において自社製品 C や D、あるいは EC2 上に OSS をホストする構成、AWS を利用した 3rd Party SaaS を組み合わせることで、他社製品 B よりもお客様に適したソリューションを構成できたかもしれません。自社製品 A も、カリカリにチューニングすることで、B よりもお客様に適した価値を提供できた可能性があります。

もしそうした選択肢が存在したのであれば、そのコンペは「AWS の持つ Capability で勝てたかもしれない案件」です。このあたりに、Solutions Architect というロール名の由来があるのかもしれません。
この場合には、シンプルにソフトウェアアーキテクト/ソフトウェアエンジニアとしての実力がコンペの勝敗に影響します。そのため、SA がパフォーマンスチューニングや最適化に本気で取り組むケースも珍しくありません。そのようなチューニングレベルになると私の場合は、私自身の技術スキルでは、お客様の望みを叶えられないというような状況も少なくありません。そのような場合には、サービスチームの SDE にがっつり案件支援にご協力をいただくようなパターンもあります。
また、自社製品 A に足りない機能を、ちょっとした Python のプログラムを Lambda 上で動かすだけで補えるのであれば、そのプロトタイプをデモとしてお見せする、参考程度にご共有することで開発を支援するといったアプローチも取れます。

いずれにせよ重要なのは、「AWS が、お客様の視点から見て本当に価値のあるプラットフォームであり続けること」ですし、SA がやるべきはお客様のプロジェクトを成功に導くためのソフトウェアエンジニアリングです。
そのためには、単に AWS の強みを訴求する「プロダクト営業」には一切の価値がありません。 AWS のセールス活動を成功に導けるのは、真にソフトウェアエンジニアとして振る舞う SA だけです。

だから AWS の SA は「AWS の利用額を増やすために、お客様のビジネスを成功に導くソフトウェアエンジニア」なのです。

## SA としてより必要とされるのは技術知識よりも巻き込み力

前述したように、確かに SA は AWS をご利用のお客様のビジネスに最大限貢献する、ソフトウェアエンジニアです。
しかし、一般的なソフトウェアエンジニアほど特定の技術分野に対して深い技術スキルが求められるかというと、それ以上に巻き込み力が重要であると考えています。

冒頭でも説明したように AWS における Solutions Architect のロール定義が「何をするか？」ではなく「何を得るか？」で定義されており、常にビジネスアウトカムを見られるような職種です。

例えば、前述したようなパフォーマンスチューニングによって、重要なお客様のシステムのコスト最適化ができる時に SA 自身がその技術について詳細に理解していて、解決できるのであれば、それはそれで良いでしょう。パフォーマンス改善と言っても、SQL くらいであれば、一般的な SA でもある程度の改善案を出すことができる人が多いと思いますし、それこそ CQRS のような別の手法を提案することもできるでしょう。しかし、そのワークロードが Spark の分散処理だった場合に、CPU 利用率のカリカリチューニングができる SA というのはごく稀です。このように担当 SA の技術スキルだけではお客様を幸せにできない、ビジネスへ貢献できないという場面は少なくありません。

具体的に上記のようなシチュエーションでは、Glue のサービスチームの SDE や、Glue の Expert である Cloud Support Engineer など、社内でもその技術に特化して明るいエキスパートを巻き込みます。
その際に SA として求められるのは二点です。一つ目が Justification の説明、二つ目がエキスパート集団の Organize です。
一点目は巻き込まれたエキスパートからすれば、ボランティア活動となってしまうためそれがボランティアとならないように社内説明をすることが重要です。具体的には、お客様が置かれている状況を適切なナラティブで説明し、いかにこの案件における技術的障壁を取り除くことが重要か。エキスパートの支援を持ってして対処するべき事かということを説明します。これは巻き込んだエキスパートの上司やさらにその上にも通じるようなビジネス的な側面で記述し、活動の正当性をアピールする必要があります。
二点目はエキスパートはお客様の環境やワークロードに詳しくないという点です。単に特定の技術領域に明るいだけで課題解決ができるほど、世の中のシステムはシンプルではありません。適切なエキスパートに対して、現状のシステムアーキテクチャの Overview から、現在どういったことが技術的な障壁になっていて、どのような方針を考えているのか。など、技術者の知見を実際の課題解決に繋げられるように盤面を整地します。これらはあらゆる技術者に通じるよう、技術的な側面で記述するいわば該当する改善プロジェクトに対する Design Doc を書くようなイメージです。

実際に私の案件でもあらゆるエキスパートを巻き込む必要があり、サービスチームの SDE から Specialist SA, Cloud Support Engineer, Professional Services など、40 人程度を特定の 1 つのプロジェクトに巻き込んだ経験もあります。
そのような大規模な巻き込みをするためには、貴重なエキスパートのリソースを巻き込むための正当性の説明、巻き込まれた多くのエンジニアが状況を理解できるような、お客様の As-Is / To-Be の説明が求められます。私はそれらに対して、常に Markdown でのナラティブと PowerPoint で記述した図での説明を組み合わせて記述するようにしています。

また、巻き込み先は技術者だけでなく、リーダーシップや営業チームなどの場合も少なくありません。SA が解決する課題はソフトウェアの側面だけではないからです。お客様の組織上のサイロを解決したい場合、お客様の将来に関するビジョンを実現するために、AWS のリーダーシップを含めて長期的なパートナーシップ契約を結ぶ必要がある場合など、様々な局面で多くの人を巻き込む必要があります。

そのため SA は常にお客様の立場で物事を考え、お客様のプロジェクトの成功のための手段を考えるソフトウェアエンジニアとして振る舞う一方で、あらゆる製品と技術ドメインに対して詳しくなることは不可能です。それよりも、一人のソフトウェアエンジニアとしての立場で、大量のナラティブ（ドキュメント）を早くわかりやすく書き、エキスパートや営業チーム、リーダーシップを巻き込んでいく力こそがお客様のビジネスを AWS の立場として成功に導くために必要な力であるように思います。
